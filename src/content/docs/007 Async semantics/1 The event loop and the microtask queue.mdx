If you had to argue your answer with a colleague, what would you say? The `setTimeout` is set to `0` — does that mean it fires immediately? Chapter 7 is about predicting and shaping async execution. This first lesson installs the runtime model that the rest of the chapter — Promises, `async`/`await` patterns, cancellation — rides on. The single thread is doing a lot of things at once; the question is which one runs next, and the answer is mechanical.

## The three-part runtime

JavaScript runs on a single thread. The runtime decides what runs next using one stack and two queues — a structure called the <Term definition="The runtime mechanism that picks the next chunk of work to run: one macrotask, then a full drain of the microtask queue, then optionally render, then the next macrotask.">event loop</Term>. These three pieces have different rules, so they get named individually.

<ThreePartRuntime />

The **call stack** is where synchronous code runs. When a function is called, a frame is pushed onto the stack; when it returns, the frame pops. Nothing else runs while the stack is non-empty. This is the part of JavaScript you have been working with from the start — it is just the function-call mechanism.

The <Term definition="A continuation scheduled by a settled Promise, an `await` resumption, or `queueMicrotask`. Microtasks drain completely between two macrotasks.">microtask</Term> queue holds Promise continuations — the callback passed to `.then`, the code that comes after each `await` — and any callback handed to `queueMicrotask(fn)`. The defining rule: the microtask queue is **drained completely** between every two macrotasks. The loop does not move on until it is empty.

The <Term definition="A unit of work scheduled via `setTimeout`, `setInterval`, I/O completion, message events, or user input. Processed one per event-loop iteration.">macrotask</Term> queue (sometimes just called the "task" queue) holds `setTimeout` and `setInterval` callbacks, I/O completion callbacks, message events, and user-input handlers. The defining rule: the loop runs **one** macrotask per iteration.

The asymmetry between the two queues is the single most load-bearing fact in this lesson: macrotasks are processed **one per loop iteration**, microtasks are processed **until the queue is empty**. A microtask scheduled by another microtask runs in the same drain. Macrotasks have to wait their turn.

## The tick recipe

The loop is a four-step algorithm you can run on paper.

<Steps>

1. **Run one macrotask to completion.** The initial evaluation of your script counts as the first macrotask. Synchronous code pushes and pops frames on the call stack until the stack is empty.

2. **Drain the microtask queue.** Run every microtask one after another. If a microtask schedules another microtask, that one runs in this same drain — the queue is fully empty before the loop moves on.

3. **(Browser only) Render, if it is time.** The renderer can opt to paint between macrotasks. The exact policy is the browser's call; for our purposes it is enough to know it lives here, after the microtask drain and before the next macrotask.

4. **Pick the next macrotask.** Go back to step 1.

</Steps>

The recipe is also why you cannot afford a runaway microtask chain. Step 2 says the loop will not advance until the microtask queue is empty — so a chain of microtasks that keep enqueueing more microtasks blocks rendering, blocks I/O, blocks user input. Every other task in the system waits. We will cover how to yield long work — by intentionally scheduling a macrotask between batches — when scheduling depth earns its keep in the performance unit later in the course. For now, just hold the cost: the loop's fairness across queues is something microtask-heavy code can break.

## Walking one tick: the canonical interleaving

A diagram of the queues is fine for the vocabulary, but it cannot show *cause and effect* — which line of code mutated which queue, and which queue is drained next. The widget below plays a small program step by step. Each click of "Next" advances one meaningful event-loop step; you can see the call stack push and pop, items enter the microtask and macrotask queues, the queues drain in their respective rules, and the console fill up.

The program is short. Read it once, then step through the widget.

```js
console.log('sync 1');

setTimeout(() => console.log('macro'), 0);

queueMicrotask(() => console.log('micro'));

const f = async () => {
  console.log('sync 2');
  await Promise.resolve();
  console.log('micro 2');
};
f();

console.log('sync 3');
```

<EventLoopWalker />

Three things to notice as you step through. First, **every synchronous statement ran before any queued callback** — `sync 1`, `sync 2`, `sync 3` print before `micro`. The top-level script is itself a macrotask, and the runtime finishes it before draining anything. Second, **the microtask drain ran every microtask before the timer** — `micro` and `micro 2` both printed before `macro`. This is the asymmetry from the previous section made concrete: macrotasks wait their turn; microtasks empty the queue. Third — and this is the move that catches every newcomer — **`await Promise.resolve()` still scheduled a microtask**. The Promise was already resolved by the time `await` saw it, but the continuation did **not** continue inline. It went through the queue. The next section explains why.

## `await` is microtask-paced, not blocking

If you read `await p` as "the function pauses on this line," you are reading it wrong. The correct mental model — the single biggest reframe in this lesson — is:

**`await p` does not block the thread. It pauses the surrounding `async` function and schedules its continuation as a microtask when `p` settles.**

The function is not stuck on that line. The function **returns** at that line. What was *after* the `await` is not "the next statement" anymore; it is a callback the runtime will invoke later, on the microtask queue. Three consequences fall out of this. Each one is a place beginners get confused, so it is worth seeing the three of them as one chord.

**Consequence 1: code before the first `await` runs synchronously on the caller's stack.**

```ts
const greet = async () => {
  console.log('inside, before await');
  await Promise.resolve();
  console.log('inside, after await');
};

console.log('before call');
greet();
console.log('after call');
```

Output: `before call`, `inside, before await`, `after call`, `inside, after await`. The body of `greet` runs top-to-bottom *on the caller's stack* until it hits the `await`. Then `greet` returns a pending Promise to the caller, and the caller's synchronous code (`console.log('after call')`) keeps running. The line after the `await` is now a microtask, and it runs after the caller's synchronous tail finishes.

The reflex to install: when you write an `async` function, the part *before* the first `await` is not asynchronous at all. It is synchronous code that happens to be in an `async` function.

**Consequence 2: a pre-resolved Promise does not skip the queue.**

This is the misconception that fools more newcomers than any other, so it gets stated explicitly:

```ts
setTimeout(() => console.log('macro'), 0);
Promise.resolve().then(() => console.log('micro'));
console.log('sync');
```

Output: `sync`, `micro`, `macro`. The Promise was already resolved by the time `.then` ran, but the callback was **still** scheduled as a microtask. It was never invoked inline. This is what guarantees the loop's ordering rules: if pre-resolved Promises could continue inline, the microtask-vs-macrotask split would depend on whether the Promise happened to be settled yet, and the loop's drain order would be undefined. The runtime closes that hole by always queueing.

This is also why a `setTimeout(..., 0)` cannot beat an awaited resolved Promise. The timer's callback is a macrotask; it has to wait for the microtask drain. No matter how fast the runtime gets at firing 0ms timers, a microtask scheduled before it will always run first.

**Consequence 3: an `async` function with no `await` still returns a Promise.**

```ts
const f = async () => 42;

const result = f();
console.log(result); // Promise { 42 }
console.log(await result); // 42
```

The `async` keyword *is* the Promise wrapping. The body ran synchronously — `42` is the return value at the function level — but the value the caller receives is a Promise. The Promise resolves on the next microtask. There is no "fast path" where an `async` function without `await` skips the Promise wrap.

### `queueMicrotask(fn)`: the explicit microtask scheduler

The two ways we have already seen to schedule a microtask are `Promise.resolve().then(fn)` and the code after an `await`. There is a third, more explicit one: `queueMicrotask(fn)` schedules `fn` to run at the next microtask point, after the current synchronous code but before any pending macrotask.

You will rarely reach for this in app code. Where you see it is in **library code** that needs to batch work — a state library scheduling subscriber notifications, the React scheduler internally, an event-bus implementation that wants to coalesce listeners — when a callback must run "after this synchronous work but before any rendering or I/O" and the author does not want the cost of allocating a settled Promise. The rule of thumb: **recognize `queueMicrotask` when reading library code; do not reach for it in app code**. App-level async work uses `await` and the Promise combinators that the next lesson covers.

## Node specifics

Everything in this lesson — the call stack, the two queues, the tick recipe, `await` as a microtask scheduler — holds in both Node and the browser. The microtask/macrotask model is universal.

Node adds two runtime-specific extras worth name-recognizing. `process.nextTick(fn)` schedules `fn` on a sub-queue that drains **before** the microtask queue on each tick. `setImmediate(fn)` schedules `fn` on a separate macrotask family that fires **after** I/O callbacks in the loop iteration. Neither exists in browsers.

:::caution
`process.nextTick` recursion starves the event loop. A callback that calls `process.nextTick` again will keep the loop pinned in the next-tick drain — no I/O callback, no timer, no microtask will run until the recursion stops. App code in this course reaches for `setTimeout(fn, 0)` when it needs a macrotask and never for `process.nextTick`. `setImmediate` is library territory (Node internals, some test runners); name-recognize, do not reach for.
:::

Node's event loop has explicit internal phases — timers, pending callbacks, poll, check, close — but the course's app code never depends on the phase split. The two-queue model is enough.

```js
console.log('a');
Promise.resolve().then(() => console.log('b'));
setTimeout(() => console.log('c'), 0);

```

<PredictWhy>
Synchronous run prints `a`. As the script runs it also enqueues work: `.then(b)` queues a microtask, `setTimeout(c, 0)` queues a macrotask, `queueMicrotask(d)` queues a second microtask. Because the `.then` ran *before* the `queueMicrotask` line, the microtask queue is `[b, d]` in that order. The script's macrotask finishes; the runtime drains microtasks FIFO: `b`, then `d`. Then one macrotask: `c`. The `0` on the timer does not race the microtask queue — microtasks are drained completely before any macrotask runs.
</PredictWhy>

## Practice: three traces

Three programs of increasing complexity. The point is not to memorize outputs; the point is to run the tick recipe in your head and have it produce the right answer.

<PredictOutput
  instructions="Two awaits and a timer. Trace it step by step."
  expected={`1

3

>

```ts
const f = async () => {
  console.log('1');
  await Promise.resolve();
  console.log('2');
  await Promise.resolve();
  console.log('3');
};

setTimeout(() => console.log('timer'), 0);
f();
console.log('script end');
```

<PredictWhy>
`setTimeout` enqueues a macrotask. `f()` runs synchronously up to the first `await` — `1` prints — then `f` returns, with `console.log('2')` and the rest as a queued microtask. The script's synchronous tail runs — `script end`. Now the microtask drain: the continuation runs, prints `2`, hits the second `await`, returns again with `console.log('3')` queued as another microtask. The drain continues — `3` prints. Microtask queue is empty. Next macrotask: `timer`.
</PredictWhy>

</PredictOutput>

<PredictOutput
  instructions="A microtask schedules another microtask. What does the drain do?"
  expected={`sync
outer micro
inner micro
timer`}
>

```ts
queueMicrotask(() => {
  console.log('outer micro');
  queueMicrotask(() => console.log('inner micro'));
});
setTimeout(() => console.log('timer'), 0);
console.log('sync');
```

<PredictWhy>
Synchronous run prints `sync`. Microtask queue: `[outer micro]`. Macrotask queue: `[timer]`. The drain starts. `outer micro` runs — and *enqueues* `inner micro`. The drain does not stop; it runs until the queue is empty. So `inner micro` runs next, in the same drain. Only then does the loop pick the macrotask: `timer`. The "drain completely" rule made concrete.
</PredictWhy>

</PredictOutput>

<PredictOutput
  instructions="A macrotask that schedules a microtask and another macrotask. Each loop iteration is `macrotask → full drain`."
  expected={`sync
A
B
C`}
>

```ts
setTimeout(() => {
  console.log('A');
  Promise.resolve().then(() => console.log('B'));
  setTimeout(() => console.log('C'), 0);
}, 0);
console.log('sync');
```

<PredictWhy>
The script is the first macrotask: `sync` prints. Macrotask queue: `[setTimeout(A)]`. Microtask queue empty, drain is a no-op. Next iteration: pop the macrotask. It runs — `A` prints, `.then(B)` enqueues a microtask, `setTimeout(C, 0)` enqueues a macrotask. Macrotask done. Drain microtasks: `B` runs. Microtask queue empty. Next iteration: pop the macrotask: `C`. The lesson within the lesson: each loop iteration is `macrotask → full microtask drain`, so any microtask queued inside a macrotask runs *before* the next macrotask — even one that was queued earlier in the same iteration.
</PredictWhy>

</PredictOutput>

## Three consequences of `await`-as-microtask

Six statements to close out. Each one probes one of the three consequences from above.

<TrueFalse instructions="Each statement is about the runtime model from this lesson — call stack, microtask queue, macrotask queue, and the tick recipe.">

  <Statement answer="true">
    Code before the first `await` in an `async` function runs synchronously on the caller's stack.
    <TfWhy>The function body executes top-to-bottom until it hits an `await`. Only then does it return a pending Promise to the caller. The code before the `await` is not asynchronous at all — it just happens to live in an `async` function.</TfWhy>
  </Statement>

  <Statement answer="false">
    Awaiting a Promise that is already resolved continues execution inline, without yielding to the event loop.
    <TfWhy>Even a pre-resolved Promise schedules its continuation as a microtask. The continuation is never invoked inline. This is what guarantees the microtask-vs-macrotask ordering — if pre-resolved Promises could skip the queue, the drain order would be undefined.</TfWhy>
  </Statement>

  <Statement answer="true">
    A `setTimeout(fn, 0)` cannot run before any pending microtask, because the microtask queue is drained completely between two macrotasks.
    <TfWhy>The loop runs *one* macrotask, then drains *all* microtasks. A 0ms timer is still a macrotask; no matter how fast the runtime is at firing it, a microtask scheduled before it runs first.</TfWhy>
  </Statement>

  <Statement answer="false">
    An `async` function with no `await` runs synchronously and returns its value directly, without wrapping it in a Promise.
    <TfWhy>Every `async` function returns a Promise — the `async` keyword *is* the wrapping. The body runs synchronously, but the return value the caller receives is a Promise that resolves on the next microtask.</TfWhy>
  </Statement>

  <Statement answer="true">
    A microtask that schedules another microtask runs the new one before the next macrotask.
    <TfWhy>The drain runs until the microtask queue is empty. Microtasks enqueued during the drain join the same drain. The loop does not move on to the next macrotask until every microtask — including the ones queued mid-drain — has run.</TfWhy>
  </Statement>

  <Statement answer="false">
    `queueMicrotask(fn)` and `setTimeout(fn, 0)` are equivalent ways to defer a callback to the next event-loop iteration.
    <TfWhy>`queueMicrotask` enqueues on the **microtask** queue, which runs before any macrotask. `setTimeout(fn, 0)` enqueues a **macrotask**, which has to wait for the microtask drain and one full loop iteration. They differ in priority, not just in syntax.</TfWhy>
  </Statement>
