In the last lesson you wired a form to a Server Action with a single prop: `<form action={createInvoice}>`. Submit, and React serializes the inputs into `FormData`, posts to the action, and runs it. That part works. But the form you built is half-finished on purpose, and it's broken in three ways you'd never ship:

1. The action takes about 400ms to round-trip. You click "Save invoice" and nothing on the screen moves. So you click again — and now there are two invoices.
2. The action returns `{ ok: false, error: { fieldErrors: { email: ['Enter a valid email address.'] } } }`. The user typed a bad email, the server caught it, and the user sees... nothing. The form threw the action's answer away.
3. They fix the email and resubmit successfully. The old "Enter a valid email address." should disappear. There's no code to clear it.

Every one of these is the same underlying gap: the form posts to the action but never *listens* to it. It doesn't know the action is in flight, and it doesn't read what comes back. One hook closes all three gaps at once — <Term definition="React hook for the component that owns a form: it wraps the action and tracks each submit.">`useActionState`</Term>. It hands you three things: the **latest result** the action returned, an **in-flight flag**, and a **bound action** you put on the form in place of the raw one.

By the end of this lesson you'll have the canonical form shape — the exact wiring every form in the rest of the course copies. This is the central mechanics lesson of the chapter, so the wiring is the point and we'll build it one piece at a time.

One thing to set aside first. A submit button buried three components deep that needs the pending state without you drilling a prop through every layer is a different problem, solved by a different hook (`useFormStatus`) in the next lesson. And the case where you want the screen to update *before* the server even answers is `useOptimistic`, two lessons out. This lesson is the form owner's hook — the one the form component itself calls.

## The action signature has to change first

You can't call the hook without changing the action, so this comes before anything you render. Think of it as the price of admission.

When you wrap an action in `useActionState`, the hook prepends an argument. The action's first parameter is no longer `FormData` — it's the **previous state**, the value the hook last handed back. `FormData` slides down to second. The signature you wrote last chapter changes shape:

<CodeVariants>
  <CodeVariant label="Before">
    <div data-mark-color="red">

    ```tsx del={2} title="app/invoices/actions.ts"
    export async function createInvoice(
      formData: FormData,
    ): Promise<Result<Invoice>> {
      // ...parse, authorize, mutate, revalidate, return
    }
    ```

    </div>
    **The shape from the last lesson:** the action takes `FormData` directly, as its only argument.
  </CodeVariant>

  <CodeVariant label="After — wrapped">
    <div data-mark-color="green">

    ```tsx ins={2-3} title="app/invoices/actions.ts"
    export async function createInvoice(
      prevState: Result<Invoice> | null,
      formData: FormData,
    ): Promise<Result<Invoice>> {
      // ...parse, authorize, mutate, revalidate, return
    }
    ```

    </div>
    **Wrapped in the hook:** `prevState` is now the first parameter; `FormData` is second.
  </CodeVariant>
</CodeVariants>

That's the whole change to the action. The body is untouched: it still reads the `formData` argument with `Object.fromEntries(formData)` and runs it through `safeParse`, exactly as you built it in the last chapter. The parse, the authorization check, the database write, the revalidation, the returned `Result` — none of it moves. Only the parameter list grows by one.

So what is `prevState` for? It's the result the action returned the *last* time it ran — `null` on the very first render, before any submit. For ordinary create-and-update forms you ignore it. The new result simply replaces the old one; the action doesn't care what came before. You'd only read it when the action's output genuinely depends on its own history — a multi-step wizard that accumulates answers across submits, the kind of flow the next chapter is built around. For everything in this chapter, treat `prevState` as a parameter you're required to declare and free to ignore.

:::note
The last chapter already named this exact signature when it introduced the three ways to call an action — it called `prevState` "the framework-bound first argument" and promised the wiring would come later. This is later. Nothing new is being invented; the shape you were told about is now plugged in.
:::

## Wiring the hook: state, formAction, isPending

With the action ready, you can call the hook. It returns three things, and the cleanest way to meet them is all at once — see the whole shape, then spend the rest of the lesson using each piece:

<CodeTooltips tooltips={{
  state: 'The latest Result the action returned, or the initial value (null here) before any submit.',
  formAction: "A bound version of the action. Hand THIS to the form's action prop — not the raw createInvoice.",
  isPending: 'true from the moment React calls the action until its Result comes back. Drives in-flight UI.',
  useActionState: 'React hook. Wraps an action, owns its latest result and in-flight flag, returns a bound action to hand the form.',
}}>
```tsx
const [state, formAction, isPending] = useActionState(createInvoice, null);
```
</CodeTooltips>

The hook comes from React itself: `import { useActionState } from 'react';`. Hold that detail — the *next* lesson's hook, `useFormStatus`, imports from `react-dom`, not `react`, and mixing them up is a common stumble. `useActionState` is `react`; pending state for the form owner.

It's also client-only. That's why the form file carries `'use client'` at the top — which it already does, since the last lesson made the form a Client Component to use the `action` prop. No new directive; just a reason the existing one has to stay.

The second argument, `null`, is the **initial state** — the value `state` holds before the first submit. For a create form, `null` is the natural reflex: there's no result yet. It pairs neatly with how you'll read errors in a moment — every error check is gated on `state?.ok === false`, and that's simply `false` while `state` is still `null`, so nothing renders on the first paint. There's one alternative worth naming once: an *edit* form can seed the initial state with the existing record as `ok(invoice)`, so `state.data` carries the server-confirmed values across repeated saves. That's a project-chapter concern; this chapter standardizes on `null` for its first form, and so will you.

There's a rarely-used third argument, `permalink` — a URL for the no-JavaScript fallback path. Omit it by default; the progressive-enhancement lesson at the end of the chapter is where it earns a sentence. Don't reach for it now.

Now the single most important line in the whole lesson — and the one people get wrong first. The form must use the **bound** `formAction`, not the raw `createInvoice`:

<CodeVariants>
  <CodeVariant label="Broken">
    <div data-mark-color="red">

    ```tsx "createInvoice"
    <form action={createInvoice}>
    ```

    </div>
    **The raw action runs, but the hook never sees the submit** — so `state` and `isPending` never update, and the form stays dead.
  </CodeVariant>

  <CodeVariant label="Correct">
    <div data-mark-color="green">

    ```tsx "formAction"
    <form action={formAction}>
    ```

    </div>
    **The bound `formAction` routes the submit through the hook's state machine,** so every return value updates on each submit.
  </CodeVariant>
</CodeVariants>

This is the same point the last lesson drilled with `<form action={fn}>`: the function reference *is* the contract. Here that point is sharper. `createInvoice` and `formAction` both run your action — but only `formAction` reports back to the hook. Hand the form the raw one and the action still fires, the invoice still saves; you just get nothing back, because the hook never learns a submit happened. `state` stays `null`, `isPending` stays `false`, and you're back to the broken form from the last lesson with extra steps. The bound action is the whole point of calling the hook.

## Rendering the result: the banner and field errors

Now the form can read what the action sends back. This is where the `Result` contract from the last chapter pays off — and where one access path is worth getting exactly right, because it's the detail most people overcomplicate.

Start with the discriminator. `state` is a <Term definition={"A union where one shared field — here, ok — tells you which branch you're in.\nChecking ok === true unlocks data; ok === false unlocks error.\nTypeScript narrows the type once you've checked the tag."}>discriminated union</Term> — `{ ok: true; data }` on success, `{ ok: false; error }` on failure — and it might also be `null` before the first submit. TypeScript won't let you touch `state.error` until you've proven you're on the failure branch. So the gate on every error read is `state?.ok === false`. Skip it and you get both a compile error and a runtime crash: `null.error` and `{ ok: true }.error` are both nonsense. This is the same discriminated-union reflex you built much earlier in the course; the `ok` tag is what selects the branch.

With that gate, the form-level banner is a single gated paragraph — a plain `<p role="alert">` carrying the action's message:

<div data-mark-color="blue">

```tsx "state.error.userMessage"
{state?.ok === false && (
  <p role="alert" className="text-destructive text-sm">
    {state.error.userMessage}
  </p>
)}
```

</div>

Notice what the form does *not* do here: it doesn't write any error copy. The `userMessage` was authored by the action — the last chapter made this a hard rule. The action owns the human sentence for a business-rule failure ("That invoice number is already in use."); the form renders it verbatim. No rephrasing, no "Something went wrong." fallback invented at the UI layer. If a `userMessage` is missing, the bug is in the action that forgot to return one — not in the form. The form is a renderer, not an author.

Field errors follow the same discipline, but the access path deserves a closer look. Let's pull it into a small presentational component so the path lives in exactly one place and the form's JSX stays readable. Step through it:

<AnnotatedCode lang="tsx" maxLines={18} code={`
// field-error.tsx — presentational, no hook, no 'use client'
export const FieldError = ({
  id,
  messages,
}: { id: string; messages?: string[] }) => {
  if (!messages?.length) return null;
  return (
    <p id={id} role="alert" className="text-destructive text-sm">
      {messages[0]}
    </p>
  );
};

// call site, inside new-invoice-form.tsx
<FieldError
  id={emailErrorId}
  messages={state?.ok === false ? state.error.fieldErrors?.email : undefined}
/>
`}>
  <AnnotatedStep meta="{2-5}" color="blue">
    A plain presentational component — an arrow bound to `const`, no hook, no client directive. It takes a stable `id` and an optional `messages` array. The prop type is `string[]` because that's exactly what a field maps to in the result; the component receives the field's whole array of messages.
  </AnnotatedStep>

  <AnnotatedStep meta="{6}" color="blue">
    If there are no messages, render nothing. The `?.length` guard handles both an absent array and an empty one in a single check.
  </AnnotatedStep>

  <AnnotatedStep meta="{7-11}" color="blue">
    Render the first message. A field's array can hold several, but one line under the input is the convention; pick `messages[0]`. `role="alert"` announces the error to assistive tech.
  </AnnotatedStep>

  <AnnotatedStep meta={`"state.error.fieldErrors?.email"`} color="orange">
    Here's the path, and the one detail to nail. `fieldErrors` is a flat map — a `Record<string, string[]>` the action built from the parse failure. A field name maps *directly* to its array of strings. So you read `fieldErrors?.email` and you have the messages. There is no extra `.errors` step under the field — the field *is* the array, not an object wrapping one. Optional-chain the map (it's absent on success), and let `<FieldError>` pick the first message itself.
  </AnnotatedStep>
</AnnotatedCode>

That fourth step is the whole reason this section exists, so let me say it plainly. The action built `fieldErrors` by calling `z.flattenError(parsed.error).fieldErrors` — a deliberate choice the last chapter argued for, precisely so the value would be a *flat* `Record<string, string[]>`. The flat shape is what makes the read short: field name in, array of messages out. <Term definition={"Zod's flat projection of a parse failure. Returns { formErrors, fieldErrors }; fieldErrors maps each field name straight to its string[] of messages — no nesting under each field."}>`fieldErrors`</Term> is `fieldErrors?.fieldName` and you're done. If you ever find yourself reaching for `fieldErrors.email.errors` or digging into a nested object per field, you've reconstructed a shape the last chapter specifically chose *not* to ship. Resist it. The field maps straight to the array.

Step back and look at where the strings actually come from, because it clarifies the form's job. The *text* of a field error — "Enter a valid email address." — was authored once, in the Zod schema, when you declared the field's validation. The action's parse failure runs that schema; `flattenError` projects the messages into the flat map; the form reads the map and renders. One set of strings, flowing through three layers — schema authors, action projects, form renders. The form never invents copy. It's the last stop, not the source.

One accessibility detail the project will inherit. A field with a possible error should tell assistive technology two things: that it's currently invalid, and where to find the message. That's `aria-invalid` on the input and `aria-describedby` pointing at the `FieldError`'s `id`. The IDs have to be stable across the server render and the client hydration, so you mint them with React's `useId()` hook rather than hardcoding strings. You'll see this on the email field in the assembled form below; the earlier accessibility lesson is the baseline owner if you want the full rationale.

And symptom 3 from the opening — stale errors lingering after a fix — turns out to need no code at all. The next submit replaces `state` with a fresh `Result`. If that submit succeeds, `state.ok` is now `true`, and every block gated on `state?.ok === false` simply stops rendering. The old field error and the old banner vanish because the condition that drew them is gone. There's no manual reset to write; the new result clearing the old one is just how the hook works.

## Pending state: disabling the button and stopping double-submits

`isPending` is the smallest of the three returns and it resolves the very first symptom. It's `true` from the instant React invokes the action until the `Result` comes back. Wire it to the submit button:

<div data-mark-color="blue">

```tsx "disabled={isPending}"
<button type="submit" disabled={isPending}>
  {isPending ? 'Saving…' : 'Save invoice'}
</button>
```

</div>

That single `disabled={isPending}` is also your double-submit fix, and it's automatic. A disabled button doesn't fire its click. So the impatient second click — the one that created a second invoice — lands on a button that's already disabled while the first submit is in flight. React swallows it. The label swap from "Save invoice" to "Saving…" tells the user *why* the button went quiet, which is the other half of the fix: the screen finally moves when they click.

If you want to freeze the *entire* form during submit — every input, not just the button — the cleanest blunt instrument is to wrap the controls in `<fieldset disabled={isPending}>`. A disabled fieldset disables every form control inside it in one stroke. It's optional; disabling the submit button is the minimum that prevents the double-submit. Reach for the fieldset when half-edited inputs mid-flight would confuse the user.

Be clear-eyed about the limit, though. `isPending` stops the *double-click* race — one user, one impatient finger. It does nothing for a network retry, a refresh-and-resubmit, or two browser tabs. For mutations that must never run twice because they cost money or send mail, the idempotency-key pattern from the last chapter is still the real defense. `isPending` is a UX guard, not a correctness guarantee.

Step 4 is worth pausing on, because it's a behavior you get for free from the last lesson. Recall that React auto-resets an uncontrolled form *on success*. On **failure** it does the opposite — it leaves the inputs alone. The values the user typed stay put. That's exactly what you want: the email was rejected, so the user fixes that one field and resubmits without retyping the other five. The failure path keeping the data isn't an accident; it's the platform matching the obvious UX.

(There's a sibling case — keeping the typed values after a *successful* save, the way an edit form should — that needs `state.data` fed back as `defaultValue`. The last lesson promised this fix would land here; it's a small enough piece that the project chapter writes it in full, on a real edit form. ## The canonical form shape

Time to assemble the pieces into one component — the shape the rest of the course copies. Read it as a whole first, then step through the parts:

<AnnotatedCode lang="tsx" maxLines={18} code={`
'use client';

import { useActionState, useId } from 'react';
import { createInvoice } from './actions';
import { FieldError } from './field-error';

export const NewInvoiceForm = () => {
  const [state, formAction, isPending] = useActionState(createInvoice, null);
  const emailErrorId = useId();
  const emailErrors =
    state?.ok === false ? state.error.fieldErrors?.email : undefined;

  return (
    <form action={formAction} className="space-y-4">
      {state?.ok === false && (
        <p role="alert" className="text-destructive text-sm">
          {state.error.userMessage}
        </p>
      )}

      <label className="block">
        Customer email
        <input
          name="email"
          type="email"
          aria-invalid={Boolean(emailErrors)}
          aria-describedby={emailErrorId}
        />
      </label>
      <FieldError id={emailErrorId} messages={emailErrors} />

      <label className="block">
        Total
        <input name="total" type="number" step="0.01" />
      </label>

      <button type="submit" disabled={isPending}>
        {isPending ? 'Saving…' : 'Save invoice'}
      </button>
    </form>
  );
};
`}>
  <AnnotatedStep meta={`"'use client'"`} color="blue">
    The form is a Client Component, because `useActionState` is a client-only hook. The directive sits at the top of the file.
  </AnnotatedStep>

  <AnnotatedStep meta="{8}" color="blue">
    One hook owns all three concerns — `state`, the bound `formAction`, and `isPending` — initialized with `null`, meaning no result yet.
  </AnnotatedStep>

  <AnnotatedStep meta={`"action={formAction}"`} color="orange">
    The bound action goes on the form — never the raw `createInvoice`. This is the wire that connects the form to the hook's state machine.
  </AnnotatedStep>

  <AnnotatedStep meta="{15-19}" color="blue">
    The form-level error: gated on `ok === false`, rendering the action's `userMessage` verbatim. The form authors no copy of its own.
  </AnnotatedStep>

  <AnnotatedStep meta={`{21-30} "emailErrors"`} color="orange">
    The email field plus its `FieldError`. `emailErrors` is read once at the top (lines 10-11) — the flat `fieldErrors?.email` path, guarded by the discriminator. The `name` is the contract with the schema, the same string the action parses; `aria-invalid` and `aria-describedby` (with a `useId()` id) wire the field to its message for assistive tech.
  </AnnotatedStep>

  <AnnotatedStep meta="{37-39}" color="blue">
    The submit button driven by `isPending`: disabled while in flight, with a label that swaps to 'Saving…'. The disabled state is also the double-submit guard.
  </AnnotatedStep>
</AnnotatedCode>

That's it. That component is the deliverable of this lesson, and it's the reason the lesson exists: **every form in the rest of the course is this shape.** The project chapter extends it — more fields, a reusable submit button factored out in the next lesson, optimistic updates layered on in the lesson after where they earn their weight — but the skeleton never changes. Client Component, hook at the root with `null`, bound action on the form, banner gated on the discriminator, named inputs each with a `FieldError` reading the flat map, submit button driven by `isPending`. Internalize this one and you've internalized the chapter.

## Practice: wire the hook yourself

The syntax here is load-bearing, and wiring it once by hand sticks better than reading it. To make every connection *visible the instant the form paints*, this form picks up mid-story: the user already submitted once with a bad email and the server rejected it, so the form re-renders with that **prior failure as its initial state** — the seeded-state move the edit-form aside mentioned, here in miniature. Everything you need to render is already in `state` from the first paint; no submit required to see your wiring work. The same four connections from the canonical shape apply.

  expect(formText()).toContain('Check the form.');
});

test('with a settled result (isPending false) the button is enabled and reads "Save profile"', () => {
  // state is a settled failure, so nothing is in flight and isPending is false:
  // the button must be enabled with its idle label. This catches a hard-coded
  // disabled and a label missing the isPending ? 'Saving…' : 'Save profile' swap.
  expect(getButton()?.disabled).toBe(false);
  expect(getButton()?.textContent).toContain('Save profile');
  expect(getButton()?.textContent).not.toContain('Saving');
});
`}
/>

If you get stuck, the four tasks map one-to-one onto the canonical form above — the hook call (seeded here with `previousResult` instead of `null`), the `action={formAction}` swap, the `disabled={isPending}` button, and the guarded `fieldErrors?.email?.[0]` read.
