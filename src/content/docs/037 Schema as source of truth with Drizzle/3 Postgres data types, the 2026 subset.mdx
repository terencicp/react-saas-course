Last lesson you wrote `uuid()`, `text()`, `integer()`, and `timestamp()` into your first table, and you took every one of them on faith. You never asked why that type was the right one. This lesson answers that question.

Postgres ships around forty built-in types, and that number scares people into thinking they need to learn all forty. They don't. A 2026 SaaS app reaches for about eight of them, day in and day out. The real skill isn't memorizing the catalogue; it's knowing those eight defaults cold, and knowing the one tempting-but-wrong type that shadows each of them. Almost every "we stored money as a float and the books don't balance" incident traces back to picking the type that *looked* right.

That's the frame for the whole lesson. A SaaS app has many kinds of column: a name, a price, a timestamp, an ID, a status, a flexible payload, a list of tags, an IP address. For each one you'll learn the correct Postgres type, the Drizzle builder that maps to it, and the trap in the obvious alternative. We keep building the same domain from last lesson, `organizations` and `invoices`, and add the columns a real invoice actually needs: an amount, a status, dates, a tag list, a webhook payload.

One promise up front, so you know where all of this is heading: the whole lesson condenses to a single lookup table at the end, with the kind of column on the left and the builder you reach for on the right. That table is what you'll keep. Everything between here and there is the reasoning that makes each row of it stick.

## Names and free text: `text` is the default

Start with the simplest column there is: a name, a note, a description, any human-readable string. The rule is flat, and it surprises almost everyone coming from another database.

**Strings are `text`, with no length number.**

If you've written SQL before, your hand is already reaching for `VARCHAR(255)`. That reflex is muscle memory from MySQL and SQL Server, where an unbounded string carries a real cost. In Postgres it doesn't. A `text` column and a `varchar(n)` column have **identical performance and identical storage**: Postgres stores them the exact same way under the hood. The length cap buys you nothing at the database level. This is the single most counterintuitive fact in this lesson, so it's worth pausing on.

The reflex isn't just neutral, it's actively harmful. Compare the two.

<CodeVariants>
  <CodeVariant label="varchar(n) — the reflex">
    <div data-mark-color="red">

    ```ts "varchar({ length: 255 })"
    name: varchar({ length: 255 }),
    ```

    </div>

    **Feels safe, buys nothing.** It looks like type safety, as if it guaranteed names can't exceed 255 characters, but that 255 is arbitrary. It makes nothing faster or smaller, and the day the business says a name can be 300 characters you ship a database migration to lift a limit that never earned its keep.
  </CodeVariant>

  <CodeVariant label="text — the default">
    <div data-mark-color="green">

    ```ts "text()"
    name: text(),
    ```

    </div>

    **Variable length, same performance.** No cap to maintain, no migration when the limit changes. If there genuinely *is* a maximum length, it's a validation rule rather than a storage rule, and it lives somewhere else entirely.
  </CodeVariant>
</CodeVariants>

Be precise about that last point, because it's the real lesson here. A length limit is a **user-experience concern**, not a storage concern. "Names must be 100 characters or fewer" exists to give the user a friendly error in a form, not to make Postgres truncate their data mid-word. So the cap belongs at the edge of your system, in the validation layer, where a too-long name produces a clear message before it ever reaches the database. You'll build that layer with <Term definition="A TypeScript-first schema validation library. You declare the shape and rules of incoming data once, and it validates and types it at your API boundary.">Zod</Term> in a later chapter. There is one source of truth for "max 100 chars," and it's the validator, not the column.

You'll see this division of labor again and again: the database type says what *can* be stored, and the validator says what's *allowed* in. They're partners, not substitutes. The column stays `text`; the rule lives in Zod.

That's why `varchar(n)` is never the right answer in this course, and `char(n)`, its fixed-width cousin, even less so. `char(n)` blank-pads every value out to `n` characters, which is almost never what anyone wants. Name them once, here, as the types you now know to walk past.

## Money and numbers: the four-way decision

This is the highest-stakes section in the lesson, so read it twice. Get a text column wrong and a name is slightly too long. Get a money column wrong and your numbers are quietly, permanently incorrect, with nothing crashing to tell you.

There are four number types in play, and the whole section is one decision among them: `integer` and `bigint` for whole numbers of different sizes, `numeric` for money and anything where exactness is the point, and the floating-point pair (`real` and `double precision`), which you'll learn to keep *away* from money. We'll start with the most expensive mistake and work outward.

Start with the disaster, because money is where the real damage lives. Suppose you store a price as a floating-point number, the same kind of number JavaScript uses for `0.1 + 0.2`. Run that in any console and you get `0.30000000000000004`, because floats can't represent most decimal fractions exactly; they store the closest binary approximation. One price with one tiny rounding error doesn't matter. But sum thousands of invoice lines and the errors accumulate, the cents drift, and the total a customer is charged stops matching the sum of the rows. The books don't balance, and the code never threw an error: it just computed the wrong answer. That's the worst kind of bug, invisible until an accountant finds it.

The fix is a type built for exactly this. Here's the money decision laid bare.

<CodeVariants>
  <CodeVariant label="Float — the silent corruption">
    <div data-mark-color="red">

    ```ts "doublePrecision()"
    amountDue: doublePrecision(),
    ```

    </div>

    **Looks fine, runs fine, lies.** `doublePrecision` (and its smaller sibling `real`) store binary floating point. The math executes without error, the totals are subtly wrong, and the bug ships precisely because nothing complains. Never use it for money.
  </CodeVariant>

  <CodeVariant label="numeric — exact">
    <div data-mark-color="green">

    ```ts "numeric({ precision: 12, scale: 2 })"
    amountDue: numeric({ precision: 12, scale: 2 }).notNull(),
    ```

    </div>

    **Exact decimal, the course default for prices.** <Term definition="The exact decimal type — Postgres NUMERIC / DECIMAL. Stores the actual digits, not a binary float, so there is no rounding error.">`numeric`</Term> stores the literal digits, so there's no binary rounding to corrupt the answer. `precision: 12, scale: 2` means up to twelve significant digits total, two of them after the decimal point, so ten digits before the decimal and two after. That comfortably covers any SaaS invoice amount.
  </CodeVariant>
</CodeVariants>

That `amountDue` upgrade is deliberate, and worth calling out: last lesson you stubbed `amountDue` as an `integer` placeholder, and from here on money is `numeric`. The `integer` was a hold-the-spot value while you learned the table shape. Now you know the right type, and `numeric({ precision: 12, scale: 2 })` is the standing default for any price column in the course.

Two new words just showed up, and they're worth a beat. <Term definition="The total number of significant digits stored — both sides of the decimal point combined.">Precision</Term> is the total count of significant digits the column stores. <Term definition="The number of digits stored after the decimal point.">Scale</Term> is how many of those sit after the decimal point. So `precision: 12, scale: 2` means twelve digits total, two of them cents.

### The surprise: `numeric` comes back as a string

Here's a detail that trips people up the first time, and it's the same shape of surprise you met last lesson. When you read a `numeric` column, Drizzle hands it to your TypeScript as a **`string`**, not a `number`.

That feels wrong until you see why it's right. A JavaScript `number` *is* a 64-bit float, the very thing `numeric` exists to avoid. If Drizzle handed you a `number`, it would push your exact decimal back through floating-point math and reintroduce the rounding bug you chose `numeric` to escape. So instead it gives you the digits faithfully, as a string, and you do money arithmetic with a decimal library that never touches raw `+`. The string isn't a defect; it's the precision guarantee reaching all the way up into your TypeScript. Never `parseFloat` a money value to add it.

Recall the "two worlds" idea from last lesson, where the TypeScript name `amountDue` and the SQL name `amount_due` differ across the boundary. This is the same boundary showing up in a second way: not the *name* this time, but the *runtime type*. `numeric` is the one type where TypeScript and SQL disagree about what the value is, and now you know it's on purpose. (You'll see exactly how Drizzle decides this when we reach inferred types at the end of the chapter. For now, just expect the string.)

### `integer` versus `bigint`

Not every number is money. For plain counts, like a quantity, a number of seats, or a retry counter, reach for `integer`. It's a 32-bit signed integer, so it tops out around ±2.1 billion, which is plenty for almost anything you'd count in an app.

When a value can outgrow two billion, such as a high-volume event counter or a Unix timestamp stored as milliseconds, step up to `bigint`, the 64-bit integer. One wrinkle to know about: a true 64-bit integer is larger than JavaScript's `number` can safely hold, so Drizzle's `bigint` builder takes a `mode`. `bigint({ mode: 'number' })` gives you an ordinary `number` and is safe up to about 2⁵³; `bigint({ mode: 'bigint' })` gives you a JavaScript `BigInt` for the full 64-bit range. Reach for `mode: 'number'` by default and only switch when you genuinely need the top end. We won't dwell on `bigint` here; it comes back as a primary-key strategy in a couple of lessons, where the choice has real weight.

## Booleans: two flags or one enum

`boolean` for true or false. That part is obvious, and short:

```ts
isArchived: boolean().notNull().default(false),
```

The real content here isn't the type; it's a modeling decision that comes up the moment you have more than one true/false fact about a row. Do you reach for several booleans, or one of something else?

The test is whether the facts are independent. Take an invoice. `isPaid` and `isSent` are **orthogonal**: an invoice can be sent but unpaid, paid but never sent, both, or neither. Every combination is a real state, so two booleans is exactly right; they vary freely.

Now take a different fact about that invoice: its *status*, which is exactly one of `draft`, `sent`, `paid`, or `void` at any moment. These are **mutually exclusive**, since it can't be both `draft` and `paid`. Model that with booleans (`isDraft`, `isPaid`, `isVoid`, and so on) and you've invited nonsense: nothing stops two of them being `true` at once, an impossible state your database now permits. A mutually-exclusive set of states wants a single column that holds exactly one value, and that's an enum, which is two sections away. Use booleans when the facts are independent, and one enum when exactly one can be true at a time.

## Time: `timestamptz` is the only timestamp

Every timestamp you write in this course follows one hard rule:

**Every timestamp is `timestamp({ withTimezone: true })`.**

That maps to the Postgres type <Term definition="Postgres timestamp with time zone. Stores an absolute instant in UTC and converts on input and output, so it means the same moment everywhere.">`timestamptz`</Term>, which stores an absolute instant in UTC and converts on the way in and out. There's a plain `timestamp` without the time-zone option, and it is *never* the right answer here. It's the single most common timezone bug in Drizzle codebases, so let's make the bug vivid before the rule.

Picture a `createdAt` stored as a plain `timestamp`. On your laptop it works flawlessly: you write "2pm," you read "2pm." Then you deploy to a server in another region, or a user in a different timezone loads the row, and "created at 2pm" silently means 2pm in some other zone, off by hours. The data looks fine. It type-checks. It passes every test you ran on one machine. The bug stays hidden until the exact moment a region boundary gets crossed, which is usually production.

<CodeVariants>
  <CodeVariant label="Plain timestamp — the hidden bug">
    <div data-mark-color="red">

    ```ts "timestamp()"
    createdAt: timestamp(),
    ```

    </div>

    **A wall-clock with no zone.** It stores "2pm" with no record of *2pm where*. Correct on exactly one machine, and wrong the day you deploy across regions or a user in another timezone reads it. It stays invisible until that day.
  </CodeVariant>

  <CodeVariant label="timestamptz — always">
    <div data-mark-color="green">

    ```ts "{ withTimezone: true }"
    createdAt: timestamp({ withTimezone: true }).defaultNow().notNull(),
    ```

    </div>

    **Stores an absolute instant in UTC.** Converts on read, so it means the same moment in every timezone on earth. This is the only timestamp shape the course writes. (`.defaultNow()` fills in the current time on insert and `.notNull()` forbids an empty value. Both are named here at face value; the next lesson covers them. This is just the canonical `createdAt` line, so it's familiar when you see it.)
  </CodeVariant>
</CodeVariants>

The rule isn't dogma. An event happened at one real instant, and `timestamptz` is the only type that records *which* instant. There is exactly one honest exception, worth knowing so you recognize it when it genuinely applies: a recurring **local** time of day that must stay local no matter the zone. "The store opens at 9am local time" is *not* a single instant; it's 9am in Tokyo and 9am in Berlin, different moments. You model that as a `time` column plus a separate `timezone` text column, never a `timestamptz`. It's the rare carve-out, not a tool you reach for often.

### Calendar dates and time-only

Not everything about time is an instant. A few more time-shaped types round this out. They're still "time," so they live here rather than in their own section.

A `date` is a calendar day with no time and no zone: a due date, a billing date, a birthday. It's the right type whenever the answer is "a day," and it comes with a trap worth naming. **Don't model a calendar day as a `timestamptz` at midnight.** Midnight *where?* The instant "2026-03-14 00:00" lands on different calendar days depending on the reader's zone, so a birthday stored that way can shift by a day. A `date` has no zone and means the same day everywhere, which is exactly why it exists.

Two more you only need to recognize: `time` is a time of day with no date attached, and `interval` is a span of time (three days, two hours). Name them and move on.

One pattern to flag and defer: every SaaS table tends to carry both a `createdAt` and an `updatedAt`. Typing those two timestamp columns onto every single table by hand gets repetitive fast. There's a tidy way to factor them out into a reusable set of columns, and that's the next lesson's job. Know it's coming; don't build it yet.

## IDs: `uuid` for surrogate keys

Every row needs an identifier. The type for it is `uuid`, which holds a 128-bit <Term definition="Universally Unique Identifier — a 128-bit value, written as 32 hex digits, designed to be unique without a central coordinator.">UUID</Term> and, like the other types that surprised you, **maps to a TypeScript `string`**.

What's worth recognizing here is that a `uuid` column can generate its own default value, and there are two generators you'll see in the wild. Either of these goes on the `id` column.

<div data-mark-color="blue">

```ts ".defaultRandom()" "uuidv7()"
id: uuid().defaultRandom(),
id: uuid().default(sql`uuidv7()`),
```

</div>

The first, `.defaultRandom()`, calls Postgres's built-in `gen_random_uuid()` and produces a fully-random <Term definition="A UUID whose bits are essentially all random. The classic, fully-random UUID.">UUIDv4</Term>. The second produces a <Term definition="UUID version 7 — its leading bits are a timestamp, so v7 IDs sort in creation order. Native in Postgres 18.">UUIDv7</Term>, whose leading bits encode a timestamp so the IDs sort by creation time, which is gentler on the database's indexes. Postgres 18, which this course pins to, ships `uuidv7()` natively, with no extension to install.

Which one you should *choose* is a real decision with a real trade-off, and it gets its own treatment a couple of lessons from now, alongside the question of whether to use `uuid` at all or a `bigint` identity instead. Don't reach for the why yet. The job right here is recognition: `uuid` is the type, it surfaces as a string, and it has these two default generators.

## A fixed set of choices: `pgEnum`

Recall the invoice `status` from the booleans section, exactly one of `draft`, `sent`, `paid`, or `void`. That's the textbook case for an enum, and Drizzle spells it with `pgEnum`. It's shaped differently from every builder you've seen so far, so look closely.

```ts
export const invoiceStatus = pgEnum('invoice_status', [
  'draft',
  'sent',
  'paid',
  'void',
]);

export const invoices = pgTable('invoices', {
  // ...
  status: invoiceStatus().notNull().default('draft'),
});
```

It's a **two-step shape**. The call `pgEnum('invoice_status', [...])` does two things at once: it declares a real, named Postgres enum type called `invoice_status`, *and* it returns a column builder. You `export` that builder at the top of the file, where it's part of your schema's surface now, just like a table, and then you call `invoiceStatus()` down in the column map, exactly the way you'd call `text()` or `uuid()`. Declare it once, then use it as a column.

The payoff arrives later, when you read rows back: the allowed values come through to TypeScript as a union, `'draft' | 'sent' | 'paid' | 'void'`. The compiler then knows a status can only ever be one of those four, so a typo like `'snet'` won't compile. That's <Term definition="A design principle: shape your types so that invalid states simply cannot be expressed. The compiler rejects them before they can occur.">making impossible states unrepresentable</Term>, a principle you've met before, enforced right at the database boundary.

So when do you reach for `pgEnum`, and when does it stop being the answer? Reach for it when the set of values is small, stable, and mutually exclusive, which an invoice status fits perfectly. The alternative is a **lookup table**: a separate table with one row per allowed value. You promote to a lookup table the moment the values need their own data hanging off them, like a display label, a color, or a sort order, or the moment non-engineers need to add and remove values at runtime, or the set just keeps growing. Metadata, runtime editing, or open-endedness is the signal to graduate from an enum to a table.

One sharp watch-out decides borderline cases. **Enum values are easy to add and painful to remove.** Adding `archived` to the list later is a small change, but *dropping* a value is a harder migration: you have to prove first that no row anywhere still uses it. So enums are for genuinely stable sets. If you can already picture yourself deleting one of these values someday, that's a hint it wanted to be a lookup table. (The mechanics of changing enum values live in a later migrations chapter, named but not taught here.)

## A flexible blob: `jsonb` with `$type`

Sometimes a column's shape isn't yours to define: an incoming webhook body from Stripe, the details bag on an audit-log entry, a grab-bag of per-tenant settings, data that's genuinely shapeless or dictated by a third party. For that, Postgres has `jsonb`, which stores structured JSON in a **binary, indexable, queryable** form.

The course rule is short: **`jsonb`, never `json`.** Postgres has both. Plain `json` stores the raw text of the document; `jsonb` parses it into a binary representation that the database can actually index and query into. There's no upside to plain `json` for what you'll build, so name it once as the foil and always reach for `jsonb`.

Drizzle adds a sharp move on top. By default, a JSON column comes back to TypeScript as `unknown`, because the database has no idea what's inside, so neither does the compiler, and every place you read it has to cast first. You can do better: `.$type<WebhookEvent>()` annotates the column with the TypeScript shape you expect, so reads come back typed. This is the one line in the lesson where three separate things happen at once, so step through it.

<AnnotatedCode lang="ts" code={`
import { jsonb, pgTable } from 'drizzle-orm/pg-core';
import type { WebhookEvent } from '@/lib/webhooks';

export const webhookDeliveries = pgTable('webhook_deliveries', {
  // ...
  payload: jsonb().$type<WebhookEvent>().notNull(),
});
`}>
  <AnnotatedStep meta={`"jsonb()"`} color="blue">
    The builder. `jsonb` stores binary, indexable JSON, and the course never uses plain `json`. On its own, a read of this column would land in your code as `unknown`.
  </AnnotatedStep>

  <AnnotatedStep meta={`".$type<WebhookEvent>()"`} color="blue">
    The annotation. It tells Drizzle the TypeScript shape of what's inside, so reads come back as `WebhookEvent` instead of `unknown` and you skip a cast at every read site. This is the Drizzle-specific move worth knowing.
  </AnnotatedStep>

  <AnnotatedStep meta={`".$type<WebhookEvent>()"`} color="orange">
    The catch, and it matters. `$type` is a **compile-time promise, not a runtime check.** Postgres stores whatever bytes you write, so if the real payload doesn't match `WebhookEvent`, the compiler won't catch it. Validating the actual shape on the way in is Zod's job at the boundary, in a later chapter. `$type` says what you *expect*; Zod is what *enforces* it.
  </AnnotatedStep>
</AnnotatedCode>

That third step is easy to miss and easy to get burned by, so don't let it blur past: a `$type` annotation is a claim TypeScript *trusts*, not a guarantee Postgres *verifies*. The database is happy to store a payload that doesn't match, and your reads will confidently hand you the wrong type. The runtime gate is always Zod.

Now the rule that keeps `jsonb` from quietly rotting your schema. Reach for it for data that is genuinely shapeless or third-party-defined: webhook bodies, audit-log details, flexible metadata. **Skip it for anything you'll filter, sort, or join on.** The heuristic is simple: the moment you find yourself reaching *into* the JSON inside a `WHERE` clause more than occasionally, that field wanted to be a real column. A `jsonb` value you keep querying into is **normalization debt**; promote it to a column and the database can index it properly. (Querying into `jsonb` with its special operators is a skill for a later chapter. Here, just know that needing to is the signal you reached for `jsonb` when a column was the right call.)

## A small ordered list: `text().array()`

Postgres has native array columns, and Drizzle exposes them by chaining `.array()` onto a builder: `text().array()` is an ordered list of strings in a single column, `integer().array()` a list of integers, and so on. The classic example is a `tags` column:

```ts
tags: text().array().notNull().default([]),
```

This is the lightweight option, perfect when you need a small, ordered list of plain scalars and a whole separate table would be overkill. But its limitation is exactly what tells you when to stop using it: **array elements aren't real database rows.** They can't be foreign keys, they can't be `JOIN`ed, and they get no referential integrity. Postgres sees a bag of strings, not connections to anything.

So the moment those tags need to become *entities*, with their own id, color, and owner, renamed in one place and reflected everywhere, or the moment you need to efficiently answer "every invoice tagged `urgent`" at scale, the array has outgrown itself and wants a <Term definition="A table whose job is to connect rows of two other tables — one row per link.">**junction table**</Term>. That graduation is a whole lesson later in this chapter. The trigger to make the jump is a single question: would I ever want to `JOIN` on these, or give them attributes of their own? If yes, it was always a junction table, and the array was a postponement. Arrays are tempting precisely because they're so easy to reach for, and a `text[]` of tags the product later wants to rename, recolor, or filter the whole list by is a junction table you put off.

## Network addresses: `inet`

One small choice quietly marks an experienced hand. When you store an IP address, like an audit log's actor IP or a request's origin, most people default to `text`. Postgres has a real type for it:

```ts
actorIp: inet().notNull(),
```

<Term definition="The Postgres type for an IP address — both IPv4 and IPv6 — that the database understands as a network, not just text.">`inet`</Term> stores IPv4 and IPv6 addresses as genuine network values, which means the database can sort them correctly and run subnet and range queries (such as "is this IP inside that block?") that a plain string simply can't. It's a tiny upgrade with no downside: store IPs as `inet`, not `text`, and they're real addresses the database understands rather than opaque strings. This pays off when audit-log tables arrive later in the course.

And the single sharpest trap, as a question, because this is the one that silently corrupts data if you get it wrong.

<MultipleChoice>
  An invoice amount should be `numeric`, never `double precision`. What goes wrong if you store it as `double precision`?

  <McqChoice correct>The math runs without complaint, but each value is stored as the nearest binary approximation of the decimal, so summed amounts drift by fractions of a cent and totals stop matching the rows.</McqChoice>
  <McqChoice>`double precision` can't hold a number large enough for a realistic invoice total, so big amounts overflow.</McqChoice>
  <McqChoice>Postgres rejects the insert outright, because `double precision` columns refuse any value with digits after the decimal point.</McqChoice>
  <McqChoice>`double precision` truncates every value to a whole number on write, silently dropping the cents.</McqChoice>

  <McqWhy>The damage is silent and cumulative. Each float keeps only the closest binary approximation of the decimal; across many rows those approximations add up, the cents drift, and the books stop balancing — and nothing throws, which is exactly what makes it dangerous. `numeric` stores the literal digits instead. (Floats happily hold cents and span an enormous range, so neither overflow nor truncation is the real problem.)</McqWhy>
</MultipleChoice>

## Closing

Here's the whole lesson in short: a 2026 SaaS reaches for about eight Postgres types, and each is shadowed by one tempting-but-wrong alternative you now know to walk past. Use `text`, not `varchar`. Use `numeric`, not a float. Use `timestamptz`, not a plain `timestamp`. Reach for `jsonb` typed with `$type` for genuinely shapeless blobs, but a real column for anything you'll filter on. Use `pgEnum` for a small, stable, mutually-exclusive set of states, and `text().array()` for a light scalar list that graduates to a junction table the day it grows attributes. Use `inet` for IP addresses. Keep the defaults table within reach; it's the practical residue of everything here.

Every column in your schema now has a *type*. What it doesn't yet have, decided deliberately, is its three per-column modifiers: can it be null, does it carry a default, is it generated? Those are the calls you've been writing on faith all lesson, like `.notNull()` and `.defaultNow()`, and the next lesson finally explains them, starting with the default that quietly shapes every well-built table: not-null until proven otherwise.
