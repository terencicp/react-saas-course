Right now your `invoiceLineItems` table has an `invoiceId` column that is *supposed* to point at a row in `invoices` — but it's just a `uuid`, and nothing enforces the promise. You could insert a line item whose `invoiceId` matches no invoice at all, and when an invoice is deleted, its line items sit there pointing at a row that no longer exists. A foreign key fixes both halves of that: it makes Postgres **enforce the link** on every write, and it lets you **decide what happens to the children when the parent is deleted**. That second half is the real lesson — a four-way choice you make per relationship, and the everyday reframe of when deletions even happen. Last lesson ended on "every foreign key points at exactly one primary key"; this is the foreign-key end of that sentence.

## A foreign key is a promise the database keeps

Before we touch deletion behavior, get the foreign key working as a pure guarantee. Here's the whole declaration, attached to the column it lives on:

```ts title="db/schema.ts"
organizationId: uuid().notNull().references(() => organizations.id, { onDelete: 'restrict' }),
```

Three pieces, left to right. The column itself (`uuid`, matching the type of the primary key it points at). The `.references(() => organizations.id, …)` call, which names the target — table and column. And the `onDelete` option, which sets the deletion policy. Ignore `onDelete: 'restrict'` for one more section; we'll spend the whole next section on what those values mean. For now, focus on the link.

Notice the reference is a *callback*: `() => organizations.id`, not `organizations.id`. That arrow isn't decoration. When two tables reference each other — and in a real schema they will — whichever module loads second hasn't finished defining its tables yet, so reaching for the value directly throws a "cannot access before initialization" error. Wrapping it in a function defers that lookup until both tables exist. Drizzle asks for the callback precisely so you never hit that trap.

So what does declaring this actually buy you? Three things:

1. Postgres **rejects any write** — insert *or* update — whose `organizationId` doesn't match an existing `organizations.id`. An <Term definition="A child row whose referenced parent row no longer exists — a line item whose invoice is gone, a task assigned to a deleted user.">orphan row</Term> is now impossible to create. The database refuses it at the boundary.
2. The **column types must match the referenced primary key exactly** — a `uuid` foreign key points at a `uuid` primary key, a `bigint` at a `bigint`. Get this wrong and the schema fails to apply; it's caught up front, never as a runtime surprise. (This is why last lesson's primary-key shapes matter here: the foreign key inherits the type of whatever it points at.)
3. The foreign-key column is **not indexed automatically**. Declaring the constraint creates the *rule*, not an index — and without one, both the cascade deletes you're about to meet and ordinary "find every invoice for this org" lookups degrade to full-table scans as the table grows. Just file that away for now; adding the index is the first thing we do in the indexing chapter.

The payoff of all this is a single reflex worth stating plainly: **the foreign key moves orphan-prevention out of your application code and down to the database boundary.** Without it, every code path that inserts a line item has to first check "does this invoice exist?" — and the one path that forgets ships the bug. With it, the question is unnecessary: a bad write would have failed before your code ran. That is the same theme the next lesson extends to uniqueness and value checks — the database is the last line of defense, not the first thing you skip.

One shape you'll see named but rarely write: a foreign key that spans *multiple* columns uses the table-level `foreignKey({ columns, foreignColumns })` helper instead of `.references()`. Because every table in this course keys on a single surrogate `id`, single-column foreign keys are almost always the shape you want — recognize the multi-column form, don't reach for it.

This guarantee — that every foreign-key value points at a real row — is <Term definition="The database guarantee that every foreign-key value points at a row that actually exists. Postgres enforces it on every insert, update, and delete.">referential integrity</Term>, and it's the floor every relational schema stands on.

## Choosing what happens on delete: the four ON DELETE rules

Now the decision. `onDelete` controls one specific moment: what Postgres does to the **child** rows when the **parent** row they point at gets deleted. There are four possible answers, and choosing between them is the modeling call — it comes from the *meaning* of the relationship, not from a default you accept without looking.

- **`cascade`** — delete the children too. Reach for it when the child **has no meaning without its parent**: a line item under an invoice, a junction row under either table it joins. This is the *ownership* relationship. The mental check is blunt: "if the parent is gone, is the child garbage?" If yes, cascade.

- **`set null`** — keep the child, but null out its pointer. Reach for it when the relationship is **optional and orphaning is exactly what you want**: an invoice's `assignedToId` when the assigned user is offboarded. The invoice survives — it's just unassigned now. This **requires the column to be nullable** (no `.notNull()`), and that's not a quirk: calling `set null` on a `.notNull()` column is a contradiction Postgres will reject, because the rule's whole job is to write a null the column forbids. This is the reason last lesson's `assignedToId` was left nullable.

- **`restrict`** — **block the delete** while any child still exists. Reach for it when removing the parent out from under live children would be a mistake the system should refuse: an organization that still has invoices. The delete throws; whoever wanted it gone must remove or reassign the children first — or, far more often, soft-delete the parent instead, which is the next section. One subtlety to put on the page, because the default is a trap: if you write *no* `onDelete` clause at all, Postgres doesn't fall back to `restrict` — it falls back to <Term definition="Postgres's implicit ON DELETE default when no clause is written. Like restrict, it blocks the delete while children exist — but its check can be deferred to the end of the transaction. The course writes restrict explicitly instead.">`NO ACTION`</Term>, a near-twin that also blocks but defers its check to the end of the transaction rather than checking immediately per statement. The course always writes `restrict` explicitly, so the blocking intent is on the page instead of inferred from an omission; the deferred-vs-immediate difference between the two is out of scope.

- **`set default`** — set the child's pointer to a column default. It exists for completeness, but it needs a sentinel default row to point at and rarely earns that wiring. Know the name; you'll almost never use it.

That's the menu. The harder skill is the *order* you ask the questions in, because the rule falls out of the answers. Walk this decision — each branch commits to the next question, and the leaf names the rule plus its canonical example.

  <Leaf id="leaf-cascade" verdict="onDelete: 'cascade'">
    Delete the children with the parent. The child is *owned* — it's garbage the moment the parent is gone. Example: `invoiceLineItems.invoiceId → invoices.id`. Watch the blast radius when the graph branches: cascading an org could sweep its invoices, their line items, their tags…
  </Leaf>

  <Leaf id="leaf-restrict" verdict="onDelete: 'restrict'">
    Block the parent delete while any child exists; the statement throws. Example: `invoices.organizationId → organizations.id`. The caller must reassign or remove the children first — or, in practice, soft-delete the parent so the delete never runs at all.
  </Leaf>

  <Leaf id="leaf-set-null" verdict="onDelete: 'set null'">
    Keep the child, null its pointer. The relationship is optional and orphaning is the goal. Example: `invoices.assignedToId → users.id` — offboarding a user un-assigns their invoices instead of destroying them. (`set default` is the rare sibling here: point at a sentinel row instead of null. Almost never worth it.)
  </Leaf>

  <Leaf id="leaf-contradiction" verdict="Make it nullable first — or reconsider">
    `set null` needs a column that can hold a null; a `.notNull()` foreign key can't. If the relationship is genuinely required, you don't want `set null` — you want `restrict` (block the delete) or `cascade` (let the child go with the parent). The "required + set null" combination is a modeling mistake, not a configuration to fix.
  </Leaf>
</StateMachineWalker>

The whole lesson lives in that question order — ownership, then blockability, then nullability — not in any single leaf. Memorizing a lookup table of four rules ages badly; internalizing the order means you can derive the right rule for a relationship you've never seen.

## `onUpdate`, and why this course never needs it

`references()` takes a second policy alongside `onDelete`: an `onUpdate` rule that fires when the **referenced primary-key value itself changes**. In this course it's dead weight. Primary keys here are immutable surrogates — UUIDv7, `bigint` identity — and they never change once minted, so `onUpdate` has nothing to fire on. The only place it earns its keep is mutable *natural* keys, which is exactly the anti-pattern last lesson steered you away from. So treat it as a non-decision: if you ever find yourself reaching for `onUpdate`, that's a signal your primary key should have been a surrogate, not a value that drifts.

## Hard delete vs. soft delete: which deletions even happen

Cascade quietly assumes something you should question: that the parent is **hard-deleted** — the row physically removed with a `DELETE` statement. In a lot of SaaS, that's the *rare* path.

The common path is **soft delete**. Instead of removing the row, you stamp it with a `deletedAt` timestamp — the nullable column already sitting in your `db/columns.ts` from the column-modifiers lesson. The row stays in the table; your queries learn to filter out the stamped ones, so the record is gone from the user's point of view but recoverable, auditable, and still there for anything that references it historically. And here's the part that reframes everything you just learned: because the row is never `DELETE`d, **`onDelete` never fires**. On the soft-delete path, `cascade` and `restrict` are simply irrelevant — they're rules about a statement you're not running.

So the senior split is a per-table decision about *which kind of deletion a table even gets*:

- **Hard delete with cascade** for genuinely scrubbing a relationship graph — tenant offboarding, where an org and everything it owns must be permanently erased (a GDPR deletion request, say), or ephemeral owned children where no audit trail matters.
- **Soft delete** for almost everything user-facing — invoices, customers, anything where "deleted" has to be recoverable or referenced later. Here `onDelete` is a backstop for the rare true purge, not the everyday path.

The reconciliation people miss: declaring `onDelete: 'cascade'` and soft-deleting are **not in conflict**. The foreign-key rule defines what happens *if* a hard delete ever runs — a purge job, an admin action, a GDPR erase. Your normal application flow soft-deletes, so the rule simply doesn't trigger. You still choose `onDelete` deliberately for every foreign key, even in a soft-delete-first app, because you're choosing what the rare purge does, not what your everyday delete does.

:::note
Choose `onDelete` for the purge that *might* happen; choose soft delete for the deletes that *actually* happen day to day.
:::

## Wiring an invoice's relationships end to end

Time to cash the decision out on real tables. The point of this example is the one that's easy to miss until you see it: **one entity can legitimately carry three different `onDelete` rules at once**, because the rule is per-edge, not per-table. Here's `invoices` and its line items with every foreign key wired. Step through it — each step lands on one foreign key, its rule, and the one question from the walker that picked it.

<AnnotatedCode lang="ts" maxLines={18} code={`
export const invoices = pgTable('invoices', {
  id: uuid().primaryKey().default(sql\`uuidv7()\`),
  organizationId: uuid()
    .notNull()
    .references(() => organizations.id, { onDelete: 'restrict' }),
  assignedToId: uuid()
    .references(() => users.id, { onDelete: 'set null' }),
  amountDue: numeric({ precision: 12, scale: 2 }).notNull(),
});

export const invoiceLineItems = pgTable('invoice_line_items', {
  id: uuid().primaryKey().default(sql\`uuidv7()\`),
  invoiceId: uuid()
    .notNull()
    .references(() => invoices.id, { onDelete: 'cascade' }),
  description: text().notNull(),
  amount: numeric({ precision: 12, scale: 2 }).notNull(),
});
`}>
  <AnnotatedStep meta={`{3-5} "restrict"`} color="blue">
    `organizationId → organizations.id`, **`restrict`**. The walker's first question: does an invoice mean anything without its org? Yes — so cascade is wrong. Second question: should you be able to delete an org with live invoices? No. The org gets blocked from deletion (and is soft-deleted in practice anyway). `.notNull()`, because every invoice belongs to an org.
  </AnnotatedStep>

  <AnnotatedStep meta={`{6-7} "set null"`} color="orange">
    `assignedToId → users.id`, **`set null`**. Optional pointer: an invoice can stand alone, deleting the user *should* be allowed, and the pointer should just clear. Note the missing `.notNull()` — `set null` requires a nullable column, which is exactly why this column was left nullable last lesson. Offboard a user and their invoices go unassigned, not deleted.
  </AnnotatedStep>

  <AnnotatedStep meta={`{13-15} "cascade"`} color="green">
    `invoiceId → invoices.id`, **`cascade`**. A line item is meaningless without its invoice — pure ownership, so the first question alone settles it. Delete the invoice and its line items go with it, in one declarative rule instead of cleanup code in every delete path. `.notNull()`, because a line item with no invoice is the orphan the foreign key exists to forbid.
  </AnnotatedStep>
</AnnotatedCode>

One table, three rules — `restrict`, `set null`, `cascade` — each correct for its own edge. That's the whole "aha" of this section.

Two threads worth pulling, briefly. There's a fourth relationship coming: invoices and tags are many-to-many, joined through an `invoiceTags` table whose two foreign keys both `cascade` — a junction row is owned by *both* endpoints, so it should vanish when either goes. Building that junction is the next lesson's job; just note the cascade-on-both for now. And notice `organizationId` again: every tenant-owned table will carry exactly this `restrict` foreign key to `organizations`. It's where multi-tenancy *starts* in the schema — actually *enforcing* that an org only ever sees its own rows is a query-layer concern much later in the course. For now, the foreign key is the seed.

Last reminder in context, since it bites here specifically: deleting an org has to scan `invoices` for matching `organizationId` rows to apply that `restrict` check — and with no index on the column, that's a full-table scan. The indexing chapter fixes it; the gotcha is just easier to feel now that you can see the relationship it slows down.

```ts title="db/schema.ts"
  id: uuid('id').primaryKey(),
  organization_id: uuid('organization_id')
    .notNull()
    .references(() => organizations.id, { onDelete: 'restrict' }),

  id: uuid('id').primaryKey(),
  invoice_id: uuid('invoice_id')
    .notNull()
    .references(() => invoices.id, { onDelete: 'cascade' }),
  description: text('description').notNull(),
  amount: numeric('amount', { precision: 12, scale: 2 }).notNull(),
});
```

Walk the rules back to the walker's questions. `organization_id → organizations.id` is **`restrict`**: an invoice means something without its org, and you should *not* be able to delete an org with live invoices — so the delete is blocked (probe 1 throws, as it must). `invoice_id → invoices.id` is **`cascade`**: a line item is owned by its invoice, so it goes when the invoice goes (probe 2 deletes the invoice and finds no orphaned line item). `assigned_to_id → users.id` is **`set null`**: the pointer is optional, so offboarding the user clears it instead of deleting or blocking the invoice (probe 3 deletes the user and finds the invoice intact with a null assignee) — which only works because the column is nullable.
