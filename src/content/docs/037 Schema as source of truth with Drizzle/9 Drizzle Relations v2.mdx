Every detail page in this app wants the same thing: an invoice, but not the bare row ‚Äî the invoice *with* its line items and *with* its tags, handed over as one typed object you can render straight into the page. So you reach for the call that ought to give you exactly that:

```ts
const invoice = await db.query.invoices.findFirst({
  with: { lineItems: true, tags: true },
});
```

It does not work yet. And that should bother you, because you've already declared the foreign keys that connect every one of these tables ‚Äî `invoiceLineItems.invoiceId` points at `invoices.id`, the `invoiceTags` junction points at both `invoices` and `tags`. The links *exist* in the database. So why can't the query builder follow them?

Because a foreign key and a traversal are two different things, declared for two different audiences ‚Äî and this lesson is about the second one. By the end you'll have written a new file, `db/relations.ts`, that turns the inert links from the last few lessons into a graph the query builder can walk. You won't write the queries themselves here; those come in the next chapter. You'll build the *map* they read.

## Why the foreign key isn't enough

Go back to what a foreign key actually is. When you wrote `.references(() => organizations.id, { onDelete: 'restrict' })`, you handed Postgres a rule it enforces on every write: reject any invoice whose `organizationId` doesn't match a real organization. That's the whole job of the constraint. It's a promise the database keeps ‚Äî but the promise is about *writes*. It guards what can be inserted and what happens on delete. It says nothing, to anyone, about how to *read* across the link.

The relational query builder ‚Äî the `db.query.‚Ä¶` API you reached for above ‚Äî needs that second thing: a declaration of which edges it's allowed to walk and how to walk them, written in TypeScript, where it lives. The foreign key is in the database; the query builder is in your Node process. They never see each other's world. So you declare the traversal separately.

That gives the data layer two files that look like they overlap but don't:

- `db/schema.ts` ‚Äî **what's in the database.** Tables, columns, foreign keys. This is the only file Postgres ever sees: it becomes migrations in a later chapter, and every constraint you've written lives here.
- `db/relations.ts` ‚Äî **how the query builder walks it.** A pure-TypeScript graph. It never touches Postgres, emits no SQL, and generates no migration. Its one and only consumer is the `db.query` builder.

<TwoLayers />

So here is the sentence to carry through the rest of the lesson: the foreign key and the relation are **two declarations of the same edge, for two different audiences** ‚Äî the database's integrity engine and Drizzle's query builder ‚Äî and **declaring one never declares the other.** This is the single most common beginner bug with Drizzle: `db.query.invoices.findFirst({ with: { tags: true } })` comes back with `tags: undefined`, the developer stares at the foreign keys sitting right there in the junction table, and can't see what's wrong. Nothing is wrong with the foreign keys. The relation was simply never declared. Hold onto that; it's the whole lesson.

## The relations file and `defineRelations`

The `db/` folder already holds `schema.ts` (your tables), `columns.ts` (the shared column helpers from earlier), and `index.ts` (the client, wired up properly in a later chapter). Add a fourth file beside them: `db/relations.ts`. Its shape is small and fixed.

You import the tables from `./schema`, pass them as one object into <Term definition="The Relations v2 API. Declares the entire traversal graph in a single call, keyed by table name.">`defineRelations`</Term>, and export what comes back as a single `relations` const:

```ts
import { defineRelations } from 'drizzle-orm';
import * as schema from './schema';

export const relations = defineRelations(schema, (r) => ({
  // one entry per table, each listing that table's relations
}));
```

Two arguments, and it's worth being precise about each because everything you write goes inside the second one.

The **first argument** is the schema object ‚Äî every table, passed together: `organizations`, `invoices`, `invoiceLineItems`, `users`, `tags`, `invoiceTags`, `memberships`. Importing the whole module as `schema` and handing it over in one go is what gives the builder its knowledge of every table and every column. That's where autocomplete comes from in the next part.

The **second argument** is a callback, `(r) => ({ ‚Ä¶ })`. It returns a map keyed by **table name**, and each value is an object describing that table's relations. The `r` parameter is the relation builder, and you'll use it three ways: `r.one.<table>(‚Ä¶)` for a relation that resolves to a single row, `r.many.<table>(‚Ä¶)` for one that resolves to an array, and `r.<table>.<column>` to point at a specific column when you need to spell out the join.

Where does this file plug in? Into the <Term definition="The relational query builder ‚Äî the db.query.* findMany / findFirst API that reads the relations graph to assemble nested objects.">relational query API</Term>. When the client gets created, the relations get handed to it on the side:

```ts
export const db = drizzle(client, { relations });
```

That one option is what carries the graph into the `db.query` builder ‚Äî without it, `db.query` has no map and every `with` comes back empty. The rest of the client setup (the connection, the pool, reading the URL from the environment) is a later chapter's job; here, that single `{ relations }` line is the only part that matters. Now let's fill the callback in, simplest edge first.

## One-to-many: an organization and its invoices

Start with the relationship that makes up most of every real schema: one organization, many invoices. The foreign key already exists ‚Äî `invoices.organizationId` references `organizations.id` ‚Äî and a relationship like this has two ends, so you declare both, on both tables.

From the organization's side, it has many invoices. From an invoice's side, it belongs to one organization. Read those two English sentences and the code is almost a transcription of them:

<AnnotatedCode lang="ts" code={`
export const relations = defineRelations(schema, (r) => ({
  organizations: {
    invoices: r.many.invoices(),
  },
  invoices: {
    organization: r.one.organizations({
      from: r.invoices.organizationId,
      to: r.organizations.id,
    }),
  },
}));
`}>
  <AnnotatedStep meta="{2-4}" color="blue">
    The `organizations` key, listing one relation: `invoices: r.many.invoices()`. `r.many` says this resolves to an *array* ‚Äî an org has many invoices ‚Äî and the call names the target table. That's the entire declaration on this side; no columns yet.
  </AnnotatedStep>

  <AnnotatedStep meta="{5-10}" color="green">
    The `invoices` key, with its `organization` relation. `r.one` says this resolves to a *single object*, not an array ‚Äî an invoice belongs to exactly one org. The braces hold the part the `many` side didn't need: the explicit join.
  </AnnotatedStep>

  <AnnotatedStep meta={`"from: r.invoices.organizationId"`} color="orange">
    `from` is the column on *this* table that points outward ‚Äî the one that *holds the foreign key*. It's `invoices.organizationId`, the same column you wrote `.references()` on. Whenever you see `from`, read "the FK-holding side."
  </AnnotatedStep>

  <AnnotatedStep meta={`"to: r.organizations.id"`} color="violet">
    `to` is the column it points *at* ‚Äî the referenced primary key, `organizations.id`. `from ‚Üí to` runs in the exact same direction as the foreign key itself: `organizationId ‚Üí organizations.id`. The relation just restates that arrow for the query builder.
  </AnnotatedStep>
</AnnotatedCode>

Notice the names. The `one` side is **singular** ‚Äî `organization` ‚Äî and the `many` side is **plural** ‚Äî `invoices`. That's not decoration: the relation key is the exact word you'll type inside `with: { ‚Ä¶ }` when you finally write the query. You want `with: { organization: true }` and `with: { invoices: true }` to read like the data they return, so a singular key returns one object and a plural key returns an array. Name them to match what comes back.

### Which side carries `from` and `to`

Look again at where `from` and `to` actually appeared: only on the `invoices` side ‚Äî the `r.one` relation, the one holding the foreign-key column. The `organizations` side was a bare `r.many.invoices()` with no `from`, no `to`.

That's deliberate, and it's the rule worth fixing in memory: **`from`/`to` go on the side that holds the foreign key.** A foreign key lives on exactly one of the two tables (here, `organizationId` is on `invoices`), so the explicit join belongs there. The other side ‚Äî the reverse direction ‚Äî doesn't need it, because Drizzle can infer it. That's the next section's whole point, and it's the headline reason this course teaches the v2 API rather than the old one.

## One declaration, both directions

In the previous version of Drizzle's relations API, you had to spell out the join on **both** ends ‚Äî `fields` and `references` repeated on the organization side *and* the invoice side, the same two columns named twice in opposite order. It was easy to get the direction backwards on one of them, and the redundancy bought you nothing.

The v2 API drops that. You pin the edge **once**, on the foreign-key-holding side (with `from`/`to`), and declare the reverse side bare. Drizzle matches them by table and infers the reverse direction for you. So in practice the reverse side is just a name and a target:

<CodeVariants>
  <CodeVariant label="Bare reverse">
    <div data-mark-color="green">

    ```ts {2}
    organizations: {
      invoices: r.many.invoices(),
    },
    ```

    </div>
    **The common case.** No `from`/`to` ‚Äî Drizzle reads the forward `invoices.organization` declaration and infers this reverse join from it. This is what you'll write almost every time.
  </CodeVariant>

  <CodeVariant label="Explicit reverse">
    <div data-mark-color="orange">

    ```ts {2-5}
    organizations: {
      invoices: r.many.invoices({
        from: r.organizations.id,
        to: r.invoices.organizationId,
      }),
    },
    ```

    </div>
    **Legal, occasionally clearer, but redundant** when the other side already pins the edge. Reach for it only when there's genuine ambiguity to resolve ‚Äî see the note below on two foreign keys between the same pair of tables.
  </CodeVariant>
</CodeVariants>

The bare reverse is the default. But notice that you *did* declare it ‚Äî you didn't leave `organizations` without its `invoices` relation just because the forward side already pinned the join. Declare **both** ends, even when one is a bare line, because the cost of omitting the reverse isn't an error you'll catch. It's a silent gap: a relation that simply doesn't exist. Six weeks later someone writes `org.invoices` in a `with`, gets nothing back, and burns an afternoon wondering why the link "isn't working" when the real answer is that this *direction* of it was never declared. Declare both, so the graph is walkable whichever way the next reader comes at it.

### When one name isn't enough: `alias`

There's one case where Drizzle genuinely can't sort the relations out by table, and the domain already contains it. An invoice points at `users` twice ‚Äî once through `assignedToId`, and (when the app grows) potentially again through a `createdById`. Two foreign keys, same pair of tables. If you declare two relations to `users`, "match by table" is ambiguous: which relation owns which column?

You disambiguate by hand with `alias` ‚Äî a label on each relation so they stay distinct:

```ts
invoices: {
  assignee: r.one.users({
    from: r.invoices.assignedToId,
    to: r.users.id,
    alias: 'assignee',
  }),
  creator: r.one.users({
    from: r.invoices.createdById,
    to: r.users.id,
    alias: 'creator',
  }),
},
```

That's the entire mechanic: when two relations connect the same two tables, give each an `alias` so the `with` keys stay unambiguous (`with: { assignee: true, creator: true }`). One example is enough ‚Äî modeling tangled multi-edge graphs is a topic of its own, and you only need to recognize the escape hatch when you trip over it.

## Many-to-many: walking the junction with `through`

The junction table you built last lesson ‚Äî `invoiceTags`, two foreign keys and a composite primary key ‚Äî was, in its own words, inert. It records that *this* invoice is linked to *this* tag, but nothing yet walks those links. The lesson ended on exactly that cliff-hanger. This is the walk.

Here's the trap, and it's worth naming because it's the natural guess: you might expect a top-level option, something like `r.many.tags({ through: invoiceTags })` ‚Äî "go to tags, through the junction." That's not the shape. The junction isn't a single waypoint; getting from an invoice to its tags is *two* hops ‚Äî into the junction, then out of it ‚Äî so you name the junction column on **each** hop. `.through()` chains onto both the `from` and the `to`:

<AnnotatedCode lang="ts" code={`
invoices: {
  tags: r.many.tags({
    from: r.invoices.id.through(r.invoiceTags.invoiceId),
    to: r.tags.id.through(r.invoiceTags.tagId),
  }),
},
`}>
  <AnnotatedStep meta="{1-6}" color="blue">
    Still `r.many` ‚Äî an invoice has many tags, the array shape is unchanged. What's new is entirely inside the braces: the join now goes *through* a third table, so each side gains a `.through()`.
  </AnnotatedStep>

  <AnnotatedStep meta={`"from: r.invoices.id.through(r.invoiceTags.invoiceId)"`} color="orange">
    The entry hop. Start at this table's primary key, `invoices.id`, and go *through* its column in the junction, `invoiceTags.invoiceId`. Read it as: "from `invoices.id`, into the junction via `invoiceTags.invoiceId`."
  </AnnotatedStep>

  <AnnotatedStep meta={`"to: r.tags.id.through(r.invoiceTags.tagId)"`} color="violet">
    The exit hop. Arrive at the target's primary key, `tags.id`, coming *out through* the other junction column, `invoiceTags.tagId`. The symmetry is the point: both sides chain `.through()`, each naming *its own* foreign-key column in the junction ‚Äî `invoiceId` on the way in, `tagId` on the way out.
  </AnnotatedStep>
</AnnotatedCode>

Read the whole thing aloud and it narrates the path: from `invoices.id`, hop *through* `invoiceTags.invoiceId`, into the junction; to `tags.id`, arriving *through* `invoiceTags.tagId`, out the other side. The junction gets named twice because the builder traverses it as two joins. And notice what you *didn't* write: nothing on `invoiceTags` itself. For this traversal the junction needs no relations of its own ‚Äî the `.through()` calls reach into it from the outside. (The reverse side, an invoice list hanging off each tag, is the mirror image: `tags: { invoices: r.many.invoices({ ‚Ä¶ }) }` with the two hops swapped.)

:::note
This is the headline reason the course standardizes on the v2 API. In the old version, walking a junction meant declaring relations *on the junction table* pointing at each parent, then chaining two separate `with` clauses to hop through it. v2's `.through()` collapses that into one direct `invoices.tags` relation ‚Äî the junction stays invisible at the call site. You'll meet v1 only in older tutorials and codebases; recognize the double-hop shape so it doesn't confuse you, and write v2.
:::

Here's what this one declaration buys. In the next chapter, `db.query.invoices.findFirst({ with: { tags: true } })` returns the invoice with a `tags: Tag[]` array hanging off it ‚Äî the junction table nowhere in sight, the two hops folded away by the builder. You don't write that query in this lesson. But that's the payoff the `.through()` chain exists to enable, and it's why the junction stopped being inert the moment you declared this relation.

### The junction that became an entity

Last lesson drew a line between two kinds of linking table, and that line has a querying consequence worth landing now. A **pure junction** like `invoiceTags` ‚Äî nothing but two foreign keys ‚Äî is what `.through()` is for: you walk *through* it because it carries no data you'd ever want on its own.

But `memberships` was different. It started as a user-to-organization link and grew a `role`, a surrogate `id`, its own timestamps ‚Äî it became a first-class entity, a thing in the domain rather than a wire between two things. So you *don't* reach through it. You relate to it directly, the same way you'd relate to any table: one-to-many from each parent into `memberships` (an org has many memberships, a user has many memberships), then `r.one` from `memberships` back to each parent. You query the membership as itself ‚Äî its `role` is data you want ‚Äî instead of treating it as a hidden hop. Pure junction, reach *through*; promoted entity, relate *to*. That's the querying half of the decision you made when you gave it an `id`.

## Self-referential relations

One more shape, briefly, because it looks exotic the first time and turns out to be nothing new. Sometimes a table relates to *itself*: a comment that replies to another comment, a category nested under a parent category, an org chart where each employee reports to another employee. Same table on both ends of the edge.

You declare it with the same `from`/`to` you already know ‚Äî the only difference is that both columns live on one table:

```ts
comments: {
  parent: r.one.comments({
    from: r.comments.parentId,
    to: r.comments.id,
  }),
  replies: r.many.comments({
    from: r.comments.id,
    to: r.comments.parentId,
  }),
},
```

With a `parentId` column that references `comments.id`, `parent` walks *up* (a comment's one parent) and `replies` walks *down* (a comment's many children). Because both relations connect `comments` to `comments`, this is exactly the two-edges-same-pair case from earlier ‚Äî so if you add a second self-edge, reach for `alias` to keep them apart. What this does *not* cover is fetching a whole tree to arbitrary depth; recursion is a query concern, not a relations-declaration one, and it belongs with the queries in the next chapter.

## When to skip the relations layer

You've just learned a tool that makes nested reads effortless, so it's worth saying plainly when *not* to reach for it ‚Äî before you start forcing every read through it in the next chapter.

The relational API earns its keep when the read is a **tree**: an entity and its children, shaped as a nested object. "An invoice with its line items and tags," "an organization with its members" ‚Äî that's the exact shape `defineRelations` and `db.query.‚Ä¶({ with })` exist for, and it's the safe default for reads that fan out across related tables.

It is *not* the tool for reads that aren't trees. Aggregates ‚Äî `COUNT` of invoices per org, `SUM` of line-item totals ‚Äî filters that select *on* a joined table, projections that pull a handful of columns from several tables at once: those are hand-written joins (`db.select(...).leftJoin(...)`), and you'll meet them in the next chapter too. The relational builder plans its own SQL toward assembling nested objects; it isn't built to shape arbitrary result sets.

One honest caveat, named not solved: because the relational API plans its own SQL, a deeply nested `with` can emit a heavier query than you'd guess from how innocent the call looks. The flag that lets you *see* the SQL it generates is the `logger: true` option from the earlier `pgTable` lesson; reading the emitted query ‚Äî and later, running `EXPLAIN ANALYZE` on it ‚Äî is how you catch a `with` that grew teeth. For now, just know the relational API is one tool with a clear job: nested tree reads. Use it there, drop to joins elsewhere.

## A version note worth keeping straight

Everything in this lesson is the **v2** relations API: `defineRelations`, the single-call graph, `from`/`to`/`through`. It ships in `drizzle-orm@1.0.0-beta` ‚Äî the forward-looking version this chapter committed to from the start.

The reason to mention it at all is so older code doesn't confuse you. The **v1** API is now labelled `[OLD]` in Drizzle's docs: it used a per-table `relations(invoices, ({ one, many }) => ‚Ä¶)` helper, its import has moved to `drizzle-orm/_relations`, and its query builder is now `db._query` rather than `db.query`. You'll write none of it ‚Äî but you'll see it in tutorials and existing repos, and recognizing the per-table shape (and the `_` prefixes that mark it as legacy) is enough to not get thrown. v2 is what you write, for the reasons this lesson showed: one declaration per edge instead of two, native junction traversal, and cleaner type inference.
