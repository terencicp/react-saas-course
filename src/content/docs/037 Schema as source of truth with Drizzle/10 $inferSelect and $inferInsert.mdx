Over the last several lessons you built the whole thing: the `organizations` table, the `invoices` table, the line items, the tags, the junctions, the relations. Every column, every type, every constraint now lives in one file. And when this chapter opened — "the schema is the source of truth" — I made you a promise: that this `db/schema.ts` would be the *root of a derivation tree*, that every typed shape your app needs downstream would be **generated** from it rather than retyped by hand. That the hand-written `type Invoice` you'd otherwise scatter across five files was a smell, because it's a copy that rots the instant someone renames a column.

This lesson is where that promise comes due. You have a Server Component that needs an `Invoice[]` prop. Where does that `Invoice` type come from? Not from a hand-typed interface — from the schema, in one line. By the end you'll be able to delete any `type Invoice` you'd ever write and replace it with a single expression that *cannot* drift, because it's reading the table you already built. Two helpers do this — `$inferSelect` and `$inferInsert` — and they're the wires that make the whole "change one file, every consumer's type checker lights up" loop actually work.

## `$inferSelect`: the row you read back

Here's the first helper, and it's the one you'll reach for most. Every table you've built — `invoices`, `organizations`, all of them — carries a property called `$inferSelect`. It resolves to the TypeScript type of one row, exactly as you'd get it back from a <Term definition="db.select() reads rows out of the database.">`db.select()`</Term>. You don't compute it, you don't maintain it — Drizzle derives it from the column definitions you already wrote.

One thing to get right immediately, because beginners trip on it: `$inferSelect` lives at the **type level**, not the value level. You reach it through `typeof`:

```ts
type Invoice = typeof invoices.$inferSelect;
```

You never write `invoices.$inferSelect` as a runtime value — there's nothing to run, it's purely a type-side projection of the table. Read `typeof invoices.$inferSelect` as "the type of the rows in the `invoices` table," and you'll have it.

That one line is the whole payoff of this chapter, cashed in. Anywhere you'd have hand-typed this:

```ts
type Invoice = {
  id: string;
  organizationId: string;
  amountDue: string;
  status: 'draft' | 'sent' | 'paid' | 'void';
  assignedToId: string | null;
  createdAt: Date;
  tags: string[];
};
```

You write this instead, and the entire category of "stale hand-typed row interface" disappears:

```ts "typeof invoices.$inferSelect"
type Invoice = typeof invoices.$inferSelect;
```

The two produce the *same* type today. The difference is that the second one re-derives itself the moment you touch the schema, and the first one quietly lies the moment someone else does.

So what does that inferred type actually contain? Every column maps to a TypeScript member by a fixed set of rules — the same per-column builders you chose back when you picked your data types. Let me walk the `invoices` table one column at a time and show you exactly how each one lands.

## `$inferInsert`: only what you must supply

`$inferSelect` was the easy half — one row, one-to-one with what's stored. Its sibling is where the interesting idea lives.

`$inferInsert` is the type that <Term definition="db.insert(invoices).values(...) adds a new row.">`db.insert(invoices).values(...)`</Term> accepts — the shape of a row you're *creating*. Same access pattern, type level, through `typeof`:

```ts
type NewInvoice = typeof invoices.$inferInsert;
```

And here's the thing that surprises people: `NewInvoice` is *not* the same as `Invoice`. It's narrower. The row you read back has `id`, `createdAt`, and `status` fully populated — but when you *insert*, you don't supply those. The database fills them in. So why would the insert type force you to provide a value the database is going to generate anyway?

It doesn't. The asymmetry between read and write isn't Drizzle being clever — it's the schema's own facts projected into TypeScript. A column with a default, a column the database computes, a column the app is genuinely responsible for: each of those is a different obligation, and `$inferInsert` encodes all three. Three rules govern how a column on `$inferSelect` becomes (or doesn't become) a member on `$inferInsert`. They mirror the column-modifier checklist you already know, so they should feel familiar.

**Rule 1 — has a default → optional.** Any column with `.default(...)`, `.defaultNow()`, or `.$defaultFn(...)` becomes *optional* on insert. You *may* pass a value; if you don't, the database (or Drizzle) fills it. The headline case is the primary key: `id: uuid().primaryKey().default(sql\`uuidv7()\`)` has a default, so it's optional on `$inferInsert` — you almost never pass an `id` yourself, you let the database mint a fresh UUIDv7. Same for `createdAt` (a `defaultNow()` SQL default) and `status` (a `.default('draft')`). The two flavours of default — a SQL `DEFAULT` clause the database applies, versus a `$defaultFn` Drizzle runs in JS before sending the insert — both land the column in the same place: optional. The *mechanism* differs; the *type* is identical.

**Rule 2 — generated → omitted entirely.** A column declared `.generatedAlwaysAs(...)` or `generatedAlwaysAsIdentity()` is *absent* from `$inferInsert`. Not optional — gone. The database computes it from other columns, so supplying one is a type error, not a courtesy. Think of the `emailLowercased` column on `users`: it's a `.generatedAlwaysAs(...)` column computed from `email`, so it never appears on the insert type. Keep this sharply distinct from Rule 1 — **optional is not omitted.** A defaulted column is there and you may skip it; a generated column isn't there at all.

**Rule 3 — `.notNull()` with no default → required.** What's left over. A column that's `NOT NULL` and has no default is one the app *must* provide — there's no value for the database to fall back on. On `invoices`, that's `organizationId` and `amountDue`: every insert has to carry both, and the type enforces it. Everything else falls out of Rules 1 and 2.

There's one nuance here that catches people, and it's worth slowing down for. A defaulted column is made **optional**, which means `undefined` is fine — you may omit it. It does *not* become **nullable**. Passing `null` is only legal if the column is *itself* nullable. So `createdAt?: Date` (you may skip it) is a completely different thing from `assignedToId: string | null` (you may pass `null`). `?: T` and `: T | null` look similar and mean opposite things — one is "you don't have to provide this," the other is "you may provide nothing-on-purpose."

Now let me put the two shapes eye to eye so the diff is unmissable. I'll switch to the `users` table for a moment, because it's the one that exercises all three rules at once — it has defaulted columns (`id`, `createdAt`), a required one (`email`), and the generated `emailLowercased` from earlier in the chapter.

## Why read and write are two shapes

Step back from the mechanics for a second, because the *reason* matters more than the rules — once you have the reason, you can predict the insert shape of a column you've never seen.

Here it is in one line: **read returns everything stored; write accepts only the subset the app is responsible for.** The database owns the rest — the values it defaults, the columns it computes. `$inferSelect` is the full row because reading gives you the full row. `$inferInsert` is narrower because inserting is only your half of the work; the database does the other half.

This is precisely the thing a hand-written interface *can't* express. To say "this field is required when you read it but optional when you write it," you'd need two separate types, both maintained by hand, both drifting independently. The schema says it once. One `pgTable` declaration knows both shapes, because the facts that distinguish them — which columns have defaults, which are generated, which are `NOT NULL` — are facts the schema already records. That's the deeper *why* behind "generate, don't hand-copy": it's not just less typing, it's expressing a relationship plain TypeScript has no way to state.

The picture makes it concrete. Think of the row as a set of column chips. Reading hands you all of them; writing hands you a strict subset — defaulted chips optional, generated chips gone.

This same split shows up again later, one layer down. When you generate runtime validators from the schema (a library called drizzle-zod, a couple of units from now), it ships *two* generators — `createSelectSchema` and `createInsertSchema` — for exactly this reason. Read and write are two shapes everywhere the schema touches: at the type level here, at the validation level there. Same asymmetry, same single source.

## Place the types next to the table, name them well

You've seen the line. Now, where does it live, and what do you call it? There's a convention, and the project chapters lean on it, so let's pin it down.

**Put the type exports directly under the table they derive from**, right there in `db/schema.ts`:

```ts
export const invoices = pgTable('invoices', {
  id: uuid().primaryKey().default(sql`uuidv7()`),
  organizationId: uuid().notNull(),
  amountDue: numeric({ precision: 12, scale: 2 }).notNull(),
  status: invoiceStatus().notNull().default('draft'),
  createdAt: timestamp({ withTimezone: true }).notNull().defaultNow(),
});

export type Invoice = typeof invoices.$inferSelect;
export type NewInvoice = typeof invoices.$inferInsert;
```

The type and its source sit in one file, three lines apart, so nobody ever hunts for "where's the `Invoice` type." And note what these two `export type` lines are: they're *re-exports of a derived shape*, not hand-copies. Back in the first lesson of this chapter I drew a line between "retyping the schema" (forbidden — it drifts) and "deriving from the schema" (encouraged). These pass the test cleanly: they restate *zero* field names. This is the one place writing a `type` for a row is not just allowed but correct.

The naming convention — memorize it, because it's the course standard and the projects reuse it verbatim:

- **The select type is the entity noun**: `Invoice`, `Organization`, `Membership`.
- **The insert type is the noun with a `New` prefix**: `NewInvoice`, `NewOrganization`, `NewMembership`.

Downstream files pull these in as type-only imports — `import type { Invoice } from '@/db/schema'` — because they're type-only symbols and your TypeScript config enforces the `import type` form for those. One line; nothing more to it.

One historical footnote so you recognize it in the wild.

:::note
You'll see an older spelling in legacy code and on Stack Overflow: `InferSelectModel<typeof invoices>` and `InferInsertModel<typeof invoices>`, imported from `drizzle-orm`. These are generic helpers that produce the *exact same types* as `typeof invoices.$inferSelect` / `$inferInsert` — just the older way to write them. The course uses the `$infer*` property form throughout: it's terser, and it's what Drizzle's own docs lead with. If you meet `InferSelectModel`, read it as "the same thing, spelled the long way."
:::

## Compose new shapes without restating fields

So far you've replaced hand-typed rows one-for-one. Here's where it gets genuinely powerful — and where the real-world payoff lives. Most of the time you don't want the *whole* row. A list view wants three columns plus the org's name. An update form wants a partial. The temptation is to hand-type those narrower shapes. Don't — build them *from* the inferred types, and they inherit the same can't-drift guarantee.

The rule: **every derived shape roots in `$inferSelect` or `$inferInsert`.** You never restate a field name; you compose with the utility types you already know. Here's the canonical example — a summary type for a dashboard list:

<AnnotatedCode lang="ts" maxLines={14} code={`
type InvoiceSummary = Pick<Invoice, 'id' | 'status' | 'amountDue'> & {
  organizationName: Organization['name'];
};

// The 80% update shape — a partial of the insert type:
type InvoiceUpdate = Partial<NewInvoice>;

// The precise update shape — id required (which row?), the rest optional:
type PreciseUpdate = { id: Invoice['id'] } & Partial<Omit<NewInvoice, 'id'>>;
`}>
  <AnnotatedStep meta={`"Pick<Invoice, 'id' | 'status' | 'amountDue'>"`} color="green">
    `Pick` pulls those three members off the inferred `Invoice` — **names and their types**, both, straight from the source. Rename `amountDue` in the schema and this line breaks at compile time too. Zero field types restated.
  </AnnotatedStep>

  <AnnotatedStep meta={`"Organization['name']"`} color="blue">
    Indexed access: `Type['field']` reads one field's type directly out of the source type. `organizationName` gets whatever `Organization.name` is, without you naming `string`. The join column borrows its type from the column it joins to.
  </AnnotatedStep>

  <AnnotatedStep meta="{6}" color="orange">
    `Partial<NewInvoice>` makes every insert field optional — the 80% answer for an update payload, where the caller sends only the columns that changed.
  </AnnotatedStep>

  <AnnotatedStep meta="{9}" color="violet">
    The precise shape, for when you want the type to enforce "you must say *which* row." `id` is pinned required via indexed access; `Omit` drops `id` from the insert type, then `Partial` makes the remaining columns optional. More precise, slightly more machinery — reach for it when the looseness of `Partial<NewInvoice>` actually bites.
  </AnnotatedStep>
</AnnotatedCode>

Look at what's *not* in that block: not a single `string`, not a single `'draft' | 'sent' | …`, not one field type written by hand. `Pick`, `Omit`, `Partial`, the `&` intersection, the `Type['field']` indexed access — these are the five tools of composition, and you met all of them in the TypeScript lessons. Here they're doing real work: every shape `InvoiceSummary`, `InvoiceUpdate`, and `PreciseUpdate` describes is *anchored* to the schema. Rename a column and they all break at the type checker — which is exactly what you want, because a break at compile time is a bug you never shipped.

On the update shapes: `Partial<NewInvoice>` is the answer you'll reach for most. The `id`-pinned variant is more correct — it won't let you build an update that forgot to say which row — but don't reach for it reflexively. Name the trade-off, pick the simple one until precision earns its keep, and move on.

## One type, one source: the prop pattern

Let me show you the whole thing flowing, end to end, because "one type, one source" is abstract until you watch a single name thread every layer.

A Server Component reads invoices from the database and hands them to a Client Component to render. (The query itself is the next chapter's job — here, just picture its result.) Watch the `Invoice` type:

<div data-mark-color="blue">

```ts /\bInvoice\b/
// db/schema.ts
export type Invoice = typeof invoices.$inferSelect;

// db/queries/invoices.ts — the next chapter owns the query body
export const listInvoices = async (): Promise<Invoice[]> => {
  // ...
};

// app/invoices/invoice-list.tsx
type Props = { invoices: Invoice[] };

export const InvoiceList = ({ invoices }: Props) => {
  // render the rows
};
```

</div>

The *same* `Invoice` appears four times — the schema export, the query's return annotation, the component's `Props`, the destructured parameter — and it's one declaration the whole way down. The schema defines it; everything else borrows it. Now run the counterfactual from the first lesson of this chapter: in the world where `InvoiceList` hand-typed its own prop interface, a column rename in the schema would update the query but leave that prop quietly stale — a `Props` that no longer matches the data flowing into it, and a type checker that says nothing because the two were never connected. Thread one type through instead, and the rename breaks every layer at once, which is the only way you find out *before* production does.

And the chain keeps going past the type. The same `invoices` table feeds the query's row type, the component prop, *and* — when you get to forms and Server Actions — the runtime validator (drizzle-zod again) the action parses its input with. Schema table → `$inferInsert` for the type → drizzle-zod for the validator → both consumed by the Server Action that writes the row. Every layer aligns because every layer reads the same root.

## When inference is wrong, and the relations gap

Inference is excellent, but it's not magic, and an experienced engineer knows where it stops. Four honest edges — none of them defects, all of them things you should *expect* so they don't surprise you later.

**`jsonb` without `$type` infers as `unknown`.** A bare `jsonb()` column tells the compiler nothing about its contents, so `$inferSelect` gives you `unknown` — correct, but useless to work with. The fix isn't a cast at the read site; it's upstream, on the column, exactly where you learned it: claim the shape with `$type<...>()`.

<CodeVariants maxLines={14}>
  <CodeVariant label="Without $type">
    <div data-mark-color="red">

    ```ts {6}
    export const webhookDeliveries = pgTable('webhook_deliveries', {
      payload: jsonb().notNull(),
    });

    type Delivery = typeof webhookDeliveries.$inferSelect;
    // payload: unknown   ← nothing to work with
    ```

    </div>
    **Inference is only as good as what you told the schema.** A bare `jsonb()` carries no shape, so the row's `payload` is `unknown` — you'd have to narrow it by hand at every read.
  </CodeVariant>

  <CodeVariant label="With $type<WebhookEvent>()">
    <div data-mark-color="green">

    ```ts "WebhookEvent"
    export const webhookDeliveries = pgTable('webhook_deliveries', {
      payload: jsonb().$type<WebhookEvent>().notNull(),
    });

    type Delivery = typeof webhookDeliveries.$inferSelect;
    // payload: WebhookEvent   ← the real shape
    ```

    </div>
    **`$type<T>()` is a compile-time promise** about the stored shape. `$inferSelect` now resolves `payload` to `WebhookEvent`. Postgres still stores bytes and Zod still validates the data on the way in — `$type` only informs the *type*, it doesn't enforce anything at runtime.
  </CodeVariant>
</CodeVariants>

That `$type` claim is a *promise to the compiler*, not a runtime guarantee — Postgres stores whatever bytes you write, and validating that the bytes actually match `WebhookEvent` is Zod's job at the boundary. Same idea applies to the next two edges: when inference gives you something too wide, the fix is in the *schema*, not a cast at the consumer.

**`numeric` stays `string`.** You met this already — `amountDue: string`, not `number`. It's listed here too because it's the same family of "the type isn't what a beginner expects," but it is *deliberate*: the string preserves arbitrary precision, and the chapter doesn't "fix" it because there's nothing broken. Read `numeric → string` as the money guarantee, every time.

**An enum-like `text` column gives you `string`, not a union.** If a column holds a fixed set of values but you declared it as plain `text()`, `$inferSelect` infers `string` and you've lost the closed set — the compiler will happily let `'pending'` sit in a column that only ever holds `'draft' | 'sent' | 'paid' | 'void'`. The fix is upstream: make it a `pgEnum`, and the union comes back for free. Don't paper over it with a cast at the read site — fix it in the schema, and every consumer inherits the narrow type.

**`$inferSelect` is flat — it does not include relations.** This is the big one, and the most common wrong expectation. An inferred `Invoice` has `organizationId: string` — the raw foreign-key column — and *that's it*. There is no nested `organization` object, no `lineItems` array, no `tags`. The relations you declared in the previous lesson have their *own* inferred result types, produced by the relational query API when you actually ask for them (`db.query.invoices.findMany({ with: { tags: true } })`, next chapter's territory). `$inferSelect` is the flat stored row; nested shapes are a *query-time* concern, by design.

:::caution
`$inferSelect` never carries relations. `typeof invoices.$inferSelect` gives you the flat row — foreign keys as bare `string` columns, no nested objects. If you need an invoice *with* its line items and tags as one typed object, that's a different shape entirely, inferred from the relational query (`db.query.…({ with })`) in the next chapter. Don't expect `with: { tags: true }` to show up in `$inferSelect` — it can't, because the flat row knows nothing about the join.
:::

One last note in the same spirit: if a query selects only *some* columns rather than the whole row, its result type is narrower than `$inferSelect` — and you should let it infer rather than restating *that* shape by hand either. Custom-select result types are the next chapter's topic; just know the principle ("don't hand-type what the query already infers") extends there too.

## The codebase rewrites itself

Here's the loop you can now picture from end to end. You change a column in `db/schema.ts` — rename it, retype it, make it `NOT NULL`. The inferred types update on their own, because they *are* the schema. You run the type checker. And every consumer that touched the old shape lights up red at once: the query's row type, the component prop, the composed `InvoiceSummary`, the update payload. You walk the squiggles, fix each one, and ship — having found every break at edit time instead of in production.

That's the promise from the first lesson of this chapter, now *true*. Back then it was a claim: "change one file, every downstream layer's type checker catches the drift." `$inferSelect` and `$inferInsert` are the wires that make the claim hold — they're what connect the schema to everything that reads it. The first lesson named the principle; the lessons in between built the file; this one made every downstream shape a *branch* of it. So the hand-typed `type Invoice` isn't just discouraged anymore — it's *unnecessary*, because the one-line derivation is strictly better in every way that matters: shorter to write, impossible to drift, and free to maintain.

That's the chapter. You have a schema that's genuinely the source of truth, and the two helpers that make "source of truth" mean something. The quiz next will let you check what stuck.
