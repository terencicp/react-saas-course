The last lesson handed you the page's rooms — `<header>`, `<nav>`, `<main>`, `<footer>`, and a heading outline inside them. Now you wire what the user clicks inside those rooms. Take one ordinary screen in the Acme invoicing app: it has a **Save** button that writes the open invoice, a **Go to dashboard** action that takes you to `/dashboard`, and a short **list of plan features** in the sidebar. Three things on screen — and the naive instinct reaches for the same tool for all three: a `<div onClick>` for Save, a `<button>` for the navigation because that's the tag you styled last, a stack of `<div>`s for the features. It renders. It looks right.

Here is that screen written two ways. Read the first sentence of each tab before the code.

Open both in a browser and you cannot tell them apart. That is the trap. The two versions render the same pixels and behave completely differently the moment anyone uses a keyboard, a screen reader, or a right-click — and the difference is invisible to the person who wrote it. This lesson teaches the one decision that separates them, and it is a decision, not a default: **behavior picks the element, the class picks the look.** A control that *performs an action* is a `<button>` even when it's styled to look like a link. A control that *goes to a URL* is a link even when it's styled to look like a button. The look is a separate knob you'll reach for in the next chapter with Tailwind — it never changes which element is correct. That is the same split the last lesson left you with, where the element carried the page outline and the class carried the look; here it carries the page's *interactivity*.

By the end you'll pick correctly by reflex across three families — the things that **do** something, the things that **go** somewhere, and the things that are a **sequence** — and you'll know exactly how much free behavior you'd throw away by guessing wrong.

## Buttons perform actions

A <Term definition="The HTML element for a control that performs an action in place — submitting, toggling, opening.">`<button>`</Term> is a control that performs an action *right here on this page*: submit a form, open a modal, delete a row, toggle a sort order. The defining trait is that it acts and you stay put — nothing about a `<button>` is navigation. (Navigation is the next section.) When the thing you're building does something on click and doesn't take the user to a URL, it's a button, full stop.

So why not a `<div onClick>`? It fires the same handler. The answer is everything the browser does for a real `<button>` that you'd otherwise write by hand:

- **Keyboard focus.** A `<button>` is in the `Tab` order automatically. A `<div>` is not.
- **Keyboard activation.** `Enter` *and* `Space` fire a `<button>`'s click, the convention every keyboard user expects. A `<div>` responds to neither.
- **A focus ring.** The browser draws a visible focus indicator when the button is tabbed to, so a keyboard user can see where they are.
- **The role.** A screen reader announces it as "button," so a non-sighted user knows it's actionable and how to activate it. A `<div>` announces nothing.
- **Disabled handling.** Setting `disabled` removes the button from the activation path and dims it, in one attribute.

Every one of those is behavior you don't write. Reach for `<div onClick>` and you've quietly signed up to rebuild all five by hand — and the cost of that is the whole back half of this lesson.

:::note
One thing the browser's default `<button>` look does *not* survive in this course: the grey native button chrome. Tailwind ships a base reset called <Term definition="Tailwind's base layer, applied automatically. It zeroes default browser styling — button chrome, list bullets, heading sizes — so every element starts visually neutral and your utility classes paint it.">Preflight</Term> (you'll meet it in the next chapter) that strips that default styling, so in this course buttons start visually bare and your utility classes paint them. The *element* is still a button with all five behaviors above — only the paint is gone.
:::

### `<button type>` is not optional

Here is the single most common bug in this whole territory, and it is worth slowing down for. A `<button>` placed **inside a `<form>` defaults to `type="submit"`** — meaning a click submits the surrounding form, whether or not that's what you intended. The attribute takes one of three lowercase values (lowercase matters — `type="Submit"` is silently wrong, the same kind of watch-out you saw with JSX attribute casing):

- `type="submit"` — submits the form. This is the default inside a form.
- `type="button"` — inert by default; runs only its own `onClick`, submits nothing.
- `type="reset"` — clears the form back to its initial values. Rare, surprising to users, and an anti-pattern in practice — know it exists and move on.

The reflex an experienced developer carries: **every `<button>` declares its `type` explicitly.** Not because the default is wrong, but because the default is *invisible* and the failure is expensive. Picture a form with two buttons side by side — **Save** and **Cancel** — and the Cancel button has no `type`. The user fills in half an invoice, decides to bail, clicks Cancel — and because Cancel defaulted to `submit`, the form submits the half-finished invoice instead of abandoning it. The user lost their work and nobody wrote a single line of buggy logic to cause it. One missing attribute did.

In the next lesson you'll wire this form to a real Server Action; here it's just the shell, so you can watch the one-attribute fix.

### Disabled buttons must say why

`disabled` is a boolean attribute — write `disabled` to turn it on, omit it to leave it off, the same on-or-off prop shape you've already seen in JSX. A disabled button can't be activated, the browser dims it, and it drops out of the activation path entirely.

The watch-out is about the user, not the syntax. A disabled button is often *also skipped in the `Tab` order*, and it offers the user no explanation for why it won't work. So a customer stares at a greyed-out **Send invoice** button with no idea that it's waiting on a required field three rows up. The rule: never hide a critical action behind a bare `disabled` without surfacing the reason somewhere the user can find it — an inline message, a tooltip, or wiring the button to a hint with `aria-describedby` (you'll meet that attribute properly in the data-and-aria lesson). Disabling a control is easy; leaving the user stranded behind it is the actual mistake.

### Icon-only buttons need a name

Plenty of buttons in a real app have no text — a trash can to delete a row, a copy glyph, an X to dismiss. They look obvious to a sighted user. To a screen reader, a `<button>` whose only child is an icon has no <Term definition="The label assistive technology announces for a control. It comes from the element's text content, or from an aria-label / aria-labelledby when there's no visible text.">accessible name</Term> — the user hears "button," nothing more, with no way to know what it does. The fix is one attribute: `aria-label` gives the button a name that assistive tech reads aloud, without putting any visible text on screen.

<CodeTooltips tooltips={{
  'aria-label': "The accessible name a screen reader announces. Required when the button has no visible text child — without it the user just hears 'button'.",
  type: 'Always declare it. button keeps this inert to any surrounding form.',
}}>
```tsx
<button type="button" aria-label="Delete invoice" onClick={remove}>
  <TrashIcon />
</button>
```
</CodeTooltips>

One trap to bank for later: `aria-label` *replaces* the accessible name, so putting it on a button that already has visible text overrides that text — the screen reader reads the label and ignores the words on screen. Only reach for it when there's no visible text to begin with. That's the full extent of ARIA you need today; the data-and-aria lesson goes deep on the rest.

## Links navigate to URLs

A link is the mirror image of a button: where a button acts and keeps you put, a <Term definition="The anchor element — a control that navigates to a URL.">link</Term> **takes the user to a URL**. That URL might be another page in your app, an external site, a downloadable file, or a spot further down the current page. If the job is "go somewhere," it's a link.

### `href` is what makes a link a link

The single most important thing about an `<a>` is its `href`. An `<a>` *with* an `href` is a real link — it's focusable, `Enter` activates it, right-click offers "Open in new tab" and "Copy link," `Cmd`/`Ctrl`-click and middle-click open it in a background tab, the destination shows in the status bar when you hover, and a search crawler can follow it. An `<a>` *without* an `href` is inert: not focusable, not activatable, no role — a dead element wearing link clothes.

That cuts both ways. If you find yourself writing an `<a>` with an `onClick` and no `href`, stop — a control that acts on click is a `<button>`, not a stripped link. The presence of `href` is the tell: it means "this goes somewhere." No destination, no link.

### `target="_blank"` travels with `rel`

By default a link replaces the current page. Add `target="_blank"` and it opens in a new tab instead — the right call for an external site you don't want to yank the user away to. But `target="_blank"` carries a security and privacy obligation, and pairing it with `rel="noopener noreferrer"` is the reflex:

- `noopener` severs the new page's handle back to the page that opened it. Without it, the opened site can reach back through `window.opener` and silently redirect your original tab to a phishing clone — an attack called <Term definition="An attack where a page opened via target=_blank uses window.opener to redirect the original tab to a look-alike phishing page while the user is reading the new one.">tabnabbing</Term>.
- `noreferrer` strips the `Referer` header, so the destination isn't told which of your pages sent the visitor.

Modern browsers now imply `noopener` for `target="_blank"` on their own, so the worst case is mostly closed by default. Writing the `rel` explicitly is still the reflex — defaults shift, older embedded webviews lag, and a senior reviewer doesn't want to audit which environments are safe. State the intent; don't lean on the browser to guess it.

Two more `rel` values you'll see: `nofollow` on user-generated links (a comment, a profile bio) tells crawlers not to lend your site's ranking to whatever a stranger pasted, and `external` is mostly a styling hook. Neither is load-bearing the way the security pair is.

### Files and in-page jumps

Two smaller link jobs round this out. `<a download href="/report.pdf">` turns a link into a file download instead of a navigation — an optional value (`download="invoice.pdf"`) renames the saved file. And an `href` that starts with `#` points at an element on the *same* page by its `id`: `<a href="#pricing">` scrolls to whatever element has `id="pricing"` and updates the URL hash, no new page load. That's how a table of contents and the "skip to main content" link from the last lesson work. The target just needs a unique `id`, and you can soften the jump into a smooth scroll with one CSS line (`scroll-behavior: smooth`, which Tailwind exposes as `scroll-smooth`).

### `<Link>` is `<a>` plus client-side navigation

For links *inside your own app*, Next.js gives you a better anchor. <Term definition="Next.js's navigation component, imported from next/link.">`<Link>`</Term> is not a separate thing to learn — it's an `<a>` with a superpower. `<Link href="/dashboard">` **renders to a plain `<a href="/dashboard">` in the HTML the browser receives**, so everything you just learned about real anchors still holds: the crawler follows it, `Cmd`-click opens a new tab, "Copy link" works, and if JavaScript fails to load the link still navigates the old-fashioned way.

What it *adds* is <Term definition="A route change handled by JavaScript on the client, swapping only the parts of the page that changed — no full-document reload, no white flash.">soft navigation</Term>: clicking it changes the route without a full-document reload. No white flash, no re-downloading the whole page — just the parts that changed. The decision rule is flat:

> Internal routes use `<Link>`. External links use a plain `<a target="_blank" rel="noopener noreferrer">`.

The trap here is a link wearing button clothes. You'll see `<button onClick={() => router.push('/dashboard')}>` in real codebases — a button whose entire job is to navigate. It works on click, and it's wrong: it's not a real link, so it loses copy-link, middle-click, and crawlability, and an experienced reviewer flags it on sight. If the control goes to a URL, it's a `<Link>`, no matter how button-like it looks.

## Choosing the element: button, link, or div

You now have both halves, so here's the decision they were building toward. It's one sentence: **match the element to the behavior, not the look.** Everything in this lesson collapses into that, and two corollaries you should be able to recite cold:

- A `<button>` styled to look like a link still *acts*. It's a button.
- An `<a>` styled to look like a button still *navigates*. It's a link.

The styling is downstream and changeable. The element encodes what the control *does*, and that's the part that has to be right.

### What `<div role="button">` really costs

The most honest way to see why the semantic element wins is to try to fake it. Suppose you insist on a `<div>` and want it to behave like a button. Here's the bill:

- `role="button"` so assistive tech announces it correctly.
- `tabIndex={0}` so it's reachable by `Tab`.
- An `onKeyDown` handler that fires on `Enter` *and* `Space` — and calls `preventDefault` on `Space`, or the page scrolls instead of activating.
- Focus-ring CSS, because you've lost the browser's default one.
- `aria-pressed` if it's a toggle.
- `cursor-pointer`, because a `<div>` doesn't get the pointer cursor.

That is the exact list a real `<button>` hands you for free, now rebuilt by hand — more code, more surface area for bugs, and a near-certainty you'll forget the `Space`-key `preventDefault` or the focus ring. The verdict is blunt: **never.** Reach for `<button>` and restyle it. (The same logic kills `<div role="link">` — if it navigates, it's an `<a>`.)

## Lists group related items

The third family is the lightest, because you already met it: <Term definition="An unordered list — a bullet list where order doesn't carry meaning.">`<ul>`</Term> for an unordered sequence, <Term definition="An ordered list — a numbered list where order is meaningful.">`<ol>`</Term> for an ordered one, and <Term definition="A list item — the only valid direct child of a <ul> or <ol>.">`<li>`</Term> for each item. The one structural rule: `<li>` is the *only* thing that goes directly inside a `<ul>` or `<ol>` — nothing else belongs there as a direct child. Any sequence of related, parallel items is a list: navigation links, a feature grid, a comment thread, audit-log entries.

You render them with the `.map` pattern from the JSX lesson — and the key rule comes along unchanged: **one `key` per `<li>`, tied to the data's identity, never the array index.**

<div data-mark-color="green">

```tsx "key={feature.id}"
<ul>
  {features.map((feature) => (
    <li key={feature.id}>{feature.name}</li>
  ))}
</ul>
```

</div>

Lists can nest for hierarchy — a `<ul>` inside an `<li>` for a file tree or a threaded comment — and `<ol>` takes attributes like `start`, `reversed`, and `type` when you need to control the numbering, though you'll usually leave that to the content. Recognize them; you won't reach for them often.

### List, or just a row of divs?

Not everything that sits in a row is a list, and over-using `<ul>` is as much a smell as under-using it. The test an experienced developer asks is short: **would a screen-reader user usefully hear "list, three items" here?** If yes, it's a list. Navigation links, feature cards, comments under a post — related and parallel, so they're a list, and the "N items" announcement is genuinely useful. A logo next to a sign-in button in the header, or a hero headline beside its call-to-action button — unrelated things that merely happen to be near each other — are *not* a list. They're just elements in a container.

The watch-out is the inverse of the button trap: a nav built as a bare row of `<a>`s renders fine and silently drops the "list of N items" announcement that tells a screen-reader user how many destinations there are.

### The nav-list pattern

This ties the whole chapter together. The last lesson named the primary navigation as a list but didn't build it — here it is, and it composes four things you already know. The canonical SaaS nav is a `<nav>` landmark wrapping a `<ul>` of `<li>`, each holding a `<Link>`.

<AnnotatedCode lang="tsx" maxLines={14} code={`
const links = [
  { href: '/dashboard', label: 'Dashboard' },
  { href: '/invoices', label: 'Invoices' },
  { href: '/customers', label: 'Customers' },
];

function PrimaryNav() {
  return (
    <nav aria-label="Primary">
      <ul className="flex gap-4 list-none">
        {links.map((link) => (
          <li key={link.href}>
            <Link href={link.href}>{link.label}</Link>
          </li>
        ))}
      </ul>
    </nav>
  );
}
`}>
  <AnnotatedStep meta="{9}" color="blue">
    The `<nav>` landmark from the last lesson, named with `aria-label` so a screen reader can tell this nav from any other on the page.
  </AnnotatedStep>

  <AnnotatedStep meta="{10}" color="green">
    A real semantic list — so assistive tech announces "list, 3 items." `flex gap-4` lays it out horizontally; `list-none` drops the bullets. The semantics survive the styling: it's still a list, just without visible markers.
  </AnnotatedStep>

  <AnnotatedStep meta={`{11-12} "key={link.href}"`} color="violet">
    `.map` over the data, one `<li>` per link, each keyed by its `href` — the stable, data-tied key from the JSX lesson, not the index.
  </AnnotatedStep>

  <AnnotatedStep meta="{13}" color="orange">
    An internal `<Link>` inside each `<li>`: soft navigation, and still a real `<a href>` underneath.
  </AnnotatedStep>
</AnnotatedCode>

That `flex gap-4 list-none` is the one piece of styling you'll see in this lesson's examples, and it's deliberate — it's the only way to show you the canonical "semantic `<ul>`, custom horizontal look" shape. Everywhere else the examples stay intentionally bare. Real components reach for Tailwind utilities and a shared `<Button>` component, both of which come later in the course; don't read the under-styling here as the finished form.
:::

The payoff is the synthesis: the `<nav>` landmark, the list semantics, the data-tied keys, and `<Link>` navigation — four ideas from across this chapter, composed into the one piece of UI nearly every SaaS app ships, with the bullets gone but the meaning intact.

## Buttons and links inside a form

One last thread to close, and it's a handoff to the next lesson on forms. You've already seen that a `<button>` inside a `<form>` defaults to `type="submit"`. Two facts make the explicit-`type` reflex non-negotiable in form context: that default, plus the fact that pressing `Enter` in almost any text input *also* submits the form. So a form full of buttons — a Save, a Cancel, a "toggle advanced options" — submits on the first stray `Enter` or the first untyped button click unless every non-submit button is marked `type="button"`. That's the whole reason the reflex exists.

You'll occasionally see a submit button carry its own `formAction`, `formMethod`, `formEncType`, `formNoValidate`, or `formTarget` — these override the parent form's matching attributes *for that one button*, so two submit buttons can post to different places. Recognize them; you'll reach for them almost never. The real form machinery — inputs, labels, the `name`-to-data contract, and the Server Action that consumes it — is the next lesson.

## Wrap-up

Three families, one decision. A control that *acts* is a `<button>` — and you declare its `type` so a form can't hijack the click. A control that *goes somewhere* is a link — a `<Link>` for internal routes, a plain `<a target="_blank" rel="noopener noreferrer">` for the outside world — never a `<div>` or a button in disguise. A sequence of related, parallel items is a `<ul>`/`<ol>` with one data-tied `key` per `<li>`. Behind every one of those choices sits the same rule the whole lesson hung on: behavior picks the element, the class picks the look. Pick by behavior and the browser hands you keyboard support, focus, roles, and the right announcements for free; guess by appearance and you've signed up to rebuild all of it by hand. The next lesson takes the form you kept glimpsing here and wires it into a real submission.
