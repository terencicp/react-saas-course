You sit down to build a sign-up form for Acme, your invoicing app. If you learned web development from a tutorial in the last decade, your fingers already know what to type: a `useState` for the email, another for the password, a `value` and `onChange` wired to each input, and a submit handler that reads all that state back out and assembles an object to send. That is roughly thirty lines of plumbing per form, and you have written all of it before validating a single field.

There is another way to start, and it is the one an experienced engineer reaches for: a plain `<form>` of labeled inputs, each carrying a `name`, a `<button type="submit">`, and a server that reads the submission and validates it. No `useState`. No submit handler. The browser already knows how to collect the fields and send them; you are spending those thirty lines reimplementing a feature the platform ships for free.

This lesson installs that second instinct. We are not wiring up form *behavior* yet ā€” submitting to a server, showing pending spinners, rendering errors all come later. Today is about the markup: which elements, which attributes, and the one idea everything hangs off ā€” **a form is a contract the server reads.** The set of `name` attributes you put on the form and the keys your server expects are the same list, and an experienced engineer writes them together. By the end you will be able to hand-author the full Acme sign-up form ā€” every field with the right element, a `name`, a `<label>`, the right keyboard, and cheap validation ā€” and explain why the submit button works without a line of JavaScript.

## A form is a contract, not a pile of inputs

Before any element detail, hold the whole round-trip in your head as four steps. It is small, and it is the spine of the entire lesson.

1. The user fills in some labeled fields and clicks submit.
2. The browser walks every control in the form, reads each one's `name` and current value, and packs them into a <Term def="A browser object that holds a form's fields as name/value pairs. You met it in chapter 15 as a body you can hand to `fetch`; here you see where it comes from ā€” the browser builds one from your named inputs.">`FormData`</Term> object.
3. That `FormData` crosses the wire to your server.
4. On the server, a <Term def="A TypeScript-first schema validation library. You describe the shape you expect; it checks an unknown value against that shape and gives you back a typed object or a list of errors. Full treatment in a later chapter.">Zod</Term> schema validates the `FormData` and hands you back a clean, typed object.

Here is the punchline. Steps 2 and 4 speak the *same vocabulary* ā€” the `name` strings. The browser keys the data under `name="email"`; the schema looks for a key called `email`. Get those two lists to agree and the form round-trips with almost no glue code between them. Get them to disagree ā€” a typo, a rename on one side only ā€” and a field the user dutifully filled silently never arrives. That shared list of strings is the contract. It is why an experienced engineer writes the form and the schema in the same sitting, as one design, rather than building the form and then figuring out how to read it.

One more thing to name before we touch markup, because it changes how you weigh every decision in this lesson. A native HTML form is <Term def="A page works as plain HTML first; JavaScript layers extra behavior on top. If the JS never loads, the baseline still functions.">progressive enhancement</Term> by default. The round-trip you just saw happens whether or not your JavaScript bundle has loaded ā€” the browser submits the form on its own. Everything React adds on top of forms later (richer pending states, inline errors, optimistic updates) is built *on top of* this working baseline, never as a replacement for it. That is the order to keep in your head: the platform default is the floor, and the fancier tools are conditional upgrades you add when a form earns them.

The wire between the browser and your server ā€” whether that is a <Term def="A function that runs on the server and is called directly from a form's `action`. React handles the network round-trip for you. Covered in a later chapter.">Server Action</Term> or a plain POST endpoint ā€” is a later chapter's job. For now, just know the boundary is there and that both sides of it agree on the `name` strings.

## The `<form>` element and the submit path

The contract needs a container, and that is the `<form>` element. It does two jobs: it marks which controls belong together, and it owns *what happens on submit*. Three of its attributes are decisions worth understanding.

**`action`** says where the contract is sent. In plain HTML it is a URL string. In a 2026 React app it is something more interesting ā€” a *function*, the Server Action, and React wires up the network call for you: `<form action={createAccount}>`. That is the default you will reach for, and building it out is a later chapter; for now, recognize the shape. If you omit `action`, the form posts back to the current URL.

**`method`** is `get` or `post`. With `get` (the default) the field values are appended to the URL as a query string ā€” right for a search or filter form, where the submission *is* a navigation you might bookmark or share. With `post` the values travel in the request body, hidden from the URL. The decision rule is one sentence: **anything that changes server state is `post`.** Signing up, creating an invoice, deleting a customer ā€” all `post`.

**`encType`** controls how the body is encoded, and you will almost never touch it. The default handles ordinary fields. There is exactly one trigger to change it: the moment your form contains an `<input type="file">`, switch to `multipart/form-data` so the file's bytes can ride along. That is the whole rule ā€” `encType` is the file-upload knob and nothing else.

Now the submit path, without a line of JavaScript. A `<button type="submit">` inside the form ā€” or simply pressing `Enter` while focused in a text field ā€” triggers the browser's native submit. The browser serializes the named controls and sends them per `method` and `action`. You met `type="submit"` in the previous lesson; the only thing to add here is *why every button in a form needs its `type` spelled out*. A button with no `type` inside a `<form>` defaults to `submit`, so a "Cancel" or "Show password" button you meant to be inert will quietly fire the form instead. Declare `type` on every button and that whole class of bug disappears.

<Aside type="caution">
You cannot nest one `<form>` inside another ā€” it is invalid HTML, and the browser silently drops the inner form rather than telling you. If two submission paths feel like they want to nest, the fix is either two sibling forms or one form with multiple submit buttons. Never a form inside a form.
</Aside>

Here is the Acme sign-up form as a bare shell. It is deliberately incomplete ā€” the inputs have no labels and no `name`s yet, because those are the contract, and the contract is the next two sections. Think of this block as a seed we will grow field by field.

```tsx
<form action={createAccount} method="post">
  <input type="email" />
  <input type="password" />
  <button type="submit">Create account</button>
</form>
```

## The `name` attribute is the contract

Of every attribute in this lesson, this is the one. `name` is the **key** under which a control's value shows up in `FormData`. The relationship is brutally simple and has no middle ground: an input *with* a `name` contributes one entry to the submission; an input *without* a `name` contributes nothing. Not an empty string, not a null ā€” nothing. The control is invisible to the submission entirely.

That is the single most common forms bug, and it is nasty precisely because it is silent. Picture a user typing their organization name into a field, seeing it on screen, clicking submit ā€” and the value never reaches your server, because someone forgot the `name`. No error fires. The form "works." The data is just gone. When you internalize that `name` is the key, this bug stops happening to you, because you stop thinking of `name` as optional chrome and start thinking of it as the wire the value travels down.

So how should you name things? In a codebase built on Server Actions and Zod ā€” which is to say, the stack this course teaches ā€” this is a real decision with a house rule. The schema keys ultimately derive from your database columns, which TypeScript reads as `camelCase`. So the `name` attributes should be `camelCase` too: `name="organizationName"` lines up exactly with the `organizationName` key in your Zod schema and the `organizationName` field on your typed object. One vocabulary across the form, the schema, and the table. Kebab-case names like `name="organization-name"` are perfectly legal HTML, but they force you to translate at the boundary ā€” to map `organization-name` back to `organizationName` somewhere ā€” and that translation layer is pure friction you can delete by just naming the field `organizationName` from the start.

Time to make the silent-drop bug something you have felt rather than just read about. The following exercise gives you a small Acme form wired against a Zod schema, but the contract is broken in three places: two inputs are missing their `name` entirely, and one has a `name` that doesn't match the key the schema expects. Your job is to repair the contract so every field reaches the server under the right key.

## Labels are a second contract ā€” with the user and their tools

A form signs a second contract, and `<label>` is where it signs. It is tempting to read a label as decoration ā€” the little word sitting next to the box ā€” but it is load-bearing infrastructure for three different audiences. For a screen-reader user, the label is what gets announced when focus lands on the field; without it, the input is "edit text, blank" and the user has no idea what to type. For someone with a motor impairment, the label *is* a click target ā€” tapping the word "Email" focuses the input, which is a much larger target than the box alone. And for the password manager or browser autofill engine, the label text is a strong signal for what the field holds. A labeled field is a field the whole ecosystem can understand.

There are two ways to associate a label with its input. The explicit form connects them by id:

```tsx
<label htmlFor="email">Email</label>
<input id="email" name="email" type="email" />
```

The implicit form wraps the input inside the label, and the association is the nesting itself:

```tsx
<label>
  Email
  <input name="email" type="email" />
</label>
```

Both are valid. The one to reach for is the explicit `htmlFor`/`id` pairing, for an unglamorous reason: it survives refactors. The link holds no matter how the markup moves around ā€” you can wrap the input in three more `<div>`s and the association doesn't break ā€” and the form libraries you will meet in a later chapter are built around id-based wiring. (Recall from earlier in this chapter that `htmlFor` is just the JSX spelling of HTML's `for`, which is a reserved word in JavaScript.)

Now the distinction that trips up nearly everyone, because two attributes here look like they do the same job and do not. `htmlFor` takes the input's **`id`**, never its `name`. They are different keys for different machines:

The reason `id` and `name` are easy to conflate is that they so often hold the same string ā€” `id="email"` and `name="email"` ā€” which makes them look interchangeable until the day a radio group breaks the illusion. We will hit that case shortly; for now hold the two ideas apart: **`id` is the field's address for the label; `name` is its key for the server.**

One more thing a `<label>` is *not*: a placeholder. A `placeholder` is the grey hint text inside an empty input, and reaching for it instead of a label is a dated anti-pattern. It vanishes the instant the user starts typing ā€” so it is gone exactly when someone might want to re-check what the field was for. Screen readers treat it inconsistently. Its contrast is usually too low to read comfortably. A placeholder can *supplement* a label with an example ("e.g. ada@acme.com"), but it can never *replace* one.

<Aside type="tip">
The two non-negotiables for every input: a `name` (so the value reaches the server) and a `<label>` (so the user and their tools can understand the field). If you find yourself writing a bare `<input>` with neither, you have skipped both contracts.
</Aside>

## Typed inputs: the `type` attribute is a UX contract

The `<input>` element wears many hats, and the `type` attribute chooses which one. It picks the on-screen keyboard the mobile user sees, the autofill suggestions the browser offers, the native picker (a date wheel, a color swatch), and a layer of cheap validation. What it does **not** do ā€” and this is the fact that catches everyone ā€” is change the kind of value that reaches your server. Every input value arrives at the server as a *string*, no matter the `type`. This is the other half of our spine: `type` carries the UX, the same way `name` carries the contract.

The SaaS-relevant types are lookup material ā€” you will return to this table rather than memorize it. Reach for the row that matches what the field *means*.

| `type` | Use it for | Worth knowing |
| --- | --- | --- |
| `text` | Names, free single-line text | The default. `maxLength`/`minLength` bound the length. |
| `email` | Email addresses | Email keyboard on mobile; a loose `@` shape check. |
| `password` | Secrets | Masks the input. Pairs with `autoComplete` (next section). |
| `number` | Quantities, amounts | Still arrives as a string (`"42"`). Use `min`/`max`/`step` to bound it ā€” `maxLength` is ignored here. |
| `tel` | Phone numbers | Telephone keypad on mobile. No format enforcement ā€” formats vary by country. |
| `url` | Links, websites | URL keyboard; a loose URL shape check. |
| `date` | Calendar dates | Native date picker. Value is an ISO string, `yyyy-mm-dd`. |
| `time` | Times of day | Native time picker. |
| `datetime-local` | A local date *and* time | Combined picker, no timezone. |
| `checkbox` | A boolean or a multi-pick option | Submits its `value` only when checked (see the next section ā€” this one bites). |
| `radio` | One choice from a set | Several radios share one `name`; only the checked one submits. |
| `file` | File uploads | Uncontrolled ā€” React can't set its value; read it from a ref. Needs `multipart/form-data`. |
| `hidden` | Carrying data the user doesn't edit | The classic use: an invoice `id` on an edit form, so the server knows which record to update. |

A note on `date`: the value is a plain ISO string like `"2026-06-01"`, and proper date handling on the JavaScript side has its own chapter much later ā€” for now, an ISO string is all you need to know it sends. And `type="submit"` does technically exist as an input, but prefer `<button type="submit">`: a button can hold rich content (an icon, a spinner) where an input submit can only hold a flat string label.

Two facts from that table earn their own emphasis because they actively trip people up.

First: **every input value is a string in `FormData`** ā€” yes, even `type="number"`. The user types `42`, and your server receives the string `"42"`, not the number `42`. The coercion happens on the server, where your Zod schema turns it into a real number (the tool for that is `z.coerce.number()`, and its chapter comes later). If you ever expect `type="number"` to hand JavaScript an actual number, this is the assumption to unlearn now.

Second: **`type` is UX, not a server guarantee.** `type="email"` nudges the mobile keyboard and runs a friendly browser check, but it does not *force* a valid email to arrive. A scripted client, or a malicious one, can POST your endpoint with anything at all and never touch your `<input>`. That is the bridge to the last section of this lesson ā€” but plant the seed now: the `type` is a courtesy to honest users, not a wall against dishonest ones.

## Beyond `<input>`: textareas, selects, and grouped controls

`<input>` is not the whole vocabulary. A few other controls round it out, and the through-line for every one of them is the same question: *what does this contribute to `FormData`?* That is where the contract gets subtle, so we will tie each element straight back to the key/value model.

**`<textarea>`** is multi-line text ā€” a note on an invoice, a description. It takes a `name` like any control and a `rows` attribute for its visible height. The one React-specific wrinkle: where plain HTML puts a `<textarea>`'s initial value between its tags, React reads it from a `defaultValue` prop instead. (`defaultValue` is the uncontrolled spelling ā€” the controlled version is a later chapter.)

```tsx
<textarea name="notes" rows={4} defaultValue="" />
```

**`<select>` and `<option>`** build a dropdown. The `name` goes on the `<select>`; what lands in `FormData` is the `value` of whichever `<option>` is chosen. Give every option an explicit `value` ā€” if you omit it, the option submits its visible text content instead, which couples your data to your copy. Acme's plan picker:

```tsx
<select name="plan" defaultValue="free">
  <option value="free">Free</option>
  <option value="pro">Pro</option>
  <option value="scale">Scale</option>
</select>
```

Add the `multiple` attribute and one `name` can yield several entries at once ā€” at which point the server reads them with `FormData.getAll(name)` rather than `.get(name)`. But the single-select dropdown above is the common case.

**The checkbox is the trap that catches everyone.** An `<input type="checkbox">` contributes its `value` to `FormData` **only when it is checked.** When it is unchecked, it contributes *nothing* ā€” not `"false"`, not the key with an empty value, but no entry at all. The key is simply absent from the submission. And if you don't set a `value`, a checked box submits the string `"on"`, which is rarely what you want ā€” so reach for an explicit `value`:

```tsx
<input type="checkbox" name="acceptedTerms" value="true" />
```

This checked-or-absent asymmetry is exactly why a checkbox can't be read naively as a boolean on the server ā€” "missing key" has to be understood as "false," which is a small coercion your Zod schema handles (its chapter comes later). Name the asymmetry now; the fix lives downstream.

**Radio groups** run on one rule, and it is the whole mechanic: several `<input type="radio">` controls that **share a single `name`** form one group, and only the checked one's `value` is submitted, once, under that shared `name`. This is the case promised earlier where `id` and `name` come apart ā€” every radio in the group needs its own unique `id` for its own label, but they all share the one `name` that defines the group.

```tsx
<input type="radio" id="billMonthly" name="billing" value="monthly" />
<label htmlFor="billMonthly">Monthly</label>
<input type="radio" id="billYearly" name="billing" value="yearly" />
<label htmlFor="billYearly">Yearly</label>
```

**`<fieldset>` and `<legend>`** group related controls into one logical unit ā€” the radio group above, or a billing-address block. The `<legend>`, which must be the first child, names the group, and a screen reader announces it before each control inside. When several controls answer *one* question, this is the element to reach for, not a `<div>`. (Its default border is stripped by the base CSS reset you will meet in the next chapter, so it won't look boxed-in by default.)

## `autoComplete`: the autofill contract

This section is short and pays off out of proportion to its length. Autofill is invisible right up until it misfires ā€” and a smooth autofill is something experienced engineers actually tune, because it directly lifts conversion on a sign-up form. If `name` is the contract with your server, `autoComplete` is the contract with the *autofill engine*: the attribute the browser and password managers read to offer the right saved value and to know what to store when the user enters a new one.

You set it to a semantic token that names what the field holds. The values you will reach for in a SaaS app: `email`, `username`, `current-password`, `new-password`, `given-name`, `family-name`, `name`, `organization`, `street-address`, `postal-code`, `country`, and `tel`. (There is also a credit-card family ā€” `cc-number`, `cc-name`, `cc-exp` ā€” worth recognizing when you build a billing form.)

The password pair is the one to understand rather than just copy. `autoComplete="current-password"` on a sign-in form tells the password manager "fill the *saved* password here." `autoComplete="new-password"` on a sign-up or change-password form tells it "this is a *new* secret ā€” offer to generate a strong one and save it." Same input type, opposite intent, and the token is how the browser tells them apart. Get this pair right and password managers do exactly the helpful thing; get it wrong and they fill a sign-up form with the user's existing password.

Which brings us to the anti-pattern. `autoComplete="off"` on a password field is a *regression*, not a security measure ā€” it fights the password manager your users depend on, nudging them toward weaker, reused, hand-typed passwords. Use the semantic token instead. The narrow legitimate use for `off` is a genuinely one-time, never-reused value ā€” a two-factor code field is the usual real example, because there is nothing to save and nothing to fill.

The following two tabs show the same email and password fields without and with the autofill tokens, so you can see exactly what changes ā€” which, fittingly for an invisible feature, is just a couple of attributes.

<CodeVariants>
  <CodeVariant label="Without autofill">
    ```tsx
    <label htmlFor="email">Email</label>
    <input id="email" name="email" type="email" />

    <label htmlFor="password">Password</label>
    <input id="password" name="password" type="password" />
    ```
    **The form works, but the browser is guessing.** With no `autoComplete`, it has only the field names to go on ā€” a password manager may fill the wrong field or fail to offer to save the new credentials.
  </CodeVariant>

  <CodeVariant label="With autofill">
    <div data-mark-color="green">

    <input id="email" name="email" type="email" autoComplete="email" />

    <label htmlFor="password">Password</label>
    <input id="password" name="password" type="password" autoComplete="new-password" />

    </div>
    **Two attributes, a markedly better sign-up.** `autoComplete="email"` lets the browser offer the user's saved address; `autoComplete="new-password"` tells the password manager this is a sign-up, so it offers to *generate and save* a strong password rather than fill an old one.
  </CodeVariant>

## Native constraints are UX, not security

Everything so far has been building toward one boundary, and it is the most important judgment in the lesson. HTML gives you a set of validation attributes that run *in the browser*, before the form is sent. They are genuinely useful ā€” they catch honest mistakes early and save the user a pointless round-trip. But they are not, and can never be, a security boundary. Here is the principle, and you should be able to recite it: **native constraints make the form pleasant; the server-side Zod schema makes it safe.** Conflating the two is the classic junior security hole.

The constraints themselves are cheap to add (their depth is a later chapter):

- **`required`** ā€” blocks the native submit and shows a localized browser message if the field is empty.
- **`type`-based checks** ā€” `email`, `url`, and `number` enforce a loose shape for free.
- **`min` / `max`** ā€” bound a number or a date to a range.
- **`minLength` / `maxLength`** ā€” bound a string's length. (Remember `maxLength` is ignored on `type="number"` ā€” use `min`/`max` there.)
- **`pattern`** ā€” a regex the value must match. Worth recognizing, but rare to reach for in 2026: the same rule reads far more legibly as a Zod schema on the server, and the client check is only a courtesy anyway.

Now the sentence the whole lesson has been pointed at, stated plainly: **client-side constraints can always be bypassed.** Devtools can delete the `required` attribute. A script can POST your endpoint and never load the page. A browser with JavaScript disabled skips the checks. A `curl` command doesn't even have a form. So your `required`, your `type="email"`, your `pattern` ā€” every one of them is a hint to honest users, and *none* of them defends your system. Every constraint you put on the markup is mirrored by a rule in your server-side Zod schema, and the Zod rule is the one that actually holds the line. The browser check exists to be fast and friendly; the server check exists to be true. This is the experienced engineer's reflex in one move: validate on the client for the user, validate on the server for the system, and never trust the client.

Make the geometry of that concrete in the following diagram. There are two paths to your server. The honest path goes through the browser's constraint check first, then the server's Zod check. The hostile path skips the browser entirely. Notice that both paths still hit Zod ā€” that is the point.

<Aside type="caution">
`disabled` and `readOnly` look similar and submit differently. A `disabled` input is non-interactive **and dropped from the submission** ā€” its value never reaches the server, which surprises people. A `readOnly` input is also non-interactive but **is** submitted. So when you want to show a value the user can't edit *and still send it* ā€” a pre-filled email on a confirmation step, say ā€” reach for `readOnly`, not `disabled`.
</Aside>

## Putting the contract together

Here is the whole Acme sign-up form, assembled ā€” every field labeled and named in `camelCase`, typed by meaning, autocompleted, and lightly constrained, with the submit button that fires the whole thing. Read it as the contract made complete: each `name` is a key your server will read, each `<label>` is a promise to the user, each `type` shapes the input, and each constraint is a courtesy the server will re-check anyway.

<AnnotatedCode lang="tsx" maxLines={18} code={`
<form action={createAccount} method="post">
  <label htmlFor="email">Email</label>
  <input id="email" name="email" type="email" autoComplete="email" required />

  <label htmlFor="password">Password</label>
  <input
    id="password"
    name="password"
    type="password"
    autoComplete="new-password"
    minLength={8}
    required
  />

  <label htmlFor="organizationName">Organization name</label>
  <input id="organizationName" name="organizationName" type="text" required />

  <label htmlFor="plan">Plan</label>
  <select id="plan" name="plan" defaultValue="free">
    <option value="free">Free</option>
    <option value="pro">Pro</option>
    <option value="scale">Scale</option>
  </select>

  <label>
    <input type="checkbox" name="rememberMe" value="true" />
    Remember me
  </label>

  <label>
    <input type="checkbox" name="acceptedTerms" value="true" required />
    I accept the terms
  </label>

  <button type="submit">Create account</button>
</form>
`}>
  <AnnotatedStep meta="{1}" color="blue">
    The container and the submit path. On submit, the contract is sent to `createAccount` as a `post` ā€” everything inside the `<form>` rides along.
  </AnnotatedStep>

  <AnnotatedStep meta={`/name="[\\w]+"/`} color="green">
    The keys. This set of `name` strings is the exact list your server's Zod schema expects ā€” the form and the schema are one vocabulary, written together.
  </AnnotatedStep>

  <AnnotatedStep meta={`/(?:htmlFor|id)="[\\w]+"/`} color="blue">
    Every input is labeled, and the explicit `htmlFor`/`id` link survives refactors. (The "remember me" and terms checkboxes use the implicit form ā€” the `<input>` nested inside the `<label>`.)
  </AnnotatedStep>

  <AnnotatedStep meta={`/(?:type|autoComplete)="[\\w-]+"/`} color="orange">
    The UX layer ā€” `type` picks the keyboard and native picker, `autoComplete` drives autofill. The `new-password` token tells the password manager this is a sign-up, so it offers to generate a strong secret.
  </AnnotatedStep>

  <AnnotatedStep meta={`"required" "minLength={8}"`} color="violet">
    Cheap native constraints that catch honest mistakes before a round-trip. Every one is mirrored by a rule in the server-side Zod schema ā€” and the Zod rule is the one that actually defends the system.
  </AnnotatedStep>

Step back and the spine is the whole lesson: the `name` attributes are the keys, the browser serializes the named controls into `FormData`, the server validates that `FormData` with a Zod schema whose keys are the same strings, and native constraints are a courtesy to the user, never a guarantee to the system. Write the form and the schema as one design and the round-trip almost wires itself.

What you have not built yet is the wiring, and that is deliberate. Connecting `action` to a Server Action and reading the `FormData` on the server is a later chapter; authoring the Zod schema that validates it is another; and the React hooks that layer richer pending states, inline errors, and optimistic updates on top of this native baseline come after that. You now have the surface those chapters wire onto ā€” and the judgment to know that the markup you wrote here is the contract, and everything else is enhancement.
