Picture one screen from Acme, the invoicing app you've been building markup for across this chapter: an audit log. Each row records something that happened to an invoice. At the end of every row sits a small trash icon to delete it. Above the log, a button toggles whether the rows are sorted newest-first or oldest-first.

Now look at four ordinary-seeming pieces of that screen, because each one hides a decision.

The delete button at the end of a row is a single icon — no words. A click on it has to know *which* invoice it belongs to. The sort toggle has a direction, and right now the only thing that knows the direction is the arrow you can see. The audit log is a grid of rows and columns. And a screen reader user, who never sees the trash glyph, still needs to be told that button deletes an invoice.

Four pieces, and the thing they have in common is the thread that runs through this whole lesson: each one is markup you write for a reader who *isn't* the person looking at the screen. Sometimes that reader is your own JavaScript. Sometimes it's a screen reader. Sometimes it's the browser's layout and accessibility engine. The chapter has been training one reflex — every element, prop, and attribute is a decision read by someone you'll never watch use it — and this lesson makes the *reader* the thing you sort by.

So the organizing question, the one to keep asking as we go, is simply: **who reads this attribute?** There are three answers, and three tools.

## Custom attributes your scripts read: `data-*`

Start with the easy one. `data-*` is the most mechanical surface in this lesson, and once it's in place it sets up the audit-log table at the end.

### The model

Any attribute whose name starts with `data-` is valid HTML. The browser will happily store it on the element and hand it back to you, but it will never *render* it, never style off it by default, and — this is the whole point — never assign it any meaning. A `data-invoice-id` means whatever your code decides it means and nothing more.

That's the deal: `data-*` is a private channel from your markup to your own code. The browser stays out of it because the channel is *yours*. Compare that to every other attribute you've met — `href`, `type`, `scope`, `name` — where the browser has opinions and behavior baked in. `data-*` is the deliberate opposite: a place to stash structured information on an element with a guarantee that no one but you is listening.

### Reading it back

You met the reader for this in chapter 14, when you walked the DOM: every element exposes a <Term definition="The DOM property that exposes an element's data-* attributes as an object. element.dataset.invoiceId reads the data-invoice-id attribute.">`dataset`</Term> object holding its `data-*` attributes. You don't need to re-learn the DOM traversal; you just need the one translation rule that trips everyone up the first time.

The attribute is written in kebab-case in your markup. The `dataset` reads it back in camelCase. `data-invoice-id` becomes `dataset.invoiceId`. The DOM does that conversion for you, automatically, in both directions — you never write the camelCase form in your markup, and you never write the kebab form in `dataset`.

And in JSX specifically, `data-*` is one of the two attribute families that pass straight through unchanged. Everywhere else JSX renames things — `class` became `className`, `for` became `htmlFor` — but `data-*` and `aria-*` keep their kebab-case spelling exactly as you'd write them in plain HTML. So the JSX you write and the attribute the browser receives are identical here.

The clearest way to see the round trip is to put the JSX and the reader side by side on one element. Hover the two highlighted attributes below.

<CodeTooltips tooltips={{
  'data-invoice-id': 'Read back in JS as element.dataset.invoiceId. Kebab in the markup, camelCase in dataset — the DOM converts it for you.',
  'data-status': 'Read back as element.dataset.status. This is also the hook a Tailwind data-[status=paid]:… variant styles off — same attribute, two readers.',
}}>
```tsx
<li data-invoice-id={invoice.id} data-status={invoice.status}>
  {invoice.summary}
</li>
```
</CodeTooltips>

One more thing about what comes back: it's always a string. The DOM stores attributes as text, so `data-count={5}` round-trips as `dataset.count === "5"`, not the number `5`. If you wrote `<div data-count={5}>` you'd read `"5"` back and have to parse it yourself. That's the same numbers-are-strings rule you saw for attributes generally — `data-*` is no exception.

### Where you'll actually reach for it

The model is simple; the value is knowing *when* a senior reaches for `data-*` instead of inventing something. Here's the daily tour. You don't need the depth on any of these yet — each one is owned by a later chapter — but seeing the set is what makes `data-*` click as a tool rather than a curiosity.

The first and most important use is **event delegation**, which you met in chapter 14. Instead of attaching a click handler to every row in a list — fifty rows, fifty listeners — you attach *one* listener to the container and let the click bubble up to it. When it fires, you ask: which element did this actually start on, and what should happen? `data-*` carries the answer.

<div data-mark-color="blue">

```tsx "[data-action]" "[data-invoice-id]" "'data-invoice-id'" "'data-action'"
function handleListClick(event: MouseEvent) {
  const button = (event.target as HTMLElement).closest('[data-action]');
  if (!button) return;

  const row = button.closest('[data-invoice-id]');
  const invoiceId = row?.getAttribute('data-invoice-id');

  if (button.getAttribute('data-action') === 'delete' && invoiceId) {
    deleteInvoice(invoiceId);
  }
}
```

</div>

One listener reads `closest('[data-action]')` to find what was clicked, walks up to the row's `data-invoice-id` to find which record, and acts. The `data-*` attributes *are* the routing table. This is the canonical reason you'll see `data-*` on list rows in real SaaS code.

The rest of the tour is quicker:

- **Test selectors.** Playwright and other end-to-end tools can find an element by a `data-testid` you put there for exactly that purpose. One caveat worth installing now, because it's the most common over-reach: a `data-testid` is the *fallback*, not the default. A good test first tries to find things the way a user would — "the button labelled Delete invoice", "the field labelled Email" — and only reaches for a `data-testid` when there's no accessible handle to grab. You'll see why when you reach testing; for now, just don't sprinkle `data-testid` on everything.
- **Analytics hooks.** A single delegated listener can read a `data-event-name` off whatever was clicked and fire a tracking event, instead of wiring analytics calls into every handler by hand.
- **Tailwind state variants.** This is the one to *recognize* rather than reach for. Tailwind can style an element conditionally off a `data-*` attribute — `data-[state=open]:rotate-180` rotates a chevron when its element carries `data-state="open"`. When you start using shadcn and Radix components, you'll notice `data-state` attributes all over their output. That's why: they're driving the styling. The element changes a `data-state` attribute, and the CSS reacts to it.

### The reflexes

Two lines to carry forward. First, `data-*` carries hooks *for code* — an id, an action name, a state flag your scripts or styles read. It never carries text the user is meant to see (that's a text node) and it never carries meaning for a screen reader (that's the next section's job — and a frequent, costly mix-up). Second, the `data-` prefix is mandatory. An invented bare attribute like `<div rowid="42">` is not valid HTML and won't show up in `dataset` at all, so the round trip silently breaks. The prefix is the price of the private channel.

:::note
You'll meet `data-state` again, properly, when you reach Tailwind's state variants and the shadcn/Radix component primitives. Here you only need to recognize it: it's a `data-*` attribute a component sets so the styling can react to it.
:::

## Signals for assistive tech: `aria-*`

Now the consumer changes, and so does the kind of thinking the tool demands. `data-*` was mechanical — learn the prefix, learn the camelCase rule, done. <Term definition="Accessible Rich Internet Applications. A set of HTML attributes that describe an element's role, name, and state to assistive technology.">ARIA</Term> is not mechanical. The attribute names are easy. Knowing *when not to use them* is the entire skill.

### The first rule: reach for the element first

Here is the rule every experienced developer has internalized, stated as bluntly as it deserves: **no ARIA is better than bad ARIA.** A wrong ARIA attribute doesn't fail loudly the way a syntax error does — it quietly lies to a screen reader, and the only person who finds out is the user you were trying to help.

To see why the rule holds, recall what you learned about the <Term definition="The structured representation the browser builds from your HTML — a tree of roles, names, and states that assistive tech reads instead of the visual pixels.">accessibility tree</Term> a few lessons back. The browser builds it from your markup, and semantic elements populate it *for free*. A `<button>` enters the tree already announced as a button. A `<nav>` enters as a navigation landmark. An `<input>` wired to a `<label>` enters as a named field. You didn't add a single ARIA attribute to get any of that — the element did the work.

So what is ARIA *for*? Two things. It fills gaps the native element genuinely can't express — a state, a name, a relationship the HTML has no built-in way to convey. And it can *override* an element's defaults — which is exactly where it goes wrong, because overriding a correct default replaces something true with something you now have to keep true by hand.

That gives you the order to ask questions in, and you'll apply it for the rest of this section:

1. Is there a semantic element that already conveys this? Use it. You're done.
2. If not, is there an `aria-*` attribute that adds the missing name, state, or relationship? Add the minimum.
3. Changing what an element fundamentally *is* with a `role` is the last resort, and in 2026 it's rarely the right call — you almost always have a better element to start from.

Every beginner ARIA disaster traces back to skipping step 1 — slapping `role` and `aria-*` onto a `<div>` to rebuild a button that `<button>` would have handed you complete. Install the order before the attributes, and you've learned the hard part.

This section is the *daily-reach subset* of ARIA — the handful you'll touch most weeks. The full surface (every role, every state, the authoring practices for complex widgets) gets its own treatment later in the course. When you need the complete catalog, that's where it lives. Here, breadth would be padding.
:::

### The attributes you'll reach for most weeks

Grouped by the *job* they do, because that's how you'll recall them — you won't think "I need `aria-label`", you'll think "this control has no visible text". Each comes with the place on the Acme screen where it earns its keep.

**Naming a control that has no visible text.** This is the home of that deferred icon-only button. Your trash button in the audit log is just an icon — visually obvious, but a screen reader has nothing to announce, because there's no text inside it. Give it a name:

```tsx
<button aria-label="Delete invoice" onClick={() => deleteInvoice(invoice.id)}>
  <TrashIcon />
</button>
```

`aria-label` takes a literal string and supplies the accessible name. The watch-out lives right here: `aria-label` *overrides* any visible text. On a button that already reads "Save", adding `aria-label="Submit"` means the screen reader says "Submit" while the eye reads "Save" — two sources of truth, drifting apart. So the rule is narrow: only label the *text-less*. If there's visible text, the element is already named.

When the name you want already exists *as another element on the page*, point at it by `id` instead of retyping it:

```tsx
<section aria-labelledby="invoices-heading">
  <h2 id="invoices-heading">Invoices</h2>
  {/* … */}
</section>
```

`aria-labelledby` names an element *by reference* — the section borrows its name from the heading it already shows. One source of truth: change the heading text and the accessible name follows. When a visible label exists, prefer this over `aria-label`.

**Pointing one element at another for description.** Where `aria-labelledby` supplies the *name*, `aria-describedby` attaches *secondary* text — the longer explanation, the hint, the error message:

```tsx
<input aria-describedby="email-hint" />
<p id="email-hint">We'll only use this for billing receipts.</p>
```

The canonical use is a form field pointing at its error message, and you'll wire that fully — alongside `aria-invalid` and the alert role — when you reach forms and validation in depth. (Remember the sign-in form a couple of lessons back deliberately rendered no error feedback yet; this is the piece that was being held back.) For now: `aria-describedby` is how a field says "this other element describes me."

**Telling the tree about a state the eye can already see.** Visual state — which nav item is active, whether a panel is open, whether a toggle is pressed — is invisible to a screen reader unless you say so.

- `aria-current="page"` marks the one nav link that points at the page you're already on — the highlighted item in the Acme sidebar. (It takes other values too — `step`, `date`, or a bare `true` — for wizards, calendars, and the like.)
- `aria-expanded` plus `aria-controls` belong on a <Term definition="The accessibility name for a control that shows and hides a section of content — an accordion header, a 'more details' toggle, a menu button.">disclosure</Term> — a control that shows and hides a panel. `aria-expanded="true|false"` announces the open/closed state; `aria-controls="panel-id"` names the panel it governs.
- `aria-pressed="true|false"` is for a toggle button — exactly the sort-direction toggle from the top of this lesson, or a bold button, or a favorite star. It tells the tree the button is a two-state switch and which state it's in. (A checkbox or a link is a different thing — use those elements when you mean them; `aria-pressed` is specifically "this *button* is currently on or off".)

```tsx
<button aria-pressed={isDescending} onClick={toggleSort}>
  Sort
</button>
```

**Hiding the decorative.** Sometimes the most accessible thing you can do is remove something from the tree. `aria-hidden="true"` does that. The textbook case is a decorative icon sitting *next to* visible text — a chevron beside the word "Filters". The eye wants the glyph; the screen reader would announce it as noise on top of the word. Hide the icon, keep the text:

```tsx
<button aria-pressed={isDescending} onClick={toggleSort}>
  Sort
  <ChevronIcon aria-hidden="true" />
</button>
```

Now the two cases for an icon line up cleanly. Icon *with* text? `aria-hidden` the icon — the text is already the name. Icon *alone*? `aria-label` the button — the icon can't speak for itself.

The load-bearing watch-out: **never put `aria-hidden="true"` on a focusable or interactive element.** Doing so creates an element you can still reach by pressing Tab but that has been erased from the tree the screen reader navigates — a control a keyboard user lands on and the screen reader refuses to announce. It's a trap, and it ships as a real bug more often than almost anything else in ARIA.

**Changing what an element *is*.** The `role` attribute overrides an element's built-in identity. In 2026 you reach for it rarely, because step 1 of the rule almost always hands you a better starting element. The one family of roles that genuinely earns its place is for live regions, next. (You might also see `role="dialog"` in the wild — and even that loses to the native `<dialog>` element and to Radix's primitives, which you'll meet later.)

### Announcing things the user didn't trigger: live regions

One more idea, named here so you recognize it, with the depth deferred. Everything above is about *static* markup — what's on the page when it loads. But a SaaS app constantly produces content *after* load that no click of the user's created: a "Saved" toast slides in, an inline validation message appears, a search result count updates. To a sighted user these are obvious. To a screen reader, by default, they're silent — the user's focus is elsewhere and nothing told the tree to speak.

A <Term definition="A region of the page marked so that assistive tech announces changes to its content automatically, without the user moving focus to it.">live region</Term> is the fix. You mark a region to announce its own changes:

- `aria-live="polite"` — announce when the user is idle, don't interrupt. Toasts and status messages.
- `aria-live="assertive"` — interrupt immediately. Reserve it for genuinely critical messages.
- `role="alert"` is the experienced developer's shorthand for an assertive live region, and it's the standard reach for an inline form error.

The one watch-out to carry: the region has to already exist in the DOM *before* the content drops into it. A live region the browser has been watching announces the change; a region that appears at the same moment as its content gives the browser nothing to compare against, and the announcement is missed. You'll wire live regions properly later in the course — for now, recognize the term and the trap.

You've now met both attribute families written for a reader who isn't the sighted user — `data-*` for your scripts, `aria-*` for assistive tech. ## When the data is genuinely tabular: `<table>`

The third reader is the browser's own layout-and-accessibility engine, and the third tool is the `<table>` element family. This is also where the chapter's threads converge — keys, `data-*`, and ARIA all show up in one artifact. But before any tag, there's a decision, and getting the decision right is most of the value.

### Is this actually a table?

`<table>` has two failure modes, and they pull in opposite directions. The old one — fading but not gone — is using a table to *lay out a page*, because decades ago that was the only way to get columns. The modern one is the overcorrection: building real tabular data out of a pile of `<div>`s because tables felt old-fashioned, and losing every bit of structure the element would have given the accessibility tree.

So you need a positive test, not a vibe. The test for <Term definition="Data shaped as a grid of like records: every row is the same kind of thing, every column the same attribute, and each cell is addressed by its (row, column) position.">tabular data</Term> is: **rows and columns of related records, where every position is indexed by (row, column).** Every row is the same *kind* of thing. Every column is the same *attribute* measured across all the rows. If that's true, it's a table.

Here's the sharp version of the test, the one to keep — the <Term definition="Swapping a grid's rows and columns: attributes run down the side instead of across the top, and vice versa.">transpose</Term> check: would *swapping the rows and columns* — putting attributes down the side and records across the top — produce a different but still *meaningful* view of the same data? For a genuine table, yes: a billing breakdown reads sensibly either way. For a page layout or a list of cards, transposing it is nonsense. If the transpose is meaningful, you have tabular data.

On the Acme surface, the clear *yes* cases are the ones you'd expect: the audit log, an invoice's line items, a billing breakdown, a metrics grid — all grids of like records. The *no* cases each have a better element, and naming them is the fastest way to keep `<table>` in its lane:

- Laying out a page or a section? That's CSS grid, which you'll meet soon. (This is the 1990s table-for-layout reflex; bury it.)
- A list of cards — one product, one teammate, one notification per tile? That's a `<ul>` and a grid, the list elements from earlier in the chapter.
- A form's two-column field arrangement? That's a `<form>` plus CSS grid.

The line under all of it: a table is for *data you'd compare across rows*, never for *positioning boxes on the page*. Walk the decision once below — it's the order a senior actually asks the questions in.

### The canonical accessible invoice table

Once the decision says "table", there's one shape a 2026 SaaS reaches for, and it folds in every thread this chapter has been pulling. Learn the anatomy as a labeled structure rather than a spec dump:

- `<table>` is the container.
- `<caption>` is the table's name. It's announced first, before the data, so the screen reader user knows what they've landed in. It goes directly inside `<table>`, before everything else.
- `<thead>` holds the header row. Inside it, each column header is a `<th scope="col">`.
- `<tbody>` holds the data rows. In each row, the cell that *identifies* the row — the invoice number — is a `<th scope="row">`, and the rest are plain `<td>`.
- `<tfoot>` is the footer row. Rare in a dashboard, but common on an actual invoice — it's where the "Total" row lives.

(You may also run into multiple `<tbody>` sections and `<colgroup>`/`<col>` for grouping and styling columns. Recognize them; you won't need them today.)

The accessibility hinges almost entirely on one attribute: <Term definition="The th attribute that declares whether a header labels its column (scope=&quot;col&quot;) or its row (scope=&quot;row&quot;), wiring each data cell to the right header.">`scope`</Term>. It's load-bearing, not decorative. `scope="col"` and `scope="row"` are what wire each data cell to its headers, so a screen reader reads a cell as **"Amount: $200"** instead of a bare "$200" with no idea which column it came from. Without `scope`, the table is a grid of disconnected numbers to anyone not looking at it. With it — and with the `<caption>` for the name — the screen reader announces the table on entry ("table, four columns, fifty rows"), names it, and ties every value to its header.

Now watch the chapter's threads land in the same element. The walkthrough below builds the complete table from a `.map`; each step calls out where an earlier idea pays off.

<AnnotatedCode lang="tsx" maxLines={18} code={`
<table>
  <caption>Recent invoices</caption>
  <thead>
    <tr>
      <th scope="col">Invoice</th>
      <th scope="col">Client</th>
      <th scope="col">Status</th>
      <th scope="col" className="text-right">Amount</th>
    </tr>
  </thead>
  <tbody>
    {invoices.map((invoice) => (
      <tr key={invoice.id} data-invoice-id={invoice.id}>
        <th scope="row">{invoice.number}</th>
        <td>{invoice.client}</td>
        <td>{invoice.status}</td>
        <td className="text-right">{invoice.amount ?? '—'}</td>
      </tr>
    ))}
  </tbody>
</table>
`}>
  <AnnotatedStep meta="{2-10}" color="blue">
    The `<caption>` is the table's accessible name, announced first so the user knows what they've landed in — and there's no `aria-label` on the table, because the caption already names it (element first). Each `<th scope="col">` in the `<thead>` labels its column.
  </AnnotatedStep>

  <AnnotatedStep meta={`{12-13} "key={invoice.id}"`} color="blue">
    The `.map` over the data. `key={invoice.id}` lets React's reconciler track each row by identity — the keys rule from the JSX lesson: stable, tied to the data, never the array index.
  </AnnotatedStep>

  <AnnotatedStep meta="{14}" color="blue">
    The invoice number is the cell that *identifies* the row, so it's a `<th scope="row">`, not a `<td>`. That's what lets a screen reader say "row INV-1029" as it reads across.
  </AnnotatedStep>

  <AnnotatedStep meta={`"data-invoice-id={invoice.id}" "text-right"`} color="green">
    `data-invoice-id` is this lesson's delegation hook — one listener on the table can find which invoice a click hit. The numeric Amount column gets `text-right` (one deliberate Tailwind utility; numbers read better right-aligned).   </AnnotatedStep>

  <AnnotatedStep meta={`"{invoice.amount ?? '—'}"`} color="blue">
    The empty-cell convention. When a value is missing, render an explicit em-dash rather than leaving the cell blank, so "no amount" reads as a deliberate value, not a hole the screen reader skips silently.
  </AnnotatedStep>

### Keeping tables usable on small screens

A table with six columns does not fit a phone, and this is exactly where beginners reach for the fix that quietly destroys everything you just built. So learn the safe reaches and the one forbidden move.

The safe default is to let the table scroll sideways. Wrap it in a horizontally scrollable container and the columns that don't fit slide into view on swipe:

```tsx
<div className="overflow-x-auto">
  <table>{/* … */}</table>
</div>
```

(That `overflow-x-auto` is one deliberate Tailwind utility — it's what makes the wrapper scroll.) For longer tables, the `<thead>` can stay pinned to the top while the body scrolls underneath with `sticky top-0`. And for genuinely complex tables, the more involved option is to switch to a card-per-row layout below a phone-sized breakpoint — one card per record instead of a cramped row — using Tailwind's responsive variants, which you'll meet later.

Here's the forbidden move, and the reason it's forbidden. To force a table into a vertical stack on mobile, it's tempting to set `display: block` on the `<tr>` and `<td>` elements. **Don't.** The moment a table row stops being laid out as a table row, the browser stops treating it as one in the accessibility tree — the "table, fifty rows" announcement vanishes, and so does every header-to-cell association `scope` bought you. You'd have made the table look fine for the eye and gone completely dark for the other reader. That's the whole spine of this lesson in one bug: the layout you give the eye must never blind the reader who can't see it.

:::caution
Never set `display: block` (or any non-table display) on `<tr>`, `<th>`, or `<td>` to force a mobile stack. It strips the table semantics from the accessibility tree. Scroll the table, pin the header, or swap to a card layout at a breakpoint instead.
:::

## Where this leaves you

Three surfaces, one question. When you reach for a custom attribute now, the first thing to ask isn't *what's it called* — it's *who reads it*. Your own scripts, and it's `data-*` with the kebab-to-camel `dataset` rule. Assistive tech, and it's `aria-*` — element first, the minimum patch second, and `role` almost never. The browser's layout-and-accessibility engine, and the question comes before any tag: is the data genuinely a grid of records? If it is, a `<caption>`, `<th scope>`, keys, and a `data-*` hook make it the shape a 2026 SaaS ships. The deeper ARIA surface, live regions in full, and form-error wiring all come back later in the course; you've installed the reflexes they build on.
