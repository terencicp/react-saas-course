Here is a `page.tsx` for an invoices dashboard. It is a Server Component, it is `async`, and its whole body is essentially one line:

```tsx title="app/invoices/page.tsx"
export default async function InvoicesPage() {
  const invoices = await db.invoices.find(); // data layer: Unit 5

  return <InvoiceTable invoices={invoices} />;
}
```

You already know how to write this. You wrote async Server Components that read data on the server in the chapter on the server/client boundary, and you wrapped slow reads in `<Suspense>` in the chapter on loading and streaming. So here is the question this lesson exists to answer, and notice that it is not a question about *how* to write the component — you can already do that. It is a question about *when and where* the component runs: does this page render once at **build time** and get served from a file, or does it render fresh on every **request**, or something in between? And whatever the answer is — what in the code decided it?

In Next.js 15 the honest answer was "it depends." It depended on a flowchart of implicit triggers you had to hold in your head, and the trigger that flipped the decision could be buried three components deep where the `page.tsx` gave no hint of it. Next.js 16 deletes that flowchart and collapses it to a single rule. By the end of this lesson you will be able to look at any route in your app and say what renders where, and why "dynamic by default" is the more honest default — not a performance smell, but the correct shape for most of what you will build.

## The old model: rendering inferred from your code

You will meet the Next.js 13–15 model in older codebases, in blog posts, and in every migration guide written before 2026, so it is worth one tight pass to recognize it — and then you can drop it, because you will never write it.

The default was the **inverse** of what you are about to learn. A route was **statically prerendered at build time** unless something tripped it into dynamic rendering. The framework rendered your page once during `next build`, stored the resulting HTML and data payload — this stored result had a name, the <Term definition="The Next.js 13–15 name for the build-time-rendered HTML and RSC payload of an entire route, stored once and served on every request until it was revalidated.">Full Route Cache</Term> — and served that same payload to every visitor until it was revalidated. Fast, but only correct for pages that look the same for everyone.

The problem was *how* a route escaped that default. The triggers were **implicit**. Reading `cookies()` or `headers()`, doing an uncached `fetch()`, reading `searchParams`, or adding `export const dynamic = 'force-dynamic'` to the file — any one of them silently flipped the **entire route** from static to dynamic. You did not write "this route is dynamic" anywhere. The framework inferred it from your API usage.

And here is where it hurt. The trigger did not have to live in your `page.tsx`. A component three levels down, written by someone else, could reach for `cookies()` to read a session — and that single call, invisible from the top of the tree, flipped the whole route dynamic. To answer "does this page render at build or at request time?" you could not read one file. You had to audit the entire subtree hunting for a dynamic API call, because the decision was real but it was *invisible*. A junior could not predict, from reading the page, what shipped where.

That is the shape of the problem: **implicit, route-wide, invisible.** Hold onto that shape, because Next.js 16 attacks all three.

## The Next.js 16 default: dynamic, with caching as opt-in

Here is the new rule, and it is the load-bearing idea of the whole lesson.

With <Term definition="The Next.js 16 config flag that makes every route dynamic by default, turns on the 'use cache' opt-in caching model, and enables Partial Prerendering.">`cacheComponents`</Term> turned on, **every route renders at request time by default.** That `InvoicesPage` from the top? It is dynamic. It can read `cookies()`, await `searchParams`, hit the database — without flipping any flag, because *there is no flag to flip.* It is already dynamic. The thing you spent the old model trying not to trip is now simply the floor you start from.

You turn the model on with one line in `next.config.ts`:

```ts title="next.config.ts"
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  cacheComponents: true,
};

export default nextConfig;
```

The inversion is worth saying in one breath, because it is the entire mental shift: **pre-16 you avoided *tripping* dynamic; post-16 you *add* caching.** Dynamic is the default you start from, and caching is something you build up from there, piece by piece, on purpose. You opt a piece of the page back into build-time rendering with a directive — it is called `'use cache'`, and it sits in the same family as the `'use client'` and `'use server'` directives you already know. That is genuinely all I want you to take from it right now: caching is **opt-in**, and the opt-in has a name. Its full anatomy — where you put it, what it caches, the rules it imposes — is the lesson after next.

:::note
**Next.js 16 under Cache Components: dynamic by default, caching by opt-in.** That single sentence is the model. Everything else in this chapter is a detail hanging off it.
:::

Two things to file away for when you are reading real code. First — and this might surprise you — you have *already* been running under this model. The course starter you have used since the App Router chapter ships with `cacheComponents: true` from day one. You have been writing dynamic-by-default routes this whole time; this lesson is just the first time it has a name. Second, if you see `experimental.dynamicIO` or `experimental.useCache` in an older `next.config.ts`, those were the experimental ancestors of this flag. They are gone in Next.js 16, folded into the single top-level `cacheComponents`. You do not need to learn them — just recognize them as the old spelling.

Now, the most important reframe in this lesson, and the one most likely to fight your instincts. A route with **no caching anywhere is fully dynamic, and that is the correct, common shape** for an authenticated SaaS surface. Your invoices dashboard, the settings page, anything scoped to a logged-in user or an organization — these *should* be fully dynamic. They show different data to every user; there is nothing meaningful to prerender. If you are carrying a reflex from the static-site era that says "dynamic = slow = bad," put it down. Here, dynamic is not a failure mode. It is the right answer most of the time. Caching is the targeted exception you reach for on the parts of the app that genuinely *are* the same for everyone — and we will get to exactly which parts those are.

## The mental model: a route is a tree of dynamic and cached subtrees

Let me give you the picture your model should crystallize around. You already reason about a route as a tree of components — that is the same tree from the server/client boundary chapter, the one you colored by *environment* (server vs. client). We are going to color the very same tree, but this time by *render-time disposition*: dynamic vs. cached.

Start simple. **Every node in the tree is dynamic by default.** That is the whole tree, one color, before you do anything.

Now you mark one node `'use cache'`. That node — *and all of its children* — become a single cached entry. The subtree renders once, its output is stored, and on later matching requests it is served straight from cache instead of re-rendering. So a cached node is not a lone cached component; it is a cached *region* of the tree, root and descendants together.

Which leads to the one hard rule of this model, the rule everything else follows from:

**Dynamic content cannot live inside a cached subtree.** If you mark a component `'use cache'` and somewhere inside it — or inside any of its children — something awaits request data, the framework rejects it. Not at runtime, not silently: it **fails the build** with a clear error pointing at the offending read. And the reason is almost obvious once you say it out loud: a cached result is computed once and reused; request data is different on every request. You cannot freeze "the current user's cookies" into a value that gets served to everyone. A cached subtree has to be **pure of request data**.

So what do you do when a page needs *both* — a cached part and a part that depends on the request? You do **not** try to make one component half-cached. You **lift the dynamic work out to a sibling** and give it its own boundary. Keep the cached subtree pure; put the request-dependent work next to it, not inside it. Split the responsibilities across siblings rather than tangling them in one node.

And at that sibling level, the page mixes completely freely. A cached header, next to a dynamic invoices table, next to a cached footer ad — three children of the same route, three different dispositions, one URL. The mixing is not the exception; it is how a real page is built.

## The Suspense boundary is the seam

You may have noticed the diagram drew the `<Suspense>` boundary as the line between the cached region and the dynamic sibling. That is not decoration. The Suspense boundary you learned in the chapter on loading and streaming is now doing a second job, and recognizing that it is the *same* primitive — not a new one — is what makes this whole model cheap to learn.

Here is the second job. When a route has a cached shell and a dynamic sibling, the `<Suspense>` boundary around the dynamic part is what lets the cached shell **ship to the browser immediately** while the dynamic hole **streams in** once its data resolves. Without that boundary, there is no seam — the whole route has to wait for the slowest dynamic read before anything reaches the user, and you have thrown away the entire benefit of caching the shell.

Stay with the reuse for a second, because it is the point. This is the exact same `<Suspense>` from before, riding the exact same streaming transport — the App Router flushing the shell first and resolved boundaries afterward over one HTTP response. Nothing new. The boundary that meant "show a fallback while this child loads" now *also* means "this is where the static part of the page ends and the dynamic part begins." One primitive, two readings, and you already know the primitive.

The full rendering shape — a cached static shell flushed instantly with dynamic holes streaming into it — has a name, <Term definition="A Next.js rendering mode where a cached static shell ships immediately and dynamic holes stream into it through Suspense boundaries. The default under Cache Components.">Partial Prerendering</Term>, and it is the very next lesson. I am naming it here only so the term is not a stranger when you meet it, and so you can see *why* the next lesson builds directly on a boundary you already understand. For now, the one thing to hold: the seam between cached and dynamic is a Suspense boundary you already know how to draw.

## Where dynamic comes from: the explicit signals

So a code path is dynamic by default — but the framework still has to *know* which paths actually touch the request, so it knows what it cannot prerender. Where does that signal come from now? This is the part that pays off the whole "explicit beats implicit" promise, so let me be precise.

Under Cache Components, a code path becomes dynamic by **awaiting a <Term definition="A value that only exists when a real request arrives: the URL's query string, the user's cookies, the request headers. Reading one forces dynamic rendering.">request-time API</Term>.** There is a small, closed set of them, and that is the entire list:

<DynamicSignalsInventory />

Look at what that list buys you. **The dynamic signal is always an `await` on one of those APIs, sitting visibly in the source.** There is no hidden third channel — no uncached `fetch` that silently flips the route, no deep child reaching for something the page can't see. If you want to find every dynamic dependency of a route, you search for these awaits. That is the legibility win made concrete: the question that used to require auditing an entire subtree is now answered by reading the code.

And the static analysis cuts the other way too, enforcing the purity rule from the tree section but stated now at the API level: a `'use cache'` function that tries to await any of these **fails the build.** Cached output and request data are incompatible, so the framework refuses to compile the combination. Same rule as before — request data cannot live in a cached subtree — now you can also see exactly *which lines* the framework is watching for.

<details>
<summary>Why each chip lands where it does</summary>

**Forces dynamic** — each one awaits a request-time API, so it can only be answered once a real request arrives:

- `await cookies()` reads the request's cookies.
- `await searchParams` reads the URL's query string, which only exists per request.
- `await headers()` reads the request headers.

**Can be cached** — none of these touch request data, so the same output is correct for every visitor:

- Rendering a static marketing header touches no data at all.
- Turning a Markdown string into HTML is a pure transformation — identical on every request.
- `await db.invoices.find()` is the one that feels like a trap. A database read touches **no request-time API**, so by this lesson's one rule it belongs in *Can be cached* — it is cacheable once you wrap it in `'use cache'`. Left alone in a dynamic-by-default page it simply runs fresh on every request; that is fine too. The deciding question is never "does it hit the network?" — it is "does it read request data?", and a bare query does not.

</details>

## The escape hatch: `connection()`

The signals above cover almost everything — but there is one gap, and it is worth naming because it is the chapter's "explicit beats implicit" thread taken to its logical end.

Some code *must* run at request time but has **no request-time API to give it away.** Think of generating a random ID, reading `Date.now()` for a freshness stamp, or calling a third-party SDK that lazily reads `process.env` the moment it is invoked. None of those await `cookies()` or `searchParams`, so the static analyzer has no way to infer "this has to be dynamic." Left alone, it might happily try to prerender that code at build time and bake in a build-time random number or timestamp — which is not what you meant.

For exactly these cases there is <Term definition="A Next.js function from next/server; awaiting it marks everything below as dynamic. For request-time code the framework cannot otherwise detect.">`connection()`</Term>, from `next/server`. Awaiting it is the explicit declaration "everything below this line is dynamic":

```ts
await connection();
```

That is the whole idea. It is the *manual* version of the signal the framework usually detects on its own — you reaching in to say "trust me, this is per-request" when the code gives no other clue. Code that runs after it and produces per-request values typically lives inside a `<Suspense>` boundary so it can stream as a hole, the same shape as every other dynamic sibling. The worked usage is a later lesson; here, just file `connection()` as the named escape hatch. The framing to keep is the senior one: where the old model would have caught this kind of code *by accident* through some incidental dynamic API, the new model makes you say it *on purpose* — and on-purpose is more legible than by-accident.

## Reading what shipped: the build log

One habit closes the loop on "who decides, and how do I confirm it." Do not assume what shipped where — read it.

Even though every route is dynamic by default, the build still runs a **prerender pass.** During `next build`, Next.js renders every `'use cache'` boundary it can resolve *without* request data and stores the result. Then at request time, the cached parts serve from cache (or from a CDN) while the dynamic parts run fresh. That build-time pass is precisely what makes the instant shell possible — the cached HTML already exists before the first user arrives. (The full mechanics are the next lesson; for now, just know the pass runs.)

Because the pass *runs* your cached components at build time, there is one consequence worth stating plainly now: a `'use cache'` component that throws during that pass **fails the whole build**, not just one request. A bug in cached, prerendered code is a build-breaker, which is good — you find out at deploy time, not in production.

And the payoff for the senior reader: the build log **labels each route's segments** — which are prerendered (static), which are dynamic, which stream. The move is to *read the log to confirm what actually shipped where*, rather than trusting your mental model of it. That is how you turn this whole abstract picture into something you can verify on your own project. The build output looks roughly like the following (the exact glyphs vary by Next.js version — read it for the *disposition per route*, not the symbols).

```bash frame="terminal"
Route (app)                              Size     First Load JS
┌ ○ /                                    1.2 kB         98.3 kB
├ ○ /pricing                             0.9 kB         97.1 kB
├ ◐ /dashboard                           3.4 kB        102.5 kB
└ ƒ /invoices                            2.1 kB        100.4 kB

○  (Static)    prerendered at build, served from cache
◐  (Partial)   cached shell ships instantly, dynamic holes stream in
ƒ  (Dynamic)   rendered fresh on every request
```
