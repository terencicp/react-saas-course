The invoices list page you built shipped fast. Fifty seeded rows, the page renders instantly, everyone moves on. Then a customer grows, the table fills to two hundred thousand invoices, and the same page takes three seconds to load.

You already know the candidate fixes. Earlier in this chapter you declared a composite index for exactly this query, and you ruled out N+1 — this is one query, not two hundred round-trips. But *how do you know* the index is the fix? And once you ship it, how do you confirm it actually worked, rather than sitting unused while the database keeps scanning the whole table?

"Feels slow" is not evidence. "Feels fast now" is not proof. There is exactly one acceptable answer to "why is this query slow, and did my change help" — you ask Postgres directly. By the end of this lesson you can take any suspect query, ask the planner how it ran, read the answer top to bottom, name the expensive part, and run the loop that turns a guess into a fix.

One rule frames everything that follows: **a query says *what* you want, the planner decides *how* to get it, and `EXPLAIN ANALYZE` shows you the *how* with real numbers attached.** Hold onto that sentence — every tool in this lesson is a different view of it.

## The planner decides how, EXPLAIN shows the decision

A SQL query is <Term definition="A query that states the result you want, not the steps to compute it. You describe the what; the database figures out the how.">declarative</Term>. When you write `select * from invoices where status = 'pending' order by created_at desc limit 20`, you have described the *result* — twenty pending invoices, newest first. You have said nothing about *how* to produce it. Should the engine read every row and discard the non-pending ones? Walk an index straight to the pending rows? Sort the whole table or use an index that already keeps it sorted?

That decision belongs to the <Term definition="The part of Postgres that turns a SQL query into an execution plan. It compares the available access paths and picks the one with the lowest estimated cost.">query planner</Term>. For any query, the planner enumerates the access paths available — scan the table, walk this index or that one, which algorithm to join with — estimates the cost of each, and picks the cheapest. This is the same machinery you met when you declared indexes: an index doesn't change *what* comes back, only *how the engine reaches it*. Two plans for the same query return byte-identical results at wildly different speeds.

`EXPLAIN` is how you read that decision. Prefix any query with it and Postgres prints the plan it *would* run — without running it:

```sql
explain select * from invoices where status = 'pending';
```

You get the chosen plan annotated with <Term definition="The planner's predicted cost of a plan node, in an arbitrary internal unit. Useful only for comparing nodes or plans against each other — never milliseconds.">cost estimates</Term>: the planner's guess at how expensive each step is. Fast to run, because the query never executes. But a guess is only half the story.

`EXPLAIN ANALYZE` is the other half. It **actually runs the query** and prints the same plan, now annotated with the real timings and row counts measured during execution, sitting right beside the estimates:

```sql
explain analyze select * from invoices where status = 'pending';
```

This is the one to reach for. The estimate tells you what the planner *thought*; the actuals tell you what *happened*; and the gap between them is, more often than you'd guess, the bug itself. A plan that estimates ten rows and actually returns a million made its decisions on bad information.

One number trips up everyone the first time. Those cost estimates look like `cost=0.00..25.40`, and the instinct is to read them as milliseconds. They are not. Cost is an **arbitrary planner unit** — useful only for comparing two nodes or two plans against each other, never as an absolute. The real time is a different field, `actual time=`, which `ANALYZE` adds and `EXPLAIN` alone does not. When you want milliseconds, you want `actual time`, which means you want `ANALYZE`.

## Running it: copy the SQL, prefix it, read the output

Getting a plan is the easy part — the skill is in reading it, which is the rest of this lesson. So keep the mechanics short.

The everyday path is a reflex you can already half-do. Turn on `logger: true` on your Drizzle client, run the page, and copy the exact SQL Drizzle logged. Paste it into a SQL console — Neon's SQL editor, `psql`, or Drizzle Studio — and prefix it with `explain (analyze, buffers)`. Drizzle builds the query; you diagnose the raw SQL it emits. That separation is the point: you don't need the ORM in the loop while you're reading plans.

You can also run a plan from inside your application code when you want a one-off probe in a script:

```ts
const plan = await db.execute(
  sql`explain (analyze, buffers) select * from invoices where status = 'pending'`,
);
```

Be honest about the friction. You're hand-writing the SQL (or interpolating it), and what comes back is rows of plan *text* — not a typed result shape you can work with. It's fine for a quick probe, but for interactive diagnosis the SQL console is cleaner. You'll read plans far more often than you'll script them.

Notice the `(analyze, buffers)` in both. <Term definition="An EXPLAIN option that reports how many disk pages each node read from RAM (hit) versus from disk (read). The best single signal for whether slowness is a cache miss.">BUFFERS</Term> adds, to every node, a count of how many pages came from RAM versus from disk — the single best signal for whether a query is slow because it's missing the cache. The catch is a version split. Postgres 18, your production target on Neon, turns `BUFFERS` on automatically whenever `ANALYZE` runs. Older Postgres — including the one running inside the exercises on this page — does not, so you have to ask for it explicitly. Rather than remember which is which, adopt one habit: **always write `(analyze, buffers)`.** It's a no-op where buffers are already on, and mandatory everywhere else. One portable reflex, correct on every version.

<Aside type="caution">
`EXPLAIN ANALYZE` *executes the query.* On a `SELECT` that's harmless. On an `UPDATE`, `DELETE`, or `INSERT` it **performs the write** — the rows really change. To get a plan for a destructive statement without committing it, wrap it in a transaction you roll back:

```sql
begin;
explain (analyze, buffers) delete from invoices where status = 'draft';
rollback;
```

Or drop `ANALYZE` and use plain `EXPLAIN` for an estimate-only plan that never touches the data. This is a real production footgun — `EXPLAIN ANALYZE DELETE` on the wrong window deletes for real.
</Aside>

## Reading the plan tree top to bottom

This is the skill the whole lesson hangs on, so slow down here.

A plan is a **tree, printed as indented text.** Each line is a node — an operation like "scan this table" or "join these two inputs" — followed by its estimates and, with `ANALYZE`, its actuals. <Term definition="A read of a plan that treats indentation as tree depth: deeper-indented nodes are children that run first, and their output feeds the less-indented parent above them.">Indentation is depth.</Term> A node indented under another is its *child*.

Here is the part beginners get backwards, every time, so read it twice: **the deepest, most-indented node runs *first*.** Its output feeds its parent one level up. Results flow *upward* until they reach the top line, the root — which is the *last* thing that happens and the result you actually receive. The natural instinct is to read the top line first and stop there. The top line is the finish line, not the start.

When you read a plan, this is the recipe:

<Steps>
1. Find the deepest node. That's where data first enters the query — a table scan or an index walk.

2. Walk outward and up, one level at a time, watching how `actual rows` changes at each node. Rows get filtered out, joined, sorted, aggregated as they climb.

3. Find the node where the time concentrates, or where the row count explodes or collapses unexpectedly. That node is your suspect.
</Steps>

## The four numbers that decide

A plan node carries a dozen fields. An experienced engineer reads four of them and skims the rest. Learn these against the position they occupy in a real node line and you'll never be lost in the output again.

<AnnotatedCode lang="text" code={`
Index Scan using idx_invoices_org_created on invoices
  (cost=0.42..18.30 rows=12 width=72) (actual time=0.015..0.040 rows=180 loops=200)
  Buffers: shared hit=120 read=8
`}>
  <AnnotatedStep meta={`"rows=12" "rows=180"`} color="blue">
    Estimated `rows` is the planner's guess from table statistics; `actual ... rows` is what really came back. When actual is off from estimated by more than ~10×, the planner chose its plan on bad information — stats are stale or the predicate is too hard to estimate. Fix: run `ANALYZE <table>;` and re-check. Here 12 vs 180 is a 15× miss — a red flag.
  </AnnotatedStep>

  <AnnotatedStep meta={`"loops=200"`} color="orange">
    This node executed 200 times. The displayed time and rows are *per loop*, so the real cost is `loops × time` and the real row total is `loops × rows`. A node that looks cheap at `0.040ms` is expensive at `loops=200` — about 8ms, 36,000 rows touched. The most-missed number on the page, and the database-side cousin of N+1.
  </AnnotatedStep>

  <AnnotatedStep meta={`"actual time=0.015..0.040"`} color="violet">
    Two numbers — startup time before the first row (`0.015`), then total time to the last row (`0.040`), *per loop*. Read it together with `loops=` to get the true cost; alone it understates a looped node.
  </AnnotatedStep>

  <AnnotatedStep meta={`"shared hit=120 read=8"`} color="green">
    120 pages served from RAM (`hit`), 8 fetched from disk (`read`). On a query that should be hot, a lot of `read` means the data doesn't fit in cache or the index isn't covering. And these counts are *per loop* too — multiply by `loops=` for the real I/O.
  </AnnotatedStep>
</AnnotatedCode>

Two of those four numbers deserve a closer look, because they hide the most bugs.

The **estimate-versus-actual gap** is the planner grading its own homework. It chose this plan because it predicted these row counts, drawn from <Term definition="Sampled summaries of each column's distribution that the planner uses to estimate row counts. Refreshed by ANALYZE and, on a schedule, by auto-vacuum.">statistics</Term> — a sampled picture of how your columns are distributed. When the prediction is wildly off, the picture is out of date. The fix is to refresh it: <Term definition="The SQL command ANALYZE <table>; refreshes the planner's statistics for that table. Not the same as the ANALYZE option inside EXPLAIN ANALYZE, which runs and times a query — a common name collision.">ANALYZE</Term> the table and run the plan again.

<Aside type="note">
The word `ANALYZE` does two unrelated jobs, which catches people out. `EXPLAIN ANALYZE` — the option — *runs and times* a query. The standalone `ANALYZE invoices;` command *refreshes statistics* for a table. Same word, different machinery. Auto-vacuum runs the statistics kind on a schedule for you; you reach for it by hand when a plan's estimates have drifted and you don't want to wait.
</Aside>

And `loops=` is the silent multiplier. A line reading `(actual time=0.01..0.04 rows=5 loops=200)` did not touch 5 rows in 0.04ms — it ran two hundred times, so the honest cost is 200 × 0.04ms and 200 × 5 rows. Postgres reports per-loop and trusts you to do the multiplication. Skip it and a node that looks free turns out to dominate the query.

## The node types you'll meet

You don't need the full catalogue of plan nodes — Postgres has dozens, and most never appear in a SaaS app. You need the handful that map to the query shapes you already write. Here they are, grouped by what they do, each tied to the query that produces it.

**Scans** are how rows first enter a plan.

- **`Seq Scan`** reads every row in the table, top to bottom. Perfectly fine on a small table. A smell on a large one when a `WHERE` clause should have let an index skip most of it.
- **`Index Scan`** walks a B-tree index to find matching keys, then fetches those rows from the table. This is the shape a selective predicate produces once the right index exists — the payoff of the indexes you declared earlier.
- **`Index Only Scan`** answers the whole query from the index alone, never touching the table, because the index already contains every column the query needs. That's a <Term definition="An index that contains every column a query reads, so the query can be answered from the index alone — no trip back to the table.">covering index</Term> (built with `INCLUDE` — named here, drilled later).
- **`Bitmap Index Scan`** paired with **`Bitmap Heap Scan`** combines one or more indexes through a bitmap before fetching rows — the planner's choice for medium-selectivity or multi-predicate queries, where a plain index scan would jump around the table too much.

**Joins** combine two inputs. The planner picks the algorithm; you read which one it picked.

- **`Nested Loop`** takes each row from the outer input and probes the inner input for matches. It wins when the outer set is small. It turns dangerous when the inner side is an unindexed scan run many times over — watch the inner node's `loops=`.
- **`Hash Join`** builds a hash table from one input, then probes it with the other. The workhorse for medium-to-large equality joins.
- **`Merge Join`** zips together two inputs that are already sorted on the join key.

**Post-processing** nodes shape rows after they're gathered.

- **`Sort`** orders rows. Watch for `Sort Method: external merge Disk: …` — that means the sort didn't fit in memory and spilled to disk, which is slow.
- **`Aggregate`**, **`HashAggregate`**, and **`GroupAggregate`** are the fold behind `count`, `sum`, and `group by`.
- **`CTE Scan`** and **`Subquery Scan`** read the results of a CTE or a subquery.

## From plan to diagnosis

Now the payoff. Reading a plan is a means to an end, and the end is recognition — seeing a pattern and knowing the fix. Each pattern below reads the same way: *what you see in the plan → what it means → the fix you already learned.*

- **A `Seq Scan` with a `Filter` on a large table, where `actual rows` is far smaller than the rows scanned.** The engine read the whole table and threw most of it away. The tell is a large <Term definition="A plan line reporting how many rows a node read and then discarded because they failed the WHERE filter. A large value next to a Seq Scan means the engine is reading rows just to throw them away.">`Rows Removed by Filter`</Term> line. The fix is the missing or unused index on the filter column.
- **An `Index Scan` returning far more rows than the final result needs.** The index is used but isn't selective enough, or it's the wrong composite for this query. A better composite — leading with the column the query filters on — may help.
- **A `Sort` consuming most of the query's time, especially `Sort Method: external merge`.** No index produces the order the query asks for, so Postgres sorts from scratch (and spills to disk if it's big). The fix is a composite index in the query's exact sort direction, so the rows arrive already ordered and the `Sort` node disappears entirely.
- **A `Nested Loop` with a high `loops=` count on its inner side, scanning an unindexed table.** This is the database-side shape that *looks* like N+1 — and here's the line you must not blur. Application N+1, from earlier in this chapter, is two hundred separate round-trips, invisible to any single plan, found only by counting queries in the log. This is *one* query whose join re-probes an unindexed table once per outer row. Because it's a single statement, the plan *does* show it. The fix is the foreign-key index on the join column.
- **Estimates off from actuals by orders of magnitude.** Stale statistics. Run `ANALYZE <table>;` and re-check before changing anything else.

<Aside type="note">
Keep the two N+1s straight. **Application N+1** is a *round-trip count* problem — N parameterized queries fired from your code — and `EXPLAIN ANALYZE` is the *wrong* tool for it, because each individual statement's plan looks perfectly healthy. You catch it with `logger: true` and a count of statements. The **database-side nested loop** with a high `loops=` is one statement with a bad join, and the plan shows it plainly. Same shape, different layer, different tool.
</Aside>

## One change, one re-run: the diagnostic loop

Everything in this lesson collapses into a single discipline, and it's worth naming on its own so it lands as a habit rather than a footnote.

<Steps>
1. **Measure.** Run `EXPLAIN (ANALYZE, BUFFERS)` on the slow query. Read it top to bottom.

2. **Hypothesize.** Name *which* node is expensive and *why* — "the `Sort` dominates because no index produces this order."

3. **Change exactly one thing.** Add one index, rewrite one clause, tighten one `WHERE`. One.

4. **Re-run and confirm.** Run `EXPLAIN (ANALYZE, BUFFERS)` again and check that *the node you predicted* changed the way you predicted. Then repeat if it's still slow.
</Steps>

The non-negotiable rule is the "exactly one" in step 3. **Change two things at once and you've destroyed the signal** — when the query gets faster, you can't say which change did it, and you may have shipped a useless second index that taxes every write to that table for no benefit. That discipline is the whole difference between a fix and cargo-cult tuning, where indexes get sprayed at a problem and nobody can later say which ones matter.

One trap will undermine the loop before you even start it: the data you measure against. A plan on a hundred dev rows tells you almost nothing about the plan on ten million production rows, because **the planner changes strategy as the table grows** — a `Seq Scan` that's cheapest at a hundred rows loses to an `Index Scan` at a hundred thousand. It's the same `Seq Scan`-to-`Index Scan` swap you just toggled in the exercise, except driven by volume rather than the index — and it's why you reach for production-shaped data: a realistically seeded volume, or a production-shaped branch on Neon. Diagnose against a toy table and you'll "fix" problems that don't exist and miss the ones that do.

## Worked example: the cursor-pagination query, before and after

Time to cash in the debt. Earlier in this chapter you declared a composite index for cursor pagination, and the lesson promised you'd be able to *prove* it earns its weight. This is that proof.

The query is the cursor-pagination one you wrote: a page of invoices for an organization, ordered newest-first, fetched with a compound cursor and a limit. In SQL — since `EXPLAIN` runs on SQL, not on the Drizzle builder — it comes out roughly like this:

```sql
select * from invoices
where organization_id = $1
  and (created_at < $2 or (created_at = $2 and id < $3))
order by created_at desc, id desc
limit 21;
```

The index you declared for it is `idx_invoices_org_created_at_id`, on `(organization_id, created_at desc, id desc)` — the tenant column first, then the sort key and its tiebreaker in the exact direction the query orders. The two tabs below are this query's plan before that index exists and after, against a populated table.

  <CodeVariant label="Before the index">
    <div data-mark-color="red">

    ```text "Seq Scan" "Sort  (actual time=88.2..88.2 rows=21 loops=1)" "read=2304"
    Limit  (actual time=88.2..88.3 rows=21 loops=1)
      ->  Sort  (actual time=88.2..88.2 rows=21 loops=1)
            Sort Key: created_at DESC, id DESC
            Sort Method: external merge  Disk: 9512kB
            ->  Seq Scan on invoices  (actual time=0.02..71.4 rows=41204 loops=1)
                  Filter: (organization_id = 1)
                  Rows Removed by Filter: 158796
                  Buffers: shared hit=180 read=2304
    Execution Time: 90.1 ms
    ```

    </div>
    **Two suspects, bottom-up.** The `Seq Scan` reads all 200,000 rows and throws away 158,796 — `Rows Removed by Filter` confirms it. Then the `Sort` orders 41,204 rows from scratch and spills to disk (`external merge`). 2,304 pages read from disk. 90ms.
  </CodeVariant>

  <CodeVariant label="After the index">
    <div data-mark-color="green">

    ```text "Index Scan using idx_invoices_org_created_at_id on invoices" "read=0"
    Limit  (actual time=0.05..0.34 rows=21 loops=1)
      ->  Index Scan using idx_invoices_org_created_at_id on invoices
            (actual time=0.04..0.31 rows=21 loops=1)
            Index Cond: (organization_id = 1)
            Filter: ((created_at < $2) OR ((created_at = $2) AND (id < $3)))
            Rows Removed by Filter: 3
            Buffers: shared hit=24 read=0
    Execution Time: 0.39 ms
    ```

    </div>
    **One node.** The `Index Scan` walks `idx_invoices_org_created_at_id` for `organization_id = 1`, and the `Limit` halts it after 21 rows — the index already holds them in `created_at desc, id desc` order, so the engine never reads the rest. The cursor's `OR` predicate can't narrow a B-tree, so it rides along as a `Filter` that skips the few rows newer than the cursor (`Rows Removed by Filter: 3`). **No `Sort` node at all.** Zero disk reads. 0.39ms — over two hundred times faster.
  </CodeVariant>
</CodeVariants>

Read the two plans side by side and two things jump out. The headline metric is the disk reads: `read=2304` collapsed to `read=0`. Before the index, Postgres dragged thousands of pages off disk to find one page of results; after, it walks straight to them in cache. The structural win is the missing node — the `Sort` is simply *gone*. The index keeps the rows in `created_at desc, id desc` order already, so there is nothing left to sort. That's the whole reason cursor pagination and its composite index belong together: the index doesn't just speed the lookup, it deletes an entire stage of work.

And notice what this was: the loop, run once. One change — the index. One re-run. The predicted node — the `Sort` — gone, exactly as predicted. That's the discipline, and that's the proof the earlier lessons deferred to this one.

## Beyond the console

You won't hand-run `EXPLAIN` on production all day. A few production-grade affordances do this for you, and it's worth recognizing them by name now so you reach for them later.

<Term definition="A Postgres extension that automatically logs the execution plan of any query slower than a configured threshold. The production version of running EXPLAIN by hand.">auto_explain</Term> is a Postgres extension that logs the plan of any query slower than a threshold you set — `auto_explain.log_min_duration = '500ms'`, and every slow query records its own plan to the log without you lifting a finger. It's the production version of this lesson's manual loop: instead of guessing which query is slow and running `EXPLAIN` on it, you let the slow ones announce themselves. It's a server-config concern, so it's deferred — but it's the answer to "how do I do this at scale."

One tier up, hosted tools like **pganalyze** and **Datadog database monitoring** collect plans *and* trend them over time, so you can see a query degrade across a deploy. And the consoles where you'll actually run the commands from this lesson are **Drizzle Studio** and the **Neon SQL editor** — both covered when you set them up.
