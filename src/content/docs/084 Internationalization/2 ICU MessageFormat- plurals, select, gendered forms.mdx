Last lesson left a loose thread. The inbox line read `You have {count} unread messages`, and `{count}` sat there as a plain named slot. Fine when the count is always plural — but the moment it can be `1`, the grammar breaks: "You have 1 unread messages" is wrong, and so is every count-shaped string in your app.

Here's the reach you'd write today, and it feels disciplined. The catalog gave you the sentence, so surely the singular-vs-plural choice is yours to make in code:

```tsx
// The catalog owns the sentence — but the plural is ours to handle, right?
const noun = count === 1 ? t('inbox.message') : t('inbox.messages');
return <p>{t('inbox.unread', { count, noun })}</p>;
```

This is the same mistake the last lesson killed for word order, walking back in dressed as a different problem. There you learned not to concatenate fragments because translators reorder sentences; here the ternary hard-codes *English's* idea of plural — one form for `1`, one for everything else — straight back into the component the rule just cleaned. English has two plural forms. Russian has four. Arabic has six. Welsh has six. Chinese has one. A ternary can't express any of that, and you can't write one ternary per language without turning your component into a grammar textbook.

The fix is the same shape as last lesson: push the logic off the JavaScript heap and into the catalog string, where per-language rules already live. The string carries the branching; the runtime picks the right branch for the current locale; your call collapses back to a single value:

<div data-mark-color="green">

```tsx "{ count }"
return <p>{t('inbox.unread', { count })}</p>;
```

</div>

That string lives in the catalog, and the syntax it speaks is <Term definition={"ICU MessageFormat.\nA Unicode-standardized syntax for putting locale-aware choice logic inside a single translation string."}>ICU MessageFormat</Term>. By the end of this lesson you'll read and author it for three kinds of choice — counts, ordinals, and free variants like gender — including the nested case where a notification needs both a person *and* a count. You'll also know the four ways these strings break, three of which fail silently.

## What ICU MessageFormat is

Last lesson, the three-party contract had a clean division: you hold the key, the catalog holds the text, the translator owns the catalog. The text was always a flat sentence with named slots. ICU MessageFormat keeps that contract exactly — same key, same translator — but gives the catalog value a richer vocabulary. The value stops being a flat sentence and becomes a tiny template the runtime interprets.

Here is the full shape for the inbox line:

```text
{count, plural, =0 {No unread messages} one {# unread message} other {# unread messages}}
```

When that string renders, the runtime does four things. It reads the variable — `count` — from the object you passed to `t()`. It looks up which plural category that number falls into *for the current locale*. It picks the matching branch. And it substitutes the `#` token with the locale-formatted number. None of that logic is in your component; all of it is in the string the translator can edit.

The "ICU" in the name is <Term definition={"International Components for Unicode.\nThe reference C and Java implementation whose message syntax the whole ecosystem adopted."}>International Components for Unicode</Term> — the reference implementation everyone copied. The rules it consults come from <Term definition={"Unicode Common Locale Data Repository.\nThe open dataset of per-language formatting and grammar rules that every Intl formatter and ICU implementation reads."}>CLDR</Term>, the dataset behind every locale-aware feature in the platform. `next-intl`, the library you met by name last lesson, ships an ICU MessageFormat parser, so you write these strings in the catalog and the library applies them.

ICU can express five kinds of choice. It's worth naming the whole set up front so you have the map, even though this lesson only teaches part of it:

- **`plural`** — for cardinal counts ("3 messages").
- **`selectordinal`** — for ordinals ("3rd place").
- **`select`** — for free string variants (gender, role, notification type).
- Inline **`number`** and **`date`** formatting — currencies, dates, percentages.

This lesson teaches the first three at depth. The last two — formatting a number or a date *inside* a message — exist, but the experienced reach is not to use them. When a string needs a formatted currency or a localized date, you make the formatter call explicitly at the seam and pass the result in, rather than burying a format skeleton inside the ICU string. That formatter family — `Intl.NumberFormat`, `Intl.DateTimeFormat`, and friends — is the whole subject of the next lesson. Name it, set it aside, and don't go looking for inline number syntax here.

## Anatomy of a plural message

The `plural` form does the most work, so dissect it once, carefully. Here is the inbox message again, broken into its parts.

<AnnotatedCode lang="text" code={`
{count, plural, =0 {No unread messages} one {# unread message} other {# unread messages}}
`}>
  <AnnotatedStep meta={`"count"`} color="blue">
    The **variable**. It names which value drives the choice, and it matches the key in the object you pass: `t('inbox.unread', { count })`. This is the one and only thread back to your component — change the count, change the branch.
  </AnnotatedStep>

  <AnnotatedStep meta={`"plural"`} color="violet">
    The **selector keyword**. `plural` means *treat this value as a cardinal number and pick a branch using the locale's plural categories*. There are two other keywords — `selectordinal` and `select` — coming later; `plural` is the one you'll reach for most.
  </AnnotatedStep>

  <AnnotatedStep meta={`"=0 {No unread messages} one {# unread message} other {# unread messages}"`} color="blue">
    The **branch table**: a list of `selector {message}` arms. Two kinds of selector appear here — exact matches like `=0`, tried first, and CLDR keywords like `one` and `other`, tried after. `other` is **mandatory**; it's the catch-all when nothing else matches.
  </AnnotatedStep>

  <AnnotatedStep meta={`"#"`} color="orange">
    The **`#` token**. Inside a branch, `#` is the count rendered as a locale-aware number — with grouping and decimals applied. Note it's `#`, not `{count}`. That distinction is the single most common authoring bug, and it's the whole next section.
  </AnnotatedStep>
</AnnotatedCode>

Step back and notice what your component looks like next to all of that: nothing changed. The call is still `t('inbox.unread', { count })`. Every decision about *how* the string varies with the count lives in the catalog, exactly where the last lesson put decisions about *how* the string varies with word order. The component passes data; the catalog holds grammar.

## `#` is the formatted count, `{count}` is a bug

That `#`-versus-`{count}` distinction deserves its own beat, because getting it wrong fails silently and survives code review.

Inside a plural branch, `#` is a special token: it stands for the selector value — `count` — formatted as a locale-aware number. You don't name the variable again; `#` already knows it's the count, and it knows the locale, so `12345` renders as `12,345` in `en-US` and `12 345` in `fr-FR` for free.

Write `{count}` instead and two things go wrong at once. `{count}` re-interpolates the raw variable as an ordinary named placeholder — so you get the number, but unformatted, with no grouping. And because you've left a `{count}` where the `#` was meant to go, the natural mistake is to end up with the variable printed *twice*: a branch reading `{count} {count} messages` renders "5 5 messages." The output is wrong, but it's not an error — nothing throws, the page renders, and a reviewer skimming the catalog sees a plausible-looking string. The only defense is knowing the rule.

Predict what the following program prints. One message uses `#` correctly; a sibling message makes the `{count}` mistake.

<PredictOutput expected={`1,234 unread messages
1234 1234 unread messages`}>

The locale is `en-US`. `format(message, values)` runs an ICU message the way `t()` would.

```ts
const count = 1234;

// Correct: `#` renders the formatted count.
const right = format('{count, plural, other {# unread messages}}', { count });

// Bug: `{count}` re-prints the raw variable, and it's doubled.
const wrong = format('{count, plural, other {{count} {count} unread messages}}', { count });

console.log(right);
console.log(wrong);
```

## One language has two forms, most don't

Look back at the inbox message and you'll see it carries three branches: `=0`, `one`, `other`. The `one` and `other` are not arbitrary names — they're <Term definition={"A named bucket CLDR assigns to numbers in a language: one of zero, one, two, few, many, other. Which buckets exist, and which numbers fall into each, is decided per language."}>plural categories</Term>, and English happens to use exactly two of them. That two-form world feels universal because it's the only one most of us grew up with. It isn't. CLDR declares, per language, which categories exist and which numbers fall into each — and the variation is dramatic.

The runtime handles all of it. You write one message shape for your *source* language; the translator supplies the branch set *their* language needs; CLDR decides which branch a given number lands in. You never enumerate languages or write a rule. The following tabs show the same inbox message across four languages — watch the branch set change while your call stays identical.

The point to hold onto: plural categories are a property of the *language*, not a universal you can hard-code. And across all four tabs, your component never changed — it called `t('inbox.unread', { count })` every time. Only the catalog branch set differs, and that's the translator's surface, not yours. These are <Term definition={"A plain counting number — 1, 2, 3, 100. The kind of number the `plural` keyword branches on."}>cardinal</Term> numbers; the next keyword, for positions, branches differently.

## `Intl.PluralRules` is the engine

It's worth seeing the machinery under the message, so the per-language behavior doesn't feel like magic. The thing that maps a number to a plural category is a native browser API: `Intl.PluralRules`. Every ICU MessageFormat implementation calls into it (or a polyfill) to choose the branch.

```ts
new Intl.PluralRules('en-US').select(1); // 'one'
new Intl.PluralRules('en-US').select(2); // 'other'
new Intl.PluralRules('ru-RU').select(5); // 'many'
```

You'll rarely call this yourself — ICU does it for you when it picks a branch. But running it once proves the categories are real and shipping in your runtime today, not invented by a library. (Pass `{ type: 'ordinal' }` to switch it to ordinal categories — that's the engine behind the next section.)

Prove it to yourself. Implement `categoryFor` so it returns the plural category for a number in a given locale, then watch Russian hand back `few` and `many` from one line of standard JavaScript.

## Exact-match overrides for product copy

Back to that `=0` in the inbox message. CLDR categories cover *grammar* — but sometimes you want different *wording* for a specific number, and that's a product decision, not a grammatical one. "No unread messages" reads better than "0 unread messages." "You" reads better than "1 person" in some narrative copy. ICU's exact matches — `=0`, `=1`, `=2` — handle this. They're literal value matches, tried *before* the category keywords, so `=0` wins for exactly zero and `one`/`other` handle the rest. The convention is to list exact matches first, then keywords, then `other`.

There's a sharp rule hiding here, and it's the difference between code that works in English and code that works everywhere: **reach for the CLDR keyword by default, and use an exact match only when the wording itself changes.** It is tempting to write `=1` instead of `one`, since in English `one` only ever matches the number 1 — they look interchangeable. They aren't. In some languages the `one` category covers more than the literal 1 (numbers ending in 1, like 21 and 31, can land in `one`). Write `=1` and you catch *only* the literal 1; every other number that should have used the singular form silently falls through to `other`. It passes every test you'd write in English and breaks the moment a translator localizes it.

<CodeVariants>

    ```text "=1"
    {count, plural, =1 {# unread message} other {# unread messages}}
    ```

    </div>
    **The trap.** Works perfectly in English, where `one` only ever means 1. But `=1` matches *only* the literal 1 — in a language whose `one` category also covers 21, 31, … those numbers silently drop to `other` and get the wrong form.
  </CodeVariant>

  <CodeVariant label="CLDR keyword + =0 override">

  </CodeVariant>
</CodeVariants>

The override lives in the source-locale catalog as a product call. Translators inherit the structure and decide, per language, whether the same carve-out makes sense for them.

## Ordinals need `selectordinal`

The inbox count is a cardinal — *how many*. Ranks and positions are a different kind of number — *which place* — and they branch on a different set of rules. "1st, 2nd, 3rd, 4th" is the canonical English example, and it's the second selector keyword: `selectordinal`.

<div data-mark-color="violet">

```text "selectordinal"
{rank, selectordinal, one {#st place} two {#nd place} few {#rd place} other {#th place}}
```

</div>

`selectordinal` works exactly like `plural` — same `#` token, same `other`-is-mandatory rule — but it consults CLDR's *ordinal* rules instead of cardinal ones. The two rule sets disagree, which is the counter-intuitive part. The same numbers land in different categories depending on which kind you ask for:

```text
n:          1      2       3       4
cardinal:   one    other   other   other     → 1, 2, 3, 4 (no suffix logic)
ordinal:    one    two     few     other     → 1st, 2nd, 3rd, 4th
```

So `2` is cardinal-`other` but ordinal-`two`, and `3` is ordinal-`few`. That's why "2 messages" but "2nd place." Other languages differ entirely — German writes "1.", "2.", Japanese uses 番 — and again, the translator picks the branches their language needs while you write one `t('leaderboard.rank', { rank })`. Reach for `selectordinal` for leaderboards, rankings, positions, "your 3rd invoice." These are <Term definition={"A position number — 1st, 2nd, 3rd. Distinct from a cardinal counting number, and it branches on a different CLDR rule set."}>ordinal</Term> numbers.

## `select` for variants the data already carries

The third keyword changes what you branch on. `plural` and `selectordinal` branch on a *number*, using categories the runtime computes from CLDR. `select` branches on a *string*, using literal matches you define — and like the other two, it needs a mandatory `other` fallback. This is the keyword for variants like gender, notification type, or role.

Here's the new running example for the rest of the lesson — a "liked your post" notification that varies by the actor's gender:

<div data-mark-color="violet">

```text "select"
{gender, select, male {He liked your post} female {She liked your post} other {They liked your post}}
```

</div>

The selector is a string — `'male'`, `'female'`, anything — and each branch is a literal match against it. `other` catches everything you didn't list, which makes it the right home for "prefer not to say," unknown, or any value the data doesn't pin down. The same shape works for a notification `type` (`invoice`, `payment`, `other`) or an organization `role` (`admin`, `member`, `other`).

Now the rule that matters most, the one that separates a correct `select` from a bug: **`select` renders a distinction the data already carries — never one you infer.** The classic mistake is reaching for the user's name to *guess* gender. Don't. The selector value comes from a column your database actually stores. If your data doesn't record gender, then the message has only an `other` branch and "They liked your post" — and that's not a gap to paper over, it's the correct, complete answer. Many English source strings collapse to gender-neutral writing for exactly this reason; a German catalog might carry all three branches because German pronouns and nouns inflect, but the *data* feeding the selector is the same either way.

## Gendered counts: nesting select over plural

Real notifications combine both dimensions. "He has 3 new messages" needs a person *and* a count — a `select` *and* a `plural`. ICU composes them by nesting: a `select` whose every branch contains a complete `plural` message.

<AnnotatedCode lang="text" maxLines={6} code={`
{gender, select,
  male {{count, plural, one {He has # new message} other {He has # new messages}}}
  female {{count, plural, one {She has # new message} other {She has # new messages}}}
  other {{count, plural, one {They have # new message} other {They have # new messages}}}}
`}>
  <AnnotatedStep meta={`"gender" "select"`} color="violet">
    The **outer selector**: a `select` on `gender`, with the three string branches you already know — `male`, `female`, `other`. Nothing new yet; this is the select from the last section.
  </AnnotatedStep>

  <AnnotatedStep meta="{2}" color="blue">
    Each branch's **body is itself a complete message** — here, a full `plural` on `count`. The `male` arm isn't a string; it's a whole nested ICU message that runs only when `gender` is `'male'`.
  </AnnotatedStep>

  <AnnotatedStep meta={`"one {He has # new message} other {He has # new messages}"`} color="blue">
    Inside, the familiar **plural branch table**: `one` and `other`, picked by the count's CLDR category. The outer select chose the pronoun; the inner plural chose the noun form.
  </AnnotatedStep>

  <AnnotatedStep meta={`"#"`} color="orange">
    Two levels deep, **`#` still means the formatted count**. The nesting changed nothing about the token — it's the count, locale-formatted, exactly as before.
  </AnnotatedStep>
</AnnotatedCode>

When you nest, put the lower-cardinality dimension on the *outside*. Here gender has three values and plural multiplies inside each, so `select` (gender) wraps `plural` (count). It's a readability heuristic, not a hard law — the real test is whether the translator can follow the structure — but coarser-cut-outside is the shape that usually reads best. And your call stays as flat as ever, two values and one key:

<div data-mark-color="green">

```tsx "{ gender, count }"
return <p>{t('notification.newMessages', { gender, count })}</p>;
```

</div>

This is the recurring refrain of the whole lesson, and the nested case makes it vivid: even two levels deep, the translator only ever edits the text inside the innermost arms — "He has # new message," "She has # new messages." They never touch the structure, never see a brace they didn't write, never read a line of JavaScript. The ICU syntax is the boundary; everything inside a branch is just text.

## Authoring an ICU message: the rules that bite

You can now read these strings. Authoring them correctly is its own skill, because three of the four common mistakes are silent. Here are the rules, each tied to the failure it prevents:

1. **`other` is mandatory.** Leave it out and the parser throws when it loads the message — a real, loud error. This is the one mistake the tooling catches for you; the other three don't throw.
2. **Use `#` for the count inside plural branches, never `{count}`.** Get it wrong and you ship "5 5 messages" with no error in sight.
3. **CLDR keyword by default, exact match only for wording overrides.** `=1` in place of `one` works in English and quietly drops 21, 31, … to the wrong branch in other languages.
4. **`select` needs the data to carry the value.** Branch on a column your database stores; never infer the distinction from a name.

Two smaller hazards round out the set. ICU treats `{`, `}`, and `'` as syntax, so a literal apostrophe or brace inside your text has to be escaped — and JSON editors don't know ICU's escaping rules, so the safe move is to validate a catalog by running it through the actual runtime, not by reading it. And while every serious library handles all three keywords, some that *advertise* "ICU support" skip `selectordinal` or `select`; `next-intl` covers all three, but if you ever swap libraries, confirm against the runtime rather than the marketing. (If you ever let content editors write these strings in a database, validate the ICU syntax at write time — but DB-stored messages are a separate topic.)

## MessageFormat 2, in one breath

One term to recognize in the wild. Unicode is finalizing <Term definition={"MessageFormat 2.\nThe next version of the ICU message syntax, with explicit .input / .local / .match declarations and first-class named-formatter integration. Standardizing as of 2026."}>MessageFormat 2 (MF2)</Term> — a cleaner syntax with explicit declarations and built-in formatter integration. As of 2026 most i18n libraries, `next-intl` included, still target MF1, the syntax you just learned. When the ecosystem moves to MF2 the migration is mechanical, so nothing here is wasted effort. You don't need the MF2 syntax today; you only need to recognize the name when it comes up.

## Recap

Count-shaped and person-variant strings stay single keys — the *logic* about how they vary moves into the catalog as ICU MessageFormat. `plural` and `selectordinal` branch on a number using the CLDR categories the runtime computes; `select` branches on a string the data already carries. `#` is the formatted count, `other` is always required, and the translator only ever edits the text inside the branches. Your component call never carries any of it — it stays flat: `t(key, { count })`, `t(key, { gender, count })`, no ternary, no inference.

There's one thing ICU deliberately handed off: formatting a number or a date *inside* a message. The next lesson picks that up directly — `Intl.NumberFormat` for currencies and percentages, `Intl.DateTimeFormat` for the Temporal values from earlier in the chapter, `Intl.RelativeTimeFormat` for "3 days ago" — the native formatter family that powers every locale-aware render in your app.
