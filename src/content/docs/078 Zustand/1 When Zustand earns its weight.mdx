By now you hold client state five different ways, and not once have you reached for a global store.
A toggle lives in `useState`.
Related fields that change together live in `useReducer`.
A value a sibling needs gets lifted and passed through Context.
A filter or a sort goes into the URL.
Server data the client caches goes to TanStack Query.

Zustand is the library people install when they think "global state," and the first thing you need to hear is that those five defaults already got there first.
So the question this lesson answers is narrow and specific: when does a feature genuinely need an in-memory store that lives across the whole app, sitting outside every one of those five homes?

The destination is a real screen.
Later in this chapter you build a four-step customer-onboarding wizard whose steps sit on separate routes — `/customers/new/step-1` through `step-4` — and every step writes into one shared draft.
That draft has nowhere good to live among the five defaults, and seeing exactly why is the whole point of the chapter.
By the end of this lesson you can run a funnel that decides, *before you install anything*, whether Zustand earns its weight, and you can name the one architectural rule that keeps it from quietly spreading into everything.

One line carries the whole lesson, so hold onto it: **if `useState` covers it, ship `useState`.**
Zustand is conditional, and it is per-feature.
It is never the app's ambient state bus.

## The five defaults already cover most client state

Before we find the gap, anchor the map.
Each of these owns a specific shape of state, and the triggers later are defined entirely by contrast with them — so this is recall, not new material.

<DefaultsMapTable />

Each of these has a boundary.
And there is a shape that falls outside all five at once: state that is genuinely shared, is not the server's, does not belong in the URL, and lives in memory across component trees that are **disjoint or split across routes** — trees with no natural common ancestor to hold it.
That shape is what the rest of this lesson is about.

## The three triggers that cross the threshold

The course accepts three reasons to bring Zustand into a SaaS codebase, and only three.
Each one names a default it crosses, and each one carries a qualifier — a senior call — that keeps it from being abused.
The qualifier is the real content here.
A junior can memorize three triggers; the skill is recognizing when a feature *isn't* one of them.

### Genuinely shared state across disjoint or cross-route trees

This is the core trigger.

Lifting plus Context works on one condition: every consumer lives under one provider, in one subtree.
The provider sits at a common ancestor, and the value flows down to everyone below it.
The moment your consumers stop sharing a natural ancestor — they sit in **disjoint trees**, or on the far sides of a **route boundary** — Context stops helping.
Think about the wizard.
Step one renders under the route segment `/customers/new/step-1`; step four renders under `/customers/new/step-4`.
React tears down step one's subtree the moment you navigate to step two.
To share the draft through Context you would hoist it to some layout high enough to wrap all four routes, then thread it back down through every intermediate segment — and every one of those intermediates becomes a Client Component passing data it never reads.
That is prop-drilling wearing a Context costume.

The same shape shows up wherever the readers genuinely scatter:

- a **command palette** whose open state is read by the topbar, by a layout overlay, and by a global keyboard handler — three subtrees that never meet;
- a **cart** read by a header badge, a slide-over panel, and a checkout page on a different route.

The senior call: this trigger is real only when the consumers genuinely cannot sit under one natural provider.
"Passing this prop down two levels feels tedious" is not the trigger.
Tedium is solved by lifting one more level.
The trigger is solved by a store because there is no level to lift to.

### Imperative actions fired from unrelated subtrees

Some state isn't really *read* across the tree — it's *commanded*.
"Open the global toast." "Fire the confirmation modal from this leaf." "Collapse the sidebar from a button buried six levels deep."

The default move is to thread a callback down from wherever the toast or the modal actually lives.
That works for one or two callers.
A small store with an `open()` / `close()` / `fire()` action surface beats threading that callback through six layers of components that exist only to pass it along.

The senior call — and this is the most-abused trigger, so the gate matters: the trigger is real only when **more than two unrelated subtrees** fire the action.
Two callers under a shared parent?
Lift the callback to the parent and pass it down.
You reach for a store when the callers are scattered and there is no shared parent low enough to hold the callback without dragging it across half the tree.

### Frequently mutated shared state where Context re-renders the whole tree

This trigger is about cost, and to feel it you need one fact about Context.

When **any** value on a Context changes, React re-renders **every** consumer of that Context — not just the ones that read the part that changed.
Everyone reading the Context re-renders, every time, on every update.
For state that changes rarely (a theme, a locale) that cost is invisible.
For state that mutates constantly — a cart recalculating per keystroke, a wizard validating per field — and that has many consumers each reading a *different* slice, that whole-tree re-render becomes the bottleneck.

Zustand subscribes differently.
A component reads a store through a <Term definition="A function that reads just the slice of store state a component needs, so the component re-renders only when that slice changes — not when anything else in the store does.">selector</Term> — a function picking out just the slice it needs — and it re-renders only when that selected slice changes.
Read `state.contact.email` and you re-render when the email changes — not when the billing address three slices over changes.

The asymmetry is easy to assert and hard to picture, so look at it directly.
The following widget models one parent and three children, each child reading a different field.
The tabs swap the strategy.
Click each trigger and watch the render badges.

That is the whole argument for trigger three.

But notice how narrow it is.
This is a **profiled** trigger: you reach for it when you have *measured* a re-render cost, not when you suspect one.
And it is the weakest of the three standing alone — a four-step wizard almost never has a render-cost problem worth a library.
Trigger three rarely justifies Zustand by itself.
It earns its place stacked on top of triggers one and two, when state that is already shared across disjoint trees *also* mutates often.

## What does not clear the bar

The triggers tell you when to reach for Zustand.
This section — which carries equal weight — tells you when not to, and that is where the judgment actually lives.

For each workload below, a default already owns it.
The senior move is to name that default out loud.

<NotZustandTable />

The reflex behind the table: **enumerate the default the workload would otherwise use, and only if every default is wrong does Zustand earn its weight.**
If `useState` covers it, ship `useState`.

Two overreaches account for almost every misuse of this library in real SaaS code, and both fail the reflex above.

The first is reaching for a store because "global state feels cleaner."
It doesn't feel cleaner six months later, when a new reader opens a 400-line `useAppStore` and has to trace which of twenty slices a bug touches.
This is the 2016 Redux-store-of-the-universe rebuilt in fewer lines, and it carries the same coupling cost.
Cleaner is the trap, not the win.

The second is using Zustand to cache server data — pulling a list into a store, then watching a user sit on a stale view because nothing told the store to refetch.
That is TanStack Query's entire job: it owns the cache, the refetch, and the staleness.
Before any of the five defaults *or* Zustand, the real first question is whether the client owns this state at all — and we'll formalize that check in a moment.

## The cost of bringing the library in

"Earns its weight" only means something if the weight is real.
Every store you add levies the same recurring taxes, and naming them is what turns the high bar from arbitrary into motivated.

- **A new "where does this state live?" question** that every future reader has to ask and answer. The five defaults have known homes; a store is a new place to look.
- **A per-feature store file** that readers have to find before they can change anything. (This is what the file-layout rule in the next section exists to fix.)
- **An SSR wiring trap.** On the App Router, a store written the naive way — defined once at module scope — is shared across requests on the server, so one user's draft can leak into the next user's first render. The trap is real; the *fix* is the next lesson. For now, just know the trap is there, because it's one of the reasons the bar is high.
- **A `'use client'` boundary at every consumer.** The store is client-only. Every component that reads it opts out of being a Server Component — a cost you pay on each call site, not once.
- **Reset discipline.** A store lives as long as the browser tab. Without an explicit reset on sign-out, on org-switch, and on a successful submit, last session's state bleeds into the next one in the same tab. Forget it and you have a data-isolation bug at the client layer — the same discipline you already met with `queryClient.clear()` at the tenancy boundary in the TanStack Query chapter, now your responsibility again.

A default carries none of these.
That asymmetry is the entire reason Zustand is conditional: the library has to clear the *combined* cost of all five taxes before it's worth reaching for — which is exactly why the three triggers that justify it are so narrow.

:::note
<Term definition="Server-side rendering — turning the page into HTML on the server before it reaches the browser. On the App Router it is the default for every route.">SSR</Term> is why the wiring trap exists. A store defined once at module scope is fine in a browser-only React app, where each user runs their own copy. On a server rendering many users' requests through one process, that single copy is shared — which is the leak. The next lesson fixes it with a per-request provider.
:::

## One store per feature, never one global store

This is the chapter's load-bearing architectural rule, and it lands here because you have now earned it.
State it plainly: **every Zustand store has a single feature owner, and it lives next to the feature it owns.**
There is no `useAppStore` holding everything.

Teach yourself the rule by its failure mode.
One `useAppStore` with twelve unrelated slices — cart, wizard, palette, theme, modal, toast, sidebar, and on — is the Redux-store-of-the-universe again, just shorter.
Every feature's state is tangled in one module.
Every change is a merge-conflict magnet.
Every consumer imports the world to read one field.
Per-feature stores invert all of that: each store's surface stays small, you can find it by the feature's name, and you can delete a feature by deleting its store and nothing else.

The following figure puts the two shapes side by side.

</Figure>

The convention that follows from the rule is mechanical, so just adopt it:

- **Name each store for its one owner** — `useWizardStore`, `useCartStore`, `useCommandPaletteStore`. The name tells a reader exactly which feature it belongs to.
- **Co-locate the store beside the feature that owns it** — its files live next to that feature's other code, never in a global `/store` directory that becomes the new junk drawer.

A feature that lives on its own route keeps its store in that route's private folders, beside the steps, schema, and submit action it serves:

<FileTree>
- src/app/(app)/customers/new/
  - _lib/wizard/
    - store.ts the wizard store `useWizardStore`
    - … types, slices
  - _components/
    - **wizard-store-provider.tsx** `'use client'` — wires the store in
    - …
  - layout.tsx
  - … the step pages this chapter builds toward
</FileTree>

The exact files and why they split across `_lib/` and `_components/` are the next lesson's job; what matters here is the shape — one store, named for its feature, sitting beside it.

This is the rule that keeps the chapter's own tooling from becoming the thing the chapter warns against.
It holds across every lesson here: one named store per feature, beside its feature, every time.

## Do we even own this state on the client?

There is one question that sits *before* the whole funnel, and it catches the single most common Zustand misuse in real code: using a store as a poor man's database.

Before you weigh `useState` against Context against a store, ask whether the state is actually the **server's**.
Plenty of state looks like client state and isn't:

- a draft that **must** survive a refresh is server state — write a `draft` row and load it back;
- a list view's filter is URL state — it belongs in the address bar;
- a comment thread is server state — TanStack Query owns it.

Here is the boundary that scopes Zustand precisely: **Zustand owns client-only state that is allowed to vanish on refresh by product decision.**
A store lives in browser memory.
Refresh the page and it's gone — and that is **by design**, not a limitation.
If losing the state on refresh is unacceptable, it does not belong in a store; it belongs on the server, with all the persistence and cleanup cost that implies, or in `sessionStorage` if the product can live with browser-tab durability.
Either way, surviving a refresh is a *product decision* you make deliberately — not something you bolt onto a store because you forgot to ask where the state really lived.

So the first gate of the funnel is the bluntest one: is this the server's state?
If yes, you are done before you start.

## The funnel: deciding before you install

Everything above collapses into a single ordered set of gates.
The order is the lesson — a senior asks these questions in this sequence, top to bottom, and stops at the first one that lands.
Walk it once and commit each answer; that sequence is more durable than any flat cheat sheet, because it's the order you'll actually run in your head the next time someone says "let's add a store for that."

<StateMachineWalker kind="decision" title="Does this state need Zustand?">
  <Question
    id="server"
    prompt="Is this state the server's?"
    description="Canonical data, or anything that must survive a refresh."
  >
    <Branch label="Yes — it's canonical data or must survive refresh" to="leaf-server" />
    <Branch label="No — it's client-only and may vanish on refresh" to="local" />
  </Question>

  <Question id="local" prompt="Would useState or useReducer cover it?">
    <Branch label="One component owns it" to="leaf-usestate" />
    <Branch label="It's shared across components" to="context" />
  </Question>

  <Question
    id="context"
    prompt="Do all the consumers sit under one provider in one subtree, written rarely?"
  >
    <Branch label="Yes — one subtree, low write frequency" to="leaf-context" />
    <Branch
      label="No — disjoint or cross-route, or it mutates often with many readers"
      to="url"
    />
  </Question>

  <Question id="url" prompt="Is it shareable view state — a filter, sort, search, or page?">
    <Branch label="Yes — it belongs in the address bar" to="leaf-nuqs" />
    <Branch label="No" to="leaf-zustand" />
  </Question>

  <Leaf id="leaf-server" verdict="Server Components / Server Actions (or TanStack Query)">
    It's the server's. Read it in a Server Component, write it through a Server Action — or reach for TanStack Query if one of its four triggers (polling, cross-view cache, optimistic-with-rollback, infinite scroll) applies. The client should not own a copy.
  </Leaf>

  <Leaf id="leaf-usestate" verdict="useState — ship it">
    One component owns it, nobody else reads it. This is the default. Don't lift it until a sibling actually needs it.
  </Leaf>

  <Leaf id="leaf-context" verdict="Lift state + React Context">
    Shared, but the consumers share a natural ancestor and you write to it rarely. Lift the state to that ancestor and pass it down through Context. The whole-tree re-render costs nothing at low write frequency.
  </Leaf>

  <Leaf id="leaf-nuqs" verdict="nuqs — URL state">
    Shareable view state belongs in the URL, where it survives a refresh and a shared link. Server Components read it straight from the params.
  </Leaf>

  <Leaf id="leaf-zustand" verdict="Zustand earns its weight">
    It's client-only, genuinely shared across disjoint or cross-route trees, and no default fits. Reach for Zustand — and remember the sub-rule: **one store per feature, never one global `useAppStore`**, co-located beside the feature it owns.
  </Leaf>
</StateMachineWalker>

Only state that falls through *every* gate lands on Zustand.
That is why, even in a large SaaS app, the number of stores stays small and each one is named for the feature it serves.

## Why Zustand, and not Redux, Jotai, or Valtio

You will be offered alternatives — by a teammate, by an AI agent, by the top of a search result — so it's worth one paragraph each on why the course picks Zustand for greenfield 2026 work.
This is "why this tool," not a library survey.

**Redux Toolkit** is the incumbent, and it's heavier than the job needs.
It still asks you to think in reducers, actions, and a dispatcher — ceremony that Zustand's v5 API simply doesn't have.
Pick Zustand by default in 2026; reach for Redux Toolkit only when a codebase already standardizes on it and consistency wins.

**Jotai** is atom-based: state is built bottom-up out of many small independent atoms with a derivation graph between them.
It's a genuinely different mental shape, and it fights the "a few named stores with slices" model this course teaches.
Jotai is the right call when your state really is dozens of tiny independent atoms with computed relationships — which is rare in ordinary SaaS UI.

**Valtio** takes a proxy-based, mutate-it-directly approach — nicer ergonomics in spots, with a smaller ecosystem around it.
Here's the telling detail: Zustand, Jotai, and Valtio are all <Term definition="Poimandres — the open-source collective that maintains Zustand, Jotai, Valtio, React Three Fiber, and many other React libraries.">pmndrs</Term> projects, built by the same people, and of the three it's Zustand that fits the canonical 2026 SaaS shape: a handful of named per-feature stores read through selector subscriptions.

The course's call, stated once: **Zustand v5, no detours.**

:::note
If the team you join already runs Redux, that's fine — don't rip it out to add Zustand. This chapter teaches the greenfield-2026 default, not a migration.
:::

## The screen we'll build against

The chapter has a destination, and it's worth naming concretely so the rest of the lessons have something to point at.

It's a four-step customer-onboarding wizard, living at `/customers/new/step-1` through `step-4`: contact, then billing, then preferences, then a review screen.
Run it past the funnel and it lands on Zustand — for stacked reasons, not one.
The four steps sit on **disjoint route segments** and all write into one shared draft: trigger one.
The progress indicator, the Next button, and the review screen all read from that same draft across the tree: trigger two.
No default fits, so the wizard clears the bar.

What it does *not* need from you yet is any Zustand code.
This lesson is deliberately syntax-free — trigger before tool.
The next lesson teaches the primitives a senior actually writes and the per-request provider that closes the SSR trap; the lesson after that runs the full funnel against this wizard and wires it up; and the project that follows builds the whole surface end to end.
For now you have the thing that has to come first: the gate that decides whether the library belongs here at all.
