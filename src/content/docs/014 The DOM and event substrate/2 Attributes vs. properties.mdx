Here is a small experiment you can run right now. Open any page with a text input, type the word `hello` into it, and then ask the browser two questions about that same field:

```js
input.value;                    // 'hello'
input.getAttribute('value');    // ''
```

The first call reports what you typed. The second insists the value is empty. And if you open View Source, the page's HTML still reads `<input value="">`, exactly as the server sent it. Three views of one input, and they don't agree. So which one is telling the truth?

All of them. They are answering different questions. This is the exact debt the previous lesson opened and named but didn't pay off: the HTML source is a photograph the parser took once at load time, and the DOM is the running program that has been changing ever since. That spine was abstract. This lesson is that same idea brought down to the level of a single piece of data on a single node — and once you see it there, every confusing form bug, every `defaultValue`-versus-`value` decision you'll make in React, and every hydration-mismatch error you'll see in the console becomes legible.

You will spend your career writing JSX, not `setAttribute`. So why learn the raw DOM split at all? Because the abstraction leaks, and it leaks in a small, fixed set of production places that are unreadable without this model. The goal here isn't to drill an API you'll type daily — it's to install the vocabulary that makes those leaks make sense later.

## Two slots on every node: the parsed string and the live value

Start with the simplest possible version of the model, before any rules or categories. Many of the data points on a node are stored in not one place but two, side by side:

- The **attribute** is the string the HTML parser read out of the source and recorded the moment it built the node. You reach it with `getAttribute(name)`, `setAttribute(name, value)`, `hasAttribute(name)`, and `removeAttribute(name)`. It is always a string — or `null` when the attribute isn't there at all.
- The **property** is a named field on the live element object, with a real type: a string, a boolean, a number, sometimes another object. You reach it with `element.propName`.

Two slots, one node. That picture is the whole lesson — every category and footgun that follows hangs off it, so let's make it spatial.

That over-time view is the core of everything. Hold onto the one-line version of it:

:::note
An **attribute** is a string the HTML parser captured at load time. A **property** is a typed, live value on the node object that the runtime reads and writes. Two storage slots, one node.
:::

A quick word on a term you'll meet constantly: when writing one side automatically updates the other, the property and attribute are said to <Term definition="When two stores stay in sync — writing one side automatically updates the other. Most identical-name attribute/property pairs reflect; value, checked, and selected deliberately do not.">reflect</Term> each other. The `id` and `type` rows in the figure reflect. The `value` row does not. That single word — *does this pair reflect or not?* — is most of what separates the categories you're about to learn.

## The four ways an attribute and a property relate

Here is the load-bearing part. Once you accept that an attribute and a property both exist for a given thing, there are only four ways they can relate to each other. An experienced engineer reads any HTML or JSX attribute and instantly knows which of the four buckets it falls into — that snap classification is the real skill this lesson installs. Take them one at a time, each with the smallest possible example.

### Identical name, kept in sync

The simplest case. The attribute and the property share a name, and they reflect — write either side and the other follows. `id`, `hidden`, `lang`, and `title` all work this way.

```js
element.id = 'checkout';
element.getAttribute('id');   // 'checkout' — the attribute followed
element.setAttribute('id', 'cart');
element.id;                   // 'cart' — and the property followed back
```

The reflex here is easy: just use the property. It's typed, it's live, and `element.id = 'cart'` reads better than `element.setAttribute('id', 'cart')`. You almost never touch the attribute side for these.

### Renamed properties: why JSX says className

Sometimes the attribute and property mean the same thing but spell their names differently. The attribute keeps its HTML spelling; the property gets renamed — usually to camelCase, sometimes to dodge a word JavaScript has reserved. You can't have a property literally named `class` or `for`, because those are keywords, so the DOM picks a legal identifier instead.

The ones worth committing to memory:

```text title="Attribute → DOM property"
Attribute        Property
class            className
for              htmlFor
tabindex         tabIndex
readonly         readOnly
maxlength        maxLength
```

The multi-word HTML attributes (`cellpadding`, `rowspan`, and friends) follow the same camelCase rule on the property side; you don't need to memorize the full list, just the pattern.

Now the payoff. Look at this snippet:

<CodeTooltips tooltips={{
  htmlFor: 'The DOM property name for the for attribute — for is a reserved word in JavaScript, so the platform picked a legal identifier. JSX didn\'t invent it; you already know it.',
  className: 'The DOM property name for the class attribute, for the same reason. JSX assigns to properties, so it uses the property spelling.',
}}>
```jsx
<label htmlFor="email" className="block">Email</label>
```
</CodeTooltips>

Most newcomers assume React renamed `class` to `className` on a whim, decide it's a React quirk to memorize, and quietly file it under "framework weirdness" — never connecting it back to the DOM. Here's the reframe that dissolves that: **JSX prop names follow the property side because JSX assigns to properties, not attributes.** `className` and `htmlFor` are not React inventions. They are the DOM property names you just learned. The knowledge transfers in both directions — what you know about the DOM is already knowledge about JSX. Hold onto that; it's why this substrate lesson pays off the moment you start writing React.

### Default vs. current: value, checked, selected

This is the most consequential pattern, and it's the one from the opening experiment. For a handful of properties, the attribute and the property deliberately *don't* reflect, because they're tracking two genuinely different things:

- `value` (attribute) is the *initial* value the parser saw; `.value` (property) is the *current* live value. The browser also gives you `.defaultValue`, a property that exposes the attribute side, so you can read the original even after the field has changed.
- `checked` (attribute) is the initial checked state; `.checked` is the current one, with `.defaultChecked` for the original.
- `selected` (attribute) on an `<option>` is the initial selection; `.selected` is the current one, with `.defaultSelected` for the original.

The attribute is the starting point; the property is where things are now. The moment a user types, or clicks a checkbox, or your code runs `input.value = '…'`, the two diverge — which is exactly the divergence you scrubbed through in the typing sequence above. The attribute froze at parse time; the property kept living.

:::note[Forward reference]
This split is the entire reason React has both a `defaultValue` prop and a `value` prop. `defaultValue` writes the *attribute* (the initial value, leaving the field free to change on its own — *uncontrolled*); `value` writes the *property* (React owns the current value — *controlled*). When a later chapter teaches controlled versus uncontrolled inputs, you'll recognize it as a direct consequence of this platform split, not a React invention. You don't need to act on this yet — just file the recognition.
:::

### Attribute-only and property-only

The fourth case: some data lives on only one side, so there's nothing to reflect.

**Property-only.** Some properties have no matching attribute at all. `textContent`, `innerHTML`, `nodeName`, and `tagName` are computed off the live node — there is no `textContent=""` you could write in HTML, and `getAttribute('textContent')` returns `null` because no such attribute exists.

**Attribute-only in practice.** Going the other way, `aria-*` is authored and read as attributes. Reflected properties like `element.ariaLabel` do exist on the platform now, but you don't reach for that property mirror day to day — accessibility state is set and inspected with `getAttribute('aria-label')` and `setAttribute('aria-label', '…')`, and that's the form you'll see in every codebase and tutorial. ARIA gets its own treatment in a later accessibility chapter; here, just recognize it as the attribute-authored case — there's no property bridge you'll actually reach for.

The one big exception to "attribute-only" is the `data-*` family, which *does* get an ergonomic property mirror — but its rename rule earns it a section of its own further down.

That's all four. Before moving on, sort some real attributes into them — recall is what turns this from a table you read into a reflex you have.

## Boolean attributes: presence is the value

There's one rule about a specific group of attributes that trips up nearly everyone at least once, so it earns its own section. For <Term definition="An attribute whose meaning is carried entirely by whether it is present or absent. Its value string is ignored. Examples: disabled, checked, required, hidden, multiple, readonly, selected.">boolean attributes</Term> — `disabled`, `checked`, `readonly`, `required`, `hidden`, `multiple`, `selected` — the browser cares about exactly one thing: **is the attribute present or absent?** The value string is ignored entirely.

That means all three of these *disable* the button:

<div data-mark-color="blue">

```html {3}
<button disabled></button>
<button disabled="disabled"></button>
<button disabled="false"></button>
```

</div>

Read that last one again. `disabled="false"` disables the button. The string `"false"` is just a present value, and presence is all that counts. The only way to enable the element is to *remove* the attribute, not to set it to something falsy-looking.

There's a second sting hiding in that example. `disabled="false"` isn't just non-functional — it's invalid markup. The HTML spec says a boolean attribute's only valid values are the empty string or a repeat of its own name (so `disabled` or `disabled="disabled"`), and an HTML validator will flag `disabled="false"`. Every browser still treats it as present and disables the element anyway. So `"false"` here is a double trap: it doesn't do what you meant, and it's not even legal.

The property side, by contrast, behaves the way your intuition wants: it's a real typed boolean.

```js
button.disabled;          // false  — a real boolean
button.disabled = true;   // disables it
button.disabled = false;  // enables it
```

So the two reflexes are:

- **Raw DOM:** set the property — `button.disabled = false` — because it's typed and reads correctly. Or, if you must work through the attribute, `button.removeAttribute('disabled')`. Never `button.setAttribute('disabled', 'false')`.
- **JSX (recognition only):** you pass a boolean — `disabled={isLoading}` — and React translates that into adding or removing the attribute for you. You never manage presence by hand.

Now make the footgun bite somewhere safe, so it never bites you in a real form.

</PredictWhy>

That last line is the one that bites quietly. Reaching for `Boolean(element.getAttribute('disabled'))` to "check if it's disabled" looks reasonable and is wrong for every present value, because the string came back non-empty. When you want a boolean, read the property, which already is one.

## Enumerated attributes: the browser picks a legal value

One more variation, lighter than the last. Some attributes don't accept arbitrary strings — they have a fixed set of legal values, and when the parser sees something outside that set, the *property* quietly returns a spec-defined fallback instead of the raw string. `<input type>` is the classic one:

```js
// markup: <input type="numbr">  (typo for "number")
input.getAttribute('type');   // 'numbr' — the raw bytes the parser saw
input.type;                   // 'text'  — the spec's fallback for an unknown type
```

The attribute hands back exactly what was written, typo and all. The property hands back what the browser will *actually act on*: an unrecognized `type` falls back to `text`, so `.type` reports `'text'`. Same spine, a new angle — the attribute is the raw input, and the property is the resolved, engineered value the runtime uses.

## data-* and the dataset bridge

`data-*` is the one attribute family that ships with a typed, ergonomic property mirror, and it's one you'll author constantly. Any attribute you prefix with `data-` becomes readable through the <Term definition="A DOM property that exposes all of an element's data-* attributes as a single object, with the data- prefix stripped and kebab-case names converted to camelCase. Values are always strings.">`dataset`</Term> property, with the prefix stripped and the kebab-case name folded to camelCase.

```js
// markup: <div data-user-id="42">
element.dataset.userId;       // '42'  — note: still a string
element.dataset.userId = '7'; // updates the data-user-id attribute to "7"
```

Two things to notice. The rename is mechanical — drop `data-`, turn each `-x` into `X` — so `data-user-id` becomes `dataset.userId`. And the values are always strings, because underneath it's still an attribute store; that `'42'` never becomes a number on its own.

Why would an experienced engineer reach for `data-*` in a 2026 React codebase? Three recurring places — you don't need to act on any of them yet, just recognize the shape:

- **State-driven styling.** Tailwind can style off DOM state directly — `data-[state=open]:rotate-180` reads a `data-state` attribute, not a piece of React state. The convention across the course is that this kind of styling reads DOM state, which is exactly a `data-*` attribute.
- **Event delegation.** A single handler high in the tree can route on `event.target.closest('[data-action]').dataset.action` — one listener serving many buttons, each tagged with what it does. This is the direct setup for the next lesson, where you'll build delegated handlers; `data-*` is the discriminator they route on.
- **Test selectors.** `data-testid` gives tests a stable hook that doesn't break when you restyle. Recognition only for now.

Here's the loop worth closing: you *write* `data-*` in JSX, and you *read* it as `dataset.*` in a delegation handler. Authoring side and reading side, same data, two spellings — the rename rule is the bridge between them.

## Reading both sides in DevTools

This is a debugging skill you'll use often, and it builds straight on the previous lesson's point that the Elements panel is the *live tree*, not the source. When you select an element, the panel's right-hand rail gives you both sides of every slot this lesson described:

- The **markup in the Elements tree** — the attributes written inside the tag, like `value=""` — is the *parsed strings*, the attribute side.
- The **Properties** subpanel (a tab in the right-hand rail) lists the full live property surface of the selected node (`$0`) — the property side, with real types.

The senior debugging move falls right out of this lesson: when an attribute *looks* correct but the page misbehaves, or vice versa, check the *other* side. The discrepancy is almost always the attribute/property divergence you now have a name for. Go back to the opening experiment — inspect the input you typed `hello` into, and you'll see `value=""` sitting right there in the markup while the Properties subpanel reports `value: "hello"`. There's the whole lesson in one inspection.

The fastest manual probe is the Console, using the `$0` reference for the last-inspected element:

```js
$0.value;                  // 'hello' — the live property
$0.getAttribute('value');  // ''      — the parsed attribute
```

## Where this leaks in a React codebase

Pull it together. You learned a raw-DOM split you'll rarely type by hand, so here is the payoff: the four places it surfaces in React code, each one an error or a decision that's unreadable without the model you just built. Every one of these is recognition — none of it is React you need to write yet.

- **JSX prop naming.** `className`, `htmlFor`, `tabIndex` are the property side speaking through JSX — DOM names you already know, not React inventions. This is the leak you've already seen; it's first because it's the one you'll meet on day one.
- **`defaultValue` vs. `value`.** This pair decides whether an input is uncontrolled (`defaultValue` writes the attribute) or controlled (`value` writes the property). The forms chapter owns the full story; you now know where the two prop names come from.
- **Hydration-mismatch errors.** React renders HTML on the server (the attribute side) and then hydrates against the live DOM on the client (the property side). When the two disagree — a boolean attribute, or a `value`/`checked` that differs — React logs a hydration-mismatch error. This is precisely the debt the previous lesson left open ("a hydration mismatch is the snapshot-versus-live gap"), now made exact: the snapshot is the attribute, the live state is the property, and the mismatch is them disagreeing. The App Router chapters own the full hydration story.
- **A controlled input with a server-rendered `value`.** React expects to own a controlled input's value, so a stray `value="initial"` arriving in the server HTML produces a console warning. Recognition only — but now you can read it.
