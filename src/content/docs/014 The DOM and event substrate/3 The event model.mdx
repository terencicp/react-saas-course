Here is something that should not happen, but does. You put a click listener on a button. You click the button. Your handler runs — fine, expected. But a listener you also attached to the surrounding `<form>` runs too. And one on `document` runs after that. You clicked the button. You did not click the form, and you certainly did not click the entire document. So why did three handlers fire?

Because the click did not "happen on the button." It *traveled*. Every event in the browser takes a trip through the tree — down from the top to the element you touched, then back up to the top again — and any listener sitting along that route gets a turn. The button, the form, and the document are all on the route, all had a listener, so all three ran.

That single fact is the whole lesson. Once you have it, everything else falls out of it: why those three handlers fired, the difference between "what was clicked" and "what's listening," and the most useful pattern in all of DOM event handling — *delegation*, where one listener on a parent quietly serves hundreds of children. Delegation matters far beyond raw DOM work, because it is **exactly** what React's `onClick` is doing under the hood. Learning the trip is learning what your event handlers compile down to.

The previous lessons built toward this. The first gave you the DOM as a live tree of nodes; the second split each node's data into its parsed and live halves. Now you watch that tree *react* — to clicks, typing, scrolling, every interaction a user has with the page. By the end you'll be able to write a delegated click handler with `closest()` and `data-*`, tell `preventDefault` from `stopPropagation` without guessing, and clean up listeners the 2026 way with a single `AbortController`.

One reassurance up front, same as the last two lessons: you will not write `addEventListener` in most of a React codebase — you'll write `onClick` and let React do the wiring. So why learn the raw substrate? Because `onClick` is built directly on top of it, and the few places you *do* reach for `addEventListener` are unreadable without it. Seeing the substrate is what makes the React abstraction make sense later, instead of being a black box you trust on faith.

## An event makes two trips: capture, target, bubble

Let's watch the trip happen, because a definition list won't make it stick the way the motion will.

When you click an element, the browser does *not* simply hand the event to that element. It starts at the very top of the tree — at `window` — and walks *down* the chain of ancestors toward the element you actually touched. Then, having reached it, it turns around and walks back *up* the same chain to the top. One event, two passes over the same path. The trip has three named phases, in this order:

- **Capture** — the downward leg. The event descends from `window`, through `document`, through each ancestor element, stopping just above the <Term definition="The element the user actually interacted with — the deepest node in the tree at the point of interaction. It stays the same for the entire trip.">target</Term>. The name means the outer elements get the *first* chance to intercept the event before it reaches its destination.
- **Target** — the event arrives at the element you actually interacted with. Listeners attached directly to it fire here.
- **Bubble** — the upward leg. The event rises back up from the target through every ancestor to `window`, the way a bubble rises through water. Listeners on ancestors fire on the way up.

So in the opening mystery, the click started at `window`, descended through `document` and the `<form>` down to the `<button>`, then bubbled back up through the `<form>` and `document` to `window`. Three handlers along that path, three handlers fired. No magic — just the trip.

Now the single most important practical fact about that trip — the one that decides how you write listeners every day. By default, **`addEventListener('click', handler)` registers your handler on the bubble phase.** Not capture. The handler you write with no options runs on the way *up*, which is why the form and document handlers in the intro fired (they had bubble listeners) and fired *after* the button (bottom-up order on the way up).

You opt into the capture phase with a third argument:

```js
button.addEventListener('click', handler);                 // bubble — the default
window.addEventListener('click', handler, { capture: true }); // capture — the downward leg
```

Be honest about the proportions here, because they govern where you spend attention. Production code lives almost entirely on the bubble phase. Capture is a niche reach you'll go years without typing — it exists for two real but rare jobs: intercepting an event at an ancestor *before* a child gets to handle it, and catching certain events at a parent that don't bubble on their own.

That second job points at a footgun worth knowing now, because you'll hit it the first time you try to react to focus. The `focus` and `blur` events **do not bubble** — they fire only at the target, so a listener on an ancestor never sees them. The platform's answer isn't to reach for capture; it's a second pair of events, `focusin` and `focusout`, that do exactly the same thing but *do* bubble. When you need to know that focus entered or left some region, you reach for the bubbling pair. Hold that thought — it matters again the moment we get to delegation.

Before moving on, lock in the order, because the order is the one thing that has to be automatic.

## `target` vs `currentTarget`: what was clicked vs what's listening

The trip you just learned forces a distinction that trips up nearly everyone the first time, so let's make it sharp now rather than let it bite later. Inside any handler, the event object carries two different references to elements, and they answer two different questions.

- `event.target` — **what the user actually hit.** The deepest node at the point of interaction. It is *fixed* for the entire trip: no matter which ancestor's listener is currently running, `target` is always the element the event originated on.
- `event.currentTarget` — **what *this* handler is attached to.** It *changes* as the event moves — at each step of the trip, `currentTarget` is whichever node is running its listener right now. It's the node you called `addEventListener` on.

When a listener sits directly on the element you click, the two are the same and the distinction seems pointless. It stops being pointless the instant the listener sits on an *ancestor* — which, as you're about to see, is every delegation handler. Picture one listener on a `<ul>`. Click any `<li>` inside it and the handler runs with `currentTarget` always pointing at the `<ul>` (that's what the handler is on), while `target` points at the specific `<li>` you clicked (that's what you hit).

```js
// <ul id="menu">
//   <li>Profile</li>
//   <li>Billing</li>
//   <li>Sign out</li>
// </ul>

const menu = document.getElementById('menu');

menu.addEventListener('click', (event) => {
  console.log(event.currentTarget.tagName); // 'UL'  — always the <ul> the handler is on
  console.log(event.target.tagName);        // 'LI'  — whichever item you actually clicked
});
```

Commit this one sentence to memory, because it resolves the confusion every time: **`target` is what was clicked; `currentTarget` is what the handler is on.**

There's a sting hiding in `target` that the next section depends on, so let's surface it now. `target` is the *deepest* node — and that can be deeper than you think. If your `<li>` contains a `<span>` with an icon, and the user clicks the icon, then `target` is the `<span>`, not the `<li>`. It can even be a text node in some cases. So you can't trust `target` to be the element you care about; it's merely the element where the click *landed*. The fix — climbing from wherever the click landed up to the meaningful element — is exactly what `closest()` is for, and it's the heart of the next section.

## Event delegation: one listener for many children

Now the payoff — the pattern this whole lesson has been walking toward, and the one you'll recognize everywhere once you know it.

Start with the problem it solves, because the pattern is only obvious once you've felt the pain of the alternative. Say you have a toolbar with three buttons: Save, Duplicate, Delete. The naive approach is to put a listener on each button:

```js
saveButton.addEventListener('click', handleSave);
duplicateButton.addEventListener('click', handleDuplicate);
deleteButton.addEventListener('click', handleDelete);
```

For three buttons this is fine. Now scale it up to where real apps live, and two cracks appear. First, a list of two hundred rows, each with an action button, means two hundred listeners — two hundred function references the browser holds, two hundred setup calls, all doing essentially the same job. Second, and worse: what about rows that *don't exist yet*? A new row the user adds, or a list you fetch from the server after the page loads, arrives with no listener attached. You'd have to remember to wire one up every single time the DOM changes. Miss one, and a button silently does nothing.

<Term definition="One listener on a stable ancestor that handles events for any number of descendants — including ones added after setup — by inspecting where the event came from.">Delegation</Term> dissolves both problems at once. Instead of one listener per button, you put a *single* listener on a stable ancestor that's always there — the toolbar, the `<ul>`, a container `<div>`. Every click on any descendant bubbles up to that ancestor, so the one listener sees them all. It works for the two-hundredth row exactly as well as the first, and it works for rows that didn't exist when you attached it — because the event bubbles up regardless of when the child was added. One listener, every child, present and future.

The shape is small and always the same. Walk through it part by part.

<AnnotatedCode lang="js" maxLines={16} code={`
// <div id="toolbar">
//   <button data-action="save"><span class="icon">💾</span> Save</button>
//   <button data-action="duplicate">Duplicate</button>
//   <button data-action="delete">Delete</button>
// </div>

const toolbar = document.getElementById('toolbar');

toolbar.addEventListener('click', (event) => {
  const actionEl = event.target.closest('[data-action]');
  if (!actionEl) return;

  switch (actionEl.dataset.action) {
    case 'save':
      saveDocument();
      break;
    case 'duplicate':
      duplicateDocument();
      break;
    case 'delete':
      deleteDocument();
      break;
  }
});
`}>

This pays two debts the earlier lessons left open on purpose. The first lesson promised you'd see `closest()` "doing real work when this chapter reaches event delegation" — this is that work: climbing from the clicked node up to the element that actually carries meaning. The second lesson promised that you *write* `data-*` in JSX and *read* it as `dataset.*` in a delegation handler — this is the reading side finally happening. The discriminator you tagged each button with is what the one listener routes on.

Two refinements that separate a delegation handler that works in the demo from one that works in production:

- **Climb with `closest()`, never trust `target` directly.** You saw why in the predict exercise — `target` is the deepest hit node, often a descendant of the thing you mean. `closest()` is what makes the handler robust to icons, spans, and nested markup inside each button.
- **Reach for `click`, and it covers the keyboard for free.** A native `<button>` fires a `click` event when activated by keyboard — Enter or Space — not just by mouse. So a delegated `click` handler already serves keyboard users with no extra work. This is a real reason to build on `click` rather than `mousedown`/`mouseup`, which fire only for the pointer and leave keyboard users stranded.

And recall the focus footgun from earlier: if you ever want to *delegate* focus handling — one listener on a form noticing when any field inside it gains focus — `focus` won't bubble to your container, so it'll never arrive. Use `focusin`/`focusout`, the bubbling pair, and delegation works the same way it does for clicks.

Here is the recognition that makes this the most leveraged thing in the lesson: **this is what React's event system does.** React does not attach a listener to every `<button onClick>` in your tree. It attaches a small number of listeners high up at the root and, when an event bubbles up, figures out which component it belongs to and calls your handler — delegation, run by the framework at scale. So when you write `onClick`, you're not escaping the pattern you just learned; you're using a polished version of it. (The React side gets its own chapter later — this is recognition, so the machinery feels familiar when you meet it, not new.)

## `preventDefault` and `stopPropagation` do different jobs

These two methods are confused constantly, by people well past their first year. The confusion comes from assuming they're variations on one idea — "stop the event" — when in fact they control two completely independent things. Get the two axes straight and the confusion evaporates.

`event.preventDefault()` cancels the browser's <Term definition="The browser's built-in reaction to an event: a link navigating to its href, a form submitting and reloading, a checkbox toggling, the right-click menu opening.">default action</Term> for that event. It does **not** stop the trip. The event keeps right on traveling to every other listener; you've only told the browser "don't do your built-in thing." Clicking a link still runs every bubble listener — `preventDefault` just stops the navigation. This is the one you reach for constantly.

`event.stopPropagation()` halts the trip. No further ancestors on the current leg get the event. It does **not** cancel the default action — if you `stopPropagation` on a link click but don't also `preventDefault`, the browser still navigates. (There's a stronger sibling, `stopImmediatePropagation()`, which additionally blocks any *other* listeners on the same element, not just ancestors.)

So the two are orthogonal — two independent switches. You can flip either, both, or neither.

    Two independent axes. `preventDefault` works the horizontal one (the browser's built-in reaction); `stopPropagation` works the vertical one (the trip through the tree). They never touch each other's axis — so you can flip either, both, or neither.
  </Fragment>

The everyday case lives in the `preventDefault`-only cell: a form's submit handler.

```js
form.addEventListener('submit', (event) => {
  event.preventDefault();
  submitWithFetch(new FormData(form));
});
```

Without that first line, submitting the form triggers the browser's default action — a full-page navigation that reloads everything and throws away your JavaScript state. `preventDefault` suppresses exactly that, leaving the event free to keep traveling while your own code takes over the submission. It's the single most common use of either method. (Note what it does *not* touch: it won't disable the browser's own field validation or autofill — those aren't part of the default action you're cancelling.)

Now the warning that makes this section matter, and the reason it sits right after delegation:

:::caution
**`stopPropagation` quietly breaks delegation.** Your delegated handler lives on an *ancestor* and depends on events bubbling up to it. If some child handler calls `stopPropagation`, the event never reaches the ancestor — your delegated handler simply never runs, and there is no error to tell you why. The bug is invisible at the call site: the code that broke it is somewhere else entirely. An experienced engineer almost never reaches for `stopPropagation`; when you feel the urge, treat it as a sign to reconsider the design instead. `preventDefault` is almost always the method you actually wanted.
:::

## The addEventListener options object

You've already met one option — `{ capture: true }`. That third argument to `addEventListener` is a small options object, and it has four flags worth knowing. Two of them barely come up; two are 2026 reflexes. Here they are, weighted by how often you'll actually reach for them.

- **`capture: true`** — register on the capture phase instead of bubble. You met it above; it stays niche.
- **`once: true`** — the listener removes itself automatically after it fires the first time. Handy for genuine one-shot work: a setup step that should run on first interaction, or "first scroll" telemetry. You never have to remember to tear it down.
- **`passive: true`** — a *promise to the browser* that this handler will **not** call `preventDefault`. That promise lets the browser start scrolling or zooming immediately without waiting to see whether your handler blocks it — which is a real, measurable fix for scroll jank on touch and wheel events. Modern browsers already default `passive` to `true` for `wheel`, `mousewheel`, `touchstart`, and `touchmove` — but only on the document-level nodes (`window`, `document`, and `document.body`), and Safari doesn't apply even that. So the senior reflex is to **pass `{ passive: true }` explicitly on scroll, wheel, and touch listeners on any other element** — both because it's clearer and because the automatic default doesn't reach there. You only opt *out*, with `{ passive: false }`, in the rare case where you genuinely must `preventDefault` — a custom gesture surface that needs to block native scrolling, say.
- **`signal: AbortSignal`** — the 2026 cleanup reach, and the reason this section exists. You hand the listener a signal, and aborting that signal removes the listener. This is what replaces the old ceremony of matching every `addEventListener` with a `removeEventListener`. It's important enough to get its own section next.

Here's the shape, so the syntax is concrete before we lean on it:

```js
const onWheel = () => updateParallax();
const onFirstInteraction = () => trackEngagement();

panel.addEventListener('wheel', onWheel, { passive: true });
panel.addEventListener('pointerdown', onFirstInteraction, { once: true });
```

:::caution
A `passive: true` listener that calls `preventDefault()` anyway doesn't get its way — the call is **ignored**, and the browser logs a console warning telling you so. The promise you made wins. If you truly need to cancel the default, you must register with `{ passive: false }`.
:::

You'll notice that second listener uses `pointerdown`, not `mousedown` or `touchstart`. On a touchscreen, mouse and touch events both fire for a single tap, so listening for one input type means double-handling on the other. The `pointer*` family — `pointerdown`, `pointermove`, `pointerup` — unifies mouse, touch, and pen into one event, which is what you reach for whenever you need raw pointer input. (The family runs deeper than this; for now, just know it's the one to default to.)

## One AbortController per setup site: the 2026 cleanup reflex

Listeners have to be cleaned up. A listener you attach and never remove keeps its handler — and everything that handler closes over — alive in memory, and keeps firing after you stopped caring. In a long-lived single-page app, uncleaned listeners are a classic memory leak and a source of ghost behavior. So every listener you add in code that can be torn down needs a way to be removed.

The old way to do that has two failure modes that bite real codebases, and they're worth seeing before the fix, because they're *why* the fix exists. To remove a listener with `removeEventListener`, you must pass the **exact same function reference** you added. Compare the two approaches side by side — the same three listeners, torn down the worn way and the 2026 way:

<CodeVariants maxLines={16}>
  <CodeVariant label="The removeEventListener ceremony">
    <div data-mark-color="red">

    ```js {10-12}
    const onScroll = () => updateHeader();
    const onResize = () => recomputeLayout();
    const onKeydown = (event) => handleShortcut(event);

    window.addEventListener('scroll', onScroll);
    window.addEventListener('resize', onResize);
    document.addEventListener('keydown', onKeydown);

    const cleanup = () => {
      window.removeEventListener('scroll', onScroll);
      window.removeEventListener('resize', onResize);
      document.removeEventListener('keydown', onKeydown);
    };
    ```

    </div>
    **The worn pattern.** Every listener needs a named reference held until cleanup, and every `add` needs a hand-matched `remove`. Miss one pairing and that listener leaks. Worse, an inline arrow — `addEventListener('scroll', () => …)` — has no reference you can pass back, so it can *never* be removed.
  </CodeVariant>

  <CodeVariant label="One AbortController">
    <div data-mark-color="green">

    ```js {8} "signal"
    const controller = new AbortController();
    const { signal } = controller;

    window.addEventListener('scroll', () => updateHeader(), { signal });
    window.addEventListener('resize', () => recomputeLayout(), { signal });
    document.addEventListener('keydown', (event) => handleShortcut(event), { signal });

    const cleanup = () => controller.abort();
    ```

    </div>
    **The 2026 reflex.** One controller's signal goes into every listener — across different event types *and* different targets. A single `controller.abort()` removes all three at once. No references to hold, nothing to hand-match, and inline arrows are now perfectly fine because you never need their reference.
  </CodeVariant>
</CodeVariants>

So the two failure modes of the old way are concrete: an **anonymous handler can't be removed at all** (there's no reference to pass), so inline arrows silently leak; and **book-keeping N handlers across N targets by hand** is tedious and easy to under-clean — forget one pairing and that listener lingers.

The fix is one pattern, and it's the headline takeaway of this whole lesson. Four points:

1. **One controller per setup site.** `const controller = new AbortController();` at the top of wherever you wire listeners up.
2. **Thread its signal into every `addEventListener` in that scope.** Pass `{ signal: controller.signal }` to each one — and it works across multiple event types and multiple targets with the same single signal.
3. **One `controller.abort()` in the cleanup branch** removes every listener that shares the signal, all at once.
4. The mental model, in one line: **one signal, many listeners, one shutdown switch.** Fewer references to hold, fewer ways to leak, and anonymous handlers stop being a problem.

If that `AbortController`-and-signal shape looks familiar, it should — it's the same cancellation primitive used for aborting `fetch` requests and timing out async work. The course reaches for one consistent cancellation tool across the whole stack, and this is it applied to event listeners.

One forward glance, then we move on. In React, the cleanup branch isn't a `cleanup()` function you call by hand — it's a function React calls for you when a component unmounts. But the *shape inside it is identical*: create a controller when you set up, register listeners with its signal, call `abort()` on the way out. You're learning the exact pattern the React lesson will lean on; when you get there, this will already be muscle memory rather than something new.

## What React owns, and where you still write addEventListener

Let's close the loop the chapter keeps returning to: React abstracts the substrate, but it doesn't replace it — and knowing where the abstraction ends tells you exactly when to drop down to `addEventListener`.

For anything inside your component tree, React owns the listeners and you write `onClick`, `onChange`, `onSubmit` — never `addEventListener`. Under the hood, React attaches a small set of native listeners at the **root container** where your app is mounted (the element you render into — not `document`), normalizes each event into a <Term definition="React's cross-browser wrapper around the native DOM event. It exposes the same surface you already know — preventDefault, stopPropagation, target, currentTarget — so the substrate knowledge transfers directly.">`SyntheticEvent`</Term>, and dispatches it to the right component by walking the tree. That is the delegation pattern you just learned, run by the framework at scale. Most of the native surface carries straight through the wrapper: `preventDefault`, `stopPropagation`, `target`, and `currentTarget` all work on a `SyntheticEvent` the way they do on a native one — so everything in this lesson transfers. (One bit of old code you might run across: `event.persist()`. It does nothing in modern React — it was a workaround for an event-pooling optimization that no longer exists. If you see it, it's a no-op leftover.)

So when do you still reach for `addEventListener` in a React app? Three escape-hatch sites — and recognizing them is the actual skill, because they're where the substrate genuinely leaks through. Each one lives inside React's cleanup machinery, wired with the `AbortController` pattern you just learned.

<CardGrid>
  <Card title="Global targets" icon="window">
    `window` and `document` events — a global `keydown` shortcut, `resize`, `scroll` on `window`. There's no JSX element to hang these on; the target isn't in your tree.
  </Card>
  <Card title="Third-party DOM libraries" icon="puzzle">
    A library hands you a raw DOM node and its own events — Stripe Elements, a map, a charting lib. You wire its listeners imperatively, outside React's tree.
  </Card>
  <Card title="Non-JSX browser APIs" icon="comment">
    Event-emitting platform objects that aren't elements — `MediaQueryList`'s `change`, `BroadcastChannel`'s `message`, a `WebSocket`'s `message`. Event-driven, but not through the component tree.
  </Card>
</CardGrid>

And here's the bow on the whole lesson: each of those three sites is wired with the exact pattern you just practiced — one `AbortController`, its signal threaded into every `addEventListener`, `abort()` in the cleanup branch. The substrate lesson handed you React-readiness for free. You didn't learn raw events as a detour around React; you learned the thing React is built on, and the thing you'll still reach for at its edges.
