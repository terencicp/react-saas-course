Three features land on your desk in the same week. A CSV export takes twelve seconds to build on the server, and the product wants a progress bar that actually moves. A notifications panel should light up the instant something happens, not five seconds later when the next poll fires. And an LLM endpoint streams its answer one token at a time, the way every chat UI now does.

Reach for the `fetch` call you built in the last lesson and all three resist it. Not because the call is wrong — because they aren't the shape it was built for. A hardened `fetch` is a *request-response*: you send one request, you wait, you read one buffered answer, the round-trip closes. But none of these three is a round-trip. Each is a **server-to-client live update over HTTP** — the connection stays open and the server keeps writing. The export pushes progress as it computes. The notifications channel pushes whenever there's news. The LLM pushes tokens as it generates them. The answer arrives in pieces, or it never stops arriving.

That reframe is the whole lesson. You already own request-response. The new shape is the one where the server keeps talking, and the senior questions it raises are: what primitive carries the bytes, what shape rides the wire, how does the client read a response that isn't finished — and the decision that matters most, **when does polling beat a live stream, and a live stream beat a WebSocket?**

We'll build this in two halves, because the second is made of the first. The substrate half: a `Response` body was *always* a stream of bytes — the `.json()` and `.text()` from last lesson just hid that by reading it to the end. Pull the stream yourself instead and you read chunks as they land. The channel half: Server-Sent Events, which turns out not to be a new transport at all, just a thin text convention layered on that same stream. You'll read one SSE stream by hand — to see, with your own eyes, that it's only framed text over `response.body` — before you ever reach for the browser API that does it for you.

So the first move is to stop trusting `.json()` to hide the truth from you. The body was a stream all along.

## `Response.body` is a stream you can pull

Last lesson, you read every response body with a consumer like `response.json()` or `response.text()`. Look closely at what those actually do: they read the body *to completion* — every byte, start to finish — and only then resolve with the whole thing. That's **buffering**. For a two-kilobyte JSON row it's exactly right; the body is tiny and you want all of it before you do anything. But aim that same `.json()` at a twelve-second export and it betrays you. The promise won't resolve until the last byte of the last row arrives — twelve seconds of a blank screen, then everything at once. You wanted the first byte *now*.

The body wasn't hiding. `response.body` is a <Term definition="A pull-based stream of chunks. response.body is one; iterate it with for await to read chunks as they arrive.">`ReadableStream`</Term> — a pull-based stream of <Term definition="One network frame of a streamed body. Its size is set by the network, not your data, so it rarely lines up with a message.">chunks</Term>, where each chunk is a <Term definition="A fixed-length array of bytes. Each stream chunk is one; TextDecoder turns it into a string.">`Uint8Array`</Term> of raw bytes. The consumer methods were reading that stream for you and handing back the assembled result. Skip them, reach for `response.body` directly, and the stream is yours to pull one chunk at a time.

The 2026 reflex for pulling it is a `for await` loop:

```ts
const response = await fetch('/api/export');

for await (const chunk of response.body) {
  // chunk is a Uint8Array — one network frame, just arrived
}
```

A `ReadableStream` is async-iterable on the server runtimes this course touches — Node, the Edge runtime, the renderer behind a Server Component — and in browsers it's only just become universal: Safari was the last holdout and didn't ship it until version 26.4, so global support sits around three-quarters. On the server, or in a browser app that doesn't have to support older Safari, `for await` reads the stream directly. Each turn of the loop hands you exactly one chunk: the next slice of the body that has physically arrived over the network. You act on chunk one while chunk four is still in flight. That's the difference buffering hid. (When you do have to cover older Safari, reach for the `getReader()` loop below — it's the cross-browser-safe spelling of the same read.)

:::note
TypeScript types `response.body` as `ReadableStream | null` — `null` for the rare bodyless response, like a `204`. Production code guards `if (!response.body) return` before the loop; the snippets here skip that one line to keep the streaming mechanics in focus. In the same spirit, helpers like `handle`, `notificationSchema`, and the React setters in later snippets are unimported stand-ins, exactly as the last lesson left `Result`, `ok`, and the schema.
:::

That last frame plants a flag we'll come back to. A chunk's size is decided by the network — by packet sizes, buffering, and timing — not by your data. So a chunk almost never lines up with a unit of meaning. For now, just register it: **a chunk is a network frame, not a message.**

The other spelling you'll meet in existing code, inside libraries, and anywhere older Safari is still in scope: instead of `for await`, call `response.body.getReader()` and loop on `reader.read()`, which hands back `{ done, value }` each turn. It's the same stream, read the same way — more ceremony, and you have to remember to call `reader.releaseLock()` when you bail early, but it works in every browser regardless of version. So it's not merely legacy: it's the safe cross-browser choice when your support matrix still includes Safari before 26.4. Reach for `for await` when you can, and drop to `getReader()` when you must.

<details>
<summary>The `getReader()` loop, for reference and older Safari</summary>

```ts
const reader = response.body.getReader();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  // value is the Uint8Array chunk
}

reader.releaseLock();
```

`for await` does all of this for you, including releasing the lock when the loop ends. Prefer it where you can; reach for this form when you recognize it in the wild or when your browser support matrix still includes Safari before 26.4.

</details>

## Decoding chunks back to text

You're holding `Uint8Array` chunks — raw bytes — and what you almost always want is text. So you need the bridge between the two, and a detail that, if you miss it, ships a bug invisible to every test you'll write.

The bridge is a pair of platform objects. `TextEncoder` goes from string to bytes: `new TextEncoder().encode(s)` returns the UTF-8 bytes of a string. `TextDecoder` goes the other way: `new TextDecoder().decode(bytes)` returns the string those bytes spell. We'll use the decoder now to turn chunks into text, and the encoder later when the server writes bytes onto a stream. They're the only two byte primitives this lesson needs.

Here's the detail. UTF-8 encodes most characters in a single byte, but anything beyond plain ASCII — an accented `é`, an emoji, a CJK character — takes two, three, or four bytes. And the network doesn't know or care where a character's bytes begin and end. It can cut a chunk *in the middle* of a multi-byte character: the first two bytes of a three-byte `€` ride one chunk, the last byte rides the next. Decode that first chunk on its own and the decoder sees two bytes that don't spell a complete character, gives up, and emits the replacement character `�`. The euro sign is gone, corrupted on a chunk boundary the network chose at random.

The fix is one option:

```ts
const decoder = new TextDecoder();

for await (const chunk of response.body) {
  const text = decoder.decode(chunk, { stream: true });
  // text is safe — a split character is held back, not corrupted
}

const tail = decoder.decode(); // flush any held-back bytes
```

`{ stream: true }` tells the decoder it's mid-stream: when a chunk ends on an incomplete multi-byte sequence, hold those trailing bytes back instead of guessing, and prepend them to the front of the next chunk. The split `€` gets reassembled on the call that receives its final byte. The decoder *carries that state between calls*, which is why you create it once, above the loop, and reuse the same instance every iteration — a fresh decoder per chunk would have no memory of the held-back bytes. The final bare `decoder.decode()` with no argument flushes anything still buffered at the end, in case the stream's very last character was split. So the reflex: **always `{ stream: true }` on chunks, one reused decoder, one final flush.**

:::caution
Decode chunks *without* `{ stream: true }` and the happy path looks perfect — because your test data is pure ASCII, where every character is one byte and nothing ever splits. Then a real user's name has an `é` in it, or a message has an emoji, it lands on a chunk boundary, and renders as `�` in production. The bug is invisible until the exact byte lands on the exact edge. The flag is not optional.
:::

## From chunks to messages

The decoder gives you correct *text*. It does nothing about the problem the diagram flagged: chunks are network frames, and your application thinks in *messages*. One message can span two chunks. Two messages can land in one chunk. You cannot assume one chunk equals one message — the network never promised that, and it routinely breaks it.

So how do you read whole messages off a stream that hands you arbitrary fragments? There's one pattern, and it works for *any* framed protocol you'll ever read off a byte stream — it's the direct prerequisite for parsing SSE by hand in a moment. Named once so you recognize it forever: **accumulate the decoded text into a buffer; split the buffer on the protocol's framing token; the final fragment might be incomplete, so keep it as the tail and carry it into the next iteration; emit everything before the tail.** For the SSE we're about to meet, the framing token is the blank line — two newlines, `\n\n` — that marks the end of one event.

The naive version skips the tail, and it's worth seeing it fail before you trust the fix. Picture splitting *each chunk on its own* on `\n\n`. A chunk arrives carrying `event1\n\nevent2\npart` — one complete event and the start of a second. Split it and you get `["event1", "event2\npart"]`. You emit `event1`, fine — but then you emit `event2\npart` too, a half-event with no terminator yet, as if it were whole. The rest of event two is still coming in the *next* chunk, and you've already mangled it. The event spanning the boundary is corrupted, every time.

Buffer instead, and the boundary stops mattering. Watch the buffer and its tail move across two chunks.

That's the entire move: **append, split, keep the tail, parse the rest.** The tail is the one genuinely fiddly part — it's a fragment you're deliberately *not* emitting yet, holding until its other half arrives. Get that one pointer right and framed-stream parsing stops being mysterious. The two versions, side by side:

Notice this section never looked *inside* an event — it only cares about where one ends. Splitting bytes into whole messages and understanding what a message *says* are two separate jobs. The framing token is `\n\n`; what those bytes mean is the next section's problem. Keeping them apart is what keeps each piece simple.

## The Server-Sent Events wire format

Here's the payoff for pulling streams by hand. <Term definition="A one-way server→client convention over a long-lived text/event-stream HTTP response: lines of data: … separated by blank lines.">Server-Sent Events</Term> (SSE) — the thing your notifications panel and your LLM token stream both want — is not a new transport, not a binary protocol, not a library. It's a *convention*: an ordinary HTTP response with the content type `text/event-stream` that stays open, down which the server writes lines of UTF-8 text. The client reads the stream — exactly the way you just learned — and dispatches one event per blank-line-terminated block. That's the entire idea. Because it's just text over a normal HTTP response, it rides every piece of infrastructure HTTP already has: it works through CDNs and proxies, it multiplexes over HTTP/2, and it carries your app's cookies for authentication with nothing extra. No new wire to stand up.

The text follows a tiny grammar. Each event is one or more `key: value` lines, and a blank line ends it. The block below is a literal slice of an SSE stream — every byte the server actually sent. Hover the fields to see what each one does.

<CodeTooltips tooltips={{
  'event:': 'Optional event type. Without it, the event is delivered as the default "message" type. With it, the client listens for this name specifically.',
  'id:': 'Optional event id. The browser remembers the last one it saw and replays it on reconnect via the Last-Event-ID request header — this is what makes SSE resume after a dropped connection.',
  'retry:': 'Optional reconnect-delay hint in milliseconds. Tells the browser how long to wait before reconnecting if the stream drops.',
}}>
```text
data: {"progress":10}

event: notification
id: 42
data: {"title":"Invoice paid"}

retry: 3000
data: {"progress":20}
```
</CodeTooltips>

Read it top to bottom. The first event is bare: a `data:` line with a JSON payload, then a blank line to terminate it. With no `event:` field, the client receives it as the default type, `message`. The second event is *named* — `event: notification` — so a client can listen for `notification` events specifically rather than lumping everything into the default; it also carries `id: 42`. That `id:` is SSE's quiet superpower: the browser remembers the last id it saw, and if the connection drops it sends that id back in a <Term definition="The request header the browser sends on SSE reconnect, carrying the last id: it saw, so the server can resume from there.">`Last-Event-ID`</Term> header on reconnect, so the server can resume from event 43 instead of replaying from the top. Resilience, for the cost of numbering your events. The `retry:` field just hints how long the browser should wait before reconnecting — recognize it; you'll rarely set it.

One rule governs writing the `data:` payload, and breaking it corrupts the stream. Each `data:` line is *one logical line* — a literal newline inside the payload starts what the protocol reads as a second `data:` line, and a blank line inside it terminates the event early. So you never write a raw multi-line string. You stringify:

```ts
const frame = `data: ${JSON.stringify(payload)}\n\n`;
```

`JSON.stringify` escapes any newline in the payload to `\n` *inside* the string, collapsing it to a single safe line, and you add the `\n\n` terminator yourself. The reflex: **one `JSON.stringify`'d `data:` line per event, terminated by a blank line.** Send an un-stringified payload with a newline in it and the reader either sees two `data:` lines or hits a premature terminator — the framing breaks on data you didn't sanitize.

:::note
You'll meet this exact shape again much later in the course. The streaming chat protocol the Vercel AI SDK speaks — the one behind every `streamText` token feed — is SSE underneath, this same `data:`-line convention with typed deltas layered on top. The LLM token stream from the intro isn't a parallel transport; it's this.
:::

## Emitting an SSE stream from a Route Handler

Now you write the server side. In Next.js, a streaming response lives in a **Route Handler** — a file at `app/api/<path>/route.ts` that exports a function per HTTP method. The handler returns a `Response`, and for SSE that `Response` wraps a `ReadableStream` you write events into.

:::note
You're seeing the bare bones of a Route Handler here — `export async function GET(request)` and a returned `Response`, nothing more. The full production shape (validating params and bodies with schemas, the auth wrapper, the status-code conventions) is owned by a later chapter on route handlers. Here the stream is the whole point, so everything else is stripped to the minimum.
:::

The handler builds a `new ReadableStream` whose `start(controller)` function does the writing: `controller.enqueue(...)` pushes a chunk onto the stream, `controller.close()` ends it. And here the encoder from earlier earns its place — a `ReadableStream` carries *bytes*, not strings, so every SSE frame goes through `TextEncoder` on its way out. It's the exact inverse of the decoding you did on the client: the server encodes text to bytes to send; the client decodes bytes to text to read.

Step through the handler. It streams a few progress updates, then closes — and cleans up if the client leaves first.

<AnnotatedCode lang="ts" maxLines={18} code={`
export async function GET(request: Request) {
  const encoder = new TextEncoder();

  const stream = new ReadableStream({
    start(controller) {
      let progress = 0;
      const timer = setInterval(() => {
        progress += 10;
        const frame = \`data: \${JSON.stringify({ progress })}\\n\\n\`;
        controller.enqueue(encoder.encode(frame));
        if (progress >= 100) {
          clearInterval(timer);
          controller.close();
        }
      }, 1_000);

      request.signal.addEventListener('abort', () => {
        clearInterval(timer);
        controller.close();
      });
    },
  });

  return new Response(stream, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache, no-transform',
      Connection: 'keep-alive',
    },
  });
}
`}>

  </AnnotatedStep>

  <AnnotatedStep meta="{2}" color="blue">
    Create one `TextEncoder` for the whole stream. Everything you enqueue must be bytes, and this is the string-to-bytes bridge, the mirror of the decoder on the client.
  </AnnotatedStep>

  <AnnotatedStep meta="{4-5}" color="blue">
    `new ReadableStream` with a `start(controller)` function. `start` runs once when the stream is read, and the `controller` is your handle for writing into it — `enqueue` to push a chunk, `close` to end it.
  </AnnotatedStep>

  <AnnotatedStep meta="{9-10}" color="green">
    The write itself. Build one SSE frame — `data:` + the JSON-stringified payload + the `\n\n` terminator — encode it to bytes, and `enqueue` it. This is the one-line-per-event reflex from the wire-format section, now firing once a second.
  </AnnotatedStep>

  <AnnotatedStep meta="{17-20}" color="orange">
    Cleanup on disconnect. Subscribe to `request.signal`'s `abort` event; when the client closes the tab, clear the interval and close the controller. Skip this and every client who navigates away leaves a `setInterval` — and, in a real handler, an open database cursor — running forever on your server. One leaked timer per lost client.
  </AnnotatedStep>

  <AnnotatedStep meta="{24-29}" color="violet">
    Return the stream as a `Response` with three headers. `Content-Type: text/event-stream` is the protocol opt-in — browsers and proxies key on it. `Connection: keep-alive` matters only on HTTP/1.1 (HTTP/2 and 3 ignore it). And `Cache-Control: no-cache, no-transform` carries the load-bearing token.
  </AnnotatedStep>
</AnnotatedCode>

That `no-transform` deserves a full stop on its own, because forgetting it is the single most common way SSE breaks *only in production*. A CDN or proxy sitting between your server and the user is allowed, by default, to *buffer* a response — hold the bytes and optimize delivery. Do that to a stream and you've destroyed the entire point: the proxy collects all your events, waits for the stream to end, and delivers one twelve-second-late blob. It worked perfectly on `localhost` because there was no proxy in the way. `no-transform` is the explicit instruction to every middlebox: do not buffer, do not touch, pass these bytes through as they arrive.

There's one more guard the code implies but worth stating outright: once you've called `controller.close()` — or once the abort handler has — calling `enqueue` again *throws*. So in a longer-lived handler you guard every write behind an "is this still open" check, the same way the abort listener stops the timer before anything else can fire.

Order the construction. The handler is fixed above the steps; drag the build order into place.

## Consuming SSE with `EventSource`

You read one SSE stream by hand to prove it's just framed text. Now meet the browser API that does the framing for you, because for most consuming you'll reach for it: <Term definition="The browser API that opens an SSE connection and auto-reconnects, replaying the last event id.">`EventSource`</Term>. You give it a URL, and it opens the connection, reads the stream, parses the `data:`/`event:`/`id:` grammar, and dispatches a clean event object per message — none of the buffer-and-tail loop, because it owns that internally.

The surface is small. `new EventSource(url)` opens it. `source.onmessage` fires for default (unnamed) events, with the raw payload string on `event.data`. `source.addEventListener('<name>', ...)` fires for events that carried a matching `event:` field. `source.close()` shuts it down. And the payload is still wire data, so the reflex from last lesson holds: you `JSON.parse` it and validate the shape before you trust it — never trust the wire, even when the wire is your own server.

The free win that makes `EventSource` the default: it *auto-reconnects*. If the connection drops, the browser reopens it on its own and sends back the last `id:` it saw in the `Last-Event-ID` header — so a server that numbers its events can resume exactly where it left off. You get resilient streaming for nothing.

Inside React, the lifecycle has one correct home. The subscription is a side effect with a setup and a teardown, so it lives in a `useEffect`, and the cleanup function closes the connection — the same setup-then-teardown discipline as the `AbortSignal` from last lesson. Step through it:

<AnnotatedCode lang="tsx" maxLines={16} code={`
useEffect(() => {

## When `EventSource` runs out and plain `fetch` takes over

`EventSource` is the default, but it has a hard ceiling, and one common requirement walks straight into it. Three limits, all baked into the API:

- It can't set custom request headers — no `Authorization`, no CSRF token.
- It can only issue a `GET`.
- It can't carry a request body.

Most of the time none of that bites, because the limit that matters most — authentication — has a clean answer for the common case. Browser SSE to your own server authenticates with **cookies**, and cookies travel on an `EventSource` request automatically (with `withCredentials: true` for the cross-origin case). So cookie-authenticated SSE works with `EventSource` out of the box. It's **bearer-token** auth that breaks it: there is simply no way to attach an `Authorization` header to an `EventSource`, so a stream behind a bearer token can't use the browser API at all.

When a hard limit bites, you already know the escape hatch — you built it. The response is still a `text/event-stream`, the exact same SSE byte protocol. So you consume it with plain `fetch` instead: set whatever headers you need, then read `response.body` and reframe the events yourself with the **append-split-keep-tail** loop from two sections ago. That loop wasn't an academic exercise; it's the thing you reach for the moment `EventSource` can't carry your auth. SSE is just framed text, and the framing token is `\n\n`.

<CodeVariants>
  <CodeVariant label="EventSource (default)">
    <div data-mark-color="green">

    ```ts {2}
    // Cookie-authed SSE — cookies ride along automatically.
    const source = new EventSource('/api/notifications');
    source.onmessage = (event) => handle(JSON.parse(event.data));
    ```

    </div>
    **Reach for this first.** Zero parsing code, and you get auto-reconnect with `Last-Event-ID` replay for free. The default for any server-to-client stream your own app authenticates with cookies.
  </CodeVariant>

  <CodeVariant label="fetch consumer (custom headers)">
    <div data-mark-color="orange">

    ```ts {1-3}
    const response = await fetch('/api/notifications', {
      headers: { Authorization: `Bearer ${token}` },
    });

    let buffer = '';
    for await (const chunk of response.body) {
      buffer += decoder.decode(chunk, { stream: true });
      const events = buffer.split('\n\n');
      buffer = events.pop() ?? '';
      for (const event of events) {
        handle(event);
      }
    }
    ```

    </div>
    **Drop to this only when a hard limit bites** — a bearer token, a non-`GET`, a request body. Same SSE protocol, but you reframe events by hand, and you give up the automatic reconnect and `Last-Event-ID` replay — you'd reimplement those yourself. That cost is exactly why `EventSource` stays the default. (This runs in the browser, so on a support matrix that still includes Safari before 26.4, swap the `for await` for the `getReader()` loop.)
  </CodeVariant>
</CodeVariants>

## Choosing the channel: polling, SSE, or WebSockets

You can now read a stream, emit one, and consume one. The last thing — and the one you'll carry longest — isn't a technique. It's a decision: of the three ways to push updates to a client, which does this feature actually need? Juniors reach for the most powerful one first and over-build. An experienced engineer asks the questions in a fixed order and lets the answer fall out, because each option costs more than the last.

Three options, in the order you consider them:

**Polling** is the default, and it's the one to start every "real-time" feature on. You make a normal request-response call on an interval — `setInterval`, or a polling option on a data-fetching library you'll meet later in the course — and re-fetch on a cadence. No open connections, no new infrastructure, works through every layer you already have. The trigger to leave it: the freshness the product needs is *tighter* than a cadence your server can sustain, or the fan-out — clients times cadence — becomes a load problem of its own. The 2026 reality is blunt: most SaaS features that *feel* real-time are five-to-thirty-second polling, and never need anything more.

**SSE** is the conditional reach *past* polling. You take it when the channel is server-to-client only, the payload is JSON or text, and one persistent HTTP connection per subscribed tab is a budget you're happy to pay. That's your three opening features exactly: the export progress bar, the notifications panel, the LLM token stream, a live order status. Its whole appeal is that it isn't a new transport — it rides existing HTTP infrastructure, stays CDN-friendly with `no-transform`, multiplexes over HTTP/2, and reuses your cookie auth.

**WebSockets** are the conditional reach *past* SSE, and only one thing takes you there: the channel has to be **bidirectional** — the client needs to send on the same live connection it receives on. Collaborative cursors, chat with typing indicators and delivery acknowledgements, a multiplayer canvas. These are a genuinely different transport: a separate connection model, no HTTP cache, its own authentication handshake. We name the trigger so you recognize the line being crossed — the API itself is out of scope for this course. The signal that you've crossed it is simple: the moment a feature needs the client to *push* over the live channel, SSE is no longer enough.

Walk the decision. Pick a branch at each step.

  </Leaf>

  </Leaf>

  <Leaf id="leaf-websockets" verdict="WebSockets — recognition only here">

The value is the *order*. Direction before frequency. Ask "can the client only receive?" first, and most features answer "yes" and never get near a WebSocket. Ask "how fresh?" second, and most of *those* answer "a few seconds is fine" and stay on polling. The powerful tools sit at the end of the funnel on purpose.

## What this substrate is not for

Two features look like they belong on a stream and don't. Naming them now keeps you from reaching wrong later.

:::caution[Large file downloads aren't SSE]
A big download streams *bytes*, not *events*. You read it with `response.blob()`, or pipe `response.body` to a save destination — there's no `data:` framing, no event grammar, none of this section's machinery. The file-handling chapters later in the course own it; recognize that it's a different job that happens to share the word "stream."
:::

:::caution[Long-running work is a job, not a stream]
SSE is for *bounded, timely* server-to-client updates — seconds, not minutes. Hold a connection open for the length of a long job and it turns fragile across proxies, mobile networks, and serverless platforms, all of which cap how long a stream may stay open. (Vercel's Fluid Compute, for instance, allows up to five minutes by default and sixty seconds on the Hobby plan — bounded either way, and billed for the whole time the connection is held.) Work that runs for minutes belongs on a background-job queue, which you'll meet later in the course. An SSE stream pushes progress *from* a job; it is not a substitute for one.
:::
