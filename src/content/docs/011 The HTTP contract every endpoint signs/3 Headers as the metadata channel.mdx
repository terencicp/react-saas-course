A header is not a free-form key-value pair the server tacks onto a response. It is *metadata addressed to a specific audience* — the browser, an infrastructure layer between client and server, or the application itself — and that audience reads it and acts on it before the body is ever touched. Pick the wrong header, or the wrong value, and the audience does something you didn't intend: a CDN caches a private response and replays it to the next user, a browser refuses your cookie, a rate-limit gateway lets a forged client through.

This lesson lands the three-audience model as the durable scaffold, walks the header families a SaaS engineer actually touches (content negotiation, caching, conditional requests, authorization, rate-limit signaling, the security baseline, request context, custom headers), and closes on a concrete map from header-family to the Next.js file that sets it. By the end you'll place any header into one of three audiences, pick the senior default for the common cases, and know exactly where in a Next.js app a given header needs to live.

## Headers have three audiences

The previous two lessons named the shape of an HTTP message in passing: a method or status line, a block of headers, and a body. The body carries the resource. The headers carry the metadata. The question this section answers is the one an experienced engineer asks before writing a header value: *who's going to read this?*

Every header is sent because someone will read it. That someone is one of three audiences:

- The **browser**, which enforces the header — it sets a cookie, applies a Content Security Policy, refuses an insecure transport, denies camera access to a third-party iframe.
- The **infrastructure layer** between client and server (CDN, reverse proxy, load balancer), which acts on the header — it caches a response, picks a compression codec, throttles a noisy client.
- The **application**, which reads the header to make a decision — which user is signing this request, what content type to return, whether this is a retry of an operation it has already done.

The audience is the durable property of the header. Once you know who reads it, you know where in your stack to set it, and you know what the consequence of getting it wrong is. The catalog of headers a SaaS engineer touches is wide, but the catalog is not what carries you — the audience model is. Students who memorise headers lose them inside a year; engineers who internalise the audiences can place new ones they meet later by reasoning.

A few headers blur the boundary, and that's fine. `Cache-Control` is mostly an infrastructure header — CDNs and shared caches are the design audience — but the browser also caches and reads it. `Authorization` is application-audience, but the infrastructure layer might still log it. Don't get stuck on edge cases; the mapping is one-to-one *in practice* for every header you'll touch, and the few boundary cases are called out where they come up.

## Content negotiation: what the body is, what the client wants

Three headers, one rule: name the format twice — once on the request side ("what I'm sending, what I want back") and once on the response side ("what you're actually getting").

`Content-Type` describes what the *current* body *is*. On a request: `Content-Type: application/json` says "the bytes after the headers are JSON." On a response: `Content-Type: application/problem+json; charset=utf-8` says "the body is RFC 9457 Problem Details, encoded as UTF-8." Each direction names its own body — the request `Content-Type` and the response `Content-Type` are independent.

`Accept` is the request-side header saying what the client *wants* in the response. A SaaS API client sends `Accept: application/json`. A browser navigating to an HTML page sends a longer list — `Accept: text/html,application/xhtml+xml,...` — telling the server which formats it can render. `Content-Type` and `Accept` are not the same axis: a POST request can carry `Content-Type: application/json` *and* `Accept: application/problem+json` at the same time, meaning "I'm sending JSON; if you reject me, send the error as Problem Details." Both can be present; they answer different questions.

`Content-Encoding` declares how the body was compressed before being put on the wire. The client advertises support via `Accept-Encoding`; the server picks one and stamps the response with `Content-Encoding`. The 2026 senior reach for new traffic is `zstd` — Chrome, Edge, Firefox, and Safari all support it now, Cloudflare and the other major CDNs negotiate it by default, and the bytes-per-CPU ratio is materially better than Brotli for the same fidelity. Brotli stays the cross-cutting default for older client support; gzip is the legacy fallback. You almost never set `Content-Encoding` by hand — the platform (Next.js plus the deploy target) negotiates compression and stamps the encoding for you. Setting it manually without actually compressing the body is a content-mismatch bug downstream tooling will choke on.

`Vary` is the cache-key extender. By default, a shared cache keys responses by URL. `Vary: Accept-Encoding, Cookie` tells it: "this response varies by the values of these request headers; key the cache on them too." Without `Vary: Cookie` on a per-user response, the CDN serves user A's response to user B from cache. Without `Vary: Accept-Encoding`, the Brotli body gets handed to a client that only asked for gzip and can't decompress it. The 2026 reflex: any response that varies meaningfully by request header sets `Vary`. The senior default for authenticated HTML (next section) is `private, no-store`, which makes `Vary: Cookie` moot — the response isn't cached at all — but on cacheable per-user surfaces, `Vary` is mandatory.

Two more in passing. `Content-Length` declares the body's size in bytes — the infrastructure layer reads it to frame the message, allocate buffers, and decide when the body has finished arriving; the platform sets it for you when it serializes the response, so it almost never appears in your code. `Accept-Language` carries the client's preferred locales — Unit 17's `next-intl` reads it for locale negotiation. Mention; don't expand.

Here's what content negotiation looks like on the wire. A request asks for JSON with any of three encodings, plus a cookie for identity; the response answers with JSON, picks `zstd`, and varies by both encoding and cookie so the CDN keys correctly.

```http
GET /api/invoices HTTP/3
Accept: application/json
Accept-Encoding: zstd, br, gzip
Cookie: session=abc...
```

```http
HTTP/3 200 OK
Content-Type: application/json
Content-Encoding: zstd
Vary: Accept-Encoding, Cookie
```

## Cache-Control and the senior defaults

`Cache-Control` is the header an infrastructure layer looks at first. Its value is a comma-separated list of directives, each one a knob on what caches are allowed to do with the response. The list of directives is short — six or seven worth knowing — and four senior defaults will cover almost every response you ever ship.

The directives a SaaS engineer reaches for:

- `private` vs. `public` — `private` means only the client's own cache (the browser) may store the response; `public` lets shared caches (CDNs, corporate proxies) store it too. Authenticated responses are `private`; anonymous responses can be `public`.
- `max-age=N` — how long, in seconds, the response stays fresh and may be served without revalidation.
- `s-maxage=N` — same as `max-age` but applies only to *shared* caches. Useful when you want the CDN to keep the response longer than the browser does.
- `no-store` vs. `no-cache` — **not synonyms**. `no-store` forbids any cache from storing the response at all. `no-cache` *allows* storage but forces revalidation (a conditional request — next section) on every read. The default-confusion of these two is the most common cache bug in production.
- `must-revalidate` — once a response is stale, it must be revalidated; intermediaries may not serve it stale as a fallback.
- `stale-while-revalidate=N` — once stale, the cache may serve the stale response and asynchronously fetch a fresh one. Keeps p99 latency low at the cost of one window of stale.
- `immutable` — the response will never change at this URL. Browsers may skip revalidation entirely, even on reload. The marker for hashed static assets.

Four defaults cover almost every response. Each one names the trigger and the directive string verbatim.

- **Authenticated HTML** (any page that varies by signed-in user): `Cache-Control: private, no-store`. No shared cache may touch it; the browser must not store it either, so the back-button doesn't flash signed-in content after sign-out.
- **Hashed static assets** (filename includes a content hash, like `/_next/static/chunks/main.a1b2c3.js`): `Cache-Control: public, max-age=31536000, immutable`. One year. New deploys ship new hashes — the URL itself is the version.
- **Cacheable HTML on the CDN edge** (marketing pages, blog posts, anything anonymous): `Cache-Control: s-maxage=300, stale-while-revalidate=86400`. Five minutes fresh on the CDN, a day of stale-while-revalidate behind it. The user always sees a fast response; the CDN refreshes in the background.
- **API responses that change rapidly on a logged-in surface**: `Cache-Control: private, no-store`. Same default as authenticated HTML — anything the server can't guarantee won't change between two reads.

Next.js 16 wraps all of this. The framework-level abstraction is the `'use cache'` directive paired with `cacheLife(...)` and `cacheTag(...)` — you don't usually write `Cache-Control` strings by hand. The directives in this section are the underlying HTTP contract; `'use cache'` is the wrapper that emits them. The detail lands in Unit 4. For now, recognise the wire shape so you can read it in DevTools and know what the framework produced.

:::caution
If you ship `Cache-Control: public, max-age=...` on a response that varies by signed-in user, a shared cache — Vercel's edge, Cloudflare, a corporate proxy — can store one user's response and serve it to another. This is the single worst cache header bug; one wrong directive leaks one user's data into another user's browser. The senior default for authenticated HTML is `private, no-store`. Reach for `public` only when you can prove the response is identical for every user, signed in or out.
:::

The two directives the trap usually involves are `no-store` and `no-cache`. They sound interchangeable; they aren't. Here's the difference on the wire.

## Conditional requests: ETag and If-None-Match

The conditional-request pair completes the caching story. It's the answer to the question "the response *might* be cached — has the resource actually changed?"

The pattern is small. The server stamps the response with `ETag: "v1"` — an opaque token identifying *this* version of the resource. On the next read, the client sends `If-None-Match: "v1"` in the request. If the resource hasn't changed, the server replies with `304 Not Modified` and no body. The client uses its cached copy. The savings are the body bytes — the round-trip is still paid, but the payload isn't.

Reach for it on read-mostly resources the client revisits: list endpoints, large JSON payloads, image metadata — anything that doesn't change often enough to justify re-shipping the body. The pattern fits naturally with `Cache-Control: no-cache`, where every read is conditional by design.

Two related headers earn one mention each. <Term definition="Pattern: server stamps an ETag on the response; client sends If-Match on a mutation; server replies 412 Precondition Failed if the resource has moved on. The write-side cousin of If-None-Match.">`If-Match`</Term> on a write request implements *optimistic concurrency* — "apply this mutation only if the resource is still at version v1; otherwise return `412 Precondition Failed` and let me reconcile." Useful for collaborative-edit surfaces; not the daily reach. `Last-Modified` and `If-Modified-Since` are the timestamp version of the same pattern. The 2026 senior default is `ETag` because timestamps have second-precision and can't disambiguate two writes inside the same second; `ETag` values are opaque to the client and the server can hash whatever shape it likes — row version, content hash, monotonic counter.

The exchange looks like this on the wire.

```http
GET /api/invoices HTTP/3
If-None-Match: "v1"
```

```http
HTTP/3 304 Not Modified
ETag: "v1"
```

No body on the 304. The client renders from its existing cache.

## Authorization: cookies for browsers, Bearer for machines

The first decision an application makes on every incoming request is "who is this?" — and the answer rides on either a cookie or an `Authorization` header. The two are not interchangeable, and which one to use is a decision shaped entirely by who's calling.

For first-party browser traffic, the answer is **cookies**. The browser sets a cookie at sign-in via `Set-Cookie:` on the response, sends it back automatically via `Cookie:` on every same-origin request, and clears it at sign-out. The browser also enforces protections cookies have accumulated over thirty years: `HttpOnly` keeps JavaScript from reading the cookie value, `Secure` requires HTTPS, `SameSite=Lax` defangs CSRF on top-level navigations, the `__Host-` prefix locks the cookie to one origin. Those mitigations are why first-party browser sessions live in cookies and not in a header your JavaScript writes by hand.

For programmatic clients — a mobile app, a server-to-server SDK, a public API consumer — the answer is `Authorization: Bearer <token>`. A bearer token is a raw credential carried in plaintext on every request. It has none of the browser-managed mitigations cookies have, which is exactly why it's wrong for browser sessions — and exactly *right* for clients that manage their own token lifecycle, store credentials in a vault or a keychain, and don't have a cookie store at all.

The senior 2026 default in this course is the split named directly: Better Auth (Unit 8) ships the project's session as a `__Host-`-prefixed `HttpOnly; Secure; SameSite=Lax; Path=/` cookie. Public route handlers called by non-browser clients read `Authorization: Bearer <token>` instead. The two coexist inside a single SaaS — pick by who's calling the endpoint, not by personal preference.

Two cousins worth one sentence each. `Authorization: Basic` is Base64-encoded `username:password`. Only safe over HTTPS, and in 2026 only used for internal tooling, admin scripts, and the occasional CI/CD job; never user-facing. `WWW-Authenticate` is the companion header on a 401 response that tells the client which authentication scheme the server expects (`WWW-Authenticate: Bearer`). The `authedRoute` wrapper in Chapter 57 sets it on every 401 from a route handler.

:::caution
The `Authorization` header is logged by most observability tools by default. It contains a long-lived credential. Configure your logger's redaction list to strip `Authorization` and `Cookie` before any request enters a log line. Chapter 81 covers the redaction seam.
:::

The cookie deep dive — `HttpOnly`, `Secure`, `SameSite`, `__Host-`, `Partitioned`, and the full `Set-Cookie` attribute surface — lives in Chapter 13. The two channels look like this side by side on the wire.

## Rate-limit signaling and Retry-After

When a server tells the client "slow down," it does so in two ways: by returning the `429 Too Many Requests` status code from the previous lesson's subset, and by spelling out *how much* slowing down with two headers.

The pair is `RateLimit` (the current state) and `RateLimit-Policy` (the policy that produced it). These are the response headers the IETF httpapi working group has been standardising as `draft-ietf-httpapi-ratelimit-headers`, currently at draft 11 and intended Standards Track. They are *structured fields* per RFC 9651 — lists of items with parameters, parsed by a defined grammar — not the loose `X-RateLimit-*` set the older Heroku and Twitter shape used. The 2026 senior reach is the IETF draft shape: Upstash, Cloudflare, and most modern gateways emit it.

`Retry-After` is the back-off signal the HTTP client is required to honor. The value is either a number of seconds or an HTTP-date. It appears on `429 Too Many Requests` from a rate limiter and on `503 Service Unavailable` from a load balancer telling you "I'm overloaded; come back later."

The senior reflex on the client side: on a 429, read `Retry-After` and schedule a retry. The retry must be idempotency-aware — retrying a POST without an `Idempotency-Key` is the bug the methods lesson closed. On a 503, read `Retry-After` and either retry transparently or surface a "service unavailable" message, depending on what the UX needs. Either way, you don't pick the wait time yourself; the server told you.

A 429 with all three headers looks like this on the wire.

```http
HTTP/3 429 Too Many Requests
RateLimit: "auth-sign-in";r=0;t=27
RateLimit-Policy: "auth-sign-in";q=10;w=60
Retry-After: 27
Content-Type: application/problem+json
```

Read the structured-field values: the policy named `auth-sign-in` allows 10 requests (`q=10`) per 60-second window (`w=60`); the current state on this policy is zero remaining (`r=0`) with 27 seconds to reset (`t=27`). `Retry-After: 27` says the same thing in the form every HTTP client library already reads. The `application/problem+json` body explains *why* in human terms — RFC 9457 Problem Details, from the previous lesson.

The helper that emits these headers on every limited response in your app — `rate-limit-headers.ts` — lands in Chapter 75. This lesson names the wire shape so you can recognise it.

## The security baseline: the irreducible six

Six headers form the security baseline every SaaS app sends. They are all browser-audience — the browser enforces them, the infrastructure layer doesn't touch them, the application produces them at response time. This lesson names the six and the role each plays; Chapter 81 wires them in production with the CSP nonce, the report-only rollout, and the `Reporting-Endpoints` companion. The payoff here is recognition: see one of these in DevTools and know what it's there for.

- **`Strict-Transport-Security`** (HSTS) — "always use <Term definition="HTTPS — HTTP over TLS. The 2026 default transport for every public endpoint. HSTS enforces it for a configured window.">HTTPS</Term> for this host, for the next N seconds, including subdomains." Closes the downgrade-to-HTTP MITM window an attacker uses to strip TLS.
- **<Term definition="Content-Security-Policy. Response header telling the browser which sources may execute scripts, load styles, frame the page, etc. The 2026 senior shape is nonce-based with 'strict-dynamic'.">`Content-Security-Policy`</Term>** (CSP) — "only execute scripts from these sources." The 2026 senior shape is nonce-based with `'strict-dynamic'`: every request gets a fresh <Term definition="A one-time random value generated per request. In CSP, the server emits a nonce in the header and the same nonce on each <script> tag it trusts; the browser executes only matching scripts.">nonce</Term>, the inline `<script>` tags carry it, and other scripts inherit trust from nonced scripts. Closes XSS as a class.
- **`X-Content-Type-Options: nosniff`** — "do not sniff the body to guess a `Content-Type`; trust the header." Closes the MIME-confusion attack where a `.txt` upload is sniffed as `text/html` and executed in a browser context.
- **`Referrer-Policy`** — "what to put in `Referer` on outgoing navigations." The 2026 senior default is `strict-origin-when-cross-origin`: full URL on same-origin, origin-only cross-origin, nothing on HTTPS-to-HTTP. Closes the credential-in-URL leak through referer headers.
- **`Permissions-Policy`** — "which browser features (camera, microphone, geolocation, USB, payment) this origin and its iframes may use." The senior default is deny-by-default with per-feature grants only for the features the app actually needs.
- **`frame-ancestors`** — a CSP directive, not a separate header: "who may embed this page in an iframe." In 2026 this replaces the old `X-Frame-Options` header — `X-Frame-Options` is retired because the CSP directive composes with the rest of CSP and `X-Frame-Options` doesn't.

Pair the six with `Reporting-Endpoints` — a companion response header that names URLs the browser can POST CSP and Permissions-Policy violation reports to. The lesson names it; Chapter 81 wires the report-only rollout and the endpoint on the receiving end.

All six are browser-audience and almost entirely static across requests, so most of them live in `next.config.ts` — the build-time setter. The one that needs per-request data is CSP, because the nonce has to be fresh per request. The closing section ties this back to where each header is set in the file system.

## Request-context headers and the trust-the-edge rule

Every incoming request carries a small set of headers the application reads to figure out the context — who's calling, from where, through what. Five worth naming.

- `Cookie:` — the value the browser sends. Better Auth's session middleware (Unit 8) reads it. The application reads it; the infrastructure layer doesn't.
- `User-Agent:` — what client is calling. Forensic and analytics signal only; never trust the value for an authorization decision because the client controls it. Useful for "is this a bot, a mobile browser, a server-to-server SDK" segmentation in logs.
- `Referer:` — the previous URL the user navigated from. (The header has been misspelled since RFC 1945 in 1996; the field name is canonical now. The *policy* header that limits it, `Referrer-Policy`, uses the correct spelling. The spec uses both, depending on which one you're reading.) The application reads this for analytics.
- `Origin:` — the load-bearing header for CSRF defense on Server Actions. On any cross-origin request that includes credentials, the browser sets `Origin:` to the calling origin. Server Actions check this against the app's own origin and reject mismatches. Chapter 54 owns the check.
- `Host:` (and the equivalent `:authority` pseudo-header under HTTP/2 and HTTP/3) — which hostname the client meant. The application reads it for multi-tenant subdomain routing (org A on `acme.app.com`, org B on `globex.app.com`); the infrastructure layer routes on it.

That's the easy half. The hard half is recovering the *real* client IP through a chain of proxies — and recovering it without trusting a value the client could have forged.

The setup: a request typically passes through several hops before hitting your application — CDN, load balancer, application. Your application sees only the load balancer's IP as the source IP. To recover the real client IP, each proxy appends its known-good upstream IP to `X-Forwarded-For` (or to the structured `Forwarded` header standardised in RFC 7239). The application then walks the chain.

The trap: only the rightmost entry — the one appended by the immediate edge your application talks to — is trustworthy. Every entry to the left of that hop could have been forged by the original client by sending a fake `X-Forwarded-For` header in the original request. If you rate-limit by the leftmost entry, you've given any attacker a free way to spoof their IP and bypass your limiter.

The fix is to count proxies from the right and stop at the first untrusted hop. On Vercel, that's `request.headers.get('x-vercel-forwarded-for')` — Vercel's edge appends a value it controls, and that's the entry you trust. On a custom edge, configure the equivalent.

`Forwarded` (RFC 7239) is the standardised replacement for the older `X-Forwarded-*` set: one structured header (`Forwarded: for=192.0.2.60;proto=https;by=203.0.113.43`) instead of three separate `X-Forwarded-For`, `X-Forwarded-Proto`, and `X-Forwarded-Host`. The senior 2026 default is to emit `Forwarded` from your edge; the legacy `X-Forwarded-*` set persists because most upstream tooling still reads it. Read whichever your edge actually sets; configure the edge to set the modern one.

:::caution
Reading `X-Forwarded-For` from the request without verifying which hop appended each entry is the most common spoofable-header bug in production rate limiters. The leftmost entry is whatever the original client sent — including a fake one. Count proxies from the right and stop at the first untrusted hop, or read the value your trusted edge appended under a vendor-specific name (`x-vercel-forwarded-for` on Vercel, similar on Cloudflare and Fastly).
:::

## Custom-header naming after RFC 6648

When you invent a custom header — for a webhook signature, a request ID, an internal application contract — what do you call it?

The instinct is to prefix it with `X-`. Don't. RFC 6648 deprecated the `X-` prefix in 2012. The reasoning was practical: when a custom header eventually became a standard (the `gzip` / `x-gzip` mess is the textbook case), stripping the prefix broke every consumer that had hard-coded the prefixed name. The senior 2026 reach is *no prefix* for plain application headers (`Request-Id`, `Idempotency-Key`) and a *vendor token* for third-party-defined headers — `Stripe-Signature` on webhook callbacks from Stripe, `Svix-Id` / `Svix-Timestamp` / `Svix-Signature` on Svix-delivered webhooks. Both shapes communicate ownership without colliding with the IANA registry.

GitHub still emits `X-GitHub-Event` as a legacy holdout, named once here as the exception that proves the rule. Pick the IETF draft shape when one exists (`Idempotency-Key`, `RateLimit`), pick a vendor token for proprietary semantics, pick no prefix for everything else.

## Where headers are set in a Next.js app

The closing payoff. The three audiences map cleanly onto the three places a Next.js app sets headers, and that map is the thing you'll look up in practice when shipping a new feature.

- **`next.config.ts headers()`** — for headers that are *static* across requests, or static per route prefix. HSTS, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `frame-ancestors`-via-CSP (when no nonce is needed), `Cache-Control` on `/_next/static`. The build-time setter — the value never depends on the request, so it gets configured once.
- **`proxy.ts`** (the Next.js 16 rename of what used to be `middleware.ts`) — for headers that need *per-request data*. The flagship case is the CSP nonce: a fresh random value per request, injected into both the `Content-Security-Policy` header and the `<script>` tags that carry it. Also where the request-id header, the rate-limit headers, and per-request auth headers live. Runs on the Node.js runtime in Next.js 16.
- **The response itself** — the route handler (`route.ts`), the Server Action's response, the page's `metadata` export — for headers that are *response-specific*. `Cache-Control` per route, the `ETag` per resource, `WWW-Authenticate: Bearer` on the 401 from `authedRoute`, the Problem Details `Content-Type: application/problem+json` on a 422.

Three short snippets anchor the diagram — one per setter. These are read-only; you won't write a `next.config.ts` until Unit 3, a `proxy.ts` until Unit 4 onward, or a `route.ts` until Unit 6. The point is the *shape* — which file, which export, where the header strings actually live.

The build-time setter, `next.config.ts`. The `headers()` async function returns an array of rules, each pairing a `source` path matcher with a list of `(key, value)` pairs. Same shape across every Next.js project.

```ts title="next.config.ts"
const nextConfig = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains; preload' },
          { key: 'X-Content-Type-Options', value: 'nosniff' },
          { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
        ],
      },
    ];
  },
};

export default nextConfig;
```

Static; runs at build time; applies to every matching request.

The per-request setter, `proxy.ts` at the project root. The `proxy` function runs on the Node.js runtime before the route resolves; here it generates a per-request nonce and sets it on the response's `Content-Security-Policy` and `X-Request-Id` headers.

```ts title="proxy.ts"
import { NextResponse, type NextRequest } from 'next/server';

export function proxy(request: NextRequest) {
  const nonce = crypto.randomUUID();
  const response = NextResponse.next();
  response.headers.set(
    'Content-Security-Policy',
    `script-src 'self' 'nonce-${nonce}' 'strict-dynamic'`,
  );
  response.headers.set('X-Request-Id', nonce);
  return response;
}
```

Per-request; runs on every request that matches the proxy's config; the nonce is fresh each time. The snippet uses `crypto.randomUUID()` for parity with the `Idempotency-Key` reach in Lesson 1, but a CSP nonce is strictly meant to be a fresh random value encoded as base64 per the CSP spec — Chapter 81 swaps to the canonical `crypto.getRandomValues(...)` + base64 pattern when it wires CSP in production.

The per-response setter, the route handler at `app/api/<route>/route.ts`. Each method is a named export (`GET`, `POST`, `PATCH`, ...). Headers go on the `Response` object, so they ride with the body the handler returns.

```ts title="app/api/invoices/route.ts"
export async function GET() {
  return new Response(JSON.stringify({ invoices: [] }), {
    headers: {
      'Content-Type': 'application/json',
      'Cache-Control': 'private, no-store',
      ETag: '"v1"',
    },
  });
}
```

Per-response; set on the `Response` object so they ride with this specific body.

That's the closing map: audience → setter → file. Every header in the lesson lands in one of those three slots.

Six claims about the load-bearing reasoning in this lesson — applying the audience model, the senior defaults, and the request-context traps to specific cases. Mark each one true or false.

  <Statement answer="false">
    `Cache-Control: no-cache` forbids the cache from storing the response, so it's the senior default for authenticated HTML.
    <TfWhy>`no-cache` *allows* storage but forces revalidation on every read. The directive that forbids storage entirely is `no-store`. The senior default for authenticated HTML is `private, no-store`.</TfWhy>
  </Statement>

  <Statement answer="true">
    A first-party browser session uses cookies; a server-to-server API call uses `Authorization: Bearer`.
    <TfWhy>Browser sessions live in cookies because the browser enforces `HttpOnly`, `Secure`, `SameSite`, and the `__Host-` prefix — mitigations a bearer token doesn't have.   </Statement>

  <Statement answer="false">
    Rate-limiting by the leftmost IP in `X-Forwarded-For` is safe because the proxies append in order.
    <TfWhy>The only hop you can trust is the immediate edge. Every value to the left of that hop could have been forged by the original client. Count from the right and stop at the first untrusted hop, or read the value your trusted edge appended (e.g. Vercel's `x-vercel-forwarded-for`).</TfWhy>
  </Statement>

  <Statement answer="true">
    `Vary: Cookie` on a per-user response tells a shared cache to key the cache by the request's `Cookie` value, so user A's response doesn't get served to user B.
    <TfWhy>Without `Vary: Cookie`, the cache keys by URL only and serves the first response it cached to every user that hits that URL. The auth-default `private, no-store` makes the question moot — the response isn't cached at all — but on cacheable per-user surfaces, `Vary` is mandatory.</TfWhy>
  </Statement>

  <Statement answer="false">
    Custom application headers should be prefixed with `X-` to mark them as non-standard.
    <TfWhy>RFC 6648 deprecated the `X-` prefix in 2012. The senior reach is no prefix (e.g. `Request-Id`, `Idempotency-Key`) or a vendor token (`Stripe-Signature`, `Svix-Id`).</TfWhy>
  </Statement>

  <Statement answer="true">
    The CSP nonce is set in `proxy.ts` rather than `next.config.ts` because each request needs a fresh nonce.
    <TfWhy>`next.config.ts headers()` is for static-across-requests headers. CSP with a per-request nonce needs request-time data, which means the proxy. The same rule applies to any request-id, rate-limit, or per-tenant header.</TfWhy>
