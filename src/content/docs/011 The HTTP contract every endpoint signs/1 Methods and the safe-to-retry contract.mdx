HTTP methods aren't transport markers. They're *declarations of intent*. The intent is what the CDN, the browser's prefetcher, and every retry policy in the stack reads to decide what they're allowed to do with the request. Pick the wrong method and you don't break the request — the bytes still travel, the server still answers — but you break the contract every downstream layer was built to honor. That's the kind of mismatch that turns into a duplicated charge in production.

This lesson lands four things. **Safety** and **idempotency** as the two anchor properties — the ones that decide whether a network blip is recoverable. The five-method palette (GET, POST, PUT, PATCH, DELETE) placed in a 2x2 by those properties, so you reason about methods instead of memorising them. PUT versus PATCH for updates, the pair that catches every junior at least once. And the `Idempotency-Key` header — the fix for the one cell of the grid where nothing built into the method itself can save you. By the end you'll pick a method by reasoning, not by lookup, and you'll know exactly why a retried POST can charge a card twice.

## Methods declare intent, not transport

The previous chapter walked the wire end-to-end. The fourth stage was the HTTP request — its first line shaped `METHOD path HTTP/version`, three tokens that ride on top of every QUIC stream. The first token, the method, is what this lesson is about.

Why does the method matter past "POST has a body, GET doesn't"? Because three layers between the client and your server read it and act on it before your code ever runs.

- The browser's link-rel-prefetch and prerender machinery follows GETs speculatively when it thinks the user is about to navigate. It never speculates on POST.
- CDNs cache GET responses by default and bypass POST entirely. The default cache key is `(method, URL, Vary headers)` — POST falls out of the cache before the question of TTL is even asked.
- HTTP clients (the browser, `fetch`, server-side SDKs) automatically retry GET-shaped requests on a network error and refuse to retry POSTs. Some libraries make that refusal load-bearing — they won't retry a POST even if you ask them to.

That's the cost of a semantic mismatch. A GET that writes will be retried on a network blip, prefetched on hover, and cached at the edge — three duplicated writes from one user action. A POST that reads bypasses every cache layer and re-runs the same server work on every navigation. Picking the wrong verb doesn't just feel wrong; it breaks every layer that read the contract.

:::caution
If you find yourself wanting a method to behave one way for the server and another way for the cache — for example, a "fast read" you want to mark as POST so it never gets cached — the answer is not the method. It's a different endpoint, or a `Cache-Control` directive on the response. The verb is not a configuration switch.
:::

## Safety and idempotency, the two anchor properties

Two definitions sit upstream of everything else in this lesson. Get them right and the 2x2 grid in the next section is obvious; get them muddled and the whole rest of the course is a slog.

A method is <Term definition="An HTTP method is safe if calling it has no observable side effect on server state.">safe</Term> if calling it has no observable side effect on server state. The word *observable* is doing work here. The server is allowed to log the call, warm a cache, increment a metric counter — those are not "observable side effects" because they don't change what a client would see on the next request. The test is what the *client* perceives, not what the server's internals do. A read endpoint is safe. A "send welcome email" endpoint is not safe, no matter how innocent its name sounds.

A method is <Term definition="An HTTP method is idempotent if making the same request N times leaves the server in the same final state as making it once.">idempotent</Term> if making the same request N times leaves the server in the same final state as making it once. Read that twice — it's about *state*, not about *response*. A `DELETE /invoices/42` is idempotent: call it once and the invoice is gone; call it five times and the invoice is still gone. The first call may return `200 OK` and the next four may return `404 Not Found` — different responses, same final state. A `POST /invoices` that creates a new invoice each time it's called is not idempotent: five calls leave you with five invoices.

This is the most common misread of idempotency in production code. People hear "idempotent" and remember "returns the same response," then they reach for `DELETE` returning 404 on the replay and decide the method isn't idempotent after all. They're checking the wrong thing. Idempotency is the property the *server state* has, not the property the *response* has.

</Figure>

Why these two properties and not others? Because each one unlocks a different downstream behavior. Safety is what caches and prefetchers act on — if a method is safe, those layers can speculate freely. Idempotency is what retry policies act on — if a method is idempotent, the network layer can retry on a transient error without asking permission. Together they map the four cells of the grid the next section builds.

## The five-method palette in a 2x2 grid

Two properties, two states each — that's four cells. Drop each of the five methods into the cell its properties pick out and the design space takes shape.

<Figure caption="The five-method palette placed by property. The grey cell has no occupants — there's no use case for a read that produces different state each time. The red cell is the one that needs Idempotency-Key to become retry-safe.">
  <MethodGrid />
</Figure>

Walk the four corners in order. The top-left is GET — the only safe method in the everyday toolkit, which is also why it's the only method that caches, prefetchers, and retry layers can act on freely. `HEAD` is also safe and idempotent and lives in the same cell; it's `GET` without the response body, used to read metadata like content length or last-modified date without paying for the payload. You'll see it again later in the course when uploading to object storage. `OPTIONS` is also safe; it's what the browser sends as the CORS preflight, covered in the next chapter.

The top-right is empty, and the emptiness is informative. There's no standard method that's safe but not idempotent — and there isn't a use case for one either. A "read" that produces different state each time it's called isn't really a read. If you find one in your design, you've named it wrong.

The bottom-left holds `PUT`, `PATCH`, and `DELETE`. They're unsafe — they change server state — but they're idempotent: retry them on a network blip and the final state lands at the same place as if the original had succeeded. That's the property the retry layer can act on. PATCH lives in this cell only when its diff is *absolute* (set this field to this value); a *relative* diff (increment this field by one) breaks idempotency, and we'll come back to that in a minute.

The bottom-right holds POST alone. POST is unsafe and not idempotent — retry it and you get duplicate work. Nothing about the method or the underlying transport makes a POST retry safe. The whole second half of this lesson is about closing that hole.

## PUT and PATCH: replace versus diff

PUT and PATCH share a cell on the grid and a use case in the design — they both update an existing resource — but they speak very different languages on the wire. Getting this pair right is the difference between a clean update endpoint and a lot of small bugs.

PUT *replaces*. You send the full resource body, the server overwrites whatever it had with what you sent. The mental model is `=` — assignment. Sending the same body twice yields the same final state by construction; that's why PUT is idempotent without needing any qualifiers.

PATCH applies a *diff*. You send a description of what to change, the server applies it. The mental model is `Object.assign(current, patch)` for the merge case, or "run this list of operations" for the json-patch case. PATCH is idempotent *only when* the diff is absolute — `status = 'paid'` re-runs to the same place no matter how many times you send it. A relative diff — `balance = balance + 100` — is not idempotent: every call adds another 100. The label "PATCH is idempotent" you'll see in some references is shorthand for "PATCH-with-absolute-diff is idempotent." Watch for the qualifier.

PATCH has a second problem PUT doesn't have: the wire format isn't determined by the method. PUT's body is the resource — that's well-defined. PATCH's body is a *description of changes* — and there are two standardised ways to describe changes, with different content types signalling which one the server should expect. Here are both, applied to the same logical operation: marking invoice 42 as paid and removing its draft note.

<CodeVariants maxLines={10}>
  <CodeVariant label="merge-patch+json">
    ```http
    PATCH /invoices/42 HTTP/3
    Content-Type: application/merge-patch+json

    { "status": "paid", "draftNote": null }
    ```
    **The default 2026 reach.** <Term definition="JSON Merge Patch (RFC 7396): the patch body is a partial JSON object; keys overwrite, null deletes. Default 2026 wire format for PATCH on object-shaped resources.">JSON Merge Patch</Term> (RFC 7396) lets you send a partial JSON object: keys overwrite, an explicit `null` deletes a key. Limitation: arrays can't be merged element-wise — to change one item in an array you have to send the whole new array. Use when your patches are field-level on object-shaped resources, which is most SaaS update endpoints.
  </CodeVariant>

  <CodeVariant label="json-patch+json">
    ```http
    PATCH /invoices/42 HTTP/3
    Content-Type: application/json-patch+json

    [
      { "op": "replace", "path": "/status", "value": "paid" },
      { "op": "remove", "path": "/draftNote" }
    ]
    ```
    **The reach for operation semantics.** <Term definition="JSON Patch (RFC 6902): the patch body is an array of typed ops (add, remove, replace, move, copy, test). Reach for it when the diff needs operation semantics merge-patch can't express.">JSON Patch</Term> (RFC 6902) is an array of typed operations — `add`, `remove`, `replace`, `move`, `copy`, `test`. Verbose but unambiguous, supports array mutation, and the `test` op lets you embed optimistic-concurrency checks in the patch itself. Use when the diff has to be precise about ordering, array changes, or conditional application.
  </CodeVariant>
</CodeVariants>

Merge-patch is the default — that's the one to reach for unless you have a reason. Json-patch is the escape hatch when the diff has to encode operations merge-patch can't express. Most SaaS update endpoints, including the invoicing endpoints you'll build later in the course, land on merge-patch.

:::caution
The most common PATCH bug in production isn't the format choice — it's sending PATCH with `Content-Type: application/json`. That's the generic JSON content type, and it doesn't tell the server *which* PATCH dialect to expect. The server has to guess, and guesses go wrong. Always set `application/merge-patch+json` or `application/json-patch+json` explicitly, or document the specific dialect your endpoint speaks. The headers lesson later in this chapter goes deeper on `Content-Type` as the content-negotiation surface.
:::

## When retries lie: the POST problem

Picture this. A user clicks "Pay" on an invoice. The client sends `POST /payments/charge` with `{ amount: 5000, currency: 'usd', source: 'tok_...' }`. The server receives the request, charges the card through Stripe, writes the payment row to the database, builds a `200 OK` response — and then the response packet gets dropped on a flaky WiFi connection. The server completed every side effect. The client never saw the answer.

What happens next? The client's network layer (or the user, hitting "Pay" again because the spinner didn't go away) retries. The server gets a second `POST /payments/charge` with the same body. From the server's perspective, this is a brand-new request with no context — it charges the card a second time. The customer is now down 100 dollars instead of 50, and you've got a refund ticket to process before the day is over.

GET-style retry doesn't fix this. GET retries are safe because the *method* is safe — the server's reaction to a duplicate GET is "send the same answer again." POST is unsafe by construction; nothing about the method or the transport makes its retry safe. The fix has to live at the application layer.

Here's the pattern. The client generates a stable identifier *per logical operation* — one ID per "the user wants to pay this invoice," not one per HTTP attempt — and sends it as an <Term definition="Request header whose value is a stable per-operation identifier (typically a UUID). The server stores (key, response) and returns the cached response on replay, making non-idempotent methods retry-safe.">`Idempotency-Key`</Term> header. The server stores `(key, response)` before sending the response back. On a retry with the same key, the server *recognises* the replay, returns the cached response, and runs no business logic at all. Same logical operation in, same response out, exactly one side effect.

Three things to nail down before moving on. First, where the key comes from. The client generates a UUID *per operation* — one per "user clicked Pay on invoice 42," not one per HTTP request. The call site is `crypto.randomUUID()`, which is available on the global `crypto` object in every modern runtime. (You'll meet UUIDv7 in the database unit as the primary-key default; the idempotency key is a UUIDv4 instead, because what we want here is randomness and opacity, not time-ordering.) The key has to live *with the operation*, not with the request — so a retry uses the same key the original sent. In a React app that often means storing the key on a mutation handle and reusing it across attempts.

Second, where the key lives on the server. The server needs a `(key, response, expires_at)` store with a unique constraint on the key column. When a request arrives, it looks the key up; if it finds it, it returns the cached response; if not, it processes the request, stores the result, and *then* sends the response. The unique constraint catches the race between two simultaneous retries trying to be the first to store the result. The full implementation — schema, TTL, unique-constraint race handling, the transactional shape — lands later in the course when you build the Stripe webhook ingestion path. Here, the contract is what matters: *the server stores the key before sending the response, and looks it up on every incoming request*.

Third, where the standard is. `Idempotency-Key` is on track to become a Standards-Track RFC — the IETF httpapi working group draft `draft-ietf-httpapi-idempotency-key-header-07` (October 2025) is the current shape, and Stripe, PayPal, and most payment processors already deploy it. The header name and value format aren't going to change between draft and RFC; you can build against the contract now without worrying about renaming later.

Here's the client side end-to-end. This is the call shape you'll be writing later, once `fetch` gets covered properly in this unit and the billing unit wires up Stripe.

<AnnotatedCode lang="ts" maxLines={12} code={`
const idempotencyKey = crypto.randomUUID();

const response = await fetch('/api/payments/charge', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Idempotency-Key': idempotencyKey,
  },
  body: JSON.stringify({ amount: 5000, currency: 'usd' }),
});
`}>
  <AnnotatedStep meta="{1}" color="blue">
    The key is generated once per logical operation. Store it on the operation handle — a React Query mutation key, a form ref, a state value — so every retry of the same operation reuses it. Generating a fresh key inside the `fetch` retry loop is the most common way to defeat the whole pattern.
  </AnnotatedStep>

  <AnnotatedStep meta={`{5-8} "Idempotency-Key"`} color="orange">
    The header rides on the request. The name is fixed by the draft; the value is your UUID. Both must travel together on every retry — drop the header on a retry and the server treats it as a fresh operation.
  </AnnotatedStep>

  <AnnotatedStep meta={`{1} /crypto\\.randomUUID/`} color="violet">
    `crypto.randomUUID()` returns a UUIDv4 — random, opaque, unique enough to be safe against collisions. The call lives on the global `crypto` object, part of the Web Crypto API covered later in this unit.
  </AnnotatedStep>
</AnnotatedCode>

:::caution
Three failure modes the implementation has to avoid. **One**: generating a new key on each network attempt. That defeats the entire purpose — every retry looks like a new operation to the server. **Two**: reusing the same key across different operations. One logical operation, one key; sharing keys means a retry of operation B replays the response of operation A. **Three**: writing the response to the store *after* sending it back. A server crash between "respond" and "persist" leaves no record of the work, and the retry re-runs it. Persist first, respond second. The route-handler and database side of all three lands in the Stripe webhook chapter; for now, hold the contract in mind.
:::

## Picking the verb is picking the contract

The whole property model — safety, idempotency, the 2x2, the Idempotency-Key — meets code at one surface: the Next.js route handler. In the App Router (covered properly later in the course), a `route.ts` file under `app/api/` exports an async function named after each HTTP method it handles. The framework reads the request line, finds the function matching the verb, and calls it. The function name *is* the contract.

```ts title="src/app/api/invoices/[id]/route.ts"
import type { NextRequest } from 'next/server';

export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  // load and return the invoice
}

export async function PATCH(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  // apply a merge-patch and return the updated invoice
}

export async function DELETE(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> },
) {
  // delete the invoice
}
```

One file, one route, three handlers. The framework dispatches `GET /invoices/42` to the `GET` function, `PATCH /invoices/42` to `PATCH`, and so on. A `POST` arriving at this route — when no `POST` is exported — gets an automatic `405 Method Not Allowed`. The verb is the contract; the handler implements it; the name on the exported function is where those two meet. Full coverage of route handlers, including the validation pipeline and the `authedRoute` wrapper that runs the auth and permission checks, comes later in the course.

There's one carve-out worth one paragraph before the recall. The HTTP spec doesn't *forbid* a GET request from carrying a body — older drafts allowed it and the current spec is silent rather than prohibitive — but intermediaries (proxies, CDNs, some HTTP libraries) silently strip GET bodies, and the request semantics ignore them. In practice, a GET body is a body that won't survive the journey. If your read needs structure — filters, sort columns, pagination — encode it in the query string (which the next chapter covers as the URL surface, including `URLSearchParams`).

:::tip
If you're tempted by "GET with a body," the real fix is usually POST with a clear name. An endpoint like `POST /searches` with the query parameters in the body, returning the result rows, is a perfectly reasonable shape — it just isn't a *read* in the HTTP sense, and you signal that honestly by using POST. Don't try to dress an unsafe operation as a safe one.
:::

## A round on the corners

Four claims to apply the property model against. Each one tests whether you're reasoning about state and contract, not pattern-matching method names.

<TrueFalse instructions="Mark each claim about HTTP methods and retries True or False.">
  <Statement answer="false">
    Calling `DELETE /invoices/42` twice in a row makes DELETE non-idempotent, because the second call does less work and returns a different status code.
    <TfWhy>Idempotency is about *state*, not response or work. After both calls, invoice 42 is gone — same final state. The fact that the second call may return 404 instead of 200 doesn't break the property; the response can differ across calls, the state cannot.</TfWhy>
  </Statement>

  <Statement answer="true">
    A `PATCH` endpoint that accepts `{ "balance": 100 }` and overwrites the balance field is idempotent.
    <TfWhy>Absolute set. Repeating the call lands at the same final state every time. The relative-diff variant `{ "delta": 100 }`, on the other hand, accumulates — five calls add 500, which breaks idempotency.</TfWhy>
  </Statement>

  <Statement answer="false">
    Generating a fresh `Idempotency-Key` for each network retry of the same charge guarantees the card is only charged once.
    <TfWhy>Exactly opposite. The whole point is that retries share the *same* key, so the server recognises the replay. A new key per attempt makes every retry look like a brand-new operation, and the server happily runs the charge again.</TfWhy>
  </Statement>

  <Statement answer="true">
    A POST endpoint that only logs the request and never mutates business state still cannot be retried safely without an idempotency mechanism.
    <TfWhy>POST is unsafe by contract — the cache, retry, and prefetch layers treat it as side-effecting whether or not the implementation actually is. Method choice declares intent; the implementation does not override the contract every downstream layer signed.</TfWhy>
  </Statement>
</TrueFalse>
