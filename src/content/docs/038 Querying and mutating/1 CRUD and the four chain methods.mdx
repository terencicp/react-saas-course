In the last chapter you made the schema the source of truth. You have an `invoices` table, an `organizations` table, the relations between them, and every TypeScript row type derived from those definitions in a single line. That's the whole shape of your data. And right now it does nothing — a table you can't read from or write to is furniture. This lesson is where the schema starts earning its keep.

Step back and ask the question an experienced engineer asks before writing a single query: *given a schema, what is the smallest set of call shapes that covers most of a feature's data access?* Because almost every feature you'll ever build does the same four things to its data. It reads rows. It filters, sorts, and pages through them. And it writes rows back. That's it. The surface is much smaller than it looks, and it's been the same four verbs since SQL was invented — <Term definition="Create, Read, Update, Delete — the four basic operations on stored data, and the four shapes you spend most of your time writing.">CRUD</Term>: create, read, update, delete. SQL spells them `INSERT`, `SELECT`, `UPDATE`, `DELETE`. Drizzle wraps each one in a typed, autocompleted TypeScript call, because Drizzle is an <Term definition="Object-Relational Mapper — a library that lets you express database queries in your programming language instead of raw SQL strings, with the schema driving the types.">ORM</Term>: it maps your tables to code instead of leaving you to concatenate SQL by hand.

By the end of this lesson you'll be able to write every basic read and write a feature needs. You'll know the four entry points (`db.select`, `db.insert`, `db.update`, `db.delete`) and the four methods that shape a result set (`where`, `orderBy`, `limit`, `offset`). You'll learn the one omission that can silently wipe a table in production. And you'll understand why SQL injection — the bug that has leaked more databases than any other — is something you essentially never have to think about on these paths. Let's start with the object that runs every one of these queries.

## The `db` client and the query builder

Every query in your app goes through one object: `db`. You import it from `db/index.ts`, alongside the tables from `db/schema.ts`. We won't build `db/index.ts` here — wiring it up to your Neon database is its own setup chapter a little later — but you should picture it as the single call site for all data access. One import, one client, every read and write.

```ts
import { db } from '@/db';
import { invoices } from '@/db/schema';
```

Here's the thing that trips people up first, so let's get it out of the way. When you call `db.select()`, **no query runs.** Nothing touches the database. What you get back is a *query builder* — an object that's collecting instructions. Each method you call on it (`.from()`, `.where()`, and the rest) tacks on another instruction and hands the builder right back to you, which is why the calls chain. The builder sits there, fully assembled but inert, until you do one specific thing: `await` it.

```ts
const allInvoices = await db.select().from(invoices);
```

That `await` is the trigger. The builder is a <Term definition="An object with a .then() method, so await works on it directly. Drizzle's query builder is thenable — awaiting it is what fires the query.">thenable</Term> — an object you can `await` — and only when you await it (or call `.execute()`) does Drizzle assemble the SQL and send it to Postgres. This laziness buys you two things worth holding onto. First, nothing fires by accident; a builder you forget to await is a no-op, not a stray query against production. Second — and this is the one that'll save you a confused afternoon later — the *order* you chain the methods in doesn't change the SQL that comes out. `.where().orderBy()` and `.orderBy().where()` produce the identical query. The builder collects the pieces and arranges them correctly no matter what order you hand them over. Pick an order that reads well and stop worrying about it.

That's the mental model. Now let's actually read some rows.

## Reading rows with `db.select`

The simplest read is the whole table:

```ts
const allInvoices = await db.select().from(invoices);
```

`allInvoices` is typed as `Invoice[]` — an array of the full row shape, every column, exactly the `$inferSelect` type you derived in the last chapter (`Invoice` is `typeof invoices.$inferSelect`). You didn't annotate it. Drizzle reads the type straight off the `invoices` table and flows it through the query, so the result is typed for free and can never drift from the schema.

One detail that is genuinely the most common first mistake, so I'll say it plainly: `.from()` is mandatory. `db.select()` on its own doesn't know what table you mean — it's half a sentence. Always pair `db.select()` with `.from(table)`.

Often you don't want every column. A list view might need only the id and the amount. You narrow the read by passing a *projection* — an object that names exactly the columns you want:

<CodeTooltips tooltips={{
  select: 'db.select() with no argument returns Invoice[] — the full row shape, every column.',
  amountDue: 'The projected result is { id: string; amountDue: string }[] — exactly the two fields you asked for, each typed from its column. amountDue is a numeric column, so its type is string.',
}}>
```ts
const rows = await db
  .select({ id: invoices.id, amountDue: invoices.amountDue })
  .from(invoices);
```
</CodeTooltips>

Now `rows` is typed as `{ id: string; amountDue: string }[]` — precisely those two fields, no more. The return type *follows the projection*. You didn't reach for `Pick<Invoice, 'id' | 'amountDue'>` or write the shape out by hand; you picked the columns and the type came with them. (Recall the carve-out from the last chapter: projecting columns you already declared is fine — it's restating a field *and* its type in a hand-written interface that's the smell.) The key on the left (`id`, `amountDue`) is the name the field gets in TypeScript; the value on the right is the column it reads from.

Notice `amountDue` comes back as `string`, not `number`. That's deliberate and it's the rule for the whole course: money is a `numeric` column, and `numeric` maps to `string` in TypeScript so you never lose a cent to floating-point rounding. You'll format it for display and never run it through `parseFloat`. Hold that thought — it'll matter the moment you write an amount back.

## Filtering with `where` and the operator helpers

A feature almost never wants *all* the invoices. It wants the sent ones, or the ones over a threshold, or the ones belonging to the organization the current user is looking at. That's what `where` is for: it filters the rows before they come back.

But `where` doesn't take a string. You don't write `where('status = sent')`. You pass it a *condition* built from a small set of helper functions imported from `drizzle-orm`. The first one you'll reach for is `eq` — equals:

```ts
const sentInvoices = await db
  .select()
  .from(invoices)
  .where(eq(invoices.status, 'sent'));
```

Read `eq(invoices.status, 'sent')` as "the `status` column equals `'sent'`." The first argument is the column; the second is the value to compare it against. There's a helper like this for every comparison you'll want. You don't need to memorize them — they read exactly like what they emit — but here's the working set so you know what's available:

<OperatorReference />

A quick watch-out hiding in those last two rows, because it bites people who don't know Postgres: `like` is **case-sensitive**. `like(invoices.status, 'SENT')` will not match a stored `'sent'`. When you want case-insensitive matching — which for user-facing search is almost always — reach for `ilike`, the case-insensitive variant Postgres gives you natively:

```ts
where(ilike(organizations.name, '%acme%'))
```

### Combining conditions

One condition is rarely enough. You filter by status *and* organization, or by one status *or* another. You combine conditions with `and(...)`, `or(...)`, and `not(...)` — each takes one or more conditions and joins them. Here's the shape you'll write more than any other in this entire course:

```ts
const orgSentInvoices = await db
  .select()
  .from(invoices)
  .where(and(eq(invoices.organizationId, orgId), eq(invoices.status, 'sent')));
```

"Sent invoices belonging to this organization." That `eq(invoices.organizationId, orgId)` is doing quiet, important work. Your app is multi-tenant — many organizations share one database — and a query that forgets to scope by `organizationId` is a query that can leak one tenant's invoices to another. So you filter by it explicitly, at the query layer, every time.

:::note
Writing this org filter by hand on every query is repetitive, and repetitive is error-prone. A later chapter on building list views folds it into a factory — a `tenantDb(orgId)` helper that scopes every query for you so you can't forget it. For now, the explicit `where` is exactly right, and writing it by hand a few dozen times is the best way to make org-scoping a reflex.
:::

### Why SQL injection isn't your problem here

Now look closely at what just happened, because it's the single most important security property of your entire data layer and you got it without doing anything.

When you wrote `eq(invoices.status, 'sent')`, that `'sent'` did **not** get glued into a SQL string. Drizzle sends the SQL text and the value to Postgres *separately*: the query goes out as `... WHERE status = $1`, and `'sent'` rides along as the bound value for `$1`. Postgres receives the structure and the data on different channels and never confuses one for the other. This is a <Term definition="A query where the SQL text and the values are sent to the database separately, with placeholders like $1 standing in for each value. The values can never be parsed as SQL.">parameterized query</Term>, and the `$1` is a <Term definition="A bound slot in a parameterized query (written $1, $2, …). The database fills it with the value at execution time — the value is never part of the SQL text.">placeholder</Term>.

Why does that matter so much? Because the value is never part of the SQL text, it can never be *interpreted* as SQL. Even if a user types `'; DROP TABLE invoices; --` into a search box and that string lands in your `eq`, Postgres treats it as a literal value to compare against — six absurd characters that match no invoice — not as a command to run. That attack, where hostile input changes what a query *means*, is <Term definition="An attack where attacker-supplied input is treated as SQL code instead of data, letting it read, change, or destroy the database. Parameterization makes it structurally impossible.">SQL injection</Term>, and it has leaked more production databases than any other single bug. Through Drizzle's helpers it is structurally impossible, because there is no string for the attacker's input to escape into.

To see why this is a property of the *tool* and not just good luck, compare the two ways you might write the same filter:

There is exactly one way to lose this protection, and it's worth naming so you recognize it in a code review: `sql.raw(userInput)` splices its argument into the query verbatim, exactly like the unsafe tab above. It exists for a narrow, legitimate purpose you'll meet at the end of this chapter — and it is never, ever pointed at user input. Everywhere else, parameterization is automatic and you can stop thinking about injection.

## Sorting and paging with `orderBy`, `limit`, and `offset`

Rows come back from Postgres in no guaranteed order unless you ask for one. `orderBy` is how you ask. You wrap the column in `asc(...)` or `desc(...)` — both from `drizzle-orm` — to set the direction:

```ts
const recent = await db
  .select()
  .from(invoices)
  .where(eq(invoices.organizationId, orgId))
  .orderBy(desc(invoices.createdAt));
```

Newest invoices first. Simple enough — but there's a subtle bug lurking here that separates code that works on your laptop from code that holds up in production, and it's worth a real beat.

### The tiebreaker, and why it's a reflex

`createdAt` is a timestamp, and timestamps collide. Two invoices created in the same millisecond — a bulk import, a burst of traffic — share a `createdAt` value. When you sort by `createdAt` alone, Postgres has no instruction for how to order rows that *tie* on that column, so it's free to return them in whatever order is convenient: one way today, the other way after a row gets updated and moves on disk. Your sort is *non-deterministic*, and you won't notice until a user swears a row jumped position, or a paginated list shows the same invoice twice.

The fix is one extra key — a unique tiebreaker, almost always the primary key:

```ts
.orderBy(desc(invoices.createdAt), asc(invoices.id))
```

Now ties on `createdAt` are broken by `id`, which is unique, so the order is fully determined — *deterministic*. Same query, same rows, same order, every time. Get in the habit of pairing any non-unique sort column with the primary key as a tiebreaker; experienced engineers do it without thinking. It's not pedantry — later in this chapter, cursor pagination depends entirely on this exact trick to page through a list without skipping or repeating rows.

### Slicing the result with `limit` and `offset`

`limit` caps how many rows come back; `offset` skips rows from the front. Together they page:

```ts
.orderBy(desc(invoices.createdAt), asc(invoices.id))
.limit(20)
.offset(40)
```

That's page three at twenty rows a page: skip the first forty, take the next twenty. It's the most intuitive way to paginate, and for small, fixed lists — an admin table of a few hundred rows — it's exactly the right tool. It also has real limits at scale and on data that changes while a user reads it, and there's a better technique for those cases. We'll make that call properly in the cursor pagination lesson later in this chapter; for now, know that `offset` is the simple default and that it has a ceiling.

Two quick things to file away, since this is the first time you're chaining three methods at once. `.limit(0)` returns an *empty* array — that's different from leaving `.limit` off entirely, which returns *everything*; don't reach for `limit(0)` thinking it means "no limit." And the order you chain `.where()`, `.orderBy()`, and `.limit()` among themselves makes no difference to the SQL, exactly as the builder section promised. Order them however reads best — most people write filter, then sort, then slice — and move on.

## Writing rows: insert, update, and delete

Writes share two threads, which is why we teach them together. Both threads matter more here than anywhere else in the lesson, because writes are where mistakes don't just return the wrong data — they *change* or *destroy* it.

**Inserting** uses `db.insert(table).values(...)`. What you pass to `.values()` is exactly the `$inferInsert` shape from the last chapter — and that shape already encodes which columns you must supply and which you can skip. Columns with a default fill themselves in: `createdAt` has `.defaultNow()`, the primary key has its own default, so you leave them out. Columns marked `.notNull()` with no default are required. Generated columns are rejected outright. You don't re-derive that asymmetry; the type already knows it.

```ts
await db.insert(invoices).values({
  organizationId: orgId,
  amountDue: '0.00',
  status: 'draft',
  dueDate: '2026-07-01',
});
```

Notice `amountDue` is the string `'0.00'`, not the number `0`. Same money rule as before — `numeric` is a string end to end, on the way in just as on the way out. To insert several rows at once, hand `.values()` an array instead of a single object; Drizzle batches them into one statement.

**Updating** uses `db.update(table).set(...).where(...)`. `.set()` takes the columns you want to change:

```ts
await db.update(invoices).set({ status: 'paid' }).where(eq(invoices.id, id));
```

**Deleting** uses `db.delete(table).where(...)`:

```ts
await db.delete(invoices).where(eq(invoices.id, id));
```

And now the most important sentence in this lesson.

### The missing `where` that wipes the table

Look at the `update` and the `delete` again. Both end in a `.where(...)`. Take it away:

```ts
await db.update(invoices).set({ status: 'void' });
await db.delete(invoices);
```

The first sets **every invoice in the table** to `'void'`. The second **empties the table.** Not for one organization — for all of them, every row, gone.

Here is what makes this dangerous rather than merely possible: Drizzle does not warn you. The types are happy. It compiles. It runs. There is no red squiggle, no thrown error, no "are you sure?" — the statement is perfectly valid SQL, and an `UPDATE` or `DELETE` with no `WHERE` clause means "every row" by definition: the database is behaving exactly as designed. The gap between "I meant to update one row" and "I updated all of them" is a single missing clause, and nothing in the type system is positioned to catch it.

:::danger[A missing `where` is a data-loss incident, not a compile error]
`db.delete(invoices)` with no `where` deletes every row in the table. The compiler won't stop you, the test suite might not stop you, and in production it's an outage and a restore-from-backup. Treat an unqualified `update` or `delete` as a bug on sight — yours or anyone's in a review.
:::

So experienced engineers build a reflex: **every `update` and every `delete` carries a `where`.** No exceptions in normal feature code — if you genuinely mean "all rows," that's rare enough to call out loudly in the code. And because reflexes slip, the ecosystem backs you up: the `eslint-plugin-drizzle` package ships two lint rules, `enforce-update-with-where` and `enforce-delete-with-where`, that flag an unqualified `update` or `delete` at lint time, before the code ever runs. Turn them on. The failure mode is severe enough that you want a machine watching for it, not just your own discipline.

### Getting the written row back with `.returning()`

There's a question every write raises: *what did I just write?* You inserted an invoice — what id did it get? You updated one — what does it look like now? The naive answer is to issue a second query: insert, then select the row you just inserted. That's two round-trips to the database and a window between them where another request could change things. There's a better way, and it's a thread that runs through the whole chapter.

Append `.returning()` to any insert, update, or delete and the statement hands the affected rows straight back, with the full `$inferSelect` shape — or a projected subset if you pass a projection, just like `select`:

```ts
const [created] = await db
  .insert(invoices)
  .values({ organizationId: orgId, amountDue: '0.00', status: 'draft', dueDate: '2026-07-01' })
  .returning();
```

One statement. `created` is the row that landed in the table — its generated `id`, its `createdAt`, everything, carrying the full `Invoice` shape — with no follow-up select. (`.returning()` hands back an *array*, so you destructure the one row out, just like a single-row read; more on that pattern at the end of the lesson.) Whenever you find yourself writing a row and then immediately selecting it back, that's the smell: reach for `.returning()` instead. Let's walk through the two write shapes you'll use most, side by side.

One more thing about deletes before you practice, because the senior default might surprise you: in most SaaS apps you rarely hard-`delete` a row at all. The common pattern is a *soft* delete — an `update` that sets a `deletedAt` timestamp instead of removing the row — so the record survives for audit and can be recovered. (That `deletedAt` column already exists on your tables from the last chapter.) A real `delete` is reserved for things like offboarding a whole tenant or expiring old audit logs. The next chapter owns soft delete and the query-time filtering it needs; for now, just know that `db.delete` is the heavier hammer and not your everyday reach.

## Reading one row vs. many

A gap you'll hit within your first hour: you have an invoice's `id`, you want *that one invoice*, and `db.select()` insists on handing you an array. There's no `db.select().findOne()`. The builder always returns rows, plural, even when you know there's at most one.

The idiom is to cap the query at one row and destructure the first element out:

```ts
const [invoice] = await db
  .select()
  .from(invoices)
  .where(eq(invoices.id, id))
  .limit(1);
```

`invoice` is now a single `Invoice` — or `undefined`, if no row matched that id. That `| undefined` is not noise; it's the type system reminding you the row might not exist. The not-found case is real (a deleted invoice, a bad id in a URL) and you'll handle it downstream, but the type makes sure you can't forget it's possible.

If you went looking for a `.findFirst()` method on `db.select()` and didn't find one, you weren't wrong to look — it exists, just not here. `findFirst` lives on Drizzle's *relational query API* (`db.query.invoices.findFirst(...)`), a separate, higher-level way to read that's built for fetching a row together with its related rows. That's the subject of a lesson coming up shortly in this chapter. On the plain SQL builder you're using now, `.limit(1)` and a destructure is the move.

That's the full toolkit. Four entry points, four chain methods, the operator helpers, parameterization you get for free, `.returning()` to skip the round-trip, and the missing-`where` failure mode burned into your reviewing instincts. Every later lesson in this chapter — joins, nested reads, aggregates, upserts, pagination — is built on these exact shapes. Get comfortable here and the rest is layering.
