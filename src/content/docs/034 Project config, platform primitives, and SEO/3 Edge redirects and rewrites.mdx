The product just renamed a whole section. What used to live under `/account` now lives under `/settings` — same screens, new address — and the change is permanent. But the old URLs don't vanish on rename day: they're in months of emails, in bookmarks, in whatever Google indexed last quarter. Every one of them needs to land the visitor on the new path. For everyone. Forever.

You already know how to write this. In the last chapter you built `proxy.ts` and sent exactly this kind of redirect from it — read the path, return a `308` to the new one. So a fair question is: why a whole new lesson for a redirect you can already write?

Because this redirect is *always true*. It doesn't read the cookie, the header, the session, or anything about who's asking — `/account` goes to `/settings` no matter what. And the moment you put an always-true rule in `proxy.ts`, you've signed the platform up to run a function on every single matched request just to send back the same redirect it could have served from the CDN edge for free. The proxy is the right home for rules that *depend* on the request. This rule depends on nothing.

So this lesson installs the **third home** for URL rules: `redirects()` and `rewrites()` in `next.config.ts` — the static, edge-applied place for rules that are the same for every visitor. By the end you'll know its pattern syntax, the one question that decides which of the three homes a rule belongs in, and two booby-traps that bite hard if you miss them — one that loops your redirect forever, and one you can't take back after launch.

One recap, because this lesson reuses concepts you already own rather than re-teaching them. You learned redirect-versus-rewrite and the 307-versus-308 distinction in the previous chapter, building rules in `proxy.ts`; you met `redirect()` and `permanentRedirect()` from `next/navigation` back in the routing chapter. This lesson is about the *config* home — and the cost reason it's the default when a rule is request-blind.

## The cheapest place to send a redirect

Let's start with the cost, because the cost is the entire reason this home exists.

A redirect in `next.config.ts` is an `async` function named `redirects` that returns an array of rules. Each rule is three fields: where to match (`source`), where to send (`destination`), and whether it's permanent. Here's the rebrand, in full:

<div data-mark-color="blue">

```ts title="next.config.ts" "permanent: true"
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  async redirects() {
    return [
      {
        source: '/account/:path*',
        destination: '/settings/:path*',
        permanent: true,
      },
    ];
  },
};

export default nextConfig;
```

</div>

The `:path*` part — `/account/:path*` matching the whole sub-tree under `/account` — is pattern syntax we'll pull apart in the next section. For now, focus on the headline, because it's the whole point: **this rule is applied at the CDN edge, with zero function invocation.** No `proxy.ts` runs. No Server Component renders. Next reads this rule table once at build time and hands it to the edge, and from then on the edge answers `/account/...` with a redirect to `/settings/...` before any of your code wakes up. The redirect is served from the same layer that serves your cached assets — about as fast and as cheap as the web gets.

Hold that against the proxy. In the last chapter you learned the proxy's cost model: every request the matcher selects pays a function round trip. That's the right price for a rule that has to *read* the request. But for a rule that reads nothing — that sends the same redirect to everyone — paying a function invocation is pure waste. You're spinning up compute to compute a constant.

The two homes, with the exact same rebrand redirect written both ways:

<CodeVariants>
  <CodeVariant label="In proxy.ts (runs every request)" icon="lucide:function-square">
    <div data-mark-color="orange">

    ```ts title="proxy.ts" {5}
    export default function proxy(request: NextRequest) {
      const { pathname } = request.nextUrl;
      if (pathname.startsWith('/account')) {
        const settings = pathname.replace('/account', '/settings');
        return NextResponse.redirect(new URL(settings, request.url), 308);
      }
      return NextResponse.next();
    }
    ```

    </div>
    **Pays a function invocation on every matched request.** The proxy is code; the platform has to run it on every matched request to find out it returned the same `308` every time. That's the right home when a rule reads the request — and pure overhead when it doesn't.
  </CodeVariant>

  <CodeVariant label="In next.config.ts (runs at the edge)" icon="lucide:zap">
    <div data-mark-color="green">

    ```ts title="next.config.ts" {4-8}
    async redirects() {
      return [
        {
          source: '/account/:path*',
          destination: '/settings/:path*',
          permanent: true,
        },
      ];
    }
    ```

    </div>
    **Zero invocation — applied at the edge.** Next bakes this rule into the edge config at build time, and the CDN answers the redirect before any function runs. Same outcome, none of the per-request cost. This is the cheaper home, and it's the default for a request-blind rule.
  </CodeVariant>
</CodeVariants>

That `permanent` field is the 308-versus-307 distinction you already know, surfaced as a boolean. `permanent: true` sends a **308** — the sticky, indexed redirect that tells search engines the page moved for good and forwards its <Term definition="The ranking value search engines pass through a link. A 308 forwards it from the old URL to the new one; a 307 does not.">link equity</Term> to the new URL. `permanent: false` sends a **307** — temporary, the old URL stays canonical. Both preserve the request method exactly, which is the whole reason Next uses this pair and never `301`/`302` (a `301`/`302` can quietly turn a `POST` into a `GET`). You derived all of that last chapter; here it's just one field. The rebrand is genuinely permanent, so `true`.

Now name the constraint that shapes everything else in this lesson. These config rules are **global and request-blind by default** — the same for every visitor, with no per-route override and no view into the session. That's simultaneously the feature and the limit. It's the feature because request-blindness is exactly what lets the edge cache and apply the rule without running your code. It's the limit because a rule that genuinely needs to know who's asking can't live here — it has to move to the proxy. Keep that line in mind; it's the spine the whole lesson hangs on.

:::caution
`next.config.ts` is read **once, at server startup**. Edit a redirect while the dev server is running, hit the route, and nothing happens — the server is still applying the rules it loaded when it booted. Before you suspect the rule is wrong, restart the dev server. "I added a redirect and it didn't fire" is, nine times out of ten, a stale config, not a broken rule — the same gotcha you met with every other key in this file, now biting on redirects exactly as promised.
:::

## Matching the request path

The `source` field is a pattern, not a literal path, and learning to read it is the one genuinely new mechanical skill in this lesson. It's worth a few minutes, because the difference between two patterns that *look* almost identical is the difference between a rule that forwards your whole rebranded section and one that silently drops half of it.

Next compiles `source` with a library called <Term definition="The pattern library Next.js compiles the source field to. Its :param and * grammar is the same one Express routes use — a colon names a segment, a star makes it repeatable.">path-to-regexp</Term> — the same `:param` and `*` grammar Express routes use. Let's build it up one rung at a time, simplest first.

<AnnotatedCode lang="ts" maxLines={14} code={`
async redirects() {
  return [
    { source: '/account', destination: '/settings', permanent: true },
    { source: '/account/:slug', destination: '/settings/:slug', permanent: true },
    { source: '/account/:path*', destination: '/settings/:path*', permanent: true },
    { source: '/invoices/:id(\\\\d+)', destination: '/bills/:id', permanent: true },
  ];
}
`}>
  <AnnotatedStep meta="{3}" color="blue">
    **Static path.** `/account` matches exactly `/account`, anchored to the start — it won't match `/team/account`. The simplest rule there is: one literal path in, one literal path out, no capture and no nesting.
  </AnnotatedStep>

  <AnnotatedStep meta="{4}" color="blue">
    **Named single segment.** `:slug` captures exactly one path segment. `/account/:slug` matches `/account/billing` but **not** `/account/billing/history` — a colon param is one segment, never a sub-tree. The captured `:slug` is reused by name in `destination`.
  </AnnotatedStep>

  <AnnotatedStep meta="{5}" color="blue">
    **Catch-all.** Add `*` and `:path*` captures zero-or-more segments: `/account`, `/account/billing`, and `/account/billing/history` all match. This is the rebrand workhorse — it forwards the entire sub-tree in one rule. (`+` means one-or-more, `?` means optional.)
  </AnnotatedStep>

  <AnnotatedStep meta="{6}" color="blue">
    **Regex-constrained.** `:id(\\d+)` matches only digits, so `/invoices/123` matches and `/invoices/abc` does not. Note the double backslash — `\\d` in the TypeScript string is the single `\d` the matcher sees. Use this to keep a rule from firing on the wrong shape of input.
  </AnnotatedStep>
</AnnotatedCode>

A couple of things worth pinning down from that walkthrough. First, **the destination references captures by name.** Whatever `:slug` or `:path*` captured in `source` gets substituted back into `destination` — that's how `/account/:path*` → `/settings/:path*` carries the entire sub-tree across. Query strings come along automatically too: a request to `/account/billing?ref=email` lands on `/settings/billing?ref=email` without you touching the query. Second, **the single-segment versus catch-all boundary is the one that trips everyone.** `:slug` stops at the first slash; `:path*` keeps going. Pick `:slug` for the rebrand and every nested URL under `/account` quietly fails to redirect — the kind of bug nobody notices until a customer reports a dead bookmark.

And now the booby-trap, because it's the one that turns a redirect into an outage. **The forward slash must come before the colon.** Write `'/account/:path*'`, never `'account/:path*'`. It looks like a typo too small to matter. It isn't.

<CodeVariants>
  <CodeVariant label="Broken — redirects forever" icon="lucide:repeat">
    <div data-mark-color="red">

    ```ts del={2}
    {
      source: 'account/:path*',
      destination: '/settings/:path*',
      permanent: true,
    }
    ```

    </div>
    **This redirects in an infinite loop.** The leading `/` is what anchors a pattern to the start of the path. Drop it and the pattern is no longer pinned to the root, so it keeps matching the path it just redirected *to* — the browser arrives at the destination, the unanchored rule matches again, and it bounces until it gives up with a "too many redirects" error. The Next.js docs name this missing slash as the canonical cause of redirect loops.
  </CodeVariant>

  <CodeVariant label="Fixed" icon="lucide:check">
    <div data-mark-color="green">

    ```ts ins={2}
    {
      source: '/account/:path*',
      destination: '/settings/:path*',
      permanent: true,
    }
    ```

    </div>
    **One character fixes it.** The leading `/` anchors the pattern to the path root, so it matches `/account/...` and nothing under `/settings/...`. The rule fires once, the redirect lands, done. Every `source` in your config starts with `/` — make it a reflex you never have to think about.
  </CodeVariant>
</CodeVariants>

## Narrowing a rule with `has` and `missing`

So far a config rule is fully request-blind. But there's a narrow band where a rule can peek at the request *without* leaving the edge — and knowing exactly where that band ends is a piece of judgment worth installing carefully.

Every rule can carry a `has` array, a `missing` array, or both. Each is a list of conditions on the request — a cookie, a header, the host, or a query parameter:

<div data-mark-color="blue">

```ts title="next.config.ts" {3}
{
  source: '/app/:path*',
  missing: [{ type: 'cookie', key: 'session_token' }],
  destination: '/welcome',
  permanent: false,
}
```

</div>

Read that rule as one sentence: *when someone hits `/app/...` and there's no `session_token` cookie, send them to `/welcome` instead.* Every entry in `has` must match **and** every entry in `missing` must not match for the rule to fire. The `value` field is optional — omit it (as above) to match on the *presence* of the cookie at all; supply a string to require a specific value (the string is regex-capable, and named groups can flow into `destination`). Beyond cookies, `type: 'host'` is the common one — routing an apex domain versus a `www` subdomain to different destinations, since the host is something the edge can read directly.

Now the line that matters, the one an experienced engineer draws sharply. **`has` and `missing` test presence and literal value at the edge — they do not decode, verify, or trust.** A `has: cookie` check can tell you a `session_token` cookie *exists*; it cannot tell you the session is valid, who the user is, or what they're allowed to see. It's a coarser cousin of the cheap presence check you do in the proxy. So the rule that gated the marketing page on a *missing* cookie above is fine: it's a UX nudge — "you look logged out, here's the welcome page" — and if it's wrong, the worst case is the real app re-checks and corrects course. But anything that *validates* a session, reads the current user, or makes a genuine authorization decision still belongs in `proxy.ts` (cheap presence check) and then the route (the authoritative check).

:::note
This is the same "presence here, real check there" boundary you drew in the previous chapter, now one rung cheaper. A config `has`/`missing` cookie check is fine for redirecting a marketing page to `/dashboard` when a session cookie *exists* — that's a UX optimization, and the dashboard re-checks. Deciding *what a user may see* is never a config rule. If the answer depends on who the user actually is, it's a proxy or route concern, not an edge match.
:::

## Rewrites: changing the implementation, not the URL

Everything so far has been a redirect — the address bar changes. A **rewrite** is the other operation you met last chapter: same `source`/`destination` shape, but the URL the visitor sees stays exactly where it is while the server quietly renders something else behind it. A redirect is visible and costs two round trips; a rewrite is invisible and costs one. You own that distinction already — here it's the same idea, expressed in config.

The shape is almost identical to `redirects()`, with one field removed:

```ts title="next.config.ts"
async rewrites() {
  return [
    { source: '/docs/:path*', destination: '/external-docs/:path*' },
  ];
}
```

There's **no `permanent`** field — a rewrite isn't an HTTP redirect status at all, it's an internal swap, so permanence doesn't apply. The other difference from `redirects()` is the return shape. Returning a **flat array**, as above, is the common case: those rules run *after* Next has tried to match a real route. For the rare time you need rewrites to run *before* route matching — or to fall through only when nothing else matched — `rewrites()` can instead return an object with `beforeFiles`, `afterFiles`, and `fallback` stages that order your rules against the filesystem. Most app teams never need it; reach for the object form only when ordering against real routes becomes the problem, and external proxies typically live in its `fallback` stage. Default to the flat array.

The canonical use is serving content that lives somewhere else under a URL on your own domain — a marketing CMS, a docs site, a help center — so the visitor stays on `app.example.com/docs/...` while the bytes come from an upstream origin. Which brings us to the one sharp edge rewrites have that redirects don't.

<CodeVariants>
  <CodeVariant label="Internal rewrite" icon="lucide:server">
    <div data-mark-color="green">

    ```ts "destination: '/cms/blog/:path*'"
    async rewrites() {
      return [
        { source: '/blog/:path*', destination: '/cms/blog/:path*' },
      ];
    }
    ```

    </div>
    **Cheap — the destination is a route on your own app.** Next just renders `/cms/blog/...` while the URL stays `/blog/...`. No external hop, no proxy cost; it's the same invisible swap you'd do for multi-tenancy, applied to internal routes.
  </CodeVariant>

  <CodeVariant label="External rewrite" icon="lucide:globe">
    <div data-mark-color="orange">

    ```ts "destination: 'https://marketing.example.com/:path*'"
    async rewrites() {
      return [
        { source: '/blog/:path*', destination: 'https://marketing.example.com/:path*' },
      ];
    }
    ```

    </div>
    **Not free — every matched request streams through your deployment to the upstream and back.** This is a real reverse proxy: your server fetches `marketing.example.com/...`, relays the response, and the visitor never sees the swap. Useful, but it has genuine cost — so the matcher has to stay tight, for the reason below.
  </CodeVariant>
</CodeVariants>

When the `destination` is an external URL like `https://marketing.example.com/:path*`, you've built a <Term definition="A server that forwards a request to an upstream origin and relays the response, with the client none the wiser. Your deployment fetches the upstream and streams it back under your own URL.">reverse proxy</Term>. Every matched request now flows through your deployment to the upstream and back — no longer free-at-the-edge, but real compute and bandwidth on every hit. That's fine when the matcher is scoped to the actual prefix you mean to proxy. It's a disaster when it isn't.

:::caution
**Keep the external matcher tight.** An unbounded `:path*` at the root — `source: '/:path*'` → an external origin — captures *everything*, including your `/_next/*` asset requests and your API routes, and proxies them all to the upstream. The app breaks in confusing ways: assets 404, API calls hit a marketing server. Always constrain to the real prefix (`/blog/:path*`, `/docs/:path*`), never the bare root.
:::

One last note on precedence, so you don't define the same rewrite twice. If both a `proxy.ts` rewrite and a config rewrite match the same request, **the proxy wins** — it runs first. So pick one home per rule. A request-blind rewrite (serving the docs site) belongs in config; a request-dependent one (the subdomain-to-org rewrite from last chapter) belongs in the proxy. Don't split a single rewrite across both files.

## `trailingSlash`: a launch-time decision you can't take back

There's one more config flag in this family, and it's unusual because its cost isn't compute — it's *permanence*. `trailingSlash` decides one thing for your entire app: whether the canonical URL form has a trailing slash or not.

<div data-mark-color="blue">

```ts title="next.config.ts" "trailingSlash: false"
const nextConfig: NextConfig = {
  trailingSlash: false,
};
```

</div>

The default, `false`, serves `/about` and treats it as canonical. Flip it to `true` and the app serves `/about/` instead, redirecting `/about` to `/about/`. That's the whole behavior — it picks one of two URL shapes for every route in the app.

Here's why it earns a careful beat rather than a passing mention. That choice bakes into *everything*: every internal link your app generates, every external backlink someone else points at you, every URL search engines have indexed, every link a customer has shared in Slack. Pick a form and the entire web settles around it. **Flip it after launch and you issue a redirect on every previously-canonical URL** — every backlink and bookmark now pays a redirect hop, forever, and search engines re-crawl the whole site to learn the new shape. It's not a crash; it's a permanent paper-cut you inflicted on yourself, and there's no clean way to take it back.

So the reflex is simple: **pick one form at the start of the project and never touch it.** The default, `false`, is the right pick for almost every SaaS — cleaner URLs, and it's the framework default, so it's what every tool and every link assumes anyway. Name it once in the config, lock it, and move on. (And notice *why* this lives in config and not the proxy: it's request-independent and global — the same rule for every visitor on every route — which is exactly the kind of rule this home is for.)

:::caution
Treat `trailingSlash` as a one-time decision made at project start. Flipping it on a live app turns every existing link to a route into a redirect hop and forces a full re-crawl — a permanent tax on links you don't even control. Decide once, lock it, leave it.
:::

## Which home does this rule belong in?

You now have all three homes in view, so it's time to assemble the thing you'll actually carry to work: the decision that routes any URL rule to the right place. Last chapter you saw a two-home version of this — proxy versus `redirect()`. This lesson completes it to three, and the new branch is the one we just spent the whole lesson on.

It comes down to **two questions, asked in a fixed order**. The order is the whole point — ask them in this sequence and a rule sorts itself.

The first question is about the request: *does this rule depend on who's asking* — the session, the geography, an A/B bucket, the host beyond simple matching? If **no** — it's the same for everyone — it belongs in `next.config.ts`, applied at the edge with zero invocation. That's the rebrand, the legacy URL migration, the marketing rename, the docs rewrite. If **yes**, you move to the second question.

The second question is about timing: *does it need to happen before the route renders, conditionally, on every request* — an auth gate, subdomain tenancy, A/B bucketing? If **yes**, it's a `proxy.ts` rule: it runs per request, reads cookies and headers and geo, and pays its invocation for the privilege. If **no** — it's the outcome of an action, or a per-page check — it's `redirect()` / `permanentRedirect()` from `next/navigation`: after a Server Action finishes, or inside a single page that looks something up and bounces.

Presence is request-blind enough for the edge; validation is not.

## The whole config, end to end

Let's consolidate everything into one config you could actually ship at this stage. It carries the always-on flags from the start of this chapter, the rebrand redirect, a cookie-gated redirect, the external docs rewrite, and the locked `trailingSlash` — under thirty lines, every entry defensible. Step through it; each block ties back to the home decision.

<AnnotatedCode lang="ts" maxLines={18} code={`
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  cacheComponents: true,
  typedRoutes: true,
  trailingSlash: false,
  async redirects() {
    return [
      { source: '/account/:path*', destination: '/settings/:path*', permanent: true },
      {
        source: '/app/:path*',
        missing: [{ type: 'cookie', key: 'session_token' }],
        destination: '/welcome',
        permanent: false,
      },
    ];
  },
  async rewrites() {
    return [
      { source: '/docs/:path*', destination: 'https://docs.acme-marketing.com/:path*' },
    ];
  },
  // Security headers (CSP, HSTS) go here once the hardening pass lands.
  // async headers() { ... },
};

export default nextConfig;
`}>
  <AnnotatedStep meta="{4-6}" color="blue">
    **The always-on flags, plus the locked URL shape.** `cacheComponents` and `typedRoutes` are on for every project (lesson 1). `trailingSlash: false` is the one-time launch decision — request-independent and global, which is exactly why it lives here.
  </AnnotatedStep>

  <AnnotatedStep meta="{9}" color="blue">
    **The rebrand — a request-blind 308.** `/account/:path*` → `/settings/:path*`, `permanent: true`. Same for everyone, forever, so it lands in config and applies at the edge for free. The catch-all forwards the whole sub-tree.
  </AnnotatedStep>

  <AnnotatedStep meta="{10-15}" color="blue">
    **A presence-gated redirect.** When `/app/...` is hit with no `session_token` cookie, bounce to `/welcome` — `permanent: false`, because it's a temporary UX nudge, not a permanent move. `missing` only checks the cookie *exists*; the real session check still happens downstream.
  </AnnotatedStep>

  <AnnotatedStep meta="{18-22}" color="blue">
    **The external docs rewrite.** `/docs/:path*` streams from the upstream marketing site while the URL stays on your domain. The matcher is scoped to `/docs` — never the bare root — so assets and API routes don't get proxied along with it.
  </AnnotatedStep>

  <AnnotatedStep meta="{23-24}" color="blue">
    **A signpost, not code.** The commented `headers()` marks where the security baseline (CSP, HSTS) lands in the hardening pass later in the course — carried as a comment so the file is honest about what it doesn't do yet.
  </AnnotatedStep>
</AnnotatedCode>

That's the artifact to carry out of here. Read top to bottom, it's the chapter-wide story in one file: the rule the platform applies before any of your code runs.

And the synthesis the whole lesson was building toward. The config file owns your app's **request-independent** URL shape — the rules that are the same for every visitor, applied at the edge for free. The proxy owns the **request-dependent** shape — the rules that read the session, the host, the bucket, and pay an invocation to do it. `redirect()` owns the **action-time** shape — the rules that fire after application code finishes its work. Three homes, one question to route between them: *does this rule depend on who's asking?* Ask it first, and the rest follows.
