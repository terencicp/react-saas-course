The Acme app renders. Every page carries its title, its description, its social card — you wired all of that in the last lesson, [Metadata and dynamic OG cards](/034-project-config-platform-primitives-and-seo/6-metadata-and-dynamic-og-cards/). So you ship it to a staging URL, run a pre-launch SEO audit, and the report comes back with a list of gaps that have nothing to do with any single page:

- No `robots.txt`. Crawlers have no instructions.
- No `sitemap.xml`. Crawlers can still follow links, but nothing tells them which URLs matter or when they last changed.
- The browser tab shows the default Vercel favicon, not the Acme mark.
- Add the app to an iPhone home screen and you get a blurry screenshot, not an icon.
- The mobile address bar is plain grey instead of the brand color.
- And the dangerous one: the staging deploy is fully indexable. Google is about to put your unfinished app in its index, competing with production for the same keywords.

These are not page-level concerns. They are *site-level* artifacts — the handful of standards-mandated files a crawler, an operating system, or a mobile browser expects to find near the root of your site, at well-known paths, regardless of which page someone is on. The old web answered each one by hand-maintaining a file in a static folder: drop a `robots.txt` here, generate a `sitemap.xml` with a script there, export a favicon at ten sizes and paste ten `<link>` tags into `<head>`. Every one of those was a thing you maintained by hand and forgot to update.

Next.js replaces all of it with **file conventions**: typed functions you write near the root of `app/`, which the platform discovers by filename, runs at build time, caches, and serves at the path the standard expects — or wires into `<head>` for you. That is the spine of this whole lesson, and it is worth stating once so you can hang every section on it:

:::tip
Each of these is the same shape: a file with a framework-reserved name that replaces a hand-maintained `/public` artifact with a **typed**, **dynamic**, **cached-by-default** function whose output the CDN serves. Typed: you return a `Robots` or `Sitemap` object, autocompleted and type-checked, so a typo is a build error. Dynamic: the function can read the environment or query the database. Cached: it runs once at build and is served from the edge — *unless* you reach for a request-time API, which silently flips it to per-request, the same Cache Components reflex from the rest of this chapter.
:::

By the end you will be able to stand up the complete root SEO and platform bundle for a new SaaS — the marketing crawl-control files, the icon set, the install manifest, the mobile viewport — knowing which filename owns which output, and carrying the reflex to keep each one pure so the platform can cache it. We will build it up file by file and assemble the whole thing into one concrete `app/` directory at the end.

## The file-convention map

Before any single file, install the map. Every file in this lesson shares the same four properties — it is discovered by its filename, returns a typed value (or is a typed image), runs as a cached route handler, and is auto-wired into `<head>` or served at a well-known URL. The only thing that changes from file to file is *which artifact it owns*. So the most useful thing you can hold in your head is not the syntax of any one file — it is the mapping from filename to output.

## robots.ts: telling crawlers where to go

A <Term definition="An automated program that fetches pages to build a search index — Googlebot, Bingbot, and the crawlers behind social-preview cards and AI tools.">crawler</Term> arriving at `app.acme.com` looks first for `/robots.txt`: a plain-text file that tells it which paths it may fetch and where to find your sitemap. That file is the <Term definition="The robots.txt format. A site-wide, voluntary protocol telling well-behaved crawlers which paths they may and may not request.">Robots Exclusion Standard</Term>, and in Next.js you no longer hand-write it. You write `app/robots.ts`, a default-exported function that returns a typed `Robots` object, and the platform serves the rendered text at `/robots.txt`.

The minimal shape is small. You declare rules — which user-agents, what they may and may not crawl — and point at your sitemap and canonical host:

```ts title="app/robots.ts"
import type { MetadataRoute } from 'next';

export default function robots(): MetadataRoute.Robots {
  return {
    rules: { userAgent: '*', allow: '/', disallow: '/api' },
    sitemap: 'https://app.acme.com/sitemap.xml',
    host: 'https://app.acme.com',
  };
}
```

That is the entire object. `userAgent: '*'` is every crawler, `allow: '/'` opens the whole site, `disallow: '/api'` keeps bots out of your route handlers, and `sitemap` and `host` hand the crawler your map and your canonical origin. The shape is typed — `MetadataRoute.Robots` autocompletes every field, and a misspelled `dissallow` is a red squiggle, not a silently-ignored line in a text file.

Note the `export default`. The course's rule everywhere else is named exports — but these file conventions are the carve-out, because the framework finds them by filename and expects a default export, exactly like `layout.tsx`, `page.tsx`, and last lesson's `opengraph-image.tsx`. Every file in this lesson follows that same framework-dictated shape.

That much is mechanical. The senior decision in this file is not the shape — it is the branch. **A preview deploy must not be indexable.** This is the gap the audit flagged, and it is a real production incident, not a style preference, so it earns the depth.

Here is the failure. Every push to a feature branch gets its own Vercel preview URL, a full production build at a public address. If your `robots.ts` returns `allow: '/'` unconditionally, that preview is fully crawlable. Google indexes your half-finished pages and — worse — those preview URLs now compete with your real domain for the same keywords. That is <Term definition="When the same content is reachable at multiple URLs, search engines split ranking signals across them and may index the wrong one. A leaked staging deploy is the classic cause.">duplicate content</Term>: search engines see two copies of your site and can rank the staging one over production.

So one `robots.ts` reads which environment it is running in and branches. Walk through it:

<AnnotatedCode lang="ts" code={`
import type { MetadataRoute } from 'next';

const ORIGIN = 'https://app.acme.com';

export default function robots(): MetadataRoute.Robots {
  if (process.env.VERCEL_ENV !== 'production') {
    return { rules: { userAgent: '*', disallow: '/' } };
  }

  return {
    rules: { userAgent: '*', allow: '/', disallow: ['/api/', '/dashboard/'] },
    sitemap: \`\${ORIGIN}/sitemap.xml\`,
    host: ORIGIN,
  };
}
`}>
  <AnnotatedStep meta="{6}" color="orange">
    The whole file pivots on this one line. `process.env.VERCEL_ENV` is set by Vercel to `'production'`, `'preview'`, or `'development'`. Every environment that is not production takes the early return.
  </AnnotatedStep>

  <AnnotatedStep meta="{7}" color="red">
    For any non-production build, return one rule: `disallow: '/'`. That blocks every crawler from every path — no sitemap, no host, nothing to index. Your previews stay invisible.
  </AnnotatedStep>

  <AnnotatedStep meta="{11}" color="green">
    Only production reaches the real rules. `allow: '/'` opens the public surface; `disallow` carves out the paths bots should never touch — the API and the signed-in dashboard.
  </AnnotatedStep>

  <AnnotatedStep meta={`{12-13} "ORIGIN"`} color="blue">
    `sitemap` and `host` point at the absolute production origin. Hard-code that canonical origin rather than deriving it from the incoming request host — the request host on a preview deploy is the preview URL, and you never want a crawler told that the canonical site lives at a throwaway address.
  </AnnotatedStep>
</AnnotatedCode>

The load-bearing detail is the discriminator on line 6, and it is the one place people get this wrong. The obvious instinct is to check `process.env.NODE_ENV === 'production'` — and that is a trap, because on Vercel a preview deploy *is* a production build, so `NODE_ENV` is `'production'` on previews too. Gating on `NODE_ENV` would leave every preview indexable, the exact bug you are trying to prevent. The signal that actually distinguishes the three is `VERCEL_ENV`. Reading `process.env` directly like this works, but it is stringly-typed and easy to typo; a later chapter wraps environment variables in a validated, typed `env` object, and this read is one of the first things to move there.

Get this branch backwards and the stakes cut the other way, which is why the discriminator is worth this much care. Ship `disallow: '/'` to *production* and you have told Google to delist your entire site. Nothing breaks, nothing errors — the build is green, the page renders — and then weeks later organic traffic quietly craters and someone goes looking for why. An invisible, slow-motion outage is the worst kind. The branch is one line; it is the most consequential line in this lesson.

:::caution
`Disallow` is a polite request, not a lock. A well-behaved crawler honors it; a hostile one ignores it and fetches the path anyway. It is not access control. Private routes — your dashboard, your API — need real authentication, which is a later unit; a robots rule only keeps honest bots from *indexing* a path, never from *reaching* it.
:::

## sitemap.ts: the index of what to crawl

`robots.txt` tells crawlers where they may go; the <Term definition="An XML file listing the URLs you want crawled, each with optional last-modified and change-frequency hints. Crawlers use it to discover and prioritize pages instead of guessing from links.">sitemap</Term> tells them where to look. It is an XML file listing the URLs you want indexed, with hints about when each last changed. You write `app/sitemap.ts`, return an array of entries, and Next.js emits valid sitemap XML at `/sitemap.xml`.

The shape is an array, one entry per URL:

```ts title="app/sitemap.ts"
import type { MetadataRoute } from 'next';

const ORIGIN = 'https://app.acme.com';

export default function sitemap(): MetadataRoute.Sitemap {
  return [
    { url: `${ORIGIN}/`, changeFrequency: 'weekly', priority: 1 },
    { url: `${ORIGIN}/pricing`, changeFrequency: 'monthly', priority: 0.8 },
    { url: `${ORIGIN}/blog`, changeFrequency: 'daily', priority: 0.6 },
    { url: `${ORIGIN}/help`, changeFrequency: 'weekly', priority: 0.5 },
  ];
}
```

Each entry needs an absolute `url` — relative paths are invalid in a sitemap. `changeFrequency` and `priority` are hints, not commands; crawlers weigh them loosely. `lastModified` is the field that earns its keep, and it is missing here on purpose, because for static marketing pages there is nothing meaningful to put. The moment your content is data-driven, that changes.

Because a real SaaS is data-driven. Your public surface is not only four hand-listed marketing routes — it is every published blog post, every public help article, every public profile, and that set grows every time someone hits publish. Hard-coding those URLs means the sitemap is stale the moment content ships. So the function goes `async` and queries the indexable rows:

<CodeVariants>
  <CodeVariant label="Static only">
    ```ts title="app/sitemap.ts"
    export default function sitemap(): MetadataRoute.Sitemap {
      return MARKETING_ROUTES.map((path) => ({
        url: `${ORIGIN}${path}`,
        changeFrequency: 'weekly',
      }));
    }
    ```
    **Fine until content ships.** Four routes, known at build, never out of date — because they never change. A blog or help center breaks this the first time someone publishes.
  </CodeVariant>

  <CodeVariant label="Marketing + public rows">
    <div data-mark-color="green">

    ```ts title="app/sitemap.ts" "async" "await" {15}
    export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
      const articles = await listPublicHelpArticles();

      const staticEntries = MARKETING_ROUTES.map((path) => ({
        url: `${ORIGIN}${path}`,
        changeFrequency: 'weekly' as const,
      }));

      const articleEntries = articles.map((article) => ({
        url: `${ORIGIN}/help/${article.slug}`,
        lastModified: article.updatedAt,
        changeFrequency: 'monthly' as const,
      }));

      return [...staticEntries, ...articleEntries];
    }
    ```

    </div>
    **Always current.** The function awaits a read of the published help articles and maps each row to an entry. `lastModified` is the row's real `updatedAt`, so an edited article tells crawlers to recrawl. Static and dynamic entries concatenate into one array.
  </CodeVariant>
</CodeVariants>

Two things about that database version matter beyond the syntax. First, `listPublicHelpArticles()` is a read helper from your data layer, not an inline query — the course keeps every database read behind a verb-named function in `db/queries/`, so the call site stays this clean and the query lives in one place. Second, and this is the rule that keeps a sitemap correct: **only publicly indexable rows belong in it.** Never authenticated data, never tenant-scoped data, never anything behind a login. Putting a private invoice URL in your sitemap both leaks its existence and is useless — a crawler can't fetch it anyway. Public means public.

Now the property that ties this file back to the rest of the chapter. `sitemap.ts` is a special route handler, and under Cache Components it is **static by default** — the build renders it once, the CDN serves that one artifact to every crawler. That is exactly what you want: a crawl is a cache hit, zero function invocations. But it is fragile in a specific way.

:::caution
Reach for a request-time API inside `sitemap.ts` — `cookies()`, `headers()`, or an uncached fetch — and you flip it from static to dynamic. Now it re-runs that database query on *every* crawler hit instead of once at build. A bot that crawls aggressively turns one cached file into thousands of live queries. Keep `sitemap.ts` pure: derive everything from build-time data, and let the build-time database read be its only data source.
:::

This is the same reflex from earlier in the chapter — static by default, request-time reads opt you out — applied to a file you might not have thought of as a render. You will see it again in a moment as the rule that governs *every* file in this lesson.

One scale note, for recognition only. Google rejects a single sitemap over 50,000 URLs or 50 MB. If you genuinely have that many indexable pages, the `generateSitemaps` function splits the output into numbered files — `/sitemap/0.xml`, `/sitemap/1.xml` — plus an index that points at them. Most SaaS apps never come close; know the threshold exists so you recognize the pattern in a large codebase, and reach for it only if you actually cross 50,000 URLs.

## Icons: favicon, icon, and apple-icon

A <Term definition="The small icon shown in a browser tab, bookmark, and history entry. Historically a single favicon.ico file at the site root.">favicon</Term> is the small mark in the browser tab. The same idea now extends across devices: a high-resolution icon for the tab, a larger one for Android and PWA installs, and a dedicated tile for the iOS home screen. Historically you exported every size, named them by hand, and pasted a wall of `<link>` tags into `<head>`. Next.js collapses all of it into three file conventions: you drop image files in `app/` with reserved names, and the platform fingerprints them, caches them, and injects the right `<link>` tags.

Three filenames, three jobs:

<Figure>
  <IconConventionsTable />
  <Fragment slot="caption">
    `favicon.ico` lives only at the root of `app/`. `icon` and `apple-icon` can sit in any route segment to override per-section. Apple touch icons must be raster — no SVG, no `.ico`.
  </Fragment>
</Figure>

The default path is the boring one, and it is the right one: drop a static image in `app/` and you are done. The platform reads the file, hashes its contents for cache-busting, caches it on the CDN, infers the `type` and `sizes` from the actual bytes, and injects the `<link>` tag. No manual `<head>`, no manual fingerprinting, no stale-icon-after-deploy problem. Reach for anything fancier only when you have a real reason — and the static path covers nearly every app.

Real devices want more than one size, though. A 32-pixel tab icon and a 512-pixel install icon are different files. You set multiple icons with a **numeric suffix** on the filename: `icon.png`, `icon1.png`, `icon2.png`, and so on, sorted lexically. The filenames carry no dimension information — the platform reads each file's real pixel size and infers the `sizes` attribute from the bytes themselves. The practical set for a SaaS covers the common targets: a high-resolution `icon` (around 512) for crisp tabs and PWA splash, a mid-size `icon` (192) for Android and home-screen installs, an `apple-icon` for the iOS tile, and the legacy `favicon.ico` for the broadest browser support.

Here is what that looks like in `app/`:

<FileTree>
- app/
  - favicon.ico legacy single-file favicon, broadest browser support
  - icon.png high-res mark, `512×512` — crisp tabs and PWA splash
  - icon1.png mid-size mark, `192×192` — Android / PWA installs
  - apple-icon.png iOS home-screen tile, raster only
</FileTree>

One device note that is not a Next.js rule. The iOS home screen scales your `apple-icon` up to fill its tile, and an undersized source renders blurry — Apple's tiles expect something generously sized (around 180×180 or larger). Next.js will not reject a small file; iOS just makes it look bad. Ship a comfortably large raster <Term definition="The icon iOS uses when you add a site to the home screen, set via an apple-icon file or a rel='apple-touch-icon' link.">apple touch icon</Term> and you avoid it. And remember the format constraint from the table: apple-icon must be raster — png or jpg — because Apple's tiles don't take SVG.

You can also *generate* an icon instead of shipping a static file: name it `icon.tsx` or `apple-icon.tsx` and default-export a function that returns an `ImageResponse`, the same generated-image API you met in the last lesson for OG cards. The reach here is narrow — per-organization branded favicons, an A/B-tested mark — and rare for app surfaces. Know it exists; the static file is the default for almost everything.

The same applies to the social-card images from the last lesson. A root-level `opengraph-image.png` (paired with `opengraph-image.alt.txt` for its alt text) is the brand-wide fallback every route inherits unless it overrides — and that root brand fallback is the piece the last lesson deferred to here, because it belongs in this site-level bundle. One watch-out: keep that fallback **root-only**. Drop `opengraph-image` files into multiple nested layouts and they collide silently, with no error to tell you which one won. One brand fallback at the root, per-page overrides where a page needs them — that is the whole rule. Everything about generating those images lives in the last lesson; this is only about where the brand fallback sits in the bundle.

<Term definition="A web app manifest — JSON describing how the app behaves when installed: name, icons, display mode, theme colors. The basis for 'Add to Home Screen' without a full Progressive Web App.">PWA</Term> is going to come up in the manifest section; we will define it properly there. For now: the icon set you just declared is exactly what the install manifest will point at.

## The viewport export: the field that isn't metadata

There is one field in this whole area that trips up nearly everyone, because it looks like metadata and isn't. It controls how your page renders on mobile and what color tints the browser chrome — and if you put it where it looks like it belongs, inside your `metadata` object, you get a build warning. So lead with the rule, because the rule is the whole point of this section:

**Viewport-affecting fields go in a separate `export const viewport`, never inside `metadata`.**

The fields in question are `width`, `initialScale`, `themeColor`, and `colorScheme`. Here is the *why*, because the why is what makes the rule stick. The `metadata` object owns the SEO and social tags — title, description, Open Graph, canonical. Those describe your page to crawlers and link unfurlers. But `width` and `themeColor` drive a different family of `<meta>` tags — `viewport`, `theme-color`, `color-scheme` — that control how the browser physically renders the page on a device. Different purpose, different cadence, different lifecycle. So Next.js split them into their own typed export rather than letting them ride along in `metadata`. The canonical shape:

```ts title="app/layout.tsx"
import type { Viewport } from 'next';

export const viewport: Viewport = {
  width: 'device-width',
  initialScale: 1,
  themeColor: '#0f172a',
  colorScheme: 'light dark',
};
```

Each field, briefly. `width: 'device-width'` with `initialScale: 1` is the <Term definition="The meta viewport tag. Tells a mobile browser to render at the device's real width instead of zooming out a desktop-width layout. width=device-width, initial-scale=1 is the universal mobile-correct default.">viewport meta</Term> default that makes a layout render at the phone's real width instead of zooming out a desktop page — the scaffold ships it, and you should not fight it. `colorScheme: 'light dark'` declares the app supports both color schemes. And `themeColor` is the <Term definition="The theme-color meta tag. Tints the mobile browser's UI — the address bar and status bar area — to a color you choose, so the chrome matches your app's surface.">theme-color</Term>: it tints the mobile browser's chrome — the Safari and Chrome address bar — to match your brand surface instead of leaving it default grey. That last one is the gap the audit flagged.

`themeColor` carries the one real decision in this section. A single color string tints the chrome one way regardless of theme. But a SaaS with both a light and a dark theme wants the address bar to match the *active* theme — dark chrome in dark mode, light chrome in light mode. For that, `themeColor` takes an array of `{ media, color }` entries keyed on the color-scheme media query:

```ts title="app/layout.tsx"
export const viewport: Viewport = {
  width: 'device-width',
  initialScale: 1,
  colorScheme: 'light dark',
  themeColor: [
    { media: '(prefers-color-scheme: light)', color: '#ffffff' },
    { media: '(prefers-color-scheme: dark)', color: '#0f172a' },
  ],
};
```

The single string is the simple default; the array is the senior pick for a theme-switching app, so the chrome never clashes with the surface beneath it.

The footgun is mechanical, which is why it is worth drilling. Putting a viewport field inside `metadata` does not error — the build still succeeds, the tag may even still emit. It *warns*, and the warning is easy to scroll past. But it is the deprecated path, gone since Next 14 and still flagged through 16, and it will eventually break. The fix is to move the field out. The clearest way to see it is side by side — the wrong shape and the right one:

<CodeVariants>
  <CodeVariant label="The warning">
    <div data-mark-color="red">

    ```ts title="app/layout.tsx" {4-5}
    export const metadata: Metadata = {
      title: 'Acme',
      description: 'Invoicing for small teams',
      themeColor: '#0f172a',
      viewport: { width: 'device-width', initialScale: 1 },
    };
    ```

    </div>
    **Deprecated, silently.** `themeColor` and `viewport` inside `metadata` trigger a build warning, not an error — the page still builds, so it's easy to miss. This shape is on the way out.
  </CodeVariant>

  <CodeVariant label="The fix">
    <div data-mark-color="green">

    ```ts title="app/layout.tsx" {6-10}
    export const metadata: Metadata = {
      title: 'Acme',
      description: 'Invoicing for small teams',
    };

    export const viewport: Viewport = {
      width: 'device-width',
      initialScale: 1,
      themeColor: '#0f172a',
    };
    ```

    </div>
    **Split into two exports.** SEO and social fields stay in `metadata`; viewport and theme-color move to their own `viewport` export. Same `layout.tsx`, two sibling exports — the warning is gone.
  </CodeVariant>
</CodeVariants>

Two sibling exports in the same `layout.tsx` — that is the shape you want, and it is where the `viewport` export lands next to the `metadata` export the last lesson set up.

There is a `generateViewport` function, the dynamic counterpart to `generateMetadata`, for the rare case of a route-dependent viewport — a per-locale `colorScheme`, say. It takes the same Promise-shaped `params` you saw with `generateMetadata`. But it carries a sharper constraint than its metadata sibling, and the constraint is the reason to almost never use it: **viewport cannot be streamed.** It affects the initial paint, so a `generateViewport` that reads runtime data blocks the *entire route* — no static shell, nothing renders until the read resolves. Metadata can stream in late; viewport can't. That makes "keep viewport static" a stronger rule than "keep metadata static." For the common case, the static `viewport` object is the only right answer.

## The web manifest: add to home screen

The last gap is the home-screen experience. When someone taps "Add to Home Screen" on the Acme app, the operating system wants a name, an icon, and a color to build the launcher tile and splash screen. That information lives in a <Term definition="A web app manifest — a JSON file the browser reads to learn how to present an installed web app: its name, icons, display mode, and colors.">web app manifest</Term>, and like everything else in this lesson it has a file convention: `app/manifest.ts`, a default-exported function returning a typed manifest object. The platform serves it at `/manifest.webmanifest` and injects `<link rel="manifest">`.

A minimal manifest reuses the brand strings and the icon set you already declared:

```ts title="app/manifest.ts"
import type { MetadataRoute } from 'next';

export default function manifest(): MetadataRoute.Manifest {
  return {
    name: 'Acme — Invoicing for small teams',
    short_name: 'Acme',
    description: 'Send and track invoices.',
    start_url: '/',
    display: 'standalone',
    background_color: '#0f172a',
    theme_color: '#0f172a',
    icons: [
      { src: '/icon.png', sizes: '512x512', type: 'image/png' },
      { src: '/icon1.png', sizes: '192x192', type: 'image/png' },
    ],
  };
}
```

The senior framing: a basic manifest costs almost nothing and unlocks real value — the app becomes installable, with a home-screen icon, a <Term definition="A manifest display mode that runs the installed app full-screen with no browser address bar or tabs, so it looks like a native app.">standalone display</Term> mode that drops the browser chrome, and splash colors that match your brand. The `icons` array points at the 192 and 512 marks you already shipped — the two sizes installers reach for.

Be clear about the boundary, though, so you don't over-scope it. A manifest is *install metadata*, not a Progressive Web App. A full PWA — service worker, offline support, push notifications — is a much deeper topic and out of scope here. The manifest gives you "Add to Home Screen" and a branded launcher; it does not make your app work offline. Name that line so you reach for exactly the cheap win and stop there.

A static `public/manifest.webmanifest` is the alternative — and the `.ts` convention wins for the same reasons every other file in this lesson does: it is typed, and it can pull brand strings and the icon list from one shared config instead of duplicating them across a hand-maintained JSON file.

## These are cached route handlers: keep them pure

Step back, because the most transferable thing in this lesson is not any one file — it is the property they all share. Every file you just met — `robots.ts`, `sitemap.ts`, `icon.tsx`, `apple-icon.tsx`, `manifest.ts`, `opengraph-image.tsx` — is a special route handler. Under Cache Components, all of them are **statically generated at build and served from the CDN by default.** Zero per-request work. A crawler hitting `/sitemap.xml`, a browser fetching `/icon.png`, a phone reading `/manifest.webmanifest` — every one of those is a cache hit on a file the build produced once.

They flip to dynamic — re-running per request — only if you make them. The trigger is the same one from the rest of the chapter: read a request-time API (`cookies()`, `headers()`, an uncached external fetch) or set a dynamic route-segment config, and the platform has to re-render the file on every request because its output now depends on the request. The whole value proposition — one cached artifact, served forever — collapses the moment a request-time read sneaks in. A `sitemap.ts` that calls `cookies()` runs its database query on every crawl. A `robots.ts` that reads a header recomputes per request. You have seen this reflex before, with the cache model earlier in the chapter and with OG-image caching last lesson; it applies to every file here. Viewport is the one place it applies *harder*, not softer.

Why harder: viewport can't be streamed — it gates the initial paint — so a runtime-reading `generateViewport` blocks the entire route with no static shell at all. A dynamic `sitemap` just costs an invocation per crawl; a dynamic viewport costs your whole page its first paint. If you genuinely need external (not request-time) data in a viewport, the escape hatch is to cache that read inside the function, but the honest answer is almost always to keep `viewport` a static object and not reach for the dynamic form.

Test the reflex. Sort each of these into whether it stays static — built once, served from the CDN — or flips dynamic, running on every request:

  <Bucket name="static" label="Stays static" description="Built once, served from the CDN" />
  <Bucket name="dynamic" label="Flips dynamic" description="Re-runs on every request" />

  <Item bucket="static">`sitemap.ts` returning a hardcoded array of routes</Item>
  <Item bucket="static">`robots.ts` branching on `process.env.VERCEL_ENV`</Item>

The discriminator the exercise drills is the one to carry out of this lesson: reading the *environment* is build-time and stays static (that is why the env-aware `robots.ts` is still a cached file), but reading the *request* — cookies, headers, live data — is what opts you out.

One practical corollary, for the generated-image files specifically. A dynamically-generated OG card or icon — one that renders through `ImageResponse` — is cold on its first request: the render takes a few hundred milliseconds the first time, before the result is cached. If a bot scrapes that URL before anyone has warmed it, the bot waits. The fix is a post-deploy hook that pings your key OG URLs to warm the CDN before traffic arrives. Static image files never need this — they are already bytes on disk. It only applies to the `ImageResponse`-generated ones from the last lesson; know the angle exists for when you reach for dynamic generation.
