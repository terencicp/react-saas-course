    You need a two-column header — logo on the left, a "Open dashboard" link on the right. You write it with `<div className="flex justify-between">`, and it looks perfect in the preview server. Why is this still the wrong call?

    <McqChoice correct>The preview renders in Chrome, which honors `flex`; Gmail discards it and collapses the two columns into one stack. Use `<Row>` and `<Column>`, which compile to the table layout every client understands.</McqChoice>
    <McqChoice>`flex` works in every client, but `<div>` is disallowed in React Email templates — you must use `<Section>` as the outer wrapper for the styles to inline.</McqChoice>
    <McqChoice>The `<Tailwind>` component would throw a build error on `flex`, so the template never compiles.</McqChoice>

    <McqWhy>The trap is that the preview lies: it's a Chrome renderer, so `flex` looks aligned there, then Gmail throws the flex away and stacks everything into one column with no warning. Horizontal layout in email is `<Row>` plus `<Column>`, which emit table cells. `<div>` itself is fine, and `<Tailwind>` doesn't error on unsupported utilities — it silently drops them, which is exactly why the failure is so easy to miss.</McqWhy>

    Your brand color is defined in the web app as an OKLCH value in a Tailwind v4 `@theme` block. What has to happen for it to render reliably on a `<Button>` across mailbox clients?

    <McqChoice correct>Mirror the token into a JS config (`emailTailwindConfig`) as a hex value, because some clients can't parse OKLCH and silently drop the whole `background-color`, leaving the button with no fill.</McqChoice>
    <McqChoice>Pass the web app's CSS `@theme` block to `<Tailwind>`'s `config` prop so the same OKLCH tokens are reused without duplication.</McqChoice>
    <McqChoice>Nothing — `<Tailwind>` converts OKLCH to a client-safe color space automatically at render time.</McqChoice>

    <McqWhy>Two things bite here. The `<Tailwind>` component is configured through a JS object, not a CSS `@theme` block, so the web tokens can't be reused directly. And a client that can't parse OKLCH doesn't fall back gracefully — it drops the property entirely, so the brand button renders with no background. The fix is a hand-maintained config mirroring each token as hex, which renders everywhere.</McqWhy>

    A teammate inlines the logo as a base64 data URL so there's "nothing to host." Beyond the image possibly not appearing, what's the more serious consequence?

    <McqChoice correct>The encoded bytes can push the message past Gmail's 102 KB limit, which clips it — hiding everything after the cut, including the footer's unsubscribe link and legal address, behind a link most people never tap.</McqChoice>
    <McqChoice>Base64 images break the auto-generated plain-text part, so the message ships HTML-only and fails spam filters.</McqChoice>
    <McqChoice>Inline images force Outlook to render at natural size, blowing past the 600px column.</McqChoice>

    <McqWhy>The hard wall is the 102 KB Gmail clip: a base64 logo can eat the budget by itself, and when Gmail clips it buries the unsubscribe link and legal address behind "View entire message" — a compliance problem on top of a visual one. The plain-text part is derived from the JSX regardless of image encoding, and the natural-size issue is about missing `width`/`height` attributes, not base64.</McqWhy>

  <MultipleChoice src="49.1">
    Which lines belong *inside* a `WelcomeEmail` template, and which belong in the Server Action that calls it? Select every statement that's correct.

    <McqChoice correct>Reading `firstName` and `verifyUrl` from props belongs inside the template — it's pure presentation.</McqChoice>
    <McqChoice correct>Building `verifyUrl` from `process.env.APP_URL` and a token belongs in the Server Action, not the template.</McqChoice>
    <McqChoice>The template should query the database for the user record when only an id is passed, to keep the call site simple.</McqChoice>
    <McqChoice correct>Keeping the template props-only is what lets the same file render unchanged in the preview, in a test, and in a real send.</McqChoice>

    <McqWhy>A template is a pure renderer: every dynamic value arrives as a prop, and it never reads env, the session, or the database. The caller computes those and passes finished values in. That discipline is exactly what makes the one file render identically across the preview server, a unit test, and production — no `if preview / else production` branching anywhere.</McqWhy>

    The preview server's dark-mode toggle shows your template looking clean in dark mode. What can you correctly conclude?

    <McqChoice correct>That the obviously-broken cases are caught, but not that dark mode is correct everywhere — the toggle only *emulates* `prefers-color-scheme: dark` and can't reproduce Gmail Android's blanket inversion. Confirm with a real test-send.</McqChoice>
    <McqChoice>That the email is confirmed correct in dark mode on every client, since the toggle renders the same way real clients do.</McqChoice>
    <McqChoice>Nothing useful — the toggle is purely decorative and should be ignored in favor of the test-send.</McqChoice>

    <McqWhy>Calibrate trust precisely: the toggle emulates a preference-respecting client, so it's great for catching the obvious failures (a white logo vanishing on an inverted background) cheaply. It's neither authoritative nor useless — it narrows the problem, the gate closes it.</McqWhy>

    You're about to declare a template done. Order matters: which check sits *last* in the loop, and why isn't it just another preview pass?

    <McqChoice correct>A test-send to real inboxes you actually open — it's the only thing that reveals what a real client does (dark-mode inversion, the Outlook VML button path) that no local render can fake.</McqChoice>
    <McqChoice>Toggling the mobile viewport — once it wraps cleanly at 375px, every other client follows.</McqChoice>
    <McqChoice>Reading the HTML view to confirm the `<Tailwind>` classes inlined — once the styles are on the wire, rendering is guaranteed.</McqChoice>
    <McqChoice>Editing `PreviewProps` to a long name and URL to stress the layout — the worst-case render proves the rest.</McqChoice>

    <McqWhy>The mobile toggle, the HTML view, and the props-panel stress test are all part of the fast inner loop — they catch layout and content issues locally and cheaply. But the loop has a ceiling: it can't reproduce a real client's inversion or Outlook's VML rendering. The test-send is the verification gate, run once at the end against inboxes you genuinely read, before the template is wired into a Server Action.</McqWhy>

  <MultipleChoice src="49.3">
    The Resend SDK auto-derives the `text/plain` body from the `react` node you send. Given that, what's the correct working habit for the plain-text version?

    <McqChoice correct>Read it in the preview's plain-text tab as a standalone message, and fix any incoherence in the HTML JSX — there's no separate text file, the text is downstream of the JSX.</McqChoice>
    <McqChoice>Hand-write a parallel `text` string for every send so you control exactly what plain-text readers see.</McqChoice>
    <McqChoice>Call `render(<T/>, { plainText: true })` on the send path to guarantee the text part is generated.</McqChoice>

    <McqWhy>Because the text is generated from the JSX, a bad text version is a bug in the HTML, and the fix lives in `welcome.tsx` — reword the copy, fix the link text, adjust an `alt`. Hand-maintaining a parallel string drifts from the HTML within a release or two. And the `plainText` flag on `render` is deprecated in favor of `toPlainText(html)`; on the send path you pass the node and get both bodies for free.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="49.3">
    The plain-text tab shows the decorative sparkle image's filename leaking in as a stray line above the greeting. What's the right fix, and what's the trap?

    <McqChoice correct>Set the decorative image's `alt` to an explicit empty string — that drops it from the text. The trap is doing the same to an *informational* image, which silences real content for screen-reader users.</McqChoice>
    <McqChoice>Delete the offending line directly in the generated text body — it's faster than touching the JSX.</McqChoice>
    <McqChoice>Shrink the image with smaller `width`/`height` so the generator emits less text for it.</McqChoice>

    <McqWhy>An explicit `alt=""` marks an image as decorative, so the generator skips it. There's no separate text file to edit — it's re-derived from the JSX every render. The trap is using `alt=""` to "clean up" an *informational* image: that empties the alt to silence the text and blinds screen-reader users to real content. `width`/`height` are layout only and have no bearing on the text body.</McqWhy>
  </MultipleChoice>

    You ship a CTA whose text only clears 4.5:1 contrast *after* a client applies your `@media (prefers-color-scheme: dark)` styles. In its light form it sits at 3:1. Is the template accessible enough to ship?

    <McqChoice correct>No — the *light* template must meet WCAG contrast on its own. Most clients ignore the dark preference (and some Gmail clients strip `<style>` blocks), so dark mode is a courtesy layer, never the contrast strategy.</McqChoice>
    <McqChoice>Yes — as long as the message clears 4.5:1 in *some* rendering, it satisfies WCAG AA for that text.</McqChoice>
    <McqChoice>Yes, provided the `color-scheme` meta tags are present so the dark styles are guaranteed to apply.</McqChoice>

    <McqWhy>A message that only reaches AA after a dark transform fails contrast for the majority of readers whose clients ignore the preference — and some Gmail clients strip `<style>` blocks outright, so the dark rules may never arrive. The light, inline-styled template has to pass on its own. Dark mode sits on top of a template that already passes; it never makes one pass.</McqWhy>
  </MultipleChoice>

    You add Tailwind `dark:` utilities to your CTA but put nothing else in `<Head>`. A user opens it in Apple Mail with the OS set to dark. What happens?

    <McqChoice correct>The dark styling is ignored — Apple Mail only applies your dark styles once the template declares it's dark-aware via the `color-scheme` and `supported-color-schemes` meta tags. The plumbing is the opt-in.</McqChoice>
    <McqChoice>Apple Mail applies the `dark:` styles automatically, because it always honors `prefers-color-scheme`.</McqChoice>
    <McqChoice>Apple Mail fully inverts the whole message regardless, so the `dark:` styles are redundant.</McqChoice>

    <McqWhy>The head plumbing is the opt-in: without `<meta name="color-scheme">` and `<meta name="supported-color-schemes">`, Apple Mail won't apply your dark styles even with the user in dark mode — your `dark:` (or `@media`) rules are simply ignored. Apple Mail is a "no transformation" client, so it doesn't blanket-invert; the styling rides on top of the declared plumbing.</McqWhy>
