The last lesson left you holding a shippable `emails/welcome.tsx` and three nagging questions. Does the heading wrap awkwardly at 600px? Is the button readable in dark mode? Is the preheader actually what you intended? You've read that template as code. You haven't seen it.

And the honest answer to all three is uncomfortable: the only place those answers are truly *real* is the inbox. Not your editor, not a browser tab — an actual mail client rendering your actual message. That's the same lesson the `flex` trap taught you: Chrome renders the JSX one way, Gmail renders the wire HTML another, and Chrome is the one lying.

But "the inbox is the only source of truth" can't mean *send from staging, alt-tab to Gmail, refresh, squint, change one word, repeat*. That loop is slow, it pollutes a real inbox with junk, it burns your send quota, and after all that it still only shows you one client out of a dozen. You'd never iterate on a web page that way — you'd never tolerate a five-minute round trip to check whether a heading wraps.

So the team works with two tools, and the whole point of this lesson is to install both and, more importantly, to teach you exactly when each one can be trusted:

- A **local preview server** — the fast inner loop. Save a file, watch it re-render in a second. This is where you do ninety percent of the work, catching layout and content problems in a sub-second cycle.
- A **test-send** — the verification gate. A real message to real inboxes you actually open, run once at the end, to confirm what no local preview can fake.

By the close of this lesson you'll have a `pnpm email dev` script, a running preview server, and a repeatable loop you run on every template before it ships. The trap to avoid is conflating the two tools — treating the fast preview as if it were authoritative. It isn't, and the moment you forget that is the moment a broken dark-mode button reaches a customer. Hold the line between *iterate* and *verify* and the rest of the lesson falls into place.

## Booting the preview server

Start with the payoff, because it arrives in two steps and one command.

<Steps>

1. Add a script to `package.json`. The `react-email` package ships a command-line binary called `email`; this one line exposes its subcommands — `dev`, `build`, `export` — through `pnpm email …`.

   ```json title="package.json"
   {
     "scripts": {
       "email": "email"
     }
   }
   ```

2. Run it.

   ```bash
   pnpm email dev
   ```

   This boots a local server at `http://localhost:3000`, scans your `emails/` directory, lists every template it finds in a left-hand panel, and renders the one you select. Open the URL, click `welcome.tsx`, and there it is — the template you wrote last lesson, rendered.

</Steps>

That moment is worth pausing on, because it's the bridge between the two lessons. The server renders your template using its `PreviewProps`. Recall that last lesson you co-located mock data right beside the component:

```tsx
WelcomeEmail.PreviewProps = {
  firstName: 'Ada',
  verifyUrl: 'https://yourapp.com/verify/abc123',
} satisfies WelcomeEmailProps;
```

That's not decoration. It's the *reason the server can show you anything at all*. A template is a pure function of its props — with no values, there's nothing to render. `PreviewProps` supplies those values, so the preview greets "Ada" and points the button at a plausible verify URL. The work you did making the template props-only pays off here: the same file that renders in the preview is the one that ships in production, just with real values swapped for Ada's.

<PreviewDefaultState />

Now the reflex that makes the whole thing valuable: **keep this server running in a second terminal whenever you touch a template.** The server isn't a one-shot render — it has a <Term tooltip="A background process that watches your source files and reacts the instant one changes — here, by re-rendering the affected template.">file watcher</Term>. Save a `.tsx`, and the preview <Term tooltip="The page updates in place from the new source, without a manual reload — the watcher pushes the change to the browser.">hot-reloads</Term> on its own. So the inner loop is just: save, alt-tab, eyeball, fix. Measured in seconds, not minutes. That difference — seconds versus minutes — is the entire reason this tool exists.

## The props panel is a live what-if

The server does more than render your `PreviewProps` once. It reads each field and surfaces it as an editable control in the UI. Change `firstName` in that panel and the preview re-renders with the new value, no code edit required.

This is small to describe and large in value, because it turns `PreviewProps` from a fixed snapshot into a live *what-if* surface — and the experienced move is to use it for the cases you'd otherwise never see. Don't just admire the happy path with `firstName: 'Ada'`. Stress the layout through the panel, which is far faster than re-running your app's Server Action with different inputs:

- Swap `firstName` for something long like `Maximilian-Alexander`. Does the heading wrap cleanly onto a second line, or does it overflow the column?
- Swap `verifyUrl` for a genuinely long URL — a real signed token can run a couple hundred characters. Does the button stay inside the column, or does the link blow out the layout?
- Try a sparse or empty-ish value to see how the template behaves when the data is thinner than you assumed.

<PropsPanelStress />

There's a watch-out that belongs right here, at the moment it bites: keep `PreviewProps` *honest*. The values should look like the real production payload — a name shaped like a name, a URL shaped like your actual verify URL — not `firstName: 'x'`. A `PreviewProps` that doesn't resemble a real send teaches you nothing about the real send. If your mock URL is ten characters and production sends two hundred, the preview will look perfect and the inbox will look broken. The panel is only as truthful as the data you seed it with.

## Three lenses: desktop, mobile, and dark

The preview chrome gives you three switches, and the clean way to think about them is as three *lenses* on the same template — the identical `welcome.tsx`, viewed under different conditions. Here they are side by side; click through the tabs.

<ThreeLenses />

### Mobile is not optional

The first switch is the desktop/mobile viewport toggle. Treat the mobile view as a non-negotiable step in the loop, because the math demands it: the majority of consumer email opens happen on phones — recent email-client surveys put it well past half — and a template that wraps beautifully at a 600px desktop column can stack into something illegible at ~375px. Two columns collapse into a cramped sandwich; a button that was comfortable now runs edge to edge; padding you tuned for desktop swallows half the screen. None of that shows up in the desktop view. So the rule is simple: always toggle to mobile before you call a template done.

### Dark mode: trust it only this far

The second switch is the dark-mode toggle, which renders your template as though the reader's OS were set to dark. This is the single most important judgment call in the lesson, so slow down for it.

The toggle renders under <Term tooltip="The CSS media feature that reports whether the user's operating system is set to a light or dark color scheme.">`prefers-color-scheme: dark`</Term> — but **it is not authoritative.** Here's why, and it matters. Real mail clients don't simply honor your dark styles; many apply their own <Term tooltip="A client-side transformation that flips light backgrounds dark and dark text light, often heuristically and beyond your control.">color inversion</Term> heuristics on top of whatever you send. Apple Mail does one thing, Gmail on Android does another, and neither matches the preview's emulation. React Email's own documentation is careful with the wording: the toggle *emulates* client inversion. Emulates, not reproduces.

So calibrate your trust precisely. The toggle is excellent for catching the *obviously* broken case — a near-white logo that vanishes against an inverted near-white background, body text that turns invisible, a button that loses all contrast. Catch those here, cheaply. But treat dark mode as *confirmed* only after a real test-send to a real client. The toggle narrows the problem; it does not close it.

That's the whole job of this section — *what the toggle shows and how far to trust it*. How to actually build dark-mode styles that survive a real client is a discipline of its own, and it's the subject of the next lesson. For now: catch the obvious here, confirm the rest at the gate.

## Reading the HTML and plain-text views

Beyond the visual render, the preview UI exposes two views of the *output* — the actual artifact your code produces — and each earns a specific glance in the loop.

The **HTML view** shows the literal markup that goes on the wire. This is where last lesson's render pipeline stops being an abstraction and becomes something you can see. Open it and you can confirm the things the visual render hides:

- That your `<Tailwind>` classes actually *compiled to inline styles*. You'll see `class="text-lg"` has become `style="font-size: 18px; …"` on the element — the inlining the pipeline promised, made literal.
- That your `<Preview>` preheader text is present in the document, sitting where the inbox will pull it from.
- That the `<head>` carries what it should.

Think of the HTML view as the "is the pipeline doing what I was promised" inspection point. When a brand color mysteriously disappears in a real client, this is the first place you look — to check whether the style inlined at all.

The **plain-text view** shows the text part — the `text/plain` half of the message that the Resend SDK derives automatically from your React node, the one you got for free last lesson. The glance here is different in kind: don't just confirm it *exists*, actually *read* it. Confirm it reads as a coherent standalone message and not as "the HTML with the tags torn off." This part serves a real audience — screen readers, clients that strip HTML, the fallback Gmail shows when it clips a long message — and a garbled text part fails all of them silently.

Handy in the loop — but last lesson's rule still governs *production*: a real send needs images at a real HTTPS URL. The `emails/static/` folder is a dev convenience, not a send-time CDN.

## The test-send is the gate

Everything so far has been the inner loop — fast, local, catching layout and content. Now the other half of the lesson, and the half people skip at their peril.

The preview UI has a **Send** button. It fires a *real* email through Resend — using the same `RESEND_API_KEY` your app uses — to an address you type in. This is not another preview. It's the verification gate, and it exists precisely because the inner loop has a ceiling it cannot break through.

Keep the two jobs distinct in your head:

- The **loop** (the preview server) is where you *iterate*. Fast, local, catches the 600px wrap and the empty preheader.
- The **test-send** is where you *verify*. Slow, real, and the only way to see what an actual client does with your message — the dark-mode inversion the toggle could only emulate, the Outlook <Term tooltip="Vector Markup Language — the legacy markup older Outlook uses to render buttons and shapes. React Email generates the fallback for you, but only a real Outlook confirms it landed.">VML</Term> button path nothing local can reproduce.

The canonical practice: when a template clears the loop, test-send it to a *spread* of real accounts you actually open — a personal Gmail, an iCloud or Apple Mail address, an Outlook.com, a Proton — and open each one in its real client. That spread is what catches the things the preview's dark toggle can't: Gmail Android's blanket inversion, Outlook rendering the button through VML. There is no shortcut here, and that's the point.

State the rule crisply, because it's the spine of the whole lesson: **the preview server is the inner loop; the test-send is the gate the template passes before it's wired into a Server Action and shipped.** Iterate in the loop. Pass the gate. Then ship.

:::caution
Two ways a test-send fools you. First, sending to an address you *don't actively open in the real client* proves nothing — a dead mailbox or a webmail you never check tells you the send succeeded, not that it rendered. Test only to inboxes you genuinely read. Second, a test-send is a *real send* against your real Resend quota — treat it like one. Don't fire a hundred of them; the loop is for iterating, the test-send is the final check.
:::

## The loop, end to end

Time to see the whole thing as one chain. Each step below catches a class of failure the previous step *cannot* — that's the structure worth memorizing, because it tells you why no step is skippable. Scrub through it.

<TheLoop />

In words, the loop is:

1. **Write the template** in `emails/welcome.tsx`.
2. **Define realistic `PreviewProps`** — without them the preview has nothing to render.
3. **Run `pnpm email dev`** in a side terminal.
4. **Save, then eyeball** — desktop, toggle mobile, toggle dark.
5. **Read the plain-text view** and confirm it's coherent.
6. **Test-send** to two real inboxes — one Gmail, one Apple Mail — for cross-client confidence.
7. **Wire it into the Server Action**: `sendEmail({ react: <WelcomeEmail … /> })`, the wrapper from the last chapter.

The skill this lesson is really teaching *is* this sequence — knowing the order, and knowing why each step is where it is. So reconstruct it from memory: drag the steps into order.

## When two dev servers want the same port

One operational gotcha to close out, because you'll hit it the first time you run both servers at once.

Both `pnpm dev` (your Next.js app) and `pnpm email dev` (the preview server) default to port 3000. Start the second one and it'll either refuse to boot or land somewhere unexpected. The fix is to move one of them, and the email server is the one to move — your app's URLs are baked into more config, so it's the more expensive thing to relocate.

```bash
pnpm email dev --port 3001
```

The `-p` flag is the short form of `--port`, if you prefer it. The two servers are fully independent — the preview server has nothing to do with your running Next.js app — so they coexist on different ports without issue.

:::tip
Write the chosen port into your project's README, next to the `pnpm email dev` instructions. The clash is invisible until someone trips on it, and that someone is usually a teammate joining the project — or you, three weeks from now, having forgotten you ever moved it. One line in the README turns a confusing failure into a non-event.
:::

## Recap

The durable takeaways from this lesson:

- `pnpm email dev` boots the preview server: a left-panel template list, a live render driven by `PreviewProps`, and a file watcher that hot-reloads on save. Keep it running in a second terminal.
- The props panel turns `PreviewProps` into a live what-if surface — stress-test long names and long URLs there, and keep `PreviewProps` shaped like the real payload or it teaches you nothing.
- The mobile viewport toggle is non-negotiable; the dark-mode toggle only *approximates* client inversion and is not authoritative.
- The HTML view shows the pipeline's inlined output; the plain-text view shows the auto-derived text part — the next lesson teaches exactly what to read it for.
- The preview server is the iteration loop; the **test-send is the verification gate** — send to real inboxes you actually open before wiring the template into a Server Action.
- Move one dev server off port 3000 with `--port 3001`, and document the choice in the README.
