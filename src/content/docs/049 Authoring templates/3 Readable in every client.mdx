You can now change the HTML of a template, watch it hot-reload in the preview, and send a real test to your own inbox to confirm it survived the trip. That loop is solid. But notice what it optimized: a single render — the HTML one, the way a sighted person sees it in a modern client, the way you saw it in the preview.

That render is one of several that ship from the same send. The person running VoiceOver hears the message read aloud and never sees a pixel of your layout. The reader on a locked-down corporate account gets the HTML stripped at the gateway and is left with text only. The Gmail-on-Android user has the operating system inverting every color, so the dark logo you designed on white turns into a photo negative. The reader at 200% zoom on a small laptop sees your 600-pixel column reflow into something you never looked at. None of these people see what the preview showed you. Each of them is on the recipient list anyway.

So the question this lesson answers is not "how do I make the HTML render?" — you solved that. It is: *what posture catches everyone who doesn't see the HTML you eyeballed?* And, more concretely, the through-line that keeps this from being an abstract accessibility lecture: *where in the template does each fix live?* Every correctness check here lands on a specific attribute or element in `welcome.tsx`. The discipline is the lesson; the syntax is small.

You will leave with three things: a plain-text version of the message that reads as a coherent message on its own, an accessibility checklist applied attribute-by-attribute to the template you already built, and dark-mode plumbing that ships *with* the template instead of getting bolted on after a support ticket. None of this is new territory so much as a redirection of skills you already have. The <Term definition="Web Content Accessibility Guidelines — the W3C standard for making web content usable by people with disabilities.">WCAG</Term> reflexes from semantic HTML — one `<h1>`, descriptive link text, sufficient contrast — are exactly the muscles you built earlier. Here they get pointed at a far more constrained surface.

## The message ships as two bodies

Every legitimate transactional email goes on the wire as a single <Term definition="A MIME container holding multiple versions of the same content, marked as interchangeable alternatives. The client renders whichever one it can or prefers.">`multipart/alternative`</Term> message carrying *two* bodies: one `text/plain`, one `text/html`. They are not two emails. They are two encodings of the same message, bundled together, and the receiving client renders whichever one it can or chooses to. You met the edge of this in the first lesson — "you get the text part for free by passing the `react` prop." Now make it concrete: that free text part is the `text/plain` body of this two-body envelope.

The text body is not optional, and it is worth knowing *who* relies on it rather than memorizing a rule. Four readers and uses:

- Screen readers and accessibility-mode clients that don't render HTML at all read the text body.
- Clients and corporate security policies that strip HTML for safety leave the recipient with text only.
- Spam and content scanners read the text part to judge the message — and a missing or empty text part is itself a faint spam signal, which ties straight back to the deliverability work from the last chapter.
- When Gmail clips an HTML message that runs past its size budget (a constraint from the first lesson — we won't re-derive it), the text part is part of what indexers and scanners fall back to.

Now the part that matters most, because it determines how you'll *work*: you do not write that text body. The Resend <Term definition="Software Development Kit — the official Resend client library you call from your application code to send email.">SDK</Term> auto-derives it from the same `react` node you pass to send — the same render that produced the dual-format output in the first lesson. Your application code calls nothing extra. You pass the component; both bodies come out the other side.

This has two consequences worth stating as reflexes, because getting them wrong is the most common way teams break their own plain-text experience.

The first: **never hand-maintain a parallel text field.** It is tempting to write a nice `text` string by hand so you control exactly what the plain-text reader sees. Don't. It drifts from the HTML within a release or two — someone changes the verify copy in the JSX, nobody updates the hand-written string, and now your two bodies disagree about what the email even says. This is the same rule from the first lesson — "pass the node, don't pre-render it" — extended to the text body. The generated text is downstream of the JSX, so it can never go stale.

The second is just a name to recognize, so you understand that the auto-text is the default you're *relying on*, not magic. You *opt out* of the generated text by passing `text: ''` or your own string to the send call. You will rarely want to. Knowing the opt-out exists is enough.

It is not a `plainText` flag on `render`. You won't reach for it on the send path; it's the helper sitting behind the preview server's plain-text tab and behind your tests.

## Reading the plain-text version

"The text is generated for you" is true, but it is not the same as "the text is good." A generator can only work with the structure and wording you gave it. So the generated text becomes an active QA habit, not a thing you set and forget.

The habit is small: open the preview server's plain-text tab — the one named in the last lesson — and read the text body as a standalone message. Not as a degraded copy of the pretty version. As *the message*, the way a screen-reader user or a plain-text client receives it. And here is the mindset shift that makes this worth your time: **a bad text version is not a text-version problem. It is a bug in your HTML JSX, and you fix it there.** There is no separate text file to edit. Every defect you see in the text tab traces back to a decision in `welcome.tsx`.

Read it against four questions, each of which points at a specific spot in the JSX.

**Are the purpose and the call to action unambiguous from the text alone?** Strip away the styled button and the layout — does the reader still know what this email is and what to do? If the verify action only makes sense *because* of a big colored button, your body copy is carrying too little. The fix is wording: beef up the `<Text>` and the heading so the message stands without the chrome.

**Are links labeled rather than bare?** A naked `https://yourapp.com/verify/abc123` sitting alone in the text is technically correct and practically worse than the same URL with context around it. The text generator places the URL right after the link's text, so the link text needs to read sensibly beside its own URL. The fix lives on the `<Link>` text — "Verify your email" reads well followed by a URL; "click here" does not.

**Do decorative images stay out of the text?** A purely decorative image should contribute nothing to the text body. Give it `alt=""` — empty, explicitly set, not omitted — and it drops out of the text cleanly. Leave the `alt` missing or junk and you get stray noise: the word "image," or a filename, littering the message. The fix lives on the `<Img>`'s `alt`.

**Does the section order in the text match the HTML?** If the footer floats up to the top of the text version, that is almost never a text problem — it is a structural bug in the JSX, a `<Container>` or nesting that's out of order. The text generator walks your component tree top to bottom; if the text order is wrong, the tree order is wrong. Fix the structure, not the text.

:::caution
There is one tempting wrong move in this exact spot, and it is worth naming before you make it. When the text version looks noisy, do **not** "clean it up" by setting an *informational* image's `alt=""`. That empties the alt to silence the text — and in doing so it blinds every screen-reader user to real content. Stripping `alt` is the fix for *decorative* noise only. For an image that carries meaning, the cure for noisy text is reworded HTML around it, never a stripped `alt`.
:::

## The accessibility checklist for transactional mail

Email accessibility is, perhaps surprisingly, *less* to remember than web accessibility — but with floors that are harder, not softer. Hold both halves of that at once.

It is shorter because the substrate fights you. Email renders on tables and inline styles, with no JavaScript and almost no working ARIA. Most of the ARIA patterns you might reach for on the web simply don't survive in a mail client, so the list of things you *can* do collapses to a small set. But the things on that short list are stricter and less forgiving, because the reader cannot restyle your message. On the web a low-vision user can bump the font, override your colors, run a reader extension. In an inbox they mostly get what you sent. So the numeric floors — font size, contrast, touch target — are hard limits you meet up front, not suggestions.

The good news is that most of these are reflexes you already have from semantic HTML and your component work — semantics, one heading per page, descriptive links, contrast. Here they're just re-pointed at the constrained surface. Walk the checklist; each item is a rule, a one-line reason, and the exact place it lives in `welcome.tsx`.

**`lang` on `<Html>`.** `<Html lang="en">` was already in the template from the first lesson — here's why it was there. A screen reader picks its pronunciation rules from `lang`. Leave it off and the reader falls back to the system locale and may mispronounce the entire message in the wrong accent. One attribute, whole-message stakes.

**`<Title>` in `<Head>`.** `<Title>Verify your email</Title>` — assistive tech announces it, and some clients show it in their "open in browser" view. This is new `<Head>` content this lesson adds to the template.

**One `<h1>`, in logical order.** Exactly one `<Heading as="h1">`, and it states the *message's purpose*; subsections drop to `as="h2"`. Putting the logo in the `<h1>` is the usual anti-pattern, and it is doubly bad in email: the `<h1>` is the very first thing a screen-reader user navigates to, so it must be the point of the message, not the brand mark. "Verify your email," not "YourApp."

**Descriptive link text.** `<Link>Verify your email</Link>`, never "click here" or "this link." Screen readers let users pull up a list of every link by its text alone, with no surrounding sentence — so each link's text has to make sense standing by itself. (This is the same point the plain-text section made about labeled links; there it was for the text reader, here it's the accessibility rationale. One discipline, two payoffs.)

**Image `alt` discipline.** Decorative image → `alt=""`, empty and explicit, never omitted. Informational image → one descriptive sentence. Logo → the brand name. And a footgun worth flagging: a *missing* `alt` often falls back to the image's filename in some clients, which is both an accessibility failure and a deliverability one. (Same rule as the plain-text section — one attribute, paying off in two places again.)

**Color contrast at WCAG AA.** 4.5:1 for body text, 3:1 for headings and large text. Default `<Text>` on a near-white background is usually fine. The failure that actually recurs is the **brand color on the call-to-action button** — the text on `bg-brand` has to clear 4.5:1 against the button's fill. This is the moment the project's `brand` / `brand-foreground` pair earns a real contrast check rather than a vibe check.

**Minimum font size.** 14px for body text, 16px on mobile. iOS Mail will auto-bump text under about 13px, but don't *rely* on the bump — design to the floor. (Sizes are predictable here because of the `pixelBasedPreset` from the first lesson, which keeps your `text-*` utilities in pixels rather than rems.)

**<Term definition="The tappable area around an interactive element. For a primary action like a CTA, at least 44×44 CSS pixels per the iOS Human Interface Guidelines.">Touch target</Term> of at least 44×44px.** A finger needs room. The `<Button>` component's default padding clears 44×44 on its own; a `<Link>` you've hand-styled to *look* like a button usually does not. That's one more reason to reach for `<Button>` for any real call to action — the bulletproof-button point from the first lesson, now with the accessibility reason attached.

**Don't carry meaning in color alone.** A red "urgent" button means nothing to someone who can't perceive the red — a screen reader announces the text, not the hue. The *text* has to carry the meaning; color only reinforces it. This is a WCAG success criterion you met with semantic HTML, unchanged here.

## Designing for dark mode you can't control

Dark mode is the hardest part of this lesson because the thing you're designing for isn't one behavior — it's three, and you don't get to choose which one a given recipient's client uses. So before any code, the model.

A mail client does exactly one of three things with dark mode:

- **No transformation.** It renders your HTML as authored and applies your own dark styles if you provided any. Apple Mail on macOS in dark mode and recent Outlook behave this way.
- **Partial inversion.** It inverts light backgrounds but preserves elements that are already dark. Brand colors usually survive; a near-white background flips to near-black. Gmail on iOS and Outlook on iOS do this.
- **Full inversion.** It inverts *everything*. Your dark-text-on-white logo becomes a photo negative, and brand CTA colors hue-shift into something you never picked. Gmail on Android in some configurations, and some Outlook installs, do this.

<Figure caption="The same message, three client behaviors. Full inversion is the destructive one: it runs the whole card through an invert-and-hue-rotate, so the brand logo comes back as a washed-out negative and the CTA lands on a color you never chose.">
  <ThreeTierDarkMode />
</Figure>

Seeing the same message wrecked three different ways gets you to the posture in one move: **assume some clients will invert, design defensively so inversion doesn't break the message, and opt into your own dark styles for the clients that honor the preference.**

And the non-negotiable that bridges back to the last section: **the *light* template must already meet WCAG contrast entirely on its own.** Dark mode is never your contrast strategy. The majority of clients ignore your dark preference anyway, so a message that only reaches AA contrast *after* a dark transform is a message that fails contrast for most of its readers. Dark mode is a courtesy layer on top of a template that already passes. It is not the thing that makes it pass.

### The plumbing

With the model in place, the syntax is small. Three pieces, all living in `<Head>`, flag the template as dark-aware. Without them, Apple Mail won't apply your dark styles even when the user is in dark mode — it needs to be told the template opted in:

- `<meta name="color-scheme" content="light dark" />`
- `<meta name="supported-color-schemes" content="light dark" />`
- an inline `<style>` with `:root { color-scheme: light dark; }`

These extend the same `<Head>` the accessibility section just added `<Title>` to. Now, the two ways to actually style for dark, with a recommendation between them.

The reliable approach for a transactional template is an <Term definition="A CSS media query that activates when the reader's client reports a dark-mode preference. The client must respect the preference for it to apply.">**`@media (prefers-color-scheme: dark)`**</Term> block in an inline `<style>` in `<Head>`, targeting class or `data-` selectors to swap the brand color and the logo. This works across the clients that respect the preference and is the one to reach for.

The other is **Tailwind's `dark:` variant** through the `<Tailwind>` component. It's supported, but narrower — it works in Apple Mail and recent Outlook and is ignored elsewhere. It's fine for a simple background-and-text swap; it is *not* a substitute for the media-query block when you need to handle real inversion. (The first lesson told you not to reach for `dark:` yet — now you can, with the head plumbing in place, but know its ceiling.)

So the minimum-viable posture for the project's transactional template is concrete: the head plumbing, plus one `@media (prefers-color-scheme: dark)` block that handles the brand-color swap and the logo. A full dark-everything redesign is a marketing-template concern and explicitly out of scope here.

There's a wrinkle worth one line. Some Gmail clients strip `<style>` blocks entirely. That means your `@media` rules are an *enhancement* — they may not arrive at all. Which is exactly why the rule above is non-negotiable: the *light* inline styles have to be correct on their own, because for some readers they're the only styles that survive.

<AnnotatedCode lang="tsx" maxLines={16} code={`

  <meta name="supported-color-schemes" content="light dark" />
  <style>{\`
    :root { color-scheme: light dark; }
    @media (prefers-color-scheme: dark) {
      .cta { background-color: #1d4ed8 !important; }
      .logo-light { display: none !important; }
      .logo-dark { display: block !important; }
    }
  \`}</style>
</Head>
`}>

  </AnnotatedStep>

  <AnnotatedStep meta="{3-12}" color="blue">
    **The dark-mode layer.** The two `color-scheme` metas flag the template as dark-aware — without them Apple Mail won't apply your dark styles at all. The inline `<style>` then carries the swap: the `@media (prefers-color-scheme: dark)` block recolors the `.cta` and flips the logo. ### The logo-on-dark problem

One specific failure deserves a name even though the project won't build the fix here. A dark-text logo on a near-white background vanishes when a client inverts to near-black — it becomes dark-on-dark, or a smeared negative. The pattern that solves it is a <Term definition="An HTML element that lets the client choose among several image sources based on conditions like the color scheme, falling back to a plain img.">`<picture>`</Term> element with a `<source media="(prefers-color-scheme: dark)">` pointing at a light-on-dark version of the logo, and a fallback `<img>` with the normal dark-on-light version.

Apple Mail and most modern webmail honor it; clients that ignore the preference still get the fallback image, so the worst case degrades to "barely visible," not "missing entirely." The project ships one brand-neutral logo that reads on either background, so you won't build the swap — but recognize the pattern when you need it.

<div data-mark-color="blue">

```tsx {2} "media=\"(prefers-color-scheme: dark)\""
<picture>
  <source srcSet="https://cdn.yourapp.com/logo-on-dark.png" media="(prefers-color-scheme: dark)" />
  <Img src="https://cdn.yourapp.com/logo-on-light.png" alt="YourApp" width="120" height="32" />
</picture>
```

</div>

## Internationalization, in one line

If you expect international readers, there is exactly one thing worth doing in the template today, and it's a single token. `<Html lang="en" dir="auto">` — adding <Term definition="A value that tells the client to infer text direction (left-to-right or right-to-left) from the first strong directional character in the content.">`dir="auto"`</Term> lets the client flip the layout to right-to-left for Arabic or Hebrew content on its own, without you maintaining a separate per-locale template. Combined with the logical-property utilities from earlier — `ps-*` and `pe-*` instead of `pl-*` and `pr-*`, which the `<Tailwind>` component supports — the message mirrors cleanly when the direction flips.

That's the whole takeaway. Real localization — translated copy, ICU plurals, per-locale templates — is a substantial topic the course handles later when it reaches internationalization properly. The project's first welcome email is English-only. But `dir="auto"` is a one-token insurance policy an experienced engineer building a SaaS that expects international users would set on day one, so it earns its single line here. Everything past it belongs to the internationalization unit.

## Recap

- Every transactional email ships as two bodies in a `multipart/alternative` envelope — `text/html` and `text/plain` — and the client renders whichever it can.
- The text body is generated by the Resend SDK from the same `react` node you send. Never hand-write it; read it in the preview's plain-text tab and fix any incoherence in the HTML JSX, where it actually lives.
- The email accessibility checklist is short but its numeric floors are strict: `lang` on `<Html>`, a `<Title>`, exactly one `<h1>` stating the purpose, descriptive link text, decorative-vs-informational `alt` discipline, 4.5:1 / 3:1 contrast, 14px / 16px-mobile body, a 44×44 CTA, and no meaning carried in color alone.
- Dark mode has three client behaviors — no transform, partial inversion, full inversion — and you design for all three because you don't control which a reader gets.
- The light template must pass WCAG contrast on its own. Dark mode is a courtesy layer behind opt-in head plumbing (`color-scheme` metas plus a `@media (prefers-color-scheme: dark)` block), never your accessibility strategy.
- `dir="auto"` on `<Html>` is the one-line internationalization insurance; everything beyond it is a later topic.
