Here is a bug report a junior on your team might file, almost word for word:

> My "+3" button calls `setCount(count + 1)` three times in the click handler, but the counter only goes up by **one** per click. Is `setState` broken?

It is not broken. The handler is reading `count` exactly the way React intends, and *that* is the surprising part — the correct behavior is the one that trips everyone. To see why, hold on to the two rules from earlier in this chapter. `UI = f(state)` says your component is a function of its inputs. The purity contract said that function must be deterministic: same inputs, same output, no reaching outside. Put those together and a sharp edge appears. If render is a function and `count` is one of its inputs, then `count` *cannot change* in the middle of a render. It is a fixed input. A constant.

That single consequence is what this lesson is about. By the end you will look at that handler and know, before you click, that it adds one — and you will own the one reflex that every React codebase leans on to make it add three instead.

You already met the tool. Earlier in the chapter you saw that `const [count, setCount] = useState(0)` declares a piece of local state: reading `count` gives you this render's value, and calling `setCount(next)` schedules a re-render. We are not going to re-open the full <Term definition="React hook that returns a [value, setter] pair for a piece of local state. The full API — typing, lazy initialization, what it is not for — comes in the next chapter.">useState</Term> story here — typing it, lazy initialization, what it is and is not for, all of that is the next chapter. This lesson studies exactly one property of that value: it is a **snapshot**.

## A render is a snapshot of state

Here is the model, and it is worth installing before you see a single fix.

When React renders your component, it takes the current value of each piece of state and *bakes it into that render* as a plain constant. For the duration of that render — inside the JSX you return, and inside every function you define along the way, every event handler and every callback — `count` is not a live variable you can watch change. It is a number that was frozen the moment the render began. If `count` was `3` when React called your function, then `count` is `3` everywhere in that render, full stop. Calling `setCount(...)` does not reach back into the render you are standing in and change it. The setter has exactly one job: ask React to render *again*, and produce a *new* snapshot with the new value.

Think of each render as a photograph. React points the camera at your state, presses the shutter, and you get a still image. Everything in that photo — the props, the state, the handlers wired to your buttons — is fixed in that one frame. When state changes, React does not edit the photo. It takes a new one.

This is not a quirk bolted onto React; it falls straight out of the two rules you already trust. A pure function handed `count = 3` must always produce the same tree. It would be incoherent — a contradiction of purity itself — for `count` to be `3` on one line and `4` three lines later within the same call. So React draws a hard line: **state changes between renders, never during one.**

:::note
Setting state asks React for a *new render*. It does not change the variable in the render you are already in.
:::

That line is the whole lesson in one sentence, and most of what follows is the same sentence said from a new angle. Recall the phase strip from the first lesson of this chapter, `Trigger → Render → Reconcile → Commit`. The frozen `count` lives inside one `Render` box. The new snapshot — the next photo — is produced in the *next* `Render` box, after a fresh `Trigger`.

Let me make that visible. Scrub through the following sequence: it follows one click from the render that holds `count = 0` through to the render that holds `count = 1`.

## Three setters, one increment

Now the bug, with the model in hand. Here is the handler from the report:

<AnnotatedCode lang="tsx" code={`
function handleTripleClick() {
  setCount(count + 1);
  setCount(count + 1);
  setCount(count + 1);
}
`}>
  <AnnotatedStep meta={`"count"`} color="blue">
    `count` is a snapshot. In this render it is one fixed number — say `0` — and all three lines read that same `0`.
  </AnnotatedStep>

  <AnnotatedStep meta="{2-4}" color="blue">
    So every line is really `setCount(0 + 1)`, which is `setCount(1)`. Three requests, all asking for the same thing: set the count to `1`.
  </AnnotatedStep>

  <AnnotatedStep meta={`"setCount"`} color="blue">
    React grants the request. The next render's snapshot holds `1`, not `3`. The reads never saw each other's writes — they all looked at the same photo.
  </AnnotatedStep>
</AnnotatedCode>

Read it through the snapshot model and the mystery evaporates. In this render `count` is `0`. So `setCount(count + 1)` is `setCount(0 + 1)` is `setCount(1)` — on all three lines. You did not write "add one, three times." You wrote "set the count to one" three times. React obliges, the next render shows `1`, and the camera metaphor pays off: three reads of the same photo can only ever see the same number.

This exact snippet is a classic React interview question, precisely because it sorts the people who think of state as a live variable from the people who think of it as a snapshot. You are now in the second group.

## Why the snapshot holds: batching and the update queue

You can predict the bug now. But the next question is the one an experienced engineer asks: *why* does the snapshot survive three setters intact? Two mechanisms underneath, and we will take the simpler one first.

The first is the **update queue**. When you call `setCount(...)`, React does not stop everything and re-render on the spot. It writes an entry onto a little queue it keeps for that piece of state, and keeps running your handler. Only after the handler finishes does React work through the queue to compute the final next value, then render once. The subtle part is *what each entry says*. When you pass a **value** — `setCount(1)` — the entry means "replace the state with `1`," and it overrides whatever was queued before it. So three `setCount(0 + 1)` calls leave three "replace with 1" entries on the queue; React applies them in order, and they all land on the same `1`. That is the bug, stated mechanically. (Hold this thought: an updater entry behaves differently, and that difference is the fix in the next section.)

The second mechanism is <Term definition="Grouping multiple state updates fired close together into a single re-render, instead of re-rendering once per setter call.">batching</Term>. React groups all the setters fired during one event into a single re-render. Click the button, fire three setters, get *one* new render — not three. This is why the snapshot is stable across those three calls: React never pauses between them to re-read state and start a fresh render, so there is no opportunity for `count` to change underneath you. Since React 18, and unconditionally in React 19, this grouping also covers setters fired later — inside promises, `setTimeout`, and `async` callbacks — so you get the same batching everywhere, not only in event handlers. This is just how React 19 behaves; there is nothing to configure.

There is one boundary worth naming. React batches the setters *within* a single event, but it does not batch across separate, intentional events. Two real clicks are two renders. The grouping is "everything one event triggers," not "everything that ever happens."

:::note
Batching is a feature, not a quirk. It is what lets you call several setters in a row without paying for several renders — and it is what prevents the screen from flickering through half-finished intermediate states.
:::

## The updater form: read from the queue, not the snapshot

When the next state depends on the previous one, you do not want to read `count` from the frozen snapshot — by the time React processes the queue, the snapshot is stale by design. You want to read the value *the queue has computed so far*. That is precisely what the updater form gives you. Instead of passing a value, you pass a function: `setCount((c) => c + 1)`. React calls that function with the pending value — the result of any updaters already queued ahead of it — and uses what you return as the next entry. Queue three of them and React threads them: `0 → 1 → 2 → 3`.

<CodeVariants>
  <CodeVariant label="Value form">
    <div data-mark-color="red">

    ```tsx del={2-4}
    function handleTripleClick() {
      setCount(count + 1);
      setCount(count + 1);
      setCount(count + 1);
    }
    ```

    </div>
    **Adds one per click.** Every line reads the same snapshot of `count`, so all three resolve to the same value. Three identical "replace" entries collapse into one. This is the buggy version.
  </CodeVariant>

  <CodeVariant label="Updater form">
    <div data-mark-color="green">

    ```tsx ins={2-4}
    function handleTripleClick() {
      setCount((c) => c + 1);
      setCount((c) => c + 1);
      setCount((c) => c + 1);
    }
    ```

    </div>
    **Adds three per click.** Each updater receives the running result of the one before it — `0 → 1 → 2 → 3` — instead of re-reading the frozen `count`. The argument `c` is the pending value, not the render's snapshot.
  </CodeVariant>
</CodeVariants>

That gives us the rule, and it is conditional — resist the urge to round it up to "always use updaters."

:::tip[When the next state depends on the previous state, use the updater form.]
`setCount((c) => c + 1)`, `setItems((items) => [...items, item])`, `setIsOpen((open) => !open)` — anything that reads the prior value to compute the next one. When the next state is *independent* of the previous — `setCount(0)` to reset, `setUser(freshUser)`, `setIsOpen(true)` — the direct form is fine and reads clearer. Reach for the updater when you need it, not as a tax on every setter.
:::

Two constraints come along with the updater function, and they are the same purity rules you already met. The updater must be **pure** — it takes the previous state and returns the next one, with no side effects and no reaching outside for values. And React reserves the right to call your updater **more than once in development** to flush out impurity, so a sloppy updater that, say, mutates a shared object will misbehave in ways that only show up in the dev environment. You will see the mechanism behind that double-call in a later chapter; for now, just keep updaters pure and it never bites you.

Time to feel it. The following counter has a "+3" button wired with the value form, so it stubbornly adds one per click. Switch it to the updater form and watch the jump from +1 to +3.

## State you set now, you read on the next render

The snapshot rule has a corollary that catches people the very first day they reach for it: the setter is **queued, not immediate**, so the line *after* it still sees the old value.

<div data-mark-color="blue">

```tsx {2}
setCount(count + 1);
console.log(count); // still the OLD count — this render's snapshot
```

</div>

Walk it through the rule one more time. The `console.log` runs on the *next line down*, but it is still inside the *current* render — the render whose `count` is frozen at the old value. The new value does not exist yet; it will only appear in the next render's snapshot, after React processes the queue and calls your component again. So the log prints the old number.

There is a corollary of the corollary, and it is the part people fight hardest: `setCount` returns nothing. There is no `const next = setCount(count + 1)` — the setter is not a function that hands you the new state. If you want to *use* the next value, you read it on the next render. (Or, often, you did not need it in state at all and could compute it directly while rendering — but that is derived state, which is next chapter's job.)

Predict this one. It is the cleanest deterministic case in the lesson.

## Stale closures: setters that run after the render

Everything so far has been one render's snapshot read inside that same render. The rule gets sharper — and scarier — when a setter runs *after* the render has finished. Timers, promises, network callbacks: code that was *defined* in one render but *runs* later, against a page that has moved on.

Picture a button whose handler schedules an increment a second from now:

<CodeVariants>
  <CodeVariant label="Stale">
    <div data-mark-color="red">

    ```tsx "count + 1"
    setTimeout(() => setCount(count + 1), 1000);
    ```

    </div>
    **Captures the snapshot.** The callback closes over `count` from the render where the timeout was *created*. Click twice fast and both timeouts captured the same `count` — so two clicks can land as a single increment. The value is a second old by the time it runs.
  </CodeVariant>

  <CodeVariant label="Safe">
    <div data-mark-color="green">

    ```tsx "(c) => c + 1"
    setTimeout(() => setCount((c) => c + 1), 1000);
    ```

    </div>
    **Reads the live queued value.** The updater ignores the captured snapshot and asks React for the current pending value when it finally runs. Two clicks, two increments, no matter how stale the render that scheduled them.
  </CodeVariant>
</CodeVariants>

The reason is a <Term definition="A function that 'remembers' the variables from the scope where it was created. A handler defined during render closes over that render's props and state — even when it runs much later.">closure</Term>. The arrow function you hand to `setTimeout` was created during a particular render, and it remembers that render's `count` — the snapshot, again, just delayed. When it fires a second later, the page may be on its third render, but the callback is still holding the number from render #0. If the user clicked twice before the first timeout fired, the second timeout also closed over that stale `count`, so both compute `0 + 1` and both set `1`. Two clicks, one increment.

The updater form fixes it for the same reason it fixed the triple-click: `setCount((c) => c + 1)` does not read the captured `count` at all. It asks React for the current value at the moment it runs, so it cannot be stale. And now we can state the rule in its general form, the one the triple-click bug was only a special case of:

:::tip[Any setter that runs after render — timers, promises, async callbacks, subscriptions, effects — should use the updater form, unless you deliberately want the snapshot.]
The triple-click bug and the stale-timeout bug are the same bug. Both read a frozen `count` when they meant to read the latest. The updater form is the one fix for both, because it reads from the queue instead of the closure.
:::

## Updating objects and arrays without mutating

There is a second way a setter can appear to do nothing at all — not "it added one instead of three," but "I called it and the screen did not change." This one is the same muscle you built in the purity lesson, now exercised at the `setState` call site. Recall React's equality rule from the first lesson of this chapter: React decides whether state actually changed using <Term definition="JavaScript's identity comparison. Primitives compare by value; objects, arrays, and functions compare by reference — two different objects are never equal, even with identical contents.">`Object.is`</Term>. Primitives by value, objects and arrays by reference.

Here is the trap:

<div data-mark-color="red">

```tsx {2}
user.name = 'Alice';
setUser(user); // no re-render — same reference, Object.is bails out
```

</div>

You changed the data — `user.name` really is `'Alice'` now. But you handed `setUser` the *same object reference* you already had. React compares the next state to the current with `Object.is`, sees the identical reference, concludes nothing changed, and skips the render. The data moved; the screen did not. This is the most disorienting variant because everything looks right in the debugger — the object holds the new value — yet the UI is frozen.

The fix is the spread reflex you learned alongside purity, now applied at the call site: never mutate what is already in state; always hand the setter a **new** object or array. Walk through the three idioms you will reach for constantly.

<AnnotatedCode lang="tsx" code={`
setUser({ ...user, name: 'Alice' });
setItems([...items, newItem]);
setItems(items.filter((item) => item.id !== id));
`}>
  <AnnotatedStep meta="{1}" color="green">
    **Update an object:** spread the old fields into a fresh `{}`, then override the one you are changing. New object, new reference — `Object.is` sees a change, React re-renders.
  </AnnotatedStep>

  <AnnotatedStep meta="{2}" color="green">
    **Append to an array:** spread the old items into a new array and add the new one. The original array is untouched; the setter receives a brand-new array.
  </AnnotatedStep>

  <AnnotatedStep meta={`{3} "filter"`} color="green">
    **Remove from an array:** `filter` returns a *new* array with the unwanted item gone. `map` works the same way for replacing one. Both produce a new reference for free.
  </AnnotatedStep>
</AnnotatedCode>

Each idiom shares one shape: produce a new reference, never edit the old one. That is the entire trick, and `Object.is` does the rest.

One nuance bites people the first time they hit nested state: **spread is shallow.** Spreading the top level gives you a new outer object, but the nested objects inside are still the *same references* you started with. If the thing you are changing lives one level down, you spread at every level on the way to it:

<div data-mark-color="blue">

```tsx "...user.address"
setUser({ ...user, address: { ...user.address, city: 'Berlin' } });
```

</div>

Worth naming the flip side, because it is occasionally useful on purpose: setting state to the very same value or reference is a *deliberate* no-op React lets you lean on — calling `setUser(user)` with an unchanged reference will correctly skip the render. The bug is when that no-op happens *by accident*, because you mutated in place and handed back the old reference when you meant to change something. Same mechanism, opposite intent.

For state nested deeply enough that hand-spreading every level turns into noise, the ecosystem's answer is a small library called Immer, which lets you write what looks like a mutation and produces an immutable update under the hood. Recognize the name so it is not a mystery when you see it; the course default is direct spreads and updater functions, and you will reach for them far more often.

Now drill the instinct that ties it all together: given a setter call, does it cause a re-render or quietly bail out? Sort each chip.

## flushSync: forcing a synchronous render

Because React 19 batches everything, there are rare moments when batching is exactly what you do *not* want. You set some state and then, on the very next line, you need the DOM to already reflect it — almost always to measure a layout or hand a freshly-updated node to a non-React library. Normal batching means the DOM has not updated yet on that next line. `flushSync` is the opt-out: it runs the render and commits it to the DOM synchronously, before the next line executes.

<div data-mark-color="blue">

```tsx "react-dom"
import { flushSync } from 'react-dom';

flushSync(() => setSelectedId(id));
// the DOM is committed here — safe to read or measure the new node
node.scrollIntoView();
```

</div>

This is a recognition-level tool, not a workhorse. You will go a long time between legitimate uses.

:::caution
`flushSync` defeats batching and forces an extra synchronous render — the exact cost batching exists to avoid. Reach for it only when a measurement or a non-React API genuinely needs the DOM updated before the next statement, never as a default. And note the import: it comes from `react-dom`, not `react`.
:::

## Choosing the shape of your state

Close with a decision you will make on almost every component: when you have several related values, do you give each its own `useState`, or bundle them into one `useState` holding an object? The snapshot and immutability material you just learned is exactly what makes this a real trade-off rather than a coin flip.

Compare a form with a first and last name. Two pieces of state — `const [firstName, setFirstName] = useState('')` and `const [lastName, setLastName] = useState('')` — or one — `const [name, setName] = useState({ first: '', last: '' })`. The cut comes down to one question: do the values share a lifetime?

- **Separate state** when the values change *independently* — a panel's `isOpen` and the search text below it have nothing to do with each other. More setters, but each update is a clean primitive set, no ceremony.
- **Grouped object state** when the values change *together* — a form draft, a settings bundle, anything you tend to read and write as a unit. Fewer setters, but every update is now a spread (`{ ...form, email }`), and you carry the shallow-spread discipline from the previous section on your back.

So it is a trade between *number of setters* and *spread ceremony*, decided by whether the values live and die together. And when a grouped object's transitions start to multiply — three or more setters that always move in concert, with rules about how one field constrains another — that is the signal to reach for `useReducer`, which gives those transitions a single named home. We will not teach it here; just know its name and the threshold that calls for it.
