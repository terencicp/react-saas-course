Back in "The event model" you learned the native event system from the ground up: `addEventListener`, the capture and bubble phases, `event.target` versus `event.currentTarget`, `e.preventDefault()` and `e.stopPropagation()`, and delegation — one listener on a shared ancestor handling clicks on every descendant. That was the browser's model, written by hand.

Now JSX hands you `onClick`, `onChange`, `onSubmit` as props you spell directly on an element. They *look* like a brand-new event system, and that's exactly the trap. They aren't. `onClick` is the native model you already know, wrapped and typed. React calls the wrapper a <Term definition="React's cross-browser wrapper around the native DOM Event. Carries the same API — type, target, currentTarget, preventDefault, stopPropagation — and bubbles through React's component tree.">synthetic event</Term>, and almost everything about it behaves the way the native event did. This lesson is the translation table: what carries over (most of it), what to type (the daily friction for React newcomers), and the three behaviors that bite in production precisely because the names are identical to native but the semantics aren't.

## The synthetic event wrapper

Start with the smallest handler there is — a button that does something when you click it.

<CodeTooltips tooltips={{
  currentTarget: 'The element the handler is attached to — here, the button. Typed against that element, so .disabled is available with no cast.',
  preventDefault: "Cancels the browser's default action for this event. Same meaning as native.",
  nativeEvent: 'The raw browser Event underneath the synthetic wrapper. Reach for it only when you need a native-only API.',
}}>
```tsx title="SaveButton.tsx"
import type { MouseEvent } from 'react';

export function SaveButton() {
  const handleSave = (e: MouseEvent<HTMLButtonElement>) => {
    e.preventDefault();
    e.currentTarget.disabled = true;
    console.log(e.nativeEvent);
  };

  return <button type="button" onClick={handleSave}>Save</button>;
}
```
</CodeTooltips>

When that button is clicked, React calls `handleSave` and hands it one argument: the event. By convention you name it `e`. Log it, and you'll see something familiar — it has `type`, `target`, `currentTarget`, `preventDefault()`, `stopPropagation()`. This is the same shape you inspected on a native event in DevTools. It is not a native event, though; it's a `SyntheticEvent`, React's wrapper around the underlying browser `Event`.

Here is the reassurance to internalize before anything else, because it kills most of the anxiety newcomers feel here: **the surface is the one you already learned.** `e.preventDefault()` still cancels the browser's default action — the form submit, the link navigation, the checkbox toggle. `e.stopPropagation()` still halts the event from travelling further up the tree. The event still <Term definition="The event travels from the element it originated on up through its ancestors, firing each handler along the way. React dispatches synthetic events through your component tree the same way.">bubbles</Term> — a click on a child fires the child's `onClick`, then its parent's `onClick`, then its grandparent's, exactly as the DOM does. You are not learning a new event model. You're learning a thin layer on top of the one you own.

If you ever need to drop down to the raw browser event, it's right there: `e.nativeEvent` is the native `Event` underneath. You'll reach for it rarely — the one realistic case is calling something that lives only on the native event, like `e.nativeEvent.stopImmediatePropagation()` (`stopImmediatePropagation` has no synthetic equivalent).

So why does the wrapper exist at all? Historically, browsers disagreed about event details, and the synthetic layer normalized those quirks so you wrote one handler that worked everywhere. By 2026 those quirks have largely evaporated — but the wrapper stayed, because normalization was never its only job. The synthetic event is also how React **integrates events with its render scheduling**. The dispatch runs through React's own tree, which lets React batch the work your handler triggers: every `setState` you fire inside one handler collapses into a single re-render, the batching you saw earlier in this chapter. The wrapper isn't legacy cruft you tolerate — it's the seam where the event system and the render system meet.

<Aside type="note">
You will find old tutorials and Stack Overflow answers warning that the event object is "pooled" — reused and wiped clean the moment your handler returns — so reading `e.target` inside an async callback gives you `null` unless you first call `e.persist()`. Ignore all of it. **React 17 removed event pooling.** Synthetic events are ordinary objects now; hold a reference as long as you like, read it whenever you like, and `e.persist()` is a no-op kept only for backward compatibility. If a guide tells you otherwise, it predates React 17 and you can close the tab.
</Aside>

## Typed handlers, parameterized by element

This is the part that trips up almost everyone new to React with TypeScript, so it gets full treatment. The question is simple: when you extract a handler to a named function, what type does the event parameter take?

The rule: **React's event types are generic over the element the handler is attached to.** You import them from `react` and parameterize them with the element type.

<CodeTooltips tooltips={{
  MouseEvent: "React's mouse-event type, generic over the element it's bound to. Covers click, double-click, mouse enter/leave, etc.",
  ChangeEvent: "React's change-event type. The input/select/textarea whose value changed, parameterized by element.",
  KeyboardEvent: "React's keyboard-event type. Carries e.key, modifier flags, parameterized by element.",
  FormEvent: "React's form-event type. Used for onSubmit on a <form>.",
}}>
```tsx
import type { ChangeEvent, FormEvent, KeyboardEvent, MouseEvent } from 'react';

const handleClick = (e: MouseEvent<HTMLButtonElement>) => {};
const handleChange = (e: ChangeEvent<HTMLInputElement>) => {};
const handleKeyDown = (e: KeyboardEvent<HTMLInputElement>) => {};
const handleSubmit = (e: FormEvent<HTMLFormElement>) => {};
```
</CodeTooltips>

Those four cover the vast majority of what you'll write. The rest of the family — `FocusEvent`, `PointerEvent`, `ClipboardEvent`, `DragEvent`, and so on — lives in React's types under the same naming, and you reach for them the same way: import from `react`, parameterize with the element.

Now the shortcut that saves you from typing any of this most of the time. **An inline handler infers its parameter type from where it sits in the JSX.** Write `<button onClick={(e) => ...}>` and TypeScript already knows `e` is `MouseEvent<HTMLButtonElement>` — it reads the type off the `onClick` prop's signature, which knows it's on a `<button>`. You annotate *only* when you extract the handler to a named `const` outside the JSX, because at that point there's no prop position left for TypeScript to read the type from.

That gives you a clean reflex:

- **Inline arrow in the JSX → no annotation.** The prop position supplies the type.
- **Extracted to a named handler → annotate the parameter.** You've left the prop position, so you state the type yourself.

This is the same TypeScript stance the whole course runs on — lean on inference at the boundaries, annotate at the seams. The two shapes side by side:

<CodeVariants>
  <CodeVariant label="Inline (inferred)">
    ```tsx "(e)"
    <button type="button" name="save" onClick={(e) => console.log(e.currentTarget.name)}>
      Save
    </button>
    ```
    **No annotation needed.** TypeScript infers `e` as `MouseEvent<HTMLButtonElement>` straight from the `onClick` prop, and `e.currentTarget` is fully typed as the button. This is the default for any handler short enough to read inline.
  </CodeVariant>

  <CodeVariant label="Extracted (annotated)">
    ```tsx "MouseEvent<HTMLButtonElement>"
    import type { MouseEvent } from 'react';

    const handleSave = (e: MouseEvent<HTMLButtonElement>) => {
      console.log(e.currentTarget.name);
    };

    <button type="button" name="save" onClick={handleSave}>Save</button>
    ```
    **You annotate the parameter.** Pull the handler out to a named `const` and there's no prop position to infer from. Note the `import type` — event types are type-only imports, and `verbatimModuleSyntax` requires the `type` keyword.
  </CodeVariant>
</CodeVariants>

One watch-out while you're importing these: pulling `MouseEvent` from `react` shadows the global DOM `MouseEvent` type in that file. That's intended — inside a React component you want React's typed version, not the raw DOM one. Don't "fix" it.

## Reach for currentTarget, not target

Here's the first behavior that bites — and it bites through TypeScript, not at runtime. Refreshing the distinction you learned natively, in one line: <Term definition="The element the event handler is attached to. Stays constant for the whole dispatch of one handler.">`currentTarget`</Term> is the element the handler is attached to; <Term definition="The deepest element the event actually originated on — the exact node you clicked. May be a descendant of currentTarget.">`target`</Term> is the deepest element the event originated on. Same as native.

The React-specific bite is in the *types*. Because the handler's prop position pins down which element it's on, **`e.currentTarget` is typed as that element** — so on an input handler, `e.currentTarget.value` is fully typed, no cast, no fuss. But **`e.target` is typed as the generic `EventTarget`**, which has no `.value`, no `.checked`, no `.name`. Reaching for `e.target.value` is a TypeScript error before you've even run the code.

<CodeVariants>
  <CodeVariant label="e.target.value">
    <div data-mark-color="red">

    ```tsx "e.target.value"
    const handleChange = (e: ChangeEvent<HTMLInputElement>) => {
      setEmail(e.target.value);
    };
    ```

    </div>
    **TypeScript error** — `Property 'value' does not exist on type 'EventTarget'`. `target` is typed as the bare `EventTarget` because, in the general case, the event could have originated on any descendant element, so React can't promise it's the input.
  </CodeVariant>

  <CodeVariant label="e.currentTarget.value">
    <div data-mark-color="green">

    ```tsx "e.currentTarget.value"
    const handleChange = (e: ChangeEvent<HTMLInputElement>) => {
      setEmail(e.currentTarget.value);
    };
    ```

    </div>
    **Clean and typed.** `currentTarget` is pinned to the element the handler sits on — the `HTMLInputElement` — so `.value` is right there. Make this your default for reading `.value`, `.checked`, `.name`, and `dataset` off the event.
  </CodeVariant>
</CodeVariants>

So the reflex is one word: **reach for `currentTarget`.** For reading values off the element a handler is bound to — the everyday case — it's the typed, correct choice, and it sidesteps the casting dance entirely.

There's a nuance worth naming so it doesn't surprise you later. `currentTarget` and `target` point at the *same element* when the handler sits directly on the element you interacted with — which is the common case, a handler right on the input. They only diverge under delegation: the handler is on a parent, and you clicked a child. That's the one situation where you'd deliberately read `target` (and narrow it, since you know which element you expect). But for reading a control's own value, `currentTarget` is the default, and a good default it is.

## Delegation happens at the React root

This is the most counterintuitive thing in the lesson, and the source of a `stopPropagation` surprise that stale tutorials get wrong. You already understand delegation from "The event model" — one listener on an ancestor handling many descendants. The new fact is *where* React puts that listener.

Here's the mental image you probably hold, and it's wrong: you imagine each `onClick` prop becoming a native `addEventListener` call on that specific button. It doesn't. **Since React 17, React attaches one listener per event type at the root container** — the single DOM node React mounts your whole tree into. Your `onClick` props aren't native listeners on each button; they're entries in React's internal tree. When you click a button, the click bubbles up the real DOM to that one root listener, and React then looks at where the click landed and **dispatches** a synthetic event back down through your component tree, firing the matching `onClick` handlers along the way.

<Aside type="note">
The root container is the DOM node React renders into, set up under the hood by `createRoot`. In a Next.js app the framework wires this up for you — you'll never call `createRoot` yourself in this course, so treat it as recognition, not a setup step.
</Aside>

Why should you care where the listener lives? Two consequences, and the second is the one that bites.

First, the benign one: **multiple independent React trees on one page don't fight.** Pre-React-17, React attached its single listener to `document`, so two separately-mounted React apps on the same page stomped on each other's `document`-level handling. Now each root owns its own listener at its own container, and they coexist cleanly. You'll rarely run two React roots in a Next.js SaaS, but this is the design reason the listener moved.

Second, the one to remember: **`e.stopPropagation()` in a React handler now reaches farther than it used to.** Because React's listener lives at the **root container** rather than at `document`, a React handler that calls `e.stopPropagation()` stops the native event from bubbling past the root — which means a native listener you attached to `document` (or to any ancestor *above* the root) **never fires**. The event is contained before it ever gets there.

This is the exact opposite of the pre-React-17 behavior, and it's why old articles mislead. Back then React listened at `document` itself, so by the time your React handler ran, the native `document` listener had *already* received the event, and `stopPropagation` inside React couldn't un-fire it. People wrote that down, and the note outlived the behavior.

The durable rule for you: a React child's `stopPropagation()` contains the event **within the root container**. So a global native `document` listener — the window/document keyboard-shortcut handlers you learned to attach natively, or a third-party DOM library listening above the root — can be silently suppressed by a React component deep in your tree. That's the surprise to watch for. (Native listeners attached *below* the root still interleave with React handlers by ordinary DOM bubbling order — nothing strange there.)

## onChange fires on every keystroke

The third divergence is short and sharp, and it confuses anyone who first read about the native `change` event on MDN. There, `change` fires only when an input *loses focus* after its value changed — you type, you click away, *then* it fires.

React's `onChange` does not work that way. **`onChange` on a text input fires on every keystroke.** Type five characters, it fires five times. Under the hood React wires `onChange` to the native `input` event, not the native `change` event — React treats `onInput` and `onChange` as the same thing. The name is borrowed from the native world; the behavior is the native `input` event's.

This isn't a wart — it's exactly what makes live-updating UI work. The canonical pattern is to push each keystroke straight into state:

```tsx
<input value={query} onChange={(e) => setQuery(e.currentTarget.value)} />
```

Every keystroke fires `onChange`, which calls `setQuery`, which re-renders with the new value — and the UI tracks the input in real time. Feel it for yourself:

## Forms and preventDefault, briefly

You'll meet `onSubmit` constantly, so let's defuse it — but only the event mechanic, because forms proper are a later topic.

A `<form>` has a default action: when submitted, the browser navigates — a full-page reload to the form's `action` URL. To handle the submit in JavaScript instead, you cancel that default with the same call you learned natively:

```tsx
<form
  onSubmit={(e) => {
    e.preventDefault();
    const data = new FormData(e.currentTarget);
    saveDraft(Object.fromEntries(data));
  }}
>
  {/* fields */}
</form>
```

Without <Term definition="Cancels the browser's default action for the event — here, the full-page form submission. Same meaning as the native call you already learned.">`preventDefault`</Term> the page reloads and your handler's work is thrown away. With it, the submit is yours. When you extract the handler, its type is `FormEvent<HTMLFormElement>`.

Now the part that keeps you from over-investing in this pattern: in 2026 you'll write `preventDefault` on a form far less often than older code does. The modern React form is built on the `<form action={fn}>` prop — you hand the form a function, and that function owns the entire submit lifecycle, `preventDefault` included, without you writing it. The course gives forms — the `action` prop, reading submitted data, validation, pending and error states — their own dedicated unit later. For now, recognize the `onSubmit` + `preventDefault` shape when you see it, and know that the `action={fn}` pattern is what retires most of it.

## Keyboard events and the e.key reflex

Keyboard handling is a small, high-value family the rest of the course leans on, so learn the one reach that matters.

**Read `e.key`.** It's a string naming the key: `'Enter'`, `'Escape'`, `'ArrowUp'`, `' '` for the spacebar, `'a'` for the A key. The older `e.keyCode`, `e.charCode`, and `e.which` are deprecated — don't reach for them. (`e.code` also exists, naming the *physical* key regardless of keyboard layout — useful for game controls — but `e.key` is the daily one.)

The canonical use is dismissing things on Escape:

```tsx
<input
  onKeyDown={(e) => {
    if (e.key === 'Escape') {
      clearSearch();
    }
  }}
/>
```

Escape-to-close on modals and menus is the everyday reach. That said, *reading the key* is all this lesson teaches — the focus management and accessibility of dismissible overlays (where focus goes when the modal closes, trapping Tab inside it) is its own topic the course handles later with shadcn's overlay components. Read the key here; manage the focus there.

For chord shortcuts like ⌘K, the modifier keys are booleans on the event: `e.metaKey` (⌘ on macOS, Windows key elsewhere), `e.ctrlKey`, `e.shiftKey`, `e.altKey`. Check `e.key === 'k' && e.metaKey` and you've got the command palette trigger.

One naming choice: reach for `onKeyDown`, not `onKeyPress`. `onKeyDown` fires before the value changes and covers *every* key — modifiers, arrows, Escape, the lot. `onKeyPress` is deprecated (it only ever fired for printable characters and was dropped from the DOM spec).

## Pointer events, the unified input primitive

When code has to handle both mouse and touch — a drag handle, a custom slider, any gesture — there's one correct reach in 2026, and it's not the old mouse-plus-touch pairing.

**Use pointer events:** `onPointerDown`, `onPointerMove`, `onPointerUp` cover mouse, touch, and pen through a single API. When you need to branch on the input type, `e.pointerType` tells you which: `'mouse'`, `'touch'`, or `'pen'`. The legacy approach — wiring up `onMouseDown` *and* `onTouchStart` and reconciling the two — double-handles every interaction and is exactly what pointer events were created to replace.

```tsx
<div
  onPointerDown={(e) => {
    e.currentTarget.setPointerCapture(e.pointerId);
    const slop = e.pointerType === 'touch' ? 12 : 4;
    startDrag({ x: e.clientX, y: e.clientY, slop });
  }}
>
  {/* draggable handle */}
</div>
```

The one genuinely useful reflex worth knowing concretely is that `setPointerCapture` call. <Term definition="Routes all further pointer events to one element until release, even after the pointer leaves the element's bounds. The reason drag handlers use pointer events.">Pointer capture</Term> tells the browser to keep sending pointer events to this element *even after the pointer moves outside it* — which is precisely what a drag needs, since your cursor races past the handle's edges the moment you start dragging. Without it, the drag drops the instant you leave the element. This is *the* reason serious drag code uses pointer events.

Building a full drag-and-drop system — reordering a list, dragging cards between columns — is what libraries like `@dnd-kit` exist for, and that's out of scope here. The takeaway is just the platform default: for anything cross-input, reach for `onPointer*`, not `onMouse*` plus `onTouch*`.

## A few closing reaches

Three small things to recognize, none of which earns its own section, but each of which you'll hit.

**The capture-phase variant.** Every handler so far has been bubble-phase — `onClick` fires as the event travels *up* the tree. The capture-phase version, firing on the way *down* before children see the event, appends `Capture` to the name: `onClickCapture`, `onKeyDownCapture`, `onPointerDownCapture`. Same capture-versus-bubble semantics you learned natively. The bubble form is your daily reach; capture is the niche case where a parent needs to intercept an event before its children get it.

**`value` is always a string.** `e.currentTarget.value` hands you a string — *always* — even on `<input type="number">`, where the DOM stores the raw text you typed, and `<input type="date">`, where it's an ISO date string. The conversion to an actual number or date is your handler's job; the event never hands you a typed `number`. So don't blindly `Number(e.currentTarget.value)` and trust it, and don't expect a `Date`. (When you reach forms later, schema validation with Zod's `z.coerce.number()` is where this coercion gets handled cleanly at the boundary — but the trap is worth knowing now.)

**`onScroll` doesn't bubble in React.** Matching the DOM, scroll events don't propagate up the tree. Attach `onScroll` to the element that actually scrolls, not to a parent expecting to catch it.

That's the whole event surface. To consolidate the lesson's one big idea — what carried over from native versus what's genuinely React's — sort these:
