Every year, WebAIM scans the home pages of the top million websites and counts the accessibility errors a machine can detect. The 2026 run found something that should stop you: pages that used ARIA averaged roughly **59** detected errors, while pages that used none averaged about **42**. More ARIA, more errors. Not a small gap, and not noise — the same direction every year the report has run.

Read that the wrong way and you conclude ARIA is harmful. It isn't. The pages reaching for ARIA are the ones attempting harder interfaces, and ARIA is the power tool they grab. But a power tool cuts both ways, and this is the first thing in this whole chapter that can make an interface measurably *worse* while looking, in the diff, like you made it *more* accessible. Everything else you have learned — semantic HTML, the shadcn primitives, the four commitments — only ever helps. ARIA is the one surface where a well-meaning junior ships a regression and feels good about it.

So this lesson is mostly about restraint. By the end you will know the **four** ARIA surfaces a SaaS engineer actually reaches for — roles, labels, descriptions, and live regions — and, more to the point, you will know when *not* to reach for any of them. You will label an icon-only button so a screen reader can read it, announce a "Saved" toast and a search-result count to someone who can't see them, and wire help text to an input. Those four things show up in every screen you build from the next chapter on.

Frame it against the previous two lessons. Semantic HTML is the first move; the shadcn primitives ship correct ARIA for the widgets they cover; together that is something like ninety percent of accessibility, handled before you type an `aria-` anything. This lesson is the other ten percent — the custom widgets and the live state the primitive doesn't know about. Most of your ARIA will be a handful of attributes. The skill is knowing which handful, and having the discipline to stop there.

## The one rule that prevents most ARIA bugs

There is a single rule that, held honestly, prevents the large majority of ARIA bugs before they exist. It comes straight from the <Term definition={"The WAI-ARIA Authoring Practices Guide — the W3C's official guidance on how to use ARIA correctly, including patterns for common widgets.\nWhen this course says 'the spec says', this is usually the source."}>WAI-ARIA Authoring Practices Guide</Term> (the APG), the W3C document that defines how ARIA is meant to be used:

> If you can use a native HTML element with the semantics and behavior you need, use it.

And the maxim the same guide leads with, the one this lesson is named after: **no ARIA is better than bad ARIA.** A control with no ARIA at all is a known quantity. A control with *wrong* ARIA actively lies to the user, and a lie is worse than silence.

To understand why ARIA goes wrong so reliably, you need one mental model, and it is the load-bearing sentence of this entire lesson:

**ARIA only changes what assistive technology announces. It never changes behavior.**

Not keyboard handling. Not focus. Not click. ARIA edits the *description* of an element that <Term definition={"Assistive technology — the software and hardware people use to operate a computer non-visually or non-manually.\nScreen readers (VoiceOver, NVDA, JAWS), switch devices, screen magnifiers.\nUsually abbreviated AT."}>assistive technology</Term> (AT — screen readers, switch devices, magnifiers) reads aloud. It does not touch what the element *does*. Hold that distinction and most ARIA mistakes become visible the moment you make them.

Here is the canonical one, the mistake the WebAIM numbers are partly made of. You need a Save control, and for whatever reason you build it on a `<div>`:

<CodeVariants>
  <CodeVariant label="div + role (broken)">
    <div data-mark-color="red">

    ```tsx "role=\"button\""
    <div role="button" onClick={save}>Save</div>
    ```

    </div>
    This renders, it clicks with a mouse, and a screen reader now announces "Save, button." That announcement is a promise: a screen-reader user hears "button" and expects Enter and Space to activate it, the way every button they've ever used does. But a `<div>` isn't in the tab order and has no key handling — it's dead to the keyboard. The `role` wrote a promise the markup can't keep. And notice the tempting wrong fix: stack on `tabIndex={0}`, then an `onKeyDown` for Enter, then another for Space, then re-create the focus behavior by hand. You'd be rebuilding `<button>`, badly, attribute by attribute.
  </CodeVariant>

  <CodeVariant label="button (correct)">
    <div data-mark-color="green">

    ```tsx "button"
    <button onClick={save}>Save</button>
    ```

    </div>
    The native element brings the role *and* the keyboard *and* the focus together, as one package, for free. The screen reader announces "Save, button," and Enter and Space genuinely work, because the browser wired them. This is the whole point in one line: ARIA gives you the announcement half; the native element gives you the announcement *and* the behavior, already in sync.
  </CodeVariant>
</CodeVariants>

That contrast is the clearest possible illustration of "ARIA is not behavior," and it is also exactly why "semantic HTML first" from the previous lesson is not a style preference. Native elements are the only thing that keeps what AT *says* about a control and what the control *does* in lockstep. The moment you split them — role on one element, behavior nowhere — you have built the lie.

So the scope of this lesson is narrow on purpose. Native HTML and the shadcn primitives cover the floor. ARIA is for four jobs, and only four:

- **Roles** that describe custom widgets — almost all of which the primitive already writes, so this is mostly a list of things you *don't* do.
- **Labels** that give a name to a control with no visible text.
- **Descriptions** that attach help or error text to a control.
- **Live regions** that announce changes a sighted user would simply see.

The rest of the lesson is those four, in order. Each one arrives the same way: here is the gap native HTML leaves open, and here is the one ARIA tool that fills it.

## Roles: the few you write, the many the primitive owns

A `role` overrides an element's *semantics* for assistive technology — it tells AT "treat this element as a such-and-such." Which immediately raises a question: when do you need to override semantics that HTML already gives you?

Mostly, you don't. Every HTML5 element ships an <Term definition={"The role an element already has, before you add any role attribute.\nA button element is implicitly a button, nav is a navigation landmark, main is the main landmark.\nA role attribute only matters when you need something different from the implicit one."}>implicit role</Term> for free. A `<button>` is already a button to AT. A `<nav>` is already a navigation region; `<main>` is already the main region; `<h1>` is already a heading. So the explicit `role` you were about to type — `<button role="button">`, `<nav role="navigation">` — is redundant with the element you already chose. Redundant ARIA is the cheap, harmless end of "bad ARIA," but it is still noise, and it is still a sign you reached for the attribute before the element.

What about the genuinely complex roles — the ones for tabs, menus, comboboxes, trees, grids? Here the previous lesson's dividend pays off directly. **You do not write these.** A shadcn `DropdownMenu` is already wired as `menu` and `menuitem`; `Tabs` is `tablist`, `tab`, and `tabpanel`; `Dialog` is `dialog`. The Radix primitive underneath owns the entire role taxonomy for the widget, and — true to the "the source is the docs" idea from the first lesson — you can open the file in `components/ui/` and see the roles sitting right there. Your job is to recognize that this machinery exists and is handled, not to reproduce it.

That leaves a short list of roles a SaaS engineer actually authors. It is short enough to give you in full:

- `role="status"` and `role="alert"` — for announcing live changes. These are the live-region surface, and they get their own section later; they're on the list so the list is complete.
- `role="presentation"` (its synonym `role="none"` does the same thing) — *strips* an element's implicit semantics. The trigger is specific: you have inherited markup that reuses a semantic element purely for layout — a `<table>` laying out a form, a `<ul>` used as a grid — and AT is announcing "table" or "list, 3 items" over what is really just visual scaffolding. The role tells AT to ignore the semantics and treat the element as a plain container.
- Landmark roles — `banner`, `navigation`, `main`, `contentinfo` — the explicit equivalents of `<header>`, `<nav>`, `<main>`, `<footer>`. You met <Term definition={"A region of the page that AT users can jump straight to — banner, navigation, main, contentinfo, and a few more.\nThe HTML5 elements header, nav, main, footer provide them implicitly; the role attribute is the fallback when you can't use the element."}>landmark</Term> landmarks back in the lesson on semantic structure. The rule with these is the same rule again: **prefer the element.** Use the `role` only in the rare case you genuinely cannot change the tag.

Notice the shape repeating across every entry: the senior move is almost always to fix the *element*, and the `role` is the escape hatch for when you can't. `role="presentation"` is the cleanest illustration — the right fix is usually "don't use a `<table>` for layout in the first place." But when you have inherited code you can't restructure today, the escape hatch is there:

<div data-mark-color="blue">

```tsx "role=\"presentation\""
{/* legacy layout table — presentation until it's rebuilt as a grid */}
<table role="presentation">
  {/* … inherited rows … */}
</table>
```

</div>

That is the entire role surface you own. Status and alert in their section; presentation as the inheritance escape hatch; landmarks only when the element is out of reach. Everything else, the primitive already wrote.

## Naming controls a screen reader can read

Every interactive control needs an <Term definition={"The string assistive technology announces for a control.\nFor a button with text, it's the text. For an icon-only button, you supply it with aria-label or visually-hidden text.\nEvery interactive control needs a non-empty one."}>accessible name</Term> — the string AT announces when the user lands on it. A sighted user reads a button's name off its visible text. AT needs that same name, and the question is only *where the name comes from*. There are four sources, and the rule that keeps them simple is the one the spec exists to enforce: **exactly one source supplies the name; you never stack them.** (When you do stack them they have a fixed override order — `aria-labelledby` beats `aria-label` beats visible text — but the moment you're relying on that order, you've already made the mistake. Supply one.)

Taken in the order you actually reach for them, from the one you want first to the one you want last:

1. **Visible text content.** The default, and the best one. `<button>Save</button>`, `<a href="/settings">Settings</a>` — the name *is* the visible text, which means the name and what the user sees can never drift apart. Whenever the control has visible text, you are already done. Add nothing.
2. **`aria-label`.** Supplies a name when there is *no* visible text to use. This is the icon-only button, coming up next — the case this whole section exists for.
3. **`aria-labelledby={id}`.** Points at *another* element's text and borrows it as the name. Reach for this when the label already exists elsewhere on screen — a section named by its own heading, say. It pairs naturally with `useId` for the id, which we'll use in the next section.
4. **`title`.** Mentioned only so you know to *avoid* it for naming. The `title` attribute shows a tooltip on mouse hover, but support for it as an accessible name is inconsistent across AT, and it never appears for keyboard or touch users at all. Don't rely on it.

One thing this chain is *not* about: form inputs. A text input gets its name from a `<label htmlFor>`, which you met in the lesson on the label contract — that is the form-specific labeling mechanism, and it is the right tool for inputs. Don't reach for `aria-label` on an `<input>` when a visible `<label>` is what the field actually wants.

### The icon-only button

Now the case that recurs in every modal, drawer, toast, and table row you will ever build — the single highest-frequency ARIA task in a shadcn codebase. You drop a close button into a dialog:

```tsx
<Button size="icon"><X /></Button>
```

To a sighted user this is obviously a close button — the X says so. To a screen reader it announces, in full: "button." That's it. No name. The user has landed on a control and has no idea what it does. This is the canonical icon-only-button miss, and it is everywhere.

Before the fix, one thing about the icon that corrects a common instinct. You might think the `<X />` needs an `aria-hidden` so the screen reader doesn't try to read the SVG. It doesn't — Lucide (the icon set shadcn ships) marks every icon `aria-hidden="true"` *by default*. The glyph is already invisible to AT. So the problem isn't that the icon is being announced; it's that *nothing* is. The name has to go on the **button**, where the interaction lives — never on the icon. There are two equivalent ways to put it there:

<AnnotatedCode lang="tsx" code={`
<Button size="icon"><X /></Button>

<Button size="icon" aria-label="Close"><X /></Button>

<Button size="icon">
  <X />
  <span className="sr-only">Close</span>
</Button>
`}>
  <AnnotatedStep meta="{1}" color="red">
    The starting point. The Lucide `<X />` is already `aria-hidden`, so AT reads only the button's role and finds no name. The screen-reader user hears "button" and is stranded.
  </AnnotatedStep>

  <AnnotatedStep meta={`"aria-label=\\"Close\\""`} color="green">
    The most direct fix. `aria-label` supplies the name straight onto the interactive element. The control now announces "Close, button." This is the reach for the common case.
  </AnnotatedStep>

  <AnnotatedStep meta="{7}" color="green">
    The equivalent alternative — real text content, hidden from the screen visually but present in the accessibility tree, so it becomes the button's name the normal way (source 1 in the precedence chain). Identical result; pick whichever reads cleaner at the call site. You'd reach for this when you want the label to live as real DOM text rather than an attribute.
  </AnnotatedStep>
</AnnotatedCode>

That `sr-only` is a Tailwind utility — <Term definition={"A Tailwind utility that hides an element visually while keeping it in the accessibility tree, so screen readers still announce it.\nUsed for text that gives AT context a sighted user gets from layout or an icon.\nFull treatment in the next section."}>`sr-only`</Term> visually hidden, still in the accessibility tree. It's the workhorse of accessible labeling and you'll see it again in a moment; for now, read it as "text only AT can perceive."

Now you put it into practice. The exercise below hands you a small toolbar where every button is icon-only and nameless. Give each one an accessible name — and watch the trap the precedence chain warns about: the one button that *does* have visible text needs nothing, so don't stack a redundant `aria-label` onto it.

## Help text and descriptions with `aria-describedby`

A name tells the user *what a control is*. Sometimes you also need to tell them something *about* it — a format hint under a field, a sentence of help text, an error message. That second category is a **description**, and AT reads it *after* the name. The mechanism is `aria-describedby`, and it works by reference: you point the control at the id (or several ids) of the elements holding the description text.

```tsx
const hintId = useId();

<input aria-describedby={hintId} />
<p id={hintId}>We'll never share your email.</p>
```

The attribute takes a space-separated list of ids, so a control can have more than one description — a hint and an error, for instance — and AT reads each referenced element in turn. Here it points at one: the input announces its name, then "We'll never share your email."

The id itself is the part worth pausing on, because hand-writing it is a real bug waiting to happen. You reached for `useId` back in the lesson on stable ids — this is one of its core uses. A description lives in a separate element from the control that references it, so the two have to agree on a string. If you hard-code `id="email-hint"` and that component renders twice on the same page — two of the same form, a list of them — you now have two elements sharing one id, and the reference is ambiguous. `useId` hands you an id that is unique per instance *and* stable across the server render and the client hydration, which is exactly the guarantee a cross-element reference needs.

<CodeTooltips tooltips={{
  useId: 'Returns a stable, unique string — the same value on the server render and the client, and different for every instance of the component.\nExactly what a cross-element id reference needs.',
}}>
```tsx
const hintId = useId();

<input type="email" aria-describedby={hintId} />
<p id={hintId}>We'll never share your email.</p>
```
</CodeTooltips>

One discipline to hold from the start: a reference and its target live and die together. If the element a `describedby` points at gets emptied or removed but the attribute stays, AT is left reading an empty description, or pointing at a node that no longer exists. Whenever you remove the hint, remove the reference in the same change.

This is deliberately the shallow version. The place descriptions earn their keep is form validation — wiring an error message to a field, pairing `aria-describedby` with `aria-invalid`, and announcing the error at the right moment. That is a whole topic, and it belongs to the forms chapter (Chapter 44), which builds it properly. Here you only need the wiring: a description is a referenced element, read after the name, with a `useId`-stable id holding the two ends together.

## Hiding the right things from the right audiences

Three attributes hide content, they are constantly confused for one another, and the confusion comes from treating "hidden" as one thing. It isn't. The question that untangles all three is a single one: **who needs to perceive this content?** A piece of content can serve sighted users, AT users, both, or neither — and there is a different tool for each combination.

- **`sr-only`** (the Tailwind utility) → in the accessibility tree, off the screen. Visible to AT, hidden from sighted users. For text that gives AT the context a sighted user already gets from layout or an icon: the skip-link text, the icon-button label from two sections ago, a "Showing results for…" line.
- **`aria-hidden="true"`** → on the screen, out of the <Term definition={"The parallel tree assistive technology actually consumes, derived from the DOM.\nEach node carries a role, a name, a state. aria-hidden removes a node and its whole subtree from this tree.\nWhat AT announces is this tree, not the DOM directly."}>accessibility tree</Term>. Visible to sighted users, hidden from AT. For purely *decorative* visuals that would be noise if announced: a decorative image, an icon sitting *next to* a visible text label.
- **`hidden` / `display: none`** → off the screen *and* out of the accessibility tree, and out of layout entirely. Hidden from everyone, because it is content that is not relevant to anyone right now — the body of a collapsed accordion panel, a tab's contents while another tab is active.

The everyday version of this is icons, and it is where the three tools click into place. Take `<Mail /> Send` — an icon beside the word "Send." The word carries the meaning; the icon is decoration. If AT announced both, the user would hear "mail, Send," which is just noise — so the icon should be `aria-hidden`. And with Lucide that is *already true* by default, which is the right default: a decorative icon next to a label needs nothing from you.

Now the apparent contradiction. A *standalone* `<Mail />` button — an icon with no text — is clearly meaningful; it does something. So shouldn't you un-hide its icon? No. The resolution is the rule you already learned: you don't expose the icon, you **name the button** (`aria-label="Compose"`). The name lives on the interactive element; the icon stays decorative in both cases. The icon being decorative and the control being meaningful are simply two different layers, and once you see that, the contradiction dissolves.

:::caution
The trap that strands users: never put `aria-hidden="true"` on a focusable element — a button, a link, an input — or on anything that *contains* one. `aria-hidden` removes the element from the accessibility tree, but it does **not** remove it from the tab order. So keyboard focus can still land there, and when it does, AT announces nothing at all. The user is stuck on a control they can't hear. If you want something gone, take it out of the DOM or make it genuinely inert — don't `aria-hidden` over a control the keyboard can still reach. (The next lesson, on focus, comes back to exactly this interaction.)
:::

  <Leaf id="leaf-ariahidden" verdict="aria-hidden (already the Lucide default)">
    A decorative visual beside a visible label is noise to a screen reader — the label already carries the meaning.
    `aria-hidden="true"` keeps it on the screen and out of the accessibility tree, and every Lucide icon ships this way by default, so usually you write *nothing*.
    Pick `sr-only` or leave it in the tree instead and AT reads "mail, Send" — the same word twice, clutter on every pass.
  </Leaf>

  <Leaf id="leaf-name" verdict="Name the button — don't un-hide the icon">
    Here's the resolution to the apparent paradox: the control *is* meaningful, but you still don't expose the icon.
    The name lives on the interactive element — `aria-label="Compose"` on the `<Button>` (or an `.sr-only` text child) — and the glyph stays decorative, exactly as in the decorative case.
    Un-hide the icon instead and AT may read the raw SVG title or nothing useful; the reliable name belongs on the button.
  </Leaf>

  <Leaf id="leaf-sronly" verdict="sr-only — in the tree, off the screen">
    The text alternative for an icon-only button, a skip-link's words, a "Showing 12 of 134 results" line a sighted user infers from the list — AT needs the text that layout or an icon conveys visually.
    `sr-only` keeps it in the accessibility tree while hiding it from sight.
    Reach for `aria-hidden` here and the control announces nothing, or the AT user never learns the result count — the information vanishes for exactly the audience that needed it.
  </Leaf>

  <Leaf id="leaf-hidden" verdict="hidden / display:none — gone for everyone">
    The body of a collapsed accordion panel, the contents of a closed modal — content that is irrelevant to *every* audience until the UI enters that state.
    `hidden` (or `display:none`) removes it from the screen, the accessibility tree, *and* layout.
    `aria-hidden` alone is the trap: it leaves the subtree in layout and in the tab order, so keyboard focus can still land on a control inside the "hidden" panel.
  </Leaf>

  <Leaf id="leaf-trap" verdict="Don't aria-hidden a focusable control">
    This is the one that strands users, so the walk ends on a *don't*.
    `aria-hidden` removes a node from the accessibility tree but **not** from the tab order — so keyboard focus still reaches the control, and when it lands there AT announces nothing at all.
    If the control should be gone, take it out of the DOM or make it genuinely inert; never `aria-hidden` over something the keyboard can still reach.
  </Leaf>
</StateMachineWalker>

## Announcing what only sighted users can see

The fourth surface is the one with real machinery behind it, and it is where most of this lesson's value sits. Here is the problem it solves. A sighted user *sees* a toast slide in, a "Saved" badge flip on, a result count change after they filter a table. An AT user perceives none of it — the page changed somewhere off to the side, silently, and nothing told them. A <Term definition={"A region of the DOM that assistive technology watches for changes.\nWhen its contents change, AT announces the new text — without the user moving focus there.\nThe mechanism for making async changes (toasts, save indicators, counts) perceivable."}>live region</Term> is the mechanism that fixes this: it is a region of the DOM that AT *watches*, so that when its contents change, AT reads the new text aloud — without the user having moved focus there at all.

A live region is defined by a small set of attributes. The walkthrough below builds a minimal one, one attribute at a time.

<AnnotatedCode lang="tsx" code={`
<div aria-live="polite" aria-atomic="true">
  {message}
</div>

<div role="status">{message}</div>
`}>
  <AnnotatedStep meta={`"aria-live=\\"polite\\""`} color="blue">
    `aria-live` says *when* AT announces a change. `"polite"` waits until the user is idle, then reads it — the default reflex for almost everything. The alternative, `"assertive"`, interrupts whatever AT is saying *right now*. Reserve assertive for genuine, must-not-miss messages; reach for polite by default.
  </AnnotatedStep>

  <AnnotatedStep meta={`"aria-atomic=\\"true\\""`} color="blue">
    `aria-atomic` says *how much* to announce. `true` reads the whole region on any change; the default (`false`) reads only the node that changed. Use `true` when the message only makes sense whole — "Saved 3 of 5" means nothing if AT reads just the "5".
  </AnnotatedStep>

  <AnnotatedStep meta={`"role=\\"status\\""`} color="green">
    You rarely hand-set those two attributes, because the roles bundle the right combination. `role="status"` *implies* `aria-live="polite"` and `aria-atomic="true"`; `role="alert"` implies `assertive` and atomic. Learn the two roles and you've learned the common cases — reach for the role, not the raw attributes.
  </AnnotatedStep>

  <AnnotatedStep meta="{1-3}" color="orange">
    The rule the next diagram is about, and the one that breaks live regions most often: the region must already exist in the DOM *before* its content arrives. AT *monitors* registered regions for changes; a region that mounts with its text already inside was never registered in time, so the change is never seen.
  </AnnotatedStep>
</AnnotatedCode>

That last rule is subtle enough, and broken often enough, that it is worth *watching* rather than just reading. The thing to internalize is non-obvious: AT announcement keys off a *mutation inside a region it is already watching* — not off a region appearing. So the order of operations matters in a way that is invisible in the rendered output. Scrub through the sequence below, which shows AT's point of view over time for two implementations side by side: one that always mounts the region empty, and one that conditionally renders it.

<DiagramSequence>
  <DiagramStep>
    <PreMountRule step={1} />
    <Fragment slot="caption">
      **Mount.** The always-mounted region renders empty — so it's in the DOM, and AT *registers it and starts watching*. The conditional version renders nothing while `msg` is falsy: there's no region for AT to watch yet. Neither announces anything; both are silent. *The whole difference is decided right here, before any text exists.*
    </Fragment>
  </DiagramStep>

  <DiagramStep>
    <PreMountRule step={2} />
    <Fragment slot="caption">
      **`setMsg('Saved')`.** Left: "Saved" is inserted *into the region AT was already watching* — AT detects the mutation and announces it. Right: `msg` turns truthy, so the region mounts *with "Saved" already inside it*. AT sees a brand-new node appear, not a change in a watched region — so it stays silent (or fires unreliably). Same text on screen; only one is heard.
    </Fragment>
  </DiagramStep>

  <DiagramStep>
    <PreMountRule step={3} />
    <Fragment slot="caption">
      **`setMsg('Saved ✓')`.** Every subsequent change to the watched region is announced too — it keeps working. The conditional region now exists, but it was never registered, so its updates stay unreliable. AT announcement keys off a mutation *inside a region it already watches* — not a region that appears with text already in it.
    </Fragment>
  </DiagramStep>
</DiagramSequence>

The fix follows straight from the diagram: always render the region, toggle only its *contents*. The two versions sit one keystroke apart, and the difference is the whole bug.

<CodeVariants>
  <CodeVariant label="Broken — conditional render">
    <div data-mark-color="red">

    ```tsx {1}
    {error && <div role="alert">{error}</div>}
    ```

    </div>
    The region only exists when `error` is truthy — so it mounts *with the text already inside*, and AT was never watching it. The error is shown to sighted users and announced to no one. This is the live-region bug, in its most common shape.
  </CodeVariant>

  <CodeVariant label="Fixed — always-mounted region">
    <div data-mark-color="green">

    ```tsx "role=\"alert\""
    <div role="alert">{error}</div>
    ```

    </div>
    The region is always in the DOM; only its *contents* toggle (it's empty when `error` is falsy). AT registered it on mount, so when `error` fills it, AT announces the change. Same markup, reordered timing — and now it works.
  </CodeVariant>
</CodeVariants>

### `status` or `alert`?

The two roles split on one axis: **`alert` interrupts; `status` waits.** `alert` cuts into whatever AT is currently reading, every single time. That is the right behavior for a genuine, time-sensitive failure — and exactly the wrong behavior for routine confirmations. Wire your "Draft saved" badge as `alert` and you've turned the screen reader into a slot machine, jackpotting over the user's reading every few seconds. The cut is about severity:

- **`role="status"`** — the default reach, for anything informational: "Saving…", "Draft autosaved", "Search returned 12 results", "Copied to clipboard."
- **`role="alert"`** — reserved for genuine, time-sensitive failures the user must not miss: "Session expired", "Payment failed", "Connection lost."

When in doubt, it's `status`. `alert` is the exception you justify, not the default you reach for.

### You usually don't build the toast

Here is the payoff for understanding all of that machinery: most of the time, you get to *not* build it. The most common live region in any SaaS app is the toast, and shadcn's default toast is <Term definition={"shadcn/ui's default toast library since 2026 (it replaced the deprecated built-in Toast component).\nIt mounts one persistent live region at the app root and injects toast content into it, so the pre-mount rule is satisfied for you."}>Sonner</Term>. (If you find an older codebase using a `Toast` component imported from `components/ui/toast`, that's the deprecated predecessor — recognize it as legacy.) Sonner renders **one persistent live region at the app root** and injects every toast's content into it. Which means the pre-mount rule — the entire diagram you just scrubbed through — is already satisfied for you. The region is always mounted; Sonner mutates its contents.

So your job collapses to the one piece of judgment that's actually yours: picking the severity.

```tsx
toast.success('Saved');
toast.error('Payment failed');
```

The severity you pick is the politeness you get — a confirmation announces calmly, a failure with the urgency it deserves. You learned the machinery not so you'd hand-build a toast, but so you'd *trust the primitive that did* — and, just as importantly, so you can build the live region yourself when the surface isn't a toast and no primitive covers it. (Sonner's own setup is a job for a later chapter; here, the discipline is the severity call.)

Because that last case — a live region you build by hand — comes up more than you'd expect, here is where it shows up in a real SaaS UI:

- **Toasts** → Sonner. Handled, as above.
- **A search or filter result count** → a `role="status"` in the results header, updated after a filter runs: "Showing 12 of 134". This is the canonical one you build yourself — there's no primitive for "the list changed, tell AT how many."
- **An optimistic save indicator** → a "Saved" badge as `role="status"` when the optimistic update lands, and a `role="alert"` on rollback: "Couldn't save — changes reverted." (Optimistic UI — updating the screen before the server confirms — gets its full treatment in a later chapter; here it's just a live-region surface.)
- **An inline form error** → a `role="alert"` beside the field, populated when validation fails. The forms chapter (Chapter 44) owns the depth.

Recognize the pattern and you'll spot every place a live region belongs — and reach for one only where the primitive doesn't already cover you.

## The four surfaces, together

Step back and the whole lesson fits on one screen. Native HTML and the shadcn primitives are the floor — they handle the overwhelming majority of accessibility before you write a single `aria-` attribute. ARIA is the narrow slice on top, and it is exactly four jobs:

- **Roles** — almost always the primitive's, not yours. *Trigger: a custom widget no semantic element or primitive covers (rare). Mostly: `presentation` to strip inherited layout semantics, landmarks when you can't change the tag.*
- **Labels** — yours, on controls with no visible text. *Trigger: an icon-only button. The name goes on the control, never the icon.*
- **Descriptions** — yours, for help and error text. *Trigger: an input that needs a hint or an error read after its name. Wire it with `aria-describedby` and a `useId` id.*
- **Live regions** — yours, for async state a sighted user would just see. *Trigger: a toast, a result count, a save indicator. Mount the region first; pick `status` over `alert` unless it's a genuine, must-not-miss failure.*

And the single reflex that ties the WebAIM numbers back to the start of the lesson, the one to carry into every screen: **before you add any `aria-*`, ask whether a native element or your shadcn primitive already does this. If it does, that's the bug — fix the markup, don't add the attribute.** That question, asked every time, is the difference between the pages averaging 42 errors and the ones averaging 59. No ARIA is better than bad ARIA.

This lesson was about what AT *announces*. The next one is about where focus *is* — and it picks up the loose thread this one left dangling: the focusable element with `aria-hidden` on it, and the larger problem of focus order, which the platform does not solve for you.
