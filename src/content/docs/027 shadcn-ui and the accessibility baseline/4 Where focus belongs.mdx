A keyboard user is working down a list of invoices. They Tab to a row, press <kbd>Enter</kbd> on its link, and the next page loads. Their focus is now sitting on the link they just clicked — except that link belongs to the page that no longer exists. The next Tab does something arbitrary: it jumps to the browser's address bar, or to the top of `<body>`, or somewhere else entirely depending on the engine. The user has lost their place. A mouse user would never notice this, which is exactly why it ships: it is invisible to the person who wrote it.

Compare that to the web you already know. On a plain multi-page site, every navigation is a full page load, and a full page load *always* resets focus to the top of the new document. You got that reset for free, on every link, without ever thinking about it. Single-page apps traded it away — they swap content without reloading the document, which is faster and smoother, and they never handed the reset back. So now there is a question the browser used to answer for you, and you have to answer it yourself: **where is focus right now, and where should it be?**

This lesson answers that question for the three situations a SaaS app creates — a modal opening, a route changing, a form submitting — and, just as important, tells you which ones the platform already solves and which ones it leaves entirely to you. By the end you will be able to locate where focus responsibility falls to you, not just memorize three snippets.

One thing first, and it is not optional: **you cannot find any of these bugs with a mouse.** A mouse click sets focus wherever it lands, so it papers over every focus mistake in this lesson. The detection tool is the one from the lesson on the four commitments — unplug your mouse, or commit to Tab and Shift+Tab only, for the rest of this page. If you don't, none of what follows will feel real.

## The two verbs of focus

Before the three situations, two primitives. Get these crisp and the rest of the lesson is just applying them.

Start with the model. At any moment, **exactly one element on the page has focus** — there is a single cursor, and it has a position. You can read where it is: the browser exposes it as <Term definition={"The one element on the page that currently has keyboard focus.\nReading document.activeElement tells you where the focus cursor is right now.\nExactly one element has focus at a time."}>`document.activeElement`</Term>, the one element with the cursor. Tab moves that cursor forward through the focusable elements **in DOM order**; Shift+Tab moves it back. That is the whole model: one cursor, walking the DOM.

Not everything is in that walk. Interactive native elements — `<button>`, `<a href>`, `<input>`, `<select>`, `<textarea>` — join the Tab order for free; that is part of what you bought when you chose them over a `<div>`. Everything else — a `<div>`, an `<h1>`, a `<section>` — is *not* focusable by default and Tab skips right over it. This is the same "semantic HTML first" reflex from the lesson on the four commitments, seen from the focus side: the reason a `<button>` is reachable by keyboard is the reason you reached for it in the first place.

That gives us the two things you can do to the cursor, and they are genuinely different jobs with genuinely different tools.

**Verb one — make a thing targetable.** You already met `tabindex` and its three values in the lesson on the four commitments, so this is just naming the one value this lesson actually uses. `tabindex="0"` joins the Tab order in DOM position (rare in a shadcn codebase — a native control almost always fits). Positive values are the anti-pattern that scrambles tab order globally; never type one. The value that matters here is `tabindex="-1"`: **focusable by script, skipped by Tab.** It is how you make a non-interactive element — an `<h1>`, a `<main>` — a legal target for focus *without* dropping it into the user's Tab sequence. The user never tabs to it; your code can still send focus there.

**Verb two — move the cursor.** `element.focus()` moves focus to an element programmatically. And there is one option you should treat as the default for any page-level focus move: `element.focus({ preventScroll: true })`. Without it, the browser scrolls the focused element into view as a side effect — which fights whatever owns your scroll position (the framework's scroll restoration, on a route change) and can yank the viewport somewhere the user didn't ask to go. With `preventScroll: true`, you move focus and leave scrolling to whoever is supposed to govern it. Reach for it by default; the page won't jump.

Here is the pairing those two verbs add up to, and it is the single most useful shape in this lesson. To focus a non-interactive element like a heading, you need **both** verbs at once — `tabindex="-1"` so it *can* receive focus, and `.focus({ preventScroll: true })` to actually move there:

<div data-mark-color="blue">

```tsx "tabIndex={-1}" ".focus({ preventScroll: true })"
const headingRef = useRef<HTMLHeadingElement>(null);

const focusHeading = () => {
  headingRef.current?.focus({ preventScroll: true });
};

<h1 tabIndex={-1} ref={headingRef}>Invoices</h1>
```

</div>

Internalize this as one fact, because it is the source of the most common focus confusion there is — "I called `.focus()` on my heading and nothing happened." Nothing happened because a bare `<h1>` is not a focus target. `.focus()` alone is half the move; `tabindex="-1"` is the other half. You almost always need both. You will see this exact pairing twice more before the lesson is over.

## Modals already solve it: the focus-trap contract

The first situation is the one the platform handles for you completely — which is the right place to start, because the job here is not to *build* anything. It is to recognize what you already get, trust it, and not break it.

When a Radix `Dialog` opens (and every overlay primitive shadcn ships is Radix underneath), it fulfills a four-part contract. This is the <Term definition={"Focus is confined to a region — Tab cannot leave it until the region closes.\nThe correct behavior for a modal dialog and the wrong behavior for anything non-modal.\nRadix ships it for every overlay primitive."}>focus trap</Term> you met named in the lesson on the four commitments, now at full depth. The four guarantees:

1. **Focus moves *into* the dialog** when it opens — to the first focusable element, or the close button.
2. **Tab and Shift+Tab cycle *within* the dialog** and cannot escape to the page behind it. Tab off the last element wraps to the first; Shift+Tab off the first wraps to the last.
3. **Esc closes the dialog and returns focus to the trigger** that opened it — so the user lands back exactly where they were.
4. **The content behind the dialog is made <Term definition={"An HTML attribute and state that makes a subtree non-focusable and hidden from assistive technology.\nRadix applies it to the page behind an open dialog so nothing back there can be tabbed to or announced."}>`inert`</Term>** — neither focusable nor announced — so Tab and the screen reader both stay inside the dialog.

That is the shadcn dividend applied to focus: `Dialog`, `AlertDialog`, `Sheet`, `Drawer`, `Popover`, `DropdownMenu`, `Command` — every overlay in `components/ui/` ships all four guarantees. Your job is *not to break them*. Don't strip the markup that carries the behavior; don't override the close-on-Esc focus return without a reason.

It is worth feeling the weight of what that primitive is doing, because the weight is the argument for never hand-writing it. A *correct* focus trap has to handle: Tab and Shift+Tab cycling at both ends of the dialog; content that changes while the dialog is open (the element you focused disappears); focus escaping through paths you don't control (browser autofill, devtools, <kbd>Cmd</kbd>+<kbd>L</kbd> to the address bar); and restoring focus to the right trigger when it closes. Each of those is a sharp edge, and a hand-rolled trap usually has the first one and none of the rest. Radix gets all four right. So does a dedicated library (`focus-trap`, `react-focus-lock`) for the rare case you are building a portal-rooted custom surface shadcn has no primitive for — and even then, you reach for a library, never your own keydown handlers. The rule is blunt: you do not write a focus trap by hand.

There is exactly one case the primitive can't guess, and it shows up constantly in a SaaS app. Guarantee 3 — return focus to the trigger — assumes the trigger still exists. But picture a delete-confirmation `AlertDialog` opened from a table row, where confirming **deletes that row**. The trigger button was *in* the row. The row is gone. There is now no element to return focus to, and the user is stranded the moment the dialog closes. Radix can't know your mutation removed the trigger, so this one is yours: give the close an explicit destination instead. Focus a stable nearby anchor — the table's heading, the "New invoice" button, the list container — and do it in the dialog's `onCloseAutoFocus`, where you can intercept the default return.

The two versions sit side by side below. The first is the everyday case where the trigger survives and Radix handles everything; the second is the delete flow where you redirect.

So: situation one is mostly recognition. The primitive owns the trap; you learn the contract so you trust it and don't undo it, and you keep one custom move in your pocket for the deleted-trigger case. The next two situations are where the platform stops helping.

## The route-change gap Next.js leaves you

This is the load-bearing section of the lesson. It is the single most common accessibility regression in single-page apps, it is invisible in an automated audit, and it is exactly where this chapter's "focus is state" thread pays off in a copyable pattern.

Here is the gap, stated precisely. On a `<Link>` click or a `router.push`, the Next.js App Router does a <Term definition={"A client-side route change that swaps the page content without a full document reload.\nFaster and smoother than a full page load — but it skips the focus reset a real page load gives you for free."}>soft navigation</Term> and moves focus **nowhere.** It updates the URL, swaps the page content, and leaves the focus cursor exactly where it was — which, after the swap, points at an element that just unmounted. This is not a hack you're working around; it is a known, long-standing limitation of the App Router. The full page-load reset the static web gave you is gone, and nothing replaced it.

So the bug is the one from the very top of this lesson, now named as a mechanism: focus is stranded on the unmounted link → the next Tab is arbitrary → and the screen-reader user is hit twice, because nothing announced that the page even changed. They are left on a dead node with no idea where they are.

The fix is small and it is the same pairing you already know, applied at the page level. You focus the new page's main heading when the route changes. That takes a tiny client component, mounted once in the layout, that watches the path and moves focus on every change:

<AnnotatedCode lang="tsx" code={`
'use client';

import { useEffect } from 'react';
import { usePathname } from 'next/navigation';

export const RouteFocus = () => {
  const pathname = usePathname();

  useEffect(() => {
    const frame = requestAnimationFrame(() => {
      const heading = document.getElementById('page-heading');
      heading?.focus({ preventScroll: true });
    });
    return () => cancelAnimationFrame(frame);
  }, [pathname]);

  return null;
};
`}>
  <AnnotatedStep meta="{1, 4}" color="blue">
    This runs in the browser and needs the current path, so it's a client component. `usePathname` is the Next.js hook that hands us the route-change signal.
  </AnnotatedStep>

  <AnnotatedStep meta={`{9-15} "[pathname]"`} color="green">
    The effect re-runs every time `pathname` changes — i.e. on every navigation. This is a *legitimate* effect: it synchronizes React with an external system (the router), which is exactly what effects are for, so it doesn't violate "you probably don't need an effect."
  </AnnotatedStep>

  <AnnotatedStep meta={`{10-13} "document.getElementById('page-heading')"`} color="orange">
    Focus must land *after* the new route paints, so we defer one frame and query the heading fresh inside the effect, rather than holding a ref to a node from the previous route that no longer exists. This is the one subtlety: get it wrong and the focus call silently no-ops.
  </AnnotatedStep>

  <AnnotatedStep meta={`"{ preventScroll: true }"`} color="violet">
    Scroll position on a route change is the framework's job (scroll restoration). `preventScroll: true` moves focus without fighting it — the cursor goes to the heading, the scroll goes where the framework decides.
  </AnnotatedStep>
</AnnotatedCode>

Notice what the target is: an `<h1 tabIndex={-1}>` (or `<main id="main-content" tabIndex={-1}>`), the exact pairing from "the two verbs," now in its first real application. The `tabindex="-1"` makes the heading a legal focus target without putting it in anyone's Tab order; the `.focus({ preventScroll: true })` moves there. And because every page has exactly one `<h1>` — the heading-hierarchy rule you own from the lesson on semantic structure — "the heading" is never ambiguous. There is one, and it's the landing spot.

While you are here, there is a second pattern that shares this exact mechanism, so it's cheap to add right now: the **skip link.** It is a link that focuses `<main tabindex="-1">` — same target shape, same `tabindex="-1"`. Its job is for keyboard users who don't want to Tab through your entire nav on every single page. You make it the *first* focusable element in the layout; it sits visually hidden until it receives focus, then reveals itself, and activating it jumps the cursor straight past the nav into the main content.

<div data-mark-color="blue">

```tsx "sr-only focus:not-sr-only"
<a href="#main-content" className="sr-only focus:not-sr-only ...">
  Skip to content
</a>
...
<main id="main-content" tabIndex={-1}>
  {children}
</main>
```

</div>

The `sr-only` is the visually-hidden utility from the lesson on ARIA, and `focus:not-sr-only` is the reveal-on-focus trick: hidden until focused, visible the moment a keyboard user Tabs to it. The skip link is cheap to build and it is a concrete differentiator on a 2026 accessibility audit — the kind of thing that surfaces in a procurement review and that automated tools, true to the "tools catch a minority" point from the lesson on the four commitments, will not flag for you.

## After a submission: move focus and announce

The third situation is form submission, and the clean way to reason about it is to forget forms for a second and ask only: *after the submit resolves, what happened to the screen?* There are three answers, and focus goes somewhere different in each.

**It succeeded and navigated.** A create flow that redirects you to the new resource, say. You don't do anything — the route-change pattern from the last section already fired, and focus is on the new page's `<h1>`. Situation three collapses into situation two. Move on.

**It succeeded and stayed put.** An inline edit, a posted comment. Focus has to go to a *sensible anchor that you choose deliberately* — usually the action that originated the submit (the edit button you can now press again), or the next field, or the success indicator. And if you render a success message, it's a `role="status"` (from the lesson on ARIA) so a screen reader actually hears it. The focus move and the announcement are two halves of one event.

**It failed with errors.** Focus moves to the **first field with an error**, and that field's error is announced through the live region. This is where focus and announcement compose into the single most useful submission pattern there is: if you focus an input that is `aria-describedby` an error element with `role="alert"`, the screen reader reads the *label, the value, and the error in one pass* — "Email, not-an-email, Enter a valid email address." One focus move, one coherent announcement, everything the user needs.

Here is just the focus-and-announce skeleton for that third case — deliberately stripped of the form machinery so you can see the two moves clearly:

<div data-mark-color="blue">

```tsx "firstErrorRef.current?.focus()" "aria-describedby={errorId}"
const firstErrorRef = useRef<HTMLInputElement>(null);
const errorId = useId(); // stable id for the label/error wiring — useId is the forms chapter's job

// Form submission + validation live in the forms chapter — this is the focus + announce half.
const focusFirstError = () => {
  firstErrorRef.current?.focus();
};

<input ref={firstErrorRef} aria-describedby={errorId} />
<p id={errorId} role="alert">Enter a valid email address.</p>
```

</div>

That is deliberately a sketch. The real machinery around it — the Server Action, `useActionState`, generating the ids with `useId`, `aria-invalid`, rendering the whole error tree — is a large topic, and it belongs to the forms chapter (Chapter 44). This lesson owns exactly one half of that: move focus to the first error, and make sure something announces it. The boundary is intentional, not an unfinished example.

Pull the three cases together and they share one rule, and it's the one genuinely new idea in this lesson:

:::tip
**Move focus and announce, or do neither.** A focus move the screen reader has nothing to say about is just disorientation — the cursor jumped and the user doesn't know why. An announcement with no focus move leaves a keyboard user's cursor stranded somewhere else while text is read out of nowhere. The two are one event. When you move focus on a disruptive change, make sure there's something to announce; when you announce, make sure focus is somewhere sensible.
:::

That rule has a sharp edge, and it's the failure mode this whole section invites. The power to move focus tempts you to move it on *every* state change — every keystroke, every re-render, every little update. That is "over-focusing," and it is actively harmful: each focus move interrupts the screen reader, so a UI that re-focuses constantly *steals its own announcements* and leaves an AT user unable to hear anything to completion.

:::caution
Move focus only on genuinely disruptive transitions — a dialog opening or closing, a route changing, a submit resolving. Not on every keystroke, not on every render. If you find yourself calling `.focus()` inside an event that fires rapidly, you are over-focusing, and you are making the experience worse for the exact users you're trying to help.
:::

### `autoFocus` — the careful reach

There's a related shortcut worth placing precisely. React's `autoFocus` prop focuses an element when it mounts. It's right in a narrow set of cases and wrong in a couple of important ones.

It's *right* on single-purpose landing screens where the user's intent is unambiguous the instant the page appears: a sign-in screen's email field, a one-input search page, a command palette that just opened. There's only one thing to do, so focusing it for the user is a kindness.

It's *wrong* on multi-section forms — focusing the first field steals focus before the screen reader has read the surrounding context the user needs — and it's wrong on dialogs, where Radix already manages initial focus and `autoFocus` just fights it. (That's the modal section again: let the primitive own initial focus.) One mechanical note to file away: `autoFocus` fires per **mount**, so a component that remounts re-fires it, which can surprise you if the element comes and goes.

## Two traps the platform sets

Two more focus mistakes that don't fit the three situations but bite constantly. They're grouped because they share a shape: **the obvious move is the wrong one.**

### Tab order follows the DOM, not the CSS

You planted this rule back in the lesson on the four commitments; here is where it bites. Tab order is DOM order, full stop. Visual reordering with CSS — `flex-direction: row-reverse`, `order: -1`, `grid-template-areas`, a `flex-wrap` reflow — moves *pixels*, not the cursor. So a user can see your sidebar on the right and your main content on the left, Tab, and land in the sidebar first, because in the DOM the sidebar comes first. The reading order they see and the Tab order they get have silently diverged.

The fix is the one you already know and it is not the tempting one. **If the focus order is wrong, the markup order is wrong — reorder the DOM.** Do not patch it with `tabindex`: a positive `tabindex` to "fix" visual order is the anti-pattern from the lesson on the four commitments, and it scrambles the rest of the page to paper over this one spot. When you need a layout that mirrors (right-to-left, say), the correct tool is logical properties from the lesson on logical layout — they flip the layout without touching source order, so the DOM order stays the reading order. The principle holds: the fix is always the DOM, never the CSS.

### `disabled` vs `aria-disabled` — discoverability decides

When you disable a control, you reach for the native `disabled` attribute. Usually that's correct: `disabled` removes the control from the Tab order entirely and announces "disabled" — a keyboard user tabs right past it, which is exactly what you want for most disabled things.

But "removes it from the Tab order" is sometimes the bug. Picture a submit button that's disabled until a form validates. A native `disabled` button is *skipped* by Tab — so a screen-reader user working down the form never lands on it, never discovers it's there, and never hears *why* they can't submit. The thing you disabled became invisible to the user most in need of an explanation.

That's the case for `aria-disabled="true"`. It keeps the element focusable and discoverable — the user can Tab to it, hear it, hear the reason if you pair it with a `role="status"` message — *but* it does not block activation the way native `disabled` does. So you take on one responsibility in exchange: **guard the handler yourself**, early-returning from `onClick` while the disabled state holds, since `aria-disabled` won't stop the click for you.

The decision is a single question, and the walker below makes you walk it in order. Default to native `disabled`; reach for `aria-disabled` only when the disabled control must stay discoverable.

    <Branch label="No — just keep it reachable" to="aria-bare" />
  </Question>

  <Leaf id="native-disabled" verdict="disabled">
    The native `disabled` attribute removes the control from the Tab order entirely and announces "disabled." The correct default for most disabled controls — a keyboard user tabs straight past it, which is exactly what you want.
  </Leaf>

  <Leaf id="aria-bare" verdict="aria-disabled + guarded handler">
    `aria-disabled="true"` keeps the control focusable and discoverable — the user can Tab to it and hear it's disabled. Because `aria-disabled` doesn't block activation, you must early-return from `onClick` while the disabled state holds.
  </Leaf>

  <Leaf id="aria-reason" verdict="aria-disabled + guarded handler + a reason">
    Same as the reachable case, plus a `role="status"` message (wired via `aria-describedby`) explaining *why* the control is inert — e.g. a submit button waiting on validation. The user reaches it, hears it's disabled, and hears the reason.
  </Leaf>
</StateMachineWalker>

## Make focus visible where it lands

One closing reminder, because a lesson about *moving* focus is incomplete without it. Most of the time you're moving focus to native elements or shadcn primitives, which already show a focus ring. But the moment you make something focusable that *isn't* natively interactive — a clickable card, a custom selectable row with `tabindex="0"` (rare, but it happens) — you've created a target a keyboard user can land on but, unless you do something, **cannot see.** A focus cursor you can't see is its own bug.

So give it a ring:

<div data-mark-color="blue">

```tsx "focus-visible:outline-none" "focus-visible:ring-2" "focus-visible:ring-ring"
<div
  tabIndex={0}
  className="rounded-lg border p-4 focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring"
>
  ...
</div>
```

</div>

Those are the canonical classes: `focus-visible:outline-none` drops the default outline, and `focus-visible:ring-2 focus-visible:ring-ring` draws a ring in shadcn's semantic `--ring` token — theme-owned, so it's the same ring the rest of your app uses, and the "audit the token" reflex from the lesson on the four commitments applies. `:focus-visible` (rather than `:focus`) is what you met in the lesson on focus styles — it shows the ring for keyboard focus and not for mouse clicks. And the one rule from the lesson on the four commitments that this closes the loop on: never *remove* a focus ring to tidy up a design. Style it. A removed ring is a keyboard user with no idea where they are.

## What you own now

Pull it together. The platform solves one of the three situations completely — the modal trap, owned by Radix — and leaves you the other two. On a route change it moves focus nowhere, so you send focus to the new heading. On a submission you move focus by outcome — to the new page, to a sensible anchor, or to the first error — and you pair every move with an announcement. Underneath all three is one model (a single cursor you can read and move) and two verbs (`tabindex="-1"` to make a target, `.focus({ preventScroll: true })` to move there). And underneath *that* is the reflex the whole lesson is really about: after any disruptive change, ask where the cursor landed, because the platform won't warn you when it landed nowhere.

<TrueFalse instructions="Each claim is about where focus is — and whose job it is to put it there.">
  <Statement answer="false">
    Next.js App Router automatically moves focus to the new page on client-side navigation.
    <TfWhy>It moves focus *nowhere* — the cursor stays on the now-unmounted element. Restoring focus to the new page's heading is the engineer's job, and it's the most-shipped SPA accessibility regression.</TfWhy>
  </Statement>

  <Statement answer="false">
    `tabindex="-1"` puts an element in the Tab order.
    <TfWhy>It makes an element focusable *by script* while keeping it *out* of the Tab order — the user can't Tab to it, but your code can `.focus()` it. That's exactly what you want for a heading you focus on a route change.</TfWhy>
  </Statement>

  <Statement answer="false">
    When a custom modal needs a focus trap, the right move is to hand-write one with keydown handlers.
    <TfWhy>A correct trap handles cycling at both ends, content changing while open, focus escaping via autofill/devtools, and return-focus restoration. Use the Radix primitive, or a library for a custom surface — never your own handlers.</TfWhy>
  </Statement>

  <Statement answer="true">
    Moving focus to an input whose error is in a `role="alert"` element (via `aria-describedby`) lets a screen reader announce the label and the error together.
    <TfWhy>This is the submission pattern to reach for — one focus move produces one coherent announcement of label, value, and error.</TfWhy>
  </Statement>

  <Statement answer="false">
    Reordering elements visually with CSS `order` also changes the Tab order.
    <TfWhy>Tab follows DOM order; CSS moves pixels, not the focus cursor. If the focus order is wrong, the markup order is wrong — reorder the DOM, never patch it with `tabindex`.</TfWhy>
  </Statement>

  <Statement answer="false">
    A button disabled with the native `disabled` attribute is still reachable by Tab.
    <TfWhy>Native `disabled` removes it from the Tab order entirely. That's exactly why `aria-disabled` exists — for the case where a disabled control must stay discoverable so the user can hear why it's inert.</TfWhy>
  </Statement>

  <Statement answer="true">
    `element.focus({ preventScroll: true })` moves focus without scrolling the element into view.
    <TfWhy>It's the default to reach for on page-level focus moves — focus goes where you send it while scroll stays governed by the framework's scroll restoration, so the viewport doesn't jump.</TfWhy>
  </Statement>
</TrueFalse>

The loose thread this lesson leaves is the one the previous lesson handed it: you now know where focus *is* and how to move it, but a screen that's loading, empty, or broken has nothing to focus *yet*. The next lesson is about those four states — loading, empty, error, populated — and designing all four instead of only the happy path.
