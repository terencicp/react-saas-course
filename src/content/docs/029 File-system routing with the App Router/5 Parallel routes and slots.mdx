You have seen this screen in every serious SaaS tool you have ever used. Gmail, Linear, an invoices dashboard: a list of things runs down one side, and a detail pane on the other side updates the moment you click a row — all at one URL you can copy, paste into a teammate's chat, and have them land on exactly the same invoice you were looking at. That is the screen to build, and you already own most of the pieces. Layouts give you the shared shell. The `[id]` segment gives the detail pane a URL per invoice. `<Link>` makes clicking a row feel instant. So here is the question an experienced engineer asks before opening the editor: **how do I render the list on the left and the selected invoice on the right as one screen, where each side gets its own URL, its own loading and error behavior, and clicking a row swaps only the detail without re-rendering the list?**

Everything you have learned so far renders *one* route tree at a URL. `/invoices/42` resolves to a single chain of layouts wrapping a single page. What this screen needs is two route trees living at the same URL at the same time — the list tree and the detail tree, resolved independently, handed to one layout together. That capability has a name in the App Router: **parallel routes**, built from folders prefixed with `@`. We will build the model in three passes. First, a slot is just another child the layout receives. Then, each slot is a full route tree of its own — the part that makes the list stay put while the detail changes. Last, the one file you must never forget, or the page works for you and 404s for everyone you share it with.

## A slot is another child the layout receives

Before any new folder rules, install the one idea everything else hangs off. Cast your mind back to layouts: the layout receives `children`, and *the framework fills it* — you never pass `children` to a layout yourself, you receive it. Here is the reframe that collapses most of the apparent novelty in this lesson. That `children` prop **is itself a slot** — the layout's unnamed slot, filled automatically with the matching segment's `page.tsx`. A parallel route just adds a *named* slot beside it, filled the exact same way.

You create a named slot by adding a folder whose name starts with `@`. Picture the invoices route with a `@detail` folder dropped in next to its page:

<FileTree>
- src/app/
  - invoices/
    - layout.tsx receives `children` **and** `detail`
    - page.tsx fills `children` — the list
    - **@detail/**
      - page.tsx fills the `detail` prop
</FileTree>

That `@detail` folder does *not* add a URL segment — there is no `/invoices/@detail` route, and we will come back to why that matters. What it does is hand the `invoices` layout a second filled prop, named `detail`, sitting right beside `children`. Look at the layout that receives it:

<AnnotatedCode lang="tsx" code={`
export default function InvoicesLayout({
  children,
  detail,
}: {
  children: React.ReactNode;
  detail: React.ReactNode;
}) {
  return (
    <div className="grid grid-cols-[1fr_2fr] gap-6">
      <section>{children}</section>
      <aside>{detail}</aside>
    </div>
  );
}
`}>
  <AnnotatedStep meta="{2-3}" color="blue">
    The layout destructures two props. `children` you already know — it's the same-segment page. `detail` is new, and here is the rule that drives this whole lesson: the prop name is the slot folder's name minus the `@`. `@detail` on disk becomes a `detail` prop in the layout. Add a `@reviews` folder and you would receive a `reviews` prop.
  </AnnotatedStep>

  <AnnotatedStep meta={`"children: React.ReactNode" "detail: React.ReactNode"`} color="blue">
    Both are typed `React.ReactNode`, the same type `children` always carries. That's the tell that a slot is filled with already-rendered UI, exactly like `children` — the framework hands you finished elements, not a component to call. One note: these prop types are hand-written here so the slot-to-prop link is impossible to miss. In production you'd type the whole parameter with the generated `LayoutProps<'/invoices'>` helper, which names the slots for you — the same generated-types family as the `PageProps` you met in the dynamic-segments lesson, and which the detail page below uses.
  </AnnotatedStep>

  <AnnotatedStep meta="{10-11}" color="blue">
    You decide *where* each slot renders. `children` goes in the left column, `detail` in the right — but you could swap them, nest them, or wrap either one. The framework decides *what's inside* each slot; you decide *where it goes*. Identical contract to `children`, just a second one.
  </AnnotatedStep>
</AnnotatedCode>

That is the entire core of the feature, and it really is this small: **add a `@x` folder, get an `x` prop.** Every other thing in this lesson — the list staying mounted, the streaming, the one file you must not forget — is a consequence of that one rule.

<Term definition={"A named region of a layout the framework fills from a matching @-folder route, sitting beside children.\nThe prop name is the folder name minus the @."}>Slot</Term> is the word for one of these named regions, and you will see it everywhere in the App Router docs. With the vocabulary in place, let's make a slot do something a plain prop can't.

## Each slot is its own route tree

Here is the payoff, and it is bigger than "a layout with two props." A `@slot` folder is not a single file you point at. It is a **complete, independent route subtree** — it gets its own `page.tsx`, its own `[id]` dynamic segments, its own `loading.tsx` and `error.tsx`. When a URL comes in, the router matches it against *every* slot independently and hands all the matches to the layout at once. That is the mechanism behind the list-plus-detail screen, so let's build the real shape:

<FileTree>
- src/app/
  - invoices/
    - layout.tsx receives `{ children, detail }`
    - page.tsx the list — fills `children` at `/invoices`
    - @detail/
      - page.tsx empty placeholder — fills `detail` at `/invoices`
      - [id]/
        - **page.tsx** the selected invoice — fills `detail` at `/invoices/42`
</FileTree>

Read that tree as two route trees stacked in one folder. The `children` tree is shallow: `page.tsx` is the list, and that is all it has. The `detail` tree is deeper: it has its *own* `page.tsx` (shown when no invoice is selected) and its own `[id]` segment (shown when one is). The URL decides what fills each tree, and crucially, it decides them **separately**. Walk the two URLs this screen lives at:

<UrlToSlotMapping />

Linger on what changed between those two steps, because it is the whole reason this feature exists. Navigating from `/invoices` to `/invoices/42`, the `children` slot's match did not change — it is `page.tsx`, the list, in both. So the framework leaves the list pane exactly as it is: same DOM, same scroll position, same client state in any toggles or filters the user touched. Only the `detail` slot resolved to something new, so only the right pane re-renders. **Click a row, navigate to a real shareable URL, and the list never flickers.** That is the behavior the entire SaaS industry hand-rolled with client-side state machines for a decade — a "selected item" kept in React state, the detail conditionally rendered, the URL out of sync with what was on screen. Parallel routes give it to you URL-backed, refreshable, and shareable, for the price of a folder.

And because each slot is an ordinary route tree, the detail page is an ordinary dynamic page — nothing special about living in a slot. It reads its slice of the URL the same way any `[id]` page does:

```tsx title="src/app/invoices/@detail/[id]/page.tsx"
export default async function InvoiceDetail({
  params,
}: PageProps<'/invoices/[id]'>) {
  const { id } = await params;
  // capture → validate → query
  return <InvoiceCard id={id} />;
}
```

Everything there is a reflex you already have: `async` component, `params` typed with the generated `PageProps<'/invoices/[id]'>` helper, `await params` because request inputs are Promises in Next 16, then the capture-validate-query path you'd run for any dynamic route. The detail slot reads `params`; the list's `page.tsx` would read `searchParams` to filter and sort — same idea on the other slot. (We are not building the filter here; URL-driven list state is its own topic later in the course. For now, just file it: each slot reads its own slice of the URL, independently.)

Step back and connect this to the layout boundary you already know. A layout stays mounted while the page beneath it changes — that is what made shared shells cheap. Parallel routes are that same idea, widened: the layout *and* the list pane both sit above the navigation that happened, so both stay mounted, while the one slot whose match changed swaps beneath them. It is nested-layout persistence with two independent below-the-fold trees instead of one.

## default.tsx and the unmatched slot

You can now build the screen, and it will work flawlessly while you click around in development. Then you ship it, a user opens a shared `/invoices/42` link in a fresh tab, and the entire page 404s. This is the single most common parallel-routes bug, and the reason it bites so reliably is that the happy path hides it completely. Let's make the failure visible so you never get caught by it.

Start with how the router actually tracks slots. It holds **one current match per slot** — one for `children`, one for `detail`. When you click a `<Link>`, that is a <Term definition={"A client-side page swap with no full document reload.\nThe router re-resolves only the slots whose match changed and keeps the previous match for the rest."}>soft navigation</Term>: the router re-resolves only the slots whose match actually changed and **keeps the previous match for the others**. That is precisely why clicking `/invoices/42` updated `detail` and left `children` untouched — `children`'s match did not change, so the router kept it.

Now the case that breaks. A <Term definition={"A full document load from the URL alone — a direct visit, a refresh, or opening a link in a new tab.\nEvery slot is resolved fresh; there is no previous match to keep."}>hard navigation</Term> — a direct visit to `/invoices/42`, a refresh, a `Cmd`- or `Ctrl`-click that opens a new tab — starts from nothing. There is no "previous match" to keep, because there was no previous page. The router has to resolve **every** slot from the URL alone. And if any slot has no route matching that URL, the framework has nothing to put there — and rather than render half a screen, it 404s the **whole route**. Not just the empty slot. The entire page.

This is why the bug is so sneaky. Clicking from `/invoices` to `/invoices/42` is a soft navigation, and the router carries `children`'s match forward, so every slot is always filled — it looks perfect. The instant someone hard-loads a URL one of your slots can't match, the page falls over. Your dev session never hits that path; your users hit it on their first click.

The fix is one file: `default.tsx`. A `default.tsx` inside a slot folder is that slot's **fallback for the unmatched case** — what the framework renders for the slot when, on a hard navigation, the URL matches none of the slot's routes. It has the same default-export shape as `page.tsx`. Watch all three cases play out:

<DefaultMechanics />

So you add the file:

<FileTree>
- src/app/
  - invoices/
    - layout.tsx
    - page.tsx fills `children` — the list
    - @detail/
      - page.tsx fills `detail` at `/invoices`
      - **default.tsx** the unmatched-slot fallback
      - [id]/
        - page.tsx fills `detail` at `/invoices/42`
</FileTree>

Carry one reflex out of this section: **every parallel slot ships a `default.tsx`.** Make it muscle memory the way you reach for a `key` on a list. What goes *in* the file depends on whether the empty state needs UI:

<CodeVariants syncKey="default-shape">
  <CodeVariant label="Return null">
    ```tsx title="src/app/invoices/@detail/default.tsx"
    export default function Default() {
      return null;
    }
    ```
    **The minimum that satisfies the rule.** The slot renders nothing on the unmatched case, and the route survives. Reach for this when an empty slot should simply show empty.
  </CodeVariant>

  <CodeVariant label="Neutral placeholder">
    ```tsx title="src/app/invoices/@detail/default.tsx"
    export default function Default() {
      return (
        <p className="text-muted-foreground">
          Select an invoice to see its details.
        </p>
      );
    }
    ```
    **The same fallback, with the empty state the user should actually see.** For a detail pane, an explicit "nothing selected yet" prompt beats a blank rectangle. Both are legitimate — pick by whether the empty state deserves UI.
  </CodeVariant>
</CodeVariants>

<Aside type="caution">
If a slot has no `default.tsx` and no matching route on a hard load, Next.js returns a 404 for the **whole page**, not just that slot. Add a `default.tsx` to every slot you create — even one that only `return`s `null`. This is the bug that passes every dev click-through and breaks the first time a real user opens a shared link.
</Aside>

One honest subtlety, because you will wonder about it. If `children` is itself a slot, does it need a `default.tsx` too? In the simple single-segment shape here, no — `children` always matches the segment's own `page.tsx`, so there is nothing to fall back from. But the framework can still fail to recover `children`'s active state on a hard load once you nest slots more deeply, and Next.js's own "missing default" guidance explicitly covers the implicit `children` slot. So the durable rule is broader than "every named slot": **a `default.tsx` is needed wherever any slot — named, or the implicit `children` — can go unmatched on a hard navigation.** "Every named slot ships one" is the practical floor that keeps you safe day to day; don't conclude from it that `children` is magically exempt.

## Slots stream on their own

One more capability falls out of "each slot is its own route tree," and it is worth naming even though we will not build it here. Because a slot is an independent tree, it can carry its own `loading.tsx`, and the framework wraps each slot in its own loading boundary. A slow query in the `detail` slot does **not** block the list from painting — the layout shell and the already-resolved list render immediately, the detail pane shows its own skeleton, and it streams in when its data is ready. Each slot loads on its own clock.

<FileTree>
- src/app/
  - invoices/
    - @detail/
      - **loading.tsx** skeleton shown only while the *detail* slot loads
      - page.tsx
      - [id]/
        - page.tsx
</FileTree>

That `loading.tsx` shows only while the *detail* slot is resolving, independently of the list — the list is already on screen.

<Aside type="note">
How that loading boundary actually works — Suspense, streaming, where to place the boundary — is a topic of its own in the next chapter. Here it is enough to know the shape: **each slot is its own loading boundary.** We are not wiring up the streaming mechanics in this lesson.
</Aside>

## @slot, (folder), and \_folder

Three folder conventions now change how routing behaves *without* adding a URL segment, and they are easy to blur together because they look similar and all "don't show up in the URL." Pin the differences down before they tangle. The discriminator to lead with: `_folder` **is not a route at all**, while `(folder)` and `@folder` *are* routes that simply contribute no URL segment — and those two differ in what they hand the layout.

| Convention | What it is | Adds a URL segment? | What the layout receives |
| --- | --- | --- | --- |
| `_folder` | Private folder — colocated, non-routable code | No — invisible to the router | Nothing; it is not a route |
| `(folder)` | Route group — organizes siblings, picks a layout | No | Normal `children` |
| `@folder` | Parallel slot — an independent route tree | No | A **named prop** beside `children` |

One line each, in the language already in play: `_folder` is "not a route"; `(folder)` is "a route minus its URL segment"; `@folder` is "a route that arrives as a named prop." That last contrast is the one to hold onto — `@detail` is **not** a route group. You never write `/invoices/@detail/...`; the `@` segment is invisible to the URL exactly like `(group)` is, but it surfaces as a *prop*, not as more `children`. And a slot is scoped to **one** layout: the layout in the same folder as the `@` directory. A parent layout or a child layout never sees it.

## When parallel routes earn their weight

You have a power tool now, and the experienced move with a power tool is knowing when *not* to reach for it. The default is still **one page**. Parallel routes are an escalation you reach for only when a specific threshold is crossed: **two or more regions of one screen each need their own URL-driven state, loading, error, or not-found behavior — and you want both reflected in a single URL you can refresh and share.** Cross that line and they are exactly right. Stay below it and they are overkill.

Three situations genuinely cross it. The first is the list-plus-detail surface we built and its cousins — any split screen where each pane carries independent URL state. The second is independent loading and error per region: a dashboard where the activity feed can throw its own error and show its own retry without taking down the KPI cards beside it, because each lives in its own slot with its own error boundary. The third is the modal-with-a-real-URL pattern, which lives in a `@modal` slot — that is the next lesson.

Now the trap, because this is exactly where beginners overuse the feature. **Two unrelated routes that merely share a layout are not parallel routes — that is nested layouts, which you already have.** A single page that fetches two things is not parallel routes either; that is one `page.tsx` doing two reads. Parallel routes earn their complexity *only* when the regions each need independent URL, loading, or error behavior. If they don't, a plain layout plus a page is simpler, and simpler wins. File parallel routes as the *escalation* from nested layouts, not a replacement for them — you climb to them when one layout with one page can no longer express what each region needs.

## Recap and what's next

Three sentences hold the whole lesson. A `@folder` gives the same-segment layout a named prop beside `children` — add a `@x` folder, get an `x` prop. Each slot is an independent route tree resolved from the same URL, which is why the list stays mounted while only the detail swaps. And every slot ships a `default.tsx`, or a hard navigation to an unmatched slot 404s the entire route.

Test the resolution model on the case that breaks people. Given the canonical invoices tree, put the events in order for a **refresh on `/invoices/42`** — a hard navigation, so every slot resolves fresh from the URL:

<Sequence instructions="A user refreshes the page on /invoices/42. Order what the router does to resolve it.">

```
src/app/invoices/
  layout.tsx
  page.tsx
  @detail/
    page.tsx
    default.tsx
    [id]/
      page.tsx
```

  <Step>The browser issues a full document request for `/invoices/42`</Step>
  <Step>The router resolves `children` against `src/app/invoices/page.tsx` (the list)</Step>
  <Step>The router resolves `@detail` against `@detail/[id]/page.tsx` — it matches, so `default.tsx` is not used</Step>
  <Step>The layout receives `{ children, detail }`, both filled</Step>
  <Step>The page renders with the list on the left and invoice 42 on the right</Step>

The next lesson takes the exact slot machinery you just learned and adds one folder that *intercepts* navigation. The result is the pattern every polished SaaS app uses: click a row and the invoice opens in a modal over the list, but the URL is real — refresh it or share it and you get the full invoice page. Same `@slot` you now understand, one new trick on top.
