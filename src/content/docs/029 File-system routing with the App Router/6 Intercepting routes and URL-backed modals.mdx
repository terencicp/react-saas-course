Picture a photo feed — a grid of thumbnails. You click one, and a lightbox opens over the grid: the photo gets bigger, the feed dims behind it, and you can still feel the page you came from underneath. Click another, the next photo slides in. Hit the back button, the lightbox closes and you're back at the grid. Instagram does this. Dribbble does this. It is one of the most familiar interactions on the web.

You already know how to build a version of it. You've written modals with local state before: a `selectedId`, a `setSelectedId`, and a `<Dialog>` that renders when something is selected.

```tsx title="feed.tsx"
'use client';

export const Feed = ({ photos }: { photos: Photo[] }) => {
  const [selectedId, setSelectedId] = useState<string | null>(null);

  return (
    <>
      <div className="grid grid-cols-3 gap-2">
        {photos.map((photo) => (
          <button key={photo.id} onClick={() => setSelectedId(photo.id)}>
            <Image src={photo.src} alt={photo.alt} width={300} height={300} />
          </button>
        ))}
      </div>

      {selectedId != null && (
        <Dialog open onOpenChange={() => setSelectedId(null)}>
          <DialogContent>
            <PhotoDetail id={selectedId} />
          </DialogContent>
        </Dialog>
      )}
    </>
  );
};
```

This works. It opens, it closes, it shows the photo. Ship it and most users will not complain. But look at what it can't do, all for the same reason — the modal lives in a component's state, and component state is invisible to the URL:

- **You can't share it.** Copy the address bar and send it to a colleague and they land on the bare feed. The thing you wanted them to see isn't in the link.
- **Refresh closes it.** Reload the page and `selectedId` resets to `null`. The modal is gone.
- **Back leaves the page.** The browser back button doesn't close the modal — it navigates away entirely, because as far as the browser knows, nothing changed when the modal opened.
- **Cmd-click does nothing useful.** There's no URL behind the photo, so you can't open it in a new tab the way you'd expect to open any other piece of content.

Every one of those is a missing URL. The modal is a real, addressable view that the user thinks of as a place — and it has no address. The fix is to give it one: a <Term definition="A URL that reproduces a specific in-app view — paste it cold and you land exactly where the link points.">deep link</Term> — so the modal becomes shareable, refreshable, and survives back and forward, while *still* rendering in context over the feed instead of yanking you off to a separate page.

Next.js ships a convention built for exactly this: **intercepting routes**. And here's the good news — you already have most of the machinery. The parallel-route slots from the last lesson do almost all the work. Today adds exactly one new thing: a folder prefix.

## Two ways to arrive at the same URL

Before any folder syntax, hold onto one idea, because the entire pattern hangs off it: a single URL can render two different things, and which one you get depends on *how you got there*.

You met the two ways to "get there" in the navigation lesson. Let me bring them back, because here they stop being trivia and become the spine of the feature.

<Term definition={"A client-side page swap with no full document reload.\nReact updates only the part of the tree that changed and leaves the rest mounted."}>Soft navigation</Term> is movement *inside* the running app — a `<Link>` click or a `router.push`. The browser doesn't reload the document. React swaps the part of the tree that changed and leaves everything else mounted, so the layout you were looking at stays exactly where it was. **This is the path interception fires on.**

<Term definition={"A full document load from the URL alone — a direct visit, a refresh, or opening a link in a new tab.\nThe browser resolves the route from scratch; there is no running app to intercept."}>Hard navigation</Term> is a full document load. Pasting a URL into a fresh tab, refreshing, Cmd-clicking into a new tab, following a link from someone's email — in every one of these the browser throws away whatever was on screen and resolves the URL from scratch, starting from nothing. **Interception does not fire here.** There's no running app to intercept; the browser just asks the server for that URL and renders whatever the server returns.

Now the rule the whole lesson rests on:

:::note
An intercepting route is the **soft-navigation** render path. The **hard-navigation** path always renders the real, standalone page.
:::

That has a direct consequence you cannot skip. Because the two navigation kinds need two different results, every intercepting route is **paired** with a non-intercepting sibling — one file for each path. The intercepter handles the in-context soft-nav render; the real page handles the standalone hard-nav render. Build the intercepter and forget its sibling and you've shipped a modal that works beautifully until the first person refreshes or pastes the link — then it 404s. That is the single most common interception bug, and we'll meet it again when we write the files.

Let me make the dual path concrete before we touch syntax. The diagram below shows the same destination — a photo with id `42` — reached two different ways. Flip between the tabs and watch what renders.

Read those two tabs against each other. The URL is identical — `/photo/42`. On soft navigation the feed is still there and the photo floats over it in a modal. On hard navigation there's no feed to float over, so you get the full page. Same address, two renders, and the *only* difference is how the user arrived. Keep this picture in your head; every code sample below is read against it.

There's one subtlety worth pinning down because it's what makes the whole thing pay off. On soft navigation, even though the modal is what renders, the browser's address bar shows `/photo/42` — the real route's URL. The URL is *masked* to the standalone page's address while the feed stays mounted underneath. That mask is the magic trick: the address bar shows a genuine, shareable URL, so the user can copy it, bookmark it, or send it — and when *that* person opens it cold, hard navigation kicks in and they get the full page. The two paths meet at the same address from opposite directions.

## The intercepting prefix

Now the syntax, and it's small. You mark a route folder as an intercepter by prefixing its name with one of four markers. The folder then says: "intercept soft navigations heading to *this* URL, and render *me* in their place instead of the real page."

The marker isn't decoration — it encodes a distance. Specifically, **how far up the URL tree the segment you're intercepting sits, measured from the intercepter's own folder.** There are four:

| Prefix | Intercepts a route… |
| --- | --- |
| `(.)folder` | at the **same** URL level |
| `(..)folder` | **one** URL level up |
| `(..)(..)folder` | **two** URL levels up |
| `(...)folder` | from the **root** `app` directory |

If you've used relative file paths in a terminal, `(.)`, `(..)`, and `(..)(..)` will feel familiar — same level, one up, two up — and `(...)` is the special "jump all the way to the top" case. The resemblance is deliberate. But there's a trap hiding inside it, and it is the single most counterintuitive thing in this lesson, so I'm going to slow down here.

:::caution
The count is **URL-segment-relative, not file-system-relative.**
:::

`(..)` means "one *URL* segment up," not "one *folder* up." Most of the time those are the same number and you never notice the difference. They diverge the moment a folder in your path contributes no URL segment — and two kinds of folder do exactly that. A parallel-route slot like `@modal` adds no URL segment (you learned this in the last lesson — `@slot` folders organize the render tree, they don't appear in the address). And a <Term definition={"A folder wrapped in parentheses, like (marketing), that organizes routes without adding a URL segment."}>route group</Term> — a `(folder)` in parens — adds no URL segment either (you learned *that* a few lessons back). Both are invisible to the prefix count.

This isn't an edge case you can file away for later. It's the case that decides the prefix for the pattern we're about to build. Our intercepter is going to live *inside* the feed's `@modal` slot. So when we count "how far up is the photo route," the `@modal` folder doesn't count — it's two folders up on disk, but only **one URL segment up**. That's why the canonical example uses `(..)photo` and not `(...)`. Let me show you the divergence directly, because seeing it once fixes it.

<Figure>
  <PrefixArithmetic />
  <Fragment slot="caption">
    `@modal` is a slot, not a URL segment — so it drops out of the count. Two folders up on disk is one URL segment up. Hence `(..)photo`, not `(...)`.
  </Fragment>

Look at the two strips. On disk, the path from `@modal` up to the `photo` route crosses two folders. On the URL, it's a single segment, because `@modal` simply isn't there. The prefix counts the bottom strip, never the top one. So `(..)photo` — one URL segment up — is correct, even though your fingers, walking the folder tree, want to type `(...)` or count two. Get this wrong and the intercepter silently never fires, which is the second-most-common bug in the whole pattern. Route groups behave identically: a `(marketing)` folder in the path is skipped for the same reason, for the same one-line rule — *no URL segment, no count*.

Hold that `(..)photo` in your head. It's the exact prefix we'll write next, and it'll show up identically in the file tree and the code.

## Wiring the modal

Time to assemble it. Almost every piece here is something you already have — the `@modal` slot and its `default.tsx` from parallel routes, the `[id]` segment and the `await params` reflex from dynamic routes, the shadcn `<Dialog>` from the accessibility chapter. The *one* genuinely new idea is the prefix, and you just learned it. Everything else is recombination.

Here's the complete file tree. Read each annotation against the dual-path picture from earlier — which file is the soft-nav render, which is the hard-nav render, which is the closed-modal fallback.

<FileTree>
- src/
  - app/
    - feed/
      - page.tsx the feed grid — the layout's `children` slot
      - layout.tsx receives the `@modal` slot as a prop, beside `children`
      - @modal/ the parallel slot the modal renders into
        - default.tsx `return null` — the closed-modal fallback on hard nav
        - (..)photo/ `(..)` — one *URL* segment up, because `@modal` adds none
          - [id]/
            - **page.tsx** the **intercepter** — renders on *soft* nav, wraps the detail in a modal
    - photo/
      - [id]/
        - **page.tsx** the **real** route — renders on *hard* nav, full standalone page
    - _components/
      - photo-detail.tsx the shared content — *both* pages import this
      - modal.tsx the dialog wrapper (Client Component)
</FileTree>

Five files do the work. Let me walk them in the order that builds understanding, starting with the one that carries the new idea.

The intercepter and the real page render the *same content* — the photo detail — and differ only in their wrapper. That's the trick that keeps the pattern honest: you write the photo-detail view once, as a shared component, and present it two ways. One path wraps it in a modal; the other gives it a full page. Compare them side by side.

<CodeVariants syncKey="intercepter-vs-real" maxLines={0}>

    import { notFound } from 'next/navigation';
    import { z } from 'zod';

    import { Modal } from '@/app/_components/modal';
    import { PhotoDetail } from '@/app/_components/photo-detail';

    export default async function PhotoModal({
      params,

      const parsed = paramsSchema.safeParse(await params);

      return (

        </Modal>
      );
    }
    ```

The wrapper is the *only* thing this file adds; the content is identical to the other tab.   </CodeVariant>

    import { z } from 'zod';

    import { PhotoDetail } from '@/app/_components/photo-detail';

    const paramsSchema = z.object({ id: z.uuid() });

    export default async function PhotoPage({
      params,
    }: PageProps<'/photo/[id]'>) {
      const parsed = paramsSchema.safeParse(await params);
      if (!parsed.success) notFound();

      return (
        <main className="mx-auto max-w-2xl p-6">
          <PhotoDetail id={parsed.data.id} />
        </main>
      );
    }
    ```

    </div>
    **Renders on hard navigation** — a full standalone page, no modal. Same `<PhotoDetail>`, just dropped into page chrome instead of a dialog. There's no feed to float over here, so the bare component *is* the page.
  </CodeVariant>
</CodeVariants>

Notice what's the same and what's different. Both pages capture `params`, validate the id, and bail with `notFound()` on a miss — that's the same discipline from the dynamic-segments lesson, and I won't re-walk it here. Both render `<PhotoDetail id={id} />`. The *only* difference is the wrapper: the intercepter wraps it in `<Modal>`, the real page sets it in page chrome. Write the detail once, present it twice. If you ever need to change what the photo view shows, you change it in `<PhotoDetail>` and both paths update together.

:::note
`<PhotoDetail>`'s data fetch is a thin placeholder here. The real data layer — `getPhoto`, queries, all of it — arrives later in the course, so treat the fetch as a stand-in for now.
:::

Next, the empty slot. Remember the pairing rule: on hard navigation, interception doesn't fire, so the `@modal` slot has no route to match. And you learned in the parallel-routes lesson what an unmatched slot does without a fallback — it 404s the *entire* route. The fix is the same `default.tsx` muscle you drilled there.

```tsx title="app/feed/@modal/default.tsx"
export default function Default() {
  return null;
}
```

`return null` means "closed modal — render nothing." On hard navigation, the real photo page renders standalone and the `@modal` slot quietly renders nothing, exactly as it should: there's no feed for a modal to float over, so there's no modal. This is the file beginners forget, and forgetting it is precisely the 404-on-refresh bug from earlier — three lines that the whole pattern leans on.

Last, the wrapper itself — and this is where the cleverest idea lives. Watch how the modal closes.

The focus trap, Esc-to-close, and return-focus all come from shadcn's `<Dialog>` — the accessibility chapter built that; here we just lean on it. Notice too what the wrapper *doesn't* do: it never imports `<PhotoDetail>`. A Client Component composes a Server Component by taking it as `children`, never by importing it directly — so the server-rendered detail is passed in by the intercepter page and renders untouched inside the dialog.

Sit with step 4, because it's the whole conceptual payoff. Opening the modal was navigation — the user clicked a `<Link>` to `/photo/42`. So closing it is navigation too: `router.back()`, which pops the URL back to `/feed`. And the moment you frame closing as navigation, every browser affordance comes along for free. The back button closes the modal, because back *is* what closes it. Forward reopens it. The URL stays truthful the entire time. You didn't wire any of that — you got it because you spent a real URL on the modal, and the browser already knows how to manage URLs.

That's the trade you've been making this whole lesson, stated plainly: the hand-rolled `useState` modal kept its open/closed state in a component, where the browser couldn't see it. This pattern keeps that state in the URL, where the browser does everything. Opening is navigation, closing is navigation, and the back button, refresh, sharing, and Cmd-click all just work.

## When intercepting routes earn their weight

This is a power tool, and power tools cost something. You're now maintaining two render paths, an extra route, and a parallel slot, for what is, on the surface, "a modal." So the senior question isn't "can I build a URL-backed modal" — you just did — it's "does *this* modal deserve one."

Reach for an intercepting-route modal when both of these are true: the UI is "show details or edit something in context, without leaving the list," **and** that detail view deserves a real URL — meaning it should be shareable, deep-linkable, refreshable, and survive back and forward. The photo feed is the textbook shape, but it's the SaaS surfaces where you'll actually reach for it most. An invoice row in a table that opens an invoice detail at `/invoices/42` — a teammate pastes that link and lands on the same invoice, modal and all. A message in an inbox that opens the full thread in context. An "edit settings" panel you can link a colleague straight to. Same pattern every time. The reflex worth building is small and durable: **when a piece of UI deserves a URL, give it one.**

Now the other half, which is just as much a senior call: most modals *don't* deserve a URL, and forcing this pattern onto them is over-engineering. A confirm-delete prompt — "Are you sure?" — is ephemeral and app-internal; nobody shares it, nobody deep-links it, and a refresh *should* dismiss it. A command palette, a form-validation popover, a transient menu — same story. Those stay a plain `useState` + `<Dialog>`, the exact pattern you opened the lesson with. That pattern isn't a beginner's shortcut you've now outgrown; for ephemeral, unshareable UI it is simply the correct answer. The skill is telling the two apart.

There's also a middle option worth naming so you recognize it when you meet it. Sometimes you want a modal to persist in the URL — survive a refresh, be shareable — but it isn't really a *detail page* of a list item, so a whole intercepted route is heavier than you need. For those, you can key a dialog off a query string instead — something like `/settings?modal=invite` — where a `searchParams` value, not a route, decides whether the dialog is open. It's lighter: no extra route, no slot, no intercepter. The course gets to `searchParams`-as-state in a later chapter; for now, just file it as the lighter cousin of today's pattern, for when you want URL persistence without a full route behind it.

    <Fragment slot="left">`(..)folder`</Fragment>
    <Fragment slot="right">Intercepts a route **one** URL level up</Fragment>
  </Pair>
  <Pair>
    <Fragment slot="left">`(..)(..)folder`</Fragment>
    <Fragment slot="right">Intercepts a route **two** URL levels up</Fragment>
  </Pair>
  <Pair>
    <Fragment slot="left">`(...)folder`</Fragment>
    <Fragment slot="right">Intercepts a route from the **root** `app` directory</Fragment>
  </Pair>
</Matching>

Finally, the highest-value trap in the lesson: counting URL segments, not folders, when a group is in the way.

<MultipleChoice>
  A dashboard nests everything under a `(dashboard)` route group. The settings page lives at `app/(dashboard)/settings/page.tsx`, and the members page at `app/(dashboard)/members/page.tsx` — so their live URLs are `/settings` and `/members`. You want clicking a row in settings to open the members view as a panel over settings, so you add a slot and put the intercepter at `app/(dashboard)/settings/@panel/<prefix>members/[id]/page.tsx`. Which prefix goes in `<prefix>`?

  <McqChoice correct>`(..)`</McqChoice>
  <McqChoice>`(...)`</McqChoice>
  <McqChoice>`(..)(..)`</McqChoice>
  <McqChoice>`(.)`</McqChoice>

  <McqWhy>Count URL segments, not folders. Walking up from the intercepter you cross `@panel` (a slot) and `(dashboard)` (a route group), and **neither contributes a URL segment** — so they're both invisible to the count. Strip them away and the live URLs tell the real story: `/settings` and `/members` are siblings, exactly **one** segment apart. That's `(..)`. The folder tree tempts you toward `(..)(..)` (two folders up) or `(...)` (root the count at the group) — but those count *disk*, and the disk count is the trap. `(.)` would mean members sits at the same level as the intercepter's own URL, which it doesn't.</McqWhy>
</MultipleChoice>

## What you can reach for now

You added one convention — the intercepting-route prefix — and turned a state-bound modal into a URL-backed view. The muscle memory to carry forward: the intercepter is a soft-navigation render override; it *always* ships alongside its non-intercepting sibling, the real page that handles hard navigation; it lives in a parallel slot, so the modal renders *beside* the page underneath, not instead of it; that slot needs an empty `default.tsx` so a hard load doesn't 404; and closing the modal is `router.back()`, not a state toggle, which is why back, forward, refresh, and sharing all work for free. The durable reflex underneath all of it — **URL-backed UI for anything worth sharing or deep-linking** — comes back in every list-and-detail surface you build from here on.

A few threads pick up later. The list-plus-detail project wires this exact pattern to real data and server-side mutations. The chapter on async UI adds `loading.tsx` and `error.tsx` at the slot boundary, so the modal can stream and recover. And the focus trapping, Esc-to-close, and return-focus the `<Modal>` leans on are all owned by the accessibility chapter you've already done — the `<Dialog>` was doing that work the whole time.
