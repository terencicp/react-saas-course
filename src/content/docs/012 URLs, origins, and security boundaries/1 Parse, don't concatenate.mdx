You learned backticks as the clean way to interpolate a string, and they are — except for one job. The moment a string is a URL, `` `${base}/users?id=${id}` `` stops being clean and becomes a latent bug. It looks fine in the editor and it works on the happy path, which is exactly why it survives review and breaks in production. The problem is the premise: **URLs look like strings, and they aren't.** A URL is a structured value with grammar and escaping rules, and the browser already ships a parser that knows all of them — the one the <Term definition="Web Hypertext Application Technology Working Group — the body that maintains the living URL, HTML, and Fetch standards browsers actually implement.">WHATWG</Term> URL standard defines. The senior move is to hand the string to that parser instead of treating the URL as a fill-in-the-blank template. This is the reflex behind every API URL you'll assemble from Units 3 through 7, so it's worth installing properly once.

By the end of this lesson you'll parse any URL into its fields, build one with `new URL()` and `URLSearchParams` with zero escaping bugs, and know which of three encoding tools belongs in which position. The `origin` field you'll name in a moment is the same one that becomes the unit of browser trust in the next lesson — so the vocabulary here is load-bearing for the rest of the chapter.

## A URL is six fields, not one string

Before you can hand a URL to the parser, you need the parser's vocabulary. Every URL decomposes into a fixed set of named fields, and once you can see them you stop seeing one opaque string. Here is the full anatomy on a single example.

<UrlAnatomy />

Each label in that diagram is a property you can read off a `URL` instance — `url.protocol`, `url.hostname`, `url.pathname`, and so on. Two of them carry a subtlety the diagram can't show. First, `url.searchParams` isn't a plain string like `url.search` — it's a **live `URLSearchParams` view** of the query, and mutating it rewrites `url.search` (and therefore `url.href`) underneath you. That live link is the hinge to the next section. Second, a URL technically has `username` and `password` fields too; this course ignores them on purpose. They're deprecated for navigation and shipping credentials inside a URL is a mistake, so treat the six fields above as the whole surface.

Parsing is the whole point, and it's one constructor call. Hand `new URL()` a string and read the fields straight off the instance.

```ts
const url = new URL('https://api.acme.com:8443/v1/invoices?status=paid#row-7');
url.hostname; // 'api.acme.com'
url.pathname; // '/v1/invoices'
url.search; // '?status=paid'
url.origin; // 'https://api.acme.com:8443'
```

No import made that work. `URL` is a global in every runtime this course ships on, which we'll come back to at the end — for now, just notice you reached for a structured value, not a regex.

## Building URLs with `new URL(input, base)`

Reading a URL is half the job; the other half is building one, and that's where concatenation does the most damage. The constructor takes a second argument for exactly this: `new URL(input, base)` resolves a relative path against a base URL. That's the canonical shape for assembling an API URL.

```ts
const base = 'https://api.acme.com'; // from a validated env var
const url = new URL('/v1/invoices', base);
url.href; // 'https://api.acme.com/v1/invoices'
```

In real code that `base` comes from a validated environment variable rather than a literal — you'll see that env story in a later unit — but the shape is the same: a relative path resolved against a base. Notice what you didn't do: you didn't worry about whether `base` ended in a slash, or whether the path started with one. The parser resolves the path against the base the way a browser resolves a link on a page, and it does it deterministically.

That determinism is the first concrete win, so it's worth seeing the failure it replaces. The classic concatenation bug is trailing-slash drift: the same logical URL assembled from two slightly different inputs produces two different strings, and now your cache, your logs, and your access checks see two URLs where you meant one.

<CodeVariants>

    </div>
    Both inputs land on the identical `href`. The parser normalizes the slashes for you — you stop thinking about them.
  </CodeVariant>
</CodeVariants>

While the parser is normalizing slashes, it normalizes two more things you'll lean on in the next lesson: it **lowercases the hostname** and **drops the default port** — `:80` for `http`, `:443` for `https` collapse to no port at all. This is exactly why origin comparisons in the next lesson are reliable: two URLs for the same server normalize to the same `origin` string instead of differing on capitalization or a redundant port.

There's a posture decision baked into the constructor: `new URL()` **throws** when the input can't be parsed. That's a feature, not a hazard, once you know where each kind of input comes from. A base URL almost always comes from configuration — an env var, a constant — so a malformed one is a *deploy* bug, not a user error. You want it to fail fast and loud at boot rather than wrapping every construction in a `try`/`catch` that swallows a misconfiguration into a vague 500 later. Let it throw.

The exception is untrusted input. When the string you're parsing came from a user — a query param, a pasted link, a redirect target — throwing is the wrong response to bad data you fully expected to receive. For that case, reach for `URL.canParse(input, base)`, the static boolean-returning guard.

<CodeTooltips tooltips={{
  canParse: 'Static method. Returns true/false instead of throwing — the guard for untrusted input.',
  URL: 'Global in every runtime — browser, Node, Edge. No import.',
}}>
```ts
const next = searchParams.get('next') ?? '';
if (URL.canParse(next)) {
  const target = new URL(next);
  // safe to use target
}
```
</CodeTooltips>

So the rule splits cleanly by trust: `new URL()` for config you control, and let it throw; `URL.canParse()` to guard before parsing anything a user could send. (`URL.parse()` exists too — same idea, but it returns `null` instead of a boolean. Reach for `canParse` when you're guarding and `parse` when you want the parsed result in one step; they're the same vintage and either is fine.)

## `URLSearchParams`: the end of escaping query strings

Here's the win, stated before the mechanics: you write `params.set('q', userInput)` and you never think about escaping a query value again. The entire pitch of `URLSearchParams` is that it owns the encoding so you don't. Every half-hour you'd otherwise spend debugging a `&` that ate the rest of a query string is a half-hour it gives back.

It builds from four kinds of input, and knowing which is which saves you from converting between them by hand. The same logical params, four ways.

<AnnotatedCode lang="ts" code={`
const fromUrl = url.searchParams;
const fromRecord = new URLSearchParams({ status: 'paid', limit: '20' });
const fromString = new URLSearchParams('status=paid&limit=20');
const fromEntries = new URLSearchParams([['tag', 'a'], ['tag', 'b']]);
`}>
  <AnnotatedStep meta="{1}" color="blue">
    The live view off a `URL` instance. This is the one you reach for most in app code: grab it, mutate it, and `url.href` updates with it. No separate object to keep in sync.
  </AnnotatedStep>

  <AnnotatedStep meta="{2}" color="blue">
    From a plain record. The quickest shape when every key has a single value. Note the values are strings — a query string has no notion of numbers.
  </AnnotatedStep>

  <AnnotatedStep meta="{3}" color="blue">
    From a raw query string, with or without the leading `?`. This is how you parse an incoming query — take the `search` off a URL someone handed you and read it structurally.
  </AnnotatedStep>

  <AnnotatedStep meta="{4}" color="blue">
    From an array of entries. The only shape that can express a repeated key, like `tag=a&tag=b`. A record can't — an object can't hold two `tag` keys.
  </AnnotatedStep>
</AnnotatedCode>

That last shape hints at something query strings do that records can't: a key can repeat. `?tag=a&tag=b` is a perfectly valid filter for "tagged a *or* b" — a list, not a scalar — and the API reflects that with two methods per operation. `append` adds a value, `set` replaces every value for a key; `get` returns the first value, `getAll` returns all of them. Iteration and insertion order are preserved, so the string comes back out in the order you built it.

```ts
const params = new URLSearchParams();
params.append('tag', 'a');
params.append('tag', 'b');
params.getAll('tag'); // ['a', 'b']
params.get('tag'); // 'a'  (just the first)
```

When you're done building, `params.toString()` hands you the finished query string — and it's correctly percent-encoded, every special character escaped to the right form. That sounds like the end of the story, but *how* it encodes is the one detail you have to understand, because it's not the same encoding you'd reach for elsewhere. That's next.

## The `%20`-vs-`+` split

Here's the surprise, and it's the one experienced developers get wrong too: the WHATWG URL standard uses **different** percent-encode sets for different positions in a URL. Your intuition says encoding is encoding — a space is a space. That intuition is the trap. The same character gets a different escape depending on where in the URL it sits.

It comes down to a clean binary. <Term definition="Replacing a character with % followed by its byte value in hex — %20 is a space. Also called URL-encoding.">Percent-encoding</Term> handles two contexts differently:

- **Path segments and the fragment** encode a space as `%20`. `encodeURIComponent` matches this set.
- **`application/x-www-form-urlencoded` query strings** — what `URLSearchParams` produces, and what an HTML form submission sends — encode a space as `+`. And because `+` *means space* on the way back, a literal plus sign has to be sent as `%2B`.

That second rule is where the bug lives. If you encode with one model and decode with the other, the round trip silently corrupts your data. The clearest way to feel it is a value that genuinely contains a plus — a search for `"a+b"` — pushed through both paths.

<CodeVariants>
  <CodeVariant label="Same model (URLSearchParams)">
    <div data-mark-color="green">

    ```ts {2}
    const encoded = new URLSearchParams({ q: 'a+b' }).toString(); // 'q=a%2Bb'
    const back = new URLSearchParams(encoded).get('q'); // 'a+b'
    ```

    </div>
    `URLSearchParams` escapes the literal plus to `%2B` on the way out and reads it back as a plus on the way in. Encode and decode with the *same* model and the value survives the round trip exactly.
  </CodeVariant>

  <CodeVariant label="Mixed models (the bug)">
    <div data-mark-color="red">

    ```ts {2}
    const encoded = new URLSearchParams({ q: 'a b' }).toString(); // 'q=a+b'
    const back = decodeURIComponent('a+b'); // 'a+b'  ← a space went in, a plus came out
    ```

    </div>
    The space became `+` on the wire, but `decodeURIComponent` reads `+` as a literal plus, not a space — so the value is corrupted. The two tools disagree about what `+` means, and the disagreement is invisible until a value contains one.
  </CodeVariant>
</CodeVariants>

The fix isn't to memorize which character maps to what. It's a posture.

:::tip
Encode and decode with the same model, end to end. The simplest version of that rule: do both with `URLSearchParams` and never hand-roll the query string. If you remember one sentence from this lesson, make it this one.
:::

## Which encoder, which position

The `%20`-vs-`+` split is the deep cut, but it sits inside a smaller, more common confusion: there are three encoding tools, each correct in exactly one place, and most bugs come from picking the wrong one. People reach for `encodeURIComponent` everywhere and double-encode values that `URLSearchParams` already handled, or reach for `encodeURI` thinking it escapes a value — it doesn't, it's for whole URLs. This table settles it.

| Tool | Escapes structural chars (`:/?#&=`)? | Use for |
| --- | --- | --- |
| `encodeURI` | **preserved** | a whole, already-trusted, well-formed URL string — rarely the right tool |
| `encodeURIComponent` | **escaped** | a single value going into a **path segment** |
| `URLSearchParams` | handled for you | **any** query-string construction |

The heuristic underneath the table is short enough to hold in your head: a value going into a path segment gets `encodeURIComponent`; anything in the query goes through `URLSearchParams`; whole-URL escaping is almost never what you want — reach for `new URL` instead. Lock it in while it's fresh.

To attach `?q=` search terms to a URL, build them with <DropdownChoice answer="URLSearchParams" options={['encodeURI', 'encodeURIComponent', 'URLSearchParams']} />. And to escape an entire URL string you already trust as well-formed — the rare case — you'd use <DropdownChoice answer="encodeURI" options={['encodeURI', 'encodeURIComponent', 'URLSearchParams']} />.

## The bugs concatenation hands you for free

Now the payoff. Every one of these is a bug that string concatenation produces silently and that `new URL` plus `URLSearchParams` retires — and they're worth seeing as a list, because the point isn't any single one, it's that you stop writing the whole category.

- **Parameter injection** — `` `?q=${q}` `` with `q = 'a&admin=true'` smuggles a second parameter into your URL; `params.set('q', q)` escapes the `&` to `%26`, so it stays part of the value. *(The most security-relevant — letting a user pick your query keys is a real attack surface.)*
- **Double-encoding** — running `encodeURIComponent` over a value `URLSearchParams` already escaped gives you `%2520` where you meant `%20` — the `%` itself got encoded.
- **Path-traversal-shaped values** — a raw `'../admin'` interpolated into a path is read as path *syntax* by the server, not as a literal segment.
- **Unicode hostnames** — the parser normalizes an <Term definition="Internationalized Domain Name — a hostname with non-ASCII characters, normalized to ASCII via Punycode.">IDN</Term> via <Term definition="ASCII encoding of Unicode hostnames, e.g. münchen becomes xn--mnchen-3ya.">Punycode</Term> (`münchen` becomes `xn--mnchen-3ya`); hand-templating ships a hostname the DNS layer can't resolve.
- **Trailing-slash drift** — the double-slash bug from the constructor section, now one entry in a longer pattern.

Read that list as five symptoms of one disease. None of them is a one-off you patch when it bites; each is a *category* of bug, and reaching for the parser retires the category, not the instance. That's the whole thesis of the lesson in one sentence: don't fix URL bugs, stop writing them.

## One reflex, every runtime

A practical relief worth stating plainly, because it removes a class of "where do I import this from" friction: `URL` and `URLSearchParams` are **globals in every runtime this course ships on** — the browser, the Node.js server, the Vercel Edge runtime. There is no import, ever. The same two lines work unchanged whether they run in a Server Component, a Route Handler, or a click handler in the browser.

And the vocabulary travels into React intact. Next.js's `useSearchParams()` hook hands you a **read-only `URLSearchParams`** instance — so `get`, `getAll`, and the multi-value model you just learned carry straight from URL strings into the component tree. You'll see the full treatment in Unit 4; for now just know it's the same object, not a new API to learn.

One more name for the forward file, no code attached: `URLPattern` is the standardized URL-matching primitive — a global now on browsers, Node, and Edge. You won't write it in this course, because Next.js's file-system router owns route matching, but it's worth recognizing when you see it: Unit 4's middleware uses pattern objects under the hood.

## Practice: build the URL

Time to operationalize the round trip. You're handed a base URL and a record, and your job is to return a fully-built, correctly-encoded URL string using `URLSearchParams` — no manual `?`, no hand-escaping. The grading hinges on the round trip: the test parses your result back out and checks that the `+` in `'a+b'` survived. If you build the query by hand with `encodeURIComponent`, the `+` decodes to a space and the test fails — which is the `%20`-vs-`+` lesson made executable.

The reference solution is short — try it before you open it.

<details>
<summary>Show a solution</summary>

```js
function buildSearchUrl(base, { q, tags }) {
  const url = new URL(base);
  url.searchParams.set('q', q);
  for (const tag of tags) {
    url.searchParams.append('tag', tag);
  }
  return url.href;
}
```

`set` for the single value, `append` in the loop for the repeated one, and `url.href` reads the finished string back out — encoding handled the whole way.

</details>
