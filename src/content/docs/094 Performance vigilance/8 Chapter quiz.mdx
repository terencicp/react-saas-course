    You ship a fix at 9am and want to know within the hour whether it improved performance. Why can't you rely on the Speed Insights field score to tell you?

    <McqChoice correct>The field score is a trailing 28-day p75 of real traffic, so a same-day change won't visibly move it for a week or two.</McqChoice>
    <McqChoice>Speed Insights only updates its dashboard once a day, so you'd see the change tomorrow morning.</McqChoice>
    <McqChoice>Field data is averaged across all users, so a single deploy is too small a sample to register.</McqChoice>

    <McqWhy>Field data is a rolling 28-day window — a stability signal, not a same-day alarm. That's exactly why regression detection has to happen *before* deploy with a lab tool like Lighthouse. (And the score is p75, not an average — that's a separate property.)</McqWhy>
  </MultipleChoice>

    A marketing page has a small header logo (first in the DOM), a full-bleed hero photo, and two product thumbnails — all above the fold. To improve LCP, which gets the `preload` prop?

    <McqChoice correct>Only the full-bleed hero photo — it's the largest painted element, and `preload` is a fixed budget you spend on one image.</McqChoice>
    <McqChoice>The header logo, because it's first in the DOM and the browser discovers it earliest.</McqChoice>
    <McqChoice>All three — they're above the fold, so front-loading every one makes the page paint sooner.</McqChoice>

    <McqWhy>The L in LCP is *largest*: the hero dwarfs the logo and thumbnails. Preloading all three splits the browser's high-priority budget so none lands as early as the hero alone would. One `preload` per page, on the element that defines LCP.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="94.3">
    You add `experimental.optimizePackageImports` to `next.config.ts`. Which packages actually belong in that list?

    <McqChoice correct>Libraries Next.js doesn't already optimize — most often your own internal packages — since lucide-react, date-fns and friends are on the default list already.</McqChoice>
    <McqChoice>Every multi-export dependency you import, including lucide-react and date-fns, to be safe.</McqChoice>
    <McqChoice>Only CommonJS packages like plain `lodash`, since those are the ones that can't be tree-shaken.</McqChoice>

    <McqWhy>Next.js maintains a default-optimized list (lucide, date-fns, recharts, the icon sets) that gets rewritten for you automatically. You only list the ones it doesn't cover — typically internal monorepo packages. And listing a CommonJS package like `lodash` won't help; the fix there is to switch to `lodash-es`.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="94.4">
    In the production bundle treemap, the single biggest tile is the React + Next runtime. What's the right move?

    <McqChoice correct>Leave it — the framework runtime is the floor every app pays, not an optimization target.</McqChoice>
    <McqChoice>It's the biggest tile, so it's the highest-value fix — trace its import chain and trim it.</McqChoice>
    <McqChoice>Code-split it with `dynamic()` so it only loads on routes that need it.</McqChoice>

    <McqWhy>Beginners see the largest rectangle and assume "biggest equals problem." The framework runtime is the baseline; there's no win there. The leads are *surprise* dependencies, heavy per-route chunks, duplicates, and a growing shared chunk — not the floor.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="94.5">
    Your Lighthouse report shows green LCP and CLS. Why does it never show you an INP value?

    <McqChoice correct>Lighthouse is a synthetic load with no user clicking anything, so there's no interaction to measure — it reports TBT as a partial lab proxy instead.</McqChoice>
    <McqChoice>Lighthouse measures INP but only surfaces it when it crosses the poor threshold, to reduce noise.</McqChoice>
    <McqChoice>INP only applies to authenticated pages, and Lighthouse audits the marketing page by default.</McqChoice>

    <McqWhy>INP is the latency of *real* interactions, and a lab run never clicks. Lighthouse reports Total Blocking Time as the partial proxy — a high TBT warns INP will likely be bad in the field — but there is no INP number in a Lighthouse report. For a real INP figure, go to Speed Insights or the DevTools Performance panel.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="94.6">
    You wrap two co-located Server Component awaits in `Promise.all`, but the second read actually needed the first's value. What happens?

    <McqChoice correct>No error — the second read runs with `undefined` and quietly produces wrong data.</McqChoice>
    <McqChoice>`Promise.all` throws immediately because it detects the missing dependency.</McqChoice>
    <McqChoice>The build fails, because Next.js validates await dependency order at compile time.</McqChoice>

    <McqWhy>This is the silent, dangerous failure: nothing throws, the page just renders on garbage. The dependency check — "does this read need the value I just awaited?" — is the only thing standing between you and that bug. If yes, keep it sequential.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="94.7">
    In a production trace, the invoice list is one fat 280ms span while the audit-log page is a staircase of fifty thin spans. Match each signature to its structural fix.

    <McqChoice correct>Fat span → add the composite `(organization_id, …)` index; staircase → collapse the N+1 with Drizzle `with` or a join.</McqChoice>
    <McqChoice>Fat span → collapse the N+1; staircase → add the missing index.</McqChoice>
    <McqChoice>Both → wrap the queries in `'use cache'` so warm requests skip the latency.</McqChoice>

    <McqWhy>One fat span means a single query doing too much work — a sequential scan a composite index removes. A staircase of thin spans means N+1: one query per row, collapsed into one statement with a relation or join. Caching only hides the cost — every cold request still pays it, and the slow shape is still there underneath.</McqWhy>
