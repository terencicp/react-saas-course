The product ships in two weeks. The team needs two things before that date arrives. First, a structural way to stop a performance regression from reaching production — something that fails a pull request before the slow code merges, not a dashboard someone notices three weeks later. Second, a one-off deep pass over the pages that actually matter — the landing page, the dashboard — so launch day doesn't ship with a four-second hero image nobody profiled.

Both of those are Lighthouse. But there's a tension to clear first, because the previous lessons of this chapter told you the opposite. [The Core Web Vitals](/094-performance-vigilance/1-the-core-web-vitals/) drew a hard line: field data is the verdict, Lighthouse is the regression catcher — chase your real-user numbers, don't chase a synthetic score. That's correct. After launch.

Before launch, there is no field data. The Core Web Vitals you see in Speed Insights come from <Term definition="Chrome User Experience Report — Google's aggregation of real Chrome users' performance data. It needs sufficient traffic to exist, so an unlaunched product has none.">CrUX</Term> — the Chrome User Experience Report, Google's aggregation of real Chrome users. CrUX needs traffic to exist, and an unlaunched product has none. So in the two weeks before ship, the synthetic lab run is not the inferior signal you'd ignore in favor of field data — it's the *only* signal you have. That resolves the apparent contradiction: Lighthouse demotes to the regression catcher once real users arrive, but until they do, it's the whole picture.

This lesson installs Lighthouse in those two postures: the **pre-launch deep audit** you run once, and the **CI regression gate** you run forever. By the end you'll have written a `lighthouserc.json` — the one config file that encodes the gate — and you'll have a threshold cheat sheet calibrated for a 2026 SaaS, so the numbers in that config aren't pulled from the air.

## What Lighthouse measures (and the one Vital it can't)

A Lighthouse run is a single page load under a fixed synthetic profile. It throttles the network to a simulated Slow 4G connection, throttles the CPU to mimic a mid-tier mobile phone, disables browser extensions, and starts from a cold cache. One run, on demand, reproducible. That is the exact opposite of field data's "real users, real devices, rolling 28-day window."

That fixed profile is *why* Lighthouse is the regression catcher. When a number moves between two runs, the network didn't get slower and the user's phone didn't change — the only thing that moved is your code. Field data can't tell you that; a Vital regression in CrUX could be a code change or just a bad week of mobile traffic. The lab controls every variable except the one you care about.

A run scores four categories: Performance, Accessibility, Best Practices, and SEO. (There used to be a fifth — PWA — but it was removed in Lighthouse 12. If a tutorial shows you five categories with a Progressive Web App score, it's stale, the same way a tutorial citing FID instead of INP is stale.)

The category you came for is Performance, and this is where the load-bearing correction lives. The Performance score is a weighted blend of five **lab metrics**:

</Figure>

Two of those five are minor and you can mostly set them aside: FCP (when the first pixel of content paints) and <Term definition="Speed Index — a lab metric measuring how quickly the page's content visually populates during load. Lower is better.">Speed Index</Term> (how quickly the page visually fills in during load) carry 10% each. The other three are where the lesson lives, and there are two things to take from them.

First, **LCP and CLS are right there.** These are the same two Vitals from [The Core Web Vitals](/094-performance-vigilance/1-the-core-web-vitals/), and between them they carry half the Performance score. That means the work you already know moves this number directly: the `next/image` `preload` from [Priority on the LCP element](/094-performance-vigilance/2-priority-on-the-lcp-element/) drives LCP, and the reserve-space discipline that keeps layout from jumping drives CLS. You are not learning a new set of fixes — you're learning a tool that scores the fixes you have.

Second, and this is the correction: that third Vital, INP, **is not on the bar.** Lighthouse cannot measure <Term definition="Interaction to Next Paint — a Core Web Vital measuring the latency of real user interactions. It is field-only: a lab run never clicks anything, so there is no INP to report.">INP</Term> — Interaction to Next Paint is the latency of real user interactions, and a lab run never clicks anything. There's no user, so there's no interaction, so there's no INP to report. What Lighthouse measures instead is <Term definition="Total Blocking Time — the sum of all main-thread blocking time over 50ms during page load. Lighthouse uses it as the partial lab proxy for INP.">TBT</Term> — Total Blocking Time, the total time the main thread was blocked for longer than 50 milliseconds during the page load. TBT is the *partial proxy* for INP: a page whose main thread is jammed loading will almost certainly be slow to respond to clicks too, so a high TBT is your pre-launch warning that INP will be bad in the field. But it's only a proxy. It captures the input-delay component of INP and nothing else.

The practical rule: **never quote an INP number from a Lighthouse report — there isn't one.** Read TBT, treat it as the early-warning signal, and remember the actual INP fix from the first lesson was *ship less client JavaScript*. TBT is how you catch that whole class of problem before you have a single real INP measurement. When you finally want a true INP number, you go to Speed Insights or the DevTools Performance panel, not Lighthouse.

:::note
The boundary, restated so it sticks. **Lighthouse** is synthetic, runs pre-deploy, and is your regression catcher plus pre-launch deep dive. **Speed Insights** is real users and the production verdict. When the two disagree after launch: field data wins for deciding *what to fix*, lab data wins for *catching regressions* before they ship.

## Three ways to run Lighthouse, and when each earns it

There are three surfaces that run Lighthouse, and they aren't interchangeable. Before any mechanics, get the decision straight — the question an experienced engineer asks isn't "what are the options," it's "what am I trying to do right now."

The first reach is the **DevTools Lighthouse panel.** The trigger: a quick, ad-hoc check while you're developing, on the exact page you're staring at. It's one click and it runs locally. Two caveats, though, and they bite people constantly. Run it against a **production build** — `pnpm build` then `pnpm start` — never `pnpm dev`. The dev bundle is unminified, unoptimized, and full of development-only code; every metric off `pnpm dev` is meaningless. And even on a production build, `localhost` over the loopback interface has near-zero network latency, so the numbers read artificially fast. Keep throttling on and treat the absolute values with suspicion; the panel is best for *relative* checks ("did this change make it worse?").

The second reach is **PageSpeed Insights** — Google's hosted Lighthouse. The trigger: the pre-launch audit of your **marketing page**. It earns that spot for one reason — it runs Lighthouse on Google's own infrastructure *and* overlays CrUX field data in the same view, tying lab and field together. There's a pre-launch nuance here that matters: for a URL with no traffic yet, the field section simply doesn't appear. You'll see a lab-only report. That's not a bug — it's the correct, expected pre-launch state, and it's the concrete face of "there is no field data yet." After launch, that section fills in and becomes the verdict.

The third reach is **`@lhci/cli`** — Lighthouse CI. The trigger: the automated regression gate. It runs Lighthouse against a built copy of your app inside CI, asserts your thresholds, and fails the build when they're missed. This is the course default for the gate and the spine of the next two sections.

The following walker encodes that decision — the order to ask the questions in, not a flat feature list. Start from what you're trying to do and let it route you to the surface.

## Two surfaces and the threshold cheat sheet

Before wiring the gate, you need the targets it asserts against — otherwise the numbers in the config are arbitrary. So: what do you aim at, and on which pages?

You do not audit every route. You pick two, because a SaaS has two performance regimes and two pages cover both. The first is the **marketing or landing page** — your highest-traffic surface, the SEO-sensitive first impression, and mostly static. The second is **one critical authenticated screen** — the dashboard home or the primary task screen, whichever ships the most JavaScript. That second page is the realistic interactive worst case: it's behind login, it's heavier, and it's where bundle bloat shows up. The rest of the app's pages are variations on one of these two; auditing both covers the static-marketing regime and the JS-heavy-app regime, and that's enough.

Here are the starting targets. Treat them as 2026 SaaS defaults you tighten quarterly, not laws.

The reasoning behind those numbers is worth spelling out, because every one of them connects to something you already know.

**LCP and CLS match the *good* Vital bands** from the first lesson — LCP ≤ 2.5 s, CLS ≤ 0.1. The lab target equals the field target here; you want the synthetic run to clear the same bar real users will be scored against.

**TBT replaces what you might have expected to be an INP budget.** There is no lab INP, so there's no INP number to assert. You budget TBT instead — the proxy — and the dashboard gets a looser 300 ms because an interactive screen legitimately runs more JavaScript on load than a static landing page.

**The JS budgets are the line straight back to the treemap.** [Reading the bundle treemap](/094-performance-vigilance/4-reading-the-bundle-treemap/) taught you to find where the bytes went; this is where you cap them. And notice the dashboard's larger JS budget *is the reason* it scores lower on Performance — more JavaScript means more main-thread work, which means higher TBT, which means a lower score. The looser targets aren't a double standard; they're the honest cost of an interactive surface.

One thing the cheat sheet deliberately does *not* say is "100 everywhere." Chasing a perfect 100 is diminishing returns. The jump from 90 to 100 often involves micro-optimizations that don't change a thing for a user on a flaky mobile connection — and that user, not the engineer on fiber, is who the score is supposed to protect. The cheat sheet encodes "good enough that you don't lose users," not a vanity number.

Before you wire any of this up, make sure the *surface class* — not the page's name — is what determines which budget applies. Sort each of these pages into the audit profile it belongs to.

## Wiring the CI gate with `@lhci/cli`

This is the recurring posture — the floor that holds as the app grows. The whole gate lives in one config file you should be able to write yourself after this section.

Install it as a dev dependency:

```bash
pnpm add -D @lhci/cli
```

The current `@lhci/cli` is the 0.15.x line, which bundles Lighthouse 12 — the version whose Performance score you just learned. It runs fine on Node 24, the course default. (Don't memorize the version literal; LHCI's versioning moves, and you'll get whatever's current.)

Everything else is one file: `lighthouserc.json`. It has three load-bearing parts — what to audit, what to assert, and where to put the report. Walk through it pane by pane.

<AnnotatedCode lang="json" maxLines={18} code={`
{
  "ci": {
    "collect": {
      "url": ["http://localhost:3000/", "http://localhost:3000/dashboard"],
      "startServerCommand": "pnpm start",
      "numberOfRuns": 3
    },
    "assert": {
      "preset": "lighthouse:recommended",
      "assertions": {
        "categories:performance": ["error", { "minScore": 0.9 }],
        "largest-contentful-paint": ["error", { "maxNumericValue": 2500 }],
        "total-blocking-time": ["error", { "maxNumericValue": 300 }],
        "cumulative-layout-shift": ["error", { "maxNumericValue": 0.1 }],
        "resource-summary:script:size": ["error", { "maxNumericValue": 358400 }]
      }
    },
    "upload": {
      "target": "temporary-public-storage"
    }
  }
}
`}>
  <AnnotatedStep meta="{3-7}" color="blue">
    The `collect` block: the two surfaces from the cheat sheet — the marketing root and the dashboard — and `startServerCommand` so LHCI boots your production app itself with `pnpm start`. It expects the app to have been built first.
  </AnnotatedStep>

  <AnnotatedStep meta="{6}" color="blue">
    `numberOfRuns: 3`. A single Lighthouse run is noisy — one slow garbage-collection pause can swing a metric. LHCI runs three times per URL and takes the median, smoothing run-to-run jitter into a number you can assert against.
  </AnnotatedStep>

  <AnnotatedStep meta="{8-9}" color="green">
    The `assert` block — the heart of the gate. `preset: "lighthouse:recommended"` pulls in Lighthouse's full recommended assertion set as a baseline, so you're checking dozens of audits for free.
  </AnnotatedStep>

  <AnnotatedStep meta="{11-14}" color="green">
    The overrides that tie the gate to your cheat sheet. `categories:performance` `minScore: 0.9` is the ≥ 90 score (scores are 0–1 here, so 0.9 = 90). Then the per-metric budgets: LCP and TBT in **milliseconds**, CLS unitless. These are the lines that fail the build when a budget busts.
  </AnnotatedStep>

  <AnnotatedStep meta="{15}" color="green">
    The structural resource budget. `resource-summary:script:size` caps shipped JavaScript — and the value is in **bytes**, so the 350 KB from the cheat sheet becomes 358400. Bust the JS budget and the pull request fails mechanically, no human judgment required.
  </AnnotatedStep>

  <AnnotatedStep meta="{18-20}" color="blue">
    `upload` with `target: "temporary-public-storage"`. Zero setup, and you get a shareable hosted report URL for every run. The heavier alternative is a self-hosted LHCI server for historical trends — name it, don't build it.
  </AnnotatedStep>
</AnnotatedCode>

Two ideas in that config carry the lesson's weight.

The first: **assert the metric budget, not the aggregate score.** The discipline of the gate is failing when LCP busts 2.5 seconds or the JS budget blows past its cap — *not* when the overall score drifts from 95 to 93. The score is diagnostic; it tells a human where to look. The assertions are the structural protection; they fail the build mechanically. This is the exact through-line the whole chapter runs: structural protection over vanity metric, the same shape as the ESLint ban on raw `<img>` in [Priority on the LCP element](/094-performance-vigilance/2-priority-on-the-lcp-element/) and `sideEffects: false` in [The barrel-export trap](/094-performance-vigilance/3-the-barrel-export-trap/).

The second: that `resource-summary` line is a **performance budget**, and budgets are the structural form of this idea. Beyond JS size, LHCI lets you cap image bytes, font bytes, total page weight, and timing budgets like LCP and FCP. The advice is one budget per resource class — and the JS budget is the load-bearing one for a SaaS, because JavaScript is what drives TBT, which drives the score.

One honest simplification: this config applies one assertion set to *both* URLs, so the marketing page gets held to the dashboard's looser 350 KB budget rather than its own 200 KB. That's fine as a starting gate — it still catches a runaway dependency. When you want a tighter per-surface budget, LHCI's `assertMatrix` lets you key different assertions to different URL patterns; it's a small extension once the basic gate is in place.

To run it, one script line in `package.json`:

```json title="package.json"
"scripts": {
  "lhci": "lhci autorun"
}
```

<Term definition="The LHCI command that chains the whole gate: healthcheck, then collect, assert, and upload, exiting non-zero if any assertion fails.">autorun</Term> is the one command that does everything — `lhci autorun` chains a healthcheck, then runs collect, assert, and upload in sequence, exiting non-zero if any assertion fails. That non-zero exit is what a CI system reads as "this build failed."

Which raises the obvious question: where does this run? The conceptual flow on a pull request that touches UI or dependencies is: build the app, start it, run `lhci autorun`, and let CI fail if any assertion does. But the thing that *orchestrates* that — the workflow file, the triggers that decide which PRs run it, the runner, dependency caching, secrets, and a logged-in test user for the dashboard route — is GitHub Actions, and you haven't met GitHub Actions yet.

:::note
GitHub Actions is taught in a later chapter, not here — the workflow/job/step model, triggers, caching, and required status checks. This lesson owns the `lighthouserc.json` and the flow it encodes; the YAML that runs it on every pull request comes later. That's why you won't see a `.github/workflows` file in this lesson — its absence is deliberate, not an omission.
:::

One more real requirement to name and defer. Your dashboard surface lives behind login, and a fresh Lighthouse run arrives with no session — so it'd be redirected to the sign-in page and audit *that* instead. Two reaches solve it: a `puppeteerScript` that logs in before the audit, or injecting a session cookie. Both depend on your auth stack and on CI secret plumbing, so the implementation waits for the GitHub Actions chapter. For now, know it's a requirement, not a detail you can skip.

Start with the **Performance score**, but only to ignore it. It's the weighted aggregate of the five metrics; its diagnostic value is entirely in the breakdown beneath it. Restating the discipline one more time: don't chase the number, find the red metric.

Drop to the **metrics strip** to see *which* metric is red. This matters because the metric tells you the fix. A red LCP sends you to the image work in [Priority on the LCP element](/094-performance-vigilance/2-priority-on-the-lcp-element/); a red TBT sends you to the client-JavaScript work in the first lesson and the bundle triage in [Reading the bundle treemap](/094-performance-vigilance/4-reading-the-bundle-treemap/). Same red color, opposite fixes.

Then **Opportunities** — and act here first. These are ranked by estimated time savings, and almost every one maps to a fix this chapter already taught. Oversized-image opportunities are [Priority on the LCP element](/094-performance-vigilance/2-priority-on-the-lcp-element/) — `next/image` with `preload` and correct sizing. Unused-JavaScript and bundle opportunities are [The barrel-export trap](/094-performance-vigilance/3-the-barrel-export-trap/) (`optimizePackageImports`) plus [Reading the bundle treemap](/094-performance-vigilance/4-reading-the-bundle-treemap/) for the treemap triage. Render-blocking resources point at font and CSS handling.

Last, **Diagnostics** — structural issues with no time estimate attached: excessive DOM size, long main-thread tasks, console errors. Long main-thread tasks here are the TBT story again, pointing at client JavaScript.

Here's the cross-tool routing that ties it all together. A red metric in Lighthouse is the *symptom*. The tool that gives you the *diagnosis* depends on the metric: bundle weight goes to the treemap from [Reading the bundle treemap](/094-performance-vigilance/4-reading-the-bundle-treemap/); a slow render time goes to the RSC waterfall view in a Sentry trace (the next lesson); a database-bound slow TTFB goes to the query work in the lesson after that. Lighthouse tells you *that* the page is slow and roughly *where*; the chapter's other tools tell you *why*.

Finally, a word on the three categories you didn't come for. Lighthouse surfaces accessibility, SEO, and best-practices gaps almost for free — missing alt text, unlabeled form fields, low contrast, missing landmarks (accessibility); missing meta tags (SEO); mixed content or insecure requests (best practices). None of these are taught at depth here — accessibility and SEO each have their own lessons elsewhere — but the audit *flags* them, and fixing each one is the relevant lesson's job. One thing worth internalizing now, though, because it's the trap teams fall into:

:::caution
Don't dismiss the accessibility findings because no users have complained. Users with accessibility needs don't file complaints — they leave, silently, and you never learn why. The flag is free; ignoring it isn't.
:::

## Pre-launch deep pass vs. the recurring gate

Step back from the tool. What this chapter is really teaching is a *cadence*, and Lighthouse runs in two of them. Getting the cadence right is the whole point — the same tool, used two different ways, solving two different problems.

The **pre-launch deep pass** is the one-off. Before ship, run PageSpeed Insights against the marketing page, the dashboard, and two or three other critical screens — signup, the primary task flow, settings. Triage every finding through the chapter's fix map: bundle bloat to the treemap, image problems to `next/image`, slow renders to the RSC waterfall, slow database queries to the query work. Ship the fixes, re-audit, and repeat until every surface clears the cheat-sheet targets. This is a deep dive you do *once* — and remember the field section of every report will be empty, because there's no traffic yet. Lab is the whole picture here.

The **recurring CI gate** is the floor. `@lhci/cli` on every pull request that touches UI or dependencies, against the two surfaces, with the budget assertions. It can't make your app fast — it has no opinions about your code — it can only stop the app from *silently regressing* past the budget as it grows. Pair it with the post-launch verdict from Speed Insights and the chapter's other recurring checks, like the weekly slow-query review in the database lesson.

There's a third move that only becomes available after launch: **calibration.** Once real field data exists, recalibrate the CI budgets against your *historical field data*, not vanity scores. If the field consistently beats a lab budget, you've earned the right to tighten it. If the field is worse than the lab predicted, your lab profile is too generous and you should trust the field for prioritization — which is the first lesson's rule, finally actionable.

    <CadenceCard posture="recurring" />

That two-cadence picture is the node this lesson occupies in the chapter: the all-three-Vitals pre-launch gate, sitting between the per-metric fix lessons before it and the server-side diagnosis lessons after. The first lesson framed performance vigilance as a pre-launch deep pass plus recurring guards; this lesson is where that framing gets its tooling.

<MultipleChoice>
  Your team merges a pull request that adds a charting library to the dashboard. Two months later, that library has quietly pushed the dashboard's bundle past its budget. Which Lighthouse posture was supposed to catch this *at merge time*?

  <McqChoice>The pre-launch PageSpeed Insights deep pass</McqChoice>
  <McqChoice correct>The `@lhci/cli` CI gate's JS resource budget, which fails the PR when shipped JavaScript busts its cap</McqChoice>
  <McqChoice>Speed Insights field data, once the 28-day window fills in</McqChoice>
  <McqChoice>A DevTools Lighthouse run on localhost</McqChoice>

  <McqWhy>The deep pass already happened before launch and is a snapshot, not a guard. Speed Insights would eventually show the regression in the field, but slowly and only after users felt it — and it can't block a merge. The DevTools run is ad-hoc and local. Only the CI gate runs on the pull request itself and fails it mechanically when the JS budget busts — that's the whole reason the budget is structural rather than a number someone eyeballs.</McqWhy>
</MultipleChoice>
