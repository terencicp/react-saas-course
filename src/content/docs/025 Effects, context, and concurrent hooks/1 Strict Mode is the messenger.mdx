A bug report lands. A user opened a page, clicked away to another tab in your app, came back, and now the screen shows numbers from thirty seconds ago. Or they got two "Saved!" toasts instead of one. Or your network panel — once you reproduce it — shows the same request firing twice on every visit. Nothing in the code looks wrong. It worked on your machine. It worked in review. It works in production *most* of the time, which is the worst kind of works.

The root cause, almost always, is the same: a component reached out to something outside React — an interval, a network request, an event listener — and never tore it down when it went away. The old one keeps running. A new one stacks on top. The leak compounds every time the user navigates back.

Here is the part that should sting a little. React tried to tell you. The first time you loaded that page in development, React ran that component through a full mount, unmount, and mount again — *on purpose* — precisely so a missing teardown would misbehave loudly, right there on your screen, weeks before any user saw it. That machinery is called <Term definition="React's dev-only wrapper that double-runs pure code and cycles effects through mount, unmount, and mount to surface impurity and missing cleanups before production.">Strict Mode</Term>, and if you've ever been confused by "why is my effect running twice?", you've already met it — you just read it as a nuisance instead of a free correctness test.

This lesson flips that reading. By the end, when you see something run twice in dev, you won't reach for a switch to make it stop. You'll read it as a signal — *this code isn't safe to run twice, so fix it* — and you'll know the exact shape of the fix. You already hold the contract Strict Mode is checking: back in [The purity contract](/023-the-render-model/3-the-purity-contract/), you learned that rendering is a pure function of props and state. Strict Mode is one of the ways React *verifies* you kept that promise. It also previews a second discipline — cleanup — that you'll learn to write properly in the next lesson. For now, the goal is narrower and more valuable: learn to trust the messenger.

## You already have it on

Strict Mode is a component. You wrap part of your tree in it, and everything inside opts into a set of development-only checks.

```tsx title="src/index.tsx"
<StrictMode>
  <App />
</StrictMode>
```

It wraps a *subtree*, so in theory you could scope it to one risky corner of your app. In practice nobody does — you wrap the whole thing once at the root and forget about it.

Except you don't even write that. If you scaffolded your project with Next.js — which every project in this course does — Strict Mode is already on. Next.js wraps your application in it by default, and has since version 13.5.1, so your Next.js 16 app has been running every Client Component through these checks since the day you created it. There's a `reactStrictMode` flag in `next.config` you *could* flip off, but it defaults on, and turning it off to silence a warning is a smell we'll come back to at the end of the lesson.

<Aside type="note">
You did not opt into this. You inherited it. Every doubled render and twice-firing effect you've seen in dev was Strict Mode doing its job — and was always trying to tell you something about your code.
</Aside>

The reason you can leave it on without a second thought is the guarantee that makes the whole feature work: **every one of these checks is development-only and stripped from your production build.** No double renders ship. No extra effect cycles run for real users. There is zero runtime cost in production, which means there is no performance argument for disabling it — the thing it would "cost" you doesn't exist where it would matter. Leaving it on is free. Turning it off only buys you the right to ship the bug it would have caught.

## What runs twice, and why

Strict Mode doesn't run *everything* twice. It runs two specific categories of code twice, and they exist for two different reasons. Holding the two reasons apart is the whole game — it's what tells you which kind of problem a doubling is pointing at.

The first category is **things that are supposed to be pure.** That's your component function body itself; the functions you hand to `useState`, `useReducer`, and `useMemo` as initializers — including the lazy initializer you met in [The useState surface and lazy initialization](/024-hooks-for-holding-state/1-the-usestate-surface-and-lazy-initialization/), the `useState(() => …)` form; and the updater functions you pass to a setter, like `setCount((c) => c + 1)`. Strict Mode runs each of these twice in dev. The logic is airtight: if a function is genuinely pure, running it a second time produces the exact same result as the first, so the doubling is *invisible*. You'd never know. The only time you notice is when the function isn't pure — when it secretly changes something in the outside world — and then the second run does that something a second time, and the damage shows. The doubling doesn't break pure code. It exposes code that was never pure to begin with.

The second category is **effects**, and here the goal is different. Strict Mode isn't checking purity — it's checking *cleanup*. On the first mount in dev, instead of just running an effect's setup once, React runs the full lifecycle: setup, then cleanup, then setup again. We'll unpack exactly what that does in the next section, because it's the heart of the lesson. React 19 extends the same treatment to <Term definition="A function passed to ref={} that React calls with the DOM node on mount and null on unmount.">callback refs</Term> — the function-form `ref` you saw in [useRef as the non-rendering escape hatch](/024-hooks-for-holding-state/5-useref-as-the-non-rendering-escape-hatch/) gets an extra call-and-cleanup cycle too. Same idea: anything that *sets something up* gets run through a teardown to prove the teardown exists and works.

Now the line that resolves the most common confusion juniors have. Strict Mode does **not** double-invoke event handlers. It does **not** double-invoke `setTimeout` or `setInterval` callbacks. A click that runs an `onClick` once still runs it once. Only render-time code and the effect-and-ref lifecycle get doubled. This is exactly why a click handler that fires a request once behaves perfectly while an effect that fires the same request runs it twice — the handler was never in the double-invocation set. If you've ever wondered why one doubles and the other doesn't, that's the bright line.

## The mount, unmount, mount cycle

This is the section that makes the next lesson feel necessary, so slow down here.

You'll meet the full effect API in the next lesson. For now, treat an effect as a black box with two parts. It has a **setup** — code that runs after the component appears on screen — and an optional <Term definition="The function an effect returns. React runs it before the next setup and on unmount, to tear down whatever the setup created.">cleanup</Term> — a function the setup returns, which React runs on the way out to undo whatever the setup did. You write `useEffect(setup, [])` and think of it as: *setup runs once when I show up, cleanup runs once when I leave.* That's the entire mental model you need today. Dependency arrays, the signature, the way effects re-synchronize when inputs change — all of that is the next lesson's job.

Here's the twist Strict Mode adds. In development, on the very first mount, React doesn't just run setup. It runs **setup, then cleanup, then setup** — a complete mount, unmount, remount, compressed into the initial render. It's simulating a user who shows up, leaves, and comes back, all in a fraction of a second, before you've even interacted with the page.

Now watch what that does to an effect that subscribes to something but forgets to unsubscribe. Say the setup starts an interval that logs a tick every second, and you didn't write a cleanup. Scrub through what happens.

<StrictModeIntervalLeak />

The cleanup isn't decoration. It's the thing that makes the second setup safe. Run the same effect with a cleanup that clears the interval, and the cleanup step actually fires — it clears timer #1 *before* setup #2 ever runs, so you're left with exactly one live timer no matter how many times the component mounts. Here are the two shapes side by side.

<CodeVariants>
  <CodeVariant label="Leaks">
    <div data-mark-color="red">

    ```tsx {2}
    useEffect(() => {
      const id = setInterval(tick, 1000);
    }, []);
    ```

    </div>
    **Leaks — no cleanup.** Strict Mode's second setup stacks a second interval on top of the first, which was never cleared; the tick fires twice.
  </CodeVariant>

  <CodeVariant label="Clean">
    <div data-mark-color="green">

    ```tsx {3}
    useEffect(() => {
      const id = setInterval(tick, 1000);
      return () => clearInterval(id);
    }, []);
    ```

    </div>
    **Clean — returns a cleanup.** Cleanup clears the first interval before the second setup runs, so exactly one timer is ever live — it survives any number of mounts.
  </CodeVariant>
</CodeVariants>

An interval is the easiest leak to *see*, but it's one of three you'll meet constantly, and they all share the same shape — a setup that reaches outside React, and a missing cleanup that should have reached back. Keep these three as your mental checklist for "what kind of leak is this":

- **An interval or timeout that's never cleared** — it fires twice (you just saw it).
- **A network request that's never aborted** — two requests go out in parallel, and whichever resolves last wins, which is how you get stale data on screen.
- **An event listener that's never removed** — the handler runs twice for every single event, because two copies are attached.

The next lesson codes all three properly. Today the point is recognition: when you see double, name which of these three it is, then write the cleanup. You don't yet need to know the precise API to know the *shape* of the answer.

## Why running twice catches impurity

That's the cleanup half of Strict Mode. The other half — running pure code twice — catches a different class of bug, and it leans directly on the purity contract you already hold, so this is shorter.

The mechanism is the same logic from the effect side, pointed at render. If your render or your initializer does something *observable to the outside world* — pushes onto an array that lives at module scope, bumps a counter, writes to `localStorage`, pings an analytics endpoint — then running it twice does that thing twice. The array has duplicate entries. The counter reads `2` when you expected `1`. The stored value is corrupted. Strict Mode didn't *cause* any of that. It revealed an impurity that was always there, lurking, ready to misbehave the moment React re-rendered the component for any reason — and React re-renders components constantly. Strict Mode just guarantees the misbehavior shows up now, on your screen, instead of intermittently, in production, when some unrelated state change happens to trigger a re-render.

The sharpest version of this is the lazy initializer you just learned, so let's name it directly. `useState(() => expensiveCompute())` runs that initializer twice in dev. If `expensiveCompute` is pure — it reads its inputs, does math, returns a value — twice is identical to once and you never notice. But if it has a side effect tucked inside, a `localStorage` write or an analytics event or a counter bump, that side effect now fires twice. The same is true of the `init` argument you pass to `useReducer`. And note the fix is *not* "stop using lazy initialization" — lazy init is correct and you should keep it. The fix is "keep the initializer pure." A side effect doesn't belong in an initializer at all; it belongs in an effect or an event handler, where it runs when it's supposed to.

The exercise below makes the doubling something you *feel* instead of something you're told. Read the program, predict what it prints, then check yourself.

The same rule covers a side effect sitting directly in the render body — an analytics call written inline in the component, say. It fires twice in dev because the body runs twice, and the fix is the one you'd reach for anyway: move it out of render into an effect or an event handler, because **render must be pure.** (There's a whole catalog of "this looks like it needs an effect but doesn't" coming a few lessons from now; for today the point is just that render is not the place for side effects.)

## Don't fight the messenger

Everything so far has been setup for this section, because this is the one place juniors reliably make the wrong move, and the move *feels* clever, which is what makes it dangerous.

Here's the scene. You see your effect run twice. You don't yet have the cleanup reflex, so you reason: "I only want this to happen once. Let me guard it." You reach for a ref to remember whether you've already run, and you bail out the second time. The guard lines below are the ones to distrust.

<div data-mark-color="red">

```tsx {1, 3-4}
const didRun = useRef(false);
useEffect(() => {
  if (didRun.current) return;
  didRun.current = true;
  const id = setInterval(tick, 1000);
}, []);
```

</div>

It works. The double-firing stops. And it is exactly wrong.

<Aside type="danger" title="This hides the leak, it doesn't fix it">
The guard suppresses the *second* setup. The *first* setup's interval is still never cleared — there's still no cleanup. You took a bug Strict Mode was showing you and made Strict Mode stop showing it to you, while the leak stayed put. You didn't fix anything; you blindfolded the smoke detector.
</Aside>

And in 2026 this is worse than just hiding a dev-time signal, because the thing Strict Mode is simulating is no longer hypothetical. React 19's concurrent features genuinely mount, unmount, and remount components *in production* — not as a dev simulation, for real users. Transitions and prefetching do it. The <Term definition="A React 19 feature, <Activity>, that hides a subtree and later restores it — cleaning up its effects when hidden and re-running them when shown again.">Activity API</Term> does it explicitly: `<Activity>` hides a piece of UI and later brings it back, running cleanups when it hides and re-running setups when it returns. A guard built on the assumption that "setup runs exactly once, forever" is simply false against how production already behaves. Strict Mode was never a dev fiction inventing a scenario that can't happen — it's a *preview* of the mount-unmount-remount that production does on its own.

So here's the rule the whole lesson has been driving at, the one sentence to carry out of here:

> **Write cleanups that make the second mount safe — never guards that try to prevent it.**

The correct response to "this runs twice" is always the same: add or fix the cleanup. Once the cleanup is right, the double-mount is *harmless* — which is the precise thing Strict Mode is checking, and the precise thing production needs from your component. You're not making Strict Mode happy. You're making your component correct, and Strict Mode is how you find out you succeeded.

(`<Activity>` is something you'll meet properly when we reach the App Router. For now you only need the one fact: production remounts components on its own, so your cleanups have to handle it.)

Time to do the fix yourself instead of reading it. The component below leaks — its effect starts something and never cleans it up. Fix it so it survives being remounted.

<ReactCoding
  hidePreview
  instructions="This effect subscribes to a 'tick' event on window but never unsubscribes, so a remount stacks a second listener and the count jumps by two per tick. Return a cleanup from the effect that removes the listener — fix the shape, don't add a one-time guard."
  starter={`import { useEffect, useState } from 'react';

export function App() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const onTick = () => setCount((c) => c + 1);
    window.addEventListener('tick', onTick);
    // The setup added a listener. Return a cleanup that removes it,
    // so a remount tears the old one down instead of stacking a new one.
  }, []);

  return (
    <p>
      Ticks: <span data-testid="count">{count}</span>
    </p>
  );
}`}
  tests={`
// act() needs this flag to flush passive effects deterministically; setting it
// lets every mount run the effect's setup and every unmount run its cleanup.
(globalThis as any).IS_REACT_ACT_ENVIRONMENT = true;

// Mount App into our own root with act() so the effect's setup runs and its
// cleanup runs on unmount — then we can replay Strict Mode's mount/unmount/mount
// for real and watch what the effect does to window's 'tick' listeners.
const fireTick = async () => {
  await act(async () => {
    window.dispatchEvent(new Event('tick'));
  });
};

const readCount = () =>
  document.querySelector('[data-testid="count"]')?.textContent;

test('a tick bumps the count once while mounted', async () => {
  const host = document.createElement('div');
  document.body.appendChild(host);
  const root = createRoot(host);
  try {
    await act(async () => {
      root.render(<App />);
    });
    expect(readCount()).toBe('0');
    await fireTick();
    expect(readCount()).toBe('1');
  } finally {
    await act(async () => {
      root.unmount();
    });
    host.remove();
  }
});

test('after a remount exactly one tick listener is live, so a tick bumps by one not two', async () => {
  // Count how many 'tick' listeners are attached to window: every add is +1,
  // every matching remove is -1. A correct cleanup keeps this at 1 across a
  // remount; a leak (or a useRef guard) leaves it at 2.
  let liveTickListeners = 0;
  const realAdd = window.addEventListener.bind(window);
  const realRemove = window.removeEventListener.bind(window);
  window.addEventListener = (type, handler, opts) => {
    if (type === 'tick') liveTickListeners += 1;
    return realAdd(type, handler, opts);
  };
  window.removeEventListener = (type, handler, opts) => {
    if (type === 'tick') liveTickListeners -= 1;
    return realRemove(type, handler, opts);
  };

  const host = document.createElement('div');
  document.body.appendChild(host);
  // Mount, unmount, then mount into a fresh root — exactly the mount/unmount/
  // mount cycle Strict Mode forces. The first unmount must tear the listener
  // down (that is the cleanup's whole job) or the second mount stacks a second.
  const firstRoot = createRoot(host);
  let secondRoot = null;
  try {
    await act(async () => {
      firstRoot.render(<App />);
    });
    await act(async () => {
      firstRoot.unmount();
    });
    secondRoot = createRoot(host);
    await act(async () => {
      secondRoot.render(<App />);
    });

    // Without a cleanup the first listener was never removed, so two are live.
    expect(liveTickListeners).toBe(1);

    // And the behaviour the user sees: a single tick bumps the count by one.
    await fireTick();
    expect(readCount()).toBe('1');
  } finally {
    window.addEventListener = realAdd;
    window.removeEventListener = realRemove;
    if (secondRoot) {
      await act(async () => {
        secondRoot.unmount();
      });
    }
    host.remove();
  }
});
`}
/>

<summary>Reveal the fix</summary>

```tsx
useEffect(() => {
  const onTick = () => setCount((c) => c + 1);
  window.addEventListener('tick', onTick);
  return () => window.removeEventListener('tick', onTick);
}, []);
```

The returned cleanup removes the exact listener the setup added, so a remount tears the old one down before adding a new one — there is never more than one live listener, no matter how many times the component mounts.

</details>

## Where the doubling does and doesn't happen

One last thing, so the doubling never surprises you in the wrong place. Strict Mode's double-invocation is a *Client Component* phenomenon. Your Next.js app is Server Components by default, and Server Components don't run under it — they execute once per request, on the server, full stop. The checks kick in only when you cross into client code (the `'use client'` boundary you'll learn in the App Router) and below.

<Aside type="note">
If you don't see doubling, you may not have crossed into a Client Component yet. Server Components run once on the server; Strict Mode's mount-unmount-mount only happens in the client tree.
</Aside>

You'll also occasionally see *other* dev-only warnings in the console — React flagging a deprecated API, an unsafe legacy lifecycle, or a string ref from some older third-party library. You won't write any of those yourself, but recognize the shape: a yellow warning naming a component or an API is React telling you, in dev, about something that'll bite later. Same messenger, different message.
