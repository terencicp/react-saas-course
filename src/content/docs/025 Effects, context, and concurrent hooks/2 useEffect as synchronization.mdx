You are building a chat feature. When the panel opens, it has to connect to a WebSocket for the current room. When the panel closes, it has to close that connection. And when the user clicks a different room, it has to drop the old connection and open a new one. No amount of `useState`, and no clever deriving-in-render, gets you there. A live socket is a thing that exists *outside* React's render — React can't compute it, only command it. Opening it, closing it, and re-pointing it when inputs change is exactly the job `useEffect` exists for.

That is the lesson: how to keep something outside React <Term definition="Make an external system's state match React's current props and state.">synchronized</Term> with your component's current props and state. We are going to open the black box you met in the previous lesson — setup plus an optional cleanup — and learn to write the setup, write the matching cleanup, and read the dependency array.

But first, a warning that has to come before any syntax, because it changes how you'll read everything that follows.

In 2020, `useEffect` was the hook you reached for constantly. Fetching data on mount? Effect. Keeping the URL in sync with a filter? Effect. Resetting a form when the selected record changed? Effect. Computing a derived value? Effect. It was the catch-all, and codebases drowned in them. In 2026, every one of those jobs moved to a better-shaped tool. What is left for `useEffect` is a narrow set of cases: synchronizing with systems React does not own. The single most common mistake a junior makes with this hook in 2026 is reaching for it at all. So we will lead with *when not to use it*, then teach the mechanics for the times one is genuinely warranted.

`useEffect` is an <Term definition="A deliberate exit from React's normal data flow, used sparingly and on purpose.">escape hatch</Term> — a deliberate step outside React's normal data flow, taken sparingly and on purpose. A later lesson, "You probably don't need an effect," is the full audit of when to take it. This lesson teaches the hatch itself.

By the end you'll be able to write a correct connect / disconnect / reconnect effect, clean up four different kinds of external resource, and sidestep the infinite-loop and stale-data traps that make effects feel cursed. We'll keep coming back to the chat room the whole way, so you're never re-orienting to a new example.

## What useEffect is for in 2026

Before the API, the decision. The reflex you want to build is: when you think "I'll add an effect here," your default answer is *no*, and you only override that default when the task is genuinely one of synchronizing with something React doesn't manage.

There are three shapes where the answer is yes. In a 2026 SaaS app, these are essentially the *only* places `useEffect` is the right tool:

1. **Non-React subscriptions** — a WebSocket, an `EventSource` for server-sent events, a `BroadcastChannel`. A live connection you open, listen to, and close.
2. **Third-party widgets that take a DOM node** — a charting library you hand a `<div>`, a map, Stripe Elements, a video player. Code outside React that needs to be created against a real element and torn down later.
3. **Browser APIs React doesn't model** — `IntersectionObserver`, `ResizeObserver`, `matchMedia`, raw scroll or resize listeners. Platform features with no React equivalent, that you subscribe to and unsubscribe from.

Every one of these has the same shape: something external gets *created* or *connected*, and later needs to be *destroyed* or *disconnected*. Hold onto that shape — it's the whole lesson.

Now the other side. Here is what used to live in effects and where each piece went. You won't learn the replacements today; the point is just to start wiring the reflex, so that when you see one of these tasks, "effect" is *not* the word that comes to mind:

- **Initial data the page needs** → a Server Component or route loader fetches it directly. (Covered when we reach the App Router.)
- **Cached or refetched server state** → TanStack Query, or `use()` for a streamed promise. (`use()` is later in this chapter; TanStack Query comes in its own chapter.)
- **State that lives in the URL** → `nuqs` / `useSearchParams`, which you already saw.
- **Form submission state** → Server Actions plus `useActionState`. (Its own unit.)
- **A value you can compute from props or state** → just compute it in render. You already know this one: derive, don't mirror.
- **Resetting state when a prop changes** → a `key` reset, which you already know too.

Notice the last two are things you already do. The 2026 narrowing isn't a pile of new rules to memorize — it's mostly consolidation of habits you've already built. Derive in render instead of syncing into state; reset with `key` instead of an effect. Those instincts were already pulling work *away* from effects. This section just names the boundary and tells you that almost everything lives on the other side of it.

## The signature, in its three forms

Start as simple as the hook gets:

```ts
useEffect(setup, dependencies?);
```

`setup` is a function — it's the code that does the synchronizing. It optionally returns a *cleanup* function, the code that undoes whatever setup did. We'll spend the next section on cleanup; for now, just hold it as "an optional function setup hands back."

The second argument, the dependency array, is where most of the behavior lives. There are three forms, and getting the *shape* of this array right is the highest-leverage skill in the whole lesson. Here they are side by side.

<CodeVariants>
  <CodeVariant label="Omitted — useEffect(fn)">
    ```ts
    useEffect(() => {
      connectToLobby();
    });
    ```
    **Almost always a bug.** With no dependency array at all, the setup runs after *every single render* — every keystroke, every parent update. You will see this in old code; treat it as a mistake until proven otherwise.
  </CodeVariant>

  <CodeVariant label="Empty [] — useEffect(fn, [])">
    ```ts
    useEffect(() => {
      connectToLobby();
    }, []);
    ```
    **The empty array means "no reactive dependencies."** Setup runs once when the component mounts, cleanup once when it unmounts. This is the "connect once, disconnect once" shape — fine when the effect genuinely never needs to re-sync.
  </CodeVariant>

  <CodeVariant label="With deps [roomId] — useEffect(fn, [roomId])">
    <div data-mark-color="blue">

    ```ts "roomId"
    useEffect(() => {
      connectToRoom(roomId);
    }, [roomId]);
    ```

    </div>
    **Re-syncs when a listed value changes.** The setup re-runs whenever any listed value changes, compared with `Object.is`. This is the "reconnect when the room changes" shape — and the one you'll reach for most. List the values the effect reads, and React re-syncs when any of them drifts.
  </CodeVariant>
</CodeVariants>

That comparison in the third form — "changes by `Object.is`" — is the same referential equality React already uses for the state bailout you met earlier. You don't need to relearn it: `Object.is` treats two values as equal when they're the same primitive or the very same object reference. React runs that exact check on each dependency to decide whether anything changed.

Here's the canonical chat effect with its parts labeled, so the moving pieces have names before we put them in motion.

<CodeTooltips tooltips={{
  useEffect: 'Runs the setup after React commits this render to the screen.\nReturns nothing, or a cleanup function.',
  '[roomId]': 'The reactive values the setup reads.\nReact re-syncs whenever one of them changes.',
  '() => connection.disconnect()': 'The cleanup.\nTears down exactly what this setup created, before the next sync and on unmount.',
}}>
```ts
useEffect(() => {
  const connection = connectToRoom(roomId);
  connection.connect();
  return () => connection.disconnect();
}, [roomId]);
```
</CodeTooltips>

One rule to plant now, because it prevents an error nearly everyone hits once: **the setup function must be synchronous, and it must not return a Promise.** Writing `useEffect(async () => { ... })` is a type error, and for a precise reason — an `async` function always returns a Promise, but React expects setup to return either nothing or a cleanup function. A Promise is neither. When you need to do async work inside an effect (and sometimes you genuinely do), the async lives in an *inner* function that you call; the setup itself stays synchronous.

```ts
useEffect(() => {
  const load = async () => {
    const data = await fetchRoomHistory(roomId);
    setHistory(data);
  };
  load();
}, [roomId]);
```

We'll come back to the async case properly when we deal with race conditions — there's a subtlety the snippet above is still missing. For now, just hold the rule: setup is synchronous; async work goes in a function setup calls.

## The lifecycle is synchronization, not "mount and unmount"

This is the heart of the lesson, and it's where the mental model you've probably absorbed from somewhere else will fight you. So let's name the wrong model and kill it.

The wrong model goes like this: `useEffect(fn, [])` is "run this when the component mounts," and the cleanup is "run this when the component unmounts." People who learned an older React vocabulary call these "componentDidMount" and "componentWillUnmount." It's a tempting picture because, for the empty-array case, it even produces the right behavior. But it's the wrong model, and it produces a specific, nasty bug the moment a dependency enters the picture.

The right model: **an effect's job is to make the outside world match the *current* props and state.** That's it. When a dependency changes, the world built from the old values is now stale, so React tears it down and builds it again from the new values. Concretely, React does **cleanup, then setup**: it runs your cleanup to undo the old sync, then runs setup to establish the new one.

So cleanup is not "the unmount handler." Cleanup runs **before every re-sync, and also on unmount.** Whenever the effect is about to run again, the previous run gets cleaned up first.

Watch it happen in the chat room. The user is in `general` and clicks `random`, so `roomId` changes from `"general"` to `"random"`. React runs the cleanup first — disconnect from `general` — and *then* the setup — connect to `random`. If you'd written the effect with the wrong mental model and skipped the cleanup, the `general` connection would never close. The user would now be receiving messages from *both* rooms, and every room switch would stack another live connection. That is the canonical leak, and it falls out directly from thinking "unmount" instead of "re-sync."

## The dependency array is a contract, not a tuning knob

It's tempting to treat the dependency array as a dial — add a value to make the effect run more, remove one to make it run less. That's the wrong frame, and it's where lying-to-React bugs come from. The array is a **contract**, and the contract is exact:

**Every reactive value the setup reads belongs in the dependency array.** A <Term definition="A prop, state, or anything derived from them; can change between renders.">reactive value</Term> is anything that can change between renders — props, state, and any value computed from them. React uses the array to detect when the outside world has drifted from the current render and a re-sync is due. If you read a value but leave it out of the array, you've broken the contract, and React can no longer tell when to re-sync.

You don't have to track this by hand. The lint rule `react-hooks/exhaustive-deps` reads your setup, finds every reactive value you reference, and flags any you forgot. It ships in the default Next.js ESLint config and runs at the `warn` level. This is one of the two React hook rules from the previous chapter that you never disable — but here's the important reframe: treat it as an **ally**, a correctness oracle, not an obstacle. When it flags a missing dependency, the experienced reflex is to *add the dependency*, not to silence the rule.

What goes wrong if you lie to it? You get a <Term definition="A function capturing values from an earlier render, used after they've changed.">stale closure</Term>. The effect's setup captured the value from the render where it last ran; if that value changes but the effect doesn't re-run (because you omitted it from the array), the effect keeps using the old, captured value forever. You met closures capturing by reference earlier in the course — this is that exact trap, now wearing an effect costume. The code doesn't error. It just quietly uses yesterday's data.

:::caution
An effect that lies about its dependencies doesn't throw an error — it silently keeps using values from the render where it last ran. Stale data with no stack trace is the worst kind of bug to chase. The lint warning is trying to prevent exactly this; listen to it.
:::

Two objections come up constantly when the lint asks for a dependency. Both feel like reasons to disable the rule. Neither is.

**"But adding it makes my effect re-run too often."** You add `currentUser` to the deps because the effect reads it, and now the chat reconnects every time the user edits their display name. That's clearly wrong — changing your name shouldn't drop your socket. But the fix is *not* to delete the dependency and lie. This is the **non-reactive trap**: the effect reads a value at *event time* (when a message arrives, you call `onMessage(msg, currentUser)`) that genuinely should not *drive* re-synchronization. The right tool carves out a non-reactive seam so you can read the latest value without listing it as a trigger. That tool is `useEffectEvent`, and it's the entire subject of the next lesson. For now, just feel the pain clearly — reading a fresh value versus re-running on its change are two different needs — and know the relief is one lesson away.

**"The lint is asking for things that never change."** Some values are stable by guarantee, and the lint already knows it — it won't ask for them. The `set` functions from `useState` and the `dispatch` from `useReducer` keep the same identity for the component's whole life (React promises this, as you saw when we covered state). Refs (`ref.current`) are the same. You can read these inside an effect without listing them, and the lint won't complain.

So the discipline is small and firm: read a reactive value, list it. If listing it causes a re-run you don't want, that's a signal you have a non-reactive read — reach for the seam (next lesson), never the `eslint-disable`.

## Four cleanups for four kinds of resource

Now the discipline that makes effects safe instead of leaky. The rule is one sentence: **every effect that creates something outside React returns a cleanup that destroys exactly that thing.** The cleanup mirrors the setup — same object, undone. If setup added a listener, cleanup removes that listener. If setup opened a connection, cleanup closes that connection.

There are four canonical pairings you'll write over and over. Here they are as a reference — setup line on top, cleanup line below, in each tab.

<CodeVariants>
  <CodeVariant label="Listener">
    ```ts ins={2}
    window.addEventListener('resize', onResize);
    return () => window.removeEventListener('resize', onResize);
    ```
    **Add a listener, remove the *same* listener.** The function reference must be identical in both calls — an inline arrow in each would create two different functions, and the cleanup would remove nothing.
  </CodeVariant>

  <CodeVariant label="Timer">
    ```ts ins={2}
    const id = setInterval(tick, 1000);
    return () => clearInterval(id);
    ```
    **Start an interval (or a timeout), clear it by its id.** This is the uncleared-interval leak from the previous lesson — now with the cleanup that fixes it. Without the clear, every mount stacks another ticking timer.
  </CodeVariant>

  <CodeVariant label="Subscription">
    ```ts ins={2}
    const unsubscribe = store.subscribe(onChange);
    return () => unsubscribe();
    ```
    **Many subscription APIs hand you an unsubscribe function when you subscribe.** Capture it and call it in the cleanup. Don't reinvent the teardown — the API already gave it to you.
  </CodeVariant>

  <CodeVariant label="Resource">
    ```ts ins={2}
    const chart = createChart(node);
    return () => chart.destroy();
    ```
    **A third-party instance — a chart, a map, Stripe Elements — gets created against a DOM node and exposes a `destroy()` (or `remove()`).** Create it in setup, destroy it in cleanup. This is the shape for every widget you mount into React.
  </CodeVariant>
</CodeVariants>

This gives you a code-review habit worth keeping for the rest of your career. Read any effect and ask one question: *what does the cleanup tear down?* If the cleanup tears down the exact thing the setup created, good. If the cleanup is empty — and nothing external was actually created — that's a smell. An effect with nothing to clean up is usually one of the anti-patterns from a later lesson wearing a costume: a derived value that should be computed in render, a handler's logic that wandered into an effect, a fetch that should be a Server Component. The empty cleanup is the tell. Flag it.

Now write one. The next exercise hands you an effect that subscribes but never cleans up. Your job is to add the cleanup and watch the leak stop.

  starter={`import { useEffect, useState } from 'react';

// A fake resize subscription. It hands back an unsubscribe function —
// the same shape real subscription APIs (sockets, stores) give you.
const subscribeToResize = (onResize: () => void) => {
  window.addEventListener('resize', onResize);
  return () => window.removeEventListener('resize', onResize);
};

const ResizePanel = () => {
  useEffect(() => {
    const unsubscribe = subscribeToResize(() => {});
    // The setup subscribed and got an unsubscribe back. Return a cleanup
    // that calls it, so closing the panel tears the listener down.
  }, []);

  return <p data-testid="panel">Resize panel mounted</p>;
};

export function App() {
  const [isOpen, setIsOpen] = useState(true);
  return (
    <div>
      <button data-testid="toggle" onClick={() => setIsOpen((open) => !open)}>
        Toggle panel
      </button>
      {isOpen && <ResizePanel />}
    </div>
  );
}`}
  tests={`
// act() flushes passive effects deterministically, so each mount runs the
// effect's setup and each unmount runs its cleanup before we measure.
(globalThis as any).IS_REACT_ACT_ENVIRONMENT = true;

// Mount App into our own host, with a probe on window's resize listeners, run
// the body, then tear the host down. The probe counts every 'resize' add (+1)
// and matching remove (-1); subscribeToResize routes through those, so the
// counter tracks exactly THIS App's panel listeners. Installing the probe
// before render and scoping queries to \`host\` keeps the measurement isolated.
const withMountedApp = async (
  run: (ctx: {
    host: HTMLElement;
    counter: { live: number };
    toggle: () => Promise<void>;
    panelIsOpen: () => boolean;
  }) => Promise<void>,
) => {
  const counter = { live: 0 };
  const realAdd = window.addEventListener.bind(window);
  const realRemove = window.removeEventListener.bind(window);
  window.addEventListener = (type, handler, opts) => {
    if (type === 'resize') counter.live += 1;
    return realAdd(type, handler, opts);
  };
  window.removeEventListener = (type, handler, opts) => {
    if (type === 'resize') counter.live -= 1;
    return realRemove(type, handler, opts);
  };

  const host = document.createElement('div');
  document.body.appendChild(host);
  const root = createRoot(host);

  const toggle = async () => {
    const btn = host.querySelector('[data-testid="toggle"]') as HTMLButtonElement;
    await act(async () => {
      btn.click();
    });
  };
  const panelIsOpen = () => !!host.querySelector('[data-testid="panel"]');

  try {
    await act(async () => {
      root.render(<App />);
    });
    await run({ host, counter, toggle, panelIsOpen });
  } finally {
    await act(async () => {
      root.unmount();
    });
    host.remove();
    window.addEventListener = realAdd;
    window.removeEventListener = realRemove;
  }
};

test('panel mounts open with exactly one live resize listener', async () => {
  await withMountedApp(async ({ counter, panelIsOpen }) => {
    expect(panelIsOpen()).toBe(true);
    expect(counter.live).toBe(1);
  });
});

test('Resize listener is still attached after the panel closes', async () => {
  await withMountedApp(async ({ counter, toggle, panelIsOpen }) => {
    // Close the panel — its effect should clean up and remove the listener.
    await toggle();
    expect(panelIsOpen()).toBe(false);
    expect(counter.live).toBe(0);
  });
});

test('Toggling the panel repeatedly stacks listeners instead of replacing them', async () => {
  await withMountedApp(async ({ counter, toggle, panelIsOpen }) => {
    let peak = counter.live;
    // Off, on, off, on, off — a correct cleanup keeps the count bouncing
    // between 0 and 1; a leak makes it climb past 1.
    for (let i = 0; i < 5; i += 1) {
      await toggle();
      peak = Math.max(peak, counter.live);
    }
    expect(peak).toBe(1);
    // Ended closed after an odd number of toggles, so nothing is left live.
    expect(panelIsOpen()).toBe(false);
    expect(counter.live).toBe(0);
  });
});
`}
/>

<summary>Reveal the fix</summary>

```tsx
useEffect(() => {
  const unsubscribe = subscribeToResize(() => {});
  return unsubscribe;
}, []);
```

The effect returns the `unsubscribe` function it was already handed, so React runs it on unmount and removes the exact listener the setup added. Closing the panel now tears its listener down instead of orphaning it, and toggling never stacks more than one.

## Two race patterns for async inside an effect

Most fetching is not an effect's job in 2026 — we said it up top, and it's worth saying once more so nothing here reads as an endorsement. But residual cases exist: an SDK method that takes an `AbortSignal`, a one-off `POST` that shouldn't be cached, a browser API that returns a promise. When async work *does* live in an effect, it carries a race condition you have to know how to kill.

The race, in the chat room: the user clicks through rooms fast. You fire a request for `general`'s history, then immediately a request for `random`'s. But the network doesn't promise order — `general`'s response might arrive *after* `random`'s, and your `setHistory` would overwrite the correct (new) data with stale (old) data. The newest click loses. That's the bug, and the fix is, once again, the cleanup.

There are exactly two patterns, and which one you use depends on whether the async call can be cancelled.

<CodeVariants>
  <CodeVariant label="Abortable">
    <div data-mark-color="red">

    ```ts {8}
    useEffect(() => {
      const controller = new AbortController();
      fetchRoomHistory(roomId, { signal: controller.signal })
        .then(setHistory)
        .catch((error) => {
          if (error.name !== 'AbortError') throw error;
        });
      return () => controller.abort();
    }, [roomId]);
    ```

    </div>
    **Use this when the call accepts an `AbortSignal`** — which is the same `AbortController` you used for `fetch` earlier in the course. Pass it the signal and abort in the cleanup. On a room switch, the cleanup aborts the in-flight request *before* the new setup fires, so the stale response never resolves into `setHistory`. The `.catch` swallows the expected `AbortError` and re-throws anything real.
  </CodeVariant>

  <CodeVariant label="Not abortable">
    <div data-mark-color="red">

    ```ts {6-8}
    useEffect(() => {
      let ignore = false;
      fetchRoomHistory(roomId).then((history) => {
        if (!ignore) setHistory(history);
      });
      return () => {
        ignore = true;
      };
    }, [roomId]);
    ```

    </div>
    **Use this when the call can't be cancelled** — don't try to stop the request, just stop *listening* to it. A local `ignore` flag starts `false`; the cleanup flips it to `true`. The resolved callback checks the flag and no-ops if this effect has already been cleaned up. The stale response still arrives, but it's ignored.
  </CodeVariant>
</CodeVariants>

The selection rule is simple: **if the API accepts a `signal`, abort; otherwise, use the `ignore` flag.** Aborting is better when you can do it — it stops wasted work, not just wasted state updates — but the flag covers everything else.

Notice the shape underneath both. The async work lives in an inner promise chain (`.then`), so the setup itself stays synchronous and returns a real cleanup — exactly the rule from the signature section, now paying off. And notice that this is *the same mechanism* as every other effect: cleanup-on-resync. We're not learning a new lifecycle; we're applying the lifecycle you already know to async. This is also why Strict Mode fires two requests in development — it's surfacing a missing abort the same way it surfaces every other missing cleanup.

## When dependencies are objects, arrays, and functions

This is the single most common real-world `useEffect` bug, and it earns its own section because the fix is genuinely non-obvious the first time you hit it. Here's the trap: a dependency whose *reference* changes every render, even though its *contents* are identical — and since `Object.is` compares identity, React sees "changed" every single time. The effect re-runs on every render. And if that effect calls `setState`, the new state triggers a render, which makes a new dependency, which re-runs the effect, which calls `setState` — an infinite loop.

The culprit is almost always an object or array literal. Every time a component renders, `{ id: 1 }` evaluates to a *brand-new* object. It's never `Object.is`-equal to the one from the last render, even though it looks identical. Walk the failure and the fix.

<CodeVariants>
  <CodeVariant label="Broken">
    ```tsx del={2,10}
    const RoomPanel = ({ roomId }: { roomId: string }) => {
      const options = { roomId, theme: 'dark' };
      return <Messages options={options} />;
    };

    const Messages = ({ options }: { options: RoomOptions }) => {
      const [messages, setMessages] = useState<Message[]>([]);
      useEffect(() => {
        loadMessages(options).then(setMessages);
      }, [options]);
      // ...
    };
    ```
    **New object every render → effect every render → `setMessages` every render → loop.** The parent builds `options` fresh on each of its renders, so `options` is a different reference each time; `Object.is` always reports a change, and the effect never stops re-running.
  </CodeVariant>

  <CodeVariant label="Fixed">
    ```tsx ins={3,6}
    const Messages = ({ options }: { options: RoomOptions }) => {
      const [messages, setMessages] = useState<Message[]>([]);
      const { roomId, theme } = options;
      useEffect(() => {
        loadMessages({ roomId, theme }).then(setMessages);
      }, [roomId, theme]);
      // ...
    };
    ```
    **Depend on the *primitive* fields the effect actually uses.** Strings and numbers compare by value under `Object.is`, so the deps only count as changed when the real values change. The loop stops.
  </CodeVariant>
</CodeVariants>

When an object or array dependency is the problem, there's a fix ladder. Try them in this order:

1. **Depend on the primitive fields you actually read.** If the effect uses `options.roomId` and `options.theme`, list those, not `options`. This is the fix above, and it covers most cases.
2. **Move the object construction so it isn't recreated each render** — or memoize it upstream. (The React Compiler, which the next chapter covers, usually handles this memoization for you; that's a reason not to reach for manual memoization first.)
3. **Restructure so the parent passes primitives** rather than assembling an object only to be torn apart again downstream.

**Functions hit the same wall.** A function defined inside the component body is a new reference on every render, just like an object literal. List it in a dependency array and the effect re-runs every render. The fix ladder, in order of preference:

1. **Move the function *inside* the effect.** If only the effect uses it, define it in the setup — then it isn't a dependency at all, and the problem vanishes.
2. **Move it *outside* the component.** If the function captures no reactive values, hoist it to module scope. Module-level functions have one stable identity forever.
3. **`useCallback`** — only when the function must be passed to a memoized child that depends on its identity. This is a narrow exception with a real cost; the next chapter covers when it's warranted. Don't reach for it reflexively.
4. **`useEffectEvent`** — when it's an event-shaped read of the latest values. That's the next lesson's tool.

The skill to build here is fast recognition. The moment you see "re-runs every render" or "infinite loop," your first thought should be *identity* — a dependency whose reference changes each render — and your first move should be to depend on the primitive instead.

Diagnose one before moving on.

<MultipleChoice>
  A parent renders `<Cart item={{ id: sku, qty: 1 }} />`, building that object fresh each render. The child loops forever:

  ```tsx
  const Cart = ({ item }: { item: { id: string; qty: number } }) => {
    const [total, setTotal] = useState(0);
    useEffect(() => {
      setTotal(priceFor(item));
    }, [item]);
    // ...
  };
  ```

  The effect fires on every render and the component never settles. Which change actually stops the loop?

  <McqChoice>Silence the warning with `// eslint-disable-next-line react-hooks/exhaustive-deps` on the line above `[item]`.</McqChoice>
  <McqChoice>Store `total` a second time in another `useState` and read from that copy.</McqChoice>
  <McqChoice correct>List `item.id` in the array in place of `item`, so the dependency is a string.</McqChoice>
  <McqChoice>Delete the `[item]` array so the effect stops comparing dependencies.</McqChoice>

  <McqWhy>The parent hands `Cart` a brand-new `item` object on every render, so `Object.is` reports it as changed each time and the effect re-runs — and since the effect calls `setTotal`, each run schedules the next render. The cure is to depend on a value that compares by *contents*, not by reference: `item.id` is a string, so the deps only register a change when the id actually differs, and the loop stops. Silencing the lint hides the warning while the loop runs on; deleting the array is strictly worse — with no array the effect runs after *every* render unconditionally.</McqWhy>
</MultipleChoice>

## useLayoutEffect and useSyncExternalStore: the two specialist variants

`useEffect` has two siblings you should be able to *recognize*, but will rarely reach for. The point of this section is to make their triggers glanceable, so when you see them in a library's source it isn't a mystery — and so you know that `useEffect` is the default you almost always want.

<TabbedContent>
  <TabbedItem label="Need to avoid a paint flicker?">
    `useLayoutEffect` is the synchronous sibling. A regular `useEffect` runs *after* the browser paints; `useLayoutEffect` runs after React commits the DOM but *before* the browser paints. That difference matters in exactly one situation: you need to measure a DOM node and change state based on the measurement *without the user seeing a flicker*. The canonical case is a tooltip — measure its rendered width, then reposition it so it doesn't spill off-screen, all before the first paint.

    Reach for it only when a visible flicker is the actual problem. Because it blocks paint, overusing it hurts performance; `useEffect` is correct and cheaper everywhere else. (There's a third, `useInsertionEffect`, that's strictly CSS-in-JS-library territory — you'll never write it.)
  </TabbedItem>

  <TabbedItem label="Reading an external store safely?">
    `useSyncExternalStore` is the correct primitive for reading a value that lives *outside* React — a `window` property, a third-party store, a `BroadcastChannel` — in a way that stays safe under concurrent rendering. It prevents <Term definition="Different parts of a single render observing different values of the same external source.">tearing</Term>: different parts of a single render observing different values of the same external source.

    You will almost never call it directly. The libraries you already use call it internally — a store like Zustand, or `useSearchParams`, are built on it. Know the name so it isn't mysterious in a library's source, and know that *if* you ever hand-integrate an external store yourself, this — not `useEffect` plus `useState` — is the right tool.
  </TabbedItem>
</TabbedContent>

To be clear: these are exceptions. `useEffect` is the default for synchronization. Reach for `useLayoutEffect` only to kill a flicker, and you'll likely go months without writing `useSyncExternalStore` at all.

## Effects and the server boundary

One last fact, and it closes the loop on everything we opened with: **effects only run in Client Components.** They don't run during server rendering, and they never run in a Server Component. You saw the `'use client'` directive in the walkthrough earlier — that boundary is exactly why the effect is allowed to run there at all.

This gives you a reflex worth more than any single piece of syntax. When a component "needs an effect," an experienced engineer's *first* question isn't "how do I write this effect?" — it's "should this even be a Client Component?" Could this be a Server Component that reads the data directly, no effect required? That question, and the Server/Client boundary that makes it answerable, is taught in full when we reach the App Router; here, just install the reflex.

And it ties the bow on the lesson's opening. An effect that doesn't run on the server means **anything that must be correct on the very first paint cannot live in an effect.** First-paint data is precisely what Server Components and `use()` (later in this chapter) are for. So the narrowing we started with isn't just a list of better tools — it's enforced by the runtime itself. Effects are for synchronizing with the world *after* the page is alive, not for getting it alive in the first place.

:::note
If you remember one thing: when you read or write an effect, ask two questions. *What external system does this synchronize with?* and *What does the cleanup tear down?* An effect that can't answer the first is probably the wrong tool. An effect that can't answer the second is probably leaking.
:::
