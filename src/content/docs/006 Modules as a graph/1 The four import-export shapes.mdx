You have been writing `import` lines for five chapters by example, without naming the shapes. This chapter treats your codebase as what it is: a **directed graph** of modules with edges drawn by your imports. This first lesson installs the reading skill the rest of the chapter depends on.

Every `import` line is one of four shapes — **named**, **default**, **side-effecting**, or **dynamic** — and an experienced engineer reads the shape before the contents. Each shape draws a different kind of edge. Two cross-cutting concerns ride on top: the `import type` discipline that keeps type-only edges out of runtime code, and the bare-specifier rule that decides where `'react'` or `'@/db'` actually points to.

## Named exports: the course default

The first shape is the one you will write nearly every time. A module exports values by name; an importer pulls those names out by name. The exporter and the importer agree on the spelling.

```ts
export const TAX_RATE = 0.08;

export const formatPrice = (cents: number) => `$${(cents / 100).toFixed(2)}`;

export type Money = { cents: number; currency: string };
```

The matching import:

```ts
import { TAX_RATE, formatPrice, type Money } from './pricing';
```

Three values, one import line, identical spellings on both sides. That spelling alignment is the entire point. Rename `formatPrice` to `formatAmount` and every caller breaks at compile time — your editor's rename refactor walks every import line and updates them. The bundler can drop any named export that no consumer imports (this is <Term definition="Bundler optimization that removes exports no consumer imports; relies on static analysis of import/export lines.">tree-shaking</Term>), so an unused helper costs nothing at runtime. And the call site reads explicitly — when you see `{ formatPrice }` in an import, you know exactly which symbol you are pulling in.

Every utility, every component, every type you author in this course is a named export. There is no exception to learn — the carve-outs all belong to the next section.

## Default exports: only when the framework demands them

The second shape exists, but you will rarely choose to author it. A module marks one export as its default; an importer picks any name it wants for that default.

```ts
export default function Page() {
  return <h1>Invoices</h1>;
}
```

```ts
import Page from './page';
```

Notice that `Page` on the import side is a name the caller chose. The exporter did not declare `Page`; it declared "the default." If you renamed the function in `./page` to `InvoicesIndex`, every import line in the rest of the codebase would keep saying `import Page from './page'` and keep compiling. The rename does not propagate, because there is no name to propagate.

That asymmetry is why named exports are the default and default exports are the exception. The exceptions in 2026 are small and framework-owned:

- Next.js App Router special files: `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx`, `template.tsx`, `default.tsx`.
- A handful of third-party libraries whose primary export ships as default — React itself, for example.
- Everywhere else: **named, not default.**

When you write a `page.tsx`, you write `export default`. Everywhere else in the codebase you write `export const` or `export function`. The named-vs-default question is settled in 2026; the senior cut is to follow the framework where the framework demands a default, and reach for named in every file you own.

## Side-effecting imports: when the file runs for what it does, not what it returns

The third shape looks almost wrong the first time you see it. There is no binding on the left of `from` — no `{ ... }`, no default name. The importer asks the runtime to **evaluate the file**, then moves on.

```ts
import 'server-only';
import './globals.css';
```

The first line registers a build-time enforcement that no client bundle can ever reach this module (the next lesson teaches what `server-only` enforces). The second line tells Next.js to include the global stylesheet in the page's CSS bundle. Neither produces a value the importer reads; the value of the statement is the **side effect** of evaluating the file.

In 2026 SaaS code, side-effecting imports are rare and intentional. The two canonical sites are the two above: a `'server-only'` or `'client-only'` guard at the top of a boundary file, and a global CSS import in `app/layout.tsx`. Anywhere else, a bare `import 'something'` is a smell — it usually means the file is doing something at the top level that the consumer cannot see from the import line. When you encounter one in code review, treat it as something to ask about, not pattern-match past.

## Dynamic imports: a value-level expression that returns a Promise

The fourth shape is structurally different from the first three. The first three are **statements** that live at the top level of a module; they declare a binding before any code runs. A dynamic import is an **expression** — it can appear anywhere a value can appear, and it returns a `Promise<Module>` that the runtime resolves when the expression evaluates.

```ts
const onClick = async () => {
  const { renderChart } = await import('./chart');
  renderChart();
};
```

The `import('./chart')` syntax looks like a function call, and for purposes of reading the code, you can think of it as one. It returns a promise; you `await` it; you destructure the module's exports off the resolved value. The edge into the module graph is still real — the runtime fetches and evaluates `./chart` when the expression runs — but the edge is **deferred**. The bundler treats it specially: it emits `./chart` (and everything `./chart` imports) as a separate chunk, fetched on demand the first time the click handler runs. (The code-splitting story is the subject of the next lesson.)

The four shapes are the spine of every import line in the codebase. The rest of this lesson is two cross-cutting concerns that ride on top of those four, and then two structural rules — re-exports and bare-specifier resolution — that decide what gets imported and from where.

## The type-only import discipline

The first cross-cutting concern is one you have already met. Back in the "Annotate the boundaries, infer the inside" lesson, the course turned on <Term definition="TypeScript flag that requires explicit `type` keywords on type-only imports and exports; prevents the bundler from erasing imports the runtime needs.">verbatimModuleSyntax</Term>, the TypeScript flag that requires every type-only import to carry the `type` keyword. The flag is on for the rest of the course. This is the lesson that names it as a recognition skill.

There are two forms you will write. Decide between them by asking: does this file need the **value** at runtime, or only the **type** at compile time?

<CodeVariants>
  <CodeVariant label="Pure type import">
    ```ts
    import type { User } from './users';

    declare function getUser(id: string): Promise<User>;
    ```
    When a file uses a symbol only as a type — in a parameter annotation, a return type, a type alias — reach for `import type`. The whole statement is erased at compile time; no edge into the module graph survives into the runtime. This is the default reach when the file genuinely does not need the value.
  </CodeVariant>

  <CodeVariant label="Mixed value + type">
    ```ts
    import { createUser, type User } from './users';

    const newUser: User = await createUser({ email: 'a@b.com' });
    ```
    When one file needs both the value and the type from the same module, use the inline `type` modifier. The value `createUser` draws a real runtime edge into `./users`; the `User` import is erased. Splitting these into two separate statements would split a logical pair across two lines for no readability gain. This is the senior reach when both surfaces matter.
  </CodeVariant>
</CodeVariants>

The same `type` keyword works on re-exports too:

```ts
export type { User } from './users';
```

That line forwards the `User` type to consumers of the current module without drawing any runtime edge. Same discipline as imports.

Why does any of this matter? Because of the silent-bug class `verbatimModuleSyntax` prevents. The flag enforces a strict rule: if an import is a type, mark it with `type`; otherwise the compiler keeps the statement in the emitted code unconditionally. This closes a gap that bites the moment a module has **side effects** — a `console.log`, a global registration, a `'server-only'` guard. Without the flag, writing `import { Logger } from './logger'` and only using `Logger` as a type lets the compiler erase the entire import statement, silently dropping the logger's initialization. The bug would not surface until production, when the logs go missing. `verbatimModuleSyntax` refuses that ambiguity at the source.

## Re-exports: the transparent forwarding form

A module can pull a binding from one module and immediately republish it under its own name. The shape:

```ts
export { createInvoice } from './actions';
export type { Invoice } from './types';
// wildcard form — used sparingly
export * from './schemas';
```

Each `export ... from` line draws a real edge into the named module (the module is still evaluated) and makes the listed bindings available from the re-exporter. This is the right reach when a domain module wants to expose a curated surface — `db/queries/invoices.ts` re-exporting the four read helpers callers actually use, hiding the fifteen internal building blocks that compose them.

Where re-exports become a problem is the **barrel file** pattern: an `index.ts` that re-exports dozens of unrelated symbols from across a directory. Barrels hurt tree-shaking (some bundlers cannot tell which downstream re-exports are unused) and degrade editor go-to-definition (the IDE walks through the barrel before reaching the real source). The course's code conventions are explicit: do not author barrels in `lib/`, `db/`, or `_lib/`. Reach for `export *` only when the surface is intentionally open — a domain module that wants to expose its full public API in one swoop.

One mechanical rule: `export *` from two modules that share a symbol name is a compile error. The compiler refuses the ambiguity rather than silently picking one. The wildcard form trades that safety for terseness; reach for it only when you trust the named modules not to collide.

## Where does `'pkg'` actually come from? Bare-specifier resolution

The second cross-cutting concern is the question every `import` line quietly answers: where does that string on the right of `from` actually point to? When the specifier starts with `./` or `../`, the answer is "relative to this file" — obvious. But the moment the specifier has no leading dot, it becomes a <Term definition="An import specifier with no `./`, `../`, or absolute path. Resolved through node_modules, the package's exports field, or tsconfig.json paths — never against the importing file's directory.">bare specifier</Term>, and the resolution rule changes. `'react'`, `'next/headers'`, `'@/db'` are all bare. Where do they resolve from?

The answer is one of three rules, and the diagram below shows all three side by side.

<BareSpecifierResolution />

Three different specifiers, three different rules. Walk them in turn.

The first row is a **package name**: `'react'`, with no slash. The runtime walks up the directory tree from the importing file looking for a `node_modules/react` folder. When it finds one, it reads `react/package.json` and looks at the <Term definition="Field in package.json that declares which files a package makes importable. Subpaths not listed here are blocked even if the file exists on disk.">exports field</Term> — a map declaring which files the package makes importable from which specifiers. In 2026, the `exports` field is the source of truth for what a package exposes, and a deep import it does not list will fail with `ERR_PACKAGE_PATH_NOT_EXPORTED` even if the file exists on disk. That refusal is intentional — it is how libraries communicate their public surface. (Older packages without an `exports` field fall back to the legacy `main` field.)

The second row is a **subpath inside a package**: `'next/headers'`, where everything after the first slash is a key the package's `exports` field must declare. Next.js ships `"./headers"` as a recognized subpath; the field points to the implementation file. The senior takeaway is the same as the first row, just sharper: the slash does not give you free access to any file in the package. If the `exports` field does not list `"./internal/something"`, then `import { x } from 'next/internal/something'` does not resolve, even if you can see the file in `node_modules/next/internal/something.js`. The subpaths a SaaS engineer reaches for daily are a small set: `next/cache`, `next/headers`, `next/navigation`, `next/server`, `zod/v4`.

The third row is a <Term definition="A tsconfig.json mapping that resolves an alias like `'@/lib/db'` to a project-relative file. Both the bundler and tsc read the same tsconfig.json and agree on the resolution.">TypeScript path alias</Term>: `'@/db'`, with the leading `@/`. This is not a `node_modules` lookup at all. The TypeScript compiler reads the `paths` field of `tsconfig.json` (the same field you set up back in the "Run TypeScript locally" lesson), sees `"@/*": ["src/*"]`, and resolves `'@/db'` to `src/db/index.ts` on disk. The bundler reads the same `tsconfig.json` and agrees on the resolution. You will see this alias on every line of every server file in the course — it lets you write the same path from anywhere in the codebase without `../../../`.

## JSON imports with the `with` attribute

One last shape that does not quite fit the four-form taxonomy but you will see in 2026 code: a JSON file imported as a module.

```ts
import config from './config.json' with { type: 'json' };
```

The `with { type: 'json' }` attribute is **mandatory** in modern Node and required by the V8 runtime for security — the runtime validates the file's MIME type before parsing it as JSON, refusing to evaluate a misnamed JavaScript file as data. The binding form is always default, because JSON modules expose only a default export (there are no named exports on a parsed JSON object). The shape is locked in.

One watch-out: this works only in static positions, at the top of a module. The runtime materializes the JSON before the module evaluates, so you cannot dynamically construct the path. For a dynamic JSON import, the attribute moves to the second argument of `import()`:

```ts
const { default: config } = await import('./config.json', { with: { type: 'json' } });
```

This is the shape for the rare case where you load a JSON file conditionally at runtime. The parse-with-Zod story at the wire boundary — fetching JSON over HTTP and validating its shape before you trust it — lives in a later chapter on JSON parsing; the form here is for **static** JSON files bundled into your code.
