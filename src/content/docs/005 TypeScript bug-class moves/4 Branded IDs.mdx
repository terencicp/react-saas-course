The chapter has been about the shape of values — discriminated unions for variants, transitions for state machines, exhaustiveness for the consumers that read them. Inside those variants, every `userId`, `invoiceId`, and `sessionId` has been a plain `string`. This lesson types the **identity** of those strings.

Picture a function that fetches an invoice by ID, and another that fetches a user. Both take a string. Both run a query. The signatures look like this:

```ts
declare function getInvoice(invoiceId: string): Promise<Invoice>;
declare function getUser(userId: string): Promise<User>;

declare const currentUser: { id: string; name: string };

// Compiles. The type system can't tell these strings apart.
await getInvoice(currentUser.id);
```

Both parameters are `string`. The structural type system sees two strings and shrugs. The call compiles. The query runs. The wrong row comes back — either the invoice whose primary key happens to match the user's ID (if one exists), or `null` and a confusing 404 (if it doesn't). In production, the user opens their dashboard and sees someone else's invoice.

The fix isn't to be more careful at the call site. The fix is to make `UserId` and `InvoiceId` **distinct types** even though they're the same shape at runtime. Brand them. From there, `getInvoice(currentUser.id)` fails to compile, the bug class disappears, and the rest of the codebase reads the brand without ceremony. This lesson installs the pattern — the declaration, the factory that creates branded values, the Zod integration at the parse seam, and the senior cut between IDs that earn a brand and strings that don't.

## Structural vs. nominal: the model the brand lives in

To understand why two `string`-typed parameters are interchangeable, you have to know what kind of type system TypeScript is. The answer shapes every move in this lesson.

TypeScript is **structural**. <Term definition="Type compatibility by **shape**. Two types with the same fields and types are interchangeable, even if their names differ.">Structural typing</Term> matches types by **shape**, not by name. If two type aliases boil down to the same underlying shape, they are interchangeable as far as the compiler is concerned. So `type Email = string` and `type UserId = string` are both, structurally, just `string`. Pass an `Email` where a `UserId` is expected, and TypeScript accepts it — the shapes match, the names are decoration.

Other languages — Java, C#, Rust, Swift — work differently. They are <Term definition="Type compatibility by **name**. A `UserId` is distinct from a `string` because it is named `UserId`, even if both share the same underlying shape.">nominal</Term>. A `UserId` is distinct from a `string` because it has a different name, regardless of shape. TypeScript has no nominal-type keyword. There is no `nominal type UserId = string` syntax. The workaround is the <Term definition="A compile-time marker (a phantom property) attached to a primitive type via intersection to give it a unique nominal identity. Erased at runtime.">brand</Term>: attach a phantom field to the type — a field that exists only at compile time — whose value carries a unique label. The label gives the type a name to refuse on.

What does that mean concretely? A branded `UserId` is, at runtime, just a `string`. At compile time, it's a `string` with an extra `__brand: 'UserId'` property bolted on via an intersection. JavaScript strings can't carry extra properties, so the property is **phantom** — it exists in the type system, never in the runtime value. But the compiler treats `string & { __brand: 'UserId' }` and `string & { __brand: 'InvoiceId' }` as distinct types because their phantom labels differ, even though both erase to `string` the moment the code runs.

That figure is the lesson's pivot. Keep it in mind: at runtime the brand is invisible, at compile time it gives the type a name. Everything else — the declaration syntax, the factory, the wire boundary — is just plumbing for that one idea.

## The brand declaration: a string with a phantom field

The course's default form is the **string-intersection brand**. Two lines per brand, one phantom field, no runtime construct. Here are two branded IDs and the call site that proves the compiler refuses the cross-call.

<AnnotatedCode lang="ts" code={`
type UserId = string & { readonly __brand: 'UserId' };
type InvoiceId = string & { readonly __brand: 'InvoiceId' };

declare function getUser(id: UserId): Promise<User>;
declare function getInvoice(id: InvoiceId): Promise<Invoice>;

declare const currentUserId: UserId;
declare const currentInvoiceId: InvoiceId;

await getUser(currentUserId);
await getInvoice(currentInvoiceId);
await getInvoice(currentUserId);
`}>
  <AnnotatedStep meta={`/& \\{ readonly __brand: '\\w+' \\}/`} color="blue">
    **The phantom property.** `string & { readonly __brand: 'UserId' }` is an **intersection type** — the value must be a `string` *and* have a `__brand` field. JavaScript can't actually carry a property on a primitive string, so the field exists only in the type system. The `readonly` modifier and the literal-typed value (`'UserId'` rather than just `string`) are deliberate: the literal makes the brand label part of the type's identity, so `'UserId'` and `'InvoiceId'` are non-assignable.
  </AnnotatedStep>

  <AnnotatedStep meta={`{1-2}`} color="green">
    **Two distinct types, same runtime shape.** Both erase to `string` when the code runs, but the literal brand labels (`'UserId'` vs `'InvoiceId'`) give them distinct identities at compile time. That distinction is the entire point — the compiler now has names to refuse on.
  </AnnotatedStep>

  <AnnotatedStep meta={`{12}`} color="red">
    **The compile error.** `getInvoice` expects an `InvoiceId`; it received a `UserId`. The diagnostic reads: `Argument of type 'UserId' is not assignable to parameter of type 'InvoiceId'`. The two literal brand labels are right there in the error message, so the on-call engineer can walk straight to the mix-up. The cross-call bug is uncompilable.
  </AnnotatedStep>
</AnnotatedCode>

A few mechanical notes on the declaration. `string & { ... }` is an **intersection type** — the previous chapter installed the operator; here it's working for us in a new role. The intersection says "the value must satisfy both sides," and while JavaScript can't actually attach a property to a primitive string, the *type* still requires the property to exist. The compiler reasons about the phantom field; the runtime never sees it.

The `readonly` modifier and the literal-typed value (`'UserId'`, not just `string`) are both deliberate. The literal type is what makes two brand labels non-assignable — change `'UserId'` to `'InvoiceId'` and you get a different type, structurally, because the literal values don't match. The `readonly` modifier is belt-and-braces: even if some pathological code surfaces the phantom property somehow, it can't be reassigned. State this once and move on — the readonly literal phantom is the form the rest of the course writes.

Where do the type declarations live? Per the project's file conventions, the brand factory and helper sit in `lib/branded.ts`. The per-entity types — `UserId`, `OrgId`, `InvoiceId`, `SessionId` — can live there too for global brands, or co-locate with the database schema once Unit 5 lands and the schema becomes the source of truth. Treat `lib/branded.ts` as the central home for this lesson.

## The brand factory: the only seam where `as` is allowed

You now have `UserId` and `InvoiceId` as distinct types. The compiler refuses to assign a bare `string` to either — and that is exactly the point. But every value that *enters* your code from outside — a database row, a request body, a URL segment, a `localStorage` read — arrives as a plain `string`. Some named place has to perform the cast from `string` to `UserId`. That place is the <Term definition="A function that converts a plain string into a branded ID, performing validation at the seam. The single place in the codebase where `as <Brand>` is allowed.">brand factory</Term>.

The simplest possible factory is a one-liner:

<CodeVariants>
  <CodeVariant label="Illustrative — bare cast">
    ```ts title="lib/branded.ts"
    export const userId = (value: string): UserId => value as UserId;
    export const invoiceId = (value: string): InvoiceId => value as InvoiceId;
    ```

    **The minimal shape.** The factory's whole job is to make `as UserId` legal in one named place. Downstream code never asserts — it imports the factory and calls it. The arrow function bound to `const` matches the course's default function form; no `function` keyword, because the factory is a thin wrapper, not a type-guard signature.
  </CodeVariant>

  <CodeVariant label="Production — Zod-parsed">
    ```ts title="lib/branded.ts"
    import { z } from 'zod';

    export const UserId = z.uuid().brand<'UserId'>();
    export type UserId = z.infer<typeof UserId>;
    export const userId = (value: string): UserId => UserId.parse(value);
    ```

    **The production form.** The factory wraps a Zod parse — the value is validated as a UUID at the seam, then re-branded by `.brand<'UserId'>()`. The schema, the type, and the factory share the name `UserId`; one identifier carries the validator, the inferred output type, and the seam that produces values of that type. Zod authoring lands in depth in a later unit; the API surface is the point here.
  </CodeVariant>
</CodeVariants>

Two things are happening across those tabs. The structural shape — the bare `as UserId` cast — is the simplest factory that satisfies the pattern's contract: one place where `as` is allowed, downstream code goes through the function. The production shape is what you'll actually write: the factory body becomes a validator, so the seam doesn't just trust the string, it *checks* the string before stamping it with the brand.

A few conventions worth pinning down. The **type** is `UserId` (PascalCase, since it's a type). The **factory function** is `userId` (camelCase, since it's a value). The pair reads naturally at the call site: `const id = userId(row.id)`, and the result is a `UserId`. This same-name-pair convention is the senior shape — keep it.

The discipline rule has one sentence, and it is the load-bearing rule for the whole pattern: **`as <Brand>` lives only inside the factory.** Every other reference to a branded ID goes through the factory. A call to `someString as UserId` outside `lib/branded.ts` is a code-review failure. The pattern's safety comes from the validation living at one named seam; bypass the seam and the bug class returns.

:::tip
The Zod 4 form `z.uuid()` is the top-level format builder. Zod 4 introduced top-level builders — `z.uuid()`, `z.email()`, `z.url()`, `z.iso.datetime()` — that replace the deprecated `z.string().uuid()` chain. New code uses the top-level form throughout.
:::

## The unique-symbol form: when the brand crosses a package boundary

There is a second brand-declaration form you'll see in the wild. It replaces the literal-typed `__brand` field with a symbol-keyed phantom property:

```ts
declare const userIdBrand: unique symbol;
type UserId = string & { readonly [userIdBrand]: never };
```

`unique symbol` is a TypeScript type that guarantees the symbol is uniquely identifiable — no other module can accidentally produce a property that collides with this one, even if a third-party type also happens to use a `__brand: 'UserId'` literal field. The trade-off is a little extra verbosity at the declaration; the call-site experience is identical.

When to reach for it: when the branded type leaves the package boundary. The trigger is publishing a library whose branded types must round-trip through codebases the author doesn't control, or when the brand must compose with `declare module` augmentations on third-party types — a topic the next chapter owns. For ordinary internal SaaS code, the string-intersection form is the course's default; the unique-symbol reach earns one mention here and you'll see it again at the right moment.

## Brands erase at the wire boundary

Here is the most important property of a brand, and the one most likely to confuse if you skip it: **the brand exists in the codebase, not in the value**. Serialize a `UserId` to JSON and the phantom property is gone — `JSON.stringify` only sees the underlying string. The receiving end of a `fetch`, a Server Action response, a database read, or a `localStorage` get receives a plain `string`. The wire never carries the brand.

That means **brands are a compile-time tool, not a runtime contract**. They don't validate the value as it crosses a network boundary. What validates the value is the **brand factory** on the receiving side — the same factory you already wrote, called once on the incoming string. The seam re-brands the value after validating it. From that line forward, downstream code holds a `UserId` again.

<WireBoundary />

The reflex an experienced engineer brings is: every place a branded value re-enters typed code from outside is a parse seam, and the factory is the parser. A Server Action that receives a `{ userId: string }` body calls `userId(input.userId)` once at the top; from that line down, the variable is a `UserId`. A `fetch` response handler does the same. A row read from the database that wasn't already typed by Drizzle does the same. This is the round-trip — branded in scope A, plain over the wire, re-branded in scope B.

This is also the seam where validation pays off the most. If the factory body is a Zod parse (the production tab from earlier in the lesson), the incoming string isn't just stamped — it's checked. A malformed ID from a hostile or buggy client fails at the factory, not three function calls later when the database query returns nothing.

## Zod integration: the schema is the source of truth

You saw the production factory in the `CodeVariants` block above. It's worth one explicit pass to name what's happening, because this is the shape the rest of the course writes.

```ts title="lib/branded.ts"
import { z } from 'zod';

export const UserId = z.uuid().brand<'UserId'>();
export type UserId = z.infer<typeof UserId>;
export const userId = (value: string): UserId => UserId.parse(value);
```

Three exports, one identifier shared between them. Let's walk through what each does.

`z.uuid().brand<'UserId'>()` creates a Zod schema that parses a string, checks it's a UUID, and tags the schema's output type with a `UserId` brand. The `.brand<...>()` method is Zod's own brand machinery, structurally compatible with the string-intersection brand the course defaults to — the output type Zod produces is `string & z.$brand<'UserId'>`, which behaves the same way for the compiler.

`z.infer<typeof UserId>` reads the inferred output type off the schema. Because the schema was branded, the inferred type carries the brand. One declaration, two artifacts: the runtime validator and the compile-time type.

`UserId.parse(value)` runs the schema on the incoming value, throwing if it's invalid and returning the branded value if it isn't. The factory wraps that one call.

The schema and the type **share the name** — both are exported as `UserId`. This is a deliberate naming carve-out. Most Zod schemas in the codebase follow the `<entity>Schema` convention (`invoiceSchema`, `createInvoiceSchema`), with the type derived as a separate identifier. Branded IDs collapse that pair: the brand and the schema are the same concept, the schema's whole purpose is to mint values of the branded type, and one identifier for the pair reads cleaner at every import site. The `<entity>Schema` rule still holds for everything else; brands are the carve-out.

## Drizzle integration, in one line

When the database schema lands (next unit), Drizzle column types accept a `.$type<T>()` modifier that overrides the inferred TypeScript type at the schema layer. A primary-key column declared with `.$type<UserId>()` makes every row read from that column return a `UserId`, not a `string`.

```ts
id: uuid('id').primaryKey().$type<UserId>();
```

The result: branded IDs flow out of `db.select()` straight into Server Actions, render functions, fetch responses — every downstream consumer reads the brand without any extra ceremony. The brand is set once at the schema and propagates through `$inferSelect` and `$inferInsert`. Database setup and the column-customization API land in the next unit; the seam is named here so you recognize it when it lands.

## Where brands earn their weight — and where they don't

Not every string deserves a brand. The pattern has a cost — declaration noise, factory imports, an extra parse step at every boundary — and you only want to pay that cost where the bug class actually exists. Three categories earn the reach:

- **Primary keys.** `UserId`, `OrgId`, `InvoiceId`, `SessionId`. The original mix-up bug from the introduction: `getInvoice(currentUser.id)` compiled because both arguments were `string`. Branding the primary keys makes the cross-call uncompilable at every call site in the codebase. This is where brands pay off the most.

- **External keys with semantic identity.** `StripeCustomerId`, `StripePriceId`, `StripeSubscriptionId`, `R2ObjectKey`. The third party owns the value's format, but your application reasons about them as distinct entities. Three paragraphs of webhook-handler code later, branding prevents passing a `StripeCustomerId` where a `StripeSubscriptionId` was expected. Stripe billing and R2 object storage land in later units; the pattern is set up here.

- **Secret-typed values.** `BearerToken`, `WebhookSecret`, `ApiKey`. Branding a secret makes it visible at the type level, so a logging wrapper or a response-body assembler can be lint-checked against accidentally including it. The brand is the type-level handle that lets `redact(response)` know which fields to strip. The security baseline unit owns the wider story; the brand is the type-level grip.

Now the other side. A `string` that holds **free-form user input** — an article title, a comment body, a search query, a display name — is **just a `string`**. Branding it adds declaration noise without preventing a real bug. There is no `ArticleTitle` that ever gets confused with a `CommentBody` in your codebase, because both are display strings that flow through the same render path. The branding tax outweighs the safety win every time.

:::tip
**The senior test.** Brand a string when **(1)** it crosses a schema boundary, **(2)** it has semantic identity — it matches a row somewhere — and **(3)** it could plausibly be confused with a different value of the same shape. If none of the three apply, the string is a string.
:::

## Exercise — brand `OrgId` and prove the cross-call fails

The starter below declares `UserId` correctly but leaves `OrgId` as a bare `string`. A `@ts-expect-error` directive sits above the cross-call `getOrgMembers(someUserId)` — its job is to assert that the line below it fails to compile. Right now the line compiles (both arguments are still structurally `string`), so the directive itself fires with `"Unused '@ts-expect-error' directive"`. Brand `OrgId` properly (and update the `orgId` factory to cast accordingly) so the cross-call fails to type-check; once that line errors, the directive becomes valid and the diagnostic clears.

<TypeCoding
  instructions={`The \`OrgId\` type is declared as a bare \`string\`, so the compiler can't tell a \`UserId\` apart from an \`OrgId\`. The \`@ts-expect-error\` directive on the cross-call currently errors with "Unused '@ts-expect-error' directive" because the line below it compiles. Brand \`OrgId\` (and update its factory) so the cross-call fails to type-check, the directive becomes valid, and all the errors go away.`}

const userId = (value: string): UserId => value as UserId;
const orgId = (value: string): OrgId => value; // BUG: factory doesn't tag

declare function getOrgMembers(id: OrgId): Promise<unknown>;

declare const someUserId: UserId;
declare const someOrgId: OrgId;

// The cross-call should fail to compile: a UserId is not an OrgId.
// @ts-expect-error
await getOrgMembers(someUserId);

// These should compile: matching brands.
await getOrgMembers(someOrgId);
await getUser(someUserId);
`}
/>

<details>
<summary>Reveal the reference solution</summary>

```ts
type OrgId = string & { readonly __brand: 'OrgId' };

const orgId = (value: string): OrgId => value as OrgId;
```

Branding `OrgId` makes it structurally distinct from `UserId` — the literal brand labels differ, so the two types are non-assignable even though both erase to `string` at runtime. The cross-call `getOrgMembers(someUserId)` now fails to type-check (`Argument of type 'UserId' is not assignable to parameter of type 'OrgId'`), the `@ts-expect-error` directive is valid (no longer "unused"), and the diagnostic clears.

</details>

The directive's behavior is the lesson's mechanic, not an aside. `@ts-expect-error` is the natural TypeScript idiom for "this line should fail to compile" — it complains when the line below doesn't error, and goes quiet when it does. The starter has the directive in place but the brand is missing; you fix the brand, the line errors, the directive becomes valid.
