You've spent the last few lessons modeling shapes that prevent bug classes â€” discriminated unions for impossible states, transitions for valid moves, brands for IDs. Every one of those patterns has the same skeleton: a runtime value (a request state, an event, an ID) and a type that describes it. So far you've written both halves by hand. This lesson cuts that work in half.

Look at the bug class first.

The rule the lesson installs sits in one sentence: **if a hand-written type alias mirrors the keys, values, or structure of a runtime value, replace it with a <Term definition="A type computed from a runtime value via `typeof`, `keyof`, or indexed access â€” not hand-written.">derived type</Term> instead.** Two sources of truth drift; one source of truth doesn't.

Three operators do the lifting: `typeof V` lifts a value into the type register, `keyof T` reads the keys of a shape, and `T[K]` reads the value at a key. Composed, they give you the two idioms every 2026 SaaS codebase reaches for daily â€” `keyof typeof OBJ` for the keys of a typed config, and `typeof ARR[number]` for the elements of a frozen array. The lesson walks each operator on the same `ROUTES` value (so each new operator is a new lens on a familiar shape), promotes the two idioms, then ties everything together on a worked permissions example you'll copy into real code.

## Two registers, one keyword

Before any operator does work, you need a mental split that the rest of TypeScript depends on. TypeScript code lives in two parallel worlds.

The first is the <Term definition="An expression evaluated by the JavaScript runtime. Persists past compilation.">value-level</Term> register â€” the JavaScript your code compiles down to. Variables, function calls, object literals, `if` statements: everything that runs at runtime. This is the part of TypeScript that *isn't* TypeScript at all; it's the JavaScript the engine executes.

The second is the <Term definition="An expression evaluated by the TypeScript compiler at compile time. Erased before the JavaScript runs.">type-level</Term> register â€” the expressions the TypeScript compiler reasons about during type-checking. `type Foo = string`, `keyof T`, `Partial<User>`, `Awaited<Promise<X>>`: all of it lives in this register and **erases** before the runtime ever sees it. There is no `keyof` at runtime; the keyword is meaningless to the engine.

Most operators belong to one register exclusively. `if`, `for`, `+`, `JSON.parse` â€” value-level only. `keyof`, `extends` (in a constraint), `infer`, the indexed access `T[K]` â€” type-level only. The two registers can never see each other directly: type expressions can't run, and runtime expressions can't appear inside a type.

<TwoRegistersTable />

Here's the trap that wastes hours every time you forget it. **The keyword `typeof` exists in both registers â€” and it means different things in each.** Position determines which.

At the value level, `typeof x` is the runtime operator you've used for years: it returns a string like `'string'`, `'number'`, or `'object'`. You write it inside `if (typeof input === 'string')` to narrow a value. The previous chapter installed it as one of the narrowing tools.

At the type level â€” inside a `type` declaration, after a colon in an annotation, anywhere the compiler is reading a type expression â€” `typeof V` is something different entirely. It's an **extractor** that reads the inferred type off a value identifier and lifts it into the type register. `typeof ROUTES` here doesn't return `'object'`; it returns the actual `{ readonly home: '/'; readonly invoices: '/invoices'; ... }` shape, ready to be processed by other type-level operators.

Same keyword, two registers, two meanings. The compiler keeps them straight by looking at position; you keep them straight by knowing which world you're in. One mechanical rule helps: the type-level `typeof` requires a **value identifier** on the right, not an arbitrary expression. `typeof ROUTES` works; `typeof getRoutes()` does not â€” there's no value identifier to read from, just a call expression. When you reach for the latter, you actually want `ReturnType<typeof getRoutes>` (which we'll name in the next lesson).

## Lifting values into types with `typeof`

The first operator. Same `ROUTES` value from the introduction; one new lens on it.

```ts
const ROUTES = {
  home: '/',
  invoices: '/invoices',
  settings: '/settings',
} as const;

type Routes = typeof ROUTES;
```

What does `typeof ROUTES` evaluate to? The literal-typed shape of the value, with every property preserved at its narrowest:

<CodeTooltips tooltips={{
  'typeof ROUTES': "Resolves to { readonly home: '/'; readonly invoices: '/invoices'; readonly settings: '/settings' } â€” the exact shape of the ROUTES value, every property readonly, every value its literal type.",
}}>
```ts
type Routes = typeof ROUTES;
```
</CodeTooltips>

That resolved shape is the *output* of the operator. Read the operator as "take the inferred type of this value, hand me back the type expression that describes it." The value `ROUTES` lives at runtime; `typeof ROUTES` is the compile-time description of what's in `ROUTES`. From there, every other type-level operator in the lesson works on this description rather than on the value directly.

One dependency from the previous chapter matters here: **`as const` must come first**, then the derivation. <Term definition="Value-site freeze â€” every literal stays at its narrowest type, every property becomes `readonly`. Required to keep literal unions narrow when deriving types from values.">`as const`</Term> is what keeps the property values at their literal types instead of widening to `string`. Without it, `typeof ROUTES` would resolve to `{ home: string; invoices: string; settings: string }`, and every downstream derivation that pulls out the path values would degrade to `string` â€” useless for narrowing a switch or matching a URL. The rule is short:

:::tip
`as const` first, then derive. Without the freeze, the literal types you wanted are already gone before you start.
:::

One thing to file away: `typeof V` also composes with utility types â€” you'll see `ReturnType<typeof saveInvoice>` to pull the return type off a function value, `Parameters<typeof saveInvoice>` to pull its argument tuple, `Awaited<ReturnType<typeof fetchInvoices>>` to unwrap the resolved value of an async function. Those compositions are the next lesson's territory. For now, just know `typeof` is the bridge: it lifts a value into the type register, and other operators take it from there.

## Reading the keys with `keyof`

The second operator. Same `ROUTES` value; one new lens.

`keyof T` produces the **union of keys** of an object type `T`. If `T` has the keys `home`, `invoices`, and `settings`, then `keyof T` is the type `'home' | 'invoices' | 'settings'`. That's the whole operator â€” given a shape, hand me back the union of its property names as a literal union.

The reach is "I have a typed config, and I need its key names as a type." The two operators compose:

<AnnotatedCode lang="ts" code={`
const ROUTES = {
  home: '/',
  invoices: '/invoices',
  settings: '/settings',
} as const;

type RouteName = keyof typeof ROUTES;
`}>
  <AnnotatedStep meta={`"typeof ROUTES"`} color="blue">
    **Inside first: lift the value.** `typeof ROUTES` reads the inferred type off the value and produces `{ readonly home: '/'; readonly invoices: '/invoices'; readonly settings: '/settings' }`. This is the shape `keyof` will read keys from.
  </AnnotatedStep>

  <AnnotatedStep meta={`"keyof typeof ROUTES"`} color="green">
    **Outside next: pluck the keys.** `keyof` walks the shape and returns the union of property names: `'home' | 'invoices' | 'settings'`. Read the compound expression inside-out â€” `typeof` lifts, `keyof` reads.
  </AnnotatedStep>
</AnnotatedCode>

This is the first of the two load-bearing 2026 idioms, and it earns a name: **`keyof typeof OBJ`**. Any time you've written `const FOO = { ... } as const` and you also want a type that's the union of `FOO`'s keys â€” `'a' | 'b' | 'c'` â€” the answer is `keyof typeof FOO`. Route names, permission keys, feature-flag identifiers, locale codes, plan tiers: every typed-config object that needs a matching literal-union type reaches for this idiom.

One mechanical note that prevents confusion later: `typeof` is *only* the lift. Once a shape is already in the type register â€” an `interface` you declared, a `type` alias, a type imported from a module â€” `keyof` operates on it directly with no `typeof` needed:

```ts
type User = { id: string; email: string; name: string };
type UserKey = keyof User;
//   ^? 'id' | 'email' | 'name'
```

No `typeof` here because `User` is already a type; nothing to lift. The senior reading is: if the right-hand side of `keyof` is a *value identifier* (something declared with `const`, `let`, or as a function), you need `typeof` in front of it. If it's already a type identifier, you don't.

## Reading values with indexed access

The third operator. Same `ROUTES` value; one final lens.

Indexed access mirrors the bracket-access syntax you already use on values, lifted into the type register. At the value level, `ROUTES['home']` is `'/'`. At the type level, `(typeof ROUTES)['home']` is also `'/'` â€” same key, same result, but now you're asking for the type at that key instead of the value at that key. The operator's full notation is `T[K]`: an expression of type `T` indexed by a key of type `K`, returning the type at that key.

The operator takes two forms, and the second is the one that unlocks the chapter.

<CodeTooltips tooltips={{
  "(typeof ROUTES)['home']": "Reads the type at a single key. Resolves to '/'. Reach for this when you need one specific value type â€” for example, the type of a single config field.",
  "(typeof ROUTES)[keyof typeof ROUTES]": "Reads the union of types across all keys. Resolves to '/' | '/invoices' | '/settings'. Reach for this when you need the full union of value types â€” every URL the routes map can produce, every value a config can hold.",
}}>
```ts
const ROUTES = {
  home: '/',
  invoices: '/invoices',
  settings: '/settings',
} as const;

type HomePath = (typeof ROUTES)['home'];
type RoutePath = (typeof ROUTES)[keyof typeof ROUTES];
```
</CodeTooltips>

The first form â€” `(typeof ROUTES)['home']` â€” is the literal lookup. You name a specific key, you get the type at that key. Useful when one slot is the answer.

The second form â€” `(typeof ROUTES)[keyof typeof ROUTES]` â€” is the union sweep. The key argument is itself a union (`'home' | 'invoices' | 'settings'`), so the result is the union of types at *every* one of those keys: `'/' | '/invoices' | '/settings'`. The compiler reads "give me the type at any of these keys" and hands back the type-union of all the answers. This is the form that gives you a literal union of *values* from a typed config â€” every URL the routes map can produce, every status a state machine can hold, every locale string an i18n config supports.

You now have three lenses on the same value. It's worth one figure to anchor them.

## Element types from arrays: `typeof ARR[number]`

The lesson's second load-bearing idiom â€” and the one whose syntax catches everyone off-guard the first time. Different shape, same principle: lift the value, then read inside it.

Start with the bug class. You have a list of supported locales:

```ts
const LOCALES = ['en', 'es', 'fr'];

type Locale = (typeof LOCALES)[number];
//   ^? string
```

`Locale` is `string`. Not `'en' | 'es' | 'fr'`. Useless for narrowing a switch. The reason is the one you saw in the introduction: without `as const`, the array's inferred type is `string[]`, and a `string[]` indexed by `number` returns... `string`. The literals you wrote are already gone before the derivation runs.

Freeze the value first, then derive:

```ts
const LOCALES = ['en', 'es', 'fr'] as const;

type Locale = (typeof LOCALES)[number];
//   ^? 'en' | 'es' | 'fr'
```

Now `LOCALES` is a <Term definition="An array type with a fixed length and per-position element types. `readonly ['en', 'es', 'fr']` is a tuple of three string literals.">tuple</Term> â€” `readonly ['en', 'es', 'fr']` â€” and indexing it by the type `number` gives the union of *all* possible element types: `'en' | 'es' | 'fr'`. That's exactly what you wanted.

Two pieces of that expression deserve a closer look, because the syntax is non-obvious until you see what it's actually doing.

The first piece is `[number]` as an index. This is **not** indexing the array at position 0, 1, or 2. It's not a runtime expression at all â€” it's type-level indexed access (`T[K]`) where the *key type* is `number`. The question you're asking is: "for any numeric index into this tuple, what type lives there?" And the answer is the union of every type that lives at any numeric position â€” `'en' | 'es' | 'fr'`. It's the same `T[K]` operator from the previous section, just with a union-typed key instead of a literal one. No new magic.

The second piece is the trigger: any time you have a value list that should double as a literal-union type â€” locale codes, permission strings, plan tiers, status enums, MIME types your uploader accepts â€” declare it with `as const` and derive the union with `typeof ARR[number]`. The pair is the senior reflex. Even if today the value list is the only thing you need, leave the type alias derived; the moment someone wants the union, it's already there.

```ts
const ROLES = ['member', 'admin', 'owner'] as const;
type Role = (typeof ROLES)[number];
//   ^? 'member' | 'admin' | 'owner'

const PLAN_TIERS = ['free', 'pro', 'enterprise'] as const;
type PlanTier = (typeof PLAN_TIERS)[number];
//   ^? 'free' | 'pro' | 'enterprise'
```

You'll see this exact shape again in the i18n chapter (locales), the RBAC chapter (role names), and the billing chapter (plan tiers). Three units, one idiom.

:::note
The Twoslash `// ^?` comments above are a convention the lesson uses to show what the compiler resolves a type to â€” they're documentation here, but the editor in the exercise below renders them as live queries you can experiment with.
:::

## Putting it together â€” a permissions config

Time to fuse all three operators on a non-trivial shape. The permissions table below is the artifact you'll actually copy into the RBAC unit, and the derivations below it are the chain the rest of this lesson exists to teach.

Start by deriving the role names from a frozen list â€” the pattern you just learned:

```ts
const ROLES = ['member', 'admin', 'owner'] as const;
type Role = (typeof ROLES)[number];
//   ^? 'member' | 'admin' | 'owner'
```

Now the permissions table itself. The keys are permission strings; the values are arrays of which roles may exercise each permission. The <Term definition="Validates against the contract while keeping every literal narrow â€” the as const satisfies T pattern from the previous chapter.">`as const satisfies`</Term> pattern from the previous chapter does both jobs at once: validates the shape against the `Record<string, readonly Role[]>` contract, *and* keeps every literal narrow for the derivations to read:

```ts
const PERMISSIONS = {
  'invoice:read': ['member', 'admin'],
  'invoice:write': ['admin'],
  'org:billing': ['owner'],
} as const satisfies Record<string, readonly Role[]>;
```

Three derivations turn that one value into a fully typed permissions API. Walk them in order:

<AnnotatedCode lang="ts" code={`
const ROLES = ['member', 'admin', 'owner'] as const;
type Role = (typeof ROLES)[number];

const PERMISSIONS = {
  'invoice:read': ['member', 'admin'],
  'invoice:write': ['admin'],
  'org:billing': ['owner'],
} as const satisfies Record<string, readonly Role[]>;

type Permission = keyof typeof PERMISSIONS;
type AllowedRole = (typeof PERMISSIONS)[Permission][number];
`}>
  <AnnotatedStep meta="{1-2}" color="green">
    **`Role` â€” the union of role names, lifted from the value list.** `typeof ROLES` lifts the `['member', 'admin', 'owner']` tuple into the type register; `[number]` reads the union of element types. The result is `'member' | 'admin' | 'owner'`, ready to constrain both the `PERMISSIONS` value's contract and the API below.
  </AnnotatedStep>

  <AnnotatedStep meta="{10}" color="blue">
    **`Permission` â€” the keys of the permissions table.** `keyof typeof PERMISSIONS` produces `'invoice:read' | 'invoice:write' | 'org:billing'`. Adding a new permission to the value automatically extends this union.
  </AnnotatedStep>

  <AnnotatedStep meta="{11}" color="orange">
    **`AllowedRole` â€” every role that appears in any permission.** Two-step lookup. `(typeof PERMISSIONS)[Permission]` indexes the table by the union of all permission keys, producing `readonly ['member', 'admin'] | readonly ['admin'] | readonly ['owner']` â€” the union of role arrays. The trailing `[number]` then reads the element type of each, collapsing to `'member' | 'admin' | 'owner'`. Renaming a role in `ROLES` fails to compile at the `PERMISSIONS` literal until you update every reference.
  </AnnotatedStep>
</AnnotatedCode>

The payoff is the contract you've now wired. Adding a new permission to the `PERMISSIONS` value updates `Permission` for free â€” every consumer of that union immediately sees the new key. Adding a role to an existing permission updates `AllowedRole` for free. Renaming a role in `ROLES` doesn't compile until the `PERMISSIONS` literal is updated, because the `as const satisfies` contract anchors the role names to the source list.

You can now type the API around this without ceremony:

```ts
const hasPermission = (role: Role, permission: Permission): boolean => {
  const allowedRoles: readonly Role[] = PERMISSIONS[permission];
  return allowedRoles.includes(role);
};
```

Read what just happened: the function's argument types come from the value. The compiler refuses `hasPermission('bogus', 'invoice:read')` because `'bogus'` isn't a `Role`. It refuses `hasPermission('admin', 'fake:permission')` because `'fake:permission'` isn't a `Permission`. The compiler â€” not the team's discipline â€” is now the maintenance burden for keeping this surface coherent.

The intermediate `allowedRoles` binding widens the per-permission tuple (`readonly ['admin']` for `'invoice:write'`) back up to `readonly Role[]` â€” the contract type. That widen is what lets `.includes(role)` accept any `Role`, not just the literals at that specific permission. The two-step read is the senior shape; the alternative (`includes(role as never)`) compiles but hides the widening from anyone reading the function.

You'll see this exact pattern again when the RBAC unit lands. The shape stays; the only thing that grows is the `PERMISSIONS` value itself.

## The reverse trap

One short rule prevents the most common misconception about derivation.

**You can derive a type from a value, but you cannot derive a value from a type.** Types are erased at compile time â€” there is nothing for `keyof T` to read at runtime, nothing for `typeof V` to produce when there's no value, nothing for `T[K]` to walk. Types are descriptions; values are the thing being described. The arrow only goes one way.

That has a senior consequence that decides every "do I declare the type or the value first?" question: **when you need both a runtime list and a compile-time union, the value is the source.** The list lives in code as an `as const` array or object; the type is derived from it. Never hand-write both.

<CodeVariants>
  <CodeVariant label="Anti-pattern â€” type declared, value listed separately">
    <div data-mark-color="red">

    ```ts {1,3}
    type Locale = 'en' | 'es' | 'fr';

    const LOCALES: Locale[] = ['en', 'es', 'fr'];
    ```

    </div>
    Both the type and the value list are hand-written. Adding `'de'` to the type and forgetting the array makes `LOCALES.includes(userLocale)` lie; adding it to the array and forgetting the type lets the literal union admit the same drift. Adding a new locale means updating both â€” and the compiler doesn't catch the case where you update one and forget the other. The two sides drift the moment one diverges.
  </CodeVariant>

  <CodeVariant label="Senior form â€” value declared, type derived">
    <div data-mark-color="green">

    ```ts {1,3}
    const LOCALES = ['en', 'es', 'fr'] as const;

    type Locale = (typeof LOCALES)[number];
    ```

    </div>
    The value is the source. The type reads off the value. Adding `'de'` to `LOCALES` extends `Locale` for free; removing one shrinks the union and breaks every consumer that still reads the missing variant. Editing the array updates the type automatically; the two can't drift because there's only one side to edit.
  </CodeVariant>
</CodeVariants>

There is one carve-out worth naming so you don't over-apply the rule. When the type comes from an **external** schema â€” Stripe's `Subscription.status` union, a third-party SDK's enum â€” and your codebase doesn't own a value list that mirrors it, hand-writing `type Status = 'active' | 'past_due' | ...` standalone is fine. The bug class fires when *both* the value list and the type alias live in your codebase and disagree. If only one of them is yours, derivation has nothing to derive from; you're back to writing the part you own.

## Where the rest of the toolbox plugs in

You've installed the three operators and the two idioms. The next lesson sits one layer above this one â€” utility types like `ReturnType`, `Parameters`, `Awaited`, `Partial`, `Pick`, and the rest of the daily-reach surface. They compose with `typeof` to pull function-shape data straight off a function value: `ReturnType<typeof saveInvoice>` gives you the return type without restating it; `Parameters<typeof saveInvoice>[0]` gives you the first argument's type; `Awaited<ReturnType<typeof fetchInvoices>>` unwraps the resolved value of an async function. **The next lesson owns the utility-type depth.** This one names the seam.

The principle also shows up across the SaaS stack as the default for typed surfaces â€” three places you'll see it in coming units:

- **Drizzle.** `typeof invoices.$inferSelect` derives the row type from the schema value. The schema is the source; the type is the derivation. You'll see this when the database unit lands.
- **Zod.** `z.infer<typeof invoiceSchema>` derives the validated type from the schema value. Same principle, different value source.
- **Next.js.** Route segment params derive their type from the route's `params` schema. Same arrow: value-side first, type-side derived.

The pattern you just installed isn't a niche TypeScript trick â€” it's the default shape every modern data layer and framework reaches for when it needs a type to track a value. The next lesson ships the utility chains that compose on top of it.

<summary>Reveal the reference solution</summary>

```ts
type Role = (typeof ROLES)[number];
type Permission = keyof typeof PERMISSIONS;
type AllowedRole = (typeof PERMISSIONS)[Permission][number];

const hasPermission = (role: Role, permission: Permission): boolean => {
  const allowedRoles: readonly Role[] = PERMISSIONS[permission];
  return allowedRoles.includes(role);
};
```

`Role` lifts the `ROLES` tuple and reads the element union with `[number]`. `Permission` reads the keys of the `PERMISSIONS` value. `AllowedRole` indexes the table at every permission key (returning the union of role arrays), then reads each array's element type with a second `[number]`. The `hasPermission` signature consumes the two unions directly â€” `'bogus'` isn't a `Role`, so the bogus call now fails to compile and the `@ts-expect-error` directive above it becomes valid.

</details>

If the three type derivations clicked without ceremony, the lesson has landed. The signature wiring is the small payoff: a function whose argument types come from the same values its body operates on, no parallel hand-maintained list to keep in sync.
