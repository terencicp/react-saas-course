You now write discriminated unions whose shape is watertight (the previous two lessons installed the variants and the transition functions between them). The variants exist, and the moves between them are typed. Good.

There's still a crack — and it doesn't live in the union. It lives in the **consumers**. The code that reads a discriminated union — a `switch`, a router, an event handler — has no compile-time link between "which variants exist" and "which variants are handled." Add a fifth variant to the union; the existing `switch` over the four old variants keeps compiling. The new variant slides through the `default` branch (a `return null`, a `console.warn`, a logged-and-ignored payload) and nobody notices until production telemetry flags the gap.

Picture a webhook handler. It receives event payloads typed as a four-variant union — `'user.created'`, `'invoice.paid'`, `'subscription.canceled'`, `'session.revoked'` — and dispatches on `event.type`. Six months in, a teammate ships a new event type, `'invoice.refunded'`, and updates the producer side. The consumer's `switch` keeps compiling because TypeScript doesn't track "every variant has a case." Refunds go unhandled for two weeks. Customer service finds the bug; the team finds the silent `default` branch.

<div data-mark-color="orange">

```ts {13}
type AppEvent =
  | { type: 'user.created'; userId: string }
  | { type: 'invoice.paid'; invoiceId: string; amount: number }
  | { type: 'subscription.canceled'; subscriptionId: string }
  | { type: 'session.revoked'; sessionId: string };

const handle = (event: AppEvent): void => {
  switch (event.type) {
    case 'user.created':          return notifyUser(event.userId);
    case 'invoice.paid':          return recordPayment(event.invoiceId, event.amount);
    case 'subscription.canceled': return cleanupSubscription(event.subscriptionId);
    case 'session.revoked':       return revokeSession(event.sessionId);
    default:                      return; // adding a fifth variant lands here, silently
  }
};
```

</div>

The fix is structural. The compiler should refuse to build the missing handler — not warn, not suggest, refuse. This lesson installs the move: end every `switch` on a discriminant with something that turns "missing variant" into a compile error. After that, the lesson installs two narrowing helpers — **type predicates** and **assertion functions** — that the rest of the course relies on for narrowing `unknown` from the wire and filtering arrays cleanly.

## The `never` floor

The whole enforcement mechanic rests on one type. <Term definition="The bottom type — no value is assignable to `never`. Used to mark unreachable code and to enforce exhaustiveness on discriminated unions.">`never`</Term> is TypeScript's **bottom type** — the type with no inhabitants. No value is assignable to `never`. A variable typed `never` cannot hold a string, a number, an object, or anything else. It is the bottom of the type lattice, and that emptiness is exactly what makes it useful here.

Here's the mechanic. Inside a `switch` on the discriminant of a discriminated union, the compiler narrows the value one variant at a time. After the first `case` returns, the remaining type loses that variant. After the second `case` returns, it loses another. When every variant has been handled, the value at the `default` branch has nothing left in its type — the compiler narrows it down to `never`.

```ts
const handle = (event: AppEvent): void => {
  switch (event.type) {
    case 'user.created':          return notifyUser(event.userId);
    case 'invoice.paid':          return recordPayment(event.invoiceId, event.amount);
    case 'subscription.canceled': return cleanupSubscription(event.subscriptionId);
    case 'session.revoked':       return revokeSession(event.sessionId);
    default:
      const remaining = event;
      //    ^? const remaining: never
  }
};
```

Now remove one of the cases. The compiler can no longer eliminate that variant, so the value at the `default` is the missed variant itself — not `never`.

<div data-mark-color="red">

```ts {6, 10}
const handle = (event: AppEvent): void => {
  switch (event.type) {
    case 'user.created':          return notifyUser(event.userId);
    case 'invoice.paid':          return recordPayment(event.invoiceId, event.amount);
    case 'subscription.canceled': return cleanupSubscription(event.subscriptionId);
    // case 'session.revoked':    handled? no.
    default:
      const remaining = event;
      //    ^?
      //    const remaining: { type: 'session.revoked'; sessionId: string }
  }
};
```

</div>

That's the whole observation. **Any time the type at the bottom of a `switch` is something other than `never`, a variant was missed.** The enforcement idea falls out of it: pass that bottom value to a function whose parameter is typed `never`. When every variant has been handled and the bottom is `never`, the call compiles. When a variant is missed and the bottom is the missed variant, the call fails — because the missed variant is not assignable to `never`. The compile error names the missing variant by its full type.

The rest of the lesson is two surface shapes for that one mechanic.

## `assertNever` — the default

The helper-function form. The whole thing is three lines:

```ts
export function assertNever(value: never): never {
  throw new Error(`Unhandled variant: ${JSON.stringify(value)}`);
}
```

Note the `function` keyword — a deliberate break from the chapter's arrow-function default. The code conventions name **type-guard signatures** as one of the carve-outs from that default, and `assertNever` rides the same carve-out because its **return type `never`** is the point of the signature. Anywhere the signature itself is the contract, the `function` form is the reach. The file lives at `lib/assert-never.ts` — one export per file, kebab-case filename matching the export.

The shape at the call site:

<AnnotatedCode lang="ts" code={`
import { assertNever } from '@/lib/assert-never';

type AppEvent =
  | { type: 'user.created'; userId: string }
  | { type: 'invoice.paid'; invoiceId: string; amount: number }
  | { type: 'subscription.canceled'; subscriptionId: string }
  | { type: 'session.revoked'; sessionId: string };

const handle = (event: AppEvent): void => {
  switch (event.type) {
    case 'user.created':          return notifyUser(event.userId);
    case 'invoice.paid':          return recordPayment(event.invoiceId, event.amount);
    case 'subscription.canceled': return cleanupSubscription(event.subscriptionId);
    case 'session.revoked':       return revokeSession(event.sessionId);
    default:                      return assertNever(event);
  }
};
`}>
  <AnnotatedStep meta="{11-14}" color="blue">
    The four handled cases. Each variant gets exactly one `case`; the handler bodies are elided — the point is that each case terminates (`return`) so the compiler can eliminate that variant from the remaining type as the `switch` walks down.
  </AnnotatedStep>

  <AnnotatedStep meta="{15}" color="green">
    The exhaustiveness check lives on this one line. `assertNever`'s parameter is typed `never`; the call only type-checks when the bottom of the `switch` has been narrowed to `never` — which happens exactly when every variant above has been handled.
  </AnnotatedStep>

  <AnnotatedStep meta="{15}" color="red">
    If a `case` is removed (or a new variant lands without a corresponding case), `event` at the `default` is no longer `never` — it's the missed variant, and the TypeScript error reads `Argument of type '{ type: "session.revoked"; ... }' is not assignable to parameter of type 'never'`. The diagnostic names the missed variant by its full type, so the on-call engineer can walk straight to the missing case.
  </AnnotatedStep>
</AnnotatedCode>

Two things are happening on that one `default` line, and both matter. The first is the compile-time check just described. The second is a runtime safety net: `assertNever` throws. If the consumer is somehow reached at runtime with an unhandled variant — a webhook from a future API version, a manually crafted JSON body, a database row populated by a deprecated codepath — the function throws with the offending value in the error message. **The compile-time check and the runtime throw are two channels of the same value, and both are part of the contract.**

The runtime throw is loud on purpose. The reflex an experienced engineer brings is: if the consumer is reached with an unhandled variant, the team needs the error in their logs immediately, not a swallowed `console.warn`. Silence at this layer is how missing-refund bugs survive for two weeks. The `Error` carries the offending value via `JSON.stringify` so whoever's on call doesn't have to chase a missing field across three log lines.

## `satisfies never` — when the runtime throw isn't needed

There's a second surface for the same mechanic. Since TypeScript 4.9, the `satisfies` operator (which the previous chapter named) lets you write the check inline without a helper:

```ts
const handle = (event: AppEvent): void => {
  switch (event.type) {
    case 'user.created':          return notifyUser(event.userId);
    case 'invoice.paid':          return recordPayment(event.invoiceId, event.amount);
    case 'subscription.canceled': return cleanupSubscription(event.subscriptionId);
    case 'session.revoked':       return revokeSession(event.sessionId);
    default:                      event satisfies never;
  }
};
```

Recall what `satisfies` does: it checks that the value on the left is assignable to the type on the right, **without** widening the value's inferred type. `event satisfies never` compiles iff `event`'s narrowed type at that point is `never`. Same enforcement as `assertNever`; the line erases at compile time, so there's **zero** runtime cost — no function call, no throw, nothing in the emitted JavaScript.

The senior question is when to reach for each. The choice is about what happens if the bottom is somehow reached at runtime.

:::tip
| Reach | Use when |
| --- | --- |
| `assertNever(event)` | The consumer is a real handler — a webhook router, a reducer, a render dispatch — and a bad payload reaching the `default` should fail loudly. The runtime throw is part of the contract. |
| `event satisfies never` | The bottom is **structurally unreachable** and the compile-time check alone is the point. The function returns before the switch ends, the switch lives inside a pure type-narrowing helper, or it's a test-fixture branch-coverage check. |
:::

The course default is `assertNever`. Production handlers want the loud throw — silence at the bottom of a real consumer is how runtime drift goes unnoticed. Reach for `satisfies never` when the bottom genuinely can't be hit; everywhere else, `assertNever`.

## `noFallthroughCasesInSwitch` — the strict-flag companion

There's one more enforcement that composes with `assertNever`, and you don't have to do anything to enable it — the course's `tsconfig.json` ships with `noFallthroughCasesInSwitch` on by default. The flag turns a `case` body that lacks an explicit `break`, `return`, `throw`, or `continue` into a compile error. So a `case` that forgets to terminate doesn't silently fall through to the next case's body.

The two enforcements are independent and they compose. `noFallthroughCasesInSwitch` makes every `case` body unambiguous; `assertNever` makes every `switch` exhaustive. Together they close both gaps — a `switch` becomes both watertight at the case level and watertight at the variant-set level. You'll see this error fire occasionally during refactors; now you know what it's saying.

## The lookup-map alternative

The `switch + assertNever` shape is the right reach when each case has nontrivial logic — a few lines, a branch, a call to a helper. But a lot of dispatch is simpler than that: each variant maps to a one-shot handler, no branching, no inline logic. For that case, a `Record` indexed by the variant literal reads cleaner than a `switch` and gives you the same exhaustiveness guarantee for free.

```ts
type EventType = AppEvent['type'];

const HANDLERS: Record<EventType, (event: AppEvent) => Promise<void>> = {
  'user.created':          handleUserCreated,
  'invoice.paid':          handleInvoicePaid,
  'subscription.canceled': handleSubscriptionCanceled,
  'session.revoked':       handleSessionRevoked,
};

await HANDLERS[event.type](event);
```

The `Record<EventType, Handler>` form requires **every** key in the union to have a handler — that's the same rule a literal-keyed `Record` carries from the previous chapter. Add `'invoice.refunded'` to `AppEvent` and the `HANDLERS` constant fails to type-check at the literal site, on the object that's missing the key. Same enforcement as `assertNever`, expressed as a completeness constraint on the record's keys instead of a bottom-of-switch check.

(The `AppEvent['type']` form is **indexed access** — reading the type of the `type` field straight off `AppEvent`. A later lesson in this chapter, "Derive types from values," installs the derivation operators properly; here it's used directly because it's the most readable way to name the discriminant union.)

The trigger to reach for one or the other:

- **`switch + assertNever`** when each case has inline logic — a few lines, a conditional, anything that wants its own block. The case body is local to the discriminant check, which reads well.
- **`Record<Variant, Handler>`** when each case is a one-to-one handler reference. The lookup table is shorter, less ceremony, and the keys document the variant set at a glance.

## Type predicates — block-scoped narrowing

The chapter so far has assumed values that already carry a type — a discriminated union you wrote, a state machine you control. The other half of working with types is values that **don't** carry a type yet — `unknown` from `await req.json()`, mixed arrays from a query that returns multiple shapes, any value that crossed a boundary without a parser. Two functions narrow these values into the typed code that follows. This is the first one.

A <Term definition="Function whose return type `value is T` tells the compiler that when the function returns `true`, the argument is `T` inside the if-block where the call appears.">type predicate</Term> is a function whose return type is `value is T`. The signature tells the compiler: "when this function returns `true`, treat the argument as `T` inside the conditional block where the call appears." The narrow lives **inside the if-block** — once execution leaves the block, the type widens back. That's the **block-scoped** rule, and it's what differentiates this helper from the next one.

The canonical shape:

```ts
function isUser(value: unknown): value is User {
  return typeof value === 'object'
    && value !== null
    && 'id' in value
    && 'email' in value;
}
```

Note the `function` keyword again — same carve-out as `assertNever`. Type predicates are the canonical case the rule was written for: the `value is User` return type is the function's whole contract, so the signature wants the form that makes it most legible.

Two uses dominate the rest of the course — narrowing a mixed-array down to one variant via `.filter`, and narrowing an `unknown` value from the wire before passing it into typed code. Walk through both in one block.

<AnnotatedCode lang="ts" code={`
type User = { id: string; email: string };
type Guest = { sessionId: string };

function isUser(value: unknown): value is User {
  return typeof value === 'object'
    && value !== null
    && 'id' in value
    && 'email' in value;
}

declare const allMembers: (User | Guest)[];
const users = allMembers.filter(isUser);
//    ^? const users: User[]

declare const maybeUsers: (User | undefined)[];
const present = maybeUsers.filter((u) => u !== undefined);
//    ^? const present: User[]

declare const payload: unknown;
if (isUser(payload)) {
  await sendWelcome(payload);
  //                ^? (parameter) payload: User
}
`}>
  <AnnotatedStep meta={`"value is User"`} color="blue">
    The signature is the type predicate's whole API surface — the `value is User` return type is what tells the compiler the function refines its argument when it returns `true`.
  </AnnotatedStep>

  <AnnotatedStep meta="{12-13}" color="green">
    The named-predicate filter. The array narrows from `(User | Guest)[]` to `User[]` because `.filter`'s typing recognizes the callback is a type predicate and uses its refined type for the result.
  </AnnotatedStep>

  <AnnotatedStep meta="{16-17}" color="violet">
    The TypeScript 5.5 inferred form. Since TypeScript 5.5 (June 2024) the compiler infers `u is User` automatically when an inline arrow callback's body is a simple refinement — no helper function needed for one-shot inline filters.
  </AnnotatedStep>

  <AnnotatedStep meta="{20-23}" color="orange">
    Narrowing `unknown` from the wire. Inside the `if`-block, `payload` is `User` and typed code can read its fields; outside the block, it's still `unknown`. That scope rule is the block-scoped narrow named above.
  </AnnotatedStep>
</AnnotatedCode>

The TypeScript 5.5 inferred form deserves a separate sentence. Before 5.5, narrowing an array with an inline arrow required either writing a named type predicate or using a type assertion at the call site. Since 5.5, when an inline arrow's body is a simple refinement of the parameter — `u !== undefined`, `typeof u === 'string'`, an `in`-check — the compiler infers the predicate without you writing it. **For one-shot inline filters whose body is a simple refinement, the inferred form is the right reach.** No helper, no annotation, the array narrows correctly.

For **named, reusable, multi-condition predicates** — anything you'd put in `lib/` and import from multiple call sites — the explicit `value is T` form is still the senior default. The signature documents the predicate's intent; the body can grow without losing the narrow.

A word on the body of `isUser` above. The chain `typeof value === 'object' && value !== null && 'id' in value && 'email' in value` is illustrative — it's the simplest body that shows the predicate's signature in action. It's also fragile: it doesn't check that `id` and `email` are strings, it drifts the moment `User` grows a field, and it can't express refinement constraints (`email` must look like an email, `id` must look like a UUID). **In production, the body is a Zod parse**, not a hand-written `in` chain. A schema-authoring chapter later in the course owns the schemas themselves; here, the signature is the lesson, not the body.

## Assertion functions — scope-wide narrowing

The second narrowing helper. Same idea — a function that narrows a value — different scope.

An <Term definition="Function whose return type `asserts value is T` tells the compiler that after the call returns (without throwing), the argument is `T` for the rest of the scope.">assertion function</Term> has the return type `asserts value is T`. The signature tells the compiler: "if this function returns (without throwing), the argument is `T` for the **rest of the scope**" — not just inside an if-block, the whole rest of the function. The narrow is **scope-wide**.

The canonical shape:

```ts
function assertIsUser(value: unknown): asserts value is User {
  if (!isUser(value)) {
    throw new Error(`Expected User, got: ${JSON.stringify(value)}`);
  }
}
```

The structural difference between the two helpers is one word: **predicates return a boolean and let the caller branch on it; assertions throw and let the caller continue with the narrowed type.** Same `function`-keyword carve-out applies for the same reason.

Read them side by side — the difference between block-scoped and scope-wide is what each call site looks like.

Two uses dominate where the assertion form earns its weight.

The first is **parse-or-throw at a service boundary**. A function reads `unknown` from a third-party SDK, asserts the shape, and continues with the narrowed value. No nested if-block, no rightward drift — the rest of the function reads as if the value was already typed:

```ts
const raw: unknown = await stripe.invoices.retrieve(id);
assertIsInvoice(raw);
return raw;
```

The second is **test fixtures and harnesses**. A custom helper like `expectUser(value)` asserts inside a test and lets the rest of the test access the narrowed value without an `if` wrapper around every assertion. The scope-wide narrow is what makes test bodies readable.

The choice between the two helpers is one question: do you want the caller to handle the false case, or do you want the call site to fail fast? Predicates hand control to the caller. Assertions take it away. Production parse boundaries usually want the assertion form because the upstream wire shape is a contract — a violation is a bug, not a branch you handle.

## Don't hand-roll the runtime check

One reminder before the exercises, because the lesson's examples could leave the wrong impression.

The bodies shown above — the `'id' in value && 'email' in value` chain inside `isUser` — are illustrative. **Production code does not author runtime validators by hand.** Hand-written chains drift from the type the moment the type grows a field, miss edge cases (nested optionals, discriminated unions, refinement constraints), and duplicate the type information that's already encoded in the schema.

The senior reflex: the schema is the source of truth. Author the schema in **Zod** once, derive the TypeScript type from `z.infer<typeof schema>`, and build the type predicate or assertion function on top of `schema.safeParse(value).success`. A later unit owns the schema authoring and the runtime/type round-trip; what this lesson installs is the **signature** the schema gets wired into, not the parser's body.
