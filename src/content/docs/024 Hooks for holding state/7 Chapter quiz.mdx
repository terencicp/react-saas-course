  <MultipleChoice src="24.1">
    A component seeds its state by reading and parsing a draft out of `localStorage`. Which call does that read-and-parse work exactly once, on mount?

    <McqChoice>
      ```tsx
      useState(parseDraft(getStoredDraft()))
      ```
    </McqChoice>
    <McqChoice correct>
      ```tsx
      useState(() => parseDraft(getStoredDraft()))
      ```
    </McqChoice>
    <McqChoice>
      ```tsx
      useState<Draft>(parseDraft(getStoredDraft()))
      ```
    </McqChoice>

    <McqWhy>Passing a *function* tells React to call it once at mount and ignore it afterward. The bare-call form runs `parseDraft(getStoredDraft())` on every render and throws all but the first result away — wasted work on every keystroke. The annotation only fixes types; it does nothing about *when* the work runs.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="24.5">
    You're holding the `setTimeout` ID that debounces a search input. Why is this a `useRef` and not a `useState`?

    <McqChoice correct>The JSX never reads the timer ID, so storing it in state would schedule a re-render that paints nothing new.</McqChoice>
    <McqChoice>A timer ID is a number, and `useState` can only hold objects.</McqChoice>
    <McqChoice>`useRef` survives re-renders while a `useState` value is reset to its initial value on each render.</McqChoice>

    <McqWhy>The deciding question is "does the JSX read it?" — no, so it's a ref. Writing `.current` never re-renders, which is exactly right for a value the user never sees. The type claim is false, and `useState` *also* survives renders — the difference is that updating it forces a render you don't want here.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="24.2">
    A junior keeps `fullName` in its own `useState` and resyncs it with a `useEffect` watching `firstName` and `lastName`. Beyond the extra render, what is the more dangerous problem with this shape?

    <McqChoice correct>`fullName` is a second copy of data that already exists, kept current by hand — add a `middleName` and forget the dependency array, and it silently goes stale with no error.</McqChoice>
    <McqChoice>String concatenation in an effect is too slow to run on every keystroke.</McqChoice>
    <McqChoice>`useEffect` can only depend on one value, so watching both names is unsupported.</McqChoice>

    <McqWhy>The real cost is the second source of truth maintained by a hand-written dependency list. A derived `const fullName = firstName + ' ' + lastName` can't drift because there's nothing to keep in sync. The concatenation is rounding error next to the render, and effects can depend on any number of values.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="24.3">
    A list view has a search filter the user types into. A teammate argues it should live in the URL as `?q=...` rather than in lifted `useState`. Which facts actually justify the URL home? **Select all that apply.**

    <McqChoice correct>After a refresh, the user expects their filter to still be applied.</McqChoice>
    <McqChoice correct>A shared link should land the recipient on the same filtered view.</McqChoice>
    <McqChoice>Two sibling components on screen both need to read the filter.</McqChoice>
    <McqChoice>The filtered rows are expensive to recompute on every render.</McqChoice>

    <McqWhy>The URL test is two questions: survive a refresh? shareable as a link? Either "yes" sends the value to the URL. "Two siblings need it" is the trigger for *lifting*, not for the URL — and recompute cost is a memoization concern, unrelated to where the source of truth lives.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="24.4">
    A submit form tracks `isSubmitting`, `isSaved`, `submitError`, and more across separate `useState`s, and a bug lets the spinner stay on after a validation failure. What is the core reason modelling this as a `useReducer` with a single `status` field fixes the class of bug?

    <McqChoice correct>Combinations like "submitting and saved at once" become unrepresentable — there's no value of `status` that means both — instead of merely discouraged.</McqChoice>
    <McqChoice>A reducer is always fewer lines of code than the equivalent `useState` calls.</McqChoice>
    <McqChoice>Putting the `await` inside the reducer guarantees the spinner is reset on every exit path.</McqChoice>

    <McqWhy>The win is collapsing several booleans into one `status` with a fixed set of legal values, so the illegal combinations can't be constructed. A reducer is often *more* code, not less — and async must never live in the reducer; it stays in the handler that brackets the `await` with dispatches.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="24.6">
    A teammate proposes using `crypto.randomUUID()` instead of `useId()` to generate the `id` that links a `<label>` to its `<input>`, reasoning it's "more unique." What goes wrong?

    <McqChoice correct>It rolls a different value on the server and in the browser, so the rendered HTML and hydration disagree and React warns about a mismatch.</McqChoice>
    <McqChoice>`crypto.randomUUID()` is too slow to call during render.</McqChoice>
    <McqChoice>UUIDs contain characters that are invalid in an HTML `id` attribute.</McqChoice>

    <McqWhy>`useId` is deterministic across the server boundary because it derives from the component's position in the tree; a random generator runs twice and produces two different strings, breaking hydration. The same `crypto.randomUUID()` is the *right* tool when minted once at item creation and stored on the data — just not for a render-time ARIA id.</McqWhy>
