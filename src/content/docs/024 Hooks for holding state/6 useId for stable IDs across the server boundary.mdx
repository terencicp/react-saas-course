Say you're building a `TextField` component — a `<label>` and an `<input>` together, the thing you'll drop into every form in the app:

```tsx
<TextField label="Email" />
```

For a screen reader to announce that field as "Email, edit text" instead of "edit text, blank", the label has to be *associated* with the input. In HTML that association is made by a shared string: the label carries <Term definition="The JSX spelling of the HTML for attribute — it names the id of the input this label describes.">`htmlFor="email"`</Term> and the input carries the matching `id="email"`. Same string on both, and the browser links them.

Here's the snag. `TextField` renders its own `<label>` and `<input>`, so `TextField` is the one that has to supply that id — and it has no idea what to use. Hardcode `id="email"` and the component works exactly once: the moment a form has two `TextField`s, you've got two `id="email"` on one page. Two elements sharing an id is invalid HTML, and the browser quietly picks one — so now both labels point at the same input, and the second field is unlabelled. The component needs a unique id it can't possibly know in advance.

You've spent this chapter deciding *where a value lives* — `useState`, the four homes, `useReducer`, a ref. This last hook isn't about holding a value at all. <Term definition="A React hook that returns a stable, unique id string, called with no arguments.">`useId`</Term> takes no arguments, returns one string, and exists to solve exactly this: handing a component a unique id to wire two DOM nodes together. The API is so small that the whole lesson is really about *why* it's a hook and not a one-liner — and about the two jobs people hand it that it's precisely the wrong tool for. By the end you'll wire a label, an input, and an error message together — correctly, and under server rendering — and you'll be able to say in one breath when *not* to touch it.

## Why you can't just make up an id

Strip the requirement down before reaching for any API. This id carries no meaning — nobody reads it, nothing depends on its contents. It only has to be **unique on the page** so two fields don't collide, and **stable enough to put on both nodes** so the label and input agree. That's it. It's plumbing.

So generate one. There are three obvious ways, and watching each fail is the whole point — the failures are what `useId` is shaped around.

The hardcoded literal is the obvious miss. It works for a single field and **collides** the moment the component is reused, which is the moment it gets useful. A reusable component cannot hardcode an id.

So make it unique per instance. `Math.random()` gives a different string every call, collision solved — except it introduces a worse bug, and to see it you need one fact about how a React page reaches the screen. Under <Term definition="SSR — the page is built to HTML on the server first, then sent to the browser.">server rendering</Term>, your components run *twice*. The server runs them once to produce the initial HTML the user sees immediately. Then the browser runs the same components again over that HTML, to attach event handlers and wire up state — a step called <Term definition="The browser re-running your components on top of the server's HTML to make it interactive; if the HTML the browser would produce does not match what the server sent, React warns.">hydration</Term>. The catch is that these two runs must agree: hydration checks that the HTML the browser produces matches what the server sent. `Math.random()` runs in both, and rolls a *different* number each time — so the server stamps `id="field-0.81"`, the browser computes `id="field-0.45"`, the attributes disagree, and React fires a hydration-mismatch warning. The field might still kind of work, or it might wire to the wrong node. Either way you've shipped a bug that never showed up while you were clicking around in dev.

`crypto.randomUUID()` is the move an experienced developer makes next, and it's worth naming precisely so you can kill the instinct. It produces a genuinely unique id, so surely it's the correct tool — but it fails for the *exact same reason* `Math.random()` did. It runs on the server, runs again in the browser, and hands back a different uuid each time. Same mismatch, plus you've reached for a cryptographic generator to label an input.

Step back and the three failures rhyme. Hardcoding isn't unique. Both generators are unique but *not the same on both sides of the server boundary*. What you need is something unique that the server and browser will independently compute to the **same** value — and they share exactly one thing that's guaranteed identical on both runs: the **shape of the React tree itself**. Same components, nested the same way, in the same order, server and client. An id derived from *where a component sits in that tree*, rather than from a random roll, comes out identical on both runs for free. That is precisely what `useId` does.

:::caution
Don't teach yourself to "fix" the random-id mismatch by moving the read into an effect or silencing the warning. The mismatch is the symptom telling you the value isn't deterministic across the boundary; the fix is a deterministic id, which is `useId`. Hydration mechanics get their own treatment when we reach the App Router.
:::

## useId: one string, identical on both sides

Here's the entire API.

<CodeTooltips tooltips={{
  useId: 'Returns a unique, stable id string, identical on the server and the client. Takes no arguments.',
  id: 'Opaque on purpose. Hand it to id / htmlFor / aria-*. Never parse it, pattern-match it, or select on it from CSS.',
}}>
```tsx
const id = useId();

<input id={id} />
```
</CodeTooltips>

No arguments. It returns one string, and three facts about that string carry everything:

- **Unique.** Each `TextField` instance on the page gets its own id, so two of them never collide.
- **Stable.** The same instance gets the *same* string on every re-render — it won't churn underneath you the way a fresh `Math.random()` would.
- **Identical on server and client.** This is the property the whole lesson turns on. The server-rendered HTML and the browser's hydration produce the same id, so there's no mismatch.

The string it returns is **opaque** — a short token like `«r1»`, format and all chosen by React. You never read it, parse it, slice a suffix off it for meaning, or write a CSS selector against it. It is a handle you pass to `id`, `htmlFor`, and the `aria-*` attributes, and nothing else.

Now the question the lesson has been circling: *why is this a hook at all, and not a plain `makeId()` function?* Because a plain function can't know where it is. A hook can. Recall from the render model that React calls your hooks in the same order on every render — that fixed call order is exactly how React keeps each `useState` matched to its own value across renders. `useId` rides the same machinery: it derives the id from the component's position in the render tree. And because the server and the browser walk the *same tree in the same order*, both arrive at the same position, and therefore the same id — no coordination, no shared random seed, no value passed across the wire. The determinism is a side effect of the tree being the tree.

<UseIdTreePosition />

One consequence to file away: because the id is tied to tree position, it is *not* permanent across a component's whole life. If a component unmounts and later remounts — for instance after a `key` reset of the kind from the last chapter — it can come back with a different id. That's harmless for ARIA wiring, which re-reads the current attributes anyway. It only surprises you if you stashed the id somewhere expecting it to live forever; don't.

`useId` works the same way under server rendering and inside Client Components — the id is generated during the render, serialised into the HTML, and matched again on hydration. The experienced-developer takeaway is the absence of a special case: you don't generate ids one way for server-rendered output and another way in the browser. The same `useId()` call is correct everywhere a component renders.

## Wiring a field: label, input, and error message

This is the pattern you'll actually copy into real code, so build the whole field.

The core is what you've already seen: one `useId()` call, its string on the input's `id`, the same string on the label's `htmlFor`. That single association is what turns "edit text, blank" into "Email, edit text" for a screen-reader user. It's also the difference between a form anyone can fill out and one that's quietly unusable for a slice of your customers.

A real field needs more than one id, though. There's the input itself, and there's usually an error message that has to be announced *as part of the field* when validation fails — wired with <Term definition="An ARIA attribute naming the id of an element that describes this one; a screen reader reads that element's text right after the field's name.">`aria-describedby`</Term>, which points at the id of the error element. You might want a hint line too. So you need several related ids, and the question is how to mint them.

The convention is to call `useId()` **once** and derive the rest by suffixing the base:

```tsx
const id = useId();
const errorId = `${id}-error`;
```

Both work — calling `useId()` twice for two ids is perfectly legal, and you'll see it in library code, so don't be thrown by it. But one call plus suffixes reads better: it says *these ids belong to the same field*, and you can see the relationship at a glance instead of tracking two independent calls. One field, one `useId`, suffixed ids.

The error node only exists when there's actually an error, so it's a conditional render. Guard it with a real boolean — `error != null && <p id={errorId}>…</p>` — the same discipline this chapter has used throughout, so an empty string or a `0` can never leak a stray node into the markup. And match the house standard for form fields while you're here: alongside `aria-describedby`, set `aria-invalid={error != null}` so assistive tech announces the field as invalid, not just that there's some text nearby.

That leaves one genuine judgement call. `aria-describedby` points at an id; when there's no error, that node isn't in the DOM, so the attribute references nothing. A dangling `aria-describedby` is harmless — screen readers ignore a missing target. So you can leave it always set and not lose sleep. The slightly cleaner move is to set it only when the error is present, so the attribute never points at a ghost. Both are defensible; the snippet below sets it conditionally so you've seen the tidy version.

<AnnotatedCode lang="tsx" maxLines={18} code={`
type TextFieldProps = {
  label: string;
  error?: string;
};

export const TextField = ({ label, error }: TextFieldProps) => {
  const id = useId();
  const errorId = \`\${id}-error\`;

  return (
    <div>
      <label htmlFor={id}>{label}</label>
      <input
        id={id}
        aria-invalid={error != null}
        aria-describedby={error != null ? errorId : undefined}
      />
      {error != null && <p id={errorId}>{error}</p>}
    </div>
  );
};
`}>
  <AnnotatedStep meta="{7-8}" color="blue">
    One call, two ids. The base goes on the input; `-error` is suffixed for the message. Calling `useId` again would also work, but suffixing keeps it visibly one field.
  </AnnotatedStep>

  <AnnotatedStep meta={`{12} "id={id}"`} color="green">

  </AnnotatedStep>

  <AnnotatedStep meta="{15-16}" color="green">
    When an error exists, the field is announced as invalid and its description points at the error text. No error, no description — `undefined` drops the attribute.

  <AnnotatedStep meta="{18}" color="green">

Notice what's *not* in that component: a single hardcoded id. Every id is generated and passed by reference. That's the reflex to build — wire two nodes by handing them the same generated string, never by typing a literal into two places and hoping they stay in sync.

Grading the *association* rather than a fixed id string is the lesson in miniature: there is no "right" id to type, only a right way to connect nodes — generate once, reference everywhere.

## Two things useId is not for

The API is small; the judgement is in knowing where it doesn't belong. Two misuses are close to universal, and both are conceptual, not stylistic — getting them wrong changes how your code behaves, not just how it reads.

**Not for list keys.** This is the one nearly everyone reaches for, so let it land hard. Recall the `key` contract from the previous chapter: a list key is *reconciliation identity*, the string React uses to track which rendered item corresponds to which across renders, so it can move and reuse DOM instead of rebuilding it. A key must come from the **data** — `item.id`, a `slug`, an id stamped on the row when it was created — and stay fixed for the life of that datum.

`useId` can't supply that, and not just as a matter of taste — it's structurally impossible. `useId` produces an id *for a component instance that already exists*. But a key has to identify an item *before* React has decided which instance to create or reuse — keying is the step that *chooses* the instances. You'd be asking React for the id of a thing it hasn't built yet, to decide whether to build it. The chicken can't lay an egg it hatched from. Keys come from your data; `useId` comes from the tree React assembles *after* keying.

So where do keys come from when the data has no natural id — a to-do the user just typed, with no server id yet? Mint a `crypto.randomUUID()` **at the moment you create the item, and store it on the item**. Note the contrast with the earlier section: `crypto.randomUUID()` was the *wrong* tool when called during render for an attribute, because it rolls fresh every render and across the server boundary. Called *once at creation* and saved onto the data, it's exactly right — now it's stable, it lives with the datum, and it survives every render after.

**Not for secrets or human-readable anchors.** `useId`'s output is stable-per-position, which is the opposite of secret. It is not random, not unguessable, and not unique across pages or sessions — load the page twice and you may get the same string. So it's the wrong tool for anything that needs cryptographic uniqueness: CSRF tokens, session ids, idempotency keys. Those come from a real generator on the server (`crypto.randomUUID()` and friends), and you'll meet them properly in the server chapters.

It's equally wrong for ids a *human* relies on — the `#pricing` anchor someone bookmarks and shares. Those have to be stable across deploys, readable, and hand-authored. An opaque, tree-derived token like `«r1»` is none of those.

Two more guardrails, both one-liners and both familiar:

- **Keep `useId` calls out of conditionals and loops** — the rule for every hook, foreshadowing the rules of hooks. The call order is what fixes the id to a tree position; if a `useId` call appears or disappears depending on a condition, its position can shift between the server and the browser, and you're back to a mismatch.
- **Don't style by these ids.** They're wiring tokens for `id` / `htmlFor` / `aria-*`, not selector targets. Reaching into CSS to match on a `useId` value is a smell regardless of what the format happens to be — style with classes, wire with `useId`.

## How libraries already do this

You will rarely write the `TextField` above from scratch, and that's the last thing worth knowing. The component libraries you'll reach for — shadcn/ui and the Radix primitives under it — call `useId` internally to wire ARIA on every input, label, and dialog they ship. When you later drop in a `<Label>` and an `<Input>` and they just announce correctly, this is the machinery underneath. The course's own form components, when you build them, reach for `useId` at every label-input pairing for the same reason.

The practical upshot lands the moment you wrap a third-party input in your own component: *your wrapper* is where the `useId` call goes, and it passes the id down to the inner element as a prop. The id is generated once, at the boundary you own, and handed inward — the same generate-once-reference-everywhere shape, one level up.
