You're building an invoices table. Every row gets a small "Copy link" button so the user can grab the invoice URL and drop it into an email without leaving the page. They click, the URL lands on their clipboard, and a "Copied" tick flashes for a couple of seconds so they know it worked. This is the most ordinary button in a SaaS app — you'll write a dozen variants of it across any real product.

So you write the obvious thing — `onClick={() => navigator.clipboard.writeText(url)}` — and ship it. It works on your machine. Then the support tickets start: for some users the button does nothing, for others it works only when the page is quick and fails when it's slow, and none of it reproduces locally. The call that looked like a one-liner has two invisible gates on it, and neither one trips during local development. **What gates this call, and why does the obvious version break in production but not in dev?** Hold that question; the whole lesson is the answer.

By the end you'll have the button that survives production: one async call, placed where the browser will actually honor it, wrapped so it degrades gracefully when it can't, and labelled so a screen reader announces the result. And you'll know precisely which two conditions decide whether the bytes ever reach the clipboard — so the next time you write this button, you write the correct version the first time.

## `navigator.clipboard.writeText` in one line

The entire write API is one method. `navigator.clipboard.writeText(text)` takes a plain-text string and writes it to the system clipboard — the same clipboard the user pastes from anywhere on their machine. It's asynchronous: it returns a Promise that resolves when the write succeeds and rejects when it fails.

```ts
await navigator.clipboard.writeText(invoiceUrl);
```

Two things to internalize before we go further. First, the write *replaces* the entire clipboard contents — there's no append, no history, no merge. Whatever was on the clipboard is gone, and the new string is all that's there. Second, `writeText` expects a string; hand it anything else and it silently coerces with `String(value)`, so passing an object lands the literal text `[object Object]` on the clipboard. Pass strings.

There's exactly one copy API you write in 2026, and this is it. If you search for "javascript copy to clipboard" you'll surface a pile of pre-2022 answers built on `document.execCommand('copy')` — the old dance of creating a hidden element, selecting its text, and calling an `exec` method synchronously. That API is deprecated and the synchronous selection behavior it leaned on is gone. Recognize it as the legacy pattern so you don't paste it into a 2026 codebase; never reach for it.

:::note
`writeText` is the whole happy path. The depth in this lesson isn't the call — it's the two conditions that have to be true *before* the call can succeed, and what you do when they aren't.
:::

## The two gates that decide whether the write lands

Before `writeText` can put anything on the clipboard, two things must be true. Miss either and the call fails — but they fail in two genuinely different ways, and telling them apart is what separates a button that works from one that mysteriously doesn't.

### Gate 1 — a secure context

The clipboard write only works in a <Term definition="A page served over HTTPS, or loaded from localhost. The browser gates powerful APIs behind it.">secure context</Term> — HTTPS, or `localhost` during development. This is the same gate you already met with `crypto.randomUUID` and `crypto.subtle` in the previous lesson, so you've seen it before; the reflex is the same and the local-dev unblock is the same `mkcert` setup.

There's one wrinkle worth flagging, because it shapes how you handle the failure later. On a plain `http://` page, `navigator.clipboard` isn't a clipboard object that rejects your call — it's simply `undefined`. So the failure isn't a rejected Promise you can `catch`; it's a `TypeError` the moment you reach for `.writeText` on `undefined`. We'll come back to why that distinction matters when we handle failures. For now: serve over HTTPS or `localhost`, and this gate is open.

### Gate 2 — transient user activation

This is the gate that bites people, because it's the one that's always open in dev and frequently closed in production. The browser will only let you write to the clipboard as the *direct result of a user action* — a click, a key press, a pointer release. No gesture, no write. The browser does this on purpose: a page that could silently rewrite your clipboard at any moment would be a security disaster, so the clipboard is a privileged surface you can only touch when the user just did something.

Here's the part that's easy to miss. That permission isn't permanent — it's *transient*. The browser sets a "the user just interacted" flag on the gesture and then clears it about a second later. So it's not enough that a click happened *at some point*; the write has to run while that flag is still live. The moment you let too much time pass — or hand control back to the browser across the wrong kind of async gap — the activation is spent, and the call rejects with a `NotAllowedError`.

<Term definition={"A short-lived 'the user just interacted' flag the browser sets on a gesture like a click or key press and clears about a second later.\nPrivileged APIs such as clipboard write require it to be live at the moment they run."}>Transient user activation</Term> is the load-bearing idea of this lesson, so let's make the invisible window visible. Step through those and the pattern resolves into one rule: **the write goes inside the gesture handler, and it runs before you hand control back to the browser for anything slow.** The first marker lands because the call is the first thing the click does. The second misses because a network round-trip outlived the activation — the `fetch` resolved a few hundred milliseconds later, in a fresh microtask, with the flag already cleared. The third never had a chance: a `setTimeout` and a mount effect both run with no gesture under them at all.

This is *why* you'll see the call sitting at the top of the click handler in the next section, not buried after some `await` that talks to the network, and never in a `useEffect`. If you genuinely need data from the server before you copy, fetch it *before* the user clicks and have the string ready, so the click handler does nothing but write.

## The canonical Copy button

Now assemble the real thing. The button has to run in the browser — it has an interactive click handler and local feedback state — so its file is a <Term definition="A file that starts with the 'use client' directive. It ships to the browser and can use interactivity, state, and browser APIs.">Client Component</Term>: the `'use client'` directive sits as the literal first line. You'll meet that directive properly later in the course when server and client components get their own chapter; for now, treat it as the marker that says "this code runs in the browser."

That client boundary is a design decision, not a formality. You make the *button itself* the client component — a tiny island — and pass it the URL as a plain string prop from a Server Component parent that stays on the server. The mistake is turning the whole page into a client component just to host one Copy button; you'd ship far more JavaScript and lose the server rendering for everything around it. Smallest interactive leaf, fed its data as props.

Here's the shape. Step through it — each step highlights one decision.

<AnnotatedCode lang="tsx" maxLines={18} code={`
'use client';

type CopyButtonProps = {
  value: string;
  label: string;
};

export const CopyButton = ({ value, label }: CopyButtonProps) => {
  const [copied, setCopied] = useState(false);
  const timer = useRef<ReturnType<typeof setTimeout> | null>(null);

  useEffect(() => () => clearTimeout(timer.current ?? undefined), []);

  const copy = async () => {
    try {
      await navigator.clipboard.writeText(value);
      setCopied(true);
      timer.current = setTimeout(() => setCopied(false), 2000);
    } catch {
      // the write rejected — degrade instead of lying (next section)
    }
  };

  return (
    <button type="button" onClick={copy} aria-label={label}>
      <span aria-hidden>{copied ? 'Copied' : 'Copy'}</span>
      <span role="status" className="sr-only">
        {copied ? 'Copied to clipboard' : ''}
      </span>
    </button>
  );
};
`}>
  <AnnotatedStep meta="{1} {8}" color="violet">
    The client-island boundary. The `'use client'` directive ships this one file to the browser; `value` is the string to copy and `label` is the accessible name, both handed down as props from a Server Component parent that stays on the server.
  </AnnotatedStep>

  <AnnotatedStep meta={`{14} {16} "navigator.clipboard.writeText"`} color="green">
    The write is the *first* thing the handler does, reached synchronously the instant the click fires — this is the activation gate from the last section, in code. No `await fetch` before it, no effect, no timeout standing between the gesture and the write.
  </AnnotatedStep>

  <AnnotatedStep meta="{15} {19} {21}" color="orange">
    The call rejects in real browsers — expired activation, denied permission — so the `catch` is not optional. What goes inside it is the whole next section; for now, know that swallowing the error silently is the bug, not the fix.
  </AnnotatedStep>

  <AnnotatedStep meta={`{9} {10} {12} "setTimeout(() => setCopied(false), 2000)"`} color="blue">
    `copied` drives the label swap; a `setTimeout` flips it back after two seconds, and its handle lives in a ref so the cleanup can cancel it if the button unmounts mid-timer. The hook mechanics land in their own chapters later — here, just see the shape: feedback that cleans up after itself.
  </AnnotatedStep>

  <AnnotatedStep meta={`"aria-label={label}" {27-29}`} color="violet">
    The visible content is the bare word "Copy" with no object, so `aria-label="Copy invoice URL"` gives a screen reader the full intent. The `role="status"` element is a live region — assistive tech announces its text when it changes, so a non-sighted user hears "Copied to clipboard" the moment the copy lands. Live regions get full treatment in a later chapter; here, plant the correct attribute.
  </AnnotatedStep>
</AnnotatedCode>

A few notes on choices in that block. The visible text is wrapped in an `aria-hidden` span and the announcement lives in a separate `role="status"` span, so a screen reader hears one clean "Copied to clipboard" instead of the visible label and the status text fighting each other. The `<button>` is a bare semantic button on purpose — in a real product you'd reach for your design system's button component, but importing one here would pull focus from the clipboard surface this lesson is actually about. The semantics that matter — `type="button"`, the accessible label, the live region — are all present; the styling is deliberately left out.

:::caution
Notice there is no path in that handler where `copied` becomes `true` without the `await` having resolved first. The success state is downstream of the successful write, never beside it. That ordering is the entire point of the next section.
:::

## When the write fails, degrade instead of lying

A Copy button that catches the error, throws it away, and *still* flashes "Copied" is worse than one with no error handling at all — it tells the user the URL is on their clipboard when it isn't, and they paste stale garbage into an email to a client. The catch branch isn't cleanup you bolt on at the end; it's a first-class part of the button's behavior. Let's make the failure path honest.

Two failure shapes are worth handling, and they are not the same kind of failure:

- **Activation expired or denied.** The Promise rejects with a `NotAllowedError` — the gesture window closed, or the browser declined the request. This is a real runtime failure that can happen to a real user, so it belongs in the `catch`. The honest response: tell them it didn't copy and give them a way to copy it themselves.
- **Insecure context.** On `http://`, recall that `navigator.clipboard` is `undefined`, so this never reaches your `catch` at all — it throws a `TypeError` when you read `.writeText`. And it's not a user error; it's a deployment misconfiguration. The fix isn't a friendly message to the user — it's making sure production is HTTPS so this can't happen, and surfacing it loudly in development (a console warning, a dev-only banner) if it ever does. Don't try to handle the insecure-context case with a `catch (e) { if (e.name === '…') }` branch; that's not where it shows up.

The recovery for the case that *can* reach a user — the rejected write — is a manual-copy affordance. When the automatic copy fails, render the value in a small read-only field with its text pre-selected, so the user can finish the job with `Cmd/Ctrl+C` themselves. The exact UI is yours — an inline field, a tiny popover — but the principle is fixed: **never show success you didn't achieve.**

The two versions below are the whole argument of this section. One is the button most people would write; the other is the one you ship.

## Reading the clipboard and rich content

Everything so far has been *writing* plain text, which is the surface you'll actually reach for. The Clipboard API has two more corners worth recognizing so you know they exist — and worth knowing you usually shouldn't reach for them.

**Reading** the clipboard from JavaScript is possible — `navigator.clipboard.readText()` returns the clipboard's text, and `navigator.clipboard.read()` returns richer content. But reads prompt the user for permission in Chromium browsers and behave differently in Safari, and the SaaS reach is thin: when you want a paste, you almost always get it for free by wiring an `<input>` or `<textarea>` to the browser's normal paste behavior, not by reading the clipboard yourself. The posture: don't read the clipboard from JS unless the feature genuinely needs it — a rich-paste-into-a-document editor, a screenshot dropped into a chat input. For an ordinary form, let the platform handle paste.

```ts
const pasted = await navigator.clipboard.readText();
```

**Rich content** is the other corner. `writeText` only does plain text; to put multiple representations on the clipboard at once you use `navigator.clipboard.write` with a <Term definition="An object holding one clipboard entry as several MIME-typed representations at once, so each paste target can pick the format it understands.">`ClipboardItem`</Term>. You hand it a map from MIME type to content, and the paste target picks the format it understands — a rich editor takes the `text/html`, a terminal takes the `text/plain`.

```ts
await navigator.clipboard.write([
  new ClipboardItem({ 'text/html': htmlBlob, 'text/plain': textBlob }),
]);
```

The 2026 reach for this is narrow: image and chart copy buttons — "copy this generated diagram as an image." If you ever build one, expect a Safari-specific wrinkle around async data and the gesture window, where the value handed to a `ClipboardItem` may need to be a Promise that resolves *inside* the gesture rather than something you awaited beforehand. Treat that as a "look it up when you need it" detail, not something to memorize now. And notice those values are `Blob`s — the binary container behind rich clipboard content. You'll meet `Blob` and `File` properly in the next lesson; they're the substrate under this and under file uploads.
