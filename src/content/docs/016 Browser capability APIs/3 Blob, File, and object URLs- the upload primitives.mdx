You're building the account settings page, and it needs a profile photo field. The shape is the one every SaaS app shares: an `<input type="file" accept="image/*">`, and the moment the user picks an image you want a thumbnail preview to appear with the filename beside it, so they can confirm they grabbed the right file before they hit save. On save, the bytes upload. You build a version of this constantly — avatar pickers, attachment previews, logo uploads, CSV imports — so it's worth getting the primitives under it right once.

So you write the obvious thing. The change handler hands you a file, you drop it into an `<img src={pickedFile}>`, and you get a broken-image icon — because a file object isn't a URL, and the browser has no idea what to do with it. You search for the fix and the top answer reaches for `FileReader`, which works but is a pile of event-handler boilerplate from an older era of the platform. And the version that *does* render the preview cleanly — `URL.createObjectURL` — quietly leaks a chunk of memory on every re-pick unless you do one more thing that nobody's first draft includes. **What are the platform primitives that turn picked bytes into a preview, and what is the one cleanup step that keeps the obvious version from leaking?** Hold that; the whole lesson is the answer.

By the end you'll have a leak-free pick-to-preview island — pick a file, see the thumbnail, see the name, no memory left behind — and a clear model of where those bytes go next when it's time to actually upload them. Three nouns build the whole thing, in a chain, so we'll take them one at a time and assemble them last.

## `Blob` — an immutable bag of bytes

Start at the bottom of the chain. A <Term definition={"An immutable, fixed-length sequence of bytes carrying a claimed MIME type.\nThe universal in-memory binary container the browser passes around — Response.blob() returns one, a fetch upload body accepts one, a File is one."}>`Blob`</Term> is the platform's container for a chunk of binary data: an immutable, fixed-length sequence of bytes. It doesn't know or care whether those bytes are an image, a CSV, or a zip — it's just the bytes, plus a label saying what kind of thing they're supposed to be. You read two properties off it: `size`, the byte length as a number, and `type`, the <Term definition={"A short string like image/png or text/csv that labels the kind of bytes. A Blob carries whatever its producer claimed — the type is a label, not a guarantee, and is never checked against the actual content."}>MIME type</Term> string.

That word *claimed* in the definition is doing real work, so underline it now: `type` is whatever the producer said the bytes are, not what they actually are. Nothing verifies it. A Blob labelled `'image/png'` can hold anything at all. We'll come back to why that matters the moment a user is involved.

You construct one with `new Blob(parts, { type })`, where `parts` is an array of pieces to concatenate — strings, `ArrayBuffer`s, typed arrays like `Uint8Array` (the byte array from the streaming chapter), or other `Blob`s. The browser stitches them together into one contiguous byte sequence.

```ts
const csv = new Blob(['id,name\n1,Ada\n'], { type: 'text/csv' });
```

When do *you* reach for `new Blob(...)`? Whenever your code is the one minting the bytes: assembling an upload payload in memory, slicing a piece out of a larger file, building generated content like a CSV export before you hand it to a download. And you've already met Blobs without naming them as a category — the rich `ClipboardItem` content from the previous lesson held `Blob`s, `Response.blob()` from the fetch chapter hands one back, and a `fetch` upload body accepts one directly. That's the senior framing to bank: **`Blob` is the universal in-memory binary container** the platform passes around. Once bytes are in a Blob, every byte-shaped API on the platform speaks the same language.

One watch-out, right here at the construct. If you write `new Blob(['hello'])` with no options object, `type` comes out as the empty string `''` — not a sensible default, just *unset*. Most consumers treat an empty type as `application/octet-stream`, the generic "some bytes, who knows" type. When anything downstream is going to branch on the content type — a server deciding how to store it, an `<img>` deciding whether it can render it — pass an explicit `type`. Don't let it default.

## `File` — a `Blob` the OS handed you

Now the second noun, and it's a small step up from the first. A `File` *is* a `Blob` — it's a subclass — so everything you just learned carries over: a `File` has `size`, has `type`, has the same byte-reading methods. It adds exactly two read-only properties on top: `name`, the filename as the operating system reports it, and `lastModified`, a timestamp in milliseconds of when the file was last changed on disk. That's the whole difference. A `File` is a `Blob` that remembers where it came from.

Here's the fact that reorganizes how you think about it: **your code never constructs a `File`. The browser does.** You don't `new File(...)` — you *receive* `File` objects, and in a SaaS app they arrive from essentially one place. When the user picks files through an `<input type="file">`, the input's `change` event populates `event.target.files` — a <Term definition={"An array-like list of File objects the browser fills from a file input or a drop. It has length and index access but no array methods — spread it ([...files]) or use Array.from before you map or filter."}>`FileList`</Term> holding everything they selected. (Drag-and-drop is the other source — it fills `event.dataTransfer.files` with `File`s the same way — but a plain file input covers the upload surface this course builds, so we'll name drop once and move on.)

Reading the picked file off that event is one line, and it has a subtlety in it. For a single-file picker you reach for `event.target.files?.[0]` — the first file in the list. The optional chain matters because if the user opens the picker and cancels, there's no selection and you'd be indexing into nothing. For a multi-file picker you iterate the list — and this is where the first real footgun lives. A `FileList` is *array-like*: it has a `length` and you can index it with `files[0]`, but **it is not an array.** It has no `.map`, no `.filter`, no `.forEach` that you'd reach for to render a list of thumbnails. Spread it into a real array first — `[...files]` — and then it behaves the way you expect.

const [file, setFile] = useState<File | null>(null);

<input

  accept="image/png,image/jpeg,image/webp"
  onChange={(event) => setFile(event.target.files?.[0] ?? null)}

  </AnnotatedStep>

  <AnnotatedStep meta={`"event.target.files?.[0]"`} color="green">
    `event.target.files` is the `FileList` the change event populates. Single-pick reads the first entry, `?.[0]`, and the optional chain is load-bearing: a cancelled picker means no selection, so it guards against indexing nothing and `?? null` falls back cleanly.
  </AnnotatedStep>

  <AnnotatedStep meta={`"setFile"`} color="orange">
    `setFile(...)` stashes the picked `File` in state; from here the rest of the lesson turns it into a preview. For a `multiple` picker you'd read the whole `FileList` instead — and because it's array-like but not an array, you spread it, `[...event.target.files]`, before you can `.map` it into a row of thumbnails.
  </AnnotatedStep>

That `accept` attribute earns its own warning, because it is the single most common thing juniors mistake for security. `accept="image/png,image/jpeg,image/webp"` tells the OS picker to gray out everything that isn't one of those types — it's a nicer experience, the user doesn't see files they can't pick. But it is **UX, not validation.** A determined user can rename a file, drop something past the picker, or hit your endpoint directly with whatever bytes they like. So the reflex, and you'll see this pattern again and again: **`accept` is UX; real content-type and magic-byte validation is security, and it lives on the server** — the upload chapters later in the course own the step where the server reads the actual leading bytes of the file and confirms they match the claimed type before trusting it. Remember the *claimed* `type` from the Blob section? This is why it was only ever a claim.

That gives us the single sentence that organizes this whole lesson, so commit it: **bytes that come from the user arrive as a `File`; bytes your own code mints are a `Blob`.** Same byte container underneath, two directions of travel. Hold that, because the next noun renders both of them the exact same way.

## `URL.createObjectURL` — a renderable handle for bytes

You have a `File` (or a `Blob`) in hand. You can't put it in an `<img src>` directly — `src` wants a URL string, and a file object is not a string. <Term definition={"A temporary string handle of the form blob:<origin>/<uuid> that maps to bytes held in browser memory. Any element that takes a URL can read it; the bytes stay pinned until the handle is revoked or the page unloads."}>`URL.createObjectURL`</Term> is the bridge. You pass it a `Blob` or a `File`, and it hands back a **string** that looks like this:

```tsx
const previewUrl = URL.createObjectURL(file);
// 'blob:https://app.example.com/9b2c…-uuid'

<img src={previewUrl} alt="" />;
```

That string is a real URL as far as the DOM is concerned. Any element that takes a URL can consume it — `<img src>`, `<video src>`, an `<a href download>` for a download link, an `<iframe src>` for an in-page preview. The element renders the bytes as if they'd been fetched from a server, except nothing was fetched and there's no server.

Here's the mechanism, and this is the conceptual crux of the lesson — get this and the rest is consequence. When you call `createObjectURL`, the browser adds an entry to an **in-memory map**: this URL string points to those bytes, held in memory. The string is cheap — it's a few dozen characters, a UUID with a prefix. But **the bytes behind it stay alive, pinned in memory, for as long as that map entry exists.** Creating the URL is what pins them. They don't get released when you stop using the string, or when the `<img>` finishes rendering, or when the variable goes out of scope. The map entry is the thing keeping them alive, and the map entry sticks around until you remove it or the page unloads. File that away — the leak we're about to meet is a direct, mechanical consequence of it.

You might wonder why this is the move and not the `FileReader` the internet pushed you toward. `FileReader` produces a <Term definition={"A data:<type>;base64,… string that embeds the file's bytes inline as base64. Self-contained but ~33% larger than the raw bytes and re-encoded on every use — an object URL is a constant-size handle instead."}>data URL</Term>: a `data:image/png;base64,…` string with the *entire file encoded inline*. For a two-megabyte photo, that's a two-and-a-half-megabyte string (base64 inflates the bytes by about a third), allocated in memory as a string, and re-materialized every time something reads it. An object URL is the opposite trade: a tiny constant-size handle no matter how big the file, with the real bytes referenced once in the browser's map rather than copied into a string. For previewing a picked file, `createObjectURL` is the 2026 default. `FileReader` is the event-driven legacy path, and the one place it still earns its weight is when you specifically need its progress events for a long read — recognize that it exists, reach for the object URL otherwise.

One more thing, and it's a deliberate correction to a reflex you've been building all chapter. The last two lessons gated their APIs behind a <Term definition="A page served over HTTPS or from localhost. Browsers gate Web Crypto, the clipboard, and some other powerful APIs to secure contexts.">secure context</Term> — `crypto.subtle`, `randomUUID`, and the clipboard write all silently fail on plain `http://`. **`Blob`, `File`, and `createObjectURL` are not in that club.** They work fine on plain HTTP; no secure context required. What *does* constrain them is different: they're **browser-only**. There is no `URL.createObjectURL` and no `<img>` on the server, so this code cannot run in a Server Component — it has to live in a client island, the same `'use client'` boundary you put the Copy button behind. Same boundary discipline, different reason: not because the API is gated, but because the API simply doesn't exist outside the browser.

## `revokeObjectURL` and where the cleanup belongs

Now the climax, and the reason this lesson exists. You've created a handle that pins bytes in memory. The mirror call releases them. <Term definition="The mirror of createObjectURL: it removes the URL's entry from the browser's in-memory map, which lets the underlying bytes be garbage-collected. Every createObjectURL has a matching revokeObjectURL.">`URL.revokeObjectURL(url)`</Term> takes a URL you created and deletes its entry from that map — which finally lets the bytes be garbage-collected. This is the cleanup reflex the whole chapter has been drilling, and it has a clean one-line form you should burn in: **every `createObjectURL` has a matching `revokeObjectURL`.** It's the same shape as aborting a request you started in the DOM chapter and clearing the `setTimeout` you started in the last lesson — every resource you open in the browser, you close.

Skip the revoke and here's exactly what happens. Each time the user re-picks a photo, your handler calls `createObjectURL` again and mints a *fresh* handle for the new file. The `<img>` swaps to the new URL and renders the new preview — looks perfect. But the *previous* URL's map entry is still there, so the *previous* file's bytes are still pinned, with nothing pointing at them and no way to ever release them. In a one-shot avatar picker that's one stale image's worth of memory leaked — annoying, survivable. In a gallery or a multi-file attachment list, it scales linearly: every re-pick, every re-render that re-creates a URL, strands another file's bytes, and the tab's memory climbs until something falls over. This is the bug the lesson is here to prevent.

To make the invisible map visible, scrub through the lifecycle below. Watch the **memory** column on the left fill with pinned byte-blocks and the live **handle** on the right point at one of them; the leak is the moment a block has no arrow pointing at it but is still sitting in memory.

So where does the revoke actually go? Two correct homes, picked by context.

For the pick-to-preview flow, it belongs in a cleanup that fires **when the file changes or the component unmounts** — so swapping to a new photo revokes the old URL, and leaving the page revokes the last one. In React that's the cleanup *return* of a `useEffect` keyed to the file. For a one-shot generated download — mint a CSV, link it, let the user click — the revoke goes right after the consumer is done with it, in the link's click handler. The principle under both: revoke once nothing needs the bytes anymore.

And now the footgun, which is its own distinct mistake and deserves equal billing with the leak: **do not revoke too early.** It is tempting to write `createObjectURL` and `revokeObjectURL` on adjacent lines — create the handle, set `<img src>`, revoke. That blanks the preview. Setting `src` doesn't synchronously read the bytes; the browser reads them a beat later, asynchronously, and if you've already revoked the URL by then the map entry is gone and the `<img>` renders the broken-image icon. This is the exact mechanism behind the second-most-common version of this bug, the one that goes *"I added the cleanup like the docs said and now my preview is blank."* The revoke runs on unmount or on URL change — *after* the element has had its chance to read the bytes — never on the line right after you created it.

A few lifecycle facts worth stating once and then trusting. A `blob:` URL is **origin-scoped**: it only works on the page that created it, you can't share one across tabs, and it doesn't survive a page reload. The browser revokes *all* of a page's object URLs automatically when the page unloads — so your manual `revokeObjectURL` is about managing memory *during the tab's lifetime*, not about cleaning up after the user closes the tab. And setting `<img src>` to a URL you've already revoked renders the broken-image icon, same as the footgun above.

Now assemble the whole thing. This is the leak-free pick-to-preview island, every piece you've met in one place — the picked `File` in state, the object URL derived from it, the preview, and the cleanup that makes it not leak.

<AnnotatedCode lang="tsx" maxLines={18} code={`
'use client';

export const AvatarPicker = () => {
  const [file, setFile] = useState<File | null>(null);
  const [previewUrl, setPreviewUrl] = useState<string | null>(null);

  useEffect(() => {
    if (!file) return;
    const url = URL.createObjectURL(file);
    setPreviewUrl(url);
    return () => URL.revokeObjectURL(url);
  }, [file]);

  return (
    <div>
      <input
        type="file"
        accept="image/png,image/jpeg,image/webp"
        onChange={(event) => setFile(event.target.files?.[0] ?? null)}
      />
      {previewUrl != null && <img src={previewUrl} alt="" />}
      {file != null && <p>{file.name}</p>}
    </div>
  );
};
`}>

Same boundary as the Copy button — but note the difference: there's no secure-context gate here, unlike the clipboard. It's a client island because the API is browser-only, not because it's gated.
  </AnnotatedStep>

  <AnnotatedStep meta="{4-5}" color="blue">
    The two state slots — the picked `File`, and the preview URL string derived from it. Two pieces of state because they change at different moments: the file on pick, the URL as a consequence.
  </AnnotatedStep>

  <AnnotatedStep meta="{9-10}" color="green">
    Inside the effect, `const url = URL.createObjectURL(file)` mints the handle once a file is present — the `if (!file) return` guards the no-selection case.
  </AnnotatedStep>

  <AnnotatedStep meta="{11}" color="orange">
    The cleanup `return () => URL.revokeObjectURL(url)` is the whole lesson in one line. It fires when `file` changes — releasing the old URL *before* the new preview renders — or on unmount. It does **not** fire synchronously after the `<img>` mounts, which is exactly why the preview survives instead of going blank.
  </AnnotatedStep>

  <AnnotatedStep meta="{16-23}" color="blue">
    The render. `accept` is UX-only; `event.target.files?.[0] ?? null` handles the cancelled-picker case; the `!= null` guards render the `<img>` only once a URL exists and the `<p>` only once a file is picked — a `null` check, not a bare-truthy test, with `file.name` shown beside the preview.
  </AnnotatedStep>

A note on what's deliberately missing from that block, so nobody "fixes" it later. The markup is bare — a plain `<input>`, a plain `<img>`, a plain `<p>` — with no design-system components and no styling. That's on purpose: it keeps your attention on the byte primitives instead of the chrome. In a real product you'd wire this to your design system and, in the upload chapters ahead, to a server action that actually ships the bytes. The hooks are shown as *shape*, not taught — `useState` holds the file and the URL, `useEffect` synchronizes with the object-URL map and cleans it up, and that cleanup-return contract is genuinely the textbook use of an effect (syncing with an external system and tearing it down). The mechanics of those hooks land properly later in the course; here, just read the shape.

:::caution
The revoke lives in the cleanup *return*, not on the line after `createObjectURL`. Writing `URL.revokeObjectURL(url)` immediately after creating the URL blanks the preview — the `<img>` hasn't read the bytes yet. Create, render, and let the effect's cleanup revoke on the *next* file change or on unmount.
:::

## Two more shapes you'll meet

The pick-to-preview island is the case you'll build most, but two relatives are worth recognizing so you spot them when they show up.

**Downloading a generated file** is the mirror image. Sometimes the app *mints* the bytes itself — a CSV export, a generated PDF or JSON — and wants to hand them to the user as a download. This is the `Blob` direction of the rule from earlier: bytes your code made, not bytes the user picked. The shape is the same `createObjectURL`, just pointed the other way: build a `Blob`, make a handle, put it on an `<a href download>`, let the click happen, then revoke. Same handle, same cleanup reflex, opposite source.

```tsx
const downloadReport = (csv: string) => {
  const blob = new Blob([csv], { type: 'text/csv' });
  const url = URL.createObjectURL(blob);

  const link = document.createElement('a');
  link.href = url;
  link.download = 'report.csv';
  link.click();

  URL.revokeObjectURL(url);
};
```

That synchronous revoke right after `link.click()` looks like it contradicts the footgun, but it doesn't: a download click captures the bytes before `click()` returns, so the handle has already done its job by the next line. The `<img>` case is different because rendering reads the bytes *later*. The rule is unchanged — revoke once the consumer is done — it's just that a download consumer finishes immediately and an `<img>` consumer doesn't.

**The upload handoff** is where the `File` you previewed actually goes in production — and it's deliberately someone else's lesson, named here once so you see the seam this lesson builds toward. The picked `File` doesn't get uploaded through your server. The client sends the file's `name`, `type`, and `size` to a server action; the action returns a short-lived <Term definition="A URL, signed by the server with a short expiry, that grants the holder permission to PUT an object straight to object storage. The bytes go browser-to-storage; the application server only signs the URL, it never handles the file.">presigned URL</Term>; the client then PUTs the **`File` itself** straight to object storage using that URL, so the bytes travel browser-to-storage and your server never touches them; finally the client tells the server "done, here's the key" so it can write the metadata row. The `File` object from this lesson is the body that PUT accepts — that's the entire reason it mattered to get these primitives right.

## Spot the leak

Time to write the cleanup yourself, because that's the one durable skill this lesson installs. The component below renders a preview correctly but leaks a `Blob` on every re-pick — the canonical bug, the first draft everyone writes. Make it leak-free without breaking the preview.

<ReactCoding
  instructions="This avatar preview leaks a Blob on every re-pick — it mints an object URL in the effect but never revokes the old one. Wire the cleanup so picking a new file (or unmounting) releases the previous URL — without blanking the preview."
  starter={`import { useState, useEffect } from 'react';

export function App() {
  const [file, setFile] = useState<File | null>(null);
  const [previewUrl, setPreviewUrl] = useState<string | null>(null);

  useEffect(() => {
    if (!file) return;
    const url = URL.createObjectURL(file);
    setPreviewUrl(url);
    // The handle is created, but nothing ever releases it.
  }, [file]);

  return (
    <div className="space-y-3">
      <input
        type="file"
        accept="image/png,image/jpeg,image/webp"
        onChange={(event) => setFile(event.target.files?.[0] ?? null)}
      />
      {previewUrl != null && (
        <img src={previewUrl} alt="" className="h-24 w-24 rounded object-cover" />
      )}
      {file != null && <p className="text-sm text-gray-600">{file.name}</p>}
    </div>
  );
}`}
  tests={`
// Record every object URL created and revoked, so the tests can check the
// cleanup wiring without depending on what the <img> visually shows.
const created: string[] = [];
const revoked: string[] = [];
const realCreate = URL.createObjectURL.bind(URL);
URL.createObjectURL = (obj: Blob | MediaSource) => {
  const url = realCreate(obj);
  created.push(url);
  return url;
};
URL.revokeObjectURL = (url: string) => {
  revoked.push(url);
};

const makeImageFile = (name: string) =>
  new File([new Uint8Array([137, 80, 78, 71])], name, { type: 'image/png' });

// Simulate the user choosing a file: set it on the input and fire the change
// event React listens for, then let the effect run.
const pick = async (name: string) => {
  const input = document.querySelector('input[type="file"]') as HTMLInputElement;
  Object.defineProperty(input, 'files', { configurable: true, value: [makeImageFile(name)] });
  input.dispatchEvent(new Event('change', { bubbles: true }));
  await new Promise((resolve) => setTimeout(resolve, 0));
};

test('picking a file shows a preview from a blob: URL', async () => {
  await pick('avatar.png');
  const img = document.querySelector('img');
  expect(img).not.toBeNull();
  expect(img?.getAttribute('src')?.startsWith('blob:')).toBe(true);
});

test('the visible preview is not revoked while it is still on screen', async () => {
  created.length = 0;
  revoked.length = 0;
  await pick('avatar.png');
  expect(revoked).not.toContain(created[0]);
});

test('re-picking releases the previous file URL (no leak)', async () => {
  created.length = 0;
  revoked.length = 0;
  await pick('first.png');
  const firstUrl = created[0];
  await pick('second.png');
  expect(revoked).toContain(firstUrl);
});
`}
/>

The middle test is the footgun guard: revoking on the line right after `createObjectURL` blanks the preview, so the cleanup belongs in the effect's `return` — released on the *next* pick or on unmount, after the `<img>` has read the bytes.
