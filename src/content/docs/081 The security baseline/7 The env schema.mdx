Back in the first project that ever held a real secret, you wrote one file — `env.ts` — and made it a quiet promise: every piece of configuration this app reads goes through here, validated, typed, once. That was the very first named side-effect boundary in the whole course, and the note at the bottom of that lesson said the idea would come back to "anchor the security baseline near the end." This is that moment. The baseline pass is here to confirm the door is still the only door.

Because here's what actually happened in between. You added auth variables, then email, then invitations, then Upstash, then Stripe, then observability — a variable here, a variable there, each under its own deadline. One file was the single source of truth on day one. A dozen variables later, the senior question is simply: *is it still?* Or has the discipline quietly eroded, one bypass at a time, the way these things always do?

The previous lesson left you a thread to pull. Its leak audit opened with a check — "no `process.env.X` outside `env.ts`" — and then said, in so many words, *the full invariant is the next lesson's*. That's this lesson. The leak audit owned where secrets escape; this one owns whether the validation discipline still holds. By the end you'll have a four-item audit you can run against any repo — including yours, today — to answer that one question.

## One file, four jobs

Before the checks, it's worth getting precise about what `env.ts` actually *is* now that it has grown, because every invariant below is really just protecting one of its jobs. When you wrote it, it had one obvious purpose. As the app matured, three more quietly piled onto the same file. It now does four things at once:

1. **It's the runtime gate.** `createEnv` runs the schema the instant the module loads. A missing or malformed variable doesn't boot a half-configured app — it fails `next build`, on your terminal, naming the variable. (You saw exactly this when you pulled `DATABASE_URL` and watched the build refuse.)
2. **It's the type boundary.** `env.DATABASE_URL` is typed `string`, never `string | undefined`. The build already guaranteed the value exists, so nothing downstream null-checks it.
3. **It's the secret/client firewall.** The `server` block is structurally walled off from the browser bundle. Import a server variable into a Client Component and you get a *build error* — not a lint warning you can ignore, a failed deploy.
4. **It's the documentation.** Open the file and you have the canonical, exhaustive answer to "what does this app need to run." Nothing else has to be true for that to be true.

All four collapse into one file. That's the whole reason the file is worth guarding.

## Invariant 1 — every access goes through the typed `env`

The first invariant is the precondition for all the others, so it's the one to lock in first: **every read of configuration is the typed import.** You write `import { env } from '@/env'` and then `env.WHATEVER`. Raw `process.env.WHATEVER` anywhere else is the smell.

The reason it's invariant number one is that a variable which never enters the schema can't be protected by *any* of the other three. It can't be split into server or client, it can't be documented, it can't be validated — it just sits there, untyped and unchecked, waiting. The schema can only guard what's in the schema.

Look at the difference. The two versions below read the same Stripe secret, and the second one is the only one the schema can see.

<CodeVariants>
  <CodeVariant label="Bypasses the schema">
    <div data-mark-color="red">

    ```ts del={1}
    const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!);
    ```

    </div>
    **Skips the boundary entirely.** The value types as `string | undefined` — note the `!` papering over it — and nothing validates it, so a missing variable sails past the build.
  </CodeVariant>

  <CodeVariant label="Goes through the boundary">
    <div data-mark-color="green">

    ```ts ins={1-2}
    import { env } from '@/env';
    const stripe = new Stripe(env.STRIPE_SECRET_KEY);
    ```

    </div>
    **Covered by the schema.** Through `env` the value is typed `string` — no `!`, no null check — and a missing `STRIPE_SECRET_KEY` would have failed the build before this line ever ran.
  </CodeVariant>
</CodeVariants>

The failure mode is worth picturing concretely, because it's the exact bug `env.ts` was installed to kill, sneaking back in through a side door. A stray `process.env.STRIPE_SECRET_KEY` deep in a handler types as `string | undefined`. It slips past the build, because nothing in the schema validates it. The app deploys clean. Then the first request that reaches that handler in an environment where the variable was never set gets `undefined`, hands it to Stripe, and 500s. You're back to the 3am outage the boundary exists to prevent — except now it's hiding in one handler instead of being caught at the door.

The check is a single grep: `process.env` across the whole repo. Every hit outside `env.ts` is a finding. The previous lesson's leak audit already greps for exactly this — here you're naming *why* it's invariant one, not just a leak vector.

There are two sanctioned exceptions, and naming them keeps you from flagging false positives. The first is `process.env.NODE_ENV` — Next.js validates it itself, and you'll genuinely need it for build-time branching (you'll use it twice later in this very lesson). The second is rare framework internals reading their own knobs, like a `next.config.ts` checking `process.env.ANALYZE` to toggle the bundle analyzer. Everything else routes through `env`.

## Invariant 2 — the server/client split is a firewall, not a label

This is the load-bearing security invariant of the lesson, so slow down here. The schema doesn't put variables in a `server` block and a `client` block as a tidy bit of organization. The split is a *firewall*. Server-only variables go in `server`; anything the browser is allowed to see goes in `client` and **must** carry the `NEXT_PUBLIC_` prefix — the only prefix Next.js inlines into the browser bundle. Cross that line the wrong way and the consequence is structural, not cosmetic.

There are two ways to cross it, and they are not equally serious.

The dangerous one is a **leak**: a `server` variable imported into a Client Component. Because that file gets bundled for the browser, the secret would be inlined into JavaScript anyone can read. Here's the part that makes the split a firewall — `@t3-oss/env-nextjs` throws at *build time* when you do this. It's not caught in code review if someone's paying attention; it's structurally impossible to ship. The defense is the split itself.

You have a second structural belt for the same job, which you've seen before: `import 'server-only'` at the very top of any module that touches a secret — the database client, an SDK constructed with a private key, the email sender.

```ts title="src/lib/db.ts"
import 'server-only'; // build error if this module is ever imported client-side
```

If any client module ever imports a file that begins with that line, the build breaks before the env package even gets a chance to fire. Two belts, same principle: turn a leaked import into a failed build rather than a discoverable-in-production breach.

The other way to cross the line is mild: a `client` variable read inside a server file. It's harmless for security — the value is already public and already inlined — it just signals confusion, since there's no reason to route a server-side read through a client variable. Worth a glance in review; not worth alarm.

Now the rule that decides which block a variable belongs in. State it once, and let it stick:

**A `NEXT_PUBLIC_*` name is a public promise.** The prefix tells Next.js to ship the value to every visitor, which means the name is a vow that the value is safe for the whole world to read.

The trap is to judge a variable by which vendor issued it instead of by the authority it grants. `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` and `NEXT_PUBLIC_SENTRY_DSN` are public *by design* — a <Term definition="A public-by-design API key that only identifies your Stripe account and grants no privileged access.">publishable key</Term> and a <Term definition="Sentry's public ingest URL — it identifies your project and accepts error events, but grants no read access.">DSN</Term>. Both only *identify*; neither grants any power. But `NEXT_PUBLIC_STRIPE_SECRET_KEY` is a self-contradicting name and a genuine breach — the prefix promises "public" while the name says "secret," and the prefix wins, shipping your key to the world. Same judgment the leak audit asked of you; the difference here is you're sorting variables into the schema, not just spotting names.

Sort it out by hand, because this is precisely where it's easy to slip. For each variable below, decide where it belongs in `env.ts` — and watch for the two that don't belong in the schema at all.

If the two "not in the schema" items gave you pause, that's the subtler half of the invariant landing. `NODE_ENV` is the framework's own variable — you read it, but you don't declare it. And a value you've hardcoded isn't configuration the environment supplies, so it has no business in `env.ts` even though it's a "setting." The schema is for what the *environment* provides, nothing more.

## Invariant 3 — the schema and `.env.example` describe the same world

The third invariant ties the schema to the one file a new contributor actually touches on day one. `env.ts` is what the app *validates*. The committed `.env.example` is what a newcomer *copies* to get started. The invariant: they must list the same variables.

When they drift, onboarding fails in the most demoralizing way possible — silently. A new developer (or an AI agent setting up the project) copies `.env.example`, fills in the blanks, runs `pnpm build`, and hits a validation error for a variable they were never told existed, because someone added it to the schema and forgot the example. The roles are worth holding straight: `.env.example` is committed, with a placeholder value and a `# source:` comment per variable; the real `.env` / `.env.local` is git-ignored and holds the live values. Keeping the two in lockstep is the invariant; turning it into an automatic check that runs on every pull request is the second belt, wired into CI later in the course.

Here's where the payoff of the first three invariants arrives. Because *every* variable goes through the schema (invariant 1), and the schema is split correctly (invariant 2), and it matches the example (invariant 3), the schema becomes something more than a validator: it's the inventory. It *is* the answer to "what does this app need to run." So let's actually look at the grown-up file — your scattered, course-long collection of configuration, gathered into one legible map.

Read that file top to bottom and you've read the app's entire dependency on the outside world: every database, every third-party service, every secret, grouped by the feature that needs it. That's invariant 3's reward — the schema doubles as the most honest documentation you have, *as long as* it stays in lockstep with what the code actually reads. Which is exactly why the orphaned `LEGACY_WEBHOOK_URL` is a real finding and not a harmless leftover: documentation that lies is worse than no documentation, because people trust it.

## Invariant 4 — `SKIP_ENV_VALIDATION` is an escape hatch, not a setting

The fourth invariant guards the one feature that can undo all the others. `@t3-oss/env-nextjs` honors a `SKIP_ENV_VALIDATION` flag: set it, and `createEnv` returns the env object without running a single schema. The whole gate, switched off.

The earlier lesson named one honest use for it. The baseline pass sharpens that to a precise rule — there are **exactly two** legitimate places for this flag:

1. A Docker or container image build where the server secrets genuinely aren't injected at build time, and you build the image now and supply the values at runtime.
2. A type-check or lint CI job that doesn't need real values to do its work.

Anywhere else, it's the bug. Used correctly, it's a single line in a build script:

```bash
# Dockerfile: secrets are injected at runtime, not build time
SKIP_ENV_VALIDATION=1 pnpm build
```

Now the failure mode that makes it dangerous, because this is the beat to land. Imagine a production build complains that a variable is missing. The lazy fix — the one that feels like progress and is actually a breach — is to set `SKIP_ENV_VALIDATION=1` in the *production runtime environment* to make the error go away. The build goes green. But you've just made the escape hatch permanent. The gate is off in production, forever. And now a genuinely missing variable doesn't surface at *deploy* as a build failure on your terminal — it surfaces at *first request* as a 3am 500, exactly the outage the boundary was installed to prevent. The escape hatch un-does the entire feature.

The reflex to build: **if the build says a variable is missing, the variable is missing.** Set it. Don't silence it.

:::caution
There's a sharper edge here. When validation is skipped, Zod `.default()` values aren't applied either — the schema doesn't run at all, so the defaults never fill in. A build that "works" under `SKIP_ENV_VALIDATION` can ship with `undefined` where you confidently expected a default value. Second reason it's strictly an escape hatch, never a setting.
:::

## When the schema grows up: per-environment variables

The four invariants assume every variable behaves the same in every environment. Two real complications break that assumption, and they only make sense now that the invariants are in hand. Keep these in your back pocket; they're the necessary tail, not the core.

### Production-only variables

Some variables are required in production but genuinely absent in development. `STRIPE_WEBHOOK_SECRET` and `SENTRY_AUTH_TOKEN` are the canonical examples — no developer needs them to run the app on their laptop, but production must have them.

You can't make them unconditionally required (every local `pnpm build` would fail for variables nobody needs locally) and you can't make them unconditionally optional (a production deploy could then ship with Sentry silently un-authed, source maps never uploaded). The fix is to branch the schema on the environment, giving each environment exactly the contract it needs:

```ts {1,4} title="src/env.ts"
const isProd = process.env.NODE_ENV === 'production';

server: {
  SENTRY_AUTH_TOKEN: isProd ? z.string().min(1) : z.string().optional(),
},
```

In production the variable is required; everywhere else it's optional. This is one of the two sanctioned `process.env.NODE_ENV` reads from invariant 1 — the exception earning its keep.

### The per-environment URL helper

The other complication is subtler. Some variables don't switch between present and absent — they hold a *different value* in each environment. The app's own base URL is the cleanest case: it's `http://localhost:3000` on your laptop, a unique per-branch URL on a Vercel preview deployment, and your real domain in production.

You can't hardcode it, and you can't shove three values into one variable. The pattern is a small helper in `/lib` that resolves the URL for whatever environment it's running in — the one place this per-environment branching lives:

<div data-mark-color="green">

```ts {5} title="src/lib/get-app-url.ts"
import { env } from '@/env';

export const getAppUrl = (): string => {
  if (process.env.NODE_ENV === 'production') return env.NEXT_PUBLIC_APP_URL;
  if (process.env.VERCEL_URL) return `https://${process.env.VERCEL_URL}`;
  return 'http://localhost:3000';
};
```

</div>

Two things deserve a callout. First, the gotcha that bites everyone once: Vercel injects `VERCEL_URL` (and `VERCEL_PROJECT_PRODUCTION_URL`) *without* the `https://` scheme — it's a bare host like `my-app-git-feature.vercel.app`. Forget to prepend the scheme and every link the helper builds is broken. Second, those `VERCEL_*` reads sit right next to a `process.env` call, which looks like an invariant-1 violation — but it isn't. Vercel injects them, they're not app secrets, and they live *only* inside this one helper. That's the narrow framework-internal exception, kept narrow so the invariant stays coherent.

The deeper point: "single source of truth" extends past validation to *derived* config. The URL is computed, not stored — and `getAppUrl()` is the one place that computation lives, for the exact same reason `env.ts` is the one place validation lives. One door, again.

## The env audit: one page, four invariants

Every check above is now a line you can run against any codebase, including yours today. This is the synthesis — the four invariants rolled into a repeatable pass, the deliverable this lesson exists to produce. Run it, and every hit is a finding to fix.

<Checklist id="env-audit">
  <ChecklistItem chip="untested">**Every access through the typed `env`.** Grep `process.env` across the repo; every hit outside `env.ts` is a finding. Allowed exceptions: `process.env.NODE_ENV` and rare framework reads like `next.config.ts`. (Invariant 1 — the check the previous lesson's leak audit forward-referenced.)</ChecklistItem>
  <ChecklistItem chip="untested">**Server/client split clean.** No `server`-block variable imported in a Client Component (the build catches it; confirm `import 'server-only'` guards every secret-bearing module), and no `NEXT_PUBLIC_*` name that grants real authority. (Invariant 2.)</ChecklistItem>
  <ChecklistItem chip="untested">**Schema and `.env.example` in lockstep.** Diff the two variable lists; every schema variable has a placeholder + `# source:` line in `.env.example`, and vice versa. Flag any orphaned schema variable no code reads. (Invariant 3.)</ChecklistItem>
  <ChecklistItem chip="untested">**No `SKIP_ENV_VALIDATION` outside the two sanctioned scripts.** Grep for it; it belongs only in the Docker build and the type-check CI job, never in production runtime env vars. (Invariant 4.)</ChecklistItem>
  <ChecklistItem chip="untested">**Production-only vars are `NODE_ENV`-conditional, and the normal build passes *without* `SKIP_ENV_VALIDATION`.** (The extensions.)</ChecklistItem>
</Checklist>

That's the audit. Run it and you've turned four invariants into a repeatable pass. It joins the rest of this chapter's deliverables — the header check, the rate-limit matrix, the audit-event catalog, the retention catalog, the consent reject-path test, and the leak audit — as one of the things the next chapter's pre-launch audit project runs against a seeded codebase.

One boundary to name and leave alone: per-customer, white-label env resolution — different configuration per tenant — is a real thing, and it's beyond where this app is. The single schema, split correctly and kept honest, is the genuine baseline at this stage. "One source of truth, kept consistent" beats "infinitely flexible, kept never."
