The last two lessons mapped the pipeline. Lesson 1 named the four network stages from URL commit to first byte; lesson 2 named the six browser stages from first byte to pixels. Both lessons used the Network panel in passing — protocol column, timing breakdown — but DevTools is much bigger than that one tab, and you'll be living inside the rest of it for the entire course.

DevTools is the feedback half of every loop you'll run as a web developer. You change a class, you reload, you look. You click a button, you see what request went out. You suspect a stale cookie, you read the jar. The panels are how you read each loop. This lesson is the tour — four panels that earn their place on a SaaS engineer's daily path, each answering one load-bearing question.

- **Elements** — what's actually rendered right now?
- **Network** — what did the server send back?
- **Console** — what does the page think it knows?
- **Application** — what's in storage right now?

A fifth panel that gets a lot of marketing — Performance and Lighthouse — is deliberately deferred. You can't tune performance on an empty teaching page; that conversation lands later in the course when there's a real app to measure. The same logic applies to a handful of other panels. We'll name them once near the end so you know where they live, then move on.

What you should leave with isn't mastery of any one panel — it's the reflex. You skim a bug report and reach the right panel without thinking. You hit each panel with two or three moves most newcomers don't know exist. The geography sticks, and the per-panel depth comes from doing the work.

Open any well-styled site in a new browser tab right now — `react.dev` works well — with DevTools docked to the side or bottom. We'll switch panels mid-lesson.

## Setting up: Chromium and React DevTools

The course teaches against <Term definition="Google's open-source browser engine — the base for Chrome, Edge, Arc, Brave, and most other 2026 browsers besides Firefox and Safari.">Chromium</Term> DevTools. Any Chromium browser — Chrome, Edge, Arc, Brave — ships the same deepest-surface DevTools, and the React DevTools extension lands here cleanly. Firefox DevTools is the cross-browser sanity check; the layout differs but the concepts transfer. Safari DevTools is what you'll reach for in the iOS-specific work the deployment unit covers later in the course. For everything else, Chromium is the default.

You'll need one extension installed before the React unit ships its first component: **React DevTools**. Installing it now means it's ready for the rest of the course. The extension adds two tabs to DevTools — **Components** and **Profiler** — that read the React tree directly rather than the DOM. We won't use them in this lesson; we'll need them later.

<Steps>

1. Open the Chrome Web Store listing for [React Developer Tools](https://chromewebstore.google.com/detail/react-developer-tools/fmkadmapgofadopljbjfkapdkoienihi) and click **Add to Chrome**. The same listing works for any Chromium-based browser.

2. Open a page that uses React — `react.dev` is the canonical test — with DevTools open. Two new tabs labelled **Components** and **Profiler** appear in the DevTools tab strip. That confirms the install.

3. If you're on Firefox, install the [same extension from Firefox Add-ons](https://addons.mozilla.org/en-US/firefox/addon/react-devtools/). Same UI, same two tabs.

</Steps>

The mechanics of those two tabs land later in the course — the render-tree drill arrives with the React unit, the Profiler returns in the performance unit. Install once, ignore for now.

## Elements — what's actually rendered?

The Elements panel shows the <Term definition="The browser's tree of typed element nodes as it stands right now — including every change React or any other JavaScript has applied since the page loaded. Different from the source HTML the server returned.">live DOM</Term>: the in-memory tree as it stands at this exact moment, after every JavaScript mutation. That distinction matters. The source HTML the server originally sent is a different thing — you read it with View Source (Ctrl/Cmd+U) — and on a React app, the two will routinely disagree. That disagreement is the *expected* mechanics, not a bug.

The right pane of the Elements panel shows the cascade for whatever element you've selected: every CSS rule that targeted it, listed in cascade order, with overridden rules struck through. This is how an experienced engineer reads the cascade in practice. The cascade *itself* — how specificity works, what the layers are, why one rule wins — gets its dedicated chapter later in the course, when Tailwind lands. Here you're learning to *read* it.

<Figure caption="Elements panel — live DOM on the left, the cascade on the right. Struck-through rules lost the cascade; the Computed tab resolves every rule down to one final value per property. The `:hov` button lets you force pseudo-states without using the mouse.">
  <ElementsPanelOverview />
</Figure>

Five moves carry the panel's daily value:

- **Inspect to find.** `Cmd+Shift+C` on macOS, `Ctrl+Shift+C` elsewhere, toggles the inspector cursor. Click any pixel on the page and the corresponding DOM node lights up in the tree. This is the reflex when a user says "this thing isn't styled right" — you start at the pixel, not at the source file.

- **Read the cascade in the Styles pane.** Rules are listed top-down by which one won; overridden rules are struck through. The struck-through line tells you *why* your rule didn't apply — usually specificity or order. When the Tailwind chapter teaches the layer ordering later in the course, this panel is where you'll read it.

- **Edit styles live, no reload.** Click any rule's value to edit it in place. The page updates as you type. The experienced engineer's heuristic: *if it works in DevTools, it'll work in the code.* Make the change here first, confirm it does what you want, then go write it. You'll save more time over a career than you'd believe.

- **Toggle <Term definition="The colon-prefixed states an element can be in — :hover, :focus, :active, :focus-visible, :focus-within. Toggling them in DevTools forces the state without needing the mouse or keyboard interaction.">pseudo-states</Term> explicitly.** Click `:hov` in the Styles pane to force `:hover`, `:focus-visible`, `:active`, or `:focus-within` on the selected element. The naive instinct is to hover the element directly to inspect its hover style — but moving the mouse to the panel kills the hover state. Forced toggles are the way through.

- **Computed tab for the final value.** When the cascade gets dense and you stop caring *why* a value won, the Computed tab shows the single resolved value the browser ended up using, plus which selector won it. Use this when the question is "what *is* the color, regardless of where it came from."

Try it now. Open the page you have in front of you, inspect any button, and click `:hov` in the Styles pane to toggle `:hover`. Watch the styles change without your mouse touching the button. Then edit the `background-color` value in the Styles pane directly — set it to `red`, or anything. The button updates with no reload. That's the dev loop tightened from minutes to milliseconds.

## Network — what did the server send back?

The Network panel logs every request the page makes — Document, Fetch/XHR, JS, CSS, Images, Fonts, WebSockets — and surfaces each as a row with timing, headers, payload, and response. Lesson 1 of this chapter used the Protocol column and the Timing breakdown; that was a slice of one panel. The rest of it — the toolbar checkboxes, the throttle, the request-detail tabs — is what you'll actually live in when you debug a SaaS app.

<Figure caption="Network panel — every request, every header, every payload. The toolbar checkboxes and the right-pane tabs carry the daily workflow.">
  <NetworkPanelOverview />
</Figure>

Seven moves separate the typical workflow from the experienced one:

- **Open the panel *before* the action that triggers the request.** Network only captures while it's open. Open afterwards and the request you wanted to see is already gone. The reflex is open first, then click — easy to forget the first dozen times.

- **Turn on Preserve log.** A redirect or a full-page navigation wipes the request log by default. With Preserve log on, every request stays visible across navigations and reloads. Turn it on as a permanent default in your DevTools settings; you'll never want it off.

- **Tick Disable cache while DevTools is open.** Otherwise the cached request returns in 1ms and you'll conclude your change worked when nothing actually went over the wire. The next user (cold cache) will see the old behavior. Reflex: cache off whenever the panel is open.

- **<Term definition="Network conditions simulated by DevTools to mimic a slower connection from the browser side. 2026 presets: Fast 4G (≈9 Mbps down, 40ms RTT), Slow 4G (≈3 Mbps, 150ms RTT), 3G (≈400 Kbps, 400ms RTT), Offline. A per-request override shipped in late 2025 via right-click on a row → Override request.">Throttle</Term> to "Slow 4G" or "3G" when testing loading states.** The 2026 presets are "Fast 4G", "Slow 4G", "3G", and "Offline" — Chrome 130+ renamed the old "Fast 3G" / "Slow 3G" presets to match modern device baselines. The reflex for "this looks fine on localhost": flip to Slow 4G, reload, watch the skeleton states actually breathe. A per-request override (right-click a row → Override request → Throttle) shipped in late 2025, useful for testing one slow API call against an otherwise fast page.

- **Filter by type to drop the noise.** *Doc* shows just the page navigation requests. *Fetch/XHR* shows your application's API calls. Most SaaS debugging happens in Fetch/XHR; the chip cuts the waterfall from hundreds of rows to the dozen that matter.

- **Read the right pane in this order.** Headers (auth, content type, status), then Payload (what your client sent), then Response (what the server sent back — the Preview tab pretty-prints JSON for you), then Timing (where the time went — folds onto the four stages from lesson 1 of this chapter). Going in any other order wastes clicks.

- **Right-click → Copy → Copy as fetch / Copy as cURL.** This is the move that turns a captured browser request into a debuggable reproduction. Paste the fetch into the Console and re-run it; paste the cURL into a terminal and re-run it. Either way you've got a JavaScript or shell version of the exact request the browser sent, ready to edit. The full `fetch` API gets its dedicated chapter later in this unit; this is your first preview of the shape it produces.

Open Network on the page you have in front of you. Tick **Preserve log** and **Disable cache**. Reload. Find the document request row at the top, right-click it, choose **Copy → Copy as fetch**. Paste it into the Console (you'll learn the Console properly in the next section). What you just pasted is the same request the browser made, expressed as JavaScript you can edit and re-run.

## Console — what does the page think it knows?

The Console is a <Term definition="Read-Eval-Print Loop — an interactive shell that evaluates one expression at a time and prints its result. The browser's JavaScript REPL runs inside the page itself, so it can read and mutate the page's actual state.">REPL</Term> inside the running page. Anything in the global scope is reachable; anything any script `console.log`s lands here; anything you type runs in the page's own JavaScript context. It's also the most under-used tool in DevTools — most newcomers call `console.log()` and stop there. The Console has a vocabulary, and learning it is the single highest-leverage move on this whole tour.

<Figure caption="Console — REPL inside the page, with log-level filtering. `console.table` and `$0` are two of the moves that separate the typical workflow from the experienced one.">
  <ConsolePanelOverview />
</Figure>

Seven moves to internalize:

- **Log levels and filtering.** `console.log` is the default; `console.info`, `console.warn`, and `console.error` print at their named levels. The dropdown at the top of the panel lets you drop everything but errors when a noisy page is talking to you. The reflex for production debugging output: use `console.warn` and `console.error` so the signal stays filterable amid the noise.

- **`console.table(arrayOfObjects)`.** Renders an actual table with one column per object key. Use this instead of `console.log` for any array of objects, ever. The cost of expanding `[Object, Object, Object, ...]` rows manually adds up over a career.

- **`console.dir(domNode)`.** Prints the full JavaScript property tree of a DOM node — properties, methods, internals — rather than the rendered HTML representation, which is what `console.log(node)` gives you. Reach for it when the question is "what JavaScript properties does this node carry?" rather than "what does it look like in the tree?"

- **`console.trace()`.** Prints the stack trace at the call site. Use it when "who called this?" is the question and you don't want to set a breakpoint to find out.

- **`$0`, `$1`, `$2`, `$3`, `$4`.** References to the last five elements you selected in the Elements panel. Click a node in Elements, switch to Console, type `$0.getBoundingClientRect()`. This is the most useful bridge between two DevTools panels, and the one most often missed.

- **`copy(value)`.** Copies any value to your clipboard. `copy($0.outerHTML)` grabs the live rendered markup of the inspected element. `copy(JSON.stringify(state, null, 2))` snapshots any complex value as pretty JSON, ready to paste into a bug report or test fixture.

- **Live evaluation as you type.** Start typing `document.queryS…` and DevTools shows you the result of the current expression in a faded preview before you hit Enter. Useful for refining a selector against the live DOM before pasting it into your code.

Those last few names — `$0` through `$4`, `copy`, plus `$_` for the last evaluated expression — are <Term definition="DevTools console-only helpers — short variables and functions that exist only in the Console, not in your scripts. $0 to $4 reference the last five elements selected in Elements; copy() puts any value on the clipboard; $_ holds the last evaluated expression. They're injected into the global scope of the Console and won't work in production code.">console utilities</Term>. DevTools injects them into the Console's global scope; they don't exist in your application code. Try to use `copy()` in a script file and you'll get a `ReferenceError`. That distinction is occasionally useful when you're debugging a copy-paste that works in Console and not in a `.js` file — the utilities don't travel.

The vocabulary, in five lines:

```js
console.table(users);        // an array of objects rendered as a real table
console.dir($0);             // full JS property tree of the inspected element
console.trace();             // stack at this line
copy($0.outerHTML);          // live rendered markup → clipboard
copy(JSON.stringify(state)); // snapshot any value as pretty JSON
```

Try the bridge right now. Switch to Elements, inspect any element on the page. Switch to Console, type `$0` — the element prints. Type `console.dir($0)` and watch its full JavaScript surface unfold. Type `copy($0.outerHTML)` and paste anywhere — the live rendered markup is on your clipboard.

## Application — what's in storage right now?

The Application panel is the storage and identity inspector. Cookies, Local Storage, Session Storage, IndexedDB, Cache Storage, Service Workers — every place the page persists data is here, and every entry can be inspected, edited, or deleted from the panel.

For day-to-day SaaS work, the surface that matters is **Cookies** (where session cookies live, when the auth chapters land) and **Local/Session Storage** (where client-state tooling and URL-state lists keep working values). The rest you'll see once or twice over the course of a project; named here so the geography is on your mental map.

<Figure caption="Application panel — every storage surface and identity slot the page touches, all editable from here. The Cookies row's attribute columns are where most auth bugs live.">
  <ApplicationPanelOverview />
</Figure>

Four regions to know:

1. The sidebar with **Manifest**, **Service Workers**, **Storage** (and its sub-items Local Storage / Session Storage / IndexedDB / Cookies / Cache Storage), and **Background Services**.
2. The **Clear site data** button under Storage at the top of the section.
3. The Cookies row in Storage — when selected, the right pane lists every cookie with its full attribute set.
4. The right pane's cookie attribute columns: `HttpOnly`, `Secure`, `SameSite`, Expires, Domain, Path.

Five moves to know:

- **Cookies — the auth surface.** Expand **Cookies** under **Storage** and click the origin you care about. The right pane lists every cookie with its full attribute set: name, value, domain, path, expires/max-age, `HttpOnly`, `Secure`, `SameSite`. Edit a value in place; double-click any attribute to flip it; delete a row with the Delete key. When the auth chapters arrive later in the course, this is where you'll read the session cookie. The reflex when "auth doesn't work": check the cookie's `SameSite` and `Secure` flags before you check the server logs. The semantics of those attributes — when they apply, why they block — get their dedicated chapter later in this unit.

- **Local Storage and Session Storage.** Same edit-and-clear surface, keyed by origin. Local Storage persists across tabs and reloads; Session Storage is per-tab and clears when the tab closes. When the URL-state list project arrives later in the course, the values you're stashing show up here.

- **IndexedDB.** Persistent client-side database. Surfaces here; mentioned in one line because when offline state earns its weight on a project, this is the panel. Not on the daily path for the 2026 SaaS stack — the platform's data fetching covers most of the surface.

- **Service Workers and Cache Storage.** Service workers register under their own sidebar entry; whatever they've cached shows in Cache Storage. Named here so you can find them; the course doesn't ship a service worker on this stack, because Server Components and the platform's data fetching cover the use cases.

- **<Term definition="The DevTools button under Application → Storage that wipes every persistence surface for the current origin: cookies, local/session storage, cache storage, IndexedDB, service workers. The fix for 'works in incognito, breaks here' — your local state has drifted from the server's expectations.">Clear site data</Term> — the nuke button.** One button at the top of the Storage section wipes cookies, every storage flavor, cache, service workers, the lot, for the current origin. It's the fix for "it works in incognito but not here" — your local state has drifted from the server's expectations and clearing is faster than diffing. Reach for it without ceremony.

Try the cookie inspector right now. Open Application → Storage → Cookies on the page you have in front of you. Click your origin. Read the attribute columns on any cookie. Look at `SameSite`, `Secure`, `HttpOnly` — those four characters in the wrong combination are why an authenticated request can silently drop its cookie on a cross-site fetch. The chapter later in this unit owns the production defaults for those attributes; the Application panel is where you'll diagnose them when something goes wrong.

## The panels we're deliberately skipping

:::note[Panels named, not toured]
A handful of other DevTools panels exist; this lesson skips them on purpose. You should know where they live so you can find them when you need them.

- **Performance and Lighthouse.** Both surface Core Web Vitals and performance traces. Both are tuned in the performance unit later in the course, against a real app. Measuring performance on an empty teaching page is theatre — you can't tune what you haven't built.
- **Sources.** The browser-side debugger — breakpoints, watch expressions, scope. The course's day-to-day is heavy on Server Components and Server Actions where the debugger lives in the Node process, not the browser. Reach for Sources when you're debugging Client Component logic at production depth; otherwise the Console covers most needs.
- **Memory, Recorder, Security.** Memory for heap snapshots when a leak actually earns the dig. Recorder for capturing a flow and replaying it as a Puppeteer script. Security for the connection's TLS state at a glance. Not on the daily path.
- **Device Mode.** The viewport simulator that toggles mobile / tablet / desktop sizes. It lives in the top-left of DevTools (the laptop-and-phone icon, or `Cmd+Shift+M` / `Ctrl+Shift+M`). The responsive Tailwind chapter later in the course owns the workflow.
:::

## What stays out of this lesson

This is a survey, not a manual; the panels each have their own deeper terrain owned elsewhere.

- Cookie attribute semantics — `HttpOnly`, `Secure`, `SameSite` — are taught in their dedicated chapter later in this unit. The Application panel surfaces them; the chapter explains them.
- The CSS cascade itself, specificity, the box model, and Tailwind layer ordering arrive in the styling unit later in the course. Elements is where you'll *read* them.
- The `fetch` API, request and response shapes, error handling — covered later in this unit. The Network panel reads them.
- React component tree inspection, prop edits via DevTools, Profiler flame charts — covered in the React unit and again in the performance unit. React DevTools is installed; the depth waits.
- Core Web Vitals, Performance flame charts, Lighthouse audits, bundle analyzers — covered in the performance unit. The panels exist; the workflow lives there.

The geography is in your head. The depth comes from doing the work, which the next lesson nudges further along — a local HTTPS setup that unblocks the secure-context APIs the rest of this unit depends on.
