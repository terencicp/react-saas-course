A teammate ships `const config = { feature: true }` at the top of a module and considers the value protected. Another module imports `config`, sets `config.feature = false` mid-request to short-circuit a code path, and the user-facing flag flips silently for everyone touching that request. `const` did nothing wrong — it never promised to freeze the value. It only promised the *name* would keep pointing at the same object.

A different week, a different PR: a binding was declared `let` "just in case" six months ago, never reassigned, and finally gets reassigned by an unrelated change. Half the rendering pipeline downstream had read the original value once and assumed it wouldn't move. Two bugs, one root: misunderstanding what `const` actually promises and reaching for `let` without earning it.

By the end of this lesson you'll have the one-sentence rule that closes both bug classes, the block-scope and Temporal-Dead-Zone semantics that keep modern JavaScript predictable, and the senior reflex of `const`-by-default that every later file in this course writes without thinking. The binding diagram from the first lesson of this chapter is the spine again — `const` locks the arrow, not the box.

## `const` locks the arrow, not the box

Here is the rule the rest of the lesson hangs on: `const` is a <Term definition="A name that points to a value. In `const x = 1`, `x` is the binding; `1` is the value.">binding</Term> that cannot be **reassigned**. It says nothing about whether the **value the binding points at** can be mutated. Two different promises about two different things, and `const` only makes the first one.

The picture is the one from the chapter opener. Two bindings, an arrow each, and either a value box or a shared object on the other end.

<BindingDiagram />

`const` makes that arrow permanent. You can't repoint the arrow at something else — that's what "no reassignment" means. But the object box on the right-hand panel? `const` has no opinion about it. If the value at the end of the arrow is mutable (and every object, array, `Map`, `Set`, and class instance is), then any code holding a reference can reach in and change its contents. The arrow stays put; the box changes underneath.

The two-line consequence is the production bug from the opener compressed into a snippet.

```ts
const config = { feature: true };
config.feature = false;
// config = { feature: false };
```

The second line works. It's a mutation through the existing arrow — reaching into the object the arrow points at and changing one of its properties. `const` has nothing to say about that. The third line, if you uncommented it, would throw `TypeError: Assignment to constant variable.` That one is a reassignment — an attempt to repoint the arrow at a *different* object. That's what `const` blocks.

For a primitive, the distinction collapses, because primitives have no inside to mutate — there's no way to "change the contents" of `42` or `'hello'`. So `const x = 42` does behave as if the value itself is frozen, just because the value can't be mutated to begin with. The illusion of `const = frozen value` works for primitives and breaks the moment the value is an object. That's where most of the bugs come from: the mental model that worked for `const PORT = 3000` quietly fails on `const config = { feature: true }`.

The defense isn't `Object.freeze` (we'll get to why later in this lesson). The defense is the immutability reflex from the first lesson of this chapter — copy-then-modify with spread or `structuredClone` — plus the compile-time guards TypeScript ships, which Chapter 4, "Typing values you know," lands properly. For now, hold onto the rule: **the arrow is locked, the box is not.**

## `const` by default, `let` when the binding must change

The senior reflex is one keyword: write `const` for every binding unless the binding genuinely has to be reassigned. Not "unless the value might change" — the value is the box, and `const` doesn't care about the box. Unless the *binding itself* needs to point at something different later. That's the only trigger for `let`.

In practice, `let` earns its place in a small, well-defined set of cases:

- An **accumulator** in a manual loop — `let total = 0;` that you increment as you walk a list. The binding has to reassign on every iteration; that's the loop. (Though the experienced reach when the shape fits is `.reduce`, which sidesteps the mutable binding entirely.)
- A **value that depends on a branch** too gnarly for a ternary — `let role; if (...) { role = 'admin'; } else if (...) { role = 'editor'; } else { role = 'viewer'; }`. Rare in well-shaped code; usually a sign the branches should be extracted into a helper that returns the value, after which the call site is `const role = pickRole(...)` again.
- A **loop counter** in a classic `for (let i = 0; ...; i++)`. Used less often than you'd think in 2026 code — `for...of`, `.map`, `.filter`, `.reduce`, and `Array.from({ length })` cover most cases — but a legitimate `let` when the index is genuinely what you need.

That's the list. Outside of those shapes, if you wrote `let` and never reassigned, change it to `const`. You won't have to enforce this manually — Biome's `useConst` rule (already enabled in the chapter's `biome.json` from Unit 1) catches the pattern at build time and tells you exactly which `let` to convert. The reflex is still worth installing as a reading habit, because reviewing a PR is a place the lint hasn't run yet and you want to see `let` and immediately ask "where does this get reassigned?"

A quick tour of the same accumulator in three forms — the form to leave behind, the form to reach for occasionally, and the form an experienced engineer reaches for first.

The pattern across the three tabs is the chapter's broader thesis showing up at the keyword level. The default reach is the most constrained form (`const` plus a method that doesn't need mutation), the acceptable fallback is the next step up (`let` plus a manual loop), and the legacy form (`var`) belongs in code you're rewriting, never in code you're authoring. Hold the same shape across every binding decision in the course and the cognitive load of reviewing your own code drops to nothing.

## Block scope and the Temporal Dead Zone

Two mechanical rules sit underneath `const` and `let`, and they're tightly enough linked that an experienced engineer thinks of them together. The first is **block scope**: every `const` and `let` binding is scoped to the nearest pair of curly braces enclosing it — a function body, an `if` branch, a `for` loop, an arrow-function body, or a bare `{ ... }` block expression. The scope is **lexical**, meaning it's determined by where the binding is written, not by the call stack that reaches it at runtime.

```ts
if (user.isAdmin) {
  const tier = 'pro';
  console.log(tier);
}

// console.log(tier); // ReferenceError — tier is not visible out here
```

That's the rule. The binding lives inside its block and nowhere else. `var` is the exception that broke this — `var` is function-scoped, so a `var` declared inside an `if` block leaks out to the surrounding function, which is the legacy bug class block scope was designed to kill.

The second rule is what stops a different bug class. Every `const` and `let` binding enters a state called the <Term definition="The period between entering a scope and reaching a let/const declaration. The binding exists (the parser saw it) but accessing it throws a ReferenceError.">Temporal Dead Zone</Term> (TDZ for short) the moment its scope starts, and stays there until the line that declares it runs. While a binding is in the TDZ, the *name* exists — the parser has already seen the declaration — but **accessing it throws a `ReferenceError`**. The engine refuses the read.

The TDZ exists because the language is protecting you from `var`'s old footgun. With `var`, accessing the binding before the declaration line silently returned `undefined`, because the engine hoisted the name and initialized it to `undefined` before any code ran. That meant a typo or a wrongly-ordered statement gave you `undefined` at runtime instead of an error — and `undefined` propagates through everything until it surfaces somewhere unrelated. The TDZ makes the same access fail loudly at the moment the bad code runs, which is the only place you can still cheaply fix it.

Seeing the engine refuse the access lands the concept harder than reading about it. The fix isn't to swap to `var` to make the error go away — the fix is to declare the binding before you use it. Which leads to the senior reflex worth installing now: **declare bindings at the top of the scope where they're used**. The TDZ rewards readers who scan a function top-down — if a binding is referenced, its declaration is above the reference. You almost never have to think about the TDZ once that habit is in place.

One paragraph on <Term definition="The compile-time process where the engine collects every name in a scope and pre-registers it before any code runs. For `var`, the name is initialized to `undefined`; for `let`, `const`, and `class`, the name is registered but unreachable until the declaration line (the TDZ).">hoisting</Term>, which is the umbrella term for what's actually happening here. In 2026 JavaScript, `const`, `let`, and `class` hoist their **name** but not their value — the binding exists from the top of the scope but is unreachable until the declaration line runs (that's the TDZ). `var` hoists its name **and** pre-initializes it to `undefined`, which is the bug class TDZ was designed to kill. Function declarations are the third form: they hoist their entire body, which is why you can call a `function foo() {}` declared at the bottom of a file from the top. You won't need any of this in day-to-day 2026 code — declare bindings before you use them and the whole machinery stays invisible.

:::note
You almost never need to think about hoisting in 2026 code. Declare every binding before you reference it and the TDZ never fires. The forms that genuinely surprise — `var` returning `undefined` before its declaration, function declarations called above their definition — aren't on the course's path; you'll only meet them in legacy code you're rewriting.
:::

## `Object.freeze`, `readonly`, and why the course skips runtime freezing

`Object.freeze` is the standard-library tool that does what `const` doesn't: it makes an object's top-level properties read-only **at runtime**. Once frozen, any attempt to write to a property either silently no-ops (in non-strict mode, which the course never runs in) or throws `TypeError` (in strict mode, which all modules and class bodies use by default). The freeze is **shallow** — same shape as the spread copies from the first lesson — so nested objects stay mutable.

```ts
const config = Object.freeze({ feature: true, env: { tier: 'pro' } });

// config.feature = false; // TypeError in strict mode — top-level write rejected.
config.env.tier = 'free'; // allowed — the freeze is shallow.
```

The course names `Object.freeze` so you recognize it in the wild, and then doesn't reach for it. The reason is straightforward: TypeScript's `readonly` modifier and `as const` (both landed properly in Chapter 4, "Typing values you know") provide the same "this can't be reassigned" guarantee at **compile time**, with **zero runtime cost**. The bug becomes a red squiggle in your editor before you ever run the code, instead of a `TypeError` that surfaces in production. `Object.freeze` is the right tool for the rare case where a runtime guarantee is genuinely required — usually a module exporting a config object that third-party code receives and could tamper with — but inside a SaaS app where you control both ends of every call, the typing tools win every time.

The senior reach: `readonly` and `as const` for compile-time immutability; `Object.freeze` only when a runtime guarantee is genuinely required, which inside an app you own end-to-end is almost never.

## `const` doesn't narrow types

One last value-model observation before the exercise, because it lands a reflex you'll need the moment TypeScript shows up in earnest. `const` on a primitive narrows the inferred type to the literal value. `const` on an object widens it. That asymmetry catches everyone the first time, and the fix is the same `as const` reach you just heard named.

Hover the underlined names in the two snippets below to see what TypeScript actually infers.

    On an object, `const` does **not** carry the narrowing through to the properties. TypeScript infers `config.feature` as `boolean`, not `true`, because the object is mutable — someone could write `config.feature = false` later (which `const` doesn't prevent, as you just saw). The type checker widens conservatively to reflect that reality.

The fix when you actually want the literal types is `as const`, which you'll meet properly in Chapter 4, "Typing values you know." One-line preview so you recognize the shape:

```ts
const config = { feature: true } as const;
// inferred type: { readonly feature: true }
```

Two things flipped: every property is `readonly` (the compile-time `Object.freeze`), and every primitive value narrows to its literal type. The chapter that owns `as const` goes into how the inference algorithm actually works; what matters here is the trigger — when you write `const` on an object literal and want TypeScript to lock the property types as well as the binding, `as const` is the reflex. The seed of that reflex starts at the value-model level, which is why it earns a mention in this chapter.

## Check yourself

Eight real bindings, three buckets. The first two buckets are the easy classifications; the third is for the cases where the value *looks* like it's changing but the binding doesn't — so `const` is the only correct call, even though a careless reader might reach for `let` on reflex. The senior install is recognizing those last two as `const` without a second thought, not as a tradeoff.

The two third-bucket cases are the senior install, and they aren't tradeoffs — `const` is the only sensible call in both. A `for...of` loop variable feels like it's changing, and the *value* is, each iteration — but the language creates a brand-new binding scoped to the loop body on every pass, so the name is never reassigned. Write `let` there and you're claiming a reassignment that doesn't exist; Biome's `useConst` rule flags it on the spot. An array built with `.push` mutates the array in place, but the arrow keeps pointing at the same array, so the binding never reassigns either. Both are `const`, full stop. Reach for `let` only when you'd genuinely write `binding = somethingElse` somewhere in the body.
