A teammate opens a pull request that updates a user record before sending it to an analytics helper. The update ships, and within an hour two pieces of UI on the dashboard start drifting out of sync — the sidebar shows the old name, the header shows the new one, and only a refresh aligns them. The diagnosis is mundane and unforgiving: the helper mutated the same object the calling component still held a reference to. The bug class behind this — and behind a long catalogue of "why did my state silently change?" stories — is treating `=` as if it duplicates a value, when what it actually does is bind a name to one. By the end of this lesson you'll know exactly which values are shared and which aren't, and the fix won't be a defensive-copy reflex — it'll be a mental model precise enough that you can predict the behavior before you run the code.

## Names and values

Here is the rule the rest of the lesson is built on: in JavaScript, `=` does not duplicate the value on its right. It binds the name on its left to that value. Whether anything gets copied depends entirely on what kind of value it is.

JavaScript splits its values into two categories, and the split is the whole story.

**Primitives** are values JavaScript holds directly. The seven of them are `string`, `number`, `boolean`, `bigint`, `symbol`, `null`, and `undefined`. When you assign a primitive to one name and then to another, the value itself is copied; the two names are independent from that moment on.

**Objects** — and that includes arrays and functions — are the other category. The name doesn't hold the object; it holds a <Term definition="A pointer-like value that identifies an object; assigning a reference doesn't duplicate the object it points at.">reference</Term> to it. The reference is a small pointer-like value identifying where the object lives. When you assign an object to another name, what gets copied is the reference, not the object. Both names now point at the same object.

<BindingDiagram />

The consequence of the right panel is what gets people. If `user` and `alias` both point at the same object, then mutating through one is visible through the other:

```ts
const user = { name: 'Ada', age: 36 };
const alias = user;

alias.name = 'Grace';
console.log(user.name); // 'Grace'
```

Notice that `const` did not prevent this. `const` locks the binding — you can't reassign `alias` to point at something else — but it says nothing about whether the value the binding points at can change. That distinction has its own lesson later in this chapter; for now, hold onto the diagram.

## What functions do with the values you pass them

A function call is just another binding. Each parameter is a fresh name on the function side, and calling the function binds that name to whatever value you passed. So the same primitive-vs-reference split applies — and it resolves the perennial "is JavaScript pass-by-value or pass-by-reference?" debate in one sentence.

**JavaScript is always pass-by-value. The value being passed for an object is a reference.**

That sounds like wordplay, but it's the difference between two bug classes every engineer has shipped at least once. Compare the two ways a function can try to "change" a user.

<CodeVariants>
  <CodeVariant label="Reassigning the parameter">
    <div data-mark-color="blue">

    ```ts {2}
    function rename(user: { name: string; age: number }) {
      user = { name: 'Grace', age: 36 };
    }

    const ada = { name: 'Ada', age: 36 };
    rename(ada);
    console.log(ada.name); // 'Ada'
    ```

    </div>
    Inside `rename`, the parameter `user` is rebound to point at a brand-new object. That rebinding only touches the function's local name. The caller's `ada` binding is untouched — it still points at the original object, which was never modified.
  </CodeVariant>

  <CodeVariant label="Mutating a property">
    <div data-mark-color="orange">

    ```ts {2}
    function rename(user: { name: string; age: number }) {
      user.name = 'Grace';
    }

    const ada = { name: 'Ada', age: 36 };
    rename(ada);
    console.log(ada.name); // 'Grace'
    ```

    </div>
    Inside `rename`, the parameter `user` and the caller's `ada` both point at the same object. The function doesn't rebind anything — it reaches through its local reference and changes a property on the shared object. The mutation is visible everywhere that object is held.
  </CodeVariant>
</CodeVariants>

The rule extracted from both tabs: a function reassigning its parameter is invisible to the caller; a function mutating a property the parameter points at is visible to the caller. Every engineer has shipped both at least once. The fix is the model, not memorization — once you can see the diagram in your head, the call-site behavior is mechanical.

## Shallow copy: the daily reach

You usually don't want to mutate a value that came from somewhere else — a function argument, a piece of state, a row you fetched from the database — because that mutation will travel back through every reference still pointing at it. But you do want to produce a modified version. The standard move is to copy first, then modify the copy. The 2026 default for that copy is the spread operator.

```ts
const next = { ...prev };
const nextItems = [...items];
```

That's it. For objects, `{ ...prev }` builds a new object with the same own enumerable properties. For arrays, `[...items]` builds a new array with the same elements. Both are one keystroke past the value you're copying, and both are what an experienced engineer reaches for nine times out of ten.

You'll occasionally see `Object.assign({}, prev)`. It does the same thing and is worth recognizing in older code; the only reason to prefer it today is a rare interaction with `Object.defineProperty` semantics, which the course doesn't reach for. `Array.prototype.slice()` shows up for the same reason on `arguments`-like objects you can't spread — also rare, also worth recognizing.

The word "shallow" is doing real work in "shallow copy." The spread copies the top level — but for any nested object, the copy gets the original's *reference*. The nested object isn't duplicated; both the original and the copy point at it. Mutate through one, the other sees it. Let's prove it.

<ScriptCoding

  name: 'Ada',

const copy = { ...original };

// Predict each value below. Replace null with the string you expect.

  originalName: null,   // what does original.name hold now?

  originalStreet: null, // what does original.address.street hold now?
  copyStreet: null,     // what does copy.address.street hold now?
};
`}
  tests={`
test("Top-level rename on the copy doesn't touch the original", () => {
  expect(prediction.originalName).toBe(original.name);
});
test('The copy holds the new name', () => {
  expect(prediction.copyName).toBe(copy.name);
});
test('Nested mutation leaks back to the original', () => {
  expect(prediction.originalStreet).toBe(original.address.street);

});

/>

<details>
<summary>Reveal the answer</summary>

```ts
const prediction = {
  originalName: 'Ada',
  copyName: 'Grace',
  originalStreet: 'Babbage Ave',
  copyStreet: 'Babbage Ave',
};
```

The spread copied the top level, so renaming `copy.name` left `original.name` as `'Ada'`. But `address` was a reference, copied as a reference — both `original.address` and `copy.address` point at the same nested object, so mutating `copy.address.street` is mutating it for `original` too.

</details>

Shallow is almost always what you want. Most state shapes you'll work with are flat or only one level deep, and a spread is enough to isolate the mutation. Reaching for a deep copy when you don't need to is wasteful — both in keystrokes and in actual work the runtime does. The only time you should escalate to a deep copy is when the value is genuinely nested and you can't predict where the mutations will land.

## structuredClone: the deep copy default

When the data shape is genuinely nested — an object with objects in it, a record holding a `Map`, a tree — a spread doesn't protect the inner references and the mutation will bleed. The 2026 answer is `structuredClone`, a global function built into Node 24 LTS and every browser the course targets. No import, no library, no hand-written tree walker.

<CodeVariants>
  <CodeVariant label="Spread (shallow)">
    <div data-mark-color="orange">

    ```ts {6}
    const original = {
      name: 'Ada',
      address: { street: 'Lovelace Ln', city: 'London' },
    };

    const copy = { ...original };
    copy.address.street = 'Babbage Ave';

    console.log(original.address.street); // 'Babbage Ave'
    ```

    </div>
    The spread copied the top level, but `address` is still shared. Mutating through `copy.address` mutates the original too.
  </CodeVariant>

  <CodeVariant label="structuredClone (deep)">
    <div data-mark-color="green">

    ```ts {6}
    const original = {
      name: 'Ada',
      address: { street: 'Lovelace Ln', city: 'London' },
    };

    const copy = structuredClone(original);
    copy.address.street = 'Babbage Ave';

    console.log(original.address.street); // 'Lovelace Ln'
    ```

    </div>
    `structuredClone` walks the whole structure and copies every nested object independently. There is no shared reference left between `original` and `copy`, so mutations stay where you make them.
  </CodeVariant>
</CodeVariants>

`structuredClone` does more than walk objects recursively. It handles <Term definition="An object whose property graph contains a path back to itself; a naive recursive copy would loop forever.">cyclical references</Term> — an object that contains a reference back to itself, which would crash a naive recursive copy — and it preserves the types that actually matter in production: `Date`, `Map`, `Set`, `ArrayBuffer`, typed arrays, and `RegExp`. It is also typically faster than the alternative you may have seen in older code (more on that in the next section).

What it can't clone is anything that carries behavior or identity beyond the data it represents.

:::caution
`structuredClone` throws `DataCloneError` if it encounters:

- Functions. You can't clone behavior.
- Class instances. The data survives, but the prototype chain doesn't — your `User` instance comes out the other end as a plain object.
- DOM nodes, `WeakMap` and `WeakSet`, and property getters or setters.

The senior rule: if `structuredClone` throws, the value is carrying behavior, and behavior shouldn't be cloned anyway. The fix is to clone the data and reattach the behavior — not to engineer around the clone.
:::

The mental model you're installing here — "what can cross a serialization boundary?" — is the same one the course will lean on later for Server Actions. Next.js serializes the arguments you pass into a `'use server'` function through a superset of this algorithm, which is why `structuredClone` is the right canonical frame even though you won't call it constantly.

## The JSON.parse(JSON.stringify(x)) you'll see in legacy code

Before `structuredClone` shipped natively, the workaround pattern was the same one-liner everywhere:

```ts
const copy = JSON.parse(JSON.stringify(original));
```

It does deep-copy. It also silently mutates the data on the way through, because `JSON.stringify` only knows about the JSON data model — anything outside it gets lost or corrupted. That's occasionally useful (when you want to force a value into wire format) and almost always a bug.

The watch-out is worth feeling rather than reading.

<PredictOutput expected={"{\n  created: '2026-01-15T10:00:00.000Z',\n  tags: {},\n  note: 'Pay before EOM'\n}"}>

```ts
const original = {
  created: new Date('2026-01-15T10:00:00Z'),
  tags: new Set(['urgent', 'invoice']),
  ratio: undefined,
  note: 'Pay before EOM',
};

const copy = JSON.parse(JSON.stringify(original));

console.log(copy);
```

<PredictWhy>
`JSON.stringify` walks the value and converts each piece to its JSON form: a `Date` becomes its ISO-string representation, a `Set` becomes a plain object with no enumerable own properties (and therefore an empty `{}`), and an `undefined` property is dropped entirely (JSON has no `undefined`). When `JSON.parse` reads the result back, every type that was converted comes back as its replacement — there is no way to recover `Date`-ness from a string, or `Set`-ness from `{}`. Reach for `structuredClone` unless you specifically want JSON's lossy normalization.
</PredictWhy>

</PredictOutput>

Recognize the pattern when you see it in a codebase you've inherited. Don't write it.

## The React-shaped reflex this builds toward

React detects state changes by reference equality. When you call a state setter, React compares the new value to the old value with `===` — which on two objects asks whether they are the same reference (the topic of the next lesson), not whether they have the same shape. So if you mutate an array or object in place and pass the same reference back, React sees no change, skips the re-render, and your UI silently drifts from your data.

The reflex is **copy then modify**, never mutate in place. The shallow-vs-deep call you make here is the same one you'll make every time you update React state. Most state shapes are flat enough that a spread does the job; the rare deeply-nested case is where `structuredClone` (or a targeted nested spread) earns its weight.

The React-specific patterns — functional setters, `useReducer`, when teams reach for a library called Immer that lets you write what looks like mutation but compiles to copy-then-modify — belong in the React chapters. Plant the language-level reflex here and they'll feel obvious by the time you meet them.

## Check yourself

One exercise to confirm you can predict cross-boundary behavior without running the code. Don't pattern-match — apply the binding model line by line.

<PredictOutput expected={`2
Dr. Ada
Ada, Paris`}>

```ts
const count = 1;
const view = count;
console.log(count + view);

const ada = { name: 'Ada' };
function promote(user: { name: string }) {
  user.name = `Dr. ${user.name}`;
}
promote(ada);
console.log(ada.name);

const original = { name: 'Ada', address: { city: 'London' } };
const copy = { ...original };
copy.name = 'Grace';
copy.address.city = 'Paris';
console.log(`${original.name}, ${original.address.city}`);
```

<PredictWhy>
Walk each line against the binding model:

1. `view` was bound to the *value* of `count`, not to `count` itself. Primitives copy on assignment, so `view` holds its own `1`. The sum prints `2` — there's nothing pointing back at the original binding to disturb.

2. `promote` receives a reference to the same object `ada` points at. Mutating a property on that object is visible to the caller, so `ada.name` is now `'Dr. Ada'`.

3. The spread copied the top level of `original`, so renaming `copy.name` did not touch `original.name` (still `'Ada'`). But `address` was a reference copied as a reference — both objects share the same nested address — so writing to `copy.address.city` also changed `original.address.city` to `'Paris'`. The combined string is `'Ada, Paris'`.
</PredictWhy>

</PredictOutput>

If you predicted all three lines correctly, you have the binding model. From here, every later concept that touches "what gets shared and what doesn't" — Server Action arguments, React state updates, structural typing — reads against this same diagram.
