Two snippets that surprise almost every developer the first time they meet them. The first compares two objects with identical contents.

```ts
console.log({ id: 1 } === { id: 1 }); // false
```

The second compares a value to itself.

```ts
console.log(NaN === NaN); // false
```

JavaScript ships three equality operators and two edge cases the spec guarantees will surprise you. The senior reflex is to reach for exactly one operator by default and recognize the rest on sight. By the end of this lesson you'll write `===` without thinking, know the two cases where it lies, and know the rare moments to break the rule.

## How `===` compares the two kinds of value

`===` asks one question: are these the same value? The answer depends entirely on what kind of value you handed it — and that's the split you already have from the previous lesson on bindings. Primitives are values themselves; objects are references to a value that lives elsewhere. `===` respects that split.

For **primitives**, `===` compares the value itself. `'ada' === 'ada'` is `true` because the two strings *are* the same value — there's no notion of identity separate from the content. `42 === 42`, `true === true`, `null === null`: all `true`, for the same reason.

For **objects**, `===` compares the reference — the arrow in the binding diagram from the previous lesson, not the contents of the box it points at. Two object literals with identical contents are two separate allocations: two different boxes, two different arrows, so `===` returns `false`. The same object reached through two names is `===` to itself, because both names point at the same box.

```ts
console.log('ada' === 'ada');                       // true
console.log({ id: 1 } === { id: 1 });               // false
const user = { id: 1 };
const alias = user;
console.log(user === alias);                        // true
```

Hold onto this sentence: `===` answers "are these the same value?" for primitives and "are these the same allocation?" for objects. It never answers "do they have the same shape?" — that's a separate question the language doesn't ship an operator for, and one we'll come back to at the end of the lesson.

## Why the course never writes `==`

There's a second equality operator, `==`, and the course never writes it.

`==` applies a coercion table before comparing — numbers get pulled into strings, objects get pulled into primitives, `null` and `undefined` are considered equal to each other but not to `false`, and the special cases stack up from there. The full table is documented and nobody on a team remembers it correctly under deadline pressure. The one occasionally-defensible idiom (`x == null` to match both `null` and `undefined` at once) is unnecessary in this course's stack — the next chapter introduces the `??` operator and explicit nullish checks that read better and don't drag the rest of the coercion table along for the ride.

This isn't a discipline problem; it's a tooling problem the project has already solved. Biome's `noDoubleEquals` rule (enabled in the canonical `biome.json` you'll set up later in the course) flags every `==` and `!=` at lint time, so a slip is a build-time error, not something you have to remember in a code review.

:::caution
If you see `==` in legacy code or in code generated by an AI assistant, change it to `===` and re-test. The behavior will change — but it'll change to the behavior you actually wanted.
:::

## The two edge cases `===` gets wrong

`===` is the right default everywhere, but there are exactly two spots where its answer is going to surprise you. Predict the output of this snippet before reading on.

```ts
console.log(NaN === NaN);   // ?
console.log(+0 === -0);     // ?

```

<PredictWhy>

`NaN === NaN` is `false` because <Term definition="The standard that defines how JavaScript's `number` type stores and computes floating-point values. The cause of every floating-point surprise; covered in depth in the next lesson on money math.">IEEE 754</Term> — the floating-point standard JavaScript's `number` type is built on — specifies that `NaN` is not equal to anything, including itself. That spec choice is what makes `NaN` a useful "this calculation was invalid" marker: once a value is `NaN`, no accidental equality check ever "matches" it back into a valid bucket. The next lesson explains why JavaScript produces `NaN` values in the first place; for now the rule is observable — never compare a value to `NaN` with `===`.

`+0 === -0` (and `0 === -0`) is `true` because the spec defines `===` that way. Signed zero exists — it matters for graphics, scientific computing, and a handful of hash-key designs — but in everyday SaaS code the distinction is invisible until it isn't.

</PredictWhy>

Both behaviors are guaranteed by the spec across every JavaScript engine you'll ever target. They aren't bugs and they don't change — they're a small tax you pay for `===` being defined precisely enough to be portable.

## `Object.is`: the same as `===`, except for those two cases

JavaScript ships a third equality form: `Object.is(a, b)`. Despite the name, it has nothing to do with objects. It exists to plug exactly one gap — it behaves like `===` for *everything* except the two edge cases above. `Object.is(NaN, NaN)` is `true`, and `Object.is(+0, -0)` is `false`. Everywhere else, the two agree.

```ts
console.log(Object.is(NaN, NaN));     // true   — diverges from ===
console.log(Object.is(+0, -0));       // false  — diverges from ===
console.log(Object.is('ada', 'ada')); // true   — same as ===
console.log(Object.is({}, {}));       // false  — same as ===
```

Two real-world places `Object.is` actually earns its keep:

- **Custom memoization keys** where bit-pattern identity matters — for instance, a memoizer that needs to treat `NaN` as a valid input distinct from "no value."
- **React's reactivity bailout.** When you call a state setter, React uses `Object.is` (not `===`) to decide whether the value changed and the component needs to re-render. For objects, this is reference identity — which is why passing a freshly-spread object (`setUser({ ...user })`) re-renders even when the contents look identical. Different allocation, different reference, `Object.is` returns `false`. We'll come back to this rule in the React chapters; the takeaway here is that it's where `Object.is` actually shows up in the wild.

The senior reflex: default to `===` everywhere. Reach for `Object.is` only when one of those two cases applies — and when you do, add a comment explaining why, because the next reader will assume it's a typo for `===`.

## `Number.isNaN` over the global `isNaN`

The `NaN === NaN` rule means you can't write `if (x === NaN)` to check whether a value is `NaN`. You need a dedicated function for that — and JavaScript ships two of them. Only one is safe.

The global `isNaN(x)` is the legacy form, and it coerces its argument to a number before testing. That coercion is almost never what the caller meant. `isNaN('hello')` is `true` — not because `'hello'` is `NaN`, but because coercing `'hello'` to a number *produces* `NaN`. `isNaN(undefined)` is `true` for the same reason. The function answers a different question than its name suggests.

`Number.isNaN(x)` skips the coercion entirely. It returns `true` only when the value handed in is the actual `NaN` value, and `false` for everything else. Run the two side by side and you can read the divergence directly.

## Structural equality isn't in the language

Notice what we haven't talked about: comparing two objects for "same shape, same values." JavaScript ships no built-in operator or function for that. Comparing `{ id: 1, name: 'Ada' }` to another object with identical fields and identical values requires either a third-party library (`fast-deep-equal`, `lodash.isEqual`) or a hand-written recursive walk. The course does not pull in a library for this in the early units.

Instead, the course teaches the patterns that avoid the question entirely. <Term definition="Comparing two objects by whether their fields and values match, recursively. JavaScript has no built-in operator for this; the course's patterns sidestep the need.">Structural equality</Term> rarely needs to be written by hand if your data model is shaped well. The three moves you'll see in this course's stack:

- **Compare by primary key.** Two invoice rows are the same invoice when `a.id === b.id`, and only then. Database-shaped data carries its identity in the ID column; equality at the application layer collapses to a primitive comparison on a string.
- **Derive a stable string key from the relevant fields.** When you need to detect changes for caching or deduplication, build a key (`` `${invoiceId}:${status}:${updatedAt}` ``) and compare strings.
- **Use discriminated unions where each variant has its own shape.** The TypeScript chapter coming up later in this unit goes deep on this pattern; comparisons that matter become comparisons on the discriminant field, not on the whole object.

Once you reach the validation and database chapters (Zod in the Forms and Validation unit, Drizzle in the Postgres unit), the comparisons that matter at the wire and the database boundary are ID-based by construction, and the "I need to deep-compare two objects" question rarely surfaces in well-shaped code.
