    A teammate's PR adds a component test for an async Server Component that fetches and renders this org's invoices. They argue it's worth testing because the list is a high-traffic surface. What's the senior response?

    <McqChoice correct>Wrong surface — Server Components aren't an RTL target in 2026; the list's bugs live at the seam (the query, the auth check, the formatter), covered by an integration test, and on a money path, end to end.</McqChoice>
    <McqChoice>Keep it, but only assert the row count — re-testing that the framework renders an array is the one safe thing to check on a Server Component.</McqChoice>
    <McqChoice>Keep it — high-traffic surfaces meet the implicit accessibility trigger, so any component on one earns a component test.</McqChoice>

    <McqWhy>Server Components are off the RTL surface entirely. Their bugs are in the data layer (caught at the seam) and, where they sit on a money path, in E2E. Asserting the row count just re-tests that Next.js can render an array. The accessibility trigger is about role-first queries on interactive client surfaces, not "any component on a busy page."</McqWhy>

    Which of these components meet a trigger to reach for RTL? **Select all that apply.**

    <McqChoice correct>A `<DateRangePicker>` shared across Reports, Invoices, and Filters.</McqChoice>
    <McqChoice correct>A multi-step subscribe form that reveals a seat-count field only on the Pro plan.</McqChoice>
    <McqChoice>A `<Section>` wrapper whose only job is to add vertical spacing.</McqChoice>
    <McqChoice>A button whose sole job is to call a Server Action that already has its own seam test.</McqChoice>

    <McqWhy>The shared picker (one bug, many regressions) and the conditional multi-step form (state transitions the seam test never sees) each hit a named trigger. The spacing wrapper has no state or branching content, and the lone action-button is already covered at the seam — both are deletions, not tests.</McqWhy>

  <MultipleChoice src="89.2">
    Under `globals: false`, you write your first `*.dom.test.tsx` files and they pass individually but start failing in a full run with "found multiple elements." What's missing from the jsdom setup file?

    <McqChoice correct>`afterEach(cleanup)` — without a global `afterEach` for RTL to auto-hook, each rendered tree stays mounted into the next test, so queries match elements from a previous render.</McqChoice>
    <McqChoice>A `ResizeObserver` polyfill — the missing browser API leaves stale nodes mounted across tests.</McqChoice>
    <McqChoice>`vi.resetAllMocks()` in `afterEach` — without it the navigation mock returns the previous test's elements.</McqChoice>

    <McqWhy>RTL only auto-cleans when the runner exposes a global `afterEach`; with `globals: false` it doesn't, so you register `afterEach(cleanup)` by hand. Skip it and trees leak across tests, and a query that should match one element finds two. Polyfills and mock-resets fix different problems, not stale DOM.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="89.2">
    Your `render` helper calls `userEvent.setup()` with no options. A teammate suggests copying `userEvent.setup({ delay: null })` from an older tutorial to speed tests up. Is that the right default?

    <McqChoice correct>No — bare `userEvent.setup()` is the correct default; the docs discourage `{ delay: null }` and warn it causes unexpected behavior, especially with fake timers.</McqChoice>
    <McqChoice>Yes — `{ delay: null }` removes inter-keystroke delays and is the recommended default for any test suite.</McqChoice>
    <McqChoice>Yes, but only in the `render` helper, never inside an individual `it` block.</McqChoice>

    <McqWhy>Bare `userEvent.setup()` is the default this suite uses. `{ delay: null }` is discouraged and interacts badly with fake timers; the one documented escape hatch for a timer-driven test is `{ advanceTimers: vi.advanceTimersByTime }`, reached for only when a timer actually forces it.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="89.3">
    A Confirm button carries class `btn-primary`, id `submit`, text `Confirm`, `aria-label="Confirm purchase"`, role `button`, and `data-testid="submit-btn"`. You're proving a user can click it. Which query belongs at the top of the ladder?

    <McqChoice correct>`screen.getByRole('button', { name: /confirm purchase/i })`</McqChoice>
    <McqChoice>`screen.getByTestId('submit-btn')`</McqChoice>
    <McqChoice>`screen.getByText('Confirm')`</McqChoice>

    <McqWhy>Role plus accessible name is exactly what assistive tech sees, so a green test proves the button is reachable and announced as "Confirm purchase" — a free accessibility assertion. The test id is invisible to users and would stay green even on an unreachable `<div>`; `getByText('Confirm')` ignores the role and could also match a tooltip, throwing on the ambiguity.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="89.3">
    You want to assert that no validation error is shown before the user submits. Which query family is correct, and why?

    <McqChoice correct>`queryBy*` — it returns `null` when nothing matches, which is the only family that lets a "not present" assertion run.</McqChoice>
    <McqChoice>`getBy*` — it's the default for any presence check, and `.not.toBeInTheDocument()` flips it to a negative.</McqChoice>
    <McqChoice>`findBy*` — absence is an async condition, so you must wait to confirm nothing appeared.</McqChoice>

    <McqWhy>`getBy*` throws the instant it finds no match — before your `expect` ever runs — so the test fails with "could not find an element" instead of cleanly asserting absence. `queryBy*` returns `null` for the negative case. `findBy*` is for elements that appear after async work, not for proving something never showed up.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="89.4">
    A subscribe-form component test mocks `createSubscription`, clicks Subscribe, and its only assertion is `expect(createSubscription).toHaveBeenCalledWith({ plan: 'pro', seats: 5 })`. It passes. What's wrong with it?

    <McqChoice correct>It checks the mock was called but never what the form did with the result — it stays green even if the form then renders a blank screen and the user gets no feedback.</McqChoice>
    <McqChoice>Calling a real Server Action from jsdom hits the database, so this assertion can't honestly pass in a component test.</McqChoice>
    <McqChoice>It stops too early — it should read back the subscription row to confirm the write actually landed.</McqChoice>

    <McqWhy>The mocked action's only job is to drive a branch; the form's job is to turn that branch into something the user sees. Assert the visible consequence (a success `status` region, or a declined-card `alert`), which survives a field rename and fails for the right reason. Reading back the row would be mocking too deep — that effect belongs to the seam test.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="89.4">
    For the date-range picker test, why pin time with `vi.setSystemTime(new Date('2026-05-14'))` before asserting that pressing `{ArrowRight}` moves focus to "15 May"?

    <McqChoice correct>The grid is rendered relative to "today" — without a fixed clock, which day "15 May" sits next to changes with the CI run date, making the focus target flaky by construction.</McqChoice>
    <McqChoice>`userEvent.keyboard` needs fake timers installed, and `setSystemTime` is what installs them.</McqChoice>
    <McqChoice>It prevents the real network clock from drifting the MSW handler timestamps out of range.</McqChoice>

    <McqWhy>A picker that reads the real clock is flaky by construction: "this month" differs depending on when the suite runs. Pinning the clock seam makes "today" deterministic so a specific day is a stable focus target. It has nothing to do with installing timers for keyboard events or with MSW timestamps.</McqWhy>
  </MultipleChoice>
