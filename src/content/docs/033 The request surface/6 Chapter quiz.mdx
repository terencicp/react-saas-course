    A teammate ships an auth gate in `proxy.ts`. It reads the session cookie, verifies it against the database, and bounces signed-out users — and they've decided that since the proxy now does the real check, the routes themselves no longer need one. What's wrong with this plan? **Select all that apply.**

    <McqChoice correct>A database read in the proxy runs on every matched request, multiplying a cost you'd otherwise pay once in the route — the classic "why did every page get slow" regression.</McqChoice>
    <McqChoice correct>Excluding a path from the matcher (or a refactor that does so) silently drops proxy coverage on Server Action POSTs to that path, so a proxy-only gate can leave a hole nothing flags.</McqChoice>
    <McqChoice>It's the right call: centralizing the authoritative check in one file is exactly what `proxy.ts` is for, and duplicating it in every route is wasteful.</McqChoice>

    <McqWhy>The proxy is a fast, non-authoritative gate, not a second application layer. A per-request DB read there is paid across the whole app, and matcher coverage is fragile — a Server Action posting to an excluded path never sees the proxy, so a proxy-only gate can open silently. The reflex is defense in depth: cheap cookie-presence check in the proxy, authoritative validation in the route's `requireUser()`, fresh every time.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="33.1">
    A Server Component reads `(await headers()).get('x-forwarded-for')` to get the client IP, and another component reads a custom `x-user-role` header to decide whether to show an admin panel. Which read is the dangerous one, and why?

    <McqChoice correct>The `x-user-role` read — it makes an authorization decision from a header, which any client can forge. Identity and permission come from the verified session, never a raw header.</McqChoice>
    <McqChoice>The `x-forwarded-for` read — IP addresses are personal data, so reading one without consent is the real risk; the role header is fine because the app set it.</McqChoice>
    <McqChoice>Neither is dangerous — headers arriving at a Server Component have already passed through the platform, so they can be trusted for any decision.</McqChoice>

    <McqWhy>Headers are attacker-controlled strings by default — anyone can send `x-user-role: admin`. Authorizing off one is a textbook privilege-escalation bug. `x-forwarded-for` is only trustworthy because a known proxy (Vercel) overwrites it, and even then it's for telemetry, not identity. The anchor: headers are for telemetry and platform signals; the session decides who someone is and what they may do.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="33.3">
    Your SaaS permanently moves `/account` to `/settings`, and separately bounces already-logged-in users away from `/login`. A reviewer says "just use `NextResponse.redirect(url)` for both and you're done." What does that miss?

    <McqChoice correct>The `/account` move should be a 308 (permanent) but `redirect()` defaults to 307; and the `/login` bounce should stay 307 because it depends on the user's session — a 308 there gets cached as "never visit /login," wrong once they log out.</McqChoice>
    <McqChoice>Both belong in `next.config.ts` instead, because any redirect is cheaper at the CDN edge regardless of whether it reads the request.</McqChoice>
    <McqChoice>Nothing — `NextResponse.redirect(url)` already sends a permanent redirect by default, which is correct for both cases.</McqChoice>

    <McqWhy>The status code is the permanence signal, and `redirect()` defaults to 307 (temporary). The `/account` rename is a genuine permanent move, so it earns an explicit 308 that lets search engines reindex. The `/login` bounce is a fact about *this user right now*, not the URL, so it stays 307 — a wrong 308 gets cached and indexed and is painful to undo. Under-commit to 307 unless a move is truly forever.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="33.4">
    Building an invoice list, you wire the status filter into `useState`, refetch in a `useEffect`, and render. It works on your screen. Why does a senior reach for URL state instead?

    <McqChoice correct>The filter is state a user expects to survive a refresh and travel in a shared link — that belongs in the URL, where a Server Component reads it directly with no client state, no effect, and no fetch waterfall.</McqChoice>
    <McqChoice>`useState` can't hold a filter value reliably across re-renders, so the URL is the only place the value stays stable.</McqChoice>
    <McqChoice>URL state is faster only because the browser caches query strings; functionally the two approaches are equivalent.</McqChoice>

    <McqWhy>The decision rule is one question: would the user expect this state back on refresh or in a shared link? Filter, sort, pagination, and active tab all answer yes, so they live in the URL — the server is then a pure function of the URL, reading and rendering with no second source of truth. The `useState`+`useEffect` version rebuilds the exact fetch-in-an-effect waterfall the effects chapter warned against, and it dies on refresh and can't be shared.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="33.5">
    A `'use client'` filter chip needs to show whether it's the active filter and, on click, change the URL to its status while keeping the existing `?sort` and `?cursor` params. Which approach is right?

    <McqChoice correct>Take the active status as a prop the server already resolved (no hook to read it), and on click seed a `new URLSearchParams` from the current query, `set` the one key, and `router.replace(..., { scroll: false })`.</McqChoice>
    <McqChoice>Read the active status with `useSearchParams` for live accuracy, and on click `router.push('?status=paid')` so the back button can step through each filter.</McqChoice>
    <McqChoice>Read the active status with `useSearchParams`, and on click `router.replace('?status=paid')` — the shortest correct write.</McqChoice>

    <McqWhy>The active state is something the server already read and passed down, so a plain prop comparison beats reaching for `useSearchParams` (which would also drag in a Suspense boundary). For the write, `router.replace('?status=paid')` overwrites the *entire* query and silently drops `sort` and `cursor`; you merge by seeding `URLSearchParams` from the current query and setting one key. `replace` (not `push`) keeps filter churn out of history, and `scroll: false` stops the viewport from jumping.</McqWhy>
  </MultipleChoice>
