It's two months after you shipped the invoicing feature. A teammate — or future-you, which is the same person with less memory — is wiring up a new screen and types `createInvoiceDraft(`. They hover the call to remind themselves what to pass and what comes back. The tooltip shows the signature and nothing else. Does it write to the database? Does it send an email? What happens when the input is malformed — does it throw, or return an error they need to handle? The signature can't say. So they open the file, read the body, reconstruct the contract in their head, and lose ten minutes. That tax gets charged to every caller, every time, forever.

A doc comment on `createInvoiceDraft` would have answered all of it in the hover, before they ever left the call site. So the obvious move is to write a doc comment on everything. That's the wrong move too — a block on every private helper buries the few that matter under noise, and the file becomes half comment. The skill this lesson teaches is the cut between those two failures: *which declarations earn a doc block, and which don't?* The tag syntax is small and you'll have it in five minutes. The judgment is the whole lesson.

This is the same thread you pulled in the previous chapter, [Docs that live next to the truth](/101-docs-that-live-next-to-the-truth/1-the-four-jobs-of-docs/): docs live next to the code they describe, and you *link* to a source of truth instead of copying it. There you named the four artifacts a repo maintains from the outside — README, AGENTS.md, ADRs, source-as-reference. Now you're zooming all the way in, to the documentation that lives *inside* the source file, on the function or type or schema itself.

Here's the payload. A three-part test for deciding yes-or-no on any declaration. A working set of five tags, and the judgment to keep it small. A writing posture built around one fact: the first sentence is what the reader actually sees. And one reflex carried over from last chapter — link, don't duplicate. The deliverable is never a generated docs website. It's a good IDE hover.

## What TSDoc is, in one pass

<Term definition="Microsoft's doc-comment standard for TypeScript. A /** ... */ block above a declaration; IDEs render it on hover at the call site.">TSDoc</Term> is the Microsoft-stewarded standard for doc comments in TypeScript: a `/** ... */` block placed directly above a declaration, with a small set of `@`-prefixed tags inside it. That's the whole format. Here's the smallest one — a single summary sentence, no tags yet:

```ts
/** Creates a draft invoice for the active organization and returns its id. */
export async function createInvoiceDraft(input: CreateInvoiceInput): Promise<Result<{ id: string }>> {
```

The reason to bother is who reads it and where. Every IDE — VS Code, WebStorm, Cursor — renders that block on **hover at the call site**. A developer using the function never has to open the file where it's defined; the contract comes to them. For a closed-source SaaS app, that hover *is* the reference documentation, and it's the only surface that matters. (There's a second consumer — tools like TypeDoc and <Term definition="A Microsoft tool that turns TSDoc comments into a published HTML reference site and an API report.">API Extractor</Term> generate full HTML reference *sites* from the same syntax — but that's for *published* libraries, where the docs are the product. We'll come back to that threshold at the end. It isn't where your effort goes on a SaaS app.)

Plant one contrast now, because the rest of the lesson leans on it. **The doc comment states the contract; the body states the implementation.** The contract is what a caller needs to know from the call site — what it does, what it assumes, what it gives back, how it fails. The implementation is *how* it pulls that off, and a caller doesn't care. In the vocabulary from last chapter, a doc comment is *reference* documentation, in code form.

## The public-surface cut: which declarations earn a block

This is the load-bearing part of the lesson. Get the cut right and the rest is mechanical.

### The inclusion test

A declaration earns a TSDoc block if **any one** of these is true:

1. **It crosses a module boundary.** It's exported from `src/lib/<feature>/` and used over in `src/app/`, it's a Server Action a component calls, it's a util imported elsewhere. The moment a declaration is read one call site away from where it's defined, its contract needs to be stated where that reader hovers.
2. **The type alone doesn't carry the contract.** There's a precondition, a side effect, an ordering requirement, or an error behavior that the signature physically cannot express. `(orgId: string) => Promise<void>` can't tell you it sends an email.
3. **The next reader is at the call site, not in the file.** A teammate or future-you who will hover the name, not open the body. If the only person who ever reads this declaration is the person editing the file it lives in, the block has no audience.

The inverse is the default, and the default is **no**. A private helper used once in the same file: the name and type are the contract. A one-liner whose body is shorter than the doc would be: the body *is* the doc. A component whose props type already tells the story: the type is the doc. None of these earn a block.

Hold onto the posture underneath the test: **the cut is the public surface.** Doc volume should track doc value. A block that doesn't add what the call site can't see isn't neutral — it's noise, and noise has a cost, because it dilutes the blocks that do matter.

### The SaaS surfaces that always earn a block

Let's make the test concrete on the exact surfaces this course has already built. These five are where the public-surface cut almost always lands on *yes*.

The richest case is a **Server Action** — the course's typed API entry point since [the Server Actions chapter](/043-server-actions/1-the-use-server-seam/). A caller hovering an action needs four things the signature can't give them: a plain summary of what it does, the precondition it assumes, the side effects it causes, and the failure modes they have to handle. Let's walk one part by part.

  async (input, ctx) => {
    // …
  },

`}>

  <AnnotatedStep meta="{4-6}" color="orange">

  </AnnotatedStep>

  </AnnotatedStep>

  <AnnotatedStep meta="{9-10}" color="green">

That one walkthrough is the anatomy. The other four surfaces follow the same pattern, so a quick look at each is enough — once you've seen the shape, you recognize it.

**Exported functions in `src/lib/<feature>/`** are consumed across the app, so their contract belongs where the reader hovers. Internal helpers *in the same module* don't — the boundary is the export, not the folder.

```ts
/**
 * Builds the R2 object key for an upload, namespaced to the org so one
 * tenant can never address another tenant's files.
 */
export const buildObjectKey = (orgId: string, filename: string): string =>
  `orgs/${orgId}/uploads/${filename}`;
```

**Drizzle schema tables** earn a block on the exported `pgTable` identifier, so that hovering the table name at a query site surfaces what the table is for. This pairs with the one-paragraph table header you met in [the source-as-doc lesson](/101-docs-that-live-next-to-the-truth/2-the-thin-readme/).

```ts
/** One row per issued invoice; tenant-scoped on `organizationId`. */
export const invoices = pgTable('invoices', {
  id: uuid().primaryKey().$defaultFn(() => uuidv7()),
  // …
});
```

**Zod schemas exposed across modules** get a one-line summary on the exported schema describing what the *shape* represents — `UserProfileInput — the shape the profile-update action accepts`. The field-level intent lives in the schema's own `.meta({ description })`, so you don't restate fields here either.

**Webhook handlers** get a block on the route handler stating the contract: which event types it accepts, the idempotency key it dedupes on, the side effects, and the response it returns. As you saw last chapter, the handler *is* the webhook contract doc.

```ts
/**
 * Stripe webhook receiver. Verifies the signature, then dedupes on the
 * event id via `processed_events` before projecting `checkout.session.completed`
 * and `customer.subscription.*` into the org's plan entitlements.
 */
export async function POST(request: Request) {
  // …
}
```

### The negative space: what does NOT earn a block

The cut only sticks once the *no* cases are as vivid as the *yes* cases. This is where over-documenting gets corrected, so read the reasons, not just the list.

- **Internal helpers** — one file, one caller. The name and type are the contract; there's no second reader to serve.
- **React components** — the props type and the component name document them. A block on every component is pure noise. The one exception: a shared design-system primitive whose prop semantics are genuinely non-obvious.
- **Trivial passthroughs and getters** — `const isAdmin = (user) => user.role === 'admin'`. The body is the doc; a comment above it would say less than the line itself.
- **Test helpers** — read by the test author and the test's reviewer, both of whom are right there in the file. Clear names are enough.
- **Type aliases that mirror a schema** — `type UserInput = z.infer<typeof userInputSchema>`. The schema is the doc; the type is just the derived view of it. A block here would be a copy of a copy.

## The first sentence is the whole doc

Look again at how a hover renders, because it dictates how you write. The IDE shows the signature, then your first sentence, prominently — and everything else sits below the fold. Most readers never scroll. So the reflex is to write the first sentence such that a reader can decide *in two seconds* whether they're at the right function: verb-first, declarative, no preamble.

The contrast is the whole teach:

- Good: `Creates an invoice draft and returns its id.`
- Bad: `This function is used to create an invoice draft.`

The bad one opens with four words of throat-clearing — "This function is used to" — that every doc comment could start with, which means they carry no information. By the time the reader's eye reaches the verb, they've scanned past noise. The good version leads with the verb and is done.

Here's what the reader actually sees in each case. The following figure mocks up the hover card the IDE renders; flip between the two tabs to feel the difference at a glance.

Everything below that first line follows one rule: the block adds only what the call site can't already see. Preconditions, side effects, the failure modes, the one example worth showing. Anything the signature already states, the block leaves out. Which brings us to the most common way that rule gets broken.

## TSDoc states the contract, not the type

If you've written doc comments in another language — Java, or plain JavaScript with JSDoc — you carry a habit that's wrong here, and it's worth naming directly because nearly everyone does it on their first TypeScript block.

The course's TypeScript style is inference-led: you annotate the parameter and return types right in the signature. That means the type is *already stated*, once, in the one place that's guaranteed to stay correct — the compiler checks it. So the doc comment must not repeat it. Compare the two tabs below — the same function, written the JSDoc way and the TSDoc way:

<CodeVariants syncKey="jsdoc-vs-tsdoc">
  <CodeVariant label="JSDoc habit">
    <div data-mark-color="red">

    ```ts "{string}" "@returns the customer object"
    /**
     * Creates a customer.
     *
     * @param {string} email - the email
     * @returns the customer object
     */
    export async function createCustomer(email: string): Promise<Customer> {
    ```

    </div>
    **The `{string}` duplicates what the signature already says, and `@returns the customer object` restates the return type.** Both drift the day the types change, and neither tells the caller anything new.
  </CodeVariant>

  <CodeVariant label="TSDoc">
    <div data-mark-color="green">

    ```ts "@param email - the billing email the customer is created against"
    /**
     * Creates a Stripe customer for the org's billing email.
     *
     * @param email - the billing email the customer is created against
     */
    export async function createCustomer(email: string): Promise<Customer> {
    ```

    </div>
    **No type in the tag — the signature owns the type.** The `@param` adds what the type can't: *why* this address matters. `@returns` is dropped because `Promise<Customer>` already says it.
  </CodeVariant>
</CodeVariants>

This is the line that separates TSDoc from JSDoc, so it's worth saying as a rule: **TSDoc tags never carry types.** No `@param {string}`. JSDoc needed types because plain JavaScript had none — the comment was the only place the type could live. TypeScript moved the type to the signature, so the tag's job shrank to the part the signature can't express: the parameter's *purpose*.

The same logic governs `@returns`. Only document the return when its *semantics* go past its type — a sentinel `null` that means "not found," a partial result, an ordering guarantee. `@returns the user object` on a function that already returns `User` is pure noise; drop it. Here's the rule in a tooltip, on a block that does it right:

<CodeTooltips tooltips={{
  '@returns': 'Documented here only because null is a sentinel — it means not found OR not in this tenant, which the type Invoice | null cannot express on its own.',
}}>
```ts
/**
 * Looks up an invoice by id within the active org.
 *
 * @returns the invoice, or `null` when it doesn't exist or belongs to another org
 */
export async function getInvoice(id: string): Promise<Invoice | null> {
```
</CodeTooltips>

## Link, don't duplicate — the reflex applied inline

Last chapter's reflex was: before you paraphrase a source of truth, ask whether you could link to it instead. That reflex moves straight into TSDoc, and it's the difference between a doc that stays correct for free and one that's *guaranteed* to go wrong.

When a function's contract is already stated somewhere structural, the block points at that place instead of restating it. A function that validates a Zod-checked input doesn't list the input's fields — it links to the schema, and the schema (with its `.meta({ description })` fields) is the field-by-field reference. A function that reads config doesn't describe the variable — it names `STRIPE_WEBHOOK_SECRET` and lets `env.ts` be the env doc. The two tabs below show the same action documented both ways:

     * @param input.notes - optional free-text notes shown on the invoice

      // …
    });
    ```

    </div>
    **This list drifts the day someone adds, renames, or removes a field.** Now there are two definitions of the shape — the schema and this prose copy — and one of them is silently wrong. The reader can't tell which.
  </CodeVariant>

  <CodeVariant label="Links to the schema">
    <div data-mark-color="green">

     * Creates a draft invoice for the active organization.
     *
     * @param input - the draft fields, validated against {@link createInvoiceSchema}
     */
    export const createInvoiceDraft = authedAction('member', createInvoiceSchema, async (input, ctx) => {
      // …
    });
    ```

    </div>
    **One source of truth.** The schema can't drift from itself, and the hover still gives the caller a one-click path to the field list — each field's intent lives in the schema's own `.meta({ description })`.
  </CodeVariant>
</CodeVariants>

State the reason as economics, because that's what makes it a reflex instead of a style preference. A duplicated field list isn't *likely* to drift — it's *certain* to, the next time the schema changes and the person changing it doesn't think to update the prose copy. And a drifted doc is worse than no doc: it actively lies to the next reader, who now has to discover it's wrong before they can ignore it. (You'll see that asymmetry made structural in a couple of lessons, when we get to keeping docs in sync at review time.) The link is the only form that can't drift, because it doesn't store a second copy of anything.

:::note
This is also *why* a `type UserInput = z.infer<typeof userInputSchema>` alias earns no block of its own: any block you wrote would just restate the schema it's derived from. Link-don't-duplicate and the public-surface cut are the same instinct seen from two angles.
:::

## `@example`, `@deprecated`, and the tags that rarely earn their weight

You've now met four of the five tags worth your time — `@param`, `@returns`, `@throws`, and the inline `{@link}`. Two more pull their weight in specific situations, and a couple exist that you'll almost never reach for. Let's close out the toolkit and, just as importantly, keep it small.

### `@example` — when one call shape earns its weight

Add an `@example` only when the call shape is non-obvious: an action with several optional parameters, or an unusual usage a reader would otherwise have to guess at. A function whose call is obvious from its signature doesn't get one — the example would just restate the signature.

When you do write one, it has to be runnable as-is. An example with a hand-wavy `// ...` in the middle isn't an example; it's a sketch, and the reader can't trust it. Here's a function whose optional second argument earns its one example:

```ts
/**
 * Archives an invoice. Pass `{ notify: true }` to also email the customer.
 *
 * @example
 * await archiveInvoice(invoice.id, { notify: true });
 */
export async function archiveInvoice(
  invoiceId: string,
  options?: { notify?: boolean },
): Promise<Result<void>> {
```

### `@deprecated` — never deprecate without a path

When you retire a declaration, keep its summary intact and add a `@deprecated` line that names the replacement: `@deprecated use createInvoiceDraft instead — removed in the next major.` The replacement path is the whole point, and the reason is what the IDE does with it.

</Figure>

The caller sees the strikethrough while typing and reads the replacement in the hover. They're redirected at the call site, before they've written a line against the dead function. A bare `@deprecated` with no path tells them to stop without telling them where to go — which just sends them into the file you were trying to keep them out of.

### The two you'll rarely type: `@remarks` and `@see`

`@remarks` exists for when a block genuinely needs a second section beyond its one-paragraph summary. Most don't. If you find yourself reaching for it to brain-dump *why* the code is shaped this way, stop — that's architectural rationale, and rationale belongs in an [ADR](/101-docs-that-live-next-to-the-truth/4-adrs---one-decision-per-file/), not in a hover the caller reads to decide how to call the function.

`@see` adds a "see also" pointer, but an inline `{@link}` in the summary usually does the job in fewer characters and in the spot the reader is already looking. Reach for `@see` only when a standalone pointer reads better than an inline one.

Here's the whole working set in one place — a reference to come back to, not something to memorize:

| Tag | Earns its place when… | Skip it when… |
| --- | --- | --- |
| `@param` | the parameter's *purpose* isn't obvious from its name and type | the name already says it (and never to state the type) |
| `@returns` | the return carries semantics beyond its type — a sentinel, a partial result, an ordering | the type already says it (`@returns the user object`) |
| `@throws` | the caller must handle a specific failure mode | the function doesn't throw anything the caller acts on |
| `@example` | the call shape is non-obvious; one runnable call clarifies it | the signature already makes the call obvious |
| `@deprecated` | a declaration is retiring — always with the replacement path | — (it's the path that makes it useful) |
| `@remarks` *(rare)* | the summary genuinely needs a second section | the one paragraph covers it — which is almost always |
| `@see` *(rare)* | a standalone pointer reads better than an inline `{@link}` | an inline link in the summary already does the job |

## The hover is the usability test (and the audit trail)

Everything in this lesson collapses into one reusable check. After you write a block, **re-read your own hover and ask: can a caller decide in five seconds whether to use this, and how?** If the hover dumps three paragraphs of internal rationale, the block has drifted — into explanation that belongs in an ADR, or into implementation narration, which belongs nowhere. Cut to what the caller needs. That's the test, and it's the whole posture in one sentence.

There's a second reason the cut earns its keep in 2026, beyond saving the next caller a trip into the file. The same block is the **brief** that anyone editing the function reads to learn its contract before they touch the body — a human reviewer two months from now, or an automated agent making a change across the codebase. If the contract is stated in the block, the editor works from it. If it isn't, they read the implementation and *infer* the contract, and inference is where wrong changes come from. The hover is the contract you hand the next editor, whoever or whatever they are. That's why a stated public surface is a property of the codebase, not a personal nicety.

One last threshold, stated once and then dropped: TypeDoc and API Extractor turn these same comments into a published HTML reference site. For a *library* — an SDK, a component package — that site is load-bearing, because the docs are the product surface the world consumes. For a closed-source SaaS app, the IDE hover *is* the reference surface, and the course doesn't reach for TypeDoc by default. If you ever publish a package, the calculation flips and you set it up then. Until then, your deliverable stays the hover.
