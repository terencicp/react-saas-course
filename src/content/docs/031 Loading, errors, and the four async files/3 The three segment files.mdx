Picture a route that shows one invoice. The folder is `app/invoices/[id]/`, and the page reads a single row by its ID and renders the detail:

```tsx title="app/invoices/[id]/page.tsx"
export default async function InvoicePage({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const invoice = await getInvoice(id);
  return <InvoiceDetail invoice={invoice} />;
}
```

That is the happy path, and it is the only path this file handles. But a real request to a real route does not have one outcome — it has four. Walk them, because each one is a different thing on the user's screen:

- The query is **in flight**. For the 800ms it takes to come back, what fills the page?
- The query **succeeds**. The invoice renders. This is the path above — handled.
- The ID matches **no invoice**. Maybe it was deleted, maybe the link was wrong. `getInvoice` returns `null`. What then?
- The query **throws**. The database is down, the connection timed out. What does the user see instead of a crash?

You already have the raw materials to handle all four. You met `<Suspense>` as the loading contract two lessons ago, you watched it stream a page in the last one, and you saw `notFound()` named back when you learned routing. You could, in principle, hand-wire a `<Suspense>` boundary and an Error Boundary around this page yourself. The App Router offers something better: three sibling files you drop next to `page.tsx`, plus one function you call. Each one is sugar over a primitive you can already name — and the goal of this lesson is to keep that mapping in view so you never treat these files as magic.

By the end, `app/invoices/[id]/` will ship all four states with `page.tsx` and three small files beside it. You are assembling parts you already know, not learning from scratch — which is exactly why this should feel lighter than it looks.

## The file beside the page

Before any single file, learn the one idea that makes all three click, because it is identical across every one of them.

A specially-named file in a route folder becomes a <Term definition="A wrapper React places around a subtree to handle one cross-cutting concern — loading, errors, or a missing resource — for everything inside it.">**boundary**</Term> that wraps that segment's `page.tsx` and everything nested below it. "That segment" means the folder the file sits in: `app/` maps the URL path onto folders, so each folder is one <Term definition="A single folder in app/ — it owns one piece of the URL path and the UI rendered for it.">route segment</Term>. Drop `loading.tsx` in `app/invoices/`, and its boundary covers `/invoices` *and* every route underneath — `/invoices/123`, `/invoices/123/edit`, all of it.

That coverage runs deep until a child segment provides its own. A `loading.tsx` inside `app/invoices/[id]/` **overrides** the one in `app/invoices/` for the `[id]` subtree — the nearer file wins, the way a more specific CSS rule beats a general one. So a boundary you place high is inherited by everything below it, and a boundary placed lower shadows the ancestor for its own branch.

This file tree shows both at once — a shared boundary up top, an override deeper down:

<FileTree>
- app
  - invoices
    - loading.tsx wraps everything under invoices/
    - error.tsx
    - **[id]**
      - page.tsx
      - loading.tsx overrides for the [id] subtree
      - not-found.tsx
</FileTree>

Hold this model separate from the three files themselves. *Where the file sits and what it covers* is one idea — the one you just learned, shared by all three. *What each file does* is three more, coming next. Keeping those apart is what stops this from feeling like four arbitrary conventions to memorise.

One reach worth stating now, because the inheritance model invites the opposite mistake: place **one boundary file per coherent visual surface, not one per route folder.** Many routes legitimately share the same loading shell, the same error panel. Dropping a `loading.tsx` into every folder by reflex gives you a pile of near-identical skeletons to maintain and a jarring cascade of fallbacks as the user navigates deeper. Put the file where the UX genuinely changes; let inheritance cover the rest.

## `loading.tsx`: the segment's Suspense fallback

Start with the in-flight state — the 800ms gap. The file that fills it is `loading.tsx`, and here is the whole truth about it: `loading.tsx` is the framework writing `<Suspense fallback={<Loading />}>` around your segment for you. That is it. You write the skeleton; Next.js wires the boundary.

The two tabs below make the sugar visible. The first is the file you actually write; the second is the boundary Next.js produces from it.

<CodeVariants maxLines={6}>
  <CodeVariant label="What you write">
    ```tsx title="app/invoices/[id]/loading.tsx"
    export default function Loading() {
      return <InvoiceSkeleton />;
    }
    ```
    **You write one default-exported component that returns a skeleton — no props, no boundary.** The skeleton mirrors the resolved layout's footprint, so nothing shifts when the content swaps in.
  </CodeVariant>

  <CodeVariant label="What Next.js wires">
    <div data-mark-color="green">

    ```tsx {1,3}
    <Suspense fallback={<Loading />}>
      {/* layout → page.tsx for this segment */}
    </Suspense>
    ```

    </div>
    **The framework wraps your segment in the exact `<Suspense>` boundary from the first lesson of this chapter — your `Loading` becomes its `fallback`.** This is conceptual; you never write this wrapper, but it is precisely the Suspense pattern you already know.
  </CodeVariant>
</CodeVariants>

Seeing the file *as* the Suspense pattern is the point — once you do, everything else about `loading.tsx` follows from what you already know about Suspense. A few specifics, tied to the running route.

**Default export, no props.** `loading.tsx` is default-exported — a deliberate exception to the course's named-exports rule, because the framework finds these files by their default export. And it receives **no props**: the framework decides when to show it, so there is nothing to pass in. A `loading.tsx` that tries to read `params` or `searchParams` is reaching for context it does not have.

**It is a Server Component.** No `'use client'`. The default in the App Router is the server, and the opt-in to the client needs a reason — a loading skeleton has none. It renders static markup once and never touches state or browser APIs. If a piece of the skeleton needs animation — a shimmer, a pulse — that animated piece is its own Client Component that the loading file *renders*; the file itself stays on the server. Keep the directive at the leaf that genuinely needs it, never on the whole skeleton.

**Skeleton over spinner.** A spinner says "something is happening." A skeleton says "*this* is what is about to appear" — the same column widths, the same number of rows, the same heights as the resolved invoice. When the real data swaps in, nothing on the page jumps, because the skeleton already reserved the exact space. That is the difference between a polished surface and one that flinches every time it loads.

:::tip
Match the skeleton to the resolved layout's footprint — same heights, same item count, same rough shape. The fallback should be a grey tracing of the final UI, so the swap from skeleton to content is invisible except for the content appearing. A skeleton that is the wrong size produces a layout shift, which is the exact jank you reached for Suspense to avoid.
:::

**Scope works exactly as you just learned.** `app/invoices/loading.tsx` covers `/invoices` and every nested route without its own loading file; a `loading.tsx` inside `app/invoices/[id]/` overrides it for that subtree. Nothing new here — it is the inheritance model from the previous section, applied to this specific file.

There is one sharp edge worth knowing before it bites you in a build log.

:::caution
This course runs with Cache Components on. Under that mode, `loading.tsx` covers the work *inside* your page — but it does **not** cover uncached, request-time data read in a **layout** above it, such as `cookies()`, `headers()`, or an uncached `fetch`. That layout data has to sit inside its own `<Suspense>` boundary, or the build fails with an error pointing you at it. The fix is the same discipline from the streaming lesson: keep slow, request-time reads out of layouts and down in the page where `loading.tsx` already covers them — or wrap them in their own boundary. The compiler is now enforcing the rule you already learned by hand.
:::

## `not-found.tsx` and the `notFound()` trigger

Next state: the ID matches no invoice. `getInvoice(id)` comes back `null`. This is the pairing people most often get wrong, so treat the *trigger* and the *file* as two halves of one mechanism — neither works without the other.

Start with the trigger, in the page. Here is the senior shape for fetch-then-guard, walked line by line:

<AnnotatedCode lang="tsx" code={`
import { notFound } from 'next/navigation';

export default async function InvoicePage({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const invoice = await getInvoice(id);
  if (!invoice) notFound();
  return <InvoiceDetail invoice={invoice} />;
}
`}>
  <AnnotatedStep meta={`{4} "await params"`}>
    **Await `params`.** Route params arrive as a Promise in Next.js 16, so you `await` them before reading `id`. You met these async request APIs in the previous chapter; here they feed the lookup.
  </AnnotatedStep>

  <AnnotatedStep meta="{5}">
    **Fetch the row.** `getInvoice(id)` returns the invoice *or* `null` — the read shape for a single record that may legitimately not exist. `invoice` is `Invoice | null` at this point.
  </AnnotatedStep>

  <AnnotatedStep meta={`{6} "notFound()"`} color="orange">
    **Guard on the miss.** When the row is `null`, call `notFound()`. It does not return a value to branch on — it <Term definition="Stops execution at the call, like an early throw — nothing after it runs.">short-circuits</Term> by *throwing* a special signal the framework catches. Execution stops on this line; nothing below runs.
  </AnnotatedStep>

  <AnnotatedStep meta="{7}" color="green">
    **Render the success case.** Because `notFound()` never returns, TypeScript knows that if execution reached this line, `invoice` was not `null` — it has narrowed `Invoice | null` down to `Invoice`. The narrowing is real, not a hope: `notFound()` is typed to return `never`, so TS treats everything after it as unreachable on the `null` branch. No `!`, no cast.
  </AnnotatedStep>
</AnnotatedCode>

That `never` return is what makes the guard pull double duty: it ships the 404 *and* it tells the type system the value below is safe. You get the runtime behaviour and the type narrowing from one call.

Now the file. `app/invoices/[id]/not-found.tsx` is what renders when `notFound()` fires inside that segment. And the reach here is **co-location**: because this file sits next to the dynamic invoice route, its UI can be specific — "This invoice doesn't exist" with a link back to the list — instead of a generic site-wide "Page not found." The route that knows it is about invoices ships the 404 that knows it too.

```tsx title="app/invoices/[id]/not-found.tsx"
import Link from 'next/link';

export default function NotFound() {
  return (
    <div>
      <h2>Invoice not found</h2>
      <p>This invoice doesn’t exist, or you don’t have access to it.</p>
      <Link href="/invoices">Back to all invoices</Link>
    </div>
  );
}
```

Like `loading.tsx`, this is a default-exported Server Component that takes **no props**. It *can* be `async` if the 404 UI needs to fetch — read a header, look up a suggestion — but it cannot read `params` or `searchParams` directly. If the message needs the missing ID, fetch that context in `page.tsx` before you call `notFound()`; if it needs a client hook like `usePathname`, do that work in a Client Component the file renders.

There are two distinct ways `not-found.tsx` gets shown, and the distinction matters:

- **You call `notFound()` after a lookup.** This is the case above. The framework renders the nearest `not-found.tsx` walking up from the segment that threw — your co-located invoice 404.
- **The URL matches no route at all.** Someone visits `/invioces` (a typo). No segment matches, so there is no `notFound()` call to make — the framework automatically renders the **root** `app/not-found.tsx`. That root file is the catch-all for every unmatched URL in the app, which is why a generic site-wide 404 belongs there and a resource-specific one belongs next to its route.

Two more things keep people out of trouble.

First, the status code, which is more subtle than "it returns a 404" — and the subtlety is a direct consequence of the streaming you learned last lesson.

:::note
A `not-found.tsx` does **not** always emit an HTTP 404. When the response **streams** (the body has already started flushing), the status line was locked at the first byte — so the page comes back **200** instead. Next.js still injects `<meta name="robots" content="noindex">` into the streamed not-found page so search engines drop it from their index anyway. Only a **non-streamed** response carries an actual **404** status. The practical takeaway: the `noindex` meta, not the raw status code, is what protects your SEO here. If you genuinely need a 404 *status* — for compliance, analytics, or a strict crawler — the existence check has to finish *before* the body starts streaming, i.e. call `notFound()` before any suspending `await` or boundary. The status header is set the instant the first chunk goes out; you cannot change it after.
:::

Second — and this trips up nearly everyone the first time — `not-found.tsx` is **not** tied to a 404 from a `fetch`. If you call some API and it answers `404 Not Found`, that HTTP status does nothing on its own; no boundary fires. *You* read that response, turn it into `null` (or a throw), and call `notFound()` yourself. The file is wired to the `notFound()` function, full stop — never to an HTTP status code that happened somewhere inside your data layer.

One name to file away and move past: there is also an experimental `global-not-found.js`, behind a config flag, for apps with multiple root layouts or a dynamic root segment. It is out of scope here — this lesson owns the segment-level trio.

## `error.tsx`: the segment's Error Boundary

Last state, and the one with the most teeth: the query throws. The database is unreachable, a timeout fires, something downstream blows up. The file that catches it is `error.tsx`.

Open with the primitive, because it is the one you were promised back in the first lesson of this chapter. Suspense, you learned there, does **not** catch errors — that is the Error Boundary's job. `error.tsx` is the framework writing a React Error Boundary around your segment. Any uncaught throw — in `page.tsx`, in a nested layout below it, in any child — renders the error UI instead of crashing the whole tree. This file is the one that fills the gap Suspense deliberately left open.

Here is the canonical shape, walked line by line:

<AnnotatedCode lang="tsx" code={`
'use client';

export default function Error({
  error,
  unstable_retry,
}: {
  error: Error & { digest?: string };
  unstable_retry: () => void;
}) {
  return (
    <div>
      <h2>Couldn’t load this invoice</h2>
      {error.digest != null && <p>Reference: {error.digest}</p>}
      <button onClick={() => unstable_retry()}>Try again</button>
    </div>
  );
}
`}>
  <AnnotatedStep meta={`{1} "'use client'"`} color="violet">
    `'use client'` is **mandatory**. Error Boundaries are class-component machinery under the hood (`getDerivedStateFromError`, `componentDidCatch`) and they hold state — both client-only. So unlike the other two files, this one *cannot* be a Server Component. Omit the directive and the build fails with a clear error. This is the *reason* `error.tsx` is the exception, not an arbitrary rule.
  </AnnotatedStep>

  <AnnotatedStep meta={`{7} "Error & \{ digest?: string \}"`}>
    **The `error` prop.** You receive the thrown error, plus an optional `digest` string. The shape is `Error & { digest?: string }` — verbatim, no `any`.
  </AnnotatedStep>

  <AnnotatedStep meta={`{13} "error.digest"`} color="blue">
    **Show the `digest`.** For errors thrown in a **Server Component**, the real message and stack never cross the wire — you get a *generic* message and a `digest`, a hash that correlates this failure with your server logs. That is the security property from the previous chapter holding the line: the full error stays on the server so secrets never leak to the browser. Surfacing the `digest` in the UI gives a user something to paste into a support ticket and you something to grep for.
  </AnnotatedStep>

  <AnnotatedStep meta={`{14} "unstable_retry"`} color="blue">
    **The retry.** `unstable_retry()` re-runs the segment — it re-fetches *and* re-renders. Wire it to your "Try again" button, because most segment errors are failed data reads and you want a genuine fresh attempt, not a re-render of the same broken state.
  </AnnotatedStep>
</AnnotatedCode>

Two of those steps deserve more than a step's worth of space.

**The `digest`, and why the message is generic.** When a Server Component throws, the error is serialized to be sent to the browser — but Next.js strips it first. The user, and your `error.tsx`, see a flat "An error occurred in the Server Components render" plus a `digest` hash; the actual message and stack trace stay server-side, written to your logs. This is deliberate: a raw error message can carry a connection string, a file path, an internal hostname — exactly the things you do not hand to whoever is poking at your app. The `digest` is the bridge across that gap. Print it in the UI, and a support ticket that says "Reference: 1a2b3c" lets you find the real stack in your logs in seconds. (Errors thrown in *Client* Components keep their original message — there is no wire to cross, so there is nothing to strip.)

**`unstable_retry()` versus `reset()`.** There are two ways to retry, and the difference decides whether retry does anything useful.

<CodeVariants maxLines={3}>
  <CodeVariant label="unstable_retry() — re-fetch and re-render">
    <div data-mark-color="green">

    ```tsx "unstable_retry"
    <button onClick={() => unstable_retry()}>Try again</button>
    ```

    </div>
    **This is the one you want:** it re-runs the segment from scratch — fresh data, fresh render — by doing a `router.refresh()` and a `reset()` together inside a transition. Because most segment errors are failed reads, the only retry that can succeed is one that fetches again.
  </CodeVariant>

  <CodeVariant label="reset() — re-render only">
    <div data-mark-color="orange">

    ```tsx "reset"
    <button onClick={() => reset()}>Try again</button>
    ```

    </div>
    **The older, narrower option** — a separate prop you destructure instead: it clears the error state and re-renders the segment, but it does **not** re-fetch. If the cause was a failed Server Component read, re-rendering with the same data just reproduces the same error — so for data failures, `reset()` is useless.
  </CodeVariant>
</CodeVariants>

State the distinction crisply, because it is the single most common mistake with this file: `reset()` means "try rendering again with the same data"; `unstable_retry()` means "go get fresh data, then try again." For Server Component data errors — which is most of what you will hit — only the second one can recover. Default to `unstable_retry()`.

One honest caveat: the `unstable_` prefix is real. The API is still settling and the name may change in a future release. But it *is* the documented default in the current version, and it does the thing you actually want, so reach for it — just do not be surprised when a future upgrade renames it.

### What `error.tsx` catches — and what it doesn't

An Error Boundary catches throws that <Term definition="Errors propagate upward through the tree until a boundary catches them, the way an exception travels up the call stack.">**bubble up**</Term> to it from below. So scope matters, and there is one piece of it that surprises everyone.

`error.tsx` **catches** a throw in `page.tsx`, in any layout *nested below* it, and in any child component. It does **not** catch a throw in the layout it sits *beside*. The reason is structural: that sibling layout is the boundary's *parent* — the boundary lives inside the layout's subtree, not around it. So if the layout itself throws while rendering, it throws *before* the boundary exists to catch it. The error sails right past.

This is not a footnote — it is a real limitation with a real consequence. Follow it up one level: a throw in the **root** layout escapes `app/error.tsx` for exactly the same reason, and crashes straight to the browser's default error screen. There is no `error.tsx` above the root to catch it. The fix is a different file, `global-error.tsx`, and it is the entire subject of the next lesson. This is a deliberate handoff, not a gap I am leaving open.

### Verify the UX in a production build, never just dev

Here is a trap that has cost people an afternoon of "but it works on my machine."

In **development**, when your segment throws, the Next.js error overlay slams up first — full message, full stack, the whole diagnostic. Your `error.tsx` renders *behind* it. So in dev you are looking at the developer experience, not the user experience.

In **production**, the overlay is gone. Only your `error.tsx` renders — with the *generic* message and the `digest` for a Server Component throw, exactly as the user will see it. The rule that follows is simple and load-bearing: **never sign off on `error.tsx` from dev mode alone.** Build the app and run it locally before you ship, so you are looking at what the user actually gets — the real copy, the real `digest`, the real layout — and not the dev overlay sitting on top of it.

### What belongs in `error.tsx`

Keep the file small and single-purpose. It owns the failure UI: a message, the `digest`, a "Try again" button, and — when you wire up monitoring — a `useEffect` that reports the error to your error-tracking service (the full integration comes much later in the course; for now, just know the report belongs here). What does **not** belong: business logic, and any data fetching that could itself throw. An `error.tsx` that throws while rendering the error is a genuinely bad day.

:::note
`error.tsx` is for the *unexpected* — a thrown exception, a crash. It is not how you handle *expected* failures like a form that fails validation. Those return a structured result from a Server Action and render inline, a separate channel you will meet when this course reaches forms. Throw for the impossible; return for the foreseeable.
:::

## How the three files wrap one segment

You have met all three files, each motivated by a state on the screen. Now see how they compose — because they are not three independent gadgets bolted on side by side. The framework assembles them into three **nested** boundaries around your segment, in a fixed order, and the order has consequences.

Scrub through the build below. Each step adds one boundary and one state, in the order you learned them — and the final frame is the whole wrapper the framework produces from your three files plus the page.

<ThreeBoundariesCompose />

Two things to carry out of that sequence.

First: **you write none of that wrapper.** Three small files plus your page, and the framework produces the entire nested-boundary structure. You never type a `<Suspense>`, never write an Error Boundary class, never wire a not-found catch. The files *are* the configuration.

Second: **the nesting order is not arbitrary — it decides behaviour.** The error boundary is outermost, which means a throw that happens *while the skeleton is showing* replaces the skeleton with the error UI; the error wins over the loading state, because it sits outside it. And the not-found boundary is innermost, inside Suspense, so a `notFound()` is caught after the loading phase, close to the page. Read the layers from the outside in and you can predict exactly what the user sees in any combination of states.

So here is the image to leave with. The finished `app/invoices/[id]/` is four files — and that is not a coincidence, it is one file per state:

<FileTree>
- app
  - invoices
    - **[id]**
      - page.tsx the populated state
      - loading.tsx the in-flight state
      - error.tsx the failed state
      - not-found.tsx the missing state
</FileTree>

Four states of a route, four files. When you look at any route from now on, the question to ask is the one this lesson trained: *where are my four states, and which file ships each?*

    Calling `reset()` in `error.tsx` re-fetches the data and tries again.
    <TfWhy>`reset()` only clears the error state and re-renders — it does not re-fetch. For a Server Component data error, re-rendering with the same data reproduces the same error. `unstable_retry()` is the one that re-fetches *and* re-renders.</TfWhy>
  </Statement>

  <Statement answer="false">
    A `fetch` that returns HTTP 404 automatically triggers `not-found.tsx`.
    <TfWhy>It does not. An HTTP 404 from a `fetch` is just a status code; nothing fires on its own. You read the response, turn it into `null` or a throw, and call `notFound()` yourself. The file is wired to the function, not to a status.</TfWhy>
  </Statement>

  <Statement answer="false">
    A `not-found.tsx` always returns an HTTP 404 status.
    <TfWhy>Only when the response is *not* streamed. Once the body has started streaming, the status is locked at the first byte, so a streamed not-found returns 200 — but Next.js injects a `<meta name="robots" content="noindex">` into that not-found page to keep it out of search results. The `noindex` meta, not the status code, is what protects SEO.</TfWhy>
  </Statement>
</TrueFalse>
