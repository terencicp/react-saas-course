In the last lesson you typed the props *contract* — what a component accepts as configuration. This lesson is about the other half of the contract: the *content*. Because a component's most powerful input isn't a string or a boolean. It's other JSX.

Here's the problem that forces the issue. You build a `<Card>` and it needs a title, a body, and a footer, so you give it three props:

```tsx
<Card title="Acme Inc." body={<p>Renews May 1.</p>} footer={<Button>Manage</Button>} />
```

That works. Then design wants a badge in the corner. Then an action menu. Then a divider. Each one is a new optional prop — `badge`, `action`, `divider` — and the call site stops reading like UI and starts reading like a config blob you have to decode. The prop list is the smell. The fix isn't a better prop; it's *no prop*. You let the consumer hand the card its content as nested JSX, and the card just decides where each piece goes:

```tsx
<Card>
  <CardHeader>
    <CardTitle>Acme Inc.</CardTitle>
  </CardHeader>
  <CardContent>Renews May 1.</CardContent>
  <CardFooter>
    <Button>Manage</Button>
  </CardFooter>
</Card>
```

That reads like HTML, because it *is* the same shape as HTML — nested elements, each in its place. By the end of this lesson you'll be able to decide between a prop and a slot, build that compound `<Card>` family from scratch, and dodge the one conditional-rendering bug that ships to production constantly. And this isn't a teaching toy: the family above is the exact shape every shadcn component takes, the one you'll import and read verbatim when we reach the shadcn library a few chapters from now. You're learning to write the thing you'll later consume.

## `children` is the universal content slot

Start from what you already do reflexively. In the last lesson, props arrived as one object and you destructured the names you wanted right in the parameter. `children` is no different — it's just another prop, and you pull it out the same way:

```tsx
({ children }: { children: ReactNode }) => { /* ... */ };
```

The only new idea is *where the value comes from*. Every other prop is passed as an attribute — `variant="primary"`, `disabled`. `children` is passed as whatever sits *between the opening and closing tags*. When the consumer writes `<Card>Anything here</Card>`, that text — and any JSX, in any amount — arrives as the `children` prop. React fills it in for you; you never pass it by name.

Wire it into the running `<Card>` spine from the last lesson — the same `cn()` merge, the same `...rest` discipline, nothing about that changes:

<CodeTooltips tooltips={{
  'children:': 'The content between the tags. Typed ReactNode — anything React can render.',
  ReactNode: 'The wide renderable type: JSX, strings, numbers, arrays, fragments, portals, and null/undefined/booleans (which render as nothing).',
  '{children}': "Place the consumer's content wherever you want it inside your own markup.",
}}>
```tsx
type Props = ComponentProps<'div'> & {
  children: ReactNode;
};

const Card = ({ children, className, ...rest }: Props) => (
  <div className={cn('rounded-lg border bg-card p-6', className)} {...rest}>
    {children}
  </div>
);
```
</CodeTooltips>

Three things are worth saying out loud about this block. First, `children` comes *in* through the props object and gets *placed* by `{children}` inside your JSX. The component decides where the content lands — here it's wrapped in the bordered, padded `<div>`, but you could put markup before or after it, or around it. The consumer brings the content; you frame it.

Second, the type. `children` is typed `ReactNode` — the deliberately *wide* "anything React can render" type. And it really is wide: it accepts JSX elements, plain strings, numbers, arrays of those, fragments, portals, and — this is the part that matters in a minute — `null`, `undefined`, and `boolean`, all three of which render as nothing. Don't read that width as sloppiness. It's the feature. It's *why* `{condition && <Thing />}` and `{items.map(...)}` both just work when you drop them into JSX: a `false` renders nothing, an array renders each element. `ReactNode` is wide on purpose so that everything you'd naturally want to render is already legal.

Third — and you'll see this in older code or in a teammate's PR — there's a narrower cousin called <Term definition="A single JSX element — the narrow renderable type. Only reach for it when a component must inspect or clone its one child; it rejects strings, numbers, and fragments.">`ReactElement`</Term>. Where `ReactNode` is "anything renderable," `ReactElement` is "*a single JSX element*," and nothing else. It's the wrong default almost every time: type `children: ReactElement` and your component breaks the moment someone passes a string, two elements, or a fragment. You reach for it only in the rare case where a component must *inspect or clone* its one child — and even then, modern React has a cleaner, typed way to do that, which is what the next lesson's `asChild` is for. The rule from the code conventions is blunt and worth memorizing: **children are `ReactNode`, never `JSX.Element` or `ReactElement`.**

:::note
That "renders as nothing" clause is doing real work. A component can return `null` to render nothing at all, and `{loggedIn ? <Dashboard /> : null}` is completely valid because `null` is a member of `ReactNode`. You'll lean on this constantly.
:::

## Compound components: regions as JSX, not props

A `<Card>` that just wraps `children` is fine when the card is one undivided blob of content. But real cards have *structure* — a header, a body, a footer, maybe an action in the top-right corner. The question is how the consumer puts content into each of those regions. The naive answer, the one this chapter opened on, is a prop per region. The senior answer is a *compound component*: the card ships not as one component but as a small, tightly-coupled **family**, and the consumer composes the family with JSX.

This is *the* shadcn pattern, so let's learn the exact family you'll be importing later. The current shadcn `<Card>` looks like this:

<FileTree>
- **Card** the outer container
  - CardHeader the top region
    - CardTitle a semantic heading
    - CardDescription muted subtext
    - CardAction the top-right slot — a button or a badge
  - CardContent the body
  - CardFooter the bottom region
</FileTree>

Read that tree as a contract, not as files — every one of those lives in a single `card.tsx`. `Card` is the container. Inside it, `CardHeader` groups the top region, and the header itself holds a `CardTitle` (a real heading), an optional `CardDescription`, and a `CardAction` for whatever sits top-right — a button, a badge, a menu trigger. `CardContent` is the body. `CardFooter` is the bottom strip. The consumer picks the parts they need, in the order they want, and skips the rest.

To feel *why* this beats a prop per region, watch the two designs side by side. The same card, built both ways:

<CodeVariants>
  <CodeVariant label="A prop per region">
    ```tsx
    <Card
      title="Acme Inc."
      description="Pro plan"
      body={<p>Renews May 1.</p>}
      action={<Button size="sm">Upgrade</Button>}
      footer={<Button>Manage</Button>}
    />
    ```
    **Every region is a prop, so the prop list grows forever.** The card has to enumerate every region it might ever hold, and the moment design wants a new one — a badge, a divider — you edit the *component* to add another optional prop and another branch in its body. The call site is a config object you decode top to bottom.
  </CodeVariant>

  <CodeVariant label="A subcomponent per region">
    ```tsx
    <Card>
      <CardHeader>
        <CardTitle>Acme Inc.</CardTitle>
        <CardDescription>Pro plan</CardDescription>
        <CardAction>
          <Button size="sm">Upgrade</Button>
        </CardAction>
      </CardHeader>
      <CardContent>Renews May 1.</CardContent>
      <CardFooter>
        <Button>Manage</Button>
      </CardFooter>
    </Card>
    ```
    **Every region is a subcomponent, so a new region is a new export, not a new prop.** The consumer places real JSX in each slot, controls the order, omits what they don't need, and a new region tomorrow is one more thin wrapper in `card.tsx` — the existing call sites never change. It reads like the markup it produces.
  </CodeVariant>
</CodeVariants>

The shift is subtle but total. In the first design, the *component* owns the list of possible regions; every new region is the component's problem. In the second, the *consumer* owns the composition; the component just supplies the building blocks. That's the whole idea behind "composition over configuration" — when a component grows a prop for every region it might hold, stop adding props and start handing out pieces.

So what does a "piece" actually look like? Each member of the family is the most ordinary thing in the world: an L1-style typed component that wraps one element, owns its own classes, and forwards `className` and `...rest`. There's nothing new to learn per part — it's the same spine, repeated. Here's the family in one file:

<AnnotatedCode lang="tsx" code={`
const Card = ({ className, ...rest }: ComponentProps<'div'>) => (
  <div className={cn('rounded-lg border bg-card p-6', className)} {...rest} />
);

const CardHeader = ({ className, ...rest }: ComponentProps<'div'>) => (
  <div className={cn('flex items-start justify-between gap-4', className)} {...rest} />
);

const CardTitle = ({ className, ...rest }: ComponentProps<'h3'>) => (
  <h3 className={cn('font-semibold leading-none', className)} {...rest} />
);

// CardDescription, CardContent, CardFooter follow the same shape.

const CardAction = ({ className, ...rest }: ComponentProps<'div'>) => (
  <div className={cn('shrink-0', className)} {...rest} />
);

export { Card, CardHeader, CardTitle, CardAction };
`}>
  <AnnotatedStep meta="{19}" color="blue">
    This one file exports a *set* of components, not one. That's the sanctioned exception to "one component per file" the last lesson named — a tightly-coupled family you never use one piece of in isolation ships together.
  </AnnotatedStep>

  <AnnotatedStep meta="{1-3}" color="blue">
    The container is the exact spine from the last lesson — `cn(base, className)`, `...rest` spread onto a plain `<div>`. Notice there's no `children` in the destructure here: `children` rides along inside `...rest` and lands on the `<div>` automatically, so you don't have to name it unless you want to place it deliberately.
  </AnnotatedStep>

  <AnnotatedStep meta="{5-7}" color="blue">
    A subcomponent is a thin wrapper. It owns *its* classes (here, the flex layout that pushes the action to the right), accepts *its own* `className`, and spreads `...rest`. Each part styles its region and nothing else.
  </AnnotatedStep>

  <AnnotatedStep meta="{9-11}" color="blue">
    `CardTitle` renders a semantic `<h3>`, not a styled `<div>`. The accessibility lives *in the subcomponent* — the consumer writes `<CardTitle>` and gets a real heading for free. (We'll lean hard on this idea in the shadcn chapter.)
  </AnnotatedStep>

  <AnnotatedStep meta={`{15-17} "ComponentProps<'div'>"`} color="blue">
    Every part is its own L1-typed component — `ComponentProps<'div'>` pulls in every native attribute. `CardAction` is the cleanest proof of the whole pattern: a brand-new region — the top-right slot — is a *new export*, never a new prop on `<Card>`.
  </AnnotatedStep>
</AnnotatedCode>

Look at what just happened across those five steps: there was no new mechanism. Every part is a function you already know how to write from the last lesson. The *pattern* is entirely in the packaging — a coordinated set of thin wrappers, exported together, each owning one region's styling. That's the lever. Compound components feel sophisticated and turn out to be the simplest thing in the chapter.

Now, *why* is this worth it? Three concrete payoffs, each tied to an edit you can picture making:

- **It scales by addition, not modification.** Design wants a `CardBadge` next month? You write one more eight-line wrapper and export it. Every existing `<Card>` in the codebase keeps working untouched. The prop-per-region card would force you to edit the component *and* risk every call site.
- **The consumer controls order and presence.** Want the footer above the content for one card, or no header at all? Reorder the JSX, drop the part you don't want. The component never dictated a fixed layout, so there's nothing to fight.
- **Each part takes a `className`.** Because every wrapper forwards `className` through `cn()`, the consumer can restyle one region in place — `<CardHeader className="bg-muted">` tints just that header. No prop to add, no fork; the styling escape hatch from the last lesson is open on every part.

One honest caveat, and you should plant it in your mental model now so you don't go hunting for a guarantee that isn't there: **nothing stops a consumer from rendering `<CardFooter>` outside of a `<Card>`.** They're just exported components. The coupling between them is convention and documentation, not the type system — TypeScript will not error if you misuse them. That's a real limitation of the pattern, and the senior response is to *document the family* so consumers know the parts belong together. (You might wonder whether the parts should share state — a context that links them. For a styling-only family like this `<Card>`, no: these parts only carry classes, and reaching for context here would be over-engineering. Context-linked compound components are a real pattern, but a later one.)

Step back and notice the bigger picture. This exact shape — a family of thin, composable parts — is what *every* modern component library is built on: shadcn, Radix, Ariakit. shadcn leaned in so hard that it now ships these composition trees right in its docs, specifically so that humans (and coding agents) assemble the parts in the right shape without forgetting a wrapper. You're not learning a quirky technique. You're learning the default form of a 2026 component library — by writing the thing you'll soon import unmodified.

## One region? Reach for a prop, not a subcomponent

So compound components are the answer. Always? No — and reaching for them *everywhere* is the equal-and-opposite mistake to never discovering them. There's a whole class of components that own exactly one named region, and for those a subcomponent is pure ceremony.

Take the `<Button>` from the last lesson. A common need: an icon before the label — a trash icon on a delete button, a plus on a create button. You *could* invent a subcomponent for it:

<CodeVariants>
  <CodeVariant label="A subcomponent (overkill)">
    <div data-mark-color="red">

    ```tsx
    <Button>
      <ButtonIcon>
        <TrashIcon />
      </ButtonIcon>
      Delete
    </Button>
    ```

    </div>
    **A whole subcomponent for a single fixed slot.** A button has exactly *one* icon position. Spinning up a `<ButtonIcon>` family member to fill it is the compound pattern applied where there's nothing to compose — ceremony with no payoff.
  </CodeVariant>

  <CodeVariant label="A prop (right-sized)">
    <div data-mark-color="green">

    ```tsx
    <Button leftIcon={<TrashIcon />}>Delete</Button>
    ```

    </div>
    **One region, so one `ReactNode` prop.** The button owns a single icon slot, so a named prop expresses it exactly — readable at the call site, and the consumer still hands in any JSX they like.
  </CodeVariant>
</CodeVariants>

This is **prop-as-slot**: a named region passed as a `ReactNode` prop. It's the same trick as `children` — the consumer hands you JSX — except the region is *named*, so the consumer (and the types) know exactly which slot it fills. Wiring it onto the `<Button>` is a one-line addition to the spine you already have:

<div data-mark-color="green">

```tsx "leftIcon?: ReactNode" "{leftIcon}"
type Props = ComponentProps<'button'> & {
  variant?: 'primary' | 'destructive' | 'ghost';
  size?: 'sm' | 'md' | 'lg';
  leftIcon?: ReactNode;
};

const Button = ({
  variant = 'primary',
  size = 'md',
  leftIcon,
  children,
  className,
  ...rest
}: Props) => (
  <button className={cn(buttonClasses({ variant, size }), className)} {...rest}>
    {leftIcon}
    {children}
  </button>
);
```

</div>

(`buttonClasses` is the stand-in from the last lesson for "variant + size → class string." Don't build it.)

Notice that both forms — `children` and `leftIcon` — are typed `ReactNode`, and both forward `className` through `cn()`. They are *the same tool*. The only thing that differs is the **number of regions**. `children` is the unnamed catch-all; `leftIcon` is a named single slot. So how do you decide, in the moment, which to reach for? There's a clean rule, and it's the load-bearing takeaway of this whole lesson:

> **Zero or one named region → prop-as-slot (or just `children`). Two or more → compound.**

With a corollary that overrides the count: if the consumer needs to *reorder*, *omit*, or *independently restyle* the regions, that pushes you toward compound even at two regions — because those three freedoms are exactly what compound components give and props don't.

A rule you only read is a rule you forget. So apply it. Walk the decision tree below for a handful of real components — at each step, answer the question, and the walker lands you on a recommendation. Try to predict each verdict before you click it.

## Conditional rendering and the `0`-falsy trap

You'll rarely render the same content unconditionally. A panel shows only when it's open; an error banner replaces the form when something breaks; a list appears only when it has items. React doesn't have a special syntax for this — it falls straight out of the JSX rules you already met, where booleans, `null`, and `undefined` render as nothing. That's the payoff of `ReactNode` being so wide: the two everyday reflexes are just JavaScript expressions in a `{}`.

```tsx
{isOpen && <Panel />}

{isError ? <Alert /> : <Content />}
```

The first is the **on/off** reflex: `{isOpen && <Panel />}`. When `isOpen` is `true`, `&&` evaluates to the right-hand side and the `<Panel />` renders; when it's `false`, the whole expression *is* `false`, which renders as nothing. That only works because `false` is a legal, render-as-nothing member of `ReactNode` — the width of the type is exactly what makes the idiom safe. The second is the **pick-one** reflex: `{isError ? <Alert /> : <Content />}` renders one branch or the other. Reach for `&&` when the choice is "show this or show nothing," and the ternary when it's "show this or show that."

Now the bug. It is, without exaggeration, the single most common conditional-render mistake shipped to production, and it hides inside the perfectly reasonable-looking `&&` reflex. Watch:

<div data-mark-color="red">

```tsx "count"
{count && <List items={items} />}
```

</div>

Reads fine. "If there's a count, render the list." But run it when `count` is `0`, and React prints a literal `0` to your page — a stray zero floating next to an empty list, in production, in front of a user. Here's the mechanism, and it's worth understanding rather than memorizing. `0` is falsy, so `&&` short-circuits and the *whole expression evaluates to `0`* — not `false`, the number `0`. And `0` is a number, and numbers are a perfectly renderable member of `ReactNode`. So React does exactly what you told it: it renders the `0`. (The empty string `''` does the same thing, for the same reason.) The `&&` idiom is safe with a *boolean* on its left, because `false` renders nothing — but a *number* on the left can short-circuit to a renderable value.

Before the fix, predict the bug yourself:

  <McqChoice>An empty `<span>` — `0` is falsy, so the whole expression drops out and nothing renders.</McqChoice>
  <McqChoice correct>A `<span>` containing the digit `0` — the user sees a stray `0` where the badge should have been.</McqChoice>
  <McqChoice>Just the `<Badge>New</Badge>` — a falsy left side is ignored and the right side renders anyway.</McqChoice>
  <McqChoice>A runtime error — `&&` can't combine a number with a JSX element.</McqChoice>

  <McqWhy>`messageCount && <Badge>New</Badge>` short-circuits on the falsy `0`, so the expression's value *is* the number `0` — not `false`, not nothing. Numbers are renderable `ReactNode`, so React dutifully prints `0` inside the span. The `<Badge>` never renders, but the `0` does. Put a real boolean on the left — `messageCount > 0 && …` — and it disappears.</McqWhy>
</MultipleChoice>

The fix is to put a *real boolean* on the left of `&&` — never a bare number or string. Three forms, all from the code conventions, pick whichever reads best at the call site:

<div data-mark-color="green">

```tsx "count > 0" "Boolean(count)" "value != null"
{count > 0 && <List items={items} />}

{Boolean(count) && <List items={items} />}

{value != null && <Field value={value} />}
```

</div>

The first compares to a number, producing a real `true`/`false` — the cleanest read when you genuinely mean "more than zero." `Boolean(count)` coerces explicitly when any non-zero count should show. And `value != null` is the one to reach for with a *nullable* value (it catches both `null` and `undefined` while letting through legitimate falsy values like `0` or `''`). The rule underneath all three: **the left side of `&&` in JSX must be a real boolean.** Get that habit and the trap simply never fires.

## Fragments group without a wrapper

A component must return *one* parent element — you can't return two siblings side by side. The lazy fix is to wrap them in a `<div>`, and most of the time that's harmless. But sometimes that extra `<div>` is actively wrong: it breaks a flex or grid layout that expected direct children, or it produces invalid HTML where the parent demands specific child tags. For those cases React gives you a wrapper that emits *no DOM node at all* — the <Term definition="Groups sibling elements without adding a DOM node of its own. The <>...</> shorthand is the everyday form.">fragment</Term>.

The cleanest motivation is a definition list. A `<dl>` expects bare `<dt>`/`<dd>` pairs as its children; slip a `<div>` between them and the HTML is invalid. So a component that renders one row of the list has to return two siblings with no wrapper:

```tsx "<>" "</>"
const PlanRow = ({ label, value }: { label: string; value: string }) => (
  <>
    <dt className="text-muted-foreground">{label}</dt>
    <dd className="font-medium">{value}</dd>
  </>
);
```

Those `<>` and `</>` are the **fragment shorthand** — an empty tag that groups its children and renders nothing of its own. The two `<dt>`/`<dd>` elements come out as direct, valid children of whatever `<dl>` the consumer drops `PlanRow` into. No wrapper, no invalid markup, no broken layout. This shorthand is the daily reach; you'll type it constantly.

There's one case where the shorthand can't be used: when the fragment is a *list item*. If you map over a list and each iteration emits two sibling elements, React needs a `key` on each — its way of tracking which item is which — and you can't put a `key` on `<>`. There you switch to the longhand `<Fragment>`:

```tsx "key={plan.id}"
{plans.map((plan) => (
  <Fragment key={plan.id}>
    <dt>{plan.label}</dt>
    <dd>{plan.price}</dd>
  </Fragment>
))}
```

We'll get to *why* lists need a `key` — and what React does with it — in the next chapter, on how components render. For now, two facts are enough: when a fragment is part of a mapped list, use `<Fragment key={...}>` instead of `<>`; and outside of lists, fragments don't take a `key` at all. Don't sprinkle `key` onto a fragment that isn't a list item — it does nothing there.

## Children as a function: the render prop you'll recognize but rarely write

There's one more shape `children` can take, and the honest framing up front is: **you will read this, and you will almost never write it.** I'm teaching it so it doesn't startle you in someone else's code, not because you'll reach for it.

So far `children` has been *content* — JSX the component places. But `children` can also be a *function*. The component owns some value — state, a subscription, the result of a fetch — and instead of rendering content directly, it *calls* `children` with that value and lets the consumer decide what to render:

```tsx {2}
<DataLoader url="/api/invoices">
  {(invoices) => <InvoiceList items={invoices} />}
</DataLoader>
```

Here `children` isn't JSX — it's `(invoices) => <InvoiceList items={invoices} />`, a function. `DataLoader` does the loading, then *calls* that function with the data, handing control of the render back to the consumer. Typed, `children` here is `(data: T) => ReactNode` — a function that takes the component's value and returns something renderable. The name for this is a <Term definition="A prop — often children — that is a function the component calls to let the consumer control what renders, passing it data the component owns.">render prop</Term>.

It's a clever pattern, and it is **not deprecated** — but in 2026 it's a recognition-level thing, not a daily tool. The reason: nearly every case that historically reached for a render prop is now a **custom hook**. `DataLoader` as a render-prop component is the old shape; `const invoices = useInvoices()` is the modern one — same capability, no nesting, no function-as-child. (We build custom hooks properly a few chapters on.) Render props survive only in the narrow case where a component *also* renders chrome around the consumer's output — a layout, a boundary — and even those are few. The takeaway is one sentence: **read it without flinching; reach for a custom hook before you ever write one.**

There's an even rarer cousin worth one breath. React ships utilities — `Children.map`, `Children.toArray` — for *iterating* a component's children. The 2026 senior instinct is to **not** iterate children: if a component needs to know about a list of things, expose a *data prop* and map over the data, not the JSX. `Children.toArray` exists for the genuine edge case where a component must inspect its children — a custom `<Tabs>` reading its `<Tab>` elements to build a tab strip — and wants stable keys while doing it. Know it's there; treat it as rare and intentional. And one trap to name explicitly: do **not** reach for `cloneElement` or `Children.map` to *inject props* into children. That's the old hand-rolled way, and the next lesson's `asChild` does exactly that with a typed, sanctioned contract.

## Composition is the first answer to prop drilling

One last idea, as a preview — it points at a problem you'll solve properly later, but the seed belongs here, because composition is the first tool you should reach for.

Picture a layout nested a few levels deep: `<Layout>` renders `<Page>`, which renders `<Header>`, which renders `<Toolbar>`. Only the `<Toolbar>` at the very bottom needs the current `user` — to show their avatar, say. But the `user` enters at the top. So you thread it down: `<Layout>` takes a `user` prop and passes it to `<Page>`, which takes it and passes it to `<Header>`, which takes it and passes it to `<Toolbar>`. Three components in the middle accept and forward a prop they have no use for, purely to relay it. That's **prop drilling** — and every one of those middle components now has `user` in its signature for no reason of its own.

There's a fix that needs no new tool, just the composition you learned today. Instead of `<Layout>` *creating* the toolbar deep inside, let it *accept* the finished toolbar as a slot — `children`, or a named `ReactNode` prop. Now the consumer wires `user` to `<Toolbar user={user} />` at the top, at the call site, and hands the assembled element down. The middle layers pass an opaque `ReactNode` through; they never see `user`, never name it, never know it exists.

The frame to hold onto: composition makes *the component that needs the prop and the component that has the prop into siblings at the call site*. They meet where the `user` already is, so nothing has to drill.

Now the honest preview part. This is not the whole story of where state lives or how to share it widely — those are real topics we treat properly later (where state should live, and Context as the tool for genuinely cross-cutting values). The senior point to plant *now* is the reflex order: when a value has to reach somewhere deep, **try composition first.** Prop drilling is not automatically a bug that demands Context — reaching for Context too early is its own smell — and composition dissolves a surprising amount of drilling for free, with tools you already have. Keep it in the front of your mind; we'll add the heavier tools to the back of it later.

## What you can do now

You came in able to type what a component *accepts*; you leave able to type what it *contains*. The throughline of the whole lesson was one senior question asked over and over — *does this region deserve a prop, or a slot?* — and the answer is a rule you can now apply on sight: zero or one region, a prop or `children`; two or more, a compound family. You can build that family — the shadcn `<Card>` you'll soon import for real — out of nothing but the thin typed wrappers from the last lesson. You can render content conditionally without printing a stray `0`. And you know that when a prop has to travel deep, composition is the first thing to try.

The next lesson takes the same `<Button>` and `<Card>` and asks a different question: what if a `<Button>` needs to render as a *link*? That's polymorphism, and it's where `asChild`, `Slot`, and `cva` enter — the last pieces that make these components the real shadcn shape.
