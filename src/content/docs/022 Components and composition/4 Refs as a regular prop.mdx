Your `<Input>` renders a real `<input>` somewhere inside it. Sooner or later a parent needs to reach that node directly — focus it the moment a form opens, scroll it into view after a validation error, measure its width. None of that goes through props or state; it goes through a <Term definition="A handle that points straight at a DOM node (or a value), exposed on a ref object's .current. Lets a parent reach the actual element a component renders.">ref</Term>, a handle that points straight at the DOM node. The parent holds the ref. The problem is getting it onto the actual `<input>` buried inside your component.

The whole lesson answers one narrow question: **how does a ref cross the component boundary?** You already hand `onClick`, `className`, and `...rest` down to the inner element. The answer in React 19 is that `ref` is now just one more of those — an ordinary prop you destructure and pass along. Before React 19 this took a dedicated API called `forwardRef` and a few lines of ceremony per component; React 19 deleted that ceremony. By the end you'll write a focusable `<Input>`, send a ref through `asChild` so it lands on a rendered `<a>`, and wire a callback ref to an observer — and you'll recognise `forwardRef` when you read it in older code without ever reaching for it yourself.

## `ref` is a regular prop now

Here's the whole model in one sentence: a function component accepts `ref` as an ordinary prop, you destructure it, and you hand it to the inner DOM element. That's it. The canonical `<Input>`:

<CodeTooltips tooltips={{
  ref: 'Now an ordinary prop. React assigns the DOM node to it after the element mounts.',
  '...props': 'Every native input attribute — value, onChange, placeholder — minus the ref we destructured out.',
  "ComponentProps<'input'>": "All props the JSX input element accepts, including ref typed as Ref<HTMLInputElement>.",
}}>
```tsx
const Input = ({ ref, ...props }: ComponentProps<'input'>) => (
  <input ref={ref} {...props} />
);
```
</CodeTooltips>

Read that destructure carefully, because the payoff is hiding in the type. You did **not** add a `ref` field to a props type. `ComponentProps<'input'>` — the element-props alias you've used since the typed-props contract lesson — *already includes* `ref`, already typed against `HTMLInputElement`. The ref rides in for free with the same alias that gives you `value`, `onChange`, and every other native attribute. Pull `ref` out of the destructure, drop it on the `<input>`, and the forward is done.

This is the arrow-bound-to-`const` form the rest of the course uses, and it's the shape you should ship. A plain `function Input({ ref, ...props }: ComponentProps<'input'>) { ... }` declaration is exactly equivalent — `ref` is a real parameter either way — but stay with the arrow form so your components all read the same.

The parent side is small, and it's the only place `useRef` shows up in this lesson:

```tsx
const SearchBar = () => {
  const inputRef = useRef<HTMLInputElement>(null);

  return <Input ref={inputRef} placeholder="Search" />;
};
```

`useRef(null)` produces the ref; passing it as `ref={inputRef}` is the whole job here. React assigns the rendered `<input>` to `inputRef.current` once it mounts, so the parent can later call `inputRef.current?.focus()`. That `.current` mechanic — how a ref stores a value, when it's safe to read, what else it's good for — gets its full treatment in the lesson on `useRef` later in the next chapter. For now treat `useRef(null)` as nothing more than the thing that produces the ref you pass down.

### `forwardRef` was the old ceremony

If `ref` is just a prop, what did this look like before? Every codebase written before 2024, every shadcn component file before mid-2025, every pre-React-19 tutorial wrapped its components in `forwardRef`. You will read it. The before/after is the single most useful thing to fix in your head:

<CodeVariants>
  <CodeVariant label="React 18 — forwardRef">
    <div data-mark-color="red">

    ```tsx del={1} del=", ref" del={3} del={4}
    const Input = forwardRef<HTMLInputElement, ComponentProps<'input'>>(
      ({ ...props }, ref) => <input ref={ref} {...props} />,
    );
    Input.displayName = 'Input';
    ```

    </div>
    **The old ceremony.** `forwardRef` wrapped the component so React could thread a second `ref` argument in beside `props`; you also paid for an explicit generic pair and a `displayName` so the component showed up named in DevTools. Six lines of boilerplate, on every component that wanted a ref.
  </CodeVariant>

  <CodeVariant label="React 19 — ref as a prop">
    <div data-mark-color="green">

    ```tsx ins="ref,"
    const Input = ({ ref, ...props }: ComponentProps<'input'>) => (
      <input ref={ref} {...props} />
    );
    ```

    </div>
    **Same behaviour, the wrapper and the second parameter are gone.** `ref` is a normal prop now — destructure it, forward it. No wrapper, no generic pair, no `displayName`. The component contract got smaller, and there's nothing magic left to remember.
  </CodeVariant>
</CodeVariants>

`forwardRef` still runs in React 19, but it logs a deprecation warning and is slated for removal in a future major version. You never write it. When you inherit a codebase full of it, the official <Term definition="An automated source-transform that rewrites code mechanically — you run a script and it edits your files, no hand-editing.">codemod</Term> `npx codemod react/19/remove-forward-ref` rewrites every `forwardRef` call to the prop form mechanically, and ESLint flags the redundant wrappers so they don't creep back in. Old ref code is convertible by a script — that's worth knowing so you don't treat a `forwardRef`-heavy repo as a rewrite.

:::note
`forwardRef` is a legacy React API that wrapped a component so it could receive a forwarded `ref` as a second argument. In React 19 it's unnecessary — recognise it in old code, never author it.
:::

Why did React bother removing it? Because `ref` used to be **reserved** by the runtime, the same way `key` is — a prop name React intercepted before it ever reached your component, so a component literally could not receive `ref` through `props`. `forwardRef` existed solely to smuggle the reserved prop back in. Making `ref` ordinary removes the special case: the contract shrinks, and the React Compiler — already enabled in this course's stack — can reason about `ref` exactly like it reasons about any other prop instead of treating it as a runtime quirk. This is the same move you'll see again with the `'use client'` and `'use server'` directives in the App Router chapter: React 19 keeps trading magic for plain, inspectable contracts. When something used to be special-cased and now just isn't, that's usually the direction.

## Typing a ref: `Ref`, `RefObject`, and `RefCallback`

Most of the time you never write a ref type at all, because `ComponentProps<'input'>['ref']` resolves it for you. You reach for the explicit type in exactly one situation: when your props type is *not* an element-props alias — a component whose props you wrote by hand that should nonetheless accept a ref to its root node. Three type names cover everything you'll meet.

<CodeTooltips tooltips={{
  Ref: 'The type an element\'s ref prop accepts. The union of an object ref, a callback ref, and null. This is the one you write when annotating a custom ref prop by hand: ref?: Ref<HTMLDivElement>.',
  RefObject: 'What useRef returns. Its .current holds the value. In React 19 .current is writable; the old MutableRefObject type is deprecated.',
  RefCallback: 'The function form of a ref. React calls it with the DOM node on mount. In React 19 it may also return a cleanup function.',
}}>
```tsx
type Ref<T> = RefObject<T> | RefCallback<T> | null;

type RefObject<T> = { current: T };
type RefCallback<T> = (node: T | null) => void;
```
</CodeTooltips>

`Ref<T>` is the one you'll actually type. When you hand-write a component's props and still want it to accept a ref, you add `ref?: Ref<HTMLDivElement>` yourself — and notice it's a *union*, not a single object type. That union (object, function, or `null`) pays off when you merge two refs onto one node later in this lesson, so hold onto it.

`RefObject<T>` is what a `useRef` call hands back; its `.current` holds the value. There's one historical wrinkle worth clearing up so older code doesn't trip you. Before React 19, `RefObject<T>` meant "`.current` is readonly" and a separate `MutableRefObject<T>` meant "writable `.current`" — two types for the same idea. React 19 collapsed them: `RefObject<T>` now has a **writable** `.current`, and `MutableRefObject` is deprecated. You only ever write `RefObject`; the only reason to know `MutableRefObject` exists is to recognise it in pre-2025 code and not copy it forward. (A `refobject-defaults` codemod handles the related `useRef(null)` argument change automatically — recognition-level, nothing to do by hand.)

`RefCallback<T>` is the function form — `(node: T | null) => void`. That's the next section, and it's where refs stop being a dull pass-through and start doing real work.

The decision rule is the whole takeaway here, so make it stick: **element-props alias → the ref type comes for free; hand-written props → annotate `ref?: Ref<T>`.** Quick check:

A component with hand-written props — not a `ComponentProps<'tag'>` alias — that should still accept a ref to its root `<div>`. Fill in the ref type.

  answers={[
    { answer: 'Ref<HTMLDivElement>', options: ['Ref<HTMLDivElement>', 'RefObject<HTMLDivElement>', 'MutableRefObject<HTMLDivElement>', 'HTMLDivElement'] },
  ]}
>
```tsx
type DropdownProps = {
  items: string[];
  ref?: ___;
};
```
</Dropdowns>

The blank is `Ref<HTMLDivElement>`. An element's `ref` prop accepts the full `Ref<T>` union — an object ref *or* a callback ref — so that's the type a hand-written prop must declare to match it. `RefObject<HTMLDivElement>` alone rejects callback refs; `MutableRefObject<HTMLDivElement>` is the deprecated pre-React-19 spelling you only ever read; and `HTMLDivElement` is the *node itself*, not a ref pointing at it.

## Ref callbacks: running code when a node mounts

A ref doesn't have to be an object. Set `ref` to a *function* and React calls it with the DOM node the moment the element mounts, handing you the node directly instead of stashing it on `.current`:

```tsx
<input ref={(node) => node?.focus()} />
```

Why would you want that? Because some work needs the DOM node *at the instant it attaches*, not whenever you happen to read `.current` later. An object ref gives you the node after the fact; a callback ref gives it to you on attach, which is exactly what you need to wire up an observer, measure the element's size, or attach a non-React event listener the moment it exists. The canonical reach — and one this course comes back to for lazy-loading images — is hooking an `IntersectionObserver` to a node so you get told when it scrolls into view:

```tsx
<div ref={(node) => {
  const observer = new IntersectionObserver(([entry]) => {
    console.log(entry.isIntersecting ? 'in view' : 'out of view');
  });
  observer.observe(node);
}} />
```

The <Term definition="Browser API that runs a callback when an element enters or leaves the viewport. Covered for lazy-loading later in the course.">`IntersectionObserver`</Term> is a browser API that fires a callback when an element enters or leaves the viewport — you'll meet it properly later when the course covers lazy-loading. Ignore its details for now; the point is that it needs the real `<div>` node to start watching, and the callback ref is what hands that node over at the right moment.

One stability note, kept light because the full story belongs to the render model in the next chapter. An inline callback ref like the one above is a brand-new function on every render. Without the cleanup return you're about to learn, React responds to a new function by running the old one with `null` and then the new one with the node, every render. If the setup is expensive, wrapping the callback in `useCallback` gives React a stable function so it stops re-running. State that as a fact for now; *why* a new function triggers a re-run is a render-model question the next chapter answers.

## Returning a cleanup from a ref callback

Wiring up an observer raises an obvious question: who tears it down? An observer you create but never disconnect keeps a live reference to a node that's left the page — a leak. React 19 added the clean answer: **a ref callback may `return` a cleanup function**, and React runs it when the node unmounts.

```tsx ins={6}
<div ref={(node) => {
  const observer = new IntersectionObserver(([entry]) => {
    onVisible(entry.isIntersecting);
  });
  observer.observe(node);
  return () => observer.disconnect();
}} />
```

The mental model is the cleanest part of this whole lesson: **a mini-effect scoped to the life of one DOM node.** Setup runs when the node attaches, the returned function runs when it detaches, and you never reach for the `useEffect` + `useRef` pairing people used to write for this. Setup and teardown live side by side in the same callback.

There's one precise behavioural detail to lock in, because it's the kind of thing that quietly changes how the old mental model maps onto the new API. When you return a cleanup, React **no longer** calls the callback again with `null` on unmount — the returned function *is* the unmount signal now. The pre-React-19 pattern was a single callback invoked twice, once with the node and once with `null`, and you branched on which you got. The React 19 pattern is setup-then-cleanup, two separate functions, no `null` call. The lifecycle:

<RefCallbackLifecycle />

One TypeScript trap rides along with this feature, and it bites people right after they upgrade, so spot it now. TypeScript in React 19 rejects a ref callback that returns anything other than a cleanup function, `undefined`, or `null`. The problem is the one-line arrow body. An arrow with no braces *implicitly returns* its last expression — so `ref={(node) => (mapRef.current = node)}` returns the *value of the assignment*, which is the node, which is not a cleanup, which is now a type error. The fix is a block body that returns nothing:

<CodeVariants>
  <CodeVariant label="Type error">
    <div data-mark-color="red">

    ```tsx {1}
    <div ref={(node) => (mapRef.current = node)} />
    ```

    </div>
    **Implicit return.** The parenthesised arrow body returns the assignment's value — the node — and TypeScript reads that as a returned non-cleanup. Compile error, the moment you upgrade to React 19.
  </CodeVariant>

  <CodeVariant label="Block body">
    <div data-mark-color="green">

    ```tsx {1}
    <div ref={(node) => { mapRef.current = node; }} />
    ```

    </div>
    **Returns nothing.** Braces make it a statement body with no return, so the callback returns `undefined` — which is allowed. The only change is two curly braces.
  </CodeVariant>
</CodeVariants>

## Putting two refs on one node

Here's the genuinely tricky case, and it's where the `Ref<T>` union from earlier pays for itself. Suppose your `<Input>` needs to keep *its own* internal ref on the `<input>` — to focus itself when some event fires — **and** it still has to forward the caller's `ref` to that same `<input>`. Two refs, one node. The `ref` attribute takes exactly one value, so you can't write `ref={a} ref={b}`. A callback ref solves it: one function, fired with the node, writes into both refs.

<AnnotatedCode lang="tsx" maxLines={18} code={`
const Input = ({ ref, ...props }: ComponentProps<'input'>) => {
  const internalRef = useRef<HTMLInputElement>(null);

  return (
    <input
      ref={(node) => {
        internalRef.current = node;
        if (typeof ref === 'function') ref(node);
        else if (ref) ref.current = node;
      }}
      {...props}
    />
  );
};
`}>
  <AnnotatedStep meta="{2}" color="blue">
    The component owns an internal ref — say, to focus the field after a validation error. It needs to point at the same `<input>` the caller's ref points at.
  </AnnotatedStep>

  <AnnotatedStep meta="{6-10}" color="blue">
    One callback ref, fired with the node, is the seam where every ref that needs this node gets written. You can't pass two refs to one `ref` attribute — so you fan one out by hand.
  </AnnotatedStep>

  <AnnotatedStep meta="{7}" color="green">
    Wire the component's own ref: assign the node straight onto `internalRef.current`.
  </AnnotatedStep>

  <AnnotatedStep meta="{8-9}" color="orange">
    Now the caller's `ref`. It might be a callback (`typeof ref === 'function'` → call it with the node) or an object (`ref.current = node`) or `null` (do nothing). That three-way branch is the `Ref<T>` union from earlier — this is exactly why `Ref<T>` is a union and not just an object.
  </AnnotatedStep>
</AnnotatedCode>

That branch is correct, and it's also boilerplate you don't want to retype on every component that forwards a ref. So teams pull it into a helper — a <Term definition="Helper that fans one DOM node out to several refs — write it in ~5 lines or import react-merge-refs.">`mergeRefs`</Term> function, roughly five lines, or the `react-merge-refs` package — and the call site collapses to:

```tsx
<input ref={mergeRefs([internalRef, ref])} {...props} />
```

Recognise the pattern, reach for the utility. One caveat to file away: a fully correct merge in React 19 also threads the cleanup return through each ref, and a hand-rolled five-liner usually skips that — the libraries handle it (the `react-best-merge-refs` package goes out of its way to get the React 19 cleanup path right). When a node needs more than one ref, prefer a maintained helper over hand-branching every time.

## `useImperativeHandle`: the rare escape valve

Everything so far hands the *DOM node* to the parent. Occasionally you want to hand it a curated set of *methods* instead — `ref.current.open()`, `ref.current.scrollToBottom()`, `ref.current.clear()` — rather than the raw node, either because the imperative surface is deliberately small (a `<Dialog>` that exposes `open()` and `close()` and nothing else) or because handing out the live node wholesale would be wrong. <Term definition="React hook that lets a component expose a custom object of methods on its ref, instead of a DOM node. Rarely the right reach.">`useImperativeHandle`</Term> builds that custom handle:

```tsx {11-16}
type FancyInputHandle = {
  focus: () => void;
  clear: () => void;
};

const FancyInput = (
  { ref, ...props }: ComponentProps<'input'> & { ref?: Ref<FancyInputHandle> },
) => {
  const inputRef = useRef<HTMLInputElement>(null);

  useImperativeHandle(ref, () => ({
    focus: () => inputRef.current?.focus(),
    clear: () => {
      if (inputRef.current) inputRef.current.value = '';
    },
  }));

  return <input ref={inputRef} {...props} />;
};
```

Notice the type change: the parent's `ref` is `Ref<FancyInputHandle>` now, not `Ref<HTMLInputElement>`. The parent calls `ref.current?.focus()` and gets your curated method, never the node.

:::caution
**Prefer exposing the node.** Reach for `useImperativeHandle` only when you genuinely need a *curated method surface* and handing out the DOM node is the wrong call. The common mistake is wrapping `{ focus }` in a handle when the parent could just receive the `<input>` ref and call `.current.focus()` itself — that's strictly more code for strictly less capability. Most components that reach for `useImperativeHandle` shouldn't.
:::

This lesson shows you the shape and the rule, nothing more. The API at depth — the dependency-array third argument, the patterns for a real custom hook — lands later when a custom hook earns it, in the hooks chapter.

## Refs through `asChild` and `Slot`

Time to close the loop the polymorphism lesson left open. Recall the canonical `<Button>`: it switches `Comp = asChild ? Slot : 'button'` and spreads `{...props}` onto `Comp`. Here's the thing — `{...props}` now carries `ref`, because `ref` is just a prop. And Radix <Term definition="Radix component that merges a parent's props, classes, handlers, and ref onto its single child element.">`Slot`</Term>, as part of the same merge that concatenates `className` and composes event handlers (the merge you learned in the polymorphism lesson), *forwards the parent's ref onto its single child element*. So the ref half of that merge comes for free. Watch what that buys you:

<CodeTooltips tooltips={{
  Slot: "Radix component that merges a parent's props, classes, handlers, and ref onto its single child element.",
  asChild: 'When true, the component renders as its child instead of its default element — and forwards everything, ref included, onto it.',
}}>
```tsx
const Button = ({ asChild, className, ...props }: ButtonProps) => {
  const Comp = asChild ? Slot : 'button';
  return <Comp className={cn(buttonVariants(), className)} {...props} />;
};
```
</CodeTooltips>

Now point a ref at it through `asChild` and watch where it lands:

```tsx
<Button asChild ref={buttonRef}>
  <Link href="/dashboard">Open dashboard</Link>
</Button>
```

`buttonRef` lands on the rendered `<a>` — the element `<Link>` produces — and you wrote *nothing* extra to make that happen. The ref-as-prop flows through `{...props}` into `Slot`, and `Slot` drops it on the child. No special-casing anywhere in the path. That's the entire payoff of removing the magic: because `ref` is an ordinary prop, it travels through `{...props}` spreads and through `Slot`'s merge exactly like `className` does. `asChild`, `Slot`, `cva`, and ref-as-prop stop being four separate tricks and read as one coherent contract.

This is the exact surface every shadcn primitive in the component-library chapter is built on — ref-as-prop plus `Slot` forwarding — which is why a ref works through their `asChild` prop without any of them doing anything special. You now know how to write the thing those components are made of.
