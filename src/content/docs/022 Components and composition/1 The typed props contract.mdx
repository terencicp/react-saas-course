Here are two ways to write the same Save button. Read both, and notice which one you'd rather inherit a year from now.

<CodeVariants>
  <CodeVariant label="Coffin">
    ```tsx
    <Button isPrimary isLarge withIcon iconName="save" iconPosition="left" loading={false}>
      Save
    </Button>
    ```
    **Every prop here is independent, and nothing constrains anything.** Six props, every one a boolean or a stringly-typed flag. Nothing stops you writing `isPrimary` and `isSecondary` both `true`, or pairing `iconName="save"` with `withIcon={false}`. And the list only grows — every design tweak adds another boolean to remember.
  </CodeVariant>

  <CodeVariant label="Contract">
    ```tsx
    <Button variant="primary" size="lg">
      Save
    </Button>
    ```
    **Two named axes, each with a fixed set of mutually-exclusive options.** The *appearance* lives on `variant`, the *scale* lives on `size`, and `primary` and `secondary` can't both be set because they're the same prop. There's nothing to misconfigure, and the editor autocompletes the allowed values.
  </CodeVariant>
</CodeVariants>

The first one is a coffin. Not because any single prop is wrong, but because the *shape* is wrong: every prop is an independent flag, the flags don't constrain each other, and the only way to learn what's legal is to read the source. The second is a contract — a small, named surface that says exactly what it accepts and rejects everything else before the code even runs.

Up to now you've styled *elements*: a `<button>`, a `<div>`, a `<section>` that you owned at the call site and dressed with Tailwind. This is the lesson where you stop styling tags and start writing the thing other code calls. A React component is a function that takes props and returns JSX, and its props are a contract — a promise about what it accepts. This lesson installs the form an experienced developer reaches for when writing that contract in 2026, and the four disciplines that keep it small: a typed props contract, variant unions instead of boolean piles, native-attribute inheritance, and an always-open `className`.

You already have every piece this rests on. You can write JSX. You can type a union and a discriminated union. You can merge class strings with `cn()`. This lesson is where those three meet on a function boundary. By the end you'll have written the canonical `<Button>` that the rest of the chapter extends — the exact component the next lessons layer `children`, `Slot`, and `cva` onto.

## A component is a typed function of props

Strip away every refinement and a React component is just this: a function that takes props and returns JSX. The 2026 form is an arrow function bound to `const`, a PascalCase name, props destructured right in the parameter, and no return type — inference supplies it.

At its absolute simplest, one prop, typed inline:

```tsx
const Greeting = ({ name }: { name: string }) => <p>Hello, {name}</p>;
```

That works, but you won't usually inline the type. Once a component has more than one prop, you pull the shape out into a named `type` called `Props`:

<AnnotatedCode lang="tsx" code={`
type Props = {
  label: string;
  tone: 'neutral' | 'success' | 'warning';
};

const Badge = ({ label, tone }: Props) => (
  <span className={cn('rounded-full px-2 py-0.5 text-xs', toneClasses[tone])}>
    {label}
  </span>
);
`}>
  <AnnotatedStep meta="{6}" color="blue">
    A component is an arrow function bound to `const`, named in PascalCase. That casing is load-bearing, not style: write `badge` lowercase and JSX treats `<badge>` as a literal DOM tag and silently drops every prop you pass it. PascalCase is how React knows this is your function, not an HTML element.
  </AnnotatedStep>

  <AnnotatedStep meta={`"{ label, tone }: Props"`} color="blue">
    Props arrive as one object. Destructure the names you want directly in the parameter and annotate that parameter with your `Props` type. This is the form you'll write for every component in the course.
  </AnnotatedStep>

  <AnnotatedStep meta="{1-4}" color="blue">
    The convention is `type Props` for component props, never `interface` — the same call you met when you first weighed `type` against `interface` (`type` by default, `interface` only for declaration merging, which props never do). Note there's no return type on `Badge`: inference already knows it returns a `JSX.Element`, and annotating it is noise.
  </AnnotatedStep>
</AnnotatedCode>

Two details worth saying out loud. First, that PascalCase rule is not a style preference — it changes behavior. A lowercase component name makes React render a literal DOM element of that name and throw your props away with no error. Name components `Badge`, `SubmitButton`, `PriceTag`; never `badge`.

Second, you don't annotate the return type. The body returns JSX, TypeScript infers the <Term definition="The type a component returns — a single rendered React element. You almost never write this type by hand; inference supplies it.">`JSX.Element`</Term> for you, and writing `: JSX.Element` after the parameter list just adds noise the compiler already knows. Annotate the inputs; let the output infer.

## Typing props: strings, unions, and function signatures

The `Props` type is where the contract gets written, so it's worth knowing the vocabulary it's built from. You already know TypeScript, so this is application, not instruction — four shapes cover almost every prop you'll ever type.

<CodeTooltips tooltips={{
  "'primary' | 'secondary' | 'destructive'": 'A finite set of allowed strings. TypeScript rejects any value that isn\'t one of these three.',
  'icon?': 'Optional prop — the value may be undefined. The caller can leave it out.',
  MouseEvent: "React's synthetic event type — not the DOM MouseEvent global. Import it from react.",
}}>
```tsx
type Props = {
  title: string;
  count: number;
  variant: 'primary' | 'secondary' | 'destructive';
  icon?: string;
  onClick: (event: MouseEvent<HTMLButtonElement>) => void;
};
```
</CodeTooltips>

Three of those are familiar on sight: a `string`, a `number`, an optional prop marked with `?`. The fourth carries the one genuinely new piece.

The **union of string literals** — `variant: 'primary' | 'secondary' | 'destructive'` — is the workhorse of component props, and it's the shape the next section pays off in full. It says "this prop is exactly one of these three strings," and the editor will autocomplete them and reject anything else. Any time a prop has a small, fixed set of legal values, this is the type you reach for.

The **function prop** is typed by its signature. `onClick: (event: MouseEvent<HTMLButtonElement>) => void` says the component hands its caller a click event and expects nothing back. The new detail is `MouseEvent`: it's React's *synthetic* event type, imported from `react`, not the `MouseEvent` that lives on the DOM global. The generic argument matters — `MouseEvent<HTMLButtonElement>` types `event.currentTarget` as a button element, so reading `event.currentTarget.disabled` type-checks. Get the element wrong and that access breaks.

One reflex to carry onto this surface: **never type a prop `any`**. It's the same instinct you already have — reach for the specific shape, or `unknown` and narrow at the boundary. `any` on a prop throws away the entire contract for that field.

## Defaults belong at the destructure

One idea, sharply: default values go in the parameter destructure, right next to the prop they default.

```tsx
const Button = ({ variant = 'primary', size = 'md' }: Props) => {
  // ...
};
```

Two things an experienced developer knows here. First, `defaultProps` — the old object you'd hang on a component to supply defaults — is **gone** for function components; React 19 removed it. You won't see it in new code. Defaulting at the destructure is the only form now.

Second, this beats a falsy check in the body. You might be tempted to write `const v = variant || 'primary'` instead, but that's the `||` trap on a new surface: `||` fires on *any* falsy value, so a legitimate falsy prop gets silently overridden, and the default ends up scattered away from the signature where you can't see it. The destructure default fires only on `undefined` — pass nothing, or pass `size={undefined}`, and you get `'md'`; pass any real value and it stands. That `undefined`-only behavior is exactly why the default sits cleanly beside the prop name, and why a union prop never has a falsy member to trip over.

## From boolean piles to a variant union

This is the move the first half of the lesson has been building toward, so give it a moment. Go back to the coffin's appearance flags — `isPrimary`, `isDestructive`, `isGhost` — and collapse them into a single prop:

<CodeVariants>
  <CodeVariant label="Three booleans">
    <div data-mark-color="red">

    ```tsx {2-4}
    type Props = {
      isPrimary?: boolean;
      isDestructive?: boolean;
      isGhost?: boolean;
    };
    ```

    </div>
    **Eight states, three of them real.** Three booleans encode 2³ = 8 combinations, and most are nonsense — `isPrimary` and `isGhost` both `true`, or all three `false`. The type permits every one of them.
  </CodeVariant>

  <CodeVariant label="One union">
    <div data-mark-color="green">

    ```tsx {2}
    type Props = {
      variant?: 'primary' | 'destructive' | 'ghost';
    };
    ```

    </div>
    **Three states, exactly the three that are real.** One axis, one prop. The variants are mutually exclusive by construction — you cannot set two at once — and there's nothing illegal left to represent.
  </CodeVariant>
</CodeVariants>

Walk the reasoning, because this is a judgment call you'll make over and over.

**Mutually exclusive by construction.** Three booleans give you 2³ = 8 combinations, and five of them are meaningless — anything with two flags on, plus the all-off state. The union gives you exactly three: the appearance is one value, so two appearances at once isn't a bug you have to *remember* to avoid, it's a state the type can't even spell. This is the make-illegal-states-unrepresentable principle you've seen before, now living on a props boundary.

**Exhaustive.** Because the variant is a closed union, a `switch` or lookup over it can be checked for completeness. Add a `'link'` variant later and TypeScript surfaces every place that has to handle the new case. Three independent booleans give you no such safety net — nothing connects them, so nothing can be checked.

**One axis, one prop.** The same logic applies to scale: `size: 'sm' | 'md' | 'lg'` over `isSmall` / `isLarge`. Whenever a set of options is mutually exclusive — one appearance, one size, one alignment — they belong on one union prop, not spread across booleans.

There's a seam here worth naming. These unions are the *source* of the variant table that `class-variance-authority` reads later in this chapter — the `variant` and `size` props you're defining now become the keys `cva` maps to class strings. You're building the input that tool will consume; keep the shape clean.

Here's the trap beginners fall into once this clicks: turning *everything* into a union. The rule isn't "always collapse." A union is for **one axis with mutually-exclusive options**. A boolean is for an **independent on/off flag** that has nothing to do with any other prop — `disabled`, `loading`, `fullWidth`. Those are orthogonal switches; forcing them into a union would be the same mistake in the other direction. Train the judgment with the next exercise.

## Inheriting native attributes with ComponentProps

Now the centerpiece. Your `<Button>` wraps a native `<button>`, and a native `<button>` accepts a *lot*: `disabled`, `type`, `aria-label`, `onClick`, `onFocus`, `form`, `name`, every standard attribute and every event handler. If the consumer can't pass those through your component to the real element underneath, your `<Button>` is a downgrade from the raw tag. And hand-typing each one in `Props` is both endless and always one attribute behind.

The reach is one type and one spread, and they're really a single idea.

First, reach for a React utility type called <Term definition={"ComponentProps<'button'> is every prop the JSX <button> accepts — attributes, event handlers, and ref.\nComponentProps<typeof X> is every prop the component X accepts."}>`ComponentProps`</Term>. It pulls in *everything* the JSX `<button>` accepts, and you intersect it with the props your component adds on top:

```tsx
type Props = ComponentProps<'button'> & {
  variant?: 'primary' | 'destructive' | 'ghost';
  size?: 'sm' | 'md' | 'lg';
};
```

`ComponentProps<'button'>` is the type of every attribute, every event handler, and — in React 19 — the `ref` too, all under one alias. You never list them; the alias *is* the list, and it stays correct as the platform evolves.

Then, in the body, destructure the props your component actually *consumes*, and spread the rest straight onto the element:

<AnnotatedCode lang="tsx" code={`
type Props = ComponentProps<'button'> & {
  variant?: 'primary' | 'destructive' | 'ghost';
  size?: 'sm' | 'md' | 'lg';
};

const Button = ({
  variant = 'primary',
  size = 'md',
  className,
  ...rest
}: Props) => (
  <button className={cn(buttonClasses({ variant, size }), className)} {...rest} />
);
`}>
  <AnnotatedStep meta="{1-4}" color="green">
    The intersection pulls in every native button prop and adds your two. The component now accepts everything a real `<button>` does, plus `variant` and `size`.
  </AnnotatedStep>

  <AnnotatedStep meta="{9}" color="green">
    `className` comes *in* through `ComponentProps<'button'>`, so you must pull it out by name. Leave it inside `...rest` and you can't merge it deliberately — it would land on the element raw and fight your own classes.
  </AnnotatedStep>

  <AnnotatedStep meta={`"cn(buttonClasses({ variant, size }), className)"`} color="green">
    Compute your component's classes first, then pass the caller's `className` as the *last* argument so it wins on conflicts. Same `cn()`-className-last rule you already use, applied on the new props surface.
  </AnnotatedStep>

  <AnnotatedStep meta={`"{...rest}"`} color="green">
    `...rest` is everything you *didn't* destructure — `disabled`, `type`, `onClick`, `aria-label`, all of it — spread directly onto the native element. The consumer's native attributes reach the real button with zero per-attribute typing.
  </AnnotatedStep>
</AnnotatedCode>

(`buttonClasses` here is a stand-in for "the function that turns variant + size into a class string" — that's what `cva` becomes in a later lesson. Don't build it; just reference it.)

The `className`-plus-`...rest` discipline at the heart of that block has one ordering rule you have to get right, and it's worth seeing *why* rather than memorizing it. Because `ComponentProps<'button'>` *includes* `className`, the caller's `className` is sitting inside `...rest` unless you pull it out. If you forget to destructure it and just write `<button className={cn(buttonClasses(...))} {...rest} />`, the spread re-injects the raw caller `className` and either clobbers your computed classes or sets the attribute twice. The fix is structural: destructure `className` out of the rest, compute `cn(yourClasses, className)`, and set that on the element. Pull it out, merge it last, spread the remainder. That ordering is the whole game.

That <Term definition="Rest destructuring — gathers every prop you didn't name into one object you can then spread onto another element.">`...rest` spread</Term> is what makes the component a thin, faithful wrapper instead of a walled garden. Anything the consumer needs from the native button, they get, without you anticipating it.

### ComponentProps vs the older forms

You'll meet two other ways to type native button props in code written before 2025 or in older shadcn components. Recognize them, know why `ComponentProps` is the default now.

<CodeVariants>
  <CodeVariant label="ComponentProps">
    ```tsx
    type Props = ComponentProps<'button'> & {
      variant?: 'primary' | 'destructive' | 'ghost';
    };
    ```
    **The 2026 default.** One alias, no element type to name, and it pulls *everything* the JSX `<button>` accepts — attributes, handlers, and `ref`. This is what you write.
  </CodeVariant>

  <CodeVariant label="ButtonHTMLAttributes">
    ```tsx
    type Props = ButtonHTMLAttributes<HTMLButtonElement> & {
      variant?: 'primary' | 'destructive' | 'ghost';
    };
    ```
    **The older per-element form.** It works, but it's more verbose and you name the element twice — once in the type, once in the generic. No reason to reach for it over `ComponentProps`.
  </CodeVariant>

  <CodeVariant label="HTMLAttributes">
    <div data-mark-color="red">

    ```tsx
    type Props = HTMLAttributes<HTMLButtonElement> & {
      variant?: 'primary' | 'destructive' | 'ghost';
    };
    ```

    </div>
    **The base attributes only — a subtle bug.** `HTMLAttributes` carries the attributes *every* element has, but **not** button-specific ones like `type` or `disabled`. Reach for this expecting button props and they silently won't be there.
  </CodeVariant>
</CodeVariants>

The takeaway is short: `ComponentProps<'button'>` for the wrapped-element props, every time. The other two are recognition targets, and the last one is a trap to avoid.

## Typing wrappers and component references

Two type helpers you'll reach for less often, worth recognizing now. They're duals — one goes from a component to its props, the other from props to a component — and they're easy to mix up, so it's worth seeing them side by side.

The first types a **wrapper**. Say you build a `<ConfirmButton>` that's just your `<Button>` pre-set to the destructive variant, plus a `confirmLabel`. You don't want to restate every variant and native attribute the inner Button already exposes — you want to *inherit* them. `ComponentProps<typeof Button>` does exactly that: it extracts the full props type of an existing component.

The second types a **component reference** — the component itself, not its props. Where `ComponentProps<typeof X>` pulls *props out of* a component, `ComponentType<P>` is the type *of a component* that accepts props `P`. You reach for it when a component is a value you store or pass: a registry that maps names to components, or a prop that *is* a component — like accepting a Lucide icon to render inside a button without instantiating it at the call site.

<CodeVariants>
  <CodeVariant label="Wrap a component">
    ```tsx
    type Props = ComponentProps<typeof Button> & {
      confirmLabel: string;
    };

    const ConfirmButton = ({ confirmLabel, ...rest }: Props) => (
      <Button variant="destructive" {...rest}>
        {confirmLabel}
      </Button>
    );
    ```
    **`ComponentProps<typeof Button>` inherits the inner component's whole surface.** Every variant, size, and native attribute Button exposes flows through — you only declare what `ConfirmButton` adds.
  </CodeVariant>

  <CodeVariant label="Accept a component">
    ```tsx
    type Props = {
      icon: ComponentType<{ className?: string }>;
      label: string;
    };

    const IconButton = ({ icon: Icon, label }: Props) => (
      <button>
        <Icon className="size-4" />
        {label}
      </button>
    );
    ```
    **`ComponentType<P>` is the type of a component itself.** You pass the icon component in (not an element), rename it to a PascalCase local, and render it where you choose.
  </CodeVariant>
</CodeVariants>

## className is the styling contract, not an option

This one is short, but it's a discipline, not a mechanic — which is why it gets its own heading. The rule:

> Every component the project ships accepts an optional `className` and merges it `cn(..., className)` at its outermost element.

Your `<Button>` already does this; here it's named as a *rule* so you apply it everywhere, not just to buttons.

```tsx
<button className={cn(buttonClasses({ variant, size }), className)} {...rest} />
```

The reasoning is about reuse. A component that *doesn't* accept `className` looks fine right up until the first consumer needs one different margin — and then they're stuck forking the component or smothering it in a wrapper `<div>` just to nudge it. Accepting `className` and merging it last (so the caller wins on conflicts, exactly as in the `cn()` rule you already know) costs one line and keeps the component usable in every layout it lands in. Never lock styling down. The escape hatch is the feature.

You don't need a new exercise for this — the test in the previous exercise already guards that a caller's `className` survives onto the element. That assertion *is* this rule, enforced.

## Discriminated unions for mutually-exclusive props

The variant union handled "pick one of N appearances." There's a sharper tool for a different shape: when two props must *never* both be set, and one of them must be.

The canonical case is a thing that's either a link or a button. A link needs an `href`; a button needs an `onClick`. Setting both is a bug, and setting neither is a different bug. Two optionals plus a runtime guard *describes* the rule but doesn't *enforce* it — nothing stops a caller passing both. A discriminated union makes the illegal combination a **compile error**:

<CodeVariants>
  <CodeVariant label="Two optionals">
    <div data-mark-color="red">

    ```tsx {2-3}
    type Props = {
      href?: string;
      onClick?: () => void;
    };
    ```

    </div>
    **Describes the rule, enforces nothing.** Both props optional means a caller can pass both, or pass neither, and the body has to guard at runtime for shapes the type already let through.
  </CodeVariant>

  <CodeVariant label="Discriminated union">
    <div data-mark-color="green">

    ```tsx {2-3}
    type Props =
      | { href: string; onClick?: never }
      | { onClick: () => void; href?: never };
    ```

    </div>
    **Makes the illegal combo a compile error.** Each branch permits exactly one of the two and forbids the other with `?: never`. Passing both `href` and `onClick` matches neither branch — TypeScript rejects the call before it runs. So does passing neither.
  </CodeVariant>
</CodeVariants>

Be honest about the cost. Discriminated-union props need a discriminant or careful narrowing in the body to tell the branches apart, and that's friction every time you read the component. So reach for this only when the mutual exclusion is *real* — most props are happily independent and don't need it. The link-or-button case earns it; a typical button's props don't.

This is also the *typed* answer to the button-vs-link question. The next lesson on polymorphic components solves the same problem a different way — by composition, letting one component *become* a link or a button — and you'll weigh the two. For now, the discriminated union is the tool when the exclusion lives in the types.

That anti-pattern from the start of the chapter — shipping both `onClick` and `href` on one flat button API and sorting it out at runtime — is exactly the shape this fixes. Don't ship both on a flat API; model the exclusion, or pick one.

  Jolly Coding builds a discriminated union from a pile of optionals and lands it on React props and button variants — including the narrowing cost named above (7 min).
</VideoCallout>

Make the type do that work. The two-optionals `Props` below lets every call site through — including the two that should be bugs. Each bad call carries an `@ts-expect-error` directive, and since the flat shape *permits* both lines, those directives are currently unused — that's the red in the diagnostics panel. Rewrite `Props` as a discriminated union and the illegal calls finally error, satisfying the directives, while the two legitimate shapes still type-check.

<TypeCoding
  instructions={`Rewrite \`Props\` so that passing both \`href\` and \`onClick\` — or neither — is a type error, while a lone \`href\` or a lone \`onClick\` still type-checks. Give each branch one required prop and forbid the other with \`?: never\`. When the union is right, the two \`@ts-expect-error\` directives stop being unused and the diagnostics clear.`}
  starter={`type Props = {
  href?: string;
  onClick?: () => void;
};

// A link: just an href. This must stay legal.
const asLink: Props = { href: '/invoices' };

// A button: just an onClick. This must stay legal.
const asButton: Props = { onClick: () => {} };

// @ts-expect-error — a link AND a button at once is a bug
const both: Props = { href: '/invoices', onClick: () => {} };

// @ts-expect-error — neither a link NOR a button is also a bug
const neither: Props = {};
`}
/>

<details>
<summary>Reveal the discriminated union</summary>

```ts
type Props =
  | { href: string; onClick?: never }
  | { onClick: () => void; href?: never };
```

Each branch makes exactly one of the two props required and forbids the other with `?: never`. `asLink` matches the first branch — `href` is present, `onClick` is absent, which satisfies `onClick?: never`. `asButton` matches the second the same way. `both` matches *neither* branch: its `onClick` violates the first branch's `onClick?: never`, and its `href` violates the second's `href?: never` — so TypeScript rejects it, and the `@ts-expect-error` above it is now satisfied. `neither` (`{}`) also matches no branch, because each branch has one *required* prop that an empty object can't supply. Both directives go quiet, and the diagnostics panel is clear.

</details>

## Generic components, lightly

The last technical tool, scoped tight: you'll write this rarely and recognize it always. Sometimes a component has to work for *any* item type — a `<List>` that doesn't care whether it's rendering invoices, users, or tags. That's a generic component.

It takes the data and a function that renders one item:

<CodeTooltips tooltips={{
  'extends { id: string }': 'Constrains the type parameter to any type with an id, and resolves the JSX-vs-generic ambiguity: in a .tsx file a bare <Item> reads as a JSX tag, so the extends clause (or a trailing comma) tells the compiler it\'s a type parameter.',
  ReactNode: 'Anything React can render — JSX, strings, numbers, arrays, null. The wide return type; taught in depth in the next lesson.',
}}>
```tsx {6}
type ListProps<Item> = {
  items: Item[];
  render: (item: Item) => ReactNode;
};

const List = <Item extends { id: string }>({ items, render }: ListProps<Item>) => (
  <ul>
    {items.map((item) => (
      <li key={item.id}>{render(item)}</li>
    ))}
  </ul>
);

const InvoiceList = ({ invoices }: { invoices: Invoice[] }) => (
  <List items={invoices} render={(invoice) => <span>{invoice.number}</span>} />
);
```
</CodeTooltips>

There are two things in that signature worth a beat. The first is the constraint: `<Item extends { id: string }>` says the list works for any type *that has an id*, which is what lets you write `key={item.id}` — a stable key tied to each row's identity, the same `<T extends ...>` instinct you reach for with any constrained generic. The second is what that opening `<` would mean *without* the `extends` clause. In a `.tsx` file, a bare `<Item>` reads as an opening JSX tag — the generic and JSX syntaxes collide. The `extends` clause resolves it (and when you don't need a constraint, a trailing comma — `<Item,>` — does the same job); it's the detail that trips people who know TS generics cold but have never written one inside `.tsx`.

The payoff is at the call site. Look at `InvoiceList`: it passes `invoices`, and the component *infers* that `Item` is `Invoice` from the array — so inside `render`, the `invoice` parameter is fully typed as an `Invoice` with zero annotation at the call site. Full type-safety, nothing to declare. That's the whole reason the generic exists.

Keep this in the recognition drawer. It's a confidence builder — the type system can do this — far more than a daily tool.

## One component per file (and the one exception)

Close on where this lives. The conventions are short:

- **One component per file**, by default. The filename is kebab-cased and matches the export: `button.tsx` exports `Button`, `confirm-button.tsx` exports `ConfirmButton`.
- **Internal sub-components** used only by that one file live in the same file. Don't split out a helper nobody else imports.
- **The exception:** a tightly-coupled compound set ships as one file that exports the whole surface. `<Card>`, `<CardHeader>`, `<CardContent>`, `<CardFooter>` all come from `card.tsx`, because you never use one without the others.

<FileTree>
- components/
  - ui/
    - button.tsx exports `Button` — *the default:* one component per file
    - input.tsx exports `Input`
    - **card.tsx** exports `Card`, `CardHeader`, `CardContent`, `CardFooter` — *the exception:* a compound set in one file
</FileTree>

That compound pattern is the canonical shadcn shape, and it's exactly what the next lesson teaches — *how* a set of components composes into one. Here it's just the file-layout exception to recognize.

## What you'll see in old code, and won't write

You now have the full shape: a component is a typed function of props, with a small variant-union API, native attributes inherited through `ComponentProps`, sensible defaults at the destructure, and a `className` that stays open. A few patterns surround this in older codebases — know them on sight, don't reach for them.

:::caution[Recognize, don't write]
- **`forwardRef`** — the old way to let a component accept a `ref`. React 19 makes `ref` a regular prop, covered in a later lesson; `forwardRef` is legacy.
- **`PropTypes`** — runtime prop validation, retired. TypeScript is the contract now; it checks before the code runs, not after.
- **Higher-order components (HOCs)** — a wrapping pattern that hooks and composition retired. You'll see `withSomething(Component)`; you won't write it.
- **Class components** — out of scope for this entire course. Function components only.
:::

The `<Button>` you built is the literal foundation the next lessons extend — composition and `children` next, then `Slot` and `cva` for polymorphic variants. The mental model carries all the way through: a React component is a typed function from props to JSX, its props are a contract, and the contract is the design. Get the shape right and everything else in this chapter is a refinement of it.
