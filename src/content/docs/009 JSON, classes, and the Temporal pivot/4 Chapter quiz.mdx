  <MultipleChoice src="9.1">
    A Server Action receives this JSON body and reads `data.amount` to charge a customer:

    ```ts
    const raw = await req.text();
    const data = JSON.parse(raw);
    return chargeInvoice(data.invoiceId, data.amount);
    ```

    What is wrong with this shape?

    <McqChoice correct>
      `JSON.parse` returns `any` by convention; the value must be typed `unknown` and narrowed by a Zod schema before any field is read.
    </McqChoice>
    <McqChoice>
      The `SyntaxError` from `JSON.parse` isn't caught at the call site â€” wrap the parse in a `try`/`catch` and return a 400 from inside the catch.
    </McqChoice>
    <McqChoice>
      `req.text()` is the wrong method â€” use `req.json()` so the runtime parses and types the body for you.
    </McqChoice>

    <McqWhy>The bug class is "I read the response and trusted the shape." `JSON.parse` hands back whatever the wire sent; the TypeScript signature says `any`, the discipline says `unknown`. Reading `data.invoiceId` and `data.amount` off the raw parse lets a malformed webhook poison the database. The cure is the two-step seam: `JSON.parse` to `unknown`, then `schema.parse` to the domain type â€” usually behind the `parseJson(raw, schema)` helper. The `SyntaxError` catch belongs once at the transport-layer wrapper, not at every call site. And `req.json()` returns `Promise<any>` â€” same discipline, different syntax.</McqWhy>
  </MultipleChoice>

    Which of these `JSON.stringify` outputs are correct? Select all that apply.

    <McqChoice correct>
      `JSON.stringify({ a: 1, b: undefined })` â†’ `'{"a":1}'`
    </McqChoice>
    <McqChoice correct>
      `JSON.stringify([1, undefined, 3])` â†’ `'[1,null,3]'`
    </McqChoice>
    <McqChoice>
      `JSON.stringify({ id: 42n })` â†’ `'{"id":"42"}'`
    </McqChoice>
    <McqChoice correct>
      `JSON.stringify({ ratio: NaN })` â†’ `'{"ratio":null}'`
    </McqChoice>

    <McqWhy>Three of the four serialization holes show up here. Object properties with `undefined` are *dropped*; array slots become `null` (positions matter, so the value can't simply vanish). `NaN`, `Infinity`, and `-Infinity` all serialize as `null` â€” indistinguishable from a real `null` on the receiver. The `BigInt` case is the odd one out: `JSON.stringify` *throws* `TypeError` on a `bigint`, it doesn't coerce to a string. The senior reach is to model 64-bit IDs as strings in the schema so no `bigint` ever reaches the serializer.</McqWhy>

  <MultipleChoice src="9.2">
    A teammate writes this class to hold a Stripe API key:

    ```ts
    class SecretHolder {
      private secret = 'sk_live_xxx';
    }
    ```

    What's the senior critique?

    <McqChoice correct>
      TypeScript's `private` is erased at compile time â€” the field is reachable at runtime via `(holder as any)['secret']` or reflection. Use `#secret` for runtime-enforced privacy.
    </McqChoice>
    <McqChoice>
      The field should be `readonly` so it can't be reassigned after construction.
    </McqChoice>
    <McqChoice>
      Hardcoded secrets belong in environment variables â€” move the value to `process.env.STRIPE_KEY` and the class is fine.
    </McqChoice>

    <McqWhy>The point of the class is privacy, and TS `private` doesn't deliver it. The keyword is a compile-time hint that's erased before the engine sees the code; bracket access, `Reflect.get`, `Object.entries`, and `JSON.stringify` all reach the field at runtime. `#secret` is enforced by the JavaScript runtime â€” invisible to bracket access, skipped by `JSON.stringify`, scoped per-class. `readonly` is a different axis (mutation, not visibility) and the env-var move is true but orthogonal to the privacy bug. The fix is `#secret`.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="9.3">
    A teammate adds this to `lib/billing/stripe.ts`:

    ```ts
    import { Temporal } from 'temporal-polyfill';

    export const createdAt = (sub: Stripe.Subscription): Temporal.Instant =>
      Temporal.Instant.fromEpochMilliseconds(sub.created * 1000);
    ```

    What's the senior critique?

    <McqChoice correct>
      Import `Temporal` from `@/lib/temporal`, never from `temporal-polyfill` directly. Two `Temporal` instances in the same process break `instanceof` and cross-instance `from()` silently.
    </McqChoice>
    <McqChoice>
      `Temporal.Instant.fromEpochMilliseconds` is the wrong factory for Unix seconds â€” call `Temporal.Instant.fromEpochSeconds(sub.created)` instead.
    </McqChoice>
    <McqChoice>
      The conversion belongs in `lib/temporal.ts` so the SDK adapter never touches `Temporal` at all.
    </McqChoice>

    <McqWhy>One seam, one import path. `lib/temporal.ts` is the only file in the codebase that imports the polyfill; every other file imports `Temporal` from there. Mixing the polyfill `Temporal` with the native `Temporal` (or two polyfill copies) in one process produces values that look identical from the outside but fail `instanceof` and `Temporal.X.from()` cross-checks â€” silent breakage. The other options are wrong on facts: there is no `fromEpochSeconds` factory (Stripe's seconds get multiplied by 1000 at the seam), and the conversion legitimately lives at the SDK adapter â€” that's exactly what "Date stays at the seam" means.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="9.3">
    True or false: `new Date('2026-05-15')` and `new Date('2026-05-15 00:00')` produce the same `Date` on every server.

    <McqChoice correct>
      False â€” the ISO date-only form parses as UTC; the space-separated form parses as the runtime's *local* timezone. The two strings produce the same instant only when the server is on UTC.
    </McqChoice>
    <McqChoice>
      True â€” both strings describe midnight on May 15, 2026; the parse normalizes them to the same millisecond count.
    </McqChoice>

    <McqWhy>A single character â€” the space versus the `T` â€” flips the timezone semantics. `'2026-05-15'` is the ISO date-only form, specified to parse as UTC. `'2026-05-15 00:00'` is the space-separated form, parsed as local time then converted back to UTC for storage. Code that ingests date strings from third parties passes its tests on a UTC server (Vercel, CI) and breaks the moment it runs in a non-UTC environment. This is the "format-determined timezone parse" bug â€” one of the eight structural flaws Temporal exists to retire.</McqWhy>
