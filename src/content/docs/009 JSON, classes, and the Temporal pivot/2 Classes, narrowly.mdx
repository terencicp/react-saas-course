A new contributor opens a pull request. The diff adds `class UserService` with five methods — two of them `static`, one of them quietly losing `this` inside a `.map` callback. The PR is well-meaning; the muscle memory comes from Java, C#, or Python, where a service class is the obvious shape. The reviewer's first thought is not "this needs tweaking." It's *do we need a class here at all?*

The answer in a 2026 SaaS codebase is almost always no. Records and functions are the default. **A `class` declaration is a carve-out, and it earns its weight at exactly three triggers:** a custom `Error` subclass (familiar from the previous chapter), a third-party SDK adapter wrapper, and the rare stateful domain object. Everything else is a function over a typed record. This lesson installs the trigger filter first, then the minimum surface to ship when a trigger fires, and finally the refusal list — the primitives your prior-language muscle memory will pull you toward but that this stack doesn't reach for.

## When a class earns its weight

The decision walks one question at a time. Before any syntax, before the choice of `readonly` versus arrow-field, you check whether the code in front of you matches one of three concrete shapes. If none of them fit, the answer is a module of functions over a typed record — the same `getUser(id)` / `User` shape the course has been writing since the modules chapter.

**Trigger 1 — Custom `Error` subclasses.** You met this shape in the previous chapter on errors: `class ValidationError extends Error` with a literal `name` discriminant the `catch` reads to route the failure. The reason it earns a class is *the runtime contract*: `throw` expects an `Error`, every framework boundary catches `Error`, and the literal `name` field is what survives across realms when `instanceof` doesn't. The platform dictates the shape; you comply. Refer back to the previous chapter for the canonical `ValidationError` body — no re-derivation here.

**Trigger 2 — Third-party SDK adapter wrappers.** Stripe ships its surface as a `Stripe` class. Better Auth, Resend, the Postgres driver — many SDKs do the same. The application wraps the vendor class in a thin one of its own (`class BillingClient`, `class EmailSender`) for three reasons stacked on top of each other: the wrapper *hides the vendor type* so it doesn't leak into business logic; it *centralizes construction* (one `apiKey`, one `apiVersion`, one set of defaults); and it *exposes application-shaped methods* (`createCheckoutSession(input)`, not `stripe.checkout.sessions.create({ ... })`). The trigger here is *carrying SDK state across calls behind an application-shaped surface*.

**Trigger 3 — The rare stateful domain object.** A long-lived in-memory entity with invariants that only methods can enforce. The textbook example is a `Cart` that totals itself, or a token-bucket rate limiter that decrements and refills. Be deliberately suspicious of this trigger: on this stack, *state lives in Postgres, in Zustand, or in React state*. Most "stateful objects" you'll be tempted to write are domain records the database owns. This trigger fires only for genuine in-memory aggregates whose invariants would otherwise become discipline rather than guarantee.

That's the whole list. Walk the filter once and the recommendation lands.

## Why records and functions are the default

Two failure modes drove the 2026 shift away from class-everything. Both bite at the boundary; both vanish when the same code is a function over a record.

**Failure mode one — `this` binding.** A method passed as a callback loses `this`. The compiler doesn't tell you. The runtime tells you when something deep in a `.map` reaches for `this.normalize` and gets `undefined`.

```ts
const users = await listUsers();
const normalized = users.map(svc.normalize);
```

If `svc.normalize` reads any field through `this`, the call site above silently strips the binding — `Array.prototype.map` invokes the function with no receiver. The class fix is to declare `normalize` as an arrow-field, but arrow fields break inheritance and pay one closure per instance instead of one method on the prototype. The function-over-record alternative has *no `this` to lose*: `users.map(normalizeUser)` closes over its arguments and reads what it needs from them.

**Failure mode two — class instances don't cross the wire.** The previous lesson on JSON named this in passing: `JSON.stringify` writes a class instance's own enumerable properties only, dropping methods, getters, and `#private` fields entirely. Every wire boundary the course will teach — Server Actions, Server Components, RSC payloads, route handler responses — serializes its outputs to records. A class instance crossing the network boundary arrives as a *record-shaped imitation* of itself, methods gone, privacy gone, identity gone. Records and functions match that shape natively; classes fight it.

:::tip
The record-plus-module-function shape is *also* what makes the server/client split trivial. The function lives in `lib/users.ts`; the record type lives next to the schema; nothing in the file needs `'use client'` until React state enters. The wire boundary inherits the same `unknown`-then-Zod discipline the previous lesson installed.
:::

So the default isn't anti-class. It's pro-record. The class triggers earn their weight only when the platform itself dictates a different shape.

## The minimum class surface

When a trigger does fire, you ship only these primitives. The discipline here is keeping the surface small enough to hold in working memory — five primitives, in a fixed order, every time.

1. **`constructor` with one options-object parameter.** Two positional arguments maximum, then options object. The constructor body is assignment only; no method calls, no validation chains, no side effects. Anything more belongs in a `static` factory.
2. **`readonly` on every field that isn't mutated after construction.** Default to `readonly` everywhere unless mutation *is* the lesson — and even then, prefer the smallest mutable surface. `readonly` is compile-time field immutability; the field's *contents* can still mutate (a `readonly` `Map` is still `.set()`-able), but the binding itself can't be reassigned.
3. **`#private` (<Term definition={"The `#name` field syntax — runtime-enforced visibility. The field can't be read via bracket access, doesn't appear in `Object.keys`, and is skipped by `JSON.stringify`. Per-class scope; two unrelated classes can both declare `#secret` without collision."}>hash-private</Term>), not TypeScript's `private` keyword.** This is the load-bearing distinction. TS `private` is erased at compile time and the field is reachable via bracket notation at runtime. `#private` is enforced by the JavaScript runtime, can't be read by bracket access, can't be reached by reflection, and survives the compile. We'll come back to this in its own section.
4. **<Term definition={"A class field whose value is an arrow function (`method = (x) => { ... }`). `this` is bound to the instance at construction, so the method survives being passed as a callback. Costs one closure per instance instead of one method on the prototype."}>Arrow-field methods</Term> only when detachment matters.** Default to regular method syntax — one copy on the prototype, no per-instance allocation. Reach for the arrow-field shape (`method = (input) => { ... }`) only when the method is passed as a callback, stored in a `Map`, or attached as an event listener — situations where the `this` binding would otherwise be lost.
5. **`static` factory methods for non-trivial construction.** When the class has multiple construction paths (`fromJson`, `fromRow`, `fromEnv`), make them `static` methods returning instances. The constructor stays one shape; the call site reads as `BillingClient.fromEnv(env)`, not `new BillingClient(env.STRIPE_KEY, '2026-04-22.dahlia', { ... })`.

These five primitives, named in order, applied to the canonical Stripe-adapter example, look like this.

One more piece outside the surface itself. When the wrapper has no per-request state — and most don't — export a configured singleton from the module:

```ts
export const billing = BillingClient.fromEnv(env);
```

Every importer reads the same instance; construction happens once at module load. The <Term definition={"A single configured instance exported from a module — `export const billing = ...`. The rest of the app imports the instance, never the class. Per-request instances only when the wrapper carries request-scoped state."}>module-level singleton</Term> is the default. Per-request instances are the rare reach for wrappers that carry request-scoped state (a per-tenant API key, a per-request idempotency context). One instance, exported, imported everywhere.

## What the surface refuses

The list above is what the class earns. This list is what it refuses — primitives your prior-language muscle memory will reach for, and the one-line reason each one stays out.

1. **Class inheritance hierarchies beyond `extends Error`.** Composition over inheritance, always. The Liskov-substitution traps and the brittle-base-class problem cost more than they buy on this stack. When you need polymorphism, reach for a discriminated union (`{ kind: 'circle'; radius: number } | { kind: 'square'; side: number }`); when you need behavior reuse, compose functions over a record.
2. **`abstract class`.** A TypeScript-only marker with no runtime guard — the `abstract` keyword is erased at compile, so an `abstract` class is just a class that the compiler refuses to instantiate directly. Reach for a `type` and a discriminated union instead.
3. **Mixins.** `Object.assign(C.prototype, M)` gymnastics, three layers deep, with stack traces nobody can read. The disease they cure is the inheritance limitation; the real cure is composition.
4. **Decorators.** TC39 Stage 3 since TypeScript 5.0 in 2023, spec-stable, widely used in NestJS and TypeORM. The 2026 SaaS stack the course teaches (Next.js + Drizzle + Better Auth) doesn't reach for them. Named once, dismissed for this stack — *not* "decorators are bad," but "this stack doesn't need them."
5. **Getters and setters.** They look like field reads at the call site but execute code. The surprise is the cost: a `for` loop reading `cart.totalCents` ten thousand times runs the totaling logic ten thousand times. Plain methods (`cart.totalCents()`) make the cost visible where it happens.
6. **Class expressions (`const C = class { ... }`).** Anonymous classes complicate stack traces and rename badly. Always `class C { }` at module scope, exported by name.

The principle behind the refusals is one sentence: **the reach that ages well is the smallest surface that makes the trigger work.** Every refused primitive solves a real problem the SaaS stack doesn't have. Lock it.

## `#private` and the runtime privacy story

Of the five primitives above, the one most likely to trip you up is the choice between TypeScript's `private` keyword and the JavaScript-runtime `#private` syntax. They look similar; they are not. The TypeScript form is a compile-time hint that erases at runtime. The `#private` form is a runtime guarantee enforced by the JavaScript engine.

The difference matters because the failure mode is structural, not stylistic. A field marked `private` in TypeScript will *appear* private when the rest of the codebase tries to read it — the type checker refuses. But the same field is reachable at runtime by anyone holding the instance, via bracket access or `Reflect.get`. If the field holds a secret (an API key, a session token), that's the bug class.

    TypeScript's `private` is a compile-time hint, erased before the engine sees the code. A static `holder['secret']` is still caught by the type checker, but the moment the access goes through `as any`, `Reflect.get`, or a computed key the checker can't resolve, the field reads cleanly. Anyone with the instance can reach it that way; reflection and `Object.entries` see it; `JSON.stringify` writes it. The privacy is a *naming convention enforced by the compiler*, nothing more.
  </CodeVariant>

  <CodeVariant label="#private (preferred)">
    <div data-mark-color="green">

    ```ts {2} "holder['#secret']"
    class SecretHolder {
      #secret = 'sk_live_xxx';
    }

    const holder = new SecretHolder();
    holder['#secret']; // undefined — the field is invisible to bracket access
    ```

    </div>
    The `#secret` syntax is enforced by the runtime. Bracket access returns `undefined`; the field doesn't appear in `Object.keys`, `Object.entries`, or `JSON.stringify`. Even reflection can't reach it. Privacy survives the compile. This is the form to reach for when the field holds anything you wouldn't want to read in a debugger across a process boundary.
  </CodeVariant>

  <CodeVariant label="Per-class scope">
    <div data-mark-color="blue">

    ```ts "#secret"
    class A {
      #secret = 'a';
      read() { return this.#secret; }
    }

    class B {
      #secret = 'b';
      read() { return this.#secret; }
    }
    ```

    </div>
    Hash-private fields are scoped to their declaring class. Two unrelated classes can both declare `#secret` and never collide — the names live in separate per-class slots, not a shared string namespace on the instance. TS `private` fields, by contrast, are just regular string-keyed properties at runtime; if two classes (or a subclass and its parent) both pick the same name, they share the slot and write through each other. `#private` is the only privacy the language *actually* gives you.
  </CodeVariant>
</CodeVariants>

:::note
Hash-private is also the only privacy `JSON.stringify` respects. Connecting back to the previous lesson: when a class instance crosses the wire, `stringify` writes only its *own enumerable* properties — `#private` fields are excluded structurally. The `BillingClient` above benefits from this for free: `JSON.stringify(billing)` produces `{}` because `#stripe` is the only field and it's hash-private — the Stripe API key can't leak through an accidental log line or an over-eager `res.json(...)`. If you need the wire shape to be something other than empty, implement `toJSON()` explicitly. The `Cart` example below does exactly that.
:::

## Putting it together — the stateful domain object

Trigger three is the rarest of the three. Most "stateful domain objects" you'll be tempted to write are records that the database, React state, or a Zustand store should own. The case that genuinely earns a class is an in-memory aggregate that *enforces an invariant through its methods* — the kind of object where refactoring it to "a `Map` plus a few functions" would turn a guarantee into a discipline.

A shopping cart is the textbook example. Adding the same SKU twice merges the quantities into one line instead of producing two. The total is derived from the lines. The wire shape is the lines and the total, packaged for serialization. Read the whole thing once and we'll walk what makes it earn the trigger.

```ts
type CartLine = { sku: string; quantity: number; unitPriceCents: number };

export class Cart {
  readonly #lines = new Map<string, CartLine>();

  addLine(line: CartLine): void {
    const existing = this.#lines.get(line.sku);
    const merged = existing
      ? { ...existing, quantity: existing.quantity + line.quantity }
      : line;
    this.#lines.set(line.sku, merged);
  }

  removeLine(sku: string): void {
    this.#lines.delete(sku);
  }

  totalCents(): number {
    let total = 0;
    for (const line of this.#lines.values()) {
      total += line.quantity * line.unitPriceCents;
    }
    return total;
  }

  toJSON(): { lines: CartLine[]; totalCents: number } {
    return { lines: [...this.#lines.values()], totalCents: this.totalCents() };
  }
}
```

Three things to notice. First, all four methods (`addLine`, `removeLine`, `totalCents`, `toJSON`) are regular methods, not arrow fields. The methods are always called as `cart.method(...)` — never passed as callbacks, never stored in a `Map` — so there's no `this` binding to preserve. Regular methods are one prototype copy, no per-instance closure; cheaper memory, same correctness.

Second, `totalCents()` is a method, not a getter. The lesson refused getters one section ago, and the refusal applies in full force here — a getter would let `cart.totalCents` read like a field while running a loop underneath. `totalCents()` makes the cost visible at the call site.

Third, `toJSON()` is the wire seam. When this `Cart` crosses an RSC payload or a Server Action response, the platform's `JSON.stringify` calls `toJSON()` and writes the *record* it returns — `{ lines, totalCents }` — not the class instance. The `#lines` Map disappears (it would serialize as `{}` otherwise), the methods disappear, the privacy disappears. On the other side, the receiver reads a typed record; the `Cart` is reconstructed only if the receiver needs the methods.

This class earns its trigger for three concrete reasons. The `#lines` Map is private state — nothing outside the class can read or write it. The invariant *quantities for the same SKU are summed, not duplicated* lives in `addLine`'s method body, not in the data — the Map's key uniqueness is what makes one slot per SKU possible, but the *merge* of an incoming line into an existing one is the work the method does, and every write path goes through it. The wire shape is the `toJSON()` record, explicit and intentional. You *could* refactor this to a module of functions over a `Map<string, CartLine>` — but then every caller hand-threads the Map, and the merge invariant becomes a discipline rather than a guarantee.

Note the rarity. Most "carts" in 2026 SaaS code live in the database or in a Zustand store; this in-memory shape fires when the cart never persists and the invariants are non-trivial. The same reasoning applies to a token-bucket rate limiter, an in-memory cache wrapper, a parser's stateful tokenizer. Three or four times across a SaaS codebase, total.

## Equality, `instanceof`, and the cross-realm trap

Two more rules close out the surface. Both are pivots from the previous chapter on errors, restated for the non-error case.

**Reference equality.** The `===` operator on two class instances compares identity, not value. Two `Cart` instances built from the same data are not equal: `new Cart() === new Cart()` is `false` even when both are empty. The cure when you find yourself needing value equality is *to use a record instead of a class*. The shape is the value; deep equality on records is well-defined; the `Cart` class earns its weight because the *invariant* is its product, not the value itself.

**`instanceof` and realms.** Recall from the previous chapter on errors: `instanceof` walks the prototype chain in the current <Term definition={"An isolated JavaScript execution context — an iframe, a Worker, a `vm.runInContext` sandbox. Each realm carries its own copy of every built-in constructor, so a value created in one realm fails `instanceof` against the same class imported from another."}>realm</Term>. Values that crossed an iframe, a Worker, or a `vm.runInContext` boundary carry the *other* realm's prototype, and `instanceof` returns `false` even when the class name matches. For `Error` subclasses, the cure was the literal `name` discriminant — a string comparison survives the realm crossing.

For the two non-error class triggers, the realm trap rarely bites. SDK adapter wrappers live on the server, are constructed once, and never cross a realm. Stateful domain objects either stay in one realm or cross a serialization seam where they're reconstructed from scratch on the other side. Inside one realm, `instanceof` is fine — `value instanceof BillingClient` narrows correctly, the prototype chain is intact. The reflex to internalize: `name` discriminants for errors that cross boundaries; `instanceof` for in-realm narrowing on the other triggers.

## Watch-outs

A checklist of the five most likely PR-review findings on class code. Carry the list.

:::caution
- `private` (TypeScript) is erased at runtime; bracket access reads the field. Reach for `#private` everywhere privacy *actually* matters.
- Passing `instance.method` as a callback strips the `this` binding silently. Bind explicitly at the call site, or arrow-field the method at declaration.
- `static` methods can't see instance fields. Common copy-paste bug when refactoring an instance method to a factory — the body still references `this.x` and the compiler doesn't always catch it cleanly.
- `JSON.stringify(instance)` drops methods and `#private` fields. Implement `toJSON()` explicitly when the wire shape needs to differ from the in-memory shape — the `Cart` above is the template.
- Using a class to "group related functions" is a smell. The module is the grouping primitive. `class UserService` is the canonical anti-pattern.
:::

    <TfWhy>`readonly` is compile-time field immutability — the binding can't be reassigned, the field's contents can still mutate. `#private` is runtime-enforced visibility — the field can't be read or written from outside the class. They solve different problems and you reach for both.</TfWhy>
  </Statement>

  <Statement answer="true">
    Arrow-field methods (`handle = () => { ... }`) are the right reach when the method is passed as a callback.
    <TfWhy>The arrow-field form binds `this` at construction, so the method survives detachment. Regular methods lose `this` when passed as callbacks. Default to regular methods (cheaper memory); arrow-field when detachment is in the call path.</TfWhy>
  </Statement>
</TrueFalse>
