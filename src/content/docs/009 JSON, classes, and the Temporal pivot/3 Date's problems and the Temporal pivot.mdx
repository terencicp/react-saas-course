A reminder-email job is scheduled to fire "every Monday at 9 AM in the user's timezone." The implementation is the obvious one: read the last fire time, add seven days of milliseconds, write the new time back.

```ts
const nextFire = new Date(lastFire.getTime() + 7 * 24 * 60 * 60 * 1000);
```

Twice a year, the email fires an hour early or an hour late. The on-call engineer can reproduce it locally only by changing the system clock to the day before a daylight saving transition. The fix the engineer reaches for — special-case the DST week, adjust by an hour — is not the fix. The bug is structural. `Date` is a count of UTC milliseconds with no concept of "9 AM in Sydney"; any code that adds twenty-four hours to a millisecond count is a <Term definition="Daylight saving time — the seasonal one-hour shift between standard time and summer time, applied differently in each region.">DST</Term> bug in waiting. The cure isn't smarter arithmetic. It's a different type.

The platform finally has one. `Temporal` advanced to TC39 Stage 4 on March 11, 2026 and shipped unflagged in Node 26 on May 5, 2026 — the course's deploy target on Vercel. This lesson catalogues why `Date` produces this kind of bug structurally, introduces the five Temporal types that replace it, and installs the three rules the rest of the course rides on. The full Temporal architecture — arithmetic surface, DST scheduling, formatting, storage codecs — lands in unit 17 on time and internationalization. Here, you learn *which type to reach for*.

## Why Date keeps causing production bugs

The DST-drift bug isn't a one-off. `Date` ships with eight design flaws baked in, grouped into three clusters: traps in the API surface, traps in how it parses and reports values, and traps in its precision. Each one has the same root: `Date` overloads three jobs — a UTC instant, a calendar date, a wall clock — onto one shape, a millisecond count plus a few formatters that read the host's local timezone. When you call a `Date` method, the type doesn't tell you which of those three roles is in play. The runtime guesses, and sometimes guesses wrong.

Read the catalogue once. Don't try to memorize all eight; the exercise after the cluster locks in recall.

### The API traps

Three behaviours where the method signatures themselves are the mistake.

```js
new Date(2026, 0, 31);
// → January 31, 2026 — month is zero-indexed, day is one-indexed

new Date(2026, 12, 1);
// → January 1, 2027 — months overflow silently

const d = new Date('2026-05-15');
d.setMonth(5);
// mutates d in place; no return value

const week = 7 * 24 * 60 * 60 * 1000;
const nextWeek = new Date(d.getTime() + week);
// 'a week' as a number; breaks across DST
```

Zero-indexed months next to one-indexed days is the first paper cut. Mutation via `setX` is the second — pass a `Date` to a function and the function can silently alter the caller's value. The third is the lack of a duration type: any span you need (`30 days`, `2 hours`) becomes a raw number of milliseconds, and that number doesn't know that "+24 hours" crosses a DST boundary into either 23 or 25 wall-clock hours.

### The semantic traps

Three behaviours where the same call means different things in different contexts.

```js
new Date('2026-05-15').toISOString();
// → 2026-05-15T00:00:00.000Z (UTC, by spec)

new Date('2026-05-15 00:00').toISOString();
// → depends on the runtime's local timezone

new Date('garbage');
// → Invalid Date — typeof is 'object', instanceof Date is true

new Date('garbage').getTime();
// → NaN

new Date().toString();
// → host-locale-and-tz-dependent; differs dev vs prod
```

The first pair is the worst of them. A *space* between the date and the time changes the parse from UTC to local. Same string shape, different timezone semantics. Production code that accepts a date string from a third party will be tested on a UTC server and break in a non-UTC environment.

The second is the `Invalid Date` <Term definition="A special 'this isn't a real value' value returned in place of throwing an error. The type system says it's valid; the runtime says it isn't.">sentinel</Term>. `new Date('garbage')` doesn't throw — it returns a `Date` instance whose internal time is `NaN`. The type system says it's a `Date`. The runtime says it isn't. Every downstream call has to defensively check `isNaN(d.getTime())` or risk silently propagating bad data.

The third is `toString`. The output depends on the host's locale and timezone. Logging dates with `toString` produces strings that look one way on a developer's laptop, another way on a Vercel build runner, and a third way in Sentry breadcrumbs. The wire is meant to be canonical; `toString` isn't.

The space-versus-`T` trap is worth predicting once. Imagine you're running the two lines below on a server whose timezone is set to UTC — what do they print?

### The precision traps

Two behaviours where the number of bits matters.

```js
new Date(99, 0, 1);
// → January 1, 1999 — legacy two-digit-year heuristic

Date.now();
// → milliseconds — Postgres timestamptz is microseconds

performance.now();
// → sub-millisecond resolution — Date's millisecond floor is coarser than either
```

The Year-100 trap is the smallest flaw on the list but the most embarrassing: `new Date(99, 0, 1)` is 1999, not the year 99. The legacy two-digit-year heuristic remains in the runtime for backward compatibility.

The precision trap matters more. Postgres `timestamptz` stores microseconds and `performance.now()` returns sub-millisecond resolution — `Date`'s millisecond floor is coarser than either, and a high-frequency event log round-tripped through `Date` loses ordering on simultaneous events.

The cure for all eight is the same: more types, not more methods. The reason `Date` is structurally unsafe is that it overloads "an instant in real time", "a calendar date", and "a wall clock" onto one shape. The replacement is to give each of those three meanings its own type, with its own API, refusing the operations that don't make sense for it.

Before you move on, sort the behaviours. Some of these are `Date` misfeatures; some are normal language behaviour you might confuse with one.

## The three canonical production bugs

The eight flaws are abstract. Three real bugs are what makes them stick. Each one is a structural consequence of `Date`'s overload, not a bad-luck edge case, and each one points at a specific Temporal type as the cure.

**The DST reminder drift.** The Monday-morning reminder email fires at 8 AM or 10 AM for half the year. The math (`+ 7 * 86_400_000`) advances seven UTC days, but the user's wall clock isn't measured in UTC days — it shifts an hour when their region enters or leaves DST. The cure: `Temporal.ZonedDateTime` paired with `add({ days: 1 })`. The type knows the user's IANA tz; the add operation knows that "a day" on a wall clock is a calendar concept, not a fixed number of milliseconds.

**The date-line invoice.** An invoice is created with `dueDate: '2026-05-15'`. The server stores it as `2026-05-15T00:00:00Z` — UTC midnight on May 15. The UI reads it back as a `Date` and renders it with the user's local timezone. For a user in Los Angeles (UTC-8), UTC midnight on May 15 is 4 PM on May *14* — the date the UI prints. The invoice the customer agreed was due on the 15th now displays as due on the 14th. The cure: `Temporal.PlainDate`. A calendar date has no time and no timezone; it's the same May 15 in Sydney, Tokyo, and Madrid, because that's what a calendar date *means*.

**The audit-log timezone smear.** A Server Component renders an audit-log row's timestamp with `new Date(row.createdAt).toLocaleString()`. The server clock is UTC. Every user — in São Paulo, in Berlin, in Honolulu — sees the same UTC string regardless of where they are. The cure: store `Temporal.Instant` in Postgres, send the ISO string over the wire, and format it on the client (or render it server-side using the user's profile timezone) at the boundary. The full pattern lands in unit 17; the choice this lesson installs is the type — `Temporal.Instant` for "when did this happen".

Three vignettes, three Temporal types named. The catalogue is next.

## Temporal, the type catalog

The Temporal proposal replaces one type with five. Each type names one of the meanings `Date` overloaded onto its single shape, and each one refuses the operations that don't make sense for it. The full surface is rich — arithmetic, comparisons, rounding, calendar systems — and unit 17 walks through it in depth. What you need here is the *map*: which type for which meaning.

Each type has a static factory. The constructor is hidden; you call `from`, `fromEpochMilliseconds`, or `Temporal.Now`.

```ts
const now = Temporal.Now.instant();
const due = Temporal.PlainDate.from('2026-05-15');
const schedule = Temporal.ZonedDateTime.from(
  '2026-05-15T09:00:00[Australia/Sydney]',
);
const sla = Temporal.Duration.from({ days: 30 });
const local = Temporal.PlainDateTime.from('2026-05-15T17:00');
```

The `from` method accepts ISO 8601 strings or option objects. The bracketed `[Australia/Sydney]` in the third line is the standard ISO 8601 extension Temporal uses to attach a timezone to a date-time.

Three properties of the catalogue carry the rest of the architecture.

**The type tells you what you have.** This is the key shift. With `Date`, you can call `due.getHours()` on what you intended as a calendar date; the API doesn't refuse. With Temporal, `PlainDate` has no `getHours()` method — the operation is meaningless on a calendar date, so it isn't there. The type system and the runtime agree on what each value is for. Ambiguity is refused at the API, not caught downstream.

**Immutability everywhere.** Every Temporal operation returns a new instance. `instant.add({ minutes: 5 })` does not mutate `instant`; it returns a new `Instant`. No method ever changes the receiver. This pairs with the `const` discipline you already use for value bindings: the value a name points at never silently changes under your feet.

**Errors over sentinels.** `Temporal.Instant.from('garbage')` throws a `RangeError`. There is no `Invalid Instant` sentinel; bad input doesn't propagate. The catch belongs at the *parse seam* — the route handler, the Server Action wrapper, the third-party adapter — the same discipline you installed for `JSON.parse` in the first lesson of this chapter. The wire is unknown until validated; Temporal `from()` rides the same pattern.

## The runtime story, May 2026

Temporal is real native code in Node 26, and it's a <Term definition="A small library that implements a not-yet-native API in JavaScript so existing code can use the same shape before the platform ships it.">polyfill</Term> everywhere else. The seam between those two is the one piece of plumbing this lesson installs.

**Stage 4 and Node 26.** Temporal advanced to TC39 Stage 4 on March 11, 2026 — the proposal is now part of the ECMAScript 2026 specification. Node 26.0.0 shipped on May 5, 2026 with Temporal unflagged in the default global. That's the course's deploy target on Vercel's Node 26 runtime. Node 26 promotes to LTS in October 2026.

**Node 24 LTS, the gap.** Until Node 26 promotes to LTS, production codebases still on Node 24 don't have native Temporal. Two polyfills are mature in 2026: `temporal-polyfill` from FullCalendar, roughly 20 KB gzipped, with near-complete spec compliance; and `@js-temporal/polyfill` from the TC39 champions themselves, in the 45–60 KB range depending on version. The course picks `temporal-polyfill` for the smaller bundle.

**Browser support, not a blocker.** Firefox 139 enabled Temporal by default in mid-2025. Chrome 144 ships it in 2026. Safari is pending. None of this is on the critical path for the course because the SaaS the course builds renders dates server-side and ships ISO 8601 strings to the browser — the browser doesn't need a Temporal runtime to display a date. Locale-aware formatting lands in unit 17 too.

## The lib/temporal.ts seam

Here's the rule the whole course depends on: there is exactly one file in the codebase that imports the polyfill, and every other file imports from that file. The convention has a name — `lib/temporal.ts` — and a shape:

```ts
// lib/temporal.ts
import { Temporal } from 'temporal-polyfill';

export { Temporal };
```

That's the whole file for a project pinned to Node 24. Two lines. The point isn't the brevity; it's the <Term definition="A boundary in the code where the application meets something external — an SDK, the database, the wire — concentrated in one place so the rest of the codebase can stay clean.">seam</Term>. Once Node 26 becomes the project's pinned runtime, those two lines collapse to one re-export from the native global — every other file in the codebase already imports `Temporal` from `@/lib/temporal`, so the swap is a single-file change.

<AnnotatedCode lang="ts" code={`
// lib/temporal.ts
import { Temporal } from 'temporal-polyfill';

export { Temporal };

export const instantFromDate = (d: Date): Temporal.Instant =>
  Temporal.Instant.fromEpochMilliseconds(d.getTime());

export const dateFromInstant = (i: Temporal.Instant): Date =>
  new Date(i.epochMilliseconds);
`}>
  <AnnotatedStep meta="{2-4}" color="blue">
    The single import path is the discipline. Every application file reaches for `Temporal` from `@/lib/temporal` — never from `temporal-polyfill` directly, never from `globalThis.Temporal`. The reason isn't aesthetic: mixing the native and polyfill `Temporal` in the same process *breaks `instanceof` checks* and *breaks cross-instance interop with `from()`*. The two values look identical from the outside and aren't the same class.
  </AnnotatedStep>

  <AnnotatedStep meta="{6-7}" color="green">
    The first codec: take a `Date` (handed back by a third-party SDK) and produce a `Temporal.Instant` the rest of the application can read. The arrow form, the explicit return type, and the one positional parameter are all consistent with the project's `/lib` helper conventions.
  </AnnotatedStep>

  <AnnotatedStep meta="{9-10}" color="orange">
    The reverse codec, for the rare path where you hand a `Date` *back* to an SDK that requires one. Application code never reaches for either of these directly — the call sites are inside SDK adapters (in unit 12, `lib/billing/` is the canonical example).
  </AnnotatedStep>
</AnnotatedCode>

For projects that toggle between Node 24 and Node 26 across environments — local dev on the new runtime, CI on the LTS — the same file picks up a one-line runtime guard so it works on either.

```ts
// lib/temporal.ts (runtime-guarded variant)
import { Temporal as TemporalPolyfill } from 'temporal-polyfill';

export const Temporal = globalThis.Temporal ?? TemporalPolyfill;
```

For a project pinned to one Node version, the simpler re-export is the right reach. The course's pinned runtime is one version at a time, so the first form is the default.

:::caution
Never `import { Temporal } from 'temporal-polyfill'` outside `lib/temporal.ts`. Later in the course, an ESLint rule (`no-restricted-imports`) will enforce this automatically; for now, the rule is by discipline. Two `Temporal` instances in the same process break `instanceof` silently.
:::

## Date stays at the seam

The pivot is type-level — but `Date` doesn't vanish from the codebase. Two cases keep it in scope, and naming them up front prevents the over-correction reflex where students try to purge `Date` from places it legitimately belongs.

**SDK ingress.** Third-party SDKs hand back `Date` instances on their callbacks and response objects. Stripe's SDK is the canonical example: its wire format is Unix seconds (10-digit integers), but the JavaScript SDK wraps them in `Date` instances on the way out, so your application code receives `subscription.created` as a `Date` object. The application's domain doesn't read that `Date` directly. It converts at the seam.

**Stopwatch measurements.** `Date.now()` and `performance.now()` are the right tools for "how many milliseconds did this operation take" — measurements where absolute time has no meaning, only the difference between two reads. The result is a number, not a value that travels into the domain. `performance.now()` is the better reach when it's available (sub-millisecond, monotonic — doesn't jump when the system clock corrects).

Everything else is Temporal. If a value gets *stored* in Postgres, *displayed* in the UI, *compared* against other moments in domain logic, or *scheduled* via a background job, it's a Temporal type. If it's a measurement seam — a stopwatch, a clock differential — `Date.now()` or `performance.now()` is fine.

The shape of the conversion is small enough to read in one pass.

<CodeVariants>
  <CodeVariant label="At the seam">
    ```ts
    // lib/temporal.ts
    import { Temporal } from 'temporal-polyfill';

    export { Temporal };

    export const instantFromDate = (d: Date): Temporal.Instant =>
      Temporal.Instant.fromEpochMilliseconds(d.getTime());

    export const instantFromUnixSeconds = (s: number): Temporal.Instant =>
      Temporal.Instant.fromEpochMilliseconds(s * 1000);
    ```
    **The only place `Date` touches `Temporal`.** Most projects only need the first function; the second is for SDKs (Stripe is the canonical example) that hand you raw Unix seconds instead of a `Date` wrapper.
  </CodeVariant>

  <CodeVariant label="In application code">
    ```ts
    import { Temporal } from '@/lib/temporal';

    type Subscription = {
      id: string;
      currentPeriodEnd: Temporal.Instant;
    };

    export const isExpired = (
      sub: Subscription,
      now: Temporal.Instant,
    ): boolean => Temporal.Instant.compare(sub.currentPeriodEnd, now) < 0;
    ```
    **No `Date` in sight.** The adapter at the seam (in unit 12 the SDK lives in `lib/billing/`) is the only place that touches the SDK's `Date` wrappers — domain code reads `Temporal.Instant` exclusively.
  </CodeVariant>
</CodeVariants>

The conversion is one direction at this stage: third-party `Date` becomes domain `Temporal.Instant`. The reverse — `dateFromInstant` for SDKs that take a `Date` as input — lands in unit 17 alongside the Drizzle codec. The principle is the same: one converter, one place.

## The three rules

The whole lesson collapses to three rules. Memorize these; the rest of the course assumes them.

<Steps>

1. **Never `new Date(year, month, day)` or `date.setX(...)` in application code.** Construction goes through `Temporal.PlainDate.from('2026-05-15')` or `Temporal.Now.instant()`. Mutation doesn't happen — every Temporal operation returns a new instance.

2. **Convert SDK `Date` to `Temporal.Instant` at the seam.** Use `instantFromDate` from `lib/temporal.ts`. No raw `Date` propagates inward from a third-party SDK.

3. **Import `Temporal` from `@/lib/temporal`.** Never from `temporal-polyfill` directly, never from `globalThis.Temporal`. One seam, one import path, one place to swap when Node 26 becomes the pinned runtime.

</Steps>

These three rules are the *pivot*. The architecture rides on top in unit 17: Drizzle codecs for `timestamptz` and `date` columns, the `users.timeZone` profile column, DST-aware schedules in Trigger.dev, and locale-aware formatting via `Intl.DateTimeFormat`. Every lesson in that unit refers back here. And one more habit you don't need to apply yet but should file away: 2026 SaaS projects don't install `date-fns`, `dayjs`, `luxon`, or `moment`. The platform is the library now.
