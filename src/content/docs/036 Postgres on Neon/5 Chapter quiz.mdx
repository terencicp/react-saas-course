  <MultipleChoice src="36.3">
    Branching a 10 GB Neon database is near-instant and adds almost no storage, while the old `pg_dump`/restore copy takes minutes and a fresh 10 GB on disk. Which single architectural fact is the root cause of that difference?

    <McqChoice correct>Neon separates storage from compute, so a branch is a new stateless compute pointed at a copy-on-write snapshot — it shares the parent's existing pages and only writes the ones it diverges on.</McqChoice>
    <McqChoice>Neon keeps every branch's compute warm in advance, so the copy is already running by the time you ask for it.</McqChoice>
    <McqChoice>Neon compresses the parent's pages before branching, shrinking the 10 GB enough to duplicate it quickly.</McqChoice>

    <McqWhy>Everything in the lesson — cheap branches, scale-to-zero, read replicas — falls out of one move: storage and compute are pulled apart. Because storage is an independent, shared log of pages, a branch starts by sharing all of the parent's pages copy-on-write and only pays for what it later changes. Warm compute is about cold-start latency, not copy cost, and nothing is compressed or physically duplicated up front.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="36.1">
    A spec says: a `user` can belong to many `project`s, and a `project` can have many `user`s. What shape does this take in a normalized schema?

    <McqChoice correct>A junction table (e.g. `project_members`) holding a foreign key to each side, keyed on the pair of those two columns.</McqChoice>
    <McqChoice>A single `project_id` foreign-key column on the `user` table.</McqChoice>
    <McqChoice>A comma-separated `project_ids` column on `user` and a `user_ids` column on `project`.</McqChoice>

    <McqWhy>"X can have many Ys *and* a Y can belong to many Xs" is many-to-many, and neither side can hold a single foreign key — so you introduce a junction table with one foreign key to each side, keyed on the composite pair. A lone `project_id` column expresses one-to-many (each user in exactly one project), which the spec contradicts. The comma-separated columns are a 1NF violation and can't be foreign-key-enforced or indexed.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="36.2">
    The course commits to a Neon dev branch as the default local database rather than Docker Postgres. What is the reasoning, and what does choosing Docker instead cost a student given how the app is wired?

    <McqChoice correct>Prod-parity prevents more bugs than offline prevents lost hours, so the default optimizes for parity — and because the app only ever reads `DATABASE_URL`, a student who prefers Docker changes one variable and loses nothing downstream.</McqChoice>
    <McqChoice>Neon branches are faster to query than a local container, and switching to Docker would require rewriting the queries and schema to target the local Postgres dialect.</McqChoice>
    <McqChoice>Docker can't run the same migrations as Neon, so a Docker student has to maintain a separate, parallel set of migration files.</McqChoice>

    <McqWhy>The default optimizes for parity because a "works on my machine" failure that surfaces in staging costs more than the rare afternoon without wifi. The whole point of the `DATABASE_URL` contract is that the choice is orthogonal to everything downstream: the same migrations, the same seed, and the same `db` client run against any of the three options. Docker's cost is real but it's drift (and losing offline only), not query rewrites or a second set of migrations — those don't change at all.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="36.4">
    Your deploy step runs `drizzle-kit migrate`, a long stream of `ALTER`/`CREATE` statements in one session. A teammate points it at the pooled `DATABASE_URL` (the `-pooler` host) to keep it consistent with the app. Why is that the wrong URL, and which client exists for this job?

    <McqChoice correct>PgBouncer transaction mode can reclaim the backend at a transaction boundary and sever the migration mid-stream — leaving a half-applied schema — so migrations take the unpooled direct URL via `dbUnpooled`.</McqChoice>
    <McqChoice>The pooled URL is read-only, so DDL statements silently no-op against it; migrations must use `dbUnpooled`, which is the only client with write access.</McqChoice>
    <McqChoice>The pooled URL caps queries at one statement per connection, so a multi-statement migration is rejected outright before any change is applied.</McqChoice>

    <McqWhy>Transaction-mode pooling hands each transaction a possibly-different backend and is allowed to reclaim a connection at any transaction boundary — fine for short app bursts, fatal for a long single-session migration it could cut partway through, leaving a half-applied schema. That hazard is the entire reason `dbUnpooled` (over `DATABASE_URL_UNPOOLED`, skipping the pooler) exists. The pooled URL isn't read-only and doesn't hard-cap statements; the failure is severed sessions, not rejected writes.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="36.4">
    A Server Component reads the invoice list to render a page; a Server Action later reads an account balance, decides, then deducts credits and inserts a charge atomically. Match each to its client. **Select all that apply.**

    <McqChoice correct>The Server Component read uses `db` — the pooled HTTP client: one fetch per query, no held connection.</McqChoice>
    <McqChoice correct>The Server Action uses `dbTx` — the WebSocket client over the *same* pooled URL — because an interactive read-decide-write needs one connection held across several steps.</McqChoice>
    <McqChoice>The Server Action must use the unpooled URL, since any transaction has to skip the pooler to hold its connection.</McqChoice>

    <McqWhy>The two axes are independent. *Which URL?* Both are ordinary app traffic, so both ride the pooled `DATABASE_URL` — the unpooled URL is for migrations and long maintenance sessions, not "transactions." *Which driver?* A one-shot render read takes HTTP (`db`); a read-modify-write that must be all-or-nothing takes the WebSocket transaction client (`dbTx`), which holds its connection only for the transaction's few milliseconds — something transaction-mode pooling handles fine.</McqWhy>
  </MultipleChoice>
