Last lesson, "Dark mode via semantic tokens," you styled a component to render correctly in both themes — *assuming* something had already put `.dark` on `<html>`. This lesson builds that something. And it's trickier than it sounds, because of one specific failure: picture a user who keeps their machine in dark mode opening your app. For a fraction of a second they see a white flash, then the page snaps to dark. That flash is the bug this whole lesson exists to kill. By the end you'll have a theme toggle that switches instantly with no flash, and you'll be able to explain *why* every piece of the setup has to be there — not just that some guide told you to add it.

That flash has a name. It's <Term definition="Flash of unstyled content. Here specifically, a flash of the wrong-themed page — the default theme paints for an instant before the correct one applies.">FOUC</Term> — a flash of the wrong-themed content before the right theme takes hold. Keep that picture in your head; everything below is downstream of preventing it.

## Why the theme can't wait for React

Before reaching for a tool, let's see exactly *when* the flash happens, because the timing is the whole problem. Build the naive version in your head — the one you'd write with only what you know so far — and watch where it breaks.

You'd put the toggle in a component, have it read the saved theme, and add `.dark` to `<html>` when the saved theme is dark. Reasonable. Now trace what actually happens when the page loads:

<Steps>
1. The server renders the page. But **the server doesn't know the user's theme** — the preference lives in the browser (in `localStorage`) and in the OS setting, and the server can read neither. So it emits HTML with the *default* theme: no `.dark` class.

2. The browser receives that HTML and **paints it**. This is the first paint — the user's first pixels — and it happens *before any of your React code runs*. So the screen shows the default (light) theme.

3. React loads and hydrates. Now your code runs: it reads the saved theme, sees `"dark"`, and adds `.dark` to `<html>`.

4. The page **repaints** in dark. The user saw light, then dark. That gap between step 2 and step 4 is the flash.
</Steps>

Stare at the root cause, because the fix falls straight out of it: **the correction runs after the first paint.** Any approach that reads the theme inside React — an effect, a provider that flips the class on mount — is structurally too late. By the time React is running, the wrong pixels are already on the user's screen. You can't fix a paint that already happened; you can only race to set the class *before* it.

So the fix has a precise shape, and we can describe it before naming any package: **the class has to be written by code that runs before the body paints.** What runs that early? A plain synchronous `<script>` in the `<head>`. The browser executes a blocking `<head>` script inline, while it's still parsing, *before* it paints the `<body>`. Slot a tiny script there that reads `localStorage` and the OS preference and writes `.dark` onto `<html>`, and the very first paint is already correct. No flash, because there was never a wrong frame to flash.

The diagram below makes that concrete. Scrub through both lanes: the top lane is the naive timeline you just traced, the bottom is the fixed one. Watch *where in time* the flash lives — it sits between "HTML arrives" and "first paint" — and watch the inline `<head>` script slot into exactly that gap to close it.

Back in "The root layout" in the previous chapter, you met `suppressHydrationWarning` as the surgical fix for one element whose mismatch is expected and harmless — and you saw it sit on `<html>` in exactly this kind of setup. One word in that timeline deserves a quick refresher, since the rest of the lesson leans on it. <Term definition="React's second pass: the browser re-runs the component tree over the server's existing HTML and attaches event handlers to it, rather than rebuilding the DOM. The deeper render model comes in a later chapter.">Hydration</Term> is React's second pass — in the browser, it walks the same component tree the server already rendered, adopts the existing HTML, and wires up event handlers. You met it in the previous chapter; the one thing that matters here is that hydration happens *after* the first paint, which is precisely why a React-based theme fix is too late.

## next-themes: the script, packaged

You could hand-write that inline script. You shouldn't — it's fiddly to get right (reading storage, falling back to the OS preference, avoiding its own flash), and it's a solved problem. The standard solution on a Next.js stack is a tiny package called <Term definition="A ~3KB library that manages light/dark/system theming for React apps: it injects the pre-paint inline script, persists the choice to localStorage, tracks the OS preference, and exposes a useTheme() hook.">`next-themes`</Term>. It's about 3KB, and its entire job is the thing the last section described. Concretely, it:

- injects that synchronous `<head>` script for you, so the class is set before the first paint;
- persists the user's choice to `localStorage`, so it sticks across visits;
- listens to the OS <Term definition="The CSS media query that exposes the operating system's light/dark setting. next-themes reads it so 'system' mode follows the OS live; its full syntax comes in a later chapter.">`prefers-color-scheme`</Term> setting, so `"system"` mode follows the OS live; and
- exposes a React hook, `useTheme()`, to read and change the theme from your components.

You install it and configure it once. You never write the inline script yourself — that's the part you're paying 3KB to skip. There are only two moving parts to learn, and the rest of the lesson wires them:

- **`<ThemeProvider>`** — wraps your app, holds the configuration, and is what injects the script.
- **`useTheme()`** — the hook your toggle button calls to read and set the theme.

Install it:

```bash
npm i next-themes
```

## Wiring the provider into the root layout

There's a placement constraint here that you already have the tools to reason about, so let's reason about it rather than copy a recipe. `<ThemeProvider>` uses React context and effects — that makes it a Client Component. But `app/layout.tsx` is the root layout: it owns the `<html>` and `<body>` tags and, as you saw in the previous chapter, it must stay a Server Component. Dropping `'use client'` at the top of the root layout to host the provider would turn your entire app into a client subtree — exactly the thing you don't want.

This is exactly the `<Providers>` pattern the previous chapter set up — a thin Client Component at `app/_components/providers.tsx` that carries the `'use client'` directive and wraps the app, which the server layout renders as a child. Back then the wrapper was essentially empty, holding a bare `<ThemeProvider>`; now you fill in its configuration. The `'use client'` boundary stops at `<Providers>`; `<html>` and `<body>` stay on the server. Here are both files.

<CodeVariants>
  <CodeVariant label="app/_components/providers.tsx">
    ```tsx title="app/_components/providers.tsx" ins={8-11}

    import { ThemeProvider } from 'next-themes';

    export function Providers({ children }: { children: React.ReactNode }) {
      return (
        <ThemeProvider
          attribute="class"
          defaultTheme="system"
          enableSystem
          disableTransitionOnChange
        >
          {children}
        </ThemeProvider>
      );
    }
    ```
    **The single client-side provider wrapper from the previous chapter, now configured** — back then `<ThemeProvider>` had no props. It's also the one spot every app-wide provider consolidates: later chapters add the data-fetching client and the i18n provider right here, beside `<ThemeProvider>`.
  </CodeVariant>

  <CodeVariant label="app/layout.tsx">

    **Unchanged from the previous chapter except for one attribute.** No `'use client'` here — the root layout stays a Server Component, rendering `<Providers>` as a child exactly as before. The only new thing is `suppressHydrationWarning` on `<html>`, and it's the one piece you should never copy without understanding — which is the next section.
  </CodeVariant>
</CodeVariants>

### Why `suppressHydrationWarning` belongs on `<html>` — and only there

Here's the twist that makes this lesson click: **the fix from the first section creates a brand-new problem, and this prop is the answer to it.** Follow the chain.

The inline script mutates `<html>`'s class *before React hydrates* — that's the entire point of it. So when React does hydrate and compares the DOM it finds against the HTML the server sent, it sees a discrepancy: the server sent `<html>` with no class, but the live DOM has `<html class="dark">`. React calls that a <Term definition="When the HTML React produces on its first browser pass doesn't match the HTML the server sent. React expects them identical; a difference normally signals a bug, so React warns. Here the difference is caused on purpose by the inline script.">hydration mismatch</Term> and logs a warning.

The thing to internalize: **that mismatch is not a bug — it's the fix working as designed.** The class genuinely *is* different, because you deliberately changed it before hydration to win the race against first paint. `suppressHydrationWarning` is how you tell React "I know this one element's attributes were changed on purpose before hydration; don't warn about it." You're not hiding a problem; you're acknowledging an intentional, external mutation.

Three precise things to hold onto, because each one is a trap if you get it backwards:

- **It goes on `<html>`** specifically, because `<html>` is the element the script mutates. The class lands there; the acknowledgement goes there.
- **It is shallow.** It silences mismatch warnings for that one element's *own* attributes only — not for any of its descendants. So it can't accidentally mask a real mismatch deeper in your tree. That shallowness is exactly why it's safe here and why you don't scatter it around.
- **It is not a general "make hydration errors go away" switch.** Slapping it on some other element to quiet a real mismatch just hides a real bug. It's licensed *here*, and only here, because the mutation is intentional and comes from outside React.

## Building the theme toggle

Now the part the user actually clicks. And here's something worth noticing: **the exact same mismatch problem comes back, one level down.** A toggle naturally wants to show the icon for the *current* theme — a sun in light mode, a moon in dark. But the server doesn't know the theme (same root cause as the flash), so if the button renders the current-theme icon during the server render, React hydrates and finds a different icon than the server sent. Mismatch on the button.

There are two clean ways out. Lead with the one that needs no React state at all, because it's the reflex for a plain light/dark toggle.

<CodeVariants>
  <CodeVariant label="CSS-only swap">
    <div data-mark-color="green">

    ```tsx title="app/_components/theme-toggle.tsx" {14-15}
    'use client';

    import { Moon, Sun } from 'lucide-react';
    import { useTheme } from 'next-themes';

    export const ThemeToggle = () => {
      const { resolvedTheme, setTheme } = useTheme();
      return (
        <button
          type="button"
          aria-label="Toggle theme"
          onClick={() => setTheme(resolvedTheme === 'dark' ? 'light' : 'dark')}
        >
          <Sun className="inline dark:hidden" />
          <Moon className="hidden dark:inline" />
        </button>
      );
    };
    ```

    </div>
    **This is the reflex for a plain light/dark toggle.** Both icons always render — the markup is byte-for-byte identical on server and client, so there's nothing to mismatch. Only CSS decides which one is *visible*, keyed off the `.dark` class the inline script already set before paint. `useTheme()` is used purely to *write* on click (`setTheme`), which runs after hydration and is always safe.
  </CodeVariant>

  <CodeVariant label="Mount-gated read">
    <div data-mark-color="blue">

    ```tsx title="app/_components/theme-toggle.tsx" {8-9, 11-13}
    'use client';

    import { useEffect, useState } from 'react';
    import { useTheme } from 'next-themes';

    export const ThemeToggle = () => {
      const { resolvedTheme, setTheme } = useTheme();
      const [mounted, setMounted] = useState(false);
      useEffect(() => setMounted(true), []);

      if (!mounted) {
        return <button type="button" aria-label="Toggle theme" className="size-9" />;
      }

      return (
        <button
          type="button"
          aria-label="Toggle theme"
          onClick={() => setTheme(resolvedTheme === 'dark' ? 'light' : 'dark')}
        >
          {resolvedTheme === 'dark' ? 'Dark' : 'Light'}
        </button>
      );
    };
    ```

    </div>
    **Reach for this only when the button's *content* depends on the theme** — e.g. a menu that shows the active theme's *label* as text. Because you must read the theme in React to render text from it, you wait until after mount: a `mounted` flag flips to `true` in an effect, and until then you render a same-size placeholder so the layout doesn't jump. `useState` and `useEffect` are taught properly in a later chapter — here you're just recognizing the shape, not mastering the hooks.
  </CodeVariant>
</CodeVariants>

The takeaway on ordering is the experienced stance: for a simple light/dark switch, the CSS-only swap is the default — it sidesteps the mismatch *by construction*, with no hooks and no placeholder. Reach for the mount-gate only when the control's content, not just its styling, has to branch on the theme.

<Aside type="note" title="next-themes is the React side only">
  `next-themes` sets the class in the browser. If you ever need the *server* to render theme-dependent content — say, a server component that branches its markup on the theme — the server has to learn the theme some other way, which means storing it in a cookie it can read. That's a deliberate boundary, not something this lesson covers; just know the cookie route exists for that narrow case.
</Aside>

You'll have noticed both toggles read `resolvedTheme`, not `theme`, to decide the next value. That choice is the one `useTheme()` subtlety that trips everyone, so it earns its own moment.

### `theme` vs `resolvedTheme`

`useTheme()` hands back an object — `{ theme, setTheme, resolvedTheme, systemTheme, themes }`. Two of those values look interchangeable and are not, and mixing them up is the classic `next-themes` bug.

The distinction is *setting* versus *result*:

- `theme` is the **setting the user picked** — and it can be the literal string `"system"`.
- `resolvedTheme` is the **concrete theme in effect right now** — always `"light"` or `"dark"`, after `system` has been resolved against the actual OS preference.

When the user has chosen "system" and their OS is dark, here's what each holds:

```tsx "'system'" "'dark'"
const { theme, resolvedTheme } = useTheme();

theme;         // 'system'  — the setting the user chose
resolvedTheme; // 'dark'    — what 'system' actually resolves to right now
```

Now the trap is obvious. A toggle that computes the next theme from `theme` breaks the moment `theme` is `"system"` — there's no sensible "opposite of system" to flip to. Always read **`resolvedTheme`** when you need to know what's actually on screen, which is why both toggles above flip on `resolvedTheme === 'dark'`. (The hook returns more than this — `systemTheme`, `themes` — but you won't need them for a light/dark toggle.)

## Verifying it works

Two checks turn this from "I think it works" into "I watched it work." They also double as a triage routine — when a teammate says "dark mode is broken," these two checks tell you *which half* is broken. This is the daily reflex; build the habit now.

<Steps>
1. **Open DevTools and inspect the `<html>` element, then toggle the theme.** The `class` attribute should flip between absent and `"dark"` on every click. What you learn from it:

   - *No class ever appears.* The problem is on this side — the provider isn't wrapping the tree, or `attribute` is misconfigured. Start at `app/_components/providers.tsx`.
   - *The class flips, but the colors don't change.* The wiring here is fine; the problem is on the *Tailwind* side from last lesson — the `@custom-variant dark` line or the token overrides in `globals.css`. **Class present but no color change points at last lesson; no class at all points at this one.** That split is the whole triage.

2. **Hard-reload the page with dark active, and watch the first frame.** No flash means the inline script is doing its job — the class is set before paint. A flash means the script isn't running: most likely the provider is missing or misplaced, or you added a competing effect-based theme setter that's fighting it. The fix is to remove the rogue setter and let `next-themes` own the class.
</Steps>

To lock in the timing model that all of this rests on, put the load sequence back in order. Drag these into the order they actually happen:

## Beyond light and dark

One extension worth naming, so you recognize it when a project needs it. Some apps ship more than two themes — a marketing site with brand variants, an app with a high-contrast mode. The model stretches cleanly: switch `<ThemeProvider>` to `attribute="data-theme"`, pass the list with `themes={['light', 'dark', 'blue', 'high-contrast']}`, and add a `[data-theme="blue"] { … }` token block in `globals.css` for each — the exact same token model as last lesson, just keyed off an attribute instead of the `.dark` class. It's rare in SaaS dashboards and common on marketing and agency sites. You won't build it here; just know the same machinery scales to it.

<Aside type="note" title="A note you can mostly ignore">
  React 19 once emitted a development warning about rendering inline `<script>` tags like the one `next-themes` injects. Current `next-themes` handles this cleanly and the warning isn't something you'll trip over in 2026 — it's mentioned only so that if you read it in an older guide, you know it's not a problem you need to chase.
</Aside>
