A real SaaS ships a light theme and a dark theme from day one. Users expect to pick, and many arrive with the OS already set to dark. So you have a concrete decision to make: how does one component render correctly in *both* themes?

You already know one tool that looks like the answer. The first lesson of this chapter, "Utility-first on JSX," introduced `dark:` as a variant gate, where `dark:bg-slate-800` applies that background only in dark mode. So the obvious move is to add a `dark:` version of every color on every element. This lesson talks you *out* of that as your default, and installs the thing experienced teams reach for instead: a small set of named colors whose values the theme decides, not the component.

One boundary up front. This lesson covers the Tailwind and CSS half of dark mode: how the colors are defined and how the swap happens. It stops at the moment a `.dark` class lands on the `<html>` element. What puts it there, meaning the toggle, the code that reads the OS preference, and the code that runs before the page paints, is the next lesson, "Theme switching with next-themes." So through this whole lesson, assume some switch out of view has put `.dark` on the root, and watch what that does to your styles. You won't build a working toggle here.

## Two themes, and the naive way to get there

Here's the obvious first attempt. You have a card, and you want it to look right in light and dark, so you pair every color utility with a `dark:` override:

<div data-mark-color="orange">

```tsx /dark:[\w-]+/
<div className="bg-white text-slate-900 border-slate-200 dark:bg-slate-800 dark:text-slate-50 dark:border-slate-700">
  …
</div>
```

</div>

It works. In light mode the card is a white surface with dark text; once `.dark` lands on an ancestor it flips to a dark-grey surface with light text. But look at what this costs, because the cost compounds across a real codebase:

- **Maintenance.** Every component re-derives the dark palette by hand. The card knows its dark colors, the dialog knows its dark colors, the sidebar knows its dark colors. There's no shared source, just the same decision re-made everywhere.
- **Drift.** It's `slate-800` on the card, but was it `slate-800` or `slate-900` on the dialog? Nobody remembers, nobody checks, and the two surfaces quietly diverge. Multiply that by every color on every component and the dark theme slowly stops looking designed.
- **Weight.** Six to ten extra utilities per element. The class string doubles, and half of it is bookkeeping that has nothing to do with what this component *is*.

Step back and notice the deeper problem: the component shouldn't *know* the dark palette at all. `slate-800` is a value, a specific grey, and the moment a component hard-codes it, that component owns a design decision it has no business owning. What the component actually wants to say is "paint me the card surface, with the on-card text color." It should ask for a *role* and let the theme answer. That shift, from naming the value to naming the role, is the whole lesson.

## Naming colors by role, not by value

A <Term definition="A color named for the job it does in the UI (background, card, primary) instead of the value it happens to be. Components reference the role; the theme supplies the value.">semantic token</Term> is a color named for *what it's for* rather than *what it is*. Instead of `white` and `slate-800`, you name roles: `background`, `card`, `primary`, `border`, `muted`. A **theme** is then just one set of values for those role names. Light is one set, dark is another. Same names, different values.

That single move dissolves all three costs at once. The component references `card`, never a grey. There's exactly one place each role's value lives, so nothing can drift. And the class string says only what the element *is*, with no `dark:` bookkeeping at all, because the role doesn't change between themes, only its value does.

The role set this course uses is the one [shadcn/ui](https://ui.shadcn.com) ships, the component library you'll build a real product surface on later in this unit, so it's worth learning the actual names. The most useful idea in the set is the pairing convention: every surface token has a matching <Term definition="The on-surface color: the text and icons that sit on top of a surface. Paired with each surface token so the two are tuned for legible contrast together, in both themes.">foreground</Term> token for the text and icons that sit on it. `card` pairs with `card-foreground`, `primary` with `primary-foreground`, `muted` with `muted-foreground`. The pair is the unit. Because the designer tunes a surface and its foreground *together*, the text stays legible on that surface in both themes, so you never put light-mode text on a dark-mode card by accident.

## Defining the tokens in globals.css

You know the model. Now for the syntax: where the two value sets live, and how `.dark` swaps from one to the other. We'll build it in two steps, the simplest form that works first and then the form you'll actually copy from shadcn, so the production shape arrives as a small, motivated step rather than a wall of boilerplate.

### The simplest form that works

Recall from "CSS-first config in globals.css" earlier in this chapter that a `--color-*` token in `@theme` mints the matching utilities, so `--color-card` is what makes `bg-card` and `text-card` exist. The most direct way to define your palette, then, is to put the light values in `@theme` and let a `.dark` block override the same variable names.

```css title="globals.css"
@theme {
  --color-background: oklch(1 0 0);
  --color-foreground: oklch(0.145 0 0);
  --color-card: oklch(1 0 0);
  --color-card-foreground: oklch(0.145 0 0);
}

.dark {
  --color-background: oklch(0.145 0 0);
  --color-foreground: oklch(0.985 0 0);
  --color-card: oklch(0.205 0 0);
  --color-card-foreground: oklch(0.985 0 0);
}
```

Those colors are written in <Term definition={"oklch(L C H): lightness 0–1, chroma (saturation), hue (0–360). Lightness is perceptually uniform, so the same step looks like the same change at any hue, which makes a dark variant 'lower the L, keep H'."}>OKLCH</Term>, written as `oklch(L C H)`, where the first number is lightness from 0 (black) to 1 (white). You'll see it throughout shadcn's palette, so it's worth one sentence: lightness in OKLCH is *perceptually uniform*, meaning a `0.1` drop looks like the same amount of darkening at any hue. That's exactly the property you want for dark mode, where the move is mostly "lower the lightness, keep the hue." `oklch(1 0 0)` is pure white and `oklch(0.145 0 0)` is near-black; the `0 0` means no chroma and no hue, a neutral grey. That's all you need here. Authoring a real palette is its own lesson later in this unit, "OKLCH, color-mix(), and the alpha syntax"; right now you're only *reading* these values.

Now for the mechanism, which is the load-bearing insight of the entire lesson, so slow down on it. The utility `bg-background` doesn't compile to a fixed color. It compiles to this:

<div data-mark-color="blue">

```css "var(--color-background)"
.bg-background {
  background-color: var(--color-background);
}
```

</div>

It reads a *variable*. So when `.dark` sits on an ancestor and re-points `--color-background` to the dark value, the CSS cascade resolves that variable to the dark color, and `bg-background` follows, along with every other utility that reads the same variable. You don't re-style anything. You change one variable, and everything depending on it re-themes at once. The component's class string never moves because *the class was never the color*: it was always a reference to a variable, and the theme owns the variable.

That indirection is the trick the whole model rests on, and it's worth seeing the swap happen rather than taking it on faith. Scrub through this:

### The production form shadcn ships

Step A works, and you fully understand it now. But the file you'll actually copy from shadcn looks a little different, and the difference is worth understanding rather than pasting blind. In shadcn's `globals.css`, the palette lives in `:root` and `.dark` as *plain* CSS variables (`--card`, not `--color-card`), and a separate `@theme inline` block bridges those plain variables into Tailwind's color tokens:

<AnnotatedCode lang="css" maxLines={18} code={`
@custom-variant dark (&:is(.dark *));

:root {
  --radius: 0.625rem;
  --background: oklch(1 0 0);
  --foreground: oklch(0.145 0 0);
  --card: oklch(1 0 0);
  --card-foreground: oklch(0.145 0 0);
}

.dark {
  --background: oklch(0.145 0 0);
  --foreground: oklch(0.985 0 0);
  --card: oklch(0.205 0 0);
  --card-foreground: oklch(0.985 0 0);
}

@theme inline {
  --color-background: var(--background);
  --color-foreground: var(--foreground);
  --color-card: var(--card);
  --color-card-foreground: var(--card-foreground);
  --radius-lg: var(--radius);
  --radius-md: calc(var(--radius) * 0.8);
  --radius-sm: calc(var(--radius) * 0.6);
}
`}>

    Line one defines what `dark` *means* as a selector. We unpack it in the next section; for now, know it's what tells Tailwind to honor the `.dark` class.
  </AnnotatedStep>

  <AnnotatedStep meta="{3-9}" color="green">
    `:root` is the **light** theme: one plain CSS variable per role, in OKLCH. Note `--card`, not `--color-card`; these aren't Tailwind tokens yet, just plain variables holding the palette. `--radius` is a non-color token along for the ride.
  </AnnotatedStep>

  <AnnotatedStep meta="{11-16}" color="violet">
    `.dark` is the **dark** theme: the same variable names, different values, and nothing else. The entire dark theme is just this block of overrides.
  </AnnotatedStep>

  <AnnotatedStep meta="{18-26}" color="orange">
    `@theme inline` is the bridge. Each Tailwind color token (`--color-card`) is mapped to the plain variable (`var(--card)`). *This* is what makes `bg-card` exist as a utility, and the `--radius-*` rows show non-color tokens deriving sizes from `--radius` with `calc()`.
  </AnnotatedStep>

The one part that deserves a real explanation is the word `inline`, because it's the subtlest thing in the lesson and it's the reason the two-layer shape works at all.

A plain `@theme` defines its tokens as variables on `:root`. So `--color-card: var(--card)` would put that line on `:root`, and `bg-card` would compile to `background-color: var(--color-card)`, a reference to `--color-card`, which is itself a reference to `--card`. The catch is *where* that inner `var(--card)` gets read. CSS resolves it in the scope where `--color-card` lives, on `:root`, where `--card` still holds its light value. So when `.dark` overrides `--card` deeper in the tree, that override never reaches `--color-card`: the hop from `--color-card` to `--card` already happened up at `:root`, in the light scope, before `.dark` was ever in play. Dark mode silently does nothing. The `inline` keyword fixes the scope: it emits the resolved `var(--card)` reference *directly* into the utility, so `bg-card` compiles to `background-color: var(--card)`. Now the `--card` lookup happens on the element itself, where `.dark` *is* in scope, so the cascade picks the dark value and the swap flows through to every utility.

So the rule is short:

:::tip
Reach for `@theme inline` when the token points at another variable you intend to override per theme. Use a plain `@theme` when the token *is* the value (a fixed brand color, a spacing step that never changes by theme).
:::

This raises a fair question: if Step A worked, was it wrong? No. Step A's `--color-card: oklch(...)` also compiles to a `var(--color-card)` read in the utility, which `.dark` can override too, so both forms reach the same cascade. Step B is the canonical one for two reasons. First, it's the exact shape you'll copy from shadcn, so learning it now means the file makes sense when you paste it. Second, it cleanly separates concerns: `:root` and `.dark` are *the palette*, the design, while `@theme inline` is *the Tailwind binding*. A designer can rewrite the whole `.dark` block without touching the binding, and the binding is what lets scalar tokens like `--radius` drive `calc()`-derived utilities. Write Step B.

A few common failures and what causes each:

- **A token isn't showing up as a utility.** It's missing from `@theme inline`, or the `--color-` prefix is wrong. As in the last config lesson, check that the token starts with a real namespace.
- **Colors don't change when `.dark` is present.** Either the `@custom-variant dark` line (next section) is wrong or absent, or you bridged a `var(--…)` token through a plain `@theme` instead of `@theme inline`, so the reference resolved in the wrong scope and `.dark` never reached it.
- **The `--color-` prefix leaked into `:root`.** `:root` and `.dark` hold *plain* variables, like `--card`. The `--color-` prefix appears *only* inside `@theme inline`. Mixing them is an easy typo to make.

## Telling Tailwind what "dark" means

One line in that file does the actual switching, and it's worth isolating so it isn't magic:

<div data-mark-color="blue">

```css "&:is(.dark *)"
@custom-variant dark (&:is(.dark *));
```

</div>

You met `@custom-variant` in the config lesson as the way to define a new variant prefix. This is its most valuable use. It defines `dark` as a variant whose selector is `&:is(.dark *)`, which reads as "this element, when it has a `.dark` ancestor." From that one definition, two things follow at once. It powers any explicit `dark:` utility you still choose to write (we'll see where that's still the right call below). And because the entire `.dark` subtree matches the selector, it's the switch under which your `.dark { … }` variable overrides take hold on every descendant. The directive is what makes Tailwind treat the `.dark` class as meaningful at all.

The class itself goes on `<html>`, and putting it there is the next lesson's job. Here, you only need to hold this in mind: *something* sets `.dark` on the root, and this line is what Tailwind reads to react to it.

You'll see one small variation between sources, so let me name it before it confuses you. shadcn ships `&:is(.dark *)`. Tailwind's own default is `&:where(.dark, .dark *)`. The only difference is specificity: `:where()` contributes zero specificity, while `:is()` contributes its argument's. In practice, `:where()` keeps dark rules maximally overridable, which is the safest default, while `:is()` is what shadcn picked and therefore what you'll paste. Learn `:is()` as the canonical shape for this course, and just know `:where()` exists as the zero-specificity alternative. Why specificity matters is the subject of a later chapter, so for now this isn't a decision to agonize over.

The order of these blocks matters: `@custom-variant` has to be defined before anything uses it, and the bridge has to come after the palette it reads. Put the file in the right order:

<Sequence instructions="Order the blocks of globals.css from top to bottom. Two constraints decide it: a directive must be defined before anything uses it, and the bridge can only read a palette that already exists.">
  <Step>`@import "tailwindcss";` — pulls in Tailwind first, so everything below has it</Step>
  <Step>`@custom-variant dark (&:is(.dark *));` — defines what `dark` means, before any block relies on it</Step>
  <Step>`:root { … }` — the light palette: one plain `--token` per role</Step>
  <Step>`.dark { … }` — the dark overrides: same names, swapped values</Step>

## When per-utility `dark:` still earns its weight

The lesson has spent most of its energy steering you away from `dark:`, so let me steer you back from over-correcting. The token model is for *systematic* color: the surfaces, text, and borders that every component shares. But some adjustments are genuinely one-off, and forcing those into the global palette is its own mistake. A few that legitimately belong inline:

- a shadow that's softer or absent in dark (`dark:shadow-none`),
- a hero gradient that flips direction or palette between themes,
- an image or illustration overlay that only appears in dark.

For these, an inline `dark:` is the right call. Promoting a one-off shadow to a named token would pollute the palette with a single-use value that no other component will ever reference, which is clutter dressed up as a system.

So here's the actual rule, the threshold to keep in your head:

:::tip
If the same `dark:` adjustment shows up in **two** components, it's no longer one-off, so promote it to a token. One use is an exception; two is a pattern.
:::

That's the same rule from earlier in the chapter, where a repeated arbitrary value is a signal the theme should grow, pointed at dark mode specifically.

One more thing the token model buys you: it composes cleanly with the state variants from the previous lesson, "Variants that read DOM state." State and theme are independent axes. A field that turns red on an invalid value writes one theme-agnostic string:

<div data-mark-color="green">

```tsx "aria-invalid:border-destructive aria-invalid:bg-destructive/10"
<input className="border-input aria-invalid:border-destructive aria-invalid:bg-destructive/10" />
```

</div>

`destructive` resolves correctly in both themes, and the `aria-invalid:` gate fires from the DOM state the field already tracks. One class string carries through every theme *and* every state, because both read off the same token set. That's the payoff of keeping color in tokens: the two axes never tangle.

Where does each of these belong: a token, or an inline `dark:`?

## What the model gives you next

Hold onto one sentence: **components ask for a role; the active theme answers; flipping the `.dark` class re-answers every question at once.** That's the entire mental model, and it scales further than light-versus-dark.

<CardGrid>
  <Card title="Non-color tokens" icon="setting">
    The same swap works for `--radius`, shadows, even font sizes; anything can differ by theme or surface. The `--radius-*` rows in `@theme inline` already did this.
  </Card>
  <Card title="Hue-shifted darks" icon="moon">
    A dark theme can shift hue, not just lower lightness. The value sets per theme are fully independent, so it costs nothing extra.
  </Card>
  <Card title="Beyond light and dark" icon="puzzle">
    Brand themes, high-contrast mode, per-tenant theming all extend the same way, via a `data-theme` attribute and more value sets.
  </Card>
</CardGrid>

Now for the handoff. Everything in this lesson assumed `.dark` was already on `<html>`. It isn't yet: nothing here sets it, since you never built a toggle, read the OS preference, or imported React. This lesson is the *what changes*: the colors, the tokens, the directive that makes the swap fire. The next lesson, "Theme switching with next-themes," is the *what flips the switch*: wiring `next-themes`, setting the class before the page paints so there's no flash of the wrong theme, and building the toggle the user actually clicks.
