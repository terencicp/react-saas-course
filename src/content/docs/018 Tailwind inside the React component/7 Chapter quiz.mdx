  <MultipleChoice src="18.1">
    A `<Badge>` is supposed to take a brand color and tint itself. It renders fine in the editor, but in the deployed build every badge comes out with no background at all ‚Äî and there's no error anywhere:

    ```tsx
    const Badge = ({ tone }: { tone: 'red' | 'green' }) => (
      <span className={`bg-${tone}-500`}>{tone}</span>
    );
    ```

    What went wrong?

    <McqChoice correct>
      Tailwind scans your source as plain *text* at build time and never runs your code, so it only sees the literal string `bg-${tone}-500` ‚Äî never `bg-red-500` or `bg-green-500`. Those classes are assembled at runtime, so their CSS is never generated. The fix is a lookup map that contains each full class name as literal source text.
    </McqChoice>
    <McqChoice>
      Template literals aren't allowed in `className` ‚Äî React strips interpolated class strings during the production build for security, which is why it works in dev but not in the deployed bundle.
    </McqChoice>
    <McqChoice>
      The classes exist, but `bg-red-500` and `bg-green-500` aren't on this project's theme scale, so the build drops them. Switching to semantic tokens like `bg-primary` would emit them.
    </McqChoice>

    <McqWhy>This is the dynamic-class trap, and it's nasty precisely because nothing errors. Tailwind generates CSS by scanning your source files as text and emitting a rule for every class string it literally sees ‚Äî it never executes your components. So `` `bg-${tone}-500` `` is, to the scanner, one opaque string it can't match to any utility; `bg-red-500` only ever exists after the code runs, which the scanner never does. No CSS, no background, no error. The rule that prevents the whole class of bug: never assemble a Tailwind class name from a variable. Write each complete class as literal text ‚Äî usually a lookup map keyed by the variable (`{ red: 'bg-red-500', green: 'bg-green-500' }[tone]`) ‚Äî so the scanner can find every name. It has nothing to do with React stripping classes or with the theme scale.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="18.2">
    You add this to `app/globals.css` and reach for `bg-brand` in a component, but the utility doesn't exist ‚Äî IntelliSense doesn't even suggest it:

    ```css
    @theme {
      --brand-color: oklch(0.62 0.19 256);
    }
    ```

    What's the first thing to check?

    <McqChoice correct>
      The namespace prefix. A `@theme` token mints utilities by its *namespace*, and `brand` isn't one Tailwind recognizes ‚Äî there's no `--brand-*` family. Lead with the real namespace: `--color-brand` mints `bg-brand`, `text-brand`, `border-brand`. Same name, same value; only the order is wrong.
    </McqChoice>
    <McqChoice>
      The token needs to be registered as a utility. Defining a CSS variable isn't enough on its own ‚Äî you also have to declare `@utility bg-brand { ... }` so Tailwind knows to generate the class.
    </McqChoice>
    <McqChoice>
      The value. `oklch(...)` colors can't be used in `@theme`; tokens there must be hex or RGB, and the OKLCH value is silently rejected, taking the whole token with it.
    </McqChoice>

    <McqWhy>In v4 a `@theme` token is a CSS variable whose *namespace* deterministically decides which utility family it joins ‚Äî and the name you choose travels straight into the utility. `--color-brand` routes through the `color` namespace and mints `bg-brand`, `text-brand`, `border-brand`, `ring-brand` with no registration step at all. `--brand-color` flips that: `brand` isn't a namespace Tailwind knows, so it builds nothing, and the canonical fix is to lead with the namespace ‚Äî `--color-brand`. So the first debugging question for any missing utility is "does the token start with a real namespace?" You never separately register utilities (that's exactly what the deterministic minting removes), and OKLCH is v4's *default* color space, not a rejected one.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="18.3">
    A reusable `<Button>` sets `px-4` internally. A consumer passes `className="px-8"` to make it wider. With the naive `` className={`... px-4 py-2 ${className}`} ``, both `px-4` and `px-8` end up on the element. Why is "the consumer's `px-8` comes later in the string, so it wins" the *wrong* mental model ‚Äî and what actually fixes it?

    <McqChoice correct>
      Class order in the `class` attribute doesn't decide the winner ‚Äî the cascade does, and among equal-specificity rules the last one *in the generated stylesheet* wins, in an order Tailwind fixes independently of your markup. So the outcome is build-dependent and silent. `cn()` fixes it because `tailwind-merge` *deletes* the losing `px-4`, leaving only `px-8` on the element.
    </McqChoice>
    <McqChoice>
      The model is right about order but `cn()` is still needed for performance ‚Äî concatenation rebuilds the whole string on every render, while `cn()` memoizes the result so the merge runs only once.
    </McqChoice>
    <McqChoice>
      It's wrong because the *first* class in the attribute always wins in CSS, so `px-4` beats `px-8`. `cn()` fixes it by reversing the string so the consumer's class lands first.
    </McqChoice>
    <McqChoice>
      The model is wrong because `px-4` and `px-8` don't actually conflict ‚Äî they set different properties. `cn()` simply keeps both, and the browser adds the two paddings together.
    </McqChoice>

    <McqWhy>Two classes that set the same property both land on the element, and the browser can't apply both ‚Äî it falls back to the cascade. Among equally-specific rules the last one *in the stylesheet* wins, and Tailwind emits its utilities in its own fixed source order, with nothing to do with where the tokens sit in your `class` attribute. So you can't reason about the winner from your own code, and it can shift between builds: silent and unpredictable. Concatenation can only ever *append* ‚Äî it never removes the loser. The fix is `tailwind-merge` (reached through `cn()`), which parses the string as Tailwind, sees `px-4` and `px-8` as one horizontal-padding conflict, and keeps only the last ‚Äî deleting `px-4` outright. It's a correctness fix, not a performance one, and `px-4`/`px-8` genuinely do conflict.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="18.3">
    Here's the corrected `<Button>`. Why must `className` be the *last* argument to `cn()` ‚Äî and what breaks if you move it first?

    <McqChoice correct>
      `tailwind-merge` keeps the *last* of each conflicting utility, so last position is exactly what lets a consumer's `px-8` delete the component's `px-4`. Move `className` first and the component's own defaults land last and win ‚Äî consumer overrides silently stop working.
    </McqChoice>
    <McqChoice>
      Argument order is just a style convention; `tailwind-merge` resolves conflicts by specificity, not position, so `className` wins from anywhere in the call. Putting it last is purely for readability.
    </McqChoice>
    <McqChoice>
      `className` must be last because `cn()` only reads its final argument for consumer overrides and treats every earlier argument as internal-only defaults that can't be overridden.
    </McqChoice>

    <McqWhy>The whole override pattern rests on one fact about `tailwind-merge`: within a conflict group, the *last* class wins. So the call is ordered defaults first, conditional variants next, consumer `className` last ‚Äî and that last slot is what makes the consumer's `px-8` delete the component's `px-4` deterministically, every build. Flip it so `className` comes first and the component's defaults now sit last; they win the merge and clobber whatever the consumer passed, so overrides quietly stop working with no error. It isn't specificity-based (position is the whole mechanism), and `cn()` doesn't treat arguments as privileged tiers ‚Äî every argument is merged the same way, last-wins.</McqWhy>
  </MultipleChoice>

    A teammate is building a settings panel and reaches for `useState` + handlers for each of these. Which ones can be done with a Tailwind variant *instead* ‚Äî no state, no handler ‚Äî because the DOM already tracks the fact? Select all that apply.

    <McqChoice correct>
      A form's outer border turns red while *any* field inside it is invalid.
    </McqChoice>
    <McqChoice correct>
      A "Delete" button hidden inside a card fades in when the card is hovered.
    </McqChoice>
    <McqChoice correct>
      A radio-card highlights when the radio it wraps is the checked one.
    </McqChoice>
    <McqChoice>
      A "Saved!" banner appears two seconds after the server confirms the write succeeded.
    </McqChoice>

    <McqWhy>The reflex is one question: can the DOM already tell me this? If yes, write a variant. A form reading its descendants' validity is `has-[:invalid]:border-destructive` ‚Äî `:has()` lets a parent react to a child. A button reading its card's hover is the `group` / `group-hover:` pair. A label reading the checked state of the radio it contains is `has-[:checked]:`. None of those need a boolean or a handler ‚Äî the browser and the DOM already hold the fact, and the selector reads it. The banner is different in kind: "the server confirmed, then two seconds passed" is a value no selector can match on, so it's genuine React state. Variants replace the state that was only ever *mirroring* a DOM fact ‚Äî not state that tracks something the DOM never knew.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="18.4">
    You write `aria-current:font-semibold` on the active nav link to bold it, but it never applies ‚Äî no error, no style. The element really does carry `aria-current="page"`. What's wrong?

    <McqChoice correct>
      `aria-current` has no built-in shorthand variant (its value can be `page`, `step`, `location`, ‚Ä¶, so there's no single boolean case to target), so `aria-current:` compiles to nothing. You need the arbitrary form: `aria-[current=page]:font-semibold`.
    </McqChoice>
    <McqChoice>
      ARIA attributes can't be styled by Tailwind at all ‚Äî they exist only for assistive tech. You have to mirror the active state into a `data-active` attribute and write `data-active:font-semibold`.
    </McqChoice>
    <McqChoice>
      `aria-current="page"` is a string, but the `aria-current:` variant matches the boolean `true`, so it only fires for `aria-current="true"`. Set the attribute to `"true"` and the shorthand works.
    </McqChoice>

    <McqWhy>Tailwind ships shorthand variants only for the *boolean* ARIA attributes ‚Äî `aria-expanded:`, `aria-pressed:`, `aria-checked:`, and the rest ‚Äî and each targets the `="true"` case. `aria-current` isn't boolean: its value can be `page`, `step`, `location`, `date`, `time`, or `true`, so there's no single case a shorthand could target, and none is generated. Writing `aria-current:` therefore compiles to nothing, silently. The arbitrary form covers it: `aria-[current=page]:font-semibold`. (`aria-invalid` is the other common one with no shorthand, written `aria-[invalid=true]:`.) ARIA *is* styleable ‚Äî that's the whole point, one attribute serving both assistive tech and your styles ‚Äî and you don't mirror it into a `data-*`. Changing the value to `"true"` would be wrong markup for a nav link.</McqWhy>
  </MultipleChoice>

    Across your app a single card surface keeps appearing as `bg-white dark:bg-slate-800`, and the dimmer "last edited" text shows up as `dark:text-slate-400` on two different pages. The chapter's dark-mode model says one of these patterns should stay an inline `dark:` and the other should become a semantic token. Which is which, and what's the rule?

    <McqChoice correct>
      Both should become tokens. The test is *reuse*, not the kind of property: the surface recurs across components and the dimmer text is already duplicated on two pages ‚Äî both are past the two-component threshold, so promoting them gives each one source of truth and stops the darks drifting.
    </McqChoice>
    <McqChoice>
      Both should stay inline `dark:`. Per-utility `dark:` is the senior default; semantic tokens are an advanced opt-in you reach for only once a design system is formally documented.
    </McqChoice>
    <McqChoice>
      The surface becomes a token (backgrounds always do); the dimmer text stays inline, because text color is a one-off styling concern that doesn't belong in the shared palette.
    </McqChoice>

    <McqWhy>The threshold is reuse, full stop ‚Äî not whether it's a background or a text color. A surface shared across components and a dimmer-text color already duplicated on two pages are both past the "if the same `dark:` adjustment shows up in two components, promote it" line: lift them into role-named tokens (`bg-card`, `text-muted-foreground`) so one value drives every instance and the dark values can't drift apart. The carve-out for inline `dark:` is for genuine *one-offs* ‚Äî a shadow that's softer in dark on a single card, a hero gradient that flips on one screen ‚Äî where promoting the value would pollute the palette with something nothing else references. Per-utility `dark:` everywhere is exactly the default this lesson argues *against*, and "backgrounds are tokens, text isn't" is not the rule.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="18.6">
    A theme setup reads the saved theme inside a `useEffect` and adds `.dark` to `<html>` after the component mounts. A user with a dark-set OS reloads and sees a white flash that snaps to dark. Why can't the effect-based approach ever fix this, and what does?

    <McqChoice correct>
      Effects run *after* hydration, which runs after the first paint ‚Äî so by the time the effect sets the class, the wrong (light) pixels are already on screen. You can't repair a paint that already happened; you have to set the class *before* paint, with a synchronous `<head>` script (what `next-themes` injects for you).
    </McqChoice>
    <McqChoice>
      The effect runs fine, but `useEffect` can't mutate `<html>` because it sits outside React's root ‚Äî switching to a `useLayoutEffect` that targets `document.documentElement` sets the class early enough to kill the flash.
    </McqChoice>
    <McqChoice>
      The server renders `.dark` correctly, but the effect *removes* it on mount and re-adds it a frame later. Deleting the redundant effect entirely lets the server-rendered class stand, with no flash.
    </McqChoice>

    <McqWhy>The flash is a timing problem. The server can't read `localStorage` or the OS setting, so it ships HTML with the default theme; the browser paints that first frame *before any React runs*; only then does hydration happen and the effect fire and flip the class, forcing a repaint. Light, then dark ‚Äî that gap is the flash. Anything that reads the theme inside React is structurally too late, because the wrong pixels are already painted. The only fix is to set the class before the body paints, which means a plain synchronous `<head>` script ‚Äî exactly what `next-themes` injects. `useLayoutEffect` still runs after paint, and the server never rendered `.dark` to begin with, so there's nothing for it to wrongly remove.</McqWhy>
  </MultipleChoice>

    <McqChoice correct>
      The pre-paint inline script rewrites `<html>`'s class before React hydrates, so React is guaranteed to find a class the server's HTML never had. The mismatch is intentional, not a defect, and this prop acknowledges it.
    </McqChoice>
    <McqChoice correct>
      It's shallow ‚Äî it silences mismatch warnings for `<html>`'s own attributes only, not its descendants, so it can't mask a real mismatch deeper in the tree.
    </McqChoice>
    <McqChoice>
      It makes React skip hydrating `<html>` entirely, so its class can never conflict.
    </McqChoice>
    <McqChoice>
      Theme-token utilities like `bg-background` need it too, or they won't match between server and client render.
    </McqChoice>

    <McqWhy>`suppressHydrationWarning` acknowledges one specific, intentional mismatch: the inline script deliberately writes `.dark` onto `<html>` before hydration to win the race against first paint, so React inevitably finds a class the server never sent. That's the fix working, not a bug ‚Äî and the prop tells React so. It's also deliberately *shallow*: it quiets warnings for that one element's own attributes and nothing below it, which is exactly why it's safe and why you don't scatter it around. It does *not* switch hydration off ‚Äî React still hydrates `<html>` normally ‚Äî and it has nothing to do with theme-token utilities, which resolve identically on server and client because CSS variables resolve the same in both.</McqWhy>
