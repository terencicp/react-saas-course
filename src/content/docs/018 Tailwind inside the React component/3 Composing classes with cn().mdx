So far you've been a *consumer* of styles. You read utility strings, you defined the tokens underneath them, and every class you wrote landed on a tag you owned. This lesson flips the role: you write a component that *other code* styles. That's the moment a small, specific bug shows up — and the moment a one-line helper called `cn()` becomes the most-typed function in your codebase.

Here's the scenario, and it's about as ordinary as front-end work gets. You build a `<Button>` once and reuse it across the app. Internally it sets `px-4` so the label has breathing room. On one screen — a hero section — you want that button wider, so at the call site you pass `className="px-8"`:

```tsx
<Button className="px-8">Get started</Button>
```

The naive way to honor that prop inside the component is to glue the consumer's string onto the end of your own:

```tsx
const Button = ({ className }: { className?: string }) => {
  return <button className={`inline-flex rounded-md bg-primary px-4 py-2 ${className}`}>...</button>;
};
```

Read that template literal and ask the question this whole lesson answers: the element now carries both `px-4` and `px-8`. **Which padding wins — and can you rely on the answer?** If you can't, you don't have a reusable component; you have a component that styles itself differently depending on details no consumer can see.

By the end of this lesson you'll write the `cn()` helper yourself, understand the two distinct jobs it does, and apply the one ordering rule — consumer `className` *last* — that makes overrides win every single time. It's the exact shape every reusable component in this course is built on.

## Two classes, one property: the bug behind the bug

Let's make the failure concrete, because `cn()` only makes sense once you've felt what it's fixing.

Go back to that naive body. With the consumer passing `px-8`, the string your component hands to the DOM is `inline-flex rounded-md bg-primary px-4 py-2 px-8`. Both `px-4` and `px-8` are valid Tailwind utilities, both set the same CSS property — horizontal padding — and both end up in the element's `class` attribute. The browser can't apply both. It has to pick one.

<CodeVariants>
  <CodeVariant label="Naive (string concat)">

    ```tsx "${className}"
    <Button className="px-8">Get started</Button>

    const Button = ({ className }: { className?: string }) => {
      return <button className={`inline-flex rounded-md bg-primary px-4 py-2 ${className}`}>...</button>;
    };
    ```

    </div>
    The consumer passes `px-8`; the component already set `px-4`. Concatenation just appends — the consumer's string is glued onto the end. **Nothing removes the old `px-4`.**
  </CodeVariant>

  <CodeVariant label="What the DOM gets">
    <div data-mark-color="red">

    ```html "px-4" "px-8"
    <button class="inline-flex rounded-md bg-primary px-4 py-2 px-8">...</button>
    ```

    </div>
    **Both survive.** The cascade — not your string order — decides which one paints. And the cascade's tiebreaker is set by Tailwind, not by you.
  </CodeVariant>
</CodeVariants>

So why can't you just trust that the *later* class in the string wins? Because the order of classes in your `class` attribute has nothing to do with which one the browser applies. CSS resolves a tie between two equally-specific rules by the <Term definition="The CSS rules that decide which of several conflicting declarations applies to an element. Among rules of equal specificity, the last one in the stylesheet wins. Covered in depth in the chapter on the cascade.">cascade</Term>: among rules of equal weight, **the last one in the stylesheet wins** — and Tailwind emits its utilities into that stylesheet in *its own fixed order*, independent of where the tokens sit in your markup. The winner is decided by where Tailwind happened to place the rule for `px-4` versus `px-8` in the generated CSS. You don't control that order, and you can't reason about it from your own code. (The cascade's mechanics are a later chapter's job; here you only need the consequence.)

That makes this bug nasty in two specific ways. It's **silent** — there's no error, because both classes are perfectly valid; nothing is technically wrong. And it's **unpredictable** — the outcome can shift between Tailwind versions or builds, and the consumer who wrote `px-8` has no way to look at their own code and know whether it took. A component whose overrides can't be reasoned about is, for all practical purposes, broken.

That points straight at the fix. Template-literal concatenation is the wrong tool for Tailwind classes for one structural reason: **it can only append, never delete.** To make an override work, you don't need to add the new class — you need something to *remove* the losing one, so only the winner is left on the element. That deleting tool is `tailwind-merge`, and you'll reach it through `cn()`.

:::caution
This is the bug behind the rule you met earlier in this chapter: never build a Tailwind class name by concatenating strings. Now you've seen why — concatenation leaves both conflicting classes on the element, and the cascade, not your code, decides the winner.
:::

Before we hand you the fix, feel the bug in your own hands. The exercise below is *meant* to be frustrating.

## clsx: deciding which classes are present

`cn()` does two jobs, and the cleanest way to understand it is to meet each job on its own first. We'll build it from the inside out. The inner job comes first.

The first job is *deciding which classes are in the string at all.* Real components don't apply a fixed set of classes — they toggle some on and off. A button dims while it's submitting; a tab gets a background when it's active. You need a way to say "include this class, but only when this condition holds." That's <Term definition="A tiny, dependency-free utility that joins truthy class inputs — strings, arrays, and objects whose keys are kept when their value is truthy — into one space-separated string. It is unaware of Tailwind.">`clsx`</Term>.

`clsx` is a tiny (~240 byte), dependency-free utility that takes a mix of inputs and joins the truthy ones into a single space-separated string. It accepts plain strings, arrays, and — most usefully — **objects whose keys are kept only when their value is truthy.** Anything falsy along the way (`false`, `null`, `undefined`, `0`, `''`) is silently dropped. That's its entire purpose: conditional assembly.

<CodeTooltips tooltips={{
  clsx: 'Joins truthy inputs into one space-separated string. Drops false, null, undefined, 0, and empty string.',
  "{ 'opacity-50': isPending }": 'Object form: the key (the class) is included only when its value is truthy.',
}}>
```tsx
clsx('rounded-md', { 'opacity-50': isPending }, isActive && 'ring-2');
```
</CodeTooltips>

This single call mixes all three input shapes — a base string that's always present, an object whose key appears only when `isPending`, and a `&&` expression that contributes `ring-2` only when `isActive`.

That object form is the one to internalize: the *key* is the class, the *value* is the condition. `{ 'opacity-50': isPending }` means "apply `opacity-50` when `isPending` is true." It reads almost like English.

But here's the catch that sets up everything next. `clsx` knows nothing about Tailwind. It doesn't understand that `px-4` and `px-8` fight over the same property — to `clsx` they're just two strings, and it keeps both:

```ts
clsx('px-4 px-8'); // -> 'px-4 px-8'
```

That's the line to remember. **`clsx` is Tailwind-blind.** It decides *which classes are present*, and it does that job perfectly — but it never deletes a conflict, because it can't see one. The thing that *can* is the second job.

## tailwind-merge: deciding which conflicting class survives

The second job is *resolving conflicts* — and for that you need a tool that reads the string the way Tailwind does. That's <Term definition="A Tailwind-aware utility that parses a class string, groups utilities by the CSS property they control, and within each conflict group keeps only the last class, deleting the rest.">`tailwind-merge`</Term>.

`tailwind-merge` parses a class string *as Tailwind*. It knows that `px-4` and `px-8` both control horizontal padding, groups them as a conflict, and within that group keeps only the **last** one — deleting the rest. Its exported function is `twMerge`:

```ts
twMerge('px-4 py-2 px-8'); // -> 'py-2 px-8'
```

`px-4` is gone. `py-2` stays, because it controls a different property and never conflicted. And critically: the winner is the *last* one in the string. That single fact — last conflicting class wins — is the entire foundation of the override pattern you're about to learn. If the consumer's class is last, the consumer wins.

`tailwind-merge` is smarter than a string match, which is what lets you trust it. It understands variant prefixes, so `hover:bg-primary` and `hover:bg-accent` conflict but `bg-primary` and `hover:bg-accent` don't — they apply in different states. It understands responsive prefixes the same way. And it understands shorthand-versus-longhand families: `p-4` (all four sides) and `px-8` (just horizontal) only *partially* overlap, so it keeps both and lets `px-8` win on the horizontal axis alone, rather than naively dropping one:

```ts
twMerge('p-4 px-8'); // -> 'p-4 px-8'
```

One honest caveat, stated once so you recognize it later: `tailwind-merge`'s conflict knowledge is tied to a specific Tailwind version — it has to know which utilities exist to know which ones fight. So the installed `tailwind-merge` must match your Tailwind v4. A mismatch fails quietly, mis-resolving conflicts. You won't hit this in practice: it ships version-pinned in the project scaffold, and current `tailwind-merge` supports v4. Just don't bump one without the other.

## Writing cn(): clsx then tailwind-merge

You now know both jobs. `clsx` decides *which classes are present*; `tailwind-merge` decides *which conflicting one survives*. Run them in that order and you have a function that flattens conditionals and then resolves conflicts in a single pass. That function is `cn()`, and its body is just the two halves nested:

<CodeTooltips tooltips={{
  ClassValue: "clsx's exported input type: string | number | boolean | null | undefined, an array of those, or an object of class-name keys to boolean values. It's everything clsx accepts.",
  twMerge: 'tailwind-merge\'s function: parses the string as Tailwind and keeps only the last of each conflicting utility.',
  clsx: 'Joins truthy inputs into one space-separated string. Drops falsy values. Tailwind-unaware.',
}}>
```ts title="lib/utils.ts"
import { type ClassValue, clsx } from 'clsx';
import { twMerge } from 'tailwind-merge';

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}
```
</CodeTooltips>

Read it from the inside out and it's literally the pipeline you just watched. `clsx(inputs)` runs **first** — flatten the conditionals, drop the falsy values — and its flat string feeds `twMerge` **second**, which resolves the conflicts last-wins. The nesting order *is* the order of the two jobs.

Two things to lock in. First, the signature: `...inputs: ClassValue[]` means `cn()` accepts any number of arguments, in every shape `clsx` accepts — because `ClassValue` *is* `clsx`'s own input type. So you can pass strings, conditionals, objects, arrays, all in one call.

Second, and this is a rule, not a preference: **you don't write this file.** `cn()` lives at `lib/utils.ts` and ships in the project scaffold from the start — this is the convention every shadcn project follows, so every component in this course imports it from the same place rather than redefining it. The <Term definition="A project path alias. The @/ prefix maps to the app's source root, so @/lib/utils resolves to lib/utils.ts regardless of how deeply nested the importing file is.">`@/lib/utils`</Term> alias gets you there from anywhere:

```ts
import { cn } from '@/lib/utils';
```

One last thing so nobody over-thinks it: `cn()` runs while your component renders, and it costs a fraction of a millisecond per call. At the scale of a SaaS UI that never matters. Don't wrap it in `useMemo`; don't try to cache it. Just call it.

## The override pattern: className last

This is the payoff — the pattern you'll reach for in nearly every reusable component you write. Here's the `<Button>` from the start of the lesson, done correctly. Its props extend the native button's, then add one of its own — `isPending`, for a submitting state:

```tsx
import type { ComponentProps } from 'react';
import { cn } from '@/lib/utils';

type ButtonProps = ComponentProps<'button'> & { isPending?: boolean };

const Button = ({ className, isPending, ...rest }: ButtonProps) => {
  return (
    <button
      className={cn(
        'inline-flex items-center rounded-md bg-primary px-4 py-2 text-primary-foreground',
        isPending && 'opacity-50',
        className,
      )}
      {...rest}
    />
  );
};
```

Look at the three arguments to `cn()`, because their order is the entire point. The base classes come first. The conditional (`opacity-50` while the button is submitting) comes in the middle. And the consumer's `className` comes **last.**

That ordering is the lesson's headline takeaway, and it's worth stating as a flat rule: **inside `cn()`, the consumer's `className` is always the last argument.** You already know why — `tailwind-merge` keeps the *last* conflicting class, so last position is exactly what lets a consumer's `px-8` delete the component's `px-4`. Move `className` earlier in the call and you'd flip it: the component's own defaults would land last and clobber the consumer, and overrides would silently stop working. Last position is non-negotiable.

And that closes the bug from the start of the lesson. `<Button className="px-8">` now wins *reliably*, because `cn` deletes `px-4` and keeps the later `px-8` — not by luck of emit order, but deterministically, every build. Let's walk the corrected version against the naive one, then step through the argument order.

Two boundaries worth setting while the pattern is fresh, so you don't over-apply it.

**Reach for `cn()` only when there's something to merge** — a conditional class, or a consumer `className`. A static, unconditional class string doesn't need it. `className="flex items-center gap-2"` is perfectly good as a plain string; wrapping it in `cn()` adds nothing. `cn()` earns its place precisely when something conditional or overridable is in play.

**`cn()` belongs on the render path.** It computes a class string for your JSX, so it runs where the JSX is built — not inside an effect or other non-render code (you'll meet effects in a later chapter; for now just know `cn()` isn't for them). This also extends the debugging reflex from earlier in this chapter — "is the class even in the DOM?" Now there's a new suspect: if a class you wrote isn't showing up, `cn()` may have merged it out, a later conflicting class having deleted it.

## Conditional classes: the four forms

You've already seen `clsx` accept strings, objects, and `&&` expressions. Those are forms of the same idea — "include this class when this holds" — and `clsx` (and so `cn()`) accepts four of them. They all flow through the same flattening step, so the choice between them is purely about which reads cleanest at the call site. Here's the quick reference.

<CodeVariants maxLines={4}>
  <CodeVariant label="&&">
    ```tsx
    cn('rounded-md px-3 py-2', isActive && 'bg-accent');
    ```
    **Use for a single on/off class — the canonical default.** Reads cleanest when one class toggles on one boolean. The left side must be a real boolean (more on that just below).
  </CodeVariant>

  <CodeVariant label="Object">
    ```tsx
    cn('rounded-md px-3 py-2', { 'bg-accent': isActive, 'opacity-50': isPending });
    ```
    **Use when several independent classes each gate on their own boolean.** Keeps the conditions aligned and scannable in one block.
  </CodeVariant>

  <CodeVariant label="Ternary">
    ```tsx
    cn('transition-transform', isOpen ? 'rotate-180' : 'rotate-0');
    ```
    **Use when the choice is genuinely two-sided — this or that, not present or absent.** Picks one of two classes rather than toggling one.
  </CodeVariant>

  <CodeVariant label="Array">
    ```tsx
    cn(['rounded-md px-3 py-2', isActive && 'bg-accent']);
    ```
    **Use when you're assembling classes from separate pieces and want to group them.** Rare; mostly for building a list up programmatically.
  </CodeVariant>
</CodeVariants>

There's exactly one watch-out here, and it carries forward from earlier in this chapter. With the `&&` form, the left side has to be a *real boolean.* `isActive && 'bg-accent'` is fine because `isActive` is true or false. But if the left side is a number or a string, a falsy value leaks *itself* into the output — `count && 'badge'` returns `0` when `count` is `0`, and now `0` is in your class string. For values that aren't already booleans, coerce: `Boolean(count) && 'badge'`, or `value != null && '...'` when you only mean to guard against `null`/`undefined`.

:::tip
The `&&` left side must be a genuine boolean. For numbers or strings use `Boolean(value) && '...'`; for nullable values use `value != null && '...'`. A bare `0` or `''` on the left leaks a falsy value straight into the class string.
:::

Beyond that one rule, the forms are interchangeable. Pick whichever reads clearest where you're standing — they all behave identically.

## Where the merge stops: cn()'s blind spots

`cn()` is reliable, but it isn't magic, and an experienced engineer knows exactly where its guarantee ends.

What `tailwind-merge` **knows** is Tailwind's own surface: the built-in utilities, the variant and responsive prefixes, the shorthand/longhand groups, and arbitrary values that follow Tailwind's conventions. Inside that surface, conflicts dedupe cleanly.

What it **can't see** is anything outside that surface. Hand-written CSS classes (`card-legacy`), classes from a third-party library (`swiper-slide`), arbitrary-property forms that don't map to a known utility group — `tailwind-merge` has no model for what CSS those set, so it can't tell when two of them fight. Two such classes targeting the same property will *both* survive, and you're back to the cascade deciding. The mitigation is the same discipline you've already been following: stay on the utility surface. Bespoke classes are a deliberate, named boundary — not your default.

There is one escape hatch, named here only so you recognize it if you ever meet it: if you ship your own custom utilities (the `@utility` directive you saw when configuring Tailwind in CSS) and need `tailwind-merge` to understand *their* conflicts, `extendTailwindMerge` teaches it new conflict groups. You'll almost never reach for it, and it's not worth a config walkthrough now — just know the name.

Use the check below to make the boundary a recognition skill. For each class pair, decide whether `tailwind-merge` can dedupe it or whether both classes survive untouched.

That's the lesson. Two things to carry out of it. First, **`cn()` is `clsx` then `tailwind-merge`** — flatten the conditionals, then resolve the conflicts, in that order. Second, **the consumer's `className` goes last** so their overrides reliably win. Those two facts are the whole composition story for a single component.

There's one more chapter to this, which you'll meet later. When a component grows *orthogonal* variants — a button that varies by size *and* style *and* state, each independently — the `&&` and object conditionals start to multiply, and you want a cleaner way to declare that whole matrix once. That's what `class-variance-authority` (CVA) does, and it pairs directly with `cn()`: CVA picks the variant classes, `cn()` merges in the override. Every shadcn primitive you'll meet uses the two together. We'll get there; for now, `cn()` and `className`-last are the foundation everything else builds on.

:::
