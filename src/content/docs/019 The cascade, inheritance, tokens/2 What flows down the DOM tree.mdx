Here is a layout you've written a dozen times by now: a `<body>` that sets the font and the text color once, and trusts every element underneath to pick them up.

```tsx title="app/layout.tsx"
<body className="font-sans text-foreground">
  <p>Your trial ends in 3 days.</p>
  <button>Upgrade now</button>
</body>
```

Open this in a browser and the paragraph behaves: it renders in your sans font, in your foreground color. The button does not. It comes out in the browser's default font — usually a slightly different sans, sometimes the system UI font — and on many browsers its text is a flat black or even a faded grey, ignoring the `text-foreground` you set right above it. Same parent, same two utilities, two completely different outcomes.

So there are two questions buried in this tiny snippet, and they turn out to be the same question. Why do the body's styles flow down to the paragraph automatically — and why does this one element opt out? By the end of this lesson you'll have a precise answer to both, and something more useful: a rule for *where every styling utility belongs* on a real project — which ones you set once high up and let fall, and which ones you write on each element by hand. The button's escape, we'll come back to; it's the most instructive part.

## Inheritance is a per-property flag

The mechanism behind "the paragraph picked up the font" has a name: inheritance. And the first thing to get right about it is the thing most people get wrong — inheritance is **not** a global "styles flow downward" behavior. It's a flag, set property by property, baked into the CSS specification itself. For every single CSS property, the spec declares one of two things: *inherited: yes* or *inherited: no*. `color` says yes. `padding` says no. That's it; there's no master switch.

Here's what that flag actually controls. After the cascade runs — the origin, layer, specificity, and source-order contest you saw in the previous lesson, *How the browser picks a winning rule* — some element ends up with *no declared value* for some property. Nobody wrote a rule that touches it. The browser can't leave the property blank, so it fills in a value, and the flag decides which one:

- If the property is **inheriting**, the browser copies the <Term definition="The final resolved value the browser actually uses for an element, after the cascade runs and relative units like em and percentages are resolved to concrete values.">computed value</Term> from the element's **parent**.
- If the property is **non-inheriting**, the browser uses the property's <Term definition="The property's spec-defined default value, used when nothing in the cascade supplies one and the property does not inherit.">initial value</Term> — the spec-defined default for that property, which is the same everywhere and has nothing to do with the parent.

Read that first bullet again, because one word in it is load-bearing: *computed* value of the parent. Not "the value you typed on the parent" — the value the parent actually resolved to after its own cascade and its own unit math. That distinction looks pedantic right now. Hold onto it; it's the entire reason one of the classic CSS bugs near the end of this lesson exists.

The payoff of inheriting properties is **reach**. A value you set on `<html>` or `<body>` doesn't stop at the first child — it falls all the way down, element to element, to every descendant, unless some rule overrides it somewhere on the way. That's why `font-sans` on `<body>` is genuinely enough for your whole app. You're not relying on a wrapper component to forward it. You set it at the top, and inheritance carries it to the deepest nested `<span>` you'll ever write, for free.

### It flows down the DOM tree, not the JSX tree

One caveat to nail down now, because it trips up almost everyone coming to React from the outline diagram in their head. Inheritance walks the rendered **DOM** tree — real parent element to real child element in the page the browser builds. It has nothing to do with how your JSX components are nested.

Those two trees are not the same shape. A `<Card>` component might be ten layers deep in your JSX, wrapping `<CardHeader>` wrapping `<CardTitle>`, but in the DOM it contributes only whatever element it renders at its root — maybe a single `<div>`. Inheritance sees that `<div>`, not the component. So you don't "pass a font down through props," and wrapping a subtree in a styling component does nothing on its own. You set an inheriting property on an **ancestor DOM node**, and it reaches the **descendant DOM nodes** below it. When you reach for inheritance, picture the rendered page, not your component file.

## What inherits, and what doesn't

This is the part to actually learn, and the good news is you don't learn it as two lists to memorize. You learn one heuristic, and the lists fall out of it. Here it is:

> **If a property describes the text, it inherits. If it describes the box — its size, position, background, or visual effects — it doesn't.**

And — this is the part that makes it stick — that split is *designed*, not arbitrary. Think about what each kind of style is for. Text styling is almost always something you want to apply to a whole region at once: pick a font, a color, a line height for an article, and you want every paragraph, list, and inline `<em>` inside it to follow without you repeating yourself. Inheritance is exactly that "apply to everything underneath" behavior. Box geometry is the opposite: no two elements share a width by coincidence, and a child inheriting its parent's `padding` would be a nightmare, not a convenience — every nested box would balloon. So the spec made the text-ish properties inherit and the box-ish ones not, because that's what's useful. Once you see it as a deliberate choice rather than trivia, you can predict the answer for a property you've never met.

Here are the two families filled in. You don't need to commit them to memory — read them as confirmation that the heuristic holds.

<InheritsVsNot />

Two entries on that list deserve a flag, because they earn the senior payoff.

The first is `cursor` and `visibility`: both inherit, even though neither feels like "text." `cursor` inherits so that setting a pointer on a clickable region covers the text and icons inside it without you re-applying it to each child. `visibility` inherits so that hiding a container hides its contents. They're the heuristic's edge cases — keep them in mind, they show up in the exercise.

The second is the one that quietly powers your whole theme: **CSS custom properties inherit.** When you write `--color-foreground` and a `.dark` block on `<html>` overrides it, that override reaches *every* component in the app — not because of any framework magic, but because custom properties are inheriting properties like any other, and `<html>` is the ancestor of everything. That's the substrate your design tokens are built on, and it gets the full treatment later in this chapter, in *Custom properties and the three-tier token model*. For now, just file the one fact: text styles flow from `<body>`, and tokens flow from `:root`.

Now connect both halves back to Tailwind, because this is where the lesson stops being theory. Tailwind is an atomic-utility system — one class, one declaration, on one element. Look at what that demands of each family. The **non-inheriting** half is per-element anyway, so writing `px-4` on each element that needs padding is exactly right; Tailwind never even tries to make box utilities cascade, because they don't. And the **inheriting** half is what lets the utility list on any given element stay short: you set `font-sans text-foreground` once on `<body>`, inheritance carries it everywhere, and you never repeat it. Inheritance is the reason your leaf elements aren't drowning in repeated typography classes. Knowing which family a property is in *is* knowing whether to hoist its utility to an ancestor or keep it local — that's an architecture decision you'll make on every component.

## Telling a property to inherit on purpose: the keywords

So far inheritance has been the *fallback* — what happens when no rule applies. CSS also lets you ask for it explicitly. There are a handful of global keywords you can set on any property to override whatever the cascade decided and force a specific resolution instead. You will write these by hand almost never on a 2026 Tailwind project, but you'll read them in other people's CSS and see them in DevTools, so recognize them.

<div data-mark-color="blue">

<CodeTooltips tooltips={{
  inherit: "Force the parent's computed value — opt this property into inheritance even if it normally wouldn't.",
  initial: "Force the property's spec-defined initial value. Note: that's the CSS default, which is often NOT what the browser visibly shows.",
  unset: 'Do the natural thing: inherit if the property is an inheriting one, otherwise use its initial value.',
  revert: "Roll back to what the browser's user-agent stylesheet would have shown — keep the browser default rather than the spec initial.",
}}>
```css "inherit" "initial" "unset" "revert"
.a { color: inherit; }
.b { margin: initial; }
.c { font-size: unset; }
.d { display: revert; }
```
</CodeTooltips>

</div>

The pair worth not confusing is `initial` versus `revert`. `initial` wipes the property to the *spec's* default, which is frequently not what you see on screen — `display: initial` is `inline`, even for a `<div>`, because the browser's own stylesheet is what made the `<div>` a block in the first place. `revert` rolls back to that browser stylesheet instead, returning the property to what the browser would have rendered. There's also `revert-layer`, which rolls back just to the previous cascade layer — a direct tie to the layer model from the last lesson — but you'll reach for it even more rarely.

On a real project you don't write these raw. You write the utility form, and Tailwind emits the declaration for you:

```tsx "text-inherit" "bg-transparent" "border-transparent"
<span className="text-inherit">…</span>      {/* → color: inherit */}
<div className="bg-transparent border-transparent">…</div>
```

`text-inherit`, `bg-transparent`, `border-transparent`, and the occasional arbitrary `[all:revert]` cover essentially every real case. So when you spot a raw `inherit` or `initial` in a hand-written rule, read it as a small yellow flag — usually a sign someone is fighting a cascade they should have organized with layers instead. Same lesson as last time: reach for structure, not for a keyword that papers over it.

## Why form controls don't listen

Now we can solve the opening mystery. Why did the paragraph pick up the body's font and color while the `<button>` shrugged them off?

Here's the mechanism, and it's not an exception to inheritance — it's inheritance working exactly as specified. The browser ships its own stylesheet, the <Term definition="The browser's built-in default CSS, applied to every page before any of your styles. It sets the baseline look of unstyled elements — margins on headings, the appearance of form controls, link colors.">user-agent stylesheet</Term>, and that stylesheet contains rules that set `font-family` and `color` **directly on form controls** — `<button>`, `<input>`, `<textarea>`, `<select>`. Once there's a directly-declared value sitting on the element itself, the cascade has a winner, and it never falls through to inheritance. Remember from the last lesson that an inherited value is the *weakest* possible source — it's literally the absence of a declaration. Any real declaration beats it. So the body's `font-family` reaches the button, finds the user-agent stylesheet already declared one *on the button*, and loses. The button isn't ignoring inheritance; it simply isn't relying on it, because something already set the value it needs.

Why do browsers do this at all? Historically, form controls were rendered by the operating system as native widgets — a real OS button, a real OS dropdown — and the user-agent stylesheet pins their typography so they keep looking like controls instead of inheriting whatever a page throws at them. It's a deliberate platform decision, not an accident, which is exactly why the fix is also deliberate.

That second tab is the whole resolution: Tailwind's **Preflight** — the reset that loads with `@import "tailwindcss"` — ships exactly one rule, `button, input, select, textarea { font: inherit; color: inherit; … }`, and all it does is *re-enable inheritance* on the controls by setting their font and color back to `inherit`. With Preflight in place, your `<body>` font reaches your buttons after all. Preflight does a great deal more than this, and the next lesson, *Preflight, the deliberately blank canvas*, is where it gets the full treatment — here, the only thing to take away is that the fix for the rebels is literally "turn inheritance back on."

So on a real Tailwind project you don't hand-set fonts on form controls and you don't fight Preflight — it already inherited them for you. And the shadcn `<Button>` and `<Input>` components you'll meet later wrap these controls with their own styling on top, so in practice you rarely touch a bare `<button>` at all. The one time you'd reach for a per-element utility is a stubborn browser-specific holdout that Preflight didn't catch.

## currentColor: the one inheritance trick you'll write by hand

There's exactly one inheritance-powered pattern you'll author directly, and you'll reach for it constantly once you have icons on the page, so it's worth its own moment.

`currentColor` is a CSS keyword that resolves to the **computed value of `color`** on the element where you use it. That's the whole idea — wherever you write `currentColor`, it means "whatever the text color is right here." And because `color` inherits, the text color right here is usually one that flowed down from an ancestor. So `currentColor` lets any property piggyback on the inherited text color and follow it around the page automatically.

The canonical use is **SVG icons**. An inline icon with `fill="currentColor"` paints itself in the text color of wherever you drop it. Put a check icon next to a line of body text and it matches the text with no effort. Set `text-blue-500` on a button, and because both the label *and* the icon read `currentColor`, both turn blue from that one utility — you colored the parent once and the icon came along. This is precisely how Lucide and shadcn icons behave by default, which is why icon color "just works" in that ecosystem without you ever styling the icon directly.

The Tailwind form is `fill-current` (→ `fill: currentColor`) and `stroke-current` (→ `stroke: currentColor`). In practice you'll often write neither, because the icon already defaults to `currentColor`, and you just set `text-*` on the parent. The exercise below makes the connection concrete — make the icon follow the label's color.

## When the page lies: inheritance look-alikes

A few effects *look* like inheritance — a parent changes and the children visibly respond — but the mechanism underneath is something else entirely. Getting these wrong produces real, confusing bugs, because the fix you reach for depends on what's actually happening. Here are the four that bite.

**`opacity` is not inherited — but it sure looks like it.** Set `opacity: 0.5` on a parent and every child fades. So people conclude opacity inherits. It doesn't. What happens is that the parent and its whole subtree get flattened into a single composited group, and *that group* is made half-transparent as one unit. The children fade as a side effect of the group being see-through, not because each child got an opacity value. The tell that proves it: a child *cannot* set `opacity: 1` to become solid again. The group is already half-transparent and the child is painted inside it, so there's nothing for the child to opt out of. Contrast that with a genuinely inherited property like `color`, which a child *can* freely override — that difference is the whole distinction, and it's worth seeing side by side.

<LooksVsActuallyInherited />

**`visibility: hidden` *is* inherited — and a child can reverse it.** Hide a parent with `visibility: hidden` and its children vanish too, because `visibility` inherits. But unlike most inherited values, a child can explicitly set `visibility: visible` and reappear, while the element still takes up its layout space. That reversibility is what makes `visibility` different from `display: none`, which does **not** inherit and removes the element from layout entirely — there's no bringing back a single child of a `display: none` ancestor, because the whole subtree is gone from the render. (When to actually use which for hiding things is a decision the box-model chapter handles; here, only the inheritance facts matter.)

**Transparent backgrounds are passthrough, not inheritance.** A child with no background of its own shows the parent's background — looks inherited. It isn't: `background-color` doesn't inherit. The default `background-color` is `transparent`, so the child is simply *see-through*, and you're looking at the parent's background *through* it. The tell: change the parent's background and what shows changes, but inspect the child and its `background-color` is still `transparent` — it never took the parent's color, it just isn't covering it.

**`em` font-size compounds — prefer `rem`.** Remember the load-bearing word from earlier: a child inherits the parent's *computed* font-size. Now picture nesting elements that each set `font-size: 1.25em`. The first is 1.25× its parent. The second is 1.25× *that* — already 1.56×. The third, 1.95×. Each level multiplies against the computed result of the level above, and the text runs away on you. This is the gotcha the "computed value of the parent" detail was setting up. The fix, and the course default, is `rem`: it's relative to the **root** font-size, not the parent, so it never compounds — and in practice Tailwind's `text-*` scale emits `rem` for you, so you get the non-compounding behavior automatically just by using the scale. The typography chapter goes deeper; the reflex to install now is `rem` over `em`.

Four claims, four mechanisms. Run the round below to lock in which is which — each statement targets one of the tells above.

## The placement rule: typography up, box-model down

Everything in this lesson converges on a single rule, and it's the thing you'll carry to every project you build. Here it is, plainly:

> Put **typography and color utilities on `<body>`** (or a high ancestor). They inherit, so they reach the whole app for free. Put **box-model and layout utilities on each element**. They don't inherit, so they have to be local.

Tokens follow the same logic one level up: your custom-property tokens go on `:root` and `.dark`, because custom properties inherit, which makes them the theming substrate the whole tree reads from — the next-lesson topic.

Look at what that produces in a real codebase. Your root layout sets the typographic baseline once, and your leaf components carry only the box and layout utilities they each individually need — with zero typography repeated, because it already flowed down.

<AnnotatedCode lang="tsx" code={`
// app/layout.tsx
<body className="font-sans text-foreground antialiased">
  {children}
</body>

// components/plan-badge.tsx
<span className="inline-flex items-center gap-2 px-3 py-1 rounded-full bg-muted">
  <CheckIcon className="size-4" />
  Pro plan
</span>
`}>
  <AnnotatedStep meta={`{2} "font-sans" "text-foreground"`} color="blue">
    Typography and color live once, high up, on `<body>`. Both inherit, so every element the layout ever renders below picks up this font and this color for free — you never repeat them.
  </AnnotatedStep>

  <AnnotatedStep meta={`{7} "px-3" "py-1" "rounded-full" "bg-muted"`} color="blue">
    Box-model and layout utilities live on the element itself. None of these inherit, so the badge — and every other leaf — declares the spacing, shape, and background it individually needs, right where it's used.
  </AnnotatedStep>

  <AnnotatedStep meta="{7-10}" color="blue">
    Now read the leaf for what *isn't* there: no `font-*`, no `text-*`. The badge's typography flowed down from `<body>` and never needed restating. That absence is inheritance keeping the utility list short — the whole point of the rule.
  </AnnotatedStep>
</AnnotatedCode>

That's the senior reframe to close on. Inheritance isn't a quirk you memorize for a quiz — it's the lever that decides *where a style lives*. Knowing what flows down is what keeps your utility lists short and your styling decisions in the right place: one element's job gets one element's utilities, and the whole app's typography gets one ancestor's utilities.

## Reading inheritance in DevTools

Same instrument as the last lesson, retargeted. When a style isn't behaving and you suspect inheritance, the browser will show you exactly where each value came from.

The **Styles** panel lists an element's own rules first, then — dimmed, below them — sections headed `Inherited from <selector>`, one per ancestor that contributed an inherited value. So you can see at a glance that the `color` on this element actually came from a rule on `body`, three ancestors up. The **Computed** panel goes one better: expand any property and it traces where the final value resolved from, and inherited properties are flagged with the ancestor they came from.

<InheritedFromPanel />

The move is the same three clicks as before: open Computed, find the property that's misbehaving, read where it resolved from. If it's an inheriting property and it's not taking effect, the panel tells you whether it inherited the value you expected or got overridden somewhere on the way down — same muscle as the cascade trace, pointed at a new target.
