You write `<h2 className="text-2xl">` and the heading renders smaller than you asked for, as if the utility never landed. Or you write a custom `.btn` class, drop a `bg-primary` utility on the same element, and the background just... doesn't take. The element ignores you.

Here is what is actually happening: two rules both set the same property on that element, and exactly one of them gets to paint. There is no ambiguity from the browser's side — it ran an algorithm, picked a winner, and moved on. The reason it *feels* like a mystery is that you have been writing the inputs to that algorithm since the last chapter without ever seeing the algorithm itself. You write `@layer` blocks, you tack a `!` onto a utility, you wrap class strings in `cn()` — every one of those is a move in a game whose rules you have not been shown.

This lesson shows you the rules. By the end you will be able to take any "why is this style not applying?" moment, trace it to the *exact* step that decided it, and fix it the way an experienced engineer would — which, you will see, is almost never by reaching for `!important`.

## The cascade is a four-gate waterfall

When two or more rules set the same property on the same element, the browser resolves the conflict with the <Term definition="The algorithm the browser uses to resolve which CSS declaration wins for each property on each element.">cascade</Term> — the algorithm that picks, for each property on each element, which value paints. It runs four steps, and the order they run in is the whole game:

1. **Origin and importance**
2. **Cascade layer**
3. **Specificity**
4. **Source order**

The trap is to read that as four factors the browser weighs together, like a scorecard. It is not. It is a *waterfall*. The browser collects every <Term definition="A single property: value pair inside a CSS rule.">declaration</Term> — every `property: value` pair from every rule — that targets this element and sets this property, then drops them through the four gates in order. The first gate that produces a single winner stops the process. Every gate after that only exists to break the ties the gate above it left behind.

So if gate 1 already singles out a winner, gates 2, 3, and 4 never run. If gate 1 leaves a tie and gate 2 breaks it, gates 3 and 4 never run. Specificity — the thing you were probably taught is *how* CSS conflicts get resolved — is gate 3. It only ever sees the conflicts that survived two earlier gates untied.

<CascadeWaterfall />

Hold onto the shape of that picture, because it reframes the entire job of debugging CSS. You are not weighing four things. You are asking one question at a time, in order, and stopping the moment one of them answers.

And here is the part that matters most for the stack you are building on. On a Tailwind v4 project, the gate that decides almost every real conflict is **gate 2, the layer**. The classic symptom — a piece of custom CSS silently beating a utility — is a layer problem nine times out of ten. The other common one is **gate 4, source order**, when two conflicting utilities slip through without being deduplicated. Specificity wars, the thing that fills Stack Overflow threads, are *rare* on a Tailwind codebase, because utilities are nearly all single-class selectors with identical specificity — there is nothing for gate 3 to adjudicate. Plant that now. The rest of the lesson pays it off.

A note on gate 1's cast of characters before we go deeper. "Origin" means *who wrote the rule*: the browser ships its own default styles (the <Term definition="The browser's built-in default styles.">user-agent stylesheet</Term> — that is where an unstyled `<h1>` gets its big bold look), the user can supply their own stylesheet, and then there is you, the author. In a SaaS app, author styles are essentially everything — your CSS and Tailwind's. Origin barely matters on its own. What makes gate 1 the *first* gate is `!important`, which can invert the normal pecking order. We will come back to exactly how, once the layers are in place to make it make sense.

## Tailwind's four layers, and the trap of leaving CSS unlayered

Gate 2 is where this lesson earns its keep, so let's open it up.

A <Term definition="A named bucket of CSS rules, declared with @layer. Layer order outranks specificity.">cascade layer</Term> is a named bucket you sort rules into, declared with `@layer`. The rule governing them is simple and absolute: **a declaration in a later-declared layer beats a declaration in an earlier-declared one, no matter how specific either selector is.** Layer order is decided *before* specificity is even consulted. That single sentence is the key that unlocks most Tailwind debugging.

You have already been generating layers without writing them by hand. The single `@import "tailwindcss"` line you added to `globals.css` in the last chapter emits four layers, in this declared order:

<LayerStack />

Walk up that stack. `theme` and `utilities` are Tailwind's; you do not author into them directly. `utilities` is declared last of the four, which is exactly why utilities are designed to win — a `utilities` rule like `text-2xl` beats a `base` rule like Preflight's `h2` reset *regardless of specificity*, because the layer gate fired first and `utilities` is the later layer. That is the answer to the opening heading bug, and it is the entire point of utilities being able to override your element baselines per element.

So far this is the system working. Now the trap.

:::caution
Any rule you write at the top level of `globals.css`, *outside* an `@layer` block, is **unlayered** — and unlayered CSS beats all four layers. A bare `h2 { font-size: 1rem }` sitting on its own in your stylesheet will silently win over `text-2xl`. This is the single most common Tailwind styling bug, and it is completely invisible until you know to look for it.
:::

Why does unlayered win? The cascade treats unlayered declarations as if they belong to a layer that was declared *after* every named one. In Tailwind v4 specifically, that places them above `utilities` — so a stray rule, or a third-party CSS file you `@import` without wrapping, overrides your component-layer and utility-layer styles alike. Nothing in your code says "this should win." It wins by accident of where it sits.

The senior reflex that falls out of this is a one-liner you should adopt permanently: **every custom CSS rule names its layer.** Concretely, you author into two of Tailwind's layers:

- **`@layer base`** for global, element-level styling — resets and element baselines. Your `h2` defaults, your `code` and `kbd` chip styles, your default link underline. Putting them in `base` means a utility can still override them per element, because `utilities` sits above `base`. That is what you want: a sensible default that any utility can beat.
- **`@layer components`** for the rare bespoke component that utility classes genuinely cannot express. The threshold matters here — reach for it only when no composition of utilities works. On a React codebase, most "components" are React components holding utility class strings, not CSS classes. You will write into `components` far less often than you think.

Here is the bug and its fix side by side. Picture a `.btn` class that a teammate styled in CSS, used in a component that also accepts utility overrides from its caller:

<CodeVariants>
  <CodeVariant label="The bug — unlayered">
    ```css del={3-5}
    @import "tailwindcss";

    .btn {
      background: var(--color-secondary);
    }
    ```
    **The override silently loses.** The component renders `<button className="btn bg-primary">`, expecting the caller's `bg-primary` to win — but `.btn` is unlayered, so it sits *above* the `utilities` layer where `bg-primary` lives. The button paints `secondary` and the override is ignored.
  </CodeVariant>

  <CodeVariant label="The fix — layered">
    ```css ins={3,7}
    @import "tailwindcss";

    @layer components {
      .btn {
        background: var(--color-secondary);
      }
    }
    ```
    **The same rule, back in the system.** Moving it into `@layer components` drops it *below* `utilities`, so `bg-primary` wins the layer gate and the caller's override takes effect again. No specificity changed, no `!important` — the rule simply rejoined the cascade.
  </CodeVariant>
</CodeVariants>

Read that fix carefully, because it is the template for almost every cascade bug you will hit on this stack. Nothing about the *selector* changed. Nothing got more specific. You did not reach for `!important`. You took a rule that had floated out of the layered system and put it back where it belongs — and the system resumed working on its own.

One last watch-out on layers, because it bites silently: the *declaration order* of layers is fixed by the `@layer` statement at the top of Tailwind's generated CSS, and that order is anchored to where your `@import "tailwindcss"` sits. If you `@import` a third-party stylesheet in the wrong place relative to it, you can shift where those external rules land in the cascade. Keep your imports deliberate.

## Specificity, demoted

Gate 3. The one you may have been taught was the main event, here taking its actual place — third in line, and on this stack, mostly idle.

Specificity is a four-part tuple, read left to right:

**(inline styles, ID selectors, class / attribute / pseudo-class selectors, element / pseudo-element selectors)**

You compare two selectors slot by slot from the left; the first slot where they differ decides it, and a higher number there wins outright no matter what the slots to its right say. A single utility class scores `(0, 0, 1, 0)`. An `#header` ID selector is `(0, 1, 0, 0)` — and because the ID slot sits to the left of the class slot, that one ID beats *any* number of classes, even `(0, 0, 50, 0)`. An inline `style={{ }}` attribute is `(1, 0, 0, 0)`, which outranks everything built from IDs or classes. The universal selector `*` contributes `(0, 0, 0, 0)` — it adds nothing.

<SpecificityTuple />

Now the demotion, stated plainly: **specificity only breaks ties *within* a single layer.** Across layers, the layer gate already chose a winner before specificity got a turn. So on a Tailwind project, specificity earns its keep almost entirely *inside* `@layer base` — the one place where your own element rules and Preflight's element rules coexist and might genuinely clash. Between a utility and your custom CSS? The layer gate fired first; specificity never ran. This is why "raise the specificity" is almost always the wrong instinct here. You would be tuning a gate that the conflict already skipped.

There is one tool worth recognizing in this gate, because Tailwind uses it constantly: **`:where()`**, the specificity-zero wrapper. Anything you put inside `:where(...)` contributes exactly zero to the tuple, no matter how complex the selector inside it is. `:where(.dark, .dark *)` matches dark-mode elements but scores `(0, 0, 0, 0)` for the wrapper itself. That is the entire point — a framework can match the elements it needs to without inflating specificity, which means the utility classes you stack on top still win their ties cleanly. If frameworks matched with normal selectors, every utility would be in a specificity fight with the framework's own rules. `:where()` defuses that.

This connects to a line already living in your `globals.css`. Last chapter you wrote a dark-mode variant — the shipped course code uses the `:is(.dark *)` form. Worth knowing precisely: `:is()` and `:where()` look similar but differ in exactly one way — `:where()` always scores zero, while **`:is()` takes the specificity of its most specific argument**. So `:is(.dark *)` contributes the specificity of `.dark`, a single class. In practice that is completely fine for the dark variant, because the `.dark` toggle lives way up on the `<html>` element and the token swap it drives never gets into a specificity duel with your utilities. Tailwind's own docs happen to show the `:where(.dark, .dark *)` form, which is the textbook specificity-zero variant. The distinction is not something you need to act on — but now you can read either form and know exactly what it scores.

:::note
You do not write `:where()` in your own application code. You *recognize* it — in framework output, in DevTools — and understand why it is there. That recognition is the deliverable, not a new authoring habit.
:::

One more consequence of the tuple, and it is a temptation worth naming. Inline `style={{ }}` scores `(1, 0, 0, 0)` and therefore beats every class-based rule you could write. It is tempting to use that as a hammer: can't get a style to apply? Slap it inline and force the win. Don't. That is the exact same anti-pattern as `!important`, just spelled with a different syntax — you would be muscling past the cascade instead of fixing why your rule lost. (Inline `style` does have a legitimate home, for genuinely *dynamic* values like a CSS custom property computed at runtime — but that is a different chapter's topic. Using it to win a static conflict is the smell.)

Score these four for yourself. Read each selector slot by slot and pick its tuple — the `:where()` line is the one that catches people.

The last two are the traps. `a:hover` is `(0, 0, 1, 1)` — the `:hover` pseudo-class lands in the *class* slot, not the element slot, so it's one class and one element, not two elements. Miss the `:where()` rule and you'd guess `(0, 0, 2, 0)` and expect it to win a tie it actually draws.

## `!important` is a smell, and the two times it isn't

Back to gate 1, and the reflex this lesson most wants to install.

`!important` is adjudicated at the very first gate — before layers, before specificity, before source order. That is precisely what makes it a bug smell on a 2026 stack. When you write `!important`, you are not winning the cascade; you are *bypassing* it. You are switching off the predictable four-gate machine and asserting a winner by force. And on a Tailwind project, the thing you are usually forcing past is not some genuinely unwinnable fight — it is a rule that is simply in the wrong layer, or in no layer at all.

Trace the causation, because it is the whole argument: `!important` is what people reach for when they don't understand the layer gate. The custom rule loses, they don't know *why* it loses, so they hit it with the biggest hammer available. Once you understand that the rule lost at gate 2 and that moving it to the right layer fixes it at gate 2, the reason to reach for `!important` evaporates. The senior fix for a losing rule is **put it in the right layer**, not bang it.

There is also a genuinely subtle reason `!important` makes a layered codebase *harder* to reason about, worth knowing as a recognition even though you should not lean on it: with cascade layers, **`!important` inverts the layer order.** Normally a later layer beats an earlier one. But for `!important` declarations, an *earlier* layer beats a later one. So in a layered codebase, `!important` doesn't just jump the queue — it flips the queue's direction, which is a second, sneakier way for it to produce surprises. Important inversion is real, but it is rare, and you do not want to be the one who introduced it.

So is `!important` ever correct? Yes — narrowly, so that "never" stays an honest rule and not a lie you'll catch this course in:

1. **Overriding third-party inline styles you don't control.** A widget that ships with `style="..."` baked onto its root element scores `(1, 0, 0, 0)` and beats every class you can write — and you can't edit its markup. An `!important` declaration outranks even inline styles, so this is the one place it is the right tool. Tailwind gives you the idiomatic spelling: the `!` modifier you met last chapter, `bg-white!`, emits an `!important` utility. This is the SaaS-relevant carve-out — third-party embeds are a fact of life.
2. **User-stylesheet accessibility overrides.** A user's *own* stylesheet forcing high contrast or larger text uses `!important` to override author styles. That is the user's CSS, not yours, and it is out of scope for the app you build — named here only so the picture is complete.

Notice both carve-outs share a shape: `!important` is correct only when the thing you are overriding is *outside your codebase* and you genuinely cannot reach it any other way. The moment the rule you are fighting is your own, `!important` is the wrong answer and a layer move is the right one.

## Source order, and why `cn()` makes it trustworthy

Gate 4, the final tiebreaker — and the one your existing tooling already neutralizes for you.

When two declarations survive all three earlier gates dead even — same origin and importance, same layer, same specificity — the browser falls back to source order: **whichever declaration appears later in the generated stylesheet wins.** For utilities this is the *common* case, not an edge case. `px-4` and `px-8` are both single-class selectors, `(0, 0, 1, 0)`, both living in the `utilities` layer. They tie on every earlier gate. So the winner comes down entirely to which one Tailwind happened to emit last in the compiled CSS.

And here is the trap that follows, the one that genuinely surprises people:

:::caution
The order of classes in your `className` string does **not** decide the conflict. `className="px-8 px-4"` and `className="px-4 px-8"` render *identically*. The winner is fixed by Tailwind's emission order in the compiled stylesheet — which has nothing to do with the order you typed the classes.
:::

Read that twice, because it inverts the intuition almost everyone arrives with. You would swear the last class in the string wins. It does not. The string order is irrelevant; the *stylesheet* order is what counts, and that is determined by Tailwind's build, not by you. Which means it is both invisible (you can't see the emission order) and build-dependent (it can shift). A genuinely nasty class of bug.

You already own the fix. Last chapter you met `cn()` — `clsx` composed with `tailwind-merge`. Its whole job, for this purpose, is to resolve utility conflicts *before they ever reach the cascade*. `cn('px-8', 'px-4')` returns just `'px-4'` — `tailwind-merge` knows `px-8` and `px-4` target the same property, so it drops the loser and emits a single surviving class. Only one `px-*` reaches the DOM. With nothing to tie, the source-order gate has nothing to adjudicate; the question that caused the bug stops existing.

So the senior reflex for utility conflicts is not to reason about source order at all. It is: run them through `cn()` and the ambiguity is gone. You do not memorize emission order; you delete the conflict. (How `cn()` and `tailwind-merge` decide which class survives is the previous chapter's territory — here, all that matters is that it makes gate 4 a non-issue.)

## Reading the cascade in DevTools

Everything so far is the model. This is the instrument that lets you *read* the model on any real element — and it is one repeatable move, not a feature tour. Learn it once and you run it for the rest of your career.

Two panels matter. The **Styles** panel shows every rule targeting the selected element, listed in cascade order, with the declarations that *lost* struck through. A line through a declaration is the browser telling you, in the plainest possible terms, "this rule set this property and a different rule beat it." The winner is the one that is not crossed out. That alone resolves a huge fraction of "why isn't this applying?" questions at a glance.

The **Computed panel** — <Term definition="The DevTools view that shows each property's final resolved value and the rule that supplied it.">the authoritative view</Term> — is where the durable skill lives. It shows the *final, resolved* value of every property — and when you expand a property, it reveals the **trace**: which specific rule supplied the winning value, and, in modern Chrome and Firefox, *which cascade layer that rule came from*. The Cascade Layers indicator (shipped in Chrome 99 and Firefox 97, universal long before 2026) labels each rule with its layer, so the distinction at the heart of this lesson — "is the winner unlayered, or is it `@layer utilities`?" — is right there to read. If a declaration carries `!important`, the panel flags that too.

<ComputedPanelTrace />

Here is the move, as a procedure to keep. Whenever a style won't apply:

1. Select the element and open the **Computed** panel.
2. Expand the misbehaving property to see its **trace**.
3. Read **which rule won and which layer it sits in.**

If the winning rule is unlayered, or sits in a later layer than you expected, you have found your bug — and you already know the fix is a layer move, not `!important`. Three clicks, no guessing.

## Closing the loop

Take it back to the heading you opened with — `<h2 className="text-2xl">` rendering at the wrong size. You now have the whole loop. The symptom is a wrong font size. You select the `<h2>`, open Computed, expand `font-size`, and read the trace. One of two things is true. Either `text-2xl` *is* winning (it's in `utilities`, the later layer, exactly as designed) and the size you're seeing really is `text-2xl` — in which case the cascade is working correctly, and the surprise is that the heading had no large default to begin with because Preflight stripped it, which a later lesson in this chapter takes up. Or some unlayered rule of your own is sitting above `utilities` and stealing the win — in which case the deciding gate is **layer**, and the fix is to move that rule into the layer it belongs in. Either way, the trace told you which gate decided, and you reached for a layer, not a bang.

That is the mental model to walk away with. The cascade is a four-gate waterfall — origin and importance, then layer, then specificity, then source order — and the browser stops at the first gate that leaves a single winner. On a Tailwind v4 app the **layer** gate decides almost everything, so the experienced engineer's fix for a losing rule is to name its layer, not to escalate specificity or drop an `!important`. And when you can't reason it out by eye, the Computed panel's trace tells you which gate decided in three clicks. The whole thing is mechanical. Nothing about it is a mystery anymore.
