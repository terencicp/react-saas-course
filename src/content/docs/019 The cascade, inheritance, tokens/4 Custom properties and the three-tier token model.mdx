You have spent a whole chapter theming with Tailwind, and there are three questions about it the chapter never actually answered — questions that should be nagging at you.

Can JavaScript change a token *while the page is live* — and if it can, does React need to know? When you flipped the theme in the last chapter, adding `.dark` to `<html>` repainted the *entire* application in one stroke — how does a single class on one element reach every color on the page? And the brand-blue you defined back when you set up `@theme`: could a customer set their *own* brand color, the way every real SaaS lets you?

Each of those questions has the same answer, and you have been staring at it the whole time without seeing it. It is the thing a `var(--color-primary)` reference actually points at. By the end of this lesson, the `var()` you have been writing on faith stops being magic: it becomes something you can read, write, scope to one section of the page, and architect into a system that survives a rebrand.

One thing this lesson is *not*: it is the machinery, not the color. What makes a *good* color value, and how animation interpolates between values, come later. Here we build the substrate every token sits on, and the pattern for organizing tokens once you have more than a handful.

## A custom property is a binding, not a constant

Here is the primitive in one line. Any CSS property whose name starts with `--` is a <Term definition="A CSS property whose name starts with two dashes. You declare it on a selector and read it back with var(). The spec name; CSS variable is the colloquial name for the same thing.">custom property</Term> — colloquially a "CSS variable," same thing. You declare it on a selector, and you read it back anywhere with `var(--name)`.

Let's build it up slowly, because the obvious mental model — "a custom property is a constant, a named value I can reuse" — is *wrong*, and getting from there to the right model is the whole point of this section.

Start with the underwhelming version. Declare a custom property on a parent, read it on a child:

```css
.toolbar {
  --gap: 1rem;
}

.toolbar button {
  gap: var(--gap);
}
```

So far this looks exactly like a constant. You named a value, you reused it. If that were all custom properties did, they would be a mild convenience and this section would be over. Two things make them something else entirely.

**First: they inherit.** Back when you learned what flows down the DOM tree, custom properties were named as members of the inheriting family — set one on an ancestor and every descendant sees it — but the consequence was deferred to here. This is that consequence. A custom property declared on `<html>` is visible to `var()` in *every* element on the page, because it inherits all the way down the tree. You don't have to re-declare it; it flows.

**Second: they are live.** This is where "constant" dies. When you change the declared value of a custom property — at any point, by any means — every `var()` that reads it recomputes, with no reload and no rebuild. A constant is resolved once and frozen. A custom property is resolved *at the moment it is used*, against the <Term definition="The final value the browser settles on for a property on a given element, after the cascade and inheritance have run. var() reads this value at use time, so changing it anywhere updates every reader.">computed value</Term> of the property on that element. Change what is under it, and every reader follows.

That is the reframe the entire lesson hangs on: **a custom property is a binding, not a constant.** `--color-foreground` is not "the value `oklch(...)`." It is a live reference — a name that the browser re-resolves at every use site, every time the value beneath it changes, anywhere in its subtree.

You cannot read your way to believing this. You have to feel it. So slide this:

Now the payoff you have been owed since the last chapter. Every token you wrote in `@theme` *is* one of these. When you defined `--color-brand`, Tailwind put it in `:root` as a plain custom property, and every utility that consumes it — `bg-card`, `text-primary`, `border-border` — compiles down to a `var()` read. The `--{namespace}-{name}` rule you learned, where `--color-primary` mints `bg-primary` and `text-primary` and `border-primary`, was minting *bindings* the whole time. You have been writing custom properties for a full chapter; you just didn't have the word for what they were.

:::note
This is also the subtlety behind `@theme inline`, which you met last chapter: `inline` substitutes the token's *written value* straight into the utility, so a literal (`--color-brand: oklch(...)`) freezes the utility and a runtime override can't reach it, while a `var()` pointing at a per-theme variable (shadcn's `--color-card: var(--card)`) keeps it live — `inline` is precisely what makes the utility resolve `var(--card)` on the element, so the `.dark` swap still flows through. The rule that matters here is the constant underneath: as long as a utility ends in a `var()` read, a runtime override reaches it.
:::

## Overriding tokens down the tree

You now have the binding. The next move is *changing what it points at* — and the cleanest way is also the one you have already shipped.

The pattern is: **declare the default on `:root`, override it on a descendant selector.** Watch what that buys you.

Three selectors, one token, re-declared. The override flows down by inheritance, and every `var(--color-primary)` below the selector re-resolves to the new value. The dark tab is not a new idea — it is *precisely* the dark-mode swap you built last chapter, finally explained. You re-declared the semantic tokens under `.dark`, and because custom properties inherit and live, the whole subtree repainted. You had already shipped a subtree override; here is the general shape it belongs to.

And the shape generalizes past dark mode. A marketing hero that wants a punchier accent? `.marketing-hero { --color-primary: ... }`. A multi-tenant app where Acme gets their brand and Globex gets theirs? `[data-org="acme"] { --color-primary: ... }`. Same mechanic, different selector — declare the token nearer the element, and the nearer declaration wins for that subtree. Per-tenant branding is one of the clearest reasons a real SaaS leans on this; you will build the organization layer that drives `[data-org]` much later in the course, so treat it as a use case you now understand, not a feature to wire up today.

Here is why this is the reflex an experienced engineer reaches for, instead of overriding colors element by element. *One* declaration re-themes an *entire* subtree, and the components inside it change **nothing** — they keep rendering `bg-primary`, blissfully unaware that the binding under them now points somewhere new. That only works because of two facts you have now assembled: utilities reference `var(--token)` (the plain-`@theme` link from the last section), and custom properties inherit (the reach from the section before that). Override-down-the-tree is those two facts paying off together.

## Reading and writing tokens from JavaScript

CSS can override a token. But the first question that opened this lesson was about *JavaScript* — changing a token while the page is live. That is a different channel, and it answers the "does React need to know?" question with a hard, load-bearing *no*.

The DOM gives you the imperative twin of the CSS you just wrote. Three calls:

```js
element.style.setProperty('--brand', value);
getComputedStyle(element).getPropertyValue('--brand');
element.style.removeProperty('--brand');
```

`setProperty` declares a custom property *on the element you call it on* — so its scope is exactly that element's subtree, the same way a CSS selector's scope is the elements it matches. Call it on `document.documentElement` and you have changed `:root`, repainting the whole page; call it on a deeper node and you have scoped the change to that branch. Writing the token from JS on an element is the imperative version of declaring that token on a selector in CSS. `getPropertyValue` reads it back. `removeProperty` deletes your override so the inherited value shows through again.

Now the point that matters more than the syntax. **These writes update pixels, not React.** A `setProperty` call repaints through the cascade with *no re-render* — React never finds out it happened. That is simultaneously the feature and the trap.

It is the feature because a color-picker can drag butter-smooth, writing the property on every pointer move, without thrashing the React tree sixty times a second. It is the trap because if a *component* needs to read that value to make a decision, the value has to also live in React state — the custom property is a one-way visual output, not a place React reads from. Beginners discover `setProperty`, write a value with it, and then can't understand why the component that "uses" the color didn't re-render. It didn't re-render because you never told React anything; you told the browser to repaint.

The senior shape threads both: a theme color-picker writes the property live for instant visual feedback *and* commits the chosen value to React state (and persistence) when the user releases. Visual channel for the drag, state channel for the truth.

So the reflexive `.trim()` is usually redundant for a single-token value — the browser already trimmed the edges. The trap is the inverse one: assuming a space *survives* and writing a comparison that "defends" against whitespace that was never there. Where whitespace genuinely bites is *inside* a multi-part value, and `.trim()` can't reach that — you normalize it yourself.

Two smaller edges round this out. Custom-property names are **case-sensitive**: `--Brand` and `--brand` are two different properties, and mixing the casing is a silent miss. And `var()` takes an optional fallback — `var(--x, 1rem)` uses `1rem` when `--x` isn't set — but in a project that *owns* its tokens you rarely need it; the fallback exists mostly for consuming tokens you didn't define and can't guarantee. Name the syntax, reach for it almost never.

One more use case worth flagging, because it is the same disease you already cured once. A multi-tenant brand color usually comes from server data, and if JavaScript writes it *after* React hydrates, the page paints once in the default theme and then snaps to the tenant's color — a flash of the wrong styling, the same <Term definition="Flash of unstyled (or wrong-styled) content. The page paints with default styling for a beat before a script corrects it. Cured by writing the value in an inline script in <head>, before first paint.">FOUC</Term> the `next-themes` setup avoided last chapter. The cure is identical: a tiny inline `<script>` in `<head>` that calls `setProperty` *before* the first paint, so the page never renders in the wrong theme. Same problem, same fix — you have met it before, so this is recognition, not new ground.

## Writing tokens from React with `style={{}}`

The JS API is the imperative path. Inside a component, you almost always want the *declarative* one — and React gives custom properties a first-class shape.

You write them in the `style` prop, exactly like any other style, just with the `--` name as the key:

```tsx
<div style={{ '--card-padding': '1.5rem' }} className="p-[var(--card-padding)]">
  {children}
</div>
```

Two halves working together. The inline `style` *declares* the custom property on this element — the React equivalent of `setProperty`. The `p-[var(--card-padding)]` utility *reads* it. And the value can differ on every instance, which is exactly the thing static utilities cannot do: `p-6` is `p-6` on every element forever, but `--card-padding` can be `1.5rem` here and `3rem` there, driven by a prop.

That bracket form — `p-[var(--name)]`, `bg-[var(--name)]`, `w-[var(--name)]` — is Tailwind's **arbitrary-value** escape hatch. It is how you read a custom property that isn't on a `@theme` namespace, including any runtime variable you set with inline `style`. Use it honestly: it is *rare* in component code, because almost everything you style comes from a semantic token you already have. It earns its place for genuine one-offs — a single element whose width or transform is driven by a runtime value — and nowhere else. The semantic token is the rule; the arbitrary `var()` is the exception.

This is the right moment to settle an apparent contradiction. When you learned about the cascade, the rule was: never reach for inline `style` to win a *static* conflict — that is a smell, because it sidesteps the whole layer system. This is the legitimate inverse. Inline `style` is the *correct and only* tool for a **dynamic, per-instance value**, because there is no static class that can carry a value computed at render time. The two rules don't clash: static conflict, use the cascade; per-instance value, use inline `style`. The deciding question is whether the value is known ahead of time or computed per instance.

## Designing the token system: primitive, semantic, component

You can now read a custom property, write it from CSS, JavaScript, and React, and scope it to any subtree. The substrate is yours. The remaining question is the architectural one — the part that is actual senior work: *how do you organize hundreds of these so a design system stays changeable as the product grows?*

The answer the industry converged on is three tiers, and the cleanest way to understand them is to motivate each by a problem the previous tier can't solve. The named design decisions you are organizing have a cross-tool name — <Term definition="A named design decision (a color, a spacing, a radius) stored so it can be referenced and changed in one place. The cross-tool concept behind CSS custom properties used for design.">design tokens</Term> — and the three tiers are how you stop them from rotting as the system grows.

**Tier one — primitives: the raw palette.** `--gray-50` through `--gray-950`, `--blue-500`, `--spacing-4`. These are *values, not meanings* — the literal colors and spacings, with names that describe what they look like. This is Tailwind's default palette, the one you got for free the moment you imported Tailwind. A primitive knows it is blue; it has no idea what blue is *for*.

That is the problem the next tier solves.

**Tier two — semantic tokens: the component contract.** `--color-foreground`, `--color-primary`, `--color-destructive`, `--color-muted`. A <Term definition="A token that names a role (foreground, primary, destructive) and points at a primitive value. The tier components read. Re-pointing it at a different primitive — as .dark does — re-themes everything that reads it.">semantic token</Term> names a *role* and *points at* a primitive. `--color-primary` doesn't say "blue"; it says "the primary action color," and it happens to resolve to `--blue-600` today. **This is the tier components read.** It is exactly the token set shadcn ships — the `bg-primary` / `bg-destructive` / `bg-muted` vocabulary you will meet in full when you bring shadcn into the project later. And it is why the dark swap worked: `.dark` doesn't touch any component; it re-points the *semantic* token at a *different primitive* (`--color-primary` goes from `--blue-600` to `--blue-400`), and every `bg-primary` follows the binding to the new color.

**Tier three — component tokens: the rare exception.** `--button-primary-bg`, `--card-padding`. A token scoped to a single component, for the genuine cases where the semantic tier can't express what one component needs. Reach for this tier *rarely* — most components live their whole lives on the semantic tier and never define a component token. When you see one, it should feel like a deliberate exception, not a default.

The chain is the lesson, so see it resolve:

<TokenResolutionChain />

`bg-primary` reads the semantic token, which points at the primitive, which holds the value — three hops of indirection, and that indirection is the entire payoff. Look at the dark strand: the same `bg-primary`, the same component, but the middle pointer now aims at a lighter primitive, so the button lands on a different color *with the component untouched*. The component never knows. That is the property you are buying.

Which makes the load-bearing rule unavoidable, and it is the senior reflex this whole section exists to install: **components reference the semantic tier, never primitives.** `bg-primary`, never `bg-blue-600`. Direct primitive use in a component is a code smell — full stop. The why is a production story you can feel: a rebrand means changing the semantic-to-primitive pointer in *one place*, and the whole app re-themes. If components hard-code `bg-blue-600` instead, a rebrand becomes a find-and-replace across the entire codebase that is never quite complete — there is always one `bg-blue-600` hiding in a component nobody opened this quarter, still blue after everything else turned purple. The indirection is what makes the change one line instead of a thousand.

In CSS, the chain is literally one token reading another:

<div data-mark-color="green">

```css "var(--blue-600)" "var(--color-primary)"
:root {
  /* Primitive — a value */
  --blue-600: oklch(0.55 0.2 264);

  /* Semantic — points at the primitive (this is the indirection) */
  --color-primary: var(--blue-600);

  /* Component — points at the semantic; rare, shown for completeness */
  --button-primary-bg: var(--color-primary);
}
```

</div>

That `--color-primary: var(--blue-600)` — a custom property whose value is *another* custom property — is the indirection made literal. The semantic token doesn't *hold* a color; it *points at* the primitive that does. Re-point it once, everything downstream follows.

## Naming tokens for purpose, not appearance

Token systems rot at the naming layer, so naming earns its own short section. Treat these as decisions, not style preferences.

**Pair every surface with its foreground.** The convention is `--color-{role}` plus a matching `--color-{role}-foreground` for the text that sits on it: `--color-primary` and `--color-primary-foreground`, `--color-destructive` and `--color-destructive-foreground`. The pairing guarantees that text on a colored surface stays legible no matter how the surface color changes — you never put primary text on a primary background by accident. This is how shadcn keeps every surface readable, and you'll see the full set when you adopt it.

**Express states with opacity, not new tokens.** A hover state is usually `bg-primary/80` — the same token at 80% opacity — rather than a separate `--color-primary-hover` token. Fewer tokens to define, fewer to keep in sync, same result. That is the 2026 default: reach for the opacity modifier before you mint a state token.

**The same purpose rule covers every namespace.** `--spacing-{role}`, `--radius-{role}`, `--shadow-{role}` — all named for what they are *for*, never what they *are*.

And the rule under all of it: **the name describes what the token is for, never what it looks like.** `--color-destructive`, not `--color-red`. A destructive button is red today, but if the brand refresh next quarter makes destructive actions orange, a token named `--color-red` that now holds an orange value is a lie you have to read past every time — and renaming it is a breaking change rippling through every component that referenced it. The token name is an API. Name it after the durable thing — the *purpose* — not the volatile thing — the *color* — and it survives the rebrand that the color never will.

## `@property`: giving a token a type

One last, narrow corner — recognition-level, and gated by a real trigger.

To the browser, a plain custom property is an untyped string. It doesn't know `--gradient-angle: 0deg` is an *angle*; it just knows it is some text. And because it can't tell what *kind* of value it holds, it **can't interpolate it** — so a CSS transition or animation on a bare custom property *snaps* from one value to the next instead of animating between them. That is the problem `@property` solves: it tells the browser the type, so the browser can interpolate.

You register the property with an at-rule:

```css
@property --gradient-angle {
  syntax: '<angle>';
  inherits: false;
  initial-value: 0deg;
}
```

Three fields. `syntax` is the type — `<angle>`, `<color>`, `<length>`, or `*` for "any." `inherits` says whether the value flows down the tree like a normal custom property. `initial-value` is the starting value, and it is required unless `syntax` is the universal `*`. Once registered, the browser knows `--gradient-angle` is an angle and will smoothly interpolate it from `0deg` to `360deg` instead of jumping.

The 2026 reach here is genuinely narrow: **animatable custom properties.** A gradient angle that spins, a custom property that drives a transform and needs to interpolate between values — that is what `@property` is for. For a static token, plain `@theme` and `:root` are all you need; `@property` only earns its keep when the value must *animate*. So the recognition to carry away is a debugging instinct: **a token animates but snaps instead of easing → it is an unregistered custom property.** The actual keyframe and transition syntax comes later, when the chapter on motion arrives; here you only need to know the enabling primitive exists and what crossing-the-threshold looks like.
