A teammate opens a pull request. It's one line: they add a logging helper to the "Mark as paid" button so they can see when a click fires. The helper imports a couple of utilities, one of which — three files away, where nobody was looking — imports the database client. The tests pass. `next build` is green. The PR merges.

The database client is now in the public JavaScript bundle. Its connection logic, and the URL it connects to, ship to every visitor's browser, readable in the Network tab by anyone who opens DevTools. Nobody typed anything that looked wrong. The button already had its `'use client'` directive, and the directive did exactly its job — it drew the boundary and pulled everything the file imports across it. The leak happened *because* the boundary worked, not despite it.

That's the gap this lesson closes. You've been typing `'use client'` since the last lesson on the strength of one fact: it marks a file, and the mark flows downward to everything the file imports. That fact is true and it is not enough. A directive is a boundary *marker* — it tells the framework where the line is. It is not a boundary *guard* — it never checks whether the things on the client side of that line have any business being there.

By the end of this lesson you'll be able to do three things. First, read and place both directives — `'use client'` and its sibling `'use server'` — with their exact rules, including the trap that makes one of them fail in total silence. Second, name *why* the framework makes you write a literal string instead of figuring the boundary out for you. Third, make a misplaced server import fail loudly at `next build` instead of slipping into production the way the one above did. The last lesson promised this lesson would be "that directive in full." Here it is — both of them, plus the one-line tools that turn the gap into a build error.

## `'use client'` in full

You already own the headline fact about `'use client'`: it marks the entry point into the client subgraph, and everything that subgraph imports travels into the browser with it. That part is settled. What the last lesson deferred — and what trips people in real codebases — is the part that isn't about *what the directive does* but about *what counts as the directive at all*.

Here's the rule. `'use client'` is a <Term definition={"A literal string at the head of a module or function that the bundler reads as an instruction before it processes the code.\nIt is not executable code in the usual sense — it configures how the module is treated, not what it does at runtime."}>directive</Term>: a bare string literal that sits at the head of the file, above every `import` and every statement. Not a function call, not an import, not a special comment — a string, on its own line, ending in a semicolon. Single or double quotes both work. **Backticks do not** — `` `use client` `` is a tagged template expression, which is a completely different thing the JavaScript engine evaluates, not a directive the bundler recognizes.

One allowance is worth stating precisely, because the rule is easy to over-remember as "line one, no exceptions." The directive must come before any import or executable code — but *comments may precede it*. A license header or a `// @ts-nocheck` pragma above the directive is fine. The reason is mechanical: the bundler reads this string as a module-level instruction before it processes the module's body, and comments aren't part of the body, so they don't get in the way. "Before any import or statement" is the rule; "literally the first line" is a near-enough approximation that breaks the moment a file has a header comment.

The following two tabs show the placement that works and the placement that quietly doesn't.

<CodeVariants>
  <CodeVariant label="Correct" icon="approve-check">

    ```tsx title="app/invoices/_components/mark-paid-button.tsx" {1}
    'use client';

    import { useState } from 'react';

    export function MarkPaidButton() {
      const [isPaid, setIsPaid] = useState(false);
      // ...
    }
    ```

    </div>
    **The directive sits above everything.** The bundler reads it before the module body and marks this file — and everything it imports — as client. This is the only placement that does what you think it does.
  </CodeVariant>

  <CodeVariant label="Wrong — and silent" icon="error">
    <div data-mark-color="red">

    ```tsx title="app/invoices/_components/mark-paid-button.tsx" {1,3}
    import { useState } from 'react';

    'use client';

    export function MarkPaidButton() {
      const [isPaid, setIsPaid] = useState(false);
      // ...
    }
    ```

    </div>
    **Because an import comes first, the string is no longer at the head.** It's now just a regular expression statement in the middle of the module, and the bundler ignores it entirely. The file is treated as a Server Component — with no error. The `useState` on the next line will throw the first time this component actually renders, and only then.
  </CodeVariant>
</CodeVariants>

That second tab is the most important thing in this section, so let me draw it out. A directive is *just a string*. There is no symbol to resolve, no import to fail, no type to check. TypeScript has nothing to say about it, because as far as the type system is concerned a string in your file is a perfectly valid expression. So every way you can get the string slightly wrong leaves the file silently classified as a Server Component, with no complaint from any tool:

```tsx title="directive look-alikes — all silently ignored"
'use cleint';   // typo: ignored, the file stays a Server Component
'use-client';   // hyphen, not a space: ignored
`use client`;   // backticks make it a tagged template, not a directive: ignored
'use client'    // with an import above it: no longer at the head, ignored
```

None of those error. The bundler sees a string it doesn't recognize as a directive, shrugs, and treats the file as the default: a Server Component. The file *looks* like a Client Component to you — it has hooks, it has an `onClick` — but it isn't marked as one, so the first `useState` or event handler it hits crashes at render time. And because Server Components only render when something actually renders them, the crash can hide in a code path you don't exercise until a user clicks the wrong thing in production.

This is why experienced engineers have a small, almost superstitious habit here: **never hand-type the directive.** Copy it from a file you know is correct, or let your editor's snippet insert it. The string is too important and too unguarded to retype from memory and hope you got it right.

:::caution
A typo'd directive is the worst class of bug: invisible at build time, silent in code review, and triggered only when the wrong render path runs. There's no red squiggle to save you — the cost of a single transposed letter is a component that ships as the wrong kind and fails in front of a user.
:::

One last rule, and it's a reassuring one. Once a `'use client'` file has crossed into the client subgraph, every module it imports is *already* client — so writing `'use client'` again at the top of one of those imported files does nothing. It isn't wrong, just redundant. The boundary is a one-way door: you cross it once, at the top of the entry file, and everything below is on the other side. You never need to "re-declare" the client side deeper in the tree, and a stray extra directive down there is harmless noise, not a second boundary.

## `'use server'` and the asymmetry with `'use client'`

There's a second directive, `'use server'`, and the moment you see the name your brain will fill in the wrong meaning. `'use client'`, `'use server'` — the symmetry is right there in the words, so the obvious guess is that one marks Client Components and the other marks Server Components. That guess is wrong, and it's wrong in a way that causes real confusion, so let's kill it before looking at a single line of code.

Start from a fact you already have from the first lesson: **Server Components need no directive at all.** They are the default. Every file under `app/` is a Server Component until a `'use client'` boundary above it says otherwise. So `'use server'` *cannot* be "the thing that makes a Server Component" — there is no such thing to make, because Server Components are what you get for free by writing nothing. A file with `'use server'` at the top is not "extra-server" or "more of a Server Component." It is something else entirely.

Here's the actual shape of the two directives, side by side:

- `'use client'` marks **where a component runs**. It says "this file and its imports ship to the browser and run there." It's about *location*.
- `'use server'` marks **a function the browser is allowed to call**. It says "this function stays on the server, but a Client Component may invoke it, and when it does, the call crosses the network and the body runs server-side." It's about *exposure*.

One is a statement about *where code lives*. The other is a statement about *what the client is permitted to reach across the wire and trigger*. They share three letters of syntax and almost nothing of meaning.

The thing `'use server'` marks has a name: a <Term definition={"A server-side function exposed to Client Components as a callable endpoint, marked with 'use server'.\nCalling it from the browser triggers a network request; the body runs on the server. Covered in full in the forms chapter."}>Server Action</Term>. It's the framework's built-in <Term definition={"Remote procedure call — invoking a function that runs on a different machine as if it were a local function call, with the network round-trip hidden behind the call.\nServer Actions are React's RPC mechanism between the browser and the server."}>RPC</Term> mechanism: a Client Component calls what looks like an ordinary async function, but the function body runs on the server. Under the hood the client only ever holds an opaque reference to it; invoking that reference fires a network request, the server runs the real function, and the result comes back. It's the one function-shaped thing that's allowed to cross the boundary you've been drawing — *exactly what* is allowed to cross is the next lesson's subject, and the full Server Action surface (validation, error handling, what it returns, how it wires to a form) is a whole later chapter. Here you only need its two placements.

<CodeVariants>
  <CodeVariant label="File-level (_actions.ts)">

    ```tsx title="app/invoices/_actions.ts" {1}
    'use server';

    export async function archiveInvoice(id: string) {
      // mutation logic: the forms chapter
    }

    export async function markInvoicePaid(id: string) {
      // mutation logic: the forms chapter
    }
    ```

    </div>
    **The directive at the top of the file makes *every* export a Server Action.** The convention is to gather them in a file named for the job — here, the invoices' actions. Each is now callable from a Client Component; the bodies are stubs because the real work belongs to a later chapter.
  </CodeVariant>

  <CodeVariant label="Inline (one-off)">
    <div data-mark-color="blue">


    </div>
    **When you need just one action, scoped to where it's used, put `'use server';` as the first line of an async function body** instead of at the top of a file. The page itself has no directive — it's a Server Component, the default. Only the inner `archiveInvoice` function becomes a Server Action.
  </CodeVariant>
</CodeVariants>

So: file-level `'use server';` turns every export in the file into a Server Action, the convention being to collect them in an `_actions.ts` file. Inline `'use server';` as the first line of an async function body turns that single function into a Server Action, scoped to where it's defined. Both placements; both bodies are stubs, because everything *inside* a Server Action — the part that does the work — is owned by the forms chapter. Resist the urge to fill them in.

## Why a literal string and not framework magic

Step back from the mechanics for a second, because there's a design decision hiding in plain sight, and it's worth seeing clearly. The framework *could* have figured the boundary out on its own. It could scan every file, notice the ones that call `useState` or attach an `onClick`, and silently classify those as client. Or it could key off the filename — anything ending in `.client.tsx` becomes a Client Component, no string required. Both are technically possible. The framework deliberately does neither. Instead it makes *you* write a literal string at the top of the file.

That choice is the clearest example of an idea you'll meet over and over in this stack, so this is the moment to name it: **prefer explicit over magic.** When the framework could either infer something for you or make you state it outright, the senior preference is to state it outright — because a thing you can read beats a thing you have to deduce.

Feel the difference concretely. Imagine the inference version were real, and you open an unfamiliar file. To answer "does this run on the server or in the browser?" you'd have to scan the whole file for any hook, any event handler, any browser API — and then mentally run the bundler's classification rules over what you found. With the directive, that question is the *first line you read*. The answer is handed to you before you've parsed a single piece of logic. The explicit version moved the boundary from "something you compute" to "something you see."

And "something you see" pays off in three places that matter:

- **In the source.** The first line of every file tells you which side it's on, every time you open it. No deduction.
- **In code review.** When a diff adds `'use client'` to a file, the reviewer *sees a leaf cross into the bundle* — it's right there in the changed lines, a visible, discussable event. With inference, the same change would be a silent side effect of adding a hook somewhere, invisible in review.
- **In git history.** The exact commit where a file became a Client Component is a one-line change you can `git blame`. The moment is recorded. With magic, there's no moment to point at — just code that started bundling differently and nobody can say exactly when or why.

This isn't a one-off rule for directives. It's a stance that runs through the whole course, and you'll see the same instinct each time the framework offers to guess on your behalf. You'll write explicit dependency arrays on effects rather than letting the framework guess what an effect depends on. You'll declare explicit Zod schemas at the edges of your system rather than trusting whatever shape the data happens to arrive in. You'll annotate what a Server Action returns rather than leaning on inference at a boundary that crosses the network. Same principle every time: at the points that matter, make the behavior legible where you read it, not hidden in a tool's inference.

The directive is this principle's *statement* — the string you write down. The rest of this lesson is the principle's *enforcement* — the tools that make the explicit boundary one the framework will actually hold you to.

## When convention isn't enough: the leak the directive can't stop

Now back to the leak from the opening, because we can finally explain exactly why the directive didn't catch it.

Here is the thing the directive does, stated as narrowly as it's true: `'use client'` says "this file and everything it imports are client." That's the whole promise. Read it again and notice what it does *not* say. It does not say "and I'll check that everything it imports *should* be client." It marks the boundary; it does not inspect what ends up on either side of it. A server-only module — a database client, a helper that reads a secret, a file that uses Node's `fs` — that gets imported into a client file, whether directly or several hops down a chain of innocent-looking imports, is dragged across the boundary by the exact propagation rule you learned last lesson. The directive worked perfectly. The *import* was the mistake, and the directive has no opinion about imports.

The dangerous word there is *transitive*. A <Term definition={"A module you don't import directly, but pull in through a chain — A imports B, B imports C, so A transitively depends on C.\nThe reason a leak can hide several hops below the file you actually edited."}>transitive import</Term> is one you never wrote down yourself: your file imports a helper, the helper imports another, and four files later something pulls in the database client — and all of it counts as "imported by" your file as far as the bundler is concerned. That's why these leaks hide. The damage isn't in the import you can see; it's at the bottom of a chain you'd have to follow by hand.

When that happens, you get one of two outcomes, and they're both bad:

- **The loud-but-cryptic case.** The dragged-in module reaches for a Node-only API like `fs` or `net` that simply doesn't exist in a browser. The build *might* crash — but the error points at the missing browser global, deep in some dependency, nowhere near the import you actually got wrong. You'll spend an afternoon debugging a symptom three files away from the cause.
- **The silent case, which is worse.** The module is pure JavaScript — it bundles fine. No missing global, no crash. And now the code that reads your secret, possibly with the literal secret value inlined right into it, ships in the public bundle. The build is green. The deploy succeeds. The leak is completely silent, exactly like the one that opened this lesson.

You've seen this shape before. The first lesson drew the asymmetry: importing a Server Component into a Client Component fails *loudly* at build time, but passing a secret as a prop leaks *silently*. The last lesson sharpened it: the framework guards the loud failures, not the silent ones. This leak is the same asymmetry one more time — a silent failure the framework won't catch for you. The difference is that this time there's a tool that converts it into a loud one. First, though, see how a real leak hides, because it's worse than "someone imported the database into a button."

Three imports. Every single one of them looks reasonable on its own line — a button needs a label, a label needs a total, a total needs tax tables. No individual file is obviously wrong, which is precisely why no human reviewer flags it and why the build sails through. The mistake only exists in the *chain*, and chains are exactly what humans are bad at tracking and machines are good at. So let's get a machine to track it.

## `server-only` and `client-only`: turning a leak into a build error

The fix is one line, and it goes at the top of the file that must never reach the browser. For the database client, that's `db/index.ts`:

<div data-mark-color="green">

```ts title="db/index.ts" {1}
import 'server-only';

import { drizzle } from 'drizzle-orm/node-postgres';

import { env } from '@/env';

export const db = drizzle(env.DATABASE_URL);
```

</div>

That `import 'server-only';` line is a <Term definition={"An import with no binding — import 'x'; rather than import { thing } from 'x'.\nYou import it for what it does when the module loads, not for any value it gives you. import './globals.css' is another example."}>side-effecting import</Term>: you're not pulling a value out of it, you're importing it purely for what its presence *does*. And what it does is simple and brutal. `server-only` is rigged so that if the module importing it ever ends up in the client bundle, the build fails. So the moment a Client Component imports `db/index.ts` — directly, or three transitive hops away like the chain you just saw — `next build` stops and prints an error naming the offending import path. The silent leak becomes a loud, pre-deploy failure that points at the exact chain. The afternoon of debugging, the secret in production: both gone, traded for one line and a build error you see before you ever deploy.

`client-only` is the mirror image. `import 'client-only';` goes at the top of a module that must never run on the *server* — a helper bound to a browser API, or a third-party library that reads `window` the instant it's imported. Same mechanism, opposite direction: if such a module gets pulled into a Server Component, the build fails instead of crashing at render time with a baffling `window is not defined`. You'll reach for `client-only` far less often than `server-only`, because most code is perfectly safe to run on a server — but when you're wrapping a stubbornly browser-only library, it earns its line by failing fast and pointing at the cause.

A precise note on how these actually work, because the detail shifted recently and stale advice is everywhere. Next.js recognizes both `server-only` and `client-only` *internally* — it's the framework, not the package, that produces the build error, and Next.js ships its own type declarations for both. As of current Next.js, **installing the npm packages is optional**: the `import 'server-only';` line does its job whether or not the package is in your `node_modules`. You may still run `npm install server-only client-only` (usually as devDependencies) so your linter doesn't flag the import as unresolved — but don't think of the install as the thing protecting you. The *import line* is the contract; the protection comes from the framework reading it.

:::note
Older guides tell you that you *must* install `server-only` and `client-only` for them to work. That was once true and isn't anymore — Next.js handles both internally. Install them if you want clean linting, but the line is what matters, not the package.
:::

This is where the principle from the last section becomes concrete. The directive made the boundary *explicit*; `server-only` makes it *enforced*. And that turns it into a habit worth adopting now and keeping for the rest of the course: **every module that must stay on the server opens with `import 'server-only';`.** Your database client, your auth helpers, your email sender, your billing adapter, every `_actions.ts` file — all of them start with that line. The course's code conventions already bake this in: every SDK adapter under `lib/` begins with it. The cost is one line per file. The payoff is that the bundle *cannot* silently ship server code, ever — a careless import six months from now becomes a build error a reviewer sees, not an incident a customer reports.

<div data-mark-color="green">

```ts title="lib/auth.ts" {1}
import 'server-only';

// the Better Auth server instance

export const auth = /* ... */;
```

</div>

Now the second confusion to kill, and it's the one that catches people who *have* learned both `'use server'` and `server-only` — because the words overlap, the mind blends them. They are opposites. Watch:

- `import 'server-only';` says **"this file errors if it reaches the client."** It's a prohibition — a slammed door. The contents are off-limits to the browser, full stop.
- `'use server';` says **"every export here is a Server Action the client can call."** It's an exposure — an opened window. The contents are *deliberately reachable* from the browser, as callable endpoints.

One forbids the client from getting near the module. The other hands the client specific functions to invoke. A file would essentially never want both, because they're contradictory intentions. The two tabs below put them next to each other so the contrast lands.

    <Fragment slot="left">`app/invoices/page.tsx` — a plain page that fetches and renders invoices</Fragment>
    <Fragment slot="right">*nothing — a Server Component is the default*</Fragment>
  </Pair>
</Matching>

## Recap and what's next

Three things to carry forward.

- **The directives are literal strings with strict placement, and they do different jobs.** `'use client'` marks a component that runs in the browser; it must sit above all imports, it propagates to everything the file imports, and a typo in it fails *silently*. `'use server'` marks a Server Action — a function the client can call, executed on the server — and has nothing to do with Server Components, which need no directive at all. Parallel names, opposite jobs; don't blend them.
- **Explicit over magic.** The framework makes you write the boundary as a string you can read, review, and `git blame` rather than inferring it behind your back. That's a senior stance you'll meet again across the stack — explicit effect dependencies, explicit Zod schemas at the edges, explicit return types on actions.
- **`server-only` and `client-only` are the enforcement.** One side-effecting import turns a silent server-code leak into a loud `next build` error that names the offending chain. Adopt the habit now: every module that must stay on the server opens with `import 'server-only';`.

Two threads stay open for the rest of this chapter. The next lesson is *exactly what's allowed to cross the wire* when you pass props to a Client Component — the serialization rules behind "props must be serializable," and the secrets-in-props leak shown in full. After that comes hydration's failure modes: what happens when the server render and the browser render disagree. And the full Server Action surface you kept stubbing out — validation, errors, return shapes, form wiring — is owned by a later chapter on forms and Server Actions; today you only needed the directive that marks one.
