A reviewer opens a PR and sees this call: `createUser('alex', 'a@x.com', true, false, true)`. Which boolean is `admin`? Which is `sendWelcomeEmail`? Which is `acceptedTerms`? The reviewer has to jump to the function definition to find out — and even after they find out, they still have to count argument positions to be sure the call site matches. A teammate who reorders the booleans tomorrow produces a silent, catastrophic bug. The next person who needs a `marketingOptIn` parameter breaks every caller in the codebase.

The fix is one sentence: **two positional parameters max, then switch to a single options object.** That's the rule the rest of this lesson installs.

The previous lesson "Arrow by default, declaration on demand" picked the *form* of every function the course writes — arrow expressions bound to `const`. This lesson picks the *shape* of the parameter list inside that form. You'll see why two is the threshold, why an options object dissolves the readability problem, the exact firing rule for parameter defaults that catches every returning developer (`undefined` only, never `null` or `0`), the TypeScript ordering rule that the options object also dissolves, rest at the signature paired with spread at the call site, the field-level default pattern every paginated function in the course will use, and a refactor exercise where you take a five-positional-arg function and convert it.

## Two parameters max, then switch to an options object

The fix lives next to the failure mode. Both tabs below define the same `createUser` function — same five inputs, same intent. Compare the call site.

The rule, stated once: **past two positional parameters, switch to an options object.** Two is the threshold, not "two or three." Three positional parameters already starts to fray — by argument three the reader is counting positions instead of reading meanings.

There is one exception, and you have to name it because the language itself relies on it. Some functions take positional arguments because the positions are **semantic** — reordering them changes what the function does. A comparator `(a, b) => number` has `a` on the left and `b` on the right; `Array.sort` calls it that way, you don't get to switch the slots. A reducer `(acc, x) => acc` has the accumulator first and the current item second; that's the contract `Array.reduce` invokes. A range helper `(min, max) => …` has the lower bound first by convention. The trigger to keep positional past two is the function's identity *requires* a fixed argument order — mathematical conventions, callback contracts the runtime invokes, binary operators. Most application code doesn't qualify. If you're writing a `createUser`, a `sendEmail`, a `chargeCard`, the positions are not semantic; they're administrative. Options object.

There's a deeper signal hiding inside the same smell. When a function reaches four or five parameters, an experienced engineer reads that as "this function does too many things" *before* "this function needs an options object." Sometimes the right refactor isn't to box the arguments — it's to split the function. `createUserAndSendWelcomeEmail` is two operations pretending to be one; the cleaner shape is `createUser` returning the user, then `sendWelcomeEmail(user)` as a separate call. The options-object pattern is the primary teach of this lesson; the split-the-function move is the deeper heuristic you'll reach for as your sense for cohesion sharpens.

The options-object shape isn't just a code-style preference — it's the call shape the rest of the course writes. Server Actions (in the chapter on the Server Action surface) take a single options-shaped argument. React component props (in the React chapters) are an options object spelled with JSX. Drizzle's query builder (in the Postgres unit) accepts options at every chain step. The form was decided here, once.

## Parameter defaults fire only on `undefined`

The mechanics beat the lesson can't skip. The rule is one sentence and you need it in muscle memory before you write the first signature with a default.

**A parameter default fires only when the argument is `undefined`. Not `null`, not `0`, not `''`, not `false`.** Every value other than `undefined` flows through untouched, even the falsy ones.

This catches developers returning to TypeScript from older JavaScript habits. Code from before 2015 commonly defaulted with `||`: `const pageSize = arg || 20`. That form fires on every falsy value — `0`, `''`, `false`, `null`, *and* `undefined` — and it has bitten production countless times by replacing a deliberate `0` with a default `20`. Parameter defaults are narrower and more predictable on purpose; only `undefined` triggers them. (The same nullish-vs-falsy split shows up in the `??` versus `||` operator pair, which gets its own lesson later in this chapter. Parameter defaults are the signature-level form of the same idea.)

Here are the four cases worth knowing, in one block:

```ts
const greet = (name: string = 'friend') => `Hello, ${name}!`;

greet();                  // 'Hello, friend!'
greet(undefined);         // 'Hello, friend!'
// @ts-expect-error — demonstrating the runtime rule: null is not undefined, so the default does not fire.
greet(null);              // 'Hello, null!'
greet('');                // 'Hello, !'
```

The first two calls trigger the default — no argument and explicit `undefined` are the same thing to the parameter binding. The third passes `null`, which is not `undefined`, so the default does not fire and the empty body interpolates `'null'`. The fourth passes an empty string, which is falsy but defined; the default does not fire.

## Required parameters before optional — and how the options object dissolves the rule

TypeScript enforces an ordering rule on the parameter list: a required parameter cannot follow an optional one. `(options?: Options, url: string)` is a compile error. The reason is that the call site has no way to skip the optional and supply the required — positions don't allow it.

The workaround, in 2026 practice, is rarely "reorder so required comes first." It's the options object you already know:

```ts
const fetchPage = (options: {
  url: string;
  headers?: Record<string, string>;
  timeoutMs?: number;
}) => {
  // ...
};
```

On a single options-object parameter the required-before-optional rule disappears entirely — fields on an object are unordered, and the `?` modifier on each field marks just that field as omittable without affecting the others. `url` stays required; `headers` and `timeoutMs` are optional. The call site reads cleanly with any subset of the optional fields, or none of them.

TypeScript does allow one carve-out: an optional positional parameter *with a default value* (`url: string = 'http://localhost'`) can come before a required one — but the call site has to pass `undefined` explicitly to skip it, which is the same readability problem positional booleans have. The course doesn't reach for that form; the options object reads better in every case the carve-out would cover.

## Rest at the signature, spread at the call site

The other mechanics beat. Two pieces of syntax that look identical — three dots — and do opposite things depending on which side of the function boundary they're on. The mental model is one sentence: **rest binds positions to an array on receive; spread unpacks an array to positions on send.** Same dots, opposite directions.

The rest parameter at the signature side collects every trailing positional argument into a single array:

```ts
const tag = (label: string, ...ids: string[]) => {
  for (const id of ids) {
    console.log(`${label}: ${id}`);
  }
};

tag('user', 'u_1', 'u_2', 'u_3');
```

`...ids` collects the three trailing strings into an array typed `string[]`. The rest parameter has one structural restriction: it must be the **last parameter** in the signature. Anything after it has no positions left to bind. You can have any number of regular parameters before it, but only one rest parameter, and it has to come last.

On the call-site side, spread reverses the operation — it unpacks an array back into individual positional arguments:

```ts
const ids = ['u_1', 'u_2', 'u_3'];
tag('user', ...ids);
```

The spread expands the array in place. `tag` receives the same four arguments as the earlier call. Same three dots, opposite direction.

The most common reach for both in 2026 SaaS code is the **wrapper pattern** — a function that decorates another function (logging, timing, error-translation) and forwards every argument through:

```ts
const logged = (...args: Parameters<typeof baseFn>) => {
  console.log('calling baseFn with', args);
  return baseFn(...args);
};
```

`Parameters<typeof baseFn>` is a TypeScript utility type that extracts the argument tuple of `baseFn` and types `args` as that exact shape — so `logged` always matches `baseFn`'s signature, no matter how `baseFn` changes. The depth of typed wrappers and the generics that make them really shine lives in a later TypeScript chapter; the runtime shape is just rest on the inside, spread on the way back out.

One legacy note. Older JavaScript used a magic `arguments` object inside `function` declarations to collect call-site arguments. It's array-*like* but not actually an array, doesn't exist inside arrow functions at all, and rest parameters replace it completely. The course never writes `arguments` — it's named once here so you recognize it when you stumble across it in old code.

## Defaults inside the options-object pattern

The compositional form. Every paginated list function in the course will use this exact shape, so it earns its own short section.

```ts
const listInvoices = ({
  pageSize = 20,
  sort = 'asc',
}: {
  pageSize?: number;
  sort?: 'asc' | 'desc';
} = {}) => {
  // ...
};

listInvoices();
listInvoices({ pageSize: 50 });
listInvoices({ pageSize: 50, sort: 'desc' });
```

Two composition pieces working together. The **field-level defaults** (`pageSize = 20` and `sort = 'asc'` inside the destructure) supply per-field fallbacks the same way regular parameter defaults do — they fire when the field is `undefined`, which includes the case where the caller omits the field entirely. The **trailing `= {}`** on the parameter itself is the part most students miss on first read: it makes the function callable with no argument at all. Without it, `listInvoices()` would crash trying to destructure `undefined`, because you can't pluck fields out of nothing. The `= {}` says "if no argument, use an empty object as the source for destructuring" — and then every field-level default gets its chance to fire.

The `{ pageSize = 20, sort = 'asc' } = ...` half of that signature is **parameter destructuring** — the mechanic owned by the lesson on destructuring later in this chapter. Here you're reading the canonical shape every paginated function in the course writes; the lesson on destructuring derives why it works and shows the four extensions (rename, nested, rest at destructure, default at destructure) in depth.

Now the hands-on half. The function below has four positional parameters — past the rule, with one already defaulted and one already optional. Refactor it to take a single options object. The required fields are `street` and `city`; `country` defaults to `'US'`; `zip` has no default and is omitted from the output when missing. The tests pin both the structural shape and the firing rule from earlier in the lesson — pay attention to the last one.

<ScriptCoding
  runner="sandpack"
  instructions="Refactor formatAddress to take a single options object. Required fields: street, city. Optional fields: country (defaults to 'US'), zip (no default — omit from the output when missing). The function returns the parts joined by ', '."
  starter={`export const formatAddress = (
  street: string,
  city: string,
  country: string = 'US',
  zip?: string,
): string => {
  const parts = [street, city, country];
  if (zip) parts.push(zip);
  return parts.join(', ');
};
`}
  tests={`
test('uses default country when omitted', () => {
  expect(formatAddress({ street: '1 Main', city: 'Austin' }))
    .toBe('1 Main, Austin, US');
});
test('caller can override country', () => {
  expect(formatAddress({ street: '1 Main', city: 'Berlin', country: 'DE' }))
    .toBe('1 Main, Berlin, DE');
});
test('zip is appended when provided', () => {
  expect(formatAddress({ street: '1 Main', city: 'Austin', zip: '78701' }))
    .toBe('1 Main, Austin, US, 78701');
});
test('passing undefined for country still gets the default', () => {
  expect(formatAddress({ street: '1 Main', city: 'Austin', country: undefined }))
    .toBe('1 Main, Austin, US');
});
`}
/>

The fourth test is the deliberate one. It pins the `undefined`-only firing rule into the refactored shape: a caller who *explicitly* passes `undefined` for `country` still gets the default. The field-level default written as `country = 'US'` inside the destructure is the canonical form that satisfies it — the destructure pulls the field out as `undefined`, the default fires, and the function reads `'US'`. Same firing rule as a plain parameter default, applied one level deeper.

The reflex you should leave with: **two positional max, options object past that, defaults fire on `undefined` only, rest receives an array, spread sends one.** When you write your first Server Action it will take a single options object, decided here. When you write your first React component its props will be an options object spelled with JSX, also decided here. When you write your first Drizzle list query it will use the field-level default form, decided here. The shape is settled once; the rest of the course writes it without revisiting.
