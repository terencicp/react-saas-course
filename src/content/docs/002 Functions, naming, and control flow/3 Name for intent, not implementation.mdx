Open any production repo and three bugs are waiting at the names. A variable called `data` whose shape nobody can know without opening three files. A function called `processOrder` whose body the next reader has to read end-to-end to learn what "process" means in this codebase. A boolean called `notDisabled` that someone negates six months later as `!notDisabled` — a double negative that reads as "enabled" but never quite says so, and the reader cancels the negations wrong.

None of these are style preferences. They are documentation failures. The name is the only piece of code the next reader sees before they read the implementation, and a bad name is a coordination cost paid on every PR by every future reader, forever. The fix is one principle plus three categories of failure you'll learn to spot in a diff.

The principle, stated in one sentence:

<Aside type="tip">
  **A name says what the value is or what the function does, not how it's computed.**

  *<Term definition="The course's running list of senior-mindset principles — small set of decisions that shape a 2026 SaaS codebase before any syntax is involved. Each one earns its place by preventing a real bug class, not by being a style preference.">Architectural Principle #4</Term>* in the course's running list of senior-mindset principles.
</Aside>

This is the fourth principle the course installs explicitly. Like the `const`-by-default principle from the previous chapter, it's a *discipline*, not a syntax feature — TypeScript can't catch a misleading name, and Biome can lint a few of the rougher edges but most of the work lives in the human reflex you'll build today. The previous lesson "Signatures that stay readable past two parameters" picked the *shape* of the parameter list; this lesson picks the *labels* that hang on every binding inside that shape.

The path through: four naming surfaces (variables, functions, parameters, types), the boolean-prefix convention that tells the reader a value's type before they read the annotation, three classes of bad names that account for almost every naming failure you'll see in code review, and a short pair of rules on abbreviations and consistency. It closes with two recognition reps — a sorting drill on abbreviations and a PR review where you flag the bad names in a small file.

## The principle and its asymmetry

The word that does the work in the principle is *asymmetric*. Read it again:

> A name says what the value is or what the function does, not how it's computed.

The principle tolerates one direction and forbids the other. A name that is *vague but fitting* — `user`, `total`, `invoices` — is acceptable; the reader learns what the value is, even if they don't learn every detail about it. A name that *leaks the implementation* — `userArray`, `totalReducer`, `invoicesQueryResult` — is the smell; the reader learns how today's code happens to compute the value, and a future refactor breaks the name. The principle isn't "be specific." It's "don't leak how; do say what."

The same situation written both ways makes the asymmetry concrete. Both tabs below define a variable holding the pending invoices for a customer. The function call is identical in both; only the names change.

<CodeVariants>
  <CodeVariant label="Vague but fitting">
    <div data-mark-color="green">

    ```ts
    const invoices = await getPendingInvoices(customerId);
    ```

    </div>
    `invoices` is unspecific — it doesn't say the source, the filter, or the type. That's fine. The name fits what the value *is*: a collection of invoices. If `getPendingInvoices` is reworked tomorrow to read from a cache or paginate, the variable name still fits. Vague is acceptable when the value's identity is what the reader needs.
  </CodeVariant>

  <CodeVariant label="Implementation-leaking">
    <div data-mark-color="orange">

    ```ts
    const invoicesArray = await getPendingInvoicesQueryResult(customerId);
    ```

    </div>
    Both names leak today's implementation. `invoicesArray` pins the reader to the array representation — switch to a Map, a generator, or a paginated cursor and the name lies. `getPendingInvoicesQueryResult` pins the reader to the DB-query origin — switch to a cache hit or a third-party API call and the name lies. The names encode *how*, not *what*.
  </CodeVariant>
</CodeVariants>

One nuance to plant before moving on. "Vague is acceptable" doesn't mean "vague is the goal." Vagueness is a direction the principle tolerates, not a target it promotes. The right name is *as specific as the value warrants* — `pendingInvoices` reads better than `invoices` when the pending filter is the point of the variable. As specific as warranted, never more implementation-coupled than needed.

## The four naming surfaces

Every name in a 2026 codebase lives on one of four surfaces: variables, functions, parameters, types. The principle is the same across all four; the local rules and the payoff differ.

### Variables — nouns, concrete over abstract

Variables are nouns. Concrete beats abstract. `pendingInvoices` beats `data`; `activeUser` beats `obj`. The one twist that's worth stating explicitly: **length is proportional to scope.** A one-line callback parameter can be `x` — its scope is the expression it lives in. A module-level constant cannot — a reader miles away from the assignment needs to know what it is.

```ts
const totalCents = invoices.reduce((sum, invoice) => sum + invoice.cents, 0);

const pendingInvoicesForCurrentMonth = await db
  .select()
  .from(invoicesTable)
  .where(/* customerId + status filter */);
```

`sum` and `invoice` are fine inside the one-line `.reduce` — the binding's life is one expression and the position in the callback (accumulator first, item second) tells the reader what each one is. `pendingInvoicesForCurrentMonth` carries its full intent because it lives at module scope, where the reader who jumps to its first use has none of the context the writer had at assignment time. The rule reads in one direction: short names buy their brevity with the short scope they live in.

### Functions — verbs that signal the kind of operation

Functions are verbs or verb phrases. The verb does double duty — it tells the reader the function *does* something, and it signals the *kind* of operation:

- `load`, `fetch`, `get` for reads
- `create`, `update`, `archive` for writes
- `parse`, `validate` for transformation
- `format`, `render` for output projection

A reviewer skims the verbs and knows the function's category before reading its body. That category-at-a-glance is what makes a file readable at scrolling speed.

```ts
const invoice = await getInvoice(id);
const validated = parseCreateInvoiceInput(formData);
const formatted = formatCurrency(invoice.cents, 'USD');
await archiveInvoice(invoice.id);
```

Each verb signals the operation's shape — `get` is a read returning the row or null, `parse` is a transformation from raw input to a validated value, `format` is an output projection, `archive` is a write. The reader doesn't need to open any of those function bodies to know what category they're in.

The course doesn't legislate a single verb glossary. `fetchInvoice` and `loadInvoice` and `getInvoice` all communicate intent; the choice between them is a *team-codebase* call. The course's own convention (used in every later chapter) is `getInvoice` for single-record reads, `listInvoices` for collections, `requireInvoice` for reads that throw on miss, and verb+noun for Server Actions like `createInvoice`. The rule isn't which verb you pick — it's that you pick one verb per concept and stick to it across the codebase. The consistency section at the end of the lesson revisits this.

### Parameters — same rules, public surface

Parameters follow the variable rules with one tighter constraint: parameter names appear in the function's <Term definition="Everything about a function that's visible to a caller without opening its body — the name, the parameter list, the return type, and the TSDoc. Anything you'd see in an IDE tooltip or an error message counts.">**public surface**</Term>. TypeScript shows them on hover, in error messages, and in IDE tooltips. A vague parameter name pollutes every call site that uses the function — every developer who hovers the function sees the bad name and pays a tiny readability cost.

The contrast lands fastest when you imagine the IDE tooltip the next developer sees.

<CodeVariants>
  <CodeVariant label="Self-documenting">
    <div data-mark-color="green">

    ```ts
    const createUser = ({ name, email, role }: CreateUserInput) => {
      // ...
    };

    createUser({ name: 'alex', email: 'a@x.com', role: 'admin' });
    ```

    </div>
    On hover the IDE shows the destructured fields `name`, `email`, `role`. The call site reads as English — every argument names itself. A reviewer skimming the call needs zero context from the function body.
  </CodeVariant>

  <CodeVariant label="Opaque on hover">
    <div data-mark-color="orange">

    ```ts
    const createUser = (a: string, b: string, c: string) => {
      // ...
    };

    createUser('alex', 'a@x.com', 'admin');
    ```

    </div>
    On hover the IDE shows `(a: string, b: string, c: string)`. The reader has to open the function body to learn which string is the name, which is the email, which is the role. Every call site is a small mystery — and a teammate who swaps two strings ships a silent bug because the types still match.
  </CodeVariant>
</CodeVariants>

The destructured shape `({ name, email, role }: CreateUserInput)` is the canonical parameter form for any function that takes more than one input — the previous lesson "Signatures that stay readable past two parameters" picked it as the options-object default. The lesson on destructuring later in this chapter owns the mechanics of how the destructure works. The naming rule here applies whether you destructure at the signature or pull the names off the options object on the first line of the body: every name in that destructure shows up on hover.

### Types and type members — PascalCase nouns, no noise suffixes

Types are PascalCase nouns. Fields are camelCase. The course never writes `Type` or `Interface` as a suffix and never writes `I` as a prefix.

```ts
type Invoice = {
  id: string;
  customerId: string;
  status: 'paid' | 'pending' | 'overdue';
  amountCents: number;
};
```

The alias name is a noun — `Invoice`, not `InvoiceType`. The `type` keyword on the left is already the marker that the right-hand side is a type; suffixing the name with `Type` adds noise the reader has to skip. The same logic kills the `I`-prefix convention from older Java and C# codebases (`IUser`, `IInvoice`) and the <Term definition="A naming style that encodes a value's type or scope in the identifier itself, like bIsAdmin for a boolean or EStatus for an enum. Invented for early C and Visual Basic codebases where the compiler couldn't tell you the type at a glance.">Hungarian notation</Term> markers from older C codebases (`EStatus` for enums, `bIsAdmin` for booleans). The TypeScript community dropped both years ago because they're disambiguations that don't disambiguate anything — TypeScript already knows what's a type and what's a value.

If you see these conventions in third-party code, recognize them and don't replicate. The type-system depth that backs the `type`-vs-`interface` choice, branded types, and the rest of the type surface lands in the chapter on TypeScript object types.

## Booleans get a verb prefix that names the truth condition

The boolean-prefix convention is small but high-leverage. It's a **visual contract**: when the reader sees `is`, they know the value is a boolean before they read the annotation. The contract is for reading speed, not for typing discipline.

Five prefixes cover the surface:

- **`is*`** — current state. `isAdmin`, `isLoading`, `isPublished`.
- **`has*`** — possession or membership. `hasUnpaidInvoices`, `hasAccess`, `hasErrors`.
- **`can*`** — permission or capability. `canEdit`, `canDelete`, `canRetry`.
- **`should*`** — conditional intent (often used for behavior gating). `shouldRetry`, `shouldRevalidate`, `shouldOpenOnMount`.
- **`will*`** — future state or pending behavior. Rare; reach for it only when the timing matters.

Picking the right prefix isn't a precision exercise — there's often more than one acceptable choice. The discipline is to use *some* prefix from this set, every time, so the reader can read the truth condition at a glance.

```ts
const isAdmin = user.role === 'admin';
const hasUnpaidInvoices = invoices.some((invoice) => invoice.status === 'pending');
const canEditInvoice = isAdmin || invoice.ownerId === user.id;
```

Three booleans, three prefixes, three truth conditions. Each name reads as a complete sentence the value answers — "is this user an admin?", "does this collection have unpaid invoices?", "can this user edit this invoice?". The reader doesn't have to backtrack to the assignment to know what kind of value they're looking at.

:::caution
The prefix is a contract that says "this is a boolean." Naming a non-boolean with one of these prefixes breaks the contract. A string named `isAdminLabel` reads as a boolean and lies — call it `adminLabel` instead. The contract works in one direction: every boolean gets a prefix, but not every value with a prefix-shaped name is a boolean. Don't decorate non-booleans with these prefixes for "consistency"; the cost is paid by every future reader who has to double-check the type after the name promised otherwise.
:::

## The three classes of bad names

The lesson's center. Almost every naming failure in a code review falls into one of three categories. Once you can spot the category, you can name the fix without thinking. The classes are:

1. **Implementation-leaking** — the container or origin is in the name.
2. **Vague abstractions** — the name fits any value and communicates none.
3. **Negated booleans** — the name carries a negation that compounds at use sites.

Each one gets its own subsection with the smell, the fix, and a short worked example.

### Implementation-leaking — the container is in the name

The smell: the container or representation appears in the name. `userArray`, `customerMap`, `loadingFlag`, `invoicesQueryResult`. The name encodes how the value is stored or where it came from, which means the name *will* lie the moment someone changes either of those.

The fix: name what's *in* the container, not the container itself. Plurality is enough to signal a collection. The boolean prefix is enough to signal a flag. The destination of the value is enough to signal its purpose.

<CodeVariants>
  <CodeVariant label="Container in the name">
    <div data-mark-color="orange">

    ```ts
    const customerArray = await listCustomers();
    const loadingFlag = false;
    const invoicesQueryResult = await db.select().from(invoicesTable);
    ```

    </div>
    Every name leaks today's implementation. Switch `customerArray` to a Map next month and the variable name lies. Replace `loadingFlag` with anything boolean-ish (a finite-state machine value, a Promise pending status) and the name lies. Move `invoicesQueryResult` to a cached read and the name lies.
  </CodeVariant>

  <CodeVariant label="What's in the container">
    <div data-mark-color="green">

    ```ts
    const customers = await listCustomers();
    const isLoading = false;
    const invoices = await db.select().from(invoicesTable);
    ```

    </div>
    Plurality (`customers`) and the boolean prefix (`isLoading`) do the work the container suffix was trying to do — without coupling the name to today's representation. Switch the array to a Map, the flag to a state machine, the DB query to a cache hit, and every name still fits.
  </CodeVariant>
</CodeVariants>

### Vague abstractions — names that fit anything fit nothing

The smell: names that could attach to any value in the codebase: `data`, `info`, `result`, `manager`, `helper`, `util`, `handler`. They communicate nothing because they communicate everything.

The fix: replace with the concrete thing the value actually is. `data` becomes `weeklyMetrics`. `result` becomes `validatedInput`. The substitution is mechanical — read the source of the value and use that as the noun.

```ts del={1,4} ins={2,5}
const data = await fetchWeeklyMetrics();
const weeklyMetrics = await fetchWeeklyMetrics();

const result = validateInput(form);
const validatedInput = validateInput(form);
```

Both renames make the next reader's job easier — they see what the value represents without opening the function that produced it.

There's a deeper signal here worth naming. When a file's central object is genuinely called `manager` or `helper`, that's a refactor signal, not just a naming smell. A `UserManager` class usually turns out to be three or four unrelated operations bundled together — read the methods and you'll find a `findUser`, a `sendEmail`, a `validatePermission`, and a `logAudit` cohabiting one class because the author didn't have a name for any of them at the time. The course's discipline of placing pure helpers in `/lib` with verb-led names (`buildObjectKey`, `parseCursor`, `roleAtLeast`) is the structural answer: if you can't name the abstraction, you don't have one — you have a bag of operations that each deserve their own function.

### Negated booleans — double-negative footguns

The smell: a boolean named with a negation in the name itself. `notDisabled`, `isNotLoading`, `noErrors`. The name carries the negation as a permanent feature of the binding, which means it compounds with the negation operator (`!`) at every use site. `!notDisabled` reads as a double negative that means "enabled" — but it doesn't *say* so. Six months later a reader pattern-matches `!notDisabled` to "not disabled" and gets it wrong.

The fix: name the *positive* condition. `isEnabled` instead of `notDisabled`. `isLoading` instead of `isNotLoading` (and negate at the use site as `!isLoading` when you need the inverse). `hasErrors` instead of `noErrors`. The negation now lives at the use site only — and there it reads in one pass.

<AnnotatedCode lang="ts" code={`
if (!notDisabled) submitButton.disabled = true;
if (!isEnabled) submitButton.disabled = true;
`}>
  <AnnotatedStep meta="{1}" color="orange">
    `!notDisabled` is a double negative — the reader has to mentally cancel the two negations to know which condition fires the body. Six months from now a teammate cancels them wrong and the button never disables when it should. The negation in the name compounds with the negation operator at the use site; the meaning lives in the gap between them.
  </AnnotatedStep>

  <AnnotatedStep meta="{2}" color="green">
    `!isEnabled` reads in one pass — "if not enabled, disable the button." The negation operator at the use site pairs with a positively-named boolean and the meaning is unambiguous. Same runtime behavior; one reading instead of two.
  </AnnotatedStep>
</AnnotatedCode>

The Code conventions doc bans negated booleans outright across the course's codebase. Biome doesn't ship a built-in rule for this (you'd write a custom lint to catch it), but the PR-time review reflex is the safety net — the next exercise gives you reps on it.

## Abbreviations and consistency

Two short disciplines that round out the naming rules.

### The abbreviation rule

Don't abbreviate unless the abbreviation is **more common than the spelled-out form** in the domain. Acceptable: `url`, `id`, `db`, `api`, `http`, `jwt`, `ms` (milliseconds), `auth`, `env`. These read instantly to any web developer — spelling them out (`uniformResourceLocator`, `identifier`, `database`) would be the surprising choice.

Never invent new abbreviations. Not `usr` for user, not `prfl` for profile, not `qty` for quantity, not `acct` for account. The reader-cost of disambiguating an unfamiliar abbreviation is *always* higher than the writer-cost of typing the full word — especially in 2026 where every editor autocompletes. You save three keystrokes and tax every future reader who has to pause and guess.

### The consistency rule

When two names both pass the principle — `fetchUser` and `loadUser`, `customers` and `customerList`, `validate` and `check` — the choice between them is a team-codebase call, not a course call. Both communicate intent; either is acceptable in isolation.

The rule the course legislates: **pick one across the codebase and stick to it.** Drift between synonyms in the same repo is the real smell, not the choice itself. A codebase that uses `fetchUser` in one file and `loadUser` in the next forces every reader to keep two mental models of "the read operation" running side by side, and the cognitive cost compounds across hundreds of files. Pick one verb per concept, write it down in the team's conventions doc, and let the linter or a custom check enforce drift if the volume warrants it.

This is the same principle the previous chapter applied to function form: a team that rotates randomly between three function forms in the same hundred lines isn't picking each one for a reason — they're just reaching for whichever form their fingers offered first. The same rotation, applied to verbs, ships the same readability cost.

## Spot the bad name

The closing exercise. The point of this lesson isn't to memorize every rule — it's to leave with a reflex that fires when you see a bad name in a diff. Three filters to run on every name you review: does it leak the implementation? does it fit any value? does it carry a negation?

Review the PR below. Three of the five named values violate one of the three bad-name classes. Two are fine. Click the offending lines and leave inline review comments — name the class and propose the fix.

<CodeReview instructions="Review this PR. Flag every name that violates the principle — name for intent, not implementation. Three of the five named values have problems. Two are fine.">
  <ReviewFile name="src/invoices/summary.ts">
    ```ts
    import { db } from '@/db';
    import { invoicesTable } from '@/db/schema';

    export const getInvoiceSummary = async (customerId: string) => {
      const invoicesQueryResult = await db
        .select()
        .from(invoicesTable)
        .where(/* customerId + pending status filter */);

      const data = invoicesQueryResult.filter((invoice) => invoice.status === 'pending');
      const notPaid = data.length > 0;
      const totalCents = data.reduce((sum, invoice) => sum + invoice.cents, 0);

      return { pending: data, hasPending: notPaid, totalCents };
    };
    ```
  </ReviewFile>

  <ReviewIssue file="src/invoices/summary.ts" line={5} kernel="implementation-leaking name; pins the variable to the DB-query origin even though the value is just a list of invoices — rename to `invoices`">
    This is the **implementation-leaking** class. `invoicesQueryResult` encodes that today's value came from a DB query — switch it to a cache hit, an API call, or a paginated cursor and the name lies. The fix is `invoices`. The value is just a list of invoices; the source is the function's job to know, not the variable's. The principle's asymmetry from earlier in the lesson: vague-but-fitting is acceptable, implementation-leaking is the smell.
  </ReviewIssue>

  <ReviewIssue file="src/invoices/summary.ts" line={10} kernel="vague abstraction; `data` fits anything and communicates nothing about the filter that was applied — rename to `pendingInvoices`">
    This is the **vague-abstraction** class. `data` could attach to any value in any file. The reader has no way to know from the name alone that this binding holds the *pending* subset — they have to read the `.filter` call to find out. The fix is `pendingInvoices`. The name now matches what the value is, including the filter that produced it. As specific as the value warrants.
  </ReviewIssue>

  <ReviewIssue file="src/invoices/summary.ts" line={11} kernel="negated boolean; compounds with `!` at use sites into a double negative — name the positive condition (`hasPending`)">
    This is the **negated-boolean** class. `notPaid` is a boolean named with a negation in the name. Any future use site that needs the inverse becomes `!notPaid` — a double negative that means "paid" but doesn't say so. The fix is `hasPending` (or `hasPendingInvoices` for full clarity). The name now states the positive condition the value answers, and the negation operator at the use site reads in one pass.
  </ReviewIssue>

  <ReviewWhy>
    Three classes, one reflex. When you see a name in a diff, run the three filters before you read the implementation — does the name leak the container, the query origin, the representation? Does it fit any value? Does it carry a negation? The names you can't justify in a single sentence are the ones to flag. The names you can — `getInvoiceSummary` (verb-led function, intent-named), `totalCents` (concrete, prefix-less because totals aren't booleans) — leave alone. Leaving correct names un-flagged is half the recognition reflex.
  </ReviewWhy>
</CodeReview>

The reflex you should leave with: every name in the codebase passes three filters before it lands. Does it leak the implementation? Does it fit any value? Does it carry a negation? Three filters, no exceptions. When the reflex is sharp, you spot the bad name in the diff before you read what the line does — and the review comment writes itself.
