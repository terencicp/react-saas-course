Open a file from a team that never picked a function rule and you'll see all three forms in the same hundred lines. A `function declareUser() { ... }` at the top. A `const handleClick = function (e) { ... }` halfway down. A `(name) => name.toUpperCase()` inside a `.map`. Every one of them is legal. None of them was chosen for a reason â€” the author reached for whichever form their fingers offered first. That rotation, not any single form, is the smell. A reviewer can't tell whether `function` carried meaning or was the next reflex.

The fix is one sentence: pick one default, name the triggers that flip it. The course's default is `const fn = (args) => â€¦`. Three narrow triggers earn a `function` declaration. Everything else is `const` + arrow.

This lesson rides the `const`-by-default reflex from the previous chapter's lesson "const binds, it doesn't freeze" straight into function form â€” if every binding is `const`, then every function is a value bound to `const`, which spells `const fn = (args) => â€¦`. You'll see why arrow `const` wins as a default, the one syntax gotcha that catches everyone in their first week, the three triggers that earn `function`, two ancillary forms named for recognition only, and a sorting drill that installs the reflex.

## Why arrow `const` is the 2026 default

Here is the canonical shape. Every function in the rest of the course will look like this, unless one of three triggers flips it.

```ts
const greet = (name: string) => `Hello, ${name}`;
```

An arrow expression bound to `const`. Read left-to-right: the name (`greet`) is a `const` binding, the right-hand side is a function value, the arrow connects the parameter list to the body. The form is the natural continuation of the previous chapter â€” `const`-by-default means `const fn = â€¦` by default, and the right-hand side that fits a `const` is an arrow expression.

Three reasons this wins as the default:

- **It continues the `const` reflex from the previous chapter.** Functions are values. If every binding is `const` unless it has to reassign, then the function form follows: `const fn = â€¦`. You're not learning a new rule â€” you're applying the one you already have.
- **No `this` rebinding surprise inside callbacks.** Arrow functions inherit `this` from the enclosing scope. `function` expressions get their own `this` based on the call site, which used to force developers to write `.bind(this)` on every callback before 2015. The arrow's inherited-`this` rule is exactly what `.map`, event handlers, and `setTimeout` want by default. The full rule gets a short section of its own later.
- **Biome's `useArrowFunction` rule is the build-time backstop.** The rule auto-fixes non-`this`, non-generator `function` expressions into arrow form, and deliberately leaves top-level `function` declarations alone (those are the triggers covered in the next section). Same shape as the `useConst` rule from the previous chapter: you write what you mean and the linter catches the cases where reflex slipped.

"Default" is not "always." Defaults exist so you can think about the exceptions. The job of this lesson is to install the default *and* the three named cases where it flips. Anything outside those three is `const` + arrow.

## Implicit return, block body, and the parens trap

Arrow functions have two body shapes. The choice between them is mechanical â€” one trigger flips it â€” and once you've seen the rule you'll never have to think about it again.

<CodeVariants>
  <CodeVariant label="Single expression">
    <div data-mark-color="green">

    ```ts
    const double = (x: number) => x * 2;
    ```

    </div>
    **One expression, no braces, no `return`.** The expression's value is what the function returns. Reach for this form when the body is one line.
  </CodeVariant>

  <CodeVariant label="Statements needed">
    <div data-mark-color="blue">

    ```ts
    const double = (x: number) => {
      const result = x * 2;
      return result;
    };
    ```

    </div>
    **Braces and `return` the moment the body needs a statement** â€” an intermediate `const`, an `if`, a loop, a side effect. The block form scales up; the implicit-return form does not.
  </CodeVariant>
</CodeVariants>

The senior reflex: reach for the implicit-return form for one-liners, switch to the block body the moment the body needs a statement. Don't twist a `return` into a comma expression or a nested ternary just to keep the one-line shape â€” that's a readability tax the next reader pays.

There's one syntax gotcha in arrow-`const`-land, and it's the one that catches every beginner in their first week. The good news: once you've seen it, you'll never write it again.

<AnnotatedCode lang="ts" code={`
const wrong = (name: string) => { name: name.trim() };
const right = (name: string) => ({ name: name.trim() });
`}>
  <AnnotatedStep meta="{1}" color="orange">
    The braces parse as a **block body**, not an object literal. The parser reads `name:` as a *labeled statement* (an obscure JavaScript feature) and `name.trim()` as the expression inside it. No `return`, so the function returns `undefined`. With Biome and a strict `tsconfig` the unused label is usually flagged at build time, but in older or looser configurations the bug is silent â€” no syntax error, no type error, just an `undefined` flowing downstream until something breaks.
  </AnnotatedStep>

  <AnnotatedStep meta="{2}" color="green">
    Parens around the object literal disambiguate. The braces are now an object; the parens are the expression the arrow returns. The senior reflex: wrap object-literal returns in parens. Once you have the reflex, the trap stops existing.
  </AnnotatedStep>
</AnnotatedCode>

:::note
This is the only syntax gotcha in arrow-`const`-land â€” everything else is straight-line. The reason it exists is parser ambiguity: `{` can begin a block or an object literal, and the parser commits to "block" inside an arrow body. Parens force the expression reading.
:::

## The three triggers that earn a `function` declaration

These are the only cases where `function` earns its keyword in a 2026 SaaS file. Each subsection is one tight pattern: the trigger, the snippet, and the honest framing of how often it actually shows up.

### Hoisting â€” top-of-file helpers used above their declaration

A `function` declaration hoists its full body to the top of the scope. The function is callable from any line in the module, including lines above it. An arrow `const` lives in the <Term definition="The period between entering a scope and reaching a let/const declaration. The binding exists but accessing it throws a ReferenceError.">Temporal Dead Zone</Term> (introduced in the previous chapter's lesson "const binds, it doesn't freeze") until the line it's declared on â€” it cannot be referenced before that line.

```ts
const render = (node: TreeNode) => `<li>${formatLabel(node)}</li>`;

function formatLabel(node: TreeNode) {
  return node.label.trim();
}
```

`formatLabel` is callable on line 1 because the `function` form hoists its whole body. If you rewrote `formatLabel` as an arrow `const`, the reference on line 1 would throw `ReferenceError: Cannot access 'formatLabel' before initialization`.

The honest framing: this trigger is rare in 2026. Most code is import-first and module-scoped â€” helpers come from other files, not from below in the same file. You'll see it most often in **recursive helpers** (the next subsection) and a handful of small in-module utilities where flipping the declaration order would hurt readability.

### Named recursion â€” a helper that refers to itself

A recursive function needs a stable name to call itself by. The `function` form gives that name directly to the function's own scope, independent of any outer binding.

```ts
function walk(node: TreeNode): void {
  for (const child of node.children) {
    walk(child);
  }
}
```

`walk` refers to itself by name. That name lives in the function's own scope, not in the outer `const` binding. The arrow alternative â€” `const walk = (node) => { ... walk(child); ... }` â€” works in practice, but the recursion routes through the outer binding instead of through the function itself. The moment another scope shadows `walk` (a test that mocks it, an inner helper that reuses the name), the recursion calls the shadow, not itself. The `function` form sidesteps the whole question.

The honest framing: tree walkers, parsers, and a small set of recursive algorithm helpers. You'll meet this shape later when traversing database relation graphs and transforming validated input.

### Type-guard and assertion signatures â€” TypeScript-shaped triggers

This is the trigger most students miss on first read. TypeScript has two signature shapes that interact with arrow vs. `function`:

- The **assertion** signature `asserts x is User` does not parse on arrow expressions at all. It is a long-standing language constraint â€” `const assertUser = (x: unknown): asserts x is User => â€¦` is a syntax error. You **must** write `function` for assertions.
- The **predicate** signature `x is User` has more nuance. TypeScript 5.5+ can infer a predicate from a simple-bodied arrow (e.g. `const isUser = (x: unknown) => typeof x === 'object' && x !== null && 'id' in x`). But the moment the body is more than one expression, or the team wants the contract **declared** rather than inferred, `function` is the canonical form. The signature reads the contract; no inference required.

The senior reach in 2026: declare both shapes with `function`. The predicate sits next to its assertion sibling and reviewers read the contract at a glance. Here's the pair in a single block â€” narrowing fires the moment the predicate returns true.

## Two forms to recognize, not reach for

Two more function forms exist in 2026 TypeScript. Neither is a trigger; both are recognition-only. You'll see them in third-party code and the occasional config object â€” knowing the shape means you won't "fix" them to an arrow.

### Method shorthand in object literals

When a function lives inside an object literal as a method, the language lets you drop the colon and the `function` keyword:

```ts
const config = {
  greet(name: string) {
    return `Hello, ${name}`;
  },
};
```

This form shows up inside config objects, database relation definitions, validation schema transforms, and route-handler exports â€” anywhere a function lives as a method on an object literal. It's not a separate trigger; it's a syntactic shortcut. Method shorthand binds `this` the same way `function` does (own `this` based on call site), which is the correct behavior for the object-method shape. Recognize the form; don't "fix" it to an arrow.

### `function` expressions

The other ancillary form: a `function` expression assigned to a `const`.

```ts
const factorial = function fact(n: number): number {
  return n <= 1 ? 1 : n * fact(n - 1);
};
```

The internal name (`fact`) is only visible inside the function body â€” useful for self-reference inside an expression that for some reason can't be a `function` declaration. Almost always replaced by an arrow expression or a `function` declaration. Named here so you recognize it in older code or third-party libraries; not a form the course writes.

## The `this` rule, stated and dropped

Arrow functions inherit `this` from the enclosing scope. `function` forms â€” declarations, expressions, method shorthand â€” get their own `this` based on the call site. That's the whole rule.

If you read any JavaScript article from before 2015, `this` reads as one of the language's worst footguns. The reason: `function` expressions used inside callbacks got their own `this`, usually `undefined` or the wrong thing, and developers had to write `.bind(this)` on every callback to fix it. Arrow functions in 2015 removed the footgun by inheriting `this` from where the arrow was written â€” exactly what callbacks want. The pain is real history; the fix is the boring default you're already using by writing arrow `const`.

The course barely uses `this` at all. The stack you'll learn is functional: components are functions, database queries are functions, Server Actions are functions. The only `this` you'll meet is in third-party class-based code (some SDKs) and the rare custom `Error` subclass. The rule is named so you're not surprised when you see it â€” not derived, because you won't write it.

:::note
If you find yourself reaching for `.bind(this)` or wondering what `this` resolves to in your code, you've drifted into class-OOP territory the course doesn't teach. Step back, write a function that takes what it needs as parameters, and the question disappears.
:::
