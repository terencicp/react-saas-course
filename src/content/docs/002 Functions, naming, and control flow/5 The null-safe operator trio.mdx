Two short bugs from the same root. The first: a defensive chain `if (user && user.profile && user.profile.address) { … }` that the team rewrites once per page until someone reaches for a three-character fix. The second: `const pageSize = input.pageSize || 20` — works fine until the day a caller passes `pageSize: 0` to mean "show no rows" and your function silently renders twenty rows instead. Both are about *the value might be missing*, and both have a precise fix in a trio of operators that has shipped in every browser since 2020.

That trio is what this lesson installs. **`?.`** — optional chaining, for nullable property and call access. **`??`** — nullish coalescing, for "use a default when the value is missing." **`??=`** — nullish coalescing assignment, for lazy initialization. Plus one closing rule the language picked deliberately: JavaScript rejects `??` mixed with `||` or `&&` without explicit parentheses, and the fix is one set of parens.

What's ahead: `?.` with its per-link semantics and the overuse trap that silences the type system; `??` with the falsy-vs-nullish distinction as the lesson's payoff (and the three concrete bugs `||` ships); `??=` for the cache-or-compute pattern; the parens rule for mixed expressions. The previous lesson on "Flat control flow" gave you guard clauses for `null` and `undefined` cases. This lesson gives you the operators that handle the same shape inline — no `if`, no return, no extra indentation. And it builds on the same anchor as the `==` ban from the chapter on TypeScript fundamentals: **the language distinguishes nullish from falsy on purpose, and you should too.**

## `?.` for nullable property and call access

The first operator. The shape lives in one line — the same chain from the intro's bug, refactored.

```ts
const city = user?.profile?.address?.city;
```

Each `?.` checks the value to its **left** before stepping right. The first nullish link <Term definition="The expression stops evaluating at the first link that returns the early value. For ?., that's the first nullish link returning undefined; for ??, that's a non-nullish left returning itself.">short-circuits</Term> the entire expression to `undefined`. No more nested `&&` chains; no more `Cannot read property 'address' of undefined` at runtime. <Term definition="The ?. operator family — ?., ?.(), ?.[…] — for accessing a property, calling a method, or indexing when the receiver might be null or undefined.">**Optional chaining**</Term> covers three access shapes — property, method call, and index — each with its own punctuation.

<CodeVariants>
  <CodeVariant label="Property access">
    ```ts
    const city = user?.profile?.address?.city;
    ```
    **Property access.** The chain short-circuits at the first nullish link and the whole expression resolves to `undefined`. Reads at every dotted hop where the receiver might be missing.
  </CodeVariant>

  <CodeVariant label="Method call">
    ```ts
    const greeting = user?.greet?.();
    ```
    **Method call.** The `?.()` form checks that `user.greet` is not nullish before calling. Useful for optional callbacks — an `onSuccess?.()` prop on a component, a `cleanup?.()` returned from an effect, a feature-flagged method on a third-party SDK.
  </CodeVariant>

  <CodeVariant label="Indexed access">
    ```ts
    const first = invoices?.[0];
    ```
    **Indexed access.** The `?.[…]` form checks `invoices` before indexing. Reaches at any optional array or object access where the bracket notation is required — dynamic keys, numeric indices, anything not expressible as a static dotted path.
  </CodeVariant>
</CodeVariants>

### Each `?.` only guards the link to its left

The senior reflex this section installs: **a `?.` at one link doesn't protect later links.** Each operator checks exactly one step. Place it once at the front of a chain and the rest of the chain is still vulnerable.

```ts
const city = user?.profile.address.city;
```

This looks defensive but isn't. The `?.` after `user` only guards the access to `profile`. If `user.profile` is nullish, the chain throws at `.address` — and it throws with the same `Cannot read property 'address' of undefined` you were trying to avoid. The fix is a `?.` at every link that could be missing:

```ts
const city = user?.profile?.address?.city;
```

Now every step checks its receiver before continuing. The rule reads in one sentence: **question-dot at every nullable link, not just the first one.** When you're reading code in review and you see a chain with one `?.` followed by plain dots, the first reflex is "does the type allow the middle to be nullish?" — if yes, the chain is one missing field away from a runtime error.

### The overuse trap: don't silence the type system

The opposite reflex is the more dangerous one. The naive teaching of `?.` is "sprinkle it whenever you're not sure" — and that's exactly the wrong rule, because it *silences* the type system on every line you add it. The point of `?.` is to handle nullables the type acknowledges. The point of TypeScript is to fail loudly when a non-nullable becomes nullable later. Reach for `?.` on a field the type says is required, and you've turned that future signal off.

```ts
type Order = {
  customer: Customer;
  lines: Line[];
};

const orderTotal = order.customer?.lines.length;
```

`customer` is required by the type — there's no `?` after it, so it's `Customer`, not `Customer | undefined`. Writing `order.customer?.lines` reads as cautious; it isn't. The day someone marks `customer` as optional to support draft orders, every existing call site silently returns `undefined` instead of failing at the line that needs to be re-examined. The discipline:

**`?.` where the type acknowledges nullable; let TypeScript fail at the call site otherwise.**

The narrowing chapter later in this unit owns the full story on how TypeScript flows nullability through `?.` chains — and how the right reach for a "value might or might not be there" branch is often a guard clause that narrows before the access, not a `?.` after it. For now, the recognition pattern: a `?.` on a non-nullable field is a code smell, not extra safety.

:::tip
Biome's `useOptionalChain` rule (in the recommended preset) flags long `&&` chains that exist only to guard a property access and offers an autofix to the `?.` form. The linter handles the *form*; the senior handles the *discipline* of where to place it.
:::

## `??` for "use a default when the value is missing"

The second operator and the lesson's biggest payoff. The rule fits in one sentence: **`??` returns the right operand only when the left is `null` or `undefined`. `||` returns the right operand for any falsy left.** That's the entire distinction. Everything below is consequences.

<Term definition="The two values JavaScript treats as 'missing': null and undefined. Distinct from the larger 'falsy' set, which also includes 0, '', false, and NaN.">**Nullish**</Term> is the right concept for "the value is missing" — the absence of a value. **Falsy** is the right concept for "the value is missing **or** the value is one of several legitimate-but-empty things" — empty string, zero, false, `NaN`. The two sets overlap on `null` and `undefined`; everywhere else, they diverge. `??` picks the first set; `||` picks the second. When you want a default to fire only when the field is absent, `??` is the operator. `||` fires on absence *and* on every legitimate zero, empty string, or `false` the caller might pass — and you almost never want both.

### Three concrete bugs `||` ships

Each tab below is a real bug the `||` reflex produces. Same shape every time: the right operand is the "default," the left operand can legitimately be a falsy value, and `||` overwrites the user's intent.

  </CodeVariant>

    An empty string is often a value the user deliberately cleared — a form field they emptied before submit. `||` treats it as missing; `??` respects the user's choice and only defaults when the field wasn't submitted at all.
  </CodeVariant>

    The senior tell that `||` is in the wrong lane: the right operand is a boolean literal. If the left is also boolean, falsy-vs-nullish is the entire question, and `||` will overwrite every deliberate `false` the user sets.

### The senior rule, restated

**For "use a default when the value is missing," reach for `??`. Reach for `||` only when "any truthy value short-circuits" is exactly what you want** — and in 2026 SaaS code, that's rare enough that the senior writes `??` by reflex and uses `||` deliberately when it appears. The same semantic backs the parameter-defaults rule from "Signatures that stay readable past two parameters": parameter defaults fire only on `undefined`. `??` is that rule, available as an operator anywhere in the body of a function. If `??` feels familiar — it should. It's the same "the field is missing" semantic in a different syntactic position.

## `??=` for lazy initialization

The third operator. The shape: **`x ??= y` assigns `y` to `x` only when `x` is currently nullish.** Same nullish-not-falsy semantic as `??`, applied to assignment. The canonical use is the cache-or-compute pattern in three characters.

```ts
const cache: Record<string, number> = {};

const getOrCompute = (key: string) => {
  cache[key] ??= computeExpensive(key);
  return cache[key];
};
```

One line, one read, one conditional write. The first call for a key writes; every subsequent call for that key reads the cached value. The `??=` form expresses the intent — "fill this slot if it's empty" — without the boilerplate of an explicit `if (cache[key] === undefined) cache[key] = …` or the verbose `cache[key] = cache[key] ?? …` rewrite of the same idea. It's the reach for any "initialize once, reuse" pattern that doesn't need a full memoization library.

### The right-hand side is evaluated lazily

`??=` only evaluates its right operand when the left is nullish — so `computeExpensive(key)` doesn't run on cache hits. That matters. If the right-hand side is a function call, an API request, or a database read, the assignment operator's laziness is doing real work: skipping the call entirely when the slot is already filled. The naive long-hand form `cache[key] = cache[key] ?? computeExpensive(key)` happens to behave the same way (because `??` itself short-circuits), but it reads the cache twice and writes unconditionally. `??=` says the intent in one read and one possible write.

### `||=` and `&&=` exist, but you write `??=`

For completeness: `||=` assigns when the left is *falsy* (and so ships the same `0`/`''`/`false` traps as `||`); `&&=` assigns when the left is *truthy*. Both are valid JavaScript, both occasionally fit a real pattern. The course writes `??=` by reflex because nullish-not-falsy is almost always what the caller wants — the same reasoning behind picking `??` over `||`. Name `||=` and `&&=` once for recognition; reach for `??=` for "fill if empty."

## Mixing operators — the parens rule

One last mechanic before the lesson closes. JavaScript intentionally **rejects** `??` mixed with `||` or `&&` without explicit parentheses. `a || b ?? c` is a syntax error, not a precedence quirk you can memorize away.

```ts
const value = a || b ?? c;
```

The reason the language did this: the precedence was ambiguous enough — does `||` bind tighter than `??`? does `??` bind tighter than `&&`? — that the spec authors picked "require explicit grouping" over "pick one and let half the readers get it wrong." The error is the language refusing to guess at the author's intent. The fix is one set of parens, either way you meant it:

```ts
const value = (a || b) ?? c;

const value = a || (b ?? c);
```

Two readings, two meanings. The first groups the `||` first — pick the truthy of `a` or `b`, then fall back to `c` only if the result is nullish. The second groups the `??` first — use `a` when truthy, otherwise compute `b ?? c` (i.e. `b` when not nullish, `c` when it is). Same operators, different shape — and the parser refuses to pick one without you saying so.

### The broader parens reflex

You don't memorize JavaScript's full operator-precedence table — there are too many operators and the cost of remembering them is higher than the cost of typing parens. The senior reflex: **when an expression mixes operators of different kinds, add parens that state the intent.** Even when the parser doesn't require them, the parens document the author's reading to the next reviewer.

```ts
const score = a ?? b + 1;

const score = (a ?? b) + 1;
```

The first line parses as `a ?? (b + 1)` — almost certainly not what was meant. The second line says the intent out loud: take `a` if it's set, otherwise `b`, then add one. The parens cost nothing; the misread precedence costs a runtime bug that's invisible in casual review.

The rule generalizes: any time `??` shares an expression with `||`, `&&`, `+`, `-`, or any other binary operator, parens make the grouping explicit. The parser accepts both readings — your job is to pick the one you meant.
