Two short bugs from the same root. The first: a `sendInvoice` handler receives an object — `{ customerEmail, amountCents, internalNotes }` — and forwards the whole thing to `email.send(input)`. The customer-facing email now includes the internal admin-only notes meant for the finance team, and nobody noticed because the leak is invisible at the call site. The second: a React component writes `const { name: customerName } = props` to avoid clashing with an outer `name` binding. The author flips the rename direction by mistake; now the renamed `customerName` is never created and the local `name` shadows the wrong identifier. Both bugs come from the same place: a function pulled fields out of an object without naming exactly which ones, in exactly which direction.

That fix is destructuring. <Term definition="The syntactic form that pulls fields out of an object (or positions out of an array) and binds them as local variables in one expression. Same shape works at the signature, after assignment, and inside function bodies.">**Destructuring**</Term> is the syntactic consumer of the options-object pattern from the previous lesson "Signatures that stay readable past two parameters" — the caller writes an object, and the function picks the fields it needs by name. Once you're past the React unit, every component, every Server Action, every Drizzle query helper takes its input this way, and the course never re-explains the form.

What's ahead: object destructuring with its four extensions (rename, default, combined, rest); array destructuring for its two daily reaches; the signature-level destructure as the canonical Server Action and React-prop shape; and the destructure-then-rebuild pattern as the structural fix for the wholesale-forward leak.

## Object destructuring and its four extensions

The base form, against a small `customer` object we'll reuse across this section:

```ts
const customer = {
  id: 'cust_123',
  name: 'Alex',
  email: 'alex@example.com',
  pageSize: 0,
};

const { name, email } = customer;
```

That's the whole base. Two `const` assignments by key, written as one expression. Field name on the left, source object on the right; the local bindings created are `name` and `email`, holding the values they had on the object. The mental model fits in one sentence — and once you have it, the four extensions are the same shape with one piece added.

The variants below all run against the same `customer` object above. The only thing changing from tab to tab is which extension you reach for.

<CodeVariants>
  <CodeVariant label="Rename">
    ```ts
    const { name: customerName } = customer;
    ```
    **Rename.** Reach for this when the outer scope already binds `name` (which would shadow), or when the local context wants a more specific name than the field. The syntax reads backwards from the object-literal shorthand you might expect: left of `:` is the field name in the source, right of `:` is the local binding being created. `customerName` is now in scope; `name` is not. This is the part students mis-read on first contact — re-read this paragraph if it surprised you.
  </CodeVariant>

  <CodeVariant label="Default">
    ```ts
    const { pageSize = 20 } = customer;
    ```
    **Default.** Reach for this when the field may be missing and you have a sensible substitute. Same nullish-vs-undefined semantic as parameter defaults from the previous lesson — the default fires **only on `undefined`**, never on `null`, `0`, `''`, or `false`. With the `customer` object above where `pageSize: 0`, the default does *not* fire — `pageSize` is bound to `0`, preserved as the caller intended. Same reasoning as `??` over `||` from the previous lesson — the language draws the line at "missing," not "falsy."
  </CodeVariant>

  <CodeVariant label="Combined rename + default">
    ```ts
    const { pageSize: limit = 20 } = customer;
    ```
    **Combined rename + default.** The experienced form when local naming and a default are both wanted. Read order: **rename first, then the default applies to the renamed binding.** The default expression sees the renamed identifier (`limit`), not the original field name (`pageSize`). Useful when an inbound field has a generic name and the local context wants something more specific without losing the missing-field safety net.
  </CodeVariant>

  <CodeVariant label="Rest">
    ```ts
    const { id, ...rest } = customer;
    ```
    **Rest.** Pulls `id` out by name and collects every remaining field into a new object bound to `rest`. The single most common use in 2026 SaaS code is omitting a field before forwarding the others — covered in depth in the destructure-then-rebuild section below. One syntactic rule: `...rest` must be the *last* element in the pattern; the parser rejects it anywhere else.
  </CodeVariant>
</CodeVariants>

The shape that trips first-time readers is the rename. Lock the left-versus-right reading now, before it ships a shadow bug in real code.

<MultipleChoice>
  Given the line below, which name is now bound as a local variable, and which name was the original field on the source object?

  ```ts
  const { foo: bar } = obj;
  ```

  <McqChoice>
    `foo` is the new local binding; `bar` was the field name on `obj`.
  </McqChoice>
  <McqChoice correct>
    `bar` is the new local binding; `foo` was the field name on `obj`.
  </McqChoice>
  <McqChoice>
    Both `foo` and `bar` are now bound as local variables.
  </McqChoice>
  <McqChoice>
    Neither is bound; this is a syntax error.
  </McqChoice>

  <McqWhy>The destructure rename reads **field-on-source `:` local-binding**. The field that existed on `obj` is `foo`; the variable the line creates in your scope is `bar`. The reading direction is the reverse of an object-literal shorthand (`{ bar: foo }` in an object literal would store `foo`'s value under the key `bar`), which is why the rename is the single most mis-read piece of destructuring syntax. If the answer above surprised you, that's the part to slow down on.</McqWhy>
</MultipleChoice>

## Array destructuring: the two daily reaches

The companion form, much shorter. Where object destructuring picks fields **by name** and is order-independent, array destructuring binds **by position**. The shape:

```ts
const pair = ['admin', true];
const [role, isActive] = pair;
const [, isActive2] = pair;
const [first, ...rest] = ['a', 'b', 'c'];
```

Three lines, three forms. The first binds the two positions of the array to local variables in order. The second uses a comma-hole to skip the first position and bind only the second — useful when you only need one slot out of a tuple. The third is the head-and-rest split: bind the first item, collect every remaining item into a new array.

The trigger for reaching at all is the discipline: **use array destructuring when the data is meaningfully ordered.** A `useState` return value where position 0 is the state and position 1 is the setter. An `Object.entries` pair where position 0 is the key and position 1 is the value. An HTTP method-and-path tuple where the order is part of the contract. Don't reach for it on a plain array of records where position carries no semantic — there, you're better off iterating and naming the item.

:::note
Array destructuring is **less common than object destructuring in SaaS code** — most domain data is shaped (records, options objects, props), not positional. The reason to learn the form is the React hooks surface in Unit 3, where `useState` returns a tuple by design and the destructure is how every component reads it. The first time you write `const [count, setCount] = useState(0)`, the form is already decided.
:::

You'll meet this form in two specific places later: `useState` returns `[value, setter]` tuples that you destructure positionally in the chapter on the React render model, and `Object.entries` yields `[key, value]` pairs that you destructure inside `for...of` in the next chapter on containers.

## Signature-level destructure: the canonical 2026 shape

This is the headline. The literal form every Server Action, every React component, and every options-object helper in this course will write. The destructure happens **at the parameter position** — the function unpacks its options object the moment the call arrives, and the body reads the same identifiers the call site wrote.

<CodeTooltips tooltips={{
  'customerId: string;': "Inline object-type annotation — the function's parameter type is defined right at the parameter position, no separate `type` alias needed. Full mechanics in the chapter on TypeScript object shapes.",
  'notes?:': "The `?` marks the field as optional in the type — the caller may omit it. The field-level default below fires only when it's omitted, same rule as parameter defaults from the previous lesson.",
  "notes = ''": "Field-level default — fires only when the caller omits the field or passes `undefined`. Anywhere else the caller's value is preserved.",
}}>

```ts
const createInvoice = ({
  customerId,
  amountCents,
  notes = '',
}: {
  customerId: string;
  amountCents: number;
  notes?: string;
}) => {
  // body reads the same names as the call site
};
```

</CodeTooltips>

Three payoffs make this shape the default every framework in the course expects:

1. **Names match the call site.** The caller writes `createInvoice({ customerId, amountCents })`; the body reads `customerId` and `amountCents`. No `const customerId = options.customerId` boilerplate, no mental remap between the call and the implementation. Both sides of the function share one identifier vocabulary.
2. **Adding a field never breaks existing callers.** Field order is irrelevant — the destructure picks by name. A new optional field on the type adds a capability without touching any call site that doesn't use it.
3. **Defaults live at the signature.** The body never starts with `const notes = options.notes ?? ''`. The default is *at the field*, visible in the signature where a reviewer looks for the contract.

This is the shape every options-object function in the rest of the course will write. React component props in the chapter on components and composition are this shape, spelled in JSX. Server Action input in the chapter on the server / client boundary is this shape. The Drizzle query-builder helpers in the Postgres unit take their options this way. Once the form is in your fingers, you'll reach for it on every function past two parameters.

One last call to make explicit: **signature destructure is the default for API-shaped functions.** The alternative — destructure on the first line of the body, with the parameter named `options` — is the right reach only when the original `options` reference needs to stay in scope (for logging the full payload, forwarding it to an SDK, or referencing it more than once before the destructure can cover every use). Don't dwell on the variant; reach for the signature form first.

## Destructure-then-rebuild: the no-accidental-forwarding pattern

The mechanics above are the *form*. This section is the *discipline*, and it's the structural fix for the wholesale-forward bug from the lesson's first paragraph.

**When forwarding data to a downstream call, destructure exactly the fields needed and rebuild the object literal that goes downstream. Never forward the original object wholesale.**

Walk the before-and-after. The shape is identical; the data flowing out is what changes.

<CodeVariants>
  <CodeVariant label="Wholesale forward leaks internalNotes">
    <div data-mark-color="orange">

    ```ts {3}
    const sendInvoice = (input: SendInvoiceInput) => {
      db.insert(invoicesTable).values(input);
      email.send({ to: input.customerEmail, ...input });
    };
    ```

    </div>

    A `sendInvoice` handler receives an invoice payload — `{ customerEmail, amountCents, internalNotes }` — and forwards the whole `input` to two downstream callees. The database insert is the row's genuine home. The `email.send` call is the leak: the customer-facing email body now interpolates every field on `input`, including `internalNotes` ("flag for collections — third late payment"). Nothing at this call site says "internalNotes" — that's exactly why the bug is invisible to a reviewer skimming the function. Worse, when someone upstream adds a new admin-only field to `SendInvoiceInput` (a `riskScore`, a `discountReason`, a `routingMemo` from a future invoicing flow), it silently lands in the customer's inbox too, with zero code change here.
  </CodeVariant>

  <CodeVariant label="Destructure to the exact downstream shape">
    <div data-mark-color="green">

    ```ts {1,3}
    const sendInvoice = ({ customerEmail, amountCents, internalNotes }: SendInvoiceInput) => {
      db.insert(invoicesTable).values({ customerEmail, amountCents, internalNotes });
      email.send({ to: customerEmail, amountCents });
    };
    ```

    </div>

    The signature destructures every field by name. The downstream calls receive object literals built from those named fields — and only those. `email.send({ to: customerEmail, amountCents })` cannot receive `internalNotes` because the developer didn't write it into the object. When `SendInvoiceInput` grows a new admin-only field tomorrow, the email line still ships exactly `{ to: customerEmail, amountCents }` — the new field literally has no path to it. The form is structural: no leak is possible from this shape without the developer choosing to put it there.
  </CodeVariant>
</CodeVariants>

This is the same principle as the options-object pattern from the previous lesson — **make the surface area visible at the call site** — applied to the *outgoing* side of a function. The options object made the inputs explicit; the rebuilt object literal makes the forwarded fields explicit. Both win for the same reason: a new field added somewhere else can't silently reach a new place. It only flows where a developer, today, types its name.

The rest extension from earlier in the lesson now earns its load-bearing use: omit a specific field by name and forward everything else deliberately.

```ts
const { internalNotes, ...customerSafeFields } = invoice;
return customerSafeFields;
```

This is the canonical "strip a sensitive field" idiom. It's the shape a `serializeInvoice` helper takes before sending a record to the client, the shape an email adapter takes before composing a customer-facing message, and the shape any "send to third party" function takes when most of the record is safe but one field isn't. Note the asymmetry: when you're *picking* fields to forward, name them positively (`{ customerEmail, amountCents }`); when you're *omitting* one specific field from an otherwise-safe shape, the rest pattern is the cleanest reach. Same principle, different direction.

The Server Actions chapter later in the course applies this same discipline to the fields passed into Drizzle's `.values(…)` call — only what was destructured by name reaches the table. The form is decided here; the application is the rest of the course.

## The nested-destructure watch-out

One mechanic to flag before the closing exercise. Destructuring can nest — you can reach two or three levels into an object in a single pattern — and the moment a nullable link sits in the middle of that chain, the form throws at runtime.

```ts
const { profile: { address: { city } } } = user;
```

That line throws if either `user.profile` or `user.profile.address` is nullish — the destructure machinery has to dereference each level to reach `city`, and dereferencing `undefined` is a `TypeError`. The fix isn't a clever nested-destructure trick; the fix is the `?.` operator from the previous lesson:

```ts
const city = user.profile?.address?.city;
```

Optional chaining was built for exactly this shape — "step through nullable links and short-circuit safely" — and it reads more cleanly at every level past the first.

The rule: **write shallow destructures by default and reach for `?.` for nullable paths.** Nested destructure is legal and occasionally the right call when the shape is genuinely required (a hardcoded config, a result from a schema-validated parse). Past one level of nesting, the reader cost spikes and the `?.` form usually wins on clarity *and* safety. Pick the form that matches the type's nullability story — the same discipline from the previous lesson's overuse-trap section, applied here.

## Recognize the forms

Close the lesson by sorting six destructure snippets into the three buckets that map to the three pieces of the model: **syntactic validity** (does the parser accept it?), **semantic correctness** (does the default fire when you'd expect?), and **runtime safety** (does it throw on the given input?).

<Buckets twoCol instructions="Sort each destructure snippet by what it would do — given the input noted in the comment, where applicable.">
  <Bucket name="valid" label="Valid" description="Parses fine, won't throw on the given input." />
  <Bucket name="wrong" label="Syntactically wrong" description="The parser rejects this; TypeScript flags it before runtime." />
  <Bucket name="throws" label="Throws at runtime" description="Parses fine; throws a TypeError when a nullish link is dereferenced." />

  <Item bucket="valid">
    ```ts
    const { name, email } = customer;
    ```
  </Item>
  <Item bucket="throws">
    ```ts
    // Given: user.profile = null
    const { profile: { address } } = user;
    ```
  </Item>
  <Item bucket="valid">
    ```ts
    const { name = 'Anon' } = { name: '' };
    ```
  </Item>
  <Item bucket="wrong">
    ```ts
    const { ...rest, id } = customer;
    ```
  </Item>
  <Item bucket="wrong">
    ```ts
    const { id: , name } = customer;
    ```
  </Item>
  <Item bucket="valid">
    ```ts
    // Given: order.customer = undefined
    const { customer: { id } = {} } = order;
    ```
  </Item>
</Buckets>

Sorting these six is the recall check for the three pieces of the model: the *form* (what the parser accepts), the *semantics* (defaults fire only on `undefined`), and the *runtime safety* (whether the chain dereferences a nullish link). Past this point in the course, every options-object function you write — every Server Action, every React component, every Drizzle helper — opens with a destructure that obeys all three rules.
