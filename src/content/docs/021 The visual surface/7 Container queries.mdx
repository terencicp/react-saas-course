In the last lesson you left a card unfinished. The same `<ProductCard>` looked right in a wide feed and cramped in a narrow sidebar, and you couldn't fix it with `md:` — because both slots share one viewport, and a media query only ever reports the size of the *screen*. The card's problem was never the screen. It was the card's own width, and the viewport has no way to tell you that. Container queries close that gap: you mark a wrapper as measurable with `@container`, and the children inside ask *that wrapper's* width with `@md:flex-row` instead of the window's. By the end of this lesson you'll have one card, authored once, that lays itself out correctly in the dashboard grid, the sidebar, and the full-width feed — the parent none the wiser — plus a crisp rule for when this beats a media query. Start by dragging a container narrower and watching the card decide its own layout.

## A component that adapts to its slot, not the screen

Here's the target. A product card that stacks its image on top of its text when its slot is narrow, and sits image-left, text-right when its slot is wide — and it figures that out *itself*, without the parent passing down a `variant="compact"` prop, without a `useState`, without measuring anything in JavaScript. The component owns its own breakpoint. Drop it in a three-column dashboard grid and each cell is narrow, so it stacks. Drop it in a sidebar and it stacks. Drop it in a full-width feed row and it goes horizontal. One component, three correct layouts, and the parent never told it which.

The shape that makes this work is two pieces. First, the parent **opts in** to being measurable:

```tsx
<div className="@container">…</div>
```

That `@container` is the wrapper saying *"I am a box whose width can be queried."* Second, a child inside it **queries that width**:

```tsx
<div className="flex flex-col @md:flex-row">…</div>
```

That reads *"stack by default; go side-by-side once my container passes the `@md` width."*

Hold on the difference between `@md:` and the `md:` you learned in the last lesson, because it's the whole idea in one character. In `md:flex-row`, the `md` is the **screen** — it fires at a viewport width. In `@md:flex-row`, the `@md` is **this container** — it fires at the wrapper's width. The `@` prefix is the entire tell. Same grammar, different yardstick: one measures the window, the other measures the box the component happens to land in.

Before naming any of the underlying CSS, get the effect in your hands. The slider below drives a real container's width. The card inside reacts through genuine container-query CSS — not a fake, an actual `@container` rule. Drag it from narrow to wide and watch two things happen at once: the layout flips from stacked to side-by-side, and the title grows smoothly with the box.

What you just did is the entire lesson in one loop: you changed the box's width and the component re-laid-itself-out. The wrapper had a property that said "measure me," and the card inside read that measurement and reacted. Everything from here is putting precise names on those two moves — starting with what the wrapper is actually measuring, which is its <Term definition="The element's width axis in writing-mode-aware terms. For left-to-right Latin text, the inline axis is the horizontal one — the direction the text flows.">inline-size</Term>.

## The model: a parent opts in, descendants ask

Now that you've felt it, name it precisely. A container query can't just point at any element and ask its width. An ancestor has to first *declare itself queryable*, and it does that with the `container-type` property. Think of it as a contract: measuring a box costs the browser extra layout work, so you opt in per-subtree rather than paying that cost on every element on the page. That's why the parent declaration exists at all, and it's also why the most common beginner bug is forgetting it — more on that in a moment.

The value you reach for almost every time is `container-type: inline-size`. It makes the container queryable on its **inline (width) axis only**, and leaves its block size — its height — content-driven, free to grow with whatever's inside. That asymmetry is exactly what you want in practice. You almost always care about *"how wide is my slot"* while letting the box get as tall as its content needs. Querying width while height flows naturally is the 99% case, which is why it's the default. In Tailwind, **`@container` compiles to `container-type: inline-size`** — that single utility is the opt-in.

Once an ancestor is a container, any descendant can query it. A rule written as `@container (width >= 28rem) { … }` measures *that ancestor*, not the viewport. In Tailwind that's the family of `@`-prefixed variants — `@md:flex-row`, `@lg:grid-cols-2`, and so on — each compiling to a `@container (width >= <size>)` block. The `@` is what separates a container variant from a viewport variant. That's the payoff of the variant grammar from the last lesson: you already read `md:` as "at a screen width," so `@md:` reads as "at a container width" with no new mental machinery.

One rule that trips everyone up at first: **the container is always an ancestor — a query never reads the element it's written on.** The `@container` declaration goes on a wrapper *up the tree*; the `@md:` query goes on a descendant *inside* it. Put both on the same div and nothing happens, because that div is asking about its own width, and a container query only ever looks upward for the nearest container.

Here's the two-piece pattern in Tailwind, walked one part at a time so you can see the CSS each utility compiles to.

<AnnotatedCode lang="tsx" code={`
<article className="@container">
  <div className="flex flex-col gap-4 @md:flex-row">
    <img className="rounded-md @md:w-40" />
    <div className="@md:flex-1">…</div>
  </div>
</article>
`}>
  <AnnotatedStep meta={`"@container"`} color="blue">
    The opt-in. `@container` compiles to `container-type: inline-size`, so this `<article>` is now measurable on its width axis. Nothing about it *looks* different yet — it has simply declared itself queryable.
  </AnnotatedStep>

  <AnnotatedStep meta={`"flex flex-col"`} color="blue">
    The default, narrow layout: a vertical stack with the image above the text. These are unprefixed utilities, so they always apply — this is what renders until a container query overrides it.
  </AnnotatedStep>

  <AnnotatedStep meta={`"@md:flex-row"`} color="blue">
    The query. `@md:flex-row` compiles to `@container (width >= 28rem) { flex-direction: row }` — it flips to side-by-side once the *article* is at least 28rem wide, regardless of screen size. The `@` is what makes it read the container, not the viewport.
  </AnnotatedStep>

  <AnnotatedStep meta={`"@md:w-40" "@md:flex-1"`} color="blue">
    Per-element overrides inside that same `@md` query: the image takes a fixed `10rem` width while the text column takes the remaining space. Each utility carries its own `@container (width >= 28rem)` wrapper.
  </AnnotatedStep>
</AnnotatedCode>

You don't need to second-guess whether you can use this. Size container queries and the container units that go with them are <Term definition="A web feature supported across all major current browsers — safe to use without a fallback.">Baseline</Term> — cross-browser since early 2023, Baseline *widely available* since 2025 — so in 2026 they're universal, with no polyfill story. Reach for them without checking a support table.

:::caution
The number-one container-query bug is forgetting `@container` on an ancestor. When the wrapper isn't a container, the `@md:` variants don't error — they just silently never match, and your card stays stacked at every width. If your `@md:` styling isn't firing, your first move is to confirm some ancestor actually has `container-type` set. The browser isolating a subtree's layout this way so it can be measured independently is called <Term definition="The browser isolating a subtree's layout so it can be measured and queried independently from the rest of the page.">containment</Term>, and without it there's nothing to query.
:::

There are two other `container-type` values, named here only so you recognize them. `container-type: size` queries *both* axes, but it requires the container to have a defined height — which is footgun-prone, because the content inside can no longer size the box that's measuring it, and it's easy to collapse the whole thing to zero. Reach for it almost never; `inline-size` is the default. And `container-type: normal` simply removes containment. Tailwind exposes the size variant as `@container-size`, but you'll rarely write it.

## The container breakpoint scale is smaller than the viewport scale

When you reach for `@md:` you're using a breakpoint — but it is *not* the same 768px your `md:` viewport breakpoint fires at. This matters, so make it concrete: a sidebar widget at 400px wide is *roomy* for its slot but *tiny* for a screen. If the container scale reused the viewport numbers, `@md` would never fire inside anything but a full-width region. So Tailwind ships a separate, smaller scale for containers. The clearest contrast: viewport `md` is **768px**, while container `@md` is **448px (28rem)**. Same name, different yardstick — because one is measuring a screen and the other a box.

The full scale runs from `@3xs` up to `@7xl`. You don't need to memorize it; the diagram below is for orientation, and the reflex is the same one from the last lesson — *stay on the scale, and find the breakpoint by resizing until the layout breaks*, now applied to the container rather than the viewport.

When a one-off threshold is genuinely the right call, the arbitrary form is `@min-[475px]:` (and `@max-[960px]:` for the other direction) — but treat that the way you treat any bracket value: it's an escape hatch, not the first reach. If a project consistently needs another step, grow the scale instead of littering brackets, by adding to the `--container-*` namespace in your theme:

```css
@theme {
  --container-8xl: 96rem;
}
```

That mints an `@8xl` variant you can use everywhere, exactly like the built-in steps. The container scale also has `max-*` and ranged variants — `@max-md:` for "below the container's md," and `@sm:@max-md:` to target a single band — but you'll reach for those rarely; recognize them when you see them.

## Fluid component typography with cqi and clamp()

Discrete breakpoints are only half of what containers give you. The other half is **continuous** sizing, and it's one of the highest-value patterns of 2026. The mechanism is container query units, and they map cleanly onto the viewport units you already know. Where `1vw` is 1% of the viewport's width, `1cqi` is 1% of the *container's* inline size. The full set exists — `cqb` is 1% of block size, `cqw` and `cqh` are the explicit width and height forms, `cqmin` and `cqmax` are the smaller and larger of the two — but in practice <Term definition="Container query inline-size unit: 1cqi is 1% of the nearest queried container's inline (width) size.">cqi</Term> is the reach for almost everything, for the same reason `inline-size` is the default `container-type`: width drives most component sizing.

Here's where it pays off. A card title that reads at ~16px in a small card and ~24px in a large one is not a stack of breakpoints — it's a single declaration:

```css
font-size: clamp(1rem, 6cqi, 1.5rem);
```

`clamp()` takes three arguments — a minimum floor, a preferred value, and a maximum ceiling. Here the floor is `1rem` (16px), the ceiling is `1.5rem` (24px), and the preferred value is `6cqi`: 6% of the container's width. As the card grows, `6cqi` grows with it, and the title scales smoothly between its floor and ceiling. No `@md:text-2xl` step, no sudden jump at a breakpoint — the title grows *continuously* with the slot. This is exactly what you watched happen in the lab: the title you saw scale on the slider was this one `clamp(1rem, 6cqi, 1.5rem)` rule.

Now a divergence worth calling out, because it looks like it breaks a rule you've been taught. Tailwind has no first-class `cqi` utilities. You write the fluid value as an arbitrary value: `text-[clamp(1rem,6cqi,1.5rem)]` for the title, or `p-[3cqi]` for padding that scales with the card. That is *not* the bracket smell the earlier lessons warned you off. The smell is reaching for `text-[17px]` when a scale step would do — inventing a one-off where the system already has an answer. Here there is no scale step that expresses "6% of the container's width that clamps between two bounds." The value *is* a fluid expression, so the escape hatch is the correct and only tool. Stay on the scale for the things the scale covers; reach for brackets when the value is genuinely a computation.

:::caution
Container units resolve to **0** when no ancestor is a container. A `6cqi` title with no `container-type` anywhere up the tree silently becomes 0px — invisible text, no error. This is the same root cause as the missing-`@container` bug from earlier, so collapse both into one rule and remember it as one: container features — both the `@md:` variants and the `cqi` units — need a declared container above them, full stop.
:::

## Naming a container so deep children query the right one

Everything so far assumed one container. Add a second and a new problem appears. An unnamed `@container` query always binds to the *nearest* container ancestor. So when containers nest — a card inside a panel, and both are containers — a deep child that wants to react to the *outer* panel can't reach past the inner card to do it. By default a query grabs the closest container, and sometimes the closest one isn't the one you mean.

The fix is to give the container a name and query it by name. In CSS that's `container-name` alongside `container-type`, usually written with the `container` shorthand: `container: panel / inline-size`. Then a descendant addresses it explicitly: `@container panel (width >= 28rem) { … }`. In Tailwind, the name rides along as a `/` suffix on both ends — `@container/panel` on the ancestor, and `@md/panel:flex-row` on the descendant. Naming still implies `inline-size`, so you're not giving anything up.

<CodeVariants syncKey="container-naming">
  <CodeVariant label="Unnamed (ambiguous)">
    <div data-mark-color="orange">

    ```tsx "@container" "@md:flex-row"
    <section className="@container">     {/* outer panel */}
      <article className="@container">   {/* inner card */}
        <div className="flex flex-col @md:flex-row">…</div>
      </article>
    </section>
    ```

    </div>
    **The `@md:` binds to the *nearest* container — the inner card, not the outer panel.** You wanted the layout to respond to the panel's width, but the inner `@container` is closer, so it wins. There's no error; the query just silently measures the wrong box.
  </CodeVariant>

  <CodeVariant label="Named (explicit)">
    <div data-mark-color="green">

    ```tsx "/panel"
    <section className="@container/panel">  {/* named */}
      <article className="@container">
        <div className="flex flex-col @md/panel:flex-row">…</div>
      </article>
    </section>
    ```

    </div>
    **The `/panel` suffix points the query past the inner card at the named ancestor.** Now `@md/panel:` measures the outer panel regardless of any container between them. The syntax delta is tiny — a `/panel` on each end — but the targeting is what changed.
  </CodeVariant>
</CodeVariants>

Notice how small the delta is — a `/panel` on each end — and how much it changes: the query now measures the box you intended, not the one that happened to be closest. That's the whole reason naming exists, and it's also why you reach for it sparingly. Most components have exactly one container and never need a name; pull naming out specifically when the structure nests and an inner container would otherwise win. (And that's the underlying behavior to remember: an inner unnamed container *shadows* an outer one for unnamed queries — the nearest one wins — and a name is how you bypass it.)

## Choosing between viewport and container queries

You now hold both tools, which means you've reached the part of responsive design that actually takes judgment: knowing which one a given problem wants. The rule is the one the last lesson planted, now fully cashed in:

**Page-level structure → viewport query (`md:`). Component-level adaptation → container query (`@md:`).**

And the part that surprises people: most real SaaS UIs use *both at once*. The page shell is viewport-driven — the navigation switches from a hamburger to a full bar, the layout splits from one column to two — because those decisions genuinely depend on the screen. The components living inside that shell are container-driven — a card adapts to its grid cell, a sidebar widget collapses with its parent — because those depend on the slot, not the window.

When you're unsure which a given element wants, ask one question: *does this element's right layout depend on the screen, or on the space it happens to be in?* Screen means `md:`. Slot means `@md:`. The top-level app navigation depends on the screen — viewport. A `<ProductCard>` reused across the feed, the sidebar, and the grid depends on its slot — container. A stat tile that's four-across on a wide dashboard and one-across when the grid drops to a single column is changing because *the grid* changed, not because the screen did — container.

There's one more tool in this neighborhood you already know, and it's easy to confuse with container queries because both are "responsive without breakpoints." From the grid lesson, `grid-cols-[repeat(auto-fit,minmax(280px,1fr))]` makes the **container** respond to its *items* — the track count flexes to fit however many cards there are, but each card stays internally the same. A container query does the opposite: `@container` plus `@md:flex-row` makes the **items** respond to the *container* — each card re-lays-itself-out based on the width it ended up with. They're not competitors; they compose. `auto-fit` decides *how many cards fit per row*, and each card's own `@container` decides *its internal layout* given the cell width that came out of that. That pairing is the 2026 reflex for a card grid that's fluid all the way down.

The flip side of the rule is knowing when *not* to reach. The page shell rarely needs container queries; over-containerizing adds layout cost and a layer of indirection for no gain. Use viewport queries where viewport is the honest answer.

Sort these real scenarios into the tool each one calls for. The point isn't to recall the rule — it's to *apply* it, which is the skill you'll use every time you build a screen.

## Practice: make the card adapt to its slot

Time to write the canonical pattern yourself. The exercise below renders the *same* `<ProductCard>` twice — once in a narrow wrapper, once in a wide one — so a single correct card definition produces a stacked layout in the narrow slot and a side-by-side layout in the wide slot at the same time. That's the thesis of the whole lesson made literal: one component, two layouts, neither wrapper telling the card which to use. Right now the starter card stacks in both slots, which is wrong — the wide slot should go horizontal. Add the container setup and watch the card in the wide slot flip while the narrow one stays stacked.

<ReactCoding
  live
  instructions={`Make the card lay itself out from its slot. Add \`@container\` to the card root, then \`@md:flex-row\` to the inner layout (with \`@md:w-40\` on the image and \`@md:flex-1\` on the body). The same component should render stacked in the narrow slot and side-by-side in the wide slot — without either wrapper telling it which.`}
  starter={`function ProductCard() {
  return (
    <div className="rounded-lg border p-4">
      <div className="flex flex-col gap-4">
        <div className="h-28 rounded-md bg-zinc-200" />
        <div>
          <h3 className="font-semibold">Wireless Headphones</h3>
          <p className="text-sm text-zinc-500">$129</p>
        </div>
      </div>
    </div>
  );
}

export function App() {
  return (
    <div className="flex flex-col gap-6">
      {/* narrow slot — the card should stay stacked here */}
      <div className="w-64"><ProductCard /></div>
      {/* wide slot — the same card should go side-by-side here */}
      <div className="w-[34rem]"><ProductCard /></div>
    </div>
  );
}`}
  target={`function ProductCard() {
  return (
    <div className="@container rounded-lg border p-4">
      <div className="flex flex-col gap-4 @md:flex-row">
        <div className="h-28 rounded-md bg-zinc-200 @md:w-40" />
        <div className="@md:flex-1">
          <h3 className="font-semibold">Wireless Headphones</h3>
          <p className="text-sm text-zinc-500">$129</p>
        </div>
      </div>
    </div>
  );
}

export function App() {
  return (
    <div className="flex flex-col gap-6">
      {/* narrow slot — the card stays stacked here */}
      <div className="w-64"><ProductCard /></div>
      {/* wide slot — the same card goes side-by-side here */}
      <div className="w-[34rem]"><ProductCard /></div>
    </div>
  );
}`}
/>

If the card in the wide slot went horizontal while the one in the narrow slot stayed stacked, you've built a genuinely portable component. It carries its own layout logic and reads it off whatever space it's given — no prop, no state, no JavaScript measuring the DOM.

## Wrapping up

That completes the visual surface. Across this chapter you've installed everything that sits on top of layout — how text reads, how color renders, how an element decorates itself and responds to interaction, how it moves, and now how it adapts, both to the viewport and to the box it lives in. The reflex to carry forward is the one this lesson is built around: author components that lay themselves out from the space they're given, rather than depending on a parent to hand down the right variant. That's what makes a component genuinely reusable, and it's exactly the foundation the next chapter builds on when you start composing these adaptive pieces into larger ones.
