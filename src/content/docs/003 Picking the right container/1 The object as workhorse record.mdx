Two real bugs, same root.

```ts
const config: Record<string, string> = { theme: 'dark', locale: 'en' };
const userInput = 'toString';
const value = config[userInput];
// value is [Function: toString] — the prototype chain leaked
// into application logic
```

```ts
const invoice = { id: 'inv_001', amountCents: 4900, status: 'pending' };
const patch = { id: 'inv_TYPO', status: 'paid' };
const updated = { ...invoice, ...patch };
// updated.id is 'inv_TYPO' — the patch silently overwrote the ID
// and the customer just got a new invoice number
```

The first bug walked into `Object.prototype` because the code used bracket access on a key the user controls. The second overwrote a field because the spread followed the rule "right-most key wins" and the patch happened to carry an `id`. Two object-handling bugs, both prevented by reaching for the form that says what you actually mean. This lesson installs the daily reach for three jobs you do with every record-shaped value in a SaaS codebase: *building* it, *reading* it, and *reshaping* it.

## Dot by default, brackets on trigger

Dot access is the default form for reading a field off an object. It's static, type-checked against the known shape, and the form every reader expects to see.

<CodeTooltips tooltips={{
  amount: "Inferred as `number` — dot access on a typed shape returns the field's type.",
  value: "Inferred as `number | undefined` — under `noUncheckedIndexedAccess`, bracket access on a `Record<string, T>` returns `T | undefined` because TypeScript can't know which keys exist at runtime.",
  'Record<string, number>': "An object type with dynamic string keys, all of value type `number`. The mapped-type form gets its own lesson in the next chapter — for now, read it as `{ [key: string]: number }`.",
}}>
```ts
const invoice = { amountCents: 4900, status: 'pending' };
const amount = invoice.amountCents;

const fields: Record<string, number> = { amountCents: 4900 };
const value = fields['amountCents'];
```
</CodeTooltips>

Bracket access is the *conditional* form. Three triggers earn it, and only these three:

1. **The key isn't a valid identifier.** `style['background-color']`, `payload['user.email']` — anything with a hyphen, dot, or other character JavaScript identifiers can't carry.
2. **The key is held in a variable.** `obj[fieldName]` in a generic helper that walks a record by a key passed in.
3. **The key is computed at runtime.** Factory helpers, schema mappings — patterns where the key is built up from data, not written at the call site.

If none of these fires, reach for the dot. Juniors sometimes reach for brackets because they "feel more dynamic" — that's a tic, not the senior reach. The dot reads as intent (*"this field, by name"*); the bracket reads as escape (*"this key, computed somehow"*). The two convey different things to whoever reads the code next.

The type difference is the other half of the story. Under <Term definition={"The strict tsconfig flag the course pins on.\nMakes any indexed/bracket read return T | undefined instead of T,\nbecause TypeScript can't prove the key exists at runtime."}>`noUncheckedIndexedAccess`</Term>, bracket access on a <Term definition={"An object type with dynamic keys, all values of type T.\nRead `Record<string, number>` as 'object whose keys are strings, all values are numbers.'\nThe mapped-type form lands in its own lesson in the next chapter."}>`Record<string, T>`</Term> returns `T | undefined`. Dot access on a known shape returns `T`. The strictness is doing the right thing — when you're reading by a variable key, you genuinely don't know whether the key is present, and the type forces you to handle that.

## Building records: shorthand, computed keys, spread

Three sugars build object literals. Each maps to a specific shape of operation, and each shows up in production every day.

<CodeVariants>
  <CodeVariant label="Shorthand">
    <div data-mark-color="green">

    ```ts
    const name = 'Lina Park';
    const email = 'lina@acme.test';

    const customer = { name, email };
    ```

    </div>
    **Same-name fields collapse.** When the local binding's name matches the field name, write the name once. The reflex for every object literal built from local variables.
  </CodeVariant>

  <CodeVariant label="Computed key">
    <div data-mark-color="green">

    ```ts
    const fieldName = 'amountCents';
    const value = 4900;

    const patch = { [fieldName]: value };
    ```

    </div>
    **Bracket-on-the-left for a dynamic key.** Rare in 2026 app code — most keys are known at write time — but the form factory helpers and reducer patterns reach for. Same three triggers as bracket *reads*, applied to the build site.
  </CodeVariant>

  <CodeVariant label="Spread">
    <div data-mark-color="green">

    ```ts
    const base = { id: 'cus_001', name: 'Lina Park', plan: 'free' };
    const updated = { ...base, plan: 'pro' };
    ```

    </div>
    **Shallow merge, right-most key wins.** The form React state setters, Server Action input merging, and Drizzle update payloads all consume. Putting `plan: 'pro'` after `...base` overrides the base's `plan` — if the order were reversed, the override would lose.
  </CodeVariant>
</CodeVariants>

That "right-most key wins" rule is what bit the second bug at the top of the lesson. Spread is shallow — already named back in the value-model lesson, no need to re-derive — and a key on either side of the merge claims the slot if it appears later. Whoever wrote `{ ...invoice, ...patch }` was thinking *"apply the patch"*, but if the patch carried an `id` field the writer didn't expect, the merge happily honored it. The fix isn't to memorize the rule; it's to spread only the fields you mean to override, or to validate the patch shape at the boundary so a stray `id` never reaches the merge.

## Is the key — or the value — there?

Three checks sit close enough together that juniors conflate them, but they ask three different questions — *is the key present at all*, *is it present as an own key (not inherited)*, and *is there actually a usable value behind it*. The differences matter. Sorting them out is the reflex this section retires.

<CodeVariants>
  <CodeVariant label="in (legacy)">
    <div data-mark-color="green">

    ```ts
    const invoice = { amountCents: 4900, status: 'pending' };

    'amountCents' in invoice;   // true — own key
    'toString' in invoice;      // true — inherited from Object.prototype
    ```

    </div>
    **Walks the prototype chain.** Returns `true` for inherited keys too. Almost always the wrong reach in 2026 app code — the form to recognize in older codebases, not to write fresh.
  </CodeVariant>

  <CodeVariant label="Object.hasOwn">
    <div data-mark-color="green">

    ```ts
    const invoice = { amountCents: 4900, status: 'pending' };

    Object.hasOwn(invoice, 'amountCents');   // true
    Object.hasOwn(invoice, 'toString');      // false — inherited, not own
    ```

    </div>
    **Own keys only.** The 2026 reach when the question genuinely is *"is this key present on the object itself, ignoring the prototype chain"* — reading user-controlled keys from a parsed JSON payload, defending against an attacker who put `toString` in their input.
  </CodeVariant>

  <CodeVariant label="?? fallback">
    <div data-mark-color="green">

    ```ts
    const invoice = { amountCents: 4900, status: 'pending' };

    const amount = invoice.amountCents ?? 0;
    const note = invoice.note ?? 'no note';
    ```

    </div>
    **Is there a *value* here?** What application code usually wants. Asks neither "own vs inherited" nor "key present" — it asks "did I get something usable, or should I fall back?" The reflex when reading optional fields off a typed shape. `??` falls back only on `null` or `undefined`, never on `0` or `''`.
  </CodeVariant>
</CodeVariants>

That `'toString' in invoice` returning `true` is the same family of bug as the chapter opener's `config[userInput]` returning `Object.prototype.toString`. Both reach into the prototype chain through code that meant to ask about own properties. `Object.hasOwn` is what `'foo' in obj` should have been from the start — own-property only, no inheritance walk. Whenever the input to a presence check could be controlled by a user, `Object.hasOwn` is the reach.

:::note
You may see `Object.prototype.hasOwnProperty.call(obj, 'foo')` in older code — that was the awkward incantation `Object.hasOwn` replaced in ES2022. Same answer, less ceremony. Write `Object.hasOwn` in fresh code; recognize the older form when reading codebases that predate it.
:::

## The Object.* methods worth knowing

The `Object` constructor carries a handful of static methods that handle the daily reshape jobs — iterating, building from pairs, merging, freezing, grouping. They're the surface that retired the lodash habit.

### Object.keys, Object.values, Object.entries — the iteration triad

`Object.keys(obj)` returns the own enumerable string keys, `Object.values(obj)` returns the corresponding values, `Object.entries(obj)` returns `[key, value]` pairs. All three return arrays in insertion order, which drives `for...of` over an object (already met in the flat-control-flow lesson as the `for...in` replacement; the dedicated iteration treatment lands later in this chapter) and `.map` over key-value pairs.

<CodeTooltips tooltips={{
  keys: "Inferred as `string[]`, not `('amountCents' | 'status')[]`. TypeScript widens because at runtime, `invoice` could have more keys than its compile-time type — a caller might have passed a wider object. The widening is intentional, not a bug.",
}}>
```ts
const invoice = { amountCents: 4900, status: 'pending' };
const keys = Object.keys(invoice);
```
</CodeTooltips>

The `string[]` widening trips juniors who expected `keyof T`. The reason is structural typing: any object that has at least `amountCents` and `status` is assignable to that type, including objects with extra keys. TypeScript can't prove the runtime shape is exactly the declared shape, so it returns the safe answer. When you genuinely need the narrower type at a trusted boundary (typically right after parsing input you control), an assertion is the explicit escape — and you accept responsibility for the proof.

### Object.fromEntries — the inverse

`Object.fromEntries` takes an iterable of `[key, value]` pairs and builds an object. The most common use is the round trip with `Object.entries`: break the object into pairs, transform them, build a new object back.

<AnnotatedCode lang="ts" code={`
const customer = { name: 'Lina Park', email: 'lina@acme.test' };

const pairs = Object.entries(customer);
const upperPairs = pairs.map(([key, value]) => [key.toUpperCase(), value]);
const upperCustomer = Object.fromEntries(upperPairs);
`}>
  <AnnotatedStep meta="{3}" color="blue">
    Break the object into `[key, value]` pairs — an array we can iterate, map over, or filter.
  </AnnotatedStep>

  <AnnotatedStep meta="{4}" color="orange">
    Transform each pair. Any array operation is now available. Here we uppercase the key and pass the value through unchanged.
  </AnnotatedStep>

  <AnnotatedStep meta="{5}" color="green">
    Build the object back from the transformed pairs. The round trip — `entries → transform → fromEntries` — is the senior shape for any object-level reshape that would otherwise be a hand-rolled `.reduce` over the keys.
  </AnnotatedStep>
</AnnotatedCode>

This round trip retires the old `arr.reduce((acc, [k, v]) => ({ ...acc, [k]: v }), {})` form for the same reason `Object.hasOwn` retired `hasOwnProperty.call` — it reads as the operation it's doing, in two named steps instead of one long fluent line. (The reduce form also rebuilds the accumulator object on every iteration, so it's quietly `O(n²)` while `Object.fromEntries` is linear.)

The other site `Object.fromEntries` lives at is the boundary where pair-shaped data crosses into object shape — converting a `Map` to a plain object for a JSON response, or parsing a `URLSearchParams` instance into a flat object of query values. The `Map` case comes up later in this chapter, and the query-string case comes up when Server Actions land.

### Object.assign — mutates, returns the target

`Object.assign(target, ...sources)` copies the source objects' own enumerable properties onto `target`, returning `target`. The return value is the same object — not a copy.

```ts
const config = { theme: 'dark', locale: 'en' };
Object.assign(config, { locale: 'es' });
// config is now { theme: 'dark', locale: 'es' }
```

That mutation is the catch. Most of the time you wanted a non-mutating merge, and the spread form (`{ ...config, locale: 'es' }`) is the right reach. `Object.assign` earns its place only when the mutation is the point — patching a config object the caller already owns, applying defaults to an options argument inside a function body. When in doubt, spread.

### Object.freeze, Object.isFrozen — runtime immutability

`Object.freeze(obj)` makes the object read-only at runtime: writes silently fail in non-strict mode, throw in strict mode. `Object.isFrozen(obj)` is the predicate. TypeScript's `readonly` modifier — landing when type annotations get their own chapter next — is the *design-time* enforcement, and that's almost always what you want. `Object.freeze` is reserved for the rare case where a runtime guarantee matters: a shared module-level constant a careless caller might try to mutate. Already framed in the value-model lesson; named here for completeness.

### Object.groupBy — bucket rows by a string key

This is the modern beat. `Object.groupBy` (ES2024) takes an array and a callback that returns a string key, and produces an object grouping the items by key. It retires the `.reduce` pattern juniors used to write for this exact shape.

```ts
const invoices = [
  { id: 'inv_001', amountCents: 4900, status: 'paid' },
  { id: 'inv_002', amountCents: 1200, status: 'pending' },
  { id: 'inv_003', amountCents: 9900, status: 'paid' },
  { id: 'inv_004', amountCents: 3300, status: 'pending' },
];

const byStatus = Object.groupBy(invoices, (invoice) => invoice.status);
```

`byStatus` is now an object whose keys are the status values (`'paid'`, `'pending'`) and whose values are arrays of the matching invoices. One line, one operation, reads as intent. The senior reach for *"group these rows by this categorical field."*

<CodeTooltips tooltips={{
  byStatus: "Inferred as `Partial<Record<string, Invoice[]>>`.\nEvery value is `Invoice[] | undefined` even when you can see by reading the data that the input has at least one `'paid'` invoice — TypeScript can't prove the callback returned every key at least once.\nRead with `byStatus.paid ?? []` or check `Object.hasOwn(byStatus, 'paid')` before indexing.",
}}>
```ts
const byStatus = Object.groupBy(invoices, (invoice) => invoice.status);
```
</CodeTooltips>

That `| undefined` on every group is the gotcha. TypeScript doesn't track which keys the callback actually returned for the input, so even if you can see by reading the data that `'paid'` shows up, the type stays defensive. The `??` fallback handles it cleanly: `const paid = byStatus.paid ?? [];`. The pattern shows up again when grouping by a non-string key — that case earns `Map.groupBy`, covered in the lesson on `Set` and `Map` later in this chapter.

## The prototype chain in one paragraph

Every object literal you write inherits from `Object.prototype`, which is why `.toString`, `.hasOwnProperty`, and `.constructor` are available on it without you having defined them. That inherited surface is what made `config[userInput]` return `[Function: toString]` at the top of the lesson — the prototype chain leaked into application logic the moment user input touched bracket access. The senior reflex is to never trust the chain through user-controlled keys, which is the entire reason `Object.hasOwn` exists. For the rare case where you genuinely need an object with no inherited surface at all — a strict lookup table keyed by user input where any accidental prototype access would be a bug — there's `Object.create(null)`:

```ts
const safeMap = Object.create(null);
safeMap[userInput] = 'value';
// safeMap.toString is undefined — no prototype chain to leak
```

Named once, parked. The deeper reflective surface — `Object.getPrototypeOf`, `Object.defineProperty`, the legacy `__proto__` accessor — isn't on the daily senior path and lives out of scope.

## Where this lands later

The object as record shows up in every later unit. One sentence each:

- Drizzle returns row objects — `select` queries hand back arrays of records shaped exactly like the table you read from.
- React props are an object shape — every component receives `{ children, className, ... }` and destructures from it.
- Server Actions consume an options-object input — `createInvoice({ customerId, amountCents, dueDate })` is the shape.
- Zod's `.parse()` returns a typed object — the validated shape is what the rest of the system reads from.

Same form, different surface. The record is the record.
