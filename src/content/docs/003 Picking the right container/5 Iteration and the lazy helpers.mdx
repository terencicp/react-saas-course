Two production bugs, opposite ends of the same surface. The first allocates a million rows to throw most of them away. The second silently picks up a property nobody added to the object. Both are iteration bugs — the wrong loop, or the wrong surface for the shape of the source.

The fix for either one isn't a clever defensive check or an extra copy — it's reaching for the right loop and, when the source is lazy, the right helper surface. This lesson installs both: the protocol that sits underneath every `for...of`, the three static entry points for iterating an object (and the one banned form you'll flag in code review), generators in just enough detail to recognize them, and the ES2025 `Iterator.prototype` helpers that let you compose `.filter` and `.map` over a lazy source without materializing it first.

## The iteration protocol

Every container the previous four lessons installed — arrays, `Set`, `Map`, plus the `Object.entries` array — shares one contract. So do strings, `URLSearchParams`, the `NodeList` from a DOM query, the value returned by `FormData.entries()`, and the result of calling a generator. The contract is the *iteration protocol*, and it's what `for...of`, the spread operator (`[...x]`), `Array.from`, and the new `Iterator.from` all consume.

The protocol has two halves. Anything with a `[Symbol.iterator]()` method that returns an iterator is an <Term definition="An object that implements the iteration protocol via a [Symbol.iterator]() method returning an iterator. Arrays, strings, Maps, Sets, generators — anything you can put after the of in for...of.">iterable</Term>. An <Term definition="An object with a .next() method that returns { value, done } each time it's called. Once done is true, the iterator is exhausted.">iterator</Term> is anything with a `.next()` method that returns `{ value, done }` — `value` is the next item, `done` flips to `true` when there's nothing left. That's the entire surface. No class, no inheritance, no registration; if your object answers those two questions, the language treats it as iterable.

You almost never author one by hand in 2026 application code — the only realistic author site is a generator function, and we'll meet those in a moment. The point of naming the protocol is that everything else in this lesson makes sense once you can see it. `for...of` is just a sugar for *call `[Symbol.iterator]()`, then call `.next()` in a loop until `done` is true.* Spread is the same loop, collecting into a fresh array. The iterator helpers later in the lesson are operators on the iterator the protocol returns.

One short walk-through to make the shape concrete. The code below pulls values out of a plain array by hand — the same dance `for...of` does for you.

<AnnotatedCode lang="js" code={`
const arr = [1, 2];
const iter = arr[Symbol.iterator]();

iter.next();
iter.next();
iter.next();
`}>
  <AnnotatedStep meta={`{2} "arr[Symbol.iterator]()"`} color="blue">
    `Array` implements the iteration protocol — it has a `[Symbol.iterator]()` method that returns a fresh iterator pointed at the start of the array. Every iterable has this method; you almost never call it by hand because `for...of` does it for you. We're calling it explicitly here to make the contract visible.
  </AnnotatedStep>

  <AnnotatedStep meta="{4-5}" color="blue">
    Each `.next()` advances the iterator and returns `{ value, done }`. The first call returns `{ value: 1, done: false }` — the first element, and there's more. The second returns `{ value: 2, done: false }` — the last element, and the iterator hasn't realized yet that it's the last. `done` reports *after* the fact.
  </AnnotatedStep>

  <AnnotatedStep meta="{6}" color="blue">
    One more `.next()` returns `{ value: undefined, done: true }`. The iterator is exhausted; `value` is `undefined` because there's nothing to hand back. `for...of` stops the moment it sees `done: true` and never runs the body for the undefined value. That's the whole contract — call `.next()` until `done`, ignore the trailing undefined.
  </AnnotatedStep>
</AnnotatedCode>

The takeaway isn't *"now I can write iterators by hand"*; it's *"now I can read what `for...of`, spread, and `Array.from` are doing under the hood, and the lazy helpers later in this lesson have an obvious place to plug in."* The protocol is the seam between the container surface (the previous four lessons) and the consumer surface (this one).

## `for...of` as the default loop

The lesson on array methods named the four triggers that flip the choice from a `.map`/`.filter` chain to a `for...of` loop. Those triggers still apply, but they're worth re-anchoring here as *the consumer of the protocol* — `for...of` is the loop you reach for over anything iterable, which is everything in this chapter and most of the surfaces in the chapters ahead.

The mechanics are exactly what the protocol implies. `for...of` calls `[Symbol.iterator]()`, then calls `.next()` until `done`, runs the loop body with each `value`. It plays naturally with `break`, `continue`, `return`, `throw` — and, the killer feature, with `await` inside the body.

The five things `for...of` lets you write that array methods don't:

- `break` / `continue` / `return` from inside the body to short-circuit out.
- `await` inside the body for sequenced async work (the trigger from the previous lesson).
- Destructure in the binding itself — `for (const [key, value] of map)` or `for (const { id, amountCents } of invoices)`.
- The `.entries()` shape for index plus value — `for (const [i, item] of arr.entries())`.
- Multiple statements per iteration with their own local bindings, without stuffing them into a `.forEach` callback.

The four triggers from the array-methods lesson collapse the choice in one sentence: reach for `for...of` when you need async sequencing, early termination, multi-statement bodies, or both index and value. Otherwise the chain reads cleaner.

One example pulling several of those together. The invoice domain from the previous lessons — `{ id, amountCents, status, customerId, dueDate }`, this time keyed by `id` in an `invoicesById` lookup — walked until the running total crosses a threshold, returning the id where it crossed:

```ts "for (const [id, invoice] of Object.entries(invoicesById))" "return id"
const findFirstOverBudget = (
  invoicesById: Record<string, Invoice>,
  budgetCents: number,
): string | undefined => {
  let runningTotal = 0;
  for (const [id, invoice] of Object.entries(invoicesById)) {
    runningTotal += invoice.amountCents;
    if (runningTotal >= budgetCents) return id;
  }
  return undefined;
};
```

Two things from the list happen at once. The `for (const [id, invoice] of ...)` binding destructures each `[key, value]` pair as it arrives, so both halves are bound in one move. The `return id` short-circuits the loop the moment the running total crosses the threshold — the rest of the entries never get walked. Neither shape is reachable inside a `.reduce` callback — `.reduce` doesn't expose early termination, and destructuring the pair inside the callback signature is awkward for what should read as one line. `for...of` is the right tool here because of the triggers, not the syntax.

## Iterating an object's own properties

Plain objects don't implement `[Symbol.iterator]` themselves. Writing `for (const x of someObject)` throws a `TypeError` at runtime — *"someObject is not iterable."* This trips up developers coming from Python, where `for ... in dict` walks the keys natively. JavaScript splits the dance into two steps: you call one of three static methods to get an iterable view of the object, then iterate that.

The three entry points, named once in the lesson on objects and recalled here as the consumer surface:

- **`Object.entries(obj)`** returns `[key, value]` pairs. The default reach — you usually want both halves.
- **`Object.keys(obj)`** returns keys only.
- **`Object.values(obj)`** returns values only.

All three return arrays of <Term definition="A property defined directly on the object, not inherited from its prototype. Object.entries, Object.keys, and Object.values only walk own properties; for...in walks own plus inherited.">own properties</Term> — the prototype chain doesn't leak through. The senior reach for *"loop over an object"* is `for (const [key, value] of Object.entries(obj))`.

One small TypeScript wrinkle worth recalling from the objects lesson: `Object.keys(obj)` and `Object.entries(obj)` widen the key type to `string[]`, not `keyof typeof obj`. The reason is that at runtime, `obj` might have extra keys the compile-time type doesn't know about, so narrowing them as the declared keys would be a lie. In practice you read the value (which is typed correctly) and ignore the widened key, or assert at trusted boundaries.

### The `for...in` ban

`for...in` exists. The course never writes it. Here's why, in one block so you can flag it on sight in code review.

`for...in` iterates **string keys**, **including inherited enumerable ones**. Two things in that sentence go wrong in production code. First, it walks the prototype chain — every enumerable property on `Object.prototype` (and any prototype between yours and `Object.prototype`) shows up in the iteration. If a third-party script or an older polyfill adds an enumerable property to `Object.prototype`, every `for...in` loop in your app suddenly logs that property too. The cold-open bug, exactly. Second, on arrays, `for...in` yields the indices as *strings* — `"0"`, `"1"`, `"2"` — not the values, which is almost certainly not what the developer meant.

The senior rule: in 2026 application code, you write `Object.entries` (or `.keys` / `.values`) when iterating an object, and `for...of` when iterating an array. `for...in` is named once, in a code-review checklist, and you can configure a linter to flag it project-wide as a one-liner in `biome.json`.

The two forms side by side, both iterating the same `invoicesById` record:

<CodeVariants>
  <CodeVariant label="for...in (banned)">
    <div data-mark-color="red">

    ```ts "for (const id in invoicesById)"
    for (const id in invoicesById) {
      const invoice = invoicesById[id];
      if (invoice === undefined) continue;
      console.log(id, invoice.amountCents);
    }
    ```

    </div>
    Walks the prototype chain — if anything in the page has added an enumerable property to `Object.prototype`, the body runs for it too. Under `noUncheckedIndexedAccess`, `invoicesById[id]` returns `Invoice | undefined`, so the body needs a defensive narrow before reading `.amountCents`. Twice the typing for the wrong shape.
  </CodeVariant>

  <CodeVariant label="Object.entries (default)">
    <div data-mark-color="green">

    ```ts "for (const [id, invoice] of Object.entries(invoicesById))"
    for (const [id, invoice] of Object.entries(invoicesById)) {
      console.log(id, invoice.amountCents);
    }
    ```

    </div>
    Own properties only — the prototype chain can't leak through. Destructuring the pair in the binding gives you `id` and `invoice` in one move, and `invoice` is typed (the key widens to `string` per the watch-out from the lesson on objects, but the value carries its type).
  </CodeVariant>
</CodeVariants>

If you ever do need *"is this key on the object itself, ignoring the prototype?"* as a check rather than a loop, that's what `Object.hasOwn(obj, key)` is for (from the lesson on objects). For the loop case, `Object.entries` is the answer.

## Generators, in one snippet

Generators are the easiest way to author a custom iterable — and the only way you'll realistically reach for in 2026 application code. The lazy-helper section in a moment needs a non-array source to demonstrate laziness, and a generator is the cleanest source to write.

A generator function uses the `function*` syntax (note the asterisk) and the `yield` keyword inside the body. Calling it doesn't run the body — it returns an iterator. Each `yield` pauses the function and hands a value out; the next `.next()` call resumes execution right where it left off. The returned object is both an iterator (has `.next()`) and an iterable (its `[Symbol.iterator]()` returns itself), so it drops straight into `for...of` or into any of the iterator helpers we're about to meet.

```js "function*" "yield event"
function* invoiceEvents() {
  const events = [
    { id: 'evt_1', severity: 'info' },
    { id: 'evt_2', severity: 'critical' },
    { id: 'evt_3', severity: 'info' },
    { id: 'evt_4', severity: 'critical' },
  ];
  for (const event of events) {
    console.log(`yielding ${event.id}`);
    yield event;
  }
}
```

Plain JS, not TS — the generator surface is the shape we care about here, and the `console.log` is the *observable signal of laziness* for the next section. Every time the generator yields, you'll see the line print. If something downstream stops pulling values early, the `console.log` lines stop firing. That's how we'll prove the helper chain doesn't materialize the whole source.

In 2026 SaaS code, the common consumer sites for generators are anything streamed — `ReadableStream` from the Fetch API (later in the platform unit), the loop over paginated API pages (later in the async chapter). The common author site is exactly this: *"I need a lazy source for a helper chain and writing it as a generator is shorter than hand-rolling a `[Symbol.iterator]` method."* That's the whole reach. No exercise here — recognition is enough.

## The lazy iterator helpers

The central beat of the lesson. ES2025 added a layer of methods directly to `Iterator.prototype` — the same `.map`, `.filter`, `.take`, `.reduce` you know from arrays, but operating on the *iterator* itself, lazily, without materializing intermediate arrays. They land on every iterator in the language, which means they work on generators, `Map.values()`, `Set.values()`, the iterator returned by `Object.entries`, anything. Node 24 LTS (the runtime the course pins) ships them, and every modern browser does too.

The instinct after reading that paragraph is *"so I should always use these instead of array methods."* That's the wrong reach. The whole point of these helpers is to handle sources that array methods handle badly. If you can already point at an in-memory array, array methods are the right tool. Three triggers — and only these three — flip the choice toward the helpers.

**Trigger before tool.** Reach for the iterator helpers when:

1. **The source is itself lazy** — a generator, a stream, a paginated API loop. You don't want to pull all of it at once; you want each pipeline stage to pull just what the next stage needs.
2. **The source is large enough that materializing it is wasteful.** Even if you could pull it all, allocating a million-row array just to throw most of it away is the wrong shape.
3. **The pipeline short-circuits.** You only need the first N matches, or the first match that satisfies a condition. Array methods walk the whole array; the helpers stop pulling the moment the terminal step has what it asked for.

On an in-memory list of 50 invoices, `invoices.filter(isOverdue).map(toReminder)` is the right shape. Eager, readable, terminal in one step. The iterator helpers earn their place when the source pushes back against being materialized. *Trigger before tool — always.*

### The surface

The methods split into two groups by what they do. Lazy methods return a *new* iterator and run nothing — they wire up the next stage of the pipeline and wait. Terminal methods *pull* the chain: they call `.next()` on the iterator they wrap, which calls `.next()` on the one *it* wraps, all the way back to the source.

- **Lazy (returns a new iterator):** `.map(fn)`, `.filter(fn)`, `.take(n)`, `.drop(n)`, `.flatMap(fn)`.
- **Terminal (pulls the chain):** `.toArray()`, `.reduce(fn, init)`, `.forEach(fn)`, `.some(fn)`, `.every(fn)`, `.find(fn)`.
- **Entry point:** `Iterator.from(iterable)` wraps any iterable (array, set, map, generator, custom) so the helper chain becomes available on it.

Nothing in the lazy half runs until the terminal step asks. `.toArray()` pulls everything. `.find` / `.some` / `.every` pull until a match short-circuits the answer. `.take(n)` followed by `.toArray()` pulls *exactly* `n` items through the chain, plus whatever the upstream stages had to discard to find them.

### The pull-through pipeline

The visual that makes the laziness click. Each terminal call pulls one value through the chain at a time — the source yields only what the terminal needs.

<PullThroughPipeline />

Reading the diagram right-to-left: `.toArray()` asks `.take(10)` for one value. `.take(10)` asks `.filter(isCritical)` for one value. `.filter` asks the source for one value, checks it against `isCritical`, and either passes it back (if critical) or asks again (if not). The value flows back along the chain in the opposite direction the pull went. When `.take(10)` has handed out its tenth value, it answers `done: true` on the next pull and the chain stops cold. The source generator yields exactly the items it had to produce to satisfy ten passes through the filter — not one more.

That's the whole win. If the first ten critical events are in the first fifty items of a million-row stream, the source yields fifty times. The other 999,950 never happen.

### Eager versus lazy

The cold-open bug, fixed.

  <CodeVariant label="Eager (wasteful)">

    </div>
    Materializes the entire stream first, *then* filters, *then* slices. If `invoiceEvents()` can yield a million rows, the spread allocates an array of a million objects before `.filter` reads its first byte. Wrong shape for a lazy source.

  </CodeVariant>

The two shapes look almost identical on the page. The difference is in what runs, and when. The eager version reads the entire source up front; the lazy version reads one value at a time and stops the moment the terminal step has what it asked for. *Same surface area, two orders of magnitude apart in what gets allocated.*

### Predict the pulls

A short drill to make the laziness concrete. The generator below `console.log`s on each yield — that's the observable signal we set up earlier. Read the code, then predict what prints before `result` resolves.

```js
function* events() {
  const items = ['a', 'b', 'c', 'd', 'e', 'f', 'g'];
  for (const item of items) {
    console.log(`yielding ${item}`);
    yield item;
  }
}

const isMatch = (item) => item === 'c' || item === 'f' || item === 'g';

const result = Iterator.from(events()).filter(isMatch).take(2).toArray();
console.log(result);
```

<PredictWhy>`.toArray()` pulls one value at a time through the chain. The first pull walks the source until `.filter` finds a match — `a` is dropped, `b` is dropped, `c` passes, so `.take(2)` accepts it. The second pull resumes the source from where it stopped — `d` dropped, `e` dropped, `f` passes — `.take(2)` accepts it. Two items have made it through, so `.take(2)` answers `done: true` on the next pull and the chain stops. The source yields six times in total, never reaching `g`. The full array — `[ 'c', 'f' ]` — only assembles once `.toArray()` is done pulling.</PredictWhy>

</PredictOutput>

The lesson in that drill: the source generator runs only as far as the terminal step demands. Six `console.log` lines fire, not seven, because the source never had to be asked for `g`. The same code rewritten as `[...events()].filter(isMatch).slice(0, 2)` would fire all seven `console.log`s before `.filter` even started looking — and the value of `g` would be allocated for no reason.

### Rewrite the eager form

Now the inverse drill — the same shape as the cold open, with a test asserting *both* the result and the count of times the source generator yielded. The eager form passes the result check; only the lazy form passes the count check. The failing count is the lesson — you watch *"yielded 1000 times"* turn into *"yielded N times"* once the rewrite lands.

<ScriptCoding

The 'only pulls what it needs' test checks that the source generator's yield fired at most 425 times (not 1000). Rewrite the body using Iterator.from, .filter, .take, and .toArray so both tests pass. Keep the signature the same."
  starter={`const firstNMatches = (source, predicate, n) => {
  return [...source].filter(predicate).slice(0, n);
};
`}
  tests={`
let yieldCount = 0;

function* numbers() {
  for (let i = 0; i < 1000; i++) {
    yieldCount++;
    yield i;
  }
}

test('returns the right matches', () => {
  yieldCount = 0;
  const result = firstNMatches(numbers(), (n) => n % 100 === 0, 5);
  expect(result).toEqual([0, 100, 200, 300, 400]);
});

test('only pulls what it needs (yields at most 425 times)', () => {
  yieldCount = 0;
  firstNMatches(numbers(), (n) => n % 100 === 0, 5);
  // Eager pulls 1000; lazy pulls 401 (yields 0 through 400).
  console.log('source yielded ' + yieldCount + ' times');
  if (yieldCount > 425) {
    throw new Error('source yielded ' + yieldCount + ' times — the chain is still eager');
  }
});
`}
/>

<details>
<summary>Reveal the answer</summary>

```js
const firstNMatches = (source, predicate, n) => {
  return Iterator.from(source).filter(predicate).take(n).toArray();
};
```

`Iterator.from(source)` wraps the generator so the helper chain is available. `.filter` and `.take` are lazy — they wire up the pipeline but pull nothing yet. `.toArray()` is the terminal step that pulls one value at a time. The source yields only what `.take(n)` needs: for `n = 5` and the predicate `n % 100 === 0`, the generator's `yield` fires 401 times (0 through 400), not 1000.

</details>

The shape of the rewrite is what matters. The arguments don't change, the return type doesn't change, the test for *"did you return the right thing?"* doesn't change. What changes is the count of times the source had to yield to get there. *That's* the production win: not microseconds shaved off a benchmark, but a stream you can compose against without first asking *"can my server hold all of this in memory?"*

## When to stay with arrays

The most common mistake after learning the iterator helpers is reflex-reaching for them on a 50-row array. Don't. The helpers earn their place against the three triggers above; if none of them fire, you're trading readability for nothing.

Most application data fits comfortably in memory. A `.filter(isOverdue).map(toReminder)` over an in-memory array of fifty invoices is the right shape — eager, readable, terminal in one step, no wrapper. Wrapping that array in `Iterator.from(...)` adds two letters, one method call, and zero observable benefit. The one-line senior rule: *if you can already point at the array, don't wrap it. If the source yields values over time, do.*

## Async iteration, in one paragraph

When the source itself is asynchronous — a `ReadableStream` from `fetch`, server-sent events, a paginated API that returns a `Promise<Page>` per pull — the synchronous `for...of` doesn't fit. The shape is `for await (const value of asyncIterable)` — same loop body, awaits each value before binding it. You'll meet `for await...of` properly in the async chapter and again when we cover streamed `fetch`. The lazy iterator helpers have an async cousin too — async iterator helpers ship under a separate spec — but the course reaches for them only where they earn their place. For now, just recognize the shape:

```ts
for await (const chunk of response.body) {
  process(chunk);
}
```

When you see `for await` in code or in a future lesson, the only new piece is the `await` keyword between `for` and `(`. The rest is the same iteration protocol you just installed, with a `Promise` between each pull and the value.
