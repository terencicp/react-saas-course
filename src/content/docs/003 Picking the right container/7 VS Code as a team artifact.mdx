A teammate joins the repo on Monday, opens it in their editor with their personal "format on save with Prettier defaults" already on, hits save on one file, and the diff is a two-hundred-line whitespace storm. Their formatter, their indentation rule, and their auto-organize-imports behaviour all disagree with the project's. The PR is unreviewable. The fix isn't a code-of-conduct doc that asks everyone to please disable their personal settings; it's three files the repo commits so the editor reads the *project's* rules instead of the developer's.

Every lesson so far in this chapter ran in the TS Playground in your browser. From the next lesson on, you'll write `.ts` files on disk and run them locally. Before that switch lands, this lesson pins the editor — VS Code — and the team-shared config that comes with it, so the next lesson opens against a known surface. Node 24 LTS, the `.mise.toml` that pins it, and the `tsconfig.json` that drives the compiler all land in lessons coming right after this one; here, the editor and its repo-owned files are the only concern.

## Why VS Code

VS Code is the 2026 default for SaaS work: the broadest extension ecosystem, the stack's official <Term definition={"Language Server Protocol — the contract between an editor and a language tool like a type-checker or linter, so the same diagnostics can run in any editor that speaks the protocol."}>LSP</Term>s target it first (Biome, Tailwind, TypeScript), it's free, cross-platform, and the editor every junior teammate will already have or will install in five minutes. Three alternatives are worth naming once, each with the one trigger that would flip the choice:

- **Cursor** — AI-first workflows the course deliberately doesn't teach against. "How to direct AI" goes stale faster than this course can ship; the durable engineering it covers doesn't, and that's where the time goes.
- **Zed** — editing speed past the point a senior would notice; Zed's LSP support for the stack still lags VS Code's.
- **Neovim** — keyboard fluency a working dev typically doesn't have time to invest in for the marginal gain it returns.

The course commits to VS Code. Install it from the official download page; the installer is a single click on every platform.

## The minimum-viable extension set

Themes, icon packs, vim modes, productivity bundles — out of scope. Install whatever you like on the side; what follows is the set the *repo* needs the team to share. Six extensions, each earning its place against a named trigger.

<CardGrid>
  <Card title="Biome" icon="biome">
    `biomejs.biome` — Lint-and-format LSP for JS, TS, and JSON. Lands the formatter so the next time you hit save, it's the project's formatter that runs, not the editor's default.
  </Card>
  <Card title="Tailwind CSS IntelliSense" icon="seti:css">
    `bradlc.vscode-tailwindcss` — Class-name autocomplete, hover-previewed generated CSS, design-system linting. Earns its weight from the React unit onward; installed now so it never has to be revisited.
  </Card>
  <Card title="EditorConfig for VS Code" icon="seti:config">
    `EditorConfig.EditorConfig` — Reads `.editorconfig` so non-VS Code editors on the team (IntelliJ, Neovim, etc.) honour the same indent, line-ending, and final-newline rules.
  </Card>
  <Card title="GitLens" icon="seti:git">
    `eamodio.gitlens` — Inline blame on the line under the cursor, file history, side-by-side compare. Code review starts with "who wrote this and why" — GitLens puts that question one click away.
  </Card>
  <Card title="Error Lens" icon="warning">
    `usernamehw.errorlens` — Pulls TypeScript and Biome diagnostics inline next to the offending line. Tightens the feedback loop on every save — no hover needed to see what's wrong.
  </Card>
  <Card title="Pretty TypeScript Errors" icon="seti:typescript">
    `yoavbls.pretty-ts-errors` — Flattens long structural type errors into something a human reads in one pass. Earns its weight the first time a Drizzle or Zod inferred type explodes into a wall of brackets.
  </Card>
</CardGrid>

Install each one by clicking through the marketplace link in VS Code, or by running `code --install-extension <id>` in a terminal. Either works. The `extensions.json` file in the next section turns this into a one-click prompt the next time anyone clones the repo, so this is the last time anyone has to think about installing these by hand.

GitLens is the one extension on the list whose payoff isn't obvious from a single setting — its inline blame, CodeLens, and file-history views are the "who wrote this and why" tooling that code review leans on.

## The three repo-owned files

Three files live in the repo so every editor on the team behaves the same: `.editorconfig` at the root for any editor that honours the spec, `.vscode/extensions.json` so VS Code prompts to install what the project needs, and `.vscode/settings.json` for the workspace-only overrides that pin the formatter, the on-save actions, and the TypeScript SDK choice. Personal preference — theme, font, keybindings — never lives in these files.

Create the three files in the order below. Each one pins a specific layer of the editor's behaviour.

<Steps>
1. **`.editorconfig` at the repo root.** A six-rule file every modern editor honours, so the most basic things — indent, line endings, file encoding — never come down to whose machine you're on.

   ```ini title=".editorconfig"
   root = true

   [*]
   indent_style = space
   indent_size = 2
   end_of_line = lf
   charset = utf-8
   insert_final_newline = true
   trim_trailing_whitespace = true
   ```

   Six rules: 2-space indent, LF line endings (so a Windows teammate doesn't introduce CRLF), UTF-8 encoding, a final newline at the end of every file, and trimmed trailing whitespace. Honoured natively by the IntelliJ family and Neovim, and by VS Code via the EditorConfig extension above. Biome reads `.editorconfig` for indent and line-ending settings too — one source of truth, no duplicate config to keep in sync once `biome.json` lands later in the course.

2. **`.vscode/extensions.json`** with the recommendation list. The marketplace IDs of the six extensions from the previous section, in the same order.

   ```json title=".vscode/extensions.json"
   {
     "recommendations": [
       "biomejs.biome",
       "bradlc.vscode-tailwindcss",
       "EditorConfig.EditorConfig",
       "eamodio.gitlens",
       "usernamehw.errorlens",
       "yoavbls.pretty-ts-errors"
     ]
   }
   ```

   When a teammate clones the repo and opens the folder in VS Code, the editor prompts to install any of these six the teammate doesn't already have. Structural enforcement: "I forgot to install Biome" is hard to leave un-fixed when the editor itself asks every time.

3. **`.vscode/settings.json`** with the workspace overrides. The settings the project needs, scoped to this workspace only — nothing personal goes in here.

   ```json title=".vscode/settings.json"
   {
     "editor.formatOnSave": true,
     "editor.defaultFormatter": "biomejs.biome",
     "[css]": {
       "editor.defaultFormatter": "biomejs.biome"
     },
     "editor.codeActionsOnSave": {
       "source.fixAll.biome": "explicit",
       "source.organizeImports.biome": "explicit"
     },
     "typescript.tsdk": "node_modules/typescript/lib"
   }
   ```

   The first pair, `editor.formatOnSave` and `editor.defaultFormatter`, sends every save through Biome instead of whatever each teammate happens to have set as their personal default. The per-language `[css]` override is here for the day Tailwind v4 code lands later in the course — not load-bearing yet, but pinning it now means you don't have to revisit the file.

   `editor.codeActionsOnSave` lists the two Biome actions that run on save. `source.fixAll.biome` applies safe auto-fixes; `source.organizeImports.biome` sorts and dedupes imports. Both use the `"explicit"` string form, which has been mandatory since VS Code 1.83 — the older `true`/`false` values stopped working a couple of years ago and any tutorial that still shows them is stale.

   `typescript.tsdk` points the editor at the workspace TypeScript — the `tsc` installed inside the project's `node_modules`. Why it matters: every fresh open of a TS project shows a "Use Workspace Version" toast asking whether the editor should switch from its bundled TS to the one the project pins. Setting this here pre-answers that prompt, so the editor uses the same compiler that your build does. The `node_modules/typescript` path doesn't exist yet — that's a later lesson's job — but pinning it now is harmless, and the moment the dependency lands the editor picks it up.
</Steps>

:::tip
**Workspace vs User — the partition that matters.** Settings you want for *yourself* — theme, font, keybindings, mouse scroll speed, mini-map visibility — belong in **User settings** (`Cmd/Ctrl + ,` opens settings; choose the **User** tab), never in `.vscode/settings.json`. The workspace file is a team artifact: it ships in the repo, it goes through code review, it applies to every teammate. The User file is personal: it lives in your profile, nobody else sees it. Pushing a personal preference to the workspace file is one of the rare repo changes that gets reverted on PR review with no discussion.
:::

## Workspace vs user — sort it now

Six settings, two homes. Drop each one into the place it belongs — the repo's `.vscode/settings.json` (every teammate gets it) or your own User settings (only you see it). The rule is the same one the callout above states: if the *project* needs the setting, it's workspace; if *you* prefer the setting, it's user.
