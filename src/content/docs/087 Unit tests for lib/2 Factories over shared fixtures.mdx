Three tests in the same file need the same setup: an `Organization` on the `pro` plan and a `User` whose role is `admin`. You know the AAA shape now, you know arrange should be one line, and so the obvious move is to lift that setup out of the tests. You hoist a `const org = { ... }` and a `const admin = { ... }` to the top of the file and let all three tests share them. Less typing, one place to read the shape, done.

That reflex is the single most common test-data mistake there is, and it has a name and a cost we are going to make you feel before we fix it.

In the previous lesson I made you a promise: when arrange grows past three lines, the fixture moves to a factory. This is where that promise comes due. By the end of this lesson you'll write `buildUser({ role: 'admin' })` and read it out loud as "a user that's an admin" — and you'll know exactly when to reach for a factory, when to reach for a static fixture instead, and why a seed is neither.

## The shared-fixture trap

Here is the file you'd actually write. A `User` shape with a handful of fields, hoisted to the top, shared by every test below it.

<div data-mark-color="orange">

```ts title="src/lib/access.test.ts" {1}
const baseUser = {
  id: 'usr_1',
  email: 'test@example.com',
  role: 'member',
  orgId: 'org_1',
};

it('lists a member in the directory', () => {
  expect(canSeeBilling(baseUser)).toBe(false);
});

it('grants billing access to admins', () => {
  baseUser.role = 'admin';
  expect(canSeeBilling(baseUser)).toBe(true);
});
```

</div>

Read the second test. It needs an admin, so it does the natural thing — it reaches into the shared object and sets `baseUser.role = 'admin'`. The test passes. The first test passes. Run the file and it's green. Ship it.

Now someone reorders the file, or Vitest runs the tests in a different order, or a third test lands between these two. Suddenly the *first* test sees `role: 'admin'` — a value it never set, never asked for, and that contradicts its own name. It fails. Whoever picks up that failure will stare at a test called "lists a member" that's somehow looking at an admin, find nothing wrong with the test itself, re-run it, watch it pass, and write it off as flaky. The real name for what they're seeing is <Term definition="A test's outcome depends on whether, and in what order, other tests ran first.">run-order coupling</Term>: the test's outcome depends on whether, and in what order, other tests ran first.

It is not flaky. As I said in the last lesson, flake has a structural cause, and here the cause is sitting in plain sight: the two tests share one mutable object, and one of them writes to it. The outcome of the first test now depends on whether the second test ran first. That's the whole bug.

<SharedFixtureCollision />

Maybe you spot the mutation and decide to dodge it. The third test needs a plain member again, so rather than risk the shared object you copy the literal, paste it, and tweak the one field you care about. Now there are two near-identical object literals drifting in the same file. When the `User` shape gains a field, you update one and forget the other. Worse, a reader looking at a fifteen-field literal can't tell which of those fields the test actually depends on and which are just there to make the object valid. The data buries the intent.

So the diagnosis. The problem was never the data — every test legitimately needs a user. The problem is that the data is *shared* and *mutable*. There are only two ways out. You can deep-clone the object on every read, which is noisy, easy to forget, and does nothing for the readability problem. Or you can stop storing the data and start *producing* it — a function that hands back a brand-new instance every time you call it. That function is a factory, and it's the better escape because it kills both problems at once: every test gets its own object, and every test names only the field it cares about.

## The factory shape

A factory is a function that returns a fresh domain object. The whole pattern is three rules, and I'm going to introduce them one at a time, because each one is the fix to something that breaks without it.

**Rule one: every required field gets a valid default.** Start with the simplest possible `buildUser` — no arguments yet, just a function that returns a fully-formed `User`.

```ts title="src/test/factories/users.ts" {3}
const buildUser = (): User => ({
  id: 'usr_test',
  email: 'test@example.com',
  role: 'member',
  orgId: 'org_test',
  createdAt: instant('2026-01-01T00:00:00Z'),
});
```

The word that matters in "valid default" is *valid*. Look at `email: 'test@example.com'` — that's a real, schema-passing address. The temptation is to write `email: 'TODO'` or `email: 'xxx'` because it's "just a test" and the email doesn't seem to matter. Don't. The moment a test exercises validation, a placeholder email ships you a <Term definition="A test that passes while the code it covers is actually broken.">false negative</Term> — the test passes, your validator is broken, and nobody finds out. A factory default has to be an instance your production code would actually accept. Defaults respect your domain invariants: `buildUser` never produces an address that fails validation, the same way `buildInvoice` later won't produce negative line totals unless you explicitly ask for them.

**Rule two: `overrides` is the last spread.** Right now `buildUser()` always returns a member. The test that needs an admin can't get one without mutating the result — and we just spent a whole section learning why mutation is the enemy. So we let the caller pass the fields it cares about, and we spread them in last.

<div data-mark-color="orange">

```ts title="src/test/factories/users.ts" "overrides"
const buildUser = (overrides: Partial<User> = {}): User => ({
  id: 'usr_test',
  email: 'test@example.com',
  role: 'member',
  orgId: 'org_test',
  createdAt: instant('2026-01-01T00:00:00Z'),
  ...overrides,
});
```

</div>

`Partial<User>` means "any subset of `User`'s fields, all optional," so the caller can override one field or ten. Because `...overrides` comes *last*, anything the caller passes wins over the default. The payoff is at the call site:

```ts
const admin = buildUser({ role: 'admin' });
```

That line says one thing and says it loudly: a user that is an admin. The `id`, the `email`, the `orgId`, the `createdAt` — all valid, all present, all irrelevant to this test, and so all invisible. The test documents its own inputs. That's the readability win the copy-pasted literal could never give you.

**Rule three: a fresh object every call.** This one's subtle because it's about what you *don't* do. The object literal is constructed *inside* the function body, so every call evaluates it again and you get a new object every time. `buildUser()` and `buildUser()` are two different references that share no memory. That's what closes the bug class from the last section structurally — there's no shared object left to mutate.

The anti-pattern to watch for: a "factory" that builds one object at module load and hands back that same reference on every call.

<div data-mark-color="red">

```ts {1} "return cached"
const cached = { id: 'usr_test', email: 'test@example.com', role: 'member' };

const buildUser = (overrides: Partial<User> = {}) => {
  Object.assign(cached, overrides);
  return cached;
};
```

</div>

That isn't a factory — it's the shared mutable fixture from the start of the lesson wearing a function's clothes. Every caller mutates and receives the *same* object. The fresh-object-per-call rule is the entire reason the pattern works; don't optimize it away.

Here is the finished factory. Small, but every part is load-bearing, so let's walk through it.

<AnnotatedCode lang="ts" maxLines={16} code={`
import type { InferSelectModel } from 'drizzle-orm';
import { users } from '@/db/schema';
import { instant } from '@/lib/temporal';

type User = InferSelectModel<typeof users>;

export const buildUser = (overrides: Partial<User> = {}): User => ({
  id: 'usr_test',
  email: 'test@example.com',
  role: 'member',
  orgId: 'org_test',
  createdAt: instant('2026-01-01T00:00:00Z'),
  ...overrides,
});
`}>
  <AnnotatedStep meta="{7}" color="blue">
    The signature is the contract: it takes an optional subset of `User` and always returns a complete `User`. The `= {}` default lets you call `buildUser()` with no arguments at all.
  </AnnotatedStep>

  <AnnotatedStep meta="{8-12}" color="green">
    Every required field gets a *valid* default — a real email, a real role, a real instant. Valid, never a placeholder: each one is a value your production code would actually accept.
  </AnnotatedStep>

  <AnnotatedStep meta="{13}" color="orange">
    The override spread comes *last*, so the caller's fields win over the defaults. This is the line that lets a test name only the one field it's testing.
  </AnnotatedStep>

  <AnnotatedStep meta="{5}" color="blue">
    The `User` type is derived straight from the Drizzle schema, so the factory and the database agree by construction. This is where the next payoff comes from.
  </AnnotatedStep>
</AnnotatedCode>

That last step is worth dwelling on, because it's where the factory pays for itself. Notice the return type isn't a hand-written `User` interface — it's <Term definition="A Drizzle helper that infers a row's TypeScript type from a table's schema definition, so the type follows the schema automatically.">`InferSelectModel<typeof users>`</Term>, the type Drizzle infers from your schema. You met this in the database unit: the schema is the single source of truth, and the row type flows out of it. By typing `buildUser` off that same inferred type, you've wired the factory directly to the schema.

Watch what happens on a migration. You add a required `timeZone` column to the `users` table. Every place that constructs a `User` is now missing a field — including `buildUser`. TypeScript fails the build *inside the factory*, at exactly one line. You add `timeZone: 'UTC'` to the defaults once, and every test that calls `buildUser` keeps compiling, untouched. Compare that to fifty test files each holding their own user literal: one migration, fifty compile errors, fifty edits. A factory absorbs the schema change in one place. That's the second senior payoff, and it's why people who've been burned reach for factories reflexively — they pay off on the third schema migration, and there's always a third.

Put the two shapes side by side and the difference is stark.

## Where factories live

A factory is shared test infrastructure, and the course gives it one canonical home: `src/test/factories/`, one file per entity. `users.ts` exports `buildUser`, `orgs.ts` exports `buildOrg`, `invoices.ts` exports `buildInvoice`. The filename matches the entity; the export matches the file. No guessing where `buildInvoice` lives.

<FileTree>
- src/
  - test/
    - **factories/**
      - **users.ts** `buildUser`
      - **orgs.ts** `buildOrg`
      - **invoices.ts** `buildInvoice`
    - matchers/ custom `Result` matchers, from the previous lesson
    - fixtures/ static external payloads — covered below
    - clock.ts the frozen-time seam — next lesson
</FileTree>

You might wonder why factories don't follow the colocation rule from the last lesson — the one that says a test sits right next to the file it tests. The answer is in what colocation is *for*. Colocation pairs a test with its one source file so they move, rename, and delete together. A factory has no single source file. The same `buildUser` is imported by `/lib` unit tests today and by integration tests in the next chapter; it's *cross-cutting* test infrastructure, not a test of any one module. Cross-cutting test support lives under `src/test/`. You can see its other tenants in the tree above: the `toBeOkResult` and `toBeErrResult` matchers from the last lesson live in `src/test/matchers/` for the exact same reason, and the next two lessons fill in `clock.ts` and `fixtures/`.

One rule carries straight over from the last lesson: dependency direction. Factories import domain types from `@/db` or `@/lib` — never from `app/**`. A factory reaching up into your route handlers or pages reverses the dependency arrow (test support should depend on the domain, not on the application surface) and trips the same `no-restricted-paths` lint rule that guards your `/lib` tests. Point the imports down toward the schema, never up toward the app.

## Composing factories

An `Invoice` has a customer. You could make every invoice test build a full `User` first and pass it in, but that's friction the factory should absorb. So `buildInvoice` defaults its `customer` to a call to `buildUser`:

<div data-mark-color="blue">

```ts title="src/test/factories/invoices.ts" {4}
export const buildInvoice = (overrides: Partial<Invoice> = {}): Invoice => ({
  id: 'inv_test',
  status: 'draft',
  customer: buildUser(),
  total: { amount: 1000, currency: 'USD' },
  createdAt: instant('2026-01-01T00:00:00Z'),
  ...overrides,
});

const paid = buildInvoice({ status: 'paid' });
const vipInvoice = buildInvoice({ customer: buildUser({ email: 'vip@acme.test' }) });
```

</div>

The call sites now read as a tree of overrides. `buildInvoice({ status: 'paid' })` gets a perfectly valid customer for free — you didn't ask about the customer, so you don't see it. And when a test *does* care about the customer, it reaches one level down: `buildInvoice({ customer: buildUser({ email: 'vip@acme.test' }) })`. Composition all the way down, each layer naming only what it needs.

There's one trap in composition sharp enough to earn its own warning.

:::caution
**Circular defaults loop forever.** If `buildUser` defaults its `org` to `buildOrg()`, and `buildOrg` defaults its `owner` to `buildUser()`, then constructing either one calls the other, which calls the first, which calls the other — construction never terminates and your test suite hangs. Break the cycle at one layer by defaulting a plain reference ID instead of a nested object: `buildUser` defaults `orgId: 'org_test'`, not a nested `buildOrg()`. The rule: most factories stay one level deep, and across any cycle you reference an ID, never a nested build.
:::

A word on override depth. `Partial<User>` covers flat objects, which is the overwhelming common case. When you genuinely need to override a nested field, give that one factory a hand-written shape for it — `{ customer?: Partial<User> }` — rather than reaching for a recursive `DeepPartial<T>` type. `DeepPartial` exists, and you'll see it in the wild, but most factories never need it, and reaching for it early buys you a confusing type for a problem you don't have. Stay flat until a real case forces your hand.

## Deterministic defaults: sequences, not randomness

Here's a move that looks helpful and is poison: reaching for `Math.random()` or `crypto.randomUUID()` inside a factory default to make each user "more realistic."

Don't. A random default means a failing test can't be reproduced. The run that failed had a value you'll never see again; you re-run, get a different value, the test passes, and you've learned nothing. This is the literal source of "flaky, can't repro" bug reports. Unit tests want *deterministic* data — `email: 'test@example.com'` debugs far better than a faker-generated `wilma.lakin@example.org` that's different every run. A factory default is a fixed, known value on purpose.

"But some tests need *distinct* values" — and that's true. Integration databases in the next chapter reject duplicate emails, so a test inserting three users needs three different addresses. The answer to *uniqueness* is not *randomness*. It's a sequence helper: a counter that hands out the next number each time you ask.

<div data-mark-color="green">

```ts "userSeq.next()"
const sequence = () => {
  let n = 0;
  return {
    next: () => {
      n += 1;
      return n;
    },
  };
};

const userSeq = sequence();
const a = buildUser({ email: `user-${userSeq.next()}@test.com` });
const b = buildUser({ email: `user-${userSeq.next()}@test.com` });
```

</div>

The sequence is <Term definition="Only ever increases, never repeats a value.">monotonic</Term> — it only ever counts up, so the values are unique — and it's reproducible — run one always gets `user-1`, `user-2`, the run after gets the same. Unique *and* deterministic, which is exactly what randomness can't give you. The one discipline: the sequence resets per test or per worker, never as global module state. A counter shared across the whole module is shared mutable state by another name, and you've reintroduced the run-order coupling from the start of the lesson — test order now decides which numbers a test sees.

You may have noticed the factory's `createdAt` is a hardcoded instant and the `id` is a fixed string, not a live `Temporal.Now.instant()` or a fresh UUID. That's deliberate, and it's the same principle: time and IDs in a factory default are either literals or pulled from a seam your tests control — never a live call to the wall clock or the UUID generator. The machinery for *pinning* time and IDs across a whole test — fake timers, the clock module, the ID seam — is the next lesson's entire subject. For now, just hold the rule: factory defaults are frozen, not live.

One last placement note, because it seeds the next section. `@faker-js/faker` is a real tool with a real job — generating realistic *seed* data for a development database, which you met back in the database unit. It is not a unit-testing tool. Unseeded faker in a unit test is the flake source we just spent this section avoiding. Faker for dev seeds; literals and sequences for tests.

## Factories vs fixtures vs seeds

Faker-for-seeds points at a distinction that trips up real teams constantly. Three words — *factory*, *fixture*, *seed* — get used interchangeably, and they are not interchangeable. You'll hear all three on the job. Here's each one by what it *is* and when you reach for it.

<FactoryFixtureSeedTable />

A **factory** is a *function* that produces a fresh instance with overrides. You reach for it for per-test domain entities — every user, org, and invoice you've built so far.

A **fixture** is *static* data, used verbatim — typically a JSON file you load and pass straight through. You reach for it when the data is an external payload whose exact bytes matter and which you neither author nor want to paraphrase.

A **seed** is code that *populates a database* with realistic volume and distribution — fifty invoices across a spread of statuses so the dev list page looks real. You met seeds in the database unit, built with `drizzle-seed` in `scripts/seed.ts`. The boundary is clean: a seeder is for *datasets and distributions*, a factory is for *individual per-test rows*. Same conceptual building block, completely different scale and purpose.

Why does conflating them matter? Because each confusion is a bug you already know. A "fixture" that gets mutated is just the shared mutable object from the start of this lesson — it should have been a factory. A "seed" reused as a per-test fixture is shared mutable state across tests — run-order coupling again. The words name different tools for different jobs; using the wrong word leads you to the wrong tool.

### Static fixtures for external payloads

So when *is* a static fixture the right call, and not a factory? Exactly when the payload comes from outside your system and its precise shape is the thing under test.

The clearest case is a webhook. To test how you handle a Stripe `checkout.session.completed` event, you need a byte-realistic copy of what Stripe actually sends. You capture it once from Stripe's test dashboard, save it to `src/test/fixtures/stripe/checkout-completed.json`, and import it in the test. Why static and not a `buildStripeEvent` factory? Because the *exact* shape matters — signature verification, which you'll do in the next chapter, hashes the raw bytes, so a paraphrased payload would verify differently from the real one. You don't author Stripe's schema and you don't want to approximate it. The same logic applies to signed JWTs, Resend bounce webhooks, and S3 event notifications: capture the real thing, store it, use it verbatim.

The maintenance rule follows from the same principle. When a fixture goes stale, you update it by *re-capturing* from the source, not by hand-editing past the single field your test is about. A hand-drifted fixture quietly lies about the real payload shape, and a test against a lie is worse than no test.

One last name to retire. You'll come across the *object mother* pattern — a function like `anExpiredInvoice()` that returns a ready-made entity. It's the same idea as a factory with one more layer of indirection: `buildInvoice({ status: 'expired' })` reads just as clearly and you don't have to write a new named function for every variant. The override-spread factory already does the object mother's job, so the course uses the factory. Now you'll recognize the term when someone says it, and you'll know it's not something new to learn.

Sort these into the right bucket. The split you want is: per-test domain entity is a factory, captured external payload is a fixture, bulk realistic database data is a seed.

## Build a user factory

Time to write one yourself. Implement `buildUser` so that it satisfies all three rules: valid defaults, an overrides-last spread, and a fresh object on every call. The tests check each rule directly — including the one that catches a cached object, so the fresh-instance rule isn't optional.

<ScriptCoding
  runner="sandpack"
  maxHeight={360}
  instructions="Implement buildUser so a no-argument call returns valid defaults, a single override changes only that field, and every call returns a brand-new object. Construct the object literal inside the function body. (In your real file this is typed buildUser(overrides: Partial<User> = {}): User — here it runs untyped so the bundler can execute it.)"
  starter={`export const buildUser = (overrides = {}) => {
  // Return a valid User: { id, email, role, orgId }.
  // Defaults below, overrides last, fresh object every call.
  return {};
};`}
  tests={`
test('a no-argument call returns valid defaults', () => {
  const user = buildUser();
  expect(user.email).toBe('test@example.com');
  expect(user.role).toBe('member');
});

test('an override changes only the named field', () => {
  const user = buildUser({ role: 'admin' });
  expect(user.role).toBe('admin');
  expect(user.email).toBe('test@example.com');
});

test('every call returns a fresh object', () => {
  const a = buildUser();
  const b = buildUser();
  expect(a).not.toBe(b);
  a.role = 'admin';
  expect(b.role).toBe('member');
});
`}
/>

The third test is the one that matters most. If you built the object once and returned it from a closure, the first two tests would pass and that one would fail — mutating `a` would change `b`, because they'd be the same object. That failure is the whole point: it's the cached-object anti-pattern caught in the act. A factory you can't mutate into a shared-state bug is a factory that's actually doing its job.

## Where this goes next

You now have the move that recurs in every chapter from here on: test data is produced by a function that returns a fresh instance, never stored as a shared constant. You can build a typed factory, compose factories without looping, keep defaults deterministic, and pick the right one of factory, fixture, or seed for the job in front of you.

A note for when your factories get bigger. There are libraries for this — `fishery`, `rosie`, and friends — that give you a builder DSL. The 2026 reach for a Next.js SaaS is to hand-roll it: a factory is about twenty lines, and a library is a dependency plus a small DSL to learn for something you can write in a function. Reach for a library only when your factories have grown into a genuine domain DSL, which is rare. And resist the chained-builder shape some libraries encourage — `buildUser().withRole('admin').build()` — it's three calls and a fluent interface to express what `buildUser({ role: 'admin' })` says in one. The override-spread shape is enough.

Every factory in this lesson returned a plain in-memory object — no database, no `await`, nothing inserted. That's deliberate: a `/lib` unit under test is a pure function, and its inputs are plain objects. The factory that *inserts* into a real database — the signed-in-user fixture, the rolled-back org seed — is the integration layer's job, and it's coming in the next chapter. Right next door, the following lesson takes the deterministic-defaults thread we started here and makes it rigorous: pinning time, IDs, and randomness behind seams your tests control.
