A user fills in a form and hits submit. On the server, a function reads `formData.get('quantity')` and gets back the string `"3"` ŌĆö not the number `3`. `formData.get('archived')` comes back as `"on"`, or as nothing at all. `formData.get('issuedAt')` is an ISO string like `"2026-01-15T10:30:00Z"`, not a `Date`. But the code on that side of the wire wants a `number`, a `boolean`, a `Date` ŌĆö the typed domain you've spent five lessons learning to describe with Zod. Everything you wrote ŌĆö the builders, the formats, the checks, the derived shapes ŌĆö describes that typed domain. The wire between the browser and the server carries none of it. It carries strings.

This lesson is the bridge. By the end you'll write `invoiceFormSchema` ŌĆö the form-input schema every action in this unit reuses ŌĆö and the one line, `safeParse(Object.fromEntries(formData))`, that turns a bag of strings into a validated, typed object. More importantly, you'll learn the four places the obvious version of that bridge silently corrupts your data, and how to feel the trap coming before you reach for the wrong tool.

## Everything off a form is a string

Start with the constraint everything else works around. When a browser submits a `<form>`, it serializes every control to text and sends it across. The server reconstructs a <Term term="A browser API that models a submitted form as a key/value collection. Every value is a string, except file uploads, which are File objects.">FormData</Term> object ŌĆö a key/value collection mirroring the form ŌĆö and every value you pull out of it is the same shape:

<CodeTooltips tooltips={{
  quantity: 'formData.get(name) returns FormDataEntryValue = string | File.\nA number input is still a string here.',
  archived: 'Returns the string "on" when checked, or null when the key is absent.\nNever a boolean.',
  issuedAt: 'An ISO datetime string. Not a Date ŌĆö a Date cannot travel over HTTP.',
}}>
```ts
// <form>
//   <input name="quantity" type="number" />
//   <input name="archived" type="checkbox" />
//   <input name="issuedAt" type="datetime-local" />
// </form>

const quantity = formData.get('quantity'); // FormDataEntryValue ŌåÆ "3"
const archived = formData.get('archived'); // FormDataEntryValue ŌåÆ "on" | null
const issuedAt = formData.get('issuedAt'); // FormDataEntryValue ŌåÆ "2026-01-15T10:30:00Z"
```
</CodeTooltips>

Look at the type the platform hands you: `FormDataEntryValue`, which is `string | File`. There is no `number` on the wire. No `boolean`. No `Date`. No array. A number input that *looks* like it holds a number gives you `"3"`. The checkbox gives you `"on"` or nothing. The date picker gives you an ISO string. This isn't a Zod fact ŌĆö it's the HTML form specification. A `<form>` can only carry text, and one other thing: files.

That gives you a picture worth keeping in your head. On the left, a browser form with controls that *look* typed ŌĆö a number spinner, a checkbox, a date picker. They cross the wire. On the right, the server, holding the same fields as quoted strings.

<FormDataBoundary />

Two more shapes to name before moving on, because they get their own sections later. A field can appear more than once ŌĆö a multi-select, or several checkboxes sharing one `name`. For those, `formData.getAll(name)` returns a `string[]`, every value, in order. And a file input on a <Term term="The form encoding used when a submission carries a file. It splits the body into parts, one per field, so binary file data can ride alongside the text fields.">multipart/form-data</Term> form gives you a `File` ŌĆö the single value off a form that *isn't* a string. Hold those two; the rest of this lesson assumes everything else is text.

## The first move: `Object.fromEntries(formData)`

You could call `formData.get` field by field, but you'd be hand-unpacking the form before Zod ever sees it. The schema already knows the field names ŌĆö let it do the unpacking. `FormData` is iterable as `[key, value]` pairs, so `Object.fromEntries` turns the whole thing into a plain object in one line:

```ts
const raw = Object.fromEntries(formData);
// { quantity: '3', archived: 'on', notes: 'net 30 terms' }
```

That `raw` object ŌĆö every value still a string ŌĆö is exactly what you'll hand to `safeParse`. One call collects the entire form; the schema validates and coerces the whole thing in a single pass. This is the shape every form-consuming function in this unit opens with, so it's worth getting the reflex into your fingers.

But `Object.fromEntries` has one sharp edge, and it's sharp enough to be the first of this lesson's four traps. It's a property of `fromEntries` itself, not of Zod, which is why it bites here at the very start. When a key appears more than once, `Object.fromEntries` keeps **only the last value**. Every earlier value is silently dropped.

<CodeVariants>
  <CodeVariant label="Naive ŌĆö last value wins">
    <div data-mark-color="red">

    ```ts {1}
    const raw = Object.fromEntries(formData);
    // tags: 'paid'  ŌåÉ only the LAST checked value survived
    ```

    </div>
    This looks complete, but `tags` was checked twice. `fromEntries` collapsed both to the last one. The schema's `z.array(z.string())` will receive a single string `'paid'` ŌĆö not an array ŌĆö and fail with a confusing "expected array, received string." The bug is upstream of Zod; the schema is innocent.
  </CodeVariant>

  <CodeVariant label="Correct ŌĆö getAll for arrays">
    <div data-mark-color="green">

    ```ts {3}
    const raw = {
      ...Object.fromEntries(formData),
      tags: formData.getAll('tags'),
    };
    // tags: ['urgent', 'paid']  ŌåÉ both values, as an array
    ```

    </div>
    Spread `fromEntries` for the scalar fields, then override each multi-valued key with `getAll`, which returns the full `string[]`. Now the schema receives the array it expects.
  </CodeVariant>
</CodeVariants>

The fix is mechanical: spread `Object.fromEntries` for the ordinary single-value fields, then override each multi-valued key with `formData.getAll`. The cost is that you have to *know* which fields are multi-valued ŌĆö there's no automatic detection. The convention that saves you is to name those inputs deliberately (a plural like `tags`) and remember that any field whose schema is a `z.array(...)` needs a `getAll`. Forget it, and the schema receives one lonely string where it wanted an array, and the error message points at the schema when the real bug is here, two lines up.

## `z.coerce`: a transform that runs a constructor

Now the bridge itself. You have a `raw` object full of strings; the schema wants numbers and dates. The tool is `z.coerce`, and it's not a new idea ŌĆö you already learned it in [Checks and transforms](/042-zod-4---the-validation-contract/3-checks-and-transforms/), you just didn't have its name.

Recall what a transform does: it runs a function on the input before the inner schema validates, and it changes the inferred output type. `z.coerce` is exactly that, with the transform function fixed to a JavaScript constructor. `z.coerce.number()` is `z.number()` with a built-in transform that calls `Number(input)` first. The chain is the same every time:

- `z.coerce.number()` runs `Number(input)`, then validates a number.
- `z.coerce.date()` runs `new Date(input)`, then validates a date.
- `z.coerce.bigint()` runs `BigInt(input)`, then validates a bigint.
- `z.coerce.string()` runs `String(input)`, then validates a string.

The input is accepted as `unknown`, the constructor runs, and the inner schema checks the *result*. That last detail matters: any `.positive()` or `.min()` you chain validates the coerced value, not the original string.

This is also where the input/output type split from [Derive, don't duplicate](/042-zod-4---the-validation-contract/4-derive-dont-duplicate/) stops being abstract. A `z.coerce.number()` schema has **input** `unknown` and **output** `number`. The form sends a string; the parsed value is a number; the two types are genuinely different. That's the case `z.input` and `z.output` exist for. The form contract ŌĆö what the `<form>` is allowed to send ŌĆö is `z.input<typeof schema>`. The validated value your code works with afterward is `z.output<typeof schema>` (which is what `z.infer` gives you). Coercion is the reason a form schema has two faces.

Here's the schema this whole unit is built on. Read it one field at a time.

The line every form-consuming function in this unit opens with assembles those two moves into one:

```ts

```

`Object.fromEntries` collects the form into a string-valued object; `safeParse` coerces and validates it against the schema. After this line, `parsed` is either a success holding a fully typed invoice or a failure holding the issues ŌĆö the `parsed.success` branch, the error rendering, the database write all come next chapter, where you wire this schema into an actual Server Action. For now, every snippet stops right here, at the `safeParse`. This one line is the whole reason the lesson exists: it's the seam where untrusted strings become trusted, typed data.

## The boolean trap: `z.coerce.boolean()` is wrong for checkboxes

The headline trap. A checkbox is the most common boolean on any form, and the obvious schema for it ŌĆö `z.coerce.boolean()` ŌĆö is wrong in a way that's worse than throwing: it never fails.

Start with the wire shape, precisely, because the danger comes from it. A checked checkbox sends `name=on`. An **unchecked checkbox sends nothing at all** ŌĆö the key is simply absent from the FormData, so after `Object.fromEntries` the field is `undefined`. Not `"off"`, not `"false"`, not `""`. Present-or-absent, never present-with-a-false-value.

Now `z.coerce.boolean()`. Its transform calls `Boolean(input)`, and `Boolean` treats *every* non-empty string as true. That has two consequences, and both are silent:

- **It inverts any literal boolean string.** `Boolean('false')` is `true`. `Boolean('off')` is `true`. A field carrying the literal text `"false"` or `"off"` coerces to `true`. Checkboxes never send those (they send `"on"` or nothing), so this one doesn't bite the checkbox directly ŌĆö but the instant you have a `<select>` or a hidden field whose value is the word `"true"`/`"false"` or `"on"`/`"off"`, `z.coerce.boolean()` flips half your data the wrong way.
- **It can never reject a bad value.** `Boolean(anything)` always returns a real boolean, so the inner `z.boolean()` always sees a valid boolean and the parse *always succeeds*. `'on'`, `'false'`, `'off'`, `''`, even the absent `undefined` all sail through ŌĆö `undefined`ŌåÆ`false`, every non-empty stringŌåÆ`true`. There is no input `z.coerce.boolean()` rejects, so it can't catch a malformed value, and it silently conflates the absent-when-unchecked case with a deliberate `false`.

:::caution
`z.coerce.boolean()` is `Boolean()` in a wrapper: it accepts *everything* and rejects *nothing*. If you've seen it described as a way to "validate" a boolean off a form, that's the trap ŌĆö it doesn't validate, it just truthiness-tests, and truthiness is the wrong question for form strings.
:::

So the obvious tool would invert any `"false"`/`"off"` it ever saw and can never reject garbage. The correct shape is a `z.preprocess` ŌĆö a transform that maps the wire value to a real boolean *before* `z.boolean()` validates it:

```ts
archived: z.preprocess((value) => value === 'on' || value === true, z.boolean()),
```

Walk the pieces. `value === 'on'` catches the checked checkbox. `|| value === true` catches a programmatic or JSON caller that already sent a real boolean. Anything else ŌĆö the absent `undefined`, an empty string, a stray `"false"` ŌĆö falls through to `false`. The result is always a genuine boolean, which the inner `z.boolean()` then validates. The unchecked checkbox becomes `false`, exactly as a user expects, and the schema means *exactly* `'on'` rather than "anything truthy."

There's one more boolean tool worth knowing so you don't reach for the wrong one. Zod 4 ships `z.stringbool()`, built for strings that *spell* a boolean: it accepts `"true"`/`"false"`, `"yes"`/`"no"`, `"on"`/`"off"`, `"1"`/`"0"`, case-insensitively (and you can customize the word lists with `{ truthy, falsy }`). It's the right reach for a `<select>` or text field whose value is literally the word "true." But it expects a *present string* ŌĆö it doesn't model the absent-when-unchecked checkbox. So the line is clean: **`z.stringbool()` for a string that spells a boolean; `z.preprocess` for a checkbox that's present-or-absent.** And `z.coerce.boolean()` for neither, when the source is form data.

## The empty-string trap: optional numbers that become zero

The second trap is the same shape as the first ŌĆö a successful parse that produces the wrong value ŌĆö but on a field you might not think to check. It never throws, never turns a row red. It just writes a wrong number to your database and moves on.

Here's the footgun. `Number('')` is `0`. Not `NaN`, not an error ŌĆö a clean, plausible zero. An empty text or number input submits the empty string `''` (the key is present, the value is blank). So `z.coerce.number()` on a blank field runs `Number('')`, gets `0`, and that `0` sails through any `.nonnegative()` or `.min(0)` you put after it. Picture a user who leaves an optional "discount" field empty. They meant "no discount." The schema records a discount of zero ŌĆö a real, applied, zero-percent discount. The parse succeeds. Nothing anywhere signals that a blank just became a number.

The boundary is worth drawing, because `z.coerce.number()` isn't wrong everywhere. On a **required** field with a sensible floor ŌĆö a price that must be `.positive()`, a quantity that's `.min(1)` ŌĆö a blank submission *should* fail, and it does: `Number('')` is `0`, and `0` fails `.positive()`, so the user gets the validation error they should. The trap is specific. It bites **optional** numerics, where "blank" is a legitimate answer meaning "absent" and the schema quietly translates it to "zero" instead.

The fix is to decide, in the schema, that a blank means absent ŌĆö by mapping the empty string to `undefined` *before* coercion runs.

<CodeVariants>
  <CodeVariant label="Naive ŌĆö silent zero">
    <div data-mark-color="red">

    ```ts {2}
    const schema = z.object({
      discount: z.coerce.number().nonnegative().optional(),
    });
    // { discount: '' }  ŌåÆ  { discount: 0 }   ŌåÉ blank became a real zero
    ```

    </div>
    `.optional()` looks like it handles the blank case, but the blank isn't `undefined` ŌĆö it's the string `''`. `z.coerce.number()` runs `Number('')` and gets `0`, which passes `.nonnegative()`. The `.optional()` never fires. A user who entered nothing now has a zero discount, and the parse succeeded, so nothing warned you.
  </CodeVariant>

  <CodeVariant label="Correct ŌĆö empty maps to undefined">
    <div data-mark-color="green">

    ```ts {2-5}
    const schema = z.object({
      discount: z.preprocess(
        (value) => (value === '' ? undefined : value),
        z.coerce.number().nonnegative().optional(),
      ),
    });
    // { discount: '' }  ŌåÆ  { discount: undefined }   ŌåÉ absent, as intended
    ```

    </div>
    The preprocess strips the empty string to `undefined` first; *then* the optional-number schema sees genuine absence and leaves it `undefined`. Now blank means absent, and every consumer reads `number | undefined` ŌĆö they can't mistake a blank for a zero, because the schema already made the call.
  </CodeVariant>
</CodeVariants>

There's a second shape for the same intent, heavier but more explicit at the value level ŌĆö a union that names the empty string and transforms it away:

```ts
discount: z.union([
  z.literal('').transform(() => undefined),
  z.coerce.number().nonnegative(),
]).optional(),
```

Reach for the union form when you want "the empty string means nothing" to read as a deliberate value-level rule rather than a preprocessing step; reach for the `z.preprocess` form for the common case. Either way, the senior move is the same: the schema is where "blank means absent" gets decided, once. Encode it there and every consumer of the parsed value sees `number | undefined` and physically cannot treat a blank as a zero ŌĆö the wrong interpretation never escapes the boundary.

This trap is best *read*, not graded ŌĆö because the naive schema and the fixed one both parse a blank successfully; only the produced value differs. ## The date trap: `z.coerce.date()` is too loose about format

The third trap isn't that `z.coerce.date()` lets garbage through ŌĆö feed it `"not-a-date"` and it rejects cleanly. The trap is the opposite: it's *too lenient* about what counts as a date, and it accepts a string your contract should refuse.

`z.coerce.date()` runs `new Date(input)`. `new Date` happily parses far more than a full ISO 8601 timestamp ŌĆö a bare date with no time, `"2026-01-15"`, parses straight to a `Date` (midnight UTC). That's the problem. If your contract is "an invoice's `issuedAt` is a precise instant" but the form sends a date-only `"2026-01-15"`, `z.coerce.date()` says yes and silently invents a `00:00:00Z` time you never agreed to. The schema accepted a shape it should have named and rejected. `z.coerce.date()` validates *that the input is parseable as some date*, not *that it's the date format your contract specifies* ŌĆö and on a boundary, the format is part of the contract.

The tool that names the format is `z.iso.datetime()`: it requires a full ISO 8601 timestamp with a time component and a `Z`, and it rejects a date-only string outright. Both reject true garbage like `"not-a-date"`; the difference is the date-only case, where `z.coerce.date()` is lenient and `z.iso.datetime()` holds the line.

See the looseness directly. The exercise below is inverted on purpose: the starter uses bare `z.coerce.date()`, so the date-only fixture ŌĆö which the contract should refuse ŌĆö wrongly **passes (stays green when it should be red)**. You fix a false *pass* by switching to the format that names the contract.

  schemaName="issuedAtSchema"
  instructions="Backwards on purpose. So `z.iso.datetime()` is how you name the format. But you have a second choice to make ŌĆö whether the action wants the date as a `string` or a `Date` ŌĆö and which one you pick depends entirely on where the date goes *next*. That's the rule this whole section reduces to: **pick by where the date is used.**

<TabbedContent>
  <TabbedItem label="Stays a string ŌåÆ z.iso.datetime()">
    ```ts
    issuedAt: z.iso.datetime(),
    // output type: string
    ```

    Reach for this when the date stays text until the database. `z.iso.datetime()` validates the *string format* ŌĆö a full ISO 8601 timestamp, with a time component and a `Z` ŌĆö and infers as `string`. The action receives a `string`; if the next stop is a `timestamptz` column, Drizzle converts it on write. No JS `Date` needed.
  </TabbedItem>

  <TabbedItem label="Needs a Date in JS ŌåÆ z.iso.datetime().transform(...)">
    ```ts

    // output type: Date
    ```

    Reach for this when downstream code does date math or formatting in JavaScript. It validates the full-timestamp string *strictly first*, then constructs the `Date` ŌĆö so the loose formats `z.coerce.date()` would have waved through are already rejected before any `Date` exists.   </TabbedItem>
</TabbedContent>

If the date stays text all the way to the database, use `z.iso.datetime()`: it validates the format and infers as `string`, and the action passes a string straight to a `timestamptz` column. If JavaScript code needs a real `Date` to do math or formatting, use `z.iso.datetime().transform(s => new Date(s))`: it validates the strict format first, *then* constructs the `Date`, so a date-only or otherwise off-contract string is rejected before any Date is built. Either way you've named the exact format the contract requires. Bare `z.coerce.date()` is the one to avoid on a boundary: it accepts whatever `new Date` can parse, which is looser than the contract, and you find out only when a half-specified date has already become a row. That's the trap.

## Validating uploaded files

The last seam is the one non-string value off a form. A file input on a `multipart/form-data` form makes `formData.get('avatar')` return a `File` instance ŌĆö the only place the wire carries something other than text.

The validator at this seam is `z.instanceof(File)`, and you constrain it the same way you'd constrain anything else: with refinements. The two checks that matter at the boundary are size and type.

```ts
const MAX_BYTES = 5 * 1024 * 1024;
const ALLOWED_TYPES = ['image/png', 'image/jpeg', 'image/webp'];

const avatarSchema = z
  .instanceof(File)
  .refine((file) => file.size > 0, { error: 'No file uploaded' })
  .refine((file) => file.size <= MAX_BYTES, { error: 'File too large' })
  .refine((file) => ALLOWED_TYPES.includes(file.type), {
    error: 'Unsupported file type',
  });
```

`file.size` is the byte count; `file.type` is the file's <Term term="The content-type string a File carries, like image/png. The browser sets it from the file; treat it as a hint, not a guarantee.">MIME type</Term> ŌĆö a string like `image/png`. The `file.size > 0` refine earns its place because an empty file input yields a zero-byte `File` in some browsers rather than nothing at all, so "no file" and "a file" both arrive as a `File`; the size check is how you tell them apart. For a genuinely optional file field, add `.optional()` and let that guard handle the empty case.

Two things to keep this in scope. First: `File` is a Web API global, not a browser-only one. Next.js runs Server Actions and route handlers on a runtime that provides `File`, so this validation works server-side ŌĆö you're not reaching for a polyfill. Second, and more important: this validates the file's *shape* at the boundary and nothing more. The real upload story ŌĆö presigned URLs, object storage, streaming the bytes somewhere durable ŌĆö is a whole chapter later in the course. Here, a file is just one more field the schema checks before the rest of the action runs. Validate that it's a file, that it's not too big, that it's a type you allow, and move on.

## Putting the bridge together

Pull the patterns into the schema you'll actually carry forward. This is `invoiceFormSchema` with the traps designed out: the coerced number, the strict date, the preprocess boolean for the checkbox, the optional notes.

- A **number** off a form ŌåÆ `z.coerce.number()` ŌĆö but an **optional** number maps empty to `undefined` first, or you'll write a silent zero.
- A **boolean checkbox** ŌåÆ `z.preprocess(v => v === 'on' || v === true, z.boolean())`, never `z.coerce.boolean()`.
- A **date** ŌåÆ `z.iso.datetime()` if it stays text, or the strict `.transform(s => new Date(s))` if you need a `Date` ŌĆö never bare `z.coerce.date()`.
- An **array** field ŌåÆ `formData.getAll(name)`, not `Object.fromEntries`.
- A **file** ŌåÆ `z.instanceof(File)` with size and type refinements.
