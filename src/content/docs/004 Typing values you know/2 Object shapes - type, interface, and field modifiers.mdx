The previous lesson installed the vocabulary for scalar values — primitives, literal unions, and the four corners. This lesson does the same for the next most common type you'll author: the object shape. Two production failure modes anchor it.

<CodeVariants>
  <CodeVariant label="The codebase uses both">

    ```ts
    // src/lib/auth.ts
    type User = {
      id: string;
      email: string;
    };

    // src/lib/profile.ts
    interface User {
      id: string;
      email: string;
    }
    ```

    You open two files in the same repo. One declares `User` with `type`; the other with `interface`. They produce the same shape at the call site, so neither one is wrong — but they can't both be canonical. New engineers waste an hour every onboarding deciding which form to write next. The fix is one rule with one narrow exception, and this lesson installs both.
  </CodeVariant>

  <CodeVariant label="The readonly that wasn't">
    <div data-mark-color="red">

    ```ts "invoice.lines.push(line)"
    type Invoice = {
      id: string;
      lines: InvoiceLine[];
    };

    const addLine = (invoice: Invoice, line: InvoiceLine) => {
      invoice.lines.push(line);
    };
    ```

    </div>
    `invoice` is meant to be the source of truth for a rendered invoice. Then a downstream helper appends a line to the array in place, the parent component holds a stale reference for a render cycle, and two screens display different totals before anyone notices. The fix is a `readonly` modifier — but `readonly` only locks one of the two things engineers assume it locks. This lesson lands the boundary.
  </CodeVariant>
</CodeVariants>

Both failures are vocabulary failures, just like the typo and `any` bugs in the previous lesson. The engineer reached for the wrong keyword or skipped a modifier the language already provides.

Four senior reflexes come out of this lesson:

- `type` by default; `interface` only for declaration merging.
- `?` for optional fields, with the `| undefined` distinction the strict tsconfig will eventually enforce.
- `readonly` on a field locks the binding, not the value behind it.
- `readonly T[]` for array immutability, and `Readonly<T>` for the per-field shorthand.

What this lesson doesn't cover: dynamic keys (next lesson), unions and intersections, narrowing (the lessons after that). We stay on a single object shape with known field names.

## `type` by default

Here's the rule plainly: **`type` is the default for every alias the course writes — objects, primitives, tuples, unions, intersections, generics, conditional types, mapped types.** The project's code conventions pin the rule because the legibility win is real: the team never debates the call.

```ts
type User = {
  id: string;
  email: string;
  name: string;
};
```

That's the canonical shape, and it's what almost every type declaration in this course looks like. One keyword, alphabetical-ish fields, semicolons on the field lines (or commas — Biome handles whichever you write).

The mechanical reason `type` wins as the default is that it scales. `interface` declares only object shapes and classes. `type` declares all of that plus everything else the chapter will introduce — unions in a couple of lessons, intersections in the same lesson, conditional and mapped types in the next chapter, generic helpers throughout. Defaulting to `type` removes a decision you'd otherwise make every time you reach for a non-object shape.

A short vocabulary note: the word <Term definition="A name bound to a type expression. The type itself is structural — the alias is just a label TypeScript uses when reporting errors and showing hover info.">alias</Term> shows up everywhere in the TypeScript docs and in this chapter. `User` above is a `type` alias for `{ id: string; email: string; name: string }`. The alias has no runtime presence; it's erased at compile time.

## `interface` earns one trigger: declaration merging

There's exactly one job `type` can't do: <Term definition="Two declarations with the same name in the same scope combine into one. interface allows it; type rejects it as a duplicate identifier. The only practical use is augmenting a type that another package shipped.">declaration merging</Term>. Two `interface` declarations of the same name in the same scope merge into one combined shape. Two `type` declarations of the same name are a duplicate-identifier error.

<CodeVariants>
  <CodeVariant label="Two type — duplicate identifier">
    <div data-mark-color="red">

    ```ts
    type User = {
      id: string;
    };

    type User = {
      email: string;
    };
    // Error: Duplicate identifier 'User'.
    ```

    </div>
    Two `type` declarations with the same name collide. The compiler refuses to choose between them.
  </CodeVariant>

  <CodeVariant label="Two interface — silently merged">
    <div data-mark-color="green">

    ```ts
    interface User {
      id: string;
    }

    interface User {
      email: string;
    }

    const user: User = { id: '1', email: 'lina@example.com' };
    ```

    </div>
    Both declarations merge into one `User` with both fields. This is the feature when you mean it — and a silent footgun when two unrelated declarations accidentally collide in the same scope.
  </CodeVariant>
</CodeVariants>

The canonical case for the merge is augmenting a type that a third-party package ships. Better Auth exports a `Session` type, and your app needs to add a `currentOrgId` field to it. Next-intl exports an `AppConfig` type, and your app fills in the shape of your translation messages and locale list. The mechanism is a `declare module 'package-name'` block containing an `interface` declaration that targets the package's existing type — every import of the package, anywhere in your codebase, sees the merged shape.

You won't write a `declare module` block in this chapter. Lesson 4 of chapter 6 owns the full pattern. The point here is the **decision**: write `type` by default; reach for `interface` only when you're inside a `declare module` block augmenting a third-party type.

:::note
Module augmentation is the full pattern (lesson 4 of chapter 6). This lesson just names the trigger so you read the move correctly when you meet it.
:::

## The non-differences

Three differences between `type` and `interface` show up in blog posts and Stack Overflow threads. None of them change the senior call at this lesson's scale, so let's dispatch them in one place and stop thinking about them.

**`extends` vs `&`.** `interface B extends A` and `type B = A & { /* extra fields */ }` produce structurally equivalent types for everything this course will type. The course composes with `&` because it's the only form that works for unions too. Composition lands a few lessons from now.

**Semicolons, commas, trailing commas.** Cosmetic. Biome enforces a single style on save; you write whichever feels natural and stop reading it back.

**Performance at extreme scale.** `interface` checks marginally faster than very large unions of intersections in pathological cases — the kind library authors hit. At a single-app SaaS scale, the difference doesn't bite.

The only difference that matters for the senior call is declaration merging. Everything else is style.

## The `?` modifier: optional fields

A `?` after a field name marks the field as **optional** — the property may be absent from the object entirely.

```ts
type User = {
  id: string;
  email: string;
  name?: string;
};

const anon: User = { id: '1', email: 'anon@example.com' };
const lina: User = { id: '2', email: 'lina@example.com', name: 'Lina' };
```

Both literals satisfy `User`. The first omits `name`; the second sets it. That's the basic move, and it covers the common case — fields a database column allows `NULL` on, fields the user hasn't filled in yet, fields a particular API response doesn't return.

What's worth knowing is that there are actually three closely-related forms — `name?: string`, `name: string | undefined`, and `name?: string | undefined` — and the differences between them are real, even when TypeScript sometimes lets you slide between them.

<CodeVariants>
  <CodeVariant label="name?: string">

    ```ts
    type User = {
      id: string;
      name?: string;
    };

    const user: User = { id: '1' };
    ```

    `name` may be absent from the object. `'name' in user` is `false`. `Object.keys(user)` returns `['id']`. `JSON.stringify(user)` produces `'{"id":"1"}'` — no `name` key in the output.
  </CodeVariant>

  <CodeVariant label="name: string | undefined">

    ```ts
    type User = {
      id: string;
      name: string | undefined;
    };

    const user: User = { id: '1', name: undefined };
    ```

    `name` must be present in the literal — even if its value is `undefined`. `'name' in user` is `true`. `Object.keys(user)` returns `['id', 'name']`. `JSON.stringify(user)` still produces `'{"id":"1"}'` because `JSON.stringify` drops `undefined` values, which is its own surprise.
  </CodeVariant>

  <CodeVariant label="name?: string | undefined">

    ```ts
    type User = {
      id: string;
      name?: string | undefined;
    };

    const absent: User = { id: '1' };
    const present: User = { id: '2', name: undefined };
    ```

    The combined form — accepts both absence and an explicit `undefined` value. The most permissive shape; under the project's default tsconfig, this is what TypeScript treats `name?: string` as equivalent to.
  </CodeVariant>
</CodeVariants>

The runtime difference between "absent" and "present-but-`undefined`" is real and it doesn't go away just because the compiler sometimes ignores it. The `in` operator separates them. `Object.keys` enumerates the second but not the first. `JSON.stringify` drops `undefined` values silently. A library that iterates with `for (const key in obj)` will skip the absent field and visit the explicit-`undefined` one.

Under plain `strict`, TypeScript treats `name?: string` and `name: string | undefined` as assignable to each other at object-literal sites. You can write `{ name: undefined }` for a `name?: string` field and the compiler shrugs. That's a deliberate ergonomic choice — and the cost is that the runtime distinction the type system describes isn't enforced.

Later in the course, when the project pins the full strict tsconfig (lesson 5 of chapter 24), one of the flags it turns on is <Term definition="TypeScript compiler flag that enforces the distinction between optional (absent) and present-but-undefined fields. Without it, the compiler treats name?: string and name: string | undefined as interchangeable.">`exactOptionalPropertyTypes`</Term>. With that flag on, a `name?: string` field rejects the value `undefined` — the field must be either absent or a `string`. The compiler finally models what the runtime already did.

:::tip
With `exactOptionalPropertyTypes` on, write `?` when a field can be absent or a typed value, and `| undefined` only when the API contract genuinely requires the property to be present with the value `undefined`. Don't combine the two unless you mean both.
:::

## The `readonly` modifier: locks the binding, not the value

`readonly` before a field name forbids reassignment of the property after the object is constructed. It does not freeze the value the property points at. The mental model is identical to `const` from the JavaScript value-model chapter: the binding is locked, the value is not.

This is the field-level rule and the place most engineers form a wrong intuition about it. Walk through it line by line.

<AnnotatedCode lang="ts" code={`
type Invoice = {
  readonly id: string;
  readonly issuedAt: Date;
  readonly customer: { name: string; email: string };
};

const invoice: Invoice = {
  id: 'inv_1',
  issuedAt: new Date(),
  customer: { name: 'Lina', email: 'lina@example.com' },
};

invoice.id = 'inv_2';
invoice.customer = { name: 'Mara', email: 'mara@example.com' };
invoice.customer.name = 'Mara';
`}>
  <AnnotatedStep meta="{1-5}" color="green">
    **The declaration.** Three `readonly` fields. The shape says: once an `Invoice` is constructed, you cannot reassign `id`, `issuedAt`, or `customer` on it. That's what's locked.
  </AnnotatedStep>

  <AnnotatedStep meta="{13-14}" color="red">
    **The two reassignments that fail.** Writing to `invoice.id` is the obvious case — the type says `readonly`, the compiler refuses. Writing to `invoice.customer` is the same rule: even though the new value structurally matches `{ name: string; email: string }`, you're reassigning the property, and `customer` is `readonly`.
  </AnnotatedStep>

  <AnnotatedStep meta="{15}" color="orange">
    **The mutation that slips through.** `readonly customer` locks the `customer` property, not the object the property points to. The nested `name` field on that inner object is mutable. `invoice.customer.name = 'Mara'` compiles cleanly — and the customer's name is silently overwritten. This is the bug the introduction described.
  </AnnotatedStep>
</AnnotatedCode>

The rule fits in a sentence: `readonly` locks the binding from the alias to the value, not the value itself. It's the field-level analogue of `const`. The same rules you internalized for `const` apply here, one level deeper into the data.

For nearly all SaaS code, the field-level `readonly` is the right reach. React's rendering model already discourages mutating values it owns, the framework re-runs components with fresh references on every render, and the cases where a nested write actually bites are rarer than blog posts make them sound. When you do need to lock the nested value too — typed config tables, frozen registries, lookup tables you want to guarantee immutable — `as const` (lesson 7 of this chapter) is the right move. There's a deeper "deep readonly" pattern for the rare cases where neither approach fits, but you don't need it today.

## Array-level readonly: `readonly T[]` and `Readonly<T>`

Now return to the introduction's `Invoice` with `lines`. The fix that ships in real codebases isn't a `readonly` on the field — it's a `readonly` on the *array type*.

<CodeVariants>
  <CodeVariant label="Mutable array — readonly lines: InvoiceLine[]">
    <div data-mark-color="red">

    ```ts "invoice.lines.push(line)"
    type Invoice = {
      readonly id: string;
      readonly lines: InvoiceLine[];
    };

    const addLine = (invoice: Invoice, line: InvoiceLine) => {
      invoice.lines.push(line);
    };
    ```

    </div>
    `readonly` on the field locks the `lines` reference — `invoice.lines = []` would error. But the array methods aren't constrained: `.push`, `.pop`, `.splice`, `.sort`, index-write all still compile. The bug from the introduction is back.
  </CodeVariant>

  <CodeVariant label="Immutable array — readonly lines: readonly InvoiceLine[]">
    <div data-mark-color="green">

    ```ts ins="readonly InvoiceLine[]"
    type Invoice = {
      readonly id: string;
      readonly lines: readonly InvoiceLine[];
    };

    const addLine = (invoice: Invoice, line: InvoiceLine) => {
      invoice.lines.push(line);
      // Error: Property 'push' does not exist on
      // type 'readonly InvoiceLine[]'.
    };
    ```

    </div>
    The array element type is now `readonly InvoiceLine[]`. The mutating methods disappear from its surface; the read methods (`.map`, `.filter`, `.find`, indexed read) stay. The push errors at compile time, not after a stale-render incident.
  </CodeVariant>
</CodeVariants>

`readonly T[]` forbids `.push`, `.pop`, `.shift`, `.unshift`, `.splice`, `.sort`, `.reverse`, and direct index-write. It leaves every read method intact. The longer-named equivalent is `ReadonlyArray<T>` — same type, written as a utility. The course writes `readonly T[]` for concision.

When you have an existing object type and you want a fully-frozen variant of it without manually retyping every field, TypeScript ships a per-field shorthand:

```ts
type Invoice = {
  id: string;
  issuedAt: Date;
  customer: { name: string; email: string };
};

type FrozenInvoice = Readonly<Invoice>;
```

`Readonly<T>` applies `readonly` to every top-level property of `T`. `FrozenInvoice` here is exactly equivalent to writing `readonly` in front of each of `Invoice`'s three fields. The trigger is "I already have a type and I want a variant where every field is locked." It's shallow — the `customer.name` field is still mutable, same as the manual-`readonly` case from the previous section. The full utility-type surface (`Pick`, `Omit`, `Partial`, and friends) lands in the next chapter; `Readonly<T>` shows up here because it's the per-field shorthand for the rule this section just installed.

One forward note: when the object is a literal config you're authoring inline — a routes map, a permissions table — `as const` (lesson 7 of this chapter) is the right reach. It locks every property at the narrowest literal type *and* applies `readonly` everywhere in one keyword. You don't need it for the `Invoice` derived from a database row; you'll want it for the literal config you write next to the call site.

## Excess property checks

There's one more rule worth naming once, because if you don't know it exists you'll find it confusing the first time it bites.

```ts
type User = { id: string; email: string };

const direct: User = { id: '1', email: 'lina@example.com', role: 'admin' };
//                                                          ^^^^
// Error: Object literal may only specify known properties,
// and 'role' does not exist in type 'User'.

const draft = { id: '1', email: 'lina@example.com', role: 'admin' };
const flowed: User = draft;
```

The line declaring `direct` errors. The line assigning `draft` to `flowed` compiles cleanly — the extra `role` field is silently along for the ride.

The reason is TypeScript runs an extra check on **object literals assigned directly to a known type**. If the literal has extra properties the target type doesn't declare, the compiler flags them. When the same value flows through a variable first, the check is skipped — the variable's *inferred* type already structurally matches `User` (a `User` is allowed to have extra properties at runtime; you just can't put them in the literal), and TypeScript doesn't re-check at the second assignment.

The literal-site error is a feature. It catches the typo `emial` instead of `email` at the most informative spot — where you wrote it, not where it's read three files away. The variable-flow loophole isn't a bug either; it's the language preferring structural compatibility over strict literal matching once a value has a name.

The senior reflex when the variable-flow path is the bug: type the source variable (`const draft: User = { ... }`) so the literal-site check fires there too. The wrong reflex is to silence the literal-site error with `as User`. Type assertions don't fix anything; they hide it. Lesson 6 of this chapter unpacks that posture in full.

## One sentence on dynamic keys

Everything in this lesson assumed objects with **known field names** — `id`, `email`, `name`, `lines`. Objects with **dynamic keys** — a cache keyed by user ID, a lookup table from status to label, a JSON shape whose keys aren't known at design time — need a different form: an index signature or `Record<K, V>`. Next lesson owns that surface in full.

## Pick the right modifier combination

You're typing a `User` row for an invoicing app. The project's `tsconfig` has `exactOptionalPropertyTypes` on, so the `?` vs `| undefined` distinction is enforced. For each field, pick the modifier combination that matches the field's contract.

<MultipleChoice>
  **`id`** — a UUID assigned at row creation, never reassigned in app code, and always present on every row. Which declaration matches the contract?

  <McqChoice>`id: string`</McqChoice>
  <McqChoice>`id?: string`</McqChoice>
  <McqChoice>`readonly id?: string`</McqChoice>
  <McqChoice correct>`readonly id: string`</McqChoice>

  <McqWhy>The field is assigned once at row creation and never reassigned, so the binding is locked with `readonly`. It's always present on the row, so no `?` — absence isn't a valid state.</McqWhy>
</MultipleChoice>

<MultipleChoice>
  **`email`** — captured at sign-up, always present on every row, and never reassigned after creation (email changes go through a separate `pendingEmail` workflow). Which declaration matches the contract?

  <McqChoice>`email: string`</McqChoice>
  <McqChoice>`email?: string`</McqChoice>
  <McqChoice>`email: string | undefined`</McqChoice>
  <McqChoice correct>`readonly email: string`</McqChoice>

  <McqWhy>Same shape as `id`. The row guarantees `email` is present, so no `?` — absence isn't a valid state. App code never reassigns it, so `readonly` locks the binding.</McqWhy>
</MultipleChoice>

<MultipleChoice>
  **`name`** — the user may not have set one yet. When the field hasn't been filled in, it's absent from the row entirely (no `null`, no `undefined` value — just no key). App code never reassigns it after the row is loaded. Which declaration matches the contract?

  <McqChoice>`name: string`</McqChoice>
  <McqChoice>`name?: string`</McqChoice>
  <McqChoice>`name: string | undefined`</McqChoice>
  <McqChoice correct>`readonly name?: string`</McqChoice>

  <McqWhy>Absence is a valid state, so `?`. The row is never rewritten in app code, so `readonly` locks the binding. Under `exactOptionalPropertyTypes`, `?` means absent or a `string` — exactly the contract here. `name: string | undefined` would force the key to be present, and a bare `name?: string` would leave the binding reassignable.</McqWhy>
</MultipleChoice>

<MultipleChoice>
  **`lastSeenAt`** — present on every row, but the database column allows `NULL`. When the user has never signed in, the field is present with the value `null`. App code never reassigns it. Which declaration matches the contract?

  <McqChoice>`lastSeenAt: Date | null`</McqChoice>
  <McqChoice>`lastSeenAt?: Date`</McqChoice>
  <McqChoice>`lastSeenAt?: Date | null`</McqChoice>
  <McqChoice correct>`readonly lastSeenAt: Date | null`</McqChoice>

  <McqWhy>The row guarantees the key is present, so no `?` — `null` is the value, not absence. `Date | null` models the database `NULL`. App code never rewrites the field, so `readonly` locks the binding. A bare `lastSeenAt: Date | null` would leave the binding reassignable; `lastSeenAt?: Date` would lose the `null` and model absence instead.</McqWhy>
</MultipleChoice>

<MultipleChoice>
  **`overrides`** — a list of role overrides loaded with the row. App code must not mutate the array (no `.push`, no index-write) and must not reassign the field. Which declaration matches the contract?

  <McqChoice>`overrides: RoleOverride[]`</McqChoice>
  <McqChoice>`readonly overrides: RoleOverride[]`</McqChoice>
  <McqChoice>`overrides: readonly RoleOverride[]`</McqChoice>
  <McqChoice correct>`readonly overrides: readonly RoleOverride[]`</McqChoice>

  <McqWhy>Two locks, two layers. `readonly` on the field locks the binding so app code can't reassign `user.overrides`. `readonly RoleOverride[]` removes `.push`, `.pop`, and `.splice` from the array's surface. Either one alone leaves the other layer mutable.</McqWhy>
</MultipleChoice>

If you can make the call on each of those five fields without thinking, the `?` / `| undefined` / `readonly` / `readonly T[]` matrix is installed. The wrong answers aren't decoy noise — each one is a real mistake you'll catch yourself making in the first month of writing types.
