The previous lessons gave you the vocabulary for a single value or shape — primitives, literal unions, named-field objects, tuples, dynamic-keyed records. With that surface in hand, you can name every type for one value. What you can't do yet is **compose** the shapes you already know into the kinds of types real codebases ship every day. Three quick scenarios anchor what that means in practice.

```ts
type FormattedAmount = string | number;

type UserOrMiss = User | null;

type AuthedRequest = BaseRequest & { token: string };
```

Three compositions. A formatter parameter that accepts either a numeric amount or a pre-formatted string. A query result that returns a `User` row when the lookup hits and `null` when it misses. A request payload built from a base type with an auth token bolted on. Each one takes shapes you already know — a primitive, a `User`, a `null` value, a `BaseRequest` — and combines them with one of two operators.

The operators read in plain English. The `|` reads as **or** — a value of `A | B` is *one of* the alternatives. The `&` reads (on shape types) as **plus** — a value of `A & B` has the fields of *both* shapes. This lesson teaches both operators, the senior call between them, and the bug class each prevents. It also plants the seed for the **discriminated union** — the pattern the next chapter builds out — by showing the shape and the bug it fixes.

One preview. Composing a shape union is the easy part. **Reading** a field on the result is where the first compile error fires, and the fix isn't to widen the type or assert past it. The fix is to narrow, and the next lesson owns the full narrowing surface. The compile error is the lesson, and the section on the landmine plants the rule.

## How to read `|` and `&`: values and fields

Before any composition, install the mental model. `|` and `&` are set operations on types. A type, read set-theoretically, is the set of values it admits — `string` is the set of all strings; `'draft'` is the set with exactly one inhabitant. The two operators combine those sets:

- `A | B` — the <Term definition="The | operator on types. A | B is a value that is either an A or a B. The set of inhabitants is the set-theoretic union — it grows.">union</Term> of inhabitants. A value of `A | B` is a value of `A` *or* a value of `B`. The set of values grows.
- `A & B` — the <Term definition="The & operator on types. A & B is a value that satisfies both A and B. On shape types, the value has the fields of both — the set of fields grows even though the set of values shrinks.">intersection</Term> of constraints. A value of `A & B` satisfies *both* `A` and `B`. On shape types, the value has the fields of *both* — and that means the set of values shrinks (fewer values satisfy two constraints than one), while the set of fields grows.

<UnionIntersectionVenn />

The picture and the prose hold for **primitive** types directly: `string | number` admits every string and every number, so its inhabitant set is genuinely larger than `string` alone. `'red' & 'blue'` admits zero values, because no string is simultaneously `'red'` and `'blue'`.

For **shape** types, the same set picture applies to the *values* — but day-to-day reading focuses on the **fields**. `{ id: string } & { email: string }` has both `id` and `email`, because a value satisfying both shape constraints must carry every field either constraint names. That inversion — "intersection on shapes grows the field set" — is the part that trips beginners. Read it twice: the *value* set shrinks (fewer objects satisfy two shape constraints than one), and at the same time the *field* set grows (the values that do satisfy both have to carry every named field). One operator, two simultaneous facts.

The intersection of types that don't overlap is `never` — the type with no inhabitants. `'red' & 'blue'` is `never` because no string is both literals; `string & number` is `never` because no value is both a string and a number. You'll see this in TypeScript error messages occasionally. The full `never` story — including the exhaustiveness check it powers — lives in later lessons.
:::

## Unions in practice: literals, primitives, shapes, and nullables

A union value is **one of** the alternatives, not all of them. The set of values grows; the set of fields you can read without narrowing is exactly the **intersection of fields** common to every alternative. Four union shapes cover everything you'll write in 2026 SaaS code — walk them in order.

<CodeVariants>
  <CodeVariant label="Literal union">
    ```ts
    type Status = 'draft' | 'sent' | 'paid';

    const next: Status = 'sent';
    ```
    A literal union admits exactly the listed values — three strings here, not every string. The rule from the lesson on primitives, literals, and the four corners applies: finite known domain, literal union, not the primitive. The typo `'pendng'` is a compile error because the value isn't a member of the union.
  </CodeVariant>

  <CodeVariant label="Mixed-primitive">
    ```ts /typeof amount === 'number'/
    const format = (amount: string | number): string => {
      return typeof amount === 'number' ? amount.toFixed(2) : amount;
    };
    ```
    The read-without-narrowing rule fires here. `.toFixed` lives on `number`, not on `string`, so the direct call `amount.toFixed(2)` without the `typeof` check would error. The `typeof` check inside the conditional narrows `amount` to `number` on the true branch and to `string` on the false branch — a preview of the next lesson, which names every narrowing form.
  </CodeVariant>

  <CodeVariant label="Shape union">
    ```ts "value.name" "value.email"
    type User = { id: string; email: string; name: string };
    type Guest = { id: string; name: string };

    const greet = (value: User | Guest): string => {
      return `Hi, ${value.name}`;
      // return value.email;  // error: email is on User, not Guest
    };
    ```
    `value.name` reads cleanly because `name` lives on both variants. `value.email` errors because `Guest` doesn't have it. That's the **shape-union landmine** the next section names as the lesson's load-bearing rule. The fix isn't to widen the type or assert past it — it's to narrow first.
  </CodeVariant>

  <CodeVariant label="Nullable union">
    ```ts "User | null" /u !== null/
    const getUser = (id: string): User | null => {
      // ...
    };

    const u = getUser('usr_01');
    // u.email                            // error: u may be null
    if (u !== null) console.log(u.email); // ok after narrowing
    ```
    A function returning `User | null` says explicitly "this may miss" — the caller can't read `.email` without narrowing past the `null` first. Nullable unions are unions; same operator, same rule. The `?:` field shorthand from the lesson on object shapes and modifiers types the field as `T | undefined`, and the same narrowing applies.
  </CodeVariant>
</CodeVariants>

The pattern across all four tabs is the same rule applied to different shapes:

**On a union, you can read a field — or call a method — only if it exists on every variant. Anything else is a compile error, and the fix is to narrow the value before reading.**

That's the <Term definition="On a union of shapes, only fields common to every variant are readable without narrowing. To read a non-shared field, narrow the value with a runtime check the language tracks.">shape-union access rule</Term>. It's load-bearing for everything that follows in this chapter and the next.

## The shape-union landmine: narrow, don't widen

The rule above explains why one specific bug pattern keeps surfacing. A function takes `value: User | Guest` and reads `value.email`. `Guest` doesn't have `email`. The compiler errors, and the engineer reaches for the wrong fix. Three responses suggest themselves; only one earns the senior reflex.

<CodeVariants>
  <CodeVariant label="Widen the parameter (wrong)">
    <div data-mark-color="red">

    ```ts "{ email?: string; name: string }"
    const greet = (value: { email?: string; name: string }): string => {
      return `Hi, ${value.email ?? value.name}`;
    };
    ```

    </div>
    The first beginner impulse — silence the compiler by widening the type until the read compiles. The error goes away, but the type stops carrying information. The parameter no longer documents what the function accepts; a future call site that hands in a `User` and a future call site that hands in a `Guest` both compile, and so does any other object with a `name` field. The bug isn't fixed — it's hidden.
  </CodeVariant>

  <CodeVariant label="Assert past the type (wrong)">
    <div data-mark-color="red">

    ```ts "value as User"
    const greet = (value: User | Guest): string => {
      return `Hi, ${(value as User).email}`;
    };
    ```

    </div>
    The second beginner impulse — lie to the compiler with `as`. The cast tells TypeScript "trust me, this is a `User`", but the runtime value is unchanged. The moment a real `Guest` flows through, `value.email` is `undefined` and the next line that touches it crashes. Type assertions don't validate at runtime; they only change what the compiler believes. The bug is now invisible until production. The next lesson names the three legitimate triggers for `as`; this isn't one of them.
  </CodeVariant>

  <CodeVariant label="Narrow with a runtime check (right)">
    <div data-mark-color="green">

    ```ts "'email' in value"
    const greet = (value: User | Guest): string => {
      if ('email' in value) {
        return `Hi, ${value.email}`;
      }
      return `Hi, ${value.name}`;
    };
    ```

    </div>
    The senior reflex — narrow with a runtime check the language tracks. The `'email' in value` check fires at runtime (`in` is a real JavaScript operator), and TypeScript tracks the narrowing into the `if` branch: inside the block, `value` is typed as `User`, so `value.email` is readable. The fallthrough branch types `value` as `Guest`. Both branches are type-safe and runtime-safe. The next lesson covers the full narrowing surface (`typeof`, `in`, `instanceof`, `Array.isArray`, discriminant equality, custom predicates); for now, the direction is what matters: widen-or-assert is wrong, narrow is right.
  </CodeVariant>
</CodeVariants>

The chapter's rule, landing in its compositional context:

**Never widen a shape union to read a non-shared field. Never assert past it. Narrow it instead.**

:::tip
The same rule applies to nullable unions. To read `.email` on a `User | null` value, narrow past the `null` first (`if (u !== null)`). The shape-union rule and the nullable-narrow rule are the same rule applied to different unions.
:::

## Intersections on shapes: composing field sets

Unions handle "one of several alternatives." Intersections handle the other half of composition — "a shape with fields from multiple sources." The set-theoretic reading from earlier sets up the inversion that confuses beginners. You read `&` as **and**: `A & B` is "satisfies `A` and `B`." That's correct, but the day-to-day usage on shape types is **field composition** — the resulting type carries every field either constraint named, because a value satisfying both shape constraints must carry all the fields either constraint demands.

```ts
type WithId = { id: string };
type WithEmail = { email: string };

type IdAndEmail = WithId & WithEmail;
// equivalent to: { id: string; email: string }

const u: IdAndEmail = { id: 'usr_01', email: 'a@example.com' };
```

A value of `IdAndEmail` has `id` and `email` simultaneously. A literal missing either field is a compile error at the assignment site — the same posture as `Record<LiteralUnion, V>`'s completeness check from the previous lesson, applied to fields instead of keys.

Three production sites earn the reach for `&` in 2026 SaaS code.

**Request payload composition.** A base type captures the fields every request carries (`requestId`, `timestamp`), and each route's payload composes that base with route-specific fields:

```ts
type BaseRequest = { requestId: string; timestamp: string };

type CreateInvoiceRequest = BaseRequest & {
  customerId: string;
  lines: InvoiceLine[];
};
```

The base type is reusable across every route in the codebase; the route-specific extension stays local to the file that ships it. The intersection composes them at the call site without forcing a class hierarchy or a generic helper.

**Extending a third-party type with project-local fields.** When a library exports a type and you need the same shape plus one or two project-local fields, `LibraryType & { localField: T }` is the senior reach. A different tool — declaration merging via `interface`, named in the lesson on object shapes and modifiers — handles the case where the library's *own* type needs to be augmented everywhere the library is used. Two different jobs: `&` composes at the type-use level, while declaration merging modifies the original declaration module-wide. For a project-local extension that shouldn't leak into every other consumer of the library type, reach for `&`.

**Discriminated union variants.** Each variant of a discriminated union is often a base shape composed with a discriminating extension. `{ status: 'success' } & { data: User }` is equivalent to `{ status: 'success'; data: User }` — the inline form and the `&` form are interchangeable. Writing the `&` form makes the **discriminant** visually distinct from the payload, which is worth its weight when the variants are long. The next section seeds the pattern in full.

:::note
The intersection of incompatible types is `never`. `string & number` is the empty set — no value is both a string and a number — and the compiler types it accordingly. The same goes for any pair of literal types that don't overlap. The full `never` story — especially the exhaustiveness check that uses it to prove every case of a discriminated union is handled — lives in the next lesson and in the next chapter.
:::

## Discriminated unions, seeded

You've seen unions and intersections separately. One pattern combines them — a union of object types where every variant carries a literal-typed field that names which variant it is. The field is the discriminant; narrowing on it separates the variants at compile time. This is the **discriminated union**, and it's the shape most production TypeScript code uses to model "this value is in one of N states."

The pattern earns its weight by replacing a worse design. The bug discriminated unions fix is a payload where every field is optional and a boolean flag tries to signal which fields are present.

<CodeVariants>
  <CodeVariant label="Flag-boolean payload — the bug">
    <div data-mark-color="red">

    ```ts "r.data!.name"
    type FetchResult<T> = {
      isLoading: boolean;
      data?: T;
      error?: Error;
    };

    const render = (r: FetchResult<User>): string => {
      if (r.isLoading) return 'Loading…';
      return r.data!.name;
    };
    ```

    </div>
    The shape a beginner reaches for — three states (loading, success, error) collapsed into one record with optional fields and a boolean flag. The type system can't tell that `data` is guaranteed present when `isLoading` is false and `error` is absent: every field is optional, so every read needs an assertion (`r.data!.name`). The `!` is the smell — it's a runtime assertion that the type system can't verify, and if the data layer ever returns a result where neither `isLoading` nor `data` nor `error` line up the way the renderer assumes, the crash lands in production.
  </CodeVariant>

  <CodeVariant label="Discriminated union — the fix">
    <div data-mark-color="green">

    ```ts "status: 'loading'" "status: 'success'" "status: 'error'"
    type FetchResult<T> =
      | { status: 'loading' }
      | { status: 'success'; data: T }
      | { status: 'error'; error: Error };

    const render = (r: FetchResult<User>): string => {
      if (r.status === 'loading') return 'Loading…';
      if (r.status === 'error') return `Error: ${r.error.message}`;
      return r.data.name;
    };
    ```

    </div>
    Each variant carries a literal-typed `status` field. Inside `if (r.status === 'loading')`, TypeScript narrows `r` to the loading variant — no `data`, no `error`, just the discriminant. Inside `if (r.status === 'error')`, `r.error` is typed as `Error`, present and readable. On the fallthrough, `r` is the success variant and `r.data` is `T`, no assertion needed. The discriminant carries information from the runtime check into the compile-time narrowing, and the `!` from the previous tab disappears entirely.
  </CodeVariant>
</CodeVariants>

The pattern in one sentence:

**A discriminated union is a union of object types where each variant carries a literal-typed field that names which variant it is. The field is the <Term definition="The literal-typed field on every variant of a discriminated union that identifies which variant a value is. Narrowing on the discriminant separates the variants at compile time.">discriminant</Term>; narrowing on it separates the variants at compile time.**

Three forward links worth naming so you know where this lands:

- **Narrowing on the discriminant.** The `if (r.status === 'loading')` motion is equality narrowing — one of the five narrowing forms the next lesson owns. This lesson shows the motion in passing; the next names every form and walks the three legitimate triggers for `as` and `!`.
- **The exhaustiveness check.** When every variant of a discriminated union has been handled, the unhandled branch's type is `never`. Code in that branch becomes a way to prove at compile time that nothing was missed — and adding a new variant to the union later turns every unhandled site into a compile error pointing at the missing case. The full pattern lives in the next chapter.
- **The full discriminated-union surface.** Fetch results, form states, command and event payloads, `Result<T, E>` types for error handling. The next chapter takes the shape you saw above and builds out the patterns most production TypeScript code uses to keep state legible. This lesson plants the shape so the next chapter has language to reference.

:::tip
A discriminant only narrows if it stays a literal type. A value inferred as `string` (via property widening, named briefly in the lesson on primitives and four corners) won't narrow on equality with a specific literal. A later lesson in this chapter covers `as const` — the operator that fixes inline-literal discriminants — and `type` aliases of variants carry the literal type in the declaration, so they narrow correctly out of the box.
:::

## Author a discriminated `Result<T>` shape

Time to write one. The exercise asks you to declare a `Result<T>` type as a discriminated union with two variants — success carrying a value, failure carrying an error — and prove the type system catches the access-without-narrowing read. The `@ts-expect-error` directives validate themselves: they fail the build if the targeted line *doesn't* error, so you can use them as a compile-time check that the union access rule is firing.

<TypeCoding
  instructions="Declare Result<T> as a discriminated union with `ok` as the literal discriminant — `ok: true` carries `value: T`, `ok: false` carries `error: Error`. The two @ts-expect-error directives must trigger (proving the shape-union access rule fires before narrowing), and the ^? query inside the `if (r.ok)` branch must resolve `r.value` to the inner record type."
  starter={`// Declare a discriminated union called Result<T>:
//   - ok: true variant carries 'value: T'
//   - ok: false variant carries 'error: Error'
type Result<T> = never;

declare const r: Result<{ id: string; name: string }>;

// These must fail to type-check (the @ts-expect-error directives verify):

// @ts-expect-error — cannot read .value without narrowing
r.value;

// @ts-expect-error — cannot read .error without narrowing
r.error;

// This must succeed once the type is correct:
if (r.ok) {
  const x = r.value;
  //    ^?
}
`}
  expectedQueries={[
    { line: 19, contains: 'name: string' },
  ]}
/>

When the exercise grades green, you've felt three things in your fingers. You wrote a discriminated union. You proved the type system blocks the unnarrowed read on `.value` and `.error`. And you watched the narrowed branch resolve `r.value` to the inner type. Those three motions — author, block, narrow — are the rhythm the next chapter turns into a production pattern.

## Decide which operator to reach for

Six composition scenarios, two operators. Sort each one by whether the value is *one of* several shapes (`|`) or has fields of *both* (`&`). Some are obvious; one is subtle. The subtle case is the lesson's bridge to the next chapter.

  <Item bucket="intersection">A third-party `Session` extended with a project-local `tenantId: string` field</Item>
  <Item bucket="union">A `Status` field that's `'draft' | 'sent' | 'paid'`</Item>

The exercise's subtle case is item 4 — the discriminated `Result<T>`. The outer type is a **union** of two variants, so the sort answer is `|`. But each variant is *internally* a shape that could be written as an intersection (`{ ok: true } & { value: T }` reads the same as `{ ok: true; value: T }`). Both operators in one type; both senior reflexes in one shape. That's the bridge to the next chapter, where discriminated unions become the central pattern.

The senior reflex in one move:

- **`|` for alternatives** — the value is one of several shapes. Literal unions, mixed-primitive unions, shape unions, nullable unions, the outer shape of a discriminated union.
- **`&` for field composition** — the shape has the fields of both. Payload composition, third-party type extension, the inside of a discriminated union's variants.

Read every composed type as a set first. `User | null` is "either a `User` or `null`." `BaseRequest & { token: string }` is "a value that has every field `BaseRequest` declares plus a `token`." The operator is the punctuation; the set-theoretic reading is the language.
