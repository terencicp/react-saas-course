The last two lessons taught the two object shapes you author from declared field names — `type User = { id: string; email: string }` and the positional tuple. This lesson installs the third. Three quick scenarios to anchor what "dynamic keys" actually means in production code:

```ts
const userCache = {
  'usr_01j8aa': { id: 'usr_01j8aa', email: 'a@example.com' },
  'usr_01j8ab': { id: 'usr_01j8ab', email: 'b@example.com' },
};

const statusLabels = {
  draft: 'Draft',
  sent: 'Sent',
  paid: 'Paid',
};
```

A cache keyed by user ID. A lookup from status to display label. And — not shown above but the third shape this lesson covers — a JSON payload parsed from the wire, where the keys are whatever the third party decided to send. All three are objects whose keys aren't declared up front like `id` or `email` were in the previous lessons. The keys **come from data**: from the IDs your database minted, from a finite set of statuses your domain agreed on, from the shape a webhook chose to ship.

Three shapes, three correct types — but which type for which shape? Reaching for the same form for all three is the vocabulary failure this lesson prevents. Type the status lookup too loosely and a missing key ships, then surfaces as a broken label three components away. Type the cache too tightly — listing every user ID at design time — and the type stops compiling the moment a new user signs up. Two forms exist for dynamic keys: the **index signature** (`{ [key: string]: V }`) and the **`Record<K, V>` utility type**. By the end of this lesson you'll have one rule that picks between them, plus the strict-mode read-site behavior that makes the finite-key case the senior reach.

## The two forms: index signature and `Record<K, V>`

The bare syntax first. An <Term definition="An object-type member that declares the type of every dynamic key. Syntax: { [keyName: KeyType]: ValueType }. The keyName slot is pure documentation surfaced in tooltips; the constraint is the KeyType.">index signature</Term> is an object-type member that says "every key of this type maps to a value of this type." You write it as `{ [userId: string]: User }`. The `userId` part is a label — the same all-or-nothing tooling rule from the previous lesson's tuple labels: pure documentation, surfaced in editor tooltips. The `string` is the **key constraint**. The `User` is the value type.

The <Term definition="Utility type for objects whose keys are of type K and values of type V. With K = string, equivalent to an index signature; with K a literal union, requires every key to be present.">`Record<K, V>`</Term> utility type is the same idea wrapped in a generic: `Record<string, User>`. TypeScript ships it built-in; you don't import anything.

<CodeVariants>
  <CodeVariant label="Index signature — { [userId: string]: User }">

    ```ts
    type User = { id: string; email: string };

    type UserCache = { [userId: string]: User };

    const userCache: UserCache = {
      'usr_01j8aa': { id: 'usr_01j8aa', email: 'a@example.com' },
      'usr_01j8ab': { id: 'usr_01j8ab', email: 'b@example.com' },
    };
    ```

    This is the form you'll meet in legacy types and in mixed shapes where named fields sit next to the dynamic surface (covered later in this lesson). The bracket syntax reads "for any key of type `string`, the value is a `User`."
  </CodeVariant>

  <CodeVariant label="Record — Record<string, User>">

    ```ts
    type User = { id: string; email: string };

    type UserCache = Record<string, User>;

    const userCache: UserCache = {
      'usr_01j8aa': { id: 'usr_01j8aa', email: 'a@example.com' },
      'usr_01j8ab': { id: 'usr_01j8ab', email: 'b@example.com' },
    };
    ```

    Identical shape at the type-system level. `Record<string, User>` reads "object whose keys are strings, whose values are `User`" without the bracket-syntax noise. The course prefers `Record<string, V>` for legibility and reaches for the index-signature form only when the mixed-shape case demands it.
  </CodeVariant>
</CodeVariants>

The two forms are **interchangeable** at the type-system level when the key constraint is `string`. Any assignment that compiles against one compiles against the other; either gets the same hover, the same autocomplete, the same errors. The senior call between them is pure legibility — `Record<string, V>` reads first, so prefer it unless the index-signature syntax is the only one that fits the shape (the mixed-fields case later in this lesson).

:::note
`Record<K, V>` generalizes — the first parameter can be any <Term definition="The built-in type for any valid object key: string | number | symbol. Record<PropertyKey, V> is the broadest dynamic-keyed shape.">`PropertyKey`</Term>: `string`, `number`, or `symbol`. `Record<number, V>` is rare in SaaS code (arrays handle nearly every numeric-indexed case); `Record<symbol, V>` is rarer still. This lesson uses `string` keys throughout, and the literal-union form in the next section.
:::

## `Record<LiteralUnion, V>` and the completeness payoff

Now the second of the three shapes from the intro: the status-to-label lookup.

```ts
const statusLabels = {
  draft: 'Draft',
  sent: 'Sent',
  paid: 'Paid',
};
```

The keys here are not open. There are three of them and only three — `'draft'`, `'sent'`, `'paid'` — the same literal union you used to type an invoice's `status` field in the first lesson of this chapter. If you type this with `Record<string, string>`, the type system can't tell whether `'paid'` is present, because every string is a valid key. If you type it with `Record<'draft' | 'sent' | 'paid', string>`, the missing-key error fires at the literal site — the same place excess-property checks fire when an unexpected field appears.

Walk through the three states.

<AnnotatedCode lang="ts" code={`
type Status = 'draft' | 'sent' | 'paid';

const labelsLoose: Record<string, string> = {
  draft: 'Draft',
  sent: 'Sent',
  // 'paid' missing — no error
};

// @ts-expect-error — Property 'paid' is missing
const labelsStrict: Record<Status, string> = {
  draft: 'Draft',
  sent: 'Sent',
};

const labels: Record<Status, string> = {
  draft: 'Draft',
  sent: 'Sent',
  paid: 'Paid',
};
`}>
  <AnnotatedStep meta="{3-7}" color="red">
    **`Record<string, string>` — accepts anything, catches nothing.** The key constraint is `string`, so any object whose keys are strings and values are strings passes. The `'paid'` key is missing and TypeScript stays quiet. The bug ships: the consumer reads `labelsLoose.paid`, gets `undefined`, and the UI displays the raw status code or — worse — crashes inside a string method called on `undefined`. This is what the rest of the lesson prevents.
  </AnnotatedStep>

  <AnnotatedStep meta="{9-13}" color="orange">
    **`Record<Status, string>` — every member of the union must appear.** The literal-union key forces completeness at the literal site. The `@ts-expect-error` on the line above the declaration acknowledges the missing `'paid'` — the compiler error fires at the assignment site, in the file the value is defined in, not three components away when something tries to read `labels.paid`. This is the **completeness check** the literal-union form earns.
  </AnnotatedStep>

  <AnnotatedStep meta="{15-19}" color="green">
    **`Record<Status, string>`, complete.** All three keys present, no error. Shipping shape. And the payoff continues past the literal site: the day a teammate adds `'overdue'` to the `Status` union, every `Record<Status, V>` initializer in the codebase becomes a compile error pointing at the missing key. That's a feature — the type system finds every site you need to update, no grep required.
  </AnnotatedStep>
</AnnotatedCode>

The rule the walkthrough installs: **if the keys are finite and known at design time, type the object as `Record<LiteralUnion, V>`.** Same posture as the literal-union rule for primitives — the union is what turns a generic shape into a checked one. The completeness check at the literal site is the first half of the payoff. The second half — what happens at the read site — is the next section, and it's the bigger one.

:::tip
Treat the compile errors a Status union grows as a feature, not a chore. Each one is the type system pointing at a site that needs updating; the alternative is a missing key that compiles fine and breaks at runtime in production.
:::

## The `noUncheckedIndexedAccess` divergence

The course's tsconfig ships with `noUncheckedIndexedAccess: true` — the same flag introduced in the first lesson of the previous chapter, and one piece of the full strict-mode build-out the course revisits in chapter 024. That flag changes how index reads are typed. Under the flag, **any read through an index signature returns `V | undefined`** — TypeScript no longer assumes the key is present just because the type says every key maps to a `V`. The `Record<LiteralUnion, V>` form is the exception: every key in the union is guaranteed present, so the read returns `V` directly.

That divergence is the central payoff. Two tabs make it visible at the type level.

<CodeVariants>
  <CodeVariant label="Open keys — User | undefined at the read site">

    ```ts
    type User = { id: string; email: string };

    const userCache: Record<string, User> = {};

    const u = userCache['usr_01j8aa'];
    //    ^? User | undefined
    ```

    Under `noUncheckedIndexedAccess`, the read returns `User | undefined` regardless of whether the key looks "obvious" — the type system is being honest that an open-keyed object has no way to prove the key is present. Before you use `u`, you narrow: `if (u) { ... }`. The full narrowing surface lands two lessons from here; the dynamic-keyed-object subset is at the end of this lesson.
  </CodeVariant>

  <CodeVariant label="Finite keys — string at the read site">

    ```ts
    type Status = 'draft' | 'sent' | 'paid';

    const labels: Record<Status, string> = {
      draft: 'Draft',
      sent: 'Sent',
      paid: 'Paid',
    };

    const status: Status = 'draft';
    const label = labels[status];
    //    ^? string
    ```

    The read returns `string` directly — no `| undefined`. Every key in the `Status` union is guaranteed present (the completeness check from the previous section earned that guarantee), so the type system doesn't widen at the read site. No narrowing required: the value is ready to use the moment you read it.
  </CodeVariant>
</CodeVariants>

Read the two `^?` annotations side by side — `User | undefined` versus `string` — and the senior call writes itself: **use `Record<LiteralUnion, V>` when the keys are finite and known; use `Record<string, V>` (or the index signature) when the keys are open.** The two forms are interchangeable at the type-system level when `K = string`; the divergence appears when `K` is a literal union, and the divergence is what earns the senior reach.

:::caution
The `Record<LiteralUnion, V>` read-site guarantee depends on the key variable being typed as the union, not as `string`. `labels[someString]` reverts to `string | undefined` because the type system can't prove `someString` is actually one of `'draft' | 'sent' | 'paid'`. Type your key variable as the union, or narrow it before the lookup.
:::

## Mixed shapes: known fields plus a dynamic surface

Occasionally a type has both **named fields** (the previous-lessons territory) and a **dynamic surface** (this lesson's territory) on the same object. A serialized configuration with a `name` field plus arbitrary metadata. An event payload with a `type` discriminant plus whatever metadata the producer chose to attach. The index-signature syntax is the form that admits the combination; `Record<K, V>` doesn't have a slot for named fields, so the mixed shape lives in the index-signature form.

<CodeTooltips tooltips={{
  '[key: string]: string | number': 'The index signature covers every key not listed above it — retries and region here, and any other string-keyed value the caller chooses to attach. Its value type (string | number) is the constraint every dynamic value must satisfy.',
  'name: string': 'Every named field type must be assignable to the index signature value type. name: string is fine because string is part of string | number. A name: boolean would error at this line — TypeScript enforces that the dynamic surface covers the named fields too.',
}}>
  ```ts
  type Metadata = {
    name: string;
    createdAt: string;
    [key: string]: string | number;
  };

  const m: Metadata = {
    name: 'project-alpha',
    createdAt: '2026-05-26T10:00:00Z',
    retries: 3,
    region: 'us-east-1',
  };
  ```
</CodeTooltips>

That last point is the rule worth remembering: **the index signature's value type is a ceiling for every named field on the same object.** If the index signature is `string | number` and a named field is `boolean`, TypeScript errors at the named field's declaration — the dynamic-surface constraint reaches into the named-field section of the same type.

:::note
Mixed shapes are rare in 2026 SaaS code. The senior reach is usually to split the type into a named-field section and a `Record`-typed metadata field: `{ name: string; metadata: Record<string, string | number> }`. That form keeps the named fields free of the index-signature constraint and reads more honestly — the `metadata` field is explicitly the dynamic part. Recognize the mixed form when you meet it in library types; reach for the split form in your own code.
:::

## Reading dynamic-keyed objects: existence checks

Every read through an index signature under `noUncheckedIndexedAccess` returns `V | undefined`. Before you use the value, you narrow. Two narrowing forms cover the dynamic-keyed cases; the full narrowing surface — `typeof`, `instanceof`, discriminant equality, type predicates — lands two lessons from here.

```ts
type User = { id: string; email: string };

const cache: Record<string, User> = {};
const key = 'usr_01j8aa';

const maybeUser = cache[key];
if (maybeUser !== undefined) {
  // maybeUser is User here — narrowed by the explicit undefined check
}

if (key in cache) {
  // The `in` check confirms the key exists at runtime,
  // but TypeScript still types cache[key] as User | undefined here.
  const u = cache[key];
}
```

The **`!== undefined` check on the read** is the narrowing reflex for an index-signature object. Pull the read into a `const`, check it against `undefined`, and the `const` is typed as `V` inside the branch. This is the form to reach for whenever you need the value typed without the `| undefined`.

The **`in` operator** is the right runtime check for "does this key exist on the object" — it's what JavaScript ships for the question. It does *not* propagate as a type narrowing across an index signature, though: inside `if (key in cache)`, a subsequent `cache[key]` read still types as `V | undefined`. Use `in` when you want to test existence (for example, before writing to the key); use the `!== undefined` form on a captured read when you want the narrowed type.

:::note
The full narrowing surface — `typeof`, `instanceof`, `Array.isArray`, discriminant equality, custom type predicates — lives two lessons from here. The two forms shown above are the dynamic-keyed-object subset; they're enough to read and use a `Record<string, V>` correctly today.
:::

## Decide which form to reach for

You've seen both forms, the completeness payoff, and the read-site divergence. The exercise installs the call as a sort. For each shape, decide whether the keys are **open** (no design-time enumeration — IDs, UUIDs, anything from the wire) or **finite** (every member known when you write the code — statuses, methods, locales). The bucket choice follows from the answer.

<Buckets twoCol instructions="Sort each shape by whether the keys are finite and known at design time, or open and minted at runtime.">
  <Bucket name="open" label="Record<string, V> or index signature" description="Open keys — minted at runtime" />
  <Bucket name="finite" label="Record<LiteralUnion, V>" description="Finite keys — every member known at design time" />

  <Item bucket="open">Cache from user ID to user record</Item>
  <Item bucket="finite">Status (`'draft' | 'sent' | 'paid'`) to display label</Item>
  <Item bucket="open">Arbitrary JSON parsed from a third-party webhook</Item>
  <Item bucket="finite">HTTP method (`'GET' | 'POST' | 'PUT' | 'DELETE'`) to handler</Item>
  <Item bucket="finite">i18n messages keyed by locale (`'en' | 'es' | 'fr'`)</Item>
  <Item bucket="open">Drizzle row lookup table keyed by primary-key UUID</Item>
</Buckets>

The two subtler items: the webhook payload is `Record<string, unknown>` — open keys, and the values are also unknown until you parse them (the `unknown` reach from the lesson on primitives, literals, and the four corners). The Drizzle lookup table is `Record<string, Row>` — the keys are open (UUIDs are minted at insert time), but the value is the typed row from `$inferSelect`, the thing the database hands you. Both are open at the key level; the divergence is at the value level, not at the form-choice level.

The senior reflex in one move: **finite keys, `Record<LiteralUnion, V>`** — compile-time completeness, safe reads, refactor errors when the union grows. **Open keys, `Record<string, V>` or the index signature** — narrow the read with `in` or `!== undefined` before using the value. One rule, two cases.
