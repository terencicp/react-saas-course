Both bugs have the same shape. The engineer reached for the widest type the language admits — `string` for a field that's actually one of three values, `any` for a payload that should have been treated as data from outside the type system. The compiler would have caught both, if the type vocabulary had been right.

This lesson is that vocabulary. By the end you'll reach for a literal union when the runtime values are finite and known, reach for `unknown` instead of `any` at every value-from-outside boundary, recognize the one trigger that earns `never`, and the one trigger that earns `void`. The primitives themselves are review — you've written values in each shape across the three chapters that came before this one. The new moves are the literal-narrowing layer on top, and the four corner types that sit outside the usual primitive surface.

## The seven primitives, anchored to typeof

TypeScript's primitive types share their names with the JavaScript primitives you already know — except for `null`, which `typeof` reports as `'object'` for legacy reasons named back in the JavaScript value-model chapter.

```ts
const name = 'Lina';                  // string,    typeof 'string'
const amount = 4900;                  // number,    typeof 'number'
const isPaid = true;                  // boolean,   typeof 'boolean'
const tenantId = 9007199254740993n;   // bigint,    typeof 'bigint'
const slot = Symbol('slot');          // symbol,    typeof 'symbol'
const next = null;                    // null,      typeof 'object' (legacy)
const missing = undefined;            // undefined, typeof 'undefined'
```

Those seven names are the full primitive surface of the type system. Everything else you'll type in this chapter either narrows one of them (literal types, next section), composes them (object shapes, tuples, and unions — lessons later in this chapter), or sits at the four corners (later this lesson). No grand tour; you already know what each primitive is for.

:::note
`bigint` earns its weight only at the IEEE-754 boundary — the value-model chapter named the threshold (values beyond `Number.MAX_SAFE_INTEGER`). The course writes `bigint` at database row IDs and high-precision arithmetic seams, and that's it. `symbol` is rarer still and you won't reach for it in application code; recognize it on the list, move on.
:::

## Literal types: a primitive narrowed to one value

A literal type is a primitive type narrowed to *exactly one value*. `'pending'` is a type whose only <Term definition="The set of runtime values that satisfy a type. `string` has infinite inhabitants (every possible string); `'pending'` has exactly one (the string `'pending'`); `never` has zero.">inhabitant</Term> is the string `'pending'`. `42` is a type whose only inhabitant is the number `42`. `true` is a type whose only inhabitant is `true`. At runtime, a literal-typed value looks identical to a primitive-typed one; the difference lives entirely at compile time.

<CodeTooltips tooltips={{
  ": 'pending'": "The type is exactly 'pending', not string. The only value assignable here is the string 'pending'.",
  ": 3": "The type is exactly 3, not number. No other numeric value is assignable.",
  ": true": "The type is exactly true, not boolean. The literal false would be a type error.",
}}>

```ts
const status: 'pending' = 'pending';
const retries: 3 = 3;
const isPaid: true = true;
```

</CodeTooltips>

A literal type used alone is rarely useful — a variable that can only ever hold `'pending'` is just a `const` with extra ceremony. Literal types earn their weight when you compose several of them into a union.

## Literal unions: the senior reach for finite domains

Here's the rule, plainly: **if the runtime values are finite and known at design time, the type is a literal union, not the primitive.** An invoice status is one of `'draft' | 'sent' | 'paid'`, not `string`. An HTTP method is one of `'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE'`, not `string`. A side of an order is `'buy' | 'sell'`, not `string`. Every time you write `string` for a field whose set of values you could enumerate in five seconds, you've handed the compiler less information than it needs to catch your typos.

Watch the first bug get fixed.

<CodeVariants>
  <CodeVariant label="Before — status: string">
    <div data-mark-color="red">

    ```ts "string" "'pendng'"
    const setStatus = (status: string) => {
      invoice.status = status;
    };

    setStatus('pendng');
    ```

    </div>
    The annotation says any string is valid. `'pendng'` is a string. The call compiles, the code ships, the downstream `=== 'paid'` check silently fails forever.
  </CodeVariant>

  <CodeVariant label="After — status: 'draft' | 'sent' | 'paid'">
    <div data-mark-color="green">

    ```ts "'draft' | 'sent' | 'paid'" {5-7}
    const setStatus = (status: 'draft' | 'sent' | 'paid') => {
      invoice.status = status;
    };

    setStatus('pendng');
    // Error: Argument of type '"pendng"' is not assignable to
    // parameter of type '"draft" | "sent" | "paid"'.
    ```

    </div>
    The annotation names the closed set. The typo is now a compile error pointing at the literal — no review, no shipped bug, no late-night incident.
  </CodeVariant>
</CodeVariants>

The payoff goes further than catching the typo. Every site that consumes the value gets autocomplete for the three members — your editor offers `'draft'`, `'sent'`, `'paid'` and refuses to suggest anything else. Every site that produces a value gets a compile error for any string outside the set. And when you rename `'paid'` to `'settled'` six months from now, every call site that still uses the old literal lights up red — refactoring a closed domain becomes a typed find-and-replace instead of a grep-and-pray.

Do this once before reading on. The reflex doesn't install until you've made the change yourself.

## Where literal types come from

Once you start reaching for literals, you'll hit the surprise that bites every junior the first month: `const config = { status: 'draft' }` doesn't preserve the literal `'draft'` — TypeScript infers `{ status: string }`. The reason is worth knowing now, even though the fix lives a few lessons away.

<AnnotatedCode lang="ts" code={`
const status: 'draft' = 'draft';
const direct = 'draft';
const tuple = ['draft', 'sent'] as const;
const config = { status: 'draft' };
`}>
  <AnnotatedStep meta="{1}" color="green">
    **Written annotation.** The annotation itself names the literal type. The variable's type is exactly `'draft'`, no inference involved. The most explicit form, and the one you reach for in function parameters and exported types.
  </AnnotatedStep>

  <AnnotatedStep meta="{2}" color="green">
    **`const` binding on a primitive.** A `const` bound directly to a primitive literal infers the literal type — `direct` is `'draft'`, not `string`. (Swap `const` for `let` and the type widens to `string`, because the variable could be reassigned to any string.) This is the `const`-binds-the-name rule from the value-model chapter doing useful work.
  </AnnotatedStep>

  <AnnotatedStep meta="{3}" color="green">
    **`as const`.** The value-site freeze, lifting an entire literal into its narrowest type — here a `readonly ['draft', 'sent']` tuple of literals, not a `string[]`. Full treatment lands later in this chapter; you only need to know the form exists.
  </AnnotatedStep>

  <AnnotatedStep meta="{4}" color="orange">
    **The non-source.** Object properties *widen* — `config.status` is `string`, not `'draft'`. Every literal you put inside an object literal gets widened at inference to its base primitive, because TypeScript assumes the property might be reassigned. This is the surprise that bites; the fix is `as const` or `satisfies`, both later in this chapter.
  </AnnotatedStep>
</AnnotatedCode>

The technical name for what's happening on the last line is <Term definition="TypeScript broadens an inferred literal to its base primitive when the value lives somewhere it could be reassigned — most notably object properties and `let` bindings. `'draft'` widens to `string`; `3` widens to `number`; `true` widens to `boolean`. The fix is `as const` (freeze the inferred type) or a written annotation (state the type yourself).">widening</Term>. You don't need to fight it yet — you need to recognize it. When you write a config object and the autocomplete shows `string` where you expected the literal, that's widening, not a bug.

## The four corners

Four types sit outside the usual primitive surface, at the edges of the type system. Each has exactly one trigger that earns it. You'll leave this section with one sentence per corner, not a tour.

<FourCornersLattice />

### `any` — the unsound escape

`any` disables type checking on the value and propagates outward. Reading `.email` on an `any` payload compiles; calling it as a function compiles; passing it as a `number` argument compiles. The compiler stops looking. That's why the second opening bug shipped — `payload.email.toLowerCase()` compiled cleanly, even though the runtime value was `null`.

The course never writes `any`. If a third-party library forces an `any` into your code, you accept the value as `unknown` and narrow it before doing anything with it.

<CodeVariants>
  <CodeVariant label="Unsafe — payload: any">
    <div data-mark-color="red">

    ```ts "any" "payload.email.toLowerCase()"
    const welcome = (payload: any) => {
      const email = payload.email.toLowerCase();
      return `Welcome, ${email}`;
    };

    welcome({ email: null });
    ```

    </div>
    `any` silences the type checker on every read. The compiler is fine. The runtime crashes the moment `email` is `null`.
  </CodeVariant>

  <CodeVariant label="Safe — payload: unknown">
    <div data-mark-color="green">

    ```ts "unknown" {2-3}
    const welcome = (payload: unknown) => {
      const email = payload.email.toLowerCase();
      // Error: 'payload' is of type 'unknown'.
      return `Welcome, ${email}`;
    };
    ```

    </div>
    `unknown` accepts every value too, but refuses to let you read off it without narrowing first. The compile error points at the exact bug the runtime would hit. The narrowing lives in the next subsection.
  </CodeVariant>
</CodeVariants>

Biome — the linter the course pins in the strict baseline a few chapters from now — has a `noExplicitAny` rule that catches `any` annotations automatically. You won't need to police them by hand once the toolchain is in place.

### `unknown` — the sound top

`unknown` is the right type for any value that arrived from outside the type system. JSON parsed from the wire. A read from `localStorage`. The bound parameter of a `catch` clause. The return of a third-party SDK that ships without types. Anywhere a value crosses into your code from a place TypeScript can't see, `unknown` is the type that arrives.

The trigger is the friction: you cannot read or call an `unknown` value without narrowing it first. That's the feature. Every assumption about the value's shape has to become code the compiler can see.

```ts
const readEmail = (payload: unknown): string | null => {
  if (typeof payload === 'object' && payload !== null && 'email' in payload) {
    const email = payload.email;
    return typeof email === 'string' ? email : null;
  }
  return null;
};
```

Three checks before the read: the value is an object, it isn't `null` (because `typeof null === 'object'`, the legacy bug from the JavaScript value-model chapter), the property `'email'` exists on it. Then the read produces an `unknown`, so another `typeof` check pins it to `string` before returning. Every shape assumption is visible in the code; an unexpected wire shape can't smuggle a runtime crash through.

The narrowing surface gets its own lesson later in this chapter. For now, recognize the posture: the verbosity is the contract. In production, a Zod schema parse at the wire boundary (when Zod lands a few chapters from now) replaces the hand-rolled narrowing with a structured one — but the posture is identical. Every Zod schema is a more legible version of exactly this dance.

### `never` — the bottom

`never` is the type with no inhabitants. No runtime value satisfies it. It shows up in two places: as the return type of a function that always throws (it never returns a value), and as the type of a variable in a branch of code that the language has fully narrowed away.

```ts
const fail = (message: string): never => {
  throw new Error(message);
};

type Status = 'draft' | 'sent' | 'paid';

const statusLabel = (status: Status): string => {
  if (status === 'draft') return 'Draft';
  if (status === 'sent') return 'Sent';
  if (status === 'paid') return 'Paid';
  return status;
};
```

By the time control reaches the last `return status` line, every member of the union has been handled. TypeScript narrows `status` to `never` there — the only inhabitants of `Status` were the three literals, and they're all accounted for. The fact that `status` has type `never` at that line is what the exhaustiveness pattern in the next chapter will leverage to turn "I forgot to handle a case" into a compile error. You don't need to write that pattern yet; you need to know that `never` is what makes it work.

### `void` — function-return-only

`void` means "the caller should not rely on the return value of this function." It's a marker, not a value. And it's distinct from `undefined` in a way that bites if you confuse them: `undefined` is a value the caller can read; `void` is a contract that says *whatever the function returns, ignore it*.

The canonical site is a callback whose return the framework throws away.

```ts
const on = (event: string, handler: () => void): void => {
  // the framework discards the handler's return value
};

on('click', () => 42);
```

The callback typed as `() => void` returns `42`, and that's fine. The contract says the caller will discard the return regardless, so the callback is free to return anything it wants. You'll meet this pattern at every event handler in the browser, at every `Array.prototype.forEach` callback, at every fire-and-forget custom hook callback.

One trap to avoid: don't write `Promise<void>` to mean "an async function that resolves with no payload." That's what `Promise<undefined>` is for, or — far more commonly — just letting the inferred type carry it through. `void` on a `Promise` means "discard whatever this promise resolves to," which is rarely what you mean.

## Branded primitives and the enum non-reach

Two moves the lesson is explicitly *not* making, named so you know the gap exists.

**Branded primitives.** A `string` that represents a user ID and a `string` that represents an org ID are the same type to the compiler — `getUser(orgId)` compiles, and the bug is the kind that costs a customer's data. The fix is a brand (using TypeScript's `unique symbol` form) that makes the two strings distinct types even though they're identical at runtime. That pattern earns a full lesson in the next chapter; for now, recognize the gap and know there's a name for the move.

**The `enum` we don't write.** TypeScript ships an `enum` keyword. The course doesn't use it. Literal unions cover every case more cleanly — they integrate with JSON (an `enum` member's value isn't its name, which causes friction at the wire), they don't emit runtime code (an `enum` compiles to a runtime object that ships in your bundle), and they read as values where `enum` members read as references. You'll meet `enum` in legacy codebases. Recognize the form, write a literal union when you author fresh code.
