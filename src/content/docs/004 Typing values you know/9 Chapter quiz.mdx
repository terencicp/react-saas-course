  <MultipleChoice src="4.1">
    A webhook handler receives a payload whose shape isn't known at compile time. Two engineers reach for different annotations:

    ```ts
    const handleA = (payload: any) => payload.user.email;
    const handleB = (payload: unknown) => payload.user.email;
    ```

    Which best describes the operational difference between the two?

    <McqChoice>They are equivalent — both compile, and only `handleB` errors at runtime.</McqChoice>
    <McqChoice correct>`handleA` compiles and may crash at runtime; `handleB` fails at compile time until the value is narrowed.</McqChoice>
    <McqChoice>`handleB` compiles but disables type-checking on `payload`; `handleA` requires narrowing before any read.</McqChoice>

    <McqWhy>`any` disables type-checking on every read off the value, so `handleA` compiles and ships — and crashes the moment the payload's shape doesn't match. `unknown` is the sound top: it accepts every value but refuses every read until the code narrows the shape the compiler can see. That refusal is the feature.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="4.2">
    The course defaults to `type` for every alias. Which scenario is the one trigger that earns a reach for `interface` instead?

    <McqChoice>Composing a request payload from a base shape and a route-specific extension.</McqChoice>
    <McqChoice correct>Augmenting a third-party type (for example Better Auth's `Session`) inside a `declare module` block.</McqChoice>
    <McqChoice>Declaring an object shape whose fields might later need optional or `readonly` modifiers.</McqChoice>

    <McqWhy>Declaration merging is the one job `type` can't do — two `type` declarations with the same name are a duplicate-identifier error. The canonical use is augmenting a type a package shipped via `declare module` plus an `interface`. Composition uses `&`; modifiers work identically on either form.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="4.3">
    `useState` returns `[value, setter]` instead of `{ value, setValue }`. What's the senior reason the tuple shape wins here?

    <McqChoice>A tuple is faster at runtime than a named-field object.</McqChoice>
    <McqChoice correct>The caller destructures-and-renames on every use, so the positional shape carries no naming penalty — each call site picks the names it wants.</McqChoice>
    <McqChoice>TypeScript can't infer object property types from generic hooks, so a tuple is the only shape that works.</McqChoice>

    <McqWhy>The tuple earns its weight precisely when the caller will destructure-and-rename on every use site. Two callers in the same file can write `[isOpen, toggleOpen]` and `[isHovered, toggleHover]` from the same hook without collision. Past two slots, names win and the object is the senior call.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="4.4">
    You're typing a lookup table from invoice status to display label. The status is `'draft' | 'sent' | 'paid'`. Under `noUncheckedIndexedAccess`, which type catches a missing key at the literal site **and** reads back as `string` (not `string | undefined`)?

    <McqChoice>`Record<string, string>`</McqChoice>
    <McqChoice>`{ [status: string]: string }`</McqChoice>
    <McqChoice correct>`Record<'draft' | 'sent' | 'paid', string>`</McqChoice>

    <McqWhy>The literal-union form earns two payoffs at once. Completeness fires at the literal site if any member is missing, and reads return `V` directly (no `| undefined`) because every key in the union is guaranteed present. The `Record<string, ...>` and index-signature forms both admit any string and force a `| undefined` at every read.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="4.5">
    A function takes `value: User | Guest`. `User` has an `email` field; `Guest` doesn't. You need to read `email` when it exists. Which moves are correct senior reflexes? **Select all that apply.**

    <McqChoice correct>Narrow with `if ('email' in value)` before the read.</McqChoice>
    <McqChoice>Widen the parameter to `{ email?: string; name: string }` so the read compiles.</McqChoice>
    <McqChoice>Write `(value as User).email` — you know the call sites only pass users with emails.</McqChoice>
    <McqChoice correct>Add a literal-typed discriminant to each variant and switch on it.</McqChoice>

    <McqWhy>The shape-union access rule: only fields common to every variant are readable without narrowing. Narrow with a runtime check the language tracks (`in`, or equality on a discriminant). Widening hides the bug; asserting lies to the compiler — the runtime value is unchanged, and a real `Guest` will crash the next line that touches `email`.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="4.6">
    Which of the following is **not** one of the three legitimate triggers for reaching for `as`?

    <McqChoice>A `userSchema.parse(raw)` boundary where the parser validates `unknown` and returns a typed value.</McqChoice>
    <McqChoice>A `document.querySelector('button[type="submit"]') as HTMLButtonElement` in a tightly-scoped one-shot.</McqChoice>
    <McqChoice correct>Casting a `User | Guest` parameter to `User` because most call sites pass users.</McqChoice>
    <McqChoice>A `cache.get(id) as User` immediately after `cache.set(id, user)`, where the type system can't track the round-trip.</McqChoice>

    <McqWhy>The three legitimate triggers are boundary parse-then-trust, the proof gap TypeScript can't see, and the DOM/third-party type seam. Asserting past a union because "most callers" send a `User` is the opening bug of the lesson — the runtime value is unchanged, and the first real `Guest` will crash the next line.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="4.7">
    Consider the four scenarios below. Which calls for the combined `as const satisfies T` idiom?

    <McqChoice>The signature of an exported function — `(input: { email: string }) => void`.</McqChoice>
    <McqChoice correct>A module-level routes map whose keys must cover every `RouteName` and whose values must remain literal paths so `(typeof ROUTES)[keyof typeof ROUTES]` narrows.</McqChoice>
    <McqChoice>A local variable holding the result of a `reduce` over a list of numbers.</McqChoice>

    <McqWhy>Typed configs are the canonical site for the combined idiom. `as const` keeps the literal paths so `(typeof ROUTES)[keyof typeof ROUTES]` reads the narrow union; `satisfies Record<RouteName, string>` checks completeness without re-widening. An annotation on the same value would erase the literals; `as const` alone would skip the contract check.</McqWhy>
  </MultipleChoice>

  <MultipleChoice src="4.8">
    Which line is the wrong reach under the boundary rule and `verbatimModuleSyntax`?

    <McqChoice>
      ```ts
      const total = items.reduce((acc, item) => acc + item.price, 0);
      ```
    </McqChoice>
    <McqChoice>
      ```ts
      export const createInvoice = async (
        input: CreateInvoiceInput,
      ): Promise<Result<Invoice>> => { /* ... */ };
      ```
    </McqChoice>
    <McqChoice correct>
      ```ts
      import { User } from './user';

      const greet = (user: User) => `Hello, ${user.name}`;
      ```
    </McqChoice>

    <McqWhy>Under `verbatimModuleSyntax`, a value import for a name used only as a type is a compile error — write `import type { User }` so the compiler erases the import and the bundler never sees it. The `reduce` is the canonical "infer the inside" case; the exported async function annotates parameters and return type at the boundary, exactly where the rule wants them.</McqWhy>
  </MultipleChoice>
