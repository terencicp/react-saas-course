You've been consuming a tuple every day, probably without naming the shape. Open any React component you've seen so far and you'll find a line like this one:

```ts
const [count, setCount] = useState(0);
```

That return is a tuple. Two slots, fixed length, different type per position — a `number` at index 0, a setter function at index 1. The caller destructures it and immediately renames both halves to whatever the local component cares about: `count` and `setCount` here, `isOpen` and `setIsOpen` in the next file, `email` and `setEmail` in the form below that.

That raises a question worth holding in mind for the rest of this lesson: **why does `useState` return a tuple instead of a named-field object like `{ value, setValue }`?** Hold the question; the body answers it.

A tuple is what TypeScript calls an array shape **where position matters and length is fixed**. You already know arrays from the chapter on picking the right container, and you already know destructuring from the chapter on functions, naming, and control flow — a tuple is the type-level rule that turns "this array has some elements" into "this array has exactly these elements, in this order, with these types." Three things this lesson installs: the syntax (next section), the element labels that turn position into documentation, and the decision rule for when a tuple beats a named-field object.

One bug-anchor to keep on the table before we start. Position is fragile. A function returns `[id, name, email, role, createdAt]` — five strings and a `Date`. Six months later a teammate inserts `lastSeenAt` at position 3. Every call site that destructured by position silently shifts: `role` is now an email, `email` is now a role, and a downstream `role.includes('admin')` either crashes or — worse — silently lets the wrong user pass an authorization check. Labels (a TypeScript 4.0 feature this lesson writes by default) document the positions so the tooling surfaces them; the **three-position rule** prevents the bug from getting that far. The lesson lands both.

## Tuple syntax: positions, types, and the array contrast

A tuple type is `[T1, T2, T3, ...]` — square brackets with one type per position. The annotated value is a plain array at runtime: `Array.isArray(tup)` is `true`, `.map` and `.length` work the way you'd expect. The tuple part lives only at the type level, where TypeScript pins the **length** and the **type at each position**.

The shape to internalize is the contrast between a tuple and a same-bracket-looking array.

<CodeVariants>
  <CodeVariant label="Tuple — [string, number]">

    ```ts del={3-5}
    const pair: [string, number] = ['draft', 3];

    const wrongType: [string, number] = ['draft', 'three'];
    const wrongLength: [string, number] = ['draft', 3, true];
    const wrongOrder: [string, number] = [3, 'draft'];
    ```

    Length is fixed at 2. Position 0 must be a `string`, position 1 must be a `number`. The first line is the only one that compiles; the three below it all error — wrong type at a position, wrong length, wrong order. The compiler reads the shape strictly.
  </CodeVariant>

  <CodeVariant label="Array — (string | number)[]">

    ```ts
    const a: (string | number)[] = ['draft', 'three'];
    const b: (string | number)[] = ['draft', 3, 5];
    const c: (string | number)[] = [3, 'draft'];
    ```

    Every value the tuple rejected is fine here. Any length, any mix of the two element types, any order. This is what you write when length is unknown and the slots are interchangeable — they look similar at the bracket level, but the type-level constraint is different.
  </CodeVariant>
</CodeVariants>

At runtime, both forms are plain arrays — TypeScript erases the tuple constraint at build time, exactly like it erases every other type. The tuple is purely a compile-time agreement: "this array has exactly two elements, this type at position 0, this type at position 1." Nothing at runtime knows the difference.

:::tip
A tuple **is** an array. `.map`, `.filter`, `.length`, indexed read all work and return what you'd expect. The tuple-specific behavior — the per-position type and the fixed length — lives entirely at the type level.
:::

## Element labels: position with documentation

Open the bug from the introduction. `getUserRow` returns five strings and a `Date`. Without labels, every call site has to remember the order from outside the signature — and TypeScript can't catch a swap, because every wrong order still type-checks.

<CodeVariants>
  <CodeVariant label="Unlabeled — order is implicit">
    <div data-mark-color="red">

    ```ts "role, email"
    const getUserRow = (userId: string): [string, string, string, string, Date] => {
      // ...
    };

    const [id, name, role, email, createdAt] = getUserRow('u_1');
    ```

    </div>
    `role` and `email` are swapped at the destructure. The compiler shrugs — every position is the same type, so any permutation passes the check. The bug surfaces three files away when `role.includes('admin')` reads an `@` and silently doesn't match anything.
  </CodeVariant>

  <CodeVariant label="Labeled — names live in the signature">

    ```ts ins={2-6}
    const getUserRow = (userId: string): [
      id: string,
      name: string,
      email: string,
      role: string,
      createdAt: Date,
    ] => {
      // ...
    };

    const [id, name, email, role, createdAt] = getUserRow('u_1');
    ```

    Each position carries a name at the type level. The names show up in editor tooltips when you hover the call, and autocomplete suggests them in order at the destructure site. The compiler still can't enforce that you pick the matching local names — but the labels turn position into documentation that lives where the value lives.
  </CodeVariant>
</CodeVariants>

A <Term definition="A tuple type where each position carries a name at the type level. Names appear in editor tooltips and at destructure-site autocomplete. They don't affect type compatibility — [a: string, b: number] and [string, number] are the same type.">labeled tuple</Term> is exactly that: a tuple where each position has a name. The syntax is `[name: type, name: type, ...]`. Two rules to internalize:

**All-or-nothing.** Label every position or label none. `[a: string, number]` is a syntax error. TypeScript won't let you mix the two forms inside one tuple.

**Past length 2, label by default.** The two-element case — `[value, setter]`, `[key, value]`, `[error, result]` — is conventional enough that labels are noise. Past two, an unlabeled tuple is a future call-site bug waiting for a refactor. Three is the threshold where the cost of remembering position surpasses the cost of authoring the names.

One thing labels do **not** do: change type compatibility. `[string, number]` and `[a: string, b: number]` are the same type as far as assignability is concerned. The labels are pure documentation that the editor surfaces. That's the design — labels are a tooling feature, not a type-system feature.

## Readonly tuples

Prefix a tuple type with `readonly` and the same shape as `readonly T[]` from the previous lesson kicks in: the binding is locked at the type level, and the array methods that would mutate the tuple (`.push`, `.pop`, `.splice`, `.sort`, `.reverse`, direct index-write) drop off the type's surface.

```ts del={5-6}
type Pair = readonly [string, number];

const p: Pair = ['draft', 3];

p[0] = 'sent';
p.push('extra');
```

Both of the last two lines error. `p[0] = 'sent'` is the index-write you can't do on a `readonly` tuple. `p.push('extra')` is the array-mutation method that doesn't exist on the `readonly` form. Read methods — `.map`, `.filter`, `.length`, indexed read — stay intact, exactly like `readonly T[]`. If a function takes a `readonly [string, number]`, it's promising not to mutate the tuple in place.

One related move worth seeing once, because you'll meet it in the wild and recognize it: `as const` on an inline array literal produces a readonly tuple of literal types.

```ts
const statuses = ['draft', 'sent', 'paid'] as const;
//    ^? readonly ['draft', 'sent', 'paid']
```

Without `as const`, the inferred type of that array literal is `string[]` — TypeScript widens to a mutable array because that's the safer default for the average case. With `as const`, the same literal becomes a `readonly` tuple where each element keeps its narrow literal type. That's the seed for the typed-config pattern a later lesson in this chapter (Keeping literals narrow) builds out; you don't need to reach for it yet.

:::note
Don't reach for `as const` here. This lesson is the type-level vocabulary; the full `as const` move lands later in this chapter, in the lesson on keeping literals narrow. Recognize the form when you see it; the senior pattern comes later.
:::

## Optional and rest positions

Two more pieces of syntax exist for the cases where the tuple's length is **almost** fixed. You won't reach for either of these much in application code; you'll meet them in library types and need to read them correctly.

```ts
type RangeOrPoint = [start: number, end?: number];
const point: RangeOrPoint = [3];
const range: RangeOrPoint = [3, 7];

type Message = [header: string, ...payload: number[]];
const ping: Message = ['ping'];
const burst: Message = ['ping', 1, 2, 3];
```

**Optional positions** use `?` after the label (or after the type when there's no label): `[start: number, end?: number]` accepts a length-1 or length-2 tuple. The same `?`-on-fields rule from the previous lesson applies — optional positions must follow required ones. A required slot can't sit after an optional slot.

**Rest positions** use `...T[]` for "any number of additional elements of this type": `[header: string, ...payload: number[]]` is one string followed by any number of numbers. A tuple has at most one rest position. Since TypeScript 4.2 the rest doesn't have to be last (`[string, ...number[], boolean]` is legal), but the fixed-then-rest form is the canonical shape because it destructures cleanly: `const [header, ...payload] = m`. Other arrangements surface in library types; recognize them, but don't reach for them when you're authoring application code.

:::note
Optional and rest can compose (`[a: string, b?: number, ...rest: boolean[]]`) but the shape gets hard to read past two non-fixed elements. When it gets that complex, the senior reflex is to switch to an object. The decision-rule section at the end of the lesson covers the call.
:::

## The three sites tuples earn their weight

Time to answer the `useState` question. A tuple earns its weight when **the caller will destructure-and-rename on every use**. The rename happens at the destructure, so the positional shape carries no naming penalty — the caller writes the names they want, and the function ships positions. Three sites in 2026 SaaS code fit this pattern. Almost everything else wants an object.

### Custom hook returns

A hook that returns ordered state plus an action returns a tuple. `useState`'s `[value, setter]` is the precedent; the project's hook conventions follow it for any hook with two slots the caller will rename. Here's the canonical pattern:

```ts
const useToggle = (initial = false): readonly [boolean, () => void] => {
  const [on, setOn] = useState(initial);
  const toggle = () => setOn((v) => !v);
  return [on, toggle] as const;
};

const [isOpen, toggleOpen] = useToggle();
const [isHovered, toggleHover] = useToggle();
```

Two callers, two completely different names — `isOpen`/`toggleOpen` and `isHovered`/`toggleHover` — from the same hook. The tuple shape lets the call site's naming **be the API**. If `useToggle` returned `{ on, toggle }` instead, each caller would either use the literal names (a collision the moment two toggles live in one component) or rename at the destructure (`const { on: isOpen, toggle: toggleOpen } = useToggle()`) — strictly more verbose for the same outcome.

A <Term definition="Destructuring with a rename — const [open, toggle] = useToggle() lets the caller name the bindings without the hook prescribing them. This is the reason useState returns a tuple instead of an object.">destructure-and-rename</Term> is the operation that makes the tuple's positional shape work. Without it, the tuple shape is a worse object. With it, the tuple is the right call.

One ceiling rule: **two elements is the comfortable maximum for a tuple hook return.** Past two, names win — `useForm()` returns `{ register, handleSubmit, formState, ... }`, not a six-tuple. Three is the boundary where labels stop being enough and the senior reflex flips to an object. You'll meet this rule again later in the course when you author your own hooks for the project.

### `Object.entries` and `Map` iteration

`Object.entries(obj)` returns an array of `[key, value]` tuples. `map.entries()` returns the same pair shape over a `Map`. So does `array.entries()` and `set.entries()`. The pattern is everywhere in the language, and the tuple recognition is what lets the destructure read cleanly:

```ts
const statusLabels = { draft: 'Draft', sent: 'Sent', paid: 'Paid' };

for (const [status, label] of Object.entries(statusLabels)) {
  console.log(`${status}: ${label}`);
}

const counts = new Map<string, number>([
  ['draft', 3],
  ['sent', 7],
]);

for (const [status, count] of counts.entries()) {
  console.log(`${status}: ${count}`);
}
```

The destructure in the `for...of` head names both halves of the pair without an intermediate variable. You should leave able to read `for (const [a, b] of x.entries())` anywhere — the language ships this shape across `Object`, `Map`, `Set`, and `Array`. The two-slot tuple is exactly the case where labels would be noise, and the two-call-site rename rule earns its weight: the same `Object.entries` call gets `[status, label]` in one loop and `[status, count]` in another.

:::caution
`Object.entries` is typed as `[string, T][]` — the key is always `string`, even when the object has a literal-keyed shape like `{ draft: ...; sent: ...; paid: ... }`. The compiler widens the key to `string` at the entries call. Workarounds exist (typed `entries` helpers, narrow casts) but the senior reflex is to live with `string` for iteration and reach for `Record<LiteralUnion, V>` (next lesson) when the typed key matters at the access site.
:::

### Go-style `[error, value]` result wrappers

A third pattern you'll meet in libraries — and the one you should read but not author for new code. Some error-as-value libraries return a `[error, value]` tuple where the caller destructures both halves and branches on the error:

```ts
const [err, user] = await tryFetch('/api/me');
if (err) {
  return notFound();
}
```

The form reads cleanly when the caller knows the convention — error at position 0, value at position 1, exactly one of them set. You'll see it in `neverthrow`, in hand-rolled `tryCatch` helpers, in fetch wrappers from older codebases. Recognize it; destructure it correctly.

For **new** code, the course prefers a discriminated `Result` type:

```ts
type Result<T> = { ok: true; value: T } | { ok: false; error: Error };
```

The shape lands later in this chapter (in the lesson on unions and intersections) as a seed, and gets built out in the chapter on errors-as-values. It scales past two slots, narrows cleanly on `result.ok`, and gives the consumer a name for the field it's reading. The Go-style tuple is the read-it-correctly case; the discriminated `Result` is the write-it-this-way case.

## When to reach for an object instead

You've seen the three sites. Now the flip rule: **if the call site won't destructure-and-rename, or if the positions are easy to swap by accident, use a named-field object.** That rules out most "return multiple values" cases — a function returning `{ id, email, role }` is an object because the caller uses the names as-is. It also rules out tuples past three elements, where the cost of remembering position outweighs the conciseness gain.

Two short rules sit under the flip:

**Three positions is the rough boundary.** Coordinates (`[x, y, z]`) survive — the names at the call site are the same as the convention names, labels would be noise, and swapping is unlikely because the meanings are positional in the domain. Past three, the cost of remembering positions outweighs the conciseness, and the senior call is an object every time.

**If the slots have different meanings to different callers, it's an object.** A single shape that ships `[id, name, email]` to a user-card component and `[id, email, role]` to an auth check is two different signatures masquerading as one tuple. Two functions, two shapes — usually objects, possibly two tuples, never one.

## Write a labeled readonly tuple return

Now author the canonical shape once by hand. The starter below defines a `useDisclosure` hook that returns three slots — an `isOpen` boolean, an `open` action, and a `close` action — but the return type widens to `(boolean | (() => void))[]`, the array shape, not the labeled tuple the caller needs.

Your job: make the return type a labeled readonly tuple of `[isOpen, open, close]`. Two valid solutions exist. You can annotate the return type explicitly as `readonly [isOpen: boolean, open: () => void, close: () => void]`, or you can add `as const` to the returned array literal and let inference recover the shape. Either path lands the same type; the `^?` query on `Return` below confirms the shape.

<TypeCoding
  instructions="Type useDisclosure so its return is a labeled, readonly tuple of [isOpen, open, close]. The ^? query on Return should resolve to a type starting with `readonly [` — the readonly tuple form, with boolean at position 0 and () => void at positions 1 and 2. Two valid solutions: annotate the return type explicitly with labels (`readonly [isOpen: boolean, open: () => void, close: () => void]`), or add `as const` to the returned array literal and let inference do the rest. (The useState stub mirrors React's signature so the exercise stays type-only.)"
  starter={`// Minimal stub of React's useState so this exercise stays type-only.
declare function useState<T>(initial: T): [T, (next: T) => void];

const useDisclosure = (initial = false) => {
  const [isOpen, setIsOpen] = useState(initial);
  const open = () => setIsOpen(true);
  const close = () => setIsOpen(false);
  return [isOpen, open, close];
};

// The ^? query below confirms the return shape.
type Return = ReturnType<typeof useDisclosure>;
//   ^?
`}
  expectedQueries={[
    { line: 13, contains: 'readonly [' },
  ]}
/>

The two paths land the same type but read differently. The explicit annotation is the senior reach when the hook is exported from a shared file — the contract is visible in the signature without opening the body, and a teammate reading the export reads the labels in the tooltip. The `as const` form is fine for hooks scoped to a single file where the body is the documentation. Both are correct; both ship. The `as const` move comes back in full later in this chapter, in the lesson on keeping literals narrow.
