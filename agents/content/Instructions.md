# Project Instructions — Web Dev Course (2026)

## What this project is

We're building an **opinionated web development course** for **junior-to-mid developers who had some exposure to web development before, stepped away into other areas of software, and now want to come back**. They've written code, deployed things, so the course doesn't teach basic concepts a junior dev already knows. But beyond that floor, the course **does not assume they remember the fundamentals properly**. Every prerequisite the course depends on is covered inside it, at adult depth, without bootcamp scaffolding. A returning dev with rusty foundations can start at lesson 1 and end production-ready without ever opting into a "beginner track."

The committed stack, the architectural principles, the SaaS patterns, and the documentation practices are codified in `Table of contents.md`. Treat the TOC as the canonical source for *what* gets taught and *in what order*. This file is about *how to think and work on the project* — the posture every decision should pass through.

## Reference documents

- **`Table of contents.md`** — the canonical curriculum: 23 units enumerated down to the lesson, with the seven architectural principles and fifteen SaaS patterns tagged at the chapter where each is introduced. The *what gets taught and in what order* spine.
- **`Pedagogical approach.md`** — the *how* of teaching: lesson shape, exercise/quiz/sandbox/diagram/video decision rules, the small-projects model that lives at the end of select units, and the per-unit guidance. Sibling to this file; this file sets posture and `Pedagogical approach.md` operationalizes it.

## The two philosophical pillars

Every decision passes through these two filters. If a suggestion doesn't serve them, cut it.

### 1. Senior mindset over syntax

Code itself has become commodity. The senior contribution to a 2026 SaaS codebase isn't keystrokes — it's *what gets decided before code is written* (planning, system design, data modeling, API contract design) and *how the code is shaped so it stays changeable* (architectural patterns that hold up as the system grows). Every lesson foregrounds **decisions and reasoning**, not syntax. When teaching a feature, lead with *why this approach, when does it break, what would a senior do differently.* Code samples illustrate those decisions; they aren't the lesson.

AI-driven development is the daily reality of 2026 SaaS engineering and the reason this course exists in this shape. The default rule: **don't name AI in lesson material unless the feature, convention, or product surface being taught is itself AI-related.** AI capability moves fast — any "how to direct AI" lesson ages out before the course ships — and the actual senior skills apply identically whether code is typed by a human or generated by a model. Teach the durable skills properly and AI-readiness happens automatically.

### 2. Minimum viable 2026 stack

Teach the **smallest** set of technologies a SaaS startup would actually ship with in 2026. No redundancy, no "for completeness" detours, no historical context that doesn't change a present-day decision. If two tools do the same job, teach only the dominant 2026 startup default — never a comparison. The course's promise is that learners exit ready to build production apps with the stack senior engineers at SaaS startups actually use, not a museum tour of how we got here.

## How to think about the curriculum

### Prerequisites and pacing

**Skip the trivial baseline. Teach everything else properly.** The compression isn't in the depth of any single concept — it's in the *absence of the bootcamp preamble that surrounds it*. No patient "what is a function" detours.

**Coverage: production-surface exhaustive.** Teach every language and platform feature that shows up in code an experienced dev writes for this stack, at adult depth and without assuming the learner remembers it. A topic earns its place by being used in production. Returning devs have unpredictable gaps — a course that quietly assumes any specific gap is the course they bounce off in week three. The cut isn't *which features to include*; it's *which versions to teach* (current, not historical) and *how deep to go on academic edges* (not deep — features no production SaaS codebase reaches for stay out).

**Frame fundamentals against the senior judgment they unlock — where an anchor exists.** Where a fundamental has a clear downstream payoff, lead with it: the lesson lands as a senior-mindset move rather than a reference page. Where the payoff isn't a single named call, teach the feature on its own merits — the senior framing is still in the *how* (decisions before syntax, trade-offs named, code as illustration).

**Teach the form learners will actually write.** When a typed, wrapped, or framework-native form has displaced the underlying primitive in 2026 SaaS work, teach the form they'll write — not the underlying primitive as a separate preamble. The underlying layer doesn't get pretended away; it gets named at the call sites where it bites. (Teach TS and JS as one language; HTML through JSX; CSS through Tailwind; Postgres through Drizzle.)

### Defaults and conditionals

The stack splits picks into two kinds: **defaults you reach for first** and **conditional power tools** you only introduce when the simpler default is no longer enough. The pedagogical job is to **teach the platform default first and name the trigger condition that justifies crossing the line.** When introducing any conditional tool, open the lesson by naming the threshold the platform default crosses to make this tool worth its weight. The senior-mindset payoff is in the trigger reasoning, not the tool's API surface. Treat conditionals as power tools, not prerequisites — that distinction is the senior-mindset pillar made operational.

### Working with the committed stack

The stack was researched and locked in May 2026. Tools shift, so **re-verify any specific claim (versions, pricing, library status) before stating it as fact in lesson content** — but **don't relitigate the picks themselves** without a clear reason. If proposing anything outside the committed list, **justify it in startup-shipping terms**: who actually uses it, what does it replace, why is the redundancy worth it.

A few picks are genuinely opinionated rather than industry-consensus. When teaching them, briefly name *why we chose this* so learners understand the philosophy, not just the tool. The opinionated picks all earn an ADR for the same reason — six-months-from-now-you will ask "why did we do this?"

### Principles, patterns, and documentation practices

The course teaches seven architectural principles, fifteen SaaS application patterns, and a documentation discipline. They are introduced **inline at the moment they're most legibly demonstrated**, not bundled into a "fundamentals" preamble. Each one is load-bearing throughout the course — when a lesson involves writing code, the example code should embody these, not just reach a working result. **Reviewing example code against the principles and patterns is part of "is this lesson done?"**

Two named carve-outs to "use the framework's conventions, don't invent parallel ones" exist — one for the authz boundary, one for the billing interface. Both are documented where they're introduced. Don't generalize the carve-out pattern; it's the exception that names itself, not a license for new ones.

## Delivery format

The course ships as an **Astro Starlight site**, and lessons are authored as **MDX from day one** — never plain Markdown promoted later. Every lesson can reach for components without a format migration, so authoring decisions never get distorted by "this would need MDX, let's skip it."

**Component sourcing has an order of preference.** Reach for Starlight's shipped components first (Aside, Tabs, Card, Steps, FileTree, LinkCard, and the rest of the built-in set). When a built-in doesn't fit, reach for the **pre-built custom component set** maintained alongside the site. Only when both fall short does a lesson get to propose a *new* component — same posture as the rest of the stack: the platform default first, the custom tool only when the default has been crossed.

**New components follow a spec-it / build-it-separately workflow.** When a lesson needs a component that doesn't exist yet, the lesson plan names it — purpose, props, behavior, the lesson it serves, whether it's likely to be reused — and stops there. The component itself is built outside the lesson and wired in once available. Don't inline ad-hoc JSX as a workaround; if it's worth using, it's worth specifying so it can be reused and reviewed.

## How to communicate with the user in chat

Default to **detailed and thorough**. Walk through reasoning, name the trade-offs, leave nothing implicit. Curriculum work especially benefits from showing the why behind sequencing and scoping choices.

## Working habits

- **Verify 2026 reality with web search.** Before recommending a specific tool, version, library state, framework feature, deprecation, or pricing claim, web-search to confirm. Training-data instincts about the JS ecosystem age fast — assume any specific claim needs validation. General patterns don't need a search every time; specific names, versions, and "is X still the default" claims do.
- **Flag conflicts, don't fix them silently.** If something in the existing course materials conflicts with these rules — or with the TOC — surface it, propose a fix, and wait for approval before changing it.

## What to avoid

- Teaching deprecated patterns "for context."
- Framework or library comparisons that don't change a shipping decision.
- Treating the learner as a *brand-new* beginner — no patient "what is a function" walkthroughs, no celebratory tone for completing trivial steps, no bootcamp scaffolding around fundamentals.
- Adding a tool to the curriculum without justifying it in startup-shipping terms.
- Stating any specific 2026 fact (version, default, deprecation, pricing) without web-verifying first.
