# Chapter 13.1 — Background work — defaults and Trigger.dev

## Chapter framing

Chapter 13.1 lands the senior question students dodged through Units 1 to 12: where does "work that runs after the user's request" actually live, and when does shipping it inside the request itself stop being defensible. The chapter walks the ladder from cheapest to heaviest — inline `await` inside a Server Action, `after()` for fire-and-forget side effects on the same invocation, Vercel Cron for scheduled jobs, then Trigger.dev v4 once durability, multi-step execution, fan-out, long-running compute, or human-in-the-loop pauses enter the picture. The threshold is named once and revisited at every escalation: code stays at the lowest tier that meets the durability, latency, and time-budget requirements. The chapter inherits the idempotency discipline from 12.1.4 — every job that can retry carries a stable key — and the dedup-and-transact shape from 12.1.2 — every external side effect (Resend send, Stripe call, R2 write) is guarded by a unique constraint or a provider idempotency key.

Threads that run through every lesson: the request-response budget is sacred — user-visible work blocks the response, everything else does not; durability is the senior word for "this work survives a crash, a redeploy, a function timeout, and a transient downstream outage"; the platform default is the right answer until a named threshold is crossed (function time limit, retry semantics, multi-step orchestration, human pauses, fan-out); idempotency is not a Trigger.dev feature but the same DB-constraint discipline from 12.1, expressed through `idempotencyKey` at the trigger and wait boundaries; retries are exponential with jitter, never linear; observability comes free with Trigger.dev and must be built by hand at lower tiers, which is part of the threshold calculus. The chapter ships seven teaching lessons plus a quiz, ordered from the platform default outward: the lessons before Trigger.dev exist so the student can defend *not* reaching for it; the Trigger.dev lessons exist so the student can wield it without dragging in features they do not need.

## Lesson 13.1.1 — Inline async work and `after()` in Server Actions

Topics to cover:

- **The senior question.** A user clicks "Send invitation". The Server Action inserts the row, sends the email, and returns. What part of this work belongs inside the request-response cycle, what part belongs after the response is sent, and what part should not be on the request path at all? The lesson lands the 0-tier — when the right answer is "do it inline, return success" — and the 0.5-tier — when `after()` is the structural fix.
- **The default — sequential `await` inside the action.** A Server Action that does a DB write and one external call (one Resend send, one Stripe API call) lives entirely inside the action body, returns the Result shape from 7.2 after both complete. The senior anchor: synchronous-from-the-user's-perspective is the cheapest, most observable, most debuggable shape. The user sees the success state only after the side effect succeeded — no "queued for sending" half-state, no notification dispatcher to debug, no separate worker to deploy.
- **The thresholds that break the default.** Four named limits make inline async stop being defensible: (1) downstream latency that bloats the user-visible response (Resend p99 ~400ms is fine; a third-party API at 5s is not); (2) the function time limit — Vercel fluid compute caps at 14 minutes on paid plans, 1 minute on Hobby, but every second of that is on the user's request; (3) work that must continue if the user closes the tab — fire-and-forget cannot use the request scope; (4) work that must survive transient downstream failures with retries — the action returns to the user once, the retry needs somewhere else to live.
- **`after()` — Next.js 16's after-response primitive.** `import { after } from 'next/server'`. Schedules a callback to run after the response is sent, inside the same serverless invocation, before the function shuts down. Stable since Next.js 15.1. Usable in Server Components, Server Actions, Route Handlers, and Proxy. The senior anchor: `after()` is for "the user does not need to see this happen, but it must happen on this same invocation" — analytics events, structured logs that depend on the rendered response, fire-and-forget cache warming.
- **What `after()` is not.** It is not a job queue. It runs once, in the same function invocation, bounded by the same `maxDuration`. If the function times out or crashes after the response, the `after()` callback is lost. No retries, no durability, no cross-process visibility. The lesson pins this hard — the most common 2026 mistake is to reach for `after()` for work that needed Trigger.dev. The threshold: if losing the callback once in a thousand is acceptable, `after()` is fine; if it is not, escalate.
- **`after()` in the request lifecycle — what students must know.** `after()` extends the invocation past the response via Vercel's `waitUntil` primitive. On Vercel fluid compute, the invocation stays alive up to `maxDuration` even after the response ships. On self-hosted Node, the same behavior; on platforms without `waitUntil`, `after()` is a no-op or runs synchronously. Names the platform dependency.
- **Reading request data inside `after()`.** Inside Route Handlers and Server Actions, `cookies()` and `headers()` work inside the callback. Inside Server Components, they do not — read them before the `after()` call and close over the values. The reason is partial prerendering — Next needs to know what part of the tree is dynamic. The lesson names the gotcha and the fix (close over the value), restates briefly from 5.4.
- **The mental model — three latency tiers.** Tier 0: blocks the response (`await` inline). Tier 0.5: same invocation, after the response (`after()`). Tier 1+: outside the invocation entirely (Vercel Cron, Trigger.dev). The lesson draws the ladder once and the rest of the chapter climbs it.
- **Worked example — invite flow.** The `inviteMember` Server Action: insert the `org_invitations` row inside a `db.transaction`, send the email inline with `await sendEmail(...)`, write the audit log inside the same transaction, return Result. Then a variant: insert the row, return success immediately, schedule the audit log write via `after()`. Names which variant is right (the first — the user wants to know the invite actually sent), shows the second as the shape but flags it as the wrong default for this case.
- **Where `after()` actually earns its weight.** Logging the user agent and structured fields after a checkout completes, warming a `cacheTag` for a list the user is about to navigate to, posting a structured analytics event to PostHog from a Server Action where the analytics call's failure must not roll back the DB transaction.
- **The error-handling rule for `after()` callbacks.** Errors inside `after()` do not propagate to the user — the response is already sent. They must be caught and logged via the structured logger from 20.1, or they vanish. The senior anchor: `after()` is not a fire-and-forget guarantee, it is a fire-and-log discipline.
- **Watch-outs:** putting the actual user-visible side effect inside `after()` so the user sees "success" before the email sends — race condition between the user reloading and the email arriving, and no path to surface failure; using `after()` for work that exceeds `maxDuration` — silently truncated; using `after()` to "fix" slow Server Actions by hiding the latency — the action is still slow, the user still waits, you just lost the failure path; relying on `after()` for retries — there are none, one shot only; calling `cookies()` inside an `after()` callback in a Server Component — runtime error, read it outside.

What this lesson does not cover:

- Scheduled jobs and cron — Lesson 13.1.2.
- Durable retries and multi-step orchestration — Lessons 13.1.3 through 13.1.6.
- The Server Action canonical shape and Result type — Chapter 7.2.
- `cacheTag` / `revalidateTag` mechanics — Chapter 5.4 (and Chapter 15.1).
- Structured logging and observability of background errors — Chapter 20.1.
- The notification dispatcher — Chapter 14.1 (forward note only).

Estimated student time: 35 to 45 minutes. The "do not over-engineer" lesson — most students will write more inline awaits than Trigger.dev tasks and need to know why that is correct.

## Lesson 13.1.2 — Vercel Cron — scheduled jobs by default

Topics to cover:

- **The senior question.** A SaaS has a handful of recurring jobs — nightly digest emails, weekly metric rollups, expired-trial sweeps, stripe webhook reconciliation. They run on a schedule, they do not need to retry across hours, they fit comfortably inside a function timeout. Where does this work live? The lesson lands Vercel Cron as the 2026 default for schedules and pins the threshold that bumps a job up to Trigger.dev.
- **What Vercel Cron is, mechanically.** A `vercel.json` entry that maps a cron expression to a Route Handler path. Vercel's scheduler hits the path at the cron cadence; the handler runs as a normal serverless invocation, bounded by the same `maxDuration` as any other route. Names the topology: scheduler → HTTP GET to `/api/cron/<name>` → function executes → returns.
- **The `vercel.json` shape.** `{ "crons": [{ "path": "/api/cron/daily-digest", "schedule": "0 9 * * *" }] }`. Five-field cron expressions in UTC. The senior anchor: write UTC, never local time — the deploy region does not matter, the schedule does.
- **Where the handler code lives.** `app/api/cron/daily-digest/route.ts` exporting `GET`. Inside the handler, do the work, return a 200. The lesson names the convention (one folder per cron job, one `route.ts` per folder) and the reason (it shows up in the deploy preview and in logs grouped by path).
- **Auth — the `CRON_SECRET` discipline.** A cron handler is a public URL on the internet. Anyone can hit it and trigger the job. The protection: Vercel sets an `Authorization: Bearer ${CRON_SECRET}` header on every cron invocation; the handler checks the header against the env entry and returns 401 if it does not match. Pins this as non-optional — the same trust-boundary discipline as 12.1.1, expressed for cron.
- **The handler's verify-first shape.** The canonical cron handler: read `Authorization` header, constant-time-compare against `process.env.CRON_SECRET`, return 401 if it does not match, then do the work. Restates the constant-time compare from 12.1.1; the secret is short-lived enough that timing leaks are mostly theoretical, but the discipline is uniform.
- **Idempotency for crons.** Vercel guarantees at-least-once cron delivery, not exactly-once. A network blip can produce two invocations within seconds. The cron handler runs the same dedup-and-transact pattern from 12.1 with a key like `cron:<name>:<yyyy-mm-dd>` — one row in `processed_events` per cron tick. Most cron jobs that mutate idempotent state (e.g., "send digests to anyone who has not received one in the last 24 hours") do not strictly need this; jobs that send emails or charge cards do.
- **Time budget — same as any function.** A cron job is just a function invocation. Fluid compute on Pro caps at 14 minutes; Hobby caps at 1 minute. A digest job that emails 50 000 users one-by-one will time out. The senior anchor: if the work fits in the budget, Vercel Cron is fine; if it does not, the cron handler's job becomes "enqueue a Trigger.dev fan-out task" and the actual work moves out. Names the pattern as the bridge to 13.1.3.
- **What Vercel Cron does not give you.** No automatic retries on failure (a 5xx response is logged but not retried). No backoff. No visibility into intermediate state across a multi-step job. No human pauses, no waitpoints. No fan-out primitive — a single invocation per tick. Names each absence and ties it to the threshold for escalation.
- **Cron expressions students will actually write.** `0 9 * * *` daily at 09:00 UTC; `*/15 * * * *` every fifteen minutes; `0 0 * * 0` weekly Sunday midnight UTC; `0 0 1 * *` monthly first-of-month. Notes that Vercel does not support seconds in cron expressions and does not support the `L` / `W` extensions some schedulers offer.
- **Cost shape.** Vercel Cron is metered as function invocations plus compute time. A job that runs every minute costs 43 200 invocations per month; a daily job costs 30. The senior anchor: frequency drives cost, not work — a once-a-minute "do nothing" job is more expensive than a daily heavy job.
- **Worked example — the trial-expiry sweep.** `app/api/cron/sweep-trials/route.ts`. Cron `0 * * * *` (hourly). Handler: verify `CRON_SECRET`, open transaction, `UPDATE plan_entitlements SET status = 'expired' WHERE plan = 'trial' AND trial_end_at < now() AND status = 'active' RETURNING organizationId`, audit-log each row, return 200 with the count. The full shape — the trust check, the SQL-as-the-work, the audit log, the structured return. Closes the 11.2 lifecycle-aware UPDATE thread.
- **Local development for crons.** Vercel does not run crons against `next dev`. The development loop is "hit the URL manually with `curl -H 'Authorization: Bearer $CRON_SECRET' http://localhost:3000/api/cron/<name>`". The lesson names this and the workflow — write the handler as a normal route, test it via curl, deploy and watch the Vercel cron logs.
- **Watch-outs:** skipping `CRON_SECRET` because "it's just a cron URL" — public URLs get found within hours; using local time in the cron expression — schedules drift with DST; writing the actual work inline in the cron handler and exceeding `maxDuration` — silent truncation; not handling the at-least-once case for jobs with side effects — duplicate emails; checking `process.env.CRON_SECRET` inside the handler without a constant-time compare and without the early return — wastes the trust boundary; running crons more often than they need to "to be safe" — every minute when daily would do — pure waste.

What this lesson does not cover:

- Dynamic schedules per-tenant or per-row — Lesson 13.1.4 (Trigger.dev `schedules.create`).
- Durable multi-step jobs — Lessons 13.1.4 through 13.1.6.
- Fan-out and concurrency control — Lesson 13.1.4.
- Long-running jobs that exceed function time — Lesson 13.1.3.
- The full Stripe webhook reconciliation flow — Chapter 12.

Estimated student time: 35 to 45 minutes. The "the platform already does this" lesson — and the lesson that proves the student can defend not reaching for Trigger.dev for schedule alone.

## Lesson 13.1.3 — When Trigger.dev earns its weight

Topics to cover:

- **The senior question.** Inline `await` and `after()` cover same-invocation work. Vercel Cron covers scheduled invocations. What workloads do those two combined still fail at, and what specifically does Trigger.dev provide that earns the operational cost of a second platform? The lesson names the threshold explicitly so the student can defend the decision both ways.
- **The five trigger conditions.** Each is a named, observable property of the workload, not a vibe. **(1) Duration past the function time limit** — a job that needs more than 14 minutes on Pro fluid compute (or 1 minute on Hobby) cannot run on Vercel. **(2) Multi-step orchestration with intermediate state** — a job that does step A, waits for an external response or a delay, then does step B, where re-running A on failure would be incorrect or expensive. **(3) Automatic retries with backoff** — a job that must survive a transient downstream outage by retrying on its own schedule, not on the user's. **(4) Fan-out** — a single "send digests" trigger that spawns 50 000 child runs, one per user, with concurrency control. **(5) Event-driven and human-in-the-loop pauses** — a job that blocks waiting for a third-party callback URL, a human approval, or a wall-clock delay measured in hours or days.
- **The conditions that do not justify Trigger.dev.** A slow API call (under a minute) does not — use `after()` or accept the latency. A nightly job that fits in 5 minutes does not — use Vercel Cron. "I want a separate worker for cleanliness" does not — the platform is not your aesthetic. The senior anchor: the threshold is operational, not aesthetic.
- **The 2026 alternatives — who they are and when they win.** **Inngest** — strong serverless-native event system with step functions, similar shape to Trigger.dev, slight edge for teams already deep in event-driven design. **Vercel Queues** (in beta March 2026) — a managed message-queue topic/subscriber model on Vercel, good for fire-and-forget pub/sub but lighter on multi-step orchestration than Trigger.dev. **BullMQ + Redis** — self-managed, full control, requires running Redis and a worker process, wins when you are already on Render/Railway with persistent infra. **AWS SQS + Lambda** — enterprise-scale, heavy operational surface, wins inside an AWS estate. The course's pick is Trigger.dev v4 because the developer experience (typed payloads, native waitpoints, visible run timelines, local-CLI debugging) is the best fit for a small team in 2026, and the open-source license means students can self-host if cost shifts. Names the alternatives once; does not drill.
- **What Trigger.dev v4 provides, listed against the five conditions.** Durable runs that survive worker crashes and redeploys. Per-task retries with exponential backoff and jitter, configured declaratively. `wait.for` and `wait.until` as language primitives — the run pauses, the worker frees up, the run resumes when the wait completes. Waitpoints for external callbacks and human approval. Code-defined queues for concurrency control and fan-out. Typed payloads via `schemaTask`. A web dashboard showing every run, every step, every retry, every payload. A local dev CLI that streams live logs and lets the student kill the worker mid-run to prove durability.
- **The architectural shape — Trigger.dev next to the app, not inside it.** Trigger.dev runs as a separate service (cloud or self-hosted). The app triggers tasks via the Trigger.dev SDK over HTTPS. Tasks live in a `src/trigger/` folder, deploy via the Trigger.dev CLI alongside Vercel deploys. Names the deploy topology: two CI steps (`vercel deploy` and `trigger deploy`), one shared codebase, types flow between them via the SDK.
- **The cost shape — runs, run-minutes, and concurrency.** Trigger.dev cloud bills per run plus per run-minute plus concurrency seats. The free tier covers small SaaS workloads (5 000 runs/month, 6 hour-equivalent compute). The senior reach: track per-task run count and run duration in the Trigger.dev dashboard the same way you'd track function invocations on Vercel — the bill grows linearly with work.
- **The "should I bring this in" decision tree.** A short worked decision walk: "I need to send a single email synchronously" → inline await. "I need to log analytics after the response" → `after()`. "I need to run a nightly job that finishes in 5 minutes" → Vercel Cron. "I need to send 50 000 emails on a schedule" → Vercel Cron triggers a Trigger.dev fan-out task. "I need a job that waits for a webhook from a third party" → Trigger.dev waitpoint. "I need a multi-hour data export" → Trigger.dev task with `wait.for` between pages. Mermaid flowchart shape with the threshold at every fork.
- **The course's actual triggers — preview of Chapter 13.2 and 13.4.** The export job (13.2): multi-step, paginated, must resume on worker crash, sends an email at the end — five trigger conditions all present, Trigger.dev wins. The R2 upload (13.4): not Trigger.dev — direct presigned PUT, inline. Names the contrast: not every job is a Trigger.dev job.
- **The org-versus-app separation.** Trigger.dev tasks run with no Better Auth session and no `tenantDb` middleware; the org context must be passed in the payload and re-derived inside the task. The lesson names this constraint and the implication: every task payload is `{ organizationId, ... }`, every DB call inside the task uses `tenantDb(organizationId)` explicitly. Foreshadows the schema-task shape in 13.1.4.
- **Watch-outs:** reaching for Trigger.dev before any of the five conditions actually triggers — operational complexity for nothing; assuming Trigger.dev tasks share request context with the calling code — they do not, pass the org ID explicitly; running Trigger.dev tasks against the same Postgres without considering connection pool contention with the main app — names the symptom, points at PgBouncer / Drizzle's pool config from 6.4; using Trigger.dev for work that needs to happen *before* responding to the user — you have inverted the model, the user is now waiting on the task; comparing Trigger.dev cost per run to Vercel cost per invocation without comparing what each does — they are not the same unit.

What this lesson does not cover:

- The Trigger.dev SDK and task definitions — Lesson 13.1.4.
- Durable execution mechanics (retries, waits) — Lessons 13.1.5 and 13.1.6.
- Self-hosting Trigger.dev — out of scope for this chapter; named as an option.
- Detailed comparison with Inngest, BullMQ, Vercel Queues beyond the "when to pick" line — out of scope.
- Database connection pooling under high concurrency — Chapter 6.4 owns the topic.

Estimated student time: 40 to 50 minutes. The "defend the decision" lesson — equips the student to say "no, Vercel Cron is enough here" and "yes, Trigger.dev because of this exact property".

## Lesson 13.1.4 — Trigger.dev v4 primitives — tasks, schemaTask, queues, schedules

Topics to cover:

- **The senior question.** Trigger.dev v4 is the chosen runtime. What is the minimum API surface a student must know to define, trigger, type, and operate a task, and where is the v3-to-v4 break the student must know about because the docs and AI search results are full of v3 examples? The lesson lands the v4 mental model, the four primitives the rest of the chapter uses, and the v3 traps to avoid.
- **The directory and setup shape.** `src/trigger/` folder, one task per file or grouped by domain. `trigger.config.ts` at the repo root declares the project ref, the runtime, the build target, and the queues. `npx trigger.dev@latest init` for the first-time setup. The CLI commands students will run: `pnpm trigger dev` for the local worker, `pnpm trigger deploy` for production. Names the topology once; lesson 13.2.2 walks the starter.
- **`task()` — the base primitive.** `import { task } from '@trigger.dev/sdk'`. A task is `task({ id: 'send-digest', run: async (payload, { ctx }) => { ... } })`. The `id` is the durable identity — it persists across deploys, runs reference it by ID. The `run` function is the body. The senior anchor: the `id` is API; treat it like a route path, do not rename casually.
- **`schemaTask()` — the typed variant, default for everything in the course.** `import { schemaTask } from '@trigger.dev/sdk'` plus a Zod 4 schema for the payload. `schemaTask({ id: 'export-csv', schema: z.object({ organizationId: z.uuid(), since: z.iso.date() }), run: async (payload, { ctx }) => { ... } })`. Payload is parsed and validated before the task body runs; an invalid payload throws at trigger time, not three minutes into the run. The senior anchor: every payload from the app to a task is Zod-validated — same discipline as Server Action inputs from 7.1. Names the v4 support for Standard Schema (Zod, Valibot, ArkType) so a future swap is structural.
- **Triggering a task from the app.** `import { tasks } from '@trigger.dev/sdk'` plus `await tasks.trigger<typeof exportCsv>('export-csv', { organizationId, since })`. The generic carries the inferred payload type from the task definition — calling code gets type errors on payload shape. Returns a `handle` with `id` (the run ID) used for status checks and cancellation. The lesson pins: `tasks.trigger` is fire-and-forget — it returns once the run is enqueued, not once it completes. Use `tasks.triggerAndWait` only inside another task body, never from request code.
- **The `ctx` argument — what the task has access to.** `ctx.run.id` (the durable run ID, the natural idempotency key), `ctx.attempt.number` (which retry this is), `ctx.environment` (dev / staging / production). No request context, no session, no cookies, no headers — the task is its own world. Restates the org-separation rule from 13.1.3.
- **Queues — the v4 break from v3.** In v3, queues were declared at trigger time. In v4, queues are declared in code, at the project level, before the deploy that uses them. `trigger.config.ts` exports `queues: [{ name: 'export', concurrencyLimit: 5 }]`. The task references the queue by name in its definition. The senior anchor: queues are infrastructure now; treat them like database tables — declared in code, migrated at deploy time. The break is the most-likely-to-bite v3-to-v4 trap because old examples in search results will not work.
- **Concurrency limits.** A queue with `concurrencyLimit: 5` means at most 5 runs from that queue run simultaneously, regardless of how many are enqueued. Extra runs queue and wait. The senior reach: concurrency is the back-pressure primitive — set it to the smallest number that keeps downstream services happy (the Resend rate limit, the third-party API quota, the DB pool size). Names the rule of thumb: concurrency at the queue, not at the task body.
- **Per-tenant queues — the multi-tenant pattern.** A single static queue serializes all tenants and a single noisy tenant starves everyone. The v4 pattern is dynamic queues keyed by `organizationId`: `await tasks.trigger('export-csv', payload, { queue: { name: `org-${organizationId}`, concurrencyLimit: 1 } })`. Each org gets its own queue, sequential per org, parallel across orgs. The lesson names this as the SaaS-tenant pattern; the project (13.2) demonstrates.
- **Scheduled tasks — `schedules.task` and `schedules.create`.** Two forms. Static: `schedules.task({ id: 'nightly-digest', cron: '0 9 * * *', run: async () => { ... } })` — one global schedule, declared in code. Dynamic: `await schedules.create({ task: 'nightly-digest', cron: '0 9 * * *', externalId: organizationId })` — created at runtime per tenant or per resource, with the `externalId` so the schedule can be looked up and canceled later. Names the trade: static for one-shape global jobs, dynamic for per-tenant cadences.
- **Schedules versus Vercel Cron — the second-look.** A scheduled Trigger.dev task earns its place when the schedule needs to be dynamic (per-org cadence) or when the schedule's work needs the durability of Trigger.dev anyway. A static daily job that fits in Vercel's time budget stays on Vercel Cron. The senior anchor: do not migrate a working Vercel cron to Trigger.dev for uniformity.
- **The `init` lifecycle hook.** Tasks can declare an `init` function that runs once per run before the body, returning a value passed to the body. Useful for setting up a DB client, fetching a config row, opening a connection. The lesson names it once with a one-line example (`init: async () => ({ db: createDbForOrg(payload.organizationId) })`) and moves on.
- **What the dashboard shows.** Every run with payload, status (queued, executing, completed, failed, retrying), start time, duration, every log line, every retry's stack trace, every wait state. The lesson pins: this is observability for free — the equivalent on Vercel Cron is `console.log` plus a hope. The free observability is part of the threshold from 13.1.3.
- **Worked example — the `notify-org-members` task.** `schemaTask({ id: 'notify-org-members', schema: z.object({ organizationId: z.uuid(), eventType: z.string() }), queue: 'notifications', run: async ({ organizationId, eventType }, { ctx }) => { /* read members, send emails */ } })`. Triggered from a Server Action via `tasks.trigger`. The full lifecycle: action triggers, task enqueued, worker picks up, run executes, logs visible in dashboard. The student sees the shape they will use in 13.2.
- **Watch-outs:** copy-pasting a v3 example that declares the queue at trigger time — does not work in v4, queues are code-defined now; using `task` instead of `schemaTask` and then validating the payload inside the body — duplicates work, hides errors past the queue; triggering a task from inside another task with `tasks.trigger` instead of `tasks.triggerAndWait` when you needed the result — fire-and-forget when you wanted a child step; setting concurrency at the task level (does not exist in v4) instead of at the queue; calling `tasks.triggerAndWait` from a Server Action — the action blocks on the task and times out; renaming a task's `id` and not migrating in-flight runs — orphaned runs in the dashboard; placing tasks outside `src/trigger/` and wondering why the CLI does not find them.

What this lesson does not cover:

- Retries, exponential backoff, and `wait.for` / `wait.until` — Lesson 13.1.5.
- Waitpoints for external callbacks and human approval — Lesson 13.1.6.
- The project walkthrough and starter — Chapter 13.2.
- Self-hosting Trigger.dev infrastructure — out of scope.
- The full Trigger.dev pricing model — referenced, not drilled.
- The `init` lifecycle in depth (DB pooling patterns) — named, not drilled.

Estimated student time: 55 to 65 minutes. The "I can define and run a task" lesson — biggest lesson in the chapter because it is the API surface every later lesson assumes.

## Lesson 13.1.5 — Durable execution — retries, waits, idempotency

Topics to cover:

- **The senior question.** A task runs for ten minutes. The worker crashes at minute eight. A third-party API rate-limits the third call inside the task. What does Trigger.dev do, what does the student have to do, and where do production bugs hide in the seam between the two? The lesson lands the durable-execution mental model and pins idempotency at every external-call boundary.
- **What "durable" actually means.** A durable run survives worker crashes, redeploys, and platform restarts. The runtime checkpoints the run's state at every `await wait.*`, every `await tasks.triggerAndWait`, every `await ctx.run.*`, and at the end of every retry attempt. On crash, the run resumes from the last checkpoint on a new worker. The senior anchor: durability is a property of the *boundaries between steps*, not of the steps themselves — a long synchronous CPU loop in one step is not durable, the worker dies and the loop restarts from the beginning. Pins the implication: split long work into small steps separated by `wait` or `triggerAndWait`.
- **Retries — the declarative shape.** Every task has a `retry` block: `retry: { maxAttempts: 5, factor: 1.8, minTimeoutInMs: 1000, maxTimeoutInMs: 60000, randomize: true }`. Exponential backoff with jitter. A throw inside `run` triggers a retry with the configured backoff. `ctx.attempt.number` tells the body which retry it is on. The senior anchor: retries are the runtime's job, not the body's — never `try/catch` around an external call to "retry inside" when the runtime already does it for free.
- **What gets retried, what does not.** Any thrown error retries by default. The exception: errors deliberately thrown as `AbortTaskRunError` skip retries and fail the run immediately — for unrecoverable errors like a 400 from a downstream API with a bad payload. The lesson names the rule: retry on transients (5xx, network, rate-limit), abort on permanents (4xx with a stable payload, validation failures, programmer errors).
- **Run-level versus call-level retries.** Run-level retries restart the task from the top — every step re-runs. Call-level retries (the runtime's automatic retries for SDK calls that hit rate limits) restart only the failing call. The implication for the student: a step with side effects (a DB write, a Resend send) inside a task that retries from the top will execute twice unless guarded by an idempotency key. Names the cause of every "duplicate email" production bug.
- **`wait.for(duration)` — wall-clock waits.** `await wait.for({ minutes: 5 })` checkpoints the run, frees the worker, and resumes after the duration. The duration is wall clock — survives crashes, survives redeploys. The senior reach: use this for "wait for a third-party rate limit window to reset" or "wait for a backend job to finish" or "wait for a polling interval between pages of an export". Names the contrast with `setTimeout` inside the body — `setTimeout` holds the worker for nothing and dies on crash.
- **`wait.until(date)` — wait to a wall-clock time.** `await wait.until({ date: new Date('2026-12-01T09:00:00Z') })`. Same checkpoint semantics. Useful for "send the welcome email 24 hours after signup" or "expire the trial at the end of the period".
- **`idempotencyKey` on `tasks.trigger` and `wait.for` / `wait.until`.** Every trigger and every wait accepts an `idempotencyKey` and `idempotencyKeyTTL`. Same key within the TTL window returns the same run handle / waitpoint — does not enqueue a new run, does not start a new wait. The senior anchor: this is the dedup-and-transact discipline from 12.1.4, re-expressed as a runtime primitive — the same DB-constraint logic, hosted by Trigger.dev. Use `ctx.run.id` as the key when retrying a step within a run; use a stable business key (`org:${orgId}:export:${date}`) when triggering from the app.
- **`idempotencyKeys.create` — first-class key construction.** `import { idempotencyKeys } from '@trigger.dev/sdk'`. `await idempotencyKeys.create([organizationId, eventType, day])` produces a stable key from a key array. Names the helper once; the project uses it.
- **Cross-step idempotency — the pattern.** A task that emails N users in a loop, called inside a run that may retry, needs each per-user send guarded by its own idempotency key. Pattern: `for (const user of users) { await tasks.triggerAndWait('send-one', { userId: user.id }, { idempotencyKey: `${ctx.run.id}:user:${user.id}` }) }`. A retry of the outer run re-issues the same keys; the runtime returns the prior result instead of re-sending. Closes the idempotency thread from 12.1.4 with a concrete loop shape.
- **Run priority — queue-ordering offset.** Every `tasks.trigger` accepts a `priority` integer. Higher priority runs jump ahead in the queue, modulo concurrency limits. The senior reach: priority for user-initiated work (a user clicked "export now"), default for scheduled work, low for backfills. Names the API, names the rule, does not drill.
- **Lifecycle hooks for the senior cleanup.** `onSuccess` and `onFailure` per task — run after the body succeeds or after all retries exhaust. Useful for the "always write a final audit log" or "always clean up the temporary R2 object" pattern. Names the hook; flags that the hook itself is not retried, so the work inside should be idempotent.
- **Cancellation — `runs.cancel`.** A run can be canceled from the dashboard or via `runs.cancel(runId)`. Inside the body, `ctx.run.abortSignal` is an `AbortSignal` that fires on cancel — propagate it to `fetch` calls to short-circuit cleanly. The senior anchor: cancellation is cooperative; the body must check the signal at step boundaries or it runs to completion.
- **Worked example — paginated export with retries.** A `schemaTask` that exports an org's invoices to CSV in batches of 500. Inside the body: `for (let page = 0; page < totalPages; page++) { await pageStep.triggerAndWait({ orgId, page }, { idempotencyKey: `${ctx.run.id}:page:${page}` }); await wait.for({ seconds: 2 }); }`. A worker crash mid-export resumes on a new worker, the completed pages return cached, the next page re-executes. The student sees the durability story end to end. Same shape as the project (13.2) build steps.
- **Watch-outs:** retrying a body that calls Resend without an idempotency key — guaranteed duplicate emails across the retry boundary; using `setTimeout` instead of `wait.for` — worker held idle, durability lost; throwing `Error` for a permanent failure that should not retry — five retries of a bad payload wasted; using `wait.until` with a date in the past — fires immediately, not "skipped"; assuming `ctx.run.id` is stable across child runs — every `triggerAndWait` spawns a new run with a new ID, use the parent's `ctx.run.id` for keys that must align; mixing run-level and call-level retries by adding `try/retry` inside the body — the runtime's retry already covers it, the manual loop is redundant; not propagating `ctx.run.abortSignal` to fetches — cancellation does not stop in-flight HTTP.

What this lesson does not cover:

- Waitpoints for external callbacks and human approval — Lesson 13.1.6.
- The full task definition and queue setup — Lesson 13.1.4.
- Webhook signature verification and dedup at the route handler — Chapter 12.1.
- Database transaction semantics inside tasks — Chapter 6.4.
- Cancellation UI patterns and surfacing run state to users — Chapter 14.1 (notifications).

Estimated student time: 50 to 60 minutes. The "I can write a task that survives production" lesson — every senior-level production task hangs on this material.

## Lesson 13.1.6 — Waitpoints — external callbacks, timeouts, human-in-the-loop

Topics to cover:

- **The senior question.** A task triggers a third-party long-running job (a video render, a payment authorization, a partner API import) and must resume only when the third party calls back. Or a workflow needs a human to approve a refund before continuing. Or N parallel sub-jobs must all finish before a final aggregation step. Where does the run live in the meantime, and how does the resume work without polling? The lesson lands waitpoints — the v4 primitive that makes these patterns first-class.
- **What a waitpoint is.** A durable resumable token. The run hits `await wait.forToken(...)`; the runtime checkpoints, frees the worker, and parks the run on the token. The token is completed externally — via an HTTP callback, a programmatic SDK call, a timeout — and any run blocked on it resumes. The senior anchor: waitpoints replace the "poll a status endpoint until done" pattern with structural pausing — no polling, no wasted compute, no missed completions.
- **The two-direction relationship — many runs can wait on one token, one run can wait on many tokens.** A single waitpoint can block multiple runs (e.g., a feature-flag flip blocks every run waiting on "feature enabled"). A single run can wait on multiple waitpoints (e.g., "any of these three sub-jobs finishing" or "all of these N sub-jobs finishing"). The lesson names the topology once with a small diagram.
- **`wait.forToken` — the basic shape.** `const token = await wait.createToken({ timeout: '24h' })`. `token.publicAccessToken` is the URL-completion handle, `token.id` is the internal ID. Pass `token.publicAccessToken` to the third party as the callback URL. Then `await wait.forToken<{ result: string }>(token.id)` inside the task body. The runtime resumes the run with the value when the third party posts to the callback URL. The generic types the resumed payload.
- **Completing a token externally — the callback URL.** Trigger.dev exposes a public endpoint per token: `https://api.trigger.dev/v1/tokens/${publicAccessToken}/complete` accepts a POST with the completion payload. The third party posts directly. The senior anchor: this is the cleanest "we hand off to a third party and wait for them to call back" surface in the stack — no inbound webhook handler to write, no dedup table for the callback, the runtime handles it.
- **Completing a token programmatically.** `await wait.completeToken(tokenId, { result: 'approved' })` from anywhere — another task, a Server Action, a Route Handler. Useful for the human-in-the-loop pattern — the human clicks "approve" in the admin UI, the Server Action calls `wait.completeToken`, the run resumes.
- **Timeouts on waitpoints.** Every waitpoint can carry a `timeout` (`'24h'`, `'7d'`, ISO duration). On timeout, the wait resolves with a timeout marker — `{ ok: false, timedOut: true }` — and the run resumes with the explicit signal. The senior reach: every external waitpoint has a timeout; no "wait forever" in production. Names the rule.
- **Human-in-the-loop — the approval workflow shape.** A worked example: a refund task triggers, creates a waitpoint with a 48-hour timeout, writes a row into `pending_approvals` carrying the token ID, sends a Slack notification to admins, then awaits the token. An admin opens the admin UI, clicks "Approve" or "Reject", the Server Action calls `wait.completeToken(tokenId, { decision })`, the task resumes and applies the decision. The senior anchor: the run is alive but consuming nothing — the worker is free, no polling, the token just sits.
- **Third-party-callback — the integration shape.** Worked example: a video transcoding task posts to the partner with `callback_url = token.publicAccessToken`, then `await wait.forToken(token.id)`. The partner finishes, posts back, the run resumes with the partner's response payload. The senior reach: the partner's API surface and the wait token URL are perfectly matched — no glue webhook handler needed in the SaaS.
- **Multiple waitpoints — `wait.forWaitpoint` with an array.** `await wait.forWaitpoint([token1.id, token2.id, token3.id], { all: true })` blocks until all complete; `{ all: false }` (the default) is "any". Useful for fan-out aggregation: trigger N sub-tasks, each completes its own token on success, the parent waits on all. Names the API; the alternative shape uses `triggerAndWait` with `Promise.all` for typed sub-task results.
- **Waitpoints versus `wait.for` versus `triggerAndWait` — when each fits.** `wait.for` for time-based pauses with no external completion. `triggerAndWait` for "do a child task, get its result" — the runtime manages the waitpoint internally. `wait.forToken` for "an external system or human will signal completion" — the student manages the token. The lesson names the decision tree once.
- **Idempotency on token completion.** Tokens are completed once; a second completion call is a no-op (returns the original result). The senior anchor: the at-least-once delivery story for the callback URL is handled by the runtime — no dedup table needed on the SaaS side for the callback itself, only for the work the resume triggers.
- **Observability — what the dashboard shows for waited runs.** A run blocked on a token shows up as "Waiting" with the token ID, timeout countdown, and the URL the third party should call. The senior reach: when a third-party integration goes silent, the dashboard is the first place to look — the token tells you exactly what the run expects.
- **The `metadata` channel — passing data to a waiting run.** Inside the task body, `ctx.run.metadata` is a mutable object the run can write to and the dashboard renders live. Useful for "show progress 47 of 200" to a watching admin or to a polling client. Names the API; the project (13.2) uses it for visible run progress.
- **Worked example — the third-party render callback.** A `schemaTask` that triggers a partner video render and waits for completion. Body: create token with 6h timeout, POST to partner with `callback_url = token.publicAccessToken`, `wait.forToken<{ renderUrl: string }>(token.id)`, then save the `renderUrl` to the DB. The lesson shows the entire shape — 30 lines of task body, structural pause, no polling, full durability.
- **Watch-outs:** waitpoints without a timeout — the run lives forever waiting on a third party that went silent; trusting the third party to call the right URL — log the token creation and the resume, alert if the wait expires; completing a token from inside a transaction that may roll back — the wait resumed but the DB state did not commit, split the completion call out of the transaction; using `wait.forToken` for a wait that is actually time-based — use `wait.for`; passing the internal `token.id` to the third party instead of the `publicAccessToken` — exposes the internal ID, the partner cannot complete the token through the public URL; assuming a completed token is one-shot per run — it is one-shot per token, multiple runs can listen and all resume.

What this lesson does not cover:

- The task definition primitives — Lesson 13.1.4.
- Retry and idempotency mechanics — Lesson 13.1.5.
- Building the admin UI for human approval — Chapter 14 (notification dispatcher) and beyond.
- The full third-party-integration handshake on the partner side — partner-specific, out of scope.
- Self-hosting Trigger.dev so the waitpoint callback URLs resolve internally — out of scope.

Estimated student time: 45 to 55 minutes. The "I can pause and resume" lesson — the v4 primitive students will reach for the moment a real workflow has a human or a callback in it.

## Lesson 13.1.7 — Trigger.dev in our app — picking the workloads

Topics to cover:

- **The senior question.** The student has the threshold, the primitives, retries, waits, and waitpoints. Where in the course's actual app does Trigger.dev get wired in, and where deliberately not? The lesson is short and operational — it names the three workloads the course ships against Trigger.dev, names the workloads the course explicitly keeps on the platform default, and pins the env / config surface students will see in the project.
- **The three workloads the course wires through Trigger.dev.** **(1)** The CSV export job (Chapter 13.2 project) — multi-step, paginated, time-budget-bursting, sends an email at the end, must survive worker crashes — all five trigger conditions present. **(2)** The Stripe reconciliation sweep (forward note — extends 12.1 / 12.2) — a nightly schedule that reads Stripe state for any org with `lastEventAt` older than 24 hours and reconciles drift; durability matters because partial reconciliation is wrong. **(3)** The notification dispatcher fan-out (Chapter 14.2 forward note) — one event fans out to N channels per N users, per-org queue serializes, the dispatcher runs as a Trigger.dev task triggered from the `notifyEvent` Server Action helper.
- **The workloads the course keeps on the platform default.** Resend send from a Server Action inside `inviteMember` — inline, single call, finishes in 200ms; no Trigger.dev. The hourly trial-expiry sweep from 13.1.2 — fits in Vercel's time budget, idempotent SQL UPDATE, no need to escalate. Audit log writes — inside the action's transaction, never deferred. Analytics events from a Server Action — `after()` callback, fire-and-forget. Names each, names why it stays where it is.
- **The env surface students will see.** `TRIGGER_SECRET_KEY` for the SDK at the app side, `TRIGGER_PROJECT_REF` in `trigger.config.ts`. `CRON_SECRET` for Vercel crons. Foreshadows the `.env.example` of 13.2.
- **The deploy surface.** Two deploy commands in the SaaS CI: `vercel deploy` and `pnpm trigger deploy`. The order matters when a Server Action triggers a new task version — deploy Trigger.dev first, then the app, so the app never references a task version that has not landed. Names the rule; deeper coverage in Chapter 21 (CI/CD).
- **The cost model in operational terms.** Per-run cost is small; per-run-minute cost adds up for long-running tasks; concurrency seats add up for high-fan-out workloads. The senior reach: monitor the Trigger.dev dashboard's per-task run count weekly the same way you monitor Vercel function invocations — anomalies (a task running 10x more than expected) usually indicate a missing idempotency key or a retry storm.
- **The "two services, one codebase" mental model.** Tasks live in `src/trigger/`, the SDK is a single import, types flow. The student's Postgres is shared — same Drizzle schema, same `tenant-db.ts` helper, same audit log table. The task body calls the same `lib/email.ts`, the same `lib/billing.ts`, the same DB helpers as the app. Names the shape: Trigger.dev is a *runtime* for code that already lives in the codebase, not a separate codebase.
- **The "I do not have a Trigger.dev cloud account yet" path.** A short note: Trigger.dev v4 is open-source under Apache 2.0; self-host is an option once a SaaS scales past the free tier or has data-residency constraints. The course uses cloud for simplicity; the architectural pattern does not change. Names the off-ramp; does not drill.
- **Forward note for Chapter 13.2.** The project ships the CSV export task end-to-end: starter clone, schema-task definition with payload validation, paginated `triggerAndWait` per page with per-page idempotency keys, final email send, dashboard verification, mid-run worker kill to prove durability, parallel-trigger test to prove per-org serialization.
- **Watch-outs:** dragging every async Server Action into Trigger.dev "for consistency" — operational complexity for nothing, against the threshold from 13.1.3; deploying the app before deploying the new task version — `tasks.trigger('new-task-id')` fails at runtime; running the local Trigger.dev dev CLI against the production Trigger.dev project — production runs trigger on every save; sharing the Trigger.dev `TRIGGER_SECRET_KEY` between staging and production — same blast radius mistake as sharing webhook secrets in 12.1.1.

What this lesson does not cover:

- The export task's actual code — Chapter 13.2 project.
- The notification dispatcher implementation — Chapter 14.
- Stripe reconciliation — Chapter 12 forward note.
- Deploy pipelines and order of operations — Chapter 21.
- Self-hosting Trigger.dev infrastructure — out of scope.
