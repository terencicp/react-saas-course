# Chapter 2.9 — Practical odds and ends

## Chapter framing

Chapters 2.1 through 2.8 installed the language: values and references, the function and control-flow surface, the collection types, the type system and the moves that prevent bug classes, the module graph, the async runtime, and errors as a first-class concern. What's left is the small set of practical surfaces a senior reaches for regularly that didn't earn a whole chapter of their own — JSON at the wire, classes in the rare places they still belong, and the `Date` situation now that Temporal is unflagged in the runtime the course targets. This is the cleanup chapter: short, decisive, and tightly scoped. Every lesson resolves a specific question the student will hit on their first week shipping production code, and every lesson plants a forward reference to the unit that owns the full surface.

The senior framing for the chapter: **defaults, decisions, and where the platform finally caught up.** JSON is the wire format every API the course will touch uses; the senior writes `JSON.parse` and `JSON.stringify` by reflex and knows the three values that don't round-trip (`Date`, `undefined`, `BigInt`) and the seam where Zod (Unit 7.1) reasserts type safety after parse. Classes are not the default for application code on this stack — function composition is — but they earn a narrow place: custom `Error` subclasses (which Chapter 2.8 already taught the consumer side of) and the rare adapter wrapping a class-based SDK. The student needs the syntax and the trigger, not the OOP doctrine. `Date` has been a known liability for fifteen years and the language finally fixed it: Temporal reached Stage 4 in March 2026 and shipped unflagged in Node 26 (May 2026), which is the runtime the course pins to. The student leaves with the runtime-specific call (native Temporal on Node 26, polyfill on Node 24 LTS) and the heuristic for when `Date` is still acceptable.

Threads that must run through every lesson:

- **JSON is the wire format, not a serialization library.** Every API the course will touch — `fetch` request and response bodies (Unit 3.6), Next.js Route Handlers (Unit 7.5), Server Action serialization across the network boundary (Unit 7.2), Stripe and webhook payloads (Unit 12), Trigger.dev job inputs (Unit 13.1), the localStorage value (Unit 3.7.5) — uses JSON. The student writes `JSON.parse` and `JSON.stringify` by reflex and reaches for Zod (Unit 7.1) the moment the parsed value crosses a trust boundary. The chapter installs the language primitives so Zod can do the type-narrowing work in Unit 7.
- **Classes are not the default.** The course writes functions, types, and discriminated unions. Classes show up in two narrow places: custom `Error` subclasses (2.8.2 owns the senior shape, this lesson restates the syntax) and adapters wrapping a class-based SDK (Stripe's SDK in Unit 12.2, AWS SDK clients in Unit 13.3, the rare third-party that publishes a class-based API). The lesson teaches the minimum class surface a senior writes — the constructor, `readonly` and `private` fields, the `static` factory method as the async-init pattern — and refuses to teach inheritance hierarchies, abstract classes, mixins, decorators, and the full OOP vocabulary. The reader spots a class in third-party code and reads it; they don't author class hierarchies in application code.
- **The Temporal pivot is the senior 2026 call.** `Date`'s known problems — month math that mutates, timezone-as-second-class, no real duration type, the `getMonth` zero-indexing trap, the implicit-coercion-to-NaN failure mode — are the kind of paper cuts that aggregate into production bugs at the SaaS scale where users are in 200 timezones. Temporal is the spec answer, Stage 4 since March 2026 and unflagged in Node 26. The course targets Node 26 (May 2026 release), so Temporal is native on the runtime the student deploys to. The lesson teaches the pivot — `Date` for "what time is it right now" and for the `Date` objects third-party APIs hand back, Temporal for everything that involves arithmetic, comparison, formatting in a specific timezone, or duration math. Full treatment lands in Chapter 18.1; this lesson plants the seed and names the polyfill path for projects still on the Node 24 LTS line.
- **The chapter is short by design.** The TOC bullets are tight; the lessons are tight. The student leaves the chapter in 90 to 120 minutes of focused work, plus the quiz. The unit closes here — Unit 3 (HTTP and the browser platform) opens on the other side. The chapter's role is to clean up the practical surfaces that didn't belong inside any of the earlier chapters and to set up the forward references that the rest of the course will resolve.
- **Senior anchors for later units are seeded here.** JSON parsing as the wire-boundary input lands on Zod's `safeParse` in Unit 7.1 — the chapter names the seam. Custom `Error` subclasses land again everywhere the course handles failures (Unit 7.2, Unit 12.1, Unit 17.1). The class-based SDK adapter pattern lands in Unit 12.2 (the Stripe SDK), Unit 13.3 (S3/R2 clients), and Unit 23 (the Vercel AI SDK's `LanguageModel` class). Temporal lands in full in Chapter 18.1.
- **Naming for intent stays operating.** A parsed JSON value is named for what it represents post-narrow (`const config = parseConfigJson(raw)`, not `const data = JSON.parse(raw)`). A custom error is named for the failure (`InvoiceNotFoundError`, not `MyError`). A Temporal value is named for what it models (`const dueDate = Temporal.PlainDate.from(...)`, not `const t = ...`). The 2.2.3 reflex runs through every snippet.

This chapter ships small standalone snippets in `ScriptCoding`, a couple of `TypeCoding` blocks for the JSON-parse `unknown`-to-typed narrow, an `AnnotatedCode` for the class-based `Error` subclass and the SDK adapter, a `CodeVariants` for the `Date`-vs-Temporal contrast, and a `TabbedContent` for the JSON gotcha reference card. No diagrams beyond a one-shot Mermaid for the JSON serialization gotcha matrix if it earns its weight. The student finishes the chapter able to write `JSON.parse` and `JSON.stringify` correctly including the gotchas, recognize when a class earns its weight in 2026 application code, and reach for Temporal over `Date` for any arithmetic or timezone-aware operation.

The chapter ordering follows the layering. JSON comes first because it's the most-used surface and because the wire-boundary discipline (parse, then validate, then trust) is the lesson the rest of the course leans on. Classes come second as the narrow surface, with the explicit framing that the lesson is "where they show up" — not OOP. Temporal comes third because the pivot depends on the student understanding that the language is shifting under their feet (Stage 4, March 2026; unflagged, Node 26, May 2026), and the lesson points forward to Chapter 18.1 for the full treatment. The quiz closes the chapter and the unit.

## Lesson 2.9.1 — JSON at the wire boundary

Topics to cover:

- The chapter-opening senior question: every API the course will touch uses JSON — `fetch` bodies, Route Handler responses, Server Action payloads, Stripe webhooks, Trigger.dev job inputs, the `localStorage` value. What's the exact form the senior writes by reflex, what are the values that *don't* round-trip cleanly through JSON, and what's the seam at which a parsed value stops being `unknown` and starts being a typed domain value the rest of the codebase can trust. The lesson installs the language primitives and names the Zod seam (Unit 7.1) that closes the type story.
- **The two primitives**, named concretely with the senior-default signatures:
  - **`JSON.stringify(value, replacer?, space?)`** — serializes a JavaScript value to a JSON string. The senior reach: positional value-only call in the common case (`JSON.stringify(payload)`), `space` argument set to `2` only when humans will read the output (debug logs, dev-time config dumps, never production hot paths). The `replacer` is rare and lands when a specific value type needs custom serialization (a `BigInt`-aware replacer, redacting a secret field).
  - **`JSON.parse(text, reviver?)`** — parses a JSON string into a JavaScript value. The senior reach: positional text-only call; the `reviver` is rare. The return type is `any` from the lib types — the senior reflex is to *immediately* narrow with Zod (Unit 7.1) before reading any field. The course's `tsconfig.json` from 1.4.3 has `noImplicitAny` on, but `JSON.parse`'s `any` return slips through; the reflex is to type the receiving variable as `unknown` (`const raw: unknown = JSON.parse(text)`) and let the compiler force the narrow.
- **The serialization gotcha matrix** — the four value classes that don't round-trip cleanly through JSON, each with the senior fix:
  - **`undefined`** — `JSON.stringify({ a: 1, b: undefined })` produces `'{"a":1}'`. The `undefined` field is *dropped* silently. The senior watch-out: a payload that includes an optional field set to `undefined` reads as "this field doesn't exist" on the receiving side, not "this field exists with no value." The fix when the distinction matters is `null` (`{ a: 1, b: null }` round-trips as `{ a: 1, b: null }`); the senior default for the SaaS API contract is `null` for "present but empty" and absent for "not present." Names the JSON-array side too: `JSON.stringify([1, undefined, 3])` produces `'[1,null,3]'` — `undefined` becomes `null` in arrays because array indices can't be holes in JSON.
  - **`Date`** — `JSON.stringify(new Date())` produces an ISO 8601 string (`'2026-05-13T14:00:00.000Z'`) because the `Date.prototype.toJSON` method exists and `JSON.stringify` calls it. The asymmetry: `JSON.parse('"2026-05-13T14:00:00.000Z"')` produces the *string*, not a `Date` — the parse side has no way to know the string should be a date. The senior fix is to parse the JSON, then validate with Zod's `z.iso.datetime()` (Unit 7.1) and use Zod's transform or a manual `new Date(value)` after the schema narrows. Names the second asymmetry: `Date` round-trips through ISO strings on the wire, but Temporal types (Chapter 18.1) need explicit `toString`/`Temporal.X.from` because Temporal types intentionally don't auto-coerce. The student installs the reflex: dates on the wire are strings; the domain type is what the schema produces, not what `JSON.parse` returns.
  - **`BigInt`** — `JSON.stringify(1n)` throws `TypeError: Do not know how to serialize a BigInt`. The senior reach: BigInt is rare in 2026 SaaS application code (cents-as-integers from 2.1.3 fits in `Number`, IDs are strings or UUIDs, only the rare large-counter or cryptographic use case actually needs `BigInt`). When it does land, the senior pattern is to serialize as a *string* with a known shape and convert back on parse, using a Zod schema with a `transform` to do the narrowing:
    ```ts
    const userIdSchema = z.string().transform((s) => BigInt(s));
    ```
    Names the alternative: a `replacer` function that detects `bigint` and serializes to `{ $bigint: value.toString() }`, paired with a reviver. The senior call is the schema-driven form; the wrapping-object form earns its weight only when the JSON is consumed by code that doesn't share a schema.
  - **`NaN`, `Infinity`, `-Infinity`** — `JSON.stringify(NaN)` produces `'null'`, `JSON.stringify(Infinity)` produces `'null'`. The IEEE 754 non-finite values aren't representable in JSON. The senior reach is to never let `NaN`/`Infinity` reach the JSON boundary — the 2.1.3 cents-as-integers discipline plus Zod's `z.number().finite()` at the wire close the gap.
- **Other things that don't round-trip**, named in one paragraph so the student recognizes the failure mode:
  - **Functions**, **Symbols**, and **class instances**. `JSON.stringify(() => 1)` produces `undefined` (dropped in objects, `null` in arrays). A class instance loses its prototype on stringify; the parse side gets a plain object. The senior reflex: JSON is for data, not for behavior; if a domain value has methods, reconstruct it through a factory on the receiving side.
  - **`Map`, `Set`, `RegExp`**. None have a `toJSON` method; they serialize to `{}` (empty object). The senior reach when one needs to cross the JSON boundary is an explicit conversion (`Array.from(map.entries())` for `Map`, `Array.from(set)` for `Set`); the receiving side reconstructs from the array shape. Rare in application code that owns both ends; named for recognition.
  - **Cyclic references**. `JSON.stringify({ a: someObj })` where `someObj.b === someObj` throws `TypeError: Converting circular structure to JSON`. The senior reflex is to design the serialized shape to be a DAG (no cycles); the `structuredClone` primitive (2.1.1) handles cycles but isn't a JSON solution.
  - **Prototype methods**. The serialized output contains *own enumerable properties* only — inherited methods, getters defined on the prototype, and non-enumerable properties are dropped. Rare to bite in application code; named once.
- **`toJSON` as the extension hook** — the convention `JSON.stringify` honors. Any object whose `toJSON` method returns a value gets that returned value serialized instead of the object itself. `Date` uses it (returns the ISO string); the Temporal types use it (each returns its ISO 8601 form); the senior reach in application code is rare — a custom domain type that crosses the JSON boundary in a known shape. Named here so the student isn't surprised when an SDK's class produces unexpected JSON output.
- **The wire-boundary discipline** — the lesson's load-bearing decision. JSON crossing into the application is *untrusted* and the static type is `any`. The senior pattern is the four-step recipe:
  1. **Parse** the JSON into an `unknown` (type the receiving variable, not the call: `const raw: unknown = JSON.parse(text)`).
  2. **Validate** with Zod's `z.parse` or `z.safeParse` against a schema (full treatment in Unit 7.1). The schema narrows `unknown` to the domain type.
  3. **Reconstruct** any types that don't round-trip cleanly (dates from ISO strings via the schema's `transform`, BigInt from strings, etc.).
  4. **Trust** the result from this point onward in the function — the schema's output type is the type system's contract.
  The lesson names the recipe without teaching Zod's schema authoring (Unit 7.1 owns it). The student writes the `unknown` annotation by reflex and recognizes the seam where Zod earns its weight.
- **`JSON.stringify`'s third argument and the dev-vs-production split**:
  - `JSON.stringify(value, null, 2)` produces pretty-printed JSON. The senior reach is dev-only — pretty output costs CPU and bandwidth in production and is invisible to consumers. Use it for debug logs (Unit 20.1) and for human-readable config dumps; don't use it for API responses or wire payloads.
  - The `replacer` argument as a function. Two senior reaches when it lands: a redaction filter (`(key, value) => SENSITIVE_KEYS.has(key) ? '[REDACTED]' : value` for log lines), and the BigInt-to-string serializer named above. Rare overall; the schema-driven form is the senior default for the BigInt case.
- **`structuredClone` vs. JSON round-trip for deep copying**, named in one paragraph. The legacy reach `JSON.parse(JSON.stringify(obj))` for deep cloning was the pre-2022 pattern. The 2026 senior reach is `structuredClone(obj)` (2.1.1 introduced it) — handles cycles, preserves `Date`/`Map`/`Set`/`RegExp`/`ArrayBuffer`/typed arrays, faster than the JSON round-trip. The student never reaches for the JSON-roundtrip clone in 2026 code; the lesson names the legacy pattern for recognition.
- **JSONC, JSON5, and Trailing commas**, named in one paragraph. `tsconfig.json` and a handful of other config files use JSON-with-comments (JSONC); `JSON.parse` rejects them. The senior reach: the rare config-reader uses a JSONC-aware parser (`jsonc-parser` or the framework's bundled one); application data on the wire is strict JSON. The student doesn't enable JSONC for their own data formats — strict JSON is the wire contract.
- **The watch-outs a senior names**:
  - **`JSON.parse` returns `any`** by the lib types, which silently leaks unsafe access into the rest of the function. The reflex: annotate the receiving variable as `unknown` and let the compiler force the narrow.
  - **Stringify-then-compare** as equality. `JSON.stringify(a) === JSON.stringify(b)` produces wrong answers when object keys are in different insertion order, when one side has an `undefined` field the other doesn't, when one side has a `NaN`. The senior reach for deep equality is a small recursive walker or `dequal` (Unit 19.2 names it for tests); JSON-string comparison is the bug class.
  - **`JSON.stringify` is not sort-stable**. Object keys serialize in *insertion order* per the spec. Stringifying the same object after spread-assignment can produce different strings if the spread reorders keys. The senior watch-out: never use stringified JSON as a cache key without a stable-stringify helper.
  - **Logging an `Error` directly**. `JSON.stringify(new Error('oops'))` produces `'{}'` — `Error`'s enumerable own properties are empty by spec. The senior reach for log lines is to pluck `name`, `message`, `stack`, and `cause` explicitly (Chapter 2.8's `ensureError` plus the cause-chain walker), or let the structured logger (Unit 20.1) handle Error instances natively.
  - **Large-input parse and prototype pollution**. `JSON.parse` doesn't pollute `Object.prototype` directly (the `__proto__` key in JSON parses as a regular string property, not as a prototype assignment, since Node 17 / V8 9.6). The senior watch-out: any code path that then *assigns* parsed values into an object with `Object.assign` or spread can re-introduce pollution; Zod's schema enforces the shape and kills the pollution surface. Named for recognition; Unit 7.1 owns the schema discipline.

What this lesson does not cover:

- Zod schema authoring at depth — `z.object`, `z.array`, `z.string`, transforms, `safeParse` (Unit 7.1).
- The full Fetch API surface and response-body parsing (Unit 3.6).
- Server Action serialization across the network boundary (Unit 7.2).
- Streaming JSON parsers, NDJSON, or JSON Lines (out of scope for the course).
- The `Intl` formatters and locale-aware number/date formatting (Unit 18.2).
- `structuredClone`'s full mechanics — 2.1.1 owns the introduction; this lesson names the legacy contrast.
- Production-grade redaction for log lines (Unit 20.1).
- Custom binary serialization formats (out of scope).

Pedagogical approach:

Mechanics archetype with a load-bearing wire-boundary recipe. The lesson opens with the senior question — "you `JSON.parse` a webhook payload; what's the type of the result, and what's the first thing you do with it?" — and a small `TypeCoding` block where the student parses a payload, tries to read `result.amount`, and sees `any` flowing through. The student then annotates the receiver as `unknown` and watches the compiler refuse the unsafe access. The recipe-step-1 reflex is concrete.

A `TabbedContent` block organizes the four gotcha classes (`undefined`, `Date`, `BigInt`, `NaN`/`Infinity`) as a reference card. Each tab has the stringify output, the parse asymmetry, and the senior fix. The student leaves with a one-glance reference they return to when designing API payload shapes.

A `ScriptCoding` block has the student exercise the `undefined`-disappears behavior: they stringify `{ a: 1, b: undefined }`, observe `'{"a":1}'`, stringify `{ a: 1, b: null }`, observe the `null` survives, and stringify `[1, undefined, 3]`, observe `[1, null, 3]`. The gotcha lands in muscle memory. A second `ScriptCoding` exercises the `Date` round-trip: the student stringifies a `Date`, parses the resulting string, observes that the parsed value is a string, then narrows with a small inline `z.iso.datetime()` schema (with the schema given — Unit 7.1 hasn't landed yet) that transforms the string back to a `Date`. The seam where Zod earns its weight is concrete.

A `CodeVariants` block contrasts three approaches to a payload with a `BigInt`: tab 1 is the naive `JSON.stringify` that throws, tab 2 is a custom `replacer` that wraps the BigInt in `{ $bigint: value.toString() }`, tab 3 is the senior reach — the schema-driven approach where the BigInt is a string on the wire and Zod's `transform` converts on parse. The annotation on each tab names the trigger and the failure mode; tab 3 is the default.

A `CodeReview` exercise on a 20-line function that parses a webhook payload, reads fields off the parsed result directly without narrowing, stringifies an `Error` for logging, and uses `JSON.stringify(obj) === JSON.stringify(other)` as an equality check. The student leaves a comment per bug with the senior fix.

A `Buckets` exercise sorts ten values into "round-trips cleanly through JSON," "round-trips with caveats," or "doesn't round-trip" — a plain object with strings and numbers, a `Date`, a `Map`, a function, a `BigInt`, `null`, `undefined`, a cyclic object, a class instance, a `Set`. The student names the failure mode for each non-clean case.

Close with a small `PredictOutput` exercise: given five `JSON.stringify` calls on tricky values (an object with a `toJSON` method, an array with `undefined` and `Symbol()` elements, a nested object with a cyclic reference, etc.), predict each output (or the thrown error). The mechanics are airtight.

Estimated student time: 45 to 55 minutes.

## Lesson 2.9.2 — Classes, narrowly: where they still earn their weight

Topics to cover:

- The senior question: the course has written exclusively functions, types, and discriminated unions for eight chapters. Where do classes still show up in a 2026 SaaS codebase, what's the minimum class surface a senior writes, and why is everything else still a function. The lesson is deliberately narrow — three sites where classes earn their weight, two pieces of class syntax the reader needs to recognize, and the explicit refusal to teach inheritance hierarchies, abstract classes, mixins, decorators, and the rest of the OOP vocabulary that doesn't belong in this stack's idiom.
- **The default**, restated for the chapter's framing. The course writes functions, types, and discriminated unions because that's what idiomatic 2026 application code on this stack reaches for. React components are functions. Server Actions are async functions. Drizzle queries are function calls. Zod schemas are values. The function form composes with the rest of the language (currying, partial application, higher-order helpers) and with the type system (`Awaited`, `Parameters`, `ReturnType`, generic inference) in ways classes don't. The senior call is "function first; class only when the trigger lands."
- **Site one: custom `Error` subclasses**, restated from 2.8.2. The full senior shape lives in Chapter 2.8.2; this lesson restates the class syntax in isolation so the student sees a class declaration as the deliverable, not as a chapter-2.8 idiom:
  ```ts
  class InvoiceNotFoundError extends Error {
    readonly name = 'InvoiceNotFoundError' as const;
    constructor(readonly invoiceId: InvoiceId, options?: ErrorOptions) {
      super(`Invoice ${invoiceId} not found`, options);
    }
  }
  ```
  The lesson names the pieces — `class X extends Y`, the `constructor`, the `super` call, the `readonly` parameter property — and trusts 2.8.2 for the senior shape and discriminant pattern. The student writes one such class in the exercise, then moves on.
- **Site two: the adapter wrapping a class-based SDK**. The 2026 reality is that the JavaScript ecosystem still has class-based SDKs from libraries that predate the function-first idiom — Stripe (the `Stripe` class, full surface lands in Unit 12.2), AWS SDK clients (`S3Client`, `SQSClient`, lands in Unit 13), the Vercel AI SDK's `LanguageModel` instances (Unit 23), `EventEmitter`-based libraries, the rare ORM. The senior reach when a class-based SDK lands in the codebase is not to expose the class directly to the rest of the application; it's to wrap it in a small adapter module that returns a function-shaped API:
  ```ts
  // lib/stripe.ts
  import Stripe from 'stripe';

  const stripe = new Stripe(env.STRIPE_SECRET_KEY);

  export const createCheckoutSession = async (input: CheckoutInput): Promise<CheckoutSession> => {
    const session = await stripe.checkout.sessions.create({
      mode: 'subscription',
      // ...
    });
    return session;
  };
  ```
  The class instance is constructed once at module load (the singleton — 2.6.2's module-graph lesson cashes in here), and the rest of the application imports `createCheckoutSession`, not `stripe`. The adapter is what gives the application a function-first surface even when the SDK underneath is class-based. The trigger: any third-party class whose API is awkward to reach through directly (its methods take options the application doesn't care about, its types are loose where the application's contract is tight, the SDK changes API shape across versions). Named in one paragraph; the full Stripe adapter lands in Unit 12.2.
- **Site three: the rare custom domain class with state and methods**, named for honesty. There's a narrow tier of application code where a class is the right reach — a `RateLimiter` instance carrying token-bucket state (Unit 15.3 uses one, though Upstash's library wraps it), a `RetryPolicy` with a `next()` method that returns the next delay, a `Connection`-style abstraction over a long-lived resource. The trigger: an object that has *identity* (two instances are not interchangeable), *state* that mutates over its lifetime, and a *bounded set of methods* that operate on that state. Most application code has none of these three; the senior reach is a function with a closure over the state when one does. The class form earns its weight when the state is large enough that the closure becomes awkward or when the consumer wants to discriminate on `instanceof`. The lesson names the trigger and moves on — most students won't write a custom class outside of an `Error` subclass for years, and that's correct.
- **The minimum class surface a senior writes**:
  - **The `class X { ... }` declaration**. The body holds field declarations, the constructor, and method declarations. No `;` between members (whitespace-significant only in the trivial sense). Named declarations only — class *expressions* (`const X = class { ... }`) are out of scope; the named declaration is the senior default.
  - **The `constructor`**. Runs on `new X(args)`. Initializes `this` from the parameters. The lesson teaches the parameter-property form (`constructor(readonly id: string) { super(); }`) as the senior shorthand — declares the field and assigns it in one line. The watch-out, named once: the parameter-property form requires TypeScript and is erased to a regular assignment in the compiled output. Native JavaScript classes don't support the syntax. Since the course compiles every TypeScript file, this is a non-issue; named for honesty.
  - **Field declarations**. `readonly id: string`, `private state: 'idle' | 'running'`. The `readonly` modifier is the senior default for "this field is set once in the constructor and never reassigned." The `private` modifier (TypeScript) and the `#field` form (JavaScript) both produce private fields; the course writes `#field` for *runtime* privacy and `readonly` for compile-time-only constraints. The lesson names the difference: `private` is a TypeScript-only check that erases at compile time and doesn't prevent runtime access via bracket notation; `#field` is a JavaScript-native private slot that's invisible to outside code at runtime. The senior reach in 2026 is `#field` for fields that must be truly inaccessible (a private auth token in an SDK adapter), `readonly` for fields that are public but immutable.
  - **Methods**. `methodName(args) { ... }` inside the class body. Bound to the instance; `this` refers to the instance. The watch-out, named with the canonical bite: a method passed as a callback (`array.forEach(this.handleItem)`) loses its `this` binding because `forEach` calls the function in a context without an instance. The senior fix in 2026 is the **arrow-function class field** form (`handleItem = (item) => { ... }`) — declares the method as a field initialized to an arrow function whose `this` is lexically captured at construction time. The arrow-field form is what makes a class method usable as an event handler without `.bind(this)` ceremony. Named once with the diff.
  - **`static` methods and the static factory pattern**. A `static` method on the class doesn't take `this` (it takes the class itself as `this`). The senior reach: the *static factory method* for the async-init pattern Chapter 2.8.2 named in passing. A class constructor can't `await`; an async initialization moves to a static method that does the async work and returns a constructed instance:
    ```ts
    class DatabaseClient {
      private constructor(private readonly conn: Connection) {}
      static async create(url: string): Promise<DatabaseClient> {
        const conn = await connect(url);
        return new DatabaseClient(conn);
      }
    }
    ```
    The `private constructor` plus the `static async create` pattern names "the only way to construct one of these is through the factory, which does the async work." Rare in application code; named because it lands in adapters wrapping async-init SDKs.
  - **`extends`** for inheritance. Named with the narrow trigger: subclassing `Error` (the chapter's primary class site), subclassing `EventEmitter` from Node's `events` module when wrapping a Node-style event API, subclassing a class the SDK author intended to be subclassed. The senior call is to *not* invent class hierarchies in application code; composition through functions and discriminated unions is the idiom. The lesson names `super(args)` (calls the parent constructor) and `super.method(args)` (calls the parent method) once each for recognition.
- **What the lesson refuses to teach** — the explicit "we are not teaching OOP" cut, named so the student knows the omission is deliberate:
  - **Inheritance hierarchies** beyond `extends Error` and the rare SDK base class. The course doesn't write `class AbstractService extends BaseDomain extends ...`; the depth-of-three hierarchy is the bug class composition fixes.
  - **Abstract classes** (TypeScript `abstract class` and `abstract method`). Out of scope; the discriminated union plus the function-based interface is the senior 2026 reach for "this is a contract multiple implementations must honor."
  - **Mixins** and the `Object.assign` mixin pattern. Out of scope; composition through functions handles the use case.
  - **Decorators** (TypeScript Stage 3 decorators, the legacy `experimentalDecorators` flag). Out of scope. The course's tsconfig doesn't enable them. The senior reach in 2026 application code is higher-order functions; decorators land in framework code the student consumes (the next-intl router, the Drizzle schema definitions through a different mechanism), not in application code the student writes.
  - **`get`/`set` accessors**. Out of scope. The senior reach for "compute on read" is a method; for "validate on write" is a function that returns a new value. The mutable property with side-effecting setters is a bug class.
  - **`Symbol.iterator` and custom iteration protocols**. Out of scope for this lesson; 2.3.5 covered the consumer side.
  - **`Reflect`, `Proxy`, and meta-programming**. Out of scope for the course. The student recognizes the names in library source (TanStack, Drizzle internals) and doesn't reach for them in application code.
- **Class instances and `instanceof`**, restated from 2.8.2. The class declaration installs a constructor function on the runtime and a prototype on its instances; `instanceof` checks the prototype chain. The senior reach when the class is in scope is `instanceof X`; when the prototype chain is unreliable (cross-realm, structured-clone boundaries), the fallback is `error.name` or a duck-typed check (2.8.2 owns the gotcha).
- **The watch-outs a senior names**:
  - **`this` rebinding on method passing**. The single most common class bug. The fix is the arrow-function class field form named above.
  - **Constructors that do async work**. Constructors can't `await`; the partially-constructed object is in a broken state. The fix is the static factory method.
  - **Inheritance for code reuse**. The "DRY by extending a base class" reflex is a 2010-era pattern that doesn't hold up; composition through small functions or a discriminated-union-typed config object scales better. Named once.
  - **Singleton-by-class-pattern** (a class with a private constructor and a `static getInstance()` method). The 2026 senior reach is the module-level singleton — declare the value at module scope, export the functions that close over it. The class-based singleton adds ceremony with no win.
  - **Classes for "data shapes"**. A class with no methods, just fields, is a `type` or `interface` written incorrectly. The senior reach is the type alias or interface; the class adds runtime cost (the constructor function, the prototype) for no gain.

What this lesson does not cover:

## Lesson 2.9.3 — `Date`'s problems and the Temporal pivot

Topics to cover:

- The senior question: `Date` has been the JavaScript date type since 1995, it has known design problems that have aggregated into production bugs across every SaaS that's ever shipped, and in 2026 the language finally fixed it. Temporal reached TC39 Stage 4 in March 2026 and shipped unflagged in Node 26 (May 2026 release) — the runtime the course targets for deploys. What's the senior 2026 default, when is `Date` still acceptable, and what's the path for projects still pinned to the Node 24 LTS line? The lesson installs the pivot and trusts Chapter 18.1 for the full timezone/DST/profile-tz/`timestamptz` treatment.
- **`Date`'s known problems**, named concretely so the student knows *what* Temporal fixes:
  - **The constructor signature is overloaded and confusing.** `new Date()` is "now." `new Date(timestamp)` takes a millisecond Unix epoch. `new Date(isoString)` parses an ISO 8601 string (with a long list of implementation-specific edge cases that historically diverged across browsers, though modern engines have converged on the spec). `new Date(year, month, day, ...)` takes individual components — and `month` is *zero-indexed* (January is `0`, December is `11`), which is the bug class every junior writes once and every senior has caught in code review three times. The senior reach: prefer `Date.UTC(year, month, ...)` for the components form when the input is in components, and treat any positive integer that doesn't look like an epoch as a likely bug.
  - **Mutability.** `date.setMonth(date.getMonth() + 1)` mutates the date in place. The senior bug class: a function that takes a `Date` parameter and "adjusts" it in place, accidentally mutating the caller's value. The reflex in 2026 has been to clone (`new Date(date.getTime())`) or to spread through `structuredClone` (2.1.1). Temporal makes every type *immutable* by design — every operation returns a new instance.
  - **Timezone as a second-class citizen.** A `Date` is internally a Unix timestamp (UTC). Every "local" method (`getFullYear()`, `getMonth()`, `getHours()`) returns the value in the *runtime's* timezone — which on a server is whatever `process.env.TZ` is set to, typically UTC, but the production-vs-development divergence is the bug class that bites in CI when the developer's machine is in Pacific time and the test passes locally but fails in UTC-set CI. The senior reach in 2026 production code is to *never* call the local methods on a `Date` for business logic — use `getUTC*` for the rare case where the field is needed in a fixed timezone, or use Temporal where the timezone is named explicitly in the type.
  - **No real duration type.** Adding "one month" to a `Date` is hand-rolled month math with edge cases at month-end (January 31 + 1 month = ?). The conventional `date.setMonth(date.getMonth() + 1)` produces March 3 in non-leap years (because the runtime clamps the day). Date arithmetic is the bug class Temporal fixes with `Temporal.Duration` and `add`/`subtract` methods that take explicit `largestUnit` and `smallestUnit` options and clamp explicitly.
  - **The `Invalid Date` sentinel.** `new Date('not a date')` produces a `Date` object whose `getTime()` returns `NaN` — but the value still passes the `instanceof Date` check. Every method then returns `NaN` or `'Invalid Date'` and the function silently fails. The senior reach: `isNaN(date.getTime())` after construction to validate, or never construct from untrusted input — use Zod's `z.iso.datetime()` to validate the string first. Temporal's `from()` methods throw on invalid input instead of returning a broken instance, which is the design fix.
  - **`Date.parse` is implementation-defined for non-ISO strings.** The 2026 senior reach is to never call `Date.parse` directly — accept ISO 8601 strings on the wire (validated by Zod) and parse them with `new Date(isoString)` only after the schema confirms the shape. For anything else (a user-typed date in the application's input format), use a parser the application owns.
- **What Temporal is, in one paragraph.** A new top-level `Temporal` namespace on the runtime with a set of immutable types: `Temporal.Instant` (a Unix timestamp), `Temporal.ZonedDateTime` (an instant plus a named IANA timezone), `Temporal.PlainDate` (a date with no time, no timezone — a calendar date), `Temporal.PlainTime` (a time of day, no date, no timezone), `Temporal.PlainDateTime` (date + time, no timezone — the "wall clock" type), `Temporal.Duration` (an explicit duration with `years`/`months`/`weeks`/`days`/`hours`/`minutes`/`seconds`/`milliseconds`/`microseconds`/`nanoseconds`), `Temporal.TimeZone`, `Temporal.Calendar`, and a `Temporal.Now` namespace for "what time is it." Each type has explicit methods for construction (`Temporal.PlainDate.from('2026-05-13')`), arithmetic (`.add({ months: 1 })`), comparison (`.equals()`, `.since()`, `.until()`), and serialization (`.toString()` produces an ISO 8601 string, `toJSON()` does the same). The types are immutable; every operation returns a new instance. The senior 2026 reach for anything that involves arithmetic, comparison, formatting in a specific timezone, or duration math is Temporal.
- **The senior 2026 default**:
  - **Use `Date`** only for two things: "what time is it right now" at the boundary where you hand off to a library or API that takes a `Date` (the `new Date()` constructor as the "now" primitive); receiving a `Date` from a third party that hands one back and converting to Temporal immediately (`Temporal.Instant.fromEpochMilliseconds(date.getTime())`). Don't do arithmetic on `Date` instances. Don't format `Date` instances directly for display (use `Intl.DateTimeFormat` from Unit 18.2 with a Temporal value or with the boundary-`Date`).
  - **Use Temporal** for everything else. Domain values are Temporal types — a `dueDate` is `Temporal.PlainDate`, an `expiresAt` is `Temporal.ZonedDateTime`, a `billingInterval` is `Temporal.Duration`. Database columns deserialize through a Drizzle codec (Unit 6.2) into the right Temporal type; the application code never sees a `Date`. The full pattern lands in Chapter 18.1; the lesson installs the reflex.
- **The runtime call**, named with the version cut:
  - **Node 26 (May 2026)** ships Temporal unflagged. The student's project targets Node 26 for deploys (the course's tsconfig and engines field from Chapter 1.4 set this). No polyfill needed. `Temporal` is a global; the code reads `Temporal.PlainDate.from('2026-05-13')` directly.
  - **Node 24 LTS (the production line still in use through October 2026 when Node 26 promotes to LTS)** doesn't ship Temporal unflagged. Two polyfills exist: `temporal-polyfill` (FullCalendar, ~20KB compressed, the modern reach in 2026, near-perfect spec compliance), and `@js-temporal/polyfill` (the TC39 champion's polyfill, full spec, larger bundle). The senior call in mid-2026 if the project is on Node 24 LTS: use `temporal-polyfill` for the smaller bundle and equivalent semantics. Import as `import { Temporal } from 'temporal-polyfill'` and the rest of the code is identical to native Temporal. The student installs the polyfill once in the project's `db.ts` or a small `lib/temporal.ts` re-export and imports `Temporal` from that path everywhere else; when the project upgrades to Node 26, the re-export switches to the global and no other file changes.
  - **The browser line.** Chrome 144, Firefox 139, and Edge 144 ship Temporal natively. Safari blocks the implementation as of May 2026, which means real-world cross-browser support is roughly 70% of users. The senior reach for SSR-rendered apps (the course's Next.js 16 default) is to do all Temporal work on the server where Node 26 has it natively and serialize ISO 8601 strings to the client; the client renders the strings through `Intl.DateTimeFormat` (Unit 18.2). The polyfill ships to the client only when the application needs client-side date arithmetic on Safari, which the course's chapter-end projects don't.
- **The minimum Temporal surface the chapter installs** — the lesson teaches enough Temporal that the student recognizes the types and can write a small "due date plus N months" computation. Full surface lands in 18.1:
  - **`Temporal.Now.plainDateISO()`** — today's date in the runtime's calendar. The senior reach for "what's today" in a date-only context.
  - **`Temporal.Now.zonedDateTimeISO(timeZone)`** — now in a specified timezone. The senior reach for "what time is it for the user" where the timezone is the user's profile timezone (Chapter 18.1).
  - **`Temporal.PlainDate.from(iso)`** — parse an ISO 8601 date string into a `PlainDate`. Throws on invalid input.
  - **`.add({ months: 1 })` / `.subtract({ days: 7 })`** — immutable arithmetic. Returns a new instance. The senior reach: every "X plus N months" lives here, never on `Date`.
  - **`.toString()`** — ISO 8601 serialization for the wire and the database.
  - **`Intl.DateTimeFormat` interop** — Temporal types implement `toJSON` and integrate with `Intl.DateTimeFormat`'s `format()` for display. Named once; Unit 18.2 owns the locale-aware formatting story.
- **The watch-outs a senior names**:
  - **`Temporal.X.from(string)` throws on invalid input.** The senior reflex: validate the string with Zod first (Unit 7.1) before passing to Temporal. The schema produces a known-good string and Temporal's `from` succeeds.
  - **Don't mix `Date` and Temporal arithmetic in the same code path.** The conversion cost is real (every conversion serializes through an ISO string or an epoch millisecond); pick a type and stay in it. The senior reach is "Temporal at the domain boundary; `Date` only at the third-party seam."
  - **Drizzle and Postgres `timestamptz`** — Chapter 18.1 owns the full storage story. Named here once: Postgres stores everything in UTC as `timestamptz`; the Drizzle codec converts to `Temporal.Instant` on read. The student doesn't write the codec; the project chapters set it up.
  - **The polyfill bundle.** The 2026 senior reach is to gate the polyfill import behind the runtime check (`typeof globalThis.Temporal === 'undefined'`) when the project supports both Node 24 LTS and Node 26 — production runs on Node 26 (no polyfill loaded), but the build target for the Node 24 LTS path or for older browsers includes the polyfill. Named in one paragraph; the project chapters land the configuration.
  - **`Temporal.ZonedDateTime` vs. `Temporal.Instant` vs. `Temporal.PlainDateTime`** — the three "datetime" types are not interchangeable and each models a different concept. An `Instant` is "a point in real time" with no timezone (a Unix epoch). A `ZonedDateTime` is "a point in real time plus the timezone we're going to display it in" (an `Instant` plus an IANA timezone). A `PlainDateTime` is a "wall clock" datetime with no timezone — a meeting scheduled for 9 AM on May 13 in *whatever* timezone the reader is in. Picking the wrong type is the senior bug class Temporal fixes by *forcing* the choice. Chapter 18.1 owns the full decision; named once here so the student knows the choice exists.
- **What stays unchanged**, named for clarity. The wire format is still ISO 8601 strings. The database column is still `timestamptz`. The JSON payload still serializes dates as strings (and Temporal types serialize through their `toJSON` method to the same ISO 8601 form). What changes is *the in-memory type the application uses to do arithmetic, compare, and format*. The migration from a `Date`-heavy codebase to a Temporal-heavy one is mechanical: replace the in-memory type, leave the boundaries alone.

What this lesson does not cover:

- The full Temporal API surface — `Temporal.RoundOption`, `Temporal.DifferenceOptions`, calendar systems beyond the ISO calendar, the full `with()` and `withFields()` mutation surface (Chapter 18.1).
- UTC storage in Postgres `timestamptz` and the Drizzle codec at depth (Chapter 18.1).
- The user's profile timezone and the per-request locale derivation (Chapter 18.1, Chapter 18.2).
- DST transitions and recurring-job edge cases (Chapter 18.1).
- `Intl.DateTimeFormat`, `Intl.RelativeTimeFormat`, `Intl.NumberFormat` and locale-aware formatting (Chapter 18.2).
- `next-intl` and the 2026 i18n routing surface (Chapter 18.2).
- `date-fns`, `dayjs`, `luxon`, `moment` — the legacy date libraries the senior reach replaced. Named once and dismissed.
- Calendar systems beyond ISO (Buddhist, Islamic, Japanese — Temporal supports them, the course doesn't teach them).

Pedagogical approach:

Decision archetype with a Reference beat for the Temporal surface. The lesson opens with the senior question — "you need to compute the expiration date for a 30-day trial. What's the form?" — and a `CodeVariants` block with two tabs. The annotation names the senior call — tab 2 is the 2026 default — and the lesson cements it.

A small `ScriptCoding` block has the student walk through `Date`'s pain points in code they can run: construct a `new Date(2026, 1, 31)` (which the runtime renders as March 3 because `setMonth` clamped), observe the surprise, then construct `Temporal.PlainDate.from('2026-02-31')` and observe the `RangeError`. An `AnnotatedCode` block walks a small "trial expiration" function written with Temporal — `Temporal.Now.plainDateISO()`, `.add({ days: 30 })`, `.toString()` for the wire. The annotations name the immutable arithmetic, the `from`/`toString` symmetry, the ISO 8601 wire format. The student sees the surface and knows it's small.

A `TabbedContent` block organizes the runtime-cut decision: tab 1 is "I'm targeting Node 26 (the course default) — Temporal is global, no polyfill"; tab 2 is "I'm on Node 24 LTS — install `temporal-polyfill`, import from the package, identical code"; tab 3 is "I'm doing client-side date arithmetic in a Safari-supported app — the polyfill ships to the client, ~20KB compressed." Each tab has the setup command, the import line, and the canonical site. The student picks the right path for their project and moves on.

A `CodeReview` exercise on a 25-line function that uses `new Date()` for arithmetic, calls `setMonth` for an "add a month" operation, and parses a user-typed date with `new Date(userInput)`. The student leaves comments naming the three bugs (mutation, month-end clamping, implementation-defined parse) and proposes the Temporal-based rewrite.

A `Buckets` exercise sorts ten operations into "stay on `Date`" or "pivot to Temporal" — `new Date()` to capture "now" for a log line, a duration computation between two dates, formatting a date for the user's locale, parsing an ISO string from an API, comparing two timestamps for equality, adding one month to a billing date, getting the day of the week, converting a third-party SDK's returned `Date` to a domain value, etc. The decision discipline is the deliverable.

## Lesson 2.9.4 — Quizz
