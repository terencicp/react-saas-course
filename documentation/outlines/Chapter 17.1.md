# Chapter 17.1 — Error discipline

## Chapter framing

Chapter 17.1 opens Unit 17 — the pre-launch audit pass — by naming the two error commitments that have been operating quietly since Unit 2 and turning them into explicit, named rules every later seam audits against. The student arrives with the throw-vs-return decision from 2.8.1, the `unknown`-narrow and `Error.cause` from 2.8.2, the canonical `Result` shape and the throw-at-framework-edge rule from 7.2.3, the `authedAction(role, schema, fn)` wrapper from 10.2.2, the webhook signature-verify-then-dedup transaction from 12.1.1–12.1.2, the rate-limiter fail-open on auth from 15.3.3, and Sentry capture plus structured logs foreshadowed for 20.1. What's missing is a single name for two patterns that already run through the codebase. **Errors fail closed** — any check that gates access (`requireRole`, paywall, tenancy filter, signature verify) refuses by default, and an exception inside the check is treated as a refusal, not an allow. **User message and operator message are different artifacts** — what reaches the user is generic, operator-safe, and free of internal IDs / stack traces / DB constraint names; the diagnostic detail goes to Sentry, the structured log, and the audit log. Lessons 17.1.1 and 17.1.2 install each commitment as a senior rule with the failure mode it prevents. Lesson 17.1.3 walks the six existing seams — `authedAction`, `authedRoute`, page checks, the webhook receiver, the rate limiter, and the `error.tsx` boundaries — and points at where each commitment lands or should land. Lesson 17.1.4 is the chapter quiz.

Threads that run through every lesson. The chapter is an **audit pass, not new code** — the patterns exist; the work is naming them, finding the seams that drift, and committing the discipline. The two commitments are **load-bearing, not stylistic** — the fail-closed rule is the difference between "the authz check threw and the user got the resource" and "the check threw and the user got 403"; the message-split rule is the difference between "the user sees `duplicate key value violates unique constraint invoices_org_id_slug_key`" and the user sees "That slug is already taken." Both are **structural defenses, not cleanup conventions** — they live in helpers and wrappers (`requireRole`, `safeAction`, the `Result` mapper, the Sentry `beforeSend` scrubber, the 500-fallback in `error.tsx`) where forgetting them requires actively reaching past the helper. The student leaves the chapter able to look at any seam in the codebase and answer two questions: does this fail closed when something throws inside the check, and is the message the user sees safe to read aloud at a customer support call? The seam-by-seam walk in 17.1.3 cashes that vocabulary in on every code path the course has already shipped.

## Lesson 17.1.1 — Refuse by default

Every gate that controls access — authorization, tenancy, paywall, signature verify — treats an exception inside the check as a refusal, with the throw-and-catch structure that makes fail-open hard to write.

Topics to cover:

- **The senior question.** An `authedAction` runs `requireRole('admin')` and proceeds with the mutation. The role check reads the session, queries the membership table, compares the role, returns `'admin' | 'member' | 'owner'`. What happens when the membership query throws — a Postgres connection blip, an unexpected `null`, a TypeScript-impossible state the runtime hit anyway? Two architectures answer differently. **Fail-open** lets the action continue because the exception bubbled past the check; the user's role was "unknown" and the system defaulted to "probably fine." **Fail-closed** treats the exception as "the check did not succeed" and refuses. The lesson installs fail-closed as the load-bearing commitment for every authorization-shaped check in the codebase, names the failure modes when it slips, and walks the structural form that makes it the only path the runtime can take.
- **What "fail closed" means precisely.** A check that gates access has three legal outcomes — *allow*, *deny*, *check itself failed*. Fail-closed collapses the third into the second: an exception thrown inside the check is treated as a refusal, not as "we don't know, let it through." The action body never runs if `requireRole` threw; the user sees a 403-shaped response; the operator sees the original error in the log. The opposite — fail-open — is the bug where the exception escapes the check, the action proceeds, and the unauthorized user gets the resource because the system gave up on knowing.
- **The classes of checks the rule covers.** **(1) Authorization gates** — `requireRole`, `requireOrgUser`, `authedAction`, `authedRoute`; the membership lookup, the role comparison, the active-org resolution from the session (10.1.1). **(2) Tenancy filters** — every query that asserts `orgId = $1`; the `tenantDb(orgId)` helper (10.1.2) and the visibility helper (`active()`, `archived()` from 11.2.2); the join-time `activeFilter(table)` predicate. **(3) Paywall and entitlement checks** — the plan lookup, the feature-flag read, the seat-count compare from billing (12.2). **(4) Signature verification** — the HMAC compare on every webhook (12.1.1); the constant-time compare from Web Crypto (3.7.1). **(5) Idempotency claims** — the `INSERT ... ON CONFLICT DO NOTHING` on `processed_events` (12.1.2); a thrown dedup check refuses the work rather than running it twice. **(6) CSRF and origin checks** if/when the framework's default doesn't cover the call shape (Server Actions ship CSRF protection out of the box — named once, the rule still applies anywhere a hand-rolled check lands). The lesson names each class and points at the chapter that owns the primitive.
- **The decision question every check answers.** *If this check throws — Postgres dropped the connection, the JWT library threw on a malformed claim, the membership row was deleted between the session lookup and the role read, an `assertNever` fired — does the user get the resource or get refused?* The senior answer is always refuse. The structural form makes the wrong answer hard to write.
- **The failure modes when fail-closed slips.** **(a)** A naked `try { await requireRole('admin'); } catch { /* log and continue */ }` lets the action run when the role check fails — the canonical fail-open bug; named so the student recognizes it on review. **(b)** A `requireRole` that returns `boolean` with `false` for both "not admin" and "check threw" — the action body branches on `if (!isAdmin) return err(...)`, the exception escaped the helper, the action proceeded; the fix is to never let the check return on exception, either it returns the role or it throws. **(c)** A tenancy filter implemented as `orgId === input.orgId || isSuperAdmin(user)` where `isSuperAdmin` reads from a misconfigured table and throws — the `||` short-circuits to truthy on rejection in some flawed patterns; the fix is one filter, no `||` carve-outs, the carve-out lives in a different code path with its own gate. **(d)** A signature verify that returns `false` on both bad signature and HMAC library exception — the route handler can't distinguish; the senior reach is to throw on exception (treated as 500 by the framework) and return 401 only on a real signature mismatch. **(e)** A paywall check that catches the Stripe API timeout and lets the request through to "not block the user" — the fail-open default that lets free-tier users hit paid endpoints; the fix is fail-closed with a circuit-breaker fallback that explicitly serves a cached entitlement, not a silent allow.
- **The structural shape — the check throws on its own failure, the wrapper catches and converts to refusal.** The senior 2026 form. `requireRole(role: Role): Role` (or `requireOrgUser(): { user, orgId, role }`) — declared to throw, never to return a "we don't know" sentinel. Callers either consume the return (and know the check succeeded) or wrap the call in `authedAction`'s machinery. `authedAction` catches the throw in one place — the wrapper body — and converts it to the canonical `Result`'s failure branch (`{ ok: false, error: { code: 'unauthorized', userMessage: '...' } }`) or to the route handler's 401/403 response. The student writes neither the try/catch at the call site nor the explicit refusal — both live in the wrapper. The carve-out from Architectural Principle #5 (auth wrappers are the named exception to "don't invent parallel routing") cashes in here as defense-in-depth.
- **The `Result` failure shape vs. the throw — restated where it lands here.** From 7.2.3: throw at the framework edge for *unexpected* failures (DB down, programming errors, the framework's `notFound`/`redirect` control-flow exceptions). Return `Result` for *expected* failures the caller branches on. Fail-closed applies to both — an unexpected throw inside a check gets caught by the framework's nearest `error.tsx` (5.3.3) and the user sees a generic error, *not* the resource; an expected refusal returns `{ ok: false, error: { code: 'unauthorized' } }`. The student sees both branches reach the same outcome: the user does not get the resource.
- **The "every doubt is a deny" mental model.** If the check cannot prove the request is allowed, it denies. The verb "prove" matters: a check that read stale data, or read partial data, or read `null` for a column it should never be `null`, or hit a code path the developer didn't write a branch for — all of those are "could not prove." The fail-closed reflex is the senior reading of the catch site: the binding is `unknown`, the check did not run to completion, the answer is no.
- **The Server-Component / page-boundary shape.** Pages and layouts use `requireOrgUser()` at the top, the helper throws on no-session / no-active-org. The framework's `error.tsx` catches the throw and renders the generic fallback. `notFound()` and `redirect()` (5.1.4) are *not* errors — they are control-flow primitives the framework owns. The lesson surfaces the distinction: a thrown `Error` from a layout is fail-closed; a thrown `notFound()` is the framework's 404 plumbing. The fail-closed rule applies to genuine exceptions; the `notFound`/`redirect` primitives are framework-blessed control flow that the rule passes through.
- **The webhook seam — fail-closed before dedup, before business logic.** The signature verify (12.1.1) is the gate. A throw inside HMAC compare (malformed `Stripe-Signature` header, an unparseable timestamp) returns 400 — refuse. A throw inside the dedup `INSERT ... ON CONFLICT` returns 500 — refuse, the provider retries. A throw inside the business logic returns 500 — refuse, the provider retries, idempotency from 12.1.4 makes the retry safe. The whole receiver is fail-closed; the lesson points at each layer.
- **The rate-limiter seam — the named exception to fail-closed.** From 15.3.3: the auth-path limiter is **fail-open** because a Redis outage locking every user out of their account is worse than the brief abuse window. The lesson surfaces the exception and the reasoning so the student doesn't paint fail-closed as universal — the rule is fail-closed by default, the carve-out is documented in `safeLimit` with a logged event and an alert. Other high-stakes limiters (admin actions, billing webhooks the customer cannot retry) may flip the default to fail-closed; the policy lives in one helper, not at every call site. The senior framing: fail-closed is the default discipline, fail-open is a deliberate carve-out with a written reason.
- **Idempotency and fail-closed compose.** A retried operation can fail closed safely because the retry is idempotent (12.1.4). The webhook receiver throws on a transient DB error, refuses the work, the provider retries, the dedup constraint catches the duplicate, the business work runs once. The lesson surfaces the composition so the student sees fail-closed and idempotency as paired primitives — refuse aggressively, retry safely.
- **The Server Action wrapper's responsibility — one place to lint.** `authedAction(role, schema, fn)` wraps every Server Action; its body is the only place the catch-and-refuse machinery lives. The wrapper does (paraphrased): parse with the schema → run `requireOrgUser` → run `roleAtLeast(role)` → call `fn(ctx, input)` → return the `Result`. Every step is fail-closed: parse failure returns `{ ok: false, code: 'invalid_input' }`; session miss throws (caught by `error.tsx`); role too low returns `{ ok: false, code: 'unauthorized' }`; `fn` throws → caught by `error.tsx`; `fn` returns `Result.err` → returned as-is. The lesson reads the wrapper's source as the canonical fail-closed implementation and names "one place to lint" as the architectural reason.
- **The "trust the wrapper, don't bypass it" rule.** A Server Action that doesn't go through `authedAction` is the bug. A route handler that doesn't go through `authedRoute` (10.2.3) is the bug. The lesson names the audit step: grep for `'use server'` in files that don't import `authedAction`; grep for `export const POST = async` in `route.ts` files that don't import `authedRoute`. Each hit is reviewed; legitimate exceptions (the public sign-up route, the webhook receiver with its own signature verify) are named, the rest are migrated. The rule is structural: when the wrappers run, fail-closed is the only path.
- **What does *not* fail closed — read-only display paths with sensible defaults.** A feature flag check that defaults to `off` on Redis miss is fail-closed (the feature stays disabled). A theme preference that defaults to `'system'` on read failure is *neither* — there's no security boundary; the default is the senior product call. The lesson names the boundary: the rule applies to access decisions (who can read, who can write, what tenancy enforces); product defaults outside the security boundary are a different decision the student should not conflate.
- **Watch-outs.** The empty-catch around an action body silently swallows authorization failures; log-and-continue is fail-open dressed as discipline (log *and* return the refusal); treating `notFound()` as an error to catch is a category error — it's framework control-flow, re-throw; a `requireRole` that returns `null` on exception is fail-open with extra steps; a paywall that fails open "for UX" lets free-tier users hit paid endpoints during outages; copy-pasting `authedAction` into a "lighter" wrapper creates a parallel code path that drifts.

What this lesson does not cover:

- The user-message vs. operator-message split — Lesson 17.1.2.
- The seam-by-seam audit walk where each commitment lands — Lesson 17.1.3.
- Security headers (CSP, HSTS) — Chapter 17.2.1.
- Rate limiting at depth — Chapter 15.3 owns the wiring; this lesson names the fail-open carve-out.
- The audit-logs schema and append-only enforcement — Chapter 10.2.5 introduced; Chapter 17.2.3 revisits.
- Cookie consent and GDPR — Chapter 17.2.4–17.2.5.
- The audit project — Chapter 17.3.
- Sentry capture wiring — Chapter 20.1.1.

## Lesson 17.1.2 — Two audiences, two messages

Every error splits into a sanitized user string (no IDs, no stacks, no constraint names) and a fully-instrumented operator record (cause chain, ctx, redacted input), diverging at the wrapper and never at the UI.

Topics to cover:

- **The senior question.** A Server Action mutates a row, the database raises `duplicate key value violates unique constraint "invoices_org_id_slug_key"`. The action's catch block has the error. Two audiences are downstream. The *user* will see whatever the form renders next — they need a string that tells them what to do. The *operator* (the on-call engineer, the support rep reading Sentry, the auditor reading the log) needs the original error, the input that caused it, the stack, the cause chain, the tenant context. What reaches each audience, and what's the structural form that makes the wrong thing never reach the user? The lesson installs the **two-message rule**: every error is two artifacts — a sanitized user message and a fully-instrumented operator record — and they diverge at the seam (the action wrapper, the route handler wrapper, the page error boundary), never at the UI.
- **The two audiences, named precisely.** **The user** is the human reading the rendered string in the form or the error page. They want a sentence in their language (i18n lands in Unit 18; the principle is the same) that names what happened in product terms and tells them what to try. They cannot act on a constraint name, a stack trace, an internal ID, or "an error occurred (code: 0x4a)." **The operator** is the engineer (or the LLM agent the senior delegates to) reading Sentry, the structured log, the audit log. They want the original error, the stack, the cause chain, the request id, the user id, the org id, the action name, the input shape (parsed, never raw), and the timestamp. Conflating the two audiences is the failure mode the rule prevents.
- **What the user message is allowed to say.** A short, plain sentence in product terms. *"That slug is already taken."* *"You don't have access to this organization."* *"This invoice is no longer in draft and can't be edited."* *"Something went wrong. Please try again or contact support."* (the last is the fallback for genuinely unexpected failures the system can't disambiguate). The 2026 senior reflex: the message is a string the support team would read aloud on a call — that's the read-aloud test. If the support rep would have to translate it ("the constraint name means…"), it's not user-shaped.
- **What the user message is forbidden from carrying.** Internal IDs ("Invoice 7a9c-... could not be created"); stack traces or any source-code reference; DB error codes ("DB constraint X failed", "Error 23505"); third-party error strings ("Stripe API: webhook signature invalid"); the affected tenant or other users' data ("Another organization already owns this slug"); raw input echoed back unless the field-level error needed it ("'email' is invalid" is fine; "'<script>...' is invalid" echoes the attack); environment names ("staging server returned 500"); secrets, tokens, session IDs (obvious but worth saying once). The lesson surfaces each class with one canonical leak and the safe equivalent.
- **The operator record's shape — what gets captured.** A structured event the logger / Sentry / the audit log all consume the relevant pieces of. The fields: `error.name`, `error.message`, the cause chain walked (from 2.8.2's while-loop), the stack; `action` (the action name); `userId`, `orgId`, `role` from `ctx`; the *parsed* input (Zod-cleaned, post-strict-object); `requestId` (correlation ID — Unit 20.1.2 owns the discipline; named here as the join key); the timestamp; the route or referrer if the framework surfaces it. The senior reach: capture once, in the wrapper's catch, before the user message diverges; the structured event becomes the single source of truth the operator reads.
- **Where the two artifacts diverge — the wrapper, not the UI.** `authedAction`'s catch block, `authedRoute`'s catch block, and the page's `error.tsx` are the three seams where the split lands. Each does the same thing: capture the operator record (Sentry + structured log + audit-log entry where the action's domain expects one), then map to the user-visible artifact (the `Result`'s `userMessage`, the route's RFC 9457 Problem Details body, the error page's generic copy). The student sees the split as a property of the wrapper, not a discipline applied at each call site.
- **The mapping table — error class → user message.** A small dispatch in `/lib/error-mapping.ts`. Known cases get a tailored user message: `ZodError` → `'Check the highlighted fields.'` (and `fieldErrors` populated from `z.treeifyError`); Postgres `23505` unique violation → `'That value is already taken.'` (and the action layer overrides per-feature: `'That slug is already taken.'`); Postgres `23503` foreign-key violation → `'Related record is missing.'`; the action's own domain error classes (from 2.8.2's pattern — `InvoiceNotFoundError`, `PaymentDeclinedError`, `WebhookSignatureMismatchError`) → their tailored messages with no operator detail; the unmatched fallback → `'Something went wrong. Please try again.'` plus an `code: 'internal'` for analytics grouping (20.1). The lesson names the mapping as the senior's single place to lint: every new error class lands here once, every action calls into the mapper, the mapper enforces the split.
- **The `code` plus `userMessage` shape, restated.** From 7.2.3: `{ ok: false, error: { code: string; userMessage: string; fieldErrors? } }`. The `code` is the stable machine identifier (`'conflict'`, `'unauthorized'`, `'not_found'`, `'plan_limit'`, `'internal'`, `'invalid_input'`); the `userMessage` is the human string. The two are independent — analytics groups on `code`, the UI renders `userMessage`. The senior reach: never use the `code` as the user message ("Error: conflict") and never use the `userMessage` as the analytics group (translated strings fragment the grouping).
- **The route-handler form — RFC 9457 Problem Details.** From 7.5.2: a JSON body of shape `{ type, title, status, detail, instance, errors? }`. `title` is operator-honest but user-safe ("Conflict"); `detail` is the user-visible string ("That slug is already taken."); the `errors` field carries field errors. The same split lands at a different artifact — the wrapper writes the body, the operator log captures the original. The lesson names the symmetry: actions return `Result`, routes return RFC 9457, both are forms of the same split.
- **The page-boundary form — `error.tsx` and `global-error.tsx`.** From 5.3.3 / 5.3.4: `error.tsx` is the segment-level boundary; `global-error.tsx` is the root fallback. Both render generic copy by default (the framework's default page is fine for many products; many SaaS replace it with a branded shell). The structural rule: the `error.tsx` component receives an `error` prop *with the message sanitized by Next.js* in production builds (the framework hashes the digest, the user sees nothing actionable except "Try again"). The senior reach: do not surface `error.message` to the user inside `error.tsx`; render product-shaped copy, reach for a `Sentry.captureException(error)` call inside the component's `useEffect` (Sentry's React integration auto-captures, but the explicit call is the senior anchor), and keep a "Try again" reset button per the framework convention.
- **What `error.tsx` is allowed to expose to the user.** A generic copy: *"Something went wrong. We're looking into it."* (or product-shaped equivalent). A "Try again" button (the framework's `reset()`). A "Go to dashboard" link as a recovery affordance. Maybe a stable `digest` value the user can quote to support ("Reference: 4b1c2…"). What it's not allowed to expose: the message, the stack, internal IDs from the error, the action name, the tenant. The 2026 platform default already enforces most of this in production builds; the senior anchor is to design the boundary as if the framework didn't, so dev-mode habits don't ship as prod leaks.
- **The `digest` channel — the operator's join key.** Next.js attaches a `digest` to errors caught by `error.tsx` in production (a stable hash of the error). The user can quote the digest to support; the operator looks it up in Sentry / the structured log to find the full event. The senior reach: surface the digest to the user as the *only* internal reference, never as the message. The 2026 form: a "Reference: <digest>" line in small text under the generic copy. The lesson names the digest as the right kind of operator detail the user is allowed to see — opaque, non-leaky, joinable.
- **Sanitization at the operator boundary — Sentry's `beforeSend` and the structured-log redactor.** Sentry's JS SDK ships `beforeSend(event)` for the SDK-level scrub before the event leaves the runtime; the structured logger ships a redactor for the same purpose. The senior reach: configure both with the project's known sensitive keys (`password`, `token`, `secret`, `apiKey`, `authorization`, `cookie`, `creditCard`, `ssn`, plus the application's own — `paymentMethodId`, `webhookSecret`, etc.) and a generic regex for credit-card-shaped strings. Sentry's defaults already filter common keys; the project's `beforeSend` adds the application-specific list. The lesson names the helper as the **single** place the redaction list lives — every log line and every Sentry event go through it. Wiring lands in Chapter 20.1; the rule lands here.
- **What the operator record never carries.** Passwords (the action's input schema strips password fields from the parsed object the wrapper logs — never log `input` for a sign-in or password-change action; log the action name and the user id only); session cookies / tokens; the user's full PII when the action doesn't need it; a third party's API key. The senior reach: pre-decide which fields the action's structured log includes per action — the wrapper's logger receives the *redacted* parsed input, not the raw `formData`. The course's `safeAction` builds a per-action redactor declared next to the schema.
- **The "operator-honest, user-opaque" rule for log lines.** A log line *can* say *"insert into invoices failed: duplicate key on invoices_org_id_slug_key — orgId=org_..., slug='budget-2026', userId=usr_..."* — the operator needs all of it. The same incident's *user* artifact is just *"That slug is already taken."* The lesson reads the two lines side-by-side as the chapter's reference shape. The contract: the operator side is rich, the user side is sanitized, the wrapper splits them.
- **The audit-log angle.** Audit-log entries (10.2.5) capture the user-initiated events ("invoice.created", "member.removed") as part of the normal write path, *inside the transaction*. An audit log entry is *not* an error log; the lesson names the distinction so the student doesn't conflate them. When an action throws, the audit-log entry rolls back with the transaction — the operator sees the failed attempt only in Sentry / structured logs, not in the audit log. Chapter 17.2.3 revisits the audit-log discipline at the security baseline; the read here is the boundary.
- **Field errors are still user messages.** The `fieldErrors` returned by Zod's `safeParse` → `z.treeifyError` are user-visible strings ("Email is invalid", "Must be at least 8 characters"). The senior reach: the Zod schema authors them with user-shaped copy (Chapter 7.1.5 owns the `error` option that customizes messages). The default Zod messages are good 2026-grade defaults; the senior anchor is to override per-field where the default reads engineer-ish ("Expected string, got number" is engineer-ish; "Must be a number" reads better). Field errors flow through the same `userMessage` channel — the lesson surfaces that they don't bypass the split.
- **The 404 / 403 distinction — tenant leaks and the "404 over 403" rule.** From 10.2.3: when a request to `/invoices/[id]` hits an invoice that exists but belongs to another tenant, the senior reach is to return **404, not 403** — 403 ("forbidden") admits the resource exists in the system, which is itself a leak (the attacker now knows the ID is valid). The user-message side: "Not found." The operator side: a structured event with `code: 'cross_tenant_attempt'`, the user, the org, the requested ID — useful intel for security review (Unit 17.2). The lesson names the asymmetry: the user sees a generic 404, the operator sees the truth. The fail-closed rule from 17.1.1 applies; the message-split rule adapts the user-facing artifact to not leak.
- **The "no inner exception in the user message" rule.** When an action catches a downstream error and wraps it (`new InvoiceCreationError('...', { cause: dbError })` from 2.8.2), the `userMessage` is authored at the *outer* layer, never derived from the inner cause's message. The cause chain is for the operator; the user sees the outer error class's user-shaped string. The lesson surfaces the rule: the catch-and-rewrap from 2.8.2 implicitly enforces the split — `cause` goes to the operator chain, the new error's `userMessage` field is the user side, the two never meet.
- **The webhook receiver's message-split shape.** A webhook receiver returns 400 for signature failures, 409 for dedup-stored duplicates, 200 for successful processing, 500 for unexpected failures (the provider retries). The "user" of a webhook receiver is the provider (Stripe, Resend); the "operator" is still the on-call engineer. The split applies: the response body to the provider is the RFC 9457 shape with minimal detail (`{ type, title, status }`), and the structured log carries the full event details (the `Stripe-Signature` header parsed, the timestamp delta, the event id, the type, the org id resolved from the customer). The lesson surfaces the symmetry: every seam has a "user" (some external caller) and an "operator," every seam splits.
- **The "do not leak Next.js internals" rule.** `error.tsx` in production already filters; in dev the message comes through. The senior reach is to assume production behavior when authoring the file (use a generic copy by default, never include `error.message` in the rendered JSX even when "it's just for dev"). The Vercel-side production build strips the message in serverless responses; the senior anchor is to not depend on that stripping — author the copy as if the message will leak.
- **Watch-outs.** Echoing `error.message` "for dev" in `error.tsx` ships as a prod leak when someone forgets to remove it; rendering Zod's raw issue tree in the form leaks schema structure to the DOM (render the field-error strings only); including `cause.message` in the user-rendered string defeats the wrap; logging the raw `formData` includes passwords and CSRF tokens (log the parsed-and-redacted input only); using HTTP 500 as the catch-all hides the failure class from monitoring (match 401 / 403 / 404 / 409 / 422 / 500); Sentry's default scrubbing list is project-blind — extend `beforeSend` with the application's domain-specific keys; pretty-printed JSON logs are unqueryable in Log Drains, ship newline-delimited.

What this lesson does not cover:

- The seam-by-seam walk where the two commitments land in existing code — Lesson 17.1.3.
- Sentry SDK setup, breadcrumbs, source-map upload — Chapter 20.1.1.
- Structured-log discipline, request correlation IDs, and Vercel Log Drains — Chapter 20.1.2 / 20.1.4.
- The "log inputs you'd want at 3am" rule — Chapter 20.1.3.
- Internationalization of user messages — Unit 18.2.
- The audit-log table and append-only enforcement at depth — Chapter 17.2.3.

## Lesson 17.1.3 — Walking the six error seams

A seam-by-seam audit of `authedAction`, `authedRoute`, page-level `requireOrgUser`, the webhook receiver, the rate limiter, and the `error.tsx` boundaries, naming where each commitment lands and what to grep for to catch bypasses.

Topics to cover:

- **The senior question.** Two commitments are now named — **fail closed**, **two messages**. The codebase has, by this point, six seams where errors flow: `authedAction` at the Server Action boundary; `authedRoute` at the route-handler boundary; the page / layout `requireOrgUser()` check; the webhook receiver; the rate-limiter call; the page-level `error.tsx` and `global-error.tsx` boundaries. The lesson walks each seam and answers two questions per seam: **where does fail-closed live, and where does the message-split happen?** The output is a mental map of "every error path is one of these six shapes, every shape has a documented commitment landing." The walk is the audit pass the chapter exists for; the project in 17.3 turns the walk into a check-list.
- **Seam #1 — `authedAction(role, schema, fn)` at the Server Action boundary.** The canonical surface from 10.2.2. **Fail-closed landings:** the schema parse throws / returns failure → wrapper returns `{ ok: false, code: 'invalid_input' }`; `requireOrgUser()` throws on no session / no active org → caught by the framework's `error.tsx`; `roleAtLeast(role)` throws on too-low role → wrapper returns `{ ok: false, code: 'unauthorized' }`; `fn(ctx, input)` throws → wrapper captures, logs, returns `{ ok: false, code: 'internal' }` (the operator's Sentry captures the cause chain, the user sees the generic copy). **Message-split landings:** the wrapper's catch block runs the error-mapping helper, produces the `userMessage`, runs the redactor on the operator log payload (action name + redacted parsed input + ctx). Both commitments land in the wrapper body — *one* file, *one* place to lint. The lesson reads the wrapper's source as the canonical implementation.
- **The audit step for seam #1.** Grep for files containing `'use server'` whose exported actions don't go through `authedAction`. Each hit is a candidate: legitimate exceptions (the public sign-up server action — different wrapper) get documented; the rest are migrated. The lesson names this as the audit project's first finding category.
- **Seam #2 — `authedRoute(role, schema, fn)` at the route handler boundary.** Symmetric to seam #1 but the framework artifact is `Response`, not `Result`. **Fail-closed landings:** mirror seam #1 — parse → 422, no session → 401, role too low → 403, cross-tenant resource → 404 (the leak-avoiding shape from 10.2.3), business `Result.err` → mapped to the matching HTTP status, unexpected throw → 500. **Message-split landings:** the wrapper writes the RFC 9457 Problem Details body (`type, title, status, detail, instance, errors?`), runs the redactor on the operator log payload (route + method + headers minus auth/cookie + parsed input + ctx). The lesson reads `authedRoute`'s body and points at each line implementing one or both commitments.
- **The audit step for seam #2.** Grep for `route.ts` files exporting `GET`/`POST`/`PUT`/`PATCH`/`DELETE` that don't import `authedRoute`. Same triage: legitimate exceptions (the webhook receivers, the public auth callbacks) are named, the rest are migrated. The webhook receivers don't use `authedRoute` — they use a *different* wrapper (seam #4) with its own commitments.
- **Seam #3 — page and layout `requireOrgUser()`.** Server Components and layouts at protected route segments call `requireOrgUser()` near the top. **Fail-closed:** the helper throws on no session / no active org. **What catches the throw:** the framework's nearest `error.tsx`; the `proxy.ts` two-layer gate from 9.4.1 catches the no-session case at the perimeter and redirects to `/login`, so the in-component throw should fire only for the rare "session expired between proxy and render" race or for genuinely unexpected failures. **Message-split:** `error.tsx` renders generic copy; Sentry captures the cause chain via the integration's auto-instrumentation. The lesson points at the symmetry: the page boundary's fail-closed is implicit in "throw and let the framework catch"; the message-split is structural in "the user never sees the message."
- **The audit step for seam #3.** Grep for protected Server Components and layouts not calling `requireOrgUser()` (or its equivalent) at the top. The proxy gate (9.4.1) is the outer ring; the in-component check is the inner ring — both run, the inner one catches the cases the proxy can't (the user lost org membership while the page was open, the active org switched in another tab). The senior anchor: defense in depth. Pages that read tenant-scoped data without the inner check are findings.
- **Seam #4 — the webhook receiver (`/api/webhooks/*`).** The signature-verify-then-dedup pattern from 12.1.1–12.1.2. **Fail-closed landings:** raw body read fails → 400; signature parse fails → 400 (the structured log captures the malformed header); HMAC compare fails → 401 (constant-time compare from 3.7.1; throw on library exception is also fail-closed → 500 with the provider retrying); the `INSERT ... ON CONFLICT DO NOTHING` returns "already processed" → 200 (idempotent success, not a refusal); the business work throws inside the outer transaction → 500, the provider retries, the dedup constraint catches the duplicate on retry. **Message-split landings:** the response body to the provider is RFC 9457-shaped (`type, title, status` minimal); the structured log carries the full event (the parsed `Stripe-Signature`, the timestamp delta, the event ID, the type, the resolved tenant, the cause chain on throws). The lesson surfaces that the "user" here is the provider — same split rules, different audience.
- **The audit step for seam #4.** Each webhook receiver in `/api/webhooks/*` is visited. Confirm: raw body read before any JSON parse, signature verify with constant-time compare, dedup INSERT before the business work, business work inside a transaction with the dedup, status codes match the failure class. The Resend bounce/complaint receiver (12.1.5) follows the same shape with a Svix signature verify. Findings are receivers that JSON-parse before signature verify (the canonical bug), that catch and 200 on failure (fail-open dressed as "don't make the provider retry"), or that include the full Stripe payload in the response body (a leak the provider doesn't need).
- **Seam #5 — the rate-limiter call (the named fail-open carve-out).** From 15.3.3: the auth-path limiter is fail-open. **The carve-out's documentation:** the `safeLimit(limiter, key)` helper wraps the `limit()` call, catches the throw, logs `{ event: 'rate_limit_unavailable', limiter, key }` at error level, returns `{ success: true, ... }`. **Message-split:** the user-facing 429 body is the user-safe "Too many attempts. Please try again later." (same regardless of which gate tripped, no "your email" leak); the structured log captures which limiter, which key, the operator's full diagnosis. The lesson names the carve-out as the chapter's one explicit exception to fail-closed, with the policy documented in the helper. Other limiters (admin, billing-critical) may flip to fail-closed; the policy lives in one place.
- **The audit step for seam #5.** Grep for `limit(` calls that don't go through `safeLimit`. The carve-out exists once, the helper is the lint target. Findings are direct `limiter.limit()` calls in handlers — they bypass the documented policy.
- **Seam #6 — `error.tsx` and `global-error.tsx`.** **Fail-closed landings:** the framework catches the thrown error; in production builds, Next.js redacts the message and ships a `digest`; `global-error.tsx` is the root-layout fallback for errors `error.tsx` can't catch (root layout itself throwing). The "fail-closed" reading here: the user never gets the resource — they get the boundary. **Message-split landings:** the boundary renders generic copy; the `digest` is the user-facing join key (the only operator detail exposed); Sentry captures the full error via the integration (or via an explicit `Sentry.captureException(error)` in a `useEffect` for the senior anchor). The lesson reads the canonical `error.tsx` shape: a "Something went wrong" headline, a "Try again" button calling `reset()`, a small "Reference: {digest}" line.
- **The audit step for seam #6.** Each route segment with sensitive data has an `error.tsx`. The root has `global-error.tsx`. None render `error.message` to the user. Each calls Sentry capture (auto via the integration or explicitly via `useEffect`). Findings: segments without an `error.tsx` (they fall through to a parent; usually fine but worth confirming); `error.tsx` files that surface the message; missing Sentry capture in `global-error.tsx` (the root case is the most exotic — the rendering already failed, the senior reach is the explicit `captureException` call).
- **The `notFound()` / `redirect()` carve-out.** Restated from 5.1.4: these are framework control-flow primitives, not errors. The `error.tsx` boundary does *not* catch them — `not-found.tsx` and the framework's redirect handler do. The fail-closed rule still applies: a thrown `notFound()` from a Server Component aborts rendering and routes to the not-found page; the user doesn't get the resource. The message-split applies: the not-found page renders generic copy, no internal IDs, no leak of which ID was requested.
- **The "code" channel and analytics alignment.** Every error returned through any seam carries a stable `code` (or, for RFC 9457, the `type` URN). The analytics layer (PostHog from Unit 20.2, gated behind the consent flag from 17.2.5) groups error events by code. The senior anchor: the code is the contract between layers — the action wrapper, the route wrapper, the webhook receiver, and the error boundary all reach into the same enumerated set. The lesson surfaces the alignment: a leak in any single seam (a hand-rolled error with a free-form `code: 'something_failed'`) fragments the dashboard.
- **The audit deliverable — six paragraphs.** The lesson's exercise: the student writes six paragraphs, one per seam, naming the file(s) where the seam lives, the wrapper or helper that owns it, where fail-closed lands, where message-split lands, and one concrete commit (or PR) that touched the seam. The output is a map of the codebase's error surface that 17.3's project audit consumes. The exercise is graded on accuracy of seam location and rule landing, not prose.
- **The "every error path is one of these six" rule.** A new feature lands; its error paths must fit one of the six shapes — Server Action / route handler / page check / webhook receiver / rate limiter / page boundary. A seventh shape (a hand-rolled "API client" that hits an internal endpoint with hand-written fetch, a custom protocol receiver) means adding a seventh seam to the list with its own wrapper and its own commitments documented. The senior framing: the seam catalog is the architectural surface; growing it is a deliberate call, not a quiet drift.
- **The forward references.** Chapter 17.2 walks the *security baseline* against the same seams — headers, rate limits, audit-log gaps, secrets, GDPR, dep hygiene. Chapter 17.3 is the project that runs the full audit (error discipline + security baseline) against a seeded codebase with known findings. Chapter 20.1 wires Sentry and the structured logger that consume the operator side of the message split. Chapter 21.2 (CI on GitHub Actions) wires the lint rules that catch the seam-bypass patterns at PR time. Each unit is named once.
- **Watch-outs.** A seam "almost like `authedAction` but with an extra parameter" should extend the wrapper, not fork it (parallel wrappers drift); a webhook receiver with a custom logger fragments the operator surface (the structured logger takes per-event tags); a page check inside a Client Component bypasses the Server Component perimeter; an `error.tsx` that toggles "Show details" is a leak vector by design; copy-pasting the wrapper to "customize" one action means the audit pass misses the copy (extend, never fork); `console.log(error)` lines leak into production branches — use the structured logger's `debug` level.

## Lesson 17.1.4 — Quizz

Top 10 topics to quiz:

- The fail-closed rule — any check that gates access (authorization, tenancy, paywall, signature verify) refuses by default, and an exception inside the check is treated as a refusal, not as an allow; the canonical failure mode of fail-open dressed as discipline (the empty-catch, the `boolean`-returning helper that swallows the throw).
- The two-message rule — every error is two artifacts, a sanitized user message (read-aloud-safe, no IDs / stacks / DB error names / cross-tenant leaks) and a fully-instrumented operator record (cause chain, ctx, parsed-and-redacted input, request id); the wrapper / boundary is where they diverge, never the UI.
- The six seams where the two commitments land — `authedAction`, `authedRoute`, the page / layout `requireOrgUser()` check, the webhook receiver, the rate limiter (the documented fail-open carve-out), and the `error.tsx` / `global-error.tsx` boundaries; every error path fits one of these six shapes.
- The named exception to fail-closed — the auth-path rate limiter is fail-open by deliberate policy because a Redis outage locking every user out is worse than the brief abuse window; the policy lives in one helper (`safeLimit`) with a logged event, not at every call site.
- The structural form that makes fail-closed automatic — the check throws on its own failure (never returns a sentinel), the wrapper catches in one place and converts to a refusal (`Result.err` for actions, the matching HTTP status for routes, the framework's `error.tsx` for pages); "one place to lint" is the architectural reason.
- The error-mapping helper — known error classes (`ZodError`, Postgres `23505` / `23503`, the action's domain-error classes, the unmatched fallback) get tailored user messages and stable `code` values; one mapper file is the single source of truth.
- The `code` + `userMessage` separation — `code` is the stable machine identifier (`'conflict'`, `'unauthorized'`, `'not_found'`, `'invalid_input'`, `'plan_limit'`, `'internal'`), `userMessage` is the human string; analytics groups on `code`, the UI renders `userMessage`, never use one as the other.
- The "404 over 403" rule — cross-tenant resource access returns 404, not 403, so the response doesn't admit the resource exists; the operator log captures the `cross_tenant_attempt` event for security review.
- The `error.tsx` boundary's discipline — render generic copy, never surface `error.message`, expose at most the framework's `digest` as a user-quotable reference, call `Sentry.captureException` (or rely on the integration) for the operator side; the message is forbidden from carrying internal IDs / stack traces / third-party error strings.
- The redactor — the single helper that strips known sensitive keys (`password`, `token`, `secret`, `apiKey`, `authorization`, `cookie`, plus the application's own) from every operator-side artifact (Sentry events via `beforeSend`, structured logs, audit-log payloads); the action wrapper logs the redacted parsed input, never the raw `formData`.
