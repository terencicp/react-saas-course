# Chapter 9.4 — Auth at request time and account management

## Chapter framing

Chapter 9.3 closed the sign-in surface — every flow that mints a session. Chapter 9.4 covers the rest of the session's life: how the request-time gate decides who reaches a protected route, how a signed-in user changes the credentials that signed them in, how the user (and the system) audit and revoke sessions across devices, how the user deletes the account on demand with the verification ceremony the 2026 baseline expects, and what browser-security defaults the stack already buys without extra code. The student leaves with a production-shaped `proxy.ts` gate, a `/settings/security` surface that handles credential mutation through Better Auth's APIs at the senior elevation tier, an active-sessions list with per-row revocation, a delete-account flow with verification + cascading revocation + retention discipline, and a clear-eyed read on what CSRF/XSS mitigations React + Next.js + Better Auth already encode.

Threads that run through every lesson. The proxy is for redirects on cookie-presence; real authorization decisions live downstream at the action boundary (the 5.5.2 rule honored). Every credential-mutation endpoint is elevated — `freshAge` from 9.2.3 is the senior tier and current-password / re-authentication prompts close the gap when the session is stale. Session revocation is non-negotiable on every credential change (password reset already does it from 9.3.4; password change and email change repeat the discipline; account deletion is the maximal case). Enumeration discipline from 9.3 carries forward — change-email returns the same response shape whether the new email is taken. The send side of any new email (change-email confirmation, "new device signed in", delete-account verification, post-change notice) still flows through the Unit 8 pipeline. The `verification` table from 9.2.2 keeps doing the same job — change-email tokens, delete-account tokens, all hashed at rest, all one-time-use, each in its own `identifier` namespace. Browser-security defaults are framed as "what the stack already ships, what would undo them" — the lesson teaches the structural defense and the one footgun (`dangerouslySetInnerHTML`), not a survey of headers. Full CSP / security-header configuration is Chapter 17.3's territory and named once. Five teaching lessons plus a quiz.

## Lesson 9.4.1 — The protected-routes gate in `proxy.ts`

Topics to cover:

- **The senior question.** A signed-out user types `/dashboard` into the address bar. Where does the redirect-to-sign-in decision live, what does it read, and why is the proxy explicitly *not* the place to enforce authorization? 9.2.4 stood up the minimum gate; this lesson upgrades it to the production-shaped pattern: matcher design, cookie-presence-only reads (no DB), the `?next=` round-trip, the "auth pages refuse signed-in users" inverse gate, and the strict scope rule that keeps authz at the action boundary.
- **The two-layer gate.** Layer one is the proxy — coarse, cookie-presence-only, cheap. Runs on every matched request including prefetches, so it must not touch the database. Layer two is the layout/action — fine, full session read, full authz. The proxy's only job is "no cookie → redirect"; the layout/action's job is "valid session → who is this and may they do X." Conflating the two is the canonical bug: a proxy that does role checks becomes the entire security model, then a missed matcher entry silently opens a route.
- **Cookie-presence-only inside the proxy.** Inside `proxy.ts`, use `getSessionCookie` from `better-auth/cookies` (or read the cookie directly off `request.cookies`). Do *not* call `auth.api.getSession` from the proxy — it round-trips to Postgres on every prefetch, and Next.js 16 will prefetch protected routes on hover. The optimistic check is "is there a session-shaped cookie at all"; the real validation happens once when the layout renders. Forgery isn't a concern at this layer — the layout's `getCurrentUser` is the validating read.
- **The `matcher` design.** Two strategies: allowlist matched protected paths (`['/dashboard/:path*', '/settings/:path*', '/billing/:path*']`) or matchall-minus-public (regex matcher that excludes `/`, `/sign-in`, `/sign-up`, static assets, `/api/auth/:path*`). The senior call: allowlist when the app has a small set of protected sections; matchall-minus when most of the app is gated. Either is correct; the failure mode is mixing — a partial allowlist leaves new routes unprotected by default. Pick one and document it.
- **The `?next=` round-trip and open-redirect closure.** When the proxy redirects, it preserves the original destination: `redirect(`/sign-in?next=${encodeURIComponent(request.nextUrl.pathname + request.nextUrl.search)}`)`. After sign-in, the form reads `next`, validates it against the allowlist (paths that start with `/` and not `//`, no absolute URLs, no protocol-relative), and redirects there. The 5.5.3 rule applied at the call site — never reflect `next` raw, never trust user-supplied URL into `redirect()`. The senior fallback: invalid `next` → redirect to `/dashboard`, never to the raw value.
- **The inverse gate — auth pages refuse signed-in users.** A signed-in user hitting `/sign-in` should redirect to `/dashboard`, not show the sign-in form (broken UX, sometimes a re-auth security smell). Add the inverse rule in the same proxy: matched path is an auth page, cookie present → redirect to `/dashboard`. Same matcher logic, opposite direction.
- **What runs on the Node runtime in Next.js 16.** `proxy.ts` runs on Node by default in Next.js 16 (the Edge-runtime constraint that limited `middleware.ts` in Next.js 15 and earlier is gone). Better Auth's `getSessionCookie` works either way — it's pure cookie parsing — but the Node default removes the historical reason to keep DB access out of the proxy. The senior rule still holds for performance, not capability: optimistic cookie checks in the proxy, validating reads in the layout.
- **Composition with other proxy responsibilities.** The proxy may also handle i18n routing (18.2), feature flags, A/B test bucketing. Keep each responsibility a small function and chain them — `await authGate(request); await i18nGate(request)`. The senior reflex: one file, many gates, returns the first redirect or the `NextResponse.next()`. Don't split proxies across files; Next.js only runs one.
- **The session-touch reflex.** Better Auth's sliding renewal (`updateAge` from 9.2.3) updates `session.expiresAt` when the next mutating call happens. Some teams update the cookie in the proxy on every request; in this stack, don't — the cookie cache plus the renewal-on-action shape is enough, and a proxy-level touch adds a write to every page load.
- **Layout-level read remains the truth.** After the proxy lets the request through, the protected layout still calls `requireUser()` (9.2.4). That call validates the cookie against the database (or cookie cache), reads `user`, and either renders the layout or redirects. The proxy is the perimeter; the layout is the door check. The senior reflex: never assume the proxy's cookie check guarantees a valid session — the cache might be stale, the cookie might be forged, the user might have been deleted; the layout's read is the source of truth.
- **Watch-outs.** Calling `auth.api.getSession` from the proxy — DB load on every prefetch and route change; doing role checks in the proxy — authz leaks when the matcher misses; reflecting `?next=` without allowlist validation — open-redirect to attacker domain; redirecting signed-in users away from `/sign-in` *into* a loop with another proxy rule — test signed-in and signed-out matrix for every matcher path; forgetting the inverse gate so a signed-in user sees a sign-in form (and clicking it sometimes invalidates their current session); putting the proxy file under `src/app/` instead of project root or `src/` — Next.js won't pick it up; relying on `request.nextUrl.pathname` alone for the matcher when the matcher already filters — duplication that drifts.

What this lesson does not cover:

- The full session read inside the layout / action (9.2.4 owns the call shape).
- Authorization, role checks, org-scope filtering (Chapter 10.2's `authedAction`).
- The full `?next=` form-side handling and open-redirect rule (5.5.3 owns the surface; this lesson honors it at the call site).
- The active-sessions list / per-session revoke UI (9.4.3).
- Rate limiting the redirect or the sign-in surface (Chapter 15.3).
- Multi-tenant org-aware routing (Chapter 10.1).

Estimated student time: 35 to 45 minutes. Pattern + decision hybrid; the matcher-design table ("allowlist vs matchall-minus") and a sequence diagram for "request → cookie present? → matched? → redirect or pass-through → layout's real read" earn their weight.

## Lesson 9.4.2 — Credential changes and the elevation tier

Topics to cover:

- **The senior question.** Signed-in user wants to change their password or their email. What's the elevated-action discipline that distinguishes "current session may proceed" from "make me re-authenticate first," what's the post-change session-invalidation move, and how does change-email reuse the verification primitives from 9.3.3 without re-inventing them? Stand up `/settings/security`, wire `changePassword` and `changeEmail` against Better Auth, surface the elevation re-prompt when the session is stale.
- **The elevation tier — `freshAge` reread.** From 9.2.3: `freshAge` (default 1 day) is the "this session was recently authenticated" window. Inside it, sensitive actions proceed; outside it, the user re-authenticates first. The senior reflex: every credential-mutation endpoint *and* every irreversible-action endpoint reads `freshAge` and gates accordingly. Library handles the gate when the endpoint is library-owned (changePassword, changeEmail, twoFactor.enable/disable from 9.3.6, addPasskey from 9.3.7); app-owned destructive actions (delete-account, export-data) reach the same gate explicitly via the `freshAge` helper.
- **`changePassword` — the call, the knob, the side effect.** `authClient.changePassword({ currentPassword, newPassword, revokeOtherSessions: true })`. Library verifies `currentPassword` (constant-time Argon2 compare from 9.3.1), enforces `minPasswordLength`, hashes `newPassword`, updates the `'credential'` account row. `revokeOtherSessions: true` is the non-negotiable senior default — "a credential change means the old credential might be compromised, every other session must die." Defaults to false in the library; flip it to true at the call site, every time. Surface the `'invalid-credentials'` result when `currentPassword` is wrong (same enumeration discipline as sign-in — same shape regardless of cause).
- **The "current password" prompt — the elevation pattern at the form.** Even with a fresh session, the change-password form requires the current password. Two reasons: defense-in-depth (a borrowed laptop with a logged-in session can't change the password without it), and library enforcement (the `currentPassword` field is required). Don't reach for `freshAge`-only elevation here — the current-password check is the right tier; `freshAge` covers actions that *don't* have an existing credential to re-prove (passkey add, 2FA disable).
- **`changeEmail` — enable, call, verification ride.** `changeEmail: { enabled: true, sendChangeEmailVerification: async ({ user, newEmail, url }) => sendEmail({ to: user.email, ... }) }` on the `user` config. Disabled by default in the library. Call: `authClient.changeEmail({ newEmail, callbackURL: '/settings/security' })`. Library sends a verification link to the *current* email (not the new one) — confirming the legitimate user controls the existing inbox, then flips on click. Senior call: also send a "your email is being changed to X" notice to the current address and, after the flip, a "your email was changed" notice to both addresses. Library's send is the verification token; the notices are app code through Unit 8.
- **Why verify on the old email, not the new.** Two threat models. Sending to the new email proves the user controls the new inbox. Sending to the old email proves the user *currently* controls the existing account — closes the "attacker briefly got a session, tries to lock the user out by changing email" path. The 2026 library default is "verify on current address"; some configurations support both (dual-verification: current address confirms the request, new address confirms ownership). For most SaaS, current-address verification plus the post-change notice to both addresses is enough.
- **Email-change row in `verification`.** Same table as 9.3.3, different `identifier` namespace — `change-email:<userId>:<newEmail>`. One-time use, library-default expiry (typically 1 hour), token hashed at rest. Reuses every primitive from email-verification; the schema does the work.
- **Post-change session disposition for email change.** Email change doesn't carry the same "credential compromise" semantics as password change — the password still works. Library doesn't revoke sessions on email change by default. Senior call: revoke other sessions anyway if the product's threat model treats the email as a sign-in identifier (it is — `signIn.email({ email, password })`). The "your email was changed" notice should include a "wasn't you?" link that nukes all sessions and forces a password reset.
- **The `/settings/security` page anatomy.** Three forms on one page: change password (current, new, confirm), change email (current shown read-only, new), and a links section to 2FA setup (9.3.6) / passkeys (9.3.7) / active sessions (9.4.3). Each form is its own Server Action with its own `useActionState`. The shape mirrors every form pattern from 7.2 / 7.4 — Zod on input, `Result` discriminants on output (`'ok'`, `'invalid-credentials'`, `'email-taken'`, `'requires-re-authentication'`).
- **The re-authentication prompt — the `'requires-re-authentication'` branch.** Action calls a library endpoint, library returns the "session not fresh" error. Action returns `'requires-re-authentication'`. UI replaces the form with "for security, sign in again to continue" — a mini sign-in form that, on success, returns the user to `/settings/security` with the action retryable. Library's `freshAge` enforces; app surfaces the UX. Don't reach for a custom re-auth modal that hand-rolls password verification — call `signIn.email` and let it issue a fresh session.
- **Email-change-during-unverified-account — the small edge.** If a user signed up with email X (unverified), then changes to email Y before verifying X, the `verification` table has stale rows for X. Library cleans up; senior reviews: the test case "sign up → don't click verify → change email → click old verify link" should produce "invalid or expired link" — same shape as every verification failure (enumeration discipline).
- **OAuth-only users and `changePassword`.** A user who signed up via Google (9.3.8) has no `'credential'` account row. `changePassword` returns `'no-password-set'` (or library equivalent). UX surfaces "you sign in with Google — set a password to enable email+password sign-in." Setting a password is the `setPassword` call (library surface), which adds the `'credential'` row. Don't conflate with `changePassword`; the senior reflex catches the OAuth-only state.
- **Watch-outs.** Defaulting `revokeOtherSessions: false` on password change — the entire point of the move undone; verifying change-email on the *new* address only — opens the email-takeover vector; treating elevation as "session is signed in" instead of "session is fresh" — destructive actions go through stale sessions; logging the new password or the current password anywhere; surfacing `'invalid-credentials'` and `'email-taken'` with distinct error copy — enumeration; rolling a custom re-auth modal that calls a non-library endpoint to verify the password — bypasses the rate limit and fixation defenses on `signIn.email`; not sending a notification email on email change — silent change is a takeover-detection failure; allowing `changeEmail` to land without verification — bypasses the entire flow.

What this lesson does not cover:

- Initial password reset for forgotten passwords (9.3.4 owns).
- 2FA enable / disable elevation (9.3.6 owns).
- Passkey add / remove elevation (9.3.7 owns).
- The send pipeline anatomy (Chapter 8.3).
- Account deletion / data export (Chapter 17 owns the destructive-action wrappers).
- Audit logging for credential changes (Chapter 10.2).
- Rate limits on credential endpoints (Chapter 15.3).

Estimated student time: 40 to 50 minutes. Pattern + mechanics hybrid; the elevation-decision tree ("current-password prompt vs `freshAge` re-auth vs both") and a sequence diagram for "change-email → verify current address → click → flip → notice to both" earn their weight.

## Lesson 9.4.3 — Session management across devices

Topics to cover:

- **The senior question.** A user has the app open on a phone, a laptop, and a desktop. Each is a `session` row. They want to see the list ("am I signed in on the work laptop I left at the office?"), revoke one ("yes, kill that one"), and have a single "sign out everywhere" button for the worst case. What does the library provide, what does the senior surface look like, and how does this compose with the credential-change revocation from 9.4.2? Build the `/settings/security/sessions` page with list + per-row revoke + sign-out-everywhere.
- **The data model — already in place.** `session` table from 9.2.2: one row per active session, columns `id`, `userId`, `token`, `expiresAt`, `ipAddress`, `userAgent`, `createdAt`, `updatedAt`. The list query is `SELECT * FROM session WHERE userId = ? ORDER BY updatedAt DESC`. Library exposes it via `auth.api.listSessions({ headers })` (server) or `authClient.listSessions()` (client) — same call, same shape, same typed return.
- **Listing sessions — the call and the row UX.** `authClient.listSessions()` returns the array of `session` rows for the current user. Per-row UX surfaces: device guess (parsed from `userAgent` — "Chrome on macOS", "Safari on iPhone"; library may ship a helper, otherwise a small parser like `ua-parser-js` in a server util), approximate location (parsed from `ipAddress` — "San Francisco, CA" via a GeoIP lookup; conditional on privacy posture and product), `createdAt` ("signed in 3 days ago"), `updatedAt` ("last active 5 minutes ago"), and a "this session" badge on the row whose token matches the current cookie. The current-session detection is the senior reach — never let the user accidentally revoke the session they're using without an explicit confirm.
- **Revoke a single session — the call.** `authClient.revokeSession({ token })`. Library deletes the row; the cookie holding that token becomes invalid immediately (on next request the validating read returns null and the proxy redirects). Library accepts `id` or `token` depending on version — review the call shape. Senior reflex: confirm in the UI ("sign out the session on Chrome / macOS?") before issuing the revoke, especially for non-current sessions (irreversible).
- **Revoke all other sessions — the call and the trigger surface.** `authClient.revokeOtherSessions()` deletes every session except the current one. Library composes naturally with the `revokeOtherSessions: true` flag on `changePassword` (9.4.2) — same endpoint, called automatically on credential change, callable directly from the UI ("sign out everywhere except this device"). Senior reflex: this is the "I think someone got into my account" button; surface it prominently on `/settings/security` alongside the per-session list.
- **Revoke a user's every session — the full sign-out-everywhere.** `authClient.revokeSessions()` — deletes every row, including the current. Library clears the current cookie via `nextCookies`; the user is signed out and redirected to `/sign-in`. The senior call: this is the "yes really, log me out of everywhere now" button — distinct from `revokeOtherSessions`. UX: a single confirm ("you'll need to sign in again on this device too") then the call.
- **The cookie-cache stale-read window.** From 9.2.3: if the cookie cache is on (5-minute default `maxAge`), a revoked session's cookie still shows as valid for up to 5 minutes on the cached path. Senior trade: shorten `maxAge` for products where revocation latency matters, or disable cookie cache on the `/settings` subtree (the matcher in 9.2.3's config). Don't pretend the cache is gone — the staleness window is real and the user-facing copy should hint at it ("session was revoked; may take up to 5 minutes to take effect on currently-open tabs").
- **The active-sessions list as a security audit surface.** "When did you sign in from Bangalore? You're in Boston." That recognition is the user-facing detection layer for compromised credentials. Senior reach: send a "new device signed in" email through the Unit 8 pipeline when `signIn.email` (or any sign-in) lands a new IP+userAgent combination not seen before. Library may ship the hook; if not, the `signIn.after` callback (or equivalent) is the install point. Pair with the active-sessions list — the email links to the list with a "wasn't you?" CTA that revokes all other sessions and forces a password reset.
- **The "multi-session" plugin — different feature.** From 9.2.3: `multiSession()` lets one browser hold sessions for multiple user accounts simultaneously (Gmail account-switcher style). Different from "list my sessions across devices" — the active-sessions list is built-in and needs no plugin; multi-session adds a UI for switching the active session within a single browser. Name once, don't enable unless the product needs the account-switcher pattern.
- **`maximumSessions` — the cap knob.** Library option (typically `session: { maximumSessions: N }` or via the `multiSession` plugin) caps how many active sessions per user. When `N+1` arrives, the oldest by `updatedAt` is evicted. Senior call: most SaaS leaves uncapped; cap when compliance requires it or when one-account-many-share abuse is the threat. Default is fine for the year-1 build.
- **What happens server-side on revoke.** Single statement: `DELETE FROM session WHERE id = ?` (or `token = ?`). No further work — no token blacklist, no JWT rotation, no cache invalidation beyond the cookie-cache window. The opaque-session-id model (9.1.2) makes revocation cheap; the row's absence is the revocation. Senior contrast: a JWT-based session would need a denylist that every request consults, doubling the lookup cost — one more reason the cookie+row default earns its place.
- **The "remember me" sessions in the list.** `rememberMe: true` sessions (9.3.2) have longer `expiresAt`; `rememberMe: false` have session-only cookies with shorter server-side lifetime. Both show up in the list identically. Senior reflex: don't surface the distinction in the UX — users don't think in "remember me" terms; they think "the laptop I was on yesterday."
- **Watch-outs.** Revoking the current session by accident (no confirm, no current-session badge) — user gets booted mid-form; not surfacing approximate location and device — list is unreadable, no audit value; logging `ipAddress` / `userAgent` somewhere that leaks in error breadcrumbs (Sentry); putting the active-sessions list behind a flow that itself doesn't elevate — a borrowed laptop can revoke the legitimate user's other sessions; conflating `revokeOtherSessions` and `revokeSessions` in copy ("sign out everywhere" should be unambiguous about whether the current session goes too); assuming revoke is instant when the cookie cache is on — see the 5-minute window; building the list without a server-side query and trying to pass it from the client — `listSessions` is a server-trusted read, the array shouldn't be rebuilt client-side from any client-readable state; missing the "new device" notification email — the active-sessions list is detection only if the user happens to look.

What this lesson does not cover:

- Cookie cache configuration (9.2.3).
- Audit-log table that records sign-in / revoke events (Chapter 10.2).
- IP-based rate limiting and "new IP" anomaly detection beyond the email (Chapter 15.3).
- Account-takeover full incident-response runbook (out of scope; named in passing).
- The multi-session plugin's account-switcher UI (named, not built).

Estimated student time: 35 to 45 minutes. Mechanics + pattern hybrid; the active-sessions list is mechanical, the current-session detection and the cookie-cache staleness window are the teaching weight; a small comparison table for `revokeSession` / `revokeOtherSessions` / `revokeSessions` earns its weight.

## Lesson 9.4.4 — Account deletion with verification and cascading revocation

Topics to cover:

- **The senior question.** A user wants to delete their account. The 2026 baseline (GDPR Article 17, CCPA, every other right-to-erasure regime) makes "I want my account gone" a product obligation, not a feature. Three failure modes the senior closes: a stolen cookie clicking "delete" silently nukes the legitimate user; the cascade leaves dangling rows in tables that referenced the user (audit logs, billing records, content in shared spaces); the verification ceremony differs between password-account and OAuth-only users and the implementation collapses one path. Configure Better Auth's `user.deletion`, wire the verification email, ship the destructive-confirmation UI, name the retention discipline.
- **What `user.deletion` ships.** Disabled by default in the library. Three configuration knobs:
  - `enabled: true` — turns the surface on. Library exposes `authClient.deleteUser(...)` and the `/api/auth/delete-user/callback` route.
  - `sendDeleteAccountVerification: async ({ user, url, token })` — the callback the library invokes when verification is needed. Wire to send through the Unit 8 pipeline with a `DeleteAccountTemplate` React Email template (one CTA, expiry note, "if you didn't request this, ignore" line).
  - `beforeDelete: async (user)` / `afterDelete: async (user)` — hooks for app-owned work that must precede or follow the cascade (archival, audit-log row, downstream cleanup).
- **The two verification paths.**
  - *Password account.* `authClient.deleteUser({ password })`. Library verifies the password against `account.password` in constant time. On success, fires the cascade immediately — no email round-trip. The password *is* the fresh proof of intent. Senior default for any user with a `'credential'` row.
  - *No-password account (OAuth-only / magic-link-only).* `authClient.deleteUser({})` with no password. Library generates a CSPRNG token, stores hashed in a `verification` row keyed by `delete-account:<userId>`, invokes `sendDeleteAccountVerification`. User receives email with link to `/api/auth/delete-user/callback?token=...`. Click hits the library route, hashes incoming token, validates expiry and uniqueness, fires the cascade. The email click is the proof of intent.
  - Library picks the path automatically based on whether the call carries a password and whether a `'credential'` account exists.
- **The fresh-session requirement at this endpoint.** Library checks `freshAge` (9.2.3) on `deleteUser` — a session older than the window is refused unless the password is supplied (password path proves freshness directly). The reason: an attacker holding a stale-but-valid cookie can't delete without either re-authenticating or supplying the password. The elevation tier from 9.1.1 / 9.4.2 made operational on the maximum-blast-radius action.
- **What cascades when the delete fires.** In order:
  1. `beforeDelete(user)` runs — the senior archival hook. App code copies what must survive (audit-log identity snapshot, billing reference) into tables that don't FK-cascade from `user.id`.
  2. Every `session` row for the user is deleted (`ON DELETE CASCADE` on `session.userId` from 9.2.2).
  3. Every `account` row for the user is deleted — credential hashes, OAuth provider links, all gone.
  4. Every `verification` row keyed by the user's email or user ID.
  5. Plugin-owned rows tied to the user — `passkey` (9.3.7), `twoFactor` (9.3.6), `member` (Unit 10), `invitation` (Unit 10) — each plugin declares its own cascade.
  6. The `user` row itself.
  7. `afterDelete(user)` runs — analytics, third-party cleanup, anything that needs the user's identity but doesn't block deletion.
- **What the app retains, by senior call.** The cascade is the *easy* half; the retention decisions are the senior half.
  - *Audit-log rows* (Chapter 10.2.5) — append-only, referenced by `user.id` as a string, *never* FK-cascaded. The user is gone; the audit trail remains with the string ID dangling. Reason: audit history outlives the actor.
  - *Billing records* (Unit 12) — Stripe customers, invoices, payment ledger. Tax law typically requires 7-year retention. The `stripeCustomerId` lives on a side table or is denormalized; deletion nulls the link, the financial trail remains.
  - *Content in shared spaces* — a post in an org, a comment on someone else's invoice. Product decision: keep with author anonymized ("Deleted user"), or hard-delete. The schema is decided in Unit 6 / Unit 10; this lesson surfaces the decision, not the wire.
  - *Personal PII the law requires erased* — direct messages, private notes, anything the regulator calls "personal data." Cascade-delete or scrub at `beforeDelete`. GDPR's right-to-erasure is for *personal* data; business records have their own retention allowance.
  The senior framing: the schema's cascade vs. nullify vs. no-relationship choices from Chapter 6 decide what cascades; this lesson is the moment those decisions get exercised.
- **The `beforeDelete` hook in practice.** Three patterns the year-1 SaaS needs:
  - Write the `audit_logs` row recording the deletion (`actor`, `action: 'account.deleted'`, `targetUserId`, `timestamp`) before the user row is gone — so the audit query can still find context. Chapter 10.2 owns the schema; this lesson names the hook as the write point.
  - Copy the user's `stripeCustomerId` to a `deleted_users_billing` side table so Unit 12's webhook handlers still resolve the customer after deletion.
  - Anonymize the user's authored content — `UPDATE posts SET author_label = 'Deleted user', author_id = NULL WHERE author_id = <userId>`.
  - Refuse the delete with a thrown error if the user is the last owner of an org (Unit 10) — "transfer ownership first." Library propagates the throw as a rejected `deleteUser` Result.
- **The destructive-confirmation UI.** Below change-password / change-email / sessions on `/settings/security`, a "Danger zone" block visually separated (border, color, divider). The "Delete account" button opens a modal with:
  - The irreversibility statement ("permanently delete your account, all your data, and sign out every device — this cannot be undone").
  - A type-to-confirm input that requires the user's email — friction defense against muscle-memory clicks and accidental keyboard activation.
  - The password field (password accounts) or the "we'll email you to confirm" copy (OAuth-only).
  - A final "Delete my account" button, disabled until the confirm input matches and the password (if applicable) is non-empty.
  Senior reflex: never ship single-click destructive actions; the type-to-confirm friction is the structural defense against the borrowed-laptop / stolen-cookie cases.
- **The post-delete redirect and UX.** Password path: the cascade revoked the current session, the cookie is invalid, the proxy's next read returns null, redirect to `/sign-in` — but the senior UX intercepts this with a public `/account-deleted` page that confirms "your account has been deleted on [date]" and surfaces a "contact us" link for support inquiries. Email-verification path: the user clicks the link from email, lands on the library's callback route which fires the cascade, then redirects to the same `/account-deleted` page.
- **The notification email.** Distinct from the verification email. Verification *confirms intent*; notification *confirms completion*. Send the post-delete notification to the user's email (the address that was on file before the delete) immediately after `afterDelete` returns — "your account was deleted on [date]; if this wasn't you, contact support immediately." Senior baseline; not strictly required but standard 2026 practice. Skip when the verification email *is* the notification (i.e., the OAuth path where the click confirms the action — but even then, a separate confirmation reduces support load).
- **OAuth provider deletion considerations.** Removing the `account` row that links to Google/GitHub doesn't tell the provider — Google still has the user, the user just can't sign into this app with Google anymore. Two senior reaches:
  - If the app holds provider access/refresh tokens for ongoing API use (rare in pure-login flows from 9.3.8), call the provider's token-revocation endpoint in `beforeDelete` so the tokens stop working at the source.
  - For OAuth-only accounts going through the email-verification path, the verification email is sent to `user.email` (the address Google/GitHub reported at sign-in). If the user's provider email has changed since (and `user.email` is stale), the verification email goes to the wrong address. The senior reflex: surface "we'll email confirmation to ada@acme.com — change your email first if this isn't current" copy in the modal.
- **Org-owner caveat — named for Unit 10.** A user who owns an organization can't be cleanly deleted while the org has data — the cascade would orphan the org or leak it. `beforeDelete` refuses with a typed error; the UX surfaces "transfer ownership first" with a link to the org-ownership transfer flow (Chapter 10.2.4). For year-1 SaaS without orgs yet, ignore; once Unit 10 lands, the hook is mandatory.
- **The `verification` table at this surface.** Same table from 9.2.2, namespace `delete-account:<userId>`. Hashed CSPRNG token, library-default expiry (1 hour), single-use, row deletes on consumption. The pattern from 9.3.3 (verify-email) and 9.4.2 (change-email) reused yet again — one table, three flows.
- **Watch-outs.** Enabling `deleteUser` without wiring `sendDeleteAccountVerification` for OAuth users — the email-verification path silently fails; FK-cascading `audit_logs` from `user.id` (the audit history vanishes with the actor, defeats the audit purpose); shipping a single-click destructive button without the type-to-confirm friction (the borrowed-laptop disaster scenario); putting the archival in `afterDelete` instead of `beforeDelete` (the user row is gone, the archival can't read the data it needs); allowing the delete to land while the session is stale and the user is OAuth-only (the `freshAge` check is the only proof of intent on that path); calling a raw schema `DELETE FROM users WHERE id = ?` from a custom Server Action instead of `auth.api.deleteUser` (bypasses every library defense and every plugin's cascade); not sending the post-delete notification (silent destruction is a takeover-detection failure if the legitimate user wasn't the one who clicked); logging the verification token in URL analytics; cascading `stripeCustomerId` from `user` (loses the billing trail tax law requires); ignoring the org-owner case (orphaned org in Unit 10).

What this lesson does not cover:

- The `audit_logs` table schema and append-only discipline (Chapter 10.2.5).
- Stripe customer-record retention patterns and the side-table shape (Unit 12.2).
- Organization ownership transfer (Chapter 10.2.4).
- The full data-lifecycle / right-to-erasure compliance surface (Chapter 21).
- GDPR Article 17 / CCPA in legal depth (out of scope; the engineering shape is here).
- React Email template anatomy (Chapter 8.2).
- The `verification` table mechanics (9.2.2, 9.3.3).

Pattern + decision hybrid; a sequence diagram for the two-path delete flow (password-immediate vs. email-verification) earns its weight, and a small table mapping "what cascades / what's retained / why" is the load-bearing reference for the cascade-vs-retention decision.

## Lesson 9.4.5 — Browser security defaults: what the stack ships, what would undo them

Topics to cover:

- **The senior question.** Every SaaS post-mortem reaches CSRF and XSS eventually. What defaults does this stack already ship — React 19's auto-escaping, Next.js's same-origin shape, Better Auth's `SameSite=Lax` + `__Host-` cookie + `trustedOrigins` — and what code on the team's side would undo those defaults? The lesson is structural, not a header survey: name the defense, name the one footgun per category, and frame the full CSP / security-header configuration as Chapter 17's territory (named once here).
- **CSRF — the threat in one paragraph.** An attacker hosts `evil.com` with a form that POSTs to `app.example.com/api/transfer`. If the user is signed into the app and visits `evil.com`, the browser attaches the app's session cookie to the cross-site request. Without a mitigation, the action runs as the user. The 2026 mitigation is structural — the cookie itself refuses to ride on cross-site requests.
- **The default — `SameSite=Lax` on the session cookie.** From 9.2.3 / 9.1.2: Better Auth sets the session cookie with `SameSite=Lax` by default. The browser sends `Lax` cookies on top-level GETs (clicking a link to the app) but blocks them on cross-site POSTs (an attacker-form-driven POST from `evil.com`). The session cookie isn't attached, the action runs without a session, the action returns 401. The mitigation is the cookie attribute; the application doesn't need a CSRF token layer for the standard same-origin Next.js form / Server Action shape.
- **What would undo the default.** Flipping `SameSite=None` (some teams reach for it to "fix" a third-party embed and unflip everything else). Mounting auth endpoints under a domain whose cookie isn't `__Host-`-prefixed and adding `Domain=` so a sibling subdomain can attack. Building a cross-site browser flow (a marketing site that POSTs into the app) and "fixing" the cookie attribute instead of fixing the architecture. The senior reflex: any time `SameSite=Lax` feels like a problem, the architecture is the problem, not the attribute.
- **The Server Actions backstop.** Next.js 16 Server Actions add a built-in origin check — the framework refuses Actions whose `Origin` header doesn't match an allowed origin. A second layer behind `SameSite=Lax`. If a future browser quirk or a bizarre embed flow leaks a cross-site POST past `Lax`, the framework still rejects. Configured via `serverActions.allowedOrigins` in `next.config.ts` for the rare deliberate cross-origin Action call. Senior reflex: don't disable, don't widen casually, treat any addition as a security review.
- **The `trustedOrigins` allowlist — Better Auth's slot.** From 9.2.3: Better Auth refuses to set auth cookies on requests from origins outside `trustedOrigins`. Defaults to `[baseURL]`. Add mobile-webview / browser-extension origins explicitly when those clients exist; leave the default for a vanilla same-origin SaaS. Never `['*']`; the senior reads any addition.
- **CSRF tokens — the legacy layer this stack doesn't ship.** Cookie-to-header double-submit and synchronizer tokens are the pre-`SameSite` mitigation. Better Auth doesn't ship them by default in 2026 because `SameSite=Lax` plus the Server Actions origin check closes the threat. Name the older pattern so the student recognizes it in legacy code; don't add it to this stack.
- **The cross-origin endpoint — when CSRF tokens earn the call.** A genuinely cross-origin auth endpoint (mobile webview hitting `api.example.com` from a different origin) breaks `SameSite=Lax`'s assumption. Mitigations: keep the cookie SameSite=Strict for the highest-stakes endpoints; require a custom header on cross-origin requests (a header the attacker site can't forge without CORS preflight cooperation); switch the auth model for those clients to a Bearer token in `Authorization`. Senior call: solve the architecture first, layer tokens only if the architecture demands cross-origin.
- **XSS — the threat in one paragraph.** Attacker gets script into the page via untrusted input rendered as HTML. Script runs in the victim's origin, reads cookies (unless `HttpOnly`), reads `localStorage`, fires authenticated requests as the user. `HttpOnly` on the session cookie defends the cookie read; the rest is mitigated by not letting attacker input become HTML.
- **The default — React 19 escapes by default.** Every `{value}` in JSX is escaped — strings render as text, not HTML. The attacker's `<script>alert(1)</script>` becomes the literal string `&lt;script&gt;alert(1)&lt;/script&gt;` in the DOM. The structural defense is the framework's escaping; the app gets it free for every `{value}` it writes. Same discipline for `<a href={url}>` — React rejects `javascript:` URLs in JSX-rendered attributes (with a warning in dev, hard reject planned), closing the inline-script vector through href manipulation.
- **The footgun — `dangerouslySetInnerHTML`.** The one prop that bypasses the auto-escape. Writing `<div dangerouslySetInnerHTML={{ __html: userInput }} />` injects the string as HTML. Any path that lets attacker input reach that prop is an XSS hole. Senior reflexes: never reach for it without a load-bearing reason; when reaching for it, sanitize the input first (`DOMPurify` is the 2026 standard, server-side sanitization preferred since RSC); when sanitizing, allowlist tags/attrs (whitelist, not blocklist); when the source is "trusted" (your own CMS), still sanitize — the senior assumes every input is reachable by some path.
- **Sanitization shape.** `DOMPurify.sanitize(html, { ALLOWED_TAGS: ['p','br','strong','em','a','ul','ol','li','h1','h2','h3'], ALLOWED_ATTR: ['href'], ALLOWED_URI_REGEXP: /^https?:\/\//i })`. Run server-side in a Server Component (`isomorphic-dompurify` provides the Node API) so the un-sanitized HTML never reaches the browser. The sanitized string goes into `dangerouslySetInnerHTML`. Senior reflex: the sanitization config is part of the security posture — review it like a firewall rule.
- **The other XSS vectors React doesn't auto-close.** `href` / `src` with untrusted URLs (validate scheme allowlist — `https:`, `mailto:`, relative paths only); inline `style={{ ... }}` is fine, but `style` as a string (`<div style="background:url(javascript:alert(1))">`) only happens through `dangerouslySetInnerHTML` — already covered. `eval`, `Function`, `setTimeout(stringArg)` — never reach for any of them with user input; ESLint catches.
- **`HttpOnly` and the `localStorage` line.** `HttpOnly` (set by Better Auth on the session cookie) prevents JavaScript from reading the cookie. The trade: any auth credential the team puts in `localStorage` for "convenience" is readable by every script — a successful XSS reads it. The senior rule: session cookies stay `HttpOnly`; tokens never go in `localStorage`; if the architecture needs a JS-readable token, name the threat and pick the layer (in-memory only, short-lived, with refresh).
- **CSP — the defense-in-depth layer, named not configured here.** Content Security Policy headers (`Content-Security-Policy: default-src 'self'; script-src 'self'; ...`) close the gap when sanitization fails — even an injected `<script>` won't execute if the policy forbids inline scripts and only allows scripts from `'self'`. The right reach for production; the configuration depth (nonces, hashes, `strict-dynamic`, report-only mode for rollout) is Chapter 17.3's territory. This lesson names the header and the position in the stack — second layer behind escaping/sanitization, mandatory for SaaS handling sensitive data.
- **The other security headers — Chapter 17.3.** `Strict-Transport-Security`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `Cross-Origin-*` — the full security-header surface lands in Chapter 17.3. This lesson names them once: "Next.js's `next.config.ts` ships them via the `headers()` function; senior shape is `Referrer-Policy: same-origin` (don't leak the path to third parties), `X-Content-Type-Options: nosniff` (always), HSTS preloaded for production." Don't try to ship the full set here; honor the cross-reference.
- **Watch-outs.** Flipping `SameSite` to `None` to fix a cross-origin embed — undoes the whole CSRF defense; adding `Domain=example.com` to the session cookie so a marketing subdomain can read it — every subdomain is now an attack surface; using `dangerouslySetInnerHTML` with un-sanitized rich-text editor output; sanitizing client-side only — the un-sanitized HTML still shipped to the browser; putting auth tokens in `localStorage` for "convenience"; copying CSP from a Stack Overflow answer ("`script-src * 'unsafe-inline' 'unsafe-eval'`" defeats the entire point); writing the full security-header config in this chapter and conflicting with Chapter 17.3; allowing `target="_blank"` without `rel="noopener noreferrer"` (window-opener vector — React 19 helps but explicit is better); rendering `<a href={userUrl}>` without scheme validation — `javascript:` href slips through on older React versions and is a footgun review point on every PR.

What this lesson does not cover:

- The full security-header configuration in `next.config.ts` (Chapter 17.3).
- Content Security Policy depth — nonces, hashes, `strict-dynamic`, report-only rollout (Chapter 17.3).
- Subresource Integrity for third-party scripts (Chapter 17.3).
- Server-side request forgery, log injection, deserialization attacks (Chapter 17.3 / 17 generally).
- The OWASP top-10 survey — covered by example, not by enumeration.
- Cross-origin resource sharing (CORS) config for API endpoints exposed to other origins (named once; full coverage where the cross-origin product surface appears).

Estimated student time: 35 to 45 minutes. Concept + pattern hybrid; the "default + footgun" pairing for each category is the teaching weight; a diagram for "attacker form on evil.com → browser drops SameSite=Lax cookie → 401" earns its weight, and the sanitization snippet is the one inline example.
