# Chapter 20.1 — Error monitoring and structured logs

## Chapter framing

Chapter 20.1 is the first half of the observability story — the half that answers "what broke, when, and why" when the app is in production and the only artifact is what the system wrote down. By this point the student has shipped Server Actions, webhooks, background jobs, billing, auth, RBAC, rate limiters, and a cache invalidation discipline; every one of those surfaces fails in production in ways the dev loop never reproduces. The chapter turns scattered `console.log` and "it worked on my machine" into the two-pillar production discipline a senior actually runs: **Sentry** owns exceptions and the rich context around them (de-minified stack traces, the user / org / route who hit them, the breadcrumb trail), and **structured logs** own the request-shaped record of intentional events (auth gate hit, webhook received, action accepted, rate limit fired) carrying a request correlation ID so 3am triage can stitch the story from one search. Both flow off the box via Vercel Drains into a searchable destination; both are useless without the senior call about what gets logged and what stays out.

Threads that run through every lesson: two distinct surfaces — exceptions (Sentry) and events (logs) — conflating them is the failure mode; **everything carries the correlation ID** that flows from the edge through every log line, every Sentry breadcrumb, every audit row, every webhook delivery, so one search reconstructs the failure end to end; the user-message-vs-operator-message split from Chapter 17.1.2 decides which surface gets which detail (operator detail to Sentry / logs, user message stays safe); PII and secrets are excluded by structural enforcement (Pino `redact`, Sentry `beforeSend`), not by code-review hope, because GDPR posture from Chapter 17.2.4 reaches into observability; structured logs are JSON, never strings; the destination matters — platform defaults retain for hours, production-grade SaaS pipes to a searchable store; the live debugger earns its weight on the server-action-failed-and-the-logs-don't-tell-me-why case, with structured logs and Sentry breadcrumbs telling you where to attach. The chapter ships four teaching lessons plus a quiz; the project (Chapter 20.4) wires Sentry plus PostHog into the running app.

## Lesson 20.1.1 — Sentry: error capture, source maps, releases, and breadcrumbs

Topics to cover:

- **The senior question.** An exception just fired in production — a `TypeError` deep in a Server Action, a `PostgresError: violates foreign key constraint`, an unhandled rejection in a Trigger.dev job. What signal reaches the on-call engineer, and what context comes with it so the fix takes minutes? The lesson lands Sentry as the canonical exception sink, configured the way a senior actually configures it.
- **Why Sentry, why now.** In 2026 the SaaS market for application error monitoring has consolidated; Sentry covers JS/TS/Node/browser, integrates with Next.js 16's instrumentation, supports server- and client-side tracing, and ships a first-class `withSentryConfig` build wrapper. Alternatives exist (Rollbar, Bugsnag, Datadog APM if already a Datadog shop); name them once. The course's call: Sentry is the default — one integration, both runtimes, source-map upload built in.
- **The two surfaces Sentry owns — exceptions and (optionally) tracing.** Exceptions are the load-bearing case — every unhandled throw, every `captureException`, every React error-boundary catch. Tracing (spans, performance, RUM) is conditional and not enabled in the chapter's project. The reflex: errors first; tracing when the team is investigating latency, not before.
- **Install and wire — the May 2026 shape.** `pnpm add @sentry/nextjs`. The wizard (`pnpm dlx @sentry/wizard@latest -i nextjs`) writes `sentry.client.config.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts`; `instrumentation.ts` at the project root imports them so each runtime initializes before any user code; `next.config.ts` wraps in `withSentryConfig(nextConfig, { ... })` for the build-time pieces. Four-file shape; the wizard is the starting point and the diff is hand-edited from there.
- **Environment variables.** `SENTRY_AUTH_TOKEN` (build-time, source-map upload, server-only secret), `NEXT_PUBLIC_SENTRY_DSN` (runtime, public-safe by design), `SENTRY_ORG` / `SENTRY_PROJECT` (build-time, upload target). All flow through the `@t3-oss/env-nextjs` typed env from Chapter 1.4.
- **Source maps — readable stack traces in production.** Minified production bundles point at `chunks/a8f3e2c1.js:1:24091`; source maps map back to `app/.../page.tsx:42`. The build wrapper uploads maps to Sentry and *deletes the client-side maps from the deployed bundle* after upload; server-side maps stay in `.next/server/` because the Node runtime needs them at runtime. The customer-facing bundle never ships the map (no IP leakage), the Sentry UI shows the original code. Name `widenClientFileUpload` for the case the default upload pattern misses files.
- **Releases — tagging every error with the commit that introduced it.** Sentry's "release" is a build identifier (commit SHA by convention) attached to every event. The build wrapper sets `release` from `VERCEL_GIT_COMMIT_SHA` automatically on Vercel. The payoff: an error spiking after a deploy is filterable to "errors first seen in release `abc1234`," and source-map lookup uses the right map for that release. The discipline: ship a release identifier on every build.
- **Sessions and release health.** Sentry's "release health" tracks crash-free sessions per release; the wizard turns it on by default. The dashboard the team checks after every deploy ("did this release lower crash-free %?"). Defer the tour.
- **Breadcrumbs — the trail before the exception.** A breadcrumb is a small structured record of an event before the exception: a click, a navigation, a fetch, a console call, a custom `Sentry.addBreadcrumb({ category: 'auth', message: 'requireRole passed', level: 'info', data: { role: 'admin' } })`. Sentry collects ~100 automatically; the team adds crumbs at the seams that matter: every guard hit, every Server Action entry/exit, every webhook accept, every billing-side-effect dispatch. When the exception lands, the breadcrumb panel reads as a small log of "what the system was trying to do."
- **`beforeSend` and `beforeBreadcrumb` — structural PII / secret exclusion.** Every Sentry SDK runs every event through `beforeSend(event, hint)`; same for `beforeBreadcrumb`. The senior pattern: a centralized scrubber in the config files redacts known PII fields (`email`, `phone`, `address`), drops `Authorization` and `Cookie` headers, and removes secret-shaped patterns (`sk_`, `rk_`, JWT-like). Chapter 17.2.4's GDPR rule enforced at serialize-time, not code-review-hope. The project (20.4) wires it.
- **Capturing manually — when to call `Sentry.captureException` vs. throw.** Unhandled exceptions are captured automatically; the manual call is for cases where the error is intentionally caught and recovered but *still operationally interesting* — a webhook signature verification that failed (rejected, returned 401, the operator needs to know the rate is climbing), a non-fatal degradation (one of three notification channels failed). Pattern: `try { ... } catch (e) { Sentry.captureException(e, { tags: { surface: 'webhook', provider: 'stripe' } }); return rejected(...); }`. Capture caught-but-interesting errors with rich `tags`, never silently swallow.
- **`Sentry.setUser`, `setContext`, `setTag` — enriching events at the seam.** At the request boundary after auth resolves, attach `Sentry.setUser({ id: userId, orgId, role })` — *never* email or name; the id is enough to correlate. Every event captured during that request inherits the context. The senior pattern: a `withSentryRequestContext` helper in `lib/observability/sentry.ts` runs at the action / route-handler boundary, sets user + org + correlation-id. The audit-log surface from Chapter 10.2 and the action wrapper from Chapter 7.2 are the wire points.
- **The React error boundary integration.** `<Sentry.ErrorBoundary fallback={<Crashed />}>` catches React render errors and reports them with the component stack. Use it on the App Router's `error.tsx` segments to centralize capture, or wrap a critical subtree (the dashboard shell) for finer fallback. The fallback is the user-message side of the 17.1.2 split.
- **Server Action and route-handler instrumentation.** The Next.js SDK auto-wraps Server Actions and route handlers via `instrumentation.ts` — exceptions surface without manual `captureException`. The wrapper is automatic; the manual call is for caught-but-interesting cases only.
- **Sampling — when 100% is wrong.** For errors, sample at 100% (every exception matters). For transactions / tracing, sample at 10–25% on a busy app to control cost. The 2026 default: `tracesSampleRate: 0` on first wire (tracing off), `tracesSampleRate: 0.1` when the team starts using performance views. Errors are never sampled down.
- **The Sentry tunnel — bypassing client-side ad blockers.** `tunnelRoute: '/monitoring'` in `withSentryConfig` routes browser-side reports through your own domain so ad blockers don't drop them. The senior move that captures errors from users who'd otherwise be silent.
- **The deploy-time release pipeline.** On Vercel: `next build` runs, the Sentry wrapper uploads source maps tagged with the commit SHA release, deletes client maps, deployment goes live, the runtime SDK sends errors with `release: VERCEL_GIT_COMMIT_SHA`. Verification: trigger one deliberate `throw new Error('sentry-test')` from a Server Action, confirm the event lands within 60s, the stack trace shows TS lines not minified frames, the breadcrumb panel shows the auth-and-action trail.
- **Watch-outs:** committing the DSN as a server-side secret — it's public by design, use `NEXT_PUBLIC_SENTRY_DSN`; forgetting `SENTRY_AUTH_TOKEN` in production — source maps don't upload, no build failure flags it; logging exception messages to the structured log *and* throwing them so Sentry double-captures — capture in one place; leaving `tracesSampleRate: 1.0` in production — every request becomes a transaction event; `Sentry.setUser({ email })` — PII in every event, GDPR violation by structure, use `id` only; per-request `Sentry.init` — initialize once per runtime; expecting the wizard's defaults to be the production answer — the wizard is the start, the `beforeSend` scrubber and the release pipeline are the senior diffs.

What this lesson does not cover:

- The structured-log surface — Lesson 20.1.2.
- The log destination (Vercel Drains) — Lesson 20.1.3.
- Server-side debugging — Lesson 20.1.4.
- The user-message-vs-operator split itself — Chapter 17.1.2 owns it.
- The audit log as a discipline — Chapters 10.2 and 17.2.3.
- Sentry tracing / performance / RUM beyond a one-line mention — out of scope.
- Session replay — Sentry ships it; the course defers to PostHog session replay in Chapter 20.2.

Estimated student time: 60 to 75 minutes. Mechanics plus the senior config diffs land here; the project (20.4) wires the same shape.

## Lesson 20.1.2 — Structured logs, correlation IDs, and the PII boundary

Topics to cover:

- **The senior question.** A Server Action succeeded, then a webhook arrived and failed, then a background job retried twice and gave up, then a user opened a support ticket saying "the export didn't run." What's the trail that connects these five surfaces into one story at 3am? The lesson lands the structured-log discipline: every event is a JSON line, every line carries the correlation ID that runs through the request, the schema is consistent enough that `correlationId:"abc123"` returns the full timeline.
- **Sentry vs. logs — two surfaces, two jobs.** Sentry owns *exceptions* — unintended failure with a stack trace, small audience (errors-per-deploy scale), rich per-event context, alerting attached. Logs own *events* — intentional records of "the system did X" or "the system rejected Y," large audience (millions of lines per day on a small SaaS), simple flat structure, search-driven exploration. Routing all `console.log` to Sentry blows the quota and buries real errors; routing exceptions into the log stream loses stack-trace de-minification and breadcrumb context. Exception goes to Sentry, event goes to the log.
- **Why structured, not strings.** A string log (`'User 123 failed to update invoice 456 because of rate limit'`) is unreadable at scale: no field to filter on, no aggregation. A structured log (`{ level: 'warn', msg: 'rate_limit_rejected', userId: '123', invoiceId: '456', orgId: 'org_789', correlationId: 'req_abc', surface: 'auth_signin' }`) is a row in a table — filter, aggregate, alert. Every log is JSON; no string formatting; the fields are the schema.
- **The logger choice — Pino as the May 2026 default.** `pnpm add pino`. Pino's the high-throughput Node logger that ships JSON natively, supports child loggers for per-request context, and integrates with every searchable destination via line-delimited JSON. Alternatives (Winston, Bunyan) — named once. Pino in Node, never `console.log` past the dev loop. In the browser, intentional client events go to PostHog (Chapter 20.2); page exceptions go to Sentry. This lesson focuses server-side.
- **The `lib/observability/logger.ts` shape.** One module exports the base logger configured once: `pino({ level: env.LOG_LEVEL ?? 'info', formatters: { level: (label) => ({ level: label }) }, redact: { paths: [...], censor: '[REDACTED]' } })`. The redaction paths are the structural PII / secret exclusion — `authorization`, `cookie`, `req.headers.cookie`, `user.email`, `user.phone`, `password`, `token`, anything under a `secrets` key. Pino's `redact` config does this at serialize-time, before the line leaves the process, so a misroute can't leak.
- **The correlation ID — one ID, the whole request.** Every incoming request gets a correlation ID at the edge. On Vercel the platform sets `x-vercel-id`; middleware reads it (or generates `crypto.randomUUID()` for local dev), attaches it to per-request context, every log line in that request inherits it via a child logger. The same ID flows into the Sentry breadcrumb context (`Sentry.setTag('correlationId', id)`) so a Sentry event and the log lines link to each other. The correlation ID is *the* primary key of a production incident — every audit row, every webhook delivery row, every notification row, every log line carries it.
- **Per-request child loggers — AsyncLocalStorage as the carrier.** Node's `AsyncLocalStorage` lets the request context flow through async calls without prop-drilling. A `middleware.ts` or per-handler wrapper creates the context once; `getRequestLogger()` returns a child logger pinned to it. Every Server Action, webhook handler, background-job step reads the same child logger. Name `AsyncLocalStorage` once; the project chapter ships the helper.
- **The log schema — fields that earn their place.** Every line carries `level`, `msg`, `time`, `correlationId`, plus optional `userId`, `orgId`, `surface` (action / route / webhook / job), and `outcome` (`accepted`, `rejected`, `committed`). The `msg` is a short snake_case event name (`invoice_updated`, `webhook_received`, `rate_limit_rejected`), never a sentence. Fields beyond the standard set are surface-specific and live in a nested `data` object so the top-level shape stays consistent. The schema is the documented contract; adding a top-level field is a versioning decision.
- **The five log levels.** `debug` (dev-only, not shipped to prod by default), `info` (intentional event the system completed — "action succeeded," "webhook accepted"), `warn` (intentional rejection or degradation — "rate limit fired," "fallback used"), `error` (something failed the operator needs to know about; usually paired with `Sentry.captureException`), `fatal` (the process is going down — rare in a hosted Next.js app). Lean on `info` for the happy path, `warn` for the rejection-by-design, `error` for the surprises. Quiet logs are useless logs; verbose `info` at the seams is the point.
- **The "log inputs you'd want at 3am" rule.** At every state-changing seam, log the inputs that determine *whether the request was right*. For an action: `{ userId, orgId, action: 'updateInvoice', invoiceId, fields: ['amount', 'status'] }` — enough to answer "did the right user touch the right row" without storing old or new values. For a webhook: `{ provider, eventType, externalId, signatureValid, idempotencyOutcome }`. For a job: `{ jobName, payloadHash, attempt }`. Log enough that someone reading at 3am can answer "should this have happened?" without re-running the query.
- **The PII / secrets exclusion — same boundary as Sentry, enforced differently.** Pino's `redact` config replaces listed paths with `'[REDACTED]'` at serialize-time. Chapter 17.2.4's rule: log the *ids* and *types*, not the *values*. Never log passwords, tokens, session cookies, Stripe secrets, webhook signing secrets. The audit log (Chapter 10.2) follows the same rule and has the same structural enforcement.
- **What never goes in a log line.** Email addresses, phone numbers, mailing addresses, plain-text passwords, API keys, OAuth tokens, JWTs, session cookies, raw webhook signing secrets, full request bodies for any endpoint touching auth or billing, credit card details (PCI scope you don't want), child PII, anything that ends up on a Slack channel the support team reads. The `redact` list catches the routine cases; senior code review catches the rest.
- **The `data` payload and depth limits.** A nested object thousands of fields deep blows the line size and slows the search. Keep `data` shallow (one or two levels), under ~1KB serialized. Bigger payloads belong in S3 / R2 (Chapter 13.3) with the log line carrying the object key.
- **The canonical log sites.** The shape repeats: every Server Action wrapper from Chapter 7.2 logs entry (`action_entered`), validation (`action_validated` or `action_rejected`), commit (`action_committed`), with exceptions flowing to Sentry. Every webhook handler (Chapter 12.1) logs signature outcome, idempotency outcome, dispatch result. Every background job (Chapter 13.1) logs enqueue, attempt-start, attempt-end. Every notification dispatch (Chapter 14.1) logs channel, recipient (user id, never email), outcome. Six surfaces, one schema, one correlation ID.
- **The audit log relationship.** The audit log from Chapter 10.2 is a *persisted* structured log: a row in `audit_events` with user, org, action, target, outcome, written transactionally with the mutation. The application log is an *ephemeral* structured log: a JSON line streamed to the destination, retained per policy, useful for triage and trend analysis. Both carry the correlation ID. Every audit-worthy action writes to both — the row for compliance, the line for ops. The two don't replace each other.
- **Local dev vs. production output.** In local dev, pretty-print with `pino-pretty` for the human reader (colored, indented, schema unchanged). In production, raw JSON to stdout; the platform's drain forwards from there. Pretty-print transport gated on `NODE_ENV !== 'production'`. The schema is identical; only the presentation changes.
- **Sampling for high-volume routes.** A single hot route (dashboard polling, health check) writing one `info` line per request multiplies fast. Sample at 1–10% on truly noisy success cases (`if (Math.random() < 0.05) log.info(...)`), full-fidelity on warn / error. Never sample errors; sample only the I'm-healthy hum.
- **Watch-outs:** string-formatted logs unsearchable at scale; per-line `correlationId: crypto.randomUUID()` inside a handler instead of the AsyncLocalStorage carrier — every log line in one request has a different ID, the trail breaks; logging full request bodies "for completeness" — PII / secret leak; `console.log` left in production code — bypasses redaction and structured shape; capturing user's email as `userId` because "easier to search" — emails are PII, ids aren't; logging at `debug` in production — cost up, signal flat; treating logs as the alerting surface — alerts go on Sentry events and metric thresholds, not log-content matching; loggers initialized per-handler instead of once at the module — config drift; failing to redact the redaction allow-list itself (`tokenPrefix: '[REDACTED]'` plus `tokenSuffix: 'abc123'` is the leak by reconstruction).

What this lesson does not cover:

- The Sentry side — Lesson 20.1.1.
- The destination and Drains pipeline — Lesson 20.1.3.
- Live debugging — Lesson 20.1.4.
- The audit log table and its schema — Chapter 10.2.
- The user-message-vs-operator split — Chapter 17.1.2.
- PostHog product analytics — Chapter 20.2.
- Distributed tracing / OpenTelemetry spans — out of scope; Vercel Drains supports OTel for teams that need it.

Estimated student time: 50 to 60 minutes. The discipline is the lesson; the small Pino config and the AsyncLocalStorage carrier are the load-bearing wiring.

## Lesson 20.1.3 — Log destinations: Vercel Drains and reading production logs

Topics to cover:

- **The senior question.** The logger writes structured JSON to stdout. Where does it go from there, how long does it stay, and what does the workflow look like when an on-call engineer needs to answer "what happened to user abc at 14:32 UTC"? The lesson lands the destination pipeline and the searching discipline.
- **Why platform defaults aren't enough.** Vercel's built-in function logs surface for ~1 hour on Hobby, ~1 day on Pro, and aren't structured-search-friendly past basic text filters. For triage in the moment they're fine; for "what happened last week to the customers who churned" they're unusable. Production logs need a destination that retains for at least 30 days and supports field-based search.
- **Vercel Drains — the 2026 way off the platform.** Vercel Drains (GA in 2026) is the unified streaming primitive for logs, traces, web analytics, and performance metrics out of the deployment into the destination of choice. One Drain configuration, multiple data types. Logs flow as line-delimited JSON; traces as OpenTelemetry. Available on Pro and Enterprise; billed at $0.50/GB exported. Configure once, every log line lands in the destination automatically.
- **Destination choices.** The 2026 SaaS shortlist: **Axiom** (purpose-built for serverless / Vercel, generous free tier, native Drain integration), **Better Stack / Logtail** (good DX, alerting built in, fair pricing), **Datadog** (only if the team is already a Datadog shop), **AWS CloudWatch / GCP Cloud Logging** (only if compliance demands the cloud). The course's default: Axiom. Pick one, write the queries, don't shop around mid-incident.
- **Configuring the Drain.** Vercel dashboard -> Drains -> New Drain -> data type (Logs) -> destination (Axiom, Better Stack, or generic HTTPS endpoint with a token) -> environments (Production, Preview, Development) -> save. **Production-only by default** — preview deployments are noisy, dev is local. Turn on preview drains when a feature in preview is actively being debugged; turn them off after.
- **Trace and log correlation — what Vercel adds for free.** Vercel enriches log entries with `traceId` and `spanId` on the way out. When the team turns on tracing, the IDs connect logs to traces in the destination without manual instrumentation. The platform is attaching distributed-tracing primitives; the application code only needs the correlation ID for its own request shape.
- **Pricing and what it means for discipline.** $0.50/GB exported is cheap until a `debug`-level firehose makes it expensive. `LOG_LEVEL=info` in production (not `debug`), sampling on noisy success endpoints, a monthly Drains-cost dashboard the team watches. Structure cost in proportion to value; a million `health_check_ok` lines a day cost real money and tell nothing.
- **OpenTelemetry as the optional next step.** Vercel Drains support OTel for distributed tracing — name once. The course's stack ships logs and Sentry errors; teams that need request-flow visualization across services add OTel later.
- **The "reading production logs in anger" workflow.** A senior runs this every time: (1) pull the correlation ID from the user's report — the user-facing error UI shows it as "Error ID: abc123." (2) Open the destination, run `correlationId:"abc123"` — the request's full timeline appears as 5–20 lines ordered by time. (3) Find the `level:warn` or `level:error` line — the rejection or failure shows in `msg` and the `data` payload. (4) Cross-reference the Sentry event with the same correlation ID — breadcrumb and stack trace are there. (5) Decide: bug, expected rejection, infrastructure blip. Total time from report to root cause: 2–5 minutes when the discipline holds, 2 hours when it doesn't.
- **Exposing the correlation ID to the user — closing the loop.** Chapter 17.1.2's user-message side says "what reaches the user is operator-safe." The senior addition: the user message includes the correlation ID as a short tag (`Error ID: abc123`). Support copies it into the ticket; engineering pastes it into the log search. Every error UI renders `{correlationId}` as a small monospace tag.
- **Common search patterns.** `correlationId:"req_abc"` — the one-request timeline. `level:error AND surface:webhook AND time:[now-1h]` — the last hour of webhook failures. `msg:"rate_limit_rejected" AND time:[now-15m]` — the rate-limit pulse during a suspected attack. `userId:"usr_123" AND time:[now-24h]` — what one user did today. Document the top five in the team's runbook.
- **Alerting — what gets wired to a Slack channel.** Sentry alerts on new error issues and error spikes (its job). Destination alerts (Axiom / Better Stack) fire on log-volume anomalies and rate thresholds — `rate(level:error) > 10/min` is the canonical one. Not every line is an alert; < 10 alert rules. More than that and the team will train itself to ignore them.
- **Retention and the GDPR thread.** Destinations retain by tier (Axiom configurable, Better Stack 7–30 days on most plans). Chapter 17.2.4's GDPR posture: redaction at write-time keeps PII out of the logs, retention bounds bound the worst case. Audit the redaction quarterly by sampling a hundred lines.
- **Local dev — read your own logs.** Pretty-printed Pino lines stream to the terminal — the dev's log destination is the terminal scrollback. No Drain, no destination cost. Develop with the same `info`-rich surface you'd ship; don't write logs that exist only for production.
- **Watch-outs:** Drain attached to Preview without a reason — bills climb, signal stays; alerts on every log line — alert fatigue; relying on the platform's 1-hour log view for anything past today — destination is what production-grade triage uses; correlation ID generated in middleware but not propagated into AsyncLocalStorage — every log line gets a fresh value, the search returns nothing; `debug` in production "just for now" — left on, the bill is real next month; destination running out of quota silently — Drain backs up, lines drop, the engineer concludes the bug didn't fire (it did); the support team copy-pasting context around the error ID into Slack without redacting — the correlation ID is fine, the surrounding context might not be.

What this lesson does not cover:

- Sentry — Lesson 20.1.1.
- The log discipline itself — Lesson 20.1.2.
- Live debugging — Lesson 20.1.4.
- Distributed tracing beyond a one-line mention — out of scope.
- The destination's UI in depth (Axiom's APL query language, Better Stack's dashboards) — out of scope; the queries shown are universal.
- Self-hosted ELK / Loki stacks — out of scope; the course's call is managed.

Estimated student time: 35 to 45 minutes. The destination-and-workflow lesson; the discipline from 20.1.2 needs somewhere for the lines to land.

## Lesson 20.1.4 — Live debugging Server Actions and route handlers

Topics to cover:

- **The senior question.** A Server Action returns the canonical rejection ("Something went wrong"); the structured log shows `action_rejected` with a correlation ID; the Sentry event shows a stack trace pointing at a Drizzle query that built a `WHERE` clause from a variable that turned out to be `undefined`. Logs and Sentry told you *where*; they didn't tell you *why* the variable was `undefined`. The lesson lands the live-debugger workflow that complements the observability surface — when to attach, how to attach, and what the breadcrumb-plus-log trail tells you about where the breakpoint should sit.
- **When the live debugger earns its weight.** Most production triage happens from logs and Sentry alone — the correlation ID, the breadcrumb trail, the stack trace are enough. The debugger earns its weight on **reproducibility-locally cases**: a Server Action that fails in a way the test suite doesn't cover, a webhook handler whose signature verification falls over on a specific payload shape, a server-side render that throws only when the session has a particular RBAC mix. Read logs first, pull the correlation ID, reproduce the input shape, attach the debugger to the local server, set the breakpoint where the Sentry stack trace points, step.
- **`node --inspect` and the Next.js dev server.** Next.js's `dev` command supports `NODE_OPTIONS='--inspect' next dev` — Node opens an Inspector port (default 9229) the debugger attaches to. Show the package.json script pattern: `"dev:debug": "NODE_OPTIONS='--inspect' next dev"`. The debug script lives next to the regular dev script.
- **Attaching from VS Code — the `launch.json` shape.** A `.vscode/launch.json` configuration with type `'node'`, request `'attach'`, port 9229, plus `skipFiles: ['<node_internals>/**']`. One config, hit F5, breakpoint on the line the stack trace pointed at. Senior addition: a second config that *launches* `next dev` directly with `runtimeArgs: ['--inspect']` so the workflow is one keypress.
- **Attaching from Chrome DevTools.** `chrome://inspect` lists the Node target; clicking opens a DevTools window with full breakpoint, watch, and console support. Useful when VS Code isn't open; the IDE flow is the default.
- **The Server Action breakpoint workflow.** The action's source in `app/.../actions.ts` exports an `async function` with `'use server'`. Set a breakpoint on the first line of the function body. The action is invoked from `<form action={...}>`; the server runs the function on submit. The debugger hits the breakpoint before any code runs; step over the auth gate, the Zod parse, the Drizzle query. Inspect parameters and the AsyncLocalStorage context (correlation ID, user / org IDs). Combine with the structured-log entry to confirm "what the log line reports matches what the variable actually holds."
- **The route-handler and webhook breakpoint workflow.** Route handlers in `app/api/.../route.ts` work the same way — breakpoint on the first line of `export async function POST(request)`, trigger from `curl` or the in-app client, step. For webhooks the trigger is the provider's test-event tool (Stripe CLI's `stripe trigger`, Chapter 12 pattern); the payload shape is real, the debugger sees it before idempotency or signature verification runs. The debugger is symmetric across Server Action and route handler.
- **The "server action failed and the logs don't tell me why" workflow.** (1) Pull the correlation ID from the user's error UI or Sentry event. (2) Open the destination, find the request's log trail; note `surface`, `userId`, the `data` payload on the rejection line. (3) Open the Sentry event; read the breadcrumb trail (`requireRole_passed`, `zod_parsed`, `drizzle_query_started`) and stack trace. (4) Stack trace points at line N of `app/.../actions.ts`. (5) Reproduce locally: sign in as a user with the same RBAC mix, replay the form with the same input from the log's `data`. (6) Set a breakpoint at line N; submit; step. (7) The bug is the `undefined` `orgId` because the breadcrumb shows the gate passed on the user's *primary* org but the action read from a stale URL parameter pointing at a different org. (8) Fix the gate or the parameter source. (9) Add a log line at the missing step so next time the breadcrumb shows the gap directly. The debugger is the last 20% — the logs and Sentry are the first 80%, and the post-fix log addition closes the loop so the next bug of this shape is read in the logs alone.
- **How logs and Sentry breadcrumbs interleave with the debugger.** The breakpoint is hit; the debug console writes `getRequestLogger().info({ checkpoint: 'A', data: { orgId } })` into the local dev terminal *at the breakpoint position*. When stepping doesn't explain the value, drop a log inline from the debugger so the production version of the same flow surfaces the same line. The bug-fix PR includes the new log line; production triage of the next sibling bug starts there.
- **Conditional breakpoints.** A handler runs ten thousand times in a dev session and the bug fires once. Conditional breakpoints (`userId === 'usr_abc'`, `correlationId === 'req_xyz'`) pause only on the matching request. Condition on the correlation ID from the failing production case, replay the input shape locally, the debugger hits when the bug fires.
- **Log points — debugging without pausing.** VS Code log points: gutter annotations that log a templated message every time the line executes, without breaking. Useful for understanding flow without stopping the server. Less ergonomic than a real log line; use for transient exploration, structured logs for the durable surface.
- **Edge runtime — the debugger limitation.** Next.js 16's Edge runtime (V8 isolates, not Node) doesn't support `node --inspect` directly; debugging edge code means running the same logic under the Node runtime locally or relying on logs alone. Keep auth gates and Server Actions on Node by default; the Edge runtime is for narrowly-scoped middleware.
- **Production debugging — what's *not* on the table.** Attaching a debugger to a production Vercel deployment isn't a workflow; function instances are ephemeral, ports aren't exposed. Production debugging is logs, Sentry, and reproducing locally — exactly the workflow above. Don't shop for a "live attach to prod" tool; the discipline is reproduction.
- **The breakpoint that should have been a log line.** Every time the debugger reveals a value the senior reaches for, ask: "would I want this in the production log?" Often yes — `requireRole_passed` should log the role, the action entry should log the parameter shape, the webhook accept should log the idempotency outcome. The loop: bug found by debugger -> log added at the gap -> next sibling bug found by log search alone.
- **Watch-outs:** committing the `--inspect` flag into the production `start` script — opens a debug port on production; debugger paused on a long-running breakpoint while another dev hits the same dev server — both stall; `console.log` peppered through the code during debugging and forgotten in the PR — bypasses the structured log surface and redaction; relying on the debugger for every production bug — slow, the discipline is logs first; conditional breakpoints on values the breakpoint itself mutates — confusing pause behavior; attaching from VS Code without `sourceMaps: true` — breakpoints land on the wrong line of the transpiled output; debugger left attached when running the test suite — slow, sometimes hangs.

## Lesson 20.1.5 — Chapter quiz

Top 10 topics to quiz:

- The two-surface split — Sentry for exceptions (rich per-event context, small audience, alerting attached), structured logs for events (JSON, large volume, search-driven) — and why conflating them is the failure mode.
- Source maps in production — uploaded to Sentry at build time, client maps deleted from the bundle after upload, server maps kept for runtime reporting, `SENTRY_AUTH_TOKEN` required.
- Releases tagged from `VERCEL_GIT_COMMIT_SHA` so errors filter by deploy; the "errors first seen in release X" workflow.
- Breadcrumbs as the trail before the exception — automatic capture (clicks, fetches, navigations, console) plus custom seam-of-interest crumbs (auth gate, action entry, webhook accept), enriched via `Sentry.setUser({ id })` and never email.
- `beforeSend` and `redact` as structural PII / secret exclusion — Chapter 17.2.4 enforced at serialize-time, never code-review hope.
- Correlation IDs flowing from edge through AsyncLocalStorage child loggers — one ID for the whole request, the same ID on Sentry tags, audit rows, and every log line; the user-facing error UI exposes it as "Error ID."
- The structured-log schema — `level`, `msg` (snake_case event name), `correlationId`, `userId`, `orgId`, `surface`, `outcome`, plus a shallow `data` payload; the schema is the contract.
- "Log inputs you'd want at 3am" — log ids, types, outcomes; never values that are PII, secrets, or full request bodies; audit row and application log line both carry the correlation ID.
- Vercel Drains as the way off the platform — Logs-to-Axiom (or Better Stack) on production-only by default, $0.50/GB as the cost lever for level and sampling discipline.
- The "production logs in anger" workflow — correlation ID from user report -> destination search -> Sentry event lookup -> reproduce locally -> attach debugger if needed -> add the missing log line in the fix so the next bug surfaces in the search alone.
