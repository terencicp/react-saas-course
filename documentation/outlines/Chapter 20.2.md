# Chapter 20.2 — Product analytics

## Chapter framing

Chapter 20.2 is the second half of the observability story — the half that answers "what are users doing, what's working, and which slice of users sees which version of the product." Chapter 20.1 owns failure (exceptions to Sentry, events to structured logs); this chapter owns the intentional-behavior signal that no log line carries and no Sentry event surfaces. By this point the student has shipped a real SaaS — auth, billing, RBAC, webhooks, jobs, notifications, a cookie consent gate from Chapter 17.2.5 — and the product question is now "are users adopting the feature we just shipped, where do they drop off, and can we roll the next change to 10% of orgs before everyone." The chapter lands the two-product split: **Vercel Analytics** as the zero-config default for traffic and Core Web Vitals (covered through Speed Insights — performance metrics live here, the Core Web Vitals optimization work is Chapter 20.3); **PostHog** as the conditional power tool the moment the team needs product-event analytics, feature flags, experiments, or session replay.

Threads that run through every lesson: the default-before-conditional posture — Vercel Analytics covers the "how much traffic, which pages, what's slow" question without configuration and stays that way; PostHog earns its weight only when a question Vercel Analytics can't answer arrives (per-event funnels, flag-gated rollouts, recorded sessions, A/B tests); the minimum-viable-stack philosophy — PostHog folds four tools (analytics, flags, replay, experiments) into one platform and one consent gate, which beats Mixpanel + LaunchDarkly + Hotjar + Optimizely on integration cost alone; **everything non-essential is gated by the Chapter 17.2.5 cookie consent gate** — events fire only after `analytics` consent, session replay loads only after `replay` consent, the same `useConsent()` hook owns all of them; the user/org identity model from Chapter 8 and Chapter 10 maps to PostHog's `distinct_id` + `group` model — log in identifies, log out resets, the org is a group; PII discipline from Chapter 17.2.4 carries — id and role, never email or name, in PostHog events just like Sentry events; feature flags are not just rollout toggles, they're the structural enforcement that makes "kill switch on a bad change" a one-click action; experiments are flags with a measurement plan attached, and the measurement plan must be written before the flag is created. The chapter ships five teaching lessons plus a quiz; the project (Chapter 20.4) wires PostHog end-to-end into the running app.

## Lesson 20.2.1 — Vercel Analytics: the zero-config default

Topics to cover:

- **The senior question.** The app is live. The team needs to answer "is anyone using this, which pages are hot, how fast is the experience, where are users coming from" without spending a sprint on instrumentation. The lesson lands Vercel Analytics + Speed Insights as the default that covers those questions out of the box and tells the student where the default stops earning.
- **Two products, one platform.** Vercel ships two analytics surfaces: **Web Analytics** (page views, visitors, top pages, referrers, country, device, browser, OS — the "who's visiting and what are they reading" view) and **Speed Insights** (Core Web Vitals — LCP, INP, CLS — sampled from real user devices, the Real-User-Monitoring view). Different SDKs, different dashboards, one platform. Both run privacy-first: no cookies, anonymized data, GDPR-compliant by default, so neither is gated by the cookie consent banner.
- **Why these are the defaults, not the destination.** Both deliver immediate signal for one `pnpm add` and one component import. Cost: free tier covers small SaaS (Web Analytics free up to a generous monthly event quota on Hobby/Pro, Speed Insights free on Pro). Trade-off: Web Analytics aggregates traffic, it doesn't follow users — no funnels, no per-user behavior, no event semantics past page views and (on Pro) a flat `track('event_name', { data })`. The moment the question is "what fraction of users who saw the pricing page started checkout," Vercel Analytics is the wrong tool — that's the 20.2.2 trigger to PostHog.
- **Install — the May 2026 shape.** `pnpm add @vercel/analytics @vercel/speed-insights`. In the App Router root layout, render `<Analytics />` and `<SpeedInsights />` from their respective entry points. Both are client components and self-mounting; no provider, no configuration unless the team needs to override. Vercel's Agent installer can wire both automatically — name it, don't lean on it; the senior writes the diff by hand once to know what landed.
- **Web Analytics — what the dashboard shows.** Top pages by views, unique visitors over time, referrer breakdown (which marketing source drove which page), country distribution, device/browser/OS share. Vercel hashes visitor identity daily; no cross-day user tracking. Filters by path, country, device. The marketing team's first dashboard; the engineering team's traffic sanity check after a deploy.
- **Custom events on Vercel Analytics.** Pro and Enterprise plans expose `track(name, properties?)` from `@vercel/analytics`: `track('signup_completed', { plan: 'pro' })`. Useful for a small number of business-critical conversions when PostHog hasn't been added yet (or when the team has decided not to add PostHog). The trade-off: events aren't joinable to users, funnels aren't a primitive, retention isn't a primitive — `track()` is closer to a Web-Analytics-flavored counter than a product analytics event. If the team is reaching for more than three or four `track()` calls, the 20.2.2 trigger has fired.
- **Speed Insights — what RUM gives you.** Core Web Vitals sampled from real user sessions: LCP (Largest Contentful Paint, the load-feel metric), INP (Interaction to Next Paint, the input-responsiveness metric that replaced FID), CLS (Cumulative Layout Shift, the visual stability metric). Each metric reports the 75th-percentile value per route, with a histogram and trend over time. Filterable by route and device. The dashboard surfaces "your /pricing route's LCP regressed from 1.8s to 2.6s on mobile after the last deploy" — actionable signal that lab tools (Lighthouse runs in CI) can't produce because they don't see real devices and real network conditions.
- **The RUM-vs-lab split.** Lighthouse is lab (Chapter 20.3 owns the lab tooling): controlled environment, deterministic, useful for catching regressions in CI. Speed Insights is field: real users, real devices, real networks, percentile-distributed. Lab finds bugs; field tells you whether bugs reach users. Both run; neither replaces the other.
- **Cost and quota discipline.** Web Analytics: free monthly event quota; events past the quota meter at a small per-event price. A custom-event-heavy app blows through the quota fast — another argument for PostHog when the question is event-rich. Speed Insights: free on Pro; the cost lever is the `sampleRate` if traffic is very large.
- **What Vercel Analytics doesn't cover — the boundary to PostHog.** No per-user behavior trail. No funnel definition ("step 1 -> step 2 -> step 3, with drop-off rates"). No retention analysis. No cohort definition. No feature flags. No session replay. No experiments. Not designed for product-management questions; designed for traffic and performance questions. When the PM asks any of the above questions, PostHog is the answer (Lesson 20.2.2).
- **The GDPR posture and why neither needs the consent gate.** Both products anonymize at the edge — no cookies, no fingerprinting, no cross-session user identity. The EU regulators' guidance (and Vercel's documented stance) is that anonymized, aggregate analytics don't require consent in most EU jurisdictions. The cookie consent gate from Chapter 17.2.5 is for *non-essential* third parties that *do* identify; Vercel Analytics fires on every visit regardless of consent. PostHog is the opposite — wait for consent (Lesson 20.2.5).
- **The trigger to graduate to PostHog.** Three observable signs the question Vercel Analytics can't answer has arrived: the PM asks for a funnel, the team wants to roll a feature to 10% of orgs, support hands engineering a vague "users are confused on the dashboard" report that a session replay would resolve in 90 seconds. Any one is the 20.2.2 threshold.
- **Watch-outs:** wiring `track()` for product-grade event analytics — wrong tool, no joins to users, the work has to be redone in PostHog; assuming Speed Insights replaces Lighthouse in CI — they're complementary, the lesson 20.3.5 lab pass still runs; turning on Speed Insights with default sample rate on a million-page-views-a-month app — review the quota; treating Vercel Analytics as "we have analytics, we're done" — the PM's first funnel question is days away.

What this lesson does not cover:

- PostHog and the moment it earns its weight — Lesson 20.2.2.
- Core Web Vitals optimization mechanics (LCP/INP/CLS each, what moves the needle) — Chapter 20.3 owns the optimization work.
- Lighthouse and lab tooling — Chapter 20.3.
- Cookie consent mechanics — Chapter 17.2.5.
- Sentry tracing for performance — out of scope; Speed Insights is the default RUM.

Estimated student time: 35 to 45 minutes. Setup/wiring + reference, short by design — the default is meant to be cheap to land.

## Lesson 20.2.2 — When PostHog earns its weight

Topics to cover:

- **The senior question.** Vercel Analytics is wired and answering traffic and performance questions. The PM walks in with: "what fraction of trial signups upgrade within 14 days, and is the new onboarding flow improving that number." Vercel Analytics can't answer; the team needs an event-level analytics platform with user identity, funnels, flags, and ideally session replay so support can stop guessing. The lesson lands PostHog as the senior call, framed by the four jobs that justify adding a second analytics tool.
- **The threshold — what PostHog earns its weight against.** Four signals, any one fires the trigger: (1) **product-event analytics** — funnels, retention, cohorts, per-user event timelines that Vercel Analytics doesn't model; (2) **feature flags** — gradual rollouts, kill switches, per-segment targeting, beta gates that a deploy + revert cycle can't deliver fast enough; (3) **session replay** — recorded user sessions that turn "the dropdown is broken on Safari" reports into a 90-second video; (4) **experiments / A/B tests** — measured changes with statistical-significance reporting attached. One signal justifies PostHog; the value compounds when several land in the same quarter.
- **The minimum-stack call — why one platform beats four.** The historical SaaS shape was Mixpanel (analytics) + LaunchDarkly (flags) + Hotjar (replay) + Optimizely or VWO (experiments) — four contracts, four SDKs, four consent banners, four user-identity stitches. PostHog ships all four in one platform with one SDK, one consent gate, one user identity, one billing surface. For an early-stage SaaS the integration cost saved is the load-bearing reason; for a series-A team the same logic holds until the team has explicit reasons to pull one tool out. The 2026 default: PostHog for all four jobs from day one of the trigger firing.
- **The competitive landscape, named once.** Mixpanel and Amplitude are credible product-analytics-only alternatives — pick if the team has heavy event analytics needs *and* an existing flag/replay vendor. LaunchDarkly is the flag heavyweight — pick if the team needs enterprise approval workflows, audit trails specific to flags, or compliance certifications PostHog doesn't meet. Hotjar/FullStory for replay-only; Statsig as a PostHog-shaped competitor leaning heavier on experiments. Datadog RUM if the team is already a Datadog shop. The course's call: PostHog covers the four jobs at the scale a starting SaaS operates at; revisit at series B.
- **Hosted vs. self-hosted.** PostHog Cloud (US or EU region) is the default — managed, fast, the EU region keeps EU user data in the EU and removes a GDPR data-transfer question. Self-hosting is open source and supported; reach for it only when compliance demands the data never leaves a specific cloud account, or when monthly event volume makes managed cost the dominant line item. The course's call: PostHog Cloud EU (or US matching the company's data-residency posture).
- **Pricing shape and what to know up front.** PostHog meters per-product: events, recordings, flag decisions, experiments — each with a generous free tier and per-unit pricing past it. The cost levers worth knowing: session replay is the biggest line item by far (recordings are heavy); event sampling and replay sampling are the levers; feature-flag local evaluation (the server-side SDK evaluating flags without a network round-trip per check) keeps flag-decision cost flat. The senior posture: turn on the products you need, leave session replay sampled, watch the bill the first month.
- **The four products, one-line each.** **Analytics** — event capture, autocapture (clicks, pageviews, form submits without instrumentation), funnels, retention, cohorts, dashboards. **Feature flags** — boolean and multivariate flags, per-user / per-group targeting, gradual rollouts, local evaluation server-side, the kill switch surface. **Session replay** — recorded user sessions with DOM, network, console, and console-error tracks; the support-and-UX-debug tool. **Experiments** — flags with a measurement plan and significance reporting; the disciplined A/B-test surface. Each gets its own lesson (20.2.3 events and identity, 20.2.4 flags and experiments, 20.2.5 wiring + session replay).
- **PostHog and the existing stack.** PostHog ships side-by-side with Vercel Analytics (different jobs, no overlap) and Sentry (different jobs, partial overlap on browser errors — keep Sentry for errors, PostHog for behavior). The consent gate from Chapter 17.2.5 owns *all* PostHog calls; the user identity from Chapter 8 maps to `posthog.identify(userId)`; the org from Chapter 10 maps to `posthog.group('organization', orgId)`. PostHog isn't a replacement, it's an addition with explicit boundaries.
- **The "don't reach for it yet" cases.** The team is pre-launch with zero users — PostHog has nothing to measure, wire it the week before launch. The team's question is "is the marketing site fast" — Vercel Analytics + Speed Insights, not PostHog. The team has one product-event question — `track()` on Vercel Analytics may be enough for a quarter while the team builds the case for PostHog. The trigger is "we keep hitting questions we can't answer with what we have," not "the cool startups use PostHog."
- **The decision path, summarized.** Is the question traffic/performance? Vercel Analytics. Is the question per-user behavior, funnels, retention, or cohorts? PostHog Analytics. Is the question "roll this to 10% first"? PostHog Flags. Is the question "I can't tell what users are doing"? PostHog Session Replay. Is the question "did this change move the metric"? PostHog Experiments. Each branch is one of the four jobs that justify the platform.
- **Watch-outs:** wiring PostHog without the consent gate — fires on every visit, GDPR violation; running PostHog *and* Mixpanel *and* LaunchDarkly because the team didn't consolidate — four bills, four banners, two stitched user identities; turning on session replay with 100% sample rate on launch — the bill arrives the next month; treating feature flags as a deployment substitute — flags decay if not retired, the 20.2.4 lifecycle discipline matters; using PostHog as the errors surface — Sentry's job, PostHog doesn't de-minify stacks or send alerts on new issues the way Sentry does; identifying users by email — PII in every event, same rule as Chapter 20.1.

What this lesson does not cover:

- The PostHog event mechanics — Lesson 20.2.3.
- Flags and experiments — Lesson 20.2.4.
- Wiring and the consent gate integration — Lesson 20.2.5.
- Vercel Analytics — Lesson 20.2.1.
- The Core Web Vitals work — Chapter 20.3.
- Sentry — Chapter 20.1.

Estimated student time: 30 to 40 minutes. Decision lesson; threshold up front, alternatives compared, the four jobs as the load-bearing frame.

## Lesson 20.2.3 — PostHog events: autocapture, identify, and custom events

Topics to cover:

- **The senior question.** The team has decided PostHog earns its weight. Before any flag or replay or experiment, the question is: which events does the app *send*, what does each event carry, and how do those events tie back to the user and org the request belongs to? The lesson lands the event model that the rest of PostHog rides on.
- **The PostHog event model, one paragraph.** A PostHog event is a named record (`name: string`) with a `distinct_id` (the user identifier), a `timestamp`, and a `properties` object. Groups (orgs, accounts, teams) attach to events via `$groups: { organization: orgId }`. Funnels, retention, cohorts, and experiments all derive from queries over this event stream. The mental model: each event is a row in a wide table; PostHog handles the schema-on-read.
- **The two SDKs — `posthog-js` (browser) and `posthog-node` (server).** `pnpm add posthog-js posthog-node`. The browser SDK autocaptures clicks, pageviews, and form submits and exposes `posthog.capture()`. The server SDK has no autocapture (no DOM) and is the source of record for events that must fire on commit — checkout completed, plan changed, invitation accepted. The senior call: events that represent *state changes* fire from the server inside the Server Action / route handler, after the database commit; events that represent *intent* (button clicked, page visited, form started) fire from the browser. Twin sources, clear boundaries, no duplicate counting.
- **Autocapture — the cheap baseline.** With one SDK install, every click, pageview, form submit, and `rageclick` on the site is captured with the DOM ancestor selector chain. The PM gets a heatmap of clicks on day one without engineering instrumentation. Autocapture is the *starting* layer; named events are the *senior* layer. The reason: autocapture's selector chain (`.div > .form > .button:nth-child(2)`) breaks the moment the markup changes, so funnels built on autocapture rot. Funnels worth alerting on are built on named events.
- **Named events — the contract.** The naming convention worth committing to: `noun_verbed` (`invoice_created`, `subscription_upgraded`, `invitation_accepted`), past tense, lowercase, underscore-delimited. The reason: events are *things that happened*. Same shape as the structured-log `msg` field from Chapter 20.1.2; the symmetry is intentional — the same name appears in the log line, the audit row, and the PostHog event for the same business action, so the three surfaces line up at search time.
- **The event property schema.** Properties stay flat and shallow — `{ plan: 'pro', orgId, invoiceCount: 3 }` — not nested objects. PostHog's UI surfaces top-level keys as filterable fields; deeply nested values are second-class. Use snake_case keys to match the event name convention. PII rule (same as Sentry, same as logs): ids and types, never names or emails. Numeric values for things that get aggregated (`amount_cents`, `seat_count`), enums for things that get grouped (`plan`, `role`, `surface`).
- **`identify` — tying events to users.** Before the user logs in, events fire against an anonymous `distinct_id` PostHog generates and stores in a cookie. On login, the app calls `posthog.identify(userId, { plan, createdAt, signupSource })`. PostHog stitches the prior anonymous session to the now-known user so the pre-login pageviews are credited to them. The properties on `identify` are *user-level* (set once or updated rarely) — `plan`, `role`, `signupSource`, `createdAtCohort`. The senior pattern: identify in a layout-level effect right after the session resolves; never identify with email.
- **`group` — the org axis.** SaaS is multi-tenant; behavior at the org level matters at least as much as the user level. `posthog.group('organization', orgId, { plan, seatCount, createdAt })` attaches the org to every subsequent event. Funnels can filter by `organization.plan` or `organization.seatCount > 5`. The Chapter 10 `requireOrgUser` resolves both ids; both flow into the layout-level wiring. Multiple group types are allowed (`organization`, `project`, `team`), the senior call is to pick one or two and keep them stable.
- **`reset` — the logout case.** On logout, `posthog.reset()` clears the identity and starts a fresh anonymous `distinct_id`. Forgetting this means the next user on the shared device gets stitched into the logged-out user's identity — the funnel gets garbage, the consent state may get confused. The reset is part of the auth signout handler from Chapter 8.
- **Server-side capture from a Server Action — the canonical pattern.** Inside a Server Action, after the database commit, call `posthogServer.capture({ distinctId: userId, event: 'invoice_created', properties: { invoiceId, amount_cents, plan }, groups: { organization: orgId } })`. The server SDK is a `PostHogClient` singleton from a `lib/observability/posthog.ts` module, initialized once with the project API key. Server-side capture matters because the browser may not stick around for the response (mobile background, fast tab-close), and state-change events must not be silently dropped.
- **The action wrapper integration.** The Chapter 7.2 `authedAction` wrapper already logs to the structured-log stream on entry, validate, and commit. The senior addition: on commit, the wrapper accepts an `analyticsEvent?: string` argument and fires the PostHog capture with the resolved user, org, and correlation ID in properties. One line per action: `authedAction({ ..., analyticsEvent: 'invoice_created' }, async (ctx, input) => { ... })`. The structural enforcement is that the wrapper, not hand-written code at each call site, decides when the event fires.
- **The correlation ID thread.** The same `correlationId` from Chapter 20.1.2 attaches to every server-side PostHog capture as a property. When the support trail starts at "Error ID: abc123," the PostHog timeline shows the user's events that share that ID, the structured log shows the request lines, and the Sentry event shows the stack. Three surfaces, one ID, one timeline.
- **The "flush before exit" rule on server-side capture.** Vercel functions are ephemeral; the SDK batches events and may have unflushed events when the handler returns. After every server-side `capture`, the senior pattern is `await posthogServer.flush()` (or `shutdown()` if the request is the last in the worker). Skipping this loses events. The PostHog Node SDK exposes `flush()` and the documentation calls this out as the Vercel-specific watch-out.
- **The page-view event — autocapture vs. explicit.** Autocapture sends `$pageview` on full page loads. The App Router does client-side navigation between routes; the senior pattern is a small `PostHogPageView` client component listening to `usePathname` and firing `posthog.capture('$pageview')` on every change. Without it, half the navigation in a SPA-shaped Next.js app goes unrecorded. The wiring chapter (20.2.5) ships the component.
- **Watch-outs:** identifying with email — same PII rule as everywhere; firing the same event from both client and server — double counting in every funnel; defining events at the call site with ad-hoc names (`invoice-create`, `invoice_creat`, `created_invoice`) — funnels never match, the team gives up; nesting properties three levels deep — PostHog can't filter them; forgetting `flush()` server-side — events lost in the function cold-start gap; forgetting `reset()` on logout — identity bleed; capturing autocapture in production with no naming convention layered on top — every dashboard built on autocapture rots in the first markup refactor.

What this lesson does not cover:

- Feature flags and experiments — Lesson 20.2.4.
- Session replay — Lesson 20.2.5.
- The consent gate wiring — Lesson 20.2.5.
- Vercel Analytics events — Lesson 20.2.1.
- Sentry breadcrumbs vs. PostHog events (different surfaces) — Chapter 20.1 owns Sentry.
- Audit logs as the immutable record — Chapter 10.2.

Estimated student time: 50 to 60 minutes. Mechanics-heavy; the event model and the server-side capture pattern are load-bearing for the next two lessons.

## Lesson 20.2.4 — Feature flags and experiments

Topics to cover:

- **The senior question.** The team is about to merge a change that touches the checkout flow. The change passes review and CI; the senior is still nervous because checkout is the money path. The deploy-and-revert model is too slow (a revert is minutes, the bug is dollars). The lesson lands feature flags as the kill switch and gradual-rollout primitive, then experiments as flags-with-a-measurement-plan.
- **What a feature flag is, structurally.** A named server-evaluated boolean (or multivariate value) the application reads at the decision point. The app code branches on the flag; the flag's value can change without a redeploy. Same primitive whether it gates a rollout (10% on Monday, 50% Wednesday, 100% Friday), a per-org beta (these five customers get the new editor), or a kill switch (the new checkout flow goes false in 30 seconds when Sentry alerts fire).
- **The four reasons a flag earns its weight.** (1) **Gradual rollout** — ship to a percentage, watch error and conversion metrics, increase. (2) **Kill switch** — a config flip beats a code revert by an order of magnitude on time-to-mitigate. (3) **Targeted access** — beta, paid-tier gating in code (orthogonal to the billing gate from Unit 16, used for "this customer asked for early access"), per-segment rollout. (4) **Experimentation** — randomized assignment with a measurement plan, covered later in the lesson. Any one earns the flag; the team that wires PostHog gets all four cheaply.
- **PostHog flag types — boolean vs. multivariate.** Boolean flags: on/off, the rollout / kill-switch case. Multivariate flags: named variants with weights (`control` 50%, `variant_a` 25%, `variant_b` 25%) — the experiments case and the multi-arm rollout case. Both share targeting rules; the variant case is the experiment substrate.
- **Targeting rules — the senior shape.** PostHog flag targeting evaluates conditions in order: "release to 10% of users," "release to users where `plan = 'enterprise'`," "release to users in cohort `power_users`," "release to organization with id in list X." Conditions on user properties (set via `identify`), group properties (set via `group`), and cohorts (saved query results) all compose. The senior pattern: rollout starts with the smallest credible audience (internal team org, then beta orgs), expands by percentage, watches metrics, holds at 100% for a release before retiring.
- **Server-side vs. client-side flag evaluation.** Client-side: the browser SDK fetches the flag value once per page load. Round-trip latency, ad-block risk, the "flag value isn't ready when the component renders" flicker problem. Server-side: `posthogServer.isFeatureEnabled('new-checkout', userId, { groups })` evaluates in the route handler / Server Action / RSC, the page renders with the resolved value, no flicker. **Server-side is the senior default for any flag a user can see.** Client-side is for "is the beta banner visible" cases where flicker is invisible.
- **The bootstrap pattern — eliminating client flicker when the flag must be read in a client component.** Server-side evaluation happens in the root layout (or the page wrapper); the resolved flag values pass to a `PostHogProvider` as `bootstrap={{ distinctID, featureFlags }}` so the client SDK reports the same values on first paint that the server did. Without this, a client-component branch on `useFeatureFlagEnabled('x')` returns `undefined` for the first render and the correct value on the second. Same value, two sources, one render.
- **Local evaluation — the cost lever.** `posthogServer` can evaluate flags locally if it's been seeded with the flag definitions; otherwise every check round-trips to PostHog. Local evaluation is enabled by default in the Node SDK, refreshes flag definitions periodically (30 seconds by default), and turns flag checks into in-memory operations. The cost win matters because a busy Server Action might check three flags; without local evaluation that's three network round-trips per request.
- **The flag in the application code — three call patterns.** **In an RSC / Server Action:** `if (await posthogServer.isFeatureEnabled('new-checkout', userId, { groups: { organization: orgId } })) { /* new code */ } else { /* old code */ }`. **In a client component:** `const enabled = useFeatureFlagEnabled('new-checkout')` with `PostHogProvider` mounted upstream. **In middleware (edge):** PostHog ships `posthog-node` evaluation that works in the Edge runtime for flag-gated routing decisions; useful for "redirect 10% of users to the new landing page."
- **Multivariate flags — the named-variants case.** `await posthogServer.getFeatureFlag('checkout-experiment', userId, { groups })` returns the variant key string (`'control'` | `'variant_a'` | `'variant_b'`) or `false` if the user isn't in the experiment. Branch on the string. The variant assignment is sticky per user; the same `distinctId` always gets the same variant within an experiment, even across server and client.
- **The flag lifecycle — what the senior actually does.** Create the flag with targeting set to 0% rollout. Deploy code that checks the flag (both branches present, default to old behavior). Ship the deploy with the flag still at 0% (the change is dormant; the code path is exercised against zero users). Roll to 1% / 10% / 50% / 100% over days, watching metrics. After 100% is stable for a week, **delete the old branch and retire the flag.** The retirement step is the discipline most teams skip; the code grows fences nobody remembers building. The team review surfaces stale flags monthly.
- **Kill-switch flags — the always-on case.** Some flags exist permanently as kill switches for high-risk subsystems (the AI generation endpoint, the webhook fan-out, the new search index). They live as `feature_killswitch_ai_generation` style names with default 100% on, the off-state being the operator's emergency stop. Different lifecycle: never retired. The senior call is keeping them rare and obvious.
- **Experiments — flags with a measurement plan.** An experiment is a multivariate flag (`control` vs. `variant`) with a *predeclared* success metric (`event:checkout_completed`, "we expect a 5% lift," "we'll run for two weeks or until 10,000 sessions"). PostHog wires the random assignment, the exposure tracking, the result with statistical-significance reporting. The discipline: write the measurement plan *before* creating the flag; without it the experiment becomes a flag the team forgets to interpret.
- **Exposure events — the load-bearing detail.** PostHog's experiments need to know *which users saw which variant*; the SDK emits a `$feature_flag_called` event automatically when a flag is read, and the experiment uses these as exposure. Edge case: `useFeatureFlagPayload` does not emit the exposure event, so read with `useFeatureFlagEnabled` or `useFeatureFlagVariantKey` for any flag that drives an experiment. Quiet detail, expensive to discover after a two-week experiment returns no signal.
- **What experiments aren't for.** Marketing copy A/B tests on the landing page — PostHog can do it, but a dedicated tool (or no tool — three credible variants ranked by gut from a marketing team) often beats the engineering overhead. The cases that pay: pricing experiments, onboarding flow changes, feature placement on the dashboard. The senior call is fewer, higher-impact experiments, each with a written hypothesis.
- **Watch-outs:** flags created and never retired — the code accumulates fences and branches; rolling out to "10% of organizations" via a *user-property* flag — the rollout is per user not per org and a single org sees mixed behavior across its members; client-side flag check without bootstrap — flicker on every load; experiments running with the success metric chosen after looking at the data — invalidates the result by structure; using flags as a substitute for environment-specific config — env vars exist for a reason; flag check inside a hot loop without local evaluation — latency cost; `useFeatureFlagPayload` alone for an experiment — no exposure, no result.

What this lesson does not cover:

## Lesson 20.2.5 — Wiring PostHog: session replay and the consent gate

Topics to cover:

- **The senior question.** The four PostHog products (events, flags, replay, experiments) are decided. The team is ready to wire it. Two non-negotiables: (1) the cookie consent gate from Chapter 17.2.5 owns every non-essential analytics call, so PostHog can't fire on a user who hasn't consented; (2) the user and org identity from Chapter 8 and Chapter 10 flow into every event without duplication. The lesson is the wiring that lands both, plus session replay turned on under the same consent.
- **The file shape.** A `lib/observability/posthog.ts` server module exporting the `PostHogClient` singleton and the `flush`-aware capture helpers. A `components/posthog-provider.tsx` client component that mounts `PostHogProvider` and the conditional `posthog.init` inside it, gated by the consent hook. A `components/posthog-pageview.tsx` client component that fires `$pageview` on App Router navigations. Wiring in the root layout: server-side identify in an RSC, client provider with bootstrap, page-view tracker, all wrapped by the consent state. Four small files, one wiring point.
- **The consent gate — the single source of truth.** Chapter 17.2.5 ships a `useConsent()` hook that returns `{ analytics: boolean, replay: boolean, marketing: boolean }`. The senior pattern: PostHog initializes with `opt_out_capturing_by_default: true`. When `consent.analytics` becomes true, call `posthog.opt_in_capturing()`. When it becomes false (user withdrew), `posthog.opt_out_capturing()` and `posthog.reset()` so the in-memory identity clears. The same hook gates `posthog.startSessionRecording()` against `consent.replay`. **One hook, one place, no duplicated checks at call sites.**
- **The `posthog.init` configuration — May 2026 defaults.** `posthog.init(env.NEXT_PUBLIC_POSTHOG_KEY, { api_host: '/ingest', ui_host: 'https://eu.posthog.com', capture_pageview: false, autocapture: true, persistence: 'localStorage+cookie', opt_out_capturing_by_default: true, disable_session_recording: true, person_profiles: 'identified_only', loaded: (posthog) => { ... } })`. The flags explained: `capture_pageview: false` because the App Router needs the custom page-view component; `disable_session_recording: true` so replay starts only after the replay consent fires; `person_profiles: 'identified_only'` keeps anonymous users out of the person-profile billing dimension; `api_host: '/ingest'` routes through the reverse proxy.
- **The reverse proxy — bypassing ad blockers.** Ad blockers maintain lists of analytics domains; PostHog's `app.posthog.com` is on every list. The senior pattern is a Next.js rewrite in `next.config.ts` that aliases a non-obvious path (`/ingest/*` -> `https://eu-assets.i.posthog.com/static/*` for the script, `/ingest/decide` -> `https://eu.i.posthog.com/decide` for runtime calls). The browser sees requests to your own domain; ad blockers don't catch them. Capture rate climbs 10-30% on consumer-facing apps. Don't use obvious path names (`/analytics`, `/posthog`, `/track`) — those are blocked too. Cost note: rewrites consume function bandwidth on Vercel; for very high-traffic sites, PostHog's managed reverse proxy or a Cloudflare Worker is cheaper.
- **Environment variables — the typed env entries.** `NEXT_PUBLIC_POSTHOG_KEY` (project API key, public by design, browser SDK uses it), `NEXT_PUBLIC_POSTHOG_HOST` (the proxy path, defaults to `/ingest`), `POSTHOG_PROJECT_ID` (server-side, for the management API if used). All flow through `@t3-oss/env-nextjs` from Chapter 1.4. The personal API key for management calls (creating flags from code, a rare case) is `POSTHOG_PERSONAL_API_KEY`, server-only.
- **The identify-once pattern in the layout.** In the root layout (an RSC), resolve the session via the Chapter 8 auth helper. If signed in, pass `{ userId, orgId, role }` to the `PostHogProvider` client component as props. The provider's `useEffect` runs `posthog.identify(userId)` and `posthog.group('organization', orgId)` once after `posthog.init`. On signout, the client signout handler calls `posthog.reset()`. The senior call: identify *server-resolved* (the source of truth) and *idempotent* (re-runs are safe).
- **The page-view component.** A small `PostHogPageView` client component reads `usePathname()` and `useSearchParams()`, and on change fires `posthog.capture('$pageview', { $current_url: window.location.href })`. Mounted once in the layout, runs on every App Router navigation. Without this, a SPA-shaped Next.js app loses navigation tracking after the first load.
- **Session replay — what gets recorded.** Replay captures DOM mutations, network requests, console output, and console errors as a serialized stream the PostHog UI replays as video. Not actual video — DOM playback, low bandwidth, searchable. The PII surface is real: by default, replay records *everything* the user sees, including form inputs. The senior config: `session_recording: { maskAllInputs: true, maskTextSelector: '[data-private]', maskInputOptions: { password: true, email: true } }`. Every input is masked unless explicitly opted in (`data-public` selector pattern), every text node with `data-private` is masked, password and email inputs always masked. The cookie consent gate's `replay` permission gates the entire feature.
- **Replay sampling — the cost lever.** Replay is expensive. PostHog supports `session_recording: { sampleRate: 0.1 }` — record 10% of sessions sampled at session-start. Alternatives: `linked_flag` (only record sessions where a flag is on), `minimum_duration_milliseconds: 30000` (skip sessions under 30s, useful for filtering bounces). The senior default: 10% sampling on launch, ramp based on the bill; raise the minimum-duration floor to discard hard bounces.
- **The "start replay only after consent + identify" sequence.** When `consent.replay` becomes true *and* `posthog.identify` has been called, call `posthog.startSessionRecording()`. If recording started anonymously and the user logs in later, PostHog stitches the session to the user; the senior pattern works either way. Watch for the case where the user grants replay consent in the middle of a session — PostHog handles `startSessionRecording()` mid-session correctly.
- **Sentry interop — one-line mention.** PostHog can attach Sentry events to the replay timeline (the `posthog-js/sentry-integration` shim). Wires up if both are present so a Sentry error in the dashboard links to the replay segment where it fired. The course doesn't enable it by default; surface for the project chapter if the audit demands it.
- **The verification path the project (20.4) will follow.** Open the app in an incognito window. Decline consent — `posthog.opt_out_capturing` fires, no events leave the browser (verified in the Network tab, filter `/ingest`). Accept analytics consent — `$pageview` and subsequent events appear in the PostHog Live Events feed within 5 seconds. Sign in — `identify` fires with the user id, the org `group` is set, subsequent events show the user/org context in the PostHog UI. Accept replay consent — the session recording begins, visible in the Recordings tab within a minute. Withdraw consent — `opt_out_capturing` and `reset` fire, the next event doesn't transmit.
- **What the GDPR audit looks like, briefly.** Chapter 17.2.4 frames the data subject access / deletion case. PostHog ships an EU region (data stays in the EU) and an admin UI for deleting per-`distinct_id` data; the senior wire-up keeps the `distinct_id` equal to the application user id, so the deletion request maps trivially. The pattern: on user-deletion request from Chapter 17.2.4, the deletion handler also calls PostHog's deletion API for that `distinct_id`.
- **Watch-outs:** `posthog.init` outside the consent gate — events fire pre-consent, GDPR violation; default replay config — every form input recorded in plaintext, the credentials and the credit-card number are now in PostHog; obvious proxy path (`/track`, `/analytics`) — ad blockers catch them; `disable_session_recording: false` at init — replay starts before the replay consent is granted; identify called before init — silent no-op, no identity attached; identify in a client component without server-resolved props — race with auth resolution, sometimes fires anonymously; reverse proxy on a high-traffic site without watching the bandwidth bill — the cost shifts from PostHog to Vercel; forgetting `posthog.reset()` on logout — the next user inherits the prior identity; running both EU and US PostHog projects accidentally — the team has two billing buckets and two datasets, the funnels don't match.

What this lesson does not cover:

- The cookie consent banner UI and storage — Chapter 17.2.5 owns it.

- Vercel Analytics — Lesson 20.2.1.
