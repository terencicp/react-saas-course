# Chapter 20.2 — Product analytics: traffic, events, flags, replay

## Chapter framing

Chapter 20.2 installs the product analytics layer that sits next to the error/log floor from Chapter 20.1: where the prior chapter answers *what broke*, this one answers *what users do, which cohorts convert, which flag's variant won, and which UX bug only session replay can show.* The student arrives with the consent gate already built (Chapter 17.2.5) — the `useConsent()` source of truth, the four-state machine, PostHog's `opt_out_capturing_by_default` plus dynamic SDK import — and Vercel deployment context from Unit 6's stack walkthrough. The chapter installs Vercel Web Analytics and Speed Insights as the cookieless default, names the threshold past which event-level analytics earns its weight, and wires PostHog as the one platform that folds events, flags, replays, and experiments — the minimum-stack philosophy in action against the alternative shape of "Mixpanel for events plus LaunchDarkly for flags plus FullStory for replay plus Statsig for experiments."

Threads through every lesson. **The default is Vercel Web Analytics plus Speed Insights** — cookieless, no banner-blocking, free on every plan, covers traffic and Core Web Vitals without a single event call. **PostHog earns its weight at the threshold where event-level questions or flag-gated rollouts start mattering** — pre-PMF marketing pages don't need it; a product with onboarding funnels, paywall conversions, and gradual feature releases does. **Nothing fires before consent** — every PostHog reach in this chapter routes through the `useConsent()` gate from 17.2.5, the SDK is dynamically imported only when consent flips to accepted, and the four-state machine is the contract. **One platform, four primitives** — PostHog Cloud EU as the default region (GDPR posture), events plus flags plus replay plus experiments share one SDK, one DSN, one identity model. **Identity is the load-bearing primitive** — `posthog.identify()` after sign-in, anonymous events stitched to the identified user via the distinct ID handshake, feature flags evaluated against user properties not raw cohorts. **Server-side bootstrap is the App Router default** — flags evaluated in middleware or layout, passed to the client as bootstrap data, hooks return real values without a network round-trip and without flash-of-default-variant. Every lesson ties back to a concrete senior question — "did the new pricing page lift trial-to-paid?", "is the new onboarding flow live for 20% of new orgs?", "why did this user rage-click and bail?" — so the discipline reads as the answer to a known cost.

## Lesson 20.2.1 — The cookieless floor: Vercel Analytics and Speed Insights

Install Vercel Web Analytics and Speed Insights as the cookieless default that ships before any event taxonomy decision, covering traffic, top pages, and Core Web Vitals without a consent banner.

Topics to cover:

- **The senior question.** The product just shipped to production. Before any event taxonomy is designed, three questions need answers: how much traffic does the app get and from where, which pages convert and which leak users, and is the experience fast enough that the Core Web Vitals don't punish SEO and conversion. Two installs answer all three without writing an event. The lesson installs Vercel Web Analytics and Speed Insights as the floor that ships before any heavier analytics decision.
- **Why this is the default, not the conditional path.** Cookieless (no banner blocking on the marketing site), free on every Vercel plan, two `<script>` injections via `@vercel/analytics` and `@vercel/speed-insights`, no event taxonomy to maintain. The threshold to flip past this: event-level product questions or flag-gated rollouts. Until then, this is the whole stack.
- **What Web Analytics covers.** Page views, unique visitors, top pages, top referrers, country, OS, browser, device class. Aggregated and anonymized at ingest; no personal data leaves the user's device. Resilient intake (the v2 packages) discovers ingest endpoints dynamically to survive ad-blocker patterns. Custom events are supported but the lesson treats them as out of scope — once the team needs custom events, PostHog earns its weight (Lesson 20.2.2).
- **What Speed Insights covers.** Real-user Core Web Vitals — LCP, INP, CLS, TTFB, FCP — sampled from real production traffic, scored against the 75th-percentile thresholds Google uses for Search. The performance vigilance chapter (20.3) uses Speed Insights as the input dashboard; this lesson ships the wire.
- **The install — `@vercel/analytics` and `@vercel/speed-insights`.** Two packages, two component injections in `app/layout.tsx` (`<Analytics />` and `<SpeedInsights />`), one enable-toggle in the Vercel project's dashboard per environment. Verify in production by triggering a page view and watching the dashboard within ~30 seconds. The Vercel Agent can perform the install end-to-end in 2026; the lesson walks the manual install for understanding.
- **The cookieless guarantee and the consent gate.** Web Analytics doesn't set cookies and doesn't fingerprint; under most jurisdictions it falls under "essential" or "legitimate interest" for traffic analytics and doesn't need consent. Speed Insights is the same shape. Neither passes through the `useConsent()` gate from 17.2.5; PostHog does. State this explicitly so the student doesn't gate Vercel Analytics by reflex and lose the pre-consent traffic data.
- **Environment scoping.** Enable in production by default. Preview deployments can opt in for QA on real traffic; development should stay off. The dashboard is per-environment.
- **The threshold past Vercel Web Analytics.** When the team needs to answer *who* (identified user, not anonymous visitor), *what they did* (event with properties, not page view), *across sessions* (funnel from sign-up to first paid invoice), *gated by flag* (the new onboarding variant), or *in replay* (this user's actual interaction). Each of these crosses the threshold. The next lesson lands the decision.
- **What about Google Analytics 4.** Named once as the historical default; the course doesn't reach for it. GA4's data model (sessions, hits, audiences) is awkward for product analytics, the consent banner UX is heavier, the data export shape is rigid. The 2026 SaaS default is Vercel for traffic + PostHog for product analytics; GA4 stays in the marketing-stack conversation, not engineering.
- **Watch-outs.** Forgetting to enable Web Analytics in the Vercel dashboard after the package install (the script ships but the data sinkholes); shipping `@vercel/analytics` on a static export and assuming it works — it needs Vercel's edge to ingest, an external host won't catch it; gating Vercel Web Analytics behind the consent banner by mistake — kills traffic data for no compliance benefit; assuming Speed Insights data is real-time — there's a ~30 second to ~1 minute latency at first; relying on Web Analytics' custom events as a product analytics layer — the schema is shallow, no funnels, no cohorts, the next lesson covers the right reach.

What this lesson does not cover:

- The decision to add PostHog — Lesson 20.2.2.
- PostHog wiring, events, flags, replay, experiments — Lessons 20.2.3 to 20.2.6.
- Core Web Vitals at depth — Chapter 20.3.
- The cookie consent gate — Chapter 17.2.5.
- Server-side performance traces — Chapter 20.3.

Estimated student time: 25 to 35 minutes. The shortest lesson in the chapter; setup-and-decision shape.

## Lesson 20.2.2 — When PostHog earns its weight

Name the four needs (events, flags, replay, experiments) that cross the threshold past Vercel Analytics, and pick PostHog Cloud EU as the one-platform answer over the four-vendor alternative.

Topics to cover:

- **The senior question.** The marketing site has Vercel Web Analytics; traffic is flowing; the product team wants to know whether the new onboarding flow improved trial-to-paid conversion, the engineering team wants to roll out the new billing UI to 10% of orgs first, and support has three users this week claiming the dashboard "just broke" with no console error. None of these questions are answerable from Vercel Web Analytics. The reach is event-level product analytics with identified users, feature flags for gradual rollouts, and session replay for the bugs that don't throw. The lesson installs the decision: which platform, why one platform.
- **The four needs that cross the threshold.** (1) **Event-level analytics** — funnels, cohorts, retention curves, properties on every event (`plan: 'pro'`, `org_seats: 12`); answers *what users do across sessions.* (2) **Feature flags** — gradual rollouts, kill switches, per-cohort releases, A/B substrate; answers *who sees what feature when.* (3) **Session replay** — privacy-masked playback of the user's actual interactions; answers *what they did before the rage-click.* (4) **Experiments** — statistical A/B tests with primary and secondary metrics, automated significance calls; answers *did this change move the metric.* Each need on its own is a real SaaS line item; together they decide the platform.
- **The minimum-stack play — one platform, four primitives.** The alternative shape: Mixpanel or Amplitude for events ($), LaunchDarkly or Statsig for flags ($$), FullStory or Hotjar for replay ($$), Statsig or Optimizely for experiments ($$). Four contracts, four SDKs in the bundle, four identity models to keep aligned, four dashboards. PostHog folds the four into one platform, one SDK, one identity, one dashboard — the minimum-stack thesis applied. Quote the cost shape: PostHog's free tier (1M events, 5k replays, generous flag evals per month) covers a pre-PMF SaaS; paid tiers scale per-primitive.
- **Why PostHog over the alternatives — the credible competitors named once.** **Mixpanel** — strongest event analytics depth; no flags or replay; reach for it when event analytics is the only need and the depth matters more than folding. **Amplitude** — same shape as Mixpanel, deeper cohort tooling, enterprise pricing. **Statsig** — strong experiments and flags, weaker event UX, growing replay; reach for it when experiments are the load-bearing primitive (A/B is the team's daily reach). **LaunchDarkly** — flag specialist, no events, no replay; reach for it when flag scale (10k+ flags, complex targeting trees) outgrows PostHog's flag UX. **June.so / Pirsch / Plausible** — privacy-first traffic analytics in the Vercel Web Analytics tier, not product analytics. The course default is PostHog; the alternatives stay in the conversation only when the trigger condition fits.
- **Self-hosted vs. PostHog Cloud — PostHog Cloud EU as the GDPR default.** PostHog Cloud EU runs in Frankfurt, automatically disables IP capture for new projects, and covers the GDPR data-residency commitment without a DPA dance. PostHog Cloud US is the alternative for non-EU teams. Self-hosted PostHog exists (open-source) but the operational cost — Postgres, ClickHouse, Redis, Kafka — outweighs the savings for any team under ~50M events/month. The course default: PostHog Cloud EU.
- **What the install pulls in — the SDK shape preview.** `posthog-js` for the browser (events, flag client, replay capture), `posthog-node` for the server (server-side event capture, server-side flag evaluation), the new `@posthog/next` package in 2026 that wraps both with App Router conventions (`bootstrapFlags` for SSR-evaluated flags, middleware for distinct-ID cookie seeding, an optional proxy route). The next lesson lands the wiring.
- **The consent-gate constraint from 17.2.5.** PostHog handles personal data (distinct IDs, IP addresses, event properties that may carry user state) and falls under non-essential under GDPR. Nothing from PostHog fires before the user accepts. The wiring in this chapter routes through the `useConsent()` source of truth; the SDK is dynamically imported only on accept; `opt_out_capturing_by_default: true` is the safety floor in case the gate is bypassed. The four-state machine (unknown / accepted / rejected / pending) governs every reach.
- **The cost shape and quota discipline.** PostHog bills per ingested event, per replay session, per flag evaluation (in volume tiers). Quota-bust shapes: a `posthog.capture()` inside a render or a server-side loop, a missing distinct ID stitching producing two events per real action, replay recording every session on a B2C site at scale. Budget by primitive; the chapter's defaults assume the team picks two of the four primitives at first (events + flags), adds replay for the bug-class it's meant for, and reserves experiments for the metric-led moment.
- **The decision frame for the student to take.** Pre-PMF marketing site → Vercel only. Post-PMF product with identified users and rollout needs → Vercel + PostHog. The threshold isn't arbitrary; it's the four needs above. Name the four; if zero are real, don't add PostHog yet.
- **Watch-outs.** Adding PostHog at day zero on a pre-PMF marketing site — every event call is dead weight; the schema rots before anyone reads it; gating Vercel Analytics by the consent banner because PostHog is — they don't share the rule; assuming PostHog replaces Sentry — the two are complementary (Sentry for thrown errors, PostHog for behavior and the bug-class without an error); shipping all four primitives at install — pick the two that earn weight now, add the rest when the trigger fires; picking PostHog Cloud US when EU users dominate — the GDPR posture is harder to undo than to set up correctly; running self-hosted PostHog without a plan for ClickHouse on-call.

What this lesson does not cover:

- The actual PostHog wiring — Lesson 20.2.3.
- Event taxonomy and the identify pattern — Lesson 20.2.4.
- Feature flags at depth — Lesson 20.2.5.
- Session replay — Lesson 20.2.6.
- The consent gate's internals — Chapter 17.2.5 owns it.

Estimated student time: 30 to 40 minutes. Decision archetype; the threshold and the four-needs frame is the load-bearing content.

## Lesson 20.2.3 — Wiring PostHog through the consent gate

Wire `@posthog/next` end-to-end in the App Router with a consent-gated dynamic import, the `/ingest` proxy route, `opt_out_capturing_by_default` as the safety floor, and a verified pre- and post-consent test.

Topics to cover:

- **The senior question.** PostHog earns its weight (Lesson 20.2.2). How does the SDK arrive in the App Router, route every reach through the `useConsent()` gate from 17.2.5, identify users on sign-in, and not flash a default-variant before the server-side flag bootstrap lands. The lesson installs the wire end-to-end and the install ends in a state where one test event reaches PostHog post-consent with the right identity attached and no calls fire pre-consent.
- **The packages — `posthog-js`, `posthog-node`, `@posthog/next`.** `posthog-js` is the browser SDK; `posthog-node` is the server SDK; `@posthog/next` (2026) is the App Router wrapper that ties them together with middleware, `bootstrapFlags`, and the optional proxy route. The course reaches for `@posthog/next` as the default install. Name the underlying packages so the student understands what `@posthog/next` is wrapping.
- **Setup — the env variables.** `NEXT_PUBLIC_POSTHOG_KEY` (the project's public-write key — safe to ship to the client), `NEXT_PUBLIC_POSTHOG_HOST` (`https://eu.i.posthog.com` for EU Cloud, the default), `POSTHOG_PERSONAL_API_KEY` (server-side, for flag evaluation outside the public-write path — `@posthog/next` reads it for `bootstrapFlags`). All three pass through the typed `env` from Chapter 1.4.5; the public ones land in the client bundle, the personal key never does.
- **The `PostHogProvider` and the consent-gated dynamic import.** The provider component (client-side, `'use client'`) reads `useConsent()` from 17.2.5. When state is `accepted`, the component dynamically imports `posthog-js` (`const { default: posthog } = await import('posthog-js')`), initializes with `posthog.init(key, { defaults: '2025-05-24', api_host, opt_out_capturing_by_default: true, persistence: 'localStorage+cookie', capture_pageview: 'history_change' })`, and exposes the client via context. When state is `rejected`, the provider renders children without the SDK reach. When state is `pending` or `unknown`, the same — nothing fires.
- **`opt_out_capturing_by_default: true` as the safety floor.** Even after the dynamic import, the SDK is opted-out by default; the provider explicitly calls `posthog.opt_in_capturing()` only when consent is `accepted`. If the gate logic is ever bypassed by mistake, the SDK stays silent. The defense in depth from 17.2.5 lands structurally.
- **The `defaults: '2025-05-24'` config option.** PostHog's `defaults` knob (introduced 2025) bundles the recommended settings for autocapture, pageview tracking, error tracking, and exception capture; pin a specific date so a future PostHog default change doesn't silently flip behavior. The course pins to the May 2025 defaults (the snapshot the chapter was written against); revisit when PostHog ships a new default set.
- **App Router integration — the layout shape.** `app/layout.tsx` wraps children in `<PostHogProvider>` which wraps in `<ConsentProvider>` (the 17.2.5 component). The order matters: `ConsentProvider` outermost so `PostHogProvider` can read `useConsent()` on every render. Both are client components imported into the root server-component layout; the layout itself stays a server component.
- **Pageview tracking with the App Router.** `capture_pageview: 'history_change'` (the modern default since 2024) auto-captures pageviews on `history.pushState` calls — which is how `next/link` and `useRouter().push()` navigate. The legacy `capture_pageview: true` only fires on hard navigation; the `history_change` mode is the App Router fit. Verify by clicking a `<Link>` and watching the pageview land.
- **The proxy route — `/ingest/[...path]`.** Ad-blockers heuristically block `i.posthog.com`; a same-origin proxy route relays the requests through the app's own domain. `@posthog/next` ships a route handler factory for `app/ingest/[...path]/route.ts`. Set `api_host: '/ingest'` in `posthog.init` and `ui_host: 'https://eu.i.posthog.com'` so links in the SDK (replay viewer, feature flag console) still point at the PostHog UI. The proxy is recommended in 2026, not optional — ad-blocker coverage has crept up.
- **Server-side capture — `posthog-node` for the rare server event.** Most events fire client-side. The reach for `posthog-node` is the server-side moment with no client (a Stripe webhook completing a checkout, a Trigger.dev job firing on a schedule). The pattern: `import PostHog from 'posthog-node'`; one process-scope client; `client.capture({ distinctId, event, properties })`; `await client.shutdown()` at the end of short-lived serverless functions to flush. `flushAt: 1, flushInterval: 0` to disable batching since Vercel functions terminate fast. Server-side capture has no consent gate by definition (it's not running in the user's browser); the moral consent gate is still the user's choice — only server-fire events for users who've accepted client-side, or for users without a session at all (anonymous webhook event identified by the upstream ID).
- **Verifying the wiring.** Three checks. (1) Reject in the consent banner; open DevTools Network; click a `<Link>` — no requests to `/ingest` or to `posthog.com`. (2) Accept; click a `<Link>` — one pageview request to `/ingest`. (3) Open PostHog's Live Events tab; the pageview lands with the project's distinct ID. The verification is the lesson's gate; without it, the wiring isn't trusted.
- **Watch-outs.** Forgetting `opt_out_capturing_by_default: true` so the dynamic import alone fires events; the proxy route forgetting to forward all paths (`[...path]` not `[path]`); the consent gate reading from a different source of truth than the rest of the app (split-brain consent); `posthog.identify()` before consent (still a personal-data write); shipping the personal API key to the client by misnaming the env (`NEXT_PUBLIC_POSTHOG_PERSONAL_KEY` — never); enabling `capture_pageview: true` and getting only hard navigations; running self-hosted PostHog and pointing `NEXT_PUBLIC_POSTHOG_HOST` at the wrong subpath; assuming `@posthog/next` is a drop-in for every option in `posthog-js` — read the wrapper's API surface, not the raw SDK docs; not awaiting `client.shutdown()` on server-side capture and losing events on Vercel function termination; the consent banner outside `PostHogProvider`'s tree so `useConsent()` returns the default (read 17.2.5's wiring carefully).

What this lesson does not cover:

- The consent gate's internals — Chapter 17.2.5.
- Event taxonomy and what to track — Lesson 20.2.4.
- Feature flags and bootstrapping — Lesson 20.2.5.
- Session replay — Lesson 20.2.6.
- A/B experiments — Lesson 20.2.5 (folded under feature flags).
- The typed `env` shape — Chapter 1.4.5.

Estimated student time: 45 to 55 minutes. Setup-and-wiring lesson; the consent-gated dynamic import plus the proxy route plus identify is the load-bearing wiring.

## Lesson 20.2.4 — Events, properties, and the identify handshake

Install the Object-Action event taxonomy, a typed `track()` helper backed by an event dictionary, the person/event/super-property split, and the `identify`/`reset` handshake that stitches anonymous to known users and orgs.

Topics to cover:

- **The senior question.** The SDK is wired (Lesson 20.2.3). What events does the team fire, what properties travel on them, when does anonymous become identified, and how does the team avoid the six-month-later state where no one remembers what `clicked_thing` meant or which property carried the plan. The lesson installs the event taxonomy discipline and the identify handshake — the two contracts that govern every reach.
- **The taxonomy — Object-Action, snake_case, past tense.** The 2026 SaaS convention: `object_action` in snake_case, past tense, lower-case (`invoice_created`, `plan_upgraded`, `checkout_completed`, `paywall_viewed`). Past tense because events describe what happened; snake_case because it's lower-friction across SQL queries and dashboard filters; Object-Action because it sorts and groups predictably in the PostHog event browser. Reject patterns: title case, ambiguous names (`clicked_button`), present-tense imperatives (`upgrade_plan`), nouns alone (`invoice`).
- **The event dictionary as the source of truth.** One file in the repo (`lib/analytics/events.ts`) exports a typed `EventName` union and a `properties` type per event. The dictionary is reviewed in PR like any contract. The single-file shape means renames, removals, and new events show up in `git log`; a six-month-later "what does this event mean" lookup has a documented answer. Pair with PostHog's Event Dictionary feature for the runtime view of which events have actually been seen.
- **The typed `track()` helper.** The reach: `track('invoice_created', { invoiceId, amountCents, plan })` where the second arg is statically typed against the dictionary. A typo or a missing required property fails the build. Wraps `posthog.capture()` underneath; the public API never lets a raw string-named event escape into production. The pattern: one helper, typed dictionary, no direct SDK calls in feature code.
- **Properties — what travels on every event.** Domain properties per event (`invoiceId`, `amountCents`) plus a small set of super-properties set once per session via `posthog.register({...})` (`plan`, `org_id`, `seats`, `role`). Super-properties auto-attach to every event; don't pass them on every call. Sensitive properties never travel (no email in event properties, no IPs; identity carries email via `setPersonProperties` which routes to the identified person, not every event).
- **`posthog.identify()` — the anonymous-to-known handshake.** Before sign-in, PostHog generates an anonymous distinct ID and stores it in localStorage/cookie; every event fires under that ID. On sign-in, the app calls `posthog.identify(userId, { email, org_id, plan, created_at })`. PostHog stitches all prior anonymous events under that distinct ID to the now-identified user. The handshake works once per session; calling `identify` with a different ID later requires a `reset()` first (the sign-out reach).
- **`posthog.reset()` on sign-out.** Clears the distinct ID, the super-properties, and the identity link. Next session starts fresh anonymous. Without `reset()`, two users on the same browser pollute each other's funnels. The pattern: sign-out flow calls `posthog.reset()` after the server-side session destroy.
- **Person properties vs. event properties — the split.** Person properties live on the user (`posthog.setPersonProperties({ plan, role, created_at })`) and are queryable across all their events. Event properties live on the individual event (`amount_cents`, `invoice_id`). The decision: properties that describe *the user as of now* go on the person (with `$set` for first-seen or `$set_once` for monotonic); properties that describe *this specific event* go on the event. Updating a person property doesn't rewrite past events.
- **Group analytics — orgs as first-class.** The SaaS shape: events belong to a user but the metrics matter at the org level (MRR by org, feature adoption per org). PostHog's group analytics maps users to organizations; `posthog.group('organization', orgId, { name, plan, seats })` ties the user's events to the org. Funnels and cohorts can pivot on group. The reach: identify the org on sign-in (or on org switch) alongside the user identify.
- **Server-side capture and the distinct-ID join.** When a server-side event needs to fire (the Stripe webhook on plan upgrade, the Trigger.dev job completing an export), the server SDK needs the user's distinct ID. The reach: store the PostHog distinct ID on the user row (or read from the request cookie via `@posthog/next`'s middleware) and pass it to `client.capture({ distinctId, event, properties })`. Without the distinct ID, the server event creates a fresh anonymous person and the funnel breaks.
- **Autocapture — the convenience and the cost.** `posthog-js` autocaptures clicks, form submits, and inputs by selector (`button[data-attr=upgrade]`). Convenient for marketing pages where the team didn't pre-plan events; costly for product pages where it doubles up with named events and inflates the event count. The default in the chapter: autocapture stays on for the marketing surfaces, off (`autocapture: false`) for the authenticated app where the team writes named events. Use `data-ph-no-capture` on elements that should never autocapture.
- **The first three events for a SaaS app.** A worked example. (1) `user_signed_up` — fired client-side after successful registration, person properties set, group identified to the org. (2) `paywall_viewed` — fired when the paywall modal opens, properties: `feature`, `plan_required`. (3) `plan_upgraded` — fired server-side from the Stripe webhook with `distinct_id` read from the user row, properties: `from_plan`, `to_plan`, `amount_cents`. These three feed the trial-to-paid funnel; the lesson uses them as the worked example.
- **Watch-outs.** Naming events inconsistently across the codebase — the dictionary stops it; firing an event in a render — re-renders multiply events, every event call goes in a handler or an effect; missing `posthog.reset()` on sign-out — funnels conflate users; calling `identify` with a different ID without `reset()` — PostHog raises and the identify is dropped; sending high-cardinality properties as event props (free-text descriptions) — the schema bloats; emailing PII in event properties — person properties with `setPersonProperties` is the right surface and even there email is the operator-side identifier per 17.1.2; forgetting group analytics — the org-level metrics are the load-bearing ones for a B2B SaaS; autocapture on the authenticated app inflating the event count past the free tier; the typed `track()` helper bypassed by a `posthog.capture()` import elsewhere — ESLint rule to forbid the raw import outside the helper file; server-side capture with no `await client.shutdown()` losing events.

What this lesson does not cover:

- Feature flags and gradual rollouts — Lesson 20.2.5.
- Session replay — Lesson 20.2.6.
- The consent gate — Chapter 17.2.5.
- The audit log (separate from product analytics) — Chapter 17.2.3.
- Marketing attribution / UTM parsing — out of scope; PostHog's defaults cover it.

Estimated student time: 45 to 55 minutes. Pattern-and-decision lesson; the typed `track()` helper plus the identify handshake plus the property split is the load-bearing content.

## Lesson 20.2.5 — Flags, rollouts, and experiments on one primitive

Ship feature flags as kill switches, percentage rollouts, and metric-driven A/B experiments, with server-side `bootstrapFlags` to kill the flash-of-default-variant and a stale-flag deletion discipline.

Topics to cover:

- **The senior question.** A new onboarding flow ships; the team wants it live for 10% of new orgs first, then 50% if metrics look good, then 100%. A flag turns the new code path on per cohort, the deploy is decoupled from the release, the rollback is one toggle in PostHog (not a revert and redeploy). For the same flag, a metric-driven A/B test (variant A vs. variant B, trial-to-paid as the primary metric) closes the loop. The lesson installs feature flags as the rollout substrate and experiments as the metric-driven variant of the same primitive.
- **The flag types — boolean, multivariate, JSON payload.** **Boolean** — the default; on/off per user. **Multivariate** — `control`, `variant_a`, `variant_b`; the substrate for A/B tests. **JSON payload** — the flag returns a structured value (a config object: `{ price_cents: 2900, currency: 'EUR' }`) that ships per cohort. Reach for booleans first; multivariates when there's an experiment to run; JSON payloads when the flag is governing config not just a code branch.
- **Targeting — release conditions.** Per-user, per-property (`plan = 'pro'`), per-group (`organization.seats > 10`), per-percentage (10% rollout based on a deterministic hash of the user's distinct ID), per-cohort (PostHog's cohort feature — users matching a property predicate). Combine with AND/OR. The targeting UI is in PostHog; the code reads the flag value, never the targeting rule.
- **The two-bug failure mode — flash of default variant.** Client-side flag fetch on first paint: the SDK loads, fires a network request to PostHog, gets the variant, then re-renders. The user sees the default variant for ~200ms then a flash to the assigned one. For experiments, the flash poisons the metric (the user saw control then variant). For rollouts, the flash is a UX bug. The fix: **server-side bootstrap.**
- **Server-side bootstrap — `bootstrapFlags` via `@posthog/next`.** The reach: in the App Router root layout or in a middleware, evaluate the flags server-side using `posthog-node` and pass the evaluated values to the client as `bootstrap` data on `posthog.init`. The client SDK initializes with the flag values already populated; `useFeatureFlagValue('new_onboarding')` returns the real value on first render — no flash, no network round-trip. `@posthog/next`'s `bootstrapFlags` option does this end-to-end; the lesson walks the flow.
- **The distinct ID for server-side evaluation.** The flag evaluation needs the distinct ID to compute the per-user variant deterministically. `@posthog/next`'s middleware reads or seeds the distinct ID cookie at the request boundary; the server flag evaluation reads from the cookie; the client SDK reads from the same cookie. One identity across the SSR boundary, no skew. For authenticated routes, the user's known ID supersedes the cookie ID after sign-in (with `posthog.identify()` stitching the prior anonymous events).
- **Server-side flag evaluation patterns.** Two reaches. (1) **In middleware** — for routes where the flag controls a redirect or a layout swap (read the flag, set a response header, the layout reads the header). (2) **In a server component** — for routes where the flag controls a UI fork (`const variant = await posthog.getFeatureFlag('new_onboarding', distinctId)`; conditional render). Both pull from `posthog-node`; both pass the value to the client via bootstrap or a server prop.
- **The local-evaluation cache for server-side flags.** `posthog-node` can fetch the full flag config once (the targeting rules) and evaluate locally without a per-request network call. Set `personalApiKey` and enable local evaluation; the SDK refreshes the rules every 30s. Avoids a hot read on every page render. The `@posthog/next` defaults handle this; surface it so the student knows why server evaluation doesn't tank latency.
- **The `useFeatureFlagValue` and `useFeatureFlagPayload` hooks.** Client-side: `const flag = useFeatureFlagValue('new_onboarding')` returns the variant string or boolean (or `undefined` while loading — with bootstrap, never `undefined`). `useFeatureFlagPayload` for JSON payload flags. The hooks re-render when PostHog updates the flag remotely (a rollout percentage flip propagates without a redeploy).
- **The `$feature/...` super-property on every event.** Once a flag is evaluated, every subsequent event auto-attaches `$feature/new_onboarding: 'variant_a'`. This is what makes funnels and experiments work — the event store knows which variant the user was on when they fired the event. Don't fight this; rely on it for cohorting.
- **From flag to experiment — the metric-led setup.** An experiment is a flag with a metric attached: primary metric (`plan_upgraded` fires within 14 days of `paywall_viewed`), secondary metrics (event count, retention), and a statistical readout. The PostHog UI ships the experiment scaffolding; the code is the same as a multivariate flag (`useFeatureFlagValue('paywall_copy_test')`). The discipline: pick the primary metric *before* the experiment, sample size and significance threshold *before*, document the hypothesis in the experiment description. Stopping when the readout flickers green is the classic noise-as-signal failure.
- **Kill switches — flags for risky shipping.** The pattern: gate every newly shipped non-trivial feature behind a flag for the first week, default off, ramp to 100% once metrics stabilize. When something goes wrong, the flag flips off — no deploy, no rollback. The flag is the structural reach for incident response on a feature release. After a feature is stable, deleting the flag is also a discipline; stale flags rot in the code as `if (flag) { ... } else { ... }` forks that no one remembers to clean up. PostHog surfaces stale flags via the "Feature flag activity" view.
- **The kill-switch-vs-experiment-vs-rollout decision.** Three patterns, one primitive. **Kill switch** — boolean, default off, target = 100% of internal users until verified then 100% of all users; lifespan: weeks; delete after stabilization. **Rollout** — boolean, target = percentage ramp; lifespan: weeks; delete after 100%. **Experiment** — multivariate, target = 50/50 split, metric attached; lifespan: 2–4 weeks; convert to rollout (or delete the losing branch) after significance.
- **The stale-flag audit.** A quarterly pass to delete flags whose code branches have collapsed (everyone on `variant_a`, the `else` branch is dead code). PostHog's "Last evaluated" surface, plus a grep for the flag name across the codebase, plus the PR to remove the `if (flag)` fork. The audit reads as housekeeping but keeps the analytics and code clean.
- **Watch-outs.** Forgetting server-side bootstrap and shipping the flash-of-default for an experiment — the experiment's first day of data is poisoned; reading the flag in a server component without `posthog-node` local evaluation — every page render is a network call; the bootstrap distinct ID not matching the client distinct ID (different cookies, different evaluations) — variant flips between SSR and hydration; stale flags creating dead code branches that aren't tested; running an experiment without a pre-declared primary metric — the readout is unfalsifiable; stopping an experiment early on the first green day — false positive rate spikes; flag targeting on PII as a property (`email` matching) — keep targeting on bounded properties (`plan`, `seats`, `created_at`); the percentage rollout being non-deterministic per user (re-bucketing on every visit) — PostHog handles this via deterministic hashing on distinct ID, don't roll your own; experiments using non-PostHog events as the primary metric — the join breaks; deleting a flag while a deploy is in flight that still reads it — flag a stale flag, deploy, then delete.

What this lesson does not cover:

- Event taxonomy and identify — Lesson 20.2.4.
- Session replay — Lesson 20.2.6.
- Statistical experiment math at depth — out of scope; PostHog computes; the lesson covers the discipline.
- LaunchDarkly migration — out of scope.
- Backend-only flag systems (Unleash, OpenFeature) — named once; out of scope.

Estimated student time: 50 to 60 minutes. The chapter's largest lesson; flags plus experiments plus bootstrap is dense by design — these primitives ride together.

## Lesson 20.2.6 — Session replay with masking by default

Turn on PostHog session replay with the mask-vs-block masking catalog, sampling discipline, consent-gated start, and a replay-to-bug-fix workflow for the UX bugs that throw no error.

Topics to cover:

- **The senior question.** Three users this week emailed support saying the dashboard "broke." No console error in Sentry; the structured logs show every request returning 200; the user can't repro it on a call. The bug is a UX one — the user clicked the wrong button because the modal was visually misaligned on their viewport, or the click target was hidden behind a tooltip. Session replay is the surface that answers it: a privacy-masked, deterministic playback of the user's actual interactions with the rendered DOM. The lesson installs replay as the fourth PostHog primitive and the privacy posture that makes it shippable.
- **What session replay records — and doesn't.** DOM mutations, mouse moves, clicks, scrolls, viewport size, console errors, network request metadata (URL, status, duration — not bodies by default). It does *not* capture video; it captures the DOM and replays it in a deterministic player. This is why masking works — the player renders what was *recorded as the DOM*; masked elements were never in the recording.
- **Masking is structural and on by default.** PostHog's 2026 default: all `<input>` elements are masked at capture (the value is replaced with `***` in the recording). All text inside `<input>`, `<textarea>`, `<select>` is masked. Body text is captured unless explicitly masked. The shift from earlier defaults (capture-by-default for text, mask-by-default for inputs) is the post-GDPR direction; the lesson names the default explicitly.
- **The four masking levers.** (1) `session_recording: { maskAllInputs: true, maskTextSelector: '.sensitive, [data-sensitive]', blockSelector: '.never-record, [data-no-replay]' }` in `posthog.init`. (2) The `ph-no-capture` CSS class — elements with this class are masked at recording time. (3) The `data-ph-capture-attribute-*` attributes for explicit unmask of specific text. (4) The `mask_text_class` and `mask_text_selector` config for class- or selector-based masking. The four levers cover the spectrum from "mask everything I don't whitelist" to "capture everything I don't blacklist." Default to the conservative posture: mask all inputs, mask anything classed `.sensitive`, block anything classed `.never-record`.
- **The mask vs. block distinction.** **Mask** — the element is recorded with its content replaced (the DOM structure, position, and dimensions are preserved; the text is `***`). The replay shows the user *interacting with* a masked field. **Block** — the element is not recorded at all; the replay shows an empty placeholder of the same size. Reach for *mask* when the structure matters (the user typed in the password field — show that they typed, not what); reach for *block* when the presence of the element shouldn't be on the recording (a third-party iframe carrying PII, a customer's billing details).
- **What to mask by class — the catalog.** A small set of repeat targets. `.sensitive` — everything classed sensitive (the team uses this class on PII fields). Customer name, address, phone in profile pages. Free-text fields where customers paste anything (notes, descriptions). Credit-card iframes (Stripe Elements is third-party-iframed and already isolated, but classing the wrapper as `.never-record` is belt-and-suspenders). The audit log preview (operator PII visible). Class the targets; the masking config is one line.
- **The consent gate's reach on replay.** Replay is in scope for the same consent gate as events (Lesson 20.2.3's wiring). Replay only records on accepted consent. `disable_session_recording: true` until consent; `posthog.startSessionRecording()` after `posthog.opt_in_capturing()`. The four-state machine governs replay the same way it governs events.
- **Sampling — `session_recording: { sampleRate: ... }`.** Recording every session at scale burns the quota and turns the inbox into a haystack. Default: 100% recording of identified users on paid plans (high-signal, low-volume), 10% sample of anonymous traffic. Adjust per the team's reach. Alternative: trigger-based recording — only record sessions where a specific event fires (a `paywall_viewed` or a `support_chat_opened`) — `record_user_on_event` or PostHog's recording trigger UI.
- **Console log and network capture.** PostHog records `console.error` lines and network request metadata into the replay timeline; these light up next to the DOM playback. The reach: the operator scrubs to the moment of the user's bail, sees the 500 in the network panel and the console error, pivots to Sentry by the request URL. **Don't enable network body capture by default** — bodies carry PII; the URL and status are sufficient unless a specific debugging cycle calls for more.
- **PostHog Cloud EU and replay storage.** Replays live in the same region as the rest of the data; EU Cloud stores in Frankfurt. Retention defaults: replays are retained 30 days on the free tier, longer on paid tiers. Replays are deletable per user (the GDPR deletion path from Chapter 17.2.4 includes a PostHog deletion call — `posthog.deletePerson(distinctId)` removes the user's events and replays).
- **The replay-to-bug-fix workflow.** Modeled end-to-end. A user reports "the upgrade button does nothing." The on-call: (1) opens PostHog, filters replays by `distinctId` (the user's PostHog ID looked up from the user row), (2) finds the session, scrubs to the click on the upgrade button, (3) sees the modal opens but immediately closes — the DOM mutation timeline shows a stray `click` event firing on the backdrop, (4) the bug is a missing `e.stopPropagation()` on the modal content, (5) the fix is one line. The replay made the bug reproducible without a call to the user.
- **The decision: when not to record.** Internal admin tools (a separate PostHog project, no replay), customer-data export flows (the entire flow blocked), authentication forms (default masking covers passwords; the username can stay captured for support pivot), payment forms (Stripe iframe is third-party-isolated; classing the wrapper as `.never-record` keeps the noise down). The default is *record with masking*; the negative space is the catalog above.
- **The privacy review surface.** Before shipping replay to production, the team runs through a "what shows in a replay" review — open a session, scrub through, identify any unmasked PII, add the masking class. The review reads as a 30-minute one-off plus a quarterly recurrence. Replay is the easiest PostHog primitive to misconfigure into a GDPR violation; the review is the discipline.
- **Watch-outs.** Forgetting to mask a field that becomes PII after a feature ships (a new "notes" textarea collecting customer addresses) — the masking config drifts; relying on default masking and shipping a custom non-input form (a `<div contenteditable>`) that the default doesn't catch; enabling network body capture for debugging and forgetting to disable — bodies leak indefinitely; recording 100% of anonymous traffic and burning the quota in two days; replay surfacing in a screen-share with a customer — the operator sees PII even with masking on, treat replay as operator-side access; replay disabled by an ad-blocker for some users — accept the partial coverage, the masked sessions are still load-bearing; replay running pre-consent — verify with DevTools that no `/ingest` calls fire before accept; replay capturing a Stripe iframe's content by accident — third-party iframes are already isolated from the parent DOM, but the wrapper is in scope, class it; the replay UI showing the wrong distinct ID stitch (anonymous-then-identified) — `posthog.identify()` stitches replays the same way it stitches events, the worked flow handles it.

What this lesson does not cover:

- Sentry's session replay (the alternative; the course picks one, this chapter chose PostHog) — named once in 20.1.1.
- The cookie consent gate's internals — Chapter 17.2.5.
- GDPR deletion mechanics — Chapter 17.2.4 (this lesson points to it).
- Real-user performance monitoring (the Speed Insights surface) — Lesson 20.2.1 and Chapter 20.3.
- Replay storage at self-hosted scale — out of scope (course uses PostHog Cloud).

Estimated student time: 35 to 45 minutes. Pattern-and-decision lesson; the masking catalog and the bug-fix workflow are the load-bearing content.
