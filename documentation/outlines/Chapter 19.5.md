# Chapter 19.5 — E2E (conditional)

## Chapter framing

Chapter 19.5 is the **top, narrowest band of the honeycomb** from 19.1 — conditional in the same shape as 19.4 and the Chapter 16 client-state tools. The default for a Next.js 16 SaaS in 2026 is **zero or near-zero E2E tests in year one**: the runtime cost is seconds-per-test instead of milliseconds, the flake budget is structurally larger (real browser, real network, real third-party UIs), and the bug-density argument from 19.1.2 points at the seam, not the browser. Playwright earns its weight only on the 20–30 paths where failure costs real money — sign-in to a paid surface, the Stripe Checkout redirect and return, the webhook-driven plan flip the user sees after paying, the invitation-acceptance flow that grants org access. Everything else stays at the seam (19.3) or at the component (19.4) or off the test menu entirely. The lesson the chapter installs is the **money-path filter**: if the path doesn't touch money, identity, or data the user can't recover from a mistake on, Playwright is the wrong tool.

Threads through every lesson. **Playwright is pinned to the latest 1.x line, May 2026** — `@playwright/test` with TypeScript, three browsers configured (Chromium, WebKit, Firefox) but only Chromium runs in CI default; WebKit and Firefox are opt-in for the auth and checkout paths only. **Tests run against a production build** (`next build && next start`) via `webServer` in `playwright.config.ts`, never against `next dev` — the rendered output the test asserts on must be the output users get. **Authentication is `storageState`** — one `auth.setup.ts` signs in real users (one per role) and writes JSON state files; every other test inherits via `use: { storageState: '...' }` and never re-logs-in. **The test database is a separate Playwright-owned Postgres** with deterministic seed data — Playwright cannot share the per-worker DB lifecycle from 19.3 because the server it drives is a separate process. **MSW does not run in E2E** — the boundary that mattered at the integration layer is *the actual third party* at this layer, with the exception of Stripe Checkout where the test card and test-mode keys are the contract. **`trace: 'on-first-retry'`, `retries: 1` in CI, `0` locally** — the trace viewer is the debugger; flake gets a structural fix, not a retry bump. **Money paths are the canon** — every example lands on auth, checkout, the webhook-to-UI loop, or invitation acceptance, the four paths the project chapter (19.6) will harden.

## Lesson 19.5.1 — When Playwright earns its weight

Topics to cover:

- **The senior question.** A new feature lands. The team's reflex from the test-pyramid era is "write an E2E test for the new flow." Six months in, the suite has 80 Playwright tests, CI runs 25 minutes, a third of them are flaky on Tuesday mornings, the team learns to retry-until-green, and the bug that ships to production wasn't covered by any of the 80. The reach is the inverse rule: **E2E is off by default**; it earns weight only against the money-path filter. Playwright is a **money-path tool, not a coverage tool.**
- **The money-path filter, named.** A test earns the runtime cost when failure on that path means (1) money moves wrong (a charge fires twice, a plan downgrade silently fails, a refund doesn't reach the customer), (2) identity breaks (sign-in is broken on production and users can't reach the app), (3) data the user can't recover from is lost or exposed (invitation grants the wrong org access, an export omits records). Anything else stays at the seam (19.3) or the component (19.4) or off the menu entirely.
- **The 20–30 paths shape.** A mid-stage SaaS has under thirty money paths. The first ten are universal: sign-in (email+password, OAuth providers used), sign-out, the checkout redirect to Stripe, the return from Stripe and the plan flip in the UI, invitation acceptance with seat grant, password reset, the primary "create-the-thing-customers-pay-for" path. The next ten are app-specific (the export the user buys the plan for, the report the auditor needs). Past thirty, the test budget is being misallocated.
- **Year-one zero is common.** A small team shipping fast on this stack with disciplined integration tests (19.3) and a sentry-led production observability surface (Unit 20) is *correct* to ship year one with zero Playwright tests. The integration suite catches the seam bugs; production catches the unknown ones; manual smoke-testing covers the rest until the team can afford the runtime cost. The trigger language tells them when to start, not "we should have E2E."
- **What E2E catches that nothing else does.** The full stack composed: routing, middleware, server actions, database, third-party (Stripe, the email service), the rendered HTML, the JS hydration, the cookie store, the cross-page navigation. A bug that lives in *the composition* — a redirect that loops, a session that doesn't survive the Stripe round-trip, a webhook arriving before the UI re-fetches — only the browser-driven test sees. Worth the seconds-per-test on the paths that matter.
- **What E2E does NOT catch better than cheaper tests.** Form validation branches (component test, 19.4). The Server Action's database write (integration test, 19.3.7). The webhook signature verification (integration test, 19.3.6). The accessible name on a button (component test or a separate axe-core audit). Reaching for E2E for these is the trap that creates the slow-flaky-suite anti-pattern.
- **The cost shape, in numbers.** Unit test: 5 ms. Integration test with rollback: 20–80 ms. Component test in jsdom: 100–300 ms. Playwright test: 2–10 seconds local, sometimes more in CI. A 30-test suite costs roughly 2 minutes in CI when parallelized; a 200-test suite is a 20-minute pole. The discipline is the count, the same way 19.4 disciplines the component count.
- **The flake budget is structurally larger.** A real browser, a real network, a real third-party UI (Stripe Checkout), real timing all combine into a flake surface integration tests don't have. The 2026 reach is *structural fixes only*: better locators (role-first, not CSS), `expect.poll` and auto-waiting locators instead of `waitForTimeout`, fixtures over ad-hoc setup, `trace: 'on-first-retry'` so flakes get a debugger, `retries: 1` so the first re-run differentiates flake from real failure. Adding `retries: 3` to make CI green is the anti-pattern; it hides real bugs.
- **The senior's "before you write this test" checklist.** (1) Is this a money path under the filter? Stop if no. (2) Is the same composition already covered by an integration test plus a component test that, combined, would catch the same bug? Stop if yes — composition is the only justification. (3) Does the path cross a third party Playwright can actually drive (Stripe Checkout test mode, an OAuth provider with a test account)? Stop if no — write the seam test instead. (4) Will the test be deterministic without sleeps? Stop if no — fix the test or the app before adding the test. The four-step gate sits in front of every new Playwright file.
- **Why "production build, not `next dev`."** Dev mode runs different code paths: no static optimization, dev-only error overlays, dev-only middleware behavior, no minified hydration. An E2E test that passes against `next dev` and fails against `next start` (or the reverse) is the bug that ships. Playwright's `webServer` config points at `next build && next start` from lesson one.
- **What the chapter teaches and what it doesn't.** Teaches: the trigger filter (this lesson), Playwright's primitives in the 2026 idiom (19.5.2), the four money paths in the course's app (19.5.3). Does not teach: visual regression (Chromatic, Percy), accessibility audits at depth (axe-core), load testing (k6), mobile-app E2E (Detox/Appium), every Playwright API surface (the docs own that). The cut keeps the chapter the size of the trigger.
- **Forward references.** 19.5.2 owns Playwright config, the `webServer`, fixtures, `storageState`, locators, `expect` with auto-waiting, the trace viewer. 19.5.3 walks the four canonical money paths in the course's app. 19.5.4 is the quiz. The project chapter 19.6 hardens the Stripe checkout money path end-to-end.
- **Watch-outs.** Writing an E2E test "to be safe" without naming the money-path trigger — slow flaky suite by month six; adding `retries: 3` to keep CI green — hides real flake causes; running tests against `next dev` for speed — the bug ships from the build mismatch; using `page.waitForTimeout(...)` to wait for state — replace with `expect(locator).toBeVisible()` or `expect.poll`; logging in with the UI in every test instead of `storageState` — multiplies runtime; treating Playwright as a manual-testing replacement — manual exploratory testing still owns the unknown-unknowns.

What this lesson does not cover:

- Playwright config, fixtures, `storageState`, locators, trace viewer — Lesson 19.5.2.
- The course's money paths and how each is tested — Lesson 19.5.3.
- Visual regression, load testing, accessibility audits at depth — out of scope.
- The Stripe-checkout project — Chapter 19.6.

Estimated student time: 30 to 40 minutes. Decision lesson; the rest of the chapter sits under the trigger this one installs.

## Lesson 19.5.2 — Playwright primitives in the 2026 idiom

Topics to cover:

- **The senior question.** Playwright is greenlit for the four money paths (19.5.1). What's the smallest configuration surface that runs them against a production build, signs the test users in once, runs in CI without flake, and produces a trace when something fails? The reach: `playwright.config.ts` with `webServer`, a `tests/e2e/` directory, an `auth.setup.ts` writing `storageState`, role-first locators, `expect`'s auto-waiting matchers, `trace: 'on-first-retry'`.
- **Install and layout.** `pnpm create playwright` (or `pnpm dlx`) on first run; results in `playwright.config.ts`, `tests/e2e/` (the course's home), `playwright-report/` (gitignored), `test-results/` (gitignored), and Playwright browser binaries in pnpm's store. The course pins to the current 1.x line — `@playwright/test` with TypeScript out of the box.
- **`playwright.config.ts` — the load-bearing surface.** `testDir: 'tests/e2e'`, `fullyParallel: true`, `forbidOnly: !!process.env.CI`, `retries: process.env.CI ? 1 : 0`, `workers: process.env.CI ? 2 : undefined`, `reporter: process.env.CI ? [['github'], ['html']] : 'list'`, `use: { baseURL: 'http://localhost:3000', trace: 'on-first-retry', screenshot: 'only-on-failure', video: 'retain-on-failure' }`. Every line earns its place — they map to the discipline from 19.5.1.
- **`webServer` config.** `webServer: { command: 'pnpm build && pnpm start', url: 'http://localhost:3000', reuseExistingServer: !process.env.CI, timeout: 120_000 }`. Production build only. `reuseExistingServer` lets local iteration reuse a running server; CI always builds fresh.
- **Browser projects.** `projects: [{ name: 'setup', testMatch: /.*\.setup\.ts/ }, { name: 'chromium', use: { ...devices['Desktop Chrome'], storageState: '...' }, dependencies: ['setup'] }]`. Chromium is the default in CI. WebKit and Firefox are added as opt-in projects (`PLAYWRIGHT_PROJECTS=all`) for the auth and checkout paths only — cross-browser coverage on money paths, single-browser everywhere else. The dependency on `setup` is how `storageState` arrives.
- **`auth.setup.ts` — the one-time sign-in.** A test file in the `setup` project. Imports the e2e fixtures, navigates to `/sign-in`, fills the email and password for the test user, awaits the post-login redirect, then `await page.context().storageState({ path: '.auth/owner.json' })`. One setup file per role the suite needs — owner, member, admin. The `.auth/` directory is gitignored — these files contain real session cookies. The course's reach: one user per role, all created by a seed script (next bullet) so the credentials are stable.
- **The E2E test database and seed.** Playwright drives a real server that needs a real DB. The course's reach: a separate Postgres database (`saas_e2e`) — not the dev DB, not the integration-test per-worker DB. Before the Playwright run, a `pnpm db:e2e:reset` script runs migrations and a deterministic seed that creates the test users (`owner@e2e.test`, `member@e2e.test`), their organizations, and a baseline of records. CI runs the reset once before the suite; local runs reuse unless the schema changed. Tests assume the seed and write incrementally; the reset is the isolation seam between full runs.
- **Locators, role-first.** Same priority ladder as RTL (19.4.3) — `page.getByRole('button', { name: /sign in/i })`, `page.getByLabel(/email/i)`, `page.getByText(...)`, `page.getByTestId(...)` last resort. Playwright's locators are *auto-waiting*: `await page.getByRole('button', { name: /pay/i }).click()` waits until the button is attached, visible, stable, and enabled before clicking. No `waitForSelector`, no manual `waitForTimeout`. The locator API is the flake-prevention surface.
- **`expect` with auto-waiting matchers.** `await expect(page.getByRole('heading', { name: /dashboard/i })).toBeVisible()` polls until the assertion holds or the 5-second default times out. `toBeVisible`, `toHaveURL`, `toHaveText`, `toBeEnabled`, `toBeChecked` are the common reach. `expect.poll(fn)` is the escape hatch for non-DOM polling — wait for a database row, wait for a webhook to land. Both replace every legacy "sleep then assert" pattern.
- **Fixtures over `beforeEach`.** Playwright's `test.extend<...>({ ... })` builds custom fixtures — a `signedInPage` fixture that returns a page with `storageState` already applied, a `seedInvoices` fixture that inserts records and yields them. Fixtures compose, are typed, and are per-test by default — same discipline as Vitest's per-test isolation from 19.3.2. The course writes a small `e2e-fixtures.ts` exporting `test` and `expect` from a single import; every test file imports from there, not from `@playwright/test` directly.
- **The trace viewer is the debugger.** With `trace: 'on-first-retry'`, a failed test produces a `trace.zip`. `pnpm exec playwright show-trace test-results/.../trace.zip` opens the GUI: DOM snapshot at every action, network log, console log, screenshot, source-mapped stack. No `console.log` debugging. In CI, the trace is uploaded as a GitHub Actions artifact; the PR reviewer downloads and inspects. This is *the* 2026 Playwright debugging workflow.
- **Codegen and the VS Code extension.** `pnpm exec playwright codegen http://localhost:3000` opens a browser that records actions as Playwright code with locators pre-generated. Useful for the first draft of a new test; the senior reach is then to rewrite the locators for stability (codegen sometimes picks CSS selectors over role-based ones — fix in review). The VS Code extension exposes Run/Debug per test, live trace, and locator picker — named once.
- **Sharding in CI.** When the suite passes 10–15 minutes, sharding kicks in: GitHub Actions matrix with `--shard=1/4`, `--shard=2/4`, etc., runs four parallel jobs. Below 10 minutes, the overhead isn't worth the artifact-stitching complexity. The course's chapter-end suite is small enough to skip sharding; the project chapter mentions the seam.
- **Network — Playwright vs. MSW.** The integration suite's MSW boundary (19.3) does not run here — Playwright runs in a separate process. Inside the browser, Playwright can `page.route(url, handler)` to mock specific network requests (a deterministic Stripe redirect URL, a deterministic third-party API response in a test that's not about that third party). For Stripe Checkout itself, the path is to drive the actual `checkout.stripe.com` test-mode page with the `4242 4242 4242 4242` test card — Stripe is part of the system under test, not the mock. For OAuth providers the seam is more brittle and the course's reach is a per-provider judgment call (named in 19.5.3).
- **Mocking time and date.** Playwright supports `page.clock.install({ time: ... })` and `page.clock.fastForward(...)` to control `Date.now()` in the browser context — same intent as Vitest's `vi.setSystemTime` from 19.2.3, different surface. Reach when a test asserts on trial-end dates or any time-based UI.
- **Cleanup and per-test isolation.** Every test gets a fresh browser context (Playwright's default). `storageState` injects auth without UI login. Tests should be independent — no implicit "test A runs first or test B breaks." If a test writes to the DB, it cleans up in `afterEach` *or* asserts only on its own org's data so other tests don't conflict; the seed is sized so that per-test writes don't collide.
- **What this lesson does NOT cover.** Reading the trace viewer output in depth — the docs own it; the lesson names where to find traces and what they show. Custom reporters — pinned to GitHub + HTML. Mobile emulation — out of scope for SaaS-focused. Component testing in Playwright (`@playwright/experimental-ct-react`) — RTL owns the component layer (19.4); the course doesn't dual-pin.
- **Watch-outs.** Running tests against `next dev` "for speed" — different code paths, ship the bug; `page.waitForTimeout(2000)` — replace with auto-waiting matchers; CSS locators (`page.locator('.btn-primary')`) — refactor breaks the test; logging in with the UI on every test — multiplies runtime by 5–10×; committing `.auth/*.json` files — session cookies in git; `retries: 3` to mask flake — hides real causes, fix the locator or the wait; running the integration test DB and the e2e DB on the same database — write conflicts; assertion without `await` — `expect(locator).toBeVisible()` without await silently passes; one mega-test for the whole checkout flow — first failure hides everything after, split into named steps; ignoring trace artifacts on CI failure — the entire debugging seam stays unused.

What this lesson does not cover:

- The course's money paths and how each is tested — Lesson 19.5.3.
- Visual regression — out of scope.
- Mobile emulation — out of scope.
- Component testing in Playwright — the course uses RTL for that layer (19.4).
- Project-chapter Stripe-checkout build — Chapter 19.6.

Estimated student time: 50 to 60 minutes. Setup-and-wiring lesson; the surface installed here drives everything in 19.5.3 and 19.6.

## Lesson 19.5.3 — Playwright in our app — the four money paths

Topics to cover:

- **The senior question.** 19.5.1 named the trigger; 19.5.2 named the surface. Which paths in the course's app actually pass the filter, and what does the test for each look like? Pattern lesson; the catalog is the canon the team extends without re-litigating the trigger. Four money paths, ordered by how universally they apply.
- **Money path 1 — sign-in (Unit 9).** The first money path of every SaaS. Failure means users can't reach the product they're paying for. Behaviors to assert: `/sign-in` renders with email and password fields; submitting valid credentials redirects to `/dashboard` and the user's name is visible; submitting invalid credentials renders an error alert with the right accessible name and stays on `/sign-in`; the rate limiter from Unit 15 blocks the sixth attempt and surfaces the lockout copy. Uses `storageState` *not* for this test (the test exercises login itself) — uses fresh contexts. Cross-browser: this is one of the two paths that runs in WebKit and Firefox too. Watch-out: rate-limit state must reset between test runs — the seed script clears the Upstash counters for `*@e2e.test` emails.
- **Money path 2 — Stripe Checkout round-trip (Unit 12).** The canonical money path. Failure means a user pays and doesn't get the plan, or doesn't pay and does. Behaviors to assert: signed-in owner navigates to `/billing`, clicks "Upgrade to Pro," lands on `checkout.stripe.com` (assert URL); fills the test card `4242 4242 4242 4242` with future expiry, any CVC, any ZIP; submits; returns to `/billing/success` (assert URL); the UI reflects the plan as "Pro" (read from the DOM, which is the user's view of truth). The Stripe-test-mode UI is part of the system under test — Playwright drives the iframe locators Stripe exposes for test-card entry. The webhook arriving and flipping the plan in Postgres is the seam-test job (19.3); the *test that the UI shows it* is this one. The integration test from 19.3.6 covers the webhook in isolation; this E2E test covers the composition. Use `storageState` for the owner user. Single-browser is acceptable here for the cost; the project chapter (19.6) revisits.
- **Money path 3 — invitation acceptance (Unit 10).** Failure means a team member is granted access to the wrong org, or denied access to the right one — data-boundary risk with multi-tenant blast radius. Behaviors to assert: a fresh user receives an invitation email (intercepted via the Resend test webhook or seed-inserted token — the email send itself is the seam-test job); the user clicks the invite link with the signed token; lands on a sign-up form pre-filled with the invited email; submits credentials; lands on the org's dashboard with the assigned role visible in the UI; tries to access another org's resource and gets a 404 (the multi-tenant guard from Unit 10). This test runs without `storageState` — the invitee is a new user each run, created and torn down per test. The expired-token case is a separate, smaller test asserting the rejection alert renders.
- **Money path 4 — the primary "thing customers pay for."** App-specific; the course's working app is the invoice/billing surface, so the money path is "create an invoice, send it, the recipient pays via Stripe, the invoice flips to paid in the UI." The pattern is what generalizes: identify the one or two end-to-end loops where every layer must align for the customer to get value. For each, the test signs in as the right role (`storageState`), exercises the create surface, walks the third-party round-trip if any, returns, and asserts the user-visible outcome. The catalog above is illustrative; in a real codebase, this slot is filled with one or two app-specific paths.
- **What does NOT make the catalog.** Form validation branches — component-test territory (19.4). Search results, filter combinations, pagination — URL-state integration tests (19.3). The settings page, the docs page, the marketing landing — no money flows. Server-Action behavior in isolation — integration tests (19.3.7). Components rendering correctly in a specific locale — component tests (19.4.4). Every one of these reaches has a cheaper, more reliable home.
- **OAuth providers — the conditional reach.** If the app's primary sign-in is "Sign in with Google" (Unit 9), the E2E test either uses a Google test account (real OAuth round-trip — slow, sometimes brittle, against Google's ToS for automation in many cases) or stops at the OAuth redirect URL and asserts the redirect URL parameters are correct (faster, lower-fidelity but typical). The course names the trade-off and recommends the redirect-URL assertion for most teams; full OAuth round-trip belongs in a quarterly manual-QA pass, not the per-PR test suite.
- **Stripe Checkout iframe selectors.** Stripe's hosted checkout page uses iframes for card entry. Playwright drives them via `page.frameLocator('iframe[name="card-element"]').getByLabel(/card number/i).fill('4242...')`. The locators are stable enough across Stripe's UI evolution; pin to role+name within the frame, not CSS within the frame. Stripe Test Clocks (the API surface for fast-forwarding subscription billing) are the integration-test seam (19.3), not the E2E reach — driving an end-to-end billing cycle in a single test inflates runtime past the budget. Named once.
- **Per-test data hygiene.** Each money-path test creates data with a unique identifier (`invoice-${test.info().title}-${Date.now()}`) so parallel tests don't collide on the seeded DB. The owner user's org is shared across tests but the records inside are addressed by name. Cleanup is deferred — the next full reset (`pnpm db:e2e:reset`) is the canonical clean.
- **CI integration shape.** A `playwright.yml` workflow runs after the build job; depends on the DB being reset; uploads the HTML report and trace artifacts on failure; comments the trace link on the PR. The four-path suite runs in 3–6 minutes on Chromium-only; adding WebKit and Firefox for sign-in and checkout brings it to 8–10 minutes. Past 15 minutes, shard.
- **The reviewer checklist for a new Playwright test PR.** (1) Which money path does this cover? Name it in the PR description; if you can't, the test doesn't pass the filter. (2) Is it role-first locators throughout? Any `data-testid` or CSS locator needs justification. (3) Does the test use `storageState` instead of UI login? (4) Does it pass with `--retries=0` ten times in a row locally? (5) Does it touch a third party — if Stripe, is the test-mode key in use; if not, why is this an E2E? (6) Has a `trace.zip` been generated and reviewed against the assertions? Six checks, fast.
- **The "year-one zero, year-two five" trajectory.** A team starting Playwright in year two reaches first for sign-in (path 1) and Stripe checkout (path 2). Paths 3 and 4 follow when the team grows past one engineer being able to manually verify them every release. The four-path catalog is the *destination*, not the day-one count. A senior reviewing a junior's first Playwright PR pushes for fewer tests, better-chosen, not more.
- **Watch-outs.** Writing a Playwright test for every page "to have coverage" — slow suite by month six; using the dev server with `next dev` instead of `next start` — environment mismatch; logging in via the UI on every test — `storageState` is the reach; asserting on iframe internals beyond what Stripe officially documents — fragile to Stripe UI changes; running the E2E DB against shared infra (dev or integration) — write conflicts; "smoke" tests that only check that pages render — covered better by a curl-based health check; reaching for visual snapshot tests inside Playwright — separate tool (Chromatic) if the team can afford it, otherwise off the menu; building a Page Object Model framework before there are 30+ tests — premature; using `test.skip` to disable a flaky test instead of fixing the locator or the wait — flake compounds.

What this lesson does not cover:

- Playwright configuration, fixtures, `storageState`, locators, trace viewer — Lesson 19.5.2.
- Visual regression tooling — out of scope.
- Stripe Test Clocks for full billing-cycle simulation — integration territory (19.3), named once.
- Component-level interactive testing — Chapter 19.4.
- The Stripe-checkout project hardening — Chapter 19.6.

Estimated student time: 50 to 60 minutes. Pattern lesson; the four-path catalog is the canon the team extends.

## Lesson 19.5.4 — Chapter quiz

Top 10 topics to quiz:

- The money-path filter — money moves wrong, identity breaks, or unrecoverable data loss/exposure; everything else stays at the seam or the component layer; E2E is off by default for a Next.js SaaS in 2026.
- Year-one zero is correct for a small team with disciplined integration tests and production observability; 20–30 money paths is the upper bound, not the target.
- Playwright must run against the **production build** (`next build && next start`) via `webServer`, never against `next dev`; dev mode is a different code path and the environment mismatch ships bugs.
- `storageState` is the auth reach — one `auth.setup.ts` per role writes a JSON state file, every other test inherits via the project `dependencies` config; UI login per test is the anti-pattern.
- Locator priority — role+name first, then label, text, `getByTestId` last resort; locators are auto-waiting, so `waitForTimeout` and manual `waitForSelector` are replaced by `expect(locator).toBeVisible()` and auto-waiting matchers.
- `trace: 'on-first-retry'`, `retries: 1` in CI, `0` locally — the trace viewer is the debugger; bumping retries to mask flake is the anti-pattern; structural fixes only.
- The E2E test database is a *separate* Postgres from dev and from the integration per-worker DBs; deterministic seed creates the role users; `pnpm db:e2e:reset` is the isolation seam between full runs.
- Network at the E2E layer — MSW does not run; third parties are part of the system under test; Stripe is driven via test mode with the `4242` test card; OAuth providers are usually stubbed at the redirect URL, not the full round-trip.
- The four canonical money paths in a SaaS — sign-in, Stripe Checkout round-trip with plan flip in UI, invitation acceptance with seat grant, the app-specific primary value loop; the catalog is the canon and the reach is fewer-better tests.
- E2E vs. integration vs. component coverage rules — composition bugs to E2E, seam bugs to integration (19.3), branching-content and accessibility bugs to component (19.4); duplicating coverage at a higher cost is the anti-pattern.
