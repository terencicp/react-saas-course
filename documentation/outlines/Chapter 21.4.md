# Chapter 21.4 — Schema migrations against a live app (SaaS pattern #11)

## Chapter framing

Chapter 21.4 closes the loop Chapter 6.5 opened. 6.5.1 taught the daily `generate` → `migrate` mechanic; 6.5.2 named the production hand-edits and the three column-change patterns that bite (NOT NULL on existing rows, rename-in-place, type-rewrite). What 6.5.2 deferred is the choreography across multiple deployments: when the app is live, the schema and the code change at different moments, and any change that breaks the old code reading the new schema or the new code reading the old schema is an outage. This chapter installs the expand → migrate → contract cadence as the senior reflex for that class of change, names the triggers (which migrations actually need three deploys and which ship in one), and lands the verification step that runs the migration against the per-PR Neon branch from 21.3.5 before the merge.

Threads that run through every lesson. **The deploy is not atomic with the migration**, even on Vercel — for the seconds-to-minutes window between the migration applying and the new function pool fully warming, both code versions hit the database; the schema must work for both. **Forward-only, three deploys minimum** for any destructive change — the old shape stays alongside the new shape until the new code has been live long enough that the old shape is unread; then a separate deploy drops it. **Dual-write is application-layer choreography**, not a database trick — the migrate phase writes to both columns from the server action so the new column never falls behind while the backfill is running. **The Neon preview branch (21.3.5) is the rehearsal stage** — every migration in the cadence is exercised against a copy-on-write branch of production data before the PR merges, and the rehearsal catches the failure modes the cadence is designed to prevent. **The cadence has a cost** — three PRs, three reviews, three deploys, two weeks elapsed for a column rename — and the senior diff is recognizing which migrations need it (so the cost is paid) and which don't (so it isn't). **The cadence is the prevention; rollback is the cure** — 21.3.7's instant rollback doesn't undo a forward-only migration, which is precisely why every step in the cadence must leave production runnable on the *previous* deploy.

## Lesson 21.4.1 — The expand → migrate → contract cadence

Topics to cover:

- **The senior question.** A live app means there's always a request mid-flight against the database. A migration that changes the *shape* the app expects — drops a column, renames it, flips it to NOT NULL — leaves some requests broken in the seconds the deploy and migration are racing. The cadence is the three-step pattern that lets that change ship without an outage: expand the schema, migrate the data while the app dual-writes, contract once the old shape is unread.
- **Why one deploy isn't enough.** Vercel deployments are atomic at the alias-swap (21.3.1) but the function pool warms over seconds and the database is shared across both versions. For a non-trivial window, the v1 function fleet (still reading the old column name) and the v2 function fleet (reading the new name) both hit the same database. If the migration ran before the swap, v1 reads fail; if after, v2 reads fail. The schema has to satisfy both code versions for that window — that's the constraint the cadence solves.
- **Expand — the first deploy.** Add the new shape *alongside* the old. New nullable column, new table, new index — additive only, never destructive. Migration ships; old code continues working untouched (the new column is nullable, the new table is unread, the new index is unused). The PR contains: the schema edit, the generated SQL, and ideally no application-code change. Production is now running v1 code against a schema that has both shapes. Reversible by `git revert` of the PR's app-code changes (which are empty) plus a forward-fix migration that drops the new column (cheap because nothing reads it yet).
- **Migrate — the second deploy.** Application code starts writing to both columns (`dual-write`) and reading the new column where possible, with a fall-through to the old column if the new is null. A backfill script populates the new column for historical rows. The PR contains: the server-action edits to dual-write, optionally the backfill as a one-shot script, optionally a feature flag to gate the read path. Production now has v2 code writing to both columns; old data has been backfilled; new data is being written to both. **The system is now correct against either schema state** — this is the safety the cadence buys. Reversible by `git revert` because the old column still has the truth.
- **Contract — the third deploy.** Remove the old shape. Drop the old column, drop the dual-write code, drop the fall-through read. The PR contains: the schema edit, the generated SQL with the `DROP COLUMN`, and the app-code cleanup. Production runs only against the new shape. The cadence is complete. Reversible only by a forward-fix that re-adds the column and backfills from somewhere — at this point, rollback past contract is a data-recovery problem, not a deploy problem.
- **The dual-write pattern in app code.** Inside the server action (Unit 7) or repository helper that mutates the affected row, write to both columns in the same SQL statement. Drizzle's `update` and `insert` take both fields; one `UPDATE` sets both. The dual-write is structural, not opt-in — it lives in the same code path the application already uses, so every mutation hits both columns whether the developer remembers or not. The dual-write code is born to be deleted; the contract deploy removes it.
- **The backfill — bounded, batched, idempotent.** A separate script (`scripts/backfill_<change>.ts`) updates historical rows in batches of 1,000–10,000 rows so a single transaction doesn't lock the table for minutes. The script is idempotent — running it twice doesn't double-write — typically via a `WHERE new_column IS NULL` guard. Reach for a one-shot script run from the developer's machine against the production unpooled URL for small datasets; reach for Trigger.dev (Unit 13) for backfills measured in millions of rows where the work needs to be observable and resumable.
- **The dual-read fall-through.** While the backfill is in progress, some rows have the new column populated and some don't. The read path needs `coalesce(new_column, old_column)` or an explicit `IF NULL` branch so the application sees a consistent value regardless of which the row has. Lives in the query helper, not scattered across call sites. Removed in the contract deploy.
- **Feature flags as the optional fourth lever.** When the new column powers a new feature (not just a schema change), gate the read path behind a flag (PostHog from Unit 20.2). The flag lets the team enable the new behavior in stages — internal users first, then a percentage, then everyone — without another deploy. Flag flip is the fastest rollback if the new column has a data-quality issue. The flag is also deleted in the contract deploy, or shortly after.
- **The migration files across the three PRs.**
  - Expand PR's migration: `ALTER TABLE invoices ADD COLUMN customer_id uuid REFERENCES customers(id)` (nullable, no `NOT NULL`).
  - Migrate PR's migration: usually none (the backfill is app-code, not DDL); occasionally an index added concurrently on the new column.
  - Contract PR's migration: `ALTER TABLE invoices DROP COLUMN customer_name` plus `ALTER TABLE invoices ALTER COLUMN customer_id SET NOT NULL`.
- **Forward-only — the rule restated.** No down migrations (6.5.2). Each step in the cadence is a forward migration that leaves the system runnable. Rolling back the expand deploy: forward-fix migration drops the new column. Rolling back the migrate deploy: `git revert` the app-code PR (data in the new column doesn't hurt; the read path falls through to the old). Rolling back the contract deploy past the column drop: forward-fix that re-creates the column and backfills from a known source — expensive and rare.
- **Wall-clock pacing.** Each deploy needs time to soak before the next one. Expand: ship and wait at least until the next quiet period (overnight, weekend); confirm no errors against the additive schema. Migrate: ship, run the backfill, confirm the new column matches the old for a sample of rows, wait until you trust the dual-write has been universal across the function fleet for a full deploy cycle. Contract: ship. A full cadence typically takes one to three weeks of calendar time; the calendar is the safety margin, not theatre.
- **What the cadence does not solve.** Concurrent business-logic changes — when the schema migration is bundled with a behavior change that breaks for partial rollouts, the cadence still works for the schema but the behavior needs its own feature-flag plan. Cross-tenant data corrections — when the migration also has to repair existing data that was wrong, the backfill becomes a data migration with its own correctness story. Both are out of scope here; named so the student doesn't confuse "the cadence shipped" with "the change is safe."
- **The senior framing.** The cadence is the SaaS-pattern-#11 contract: every schema change that crosses the trigger map in 21.4.2 ships in three PRs minimum, with the migration files, the app code, and the backfill all reviewed under the discipline. The cost is real; the cost is also the price of changes that don't break production.
- **Watch-outs:** thinking the alias swap is atomic with the migration (it isn't — the schema commits before all instances of the new code are warm and after some of the old code is still serving requests); skipping dual-write because "the backfill is quick" (a write to the old column between the backfill and the contract deploy leaves a row inconsistent); making the backfill a single transaction (table lock, dead app); dropping the old column in the same PR as the rename (the canonical "I broke prod" pattern from 6.5.2); reaching for the cadence on a change that doesn't need it (21.4.2 owns the trigger map); the cadence makes every step reversible only if each PR is small enough that `git revert` is sane — bundling the dual-write with three other feature changes erases the reversibility.

## Lesson 21.4.2 — When the cadence earns its weight: the trigger map

Topics to cover:

- **The senior question.** The cadence costs three PRs and weeks of calendar time. Most schema changes don't need it — adding a nullable column, adding a new table, adding an index concurrently, all ship in one deploy and one PR. The senior diff is the call: does this change cross the trigger, or not? The lesson lands the trigger map so the student can read a schema diff and answer in thirty seconds.
- **The one-deploy list — additive and unread.** Ship in a single PR/deploy with no special choreography:
  - **New table.** Old code can't read what doesn't exist in its compiled queries; no read collision.
  - **New nullable column with no application read.** The schema diff is `ADD COLUMN`; old code is unaware; new code writes when ready.
  - **New nullable column the new code reads.** Old code unaware; new code reads `null` for historical rows and handles it (often via `coalesce` or a default in the helper).
  - **New index, `CONCURRENTLY`.** Reads and writes both succeed during the build; no app-code coordination needed. The `CONCURRENTLY` hand-edit and statement-breakpoint dance from 6.5.2 is required.
  - **New unique constraint via `CONCURRENTLY` index then constraint promotion.** The two-step that avoids the locking constraint creation; still one PR if the data already satisfies the constraint.
  - **New CHECK constraint with `NOT VALID` then `VALIDATE CONSTRAINT`.** Two statements, one PR; the second statement scans without blocking writes.
  - **Adding a foreign key with `NOT VALID` then `VALIDATE`.** Same shape — add the FK in a non-validating form, validate in a second statement that scans without long locks.
  - **New enum value (`ALTER TYPE ... ADD VALUE`).** Postgres 12+ allows it outside a transaction; one PR, plus the statement-breakpoint marker (6.5.2).
- **The three-deploy list — the cadence applies.**
  - **Column rename.** The canonical case. Old code reads `customer_name`; new code reads `customer_full_name`; the deploy window has both fleets live. Cadence: expand with the new column, migrate with dual-write and backfill, contract by dropping the old column. Three PRs.
  - **Column type change that rewrites the table.** `varchar(50)` → `text` is a metadata change in Postgres (free). `int` → `bigint` rewrites the table and holds `ACCESS EXCLUSIVE` (lock the whole table for the rewrite). `text` → `int` may fail per-row. The cadence applies for anything that rewrites: add a new column with the new type, dual-write, backfill, switch reads, drop the old column. Three PRs.
  - **Dropping a column the running code still reads.** Even if the new code stops reading it, the old code does. Cadence collapses to two deploys: deploy 1 ships code that no longer reads the column; deploy 2 drops the column. The expand step is empty (no new shape); the migrate step is "remove the read"; the contract step is the drop. Often called "the read-then-drop" two-step.
  - **Adding `NOT NULL` to an existing column.** The naive `ALTER COLUMN ... SET NOT NULL` fails if any row is null, and locks the table while scanning. The cadence: deploy 1 ensures application code writes the column on every insert (no more nulls created); backfill nulls to a sensible value; deploy 2 adds a `NOT NULL` `CHECK ... NOT VALID` constraint, then `VALIDATE` in a separate transaction; deploy 3 promotes to `SET NOT NULL` (now fast because the constraint already proves no nulls). Three PRs.
  - **Adding a foreign-key column that becomes required.** The 21.5 project's case. Expand: add `customer_id uuid` nullable with the FK constraint as `NOT VALID`. Migrate: dual-write `customer_id` alongside the existing `customer_name`, backfill, validate the FK. Contract: drop `customer_name`, promote `customer_id` to `NOT NULL`. Three PRs. This is the exact shape Chapter 21.5 walks through.
  - **Changing the primary key.** The most expensive class; the cadence works in principle but in practice often requires a maintenance window or a read-replica swap. Out of scope for the course; named for recognition.
- **The middle list — sometimes cadence, sometimes not.**
  - **Renaming a table.** Postgres `ALTER TABLE RENAME TO` is metadata-only and fast, but the app-code window has the same problem as a column rename. Cadence applies if the table has live writes; for a write-rare lookup table, a maintenance-window cutover may be cheaper than three PRs.
  - **Adding a default to an existing column.** Postgres 11+: `ALTER TABLE ... ALTER COLUMN ... SET DEFAULT` is metadata-only — historical rows keep their existing value, future inserts get the default. One PR. The senior reflex: confirm the Postgres version supports the fast path; older Postgres rewrites the table (the course's Neon is current, so the fast path is the default).
  - **Removing a default.** Metadata only, one PR.
  - **Tightening a `CHECK` constraint.** If the new constraint matches all existing rows, `NOT VALID` + `VALIDATE` ships in one PR. If existing rows fail it, the cadence: deploy 1 backfills the violating rows, deploy 2 adds the constraint.
- **The decision tree the student internalizes.** Three questions in order:
  - (1) **Is the change additive only — does old code keep working as-is?** If yes, one PR. If no, continue.
  - (2) **Does any one statement hold `ACCESS EXCLUSIVE` for long?** Index without `CONCURRENTLY`, type rewrite, validating constraint without `NOT VALID`. If yes, redesign the migration to avoid the long lock — usually via the `NOT VALID`/`CONCURRENTLY` pattern — then re-ask (1).
  - (3) **Does the running app code read or write a shape that's about to disappear?** If yes, three deploys (expand-migrate-contract). If no, you're done in one.
- **The "what kind of trigger map mistake costs you" framing.** False positive — reaching for the cadence on an additive change — wastes a week of calendar time but no production damage. False negative — shipping a column drop or rename in one deploy — is an outage. The asymmetry biases the senior reflex toward the cadence when uncertain.
- **Drizzle Kit's role in the trigger map.** The generated SQL is the diff; reading it answers question (1) and (2). When Drizzle Kit emits a `DROP COLUMN` next to an `ADD COLUMN` with similar shape, it's the disambiguation prompt (6.5.1's true-rename vs. remove-and-add). The reviewer's job is to recognize the diff and route the PR through the cadence accordingly.
- **The `NOT VALID` + `VALIDATE` two-step in detail.** Postgres lets you add a CHECK or FK constraint as `NOT VALID`, which only applies to new rows — instant, no table scan, no long lock. A subsequent `ALTER TABLE ... VALIDATE CONSTRAINT` scans existing rows but takes only a `SHARE UPDATE EXCLUSIVE` lock (concurrent reads and writes proceed). The pattern is one PR for non-strictening changes, the cadence for strictening ones.
- **Tooling that automates the cadence — pgroll, named and dropped.** Open-source tool from Xata that runs both schema versions side-by-side and handles the cadence automatically by maintaining shadow views. Reach if a team is doing many cadence-class migrations a quarter and wants the choreography off their plate. Course default is hand-rolled cadence with Drizzle Kit — the discipline is the lesson, and one tool more is one tool too many for the course's stack. Named so the student knows the option exists.
- **Watch-outs:** treating every PR as cadence-worthy (cost without benefit); shipping a "small" rename in one deploy because "no one will hit it during the window" (the cutover is exactly when staging tools, ops scripts, and cron jobs hit it); confusing `NOT NULL` with `NOT VALID` (different keywords, different semantics — `NOT VALID` is the validate-later flag, `SET NOT NULL` is the constraint promotion); skipping the `CONCURRENTLY` on a 10M-row index because "the trigger map said one PR is enough" (the trigger map and the locking checklist are independent — both must pass); reading the trigger map as a rule book instead of a decision aid (the senior call is the read of the specific change, not pattern-matching on the migration name).

What this lesson does not cover:

- The cadence itself — Lesson 21.4.1.
- Testing migrations on a preview branch — Lesson 21.4.3.
- `CREATE INDEX CONCURRENTLY` and statement-breakpoints in detail — Chapter 6.5.2.
- Postgres locking model at depth — out of scope; the lesson treats lock classes as named primitives.
- pgroll setup — out of scope; named once.
- Foreign-key reference design (cascade behaviors, indexed FKs) — Chapter 6.2 and 6.4.

Estimated student time: 35 to 45 minutes. Decision; the three-question tree and the trigger map are the load-bearing artifacts.

## Lesson 21.4.3 — Rehearsing the migration on a Neon preview branch

Topics to cover:

- **The senior question.** Every step in the cadence is a hypothesis: the migration applies cleanly against production-shaped data, the app code reads and writes correctly while both schemas coexist, the backfill completes in a reasonable time without locking the table. The hypothesis gets tested before the PR merges, against a copy-on-write Neon branch of production (21.3.5) — the most production-like environment that's cheap to break.
- **The Neon preview branch as the rehearsal stage.** From 21.3.5, every PR's preview deployment gets a Neon branch off `main` (production's branch). It carries production-shaped data — same row counts, same distribution, same indexes — and it's isolated from production by Neon's copy-on-write. The branch is the rehearsal stage that costs nothing and breaks nothing.
- **What the preview deployment exercises automatically.** The build command from 21.3.5 runs `pnpm db:migrate` against the preview branch before `next build`. So the moment the PR's commit lands, the migration runs against production-shaped data. A failure here — bad SQL, type mismatch, lock timeout — fails the build, posts to the PR, and never reaches production. The first verification is automatic.
- **What the developer verifies manually — the rehearsal checklist.** Four checks per cadence step.
  - **(1) The migration applied without error.** Vercel build logs show the `db:migrate` step succeeded; the new migration appears in the branch's `__drizzle_migrations` table. Studio confirms the schema diff matches the PR's SQL.
  - **(2) The migration finished in reasonable time.** A migration that takes ten minutes against a production-shaped branch will take longer against production proper (busier; smaller window for locks). If the preview-branch run is slow, the migration plan needs revisiting — usually `CONCURRENTLY` was missed, or a backfill should have batched.
  - **(3) The app still works against the new schema.** Open the preview URL, walk the critical paths: list pages render, mutations succeed, dashboard counts match. The preview branch is fully populated; if a query breaks, it breaks visibly.
  - **(4) The old shape, where it still exists, still works.** For the expand step, query the old column directly via Studio — it's still readable, still writable. For the migrate step, hit the production code paths via the preview URL — every mutation writes to both columns; every read returns the new value if present, the old value if not.
- **The dual-write verification — direct SQL.** Open Studio against the preview branch after exercising a mutation through the preview URL: the modified row has both `customer_name` (old) and `customer_id` (new) populated. The dual-write is doing its job. If only one is populated, the dual-write code is wrong — fix and re-push.
- **The backfill rehearsal.** The migrate step's backfill script runs against the preview branch. Time it (`time pnpm tsx scripts/backfill_customer_ids.ts`). Extrapolate against production row counts — a 200K-row backfill on the preview branch in 90 seconds is roughly 900 seconds on a 2M-row production table (linear approximation; reality is sub-linear thanks to OS cache). If the projection exceeds an acceptable window, batch smaller or move the backfill to Trigger.dev (Unit 13).
- **The lock-contention rehearsal — `auto_explain` and `pg_locks`.** For migrations on tables with high write volume, the preview branch can be loaded with synthetic write traffic (a small script hammering the relevant mutation) while the migration runs. If the migration holds a long lock, the write traffic stalls; Neon's metrics show the spike. The rehearsal catches the "this looked fine in dev but stalled production" failure mode. Reach for this verification when the table has write traffic measured in writes-per-second, not per-minute.
- **The failure modes the cadence is designed to catch — and what to look for.**
  - **Expand step's failure mode: the new column's FK constraint rejects valid backfill values.** The rehearsal catches it because the backfill runs against production-shaped data; FK violations surface immediately. Fix in the migrate PR before merging.
  - **Migrate step's failure mode: dual-write skips a code path.** The rehearsal catches it via direct Studio inspection — a row written through one path has both columns populated; a row written through a missed path has only the old. Fix the missed mutation site, push, re-verify.
  - **Migrate step's failure mode: backfill takes too long.** The rehearsal times it; the senior reach is to batch smaller or move to Trigger.dev. Better to discover at the preview stage than during the production backfill.
  - **Contract step's failure mode: something still reads the old column.** The rehearsal catches it because dropping the column breaks the read; the preview URL surfaces the broken page. Search the codebase for the old column name — Drizzle's type system catches most of these at build time (the `customer_name` field no longer exists on the type), but raw SQL or external integrations slip through.
  - **Cross-step failure mode: a CI job (test database, seed script) still references the old column.** The CI build fails; the PR is blocked until tests are updated. The unit's CI gate (21.2) is the second line of defense after the type system.
- **The CI gate's role.** Type-check and tests run against the codebase's expectations; the migration runs against the preview database. Both must pass before merge. CI typecheck catches every dropped-column read that goes through Drizzle's typed query builder. Integration tests against the preview branch catch the rest.
- **The data-integrity diff — what changed in the rehearsal.** After the migration runs against the preview, a small SQL audit: `SELECT count(*) FROM invoices WHERE customer_id IS NULL` (should be zero after backfill); `SELECT count(*) FROM invoices WHERE customer_name IS DISTINCT FROM (SELECT name FROM customers WHERE id = customer_id)` (should be zero if the backfill maps correctly). The diff catches "the backfill populated the column but with wrong values" — silent corruption that no schema check finds.
- **The "promote the rehearsal to production" handoff.** Once the rehearsal checklist passes: merge the PR. Vercel re-runs `db:migrate` against production's `main` branch as part of the production build. Because the preview-branch run already validated the migration, the production run is a repeat — same SQL, same migration runner, just against the production data this time. Confidence comes from the rehearsal having been a fair copy.
- **The "what the rehearsal can't catch" list.** Concurrency at production scale — the preview branch sees zero real traffic during the rehearsal, so a migration that's slow under write contention may look fast. Anomalies in production data the preview branch sampled past — branched data is a snapshot; data added since the branch is missing. Long-running production transactions that block the migration — those only exist in production. The rehearsal is necessary, not sufficient; the senior reach for the high-stakes migration is a low-traffic deploy window plus a watchful eye on Sentry and Neon metrics during the run.
- **Tooling reach for sensitive migrations.** For migrations on the highest-traffic tables, the rehearsal cycle gets one more step: take a fresh Neon branch off production at the moment the PR is ready, run the cadence against it under synthetic load, observe. The Native Vercel Integration creates the branch on PR open, but a manual `neonctl branches create --parent main` produces a fresh one at any moment. Reach when the rehearsal's first run revealed a question that the data snapshot answers.
- **Watch-outs:** trusting the build's automatic migration step as the only verification (it confirms SQL applies; it doesn't confirm correctness); preview branch missing the migration runner because the build command wasn't overridden in 21.3.5 (without `pnpm db:migrate &&`, the preview has the new code against the old schema — exactly the broken state the cadence prevents); rehearsing only the migrate step (each of expand/migrate/contract has its own failure modes and gets its own rehearsal); skipping the dual-write inspection in Studio (the type system doesn't enforce that the dual-write code path is reached for every mutation site); the preview branch's row counts not actually matching production (Neon copy-on-write is full data, but a recent prod backfill on `main` may have populated a column the branch was taken before — `neonctl branches create` from a known-good moment is the fix); reusing a stale preview branch after `main` has advanced significantly (close and re-open the PR to get a fresh branch, or `neonctl branches reset`); assuming the production deploy's migration "can't fail because the preview's worked" (it almost always works, but the canary-watch-Sentry discipline is still the rule).

- The cadence itself — Lesson 21.4.1.
- The trigger map — Lesson 21.4.2.
- The Neon preview-branch mechanic — Chapter 21.3.5.
- The CI gate (typecheck, lint, test, build) — Chapter 21.2.

## Lesson 21.4.4 — Chapter quiz

Top 10 topics to quiz:

- The deploy-and-migration race — the alias swap is atomic but the function fleet warms over seconds and both code versions hit the database during the window; the schema must satisfy both.
- The three steps of the cadence — expand (add the new shape alongside the old, nullable), migrate (dual-write from app code, backfill historical rows, dual-read with fall-through), contract (drop the old shape, drop the dual-write, drop the fall-through); each ships in its own PR.
- Dual-write as application-layer choreography — server actions write both columns in the same statement; it's not a database trick, it's a code pattern that survives until the contract deploy deletes it.
- The backfill discipline — bounded batches (1K–10K rows), idempotent via `WHERE new_column IS NULL`, run from a script for small datasets, moved to Trigger.dev for million-row jobs.
- Forward-only migrations and the rollback story — `git revert` rolls back the app code; the next forward-fix migration rolls back the schema; the contract step's column drop is the only irreversible move (rollback past contract is a data-recovery problem, not a deploy problem).
- The trigger map's one-deploy list — new table, new nullable column, new index `CONCURRENTLY`, new constraint with `NOT VALID` + `VALIDATE`, new enum value; additive-only changes that don't break old code.
- The trigger map's three-deploy list — column rename, table-rewriting type change, dropping a column the running code still reads, adding `NOT NULL` to an existing column, adding a required FK column.
- The decision tree — (1) is the change additive-only, (2) does any one statement hold a long lock, (3) does running code read or write a shape about to disappear; three-deploy cadence only when (3) is yes.
- The Neon preview-branch rehearsal — Vercel's build command runs `db:migrate` against the per-PR branch; the developer verifies the migration applies, completes in reasonable time, app works against the new schema, old shape still works where it should; failure modes surface before merge.
- The failure modes the cadence is designed to catch — FK rejection in the backfill, dual-write skipping a mutation site, backfill taking too long, contract step dropping a column still read by something; what the rehearsal can't catch (production-scale concurrency, post-branch data anomalies).
