# Chapter 11.1 ‚Äî URL-state list views (SaaS pattern #7)

## Chapter framing

Chapter 11.1 is where the URL-state vocabulary from Chapter 5.5.4 and the cursor-pagination shape from Chapter 6.3.6 cash in as the SaaS list view ‚Äî the screen every internal tool, dashboard, and admin app eventually becomes. The student arrives knowing `params` vs. `searchParams`, knowing how to read them in a Server Component, knowing how to write them with `router.replace`. This chapter lands the full pattern: filter, sort, search, and pagination all living in the URL together; the production layer (`nuqs`) that makes typed parsers and the read/write pair tolerable past two parameters; the search-input pattern that doesn't fire a query on every keystroke; the cursor-vs-offset call; and the share-and-refresh contract every senior expects to hold. The output is the mental model and the syntactic vocabulary the project chapter (11.3) wires into a real CRUD surface.

Threads that run through every lesson: the URL is the source of truth for any view state a user expects to survive a refresh, a share, or a back button ‚Äî and component state for everything else; the server reads and validates `searchParams` at the page boundary, the client writes via `router.replace` (not `push`) for in-page state changes so the back button leaves the page rather than undoing the last chip; Zod parses every URL read because `searchParams` is user-controlled and `string | string[] | undefined` is the raw shape; `nuqs` is the production reach past two or three parameters ‚Äî typed parsers, shared cache between server and client, `useQueryState`/`useQueryStates` on the client; `useDeferredValue` (or `useTransition` around the navigation) is the React 19 reach for keeping search inputs responsive without a `setTimeout`-based debounce, with optional `nuqs` `throttleMs` on the URL write itself; cursors beat offset by default ‚Äî stable under inserts, indexable, opaque base64 of the sort key plus tiebreaker ‚Äî and offset earns its weight only when the result set is small and known-bounded; defaults are implicit, not in the URL, to keep links short and bookmarks readable; the share-and-refresh test is the canonical verification ‚Äî open the URL in a new tab, see the same view. The chapter ships four teaching lessons plus a quiz, ordered by dependency: the pattern and the URL-vs-state decision, then filter and sort, then search, then pagination, then the quiz.

## Lesson 11.1.1 ‚Äî The list-view anatomy

Teaches the four-pillar SaaS list pattern (filter, sort, search, paginate) as URL state with a server-reads-and-writes-client division, `router.replace` over `push`, the share-and-refresh contract, and the `nuqs` `searchParamsCache` setup that becomes the chapter's reference shape.

Topics to cover:

- **The senior question.** A SaaS list screen needs to be filterable, sortable, searchable, paginated, and shareable. Where does each piece of state live, what crosses the server/client boundary, and what does the URL contract guarantee to the user? The lesson lands the full pattern as one mental model before any single mechanic gets full treatment.
- **The four pillars of a SaaS list view.** Filter (what subset is shown), sort (in what order), search (free-text within the subset), pagination (which page of the result). The lesson names all four as URL state by default and treats them as one coherent shape, not four independent decisions.
- **The URL-vs-state decision rule, restated for list views.** State that should survive a refresh, be shareable, or appear in browser history belongs in the URL. Transient interaction state (the open/closed state of a filter dropdown, hover, focus, in-progress text the user is typing but hasn't committed) belongs in component state. The reflex question for each piece of view state: would the user expect this back if they refreshed?
- **The share-and-refresh contract ‚Äî what the URL guarantees.** Open the URL in a new tab: the user sees the same filtered, sorted, paginated view. Refresh: same view. Copy and paste into Slack: coworker sees the same view (subject to auth). Back button: returns to the previous filter/sort/page combination. The lesson names this contract as the acceptance test for any URL-state implementation ‚Äî if a click changes the result but not the URL, the contract is broken.
- **The read-on-server, write-on-client division ‚Äî list-view edition.** The page is a Server Component; it reads `props.searchParams`, validates with Zod, queries the database, renders the table. Client Components handle the inputs ‚Äî the filter dropdowns, the sort header, the search input, the page-next button ‚Äî and call `router.replace` to update the URL. Server re-renders, client re-hydrates the table. No `useEffect` to sync state to URL, no client-side data fetch, no waterfall.
- **`router.replace` vs. `router.push` ‚Äî the senior call for list state.** Filter/sort/search/page changes use `replace` so the back button leaves the list page rather than undoing the last chip. `push` is reserved for genuine navigation (clicking a row, opening a detail page). Default: `replace` with `{ scroll: false }` for in-page state changes.
- **The page-shape anatomy.** A worked composition: a Server Component page that parses `searchParams`, fetches the list, renders a `<Filters />` and a `<Sort />` and a `<Search />` and a `<Table />` and a `<Pagination />`. The first four are Client Components that take the current parsed value as a prop (server-derived) and call `router.replace` on change. The table is server-rendered with the data. The architecture is canonical for the rest of the chapter.
- **`nuqs` as the production layer ‚Äî when it earns its weight.** Hand-rolled `URLSearchParams.set(...)` plus Zod parse plus `router.replace` works for one parameter, gets repetitive for two, and becomes a tax past three. `nuqs` provides typed parsers (`parseAsString`, `parseAsInteger`, `parseAsStringEnum`, `parseAsArrayOf`), defaults, the `useQueryState`/`useQueryStates` Client hooks, and `createSearchParamsCache` for the server side ‚Äî one source of truth for both sides of the boundary. The threshold: any list view with filter + sort + search + pagination crosses it. The lesson names `nuqs` as the canonical pick and uses it for the rest of the chapter.
- **The `nuqs` setup ‚Äî `NuqsAdapter` once, parsers shared.** Wrap the root layout in `<NuqsAdapter>`. Define parsers in a `searchParams.ts` module next to the page ‚Äî these are reused by `createSearchParamsCache` on the server and `useQueryStates` on the client, so the schema is one definition. Stamp this pattern; subsequent lessons reuse it.
- **Default values ‚Äî implicit in the URL, explicit in the parser.** A parser declares `.withDefault('-createdAt')` for sort; the URL omits the parameter when it matches the default. `nuqs` strips defaults from the URL automatically. The user-visible win: short, readable URLs that only reflect what's different from the baseline. The senior reflex: pick defaults that match the most common view so the empty URL is the home state.
- **Validation as defense at the boundary.** Even with `nuqs` parsers, the senior check: a `parseAsStringEnum(['draft', 'paid', 'overdue'])` rejects garbage in the URL and falls back to the default. Without validation, a hand-typed `?status=DROP%20TABLE` flows into the database query. `nuqs` parsers are Zod-shaped ‚Äî they validate-or-default by design. Reach for plain Zod only when the parameter is structured beyond what `nuqs` provides.
- **The server-side `searchParamsCache`.** `createSearchParamsCache({ status: parseAsStringEnum(...), sort: parseAsStringEnum(...).withDefault('-createdAt'), q: parseAsString.withDefault(''), cursor: parseAsString })` produces a typed parser. The page calls `await searchParamsCache.parse(props.searchParams)` once at the top, gets a typed object, passes the relevant slices to the query function and to the Client Components for their controlled-input defaults. Shared cache means the client `useQueryStates(parsers)` and the server `searchParamsCache.parse(...)` agree on shape by construction.
- **What URL state is not ‚Äî restated for list views.** Not the place for secrets, large blobs, or the contents of the row being edited (that's form state). Not the place for the in-progress text of a debounced search (lesson 11.1.3 covers the typed-vs-committed split). Not the place for state the user can't reasonably understand from the URL ‚Äî opaque blobs other than cursors are a smell.
- **Diagram ‚Äî the data flow.** Client input change to `router.replace` to URL change to Server Component re-render to new query result to streamed table to client re-hydration. The diagram pins the round-trip and the absence of client-side fetching.
- **Worked example ‚Äî the filtered, sorted, searchable, paginated invoice list, end-to-end.** The page, the `searchParamsCache`, the `<Filters />` Client Component, the database query call. The example is the chapter's reference shape; later lessons drill into each piece.
- **Watch-outs:** every keystroke writing to the URL produces history-entry spam ‚Äî use `replace` and a deferred-value pattern for search (lesson 11.1.3); writing to the URL while a Server Action is in flight can race the navigation ‚Äî finish the action, then read the new URL on next render; `router.push` instead of `replace` for chip changes creates 50 history entries on a noisy session ‚Äî pick `replace`; defaults in the URL grow the link unnecessarily ‚Äî let `nuqs` strip them; mixing hand-rolled `URLSearchParams` writes with `nuqs`-managed parameters works but loses type safety on the manual side ‚Äî commit to one approach per page; scroll resetting to top on every filter change is the wrong UX for a long list ‚Äî `{ scroll: false }` on the navigation; sharing a URL that points to a soft-deleted or restricted row leaks information via the 404 timing ‚Äî the auth check happens at the route, not the URL, and the lesson surfaces this is handled in Chapter 9 / 10.

What this lesson does not cover:

- The `useRouter`/`useSearchParams` hook mechanics ‚Äî Chapter 5.5.5.
- The `searchParams` server read mechanics ‚Äî Chapter 5.5.4.
- Cursor pagination at the database layer ‚Äî Chapter 6.3.6.
- The soft-delete visibility toggle in the URL ‚Äî Chapter 11.2.
- The Server Action that mutates and revalidates the list ‚Äî Chapter 7.2.
- Auth and tenancy scoping on the list query ‚Äî Chapters 9 and 10.

Estimated student time: 40 to 50 minutes. The chapter's center of gravity ‚Äî every later lesson builds on this shape.

## Lesson 11.1.2 ‚Äî Filter shapes and sort encoding

Teaches the four filter shapes (single-value enum, multi-value array, range as two parameters, boolean toggle with omitted default), the `-key` sort string with its enum-constrained indexed columns, the cursor-reset invariant on filter or sort change, and active-filter chips with a "clear all" affordance.

Topics to cover:

- **The senior question.** A list view needs a status filter (single value from an enum), a tag filter (multi-select from a set), and a sort selector (one of a small set of sort keys with direction). How are each shaped in the URL, parsed on the server, written from the client, and combined without trampling each other? The lesson lands the three filter shapes (enum, multi, range) and the sort shape, plus the rule that writing one parameter never resets the others.
- **Single-value enum filter ‚Äî the canonical shape.** `?status=paid` reads as `parseAsStringEnum(['draft', 'paid', 'overdue']).withDefault(null)` ‚Äî null when not in the URL, typed when present. The Client `<StatusFilter />` is a controlled dropdown that calls `setStatus('paid')` (the `nuqs` setter) which writes `?status=paid` and triggers a server re-render. The shape is the model for every single-value filter.
- **Multi-value filter ‚Äî `parseAsArrayOf` and the encoding.** `?tags=billing,urgent` reads as `parseAsArrayOf(parseAsString).withDefault([])`. `nuqs` defaults to comma-separated; the alternative repeated-key form (`?tag=billing&tag=urgent`) is supported via `parseAsArrayOf(...).withOptions({ separator: ... })`. The senior call: comma-separated is shorter for the URL and more readable, repeated-key is more web-platform-orthodox ‚Äî pick comma unless interoperating with an external system that expects the other.
- **Range filter ‚Äî date and number ranges.** A "created between" filter uses two parameters (`?createdFrom=2026-01-01&createdTo=2026-03-31`) parsed with `parseAsIsoDate.withDefault(null)`. A single parameter encoding both bounds (a JSON blob or a custom format) is brittle and harder to read ‚Äî two parameters is the senior reach. The query function takes the parsed values and applies `gte`/`lte` predicates.
- **Boolean toggle ‚Äî `parseAsBoolean` and the implicit-default rule.** A "show archived" toggle reads as `parseAsBoolean.withDefault(false)`. The URL only carries the parameter when it's true (`?showArchived=true`); the default is omitted. The pattern is the model for the soft-delete visibility toggle Chapter 11.2 builds on.
- **The sort parameter ‚Äî single string with leading `-` for direction.** `?sort=-createdAt` or `?sort=total` reads as `parseAsStringEnum(['createdAt', '-createdAt', 'total', '-total', 'customer', '-customer']).withDefault('-createdAt')`. The leading `-` for descending is a convention that scales: the query function splits the string, builds the Drizzle `orderBy` clause. Alternative: separate `sortKey` and `sortDir` parameters ‚Äî two params, more bytes, no benefit. The course picks the single-string convention.
- **Why sort is enum, not free string.** The sort value flows into the query's `orderBy` clause. Accepting a free string is a SQL-shape injection vector even if the value is parameterized ‚Äî the planner can't index every possible column, and unindexed sorts blow up at scale. The enum constrains to the columns with composite indexes (6.4.1) and pairs with cursor encoding (11.1.4).
- **The reset rule ‚Äî sort change resets pagination, filter change resets pagination.** Changing the sort with a `cursor` from the old sort in the URL produces undefined results (the cursor decodes against a different sort key). The Client setter, on sort change, explicitly clears the cursor: `setQueryStates({ sort: '-total', cursor: null })`. Same for filter changes that shrink the result set ‚Äî the cursor from the broader filter is no longer valid. The lesson names this as the canonical bug source and the structural fix.
- **The "writing one without trampling the others" pattern.** `nuqs`'s `useQueryStates(parsers)` returns a single setter that merges ‚Äî `setState({ status: 'paid' })` updates only `status`, leaves `sort`, `cursor`, `q` unchanged. The hand-rolled version requires constructing a new `URLSearchParams` from the current one and updating selectively ‚Äî the helper is one line, `nuqs` does it for free.
- **Active-filter chips ‚Äî the affordance pattern.** Above the table, render a row of chips reflecting the active filters: "Status: Paid (x)", "Tag: Billing (x)". Each chip's "x" calls `setState({ status: null })` to clear that filter. The user sees what's filtered and undoes any piece independently. The chip list reads its props from the parsed server-side state, not from `useQueryStates` ‚Äî pure-prop is simpler and renders without hydration.
- **The "clear all" reset.** A "Clear filters" button calls `setState({ status: null, tags: [], q: '', cursor: null })`, which `nuqs` strips from the URL (defaults are implicit), returning the user to the canonical empty-URL list view. The senior reach: never `router.push('/invoices')` to clear ‚Äî that pushes a history entry and triggers a full segment re-render. Use the setter.
- **Sort UI ‚Äî sortable column headers vs. a dropdown.** Two canonical affordances: clickable column headers with up/down chevrons (cycle through asc/desc/none) and a dropdown listing the sort options. Headers are denser; dropdown is more discoverable for non-table list views (card grids). Both call the same setter. The lesson names the choice as UX-driven.
- **What about server-side default sort?** The default sort (`-createdAt`) lives in the parser, so the server reads the same default the URL would. The query function gets the resolved sort; the database query runs with the right composite index (6.4.1). The senior anchor: defaults are one source of truth ‚Äî the parser.
- **Composite indexes pair with sort choices ‚Äî restated from 6.4.1.** Every sort option in the enum corresponds to a composite index `(orgId, sortKey, id)` so the query plan stays an index scan. Adding a sort option without the matching index is the easy regression. The senior call: the sort-enum list is gated by the indexes that exist; new sort options ship with their migration.
- **Worked example ‚Äî the filter and sort row.** A small Client Component renders a status dropdown, a tag multi-select, and a sort dropdown, all driven by `useQueryStates(parsers)`. On change, the setter writes; the server re-renders. The example is the next building block on top of 11.1.1's anatomy.
- **Watch-outs:** changing sort without clearing the cursor produces nonsense pages ‚Äî bake the clear into the setter; multi-select with comma-separated values trips when the values themselves contain commas ‚Äî pick a separator that can't appear in the value or escape; boolean parameters left in the URL at their default value clutter the URL ‚Äî strip via `withDefault(false)` and let `nuqs` omit; sort enums that grow without matching indexes is a perf bug ‚Äî review the index list with the enum list; date ranges parsed without timezone normalization confuse cross-region users ‚Äî handle as UTC at the URL boundary, format in the user's timezone in the UI (Chapter 18 owns the timezone discipline); filter chips that read from `useQueryStates` on the client when the parsed server state is already available re-render unnecessarily ‚Äî pass as prop; `null` vs. `undefined` in `nuqs` setters has meaning ‚Äî `null` clears the parameter, `undefined` leaves it untouched.

What this lesson does not cover:

- The search parameter (`q`) and the debounce/defer pattern ‚Äî Lesson 11.1.3.
- The pagination cursor ‚Äî Lesson 11.1.4.
- Soft-delete visibility as a filter (`showArchived`) wired into the base query ‚Äî Chapter 11.2.
- The Drizzle `where`/`orderBy` mechanics ‚Äî Chapter 6.3.
- Multi-tenancy scoping (`organizationId` filter that's always applied) ‚Äî Chapter 10.
- Saved views and named filter presets ‚Äî out of scope for this chapter.

Estimated student time: 45 to 55 minutes. The filter/sort row is the most-frequent client interaction on the list view.

## Lesson 11.1.3 ‚Äî Typed input, committed URL

Teaches the typed-vs-committed split for search inputs, React 19's `useDeferredValue` plus `useTransition` for input rhythm, `nuqs` `throttleMs` for URL-write rhythm, the `replace`-only history policy, and the `isPending` loading affordance.

Topics to cover:

- **The senior question.** A search input lets the user type a query; results filter as they type. Every keystroke should not run a database query, should not write a URL, should not flood the back-button history. What's the right rhythm ‚Äî debounce, throttle, or React 19's `useDeferredValue` ‚Äî and which of "typed" and "committed" lives in the URL? The lesson lands the typed-vs-committed split and the React 19 native pattern.
- **The two states ‚Äî typed and committed.** "Typed" is the current input value (what's in the box right now); "committed" is the value the server is querying against (what's in the URL). They diverge during typing; they converge after a brief settling period. The decision: typed lives in component state, committed lives in the URL. Reading the URL on the server queries against the committed value; the input itself is a controlled component whose `value` is the typed state.
- **The React 19 pattern ‚Äî `useDeferredValue` over the input.** `const [typed, setTyped] = useState(propFromUrl)` for the input; `const deferred = useDeferredValue(typed)` produces a value that lags during heavy renders; an effect (or a direct write in the change handler with `startTransition`) syncs `deferred` to the URL via `setQuery(deferred)`. The win: React decides the rhythm, not a fixed `setTimeout`. Adaptive on slow devices, immediate on fast ones.
- **`useTransition` around the navigation ‚Äî keeping the input responsive.** `const [isPending, startTransition] = useTransition()`; `startTransition(() => setQuery(typed))` marks the URL write and resulting server re-render as a transition ‚Äî React keeps the input responsive while the heavier work runs. The `isPending` boolean drives a subtle loading indicator on the table. The senior reach: `useTransition` around any navigation that triggers a server-side render of expensive content.
- **The classic debounce ‚Äî when it still earns its weight.** A 250ms `setTimeout`-based debounce on the URL write is the older pattern; `useDeferredValue` covers most of its territory in 2026 but a hand-rolled debounce is still right when the threshold matters externally (rate-limited search API, expensive analytics fire). The lesson names the trade-off: prefer `useDeferredValue` for UI responsiveness, reach for explicit debounce when the cost is external and quantifiable.
- **`nuqs` throttle on the write ‚Äî the second layer of rhythm.** `useQueryState('q', parseAsString.withDefault('')).withOptions({ throttleMs: 200 })` rate-limits the URL write itself, complementing the React-side deferred value. The combination: the input is always responsive (React handles), the URL writes at most every 200ms (nuqs handles), the history entry count stays sane. Default `throttleMs` of 50ms in `nuqs` is fine for most cases; bump for search.
- **History entry policy ‚Äî `replace` plus `history: 'replace'`.** Every URL write during typing uses `replace`, not `push` ‚Äî otherwise a single search session leaves dozens of back-button states. `nuqs` exposes `history: 'replace'` on the parser options. The senior anchor: only the *first* search write per "session" could conceivably be a `push`, and even that's usually wrong ‚Äî `replace` throughout.
- **The empty-query case ‚Äî omit from URL.** An empty search box should produce an empty URL parameter, not `?q=`. `parseAsString.withDefault('')` plus `nuqs` default-stripping handles this. The shareable URL when nothing is searched matches the home view; the URL only carries `?q=...` when there's something to share.
- **Server-side search shape ‚Äî what the query does.** The server reads `q` from the parsed `searchParams`, passes it to the query function. The Drizzle query applies `ilike` for small datasets, full-text search (6.3.8) for production scale, or a dedicated search index for very large sets ‚Äî the lesson names the spectrum and points to 6.3.8 for the FTS mechanics. The URL-state side is shape-agnostic; what changes is what the query function does with the value.
- **The search-and-pagination interaction ‚Äî cursor reset on `q` change.** The cursor encodes a position in the current ordered, filtered result; changing `q` invalidates that position. The setter for `q` clears the cursor, same shape as the filter/sort reset rule from 11.1.2. The pattern is consistent across every parameter that changes the result set.
- **Loading state during search ‚Äî `isPending` in the input chrome.** The `useTransition` `isPending` boolean drives a subtle spinner inside the input or a fade on the table. The user sees the system is working without the table jumping. The senior anchor: never block the input on the search query ‚Äî typing must always be instant.
- **Accessibility ‚Äî `role="searchbox"`, `aria-controls`, the live-region announcement.** The input gets `role="searchbox"` (or is a native `type="search"`) and `aria-controls` pointing to the results table's id. Screen-reader users get an `aria-live="polite"` region that announces "5 results" when the table updates. The lesson names the contract; Chapter 4.11 owns the accessibility primitives at depth.
- **Worked example ‚Äî the search input wired end-to-end.** A `<SearchInput />` Client Component takes the URL's `q` as a prop for the initial value, holds typed state, uses `useDeferredValue` and `useTransition` to sync to the URL via `nuqs`'s setter. On the server, the page parses `q`, runs the query, renders the table. The example uses 30 lines total and shows every piece.
- **The "URL write on blur" alternative ‚Äî when to reach.** For a slow data source or a single-input search where typing-as-you-go is unwanted, write to the URL only on blur or on Enter. The pattern is a `setQuery(typed)` call inside `onBlur` and `onKeyDown` for Enter. Names a case; doesn't override the default.
- **Watch-outs:** writing the URL on every keystroke creates a history-entry nightmare and triggers a server re-render per character ‚Äî use `replace` and defer; `useDeferredValue` on the URL write itself instead of on the rendered value defeats the point ‚Äî the deferred value drives the URL, the typed value drives the input; an effect that watches typed state and writes the URL without a transition wrapper janks the input on slow devices ‚Äî wrap with `useTransition` or use the `nuqs` shallow option; controlled input value drifting from URL state on prop change (the user pastes a URL, the input must update) ‚Äî sync the input state in `useEffect` watching the URL prop, or use `useQueryState` directly which handles it; the URL's `q` and the typed state can momentarily disagree (the user is typing; the URL hasn't caught up yet) ‚Äî that's correct, not a bug; full-text search ranking surprises the user who expected substring match ‚Äî match the algorithm to the user's mental model; placing the search input in a form that submits to the URL on Enter (the older pattern) is fine but the React 19 deferred-value flow is the 2026 senior pick.

What this lesson does not cover:

- Postgres full-text search mechanics ‚Äî Chapter 6.3.8.
- The N+1 problem if search joins line items per row ‚Äî Chapter 6.4.2.
- Real-time search results with a typeahead/autocomplete dropdown ‚Äî separate UX, out of scope.
- Rate-limited external search APIs (Algolia, Typesense) ‚Äî name as alternatives; out of scope for the course.
- Server Actions that update the list ‚Äî Chapter 7.2.
- The accessibility primitives ‚Äî Chapter 4.11.

Estimated student time: 40 to 50 minutes. The search input is the most user-facing piece of the list view's interaction model.

## Lesson 11.1.4 ‚Äî Cursor by default, offset when small

Teaches the cursor-vs-offset decision for list pagination, the opaque base64 cursor shape with sort key plus tiebreaker, the next-extra-row trick for `hasNext`, cursor versioning against sort, the row-inserted-during-pagination failure mode under offset, and the "position not snapshot" contract for shared URLs.

Topics to cover:

- **The senior question.** A list of invoices has 50,000 rows. The user paginates through it. What's in the URL ‚Äî a page number, an offset, a cursor ‚Äî and what are the consequences for share-and-refresh, for "jump to page N", and for what happens when a row is inserted between page loads? The lesson lands cursor-as-default with offset-as-conditional, the URL encoding for each, and the share-and-refresh contract.
- **The cursor-vs-offset decision, restated from 6.3.6.** Cursor: opaque base64 of the last row's sort key plus tiebreaker; stable under inserts; indexable; can't jump to page N. Offset: integer "skip this many rows"; supports random page access; degrades linearly with depth; unstable under inserts (rows shift). The senior default: cursor. The exception: small, known-bounded result sets where the user expects "go to page 5" UX and the offset stays under a few hundred ‚Äî admin tables, settings lists. Production list views default to cursor.
- **The cursor URL shape ‚Äî opaque base64.** `?cursor=eyJpZCI6NDIsImNyZWF0ZWRBdCI6IjIwMjYtMDQtMTUifQ==` ‚Äî the user sees a blob, the server decodes it. The senior choice: keep it opaque so users don't construct cursors by hand and so the encoding can evolve without breaking pasted links (version the cursor's internal shape). The decoding lives in the parse helper; if decode fails (malformed cursor, version mismatch), the page falls back to the first page silently.
- **The offset URL shape ‚Äî `?page=5` or `?offset=80`.** Page-based (`?page=5` with implicit `pageSize=20`) reads naturally and the user can edit it. Offset-based (`?offset=80`) is honest about what's happening. The course picks `?page=N` for offset cases ‚Äî it's the user-facing pattern Slack, Linear, and Stripe use for small lists. `pageSize` lives in the parser, not the URL, unless the user can pick.
- **The cursor encoding ‚Äî what goes in the blob.** The last row's sort key (the indexed column the query orders by) plus the tiebreaker (`id` is the universal one). For a sort by `-createdAt` with `id` tiebreaker: `{ createdAt: '2026-04-15', id: 42 }`. The next-page query is `where (createdAt, id) < (cursor.createdAt, cursor.id) order by createdAt desc, id desc limit pageSize` ‚Äî composite-index-driven, fast at any depth.
- **Bidirectional pagination ‚Äî `before` and `after` cursors.** The blob carries a direction marker, or two parameters (`?after=...` or `?before=...`) decide the direction. The course picks two parameters for clarity: `?cursor=...` is the next-page reach; `?before=...` is the previous-page reach when a "previous" affordance exists. Many list views only need next-only (infinite-scroll style) ‚Äî start there, add `before` when product needs it.
- **The "page size" question ‚Äî fixed vs. user-selectable.** Fixed page size lives in the parser (`pageSize: 20`) and isn't in the URL. User-selectable page size adds `?pageSize=50` to the URL (clamped to a max in the parser, `parseAsInteger.withDefault(20)` plus a Zod range check). The senior default: fixed. Page-size selectors are a power-user feature; ship the default first.
- **The Client Component shape ‚Äî `<Pagination />`.** Receives `cursor`, `hasNext`, `hasPrev` (server-computed) as props; renders "Next" and "Previous" buttons; on click, calls `setCursor(newCursor)` via `nuqs`. The server returns one extra row beyond the page size to compute `hasNext` cheaply (a senior trick from 6.3.6 restated here). `hasPrev` is derived from "is there a `before` value" ‚Äî true if the URL has a cursor at all.
- **The "jump to page N" affordance for offset case.** When offset is the right choice, the Client renders numbered page links (`1, 2, 3, ... 25`) and calls `setPage(N)`. Each is a `<Link>` to the same path with `?page=N`, or a `setPage(N)` setter call. The senior anchor: page links use `replace` for in-list navigation, same rule as filter/sort.
- **The "row inserted between page loads" problem ‚Äî why cursor wins.** With offset, a new row inserted at the top shifts every row's offset; page 2 now shows row 19 again (which was on page 1). The user sees a duplicate. Cursor pagination is immune: the next-page query asks for "rows after this sort key", which is stable. The lesson names this as the production failure mode that breaks offset and makes cursor the default.
- **The "go back to the first page" reset.** Whenever filter, sort, or search changes, the cursor (or page) is cleared ‚Äî same reset rule as 11.1.2 and 11.1.3, applied here too. The setter for any change to filter/sort/search bundles `cursor: null` (or `page: 1`) in the same `setQueryStates` call. The pattern is the chapter's invariant.
- **Total-count display ‚Äî when to compute, when to skip.** Showing "Showing 21-40 of 50,000" is friendly but costs an additional `count(*)` query. The senior reach: skip the total for cursor-paginated views (just show "21-40" with a "Load more" or "Next"), compute the total for offset-paginated small lists where it fits. For very large tables, the count itself can be slow without a separate optimization ‚Äî name and defer to Chapter 6.4.
- **The share-and-refresh consequence ‚Äî cursor URLs are stable, but the underlying data isn't.** A shared cursor URL takes the coworker to the rows after the encoded sort key, in the current data state. If rows were inserted, the coworker sees rows the original user didn't. If rows were deleted, fewer rows. This is correct for the cursor contract: "rows after this position", not "the exact rows I was looking at". The senior anchor: the URL guarantees the position, not the snapshot.
- **Cursor versioning ‚Äî handling sort changes gracefully.** A cursor encoded for `-createdAt` decoded against a `-total` sort produces wrong results. The cursor blob can carry a version tag identifying the sort key it was encoded for; the parse helper validates and falls back to the first page on mismatch. The senior pattern: version cursors against the sort, and the setter for sort changes clears the cursor anyway as the structural defense.
- **Worked example ‚Äî the pagination row wired end-to-end.** A Server Component query that returns `{ rows, nextCursor, prevCursor }`. The page renders the table with `rows`, passes `nextCursor`/`prevCursor` to `<Pagination />`. The Client Component reads its current cursor from `useQueryState('cursor', parseAsString)`, calls `setCursor(nextCursor)` on click. The full pattern, including the reset-on-other-change rule.
- **Watch-outs:** offset pagination on a large table is a perf bug waiting to happen ‚Äî the database scans `offset + limit` rows every time; rows-inserted-during-pagination produces phantom duplicates under offset ‚Äî switch to cursor; cursor blobs that aren't opaque (raw JSON in the URL) leak internal column names and invite hand-edits ‚Äî base64-encode at minimum; cursor decode failures must fall back to the first page silently ‚Äî throwing produces a bad UX; "load more" infinite-scroll without a URL update means the user can't share or refresh to the same scroll position ‚Äî the senior reach is to push the cursor on every load-more (with replace to avoid history spam); a `pageSize` parameter without an upper-bound clamp lets a client request millions of rows in one query ‚Äî clamp at the parser; total-count queries can dominate page latency at scale ‚Äî skip when not user-facing.

- Cursor mechanics at the database/query layer ‚Äî Chapter 6.3.6.
- Composite indexes for cursor pagination ‚Äî Chapter 6.4.1.
- Infinite scroll with intersection observers ‚Äî out of scope.
- Real-time list updates (new rows appearing live) ‚Äî out of scope, see notifications chapters.
