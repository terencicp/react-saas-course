# Chapter 18.2 — Internationalization

## Chapter framing

Chapter 18.2 closes Unit 18 with the formatting and translation half of the time-and-i18n discipline. The student arrives with the substrate from 18.1: every instant lives in `timestamptz` and is read as `Temporal.Instant`; every calendar day lives in `date` and is read as `Temporal.PlainDate`; the user's IANA timezone is a profile column passed explicitly into every formatter. What's missing is the discipline that surrounds the *human-readable* edge: translation strings as keys with interpolation (never concatenation), ICU MessageFormat for the plural/select rules every language needs, the `Intl.*` family as the native engine that powers every locale-aware render, an explicit locale-negotiation order, `next-intl` as the 2026 Next.js wiring, and the i18n-SEO surface that lets Google route the right locale to the right user. The lesson chain installs **i18n as a discipline before it's a feature** — even a single-locale launch ships through the same shape so the second locale is one PR, not a refactor.

Threads that run through every lesson. **Translation keys are the wire format; strings are the rendered output** — the application code holds `t('invoice.pastDue.title')`, the message catalog holds the localized text, and string concatenation in component code is the bug class the chapter exists to prevent. **The `Intl.*` family is the native engine** — every formatter in the codebase is `Intl.NumberFormat`, `Intl.DateTimeFormat`, `Intl.RelativeTimeFormat`, `Intl.PluralRules`, `Intl.Collator` (and the supporting `Intl.ListFormat`, `Intl.DisplayNames`); `next-intl`'s formatter hooks are convenience wrappers over the same engine, not a replacement. **Locale is a profile column, paired with `users.timeZone` from 18.1.3** — derived from the browser at sign-up, negotiated against `Accept-Language` as the fallback chain, validated against a supported-locales constant, and passed explicitly to every formatter and to `next-intl`'s middleware. **`hreflang` and per-locale URLs are the SEO contract** — search engines route locales by the URL shape and the `alternates.languages` metadata; the canonical URL is the localized URL, not the default. The chapter ships six teaching lessons plus the quiz; wiring sits in `lib/i18n.ts`, `routing.ts`, `proxy.ts`, `messages/{en,fr,de,...}.json`, `app/[locale]/layout.tsx`, and the per-page `generateMetadata`.

## Lesson 18.2.1 — Translation keys, interpolation, never concatenation

Topics to cover:

- **The senior question.** A component renders `<p>Welcome, {user.name}! You have {count} unread messages.</p>`. A second sprint adds French; the developer reaches for `'Bienvenue, ' + user.name + '! Vous avez ' + count + ' messages non lus.'` and the bug class begins — placeholders frozen in source-language order, plural agreement hand-rolled, no place for the translator to land. The lesson installs the discipline: every user-visible string is a **key** the application code holds and a **translation entry** in a per-locale catalog the translator owns, with **named interpolations** the catalog can reorder, and **never** runtime string concatenation across translatable fragments.
- **The three artifacts and where each lives.** **(1) The key** — a stable identifier in component code (`t('inbox.unreadHeader')`); chosen by the engineer, never translated. **(2) The catalog entry** — `messages/en.json`, `messages/fr.json`, one entry per key, the translator's surface; the value carries ICU MessageFormat syntax (Lesson 18.2.2 owns plurals/select; this lesson establishes the shape). **(3) The render** — the `t(key, values)` call resolves the key, applies values, returns the localized string. Three artifacts, one per role (engineer, translator, runtime).
- **Key naming — flat namespaced dot-paths.** The 2026 senior reach: 2–3 levels deep, dot-separated, scoped by feature, never deeper. `invoice.list.emptyState`, `settings.profile.timezoneLabel`, `auth.signIn.submit`. Avoid deep trees (`page.dashboard.widgets.invoices.list.header.title` is rewrite-bait when the UI moves); avoid single-bucket globals (`buttons.save` reused across 40 contexts produces translations that fit none). The grammar: `feature.surface.role` — feature is the route or module, surface is the section, role is what the string does (`title`, `description`, `cta`, `error`, `placeholder`).
- **One string, one key — the duplication-by-design rule.** A "Save" button on the profile page and a "Save" button on the billing page get *two* keys (`settings.profile.save`, `settings.billing.save`) even though English shows the same word. French may render "Enregistrer" in one context and "Sauvegarder" in another; German may need a different verb for billing. Translators need the context to decide. The bug class is the globalized `common.save` that produces wrong translations the moment a language disagrees with English on context.
- **Named placeholders, never positional.** `'Hello, {name}'` — never `'Hello, %s'` or `'Hello, $1'`. Translators reorder freely; languages that put the verb first (German), the object first (Japanese), or interleave honorifics need the freedom. Positional placeholders couple the catalog to source-language word order — the bug class shows up as "Hello, John" in English and "John, hello" being literally impossible in the translator's UI.
- **The `t(key, values)` signature.** `t('invoice.greeting', { name: user.name })` returns the rendered string with `{name}` substituted. Missing keys are a build-time error (next-intl's TypeScript generation, Lesson 18.2.5; without it, missing keys fall back to the key string and ship a leak). Missing values at runtime are a development error — `t('invoice.greeting')` when the message is `'Hello, {name}'` raises in dev, silently renders `'Hello, {name}'` in production. Type the values object; the type comes from the message itself (next-intl's compile-time inference).
- **No string concatenation across translatable fragments.** The senior rule: a translatable string is *one* `t()` call, with placeholders for the variable parts. Wrong: `t('inbox.youHave') + ' ' + count + ' ' + t('inbox.messages')`. Right: `t('inbox.unreadCount', { count })` with the message `'You have {count} messages'`. The wrong form makes plural rules impossible (Lesson 18.2.2), strips the translator's context, and produces grammatically broken renderings in languages where adjectives precede nouns or word order shifts. The lint reflex: a `+` between two `t()` calls, or a `t()` call with a JS-template-literal-built string, is a finding.
- **The "Trans" / rich-text shape for embedded markup.** A string like *"By signing up, you agree to our [Terms](/terms) and [Privacy Policy](/privacy)."* mixes translatable text and interactive elements. The senior reach: ICU's tag syntax in the message (`'By signing up, you agree to our <terms>Terms</terms> and <privacy>Privacy Policy</privacy>.'`) with a render-callback at the call site (`t.rich('auth.signUp.terms', { terms: (chunks) => <Link href="/terms">{chunks}</Link>, privacy: ... })`). The translator reorders, the engineer keeps control of the markup. The wrong form is splitting the sentence into three keys and concatenating with JSX — the link's placement is frozen in English word order.
- **Pluralization is a separate concern — preview.** A count-aware message needs plural rules (one/few/many/other), not a ternary on `count === 1`. Hand-rolled plurals fail on Slavic languages (Russian: 1 / 2-4 / 5+), Arabic (six categories), Welsh (six categories). Lesson 18.2.2 installs ICU MessageFormat. The discipline here: a count-shaped string always uses ICU plural, never `${count === 1 ? 'message' : 'messages'}`.
- **The message catalog as a checked-in JSON file.** `messages/en.json` is the source of truth and the dev fallback; `messages/fr.json`, `messages/de.json` ship beside it. PRs that add a key add it to all locales (the build fails if a locale is missing the key — next-intl's strict mode). Translators consume these JSONs directly (or through a TMS — Crowdin, Lokalise, Phrase — that round-trips JSON). The "missing translation" UI bug is structurally prevented at build time.
- **Source-of-truth locale and translation lifecycle, named once.** English (or whichever the team writes the app in) is the source locale; new keys land in `en.json` first; other locales mark the entry as "needs translation" and either fall through to source (graceful degradation) or block the PR (strict). 2026 senior default: strict in production, graceful in development. The translation flow (PR → export JSON → TMS → translator → re-import JSON → PR) is a build-step concern; the discipline this lesson installs is upstream of that flow.
- **String length and the truncation pitfall.** German strings run 30–40% longer than English on average; Chinese 30% shorter. Designs that fit "Save" in 40px will overflow "Speichern" or break on "保存" baseline alignment. The senior reach: never hard-code container widths around source-language strings, use `min-content` / `max-content` / `flex` with wrap, and test the longest-locale layout (German, Finnish, Russian) before launch. Named here as a "discipline that emerges from real translation," fully owned by the design system.
- **RTL languages — the chapter's scope-out.** Arabic, Hebrew, Persian, Urdu read right-to-left, swap layout direction, mirror icons. The HTML `dir="rtl"` attribute and Tailwind's logical properties (`ms-*` / `me-*` instead of `ml-*` / `mr-*`) are the wiring. Named once; the chapter's projects don't ship an RTL locale; the principle is named so the student recognizes the cut.
- **Watch-outs.** A string in JSX without `t()` is the bug class the audit walks for — grep for hard-coded quoted strings in `<>` / `<Button>` / `<Heading>` blocks; emoji and short symbols (✓, ×, →) are usually safe but not always (commas, periods, ellipsis vary by locale — the ellipsis character changes); date / number formatting in source code with `.toLocaleString()` or template literals is a separate bug class (Lesson 18.2.3 owns it); concatenating a number with a translated noun (`${count} messages`) without ICU plural breaks every non-English language; reusing `common.cancel` across "cancel form" and "cancel subscription" produces wrong translations in languages where the verb differs; a key whose value carries HTML (`'<b>Welcome</b>'`) couples the translator to markup — use `t.rich` and never let HTML into the message string; translators rewriting placeholders (`{name}` → `{user}`) breaks runtime; the catalog's JSON formatter strips quote escaping silently — pin the formatter and review diffs.

What this lesson does not cover:

- ICU MessageFormat plurals, select, and gendered forms — Lesson 18.2.2.
- The `Intl.*` formatters for numbers, dates, relative times — Lesson 18.2.3.
- Locale negotiation and the `users.locale` profile column — Lesson 18.2.4.
- `next-intl`'s setup, hooks, and message-typing — Lesson 18.2.5.
- `hreflang` and per-locale SEO — Lesson 18.2.6.
- Translation-management-system integration (Crowdin, Lokalise, Phrase) — out of scope, named once.
- RTL layout and bidirectional text — out of scope; named once.

Estimated student time: 40 to 50 minutes. The chapter's discipline-installation lesson; every later lesson references the key/catalog/render split established here.

## Lesson 18.2.2 — ICU MessageFormat: plurals, select, gendered forms

Topics to cover:

- **The senior question.** A notification says "You have 1 message" or "You have 5 messages." In English, the rule is "one vs. other" — a ternary on `count === 1` produces the right answer. In Russian, the rule is "one" (1, 21, 31), "few" (2-4, 22-24), "many" (5-20, 25-30), "other" (decimals); the ternary produces grammatically broken text for half the integers. In Arabic, six categories (`zero`, `one`, `two`, `few`, `many`, `other`). In Welsh, six different categories. Hand-rolling per-language plural logic in component code is the bug class. The senior reach is **ICU MessageFormat** — the catalog entry carries the plural rules, the runtime applies CLDR's plural categories per locale, the engineer writes one `t(key, { count })` call.
- **What ICU MessageFormat is.** A string syntax defined by Unicode (in CLDR; ICU is the C++/Java reference implementation; every modern i18n library implements a subset). The catalog string carries logic: `'{count, plural, =0 {No messages} one {# message} other {# messages}}'`. The runtime parses the string, looks up the plural category for `count` in the current locale, picks the matching branch, substitutes `#` with the formatted number. Five core features the chapter uses: **plural** (cardinal), **selectordinal** (ordinal — "1st", "2nd"), **select** (free enum on a string — gender, role, type), **number formatting** (`{amount, number, ::currency/USD}`), **date formatting** (`{date, date, ::yyyyMMdd}`). The chapter ships the first three at depth; number / date formatting hand off to `Intl.NumberFormat` / `Intl.DateTimeFormat` directly (Lesson 18.2.3) — both work, the senior reach is the explicit formatter call at the seam.
- **The `plural` form, anatomized.** `{count, plural, =0 {No messages} one {# message} other {# messages}}`. Three pieces. **(1)** The variable name (`count`) — matches the value passed to `t()`. **(2)** The selector keyword (`plural` for cardinal counts; `selectordinal` for ordinal). **(3)** The branch table — exact matches (`=0`, `=1`, `=2`) tried first, then CLDR category keywords (`zero`, `one`, `two`, `few`, `many`, `other`). `other` is mandatory — it is the fallback when no other branch matches. `#` inside a branch is the count formatted as a number (locale-aware grouping, decimals); to opt out, omit `#` and use a literal. `{#, number, ::compact-short}` formats the count with the compact form (`1.2K`).
- **CLDR plural categories — what each language gets.** The Unicode CLDR (Common Locale Data Repository) declares plural categories per language. English: `one` (1), `other` (everything else). French: `one` (0, 1), `many` (large numbers), `other`. Russian: `one`, `few`, `many`, `other`. Arabic: `zero`, `one`, `two`, `few`, `many`, `other`. Welsh: `zero`, `one`, `two`, `few`, `many`, `other`. The translator picks the branches their language needs; the catalog file's entry shapes per locale. English `en.json` carries two branches (`one`, `other`); Russian `ru.json` carries four (`one`, `few`, `many`, `other`); the rendering runtime picks the right branch by querying CLDR's rules.
- **`Intl.PluralRules` as the engine.** The native API: `new Intl.PluralRules('ru').select(5)` returns `'many'`. Every ICU MessageFormat implementation calls into this (or a polyfill) at branch-resolution time. The student rarely calls it directly — the message handles it — but the API is what makes "the catalog says `few` for Russian" actually work. Custom plural logic in component code (`count === 1 ? 'message' : 'messages'`) is wrong because it ignores CLDR; the senior reach delegates to the standard.
- **The exact-match overrides — `=0`, `=1`, `=2`.** Some products want "No messages" instead of "0 messages" for clarity (English category `other` would render "0 messages" otherwise). Exact-match branches override category lookup. Order in the syntax doesn't matter; exact matches take priority. Convention: list exact matches first, then category branches, then `other`. The `=0` / `=1` carve-outs are senior product calls, not a translation step — declare them in the source-locale catalog and let translators decide whether their language needs the same overrides.
- **The `selectordinal` form for ordinals.** `{rank, selectordinal, one {#st place} two {#nd place} few {#rd place} other {#th place}}`. English's ordinal suffix rule is the canonical use case ("1st", "2nd", "3rd", "4th"). Other languages have different rules (German uses "1.", "2.", "3." — the period is the suffix; Japanese uses 番). The `selectordinal` keyword consults CLDR's ordinal plural rules (distinct from cardinal). Reach for it when displaying ranks, positions, race results, leaderboard order.
- **The `select` form — free enums.** `{gender, select, male {He liked your post} female {She liked your post} other {They liked your post}}`. The selector is a string; branches are literal matches with `other` as fallback. Use cases: gender (when the data carries it; never infer from the name), notification type (`{type, select, mention {You were mentioned} reply {You got a reply} other {Activity}}`), organization role (`{role, select, owner {...} admin {...} member {...}}`). The English source often collapses `male` / `female` / `other` to the same string (gender-neutral writing); the German catalog renders three forms because German nouns and pronouns inflect.
- **Combined plural + select — gendered counts.** Two-level nesting: `{gender, select, male {{count, plural, one {He has # message} other {He has # messages}}} female {{count, plural, one {She has # message} other {She has # messages}}} other {{count, plural, one {They have # message} other {They have # messages}}}}`. The senior framing: nest the *outer* selector by the higher-cardinality dimension (usually gender / type), the inner by plural. The translator's surface is the entire string; the engineer's call is one `t('notification.likedYourPosts', { gender, count })`.
- **MessageFormat 2 — named once, not the default yet.** Unicode is finalizing MessageFormat 2 (MF2), a successor syntax with explicit `.input`, `.local`, `.match` declarations and named-formatter integration. MF2 is in ICU but most i18n libraries (next-intl included) still target the original MessageFormat (sometimes called MF1) as of May 2026. The lesson teaches MF1 because that's what ships; MF2 is named as the path forward. Migration is mechanical (the same plural / select branches express the same logic) when next-intl and the broader ecosystem adopt MF2.
- **The "translator never writes JS" rule.** A translator opens the catalog JSON, sees `'{count, plural, one {# message} other {# messages}}'`, and edits the localized strings inside the branches. They never touch JS, never see component code, never reach for a regex. The ICU syntax is the boundary; everything inside the branches is text, everything outside is the framework. A message that requires the translator to understand a ternary or a callback failed the rule.
- **Watch-outs.** Missing the `other` branch raises a runtime parse error in every ICU implementation — `other` is mandatory; the lint catches it; many TMS validators flag it before the catalog ships; using `=1` instead of `one` in English works but breaks the moment the locale is German (where `one` covers `1` only, but ordinal forms differ) — prefer CLDR categories, use exact-match only for narrative overrides; rendering `{count}` instead of `#` inside a plural branch produces "You have 5 5 messages" (the variable and the count both render); using `select` for gender by reading from `user.name` (regex on suffix) is the bug — the data must carry the gender or the message must be gender-neutral; nesting plural inside plural is rare but legal — re-read the syntax three times before shipping; ICU's quote-escaping rules (`'` inside a literal) trip up JSON editors — always validate the catalog with the runtime; some libraries advertise "ICU support" but skip `selectordinal` or `select` — confirm with the actual runtime; storing the message in the database (CMS pattern) means a content editor can break the ICU syntax — validate at write time with a parse step.

What this lesson does not cover:

- The translation key / catalog discipline — Lesson 18.2.1.
- `Intl.PluralRules` standalone surface for non-message use — Lesson 18.2.3 names it.
- `Intl.NumberFormat` and `Intl.DateTimeFormat` standalone — Lesson 18.2.3.
- Locale negotiation — Lesson 18.2.4.
- `next-intl`'s `useTranslations` / `t.rich` / inline formatters — Lesson 18.2.5.
- MessageFormat 2.0 syntax at depth — out of scope; named once.
- CLDR's full plural rule table per language — link out, not covered.

Estimated student time: 40 to 50 minutes. The chapter's load-bearing rule for any string with a count or a gendered variant.

## Lesson 18.2.3 — `Intl.NumberFormat`, `DateTimeFormat`, `RelativeTimeFormat`, `Collator`

Topics to cover:

- **The senior question.** A balance is `1234.56`. English rendering: `'$1,234.56'`. German rendering: `'1.234,56 €'`. French rendering: `'1 234,56 €'` (with a non-breaking thin space). A "due 3 days ago" string is `'il y a 3 jours'` in French, `'vor 3 Tagen'` in German. Sorting "ä" — does it land with "a" (German) or after "z" (Swedish)? Every one of these is a *runtime decision*, locale-dependent, and the senior 2026 reach is **the native `Intl.*` family**, not a library, not a `toLocaleString()` shortcut without arguments. The lesson surveys the five formatters every SaaS reaches for and names the supporting two (`ListFormat`, `DisplayNames`) once. The chapter's `next-intl` wiring (Lesson 18.2.5) wraps these — knowing the underlying engine is the senior anchor.
- **The shape every formatter shares.** `new Intl.X(locales, options)` returns a formatter instance. `.format(value)` returns the string; `.formatToParts(value)` returns an array of tokens (useful when you need to style or interpolate parts); some have `.formatRange(a, b)` for ranges and `.formatRangeToParts(a, b)` for parts. **Construct once, reuse** — formatter construction is the expensive step; memoize at module scope or behind a `getFormatter(locale, options)` cache. A per-render `new Intl.NumberFormat(...)` in a Server Component is wasted CPU.
- **`Intl.NumberFormat` — the daily reach.** **(1) Currency.** `new Intl.NumberFormat(locale, { style: 'currency', currency: 'USD' })`; the currency code is *data*, not a UI concern (the SaaS's invoice carries USD, EUR, JPY — the format renders per-locale convention, the currency is preserved). The `currencyDisplay` option controls `'symbol'` (default; `'$'`, `'€'`), `'narrowSymbol'` (`'$'` instead of `'US$'`), `'code'` (`'USD'`), `'name'` (`'US dollars'`); senior default is `'narrowSymbol'` for compact UI, `'name'` for verbose receipts. **(2) Percent.** `style: 'percent'` with `maximumFractionDigits`; the input is the *fraction* (`0.15` → `'15%'`). **(3) Decimal.** `style: 'decimal'` with `minimumFractionDigits` / `maximumFractionDigits` for fixed-precision rendering (analytics numbers). **(4) Compact.** `notation: 'compact'` produces `'1.2K'`, `'3.4M'`, `'1.2 mil'`; pair with `compactDisplay: 'short'` (default) or `'long'`; reach for it in dashboards, social-style counters. **(5) Units.** `style: 'unit'` with `unit: 'kilometer'` / `'megabyte'` / `'hour'`; the spec lists supported units (the chapter names file-size and duration as the SaaS reach). The senior call: every currency / percent / large-number render in the codebase goes through `Intl.NumberFormat`; `value.toFixed(2)` is the bug class.
- **`Intl.DateTimeFormat` — the renderer for Temporal values.** From 18.1: the application holds `Temporal.Instant` / `Temporal.PlainDate` / `Temporal.ZonedDateTime` in memory and renders at the edge. **Temporal interop.** Modern `Intl.DateTimeFormat` accepts Temporal types natively — `formatter.format(instant)` works for `Temporal.Instant` (uses the formatter's `timeZone` to resolve), `Temporal.PlainDate` (uses the formatter's default fields), `Temporal.ZonedDateTime` (uses the value's tz unless the formatter overrides). The student writes `formatter.format(invoice.createdAt)` where `createdAt` is `Temporal.Instant`; no `new Date()`, no `.toISOString()` round-trip. **The `timeZone` option is mandatory in SaaS code.** Restated from 18.1.3: a formatter constructed without `timeZone` defaults to the runtime's tz — UTC on Vercel, the developer's local on `pnpm dev`. The wrapper `formatDate(value, { locale, timeZone, ...options })` *requires* both — TypeScript enforces the shape, runtime checks the value. **Style presets vs. component options.** Two ways to specify shape: presets (`dateStyle: 'short' | 'medium' | 'long' | 'full'` paired with `timeStyle`) — the senior default for most renders; the locale picks the format; component options (`year: '2-digit'`, `month: 'short'`, `day: 'numeric'`, `hour: '2-digit'`) when control matters — used together with presets is an error. **`formatRange`.** Renders date ranges idiomatically — "Jan 5–7, 2026" in en-US, "5–7 janv. 2026" in fr-FR. Reach for it when displaying booking windows, billing periods, event durations.
- **`Intl.RelativeTimeFormat` — "3 days ago" the right way.** `new Intl.RelativeTimeFormat(locale, { numeric: 'auto' })`; `formatter.format(-3, 'day')` returns `'3 days ago'` in en-US, `'il y a 3 jours'` in fr-FR. `numeric: 'auto'` allows special-cases ("yesterday", "tomorrow", "now"); `numeric: 'always'` always returns the numeric form ("1 day ago"). The senior pairing with Temporal: `since` / `until` returns a `Duration`, the largest non-zero unit becomes the formatter's input, the sign matters (negative for "ago", positive for "in"). A small `formatRelative(instant, { locale, now })` helper hides the duration-to-unit logic; the formatter itself is the rendering primitive. Watch-out: don't chain "3 minutes ago" with auto-update on a timer that re-renders the whole page — use the user's tz and a stable "now" anchor per request to avoid hydration mismatches (18.1.1's rule); for live timestamps, a small client-island that re-renders every minute is the senior shape.
- **`Intl.PluralRules` — standalone use, restated.** Lesson 18.2.2's plural messages call into this engine. Direct use is rare — sometimes a custom branching scheme that doesn't fit ICU MessageFormat (e.g., choosing between two icons by plural category). `new Intl.PluralRules('en').select(1)` → `'one'`; `.select(5)` → `'other'`. Cardinal by default; pass `{ type: 'ordinal' }` for ordinal categories. Reach for the formatter rarely; reach for the spec it implements (CLDR plural rules) on every count-shaped UI.
- **`Intl.Collator` — locale-aware sort and search.** `new Intl.Collator(locale)` returns a comparator (`collator.compare(a, b)` for `Array.sort`). The senior reach: every user-facing list sort that operates on translated strings, user names, or anything not pure ASCII. **Sensitivity.** `sensitivity: 'base'` ignores accents and case (`'a' === 'á' === 'A'`) — the senior default for *search* matching; `'accent'` distinguishes accents, ignores case — the senior default for sort that respects diacritics; `'case'` distinguishes case only; `'variant'` (default) distinguishes everything. **`usage`.** `usage: 'sort'` (default) for ordering; `usage: 'search'` for substring / equality matching — the latter is the senior reach for "does this query match this user's name." **`numeric`.** `numeric: true` enables natural-numeric sort (`'item2'` before `'item10'`); the senior default for filenames, version strings, anything with embedded numbers. **`caseFirst`.** Rarely needed. The wrong reach is `string.localeCompare(other)` — works but constructs a fresh collator per comparison; for large sorts, construct the collator once and pass `collator.compare` to `.sort`.
- **`Intl.ListFormat` — natural-language lists.** `new Intl.ListFormat('en', { style: 'long', type: 'conjunction' })`; `formatter.format(['Alice', 'Bob', 'Carol'])` → `'Alice, Bob, and Carol'` (en-US Oxford comma); same in French → `'Alice, Bob et Carol'`. Three types: `'conjunction'` ("and"), `'disjunction'` ("or"), `'unit'` (compact list — "1h 30m"). Three styles: `'long'`, `'short'`, `'narrow'`. Reach for it whenever joining a list of items in user-visible copy — never `array.join(', ')` for a translatable surface.
- **`Intl.DisplayNames` — language, region, currency, script names.** `new Intl.DisplayNames('en', { type: 'language' })`; `.of('fr')` → `'French'`; `.of('fr-CA')` → `'Canadian French'`. Types: `'language'`, `'region'`, `'currency'`, `'script'`, `'dateTimeField'`. Reach for it in language pickers (the dropdown showing `'Français'` next to `'English'` is `displayNames.of('fr')` with `locale: 'fr'`), country selectors, currency selectors. Two senior reaches: render each locale's name *in its own locale* (the dropdown reads correctly for a non-English user — `'Français'` not `'French'`), and render in the *current* user's locale for region / currency labels.
- **`Intl.Segmenter` — named once.** Locale-aware text segmentation by grapheme, word, sentence. Reach for it when truncating user-generated text (`'こんにちは'.length` is 5 in JS; visual graphemes need `Segmenter` to count correctly), tokenizing search inputs in CJK languages. Out of the chapter's daily-reach scope; named for awareness.
- **The cache pattern, restated.** `getNumberFormatter(locale, options)` / `getDateFormatter(locale, options)` / `getCollator(locale, options)` keyed by `JSON.stringify(options)` + `locale` in a module-scope `Map`. next-intl's `useFormatter` / `getFormatter` hooks (Lesson 18.2.5) do this internally; standalone code does it explicitly. The cost matters at scale — formatter construction is double-digit milliseconds; per-row in a 1000-row table that's seconds.
- **The "locale is a contract, not a string" rule.** Every formatter takes a BCP 47 locale tag. `'en'` is incomplete in product code — different conventions for date and currency between `'en-US'` and `'en-GB'` (`'$'` vs. `'£'`, `MM/DD/YYYY` vs. `DD/MM/YYYY`). The profile column (Lesson 18.2.4) stores the full tag (`'en-US'`, `'fr-FR'`); the formatter receives it directly. `'en'` as a stored value is a smell — defaults to `'en-US'` per CLDR but renders inconsistently across browsers.
- **Watch-outs.** A `value.toLocaleString()` call without `locales` and `options` uses the runtime's locale and the default format — the same bug class as `Intl.DateTimeFormat()` with no arguments; constructing the formatter inside the render is a perf bug at scale; passing a `Date` to `Intl.DateTimeFormat` when the codebase otherwise uses Temporal is a type-discipline leak — convert at the seam; rendering currency with hard-coded `'$'` instead of `Intl.NumberFormat` with the currency code drops to wrong rendering for any non-USD account; rendering a "3 days ago" string from a `Date.now() - timestamp` math (without Temporal's `since` or a calendar-aware diff) drifts on DST boundaries; sorting by `string.localeCompare` per element in a sort callback is N collator constructions — use one `Intl.Collator` and pass its `.compare`; `Intl.RelativeTimeFormat` with `numeric: 'always'` for "live" timestamps that hit zero displays "0 minutes ago" — handle the zero case explicitly; `formatter.format(NaN)` returns `'NaN'` in en-US, the locale's NaN representation elsewhere — guard against invalid inputs at the seam; the CLDR data is bundled with Node / the browser, so different runtimes can produce slightly different output for edge formats — pin the runtime in CI for tests that snapshot strings.

What this lesson does not cover:

- The translation-key discipline — Lesson 18.2.1.
- ICU MessageFormat plurals / select — Lesson 18.2.2.
- Locale negotiation and the `users.locale` profile column — Lesson 18.2.4.
- `next-intl`'s formatter hooks — Lesson 18.2.5.
- The hreflang / SEO surface — Lesson 18.2.6.
- Bidirectional text shaping (BiDi), RTL layout — out of scope.
- `Intl.Segmenter` at depth — named once.

## Lesson 18.2.4 — Locale negotiation: profile preference plus `Accept-Language`

Topics to cover:

- **The senior question.** A French user signs in from a German hotel's network; the browser sends `Accept-Language: de-DE,de;q=0.9,en;q=0.7`. The user has `'fr-FR'` saved in their profile. A second user — anonymous, hitting the marketing site — sends `Accept-Language: pt-BR,pt;q=0.9`. The product supports `'en-US'`, `'fr-FR'`, `'de-DE'`, `'es-ES'`, `'pt-PT'`. What locale renders for each request, and what's the negotiation order that makes the call deterministic? The senior reach: a single **resolution chain** with explicit priorities — profile → cookie → `Accept-Language` (best-match against supported) → default — that runs once per request in middleware, lands on the request as a typed `locale`, and is the input to every formatter, every `t()` call, every `<html lang>`.
- **The five inputs the negotiator can read.** **(1) URL prefix** — `/fr/dashboard` carries the locale explicitly; this is the strongest signal because the user (or the previous middleware run) put it there. **(2) Profile column** — `users.locale`, set by the user on the settings page. **(3) Cookie** — the locale-switcher writes `NEXT_LOCALE` (or next-intl's named cookie); persists the choice for anonymous users and short-circuits the negotiation for repeat visits. **(4) `Accept-Language` header** — the browser's preference list with q-values, BCP 47 tags. **(5) Geo-IP** — Vercel's `request.geo.country` (or Cloudflare's equivalent); inferring "the user is in France, default to French" is a *fallback*, not a primary signal — a French speaker in Brazil is real; an English-speaking expat in Tokyo doesn't want Japanese. The chapter's senior reach: **never use geo-IP as a primary signal**; named once.
- **The resolution chain, in order.** **(1)** If the URL has a locale prefix (`/fr/...`), that's the locale. **(2)** Else, if the user is authenticated, `users.locale` from the session. **(3)** Else, the `NEXT_LOCALE` cookie. **(4)** Else, best-match `Accept-Language` against the supported-locales constant. **(5)** Else, the default locale (`'en-US'`). The chain runs in middleware (`proxy.ts` or next-intl's middleware); the result is the request's locale; the URL gets rewritten to the prefixed form (`/dashboard` → `/en-US/dashboard`) so downstream rendering is uniform. The chain is deterministic; the audit step is to grep for any locale read that doesn't come from the middleware's resolved value.
- **BCP 47 best-match against the supported set.** RFC 4647 defines two matching algorithms — `Lookup` (find the most specific supported tag) and `Filter` (find all supported tags). For locale negotiation, `Lookup` is the senior reach. The algorithm: parse `Accept-Language` into ranked tags with q-values (`de-DE;q=0.9,de;q=0.7,en;q=0.5`); for each tag in priority order, try the exact tag against the supported set; if no match, fall back to the base language (`de-DE` → `de`); if no match, try the next ranked tag; if nothing matches, return the default. **The implementation surface in 2026.** `Intl.LocaleMatcher` is a TC39 Stage 1 proposal, not yet native; the senior 2026 reach is `@formatjs/intl-localematcher` (the proposal's reference ponyfill, used by every major i18n library) or next-intl's built-in matcher (which wraps the same algorithm). The chapter's wiring delegates to next-intl's matcher in middleware; standalone code that runs outside the middleware (a Server Action, a CLI script) imports the ponyfill directly.
- **The `users.locale` column, paired with `users.timeZone` from 18.1.3.** `text` on `users`, NOT NULL with `'en-US'` default at backfill. Stores a full BCP 47 tag (`'en-US'`, `'fr-FR'`, `'pt-BR'`) — never just the language code. Validated at the write edge against `SUPPORTED_LOCALES` (the `as const` array exported from `lib/i18n.ts`); the Zod schema is `z.enum(SUPPORTED_LOCALES)`. The profile UI presents a `<select>` of supported locales with each option's name rendered in its own locale (`Intl.DisplayNames` from Lesson 18.2.3). Seeded on sign-up from the browser (`navigator.language` or the negotiated value from `Accept-Language`); editable on the profile page; propagates to every server-rendered formatter via the session.
- **Locale and timezone are independent.** Restated from 18.1.3. A Berlin user can read in `'en-GB'` and operate in `'Europe/Berlin'`. A San Francisco user can read in `'es-ES'` and operate in `'America/Los_Angeles'`. Two columns, two pickers, no inference. The product's senior reach: never derive locale from timezone or vice versa; both are user-asserted, both default to plausible values, both editable.
- **The marketing-site / anonymous-user shape.** No profile; the chain collapses to URL prefix → cookie → `Accept-Language` → default. The cookie is the senior reach for "the user picked French on the marketing site, the SEO crawler does not get redirected, the next anonymous visit honors the choice." Next-intl 4 defaults the locale cookie to a *session cookie* (expires on browser close) and *only writes it when the user picks a locale that differs from `Accept-Language`* — both choices are GDPR-driven (a cookie that just mirrors the header is "strictly necessary"; a cookie that overrides it carries consent implications). The senior framing: the locale cookie is essential-for-functionality, not consent-gated.
- **What `Accept-Language` does and does not carry.** Carries: ranked BCP 47 tags with optional q-values. Does *not* carry: timezone, currency preference, calendar system, region of residence (`en-US` from a German user on vacation is real; `en` without a region from many corporate browsers is more common than expected). The senior framing: `Accept-Language` is *one* signal among five, not the truth. Falling through to `Accept-Language` without best-match against the supported set produces the canonical bug — the browser sends `'pt-BR'`, the supported set is `['pt-PT', 'en-US']`, naive code defaults to `'en-US'` instead of falling to `'pt-PT'` (same base language, closest match).
- **The locale-switcher UI.** A dropdown rendered in every layout (header, footer, settings page); selection writes the cookie, calls a Server Action to update `users.locale` for authenticated users, and navigates to the same path under the new prefix (`/fr/dashboard` → `/de/dashboard`). next-intl's navigation hooks (`useRouter`, `usePathname`) handle the prefix rewrite. The display: each option in its own language ("English" / "Français" / "Deutsch") so a non-English user reads their language correctly.
- **The `<html lang>` attribute — the load-bearing accessibility hook.** Set on the root layout: `<html lang={locale}>`. Screen readers, hyphenation, browser spell-checkers, search engines, machine translation all key on this. The middleware-resolved locale lands in the layout; mismatch (`<html lang="en">` while rendering French content) breaks accessibility and signals a routing bug. Pair with `dir={isRTL(locale) ? 'rtl' : 'ltr'}` when the supported set includes RTL languages (out of chapter scope; named).
- **The `Content-Language` response header.** Mirrors `<html lang>` — `Content-Language: fr-FR` on the response. Crawlers and proxies key on it; cheap to set in middleware; the senior reach when the response represents content in a specific locale.
- **The "every render reads the resolved locale" rule.** The negotiation runs once per request, in middleware; downstream code reads the resolved value from the request context (next-intl's `useLocale()` / `getLocale()`). No component, no formatter, no helper re-reads `Accept-Language` or `navigator.language`. The audit grep: `Accept-Language` outside the middleware, `navigator.language` outside the sign-up form's tz-detection-style capture — both are findings.
- **Watch-outs.** Trusting `Accept-Language` without best-match against the supported set ships the default locale to every non-exact match; storing a language-only tag (`'en'`) and then formatting currency in en-US convention for a UK user is wrong — store full BCP 47; auto-redirecting based on geo-IP (the user lands on `/`, gets bounced to `/fr/` because they're in France) breaks deep links and SEO crawlers — never auto-redirect on geo; updating the profile locale silently without updating the cookie leaves anonymous-session-style navigation in the old locale; the cookie name collision between next-intl, custom code, and a TMS's preview tool causes ghost overrides — name explicitly (`NEXT_LOCALE`); locale-switching on a deep route should preserve the path (`/fr/invoices/abc` → `/de/invoices/abc`), not bounce to home — next-intl's `usePathname` handles this; a 301 redirect from non-prefixed to prefixed URLs is the SEO standard but breaks if every request lands at `/` and bounces — gate on whether the URL already has a locale prefix; rendering `<html lang>` from the locale *cookie* instead of the resolved value on the server produces hydration mismatch when the cookie disagrees with the middleware result.

What this lesson does not cover:

- The translation key / catalog discipline — Lesson 18.2.1.
- ICU MessageFormat — Lesson 18.2.2.
- `Intl.*` formatters — Lesson 18.2.3.
- `next-intl`'s middleware, routing, and hooks — Lesson 18.2.5.
- The hreflang / SEO surface — Lesson 18.2.6.
- Geo-IP-driven routing patterns at depth — out of scope; named as anti-pattern.
- `navigator.languages` plural-form (multi-language readers) — out of scope; named once.

Estimated student time: 40 to 50 minutes.

## Lesson 18.2.5 — `next-intl`: wiring the discipline in Next.js 16

Topics to cover:

- **The senior question.** Lessons 18.2.1 through 18.2.4 installed the discipline — keys, ICU messages, `Intl.*`, locale negotiation. The wiring question is which library lands the discipline in Next.js 16's App Router with the least drift. The senior 2026 default is **`next-intl`** — actively maintained, native Server-Component support, App-Router-shaped, ~2KB client bundle, ICU MessageFormat built in, navigation APIs (`<Link>`, `redirect`, `usePathname`, `useRouter`) that preserve the locale prefix, build-time message typing, and compatibility with Next.js 16 confirmed. Alternatives — `react-intl` / FormatJS (full ICU but no App-Router routing primitive), `next-i18next` (Pages-Router-shaped; mechanical migration to App Router but doesn't match next-intl's Server-Component ergonomics), Lingui (compile-step heavy) — exist; the chapter's reach is next-intl. The lesson lands the wiring as a Setup/Wiring archetype: every file, every named export, every option that matters.
- **The file shape — six places.** **(1)** `lib/i18n/routing.ts` — `defineRouting({ locales, defaultLocale, localePrefix })` is the central config; supported locales (the `as const` array from Lesson 18.2.4), the default, and the prefix mode. **(2)** `lib/i18n/navigation.ts` — `createNavigation(routing)` returns the typed `Link`, `redirect`, `usePathname`, `useRouter` that respect the locale prefix; every navigation in the app imports from here, not from `next/link` / `next/navigation`. **(3)** `proxy.ts` (Next.js 16's renamed middleware) — `createMiddleware(routing)` runs the negotiation chain from Lesson 18.2.4 and rewrites the URL. **(4)** `i18n/request.ts` — the per-request config callback that resolves the locale (via `getRequestConfig`), loads the message catalog (`messages/${locale}.json`), and returns `{ messages, timeZone, now }`. The `timeZone` is the user's `users.timeZone` (18.1.3) when authenticated, falling back to the org or default. **(5)** `app/[locale]/layout.tsx` — the root layout consumes the locale param, calls `setRequestLocale(locale)` for static rendering, wraps children in `<NextIntlClientProvider>` (only when client components consume translations), sets `<html lang={locale}>`. **(6)** `messages/{en,fr,de}.json` — the catalogs from Lesson 18.2.1.
- **`defineRouting` — the locale list and prefix mode.** `defineRouting({ locales: ['en-US', 'fr-FR', 'de-DE'] as const, defaultLocale: 'en-US', localePrefix: 'as-needed' })`. **Prefix modes.** `'always'` — every URL is prefixed (`/en-US/dashboard`); deterministic but ugly for the default locale; senior reach for products that ship multi-locale from day one. `'as-needed'` — the default locale is unprefixed (`/dashboard`), others are prefixed (`/fr-FR/dashboard`); the senior reach for products with a strong default-locale market; pairs cleanly with SEO. `'never'` — locale is resolved entirely from the cookie / `Accept-Language`; senior reach is rare (apps that don't care about per-locale URLs, mostly behind-auth tools); SEO suffers because the same URL serves different content. The chapter's default: `'as-needed'`.
- **`createMiddleware` — the negotiation runner.** `export default createMiddleware(routing);` in `proxy.ts`. The middleware reads `Accept-Language`, the cookie, the URL prefix, runs the BCP 47 best-match against `routing.locales`, rewrites the URL to the prefixed form (or canonical unprefixed form for the default), and persists the cookie. The chain from Lesson 18.2.4 is implemented inside; the senior knob is to compose with other middleware (auth, security headers) using Next.js's `proxy.ts` chaining pattern (Unit 5.5 / 9.4 / 17.2.1 own the composition shape).
- **`getRequestConfig` — per-request message and tz resolution.** `i18n/request.ts` exports `default getRequestConfig(async ({ requestLocale }) => { ... })`. Returns `{ locale, messages, timeZone, now }`. **Loading messages.** `messages: (await import(`../../messages/${locale}.json`)).default` — dynamic import, code-splits per locale, only the active locale ships. **Timezone.** Read from `users.timeZone` via the auth session for authenticated requests; default to `'UTC'` for anonymous routes. The lesson lands the senior call: the request config is the single seam where locale, tz, and message catalog converge; downstream code never reaches for `Accept-Language` or the cookie directly.
- **`setRequestLocale` and static rendering.** Next.js 16 prefers static rendering for layouts and pages by default. next-intl reads the locale from the request via `headers()` (dynamic) unless `setRequestLocale(locale)` is called at the top of the layout / page — which writes the locale to a per-request store, makes downstream `useTranslations` calls work without `headers()`, and re-enables static rendering. The senior rule: every `app/[locale]/**/page.tsx` and `app/[locale]/**/layout.tsx` starts with `setRequestLocale(locale)`. Skipping it converts the page to dynamic silently — the Lighthouse perf hit is real and the bug is invisible at code review without the rule.
- **`generateStaticParams` for prerender.** The root locale layout exports `generateStaticParams() { return routing.locales.map((locale) => ({ locale })); }`. Each locale gets prerendered at build time. Pairs with `setRequestLocale` — without both, only the default locale prerenders, the others fall to dynamic.
- **`useTranslations` (client and server) vs. `getTranslations` (server async).** Two APIs, same engine. **`useTranslations(namespace)`** is the sync hook — works in Server Components and Client Components, returns a `t` function. The senior default for *most* rendering. **`getTranslations({ locale, namespace })`** is the async function — for `generateMetadata` (which is `async` and not a component), Server Actions outside the component tree, route handlers. Returns the same `t`. The student's mental model: `use*` in the render tree, `get*` outside it.
- **The `t.rich` API for embedded markup.** From Lesson 18.2.1's "Trans" preview: `t.rich('terms.line', { link: (chunks) => <Link href="/terms">{chunks}</Link> })` with the message `'By signing up, you agree to our <link>Terms</link>.'`. Returns a `ReactNode`. The senior reach for any string with embedded interactive elements; replaces the "split into three keys" anti-pattern.
- **`useFormatter` / `getFormatter` — the `Intl.*` wrapper.** Convenience hooks over `Intl.NumberFormat`, `Intl.DateTimeFormat`, `Intl.RelativeTimeFormat`. `format.number(amount, { style: 'currency', currency: 'USD' })`, `format.dateTime(instant, { dateStyle: 'long', timeZone })`, `format.relativeTime(instant, now)`. The locale and the request's tz are wired in automatically; the engineer passes the options. The senior framing: `useFormatter` is the daily reach inside next-intl's tree; standalone `Intl.*` (Lesson 18.2.3) is the form for code outside the tree (a utility module, a CLI script, a test fixture). Both call into the same engine.
- **The `Messages` global type — compile-time key safety.** next-intl 4 ships a TypeScript inference path: declare `interface AppMessages extends Messages {}` and `declare global { interface IntlMessages extends AppMessages {} }`, with `AppMessages` typed from the source-locale JSON (`typeof import('../messages/en.json')`). `t('invoice.greeting')` is typed; `t('invoice.greting')` (typo) is a compile error; missing values for placeholders are caught at the type level. The senior rule: enable the inference; without it, missing keys ship as `'invoice.greting'` strings to production. The build step that turns `en.json` into the type is in the tsconfig / next.config alignment; the chapter installs it.
- **Server Actions and route handlers — `getTranslations` and `getLocale`.** Both are async. `'use server'` action: `const t = await getTranslations({ locale: await getLocale(), namespace: 'inbox' });` resolves the locale from the request context, loads the namespace, returns the typed `t`. The same pattern in `route.ts`. The point: actions and routes are *outside* the React render tree, so the hooks don't apply; the `get*` async siblings cover them with the same surface.
- **The "every component goes through next-intl" rule.** Once next-intl is wired, every string the user reads goes through `t()` (with optional `t.rich`), every number / date / relative-time goes through `format.*`, every navigation goes through the typed `Link` / `redirect`. The audit grep: hard-coded JSX strings, `Date.prototype.toLocaleString`, `Intl.NumberFormat` instantiated outside `useFormatter` / `lib/format.ts`, `next/link` imported in `app/[locale]/...`. Each hit is a finding; the discipline is structural.
- **The static-vs-dynamic-rendering decision.** Public marketing pages → static + `setRequestLocale` + prerender per locale → CDN-cacheable, fast. Authenticated app pages → dynamic (auth lookup forces it anyway) → the request-config resolves the user's tz / locale per request. The rule: never call `headers()` / `cookies()` inside a public layout without intending to opt out of static. next-intl honors the boundary; the developer keeps it intact.
- **Watch-outs.** Forgetting `setRequestLocale` in a static-rendered page silently converts to dynamic — the perf regression is invisible without monitoring; importing from `next/link` instead of `lib/i18n/navigation` skips the locale-prefix rewrite — broken links to default-locale routes from non-default-locale pages; loading every locale's messages eagerly (`import en from '../messages/en.json'`) ships the full catalog to every user — use dynamic `import(`messages/${locale}.json`)`; the `NextIntlClientProvider` wrapping the entire app sends the *entire* catalog to the client — wrap only the subtrees that need it, and pass the slice via `messages={pick(messages, ['nav', 'header'])}`; nesting `getTranslations` calls inside a render loop is N async lookups — call once at the top, pass the `t` down; the message catalog growing large (10K+ keys) tanks build-time type generation — split into namespaces, lazy-load per route; relying on next-intl's runtime ICU parser for every render is fine at 2026 scale but hot paths can pre-compile via the optional SWC plugin / build step (named once).

What this lesson does not cover:

- The translation-key discipline — Lesson 18.2.1.
- ICU MessageFormat at depth — Lesson 18.2.2.
- The `Intl.*` engine — Lesson 18.2.3.
- Locale negotiation algorithm details — Lesson 18.2.4.
- `hreflang` and per-locale SEO — Lesson 18.2.6.
- Translation-management integration — out of scope; named once.
- next-intl's `formats` global (locale-default formatter presets) — named, deferred to project chapter 18.3.

## Lesson 18.2.6 — i18n SEO: `hreflang`, per-locale sitemaps, canonical URLs

Topics to cover:

- **The senior question.** A French user Googles "logiciel de facturation" from Paris; the SaaS has a French marketing page (`/fr-FR/billing`) and an English original (`/billing`). Without the right SEO wiring, Google shows the English page in the French SERP, the user bounces, and the team blames "the translation isn't ranking." The fix is structural: `hreflang` tags tell Google which URLs are which-language alternates; `alternates.canonical` declares the page's canonical URL (the locale-prefixed one, not the default); a per-locale sitemap entry surfaces each variant for crawl; Open Graph `og:locale` + `og:locale:alternate` carry the same signal for social sharers. The lesson lands the full surface — Next.js 16's `generateMetadata`, sitemap, OG image conventions — as a Pattern archetype: code block named for what it prevents (wrong-locale serving), the failure mode without it (organic French traffic landing on English content), and the structural form that makes the bug hard to write.
- **What `hreflang` is, precisely.** A `<link rel="alternate" hreflang="fr-FR" href="https://app.example.com/fr-FR/billing">` tag (one per alternate locale, including a self-reference) tells Google that the page has a French variant at that URL. Google reads the tag, indexes both pages, serves the right one based on the user's locale signals (browser locale, Google account region, search query language). Without `hreflang`, Google treats each locale as a separate page and may rank only the one with the most backlinks. With `hreflang`, all variants share authority and the right one ranks per query.
- **`hreflang` rules — the three that catch teams.** **(1)** Self-reference is mandatory — the French page lists itself as `hreflang="fr-FR"` alongside the English alternate. **(2)** Bidirectionality is mandatory — if `/fr-FR/billing` lists `/billing` as `hreflang="en-US"`, then `/billing` must list `/fr-FR/billing` as `hreflang="fr-FR"`. Google validates both sides; one-sided declarations are silently ignored. **(3)** The `x-default` tag (`hreflang="x-default"`) declares the fallback for users whose locale doesn't match any alternate — typically the English page or a locale-picker landing page. The chapter's senior reach: the default locale's URL is the `x-default`, declared on every page.
- **Next.js 16's `alternates.languages` metadata.** The framework wires `hreflang` through the `Metadata` object returned from `generateMetadata`. Shape: `alternates: { canonical: '/fr-FR/billing', languages: { 'en-US': '/billing', 'fr-FR': '/fr-FR/billing', 'de-DE': '/de-DE/billing', 'x-default': '/billing' } }`. Next.js generates the `<link rel="alternate" hreflang="...">` tags from this object. The senior pattern: a `generateAlternates(pathname)` helper in `lib/seo.ts` that, given the canonical path, produces the full languages map by mapping over `routing.locales`. Every `generateMetadata` calls it; missing pages are findings.
- **The canonical URL — the locale-prefixed one.** `alternates.canonical` declares the page's authoritative URL. The senior 2026 rule: **the canonical is the locale-specific URL, not the default**. A French page's canonical is `/fr-FR/billing`; an English page's canonical is `/billing` (or `/en-US/billing` under `localePrefix: 'always'`). The wrong reach — canonicalizing all locale variants to the default — tells Google "these are duplicates of English, only rank the English one" and kills the French page's discoverability. Each locale variant is its own canonical; `hreflang` declares the relationships; Google handles the rest.
- **Sitemap — index plus per-locale.** A sitemap index at `/sitemap.xml` points to per-locale sitemaps (`/sitemap-en-US.xml`, `/sitemap-fr-FR.xml`, ...). Each per-locale sitemap lists the URLs for that locale with their `<lastmod>` and `<priority>`. The senior reach with Next.js 16: a `sitemap.ts` route that returns the array of `{ url, lastModified, changeFrequency, priority, alternates: { languages: {...} } }`; Next.js's typed `MetadataRoute.Sitemap` supports `alternates.languages` directly, emitting `<xhtml:link>` tags inside each `<url>` entry — the alternate way of declaring `hreflang`, equivalent to the `<head>` tags, often easier for large sites. Submit `/sitemap.xml` to Google Search Console; the index resolves the per-locale variants.
- **`generateMetadata` — the daily reach.** Per-page, async, returns `Metadata`. The senior pattern, paraphrased: read the `locale` and route params from the `params` argument; call `await getTranslations({ locale, namespace: 'Metadata' })` to fetch the localized title, description, OG title; call `generateAlternates(pathname)` for the languages map; assemble the `Metadata` object with `title`, `description`, `alternates`, `openGraph`, `twitter`. The function is the single seam where i18n, SEO, and social cards converge. The lesson reads one canonical implementation and names the audit step: every page in `app/[locale]/` has a `generateMetadata` that calls both `getTranslations` and `generateAlternates`.
- **Open Graph `og:locale` and `og:locale:alternate`.** Facebook, LinkedIn, Slack, X read these tags for social sharing. `og:locale` is the page's locale in the underscore form (`'fr_FR'`, not `'fr-FR'`); `og:locale:alternate` is repeated per alternate. The Next.js `Metadata.openGraph.locale` and `Metadata.openGraph.alternateLocale` accept the underscore form directly. The senior reach: derive both from the same `routing.locales` constant via a `bcp47ToOgLocale(locale)` helper; one source of truth, two formats.
- **Locale-aware OG images.** A share card carries title, description, and an image. The image is locale-aware when it contains text — "Billing software" in English, "Logiciel de facturation" in French. Next.js's `ImageResponse` from `opengraph-image.tsx` accepts `params.locale` and renders per-locale text via `getTranslations`. The senior framing: per-locale OG images are a *strong* SEO/social signal for translated marketing pages, a weak one for behind-auth pages (the OG image rarely surfaces). The chapter's reach: per-locale OG images on the marketing site, default-locale OG image inside the app.
- **The `lang` attribute on the `<html>` element, restated.** From Lesson 18.2.4: `<html lang={locale}>` is the load-bearing accessibility / SEO signal. Search engines key on it; mismatches between `<html lang>` and `hreflang` produce ranking inconsistency. The middleware-resolved locale must drive both; the audit step is to read the rendered HTML for any deployed locale and confirm the `lang` attribute matches the URL prefix.
- **`robots.txt` and per-locale indexing.** The senior 2026 default: allow all locales to be indexed; the `hreflang` tags handle disambiguation. The carve-out: locales in soft-launch / beta can be `noindex` via the page's `metadata.robots = { index: false }`. Never block via `robots.txt` for indexing decisions — `robots.txt` blocks crawling, which prevents Google from seeing the page exists; for "don't rank this yet," use `noindex`.
- **The "untranslated content" rule.** A page whose translation hasn't shipped yet — the English source exists, French is empty. Two senior options. **(a)** Fall through to the source locale (`en-US`) and serve it under `/fr-FR/...`; do *not* list this URL as a French `hreflang` alternate because the content is English; Google sees duplicate content otherwise. **(b)** `noindex` the untranslated locale until the catalog ships. The chapter's senior default: (b) for marketing, (a) for app — marketing translation quality matters for SEO, app fallback prevents broken UX. Document the choice in `lib/i18n.ts` next to `SUPPORTED_LOCALES`.
- **The marketing site vs. the app, restated for SEO.** Marketing pages (`/`, `/pricing`, `/features`) need the full `hreflang` / sitemap / OG-image surface — they're the SEO target. Authenticated app pages (`/dashboard`, `/settings`) are `noindex` by default (`robots: { index: false }` in their layout's metadata); `hreflang` is unnecessary for them; the locale resolution still runs but the SEO surface is dark. The chapter installs the discipline on both because the audit applies to both; only the marketing surface ships the public `hreflang` / sitemap entries.
- **Search Console validation.** Submit the sitemap index. Read the "International Targeting" report. Fix `hreflang` errors (most common: missing return links, malformed BCP 47 tags, missing `x-default`). The senior reach: a CI smoke test (Unit 21.2.3) that fetches the rendered HTML for each locale and asserts the `<link rel="alternate">` tags match the expected set; the validation runs on every PR, not once a quarter from Search Console.
- **Watch-outs.** Canonicalizing all locales to the default locale kills the translated pages' rankings — each variant is its own canonical; missing the self-referencing `hreflang` causes Google to drop the relationship silently; one-sided `hreflang` (English page lists French, French doesn't list English) is silently ignored; using `'fr'` instead of `'fr-FR'` is technically valid BCP 47 but produces vague targeting — match the locale tag the URL uses; `og:locale` in hyphen form (`fr-FR` instead of `fr_FR`) is a frequent typo — the Next.js metadata typing won't catch it; serving the same content under different locale URLs without translating ("duplicate content") triggers Google's near-duplicate detection — translate or `noindex`; submitting per-locale sitemaps individually instead of the index works but loses authority signaling — submit the index; serving the locale-detection redirect to crawlers (`/` → `/fr/` based on geo-IP) prevents the crawler from indexing `/` — never redirect crawlers based on `Accept-Language` or geo; lazy-loading the OG image renderer at request time on every share is fine, but a custom `ImageResponse` that fetches translations per request adds latency — cache aggressively or pre-generate at build for the marketing surface.

## Lesson 18.2.7 — Chapter quiz

Top 10 topics to quiz:

- The translation-key discipline — keys in component code, ICU MessageFormat strings in per-locale catalogs, named interpolations the catalog can reorder; no string concatenation across translatable fragments; one string / one key (no globalized `common.*` reuse); flat namespaced dot-paths (2-3 levels); `t.rich` for embedded markup.
- ICU MessageFormat surface — `plural` with CLDR categories (`zero` / `one` / `two` / `few` / `many` / `other` plus `=0` / `=1` exact-match overrides); `selectordinal` for ordinals; `select` for free enums (gender, role, type); nested plural-inside-select for gendered counts; `other` is mandatory; `Intl.PluralRules` as the underlying engine; MF2 named as the path forward.
- The `Intl.*` family — `NumberFormat` (currency, percent, compact, units), `DateTimeFormat` (Temporal-native, mandatory `timeZone`, presets vs. component options, `formatRange`), `RelativeTimeFormat` (`numeric: 'auto'`, Temporal `since` / `until` pairing), `Collator` (sort/search with `sensitivity` and `numeric` options), `ListFormat`, `DisplayNames`; the "construct once, reuse" rule.
- Why `value.toLocaleString()` without arguments is the bug class — same shape as `Intl.DateTimeFormat()` no-argument trap (18.1.3); runtime-locale, runtime-tz, silently wrong on Vercel; the structural defense is the typed `formatX(value, { locale, timeZone, options })` wrapper.
- Locale negotiation chain — URL prefix → profile (`users.locale`) → cookie → `Accept-Language` best-match → default; BCP 47 `Lookup` algorithm via `@formatjs/intl-localematcher` (`Intl.LocaleMatcher` still TC39 Stage 1 in May 2026); geo-IP is *never* a primary signal; `users.locale` and `users.timeZone` are independent profile columns.
- The `<html lang>` / `Content-Language` / cookie / URL-prefix coherence — all driven by the middleware-resolved locale; `Accept-Language` is one signal, not the truth; locale cookie as session-by-default (next-intl 4) and only written on user override.
- next-intl wiring shape — `defineRouting` (locales, defaultLocale, `localePrefix: 'as-needed'`), `createMiddleware` in `proxy.ts`, `getRequestConfig` resolving messages + tz, `setRequestLocale` for static rendering, `generateStaticParams` per locale, `useTranslations` / `getTranslations`, `useFormatter` / `getFormatter`, typed `Link` / `redirect` from the navigation module, the `IntlMessages` global type for compile-time key safety.
- Static vs. dynamic rendering with next-intl — `setRequestLocale` is mandatory for static; missing it silently converts to dynamic; marketing static + per-locale prerender, app dynamic by auth requirement; client-provider scope is the smallest subtree, with `pick(messages, ...)` to limit the payload.
- `hreflang` and `alternates.languages` — self-reference plus bidirectional declarations, `x-default` for the fallback, canonical is the locale-specific URL not the default, `generateAlternates(pathname)` helper, per-locale sitemap entries with `MetadataRoute.Sitemap`'s `alternates.languages`, OG `og:locale` underscore form, locale-aware OG images via `opengraph-image.tsx` for the marketing surface.
- The audit grep set — hard-coded strings in JSX (`<>...</>`), `+` between `t()` calls or template literals with translated fragments, `Intl.X()` without arguments, `Date.prototype.toLocaleString` without options, `next/link` imports inside `app/[locale]/...`, `Accept-Language` / `navigator.language` reads outside middleware / the sign-up form, canonical pointing to the default locale, missing self-reference in `hreflang`, untranslated catalog keys in the source-locale build.
