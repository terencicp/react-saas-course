# Chapter 15.3 — Rate limiting and shared session-shaped data (Upstash)

## Chapter framing

Chapter 15.3 lands the second baseline the moment a SaaS leaves localhost: every abusable endpoint is rate-limited, and the limiter's state lives in a process-external store the entire fleet shares. By this point the student has Better Auth on email+password (Unit 9), Server Actions and route handlers as the canonical write seams (Unit 7), the cache discipline from 15.1 and 15.2, and webhooks (Unit 12). What's missing is the senior-shaped answer to three questions: which controls run before launch when there is no Redis yet, what flips the decision so `@upstash/ratelimit` becomes non-negotiable, and what's the sliding-window-on-Upstash shape the app uses for sign-in, sign-up, password reset, and public APIs. The chapter teaches rate limiting as a security baseline (foreshadowing SaaS pattern #12 in Unit 17.2) — a named seam at the action / route-handler boundary, per-IP and per-key strategies for the auth flows, RFC-shape `RateLimit-*` response headers, and the dual-keying rule that one attacker cannot lock out a victim by hammering their email. The output is the mental model and the API surface (`limiter.limit(key)`, the per-endpoint key strategy, the 429 response shape) that Chapter 15.4 wires onto the Unit 9 auth flows.

Threads that run through every lesson: rate limiting is the security baseline's checkbox most often missed because nothing breaks until someone abuses it — the senior reflex is to ship limiters with the auth flows from day one, not after the first incident; the threshold question that decides which tool runs is the audience and the abuse model — Vercel WAF for crude IP-level brute force at the edge, application-level Upstash for per-key (per-email, per-user, per-org) limits the WAF cannot see; Upstash Redis is the 2026 default because it is connectionless (HTTP/REST), free-tier capable, and works in both the Node runtime and edge contexts where TCP is unavailable, with sub-ms p50 in the same region; `@upstash/ratelimit`'s sliding-window algorithm is the default — token bucket and fixed window are named once with the cases that flip them; the limiter is declared once at module scope so the `@upstash/ratelimit` in-memory cache works across hot invocations; the auth-endpoint key strategy is per-IP **and** per-email (or per-username) sent as two independent limiter calls, both gates, because per-IP alone lets a botnet through and per-email alone is the lockout vector; every limited response carries `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset`, and `Retry-After` per the IETF draft so clients and tests can read the state; 429 bodies stay user-safe (no "wrong password" hint, no "we throttled this IP") and the operator log captures the truth; the limiter never fails closed silently — a Redis outage is a logged event and a senior call between fail-open with an alert vs. degraded mode, with this course defaulting to fail-open-and-alert on the auth path because locking users out of their own accounts is the worse failure. The chapter ships three teaching lessons plus a quiz, ordered by dependency: the threshold and the two tools, the Upstash primitives and the algorithms, the wiring on the auth surface, the quiz.

## Lesson 15.3.1 — When edge controls are enough and when Upstash earns its weight

Topics to cover:

- **The senior question.** A pre-launch demo running on Vercel's preview URL needs some protection against a crawler hammering the sign-up form, but the team has not provisioned Redis yet. A public-URL launch with email+password is a week away. Which controls run today, what flips the decision so application-level rate limiting becomes non-negotiable, and what does each tool actually catch that the other does not? The lesson lands the threshold and the layering.
- **The two layers — edge controls and application controls.** Edge controls (Vercel WAF rate limiting rules, Cloudflare's equivalent) sit in front of the application and see IPs, paths, and headers. Application controls (`@upstash/ratelimit` inside Server Actions and route handlers) sit after auth and see the user, the email, the org, and the request body. Each catches what the other cannot: the edge stops a single IP from crawling the whole site; the application stops one attacker from trying ten thousand passwords against one victim's email from ten thousand IPs. A 2026 SaaS runs both.
- **Pre-launch — edge controls (light) are enough.** Before the public URL ships, the demo lives behind a Vercel preview or a private link, the abuse surface is mostly bots and the occasional probe. Vercel WAF's rate-limiting rules — declared in the dashboard or via the Vercel Firewall SDK — cover the basics: per-IP request budgets on `/api/*`, on the sign-in / sign-up paths, on any costly route handler. The rules take effect without redeploy; the cost is zero on the free tier; the senior call is to ship them with the first preview, not to wait. The lesson surfaces what a 2026 WAF rule looks like (path + IP-based, log or deny) without writing the rule itself — the wiring is two clicks in the dashboard, the architectural point is "the edge is the cheap first layer."
- **The threshold that flips the decision — the public URL with email+password.** The moment the app is reachable at a custom domain with a sign-in form, application-level rate limiting on the auth endpoints is non-negotiable. The reason is the email-lockout / credential-stuffing math: per-IP limits at the WAF do not see the email, so an attacker rotating IPs hammers one victim's email indefinitely; a per-IP limit tight enough to stop them locks out the office NAT. The fix is two independent limits per request — per-IP **and** per-email — applied inside the action where both are visible. The lesson names the trigger as the public URL plus the auth surface, not vague "scale" or "growth."
- **Why Upstash Redis is the 2026 default for the application layer.** The state a sliding-window limiter needs (per-key counters with sub-second precision and TTLs) lives across serverless invocations, which rules out in-memory state. The store must be reachable from every runtime the app uses — Node serverless, edge functions, Trigger.dev workers. Upstash Redis ships as HTTP/REST so it works in edge contexts where Node TCP is unavailable, scales-to-zero (no idle cost), has a generous free tier sized for tens of thousands of operations per day, and is the 2026 industry default in the Vercel ecosystem (`@upstash/ratelimit` is published and maintained by Upstash; the Vercel Marketplace integration ships the env vars). The lesson names alternatives once — Cloudflare KV / D1, self-hosted Redis on Fly, Vercel KV — and the conditions to reach for them (existing infra, latency to a non-Vercel region). For a 2026 SaaS on this stack, Upstash is the default reach.
- **The three further reasons Upstash earns its weight.** Rate limiting is the trigger; once Upstash is in the stack the same Redis serves two other workloads cheaply. **(1)** Cross-process cache for values that are too small / too hot for the Next.js data cache and need eventual consistency across regions (feature-flag fan-out, ephemeral session-shaped data the dispatcher reads, computed metric counters). **(2)** Shared session-shaped data — single-use tokens (password reset, magic link, verification), short-lived rate counters for non-abuse cases (the "send email" cooldown the auth flow shows), CSRF state, captcha challenges. **(3)** A foothold for pub/sub when the dispatcher upgrades to coalesce (named, deferred to 13.1 + 14.1.4). The lesson surfaces all three so the student knows the Redis they spun up for rate limiting earns more than one use, but the *trigger* of the chapter is still the auth endpoints.
- **What Upstash is not.** Upstash is not a Postgres replacement (no relational data, no transactions across keys, no joins); it is not a durable queue (Trigger.dev owns that — 13.1); it is not the Next.js data cache (the cacheTag world owns that — 15.1/15.2). Reaching for Upstash to "speed up the database" is the wrong trigger — that work belongs in the index plan (6.4.1) or in the cached read (5.4.3, 15.1.1). The lesson names the negative space so the student is not tempted to over-reach.
- **The naming — Redis vs. Upstash vs. the SDKs.** Redis is the protocol and data structures. Upstash Redis is the managed service. `@upstash/redis` is the HTTP/REST client. `@upstash/ratelimit` is the rate-limiting library that uses `@upstash/redis` internally. The 2026 senior names each correctly because the npm packages map to those names and the docs refer to them precisely.
- **Provisioning — one shape the course commits to.** Create an Upstash Redis database via the Vercel Marketplace integration (which writes `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` to the project's env in all three environments — production, preview, development), or via the Upstash dashboard for non-Vercel deploys (manual env wiring). Region picked to match the primary Vercel region for the lowest p50. The lesson names the integration as the senior default; the project's starter assumes the env vars are present.
- **The "ship both layers" rule.** The pre-launch edge rule does not go away when Upstash lands. The WAF stays as the cheap outer ring (IP-level brute force, scrapers); the application limiters add the inner ring (per-email, per-user, per-org). Removing the WAF when the app limiter ships is a regression — they catch different shapes. The lesson names the layering as the steady-state architecture, not a transition phase.
- **The fail-mode question — what happens when Upstash is unreachable.** The auth endpoint's `limiter.limit(key)` call can throw or time out if Upstash is down. The senior call is between **fail-open** (allow the request, log loudly, alert on rate) and **fail-closed** (reject the request as if rate-limited). This course defaults to fail-open on the auth path because a Redis outage locking every user out of their account is worse than the brief abuse window. The lesson names the rule, names the trade-off, and notes that high-value endpoints (admin-only privileged actions, billing webhooks the customer cannot retry) may flip the default to fail-closed. The implementation is a `try/catch` around the `limit()` call with the policy decided in one helper, not at every call site.
- **Watch-outs:** shipping the app to the public URL with only the WAF rule and no application limiter leaves the lockout-by-email vector open — both layers ship together; treating the WAF rate-limit rule as enough for `/api/auth/*` is the most common mistake in this stack — the per-email check the action does is invisible to the WAF; provisioning Upstash in a region distant from the Vercel region adds 30-80ms per `limit()` call — co-locate; in-memory state for "lightweight" limiters in serverless functions appears to work in dev and collapses under any non-trivial traffic — every serverless invocation has its own memory; using the Upstash SDK with TCP-style clients (`ioredis`) in edge functions silently fails — the HTTP/REST client is the only one that works everywhere; depending on the WAF for application-shaped limits (per-user, per-org, per-feature) is asking the wrong layer — those keys exist only after auth.

What this lesson does not cover:

- The sliding-window-vs-token-bucket-vs-fixed-window algorithm decision — Lesson 15.3.2.
- The actual `@upstash/ratelimit` API surface and call shape — Lesson 15.3.2.
- The per-IP + per-email dual-keying on sign-in — Lesson 15.3.3.
- The Vercel Firewall SDK and writing WAF rules in code — out of scope, named once.
- Upstash as a cross-process cache for the cache layer — named here, owned by 15.1's "the Vercel data cache" scope; not implemented in this course.
- Self-hosted Redis on Fly / Railway / EC2 — named once, deferred.
- The security baseline audit pass — Chapter 17.2.

Estimated student time: 40 to 50 minutes. The chapter's threshold lesson — every later mechanic assumes the public-URL-plus-auth trigger has fired.

## Lesson 15.3.2 — Upstash Redis and `@upstash/ratelimit` primitives

Topics to cover:

- **The senior question.** Given an Upstash Redis database and `@upstash/ratelimit` installed, what's the API surface the app calls, which algorithm is the default and what flips it, where does the limiter live in the file tree, and what does a `limit(key)` call actually return that the route handler uses to shape the response? The lesson lands the minimum primitive vocabulary the student carries into the wiring lesson.
- **The Upstash Redis client — one line.** `import { Redis } from '@upstash/redis'; const redis = Redis.fromEnv();` reads `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` from the environment and returns a connectionless client. The client is safe to share across requests; no pool, no connect lifecycle. The lesson surfaces the call shape and the `fromEnv()` convention because the project's `env.ts` (Chapter 1.4.5) already validates the two variables.
- **`@upstash/ratelimit` — what it adds on top.** A thin wrapper that turns Redis into a sliding-window / token-bucket / fixed-window limiter with one `limit(key)` method. The library handles the counter math, the TTL, the in-memory caching layer (so a single hot serverless invocation doesn't hit Redis on every call), and the analytics opt-in. The student writes the configuration; the library writes the Lua.
- **Declaring a limiter — the shape.** A small file `lib/rate-limit.ts` exports one or more `Ratelimit` instances declared at module scope. Each instance is a small object: `new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(10, '1 m'), prefix: 'rl:signin', analytics: true })`. The fields: `redis` is the shared client, `limiter` picks the algorithm and the budget, `prefix` namespaces the keys in Redis (so `signin:` and `signup:` keys never collide), `analytics: true` populates the Upstash dashboard's per-key timeline. The lesson surfaces every field and what it means.
- **Module-scope declaration — the caching reason.** The library caches recent keys in process memory; declaring the limiter at module scope means the same instance is reused across hot invocations of the same serverless function, and the cache survives. Declaring it inside the handler creates a fresh limiter (and a cold cache) on every request, defeating the optimization. The lesson surfaces the rule and the small footgun.
- **The three algorithms — sliding window is the default.** **Sliding window** (`Ratelimit.slidingWindow(max, window)`) — weighted count across two adjacent fixed windows, smoothest experience, the chapter's default. **Token bucket** (`Ratelimit.tokenBucket(max, refillRate, refillInterval)`) — refills at a steady rate, allows bursts up to bucket size, reach when the workload is a bursty API where a steady drip is correct (a content-generation endpoint that hits an LLM). **Fixed window** (`Ratelimit.fixedWindow(max, window)`) — simpler, cheaper, less smooth (the "thundering minute boundary"), reach when the cost of a few extra requests at the boundary is acceptable. The lesson names the three, picks sliding window as the default for the auth surface, names the cases that flip to the other two.
- **The `limit(key)` return shape — what the handler reads.** `const { success, limit, remaining, reset, pending } = await ratelimit.limit(key)`. `success` is the boolean the handler branches on. `limit` is the configured budget. `remaining` is what's left in the current window. `reset` is the Unix timestamp (ms) when the window rolls over. `pending` is a promise the handler can `ctx.waitUntil()` (or `await`) to flush the analytics write without blocking the user response. The lesson surfaces every field and where each goes in the HTTP response.
- **The RFC-shape response headers — the IETF rate-limit draft.** Every limited response carries `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset` (delta-seconds), and on a 429 also `Retry-After`. The IETF draft (`draft-ietf-httpapi-ratelimit-headers`) is the 2026 convention; clients and load tests read it. The lesson surfaces the header set, the precedence rule (`Retry-After` takes precedence when both are present), and the senior reflex of writing them on every response, not only the 429s — clients can use the count on a successful response to back off proactively.
- **Key design — what goes into the string passed to `limit()`.** The key is the identifier the limiter counts under. For per-IP: the client IP from `request.headers.get('x-forwarded-for')` (Vercel sets it; the lesson notes the parse rule). For per-email: a normalized email (`email.trim().toLowerCase()`). For per-user: the authenticated user id. The key string does **not** need the `prefix` (the limiter prepends its own). Keep keys lowercased and length-bounded; never embed PII beyond the email; never embed secrets. The lesson surfaces the convention and the senior reflex of running normalization once (the `lib/keys.ts` helper) so two call sites can't accidentally pick different shapes for the same intent.
- **The cost model — what each `limit()` call costs.** One Upstash request per `limit()` call by default, near zero when the in-memory cache hits (the library decides). The free tier (10k commands/day at the time of writing) covers a small SaaS for free; the next tier is pay-per-100k. The lesson names the order-of-magnitude so the student doesn't fear adding limiters. The optimization to know: ephemeral cache mode (`ephemeralCache: new Map()`) cuts Redis calls on bursty hot keys; the analytics write is the other Redis cost and `pending` lets it not block the user response.
- **Latency floor — what to expect.** Same-region Upstash adds ~5-15ms p50 to a request, ~25-40ms p99. Cross-region adds 50-100ms. The lesson names the budget so the student knows what to look for in observability (Unit 20.1) and what to do about it (co-locate the regions). On an auth endpoint already paying for a database round-trip and a password hash, the limiter latency is small relative to the request budget.
- **`Deny` lists and timeout helpers — named once.** `@upstash/ratelimit` exposes `Ratelimit.deny()` for hard-coded blocklists (a known abuse IP, a sanctioned country) and per-call `timeout` config to bound the failure mode. Named for recognition; the course's auth surface relies on the limiter and on Better Auth's existing security primitives rather than a hand-rolled deny list. Reach for the deny list when a specific abuse signal needs an out-of-band block.
- **Multi-region replication — named once.** Upstash supports global replication for multi-region apps; the limiter reads from the nearest replica. The cost is eventual consistency on counts (a hot key in two regions can briefly exceed the budget). Named for recognition; the course's single-region deploy doesn't need it. Reach when the Vercel deploy is multi-region.
- **Watch-outs:** declaring the limiter inside the handler defeats the in-memory cache and makes every call hit Redis — module scope; reading `request.ip` without falling back to `x-forwarded-for` (or trusting the wrong header on a non-Vercel host) breaks per-IP limits in production — the lesson surfaces the parse helper; passing a non-normalized email to `limit()` lets the attacker bypass per-email limits by varying case — normalize once at the boundary; setting `analytics: false` saves the writes but blinds the Upstash dashboard for incident review — keep it on for the auth surface; the `pending` promise is fire-and-forget — `await`ing it on the request path adds the analytics write to the user response; forgetting `prefix` makes two limiters share the same Redis keyspace and corrupt each other's counts — every limiter gets a distinct prefix; an excessively long window with a tight budget creates a "stuck" limiter (one user hits it once an hour ago, still blocked) — pick windows in minutes for auth, not hours.

What this lesson does not cover:

- The per-IP-and-per-email dual-keying applied to sign-in — Lesson 15.3.3.
- The 429 response body shape and the user-safe error contract — Lesson 15.3.3.
- The fail-open fallback wired in code — Lesson 15.3.3.
- The Upstash dashboard tour and incident review workflow — out of scope; named once.
- Upstash Redis primitives beyond the limiter (`SETEX`, `INCR`, pub/sub) — out of scope; named when other chapters reach for them (single-use tokens are an example a senior would build on `SETEX`, deferred).
- Latency tuning per region and the multi-region edge case — out of scope.

Estimated student time: 45 to 55 minutes. The mechanics lesson — every primitive in 15.3.3 and the 15.4 project leans on this surface.

## Lesson 15.3.3 — Wiring `@upstash/ratelimit` on the auth endpoints

Topics to cover:

- **The senior question.** Three limiters need to land on the Better Auth surface: sign-in, sign-up, password reset. Sign-in needs per-IP **and** per-email so an attacker rotating IPs against one victim's email hits the per-email cap and the office NAT doesn't trip its limit on legitimate traffic. Sign-up and password reset need per-IP (and per-email where a target email is identifiable). Where in the request lifecycle do the calls go, what budgets and windows fit each endpoint, and how is the 429 response shaped so it carries the headers without leaking the truth to the user? The lesson lands the wiring pattern Chapter 15.4 implements end-to-end.
- **The three limiters — declared once in `lib/rate-limit.ts`.** Three `Ratelimit` instances at module scope, each with its own prefix and its own budget. Reference shape: `signInLimiter = new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(10, '1 m'), prefix: 'rl:signin', analytics: true })`; `signUpLimiter = new Ratelimit({ ..., limiter: Ratelimit.slidingWindow(5, '10 m'), prefix: 'rl:signup' })`; `resetLimiter = new Ratelimit({ ..., limiter: Ratelimit.slidingWindow(3, '15 m'), prefix: 'rl:reset' })`. The budgets are senior calls, not arbitrary — sign-in tolerates more attempts (legitimate typos), sign-up and reset are rarer events so tighter. Each project tunes from monitoring; the defaults are the starting point.
- **The dual-keying rule for sign-in — both gates must pass.** Inside the sign-in action / route handler, before Better Auth's verify call: `const ip = getClientIp(headers);` and `const email = normalizeEmail(input.email);`. Two limiter calls — `await signInLimiter.limit('ip:' + ip)` and `await signInLimiter.limit('email:' + email)` — and the request is rejected if **either** returns `success: false`. The per-IP gate stops crude single-source brute force; the per-email gate stops the credential-stuffing-via-botnet pattern. The composite key prefix (`ip:` vs. `email:`) keeps the two namespaces distinct under the same limiter's prefix. The lesson surfaces the order: cheaper check first (IP, no normalization), then per-email; both calls are awaited because both must pass.
- **Where the call lives — the seam inside the action.** The limiter runs **before** any expensive work (no database read, no password hash) and **before** Better Auth's handler runs. Putting it after the password verify defeats the purpose — the attacker has already paid the computation budget the limiter exists to cap. The canonical shape: `signInAction = async (input) => { const ip = ...; const email = ...; const ipLimit = await signInLimiter.limit('ip:'+ip); if (!ipLimit.success) return rateLimitResponse(ipLimit); const emailLimit = await signInLimiter.limit('email:'+email); if (!emailLimit.success) return rateLimitResponse(emailLimit); return betterAuth.signIn(input); }`. The lesson surfaces the order as the architectural rule: gate first, work second.
- **The `RateLimit-*` response headers — applied on every response, not only 429s.** A small helper `rateLimitHeaders(result)` returns the four headers (`RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset`, plus `Retry-After` on rejected responses). The successful sign-in response includes the headers from the per-IP limiter (the more informative budget for the client). The rejected response includes the headers from whichever limiter rejected. The lesson names the IETF draft as the standard, names the `Retry-After`-takes-precedence rule, and names the senior reflex: write the headers from the limiter result, do not hand-compute them.
- **The 429 response body — user-safe, operator-honest.** The body returns a stable, non-leaky message: `{ error: 'Too many attempts. Please try again later.' }`. It does not say "your IP is rate-limited" or "this email is locked" — both leak information to an attacker (the email message confirms the email exists in the system, even when sign-in normally doesn't). The operator log line captures the truth: `{ limiter: 'signin', key: 'email:user@example.com', remaining: 0, reset: ... }`. The user sees the same string regardless of which gate tripped. The lesson surfaces the contract: 429 stays opaque, structured logs (Unit 20.1) carry the diagnosis.
- **The per-IP-only sign-up and reset endpoints.** Sign-up rate limits per-IP because the email is the attacker's choice (no victim email to protect). Password reset limits per-IP **and** per-email (the email is the victim's; the per-email gate prevents an attacker from spamming a victim's inbox with reset emails). The reset endpoint has a tighter budget than sign-up because the cost of abuse (deliverability hit from Resend, user inbox noise — Chapter 8.1.4 owns the suppression story) is concrete. The lesson surfaces the three endpoints side-by-side with their key strategies.
- **Better Auth's built-in rate limiter — the integration call.** Better Auth ships an in-memory rate limiter on by default that protects all its endpoints with a coarse budget. The 2026 senior turns the built-in off (or configures the `secondary-storage` adapter pointing at the Upstash Redis client) and runs the application-level limiters described above instead, because (a) the in-memory limiter does not work across serverless invocations, (b) the budgets are not configurable per-endpoint with the per-key shape the auth surface needs, and (c) the limiter lives outside the Server Action seam so the team has one place to lint. The lesson names the choice: replace Better Auth's built-in with the `lib/rate-limit.ts` module, wire the limiters at the action boundary. The fallback path (use Better Auth's `secondary-storage` pointed at Upstash) is named for teams that want the framework to manage the limiter rules.
- **Reading the client IP — the parse helper.** A small `getClientIp(headers: Headers): string` lives in `lib/keys.ts`. On Vercel: read `x-forwarded-for`, take the **first** entry (the original client; subsequent entries are proxies), fall back to `x-real-ip`, fall back to `'unknown'`. Trust the header only because Vercel sets it; on a self-hosted deploy the trust boundary is the load balancer. The lesson surfaces the parse, surfaces the trust boundary, and names the senior reflex of not trusting `x-forwarded-for` on platforms that don't strip the client-supplied value.
- **Email normalization — the parse helper, restated.** `normalizeEmail(email: string): string` trims, lowercases, optionally strips `+` aliases for the limit key (a senior call — `+` aliases on Gmail are the same mailbox and a target attacker can vary the alias to bypass the per-email gate; aggressive normalization risks treating distinct addresses on `+`-supporting providers as the same key). The course's default: trim and lowercase; do not strip `+`. The lesson surfaces the decision, names the trade-off, and notes that the normalization used for the limiter key must match the normalization used for the database lookup so the limiter and the lookup count the same identifier.
- **The fail-open fallback — what happens when `limit()` throws.** A `try/catch` around the `limit()` call. The catch logs a structured event (`{ event: 'rate_limit_unavailable', limiter, key }`) at error level so monitoring (Unit 20.1) alerts, and returns `{ success: true, ... }` so the request proceeds. The course's default is fail-open on the auth path; the lesson surfaces the helper `safeLimit(limiter, key)` that wraps the call and the policy decision, so the choice is one place to flip if the team wants fail-closed for higher-stakes endpoints. The lesson names the operator's reflex: a sustained `rate_limit_unavailable` alert is a Upstash incident, not a quiet drift.
- **The `pending` promise and `waitUntil` — non-blocking analytics.** Each `limit()` call returns a `pending` promise that flushes analytics to Upstash. On Vercel Functions, the senior pattern is `event.waitUntil(result.pending)` (Node runtime) so the user response returns before the analytics write completes. The lesson surfaces the integration, names the speedup (5-10ms shaved off the user-visible response), and notes that the analytics write is best-effort by design.
- **The dashboards — what the operator watches.** Upstash dashboard's analytics tab shows per-prefix rate over time, top keys, and reject rates. The senior watches: reject rate on `rl:signin` (a sustained spike is an attack or a buggy client), top keys on `rl:reset` (one email dominating means a victim is being targeted — escalate), and the Redis op count (capacity planning). The lesson names the surfaces, defers the alerting wiring to Unit 20.1.
- **Worked walkthrough — sign-in end-to-end.** Show the action's source flow: parse the input, extract `ip` and `normalizedEmail`, call `safeLimit(signInLimiter, 'ip:'+ip)` then `safeLimit(signInLimiter, 'email:'+normalizedEmail)`, return the 429 with headers if either fails, call Better Auth's `signIn`, attach the rate-limit headers to the success response. The walkthrough is the chapter's reference pattern; 15.4 implements it for real.
- **The fan-out for other abusable surfaces — named for recognition.** Public APIs (when they exist), webhook receivers (Chapter 12.1 owns idempotency; rate limiting at the receiver protects against burst-amplification attacks on the downstream system), file uploads (Unit 13.3), AI generation endpoints (Unit 23). Each follows the same shape: module-scope limiter, key strategy named, dual-keying when a victim identifier is involved, RFC-shape headers, fail-open. The lesson names the pattern as the rest of the course's reach point.
- **Watch-outs:** running the limiter after the password hash or after the database lookup pays the cost the limiter is meant to cap — gate first; checking `success` on only one of the two sign-in calls leaves the unchecked vector open — both gates, every request; trusting `x-forwarded-for` on a non-Vercel host or behind a misconfigured proxy lets the attacker spoof their IP — gate the trust at the platform boundary; returning a different 429 body for "IP exceeded" vs. "email exceeded" leaks which gate tripped — keep the body identical; setting the per-email budget tighter than the per-IP budget on sign-in lets an attacker abuse the office NAT to lock out a victim — keep per-email looser than per-IP (or comparable) and let the per-email cap protect against the stuffing pattern, not against the office; missing `Retry-After` on a 429 means clients (and tests) cannot back off correctly — write all four headers; fail-closed on the auth path under a Redis outage takes the whole user base offline — fail-open with alerts is the senior call here.

What this lesson does not cover:

- The verify recipe — running 11 requests, checking headers, observing the dashboard — Chapter 15.4.5.
- The Better Auth integration code in detail — Chapter 9 owns the action shape; this lesson surfaces the seam.
- The structured-log line shape and the alerting rule — Unit 20.1.

## Lesson 15.3.4 — Chapter quiz

Top 10 topics to quiz:

- The two layers of rate limiting and what each catches — edge (Vercel WAF) for IP-shaped abuse at the perimeter, application (`@upstash/ratelimit`) for per-key (per-email, per-user, per-org) limits the WAF cannot see; the senior reflex is to ship both.
- The threshold that flips application-level rate limiting from optional to non-negotiable — the moment the app is reachable at a public URL with email+password.
- Why Upstash Redis is the 2026 default for the application limiter — connectionless HTTP/REST works in Node and edge runtimes, scale-to-zero free tier, sub-ms p50 in-region, library maintained alongside the SDK; the three further workloads it earns (cross-process cache, shared session-shaped tokens, dispatcher pub/sub later) are upside.
- The three `@upstash/ratelimit` algorithms — sliding window as the chapter's default (smoothest experience), token bucket for bursty APIs, fixed window for cheap-and-acceptable boundary noise.
- The module-scope declaration rule for limiters — declared once in `lib/rate-limit.ts` so the in-memory cache survives across hot invocations; declaring inside the handler defeats the optimization.
- The `limit(key)` return shape — `{ success, limit, remaining, reset, pending }` and what each field maps to in the HTTP response; `pending` flushed via `waitUntil` to keep analytics off the user response path.
- The dual-keying rule on sign-in — per-IP **and** per-email as two independent gates, both must pass, because per-IP alone misses credential stuffing across IPs and per-email alone is the lockout vector; both call paths use the same limiter with distinct key prefixes (`ip:` vs. `email:`).
- The RFC-shape response headers — `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset`, and `Retry-After` on rejections — written on every response (success or 429); `Retry-After` takes precedence per the IETF draft when both are present.
- The 429 body contract — user-safe, identical regardless of which gate tripped, never leaks "your email" or "your IP"; the structured operator log captures the truth.
- The fail-open policy on the auth path — `try/catch` around `limit()`, log the `rate_limit_unavailable` event, allow the request; the alternative (fail-closed) takes the user base offline on a Redis incident and is the wrong default for sign-in / sign-up / reset.
