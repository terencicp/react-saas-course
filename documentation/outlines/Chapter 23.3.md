# Chapter 23.3 — Tools, agents, and generative UI

## Chapter framing

Chapter 23.2 wired the four AI SDK Core primitives (`streamText`, `generateText`, `generateObject`, `streamObject`) and the three UI hooks (`useChat`, `useCompletion`, `useObject`) into the Next.js 16 route-handler seam, with `UIMessage` parts as the rendering source of truth. Chapter 23.3 is where the LLM stops being a string generator and becomes a controller that calls into the app's own functions: it queries the database, hits internal APIs, returns typed objects to render as custom React components, and runs across multiple steps until the task is done. The chapter teaches the three pieces that compose that capability — tool calling and the agentic loop, generative UI via tool parts, and embeddings for retrieval — at the senior posture: each piece is a sharp knife with a clear trigger, not a default reach.

Threads through every lesson: **tools are the LLM's only safe doorway into the app's state** — every database read or write the model triggers passes through a tool whose `execute` runs server-side under the same `authedRoute` + org-scope + quota stack as any other handler, never through the model itself or through a string the model emits; **the agentic loop is the v5 `stopWhen` shape** — `stepCountIs(n)`, `hasToolCall(name)`, and custom predicates replace v4's client-side `maxSteps`, and the senior reflex is to set an explicit step cap on every multi-step call (the cost-cap discipline from 23.1.2 generalized to step count); **Zod is the single contract across the chapter** — `inputSchema` on each tool (renamed from v4's `parameters`), `outputSchema` for type-safe client rendering, the same Zod 4 discipline as 23.2.2's structured output and the form schemas from Chapter 7.1; **generative UI is the tool-parts rendering pattern, not `ai/rsc`** — the model emits typed tool calls, the route handler streams them as `tool-<name>` parts on the `UIMessage`, the client switches on `part.type` and `part.state` to render bespoke React components; `streamUI` and `ai/rsc` are named once as the experimental alternative and not taught; **embeddings are a conditional reach** — RAG earns its weight only when the corpus is large enough that prompt-stuffing breaks, and Drizzle + pgvector is the default path for SaaS that already runs Postgres; **everything composes with prior chapters** — tools call Drizzle queries from Unit 8, ride on the authz primitives from Unit 10, and write audit events from 23.1.2 inside `onFinish` and `onStepFinish`.

## Lesson 23.3.1 — Tool calling and the agentic loop

Topics to cover:

- **The senior question.** Once the model needs to do more than transform text — answer a question grounded in the user's data, take an action, or chain a sequence of operations — the model itself can't reach into the app. Tools are how. The lesson is the `tool` helper, the v5 `inputSchema` / `outputSchema` contract, the `execute` function and where it runs, the `stopWhen` agentic loop and its built-in predicates, and the senior posture that the model is a planner and the tool is the only doorway to state.
- **What a tool is — the three-part definition.** A tool is `tool({ description, inputSchema, execute })`. The `description` tells the model when to reach for it (the most under-rated field — a vague description means the model picks the wrong tool or no tool); `inputSchema` is a Zod schema the SDK serializes to JSON Schema and sends to the model, and the SDK validates the model's tool-call arguments against it before `execute` runs; `execute` is the async function that returns the tool's result. v5 renamed `parameters` to `inputSchema` and added optional `outputSchema` for end-to-end typing; name both renames once.
- **Where `execute` runs — the trust boundary.** Every `execute` runs server-side in the same Next.js 16 route handler that called `streamText`. The senior consequence is that tools sit inside the `authedRoute` wrapper from 10.2.3 and have access to the session, the org scope, and the audit logger — the model can ask the tool to query invoices, but it is the tool's code (not the model) that checks `session.orgId` against the requested `invoiceId`. The model never sees a database connection, an API key, or a row it isn't allowed to see; the tool projects the safe slice. Name this rule explicitly; it's the load-bearing one.
- **The minimal tool — inline shape.** A short example: `getInvoiceById = tool({ description: 'Look up an invoice by ID for the current organization', inputSchema: z.object({ invoiceId: z.string().uuid() }), execute: async ({ invoiceId }) => db.query.invoices.findFirst({ where: and(eq(invoices.id, invoiceId), eq(invoices.orgId, session.orgId)) }) })`. The shape is the lesson: schema in, server-side query out, org-scope enforced inside `execute`. Show the call site passing `tools: { getInvoiceById }` to `streamText`.
- **Tool calls in the `UIMessage` parts protocol.** When the model invokes a tool, the stream emits a `tool-<name>` part on the assistant message with three lifecycle states: `input-streaming` (arguments arriving), `input-available` (arguments parsed, `execute` running), `output-available` (result returned), plus `output-error`. The state transitions live in the same parts array the lesson 23.2.3 introduced. The client can render each state — a loading spinner during `input-available`, the result during `output-available`. Name the states; the actual custom rendering is 23.3.2.
- **The agentic loop — `stopWhen` and step caps.** A multi-step call sends the prompt to the model, the model emits a tool call, the SDK runs `execute`, appends the result as a tool-result message, and asks the model again — looping until the model produces text without a tool call or a stop condition fires. v5 made this a server-side concern: `streamText({ tools, stopWhen: stepCountIs(5), messages })`. Without `stopWhen`, the SDK defaults to `stepCountIs(20)`; with a single step, the model can't react to the tool's result and the answer never reads the data. The senior cut: `stepCountIs(5)` for most workloads, `stepCountIs(2)` for one-tool-plus-summary patterns, `stepCountIs(10)` only when the workload is genuinely multi-tool. v5 deliberately moved this off the client (v4's `maxSteps`); the route handler owns the cap.
- **Built-in stop conditions — `stepCountIs`, `hasToolCall`, custom predicates.** `stepCountIs(n)` is the default reach: a hard ceiling on steps. `hasToolCall('finish')` is the explicit-completion pattern — define a `finish` tool whose only job is to signal "I'm done," and the loop stops when the model calls it; useful when the workload has a clear termination state. Custom predicates `({ steps }) => boolean` let the senior cap by token usage or by elapsed time. Name all three; the default is `stepCountIs`.
- **`onStepFinish` — the per-step audit and quota seam.** `streamText({ onStepFinish: ({ usage, toolCalls, toolResults, finishReason }) => ... })` fires after each step in the loop, with the usage and the tool calls/results for that step. The senior reflex: this is where per-step audit events land (`llm.step.completed` with the tool name and arg shape), where the rolling token counter is incremented against the user's quota mid-loop, and where a runaway loop can be detected and stopped (a 5-step loop that's already burned 50k tokens is a bug). `onFinish` from 23.2.1 still fires once at the end with the aggregate; the two callbacks compose.
- **`prepareStep` — adapting the call mid-loop.** `prepareStep({ stepNumber, steps }) => Partial<CallSettings>` lets the route handler change settings between steps: swap to a cheaper model for follow-up steps, narrow the tool set after the first step, compress the conversation history when it grows past a threshold. The senior trigger is real but narrow: reach for `prepareStep` only when a single static call shape genuinely doesn't fit (the common cases are "first step plans with a smart model, follow-ups execute with a fast one" and "after the database is queried, drop the query tools to prevent re-querying"). Name the hook; do not over-teach it.
- **Tool-result shape and feeding it back to the model.** The model sees tool results as JSON. The senior reflex is to project, not to dump — a tool that returns a 200-row query result blows the input context on the next step. Return the minimal shape the model needs to answer (totals, top-N, a derived summary), not the raw row. The `outputSchema` is the contract that locks this discipline at the type level. Cross-link to 23.1.2's cost discipline — input tokens compound across loop steps.
- **Error handling inside `execute` — return, don't throw.** When the tool's underlying work fails (database error, permission denied, malformed argument), the senior pattern is to return a typed error result the model can read and react to, not to throw. A thrown error breaks the stream and the user sees a 500; a returned `{ error: 'invoice_not_found' }` lets the model say "I couldn't find that invoice, can you check the ID?" Wrap database calls and external fetches in try/catch inside `execute`; let only programmer errors bubble. Name the pattern; show the shape.
- **Tool choice — `toolChoice: 'auto' | 'required' | 'none' | { type: 'tool', toolName }`.** The default `'auto'` lets the model decide; `'required'` forces a tool call on the first step (useful when the workload demands tool grounding and free-text would be a regression); `'none'` disables tools for this call. Named tool choice forces a specific tool. The senior reach is `'auto'` by default; `'required'` for "must ground in data" surfaces; the rest is rare.
- **Composing tools with the prior authz stack — the worked posture.** Every tool in this course's app sits inside the same handler `authedRoute('member', schema, fn)` wraps. The `session` is in scope inside the handler; the tool's `execute` closes over it. The org-scope filter (`eq(invoices.orgId, session.orgId)`) goes in every tool that touches a tenant-owned table; the audit-event write goes in `onStepFinish`. The senior reflex: an unauthenticated tool is a bug class.
- **A short diagram (mermaid) — the agentic loop.** User message → route handler → `streamText({ tools, stopWhen })` → model decides (text or tool call) → if tool call: SDK validates input against `inputSchema` → `execute` runs server-side → result fed back as tool-result message → loop continues → eventually model returns text → `onFinish` → `toUIMessageStreamResponse`. The diagram makes the loop's seams and the tool's server-side residence visible.
- **Watch-outs:** v4 `parameters` instead of v5 `inputSchema` — renamed, type errors on first compile; client-side `maxSteps` from v4 — replaced by server-side `stopWhen`; omitting `stopWhen` and getting the default 20-step ceiling — burns budget on workloads that should cap lower; a tool's `execute` that queries without org-scope — cross-tenant leak, the classic Unit 10 bug class moved into AI; throwing instead of returning errors from `execute` — kills the stream, bad UX; returning huge raw results from `execute` — input-token blowout on the next loop step; a vague `description` field — the model picks the wrong tool or no tool; templating the tool name from user input — never; a tool that performs a destructive action (delete, charge) without a confirmation step — design the surface so destructive tools require a UI-side confirmation before the next agent step.

What this lesson does not cover:

- Rendering tool parts as custom React components (the generative UI pattern) — Lesson 23.3.2.
- Embeddings, vector stores, and RAG — Lesson 23.3.3.
- Multi-modal tools (file/image inputs) — out of scope.
- MCP (Model Context Protocol) servers and dynamic tools — out of scope; named in the SDK but rare in 2026 SaaS.
- Human-in-the-loop confirmation flows — named as a watch-out only.
- The worked invoice Q&A surface — Chapter 23.4.

Estimated student time: 50 to 60 minutes. The chapter's heaviest lesson; the tool primitive carries the rest of the chapter.

## Lesson 23.3.2 — Generative UI via tool parts

Topics to cover:

- **The senior question.** The model has just called a tool that returned a structured invoice summary, or a chart-ready data shape, or a calendar slot. Rendering that as JSON in a chat bubble is wrong; rendering it as a bespoke React component (invoice card, chart, slot picker) is the lesson. How does the client know which component to render, with which props, in which state? The lesson is generative UI built the canonical 2026 way: tool parts on the `UIMessage`, switched on `part.type` and `part.state` in the client, rendering typed components with the tool's `outputSchema` as the props contract.
- **Why this path, not `ai/rsc`.** The TOC names `streamUI` / `useUIState` / `useAIState` from the `ai/rsc` package as a generative UI approach. In 2026 that package is still marked experimental in the AI SDK docs and the team's documented recommendation is to use AI SDK UI (`useChat` + tool parts) for production. The senior call follows that recommendation: the tool-parts path composes with everything 23.2.3 already taught (the same `UIMessage`, the same `useChat`, the same route handler), keeps the rendering on the client where React 19's primitives already live, and survives the `ai/rsc` API churning. Name `ai/rsc` as the experimental alternative, name the senior reach, and move on.
- **The tool-parts rendering pattern — the canonical shape.** Every assistant message is `{ id, role, parts: UIMessagePart[] }`. When the model calls `getInvoiceById`, the assistant message gets a part with `type: 'tool-getInvoiceById'` that moves through `state: 'input-streaming' | 'input-available' | 'output-available' | 'output-error'`. The client's render is `message.parts.map((part) => switch (part.type) { case 'tool-getInvoiceById': return <InvoiceCard state={part.state} input={part.input} output={part.output} /> })`. The model's tool-call decision drives the component choice; the tool's `output` is the component's data prop. Show the switch inline; this is the load-bearing snippet.
- **The three states the client renders — UX per state.** `input-streaming` is the brief moment the model's argument tokens arrive; the senior surface usually skips a render here. `input-available` is "tool is running" — render a tool-specific skeleton (`InvoiceCard.Skeleton`, not a generic spinner). `output-available` is the real render. `output-error` shows a sanitized failure state (the same UX discipline from 23.2.3 — never render the raw error string). The pattern names per-tool skeletons; a chat with five tools shows five different skeletons, not one generic shimmer.
- **End-to-end typing — `outputSchema` and the typed `UIMessage`.** v5 lets the student type the chat: `useChat<MyUIMessage>(...)` where `MyUIMessage` is the union of tool parts derived from `InferUITools<typeof tools>`. The result is that `part.output` is correctly typed as the Zod schema's inferred shape, not `unknown`. The senior reflex: define the tool registry once in `lib/llm/tools.ts`, infer the message type, pass it to `useChat` — auto-complete on every tool part. Name the helper; show the inferred type briefly.
- **Where the tool registry lives — the file convention.** Tools live in `lib/llm/tools.ts` (mirrors `lib/llm/models.ts` from 23.1.3 and `lib/llm/prompts.ts` from 23.2.1). The route handler imports `import { tools } from '@/lib/llm/tools'` and passes `tools` to `streamText`. The client component imports the inferred message type from the same module. The senior reflex: tools are first-class app code, version-controlled, lint-covered, the same as routes; no inline definitions in route handlers.
- **The data-flow pattern — tool returns app-shape, component renders app-shape.** The tool's `outputSchema` is the same shape the React component takes as props. The senior cut is to design the tool's output to match the component's prop contract, not to reshape inside the component. An `getInvoiceById` tool that returns `{ id, customerName, total, dueDate, status }` feeds `<InvoiceCard />` directly; an `getMonthlyRevenue` tool returns `{ months, totals }` shaped for `<RevenueChart />`. The tool and the component are co-designed; the schema is the contract.
- **Streaming partial tool output — when `output` is large.** For long-running tools (a multi-row report, a paginated query), the tool's `execute` can yield partial results via the v5 generator shape. The client sees `output` populate progressively, like a streaming object. Mention the capability; the default is to return once and design the schema small.
- **Side-effecting tools and the confirmation pattern.** When the tool's effect is destructive (sending an invoice, deleting a row, charging a card), the senior pattern is to split the action into two tools: `proposeInvoiceSend` returns a preview shape the client renders as a confirmation card with explicit buttons; `confirmInvoiceSend` does the actual work and is gated on the client-side confirmation (the client only sends the second tool call after the user clicks). The model never confirms destructive actions on its own; the human is the loop closer. Name the pattern; the worked example lands in 23.4 if relevant.
- **Composability — tool parts and text parts in the same message.** A single assistant message can contain text parts and multiple tool parts in render order. The client's render walks the array; the model's narration sits between tool renders ("Here's the invoice you asked about. Want me to send a reminder?"). The senior UX rule: tools render where the model said they should, not all at the bottom; the parts array order is meaningful.
- **Where this composes with the auth and audit stack.** Tools authorize inside `execute` (23.3.1's rule). Audit events still write inside `onStepFinish`. The Server Component shell still loads `initialMessages`. Persistence still saves `UIMessage[]` (which now includes tool parts and their typed outputs) on `onFinish`. None of those seams change because tool parts are added — they ride the same protocol.
- **A short diagram (mermaid or component-tree SVG) — the tool-part lifecycle.** Model emits tool call → stream part `type: 'tool-X', state: 'input-streaming'` → arguments parsed → `state: 'input-available'`, `execute` runs server-side → result returned → `state: 'output-available'`, `output` on the part → client switches on `part.type`, picks the React component, passes typed props. The diagram threads the server boundary (execute) and the client boundary (component switch).
- **Watch-outs:** reaching for `ai/rsc` and `streamUI` because they're in the docs — experimental, the senior path is tool parts; rendering tool parts only on `output-available` and skipping the skeleton — feels unresponsive on slow tools; per-tool skeletons replaced with a single generic spinner — loses the affordance the tool-parts model affords; defining tools inline in the route handler — breaks the typed-message inference; reshaping the tool output inside the component instead of in the tool — the schema is the contract, the component shouldn't know its source's quirks; destructive tools without a confirmation pattern — the model will eventually delete the wrong thing; not persisting tool parts when saving messages — the next mount loses the rendered surface; rendering raw JSON for unknown tool types — always have a default case that fails gracefully.

What this lesson does not cover:

- The `ai/rsc` package and `streamUI` / `useUIState` / `useAIState` — named once as the experimental path, not taught.
- Custom data parts (non-tool typed parts) — named in passing.
- Reasoning-trace rendering — out of scope.
- Multi-modal generative UI (image generation tools rendering image parts) — out of scope.
- The worked surface against the invoice data — Chapter 23.4.

Estimated student time: 35 to 45 minutes. A mechanics-and-decision lesson — short because it leans on 23.3.1's tools and 23.2.3's parts.

## Lesson 23.3.3 — Embeddings and retrieval-augmented features

Topics to cover:

- **The senior question.** When does an LLM-backed surface need retrieval — pulling relevant snippets from a corpus and feeding them as context — versus simply prompting the model with the user's question and trusting its training? The lesson is the threshold for RAG, the AI SDK's two embedding primitives (`embed`, `embedMany`), `cosineSimilarity`, and the default 2026 storage path for SaaS that already runs Postgres (pgvector via Drizzle). The depth is "enough to ship a 2026 SaaS RAG feature competently," not a vector-database deep dive.
- **The trigger — when retrieval earns its weight.** Senior threshold: the corpus is internal (the model can't have trained on it), and either large enough that the full corpus does not fit in the context window, or small but updates fast enough that recency matters. Internal docs, customer-specific knowledge bases, the user's own data corpus, transcripts. Below the threshold — a 5-page FAQ — paste it in the system prompt; do not build a vector store. Above — the user's 10,000 invoices, the company's 200-page handbook — RAG is the path. The cost discipline from 23.1.2 reinforces the cut: prompt-stuffing N tokens of context on every call burns budget; retrieval pays the embedding cost once and the per-query cost is bounded.
- **Embeddings in one paragraph — the senior-level mental model.** An embedding is a fixed-length vector of floats produced by an embedding model; texts with similar meaning map to vectors with high cosine similarity. The senior need not internalize the geometry — they need to know that embedding the corpus once and embedding each query at retrieval time, then finding nearest neighbors, is how "semantic search" works. Name the model dimension as a number to budget around (e.g., 1536 for text-embedding-3-small) — it sizes the storage. Do not teach the math; teach the shape.
- **`embed` and `embedMany` — the SDK's primitives.** `embed({ model: openai.embedding('text-embedding-3-small'), value: 'one string' })` returns `{ embedding, usage }`. `embedMany({ model, values: ['a', 'b', 'c'] })` returns `{ embeddings, usage }` and is the batch entry point — the senior reach for indexing the corpus. Both pass through the same `lib/llm/models.ts` registry from 23.1.3 (a separate named export, `embeddingModel`). Show the inline shape; the corpus-indexing job is a one-shot async function, not a route handler.
- **Storing vectors — pgvector + Drizzle as the 2026 default.** The course already runs Postgres with Drizzle (Unit 8). pgvector is a Postgres extension that adds a `vector` column type and similarity operators (`<->` for L2, `<=>` for cosine). Drizzle has a `vector()` column helper. The schema shape: a `document_chunks` table with `id`, `documentId`, `content text`, `embedding vector(1536)`, plus a vector index (`ivfflat` or `hnsw`) on `embedding`. Show the Drizzle column definition inline; do not show the index DDL — name it once. The senior reflex is to stay on the same database the app already runs; reach for a dedicated vector DB (Pinecone, Upstash Vector, Qdrant) only when the corpus exceeds what pgvector handles at the team's scale (tens of millions of vectors) or when the workload genuinely needs a managed service.
- **The two-phase pipeline — index and query.** Indexing: chunk the document, `embedMany` the chunks, insert rows with the embeddings into `document_chunks`. Query: `embed` the user's question, `select` the nearest K chunks from Postgres ordered by `embedding <=> queryEmbedding`, pass those chunks to `streamText` as part of the system prompt. The senior cut: chunking is a problem domain (paragraph chunks for prose, fixed-token chunks for code, sentence chunks for transcripts); the SDK doesn't ship a chunker — `@langchain/text-splitters` or a hand-rolled splitter is the call. Name the choice; don't get lost in chunker pros and cons.
- **`cosineSimilarity` — the in-memory utility for small corpora.** `cosineSimilarity(a, b)` returns a number between 0 and 1. For corpora small enough to hold in app memory (a few hundred chunks), brute-force comparison is fine: load all embeddings, score each, pick top-K. Above that scale the database does it. Name the function; the senior reaches for it only when the workload is genuinely small or when scoring two specific vectors outside the retrieval flow.
- **The RAG query pattern in a route handler — the worked shape.** A short example: in `app/api/chat/route.ts`, before calling `streamText`, embed the user's question, run a Drizzle query that joins `document_chunks` on cosine similarity and returns the top 5 chunks, format them into a `relevantContext` string, and inject it into the system prompt as `Relevant context:\n${relevantContext}`. The user's question stays in `messages` (untrusted data); the retrieved context goes in the system prompt (trusted because the retrieval was authorized server-side). Cross-link to 23.2.1's prompt-injection rule — the system prompt remains the controller, retrieval just enriches it.
- **Retrieval as a tool — the agentic alternative.** Instead of always retrieving before the call, define a `searchKnowledgeBase` tool that the model decides to call when the question requires retrieval. The tool's `execute` does the embed-and-query; the result feeds back through the agentic loop from 23.3.1. The senior cut: pre-retrieval is the default for chat surfaces where every turn likely needs context; tool-based retrieval is the call for mixed surfaces where some questions are general and some need lookup. Both compose; pick the one that matches the workload.
- **Authorization on retrieval — the multi-tenant constraint.** Every `document_chunks` row carries an `orgId` (or `userId`), and every retrieval query filters by `session.orgId` (Unit 10's discipline applied to the vector table). A retrieval that returns another org's chunks is the worst cross-tenant leak shape — the model will then quote them in the answer. Name the rule; the watch-out is non-negotiable.
- **Re-embedding and corpus updates — the operational reality.** Embedding models change; switching from `text-embedding-3-small` to a newer model means re-embedding the whole corpus. The senior tracks the model identifier on each row (`embedding_model text`) so a rolling re-index is possible. Document updates trigger a chunk-level re-embed; deletes cascade to chunks. Name the discipline; do not teach the migration mechanics.
- **A short diagram (mermaid) — the RAG flow.** Documents in → chunker → `embedMany` → pgvector table. User question → `embed` → Drizzle similarity query → top-K chunks → system prompt enrichment → `streamText` → answer. The diagram shows the two phases and the seam.
- **Watch-outs:** building RAG for a corpus that fits in the system prompt — over-engineering, just paste it; using a separate vector database when pgvector handles the scale — extra moving part, extra cost, extra failure mode; embedding without an `orgId` filter — cross-tenant leak via retrieved chunks; forgetting to re-embed when the embedding model changes — dimension mismatch or silently degraded retrieval; chunk sizes too large (full pages) — retrieval returns mostly-irrelevant text and the answer drifts; chunk sizes too small (one sentence) — retrieval loses surrounding context; not deduplicating chunks at insert time — the same passage appears in top-K multiple times; mixing pre-retrieval and tool-based retrieval in the same surface without thinking through which fires first — the model may try to call `searchKnowledgeBase` when context is already in the prompt; trusting retrieved content as authoritative — RAG grounds answers, it does not validate them; running embedding on every keystroke in a search box — debounce or batch.

What this lesson does not cover:

- The math of embeddings, dimensionality reduction, or the geometry of vector spaces — out of scope.
- Production vector index tuning (HNSW parameters, IVF lists) — out of scope.
- Hybrid retrieval (BM25 + vector) and re-ranking — named as more advanced; the chapter teaches the default path.
- Managed vector databases (Pinecone, Upstash Vector, Qdrant) — named once as alternatives.
- Multi-modal embeddings (image, audio) — out of scope.
- Long-context-window models as a replacement for retrieval — named in the threshold discussion; the cut still favors retrieval at scale.

Estimated student time: 40 to 50 minutes. A decision-heavy lesson with one worked path.
