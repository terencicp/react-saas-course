# Chapter 12.1 — Webhook ingestion

## Chapter framing

Chapter 12.1 turns the route-handler boundary (Chapter 7.5) into a hardened webhook ingestion seam — the public POST endpoint a third party (Stripe, Resend, GitHub) calls and that mutates the database on the strength of a signed payload. The student arrives with route handlers, the canonical Server Action Result and transaction shape from Chapter 7, the `Idempotency-Key` HTTP semantic from 3.2.1, Web Crypto from 3.7.1 (HMAC sign/verify, constant-time compare), the `email_suppressions` table from 8.1.4, and a working Drizzle + Postgres + Server Action stack. The chapter teaches the four disciplines every webhook handler needs the moment it touches production: verify the signature against the raw body in constant time, dedupe replays through a unique-keyed `processed_events` table, wrap the dedup-and-business work in a single outer transaction so partial state never lands, and decide deliberately about out-of-order delivery and the redirect-versus-webhook race the user creates the moment they bounce back from a payment provider. The chapter then names idempotency as the unifying discipline (the same unique-key constraint that dedupes a webhook also dedupes a double-clicked Server Action and a retried background job), and applies the whole shape to a second case — Resend bounce and complaint webhooks populating `email_suppressions` — so the pattern transfers cleanly.

Threads that run through every lesson: a webhook endpoint is a public POST whose only authentication is a signature, treat the payload as untrusted until verified, treat the body as raw bytes until verified, and never rely on the framework's JSON parser sitting between the bytes and the HMAC; signature verification must be constant-time and must include a timestamp tolerance to bound replay; the `event.id` from the provider is the natural idempotency key — store it in a table with a `UNIQUE` constraint, the `INSERT ... ON CONFLICT DO NOTHING` is both the check and the claim; the dedup insert and the business mutation share one transaction so a crash between them is impossible; the handler returns 2xx fast after persisting and lets retries handle the rest — long work goes to a background job (named, deferred to Chapter 13); out-of-order events are a property of the network, not a bug, the handler reconciles using monotonic provider timestamps or by always reading the canonical state from the provider's API; the redirect-from-checkout running before the webhook is a guaranteed race, the senior call is "the webhook is the only writer, the redirect renders pending state and refreshes"; Stripe is the worked example, but the shape — verify, dedupe, transact, ack, reconcile — is provider-agnostic; the chapter ships five teaching lessons plus a quiz.

## Lesson 12.1.1 — Signature verification at the route-handler boundary

Topics to cover:

- **The senior question.** A POST arrives at `/api/webhooks/stripe` with a JSON body and a `Stripe-Signature` header. Anyone on the internet can hit this URL — there are no cookies, no session, no Better Auth wrapper. The only thing standing between the attacker's forged payload and a database mutation is whether the HMAC in the header matches what your endpoint secret produces over the raw body, and whether you compare those two strings without leaking timing information. The lesson lands signature verification as the first line of every webhook handler the course writes, with the four mechanics that have to be right or the verification is theater: raw body bytes, the HMAC scheme, the timestamp tolerance, the constant-time compare.
- **The route-handler shape — POST, raw bytes, no body parser.** In Next.js 16's App Router, `/app/api/webhooks/stripe/route.ts` exports `async function POST(req: Request)`. Read the body with `await req.text()` (or `await req.arrayBuffer()` for binary payloads) — never `await req.json()`. The framework does not body-parse a `Request` automatically (unlike older Pages-API routes), but a hand-rolled JSON parse anywhere upstream of the HMAC breaks the signature: `JSON.stringify(JSON.parse(body))` re-orders keys, drops whitespace, and produces a different byte sequence than what was signed. The 2026 reflex: raw `text()` first, then the signature check, then the parse.
- **The signature header anatomy — `t=<ts>,v1=<hmac>`.** Stripe's `Stripe-Signature` header is a comma-separated list of `key=value` pairs: `t=` is the timestamp the event was signed at (Unix seconds), `v1=` is the HMAC-SHA256 of `${t}.${rawBody}` keyed with the endpoint secret. The verifier parses the header, computes the expected HMAC, and compares. The lesson names the format once so the student recognizes the shape when other providers (GitHub uses `X-Hub-Signature-256: sha256=<hex>`, Resend uses a Svix-flavored `webhook-signature` header) ship a near-identical pattern. The course reaches for the provider SDK's verifier where it exists (`stripe.webhooks.constructEvent`), and falls back to a hand-rolled Web Crypto verifier where it doesn't.
- **Closing the Web Crypto thread from 3.7.1.** Chapter 3.7.1 taught `crypto.subtle.sign('HMAC', ...)` and the constant-time comparison helper. This lesson is the load-bearing call site that thread foreshadowed. The hand-rolled verifier the lesson shows reaches for `crypto.subtle.importKey` to load the endpoint secret as an `HMAC`-typed key, `crypto.subtle.sign` to compute the digest over `${t}.${body}`, and `crypto.timingSafeEqual` (Node) or a manual constant-time loop over `Uint8Array`s for the comparison. The student writes verification once by hand to see the mechanics, then uses `stripe.webhooks.constructEvent` for the Stripe case because the SDK gets the edges right.
- **Why constant-time comparison matters here.** A `===` string compare short-circuits on the first mismatched byte. An attacker who can submit thousands of requests per second and measure response timing can — in theory — narrow the correct signature byte-by-byte. The mitigation is comparing every byte regardless of mismatch. The senior anchor: this is the single timing-attack-mitigation pattern in the course, named in 3.7.1, exercised here. Don't ship a webhook handler that uses `===` on the signature even when the rest of the stack looks correct.
- **The timestamp tolerance — bounded replay window.** Stripe rejects events whose `t=` is more than five minutes old by default (`tolerance: 300` in `constructEvent`). The reasoning: a captured signed request stays valid forever without a tolerance, even after the endpoint secret is correct, because the signature is over the body and the timestamp — neither changes. With a tolerance, an attacker who copied a request from your logs has at most five minutes to replay it. The lesson names the default, names the trade-off (clocks must be reasonably synchronized; NTP on the server is enough), and names when to widen it (testing with a captured payload from yesterday — never in production).
- **The 400-on-failure contract.** When the signature is invalid (header malformed, HMAC mismatch, timestamp outside tolerance), the handler returns `400 Bad Request` with an RFC 9457 Problem Details body (Chapter 3.2.2). It does not return 401 — the request has no caller identity to authenticate, the issue is that the payload is malformed. It does not return 200 — that signals success to the provider and prevents the retry that would deliver a legitimately re-signed event. It does not throw past the handler — `error.tsx` is not the audience, the webhook caller is. The senior anchor: webhook errors return small, sanitized, RFC 9457 bodies the operator can read in the provider's webhook dashboard.
- **The "verify before anything else" rule.** No logging of the body, no JSON parse, no database lookup, no telemetry until the signature passes. The reasons: logging an unverified body persists attacker-controlled bytes to logs (and to log-aggregator quotas), parsing JSON before the HMAC is the same `JSON.stringify(JSON.parse(body))` bug above, hitting the database on every unauthenticated POST is a free DOS amplifier. The 2026 reflex: the handler's first three lines are `read body, read header, verify`. Everything else lives in the success branch.
- **Where the endpoint secret lives — env, not config.** `STRIPE_WEBHOOK_SECRET` is an environment variable, separate from `STRIPE_SECRET_KEY`, scoped per endpoint (Stripe lets you register multiple webhook endpoints, each with its own secret). The lesson names the env entry the project chapter (12.3) wires; the rotation discipline (Chapter 17.2 owns the broader secrets story) is named once here.
- **Local development — `stripe listen` as the local loop.** A webhook handler is unreachable from `https://api.stripe.com` while it runs on `localhost`. The senior fix is the Stripe CLI's `stripe listen --forward-to http://localhost:3000/api/webhooks/stripe`, which opens a long-lived connection to Stripe, receives every event your test account fires, and POSTs it to the local URL. The CLI prints a *separate* signing secret (`whsec_...` for the listener) the local app uses for verification — different from the production endpoint's secret. The two-secret reality (one for `stripe listen`, one for the deployed endpoint) is the failure mode developers stumble on: the handler returns 400 in dev because the production secret is in `.env`. The lesson names the convention: `STRIPE_WEBHOOK_SECRET` in `.env.local` is the listener's secret, the deployed endpoint's secret lives in the platform's environment.
- **`stripe trigger` for replay-style testing.** Once `stripe listen` is forwarding, `stripe trigger checkout.session.completed` (or `payment_intent.succeeded`, etc.) fires a canned event of that type into the listener, which forwards it to the handler with a real signature. The student can re-trigger the same event type as many times as needed to exercise the dedup path (lesson 12.1.2). The senior anchor: a webhook handler that has never been exercised twice from `stripe trigger` is not done.
- **What the handler does after a successful verify.** Parse the body (`JSON.parse(rawBody)` is now safe — the HMAC proved the bytes), narrow by `event.type`, route to the per-type handler that owns the business logic. Lesson 12.1.2 picks up here with the dedup table; this lesson stops at "parsed and ready to dispatch."
- **The `event` shape — what a Stripe event carries.** The parsed event is `{ id: 'evt_...', type: 'checkout.session.completed', created: <unix>, data: { object: <the resource> } }`. The lesson names the fields the rest of the chapter reaches for: `event.id` (the dedup key, 12.1.2), `event.type` (the dispatcher), `event.created` (the ordering anchor, 12.1.3), `event.data.object` (the payload the handler reads). The student does not need the full Stripe TypeScript type — the SDK's `Stripe.Event` narrows by `type` and the handler's `switch` cases pick up the concrete `object` shape from there.
- **Worked example — the verification skeleton.** A `/app/api/webhooks/stripe/route.ts` POST that reads the body, calls `stripe.webhooks.constructEvent(rawBody, signature, secret)` inside a try/catch, returns 400 with a Problem Details body on failure, dispatches by `event.type` on success. The dispatcher's body is a comment (`// dedup, transact, handle — next lessons`) so the file compiles and runs end-to-end. The student fires `stripe trigger payment_intent.succeeded` and sees the dispatch log without writing anything to the database yet.
- **Watch-outs.** Calling `await req.json()` instead of `await req.text()` corrupts the body for signature verification — the framework's helpers re-stringify; using `===` to compare the computed HMAC and the header value is the timing leak — reach for `timingSafeEqual` or the SDK's verifier; setting the route to `export const dynamic = 'force-static'` or any caching directive on a webhook handler is the silent footgun — webhook routes are dynamic by definition, and a cached 200 from a stale handler makes the provider stop retrying; importing the Stripe SDK at the top of a non-webhook server file ships the SDK into every request bundle — keep webhook code in `/app/api/webhooks/*`; logging the raw body before verification persists attacker payloads to log aggregators; the same endpoint URL receiving events from `stripe listen` *and* a deployed test endpoint at the same time (because both are registered against the same Stripe test mode) double-delivers events to whichever is listening — turn off the deployed test endpoint while running `stripe listen` locally; setting a different `tolerance` in `constructEvent` without naming the reason is a smell — the default is correct unless you're replaying captured payloads in CI.

What this lesson does not cover:

- The dedup table and the `INSERT ... ON CONFLICT DO NOTHING` pattern (12.1.2).
- The outer transaction wrapping dedup plus the business mutation (12.1.2).
- Out-of-order events and the redirect race (12.1.3).
- Resend signature verification specifically (12.1.5 — same shape, named there).
- Secret rotation and the platform-level secrets manager (Chapter 17.2).
- Background-jobs offload for slow handlers (Chapter 13.1, named once).
- Long-term replay of historical events from Stripe's event log (out of scope; named).

Estimated student time: 50 to 60 minutes. The lesson is dense because the verification edges are exactly the edges where production webhook handlers fail, and they all have to be right at once.

## Lesson 12.1.2 — Dedup and the outer transaction

Topics to cover:

- **The senior question.** The signature verified. The event is `checkout.session.completed`, the handler is about to flip a `subscription_status` row and grant plan entitlements. Stripe guarantees *at-least-once* delivery — the same `event.id` will arrive again if your handler is slow, returns non-2xx, or the network blips. Without dedup, every retry runs the business logic again: the second retry of `invoice.paid` charges the user a second time in your audit log, the second retry of `customer.subscription.created` issues a second welcome email, the second retry of any side-effect-bearing handler is the production-grade pager incident. The lesson lands the unique-keyed `processed_events` table as the structural dedup, the `INSERT ... ON CONFLICT DO NOTHING` reflex, and the outer transaction that binds dedup and business work together so a crash between them cannot leave the database in a partial state.
- **Why provider retries exist and what they look like.** Stripe retries non-2xx responses with exponential backoff for up to 72 hours (the documented window; the student verifies by reading the dashboard's retry log on a deliberately-failed event). Retries are not a bug to suppress — they are the reliability mechanism that lets the network fail without losing events. The handler's job is to be safely retryable: idempotent on `event.id`, fast enough that the provider's timeout doesn't trip the retry on a successful run, honest about failure (return 5xx, let it retry; don't return 200 after a partial mutation).
- **The `processed_events` table — schema and indexes.** A Drizzle table with two columns the dedup cares about: `eventId: text('event_id').notNull()` and `processedAt: timestamp('processed_at', { withTimezone: true }).defaultNow().notNull()`. A `UNIQUE` constraint on `(provider, eventId)` (or just `eventId` if you keep one table per provider — the lesson picks the multi-provider shape because Resend webhooks land in 12.1.5). A `provider` column (`'stripe' | 'resend'`) as a `text` enum. The senior call: keep the table minimal — no body, no status, no retries column — the unique constraint *is* the dedup mechanism, anything else is logging that belongs in 17.2's audit table.
- **The `INSERT ... ON CONFLICT DO NOTHING` reflex.** The Drizzle shape: `await tx.insert(processedEvents).values({ provider: 'stripe', eventId: event.id }).onConflictDoNothing()`. The senior anchor: the insert is *both* the check and the claim — no read-then-write race, no `SELECT WHERE event_id = ...` step that could pass two concurrent invocations. If two retries of the same event arrive at the same instant, the database's unique constraint serializes them; one INSERT lands, the other returns zero rows affected.
- **Reading the "was it new" signal.** Drizzle's `.returning()` after `onConflictDoNothing` yields the inserted row on success and an empty array on conflict. The handler branches: if the array is empty, the event has already been processed, return 200 immediately (the provider should stop retrying). If the array has one row, this is the first time, fall through to the business work. The senior reach: return 200 on the duplicate path so the provider does not retry forever on an event the handler legitimately decided to drop.
- **The outer transaction — dedup and business work as one unit.** The dedup INSERT and the business UPDATE (granting plan entitlements, writing the subscription row, etc.) live inside a single `db.transaction(async (tx) => { ... })`. The reason is not idempotency-against-retry — that's the unique constraint's job — but atomicity against crash: if the dedup INSERT lands and then the process dies before the business UPDATE commits, the next retry sees the dedup row and skips the work, and the user never gets their entitlement. Wrapping both in one transaction ties their fates: both commit or both roll back, the next retry sees a clean slate.
- **The exact shape.**

  ```
  await db.transaction(async (tx) => {
    const inserted = await tx.insert(processedEvents)
      .values({ provider: 'stripe', eventId: event.id })
      .onConflictDoNothing()
      .returning();
    if (inserted.length === 0) return; // already processed, commit empty txn
    await handleEvent(tx, event);
  });
  return new Response(null, { status: 200 });
  ```

  The duplicate path returns from the callback without doing work; the transaction commits empty (no-op). The senior anchor: the handler's outer shape is the same for every event type — verify, transact, dispatch — the per-type handler is what varies inside `handleEvent`.
- **Why the per-type handler takes `tx` and not `db`.** Passing the transaction handle into the business handler means every write inside `handleEvent` participates in the outer transaction. A `handleEvent` that imports `db` directly bypasses the transaction — its writes commit independently, and the failure mode reappears. The convention from Chapter 7.2.5 (transactions take `tx`, the rest of the codebase takes `db`) holds here.
- **No external side effects inside the transaction — the 7.2.5 rule, restated.** Sending a Stripe API call, firing a Resend email, or dispatching a Trigger.dev job from inside the transaction is the production landmine: the external call lands, the transaction rolls back, the external state and the database diverge. The senior reach: collect the side-effect descriptors during the transaction (a list of jobs to dispatch, a list of emails to send), commit, then fire them after. For webhooks specifically, the simpler reach is to let the next webhook (or the next user action) trigger the side effect — the database is the source of truth, and downstream effects re-derive from it. Long-running work goes to a background job (Chapter 13), named once.
- **What "the business work" looks like in this chapter.** For Stripe: update the org's `subscription_status`, write or update the `plan_entitlements` row (the derived view from 12.2.4), record an audit log entry (Chapter 17.2.3 owns the audit table; the lesson names the slot). For Resend: insert into `email_suppressions` (8.1.4). The lesson keeps the worked example small — a `subscription_status` update — so the dedup-and-transaction shape is the point, not the entitlement computation (12.2 owns that).
- **The "happy path is 2xx, otherwise 5xx" response contract.** The handler returns:
  - 200 on first-time success (the work committed).
  - 200 on duplicate (the unique constraint caught the replay).
  - 400 on signature failure (12.1.1).
  - 5xx on any unhandled exception inside the transaction (let it propagate — the provider retries, the next attempt either succeeds or hits the dedup).
  Never 200 on a half-finished write. Never 4xx on a transient error — the provider will not retry a 400 or 422.
- **The retention question — how long to keep dedup rows.** Stripe's retry window is 72 hours; Resend's is shorter. A daily job (Chapter 13) deletes rows older than seven days from `processed_events`. Without retention, the table grows without bound (an event fires every few seconds at scale); the unique-constraint index is what carries the read cost, and indexes on growing tables get expensive. The senior anchor: name retention at the schema-design moment, not after the first slow query. The retention helper itself lives in the job (Chapter 13.1); this lesson names the slot.
- **The "row in `processed_events` is the audit, not the table itself" distinction.** The dedup table records that an `event.id` was seen and processed. It does *not* record what the event did — that's the audit log's job, and the audit log carries the foreign key to the dedup row. The senior anchor: one table per concern. A `processed_events` table with a `payload` column and a `status` column has grown into the audit log; refactor to two tables before that happens.
- **The "what if the per-type handler is missing?" path.** `event.type` arrives with a value the handler doesn't know — Stripe adds new event types regularly. The dispatcher returns 200 (the event was received and acknowledged, no business work needed), logs the unknown type at info level (so the operator can decide whether to add a handler), and the dedup row records the receipt. The senior anchor: unknown event types are not errors — they're new product surface the codebase hasn't subscribed to. Returning 5xx for unknown types causes Stripe to retry forever on events you'll never handle.
- **Worked example — the assembled handler.** The full POST function from 12.1.1, now with the transaction-wrapped dedup, a `dispatch(tx, event)` function that switches on `event.type` and handles two cases (`checkout.session.completed`, `customer.subscription.updated`), and the rest of the cases logged-and-acked. The student fires `stripe trigger checkout.session.completed` twice in a row from the CLI and sees the first call mutate the database, the second call return 200 without mutating.
- **Watch-outs.** A handler that does the dedup INSERT outside the transaction and the business work inside is the original failure mode in disguise — the dedup commits independently, a rollback of the business work leaves a "processed" event that did nothing; reading the dedup row before inserting (the `SELECT WHERE eventId = ... THEN INSERT` pattern) is the classic race that the `ON CONFLICT DO NOTHING` reflex prevents — never SELECT-then-INSERT for deduplication; logging the event payload from inside the duplicate branch is fine, logging from inside the business branch should happen *after* the transaction commits (so the log reflects committed state); the per-type handler that calls `revalidatePath` is fine (it's just cache invalidation; nothing rolls back) but calling `revalidatePath` inside the transaction reads as a side effect from the caller's perspective — the senior reach is to revalidate after the transaction commits; assuming the provider's retry strategy is forever leads to silent data loss when the provider gives up after 72 hours and the handler had been returning 5xx the whole time — monitor 5xx rates on webhook endpoints (Chapter 20.1); a unique index on `eventId` alone (without `provider`) breaks when two providers happen to use overlapping ID formats — keep the composite key even with one provider; deleting from `processed_events` while a retry of an old event is in flight loses the dedup — pick a retention window longer than the longest provider's retry window.

What this lesson does not cover:

- Signature verification (12.1.1, prior).
- Out-of-order events and the redirect race (12.1.3, next).
- The full event taxonomy for Stripe (`checkout.session.completed`, `customer.subscription.*`, `invoice.*`) — Chapter 12.2 owns the billing event vocabulary.
- The retention job's implementation (Chapter 13.1).
- The audit log table schema and the link from `processed_events` to audit rows (Chapter 17.2.3).
- Background-job offload for slow per-type handlers (Chapter 13.1).

Estimated student time: 45 to 55 minutes. Pattern archetype. The transaction-plus-unique-constraint pair is the load-bearing mechanism the rest of the chapter and Unit 12 leans on.

## Lesson 12.1.3 — Out-of-order events and the redirect-versus-webhook race

Topics to cover:

- **The senior question.** Stripe sends `customer.subscription.created` and `customer.subscription.updated` in the same instant when a user starts a trial. They arrive in your handler in the opposite order — the `updated` event with `status: trialing` lands first, then the `created` event with `status: incomplete`. The naive handler writes whichever lands last; the user's row ends up in `incomplete` and the UI says "please complete your purchase" five seconds after they did. Separately: the user completes Stripe Checkout and is redirected to `/billing/success`. The redirect runs in 300ms; the `checkout.session.completed` webhook usually lands first, but not always. The success page reads the database, finds no entitlement yet, and tells the user the purchase failed. The lesson lands the two failure modes — out-of-order events and the redirect-versus-webhook race — and the senior responses: order by the provider's monotonic timestamp (or always re-read from the provider's API), and "the webhook is the only writer, the success page polls."
- **Out-of-order events as a property of distributed delivery.** Stripe explicitly does not guarantee ordering — events fan out through their delivery infrastructure, retries reorder relative to first-attempt delivery, and a 5xx on the first attempt can cause the redelivery to arrive after later events were already processed. The provider's documentation names this; the senior reflex is to design every handler under the assumption that events arrive in any order. The naive "the last event wins" is a bug that hides until production traffic exposes it.
- **The `event.created` timestamp as the ordering anchor.** Every Stripe event carries `created` (Unix seconds, set when the event was generated at Stripe). The handler can compare `event.created` against a `last_event_at` column on the entity being mutated and skip writes from older events. The shape:

  ```
  UPDATE subscriptions
    SET status = $newStatus, last_event_at = $eventCreated
    WHERE id = $subscriptionId
      AND last_event_at < $eventCreated;
  ```

  Zero rows affected means a newer event already won; one row means this event was authoritative. The senior anchor: the entity row owns its own ordering high-water mark, the column is `last_event_at` (not `updated_at` — they may diverge if non-webhook updates happen), and the comparison is in the `WHERE` clause not in application code (no read-modify-write race).
- **The "re-read the provider's API" alternative.** Instead of trusting the event payload, the handler fetches the canonical state from Stripe: `stripe.subscriptions.retrieve(subscriptionId)`. The fetched record reflects current state, not the (possibly stale) event payload. The trade-off: an extra round-trip per webhook, an extra dependency on Stripe's API uptime, and a tighter rate-limit budget. The senior call: re-read when the event is high-stakes and the payload is partial (subscription state transitions, invoice paid/failed), use the payload directly when the event is self-contained and idempotent (a `customer.created` you just persist). For the course default: re-read on `customer.subscription.*` events because the trial/active/past-due/cancelled transitions are the bug factory; trust the payload on `customer.created` and `checkout.session.completed`'s simpler shape.
- **The "ignore some events, derive from others" reach.** Many SaaS billing integrations subscribe to a small set of authoritative events and derive everything else. Example: subscribe only to `customer.subscription.updated` (or `customer.subscription.created`) plus `invoice.payment_succeeded` and `invoice.payment_failed`; ignore the dozens of other subscription-lifecycle events because their state is reflected in the next `updated`. The senior anchor: fewer event types in the dispatcher means fewer ordering interactions to reason about. Chapter 12.2 owns the per-event taxonomy; this lesson names the discipline.
- **The redirect-versus-webhook race, precisely.** A user clicks "Subscribe" on the pricing page, goes to Stripe Checkout, pays, and Stripe redirects them to `/billing/success?session_id=cs_...`. Stripe also fires `checkout.session.completed` to the webhook endpoint at roughly the same moment. The redirect is a browser navigation, takes a TLS round-trip plus a 300 OK response from your app server; the webhook is a server-to-server POST from Stripe's infrastructure to your app. Both arrive within hundreds of milliseconds of each other, in either order. The success page reads the database immediately; the webhook is what wrote the entitlement row. The race: success page reads before webhook writes, user sees "purchase failed."
- **The senior response — "webhook is the only writer."** The success page does not call `stripe.checkout.sessions.retrieve(sessionId)` and write the entitlement itself. Two writers (the success page and the webhook) racing on the same row is the production bug the entire chapter is trying to prevent — both run the business work, the dedup table doesn't help because they have different "event IDs," and the entitlement row ends up half-written from one path and half-from the other. The course default: the webhook is the sole writer, the success page is read-only.
- **The success page's UX — pending state with polling.** The success page renders a "Confirming your purchase..." state, polls a server endpoint (or refreshes the Server Component, see below) every second or two, and renders the success state as soon as the entitlement row appears. After 10-15 seconds without confirmation, it falls back to "Your purchase is being processed — check your email" and a manual refresh button. The senior anchor: the redirect is a user-experience touchpoint, not a state-mutation seam.
- **The Server Component refresh pattern in Next.js 16.** The success page is a Server Component that reads the entitlement row. The client-side wrapper calls `router.refresh()` on a `setInterval` until the row appears, at which point the server re-renders with the success state and the interval clears. No new endpoint needed; the existing Server Component query is the read. The senior reach: keep the read in the Server Component, don't invent a separate "status check" route handler.
- **The "what if the webhook takes too long" failover.** If the entitlement row hasn't appeared after, say, 30 seconds, the success page can fall back to fetching the canonical state from Stripe and writing the entitlement itself — but this is an explicit, named fallback path, not the happy path. The fallback writes the same `event.id` through the same `processedEvents` dedup so the eventual webhook arrival is a no-op. The senior anchor: even the fallback uses the same idempotency mechanism — there's one writer pattern, one dedup table, two callers.
- **Out-of-order with the redirect race — the compound case.** The user pays, the success page reads (before the webhook lands), the user reloads the success page (it polls), in between the `checkout.session.completed` and a later `customer.subscription.updated` both arrive, the second one's `last_event_at` is higher, the page reflects the latest state correctly. The compound shape works because the writer is one (the webhook handler), the dedup is one (`processedEvents`), the ordering is one (`last_event_at` on the entity). The lesson assembles the three threads into a single mental model: one writer, dedup on event ID, order on event timestamp.
- **The Stripe Workbench `created` ordering for replays.** When using `stripe trigger` or replaying historical events from the dashboard, the `created` timestamps reflect when the original event fired, not when the replay was triggered. The `last_event_at` comparison still works — replaying an old event of a type that has since been superseded will write zero rows (the entity's high-water mark is already past the replay's `created`). The lesson names this as the "replay is safe, even out-of-order" property the design buys for free.
- **Worked example — the subscription update handler.** A `handleSubscriptionUpdated` function inside the dispatcher from 12.1.2. The handler optionally re-reads `stripe.subscriptions.retrieve`, then runs the conditional UPDATE with the `last_event_at < $created` predicate. The success page, a Server Component that reads `plan_entitlements`, with a Client Component wrapper that calls `router.refresh()` on an interval until the row appears. The student runs `stripe trigger checkout.session.completed` and watches the success page transition from pending to confirmed.
- **Watch-outs.** Writing entitlements from both the webhook and the success page is the canonical billing race — pick one writer, document the choice; comparing `event.created` against the row's `updated_at` instead of a dedicated `last_event_at` causes non-webhook updates (an admin tool, a manual SQL fix) to make later webhooks no-ops — keep the columns separate; trusting the payload for state-transition events (`customer.subscription.updated`) skips the re-read and exposes the out-of-order bug — re-read on transitions, trust the payload on simple creates; polling the success page every 100ms hammers the database — one to two seconds is the right cadence, with a circuit-breaker after ~15s; long-polling implementations on Server Components break the streaming model — stick to `router.refresh()` on `setInterval`; the fallback that writes from the success page without going through the same dedup table reintroduces the two-writer bug; subscribing to every event type Stripe fires multiplies the ordering reasoning surface — name the small authoritative set in 12.2 and dispatch on those.

What this lesson does not cover:

- The full Stripe event taxonomy (`checkout.session.*`, `customer.subscription.*`, `invoice.*`) — Chapter 12.2 owns the per-event mapping.
- The `plan_entitlements` derived view schema — Chapter 12.2.4.
- The Server Component / Client Component split for the pending-state pattern — Chapter 5.2 owns the boundary; the lesson reaches for the pattern named there.
- `useOptimistic` for pre-confirming the purchase before the webhook lands — out of scope for billing because the optimism is dangerous; named as the anti-pattern.
- Background-job offload for the per-event handler — Chapter 13.1.

Estimated student time: 50 to 60 minutes. Decision archetype. The hardest senior-mindset content in the chapter; the failure modes are subtle and the fix is architectural, not syntactic.

## Lesson 12.1.4 — Idempotency as a unifying discipline

Topics to cover:

- **The senior question.** The chapter so far has dedupe a webhook event by `event.id` through a unique constraint. Chapter 7.2.5 foreshadowed a `crypto.randomUUID()` hidden field on Server Action forms to prevent double-submit. Chapter 13 (background jobs) is going to retry a queued export job after a worker crash. Three different mechanisms, three different call sites — and they're all the same pattern: a stable client-supplied (or provider-supplied) key, a unique constraint on a database table, an INSERT that doubles as a claim. The lesson lifts idempotency from "the webhook chapter's mechanism" to the cross-cutting discipline every side-effect-bearing seam reaches for, names the three call sites, and consolidates the unique-on-key DB constraint as the single mechanism.
- **What "idempotent" actually means at this seam.** An operation is idempotent when calling it twice produces the same observable state as calling it once. "Same state" means: same database rows, same external effects fired exactly once, same user-visible outcome. Idempotency is not "the operation succeeded twice"; it's "the second call did nothing because the first one already happened, and we knew that." The senior anchor: idempotency is a property of the *seam*, not of the operation — a single write to a `versions += 1` counter is non-idempotent at the SQL level but becomes idempotent at the action level when wrapped with a unique-keyed claim.
- **The three call sites the course has or will reach.**
  - **Webhooks (this chapter).** Provider-supplied key (`event.id`), table is `processed_events`, INSERT happens at the start of the handler.
  - **Server Actions on side-effect-bearing creates (Chapter 7.2.5 foreshadowed).** Client-supplied key (`crypto.randomUUID()` in a hidden form field), table is `processed_actions` (or per-resource: e.g., the `idempotency_key` column on the resource's own table for "create invoice" — same constraint shape, different table), INSERT happens after parse-and-authorize and inside the action's transaction.
  - **Background jobs (Chapter 13 foreshadowed).** Job-supplied key (the job's ID, durable across retries — Trigger.dev exposes `ctx.attempt` and a stable `taskId`/`runId`), table or in-place constraint (depends on the job's effect), the check happens at the start of the job body.
  Each call site has a different *source* of the key but the same *mechanism* — unique constraint, INSERT-or-conflict, branch on rows-affected.
- **The unique-constraint mechanism, consolidated.** The pattern, in three lines, agnostic of caller:

  ```
  await tx.insert(idempotencyTable).values({ key, scope }).onConflictDoNothing();
  const isNew = result.rowsAffected === 1;
  if (!isNew) return cachedOrEmptyResponse;
  ```

  The constraint is the synchronization primitive. The INSERT is the claim. The "rows affected" tells the caller whether to do work or return.
- **The 7.2.5 server-action idempotency surface, now built.** A `processed_actions` table with `(orgId, actionName, key)` as the unique tuple. The Server Action takes a `formData` whose hidden field contains the key, parses it (the schema validates it's a UUID), starts a transaction, inserts into `processed_actions`, branches on conflict. On conflict, the action returns a `Result` describing the prior outcome — but "prior outcome" is the senior question: where does the original result live so the second call can return it? Three answers, picked by use case:
  - **The "just return ok" reach.** If the result the user needs is recoverable from the database state (the row was created, the user can re-fetch it from the list view), the duplicate path returns `ok({ deduplicated: true })` and the form redirects to the list. Cheapest, works for most CRUD.
  - **The "store the result" reach.** A `result_payload` JSONB column on `processed_actions` keyed by the key; the first call writes it, the second call reads and returns it. Right when the action's return value is hard to re-derive (a payment confirmation number, a generated token shown once).
  - **The "store the resource ID" reach.** The action writes the created resource ID into `processed_actions.result_resource_id`; the duplicate call looks up the resource and returns it. The middle ground.
  The course default for non-billing CRUD: the "just return ok" reach. The course default for money-moving or token-issuing actions: the "store the resource ID" reach. The senior anchor: pick once per action, document the choice.
- **The "where does the key live in the form" mechanics.** The Server Component renders the form with `<input type="hidden" name="idempotencyKey" value={crypto.randomUUID()} />`. The key is generated server-side per render — every form load gets a new key. A double-click on submit (or a network retry) sends the same key both times because the form value is stable. The senior reach: do not generate the key client-side from `crypto.randomUUID()` in a `useEffect` — hydration changes the value, the second render gets a different key, the protection is gone.
- **When idempotency keys aren't needed.** Read-only Server Actions (none in the canonical shape, but Server Components that call functions with `cache()` are fine without keys). Actions whose database mutation is itself naturally idempotent: `UPDATE preferences SET theme = 'dark' WHERE userId = ...` is idempotent at the SQL level — running it twice produces the same row. Actions that always overwrite a single row keyed by a stable identity (`UPSERT a profile, replacing the whole document`) — idempotent by construction. The senior call: add the key when the operation creates a new row, fires an external side effect, or has any "did this already happen?" character; skip when the database semantics make it free.
- **The `Idempotency-Key` HTTP header — the route-handler corollary.** Public route handlers (Chapter 7.5) that accept POSTs from external callers should respect an `Idempotency-Key` request header (RFC 9110 names the convention; the chapter named the contract at 3.2.1). The route handler reads the header, validates it's a UUID or similar, runs the same `processed_actions`-shape dedup. Stripe's API itself accepts `Idempotency-Key` on every POST — the lesson names this so the student understands the discipline is symmetric: providers expect it from callers, providers supply it back through `event.id` to webhook consumers.
- **Retention and cleanup for `processed_actions`.** Same as `processed_events` — a daily job deletes rows older than the action's retry window (often 24 hours; never less than the longest realistic network retry). The retention column is the `created_at` timestamp on the table; the job is one DELETE. The senior anchor: idempotency tables grow without bound by default; retention is part of the schema, not a separate concern.
- **The "idempotency vs. concurrency" distinction, restated.** Chapter 11.2.3 taught the `version` column as the precondition for optimistic concurrency — two different writes clobbering each other on the same row. Idempotency keys (this lesson) prevent the same write from landing twice. Both can apply to the same Server Action: the idempotency key dedupes a retry, the version check detects the cross-user collision. The two patterns compose; they don't substitute. The lesson re-states the distinction so the student doesn't conflate them.
- **The "idempotency is a contract, not a feature" framing.** Every action in the course that creates a row or fires an external effect documents its idempotency: which key it accepts, what scope the key is unique under, what the duplicate-call response is. The form layer and the action layer both know the contract — the form supplies the key, the action enforces it, both rely on the unique constraint. The senior anchor: when the contract is implicit, double-clicks become production incidents; when it's explicit, the same mechanism prevents the bug everywhere.
- **The unified table that doesn't exist — and why.** A natural temptation: one `idempotency_keys` table with a `scope` column for `'webhook' | 'action' | 'job'`. The course resists: webhooks come from providers (no `orgId` necessarily), actions are tenant-scoped, jobs are background-scoped, retention windows differ. Two or three small tables (`processed_events`, `processed_actions`, plus whatever per-job dedup mechanism Chapter 13 picks) are cleaner than one generic table with a discriminator. The senior anchor from Architectural Principle #5: small concrete tables beat one generic table with conditional indexes.
- **Worked example — the server action with an idempotency key.** A `createInvoice` action from Chapter 7's worked example, now with the hidden-field idempotency key, the `processed_actions` insert at the top of the transaction, the conflict branch returning the prior invoice ID via `result_resource_id`. The student fires the form twice in quick succession (the lesson includes a small simulator: a button that submits the form twice in 50ms) and watches one row land plus two `200 ok` responses, both pointing at the same invoice.
- **Watch-outs.** Client-generated idempotency keys via `useState(() => crypto.randomUUID())` get a new key on re-render — server-render the key into a hidden field; the same key used across two different actions ("create invoice" and "delete invoice" both write the same key) is a contract bug — keys are scoped by action name, the constraint enforces it; the "store the result" pattern with a `result_payload` JSONB column tempts callers to evolve the schema of the stored result without versioning — name the contract; the unique constraint without a retention job is a slow leak — the index grows, queries slow, the operator notices six months in; the action that does its idempotency check *after* the database mutation provides no protection — the INSERT happens first, every time; idempotency keys re-used across tenants because the unique constraint forgot the `orgId` column is a cross-tenant correctness bug — composite uniqueness is the default; the temptation to "just use `updatedAt = now()` as the dedup key" doesn't work — it changes on every successful write, so it dedupes nothing.

What this lesson does not cover:

- The Trigger.dev job-specific idempotency surface — Chapter 13.1.
- The route-handler implementation of the `Idempotency-Key` header — Chapter 7.5 introduces the header, this lesson names the dedup mechanism.
- Optimistic concurrency (version columns) — Chapter 11.2.3 (named for the contrast).
- The audit log link from `processed_actions` to the audit table — Chapter 17.2.3.
- Cross-region replication of the idempotency table — out of course scope.

Estimated student time: 45 to 55 minutes. Synthesis archetype. The lesson that earns its weight by naming the discipline across three previously-disconnected chapters; the implementation is small because the mechanism is shared.

## Lesson 12.1.5 — Applying the pattern: Resend bounce and complaint webhooks

Topics to cover:

- **The senior question.** Chapter 8.1.4 named an `email_suppressions` table and the discipline "never send to a suppressed address." The handler that populates the table was deferred to this chapter. The user signs up, the welcome email bounces (the address typo'd, the mailbox is full, the domain rejects). The user marks a campaign as spam (a complaint). Both events arrive as Resend webhooks. The senior question: is the ingestion shape different from Stripe's? The lesson lands "no — verify, dedupe, transact, ack — the same shape with a different provider, a different SDK verifier, a different per-type handler," and closes the email deliverability loop from Unit 8.
- **The Resend webhook payload.** Resend ships webhook events with types like `email.sent`, `email.delivered`, `email.bounced`, `email.complained`, `email.opened`, `email.clicked`. The lesson focuses on `email.bounced` and `email.complained` because those are the suppression-list inputs; the rest are useful for analytics (Chapter 20.2) but don't write to suppressions. The event payload includes `data.email_id`, `data.from`, `data.to`, `data.subject`, and (for bounces) a `bounce.type` (`hard` or `soft`).
- **Hard versus soft bounces — the suppression decision.** A hard bounce is permanent (the address doesn't exist, the domain doesn't exist, the recipient explicitly rejected). A soft bounce is transient (the mailbox is full, the recipient's server is overloaded). The senior reach: suppress on hard bounce *and* on complaint; do not suppress on soft bounce (the address may work tomorrow). Resend's payload distinguishes the two through `bounce.type`; the handler reads it.
- **The signature verification — Svix-flavored.** Resend's webhooks are signed using a Svix-style scheme (because Resend uses Svix under the hood for webhook infrastructure). The headers are `svix-id`, `svix-timestamp`, `svix-signature`. The verification surface is similar to Stripe's: HMAC-SHA256 over `${id}.${timestamp}.${rawBody}`, constant-time compare, timestamp tolerance. Resend ships a `verifyWebhook` helper in `@resend/node` that takes the request and the secret; the lesson reaches for it the same way 12.1.1 reached for `stripe.webhooks.constructEvent`. The senior anchor: the *mechanism* is universal (HMAC over raw body, constant-time), the *header format* is per-provider — use the SDK's verifier where it exists.
- **The route handler — `/app/api/webhooks/resend/route.ts`.** A POST handler with the same five-line shape as the Stripe handler: read raw body, verify, dedupe via `processed_events` (with `provider: 'resend'` — the multi-provider key from 12.1.2 earns its weight here), dispatch by `event.type`, ack 200. The student writes the file; the lesson highlights how little code it takes because the shape was generalized in 12.1.1 and 12.1.2.
- **The per-type handlers — `handleBounce` and `handleComplaint`.** Both write to `email_suppressions` with `INSERT ... ON CONFLICT DO NOTHING` on `(email, reason)` so the same address bouncing twice doesn't fail the constraint. The schema fragment for `email_suppressions`: `email` (text, lowercased before write — the suppression check is case-insensitive), `reason` (enum: `hard_bounce`, `complaint`), `suppressedAt` (timestamp). Same outer transaction as Stripe; same dedup-first ordering; same 200 response.
- **The closure with the send-side check from 8.1.4.** Chapter 8.1.4 named the discipline "every send checks the suppression list first." The check, restated: before calling `resend.emails.send`, query `email_suppressions WHERE email = $lowercased`; if a row exists, return `{ ok: false, code: 'suppressed' }` without sending. The lesson names the helper from 8.1 (`isSuppressed(email)`) and shows the symmetry: the webhook handler writes to the table, the send helper reads from it, the discipline is one round-trip.
- **The "what about transactional emails to a suppressed address?" carve-out.** Some transactional emails (password reset, security alerts, account-deletion confirmations) the user *needs* even if they previously bounced or complained. The senior reach: a `bypassSuppression` flag on the send helper, used by a small whitelist of critical transactional flows, with an audit log entry every time it fires. The lesson names the carve-out; the implementation is two more `if` branches and an audit-log row.
- **The "we don't need to dedupe sent/delivered" reasoning.** `email.sent` and `email.delivered` are analytics events — they don't mutate user-facing state. Subscribing to them and ignoring them inside the dispatcher (return 200, no-op) is fine; passing them into the dedup table is unnecessary. The senior call: subscribe narrowly. The narrower the subscription, the less the handler does. If product analytics (Chapter 20.2) needs them, they go to a separate analytics endpoint, not this handler.
- **The webhook endpoint registration — Resend dashboard.** The endpoint URL is registered in the Resend dashboard, the signing secret is shown once at creation and copied into `RESEND_WEBHOOK_SECRET`. The lesson names the env entry, names the rotation procedure (rotate at the dashboard, deploy the new secret, then revoke the old one — there is a short window where both must be accepted; Resend supports a secondary secret slot for this).
- **The same pattern, with a different SDK — generalization.** The lesson explicitly names the shape: verify (provider SDK or hand-rolled HMAC) → dedupe via `processed_events` (composite key includes provider) → outer transaction → per-type handler → 200. Switching from Stripe to Resend to (hypothetically) GitHub or Slack changes only the verifier and the per-type bodies — the surrounding chassis is invariant. The senior anchor: the chassis is the lesson; the provider-specific bits are details.
- **The retention symmetry.** `email_suppressions` rows don't have a retention window — a suppressed address stays suppressed until manually un-suppressed by an admin (a user requests to be removed from the suppression list, the admin tool runs the DELETE). `processed_events` rows for Resend events do have retention (same seven-day window as Stripe events). The senior anchor: data lifecycle is per-table, not per-provider.
- **Worked example — the assembled Resend handler.** The route file. The `handleBounce` and `handleComplaint` functions. The migration that adds `email_suppressions` if not already present from 8.1.4. The student fires a test webhook (Resend's dashboard has a "send test event" button per type) and watches the suppression row land. The student then re-fires the same event and confirms the dedup branch returns 200 without a second INSERT (the `ON CONFLICT DO NOTHING` on `email_suppressions` would also catch it, but the `processed_events` check fires first and is the contract).
- **Watch-outs.** Forgetting the `provider` discriminator in the dedup table makes Stripe and Resend events collide if their IDs ever overlap — composite key is the protection; subscribing to `email.sent` and writing it to the suppression flow by accident suppresses every sent address — narrow the dispatcher to bounce and complaint; storing the bounce-reason text from Resend in `email_suppressions` without enum-narrowing tempts unbounded growth — pick the two enum values and discard the rest of the payload; case-sensitive comparison on email addresses breaks the suppression read (the user typed `User@Example.com`, the webhook delivered `user@example.com`) — lowercase on write *and* on read; bypassing the suppression check from a non-critical flow is the deliverability bug Chapter 8.1.4 introduced the table to prevent — name the carve-out, gate it on a flag; running this handler without `processed_events` dedup means a Resend retry storm during an outage double-suppresses every bouncing address — the dedup table is non-optional.

What this lesson does not cover:

- The `email_suppressions` table schema in detail — Chapter 8.1.4 owns it.
- React Email composition (Chapter 8.2) and the welcome email template (Chapter 8.3).
- Analytics on email delivery rates — Chapter 20.2.
- The audit log entry on `bypassSuppression` calls — Chapter 17.2.3.
- GitHub or Slack webhook ingestion — named as same-shape; not implemented in the course.

Estimated student time: 35 to 45 minutes. Pattern archetype, applied. Shorter than the earlier lessons because the chassis is reused; the lesson's value is showing the transfer cleanly.

## Lesson 12.1.6 — Chapter quiz

Top 10 topics to quiz:

- Signature verification at the webhook boundary — raw body (`req.text()`, never `req.json()`), HMAC-SHA256 over `${timestamp}.${body}`, constant-time compare, the timestamp tolerance as the replay window, 400 with RFC 9457 on failure (never 401, never 200).
- The two-secret reality of `stripe listen` — the CLI's signing secret in `.env.local`, the registered endpoint's secret in production; `stripe trigger` for replay-style testing.
- The `processed_events` table — schema (`provider`, `eventId`, `processedAt`), the composite unique constraint, the `INSERT ... ON CONFLICT DO NOTHING` that is both the check and the claim, the empty `.returning()` array as the duplicate signal.
- The outer transaction — why dedup INSERT and business UPDATE must commit together, the "external side effects after commit, never inside" rule (carried forward from 7.2.5), the per-type handler taking `tx` not `db`.
- The webhook response contract — 200 on first-time success, 200 on duplicate, 5xx on unhandled exception (let the provider retry), 400 on signature failure; never 200 after a partial mutation.
- Out-of-order events and the `last_event_at` ordering anchor — comparing `event.created` in the WHERE clause, the "re-read from the provider's API" alternative for state-transition events, the "subscribe to a narrow authoritative set" discipline.
- The redirect-versus-webhook race — "the webhook is the only writer," the success page reads-and-polls (`router.refresh()` on an interval), the explicit fallback that re-uses the same dedup table.
- Idempotency as a unifying discipline — the same unique-keyed claim across webhooks (provider's `event.id`), Server Actions (form-supplied UUID in a hidden field), and background jobs (job's stable run ID); the three "where does the prior result live" patterns.
- The `Idempotency-Key` HTTP header for public route handlers — the symmetric contract (providers supply it back through `event.id`, callers send it on POSTs), and the same `processed_actions`-shape dedup.
- Resend bounce/complaint ingestion — the identical chassis applied to a different provider (Svix-style headers, `email.bounced` with `bounce.type: 'hard'`, `email.complained`), the closure with `email_suppressions` from 8.1.4 (write-on-webhook, read-on-send, lowercase on both sides), the `bypassSuppression` carve-out for critical transactional emails.
