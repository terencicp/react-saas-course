# Chapter 14.1 — Notifications as a centralized layer (SaaS pattern #10)

## Chapter framing

Chapter 14.1 lands the dispatcher as the one named seam for user-facing notifications across the app. By this point the student has email plumbing (Unit 8), an orgs and members schema with an audit trail (Unit 10), invitations and billing flows (Units 10 and 12), and the Server Action boundary as the canonical write seam (Unit 7). What's missing is the rule that keeps every one of those flows from growing its own email-or-inbox code at the call site. The chapter teaches the dispatcher pattern in the small — one event in, channels resolved against user preferences, fan-out to email (Resend) and an in-app inbox (a `notifications` table), with dedup on rapid duplicates — and lands the conceptual line between user-facing notifications and operator-facing audit logs. The output is the mental model and the API surface (`dispatch(event)`, the `notifiable_events` registry, the prefs schema) that Chapter 14.2 wires into three real call sites.

Threads that run through every lesson: the dispatcher is the only place that knows about channels — call sites fire one event and don't import Resend or touch the `notifications` table; the `notifiable_events` registry is the single source of truth for what events exist, what channels they default to, and what templates they render; user preferences are read once inside the dispatcher, never re-checked at the call site, with a default-on rule for missing prefs so a new event type doesn't go silent at launch; notifications and audit logs are different tables for different audiences — a role change writes one of each, a failed-login attempt writes only an audit log, a "welcome aboard" writes only a notification; dedup is a 60-second window keyed by `(event_type, subject_id, recipient_user_id)` and lives inside the dispatcher so call sites can fire freely without coordinating; the dispatcher is invoked from the Server Action seam (synchronous in v1; the durable-queue upgrade is named and deferred to Unit 13a); email failures don't block the in-app write and vice versa — channels are independent. The chapter ships four teaching lessons plus a quiz, ordered by dependency: the pattern and the registry and the user-vs-audit line, then the two channel sends, then preferences and the resolution rule, then dedup, then the quiz.

## Lesson 14.1.1 — The dispatcher seam and the notifiable-vs-logged line

Topics to cover:

- **The senior question.** A growing SaaS gains channels (email, in-app inbox, eventually push and SMS) and gains events that need them (invitations, role changes, billing past-due, mentions, comments). If every call site directly calls `sendEmail` and inserts an inbox row, the channel-specific code multiplies, preferences get checked in some places and skipped in others, and adding a channel means touching dozens of files. Where's the seam, what's the API at the call site, and what does the dispatcher own?
- **The pattern named — one seam, one event, many channels.** The dispatcher is a single function (`dispatch({ event, recipients, payload })`) called from Server Actions and webhook handlers. Call sites describe *what happened* and *who should know*; the dispatcher decides *how* (which channels, which template, whether dedup applies, whether prefs allow). The architectural principle is the same one Chapter 7.2 lands at the action boundary and Unit 12.1 lands at the webhook seam — the named choke point that makes the cross-cutting concern a one-place edit.
- **The `notifiable_events` registry — the source of truth for event types.** A typed registry (a TypeScript map keyed by event type) names every event the app can fire, its default channels (email, inbox, both), its template reference, its dedup window, and its preference category. New event types are added in this one file; the dispatcher reads from it; the prefs UI reads from it; the templates are referenced by it. The registry-as-source pattern keeps the system enumerable — a senior reading the file knows every notification the app can send.
- **What lives in the registry — anatomy.** `event_type` (string literal, e.g., `'org.invitation.sent'`), `channels` (default set, e.g., `['email', 'inbox']`), `template` (a reference to a React Email component for email, an inbox-row formatter for in-app), `preferenceCategory` (the category the user toggles in settings, e.g., `'team'`), `dedup` (window and key shape, e.g., `{ windowSeconds: 60, keyBy: ['subject_id'] }`), and `description` (for the prefs UI label). Keep the registry small in v1 — three to five events — and let it grow.
- **The dispatcher's contract — input, side effects, output.** Input: a typed event with `type`, `recipientUserIds` (always a list, even when one), `subjectId` (the entity the event is about — invitation id, invoice id, member id), and a typed `payload` for template rendering. Side effects: writes inbox rows for recipients with inbox enabled in prefs, enqueues email sends for recipients with email enabled, records the dispatch in the dedup table. Output: a typed result the caller can log (count sent, count deduped, count suppressed by prefs). The dispatcher never throws on per-channel failure — it logs and continues; an inbox write failing must not kill the email send.
- **Where the dispatcher is called from — the Server Action seam.** The canonical call shape: inside the `authedAction` Server Action that performs the mutation (Chapter 10.2.2), after the database write succeeds and inside the same transaction or immediately after commit, call `await dispatch({ ... })`. Webhook handlers (Chapter 12.1) call the same dispatcher after their idempotent state transition. Background jobs (Chapter 13.1) can call it too. The dispatcher is platform-neutral about the caller; the rule is "fire after the state change is durable."
- **Synchronous in v1, durable later — the threshold.** The dispatcher in this chapter runs in-line with the request. The senior knows the upgrade: when send volume grows, when email sends start adding latency to user-visible actions, or when the system needs retries that survive a process crash, the channel sends move behind a Trigger.dev queue (Chapter 13.1) with the dispatcher writing only the event row and enqueueing the per-channel jobs. The lesson names the trigger and defers the implementation — v1 ships in-line because it's enough for tens of events per minute and one place to upgrade later.
- **The notifiable-vs-logged distinction — the line that decides which table.** `notifications` (the in-app inbox) is user-facing — the user opens it, reads it, marks it read, expects relevance. `audit_logs` (Chapter 10.2.5) is operator-facing — the org admin reviews it for compliance, security, and incident response, the user never sees it. Some events write both (role changed: the demoted member gets a notification, the admin sees the audit row). Some write only one (failed-login attempts go to audit logs, never to the user's inbox; "welcome aboard" goes to the inbox, never to audit logs). The senior rule: ask who reads it.
- **Worked examples of the distinction.** Invitation sent: notification to the invitee (so they see it in inbox if they're already a user), audit log on the inviting org. Role changed: notification to the affected member, audit log on the org. Billing past-due: notification to org owners, audit log on the org. Login failed: audit log only. Password changed: notification to the user (security signal), audit log on the user account. The rule maps cleanly once the audience question is asked.
- **What's not a notifiable event — the cut.** Internal state changes the user never needs to see (cache invalidation, background job completion that doesn't affect them, automated cleanup), high-frequency events that would spam (per-keystroke draft saves), and operator-only events (an admin's `EXPLAIN ANALYZE` ran). The default is "no notification" unless the senior question — does the user need to know? — answers yes. New event types should justify themselves in the registry's description field.
- **The recipients shape — orgs, roles, individuals.** The dispatcher takes `recipientUserIds` as a resolved list, not "the org's owners" or "anyone with the billing role." Resolution is the caller's job (with helpers like `getOrgMembersByRole(orgId, 'owner')`) because the caller knows the business rule. The dispatcher staying audience-neutral keeps it from growing org-shaped knowledge it doesn't need.
- **The `notifications` table shape — what an inbox row carries.** `id`, `user_id` (recipient), `org_id` (the org context, nullable for personal events), `event_type` (matches the registry), `subject_id` (the entity), `title` and `body` (rendered at dispatch time so the inbox UI is a pure render of the row), `read_at` (nullable timestamp), `created_at`, and a `payload` JSONB for any structured data the UI needs (link target, actor name). Render-at-dispatch over render-at-display so changes to the actor's name don't retroactively rewrite the inbox; capture the moment.
- **The seam diagram.** Call site (Server Action / webhook) -> `dispatch(event)` -> registry lookup -> preference read -> dedup check -> per-channel fan-out (inbox insert, email enqueue). A second arrow from the same call site to `audit_logs` for operator-facing events. The diagram pins the two tables, the two audiences, and the dispatcher as the one seam.
- **Watch-outs:** importing Resend or `sendEmail` from a Server Action other than the dispatcher leaks channel knowledge and breaks the seam — lint or grep for direct sends outside `lib/notifications/`; calling `dispatch` inside an open transaction without committing first risks sending a notification for a state that rolled back — call after commit, or use a transactional outbox pattern (named, deferred); putting prefs checks at the call site is a regression — the dispatcher owns the check; mixing notifications and audit logs in one table to "save a query" couples two different audiences and produces a UI that has to filter — keep them separate; a notification with a stale title because the actor renamed themselves after the dispatch is correct, not a bug — the row records the moment.

What this lesson does not cover:

- The per-channel send mechanics (Resend call, inbox row writer) — Lesson 14.1.2.
- The `user_notification_preferences` schema and read — Lesson 14.1.3.
- Dedup window mechanics — Lesson 14.1.4.
- The inbox UI (rendering, mark-as-read, unread badge) — out of scope; lives in the Chapter 14.2 starter's inspector panel.
- Push, SMS, Slack as channels — named as future additions, not implemented.
- The durable-queue upgrade (Trigger.dev workers) — named, deferred to Unit 13.1.
- The audit-log mechanics — Chapter 10.2.5.

Estimated student time: 45 to 55 minutes. The chapter's center of gravity — every later lesson plugs into this seam.

## Lesson 14.1.2 — The two channels — email and the in-app inbox

Topics to cover:

- **The senior question.** Once the dispatcher decides a recipient gets the event on email and in the inbox, how do those two sends work, what failure modes are independent, and what's the shared shape (the template reference, the rendered payload) versus the channel-specific shape (the email subject, the inbox title)? The lesson lands the two channel functions and the rule that channel failures are independent.
- **The channel function signature — uniform across channels.** Every channel exports a single function with the same shape: `sendEmail({ recipient, event, payload })` and `writeInboxRow({ recipient, event, payload })`. Both take the resolved recipient, the registry entry for the event, and the typed payload. The uniform shape lets the dispatcher loop over enabled channels without branching: `for (const channel of enabledChannels) await channels[channel](args)`. Adding push later means a new function with the same signature.
- **The email channel — calling into the Unit 8 wrapper.** The email function looks up the email template from the registry entry (a React Email component reference), renders it with the payload, calls `sendEmail` from `lib/email.ts` (Chapter 8.3.3) with the right subject, recipient address (resolved from the recipient's user record), and tags. Suppression checks (Chapter 8.1.4) happen inside the email wrapper, not in the channel function — the channel stays thin. The lesson surfaces the layering: dispatcher -> channel function -> email wrapper -> Resend.
- **The subject and the "from" name — registry-driven.** The email subject lives in the registry per event type (with template-variable interpolation for the actor or org name); the `from` name follows the transactional/marketing split (Chapter 8.1.3) — transactional events use the transactional sender. Keeping subject lines in the registry means new translations or rephrasings happen in one file.
- **The in-app inbox channel — the insert.** The inbox function writes one row to `notifications`: `user_id`, `org_id`, `event_type`, `subject_id`, `title`, `body`, `payload`, `created_at`. Title and body are rendered at insert time from the registry's inbox formatter (a small function `(payload) => ({ title, body })`) so the row is a snapshot — the inbox UI is a pure render. The insert is one statement, no joins, indexed on `(user_id, created_at desc)` for the inbox feed query (Chapter 6.4.1's composite-index discipline applies).
- **Why render-at-dispatch, not at display.** Rendering at dispatch captures the moment: actor names, org names, and amounts in the row are what they were when the event fired, not what they are when the user reads the inbox. The alternative (render-at-display) requires joining live data, which gets wrong when a user is later deleted or an amount changes, and makes the inbox query expensive. The senior reach: snapshot strings, link with stable ids in `payload` so the UI can still navigate.
- **The unread state and the `read_at` column.** `read_at` is nullable; null means unread. The unread badge is a single count query (`select count(*) from notifications where user_id = ? and read_at is null`). Mark-as-read is a single update setting `read_at = now()`. The lesson sketches the queries; the UI is in the project chapter's starter inspector.
- **The independence rule — one channel failing doesn't kill the other.** The dispatcher calls each channel function inside a try/catch and logs failures. An email send that hits a Resend 5xx leaves the inbox row written; an inbox insert that hits a database hiccup leaves the email sent. The lesson names the rule: notifications are eventually-consistent across channels by design, and a single channel's failure doesn't fail the action that triggered it. Retries belong in the durable-queue upgrade, not in v1.
- **The "fire after commit" rule, restated.** A notification for an action that rolled back is worse than a missed notification. The dispatcher is called after the action's database transaction commits, not from inside it. The pattern in code: the action `await tx.commit()` (or the implicit Drizzle commit), then `await dispatch(...)`. The transactional-outbox alternative (write a `pending_notifications` row inside the transaction, drain it from a worker) is named and deferred — v1's after-commit call is enough.
- **The recipient resolution — email address from the user record.** The email channel needs an email address; the user record holds it (Better Auth's `users` table). Reach for a small helper `getUserEmail(userId)` rather than passing addresses through call sites. The lesson surfaces the pattern: the dispatcher receives user ids, channel functions resolve to channel-specific identifiers.
- **The unsubscribe link in the email — required for the bulk-sender bar.** Per Chapter 8.1.2 and the 2026 deliverability bar, every transactional email beyond strict 1:1 needs a one-click unsubscribe header and a body link. The link routes to `/settings/notifications` with a token that pre-selects the category to disable. The lesson surfaces the requirement; the prefs UI is the project chapter's surface.
- **The inbox feed query — the user's inbox.** A single Server Component query: `select ... from notifications where user_id = ? order by created_at desc limit 50`. Cursor pagination (Chapter 6.3.6) for older entries. Composite index `(user_id, created_at desc)` keeps it fast. The lesson sketches the shape; the UI lives in the project.
- **Worked example — the role-change event firing both channels.** Registry entry for `'org.member.role_changed'` with channels `['email', 'inbox']`, an email template showing the new role and who changed it, an inbox formatter producing title `'Your role changed to admin'` and body. The dispatcher calls both channel functions; the email lands in the inbox, the row appears in the in-app inbox. The example threads the registry, the two channel functions, and the dispatcher's fan-out.
- **Watch-outs:** rendering email and inbox content twice (once per channel) duplicates the message — render the payload once in the dispatcher and pass the rendered strings to channels that need them; sending the email before the inbox row creates a window where the user reads the email, clicks through, and sees no inbox entry — order the inbox insert first when feasible; passing the raw `payload` to the email template without sanitization opens an HTML-injection vector via React Email's tags — React Email escapes by default but custom raw-HTML props don't; the inbox query without the composite index falls back to a sort on a large table — the senior reflex is to ship the index with the table; missing the unsubscribe link fails the bulk-sender bar — the email channel function asserts its presence; calling the email wrapper directly from a non-dispatcher call site is the regression the chapter exists to prevent — leave the comment, ship the lint rule, grep for it.

What this lesson does not cover:

- The email wrapper itself — Chapter 8.3.3.
- The React Email composition — Chapter 8.2.
- Suppression and deliverability — Chapter 8.1.
- The inbox UI and mark-as-read interaction — out of scope; project starter provides.
- Push, SMS, Slack channels — named as future additions.
- Durable retries on channel failure — Unit 13.1.

Estimated student time: 40 to 50 minutes. The two channel functions and the rule of independence.

## Lesson 14.1.3 — Preferences resolved once, default-on for missing rows

Topics to cover:

- **The senior question.** Users want to control which events they get on which channels. The dispatcher must read those preferences once per dispatch and resolve "channels for this recipient" before sending anything. Where do prefs live, how are they shaped, what's the default for a user who has never touched their settings or a new event type that didn't exist when the user signed up, and why is the check in exactly one place?
- **The schema — `user_notification_preferences`.** One row per `(user_id, category)` pair, with a JSONB or three-boolean column for channels (`email`, `inbox`, and reserved-for-future `push`). The category (e.g., `'team'`, `'billing'`, `'security'`) groups events that share a toggle — users don't want to manage every event type individually. Categories are defined in the registry; each event type names its category. The lesson sketches the Drizzle schema with a unique constraint on `(user_id, category)`.
- **Why category-level, not event-level — the granularity decision.** Per-event prefs produce a settings UI with twenty toggles users won't manage; category-level prefs (four to six categories) match the user's mental model ("team activity," "billing," "security alerts," "product news"). Categories also let new event types ship into an existing category without a migration of every user's prefs. The senior call: aggregate the toggles, expose the categories.
- **The default-on rule for missing rows.** A user with no row for `'team'` is treated as opted-in to all channels for that category. The reasoning: silence-by-default is worse than friction — users miss invitations, role changes, and billing alerts, then complain. The corollary: opt-out is the user action that creates a row. New event types in an existing category inherit the user's existing toggle. New categories default to on. The lesson names the rule and the trade-off (one user complaint per category vs. systemic silence).
- **Where the check happens — inside the dispatcher, exactly once.** The dispatcher, before fanning out to channels, reads the recipient's prefs for the event's category (one query, or a batched query for multi-recipient events). For each channel in the event's default channels, the dispatcher keeps the channel only if the pref allows it. Call sites never check prefs; channel functions never check prefs; the dispatcher is the one place. The lesson restates the seam principle: a cross-cutting concern checked in one place is one place to fix.
- **The resolution shape — `channels = registryChannels.filter(c => prefs[category]?.[c] ?? true)`.** A one-line filter with the default-on rule baked in via `?? true`. The expression is small enough to read in passing; the discipline is keeping it in the dispatcher and nowhere else.
- **The prefs read — one query, batched across recipients.** For an event with five recipients, the dispatcher reads five users' prefs for the event's category in one `where user_id in (...) and category = ?` query rather than five round-trips. The senior anchor: N+1 prevention applies to the dispatcher too. The query is small and indexed; the cost is negligible.
- **Per-channel mute beats global mute — the granularity the user expects.** A user can mute email but keep the inbox on (the common pattern: "I see it in the app, stop emailing me"). The schema supports this directly because each pref row has independent channel booleans. The lesson names the case as the most-common toggle in practice.
- **The "always-on" override for critical categories.** Security-critical events (password changed, login from new device, payment failed in a way the user must act on) should not be mutable to "off" on all channels. The registry can mark a category `criticalChannel: 'email'` meaning email stays on even when the user toggles it. The settings UI shows these toggles disabled with an explanatory tooltip. The lesson names the pattern; the project chapter wires one example.
- **Quiet hours and digest mode — when they earn their weight.** A 2026 senior knows the patterns exist (per-user quiet hours, daily/weekly digest rolling up low-urgency events) and that they're product-shaped rather than schema-shaped. They live in the dispatcher or a downstream digest worker. The lesson names them as out of scope for the project but as the canonical next step when the inbox grows noisy. Reach for them when user-research shows the noise problem; not before.
- **Webhook-driven external prefs — the email-suppression interaction.** A user who hits "unsubscribe" in an email triggers a Resend webhook (Chapter 12.1) that writes to `email_suppressions` (Chapter 8.1.4). The dispatcher already short-circuits via the email wrapper's suppression read, so the user is also functionally opted out of email at the channel layer. The lesson notes the two pathways exist (user toggles prefs, user clicks unsubscribe) and converge on "no email"; the prefs UI shows the user the suppression status with a way to re-subscribe.
- **The prefs UI — where it lives, what it shows.** A `/settings/notifications` page renders the categories from the registry with their current toggles per channel. Server Component reads, Server Action writes. The lesson sketches the shape; the project chapter's inspector ships a simplified version for verification.
- **The token-pre-selected-disable from unsubscribe links.** The unsubscribe link in transactional emails carries a signed token (HMAC-signed `{ user_id, category, exp }`) that the `/settings/notifications` page consumes to toggle that category's email off before rendering. The senior call: never trust unsigned tokens in unsubscribe URLs; the signature prevents an attacker from disabling another user's notifications by guessing ids.
- **Worked example — the prefs read in the dispatcher.** A small function `resolveChannels({ event, recipient })` returns the filtered channel list; the dispatcher loop calls it once per recipient. The function reads prefs, applies defaults, applies critical-channel overrides, and returns the final list. Twenty lines, one query, every rule in one place.
- **Watch-outs:** defaulting prefs to off when a row is missing produces silent regressions when new event types ship — default on; checking prefs at the call site is the regression to grep for — the dispatcher owns the check; per-event prefs produce a settings UI no one uses — aggregate to categories; making critical events mutable invites the support ticket "I disabled all emails and now my password reset didn't arrive" — mark categories as having a critical channel; unsubscribe links without HMAC let attackers disable victims' emails — sign the tokens; reading prefs per recipient in a loop is the N+1 the chapter already taught against — batch the read; users who toggle prefs in one tab while a dispatch runs in another see a race where the just-disabled channel still fires — acceptable, the next dispatch reads fresh.

What this lesson does not cover:

- The full prefs settings UI — project chapter's inspector provides.
- Quiet hours and digest implementations — named, deferred.
- The Resend unsubscribe webhook handler — Chapter 12.1.
- Email suppression read — Chapter 8.1.4.
- Token signing primitives — Chapter 9 / Chapter 17.2 owns the HMAC discipline.
- Per-org admin overrides on member prefs — out of scope.

Estimated student time: 40 to 50 minutes. Preferences are where the dispatcher pattern earns most of its weight — one read, one place, every channel decided.

## Lesson 14.1.4 — Dedup and coalesce on rapid duplicates

Topics to cover:

- **The senior question.** A user clicks "Resend invitation" five times in two seconds, or a webhook retries three times because the worker timed out, or two admins simultaneously demote the same member. Without protection, the dispatcher fires five emails and five inbox rows; the user is annoyed, the deliverability hit is real. What's the window, what's the key, where does the bookkeeping live, and what's the difference between dedup (drop the duplicate) and coalesce (collapse a burst into one)?
- **The 60-second window — the default that handles 95% of cases.** The dispatcher keeps a `notification_dedup` table with `(event_type, dedup_key, recipient_user_id, fired_at)` and a covering index. Before firing, it checks for a row with the same key in the last 60 seconds; if present, it skips. After firing, it inserts a row. 60 seconds catches double-clicks, webhook retries, and concurrent admin actions; longer windows risk dropping legitimate repeats; shorter windows let bursts through. The lesson names 60 as the default and the threshold (high-frequency events like comments or mentions need longer windows defined per event type in the registry).
- **The dedup key — per-event-type configurable.** The registry entry defines `dedup.keyBy` as an array of payload field names. For `'org.invitation.sent'`, the key is `subject_id` (the invitation id) — re-sending the same invitation dedupes. For `'org.member.role_changed'`, the key is `subject_id` plus the new role — a demotion-then-promotion-then-demotion within a minute produces one notification per transition. The key is registry-defined because the business meaning of "duplicate" varies per event.
- **Why per-recipient — same event, different audiences.** The dedup row includes `recipient_user_id` because dedup is per-recipient: a role change notifies the affected member and audit-logs to the admin; dedup applies independently. Two recipients getting the same event is not a duplicate. The composite key is `(event_type, dedup_key, recipient_user_id)`.
- **Dedup vs. coalesce — the conceptual line.** Dedup drops the duplicate outright (the 60-second pattern above). Coalesce collapses a burst into a single notification with summary content ("Jane commented 5 times on Invoice #42") — a different shape, useful for noisy event types. The lesson names coalesce, sketches its implementation (collect events into a pending bucket, flush on a timer or threshold), and defers it to "when comments or mentions ship" — v1 ships dedup only.
- **Where the bookkeeping lives — a dedicated `notification_dedup` table.** A small table with TTL behavior: rows older than the longest configured window can be deleted nightly by a cleanup job (a single delete query). The table is indexed on `(event_type, dedup_key, recipient_user_id, fired_at desc)` so the dedup check is one indexed read. The alternative (Redis with a TTL) is the right reach when the dispatcher hits high enough throughput that a database write per dispatch is the bottleneck; named, deferred to Unit 15.3 (Upstash) as the upgrade trigger.
- **The check-and-insert race — and why it's acceptable here.** Two concurrent dispatches for the same event both check, both see no row, both insert, both send. The race exists; the cost is one duplicate notification per concurrent firing. The senior call: a unique constraint on `(event_type, dedup_key, recipient_user_id, fired_at_minute)` would close the race but at the cost of insert failures the dispatcher would have to handle. For v1, the race is acceptable because the failure mode (one extra notification per rare concurrent burst) is much less bad than a missed notification under contention. The senior knows the lock-it-down upgrade exists.
- **The dedup check's place in the flow — before channel fan-out, after prefs.** The order inside the dispatcher: registry lookup -> prefs read -> dedup check (if duplicate, skip per-recipient) -> channel fan-out -> dedup row insert. Prefs first because a muted recipient doesn't need a dedup row written; dedup before channels because the goal is to skip the sends. The lesson surfaces the order as a small flowchart inside the dispatcher.
- **The "first one wins" semantics — what the user sees on a burst.** When five "resend invitation" clicks fire in two seconds, the first writes the inbox row and sends the email; the next four are dropped. The user sees one notification, the email recipient gets one email. The first event's payload wins — if the bursting events differ in payload (rare), the senior call is whether the registry's dedup key captures the right uniqueness. The lesson names the rule and the diagnostic (if the wrong event wins, the dedup key is too broad).
- **Webhook retry interaction — same idempotency story, different layer.** Chapter 12.1 lands webhook idempotency at the handler boundary: a duplicate webhook delivery produces the same state, no second event fires from the database. But if the handler does fire `dispatch()` and the webhook hits twice with different request ids, the dispatcher's dedup catches it. The two layers compose: webhook idempotency prevents the state churn, dispatcher dedup prevents the notification churn. The lesson names the layering.
- **Observability — counting deduped events for sanity.** The dispatcher's return value reports `{ sent, deduped, suppressedByPrefs }` per channel; structured logs (Chapter 20.1) capture the counts. A senior watches the dedup rate: a sudden spike means a call site is firing duplicates that shouldn't exist (a bug); a steady low rate means the system is working. The lesson surfaces the metric, defers the dashboard to Unit 20.
- **The cleanup job — pruning the dedup table.** A nightly Trigger.dev job (Chapter 13.1) deletes dedup rows older than the longest configured window plus a buffer. Single delete, indexed, fast. Without cleanup the table grows unbounded; with cleanup it stays small. The lesson names the job; the project's verify step doesn't exercise it.
- **Worked example — the rapid-fire test.** A small test that fires the same event five times in a tight loop and reads the resulting inbox and email count. The result: one inbox row, one email queued, four `deduped` counts in the return value. The example is the chapter's verification shape; the project chapter wires it as a button on the inspector.
- **Watch-outs:** a dedup key that's too narrow (e.g., includes a timestamp) defeats the purpose — every event is unique, nothing dedupes; a dedup key that's too broad (e.g., only `event_type`) collapses unrelated events for different subjects — include the subject; checking dedup before prefs writes dedup rows for users who never got the notification, making the count meaningless — prefs first; window in seconds matching cron cadence (e.g., 60s window with 60s cron retries) puts retries right at the edge — pick a window comfortably larger than the retry cadence; the dedup table without an index makes the check linear — ship the index with the table; cleanup that runs in the request path adds latency — keep it nightly and out of band; a sudden dedup-rate spike treated as "system working" is a missed bug signal — alert on the delta, not the floor.

What this lesson does not cover:

- Coalesce/digest implementations — named, deferred.
- Redis-backed dedup — Unit 15.3 names the upgrade.
- The cleanup Trigger.dev job — Unit 13.1.
- Webhook idempotency primitives — Chapter 12.1.
- Per-channel rate limits (Resend throttling, etc.) — owned by the channel wrapper, out of scope here.
- Observability dashboards on dedup metrics — Unit 20.

Estimated student time: 35 to 45 minutes. The smallest of the four teaching lessons; one mechanic with a clear failure mode.

## Lesson 14.1.5 — Chapter quiz

Top 10 topics to quiz:

- The dispatcher as the one named seam — call sites fire `dispatch(event)` and never import a channel directly; channels and prefs are the dispatcher's concern only.
- The `notifiable_events` registry as the source of truth — what fields it carries (channels, template, category, dedup key, description), and the rule that new events are added in one file.
- The notifiable-vs-logged-event line — `notifications` is user-facing, `audit_logs` is operator-facing; some events write both, some write only one; the audience question decides.
- The dispatcher's contract — input shape (event, recipients, subject id, payload), side effects (inbox writes, email enqueues, dedup row), return value (sent / deduped / suppressed counts), and the rule that per-channel failures don't kill the dispatch.
- The "fire after commit" rule — the dispatcher runs after the action's transaction commits, not from inside it; the transactional-outbox alternative is named, deferred.
- The uniform channel-function signature (`(recipient, event, payload) => Promise<void>`) and channel independence — one channel failing leaves the other's work intact.
- Render-at-dispatch over render-at-display for inbox rows — the row is a snapshot of the moment; the UI is a pure render; live joins are the regression.
- The category-grained `user_notification_preferences` schema with per-channel booleans, the default-on rule for missing rows, and the registry's `criticalChannel` override for security events.
- Preferences resolved once inside the dispatcher, batched across recipients — call sites and channel functions never read prefs.
- Dedup as a 60-second window in `notification_dedup` keyed by `(event_type, dedup_key, recipient_user_id)`, dedup key configured per event in the registry, dedup-vs-coalesce as different shapes, dedup-then-channels as the order inside the dispatcher.
