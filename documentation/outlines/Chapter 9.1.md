# Chapter 9.1 — Auth concepts

## Chapter framing

Unit 9 wires Better Auth into the data backplane from Unit 6 and the action boundary from Unit 7, but every Better Auth knob the student turns later in the unit (session table shape, cookie attributes, OAuth callback wiring, `proxy.ts` gating) decodes one of three underlying ideas: the authn/authz split, the session-versus-token storage model, and the OAuth 2.1 authorization-code-with-PKCE flow. Chapter 9.1 installs those three ideas before the unit touches a library. The chapter does not configure Better Auth, write a sign-in form, or read a cookie at request time — those are owned by 9.2, 9.3, and 9.4 respectively. The lessons here are conceptual; the code samples are minimal and illustrate the protocol shape, not a working implementation.

Threads that must run through every lesson: authentication answers "who is this request from" and authorization answers "is this request allowed" — they are different problems with different solutions, and conflating them is the first source of auth bugs in a junior codebase; the 2026 senior default for web SaaS is server-stored sessions referenced by an opaque cookie identifier, not stateless JWTs in browser storage — JWTs earn their weight at machine-to-machine and cross-service boundaries, named once so the student can recognize the trigger; cookies carry the session identifier and inherit the senior defaults from Chapter 3.4.1 (`HttpOnly`, `Secure`, `SameSite=Lax`, `__Host-` prefix), this chapter names them at the auth call site without re-teaching the cookie primitive; OAuth 2.1 is the consolidated specification that retires implicit grant and password grant and makes PKCE mandatory for every authorization-code flow including confidential clients, "OAuth 2.0 with PKCE" in older docs means the same thing the student is learning here; the chapter teaches the protocol vocabulary (authorization server, resource owner, client, redirect URI, state, code verifier, code challenge) so the Better Auth provider config in Chapter 9.3.8 lands on prepared ground. The chapter ships three teaching lessons plus a quiz.

## Lesson 9.1.1 — Authentication versus authorization

Topics to cover:

- **The senior question.** A request hits a Server Action that updates an invoice. Two questions get answered before the database write happens: which user is on the other end of the request, and is that user allowed to update *this* invoice. Mixing the two — a single function that "checks if the user is logged in" and then writes — is the bug pattern that ships SaaS data leaks. The lesson installs the vocabulary that keeps the two questions separate for the rest of Units 9 and 10.
- **Authn in one paragraph.** Authentication establishes the identity of the principal making the request: this request is from user `usr_abc`. The mechanism is a credential the user proves (password, OAuth token, passkey assertion) once, and a session reference (a cookie, this chapter) that re-establishes the identity on subsequent requests without re-proving the credential. The output of authn is a `userId` (or `null` for anonymous). That is the entire contract.
- **Authz in one paragraph.** Authorization answers a separate question: given an identified principal, is the requested action permitted on the requested resource. The mechanism is a policy check at the action boundary against the resource's ownership, the user's role, or both. The output is a boolean (or a thrown 403). The input includes the `userId` from authn, the resource ID, and any role/membership lookup. Authn says *who*; authz says *whether*.
- **The two failure modes.** Skipping authn produces an unauthenticated 401 (the action ran without knowing who the caller was). Skipping authz produces an IDOR vulnerability (the action ran for an authenticated user against a resource they don't own — the classic "change the ID in the URL and read someone else's data"). Both have happened in production at companies the student has heard of; both are structural, not stylistic. The senior fix is to make each check structurally hard to skip — `proxy.ts` matchers for authn gating (Chapter 9.4.1), the `authedAction(role, schema, fn)` wrapper for authz (Chapter 10.2.2).
- **Where each check lives in this stack.** Authn happens at the request edge: `proxy.ts` reads the session cookie, resolves it to a `userId`, redirects to sign-in if absent. Server Components and Server Actions get the `userId` from a session read (Chapter 9.2.4) — no re-validation needed, the edge already did it. Authz happens at the action boundary: every Server Action and every route handler that writes runs a role/ownership check against the action's input before the database touches anything. The two layers are different files, different lessons, different mental models — by design.
- **RBAC named, deferred.** Role-based access control (owner / admin / member) is the authz model the course uses; the full treatment lands in Chapter 10.2.1. Named here so the student knows "admin" and "member" are authz concepts, not authn concepts.
- **ABAC and policy engines named, dropped.** Attribute-based access control and OPA/Cedar-style policy engines are the senior reach when RBAC stops scaling (cross-tenant rules, per-row attribute checks, time-bound permissions). Named once; the course commits to RBAC plus the tenant-aware query helper (Chapter 10.1.2) as the year-1 default.
- **Anonymous, authenticated, authorized — the three states.** A request from an unauthenticated visitor is anonymous (authn returns `null`). A request with a valid session is authenticated (authn returns a `userId`). An authenticated request that passes the action's policy check is authorized. The marketing pages are anonymous-accessible; the dashboard is authenticated-only; deleting an invoice is admin-authorized. The three states have three distinct gates.
- **Identity providers named.** The party that runs authn doesn't have to be the application: a user can sign in with Google, GitHub, or the company's SSO and the application trusts the provider's authn output. Lesson 9.1.3 walks the OAuth flow that makes this work; named here so the term "identity provider" has a referent.
- **The senior reflex.** Treat authn and authz as two checks at two layers, never one. Name them in code (`requireUser()` for authn, `requireRole('admin')` or `requireOwnership(invoice)` for authz). The course's Server Action wrapper (Chapter 10.2.2) bundles both with structural enforcement so the missing-check class of bug is hard to write.
- **Watch-outs.** "Logged in" is ambiguous — it can mean "authenticated" or "authorized to see this page," the senior reflex is to use the precise term; an `isAdmin` boolean on the user table conflates authn and authz state — the role lives on the org membership, not the user (Chapter 10.1.1); checking `session.user.role` in a Client Component is a UI hint, never a security check — the server enforces, the client renders; IDOR is the failure mode that shows up in pentests and bug bounties more than any other auth bug, and it's always a missed authz check on an authenticated path.

What this lesson does not cover:

- Session storage and cookies — Lesson 9.1.2.
- OAuth's identity provider flow — Lesson 9.1.3.
- The session table and Better Auth's schema — Chapter 9.2.
- Sign-in flows (email/password, magic link, 2FA, passkeys, OAuth) — Chapter 9.3.
- Authz at the action boundary, RBAC, the `authedAction` wrapper — Chapter 10.2.
- Multi-tenant scoping — Chapter 10.1.
- Audit logging — Chapter 10.2.5.

Estimated student time: 25 to 35 minutes. Short and foundational; the vocabulary lesson the rest of Units 9 and 10 reference back to.

## Lesson 9.1.2 — Sessions, tokens, and the cookie that carries them

Topics to cover:

- **The senior question.** Authn established that a request is from `usr_abc` on the first sign-in. How does the next request prove it's still from the same user, without sending the password again — and what shape does that proof take on disk and on the wire? The 2026 answer for web SaaS is a server-stored session, referenced by an opaque identifier carried in an `HttpOnly` cookie. The alternative (a stateless JWT in browser storage) is named, its trigger is named, and the senior default is named. The lesson installs the mental model the Better Auth session table (Chapter 9.2.1) and the `proxy.ts` session read (Chapter 9.4.1) sit on.
- **The session as a server-stored record.** A session is a row in a database table: `id` (the cookie value, a long random opaque string), `userId` (the authenticated principal), `expiresAt`, `createdAt`, plus device metadata (IP, user-agent) for the active-sessions UI in Chapter 9.4.3. Sign-in inserts a row; sign-out deletes it; every request reads it by ID. The row is the source of truth — revoking access means deleting the row, and the next request fails the lookup. This is the model Better Auth ships by default.
- **The session cookie — what's in it and what isn't.** The cookie carries the session `id` and nothing else: an opaque random string the server generated. No user data, no role, no email — those live in the database row the ID points at. The cookie's attributes inherit the Chapter 3.4.1 senior defaults: `HttpOnly` (JavaScript can't read it, so XSS can't exfiltrate it), `Secure` (HTTPS only, so a network attacker can't sniff it), `SameSite=Lax` (the default CSRF mitigation — the cookie is sent on top-level navigation but not on cross-site `fetch`), `Path=/`, and the `__Host-` prefix in production (no `Domain` attribute, locks the cookie to the exact origin). The cookie's lifetime matches the session's `expiresAt`; a `Max-Age` of zero on sign-out clears it. Named at the call site, not re-taught.
- **Sessions versus stateless tokens — the trade.** A JWT is a signed, self-describing token: the server packs the user ID, expiry, and any claims into a JSON payload, signs it, and hands it to the client; the client sends it on every request; the server verifies the signature and trusts the payload without a database read. The win is the missing read — a stateless service can authenticate a request from the token alone. The cost is revocation: a stolen JWT is valid until it expires, the server can't invalidate it without maintaining a revocation list (which puts the database read back). For browser-facing SaaS the server runs at every request anyway, the database is already in the request path, the session read costs effectively nothing, and revocation is non-negotiable — sessions win. JWTs earn their weight at machine-to-machine boundaries (a public API, an internal service mesh, a mobile app talking to a different backend) where revocation latency is tolerable and avoiding a session lookup is material. Better Auth uses sessions by default; the JWT plugin (named once) is the reach for the M2M case.
- **Cookie caching — the 2026 hybrid.** Better Auth's "cookie cache" feature signs a short-lived copy of the session data into the cookie itself (compact base64url, JWT, or JWE encoding) so the common-case read skips the database entirely and falls back to the DB read only when the cache expires or the session is explicitly invalidated. Conceptually identical to "JWT access token + refresh token" but with the refresh token replaced by a server-stored session and the access token's lifetime measured in seconds, not minutes. Named here so the student recognizes the term when they see it in 9.2.3; the senior trigger is request volume past what a single session-read-per-request can absorb.
- **Session lifetime — `expiresIn` and sliding expiry.** Two knobs the senior tunes: `expiresIn` (absolute lifetime — the session dies at this timestamp no matter what, Better Auth's default is 7 days), and `updateAge` (the sliding-expiry window — every request inside the window extends the expiry forward, so active users stay signed in indefinitely while idle sessions die on schedule). The 2026 reflex: pick the absolute lifetime against the security posture (banking is hours, consumer SaaS is days to weeks), let sliding expiry handle the active-user UX. Named here; configured in 9.2.3.
- **Multi-device sessions — many rows, one user.** Because the session is a row keyed by ID, one user can have many concurrent sessions (laptop, phone, tablet) without any extra design — each device gets its own row on its own sign-in. The active-sessions UI (Chapter 9.4.3) lists the user's rows and lets them revoke individuals. The senior call: don't gate to a single session per user unless the product genuinely requires it; multi-device is the modern default. Stateless JWTs make multi-session revocation harder, which is one more reason web SaaS picks sessions.
- **Where the cookie travels — the request anatomy.** A diagram is worth showing once: browser sends `Cookie: __Host-better-auth.session_token=<id>` on every same-origin request; the edge handler (`proxy.ts`) reads it via `cookies()` (the API named in 5.5.1), looks up the row in Postgres, attaches the resolved `userId` to the request, and passes it down. Server Components and Server Actions read the same session through Better Auth's helper — the cookie is the wire format, the database row is the source of truth.
- **The session token versus the user — never conflate.** The session ID is a request-time credential; the user record is a long-lived identity. Sessions get rotated on sign-in, deleted on sign-out, expired by the server. Users persist. A user with zero active sessions is fully valid — just signed out everywhere.
- **The CSRF posture in two lines.** `SameSite=Lax` blocks cross-site `fetch` from carrying the cookie, which kills the classic CSRF attack against state-changing endpoints. Server Actions in Next.js add an origin check on top (Chapter 9.4.4 fully). The student leaves with one rule: same-site cookies are the default mitigation, and the stack doesn't need explicit CSRF tokens. Named here; the wiring is 9.4.4.
- **The XSS posture in one line.** `HttpOnly` removes the cookie from JavaScript's reach, so an XSS bug can't grab the session and impersonate the user — the bug becomes a same-origin browser-side exploit, not an account takeover. Storing session tokens in `localStorage` (which JavaScript can read) is the failure mode the cookie discipline avoids; named once and dropped.
- **Watch-outs.** A JWT in `localStorage` is the canonical 2026-junior anti-pattern — defaulting away from cookies because "tokens are modern" inherits XSS exfiltration and revocation-latency problems for no real win in a browser-facing app; an `HttpOnly` cookie cannot be read from client JavaScript, so any code that does `document.cookie` to "check if signed in" is buggy at the premise — the server tells the client by rendering or not rendering the page, the cookie's contents are not the client's concern; `SameSite=None` is the explicit opt-out and the trigger is genuine cross-site usage (an embedded widget, a third-party iframe) — every other site picks `Lax`; long absolute lifetimes plus aggressive sliding expiry can make sessions effectively immortal — pair them with the device-management UI (9.4.3) so users have a way to revoke; the session ID is a bearer credential — anyone holding it is the user, which is why every other defense matters.

What this lesson does not cover:

- The cookie primitive itself, attributes, the `__Host-` prefix — Chapter 3.4.1.
- The Better Auth `session` table schema and column layout — Chapter 9.2.1.
- Session strategy configuration (`expiresIn`, `updateAge`, cookie cache encoding) — Chapter 9.2.3.
- Reading the session in middleware, layouts, and Server Actions — Chapter 9.2.4.
- Active-session management UI — Chapter 9.4.3.
- CSRF and XSS defenses in this stack — Chapter 9.4.4.
- JWT plugin for M2M boundaries — named here, owned by Better Auth's plugin docs (course doesn't teach it).

Estimated student time: 40 to 50 minutes. The conceptual center of the chapter; every Better Auth knob in 9.2 decodes against this mental model.

## Lesson 9.1.3 — OAuth 2.1 and the authorization-code flow with PKCE

Topics to cover:

- **The senior question.** The product needs "Sign in with Google" — the user types nothing, clicks a button, and lands on the dashboard authenticated. What protocol makes that work without the application ever seeing the Google password, what are the moving parts, and why does the 2026 specification (OAuth 2.1) make PKCE mandatory even when the client is a server-side application that already has a secret? The lesson walks the authorization-code-with-PKCE flow end to end so the Better Auth OAuth provider config in Chapter 9.3.8 (redirect URIs, scopes, the callback route) lands on understood vocabulary.
- **OAuth 2.1 in one paragraph.** OAuth 2.1 is the 2024-finalized consolidation of OAuth 2.0 and its decade of security best-practice RFCs. It removes the implicit grant and the resource-owner password credentials grant (both deprecated for being attackable), pins exact-match comparison for redirect URIs (no path prefixes, no wildcards), and makes PKCE mandatory for every authorization code flow — public clients and confidential clients alike. "OAuth 2.0 with PKCE and the BCP" in older documentation describes the same protocol; OAuth 2.1 is the cleaned-up reference. The senior reflex: when picking a library or service, verify it implements 2.1 (or equivalent), reject anything still offering implicit flow.
- **The four roles — vocabulary first.** *Resource owner* is the human user signing in. *Client* is the application requesting access (the student's SaaS app — confidential, runs on a server, has a client secret). *Authorization server* is the identity provider (Google's OAuth endpoints, GitHub's OAuth endpoints) — issues codes and tokens. *Resource server* is the API the tokens unlock (Google APIs, GitHub APIs). For sign-in alone the resource server is usually irrelevant — the protocol's output is "Google says this user is `user@example.com`," and the app maps that to its own user record. Named once so the spec terms have referents.
- **The authorization-code flow — step by step.** A diagram is essential here (Mermaid sequence). The eight steps, named explicitly:
  1. App generates a `code_verifier` (a high-entropy random string) and derives a `code_challenge` (`SHA-256(verifier)`, base64url-encoded). Stores the verifier server-side, keyed by a `state` parameter.
  2. App redirects the browser to the authorization server's `/authorize` endpoint with `response_type=code`, `client_id`, `redirect_uri`, `scope`, `state`, `code_challenge`, `code_challenge_method=S256`.
  3. Authorization server authenticates the user (their Google password, their GitHub session — not the app's problem), shows the consent screen ("This app wants to access your email and profile"), and on approval redirects the browser back to `redirect_uri` with `?code=<short-lived-code>&state=<state>`.
  4. App receives the redirect, looks up the stored verifier by `state` (and verifies `state` matches — CSRF defense), exchanges the code at the authorization server's `/token` endpoint by POSTing `code`, `redirect_uri`, `client_id`, `client_secret`, and `code_verifier`.
  5. Authorization server verifies the code-verifier hashes to the stored code-challenge (PKCE check), verifies the client secret, returns an `access_token` and `id_token` (OpenID Connect — Google issues both, GitHub access-token only).
  6. App calls the provider's userinfo endpoint (or decodes the `id_token`) to get the user's email, name, and provider-specific ID.
  7. App matches the provider ID against its `accounts` table (Chapter 9.3.8 owns the schema); if no match, creates a user record; in either case, creates the application's own session (Lesson 9.1.2) and sets the session cookie.
  8. App redirects to the post-sign-in page. The OAuth flow is over; subsequent requests use the session cookie, not the access token.
- **The state parameter — CSRF defense at the redirect.** A random server-generated value the app stores keyed to the pending OAuth attempt, sends on the `/authorize` redirect, and verifies on the callback. Without it, an attacker can craft a sign-in URL that completes against their own account and tricks a victim's browser into linking the victim's session to the attacker's identity. Better Auth handles state automatically; named here so the student recognizes the parameter in the network tab.
- **PKCE — what it adds and why 2.1 requires it everywhere.** PKCE (Proof Key for Code Exchange, pronounced "pixie") binds the authorization code to the originator. The attack it stops: an attacker intercepts the authorization code in transit (browser history, referer header, log file, a proxy) and tries to exchange it for tokens — without the verifier the attacker generated, the token endpoint rejects the exchange. In OAuth 2.0 PKCE was optional and only required for "public" clients (mobile apps, SPAs with no client secret). OAuth 2.1 makes it mandatory for confidential clients too, because the historical assumption that a server-side client's network path is uninterceptable doesn't hold up (HTTPS proxies, internal logging, supply-chain compromise). The senior reflex: PKCE on every flow, no exceptions.
- **Scopes — what the app is asking for.** The `scope` parameter on `/authorize` lists the permissions the app requests (`openid`, `email`, `profile` for sign-in; provider-specific scopes for API access). The user sees them on the consent screen. The 2026 reflex: ask for the minimum scope the feature requires, never the full account-access scope "just in case." Each provider has its own naming (Google's `openid email profile`, GitHub's `read:user user:email`); the Better Auth provider config (Chapter 9.3.8) names them per provider.
- **Redirect URI — the exact-match contract.** OAuth 2.1 requires the authorization server to compare the `redirect_uri` against its allowlist by exact string match — no prefixes, no wildcards, no path-trailing-slash tolerance. The senior implication: every environment (dev, preview, staging, production) registers its callback URL with the provider, and the app's config wires the right one per environment. Per-PR preview URLs on Vercel make this a real operational concern (Chapter 9.3.8 owns the wiring patterns); named here.
- **OpenID Connect — sign-in's superset of OAuth.** OAuth 2.1 is authorization; OpenID Connect (OIDC) is the identity layer on top. The `id_token` (a JWT) is what makes "Sign in with X" reliable — it carries the user's identifier, the issuer, the audience, and expiry, all signed by the provider. For pure sign-in (the chapter's case), the app can read the user info from the `id_token` and skip a separate userinfo call. Google and Microsoft are OIDC providers; GitHub is OAuth-only (no `id_token`, the app calls the userinfo endpoint instead). Better Auth abstracts the difference; named here so the student knows what "OIDC" means.
- **Tokens that come back — and why the app immediately stops using them.** The access token and (optionally) refresh token from the provider are credentials for the *provider's* APIs, not for the application's. Once the app has the user's identity, it discards the access token (or stores it for later API calls if the feature needs them — Chapter 13 reaches for this in the integrations build) and switches to its own session cookie. The OAuth tokens are not the session; conflating them is the protocol mistake that produces the "token in localStorage" anti-pattern from 9.1.2.
- **Account linking — multiple identities, one user.** A user might sign in with Google today and GitHub tomorrow against the same email. The `accounts` table (Chapter 9.3.8 owns the schema) stores one row per `(provider, providerAccountId)` linked to one `userId`; the resolution logic on sign-in matches by email (with the verified-email gate) or treats the second sign-in as a separate account if the email doesn't match. Named here as the why; the implementation is 9.3.9.
- **Why the application never sees the password.** A consequence worth naming: the user's Google password never touches the application's server. The provider authenticates them; the app trusts the provider's signed assertion. The security boundary moves from "the app's password storage" to "the provider's authn quality" — usually a win, occasionally a constraint (the app inherits the provider's authn weaknesses, including its account-recovery process).
- **The flows OAuth 2.1 retires — named so the student recognizes legacy.** *Implicit flow* (`response_type=token`) returned the access token directly in the redirect URL — vulnerable to leakage through browser history, no longer permitted. *Resource owner password credentials* (the app collects the user's password and sends it to the provider) defeated the whole point of OAuth — gone. *Client credentials* (machine-to-machine, no user) still exists but isn't a sign-in flow; named once. If the student sees `response_type=token` in old code, that's a security-bug rewrite, not a maintenance task.
- **Watch-outs.** The redirect URI mismatch is the single most common OAuth setup failure — exact string match, the provider's allowlist must include the exact URL the app uses, dev/preview/prod each register separately; `state` is mandatory in 2.1 — a flow that ships without it is a CSRF vulnerability waiting; reading user data from the access token (instead of the `id_token` or userinfo endpoint) is wrong even when it "works" — access tokens are opaque by spec and the app's introspection breaks if the provider changes the format; trusting an unverified email from a provider (some providers issue tokens for unverified emails) lets an attacker create a Google account at `victim@example.com`, get a sign-in flow against it, and take over a SaaS account that's keyed by email — the senior reflex is to gate sign-in on the provider's `email_verified` claim, not just `email`; PKCE is the protocol's seatbelt, not optional decoration — every flow including server-side confidential clients runs it under 2.1.

What this lesson does not cover:

- Better Auth's OAuth provider configuration, the callback route, the `accounts` table — Chapter 9.3.8.
- Account linking implementation — Chapter 9.3.9.
- The session cookie set after a successful OAuth callback — Lesson 9.1.2 (concept) and Chapter 9.2.4 (read at request time).
- OpenID Connect at depth, claims, the `id_token` signature verification — Better Auth handles it; the course doesn't unpack the JWT internals.
- Refresh tokens and long-running access to provider APIs — Chapter 13.
- SSO via SAML and enterprise federation — out of scope; the course commits to OIDC/OAuth.
- Implementing an OAuth authorization server (the other side of the flow) — out of scope.

Estimated student time: 45 to 55 minutes. The protocol-walkthrough lesson; the Mermaid sequence diagram is the load-bearing teaching element. Pairs directly with Chapter 9.3.8.
