# Chapter 9.1 — Auth concepts

## Chapter framing

Unit 9's thesis is that authentication is a contract between the browser, the server, and a session store — and that Better Auth (Chapter 9.2 onward) is a thin, opinionated wrapper around that contract, not a black box. Chapter 9.1 installs the conceptual model the rest of the unit operationalizes: the difference between *who you are* (authentication) and *what you can do* (authorization), the shape of the session/token/cookie triplet that every request carries, and the OAuth 2.1 authorization-code-with-PKCE protocol Better Auth's social providers (9.3.8) drive end-to-end. No Better Auth code, no `auth.api` calls, no database tables — those start in 9.2. This chapter writes the mental model on which 9.2 through 9.5 are cheap.

The threads that must run through every lesson: authentication answers *who is the caller*, authorization answers *is this caller allowed to do this thing*, and the two get separate chapters in this course (Unit 9 vs. Unit 10) because conflating them is the single most common source of "I checked the user is logged in" bugs that ship admin actions to free-tier users; the 2026 default for a server-rendered SaaS on this stack is a server-stored, opaque, HttpOnly-cookie-bound session, not a stateless JWT — Better Auth ships this as the default and 9.2.3 names the threshold that flips the choice; cookie attributes (`HttpOnly`, `Secure`, `SameSite`, `Path`, `Domain`, `Max-Age`) are the protocol-layer security surface the senior owns regardless of which library mints the cookie, and XSS and CSRF are concrete attacks those attributes mitigate; OAuth 2.1 collapses the older spec set into "authorization code with PKCE, always" — implicit grant and password grant are dead, PKCE is mandatory, exact redirect URI matching is required, and refresh tokens rotate; every protocol detail named here gets cross-referenced to the Better Auth surface that abstracts it so the student knows which knob to reach for later. The chapter ships three teaching lessons plus a quiz.

## Lesson 9.1.1 — Authentication vs. authorization

Topics to cover:

- **The senior question.** A Server Action receives a request to delete an invoice. Two distinct checks must pass before the row leaves the database: the caller is a real, signed-in user (not an anonymous request, not a forged cookie), and that user is allowed to delete *this* invoice (they own it, or they're an admin on the org that owns it). The two checks look similar from the call site — both read the session, both can fail with a 401 or a 403 — but they answer different questions, fail in different ways, and live in different layers of the stack. The lesson names the distinction precisely, maps each side to the unit that owns it (Unit 9 for authn, Unit 10 for authz), and names the common bug that lives in the seam.
- **Two questions, two answers, two failure modes.** *Authentication (authn)* answers "who is the caller?" — its output is a `User` row (or `null`/anonymous). Its failure is "I don't know who you are," which maps to HTTP 401 Unauthorized and, in a web app, redirects to the sign-in page. *Authorization (authz)* answers "is this known caller allowed to perform this action on this resource?" — its output is a boolean (or a thrown forbidden error). Its failure is "I know who you are, but no," which maps to HTTP 403 Forbidden and, in a web app, shows a "you don't have access" page (or, for sensitive resources, the same 404 the resource would produce if it didn't exist — covered in 10.1.2). The 401/403 distinction is the wire-protocol shadow of the conceptual line; getting it right helps clients and monitoring tools route correctly.
- **The order matters and it's always the same.** Authentication runs first. Authorization runs second, against the authenticated identity. An authorization check on `null` (no user) collapses to the authentication failure — the senior reflex is to short-circuit on missing session before evaluating any role or ownership check, so the authz code path always operates on a non-null user.
- **Where each lives in the codebase.** Authentication lives at the request edge: `proxy.ts` (Next.js's middleware, renamed in v16 — Chapter 5.5.1), Server Component layouts, and the entry of every Server Action and route handler. Better Auth (Chapter 9.2) owns the session read; the application code calls `auth.api.getSession({ headers })` once and threads the user through. Authorization lives at the action boundary: inside Server Actions and route handlers, *after* the schema parse (Chapter 7.1.5) and *after* the authentication check, before any database write. The 10.2 `authedAction(role, schema, fn)` wrapper makes the layering structural; for now the student sees the layering conceptually.
- **The third question Unit 10 owns — "is this scoped to the right tenant?"** A SaaS app's authorization has two sub-questions wrapped in one: does the user have the right role (member, admin), and does the resource belong to an organization the user is a member of? The role check is RBAC (Chapter 10.2); the org-scoping check is the multi-tenant query helper (Chapter 10.1.2). Both are authorization. The chapter names them once so the student knows the model expands; the actual code lands in Unit 10.
- **The canonical bug — "the user is logged in, so they can do this."** A junior implementation reads "session exists" at the page or layout, hides the delete button when there's no user, and skips the authorization check on the underlying Server Action because the UI "already gated it." The bug: the Server Action is a public endpoint reachable with any HTTP client; UI gating is decoration, not enforcement. The senior reflex: every action re-checks authn and authz at its own entry, independent of what rendered the UI. The structural fix is the action wrapper in 10.2; the rule lands here.
- **Anonymous and partial-auth states.** Not every page is gated. Marketing pages, sign-in/sign-up pages, public read-only routes, and the post-sign-up "verify your email" page all serve unauthenticated or partially-authenticated requests. The senior frame: authentication has more states than "in" and "out" — there's anonymous, signed-in-but-email-unverified, signed-in-and-verified, signed-in-with-2FA-required-but-not-yet-passed, and signed-in-with-active-elevated-session (for reauthentication-required actions like changing the password — covered in 9.4.2). The application's gating policy is a state-to-route mapping; Chapter 9.4.1 wires the matcher.
- **Identification vs. authentication — the line one level deeper.** Identification is the claim ("I am `ada@example.com`"), authentication is the proof (a verified password, a passkey signature, an OAuth-provider assertion). The lesson names the distinction once because it shows up in two places: email enumeration (an unauthenticated sign-in endpoint that responds differently for "user not found" vs. "wrong password" leaks identification; the senior reflex is to return the same error for both — covered in 9.3.2), and account linking (Chapter 9.3.9), where two identifications (Google, GitHub) authenticate the same user.
- **Authorization patterns, named once and deferred.** RBAC (role-based) — the default for SaaS, every member has a role per org, the role gates the action. ABAC (attribute-based) — gates against arbitrary attributes (plan tier, region, feature flag), the senior reach when roles don't cut finely enough. ReBAC (relationship-based) — Zanzibar-style "user X has relation R to resource Y," the reach for collaborative apps with deep sharing (Google Docs, Notion). The course defaults to RBAC plus organization scoping; ABAC and ReBAC are named in 10.2 and dropped. Listed here so the student has the vocabulary when they see it elsewhere.
- **The "minimum viable" authentication for a 2026 SaaS — what 9.3 will build.** Email + password with verification, password reset, magic-link as an alternative, 2FA/TOTP for sensitive accounts, OAuth providers (Google, GitHub) for the no-password path, passkeys for the post-password future, account linking when a user signs in via two providers. Better Auth ships every one of these as a plugin; the chapter list (9.3.1 through 9.3.9) maps one-to-one to plugin choices. Named here so the student sees the shape of what's coming.
- **Watch-outs.** Conflating 401 and 403 in error responses confuses clients and monitoring — keep them distinct; UI-only gating without an action-side re-check is a vulnerability waiting for any attacker with a network panel; "admin" as a single boolean column doesn't survive the first feature request — Unit 10's `org_members(role)` is the right shape from day one; checking authorization before authentication produces null-dereference bugs at best and silent allow-by-default at worst — always authn first; identifying the user by email at sign-in failure leaks which emails exist in the system — the production-correct error is the same message for "not found" and "wrong password," named here, enforced in 9.3.2.

What this lesson does not cover:

- The session/cookie/token mechanics (9.1.2).
- The OAuth 2.1 protocol (9.1.3).
- Better Auth's API surface (Chapter 9.2).
- The `authedAction` wrapper and RBAC enforcement (Chapter 10.2).
- Tenant scoping and multi-tenancy (Chapter 10.1).
- Password hashing, rate limiting, account lockout (Chapter 9.3).
- CSRF and XSS at the cookie layer (9.1.2 names the mitigations; 9.4.4 is the chapter-level summary).

Estimated student time: 30 to 40 minutes. Conceptual-archetype lesson with a single labeled diagram showing the authn-then-authz request flow. No code blocks; the model is the lesson.

## Lesson 9.1.2 — Sessions, tokens, and the cookie that carries them

Topics to cover:

- **The senior question.** A user signs in. The next request, ten milliseconds later, somehow knows that user is signed in — without re-typing the password, without an `Authorization` header, without anything the application code visibly attached. What's actually crossing the wire on every request, where does it live in the browser, where does it live on the server, and why does a 2026 SaaS on this stack default to a server-stored opaque session in an `HttpOnly` cookie rather than the stateless JWT pattern the student saw in older tutorials? The lesson installs the session-and-cookie mental model end to end and names the cookie attributes that turn the carrier into a secure one.
- **The session is a server-side row; the cookie is the pointer to it.** When the user authenticates, the server creates a session record in the database (Better Auth's `session` table, owned in 9.2.2) with a random, opaque ID, a user reference, an expiry timestamp, and metadata (IP, user agent). The server then sets a `Set-Cookie` header on the response carrying that session ID — opaque to the client, indexed on the server. Every subsequent request from the browser includes the cookie automatically; the server reads the ID, looks up the row, and resolves the user. The two halves — the database row and the cookie — are the whole protocol.
- **Why "opaque" matters.** The session ID has no meaning to the client and contains no user data. It's a random 256-bit token. The client can't forge a valid ID without finding a collision in cryptographically random space; the server can revoke a session in one DB write without coordinating with anything else. Contrast with stateless tokens (JWT) covered below.
- **The session lifecycle — four events.** Creation (sign-in succeeds, row inserted, cookie set), reads (every request, row looked up by ID), rotation (the senior pattern: rotate the session ID on privilege changes — verify email, change password, MFA pass — to invalidate any stolen pre-rotation copy), and termination (sign-out, expiry, admin revocation; row deleted or marked, cookie cleared with `Max-Age=0`). Diagram: sequence diagram with browser, server, DB (Mermaid).
- **Idle timeout and absolute timeout — two clocks.** *Idle timeout* (the cookie's `Max-Age`, refreshed on each request) closes the session after a period of inactivity — typical SaaS value is 7 to 30 days, the senior call balances UX against the credential-replay window. *Absolute timeout* (a `created_at + max_lifetime` check on the row) caps the session even on an active user — typical value is 30 to 90 days, the reach for high-sensitivity actions (banking is 8 hours; a notes app is months). Better Auth defaults to a 7-day session with a rolling update window; the knob is `session.expiresIn` and `session.updateAge` (named in 9.2.3).
- **Multi-device sessions and per-device revocation.** A user signs in on a laptop and a phone — two session rows, two cookies, one user. The session table indexed by user lets the application list active sessions ("you're signed in on these devices") and revoke any one without touching the others. Useful for the "sign me out everywhere" button and security-incident response. The full UI lands in 9.4.3; the model lands here.
- **The HTTP cookie — a key=value pair plus attributes.** A `Set-Cookie` header looks like `Set-Cookie: better-auth.session=opaque-id; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=604800`. Each attribute is the protocol-layer surface the senior owns even when Better Auth writes the header; the lesson walks them:
  - **`HttpOnly`.** The cookie is unreadable from JavaScript (`document.cookie` doesn't see it). The XSS mitigation: even if an attacker injects a script, they can't exfiltrate the session ID. The 2026 reflex: every session cookie is `HttpOnly`. No exceptions for "we need to read it client-side" — if the client needs to know the user is signed in, the server renders that state or exposes a small `/api/me` endpoint.
  - **`Secure`.** The cookie is only sent over HTTPS. The MITM mitigation: a man in the middle on plain HTTP can't see the cookie. Always on in production; Better Auth's local-dev posture allows non-Secure cookies on `http://localhost` for the dev loop.
  - **`SameSite`.** The CSRF mitigation. Three values:
    - `Lax` (the 2026 default for session cookies). The cookie is sent on same-site requests and on top-level cross-site GETs (the user clicks a link from `evil.com` to `yourapp.com`). It is *not* sent on cross-site POSTs, image requests, iframe embeds — the surface CSRF exploits. Lax is the right default for almost every web SaaS session cookie.
    - `Strict`. The cookie is *only* sent on same-site requests; even a link from another site arrives without the cookie. Tighter, but breaks the common "user clicks a link in an email and lands signed in" flow. Reach for it on banking-style admin sessions; for a normal SaaS, `Lax` is the senior default.
    - `None`. The cookie is sent on every request including cross-site. Requires `Secure`. Used only when the cookie genuinely needs to cross sites (embedded apps, cross-origin SPAs against a separate API). For a Next.js SaaS with the app and the API on the same origin, `None` is wrong.
  - **`Path`.** The cookie is sent on requests under the given path. `Path=/` for session cookies; narrower paths only for special-purpose cookies (a sign-up wizard).
  - **`Domain`.** Default (no `Domain`) restricts the cookie to the exact host that set it. `Domain=.yourapp.com` lets subdomains share the cookie — the senior reach for a multi-subdomain setup (`app.yourapp.com`, `dashboard.yourapp.com`), the wrong reach when the apex hosts the marketing site that shouldn't see auth cookies.
  - **`Max-Age` and `Expires`.** Cookie lifetime in seconds, or an absolute datetime. `Max-Age` is preferred; sets to `0` to delete. Better Auth syncs the cookie's lifetime to the session row's `expiresAt`.
  - **The `__Host-` prefix.** A cookie named `__Host-session` is only accepted by browsers if it's set with `Secure`, `Path=/`, and no `Domain`. A defense-in-depth signal that prevents a compromised subdomain from setting an apex auth cookie. Named once; Better Auth's defaults don't use it but the option is available.
- **Session vs. JWT — the two patterns and the threshold that flips.** A *session-based* auth (the default in this course) stores state in the database; the cookie is a pointer. A *stateless JWT* auth puts the user identity and a signed claim set inside the token itself; the server verifies the signature on each request without a DB lookup. Comparison the senior weighs:
  - Session: revocation is one DB write; cookie contents are opaque; every request pays a session-table lookup (cheap with proper indexing).
  - JWT: no DB lookup on the hot path; revocation requires a blocklist (which puts state back into the system); claims travel inside the token, which means stale data until expiry; rotation is complicated (refresh tokens, blacklists).
  - The 2026 reflex for a server-rendered SaaS with a database the request already touches: sessions win on simplicity and correctness. The session lookup happens in the same database round trip as the rest of the request; the revocation story is trivial; the "JWT is faster" claim doesn't survive the first time someone needs to log a user out before the token expires.
  - When JWTs earn their weight: cross-service auth in a microservice architecture (the verification cost beats the cross-service DB call), public APIs where third parties hold tokens, mobile clients with offline-tolerant flows. None of those apply to the course's target architecture. Better Auth supports a JWT plugin for these cases; 9.2.3 names the trigger.
- **Tokens that aren't sessions — refresh, access, ID, verification.** The word "token" is overloaded. The lesson names the four families the student will encounter:
  - **Session token (this lesson's subject).** Opaque; cookie-carried; identifies the request.
  - **Access token.** OAuth's bearer credential for accessing a resource server's API — short-lived, sent in the `Authorization: Bearer` header. The thing OAuth providers return; Better Auth stores it for downstream use (calling the GitHub API on the user's behalf).
  - **Refresh token.** OAuth's long-lived credential for getting new access tokens without re-prompting the user. Stored server-side, rotated on use (9.1.3 covers rotation). Never travels in a cookie that the application reads on every request.
  - **Verification token.** A one-time, single-use token in an email link or SMS — the email-verification token (9.3.3), the password-reset token (9.3.4), the magic-link token (9.3.5). Stored hashed in the DB, expires in minutes, deleted on use.
  Each has a different lifetime, a different storage location, a different threat model. Conflating them is the bug.
- **Why the JWT-stored-in-localStorage pattern is wrong in 2026.** Older tutorials show storing a JWT in `localStorage` and attaching it via `Authorization` header in `fetch`. Two failures: `localStorage` is readable by any script that runs in the page, so a single XSS exfiltrates the credential (the whole point of `HttpOnly` cookies is to prevent this); and the server-side rendering path (Server Components, Server Actions) can't read `localStorage` without a client round trip, which defeats the architecture's first principle. The 2026 reflex: cookies, always, for browser-based session credentials.
- **The two CSRF lines of defense.** `SameSite=Lax` is one; the second is Better Auth's Origin/Referer check on state-changing requests (named here, owned by 9.4.4). Together they make CSRF on a same-origin Next.js app a non-issue without anti-CSRF tokens. The historical pattern — synchronizer tokens in a hidden form field — is named once as the "what you'd do if `SameSite` didn't exist" and dropped; modern browsers (post-2020, universally by 2026) all honor `SameSite`, and the cookie attribute is the primary mitigation.
- **Cookies under Next.js 16 — the read/write surface.** Server Components read cookies via the `cookies()` helper from `next/headers` (async-cookies API, 5.5.1); route handlers write cookies on the response; Server Actions write via the same `cookies()` helper. Better Auth abstracts the read on its side — the application calls `auth.api.getSession({ headers })` and never touches the cookie directly. The lesson names the underlying surface once so the student knows where the cookie lives in the platform.
- **Watch-outs.** A session cookie without `HttpOnly` is an XSS-to-account-takeover one-liner; a session cookie without `Secure` in production is a coffee-shop-wifi MITM one-liner; `SameSite=None` without `Secure` is silently dropped by modern browsers; sharing a session cookie across the apex and a marketing subdomain via `Domain=.yourapp.com` extends the credential's blast radius to any subdomain a junior spins up; rotating the session ID on privilege changes is a common omission — Better Auth handles it on password change but the senior reflex is to verify it on every state transition that matters; JWTs in `localStorage` are wrong for the course's architecture even when the tutorial that taught them was confident; reading `document.cookie` in client code to check sign-in state doesn't work for `HttpOnly` cookies and shouldn't — render the gated UI from the server.

What this lesson does not cover:

- The Better Auth `session` table schema (9.2.2).
- The session strategy decision tree at the library level (9.2.3).
- Session reads in middleware/layouts/actions (9.2.4).
- The OAuth flow that produces access and refresh tokens (9.1.3).
- The verification-token shapes for email and magic-link (9.3.3, 9.3.5).
- Origin/Referer checking and the full CSRF treatment at the cookie attributes layer is here; the chapter-level summary lives in 9.4.4.

Estimated student time: 45 to 55 minutes. The conceptual center of the chapter — every later session-related decision references this lesson.

## Lesson 9.1.3 — OAuth 2.1 and the authorization-code flow with PKCE

Topics to cover:

- **The senior question.** The app offers "Sign in with Google" and "Sign in with GitHub." When the user clicks the button, the browser leaves the app, lands at the provider, comes back with a code, and the next page renders signed in. What actually happens between the click and the cookie? What's the protocol the buttons drive, why does it have so many steps for what feels like a single action, what's the PKCE addition that distinguishes 2026's OAuth 2.1 from the older 2.0 examples the student saw, and which parts does Better Auth's social plugin (9.3.8) hide versus which parts does the senior still configure? The lesson installs the OAuth 2.1 mental model and the authorization-code-with-PKCE flow as the one path social sign-in takes in 2026.
- **What OAuth solves and what it doesn't.** OAuth is *delegated authorization*: the user authorizes the app to act on a resource owned by a third party (read your GitHub repos, fetch your Google profile). It is *not* an authentication protocol on its own — the fact that the user can produce an access token for `provider.com` doesn't, by itself, prove who they are to the app. OpenID Connect (OIDC) is the authentication layer on top of OAuth: the provider returns an `id_token` (a JWT) alongside the access token, which carries the user's verified identity. "Sign in with Google" is OIDC; "Connect your GitHub" for API access is plain OAuth. Most modern social providers (Google, Microsoft, Apple) speak OIDC; GitHub speaks OAuth and exposes a `/user` endpoint the app calls separately to get identity. Better Auth's social plugin handles both shapes uniformly.
- **OAuth 2.0 to 2.1 — what consolidated.** OAuth 2.1 is the security-hardened consolidation of OAuth 2.0 plus a decade of best-current-practice RFCs. The deltas relative to older tutorials:
  - **PKCE is mandatory for all authorization-code clients** (not just public SPAs and mobile — confidential server-side clients too).
  - **Implicit grant (`response_type=token`) is removed.** The pattern that returned a token directly to the browser via URL fragment is dead; authorization code is the only flow.
  - **Resource Owner Password Credentials grant is removed.** The user never types their provider password into the relying app.
  - **Redirect URIs must match by exact string comparison.** No wildcards, no partial matches — covered below as the open-redirect mitigation.
  - **Refresh token rotation is required for public clients** and recommended universally — covered below.
  - **Bearer tokens in `Authorization` header only,** never in the URL query string (which leaks via referer and logs).
  The student should write to OAuth 2.1's profile; older "OAuth 2.0 in 5 steps" tutorials usually skip PKCE and one or more of the others.
- **The actors and the surface they speak.** Four parties:
  - **Resource owner** — the user.
  - **Client (relying party)** — the app (`yourapp.com`).
  - **Authorization server** — the provider's auth endpoint (`accounts.google.com`).
  - **Resource server** — the provider's API (`graph.microsoft.com`, `api.github.com`). Often the same host as the authorization server, conceptually distinct.
  The client and authorization server exchange messages over HTTP redirects and a back-channel POST; the user sees only the provider's consent screen.
- **The flow, end to end (PKCE included).** Eight steps; the student should be able to trace one full cycle.
  1. **PKCE prep on the client.** Before redirecting, the app generates a random `code_verifier` (43–128 chars from the unreserved set) and computes `code_challenge = base64url(sha256(verifier))`. The verifier is stored server-side (Better Auth keeps it in a temporary cookie or session-state record); the challenge is sent in the URL.
  2. **Authorization request.** The app redirects the browser to the provider's `/authorize` endpoint with query parameters: `response_type=code`, `client_id`, `redirect_uri`, `scope` (the permissions requested — for OIDC sign-in this is `openid email profile`), `state` (a random anti-CSRF token the app stores and verifies on return), `code_challenge`, `code_challenge_method=S256`, and for OIDC, an optional `nonce`.
  3. **User authenticates at the provider.** The provider's UI takes over — the user signs in with their provider credentials (which the app never sees), reviews the requested scopes, and approves.
  4. **Provider redirects back with a code.** The provider issues a short-lived authorization code (typically 30–60 seconds, single-use) and redirects the browser to the registered `redirect_uri` with `?code=...&state=...`. The browser arrives at the app's OAuth callback route.
  5. **State and PKCE verification on the callback.** The app verifies the returned `state` matches what it stored (CSRF defense for the OAuth flow itself), looks up the matching `code_verifier`, and confirms the redirect URI is the expected one.
  6. **Token exchange.** The app POSTs to the provider's `/token` endpoint with `grant_type=authorization_code`, the `code`, the `client_id`, the `client_secret` (for confidential clients), the `redirect_uri`, and the `code_verifier`. This call is back-channel — server to server, the browser is not in the loop. The provider verifies the code is valid, unused, issued for this client, paired with the redirect URI, and that `sha256(verifier) === stored_challenge`. If all pass, it returns `access_token`, `refresh_token`, `expires_in`, `token_type=Bearer`, and (for OIDC) `id_token`.
  7. **Identity extraction.** For OIDC, the app verifies the `id_token` JWT signature against the provider's JWKS, checks the `iss`, `aud`, `nonce`, and `exp` claims, and reads `sub`, `email`, `name`, `picture`. For non-OIDC OAuth (GitHub), the app calls the provider's `/user` endpoint with the access token to fetch identity.
  8. **Local session creation.** The app finds or creates a `users` row for the provider's `sub`, creates an `accounts` row (Better Auth's table, owned in 9.2.1, holding the provider name, the provider's user ID, the access and refresh tokens, the scopes), and creates a `session` row plus session cookie (per 9.1.2). The user is now signed in locally.
  Diagram: sequence diagram across browser, app, provider's authorization server, and provider's resource server, with PKCE values highlighted at steps 1, 2, and 6 (Mermaid).
- **PKCE in one paragraph.** PKCE (Proof Key for Code Exchange, RFC 7636, pronounced "pixie") binds the authorization code to the specific client instance that started the flow. Without it, an attacker who intercepts the authorization code (via a malicious app on the same device, a logged URL, a referer leak) could exchange it for tokens. With PKCE, the token endpoint requires the original `code_verifier` — which only the legitimate client knows — and rejects the exchange otherwise. The cryptographic guarantee: an attacker would need both the code *and* the verifier, but the verifier never leaves the client until the back-channel POST, where it's not interceptable by anything the attacker controls.
- **`state` — the CSRF defense for the OAuth flow.** PKCE protects the token exchange; `state` protects the redirect callback from CSRF (an attacker tricking the user's browser into landing on the callback URL with the attacker's code). The app stores a random `state` value at step 2, the provider echoes it back at step 4, the app verifies the match at step 5. Mismatch → reject. Better Auth handles both `state` and PKCE automatically; the senior verifies the library's default and doesn't disable it.
- **Scopes — granular permission requests.** The `scope` parameter lists the permissions the app needs. For sign-in, `openid email profile` is the standard set. For access to provider APIs, additional scopes like `read:user` (GitHub) or `https://www.googleapis.com/auth/calendar.readonly` (Google) extend the access token's reach. The senior reflex: request the minimum scope set that the feature needs and add scopes incrementally — the consent screen growing scarier on a feature release is a churn-cost. Scopes are provider-specific; Better Auth exposes a `scope` array per provider in the config.
- **Redirect URI registration and exact matching.** The app registers a `redirect_uri` (or a list) at the provider's developer console; the provider rejects any auth request whose `redirect_uri` doesn't match by exact string comparison. The 2026 reflex:
  - Register `https://yourapp.com/api/auth/callback/google` for production.
  - Register `https://preview-*.yourapp.com/api/auth/callback/google` *only if the provider supports wildcards* (most don't in 2.1) — alternatives include using a single stable production URL and routing previews differently, or using a per-environment OAuth app.
  - Never register `http://localhost:3000/...` and `https://yourapp.com/...` on the same OAuth client — dev and prod get separate clients and separate `client_id`/`client_secret` pairs.
  Mismatched redirect URIs are the most common "it works locally but fails in prod" failure mode for social sign-in; the senior debug step is "check what the provider sees vs. what's registered."
- **Open-redirect via OAuth — the historical bug.** Older OAuth implementations let the app pass a `redirect_uri` not from a strict allowlist; an attacker would craft a flow that redirects through the app and lands the user on `evil.com` with the access token in the fragment. OAuth 2.1's exact-match registration plus PKCE plus removal of implicit grant kill this class. The senior reflex: never trust a `redirect_uri` from a query parameter on your own side either — the post-sign-in "redirect back to where the user was" pattern uses a separate, validated internal `?next=` parameter (the rules: relative paths only, no protocol, no `//` prefix; covered in 5.5.3).
- **Refresh tokens and rotation.** The access token expires in minutes to an hour; the refresh token lives for days to months and is used to get new access tokens without re-prompting. OAuth 2.1's rule: refresh tokens *rotate* — every use returns a new refresh token, the old one is invalidated. If an attacker steals a refresh token and uses it, the legitimate user's next use fails (the new token from the attacker's exchange doesn't match what the user has), which surfaces the breach. The app stores the current refresh token in the `accounts` table (9.2.1) and updates it on each refresh. Better Auth handles the rotation transparently.
- **The `id_token` (OIDC) — claims and verification.** When the provider speaks OIDC, the `id_token` is a JWT carrying:
  - `iss` — the provider's issuer URL (must match the configured expected issuer).
  - `aud` — the audience, must equal the app's `client_id`.
  - `sub` — the provider's stable user ID (the value the app stores in `accounts.providerAccountId`).
  - `iat`, `exp` — issued-at, expires-at; verify `exp > now`.
  - `nonce` — the value the app sent at step 2; verify it matches.
  - `email`, `email_verified`, `name`, `picture`, locale, etc. — identity claims.
  The JWT is signed by the provider's private key; the app fetches the public key from the provider's JWKS endpoint (the URL is in the provider's discovery document at `/.well-known/openid-configuration`). The verification is canned — Better Auth does it — but the senior should know what's being verified so a misconfigured `aud` is recognizable in logs.
- **What Better Auth abstracts and what the app still configures.** Better Auth's `socialProviders` config takes a per-provider object with `clientId`, `clientSecret`, optional `scope`, and optional `mapProfileToUser`. The library handles PKCE generation, state, the redirect-and-callback routes (`/api/auth/callback/<provider>`), token exchange, ID-token verification, the `accounts` row, the session creation, and refresh-token rotation. The app configures the OAuth client at the provider, the env vars, and the optional profile mapper. The 9.3.8 lesson is the wire-up; this lesson installs the protocol model.
- **Provider-specific quirks, named once.** Each provider has at least one footgun: Google requires explicit consent prompts to re-prompt for scopes (`prompt=consent`); GitHub's `email` is sometimes null until the app calls `/user/emails` separately; Apple Sign In returns the user's name *only* on the first auth and never again; Microsoft has separate `common`, `consumers`, and `organizations` tenants. The senior reach: read the provider's quickstart once, then trust Better Auth's adapter for the rest. The chapter doesn't enumerate quirks; 9.3.8 names them per-provider as they apply.
- **What this lesson does not enable.** No code at all. The student leaves with the model — actors, steps, parameters, PKCE math, why each piece exists — so that when 9.3.8 walks `socialProviders: { google: { clientId, clientSecret } }`, they understand which step that config drives and which configuration on Google's side it pairs with.
- **Watch-outs.** Skipping PKCE because "we're a confidential server-side client" is a 2026 anti-pattern — OAuth 2.1 mandates it for all clients; trusting a `state` value the app didn't generate is the OAuth callback CSRF; trusting an `id_token` without verifying the signature, `iss`, `aud`, and `exp` is a forgery vulnerability; requesting broad scopes upfront ("just in case we add the feature later") trains users to dismiss consent screens and triggers provider review for sensitive scopes; reusing one OAuth client across environments leaks production secrets to PR previews; storing the access token in `localStorage` or a non-`HttpOnly` cookie negates every protection the flow built up — store provider tokens server-side in the `accounts` table only; the implicit grant (`response_type=token`) and the password grant exist in older docs, in OAuth 2.1 they're deprecated/removed — refuse to copy them.

What this lesson does not cover:

- The Better Auth `socialProviders` config and the per-provider env vars (Chapter 9.3.8).
- Account linking — one user, multiple provider identities (9.3.9).
- Provider-specific quickstart steps (Chapter 9.3.8).
- Calling provider APIs with the stored access token (Chapter 9.3.8 names the pattern; deep integrations are out of scope).
- OAuth as a *provider* — exposing your app's identity to other apps (Better Auth's OIDC Provider plugin; out of scope, named once and dropped).
- Device authorization grant and machine-to-machine flows — out of scope.
- SAML, the older enterprise SSO protocol — out of scope.

Estimated student time: 40 to 50 minutes. Protocol-walkthrough lesson — the diagram is the centerpiece; the student should be able to trace the eight steps unprompted by the end.

## Lesson 9.1.4 — Chapter quiz

Top 10 topics to quiz:

- Authentication vs. authorization — the two questions, the 401/403 wire-protocol mapping, and which Unit owns each (Unit 9 for authn, Unit 10 for authz).
- The order rule — authn first, authz second, always — and the "UI gating is not enforcement" reflex that puts the check at the action boundary.
- The session-and-cookie model — the session as a server-side row, the cookie as an opaque pointer, the four lifecycle events (creation, read, rotation, termination).

- Session vs. JWT — the threshold that flips the choice and why server-stored sessions are the 2026 default for a server-rendered SaaS.
- Token taxonomy — session, access, refresh, verification — and the rule that each has a different lifetime, storage, and threat model.
- OAuth 2.1's mandatory changes — PKCE for all clients, exact redirect URI matching, implicit grant removed, refresh token rotation, bearer tokens in headers only.
- The authorization-code-with-PKCE eight-step flow — PKCE prep, authorize redirect, user consent, callback with code, state/PKCE verification, token exchange, identity extraction, local session creation.
- OAuth vs. OIDC — delegated authorization vs. authentication on top; what `id_token` adds and which claims to verify.
- The `state` parameter and the redirect URI exact-match rule — the two non-PKCE CSRF and open-redirect defenses OAuth 2.1 mandates.
