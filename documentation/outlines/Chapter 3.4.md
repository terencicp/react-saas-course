# Chapter 3.4 — Cookies and the trust model (light)

## Chapter framing

Chapter 3.1 installed the request lifecycle. Chapter 3.2 installed the HTTP contract — methods, status codes, headers — and named the `Cookie` and `Set-Cookie` header pair once, deferring the trust model here. Chapter 3.3 installed the origin and same-origin policy and named `SameSite` as the modern primary CSRF defense, also deferring to here. This chapter resolves both forward references. It installs cookies as a header-pair the browser stores and replays automatically, at the depth that makes the auth chapter (Unit 9) and the security baseline (Chapter 17.2) concrete — but no deeper. The student leaves with the vocabulary to read a `Set-Cookie` header by sight, the discipline to set every attribute that matters and skip the ones that don't, and the recognition of the failure modes that bite: the cookie that won't set because `Secure` is on an HTTP origin, the cross-site fetch that drops the cookie because `SameSite=Lax`, the session that leaks because `HttpOnly` is missing, the cookie that survives a subdomain takeover because `Domain` widened the scope unnecessarily.

The senior framing: **a cookie is a header pair the browser stores and replays automatically, scoped by host and path, with attributes that govern when it's sent and who can read it.** The senior designs the attribute set before they write the `Set-Cookie` line. The default for any cookie a SaaS app sets is `HttpOnly; Secure; SameSite=Lax; Path=/`, with `Domain` omitted unless subdomains need it and the `__Host-` prefix on the cookie name when the constraints fit (no `Domain`, `Path=/`, `Secure`, HTTPS-only). Every deviation from that default has a senior trigger — `SameSite=Strict` for the few flows where cross-site navigation must drop the cookie, `SameSite=None; Secure; Partitioned` for the rare cross-site cookie that genuinely needs to ride a third-party context, no `HttpOnly` for the cookie the client framework explicitly needs to read. The chapter installs the defaults and names each trigger so the student doesn't reach for a power tool by reflex.

This chapter is one teaching lesson plus a quiz. The TOC's structure holds — cookies is conceptual surface, not a wiring lesson; the wiring lands at Better Auth in Chapter 9.2 and the request-time reads in Chapter 9.4. The lesson is meaningful on its own — by the end the student names every attribute on a `Set-Cookie` header, picks the right `SameSite` value for any cross-site scenario the application will face, knows why the `__Host-` prefix earns its weight on the session cookie, and recognizes the cookie-related failure shapes (the missing cookie on the cross-origin fetch, the cookie that's exposed to XSS because `HttpOnly` was forgotten, the `Set-Cookie` the browser silently rejected because attributes conflicted).

Threads that must run through the lesson:

- **Cookies are a header pair the browser manages automatically.** The server sends `Set-Cookie` on a response; the browser stores the cookie keyed by `(host, path, name)` per the attributes; on every matching request the browser attaches a `Cookie` header. The application never reads a cookie from a request body or writes one with `document.cookie` in 2026 SaaS code — the framework's helpers (`cookies()` in Next.js) wrap the header surface. The chapter names the headers as the underlying protocol; the framework surface is what the student writes against in Unit 5 onward.
- **The senior default is a tight set: `HttpOnly; Secure; SameSite=Lax; Path=/`.** Every cookie the SaaS app sets gets all four unless a specific trigger flips one. `HttpOnly` is non-negotiable for any cookie that carries auth or identity — JavaScript reading the session cookie is the XSS-becomes-account-takeover bug class. `Secure` is non-negotiable in production — a cookie sent over HTTP is a cookie sniffable on any shared network. `SameSite=Lax` is the modern browser default and the primary CSRF defense; the senior names it explicitly so the cookie's behavior doesn't depend on the browser's default-attribute decision. `Path=/` is the senior default unless the cookie is genuinely scoped to a subtree (rare in a SaaS where every page reads the session).
- **`__Host-` is the senior reach for the session cookie.** The prefix is a structural enforcement that prevents a class of subdomain-takeover and cookie-injection bugs. Naming the cookie `__Host-session` (and meeting the constraints — `Secure`, no `Domain`, `Path=/`) makes the browser refuse any `Set-Cookie` that doesn't honor those constraints. The cookie can't be widened to a parent domain accidentally, can't be set by a subdomain takeover attacker, and can't be set on plain HTTP. Named once at the recognition depth the auth chapter (Unit 9) will lean on.
- **Cookie scope is host + path, not origin.** The most-confused point in the chapter. The same-origin policy uses `(scheme, host, port)`; cookies use `(host, path)` — the scheme is governed by `Secure`, the port is irrelevant, and the path is governed by `Path`. A cookie set on `https://app.example.com:3000` is sent to `http://app.example.com:8080/anything` (different port, different scheme) if `Secure` is absent — a real bug class. The senior reach: name the cookie's scope explicitly, and use `__Host-` to enforce the strictest scope.
- **`SameSite` is the 2026 primary CSRF defense.** Chapter 3.3 named it; this chapter installs the values. `Lax` (the modern browser default; the senior default to set explicitly) refuses the cookie on cross-site POSTs but allows it on top-level GET navigations — the right balance for ordinary SaaS auth flows where the user lands on the app from external links. `Strict` refuses every cross-site request, even the user clicking a link to the app from email or social — too aggressive for most flows; reserve for the rare cookie that should never ride a cross-site navigation. `None` (with mandatory `Secure`) disables `SameSite` entirely — the trigger is genuine cross-site contexts (an embedded widget, a partner integration calling your API with the user's cookie). The 2026 senior reach: `None` cookies almost always need `Partitioned` too because the browser's third-party cookie phase-out otherwise blocks them.
- **The third-party cookie phase-out is the 2026 reality.** Safari (ITP) has blocked third-party cookies by default for years; Firefox (Total Cookie Protection) partitions them; Chrome ran the deprecation through 2024-2025 and now uses Privacy Sandbox primitives. The CHIPS proposal (`Partitioned` attribute) is the explicit opt-in for cookies that *should* ride third-party contexts — embedded chat widgets, SSO iframes — by keying the cookie's storage on the top-level site. The lesson installs the model at recognition depth; the course's stack (Next.js same-origin SaaS) rarely sets `Partitioned` cookies, but the student recognizes the attribute when reading docs and understands why the cross-site flows that *used* to work silently broke.
- **`document.cookie` is a 2010-era reach.** The lesson names it once for recognition — JavaScript can read non-`HttpOnly` cookies via `document.cookie`, write cookies via the same surface — and dismisses it as a 2026 senior reach. The application reads cookies through the framework's request-side helper (`cookies()` in Next.js Server Components and Route Handlers); the application writes cookies through the framework's response-side helper (`cookies().set()` in Server Actions and Route Handlers). The `document.cookie` API is for transient client-only flags the application owns explicitly — the cookie consent toggle, a theme preference where `localStorage` isn't the right reach — and even those are usually better off in `localStorage` or a server-side cookie.
- **Cookies are not the only storage option.** The lesson names the alternatives once for the boundary: `localStorage` for client-only, no-server-replay storage (theme, recently-viewed list); `sessionStorage` for tab-scoped transient state; IndexedDB for structured client data. The senior reach: cookies for anything the server needs (auth, the user's locale preference the SSR reads, the cookie consent decision); local storage primitives for client-only state. The full storage discussion lands in Chapter 3.7.5; this lesson names cookies' boundary.
- **CSRF is the failure mode `SameSite` addresses; XSS is the failure mode `HttpOnly` addresses.** Two different attacks, two different mitigations. The lesson names both at the depth that lets the student pick the right attribute for the right reason and points forward to Chapter 17.2 for the full security-baseline pass.
- **Senior anchors for later units are seeded here.** The `__Host-` session cookie lands at Better Auth in Chapter 9.2. `cookies().set()` and `cookies().get()` from Next.js land at the request-time auth reads in Chapter 9.4. The `SameSite=None; Secure` requirement for cross-origin credentialed fetches lands at Chapter 3.6 (Fetch) and any cross-origin API surface (Chapter 7.5). The cookie consent gate lands at Chapter 17.2.5 and the PostHog wiring in Chapter 20.2.4. The cookie expiration / max-age discussion lands at Better Auth's session lifecycle in Chapter 9.2.3.

This chapter ships short code snippets — a `Set-Cookie` header line, a small Next.js Route Handler reading and writing cookies through `cookies()`, a couple of `Aside` callouts naming the prefix rules — and a handful of `Buckets`, `Matching`, and `CodeReview` exercises for the discrimination work. A `Figure` with a hand-authored SVG renders the `Set-Cookie` anatomy with each attribute labeled; a `TabbedContent` block organizes the `SameSite` values as a decision card. The chapter is short by design — one teaching lesson plus the quiz — because the wiring lessons that consume this surface (Better Auth in Unit 9, the security baseline in Chapter 17.2) are where the cookie model earns its weight in code. Here, the deliverable is recognition vocabulary and the senior defaults.

## Lesson 3.4.1 — Cookies: attributes, scope, and the senior defaults

Topics to cover:

- The chapter-opening senior question: the student is wiring an auth flow. The server needs to remember the user across requests. The browser will send headers; the server will set headers. What's the senior shape of the cookie that survives this round-trip without leaking the session to XSS, riding a cross-site CSRF post, or surviving a subdomain takeover? The naive reach is `Set-Cookie: session=abc123`. The senior reach names four attributes the line must carry (`HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`) plus the `__Host-` prefix on the cookie name. The lesson installs the model, the attribute palette, and the senior defaults. Auth (Unit 9) and the security baseline (Chapter 17.2) consume this directly.
- **The cookie model in one paragraph.** A cookie is a name-value pair the server sends to the browser via the `Set-Cookie` response header. The browser stores it keyed by `(host, path, name)` plus the attribute set, and on every subsequent request to a URL that matches the cookie's scope, the browser attaches a `Cookie` request header with the name-value pairs concatenated by `; `. The server reads `Cookie` to find the value. The protocol is in RFC 6265bis (the modernization of RFC 6265, 2011); the 2026 senior reach is to know the attributes that matter and let the framework (Next.js's `cookies()` helper, Better Auth's session writer) emit the bytes.
- **The `Set-Cookie` header anatomy**, walked with a canonical example:
  ```
  Set-Cookie: __Host-session=abc123; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000
  ```
  Named left to right:
  - **`__Host-session=abc123`** — the name-value pair. The name is the cookie's identifier (within the scope); the value is opaque to the browser (the server validates it). The `__Host-` prefix is a structural enforcement (see below).
  - **`HttpOnly`** — refuses JavaScript access via `document.cookie` (see below).
  - **`Secure`** — refuses to attach to non-HTTPS requests (see below).
  - **`SameSite=Lax`** — controls cross-site attachment (see below).
  - **`Path=/`** — scopes which URLs on the host the cookie attaches to (see below).
  - **`Max-Age=2592000`** — the lifetime in seconds (30 days; see below).
- **Reading and writing cookies in 2026 SaaS code**, named where the student will write them:
  - In **Next.js 16 Server Components and Route Handlers**, the request-side surface is `await cookies()` (the helper is async since Next.js 15.0, like `params` and `headers()`). `(await cookies()).get('__Host-session')?.value` returns the cookie value. Setting is allowed in Server Actions, Route Handlers, and Middleware — not in Server Components (Server Components are render-only).
  - In **Server Actions and Route Handlers**, `(await cookies()).set({ name: '__Host-session', value: token, httpOnly: true, secure: true, sameSite: 'lax', path: '/', maxAge: 60 * 60 * 24 * 30 })` writes the cookie. The framework serializes to the `Set-Cookie` header. The `cookies()` helper's `.delete(name)` removes a cookie (by emitting a `Set-Cookie` with `Max-Age=0`).
  - In **middleware (Chapter 5.5 owns the depth)**, `request.cookies.get(name)` reads, `response.cookies.set(...)` writes — the request and response objects each have their own cookie surface so middleware can read incoming cookies and write outgoing ones in the same pass.
  - In **client-side JavaScript**, `document.cookie` returns all non-`HttpOnly` cookies for the current document as a single string (`'a=1; b=2'`), and assigning to it writes a single cookie. The 2026 senior reach: don't use `document.cookie` in application code — the cookies the application cares about are `HttpOnly` and unreadable from JavaScript anyway. Named once for recognition.
- **`HttpOnly`** — the attribute that refuses JavaScript access:
  - **What it does.** When set, the cookie is invisible to `document.cookie` and to any JavaScript API. The browser still sends the cookie on requests, but the page's scripts can't read it.
  - **Why it earns its weight.** Cross-Site Scripting (XSS) is the bug class where an attacker injects JavaScript into the application's page — through an unsanitized user comment, a vulnerable third-party script, an `innerHTML` from user input. Without `HttpOnly`, the injected script reads `document.cookie` and exfiltrates the session token. The user is now compromised. `HttpOnly` makes the session unreadable to the injected script — XSS still bites (the attacker can act *as* the user via the same-origin fetch, which sends the cookie automatically), but the session itself is never exposed.
  - **The senior default.** `HttpOnly` is set on every cookie that carries authentication, identity, or any sensitive state. The exceptions are cookies the *client* explicitly needs to read — the cookie consent decision (so JavaScript can gate analytics before the next page load), a feature flag the client reads to decide UI state. Even those are usually better off as a non-cookie storage (`localStorage`) or as a state the server reflects into the page's initial HTML.
  - **The watch-out.** `HttpOnly` is not a substitute for XSS prevention. The injected script can still make authenticated requests *as* the user via `fetch('/api/transfer-funds', { credentials: 'include' })`. Chapter 17.2's CSP (Content Security Policy) is the structural defense against XSS getting injected in the first place; `HttpOnly` is the second line of defense that limits the blast radius. Both belong in the security baseline.
- **`Secure`** — the attribute that refuses HTTP attachment:
  - **What it does.** When set, the browser only attaches the cookie to requests over HTTPS. An HTTP request to the same host gets *no* cookie. Set-Cookie attempts on HTTP origins are silently dropped (the browser refuses to store a `Secure` cookie set over HTTP).
  - **Why it earns its weight.** A cookie sent on a plain-HTTP request is sniffable on any shared network (coffee shop Wi-Fi, mobile hotspot, ISP-level interception). The session token is then a credential the attacker can replay. `Secure` makes the cookie HTTPS-only; the attacker on the network sees encrypted bytes, not the cookie.
  - **The senior default.** `Secure` is set on every cookie in production. The dev-time bite: a `Secure` cookie set on `http://localhost` won't store. The fix is `mkcert` to run dev on `https://localhost` (Chapter 3.1.3) — the seam this chapter cashes in on. The course's reference SaaS sets `Secure` on every cookie from day one; the student has `mkcert` installed since Chapter 3.1.3 specifically so this works.
  - **The watch-out.** `Secure` only controls the *attachment* of the cookie to requests — it doesn't protect the cookie from JavaScript on an HTTPS page. `HttpOnly` is the separate attribute for that. Both belong on every auth cookie.
- **`SameSite`** — the attribute that controls cross-site attachment. The 2026 primary CSRF defense, named in Chapter 3.3 and installed here:
  - **`SameSite=Lax`** — the modern browser default. The cookie is sent on:
    - Same-site requests (any request where the request's site matches the cookie's site).
    - **Top-level GET navigations** from any site (the user clicked a link to the app, typed the URL, or was redirected from an OAuth provider).
    The cookie is *not* sent on cross-site `POST`s, image embeds, iframe loads, or `fetch`/`XMLHttpRequest` from another site. This blocks the canonical CSRF attack (evil.com posts a form to bank.com hoping the user's cookie rides along) while preserving the user's experience of clicking a link in an email and landing on the app already signed in.
    - **The "lax-by-default" 2-minute window**, named once. Chromium's implementation has a quirk: when a cookie has *no* `SameSite` attribute at all, Chrome treats it as `Lax` but also allows cross-site `POST` for 2 minutes after the cookie was set (a compatibility carve-out for legacy auth flows). When the cookie sets `SameSite=Lax` explicitly, the carve-out is gone. The senior reach: set `SameSite=Lax` explicitly to opt out of the carve-out and get strict Lax semantics.
  - **`SameSite=Strict`** — the cookie is sent only on same-site requests. Even a user clicking a link from an external page to the app *won't* send the cookie on the first request. The application sees an unauthenticated request and redirects to login; the user signs in; the cookie is now set; subsequent requests carry it.
    - The trigger. Reserve `Strict` for the cookie whose presence on a top-level navigation is itself a security concern — a password-reset confirmation page, a sensitive admin action that should refuse any cross-site entry. Almost never the right default for the main session cookie; the user-experience cost (every external link reverts to a sign-in page) is too high.
  - **`SameSite=None`** — disables `SameSite` entirely. The cookie is sent on cross-site requests (subject to the third-party cookie phase-out; see below).
    - **`Secure` is mandatory.** All major browsers since 2020 reject `SameSite=None` without `Secure`. The cookie won't store.
    - The trigger. Genuine cross-site contexts — an embedded chat widget that runs in a `<iframe>` on partner sites and needs the user's session to render their messages, a partner integration that calls the SaaS API from a different origin with the user's cookie. The dev-time corollary: a Next.js app on `localhost:3000` calling an API on `localhost:4000` is technically cross-site by spec but the senior preference is to proxy through Next.js Route Handlers (Chapter 3.3.3) so production stays single-origin.
    - **`Partitioned` is almost always required alongside.** See CHIPS below.
- **`Partitioned` (CHIPS)** — the 2026 attribute for cross-site cookies that should still work:
  - **What it does.** Marks the cookie as partitioned by the *top-level site* — the cookie's storage key includes the top-level site's origin, not just the cookie's own host. A widget embedded on `partner1.com` and on `partner2.com` sees a different partitioned cookie on each, even though both load the same iframe URL. The cookie doesn't ride across top-level sites — it's effectively scoped to the embedded context.
  - **Why it exists.** Third-party cookies (cookies set in a cross-site context, like an iframe on a partner page) are blocked by Safari's ITP (since 2017), partitioned by Firefox's Total Cookie Protection, and progressively restricted by Chrome's Privacy Sandbox. The CHIPS proposal (`Partitioned` attribute) is the explicit opt-in for cookies the developer *wants* to be third-party but stored per-top-level-site — preserving the embed's per-site state without enabling cross-site tracking.
  - **The mandatory pairing.** `Partitioned` requires `Secure`, `SameSite=None`, and a host-scoped cookie (the `__Host-` prefix is recommended). The browser rejects the cookie if these aren't met.
  - **The course's reach.** The course's reference SaaS is same-origin — the auth cookie is first-party on the SaaS host and never needs `Partitioned`. The lesson installs the attribute for recognition: the student knows what it does when they read Stripe's checkout iframe docs or wire an embedded widget.
- **`Path`** — the URL-prefix scope on the host:
  - **What it does.** The cookie is attached only to requests whose URL pathname starts with the `Path` value. `Path=/` (the senior default) attaches to every request on the host. `Path=/admin` attaches only to `/admin/*` URLs.
  - **Why the senior default is `/`.** A SaaS session is checked on every page; scoping it to a path subtree just creates surprises ("why didn't my session ride this request?"). The application's authorization model — RBAC, plan entitlements, per-route gating — is at the application layer, not the cookie's scope.
  - **The watch-out.** `Path` is *not* a security boundary. JavaScript on `/admin/page` can read `document.cookie` for cookies scoped to `/` (or to `/admin`); the same-origin policy doesn't partition cookies by path. Don't rely on `Path` for separating tenant cookies — it only controls *attachment*, not *visibility within the page*. (`HttpOnly` is the visibility control; `Path` is just attachment scope.)
- **`Domain`** — the host-scope attribute:
  - **What it does without a `Domain` attribute** (the senior default). The cookie is "host-only" — it attaches only to requests for the exact host that set it. A cookie set on `app.example.com` does *not* attach to requests for `example.com` or `api.example.com`.
  - **What it does with a `Domain` attribute.** The cookie attaches to the named domain *and all its subdomains*. `Domain=example.com` set by `app.example.com` makes the cookie ride on `app.example.com`, `api.example.com`, `marketing.example.com`, and `example.com` itself.
  - **The senior default: omit `Domain`.** A cookie that's wider than needed is a cookie that leaks more than needed. The session cookie for the SaaS is set on the SaaS host and stays there. The exception is cross-subdomain auth flows — the user signs in at `accounts.example.com` and the session needs to ride on `app.example.com` — where `Domain=example.com` is the explicit reach. The 2026 senior preference even there: a same-origin auth flow on the app's primary host (skip the accounts subdomain entirely) so the cookie stays host-only and the `__Host-` prefix can apply.
  - **The watch-out.** A `Domain` cookie can be set or modified by *any subdomain* of that domain — a takeover of `forgotten.example.com` (an abandoned staging site, a content-served-by-third-party blog) can set or overwrite the session cookie for `app.example.com`. The mitigation is to omit `Domain` (the cookie stays host-only) and use the `__Host-` prefix (which forbids `Domain` structurally).
- **Cookie name prefixes** — structural enforcement of the right attribute set:
  - **`__Secure-` prefix.** Any cookie whose name starts with `__Secure-` is rejected by the browser unless it's set over HTTPS with the `Secure` attribute. Makes "this cookie is HTTPS-only" enforceable at the protocol level.
  - **`__Host-` prefix.** Stricter. Any cookie whose name starts with `__Host-` is rejected unless: set over HTTPS, has `Secure`, has no `Domain` attribute (so it's host-only), and has `Path=/`. The prefix is the senior reach for the session cookie because it makes the four guarantees structurally — the cookie is HTTPS-only, host-only (no subdomain widening), and host-root-pathed.
  - **The senior default.** The session cookie is named `__Host-session` (or whatever Better Auth's convention is — Unit 9 names the canonical form). Auxiliary cookies that need `Domain` (the rare cross-subdomain case) use the `__Secure-` prefix instead. Cookies without any prefix are non-sensitive (consent gate, theme preference).
  - **Recent attack research** — the `__Host-` and `__Secure-` prefix guarantees can be bypassed in narrow conditions (a misconfigured load balancer that strips attributes, certain proxy chains). Named once for recognition — the prefixes are an additional structural layer, not a substitute for the rest of the attribute set.
- **Expiration** — the cookie's lifetime:
  - **Session cookies** — no `Max-Age` or `Expires` attribute. The cookie is deleted when the browser session ends. The watch-out: "browser session" is increasingly browser-dependent — Chrome's "continue where you left off" feature preserves session cookies across browser restarts, defeating the model the developer assumed. The senior reach for "log the user out when they close the browser" is *not* a session cookie; it's a short `Max-Age` plus a server-side session record the application can revoke.
  - **`Max-Age=N`** — the cookie expires N seconds after it was set. The senior reach in 2026 — `Max-Age` is a clean integer the server emits and the browser interprets in its own clock-skew-aware way.
  - **`Expires=<HTTP date>`** — the cookie expires at the absolute date. The legacy form; works but depends on the client's clock being correct. If both are set, `Max-Age` wins. The senior reach: `Max-Age` only.
  - **The senior pattern.** Auth cookies have a finite `Max-Age` (Better Auth's default is 7 days; Chapter 9.2.3 owns the senior call). The session is also revocable server-side (the session table — Chapter 9.2.2) so a stolen cookie can be invalidated even before its `Max-Age` runs out. The cookie alone is not the session of record; the database is.
  - **Deletion.** The application deletes a cookie by emitting `Set-Cookie: name=; Max-Age=0; <same scope attributes>`. The browser removes the cookie. The senior watch-out: the deletion `Set-Cookie` must match the original's scope attributes (`Path`, `Domain`, `Secure`) — a delete with a different `Path` doesn't reach the cookie. Next.js's `cookies().delete(name)` handles this for cookies the framework set; manually-set cookies need manual deletion.
- **The `Cookie` request header**, named for symmetry. On each matching request, the browser concatenates every cookie that matches the request's URL (the host matches, the scheme matches `Secure`'s requirement, the path matches `Path`, the request's site matches `SameSite`'s rule) into a single `Cookie` header: `Cookie: __Host-session=abc123; consent=accepted; locale=en-US`. The framework parses this into a map; the application reads cookies by name. The senior watch-out: the `Cookie` header can grow long with many cookies, and some load balancers cap header size — keep the cookie set tight.
- **`document.cookie`**, named for recognition and dismissed:
  - The JavaScript surface for client-side cookie access. Reading returns a single concatenated string (the same format as the `Cookie` header) of non-`HttpOnly` cookies. Writing sets a single cookie via a serialized string (`document.cookie = 'name=value; Max-Age=3600; Path=/; Secure; SameSite=Lax'`).
  - The 2026 senior reach: don't use it. The cookies that matter are `HttpOnly` and invisible to JavaScript. The cookies JavaScript can read are the ones the application explicitly owns (consent, theme) — and those are better served by `localStorage` (Chapter 3.7.5) for client-only state or by a server-set cookie the SSR reflects into the initial HTML.
  - The exception. The very narrow case where the application has a non-`HttpOnly` cookie (the cookie consent decision) and the client needs to read it before the next page load to gate analytics. Named once for recognition.
- **The `cookieStore` API**, named in one paragraph as the 2026 future. The Cookie Store API (`window.cookieStore`) is the modern, Promise-based replacement for `document.cookie`. Better ergonomics (`await cookieStore.get('consent')`), event-based change notifications, async by default. As of 2026, supported in Chromium browsers (Chrome, Edge) but not yet in Firefox or Safari. Named for recognition; the application doesn't write against it in 2026 — the framework's server-side helpers are still the right reach. The senior watches the support matrix for when this becomes the default.
- **CSRF and `SameSite`**, the boundary point. Chapter 17.2 owns the full CSRF treatment; this lesson installs the cookie attribute that does the work:
  - **The 2026 reality.** `SameSite=Lax` (the modern default, set explicitly by the senior) is the primary defense. Cross-site `POST`s from `evil.com` to the SaaS no longer carry the user's cookie. The CSRF token pattern (a hidden form field the server validates) is no longer load-bearing for typical SaaS flows; the browser does the work.
  - **The narrow case where CSRF tokens still earn their weight.** Cookies with `SameSite=None` (cross-site contexts) lose the `SameSite` defense; the application must add a CSRF token for any state-changing endpoint that accepts that cookie. The course's stack is same-origin and doesn't hit this; Better Auth defaults to the senior reach. The lesson names the boundary.
  - **`Sec-Fetch-Site`** — Chapter 3.3 named this header. A Route Handler can read it as a defense-in-depth on top of `SameSite`: refuse any state-changing request whose `Sec-Fetch-Site` isn't `same-origin`. The browser sets it unmodifiably; non-browser clients omit it. Useful for the rare paranoid endpoint.
- **The 2026 third-party cookie reality, in one paragraph for the boundary.** Safari has blocked third-party cookies since 2020 (ITP). Firefox has partitioned them since 2021 (Total Cookie Protection). Chrome ran the deprecation through 2024–2025 — the original plan was full removal, but in late 2024 Google pivoted to "user choice" through a Privacy Sandbox-driven prompt, retaining the option for users to allow third-party cookies. The 2026 reality for SaaS engineers: any cookie that's expected to ride a cross-site request needs `SameSite=None; Secure; Partitioned` (and even then, may be blocked by Safari or by user setting). The senior reach: design for first-party cookies on a single SaaS origin; treat third-party cookies as a power-tool with a high trigger.
- **The watch-outs a senior names**:
  - **A cookie set without explicit attributes inherits browser defaults that vary.** Chrome and Edge default missing `SameSite` to `Lax`-with-the-2-minute-carve-out. Firefox and Safari do not. The senior reflex: never omit `SameSite`; set it explicitly so the cookie's behavior is identical across browsers.
  - **A `Set-Cookie` the browser silently rejected.** The browser drops a `Set-Cookie` without warning if the attributes conflict (`Secure` on HTTP, `__Host-` with `Domain`, `SameSite=None` without `Secure`, an oversized cookie). The cookie just isn't stored; subsequent requests have no `Cookie` header. The senior reflex when "the cookie isn't sticking": open DevTools → Application → Cookies and look at the cookie list; if the cookie isn't there, the browser rejected the `Set-Cookie`. The Console may log a warning depending on browser and version.
  - **The cookie's `Domain` parses against the public-suffix list.** A `Domain=co.uk` attribute is rejected (the registrable suffix can't be set as the scope). Named for recognition; the application doesn't usually hit this because it omits `Domain` in the first place.
  - **Multiple `Set-Cookie` headers in one response are valid and common.** A response can set five cookies in five separate `Set-Cookie` headers (HTTP doesn't permit folding `Set-Cookie` into one comma-separated line, unlike most other multi-valued headers). The framework's `Headers` interface needs `getSetCookie()` (Chapter 3.2.3 named this) to read them as an array.
  - **The cookie size cap.** Browsers cap each cookie at ~4096 bytes and each domain at ~150-180 cookies. The senior watch-out: put bulky state in a server-side session keyed by the cookie, not in the cookie itself.
  - **Cookies and `fetch` credentials.** A cross-origin `fetch` without `credentials: 'include'` doesn't send cookies regardless of `SameSite` (Chapter 3.3.3 named this). The senior reach: knowing that the cookie attribute and the fetch option both have to align — `SameSite=None` and `credentials: 'include'` together are what makes a cross-origin authenticated fetch work.
  - **Logging a `Cookie` header is a security incident.** The session token in production logs is a credential anyone with log access can replay. The structured-log shape (Chapter 20.1) redacts the `Cookie` and `Authorization` headers; the senior reach is to verify the redaction is in place.

What this lesson does not cover:

- The Better Auth session writer, the session table schema, and the request-time auth reads (Chapter 9.2, Chapter 9.4).
- CSRF tokens, the full security baseline, the cookie consent gate as a single source of truth for analytics opt-in (Chapter 17.2).
- The Fetch API's `credentials` option at depth, the cross-origin authenticated request pattern (Chapter 3.6.1, Chapter 3.3.3).
- Server-side session storage strategies (server-stored sessions vs. JWTs, multi-device sessions, revocation) (Chapter 9.2.3).
- The Next.js middleware surface — `request.cookies`, `response.cookies` — at depth (Chapter 5.5).
- `localStorage`, `sessionStorage`, IndexedDB, the Cache API (Chapter 3.7.5; out of scope here).
- OAuth state cookies, magic-link tokens, the auth-flow cookies in detail (Unit 9).
- Cookie-based feature flags, A/B test bucketing (out of scope; PostHog's own primitives in Chapter 20.2).
- The full RFC 6265bis edge cases (the `Priority` attribute, the path-matching algorithm's edge cases) — out of scope.
- The Privacy Sandbox APIs (Topics, FedCM, Storage Access) — out of scope for the course.

Pedagogical approach:

Concept-and-Reference archetype. The lesson teaches a mental model (the cookie as a header pair the browser stores per scope) and a reference surface (the attribute palette). The deliverable is recognition — the student can read a `Set-Cookie` line and name what every attribute does, and write the senior default by reflex.

Open with the senior question — "you're wiring auth; you need a cookie that survives the round-trip without leaking the session to XSS or riding a CSRF post; what's on the `Set-Cookie` line?" — and a small `MultipleChoice` exercise pitting four `Set-Cookie` lines (the naive `session=abc123` with nothing else; the half-done `session=abc123; Secure`; the close-but-missing-prefix `session=abc123; HttpOnly; Secure; SameSite=Lax; Path=/`; the senior default `__Host-session=abc123; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000`). The discrimination — the prefix earns its weight by structurally enforcing the three constraints — frames the lesson.

A `Figure` with a hand-authored SVG renders the `Set-Cookie` anatomy: the example line at the top with each attribute boxed and labeled, with a side annotation for what each attribute prevents (`HttpOnly` → XSS reads; `Secure` → network sniffing; `SameSite=Lax` → CSRF posts; `Path=/` → no surprise scoping; `Max-Age` → finite lifetime; `__Host-` → prefix-enforced strict scope). The student sees the senior default as a single image.

A `TabbedContent` block organizes the three `SameSite` values as a decision card. Tab 1 is `Lax` (the senior default; cross-site GET navigation OK, cross-site POST blocked). Tab 2 is `Strict` (cross-site nothing; trigger is the rare cookie whose presence on a top-level link is itself a concern). Tab 3 is `None` (cross-site everything; mandatory `Secure`; almost-always-pair-with `Partitioned`; trigger is genuine cross-site contexts). Each tab has the canonical SaaS scenario and a one-line watch-out.

A `Buckets` exercise sorts ten cookies into "set with `HttpOnly`" or "set without `HttpOnly`" — the session token (yes), the CSRF token (yes, it's only validated server-side), the cookie consent decision (no, JavaScript reads it for gating), a theme preference (no, but better served by `localStorage`), the OAuth state value during a sign-in flow (yes), an A/B test bucket (no, JavaScript needs to read it), a refresh token (yes), a tenant ID for routing in `proxy.ts` (yes), a "you've seen the welcome banner" flag (no), a session activity timestamp the client reads (no). The discrimination is concrete.

A second `Buckets` exercise sorts eight cookies into `Lax`, `Strict`, or `None`+`Partitioned` — the main session cookie (Lax), a password-reset confirmation flag (Strict — refuse on any cross-site arrival), the cookie consent (Lax), a third-party embedded widget's session (None+Partitioned), a CSRF token (Lax or Strict depending on the form's entry pattern), an OAuth state value during the flow (Lax), a Stripe checkout return-marker (Lax), an embedded chat widget's per-site state (None+Partitioned). The decision practice locks in.

A `Matching` exercise pairs five attributes with their failure mode if omitted — `HttpOnly` (XSS reads the session and replays it), `Secure` (network sniffer reads the session on HTTP), `SameSite` (cross-site CSRF POST rides the cookie), `Path` is `/` (subtle bug where a subpath cookie surprises the developer), `__Host-` prefix (subdomain takeover writes the session cookie). The vocabulary is locked in.

An `AnnotatedCode` block walks a tiny Next.js Route Handler — read a session cookie with `await cookies()`, then a Server Action that sets it with the full senior-default attribute set, then a Route Handler that deletes it. The annotations highlight the `httpOnly`/`secure`/`sameSite`/`path`/`maxAge` options on `.set()` and name the matching attributes the framework will emit on the `Set-Cookie` line. The student sees the bridge from the protocol-level attributes to the framework-level helper. The full auth wiring is in Unit 9 — this is just the protocol surface.

A `CodeReview` exercise on a 30-line snippet that sets four cookies with various issues: a session cookie missing `HttpOnly` (XSS exposure), an auth cookie with `Domain=example.com` set unnecessarily (subdomain-takeover exposure), a CSRF token cookie with `SameSite=None` and no `Secure` (the browser rejects it silently — bug class), a session cookie named `session` (no prefix; should be `__Host-session` to enforce the scope guarantees), a delete that emits `Set-Cookie: session=; Max-Age=0` without matching `Path` and `Domain` from the original (delete doesn't reach). The student leaves a comment per issue with the senior fix.

A `PredictOutput` exercise on three `Set-Cookie` lines: `__Host-session=abc; SameSite=Lax; Path=/` (browser rejects — missing `Secure`), `Set-Cookie: id=123; SameSite=None` (browser rejects — `None` requires `Secure`), `Set-Cookie: x=1; Secure` on `http://localhost` (browser rejects — `Secure` not allowed over HTTP, though some browsers permit on localhost for dev). The recognition of silent-rejection failure modes is concrete.

A small interactive widget (a toggle-driven simulator) shows three cookies set with different attributes and three navigation scenarios (same-origin GET, cross-site GET via link click, cross-site POST from a form on evil.com). For each combination, the widget shows whether the cookie rides. The student toggles `SameSite` between `Lax`, `Strict`, `None` and watches the matrix update. The mental model is tangible.

Close with a `TrueFalse` round of five statements: "`HttpOnly` prevents XSS attacks" (false — it limits XSS blast radius but doesn't prevent the injection; CSP is the structural prevention), "A cookie with `SameSite=None` is sent on all cross-site requests" (false in 2026 — Safari blocks, Firefox partitions, Chrome restricts unless `Partitioned` is set), "Setting `Domain=example.com` on a cookie set from `app.example.com` makes it sticker" (false — it widens the scope and is the bug class `__Host-` prevents), "`Secure` cookies can be set over HTTP on `localhost` in modern browsers" (true — browsers grant a localhost exception, but the senior reach is `mkcert` for parity with production), "The default lifetime of a cookie without `Max-Age` or `Expires` is the browser session" (true, with the caveat that "browser session" is increasingly preserved by browser-restart features). The vocabulary is locked in.

Estimated student time: 50 to 60 minutes. Load-bearing for Unit 9, Chapter 17.2, and any later cross-origin or third-party-cookie surface.

## Lesson 3.4.2 — Quiz

Top ten topics to quiz:

1. The cookie model — `Set-Cookie` response header sets, `Cookie` request header replays; the browser stores per `(host, path, name)`; the framework's `cookies()` helper wraps the headers.
2. The senior default attribute set — `HttpOnly; Secure; SameSite=Lax; Path=/` — plus `Max-Age=N` for any cookie that should expire.
3. `HttpOnly` refuses JavaScript access via `document.cookie`; mitigates XSS-becomes-session-theft; non-negotiable on auth cookies.
4. `Secure` refuses HTTP attachment; mandatory in production; the dev-time bite is `Secure` on `http://localhost` (resolved by `mkcert`).
5. `SameSite=Lax` is the modern default and primary CSRF defense — refuses cross-site POSTs, allows top-level GET navigation. `Strict` refuses cross-site entirely; `None` (with mandatory `Secure`, almost-always-`Partitioned`) is the cross-site-context opt-out.
6. The `__Host-` prefix structurally enforces `Secure`, no `Domain`, `Path=/` — the senior reach for the session cookie. `__Secure-` is the weaker prefix when `Domain` is needed.
7. Cookie scope is `(host, path)` — omit `Domain` by default (cookie stays host-only); a `Domain` cookie is settable by any subdomain (subdomain-takeover vector); `Path` is attachment scope, not visibility (`HttpOnly` controls visibility within the page).
8. Cookie lifetime — `Max-Age=N` (seconds) is the senior reach; no `Max-Age` or `Expires` makes it a session cookie (deleted on browser session end, which "continue where you left off" features increasingly preserve). Deletion is `Max-Age=0` with matching scope attributes.
9. The 2026 third-party cookie reality — Safari blocks (ITP), Firefox partitions (TCP), Chrome restricts. Cookies that ride cross-site contexts need `SameSite=None; Secure; Partitioned` (CHIPS) and may still be blocked. The senior reach: design for first-party single-origin SaaS.
10. The silent-rejection failure modes — `Secure` on HTTP, `__Host-` with `Domain`, `SameSite=None` without `Secure`, oversized cookie. The browser drops the `Set-Cookie` without an error; the cookie just isn't stored. Recognition reflex: DevTools → Application → Cookies.
