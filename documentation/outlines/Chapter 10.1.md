# Chapter 10.1 — Organizations as the tenancy model

## Chapter framing

Unit 10 turns a single-user app into a multi-tenant SaaS. Chapter 10.1 lands the tenancy model itself — the data shape that pins every row to an organization, the active-org slot the session carries forward, and the helper that makes "I forgot the org filter" structurally hard to write. The student arrives with Better Auth's `user` / `session` / `account` / `verification` tables from Unit 9, the Drizzle vocabulary from Unit 6, and the Server Action shape from Unit 7. They leave with `organizations` and `org_members` tables installed via Better Auth's organization plugin, an `activeOrganizationId` slot on the session set on sign-in and switched from a UI control, a `tenantDb(orgId)` helper composed on top of the relational query API, and a senior-grade read on the one threshold where Postgres Row-Level Security earns its operational cost — the high-stakes data class where one missed application-layer scope would be unacceptable. Chapter 10.2 owns RBAC, the `authedAction` wrapper, and the audit-log table; this chapter installs the org skeleton those layers ride on.

Threads that run through every lesson. The default tenancy enforcement in this stack is *application-layer scoping* — every read on a tenant-owned table starts with `where eq(organizationId, ...)`, and the `tenantDb` helper makes that filter the only path that compiles. RLS is the conditional power tool, not the default — named precisely, sized to the data class where the threshold is met, and otherwise off because the operational cost (connection-level session variables, pooled-connection discipline, harder local debugging) outweighs the benefit for most rows. The active organization is *session state* held server-side on the `session` row, not URL state and not client state — switching orgs is a Server Action that writes the new value and forces a redirect; the URL never carries `orgId` for tenancy decisions. Better Auth's organization plugin owns the `organization` and `member` tables and their canonical names — the student wires the plugin, the schema generates, the names are honored downstream. Every "watch out" frames the bug class that ships if the discipline isn't structural: missing-scope reads, stale active-org after a member removal, RLS without session variables, RLS with pooled connections that share state across requests. Four teaching lessons plus a chapter quiz.

## Lesson 10.1.1 — The organization data model and the active-org slot

Topics to cover:

- **The senior question.** A SaaS user signs up. They might run one company through the app, or three. Their teammates are members of the same company. Where does "the company" live in the schema, how does the session know which one the user is currently operating inside, and what does switching companies actually do — to the session, to the next request, to the in-flight cache? Stand up the org data model with Better Auth's organization plugin, add `activeOrganizationId` to the `session` table, wire the create-org / switch-org / list-my-orgs surface.
- **The shape — three tables.** `organization` (id, name, slug, logo, createdAt, metadata), `member` (id, organizationId, userId, role, createdAt), and `invitation` (covered in Chapter 10.3 — named here, schema lands now because the plugin generates all three together). Better Auth's organization plugin owns the table names and column shape; the student adds the plugin, runs `auth:generate`, the migration drops into `src/db/schema/auth.ts` next to the four Unit 9 tables. The senior reflex: don't rename the plugin's tables to match a house style — every plugin call in Better Auth assumes its own names.
- **`organization` vs. `team`.** The plugin supports a two-level hierarchy: an organization can contain teams, and a member belongs to an org with a role and optionally to one or more teams within it. For the year-1 SaaS the senior call is *organizations only* — the team layer is the conditional reach when the product surfaces "departments inside a company." Disabled by default; named once. The student configures `organization({ teams: { enabled: false } })`.
- **The `slug` decision.** Every org gets a slug because URLs and emails want one (`/o/acme/dashboard`, `noreply+acme@`). Two slug strategies: app-generated from the name with a uniqueness suffix (`acme`, `acme-2`) or user-chosen at create time with a uniqueness check. The senior default is user-chosen — companies care about their URL identity, and the uniqueness error message is part of the create flow's UX. Slug allowlist: `^[a-z0-9-]{3,32}$`, reserved slugs (`admin`, `api`, `app`, `auth`, `billing`) blocked at the action boundary.
- **The active-org slot — adding columns to the `session` table.** The plugin extends the `session` table with `activeOrganizationId` (nullable string FK to `organization.id`). One row per session — switching orgs updates the existing `session` row, doesn't mint a new one. Why on the session and not on the user: the same user can have two browser sessions open in two orgs simultaneously (the "I review my own invoices in Personal while triaging Acme's in another tab" reflex). The 9.4.3 multi-device session model composes — each device's session carries its own active-org.
- **Setting the initial active-org — the `databaseHooks.session.create` callback.** When a session row is created (sign-in, sign-up, any flow that mints a session), set `activeOrganizationId` to the user's most-recent active org if one exists, else their first membership, else `null`. The plugin exposes the hook; the student writes a four-line function that runs the lookup inside the `before` phase of session creation. Senior reflex: don't put this in the sign-in action — every entry point that mints a session has to set it, and the hook is the one place that covers every entry point structurally.
- **The `null` active-org branch — the no-orgs-yet user.** A brand-new sign-up has no memberships. Their session's `activeOrganizationId` is `null`. The protected layout reads `activeOrganizationId`; if `null`, it redirects to `/onboarding/create-org` (a page the student stubs). The senior reflex: this is the only place in the app where "signed in but no active org" is allowed; every other read on a tenant table assumes a non-null active-org and the helper enforces it.
- **Creating an organization — the call.** `authClient.organization.create({ name, slug, logo? })`. Library validates the slug, inserts `organization` + `member` (with role `owner` for the creator), updates the current session's `activeOrganizationId` to the new org. One round trip, one Server Action, one redirect to `/dashboard`. The form mirrors every Unit 7 form shape — Zod on input, `Result` discriminants on output (`'ok'`, `'slug-taken'`, `'invalid-slug'`).
- **Switching organizations — the call and the cache reflex.** `authClient.organization.setActive({ organizationId })`. Library validates the user is a member, updates `session.activeOrganizationId`, returns the new active org. The Server Action wrapping it must `revalidatePath('/', 'layout')` (or the targeted scope) after the call — every cached read on the previous org is now stale; the senior reflex is to wipe the layout cache on switch, not to hope the user reloads. `useRouter().refresh()` on the client side completes the picture.
- **The org switcher UI.** A `<Select>` or `<Command>` in the layout header reading the membership list from a server-passed prop, calling `setActiveOrgAction(orgId)` on change. The membership list comes from `authClient.organization.list()` (or the server-side `auth.api.listOrganizations({ headers })` for SSR). Senior reflex: render the switcher in the protected layout, not in every page — one mount point, one source of truth.
- **The `activeOrganizationId` on cookie-cache reads.** Better Auth's cookie cache (9.2.3) caches the session payload including `activeOrganizationId`. After `setActive`, the cookie is updated immediately by the library — the next request reads the new value. Senior watch-out: the `maxAge` window is irrelevant for org switches because `setActive` invalidates and rewrites the cookie; don't reach for a longer/shorter cache on this account.
- **Reading the active org at request time.** Server side: `const session = await auth.api.getSession({ headers }); const orgId = session.activeOrganizationId`. Inside `requireUser()` (Unit 9.2.4), extend to `requireOrgUser()` that returns `{ user, orgId, role }` and redirects to `/onboarding/create-org` if `orgId` is null. The next lesson's `tenantDb` reads `orgId` from this helper — never from a route param, never from a client-passed value.
- **The org switch and in-flight Server Actions.** Edge case: user clicks switch-org while a Server Action from the old org is in flight (a slow save). The action read the old `orgId` at its start; it completes against the old org. Senior reflex: this is correct — actions resolve in the context they started. The UI's `router.refresh()` after switch ensures the user doesn't see ambiguous state, but the in-flight write isn't a bug.
- **Watch-outs.** Storing the active org in `localStorage` or a separate cookie — drifts from the server's session truth, breaks across devices; reading `activeOrganizationId` from the URL or a route param and trusting it — bypasses the membership check; renaming the plugin's `organization` / `member` tables in a custom schema file — every plugin call breaks silently; setting the active org in the sign-in action only — sign-up and OAuth-callback paths leave it null; forgetting to `revalidatePath` after `setActive` — user sees the old org's data for the duration of the route cache; allowing a user to set-active an org they aren't a member of (would be a library bug, but never trust — the membership check belongs in the library, and the senior reads the source on first use); blocking the `/onboarding/create-org` route in the proxy's authed-pages matcher — users with a session but no org get redirect-looped.

What this lesson does not cover:

- Roles inside the org — `owner` / `admin` / `member` semantics (Chapter 10.2.1).
- The `authedAction(role, schema, fn)` wrapper (Chapter 10.2.2).
- Invitations and the accept handshake (Chapter 10.3).
- Member removal, ownership transfer, leave-org (Chapter 10.2.4).
- Teams inside an organization (named once, not built — conditional reach).
- The audit log row written on create/switch (Chapter 10.2.5).

Estimated student time: 40 to 50 minutes. Mechanics + setup hybrid; a small ER diagram of `user` / `session` / `organization` / `member` with `activeOrganizationId` arrowed earns its weight, and the sequence diagram for "click switch → server action → setActive → revalidate → next render reads new orgId" is the second visual.

## Lesson 10.1.2 — The tenant-aware query helper (SaaS pattern #1)

Topics to cover:

- **The senior question.** Every read on `invoices`, `customers`, `documents`, any tenant-owned table, must filter by `organizationId`. The bug class is one missed `where`: one developer, one fast PR, one query that returns rows from every customer's data. Application-layer enforcement is the year-1 default in this stack; how do we make missing the filter *not compile* rather than *not pass review*? Install the `tenantDb(orgId)` helper that wraps Drizzle's relational query API so every read and every write is org-scoped by construction.
- **The threat — the missing-`where` bug.** A reviewer sees `db.select().from(invoices).where(eq(invoices.id, id))` in a PR. Looks fine. The query returns the row whether it belongs to this user's org or not; the route handler renders it; the customer dashboard shows another company's invoice. The bug is invisible in code review because the absence of the org filter looks like every other "fetch by id" query. The defense isn't more review — it's a helper where forgetting the filter doesn't typecheck.
- **The architectural carve-out — Principle #5 ("consume libraries directly") gets an exception.** Drizzle is the library; the carve-out is the small wrapper that pins `organizationId` onto every relational read. The senior reflex: name the exception explicitly so the team knows when "direct Drizzle" is correct (one-off admin scripts, migration scripts, system-level reads) and when "via `tenantDb`" is correct (any request-handled read on a tenant table). This is the same shape as the `authedAction` carve-out (Chapter 10.2.2) for authz at the action boundary.
- **The helper shape — wrap, don't replace.** `tenantDb(orgId)` returns an object that exposes the subset of Drizzle calls the app actually uses, each pre-bound to inject the org filter. The relational query API is the primary surface: `tenantDb(orgId).query.invoices.findMany({ where: ... })` composes the caller's `where` with `and(eq(invoices.organizationId, orgId), ...)`. The select-builder surface (`tenantDb(orgId).select().from(invoices)`) is wrapped the same way: an intercepted `where` that always `and`s in the org predicate. Writes (`insert`, `update`, `delete`) are wrapped to set or check `organizationId`. The student writes the helper as a small typed factory in `src/lib/db/tenant.ts`.
- **What the helper has to know per table — the registry.** Some tables are tenant-owned (`invoices`, `customers`, `documents`, `org_members`); others are global (`user`, `verification`). The helper must know which is which to refuse the wrong call shape — `tenantDb(orgId).query.users.findMany(...)` should be a type error because `user` has no `organizationId`. The senior reach: a `TENANT_TABLES` type-level set keyed off the tables that carry an `organizationId` column. The build-time check is the discipline; the runtime check is the backstop ("if you reach for a table that isn't in the registry through this helper, throw").
- **The relational query API and the nested `where`.** Drizzle's `query.invoices.findMany({ with: { lines: true, customer: true }, where: ... })` joins through `customer` to the parent record — and `customer` is also tenant-owned. The helper composes the top-level `where` for the root table; the *children* are already scoped because Drizzle's `with` follows the FK and the FK target is itself an org-scoped row. The senior reflex: a child whose FK chains back to a tenant table inherits the scope through the join; a child that *doesn't* (a public lookup table, an enum-shaped reference) reads global. The helper enforces the root; the schema's FK shape carries the rest.
- **Writes — the insert default and the update/delete predicate.** `tenantDb(orgId).insert(invoices).values({ ... })` injects `organizationId: orgId` into every row; if the caller passed an `organizationId` already, the helper throws (mismatch is a bug). `update` and `delete` always `and`-in the org predicate — a successful update against another org's row should affect zero rows, not the wrong row. The senior reflex: the helper enforces zero cross-org writes by construction; the database backstops with FK constraints (`customer_id` in `invoices` is FK to `customers.id` with `customer.organization_id` matched at app layer; RLS is the database backstop, see 10.1.3).
- **The `orgId` source — never a parameter, never a query string.** The helper receives `orgId` from `requireOrgUser()` (10.1.1) — the helper trusts that the caller already validated the session and read the active org. Senior reflex: never accept `orgId` from a route param, a form field, or a client header; the only acceptable source is the server-validated session. The `authedAction` wrapper in Chapter 10.2.2 enforces this end-to-end at the Server Action boundary; the helper here is the data-layer arm of the same discipline.
- **The composition with `requireOrgUser()`.** Inside a Server Action: `const { user, orgId } = await requireOrgUser(); const db = tenantDb(orgId); const rows = await db.query.invoices.findMany({ ... })`. Three lines, no manual filter, no place to forget the scope. The senior reflex: codify this as the canonical action shape and review every PR against it — if a tenant-table read goes through `db.` directly (the unwrapped client), the review catches it because the call shape is wrong, not because the reviewer checked every `where`.
- **The `where` composition — `and()` semantics.** The helper combines its predicate with the caller's via `and()`. If the caller's `where` is `undefined`, the helper supplies just its own predicate; if the caller's `where` is `or(eq(status, 'open'), eq(status, 'overdue'))`, the helper wraps to `and(eq(organizationId, orgId), or(...))`. The senior trap: a caller writing `or(eq(organizationId, otherOrgId), eq(status, 'open'))` to deliberately escape the scope — the helper's `and` still pins the org, so the caller's `or` becomes a logical contradiction on the other-org branch; effective scope still holds. Mention this as the proof the helper is robust against accidental and curious-developer escape.
- **The "I need to read across orgs" reach.** Admin tools, system jobs, batch queries that legitimately need to span tenants. The senior call: those reach for the unwrapped `db` client (the same client `tenantDb` wraps), and the file lives in `src/lib/admin/` or `scripts/` so the unwrapped use is visible. Don't add a `tenantDb({ allOrgs: true })` mode — the escape valve normalizes the bypass; keeping it as "different client, different file" makes the cross-org call a deliberate decision.
- **Naming — `tenantDb` versus alternatives.** The pattern shows up in the field as `forTenant(orgId)`, `dbFor(orgId)`, `tenantScoped(orgId)`, `orgDb(orgId)`. The student picks the name the codebase uses; the lesson uses `tenantDb` because the chapter framing names it. The senior reflex: pick one and grep — if a second name shows up, the team is drifting.
- **What this is not — ORM forks, query interceptors, AST rewriters.** Some ecosystems reach for "patch the ORM" (Prisma extensions, Drizzle proxies that auto-inject filters via runtime reflection). The 2026 senior call in this stack is the explicit wrapper — the type signature documents the discipline, the implementation is 40 lines, the failure mode is loud (a misshape throws), and the unwrapped `db` is still available for the legitimate cross-org reach. Reach for a smarter framework only when the table count crosses the threshold the wrapper can't keep up with (and that threshold is rarely met in year-1 SaaS).
- **Watch-outs.** Adding `where` after `tenantDb` returned a builder and then *replacing* the helper's predicate by accident (Drizzle's builder API lets you reassign `where`; the helper has to detect/document this — the canonical shape is the relational query API which doesn't expose chained `where` reassignment); type-narrowing the helper down to a single table and losing the cross-table check; returning the raw Drizzle client from inside `tenantDb` for "convenience"; importing the unwrapped `db` inside Server Actions for "one quick read"; trusting `orgId` from a function parameter at the call site instead of from `requireOrgUser` — the helper's correctness is conditional on the caller; documenting `tenantDb` as "if you forget, the helper saves you" — it doesn't, it makes forgetting *not compile*; mixing `tenantDb` and unwrapped `db` reads in the same handler (review-loud, easy to miss).

What this lesson does not cover:

- The `authedAction(role, schema, fn)` wrapper (Chapter 10.2.2) — orthogonal layer at the action boundary.
- The role check inside the helper — the helper scopes data; the role check is upstream at the action.
- Cross-org admin reads — named once, the pattern is "different client, different file."
- RLS as the database backstop — Chapter 10.1.3 and 10.1.4.
- The audit-log write on tenant-table mutations — Chapter 10.2.5.

Estimated student time: 45 to 55 minutes. Pattern lesson; the wrong-then-right archetype carries the weight — show the missing-`where` bug, then the helper signature, then the helper applied. A type-shape diagram for "table registry → typed helper → org-scoped builder" earns its weight.

## Lesson 10.1.3 — When Row-Level Security earns its weight

Topics to cover:

- **The senior question.** Application-layer scoping (10.1.2) is the year-1 default. RLS is the database backstop. What's the threshold where RLS earns its operational cost, what data classes actually meet it for a typical SaaS, and what's the senior call when the team is staring at a long Stack Overflow thread on Postgres policies asking "do we need this?" Treat RLS as a power tool — name the trigger that crosses the default, then make a deliberate decision.
- **What RLS actually buys.** A policy on a table refuses rows that don't match the policy condition — `USING (organization_id = current_setting('app.org_id')::uuid)` — for every query, every connection, every code path. The application could omit the org filter entirely and the database still wouldn't return cross-org rows. The defense is total within the database boundary; it does not depend on application-layer discipline.
- **What RLS costs.** Operational overhead in three places. (1) Every connection needs a session variable set before the first query (`SET app.org_id = '...'`); the connection pool must support session-level state, and connection reuse must reset it. (2) Local debugging is harder — `psql` queries return no rows unless the session variable is set, and the "why are there no rows" loop is a recurring junior trap. (3) Policy authoring is its own discipline — wrong policies are silent failures (the policy excludes legitimate rows; the app shows empty lists; nobody notices for a week). The senior reflex: RLS is not free, and "we already have application scoping" plus a code review for the helper-usage discipline is *most of the protection at a fraction of the cost.*
- **The threshold — when RLS earns the cost.** Three triggers, any one is enough. (1) The data class is *highest-stakes* — one missed scope produces a regulatory incident (HIPAA-covered PHI, financial PII, audit logs that legal will subpoena, security-sensitive credentials). (2) The data is read by *many code paths* — admin tools, batch jobs, BI dashboards, support consoles — and the application-layer helper can't span every entry point. (3) The team is *not the only writer* — a partner integration, an external job runner, a third-party reporting tool that holds DB credentials and runs SQL the app team doesn't review. Below the threshold, application-layer scoping is sufficient.
- **What this stack defaults to and why.** For most SaaS rows — invoices, documents, comments, settings — application-layer scoping via `tenantDb` is the default. The bug class is "missing org filter"; the structural defense is the helper that makes the filter the only call shape that compiles; the failure mode is "engineer reaches for unwrapped `db` for convenience" which is review-visible because the import line is different. RLS doesn't add enough defense over this discipline to justify the cost on every table.
- **The data classes where RLS *does* earn its place in our app.** Identify the candidate set: `audit_logs` (the table that must never leak; the regulatory and forensic value is high; many code paths write to it; the cost of one missed scope is unrecoverable trust damage) and any PII or compliance-sensitive table the product adds later (e.g., `customer_documents`, `payment_methods` if you store any). The senior call for the year-1 SaaS built in this course: RLS *on `audit_logs`*, application-layer scoping everywhere else. Chapter 10.2.5 stands up the table; this chapter wires the policy in 10.1.4.
- **RLS as defense-in-depth, not the primary defense.** The senior framing: even on RLS-protected tables, the application still uses `tenantDb`. Two layers — application scope and database scope — fail independently. The bug class "an engineer wrote SQL by hand in a one-off script and forgot the org filter" is exactly what RLS catches; the application-layer helper is what catches the 99% case in the normal request path.
- **The pooled-connection footgun.** RLS policies read `current_setting('app.org_id')`. The variable is set per-connection. With connection pooling (PgBouncer in transaction-pooling mode, Drizzle's default Postgres pool, Neon's Serverless Driver pool), connections are reused across requests — and session variables persist across reuses. Two failure modes: (a) a request that doesn't set the variable inherits the previous request's value (cross-tenant leak in the worst direction); (b) a request that sets the variable doesn't reset it, and a later request on the same connection reads cross-tenant rows. The mitigation is per-request `SET LOCAL` inside an explicit transaction — `SET LOCAL` is bound to the transaction, automatically cleared on commit/rollback. The full wiring is 10.1.4; the threat is here.
- **The Neon RLS shape — `crudPolicy` and the inferred roles.** Neon's RLS integration ships a higher-level wrapper (`crudPolicy` in Drizzle) that authors select/insert/update/delete policies in one function call against Neon's predefined roles (`authenticated`, `anonymous`). For a SaaS using Neon, this is the cleanest authoring surface — policy lives in the schema file next to the table definition, not in a separate `policies.sql`. The senior reach: if the stack is on Neon, use `crudPolicy`; if on vanilla Postgres (Supabase, RDS, Docker), use Drizzle's `pgPolicy` directly. The course's default stack is Neon for production, Docker for local; both code paths work because policies are SQL that the Drizzle Kit migrations carry forward.
- **What RLS does *not* defend against.** A compromised application server with the DB credentials and the ability to `SET app.org_id` to any value reads everything. RLS is a defense against *application bugs*, not against *application compromise*. The defense against compromise is principle-of-least-privilege at the connection level (different DB roles for different surfaces, the admin role only used for migrations, the request-handler role unable to bypass RLS) — covered as a senior reach, not built in this chapter.
- **The decision tree the senior runs at table-design time.** Step 1: does this table carry data whose cross-tenant leak is unrecoverable (legal, regulatory, audit, security)? If yes → RLS. Step 2: does the table get written by code paths outside the request handler (jobs, scripts, external integrations)? If yes → RLS or extremely careful review. Step 3: otherwise → application-layer scope via `tenantDb`, code review for the helper-usage discipline, ship. Repeat per table. Senior reflex: this is a *table-level* decision, not a project-level decision — the same project can have RLS on `audit_logs` and not on `invoices` and be correctly designed.
- **The "RLS everywhere" anti-pattern.** Some shops reach for "RLS on every table" as a blanket policy. Failure modes: every developer hits the missing-session-variable trap weekly; local debugging becomes a stop-and-restart loop; policy authoring drifts as schema evolves; admin scripts grow special-case "set the variable from the script" boilerplate; the helper-discipline atrophies because "the database catches it" — until a misconfigured pool doesn't and the bug ships anyway. Senior reflex: RLS is conditional; the unconditional layer is the helper.
- **Watch-outs.** Reaching for RLS to compensate for not having `tenantDb` — wrong layer, wrong protection; declaring RLS "on" but never setting the session variable on every request — every query returns empty, the app looks broken instead of secure; setting `app.org_id` outside a transaction with a pooled connection — variable persists, cross-tenant leak; thinking "RLS protects against SQL injection" — orthogonal, parameterization protects against injection; deciding RLS-or-helper at the project level rather than the table level — both have their place; not testing the policies (a passing-empty policy is a passing-empty test); writing the policy SQL by hand instead of through Drizzle's typed authoring surface — drifts from the schema, breaks on rename; relying on `current_user` or the database role for the tenant identity when the app uses one role for all requests — the policy reads from `app.org_id`, not from `current_user`; treating RLS as "the security layer" instead of as "a security layer" — defense-in-depth is the senior frame.

What this lesson does not cover:

- The Postgres syntax for `CREATE POLICY` (lesson 10.1.4 owns).
- The Drizzle `pgPolicy` / `crudPolicy` API at the call site (lesson 10.1.4 owns).
- The session-variable wiring through Drizzle's connection (lesson 10.1.4 owns).
- DB role separation for least-privilege (named once; full coverage in Chapter 17.2).
- RLS against multi-region replicas (out of scope).
- RLS performance impact at very high row counts (named once; production tuning is Chapter 20.3 territory).

Estimated student time: 35 to 45 minutes. Decision lesson; the threshold-up-front shape is the structure (three triggers, defaulted otherwise). A small comparison table — "application-layer scope vs. RLS: bug class caught, cost, debugging story, when to reach" — earns its weight, and a decision-tree diagram for the per-table call closes the lesson.

## Lesson 10.1.4 — Postgres RLS through Drizzle: policies, session variables, and applying it to audit_logs

Topics to cover:

- **The senior question.** The decision in 10.1.3 landed: `audit_logs` gets RLS. What's the actual syntax for a policy, how does the policy read the tenant identity at query time, how does the application set that identity per-request without leaking it across pooled connections, and what's the end-to-end test that proves "another org's row doesn't come back even if I forget the `where`"? Wire the primitives and apply them to the one table this course's stack reaches for them on.
- **The policy in one paragraph.** `CREATE POLICY audit_logs_org_isolation ON audit_logs FOR ALL TO authenticated USING (organization_id = current_setting('app.org_id', true)::uuid) WITH CHECK (organization_id = current_setting('app.org_id', true)::uuid)`. Two predicates — `USING` filters the rows the query can read; `WITH CHECK` constrains the rows the query can write. `FOR ALL` covers select / insert / update / delete in one policy; separate policies per command earn their weight when the predicates diverge (rare for tenant isolation). The `, true` argument to `current_setting` returns `NULL` instead of erroring when the variable is unset — the senior reflex is *true* so an unconfigured request fails closed (predicate against `NULL` returns no rows) rather than erroring with a database-side stack trace.
- **`ENABLE ROW LEVEL SECURITY` and the bypass-by-default trap.** A policy doesn't take effect until the table is enabled: `ALTER TABLE audit_logs ENABLE ROW LEVEL SECURITY`. Even after enabling, table owners and superusers *bypass* RLS by default — `ALTER TABLE ... FORCE ROW LEVEL SECURITY` makes the policy apply to owners too. Senior reflex: enable AND force on every RLS-protected table; the senior anti-trap is "I wrote the policy, but my migration runs as the owner role and reads everything anyway, and my tests fail to catch the cross-tenant case because they also run as owner."
- **The Drizzle authoring surface — `pgPolicy` and the `.rls()` modifier.** Drizzle exposes policy authoring as table-modifier syntax: `pgTable('audit_logs', { ... }, (t) => [pgPolicy('org_isolation', { as: 'permissive', for: 'all', to: ['authenticated'], using: sql\`organization_id = current_setting('app.org_id', true)::uuid\`, withCheck: sql\`organization_id = current_setting('app.org_id', true)::uuid\` })]).enableRLS()`. One file, schema and policy together, migrations generated by `drizzle-kit generate` carry both. The senior reach for Neon: `crudPolicy({ role: authenticatedRole, read: sql\`...\`, modify: sql\`...\` })` collapses select/insert/update/delete into one call when the predicates are uniform.
- **The session variable — where it gets set, why `SET LOCAL`, why inside a transaction.** Every request that touches `audit_logs` must `SET LOCAL app.org_id = $1` *inside an explicit transaction* before any read or write. `SET LOCAL` scopes the variable to the current transaction — automatic cleanup on commit/rollback, no leak across the connection's next checkout from the pool. `SET` (without `LOCAL`) scopes to the session — persists across transactions on the same connection, the pooled-connection footgun from 10.1.3 made manifest.
- **The Drizzle wiring — the `withTenant(orgId, fn)` helper.** A small wrapper: open a transaction, `SET LOCAL`, run the callback, commit. `await withTenant(orgId, async (tx) => { await tx.insert(auditLogs).values({ ... }) })`. Every write to `audit_logs` (and every read, if reads happen) goes through `withTenant`. The senior reflex: pair this helper with `tenantDb` from 10.1.2 — `tenantDb` handles the application-layer filter for the normal request path; `withTenant` handles the DB-layer policy for the RLS-protected table. They compose; for `audit_logs` writes, the action does both (because audit writes happen from inside actions that already have an `orgId`).
- **The DB role — running the app as a non-owner.** RLS bypass by table owner is the default. The migration runs as the owner role (and bypasses, which is fine — migrations are admin operations). The request handler must run as a *non-owner* role with `USAGE` on the schema, table-level grants on the tables, and no `BYPASSRLS` attribute. The senior reach: two `DATABASE_URL`s in the env — `DATABASE_URL_OWNER` for migrations and admin scripts, `DATABASE_URL` (app-role) for everything else. Most Neon and Supabase setups ship this separation by default; vanilla Postgres needs the role created explicitly. Name it; the full role separation is Chapter 17.2.
- **The connection-pool reality in 2026.** Drizzle's default Postgres adapter ships a connection pool; Neon's serverless driver pools too. Both reuse connections across requests. `SET LOCAL` inside a transaction is the only safe primitive — anything that sets at session level outside a transaction leaks. Senior reflex: write the helper, never reach for raw `SET`, never use `pg-pool`'s "set on connect" hook for tenant variables (per-request value, not per-connection value).
- **Testing RLS — the "another org's row doesn't come back" proof.** The acceptance test: insert two `audit_logs` rows, one for org A, one for org B. Set `app.org_id = A`, query `SELECT * FROM audit_logs` (no `where`), assert one row returned, and that row is A's. Set to B, repeat, assert B's row only. Run the test against the app role, not the owner role — owner bypasses, the test passes for the wrong reason. The senior reflex: this is a database-layer integration test, not a unit test; it lives in the integration test suite (Chapter 19.3) and runs against a real Postgres.
- **The policy + the helper composition for `audit_logs` writes.** Inside a Server Action: `requireOrgUser()` → `orgId`. The action calls `withTenant(orgId, async (tx) => { await tx.insert(auditLogs).values({ organizationId: orgId, ... }) })`. Two enforcements: the application-layer `values({ organizationId: orgId })` makes the insert explicit, and the DB-layer policy's `WITH CHECK` refuses an insert whose `organization_id` doesn't match `app.org_id`. A bug in the application (forgetting the value) is caught by the policy; a bug in the policy (typo, missing enable, missing force) is caught by the application's explicit value. Both layers, both writes look the same, defense-in-depth holds.
- **Reads on `audit_logs` — when, by whom, the policy still applies.** Audit logs are write-heavy and read-rare. Reads happen from the admin / compliance UI (Chapter 10.2.5) and from compliance exports (Chapter 13.2 territory). Every read also goes through `withTenant` so the policy filters; the admin UI shows only the current org's audit log. Senior reach: a super-admin role that needs cross-org audit reads bypasses via a different DB connection (the owner role, or a role with `BYPASSRLS`); that surface lives outside the normal request handler and the cross-org reach is explicit.
- **What changes in `drizzle.config.ts` and the migration.** Nothing in the config — Drizzle Kit picks up `pgPolicy` and `.enableRLS()` from the schema file and emits the policy CREATE and `ALTER TABLE` statements in the generated SQL. The migration is committed; review the generated SQL to confirm `ENABLE ROW LEVEL SECURITY` and `FORCE ROW LEVEL SECURITY` (some Drizzle versions need the `FORCE` added manually — verify on first generation; if missing, add a follow-up migration with the explicit `ALTER`). The senior reflex: read the generated SQL on first RLS migration; don't trust the codegen blindly when the cost of being wrong is total cross-tenant exposure.
- **The fail-closed default — why `current_setting('app.org_id', true)::uuid` and not `::text`.** Casting to `uuid` forces a type check; an unset variable returns `NULL`, the comparison against `NULL` is `NULL`, the row is excluded. A non-UUID value (a typo, a `'all'` literal someone tried for "give me everything") fails the cast and errors out — the request fails loudly rather than returning surprising rows. Senior reflex: when the variable's domain has structure, cast to the structured type; treating it as `text` and comparing to `text` opens the door for "what if I set it to `'%'` and the policy uses `LIKE`."
- **Local development — the "why are there no rows" loop.** A new developer clones the repo, runs the seed, opens `psql`, queries `audit_logs`, gets nothing. The variable isn't set. The senior fix: a `dev:psql` script that connects with the owner role for ad-hoc exploration, or a `\set` snippet in `~/.psqlrc` that sets `app.org_id` to the seed's first org. Document the recipe in the project README. Don't grant `BYPASSRLS` to the app role to "make development easier" — defeats the entire purpose.
- **Watch-outs.** `SET` without `LOCAL` on a pooled connection — leak across requests; missing `FORCE ROW LEVEL SECURITY` — owner bypasses, the migration script and any admin code that runs as owner read everything; running tests as the owner role — every policy test is a false positive; setting `app.org_id` from a request header or query string — the entire defense bypassed by a client; running `audit_logs` writes outside `withTenant` — the policy refuses with a check-constraint-like error, the action fails, the audit row is lost (and the action might be the only signal that something went wrong); writing the policy in a hand-edited migration file and forgetting to mirror in the Drizzle schema — drifts, the next generated migration overwrites; granting the app role `BYPASSRLS` for "convenience" — defeats; relying on `current_user` for the tenant rather than `app.org_id` — the app uses one DB role for all tenants, `current_user` is constant; assuming Neon's `crudPolicy` covers both reads and writes when the predicates diverge — re-read the API, split into separate policies when they do.

What this lesson does not cover:

- The schema and write surface for `audit_logs` itself — what columns, what events trigger writes (Chapter 10.2.5 owns).

- Multi-region replication and RLS interaction (out of scope).
- Other tables getting RLS later — the same recipe applies; the senior decision per table is 10.1.3's territory.
