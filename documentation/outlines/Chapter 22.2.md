# Chapter 22.2 — Docs that live in the code

## Chapter framing

Chapter 22.1 named the four artifacts a SaaS repo maintains (README, AGENTS.md, ADRs, source-as-reference). This chapter zooms in to the documentation that lives *inside* the source files — TSDoc on the functions, types, and schemas that form the codebase's public surfaces, and inline comments on the lines that need a *why* the code can't speak. The chapter's job is to lock down two cuts the senior makes every day: which declarations earn a TSDoc block (and which don't), and which lines earn an inline comment (and which don't), so that doc volume tracks doc value and the comments in the codebase are the ones a reviewer would write themselves.

Threads through every lesson: TSDoc is reference documentation in code form — it documents the *contract* of a public surface, not the implementation behind it; comments answer the *why* the code can't (the constraint, the workaround, the non-obvious trade-off) and are part of the code, never collateral that gets stripped in a refactor; the structural enforcement that keeps every doc artifact accurate is the same — the doc ships in the PR that changed the thing it describes, or it's already wrong; volume is not the goal — a TSDoc block on every private helper or a `// increment counter` above `counter++` is anti-value, signal-per-line is the senior posture; the same `link, don't duplicate` reflex from 22.1 applies inline: a TSDoc that paraphrases the Zod schema drifts, a TSDoc that points at the schema doesn't.

## Lesson 22.2.1 — TSDoc the public surface
Learn which declarations earn a TSDoc block, the minimal 2026 tag set, the first-sentence-is-the-hover writing posture, and the link-don't-duplicate reflex.

Topics to cover:

- **The senior question.** Which declarations in this codebase deserve a TSDoc block, and which don't? The default answer in 2026 is the wrong one for most teams: developers either TSDoc everything (and the noise drowns the signal) or TSDoc nothing (and the reviewer two months later has to read the implementation to learn the contract). The cut is the *public surface* — the declarations one call site away from the one that owns them are the ones whose contract needs to be stated.
- **What TSDoc is — the 2026 standard.** TSDoc is the Microsoft-stewarded standard for doc comments in TypeScript; `/** ... */` blocks with a small set of block tags (`@param`, `@returns`, `@example`, `@throws`, `@deprecated`, `@remarks`, `@see`). IDEs (VS Code, WebStorm, Cursor) render TSDoc on hover for any consuming call site; TypeDoc and API Extractor consume the same syntax to generate reference sites for libraries. Name the standard once; the rest of the lesson is *when* and *what to write*, not syntax tour.
- **The minimal tag set the course uses.** The 2026 SaaS reach is small: a one-line summary (the first sentence), `@param` only when the parameter's purpose isn't obvious from its name and type, `@returns` only when the return shape carries semantics the type doesn't capture (a sentinel `null`, a partial result), `@throws` for the failure modes the caller must handle, `@example` for the one call shape that the reader needs to see, `@deprecated` with the replacement path. Skip `@remarks` unless the summary is genuinely two-section. Skip `@see` if a single inline link works.
- **The public-surface cut — the inclusion test.** A declaration earns a TSDoc block if (a) it's exported across a module boundary (`src/lib/<feature>/` to `src/app/`, a Server Action to its calling component, a util to anything), (b) the type alone doesn't communicate the contract (preconditions, side effects, ordering, error semantics), or (c) the next caller is a teammate, an AI agent, or future-you reading the IDE hover. A private helper inside the same file, a one-line utility whose body is shorter than the comment, a React component whose props interface already tells the story — none of these earn a block.
- **The SaaS surfaces that always earn TSDoc — the map.**
  - *Server Actions.* The course's API contract (Chapter 7.2). The action function gets a one-paragraph summary of what it does, the auth/tenant precondition (`assumes called inside an authenticated org context`), the side effects (DB writes, email sends, audit log entries), and the failure modes the caller surfaces (`@throws ValidationError when the Zod parse fails`). The Zod schema lives next to the action and is its own reference — link to it from the TSDoc, don't paraphrase column-by-column.
  - *Public functions in `src/lib/<feature>/`.* The feature module's exported functions are consumed across the app; their contracts deserve to be stated in the place a reader hovers. Internal helpers in the same module don't.
  - *Drizzle schema tables.* A one-paragraph header comment per `pgTable` declaration (Chapter 22.1.2's pattern). TSDoc on the exported table identifier so IDE hover at the call site surfaces the table's purpose.
  - *Zod schemas exposed across modules.* The schema's `description` (Zod's built-in) carries field-level intent; the exported schema gets a TSDoc summary for what the shape represents (`UserProfileInput — the shape accepted by the profile update action`).
  - *Webhook handlers.* The Stripe/Resend/auth webhook route file gets TSDoc on the handler describing the contract: which event types it accepts, the idempotency key it uses, the side effects, and the expected response shape. The handler IS the webhook contract doc (Chapter 22.1.2).
- **What does NOT earn TSDoc — the negative space.**
  - *Internal helpers.* A function used in one file by one caller is private; the type and name carry the contract.
  - *React components.* Props are documented by the props type (TypeScript) and component name. A TSDoc block on every component is noise. The exception is a shared design-system primitive where the prop semantics are non-obvious.
  - *Trivial getters/setters/passthroughs.* `const isAdmin = (user) => user.role === 'admin'` doesn't need a TSDoc block; the body is the doc.
  - *Test helpers.* Test code is read by the test author and the reviewer of the test; clear names suffice.
  - *Type aliases that mirror a Zod schema.* `type UserInput = z.infer<typeof userInputSchema>` doesn't need a TSDoc block; the schema is the doc, the type is the derived view.
- **The first sentence is the whole doc — the writing posture.** IDE hover shows the first sentence first; the rest is below the fold. The senior reflex is to write the first sentence so the reader can decide *in two seconds* whether they're at the right function. Verb-first, declarative, no preamble (`Creates an invoice draft and returns its id` — not `This function is used to create an invoice draft`). The rest of the block adds the parts the call site can't see.
- **The "link, don't duplicate" reflex applied to TSDoc.** When the function takes a Zod-validated input, the TSDoc summary says "validates the input against [`userInputSchema`](link)" and lets the schema be the field-by-field doc. When the function reads a config value, the TSDoc says "reads `STRIPE_WEBHOOK_SECRET` from `env.ts`" and lets the env file be the env doc. Duplicating field lists guarantees they drift; the senior reflex is to link.
- **TSDoc and inferred types — when annotation moves to the doc.** The course's TypeScript style is inference-led (parameter/return types annotated, locals inferred). TSDoc doesn't repeat the type — `@param email - the user's email` is fine, `@param {string} email` is not (TSDoc doesn't require types in the tag because the TypeScript signature already carries them; that's the differentiator from JSDoc). State the convention once.
- **`@deprecated` and the supersession path.** A deprecated declaration keeps its TSDoc summary intact and adds `@deprecated use createInvoiceV2 instead — this version is removed in the next major.` IDEs strike through deprecated identifiers at call sites; the call site sees the alternative without opening the file. The senior reflex: deprecate with a path, never just deprecate.
- **`@example` — when one call sample earns its weight.** A complex Server Action with three optional parameters or an unusual call shape earns one `@example` showing the typical usage. Trivial functions don't. The example is runnable as-is (Chapter 4 of the pedagogy doc's code-sample rule applies to TSDoc snippets too).
- **The IDE hover as the senior usability test.** A TSDoc block works if a reader hovering the call site can decide in five seconds whether to use the function and how. If the hover dumps three paragraphs of internal rationale, the block has migrated into explanation territory (which belongs in an ADR, Chapter 22.1.4) or implementation narration (which is anti-value). The senior reflex: re-read the hover, cut what the caller doesn't need.
- **TSDoc as the audit trail for AI-edited code.** An AI agent editing a function reads the TSDoc to understand the contract; if the contract isn't stated, the agent reads the body and infers (often wrongly). The TSDoc block is the *brief* the agent gets — the same brief a human reviewer reads. Don't name AI in lesson material; the property earns its mention here because the same brief serves both audiences.
- **TypeDoc and auto-generated reference — when it earns its weight.** TypeDoc/API Extractor generate HTML reference sites from TSDoc; load-bearing for published SDKs and component libraries, not for closed-source SaaS (the IDE hover is the reference surface). State the threshold once; the course's stack doesn't reach for TypeDoc by default. For an open-source library author the calculation flips.
- **Watch-outs:** TSDoc-ing every private helper "for completeness" — the file becomes 50% comment and the public-surface signal drowns; duplicating the parameter type in the `@param` tag — JSDoc-style, redundant with TypeScript; writing TSDoc that narrates the implementation (`// then we map over the array and filter…`) — that's not a contract, the body is; leaving stale TSDoc when the signature changes — the lint rule (`eslint-plugin-tsdoc` if used, or the PR review) catches it but the discipline is structural (Lesson 22.2.3); using `@returns` to repeat what the return type already says (`@returns the user object`) — only document the return when the *semantics* go beyond the type; one TSDoc block trying to cover three overloads — split or use `@remarks`; treating TSDoc as a place to brain-dump architectural rationale — that's ADR territory.

What this lesson does not cover:

- Inline `//` comments and the why-not-what rule — Lesson 22.2.2.
- The PR-review discipline that keeps TSDoc in sync with signatures — Lesson 22.2.3.
- The Zod schema's `.describe()` mechanic for field-level docs — covered in Chapter 7.2; one-line reference from this lesson.
- TypeDoc HTML generation setup — out of scope; the IDE hover is the deliverable.
- ESLint plugin configuration for TSDoc — out of scope; the discipline is review, not lint.

Estimated student time: 35 to 45 minutes. The chapter's longest lesson — the *cut* is the load-bearing teach, the tag list is small.

## Lesson 22.2.2 — Comment the why, not the what
Learn the why-not-what rule for inline comments, the four kinds that earn their place, the negative space to avoid, and the reflex to carry comments through refactors or promote them to structural enforcement.

Topics to cover:

- **The senior question.** When does a line of code earn an inline comment, and when does it not? The default in most codebases is wrong both ways: junior code over-comments (`// increment i`), and reactionary "self-documenting code" minimalism strips comments that carried real context. The senior cut is the *why* — comments document the constraint, the workaround, the non-obvious trade-off that the code itself can't express. The *what* is the code's job.
- **The why-not-what rule — the inclusion test.** A comment earns its place when a reasonable reader looking at the code alone would ask "why is it written this way?" and the answer isn't visible. If a reader can answer the question by reading the line, the comment is noise. If the answer requires knowledge that lives outside this file (a Postgres quirk, a Stripe API gotcha, a deliberate ordering, a bug the workaround prevents), the comment is the only place that knowledge can live near the code it constrains.
- **The four kinds of comment that earn their place.**
  - *The constraint comment.* `// Postgres truncates timestamp to microseconds; round client-side first so equality compares hold.` The constraint is external, the line looks weird without it.
  - *The workaround comment.* `// Stripe webhook payloads can arrive out of order; the event_id is the dedup key, not the timestamp.` Names the failure mode the workaround prevents.
  - *The intentional-deviation comment.* `// Not using the helper here: it adds a join that doubles the query plan cost on tenant_invoices.` Names the path not taken and why.
  - *The "load-bearing weirdness" comment.* `// Order matters: the audit log entry must commit before the email enqueue, or a crash mid-transaction loses the audit row but keeps the email.` Documents the ordering as part of the contract.
- **What does NOT earn a comment — the negative space.**
  - *Restating the code.* `// loop over invoices` above `for (const invoice of invoices)`. The code already says this.
  - *Section dividers.* `// === HELPERS ===`. If the file needs section dividers, the file is too long; split it.
  - *Author/date stamps.* `// Created by Maria on 2024-03-15`. `git blame` is the author tool; comments aren't.
  - *Commented-out code.* Code that's been commented out is dead; delete it. `git log` is the recovery tool.
  - *TODO without an owner or ticket.* `// TODO: fix this`. A real TODO has either a ticket link (`// TODO(SAAS-1234): fix this`) or an owner and a deadline; bare TODOs accumulate forever. The senior pattern: prefer issue tracker over inline TODOs; reserve inline TODOs for the rare in-flight work that ships in the same sprint.
- **`// why` vs. TSDoc — the boundary.** TSDoc documents the *contract* of a public surface (read from the call site). Inline comments document the *reasoning* behind one specific line or block (read from inside the file). A reader hovering a function shouldn't see "we chose this algorithm because…" — that's not the contract. A reader inside the function looking at a non-obvious line *does* need to see it. The two surfaces are read by different people at different times.
- **`// why` vs. ADR — the boundary.** An inline comment names a *local* rationale that lives next to the line it explains; an ADR names an *architectural* rationale that affects multiple files. "Why is `await sleep(50)` here?" — inline comment, naming the race condition. "Why do we use Drizzle?" — ADR, not a comment on the schema file. The cut is scope: one line vs. the whole codebase.
- **Comments as part of the code — the "never strip in a refactor" rule.** When a function gets refactored — extracted, renamed, simplified — the comments that explained the *why* of its original lines need to move with the lines they explained, not get deleted as collateral. The failure mode is the senior-written `// don't reorder these two await calls — the audit row must commit first` comment getting stripped when a junior cleans up the function, and the bug returning three releases later. The senior reflex: if a refactor would strip a comment, *first* understand what the comment was preventing, *then* either carry the comment with the code or replace it with a structural enforcement (a test, an assertion, a type) that makes the bug hard to write. Comments are the cheapest form of structural enforcement; they're a fallback when the type system can't express the constraint.
- **The "make the bug hard to write" upgrade path.** When a comment is doing real work (preventing a real bug), the senior asks "can I move this from prose to enforcement?" Order-dependent code can be made into a single transactional function. A `// remember to validate` can become a Zod parse. A `// this must be called after auth` can become a function that takes an authenticated-session-typed argument. The comment is the bridge between "the constraint exists" and "the type system enforces the constraint." Don't leave it as a comment forever if you can promote it.
- **Comment density is a code smell — the heuristic.** If a function needs three or more `// why` comments to be understood, the function is probably doing too much or has the wrong abstraction. Comments are signal *that* something is non-obvious; their density signals where the design has room to improve. The senior reflex: when you find yourself writing the third comment in one function, ask whether the function should be split or the abstraction shifted before adding the comment.
- **The voice of a good comment — the writing posture.** Direct, declarative, names the constraint or consequence in one line. The reader has already read the code; the comment adds the part the code can't show. No hedging, no apologies, no narration of the code's structure. "Stripe `payment_intent.succeeded` can fire before `invoice.payment_succeeded`; dedup on payment_intent_id" is the shape; "We need to handle the case where Stripe events arrive in an unexpected order" is the failure mode (verbose and not load-bearing).
- **Language conventions — the small ones that compound.** Single-line `//` for one-line constraints. Block `/* ... */` reserved for TSDoc only (multi-line `//` for prose comments). No ending punctuation needed for fragments; punctuate full sentences. Lowercase first letter for fragments is fine. Match the file's style.
- **Watch-outs:** restating the code (`// set x to 5` above `x = 5`); commented-out code that "might be useful later" — it won't be, delete it; large prose explanations inline that would be an ADR — promote it; TODO comments without ticket numbers or owners that accumulate; comments that lie because the code changed but the comment didn't — the structural fix (Lesson 22.2.3) is reviewer attention; "clever" comments (humor, references) that age badly and confuse the reader who doesn't share the reference; commenting *what* a regex does instead of *why* the regex is needed — the regex's `what` belongs in a unit test, the comment names why it's there; the comment that explains a workaround for a bug that was fixed three years ago — the workaround should be removed, the comment is a fossil.

What this lesson does not cover:

- TSDoc syntax and the public-surface cut — Lesson 22.2.1.
- How comments get caught (or missed) in PR review — Lesson 22.2.3.
- Issue tracker discipline for TODOs — out of scope; the lesson states the rule (link a ticket or own a deadline) and stops.
- Self-documenting code refactoring patterns at length — Chapter 22.3 covers code review's role in surfacing them.

Estimated student time: 30 to 40 minutes. A *Pattern* lesson — the cut is the lesson, the examples are short and load-bearing.

## Lesson 22.2.3 — Docs ship in the PR, or they're already wrong
Learn the five-artifact reflex at PR-open time, the reviewer's doc checklist, the PR-template scaffolding, and the boundary where automation catches mechanical drift but review catches semantic drift.

Topics to cover:

- **The senior question.** Every artifact this unit has named — README, AGENTS.md, ADRs, TSDoc on public surfaces, inline `// why` comments — is one merge away from being silently wrong. The PR that changed the test command, renamed the Server Action, added the env var, swapped the ORM, added the ordering constraint — that PR either updates the docs that describe what changed, or it ships drift. Which artifacts need to change in which PRs, and how does the review surface enforce it?
- **The "docs in the PR" rule — the discipline.** A code change that breaks a doc claim must update the doc claim in the same PR. Not the next PR, not a follow-up ticket, not a "we'll update the README before release" comment. The structural reason: code review is the only checkpoint where a human (or a reviewing agent) looks at the diff with attention; the next checkpoint is production, by which point the doc is wrong and nobody knows. The PR is the moment of leverage.
- **The doc-change map — which artifact moves with which kind of code change.** The senior reflex when opening a PR is to ask, for each artifact, *did this change invalidate a claim that artifact makes?*
  - *README.* Only changes when the local-dev sequence changes, the common-task commands change, or a major stack swap happens. Most PRs don't touch it.
  - *AGENTS.md.* Changes when a convention shifts (new "don't" rule, new module added to the repo layout, build/test command renamed, a new tool added to the stack). Most feature PRs don't touch it; refactor and infra PRs often do.
  - *ADRs.* A new ADR is added when an architectural decision is made (Chapter 22.1.4's three-test inclusion); an existing ADR's status changes to "superseded" when its decision is replaced. Both happen in the PR that ships the decision.
  - *TSDoc.* Changes when a function's signature, contract, side effects, or failure modes change. Adding `@throws`, updating `@param`, marking `@deprecated`, refreshing the summary sentence — all in the PR that changed the code.
  - *Inline `// why` comments.* Change when the *why* changes (the underlying constraint was fixed, the workaround is no longer needed, the ordering requirement was promoted to a transactional function). The comment moves or dies with the code it explains.
  - *Schema headers.* The one-paragraph `pgTable` header comment (Chapter 22.1.2) updates when the table's purpose, scope, or invariants change. A new column doesn't usually trigger an update; a new invariant does.
  - *`.env.example`.* Updates whenever `env.ts` adds, removes, or renames a key. The two files are siblings; they ship together.
- **The reviewer's doc checklist — what to look for.** The senior reviewing a PR runs five checks against the doc surfaces:
  1. *Did any exported function's signature, contract, or throw set change?* If yes, did the TSDoc update?
  2. *Did any environment variable get added or renamed?* If yes, did `env.ts` and `.env.example` both update?
  3. *Did any convention or repo-layout fact change?* If yes, did `AGENTS.md` update?
  4. *Did this PR make an architectural decision?* If yes, is there a new ADR? If it overturns an existing one, is the old ADR's status updated?
  5. *Was a comment stripped in a refactor?* If yes, was the constraint the comment was protecting either preserved or upgraded to enforcement?
- **The "already wrong" framing — why this matters.** A doc that contradicts the code is *worse than no doc* — it costs the next reader the time to read it, plus the time to discover it's wrong, plus the lost trust in every other doc nearby. The lesson states the asymmetry plainly: the cost of writing the doc update in the PR is fifteen minutes; the cost of discovering a wrong doc in production is days. The economics dominate.
- **Where automation helps — and where it doesn't.** Some classes of drift can be caught structurally: TypeScript signatures and TSDoc-tag mismatches catch some `@param` drift (eslint-plugin-tsdoc); a `.env.example` linter can compare keys against `env.ts`; tests that import a Server Action and exercise the success/failure paths catch contract changes the TSDoc claim missed. But the load-bearing checks — did the *prose* still match the *behavior* — are review-level. State the threshold once: lint catches mechanical drift, the reviewer catches semantic drift. Don't pad the lesson with tool tours; the discipline is the teach.
- **The PR template as enforcement scaffolding.** A short PR template with two checkboxes — *I updated the docs affected by this change* and *If I added a new dependency or env var, the README/AGENTS.md/`env.ts` updated accordingly* — forces the author to look at the doc surfaces before requesting review. The reviewer sees the checkboxes and double-checks. Not a paragraph essay; two lines that move the reflex into the workflow.
- **The "is this a doc-only PR" failure mode.** Teams that ship code-and-docs separately ("we'll write the docs after the sprint") end up shipping the docs never. The lesson states the structural counter: a PR that changes a contract without updating its doc is incomplete and gets a *block* in review (Chapter 22.3 covers the language of blocking). The doc isn't a separate concern; it's part of the change.
- **The conventional-commits and changelog connection — named once.** If the team uses conventional commits, the commit message body can carry a one-line note that the PR also touched docs; the changelog generation picks it up. Useful but not load-bearing; the discipline is the PR review, not the commit prefix. State once; move on.
- **The doc-rot detection question — what to do at the boundaries.** Some docs (the README's "Getting started" sequence, the AGENTS.md "Conventions" list) don't get touched by individual feature PRs but rot anyway over time — a quarterly review where someone follows the README from a clean clone and notes every deviation is the cheapest structural check. State the cadence once: most teams review the meta-docs every quarter, not every PR.
- **The discipline scales with team size and AI activity.** With one developer and no AI agents, doc rot is mostly self-inflicted and surfaces fast. With three developers, doc rot is a coordination tax. With three developers plus AI agents editing the codebase in parallel, every artifact's accuracy is load-bearing because the agent reads the doc to decide what's safe to change; a wrong AGENTS.md or a stale TSDoc steers the agent wrong on the next change. The discipline isn't optional in 2026 the way it was in 2016. Name the property; don't moralize.
- **The senior reflex at PR-open time — the summary.** Before requesting review, the senior asks: *what did this PR claim, and which docs make a claim about that?* The list is short — the same five artifacts, every PR — and the question is automatic after a quarter of practice. The lesson's payload is the five-artifact reflex, not the tool tour.
- **Watch-outs:** "I'll update the README in a follow-up" — the follow-up doesn't ship; the discipline is to merge the doc with the code or not merge; treating doc updates as cosmetic and skipping review on the doc parts of the diff — the doc is the contract, review it; updating the doc in a *separate* PR that lands after the code — the merge window between them is the drift window; relying solely on automated drift detection — it catches mechanical mismatches, not semantic ones; the "we have a docs day every Friday" anti-pattern — the docs ship with the code; quarterly reviews of meta-docs (README/AGENTS.md) becoming "everyone is too busy" — the cadence has to be defended; checking the doc-update box on the PR template without actually updating any docs — the reviewer's job to catch.

What this lesson does not cover:

- The specific PR-review skills (suggesting vs. blocking, the language of disagreement) — Chapter 22.3.
- Automated doc-drift CI tooling tours (Mintlify, Dosu, BlockWatch) — out of scope; the lesson names the threshold and stops.
- Changelog-generation tooling — out of scope; named once.
- The new-hire onboarding workflow (a separate use of the same doc surfaces) — out of scope.
- Versioned docs for published SDKs — out of scope; the course's surface is closed-source SaaS.

Estimated student time: 30 to 40 minutes. A *Pattern* lesson — the five-artifact reflex is the deliverable, the rest supports it.

- The why-not-what rule for inline comments — comments document the constraint, workaround, intentional deviation, or load-bearing weirdness; the code documents the *what*.
- The four kinds of `// why` comment that earn their place (constraint, workaround, intentional deviation, load-bearing weirdness) and the negative space (restating code, dividers, author stamps, commented-out code, bare TODOs).
