## Lesson 3 — The use cache directive

- **Taught** — Full anatomy of `'use cache'`: one directive with three placements (file = caches every export, all must be async; component = caches render output, props in key; function = caches return value, the workhorse), and this is where the L1/L2 opaque-marker staging ends (real bodies shown now). The cache key composes four ingredients — build ID (deploy invalidates all), function ID (code edit invalidates), serializable arguments/props, dev-only HMR hash — and same args → same key → one shared cross-request entry. The serialization contract is asymmetric (args use stricter server serialization, return values looser client serialization → can return JSX, cannot accept it as a normal arg). Pass-through lets a cached shell receive non-serializable `children`/slots/Server Actions as long as it never introspects them (resolves L2's shell-with-a-hole at the code level). Default store is in-memory LRU on the server. Senior pattern: every read is a small `'use cache'` fetcher in the data layer; wrap-to-cache replaces the deprecated implicit `fetch()` caching.
- **Cut** — Chapter outline's "async-only is a build error for sync functions" claim not taught as a rule (not in built lesson). Outline's `.next/cache` / Vercel-distributed-cache storage detail simplified to "in-memory LRU + serverless-vs-self-hosted persistence."
- **CRITICAL CORRECTION (deliberate divergence)** — The chapter outline AND `Code conventions §Caching` say "closures are forbidden / cached functions cannot capture outer variables." **This is a known error and is NOT taught.** Per official Next.js v16 docs, closure variables ARE captured and folded into the cache key (bound as if arguments). What is actually forbidden: request-time APIs inside a cached scope, and non-serializable captures. The built lesson teaches the corrected model in prose only (~lines 194-196: "Closures are captured into the key, not forbidden"; explicitly calls out that "a lot of writing on `'use cache'`" is wrong) — there is NO inline author-note comment in the shipped MDX, so downstream agents must not regress this from prose alone. Downstream lessons and any code-conventions update must follow the corrected model.
- **Debts** — Defers (with one-line teasers / named owners): `cacheLife` + `cacheTag` (L4 — shown only as the commented teaser `// cacheLife + cacheTag → next lesson` inside one fetcher; "sensible default lifetime" stated without numbers); React `cache()` per-request memoization (L5 — contrast seeded as "`'use cache'` is cross-request, `cache()` is per-request"); invalidation surface `updateTag`/`revalidateTag`/`revalidatePath` (L6); async request API *syntax* / how to read `cookies()` etc. (L7 — lesson only says they're forbidden inside cached scopes); `generateStaticParams` (chapter 034). Relies on: server/client serialization boundary + allowed/rejected types (030), `<Suspense>`/streaming (031), `connection()` + cached-can't-nest-dynamic (L1), PPR shell/holes (L2), `'use client'`/`'use server'` directive family (prior units).
- **Terminology** — `Term` glosses: **cache key** (identity a result is stored/looked-up under; same key = shared entry), **serializable** (encodable + reconstructable; the cross-boundary requirement, same as the 030 wire), **pass-through** (non-serializable value received but never introspected, so not in key, not cached), **LRU** (least-recently-used eviction, the default backend). Named-not-taught variants: **`'use cache: remote'`** (cross-instance store like Redis/KV; cache-handler config) and **`'use cache: private'`** (the only variant allowed to read request APIs; caches per-user in browser memory only, never server-stored, never prerendered — the rarely-needed escape hatch). Allowed across boundary: primitives, plain objects, arrays, `Date`, `Map`, `Set`, typed arrays/`ArrayBuffer` (+ JSX for return values only). Rejected: class instances, functions, `Symbol`, `WeakMap`/`WeakSet`, **`URL`** (it's a class instance — pass the string); `Temporal` values encode to ISO strings at the boundary (project convention, recurs course-wide).
- **Patterns and best practices** — Data layer = many small single-quoted `'use cache'` fetchers, imported by components; entries compose (cached shell + cached fetcher, each under its own key). Pass request data IN as arguments; let personalized-but-fresh content stream as a dynamic sibling hole rather than reaching for `'use cache: private'`. Pass plain serializable fields, never class/SDK instances (import `db`/loggers inside the cached function). Non-determinism (`Date.now()`, `Math.random()`, `crypto.randomUUID()`) freezes inside a cached scope — choose deliberately: defer via `await connection()` + `<Suspense>` for per-request freshness, or accept a shared occasionally-refreshed value. Cache the smallest useful slice (storage has a budget). Violations (request API in cache, non-serializable arg) are build errors, not silent slow paths.

## Lesson 5 — Per-request memoization with React cache()

- **Taught** — The two-cache-layer model is the whole lesson: React `cache()` is **per-request** (born when a render starts, dedupes within that one render, discarded when it ends — no persistence, eviction, or invalidation) vs `'use cache'` **cross-request** (persists, shared across users). `cache` is a named import from `react` (NOT `next/cache`); wrap a fn with it → memoized version that runs once per render no matter how many callers, all sharing the same returned Promise. Must sit at **module scope** (inside a component → fresh memoizer every render → silently zero dedup, no error). Memo keyed by **argument identity**: primitives by value (dedupe), objects/arrays by reference (two fresh literals = two keys = runs twice, silently). The one decision: does the work read request data (`cookies()`/`headers()`/session/`params`/`searchParams`)? Yes → `cache()` (it can't be `'use cache'` — build error, so `cache()` is the *only* option); no + want cross-request share → `'use cache'`; no + only want in-render dedup → `cache()` (rare). Both directives on one fn = legal, redundant, not rivals (different layers). Closes doors: `cache()` doesn't persist / doesn't invalidate / doesn't cross the server-client boundary / is not `useMemo` (client-fiber hook, React Compiler handles it).
- **Cut** — none significant. Outline's "doesn't dedupe across Suspense boundaries with differing args" watch-out not surfaced as a distinct point (subsumed by argument-identity rule).
- **Debts** — Defers: async request API *syntax* / `headers()` Promise shape + `React.use()` unwrap → L7 (used here as an opaque call, one tooltip). Better Auth `auth.api.getSession`, the `auth` instance, redirects, org plugin, and the full `requireUser`/`requireOrgUser` bodies → **Unit 8** (shown shaped-but-stripped with `// Unit 8` markers — do NOT flesh out). Names as known/recap: `'use cache'` cross-request + request-APIs-forbidden-in-cached-scope (L3), `cacheLife`/`cacheTag` (L4, named only to say they don't apply here), invalidation surface (L6, N/A here), React Compiler / `useMemo` (ch 024). Relies on: Server Components compose without auto-threading props (030 — the *cause* of the duplicate-read problem), async Server Components / `await` (030).
- **Terminology** — `Term` glosses: **request-scoped** ("lives only for one server render of one request, then discarded"), **memoization** ("store a fn's result, return it on a repeat call with the same input"), **serializable** (one-word recap from L3). Headline one-liner to carry forward: *"`cache()` deduplicates within one render and then forgets; `'use cache'` persists across renders and across users."* Color convention for this chapter's two-layer figure: **blue = cross-request store, violet = per-request memo** (NOT orange — orange stays reserved for dynamic holes). "request layer" (above) / "cross-request layer" (below) — request-scoped readers above, cross-request fetchers below.
- **Patterns and best practices** — **The `lib/auth.ts` session-read ladder is canonical** (matches Code conventions §Authentication, ships Unit 8): one `cache()`-wrapped `getSession` (the single cached read, reads `headers()` internally), `getCurrentUser` (→ `User | null`) built on it, `requireUser`/`requireOrgUser` siblings on the same cached read → session hits DB exactly once per render. Every Server Component imports the reader it needs and calls freely — **no prop-drilling, no fetch-at-top-and-thread-down, no Context** for server work. **Prefer primitive arguments** (`userId`/`orgId` strings) for bulletproof dedup; if an object is unavoidable, pass a stable once-per-render reference, not a fresh literal per call site; the no-argument reader (reads `headers()` itself) sidesteps argument-identity entirely. Two layers compose: request-scoped `cache()` readers on top feeding request-independent `'use cache'` fetchers (`getProductCatalog()`) below. Note: `getCurrentUser` shown with **no arguments** by design (§Authentication-aligned + cleanest illustration of the no-arg shape).

## Lesson 6 — Invalidating after a mutation

- **Taught** — Closes the cache loop: pays off L4's inert `invalidate(...)` placeholder + imports L4's `src/lib/tags.ts`. The whole lesson hangs on one question — **does the user expect to see their own change the instant this action finishes?** Yes (form in current session) → read-your-writes; No (webhook/cron/admin/out-of-band) → eventual/SWR. Four tools: (1) **`updateTag(tag)`** — new in 16, `next/cache`, *one arg only*, expires the tag so the next read **blocks** for fresh = read-your-writes; **throws outside a `'use server'` function** (Server-Action-only, because RYW needs a "next render the user is staring at"). (2) **`revalidateTag(tag, profile)`** — `next/cache`, marks stale + lazy refresh on next visit (no call-time fan-out → safe at scale), runs anywhere (Server Actions + Route Handlers + jobs), the webhook/cron tool. (3) **`revalidatePath(path[, 'page'|'layout'])`** — invalidates by URL; only for path-as-resource (sitemap/OG image/feed) where no entity exists to tag; route-coupled + has a documented over-invalidation quirk → not the default. (4) **`router.refresh()`** — client, from `useRouter()` (`next/navigation`); re-pulls the route's RSC payload without full reload (state/focus/scroll survive) but **does NOT bust the `'use cache'` store** → must pair: invalidate on server first, then `router.refresh()`. The five-seam Server Action shape (`parse → authorize → mutate → revalidate → return`) is previewed; this lesson owns **only the revalidate seam**.

## Lesson 7 — How a route declares itself dynamic (Request APIs and segment config)

- **Taught** — Chapter capstone, answers "how does a route declare itself dynamic." Two halves of one question across two eras. **(1) Legacy route segment config is gone.** The four module-scope `export const` exports (`dynamic`, `revalidate`, `fetchCache`, `runtime`) were the implicit, route-global "flag panel" the student must *recognize* (not write); under `cacheComponents: true` the first three are **rejected (build error)**. Migration table is the keystone artifact: `dynamic='force-dynamic'`→delete; `dynamic='force-static'`→`'use cache'`+`cacheLife('max')`, drop request reads; `revalidate=N`→`'use cache'`+closest `cacheLife` preset (not raw seconds); `fetchCache`→delete (directive subsumes it); `runtime='edge'`→delete, Node is default, edge logic moves to `proxy.ts`.
