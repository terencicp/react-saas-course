# Chapter 001 — The JavaScript value model

## Lesson 1 — Bindings, not boxes

- **Taught** — `=` binds names to values; primitives copy on assignment, objects copy a reference; pass-by-value with reference values explains the rebind-vs-mutate distinction; spread is the 2026 shallow-copy default; `structuredClone` is the 2026 deep-copy default; `JSON.parse(JSON.stringify(x))` named only as legacy to recognize.
- **Cut** — Did not detail `Error` non-cloneability in older runtimes; did not enumerate the full set of `structuredClone` failure modes beyond functions, class instances, DOM nodes, WeakMap/WeakSet, getters/setters.
- **Debts** — Forward-links promised: `===` reference-equality and `Object.is` (Lesson 2), `const` binding-vs-value immutability and `Object.freeze` (Lesson 6), Server Action serialization superset of structured clone (Ch. 030 L4), React state immutability + Immer (Ch. 023 L4, Ch. 024 L2). Primitive enumeration depth deferred to Ch. 004 L1.
- **Terminology** — "binding" (name → value), "reference" (`<Term>` defined), "binding diagram" (two-panel: primitive boxes vs. shared object box with arrows), "copy then modify" reflex, "what can cross a serialization boundary" frame, `DataCloneError`, "cyclical references" (`<Term>` defined).
- **Patterns and best practices** — All `.ts`, inference-led, `const` only (no `let`); semantic domain names (`user`, `address`, never `foo`); one-parameter helpers; nested-record shapes (`{ name, address: { street, city } }`) over branded/`User`-typed shapes for pedagogy; spread for shallow copies; `structuredClone` for deep; never write `JSON.parse(JSON.stringify(x))`; never mutate values received from outside.
- **Misc.** — Lesson uses a custom `BindingDiagram.astro` at `src/components/lessons/001/1/`. ## Lesson 2 — What === actually compares

- **Taught** — `===` compares values for primitives and references (allocation identity) for objects; `==` is forbidden, enforced by Biome `noDoubleEquals`; the two edge cases `NaN === NaN` is `false` and `+0 === -0` is `true`; `Object.is` as the rare escape hatch matching `===` except on those two cases; `Number.isNaN`/`Number.isFinite` over coercing globals; JavaScript ships no structural-equality operator.
- **Cut** — Did not enumerate the `==` coercion table (deliberate); did not mention `BigInt` cross-type equality; did not show `Number.isInteger`/`Number.isSafeInteger` (Lesson 3 territory); no `Date` equality; no Records/Tuples proposal; no `valueOf`/`Symbol.toPrimitive`.
- **Debts** — Forward-links promised: `??` operator and explicit nullish checks (Ch. 002 L5), IEEE 754 depth on why `NaN` exists (Ch. 001 L3), React `useState`/reactivity bailout via `Object.is` (Ch. 023), Zod string/wire validations and ID-based comparisons (Unit 6), Drizzle ID comparisons (Unit 5), discriminated unions for shape-collapsing comparisons (Ch. 005). Biome `noDoubleEquals` config with `ignoreNull: false` referenced as set up later.
- **Terminology** — "are these the same value?" (primitives) vs. "are these the same allocation?" (objects); "two edge cases `===` gets wrong"; "namespaced `Number.*` over coercing globals"; "reach for X by default, recognize Y on sight"; `<Term>` definitions added for "IEEE 754" and "structural equality".
- **Patterns and best practices** — `===` everywhere, never `==`; `Number.isNaN`/`Number.isFinite` over globals; `Object.is` only with an explanatory comment; compare by primary key (`a.id === b.id`); derive stable string keys for caching/dedup; use discriminated unions instead of deep-comparing shapes; rely on Biome lint as build-time backstop rather than discipline.
- **Misc.** — One `<Aside type="caution">` is the only place `==` appears in code blocks (cited as what-not-to-write — don't normalize away). `<MultipleChoice>` "Check yourself" uses unique variable names (`userA`/`userB`, `parsedValue`, `prevState`/`nextState`) distinct from prose names (`user`/`alias`). `<ScriptCoding>` block for `Number.isNaN` vs. global `isNaN` divergence on five inputs. ## Lesson 3 — Store cents, not dollars

- **Taught** — IEEE 754 consequence (integers exact to ±2^53−1, fractions approximate); integer-cents rule for money with `Number()` + `Math.round(dollars * 100)` round-trip + `(cents/100).toFixed(2)` display; `Number.isFinite` / `Number.isInteger` / `Number.isSafeInteger` boundary-check vocabulary; `Number('') === 0` and `Number('   ') === 0` footgun + `input.trim() === ''` guard; `BigInt` reach for 64-bit IDs / large counters / crypto only (not money), with its friction (no number mixing, no `Math.*`, `JSON.stringify` throws, distinct `bigint` TS type, `n` suffix literal); `Number()` as default conversion with `parseInt(_, 10)` for dimension strings, `parseFloat` rare, `+input` as redundant shortcut; `NaN` propagation fix is at the boundary, not downstream.
- **Cut** — No `Math.abs(a - b) < EPSILON` tolerance comparison; no decimal libraries (`decimal.js`, `big.js`); no Decimal128 / TC39 decimal; no IEEE 754 bit-layout diagram; no banker's rounding detail; `Number.parseInt`/`Number.parseFloat` namespaced aliases named in one line only.
- **Debts** — Forward-links: `Intl.NumberFormat` for locale-aware currency display (later unit, "where display localization lands"); Zod absorbs the boundary pattern as `z.string().min(1).pipe(z.coerce.number().int().finite())` (Unit 6); Stripe integration on integer cents (Unit 11); Drizzle stores money as Postgres `bigint` integer-cents column (Unit 5); `Math.*` surface (Chapter 003); `as const`/`satisfies`/branded types (Chapter 004); Temporal for ms-durations (Chapter 009).
- **Terminology** — "integer cents", "boundary discipline" / "boundary check", "minor unit" (ISO 4217), `<Term>` definitions added for "IEEE 754" (reused from L2) and "ISO 4217". The `n` suffix BigInt literal. "use `number` until a real overflow forces the switch."
- **Patterns and best practices** — Money stored as integer cents in a Postgres `bigint` column via Drizzle; `Number(input)` + `Number.isFinite` at every string→number boundary; explicit empty-string guard (`input.trim() === ''`) before `Number()`; `Math.round` (never `Math.floor`/`Math.trunc`) for cents conversion; `Number.isSafeInteger` as upper-bound guard at external boundaries; reach for `BigInt` only at the three named sites; pass `cents` to Stripe and sum cents in SQL — never floats. All snippets `.ts`, `const`-bound arrow functions, inference-led except boundary-function return-type annotations (e.g. `(input: string): number`). Variable names carry the unit (`dollars` vs `cents`).
- **Misc.** — Deliberate convention divergence: the `<ScriptCoding>` exercise uses `runner="vanilla"` with a `function dollarsToCents(input)` declaration in `.js` shape (not TS arrow) — keep as-is; the runtime check is the point. ## Lesson 4 — Why `.length` lies

- **Taught** — `string.length` counts UTF-16 code units (a serialization detail); three counts a senior picks between by intent (code units via `.length`, code points via `[...str].length`/`Array.from`, grapheme clusters via `Intl.Segmenter`); canonical segmenter one-liner `[...new Intl.Segmenter('en', { granularity: 'grapheme' }).segment(input)].length` as the 2026 default for user-facing length; `'word'`/`'sentence'` granularities named with `isWordLike` flag; never `slice(0, n)` on user-facing strings (produces mojibake); `.normalize('NFC')` once at the storage/write boundary (not at each comparison); the four normalization forms (NFC default, NFD for accent-insensitive search, NFKC/NFKD compatibility folding for search only); senior `String.prototype` working surface (`includes`/`startsWith`/`endsWith`, `at(-1)`, `slice`, `split`/`join`, `replaceAll`, `trim*`, `padStart`/`padEnd`, `localeCompare`, `normalize`); legacy surface to recognize and avoid (`substr`, `substring`, `escape`/`unescape`, HTML wrappers like `bold`/`italics`).
- **Cut** — No Unicode internals (planes, BMP, UTF-8/16/32 trade-offs); no regex Unicode flags (deferred to Ch. 003); no i18n broadly; no polyfill/feature-check discussion for segmenter; no `Intl.Collator` depth; no string performance characteristics (rope/V8 internals); no Zod string schema details.
- **Debts** — Forward links: Zod `.length()` defaults to code units, needs custom refinement using segmenter (Unit 6); display-text truncation iterating segments (later UI lesson); `Intl.NumberFormat`/`Intl.Collator`/`Intl.DateTimeFormat` as the platform-native i18n surface (later units); regex Unicode flags `u`/`v` and property escapes (Ch. 003 regex lesson); `String.raw` and tagged templates (Lesson 5 of this chapter); Drizzle `text`/`varchar` don't normalize — Zod schema or action handler normalizes before write (Unit 5/6).
- **Terminology** — `<Term>` definitions added for "UTF-16 code units" (16-bit chunks, surrogate pairs for chars above U+FFFF), "grapheme cluster" (one user-perceived character, may span multiple code points), "serialization detail" (property of storage form, not of value). "Three counts, three reaches" picker frame. "Normalize once, at the database write boundary" reflex. "NFC" / "NFD" / "NFKC" / "NFKD" named with one-line triggers. "Spread-then-`.length`" as the idiomatic segmenter count form.
- **Patterns and best practices** — Reach for `Intl.Segmenter` with `granularity: 'grapheme'` for any user-facing length check; treat segmenter as baseline (no feature check, no polyfill, no try/catch — Node 24 LTS pinned); normalize to NFC at the write boundary (storage/Zod/action handler), not at every comparison; never `slice(0, n)` on user-facing strings — iterate segments and concatenate up to limit; use `Number.*` analogue here: prefer `includes`/`startsWith`/`endsWith` over `indexOf !== -1`, `at(-1)` over `string[string.length-1]`, `slice` over `substring`/`substr`, `replaceAll` over `replace(/g)` for literal needles, `localeCompare` for user-visible sorts. All snippets `.ts`, `const`-bound arrow functions, inference-led; boundary function `countCharacters` has `: number` return-type annotation. Locale `'en'` everywhere (English-only UI default). Single quotes for strings (Biome). Result shown with comment tail (`// 1`, `// false`) except inside `<PredictOutput>` blocks (which use `console.log`).
- **Misc.** — Custom `<ThreeCountsTable />` component at `src/components/lessons/001/4/` is the visual anchor (HTML+CSS table per diagrams index, three input rows × three count columns, color-coded headers: red code units / orange code points / green grapheme clusters). `<PredictOutput>` opener on `'hello'.length`/`'🇺🇸'.length`/`'👨‍👩‍👧‍👦'.length` with expected `5/4/11`. `<ScriptCoding runner="sandpack">` exercise (deliberate divergence from L3's vanilla — segmenter call is short, matching production `.ts` arrow form is the higher-leverage choice here); six tests including ASCII/flag/family/combining-mark/empty/multi-emoji. `<VideoCallout>` to Studying With Alex Unicode video included (the outline suggested skipping video but the final lesson kept one). ## Lesson 5 — Backticks and tagged templates

- **Taught** — Template literals (backticks) as the default for any interpolated or multi-line string; `+` concatenation is a smell; backticks preserve all whitespace between them (indentation gotcha); tagged-template mechanics — `` tag`Hello, ${name}!` `` is equivalent to `tag(['Hello, ', '!'], name)`, receiving a `TemplateStringsArray` of static segments plus rest-spread interpolated values (always one more segment than value); `` sql`...` `` (Drizzle) auto-parameterizes interpolated values as bound placeholders (`$1`, `$2`), with `sql.raw` named as the rare escape hatch that never takes user input; `` dedent`...` `` (npm package) strips common leading whitespace and trims outer newlines; `String.raw` named once for recognition only.
- **Cut** — Nothing significant from the outline scope; the "off-by-one missing slashes" URL framing from the chapter outline was dropped in favor of the SQL-injection + multi-line-prompt pairing.
- **Debts** — Forward links: Drizzle `sql` full treatment incl. `sql.raw`/`sql.join`/`sql.identifier` (Unit 5); `dedent` reach for email bodies (Unit 7), system prompts (later units, incl. AI Unit 22), and test fixtures (Unit 18); the course never asks the student to author a production tag function. Course-default convention reaffirmed: single quotes for plain strings (Biome), backticks only when interpolating or multi-line.
- **Terminology** — `<Term>` definitions added for "template literal", "tagged template", "`TemplateStringsArray`", "`dedent`". "The tag is a function call wearing a string costume." - **Patterns and best practices** — Reach for backticks whenever a `${}` or newline appears; never `+`-concatenate to interpolate; every hand-written SQL query goes through the ORM's `sql` tag (Drizzle); never inline user input via `sql.raw` — only fixed allowlisted identifiers; every multi-line string over a few lines uses `dedent` (one import, source stays aligned with surrounding code, output is indentation-free); the `currency` tag example reaffirms integer-cents discipline (`(cents / 100).toFixed(2)` for display).

## Lesson 6 — const binds, it doesn't freeze

- **Taught** — `const` locks the binding (arrow), not the value (box); `const` by default, `let` only for genuine reassignment (accumulators, branch-assigned values, classic `for` counters); `var` named once for legacy recognition (function-scoped, no TDZ); block scope is lexical at `{}`; Temporal Dead Zone (TDZ) — `const`/`let`/`class` hoist name only, accessing before declaration throws `ReferenceError`; `var` hoists name + `undefined` (the bug class TDZ kills); function declarations hoist their full body; `Object.freeze` is shallow runtime freeze (rejects top-level writes in strict mode, allows nested mutation), course skips it in favor of compile-time `readonly`/`as const`; `const` narrows primitive literals (`'hello'`) but widens object property types (`{ feature: boolean }`, not `{ feature: true }`).
- **Cut** — No deep TDZ tour (class TDZ, `typeof` exception, destructuring defaults); no variable shadowing; no `const`/`let` performance comparison; no module-vs-global scope; no closures/scope chains (deferred to Ch. 002 L7).
- **Debts** — Forward links: `readonly` field modifier (Ch. 004 L2); `as const` mechanics and inference algorithm (Ch. 004 L7); closures and lexical capture (Ch. 002 L7); function declarations vs. expressions in depth (Ch. 002 L1); React state immutability + reference equality re-renders (Ch. 023 L4, Ch. 024 L2). Biome `useConst` rule named as already wired in chapter project's `biome.json` (set up in Unit 1).
- **Terminology** — `<Term>` definitions added for "binding" (reused from L1), "Temporal Dead Zone" (TDZ), "hoisting". Core metaphor: "`const` locks the arrow, not the box" — reuses L1's binding diagram (`BindingDiagram.astro`) as chapter's spine visual. "Declare bindings at the top of the scope" reflex. "The senior reach" / "the form to leave behind" framing across the three accumulator variants. Three sorting buckets: `const` / `let` / `prefer-const` (= "const, looks like let but isn't").
- **Patterns and best practices** — `const` by default for every binding; `let` only when binding genuinely reassigns; reach for `.reduce` over manual `let total = 0` loop when shape fits; `for...of` loop variable is always `const` (language rebinds per iteration); `.push`-built arrays use `const` (mutation through arrow, no reassignment); declare bindings at the top of the scope where used (TDZ-immune by construction); never write `var`; convert legacy `var` to `let` or `const`; never reach for `Object.freeze` in app code — use TS `readonly`/`as const` for compile-time immutability with zero runtime cost; `Object.freeze` only when runtime tamper-resistance is genuinely required (rare in apps you own end-to-end); rely on Biome `useConst` as build-time backstop, not manual discipline.
- **Misc.** — Reuses `BindingDiagram.astro` from `src/components/lessons/001/1/` — no new lesson-specific component.
