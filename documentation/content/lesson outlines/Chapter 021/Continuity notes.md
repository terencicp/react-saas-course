## Lesson 2 — OKLCH, color-mix(), and the alpha syntax

**Taught:** OKLCH as the token storage space (L 0–1 perceptually uniform, C 0–~0.4, H 0–360; `0 0` = neutral) with the two senior reasons — channel independence (vs. HSL lightness drift) and P3 wide gamut with automatic sRGB downmap; `color-mix(in oklch, color, mix N%)` as the runtime mixer for hover/active (`black/white N%`), tinted surfaces (`var(--card), var(--primary) 4%`), and fade-to-`transparent`, with interpolation space being load-bearing (`in srgb` = muddy midpoint); the `/N` alpha modifier revealed to compile to `color-mix(in oklab, color N%, transparent)` and why `oklab`≠`oklch` but is interchangeable when one side is transparent; `opacity-*` (composites whole subtree, makes a stacking context) vs. `/N` alpha (one declaration) as a design decision; semantic OKLCH tokens flipping in `.dark` by moving only L; `prefers-color-scheme` (OS) vs. `.dark` (site) precedence resolved by `next-themes`; WCAG 4.5:1 body / 3:1 large reflex; `currentColor`, arbitrary `bg-[oklch(...)]`, hex-is-read-only.

**Recall hooks used (must stay consistent):** OKLCH literals seen in dark-mode tokens (ch018 L5); semantic role tokens + `-foreground` pairs (ch018 L5); three-tier token model + "express states with opacity, not new `-hover` tokens" + promotion threshold (ch019 L4); `/N` modifier named in variant grammar (ch018 L1); `next-themes` toggling `.dark`, `defaultTheme="system"` (ch018 L6); `@custom-variant dark` (ch018 L5 / ch019 L1); `opacity` compositing "look-alike" (ch019 L2); stacking context (ch020 L9); `currentColor` for SVG icons/borders (ch019 L2); arbitrary-value escape-hatch ladder + "repeated `[...]` = grow the scale" (ch018 L1).

**Terminology / mental models:** "token consumer → token author" is the lesson's spine; "interpolation space" (the road the browser travels between two colors); "derive a state, don't mint a `-hover` token"; decision rule = "fade the whole thing → `opacity-*`; fade one layer, keep content crisp → `/N` alpha"; "lower the L, keep the H" for the dark swap; OKLAB = Cartesian sibling of OKLCH (no hue axis, for straight-line fades). Canonical reaches: hover = `color-mix(in oklch, var(--primary), black 8%)`; tint = `color-mix(in oklch, var(--card), var(--primary) 4%)`; backdrop = `bg-black/50`; hairline = `border-white/10`; disabled = `opacity-50`.

**Browser-support facts stated (keep coherent):** `color-mix()` Baseline Widely Available since 2023 (Chrome 111, Firefox 113, Safari 16.2) — a default tool, no caniuse check. Tailwind v4 internal alpha mix uses `in oklab`; manual tints use `in oklch`. Tailwind v4 + shadcn ship their palettes/tokens in OKLCH.

**Patterns / best practices (for project chapter):** store all color in OKLCH, never ship a new hex literal (`transparent` + `currentColor` are the only self-compiling exceptions); derive hover/active/tint with `color-mix(in oklch, …)` rather than minting `-hover` tokens; promote a recurring state to a token (e.g. shadcn `--accent`) past the repetition threshold; always pass `in oklch` to `color-mix()` (`in srgb` is a code-review smell); pick `opacity-*` vs `/N` by design intent, never `opacity` as a "make-background-see-through" shortcut; reference semantic tokens (`bg-card text-card-foreground border-border`), never `dark:bg-zinc-900` next to `bg-white`; trust the contrast ratio (DevTools / WebAIM) over the eye, lean on contrast-audited `-foreground` pairs; tokens defined in `:root` + re-pointed in `.dark` shown as fragments of `app/globals.css`, never whole files.

## Lesson 3 — Borders, radius, and the elevation scale

**Taught:** elevation as one ordered scale, not per-component decoration — borders/radius/shadows are the design system's "elevation language". Borders: bare `border` = `1px solid var(--color-border)` (works because Preflight's `--default-border-color`, not raw-CSS `currentColor`); width (`border-2`/`-4`), sides incl. logical `border-s`/`-e`, semantic color overrides (`border-input`/`-destructive`/`-primary`), `border-white/10` as the canonical dark-surface hairline; `border-dashed` is a *signal* ("empty / drop here"), not decoration; `divide-y divide-border` on the parent for between-row separators (skips outer edges, no `last:` override). Radius: `rounded-xs`(2px)→`rounded-4xl`(32px) + `rounded-full`/`-none`, per-corner/logical variants named not dwelt; the discipline = pick one or two radii, tune shadcn's single `--radius` token (components derive corners from it); `rounded-full` on non-square = pill not circle, avatar pattern = `aspect-square rounded-full`. Shadows: `shadow-2xs`→`shadow-2xl` + `shadow-none` ladder mapped one-step-per-tier (page=none, resting card=`shadow-sm`, hover lift=`shadow-md`, dropdown/tooltip=`shadow-md`, dialog=`shadow-lg`/`xl`); `shadow-2xl` on a card is the headline smell ("a dialog that forgot to open"); shadows are translucent-black so they tint over any surface; each rung stacks contact+ambient layers; colored `shadow-blue-500/50` is a deliberate brand glow; `box-shadow` is NOT clipped by `overflow-hidden`; inset family `inset-shadow-*`. Focus ring: `outline`/`ring-*` take no layout space, `border` does — that geometry (not aesthetics) is why focus rings are outlines (a border ring reflows neighbors on every focus); canonical `focus-visible:outline-2 outline-offset-2 outline-ring`, or shadcn's `focus-visible:ring-2 ring-ring/50 ring-offset-2 ring-offset-background` halo; `ring-*` is box-shadow layers under the hood (follows radius, themable gap, shares shadow channel). `drop-shadow` (a filter) follows the rendered silhouette vs `box-shadow`'s bounding box — reach only for irregular/transparent shapes (icons, cut-out logos). `backdrop-filter`/`backdrop-blur` blurs content *behind* a translucent element — the one glass-morphism reach is the sticky `bg-background/70 backdrop-blur` header.

## Lesson 4 — Pseudo-classes and the :has() parent selector

**Taught:** pseudo-classes as *names of facts the browser already tracks* (reach for `useState` only when no pseudo-class matches) — this is the CSS-layer extension of ch018 L4's "can the DOM already tell me this?" spine. Interaction set: `:hover` (pointer over), `:active` (pressed right now, mouse or finger; canonical `active:scale-[0.98]`), `:focus` (focus from *any* source incl. click — the trap), `:focus-visible` (browser heuristic = keyboard/programmatic focus, suppressed on plain click — the focus reflex on every interactive element). `:focus-within` = ancestor reacting to a focused descendant (canonical `focus-within:ring-2` form-row wrapper); framed as a "single-purpose parent selector," `:has()` is the general one. Form-state set: `:disabled` (canonical `disabled:opacity-50 disabled:pointer-events-none`), `:checked` (label-card pattern read via `:has()`), `:invalid`/`:required`/`:read-only`/`:placeholder-shown` (styling hooks only, NOT validation). `:has()` = first selector reading *downward into contents*; framed as **deleted state** (each reach retires a `useState`+handler+conditional-class); `has-[…]:` ≈ `&:has(…)`; `group-has-[…]:` named; chains past 2 conditions = flatten to a `data-*` attribute; does NOT pierce shadow DOM. `:not()` via `not-*:` form — headline reach is the **disabled-hover trap** (`not-disabled:hover:bg-accent`, since disabled buttons still fire `:hover` in some browsers). Pseudo-*elements*: `::placeholder` (`placeholder:text-muted-foreground` reflex — placeholder does NOT inherit `color`, the bug it fixes), `::selection` (optional brand polish), `::file-selector-button` (named only). Structural (`:first-child`/`:nth-child`/`:empty`, only `empty:hidden` earns its keep) + link (`:link`/`:visited` privacy-locked) + `:target` = recognition only, mostly retired by `gap`/`divide-*`. DevTools "Toggle Element State" (`:hov` button) as the senior reflex to pin a state for inspection.

**Cut:** the iOS sticky-hover gate / `@media (hover: hover)` — outline framing slated it for "the end" of this lesson, but it's reduced to one parenthetical aside (`:hover` "does nothing on a touchscreen … consequences we will come back to") and **deferred to ch021 L6** (where `@media (hover: hover)` is owned). A later lesson needing the touch-hover gate should point to L6, not here.

**Debts (deferred to later lessons):**
- Form-validation flow (Constraint Validation API, `onSubmit`, server validation) → Unit 6 / ch044. `:invalid`/`:required` here are styling hooks only; stated explicitly in-lesson.
- Shadow-DOM-internal styling of native widgets (e.g. `<select>` option list) → trusts the form library, Unit 6.
- iOS sticky-hover + `@media (hover: hover)` → ch021 L6 (see Cut).

- WCAG focus-visibility as discipline + ARIA depth → ch027.

**Recall hooks used (must stay consistent):** ch018 L4 variant grammar is the load-bearing prereq — desugaring `hover:` ≈ `&:hover`, the five-family "whose state?" map, `group-*`/`peer-*`/`data-*`/`aria-*` variants are **cross-referenced, never re-taught** (this lesson owns the pseudo-class *semantics*, L4 owns the *machinery*); `aria-disabled:`/`aria-invalid:` = the attribute-set companions to the property-set `:disabled`/`:invalid`; ch021 L3 owns focus-ring *geometry* (`ring-*`/`outline`/why rings don't reflow), this lesson owns *when* `:focus-visible` fires; `focus-visible:` discipline line carried verbatim-compatible from L3; `opacity-50` = disabled fade (ch021 L2); `text-muted-foreground`/`bg-accent`/`border-destructive`/`ring-ring` semantic tokens (ch018 L5, ch021 L2); Preflight strips default focus outline so `:focus-visible` must restore it (ch019 L3); `gap`/`divide-*` retired structural-sibling selectors (ch020, ch021 L3).

**Terminology / mental models:** pseudo-*class* = a **state** on a real element (`:hover`, `:checked`), one colon; pseudo-*element* = a **sub-part** the markup never created (`::placeholder`, `::selection`), two colons — Tailwind hides the colons but the distinction governs capability. "The DOM already knows; `:has()` reads it; the React state was always a mirror." `:focus-visible` = "the focus reflex." Two organizing axes: (1) a fact *on the element itself* vs. *an ancestor reacting to a descendant* (`:focus-within`/`:has()` are the upward ones, the ones that delete most code); (2) "name of a fact the browser tracks." Discipline line: **`focus-visible:` is the default; bare `focus:` is the bug.** Tooltip terms defined: "focus heuristic," "assistive technology," "Baseline," "shadow DOM," "pseudo-element." Canonical button string: `hover:bg-accent focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring/50 focus-visible:ring-offset-2 active:scale-[0.98]`; disabled+hover set: `not-disabled:hover:bg-accent disabled:opacity-50 disabled:pointer-events-none`.

**Browser-support facts stated (keep coherent):** `:has()` is **Baseline since December 2023** — universal, no polyfill, reach without checking caniuse. `:visited` is privacy-locked (only `color`/`background-color`/`border-color` + a few props changeable). Disabled controls don't reliably receive `:hover` in all browsers (rationale for `not-disabled:hover:`).

**Patterns / best practices (for project chapter):** `focus-visible:` (never bare `focus:`) on every interactive element; the canonical button string above is the reflex; `disabled:opacity-50 disabled:pointer-events-none` for every disablable control, paired with `not-disabled:hover:` so hover never paints on a dead control; `placeholder:text-muted-foreground` on every text input; reach for `:has()`/`:focus-within` instead of mirroring DOM state into `useState`+handler; flatten `:has()` chains past 2 conditions into a single `data-*` attribute read with `data-[…]:`; `empty:hidden` for empty-state containers; prefer `gap`/`divide-*` over structural pseudo-classes; semantic tokens in app code (exercises diverge to CDN-safe literals — do not "correct").

## Lesson 5 — Motion: transitions, keyframes, and tw-animate-css

**Taught:** motion in three escalating tiers, all on the spine *cheap-by-default / state-in-React-motion-in-CSS / read-don't-reimplement*. The **compositor budget** as the foundational rule (taught first, referred back throughout): the layout→paint→composite pipeline runs ~60×/s, so animate only `transform` + `opacity` (composite-only, GPU); `width`/`height`/`top`/`margin`=layout, `background-color`/`color`/`box-shadow`=paint, both suspect — translate-not-`top`, scale-not-`width`. **Tier 1 transitions** (`transition` watches a *curated* v4 set, NOT literal `all`; `transition-all` is the rare opt-in; reflex = name the property: `transition-colors`/`-transform`/`-opacity`/`-shadow`/`-none`), the **duration band** (`150` snappy state changes, `200`–`300` entrances/exits, `400ms+` long-form only — sluggish trap at the top), **easing** reflex (`ease-out` appearing / `ease-in` leaving / `ease-linear` spin-progress), `delay-*` for staggered reveals (named only). **Tier 2 animations** (run on their own, no trigger — the transition-vs-animation line): four built-ins mapped to surfaces (`animate-spin` loader / `animate-pulse` skeleton / `animate-ping` ripple / `animate-bounce` attention nudge); authoring a custom keyframe is the v4 CSS-first two-piece — `@keyframes <name>` + `@theme { --animate-<name>: <name> <dur> <ease>; }` (registering the token is what mints the utility), shown via a `shake` example. **Tier 3 choreography** — `tw-animate-css` (`@import` one line after Tailwind; provides `animate-in`/`animate-out` + `fade-*`/`zoom-*`/`slide-*` modifiers + the `accordion-down`/`-up`/`caret-blink` keyframes shadcn relies on; the maintained successor to deprecated `tailwindcss-animate`), driven by the **data-state seam**: Radix sets `data-state="open"|"closed"`, CSS variants (`data-[state=open]:animate-in fade-in-0 zoom-in-95` / `data-[state=closed]:animate-out fade-out-0 zoom-out-95`) read it; Radix holds the element mounted through `animate-out` so the exit plays (retires hand-rolled unmount-timing JS). The **accordion height exception** (`height: auto` can't be tweened; Radix measures real height into `--radix-accordion-content-height` and the shipped keyframes animate `height: 0`→that var). **Transform surface** (`translate`/`scale`/`rotate`/`skew`, all composite-cheap; canonical `hover:scale-105` lift, `active:scale-95`/`scale-[0.98]` press, `-translate-y-1` raise, `data-[state=open]:rotate-180` chevron); `transform-gpu` named as rare manual hint. **`prefers-reduced-motion`** as discipline (the `motion-reduce:` variant + the `motion-safe:` inverse): reduced ≠ no motion — strip *decorative* (parallax/big slides/bounce), keep *functional* (spinner = the information, focus rings); shadcn/Radix ship sane defaults, the discipline is your job on bespoke motion.

## Lesson 6 — Breakpoints and the mobile-first reflex

**Taught:** the **viewport half** of responsive design, on the spine **base = smallest screen; prefixes only *add* at wider widths, never subtract**. Reopens the ch020 L4 `grid-cols-1 md:grid-cols-2 lg:grid-cols-3` card-grid line as the debt being paid. **Mobile-first** as the default authoring direction, argued *mechanically* not morally (fewer overrides shipped/held in head; `min-width` queries stack cleanly in source order vs. `max-width` fighting the base; mobile is the larger traffic share); `max-md:` named as the rare legitimate desktop-first escape hatch. The **breakpoint scale** (exact, fact-checked Jun 2026): `sm` 40rem/640, `md` 48rem/768, `lg` 64rem/1024, `xl` 80rem/1280, `2xl` 96rem/1536 — five steps, `2xl` largest core (no `3xl`); "stay on the scale," grow/redefine it in `@theme` (`--breakpoint-md: 50rem`, add `--breakpoint-xs: 30rem`) not one-off `min-[823px]:`. Every prefix **compiles to `@media (min-width: <rem>)`** (desugaring shown on `md:grid-cols-2`); `max-*` flips to a `max-width` query (below the breakpoint), `md:max-lg:` targets a single band (768–1023) — both recognition-only, rare in production. **Breakpoints are content-driven, not device-driven** (the lesson's headline mental-model correction): a breakpoint is *the width at which this specific layout stops working*, found by dragging the browser narrower until it looks wrong; `md` 768 is a pragmatic default, never "the tablet line." Daily responsive utilities framed as **two tiers — switch the layout primitive at the breakpoint (`flex-col md:flex-row`, `grid-cols-1 md:grid-cols-2`, `block md:flex`), then scale the visual values inside (`text-base md:text-lg`, `p-4 md:p-8`, `gap-4 md:gap-8`)**; prefixes **stack with state variants** (`md:hover:bg-accent`). The **`prefers-*` family unified** as one idea ("OS/device sets a signal, CSS reads it; same `@media` machinery as `md:`, different axis"): `prefers-color-scheme`/`dark:` (re-fixes OS-vs-site precedence resolved by next-themes), `prefers-reduced-motion`/`motion-reduce:`, `prefers-contrast`/`contrast-more:`, `forced-colors:` — all recognition, reflexes already built in L2/L5. **Input-device queries (the substantive half, the L4 debt paid):** `@media (hover: hover)` — **Tailwind v4 already wraps `hover:` in it by default**, so the old iOS sticky-hover bug is *gone*, but the flip side bites — **hover is enhancement-only on touch; a hover-only affordance is invisible/unreachable on a phone** (reflex: always-visible base + hover as polish, e.g. `opacity-60 hover:opacity-100`, never `opacity-0 hover:opacity-100`); raw hand-written `:hover` does NOT get the gating free. `pointer: fine`/`coarse` (don't hide small targets behind coarse), `@media print`/`print:` (invoices/reports only), `orientation` — all recognition. **Visibility by breakpoint:** `hidden md:block` (desktop-only) / `md:hidden` (mobile-only) toggle `display`, node stays mounted in DOM (drops from a11y tree only where `display:none` is active); **vs. conditional render** (`isDesktop && …`) which unmounts entirely (gone from DOM + a11y tree, state torn down) — the ch020 L2 "is it even present?" decision keyed to breakpoints; a11y trap = shipping *both* a `md:hidden` and a `hidden md:block` nav double-announces navigation. Closes by **posing (not resolving) viewport-vs-container**: same `<ProductCard>` looks right in a wide feed, cramped in a narrow sidebar — both share one viewport, so `md:` can't disambiguate; the card needs *its own width*.

**Cut:** nothing material from the lesson-outline scope; the `Dropdowns` breakpoint-value micro-check was descoped (outline leaned against it); the `DiagramSequence` step-frame fallback for the simulator was not needed (bespoke iframe component shipped).

Most SaaS UIs use both — page shell viewport-driven, components inside container-driven.** Do not let `@container`/`container-type`/`cqi` leak backward into this lesson — L6 names+teases them exactly once.

## Lesson 7 — Container queries for component-level layout

**Taught:** the **container half** of responsive design — a component laying itself out from its *slot* not the viewport, paying off L6's `ContextlessCard` cliffhanger. The **two-piece contract**: parent opts in via `@container` (= `container-type: inline-size`, width-axis only, height stays content-driven, the 99% default), descendant queries the nearest container ancestor via `@`-prefixed variants (`@md:flex-row` = `@container (width >= 28rem) { … }`). **The `@` is the entire tell** — `md:` = screen, `@md:` = this box; same grammar, different yardstick. A query **always reads an ancestor, never the element it's written on** (both on one div = nothing fires). The **container breakpoint scale is smaller than the viewport scale** (fact-checked: `@3xs` 16rem · `@2xs` 18 · `@xs` 20 · `@sm` 24 · `@md` 28rem/448px · `@lg` 32 · `@xl` 36 · `@2xl` 42 · `@3xl` 48 · `@4xl` 56 · `@5xl` 64 · `@6xl` 72 · `@7xl` 80rem) — container `@md` 448px vs viewport `md` 768px is the anchor contrast; "stay on the scale, find the breakpoint by resizing until layout breaks" reflex carried from L6, now applied to the box. Escape hatches: `@min-[475px]:`/`@max-[960px]:` arbitrary one-offs; extend via `--container-*` in `@theme` (`--container-8xl: 96rem` mints `@8xl`); `@max-md:`/`@sm:@max-md:` ranged = recognition only. **Fluid component typography** via container query units — `1cqi` = 1% of container inline size (the reach for almost everything; `cqb`/`cqw`/`cqh`/`cqmin`/`cqmax` named once), fed to `clamp()` (the canonical `font-size: clamp(1rem, 6cqi, 1.5rem)` title that scales continuously with the slot, no breakpoint). **Named containers** for nesting: unnamed `@container` binds to *nearest* ancestor, so an inner container shadows an outer one; fix is `@container/panel` on the ancestor + `@md/panel:flex-row` on the descendant (name still implies `inline-size`) — narrow reach, pull out only when structure nests. The **viewport-vs-container decision rule resolved** (L6's deferred debt, verbatim): page-level structure → `md:`, component-level adaptation → `@md:`, most SaaS UIs use both; diagnostic question = "does this depend on the *screen* or the *space it sits in*?". **`auto-fit`/`minmax` (ch020 L4) vs container queries compose, not compete**: `auto-fit` makes the *container* respond to its *items* (track count flexes), `@container` makes the *items* respond to the *container* (each re-lays-out) — the 2026 card-grid reflex pairs them. Two footguns collapsed to one root cause (**container features need a declared container above them**): forgetting `@container` → `@md:` silently never matches; `cqi` with no container up-tree → resolves to **0px** (invisible). Baseline stated once (cross-browser since early 2023, widely available 2025, universal in 2026, no polyfill).
