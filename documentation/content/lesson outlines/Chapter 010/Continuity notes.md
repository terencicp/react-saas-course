# Chapter 010 — How a request becomes a page

## Lesson 1 — URL bar to first byte

- **Taught** — Four-stage map from URL commit to first byte: DNS (with DoH, Happy Eyeballs v2), Transport (HTTP/3 over QUIC, `Alt-Svc` discovery, fallback to HTTP/2+TLS), TLS 1.3 folded into QUIC (1-RTT fresh / 0-RTT resumed, forward secrecy, 0-RTT replay-safety), HTTP request + TTFB, and reading the DevTools Network panel (Protocol column + Timing breakdown).
- **Debts** — Promised later coverage: HTTP method/status/header contract (ch 011), origins/CORS (ch 012), cookies + `Secure`/`SameSite` (ch 013), deep TLS handshake with cipher suites/SNI/ALPN/cert chains and `mkcert` (lesson 4 of this chapter), browser-side rendering pipeline (lesson 2), other DevTools panels (lesson 3), TTFB tuning + Core Web Vitals (Unit 19), idempotency contract foreshadowed for ch 011.
- **Terminology** — `h3`/`h2`/`http/1.1` (Protocol column values), QUIC, DoH/DoT/DoQ, Happy Eyeballs v2, `Alt-Svc`, resumed connection, session ticket, early data, 0-RTT, TTFB, forward secrecy. Timing-panel labels: Queued/Stalled, DNS Lookup, Initial connection, SSL, Request sent, Waiting (TTFB), Content Download. Mental model: "a slow page is a *stage* problem — point at the stage, then debug the stage."
- **Patterns and best practices** — Senior reflex: name the dominant cost stage from a waterfall row before touching server logs. Always check whether a TTFB number is cold or warm. Only idempotent GETs are safe as 0-RTT early data.
- **Misc.** — No application code shipped; lab is opening DevTools on `cloudflare.com` or the course site. Uses Mermaid `sequenceDiagram` with `themeCSS` font-size bump for the four-stage diagram, `CodeVariants` for the 1-RTT/0-RTT/2-RTT comparison (ASCII bodies), hand-coded SVG inside `<Figure>` for the annotated Timing panel (Option B chosen over screenshot), `<Matching>` for the closing dominant-cost drill. ## Lesson 4 — HTTPS on localhost with mkcert

- **Taught** — Secure-context gate (`window.isSecureContext`) and the partial/asymmetric `http://localhost` carve-out (Secure cookies, LAN IPs, service workers); TLS 1.3 handshake in three phases (ClientHello with SNI/ALPN/key share, ServerHello+Certificate+Finished with signed transcript hash, Client Finished + first HTTP request — 1-RTT total); certificate chain (leaf → intermediate → root) and OS/browser trust store as the trust anchor; why self-signed certs fail; what `mkcert` does (one-machine-only local root CA installed into OS + Firefox trust stores, signs project leaf certs); end-to-end wiring of `mkcert -install`, `mkcert localhost 127.0.0.1 ::1`, `./certificates/` directory convention, `.gitignore`-ing `*-key.pem`, and `next dev --experimental-https` with `--experimental-https-key`/`--experimental-https-cert` flags; the `dev` (HTTP) + `dev:https` (opt-in) two-script pattern in `package.json`.
- **Debts** — Forward links: cookie `Secure`/`HttpOnly`/`SameSite` contract → ch 013; `crypto.subtle` HMAC sign/verify → ch 016 lesson 1; `navigator.clipboard.writeText` Copy button → ch 016 lesson 2; HSTS, certificate transparency, full production security baseline → ch 081. Production cert provisioning (Vercel + Let's Encrypt, auto-renew, developer never touches) named in one line. Service Workers and Push named once as the third localhost loophole, out of scope for the course's stack.
- **Terminology** — `secure context`, `SNI` (Server Name Indication), `ALPN` (Application-Layer Protocol Negotiation, lists `h3`/`h2`/`http/1.1`), `forward secrecy`, `Certificate Authority`, `cipher suite`, `cert chain`, `trust store`, `SAN` (Subject Alternative Name list), `local CA`, `leaf cert`, `rootCA-key.pem` (machine-wide secret under `mkcert -CAROOT`). Mental model: "is this secure?" is a function of *both* the URL and the API — never one question.
- **Patterns and best practices** — Senior reflex: flip every project to HTTPS on day one (5-min setup eliminates the "works on my machine, breaks in preview" bug class). Project convention: certs live in `./certificates/`, public `*.pem` committed (so a teammate with their own `mkcert -install` reuses it), `*-key.pem` gitignored per-developer. Always issue cert SANs for `localhost 127.0.0.1 ::1` together. `mkcert`'s root key is a per-machine security boundary — never copy or commit `rootCA-key.pem`. Keep both `dev` (plain HTTP for easy debugging) and `dev:https` scripts in `package.json`; point Next.js at `mkcert`-issued files explicitly rather than letting it auto-generate (team-share story is cleaner).
