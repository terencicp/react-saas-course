## Lesson 2 ŌĆö Bounding spend before the surface goes public

**Taught** ŌĆö Established the full LLM cost-discipline gauntlet: token attribution via `usage`/`onFinish`, two enforcement points (pre-call estimate-and-reject, post-call ledger write), per-user daily quota keyed by date in Redis, sliding-window rate limit for burst, `maxOutputTokens` per call, and the seven abuse shapes mapped to their primary structural defenses. Framed quota as a plan entitlement and cost as a product feature (visible counter, operator cost dashboard).

**Cut** ŌĆö The chapter outline included `durationMs` and `totalCostCents` in the `llm.call.completed` event shape; the lesson omits `durationMs` and introduces `costCents` only as a write-time computation in `lib/llm/pricing.ts`, not in the event object literal shown. The chapter outline proposed a Mermaid request-lifecycle diagram; the lesson delivered a `DiagramSequence` interactive scrubbable instead. The outline referred to `writeAuditLog`; the lesson uses `logAudit` (ch057 name).

**Debts** ŌĆö All generation primitives (`streamText`, `generateText`, `maxOutputTokens` actual syntax) deferred to Chapter 106, marked with `// model call ŌĆö Chapter 106` elision comments throughout. `stopWhen(stepCountIs(n))` for agentic loop guard named but deferred to Chapter 107. AI Gateway failover (provider 429 mitigation) forward-pointed to Lesson 3. Worked invoice-chat quota wiring deferred to Chapter 108.

**Terminology** ŌĆö `usage` object shape: `{ inputTokens, outputTokens, totalTokens }` (v5; v4 used `promptTokens`/`completionTokens` ŌĆö named for stale-tutorial recognition). `totalUsage` exists for multi-step calls (aggregates across all steps; `usage` is last-step only). `onFinish` callback: where the post-call ledger write lives; does **not** fire on aborted streams. `withLlmQuota(...)`: the thin composition wrapper that makes the quota gate structural (wraps the LLM route inside `authedRoute`). Quota key shape: `quota:llm:${userId}:${todayUtc()}` (date in key = auto reset + free history). Rate-limit key shape: `rl:llm` as the `prefix` on the module-scope `Ratelimit` instance; per-user Redis key passed as the runtime `key` arg to `safeLimit`. `safeLimit(limiter, prefix, key)`: canonical Chapter 074 burst-limiter call signature ŌĆö `limiter` is a module-scope `new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(...), prefix: 'rl:llm', analytics: true, ephemeralCache: new Map() })` instance; `prefix` and `key` are the per-call runtime args. `lib/llm/pricing.ts`: price table `Record<modelId, { inputPerM, outputPerM }>` + `costCents(model, usage)` pure function; cost computed at write time so dashboard sums cents, not tokens. `llm.call.completed`: the audit event type written per call, carrying `{ userId, orgId, surface, model, inputTokens, outputTokens }`.

**Patterns and best practices** ŌĆö Every LLM call emits a `logAudit` event tagged with `userId`, `orgId`, `surface`, and `model` ŌĆö without per-user attribution the spend spike is undiagnosable. Two enforcement points always run together: pre-call estimate (cost filter, no ledger) and post-call `onFinish` write (ledger, no ceiling) ŌĆö each catches what the other cannot; `onFinish` alone is insufficient because aborted streams produce no `onFinish` call. Rate limit (burst) and daily quota (sustained) use different Redis keys and catch different attack shapes; both ship. `maxOutputTokens` is never `undefined`; a missing cap is the same severity as a missing auth check and must be audited at every call site. Quota number sourced from `getEntitlement(orgId)`, not a constant ŌĆö cost ceiling and pricing lever are the same number. `withLlmQuota` sits structurally inside the route wrapper so quota enforcement cannot be forgotten on a new route. Log a hash + metadata for prompts, never the raw prompt (sub-processor / GDPR posture from the errors-and-security unit, Unit 16). Compute `costCents` at write time in the audit event; dashboard queries sum a pre-priced column.

**Misc.** ŌĆö The `authedRoute` wrapper (ch057) does **not** yet do quota checks; the lesson is explicit that quota is a new composition layer (`withLlmQuota`), not a claim about the existing wrapper. The chapter outline framed quota check as "inside `authedRoute`"; the lesson corrects this to composition around it ŌĆö downstream lesson 3 and Chapter 108 should follow the `withLlmQuota` composition convention. The lesson is entirely decisions/placement ŌĆö no runnable route is built; all code blocks are intentional skeletons with `// model call ŌĆö Chapter 106` elision markers.

## Lesson 3 ŌĆö One-line model swaps and the AI Gateway

**Taught** ŌĆö Established the full provider-abstraction discipline: `lib/llm/models.ts` as the single file owning all model handles (role-named, `camelCase`, plain gateway strings), the AI SDK v5 global-provider default (plain `'creator/model'` string routes through the AI Gateway with no provider import), the four gateway production features (failover, observability, unified billing, BYOK), the three triggers that flip the gateway from bare-default to configured, the embedding-portability trap, and the `generateObject`-vs-`streamText` swap-portability heuristic.

AI Gateway setup steps (dashboard, project linking, key provisioning) not taught ŌĆö named as production default; wiring is project-time work.

**Terminology** ŌĆö `lib/llm/models.ts`: the canonical artifact first committed by this lesson; exports one `export const` per role handle; `import 'server-only'` at top; **`camelCase` not `SCREAMING_SNAKE_CASE`** (model handles are the explicit camelCase carve-out in conventions). Handle = a named reference you import instead of inlining the value. `'creator/model'` slash form is the v5 default gateway string; concrete examples shipped in the lesson: `fastModel = 'openai/gpt-5-mini'`, `smartModel = 'anthropic/claude-sonnet-4.5'`, `embeddingModel = 'openai/text-embedding-3-large'`. AI Gateway = a single managed endpoint routing model calls to any provider, adding failover, usage tracking, and one billing relationship; the AI SDK's default target in v5. Global provider = SDK-level default that turns a plain model string into a routed call. BYOK = bring your own key. Embedding / vector space defined (embedding is numbers assigned to text; coordinates only meaningful within the exact model that produced them; different models use incompatible geometries). Failover config shape: `providerOptions: { gateway: { models: [...fallbacks] } }` alongside `model`. Env keys: `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `AI_GATEWAY_API_KEY` (validated in `env.ts` `server` block, `z.string().min(1)`).

**Patterns and best practices** ŌĆö `lib/llm/models.ts` exports `fastModel`, `smartModel`, `embeddingModel` (and any surface-specific handles) as plain gateway strings; every call site imports a handle, never inlines a string. Handles named by role (capability), never by vendor ŌĆö vendor-named identifiers (`gpt5ForChat`) defeat the central-file discipline by leaking the vendor into the import graph. The gateway is already on by default (bare string-through-gateway = prototype); configure it (fallback list + read the dashboard) when any one of three triggers fires. Prefer `generateObject` + Zod schema over `streamText` + prompt engineering for structured workloads ŌĆö the schema absorbs provider shape differences, making swaps cleaner. Treat `embeddingModel` as a one-way commitment: swapping it requires re-embedding the entire corpus (re-indexing project, not a config change). Provider keys auto-read from `process.env` by the SDK ŌĆö never passed at the call site; never read from a DB row, query param, or request body; missing key must fail `pnpm build` via env validator. Gateway failover preferred over per-route catch-and-retry (the try/catch is duplicate code the gateway deletes). Fallback handles (`smartFallbackA` etc.) are also role-named entries in `models.ts`.
