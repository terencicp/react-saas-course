## Lesson 2 — Layouts and route groups

**Taught:** `layout.tsx` = persistent shared shell (default export, takes framework-injected `children: ReactNode`, Server Component by default); only the root `src/app/layout.tsx` owns `<html>`/`<body>`, nested layouts return fragments/wrappers; layouts compose down the tree like nesting dolls (`<RootLayout><DashboardLayout><DashboardPage/></DashboardLayout></RootLayout>`); the layout/page render boundary — layouts stay mounted across in-subtree navigations, only the page swaps, so Client-Component state in a layout survives and page state resets; fetch at the level that owns the data; `template.tsx` = same shape but remounts per navigation (state resets, effects re-fire) for narrow triggers; route groups `(folder)` are fully routable but contribute zero URL segment; the `(auth)`/`(app)` two-group/two-layout split; one-root vs multiple-root layouts; same-URL collisions across groups crash the build.

**Cut:** the prerequisite React-only `<Shell>`-imported-per-page alternative is named/contrasted (in prose + RenderTracking `Implementation` toggle) but not coded out; multiple-root layouts named for recognition only, not built.

**Debts:** Server/Client model + `'use client'` (Ch 030); `loading.tsx`/Suspense at the layout boundary and the layout-vs-template Suspense-fallback difference (Ch 031); `metadata`/`<head>`/fonts in the root layout (Ch 034); dynamic segments for the example routes kept static (L3); navigation described in prose only, no `<Link>` (L4); parallel routes `@slot` flagged as a *different model* from nested layouts (L5); the `(auth)`/`(app)` structure carried into Unit 8 auth gate and the Ch 035 project.

**Terminology:** "the framework fills `children`, you receive them not pass them"; "nesting dolls" / "compose down the tree"; "persistent UI lives in the layout; transient per-page UI lives in the page"; "fetch at the level that owns the data"; "remount" (Term: old instance discarded, fresh one mounts); route group `(folder)` = "a route, minus its own segment" vs `_folder` = "not a route"; "one root, many shells".

**Patterns and best practices:** root layout body kept minimal-but-real — `<html lang="en">`, an inline skip-link anchor `<a href="#main-content" className="sr-only focus:not-sr-only">` paired with `<main id="main-content" tabIndex={-1}>{children}</main>` (a11y baseline asks every layout to carry one), no providers/metadata; props typed at the parameter `({ children }: { children: ReactNode })`, never `JSX.Element`/`ReactElement`; layout-specific code co-located in `(app)/_components/sidebar.tsx` (Principle #1), promoted to top-level `components/` only when a second group needs it; layouts shown as plain Server Components with no data fetch/auth/metadata (staged simplification, real machinery added in Ch 030/031 + Unit 8); `layout.tsx`/`template.tsx` are sanctioned default exports, the sidebar/header are named exports in kebab-case files.

**Misc.:** through-line artifact is the `(auth)`/`(app)` file tree; skip link is written inline (anchor + `<main id="main-content">` landmark), not an imported `SkipLink` component — the Ch 027 a11y baseline is honored by hand here, not via a shared component; `template.tsx` renders *between* layout and page and can coexist with `layout.tsx` in one segment; the senior default reach is `layout.tsx` + a route group, conditionals only on a named trigger.

## Lesson 3 — Dynamic and catch-all segments

**Taught:** `[param]` dynamic segment — bracketed folder text becomes the `params` property name (`[id]`→`params.id`), captured value is **always a string**; in Next.js 16 `params` is a **Promise** (Server Component `await`s it, Client Component reads it with React 19 `use(params)` since it can't be `async`); type route props with the framework-generated global `PageProps<'/route'>` helper (no import, autocompletes param names, can't drift on rename) — hand-written `Promise<{ id: string }>` shown once as what it expands to; nested brackets for multi-axis/tenant scoping (`/orgs/[orgSlug]/invoices/[id]` → `{ orgSlug, id }`, one `await`); `[...slug]` catch-all (`string[]`, does **not** match the bare parent → 404) and `[[...slug]]` optional catch-all (`string[] | undefined`, matches parent); the bracket-shape decision order (known-vs-variable depth first, then parent-included); the **capture → validate → query** reflex — URL is untrusted input, `paramsSchema.safeParse(await params)` with `z.uuid()`, `notFound()` (from `next/navigation`) on miss; `generateStaticParams` named as forward-reference only.

**Cut:** `useParams()` hook named in one clause but not demonstrated (prop-threading `use(params)` is the shown path); numeric-id `z.coerce.number().int().positive()` shown only as a one-line variant since the course standardizes on UUIDv7.

**Debts:** why-Promise / Cache Components / PPR rendering model (Ch 032); full `notFound()` + `not-found.tsx` + error boundaries (Ch 031); the `params`-for-identity vs `searchParams`-for-view split (Ch 033); rest of async-request APIs `cookies()`/`headers()`/`draftMode()` listed as also-Promises (Ch 032.7); navigating *to* these routes with `<Link>`/`useRouter` (L4); `generateStaticParams` at depth + `dynamicParams` (Ch 034.8); Zod 4 depth (Ch 042); `getInvoice`/data layer (Unit 5); the `assertValidLocale` type-guard alternative for enumerable params named once.

**Terminology:** "a bracketed folder is a typed hole in the URL"; "the framework fills `params` like it fills `children` — but the contents are attacker-typed text"; "capture → validate → query"; "`params` is a Promise, await it"; "reading `params.id` without `await` is the dynamic-routes equivalent of forgetting a key on a list"; "the URL bar is an input field"; `use(params)` = "the client-side counterpart to `await`"; one-line decision summary: known single → `[id]`, known multi-axis → nested, variable no parent → `[...slug]`, variable with parent → `[[...slug]]`, enumerable → literal folders.

**Patterns and best practices:** type page props with `PageProps<'/route'>` (default; hand-written annotation is the anti-pattern that rots on rename); `safeParse` (never `parse`) for untrusted URL input, validate **before** the query never after; bail with `notFound()` (404), never let a garbage id throw a 500 and never `try/catch` the throwing `notFound()`; never `Number(params.id)` (silent `NaN`) — let the schema coerce; param schema named `<entity>ParamsSchema` / inline `z.object({ id: z.uuid() })`; query helpers verb-led (`getInvoice`); page is sanctioned `export default async function`; each dynamic segment name unique along a route, one dynamic shape per level (sibling `[id]`+`[...slug]` or duplicate `[id]` crash the build); catch-all value is always an array even for one segment (`['intro']` not `'intro'`).

**Misc.:** through-line domain is invoices in a multi-tenant SaaS; catch-all examples deliberately switch to a docs site (signposted) because depth is unknown there; tenant-scoped URL shape `/orgs/[orgSlug]/invoices/[id]` is the multi-tenant pattern carried through Units 8+; the canonical page is built in three intentionally-incomplete passes (capture-only → +await/typing → +validation) — only the validation-section version is shippable; no live in-browser React exercise for the page because the iframe renders React but not the App Router (route-matching `Dropdowns`/`Matching` drills test the real skill instead).

## Lesson 4 — Navigation primitives

**Taught:** four navigation tools picked by one filter — *who triggers it, and where does the code run?*: `<Link>` from `next/link` (renders a real `<a>`, intercepts same-origin left-clicks into a **soft navigation** — client-side page swap, no reload, surviving-layout client state preserved; in-app only, bare `<a>` for external/`mailto:`; dynamic hrefs via template literal `` `/invoices/${id}` `` with `key` on the row; `replace` swaps history entry, `scroll` defaults `true` = keep-position-then-top-if-out-of-view, `scroll={false}` opts out); the prefetch model — default (`prefetch` unset / `"auto"` / `null`) fires on **viewport entry** (not hover), full route+data for static destinations, partial route to nearest `loading.tsx` for dynamic, hover only re-fetches stale; three prop values (`"auto"`/`true`/`false`), **production-only** (dev never prefetches); `useRouter` from `next/navigation` (Client Component / `'use client'` only; `push`/`replace`/`back`/`forward`/`refresh`; same soft nav as `<Link>`, code-triggered, for event handlers); the **throwing trio** `redirect()` (307; 303 inside a Server Action), `permanentRedirect()` (308; 303 in Server Action), `notFound()` (404) — all from `next/navigation`, run **during render** (not event handlers), return type `never`, throw a framework-caught signal so nothing after them runs and TS narrows below the call.

**Cut:** chapter-outline `Buckets` exercise not used; `<meta refresh>` streaming-redirect status case dropped (only 307/308/303 taught).

**Debts:** `loading.tsx` boundary that prefetch + the trio reference (Ch 031); `not-found.tsx` companion for `notFound()` (Ch 031); Cache Components / static-vs-dynamic rendering behind prefetch (Ch 032, named only); `router.refresh()` as the post-mutation revalidation tool + `revalidatePath`/`revalidateTag` (Ch 032.6); real `getSession()` auth gate (Unit 8, stubbed `// TODO(Unit 8)`); real `createInvoice` Server Action + post-action `redirect()` (Ch 043, stubbed `// TODO(Ch 043)`); the rest of `next/navigation` hooks `usePathname`/`useSearchParams`/`useParams` + active-link pattern (Ch 033.5); `searchParams`/URL-as-state (Ch 033.4); `proxy.ts` / `next.config.ts` redirect maps + redirect-vs-rewrite (Ch 033.2–3 / 034); View Transitions / `onNavigate` `<Link>` props (deferred, animations Ch 021.5). Closes the L3 loop: `notFound()` doesn't need a `return` because it throws (`never`).

**Terminology:** "who triggers the navigation, and where does the code run?" (the unifying filter); **soft navigation** (Term) vs hard navigation (full document reload); `<Link>` = "`<a>` plus a contract"; "`<Link>` for in-app, `<a>` for external"; **imperative** (`useRouter().push`, you call it) vs declarative (`<Link>`, you place it); the **throwing trio** / "they don't return, they throw"; "trust the default" (prefetch); `never` (Term, function that can't return normally); HTTP status Terms 307/308/303/404; "navigate from a handler → `useRouter`; stop and reroute during render → the trio".

**Patterns and best practices:** prefer `<Link>` whenever navigation maps to a clickable destination (free accessible anchor), `useRouter().push` only when there's no anchor; trust the prefetch default — `prefetch={true}` is not the "make it fast" button and can hammer the backend; **never `try/catch` the throwing trio** (a broad catch swallows the signal — the #1 bug; do cleanup *before* the call, never assign their "result"); `redirect()` is the 307 default, `permanentRedirect()` the rare deliberate 308 for genuine URL migrations + SEO only; auth-gate shape `const session = await getSession(); if (!session) redirect('/sign-in');`; import `useRouter` from `next/navigation` **not** `next/router` (Pages Router, named once as the wrong import).

**Misc.:** Server Action (`createInvoice`) and `getSession()` are intentional stubs marked with `// TODO(Ch 043)` / `// TODO(Unit 8)` — do not flesh out; legacy nested-`<a>`-inside-`<Link>` form named as gone since Next 13; no live in-browser navigation exercise (App Router doesn't run in the React iframe, same as L3) — drills are `StateMachineWalker` decision + `Dropdowns` + `MultipleChoice` recognition only; built figures are `PrefetchTimeline` (scrubbable 3-moment prefetch sequence) and `PrefetchValuesTable` (the three `prefetch` values), plus `CodeVariants` (soft-vs-hard `<Link>`, try/catch-vs-throw `notFound`), `AnnotatedCode` (create-invoice `useRouter` button), `CodeTooltips` (`never` return type); closing line forward-points to L5 (parallel routes — one screen rendering two routes at once).

## Lesson 5 — Parallel routes and slots

**Taught:** parallel routes — a `@x` folder gives the same-segment `layout.tsx` a framework-filled named prop `x` beside `children` (prop name = folder name minus `@`), typed `React.ReactNode`; `children` reframed as the layout's *unnamed* slot (the segment's own `page.tsx`); core rule "add a `@x` folder, get an `x` prop"; each slot is a **full independent route tree** (own `page.tsx`, `[id]`, `loading.tsx`, `error.tsx`) — the router matches the URL against every slot independently and hands all matches to the layout at once; the canonical list-plus-detail surface (`@detail` slot beside the list `page.tsx`) where navigating `/invoices`→`/invoices/42` re-resolves only `detail` and leaves `children`'s match untouched so the list stays mounted (DOM/scroll/client state preserved); the detail slot's `[id]/page.tsx` is an ordinary dynamic page (`PageProps<'/invoices/[id]'>`, `await params`, capture→validate→query); **`default.tsx`** = a slot's unmatched-case fallback on **hard navigation** — soft nav keeps prior slot matches, hard nav (direct visit / refresh / Cmd+click) resolves every slot from the URL alone and 404s the **whole route** if any slot lacks a match, so every slot ships a `default.tsx` (`return null` or a neutral placeholder); slots stream on their own clock (each is its own loading boundary) — named only, not built; the three "no-URL-segment" folder conventions disambiguated (`_folder`=not a route / `(folder)`=route minus its segment, normal `children` / `@folder`=route as a named prop); the reach-for filter — parallel routes only when 2+ regions each need own URL/loading/error behavior reflected in one shareable URL, else nested layouts (L2) or one page suffices.

**Cut:** chapter-outline's independent-loading/error-per-region dashboard case named only in prose, not built; no `<Screenshot>` of the target UI (prose carries it).

**Debts:** Suspense / streaming / `loading.tsx` boundary placement + `error.tsx` `reset` (Ch 031, hard boundary — slots named as independent loading boundaries only); `searchParams` / URL-as-view-state for the list filter (Ch 033.4, named "same idea, other slot"); intercepting routes + the `@modal` slot modal-with-real-URL pattern (L6, forward-pointed in the decision funnel and closing); the full wired list-plus-detail project with real data/Server Actions (Ch 035); Cache Components/PPR render model behind "streams in" (Ch 032). Reuses L2 (layout fills `children`, stays mounted), L3 (`[id]`/`await params`/`PageProps`/capture→validate→query), L4 (soft vs hard navigation Terms, applied at depth here).

**Terminology:** "a `@slot` is another `children`, but named"; "add a `@x` folder, get an `x` prop"; **slot** (Term: named layout region the framework fills from a matching `@`-folder, beside `children`); "the framework decides *what's inside* each slot; you decide *where it goes*"; "each slot is its own route tree"; "each slot reads its own slice of the URL"; **soft navigation** / **hard navigation** re-Termed here (soft keeps prior slot matches, hard resolves all fresh); "`default.tsx` is the difference between a shareable URL and a 404"; "every parallel slot ships a `default.tsx` — muscle memory like reaching for a `key`"; folder-convention one-liners `_folder`="not a route" / `(folder)`="a route minus its segment" / `@folder`="a route that arrives as a named prop"; "parallel routes are the *escalation* from nested layouts, not a replacement".

**Patterns and best practices:** `src/app/…` paths; `page.tsx`/`layout.tsx`/`default.tsx` are sanctioned default exports (all on the framework default-export list), every other component a named kebab-case export; type page/layout props at the parameter with generated `PageProps<'/route'>`/`LayoutProps<'/route'>` as the senior default — the hand-written `{ children, detail }: { children: React.ReactNode; detail: React.ReactNode }` in the first layout is a **deliberate teaching simplification** to make the slot→prop link visible, flagged in prose then `LayoutProps` named as the production form; `React.ReactNode` for slot/children types (never `JSX.Element`/`ReactElement`); detail slot is `export default async function` awaiting `params`; minimal real Tailwind v4 grid for the layout; durable rule stated broader than the floor — a `default.tsx` is needed wherever any slot **including the implicit `children`** can go unmatched on hard nav (don't claim `children` is exempt), with "every named slot ships one" as the practical floor.

**Misc.:** through-line stays invoices in a multi-tenant SaaS (consistent L3/L4); `<InvoiceCard id>`, `<InvoiceList>`, the "Select an invoice" placeholder are intentional thin placeholders — never implemented here, data layer is Ch 035; no live App-Router exercise (React iframe runs React not the App Router, same constraint as L3/L4) — checks are recognition `MultipleChoice` (which folder becomes a prop) + ordering `Sequence` (hard-nav resolution order on refreshing `/invoices/42`); slot is scoped to **one** layout (the same-folder one), never visible to parent/child layouts; closing forward-points to L6 (same slot machinery + one intercepting folder = modal with a real URL). **Shipped figures:** the two `DiagramSequence`-class visuals became lesson components `lessons/029/5/url-to-slot-mapping.astro` (the `/invoices`→`/invoices/42` two-URL walk) and `lessons/029/5/default-mechanics.astro` (the three-case soft/hard/`default.tsx` mechanics); plus `AnnotatedCode` (the two-slot layout), `CodeVariants` (`default.tsx` return-null vs neutral-placeholder), `StateMachineWalker` reach-for funnel, and a `VideoCallout` (Web Dev Simplified `wi8kF8UniUI`, 16-min build-along). Decision funnel's "no shareable URL" leaf shipped as "Nested layouts, or one page with parallel reads" (dropped the outline's `Promise.all` phrasing).

## Lesson 6 — Intercepting routes and URL-backed modals

**Taught:** the four intercepting-route prefixes (`(.)folder`=same URL level, `(..)folder`=one level up, `(..)(..)folder`=two up, `(...)folder`=from `app` root) and the load-bearing rule that the count is **URL-segment-relative, not file-system-relative** — `@slot` and `(group)` folders add zero URL segments so they drop out of the count; the always-paired non-intercepting sibling mapped one-to-one onto soft vs hard navigation (intercepter = soft-nav render override, real standalone page = hard-nav ground truth, forgetting it = 404-on-refresh, the #1 bug); URL masking on soft nav (address bar shows the real route's URL so the link is shareable while the feed stays mounted); the combined `@modal`-slot + interception pattern — intercepter inside `@modal/(..)photo/[id]/page.tsx`, empty `@modal/default.tsx` (`return null`), shared `<PhotoDetail>` rendered two ways (wrapped in `<Modal>` vs page chrome), and **closing the modal = `router.back()` navigation, not a state toggle** (so back/forward/refresh/share work for free); the senior three-way decision (plain `useState`+`<Dialog>` for ephemeral/unshareable → intercepting route for a list item's detail/edit view that deserves a URL → `searchParams`-keyed dialog as the lighter cousin).

**Cut:** nothing significant beyond the optional `VideoCallout` (left optional, resourcer-gated).

**Debts:** `loading.tsx`/`error.tsx` at the slot boundary so the modal can stream/recover (Ch 031); `searchParams`-as-state and the `?modal=invite` dialog alternative (Ch 033, named once not built); the real data layer — `getPhoto`, queries — behind `<PhotoDetail>`'s placeholder fetch (Ch 035 / Unit 5); the fully-wired list-plus-detail project with real data + Server Actions (Ch 035). Reuses L4 (soft vs hard navigation, `useRouter` from `next/navigation` not `next/router`), L5 (`@slot`→named layout prop, `default.tsx` hard-nav fallback), L3 (`[id]`/`await params`/`PageProps<'/route'>`/capture→validate→query with `safeParse`+`notFound()`), L2 (route groups add no URL segment), Ch 027 (shadcn `<Dialog>` owns focus trap / Esc-to-close / return-focus, cited not re-taught).

**Terminology:** **deep link** (Term: a URL that reproduces a specific in-app view); "an intercepting route is the soft-navigation render path; the hard-navigation path always renders the real, standalone page"; "same URL, two outcomes, decided by navigation kind"; "URL-masked" (soft nav shows the real route's address); "no URL segment, no count" (the prefix-arithmetic rule); "closing the modal *is* navigation" / "opening is navigation, closing is navigation"; the durable reflex **"when a piece of UI deserves a URL, give it one"** / "URL-backed UI for anything worth sharing or deep-linking"; "`searchParams`-keyed dialog" = the lighter cousin (`?modal=invite`).

**Patterns and best practices:** canonical geometry — real route at top-level `app/photo/[id]/page.tsx`, intercepter at `app/feed/@modal/(..)photo/[id]/page.tsx`, prefix is **`(..)photo`** (one URL segment up; `@modal` is two folders up on disk but invisible to the count) — keep byte-for-byte if reused; `<Modal>` is a Client Component (`'use client'`, uses `useRouter` hook + `onOpenChange` handler) built on shadcn `<Dialog defaultOpen onOpenChange={() => router.back()}>`, never imports `<PhotoDetail>` — composes the Server Component via `{children}` (Client composes Server through children, never direct import); `<PhotoDetail>` is a Server Component (default); intercepter + real page both `export default async function` awaiting `params`, typed with generated `PageProps<'/route'>` (don't hand-write the Promise annotation; the intercepter's route literal is generated by `next dev`, consumed via autocomplete not hand-typed); `default.tsx` returns `null` as the closed-modal floor (same `default.tsx` discipline as L5); framework files `page.tsx`/`layout.tsx`/`default.tsx` are sanctioned default exports, `<Modal>`/`<PhotoDetail>` are named exports in kebab-case files; `safeParse` not `parse`, `notFound()` on miss, never `try/catch` the throwing `notFound()`.

**Misc.:** through-line switches to a **photo feed** (`/feed` → `/photo/[id]`) as the primary code/diagram vehicle because the lightbox-with-a-URL is the instantly-recognizable case; SaaS reach (invoice-row→invoice-detail modal at `/invoices/42`, message thread, in-context edit) is named in prose only, not built; `<PhotoDetail>`'s fetch is a flagged thin placeholder (data layer = Ch 035); no live App-Router exercise (React iframe runs React not the App Router, same constraint as L3–L5) — checks are recognition `MultipleChoice` (which file renders on soft vs hard nav) / `Matching` (prefix→meaning) / `MultipleChoice` (route-group prefix-arithmetic gotcha) plus a `StateMachineWalker` decision aid; do NOT use `RequestTrace` for the dual-path model (it models render/stream, not navigation branching). **Shipped figures:** two lesson components — `lessons/029/6/dual-render-path/BrowserMock.astro` (a soft/hard `variant`-switched browser-window mock, rendered as the two `TabbedContent` tabs that carry the dual-render-path diagram) and `lessons/029/6/prefix-arithmetic/PrefixArithmetic.astro` (the two-strip file-system-vs-URL `@modal`-drops-out arithmetic, in a `<Figure>`); plus `CodeVariants` (intercepter-vs-real-page, same `<PhotoDetail>` two wrappers), `AnnotatedCode` (the `<Modal>` client wrapper), a `StateMachineWalker` three-way decision aid, `MultipleChoice`×3 (soft-nav render, hard-nav render, route-group prefix gotcha), `Matching` (prefix→meaning), `VideoCallout` (Dave Gray `Ft2qs7tOW1k`, 25-min end-to-end build), and three `ExternalResource` cards (Next.js Intercepting + Parallel Routes docs, Vercel NextGram example).
