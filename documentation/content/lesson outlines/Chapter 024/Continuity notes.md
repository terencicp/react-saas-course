## Lesson 2 — Derive in render, do not mirror into state

**Taught.** Derived values (full name, filtered list, counts, totals, flags, selected-from-id) are computed in the render body as plain JavaScript, never stored; the canonical anti-pattern — a `useState` mirrored from other state/props and resynced by a `useEffect` setter — costs a double render + stale frame + a dependency-array bug surface, and its cures are derive / `key`-reset / lift, never a sync effect. Closes L1's frozen-prop debt (derive branch). Names the four legitimate state triggers and a four-question derive-first filter.

**Cut.** Chapter-outline watch-outs "derived state inside a custom hook is still derived state" and "a computed value that is an effect *input* belongs in render + the effect deps, don't cache it to stabilize" were dropped (custom hooks and effect deps are not yet taught — Ch 025/026 territory).

**Debts.** `useEffect` signature/lifecycle/cleanup/deps → Ch 025 L2 (here a one-clause primer only: runs after commit, re-runs on dep change). Full "you might not need an effect" catalog (handler logic, fetching, effect chains, parent notify) → Ch 025 L4. Lift → L3 (named only). `useMemo` thresholds/decision rules → Ch 026 L3 (named once as audited escape valve). React Compiler config/badge → Ch 026 L2. Server-state cache / Server Components as real home for cached external data → Ch 032 / Unit 11. Form drafts / controlled inputs → Unit 6. `key`-reset mechanic owned by Ch 023 L5 (cited as known, not re-taught).

**Terminology.** ((derived state)) re-asserted: a value computed from props/state during render rather than stored. The smell stated abstractly: "a `useState` holding a value + a `useEffect` watching other state/props calling that state's setter." Slogans: "JavaScript before JSX"; "state is the minimum set of values from which everything else can be computed"; "reach for `useState` last, not first." Heuristic question: "Could I reconstruct this value from other state right now?" Four legitimate state triggers: user input / cached external system / captured moment in time / intentionally divergent draft. ((memoization)), ((server state)), ((React Compiler)) defined inline as one-line recognition terms. Named-once narrow exception: "adjusting state during render" (guarded `if (prop !== prev) { setPrev(prop); setX(...) }`) — last resort, prefer `key`/derive.

**Patterns.** Derived values are plain `const`s in the body; booleans as predicates (`hasErrors`, `isEmpty`, `canSubmit`). No prophylactic `useMemo`/`useCallback` — the compiler memoizes; any manual `useMemo` must carry a `// measured at …ms over …` reason comment. Measure-don't-guess discipline: wrap with `console.time`/`timeEnd`, ~1ms+ is the candidate line, React's rule of thumb is "unless looping over thousands of objects it's probably not expensive." Prop-changing-state routes to derive → `key`-reset → lift in that order.

**Misc.** Reuses the L1/L5 mark convention (orange = mirror-into-state smell, green = derived fix); all sync-effect snippets are deliberate anti-patterns shipped to be rejected — do not "fix" inline. The derive-first decision Mermaid `flowchart LR` reuses L1's four-homes leaf-label-with-owning-lesson convention (derive=this lesson, `key`-reset=render-model ch, lift=next lesson, `useState`=previous lesson) — keep consistent in L3. ## Lesson 3 — The four homes for state

**Taught.** Placement (where a value lives), not API. Four homes — local / lifted / URL / server — each motivated by consequence on the running `SearchableTable` (`query`). Colocation = state at the narrowest component above all its readers (often the common ancestor, not the leaf). Lifting mechanic taught hands-on: one `useState` at common parent, value down as prop + typed `onQueryChange` callback up, children become controlled. Three lift triggers (siblings share / ancestor reacts to child / value outlives a child's unmount). URL home recognition-only via `nuqs`. Server home recognition-only (stale-second-tab failure). The ordered top-down audit tree (server → URL → lift → local) and the two misplacement smells (under-lifted = duplicated state + sync effect; over-lifted = state above its only reader). Closes the four-homes thread's lift/URL/server branches from L1.

**Cut.** Nothing material from the chapter-outline scope dropped; URL/server/forms/context/compound-components all deliberately recognition-only per outline (each owned by a later chapter).

**Debts.** `useSearchParams`/`nuqs` full API, parsers, batching, history modes → Ch 033 L4-5 and Ch 060. Server-state cache (TanStack Query) → Unit 15 (Ch 076). Forms at depth (validation, submit, server actions) → Unit 6 (form-draft lift is recognition only here). `useReducer` / "12 useStates" → L4 (not preempted). Compound components → Ch 022 L2 (cited as already taught, one line).

**Terminology.** ((source of truth)) introduced — the one authoritative location read/written; placement bugs = two sources of truth drifting. ((colocation)) — narrowest component above all readers ("as low as possible while still high enough", not always the leaf). Slogans: "colocate first, relocate on evidence" (verbatim from L1); "lifting too eagerly is as wrong as lifting too late." Build-vs-audit reconciliation: you *build* from the leaf outward (colocate first) but *audit* from the global tiers inward (server/URL first) — same reflex, two directions. The two-question URL test: survives refresh? / shareable link? — either yes → URL. "The URL is global state with a free synchronization mechanism." ((nuqs)), ((searchParams)) defined as recognition Terms; `useQueryState('q', parseAsString.withDefault(''))` shown as the call-site shape (deliberately mirrors `useState`). **Patterns.** Children receive a typed callback `onQueryChange: (next: string) => void`, never the raw `setQuery` setter (which leaks `Dispatch<SetStateAction<…>>` and couples the child to the parent's storage) — the one exception is a tiny presentational `<Input>` that genuinely *is* the input. `on*Change` handler naming carries from L1. Derived `visible` list stays in render, not state (L2 callback). Form state lives on the form component holding one `values` object, inputs presentational via `setField` callbacks — the canonical legitimate lift. Prop-drilling 2-3 layers is fine and refactor-friendly (visible, typed path); context is for cross-cutting concerns (auth/theme/locale), not for skipping intermediates.

## Lesson 4 — useReducer when transitions multiply

**Taught.** `useReducer` as the synthesis hook that cashes in immutable updates + snapshot + purity + discriminated unions at once. Trigger threshold (verbatim rule): switch to a reducer when 3+ `useState` calls update together / a transition must hold an invariant / one value is set from many scattered handlers. Signature `[state, dispatch] = useReducer(reducer, initialState[, init])`; reducer defined outside the component, `(state, action) => State`, types live on the signature (no generics on the hook call). Action = discriminated union of *events/commands* (intent), narrowed per `case`. Reducer body = one exhaustive `switch` ending in `default: assertNever(action)`; spread `...state` first then override. The headline win: collapse `isSubmitting`/`isSaved`/`hasError` booleans into one `status: 'idle'|'submitting'|'saved'` field → illegal states unrepresentable, not just guarded. Async lives in the *handler* bracketing the await (dispatch `submit` → await → dispatch `success`|`error`), never in the reducer. `dispatch` is referentially stable (sanctioned prop-drill exception, unlike raw setters from L3). Lazy `init(arg)` runs once on mount; on a changed prop use `key`-reset, not re-init. Decision: useState (one value) → useReducer-status-machine (illegal combos) → useReducer-transition-authority (many handlers) → grouped `useState` (independent + simple) — this leaf is where L1's deferred separate-vs-grouped decision lands.

**Cut.** Chapter-outline's "time-travel-debuggable (devtools recognition only)" dropped. Updater-form-style "action computes from previous state inside reducer" point folded into the general reducer purity treatment, not called out separately.

**Debts.** Strict Mode reducer double-invoke *mechanism* → Ch 025 L1 (asserted only). `useContext` + split-state/dispatch contexts for sharing a reducer → Ch 025 L5 (corrected from chapter-outline's stale "L4"). `useTransition` wrapping `dispatch` → Ch 025 L6 (one-liner). `useActionState` + Server Actions package the start/await/resolve shape → Unit 6 (the async handler here is the teaching version). Reducer unit-testing *how* → testing chapter (named as a benefit only). XState/Redux/Zustand/Jotai/Immer = recognition only. `saveInvoice` is a stand-in, not a data-layer lesson.

**Terminology.** ((transition)) — a move from one state to the next; ((invariant)) — a rule about which combinations are legal ("never submitting and saved at once"); ((dispatch)); ((reducer)); ((side effect)); ((discriminant)) / ((payload)) re-asserted from Ch 005. Slogans: "a reducer is a state machine you write inline; reach for it when your `useState`s start coordinating, not before"; "actions name events/commands, not setters" (`'submit'` good, `'setIsLoading'` smell — caller describes *what happened*, reducer decides *what it means*); "always spread `...state` first, then override"; "reach for the reducer when the ceremony pays for itself"; "12 `useState`s = a machine you haven't modeled yet" (the loud trigger). "Setter in disguise" = a setter-shaped action.

**Patterns.** Reducer/`init` defined module-scope (outside component); `State` uses a single discriminated `status` field over parallel booleans. `default: assertNever(action)` preferred over `return state` (a same-reference return bails out the render — usually a bug). Read state straight in JSX (`state.status === 'submitting'` drives `disabled`); guard null with `state.submitError != null && …`, not a bare `&&` on a non-boolean. Async handler uses Result-style `{ ok }` return (Ch 005 convention) instead of `try/catch` in teaching snippets. `dispatch` passed to children directly (no `useCallback`); raw setters still must not be (L3).

**Misc.** Running example is the `NewInvoiceForm` async-save form (coherent across the trigger/signature/reducer/wiring/async/init sections; opens on *five* `useState`s — not the outline's "eight" — for a tighter cluster); the practice `ReactCoding` uses a separate disposable download-button machine (`idle|loading|done|error`) so the grader isn't coupled to the form. Mark convention continued: orange = smell (scattered setters, async-in-reducer, partial/same-state return), green = correct (reducer fix, handler-owns-async), red = compile error (the `assertNever` catch). The chapter-map Mermaid figure was *not* forced here (the outline notes flagged it doesn't fit a hook-selection lesson); the interactive decision uses a `StateMachineWalker` (kind="decision", not in `<Figure>`) instead. Components used: `AnnotatedCode` (reducer body walkthrough + `init`), `CodeVariants` (scattered-vs-reducer before/after; reused for the wiring section), `Buckets` (twoCol 8-chip intent-vs-setter action-name sort with `<details>` answer key), `DiagramSequence` + bespoke `AsyncStateTimeline` (4-step submit→await→success/error timing strip), `StateMachineWalker` (decision tree), `ReactCoding` tests-mode `hidePreview` (add the missing `'error'` case), `CardGrid` (state-machine/Redux/Immer/context vocabulary closer), one shipped `VideoCallout` (Cosden Solutions, useReducer from scratch), 4 `ExternalResource` cards (React useReducer ref + "Extracting State Logic" + Kent C. ## Lesson 5 — useRef as the non-rendering escape hatch

**Taught.** `useRef` as the fifth home — the first whose answer is *not state*. A ref is a stable `{ current }` box: same object every render (like the `useState` setter), `.current` freely mutable, writing it never re-renders (no subscription / `Object.is` / reconciliation). The decision rule (the headline): **"does the JSX read this value?"** Yes → state, no → ref. Two equal-and-opposite errors named: *state-in-disguise* (a `useState` whose value appears in no JSX — timer ID, scroll offset, interaction flag) and *ref-where-state-belongs* (reading a UI-driving value out of `.current`, screen goes stale). Two ref flavors: (1) instance memory (timer ID for debounce — the worked example; previous-value; render counter); (2) DOM handles via the `ref` prop, with the commit-timing trap — `.current` is `null` until React assigns the node *after commit*, so never read a DOM ref in render. Four canonical DOM reaches framed as "capabilities the DOM exposes that React doesn't": focus / measurement (`getBoundingClientRect`) / imperative media (`.play()`) / scroll — all optional-chained. The during-render rule (never read/write `.current` in the body — purity) with one exception: lazy ref init via guarded `if (ref.current === null) ref.current = …` (idempotent self-guarding write; parallels L1 eager-vs-lazy `useState`). Compiler note: it treats `.current` as opaque; the `react-hooks` lint rule (not a runtime badge) catches ref-during-render.

**Cut.** Nothing material from chapter-outline scope; `useImperativeHandle`, ref callbacks, `ref`-as-prop, `usePrevious`, IntersectionObserver-combo, module-scope-ref leak all deliberately recognition-only or named-once per outline (each owned elsewhere).

**Debts.** Effect cleanup for a ref-held timer on unmount → Ch 025 L2 (named as an obligation, mechanism deferred; the AnnotatedCode step 4 flags it as the "open thread"). `usePrevious` custom hook → Ch 026 L1 (named, not built). React Compiler surface at depth → Ch 026 L2 (here only the consequence of the purity rule). `ref` as a React 19 prop / passing a ref down / `useImperativeHandle` / ref callbacks → Ch 022 L4 (cited as already taught, recognition cross-refs). Controlled-input theory / forms → Unit 6 (controlled-input example here exists only to show what a ref must *not* do).

**Terminology.** ((useRef)) — stable mutable container whose changes never trigger re-render. ((commit)) re-asserted from Ch 023 (the moment React applies changes to the real DOM; when DOM refs get assigned). ((imperative)) — telling the element to *do* something (focus/play/scroll) vs describing what it should be. ((debounce)) — wait until input stops before acting. ((pure)) re-asserted from Ch 023 L3 as the reason for the during-render rule. The headline slogan: **"if the JSX doesn't read it, it doesn't belong in state."** "state-in-disguise" / "ref-where-state-belongs" name the two errors. `ReturnType<typeof setTimeout>` over bare `number` (correct on both Node and browser) glossed at the call site.

**Patterns.** Annotate the generic at the seam — `useRef<T | null>(null)` for anything empty-at-first (mirrors `useState<User|null>(null)` from L1); inference handles `useRef(0)`. Always optional-chain a DOM-ref read (`boxRef.current?.focus()`) — the node may be `null` (not yet committed / conditionally absent); this is the same `value != null` discipline carried from L4. Touch `.current` only in handlers and effects, never render body — the sole exception is the guarded lazy-init `if`. Never reach for a ref to re-read/re-implement what props/state own (input values, class toggling, `style.display`).

**Misc.** Small disposable snippets, no continuous-example character (matches L1/L2; the reaches — debounce, focus, scroll, previous-value — are too varied to force one). Mark convention continued: orange = smell (ref-read-via-DOM defeating controlled pattern), green = correct. ## Lesson 6 — useId for stable IDs across the server boundary

**Taught.** `useId` as the odd-one-out final hook — holds no chosen value, not a state home; the argument-less call returning one opaque string with three guarantees: unique-per-instance-on-page, stable-across-renders, identical-on-server-and-client. The id is derived from the component's *position in the render tree* (the one thing guaranteed identical on both runs), which is why it's a hook not a `makeId()` function and why the same call is correct in Server- and Client-rendered components alike. Problem framed by watching three naive fixes fail: hardcoded literal (collides on reuse), `Math.random()` at render (unique but diverges across the server boundary → hydration-mismatch warning), `crypto.randomUUID()` (same SSR failure + overkill). Canonical wiring pattern: one `useId()` per field, base id on `<input id>` + `<label htmlFor>`, suffix-derived `errorId` on the error `<p>`, wired via `aria-describedby` + `aria-invalid={error != null}`. Two anti-uses landed hard: (1) list keys — structurally impossible (keying *chooses* instances before they exist; useId ids exist only *after*), keys come from data or a `crypto.randomUUID()` minted-once-and-stored-at-creation; (2) secrets / human-readable anchors — useId is not random/unguessable/cross-session and not readable, wrong for CSRF/session/idempotency tokens (→ server `crypto.randomUUID`) and for `#pricing`-style fragments (hand-authored). Plus call-order discipline (keep useId out of conditionals/loops, foreshadows rules of hooks) and don't-style-by-these-ids. Recognition: shadcn/ui + Radix call useId internally for ARIA; wrapper-owns-the-call, passes id inward as a prop.

**Cut.** `identifierPrefix` / multi-root rendering option (chapter-outline listed it) — cut as a legacy/multi-root niche a 2026 single-root Next.js app never touches; not even a `<details>` deep-dive in the shipped lesson.

**Debts.** Hydration mechanics / mismatch reconciliation / `suppressHydrationWarning` / moving reads into effects → App Router SSR lessons (Ch 030-area); here only the mismatch *warning* named as a symptom, with an explicit caution not to "fix" it by silencing. List `key` contract cited as known from Ch 023 L2 (restated one sentence as the contrast, not re-taught). ARIA roles / live regions / `role="alert"` / focus management → Accessibility chapter (Ch 027-area); this lesson uses `htmlFor`/`aria-describedby`/`aria-invalid` only as the concrete wiring useId enables. Form field components / validation / submission / Server Actions → Unit 6 (the `TextField` is a teaching prop; `error` is a plain prop, no validation lib). `crypto.randomUUID` / CSRF / session / idempotency tokens at depth → server/security units (named as the correct alternative only). shadcn `<Label>`/`<Input>` API + Radix internals → shadcn lesson (Ch 027 L1, recognition).

**Terminology.** ((useId)) — a hook returning a stable, unique id string, called with no arguments. ((SSR / server rendering)) — page built to HTML on the server first, then sent to the browser (recognition depth only). ((hydration)) — the browser re-running your components over the server's HTML to make it interactive; mismatch → React warns (recognition depth only). ((htmlFor)) — JSX spelling of the HTML `for` attribute, names the input's `id`. ((aria-describedby)) — ARIA attribute naming the id of an element that describes this one. The id is **opaque** — current React 19.1 format is Unicode-guillemet `«r1»` (changed from old colon-wrapped `:r1:` so ids are valid CSS selectors without escaping); **all prose kept format-agnostic ("an opaque, unique string")** — do not write "ids with colons" or teach selector-escaping. Slogans: "generate once, reference everywhere" / "wire by reference, not by hardcoded value"; "the determinism is a side effect of the tree being the tree."

**Patterns.** One `useId()` per field, suffix the base for related ids (`const errorId = \`${id}-error\``) over calling useId multiple times (both legal; suffix keeps "same field" visible) — this is the house standard for form fields. `aria-invalid={error != null}` and `aria-describedby={error != null ? errorId : undefined}` wired on every form field with a possible error; conditional error node guarded with proper-boolean `error != null && <p id={errorId}>…</p>` (carries chapter-wide proper-boolean conditional-render rule). Dangling `aria-describedby` is harmless but conditionally setting it (drop to `undefined`) is the tidier senior move. For client-created data with no server id, mint `crypto.randomUUID()` once at creation and store on the item (contrast: random-at-render is wrong, random-once-stored is right). Standards-clean component shape: typed props, arrow-fn bound to `const`, named export.

**Misc.** Small disposable snippets (`TextField`), no continuous-example character (matches L1/L2/L5). Anti-pattern snippets (hardcoded id, `Math.random()` id) shipped to be rejected — not corrected inline; mark convention continued: orange = smell (hardcoded/random id), green = correct (`useId`), with red reserved for the collision/mismatch break. Components used: `CodeVariants` (3 tabs Hardcoded/Math.random/useId, first two orange, third green teaser), `CodeTooltips` (signature gloss), `Figure` (custom HTML+CSS two-panel diagram contrasting useId tree-position determinism vs Math.random divergence across the server boundary, `margin:0` reset), `AnnotatedCode` (4-step full `TextField` wiring), `ReactCoding` tests-mode `hidePreview` (wire two `TextField`s — grader asserts the *association*: `label[for]===input.id`, both non-empty, the two inputs' ids differ, `aria-describedby`===error `p.id`, `aria-invalid==="true"` only on the errored field — never a literal id string since useId output is opaque/non-reproducible), `Buckets` (twoCol, 8-chip useId / crypto.randomUUID / hardcoded-human-authored sort), one shipped `VideoCallout` (Web Dev Simplified, "The Easiest React Hook", 7 min), 4 `ExternalResource` cards (React docs useId + MDN aria-describedby + W3C WAI Labeling Controls + web.dev Accessible forms).
