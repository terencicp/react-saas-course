## Lesson 1 — JSX is property syntax for HTML

- **Cut** — the JSX whitespace-collapse rule / explicit `{' '}` (deliberately dropped, noted in outline); classic-vs-automatic runtime config.
- **Debts** — reconciliation deep treatment → Ch 023 L2; resetting state with `key` → Ch 023 L5; components/props/composition/`ref`-as-prop → Ch 022; children-as-API → Ch 022 L1; synthetic events depth → Ch 023 L6; render model → Ch 023; Tailwind/`cn()` → Ch 018; full XSS/`dangerouslySetInnerHTML` security → Ch 054 L4. Builds on Ch 014 L2 (attr-vs-property rename table), Ch 016 L1 (`crypto.randomUUID()`).
- **Terminology** — "element descriptor" (a.k.a. React element); "automatic runtime"; "intrinsic element"; "JSX transform"; Turbopack defined as "Next.js's Rust-based bundler"; "the four bug classes" framing (class/className, index keys, the `0`-trap, pass-vs-call handler); recurring spine "you write JSX, the browser receives HTML, the differences are where bugs hide".
- **Patterns and best practices** — `key={row.id}` never `key={index}` / `Math.random()`; coerce numeric LHS of `&&` to boolean (`items.length > 0 &&`); pass handler by reference, wrap in arrow for args (`() => fn(x)`); fragment over wrapper `<div>`; void elements self-close; `style` is a last resort (Tailwind is the styling default); guard property access with `?.`.
- **Misc.** — `key` is NOT inside `props` — transform hoists it as the third arg `jsx(type, props, key)`; any descriptor diagram must keep examples keyless or show `key` separately. ## Lesson 3 — Landmarks and the heading outline

- **Taught** — the page as **two cooperating-but-separate maps**: landmarks (region map) and the heading outline (content map). Seven landmark elements taught grouped by job: page-frame trio (`<header>`→`banner`, `<main>`→`main` exactly-one-never-nested, `<footer>`→`contentinfo`), `<nav>`→`navigation` (reserved for major nav blocks only), grouping pair `<section>`→`region`/`<article>` (landmark only when *named*; "six vs seven" explained by this conditional), `<aside>`→`complementary`. Element-decision tests: `<article>` = "still makes sense pasted elsewhere?" (dashboard metric card is NOT an article); `<section>` = "has/should have a heading"; sidebar-with-nav-links = `<nav>` not `<aside>`. ## Lesson 4 — Actions, navigations, and item sequences

- **Taught** — three element families chosen by behavior: **`<button>`** (acts in place — submit/open/delete/toggle, never navigation) with its five free behaviors (Tab focus, `Enter`+`Space` activation, focus ring, "button" role, `disabled` handling); the `<button type>` reflex (in-form default is `type="submit"`; values `submit`/`button`/`reset` lowercase; explicit `type` on every button so a form can't hijack a Cancel click); `disabled` boolean + the "surface the reason" rule; icon-only buttons need `aria-label` (the only ARIA taught). **Links** — `<a href>` (inert without `href`; `href` is the tell that "this goes somewhere"); `target="_blank"` pairs with `rel="noopener noreferrer"` (noopener=stops tabnabbing, noreferrer=strips Referer); `download` + fragment `#id` links (`scroll-smooth`); **`<Link>`** from `next/link` = `<a>` + soft navigation, renders to a real `<a href>`. The consolidating decision (`<button>` vs `<Link>` vs `<a>` vs `<div role="button">`): match element to behavior not look; the full hand-wired cost of faking a button with a `<div>` (`role`, `tabIndex={0}`, `onKeyDown` for Enter+Space w/ `preventDefault` on Space, focus-ring CSS, `aria-pressed`, `cursor-pointer`) → verdict never. **Lists** — `<ul>`/`<ol>`/`<li>` (`<li>` is the only valid direct child); `.map`+data-tied `key` (from L1); list-vs-divs test ("would a screen-reader user usefully hear 'list, N items'?"); the canonical nav-list pattern (`<nav aria-label>` › `<ul className="flex gap-4 list-none">` › `<li>` › `<Link>`). Form handoff: `Enter` in a text input also submits; per-submit `form*` overrides named for recognition only.
- **Cut** — nothing material; `<ol>` `start`/`reversed`/`type` and nested lists kept as one-line recognition mentions per outline.
- **Debts** — forms/`<form>`/`<input>`/`<label>`/the `name`→`FormData` contract → L5 (the Save/Cancel form shells here are illustrative only, `action={fn}` unexplained); React 19 Server Action wiring → L5 / Ch 044; full ARIA (`aria-pressed`/`aria-current`/`aria-expanded`/`role`/live regions) + `aria-describedby` → L6 and Ch 027 L3; `<table>` decision → L6; shadcn `<Button>` + CVA + `asChild`/Slot → Ch 027 L1 / Ch 022 L3; Tailwind utilities + Preflight internals → Ch 018 / Ch 019 L3 (Preflight named only); `<Link>` depth (prefetch, scroll restoration, navigation model) → Ch 029 L4; focus management + skip-link mechanics → Ch 027 L4. ## Lesson 5 — Forms as a contract with the server

- **Taught** — the form as a **contract the server reads**: the four-step round-trip (user fills labeled fields → browser serializes named controls into `FormData` → crosses wire → server validates with a Zod schema), where steps 2 and 4 share one vocabulary — the `name` strings — so form and schema are written together. Native forms = **progressive enhancement** by default (round-trip works before/without JS; React form hooks layer on top, never replace). `<form>` element + three attributes as decisions: `action` (URL string in HTML, a *function* / Server Action in 2026 React; omit = current URL), `method` (`get`=query string/navigation vs `post`; rule "anything that changes server state is `post`"), `encType` (only touched for `<input type="file">` → `multipart/form-data`). Submit path w/o JS (`<button type="submit">` or Enter); every button declares `type` (L4 cash-in). **`name` = the `FormData` key** (no `name` → silently dropped, the canonical bug); **camelCase house rule** (`name="organizationName"` aligns with Zod key / Drizzle column; kebab forces a boundary rename). **`<label>` as a second contract** (screen reader / motor / autofill); explicit `htmlFor`/`id` (survives refactors, form-lib friendly) vs implicit wrap; **`htmlFor` takes `id` not `name`** — `id`=DOM/label address (page-unique), `name`=server key (form-unique, shared by radio groups); placeholder is not a label. **`type` = a UX contract** (keyboard/autofill/picker/loose check) that never changes that **every value arrives as a string** (incl. `type="number"`→`"42"`, server coerces via `z.coerce.number()`); SaaS type reference table. Other controls tied to *what they contribute to `FormData`*: `<textarea>` (`defaultValue`, uncontrolled), `<select>`/`<option>` (explicit `value`s; `multiple`→`getAll`), **checkbox** (contributes `value` only when checked, *nothing*—absent key—when unchecked; default submit value `"on"`, use explicit `value`), **radio groups** (shared `name`, only checked one submits once — the case where `id`/`name` come apart), `<fieldset>`/`<legend>`. **`autoComplete` = the autofill contract** (semantic tokens; `current-password` vs `new-password` pairing; `autoComplete="off"` on passwords is a regression, legit only for one-off values like 2FA). **Native constraints are UX not security** — `required`/type-checks/`min`-`max`/`minLength`-`maxLength`/`pattern` (pattern rare in 2026); **client constraints can always be bypassed**, every one mirrored by a server-side Zod rule that is the only enforced checkpoint. `disabled` (dropped from submission) vs `readOnly` (submitted).
- **Cut** — nothing material; the optional end-of-lesson consolidating `ReactCoding` was not added (per-section drills sufficed); `cols` on `<textarea>` and `text/plain` encType dropped to keep the surface tight.
- **Debts** — `<form action={serverAction}>` wiring + submit lifecycle + reading `FormData` server-side → Ch 044 L1–L2; `useActionState`/`useFormStatus`/`useOptimistic` + `<SubmitButton>` → Ch 044 L3–L5; controlled-vs-uncontrolled at depth (`value`+`onChange`) → Ch 044 L1 (named one line only here); Constraint Validation API depth (`ValidityState`, `setCustomValidity`, `inputmode`) → Ch 044 L6; **Zod schema authoring + `z.coerce.number()` + checkbox `"on"` coercion + `safeParse`/`Object.fromEntries`** → Ch 042 (esp. L6) / Ch 043 L2 (referenced by name only, no schema authored); React Hook Form / Conform as the conditional past native → Ch 045; file-upload mechanics + `Blob`/`File` → Ch 016 L3 / later upload lessons (`type="file"`+`multipart` recognition only); shadcn `<Form>` + `aria-describedby`/`aria-invalid` error wiring → Ch 027 L1 / Ch 044 L6 / Ch 047 (deliberately NOT pulled in — no error feedback rendered here); Temporal/date handling → Ch 083 (ISO string only here); Preflight stripping `<fieldset>`/`<button>` chrome → Ch 019 (named only). Builds on L4 (`<button type="submit">` + in-form default), L1 (`htmlFor`/kebab `aria-*`, `.map`+key), Ch 015 L1 (`FormData` as a `fetch` body primitive — treated as recognized, here shown to be *built by the browser* from named inputs).
- **Terminology** — recurring spine extended to **"the `name` carries the contract, the `type` carries the UX, the class carries the look"** (direct continuation of L3/L4's "behavior/look are separate knobs" — keep audible); "a form is a contract the server reads" / "the contract" (the shared list of `name` strings); "the silent-drop bug" (missing `name`); "two checkpoints, one trusted" (browser check skippable, server Zod check not); "the autofill contract"; "progressive enhancement" (def given); "one vocabulary across the form, the schema, and the table"; "checked-or-absent asymmetry" (checkbox); "validate on the client for the user, validate on the server for the system, never trust the client".

## Lesson 6 — data-*, aria-*, and tables

- **Taught** — the chapter closer, organized by one question: **"who reads this attribute?"** → three consumers, three tools. **`data-*`** (your own scripts): valid HTML the browser stores but never renders/styles/assigns meaning to — a private channel to your code; read via `element.dataset` (Ch 014 cash-in) with the **kebab↔camelCase** rule (`data-invoice-id`→`dataset.invoiceId`); `data-*`/`aria-*` are the two JSX pass-through families (stay kebab); values round-trip as strings (`data-count={5}`→`"5"`). The 2026 consumer tour (recognition, each owned later): **event delegation** (one container listener + `closest('[data-action]')`, the canonical SaaS site), `data-testid` (Playwright — *fallback*, accessible queries first), analytics hooks (`data-event-name`), Tailwind `data-[state=...]` variants. **`aria-*`** (assistive tech): judgment not syntax — **"no ARIA is better than bad ARIA"**; the **element-first decision order** (1 semantic element → 2 minimal `aria-*` → 3 `role` last resort, rarely right in 2026); the a11y tree (L3 recap) populates roles/names/states for free. Daily-reach set grouped by job: name a text-less control (`aria-label` literal — *overrides* visible text, so icon-only; the L4-deferred icon-button case lands here) / name by reference (`aria-labelledby`→heading `id`, preferred when a visible label exists); describe (`aria-describedby`→hint/error); state-the-eye-sees (`aria-current="page"`, `aria-expanded`+`aria-controls` for a disclosure, `aria-pressed` toggle); hide decorative (`aria-hidden="true"` — **never on focusable/interactive**, the unreachable-trap bug); `role` override (live-region roles the main survivor). Live regions named only: `aria-live` polite/assertive, `role="alert"`=assertive shorthand, region-must-pre-exist watch-out. **`<table>`** (browser layout+a11y engine): the **decision before the element** — positive test = rows/columns of like records indexed by (row,col); the **transpose check** (would swapping rows↔columns still be meaningful? → yes = tabular); *no* cases redirected (page layout→CSS grid, card list→`<ul>`+grid, two-col form→`<form>`+grid). Canonical accessible shape as labeled anatomy: `<table>`›`<caption>` (name, announced first, no `aria-label` needed)›`<thead>`›`<th scope="col">`; `<tbody>`›`<tr>` with `<th scope="row">` identifier cell + `<td>`s; `<tfoot>` (rare/invoice Total); **`scope` is load-bearing** (wires cell→header so reader says "Amount: $200"). Synthesis artifact folds `key={invoice.id}` (L1), `data-invoice-id` delegation hook, `scope`, `text-right` numeric col, em-dash `—` empty-cell convention. Responsive: `overflow-x-auto` wrapper (safe), `sticky top-0` thead, card-per-row at a breakpoint; **forbidden move: `display:block` on `<tr>`/`<th>`/`<td>`** strips table semantics from the a11y tree.
- **Cut** — `colSpan`/`rowSpan` (chapter-outline named them for the `<tfoot>` Total row; lesson dropped to recognition-absent); `<colgroup>`/`<col>` and multiple `<tbody>` kept as a single recognition-only parenthetical; `role="status"` (outline listed it; lesson kept only `role="alert"`).
