## Lesson 2 — Walking the graph: evaluation, live bindings, and the client bundle

- **Taught** — Depth-first post-order evaluation (leaves first, once per module, errors short-circuit upward); imports as live read-only bindings to the exporter's variable (re-exports preserve, importer cannot reassign); circular dependencies (value-level top-level reads return `undefined`→`NaN`, function-body access and `import type` resolve cleanly, Y-shape extracted-shared-module fix); dynamic `import()` as the deferred-edge trigger of code splitting (the bundler splits on the `import()` form, not on `await`); `'use client'` as a client-bundle entry-point directive placed on the smallest interactive leaf; `import 'server-only'` and `import 'client-only'` as build-time enforcement; the split-by-responsibility refactor (`lib/format.ts` vs `lib/auth.ts`).
- **Cut** — None significant; followed the outline. `next/dynamic` kept to a single paragraph as scoped (depth deferred to Unit 4 ch. 030). Dropped CJS contrast to a single sentence as planned.
- **Debts** — Forward links repeated: Unit 4 chapter 030 owns full RSC server/client boundary, `'use client'` placement depth, and `next/dynamic` SSR controls (`ssr: false`); Unit 5 chapter 037 owns the Drizzle relations Y-shape pattern landing in practice; chapter 006 lesson 3 owns the module-level singleton (`let cached: Db | null = null`) and the `env.ts` fail-closed throw mechanism (referenced here); lesson 3 also owns top-level await.
- **Terminology** — "Eager edge" (static import) vs "deferred edge" (dynamic `import()`); "client subgraph" / "client bundle entry point"; "live binding" / "read-only window onto the exporter's variable"; "Y-shape" for the shared-module cycle fix; "fail-closed" startup validation (Term defined inline); "code splitting" (Term defined inline). The `server-only`/`client-only` packages are bare-specifier imports — `import 'server-only';` is the canonical authoring form.
- **Patterns and best practices** — `'use client'` (single quotes) on file's very first line, before imports; `import 'server-only';` is the first line of every module that touches secrets/DB/cookies/headers/private-key SDKs — applies structurally to `env.ts`, the Drizzle client, the Better Auth instance, every billing/email/storage adapter; pure utilities live in their own files separate from any server-seam helpers (file-per-responsibility, not aesthetic — defines where the client subgraph ends); reach for dynamic `import()` only for heavy+rare or conditional component-inside-page splits, never for route-segment splits (App Router handles those); when a value-level cycle appears, extract to a shared third module rather than papering over with function-body deferral; convert to `import type` if the cycle is purely type-level.

## Lesson 3 — Top-level await vs. lazy init

- **Taught** — Top-level `await` as a graph-level mutation that turns every upstream importer implicitly async (cascading up static-import edges to the entry); the three properties that earn top-level work (cheap, mandatory for every consumer, needed at module load); the canonical synchronous `env.ts` with `@t3-oss/env-nextjs` + Zod 4 as the reference shape for "cheap mandatory deterministic"; the canonical lazy `getDb()` shape with `let cached: ReturnType<typeof drizzle> | null = null` and a cached getter; the three-question decision tree (Expensive? → Conditional? → Mandatory?); the two wrong reaches (top-level await for a DB connection; lazy init for env validation).

- **Debts** — Forward links: Unit 4 (Suspense / streaming / async Server Components as the right tool when top-level await is the wrong tool for long/per-request waits); Unit 5 / chapter 037 (full Drizzle wiring — schema-aware return type, `tenantDb(orgId)` factory, unpooled migrations client, serverless connection-pool sizing, the production `env.ts`); Unit 11 (`getStripe()` and `lib/billing/` carve-out); Unit 12 (`getS3Client()`); Unit 14 (`getRedis()`); chapter 007 lesson 3 (async/await mechanics, microtasks, event loop). Lesson 4 of chapter 006 still owns `declare module`.
- **Terminology** — `<Term>`s introduced: *implicitly async* (upstream module that waits because a statically-imported descendant has top-level await), *render-blocker* (work that gates first paint of a server-rendered page), *module-level singleton* (value cached in a module-scoped variable, exposed via getter, one slot per process or per serverless instance), *cold start* (first request handled by a freshly-instantiated serverless function instance). Reused without redefining: `'server-only'`, fail-closed. The phrase "stateful singletons that need setup are exposed through getters, not as top-level exports" is the lesson's generalized rule and should be referenced verbatim by later SDK-adapter lessons.
- **Patterns and best practices** — `import 'server-only';` as the first line of `env.ts`, `db.ts`, and every SDK-adapter module. `env.ts` uses `createEnv` from `@t3-oss/env-nextjs` with Zod 4 `z.url()` (not legacy `z.string().url()`) and a hand-destructured `runtimeEnv` (never `runtimeEnv: process.env`) for bundler-safety. The `getDb()` shape uses `let cached: ReturnType<typeof drizzle> | null = null` at module scope (deliberate `let` carve-out from "default to `const`" — the slot mutates exactly once from `null` to built client) with a `(if (cached) return cached;` guard. Stateful singletons (DB, Stripe, Redis, S3, PostHog) exposed through `getXxx()` getters, never as top-level exports. Use `drizzle({ client, casing: 'snake_case' })` in the project's real `db.ts`. Env validation must fail-closed at module load (synchronous throw from `createEnv`) — never hide it behind a `getEnv()` getter.

## Lesson 4 — Augmenting third-party modules

- **Taught** — `declare module 'pkg' { interface X { ... } }` as TypeScript's interface-merging mechanism to extend a third-party package's published types from a project-owned `.d.ts` file; the load-bearing rule that only `interface` merges (`type` aliases are closed); the module-vs-global ambient foot-gun (a `.d.ts` file needs a top-level `import`/`export` to be a module — otherwise `declare module 'x'` declares a non-existent global); the `types/<package>.d.ts` file convention (one file per package, outside `src/`, picked up by `tsconfig.json` `include`); the three trigger questions (5+ call sites, brand/domain-type preservation, library-designed-to-be-extended) preceded by question zero (does the library ship its own extension contract?); two canonical sites — `next-intl` `interface AppConfig { Messages: typeof messages }` and a narrowing Drizzle relation augmentation; the Better Auth counter-example where `additionalFields` config + `typeof auth.$Infer.Session` is the right reach over `declare module 'better-auth'`; three anti-patterns (augmenting to bypass `| undefined`, augmenting your own app types, `declare global` for ambient values).
- **Cut** — Dropped the chapter-outline's primary Better Auth `declare module 'better-auth' { interface Session { ... } }` worked example (outdated — 2026 idiom is `additionalFields` + `$Infer.Session`); pivoted Better Auth to the counter-example role; cut the proposed `ScriptCoding` exercise (value lands in recognition not keystrokes); did not show the `declare global { interface IntlMessages }` older next-intl shape.
- **Debts** — Forward links: chapter 037 (Drizzle relations API depth — the concrete interface names for the relation-narrowing augmentation); chapter 049 (Better Auth full setup, branding `session.user.id` at the query boundary via the chapter 005 brand factory); chapter 084 (full `next-intl` setup — routing middleware, locale negotiation, `t.rich`).
- **Terminology** — `<Term>`s introduced: *declaration merging* (only `interface`s merge, not `type` aliases), *module augmentation* (the `declare module 'pkg' { ... }` pattern in a project-owned `.d.ts`), *ambient declaration* (`.d.ts` participates in the type checker without runtime code; global vs module ambient distinction), *`$Infer`* (Better Auth's type-inference helper `typeof auth.$Infer.Session`). The "library's extension contract" framing — generics, `$Infer`, builder return types, `additionalFields`-style config — is the lesson's senior payload phrase. "Question zero" precedes the three trigger questions.
- **Patterns and best practices** — Augmentation files live at top-level `types/<package>.d.ts` (kebab-case filename matching import specifier), never `types/global.d.ts` or in `src/`; always include a top-level `import type` to make the file a module (the foot-gun guardrail); use `import type` exclusively inside `.d.ts` (never runtime imports); the senior reflex is "read the library's extension contract before reaching for `declare module`"; for Better Auth specifically, define additional session fields via `additionalFields` config in `lib/auth.ts` and re-export `export type Session = typeof auth.$Infer.Session`; brand IDs at the query boundary (re-brand via the chapter 005 brand factory like `userId(session.user.id)`), never via `declare module 'better-auth'`; narrowing augmentations carry a maintenance debt the cast doesn't (the type narrowing is only as safe as the invariant the query path enforces).
