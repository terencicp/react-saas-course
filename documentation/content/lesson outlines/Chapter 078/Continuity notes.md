# Chapter 078 — Zustand

## Lesson 1 — When Zustand earns its weight

**Taught** — The decision funnel for reaching for Zustand: five prior client-state defaults (useState, useReducer, lifted+Context, nuqs, TanStack Query), three triggers that cross their collective threshold (disjoint/cross-route shared state, imperative actions from >2 unrelated subtrees, profiled re-render cost from frequently-mutated state), the cost of the library, and the per-feature architectural rule; no Zustand API introduced.

**Cut** — The decision-tree as a Mermaid diagram (replaced by StateMachineWalker component); watch-out about `persist` by default and `useStore((s) => s)` whole-store selection (deferred to lesson 2 where the API is taught).

**Debts** — SSR per-request provider (trap named here, fix promised in lesson 2); `sessionStorage` durability via `persist` middleware named conceptually as escape hatch but the word "persist" not used — full treatment in lesson 2; `createStore` vs. `create` split (lesson 2); wizard store shape and Zod-per-step gates (lesson 3); full wizard build (chapter 079).

**Terminology** — *selector*: "a function that reads just the slice of store state a component needs, so the component re-renders only when that slice changes"; *per-feature scoping rule*: every store has one feature owner and lives next to it, never a global `useAppStore`; "if `useState` covers it, ship `useState`" — the chapter's headline reflex; *Zustand is conditional and per-feature, never the app's ambient state bus*.

**Patterns and best practices** — Store co-located beside its feature route: the FileTree in the lesson shows `customers/new/_lib/wizard/store.ts` + `customers/new/_components/wizard-store-provider.tsx` (route-private `_lib/` + `_components/` pattern, not a top-level `/lib/` directory). Store named for its owner: `useWizardStore`, `useCartStore`, `useCommandPaletteStore`. Zustand earns its weight only after running the ordered funnel: server state? → useState? → lifted+Context? → URL? → then Zustand. Trigger 3 (re-render cost) is the weakest and rarely justifies Zustand alone; it earns its place stacked on triggers 1–2.

**Misc.** — Lesson 1 is deliberately syntax-free ("trigger before tool"); lesson 2 must not assume any Zustand API was introduced here. The wizard destination (`/customers/new/step-1` through `step-4`) is named once as motivation but not walked; lesson 3 runs the full funnel against it. The Context whole-tree re-render mechanism is taught via RenderTracking widget (not re-taught in prose), and the `selector` term is introduced with a Term tooltip — lesson 2 builds on it as assumed vocabulary. Reset discipline (sign-out, org-switch, submit-success) is introduced as a named cost here; lesson 2/3 must implement it explicitly. The TanStack Query `queryClient.clear()` parallel is drawn — later lessons can reference this cross-chapter pattern.

## Lesson 2 — Primitives and the per-request provider

**Taught** — The full v5 API surface: `create` vs `createStore`, the cross-request data-isolation bug (module-scoped store leaks one tenant's draft into another request), the three-file per-request provider pattern (factory / `useRef`-pinned provider / typed hook), slices with `StateCreator` typing, named selectors, `useShallow`, action stability, `reset` with the replace flag, and the middleware lineup (`persist`, `subscribeWithSelector`, `devtools`, `combine`/`redux`) named with senior reach-for-it gates.

**Cut** — Full four-slice store (only `contact-slice` shown in full; billing/preferences/meta referenced as "mirror it"); Zod-per-step validation and the `validate()` action; worked `persist` hydration-gating UI (trap named, implementation deferred); full four-step wizard run — all deferred to L3 and ch079.

**Debts** — L3 must cover: running the three-trigger funnel against the concrete wizard, the `validate()` meta-slice action and per-step Zod gates, the Server Action submit boundary as the reset trigger, and the product call for why the wizard does not persist. Ch079 fills in billing/preferences/meta slices.

**Terminology** — `createStore` from `zustand/vanilla` (vanilla store factory, no React binding); `useStore` from `zustand` (generic React hook binding vanilla store to a component via selector); `useShallow` from `zustand/react/shallow` (shallow-compares selector result to suppress spurious re-renders on object/array returns); `StateCreator<WizardStore, [], [], SliceType>` (typed slice initializer — first generic = full store, fourth = this slice's return); `WizardState` = data (slice intersection), `WizardStore` = data + store-wide actions; `Object.is` is Zustand's default selector equality; `slice` = self-contained factory for one feature area.

**Patterns and best practices** — Canonical six-file layout (co-located under the feature route, NOT `lib/`): `customers/new/_lib/wizard/{wizard-types.ts, contact-slice.ts, selectors.ts, store.ts}` (pure/type-only) + `customers/new/_components/{wizard-store-provider.tsx, use-wizard-store.ts}` (`'use client'` boundary). `selectors.ts` exports named selector functions (`selectContactEmail`, `selectCurrentStep`, etc.) — one place to update if the shape moves; ch079 must include this file. Provider mounted on the **shared route-segment layout** (`customers/new/layout.tsx`), not per page — so the store survives back/forward between steps. `'use client'` only on the provider (and consumer step components); never on the factory, slices, types, or layout itself. `useRef<WizardStore | null>(null)` with `if (storeRef.current === null)` guard (React 19 + React Compiler safe form; `useState` lazy initializer is equivalent but not standardized here). `reset: () => set(initialWizardState, true)` — replace flag required (v5 compile-checks completeness); called at submit-success, sign-out, and org-switch. Atomic selectors for 2–3 fields; `useShallow` for list/object returns. `devtools` gated behind `process.env.NODE_ENV !== 'production'`. No barrel files; import the exact slice/selector needed.

**Misc.** — The lesson explicitly flags the file placement as a **deliberate divergence** from the chapter outline's `/lib/wizard/store.ts` shorthand, forced by the no-React-in-`lib/` code convention; L3 and ch079 must preserve this exact layout. The provider uses React 19 `<Context value={...}>` syntax (no `.Provider`). The `useState(() => createWizardStore())` form is named as equivalent to `useRef` in the official guide — the course standardizes on `useRef`; don't mix. The lesson closes with a forward pointer: L3 runs the concrete funnel + product calls; ch079 builds file by file.

## Lesson 3 — The routed wizard, end to end

**Taught** — Senior reasoning that justifies Zustand on the four-step `/customers/new/step-1…4` wizard: ran the L1 funnel against it (three triggers all met, trigger 3 weakest), rejected four alternatives (URL leaks PII, server drafts are product scope not stack scope, `useState` lifting collapses routing, Redux/Jotai/Valtio already settled in L1), named the `MetaSlice` shape and the Zod-per-step derived-validity gate, the submit boundary (store assembles draft → `createCustomer` action persists → `reset()` + redirect), and the navigation preserve/lose/discard contract with reset discipline at tenancy boundaries.

**Cut** — Full implementation of `BillingSlice`, `PreferencesSlice`, billing/preferences schemas, and all four route-segment pages (reserved for ch079); `persist` hydration-gating UI (named as available escape hatch, not implemented); step transition animations/progress-bar polish.

**Debts** — Ch079 builds file by file against the contract described here: four route segments, per-step Zod schemas, four-slice store, provider on shared layout, Next-gate wired to each step, `createCustomer` Server Action, success-reset-then-redirect.

**Terminology** — *draft (state)*: "data the user is editing that isn't persisted yet; here, the in-progress customer that doesn't exist until step 4 submits"; *routed wizard*: "a multi-step flow where each step is its own URL/route segment, so back/forward and deep links work — as opposed to a modal wizard living on one route"; *tenancy boundary*: "the moment the active tenant changes (org-switch) — any per-tenant client cache must be cleared or it shows another org's data."

**Patterns and best practices** — Validity is *derived*, not stored: `selectIsStepValid` indexes a `steps` array (each entry: `{ schema, slice }`), runs `step.schema.safeParse(step.slice(state)).success` — `slice` pulls only the relevant sub-state (e.g. `state.contact`), not the full store; falls back `true` for the review step and out-of-range indexes; no stored `isValid` boolean, no `validate()` action to keep in sync. Store action (`goNext()`) and router navigation (`router.push(...)`) are paired at the call site — the slice never calls the router. Same Zod schema gates the client "Next" button (UX) and validates inside the Server Action (correctness); both fire. `reset()` called at three boundaries: submit-success, sign-out, org-switch. The fork between routed and modal wizard is explicit: routed steps → store on the shared layout; modal/single-route steps → one `useForm` + `FormProvider` (ch045 pattern). URL is the source of truth for *routing*; the store is the source of truth for which step is *valid to reach* (no skipping). Do not stash the new customer's id in the store after submit — that is server state; redirect to the detail page.

**Misc.** — The `MetaSlice`/`WizardStore` reconciliation is stated explicitly: `goNext`, `goBack`, `currentStep`, `completedSteps` live in `MetaSlice`; store-wide `reset` stays on `WizardStore = WizardState & { reset }` exactly as L2 shipped it — ch079 must match. `createCustomer` Server Action (verb+noun, no `Action` suffix) uses the five-seam shape and returns `Result<T>`; Zod 4 top-level builders (`z.email()`), flat error shape (`z.flattenError(...).fieldErrors`). The TanStack Query `queryClient.clear()` parallel at org-switch is explicitly drawn as the same discipline — future lessons can reference both as the cross-chapter pattern. Provider placement on the shared layout (not per-page) is flagged as the canonical mistake and repeated twice.
