# Chapter 087 — Unit tests for lib

## Lesson 1 — Pure-function tests, the daily shape

**Taught** — Colocation rule (`name.ts` / `name.test.ts` siblings), AAA collapse for pure functions, matcher selection by value shape, one-`describe`-per-export / one-`it`-per-branch file organization, `it.each` object form, `expect.extend` custom matchers, and the purity diagnostic ("wants setup → misclassified, move to integration").

**Cut** — The lesson outline listed the full `/lib` surface inventory (temporal codecs, i18n, auth predicates, rate-limit algorithm, cache tag composers); the lesson anchors examples to `error-mapping.ts`, `money.ts`, and `redact.ts` only. No mention of `formatInvoiceTotal` or locale/i18n lib functions as test targets.

**Debts** — Fixture factories / `build*` pattern deferred to Lesson 2; `vi.useFakeTimers` / fake-timers `beforeEach` exception named but deferred to Lesson 3; deterministic-seam mocks (clock, IDs, RNG) named but deferred to Lesson 3; `expectTypeOf` named as a future `vitest` import but not taught, deferred to Lesson 4; `toThrow` listed in matcher figure but depth deferred to Lesson 6; `toBeOkResult` / `toBeErrResult` *introduced* as the home for `Result` assertions, but their unhappy-path craft deferred to Lesson 6.

**Terminology** — `colocation` (source and test in same directory); `globals: false` (requires per-file `import { describe, it, expect } from 'vitest'`); `SameValue` (`toBe`'s equality algorithm); `deep equality` (`toEqual`); `expect.extend` (custom matcher registration API); `behavior catalog` (a test file read top-to-bottom as a list of what the function does); `unit` project (Vitest project name, glob `src/lib/**/*.test.ts`).

**Patterns and best practices** — Custom matchers live in `src/test/matchers/` (e.g. `result.ts`); threshold for a custom matcher is the same comparison written ≥3 times across files. `toBeOkResult` / `toBeErrResult` are the canonical `Result` assertion pair for the full chapter. For `Temporal.PlainDate`, assert `.year`/`.month`/`.day` or `.toString()` — never `toEqual` on the object. For `Temporal.Instant`, assert `.epochMilliseconds`. `it.each` rows must be inline literals (never `it.each(generateCases())`). A `/lib` test importing from `app/**` is caught by `no-restricted-paths` lint rule.

**Misc.** — `Result<T>` shape used throughout: `{ ok: true, data }` / `{ ok: false, error: { code, userMessage } }` with lowercase snake `code` values (e.g. `'conflict'`, `'not_found'`, `'internal'`). The `unit` vitest project and its coverage thresholds (90% lines / 85% branches on `src/lib/**`) are treated as already configured from Ch086 — Lesson 1 does not reconfigure them. Inner-loop command established: `vitest --project unit --watch`. `formatMoney` signature used in the lesson is `formatMoney(cents: number): string` (not the `bigint`/`Currency` variant from the chapter-outline brainstorm); later lessons should use this signature. `toContainEqual` / `toContain` appear in the matcher-by-shape figure component but are not mentioned in prose; they are not safe to assume as "taught" vocabulary until a later lesson names them explicitly.

## Lesson 2 — Factories over shared fixtures

**Taught** — The `buildEntity(overrides: Partial<T> = {}): T` factory pattern: three rules (valid defaults, overrides-last spread, fresh object per call), factory composition with circular-default trap, `sequence()` helper for deterministic uniqueness, and the factory/fixture/seed three-noun distinction including static JSON fixtures for external payloads.

**Cut** — The chapter outline listed `webhookEvents.ts` as a fourth factory file; only `users.ts`, `orgs.ts`, and `invoices.ts` are shown. `DeepPartial<T>` was named but explicitly not taught (stay flat until forced). `fishery`/`rosie`/`factory.ts` library DSLs named only in one closing paragraph.

**Debts** — Lesson 3 owns `lib/clock.ts`, `lib/ids.ts`, `lib/random.ts` seams and `vi.useFakeTimers`; lesson established factory time/ID defaults are literals or seam-sourced but does not teach the seam. Lesson 4 owns `expectTypeOf` type-level tests on factory return shapes. Ch088 L3 owns DB-inserting factories and integration fixtures (signed-in-user, org seed, `withRollback`). Ch088 L5 owns MSW handler factories. Ch088 L6 owns Stripe fixture signature verification (forward-referenced as the reason static JSON beats a factory for webhook payloads). Ch040 L3 owns `drizzle-seed` / `scripts/seed.ts`.

**Terminology** — `run-order coupling` (a test's outcome depends on whether/which other tests ran first); `false negative` (test passes while the code is broken); `monotonic` (sequence only ever increases, never repeats); `InferSelectModel` (Drizzle type derived from schema, used to type factory return); `object mother` pattern (named and dismissed — `buildEntity({ ...overrides })` covers it).

**Patterns and best practices** — Factories live at `src/test/factories/`, one file per entity (`users.ts`, `orgs.ts`, `invoices.ts`), one `build<Entity>` export per file. Factories import domain types from `@/db` or `@/lib`, never from `app/**` (enforced by `no-restricted-paths`). Factory defaults must be valid domain instances — no placeholder strings like `'TODO'`. Factory defaults for time/IDs are hardcoded literals, not live `Temporal.Now.instant()` / `crypto.randomUUID()`. `Math.random()` / `crypto.randomUUID()` in a factory default is forbidden — use a `sequence()` helper for uniqueness. Static external payloads (Stripe webhooks, Resend bounces, S3 events) live in `src/test/fixtures/` as captured JSON, updated by re-capture not hand-editing. `@faker-js/faker` is for dev seeds (`scripts/seed.ts`), not unit tests.

**Misc.** — The `src/test/` shared-support tree is now partially established: `factories/` (this lesson), `matchers/` (L1), `fixtures/` (referenced as forward-crumb), `clock.ts` (L3 crumb). The chapter-outline `webhookEvents.ts` factory was silently dropped; later lessons or Ch088 that need a webhook factory should create it. The lesson explicitly diverges from `Code conventions.md` (which says `db/factories.ts` with Drizzle inserts) — the canonical home for this chapter's pure in-memory factories is `src/test/factories/`, not `db/factories.ts`.

## Lesson 3 — Pinning time, IDs, and randomness

**Taught** — The seam pattern applied to all three non-deterministic sources: `lib/clock.ts` (`clock.now()`), `lib/ids.ts` (`newId(prefix)`), and `lib/random.ts` (seeded `pure-rand`); two swap shapes (injection vs. `vi.mock`); `vi.useFakeTimers()` / `vi.useRealTimers()` for scheduling code; the Temporal-not-faked gotcha; `vi.spyOn` for single-call-site stubs; the `FROZEN` canonical instant and `freezeClock()` helper in `src/test/clock.ts`; the `vi.mocked` companion set; `vi.resetAllMocks()` discipline; `no-restricted-syntax` lint rule to enforce the seam structurally.

**Cut** — DST and month-end determinism in inputs (chapter-outline bullet; dropped as a Ch083 concern about input dates, not a seam concept). `vi.spyOn(Date, 'now')` as a narrow stub was contrasted against the full clock seam in passing but not taught as a first-class pattern.

**Debts** — `vi.mock` hoisting order rules named once ("runs before imports") but depth deferred to L5. `*Async` fake-timer variants and the forgotten-`await`-with-timers trap named as a crumb, owned by L5. `mockResolvedValue` / `mockRejectedValue` introduced by name here; exercised at depth in L5 and Ch088. `expectTypeOf` on seam signatures deferred to L4.

**Terminology** — `seam` (deliberate swap point: production wires real impl, tests wire frozen/seeded); `wall clock` (`Temporal.Now` / `Date.now()`, the machine's real current time); `flaky` (test whose pass/fail changes without code changes — reframed as "broken, not flaky" when the cause is structural); `FROZEN` (`Temporal.Instant.from('2026-01-15T12:00:00Z')`, the canonical test instant shared across the suite); `seed` (fixed starting value that makes a pseudo-random stream reproducible); `jitter` (randomness added to retry backoff); `macrotask` (a `setTimeout`/`setInterval` callback run by the timer queue).

**Patterns and best practices** — Every non-deterministic source goes through a seam: `lib/clock.ts`, `lib/ids.ts`, `lib/random.ts`. Never `Date.now()`, `Temporal.Now.instant()`, `crypto.randomUUID()`, or `Math.random()` inline in `/lib`. Default to injection for `/lib` helpers (explicit, type-checked, no mock machinery); reach for `vi.mock` only when threading the dep through a deep call chain costs more than a single module-level swap. `vi.useFakeTimers()` in `beforeEach` / `vi.useRealTimers()` in `afterEach` is the only legitimate `beforeEach`/`afterEach` pair in a `/lib` test — never `beforeAll`. `vi.useFakeTimers()` patches `Date`/`setTimeout`/etc. but does NOT patch `Temporal.Now`; the `clock.now()` seam is the durable answer for all domain "now" reads. For intervals, use `vi.advanceTimersByTime(N)` or `vi.runOnlyPendingTimers()` — never `vi.runAllTimers()`. Reset module mock state between tests: `afterEach(() => vi.resetAllMocks())`; restore spies with `afterEach(() => vi.restoreAllMocks())` (`vi.spyOn` wraps the real method and `restoreAllMocks` unwraps it; `resetAllMocks` does not). Spy for one random value (`vi.spyOn(Math, 'random').mockReturnValue(v)` + `vi.restoreAllMocks()` in teardown); use seam + seed for a stream of values. A `no-restricted-syntax` lint rule bans `Date.now()` and `new Date()` (no args) outside test files and the seam modules — structural enforcement over code-review vigilance.

**Misc.** — `src/test/clock.ts` is now fully established (L2 planted it as a crumb): exports `FROZEN = Temporal.Instant.from('2026-01-15T12:00:00Z')` and `freezeClock(instant = FROZEN): void` using `vi.spyOn(clock, 'now').mockReturnValue(instant)`. `lib/ids.ts` wraps `uuidv7()` (not `crypto.randomUUID()`), consistent with the UUIDv7 project standard from Unit 5. `lib/random.ts` seam wires `xoroshiro128plus` from `pure-rand` (seeded) on the test side. The `src/test/` support tree is now fully populated for this chapter: `factories/`, `matchers/`, `fixtures/`, `clock.ts`. The `buildIdempotencyKey` exercise uses three positional params (not an options object) deliberately as a pedagogical simplification — production convention would use an options object for the third dep.

## Lesson 5 — Async tests without the forgotten-await trap

**Taught** — The forgotten-`await` silent-pass mechanism, the canonical `await expect(p).resolves` / `.rejects` async assertion form, `expect.assertions(n)` for branchy `try`/`catch` error tests, the microtask/macrotask queue split that breaks sync fake-timer advancement on awaited paths, `vi.advanceTimersByTimeAsync` and the full `*Async` timer family, `await Promise.resolve()` as a scalpel for bare-microtask flushes, cancellation testing with `AbortSignal`, `Promise.allSettled` for parallel-effect assertions, and `try`/`finally` + `afterEach(() => vi.useRealTimers())` cleanup.

**Cut** — `it.concurrent` and the context-`expect` attribution requirement were addressed only as a brief aside note, not a taught pattern. `DeepPartial` and library-DSL timers are not mentioned. The chapter outline listed Vitest 4 awaited-assertion enforcement as a teachable change; the lesson frames it as background safety net, not the lesson's core.

**Debts** — Error-shape depth deferred to Lesson 6: `toThrow(Class)` vs message, `toMatchObject({ ok: false, error: { code } })`, custom `toBeOkResult`/`toBeErrResult`, Zod issue inspection, `Error.cause` chains. Integration-level async (MSW, Drizzle `withRollback`) deferred to Ch088. Server Action async testing deferred to Ch088 L7. Playwright auto-waiting deferred to Ch090.

**Terminology** — `silent pass` / `false negative` (green test against broken code; term reused from L2); `macrotask` (a `setTimeout`/`setInterval` callback run by the timer queue; term introduced in L3, deepened here); `microtask` (a promise continuation drained to empty before the next macrotask); `event loop` (scheduler alternating microtask drain and next macrotask); `dangling assertion` (`.resolves`/`.rejects` chain built but never awaited — the silent-pass mechanism).

**Patterns and best practices** — Canonical async test shape: `it('...', async () => { await expect(p).resolves.toEqual(...); })`. Prefer `await` over `return` for async assertions (composes across multiple assertions). Default every async `it` callback to `async` even before an `await` is needed. Use `expect.assertions(n)` only for branchy `try`/`catch` tests with multiple fields worth asserting; use `await expect(p).rejects.toThrow(Class)` for single-field error tests. Any fake-timer test on an awaited path uses the `*Async` variant (`vi.advanceTimersByTimeAsync`, `vi.runOnlyPendingTimersAsync`; avoid `vi.runAllTimersAsync` on self-rescheduling intervals). `await Promise.resolve()` is a scalpel for bare `queueMicrotask`/`.then` code with no real promise to await — not a general retry mechanism. Cancellation paths (`AbortSignal`) and parallel-effect outcomes (`Promise.allSettled`) are behaviors that earn assertions. `afterEach(() => vi.useRealTimers())` is non-negotiable when fake timers are used in async tests. Release resources in `try`/`finally` so a thrown assertion cannot leave an open `AbortController` or interval. A long `testTimeout` is a smell, not a fix.

**Misc.** — Code examples use invented illustrative `/lib` functions (`refundCharge`, `fetchInvoice`, `withRetry`) — not continuations of earlier project files. `expect.fail('...')` inside a `try` block guards the no-throw path; this pattern is previewed here but the full unhappy-path toolkit (including `expect.fail` in sync tests) belongs to L6. `it.concurrent` is deliberately left off by default because concurrent tests race when sharing fake timers or module mocks.

## Lesson 6 — Asserting the unhappy path

**Taught** — Two-path rule applied mechanically (every documented failure earns its own `it` in the same `describe`); `toThrow` surface (loose→tight ladder); arrow-function trap; "class and code over message" as the lesson's central thesis; `try`/`catch` + `expect.fail` for multi-field thrown-error inspection; `toMatchObject` vs `toEqual` on `Result.err`; `expect.any` / `expect.objectContaining` asymmetric matchers; `toBeOkResult` / `toBeErrResult` custom matcher implementation (pays off L1 debt); Zod 4 `safeParse` issue assertion via `toContainEqual` + `objectContaining` on `path`/`code`; `it.each` table for the Postgres error mapper including the unknown→`'internal'` fallback row; `Error.cause` chain assertion with the tautology-trap warning; `not.toThrow` and `it.fails` (legitimate use + rot risk); audit-for-missing-failure-test as a pre-PR reflex.

**Cut** — Chapter outline mentioned asserting on a structured error `code` via regex (`toThrow(/INVALID_EMAIL/)`) alongside class assertion; the lesson instead demonstrates `toThrow(ValidationError)` + `toMatchObject({ code: 'validation' })` in the `try/catch` shape, not the regex form. The outline named `err.code` as `'NOT_FOUND'` (uppercase); the lesson canonically uses lowercase-snake codes (`'not_found'`, `'conflict'`) consistent with the course's `Result` conventions.

**Debts** — Integration-level error testing (DB serialization failures, MSW failure responses, RFC 9457 bodies) deferred to Ch088. Server Action async error testing deferred to Ch088 L7. Sentry capture in tests deferred to Ch092. Error rendering in the UI not covered here.

**Terminology** — `two-path rule` (every documented failure gets its own `it`; introduced Ch086 L4, applied mechanically here); `arrow-function trap` (passing `fn(bad)` instead of `() => fn(bad)` to `toThrow` — throw escapes before `expect` runs); `toMatchObject` (partial deep match — asserts named fields, ignores others; contrasted with `toEqual`'s exhaustive match); `toContainEqual` (asserts array membership by deep equality — used for Zod `issues`); `expect.fail` (Chai assertion surfaced by Vitest; throws immediately; used as guard in `try`/`catch` to catch a missing throw); `it.fails` (inverts test verdict; green only when the assertions fail — use only with a tracking-issue reference and a removal plan); `ZodError` (structured error from a failed `safeParse`; its `issues` array carries `code`, `path`, `format` per issue); Zod 4 `invalid_format` code (replaces Zod 3 `invalid_string` + `validation: 'email'`).

**Patterns and best practices** — `toThrow` always takes an arrow: `expect(() => fn(bad)).toThrow(Class)`. Assert class and structured code (`toBeInstanceOf` + `toMatchObject({ code })`) in the `try/catch` shape; never pin `message` strings. Use `toMatchObject` over `toEqual` for `Result.err` assertions. `expect.fail('expected fn to throw')` is mandatory on the line after the call in a `try/catch` throw test — without it, a function that stops throwing silently passes. For Zod issues, `toContainEqual(expect.objectContaining({ path, code }))` — never index positionally (`issues[0]`) or assert `message`. For the Postgres error mapper and any pure lookup: `it.each` object form with one row per documented mapping plus an explicit row for the unknown→default fallback. Assert the `Error.cause` chain to catch silent observability regressions; the test's subject is the *wrap* (outer class + cause type), not a pass-through. Custom matchers (`toBeOkResult`, `toBeErrResult`) live at `src/test/matchers/result.ts` — promote when the same assertion appears ≥3 times.

**Misc.** — `toBeOkResult` / `toBeErrResult` implementation is now fully taught; TypeScript declaration-merging to augment Vitest's `Assertion` interface is noted but not deep-dived. Zod 4 issue code enum used throughout: `invalid_type`, `too_small`, `too_big`, `invalid_format`, `not_multiple_of`, `unrecognized_keys`, `invalid_value`, `invalid_union`, `custom`. The in-page `ScriptCoding` sandbox is shim-constrained (no `toMatchObject`, no `rejects`, no `objectContaining`) — the exercise uses only `toThrow(ValidationError)` and `.not.toThrow()`; the full real-Vitest matchers are taught in prose only. The lesson draws the L4/L6 boundary explicitly: L4 proves the type system forces narrowing; L6 proves real `err` values carry the right `code`. Async error mechanics (`rejects`, `expect.assertions`) are not re-taught; the lesson refers back to L5.

## Lesson 4 — Type-level tests with expectTypeOf

**Taught** — Two-surface mental model (runtime vs. type), `*.test-d.ts` colocation, `vitest --typecheck` / `vitest run --typecheck` invocation, `typecheck.include` config slice, the full live `expectTypeOf` matcher family, `assertType` as a one-off alternative, and applying those matchers to four domain targets: discriminated unions, branded IDs, `Result<T>` contract, and generic inference.

**Cut** — Chapter outline listed `toMatchTypeOf` and `toBeAssignableTo` as first-class matchers; both are deprecated in Vitest 4 and were not taught (replaced by `toExtend`, deprecated names mentioned once in passing). The outline's `Result<T, E>` two-parameter variant was not used; the lesson canonically uses single-parameter `Result<T>`. The `err()` constructor type-pinning was covered only implicitly (via `@ts-expect-error` on unnarrowed access); no standalone `expectTypeOf(err(...))` assertion block. Drizzle `InferSelectModel` type test reduced to a one-sentence reference with no dedicated code block.

**Debts** — Lesson 6 owns unhappy-path value assertions on `Result.err` (the lesson draws the boundary explicitly: L4 proves the type system *forces* narrowing before reading `data`; L6 proves a real `err` result carries the right `code` and message). Ch088 forward-crumbed as the place `*.test-d.ts` type tests on factory return shapes will appear.

**Terminology** — `type-level test` (assertion checked by tsc, not the runner; file never executed); `*.test-d.ts` (Vitest type-test file suffix; runtime runner skips, typecheck pass picks up); `tsc --noEmit` (type-check only, no JS output); `bidirectional equality` (`toEqualTypeOf`: A assignable to B AND B assignable to A — catches widening); `toExtend` (one-way assignability; replaces deprecated `toMatchTypeOf`); `structural typing` (TS compares shapes not names — why brands need a phantom field); `phantom field` (`string & { readonly __brand: Name }` — type-system-only field giving a branded type a distinct shape).

**Patterns and best practices** — `*.test-d.ts` files colocated alongside `name.ts` and `name.test.ts`. `vitest run --typecheck` in CI; opt-in locally. `typecheck.include: ['src/**/*.test-d.ts']` in `vitest.config.ts`. Matcher decision: exact type → `toEqualTypeOf`; is-a/subtype OK → `toExtend`; object subset → `toMatchObjectType`; must-not → `.not.*`; generic leaking `any` → `.not.toBeAny()` canary; drill into function/array → `.parameters` / `.returns` / `.items` first. For branded IDs, both assertions are *negative* (brand bugs are bugs of absence): `not.toEqualTypeOf<string>()` and `not.toExtend<OtherId>()`. For discriminated unions, use `toEqualTypeOf` not `toExtend` — a widened union still satisfies `toExtend`. Always add `.not.toBeAny()` canary when pinning generic inference. `expectTypeOf(value)` with nothing chained is a silent no-op. An empty `*.test-d.ts` is always green — confirm coverage by file review. Never import a `*.test-d.ts` from a runtime test.

**Misc.** — `"test:types": "vitest run --typecheck"` is an established `package.json` script. The L4/L6 boundary is explicitly drawn in the lesson prose: same `Result`, two different surfaces — do not blur. `@ts-expect-error` polarity: the directive itself errors ("unused") when the guarded line starts to type-check — a should-fail line being corrected into passing is caught automatically, making `.not` assertions self-enforcing in exercises. The lesson's navigator example uses `formatMoney(money: Money, currency: Currency): string` — a two-parameter signature that contradicts L1's established `formatMoney(cents: number): string`; this was an illustrative snippet only (the `Money`/`Currency` types are not defined anywhere in the lesson), not a redefinition of the production signature. L1's `formatMoney(cents: number): string` remains canonical for project code.
