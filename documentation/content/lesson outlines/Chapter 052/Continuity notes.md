# Chapter 052 — Better Auth setup

## Lesson 1 — Wiring the auth instance

**Taught** — The minimum Better Auth surface: install `better-auth` (one package, sub-paths, no peer-dep dance); server `auth` instance in `src/lib/auth.ts` (`server-only` first line, `betterAuth({ database: drizzleAdapter(db, { provider: 'pg' }), plugins: [nextCookies()], secret, baseURL })`); catch-all route `src/app/api/auth/[...all]/route.ts` (`export const { GET, POST } = toNextJsHandler(auth)`); browser client `src/lib/auth-client.ts` (`export const authClient = createAuthClient()`, bare/same-origin); two env vars added to `src/env.ts` server block; the server-vs-client call split (`auth.api.*` in-process vs `authClient.*` over HTTP, both reach the same `[...all]` route); a `GET /api/auth/session` → `null` smoke test. Instance is deliberately feature-less.

**Debts** — Deferred forward: Drizzle adapter internals + four tables (`user`/`session`/`account`/`verification`) + CLI generate + migration (L2); `session`/`advanced` cookie config + `__Host-` `cookiePrefix` + `SESSION_COOKIE_PREFIX` export (L3); `auth.api.getSession({ headers })` call shape + `React.cache`-wrapped `getCurrentUser`/`requireUser` + minimum `proxy.ts` gate (L4); `emailAndPassword`/`socialProviders`/`emailVerification`/`account` config + full `authClient` method surface + `useSession` (Ch 053); `trustedOrigins` config + secret rotation ops (named only). Referred back: `__Host-` opaque session cookie (Ch 051 L2); `server-only` guard (Ch 030, same as `db/index.ts`); validated env via `@t3-oss/env-nextjs`+Zod (Ch 034); catch-all segment (Ch 029); `db` client (Ch 036); Server Actions/`proxy.ts` (Ch 033).

**Terminology** — "two faces / two ways into one instance": server instance vs browser client, bridged by one catch-all endpoint, split by where the caller runs. Slogan to carry: **"the server reads identity with `auth.api`; the browser triggers auth actions with `authClient`."** "surface area, not feature count." `null` smoke-test result framed as green light ("plumbing connected; Ch 053 turns on the water"). The illustrative mirror pair `auth.api.signUpEmail` ↔ `authClient.signUp.email` (both hit `POST /api/auth/sign-up/email`).

**Patterns and best practices** — `auth.ts` leads with `import 'server-only';`; `auth-client.ts` has neither `server-only` nor `client-only` (plain module). **`nextCookies()` is non-negotiable and must be LAST in the `plugins` array** (later plugins e.g. organizations sit before it) — omitting it = silent "I called signUp and nothing happened" (200, session row, no cookie reaches browser); surfaces in Ch 053. `secret`/`baseURL` passed explicitly from validated `env` (not auto-read magic). Both env vars in `server` block, never `NEXT_PUBLIC_*`; `BETTER_AUTH_SECRET` is `z.string().min(32)` (`openssl rand -base64 32`), `BETTER_AUTH_URL` is `z.string().url()`. Per-environment secrets via Vercel project envs (split from day one). `createAuthClient()` bare — omit `baseURL` for same-origin. Never hand-write per-feature route files — the catch-all is the contract; keep it at `/api/auth`.

Corrections applied vs chapter outline: smoke test is `GET /api/auth/session` (NOT `/api/auth/ok` — unverified); secret gen is `openssl rand -base64 32` (do NOT assert `npx @better-auth/cli secret`); `createAuthClient()` bare (outline's `baseURL: process.env.NEXT_PUBLIC_APP_URL` rejected); env lives in `src/env.ts` not `lib/env.ts`. Growth-schedule table for `BetterAuthOptions` shown to defuse "is this all?" anxiety. ## Lesson 2 — Schema and the four core tables

**Taught** — Connects Better Auth to Postgres and installs the auth data model. The schema-aware adapter delta on `src/lib/auth.ts` (deferred from L1): add `import * as schema from '@/db/schema'` (namespace import) and pass `schema` → `drizzleAdapter(db, { provider: 'pg', schema })` so the adapter resolves tables by reference, not name-guessing. Generate-don't-handwrite reflex: `npx @better-auth/cli generate --config src/lib/auth.ts --output src/db/schema/auth.ts` writes a normal, reviewable Drizzle schema file living beside `customers.ts`/`invoices.ts` in `src/db/schema/`. The four tables column-by-column (real CLI output, lightly trimmed): `user` (id text PK, `email` text unique+notNull DB-enforced, `emailVerified` boolean default false, `name`, nullable `image`, timestamps — **no password column**); `session` (`userId` FK cascade, `token` text unique = the opaque cookie session-id from Ch 051, `expiresAt`, nullable `ipAddress`/`userAgent`); `account` (`userId` FK cascade, `providerId` text discriminator = `'credential'`/`'google'`/etc., `accountId`, **nullable `password`** hash, grouped nullable OAuth-token columns); `verification` (`identifier`, `value`, `expiresAt` — **no FK to user**). FK cascade consolidation (`onDelete: 'cascade'` on session+account `userId`, none on verification). First auth migration via existing Drizzle Kit pipeline: `pnpm drizzle-kit generate --name add_auth_tables` then `pnpm drizzle-kit migrate`, review the SQL before applying.

**Cut** — Dropped from chapter-outline scope (the later-dependency-relevant ones): `additionalFields`/extending `user` with app columns (`role`/`timezone`/`locale`) — fully cut here, arrives with the features needing it (Ch 053+, Unit 9, Ch 083/084); custom index tuning beyond CLI output (non-unique `session(userId)`, composite `account(providerId, accountId)`) — cut, deferred to Ch 039 discipline / Ch 054 read; field/table-name remapping (`modelName`/`fields`) — cut to a non-configuring dismissal. `usePlural` named-and-dismissed once (project uses singular default).

**Debts** — Deferred forward: session lifetime config (`expiresIn`/`updateAge`/`freshAge`) + cookie hardening (L3); request-time session read (L4); email/password + OAuth sign-up/sign-in flows that finally write `user`/`account`/`session` rows, password hashing internals, account-linking/unlinking UI, `useSession` (Ch 053); `organization`/`member`/`invitation` + `passkey` tables (named as regenerate triggers only — Unit 9 / Ch 053); active-sessions UI rendering `ipAddress`/`userAgent` (Ch 054); expand-migrate-contract for live schema changes + CI-run migrations (Ch 097/099, one-line forward-pointers). Referred back: `casing: 'snake_case'` on the Drizzle client (Ch 037 — TS `emailVerified` ↔ SQL `email_verified`); UUIDv7 id convention via `$defaultFn` (Ch 037); `db` client built by passing `schema` to `drizzle()` (Ch 036); FK `onDelete` (Ch 037); Drizzle Kit generate/migrate + review-before-apply (Ch 040); opaque-session-token-in-cookie (Ch 051 L2).

**Terminology** — The lesson's spine: **"your identity is one row; every way you can prove that identity is a separate row"** — i.e. **`user` answers *who you are*; `account` answers *how you prove it*.** Workflow slogan: **"generate the schema, review the diff, migrate"** — never hand-author auth tables. Hard one-tool rule: **"Generate with Better Auth's CLI. Migrate with Drizzle Kit. Always."** (`@better-auth/cli migrate` exists but this stack never uses it — would fragment migration history.) `providerId` called the **discriminator**. The CLI framed as "a code generator you review, never an ORM you trust blindly." `verification`'s lack of a FK explained as "a verification row sometimes needs to exist *before* the user does."

**Patterns and best practices** — Auth tables ship in the same `src/db/schema/` dir, version control, and Drizzle Kit pipeline as domain tables (schema = source of truth). CLI output consumed as-is, no wrapper. `password` is nullable text on `account`, never on `user`. `email` uniqueness enforced at the DB (unique constraint), not app-level check-then-insert. `session.token` unique index is the per-request hot-path lookup. Cascade (`onDelete: 'cascade'`) is the standard for owned children (sessions, accounts); verification is correctly exempt. Migrations named with `--name`. Better Auth hashes passwords with **scrypt** (note: chapter outline said bcrypt — the lesson says scrypt; do not teach hashing internals). Project keeps Better Auth's singular table names + default id generation.

**Misc.** — Corrections vs chapter outline: password hashing is **scrypt** not bcrypt; the adapter `schema` option was deferred from L1 and landed here as the opening surgical delta (L1 shipped `drizzleAdapter(db, { provider: 'pg' })` only). `--output src/db/schema/auth.ts` is required (CLI defaults to root `schema.ts`). `@/db/schema/index.ts` re-exports every table = what `@/db/schema` resolves to. ## Lesson 3 — Session lifetimes and cookie hardening

**Taught** — Configures the two `betterAuth` option blocks `session` (lifetime) and `advanced` (cookie). The three independent clocks on one `session` row: **`expiresIn`** absolute wall (lib default 7d → **course 30d**, persisted on `session.expiresAt`); **`updateAge`** sliding renewal (1d, course keeps — past it a request pushes `expiresAt` out by another full `expiresIn` window; shorter = write-amplification on hot table); **`freshAge`** elevation window measured from `createdAt` not last activity (lib default 1d → **course 10 min**; this lesson only sets the value, the enforcing check lives in Ch 054). `disableSessionRefresh` named (recognition). Token rotation: fresh token minted at sign-in (structural defense vs session fixation), renewal can rotate — falls out of the clocks, not configured directly. `advanced` cookie keys: `useSecureCookies` (auto-true in prod/HTTPS, relaxes on localhost), `cookiePrefix` → **`__Host-better-auth`** (browser-enforced contract: requires Secure + Path=/ + no Domain), `defaultCookieAttributes` (`sameSite:'lax'`, `httpOnly:true`, `path:'/'`), `crossSubDomainCookies` (off; **incompatible with `__Host-` by cookie spec**). The cookie cache as the chapter's one performance reach: `session.cookieCache: { enabled:true, maxAge: 60*5 }`, a signed sibling `…session_data` cookie holding a `{user,session}` snapshot → zero DB round-trips inside `maxAge`, falls back to DB past it; read call-shape identical either way. Sign-out = `auth.api.signOut` deletes the `session` row (the row deletion is what makes a stale cookie harmless).

**Cut** — `StateMachineWalker` decision-tree for the deferred reaches was offered as optional in the outline but not built (the four `Card`s suffice). No `freshAge` elevation-flow implementation (correctly deferred to Ch 054 L2).

**Debts** — Forward: `freshAge` elevation re-prompt + action-side re-auth check (Ch 054 L2); `SESSION_COOKIE_PREFIX` consumed by `getSessionCookie` in the proxy (Ch 052 L4 — **the one forward artifact this lesson ships**); read path `getSession`/`getCurrentUser`/proxy (L4); active-sessions list + "sign out everywhere" UI + per-session revocation (Ch 054 L3, named); `revokeOtherSessions:true` on credential mutations (principle stated, wired in Ch 053/054); authz at the action boundary + `authedAction` (Unit 9/Ch 057, principle only); `secondaryStorage`/Redis (Ch 074, alongside rate limiting); `jwt()` plugin (when a 2nd service exists), `multiSession()` (Ch 054 L3), `trustedOrigins` (separate-domain client) — all recognition-only with named triggers. Referred back: opaque-token / `__Host-` / `SameSite` / server-side revocation / session fixation decided conceptually (Ch 051 L2); `nextCookies` plugin clears cookie on sign-out (Ch 052 L1); `additionalFields` extension hook (Ch 052 L2 — custom user fields absent from cache unless declared).

**Terminology** — **"Three clocks, not one"** — the central anti-trap framing (beginners collapse expiry/renewal/freshness into one timeout); each clock answers a distinct question (`expiresIn`="valid at all?", `updateAge`="extend because user's here?", `freshAge`="trust it for something destructive?"). **`freshAge` is creation-relative, NOT activity-relative** (course 10 min — do not "correct" to lib 1d). `__Host-` framed as a **browser-enforced contract, not a Better Auth feature**. `crossSubDomainCookies ⊥ __Host-` is a **cookie-spec fact** (Domain required vs forbidden). Cookie cache `strategy` default = `compact` (`jwt`/`jwe` exist, recognition only). "Bearer credential" and "write amplification" given `Term` tooltips. **`SESSION_COOKIE_PREFIX`** = the single source-of-truth constant feeding both `advanced.cookiePrefix` and the proxy's `getSessionCookie`.

**Patterns and best practices** — Course config **deliberately diverges from Better Auth library defaults** (`expiresIn` 30d, `freshAge` 10m, `__Host-` prefix) — always make the gap explicit, never present as out-of-the-box. Lifetime values written as unit math (`60*60*24*30`) not magic seconds. `SESSION_COOKIE_PREFIX` is **environment-aware**: `process.env.NODE_ENV === 'production' ? '__Host-better-auth' : 'better-auth'` — a `__Host-` cookie cannot set over `http://localhost` (needs Secure), so dev relaxes the prefix; declare the prefix once, export it, never restate the literal in the proxy (drift → `getSessionCookie` silently misses the cookie → signed-in user looks signed-out). Two non-negotiable senior rules from the cache trade: **authorization re-checked against the DB at the action boundary, never in the proxy** (proxy does cookie-presence gating only); **`revokeOtherSessions:true` on credential-mutating actions**. `useSecureCookies` left unset (auto-resolves per env). Never `trustedOrigins: ['*']`. Never replace the browser session cookie with a JWT (server-side revocation is non-negotiable). Cookie cache: on for read-heavy identity-aware pages, off/short `maxAge` for revocation-sensitive surfaces (admin/billing).

**Misc.** — Decision-archetype lesson, config-only, deliberately **read-path-free** (no `getSession`/helpers/proxy — that's L4). Where chapter outline and Code conventions disagree, **Code conventions wins** (esp. `freshAge` 10 min). Two reference comparison tables (the clocks knob-table; cache on-vs-off) plus a hand-built HTML+CSS three-clocks timeline as the load-bearing centerpiece. ## Lesson 4 — Reading the session everywhere with one call shape

**Taught** — The single request-time session read and the two helpers every downstream chapter calls. The one call shape, byte-identical everywhere: `const session = await auth.api.getSession({ headers: await headers() });` → `Promise<{ user, session } | null>`, where `null` = anonymous (no valid cookie), never error; `await headers()` (async in Next 16) is the load-bearing arg / #1 silent failure; call is in-process, reads cookie cache when warm else DB, shape identical either way. The five surfaces, one call, different `null` tails: layout/Server Component → render signed-out variant or `requireUser()` redirect; Server Action → `err('unauthorized', …)` Result discriminant at the top before any write; route handler → 401 Problem Details; Client Component → receives a server-read user as a **prop** (never reads auth itself); proxy → the deliberate exception (cookie-presence only, never `getSession`). `React.cache` vs `'use cache'` as the lesson's loudest guardrail: session read is request-scoped → wrap in `cache()` from `react`; `'use cache'` is cross-request and keys on args/captured values (not the cookie) → serves user A's session to user B = account-takeover bug. The two helpers shipped into `src/lib/auth.ts` (additions to the L1 file): private `getSession = cache(async () => auth.api.getSession({ headers: await headers() }))` (the ONLY direct call in the codebase), `getCurrentUser(): Promise<User | null>` (`session?.user ?? null`), `requireUser(next?): Promise<User>` (redirects `/sign-in?next=<encoded>` on null, else returns guaranteed user); `type User = typeof auth.$Infer.Session.user`. The minimum `src/proxy.ts`: `getSessionCookie(request, { cookiePrefix: SESSION_COOKIE_PREFIX })` presence check → `NextResponse.redirect('/sign-in')` if absent else `NextResponse.next()`, `matcher: ['/dashboard/:path*']`. Client/server split: `authClient.useSession()` → `{ data, isPending, error }`, reactive/refetch-on-focus, **display only**; client reads drive UI, server reads drive decisions; can't call `auth.api` from client (server-only import + browser doesn't pass own cookies); serializable ≠ safe to send — hand the client a trimmed `{ id, name }`, never the session row/token.

**Cut** — Outline's `databaseHooks` recognition mention dropped entirely (belongs with org plugin, Ch 056 L1 — do not introduce before then). Outline's three-layer perf ladder (`React.cache` → cookie cache → `secondaryStorage`/Redis) compressed to the request-scope dedupe point only; Redis layer not restated (lives in L3 debt + Ch 074).

**Debts** — Forward: full `proxy.ts` matcher / multi-route protected surface (Ch 054 L1 — current matcher is `/dashboard/:path*` only); `authedAction` wrapper + role/org authz that layers on top of the identity read (Ch 057 / Unit 9); **`requireOrgUser(role?)`** named at recognition level as the third helper arriving with the org plugin (Unit 9 — NOT built here, no org plugin yet); active-sessions list / sign-out-everywhere / per-session revocation (Ch 054 L3); `freshAge` elevation read at the action boundary (Ch 054 L2); 401-vs-403 wire depth (Ch 046 / Ch 054 L4); `useSession` in depth + client sign-in patterns (Ch 053); CSRF/SameSite surface (Ch 054 L4); sign-in/OAuth flows that actually create sessions (Ch 053 — this lesson assumes a session may or may not exist, never creates one). Referred back: cookie cache + `SESSION_COOKIE_PREFIX` export (Ch 052 L3 — consumed here); `auth` instance + server/client split + `authClient` (L1); `Result` type `err('unauthorized', …)` (Ch 043); Problem Details 401 (Ch 046); `proxy` rename + `matcher` (Ch 033); `React.cache` / `'use cache'` / dynamic-by-default + PPR (Ch 032); `lib/db.ts` thin-wrapper analogy (Unit 5 / Ch 036); opaque session token (Ch 051).

**Terminology** — **"One call, many tails"** / "memorize the head, vary the tail" — the spine framing (one read shape; only the `null` reaction differs per surface). **"client reads drive UI; server reads drive decisions"** — the client/server hard line. **"serializable does not mean safe to send"** — the wire-leak rule for props to Client Components. The proxy as **"a bouncer who checks whether you're holding a ticket, not whether the ticket is genuine"** — optimistic UX redirect, not a security boundary. `requireUser` = the **assertive read** (never returns on null — `redirect` throws to the framework); `getCurrentUser` = the **safe read**. "request-scoped" (`React.cache`) vs "cross-request" (`'use cache'`) the named axis.

**Patterns and best practices** — Exactly **one** place calls `auth.api.getSession` directly (the private `cache()`-wrapped `getSession` in `lib/auth.ts`); pages/layouts/actions/route handlers ALWAYS go through `getCurrentUser`/`requireUser` — the raw call skips per-request dedupe (Auth conventions: "no parallel `getSession` calls anywhere"). Session reads use `React.cache`, **never** `'use cache'` (per-user data in `'use cache'` = cross-user leak, non-negotiable). Read the session at the **highest layout where the gate belongs**, not in every leaf; a session-reading layout opts its subtree into dynamic rendering (intended, lean on PPR). Server Actions read identity at the very top before any write. The proxy does **cookie-presence gating only** — authorization is re-checked against the DB at the action boundary, never the proxy (cookie cache staleness window makes proxy reads unsafe as a security boundary). `getSessionCookie` MUST receive `{ cookiePrefix: SESSION_COOKIE_PREFIX }` — its `better-auth` default silently misses the prod `__Host-` cookie → signed-in user bounced to `/sign-in` in production only (the canonical drift bug). Client Components get a trimmed `{ id, name }`, never the session row/token. `User` type derived via `auth.$Infer.Session.user` (tracks schema, no hand-written interface). `requireUser` threads `next` via `encodeURIComponent` for post-sign-in return. Helpers live in the L1 `lib/auth.ts` (already `import 'server-only';`).

Result discriminant is `'unauthorized'` (the lesson MDX uses `err('unauthorized', …)`; outline prose said `unauthenticated` — `unauthorized` shipped).
