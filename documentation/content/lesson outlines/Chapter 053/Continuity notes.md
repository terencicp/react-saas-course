# Chapter 053 — Authentication flows

## Lesson 1 — Password sign-up

**Taught** — Turns the inert Ch 052 `auth` instance into a working email+password sign-up. The `emailAndPassword` block added to `betterAuth({...})` in `lib/auth.ts`: `enabled: true`, **`requireEmailVerification: true`** (the hinge — flips success from "session issued" to "verification email queued, no session"), `autoSignIn: false` (paired with verification; `autoSignIn:true`+`requireEmailVerification:true` = canonical broken state), `minPasswordLength: 12` (lib floor 8 → course 12; `maxPasswordLength` 128 default). **The library owns the hash** — student writes no hashing call; Better Auth hashes with **scrypt** (Node-native, memory-hard) + per-password salt, writes to `account.password` (the `account` row, `providerId:'credential'`, never `user`). Override door `emailAndPassword:{ password:{ hash, verify } }` named as measured-only (Argon2id), never day-one. The `signUp` Server Action in the five-seam shape (Ch 043 `parse→authorize→mutate→revalidate→return`): Zod `signUpSchema` (`name.min(1)`, `email: z.string().trim().toLowerCase().pipe(z.email())`, `password.min(12)`); **authorize seam intentionally empty** (public endpoint, no caller); mutate = `auth.api.signUpEmail({ body, headers: await headers() })` in `try/catch` (no `db.transaction` — library owns writes); revalidate empty (nothing cached changes); return `ok({ email })` (threads email to the success page) or `mapSignUpError(error)`. **Two call faces of one instance**: client `authClient.signUp.email(...)` returns `{ data, error }`; server `auth.api.signUpEmail(...)` **throws `APIError`** on a *genuine* failure (opposite shape) — same codes on `error.body.code`, `error.status` like `'UNPROCESSABLE_ENTITY'` (422). **User-enumeration defense** (the decision insert) — *reconciled at build, the pre-build model was wrong*: with `requireEmailVerification:true` (or `autoSignIn:false`), `auth.api.signUpEmail` on a **taken email DOES NOT throw `USER_ALREADY_EXISTS`** — it returns the **same generic success** as a fresh sign-up (and the library hashes a throwaway password it never stores, so the two paths *time-match* and can't be a side-channel). The defense is closed **at the source**, not by the `catch`. `mapSignUpError` therefore only ever sees genuinely-thrown, non-enumerating failures (`PASSWORD_TOO_SHORT`, `INVALID_EMAIL`) → distinct codes; the safe-variant ships **no `USER_ALREADY_EXISTS` branch at all** (a comment names *why* the branch is absent). **`onExistingUserSignUp`** is the dedicated seam (a callback on `emailAndPassword`) that fires server-side, off the response path, **only when this protection is active** — it's how the *real* owner gets a "did you try to sign up?" heads-up while the response stays generic. `additionalFields` hook (`user.additionalFields`, sibling of `emailAndPassword`): the **`input` flag is a security boundary** — `input:true` = user-settable on sign-up body, `input:false` = server/admin-only (canonical `role`); senior call = keep sign-up minimal (email/password/name), defer rest to onboarding. Strength-meter trust line: client `zxcvbn-ts` = encouragement (entropy, UX), server `minPasswordLength` = enforcement; meter score never sent/trusted server-side. Success state = "check your inbox" view (no session, no `/dashboard`), echoes the typed email, **rate-limited** resend `authClient.sendVerificationEmail({ email })`. **Three rows exist post-sign-up**: `user` (`emailVerified:false`), `account` (`credential`, scrypt hash), `verification` (keyed by email); no `session`.

**Cut** — Strength-meter section shipped as full prose (not the `<details>` the outline offered) but still build-free (no widget). HaveIBeenPwned k-anonymity check named only. `companyName`/`timezone` shown as deferred-to-onboarding examples, not collected. Argon2 cost parameters deliberately not taught (outline's "Argon2id memory 64MiB" is wrong — scrypt is the default; correction honored).

**Debts** — Forward: **verification token shape, the React Email template, `sendVerificationEmail` callback body, the verify-click endpoint, `autoSignInAfterVerification`** (L3 — this lesson stops at "email queued", names `sendVerificationEmail` only as the seam the email rides); sign-in + failed-attempt counter + full `Result` catalog (`invalid-credentials`/`email-not-verified`/`too-many-attempts`/`requires-second-factor`) + `rememberMe` + session rotation (L2 — L2 reads the `emailVerified` this lesson wrote false); L3 consumes the `verification` row; password reset (L4), magic links (L5), TOTP (L6), passkeys (L7), OAuth (L8), account linking (L9); change-password/email from settings (Ch 054 L2); **full rate-limit wiring** (per-IP + per-email dual-key, `safeLimit`, `RateLimit-*`) named at the resend call site only (Ch 074 L3). **Terminology** — **`requireEmailVerification:true` is "the hinge"** — one boolean from which the success page, the no-session state, AND the enumeration defense all fall out ("two birds, one flag": the defense and the calm UX are the *same* config, not separately wired). **"the library owns the hash" / "you don't hash anything"** — the section's whole takeaway. **`APIError`** = Better Auth's typed throw from every `auth.api.*` call on a *genuine* failure, carries `body.code` (e.g. `PASSWORD_TOO_SHORT`) + `status` (e.g. `UNPROCESSABLE_ENTITY`); client face returns `{ data, error }` instead. **"closed at the source"** — under this config the taken-email path is silenced *inside the library* (generic success, time-matched), so the `catch` never receives existence info; the senior touch is the **comment explaining the absent branch**. **`onExistingUserSignUp`** = the out-of-band seam to notify the real owner without leaking on the response. **`input` flag = the wall between "a field the form collects" and "a field the app assigns."** "client meter = encouragement; server `minPasswordLength` = enforcement" (the Ch 043/044 client-UX-vs-server-correctness thread). NB for later lessons: `USER_ALREADY_EXISTS` does **not** reach a sign-up `catch` under verification-on; it exists in `$ERROR_CODES` and may surface on *other* paths — ground exact strings there, don't hardcode.

**Patterns and best practices** — Email **normalized before validated**: `.trim().toLowerCase()` runs *before* `.pipe(z.email())` so `Ada@acme.com`/`ada@acme.com` collapse to one canonical address and a stray space won't fail format (else both slip past the unique index = one human, two accounts). `requireEmailVerification:true` **always** paired with `autoSignIn:false`. Sign-up's authorize seam is the one legitimately-empty seam — name it as deliberate, not forgotten. Enumeration safety is a **whole-path property** (endpoint→action→pixels): config closes it at the source, but the form must *also* not branch on a distinct "email-taken" state, AND a hand-rolled "is this email registered?" pre-check before `signUpEmail` rebuilds the oracle — any of these leaking re-opens it. To warn the real owner, use **`onExistingUserSignUp`** (off the response path), never a response branch. The friendly "already registered, sign in" message is a **legitimate deliberate trade** (support-ticket savings vs harvesting risk) — decide on purpose, default to opaque. `role` MUST be `input:false` (self-set role at sign-up = instant admin). Resend endpoint is rate-limited by design (protection exists at the call site from day one). Keep sign-up to email+password+name; defer everything else to onboarding (highest-drop-off moment). Course config diverges from lib defaults (`minPasswordLength` 12, verification on, auto-sign-in off) — present as deliberate, not out-of-the-box.

## Lesson 2 — Password sign-in

**Taught** — The `signIn` Server Action, twin of L1's `signUp`, in the same five-seam shape: reads the rows L1 wrote, issues a session on success. `signInSchema` = `email` normalized then validated (`.trim().toLowerCase().pipe(z.email())`), **`password.min(1)` NOT `min(12)`** (sign-in *checks* a password, sign-up *sets* it — re-enforcing the floor locks out users whose password predates the floor), `rememberMe: z.preprocess((v) => v === 'on' || v === true, z.boolean()).default(true)` (HTML checkbox submits `"on"`/nothing, never a bool; **never `z.coerce.boolean()`** — makes `"false"` truthy). Authorize seam intentionally empty (public endpoint, the credential check *is* the authz). Mutate = `result = await auth.api.signInEmail({ body: { email, password, rememberMe }, headers: await headers() })` — **captured in `result`, not awaited-and-discarded** (the L1→L2 change: success has a value to inspect), wrapped `try/catch`. **The core teaching: sign-in is a classifier reading TWO channels, not a boolean.** Better Auth runs a gauntlet of gates — rate limit → resolve email → find `credential` account → verify hash (constant-time) → check `emailVerified` → check 2FA → issue session — and *where the request falls out* names the outcome. Five outcomes split by classification: **Error** = `'invalid-credentials'` (wrong password OR unknown email, same shape, `INVALID_EMAIL_OR_PASSWORD`/401, thrown) + `'too-many-attempts'` (per-IP cap, 429+retry-after, thrown); **Continuation** (not a failure, hands the form a different screen) = `'email-not-verified'` (`EMAIL_NOT_VERIFIED`/403, thrown) + `'requires-second-factor'`; **Success** = `'ok'`. **The structural twist:** four outcomes arrive as a thrown `APIError` caught by `catch`, but **2FA arrives on the SUCCESS path** — `signInEmail` *resolves normally* with `{ twoFactorRedirect: true, twoFactorMethods }` (e.g. `['totp']`) and NO session. Detected with `if ('twoFactorRedirect' in result)` (TS can't infer the field). The success channel carries `type SignInOk = { status: 'signed-in'; redirectTo: string } | { status: 'second-factor'; methods: string[] }` — a discriminated union the form switches on, **distinct from the generic `Result.error.code` failure union from `lib/result.ts`**; `'requires-second-factor'` is NOT an error code, which is why modeling it as one gets you stuck. `mapSignInError(error): Result<never>` (twin of `mapSignUpError`) handles **only thrown failures** (2FA never reaches it): narrows `error instanceof APIError`, maps `INVALID_EMAIL_OR_PASSWORD`→`err('unauthorized', ...)`, `EMAIL_NOT_VERIFIED`→`err('forbidden', ...)`, `error.status === 429`→`err('rate_limited', ...)`, else `err('internal', ...)`; rate-limit keyed on numeric **status not code string** (most version-stable), comment names the enumeration reason two cases collapse. **Brute-force, honest two-layer story:** Layer 1 = per-IP rate limit **Better Auth ships this** (`/sign-in/email` ≈ **3 req/10s**, **on in prod, OFF in dev** — first 429 surprises you in prod); Layer 2 = per-account/per-email limit **you add it, core Better Auth ships NO failed-attempt lockout** (Auth0/Clerk do; Better Auth doesn't) — course wires the dual-key (`per-IP` + `per-email`) limiter in Ch 074. Each layer plugs the other's hole. **`rememberMe` controls cookie persistence ONLY, not server-side session lifetime:** checked = `Max-Age` cookie (survives browser close), unchecked = session-only cookie (memory, cleared on close); **`session.expiresAt` is set the SAME either way** — the server honors the session its full lifetime regardless; unchecking just discards the cookie pointing at it. **Session-token rotation on sign-in** = the Ch 051 L2 fixation defense cashed out at the call site, zero config — `signInEmail` mints a fresh `session.token` every success, killing any pre-planted identifier. **Sign-out** = `authClient.signOut()` / `auth.api.signOut({ headers: await headers() })`, deletes the `session` row + clears cookie; **must be POST never GET** (GET sign-out = CSRF target via `<img src>`/link). **Open-redirect closure on `?next=`:** validate against an allowlist via **`safeNext(url)` from `lib/redirects.ts`** (security baseline in code conventions) — accept only same-site `/...` paths, reject `//` (protocol-relative) and absolute `http(s)://`, else fall back to a safe default; Better Auth validates only ITS OWN redirects via `trustedOrigins`, your surrounding `next` handling is on you.

**Cut** — No per-account lockout *mechanics* taught (it doesn't exist in core; only named as the Ch 074 gap). `freshAge` elevation/step-up not touched (sign-in establishes a fresh session, doesn't check freshness — Ch 054/Unit 9). (The `remember-me-cookies.astro` two-chip figure, marked optional in the plan, *did* ship.)

**Debts** — Forward: **2FA challenge UI / the factor-verification call that consumes `twoFactorMethods`** (L6 — L2 stops at *detecting* `twoFactorRedirect` and returning `{ status:'second-factor', methods }`); **the verify-email re-send screen + `sendOnSignIn:true`** (L3 — L2 stops at *returning* `'email-not-verified'`, names `sendOnSignIn` as the seam that re-sends); **password reset / the "reset link" remedy** to lockout complaints (L4); **full production rate-limit wiring** — `safeLimit`, dual-key per-IP+per-email, `RateLimit-*` headers, Upstash (Ch 074); active-sessions list / revocation, change-password/email from settings (Ch 054 L2–L3). Referred back: `Result<T>`/`ok`/`err` + five-seam shape + `z.flattenError().fieldErrors` (Ch 043); **`useActionState` form that consumes the `Result` — Ch 044 L3** (`/044-forms-the-platform-way/3-useactionstate-pending-state-and-the-result/`, NOT Ch 045; L1 already corrected this mis-citation — the action plugs in unchanged); the two call faces + `APIError` throw semantics + `mapXError` enumeration-collapse + "library owns the hash" posture + full enumeration *threat model* (L1 — referenced, never re-derived); `account`/`credential`/`emailVerified`/`nextCookies()`/`verification` table (Ch 052); session-vs-token model + **session-fixation** threat (Ch 051 L2); FormData/Zod boundary + `z.coerce.boolean()` ban (code conventions).

**Terminology** — **"a sign-in is a classifier, not a boolean"** — the lesson's spine; reads **two channels** (resolved value for the 2FA continuation, thrown error for everything else). **"Continuation" (vs Error vs Success)** — the keystone classification: `'email-not-verified'` and `'requires-second-factor'` *look* like failures but aren't — they route the user to a different screen. **"the gauntlet"** — the ordered gates; *where the request falls out* names the `Result`. **2FA forks off the SUCCESS exit, not a failure exit** — the most-misread fact; `{ twoFactorRedirect: true, twoFactorMethods }` on resolve, narrowed via `if ('twoFactorRedirect' in result)`. **`SignInOk`** = the success-channel discriminated union (`'signed-in'` | `'second-factor'`), kept distinct from the failure-channel `Result.code`. **`safeNext`** = the open-redirect allowlist guard (`lib/redirects.ts`). Color palette inherited from L1: blue=parse, orange=library call/try-catch, green=returns/session, **violet=continuation/hinge**, red=insecure variant.

**Patterns and best practices** — **At sign-in validate `password.min(1)`, never the sign-up strength floor** (re-enforcing locks out legacy-password accounts). **`z.preprocess` for FormData checkboxes, never `z.coerce.boolean()`.** **Key version-volatile error mapping on the numeric HTTP `status` where possible** (429), and read exact code strings off the client's `$ERROR_CODES` not from memory (Better Auth renames them). **`'requires-second-factor'` belongs on the success channel, not the error union** — model continuations as success-shaped, not error codes. **Run BOTH rate-limit layers on auth endpoints** (per-IP from the library + per-email you add) — code conventions mandate the dual-key limiter; **never strip the per-email limit for "lockout complaints"** — answer with clear copy + a visible reset link. **Sign-out is always POST** (a `<form>`+submit or `onClick`, never `<a href>`). **Always route post-sign-in redirects through `safeNext`** — `?next=` is untrusted; reject `//` and absolute URLs.

## Lesson 3 — Email verification

**Taught** — Closes the sign-up→verify→in-app circuit L1/L2 opened. The `emailVerification` block added to `betterAuth({...})` in `lib/auth.ts` (a **sibling of `emailAndPassword`**, not nested), four knobs: **`sendVerificationEmail: async ({ user, url }) => …`** — the seam L1 named, now filled; library hands `{ user, url }` where `url` is the **fully-formed verify link with the token already embedded** (student mints no token, builds no URL); body is one line, `sendEmail({ to: user.email, subject, react: VerifyEmail({ url }) })` (the Unit 7 wrapper). **`sendOnSignIn: true`** (lib default `false`) — pays L2's debt: when an *unverified* user signs in, Better Auth **re-fires `sendVerificationEmail` automatically** (a verified user signing in triggers nothing), which is exactly why L2's `'email-not-verified'` branch can say "we re-sent your link" without a second call. **`autoSignInAfterVerification: true`** (lib default off) — the verify endpoint **issues a session** at click-time so the user lands signed in, no second sign-in trip; senior call **on** for consumer SaaS (click already proved inbox control), **off** for high-stakes (banking/admin) where an explicit sign-in is wanted. **`expiresIn: 60 * 60`** = 1 hour, **agreeing with the lib default** (unlike L1 which diverged) — long enough to read email after lunch, short enough a leaked link is mostly inert; reset (L4) is shorter at 10 min, magic-link (L5) shorter still because the link *is* the credential. **The token / two-secret split** (the conceptual heart): one `verification` row per pending verify (the Ch 052 L2 table — L1 *wrote* it, L3 is its first real *consumer*, reading + deleting), columns `identifier`=email, `value`=**stored hashed token**, `expiresAt`=now+1h. The raw token travels in the link; **`value` is a one-way transform of it** — never the same string. Two properties: (1) **a DB snapshot is inert** (exfiltrated `verification` table yields zero working links — the same password-on-`account` principle from L1, second kind of secret); (2) **guessing is intractable** (CSPRNG, library-chosen entropy). Lookup is **constant-time** (the L2 reflex, different token). Verification token is a **bearer token** → one-time use enforced **by row deletion** (a second click finds nothing), plus expiry-by-`expiresAt` + a periodic cleanup sweep. **The click flow** (library does all of it): the link is `…/api/auth/verify-email?token=<token>&callbackURL=<dest>` hitting the **same `[...all]` catch-all** mounted Ch 052 L1 (no new route); `callbackURL` is the one threaded from L1's `signUp.email({ callbackURL })`. Happy path: hash token → look up row by hash → check `expiresAt` → flip `user.emailVerified` true → delete row → (if `autoSignInAfterVerification`) issue session → redirect to `callbackURL`. **Failure path: all three causes (never-existed / expired / already-used) collapse to one `?error=invalid_token`** redirect param; the only UI the student writes is a small landing page handling success + that one error branch. The `VerifyEmail` template at `emails/verify-email.tsx` (`{ url }` prop, imports from `@react-email/components` consolidated package, React Email anatomy NOT re-taught) — a **transactional email does exactly one job**: one CTA button, a plain-text fallback URL (clients strip buttons), an expiry line; restraint is a *deliverability* decision, not aesthetic. **Two resend doors:** explicit (`authClient.sendVerificationEmail({ email })`, the L1 button's other end) + implicit (`sendOnSignIn`); **only the most recent link works** — each resend mints a new token and supersedes the old (one live bearer token at a time); both sit behind a rate limit (~1/min/email, mechanics deferred to Ch 074). **Enumeration extended to two new surfaces:** resend must return an **identical response to the caller** whether or not the email is a real unverified account (no "no account"/"already verified" tell); verify collapses all failures into one `?error=invalid_token` (the same move as L1's `mapSignUpError` collapsing `USER_ALREADY_EXISTS`, now a redirect param not a `Result` code). **`emailVerified: true` is NOT authorization** — it proves only "controls this inbox," the capability **floor not ceiling**; per-action authz (tenancy/role/ownership) still runs at the action boundary (Unit 9). Email-change-from-settings (Ch 054 L2) reuses this exact machinery, `identifier` scoped to the user not a bare email.

**Cut** — Token at-rest framing deliberately kept conceptual: **"one-way hash, handled by the library," NOT hard-asserted `SHA-256`/`32 bytes`/`base64url`/`256 bits`** as the outline stated (Better Auth's current internals may store a signed token; docs don't expose the mechanism — teach the security *property*, version-proof). `errorCallbackURL` (dedicated error page) named in one parenthetical only; single landing page handling `?error=` is the taught default. From-address/sender discipline named as Unit 7, not re-shown in the template.

## Lesson 5 — Magic links

**Taught** — A *conditional power tool*, not a default: the year-1 baseline stays email+password+optional 2FA, magic-link is a deliberate reach. **The decision (heaviest beat), in senior question order: cadence → MFA mandate → deliverability.** Wins on infrequent sign-in (password-reuse is the dominant risk; inbox round-trip is cheap when rare), non-technical user base swamped by reset tickets, email-centric products (newsletter/mailing-list). Loses on daily sign-in (round-trip friction every session), shaky/corporate deliverability, hard MFA mandate (password is a cleaner first-factor anchor). **Mental model: "the inbox is the credential"** — a click proves *one* thing (control of the email account), trading a *remembered* secret (password) for a *delivered* one (link); everything else falls out of it. **The plugin: two registrations** — server `magicLink()` in `lib/auth.ts` plugins array + client `magicLinkClient()` in `lib/auth-client.ts` (the server/client plugin-set match from Ch 052; forget the client half and `authClient.signIn.magicLink` won't exist). **Four knobs, three familiar + one trap:** (1) `sendMagicLink: async ({ email, url }) => sendEmail({ to, subject, react: MagicLinkEmail({ url }) })` — same shape as L3 `sendVerificationEmail` / L4 `sendResetPassword`, library mints token + builds `url`; (4) **`storeToken: 'hashed'` — THE trap, lib default is `'plain'`** (plaintext token in the `verification` row), the single most important line, invisible-until-breach. **The four-step flow:** submit (`authClient.signIn.magicLink({ email, callbackURL: '/dashboard' })` → token + `verification` row keyed `magic-link:<email>` → `sendMagicLink`) → "check your inbox" (enumeration-safe, echo email) → click (`…/api/auth/magic-link/verify?token=&callbackURL=`, same `[...all]` catch-all, no new route; wins when the user is on a different device (typing crosses a gap a click can't) or email clients mangle/pre-fetch link URLs. The token is a CSPRNG-generated bearer token.

## Lesson 6 — TOTP and recovery codes

**Taught** — The chapter's first *second-factor* lesson: adds the **possession factor** (TOTP, RFC 6238) + recovery-code fallback. **The mechanism:** a ~160-bit base32 **shared secret** generated once at enrollment, stored on both server and authenticator app, **never travels again**; both sides independently compute `HMAC(secret, floor(now/30))` truncated to 6 digits — **only the 6 digits cross the wire, never the secret**. Server accepts current window **±1** (~90s) for **clock skew**; that widened window over a 10⁶ space is **exactly why the verify endpoint must be rate-limited** (named at the call site, full wiring Ch 074). **Honest bound:** TOTP defeats leaks/reuse/credential-stuffing but is **phishable** by a live proxy relaying the code in its window — the exact line **passkeys (L7) cross** (origin-bound). **SMS is the floor below TOTP** (SIM-swap), named once to skip, never taught. **The plugin = two registrations** (server/client match, the L5 trap): server `twoFactor({ issuer: 'Acme', totpOptions: { period: 30, digits: 6 } })` in `lib/auth.ts` + client **`twoFactorClient()` registered BARE (no `onTwoFactorRedirect`)** in `lib/auth-client.ts`; forget the client half and `authClient.twoFactor.*` is undefined. **`totpOptions` defaults (`period:30`, `digits:6`, SHA1) are KEPT** (a default-honoring decision — changing them silently breaks every authenticator app; the deliberate contrast with L1's password-floor divergence). **`onTwoFactorRedirect` is the plugin's BUILT-IN alternative the course does NOT use** — its job is to bounce the browser (via `window.location.href`) to a *standalone* `/sign-in/two-factor` page when sign-in needs the second factor. ## Lesson 7 — Passkeys and WebAuthn

**Taught** — The chapter's only **cryptographic, origin-bound** credential and the line L6 promised passkeys would cross. **The spine — "the credential IS the origin":** a password is a string typed into any look-alike site, a TOTP code is six digits a live phishing proxy relays inside its window; a passkey is a **private key the browser will only sign a challenge with when the request comes from the exact registered origin** — no string to leak/retype/relay, so the phish **dead-ends at the browser's origin check, not the user's judgment** (`phishing-resistant`, structural not behavioral). **Three WebAuthn roles** (`WebAuthn`=W3C standard, `FIDO2`=same machinery): **relying party** (your server, stores public keys + issues challenges, scoped by `rpID`), **user agent** (the browser — the *only* actor that enforces the origin check), **authenticator** (holds the private key in the secure enclave, unlocks with a local gesture; *platform* = built-in phone/laptop vs *roaming/cross-platform* = YubiKey). **Two ceremonies** (the L6 enroll-vs-challenge echo — two flows, different rules): **registration** = `navigator.credentials.create()` under the hood, mints a keypair, returns public key + `attestation`, RP stores a `passkey` row; **authentication** = `navigator.credentials.get()`, RP issues a time-bound random challenge, authenticator signs it, RP verifies the signature against the stored public key. **The asymmetry that makes it safe:** the private key **never leaves the enclave** (`public-key cryptography`) — the server only ever holds the *public* key, so a full `passkey`-table snapshot authenticates nobody. The chapter's "secret-the-user-holds ≠ row-in-DB" property, **one rung stronger than L1/L3/L6's hash-at-rest** — there the server hashed a secret it briefly saw; here it **never had the secret at all**. The student writes `authClient.passkey.addPasskey` / `authClient.signIn.passkey`, **never `navigator.credentials.*`** (library wraps both). **The plugin = two registrations** (the L5/L6 trap): server `passkey()` in `lib/auth.ts` plugins array + client `passkeyClient()` in `lib/auth-client.ts`; **import-path divergence (the one genuinely new trip):** server from **`@better-auth/passkey`**, client from **`@better-auth/passkey/client`** — its OWN package, NOT core `better-auth/plugins` + `better-auth/client/plugins` where L5/L6's halves live. **Config knobs:** `rpName` (cosmetic label in the OS prompt), **`rpID`** (the hinge — the registrable domain the credential is bound to; **`origin`** is the full `https://` form), **`authenticatorSelection: { residentKey: 'preferred', userVerification: 'preferred' }`** (`residentKey:'preferred'` = ask for a *discoverable*/resident credential, what powers autofill sign-in; `userVerification:'preferred'` = require the local gesture). **The `rpID` trap:** must equal the *registrable domain the app runs on* — set it to `example.com` when the app is at `app.example.com` and **registration succeeds, every sign-in silently fails** (the single most common passkey misconfig; same origin-binding that stopped the phish, now with *you* on the wrong origin). **Schema (regen via `@better-auth/cli generate`, Ch 052 workflow):** one **`passkey` table**, *one user → many passkeys* (model many from the start). Columns: `id`, `userId`, `credentialID`, `publicKey` (the only secret-shaped thing, and it's public), `counter`, `deviceType` (`'singleDevice' | 'multiDevice'`), `backedUp` (boolean sync signal), `transports`, `name`, `aaguid`, `createdAt`. **`counter` caveat:** monotonic signing counter the library checks for clone detection, but **synced passkeys report `counter: 0` and never increment** — the check is *inert* for the common (synced) case, real only for single-device authenticators (teach as "a check the library performs," not the load-bearing guard). **Registration (default `registration.requireSession: true`):** a passkey is, out of the box, a credential you *add* to an account you're already signed in to (settings page `/settings/security/passkeys`); `authClient.passkey.addPasskey({ name })`. **NOT a Server Action** — the browser client owns the whole WebAuthn round-trip (no FormData / Zod boundary / Result discriminant); do NOT force the L1–L4 action skeleton. **Passkey-first onboarding** (no prior session) = `requireSession: false` + a server **`resolveUser`** callback — *named as the senior reach, NOT built*. Management surface: `listUserPasskeys()`, `updatePasskey({ id, name })`, `deletePasskey({ id })` — build the list assuming *many* per user with device labels so stale ones can be pruned. **Sign-in (the second ceremony, both entry points call `authClient.signIn.passkey`):** explicit button = `signIn.passkey()`; **conditional-UI autofill (the 2026 default, the most ink)** = passkeys surface *inside the email field's autofill dropdown*, no email/password typed — wired with an input marked **`autocomplete="username webauthn"`** (`webauthn` must be the **last** token) + an on-mount `signIn.passkey({ autoFill: true })` guarded by **`PublicKeyCredential.isConditionalMediationAvailable()`** (short-circuits on unsupported browsers). Underlying platform primitive = `conditional mediation` (`navigator.credentials.get({ mediation: 'conditional' })`), named at the call site; only discoverable credentials populate the dropdown (why `residentKey:'preferred'` was set). **The headline:** one passkey tap proves **possession + biometric + origin** in a single cryptographic step — what password+TOTP does in two round-trips — which is why a passkey can be *primary* sign-in, not only a second factor. Post-sign-in redirect routes through **`safeNext`** (L2's open-redirect guard). **Sign-in failure modes (landed inline):** `rpID` mismatch (where the bad config finally surfaces — suspect first if sign-in fails for everyone on ship); **`NotAllowedError`** (benign — user dismissed / no credential on this origin/device; treat as "try another way," **always keep a non-passkey path visible**); origin+counter verification (library does both; senior reflex is to *review* hand-rolled code that skips them). **The product call (two-axis):** **synced** (`multiDevice`/`backedUp:true` — iCloud Keychain / Google Password Manager / 1Password, key replicates E2E-encrypted, recovers on device loss, security rides the cloud account) is the **right consumer default**; **device-bound** (`singleDevice`/`backedUp:false` — YubiKey / pinned managed credential, never leaves the hardware, lost device = gone) is the **enterprise/high-assurance reach**; `backedUp` is a column the app can read to tier policy. Composed with primary-vs-second-factor in senior question order **stakes → role → copyability**. **Recovery (the chapter's seventh recovery-codes instance):** a user whose *only* credential is one synced passkey who loses the cloud account is locked out — same shape as L6's lost-phone; the fix is **L6's recovery codes reused as the universal fallback** (named, NOT rebuilt — passkeys are another *consumer* of the once-only reveal). The rule the whole chapter returns to: **a credential the user can lose must have a recovery path set up before they lose it** (passwords→reset, TOTP→codes, passkeys→those same codes); "no recovery path" must be a loud explicit opt-in (`single point of failure`).

## Lesson 9 — Account linking

**Taught** — Last teaching lesson; closes L8's open **refuse** branch. **The spine — "linking is a trust transfer between proofs of identity":** the schema already does the join (one `user`, many `account` rows sharing a `userId`), so the only questions are *when* a second `account` row inserts and *on whose authority*. **Config** in `lib/auth.ts` under a top-level **`account` key** (sibling of `emailAndPassword`/`socialProviders`) → `accountLinking: { enabled, trustedProviders, allowDifferentEmails }`. **The reframe (lesson's most important correction):** `enabled` defaults `true` — account linking is ALREADY ON; it is NOT the on-switch. What gated L8's refuse was the **empty `trustedProviders` list** (no default = trusts nobody = same-email OAuth refuses, safe-but-inert). So you configure *trust* + *match policy* on a running feature, not "switch linking on." Three knobs: **`enabled`** (leave `true`; `false` = a blunt "always refuse `account-not-linked` for every provider"); **`trustedProviders`** (the violet hinge — allowlist of providers you trust to auto-attach on email match; Google/GitHub qualify 2026, Twitter/X does not, no default list); **`allowDifferentEmails`** (defaults `false` = link only on exact email match). **A trusted provider auto-links on email match EVEN WITHOUT an `email_verified:true` claim** — being on the list IS the trust; Better Auth does NOT additionally require `email_verified` for trusted providers (their docs flag this as account-takeover risk). So `trustedProviders` is the *entire* security boundary — there is NO `email_verified` backstop. A provider in `trustedProviders` must also be a real `socialProviders` entry. **Two link paths.** (1) **Implicit / on-sign-in:** the L8 find-or-create lookup's `link` branch, now resolving because trust is configured — no `(google,accountId)` row → email branch finds existing `user` → provider trusted → insert `account` row (`providerId:'google'`) against same `userId` → sign in. One Google click, no consent-to-link step → it's *surprising*, so the UX obligation is a one-time "we linked your Google account" toast AND a security-grade **"new sign-in method added" notification email** (reuses L8's `databaseHooks` side-effect seam + Unit-7 `sendEmail`, not rebuilt). This is the *convenience* layer. (2) **Explicit / from-settings** (the senior-preferred *first* link): a signed-in user at `/settings/security/accounts` clicks "Connect Google" → `authClient.linkSocial({ provider, callbackURL })` → same OAuth round-trip, but the callback attaches the row to the **session's current `user`** (no find-or-create, no email match to trust). **NOT a Server Action** — `linkSocial` triggers a redirect (no FormData/Zod/Result on the start; the callback is the boundary, like L8's OAuth start); button is a tiny `'use client'` island, settings page stays a Server Component reading `account` rows. **Terminology** — **"linking is a trust transfer between proofs of identity"** — the lesson spine; each `account` row is one proof, adding one declares "same human." **"the schema already does the join"** — the shared `userId` stitches credentials for free; only the insert + authority are in question. **"`enabled` is not the on-switch; `trustedProviders` is the hinge"** — the lesson's headline reframe (linking is already on; an empty list is safe-but-inert, the over-trusting list is the danger). **"being on the list IS the trust — there is no `email_verified` backstop for trusted providers"** — the most-misread fact; the takeover risk hinges on it. **"the dangerous combination"** = `allowDifferentEmails:true` + implicit on-sign-in linking. **"the library stops the foot-gun; you make the dead end a door"** — the unlink-guard framing (default is SAFE here, the *inverse* of the chapter's usual insecure-default footgun). **`UNABLE_TO_UNLINK_LAST_ACCOUNT`** = the library's last-account rejection code (ground exact string against the installed version). **`allowUnlinkingAll`** = the override (default `false`) that DISABLES the last-method protection — never flip without a guaranteed remaining sign-in path. **`canonical email`** = the one `user.email` address (vs per-account emails on `account` rows). **`domain takeover`** (`Term`'d) = the concrete way a trusted provider's email claim goes wrong. **Patterns and best practices** — **`enabled` stays `true`; curate `trustedProviders` as a security boundary** — a provider earns the list only when you judge its identity-for-an-email genuinely trustworthy; the empty list is safe (refuses), the over-trusting list is the takeover vector. **A `trustedProviders` entry must also be a configured `socialProviders` entry.** **`allowDifferentEmails` defaults `false`; flip ONLY for a genuine multi-email product need AND only behind explicit, elevated link-from-settings** — never with implicit on-sign-in linking. **Link-from-settings is a security-posture change → gate behind `freshAge` elevation** (a stale/borrowed session can't bolt on a new sign-in method). **Fire a notification email on every new sign-in method** (the chapter's strongest instance of the "tell the human a security-relevant thing changed" discipline) — sometimes the only signal a legitimate owner gets. **Don't hand-roll the last-method count** — the library refuses by default; catch `UNABLE_TO_UNLINK_LAST_ACCOUNT` and render an actionable "set up a password first" off-ramp, never a raw error toast. **Never set `allowUnlinkingAll:true` without a guaranteed remaining sign-in path** (ships a lockout generator). **Don't conflate account-linking with canonical-email change** (shared schema, different threat models). **`linkSocial` / `unlinkAccount` are client-side redirect/calls, NOT Server Actions** — don't force the five-seam skeleton. **Reflected `callbackURL` through `safeNext`.** **`accountId`/OIDC `sub` is safe to log; the account email is not.** **Encourage ≥2 sign-in methods in settings** (the resilience the last-method guard protects).
