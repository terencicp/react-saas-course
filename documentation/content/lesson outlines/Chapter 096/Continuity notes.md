**Terminology:** three zones of mutability (local-only / pushed-to-feature-branch / merged-to-main); unreferenced / dangling commit; GC (garbage collection â€” prunes commits no branch or reflog can reach); `HEAD@{n}` notation; bisect / binary-search regression hunt; pickaxe (`-S` flag on `git log`); `rerere` (reuse recorded resolution); three-way merge; backport (cherry-pick to older release line); `fixup!` commit (created by `git commit --fixup=<sha>`); `git switch` / `git restore` as the post-2.23 split of `git checkout` (presented as stable since Git 2.51, 2025 â€” any tutorial calling them experimental is out of date).

**Patterns and best practices:** Reach for `git reflog` as the first move any time work feels lost â€” read the journal before trying to reconstruct. Prefer `git branch recover-<name> <hash>` over `git reset --hard <hash>` for reflog recovery (non-destructive parking). Always label stashes with `-m`; prefer a throwaway WIP commit on a branch over a long-lived stash. Use `git bisect run pnpm test` (automated, exit-code-driven) as the default; manual marking only for non-scriptable repros. Use `git commit --fixup=<sha>` + `git rebase -i --autosquash` in the PR review loop; set `rebase.autoSquash=true` globally. Use VS Code three-way merge editor for non-trivial conflicts; terminal is fine for trivial ones. Write commits messily while working, clean with `rebase -i` before pushing â€” the clean branch is for the reviewer. `git cherry-pick -x <sha>` to preserve lineage when picking across branches that may later merge. Never cherry-pick from `main` onto a feature branch to catch up â€” rebase instead.
