# Chapter 002 — Functions, naming, and control flow

## Lesson 1 — Arrow by default, declaration on demand

- **Taught** — Arrow-`const` as the 2026 default function form; three triggers that earn `function` declaration (hoisting, named recursion, type-guard/assertion signatures); implicit-return vs. block body; object-literal parens trap; method shorthand and `function` expressions as recognition-only; `this` rule stated and parked.
- **Debts** — `unknown`, predicate `x is T`, and narrowing land in Ch 004 L1/L6; assertion `asserts x is T` and exhaustiveness in Ch 005 L3; `Error` subclasses in Ch 009 L2; references Ch 001 L1 (binding-vs-value) and Ch 001 L6 (`const`-by-default, Temporal Dead Zone, Biome `useConst`).
- **Terminology** — "arrow `const`" = `const fn = (args) => …`; "three triggers"; "parens trap" for object-literal return; `Term` component used for "Temporal Dead Zone" reference.
- **Patterns and best practices** — Default to `const fn = (args) => …`; use `function` only for hoisting / named recursion / type predicates and assertions; wrap object-literal returns in parens; method shorthand is the right form inside object literals (don't "fix" to arrow); Biome `useArrowFunction` is the build-time backstop; avoid `this` and `.bind(this)` — pass values as parameters.
- **Misc.** — Forward-promises that React components (Unit 3), Server Actions (Ch 030 L4) will be `async` arrow `const`s exported by name — these forms are decided here, not re-explained. `<CodeVariants>` color convention reaffirmed: green = senior default, blue = acceptable alternative, orange = anti-pattern. ## Lesson 2 — Signatures that stay readable past two parameters

- **Taught** — Two-positional-parameter rule and the options-object pattern as the structural fix; the one exception (arity-meaningful functions: comparators, reducers, range helpers); deeper "too many params = too many jobs / split the function" signal; parameter defaults fire only on `undefined` (not `null`/`0`/`''`/`false`); TS required-before-optional rule and how a single options-object dissolves it (with the optional-with-default carve-out named-but-not-used); rest at the signature (`...ids: string[]`, must be last) vs. spread at the call site, mental model "rest binds positions to an array on receive; spread unpacks an array to positions on send"; wrapper pattern `(...args: Parameters<typeof baseFn>) => baseFn(...args)`; field-level defaults inside the options object with trailing `= {}` so the function is callable with no argument; `arguments` named once as legacy.
- **Debts** — Inline object-type literal and `?` optional modifier depth → Ch 004 L2; signature destructure mechanics (rename, nested, rest at destructure, default at destructure) → Ch 002 L6; `??` vs `||` nullish-vs-falsy operator pair → Ch 002 L5; typed-wrapper generics depth (`Parameters<typeof>`, variadic tuples) → Ch 005 L7; references Ch 001 L2 (falsy/nullish) and Ch 002 L1 (arrow-`const` form).
- **Terminology** — "two-positional-parameter rule" / "two-param rule" (two is the threshold, not "two or three"); "options object" / "options-object pattern"; "positional soup" (anti-pattern name); "object type literal" (Term defined inline); "field-level default"; "trailing `= {}`"; "wrapper pattern"; mental model "rest = receive into array, spread = send from array".
- **Patterns and best practices** — Past two positional params, switch to a single options-object parameter; mark each option `?` for optional; use field-level destructure defaults `({ pageSize = 20, sort = 'asc' } = {})` for any function with optional config (this is the canonical paginated-list shape); destructure-then-rebuild when forwarding (deferred to L6); never use `||` for defaults of values where `0`/`''`/`false` are valid (deferred to L5); rest parameter must be last in signature; ESLint `max-params: { max: 2 }` is the build-time backstop.
- **Misc.** — Forward-promises every Server Action (Ch 030), React component prop (Ch 022/Unit 3), and Drizzle query helper (Unit 5) takes the options-object shape installed here — not re-derived. `<CodeVariants>` reaffirmed: green = options object (senior default), orange = positional soup (anti-pattern). `PredictOutput` uses multi-line template-string `expected`. `CodeReview` plants 2 `ReviewIssue`s on one file with a `ReviewWhy` debrief. `ScriptCoding` (sandpack) `formatAddress` refactor has 4 tests; the 4th deliberately pins the `undefined`-only firing rule via `country: undefined`. ## Lesson 4 — Flat control flow: guards, ternaries, and exhaustive switch

- **Taught** — Three structural forms that keep function bodies flat: guard clauses for early exits (with "no `else` after `return`" rule + Biome `noUselessElse`); expression-level ternaries for value selection only (never side effects); nested-ternary policy (acceptable for flat literal trees, refactor when branches are non-trivial); exhaustive `switch` over discriminated unions with `noFallthroughCasesInSwitch` + `assertNever` in `default` for compile-time exhaustiveness; lookup-map alternative for uniform value-to-value mappings (with `noUncheckedIndexedAccess` `??` fallback or `Record<Keys, Value>` typing); loop reach ranking (`.map`/`.filter`/`.reduce` → `for...of` → `for (let i…)` → never `for...in`, the last shipping a prototype-chain bug).

- **Debts** — `never` bottom type derivation and narrowing depth → Ch 004 L1/L6 (referenced as "the chapter on TypeScript narrowing"); `satisfies never` / full exhaustiveness type-system story → Ch 005 L3; `.map`/`.filter`/`.reduce` depth → Ch 003 L3 ("a later chapter on container-shaped data"); `Record<K,V>` and mapped types → Ch 003 ("chapter on object types and records"); `??` operator semantics → Ch 002 L5 (next lesson); `tsconfig` strict flags (`noFallthroughCasesInSwitch`, `noUncheckedIndexedAccess`) already enabled per Ch 024 L4.
- **Terminology** — "guard clause" (Term); "discriminant" (Term: the literal-typed shared field of a discriminated union); "`never` type" (Term, deferred derivation); "`noFallthroughCasesInSwitch`" (Term); "`noUncheckedIndexedAccess`" (Term); "polarity flip" (mental model for nested→guard refactor); "expression-level" vs "side effect" framing for ternary reach; "shape of the smell" reflex; `assertNever` lives once per codebase at `lib/assert-never.ts` and is imported everywhere a `switch` over a discriminated union appears.
- **Patterns and best practices** — Indentation deeper than two levels in a 2026 SaaS function is a smell; every `switch` over a discriminated union ends with `default: return assertNever(value);`; ternaries only for assignment/return/argument value selection (use `if`/`else` for side effects); object-literal lookup with `?? fallback` (or `Record<Union, V>` typing) for uniform value-to-value mappings; `for...of` + `Object.entries` for object iteration (never `for...in`); `for...of` with `break`/`continue` when array methods can't host the body; `for (let i…)` only when the index is semantic data; `assertNever` helper signature `(x: never): never` throwing at runtime with `JSON.stringify` of the unhandled variant.

## Lesson 5 — The null-safe operator trio

- **Taught** — `?.` (property, call `?.()`, indexed `?.[…]`) with per-link semantics; the overuse trap (over-protecting non-nullable fields silences TS); Biome `useOptionalChain` as build-time backstop; `??` precise nullish-vs-falsy semantics with three concrete `||` traps (`0`, `''`, `false`); `??=` for lazy init / cache-or-compute with lazy RHS evaluation; `||=` / `&&=` named once; mixing-operators parens rule (`??` with `||`/`&&` is a syntax error without explicit parens).
- **Debts** — Narrowing depth (how TS flows nullability through `?.`/`??`) → Ch 004 L6; Zod `.default()` at schema boundary → Unit 6 (named once so `??` doesn't get generalized to that layer); `useState` lazy initializer parallel → Ch 023 (not derived here); full operator-precedence table never memorized — parens reflex instead.
- **Terminology** — "the trio" (`?.` / `??` / `??=`); "nullish" (Term: null/undefined only, vs falsy's broader set); "short-circuit" (Term); "optional chaining" (Term covering `?.`, `?.()`, `?.[…]`); "per-link rule" for `?.`; "overuse trap" for over-protecting non-nullable fields; "falsy-vs-nullish" as the semantic distinction; "parens reflex" for mixed-operator expressions.
- **Patterns and best practices** — `??` by reflex for missing-value defaults; `||` only when truthy-short-circuit is exactly the intent; `?.` at every nullable link, never just the first; `?.` only where the type acknowledges nullable (let TS fail at the call site otherwise); `??=` for cache-or-compute / "fill if empty"; explicit parens whenever `??` shares an expression with `||`/`&&` (or any binary like `+`); schema defaults belong at the schema boundary, not `??` at the call site.

## Lesson 6 — Destructuring as the API call-shape

- **Taught** — Destructuring as the syntactic consumer of L2's options-object pattern; object destructuring base form + four extensions (rename, default, combined rename+default, rest) with field-level defaults firing only on `undefined` (anchored to L2/L5); rename reads field-on-source `:` local-binding (reverse of object-literal shorthand); array destructuring's two daily reaches (positional unpack with hole-skip, head-and-rest split) reserved for meaningfully-ordered data; signature-level destructure as the canonical 2026 shape (`const fn = ({ a, b, c = '' }: { a: string; b: number; c?: string }) => …`) — names match call site, field-order irrelevant, defaults at signature; signature-destructure preferred over body-destructure unless original `options` ref needed for logging/forwarding; destructure-then-rebuild discipline as the structural fix for wholesale-forward leaks; rest extension's load-bearing use for stripping a sensitive field (`const { internalNotes, ...customerSafeFields } = invoice`); nested-destructure throws on nullish intermediate links — fix with `?.` from L5.
- **Cut** — Outline's `signUp`/`password` leak example was replaced with `sendInvoice`/`internalNotes` (kept seed-domain consistency with L2–L5; same structural point); the `for...of` + `Object.entries` array-destructure example was deferred to a forward link only (Ch 003); the parens-required assignment-expression form (`({ a, b } = obj)`) was not mentioned at all.
- **Debts** — Inline object-type literal + `?` modifier depth → Ch 004 L2 (read via `CodeTooltips` here, not derived); narrowing through destructured nullable fields → Ch 004 L6; `useState` tuple destructure → Ch 023; `Object.entries` array destructure inside `for...of` → Ch 003; React props in JSX form → Ch 022; Server Action input + Drizzle `.values(…)` destructure-then-rebuild → Ch 030 L4 / Unit 5; `assertNever`/exhaustiveness deepening → Ch 005 L3 (not relevant here but pattern referenced); references L2 (options-object), L3 (intent naming), L5 (`?.`/`??`).
- **Terminology** — "destructuring" (Term: pulls fields/positions out of object/array into local bindings in one expression); "the four extensions" (rename, default, combined, rest); "rename reads field-on-source `:` local-binding"; "signature-level destructure" / "signature destructure" as the canonical API call-shape name; "destructure-then-rebuild" as the no-accidental-forwarding pattern name; "wholesale-forward leak" as the bug name it prevents; "field-level default" (reused from L2); "shallow destructures by default" rule; "customerSafeFields" idiom name for the rest-omit pattern.
- **Patterns and best practices** — Past two params, use signature destructure with field-level defaults and inline object-type annotation — this is the literal shape every Server Action, React component, and Drizzle helper writes; when forwarding data downstream, destructure exactly the fields needed and rebuild the object literal — never forward the original object wholesale; use rest-omit (`{ sensitiveField, ...safeFields } = obj`) to strip one field and forward the rest; pick positively (`{ a, b }`) when forwarding a few, omit-with-rest when stripping one from an otherwise-safe shape; rest must be last in the pattern (parser rule); destructure rename direction is field-on-source `:` local-binding (Biome `noShadow` is the build-time backstop for rename-direction-flip bugs); array destructure only when data is meaningfully ordered (tuples, hooks returns, `[key, value]` pairs) — not on records; write shallow destructures by default and use `?.` (from L5) for nullable nested paths; default expression in combined rename+default applies to the renamed identifier, not the original field name; signature destructure is the default for API-shaped functions — reach for body destructure (with parameter named `options`) only when original ref needed for logging/forwarding.
