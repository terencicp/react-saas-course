**Terminology.** Effect = **escape hatch** (deliberate exit from React's normal data flow, used sparingly); **synchronize** (make external state match current props/state); **reactive value** (prop/state/derived-from-them, can change between renders); **stale closure** (effect using values from the render it last ran); **non-reactive trap** (reading a value at event-time that shouldn't drive resync — relief is `useEffectEvent`); **tearing** (parts of one render seeing different values of an external store — at `useSyncExternalStore`). ## Lesson 3 — useEffectEvent and the non-reactive seam

**Taught.** `useEffectEvent` (stable in **React 19.2**, imported unprefixed from `'react'`, in Next.js 16) as the **non-reactive seam**: a callback that always reads the freshest props/state when called but is excluded from dependency arrays. Core reframe — the dep array is **the list of reactive values** (reactive = a change should re-synchronize the effect), not "every variable the effect reads"; non-reactive reads (a parent's fresh `onMessage`, `currentUser`) move into an Effect Event so they're read-latest without forcing resync. Cashes lesson 2's chat-room cliffhanger: deps shrink `[roomId, onMessage, currentUser]` → `[roomId]`. Three canonical reaches: event-shaped callbacks from a third-party widget, logging/analytics fired from inside an effect, reading mutable state inside a `setInterval` tick. Four safety rules. Retired the two pre-19.2 hacks (ref mirror, `useCallback` wrap).

**Cut.** Optional `CodeTooltips` on the `useEffectEvent` line folded into annotation prose (avoided stacking two code-overlay components). `useLayoutEffect` dropped from the "callable from an Effect or another Effect Event" rule statement (kept to Effect + Effect Event for simplicity).

**Debts.** Forward-refs lesson 4 ("you probably don't need an effect" full audit) as the gate to ask *before* reaching for the seam — `useEffectEvent` is only for the residual legitimate-effect cases. Forward-refs chapter 026 lesson 1 (custom hooks — the "may create an Effect Event inside a custom hook but never hand it back to callers" nuance) and chapter 026 lesson 2 (React Compiler rejecting impure render, used to justify the no-call-from-render rule). Back-refs chapter 023 lesson 3 (purity contract) and chapter 024 (stable `useState`/`useReducer`/ref identities omittable from deps — contrasted against the Effect Event's *deliberately unstable* identity).

**Terminology.** **Non-reactive seam** (read fresh, don't re-sync — the gap the hook fills); **Effect Event** (`Term`-tooltipped: a non-reactive callback from `useEffectEvent`, reads latest props/state when called, excluded from dep arrays). **The bright line:** *does a change to this value mean "tear down and rebuild"? yes → deps; no → Effect Event.* Crisp contrast pinned as a `:::note`: **`useCallback` = stable function over a fixed snapshot; `useEffectEvent` = unstable function over the latest values** — the price of fresh reads is the restricted call site. One-sentence thesis (`:::tip`): *the dependency array is the list of values whose change should re-sync the effect; `useEffectEvent` is for the values you must read fresh but whose change must not.* `WebSocket` `Term`-tooltipped here.

**Patterns and best practices.** Effect Events are called **only from inside an Effect or another Effect Event** — never from render (impurity the Compiler rejects), never from a regular handler (handlers already close over latest values — redundant and lint-flagged). Never pass an Effect Event as a prop or return it from a hook (its unstable identity breaks anything keying on it — by design, so misuse fails loudly). Keep the body event-shaped (read latest + act; no hooks, no values meant to drive re-render). `exhaustive-deps` (eslint-plugin-react-hooks 7.x) understands Effect Events: excludes them from deps and flags calls outside allowed contexts — never `eslint-disable` to drop a non-reactive read. **Reach for the seam last:** first audit whether the effect is needed at all, then value-by-value move only the non-reactive reads — it's a scalpel, not a blanket deps exemption. Burying a genuinely reactive value (`roomId`) in an Effect Event is the mirror-image abuse (trades a loud bug for a silent broken sync).

Status must always read **React 19.2** (never invent a version, never show the old `experimental_useEffectEvent` import). ## Lesson 5 — useContext without the re-render storm

**Taught.** `useContext` as the **propagation primitive** for cross-cutting infrastructure (auth user, active org, theme, locale, feature flags, router) — answers the prop-drilling pain chapter 024 lesson 3 deferred. The three pieces: `createContext<T | null>(null)`, the React 19 bare provider `<Ctx value={…}>`, `useContext(Ctx)`. The **fail-fast consumer hook** (`useCurrentUser` wraps `useContext`, throws on `null`) so call sites read a non-nullable `T`. The footgun: **a provider value change re-renders every consumer by `Object.is`, all-or-nothing, regardless of which field is read** (derived from the chapter 023/024 bailout rule). Three mitigations as three angles on one rule (reference changes ⇒ consumers re-render): (1) **split context by concern** — the default posture, not a profiling fix; (2) **separate state from dispatch** into two contexts (dispatch is reference-stable, so dispatch-only consumers never re-render) — the canonical reducer-in-context pattern; (3) **keep the provider value reference-stable** — object-literal trap, `useMemo` shown as teaching/fallback, but the React Compiler auto-memoizes the value when the provider is pure so you write the plain literal. Server boundary (recognition only): Server Components can't `useContext`; providers live in a `'use client'` `<Providers>` island, server passes data down as props.

## Lesson 6 — Marking updates as non-urgent

**Taught.** `useTransition`, `useDeferredValue`, and standalone `startTransition` as **priority markers, not speed boosts** — the lesson's load-bearing reframe (the expensive render still costs the same; it just stops blocking urgent updates). React 19's renderer is **concurrent** (interruptible chunks): it commits urgent updates first, renders transitions in the background, and freely abandons-and-restarts in-progress transition work when a newer urgent update arrives. The **central cut**: own the setter → `useTransition` (wrap the `setX` call: `startTransition(() => setFilter(value))`); only receive the value → `useDeferredValue` (wrap at the consumer: `useDeferredValue(query)` feeds the slow child, input stays bound to live `query`). Both produce identical downstream behavior, opposite sides of the data flow. `isPending` drives spinner/dim/disable affordances. Async transitions (await inside `startTransition`, `isPending` spans the whole flow). The **three-condition reach threshold** (measurable jank ~>50ms, genuinely large work, compiler hasn't already erased it) with **no transition as the default**.

**Cut.** Nothing material from outline scope. `useDeferredValue`'s optional `initialValue`/SSR arg mentioned only in the external-resource card descriptions (not taught in body, as outline allowed). One `VideoCallout` was added (ui.dev/react.gg `edN42P_vfCI`, "The Story of Concurrent Rendering in React") — the outline declined video, but one was sourced rather than skipped.

**Debts.** Recognition-only forward refs later lessons must deliver: `use()` + `<Suspense>` mechanics (the transition-keeps-old-UI-on-screen cooperation, the `use()` primitive) → **lesson 7**; `<Suspense>` placement / fallback contract / streaming → **chapter 031**; Server Actions + `<form action>` + `useActionState`/`useFormStatus`/`useOptimistic` as **implicitly wrapped in transitions** (named as the machinery they'll recognize) → **Unit 6**; React Compiler auto-memoization + `useMemo` fallback (the deferred-value-feeds-memo compounding win) → **chapter 026 lesson 2**; the Profiler to confirm the jank threshold → **chapter 026 lesson 3**; debounce/throttle (explicitly *not* what these are) → **chapter 026 lesson 1**; `nuqs`/URL-state production search input rebuilt on this foundation → **later unit (chapter 033)**. **Terminology.** Thesis line (carry forward): **"priority markers, not speed boosts"** / **"a transition reorders work, it does not delay it."** The decision heuristic to commit: **"own the update → transition the setter; only receive the value → defer the value."** **Urgent lane** vs. **transition lane**; **interrupt-and-resume** / **abandon-and-restart** for the discarded stale background render. `Term`-tooltipped: concurrent (rendering), transition, Suspense boundary. Two named footguns: **`startTransition(setFilter)` does nothing** — you must *call* the setter inside the callback; **never defer the value the input is bound to** (defer the consumer, never the source of truth). The **post-`await` re-wrap rule**: a setter after an `await` is no longer synchronously inside the transition, so re-nest it in a fresh `startTransition` (flagged as a current React limitation).

**Patterns and best practices.** Default is **no transition** — blanket-wrapping every update is a named anti-pattern (indirection for zero payoff). The input (or whatever the user directly manipulates) **always rides the urgent lane / live state**, regardless of which hook expresses the split. Prefer `useTransition` inside a component (you get `isPending`); reach for standalone `startTransition` (imported from `'react'`) only from module scope / utilities outside a component. `useDeferredValue` is the leaner choice when a single value drives the slow component (canonical solution to the practice exercise); `useTransition` costs one extra `filter` state but hands you `isPending`. Reach only when all three threshold conditions hold; measure before assuming the compiler hasn't already made the render cheap. These are explicitly *not* debounce, *not* a perf magic wand, *not* a `<Suspense>` replacement, *not* relevant to non-React (network/`setTimeout`) work.

**Misc.** Slow-list examples use a **deliberate artificial slowdown** (busy loop / ~5,000 synthetic items) purely to make jank felt in-browser — flagged as teaching device, not production code. Async-transition examples use abstract `fetchSomething()` — **never a real Server Action** (stays inside scope). Examples client-only (`'use client'` shown, mechanics deferred to chapter 030); no error handling / Result type on async sample. Practice `ReactCoding` is tests-graded + `live` on (feeling jank vanish is the point); test names must carry the fix (error text hidden) — assert input reflects latest keystroke synchronously, list lags ≥1 render (source fallback: `useTransition`/`useDeferredValue` present), input bound to live `query` not deferred. ## Lesson 7 — Reading promises with use()

**Taught.** `use(resource)` (imported from `'react'`, stable, no flag) reads a `Promise<T>` **or** `Context<T>` **during render** and either returns the value or **suspends** the component — the one-sentence model. Four outcomes: pending → suspend to nearest `<Suspense>`; resolved → return value; rejected → throw to nearest error boundary; Context → return current value (conditionally readable). The **headline pattern**: a Server parent starts a slow read *without awaiting* (`const p = getActivity(orgId)`), passes the **unresolved promise** as a prop across the RSC wire into a `'use client'` child wrapped in `<Suspense fallback>`; the child does `const data = use(p)` and renders as if data is present — collapsing the 2020 effect+state+spinner shape to one line (loading → Suspense, errors → error boundary, works during SSR). The **stable-promise rule**: `use()` tracks promises by reference (`Object.is`); a promise created in the render body (`use(fetch(...))`) makes a new resource every render → suspend-forever loop (tell: stuck fallback + Network tab looping the same request). Two sanctioned stable sources: created in a Server Component (one run/request, stability free) or held in a stable reference; `cache()` (server) dedups same-fn-same-args within a request. The **`use()` vs TanStack Query** cut: initial server-rendered "fetch once and show it" → `use()` + Server Components; client data that polls/caches-across-views/optimistic-updates/infinite-scrolls → TanStack Query (four named triggers; they compose through the same Suspense). The **conditional capability**: `use(Context)` may sit after early returns / in branches / loops (which `useContext` cannot) — same value, **not** a different subscription model, does not fix the all-or-nothing re-render storm. **Why `use()` is exempt** from rules of hooks: regular hooks are tracked by call-order indexed slots; a promise is tracked by reference and a context by tree position, so no slot to desync — lint (`react-hooks/rules-of-hooks`) has a built-in exemption.

## Lesson 8 — Rules of hooks and the lint that enforces them

**Taught.** The **indexed-slot mechanic** as the root cause of every rule: React tracks each hook not by name/variable but by **call order** — an internal pointer starts at 0, advances one per hook call, so call N claims slot N; the next render replays the same walk and call N must re-land on slot N to read back its stored value. If the *number or order* of calls drifts, every downstream call reads its neighbor's slot and React cannot detect it (positional, no name to compare) → silently wrong state far from the cause. **Rule 1** (top level, every render): never in a conditional / loop / nested function / after an early `return`; four violation shapes (hook-in-`if`, hook-after-early-return [the one juniors actually hit — the loading-spinner guard], hook-in-`.map`, hook-in-nested-function), all fixed by the same move — *make the call unconditional, push the condition onto the value or into a child component*. **Rule 2** (caller): hooks callable only from a component body or another hook; the **`use*`-naming contract** is the entire signal React + lint use (name starts `use` + uppercase = treated as hook), enforced by name alone, not behavior. The **`use()` exception explained** (pays off lesson 7's promise): exempt because it claims **no slot** — a promise tracked by reference, a context by tree position — so there is nothing to misalign; firewalled to *exactly one* exception, generalizes to nothing. The **lint** (`eslint-plugin-react-hooks`, ships in Next.js default config): `react-hooks/rules-of-hooks` (structural, **never disabled, no false positives by construction**) + `react-hooks/exhaustive-deps` (the dep-array oracle, rare documented overrides only). The **compiler split**: the React Compiler *reduces how often `exhaustive-deps` matters* (fewer manual memo dep arrays) but *makes `rules-of-hooks` more load-bearing* — the compiler reads components assuming call-order stability holds. The **runtime safety net**: "Rendered more/fewer hooks than during the previous render" = the slot count drifted, caught a layer later than lint.
