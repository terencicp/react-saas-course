# Chapter 039 — Indexes, plans, and transactions

## Lesson 1 — Indexes that earn their weight

**Taught** — Two decision frames: the four triggers that say *add an index* (FK columns, selective `where`, `order by`/cursor keys, unique constraints) and the index-type decision tree (B-tree default; composite + leftmost-prefix; partial; expression; GIN; partial unique), plus Drizzle declaration syntax and the standing cost (write tax / disk / planner) that bounds when *not* to add one.

**Debts** — Promises later lessons cover: `EXPLAIN ANALYZE` proves each index call (039.3); the `CONCURRENTLY` migration hand-edit (040.2); generated columns as the expression-index alternative (mechanic owned by 037.4); the `ANALYZE`/auto-vacuum statistics machinery (named only, 039.3 orbit). References back to 037 (FK constraints, `.unique()`, generated columns) and 038 (joins, cursor pagination, webhook idempotency unique, `tsvector` FTS, `jsonb @>`).

**Terminology** — Mental model: index = separate sorted on-disk structure the *planner may choose*; never changes results, only the path; taxes every write. Terms defined: sequential scan, query planner, selectivity, B-tree, GIN, operator class. "Four triggers" framed as a code-review reflex. "Leftmost-prefix rule" for composite indexes. Composite ordering rule: **equality predicate first, sort/range key second, tiebreaker last**; tenant/org column always leads tenant-scoped composites. "What you get for free" trio: PK indexed, `.unique()` indexed, `.references(...)` NOT indexed (the FK trap).

**Patterns and best practices** (for the project chapter codebase):
- Every FK column gets a B-tree index by default (ship day one); never add `index(...)` for a column already `.unique()`.
- Day-one rule: ship FK + unique indexes (correctness-adjacent); every other index waits for `EXPLAIN ANALYZE` evidence — "feels slow" is not evidence.
- Index naming convention (code-standards prefix form, canonical — **not** the chapter-outline suffix form): `idx_<table>_<col>[_<col2>]` (B-tree), `idx_<table>_<col>_gin` (GIN), `idx_<table>_<col>_partial` (partial), `<table>_<col>[_<col2>]_unique` (unique). Always pass an explicit `name`.
- Partial-index / partial-unique `.where()` predicates written as raw `` sql`...` `` templates, **never** `eq()` (known Drizzle Kit bug: `eq()` emits a `$1` placeholder, producing invalid `CREATE INDEX ... WHERE` SQL). Downstream agents must not "tidy" these into `eq()`.
- Composite directional form `.on(t.createdAt.desc(), t.id.desc())` and GIN form `.using('gin', t.col)` confirmed current (Drizzle ≥0.31).
- `uniqueIndex(...).where(...)` is the tool for conditional uniqueness (one primary per org, soft-delete `unique ... where deleted_at is null`); `.unique()` can't express it.
- Prefer `jsonb_path_ops` operator class for GIN on `jsonb` queried only with `@>`.
- Low-cardinality columns (booleans, small enums) usually scan better than index; rescue is a partial index on the rare value.
- Any index added to a table with live write traffic uses `CONCURRENTLY` (rule stated here, migration mechanic deferred to 040.2).
- Canonical invoices index set (worked example, names as shipped): `idx_invoices_org_id`, `idx_invoices_customer_id` (FK); `invoices_org_external_id_unique` (webhook idempotency, doubles as external-id lookup); `idx_invoices_org_created_at_id` on `(organizationId, createdAt desc, id desc)` (cursor pagination — note `_created_at_` not `_created_`, matches 039.3's reuse); `idx_invoices_pending_partial` on `dueDate where status = 'pending'`; `idx_invoices_search_vector_gin` (FTS — full `_search_vector_gin`, not `_search_gin`).

**Misc.** — Fully built out. `casing: 'snake_case'` convention: `.on()` references camelCase TS property, emitted SQL is snake_case — do not write snake_case in `.on()`; but inside `DrizzleSchemaCoding`/live-coding widgets (which don't set client `casing`) columns are written explicitly snake_case. **Schema-coding grader confirmed working** with `uniqueIndex(...).where(...)` partial unique indexes — the documented fallback (composite-unique probe + Code/prose) was *not* needed; the graded exercise (`org_members`: FK index + composite unique + partial-unique one-primary-per-org) ships live with four probes (two-same-org-user rejected, two non-primary allowed, two primary rejected, cross-org primaries allowed). Lesson uses integer PKs in the widget (not UUIDv7). Components shipped: two custom lesson diagrams under `src/components/lessons/039/1/` (`seq-scan-vs-index.astro`, `composite-prefix-diagram.astro`), `AnnotatedCode` ×3, `CodeVariants` (full-vs-partial), `Buckets`, `MultipleChoice`, `StateMachineWalker` (index-type decision tree), `DrizzleSchemaCoding`, four `ExternalResource` cards, two `VideoCallout` (B-tree explainer + multi-column index). BRIN/GiST/SP-GiST/hash kept to a single recognition `Aside` as planned.

## Lesson 2 — Spotting and fixing N+1

**Taught** — Recognition-not-API lesson: the N+1 shape (one parent query + one child query per returned row), the four disguises it hides in (await-in-`for`-loop, `Promise.all` over an id-parameterized `.map`, RSC tree where each card fetches its own data, `findMany` + per-row `findFirst`), why `Promise.all` doesn't fix it (parallelizes JS awaits, not DB round-trips), and the structural cure — collapse children into the parent query via the relational API (`with`) or a join, both from chapter 038. Diagnosis via `logger: true` (count statements-per-render); the boundary that `EXPLAIN ANALYZE` is the *wrong* tool for N+1 (each statement's plan is fine). Rules out cache, DataLoader, and "drop `with` for performance" as non-fixes.

**Cut** — Nothing material; lesson covered the full outline scope. `unstable_cache` from the chapter outline updated to current `'use cache'` / React `cache`.

**Debts** — References back to 038.2 (joins) and 038.3 (relational API `with`) as the cure tools; 038's inferred-types thread (`Invoice & { lineItems: LineItem[] }[]` for free, no hand-written `InvoiceWithItems`). Promises later: pool-starvation / contention from concurrent connection checkout (039.4); RSC waterfall / parallel-data-at-one-layer (chapter 094, owns the component-tree N+1 at depth); `EXPLAIN ANALYZE` plan reading (039.3); caching proper (chapter 033); `columns` projection to narrow an over-fetching `with` tree.

**Terminology** — Core reflex (drilled, repeated in two tips): **"Is this query running once, or once per row?"** Layer distinction: `Promise.all` rearranges the *JavaScript-await layer*; the *database round-trip layer* underneath is unchanged. The "same query parameterized by id" (N+1) vs. "different queries in parallel" (fine) line. Metric: **statements-per-render**, not per-user (a request-scoped cache can hide repeats within one render and make the count lie). "Visual signature" of N+1 = a stack of identical logged statements differing only by bound `$1`, `$2`, `$3`. Terms defined: N+1, round-trip, connection pool (redefined from 036), DataLoader.

**Patterns and best practices** (for the project chapter codebase):
- The cure is always to reshape into one query (relational `with` for nested tree reads, hand-written join for flat/aggregate projections) — never cache, never parallelize, never loop-but-smarter.
- `Promise.all` is correct for *independent* parallel reads (invoice list + current org + notifications); the smell is specifically mapping it over the same query shape parameterized by id.
- If a `with` tree over-fetches, narrow it with a `columns` projection on the relation — dropping `with` to re-fetch children by hand is a regression, not an optimization.
- Data-access code reads like `db/queries/` verb-led helpers (e.g. `listInvoicesWithItems`).
- Design for the load, not the dev sample — an N+1 invisible at 20 seeded rows is catastrophic at 800.

**Misc.** — Fully built out. Deliberate convention divergence: uses plain `db.query.invoices` / `db.select().from(invoices)`, **not** `tenantDb(orgId)` — multi-tenancy isn't taught until Unit 10; downstream agents must not "correct" this. `casing: 'snake_case'` is on the client, so example code uses camelCase TS props (`lineItems.invoiceId`) while logged SQL shows snake_case (`line_items.invoice_id`) — lesson notes this gap explicitly. The `DrizzleCoding` exercise **shipped join-based** (the documented `with`-unsupported fallback): the widget runtime constructs `drizzle(client)` with no `relations` option, so `db.query`/`with` is unavailable — the relational-API fix is taught only via `CodeVariants` + prose, and the graded `expectedRows` is the flat joined shape (innerJoin invoices×lineItems, `ordered: true`). Widget schema uses integer PKs (not UUIDv7). Components shipped: `DiagramSequence` (5-step round-trip walkthrough) with custom `RoundtripFrame.astro` under `src/components/lessons/039/2/`; custom two-lane `TwoLanesPromiseAll.astro` (JS-await lane vs DB-round-trip lane) in a `Figure`; `CodeVariants` ×2 (sequential-vs-Promise.all both marked red; N+1-vs-two-cures); `Buckets` (twoCol, 7 chips — 4 N+1, 3 fine); `DrizzleCoding` (join-based grader); four `ExternalResource` cards. Four-shapes section uses inline `Code` fragments (no per-shape component). No `MultipleChoice` (lesson's graded checks are `Buckets` + `DrizzleCoding`).

## Lesson 3 — Reading EXPLAIN ANALYZE

**Taught** — The diagnostic skill (not a feature): one rule — a query says *what*, the planner decides *how*, `EXPLAIN ANALYZE` shows the *how* with real numbers; `EXPLAIN` (estimates, doesn't run) vs `EXPLAIN ANALYZE` (runs, adds actuals); reading the plan tree (indentation = depth, deepest node runs *first*, results flow up to the root which is last); the four numbers a senior reads (estimated `rows` vs `actual rows`, `loops=`, `actual time=start..end`, buffer `hit`/`read`); the node-type field guide (scans/joins/post-processing); plan-pattern → diagnosis → fix mapping; the **measure → hypothesize → change-exactly-one-thing → re-run** loop; the dev-vs-prod data trap (planner changes strategy with table size). Pays the debt 039.1 and 038.6 deferred via the cursor-pagination before/after worked example (Seq Scan + external-merge Sort → Index Scan, no Sort, `read=2304`→`read=0`).

**Debts** — References back to 039.1 (every diagnosis points at an index the student already learned to add; cost-of-over-indexing justifies the one-change rule), 039.2 (the N+1 boundary, restated — see Terminology), 038.6/039.1 (cursor-pagination query + `idx_invoices_org_created_at_id`, reused verbatim). Promises/defers: `CREATE INDEX CONCURRENTLY` production lock-out (040.2; adding an index *in the console exercise* is fine); Drizzle Studio mechanics (040.1, named as a console only); the seed lesson / realistic-volume seeding (040.2, named as the answer to the dev-vs-prod trap); `INCLUDE`/covering indexes at depth (`Index Only Scan` named only); `auto_explain`, pganalyze, Datadog (named for recognition).

**Terminology** — Framing sentence repeated as the spine: **"a query says what, the planner decides how, `EXPLAIN ANALYZE` shows the how with real numbers."** Terms defined: declarative, query planner, cost estimate (**arbitrary planner unit, never milliseconds** — `actual time` is the ms), `BUFFERS`, statistics, `ANALYZE` (the standalone command, distinguished from the `EXPLAIN ANALYZE` option — flagged as a name collision), heap (implied), covering index, `Rows Removed by Filter`, `auto_explain`. The **two-N+1 distinction** (critical, must stay consistent with 039.2): *application N+1* = round-trip-count problem, every individual statement's plan looks healthy, caught by `logger: true` — `EXPLAIN ANALYZE` is the wrong tool; *database-side nested loop* with high `loops=` on an unindexed inner scan = one statement the plan **does** show, fixed by the FK index. `loops=` named "the silent multiplier" (per-loop time/rows/buffers — multiply by `loops` for true cost).

**Patterns and best practices** (for the project chapter codebase):
- **Always write `(analyze, buffers)`** explicitly — portable habit: a no-op on Postgres 18 (BUFFERS auto-on with ANALYZE) but mandatory on older majors. Downstream agents must not "simplify" to bare `EXPLAIN ANALYZE`.
- The **one-change-per-iteration** discipline is non-negotiable — change two things and you can't attribute the win (and may ship a useless write-taxing index); this separates a fix from cargo-cult index-spraying.
- Diagnose against **production-shaped data** (realistic seeded volume or a Neon production-shaped branch), never a toy dev table — the planner flips strategy at scale.
- `EXPLAIN ANALYZE` *executes* the statement: for `UPDATE`/`DELETE`/`INSERT` wrap in `begin; … rollback;` or drop `ANALYZE` — real production footgun.
- In-app probe form: `db.execute(sql\`explain (analyze, buffers) …\`)` (raw `sql\`\`` template, lowercase SQL keywords); plan *output* shown verbatim mixed-case (`Seq Scan`, `Index Scan`). Everyday path is copy logged SQL → prefix → run in a SQL console, not script it.
- `auto_explain.log_min_duration` is the production version of the manual loop (slow queries log their own plans) — named, deferred to server config.

**Misc.** — Fully built out. `SQLCoding` widget runs **PGlite = Postgres 17** (`REL_17_5-pglite`), *not* the Postgres-18 production target — so BUFFERS is *not* default in exercises (the exact reason the "always write `(analyze, buffers)`" habit is taught). Exercises follow chapter-038 PGlite staging conventions (integer PKs, explicit `snake_case` column strings, seeded ids) — deliberate divergence from the app's `casing: 'snake_case'` client, do not "correct." Index added in the post-index sandbox uses `idx_<table>_<col>` (039.1 / code-standards prefix form). **Plan-flip exercise shipped as the fallback (graded `SQLCoding` not used):** the `SQLCoding` widget runs **single statements** and **resets its seed per run**, so a student can't `CREATE INDEX` then re-run `EXPLAIN` in one graded interaction, and the grader can't reliably match an `Index Scan` row out of PGlite's `QUERY PLAN` output. Instead the plan-flip ships as **two read-only `SQLCoding` sandboxes** (run-as-is, no grading): a first one over 6 rows (read which line ran first) and a second over 5,000 rows where the index is **pre-declared in the seed** (`CREATE INDEX ... ; ANALYZE invoices;` baked in) so the student observes `Index Scan` + the gone `Rows Removed by Filter` directly — paired with a graded **`MultipleChoice`** ("which node sits at the root after the index?" → `Index Scan`). ## Lesson 4 — Transactions and isolation levels

**Taught** — A transaction is a boundary with **two independent knobs**, taught strictly in order: (1) **atomicity** — all-or-nothing, on by default, the everyday reason 95% of transactions exist; (2) **isolation level** — what a *concurrent* transaction sees, the rare second question. Delivers: ACID restated (A and I are the subject; C and D come free from Postgres + your constraints); the Drizzle `db.transaction(async (tx) => …)` shape (no manual commit/rollback — return commits, throw rolls back; callback return value crosses the boundary out); the four senior triggers for a transaction (multi-row mutation, cross-row invariant the schema can't express, read-modify-write under contention, atomic state transition w/ DB side effects) + the negative space (a single statement — including bulk array insert — is already atomic); the three concurrency anomalies on a two-lane timeline (non-repeatable read, phantom read, write skew); the four isolation levels and where each earns its keep; the **constraint-first reflex** (links to 037); plus two carried-forward helpers — a 40001 serialization-failure retry wrapper and a 23505 unique-violation mapper to `Result`.

**Cut** — Savepoints / nested `tx.transaction()` (`SAVEPOINT`) cut entirely (chapter outline had it as a brief mention; outline itself flagged "prefer cutting if length runs long"). A later lesson depending on savepoints would need to re-introduce them from scratch.

**Debts** — References back to 037 (`.unique()`/CHECK/FK constraints for the constraint-first reflex), 038 (the query API used in examples; 038.5 webhook-idempotency unique index as the 23505 example), 039.1 (the `(organizationId, externalId)` unique index), 036.4 (pooled vs unpooled Neon URL, PgBouncer transaction-mode pooling), and the `Result<T>`/`ok`/`err` channel from code conventions. Promises forward: **043.5** wraps exactly this `db.transaction` shape in a Server Action (parse → authorize → transaction → revalidate → return); **043.3** cashes in `isUniqueViolation` for form-field-level error mapping; outbox-style transactional messaging is **Unit 11**. `EXCLUSION` constraints named for recognition only.

**Terminology** — Spine mental model: **"a transaction is a boundary with two knobs — atomicity (on by default, the reason you reach for one) and the isolation level (defaults to `read committed`, turn up only when a named anomaly forces it, and turning it up signs you up for a retry loop); before either knob, ask whether a constraint already does the job for free."** The **`db` vs `tx` discipline** (highest-value rule): any data-layer helper that might run inside a transaction takes the client as its **first argument** (`fn(client, …)`), typed `type DbClient = typeof db | DbTransaction`; a helper closing over module-scope `db` silently runs on a different pooled connection, outside the transaction (the bug that passes every single-dev-machine test and corrupts prod). **Optimistic concurrency** = `serializable` + retry (let everyone run, abort + retry losers; best when conflicts rare); **pessimistic concurrency** = `SELECT … FOR UPDATE` / `.for('update')` (lock the known hot row, writers queue; best when contention high and hot row identifiable). Terms defined: transaction, ACID, atomicity, durability, snapshot, non-repeatable read, phantom read, write skew, serialization failure (SQLSTATE 40001), SQLSTATE, unique violation (23505), row lock, transaction-mode pooling, pool starvation. **Accuracy point that contradicts older material:** Postgres `repeatable read` is stronger than the SQL standard — it prevents *phantoms* too (snapshot isolation), not just non-repeatable reads; only `serializable` catches write skew.

**Patterns and best practices** (for the project chapter codebase):
- Every data-layer helper callable inside a transaction takes the client first arg (`DbClient = typeof db | DbTransaction`); never reach for module-scope `db` inside a `db.transaction` callback.
- Default to no transaction (single statement is atomic) → `read committed` (the default, wrap for atomicity only) → escalate only on a named anomaly. Decision order: single statement? → multi-statement read needing coherence? → cross-row invariant a constraint can't express under contention? → hot row identifiable? The everyday answer is "no transaction" or "default level."
- Constraint-first: a `UNIQUE`/`CHECK`/FK (or partial unique index for conditional rules) is the cheaper, always-on, race-free defense; reach for a transaction-with-a-check only for invariants the schema genuinely can't express.
- **Never `await` external IO inside `db.transaction`** (no fetch/email/Stripe/R2/queue) — transaction-mode pooling holds the connection for the whole transaction duration; external IO inside starves the pool under load. Gather inputs before, fire side effects after commit. This also makes the transaction safely retryable (same discipline, two views).
- `serializable`/`repeatable read` ship **with** the 40001 retry or not at all — bare = random user 500s; retry **with backoff** (no backoff = thundering herd makes contention worse); retry the *whole closure*, bounded attempts (2–3).
- 40001 (serialization failure) → **retry**; 23505 (unique violation) → **don't retry, map to `err('conflict', …)`** (deterministic — retrying reproduces it). Narrow `unknown` in catch via `instanceof Error` + `'code' in error`, never `catch (e: any)`.
- `.for('update')` runs inside an ordinary `read committed` transaction (the row lock does the serialization work, no higher level needed); holds the lock until commit, so keep it short.
- Genuinely long-running transactions (migrations, rare batch jobs) use `dbUnpooled`; everything on the request path uses pooled `db` and stays sub-100ms.
- Canonical helper signatures introduced here (reused in 043): `isSerializationFailure(error): boolean` (40001), `withRetry<T>(run)` wrapper, `isUniqueViolation(error): boolean` (23505).
