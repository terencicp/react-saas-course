# Lesson 6 — Metadata and dynamic OG cards

- **Title (h1):** Metadata and dynamic OG cards
- **Sidebar label:** Metadata and OG cards

---

## Lesson framing

**The senior question (drives the intro).** A new invoice detail page needs three things the route tree doesn't give you: a browser-tab title, a search-engine description, and — when someone pastes the link into Slack — a preview card carrying the company logo, the invoice number, and the customer name.
This lesson names the three coordinated mechanisms that produce them: the static `metadata` export, the dynamic `generateMetadata` function, and the file-convention `opengraph-image.tsx`.

**Spine / mental-model one-liner (repeat across sections):** *"Metadata is data about the page, authored where the page lives. Static when it's known at build, `generateMetadata` when it depends on the resource, a colocated image file when the preview is a picture."* The student should leave able to: set brand-wide defaults at the root, override per-page, make a detail page's title + OG card track the resource it renders, dedupe the resource read so the page and its metadata hit the DB once, and reason about when an OG card is static (free) vs dynamic (a render).

**Pedagogical posture.**
- This is a *systems-design-over-syntax* lesson but the API surface is genuinely large; the cure for cognitive load is **ruthless scoping to the canonical SaaS reach** (`title`, `description`, `openGraph`, `twitter`, `alternates.canonical`, `metadataBase`) and a one-line tour of the rest, not a field-by-field tour. The full field table belongs in the docs we link, not in prose.
- **Build on prior chapters, do not re-teach.** Three load-bearing prerequisites are *recapped in one clause each, not derived*: `params`/`searchParams` are Promises in 16 (ch.033 l4, ch.032 l7); React `cache()` is request-scoped memoization for non-`fetch` reads (ch.032 l5); under Cache Components routes are dynamic-by-default and `'use cache'` opts a function into cross-request caching (ch.032). The dedup pattern and the Cache-Components-metadata rule are *applications* of those, framed as "you already know the tool, here's where it lands."
- **Lead with the default, then the power-tool.** Static `metadata` is the default every page reaches for; `generateMetadata` is the conditional tool whose trigger is "the title/description/card depends on the resource." Same shape for OG: the **static `opengraph-image.png` file is the default** (fastest, zero runtime cost), `opengraph-image.tsx` with `ImageResponse` is the power-tool whose trigger is "the card must show per-resource data." Name the default first in both pairs — this is the chapter's "defaults before conditionals" thread.
- **Production stakes are real and easy to dramatize.** A wrong/missing OG card is the difference between a link that gets clicked and one that looks broken in a shared channel — acquisition, not polish. Two failure modes carry genuine senior weight and must be taught as such: (1) the same resource read firing twice (page + metadata) doubling the DB hit; (2) personalizing an OG card from the session, leaking one user's data into a link anyone can open. Both get explicit treatment in-section.
- **Two verified-fresh corrections to the chapter outline (downstream must keep these).** The chapter-outline brainstorm contains two stale claims that current Next 16.2 docs (verified 2026-03-03) contradict — teach the corrected version:
  1. **OG image runtime.** The outline says "`ImageResponse` runs on Edge (no Node APIs)." **Wrong for Next 16.** Generated OG images are *statically optimized by default* (rendered at build, cached) and the official examples read local fonts/logos with `node:fs/promises` + `process.cwd()`. Teach: the file is a build-time/Node-capable special Route Handler; it only defers to request time if it reads a request API or uncached data. Do **not** tell students they can't use Node APIs.
  2. **`metadataBase` when missing.** The outline says missing `metadataBase` "produces relative URLs some platforms refuse." **More precise:** a *relative URL in a metadata field without* `metadataBase` is a **build error**; and when you don't set it, Next infers a default (`VERCEL_URL`, else `localhost:3000`). The senior reason to set it explicitly is correctness of the inferred origin (a per-deploy `VERCEL_URL` is the wrong canonical/OG origin), not "platforms refuse."

**Worked-example continuity.** Reuse the chapter's fictional **Acme** SaaS (vendor convention `@acme/*`, origins `assets.acme-cdn.com` from l2). The anchor surface is the **invoice detail page** `app/(app)/invoices/[id]/`. Root layout (introduced in l4 with a `metadata` placeholder — now expanded) sets `metadataBase: new URL('https://app.acme.com')` and the title template `'%s — Acme'`. This lesson **fills the `metadata` placeholder l4 deliberately left as `/* ... */`** — call that out so the artifact reads as one evolving file across the chapter. Resource read is `getInvoice(id)` wrapped in React `cache()`, returning the row or `null`.

**Scope boundary to hold (see §Scope).** This lesson owns `metadata` / `generateMetadata` / `opengraph-image` / `twitter-image` / `metadataBase` / `alternates.canonical`. It does **not** own `robots.ts` / `sitemap.ts` / `icon`/`apple-icon` / the `viewport` export (all l7) or `generateStaticParams` (l8). When the worked invoice page needs a `not-found.tsx` boundary, reuse it from ch.029 l4 (`notFound()`), don't re-teach it.

### `metadata`: page data authored where the page lives
**Goal:** establish the static export as the default and the merge-down model.
- Show the minimal static export from a `layout.tsx`/`page.tsx`: `export const metadata: Metadata = { title: '...', description: '...' }` with `import type { Metadata } from 'next'`. Use a plain **`Code`** block — it's two lines, no need for annotation.
- Teach **merge-down**: metadata is evaluated root → leaf and *shallowly* merged; a child key replaces the parent's. Root layout sets brand defaults; pages override what's page-specific. The subtlety students get wrong: nested objects like `openGraph` are **replaced wholesale**, not deep-merged — if a page sets *any* `openGraph` field it loses the parent's other `openGraph` fields. Teach the "pull shared OG fields into a shared const and spread" escape hatch in one sentence (per docs).
- **Diagram (small, high-value): the merge ladder.** A simple **Plain HTML + CSS** vertical strip inside `<Figure>` showing three stacked segments — `app/layout.tsx` (sets `title.template`, `metadataBase`, default `openGraph`) → `app/(app)/layout.tsx` (adds nothing / inherits) → `app/(app)/invoices/[id]/page.tsx` (sets `title`, `description`) — with a right-hand column showing the resolved `<head>` output. Pedagogical goal: make "evaluated top-down, child overrides parent" visible without a heavy engine. Keep under ~3 segments, horizontal-compact. Caption states the shallow-merge gotcha.
- **`title` string vs template.** The one piece of the static surface worth real depth because every project uses it. Show the root `title: { template: '%s — Acme', default: 'Acme' }` and a page's `title: 'Invoices'` resolving to `Invoices — Acme`. Mention `title.absolute` in one clause for the "ignore the template" case (e.g. a bare marketing landing page). Use **`AnnotatedCode`** here (2 steps, blue/green): step 1 highlights the root `template`+`default` (note: `default` is required when you set a template; the template applies to **children**, never the segment it's defined in); step 2 highlights the page's plain-string title and the resolved output in a comment. This is the one spot where directing focus to two cooperating files' pieces earns the component.
- **One-line tour of the rest of the surface.** A single sentence naming the full field set (`keywords`, `authors`, `creator`, `robots`, `icons`, `alternates`, `verification`, `openGraph`, `twitter`) and immediately narrowing: "the canonical SaaS reach is `title`, `description`, `openGraph`, `twitter`, `alternates.canonical`, and `metadataBase` — the rest you add when a specific need shows up." Point `robots`/`icons` forward to l7 in half a clause. Do **not** enumerate the field table.

### `metadataBase` and the canonical URL
**Goal:** the two URL-correctness disciplines, taught together because they're both "absolute-URL hygiene."
- **`metadataBase`** set once in the root layout: `metadataBase: new URL('https://app.acme.com')`. What it does: lets every URL-based field below it use a *relative* path that Next composes into an absolute URL. Teach the corrected facts (see framing): a relative metadata URL **without** `metadataBase` is a **build error**; when unset Next infers `VERCEL_URL` (prod) or `localhost:3000` (dev). The senior reason to set it explicitly: pin the real production origin so canonical tags and OG image URLs don't resolve to a per-deploy `VERCEL_URL` or `localhost`. One `Aside` (tip) is fine here, but the rule itself lives in prose.
- **`alternates.canonical`** — SEO discipline. When the same content is reachable from multiple URLs (query-param variants, tracking params, trailing-slash forms), the canonical tag tells engines which URL is the real one so ranking signal consolidates instead of splitting across duplicates. Show `alternates: { canonical: '/invoices' }` (relative, composed by `metadataBase`). The reflex to teach: set it on any page where `searchParams` can generate duplicate-content URLs. Keep tight — one paragraph + one code line.
- `Term` candidates in this section: **canonical URL** (the single URL you want indexed when the same page is reachable many ways), **link equity** is *already defined in l3* — reuse, do not redefine (note this for the writer).

### `generateMetadata`: when the title depends on the resource
**Goal:** the conditional power-tool and its trigger; the Promise shape; `notFound()`.
- **Trigger first.** "Static `metadata` can't know the invoice number — that lives in the database, keyed by the URL's `[id]`. When metadata depends on the resource, you reach for `generateMetadata`." This is the defaults-before-conditionals beat for metadata.
- Show the async function: `export async function generateMetadata({ params }: PageProps<'/invoices/[id]'>): Promise<Metadata>`. Recap in one clause that `params` is a **Promise** in 16 — `const { id } = await params` — and mention the `PageProps<'/route'>` typed helper (typedRoutes is on per l1) as the clean way to type it. **You cannot export both `metadata` and `generateMetadata` from the same file** — teach this as a hard rule (it's a build error), the writer should state it where the trigger is introduced.
- Use **`AnnotatedCode`** (3–4 steps) on the invoice `generateMetadata`: (1) signature + `await params`; (2) `const invoice = await getInvoice(id)`; (3) `if (!invoice) notFound()` — the existence check short-circuits to the `not-found.tsx` boundary (reuse from ch.029, half-clause recap); (4) the returned `{ title: ´Invoice ${invoice.number}´, description, openGraph: {...} }`. One block, focus directed part-by-part — exactly AnnotatedCode's job.
- **`notFound()` from `generateMetadata`** is explicitly supported (and `redirect()`); state it plainly. The existence check belongs in the cached helper so the page and metadata agree on "does this exist."

### Reading the resource once: the `cache()` dedup
**Goal:** the canonical optimization — its own section because it's a top-2 production-stakes idea and a frequent beginner miss.
- The problem, stated as a cost: `generateMetadata` runs to build the `<head>`, and the page component runs to build the body. **Both need the invoice.** Naively that's two DB reads per request for one page.
- The fix: wrap the read in React `cache()` so both calls share one result within the request. Show `lib`-or-`_lib` `getInvoice` defined as `export const getInvoice = cache(async (id: string) => { ... })`, imported by both `generateMetadata` and the page. Recap in one clause (ch.032 l5): `cache()` is *request-scoped memoization* — different from `'use cache'` (cross-request). The course uses Drizzle, so the read is a DB call, not `fetch`; **`fetch` auto-dedupes but Drizzle does not** — `cache()` is what you reach for. State that contrast explicitly (it's the reason `cache()` and not "fetch handles it").
- **`CodeVariants`** (2 tabs, before/after with `del`/`ins`): tab "Two reads" shows `getInvoice` as a plain async function called in both places (DB hit ×2, flagged in first sentence); tab "One read" wraps it in `cache()` (one hit, shared). This is the ideal A/B for CodeVariants. Keep prose to one paragraph per tab.
- **Exercise (understanding check):** a **`MultipleChoice`** — "Why does wrapping `getInvoice` in `cache()` matter on this page?" with distractors that target the common confusions (e.g. "it caches the invoice across requests" → wrong, that's `'use cache'`; "it prevents the second DB read within one request" → correct; "it's required for `generateMetadata` to compile" → wrong). One question, placed right after the variants. Grading: single correct.
- `Term` candidate: **request-scoped memoization** (one cached result per request, discarded after) — but only if `cache()` wasn't already `Term`-defined upstream; the writer should check ch.032 l5's continuity notes and reuse rather than redefine. Note this.

### Open Graph and Twitter cards in metadata
**Goal:** the `openGraph`/`twitter` fields, the 1200×630 spec, and the file-convention-overrides-field rule that sets up the next section.
- Show `openGraph: { title, description, images: [{ url, width: 1200, height: 630, alt }], type: 'website' }` and the matching `twitter: { card: 'summary_large_image', title, description, images }`. Canonical OG image is **1200×630**; declare dimensions explicitly — some scrapers reject cards with missing width/height, and an undersized image gets upscaled and looks fuzzy.
- Teach the **`twitter.card` values** in one clause (`summary_large_image` is the SaaS default for a hero card; `summary` for a small square). Don't enumerate the rest.
- **The bridge to the next section:** state the key rule from the docs — **file-based metadata overrides the `metadata` object**, so when you add an `opengraph-image.tsx`/`.png` file, you do **not** also set `metadata.openGraph.images`; the file convention wires the tags (including width/height/type and the alt from the sibling `.txt`) automatically. This is *why* the next section exists and why the two approaches don't collide.
- A short note on **how the card actually appears in Slack/Facebook**: most scrapers read the `<head>` tags; mention (one clause, no depth) that Next *streams* metadata for JS-executing bots but keeps it blocking in `<head>` for HTML-limited scrapers like `facebookexternalhit`, so social previews work without extra effort. This pre-empts the "will the bot see my dynamic title?" worry. Keep it to a sentence — depth is out of scope.

### Generating the card: `opengraph-image.tsx`
**Goal:** the file convention, the static-vs-dynamic default split, the `ImageResponse` shape, and the Satori constraints.
- **Default first: the static `opengraph-image.png`.** Drop a 1200×630 PNG in the route segment and a sibling `opengraph-image.alt.txt`; Next hashes it, serves it, and wires the tags. Zero runtime cost, smallest, fastest — **the senior pick when the card needn't be per-resource.** Name this as the default before the dynamic version (chapter thread). A root-level static `opengraph-image.png` is the brand fallback for every route that doesn't override (forward-ref to l7's root SEO bundle in half a clause).
- **Power-tool: `opengraph-image.tsx`.** Trigger: the card must show per-resource data (invoice number, customer name). The file default-exports a function returning `new ImageResponse(<jsx>, { width, height })` from `next/og`. Three exports configure it: `alt` (string — **prefer the `.txt`-equivalent `export const alt`** so it's colocated), `size = { width: 1200, height: 630 }`, `contentType = 'image/png'`. Note this is one of the framework's sanctioned default-exported files (per code conventions §Function form — `opengraph-image.tsx` is on the default-export allowlist).
- **The v16 headline change — `params` is a Promise here too.** This is the single most important version fact in the lesson. In 16 the image function receives `params` as a **Promise**, mirroring `generateMetadata`: `export default async function Image({ params }: { params: Promise<{ id: string }> }) { const { id } = await params; const invoice = await getInvoice(id); ... }`. It reuses the **same `cache()`-wrapped `getInvoice`** — so the page, `generateMetadata`, and the OG image can all share one read. Call that three-way reuse out explicitly; it's the payoff of the dedup section.
- Use **`AnnotatedCode`** (4–5 steps) on the invoice `opengraph-image.tsx`: (1) imports + the three config exports; (2) signature with `params: Promise<...>` + `await params` (color this step to flag the v16 change); (3) `await getInvoice(id)` (note: reuses the cached helper); (4) the returned `ImageResponse(<div style={{ display: 'flex', ... }}>{invoice.number} / {customer}</div>, { ...size })`; (5) the colocated `export const alt`. One block — this is the lesson's centerpiece code; AnnotatedCode keeps each part in focus.
- **`ImageResponse` and Satori — the constraints are the lesson, not a footnote.** It renders a **JSX subset** to PNG via Satori: **flexbox only (no grid)**, a limited CSS subset, inline `style` objects (no Tailwind classes resolved). Frame the constraint as intentional — the design surface is deliberately narrow so the render stays fast and deterministic. The student who tries to drop a real component in here will hit walls; set the expectation. `Term`-define **Satori** (the engine that renders JSX/CSS to SVG→PNG for OG images).
- **Fonts (corrected runtime story).** To use a brand font, **read the font file and pass it via the `fonts` option** — the docs read it with `node:fs/promises` + `process.cwd()` at build, which is exactly the Node-capable default. Show one clause: `const interSemiBold = await readFile(join(process.cwd(), 'assets/Inter-SemiBold.ttf'))`. Bundle a small subset to keep the render fast. **Explicitly correct the "Edge-only, no Node APIs" myth** here in one sentence — generated OG images are statically optimized and Node-capable by default.
- **`Code` block** (not annotated) for the static fallback variant — it's just a file placement + `.txt`, a `FileTree` is the better aid (see below).
- **Diagram (`FileTree`): the colocated OG fileset.** Show `app/(app)/invoices/[id]/` containing `page.tsx`, `opengraph-image.tsx`, `opengraph-image.alt.txt` (or the static `opengraph-image.png` + `.alt.txt` alternative) and `not-found.tsx`. Pedagogical goal: cement that the card lives *next to the page*, colocated by route segment — the file-convention mental model. Tiny, high-clarity. Could pair static-vs-dynamic via **`TabbedContent`** with two `FileTree`s ("static card" / "dynamic card").

### Keeping the card fresh: caching the OG output
**Goal:** the Cache-Components story for the OG file and the page→card invalidation tie.
- Recap in one clause: under Cache Components the OG image is a special Route Handler, **statically optimized by default** (built once, CDN-cached) unless it reads a request API or uncached data. For the invoice card, the read is the cached `getInvoice` — deterministic, so the card stays static/cacheable.
- **Tie invalidation together.** When an invoice is archived/edited, the *page* and the *card* must both refresh. Teach: tag the cached read (or the page body) so a mutation busts both. Show the shape briefly — `cacheTag(invoiceTags.record(orgId, id))` on the cached unit, and the mutating action calls `updateTag(...)` (read-your-writes) or a webhook calls `revalidateTag(tag, 'max')`. **Do not re-derive the caching model** — this is an *application* of ch.032 l4/l6; recap the tool in a clause and point there. Reference the tags-helper convention (`invoiceTags.record`) from code conventions, not inline strings. Keep this section short (it's a tie-off, not a new topic).
- **The Cache-Components + `generateMetadata` gotcha (verified, real, worth teaching).** If `generateMetadata` reads runtime data (`cookies()`, `headers()`) while the page is otherwise fully prerenderable, Next **raises an error** demanding you make the intent explicit: either `'use cache'` the metadata fetch (when it depends on external-but-not-request data) or add a dynamic marker to the page. Teach the resolution path in one short paragraph + a one-line `'use cache'` snippet. This is the bridge to the watch-out below about never personalizing metadata.

### What never goes in an OG card
**Goal:** the highest-stakes watch-out, given its own short section because it's a security/privacy finding, not a tip.
- **Never personalize an OG card (or shared metadata) from the session.** An OG card is rendered into a URL that *anyone* with the link can open — Slack unfurls it, a logged-out colleague sees it. Deriving the card from `cookies()`/the session leaks one user's data into a shared surface, and (per the gotcha above) also forces the metadata dynamic. The rule: OG/Twitter data derives **only** from `params` + the resource read, never from who's asking.
- Connect to the cache story: this is *why* the deterministic `params`-only read keeps the card static and shareable. The discipline and the performance property reinforce each other.
- This pairs naturally as a closing **`TrueFalse`** round (3–4 statements) consolidating the lesson's traps: "Setting both `metadata` and `generateMetadata` in one file is allowed" (F); "A static `opengraph-image.png` needs no `metadata.openGraph.images` field" (T); "`ImageResponse` can read a local font file with Node APIs" (T — the corrected fact); "An OG card may show the signed-in user's name" (F). Grades per-statement with review. Good recall closer.

## Code-convention notes (alignment + sanctioned divergences)

- `import type { Metadata } from 'next'` (type-only import — `verbatimModuleSyntax`). `import { ImageResponse } from 'next/og'`. Group-1 external imports, single quotes, 2-space, trailing commas. `next/*` counts as external (group 1).
- **Default exports are correct here** and the writer should *not* "fix" them to named: `opengraph-image.tsx`'s default function is on the code-conventions §Function form framework-allowlist. `generateMetadata`/`metadata` are named exports (the page's `default export` is the component, framework-dictated).
- `getInvoice` follows the **reads naming**: single-record read returns the row or `null` (`getInvoice(id)`); the existence check + `notFound()` is the page/metadata's job, matching the `requireInvoice` spirit without forcing that exact helper (the worked example can use plain `getInvoice` + inline `notFound()` for teaching clarity — note this is a deliberate simplification; a real project might use `requireInvoice`). Wrapped in React `cache()`.
- Cache tags via the `tags.ts` helper convention (`invoiceTags.record(orgId, id)`), never inline tag strings, when the invalidation tie is shown. `revalidateTag(tag, 'max')` — the **two-argument form is mandatory in 16** (single-arg is a TS error); `updateTag(...)` for read-your-writes from an action. (Both are recap-only; the action body is not built here.)
- `Temporal`/`Date`: if the OG card or metadata shows a date (e.g. invoice due date), keep it ISO-string at the RSC/JSX boundary — but a date is *not* required for the worked example; prefer invoice number + customer name to avoid dragging in the time stack.
