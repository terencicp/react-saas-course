# Chapter 051 — The auth mental model

## Lesson 1 — Authn, authz, and the 401/403 split

**Taught** — The authn (who is this request from?) vs authz (may this principal do this action on this resource?) split; the three-way refinement identification→authentication→authorization; where each check lives (authn at session-read boundary: `proxy.ts`/layouts/Server Actions; authz at the action boundary); the fixed order producing 401 (unauthenticated) vs 403 (forbidden); three principal states (anonymous/authenticated/elevated); a misframe catalog of bug classes.

**Cut** — The "2026 factor landscape" topic (passwords vs passkeys vs TOTP as senior-baseline-add per surface) was compressed to a one-line naming of the three factor *categories*; deferred to Ch 053.

**Debts** — Deferred forward: sessions + the carrying cookie (L2); OAuth flow (L3); sign-in methods + hashing params (Ch 053); Better Auth API (Ch 052); RBAC roles + `authedAction` wrapper + "never authz in components/layouts" implementation (Ch 057); elevation/re-auth impl + `freshAge` (L2 of Ch 054); CSRF/`SameSite` (Ch 054). Referred back: cookies (Ch 013), Server Actions + `proxy.ts` (Ch 033/034), `Result` discriminated union (Ch 043).

**Terminology** — Precise terms held exact (never "auth" alone): identification = unverified claim; authentication/authn = proof; authorization/authz = permission. principal = verified user row a request runs as; resource = the specific target; action boundary = server entry point of a mutation; fresh session / elevation = recently re-proven identity. Heuristic for 401 vs 403: "can a sign-in fix it?". Slogan: "the button is UX, the action is security." Cross-tenant access masked as **404** (deliberate exception to the clean 401/403 split). Elevation framed as "authentication triggered by an authorization policy."

**Misc.** — Canonical `Result` error-code union used is `'validation' | 'unauthorized' | 'forbidden' | 'conflict' | 'not_found' | 'rate_limited' | 'internal'`; the action-layer split is `'unauthorized'`↔401, `'forbidden'`↔403 (note: chapter-outline prose said `'unauthenticated'`, but lesson correctly uses `'unauthorized'`). Flagged that HTTP 401's name "Unauthorized" actually means *unauthenticated*. All code snippets are conceptual sketches (`Principal`, `can(...)`), explicitly marked not-real-API so later reviewers don't align them to Better Auth/RBAC.

## Lesson 2 — Sessions versus JWTs, and the cookie that carries them

**Taught** — Session = a *handle* (opaque ID in cookie) pointing at a server-side row that holds the truth; opaque-session vs JWT contrast with a full comparison table; revocation as the deciding property (sessions = one DELETE, pure JWT = nothing to delete until `exp`); the hybrid (short access JWT + revocable refresh token) named but waved off as not-yet for early SaaS; the hardened auth-cookie attribute-by-attribute (`__Host-`, `HttpOnly`, `Secure`, `SameSite=Lax`, `Path=/`, `Max-Age`); token entropy (≥128 bits CSPRNG) + constant-time compare; the session-row columns paired with the features they power; issue/refresh/revoke/expire lifecycle; session fixation + multi-device; the three read sites (proxy / layouts / actions).

**Cut** — JWT signing-algorithm names (HS256/RS256/EdDSA) dropped entirely from prose (just "signs with a secret it holds"); the CSRF/`SameSite` "no separate CSRF tokens needed" topic reduced to one inline clause deferring full coverage to Ch 054.

**Debts** — Deferred forward: full Drizzle `user`/`session`/`account` schema/DDL (Ch 052 L2 — note: lesson text says "next chapter"); `auth.api.getSession` call shape + `getCurrentUser`/`requireUser` (Ch 052 L4); production proxy gate / matcher / `?next=` (Ch 054 L1); `freshAge` ~10-min elevation re-prompt (named once, Ch 054 L2); active-sessions UI + cookie-cache staleness window (Ch 054 L3); full CSRF story (Ch 054); action-boundary re-reads identity/permissions from DB, never trusts cookie claims (seed for Ch 057). Referred back: cookie attribute palette + `Set-Cookie` (Ch 013); `HttpOnly`/`localStorage` XSS line + CSPRNG + constant-time compare (Ch 016).

**Terminology** — session = handle; "the source of truth lives on the server." Coat-check ticket analogy is the single spine metaphor (ticket = opaque ID; coat = server row; staff pulling the coat = revocation). opaque = random lookup key, no readable meaning; CSPRNG; JWT ("jot") = signed self-describing token; claim (`sub`/`iat`/`exp`); "signed is not encrypted" (tamper-evident, not secret; payload is base64url-readable). timing attack / constant-time compare. session fixation = pre-planted ID adopted post-sign-in, defended by regenerating token at sign-in.

**Patterns and best practices** — Canonical auth-cookie the rest of Unit 8 assumes: `__Host-session=<opaque>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000`. Defaults to hold exact: `expiresIn` 30 days, sliding `updateAge` ~1 day, token rotated on a cadence (~24h) **not per request**, `freshAge` ~10 min (forward-ref only). `Secure` relaxed only in dev/preview, never prod. Session-row minimum `(id, userId, expiresAt)`; named load-bearing columns: `lastActiveAt`, `userAgent`, `ipAddress`, `activeOrganizationId`, `impersonatedBy`. Sessions are stateful by design — that is the feature, not a tax. "Sign out everywhere" = `DELETE WHERE userId = ?`; active-device list = `WHERE userId = ? ORDER BY lastActiveAt DESC`. Never store user data (email/role) in cookie/JWT and trust it at the action boundary; never put session tokens in `localStorage`; never compare tokens with `===`.

**Misc.** — Opaque session is the 2026 default for browser SaaS; JWT is the explicitly-named special-case reach (edge-rendered routes / service-to-service only). Don't let the table read as a balanced toss-up — revocation is the hinge. All code is conceptual sketch (`findSession(token)`, illustrative cookie/JWT strings), marked not-real-API; no Better Auth/Drizzle/Next call appears until Ch 052 — do not upgrade. `proxy.ts` (Next.js 16, formerly `middleware.ts`) gates on cookie *presence* only, not real validation/authz (cookie-cache window can read a stale session).

## Lesson 3 — OAuth 2.1, PKCE, and the code-for-tokens exchange

**Taught** — OAuth = delegated *authorization* (act on user's behalf, never see password), repurposed for login by reading identity; OIDC layered on top via the `openid` scope is what does *authentication* (`id_token` + `userinfo`); the four roles (resource owner / client / authorization server / resource server) with provider playing both server roles; front-channel vs back-channel as the load-bearing trust distinction; the eight-step authorization-code-with-PKCE flow (verifier→challenge derivation, `state`, redirect-out, consent, callback-with-code, `state` check, back-channel token exchange closing the PKCE loop, token return, find-or-create user + mint own session); PKCE mechanism/threat/2.1-universal mandate; `state` as login-CSRF defense; exact-match redirect URIs + scopes-least-privilege + client-secret-per-env config surface; the access/refresh/id token taxonomy with `id_token` JWKS-verification (signature + `aud`/`iss`/`exp`).

**Cut** — Implicit/ROPC removed-grant rationale kept but compressed; provider quirks (Google/GitHub/Apple/Microsoft) named one line each as outlined (no expansion); `userinfo`-vs-`id_token` choice mentioned but not contrasted in depth.

**Debts** — Deferred forward: Better Auth `socialProviders` config + callback route `app/api/auth/callback/[provider]` + per-provider quirks detail (Ch 053 L8 — note lesson says "later"); `account` table / Drizzle schema for provider identities (Ch 052 L1); account linking + `(provider, providerAccountId)` uniqueness (Ch 053 L9); `safeNext(url)` in `lib/redirects.ts` open-redirect helper (referenced as existing, met "in the auth code later"). Referred back: authn/authz split + principal (L1); `__Host-` opaque session cookie as the flow's output + "signed ≠ encrypted" JWT readability (L2); cookies/`SameSite` CSRF seed (Ch 013/L2); Web Crypto / SHA-256 / CSPRNG (Ch 016 L1).

**Terminology** — front channel = browser/redirects/URLs, untrusted (visible in history, logs, `Referer`, extensions); back channel = direct server↔server HTTPS/TLS, where secrets are safe. Load-bearing sentence: "the `code` rides the front channel (must be useless alone); the secret + token exchange ride the back channel." PKCE ("pixy") = Proof Key for Code Exchange; `code_verifier` (held back) vs `code_challenge` = `BASE64URL(SHA256(verifier))` with method `S256`. `state` = anti-login-CSRF nonce echoed on callback. JWKS = provider's published public keys for `id_token` signature verify. "OIDC over OAuth" = the senior's two-level vocabulary. Punchline slogan: "the OAuth tokens are not your session" (they're proof-of-identity *input*; your `__Host-` cookie is the session).

**Patterns and best practices** — Canonical login scope is exactly `openid email profile` (least privilege; consent screen *is* the user's trust judgment; broader scopes go in a separate later consent). PKCE on every flow regardless of client type (OAuth 2.1, mandatory — never frame as SPA/mobile-only). Redirect URIs exact-match, no wildcards, one row per environment (prod/staging/`localhost`); trailing slash is the classic silent-break trap. Distinct `client_secret` per environment; secret rides back channel only. Redact `code` and `id_token` in logs like passwords. Never reflect untrusted `?next=` into post-login redirect — allowlist via `safeNext`. Always verify `id_token` (signature against JWKS + `aud`/`iss`/`exp`) before trusting any claim. For pure login, discard access/refresh tokens after reading identity; only store refresh token if app calls provider APIs later. OAuth callback errors stay opaque (enumeration discipline, same as password sign-in).

**Misc.** — Concept lesson, no real library API; all code illustrative (authorization URL via `AnnotatedCode`, token POST body, decoded `id_token` JSON, fabricated `google: { clientId, clientSecret, scope }` config *shape*) marked not-real-API — do not upgrade to real Better Auth calls. Better Auth named once as "what implements this." Live grants in 2026: auth-code+PKCE (focus), client-credentials (named), refresh (named), device-code (out of scope); 2.1 removed implicit + ROPC. The eight-step `DiagramSequence` (cumulative Mermaid sequence per step) is the load-bearing visual; two-lane front/back-channel figure is secondary.
