# Chapter 042 — Zod 4: the validation contract

## L1 — The eight builders

**Taught** — the eight core Zod 4 builders (primitives + `.optional`/`.nullable`/`.nullish`; `z.object`/`z.strictObject`/`z.looseObject`; `z.array`/`z.tuple`; `z.literal`/`z.enum`; `z.union`/`z.discriminatedUnion`; `z.unknown`/`z.never`), framed by the keystone idea that one schema declaration yields both a runtime parser (`.parse`/`.safeParse`) and an inferred type (`z.infer`).

**Cut** — the `z.enum` `.options` accessor (outline mentioned both `.options` and `.enum`; only `.enum` was taught/confirmed — a later lesson needing the values array for a `<select>` must confirm `.options` against the live runtime).

**Debts** — deferred to later L042 lessons: `z.email()`/string-format builders + number/string/date constraint catalog (L2, beyond `.int().positive()`); `.refine`/`.transform`/`.overwrite`/`.pipe` (L3); `.extend`/`.merge`/`.pick`/`.omit`/`.partial`/`.required`/`.readonly`, `z.input`/`z.output`, `z.record`, `z.intersection` (L4); `parse` vs `safeParse`, `ZodError`, `z.treeifyError`, the unified `error` option (L5); `z.coerce`/`z.preprocess`/`FormData` (L6); `drizzle-zod` + `jsonb` shape (L7). Server Actions themselves are Chapter 043. References back: ch009 (parse-`unknown`-then-narrow), ch004 (literal-union finite domains, illegal-states-unrepresentable, `never`).

**Terminology** — "the eight builders"; **schema = a runtime parser with a TypeScript type attached**; `z.infer<typeof schema>` is the bridge from runtime to type; "discriminator" (shared literal-typed field routing a `z.discriminatedUnion`). `z.email()` deliberately NOT used yet — email fields shown as `z.string()`.

**Patterns and best practices** —
- **Naming convention (load-bearing for all of Units 6–23):** schema const camelCase (`invoiceSchema` canonical, `createInvoiceSchema` action-input), inferred type alias PascalCase on the line directly below (`type Invoice = z.infer<typeof invoiceSchema>`), same file, in `/lib`. (Outline used PascalCase `InvoiceSchema` — that was overridden; follow camelCase.)
- `z.strictObject` is the senior default at untrusted boundaries (Server Action inputs, request bodies); default `z.object` strips unknown keys silently; `z.looseObject` only for forwarding undocumented external payloads.
- Prefer `.optional()` over `.nullable()` unless `null` is a deliberate domain value; `.nullish()` over chained `.optional().nullable()`.
- `z.enum([...])` over `z.union([z.literal(...)])` for finite string domains; pass the literal array inline (or `as const` if hoisted, else it widens to `string`).
- `z.discriminatedUnion` over `z.union` for tagged variants.

**Misc.** — canonical synthesis shape established as `createInvoiceSchema = z.object({ email: z.string(), quantity: z.number().int().positive(), status: z.enum(['draft','sent','paid','overdue']), tags: z.array(z.string()).min(1).max(100) })`; the four invoice statuses (`draft`/`sent`/`paid`/`overdue`) and this invoice-input shape recur as the chapter's running example. The `z.discriminatedUnion('kind', [...])` exemplar is a notification payload (`email`+`to` / `sms`+`phone`) — note `to` is `z.string()` here, NOT `z.email()` (deferred to L2); the discriminated-union exercise (`requestStateSchema`) tags on `status` (`loading`/`success`/`data` / `error`/`error`). v3→v4 legacy named once here: `.strict()`/`.passthrough()` → `z.strictObject`/`z.looseObject` (other v3 deprecations reserved for their own later lessons). Zod version pinned to 4.4.3 in the playground callout; `VideoCallout` videoId `ZPa9I_pvRU0` (Austin Davis) covers most of the builders.

## L2 — Formats over regexes

**Taught** — the v4 top-level format builders (`z.email`, `z.uuid` vs permissive `z.guid`, `z.url`/`z.httpUrl`, `z.cuid`/`z.cuid2`/`z.ulid`/`z.nanoid`, `z.ipv4`/`z.ipv6`/`z.cidrv4`/`z.cidrv6`, `z.iso.date`/`time`/`datetime`/`duration`, `z.jwt`, `z.e164`); number/bigint/date constraint chains (`.min`/`.max`/`.gt`/`.gte`/`.lt`/`.lte`/`.int`/`.positive`/`.nonnegative`/`.multipleOf`, plus top-level `z.int`); string constraints and normalizers (`.min`/`.max`/`.length`/`.startsWith`/`.endsWith`/`.includes`/`.regex`/`.trim`/`.toLowerCase`/`.toUpperCase`); the boundary rule that schemas validate shape/format only, cross-resource rules live at the action layer.

**Cut** — `z.base64`/`z.base64url` (in chapter-outline builder list, dropped — name if a later lesson needs binary-string validation); `z.email`'s `pattern` option and `z.url`'s `hostname` option named but not drilled.

**Debts** — forward-pointers planted: string→`Date` coercion + `z.coerce` vs `z.iso.datetime` (L6); `.transform`/`.overwrite` value/type transforms and the custom-logic-rules boundary (L3); `error` message customization / `errorContains` semantics (L5); money as `numeric`-string + decimal lib (L7); `z.uuid` strict-RFC override on `drizzle-zod`-generated `uuid` columns (L7); reading client IP from headers + open-redirect rule (Ch033); timezone-aware dates / string→`Date` in-memory story (Ch083); `email_suppressions` blocklist as the canonical action-layer cross-resource check (Ch048); cross-resource rules live in the Server Action body (Ch043, next chapter); JSON-Schema/OpenAPI as a reason top-level builders win (named once). References back: ch009 (`bigint` doesn't `JSON.stringify` cleanly; parse-`unknown`-at-the-wire).

**Terminology** — **"the format is the type, not a postfix modifier"** (the lesson's spine); **"reach for the named format builder before `.regex()`"** / "regex is the last resort"; tree-shake; the `z.uuid` (strict, app-generated IDs) vs `z.guid` (permissive, untrusted upstream) split and its migration trap (v3 `z.string().uuid()` was loose → maps to v4 `z.guid()`, not `z.uuid()`).

**Patterns and best practices** —
- Always top-level format builders; the v3 `z.string().email()` chain appears struck-through exactly once then never again.
- `z.url` accepts `javascript:`/`data:` by default — the `protocol` allowlist (`z.url({ protocol: /^https?$/ })` or `z.httpUrl()`) is non-negotiable for any URL the app renders/redirects to.
- `z.number()` rejects `NaN`/`Infinity` by default in v4 — `.finite()` is redundant (do NOT emit it; it's a v3 habit). **This overrides the Chapter 042 chapter-outline watch-out, which described v3.**
- Canonical shapes: quantity = `z.number().int().positive()`; money = `z.number().positive().multipleOf(0.01)` (schema-level approximation only, L7 owns real money); `z.int().positive()` preferred when integer-ness is the point.
- Free-text fields a stranger can submit get a `.max` (e.g. `z.email().max(254)`) as cheap abuse insurance; formats compose with constraints since a format is still a string schema underneath.
- Normalizers (`.trim`/`.toLowerCase`/`.toUpperCase`) mutate the parsed value (type stays `string`); don't expect parse output to equal input.
- `z.iso.datetime()` infers `string`, not `Date`; `z.date()` infers `Date` — pick by whether the value arrives off the wire as text or is already a `Date` in memory.
- Use the ID-encoding/CIDR builder that matches the upstream producer; don't editorialize.

**Misc.** — `signupSchema` introduced as a second running example (`email`/`token` uuid/`startAt` iso.datetime), extending L1's invoice shape. Valid-token fixtures use a v4-strict UUIDv7 (`0190d3a2-7b4e-7c1a-9f3d-2e5b8c4a1f00`); ISO datetimes use the `Z` suffix. ZodPlaygroundCallout pinned `version="4.4.3"`. Legitimate-regex exemplar: internal SKU `z.string().regex(/^SKU-\d{6}$/)` (business-invented shapes with no spec).

## L3 — Checks and transforms

**Taught** — the two extension points past the built-in catalog: refinements (custom pass/fail checks) and transforms (value reshaping). Single-field `.refine(predicate, { error })`; cross-field `.refine` on the object with `path: ['field']`; `.superRefine((value, ctx) => ctx.addIssue(...))` for one rule raising multiple issues; `.transform` (changes value AND inferred output type); `.overwrite` (value-changing but type-preserving, for normalization); `.pipe(nextSchema)` for a second full validation pass on transformed output; the v4 transform-runs-even-after-refine-fails behavior; and the pure-checks-in-schema vs side-effects-at-the-action boundary extended to custom logic.

**Cut** — nothing significant beyond what was already reserved; the chapter-outline's `error`-function-form drill was deliberately held to a one-line taste (L5 owns it).

**Debts** — forward-pointers planted: full unified `error` surface + function form, `ZodError`/issue anatomy (`code`/`path`/code-specific fields), `z.treeifyError`, `parse` vs `safeParse`/`parseAsync` (L5); `z.input`/`z.output` (the two types a transform splits) + `.pick`/`.omit`/`.extend`/`.partial` composition surviving refinements (L4); `z.coerce.number()` as the lighter alternative to `.pipe` for the string→number form case (L6); async refinements (L5 `parseAsync` + Unit 22); Server Actions + the typed `Result` error path + form rendering of `path` via `useActionState`/`treeifyError` (Ch043 next chapter, Ch044 L4); Architectural Principle #3 named-but-not-numbered (formally Ch043 L4); timezone correctness of the `Date` a transform lands (L7/Ch083). References back: L2's `.trim`/`.toLowerCase`/`.toUpperCase` normalizers (`.overwrite` is their general-purpose form); L2's first drawing of the schema-vs-action boundary (re-drawn here for custom logic).

**Terminology** — **the spine question "does this rule judge the value, or change it?"** (judge → refine, change → transform) routes every refine-vs-transform decision; **a schema is a pipeline** (base type check → constraints → refinements → transforms, each stage gating the next); **refinement = predicate, `true` means acceptable** (inverting this is the headline beginner bug); **refinements don't change the inferred type, transforms do** (the single sharpest refine/transform distinction, proved by the `^?`-doesn't-move keystone vs the `^?`-moves contrast exercise); **"every cross-field check names its `path`"** (the 2026 reflex); **checks live in a `checks` array on the schema** (v4) not a `ZodEffects` wrapper (v3).

**Patterns and best practices** —
- Single-field rule a built-in misses → `.refine`; cross-field rule → `.refine` on the object with `path`; one rule, many issues → `.superRefine` (over-engineering for a single-issue rule); convert to a different type → `.transform`; normalize keeping the type → `.overwrite`; validate transformed output with a real schema → `.pipe`.
- A cross-field `.refine` without `path` is a bug (issue anchors to object root, UI can't place it), not a shortcut.
- `.transform` reached for normalization is a type-drift bug (widens schema off `ZodString`, loses string methods) — the tell you wanted `.overwrite`.
- Write transforms safe over any structurally-accepted input; never assume an earlier `.refine` short-circuited the transform (v4 may run the transform anyway).
- Pure value-provable rules (refine/superRefine/transform/normalize) stay in the schema; database-lookup / external-call rules (uniqueness, slug-taken, plan-permits) live in the Server Action body after the parse. A schema needing a DB connection to parse has crossed the line.

**Misc.** — v3→v4 legacy named once here: refinement `ZodEffects` wrapper → `checks` array (call-site syntax `.refine(predicate, options)` unchanged, so NO before/after struck-through code — unlike L2's `.email()`). Key inconsistency to preserve, do not "harmonize": `.refine` options use `error:`, but `ctx.addIssue({ code: 'custom', message })` inside `.superRefine` uses `message:` (raw issue object). Snippets deliberately use `.parse(...)` for illustration before L5 introduces `parse` vs `safeParse` — do not "fix" to `safeParse`. Canonical transform exemplar: `z.iso.datetime().transform((s) => new Date(s))` lands a `Date` in an action's typed input. `passwordChangeSchema` ({ password: min 8, confirm }) is the keystone cross-field example; `startAtSchema` the transform contrast. The transform-after-refine trap is evidenced by a live `ZodPlaygroundCallout` plus a closing-CardGrid link to the real bug report `zod#4926` (cite as canonical source if a later lesson/quiz references it). `.pipe`'s staged validation is shown via a `DiagramSequence` (`PipeStages` component, string `"42"` → number `42`). ZodPlaygroundCallout pinned `version="4.4.3"`.

## L4 — Derive, don't duplicate

**Taught** — the derive-don't-duplicate reflex (one canonical schema, every variant derived) and the composition algebra: narrowing (`.pick`/`.omit` via `{ key: true }` masks), growing (`.extend`; fusing two named schemas via the spread `z.object({ ...a.shape, ...b.shape })`), modifier flips (`.partial`/`.partial({ key: true })`/`.required`/`.readonly`); the two specialist builders (`z.record(keySchema, valueSchema)`, `z.intersection`); the input/output type split (`z.input`/`z.output`/`z.infer`, where `z.infer` resolves to output, motivated by `z.iso.datetime().transform((s) => new Date(s))` whose `z.input` is genuinely `string` and `z.output`/`z.infer` is `Date`); plus `.describe` (field doc channel) and the recursive-schema `get`-getter, each named once.

**Cut** — nothing of substance; chapter-outline's `.merge` was demoted to a named-once deprecated legacy mention (replaced by the spread), and the outline's "`.readonly` is type-only" claim was corrected (it also `Object.freeze`s at runtime).

**Debts** — forward-pointers planted: `parse`/`safeParse` + `ZodError` shape (L5, snippets here use `.parse` for brevity); `z.coerce` mechanics + FormData-is-string-only (L6; this lesson motivates the input/output split with `z.iso.datetime().transform((s) => new Date(s))` — NOT a bare `z.coerce.date()`, whose input is `unknown`, not `string` — and forward-refs coercion mechanics to L6); `createSelectSchema`/`createInsertSchema`/`createUpdateSchema` generating the canonical schema then `.partial()`/`.omit()` on top (L7, named as "source is often generated"); cross-field `.refine` rendering its `path` via `useActionState` (Ch043/Ch044); branded IDs on a schema (Ch005 L4, not taught); `.readonly` for shared cached objects (Unit 14); `.describe` on LLM tool-input schemas (Unit 22). References back: ch004 L6 utility types (`.pick`↔`Pick`, `.omit`↔`Omit`, `.partial`↔`Partial`, `.required`↔`Required`, `.readonly`↔`Readonly` — the load-bearing scaffold, stated as "runtime twins, type comes along because it's inferred"); ch037 schema-is-source-of-truth (same ethic one layer up).

**Terminology** — **"derive, don't duplicate"** (the named principle: one canonical schema, variants derived, drift impossible-by-construction); **drift** (two representations of one fact fallen out of agreement, glossed via `Term`); **a schema is a value you can do algebra on** (take a schema, get a new schema + new type for free); **mask** (the `{ key: true }` argument shape, not an array — shared by `.pick`/`.omit`/`.partial`); **`z.infer` resolves to the output type** (phrased as behavior, NOT asserted as a verbatim doc "alias"); **intersection** (a value satisfying two schemas, the `&` to a union's `|`, glossed via `Term`).

**Patterns and best practices** —
- Never copy-and-edit a schema; derive every variant from one canonical source and let each derived name signal its role (`createUserSchema`/`publicUserSchema`/`updateInvoiceSchema`).
- Combining two object schemas → spread `{ ...a.shape, ...b.shape }` (v4-current; `.merge` deprecated, `.extend` quadratically costly when chained); adding a few fields to one base → `.extend` (same-named key overrides); anything non-object needing two schemas → `z.intersection`. Last-spread-wins on key collision.
- Derive update/PATCH schemas with `.partial()` every time — never a hand-written "every field optional" twin.
- `.pick`/`.omit` inherit the source's strictness mode; with default `z.object`, `.omit` removes a key from the shape but does NOT reject an input still carrying it (the extra key is stripped, not refused).
- `.omit` of a sensitive field (`passwordHash`) is compiler-enforced leak prevention — the inferred response type can't carry what the shape doesn't have.
- `.readonly()` is a real runtime guard (`Object.freeze` on parse) plus the `Readonly<T>` type — unlike ch004's type-only `Readonly`.
- `z.input` for a form/inbound contract that feeds a `.transform`/`z.coerce` schema; `z.output`/`z.infer` for the parsed shape the action body holds — reaching for `z.infer` on the input side is a silent bug that surfaces downstream. The canonical motivating schema is `z.iso.datetime().transform((s) => new Date(s))` (input `string`, output `Date`); do NOT cite `z.coerce.date()` for this split (its `z.input` is `unknown`).
- `z.record(keySchema, valueSchema)` needs both args in v4 (single-arg v3 form gone); default rejects unmatched keys (`z.looseRecord` passes them through); `z.enum` key triggers exhaustive-key checking. For object shapes that must preserve unknown keys, reach for `z.looseObject` (L1), not a `z.record` wrapper.

**Misc.** — `userSchema` (`email`/`passwordHash`/`name`/`avatarUrl`) is the lesson's spine entity for the derivation family; the chapter's `invoiceSchema` carries the `.partial`/PATCH beat; the input/output split uses an `issuedAt: z.iso.datetime().transform((s) => new Date(s))` field (NOT `z.coerce.date()`) so `z.input` is genuinely `string`, holding the chapter through-line. v3→v4 legacy named once with struck-through before/after: `.merge` → spread, single-arg `z.record` → two-arg. Cross-field `.refine` belongs on the *derived* schema where both fields coexist (`signupSchema = userSchema.pick(...).extend({ password, confirmPassword })`); the partial/refine footgun: `.partial()` under a cross-field refine makes it vacuous — derive before refining. Recursive pattern is the v4 `get children() { return z.array(categorySchema); }` getter (no `z.lazy`/cast; `z.lazy` still exists for compat). ZodPlaygroundCallout pinned `version="4.4.3"`.

## L5 — parse, safeParse, and the error contract

**Taught** — the four parse methods as a 2×2 of two binary choices ({throw, return} × {sync, async}): `safeParse` (boundary default, returns the discriminated-union result `{ success: true; data } | { success: false; error }`), `parse` (throws `ZodError`, trusted-edge only), `parseAsync`/`safeParseAsync` (escape hatch when a refinement is async — sync forms throw on hitting one). Plus the error contract end-to-end: `ZodError.issues` anatomy (`code`/`path`/`message` + code-specific fields like `minimum`, `keys`, `expected`/`received`), `z.treeifyError` (form-shaped nested tree mirroring input) vs `z.flattenError` (flat `{ formErrors, fieldErrors }`) vs `z.prettifyError` (log string), and the v4 unified `error` option (string + `(issue) => string` function form, the `issue.input === undefined` required-check idiom).

**Cut** — nothing of substance; `Result<T>`/`ok`/`err` and `useActionState` rendering were always reserved for Ch043/Ch044 and only the seam is shown.

**Debts** — forward-pointers planted: canonical `Result<T>` + `ok`/`err` helpers the `safeParse` failure branch maps onto (Ch043 L3) — **`{ ok: false }` placeholders in this lesson are NOT real `err(...)`, do not "fix"**; five-seam Server Action shape, `safeParse`-on-entry as first seam (Ch043 L2); form-side `path` rendering via `useActionState`/`<FieldError>` (Ch044 L3); `z.coerce`/`z.preprocess`/FormData-string boundary (L6, examples here parse already-typed objects); `z.config({ customError })` + per-parse `error` override wired for localization (Ch084); `unrecognized_keys` drift surfaced to monitoring (Ch092); async refinements at the network (Unit 22 LLM tool inputs). References back: ch004/ch009 + L1 `z.discriminatedUnion` (the `safeParse` result IS discriminant narrowing on `success`); L3's cross-field `.refine({ path })` (the `path` it sets is the issue `path` the form reads); L1 `z.strictObject` (source of `unrecognized_keys`); L3's `.refine(fn, { error })` (revealed as the same unified `error` surface).

**Terminology** — **"a parse is a fork"** (valid → typed value, invalid → `ZodError`; the mental model the whole lesson hangs on); **"return the expected, throw the unexpected"** (the two-channel rule — `safeParse` is the return channel for expected user error, `parse` the throw channel for impossible-if-correct programmer error; same rule as Code conventions §Error handling, first met operationally here); **trust boundary first, then bug-vs-outcome** (the decision *order* the StateMachineWalker encodes — there are two `safeParse` leaves, `parse` lives only at trusted ∩ should-never-fail); **"the schema authors the message, the form renders it"** (messages live on the schema, never the form component); **operator signal vs field error** (an issue with empty `path` / no matching input is logged, not rendered — the `path` tells you *whether* to render, not just where).

**Patterns and best practices** —
- `safeParse` at every untrusted boundary (form, request body, webhook, `searchParams`); `parse` only trusted server-internal (a `/lib` helper asserting a value it built, a test fixture, `env.ts` at boot). `parse` at a boundary → 500 instead of a field error (the headline beginner bug).
- A `try/catch` around `parse` is a code smell, not a fix — it re-implements `safeParse` by hand and a bare `catch` swallows every throw, miscategorizing real bugs as validation failures. Reach for `safeParse`.
- `if (!result.success) return ...` then use `result.data` past the guard — the branch is what earns the typed `.data`, no cast/`as`/`!`.
- **This course uses `z.treeifyError`** (nested, for the nested schemas a real SaaS form has) — pick one per project, don't mix with `flattenError`. Access shape is `tree.properties?.field?.errors?.[0]` with load-bearing optional chaining at every hop. `flattenError` only when the form is genuinely flat.
- Error messages authored on the schema via the unified `error` option (string for one message, `(issue) => string` for per-failure wording). v4 has no `required` issue code — a missing field is `invalid_type` with `issue.input === undefined`; that check is *the* way to author a "required" message.
- An `unrecognized_keys` issue (from `z.strictObject`) has empty `path` and no input to anchor to — the form ignores it, the action logs it (contract drift, not user error).

**Misc.** — v3→v4 legacy named twice with struck-through before/after: `error.format()`/`error.flatten()` (methods) → `z.treeifyError`/`z.flattenError` (top-level functions); the three-param `{ message, invalid_type_error, required_error }` → single `error` (the first two dropped entirely — neither maps to a real issue code — `message` deprecated-but-works). The canonical `safeParse` boundary snippet returns a `{ ok: false }` placeholder with a "fleshed out next chapter" comment, deliberately NOT `err(...)`. Running schemas reused unchanged: `createInvoiceSchema` (invoice fork exercise — starter omits `'overdue'` from the enum so one fixture fails until widened), `signupSchema` (email/password, the `ZodError` anatomy + treeify beats with two fields wrong). `ZodCoding` + any callouts pinned `version="4.4.3"`. The 2×2 grid, treeify-mirror, and v3/v4 migration blocks are authored HTML+CSS / struck-through `Code`, not graph engines. `VideoCallout` videoId `T4Q1NvSePxs` (CJ/Syntax, 8 min, `prettifyError`/`flattenError` + the `env.ts` boot crash); external CardGrid links Matt Pocock's Zod 4 tour (`xcm53k0ePmY`) + three zod.dev pages (basics, error-formatting, error-customization).

## L6 — Crossing the FormData boundary

**Taught** — the FormData→typed-domain bridge: every `formData.get(name)` is `FormDataEntryValue` (`string | File`); `Object.fromEntries(formData)` collects the form into a string-valued object, fed straight to `safeParse`; `z.coerce.{number,date,bigint,string}` reframed as L3's transform with the function fixed to a JS constructor (input `unknown`, checks run on the coerced value); the input/output split made concrete (form contract = `z.input`, parsed value = `z.output`); and the four FormData traps each with a naive→fix, plus `z.instanceof(File)` shape validation.

**Cut** — nothing of substance; the chapter-outline's `tags[]`-suffix HTML naming convention was softened to "name multi-valued inputs deliberately (a plural like `tags`)"; `z.coerce.string()` named in the list but not drilled.

**Debts** — forward-pointers planted: the `parsed.success` branch, the `Result<T>` shape, and the DB write (Ch043 — **every snippet deliberately stops at the `safeParse`, no `'use server'`/action body**); `<form action={...}>` wiring + rendering field errors back via `useActionState` (Ch044); the real upload pipeline — presigned URLs, R2, streaming (Ch068, this lesson does shape-validation only); timezone-correct `Date` handling (Ch083, pick coerce-vs-iso by usage site, don't open the tz story); `drizzle-zod` generating the base schema these form rules refine on top of (L7); `zod-form-data`/`zfd.*` deliberately NOT taught (hand-rolled patterns are the minimum-viable stack). References back: L3 (`z.coerce` IS a named transform; `z.iso.datetime().transform(s => new Date(s))` is L3's canonical date exemplar); L4 (`z.input`/`z.output`, the split coercion creates); L5 (`safeParse` default; snippets continue using it).

**Terminology** — **"the wire is strings; the schema is the bridge, and the bridge is where the type changes"** (the lesson spine); **`z.coerce` = a transform that runs a constructor** (`z.coerce.number()` ≡ `z.number()` preceded by `Number(input)`); **the four traps** (booleans, optional-numbers→zero, repeated-keys, dates-too-loose-about-format) as the senior payload; `FormData` and `multipart/form-data` and `MIME type` glossed via `Term`; **"pick the date shape by where the date is used next"**. (No `Invalid Date` `Term` — `z.coerce.date()` rejects garbage in 4.4.3, so that gloss was dropped.)

**Patterns and best practices** (the load-bearing form-boundary reflexes Unit 6's project chapter must follow) —
- Every form-consuming action opens with `const parsed = invoiceFormSchema.safeParse(Object.fromEntries(formData))` — one schema coerces+validates the whole form in one pass.
- **Number off a form** → `z.coerce.number()`; but an **optional** number maps empty→`undefined` first (`z.preprocess(v => v === '' ? undefined : v, z.coerce.number()...optional())`), else `Number('')` writes a silent `0`. Required-with-a-floor fields are safe (blank `0` fails `.positive()`).
- **Boolean checkbox** → `z.preprocess(v => v === 'on' || v === true, z.boolean())`, **never `z.coerce.boolean()`** (banned for form data). The real defect (verified vs 4.4.3): `z.coerce.boolean()` runs `Boolean(value)`, so it accepts EVERYTHING and rejects NOTHING — `undefined`→false, `'on'`/`'false'`/`'off'`/`''`→true (only empty string→false). It can never fail a parse, and it silently inverts literal-string booleans (`Boolean('false')===true`). An unchecked checkbox is an absent key (→`undefined`→false via the preprocess), not `'off'`/`'false'`.
- **String that spells a boolean** (a `<select>`/hidden field valued `"true"`/`"yes"`/`"on"`/`"1"`) → `z.stringbool()`; a real boolean from a JSON/programmatic caller → `z.boolean()`.
- **Date** → `z.iso.datetime()` if it stays text to the DB (infers `string`, a `timestamptz` column takes it), or `z.iso.datetime().transform(s => new Date(s))` if JS needs a real `Date` (validates the strict format *before* constructing) — **never bare `z.coerce.date()`** on a boundary. The real defect (verified vs 4.4.3): `z.coerce.date()` REJECTS true garbage (`'not-a-date'` fails), but it's TOO LOOSE about format — it accepts a date-only string like `'2026-01-15'` (`new Date` parses it to midnight UTC), where `z.iso.datetime()` requires a full timestamp with time + `Z` and rejects date-only. The format is part of the contract.
- **Array field** → `formData.getAll(name)` (override the key on the `fromEntries` object), since `Object.fromEntries` keeps only the last of repeated keys.
- **File** → `z.instanceof(File)` + `.refine` on `file.size` (incl. `> 0` for "actually uploaded", since some browsers yield a zero-byte File) and `file.type` (MIME); `File` is a Web API global available in Next.js Server Actions/route handlers, not browser-only.

**Misc.** — `invoiceFormSchema` is the canonical form-input artifact every Unit 6 Server Action reuses; final trap-free shape: `{ customerId: z.uuid(), total: z.coerce.number().positive().multipleOf(0.01), issuedAt: z.iso.datetime().transform(v => new Date(v)), archived: z.preprocess(v => v === 'on' || v === true, z.boolean()), notes: z.string().optional() }` + `type InvoiceFormInput = z.input<typeof invoiceFormSchema>`. **Terminology** — **"generate when the shape is a row, hand-write a `z.object` when it isn't"** (the lesson's senior payload — session payloads, webhook envelopes, third-party API responses, `searchParams` map to no table); **override map** (the optional second arg to a generator, keyed by column name, glossed via `(( ))`); **callback form** `{ number: (schema) => schema.min(1).max(50) }` (receives the *generated* base schema, chain onto it — do NOT rebuild from `z.string()` — drizzle-zod re-applies nullability *around* your result; **the safe default**) vs **direct-schema form** `{ payload: someSchema }` (*replaces* the column wholesale, drizzle-zod does **NOT** re-apply nullability — the lesson's sharpest footgun); **envelope** (a webhook's outer `{ type, data }` wrapper, glossed via `(( ))`); **int32 bounds** (glossed); **the source-of-truth chain** (`db/schema.ts` → drizzle-zod base → refine+`.omit` → `z.infer` types → Server Action inputs → form `name` attrs; a column change becomes a compile error not a runtime surprise).

**Patterns and best practices** (the canonical derive-from-table path Unit 6's project chapter must follow) —
- Match the generator to the boundary: row coming back → `createSelectSchema`; row going in → `createInsertSchema` (the one reached for most); patch → `createUpdateSchema`. Reusing the wrong one is the first slip (select forces the caller to supply `id`/`createdAt`).
- Refine on top of the generated base, never hand-write a parallel schema for a table (drift smell). Division of labor: **the generated base covers the DB's constraints (types, nullability, generated columns); you refine the API's *additional* constraints on top** (length caps, formats tighter than `text`, non-negativity the column type can't express).
- Override **callback form** when adding rules (preserves nullability); **direct-schema form** only when replacing the whole column shape, and re-add `.nullable()`/`.optional()` yourself.
- `.omit` the server-set columns (`id`/`createdAt` from DB, `organizationId` from session, `createdBy` from auth) so the input contract is exactly the user-supplied subset.
- Pair a `jsonb` column with one schema declared once next to the table: feed it to `$type<z.infer<typeof schema>>()` AND the override map — one declaration resolves to the column TS type, the insert validation, and the inferred input type.
- `CHECK` constraints are invisible to Zod — re-state the rule in both layers (DB backstop + Zod boundary guard); they don't share information.
- `numeric` infers as `string` (arbitrary-precision, no float lossiness) — convert at the boundary with a decimal lib, never on the schema.
