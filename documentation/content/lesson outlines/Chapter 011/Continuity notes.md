# Chapter 011 — The HTTP contract every endpoint signs

## Lesson 1 — Methods and the safe-to-retry contract

- **Taught** — Methods declare intent (read by CDN/prefetcher/retry layer), safety vs. idempotency (state-not-response), the 5-method 2x2 grid (GET safe+idem; PUT/PATCH-absolute/DELETE unsafe+idem; POST unsafe+non-idem; empty safe+non-idem cell), PUT replaces vs. PATCH diffs with `merge-patch+json` (default) vs. `json-patch+json` (escape hatch), the retried-POST double-charge failure and the `Idempotency-Key` UUIDv4 pattern (key per logical operation, server persists `(key, response)` before responding), Next.js route handler `export async function GET/PATCH/DELETE` as where verb-as-contract is signed, and the body-on-GET footgun.
- **Debts** — HEAD detailed in Ch.069 (R2 size reads); OPTIONS/CORS preflight in Ch.012; `fetch` API surface in Ch.015; `URLSearchParams` in Ch.012; `crypto.randomUUID()`/Web Crypto in Ch.016; Problem Details and status codes in L2; full header surface (incl. `Content-Type` as content negotiation) in L3; server-side `(key, response)` storage, TTL, unique-constraint race in Ch.063 (Stripe webhook uses `event.id` as natural key); route handler implementation + `authedRoute` wrapper in Ch.046/057; Server Actions as form-submission surface in Unit 6 (Ch.043).
- **Terminology** — "declarations of intent" frame; "safe" = no observable server-state side effect; "idempotent" = same final state after N calls (state, not response); "absolute diff" vs. "relative diff" qualifier for PATCH idempotency; "PUT = `=` assignment, PATCH = `Object.assign(current, patch)`" mental models; "Idempotency-Key" (capitalized header name, IETF draft `draft-ietf-httpapi-idempotency-key-header-07`); "one key per logical operation, not per attempt"; the red cell (unsafe+non-idem) as the cell that needs `Idempotency-Key`. `<Term>` tooltips authored: `safe`, `idempotent`, `merge-patch`, `json-patch`, `Idempotency-Key`.
- **Patterns and best practices** — UUIDv4 via `crypto.randomUUID()` for idempotency keys (distinct from UUIDv7 DB primary keys); always set explicit `application/merge-patch+json` or `application/json-patch+json` on PATCH (never generic `application/json`); persist-before-respond ordering on the server idempotency store; one Next.js `route.ts` exports one async function per supported method; structured reads go in query strings, never GET bodies; POST `/searches` is the honest shape when an operation needs a body and isn't a read.
- **Misc.** — Lesson uses `HTTP/3` on request-line examples (Ch.010 established HTTP/3 over QUIC as 2026 default). Lesson-specific components live at `src/components/lessons/011/1/` (`SafetyIdempotencyCard`, `MethodGrid`, `RetryStep`). Closing recall uses TrueFalse (Buckets already used for the grid). No external resources beyond MDN methods page, Stripe idempotency blog, and the IETF draft. No quiz at end of lesson (chapter quiz is L4).

## Lesson 2 — Status codes and the Problem Details body

- **Taught** — Status code as "declaration of outcome category"; the five classes (1xx/2xx/3xx/4xx/5xx) as semantic buckets that alerting/paging and retry policies key off; senior subset per class (1xx: 100/101/103; 2xx: 200/201+`Location`/202/204; 3xx: 301/308 vs 302/307 method-preservation split, 303 PRG, 304; 4xx: 400/401/403/404/405/409/410/422/429; 5xx: 500/502/503/504/507 with 502/503/504 as "load-balancer trio"); the three 4xx discriminations (400 vs 422 parse-vs-validation, 401 vs 403 identity-vs-permission, 403 vs 404 tenancy-hiding); 4xx/5xx on-call paging contract (5xx pages, 4xx doesn't); RFC 9457 Problem Details five core fields (`type`/`title`/`status`/`detail`/`instance`) with `application/problem+json` content type; the `errors: [{path, message}]` extension shape (Zod-issue carrier); status-line and body `status` field coherence rule.
- **Cut** — None significant; lesson followed outline scope closely. Added one VideoCallout (ByteByteGo 5-min recap) not in outline; outline explicitly noted "no video embed" but writer judged ByteByteGo bar acceptable.
- **Debts** — `problemResponse(...)` helper implementation in Ch.046 (route-handler chapter); `authedRoute` wrapper mapping 401/403/422/404 to Problem Details in Ch.057 (Unit 9 RBAC); `tenantDb` factory enforcing 404-on-cross-tenant at query layer in Unit 9; full header surface (`Location`, `Retry-After`, `Cache-Control`, `WWW-Authenticate`, conditional-request headers) in L3; CORS preflight 204 in Ch.012; i18n of `title`/`detail` in Unit 17; Sentry 500-grouping and 5xx alerting rules in Unit 19; Stripe webhook 400+`problem+json` in Ch.063; `201 Created`+`Location` shape in L3.
- **Terminology** — "declaration of outcome category"; "class-first" mental model; "senior subset" framing; "load-balancer trio" = 502/503/504; "tenancy-hiding rule" = 404 over 403 for cross-tenant; "post-redirect-get" = 303+GET reflex; "cost of dialect" framing for Problem Details motivation; "problem-type-specific extension" for fields beyond the core five; `<Term>` tooltips authored: `Problem Details` (RFC 9457), `Post-Redirect-Get`, `WWW-Authenticate`. Canonical reason phrase: **"Unprocessable Content"** (not legacy "Unprocessable Entity" — RFC 9110 2022 update).
- **Patterns and best practices** — Status code on response line and `status` field in body must match; set in one place server-side so they can't drift. Use `application/problem+json` (not `application/json`) on error responses. `type` URI is the version-stable client switch; `title`/`detail` text is free to change. `detail` is user-surfaceable — never include stack traces, raw exception messages, or internal IDs. Validation errors carry the `errors: [{path, message}]` extension built from `parsed.error.issues.map((i) => ({ path: i.path.join('.'), message: i.message }))`. Schema naming: `<verbEntity>Schema` (e.g. `createInvoiceSchema`). Problem Details `type` URI shape: `https://api.example.com/problems/<kebab-name>`. Senior defaults: 308 over 301, 307 over 302 (method-preserving redirects); 303 for PRG; 422 for validation failures; 404 (not 403) for cross-tenant access; never return 200 with `{ ok: false }` in body.

## Lesson 3 — Headers as the metadata channel

- **Taught** — Three-audience model (browser enforces / infrastructure acts / application reads) as durable scaffold; content negotiation (`Content-Type` is, `Accept` wants, `Content-Encoding` with `zstd` as 2026 reach, `Vary` extends cache key, `Content-Length`/`Accept-Language` named); `Cache-Control` directive surface (`private`/`public`, `max-age`/`s-maxage`, `no-store` vs `no-cache` non-synonym, `must-revalidate`, `stale-while-revalidate`, `immutable`) with four senior defaults (auth HTML: `private, no-store`; hashed assets: `public, max-age=31536000, immutable`; cacheable CDN HTML: `s-maxage=300, stale-while-revalidate=86400`; fast-changing API: `private, no-store`); conditional requests (`ETag`+`If-None-Match`→304, `If-Match`→412 for optimistic concurrency, `Last-Modified`/`If-Modified-Since` mentioned); authorization split (cookies for first-party browser, `Authorization: Bearer` for machines, `Basic` for internal only, `WWW-Authenticate` on 401); rate-limit signaling (IETF `RateLimit`/`RateLimit-Policy` structured fields RFC 9651, `Retry-After` on 429/503); security baseline six (HSTS, CSP, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy` deny-default, `frame-ancestors` via CSP retiring `X-Frame-Options`) plus `Reporting-Endpoints` companion; request-context headers (`Cookie`, `User-Agent`, `Referer`, `Origin` for CSRF, `Host`/`:authority`) and the trust-the-edge rule for `X-Forwarded-For`/`Forwarded` (RFC 7239); custom-header naming after RFC 6648 (no `X-` prefix; vendor token or bare); three-setter mapping (`next.config.ts headers()` static, `proxy.ts` per-request, route handler/response per-response).
- **Cut** — None significant; lesson followed outline scope closely.
- **Debts** — Cookie attributes (`HttpOnly`, `Secure`, `SameSite`, `__Host-`, `Partitioned`) and `Set-Cookie` deep dive in Ch.013; CORS (`Access-Control-Allow-*`, preflight) in Ch.012; `same-origin` tuple in Ch.012; Next.js 16 `'use cache'`+`cacheLife`+`cacheTag` in Unit 4; `authedRoute` wrapper with `WWW-Authenticate: Bearer` on 401 in Ch.057; full CSP nonce wiring + report-only rollout + `Reporting-Endpoints` endpoint in Ch.081 (including canonical `crypto.getRandomValues(...)`+base64 nonce swap); `rate-limit-headers.ts` helper in Ch.075; Server Action `Origin` CSRF check in Ch.054; HEAD on R2 in Ch.069; Better Auth's `__Host-`-prefixed `HttpOnly; Secure; SameSite=Lax; Path=/` session cookie in Unit 8; `Authorization`/`Cookie` redaction in logger in Ch.081; Stripe webhook `Stripe-Signature` in Ch.063.
- **Terminology** — "three-audience model" (browser/infrastructure/application); "metadata addressed to a specific audience" frame; "senior default" stated plainly per case; "trust-the-edge rule" / "count proxies from the right"; "load-bearing header for CSRF" (`Origin`); "irreducible six" for security baseline; "shared cache" (CDN, corporate proxy) vs "private cache" (browser); "the back-button leak" (auth HTML cached locally after sign-out); `proxy.ts` is Next.js 16 rename of `middleware.ts` (use `proxy.ts` throughout, runs on Node.js runtime; edge runtime no longer supported); structured-field shape per RFC 9651 (`"name";key=value` items with parameters); IETF `draft-ietf-httpapi-ratelimit-headers-11` as 2026 de-facto. `<Term>` tooltips authored: `If-Match`, `HTTPS`, `CSP`, `nonce`. NOT re-authored: `Idempotency-Key` (L1), `Problem Details` (L2).
- **Patterns and best practices** — `Cache-Control: private, no-store` for any response that varies by signed-in user (auth HTML and fast-changing API both); `public, max-age=31536000, immutable` only for content-hashed asset URLs; `Vary: Cookie` mandatory on cacheable per-user surfaces (moot when `no-store`); `ETag` opaque tokens over `Last-Modified` (second-precision can't disambiguate); first-party browser sessions in `__Host-`-prefixed `HttpOnly; Secure; SameSite=Lax; Path=/` cookies (not Bearer); machine clients use `Authorization: Bearer`; never log `Authorization` or `Cookie` (configure redaction); count `X-Forwarded-For` from the right and stop at first untrusted hop, or read vendor edge header (`x-vercel-forwarded-for` on Vercel); emit `Forwarded` (RFC 7239) from your edge; no `X-` prefix on custom headers — use vendor token (`Stripe-Signature`, `Svix-*`) or bare (`Request-Id`, `Idempotency-Key`); never set `Content-Encoding` by hand (platform negotiates); CSP shape is nonce-based with `'strict-dynamic'` (fresh nonce per request, same nonce on inline `<script>` tags); `Referrer-Policy: strict-origin-when-cross-origin` senior default; `Permissions-Policy` deny-by-default with per-feature grants; HSTS with `includeSubDomains; preload`; status-line `HTTP/3` on every `http` fence (chapter convention). Three-setter file rule: static headers → `next.config.ts headers()`; per-request data (CSP nonce, request-id, rate-limit, per-tenant) → `proxy.ts`; per-response (`Cache-Control`, `ETag`, `WWW-Authenticate`, `Content-Type: application/problem+json`) → route handler/Server Action `Response`. Route handler snippet uses plain `new Response(...)` (not `NextResponse.json(...)`; Unit 4 introduces `NextResponse`). `proxy` is a named export from `proxy.ts` (per Next.js 16 file convention).
