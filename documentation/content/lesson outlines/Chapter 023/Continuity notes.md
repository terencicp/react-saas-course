# Chapter 023 — The render model

## L1 — What triggers a render

**Taught:** A component is a function React re-runs; rendering = calling it to get a JSX tree, then diffing; the three (and only three) triggers (own state setter, ancestor re-render, subscribed context change) with "a prop change is NOT a trigger" as the load-bearing correction; `Object.is` as React's sole prop-equality rule (primitives by value, objects/arrays/functions by reference); inline JSX literals (`{{...}}`, `[...]`, `() => {}`) churn identity every render; the React Compiler auto-memoizes so you write natural code; renders are cheap, commit is the expense, profile before optimizing.

## L2 — Reconciliation and the key prop

**Taught:** Reconciliation = the diff between Render and Commit (new element tree vs last → minimum DOM ops), motivated by O(n³) general tree-diff → two O(n) heuristics; heuristic 1 (different type at a position → tear down + rebuild subtree, all child state/DOM lost) and 2 (same type → keep node, patch only changed attributes); siblings matched by **position** when no key, so state/refs belong to the slot not the item; the index-as-key bug (re-pins identity to position, breaks on reorder/filter/insert-not-at-end, survives append-only — hence hides in dev); `key` = data identity (`item.id`, `post.slug`) that travels with the item; generate-once-at-creation for client ids vs the fatal `key={crypto.randomUUID()}`/`Math.random()` inline-in-map (remounts every row every render); `key` is per-sibling-set unique (not global) and reserved (unreadable as `props.key`, pass id again separately); the unified remount rule (state survives only if type AND key—or position without key—all stay the same; change any one → remount = full reset); conditional-render footgun (stable type preserves state, type swap resets); fragments transparent to reconciliation, `<>` can't take a key (use `<Fragment key>`).

## L3 — The purity contract

**Taught:** A component must be a **pure function of props and state** — same inputs → same JSX tree, *and* nothing outside its local scope changes while rendering; stated as React's side of a deal (you keep render pure, React gets to run/skip/pause/restart/reorder renders freely). The two-rule contract: (1) don't mutate props/state during render (treat readonly); (2) no side effects during render. Why it's non-negotiable: React reserves four rights — Strict Mode dev double-invoke, concurrent pause/restart (abandoned renders never commit), Compiler skip-the-call, out-of-order/skipped renders — each safe against pure code, a landmine against impure. The opening `<Counter>` (`renders += 1` on a module-level `let`) breaks three ways from one cause: counts-by-2 in dev, freezes under Compiler, two mounts corrupt a shared variable. Compiler is the 2026 stakes: auto-memoizes pure components, **silently skips** violators (no error/warning) — missing DevTools badge is the only tell. Violation catalog + fixes (mutate prop → spread new array; mutate state → setter; `Date.now()`/`Math.random()` in render → pass as prop; random key/id → stable id or `useId`; ref-write in render → handler/effect; `localStorage`/`document`/`fetch` in body → handler/effect). The boundary sharpened: **never mutate something that outlives this render** — local scratch arrays (`const rows = []; rows.push(...)`) are pure; same `.push()` on `props.items` is a violation. Two homes for side effects named as a map: **event handlers** (response to user input) and **`useEffect`** (after commit, to sync with outside React) — reach-for-handler-first reflex planted.

## L4 — State is a snapshot

**Taught:** State is a **snapshot** — when React renders, it bakes each state value into that render as a frozen constant; inside one render (JSX + every handler/callback defined in it) `count` cannot change, the setter only *asks for a new render*. Stated as the direct consequence of `UI = f(state)` + purity ("state changes between renders, never during one"). The canonical bug: `setCount(count + 1)` ×3 increments by **1** (all three reads see the same snapshot → three `setCount(1)` → next render shows 1). The two mechanisms underneath: (1) the **update queue** — each setter pushes an entry; a *value* entry means "replace with this" and overrides prior entries, so three identical replaces collapse to one; (2) **automatic batching** — all setters in one event group into a single re-render (React 19: unconditionally, incl. promises/`setTimeout`/async; *not* across separate events — two clicks = two renders). The fix: the **updater form** `setCount((c) => c + 1)` — receives the pending queued value, so three stack `0→1→2→3`. Rule is **conditional**: updater when next-depends-on-prev OR setter runs after render (timers/promises/async/subscriptions/effects); direct form when independent (`setCount(0)`, `setUser(fresh)`, `setIsOpen(true)`). Corollaries: setter is queued not immediate (`setCount(x); console.log(count)` logs OLD), `setState` returns nothing. **Stale closures** — a `setTimeout` callback closes over its render's `count`; two fast clicks both capture the stale value → one increment; updater form fixes it (reads queue, not closure) — framed as the *same bug* as triple-click. Immutable updates at the call site: mutate-then-`setUser(user)` bails via `Object.is` (same reference, screen frozen); three idioms — `{ ...user, name }`, `[...items, x]`, `items.filter(...)` (and `map` to replace); spread is **shallow** (nest spreads per level); same-reference no-op is deliberate-but-bug-when-accidental. `flushSync(() => …)` from **`react-dom`** = rare opt-out forcing synchronous render+commit before next line (measurement / non-React API only). Closing design cut: separate `useState` (independent values, clean primitive sets) vs grouped object (`{ ...form, email }` values that share a lifetime, spread ceremony).

The "When sequential setters are fine" content was folded into the updater-form `:::tip` (outline allowed this) rather than its own header.

## L5 — Remounting with key

**Taught:** The L2 remount mechanism (type/position/`key` change → unmount old instance, mount fresh) *cashed in as a deliberate state-management move* — change the `key` on purpose to discard a stateful child's local state when the identity behind it changes. Spine: **a `key` is identity, and you get to choose it.** A key change does its work in the **Reconcile** box (new element's key matches no existing instance → React builds a new one); the reset is the natural consequence, not a side effect. The canonical pain: the **record-bound form** (master-detail; switch Alice→Bob and Alice's half-typed edits leak onto Bob because the reused instance's `useState(user.email)` only seeds at mount — "state belongs to the slot, not the user"). Three-way comparison: bug (no key) → wrong fix (`useEffect` that re-syncs each field on `[user.id]`: brittle/per-field, fires a beat late = flash of stale data, the "you might not need an effect" smell) → right fix `<UserForm key={user.id} user={user} />` (one attribute, can't go stale, scales to any field count for free). Two more "faces" of the same trick, taught fast as variations: **animation replay** (`<Toast key={messageId}>` remounts → CSS entrance animation re-fires, guarded `motion-reduce:`) and **button-bump reset** (`useState(0)` counter, `key={resetKey}`, `setResetKey((k) => k + 1)` on click — manufacture an identity when none exists). The durable skill = the **decision order: persist? → derive? → lift? → key** (key is the *last* reach), delivered via a `StateMachineWalker`. Honest costs: remount runs every effect cleanup→rerun, refs re-attach, animations replay, DOM recreates (imperceptible mostly, measurable for heavy mount logic); **key the smallest subtree that owns the reset target.** Two foot-guns: unstable key (`key={Math.random()}`/`Date.now()` inline → remount-every-render loop, focus lost, effects thrash) and resetting state the user wanted kept (scroll/expanded panel/sub-selection).

## L6 — Synthetic events

**Taught:** `onClick`/`onChange`/`onSubmit` are NOT a new event model — they're Ch 014's native DOM model wrapped, typed, and delegated; spine: **the surface is the one you already learned.** A `SyntheticEvent` carries the same API (`type`/`target`/`currentTarget`/`preventDefault`/`stopPropagation`, still bubbles through the React tree) plus `e.nativeEvent` escape hatch (only realistic use: `stopImmediatePropagation`, which has no synthetic equivalent). Wrapper survives 2026 not for browser-quirk normalization (evaporated) but as the **integration seam with render scheduling** — dispatch runs through React's tree so setters inside one handler batch (ties to L4). Event pooling / `e.persist()` **removed in React 17** — synthetic events are ordinary objects, hold references freely (myth retired in `:::note`). **Typed handlers are generic over the element**: `MouseEvent<HTMLButtonElement>`, `ChangeEvent<HTMLInputElement>`, `KeyboardEvent<HTMLInputElement>`, `FormEvent<HTMLFormElement>`, imported type-only from `react`; the four daily ones shown, rest (`FocusEvent`/`PointerEvent`/etc.) same pattern. **Inline arrow in JSX → no annotation** (prop position infers); **extracted to named `const` → annotate** (no prop position left) — same "infer at boundaries, annotate at seams" stance. Importing `MouseEvent` from `react` shadows the global DOM type intentionally. **Reach for `currentTarget`, not `target`**: `currentTarget` typed against the bound element (`.value`/`.checked`/`.name`/`dataset` free), `target` is bare `EventTarget` (no `.value`) — `e.target.value` is a *TS error*, not runtime; they're the same node except under delegation. **Delegation lives at the React root container** (since React 17, one listener per event type, NOT per-button, NOT at `document`) — click bubbles native DOM up to the single root listener, React then *dispatches* a synthetic event down its component tree. Load-bearing consequence: **a React child's `stopPropagation()` now stops the event before it reaches `document`** (root sits below `document`), so global native `document` listeners / third-party DOM libs above the root get silently suppressed — exact OPPOSITE of pre-17, where React listened at `document` and couldn't un-fire it (why stale articles mislead). **`onChange` fires on every keystroke** (wired to native `input`, not native `change`/blur) — the lever live-updating UI pulls. `onSubmit` + `preventDefault` taught as *event mechanic only* (cancels full-page reload), with the forward note that 2026's `<form action={fn}>` retires most manual `preventDefault`. **`e.key`** is the keyboard reach (`'Enter'`/`'Escape'`/`'ArrowUp'`/`' '`; `keyCode`/`charCode`/`which` deprecated; `e.code` = physical key); modifiers are booleans (`e.metaKey`/`ctrlKey`/`shiftKey`/`altKey`, e.g. ⌘K = `e.key === 'k' && e.metaKey`); reach `onKeyDown` not deprecated `onKeyPress`. **Pointer events** (`onPointerDown`/`Move`/`Up`) = unified mouse/touch/pen primitive (`e.pointerType`), replacing legacy `onMouseDown`+`onTouchStart`; **pointer capture** (`setPointerCapture(e.pointerId)`) keeps events flowing during a drag once the cursor leaves the element. Three closing reaches: **capture-phase** variant appends `Capture` (`onClickCapture`); **`e.currentTarget.value` is always a string** (even `type="number"`/`type="date"` — convert in handler, don't blind-`Number()`); **`onScroll` doesn't bubble** in React.

**Cut:** Outline scope delivered. Event-prop *naming-convention* taxonomy section (camelCase + `on` prefix as its own beat) folded into the typed-handlers section rather than separate. The pre-17 native-handler *ordering* nuance ("document handlers run before React's") was reframed entirely around the `stopPropagation`-reach story (the load-bearing, corrected version) rather than restated as ordering rules.

**Terminology:**
- **synthetic event** — `Term`-defined: React's cross-browser wrapper around native `Event`, same API, bubbles through React's component tree.
- **bubbles** / **currentTarget** / **target** — `Term`-defined one-line refreshers from Ch 014 (assert, don't re-derive).
- **root container** — the DOM node React mounts into (via `createRoot`, framework-wired in Next.js); where the single per-event-type listener lives; the anchor for the `stopPropagation`-reach rule.
- **pointer capture** — `Term`-defined: routes further pointer events to one element even after the pointer leaves its bounds; the reason drag handlers use pointer events.
- **`preventDefault`** — `Term`-defined refresher: cancels the browser's default action (here the full-page form submit).
- Spine phrase: **"`onClick` is the native model you already know, wrapped and typed."** Reflex pair: *inline → no annotation; extracted → annotate.* Reflex word: **reach for `currentTarget`.**
- Continuous examples: `SaveButton`/`handleSave` (minimal `onClick`), email/search controlled input (`onChange`), generic + `Mute` button (root-delegation diagram/MCQ).

**Patterns and best practices:** Handlers are `handle<Event>` arrow `const`s; event param named `e` (idiomatic for short handlers, sanctioned). Event types are **type-only imports** (`import type { MouseEvent, ChangeEvent, FormEvent } from 'react'` — `verbatimModuleSyntax` requires `type`). **Inline handlers carry no annotation; extracted handlers annotate the parameter** (inference-led TS). **Default to `e.currentTarget`** for reading `.value`/`.checked`/`.name`/`dataset` off the bound element — never cast `e.target`. Reach `onKeyDown` over deprecated `onKeyPress`; `e.key` over deprecated `keyCode`/`which`. For any cross-input (mouse+touch) interaction reach `onPointer*`, never `onMouseDown`+`onTouchStart`. All interactive examples use real semantic HTML — `<button>`/`<form>`/`<input>` with associated `<label htmlFor>` — never click handlers on `<div>`s, even throwaway demos. **Project chapters:** synthetic-event handlers and `e.key === 'Escape'` reaches should follow these (Ch 027 shadcn overlays use them idiomatically).
