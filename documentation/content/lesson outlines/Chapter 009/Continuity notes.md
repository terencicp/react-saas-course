# Chapter 009 — JSON, classes, and the Temporal pivot

## Lesson 1 — JSON at the wire boundary

- **Taught** — JSON as a boundary codec (four wire sites: Server Action body, route handler response, webhook POST, `localStorage`); the parse-to-`unknown`-then-Zod two-step discipline; four serialization holes (`undefined`, `Date`, `BigInt`, `NaN`/`Infinity`); narrow uses for `reviver` (prototype-pollution defense) and `replacer` (log redaction); the `parseJson` helper as the single seam.
- **Cut** — `JSON.stringify` signature shapes / `space` argument depth (kept to one watch-out line); `toJSON` hook on custom classes (deferred to lesson 2); stable-key-order canonicalization moved to a single watch-out bullet; ES2022 details and streaming JSON parsers fully dropped.
- **Debts** — Zod surface (`z.iso.datetime()`, `z.coerce`, `.transform()`, `.safeParse()`) deferred to chapter 042; route handler `.json()` mechanics to chapter 015; Server Action `FormData` to chapter 043; `Date` → `Temporal` codec to lesson 3 of this chapter; custom `toJSON()` on classes to lesson 2 of this chapter; pino redaction to chapter 092.
- **Terminology** — "the wire is `unknown` until validated" (load-bearing slogan); "one codec, four directions"; "the seam" (where `parseJson` lives); `<Term>` tooltips: *wire*, *Zod*.
- **Patterns and best practices** — Every JSON boundary uses `parseJson(raw, schema)` from `lib/json.ts`; explicit `: unknown` annotation on `JSON.parse` result; `SyntaxError` / `ZodError` catches at transport-layer wrappers only, never per call site; `structuredClone` for deep-clone, never `JSON.parse(JSON.stringify(...))`; 64-bit IDs (Stripe, Postgres bigint) modeled as `string` in domain types; `null` means "explicitly absent in JSON", `undefined` means "not set in TS"; production JSON uses no `space` argument; `safeReviver` strips `__proto__` and `constructor` keys at the seam.
- **Misc.** — `parseJson` helper signature uses `<Schema extends z.ZodType>` and returns `z.infer<Schema>` (the writing diverged slightly from outline's `<T>(raw, schema: ZodType<T>): T` form but is equivalent). Lesson assumes `invoiceWebhookSchema`, `draftSchema`, `webhookSchema` as placeholder names — reuse if later lessons need invoice/webhook examples. Mentions a 2025 prototype-pollution attack on "major SDKs" without naming Axios specifically (outline mentioned Axios).

## Lesson 2 — Classes, narrowly

- **Taught** — The three legitimate `class` triggers (custom `Error` subclass, third-party SDK adapter wrapper, rare stateful domain object); the five-primitive minimum surface (`constructor` with one options-object, `readonly`, `#private`, arrow-field methods only on detachment, `static` factories); the six refused primitives (inheritance beyond `extends Error`, `abstract class`, mixins, decorators, getters/setters, class expressions); TS `private` vs runtime `#private`; reference equality vs `instanceof` in-realm; `Cart` worked example with `toJSON()` paying off L1's debt.
- **Cut** — TS parameter-property sugar (`constructor(readonly id: string)`) dropped entirely; `Object.create(null)` for prototype-free records dropped (outline named once, cut as niche); circular construction watch-out dropped; `Date.now()` stopwatch note moved to L3.
- **Debts** — Server/client module boundary directives (`'use server'`, `'use client'`, `'server-only'`) named in `BillingClient` example, deferred to Unit 4 / ch 010; the canonical `BillingClient` three-method `lib/billing/` surface deferred to Unit 12; full Stripe SDK at depth deferred to Unit 12; React function-component model deferred to ch 023+; deep equality on records assumed from ch 001 L1.
- **Terminology** — "trigger before tool" (decision posture); "module of functions over a typed record" (the default shape); "the wire is records, not classes"; "module-level singleton" (`export const billing = BillingClient.fromEnv(env)`); `<Term>` tooltips: *hash-private*, *arrow-field method*, *module-level singleton*, *realm* (recall from ch 008 L2). `class UserService` named as the canonical anti-pattern.
- **Patterns and best practices** — Records + functions are the default; `class` only on the three triggers. Constructor: one options-object parameter, body is assignment-only, no validation/side-effects (move to `static` factory). `readonly` everywhere unless mutation is the lesson. `#private` (never TS `private`) for any field holding a secret or implementation detail. Regular methods by default; arrow-field only when detachment is in the call path. `static fromX()` factories over overloaded constructors. Wrappers carrying secrets get `import 'server-only'` at the top. Module-level singleton (`export const billing = BillingClient.fromEnv(env)`) for stateless wrappers; per-request instance only for request-scoped state. Implement `toJSON()` explicitly on any class that crosses the wire (the `Cart` template). `instanceof` for in-realm narrowing on non-error classes; `name` literal discriminant for errors that cross realms. Never use `class` to group related functions — the module is the grouping primitive.
- **Misc.** — Canonical SDK-wrapper example is `BillingClient` wrapping `Stripe`, with `apiVersion: '2026-04-22.dahlia'` (May 2026 current; defer to SDK's pinned literal if it diverges). Canonical stateful-domain example is `Cart` with `#lines: Map<string, CartLine>`, regular methods (not arrow fields), and an explicit `toJSON()` returning `{ lines, totalCents }`. `BillingClient` `createCheckoutSession` is arrow-field for detachment-likely posture even though the lesson's direct calls don't strictly need it — call this out if Unit 12 refactors to regular method. The lesson pays off L1's `toJSON()` debt via the `Cart` example without generalizing to "always implement `toJSON`".

## Lesson 3 — Date's problems and the Temporal pivot

- **Taught** — Eight `Date` design flaws in three clusters (API: zero-indexed months/one-indexed days, `setX` mutation, no duration type; semantic: space-vs-`T` UTC-vs-local parse, `Invalid Date` sentinel, host-dependent `toString`; precision: year-100 heuristic, millisecond-only floor); three canonical production bug vignettes (DST reminder drift, date-line invoice, audit-log timezone smear); the five Temporal types at one-line depth (`Instant`, `ZonedDateTime`, `PlainDate`, `PlainDateTime`, `Duration`); runtime story (TC39 Stage 4 March 11 2026, unflagged in Node 26 May 5 2026, Node 26 LTS October 2026, `temporal-polyfill` ~20 KB for Node 24 gap, browser status non-blocking); the `lib/temporal.ts` re-export seam with optional runtime-guarded variant; `Date` carve-out (SDK ingress + stopwatch); three muscle-memory rules.
- **Cut** — Full Temporal arithmetic surface (`add`/`subtract`/`since`/`until`/`with`/`round`) deferred to ch 083 L5; no video callout per outline but lesson includes one Pixel Grid UI VideoCallout; no third exercise (kept type-pivot focus).
- **Debts** — Drizzle codecs for `timestamptz`/`date` to ch 083 L1-L2; `users.timeZone` profile column to ch 083 L3; DST-aware Trigger.dev schedules to ch 083 L4; full arithmetic to ch 083 L5; `Intl.DateTimeFormat` locale rendering to ch 084 L3; Zod `z.iso.datetime()`/`z.iso.date()`/`.transform()` to Temporal in ch 042; ESLint `no-restricted-imports` rule to enforce single import seam (later, unnamed chapter); RSC-wire encoding of `Temporal.*` as ISO strings noted but full discussion deferred to ch 083.
- **Terminology** — "The type tells you what you have" (key shift slogan); "more types, not more methods" (the cure); "the pivot" (three rules); "the parse seam" (recall from L1, applied to `Temporal.from()`); the three bug names (DST reminder drift / date-line invoice / audit-log timezone smear); `lib/temporal.ts` as the canonical file name; `@/lib/temporal` as the canonical import path; `<Term>` tooltips: *DST*, *IANA tz*, *sentinel*, *polyfill*, *seam* (recall).
- **Patterns and best practices** — Every project gets `lib/temporal.ts` re-exporting `Temporal` from `temporal-polyfill` (Node 24) or `globalThis.Temporal ?? TemporalPolyfill` (multi-runtime). All application code imports `Temporal` from `@/lib/temporal`, never from `temporal-polyfill` or `globalThis` directly (mixing breaks `instanceof`). Construction via static factories (`from`, `fromEpochMilliseconds`, `Temporal.Now.*`) — never `new`. Every Temporal value is immutable; all operations return new instances. `Temporal.X.from()` throws `RangeError` on bad input — catch at parse seams only (route handler, Server Action wrapper, adapter), same discipline as `JSON.parse`. `Date` is forbidden in domain code; allowed only at SDK ingress (convert via `instantFromDate`/`instantFromUnixSeconds`) and stopwatch measurements (`Date.now()`/`performance.now()` — prefer `performance.now()` when available, sub-millisecond and monotonic). Codec helpers (`instantFromDate`, `instantFromUnixSeconds`, `dateFromInstant`) live in `lib/temporal.ts` only, called from SDK adapters (canonical: `lib/billing/` in Unit 12). 2026 projects do NOT install `date-fns`, `dayjs`, `luxon`, or `moment`. Type-to-meaning map: `Instant` for stored timestamps (`createdAt`, webhook arrival) → `timestamptz`; `PlainDate` for calendar dates (`dueDate`, `birthDate`) → `date`; `ZonedDateTime` for DST-aware scheduling (derived, not stored); `PlainDateTime` rare ("5 PM local" with no tz commitment); `Duration` for spans (computed, not stored).
- **Misc.** — Codec signatures pinned: `instantFromDate = (d: Date): Temporal.Instant => Temporal.Instant.fromEpochMilliseconds(d.getTime())`; `instantFromUnixSeconds = (s: number): Temporal.Instant => Temporal.Instant.fromEpochMilliseconds(s * 1000)`; `dateFromInstant = (i: Temporal.Instant): Date => new Date(i.epochMilliseconds)`. Stripe wire is Unix seconds (10-digit) wrapped as `Date` in SDK — both conversion paths exist for this reason. Canonical bug-example vignette: Sydney user sees "due May 14" when invoice was `2026-05-15T00:00:00Z` (date-line invoice). Canonical scheduled-job: "every Monday at 9 AM in user's timezone" (DST drift). Polyfill choice is `temporal-polyfill` (FullCalendar, ~20 KB) over `@js-temporal/polyfill` (~45–60 KB). ISO 8601 extension syntax `[Australia/Sydney]` used to attach IANA tz to `ZonedDateTime`. Lesson 3 components live at `src/components/lessons/009/3/`: `TemporalTypesTable`, `DateVsTemporalComparison`, `TemporalSupportMatrix`.
