## Lesson framing

Third and final teaching lesson of Ch049. Lessons 1–2 made the HTML version *render* (component vocabulary, `<Tailwind>`, the preview-server iteration loop). This lesson makes the message *readable by everyone who doesn't see the HTML the designer eyeballed*: the screen-reader user, the plain-text-only client, the OS-level color inverter, the 200%-zoom reader, the corporate gateway. The chapter outline calls this a "pattern-and-checklist lesson — the discipline is the lesson, the syntax is small." Honor that: the durable takeaway is a posture and a checklist, not new API surface.

**Archetype.** Pattern-and-checklist. Most of the lesson is *judgment* (what to check, where the check lives in the template), not *typing*. Three distinct topics under one unifying frame.

**The unifying frame (the senior question).** The intro must land one mental shift: *the render you eyeballed in the preview is one of several renders that ship.* Same message, multiple surfaces — HTML, plain text, inverted-dark, zoomed, gateway-mangled. A senior designs for the whole set up front, not after a complaint. Each correctness check lives at a specific spot in the template's JSX — that "where does the fix live" through-line is what keeps this from being an abstract a11y lecture.

**Three topics, taught separately, each with its mental model:**
1. **Plain-text fallback** — every transactional email is `multipart/alternative` (two bodies, client picks one). The text body is generated, never hand-written. The senior reflex is to *read* the generated text and treat its incoherence as a bug in the HTML JSX.
2. **Accessibility checklist** — a *tighter, shorter* subset than web a11y (no JS, no ARIA patterns, table substrate) but with *stricter* numeric floors (font size, contrast, touch target). Frame it as a per-template checklist that pays for itself, with each item mapped to the exact JSX attribute/element.
3. **Dark mode** — the three-tier client-behavior model (no transform / partial invert / full invert) is the load-bearing mental model. Defensive-by-default + opt-in head plumbing for clients that respect it. The discipline: the *light* template must already pass contrast; dark mode is a courtesy layer, never an accessibility crutch.

**Cognitive-load sequencing.** Plain text first (smallest, builds directly on L1's "you get the text part free"), then accessibility (a checklist — naturally chunked), then dark mode last (most nuanced, has the heaviest model). Within dark mode, teach the *three-tier model* before any code, then the minimal plumbing, then the logo-swap as a named-not-built reach.

**Critical factual corrections vs. the chapter outline (verified June 2026 — see Scope/Fact-check notes).** The outline leans on `render(<T/>, { plainText: true })` as the generator. That option is **deprecated**. Current reality, consistent with L1 and the continuity notes:
- The **production** plain-text body is auto-generated by the **Resend SDK** from the `react` prop — free, nothing to call. You opt *out* by passing `text: ''` or a custom string. This is the path the project uses.
- The **manual/test** utility is **`toPlainText(html)`** (imported from `react-email`, run on the output of `render()`), *not* a `plainText` flag on `render`.
- The student's day-to-day surface for *seeing* the text version is the **preview server's plain-text tab** (named in L2). This lesson teaches what to *read it for*.
Frame the whole plain-text section around "it's generated for you; your job is to read it and fix the source," and mention `toPlainText` only as the named utility behind the preview tab / unit tests. Do not teach a `render({ plainText: true })` call site.

**Continuity anchors (reuse verbatim, do not re-derive).** Canonical artifact `emails/welcome.tsx`: `type WelcomeEmailProps = { firstName: string; verifyUrl: string }`, default export `WelcomeEmail`, `WelcomeEmail.PreviewProps = { firstName: 'Ada', verifyUrl: 'https://yourapp.com/verify/abc123' } satisfies WelcomeEmailProps`. Shared `emails/email-layout.tsx` (`EmailLayout`), `emails/email-tailwind-config.ts` (default `emailTailwindConfig`, has `pixelBasedPreset`). Placeholder brand "YourApp", CTA classes `rounded-md bg-brand px-5 py-3 text-brand-foreground`. `<Html lang="en">` was already set in L1's final artifact — this lesson *explains why it's there* and adds `<Title>` + the dark-mode `<Head>` plumbing. Files kebab-case, components PascalCase. No live coding (ReactCoding can't load `@react-email/components`; Sandpack would need the package too — use static `Code`/`AnnotatedCode`/`CodeVariants` + non-coding exercises).

**Visual grammar.** Reuse L1/L2's diagram grammar where natural (horizontal lit/dimmed node strips). Two diagrams earn their place: the `multipart/alternative` split, and the three-tier dark-mode behavior. Screenshots of real client renders are not capturable now — use `{/* TODO */}` `<Screenshot>` placeholders with specified alt+caption, same convention L2 established.

**Tone.** Adult, terse, senior. Lead every subsection with the decision/why, then the small syntax. Real production stakes throughout: a missing `lang` mispronounces the whole message; a 3:1 body contrast loses the low-vision user; a dark-mode-only contrast strategy fails the majority of clients that ignore the preference.

### The message ships as two bodies

**Goal:** install the `multipart/alternative` mental model and the "text is generated, not authored" reflex.

Content:
- Every legitimate transactional email goes on the wire as one `multipart/alternative` message carrying *two* bodies: `text/plain` and `text/html`. The receiving client renders whichever it can or chooses to. This was foreshadowed in L1 ("you get the text part free by passing `react`") — now make it concrete.
- Why the text body is non-negotiable (frame as four concrete audiences/uses, not a list to memorize): screen readers / accessibility-mode clients that don't render HTML; clients and corporate policies that strip HTML for security; spam/content scanners that read the text part (a missing/empty text part is itself a faint spam signal — ties back to Ch048 deliverability); and the fallback indexers/scanners see when Gmail clips the HTML at 102 KB (the clip budget itself is L1's territory — reference, don't re-teach).
- **The generation story (the corrected core).** The Resend SDK auto-derives the text body from the `react` node — same render that produced L1's free dual-MIME output. The application code calls *nothing*. Make the senior reflexes explicit:
  - Never hand-maintain a parallel `text` field — it drifts within a release or two. (This is the L1 "pass the node, don't pre-render" rule extended to text.)
  - You *opt out* by passing `text: ''` or your own string — name this only so the student knows the auto-text is the default behavior they're relying on, not magic.
  - The manual utility, when you do need it (a unit-test assertion, a custom transform), is `toPlainText(html)` from `react-email`, run on `render()`'s HTML output — *not* a `plainText` option on `render`. One sentence; it's behind the preview tab and tests, not on the send path.

**Diagram — the two-body split.** A `<Figure>` with a small horizontal SVG or HTML+CSS visual: one `react` node on the left → `Resend SDK` → splitting into two stacked labeled boxes `text/html` and `text/plain`, both wrapped in a `multipart/alternative` envelope → a client on the right picking one. Reuse the lit/dimmed node grammar from L1's render pipeline for consistency. Pedagogical goal: cement that *one source produces both bodies automatically* and the client chooses — kills the instinct to write the text by hand. Keep well under the height cap. Caption: e.g. "One React node, two bodies on the wire. The SDK generates both; the client renders whichever it can."

`<Term>` candidates here: `multipart/alternative` (re-tooltip from L1 — student may land here directly), MIME (re-tooltip), `toPlainText` (the utility behind the preview's plain-text tab).

### Reading the plain-text version

**Goal:** turn "the text is generated" into an active QA habit — the student reads the preview's plain-text tab and fixes incoherence *in the JSX*.

Content:
- The reflex: open the preview server's **plain-text tab** (L2 named it; this is the payoff) and read the text body as a *standalone message*. The mindset shift: a bad text version is not a text-version problem — it's a bug in your HTML JSX, fixed there.
- The coherence checklist (each item paired with where the fix lives in the JSX):
  - **Purpose + CTA unambiguous from text alone.** If the verify action is unclear without the styled button, the body copy is too thin — fix the `<Text>`/heading wording.
  - **Links are labeled, not bare.** A naked `https://yourapp.com/verify/abc123` is correct but worse than context around it. The text generator inlines the URL after the link text, so write link text that reads well beside its URL — fix the `<Link>` text.
  - **Decorative images don't pollute the text.** `alt=""` (empty, not omitted) drops a decorative image from the text body; a missing or junk `alt` leaves stray "image"/filename noise — fix the `<Img alt>`.
  - **Section order matches the HTML.** A footer that floats to the top of the text usually means a `<Container>`/nesting bug in the JSX — fix the structure, not the text.
- Hammer the anti-pattern (watch-out, inline): do **not** "clean up" the text version by setting an informational image's `alt=""` — that blinds screen-reader users to real content. The fix for noisy text is reworded HTML, never a stripped `alt`. (This watch-out belongs here, not in a tips bin — it's the most tempting wrong move in this exact section.)

**Diagram — HTML vs. generated text, side by side.** Use **`CodeVariants`** (two tabs: "HTML (what you wrote)" / "Plain text (what shipped)") over a *small* slice of `welcome.tsx` — the heading, one `<Text>`, the verify `<Link>`/`<Button>`, a decorative `<Img>`. Tab 1: the JSX. Tab 2: the rendered text body (e.g. `# Welcome, Ada!` … `Verify your email: https://yourapp.com/verify/abc123`), with the decorative image correctly absent. First sentence of each tab's prose carries the framing. Pedagogical goal: make the JSX→text mapping visual so the four checklist items become "I can see where that came from." (CodeVariants over TabbedContent per the components guidance — these are two code views of the same source.) Keep ≤18 lines per tab; elide with `{/* … */}` if needed.

**Exercise — coherence triage.** A `MultipleChoice` (single-correct): show a short generated plain-text body with one defect (e.g. stray "image" line from a decorative logo with a non-empty `alt`, or a "click here" link that's meaningless in text). Ask which *JSX* change fixes it. Distractors: editing a separate text file (wrong — no such file), setting the informational image's `alt=""` (the trap), lowering the logo's width (irrelevant). Correct: give the decorative image `alt=""` / reword the link text. `McqWhy` reinforces "fix lives in the HTML JSX." Rationale: tests the *reflex* (where the fix lives), not recall of prose. Do not phrase choices verbatim from the prose.

### The accessibility checklist for transactional mail

**Goal:** a per-template checklist, each item bound to a concrete JSX edit on `welcome.tsx`, framed as "shorter than web a11y, but with stricter numeric floors."

Content — open with the framing that lowers cognitive load: email a11y is a *tighter subset* of web a11y because the substrate (tables, inline styles, no JS) blocks most ARIA patterns the student learned earlier — so the list is short — *but* the numeric floors (font size, contrast, touch target) are stricter and unforgiving because the reader can't restyle the message. Then connect: most of these are reflexes from Ch017 (semantics, headings, landmarks) and the shadcn/a11y conventions, re-pointed at email.

The checklist (teach as ~8 items; each: the rule, the one-line *why*, and the exact JSX it maps to). Present the list as prose with inline `Code`, then reinforce with one exercise — do **not** make this a wall of bullets with no synthesis:
- **`lang` on `<Html>`.** `<Html lang="en">` (already in L1's artifact — *here's why it was there*): screen readers pick pronunciation from it; missing → system-locale fallback mispronounces the whole message.
- **`<Title>` in `<Head>`.** `<Title>Verify your email</Title>` — assistive tech announces it; some clients show it in open-in-browser view. New `<Head>` content this lesson adds.
- **One `<h1>`, logical order.** Exactly one `<Heading as="h1">` = the message's purpose; `as="h2"` for subsections. The h1-as-logo anti-pattern is *doubly* bad in email — the h1 is the screen reader's first navigation target, so it must be the message's point, not the brand mark.
- **Descriptive link text.** `<Link>Verify your email</Link>`, never "click here"/"this link." Screen readers can list links by text alone — each must stand alone. (Same point the plain-text section made; here it's the a11y rationale.)
- **Image `alt` discipline.** Decorative → `alt=""` (empty, not omitted); informational → one-sentence description; logo → the brand name. Missing `alt` often falls back to the filename in clients — an a11y *and* deliverability footgun. (Cross-link the plain-text section's same rule — one discipline, two payoffs.)
- **Color contrast — WCAG AA.** 4.5:1 body text, 3:1 headings/large text. Default `<Text>` on near-white is usually safe; the recurring failure is **brand color on the CTA** — `bg-brand` text must clear 4.5:1 against the button fill. This is where the project's `brand`/`brand-foreground` pair gets a contrast sanity check.
- **Minimum font size.** 14px body, 16px on mobile. iOS Mail auto-bumps sub-13px, but don't *rely* on the bump. (Ties to L1's `pixelBasedPreset` — sizes are predictable because of it.)
- **Touch target ≥ 44×44px.** The `<Button>` component's default padding clears it; a custom-styled `<Link>` acting as a CTA usually doesn't — another reason to reach for `<Button>` (L1's bulletproof-button point, now with the a11y rationale).
- **Don't carry meaning in color alone.** A red "urgent" button means nothing to a screen reader — the *text* must carry the meaning, color only reinforces. (Maps to the WCAG SC the student met in Ch017.)

Note the course-convention nuance for downstream agents: the project's web-app a11y floor is WCAG 2.2 AA with a 24×24 *minimum* / 44×44 *primary-action* touch target (per Code conventions). For a transactional CTA — always a primary action — 44×44 is the right number to teach; state it as such, don't surface the 24px web-list minimum here (it'd muddy the email-specific guidance).

**Walkthrough — the accessible `welcome.tsx`.** Use **`AnnotatedCode`** (`lang="tsx"`, `maxLines={18}`) over `welcome.tsx` *with the a11y attributes now visible* — `<Html lang="en">`, `<Title>` in `<Head>`, the single `<h1>`, descriptive `<Link>`/`<Button>` text, `<Img alt>`, the `bg-brand`/`text-brand-foreground` CTA. Steps (colored, ≤6 lines each) walk: lang+Title (the document-level a11y), heading order, link/CTA text + touch target, image alt, the CTA contrast pair. Pedagogical goal: the checklist is *abstract* until the student sees every item as a concrete attribute on the file they already built. This is the section's anchor — the syntax is small, so one good annotated pass over the real artifact does the teaching. (Syntax agent: derive from L1's final `welcome.tsx`, add `<Title>` and confirm line numbers against the rendered block.)

**Exercise — bind the checklist to the substrate.** A `Buckets` (two-column) titled e.g. "Where does each accessibility fix live in the template?" — items are concrete edits (`lang="en"`, `<Title>`, `as="h1"`, descriptive `<Link>` text, `alt=""` on decorative image, `bg-brand` contrast ≥ 4.5:1, 16px mobile body, 44×44 CTA), buckets are the *checklist categories* (e.g. "Document & language", "Structure & links", "Images", "Sizing & contrast"). Rationale: forces the student to organize the checklist into a reusable mental schema rather than a flat list — that schema is what they'll carry to the next template. Alternative if Buckets feels forced: a `Matching` drill pairing each rule with its failure mode (missing `lang` → mispronounced; bare "click here" → useless link list; 3:1 body → low-vision miss). Prefer Buckets for the schema-building payoff; downstream agent may pick whichever fits the final prose better.

`<Term>` candidates: WCAG (re-tooltip, define AA), `alt` (decorative vs. informational distinction is the whole point — a tooltip on "decorative" may help), touch target (44×44 / iOS HIG).

### Designing for dark mode you can't control

**Goal:** the three-tier behavior model first, then the minimal head plumbing, then the defensive discipline. End with the logo-swap as named-not-built.

Content, in strict order to manage load:

1. **The three-tier model (teach before any code).** Mailbox clients do exactly one of three things with dark mode, and you must design for all three because you don't control which a recipient has:
   - **No transformation** — HTML renders as authored; your own dark styles apply if present (Apple Mail macOS dark, recent Outlook).
   - **Partial inversion** — client inverts light backgrounds, preserves dark elements; brand colors usually survive, near-white → near-black (Gmail iOS, Outlook iOS).
   - **Full inversion** — client inverts *everything*; logos become photo negatives, brand CTAs hue-shift (Gmail Android in some configs, some Outlook installs).
   The senior posture stated plainly: **assume some clients will invert; design defensively; opt into your own dark styles for the clients that honor the preference.** And the non-negotiable: **the *light* template must already meet WCAG contrast on its own** — dark mode is never the accessibility strategy, because most clients ignore your preference anyway. (This is the bridge back to the a11y section: dark mode ≠ contrast compliance.)

2. **The head plumbing (the small syntax).** Two `<meta>` tags + one inline rule flag the template as dark-aware — *without them Apple Mail won't apply your dark styles even in dark mode*:
   - `<meta name="color-scheme" content="light dark" />`
   - `<meta name="supported-color-schemes" content="light dark" />`
   - inline `<style>` in `<Head>` with `:root { color-scheme: light dark; }`
   All three live in `<Head>` — extends the `<Head>` the a11y section just added `<Title>` to. Then the two styling approaches, with the recommendation:
   - **`@media (prefers-color-scheme: dark)`** in an inline `<style>` in `<Head>`, targeting class/`data-` selectors to swap the brand color and the logo. The most reliable approach across clients that respect the preference. **This is the senior reach for a transactional template.**
   - **Tailwind's `dark:` variant** via the `<Tailwind>` component — supported but narrower (works in Apple Mail / recent Outlook, ignored elsewhere). Fine for a simple bg/text swap; *not* a substitute for the media-query block for real inversions. (Calls back L1's "don't reach for `dark:` yet" — *now* you can, with the plumbing in place, but know its ceiling.)
   - State the minimum-viable posture for the project's transactional template: head plumbing + one `@media (prefers-color-scheme: dark)` block handling the brand-color swap and the logo. The full dark-everything pass is a marketing-template concern, explicitly out of scope.

3. **The logo-on-dark problem — named, not built.** A dark-text logo on near-white vanishes when a client inverts to near-black. The reach is a `<picture>` with a `media="(prefers-color-scheme: dark)"` `<source>` (light-on-dark logo) and the fallback `<img>` (dark-on-light). Supported by Apple Mail / most modern webmail; clients that ignore the preference still get the original (worst case "barely visible," not "missing"). Name it as the pattern; the project ships one brand-neutral logo so it's not built here. One short code snippet (static `Code`) is enough to show the shape.

**Diagram — the three-tier behavior.** A `<Figure>` (or `TabbedContent` with three tabs: "No transform" / "Partial invert" / "Full invert") showing the *same* simplified email card rendered three ways — light card unchanged; light bg→dark with brand CTA intact; everything inverted with the logo gone negative and the CTA hue-shifted. HTML+CSS mock (real CSS, devtools-inspectable) is ideal here since it's literally color treatments of one layout. Pedagogical goal: the three-tier model is the lesson's heaviest concept — seeing the *same message* under three transforms makes "design for all three" visceral and motivates the plumbing. Pick `TabbedContent` if three side-by-side comparisons read better as switchable panels; pick a single `<Figure>` with three small cards in a row if they fit under the height cap (prefer the row — all-three-at-once is the point). Caption ties each panel to a representative client.

**Diagram/visual — the `<Head>` assembled.** Optional but useful: a single `Code` block (or fold into the dark-mode `AnnotatedCode` below) showing the complete `<Head>` after this lesson — `<Title>`, the two color-scheme metas, the `:root` rule, and the `@media (prefers-color-scheme: dark)` block. Goal: the student has seen `<Head>` accrete piece by piece (Title in a11y section, metas+media here) — one consolidated view prevents fragmentation.

**Walkthrough — dark-mode plumbing on `welcome.tsx`.** An **`AnnotatedCode`** (`lang="tsx"`) over the `<Head>` + the brand/logo swap, stepping: the two metas (why they unlock Apple Mail), the `:root` color-scheme rule, the `@media (prefers-color-scheme: dark)` block (the brand-color + logo swap), and one step contrasting where `dark:` would fit and its ceiling. Keep ≤6 lines prose/step. This is the only genuinely new syntax in the lesson; one careful pass earns it.

**Exercise — dark-mode judgment.** A `TrueFalse` round (3–4 statements) targeting the load-bearing misconceptions:
  - "Adding the `dark:` Tailwind variant is enough to make the template dark-mode-aware in Apple Mail." → False (needs the head metas).
  - "If the template looks good in dark mode, its light version is allowed to fail WCAG contrast." → False (most clients ignore the preference; light must pass).
  - "Gmail Android may invert your logo even though you set no dark styles." → True (full inversion).
  - "The preview server's dark toggle proves the template is correct on Gmail Android." → False (callback to L2: toggle emulates, test-send verifies).
  `TfWhy` on each. Rationale: dark mode is where beginners hold the most wrong beliefs; True/False surfaces and corrects them efficiently. (Matches L2's use of `TrueFalse`/`Statement`/`TfWhy` for the dark-toggle trust question — consistent component choice across the chapter.)

`<Term>` candidates: `prefers-color-scheme: dark` (re-tooltip from L2), color inversion (re-tooltip from L2), `<picture>`/`<source>` `media` (one-line on the swap mechanism), `color-scheme` meta (what declaring it actually signals to the client).

## Scope

**This lesson teaches:**
- The `multipart/alternative` two-body model and that the text body is SDK-auto-generated (opt-out via `text`), with `toPlainText(html)` named as the manual/test utility.
- Reading the generated plain-text version for coherence and fixing defects in the HTML JSX.
- The email accessibility checklist (lang, Title, heading order, link text, alt discipline, contrast AA, font size, touch target, no-meaning-in-color-alone), each mapped to `welcome.tsx`.
- The three-tier dark-mode behavior model, the `color-scheme` head plumbing, `@media (prefers-color-scheme: dark)` vs. the Tailwind `dark:` variant, and the `<picture>` logo swap (named, not built).
- `dir="auto"` as the single-line i18n carry-over.

**Out of scope — already taught (reference concisely, do not re-teach):**
- Component vocabulary, `<Tailwind>`, `pixelBasedPreset`, the `emailTailwindConfig`/`EmailLayout` files, the JSX→inline-table render pipeline, the 102 KB Gmail clip budget, image hosting/CDN, `<Img>` width/height-as-attributes (L1).

- WCAG semantics, landmarks, heading basics, contrast concept (Ch017 + React/shadcn a11y conventions) — re-pointed at email, not re-derived. Logical-property utilities `ps-*`/`pe-*` (Ch020) — referenced for `dir="auto"`.

**Out of scope — reserved for later / deliberately excluded:**
