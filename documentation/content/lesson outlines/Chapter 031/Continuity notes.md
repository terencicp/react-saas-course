# Chapter 031 — Loading, errors, and the four async files

## Lesson 1 — Suspense, the fallback contract

**Taught.** `<Suspense fallback={...}>` as React's declarative loading contract ("render `fallback` while any descendant suspends; render `children` when all resolve"); the two suspending shapes the student writes (async Server Component awaiting in its body; Client Component reading a server-created promise with `use()`); fallback rules (rendered synchronously, must not suspend; skeleton mirroring resolved layout to avoid layout shift); the unit-of-UX placement rule; nested boundaries → shell/partial/full reveal cascade; `key` to force re-suspend on input change.

**Debts.** Streaming transport / why parallel boundaries reveal independently → L2. `Promise.all` vs parallel boundaries (data side) → L2. `loading.tsx` / file conventions → L3. `error.tsx`, Error Boundaries, `notFound()`/`not-found.tsx` → L3. `global-error.tsx` → L4. Promise stability / `cache()` request memoization → Ch032 L5. `useTransition`/`startTransition` (the deliberate opposite of `key`) → Ch025 L5.

**Terminology.** "suspend signal" (the throw a not-ready component emits, caught like try/catch); "fallback" (UI shown while suspended); "streamed promise" (created on server, passed as prop, read client-side with `use()`); "unit of UX" (the placement diagnostic: "what should the user see resolve as a single unit?"); "all-or-nothing per boundary" (one boundary = exactly two visual states). `key` framed as React's identity hint → changed key = fresh mount = fresh suspend.

**Patterns and best practices.** Boundary placement = reveal granularity; wrap the slow read in its own boundary so fast widgets paint first. Skeletons match the resolved footprint (no layout shift). Promise for `use()` must be created on the server and passed down — never minted inside a re-rendering Client Component. Suspense's job is narrow: it does NOT catch errors, retry, dedupe fetches, or auto-skeleton.

**Misc.** Canonical example domain is invoices (`InvoiceList`, `InvoiceSkeleton`, `listInvoices()` tagged `// data layer: Unit 5`) plus a dashboard with profile (fast ~10ms) + activity feed (slow ~800ms). `ReactCoding` exercise deliberately diverges from prod conventions: simulates suspending with a `setTimeout`-backed `use(promise)` (no real server/DB in-browser), explicitly labeled a teaching stand-in. Lesson scrupulously avoids `loading.tsx` even informally (kept fresh for L3). No significant cuts from chapter-outline scope.

## Lesson 2 — Streaming a page in chunks

**Taught.** Streaming as the wire transport: server opens one HTTP response, flushes the static shell + every Suspense fallback in the first chunk, then writes a follow-up chunk per boundary as it resolves (resolved HTML + inline swap script), connection held open until all settle. Streaming is the App Router default — no flag/export; boundaries *are* the config. Parallel fetching: each unit of UX owns its own read + boundary, reads in flight concurrently, each streams on its own resolve. The sequential-`await` trap (two serial awaits = sum, not max, and nothing streams). `Promise.all` (+ one boundary) vs parallel boundaries decided by consumption (together vs adjacently). Above-boundary work blocks the first byte (layout reads cost TTFB on every route). Network-panel diagnostic + the production buffering gotcha. Streaming is HTTP chunks, not SSE/WebSockets.

**Debts.** Partial Prerendering / static-vs-dynamic rendering decision (extends streaming with a build-time CDN shell over the same boundaries) → Ch032 (named, deferred). `connection()` + full cancellation surface / `AbortSignal` threading → Ch032 L8 (mid-stream nav aborts the request but does not auto-cancel in-flight DB/`fetch` work). `cache()` / request memoization for deduping parallel reads → Ch032 L5. Real-time push (SSE/WebSockets, Pusher/Ably) → out of scope, named once. Parallel routes (`@slot` = own subtree + boundary, list+detail streams in parallel) → Ch029 L5, leaned on by Ch035 project.

**Terminology.** "the shell" (immediately-rendered non-suspended outer UI = the before-first-byte window); "chunk / flush" (partial write to an open response); "RSC payload" (the serialized stream flushed in chunks, named in Ch030); "chunked transfer encoding" (HTTP/1.1 piece-by-piece body); "concurrent / in parallel" vs serial; "reveal granularity" (how many fallbacks the user sees — the axis that separates `Promise.all` from split boundaries). Mental model: sequential = sum, parallel = max, streaming pushes first paint to the shell flush.

**Patterns and best practices.** Before writing a second `await`, ask if it needs the first's result; independent reads never go serial. Keep above-boundary work cheap (`requireUser()` auth gate is the canonical fast above-boundary read); push every slow read below a boundary. `Promise.all` rejects on any failure (correct for aggregates) — `Promise.allSettled` when survivors should still render. Confirm streaming by reading the Network panel, not inferring from render order; a buffering proxy/CDN or compression layer silently collapses streaming in prod (`proxy_buffering off` / `X-Accel-Buffering: no` — named, not taught).

**Misc.** Same dashboard domain as L1, extended with an analytics chart (~300ms) between profile (~10ms) and activity feed (~800ms). No `ReactCoding` (in-browser runtime has no server, can't demo real HTTP streaming); primary interactives are `RequestTrace` (server-render→shell→stream phases), a Mermaid `gantt` (sequential-vs-parallel on a shared time axis), `CodeVariants`, a `StateMachineWalker` (consumed-together-vs-adjacently fork), and a `CodeReview` (two seeded defects with *different* fixes: serial awaits → `Promise.all`; one top-level boundary → split). `RequestTrace` engine constraint: every awaiting `kind="server"` node must sit under a `kind="suspense"` ancestor or be `cache="static"`, else it renders a `needs <Suspense>` error (used pedagogically in the above-boundary section). No significant cuts from chapter-outline scope.

## Lesson 3 — The three segment files

**Taught.** The three sibling files as sugar over primitives the student can already name, each motivated by one of a route's four states. The shared model first: a specially-named file in a route folder becomes a *boundary* wrapping that segment's `page.tsx` + everything nested below, inherited down until a nearer file *overrides* (CSS-specificity analogy). `loading.tsx` = framework writing `<Suspense fallback={<Loading/>}>` (Server Component, default export, no props). `not-found.tsx` paired with `notFound()`: fetch-then-guard (`if (!invoice) notFound()`), `notFound()` *throws* a `never`-typed short-circuit signal that also narrows `Invoice | null` → `Invoice`; the file is a no-props Server Component. `error.tsx` = framework writing a React Error Boundary (the gap Suspense leaves), the *only* one of the three needing `'use client'` (class-component machinery + state). Composition: framework nests error → Suspense → not-found → page, order decides behaviour.

**Cut.** `ArrowDiagram` for bubbling (outline suggested it) realized instead as a hand-authored HTML+CSS nested-box `Figure` (`ErrorCatchesWhat.astro`) plus a build/sequence component (`ThreeBoundariesCompose.astro`) for composition, both at `src/components/lessons/031/3/`. No conceptual cuts — full outline scope shipped.

**Debts.** `global-error.tsx` for root-layout throws (the explicit cliffhanger: `error.tsx` can't catch the layout it sits beside, root layout throw escapes `app/error.tsx`) → L4. `global-not-found.js` (experimental, `globalNotFound` flag, multiple root layouts) → named once, out of scope. Expected-failure channel (Server Action returns a structured result, renders inline) → Ch043 (`error.tsx` is for unexpected throws only). `useEffect` reporting to error-tracking / Sentry `captureException` → Ch092 (named as the location in `error.tsx`). Metadata for error/404 pages → Ch034 L9. Builds on: async request APIs `await params` (Ch030), "full Error never crosses the wire" / secrets server-side (Ch030), Server-default/client-opt-in reflex (Ch030), Suspense + "Suspense does not catch errors" (L1), streaming + status-locked-at-first-byte (L2), `notFound()` named (Ch029 L4).

**Terminology.** "boundary" (wrapper React places around a subtree for one cross-cutting concern); "route segment" (one `app/` folder = one URL-path piece + its UI); "inherit/override" (nearer file shadows ancestor for its subtree); "short-circuit" (`notFound()` stops execution like an early throw); "bubble up" (errors propagate to nearest boundary). `digest` = hash correlating a stripped Server-Component error to server logs (surface in UI for support tickets). Retry: **`unstable_retry()`** is the current 16.2 default (re-fetch + re-render = `router.refresh()` + `reset()` in a transition) — wire this to "Try again"; **`reset()`** is the stable narrower fallback (re-render only, useless for data errors). `unstable_` prefix flagged as genuinely unstable (may be renamed).

**Patterns and best practices.** One boundary file per coherent *visual surface*, not one per route folder (inheritance covers the rest; reflexive per-folder skeletons = maintenance pile + jarring cascade). Skeleton mirrors resolved footprint (no layout shift); animation lives in a Client leaf the file renders, file stays a Server Component. Co-locate resource-specific 404s next to the dynamic segment; generic site-wide 404 belongs in root `app/not-found.tsx` (also catches all unmatched URLs). `error.tsx` stays small/single-purpose: failure UI + digest + retry + monitoring `useEffect`; no business logic, no throwing data fetches. Verify `error.tsx`/error UX in a **production build**, never dev alone (dev overlay renders on top, hides the real generic-message + digest user experience). Default exports on `loading.tsx`/`error.tsx`/`not-found.tsx` are the sanctioned exception to the named-exports rule (framework finds them by default export).

**Misc.** Canonical route is `app/invoices/[id]/` (single-invoice detail); `getInvoice(id)` returns row-or-`null` (deliberately *not* the `requireInvoice` wrapper, which would hide the `notFound()` call — `requireInvoice` named once as the prod wrapper that bundles the pattern). Status-code nuance: `not-found.tsx` emits **200 + `<meta name="robots" content="noindex">`** for a *streamed* response, **404** only for a *non-streamed* one (status locked at first byte) — the `noindex` meta, not the raw status, is what protects SEO; need a real 404 status → finish the existence check before any suspending `await`. A `fetch` returning HTTP 404 does *not* trip `not-found.tsx` — the file is wired to the `notFound()` function, never to a status code. No `ReactCoding` (framework file-convention wiring can't be modeled in the in-browser runtime — same constraint as L1/L2); assessments are `Buckets` (state→file mapping) + `TrueFalse` (the six gotchas). `error.tsx` example uses `error.digest != null &&` (explicit null check, not truthy `&&`).

## Lesson 4 — Catching the root layout

**Taught.** `app/global-error.tsx` as the single Error Boundary the framework wires *above* the root layout — the outermost boundary, last net before the browser default. Why it's needed: re-states L3's rule (boundary catches descendants, not the parent layout it sits beside) and walks it up — root layout has no `error.tsx` above it, so its throw escapes the whole tree. Two non-negotiables: (1) `'use client'` (same client-only Error-Boundary reason as `error.tsx`, one-line callback); (2) it must render its own `<html>`/`<body>` because it *replaces* the crashed root layout (no parent shell survives) — taught causally, not as a rule. The "page that must not fail" discipline: it's the last line of defense, so no data fetching / business logic / anything that can throw; only the catastrophe UI + `error.digest` + retry + the monitoring report. Synthesis: `error.tsx` (under the shell) + `global-error.tsx` (the shell itself + escapees) = complete error surface; ship both at root.

**Debts.** `useEffect`-reports-to-monitoring SDK wiring (named as `global-error.tsx`'s canonical home, deferred) → Ch092. i18n proper (only the one-sentence caveat: `global-error.tsx` can't depend on i18n loaded above the crash) → Ch084. Builds directly on L3 (`error.tsx`, its `'use client'` requirement, `error`/`unstable_retry` props, `digest`, the boundary-can't-catch-its-parent-layout cliffhanger) and L3's "four states / four files" frame.

**Terminology.** No new terms — reuses L3's "boundary," "route segment," "bubble up." Key line to reuse: *every other `error.tsx` is a boundary inside the app shell; `global-error.tsx` is the boundary around the shell.* Senior takeaway phrasing: *assume nothing above you survived.* Retry prop is `unstable_retry` (mirrors L3; outline's `reset` was superseded).

**Patterns and best practices.** Senior default config: `app/error.tsx` + `app/global-error.tsx` both at the root, plus segment-level `error.tsx` wherever a feature deserves a tailored message. `global-error.tsx` must stay the most defensive file in the codebase: minimal inline styles or a small set of Tailwind utilities (framework Tailwind layer survives), no design-system providers, no custom-font dependency, no client-only context providers, single default-locale string. `metadata`/`generateMetadata` exports unsupported in `global-error.tsx` (Client Component) — use a `<title>` element in the JSX for a tab title. Verify `global-error.tsx` in a **production build**, never `next dev` alone. Default export is the sanctioned exception to named-exports (same allow-list as the L3 convention files).

**Misc.** Corrected a stale chapter-outline claim: outline said `global-error.tsx` is "production-only by default… renders only in production." Current behavior (since Next 15.2): it *does* render in dev too, but the dev error overlay sits on top of it — so dev still shows the developer experience, not the user's. Lesson teaches the current mechanism; practical advice (verify in a prod build) is unchanged. No `ReactCoding` (framework file-convention wiring + production-vs-dev rendering can't be modeled in the in-browser React runtime — same constraint as L1-L3); sole assessment is a 5-statement `TrueFalse` (the two non-negotiables, the dev trap, no-data-fetching, error/global-error not redundant). Diagrams reuse L3's nested-box HTML+CSS style for continuity (3-step `DiagramSequence` of the escaping throw; `CodeVariants` contrasting normal-layout-is-the-document vs global-error-is-the-document). `global-not-found.js` not resurrected (L3 named and dropped it). No conceptual cuts from outline scope.
