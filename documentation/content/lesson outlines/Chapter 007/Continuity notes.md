# Chapter 007 — Async semantics

## Lesson 1 — The event loop and the microtask queue

- **Taught** — Three-part runtime (call stack, microtask queue, macrotask queue), the four-step tick recipe (macrotask → full microtask drain → optional render → next macrotask), `await` as microtask-paced suspension with three consequences (sync code before first `await`, pre-resolved Promise still queues, `async` with no `await` still wraps Promise), `queueMicrotask` recognition-only, Node extras (`process.nextTick`, `setImmediate`) named with starvation caveat.
- **Debts** — Lesson 2 owns Promise authoring and combinators; Lesson 3 owns parallel-by-default and N+1; Lesson 4 owns cancellation and `AbortSignal.timeout` (replaces legacy `Promise.race` timeouts); Chapter 008 owns `try`/`catch` discipline and `unknown`-in-catch; Unit 19 owns yielding long microtask chains (`setTimeout(0)` / `MessageChannel`); Chapter 025 owns React's scheduler (explicitly disjoint from microtask queue).
- **Terminology** — `<Term>` tooltips defined: *event loop*, *macrotask*, *microtask*. Mental models: "drain completely vs one per iteration" asymmetry; "`await` returns the function, queues the continuation" (not "pauses on the line"); "the unit of cancellation is the work, not the Promise" (forward-anchor for L4). Canonical phrase: "`await p` does not block the thread."
- **Patterns and best practices** — Snippets use `const f = async () => { ... }` arrow form (not `async function`). `.then` is the carve-out form when teaching scheduling directly; project code defaults to `await`. App code never reaches for `process.nextTick` or `queueMicrotask` directly — they are recognize-in-library-code only; reach for `setTimeout(fn, 0)` when a macrotask is needed. No return-type annotations on illustrative inline async functions (convention requires them only on exported functions).
- **Misc.** — Custom widget at `src/components/lessons/007/1/EventLoopWalker.astro` (step-snapshot driven, no JS interpreter) and `ThreePartRuntime.astro` figure. Opening puzzle (`a/b/c/d` four-line program) is resolved in-lesson with output `a, b, d, c` (note: `b` before `d` because `.then(b)` enqueues before `queueMicrotask(d)`). Two `VideoCallout`s: Lydia Hallie (mid-lesson, ~12min) and Jake Archibald "In The Loop" (end, ~35min, dated 2018 but canonical). Node's internal phases (timers/pending/poll/check/close) named in one sentence only — course app code never depends on the phase split.

## Lesson 2 — Promises: combinators and withResolvers

- **Taught** — Three-state Promise model (pending/fulfilled/rejected) with permanence of settled state; the four combinators distinguished by "what counts as done?" (`Promise.all` = every value needed, `Promise.allSettled` = every result reported, `Promise.any` = first success with `AggregateError` on full rejection, `Promise.race` = first settlement of any kind); `Promise.withResolvers<T>()` as the standardized replacement for the deferred boilerplate; `.then`/`.catch`/`.finally` named-for-recognition with two carve-outs (post-call transform, non-async-caller cleanup); unhandled-rejection rule ("every Promise has a handler"); `Promise.resolve/reject` as framework-adapter shortcuts with microtask-scheduled continuations.
- **Cut** — `Promise.try` not even named (outline allowed one-liner); `Promise.allSettled` exhaustive narrowing deferred to chapter 008; the `Promise.race` "composing custom first-to-settle semantics" use case is named but only with a single cancellation-flag snippet.
- **Debts** — Lesson 3 owns `await` patterns, parallel-by-default, N+1, `return await`, fire-and-forget discipline (the caution callout names `void someAsyncFn().catch(...)` as the fix-shape); Lesson 4 owns `AbortSignal.timeout(ms)` which replaces the legacy `Promise.race([fetch, timeoutPromise])` pattern (referenced twice — in the `race` variant and forward-link); Chapter 008 lesson 1 owns `try`/`catch` mechanics and the throw-vs-return rule; Chapter 008 lesson 2 owns custom `AggregateError` subclasses and `instanceof DOMException` cross-realm gotcha.
- **Terminology** — `<Term>` tooltips: *Promise*, *settled*, *executor*, *AggregateError*. Mental models: "what counts as done?" as the decision rule; "combinators compose Promises that already exist; `withResolvers` is for authoring a Promise something external will settle"; "the deferred pattern, standardized" for `withResolvers`. Canonical rule sentence: "Every Promise this app creates is awaited inside a `try`/`catch`, has a `.catch` attached, or is fed to a combinator whose rejection branch you've thought about (`allSettled`, `any`)."
- **Patterns and best practices** — `Promise.withResolvers<T>()` with typed generic is the project default for externally-settled Promises (never the `let resolve; new Promise(r => { resolve = r })` deferred boilerplate). `Promise.all` destructuring shape: `const [a, b, c] = await Promise.all([...])`. `Promise.allSettled` consumers must inspect each entry's `status` before reading `value`. `Promise.any` is reserved for genuine redundancy (replicas, CDN fallback), not "kind of similar" reads. Bare `someAsyncFn();` calls are forbidden — must be awaited, `.catch`-ed, or explicitly `void`-prefixed with `.catch`. `await` is the project default; `.then` only when an `async` wrapper would change a signature, `.finally` only from non-async callers.
- **Misc.** — Lesson uses two `CodeVariants` blocks: one with `syncKey="promise-combinator"` for the four combinator tabs (same user/org/invoices skeleton), one with `syncKey="with-resolvers"` for the deferred-vs-`withResolvers` before/after with red/green `data-mark-color` tints. No custom widget. One `VideoCallout` (Lydia Hallie, ~9min, on Promise constructor internals) — lesson 1's pedagogical note said "no video callout" but the published lesson includes one mid-lesson on the three-state model. `Matching` exercise has 5 pairs (not 6 as outlined — "two analytics endpoints" pair was dropped, leaving one `Promise.any` example); deliberately omits `Promise.race` from right side. `ScriptCoding` refactor exercise asserts function body contains substring `Promise.withResolvers` to verify refactor actually happened. Example code elides `'use server'`/`import 'server-only'` headers — downstream lessons should not add them when code is about Promise mechanics rather than module boundaries.

## Lesson 3 — async/await: parallel by default, sequential by dependency

- **Taught** — Dependency-check reflex ("if two awaits don't share data, they run in parallel") with `Promise.all` destructured-tuple rewrite as the canonical shape; N+1 trap inside `.map(async ...)` with three fixes by trigger (bounded `Promise.all` for N≤10, `pMap` with `concurrency` for large/user-controlled N, batched DB query with Drizzle `inArray` for N reads against same backend); `for await...of` for streams (`response.body` iteration) and paginated SDK iterators (Stripe-shaped) with the "not for parallel fan-out" watch-out; `return await` mandatory inside `try`/`catch` and best-practice outside for stack-trace fidelity; `async` function always returns `Promise<T>` (never annotate unwrapped `T`); fire-and-forget shape `void asyncFn().catch(...)` pairing `void` (lint) with `.catch` (crash prevention).
- **Cut** — `AbortSignal` parameter shape elided from `for await...of` streaming example beyond a one-sentence acknowledgement (Lesson 4 owns full treatment); historical `no-return-await` ESLint rule mentioned for context but not framed as a rule the student maintains; top-level `await` named in one line only (Chapter 006 Lesson 3 owns the decision); Node streams (`Readable`/`Writable`/pipelines) and async generators authoring fully out.
- **Debts** — Chapter 008 owns `try`/`catch` mechanics at depth (`err: unknown` narrowing, throw-vs-return, `Result<T, E>`); Lesson 4 owns full `{ signal }` parameter shape and `AbortError` discrimination; Chapter 039 Lesson 2 owns deep N+1 / Drizzle batched-query treatment (forward-linked); Chapter 066 owns durable background work as the real answer to "fire-and-forget needs durability"; Chapter 025 Lesson 7 + Chapter 031 own React `use(promise)`.
- **Terminology** — New `<Term>` tooltips: *unbounded parallelism*, *backpressure*, *N+1*. Canonical sentences: "If two `await`s don't share data, they run in parallel"; "Inside a `try`/`catch`, `return await` is mandatory"; "Always pair `void` and `.catch`" for fire-and-forget. Mental models: dependency-check reflex (read consecutive awaits, ask if next needs prior's value); "what is N?" check for any `.map(async ...)`; six-shape classification (`Promise.all` / `pMap` / `for await...of` / batched query / sequential await / fire-and-forget). Latency framing: "sum-vs-max."
- **Patterns and best practices** — `await Promise.all([...])` destructured-tuple is the project rewrite target for independent reads; `pMap` from `p-map` is the project default for bounded fan-out (concurrency cap matches downstream service tolerance, not local machine throughput); Drizzle `inArray(table.col, ids)` is the canonical batched-read shape; `return await` written consistently (project default — `no-return-await` is gone); `void asyncFn().catch((err) => log.error(...))` is the only acceptable fire-and-forget shape (both operators always paired); `async` return types declared as `Promise<T>` (never unwrapped `T`); bare `.map(async ...)` forbidden in code review unless N is statically bounded by UI.

## Lesson 4 — Cancellation with AbortController and AbortSignal

- **Taught** — Producer/consumer split (`AbortController` creates token, `AbortSignal` is read-only consumer view); the `{ signal }: { signal?: AbortSignal }` parameter shape repeated across `fetch`, Node `node:timers/promises` `setTimeout`, `addEventListener`, and authored helpers; canonical user-cancel pattern (abort previous, fresh controller per call, thread `signal` through I/O); three-way catch discrimination by `err.name` (`'AbortError'` user-cancel silent return, `'TimeoutError'` surface to user, anything else rethrow); `AbortSignal.timeout(ms)` as the 2026 replacement for `Promise.race` timeouts (rejects with `TimeoutError` not `AbortError` — deliberate UX asymmetry); `AbortSignal.any([userSignal, timeoutSignal, shutdownSignal])` composition as the canonical SaaS shape; `AbortSignal.abort(reason?)` static factory for pre-aborted signal; the guaranteed/not-guaranteed contract (stops further work, does not undo completed work).
- **Cut** — Browser callback-`setTimeout`-not-abortable trap reduced to one sentence inside the `setTimeout` tab; `crypto.subtle` `signal` support not named; `signal.reason` covered in one sentence only (no custom abort-reason authoring); `Aside` callouts published as Starlight `:::tip` and `:::caution` rather than `Aside` component.
- **Debts** — Chapter 008 L1 owns `unknown`-in-catch narrowing and `ensureError`; Chapter 008 L2 owns cross-realm `instanceof DOMException` gotcha and custom abort-reason authoring; Chapter 025 L2 owns React effect-cleanup with ref-held controllers; Chapter 039 L4 owns Drizzle/`pg` connection-level cancellation and transactional rollback as the "undo" tool; Chapter 043 owns Server Action signal threading; Chapter 046 owns route handler cancellation; Chapter 066 L5 owns Trigger.dev `ctx.run.abortSignal`; Chapter 106 owns AI SDK streaming cancellation (uses the `AbortSignal.any` composition shape).
- **Terminology** — New `<Term>` tooltips: *AbortController*, *AbortSignal*, *AbortError*, *TimeoutError*. Canonical sentences: "If an async function does I/O, its signature includes `signal`"; "The unit of cancellation is the work, not the Promise" (load-bearing site, originally seeded by L1); "Controllers are not reusable — one controller per logical operation." Mental model: two-party split with one wire — producer fans out to many consumers through a single signal value. Discriminate by `err.name === 'AbortError'` / `'TimeoutError'` (not `instanceof DOMException` — that's browser-only and breaks in Node).
- **Patterns and best practices** — Every async function that does I/O accepts `{ signal }: { signal?: AbortSignal }` in an options object — even when no current caller cancels, the parameter is part of the signature so future callers don't refactor. Thread `signal` to every I/O call inside the function (`fetch`, DB, timers). One controller per logical operation; abort the previous before creating a fresh one. Catch blocks around abortable I/O have three branches in order: `AbortError` → return early (silent), `TimeoutError` → notify user and return, else → rethrow. Use `AbortSignal.timeout(ms)` instead of `Promise.race([fetch, timer])`. Compose user-cancel + deadline + shutdown via `AbortSignal.any([...])` per-request (don't stash composed signals in module scope; if you must, attach listeners with `{ once: true }` to avoid Node listener-accumulation leak). Server-side helpers use `node:timers/promises` `setTimeout` for abortable sleeps; browser code uses `AbortSignal.timeout(ms)` with `fetch`.
- **Misc.** — Custom widget at `src/components/lessons/007/4/ProducerConsumerSplit.astro` (one diagram showing controller→signal→three consumers). One `CodeVariants` with `syncKey="signal-shape"` (four tabs: `fetch`, `setTimeout`, `addEventListener`, authored helper — repeated shape is the lesson). One `CodeVariants` with `syncKey="discrim"` (red/green wrong/right discrimination tabs). One `Matching` exercise (5 pairs, no decoys). One `VideoCallout` mid-lesson (Software Developer Diaries, ~18min). Four `ExternalResource` cards in wrap-up. No React wiring shown — names Chapter 025 L2 forward-link once.
