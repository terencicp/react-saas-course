# Chapter 2.2 prerequisites review

## Missing prerequisites

- **Lesson 2.2.1** — `unknown` type and `x is T` / `asserts x is T` predicate syntax. Quote: "function isUser(x: unknown): x is User { ... } and function assertUser(x: unknown): asserts x is User { ... }. Both forms require the function keyword today". These TypeScript-specific types and return-type forms are presented as the reason a `function` declaration is chosen over an arrow, requiring the student to read and understand them — but `unknown` is first taught in 2.4.1 and type predicates / assertion signatures are first taught in 2.4.6 and 2.5.3. Suggested source: 2.4.1 (for `unknown`) and 2.4.6 / 2.5.3 (for the predicate syntax), or a brief forward note in 2.2.1 that acknowledges the student cannot yet fully parse these signatures. Severity: medium.

- **Lesson 2.2.2** — Inline TypeScript object type literals with `?` optional-field modifier used as the canonical options-object signature form. Quote: "(options: { name: string; email: string; admin?: boolean })" and "The TypeScript ordering rule. Required parameters before optional. function fetch(url: string, options?: Options) is legal". The `?` optional modifier in object type literals and parameter position is the subject of 2.4.2 (field modifiers on `type` / `interface`), which has not been taught yet. These are presented not as incidental syntax but as the canonical form every Server Action and React prop will use. Suggested source: 2.4.2, or add a one-paragraph "reading TypeScript type annotations" orientation at the chapter level. Severity: medium.

- **Lesson 2.2.4** — `never` type required to understand `assertNever`. Quote: "function assertNever(x: never): never { throw new Error('unhandled: ' + JSON.stringify(x)); } placed in default: makes a missing case a compile error". The lesson's core exhaustiveness-pattern claim — that `assertNever` turns a missing variant into a compile error — only makes sense if the student understands that `never` is the bottom type and why an argument of type `never` is unreachable. `never` is taught in 2.4.1. Without it the student is asked to take the compile-error claim on faith. Suggested source: 2.4.1. Severity: medium.

- **Lesson 2.2.6** — Signature-level destructuring with full inline TypeScript type annotations presented as the canonical form. Quote: "function createUser({ name, email, admin = false }: { name: string; email: string; admin?: boolean }). The payoff: the function's body reads the same names as the call site uses, the defaults live at the signature, and the type annotation reads adjacent. This is the shape Server Actions consume". This combines parameter destructuring with a full inline object type literal including `?` modifiers — a form that requires both 2.4.2 (field modifiers) to understand and that is being installed as the senior default the student should now write. Suggesting the student write this form before the type-annotation syntax is formally taught creates a gap. Suggested source: 2.4.2, or confine the type annotation to a `type` alias defined above the snippet so the student is not reading inline object-type syntax cold. Severity: medium.
