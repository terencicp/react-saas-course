# Chapter 087 â€” Time, dates, and timezones

## Chapter framing

Chapter 087 lands the SaaS pattern that bites every product the moment it crosses a second timezone: **store time canonically, render it locally, and never let the runtime's `process.env.TZ` enter business logic.** Pieces are in place from earlier units â€” lesson 3 of chapter 013 planted Temporal (Stage 4, unflagged in Node 26); Chapter 041 set up the Drizzle column types; Chapter 9 wired the user profile; Chapter 070 introduced Trigger.dev `schedules`; lesson 5 of chapter 034 named tz as a hydration-mismatch cause. What's missing is the explicit, named architecture that ties them: the **storage / domain / edge** split. Postgres `timestamptz` is the storage shape for instants; `date` is the storage shape for calendar days; Temporal is the domain type the application reads and writes; the user's IANA timezone (a profile column) is the formatting input at the edge; UTC is the only timezone the server clock or recurring-job schedule references. Five teaching lessons plus the quiz; wiring lands in `db/schema.ts`, `lib/temporal.ts`, the profile settings surface, and one or two Trigger.dev schedule files. Locale-aware *formatting* (`Intl.*`, `next-intl`) is Chapter 088's territory; this chapter owns the substrate.

Threads that run through every lesson. The **three-layer split** is the load-bearing architecture and every lesson lands one layer or one boundary between them. **The user's timezone is a profile column, not a per-request derivation** â€” `Intl.DateTimeFormat().resolvedOptions().timeZone` at request time on Vercel returns the runtime's `TZ` (UTC), not the user's, and silently formats every user's data in UTC. **DST is a property of the calendar, not the timestamp** â€” `Temporal.Instant` is unambiguous, `Temporal.ZonedDateTime` is where DST math lives, and a recurring job like "9 AM weekday in America/New_York" must use a named timezone so the platform handles spring-forward / fall-back. **The Temporal pivot is the senior 2026 default** â€” every arithmetic, comparison, or formatting site in this chapter uses Temporal; `temporal-polyfill` (FullCalendar's ~20KB shim) is the carve-out for projects pinned to Node 24 LTS. Two senior rules carry through: never call `getMonth()` / `setDate()` / `Date.now()` for business logic, and never compare or format a timestamp without naming the timezone.

## Lesson 1 â€” Storage, domain, edge

Teaches the three-layer split for time: `timestamptz` (UTC) in Postgres, `Temporal.Instant` in the domain via a single codec in `lib/temporal.ts`, ISO 8601 on the wire, with `Date` confined to third-party seams.

Topics to cover:

- **The senior question.** A user clicks "create invoice" at 11:47 PM Pacific on March 8, 2026 (US DST start). The Vercel function and the Postgres on Neon both run UTC. Six months later the team adds a customer-facing audit log "in the user's timezone." Three architectures answer storage differently â€” UTC-only `timestamptz`, local-as-string, `timestamp without time zone` with implicit UTC â€” and only the first keeps rendering separable from storage. The lesson installs the canonical 2026 form: every instant-shaped column is `timestamptz` with UTC semantics; every read lands as `Temporal.Instant` (or `ZonedDateTime` when a tz attaches downstream); every render to a human picks up the user's profile timezone at the edge. Storage / domain / edge â€” the three-layer split this chapter operates on.
- **What `timestamptz` actually is.** Despite the name, it stores no timezone â€” eight bytes, microseconds since 2000-01-01 UTC. On `INSERT`, Postgres reads the session's `TimeZone`, interprets the input string as that zone, converts to UTC, stores. On `SELECT`, the same setting drives string rendering. Senior reach: set session `TimeZone` to `UTC` (Neon / Vercel defaults; explicit `SET TIME ZONE 'UTC'` in `db.ts` belt-and-suspenders); write every input as ISO 8601 with `Z`. A session set to `America/Los_Angeles` reading the same row as a UTC session produces different `text` output â€” name the canonical bug.
- **`timestamptz` vs. `timestamp` (without time zone).** The Postgres wiki's "Don't Do This" list names `timestamp` directly. `timestamp` stores no timezone interpretation at all â€” the literal string, untranslated. Two services writing "now" from different machines produce different stored values for the same wall-clock moment. The senior rule: `timestamptz` is the default; `timestamp` shows up only in rare data-pipeline cases where the source explicitly disclaims a timezone (a CSV from a "local clock, no tz" system).
- **The Drizzle column.** From chapter 041: `createdAt: timestamp('created_at', { withTimezone: true, precision: 3 }).notNull().defaultNow()`. `withTimezone: true` is the `timestamptz` flag; `precision: 3` matches JavaScript's millisecond resolution (Postgres stores microseconds; the tail drops silently on read â€” fine for SaaS audit timestamps); `defaultNow()` calls Postgres's `CURRENT_TIMESTAMP` server-side. Never reach for `new Date()` or `Temporal.Now.instant()` in TS to seed a column the DB seeds itself â€” the server clock is the authority.
- **`mode: 'string'` vs. `mode: 'date'`.** Drizzle's `timestamp` ships two read modes. `mode: 'date'` (default) returns a `Date` instance; `mode: 'string'` returns the raw ISO 8601 string. The senior 2026 reach on this stack is `mode: 'string'` â€” the application never sees `Date` directly, the codec converts the string to `Temporal.Instant`, the round-trip is explicit. The default invites the `Date` gotchas lesson 3 of chapter 013 cataloged back into the codebase.
- **The codec â€” one helper, one place to convert.** `lib/temporal.ts` owns the `instant` codec â€” a typed pair `{ fromDb(value: string): Temporal.Instant; toDb(value: Temporal.Instant): string }`. Application code reads and writes Temporal; the codec is the seam. Lives once; every `instantColumn` in the schema imports it. When the project upgrades from polyfill to native Node 26 Temporal, the codec's import line changes; every consumer keeps working.
- **What application code sees.** A row's `createdAt` is `Temporal.Instant`. The student writes `invoice.createdAt.epochMilliseconds`, `.toString()`, `.since(other)` â€” never `.getMonth()`, never `.toISOString()` on a `Date`. The lesson reads three or four canonical call-sites (list query mapping rows to a domain type, Server Action receiving an instant in input, route handler serializing the column to the wire) with Temporal as the in-memory type and ISO 8601 as the wire/db shape.
- **Wire-format symmetry.** ISO 8601 strings with `Z` are the universal carrier â€” Postgres stores instants from them, `Temporal.Instant.from(string)` parses them, `Instant.prototype.toString()` and `toJSON()` produce them, every API the course consumes (Stripe, Trigger.dev payloads, Resend) emits and accepts them. ISO 8601 strings cross *every* boundary; Temporal types live only in-memory between boundaries.
- **The microsecond-precision asymmetry, named once.** Postgres `timestamptz` stores microseconds; JS `Date` / `Temporal.Instant` carry milliseconds (or Temporal's native nanoseconds). The codec preserves up to ms; the Âµs tail drops on read. Comparing rows for strict equality across a write-read cycle fails on the Âµs tail â€” use `epochMilliseconds` or `equals()`; `precision: 3` on the column matches storage to application resolution.
- **"Never trust the client clock."** From chapter 070's idempotency context: `createdAt`, `updatedAt`, `processedAt`, `expiresAt`, every server-meaningful instant is set by the *server* â€” Postgres's `defaultNow()` or `Temporal.Now.instant()` in `db.ts`'s server-side helper. A Server Action receiving `createdAt` from the client trusts the client â€” drop it and re-stamp.
- **The hydration-mismatch trap, restated.** From lesson 5 of chapter 034: rendering a date string in a Server Component that depends on the runtime's local tz produces different output on server (UTC) and client (browser local). Fix: pass the user's *profile* tz explicitly into `Intl.DateTimeFormat(locale, { timeZone })` (the formatter from chapter 088). The implicit-runtime-local default is the hydration-mismatch class.
- **The chapter's two Drizzle â†” Temporal pairs.** This lesson installs `timestamptz` â†” `Temporal.Instant`. lesson 2 of chapter 087 adds `date` â†” `Temporal.PlainDate`. Two codecs, two column types, two domain types. `time without time zone`, `interval`, `timestamptz` arrays are named once but don't show up in the SaaS surface the course teaches.
- **What stays in `Date` (the narrow carve-out).** Two seams. Third-party SDKs that return a `Date` (Stripe SDK) â€” converted to `Temporal.Instant.fromEpochMilliseconds(date.getTime())` *immediately* at the seam, never propagated. The rare "stopwatch-style" duration measurement where the absolute value doesn't matter â€” `performance.now()` is the better reach anyway. The lesson 3 of chapter 013 rule holds: `Date` at the seam, Temporal in the domain.
- **Watch-outs.** A `timestamp without time zone` migration "just for created_at" â€” catch in PR review; reading a `timestamp` column expecting UTC produces wrong-by-an-offset results from non-UTC writers; non-UTC Postgres session `TimeZone` makes `SELECT created_at::text` produce different strings per session; `new Date()` in application code as `defaultNow()`-equivalent skews against the DB clock; `.toISOString()` on a `Date` drops nanoseconds where Temporal preserves them â€” a tell that the type discipline isn't held; Stripe's `created` is Unix seconds, not milliseconds â€” multiply by 1000 before `Instant.fromEpochMilliseconds`; serializing a `ZonedDateTime` to the wire and expecting the receiver to recover the tz: `toString()` includes `[America/New_York]`, but `Instant.from()` ignores it â€” pick the right type for the wire shape.

What this lesson does not cover:

- Date-only columns and `Temporal.PlainDate` â€” lesson 2 of chapter 087.
- The user's timezone column â€” lesson 3 of chapter 087.
- Recurring jobs and DST â€” lesson 4 of chapter 087.
- Temporal arithmetic at depth â€” lesson 5 of chapter 087.
- `Intl.DateTimeFormat` and locale-aware rendering â€” lesson 3 of chapter 088.
- `next-intl`'s timezone configuration â€” lesson 5 of chapter 088.

Estimated student time: 45 to 55 minutes. The chapter's load-bearing rule; every other lesson references the split installed here.

## Lesson 2 â€” Calendar days, not midnight instants

Teaches the second storage pair â€” Postgres `date` columns and `Temporal.PlainDate` with its codec â€” and why calendar-day semantics like `dueDate` and `birthDate` must never be modeled as midnight-UTC `timestamptz`.

Topics to cover:

- **The senior question.** An invoice has a `dueDate` â€” the day the customer agrees to pay by. The naive shape is `timestamptz` set to midnight UTC. A Sydney customer's "May 15" invoice reads as 2026-05-14 in Sydney because midnight UTC on May 15 is 10 AM the day prior there. The bug class is the *date-as-instant* mistake: a calendar date and a point in time are different domains. The rule: **calendar-day values go in a `date` column with `Temporal.PlainDate`; instant values go in `timestamptz` with `Temporal.Instant`.** Not interchangeable; the runtime won't catch a swap.
- **What a `date` column is.** Postgres `date` â€” four bytes, year/month/day, no time, no timezone. `2026-05-15` is the same in every session, on every machine. Every column whose semantic is "the day, irrespective of where the user is" â€” `dueDate`, `birthDate`, `subscriptionStartDate`, `holidayDate`, `effectiveDate` â€” is `date`. The grammar test: if "May 15" is the answer regardless of location, it's `date`; if the answer is "this exact second," it's `timestamptz`.
- **The Drizzle column and the second codec.** `dueDate: date('due_date', { mode: 'string' }).notNull()` returns ISO 8601 date strings (`'2026-05-15'`). The companion codec in `lib/temporal.ts` is the `plainDate` pair: `{ fromDb: Temporal.PlainDate.from, toDb: (d) => d.toString() }`. One line each direction â€” `PlainDate` round-trips through ISO 8601 by design.
- **Why `Temporal.PlainDate`, not `Date` or a string.** Immutable, knows nothing about timezones (can't accidentally drift), exposes `with({ year, month, day })` for component-level writes, `add({ months: 1 })` for arithmetic, `compare(other)` for sorting. A plain string fails on arithmetic. A `Date` rebinds to midnight UTC and brings the timezone gotcha the lesson exists to prevent.
- **The wire shape.** ISO 8601 calendar date â€” `'2026-05-15'`. Request bodies use the string; Zod parses with `z.iso.date()` and transforms to `PlainDate` (Unit chapter 046's surface). Responses serialize `PlainDate.toJSON()` to the same string. The wire never sees a midnight-UTC timestamp posing as a date.
- **The canonical anti-pattern.** A `dueDate` declared as `timestamptz`, the application writing midnight-UTC. The bug surfaces three places: `WHERE dueDate = '2026-05-15'` misses rows on Âµs drift; the Sydney customer sees the invoice "due yesterday"; reports grouped by date show the wrong day for half the user base. The fix is structural â€” the column type rejects the time, the Temporal type rejects the timezone.
- **Calendar arithmetic stays on `PlainDate`.** "Due in 30 days" = `dueDate.add({ days: 30 })`. "Due in 1 month" = `dueDate.add({ months: 1 })` clamps correctly at month-end (Jan 31 + 1 month = Feb 28; Temporal's documented behavior). "First of next month" = `dueDate.with({ day: 1 }).add({ months: 1 })`. Each returns a new `PlainDate`; the type rejects timezone or time-of-day input.
- **Crossing to `ZonedDateTime` when time of day matters.** "End of business on the due date in the user's tz" â€” explicit conversion: `dueDate.toZonedDateTime({ timeZone: user.timeZone, plainTime: '17:00' })` returns a `ZonedDateTime`; from there, `.toInstant()` produces the `Instant` for storage. The student writes `timeZone` and `plainTime`; nothing is implicit. lesson 4 of chapter 087 uses this exact pattern.
- **Hydration-mismatch for date-only.** A Server Component rendering `dueDate.toString()` is safe â€” ISO 8601 matches across server/client. `dueDate.toLocaleString()` or `Intl.DateTimeFormat` without `timeZone` is not â€” server UTC, client local. `Intl.DateTimeFormat(locale, { dateStyle: 'long' })` on a `PlainDate` is generally safe (no tz needed); weekday-quarter formats can differ. Name the boundary once.
- **No time component at the boundary.** Inputs arrive as ISO 8601 calendar dates. `z.iso.date()` validates the shape â€” no `T`, no `Z`, no offset. A value that arrived as `'2026-05-15T00:00:00Z'` is a smell; the schema rejects it and the upstream code path is the bug.
- **Postgres-side composability.** `date + INTERVAL '1 day'` works in SQL â€” useful for "yesterday at midnight" style queries. Lexicographic ISO order matches calendar order â€” `WHERE due_date < CURRENT_DATE` does what it says. Stay in `date` semantics; cast to `timestamptz` only when joining against an instant column (and that join is usually itself a code smell).
- **Borderline cases worked.** `subscription.startedAt` is `timestamptz` (the moment the trial began, the tier-change timeline). `birthDate` is `date` (May 5, 1990 everywhere). `event.scheduledFor` is `timestamptz` (a specific moment). "Send this report at end of day on the 15th of every month" is neither â€” a recurring rule, lesson 4 of chapter 087 owns it.
- **Watch-outs.** `timestamptz` "to keep options open" produces immediate bugs; the midnight-UTC hack drifts under any non-UTC consumer; mixing `date` and `timestamptz` in a `WHERE` clause needs an explicit cast or silently coerces; `Temporal.PlainDate.from(new Date())` inherits the runtime's local tz interpretation â€” use `Temporal.Now.plainDateISO(user.timeZone)` for user-relative "today" or `'UTC'` for server-time; accepting a `Date` from a third-party calendar API and storing year/month/day into a `date` column without converting through the source tz produces off-by-one at the day boundary â€” `Instant.fromEpochMilliseconds(date.getTime()).toZonedDateTimeISO(sourceTimeZone).toPlainDate()` is the explicit form.

What this lesson does not cover:

- The `timestamptz` column and `Temporal.Instant` â€” lesson 1 of chapter 087.
- The user's timezone column â€” lesson 3 of chapter 087.
- Recurring jobs and DST â€” lesson 4 of chapter 087.
- Temporal arithmetic surface â€” lesson 5 of chapter 087.
- `Intl.DateTimeFormat` for locale-aware rendering â€” lesson 3 of chapter 088.
- Zod's `z.iso.date()` at depth â€” Chapter 046.
- `time without time zone` and "wall clock time" columns â€” out of scope.

Estimated student time: 35 to 45 minutes. The storage / domain split is established; this is the second pair.

## Lesson 3 â€” Timezone on the profile

Teaches storing the user's IANA timezone as a `users.timeZone` column seeded from the browser at sign-up, validated with `Intl.supportedValuesOf('timeZone')`, read from the session, and passed explicitly to every formatter and scheduler â€” never derived per-request.

Topics to cover:

- **The senior question.** A monthly billing email needs to land at 9 AM in the customer's timezone. A "due in 3 days" reminder needs to count days from *their* perspective. A Server-Component-rendered activity list needs each row's timestamp formatted as the user reads it. Three architectures answer "whose timezone?". **(1)** Derive from `Intl.DateTimeFormat().resolvedOptions().timeZone` at request time â€” picks up the runtime's `TZ` (UTC on Vercel), not the user's, and silently produces UTC strings everywhere. **(2)** Derive from `Accept-Language` or geo-IP â€” `Accept-Language` doesn't carry tz; geo-IP is a privacy and accuracy lottery. **(3)** Store the user's IANA tz on the profile, read from the session, pass explicitly to every formatter and every recurring-job schedule. Only (3) scales past the second timezone.
- **The `users.timeZone` column.** `text` on `users` (or `profiles`), storing an IANA identifier (`'America/New_York'`, `'Europe/Berlin'`, `'Asia/Tokyo'`, `'UTC'`). `NOT NULL` with a `'UTC'` default at backfill. *Not* an offset (`-05:00`) â€” offsets don't carry DST rules; the same offset string means different real times across the year. IANA identifier, always.
- **The companion `users.locale` column.** Named once because chapter 088's discipline pairs them: locale (`'en-US'`, `'en-GB'`, `'fr-FR'`) is the formatting input for `Intl.NumberFormat`, weekday names, ordinals; timezone is for date/time. Independent â€” a Berlin user can read in `'en-GB'` with `'Europe/Berlin'`. Both on the profile. This chapter owns timezone; lesson 4 of chapter 088 owns locale.
- **Seeding on sign-up.** The browser knows the user's tz at sign-up (`Intl.DateTimeFormat().resolvedOptions().timeZone` on the client). Three senior shapes. **(a)** Client-side detection at the sign-up form, captured in a hidden input or Server Action payload â€” senior default. **(b)** Post-sign-up onboarding step where the user picks from a list â€” higher friction, explicit. **(c)** Fallback to `'UTC'` plus a prompt on first sign-in â€” lowest-friction migration for existing accounts. The chain: client-detected tz â†’ `'UTC'` if detection fails â†’ fixed up at profile edit. The client value is user-asserted, not authoritative; the user changes it on the profile page if needed.
- **Why not derive at request time, restated.** `Intl.DateTimeFormat().resolvedOptions().timeZone` in a Server Component on Vercel returns `'UTC'` â€” every time, for every user, because Vercel sets `TZ=UTC`. Every formatter renders in UTC, every reminder fires at midnight UTC, every audit log says "0:14 UTC" instead of "8:14 PM your time." The test suite passes locally (developer's machine isn't UTC) and the bug ships unnoticed. *Never* call the no-argument `Intl.DateTimeFormat()` in server code.
- **Reading the user's tz in code.** Two seams. **(a)** The session. `requireOrgUser()` from lesson 2 of chapter 061 returns `{ user, orgId, role }`; the `user` includes `timeZone`. **(b)** A `getCurrentUserTimeZone()` helper in `lib/user-time.ts` â€” for deeply-nested components without prop-drilling. Both pass the tz explicitly into every Temporal or `Intl` call; the global / ambient form is not allowed.
- **The org-level fallback.** Some products (B2B billing, scheduling) need an org tz â€” invoices issued from a San Francisco company are dated by the company's clock, not the recipient's. Mirror column `organizations.timeZone`; a small helper resolves "the right tz for *this* operation." Billing uses `org.timeZone`; user-facing rendering uses `user.timeZone`. Most code lands on user.
- **The profile settings UI.** A `/settings/profile` page (Chapter 9, extended in chapter 089) renders a timezone select. The list is `Intl.supportedValuesOf('timeZone')` â€” a standard API (Node 22+, every modern browser) returning the platform's known IANA names; display them with current UTC offset (`'America/New_York (UTC-04:00)'`). Submit goes through `authedAction` with a Zod schema that validates against the supported-values set â€” `z.string().refine((tz) => Intl.supportedValuesOf('timeZone').includes(tz))`. An invalid string in the column produces `RangeError: Invalid time zone specified` on every read; validation at the write edge prevents the runtime explosion.
- **"The user's clock is the user's truth."** When a user says "remind me at 9 AM tomorrow," they mean *their* timezone. Every scheduling input, every "due by end of day," every "remind me in 24 hours" interprets with `user.timeZone` as the calendar/clock context. Canonical computations: "end of day for the user" = `Temporal.Now.zonedDateTimeISO(user.timeZone).with({ hour: 23, minute: 59, second: 59 }).toInstant()`; "9 AM tomorrow for the user" = `Temporal.Now.zonedDateTimeISO(user.timeZone).add({ days: 1 }).with({ hour: 9, minute: 0, second: 0 }).toInstant()`. Both produce `Instant` for storage; the user's tz lives in the conversion.
- **Detecting tz drift.** A user signing in from a new geographic timezone is common. On sign-in, the client sends current `Intl.DateTimeFormat().resolvedOptions().timeZone`; if it differs from the profile column, surface a non-blocking banner ("It looks like you're in Tokyo. Update your timezone?"). Never *auto-update* â€” silent rebase of every scheduled email for a traveler is the bug.
- **The "tz is data, not config" rule.** The user's timezone is a *user attribute* â€” not an env var, not a deployment flag, not a per-region build setting. The canonical bug is treating the deployment region as a proxy for tz ("the EU deployment serves Europe, use Europe/Brussels by default"). Wrong: a Spanish user on the US deployment exists; data follows the user, not the infrastructure.
- **Watch-outs.** Storing a UTC offset string instead of an IANA name silently breaks across DST; the structural defense for the no-argument `Intl.DateTimeFormat()` bug is a `formatDate(value, { timeZone })` wrapper that *requires* the tz argument; UTC-fallback for *display* is fine but UTC-fallback for *scheduling* fires reminders at the wrong time â€” surface an "unset" state in the UI; `process.env.TZ` in Vercel project settings (or a Docker base image) silently changes the meaning of every `new Date()` and `Intl.DateTimeFormat()` â€” keep `TZ=UTC`; deriving tz from `Accept-Language` is a category error (`en-US` doesn't mean New York); the IANA database updates several times a year â€” pin the platform's `tzdata` via OS / Node release cadence, never paper over with hard-coded offsets; treating tz as PII for GDPR is correct (a user in `Pacific/Chatham` is a small population) â€” let lesson 4 of chapter 085's redactor handle it.

What this lesson does not cover:

- The `timestamptz` column and `Temporal.Instant` codec â€” lesson 1 of chapter 087.
- Date-only columns and `Temporal.PlainDate` â€” lesson 2 of chapter 087.
- Recurring jobs that use the user's tz for the schedule â€” lesson 4 of chapter 087.
- Temporal arithmetic surface â€” lesson 5 of chapter 087.
- `users.locale` and locale negotiation â€” lesson 4 of chapter 088.
- `Intl.DateTimeFormat` and locale-aware rendering â€” lesson 3 of chapter 088.
- The full profile settings UI shipped in the project â€” Chapter 089.
- Better Auth's session surface â€” Unit 9.

Estimated student time: 40 to 50 minutes.

## Lesson 4 â€” DST and recurring jobs

Teaches how to schedule recurring work across DST transitions using Trigger.dev `schedules.task` with a named IANA `timezone` for wall-clock jobs and UTC for internal cadence, plus `Temporal.ZonedDateTime` disambiguation and per-tenant dynamic schedules.

Topics to cover:

- **The senior question.** A daily report runs at "9 AM Eastern" for a B2B customer. Twice a year, the underlying scheduler decides what "9 AM Eastern" means when the wall clock skips (spring forward, March 8, 2026) or repeats (fall back, November 1, 2026) an hour. Three architectures: **(a)** schedule in UTC, period â€” `0 13 * * *` is 9 AM EST in winter, 8 AM EDT in summer, the wall-clock time shifts by an hour twice a year; **(b)** schedule with a wall-clock expression and a named IANA tz â€” the scheduler computes the next real instant using DST rules, the report runs at 9 AM Eastern year-round; **(c)** hybrid â€” UTC schedule with per-fire computation in application code, more work, same answer as (b). (b) is the default for user-facing wall-clock-meaningful jobs; (a) is the default for internal cadence jobs (retention from lesson 4 of chapter 085, cleanups, rollups). The wrong answer is letting the *server's* `TZ` drive interpretation â€” produces "EST in dev because the developer is in Eastern, UTC in prod because Vercel is UTC" and silent twice-a-year drift.
- **What DST does to time math.** Spring forward (March 8, 2026): wall clock jumps from 1:59 AM directly to 3:00 AM â€” the 2 AM hour does not exist. Fall back (November 1, 2026): wall clock falls from 1:59 AM back to 1:00 AM â€” the 1 AM hour exists twice. Naive wall-clock interpretation skips jobs scheduled in the gap and double-fires jobs in the repeat. Practical rule: never schedule recurring jobs in the 1 AMâ€“3 AM window of a DST-observing tz; let the scheduler's tz-aware engine pick which instant a wall-clock expression resolves to. 9 AM, noon, 5 PM are safe.
- **Temporal's DST surface.** `Temporal.ZonedDateTime` is the type that knows about DST. Two key options: `disambiguation` (`'compatible'` default, `'earlier'`, `'later'`, `'reject'`) picks which of two instants on fall-back, or what to do for a spring-forward gap; `offset` (`'use'`, `'ignore'`, `'prefer'`, `'reject'`) controls whether a stored offset overrides tz rules. Senior 2026 defaults: `disambiguation: 'compatible'` (use the later instant on fall-back, jump forward on spring-forward gap â€” what most users expect), `offset: 'use'` only when round-tripping a `ZonedDateTime` from a serialized form with both tz and offset.
- **The "schedule in wall clock, store the instant" recipe.** A user-facing recurring job is two pieces: the *rule* (wall-clock expression + IANA tz) and the *next fire instant* (a `Temporal.Instant` derived from the rule). Scheduler stores the rule; on every fire, computes the next instant from rule + current `users.timeZone`, enqueues. The fire-time computation is the single place DST math happens; everything downstream operates on `Instant`. Rule is durable; instant is recomputed every fire; tz changes on the profile propagate to the next fire automatically.
- **Trigger.dev `schedules` and the `timezone` option.** From lesson 4 of chapter 070: `schedules.task` accepts a `cron` pattern and a `timezone` argument naming an IANA zone. The platform computes the next fire using the named zone's DST rules. Canonical shape for "9 AM weekday Eastern": `schedules.task({ id: 'weekly-summary', cron: { pattern: '0 9 * * 1-5', timezone: 'America/New_York' }, run: async (payload) => { ... } })`. The `payload` carries `timestamp` (the fire instant) and `lastTimestamp` as ISO strings; the run handler converts to `Temporal.Instant` and proceeds.
- **Per-tenant dynamic schedules.** "Send me my report at 9 AM in my timezone" is per-user â€” N users, N schedules, each with that user's tz. Trigger.dev's `schedules.create` API registers at runtime; the app calls it on user opt-in and `schedules.delete` on opt-out. The `externalId` is the user id for idempotent re-registration. When the user changes `users.timeZone`, the schedule needs a `schedules.update` call gated on the diff. The propagation chain: profile-tz change â†’ recompute the schedule's tz.
- **The "schedule in UTC for internal jobs" carve-out.** The retention job (lesson 4 of chapter 085), the daily backfill, hourly metric rollups â€” these don't care about wall-clock time, only cadence. UTC is right. `0 3 * * *` with `timezone: 'UTC'`. The binary: user-facing wall-clock-meaningful â†’ user/org tz; internal cadence â†’ UTC.
- **The "compute fire instant at fire time" rule.** A schedule created in October that fires "Mondays at 9 AM Eastern" cannot pre-compute the entire year's instants ahead â€” the November 1 fall-back means November's computation needs the *current* tz database. Compute the *next* fire only, at fire time, using the current tz data. A hand-rolled `pg_cron` implementation that pre-computes a year of instants in a table is the bug class â€” when the IANA database updates mid-year, pre-computed instants drift.
- **The user's tz changes â€” what propagates.** Three classes of dependent data. **(a)** Future recurring schedules â€” re-register with the new tz; `schedules.update(externalId, { cron: { ..., timezone: newTz } })`. **(b)** Already-stored future instants â€” a one-shot job scheduled for "5 PM next Friday in the user's tz" was converted to `Instant` at scheduling time; the change doesn't retroactively shift it. The default is "past intent is honored" â€” a job scheduled when the user was in New York fires at the New-York-derived instant even after they move. The bug class is the silent retroactive shift; the cure is the explicit "update these N scheduled deliveries?" prompt. **(c)** Already-stored past instants â€” these don't change; history.
- **The IANA tz database lifecycle, named once.** `tzdata` is updated several times a year by IANA â€” countries change DST rules, new names land. Node 26 ships an embedded copy; Vercel keeps current. Don't ship a hand-rolled tz library or pin an old `tzdata`; rely on the platform's bundled data and the runtime's update cadence.
- **Idempotency on fire under DST.** From lesson 5 of chapter 070: every scheduled job carries an idempotency key. Under fall-back, a poorly-designed scheduler could fire the same wall-clock expression twice; the idempotency key (typically `${scheduleId}:${fireInstant.toString()}`) makes the second fire a no-op. Use the *instant*, not the wall-clock string, as the key â€” two `Instant` values that happen to share a wall-clock representation under fall-back are *different* instants and produce different keys; the cure is structural.
- **Testing under DST.** A Vitest fixture sets `TZ=America/New_York` and runs at the boundary of a DST transition. Construct `ZonedDateTime` values straddling each transition and assert: a "9 AM weekday" job fires at 14:00 UTC in winter and 13:00 UTC in summer, no fires in the 2-3 AM gap, no double-fires in the 1-2 AM repeat. Full test surface lands in Unit 19; DST is testable and the pattern exists.
- **Watch-outs.** A `cron` expression with no `timezone` argument silently runs in the scheduler's default (typically UTC) â€” always pass tz explicitly, even for UTC schedules (makes intent explicit); "midnight" in a DST-observing tz crosses the date boundary differently on transition days (the previous day has 23 hours, the next 25) â€” `ZonedDateTime.from(...).with({ hour: 0, minute: 0, second: 0 }).toInstant()` is the explicit form; a scheduler that re-computes the schedule's tz from the user profile on *every fire* (not just on profile change) absorbs updates without an explicit re-register â€” a valid alternative; storing `users.timeZone` and not piping it into the recurring-job registration is the canonical "profile tz exists but schedules still fire in UTC" bug; not handling the IANA-name-not-recognized case crashes the job â€” Zod-validate at the profile-write edge; assuming `Temporal.Now.zonedDateTimeISO('America/New_York')` is stable across a request is fine; assuming the *offset* it implies is stable across a request that spans midnight on a DST-transition day is wrong â€” compute offsets fresh.

- The `timestamptz` column and the `Instant` codec â€” lesson 1 of chapter 087.
- Date-only columns â€” lesson 2 of chapter 087.
- The user's timezone column on the profile â€” lesson 3 of chapter 087.

## Lesson 5 â€” Arithmetic with Temporal

Teaches the daily Temporal surface â€” the `Instant` / `ZonedDateTime` / `PlainDate` / `Duration` type catalog, `add` / `subtract` / `since` / `until` / `with` / `round`, month-end clamping, the conversion graph, the six anti-patterns to retire, and the polyfill seam for Node 24 LTS.

Topics to cover:

- **The senior question.** "Trial ends 30 days after sign-up." "Next billing is 1 month from today." "Invoices issued in the last 7 days." "How long ago was the comment posted?" Every SaaS has dozens. The 2010s reach was `date-fns` or `dayjs` over `Date`, with hand-rolled month-end clamping and the implicit local-tz gotcha. The 2026 senior reach is **Temporal, natively, no library wrapper** â€” the platform owns the surface, the types refuse the bug classes by construction, the migration off legacy code is mechanical. The lesson lands the full canonical arithmetic surface for SaaS patterns, the polyfill story for projects on Node 24 LTS, and the chapter's closing rule: never hand-roll month math, never call `Date.now()` for arithmetic, never reach for a date library on this stack.
- **The Temporal type catalog, restated.** Five types do the SaaS work. **`Instant`** â€” a fixed point in real time (UTC, the wire, the `timestamptz` column). **`ZonedDateTime`** â€” an `Instant` plus an IANA tz; knows DST; the only one safe for "9 AM in New York" math. **`PlainDate`** â€” calendar date, no time, no tz; the `date` column's domain type. **`PlainDateTime`** â€” wall-clock date and time, no tz; rarely needed in 2026 SaaS (named once; typical use case "the calendar event recurs at 9 AM in whatever tz the viewer is in"). **`Duration`** â€” explicit, immutable; year/month/week/day/hour/minute/second/nanosecond components; the input to every arithmetic operation.
- **`Temporal.Now` â€” getting "now" without ambient tz.** `Temporal.Now.instant()` â€” current `Instant` (UTC, no tz). `Temporal.Now.zonedDateTimeISO(timeZone)` â€” current `ZonedDateTime` in a named tz; the `timeZone` argument is *required*. `Temporal.Now.plainDateISO(timeZone)` â€” today's date in the named tz; argument is *required*. *Never* call `Temporal.Now.X()` without an explicit `timeZone` argument where one is accepted â€” same bug class as the `Intl.DateTimeFormat()` no-argument trap. Calling `Temporal.Now.plainDateISO()` with no argument defaults to the runtime's tz â€” a footgun the spec inherited from `Date`. Install the "always pass the tz" reflex.
- **`add` and `subtract`.** Every Temporal type with arithmetic exposes `add(durationLike)` and `subtract(durationLike)`. The `durationLike` is a named-component object: `{ days: 30 }`, `{ months: 1 }`, `{ weeks: 2, days: 3 }`, `{ hours: 24 }`. Senior shape â€” never `add(30)` (Temporal rejects), the component name is the unit. Three canonical: trial = `Temporal.Now.plainDateISO(user.timeZone).add({ days: 30 })`; next billing = `subscription.startDate.add({ months: 1 })`; window = `Temporal.Now.instant().subtract({ days: 7 })`. Immutable; each returns a new instance.
- **Month-end clamping.** "1 month from January 31" = February 28 (or 29 in leap years). `Temporal.PlainDate.from('2026-01-31').add({ months: 1 })` returns `2026-02-28` â€” spec-documented "clamp to last day of target month." Watch-out: `add({ months: 1 })` twice â‰  `add({ months: 2 })` near month-end (iterative clamps twice; single-call clamps once; Jan 31 â†’ Feb 28 â†’ Mar 28 vs. Jan 31 â†’ Mar 31). Prefer single-call larger durations when month-end semantics matter; reach for `overflow: 'reject'` when month-end *shouldn't* clamp ("Jan 31 + 1 month" should error rather than silently produce Feb 28).
- **`since` and `until`.** `a.since(b)` returns a `Duration` (positive if `a` after `b`); `until(b)` opposite sign. Three reads: "since sign-up" = `Temporal.Now.instant().since(user.createdAt)`; "days until due" = `Temporal.Now.plainDateISO(user.timeZone).until(invoice.dueDate)`; "subscription period duration" = `period.endDate.since(period.startDate)`. Pass `largestUnit` for human-readable durations (`since(other, { largestUnit: 'days' })`); months/years are ambiguous for an `Instant` (no calendar attached) â€” Temporal refuses without explicit calendar context. "days" / "seconds" for wire / DB; "months/years" only on `PlainDate` or `ZonedDateTime`.
- **`compare`, `equals`, `before`, `after`.** Static `Temporal.PlainDate.compare(a, b)` returns `-1 | 0 | 1` for `Array.sort`. Instance methods `equals(other)`, `before(other)`, `after(other)` cover the booleans. Sort stability and the `Object.is` semantics from chapter 082 carry through.
- **`round` â€” bucketing.** `instant.round({ smallestUnit: 'hour', roundingMode: 'floor' })` rounds down. Three uses: event-bucketing for analytics (`event.timestamp.round({ smallestUnit: 'hour' })` produces an X-axis grid); user-entered times snapped to 15 minutes (`input.round({ smallestUnit: 'minute', roundingIncrement: 15, roundingMode: 'halfExpand' })`); start of a calendar period â€” reach for `with({ day: 1 })` (component edit), not `round`.
- **`with` â€” component-level edits.** `date.with({ day: 1 })` returns a new `PlainDate` with day=1, others unchanged. `zonedDateTime.with({ hour: 9, minute: 0, second: 0 })` sets time of day. The senior reach for "first of the month," "9 AM exactly." Small, orthogonal API.
- **The conversion graph.** Six common conversions, all explicit. `Instant â†” ZonedDateTime`: `instant.toZonedDateTimeISO(timeZone)` / `zonedDateTime.toInstant()`. `ZonedDateTime â†” PlainDate`: `zonedDateTime.toPlainDate()` / `plainDate.toZonedDateTime({ timeZone, plainTime })`. `PlainDate â†” string`: `PlainDate.from(string)` / `plainDate.toString()`. `Instant â†” string`: `Instant.from(string)` / `instant.toString()`. No `+` coercion, no truthy-fallback. The conversion is where reasoning happens â€” "I have an Instant from the DB and I need to know what day it was for the user" is `instant.toZonedDateTimeISO(user.timeZone).toPlainDate()`; three calls, three reasoning steps.
- **`Duration` arithmetic.** Two durations add: `Duration.from({ days: 7 }).add({ hours: 12 })`. A duration applies to a date: `plainDate.add(duration)`. For a per-plan "trial length," store as `interval`-typed column or as an ISO 8601 duration string (`'P30D'`, `'P1M'`), parse with `Duration.from`, apply with `add`. Wire-format symmetry from lesson 1 of chapter 087 holds: ISO 8601 strings everywhere, Temporal in-memory.
- **What the senior never writes.** **(a)** `new Date(year, month, day)` with month zero-indexed; **(b)** `date.setMonth(date.getMonth() + 1)` (mutation + zero-indexing); **(c)** `Date.now() + 30 * 24 * 60 * 60 * 1000` for "30 days from now" (works for short durations, fails on DST boundaries â€” naive milliseconds, not calendar days); **(d)** `new Date(isoString)` for a calendar date (rebinds to midnight UTC, brings the tz gotcha back); **(e)** any call to `date-fns`, `dayjs`, `moment`, `luxon` â€” libraries correct but no longer paid for; Temporal is the stack default; **(f)** comparing dates via `toISOString()` string comparison â€” works for `Instant`, breaks for `PlainDate` if formats diverge; use `compare` / `equals`. Six anti-patterns, six Temporal-correct rewrites.
- **The polyfill, restated.** Node 24 LTS (most production SaaS through October 2026 when Node 26 promotes to LTS) installs `temporal-polyfill` (FullCalendar's ~20KB; modern reach) or `@js-temporal/polyfill` (TC39 champions; larger, full-spec). The import lives in `lib/temporal.ts` once: `export const Temporal = globalThis.Temporal ?? require('temporal-polyfill').Temporal;` (ESM equivalent). Every file imports `Temporal` from `lib/temporal.ts`, never from the polyfill directly. Upgrade to Node 26 is a one-line change at the seam.
- **The browser line.** From lesson 3 of chapter 013: Chrome 144, Firefox 139, Edge 144 ship Temporal natively (May 2026); Safari pending. Course projects do Temporal work server-side and ship ISO 8601 strings to the client where they render through `Intl.DateTimeFormat` (chapter 088). Polyfill ships to the client only when client-side date *arithmetic* is needed on Safari â€” the chapter's projects don't.
- **Watch-outs.** `add({ days: 30 })` on a `ZonedDateTime` across a DST boundary is not exactly 30Ă—24 hours later in real time (one day was 23 or 25 hours) â€” `add({ hours: 720 })` is the "exact 30 days of real time" form when that's what's intended; `since(other, { largestUnit: 'months' })` on two `Instant` values throws (months are calendar-dependent) â€” convert to `ZonedDateTime` or `PlainDate` first; comparing `PlainDate` to `ZonedDateTime` is a type error; mixing polyfill Temporal and native Temporal in the same runtime is undefined behavior â€” the `lib/temporal.ts` seam is the *only* import path; `Duration` arithmetic with mixed units (`{ months: 1, days: 5 }`) is well-defined only with calendar context â€” operating on an `Instant` ignores months; the polyfill bundle even at ~20KB is non-trivial on mobile â€” gate the import behind a runtime check; calendar systems beyond ISO (Buddhist, Islamic, Hebrew) work with Temporal but the course doesn't reach for them.

- The `timestamptz` storage column and `Instant` codec â€” lesson 1 of chapter 087.

## Lesson 6 â€” Quizz

Top 10 topics to quiz:

- The storage / domain / edge split â€” `timestamptz` (UTC) for instants and `date` for calendar days in Postgres; `Temporal.Instant` and `Temporal.PlainDate` in the domain; ISO 8601 strings on the wire; user's IANA timezone as formatting input at the edge.
- Drizzle column declarations and codecs â€” `timestamp(name, { withTimezone: true, mode: 'string', precision: 3 })` paired with the `Temporal.Instant` codec in `lib/temporal.ts`; `date(name, { mode: 'string' })` paired with the `Temporal.PlainDate` codec; one place to convert.
- The `timestamp` vs. `timestamptz` rule â€” never `timestamp without time zone`; the Postgres wiki's "Don't Do This" names it; `timestamptz` stores a UTC instant despite the misleading name.
- The "midnight UTC `timestamptz` is not a date" anti-pattern â€” calendar-day semantics belong in `date` and `PlainDate`; midnight-UTC `timestamptz` drifts under any non-UTC consumer; storage-type discipline prevents the bug class.
- The user's IANA timezone on the profile, not derived per-request â€” `users.timeZone` column with `Intl.supportedValuesOf('timeZone')` validation; `Intl.DateTimeFormat().resolvedOptions().timeZone` at request time on Vercel returns `'UTC'`; seeded on sign-up from the browser, edited from the profile UI, propagated to recurring schedules on change.
- The org-level vs. user-level tz scope â€” billing uses org tz; user-facing rendering uses user tz; independent, resolved by a small helper.
- DST and recurring jobs â€” named IANA tz for user-facing wall-clock jobs (Trigger.dev `schedules.task({ cron: { pattern, timezone } })`), UTC for internal cadence jobs (retention, cleanups); avoid the 1 AMâ€“3 AM band; recompute the fire instant at fire time, never pre-compute.
- The Temporal type-picking decision â€” `Instant` for points in real time, `ZonedDateTime` for "this clock time in this tz" (the DST-aware type), `PlainDate` for calendar days, `Duration` for periods; explicit conversion graph (`toZonedDateTimeISO(tz)`, `toInstant()`, `toPlainDate()`).
- The arithmetic surface â€” `add({ days: 30 })` / `subtract({ months: 1 })` over `Temporal.Now.zonedDateTimeISO(user.timeZone)`; `since` / `until` with `largestUnit`; `with({ day: 1 })` for component edits; `round` for bucketing; month-end clamping by default with `overflow: 'reject'` as the opt-out; the six things the senior never writes (`new Date(y,m,d)`, `setMonth`, `Date.now() + ms`, midnight-UTC for dates, date libraries, ISO-string comparison).
- The Node 26 / Node 24 LTS path â€” Temporal native and unflagged on Node 26 (May 2026, the course's deploy target); `temporal-polyfill` (FullCalendar, ~20KB) re-exported from `lib/temporal.ts` on Node 24 LTS; one-line migration at the seam; the SSR-first architecture sidesteps Safari's pending support.
