# Chapter 088 вЂ” Internationalization

## Chapter framing

Chapter 088 closes Unit 18 with the formatting and translation half of the time-and-i18n discipline. The student arrives with chapter 087's substrate: every instant lives in `timestamptz` and reads as `Temporal.Instant`; every calendar day lives in `date` and reads as `Temporal.PlainDate`; the user's IANA timezone is a profile column passed explicitly to every formatter. What's missing is the discipline that surrounds the human-readable edge: translation strings as keys with interpolation (never concatenation), ICU MessageFormat for the plural / select rules every language needs, the `Intl.*` family as the native engine that powers every locale-aware render, an explicit locale-negotiation order, `next-intl` as the 2026 Next.js wiring, and the i18n-SEO surface that lets Google route the right locale to the right user. The chain installs **i18n as a discipline before it's a feature** вЂ” even a single-locale launch ships through the same shape so the second locale is one PR, not a refactor.

Threads through every lesson. **Translation keys are the wire format; strings are the rendered output** вЂ” application code holds `t('invoice.pastDue.title')`, the catalog holds the localized text, string concatenation in component code is the bug class the chapter exists to prevent. **The `Intl.*` family is the native engine** вЂ” every formatter in the codebase is `Intl.NumberFormat`, `Intl.DateTimeFormat`, `Intl.RelativeTimeFormat`, `Intl.PluralRules`, `Intl.Collator` (with `Intl.ListFormat`, `Intl.DisplayNames` named once); `next-intl`'s formatter hooks wrap the same engine, they don't replace it. **Locale is a profile column paired with `users.timeZone`** вЂ” seeded from the browser at sign-up, negotiated against `Accept-Language` as fallback, validated against a supported-locales constant, passed explicitly to every formatter and to `next-intl`'s middleware. **`hreflang` and per-locale URLs are the SEO contract** вЂ” search engines route locales by URL shape and `alternates.languages` metadata; the canonical URL is the localized URL, not the default. Six teaching lessons plus the quiz; wiring lives in `lib/i18n/`, `proxy.ts`, `messages/{locale}.json`, `app/[locale]/layout.tsx`, and the per-page `generateMetadata`.

## Lesson 1 вЂ” Keys, catalogs, and the no-concatenation rule

Translation keys in code, per-locale JSON catalogs with named placeholders, one-string-per-key, and the `t.rich` pattern for embedded markup.

Topics to cover:

- **The senior question.** A component renders `<p>Welcome, {user.name}! You have {count} unread messages.</p>`. Adding French, the naive reach is `'Bienvenue, ' + user.name + '! Vous avez ' + count + ' messages non lus.'` вЂ” placeholders frozen in source-language order, plural agreement hand-rolled, no place for the translator to land. The lesson installs the discipline: every user-visible string is a **key** the application holds and a **translation entry** in a per-locale catalog the translator owns, with **named interpolations** the catalog can reorder, and **never** runtime string concatenation across translatable fragments.
- **What the key looks like.** Flat namespaced dot-paths: `'invoice.pastDue.title'`, `'inbox.unread.count'`, `'auth.signIn.cta'`. The namespace groups by feature (matches the component tree loosely); the leaf is the role of the string. Two or three levels deep вЂ” five becomes unsearchable. Keys are stable; the catalog value is mutable. Renaming a key is a coordinated change (component + every catalog); the senior reach is `eslint-plugin-i18n-json` flagging orphans and missing keys at lint time.
- **Why named placeholders, not positional.** `'You have {count} unread messages.'` with `{count}` reads as a named slot; the French value `'Vous avez {count} messages non lus.'` keeps the placeholder name and can move it; the German might invert subject and verb. Positional (`'You have %s unread %s'`) is the C-stdlib mistake вЂ” the translator can't move the slots, can't rename them, can't see what they mean. ICU MessageFormat (lesson 2 of chapter 088) is the syntax; named placeholders are the rule.
- **The catalog format.** One JSON file per locale: `messages/en.json`, `messages/fr.json`. Nested objects mirror the dot-paths (`{ "invoice": { "pastDue": { "title": "Invoice past due" } } }`). JSON because every TMS (Lokalise, Crowdin, Tolgee, Phrase) round-trips it cleanly. YAML is fine but the next-intl ecosystem standardizes on JSON. Catalogs sit in the repo, ship in the build, are version-controlled with the code that uses them.
- **One string per key, no `common.*` re-use.** The temptation is `'common.save'` reused across forty buttons; the bug is German's `'Speichern'` reading differently in a settings page (save preferences) vs. a checkout (place order). Different surfaces, different keys, even when the source-language string is the same. The catalog can carry duplicate values; the translator can decide per-key per-locale. The senior reach: never share a key across unrelated surfaces.
- **Source-locale completeness rule.** Every key has a value in the source locale (`en-US` on this stack); missing values are build errors. Other locales can have gaps вЂ” the runtime falls back to the source. Missing-key logs (Sentry, dev console) surface gaps to the translation pipeline. The senior framing: ship a key in source-locale-complete state; let translators ship the other locales asynchronously through the TMS.
- **The `t.rich` / `<Trans>` pattern, previewed.** When a string carries embedded JSX (`<Link>`, `<strong>`, an icon), `t.rich('terms.line', { link: (chunks) => <Link href="/terms">{chunks}</Link> })` with the message `'By signing up, you agree to our <link>Terms</link>.'` returns a `ReactNode`. Depth in lesson 5 of chapter 088; preview here because the discipline exists. The wrong reach вЂ” split into three keys (`prefix`, `linkText`, `suffix`) and concatenate JSX вЂ” produces awkward translations and inflexible word order.
- **Pluralization and gendered forms вЂ” preview.** lesson 2 of chapter 088 covers ICU plural / select. The discipline framing here: never branch on `count === 1` in component code; every count-shaped string is a single key whose catalog value carries the ICU `plural` syntax. Same for gender / role variants вЂ” the catalog is the surface, not the component.
- **What's *not* a translation key.** Non-user-visible strings вЂ” debug logs, ARIA tags for purely structural elements, machine-readable values, IDs. The audit grep is for string literals inside JSX `<>...</>`, prop values that hit `aria-label` / `title` / `placeholder` / `alt`, toast / notification text, validation messages. Everything else stays as-is.
- **Server vs. client.** Server Components reach for `getTranslations` (async); Client Components reach for `useTranslations` (sync). Both call the same engine; both speak the same key. The chapter's React boundary is the same as chapter 034 вЂ” translations cross it transparently. Catalog data flows server-only by default; the active locale's slice ships to the client when client components consume it.
- **Watch-outs.** Concatenating translated fragments reorders impossibly in many locales; using English copy as the key (`t('Welcome back!')`) breaks renaming; storing English values *as* keys is the i18next default next-intl moves away from; a single 50K-line catalog slows build and breaks merge вЂ” split by namespace; reusing `'common.save'` is the duplicate-key bug class; hard-coding locale in a key (`'fr.welcome'`) defeats the catalog; `messages.welcome.replace('{name}', user.name)` bypasses `t()` and the formatter; checking in automated translations without translator review ships "AUTO: ..." to users.

What this lesson does not cover:

- ICU MessageFormat syntax (plural / select / selectordinal) вЂ” lesson 2 of chapter 088.
- `Intl.*` formatters вЂ” lesson 3 of chapter 088.
- Locale negotiation вЂ” lesson 4 of chapter 088.
- `next-intl`'s `useTranslations` / `getTranslations` / `t.rich` at depth вЂ” lesson 5 of chapter 088.
- The TMS pipeline (Lokalise / Crowdin / Tolgee) вЂ” out of scope; named once.
- RTL layout / `dir` handling вЂ” out of scope; named once for future scaling.

Estimated student time: 40 to 50 minutes. The chapter's discipline-installation lesson; every later lesson references the key / catalog / render split established here.

## Lesson 2 вЂ” ICU MessageFormat: plurals, select, gendered forms

ICU `plural` with CLDR categories and exact-match overrides, `selectordinal` for ordinals, `select` for free enums, and nested plural-inside-select for gendered counts.

Topics to cover:

- **The senior question.** A notification says "You have 1 message" or "You have 5 messages." English has two plural forms; Russian has four; Arabic six; Welsh six; Chinese one. Hand-rolling per-language plural logic in component code is the bug class. The senior reach is **ICU MessageFormat** вЂ” the catalog entry carries the plural rules, the runtime applies CLDR's plural categories per locale, the engineer writes one `t(key, { count })` call.
- **What ICU MessageFormat is.** A Unicode-defined string syntax (in CLDR; ICU is the reference impl). The catalog carries logic: `'{count, plural, =0 {No messages} one {# message} other {# messages}}'`. The runtime parses, looks up the plural category for `count` in the current locale, picks the branch, substitutes `#` with the formatted number. Five core features: **plural** (cardinal), **selectordinal** (ordinal вЂ” "1st", "2nd"), **select** (free enum on a string вЂ” gender, role, type), **number formatting** (`{amount, number, ::currency/USD}`), **date formatting** (`{date, date, ::yyyyMMdd}`). Chapter ships the first three at depth; number / date hand off to `Intl.*` formatters directly (lesson 3 of chapter 088) вЂ” the senior reach is the explicit formatter call at the seam.
- **The `plural` form, anatomized.** `{count, plural, =0 {No messages} one {# message} other {# messages}}`. Three pieces. **(1)** Variable name (`count`) вЂ” matches the value passed to `t()`. **(2)** Selector keyword (`plural` cardinal; `selectordinal` ordinal). **(3)** Branch table вЂ” exact matches (`=0`, `=1`, `=2`) tried first, then CLDR keywords (`zero`, `one`, `two`, `few`, `many`, `other`). `other` is mandatory вЂ” the fallback. `#` inside a branch is the count formatted as a number (locale-aware grouping, decimals); to override format use `{#, number, ::compact-short}` for `'chapter 002K'`.
- **CLDR plural categories per language.** Unicode CLDR declares plural categories per language. English: `one`, `other`. French: `one` (0,1), `many`, `other`. Russian: `one`, `few`, `many`, `other`. Arabic: all six. Welsh: all six. The translator picks the branches their language needs; the catalog shape is per-locale; the runtime delegates to CLDR via `Intl.PluralRules`.
- **`Intl.PluralRules` as the engine.** `new Intl.PluralRules('ru').select(5)` returns `'many'`. Every ICU MessageFormat implementation calls into this (or a polyfill). The student rarely calls it directly; ICU handles it. Custom plural logic in component code (`count === 1 ? ... : ...`) ignores CLDR; the senior reach delegates to the standard.
- **Exact-match overrides вЂ” `=0`, `=1`, `=2`.** Some products want "No messages" instead of "0 messages." Exact matches override category lookup; convention is to list them first, then categories, then `other`. The `=0` / `=1` carve-outs are senior product calls вЂ” declared in the source-locale catalog; translators decide whether their language needs the same overrides.
- **`selectordinal` for ordinals.** `{rank, selectordinal, one {#st place} two {#nd place} few {#rd place} other {#th place}}`. English's "1st / 2nd / 3rd / 4th" is the canonical use; other languages have different rules (German "1.", "2."; Japanese з•Є). `selectordinal` consults CLDR's *ordinal* plural rules (distinct from cardinal). Reach for it for ranks, positions, leaderboards.
- **`select` for free enums.** `{gender, select, male {He liked your post} female {She liked your post} other {They liked your post}}`. Selector is a string; branches are literal matches with `other` as fallback. Use cases: gender (when the data carries it), notification type, organization role. English source often collapses variants (gender-neutral writing); German renders three forms (nouns and pronouns inflect).
- **Combined plural + select вЂ” gendered counts.** Two-level nesting: `{gender, select, male {{count, plural, one {He has # message} other {He has # messages}}} female {{count, plural, ...}}} other {...}}`. Senior framing: nest the outer selector by the higher-cardinality dimension (gender / type), the inner by plural. Translator surface is the whole string; engineer call is one `t('notification.likedYourPosts', { gender, count })`.
- **MessageFormat 2 вЂ” named, not the default.** Unicode is finalizing MF2 with explicit `.input` / `.local` / `.match` declarations and named-formatter integration. As of May 2026 most i18n libs (next-intl included) still target MF1. Migration is mechanical when the ecosystem adopts MF2.
- **The "translator never writes JS" rule.** A translator opens the JSON, sees the ICU string, edits inside the branches. They never touch JS, never see component code. The ICU syntax is the boundary; everything inside branches is text. A message that requires understanding a ternary or callback failed the rule.
- **Watch-outs.** Missing the `other` branch raises a runtime parse error; using `=1` instead of `one` works in English but breaks in German вЂ” prefer CLDR categories, exact-match for narrative overrides only; rendering `{count}` instead of `#` inside a plural produces "5 5 messages"; using `select` for gender by parsing `user.name` is the bug вЂ” the data must carry it; ICU's quote-escaping trips up JSON editors вЂ” validate the catalog with the runtime; some libraries advertise "ICU support" but skip `selectordinal` or `select` вЂ” confirm with the actual runtime; storing messages in the DB means content editors can break ICU syntax вЂ” validate at write time.

What this lesson does not cover:

- Translation-key discipline вЂ” lesson 1 of chapter 088.
- `Intl.PluralRules` standalone for non-message use вЂ” lesson 3 of chapter 088 names it.
- `Intl.NumberFormat` / `DateTimeFormat` standalone вЂ” lesson 3 of chapter 088.
- Locale negotiation вЂ” lesson 4 of chapter 088.
- `next-intl` `useTranslations` / `t.rich` / inline formatters вЂ” lesson 5 of chapter 088.
- MessageFormat 2.0 at depth вЂ” out of scope; named once.
- CLDR's full plural rule table per language вЂ” link out, not covered.

Estimated student time: 40 to 50 minutes. The chapter's load-bearing rule for any string with a count or gendered variant.

## Lesson 3 вЂ” The Intl.* formatter family

The daily-reach `Intl.*` formatters вЂ” `NumberFormat`, `DateTimeFormat` (Temporal-native, mandatory `timeZone`), `RelativeTimeFormat`, `Collator`, plus `ListFormat` / `DisplayNames` вЂ” and the construct-once-reuse rule.

Topics to cover:

- **The senior question.** A balance is `1234.56`. English: `'$1,234.56'`. German: `'1.234,56 в‚¬'`. French: `'1 234,56 в‚¬'` (non-breaking thin space). "3 days ago" is `'il y a 3 jours'` in French, `'vor 3 Tagen'` in German. Sorting "Г¤" вЂ” with "a" (German) or after "z" (Swedish)? Every one is a runtime decision, locale-dependent. The senior 2026 reach is the **native `Intl.*` family**, not a library, not a `toLocaleString()` shortcut without arguments. The lesson surveys the five formatters every SaaS reaches for and names the supporting two (`ListFormat`, `DisplayNames`) once.
- **The shape every formatter shares.** `new Intl.X(locales, options)` returns a formatter instance. `.format(value)` returns the string; `.formatToParts(value)` returns tokens (for styling parts); some have `.formatRange(a, b)` for ranges. **Construct once, reuse** вЂ” construction is expensive; memoize at module scope or behind a `getFormatter(locale, options)` cache. A per-render `new Intl.NumberFormat(...)` in a Server Component is wasted CPU.
- **`Intl.NumberFormat` вЂ” the daily reach.** **(1) Currency.** `new Intl.NumberFormat(locale, { style: 'currency', currency: 'USD' })`; the currency code is *data*, not a UI concern. `currencyDisplay`: `'symbol'` (default), `'narrowSymbol'` (`'$'` instead of `'US$'`), `'code'`, `'name'`; senior default is `'narrowSymbol'` for compact UI. **(2) Percent.** `style: 'percent'`; input is the fraction (`0.15` в†’ `'15%'`). **(3) Decimal.** `style: 'decimal'` with `min/maxFractionDigits` for fixed precision. **(4) Compact.** `notation: 'compact'` produces `'chapter 002K'`, `'chapter 017M'`; pair with `compactDisplay: 'short' | 'long'`; dashboards, KPI tiles. **(5) Units.** `style: 'unit'` with `unit: 'kilometer'` / `'megabyte'` / `'hour'`. Senior call: every currency / percent / large-number render goes through `Intl.NumberFormat`; `value.toFixed(2)` is the bug class.
- **`Intl.DateTimeFormat` вЂ” renderer for Temporal values.** From chapter 087, the app holds `Temporal.Instant` / `PlainDate` / `ZonedDateTime` in memory and renders at the edge. **Temporal interop.** Modern `Intl.DateTimeFormat` accepts Temporal types natively вЂ” `formatter.format(instant)` uses the formatter's `timeZone` to resolve; `format(plainDate)` uses the formatter's default fields; `format(zonedDateTime)` uses the value's tz unless the formatter overrides. No `new Date()`, no `.toISOString()` round-trip. **The `timeZone` option is mandatory.** From lesson 3 of chapter 087: a formatter without `timeZone` defaults to the runtime's tz вЂ” UTC on Vercel, the developer's local on `pnpm dev`. The wrapper `formatDate(value, { locale, timeZone, ...options })` *requires* both. **Style presets vs. component options.** Presets (`dateStyle: 'short' | 'medium' | 'long' | 'full'` paired with `timeStyle`) вЂ” senior default; or component options (`year`, `month`, `day`, `hour`, ...) for control; mixing is an error. **`formatRange`.** Renders date ranges idiomatically вЂ” "Jan 5вЂ“7, 2026" in en-US, "5вЂ“7 janv. 2026" in fr-FR. Booking windows, billing periods, event durations.
- **`Intl.RelativeTimeFormat` вЂ” "3 days ago" the right way.** `new Intl.RelativeTimeFormat(locale, { numeric: 'auto' })`; `formatter.format(-3, 'day')` returns `'3 days ago'` in en-US, `'il y a 3 jours'` in fr-FR. `numeric: 'auto'` allows special-cases ("yesterday", "tomorrow", "now"); `'always'` always returns the numeric form. Pair with Temporal: `since` / `until` returns a `Duration`, the largest non-zero unit becomes the formatter input, sign matters (negative for "ago"). A small `formatRelative(instant, { locale, now })` helper hides duration-to-unit logic. Watch-out: don't chain "3 minutes ago" with auto-update on a timer that re-renders the whole page вЂ” use a stable "now" anchor per request to avoid hydration mismatches; for live timestamps, a small client island re-renders every minute.
- **`Intl.PluralRules` вЂ” standalone, restated.** lesson 2 of chapter 088's plural messages call this engine. Direct use is rare вЂ” sometimes a custom branching that doesn't fit ICU (choosing between two icons by plural category). Cardinal by default; pass `{ type: 'ordinal' }` for ordinal.
- **`Intl.Collator` вЂ” locale-aware sort and search.** `new Intl.Collator(locale)` returns a comparator (`collator.compare(a, b)` for `Array.sort`). **Sensitivity.** `'base'` ignores accents and case вЂ” senior default for search; `'accent'` distinguishes accents, ignores case вЂ” senior default for sort respecting diacritics; `'case'`, `'variant'` (default). **`usage`.** `'sort'` (default) for ordering; `'search'` for substring / equality matching. **`numeric`.** `numeric: true` enables natural-numeric sort (`'item2'` before `'item10'`); senior default for filenames, version strings. **`caseFirst`.** Rarely needed. Wrong reach: `string.localeCompare(other)` constructs a fresh collator per comparison; for large sorts, construct once and pass `collator.compare` to `.sort`.
- **`Intl.ListFormat` вЂ” natural-language lists.** `new Intl.ListFormat('en', { style: 'long', type: 'conjunction' }).format(['Alice', 'Bob', 'Carol'])` в†’ `'Alice, Bob, and Carol'`; in French в†’ `'Alice, Bob et Carol'`. Types: `'conjunction'` ("and"), `'disjunction'` ("or"), `'unit'` (compact вЂ” "1h 30m"). Reach for it whenever joining a translatable list вЂ” never `array.join(', ')`.
- **`Intl.DisplayNames` вЂ” language, region, currency, script names.** `new Intl.DisplayNames('en', { type: 'language' }).of('fr')` в†’ `'French'`. Two senior reaches: render each locale's name *in its own locale* in language pickers (`'FranГ§ais'`, not `'French'`), and render in the current user's locale for region / currency labels.
- **`Intl.Segmenter` вЂ” named once.** Locale-aware grapheme / word / sentence segmentation. Reach for it for truncating user-generated text (visual grapheme count) and CJK tokenization. Out of daily-reach scope; depth in lesson 4 of chapter 005.
- **The cache pattern.** `getNumberFormatter(locale, options)` / `getDateFormatter(...)` / `getCollator(...)` keyed by `JSON.stringify(options) + locale` in a module-scope `Map`. next-intl's `useFormatter` / `getFormatter` do this internally; standalone code does it explicitly. Construction is double-digit ms; per-row in a 1000-row table that's seconds.
- **The "locale is a contract, not a string" rule.** Every formatter takes a BCP 47 tag. `'en'` is incomplete вЂ” different conventions for date and currency between `'en-US'` and `'en-GB'`. The profile column stores the full tag (`'en-US'`, `'fr-FR'`); the formatter receives it directly. `'en'` as a stored value is a smell.
- **Watch-outs.** `value.toLocaleString()` without arguments uses runtime locale + default format вЂ” same bug as `Intl.DateTimeFormat()` with no arguments; constructing the formatter inside render is the scale perf bug; passing a `Date` when the codebase uses Temporal is a type-discipline leak вЂ” convert at the seam; hard-coded `'$'` for non-USD accounts is wrong; "3 days ago" from `Date.now()` math drifts on DST; per-element `localeCompare` in a sort callback is N constructions вЂ” use one collator; `numeric: 'always'` for live timestamps that hit zero displays "0 minutes ago" вЂ” guard the zero case; `formatter.format(NaN)` returns locale-specific NaN вЂ” guard at the seam; CLDR data is bundled with the runtime вЂ” pin in CI for snapshot tests.

What this lesson does not cover:

- Translation-key discipline вЂ” lesson 1 of chapter 088.
- ICU plurals / select вЂ” lesson 2 of chapter 088.
- Locale negotiation and the `users.locale` column вЂ” lesson 4 of chapter 088.
- next-intl's formatter hooks вЂ” lesson 5 of chapter 088.
- The hreflang / SEO surface вЂ” lesson 6 of chapter 088.
- BiDi / RTL вЂ” out of scope.
- `Intl.Segmenter` at depth вЂ” named once.

Estimated student time: 50 to 60 minutes. The chapter's deepest mechanics lesson; the daily-reach surface for every locale-aware render.

## Lesson 4 вЂ” The locale resolution chain

The five-input negotiation order (URL prefix, profile, cookie, `Accept-Language` best-match, default), BCP 47 `Lookup` via `@formatjs/intl-localematcher`, and `users.locale` paired with `users.timeZone`.

Topics to cover:

- **The senior question.** A French user signs in from a German hotel; the browser sends `Accept-Language: de-DE,de;q=0.9,en;q=0.7`. The user has `'fr-FR'` saved in their profile. A second user вЂ” anonymous, on the marketing site вЂ” sends `Accept-Language: pt-BR,pt;q=0.9`. The product supports `'en-US'`, `'fr-FR'`, `'de-DE'`, `'es-ES'`, `'pt-PT'`. What locale renders, and what's the negotiation order that makes the call deterministic? Senior reach: a single **resolution chain** with explicit priorities вЂ” URL prefix в†’ profile в†’ cookie в†’ `Accept-Language` best-match в†’ default вЂ” that runs once per request in middleware, lands on the request as a typed `locale`, drives every formatter, every `t()`, every `<html lang>`.
- **The five inputs.** **(1) URL prefix** вЂ” `/fr/dashboard` carries the locale explicitly; strongest signal. **(2) Profile column** вЂ” `users.locale`, set on the settings page. **(3) Cookie** вЂ” locale-switcher writes `NEXT_LOCALE`; persists for anonymous users. **(4) `Accept-Language` header** вЂ” browser preference list with q-values. **(5) Geo-IP** вЂ” Vercel's `request.geo.country` etc.; "user in France, default to French" is a *fallback* only, never primary вЂ” a French speaker in Brazil is real; an English-speaking expat in Tokyo doesn't want Japanese. **Never geo-IP as a primary signal**; named once.
- **The resolution chain in order.** **(1)** URL has a locale prefix в†’ that's the locale. **(2)** Authenticated user в†’ `users.locale`. **(3)** `NEXT_LOCALE` cookie. **(4)** Best-match `Accept-Language` against the supported set. **(5)** Default locale. Chain runs in middleware (`proxy.ts` or next-intl's middleware); URL gets rewritten to the prefixed form so downstream rendering is uniform. Audit: grep for any locale read that doesn't come from the middleware's resolved value.
- **BCP 47 best-match against the supported set.** RFC 4647 defines `Lookup` (most specific) and `Filter` (all matching). For negotiation, `Lookup` is senior. Algorithm: parse `Accept-Language` into ranked tags with q-values; for each in priority order, try the exact tag, fall back to base language (`de-DE` в†’ `de`), try the next tag, finally return default. **Implementation in 2026.** `Intl.LocaleMatcher` is TC39 Stage 1, not yet native; senior reach is `@formatjs/intl-localematcher` (the proposal's ponyfill, used by every major i18n library) or next-intl's built-in matcher. Chapter's wiring delegates to next-intl in middleware; standalone code outside the middleware (a Server Action, a CLI script) imports the ponyfill directly.
- **The `users.locale` column, paired with `users.timeZone`.** `text` on `users`, NOT NULL, `'en-US'` default. Full BCP 47 tag вЂ” never just the language code. Validated at the write edge against `SUPPORTED_LOCALES` (`as const` array from `lib/i18n.ts`); Zod schema is `z.enum(SUPPORTED_LOCALES)`. Profile UI presents a `<select>` with each option's name rendered in its own locale (`Intl.DisplayNames`). Seeded on sign-up from the browser (`navigator.language` or the negotiated value); editable on profile; propagates to every server-rendered formatter via session.
- **Locale and timezone are independent.** A Berlin user can read in `'en-GB'` and operate in `'Europe/Berlin'`. A San Francisco user can read in `'es-ES'` and operate in `'America/Los_Angeles'`. Two columns, two pickers, no inference.
- **The marketing-site / anonymous-user shape.** No profile; chain collapses to URL в†’ cookie в†’ `Accept-Language` в†’ default. The cookie is senior reach for "the user picked French on marketing, the SEO crawler isn't redirected, the next anonymous visit honors the choice." next-intl 4 defaults the locale cookie to a *session cookie* and only writes it when the user picks a locale that differs from `Accept-Language` вЂ” both choices are GDPR-driven (a cookie mirroring the header is "strictly necessary"; an overriding cookie carries consent implications). Locale cookie is essential-for-functionality, not consent-gated.
- **What `Accept-Language` carries and doesn't.** Carries: ranked BCP 47 tags with optional q-values. Does *not* carry: timezone, currency preference, calendar system, region of residence (`en-US` from a German user on vacation is real). The senior framing: `Accept-Language` is *one* signal among five, not truth. Falling through without best-match produces the canonical bug вЂ” browser sends `'pt-BR'`, supported set is `['pt-PT', 'en-US']`, naive code defaults to `'en-US'` instead of `'pt-PT'`.
- **The locale-switcher UI.** Dropdown in every layout; selection writes the cookie, calls a Server Action to update `users.locale` for authenticated users, navigates to the same path under the new prefix. next-intl's `useRouter` / `usePathname` handle the prefix rewrite. Display: each option in its own language ("English" / "FranГ§ais" / "Deutsch").
- **`<html lang>` вЂ” the load-bearing accessibility hook.** Set on the root layout: `<html lang={locale}>`. Screen readers, hyphenation, browser spell-checkers, search engines, machine translation all key on it. Pair with `dir={isRTL(locale) ? 'rtl' : 'ltr'}` for RTL (out of chapter scope; named).
- **`Content-Language` response header.** Mirrors `<html lang>` вЂ” `Content-Language: fr-FR` on the response. Crawlers and proxies key on it; cheap to set in middleware.
- **The "every render reads the resolved locale" rule.** Negotiation runs once in middleware; downstream code reads via next-intl's `useLocale()` / `getLocale()`. No component, no formatter re-reads `Accept-Language` or `navigator.language`. Audit grep: `Accept-Language` outside middleware, `navigator.language` outside the sign-up form's tz-detection capture.
- **Watch-outs.** Trusting `Accept-Language` without best-match ships the default to every non-exact match; storing a language-only tag (`'en'`) is wrong вЂ” store full BCP 47; auto-redirecting on geo-IP breaks deep links and SEO crawlers; updating profile locale silently without updating the cookie leaves anonymous navigation in the old locale; locale-switching on a deep route should preserve the path (next-intl's `usePathname` handles it); rendering `<html lang>` from the cookie instead of the resolved value produces hydration mismatch.

What this lesson does not cover:

- Translation-key / catalog discipline вЂ” lesson 1 of chapter 088.
- ICU MessageFormat вЂ” lesson 2 of chapter 088.
- `Intl.*` formatters вЂ” lesson 3 of chapter 088.
- `next-intl`'s middleware, routing, hooks вЂ” lesson 5 of chapter 088.
- The hreflang / SEO surface вЂ” lesson 6 of chapter 088.
- Geo-IP routing at depth вЂ” out of scope; named as anti-pattern.
- `navigator.languages` plural (multi-language readers) вЂ” out of scope; named once.

## Lesson 5 вЂ” Wiring next-intl into Next.js 16

The next-intl file shape вЂ” `defineRouting`, `createMiddleware`, `getRequestConfig`, `setRequestLocale`, `generateStaticParams`, `useTranslations` / `getTranslations`, `useFormatter`, typed navigation, and the `IntlMessages` global type.

Topics to cover:

- **The senior question.** lesson 1 of chapter 088 through lesson 4 of chapter 088 installed the discipline вЂ” keys, ICU, `Intl.*`, locale negotiation. The wiring question is which library lands the discipline in Next.js 16's App Router with the least drift. Senior 2026 default is **`next-intl`** вЂ” actively maintained, native Server-Component support, App-Router-shaped, ~2KB client bundle, ICU MessageFormat built in, navigation APIs (`<Link>`, `redirect`, `usePathname`, `useRouter`) preserving the locale prefix, build-time message typing, Next.js 16 compatibility confirmed. Alternatives вЂ” `react-intl` / FormatJS (full ICU but no App-Router routing primitive), `next-i18next` (Pages-Router-shaped), Lingui (compile-step heavy) вЂ” exist; chapter's reach is next-intl. The lesson lands the wiring as a Setup/Wiring archetype: every file, every named export, every option that matters.
- **The file shape вЂ” six places.** **(1)** `lib/i18n/routing.ts` вЂ” `defineRouting({ locales, defaultLocale, localePrefix })`. **(2)** `lib/i18n/navigation.ts` вЂ” `createNavigation(routing)` returns typed `Link`, `redirect`, `usePathname`, `useRouter` that respect the locale prefix. **(3)** `proxy.ts` (Next.js 16's renamed middleware) вЂ” `createMiddleware(routing)` runs the negotiation chain and rewrites the URL. **(4)** `i18n/request.ts` вЂ” per-request config callback that resolves the locale (via `getRequestConfig`), loads `messages/${locale}.json`, returns `{ messages, timeZone, now }`. `timeZone` is the user's `users.timeZone` when authenticated, falling back to org or default. **(5)** `app/[locale]/layout.tsx` вЂ” root layout consumes the locale param, calls `setRequestLocale(locale)`, wraps children in `<NextIntlClientProvider>` (only when client components consume translations), sets `<html lang={locale}>`. **(6)** `messages/{en,fr,de}.json` вЂ” catalogs from lesson 1 of chapter 088.
- **`defineRouting`.** `defineRouting({ locales: ['en-US', 'fr-FR', 'de-DE'] as const, defaultLocale: 'en-US', localePrefix: 'as-needed' })`. **Prefix modes.** `'always'` вЂ” every URL prefixed; deterministic but ugly for the default locale; senior reach for multi-locale-from-day-one products. `'as-needed'` вЂ” default unprefixed (`/dashboard`), others prefixed (`/fr-FR/dashboard`); senior reach for products with a strong default-locale market. `'never'` вЂ” locale entirely from cookie / `Accept-Language`; rare; SEO suffers. Chapter default: `'as-needed'`.
- **`createMiddleware` вЂ” the negotiation runner.** `export default createMiddleware(routing);` in `proxy.ts`. Reads `Accept-Language`, cookie, URL prefix, runs BCP 47 best-match against `routing.locales`, rewrites URL to prefixed form (or canonical unprefixed for default), persists cookie. Composes with other middleware (auth, security headers) via Next.js's `proxy.ts` chaining pattern (Unit chapter 037 / chapter 058 / lesson 1 of chapter 085).
- **`getRequestConfig` вЂ” per-request message and tz resolution.** `i18n/request.ts` exports `default getRequestConfig(async ({ requestLocale }) => { ... })`. Returns `{ locale, messages, timeZone, now }`. **Loading messages.** `messages: (await import(`../../messages/${locale}.json`)).default` вЂ” dynamic import, code-splits per locale. **Timezone.** Read `users.timeZone` via auth session for authenticated requests; default `'UTC'` for anonymous. The single seam where locale, tz, and catalog converge; downstream code never reaches for `Accept-Language` or the cookie.
- **`setRequestLocale` and static rendering.** Next.js 16 prefers static rendering by default. next-intl reads the locale via `headers()` (dynamic) unless `setRequestLocale(locale)` is called at the top of the layout / page вЂ” writes the locale to a per-request store, makes downstream `useTranslations` calls work without `headers()`, and re-enables static rendering. Senior rule: every `app/[locale]/**/page.tsx` and `app/[locale]/**/layout.tsx` starts with `setRequestLocale(locale)`. Skipping it converts to dynamic silently вЂ” Lighthouse perf hit is real, invisible at code review without the rule.
- **`generateStaticParams` for prerender.** Root locale layout exports `generateStaticParams() { return routing.locales.map((locale) => ({ locale })); }`. Each locale prerenders at build. Pairs with `setRequestLocale`; without both, only the default prerenders.
- **`useTranslations` vs. `getTranslations`.** Two APIs, same engine. **`useTranslations(namespace)`** вЂ” sync hook, works in Server and Client Components, returns `t`. Senior default for *most* rendering. **`getTranslations({ locale, namespace })`** вЂ” async function, for `generateMetadata` (async, not a component), Server Actions outside the component tree, route handlers. `use*` in the render tree, `get*` outside it.
- **`t.rich` for embedded markup.** From lesson 1 of chapter 088: `t.rich('terms.line', { link: (chunks) => <Link href="/terms">{chunks}</Link> })` with message `'By signing up, you agree to our <link>Terms</link>.'`. Returns a `ReactNode`. Senior reach for any string with embedded interactive elements; replaces the "split into three keys" anti-pattern.
- **`useFormatter` / `getFormatter` вЂ” the `Intl.*` wrapper.** Convenience over `NumberFormat` / `DateTimeFormat` / `RelativeTimeFormat`. `format.number(amount, { style: 'currency', currency: 'USD' })`, `format.dateTime(instant, { dateStyle: 'long', timeZone })`, `format.relativeTime(instant, now)`. Locale and request tz wired automatically; engineer passes options. Daily reach inside next-intl's tree; standalone `Intl.*` (lesson 3 of chapter 088) is the form for code outside the tree (utility module, CLI, test fixture).
- **The `Messages` global type вЂ” compile-time key safety.** next-intl 4: declare `interface AppMessages extends Messages {}` and `declare global { interface IntlMessages extends AppMessages {} }`, with `AppMessages` typed from the source JSON (`typeof import('../messages/en.json')`). `t('invoice.greeting')` is typed; `t('invoice.greting')` is a compile error; missing placeholder values are caught at type level. The build step that turns `en.json` into the type is in `tsconfig` / `next.config` alignment.
- **Server Actions and route handlers.** Both async. `'use server'` action: `const t = await getTranslations({ locale: await getLocale(), namespace: 'inbox' });`. Same in `route.ts`. Actions and routes are *outside* the React render tree, so hooks don't apply; the `get*` async siblings cover them.
- **The "every component goes through next-intl" rule.** Once wired: every user-read string goes through `t()` (with `t.rich`), every number / date / relative-time through `format.*`, every navigation through the typed `Link` / `redirect`. Audit grep: hard-coded JSX strings, `Date.prototype.toLocaleString`, `Intl.NumberFormat` outside `useFormatter` / `lib/format.ts`, `next/link` in `app/[locale]/...`.
- **Static-vs-dynamic decision.** Public marketing в†’ static + `setRequestLocale` + prerender per locale в†’ CDN-cacheable. Authenticated app в†’ dynamic (auth forces it anyway). Rule: never call `headers()` / `cookies()` inside a public layout without intending to opt out of static.
- **Watch-outs.** Forgetting `setRequestLocale` silently converts to dynamic вЂ” perf regression invisible without monitoring; `next/link` instead of `lib/i18n/navigation` skips the prefix rewrite; loading every locale's messages eagerly ships the full catalog вЂ” use dynamic `import()`; `NextIntlClientProvider` wrapping the whole app sends the entire catalog to the client вЂ” wrap minimal subtrees with `messages={pick(messages, [...])}`; `getTranslations` inside a render loop is N async lookups вЂ” call once and pass `t` down; large catalogs (10K+ keys) tank build-time type generation вЂ” split into namespaces; runtime ICU parser is fine at 2026 scale but hot paths can pre-compile via the optional SWC plugin.

What this lesson does not cover:

- Translation-key discipline вЂ” lesson 1 of chapter 088.
- ICU MessageFormat at depth вЂ” lesson 2 of chapter 088.
- `Intl.*` engine вЂ” lesson 3 of chapter 088.
- Locale negotiation algorithm вЂ” lesson 4 of chapter 088.
- `hreflang` and per-locale SEO вЂ” lesson 6 of chapter 088.
- TMS integration вЂ” out of scope; named once.
- next-intl's `formats` global (locale-default presets) вЂ” deferred to project chapter 089.

Estimated student time: 50 to 60 minutes. The chapter's Setup/Wiring lesson; cashes in every prior discipline as concrete file shape.

## Lesson 6 вЂ” hreflang, per-locale canonicals, and SEO

The `alternates.languages` metadata surface with self-reference and bidirectionality, `x-default`, locale-specific canonicals, per-locale sitemaps, and OG `og:locale` with locale-aware OG images.

Topics to cover:

- **The senior question.** A French user Googles "logiciel de facturation" from Paris; the SaaS has `/fr-FR/billing` and the English original `/billing`. Without the right SEO wiring, Google shows the English page in the French SERP, the user bounces, the team blames "the translation isn't ranking." The fix is structural: `hreflang` tags tell Google which URLs are which-language alternates; `alternates.canonical` declares the page's canonical URL (the locale-prefixed one, not the default); a per-locale sitemap entry surfaces each variant for crawl; Open Graph `og:locale` + `og:locale:alternate` carry the same signal for social. The lesson lands the full surface as a Pattern archetype: code block named for what it prevents (wrong-locale serving), failure mode without it (organic French traffic landing on English), structural form that makes the bug hard to write.
- **What `hreflang` is, precisely.** `<link rel="alternate" hreflang="fr-FR" href="https://app.example.com/fr-FR/billing">` вЂ” one per alternate locale including self-reference вЂ” tells Google the page has a French variant at that URL. Google indexes both, serves the right one based on the user's locale signals. Without `hreflang`, Google treats each locale as a separate page and may rank only the one with the most backlinks. With it, variants share authority.
- **`hreflang` rules вЂ” the three that catch teams.** **(1)** Self-reference is mandatory вЂ” the French page lists itself as `hreflang="fr-FR"` alongside the English alternate. **(2)** Bidirectionality is mandatory вЂ” if `/fr-FR/billing` lists `/billing` as `hreflang="en-US"`, then `/billing` must list `/fr-FR/billing` as `hreflang="fr-FR"`. Google validates both sides; one-sided declarations are silently ignored. **(3)** `x-default` (`hreflang="x-default"`) declares the fallback for users whose locale doesn't match any alternate вЂ” typically the English page or a locale-picker landing. Chapter's senior reach: default locale's URL is the `x-default`, declared on every page.
- **Next.js 16's `alternates.languages` metadata.** Framework wires `hreflang` through the `Metadata` from `generateMetadata`. Shape: `alternates: { canonical: '/fr-FR/billing', languages: { 'en-US': '/billing', 'fr-FR': '/fr-FR/billing', 'de-DE': '/de-DE/billing', 'x-default': '/billing' } }`. Next.js generates the `<link rel="alternate" hreflang="...">` tags. Senior pattern: `generateAlternates(pathname)` helper in `lib/seo.ts` that maps over `routing.locales`. Every `generateMetadata` calls it; missing pages are findings. **Critical 2026 warning.** Next.js's `alternates.languages` can in some configurations render hreflang tags in the body вЂ” Google ignores body-level hreflang. Validate that they land in `<head>`; the framework's `generateMetadata` path is correct as long as the export is at the page / layout root and not from a client component. CI assertion: a Playwright check that production HTML has the expected number of head-level hreflang tags per page (Playwright introduced in Chapter 094).
- **The canonical URL вЂ” the locale-prefixed one.** `alternates.canonical` declares the page's authoritative URL. Senior 2026 rule: **canonical is the locale-specific URL, not the default**. French page's canonical is `/fr-FR/billing`; English page's canonical is `/billing`. Wrong reach вЂ” canonicalizing all variants to the default вЂ” tells Google "these are duplicates of English, only rank the English one" and kills the French page's discoverability. Each variant is its own canonical.
- **Sitemap вЂ” index plus per-locale.** Sitemap index at `/sitemap.xml` points to per-locale sitemaps. Each lists URLs for that locale with `<lastmod>`. Senior reach with Next.js 16: a `sitemap.ts` route returning `{ url, lastModified, changeFrequency, priority, alternates: { languages: {...} } }`; Next.js's typed `MetadataRoute.Sitemap` supports `alternates.languages`, emitting `<xhtml:link>` tags inside each `<url>` вЂ” equivalent to head tags, often easier for large sites. Submit `/sitemap.xml` to Search Console; the index resolves variants.
- **`generateMetadata` вЂ” the daily reach.** Per-page, async, returns `Metadata`. Senior pattern: read `locale` and route params from `params`; `await getTranslations({ locale, namespace: 'Metadata' })` for the localized title / description / OG title; `generateAlternates(pathname)` for the languages map; assemble `Metadata` with `title`, `description`, `alternates`, `openGraph`, `twitter`. Single seam where i18n, SEO, and social cards converge. Audit: every page in `app/[locale]/` has a `generateMetadata` that calls both `getTranslations` and `generateAlternates`.
- **Open Graph `og:locale` and `og:locale:alternate`.** Facebook, LinkedIn, Slack, X read these. `og:locale` is the page's locale in underscore form (`'fr_FR'`, not `'fr-FR'`); `og:locale:alternate` is repeated per alternate. Next.js's `Metadata.openGraph.locale` and `Metadata.openGraph.alternateLocale` accept the underscore form. Senior reach: derive both from `routing.locales` via a `bcp47ToOgLocale(locale)` helper.
- **Locale-aware OG images.** Share card carries title, description, image. The image is locale-aware when it contains text вЂ” "Billing software" in English, "Logiciel de facturation" in French. Next.js's `ImageResponse` from `opengraph-image.tsx` accepts `params.locale` and renders per-locale text via `getTranslations`. Senior framing: per-locale OG images are a strong SEO/social signal for translated marketing pages, weak for behind-auth pages. Per-locale OG on marketing; default-locale OG inside the app.
- **`<html lang>` restated.** From lesson 4 of chapter 088: load-bearing accessibility / SEO. Search engines key on it; mismatches between `<html lang>` and `hreflang` produce ranking inconsistency. Middleware-resolved locale drives both; audit step is to confirm `<html lang>` matches the URL prefix.
- **`robots.txt` and per-locale indexing.** Senior 2026 default: allow all locales to be indexed; `hreflang` handles disambiguation. Carve-out: soft-launch locales can be `noindex` via `metadata.robots = { index: false }`. Never block via `robots.txt` for indexing decisions вЂ” `robots.txt` blocks crawling, preventing Google from seeing the page exists; for "don't rank yet" use `noindex`.
- **The "untranslated content" rule.** A page whose translation hasn't shipped вЂ” English source exists, French is empty. Two senior options. **(a)** Fall through to source locale (`en-US`) and serve it under `/fr-FR/...`; do *not* list this URL as a French `hreflang` alternate (content is English; Google sees duplicate). **(b)** `noindex` the untranslated locale until the catalog ships. Chapter's senior default: (b) for marketing (SEO quality matters), (a) for app (fallback prevents broken UX). Document the choice in `lib/i18n.ts` next to `SUPPORTED_LOCALES`.
- **Marketing vs. app, restated for SEO.** Marketing (`/`, `/pricing`, `/features`) needs the full `hreflang` / sitemap / OG-image surface вЂ” the SEO target. Authenticated app (`/dashboard`, `/settings`) is `noindex` by default; `hreflang` unnecessary; locale resolution still runs but the SEO surface is dark. Discipline applies to both; only marketing ships public `hreflang` / sitemap entries.
- **Search Console validation.** Submit the sitemap index. Read "International Targeting." Fix `hreflang` errors (most common: missing return links, malformed BCP 47, missing `x-default`). Senior reach: a CI smoke test (lesson 3 of chapter 101) that fetches rendered HTML for each locale and asserts the `<link rel="alternate">` tags match expected; runs on every PR.
- **Watch-outs.** Canonicalizing all locales to the default kills translated rankings; missing self-reference makes Google drop the relationship; one-sided `hreflang` is silently ignored; `'fr'` instead of `'fr-FR'` produces vague targeting вЂ” match the URL's tag; `og:locale` in hyphen form (`fr-FR` instead of `fr_FR`) is a frequent typo Next.js's metadata typing won't catch; identical content under different locale URLs triggers near-duplicate detection вЂ” translate or `noindex`; submit the sitemap index, not per-locale individually; serving the locale-detection redirect to crawlers prevents indexing the source URL; lazy OG image rendering on every share adds latency вЂ” cache or pre-generate for marketing.

What this lesson does not cover:

- Translation-key discipline вЂ” lesson 1 of chapter 088.
- ICU MessageFormat вЂ” lesson 2 of chapter 088.
- The `Intl.*` engine вЂ” lesson 3 of chapter 088.
- Locale negotiation вЂ” lesson 4 of chapter 088.
- next-intl wiring вЂ” lesson 5 of chapter 088.
- Schema.org / structured data localization вЂ” out of scope; named once.
- A/B-testing locale variants вЂ” out of scope.
- Country-targeted regional variants (`en-US` vs. `en-GB` vs. `en-AU`) at depth вЂ” named once; chapter ships language-targeted only.
