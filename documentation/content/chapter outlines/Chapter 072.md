# Chapter 072 — Cache decisions as architecture

## Chapter framing

Chapter 072 turns the caching primitives the student already knows into a decision discipline. By this point they have the mechanics from Chapter 032 — `use cache`, `cacheLife`, `cacheTag`, `updateTag`, `revalidateTag`, `revalidatePath`, `router.refresh`, `connection()`, and the `cacheComponents: true` default — and have shipped Server Actions (Unit 6), an orgs-scoped data layer (Unit 9), webhooks (Unit 11), and the list-view pattern (Unit 10). What's missing is the senior-shaped answer to "which routes earn caching, what's the tag scheme that scales across a real SaaS, and which of the four invalidation calls fires after each mutation." The chapter teaches caching as an architectural decision: a per-route classification (always-dynamic, partially-cached, fully-static) the team commits to up front, a tag scheme keyed to the org-scoped data model so a single mutation invalidates exactly the right entries, and a four-way decision tree on the invalidation surface driven by one question — does this user expect to see their own change on the next view? The output is the mental model and the conventions Chapter 073 implements on the Unit 10 list.

Threads that run through every lesson: caching is opt-in under Cache Components and the default for authenticated SaaS surfaces is dynamic — the burden of proof is on caching, not on not caching; the cacheable surfaces in a SaaS cluster on read-heavy reference data (org membership, plan entitlements, lists viewed many times between edits, public marketing pages) and the rest stays dynamic; the tag scheme is the named contract between the cached read and the mutating write, and it must encode the same scoping the data model already enforces (org-scoped tags for org-scoped data, user-scoped tags for user-scoped data, entity-record tags for record-level invalidation); the four-way invalidation surface (`updateTag`, `revalidateTag`, `revalidatePath`, `router.refresh`) splits on two axes — read-your-writes vs. stale-while-revalidate, and tag-precise vs. path-or-tree-coarse — and the user-expectation question picks the corner; `updateTag` is Server-Action-only because read-your-writes requires the in-band redirect path the action provides; `revalidateTag` is the right tool for webhook-driven and background-job mutations because the triggering user is not the only viewer and stale-while-revalidate is the correct UX; `revalidatePath` exists for the case where the tag scheme can't capture the affected surface (rare in a disciplined codebase) and `router.refresh` is a client-side last resort for fully-dynamic routes that need a fresh server render after a non-action interaction. The chapter ships two teaching lessons plus a quiz: classification and tag scheme, then the four-way decision tree on worked SaaS cases.

## Lesson 1 — Route classes and the tag scheme

Classify every route as fully dynamic, partially cached, or fully static, then design an org-scoped tag scheme (entity, record, org, user) funneled through a `tags.ts` helper so read-side and write-side strings always align.

Topics to cover:

- **The senior question.** Open the app's `app/` directory. For each route, what's the right caching posture — fully dynamic, partially cached with a static shell, or fully static? And once a route is partially cached, what tag does each cached read carry so the mutation that changes its underlying data can invalidate it without touching anything else? The lesson lands a classification the team agrees on up front and a tag scheme that scales with the data model.
- **The default for authenticated SaaS is dynamic — and that's correct.** Under `cacheComponents: true` (lesson 1 of chapter 032), routes are dynamic until a `use cache` boundary is added. The senior reflex for an authenticated, per-user, per-org surface is to leave it dynamic: the data changes often, the audience is one user, the cached payload would have a hit rate near zero. Caching earns its weight when reads are repeated and shared; per-user authenticated reads usually are not. The lesson names the rule: prove the cache is worth it, don't assume.
- **The three classes — fully dynamic, partially cached, fully static.** Fully dynamic: every read at request time, no `use cache` anywhere; the dashboard, the inbox, the settings page, anything per-user-per-second-changing. Partially cached: a dynamic outer shell wraps cached subtrees behind Suspense boundaries; the list view with a cached header and reference-data sidebar, the detail page with a cached entity that mutates rarely. Fully static: no dynamic signal anywhere, the entire page can prerender at build; marketing pages, the pricing page (where plan content is build-time), the docs site. The lesson surfaces the three as a checklist a senior runs on every new route.
- **The cacheable-in-a-SaaS shortlist.** Read-heavy reference data with a long edit-to-view ratio: plan entitlements (Unit 11), org membership lists, role definitions, feature flags (read every request, edited rarely), public marketing pages, the pricing page, the docs site, OG image generators, route-shell metadata. Reach for `use cache` when the same value is read by many users many times between writes. The lesson lists the shortlist and the question that admits a candidate: read frequency divided by write frequency. High ratios cache; low ratios don't.
- **The not-cacheable list — the senior reflex.** Personalized lists where the URL state owns the filter (Unit 10), real-time dashboards, inbox feeds, anything behind `cookies()` or `headers()` that depends on the request, anything whose payload includes a `now()` or a counter that updates with every action. The lesson names them so the student doesn't waste time trying.
- **Why `cacheLife` profiles are a UX question, not a perf question.** The three numbers (stale, revalidate, expire) describe how long the user is willing to see slightly-old data. The 2026 built-in profiles — `'seconds'`, `'minutes'`, `'hours'`, `'days'`, `'weeks'`, `'max'` — span from "barely cached" to "effectively static." The senior call: a list of orgs the user is a member of is fine on `'hours'` because membership changes rarely; a feature-flag read is fine on `'minutes'` because rollouts are minute-scale; plan entitlements read inside an action handler want `'minutes'` with explicit invalidation on plan change. The lesson surfaces profile selection as a product question.
- **The tag scheme — the named contract between read and write.** A cached read declares one or more tags via `cacheTag(...)`. A mutating write calls `updateTag` or `revalidateTag` with the same tag string. The scheme is the convention the team commits to so the strings on both sides line up. Without a scheme, tags become free-form and stale entries linger because the write site picked a different string. With a scheme, the tag is a function of the entity and its scope and the strings are mechanical.
- **The four tag shapes — entity, record, org-scoped, user-scoped.** Entity-level tag (`invoices`) invalidates every cached read of any invoice. Record-level tag (`invoice:${id}`) invalidates a single record's cached read. Org-scoped tag (`org:${orgId}:invoices`) invalidates every invoice cached read for one org. User-scoped tag (`user:${userId}:notifications`) for user-private data. Cached reads attach the union of applicable tags (a single invoice cached read tags itself with `invoices`, `invoice:${id}`, and `org:${orgId}:invoices` so any of the three writers can invalidate it). Write sites pick the narrowest tag that captures the change.
- **Tag-string conventions.** Lowercase, colon-delimited, scope first, entity next, id last (`org:${orgId}:invoices`, `invoice:${id}`). No interpolated user input that isn't already an id; no PII; under the 256-character limit (the framework's hard cap, well below it in practice). The lesson names the conventions; the project enforces with a `tags.ts` helper.
- **The tag helper — making string typos impossible.** A `lib/cache/tags.ts` file exports small functions: `invoicesTag()`, `invoiceTag(id)`, `orgInvoicesTag(orgId)`. Cached reads import them; write sites import the same functions. The string only exists in one file. Refactoring a tag is a function rename, not a grep. The lesson surfaces the helper as the structural enforcement that keeps the scheme honest.
- **Org-scoped tags compose with the org-scoped query helper.** Unit 9's `tenantDb` (lesson 2 of chapter 056) already enforces the org boundary on every read. The org-scoped tag mirrors that boundary in the cache layer: `org:${orgId}:invoices` matches the same scope. The composition means a mutation inside `tenantDb` calls `updateTag(orgInvoicesTag(orgId))` and the cached reads scoped to that org invalidate; cached reads for other orgs survive. The lesson surfaces the alignment between the data-layer scoping and the cache-layer scoping as a design principle — the same scope shape runs through both.
- **Record-level vs. entity-level — the granularity decision.** An invoice list cached read should tag itself with `org:${orgId}:invoices` because any invoice in the org changing should invalidate it. An invoice detail cached read should tag itself with `invoice:${id}` because only this invoice changing matters. The senior call: tag at the granularity the cached read needs, not at the finest granularity available. Over-narrow tags mean the write must remember every cached read that depends on the entity, which is the bug that makes caches go stale.
- **The `fetchedAt` discipline — proving the cache works.** Every cached read emits a `fetchedAt: new Date().toISOString()` inside the cached function. The render shows it (during development, behind a flag in production). When `fetchedAt` is stable across page loads, the cache is hitting; when it advances, the cache invalidated. The senior diagnostic: any cache-related bug starts with looking at `fetchedAt`. The lesson surfaces the discipline; Chapter 073's project wires it as the verification surface.
- **The "no closures over request data" rule, restated for tags.** A `use cache` function cannot read `cookies()`, `headers()`, or any request-scoped value inside; the same rule means the tags it emits can only be functions of its arguments (lesson 3 of chapter 032). The implication: org-scoped tags require the cached function to take `orgId` as an explicit argument, not read it from the session. The lesson restates the rule because it's load-bearing for the scheme — tags are arguments, not ambient state.
- **The classification document — a one-page table the team agrees on.** Each route gets a row: path, class (dynamic / partial / static), cached subtrees if partial, tag set, `cacheLife` profile. The document lives next to `next.config.ts` and is part of the project's architecture docs. The senior call: write it down before writing the cache code; reviewing it after the fact is when the wrong decisions calcify. The lesson names the artifact and what it contains.
- **What about caching mutations themselves — the trap.** Server Actions and route handlers are dynamic by definition; they don't get `use cache`. The temptation to cache the read inside an action ("the action wants the latest invoice, let me cache the lookup") is the bug — the action wants the freshest read and a cached read inside an action will serve stale data the action then writes a delta to. Read inside actions is uncached by default and stays that way. The lesson names the temptation and the rule.
- **Watch-outs:** marking every route as partially cached "to be safe" inverts the default and creates a constant audit burden — leave authenticated surfaces dynamic until proved otherwise; tags emitted as free-form strings at call sites mean a typo in the write silently fails to invalidate — funnel through `tags.ts`; record-level tags on a list read mean every mutation has to also call `updateTag(orgInvoicesTag(orgId))` defensively — tag the list with the org-scoped tag and the mutation only needs to fire that; cached reads that close over `orgId` from `auth()` instead of taking it as an argument compile but cache one entry across orgs and leak — pass scoping in as arguments; `cacheLife('seconds')` on a frequently-read public page does almost nothing for hit rate and costs the revalidation churn — pick `'hours'` or `'days'` when the data permits; per-user cached reads with a `userId` tag have a hit rate near zero — they're effectively uncached and only add complexity, keep them dynamic.

What this lesson does not cover:

- The `use cache`, `cacheLife`, `cacheTag` mechanics — lesson 3 of chapter 032 and lesson 4 of chapter 032 own them.
- The four invalidation calls applied to worked cases — lesson 2 of chapter 072.
- The `tenantDb` and org-scoped query helper — lesson 2 of chapter 056.
- Trigger.dev background jobs as cache invalidators — Chapter 066; this lesson names them as callers of `revalidateTag`.
- Edge-cache and CDN cache layers (Vercel's data cache, the browser cache) — out of scope; the lesson is about the Next.js cache.
- The `'use cache: private'` directive and per-user variants — named once as a power tool, deferred; not part of the SaaS shortlist.

Estimated student time: 50 to 60 minutes. The classification and the tag scheme are the chapter's load-bearing decisions.

## Lesson 2 — Picking the right invalidation call

Resolve `updateTag`, `revalidateTag`, `revalidatePath`, and `router.refresh` on two axes (read-your-writes vs. eventual, tag vs. path) and trace the decision tree through worked flows: post-edit lists, post-purchase plan flips, membership changes, webhooks, and background jobs.

Topics to cover:

- **The senior question.** A mutation just landed: a row was edited, a webhook fired a plan change, a background job rebuilt a summary. Which of the four invalidation calls runs — `updateTag`, `revalidateTag`, `revalidatePath`, or `router.refresh` — and how is that decided in a way that scales as the team grows? The lesson lands a two-axis decision tree the senior runs on every mutation site.
- **The four calls at a glance.** `updateTag(tag)` — Server-Action-only, immediately expires the entry, the next read fetches fresh, used when the same user expects to see their write on the next view (read-your-writes semantics). `revalidateTag(tag, profile)` — callable from actions, route handlers, and background code, marks the entry stale with stale-while-revalidate semantics so the next read serves stale and refreshes in the background; the `profile` argument is a `cacheLife` profile (use `'max'` as the senior default) and the single-argument form is deprecated in Next.js 16. `revalidatePath(path, type?)` — invalidates by URL path or route pattern, coarser than tags, used when the affected surface doesn't have a clean tag. `router.refresh()` — client-side, re-fetches the current route's server render from the browser; used after non-action client interactions on fully-dynamic routes.
- **The two axes — read-your-writes vs. eventual, tag vs. path.** Axis one: does the triggering user expect to see the change immediately on the next view, or is stale-for-a-moment acceptable? `updateTag` and `router.refresh` give read-your-writes; `revalidateTag` and `revalidatePath` give stale-while-revalidate. Axis two: can the change be addressed by a tag, or does it require a path? Tags are precise; paths are coarse. The four-corner table: `updateTag` (read-your-writes, tag), `revalidateTag` (eventual, tag), `revalidatePath` (eventual, path), `router.refresh` (read-your-writes, no tag — coarse via the current route).
- **The user-expectation question is the primary driver.** Most picks resolve on axis one. A user editing their own profile, posting a comment, changing their plan — all read-your-writes; the user sits on the redirect and stares at the page expecting their change to be there. A webhook updating an invoice status, a background job rebuilding the daily summary, an admin in another org touching shared reference data — all eventual; no specific user is waiting on this exact write, stale-while-revalidate is the correct UX. The lesson surfaces the rule: ask who triggered the mutation and whether they're watching.
- **`updateTag` and the Server Action redirect — why the restriction.** `updateTag` only works inside a Server Action because the read-your-writes guarantee requires the in-band path: the action mutates, calls `updateTag`, redirects, and the redirect's render sees fresh data. Outside an action (route handler, background job, webhook), the framework can't sequence the redirect, so the call is rejected. The lesson surfaces the restriction as a feature: the API tells you where you are.
- **Worked case — list view post-edit.** A user edits an invoice's amount on the detail page; the action updates the row and redirects to the list. The cached list read was tagged `org:${orgId}:invoices`. The action calls `await updateTag(orgInvoicesTag(orgId))` after the database write. The redirect renders the list page; the cached read sees its tag is expired and refetches; the list shows the new amount. Read-your-writes works because the same user who edited is the one viewing. The detail page cached read was tagged `invoice:${id}`; the action also calls `await updateTag(invoiceTag(id))` so a back-button to the detail shows fresh too. The lesson walks the example end-to-end with the tag set on both sides.
- **Worked case — post-purchase plan change.** The user clicks "Upgrade to Pro," the Stripe Checkout flow redirects to the success page. The actual plan flip happens in the webhook handler (Chapter 064), not in the action — the webhook is the only writer (Chapter 063). The webhook handler calls `await revalidateTag(orgPlanTag(orgId), 'max')`. The success page renders with a `router.refresh()`-polling pattern (lesson 3 of chapter 063) because the redirect and the webhook race — by the time the user sees the success page, the webhook may not have landed. The poll catches the moment the cache flips. The lesson surfaces the case as the exception where `router.refresh` earns its weight in a client component.
- **Worked case — post-membership-change.** An admin demotes a member. The Server Action runs inside the admin's session, writes the role change, and calls `await updateTag(orgMembersTag(orgId))`. The demoted member is a different user — they don't read-their-writes from the admin's action. The next time the demoted member loads a page, the cached membership read for that org is expired and the new role is in effect. The action also calls `await updateTag(userOrgsTag(memberUserId))` to invalidate the demoted member's own "orgs I'm in" cached read. The lesson surfaces the case as the multi-recipient pattern — one action, multiple tags, each scoping its own audience.
- **Worked case — webhook-driven invoice status.** Stripe sends `invoice.payment_succeeded`. The webhook handler updates the invoice row and calls `await revalidateTag(invoiceTag(id), 'max')` and `await revalidateTag(orgInvoicesTag(orgId), 'max')`. Stale-while-revalidate is correct: no specific user is watching, the next viewer sees the updated status with at most one render of stale data. `updateTag` would error here because we're in a route handler, not an action. The lesson surfaces the case as the canonical webhook shape.
- **Worked case — background-job summary rebuild.** A Trigger.dev nightly job recomputes per-org analytics summaries. The job updates the summary row and calls `await revalidateTag(orgSummaryTag(orgId), 'max')`. No user is waiting; the next dashboard view serves the previous-day summary stale-while-fresh-loads. Trigger.dev imports `revalidateTag` from `next/cache` and calls it directly — the framework supports cross-process invalidation via the deployment's cache backend. The lesson surfaces the case and the rule: background work uses `revalidateTag`.
- **`revalidatePath` — when the tag scheme can't capture it.** Reach for `revalidatePath` when the affected surface is a route pattern rather than a data entity — invalidating an entire route tree after a sitewide config change, or invalidating a path the team hasn't tagged. In a disciplined codebase the path-based call is rare because the tag scheme covers the cases. The lesson names the call, names its niche, and notes that overuse means the tag scheme is incomplete.
- **`router.refresh` — the client-side last resort.** Used after a non-action client interaction (a `<button>` that calls `fetch` directly, a third-party SDK callback) on a fully-dynamic route to force a fresh server render. Doesn't touch the cache directly — re-runs the route. Used in the success-page-polling pattern (lesson 3 of chapter 063) to bridge the webhook-redirect race. The senior call: if you're reaching for `router.refresh` often, the work probably belongs in a Server Action where `updateTag` would be cleaner. The lesson names the call and the smell.
- **The decision-tree diagram — the on-page artifact.** A small flowchart: "Is this a Server Action?" — no -> "Is it a webhook or background job?" -> yes -> `revalidateTag`. Yes (action) -> "Does the user expect to see the change immediately?" -> yes -> `updateTag` (or `router.refresh` if no clean tag) -> no -> `revalidateTag`. "Can the change be expressed as a tag?" -> no -> `revalidatePath`. The diagram is the lesson's center of gravity; the worked cases trace paths through it.
- **`updateTag` vs. `revalidateTag` — the user-expectation framing made operational.** The same mutation can be either call depending on who triggers it. A profile edit by the user themselves: `updateTag` from the Server Action. A profile edit propagated from an external identity provider via webhook: `revalidateTag` from the route handler. The senior call: name the trigger, name the expectation, pick accordingly. The lesson surfaces the framing so the student can run it on novel cases.
- **Multiple tags per mutation — the fan-out.** A single mutation usually touches more than one cached read. The action / handler calls `updateTag` (or `revalidateTag`) once per affected tag, sequentially. The calls are cheap; missing one means a stale read. The lesson surfaces the discipline: list the cached reads the mutation affects, invalidate each. The `tags.ts` helper makes the list short and mechanical.
- **The "invalidate after commit" rule, restated for caches.** The invalidation call runs after the database transaction commits, never inside it. An `updateTag` followed by a transaction rollback leaves the cache stale-with-the-rolled-back-state. The action sequence: transaction commits, then `updateTag`s, then `redirect`. The pattern mirrors the dispatcher's after-commit rule from lesson 2 of chapter 070.
- **The redirect-then-invalidate temptation — and why it's the wrong order.** A few codebases redirect first and rely on the redirect to trigger a fresh render. Without the explicit invalidation, the cached read on the destination still serves the old value. The redirect is not an invalidation signal. The rule: invalidate, then redirect. The lesson surfaces the bug because it's a common code-review catch.
- **Observability — counting invalidations in structured logs.** Each invalidation call gets a log line: `{ action: 'updateTag', tag, source }`. A dashboard surface shows invalidations per minute by tag. A spike means a hot mutation path; a flatline on a tag whose entity is being edited means the wiring is wrong. The lesson surfaces the metric, defers the dashboard to Unit 19.
- **Watch-outs:** calling `updateTag` from a route handler errors at runtime — the framework rejects it, the bug surfaces in webhook tests, fix by switching to `revalidateTag`; calling `revalidateTag` from a Server Action where read-your-writes is required produces a one-render delay the user perceives — use `updateTag`; relying on `router.refresh` to invalidate cached reads is the common confusion — `router.refresh` re-renders the route but cached entries with valid TTLs stay cached unless their tags are invalidated; tagging the write site with a different string than the read site means the cache silently stays stale — funnel both through `tags.ts`; running invalidations inside the transaction risks invalidating-then-rolling-back; multiple `await updateTag` calls in a tight loop block the redirect — list the unique tags and call each once; webhook handlers that forget to invalidate cached reads ship updates the user never sees until the cache expires — the webhook checklist (Chapter 063) includes the invalidation step.

What this lesson does not cover:

- The `use cache` directive and cached function constraints — lesson 3 of chapter 032.
- The tag scheme design itself — lesson 1 of chapter 072.
- The webhook idempotency and out-of-order guards — Chapter 063.
- The Trigger.dev job invocation surface — Chapter 066.
- Implementation on the Unit 10 list — Chapter 073's project.
- The cross-region cache backend semantics on Vercel — out of scope.
- The `'use cache: private'` variant for per-user caches — named, deferred.

Estimated student time: 50 to 60 minutes. The decision tree is the chapter's runnable artifact; every mutation site in the project leans on it.

## Lesson 3 — Quizz

Top 10 topics to quiz:

- The senior default for authenticated SaaS surfaces is dynamic — caching earns its weight on read-heavy reference data, not on per-user-per-second views.
- The three route classes — fully dynamic, partially cached (static shell + cached subtrees behind Suspense + dynamic content), fully static — and the question that admits a candidate to the cacheable shortlist (read-to-write ratio).
- `cacheLife` profile selection as a UX question — the built-in profiles (`'seconds'`, `'minutes'`, `'hours'`, `'days'`, `'weeks'`, `'max'`) and the rule that the three numbers describe how long stale data is acceptable.
- The four tag shapes — entity, record, org-scoped, user-scoped — and the convention that cached reads attach the union so any applicable writer can invalidate.
- The `tags.ts` helper as the structural enforcement that read-side and write-side tag strings line up — string only exists in one file.
- The "no closures over request data" rule as it applies to tags — org-scoping is an argument to the cached function, not a read from the session.
- The four-call invalidation surface — `updateTag` (Server-Action-only, read-your-writes, tag), `revalidateTag` (eventual, tag, callable anywhere), `revalidatePath` (eventual, path-or-pattern), `router.refresh` (client-side, route re-render).
- The user-expectation question as the primary driver — same mutation, different call depending on whether the triggering user is watching the redirect.
- The "invalidate after commit, then redirect" sequence inside Server Actions, and why the inverse order ships stale data.
- Webhook and background-job mutations use `revalidateTag` — `updateTag` is rejected outside Server Actions, and stale-while-revalidate is the correct UX when no specific user is watching.

> **Note (`revalidateTag` in Next.js 16):** the single-argument form `revalidateTag(tag)` is deprecated — every call must pass a `cacheLife` profile as the second argument (`'max'` is the senior default), e.g. `revalidateTag(tag, 'max')`.
