# Chapter 049 — Authoring templates

## Chapter framing

Chapter 048 landed the sending infrastructure — Resend on a verified domain, SPF/DKIM/DMARC published, the `email_suppressions` read at the wrapper, the first send call that arrives in the inbox. That call passed a `<WelcomeEmail name="Ada" />` placeholder as the `react` prop. Chapter 049 is where that placeholder turns into real templates the team will ship: the React Email component primitives that compile to the inline-styled, table-based HTML mailbox providers actually render; the local preview dev server and the tight iteration loop that replaces the "send-from-staging and refresh Gmail" anti-pattern; and the plain-text fallback, accessibility, and dark-mode discipline that makes the message readable in every client, screen reader, and color scheme. The student leaves with a typed `emails/` directory of reusable components, a working preview server, and the cross-client posture every transactional template inherits.

The threads that must run through every lesson. React Email is React for the email DOM — JSX is the form, the components compile to the constrained HTML/CSS subset the worst mailbox client (Gmail iOS, Outlook Windows) will render; the Tailwind component is the styling surface (mirroring the Unit 3 Tailwind muscle memory) but its output is inline styles, not classes, and the supported utility set is narrower than the web build; the preview dev server (`pnpm email dev`) is the inner loop — every template gets eyeballed before any send, the "send-from-staging" loop is the failure mode; emails are accessibility-first by default because the audience includes screen readers and low-vision users at far higher rates than typical web pages; plain-text fallback is generated from the same JSX via the `render({ plainText: true })` utility, not hand-maintained; dark mode is opt-in through `@media (prefers-color-scheme: dark)` plus the `meta name="color-scheme"` head tag, not the `dark:` Tailwind variant alone, because most clients still don't honor it; templates live in `emails/*.tsx` colocated with the app and are passed to `sendEmail`'s `react` prop from lesson 1 of chapter 048. The chapter ships three teaching lessons plus a quiz.

## Lesson 1 — JSX for the email DOM

Teaches the React Email component vocabulary, the Tailwind styling wrapper, the JSX-to-inline-table render pipeline, and the `emails/*.tsx` props-and-`PreviewProps` file convention.

Topics to cover:

- **The senior question.** The send call in lesson 1 of chapter 048 accepted a React component as the `react` prop. What HTML actually goes on the wire when that component renders, why can't the team just write the same JSX they ship to the browser, and what's the component API that makes the constrained email-HTML subset feel like normal React? The lesson installs the React Email component vocabulary, the Tailwind component for styling, the rendering pipeline that turns JSX into inline-styled tables, and the file convention for the `emails/` directory.
- **Why email HTML is its own subset.** The mailbox client landscape — Gmail web, Gmail iOS/Android, Apple Mail macOS/iOS, Outlook Windows (Word rendering engine through 2024 and now Edge-based, but legacy installs persist), Outlook Mac, Yahoo, Proton, Spark, plus enterprise webmail — has no shared modern-CSS baseline. Flexbox, Grid, custom properties, `@layer`, container queries, viewport units — all unreliable. The interoperable baseline is roughly HTML 4 plus table layouts plus inline styles plus a small subset of media queries. React Email's job is to let the team write 2026 React while the renderer emits the 2004-shaped HTML the worst client still parses.
- **The render pipeline — JSX in, dual-format string out.** A React Email template is a server-rendered React component. `render(<WelcomeEmail … />)` (from `@react-email/render`, re-exported by `react-email`) returns `{ html, text }`: the HTML string for the multipart `text/html` part, the plain-text string for the `text/plain` part (lesson 3 of chapter 049). The SDK's `react` prop on `resend.emails.send` runs the same render internally and sets both MIME parts on the outgoing message. No CSS file is loaded; styles are inlined at render time.
- **The component primitives — the read-once vocabulary.** The React Email package ships a small set of components that map to the email-safe HTML the renderer emits. The senior reach is to use these instead of raw `<div>`/`<table>`/`<style>`:
  - **`<Html>`, `<Head>`, `<Body>`** — the document shell; `<Head>` is where `<Title>`, `<Font>`, and the dark-mode meta tag (lesson 3 of chapter 049) live.
  - **`<Container>`** — the centered, max-width wrapper (default 600 px) that frames the message column. The email equivalent of a page's main content area.
  - **`<Section>`, `<Row>`, `<Column>`** — the table-row-and-column layout primitives. `<Section>` is a vertical band; `<Row>` plus `<Column>` is the horizontal split (two-up product cards, a logo-and-CTA header).
  - **`<Heading>`, `<Text>`, `<Link>`** — the text primitives. `<Heading>` accepts `as="h1"`/`"h2"`/etc.; `<Text>` is the paragraph default with sensible line-height and margin.
  - **`<Button>`** — the rendered-as-table-cell call-to-action. The component takes `href` and renders a bulletproof button (the Outlook VML fallback is generated automatically). Pick this over a styled `<a>` for CTAs.
  - **`<Img>`** — the image with required `width`, `height`, `alt`, `src`. The width/height attributes are not optional — Outlook ignores CSS dimensions and renders at the natural size without them.
  - **`<Hr>`, `<Preview>`, `<CodeBlock>`, `<CodeInline>`** — `<Preview>` deserves emphasis: a hidden element whose text becomes the *preheader* (the gray preview text next to the subject in the inbox list). Every transactional template sets a `<Preview>` — leaving it blank lets the client pull from the first visible body line, which is usually wrong.
- **The Tailwind component — styling without inventing a third syntax.** Wrapping the body in `<Tailwind>` (from `@react-email/components`) lets the template use the familiar utility classes from Unit 3 (`text-lg`, `font-semibold`, `bg-zinc-50`, `mx-auto`, `max-w-[600px]`). At render time the wrapper compiles the used classes into inline styles per element and drops the unrecognized ones with a console warning. The senior anchor: same utility vocabulary as the web app, but the supported subset is narrower (no `flex`, no `grid`, no arbitrary-property `dark:` variants without the head-tag plumbing from lesson 3 of chapter 049) and arbitrary values (`max-w-[37.5rem]`) are first-class. A `tailwind.config` can be passed for theme tokens; for a project-shared brand palette, import the same `theme` object the web app reads from.
- **Bridging the Tailwind v4 `@theme` block to React Email's v3-shaped config.** The web app's tokens live in a v4 `@theme` block in CSS (Chapter 018), but the `<Tailwind>` component still consumes a v3-style JS config object — and many of the tokens are OKLCH, which a handful of mailbox clients don't parse. The reach is a small shared `emails/emailTailwindConfig.ts` that mirrors the brand tokens as a v3 `{ theme: { extend: { colors: { … } } } }` object with hex/RGB fallbacks for each OKLCH value, then `<Tailwind config={emailTailwindConfig}>` in every template. The student needs the shape — one TS module, one default export, hex/RGB values keyed by the same names the web app uses — not the full extraction algorithm; the file is maintained by hand as the brand palette evolves.
- **The file convention — `emails/*.tsx` colocated with the app.** React Email scans an `emails/` directory at the repo root by default. Each `*.tsx` file is one template, default-exported. The convention from the React Email CLI (lesson 2 of chapter 049) and the file convention work together — moving a template breaks the preview path, so name and place templates with deliberate intent. Subdirectories are supported in React Email 6 for grouping (`emails/auth/`, `emails/billing/`).
- **Props are the template's contract.** The default export is a React component with typed props — `interface WelcomeEmailProps { firstName: string; verifyUrl: string }` — and every dynamic value reads from props. The senior reflex: never read environment variables, the current user, or database rows from inside the template; the caller (a Server Action) computes the values and passes them as props. This keeps templates pure renderers, makes the preview server render them with mock data trivially (lesson 2 of chapter 049's `WelcomeEmail.PreviewProps`), and means the same template ships unchanged from a preview, a unit-test render, and a real send.
- **The `PreviewProps` named export — mock data lives with the template.** A template exports `WelcomeEmail.PreviewProps = { firstName: 'Ada', verifyUrl: 'https://yourapp.com/verify/abc' }` (or the equivalent `export const PreviewProps`). The preview server picks these up and renders the template with the mock values; production sends supply real values. This is the bridge between "designs in isolation" and "renders with real data" that obviates a separate Storybook for email.
- **Composing shared chunks — `EmailLayout` and the brand surface.** Most transactional templates share a header (logo + product name), a body container with consistent padding, and a footer (legal address per CAN-SPAM, support link, the year). The senior reach is one `EmailLayout` component the templates compose: `<EmailLayout><Section>…body…</Section></EmailLayout>`. The brand tokens (colors, typography, logo URL) live in a single shared component, not duplicated per template.
- **Images — host them, link them, don't embed them.** `<Img>` takes a `src` that points to a publicly accessible HTTPS URL — typically a CDN, the project's R2 bucket (Unit 13b), or the marketing site's `/og/` directory. Inline base64 images bloat the message past Gmail's 102 KB clipping threshold (the truncation that hides the unsubscribe link) and are stripped by some clients. The senior reach is one CDN-hosted logo URL referenced across templates; the explicit `width`/`height` attributes are non-negotiable for Outlook. The 102 KB clipping budget is named here as the real constraint that drives the lean-template discipline.
- **Watch-outs.** Importing components from React's web ecosystem (`<div className="flex">`, shadcn primitives) into a template — the renderer doesn't understand the resulting CSS and Gmail will not parse what does compile, the template silently breaks; using `<Tailwind>` utilities outside the supported subset — `flex` and `grid` look like they work in the preview because Chrome renders them, but Gmail collapses to default block flow; forgetting `<Preview>` — the inbox preheader pulls the first visible text, which is almost always a header that duplicates the subject; missing `width`/`height` on `<Img>` — Outlook renders at the natural pixel size, blowing past the 600 px column; reaching for inline base64 images to dodge the CDN — clipping at 102 KB hides the unsubscribe link and the legal address, hurting both deliverability and compliance; relying on `<Tailwind>` colors without naming hex fallbacks in the inline style — some clients drop unknown class-derived properties and the brand color disappears; rendering the template with `await render(...)` at the call site when the SDK accepts the React node directly via the `react` prop — extra work that defeats the SDK's built-in dual-MIME generation.

What this lesson does not cover:

- The local preview dev server and the iteration loop (lesson 2 of chapter 049).
- Plain-text generation and the accessibility/dark-mode posture (lesson 3 of chapter 049).
- The send call and `lib/email.ts` wrapper (lesson 1 of chapter 048).
- The Tailwind v4 CSS-first configuration covered in Chapter 018 — referenced, not re-taught.
- Marketing email layouts with multi-column hero sections — out of scope, the project sends only transactional mail.
- MJML and `mjml-react` — named once as the legacy alternative when extreme Outlook compatibility is required, dropped.

Estimated student time: 50 to 60 minutes. Mechanics-archetype lesson with a worked welcome-email template at the end as the running artifact.

## Lesson 2 — The preview server loop

Teaches the `pnpm email dev` server, the viewport and dark-mode toggles, the HTML and plain-text tabs, and the test-send-to-inbox cross-client verification path.

Topics to cover:

- **The senior question.** A template renders correctly in the browser when previewed as a React component, but the inbox is the only place that matters. What's the iteration loop that catches "the heading wraps awkwardly at 600 px," "the CTA button is gray in dark mode," and "the preheader is empty" *before* anyone sends from staging? The lesson installs the React Email preview server, the file-watching dev loop, the device and dark-mode preview switches, and the test-send-to-inbox path the team uses for final cross-client verification.
- **`pnpm email dev` — the inner loop.** Adding `"email": "email"` to the `package.json` `scripts` block (the `react-email` CLI binary is shipped by the `react-email` package) and running `pnpm email dev` boots a Next.js-based local server on `http://localhost:3000` that watches `emails/*.tsx` and hot-reloads on save. The left panel lists every template; the main panel renders the selected one with its `PreviewProps`; the file-watcher means the loop is "save, alt-tab, eyeball, fix." The senior reflex is to keep this running in a second terminal during any template work.
- **Editing props live in the preview UI.** The preview server inspects the template's `PreviewProps` and exposes them as editable fields in the right-side panel — change `firstName` from `Ada` to a 40-character string to see how the heading wraps, swap `verifyUrl` for a 200-character URL to see button overflow. The mock data is the template's contract; editing it in the preview is faster than re-running the app's Server Action with different inputs.
- **The viewport switcher and the mobile preview.** React Email 6 ships a viewport toggle (desktop / mobile) in the preview chrome. The mobile view is non-negotiable for the iteration loop — 70 percent of consumer mail is opened on mobile per the 2024–2025 Litmus surveys, and a template that wraps cleanly at 600 px desktop may stack illegibly at 375 px. The reach is to always check the mobile view before declaring a template done.
- **The dark-mode toggle.** The preview chrome also ships a light/dark toggle that renders the template under `prefers-color-scheme: dark`. The senior anchor: this is the preview substitute for the cross-client check, but it's not authoritative — Apple Mail and Gmail Android each apply their own color-inversion heuristics that the preview can't reproduce. Use the toggle to catch obviously broken cases (a white logo on a near-white background after the client inverts); use a final test-send to Apple Mail and Gmail to confirm.
- **The HTML and source tabs.** The preview UI exposes tabs for "Rendered," "HTML," and "Plain text." The HTML tab shows the literal output that goes on the wire — useful for verifying that the `<Tailwind>` classes compiled to the expected inline styles, that the `<Preview>` hidden text is in the document, and that the `<head>` carries the meta tags. The plain-text tab shows what lesson 3 of chapter 049's render-as-text path produces — the senior reach is to read this tab and confirm the text version is coherent, not just "the HTML stripped of tags."
- **Sending a preview to a real inbox.** The preview UI ships a "Send test" button that fires through Resend (using the same `RESEND_API_KEY` the app uses) to a recipient address the developer types in. This is the cross-client verification path — render the template, test-send to a personal Gmail, an iCloud, an Outlook.com, and a Proton account, eyeball each in the actual client. The senior anchor: the preview server is the iteration loop, the test-send is the verification gate before the template is wired into a Server Action and the action is shipped.
- **CI and the rendered-output check (optional, named once).** Mature teams render each template to HTML in CI and snapshot-test the output (`renderToStaticMarkup` plus `expect(html).toMatchSnapshot()`) to catch unintended template changes during dependency upgrades. Named here as the senior reach for a team shipping enough templates that a React Email minor-version bump risk justifies the snapshot maintenance; not part of the project chapter.
- **The "where does the dev server clash with the Next.js dev server" question.** Both `pnpm dev` (Next.js) and `pnpm email dev` (React Email) want port 3000. The reach is to override one — `pnpm email dev --port 3001` is the standard — and document the port choice in the project README. Both servers can run side-by-side; the email server is independent of the Next.js app.
- **The iteration loop, end to end.** The lesson walks the canonical loop once:
  1. Write the template in `emails/welcome.tsx`.
  2. Define `PreviewProps` with realistic mock data.
  3. `pnpm email dev` running in a side terminal.
  4. Save, eyeball at desktop, toggle to mobile, toggle to dark.
  5. Read the plain-text tab, confirm it's coherent.
  6. Test-send to two personal inboxes (one Gmail, one Apple Mail) for cross-client confidence.
  7. Wire into the Server Action via `sendEmail({ react: <WelcomeEmail … /> })`.
  The loop is the lesson. Each step has a failure mode the previous step doesn't catch.
- **Watch-outs.** Iterating on the template while the dev server isn't running and discovering at send-time that the CTA wrapped — the loop exists for a reason, run the dev server; treating the preview's dark-mode toggle as authoritative — it approximates `prefers-color-scheme: dark` but doesn't reproduce Gmail Android's blanket inversion, the final test-send is non-negotiable; sending the test to an inbox the team doesn't actively check — the test-send only works if the destination is opened in the real client; running the preview with stale `PreviewProps` that don't match the production payload — `PreviewProps` is the template's mock-data contract, keep it honest; checking only the rendered tab and never the plain-text tab — the text fallback rendering is the screen-reader and clipped-Gmail audience's surface, eyeball it; assuming the preview server uses the same Tailwind config the production app uses — it reads its own config, point both at the shared `theme` import to keep brand tokens in sync.

What this lesson does not cover:

- The component vocabulary and Tailwind component (lesson 1 of chapter 049).
- Plain-text rendering as a feature (lesson 3 of chapter 049 covers the `render({ plainText: true })` API and the accessibility implications; lesson 2 of chapter 049 only names the preview tab).
- Cross-client testing services (Litmus, Email on Acid) for full screenshot matrices — named once as the senior reach for a marketing-heavy product, out of scope for transactional templates the project ships.
- The Resend dashboard's "Preview" feature on the sent-email view — adjacent capability, not the iteration loop.

Estimated student time: 30 to 40 minutes. Setup-and-mechanics lesson with the test-send as the verify step.

## Lesson 3 — Readable in every client

Teaches the auto-generated plain-text fallback via `render({ plainText: true })`, the email accessibility checklist (lang, headings, link text, alt, contrast, font size, touch targets), and the three-tier dark-mode posture with its `color-scheme` head plumbing.

Topics to cover:

- **The senior question.** The HTML version renders. Now: the screen-reader user navigating the message with VoiceOver, the user whose mail client strips HTML for security, the Gmail Android user whose client inverts colors at the OS level, the user reading at 200 percent zoom on a 12-inch laptop, the corporate gateway that re-parses HTML and breaks layouts — all of them are part of the audience and none of them see the template as the designer eyeballed it in the preview. What posture catches every one of these, and where in the template does each correctness check live? The lesson installs the plain-text fallback discipline, the email-specific accessibility checklist (which is shorter than the web list because the substrate is simpler but with stricter constraints on font sizes, contrast, and link affordances), and the dark-mode posture that ships *with* the template, not after a complaint.
- **The plain-text fallback — what it is, why it's non-negotiable.** Every legitimate transactional email goes on the wire as a `multipart/alternative` MIME message with two parts: `text/plain` and `text/html`. The receiving client picks whichever it can or wants to render. The plain-text part exists for:
  - Screen readers that don't render HTML (`Mail Drop` on iOS in low-bandwidth mode, accessibility-mode mail clients).
  - Clients that strip HTML for security (corporate-policy plain-text-only mailboxes, security-conscious users).
  - Spam filters and content scanners that read the text part for keyword matching — a missing or empty text part is itself a small spam signal.
  - The clipping fallback when Gmail truncates at 102 KB (the "[Message clipped] View entire message" link reveals the full HTML, but search engines and indexers see the text part).
  The senior reflex: never ship a transactional send without both parts, and never hand-maintain the text part — generate it from the same JSX.
- **`render(<Template />, { plainText: true })` — the generator.** React Email's `render` utility takes an options bag; passing `{ plainText: true }` returns a Markdown-flavored plain-text string with links inlined as `Text [https://url]`, headings prefixed (`# Heading`), and structural whitespace preserved. The Resend SDK's `react` prop runs both renders internally and assembles the multipart message. The application code never calls `render` directly for the production send; the preview server's "Plain text" tab is the inspection point.
- **The text-version coherence check.** The senior reach is to read the generated text version in the preview server's plain-text tab and confirm it reads as a standalone message:
  - The subject of the message and the call-to-action are unambiguous from the text alone.
  - Links are labeled — a bare `https://yourapp.com/verify/abc-123` is correct but harder than `Verify your email: https://yourapp.com/verify/abc-123`.
  - Decorative images don't pollute the text (their `alt=""` empty alt drops them from the text version; missing or non-empty alt on decorative images leaves stray "image" in the text).
  - Section ordering matches the HTML — a footer that floats to the top in the text version usually means a `<Container>` nesting bug.
  The fix is in the template's JSX (rewording, adjusting `alt` text, reordering sections), not in a separate text file.
- **The accessibility checklist for transactional mail.** Email accessibility is a tighter subset of web accessibility because the substrate (table layouts, inline styles, no JavaScript) blocks most ARIA patterns. The checklist that pays for itself on every template:
  - **Set the language.** `<Html lang="en">` (or the recipient's locale) — screen readers use it to pick pronunciation.
  - **Set the document title.** `<Title>Verify your email</Title>` inside `<Head>` — assistive tech announces it; some clients show it in the open-in-browser view.
  - **Logical heading order.** One `<Heading as="h1">` per email (the message's purpose), `<Heading as="h2">` for any subsections. The h1-as-banner-logo anti-pattern from web design is doubly bad in email because the h1 is the screen reader's first navigation target.
  - **Semantic links.** `<Link href="…">Verify your email</Link>` with link text that describes the destination, not "click here" / "this link." Screen readers read out a list of links by their text — every link must stand alone.
  - **Image alt text.** Decorative images get `alt=""` (empty string, not omitted); informational images get a one-sentence description; the logo gets the brand name. Missing `alt` defaults to the filename in many clients, which is an accessibility *and* deliverability footgun.
  - **Color contrast.** WCAG AA — 4.5:1 for body text, 3:1 for headings and large text. Email's default `<Text>` on a near-white background is usually safe; brand colors on CTAs are the common failure point.
  - **Minimum font size.** 14 px body, 16 px on mobile. iOS Mail bumps anything below 13 px automatically; not relying on the bump is the senior call.
  - **Touch target size.** CTAs are at least 44×44 px clickable area (iOS HIG) so a thumb-tap doesn't miss. The `<Button>` component's default padding clears this; custom-styled links to act as CTAs often don't.
  - **Avoid carrying meaning in color alone.** A red "Verify" button conveys urgency to sighted users; the screen-reader user gets "Verify your email." The text must carry the meaning; color reinforces.
- **Dark mode — the three-tier posture.** Mailbox clients do one of three things with dark mode:
  - **No transformation.** The HTML renders as authored. Most desktop clients in dark theme (Apple Mail macOS dark, recent Outlook). The template's own dark-mode rules apply if present.
  - **Partial inversion.** The client detects light backgrounds and inverts them while preserving dark elements (Gmail iOS, Outlook iOS). Brand colors usually survive; near-white backgrounds become near-black.
  - **Full inversion.** The client inverts everything (Gmail Android in some configurations, some Outlook installs). Logos become photo negatives, brand-color CTAs get hue-shifted.
  The senior posture is to assume some clients will invert and design defensively, and to opt into the template's own dark-mode styles for the clients that respect them.
- **The dark-mode plumbing.** Two pieces of head plumbing flag the template as dark-mode-aware (without them, Apple Mail won't apply the template's dark styles even when the user is in dark mode):
  - `<meta name="color-scheme" content="light dark" />`
  - `<meta name="supported-color-schemes" content="light dark" />`
  Plus an inline `<style>` tag in `<Head>` with `:root { color-scheme: light dark; }`.
  With those in place, two styling approaches work:
  - **`@media (prefers-color-scheme: dark)` in an inline `<style>` in `<Head>`.** Selectors that target `[data-dark]` or specific class names, swapping colors. The most reliable approach across clients that respect the preference.
  - **Tailwind's `dark:` variant via React Email's Tailwind component.** Supported but narrower — works in Apple Mail and recent Outlook, ignored by clients that don't honor the preference. Useful for simple swaps (background color, text color); not a substitute for the inline media-query approach for complex inversions.
  The senior reach for a transactional template: the head plumbing plus one `@media (prefers-color-scheme: dark)` block in the document `<style>` that handles the brand color swap and the logo swap. The full dark-mode-everything pass is a marketing-template concern; transactional templates get the minimum.
- **The logo-on-dark problem and the `<picture>` swap.** A dark-text logo on a near-white background becomes invisible when the client inverts to near-black. The senior reach is a `<picture>` element with two `<source>` entries — one with `media="(prefers-color-scheme: dark)"` pointing at the light-on-dark logo, the fallback `<img>` pointing at the dark-on-light logo. Supported by Apple Mail and most modern webmail; ignored by clients that don't honor the preference (where the original logo still ships, so the worst case is "logo barely visible," not "logo missing"). Named here, not built — the project ships with a single brand-neutral logo.
- **Right-to-left rendering and `dir="auto"`.** For an international audience the `<Html dir="auto">` plus the logical-property utilities from Chapter 020 (`ps-*`/`pe-*` rather than `pl-*`/`pr-*` — the Tailwind component supports them) mean the template flips cleanly for Arabic and Hebrew recipients without a per-locale duplicate. Named here as the carry-over from Unit 17 (i18n) for templates that need it; the project's first welcome email is English-only.
- **Watch-outs.** Hand-maintaining a `text` field alongside the React template — it drifts within two releases, the auto-generated path through `render({ plainText: true })` is the only sustainable shape; relying on the Tailwind `dark:` variant alone without the head meta tags — Apple Mail ignores the `dark:` styles entirely without them; using `alt=""` on an informational image to "make the text version cleaner" — defeats screen readers, the cleaner-text-version fix is to reword the HTML, not strip the alt; brand colors at 3:1 contrast for body text — fails WCAG AA, the user with low vision misses the message; CTAs styled as plain links with small font — the touch target fails, the user mis-taps on mobile; testing dark mode only in the preview — Gmail Android's blanket inversion can only be caught by a real test-send to an Android device or screenshot service; relying on `prefers-color-scheme: dark` for accessibility (light-on-dark contrast) — most clients don't honor it, the *default* template must already meet WCAG contrast in its light shape; missing `lang` attribute on `<Html>` — screen readers fall back to system locale and mispronounce content; using `<table role="presentation">` manually because of an old accessibility playbook — React Email's components already set the right roles, hand-rolling them is unnecessary.

What this lesson does not cover:

- Image hosting, the 102 KB clipping budget, and CDN choices (lesson 1 of chapter 049).
- The preview server's plain-text tab as a UI feature (lesson 2 of chapter 049 named it; this lesson teaches what to *read for*).
- Marketing-email accessibility (one-click unsubscribe, footer with physical address, preferences center) — out of scope, transactional only.
- Full WCAG 2.1 audit methodology — referenced as the standard, not re-taught.
- Localized templates with `next-intl` ICU plurals — that's Chapter 085 territory; this lesson names `dir="auto"` and stops.
- BIMI and the inbox-icon brand mark — out of scope (named once in lesson 2 of chapter 048 and dropped).

Estimated student time: 45 to 55 minutes. Pattern-and-checklist lesson — the discipline is the lesson, the syntax is small.

## Lesson 4 — Quizz

Top 10 topics to quiz:

- Why email HTML is its own constrained subset — the worst-client baseline (table layouts, inline styles, no flex/grid), and what React Email's renderer does on top of JSX to produce it.
- The component vocabulary — when to reach for `<Container>`, `<Section>`, `<Row>` plus `<Column>`, `<Button>`, `<Img>`, and the role of `<Preview>` for the inbox preheader.
- The `<Tailwind>` component — what utility subset it supports, how arbitrary values and theme tokens work, and the failure modes when an unsupported utility (e.g., `flex`) is used.
- `PreviewProps` as the template's mock-data contract — why props-only templates (no env reads, no DB reads inside the component) make the preview, the test, and the production send render identically.
- The 102 KB Gmail clipping threshold — why inline base64 images are the wrong reach, and what gets hidden when a message clips (unsubscribe link, legal address).
- The `pnpm email dev` iteration loop — the file watcher, the viewport toggle, the dark-mode toggle, the HTML and plain-text tabs, and the canonical save-eyeball-fix loop.
- The test-send as the verification gate — why the preview's dark-mode toggle approximates but doesn't replace a real cross-client test (Gmail Android's blanket inversion, Outlook VML rendering quirks).
