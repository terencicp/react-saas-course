# Chapter 057 — Roles, action wrappers, and the audit trail

## Chapter framing

Chapter 056 installed the org skeleton — the `organization` and `member` tables, the `activeOrganizationId` slot on the session, the `tenantDb(orgId)` helper that pins every read to a tenant. Chapter 057 lands the other half of the authorization story: *who, within the org, is allowed to do what*, and the audit trail that records every privileged action so the system can answer "who removed this member" three months later. The student arrives with `requireOrgUser()` (returning `{ user, orgId, role }`), the Server Action shape from Unit 6 (`safeParse` → mutate → `Result` return), and the route-handler shape from chapter 046. They leave with a three-role RBAC model (`owner` / `admin` / `member`), an `authedAction(role, schema, fn)` wrapper that makes "I forgot the role check" structurally hard at the Server Action boundary, the same discipline ported to `route.ts` for non-React callers, the member-management surface (list, change role, remove, leave-org, transfer ownership) built against those primitives, and an append-only `audit_logs` table that records every privileged write — RLS-protected per lesson 4 of chapter 056, written through `withTenant`.

Threads that run through every lesson. RBAC in year-1 SaaS is *three roles, not a permission matrix* — owner / admin / member is the default; granular permissions are a conditional reach when the product surfaces "approver" and "editor" as distinct seats. The structural defense against missed role checks is the *wrapper at the boundary*, not the discipline inside the action body — `authedAction(role, schema, fn)` makes the role check the only call shape that compiles for a privileged action. The audit log is *append-only by contract and by table grants* — no UPDATE, no DELETE, ever, and the table writes happen inside the same Server Action transaction that performs the privileged write so an audit row exists if and only if the work landed. Every member-management flow surfaces a senior trap: removing the last owner, demoting yourself, transferring ownership to a non-member, the in-flight Server Action that started under the old role. The carve-out from Principle #5 ("consume libraries directly") is named once at lesson 2 of chapter 057 and applied at lesson 3 of chapter 057 — `authedAction` is the small typed wrapper that pins authz onto every action; `tenantDb` (lesson 2 of chapter 056) is its data-layer twin. Five teaching lessons plus a chapter quiz.

## Lesson 1 — Owner, admin, member

Teaches the year-1 three-role RBAC default, the authority gradient between roles, the single-owner invariant, the `roleAtLeast` helper, and extending `requireOrgUser()` to return the current member's role.

Topics to cover:

- **The senior question.** Three teammates inside one org. One signs the contract, one administers users and settings, one does the day-to-day work. What roles does the year-1 SaaS ship with, how are they stored, how do new members get the right role on invite, and when does the product need to grow past this shape into a real permissions matrix? Land the three-role default and the authority gradient between them.
- **The three roles — `owner`, `admin`, `member`.** Better Auth's organization plugin owns the role column on `member.role` (string, default `'member'`). The three roles are the year-1 default: `owner` controls billing and can transfer ownership; `admin` manages members, settings, and content; `member` operates on content but doesn't see member management or settings. The senior reflex: this gradient covers the 80% case for early SaaS; product teams reach for fine-grained permissions only when customers ask, and "ask" is the trigger (named, not built).
- **The authority gradient — what each role can do.** `owner`: everything `admin` can do plus billing, plan changes, transferring ownership, deleting the org. `admin`: invite, remove, change roles up to `admin`, edit org settings, see audit log. `member`: read/write content, change own profile, leave org. The senior reflex: the role-to-capability map is *one document*, not scattered checks; one matrix in `src/lib/auth/roles.ts` everyone reads when adding a new privileged action.
- **The single-owner invariant.** Every org has at least one owner at all times. The invariant is enforced in three places — at member removal (can't remove the last owner), at role demotion (can't demote yourself if you're the last owner), at leave-org (last owner can't leave; must transfer first). The check belongs in the helper that performs the mutation, not in the UI; the UI's job is to surface the error message. Senior reflex: the invariant is a *guard rail*, not a convention — one place to put the check, one query (`SELECT count(*) FROM member WHERE organization_id = $1 AND role = 'owner'`), one error code (`'last-owner'`).
- **The role-comparison helper — `roleAtLeast(role, required)`.** Roles form a total order: `member < admin < owner`. The helper takes the user's role and a required minimum and returns a boolean. The senior reflex: codify the order in one place (`const ROLE_RANK = { member: 0, admin: 1, owner: 2 } as const`), don't write ad-hoc `role === 'admin' || role === 'owner'` checks scattered across the codebase. The TypeScript shape is `type Role = 'owner' | 'admin' | 'member'`; the union is exported from `src/lib/auth/roles.ts` and consumed everywhere a role is named.
- **The role at request time — extending `requireOrgUser()`.** lesson 1 of chapter 056 stubbed `requireOrgUser()` returning `{ user, orgId }`. This lesson extends it to `{ user, orgId, role }` by reading `member.role` for the active org. The role is fetched once per request inside the helper, not on every check — every privileged action reads the role from the helper's return value, never re-queries. The senior reflex: one read per request, one source of truth for the role inside that request.
- **Setting the role on invitation acceptance.** When a member accepts an invitation (Chapter 058), the role recorded on `invitation.role` becomes the role on the new `member` row. The senior reflex: the inviter chose the role at invite time; acceptance is a no-decision step. This is named here so Chapter 058 has the contract; the implementation is lesson 4 of chapter 058.
- **The first-member-is-owner rule.** The user who creates the org is the owner — the `member` row written by `authClient.organization.create` has `role: 'owner'`. Better Auth's plugin handles this; the student verifies the row shape after creating an org. No code to write; reflex to know.
- **What this is not — fine-grained permissions, ABAC, custom roles.** Some shops reach for "every action has a permission, every role is a bag of permissions, customers can author their own roles" in year-1. The senior call: don't. The three-role default ships the product; the permissions layer is the conditional reach when a paying customer says "we need 'approver' that can do X but not Y." Better Auth's plugin supports `additionalFields` on `member` for custom roles and a permissions API (named once for the future); building it now is premature.
- **The Better Auth plugin call — adding roles to `member`.** The plugin's default role column is already in place; configuring `organization({ allowedRoles: ['owner', 'admin', 'member'] })` (or the equivalent option name — verify on the current version) constrains writes. The senior reach: keep the role union small and known; don't accept arbitrary strings on the column.
- **The "I'm an admin but I want to see what a member sees" trap.** Some products surface "view as member" for debugging. The senior reflex: don't fake the role in the session — build a `viewAs` query param the privileged UI honors for rendering, but every server-side authz check still reads the *real* role from `requireOrgUser`. View-as is a UI mode, not an authz mode.
- **The route-protection seam — gating pages by role.** Server Components inside `(authed)` route groups call `requireOrgUser()`; pages restricted to admins call `requireAdmin()` (a thin wrapper that calls `requireOrgUser()` and `redirect('/')` if `roleAtLeast(role, 'admin')` is false). The student writes `requireAdmin` and `requireOwner` in `src/lib/auth/guards.ts`. The next lesson generalizes this to the action boundary.
- **Watch-outs.** Storing roles in the session payload and trusting them across role changes — the session payload is set at sign-in; if an admin demotes another user, that user's existing session carries the stale role until the next session creation or until a forced refresh; the senior reflex is to read role from the DB (or a short-cookie-cached value, lesson 3 of chapter 052) inside `requireOrgUser` so role changes propagate within seconds; using boolean fields like `isAdmin: true` on `user` instead of `role` on `member` — couples the role to the user across orgs (the same user is admin in Acme, member in Beta; one row per `(orgId, userId)` is the only shape); ad-hoc `if (role === 'admin')` checks scattered through actions — review-fragile, the wrapper in lesson 2 of chapter 057 is the structural fix; allowing arbitrary role strings on the column — typos in the seed file silently create unrecognized roles that fail every comparison; assuming "owner can do everything admin can" without coding it — `roleAtLeast` is the helper that proves it; surfacing the "remove member" button to a member because the page didn't read role — gate the UI on role, gate the action on role, gate the DB write through `authedAction` (lesson 2 of chapter 057) so the user can lie about all three and the action still refuses.

What this lesson does not cover:

- The `authedAction(role, schema, fn)` wrapper at the Server Action boundary (lesson 2 of chapter 057).
- The route-handler arm of the same discipline (lesson 3 of chapter 057).
- Member listing, removal, demotion, leave-org, ownership transfer (lesson 4 of chapter 057).
- The audit-log table and what gets recorded (lesson 5 of chapter 057).
- Fine-grained permissions, ABAC, or custom roles (named once; conditional reach).
- The invitation-time role choice (Chapter 058).

Estimated student time: 35 to 45 minutes. Decision-plus-mechanics; the role matrix carries the lesson, a small ER fragment of `member.role` and a `roleAtLeast` shape close it. One short table — "what owner/admin/member can do" — earns its weight.

## Lesson 2 — The authedAction wrapper

Teaches the `authedAction(role, schema, fn)` factory that lifts session, role check, and schema parse out of every Server Action body, the `ctx = { user, orgId, role, db }` payload it threads, the `Result` return contract, and the named carve-out from Principle #5 that justifies the wrapper.

Topics to cover:

- **The senior question.** Every privileged Server Action needs three things before it touches the DB — the session is valid, the user has the required role, the input parses against the schema. Forgetting any one is a real bug class: an action that mutates without checking role ships a privilege escalation; an action that mutates without parsing ships an injection vector; an action that runs without a session does whatever the caller passed. Today the action body carries the discipline — `requireOrgUser`, role check, `safeParse`, then the work. What's the wrapper that lifts all three out of the body so the action body is *just the work*, and forgetting the wrapper doesn't compile?
- **The threat — the missing role check in a fast PR.** A new feature: `deleteCustomerAction(formData)`. The author wrote `requireOrgUser` (caught the session), wrote `safeParse` (caught the input), but forgot the role check. Members can now delete customers. The bug ships because the reviewer scanned for the three lines and saw two of them; the third was supposed to be there and wasn't. The senior reflex: the defense isn't "be careful," it's a call shape where the role is a *parameter*, not a body statement.
- **The signature — `authedAction(role, schema, fn)`.** A factory that returns a Server Action. The first parameter is the minimum required role (`'admin'`, `'owner'`, `'member'`); the second is the Zod input schema; the third is the typed business function `(input, ctx) => Promise<Result<T>>` where `ctx` is `{ user, orgId, role, db }` — `db` is `tenantDb(orgId)` already bound. Typical call: `export const removeMemberAction = authedAction('admin', removeMemberSchema, async (input, ctx) => { /* just the work */ })`. The action body becomes pure business logic; everything else lives in the wrapper.
- **What the wrapper does in order.** (1) Call `requireOrgUser()` — redirects on no session or no active org. (2) Check `roleAtLeast(ctx.role, required)` — returns `{ ok: false, error: { code: 'forbidden', userMessage: 'You don't have permission to do this.' } }` on failure (no throw — the form needs to render the error). (3) Call `safeParse(Object.fromEntries(formData))` — returns `{ ok: false, error: { code: 'invalid-input', fieldErrors } }` on failure. (4) Call `fn(parsed.data, ctx)` and return its `Result`. Five seams, one wrapper, every action shares the shape.
- **The `ctx` payload — pre-built once.** `ctx.user` from the session, `ctx.orgId` from the active org, `ctx.role` from the member row, `ctx.db = tenantDb(orgId)`. The senior reflex: the wrapper builds `ctx` once and threads it; the action body never re-queries the session, never reaches for the unwrapped `db`, never re-reads the role. The composition with `tenantDb` (lesson 2 of chapter 056) is structural — the data-layer scope and the action-layer authz arrive together.
- **The `FormData` versus typed-object call shape.** Server Actions can be invoked from `<form action={...}>` (FormData) or from `useActionState` (typed object). The wrapper accepts `FormData` by default (the canonical React 19 path); the `parse` step runs `Object.fromEntries(formData)` before `safeParse`. A second variant — `authedAction.json(role, schema, fn)` — accepts the parsed object directly for the (rarer) typed-object call path. Senior reach: most actions are FormData-shaped; name the JSON variant once.
- **The Result return contract and the type inference.** The wrapper's return type is `Promise<Result<TOut>>` where `TOut` is inferred from `fn`'s return. `Result` is `{ ok: true; data: TOut } | { ok: false; error: { code: string; userMessage: string; fieldErrors?: Record<string, string[]> } }` — the same shape Unit 6 established. The senior reflex: every error path returns through `Result` (forbidden, invalid-input, business-logic failures); only framework-edge throws (`redirect`, `notFound`, an actually-unrecoverable exception) escape as exceptions.
- **The architectural carve-out — Principle #5 gets an exception, named explicitly.** The course's stance is "consume libraries directly" — don't wrap Server Actions in a tRPC-style middleware tower for its own sake. `authedAction` is the *named* exception, paired with `tenantDb` as the data-layer arm. The carve-out exists because authz at the action boundary has a real bug class (missing role check) that a structural wrapper closes; nothing else at the action boundary justifies a wrapper. Senior reflex: when a junior reaches for "let's add another middleware to authedAction," the answer is usually "no, put it in the body" — the wrapper is small on purpose.
- **The composition pattern in practice.** Inside a feature module: define the input schema with Zod (often `createInsertSchema` from drizzle-zod plus a `.refine`), define the `fn` as `async (input, ctx) => { /* work */ return ok(row) }`, export `authedAction(role, schema, fn)`. The action body is typically 5 to 20 lines — the read, the validate-business-rules, the write, the audit log (lesson 5 of chapter 057), the `revalidatePath`, the `ok(...)` return.
- **The fail-closed reflex inside the wrapper.** If `requireOrgUser` throws (no session), the wrapper lets it propagate — `redirect('/sign-in')` is the right exit. If the role check fails, return `Result.error('forbidden')` — the form should render "you don't have permission" and stay on the page, not throw the user to a 403. If parsing fails, return `Result.error('invalid-input', { fieldErrors })` — same reason. The senior anti-trap: never log "Authorization check threw, assuming allowed" — anything that looks like authz fails closed, including bugs inside the check itself (revisited in lesson 1 of chapter 080).
- **Logging and observability at the wrapper seam.** The wrapper is the right place to attach request-scoped logging context (the user ID, org ID, role, action name) and the right place to count "forbidden" denials as a metric (a spike on `forbidden` for `removeMemberAction` is either a UI bug surfacing the button to the wrong role or someone probing the API). Senior reach: the wrapper attaches the context; the metrics dashboard (Unit 19) reads it. Don't build the metrics now; name the seam.
- **The audit-log hook — foreshadowed for lesson 5 of chapter 057.** Privileged writes record an `audit_logs` row. The natural place is *inside* the action body (the body knows what entity changed, what fields, what the diff was) — not in the wrapper, because the wrapper doesn't know which entity is being mutated. Senior reflex: the wrapper carries authz; the body carries audit; both are explicit. The lesson 5 of chapter 057 lesson lands the table and the write call.
- **What the wrapper does *not* do.** It does not check entitlements (paid plan, feature flag) — those are different checks at different layers (billing in Unit 11, feature flags as a separate concern). It does not handle rate limiting — that's Chapter 074 and lives upstream. It does not handle CSRF — Next.js Server Actions ship CSRF protection by default (the framework's `Action ID` mechanism). The wrapper is precisely "session + role + schema" and nothing else.
- **Watch-outs.** Putting the role check inside the body of the `fn` instead of in the wrapper — review-fragile, defeats the purpose; using `authedAction('member', ...)` as the default and treating it as "any logged-in user" — `member` still requires an active org, which is correct, but the wrapper isn't a substitute for an *unauthed* action factory (named once for the future); accepting `role` from `formData` and trusting it in the wrapper — the role *must* come from `ctx.role` which the wrapper read from the DB; throwing inside `fn` instead of returning `Result` and breaking the typed error contract — the form's `useActionState` won't surface a typed `fieldErrors`; mixing `authedAction` and ad-hoc actions in the same feature module — pick one and grep, or the discipline drifts; nesting `authedAction` calls (calling another action from inside an action) — actions aren't middleware; if shared logic exists, extract it to a pure function the actions share; passing `ctx` to a child function without passing it through — `ctx` is action-scoped, not module-scoped; reaching for `unstable_after()` (Next.js 16) inside the wrapper to record audit-after-response and then discovering the audit row didn't land because the function errored — audit writes belong *inside* the action's transaction, not in `after`.

What this lesson does not cover:

- The route-handler arm of the same wrapper (lesson 3 of chapter 057).
- The member-management actions that consume `authedAction` (lesson 4 of chapter 057).
- The audit-log row written inside the action body (lesson 5 of chapter 057).
- Entitlements and plan-based gating (Unit 11).
- Rate limiting on action invocations (Chapter 074).
- An unauthed-action factory for the public-form path (named once; not built).

Estimated student time: 50 to 60 minutes. Pattern lesson; the wrong-then-right archetype carries it — show the missing role check inline in an action, then the wrapper signature, then the action body collapsed to just-the-work. A sequence diagram for "form submit → wrapper → requireOrgUser → role check → parse → fn → Result" earns its weight. Live coding exercise: take a missing-role-check action and refactor through `authedAction`.

## Lesson 3 — The authedRoute twin

Teaches the route-handler port of the same discipline as `authedRoute(role, schema, fn)`, the 401 / 403 / 422 / 404 status map, RFC 9457 Problem Details responses, and sharing one business function across the Server Action and route handler seams through `/lib`.

Topics to cover:

- **The senior question.** Server Actions are the React caller's seam (Unit 6). Route handlers are the non-React caller's seam (Chapter 046) — webhooks, third-party integrations, mobile clients, anything that isn't a React form. Both seams take untrusted input. Both need session + role + schema. Today the action body shape from lesson 2 of chapter 057 isn't available because the route handler signature is `(request: Request, context)` and the response shape is RFC 9457 Problem Details, not `Result`. What's the equivalent wrapper that ports the same discipline so the missing-call class stays structurally hard at *every* untrusted-input seam?
- **The shape — `authedRoute(role, schema, fn)`.** The wrapper takes the same first two arguments as `authedAction`; the third is `(input, ctx) => Promise<Response>`. The wrapper builds `ctx` the same way (`{ user, orgId, role, db }`), parses the input from the appropriate source (path params + query + body, per lesson 2 of chapter 046), runs the role check, and calls `fn`. On failure, it returns a `Response` with the matching HTTP status — 401 for unauthenticated, 403 for forbidden, 422 for unprocessable input — and an RFC 9457 Problem Details body. The senior reach: the *response shape* differs from the action's `Result` (HTTP cares about status codes; Server Actions don't), the *discipline* is identical.
- **The input source — path/query/body, not FormData.** Route handlers receive structured input from multiple places. The wrapper's `schema` is shaped per lesson 2 of chapter 046 — `{ params: ..., query: ..., body: ... }` as a top-level object, each sub-schema parsed from its source. The wrapper composes the three reads (Next.js's typed `params`, `searchParams` from the request URL, `await request.json()` or `await request.formData()` depending on the route's content type) into the typed input passed to `fn`. The senior reflex: the cheapest parses run first (params, query) so a malformed input fails before the body is read.
- **The session source at the route boundary — cookies, the same as actions.** Route handlers receive the same cookies as actions; `auth.api.getSession({ headers: request.headers })` returns the session. The wrapper uses the same `requireOrgUser()` helper from lesson 1 of chapter 057, just sourced from the request's headers. No new session machinery; the route-handler arm is shape-compatible with the action arm.
- **The status-code map — 401 vs. 403 vs. 422 vs. 404.** No session → 401 Unauthorized; session but missing role → 403 Forbidden; session and role but schema fails → 422 Unprocessable Entity; session and role and schema pass but the entity doesn't exist in this org → 404 Not Found. The senior reflex per lesson 3 of chapter 046: prefer 404 over 403 on cross-tenant leaks — "the resource doesn't exist for you" is the right user-facing posture for "you tried to access another org's row." The wrapper returns 404 when the database read inside `fn` returns nothing on a tenant-scoped query.
- **The RFC 9457 Problem Details response.** Every error response carries `application/problem+json` with `{ type, title, status, detail, instance, fieldErrors? }`. The wrapper's failure branches build this body; the `fn` returns success responses with `application/json` via `Response.json(...)`. Senior reach: define the `Problem` Zod schema once in `src/lib/http/problem.ts`, return through a helper, every route handler's error shape matches.
- **The two seams, one discipline — when each is the right call.** A new mutation is being added; the caller is a React form on the same domain → Server Action (wrap in `authedAction`). The caller is a partner's job server posting JSON, or a webhook, or the mobile app → route handler (wrap in `authedRoute`). The senior reflex: don't expose a route handler for *internal* mutations just to "have an API" — Server Actions are the right call for React callers; route handlers are the right call when the call shape genuinely needs them. The five-trigger threshold from lesson 1 of chapter 046 applies.
- **Sharing the business function across seams.** When the same mutation must be callable from both seams (the dashboard form and the partner API both create an invoice), extract the work to a pure function `createInvoice(input, ctx): Promise<Result<Invoice>>` in `src/lib/invoices/`; the Server Action wraps it via `authedAction`, the route handler wraps it via `authedRoute`. Both seams call the same function; both seams have authz at the boundary; the function itself has no notion of HTTP or FormData. This is Principle #3 (pure functions in `/lib`, side effects at named boundaries) showing up in concrete form.
- **The idempotency-key bridge — named once, fully owned by chapter 063.** Public route handlers that accept retried writes (Stripe-shaped clients sending the same POST twice on a network glitch) honor the `Idempotency-Key` header. The wrapper has a seam for this — an optional `idempotency: true` flag that hooks `Idempotency-Key` reading and the dedup table — but the full pattern lands in Chapter 063's webhook lesson. Name the seam, point forward.
- **The CORS reality.** Same-origin route handlers (the dashboard's own app calling `/api/...`) need no CORS configuration. Public-API route handlers (the partner's job server) need CORS — `Access-Control-Allow-Origin`, `Access-Control-Allow-Headers`. The wrapper doesn't handle CORS; CORS is a Next.js middleware or `next.config.ts` concern. The senior reflex: name the seam, don't bury CORS inside `authedRoute`.
- **The HTTP semantics the wrapper preserves.** GET on `/api/customers/:id` reads — no schema body, just params. POST creates. PATCH partial-updates. DELETE removes. The wrapper doesn't constrain the verb; each route file has its own export per verb (`export async function GET(...)`, `export async function POST(...)`) and each can use `authedRoute` with its own schema shape. The status-codes-by-method table from lesson 3 of chapter 046 applies: 200 on read, 201 on create, 204 on delete, 200 on patch, 409 on conflict.
- **The streaming response edge case.** Some route handlers stream — large CSV exports, server-sent events for live data. The wrapper still runs authz at the entry; the `fn` returns a `Response` whose body is a `ReadableStream`. Senior reflex: the authz check happens before the first byte goes out; streaming responses don't escape the discipline.
- **Watch-outs.** Forgetting that route handlers are dynamic by default in Next.js 16 — the route runs per request, the session is fresh, but the `force-dynamic` decision should be explicit per lesson 4 of chapter 032; running the role check inside the route body instead of the wrapper — same review-fragile bug as lesson 2 of chapter 057, ported to a new seam; trusting `Authorization: Bearer <jwt>` headers without designing the verification — the session-cookie path is the default; bearer tokens are a separate identity model not built in this chapter; returning `Result` from a route handler — wrong shape, the caller expects HTTP status + Problem Details; mixing `authedAction` and `authedRoute` discipline (calling the action from inside the route handler) — the action runs in a different context, prefer the shared `/lib` function pattern; 200 with `{ ok: false, error: ... }` from a route handler — clients expect HTTP semantics, this breaks every standard library client; throwing redirects from a route handler — `redirect()` only works in React Server Components and Server Actions; the wrapper must `return Response.redirect(url, 303)` if a redirect is the intent (rare for API routes); silent 200 on cross-tenant read because `tenantDb` returned no rows and the handler rendered an empty list — the senior shape is "tenant-scoped read returns nothing → 404, not empty 200" when the URL named a specific entity.

What this lesson does not cover:

- The Server Action arm of the wrapper (lesson 2 of chapter 057).
- Member-management routes (lesson 4 of chapter 057 — the lesson uses Server Actions; route handlers for these aren't required).
- The audit-log write inside the route handler (lesson 5 of chapter 057 — same hook, different seam).
- Idempotency-Key handling at depth (Chapter 063).
- CORS configuration (named once; lives in `next.config.ts`).
- Bearer-token authentication and machine-to-machine identity (named once; not built).
- RFC 9457 Problem Details schema authoring at depth (lesson 2 of chapter 046 owns).

Estimated student time: 40 to 50 minutes. Pattern lesson, shorter than lesson 2 of chapter 057 because the discipline is already taught; this lesson is the *port*. A small side-by-side — `authedAction` versus `authedRoute`, same signature, different response shape — is the load-bearing visual. Exercise: take a Server Action from lesson 2 of chapter 057 and write the route-handler twin sharing a `/lib` function.

## Lesson 4 — The five member-management flows

Teaches the listing query and the four privileged actions — change role, remove member, leave org, transfer ownership — the invariants that live in the helpers, the multi-row transactions they run inside, and how a removed member's stale session resolves itself on the next request.

Topics to cover:

- **The senior question.** The org has members. The settings page needs to list them with role, allow an admin to change a member's role, remove a member, allow a member to leave, and allow an owner to transfer ownership to another member. Each flow has an invariant — at least one owner, can't remove yourself, can't demote yourself out of last owner — and each flow writes an audit row. What are the five Server Actions, what invariants live in the helper, and where does the audit write attach?
- **Listing members.** A Server Component on `/settings/members` calls `tenantDb(orgId).query.member.findMany({ with: { user: true } })` and renders the table — name, email, role, joined-at. Pagination is rare here (most orgs are under 50 seats; the unbounded grow case is Chapter 060 territory). Senior reflex: list the *current* org's members only; the scope comes from `tenantDb`. The page is read-only for `member` (no buttons), shows action buttons gated on `roleAtLeast(role, 'admin')`.
- **Changing a member's role — `changeMemberRoleAction`.** `authedAction('admin', changeMemberRoleSchema, async ({ memberId, newRole }, ctx) => ...)`. The schema validates `memberId: z.string().uuid()` and `newRole: z.enum(['owner', 'admin', 'member'])`. Invariants: an `admin` cannot promote to `owner` (owner-only operation — split into `transferOwnership` flow); an `admin` cannot change another `owner`'s role (only the owner can demote themselves, and only via the transfer flow); the last-owner invariant — can't demote the only owner; can't demote yourself if that would leave no owner. The senior reach: the demotion-self case usually fails the last-owner check rather than a separate self-demotion ban; explicit error codes (`'last-owner'`, `'cannot-demote-owner'`, `'forbidden'`) let the UI render the right message.
- **Removing a member — `removeMemberAction`.** `authedAction('admin', removeMemberSchema, async ({ memberId }, ctx) => ...)`. Invariants: can't remove yourself (use leave-org); can't remove the last owner; an `admin` can remove other `admin`s and `member`s but not `owner`s. The senior reach: removal is a hard delete on the `member` row, not a soft delete (membership isn't an entity with history; the audit log holds the trail). What happens to the removed user's `activeOrganizationId` on their session — covered as a trap below.
- **Leaving the org — `leaveOrgAction`.** `authedAction('member', emptySchema, async (_, ctx) => ...)`. Any role can leave. Invariants: an `owner` cannot leave without transferring; the action returns `{ ok: false, error: { code: 'last-owner-must-transfer' } }` if so. On success, the `member` row is deleted; if the user had this org as `activeOrganizationId`, the action picks a fallback (first remaining membership, else null) and updates the session. Redirect to `/dashboard` (or `/onboarding/create-org` if the user has no orgs left).
- **Transferring ownership — `transferOwnershipAction`.** `authedAction('owner', transferOwnershipSchema, async ({ newOwnerId }, ctx) => ...)`. The schema validates `newOwnerId: z.string().uuid()`; the new owner must be an existing member of the same org. Invariants: target must be a member; can't transfer to yourself; transfer is atomic — promote the new owner, demote the old owner to `admin` (the convention; some products demote to `member`), in one transaction. Senior reflex: the demotion target (admin vs. member) is a *product decision*; the year-1 default is admin so the former owner keeps administrative access without billing rights.
- **The transaction shape — multi-row writes in one atomic step.** Transfer-ownership writes two `member` rows in one transaction. Removal writes one `member` row plus one `audit_logs` row in one transaction. Role-change writes one `member` row plus one `audit_logs` row in one transaction. The senior reflex: `await ctx.db.transaction(async (tx) => { ... })` from lesson 5 of chapter 043 — Drizzle's transaction API; every multi-row write goes through it. The action body returns `{ ok: true, data: ... }` only after the transaction commits.
- **The session-cleanup side effect — removing a member kicks them out.** When a member is removed, their existing browser session still has `activeOrganizationId = thisOrgId`. Their next request reads `activeOrganizationId`, calls `tenantDb`, but membership-check inside `requireOrgUser` finds no row → the helper redirects to `/onboarding/create-org` (or `/dashboard` if they have other orgs). The senior reflex: don't try to "kill the session" — let the membership check do the work; the user's session is valid (they're still signed in), just the active-org branch isn't valid for them anymore. The wrapper handles this naturally because the role read happens per request.
- **The audit-log write for each flow — what to record.** Foreshadowing lesson 5 of chapter 057. Each privileged member-management action writes an `audit_logs` row inside the same transaction: actor (the acting user), action type (`'member.role-changed'`, `'member.removed'`, `'member.left'`, `'org.ownership-transferred'`), subject (the affected user), payload (the before-after role for change, the target for transfer). The full schema lands in lesson 5 of chapter 057; the pattern shows up here so the actions are complete.
- **The UI surface — what the settings page looks like.** A table of members with role dropdowns (visible for admins+, disabled for `owner` rows unless current user is owner), a "remove" button per row (gated), a "transfer ownership" action (visible to owner only), a "leave organization" button at the bottom for the current user. The senior reflex: gate the *UI* on role to avoid confusion, gate the *action* on role for the security boundary; the UI gate is cosmetic, the action gate is structural.
- **The optimistic update reflex.** Role changes are fast (one row, one transaction); a senior reach is `useOptimistic` on the dropdown so the UI updates instantly and rolls back on error. The wrong-role error code surfaces as a toast. Whether to bother with optimism here is a UX call — the work is fast enough that pending state is usually fine. Name the option; the default is the simple `useActionState` flow.
- **The race — two admins changing the same member at once.** Admin A demotes Bob to member; Admin B promotes Bob to admin; both write at the same time. The last write wins. For role changes this is acceptable (no business invariant violated, the audit log records both writes). For ownership transfer this matters more — two owners transferring simultaneously to different members would race; the senior reach is optimistic concurrency on `member.role` (the `version` column pattern from lesson 3 of chapter 061) if the product surfaces this risk. Year-1 default: last-write-wins, the audit log answers "who did what when."
- **Watch-outs.** Calling `auth.api.removeMember` from Better Auth without authz around it — the library doesn't check your role, it checks org membership; the caller's role gate belongs in `authedAction`; surfacing the "remove" button on the *current user's* row — users hit it and get confused; remove their own row from the UI gate; treating the last-owner check as "the UI prevents it" — it doesn't, the action must refuse with `'last-owner'`; running the role-change and the audit-log write in separate transactions — partial state if the audit-log write fails; rolling back the role change is the right behavior, do them together; transferring ownership to a non-member — schema must check membership in the action body, not in the schema (Zod can validate format, not existence); allowing self-promotion via `changeMemberRoleAction` — the wrapper allows it because role is `admin` and the operation looks like a role change; explicit invariant in the body refuses; missing the `revalidatePath('/settings/members')` after a write — the table doesn't refresh, the user thinks the action failed; failing to handle the "I just left the org I'm currently looking at" UI case — the `revalidatePath` plus `redirect('/dashboard')` is the right exit.

What this lesson does not cover:

- The audit-log table shape (lesson 5 of chapter 057).
- Invitations — adding new members (Chapter 058 entirely).
- Per-feature permissions inside the role (named once; not built).
- Bulk member operations (out of scope for year-1).
- Member-import from CSV / SCIM provisioning (out of scope).
- The settings page's broader scope — billing tab, integrations tab (those land in Unit 11 and elsewhere).

Estimated student time: 50 to 60 minutes. Mechanics-heavy with multiple flows; the lesson is *five small actions, one shared shape*. A flow diagram for the ownership-transfer transaction earns its weight; an inline error-code table — `'last-owner'`, `'cannot-demote-owner'`, `'forbidden'`, `'not-a-member'` — closes the lesson.

## Lesson 5 — The append-only audit_logs table

Teaches the audit table's column shape, the three-layer append-only enforcement (RLS policy, table grants, application discipline), the `logAudit(tx, event)` helper whose signature forces a transaction, the canonical event set, and the retention story.

Topics to cover:

- **The senior question.** Three months from now: a customer support ticket reads "an admin removed my account access yesterday and I don't know who or why." Or a compliance officer asks "show me every role change in this org over the last 90 days." Or a security incident demands "who exported the customer database, when, from which IP." What's the table that survives these questions, what does each row record, when does the row get written, and what's the discipline that makes "the row exists if and only if the action landed" structural?
- **The shape — append-only by contract, by grants, by RLS policy.** Columns: `id` (uuid pk), `organizationId` (uuid, RLS-scoped), `actorUserId` (uuid, nullable for system actions), `actorIp` (text, the request IP), `actorUserAgent` (text, truncated), `action` (text, the verb — `'member.role-changed'`, `'org.deleted'`, `'invoice.exported'`), `subjectType` (text, the entity type — `'member'`, `'invoice'`), `subjectId` (text, the entity id), `payload` (jsonb, the before/after diff or the operation arguments), `createdAt` (timestamp, default `now()`). The table has no `updatedAt`, no `deletedAt` — *append only*. The senior reflex: the column shape codifies the invariant; rows don't change, they just accumulate.
- **The append-only enforcement — three layers.** (1) RLS policy from lesson 4 of chapter 056 covers org isolation. (2) Add a second policy denying UPDATE and DELETE to the app role: `CREATE POLICY audit_logs_no_update ON audit_logs FOR UPDATE TO authenticated USING (false)` and the same for DELETE. (3) Application discipline — no code path in `/lib` or `/app` ever issues an UPDATE or DELETE on `audit_logs`. Senior reach: the database refuses; the application never asks. The owner role bypasses for the unlikely "we need to retract one row under legal order" case — explicit, documented, rare.
- **What gets audited — the canonical event list.** Year-1 SaaS audit set: member added (invite accepted), member removed, member role changed, ownership transferred, org created, org deleted, billing plan changed (Unit 11), data exported (large reads), settings changed (sensitive fields only — not every typo). The senior reflex: *audit the verbs that an auditor would ask about three months from now,* not every CRUD operation. Auditing every `update` on `invoices` is noise; auditing "the invoice was voided" is signal.
- **What doesn't get audited.** Read operations on normal data (people viewing dashboards isn't audit-worthy in year-1 SaaS — different in regulated verticals); high-frequency low-stakes writes (creating a comment, marking a task done); system-driven writes that have their own log trail (Stripe webhook events are recorded in `processed_events` from chapter 063, not audited again). The senior reflex: the audit log is for *privileged human actions*, not for the activity feed.
- **The write call — inside the action transaction, through `withTenant`.** From lesson 2 of chapter 057 and lesson 4 of chapter 056: the audit write happens inside the same DB transaction as the privileged mutation, scoped through `withTenant(orgId, ...)` so RLS's `SET LOCAL app.org_id` is set. Pattern: `await withTenant(orgId, async (tx) => { /* mutation */; await tx.insert(auditLogs).values({ organizationId: orgId, actorUserId: ctx.user.id, action: 'member.removed', subjectType: 'member', subjectId: memberId, payload: { previousRole: prevRole } }); })`. The senior reflex: same transaction, same scope, both layers of defense, structural.
- **The `payload` JSONB shape — diff or args.** For state changes (role change, settings edit), record `{ before, after }`. For action-style events (export, deletion), record the arguments — `{ exportType: 'customers', rowCount: 1247 }`. Senior reach: the `payload` is *human-readable forensics* — terse enough to read quickly, complete enough to reconstruct what happened. Don't dump every column; record the diff that matters.
- **The actor context — user, IP, user-agent.** Every audit row carries the actor. `actorUserId` from `ctx.user.id`. `actorIp` from `request.headers.get('x-forwarded-for')` (Vercel passes it; the senior reflex is to take the first IP and validate the shape — proxy headers are spoofable, so the IP is "informational from the edge," not a security primitive). `actorUserAgent` from `request.headers.get('user-agent')` truncated to a sane length. The audit helper reads these from a thin context object the action builds.
- **The system-action case — actor is null.** Some audit-worthy events have no human actor: a scheduled job promotes a trial to paid, a webhook from Stripe creates a subscription. Those rows record `actorUserId: null`, `action: 'system.subscription-created'`, `payload` carries the provenance (`{ source: 'stripe-webhook', eventId: '...' }`). Senior reflex: the system action is still in the audit log; the absence of an actor is itself information.
- **The audit helper — `logAudit(tx, event)`.** A small function in `src/lib/audit.ts` that takes a Drizzle transaction handle and an event payload, inserts the row. The signature forces the caller to pass a `tx` — auditing happens inside a transaction, never as a fire-and-forget call. Senior reflex: the type signature is the discipline; if it's hard to call `logAudit` outside a transaction, it doesn't happen outside one. The wrong call shape (calling on the raw `db` instead of `tx`) is a type error.
- **The retention story — how long the rows live.** Audit logs accumulate; the table can grow unboundedly. The senior call: retain for the longer of the regulatory horizon (HIPAA's six years, SOC 2's typically one year) or the product's defensible window. For year-1 SaaS without a specific regulatory commitment, two years is a reasonable default — set as a scheduled deletion job that runs against the table with the *owner role* (bypassing the no-DELETE policy explicitly). Foreshadow lesson 4 of chapter 081 (GDPR retention) for the customer-data side; audit logs themselves often have *longer* retention than the data they audit, because they describe the actions taken on that data. Name the job; don't build it now.
- **The admin / compliance read surface.** The settings page surfaces a read-only "Activity log" for admins+ — paginated, filterable by actor and action type, scoped through `withTenant` (RLS applies). The senior reflex: the read surface lives in the product itself for self-serve compliance review; the *export* path (for legal subpoena response) is a separate authenticated download that triggers its own audit event (`'audit.exported'`). The page is named here; the build is straight Server Component + Drizzle query.
- **The PII reflex.** The `payload` can contain personal data (email, name, before/after of a profile field). The senior call: this is fine — audit logs *are* sensitive and the RLS policy plus the org scope keeps them tenant-isolated. Don't try to obfuscate or hash the payload "for safety"; that breaks forensic readability and adds no security. The protection is access control on the table, not opacity on the rows.
- **The performance reflex — index strategy.** Indexes on `(organizationId, createdAt DESC)` for the per-org timeline read, on `(organizationId, actorUserId, createdAt DESC)` for the per-user filter. The senior reflex: the read patterns drive the indexes; audit logs are write-heavy, read-rare, and the per-org timeline is the common query. Avoid over-indexing — every index slows the insert side.
- **The "audit log itself" recursive question.** Who watches the watcher? In year-1 SaaS, the audit log doesn't audit its own deletions — there's no recursive table. The append-only contract plus the no-DELETE policy plus the owner-role-only retention job is the protection. Senior reflex: don't build "audit log of the audit log" — the operational answer is "the owner role is closely held and the retention job is reviewed in code."
- **Watch-outs.** Writing audit rows *after* the response is returned (via `after()` or background work) — if the background write fails, the audit row is lost and the action looks successful from the user side; audit writes must be in the same transaction as the work; reaching for "centralized middleware that audits every action" — wrong abstraction, the action body knows what to record, the wrapper doesn't; recording too much (every read, every page view) — the table becomes a debug log, not an audit log, and the signal is lost in noise; recording too little (only failures) — auditing *successful* privileged actions is the point, the failure path goes to Sentry; relying on `createdAt = now()` from the database without ordering on it for tied timestamps — same-transaction inserts can have identical timestamps; add a serial `seq` column if total ordering matters within a millisecond; allowing UPDATE on `audit_logs` for "we need to fix a typo" — never; if a row is wrong, write a corrective row; trusting the IP header without origin context — `x-forwarded-for` is informational; building the audit table without RLS — lesson 4 of chapter 056 wired RLS for this table specifically, missing this defeats the threat model; storing the IP as `text` instead of `inet` — Postgres has an `inet` type that validates the shape; senior reflex is to use it; not writing the audit-row for the system-action case because "there's no user" — `null` actor is valid and meaningful.

What this lesson does not cover:

- Re-visiting RLS on `audit_logs` (lesson 4 of chapter 056 owns).
- The full security baseline audit (Chapter 081).
- GDPR retention timers (lesson 4 of chapter 081).
- The audit-log export for legal response (named once; not built here).
- Webhook event recording — `processed_events` is a different table (Chapter 063).
- Activity feeds for users (different concept — user-facing, not compliance).
- Per-row encryption of audit payloads (not a year-1 reach).

Estimated student time: 45 to 55 minutes. Pattern + setup; the table schema, the helper, the policy extension, the call shape inside actions. A small ER fragment with `audit_logs` and arrows to `user`, `organization` carries the shape; a sequence diagram for "Server Action → withTenant(orgId, async tx => { mutate; logAudit(tx, event) })" closes the lesson.

## Lesson 6 — Quizz

Top 10 topics to quiz:

- The three-role year-1 default (`owner` / `admin` / `member`), the authority gradient, and the conditional trigger that flips to a granular permissions model (a paying customer asks for "approver" or similar).
- The single-owner invariant — where it's enforced (helper, not UI), the three flows that touch it (remove, demote, leave), and the canonical error code (`'last-owner'`).
- The `authedAction(role, schema, fn)` signature, the four wrapper steps (`requireOrgUser`, role check, schema parse, call `fn`), and the `ctx` payload (`user`, `orgId`, `role`, `db = tenantDb(orgId)`).
- The Principle #5 carve-out — `authedAction` and `tenantDb` are the named exceptions, when "direct call" is correct (cross-org admin scripts, system jobs), when "via the wrapper" is correct (every request-handled privileged write).
- The `Result` versus `Response` shape — Server Action returns `Result<T>` for in-form rendering; route handler returns HTTP + RFC 9457 Problem Details; the discipline (session + role + schema) is identical at both seams.
- The status-code map at the route handler — 401 unauthenticated, 403 forbidden, 422 unprocessable input, 404 over 403 on cross-tenant leak.
- Member-management invariants — can't remove yourself (use leave-org), can't remove last owner, can't demote last owner, can't transfer to a non-member; what each flow's audit-row records.
- The `audit_logs` append-only contract — no UPDATE, no DELETE, enforced at three layers (RLS policy, table grants, application discipline); what gets audited (privileged human actions) vs. what doesn't (reads, high-frequency writes).
- The audit write discipline — inside the same transaction as the mutation, scoped through `withTenant(orgId, ...)`, written via `logAudit(tx, event)` whose signature forces a `tx` argument so off-transaction calls don't typecheck.
- The session-after-removal behavior — removed member's session stays valid (they're signed in) but their next protected-page request fails the membership check in `requireOrgUser` and redirects; no "kill the session" call is required.
