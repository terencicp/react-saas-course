# Chapter 013 ‚Äî JSON, classes, and the Temporal pivot

## Chapter framing

Chapter 013 closes Unit 2 by installing three "value-shape" disciplines a 2026 SaaS senior reaches for at the boundary between in-memory values and everything outside them ‚Äî the wire, third-party SDKs, and the standard-library date type the language is finally retiring. Each lesson answers a *trust* question: what does a JSON string become when it crosses into our process; what is a class actually good for once records and functions own most of the surface; and what time type does the application speak now that `Date` has a replacement on the platform. The unit so far built the value model (005‚Äì007), the type discipline (008‚Äì009), the module graph (chapter 010), async (chapter 011), and errors (chapter 012); this chapter is where those threads land on the three remaining standard-library corners the SaaS developer still touches daily.

Threads that run through every lesson. **The wire is `unknown` until validated** ‚Äî `JSON.parse` returns `any`-shaped data, third-party SDKs hand back values whose types lie, and the senior reflex is to narrow at the seam with Zod (Unit chapter 046) before the value reaches the domain. **Records and functions are the default; classes are the carve-out** ‚Äî the chapter names the three legitimate class triggers and refuses the rest, locking the 2026 functional-first posture into muscle memory. **`Date` is a third-party seam, not a domain type** ‚Äî every `Date` instance the runtime hands the app gets converted to `Temporal.Instant` immediately, and every domain operation reads and writes Temporal. **One codec, one place** ‚Äî JSON revivers/replacers, class adapters, and Temporal codecs all live at the seam in a `lib/` module, never inline at call sites. The Temporal pivot is the chapter's load-bearing setup for Chapter 087 (storage / domain / edge) and Chapter 103 (live migrations that need explicit instant semantics).

## Lesson 1 ‚Äî JSON at the wire boundary

Teaches `JSON.parse` / `JSON.stringify` as the wire codec, the four serialization holes (`undefined`, `Date`, `BigInt`, `NaN`/`Infinity`), the `parse-to-unknown-then-Zod` discipline, and the rare cases where `reviver` and `replacer` earn their keep.

Topics to cover:

- **The senior question.** A Server Action receives a JSON body; a route handler returns one; a third-party webhook posts one; `localStorage` round-trips one (Server Actions: Chapter 047; route handlers: Chapter 019 / chapter 050). Four directions, one codec. The naive shape ‚Äî `const data = JSON.parse(req.body)` and start reading fields ‚Äî silently lets every wire-format lie through to the domain. The senior shape is two steps: `JSON.parse` produces an `unknown`, a Zod schema (Unit chapter 046) narrows it to the domain type, the parsed value never reaches business logic untyped. Install the two-step reflex.
- **`JSON.parse` mechanics.** Returns the JavaScript value the JSON string described. The runtime type is `any` by convention but `unknown` by discipline ‚Äî the project's `tsconfig.json` doesn't redeclare it, the senior wraps every parse in a function whose return type is `unknown`. Throws `SyntaxError` on malformed input; the catch belongs at the *transport* layer (the route handler, the Server Action wrapper), not at every call site. Empty string and missing body throw ‚Äî guard before parsing.
- **`JSON.stringify` mechanics.** Returns a string. Three signature shapes: `stringify(value)`, `stringify(value, replacer)`, `stringify(value, replacer, space)`. The third argument is *not* a debug-only nicety ‚Äî `null, 2` pretty-prints; the senior reach is `null` (no replacer) and *no* space in production, `2` only at log-site or in dev tooling. The replacer is rarely worth the indirection (named below).
- **The four serialization holes.** Each one is the same bug: the JSON grammar can't carry the value, and `stringify` either drops it or coerces. **(1) `undefined`** ‚Äî properties with `undefined` values get *dropped* from object output; array elements become `null`. The asymmetry breaks `{ a: undefined }` round-tripping. Senior reflex: `null` for "explicitly absent in JSON," `undefined` for "not set in TS." **(2) `Date`** ‚Äî `stringify` calls `toJSON()` which returns an ISO 8601 string; `parse` returns the *string*, not a `Date`. The round-trip downgrades the type; install the Temporal codec (lesson 3 of chapter 013) to read it back as `Temporal.Instant`. **(3) `BigInt`** ‚Äî `stringify` throws `TypeError`. Two reaches: `JSON.stringify(value, (_, v) => typeof v === 'bigint' ? v.toString() : v)` to ship as a string, or model the value as a string at the schema boundary (every Stripe ID, every 64-bit Postgres id). **(4) `NaN` / `Infinity` / `-Infinity`** ‚Äî serialize as `null`. The bug class is "an analytics field came back null after the math overflowed"; the cure is validation at the math site, not at the serializer.
- **Special-value table, restated.** Functions, `Symbol` keys, and `Symbol` values are *silently dropped* from objects (and become `null` in arrays for functions). Class instances serialize their *own enumerable* properties only ‚Äî methods and getters disappear, and private `#fields` disappear too (lesson 2 of chapter 013). A `Map` or `Set` serializes as `{}` (empty object) ‚Äî the senior never `JSON.stringify`s a `Map` and expects a `Map` back. Cyclic references throw `TypeError: cyclic object value`; `structuredClone` (lesson 1 of chapter 005) is the deep-clone reach, not `JSON.parse(JSON.stringify(...))`.
- **The `toJSON` hook.** Any value with a `toJSON()` method has it called by `stringify`. `Date.prototype.toJSON` returns ISO 8601. `Temporal.Instant.prototype.toJSON` (lesson 3 of chapter 013) returns ISO 8601 with `Z`. The senior reach on custom classes (the rare ones in lesson 2 of chapter 013) is to implement `toJSON` so the wire shape is explicit ‚Äî never serialize a class by default and pray.
- **`reviver` and `replacer`, narrowly.** Both are second-argument hooks that walk the value bottom-up. **Senior rule:** prefer Zod's `transform` (Unit chapter 046) over `reviver` for type reconstruction ‚Äî the schema owns the contract, the reviver lives apart. The two cases where `reviver` still earns its weight: stripping `__proto__` and `constructor` keys defensively before Zod sees them (the "prototype pollution" defense at the parse step), and reading legacy JSON that's larger than memory budgets through a streaming parser (rare). `replacer` earns its weight for one job: redacting sensitive fields in log output (`(_, v) => /password|token|secret/i.test(key) ? '[REDACTED]' : v`).
- **The parse-to-`unknown`-then-Zod discipline.** The chapter's load-bearing rule. Wrap parses behind a typed helper: a `parseJson<T>(s: string, schema: ZodType<T>): T` that calls `JSON.parse`, runs `schema.parse`, and throws on either step. Lives once in `lib/json.ts`. Every wire boundary in the codebase imports it. The bug class it kills: "I read the response and trusted the shape." Unit chapter 046 owns the full Zod surface; this lesson installs the seam.
- **`Response.json()` and `Request.json()`.** The fetch API's body methods (`response.json()`, `request.json()` in Route Handlers) call `JSON.parse` under the hood. The return type is `Promise<any>` ‚Äî same `unknown`-by-discipline rule applies. Pair with a Zod parse in the same expression: `const body = ParcelSchema.parse(await request.json())`. Course's canonical request-handler shape.
- **Bandwidth and shape ‚Äî what NOT to optimize.** `JSON` is verbose; the 2010s reach was MessagePack, Protobuf, or BSON. The 2026 senior reach for SaaS is to *not* optimize the wire format until the profiler says so ‚Äî JSON over HTTP/3 with gzip/br is fast enough for the request shapes a SaaS ships. Named once, dismissed.
- **Pretty-printing and stable key order.** `stringify(value, null, 2)` indents. Key order matches *insertion order* (the same V8 rule as `Object.keys`); cross-implementation stability is *not* guaranteed by the spec ‚Äî never use `JSON.stringify` as a content-hash input. Use a canonicalizer (`fast-json-stable-stringify` or hand-roll a sorted-keys variant) only when the hash is the lesson; otherwise the default order is fine for wire.
- **Watch-outs.** Catching `SyntaxError` at every call site is noise ‚Äî the transport-layer wrapper owns it; using `JSON.parse(JSON.stringify(x))` for deep-clone misses `Date`, `Map`, `Set`, `Symbol`, `undefined`, `BigInt`, and cycles ‚Äî `structuredClone` is the reach; passing a class instance to `stringify` and expecting the class back on the receiver ‚Äî the wire is records, the class is reconstructed by a schema; relying on `JSON.parse` to validate types ‚Äî it doesn't, it produces; the `BigInt` `TypeError` shows up only when a 64-bit ID enters serialization ‚Äî guard at the column-type layer, never let `bigint` reach `stringify` raw; `__proto__` injection via crafted JSON ‚Äî `Object.create(null)` containers and Zod schemas neutralize it, never `Object.assign({}, parsedJson)` into a prototype-bearing object; pretty-printed payloads inflate response size 1.5‚Äì2√ó ‚Äî strip in production logs and responses.

What this lesson does not cover:

- The Zod schema surface and `z.iso.date()`, `z.coerce`, `transform` ‚Äî Chapter 046.
- `Request.json()` and `Response.json()` at depth, and `Headers.get('content-type')` discrimination ‚Äî lesson 1 of chapter 019.
- Server Action input validation and the `safeAction` wrapper ‚Äî Chapter 047.
- Structured logging, JSON log lines, and the redact step ‚Äî Chapter 096.
- `structuredClone` and shared-reference semantics ‚Äî lesson 1 of chapter 005.
- `BigInt` arithmetic and Postgres's `bigint` column type ‚Äî lesson 3 of chapter 005 and Chapter 041.
- `localStorage` and storage round-trips ‚Äî lesson 4 of chapter 020.

Estimated student time: 35 to 45 minutes. The wire-boundary discipline; every later validation lesson rides this seam.

## Lesson 2 ‚Äî Classes, narrowly

Teaches the three legitimate 2026 SaaS uses for `class` ‚Äî custom `Error` subclasses, third-party SDK adapter wrappers, and the rare stateful domain object ‚Äî and the minimum class surface (`constructor`, `readonly`, `#private`, arrow-field methods, `static` factories) with everything else explicitly refused.

Topics to cover:

- **The senior question.** A new contributor reaches for `class UserService` with five methods, two of which are `static`, one of which uses `this` accidentally inside a callback. The 2026 SaaS codebase the course teaches answers most "where does this code go?" questions with a *module of functions over a typed record* ‚Äî `getUserById(id)` and `User` type, not `UserService` and `User` class. Three triggers earn a `class` declaration; everything else is a smell. The lesson installs the trigger list and the minimum-surface conventions.
- **The three triggers.** **(1) Custom `Error` subclasses** ‚Äî `Error` is a class, subclassing extends the type discrimination lesson 2 of chapter 012 installed (`name` as literal discriminant, `cause` for chain walking). Domain errors (`ValidationError`, `RateLimitError`, `AuthError`) ride this surface. **(2) Third-party SDK adapter wrappers** ‚Äî when a library (Stripe, Better Auth, an HTTP client) exposes a class and the project needs to hold state across calls or override behavior, a thin wrapper class is the right reach. The wrapper is what crosses the application boundary, not the library type. **(3) The rare stateful domain object** ‚Äî a long-lived in-memory entity with invariants enforced through methods (a `Cart` that totals itself, a `RateLimiter` token bucket). 2026 SaaS has fewer of these than older codebases; most "state" lives in Postgres, Zustand, or React state. Name them once; don't reach.
- **Why not classes for everything, restated.** Two failure modes drove the shift. **(a)** `this` binding ‚Äî methods passed as callbacks lose `this`, the fix is arrow-field methods which then break inheritance, the senior reach is functions that close over a typed record where there is no `this` to lose. **(b)** Serialization ‚Äî class instances cross the wire as records (lesson 1 of chapter 013 covers how `JSON.stringify` drops methods and `#fields`); the class adds nothing across the boundary. Records + functions match the SSR + Server Action shape the course teaches; classes fight it.
- **The minimum class surface.** When a class earns its weight, ship only these primitives. **(a)** `constructor` with one options-object parameter (the lesson 2 of chapter 006 rule extends here). **(b)** `readonly` on every field that isn't mutated post-construction; the senior default is `readonly` everywhere unless mutation is the lesson. **(c)** `#private` (hash-private) for fields the class controls ‚Äî *not* TS-only `private`, which is erased at runtime. Hash-private is enforced at runtime and survives the compile. **(d)** Arrow-field methods (`handle = (...) => { ... }`) for any method passed as a callback ‚Äî preserves `this` binding by construction. Regular method syntax for methods always called as `instance.method()`. **(e)** `static` factory methods (`static fromJson(s: string)`) for non-trivial construction; constructors stay thin.
- **What the surface explicitly refuses.** **(a)** Class inheritance hierarchies beyond `extends Error` ‚Äî composition over inheritance, the LSP traps and the brittle-base-class problem don't pay for themselves. **(b)** `abstract class` ‚Äî TS-only marker, runtime hole; reach for a type and a discriminated union instead. **(c)** Mixins (`Object.assign(Class.prototype, ...)`) ‚Äî gymnastics around the inheritance limitation, the cure for which is composition. **(d)** Decorators ‚Äî Stage 3 in 2026, real cases exist (NestJS, TypeORM, MikroORM), but the SaaS stack the course teaches (Next.js + Drizzle + Better Auth) doesn't use them; named once, dismissed for this stack. **(e)** Getters and setters (`get x() {}` / `set x(v) {}`) ‚Äî they look like field reads but execute code; the surprise is the cost. Plain methods (`getX()`, `setX(v)`) when computed access is needed. **(f)** Class expressions (`const C = class { }`) ‚Äî anonymous classes complicate stack traces; always `class C { }`.
- **Custom `Error` subclasses revisited.** From lesson 2 of chapter 012: the canonical shape is `class ValidationError extends Error { override readonly name = 'ValidationError' as const; constructor(message: string, options?: { cause?: unknown }) { super(message, options); } }`. The `as const` literal `name` is the discriminant the catch uses (`error.name === 'ValidationError'`); `cause` rides through `super`; `instanceof` works but cross-realm fails ‚Äî `name` is the load-bearing check. No methods beyond `constructor`; no fields beyond `name`. The minimum.
- **The SDK adapter wrapper shape.** Stripe's SDK exposes a `Stripe` class. The application wraps it: `class BillingClient { readonly #stripe: Stripe; constructor(apiKey: string) { this.#stripe = new Stripe(apiKey, { ... }); } async createCheckoutSession(input: CheckoutInput): Promise<CheckoutSession> { ... } }`. Reasons: (a) the wrapper's method shape is the application's contract, not Stripe's; (b) `#stripe` is hidden, no leak; (c) construction is centralized (one `apiKey`, one `apiVersion`, one set of defaults). Module-level singleton (`export const billing = new BillingClient(env.STRIPE_KEY)`) is the senior reach when the wrapper has no per-request state; per-request instance when it does (rare).
- **Static factories beat overloaded constructors.** When a class has multiple construction paths (`fromJson`, `fromRow`, `fromForm`), make them `static` methods returning instances. Constructor stays one shape. Readability win: the call site reads `Cart.fromRow(row)` not `new Cart(row, undefined, true)`.
- **Arrow-field methods vs. regular methods, the precise rule.** Arrow-field is needed when the method is detached from the instance ‚Äî passed as a callback, stored in a Map, attached as a DOM event handler. Regular method is the default everywhere else (slightly cheaper memory; one copy on the prototype, not one per instance). Heuristic: if the method's first line could be `const self = this` and that surprises you, it's an arrow-field. React class components are gone (Chapter 027); the most common arrow-field site in 2026 is the SDK adapter passing its own method as a callback into a third-party listener.
- **Class fields vs. constructor assignment.** Modern syntax: `class C { readonly id: string; constructor(id: string) { this.id = id; } }` or `class C { constructor(readonly id: string) { ... } }` (TS-only parameter-property sugar ‚Äî works but mixes declaration and assignment, the senior reaches for the explicit form). Default to explicit field declaration plus constructor body assignment; readability beats brevity here.
- **Equality and identity for class instances.** `===` is reference identity; two instances built from the same data are not equal. Comparison through a value-equality helper (`a.id === b.id`) or by collapsing to a record (`structuredClone` + deep equality) ‚Äî but the cure for "I keep needing instance equality" is to use a record, not a class.
- **`instanceof` and the cross-realm trap, restated.** From lesson 2 of chapter 012: `instanceof` walks the prototype chain in the current realm; values that crossed an iframe / worker / `vm.runInContext` boundary fail. The cure is the `name` discriminant on `Error` subclasses; on non-error classes (the rare two triggers above), the boundary is usually a serialization seam where the class is reconstructed anyway.
- **`Object.create(null)` for prototype-free records.** A specific niche: the "map of arbitrary keys, paranoid about `__proto__`" case (parsed JSON containers, header bag-style storage). Not a class topic per se; named once because it's the alternative when you want "object as dictionary" without prototype risk. `Map<string, T>` is usually the right reach instead (lesson 4 of chapter 007).
- **Watch-outs.** `private` (TS) vs. `#private` (runtime) ‚Äî TS-`private` is gone at runtime, available via bracket access or reflection; hash-private survives compile and refuses bracket access; the `class C { method() {} }` form binds `this` lexically inside the method body ‚Äî passing `c.method` strips the binding silently; `static` methods don't see instance fields ‚Äî common copy-paste bug from instance to static; `instanceof Error` fails in cross-realm contexts ‚Äî use the `name` discriminant; `JSON.stringify(instance)` drops methods and `#fields` ‚Äî implement `toJSON()` explicitly if the wire shape matters; getters that perform expensive work look free at the call site ‚Äî they're not; using a class to "group related functions" is a smell ‚Äî a module is the grouping primitive; using `extends` on anything other than `Error` is a smell on this stack; circular construction (class A's constructor news a B that news an A) is a runtime stack overflow ‚Äî factor through a static factory or out of the class.

What this lesson does not cover:

- The `Error` taxonomy, `cause` chains, and `ensureError` ‚Äî lesson 2 of chapter 012.
- Module patterns for grouping functions and `verbatimModuleSyntax` ‚Äî Chapter 010.
- Branded IDs as the type-level alternative to wrapper classes ‚Äî lesson 4 of chapter 009.
- Discriminated unions as the alternative to inheritance ‚Äî lesson 1 of chapter 009.
- Decorators (Stage 3) ‚Äî named once; out of scope on this stack.
- React class components ‚Äî gone in the 2026 React 19 surface; Chapter 027 owns the function-component model.
- DI containers (`InversifyJS`, `tsyringe`) ‚Äî out of scope; not part of the SaaS stack the course teaches.
- ES2022 `static {}` initialization blocks ‚Äî niche; named once but not taught.

Estimated student time: 40 to 50 minutes. The "when to reach for `class`" discipline; the chapter's most decision-heavy lesson.

## Lesson 3 ‚Äî `Date`'s problems and the Temporal pivot

Teaches the eight production-biting design flaws of `Date`, the senior 2026 pivot to `Temporal` (Stage 4 March 11 2026, unflagged in Node 26 shipped May 5 2026), the polyfill seam for Node 24 LTS, and the chapter-closing rule: `Date` lives at third-party seams only, `Temporal` is the domain type.

Topics to cover:

- **The senior question.** Every SaaS has timestamps ‚Äî created-at, due-by, last-login, scheduled-for. The 2010s answer was `new Date()`, hand-rolled month arithmetic, `date-fns` for the things `Date` couldn't do, and a stack of bugs (off-by-one days, DST drift, "is this UTC or local?" three layers deep). The 2026 answer is `Temporal` ‚Äî landed at TC39 Stage 4 on March 11, 2026, shipping unflagged in Node 26 on May 5, 2026, the course's deploy target. The lesson catalogs why `Date` fails and installs the pivot. Chapter 087 owns the full storage/domain/edge architecture; this lesson lands the *type*.
- **`Date`'s eight design issues.** **(1) Zero-indexed months, one-indexed days.** `new Date(2026, 0, 31)` is January 31; February is `1`. The mistake compiles. **(2) In-place mutation.** `date.setMonth(5)` mutates the instance and returns the new timestamp ‚Äî passing a date through a function can change the caller's value. **(3) Timezone as a second-class concept.** `Date` always stores a UTC instant but every formatter, every getter, every constructor parses through the runtime's local tz silently. `new Date('2026-05-15')` (ISO date) is parsed as UTC; `new Date('2026-05-15 00:00')` (space) is parsed as local. The format determines the meaning. **(4) No duration type.** "30 days" is `30 * 24 * 60 * 60 * 1000` milliseconds ‚Äî fails on DST boundaries, no semantic meaning. **(5) The `Invalid Date` sentinel.** `new Date('garbage')` returns a `Date` object whose `getTime()` is `NaN`. Type system says it's a `Date`; runtime says it isn't. **(6) Year 100 problem.** `new Date(99, 0, 1)` is year 1999, not 99 ‚Äî legacy two-digit-year heuristic baked in. **(7) Millisecond precision only.** Postgres `timestamptz` stores microseconds; `performance.now()` measures sub-millisecond; `Date` is the lossy middle. **(8) `toString()` is locale-and-tz-dependent and unstable across runtimes.** Logging a `Date` produces different strings in dev and prod; `toISOString()` is the only stable form.
- **The `Date` failure modes in production.** Three canonical bugs the course catalogs once. **(a)** "The reminder fires an hour off twice a year" ‚Äî DST transition under naive `+24h` arithmetic. **(b)** "Sydney users see invoices due yesterday" ‚Äî midnight-UTC `Date` interpreted as local across the date line. **(c)** "The audit log says 0:14 UTC for everyone" ‚Äî `toLocaleString()` in a Server Component, server clock is UTC. All three are *structural* bugs `Date`'s API invites; the cure is a different type.
- **What `Temporal` fixes.** **(a)** Five distinct types (`Instant`, `ZonedDateTime`, `PlainDate`, `PlainDateTime`, `Duration`) that name what you have ‚Äî no overloaded `Date` for moments, dates, and clock times. **(b)** Immutable ‚Äî every operation returns a new instance, no in-place mutation. **(c)** Timezone is a first-class component of `ZonedDateTime`; instants carry no tz at all; calendar dates carry no tz at all. The type refuses ambiguity. **(d)** `Duration` is a real type with named components (`{ days: 30 }`, `{ months: 1 }`). **(e)** Invalid inputs *throw* ‚Äî no `Invalid Date` sentinel. **(f)** Nanosecond precision on `Instant`. **(g)** `toString()` is stable ISO 8601 across every runtime.
- **The Temporal type catalog, named once.** lesson 5 of chapter 087 owns the full surface. This lesson names the five and the one-line "what each is for": `Instant` (a UTC point in real time, the wire/db shape), `ZonedDateTime` (an `Instant` plus an IANA tz, the DST-aware type), `PlainDate` (a calendar date ‚Äî year, month, day ‚Äî no time, no tz, the `date`-column domain type), `PlainDateTime` (wall-clock date+time, no tz, rare), `Duration` (an explicit period ‚Äî `{ years, months, weeks, days, hours, minutes, seconds, milliseconds, microseconds, nanoseconds }`).
- **The runtime story, May 2026.** Node 26 ships Temporal unflagged (released May 5, 2026 ‚Äî the course's deploy target on Vercel's Node 26 runtime). Node 24 LTS (the line still on production for most teams until October 2026, when Node 26 promotes to LTS) does *not* ship Temporal natively. Browsers: Firefox 139 enabled it by default mid-2025; Chrome 144 ships in 2026; Safari pending. The course renders dates server-side and ships ISO 8601 strings to the client (lesson 3 of chapter 088), so browser Temporal support isn't a blocker.
- **The polyfill seam.** Two polyfills exist. **(a)** `temporal-polyfill` (FullCalendar, ~20 KB gzipped) ‚Äî modern reach, tree-shakeable, near-spec-perfect. **(b)** `@js-temporal/polyfill` (TC39 champions, ~56 KB gzipped) ‚Äî full-spec, heavier. The senior 2026 reach for SaaS on Node 24 LTS is `temporal-polyfill` re-exported once from `lib/temporal.ts`: `import { Temporal } from 'temporal-polyfill'; export { Temporal };` (or the runtime-guarded `globalThis.Temporal ?? polyfill` form for projects that toggle Node versions). Every application file imports `Temporal` from `lib/temporal.ts`, never from the polyfill directly. The upgrade to Node 26 native is a *one-line change at the seam*. Mixing native and polyfill Temporal in the same process is undefined behavior.
- **The `Date` carve-out.** `Date` does not disappear from the codebase ‚Äî it remains the type third-party SDKs and the platform hand the application. Two narrow seams. **(a) SDK ingress** ‚Äî Stripe's SDK returns `Date` (or Unix seconds); convert *immediately* at the seam: `Temporal.Instant.fromEpochMilliseconds(date.getTime())` for SDK `Date`, `Temporal.Instant.fromEpochMilliseconds(stripeCreated * 1000)` for Stripe's Unix seconds. Never propagate the `Date` into the domain. **(b) `performance.now()`-style stopwatches** ‚Äî `Date.now()` is acceptable for "how long did this operation take in ms" measurements where the absolute time doesn't matter; `performance.now()` is the better reach when it's available. Anything that gets stored, displayed, compared, or scheduled is `Temporal`.
- **The conversion-at-the-seam pattern.** A small `lib/temporal.ts` owns the conversion functions: `instantFromDate(d: Date): Temporal.Instant`, `dateFromInstant(i: Temporal.Instant): Date` (for the rare case of handing back to a `Date`-expecting SDK). Application code reads and writes Temporal exclusively; the seam files (`lib/stripe.ts`, `lib/triggerdev.ts`) own the conversion.
- **Wire format, restated.** `Temporal.Instant.prototype.toString()` and `toJSON()` produce ISO 8601 with `Z`. `Temporal.PlainDate.prototype.toString()` produces ISO 8601 calendar date. Symmetric `from()` static methods on each type parse the same strings. The wire is ISO 8601; the in-memory type is Temporal. lesson 1 of chapter 013's parse-then-Zod discipline composes ‚Äî Zod 4's `z.iso.datetime()` and `z.iso.date()` parse the strings; a `.transform()` lifts to Temporal. lesson 1 of chapter 087 wires the Drizzle codec.
- **What this lesson is *not* trying to teach.** The full Temporal arithmetic surface (`add`, `subtract`, `since`, `until`, `with`, `round`) ‚Äî lesson 5 of chapter 087. The Drizzle column types and codecs ‚Äî lesson 1 of chapter 087 and lesson 2 of chapter 087. The user's timezone column ‚Äî lesson 3 of chapter 087. DST and recurring jobs ‚Äî lesson 4 of chapter 087. Locale-aware formatting (`Intl.DateTimeFormat`) ‚Äî lesson 3 of chapter 088. This lesson installs the *type* and the *pivot*. The architecture rides on top in Unit 18.
- **The three rules this chapter installs for the rest of the course.** **(1)** Never call `new Date(year, month, day)` or `date.setX(...)` in application code. **(2)** Every `Date` from an SDK gets converted to `Temporal.Instant` at the seam ‚Äî no `Date` propagates inward. **(3)** `Temporal` is imported from `lib/temporal.ts`, never from the polyfill or the global directly. Three rules; muscle memory by Unit 18.
- **Watch-outs.** `new Date('2026-05-15')` parses as UTC; `new Date('2026-05-15 00:00')` parses as local ‚Äî the format-determined ambiguity is the prototype `Date` bug; `Date.parse(s)` returns `NaN` on invalid input instead of throwing ‚Äî the same trap; `JSON.stringify(date)` returns ISO 8601 (via `toJSON`), but `JSON.parse` returns the *string*, not a `Date` ‚Äî the round-trip downgrades the type (lesson 1 of chapter 013); `Date.now()` for "30 days from now" arithmetic misses DST ‚Äî `Temporal.Now.zonedDateTimeISO(tz).add({ days: 30 }).toInstant()` is the form; mixing `temporal-polyfill` and native `Temporal` in the same runtime breaks `instanceof` and `from()` cross-type ‚Äî the seam is *the* import path, enforced by `eslint-plugin-no-restricted-imports`; `Temporal.Now.plainDateISO()` called with no `timeZone` argument falls back to runtime-local (UTC on Vercel) silently ‚Äî always pass the user's tz; assuming Node 26 native Temporal is byte-identical to the polyfill ‚Äî near-perfect, four documented deviations; treating `Temporal.Instant` as serializable to `localStorage` without explicit `toString()` ‚Äî the storage layer is strings, the conversion is at the seam; reaching for `date-fns`, `dayjs`, `moment`, or `luxon` on a 2026 project ‚Äî Temporal retired them; the call to remove existing date-library usage is a one-line `lib/temporal.ts` adoption plus a mechanical refactor.

## Lesson 4 ‚Äî Quizz

Top 10 topics to quiz:

- The parse-to-`unknown`-then-Zod discipline ‚Äî `JSON.parse` returns `unknown` by convention; every wire boundary narrows via Zod before the domain reads the value; the `parseJson<T>(s, schema): T` helper as the single seam.
- The four `JSON.stringify` serialization holes ‚Äî `undefined` dropped from objects and nulled in arrays; `Date` downgrades to ISO string on round-trip; `BigInt` throws `TypeError`; `NaN` / `Infinity` serialize as `null`.
- The senior `replacer` and `reviver` reaches ‚Äî `replacer` for log-site field redaction; `reviver` for prototype-pollution stripping; Zod's `transform` is the preferred type-reconstruction path otherwise.
- The three legitimate uses for `class` in 2026 SaaS ‚Äî custom `Error` subclasses, third-party SDK adapter wrappers, the rare stateful domain object; everything else is a function over a typed record.
- The minimum class surface ‚Äî `constructor`, `readonly` fields, `#private` (not TS `private`), arrow-field methods only when detachment from `this` is required, `static` factories over overloaded constructors.
- What the class surface refuses on this stack ‚Äî inheritance beyond `extends Error`, `abstract class`, mixins, decorators, getters/setters that look like fields, class expressions.
- `Date`'s production-biting flaws ‚Äî zero-indexed months, in-place mutation, format-determined timezone parse (`'2026-05-15'` UTC vs. `'2026-05-15 00:00'` local), no duration type, `Invalid Date` sentinel, year-100 heuristic, millisecond-only precision, unstable `toString()`.
- The Temporal type catalog at one-line depth ‚Äî `Instant`, `ZonedDateTime`, `PlainDate`, `PlainDateTime`, `Duration`; what each names and which Chapter 18 column it pairs with.
- The runtime and polyfill story ‚Äî Temporal Stage 4 March 11, 2026; native and unflagged in Node 26 (May 5, 2026); `temporal-polyfill` (~20 KB) re-exported once from `lib/temporal.ts` for Node 24 LTS; native Node 26 is the course's deploy target; mixing native and polyfill in the same runtime is undefined behavior.
- The `Date` carve-out ‚Äî `Date` lives only at third-party seams (Stripe SDK, `performance.now()` stopwatches); converted to `Temporal.Instant.fromEpochMilliseconds(date.getTime())` immediately; never propagates into the domain.
