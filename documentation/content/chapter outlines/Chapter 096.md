# Chapter 096 — Git as shipping discipline

## Chapter framing

Chapter 096 opens Unit 20 by formalizing the version-control habit the student has been using since Chapter 004 without ever naming it. Up to this point `git add`/`git commit`/`git push` have been bookkeeping next to the real work. From here on, version control is *the shipping discipline*: the branch that becomes the preview deployment in chapter 098, the squash-merge commit that becomes a release in chapter 098, the protection rule that makes the CI of chapter 097 actually enforceable, the reflog rescue that turns a botched rebase into a five-minute recovery instead of a stand-up apology. The chapter teaches Git as a senior would use it on a two-to-five-person SaaS team in 2026 — trunk-based, PR-per-change, small reviewable diffs, structural enforcement at the repo settings level.

Threads that run through every lesson. **Trunk-based and PR-per-change** is the workflow — short-lived feature branches off `main`, no long-lived `develop`/`release`/`hotfix` branches; GitHub Flow is the shape, not Git Flow. **Small, reviewable, reversible** is the unit-of-change rule — every PR fits in a senior's head in one sitting, ships behind a flag where risky, rolls back as one revert. **Structural enforcement over discipline** — branch protection and rulesets make the rule mechanically true, not a team norm. **The platform default is GitHub** — the course teaches one provider; GitLab/Bitbucket get one line. **The forty commands a senior actually types** — `add`, `commit`, `push`, `pull --rebase`, `checkout -b`, `rebase -i`, `bisect`, `reflog`, `cherry-pick`, `stash`, `revert`, `restore`; everything else gets named or cut. **Recovery is structural** — the reflog is a safety net, not a panic button; the lesson that teaches it lands *before* the student loses real work.

## Lesson 1 — Trunk-based Git for teams

The four Git objects, the everyday loop, and the rebase-locally / squash-merge-on-PR defaults that keep `main`'s history one commit per shipped change.

Topics to cover:

- **The senior question.** The student has been using `git add`/`git commit`/`git push` since Chapter 004 without thinking about it. Why does the team's workflow shape matter? Because the workflow chosen determines what merges to `main` mean (deployable vs. integration-ready), what the PR review surface looks like, and how the CI gate of chapter 097 is actually load-bearing. The lesson lands the four objects (commit, branch, staging area, remote), the trunk-based workflow as the 2026 default, and `pull --rebase` as the everyday hygiene that keeps history readable.
- **The four objects, named once.** **Commit** — a snapshot of the working tree plus author, message, parent(s); the unit of change. **Branch** — a movable pointer at a commit, cheap to create, no copy of the tree. **Staging area** (index) — the set of changes that *will* be in the next commit; lets you stage in slices when the working tree has more than one logical change. **Remote** — a named URL pointing at a hosted repository (typically `origin` pointing at GitHub); `push` and `fetch` move commits between local and remote.
- **The everyday loop.** `git checkout -b feat/invoice-status` → make changes → `git add -p` or `git add path/` → `git commit -m "..."` → `git push -u origin feat/invoice-status` → open PR. After review: `git pull --rebase origin main`, force-push if rewritten, merge. The reflex is one branch per PR, one PR per logical change.
- **The staging area as a slicing tool.** `git add -p` (patch mode) walks each hunk and asks include/skip. When the working tree has two unrelated changes (a bug fix the student noticed and the feature being shipped), the senior reflex is to stage them into separate commits or separate PRs. The staging area is *what makes that possible* — it's not bookkeeping ceremony.
- **Rebase vs. merge — the team-style decision.** Two integration shapes. **Merge** preserves both histories with a merge commit; the graph forks and joins. **Rebase** rewinds the feature branch's commits and replays them on top of `main`; the graph stays linear. Hashes change after a rebase — the rebased commits are *new* commits with the same content.
- **The 2026 team-style default.** **Rebase locally, squash-merge on PR.** Locally: `git pull --rebase origin main` keeps the feature branch up to date without merge commits cluttering history. On the PR: GitHub's "Squash and merge" button collapses the PR into one commit on `main`. Result: `main`'s history is one commit per shipped change, no internal "wip" / "fix typo" / "address review" noise, every commit on `main` deploys cleanly, `git bisect` is trivial. The merge-commit option ships when the PR's individual commits are meaningful (a multi-commit refactor where each commit is structurally a step); rare in normal feature work.
- **`git pull --rebase` as the everyday default.** The reflex set in `git config --global pull.rebase true` (or per-repo). Without it, every `git pull` on a branch with local commits makes a merge commit and clutters history with "Merge branch 'main' into feat/x" noise. With it, local commits replay on top — clean.
- **The trunk-based workflow.** One long-lived branch — `main`. Feature branches branch off `main`, live hours-to-days, merge back. No `develop`. No `release/*`. No `hotfix/*`. Releases happen by deploying `main`. Hotfixes are normal PRs against `main` that ship fast. This is **GitHub Flow** — the practical shape of trunk-based for small SaaS teams.
- **Why not Git Flow.** Git Flow's `develop`/`release`/`hotfix` branches existed when shipping was a quarterly event with QA gates. With Vercel previews per PR (Chapter 098) + CI per PR (Chapter 097) + feature flags (lesson 4 of chapter 093) for risky merges, the long-lived branches are pure overhead. Named once, cut.
- **Branch naming.** Convention, not enforcement: `feat/`, `fix/`, `chore/`, `refactor/`, `docs/` prefixes; kebab-case body referencing the work, optionally with the ticket ID (`feat/INV-412-status-filter`). Nothing technical hangs on this — it's review readability. Pre-commit hooks or rulesets *can* enforce the shape; the senior call is usually "don't bother."
- **Commit message shape.** Imperative mood ("Add invoice status filter," not "Added"). Subject line ≤ 72 chars, no trailing period. Body wraps at 72, explains *why*, not *what* the diff already shows. **Conventional Commits** as the optional structural layer — `feat:`, `fix:`, `chore:`, `refactor:`, `docs:`, `test:` — earns its weight only when paired with automated changelog generation or semantic-version bumping; for an internal SaaS that doesn't ship a public package, skip it and just write good messages. Named, deferred to the team's call.
- **`.gitignore` and `.gitattributes` as repo artifacts.** The scaffold from Chapter 004 already ships them. The senior diff: `.gitignore` ignores `node_modules/`, `.next/`, `.env*` (except `.env.example`), `*.log`, coverage outputs, OS junk (`.DS_Store`); `.gitattributes` pins line endings (`* text=auto eol=lf`) so Windows contributors don't ship `\r\n` to a Linux-deployed repo. Both committed; both rarely touched after setup.
- **The `.env` rule, again.** `.env*` ignored except `.env.example`. If a secret lands in a commit, the file's gone from working tree, *not gone from history*. Rotation is the only fix (the secret is leaked the moment it's in a remote). lesson 6 of chapter 081 owns the rotation playbook; the prevention is the gitignore line.
- **What "good history" looks like.** `git log --oneline main` reads like a changelog — one line per shipped feature/fix, clear imperative subjects, parseable scan. No merge commits, no "wip," no "fix typo," no "address review comments." Every commit on `main` is a candidate for `git revert` and a candidate for `git bisect`.
- **`git push --force-with-lease`, named.** When a rebased feature branch needs to push (history rewritten, remote is behind), `--force` overwrites the remote unconditionally — destructive if a teammate pushed in between. `--force-with-lease` checks the remote pointer matches what you fetched last; aborts if it changed. Reflex: never `--force`, always `--force-with-lease`. On personal feature branches the practical risk is low; the habit is what saves a teammate's commits the day it matters.
- **Defaults the student should set once.** `git config --global init.defaultBranch main`, `git config --global pull.rebase true`, `git config --global push.autoSetupRemote true`, `git config --global rebase.autoStash true`, `git config --global rerere.enabled true` (re-uses previous conflict resolutions on repeated rebases). One-time setup, lifelong payoff.
- **The Git GUI question.** VS Code's source-control pane, GitHub Desktop, GitKraken, Lazygit, Tower — all valid. Senior reflex: the terminal for everything *except* line-by-line staging and conflict resolution, where a visual hunk picker (VS Code's diff editor) is genuinely faster. No religion; the commands are what you need to know, the UI is interchangeable.
- **Watch-outs:** committing `.env*` — rotation, not deletion; merge commits from `git pull` without `--rebase` — clutter; `git push --force` — overwrites teammates; long-lived feature branches — diverge from `main`, painful merge; one-commit-per-typo PRs — squash-merge collapses, but local hygiene still matters; rebasing a branch others have pulled — rewrites history they have, force-with-lease still hurts them; conventional-commit tooling without changelog automation — ceremony for no payoff; using `git pull --rebase` on `main` itself when `main` was force-pushed — never happens with protection rules but the failure mode is real.

What this lesson does not cover:

- Cherry-pick, bisect, reflog, interactive rebase — lesson 2 of chapter 096.
- Pull request mechanics (review, draft, codeowners) — lesson 3 of chapter 096.
- Branch protection / rulesets — lesson 4 of chapter 096.
- CI on push/PR — Chapter 097.
- Preview deployments per PR — lesson 5 of chapter 098.
- Schema-migration PR cadence — Chapter 099.
- Submodules, subtrees, monorepo tooling — out of scope.
- GitLab/Bitbucket — named once, cut.
- Signed commits (GPG/SSH) — named once, cut; valid posture for sensitive enterprise repos, not minimum-viable.

Estimated student time: 50 to 60 minutes. Concept + Setup; the `pull.rebase` + squash-merge combination is the load-bearing senior diff.

## Lesson 2 — Reflog, bisect, and the rescue toolkit

The four power tools — `reflog` for recovery, `bisect` for regression hunting, `cherry-pick` and `revert` for surgical moves, `rebase -i` for shaping history before push — and the trigger that earns each.

Topics to cover:

- **The senior question.** A bad rebase wiped four hours of work. A teammate's commit broke a test and nobody knows which one. A bug fix lives on a feature branch that won't merge for a week but production needs it now. The lesson lands the four power tools — `reflog` for recovery, `bisect` for blame-by-binary-search, `cherry-pick` for surgical movement, `rebase -i` for shaping history *before* it leaves the branch — with the trigger that earns each.
- **The "history is editable until pushed and merged" rule.** Local commits are mutable — rewrite freely. Pushed-but-not-merged feature branch is mutable with care — your branch only, force-with-lease. Merged to `main` is immutable — never rewrite shared history. The four tools below all operate within "local or feature-branch-only" by default; the one exception (revert) operates on shared history *additively*, never by rewrite.
- **`git reflog` — the safety net.** Records every HEAD movement: commits, checkouts, resets, rebases, merges. When something looks lost — a botched rebase, a `reset --hard` that deleted local work, a force-push that overwrote a branch — the reflog still points at the pre-disaster commit. Recovery: `git reflog` to find the hash, `git reset --hard <hash>` (or `git branch recover <hash>`) to restore. Default retention 90 days for reachable commits, 30 for unreachable. **Teach this before the student needs it.** The trigger to reach for the reflog is "I think I lost work" — and by the time the student thinks that, they need to already know the command exists.
- **`git stash` — the scratchpad.** Working-tree changes that aren't ready to commit but the branch needs to switch. `git stash push -m "msg"` saves, `git stash pop` restores. Reach for it when an emergency context-switch arrives and the current work isn't commit-shaped. Watch-out: stashes pile up silently (`git stash list`); a stash older than a day is usually abandoned work that should be either turned into a commit or dropped.
- **`git bisect` — binary search for the bug.** A test was passing last week; now it fails. Which commit broke it? `git bisect start && git bisect bad HEAD && git bisect good <known-good-sha>`. Git checks out the midpoint; run the test; mark `git bisect good` or `git bisect bad`; repeat. `log(N)` checkouts on N commits — 1000 commits in 10 steps. `git bisect run <command>` automates: bisect runs the command at each step, marks based on exit code. Reach when the bug class is "regression in a known-good baseline" and the change set since is non-trivial.
- **The bisect-friendly main.** Squash-merge from lesson 1 of chapter 096 pays off here. Every commit on `main` is one shipped change; bisect lands on a single PR, not a "wip" commit mid-PR with broken intermediate state. The two disciplines are paired.
- **`git cherry-pick` — surgical commit movement.** Apply one commit from another branch onto the current branch. `git cherry-pick <sha>`. Triggers: a hotfix landed on a feature branch needs to ship now via a small PR; a fix on `main` needs to backport to a release branch (rare in trunk-based, but the case exists); pulling one commit from an abandoned branch. Watch-out: cherry-picking the same commit twice creates two commits with the same content but different hashes — diverging history; usually fine, but tools that track "has this been cherry-picked" rely on commit messages or the `-x` flag (`-x` appends "(cherry picked from commit <sha>)" to the message).
- **`git revert` — the undo-on-shared-history primitive.** A bad commit shipped to `main`. Rewriting history is off the table (others have pulled it). `git revert <sha>` creates a *new* commit that applies the inverse of the original — clean rollback that preserves the audit trail. Reflex: production rolls back via revert; feature-branch mistakes get fixed by `rebase -i` or amend. The lesson 8 of chapter 098 rollback story has two layers — Vercel-promote-previous-deployment for the deploy itself, `git revert` for the code that produced it.
- **`git rebase -i` — interactive rebase, history shaping.** Open the last N commits in an editor with one line per commit and a verb. **`pick`** — keep. **`reword`** — keep, edit message. **`edit`** — stop after applying, let you amend the working tree, continue. **`squash`** — collapse into previous, combine messages. **`fixup`** — collapse into previous, drop message. **`drop`** — remove entirely. **`reorder`** — drag lines. The senior workflow: write commits messily on the feature branch, then `git rebase -i origin/main` before push to clean up. Reach: collapse "fix typo" into the previous commit; reword a confusing message; reorder so the commits read as the logical narrative; split a too-big commit (mark `edit`, `reset HEAD~`, restage in pieces).
- **The `--autosquash` shortcut.** `git commit --fixup=<sha>` creates a commit named `fixup! <original subject>`; `git commit --squash=<sha>` does the same for squash. Then `git rebase -i --autosquash origin/main` reorders and pre-marks them. Reach: a review comment lands; the fix belongs in the original commit, not a "address review" commit. Set `git config --global rebase.autoSquash true` so `-i` always autosquashes.
- **`git commit --amend` — the one-commit revision.** Edits the most recent commit's message or content (stage changes, then `--amend`). The reflex when "I forgot to add a file" or "the message has a typo." Combined with `--no-edit` to amend content without re-opening the editor. Past that: `rebase -i` for the right tool. Watch-out: amending after push requires `--force-with-lease`; same rule as rebase.
- **`git restore` and `git switch` — the post-2.23 split.** Modern Git split `git checkout`'s many jobs. **`git switch <branch>`** changes branch (no file-operation overload). **`git restore <file>`** discards working-tree changes. **`git restore --staged <file>`** unstages. The student can keep using `git checkout`; the senior reflex prefers `switch`/`restore` because the intent is unambiguous. Named once, used in examples.
- **The conflict resolution flow.** Conflict happens during merge, rebase, cherry-pick, revert. Git marks files with `<<<<<<<`/`=======`/`>>>>>>>`. Resolve, `git add <file>`, then `git rebase --continue` (or `merge --continue`, etc.) or `git rebase --abort` to bail. VS Code's three-way conflict editor is faster than the terminal for non-trivial conflicts. `git rerere` (enabled in lesson 1 of chapter 096) records resolutions and replays them on the same conflict in repeated rebases — saves real time on long-lived feature branches.
- **The `git log` invocations a senior actually types.** `git log --oneline -20` (last 20 as one-liners), `git log --graph --oneline --all` (visual graph for "what does the history actually look like"), `git log -p <file>` (changes to one file with diffs), `git log -S "string"` (commits that added/removed a literal string — pickaxe), `git log --author=<name>` (someone's commits). The senior reflex: `git log` answers most "when did this change" questions before `git blame` does.
- **`git blame` — author and commit per line.** `git blame <file>` annotates each line with sha, author, date. VS Code's GitLens or `git blame -L 40,80 <file>` for a slice. Reach: "who/why did this line exist." Pair with `git log -p` against the resulting commit to read the whole change.
- **The "I committed to the wrong branch" rescue.** Common case. Two paths. **Path A** (haven't pushed): `git log` to grab the sha, `git reset --hard HEAD~N` to undo locally, `git checkout -b correct-branch <sha>`, `git cherry-pick <sha>` to apply. **Path B** (pushed but PR not opened): the same, plus `git push --force-with-lease` on the original branch. The reflog is the rescue if a step goes sideways.
- **The recovery muscle memory.** Five recovery scenarios as the lesson's exercise targets. (1) "Lost a commit after `reset --hard`" → reflog → reset. (2) "Wrong branch" → reset + cherry-pick. (3) "Committed a secret" → reflog as evidence, then *rotate the secret* (the file in history is irrelevant once rotated). (4) "Rebase went sideways" → `rebase --abort`; if past abort, reflog. (5) "Tests started failing yesterday and nobody knows when" → `bisect run`.
- **Watch-outs:** rewriting shared history — never on `main`, only on your own feature branches; `git push --force` without `--lease` — overwrites teammates; cherry-picking from `main` to a feature branch instead of rebasing — creates divergent history; `git reset --hard` without staging important work — reflog is the rescue but prevention is better (`git stash` first); bisect with intermediate broken commits — squash-merge prevents this structurally; `rebase -i` on a branch others have pulled — rewrites their history; relying on the reflog for old work — 90 days max, plan accordingly; `git commit --amend` then forgetting to force-push — local and remote diverge silently.

What this lesson does not cover:

- Submodule recovery — out of scope.
- `git filter-repo` / BFG for removing files from history — named once for the secret-leak case, deferred to a security playbook; rotation is the primary fix.
- Worktrees (`git worktree`) — useful for parallel branches, named once, cut.
- Signed commits — named once in lesson 1 of chapter 096, cut here.
- Git internals (objects, refs, packfiles) — useful context for debugging but not minimum-viable.
- Pull request workflow — lesson 3 of chapter 096.

Estimated student time: 50 to 60 minutes. Mechanics + Reference; the reflog-as-safety-net and the bisect-friendly main are the load-bearing reflexes.

## Lesson 3 — The pull request as designed artifact

The small-reviewable-reversible rule, the six-section description template, draft PRs, CODEOWNERS routing, and the fixup-commit-plus-squash-merge review loop.

Topics to cover:

- **The senior question.** The student has been pushing to a personal branch and merging at will. On a team, the PR is *the unit of change* — the artifact reviewers read, CI runs against, the preview deployment binds to, and the squash-merge commit memorializes on `main`. What makes a PR reviewable, what's "small enough," what does the senior write in the description, when does a draft PR earn its weight? The lesson lands the PR as a designed artifact, not a bookkeeping step.
- **The PR lifecycle.** `git push -u origin feat/X` → `gh pr create` (or web UI) → reviewers requested → CI runs (Chapter 097) → preview deployed (lesson 5 of chapter 098) → discussion → push fixups → review approves → squash-merge → branch auto-deleted. Each stage is a checkpoint; the goal is "this lands in one round of review."
- **Small, reviewable, reversible — the three-word rule.** **Small** — under ~400 changed lines is the rough ceiling for a serious review in one sitting; over it, attention drops and review becomes rubber-stamping. **Reviewable** — one logical change, no drive-by formatting, no "and while I was in there." **Reversible** — one `git revert` undoes it cleanly because it's atomic. The three constraints reinforce each other.
- **The "stack of PRs" pattern.** Big change splits into a chain: PR 1 (refactor with no behavior change) → PR 2 (extract helper) → PR 3 (use helper at the call site) → PR 4 (the feature). Each PR small, each reviewable, each merges independently. Tools: `gh pr create --base feat/step-1` chains PRs against each other; Graphite / git-spice / Sapling automate the stack mechanics. Named once; manual chaining works for most SaaS teams.
- **The PR description as the senior's diff explainer.** The diff shows *what*; the description tells reviewers *why*, *what to look at first*, and *how it was verified*. Template: **(1) What** — one-paragraph summary of the user-visible change. **(2) Why** — link to the ticket / decision / Slack thread; the problem this solves. **(3) How** — non-obvious choices in the implementation, alternatives considered. **(4) Test plan** — what the author ran (the seeded test, the manual flow, the migration on a preview branch). **(5) Screenshots / video** — for UI changes; the preview-deployment URL closes this in lesson 5 of chapter 098. **(6) Risks / rollback** — what could break, how to roll back if it does. Repos ship a `.github/pull_request_template.md` (named here, scaffolded in lesson 4 of chapter 096).
- **The "review-yourself first" reflex.** Open the PR's "Files changed" tab and read the diff before assigning reviewers. Half the comments a reviewer would have made come to mind in self-review. Leave inline comments on your own PR for context ("intentionally not handling X here because Y"). The reflex saves a review round and signals care.
- **Draft PRs — when they earn it.** Opened with "Draft" status, can't be merged, signals "feedback wanted, not approval." Reach when the approach is uncertain and an early reviewer can save a day of rework; when the PR depends on another PR not yet merged; when CI is the early-warning signal and reviewers shouldn't read yet. Don't use as a default — non-draft PRs get attention; drafts get ignored.
- **Conversational review — the comment styles.** **Suggestion** (the `suggestion` code-block fence GitHub renders as one-click apply) — concrete diff the author can accept in one click; reach for typos and small refactors. **Question** — "why this not that?"; legitimate when the reviewer doesn't yet have context, not when it's a disguised demand. **Blocking comment** — "this needs to change before approval"; mark with the "Request changes" review action so it's structural. **Nit** — explicit "non-blocking, optional"; signals the reviewer cares about the bar but won't block on style. The author's reflex: respond to every comment, even with "good catch, fixed" or "deferred to follow-up issue #N."
- **The reviewer's job in 60 seconds and 30 minutes.** **60-second pass** — read the description, scan the diff, flag obvious problems (missing tests, missing migration, leaked secrets). **30-minute pass** — re-read with full attention, follow each change to its callers, run the preview deployment, leave the review. The 60-second pass exists so dead-on-arrival PRs don't waste 30 minutes; the 30-minute pass is the actual review.
- **Fixup commits and squash-merge — the everyday loop.** Reviewer leaves a comment. Author makes the fix as `git commit --fixup=<sha>` (lesson 2 of chapter 096) or just `git commit -m "address review"`. Pushes. Reviewer re-reviews the new commit (GitHub's "Compare since last review" view). Approve. Squash-merge collapses the noise. Result: `main` shows one clean commit; the PR shows the conversation for posterity.
- **`gh` — the GitHub CLI.** `gh pr create`, `gh pr view`, `gh pr checkout 123` (check out someone else's PR locally), `gh pr review`, `gh pr merge --squash --delete-branch`. Named: not required (the web UI is fine), but the CLI is faster once muscle memory lands and integrates with shell aliases. The course defaults to the web UI; the CLI gets one line per command.
- **CODEOWNERS — automatic reviewer routing.** `.github/CODEOWNERS` maps file patterns to GitHub usernames or teams. `app/billing/* @team/billing-leads` means changes to billing files auto-request the billing leads' review. Without the protection rule (lesson 4 of chapter 096), it's a suggestion; with `require_code_owner_review` enabled, the PR can't merge without their approval. The structural enforcement comes from the rule; the file is the data. Pattern-match by ownership zones: auth, billing, schema, infra get codeowners; UI surfaces and feature work typically don't.
- **The PR template.** `.github/pull_request_template.md` populates the description box. Senior shape: the six headings above, no required-checkbox theater (a tooling-enforced "I ran tests" checkbox is bypassed in seconds; the CI status check is the structural enforcement). Ships with the repo; one-time setup.
- **Issue templates and labels — named once.** `.github/ISSUE_TEMPLATE/*` for bug reports / feature requests / RFCs; labels for `bug`/`enhancement`/`p0`/`good-first-issue`/`needs-design`. Useful as the team grows past two people; one-line mention. The course doesn't ship templates as a lesson.
- **The merge-vs-squash-vs-rebase button on GitHub.** The repo setting decides which are available; the team default is **Squash and merge only** (other options disabled in repo settings → general → pull requests). One commit per PR on `main`, no exceptions. Disable "Allow merge commits" and "Allow rebase merging" in repo settings — structural enforcement of the squash-merge discipline from lesson 1 of chapter 096.
- **Auto-merge.** GitHub's "Enable auto-merge" queues the PR to merge as soon as all required status checks pass and reviews approve. Reach when the PR is ready, you don't want to wait, and you trust CI. The repo setting "Allow auto-merge" turns it on. Named once.
- **Merge queue, named once.** GitHub's serialized merge queue for high-traffic repos (rebase each PR onto the latest `main` and re-run CI before merging, to prevent landing-broken-main races). Earns weight at higher team sizes than 2026's typical SaaS startup; named, cut at the SaaS-startup minimum.
- **The PR-to-deploy story, forward referenced.** Chapter 097 wires CI to run on every PR. lesson 5 of chapter 098 binds a Vercel preview deployment per PR with its own URL and its own Neon branch. Chapter 099 / chapter 100 ships schema migrations across a series of small PRs. Each chapter assumes the PR-as-unit-of-change discipline this lesson lands.
- **Watch-outs:** PRs over 800 lines — reviewers skim or rubber-stamp; "and while I was in there" drive-by changes — split them; PR description as "see commits" — wastes the reviewer's time; merging your own PR without review on a team — defeats the gate (lesson 4 of chapter 096 enforces); pushing to the PR branch after approval without re-requesting review — silent risk; not deleting branches after merge — stale-branch landfill; assigning four reviewers when one suffices — diffuses responsibility; bypassing CODEOWNERS via repo-admin override — visible in the audit log, breaks trust; using the merge button when squash-merge is the convention — undocumented exception.

What this lesson does not cover:

- Branch protection rules and rulesets — lesson 4 of chapter 096.
- CI status checks as required — Chapter 097.
- Preview deployments per PR — lesson 5 of chapter 098.
- PR review through the principles/patterns lens — Chapter 104.
- AI-assisted review tooling — out of scope.
- Merge-queue mechanics at scale — named once, cut.

Estimated student time: 45 to 55 minutes. Pattern + Decision; the description template and small-reviewable-reversible rule are the load-bearing senior diffs.

## Lesson 4 — Rulesets that enforce the workflow

The six-rule minimum-viable ruleset for `main` — PR required, approvals, stale-approval dismissal, code-owner review, required status checks, linear history — plus bypass actors and the `.github/` policy surface.

Topics to cover:

- **The senior question.** Lessons lesson 1 of chapter 096–lesson 3 of chapter 096 taught the workflow: trunk-based, small PRs, squash-merge. None of it survives one bad day without enforcement. A teammate force-pushes `main` to fix a typo; a Friday-evening fix lands without review; a passing CI configured wrong gets bypassed. What makes the workflow *mechanically* true rather than a team norm everyone hopes holds? GitHub branch protection rules (the legacy mechanism) and rulesets (the 2026 replacement). The lesson lands the minimum rule set every production repo ships with and the senior diff past it.
- **Rulesets vs. branch protection rules — the 2026 picture.** GitHub launched rulesets as the replacement for branch protection rules. Rulesets are more flexible (multiple branch patterns, layered enforcement, organization-level rules, bypass lists with audit trails). Branch protection rules still work; new repos default to rulesets in the UI. The course teaches rulesets; the legacy UI is named once. Both enforce the same primitives; the migration is mechanical.
- **The minimum-viable ruleset for `main`.** Six rules, each with one-line justification:
  - **(1) Require a pull request before merging.** Disables direct pushes to `main`. The structural rule. Without it, everything else is optional discipline.
  - **(2) Require approvals: 1.** One reviewer. Two-person teams: pair across PRs. Solo project: dismiss this rule and accept the trade-off, or use a personal review-self pattern. The "Required reviewer rule" went GA in February 2026, giving granular team-level enforcement on top of CODEOWNERS.
  - **(3) Dismiss stale pull request approvals when new commits are pushed.** A reviewer approved at commit A; the author pushed commits B and C; the approval no longer reflects what merges. Dismissal forces re-review of meaningful changes.
  - **(4) Require review from Code Owners.** Pairs with `.github/CODEOWNERS` from lesson 3 of chapter 096. Without this rule, CODEOWNERS is a suggestion; with it, the named owners' approval is required structurally. Senior diff: review-by-team-name lands in rulesets (GA Feb 2026) when the team grows past per-person ownership.
  - **(5) Require status checks to pass before merging.** The CI gate from Chapter 097. Lint, type-check, test, build — each listed as a required check by name. PRs can't merge until green. **Strict mode** (require branches to be up-to-date before merging) adds the up-to-date-with-`main` constraint — pairs with the rebase workflow but doubles re-runs on busy `main` branches; enable when CI is fast, disable when CI takes 10+ minutes.
  - **(6) Require linear history.** Forbids merge commits on `main`. Pairs with squash-merge from lesson 3 of chapter 096. Without it, a teammate hitting "Create merge commit" anyway pollutes history.
- **Senior-reach rules.** **(7) Require signed commits** — pairs with GPG/SSH commit signing; the audit-grade enforcement; named, optional. **(8) Require deployments to succeed before merging** — the preview deployment from lesson 5 of chapter 098 listed as a required check; structurally requires the preview built. **(9) Restrict who can push to matching branches** — explicit allow-list; rare for trunk-based but useful for release-tagged branches. **(10) Block force pushes** — on by default in rulesets; verify it's set. **(11) Restrict deletions** — verify `main` can't be deleted accidentally. All five named, one-line each.
- **Bypass actors.** Rulesets allow listing specific users / teams / apps that bypass *with audit trail*. Reach for the on-call team handling a true production emergency where the normal flow won't ship in time. The audit log records every bypass — bypass is rare and visible. Default: empty bypass list; named explicitly when a team adds an account.
- **The CODEOWNERS file in depth.** `.github/CODEOWNERS` syntax — patterns from most-general to most-specific (last match wins). Examples: `* @org/eng` (everything defaults to engineering), `/app/billing/ @org/billing-leads`, `/db/schema.ts @org/dba`, `/.github/ @org/platform`. Pattern *not* a regex — gitignore-style globs. Reach when there are clear ownership zones; cut when the team is two people without specialization. Each line is structural enforcement once paired with the rule.
- **Multiple rulesets layered.** Rulesets stack — one general ruleset on all branches (block force-push, require signed commits if the team enables it), a stricter one on `main` (the six rules above), an even stricter one on `release/*` if the team uses release branches. Trunk-based teams need exactly the `main` ruleset; the layering capability is named once.
- **`.github/CODEOWNERS` and protection rule interplay.** Without `require_code_owner_review`, CODEOWNERS auto-requests the listed reviewers but they can be ignored. With it, their approval is required. The two pieces sit in two places (file in repo, rule in settings) — the senior reflex is to verify both when introducing ownership zones.
- **Required status checks — naming the checks.** The required-checks list in the ruleset names checks *by name string*. The CI in Chapter 097 produces job names — `typecheck`, `lint`, `test`, `build`. Each gets listed as required. **Watch-out:** renaming a CI job removes it from the required list silently — the rule's check name no longer matches, the gate disappears. Reflex: when renaming a job, update the ruleset in the same PR; the platform team owns this when the repo grows.
- **Auto-deletion of head branches.** Repo setting: "Automatically delete head branches" after PR merge. Pair with squash-merge; the feature branch is one commit on `main` now, the branch is redundant. Reduces clutter, prevents stale branches that diverge for weeks.
- **The "first push to `main` is direct" exception.** The repo's very first commit (the initial scaffold from Chapter 004) is pushed directly to `main` before any protection exists. Rule lands on commit 2 onward. After that, the only way onto `main` is through a PR. Sequence: initial push → enable ruleset → PR-only from here.
- **The settings-as-code option.** Rulesets can be defined in GitHub UI or as JSON exportable via API; tools like Terraform and Probot can manage them as code. For a SaaS startup repo this is overkill — UI is fine; for an org with dozens of repos, settings-as-code earns its weight. Named, cut at the lesson's scope.
- **The `.github/` directory as the repo's policy surface.** Lives in-repo, version-controlled, reviewed in PRs. Files that land here: `CODEOWNERS`, `pull_request_template.md` (from lesson 3 of chapter 096), `ISSUE_TEMPLATE/*.md`, `workflows/*.yml` (Chapter 097), `dependabot.yml` (lesson 8 of chapter 081 / Chapter 097). The pattern: enforcement *configuration* sits in `.github/` so changes to it go through the same review as code; the enforcement *itself* (the ruleset) sits in repo settings because GitHub doesn't yet auto-apply rule JSON from the repo.
- **Worked example outline (not code).** Setting up a fresh repo's ruleset: open repo → Settings → Rules → Rulesets → New branch ruleset. Target branch `main`. Enforce status: Active. Enable: require PR (with "1 required approval" + "dismiss stale" + "require codeowner"), require linear history, block deletions, block force-pushes, require status checks (list `typecheck` / `lint` / `test` / `build`). Save. Verify by attempting `git push origin main` directly — rejected with "Protected branch update failed."
- **Watch-outs:** missing the required-status-checks list — CI runs but doesn't gate; CODEOWNERS without the rule — suggestion only; bypass-actor with no audit review cadence — defeats the visibility; required reviews of 2 on a two-person team — deadlock; strict-mode required-up-to-date with slow CI — every PR re-runs every time `main` moves; renaming a CI job without updating the rule — gate silently disappears; the initial-direct-push exception accidentally repeated later by an admin — visible in the audit log, fixable; signed-commit rule on a team without signing configured — every push blocked; ruleset disabled for "this one PR" without re-enabling — checked into the team's "ops habit" by an after-action review.

What this lesson does not cover:

- The CI workflow itself — Chapter 097.
- Preview deployments as required check — lesson 5 of chapter 098.
- GPG/SSH commit signing — named, cut.
- Settings-as-code tooling (Terraform, Probot) — named, cut.
- Organization-level rulesets and enterprise policies — named once, cut.
- AI-driven review-bot tooling — out of scope.

Estimated student time: 35 to 45 minutes. Setup/wiring + Pattern; the six-rule baseline is the load-bearing artifact.

## Lesson 5 — Quizz

Top 10 topics to quiz:

- The trunk-based / GitHub Flow workflow — one long-lived `main`, short-lived feature branches, PR-per-change, squash-merge to `main`; why not Git Flow (no `develop`/`release`/`hotfix` branches with preview-per-PR + CI-per-PR + feature flags handling the cases Git Flow was designed for).
- Rebase vs. merge as a team-style decision — rebase locally to keep history linear, squash-merge on PR to collapse internal commits, `git pull --rebase` as the everyday hygiene; when merge-commit earns its weight (multi-commit refactor where each step is structurally meaningful).
- The four objects (commit / branch / staging area / remote) and the staging area as a slicing tool — `git add -p` to commit unrelated changes separately; the senior reflex of one logical change per commit, one logical change per PR.
- The minimum global config — `init.defaultBranch=main`, `pull.rebase=true`, `push.autoSetupRemote=true`, `rebase.autoStash=true`, `rerere.enabled=true`, `rebase.autoSquash=true`; the `--force-with-lease` reflex over `--force`.
- The four recovery tools and when each earns it — `reflog` for "I lost work," `bisect` for "regression in a known-good baseline" with `bisect run` for automation, `cherry-pick` for surgical commit movement (and `revert` as the safe undo on shared history), `rebase -i` (`pick`/`reword`/`edit`/`squash`/`fixup`/`drop`) plus `--autosquash` with `git commit --fixup=<sha>` for shaping history before push.
- The "history is mutable until pushed and merged" rule — local commits free to rewrite, feature branch with `--force-with-lease` only, `main` is immutable (revert, never rewrite); why squash-merge makes `git bisect` trivial.
- Pull request as the unit of change — small/reviewable/reversible (~400 lines), the six-section description (what/why/how/test plan/screenshots/risks), draft PRs when uncertain, fixup commits + squash-merge as the everyday review loop.
- CODEOWNERS as automatic reviewer routing — gitignore-style globs, last-match-wins, suggestion without `require_code_owner_review` rule, structural enforcement with it; pair with team-level rules (GA Feb 2026 in rulesets).
- The minimum-viable `main` ruleset — require PR + 1 approval, dismiss stale approvals on new commits, require code-owner review, require status checks (listed by name string from Chapter 097's CI), require linear history, block force-pushes and deletions; senior-reach rules (signed commits, required deployments, bypass actors with audit).
- Rulesets vs. legacy branch protection rules — rulesets as the 2026 default (layered, org-level, audited bypass), branch protection rules as the legacy mechanism still supported; the `.github/` directory as the policy surface (CODEOWNERS, PR template, workflows, dependabot.yml) reviewed in PRs while the enforcement rule itself lives in repo settings.
