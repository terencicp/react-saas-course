# Chapter 096 — Error monitoring and structured logs

## Chapter framing

Chapter 096 installs the production observability floor: when an exception fires at 3am, the on-call engineer reaches Sentry for the stack trace and Log Drains for the structured request log, joins them on a correlation ID, and reconstructs what happened without redeploying the app. The student arrives with error discipline (Chapter 084) — the user/operator message split, fail-closed authorization, the six error seams — and `error.tsx` / `global-error.tsx` from Chapter 035. What's missing is the wire: how the thrown error gets out of the Vercel function and into a UI a human can query, and how the request-scoped log line that explains the context interleaves with the trace. The chapter cashes in the user/operator split structurally — operator detail flows to Sentry and the log drain, never to the response — and lands the discipline that makes the post-mortem possible.

Threads through every lesson. **Errors and logs are two surfaces of one incident** — the Sentry event and the correlated log lines share a `requestId`, joinable by humans and dashboards. **The minimum viable stack is Sentry plus Vercel Log Drains** — one error-monitoring vendor, one log destination (Axiom as the default for Vercel-native cost and ergonomics; Datadog and Better Stack named once as the conditional alternative). **Structured logging is the discipline, not the library** — JSON lines with a fixed key set, request-scoped child loggers, and a correlation ID injected at the request boundary; `pino` is the default reach in 2026 but the discipline is what travels. **PII and secrets never reach the log destination** — redaction is structural (denylist of keys at the logger, never per-call-site discipline) and audited. **Server-side debugging completes the loop** — when logs and Sentry breadcrumbs aren't enough, the engineer attaches `node --inspect` to the local server, sets breakpoints in the failing server action, and watches the variables the log line couldn't carry. Every lesson ties back to a concrete production incident shape — "the webhook silently 500'd," "the server action returned the generic toast and no one knows why" — so the discipline reads as the answer to a known cost.

## Lesson 1 — Sentry: capture, releases, breadcrumbs

Install Sentry for Next.js with the wizard, wire `onRequestError` and manual `captureException` in handled-error seams, tag events with releases and user context, and add domain breadcrumbs.

Topics to cover:

- **The senior question.** A server action throws in production. The user sees the generic toast from the operator/user split. Without instrumentation, the operator sees nothing — the function logs go to Vercel's function output, the stack trace is minified, and finding "this error happened to user X at 14:32 on the Pro plan billing flow" requires grepping a tab. Sentry's value is the inversion: the error shows up grouped by fingerprint, with a readable stack trace, the breadcrumbs that led to it, the release it shipped in, and the user/org tags. The lesson installs Sentry as the error-monitoring default for the 2026 Next.js stack.
- **Why Sentry over alternatives.** Default. Alternatives named once: Bugsnag, Rollbar, Highlight (open-core), Honeybadger. Trigger to flip: none for the course's target SaaS. Sentry's Next.js SDK is first-party-quality, integrates with the App Router's `instrumentation.ts` hook, and folds in release tracking and source-map upload in one wizard. The integrated session replay (separate lesson — chapter 097 covers PostHog's session replay; Sentry's Replay is the alternative).
- **Setup — the wizard, what it writes.** `npx @sentry/wizard@latest -i nextjs` creates `instrumentation-client.ts`, `instrumentation.ts` (server + edge), wraps `next.config.ts` in `withSentryConfig`, adds the auth-token env, and adds the upload step to the build. Walk through every file it touches, name what's load-bearing (the `onRequestError` export from `instrumentation.ts`, the `Sentry.init` server-side, the wrapped config), and what's optional (the example page, the tunnel route — keep the tunnel route, ad-blockers eat the default endpoint).
- **`instrumentation.ts` and `onRequestError`.** The Next.js 16 hook that fires for unhandled errors in server components, route handlers, and server actions. Sentry's wizard wires `Sentry.captureRequestError` as `onRequestError`. Without this hook, server-side throws caught by Next.js's framework boundary don't reach Sentry. The lesson lands the hook explicitly so the student doesn't mistake the wizard's wiring for magic.
- **Manual capture for caught errors.** The `authedAction` wrapper from prior chapters catches the throw to return the user-safe `Result.err`. The error never reaches `onRequestError`. The reach: `Sentry.captureException(err, { tags: { seam: 'authedAction', actionName }, user: { id: userId, org: orgId } })` *inside* the catch, before returning the user-safe result. Every catch-and-handle seam calls `captureException`; every uncaught throw rides `onRequestError`. The two-path rule for Sentry.
- **`withServerActionInstrumentation` for spans.** Sentry's wrapper for Server Actions; adds a span around the action call for performance tracing and groups errors by action name. Reach for it on actions that need performance traces; basic error capture works without it. Named once.
- **Source maps and release tagging.** Production code is minified; the raw stack trace reads `Function.t [as h] (chunk-abc123.js:1:42)`. Sentry's source-map upload step (run during `next build`) rebinds the trace to original source. The release name (commit SHA via Vercel's `VERCEL_GIT_COMMIT_SHA` env) tags every event so a regression maps to the deploy that introduced it. The wizard wires both; verify the auth token, the `org` and `project` slugs in `withSentryConfig`, and the `release` derivation from the env. CI uploads source maps; local dev does not (no value, slow build).
- **Releases and deploy notifications.** Sentry's "release" links events to commits and authors. With the Vercel integration, each deploy notifies Sentry. "First seen in release X by author Y" becomes the diagnostic anchor for regressions. The student wires the Sentry-Vercel integration once.
- **Breadcrumbs — what they are and what to add.** Sentry auto-captures breadcrumbs: navigation, fetch, console logs, UI clicks (via the `replayIntegration` if enabled). Custom breadcrumbs add domain context: `Sentry.addBreadcrumb({ category: 'invoice', message: 'Loaded invoice for billing', data: { invoiceId } })`. Reach for it where the failure context isn't obvious from the stack — webhook handlers (the event ID and event type), background jobs (the job ID, the input shape), multi-step server actions (which step failed). Breadcrumbs are *not* logs — they're per-event context, capped, dropped after the event fires.
- **Setting user and tag context.** `Sentry.setUser({ id, email, orgId })` after auth in the request handler (or via Sentry's scope inside the action). Sentry groups errors per user, lets the operator filter to "all errors for user X this week," and feeds the BUG SLA dashboards. **Email is operator-safe context, not PII to redact** — the user/operator split (lesson 2 of chapter 084) puts internal IDs and emails on the operator side. Custom tags (`tag.plan = 'pro'`, `tag.feature = 'invoicing'`) earn weight for filtering. Don't put secrets in tags; don't put high-cardinality values in tags (use `extra` or `context` for one-off data).
- **`Sentry.beforeSend` for redaction.** A last-line hook that runs on every event before send. Reach for it to strip PII the integration might capture (request bodies on transactions, query strings with tokens). The course's reach: strip `password`, `token`, `apiKey`, `Authorization` from event request data; let user emails and IDs through (operator side). Same redaction posture as the logger (lesson 3 of chapter 096).
- **The `error.tsx` and `global-error.tsx` boundaries — Sentry's reach.** The boundaries from Chapter 035 receive the `error` prop; the Sentry wizard imports `Sentry.captureException(error)` in `global-error.tsx`'s `useEffect`. `error.tsx` segments need the same wiring (the wizard doesn't add it). Pair with the `digest` Next.js passes to the boundary; `digest` is Sentry's correlation anchor for grouped server-side errors.
- **Client vs. server SDK.** `instrumentation-client.ts` (the client SDK), `instrumentation.ts` (server + edge). Separate DSNs not needed; both share one DSN. Client SDK auto-captures unhandled rejections, console errors with `captureConsoleIntegration`, and (optionally) Replay. Server SDK captures throws via `onRequestError` plus manual `captureException`. The two SDKs share the release tag so events from both sides on the same deploy group.
- **Sampling — `tracesSampleRate` and `replaysSessionSampleRate`.** Traces and replays cost more than error events. Default: 0% traces and 0% replays in the chapter's wiring; 100% errors. Reach for sample rates when performance traces earn weight (Chapter 098 references); session replay is owned by lesson 3 of chapter 097 (PostHog). Don't ship Sentry Replay alongside PostHog Replay — pick one.
- **Cost shape and quota discipline.** Sentry's free tier is ~5k errors / month; the team plan starts where it earns. Quota-bust shapes: a tight loop calling `captureException`, a webhook signature mismatch firing on every retry, a missing `tags` field producing high-cardinality groups. The reach: deduplicate via fingerprinting (`Sentry.withScope(scope => scope.setFingerprint([...]))`) on known-noisy errors and `Sentry.captureException(err, { level: 'warning' })` for non-actionable but trackable events.
- **Verifying the wiring.** Throw a deliberate error from a server action and from a client component; check that both events arrive grouped, with readable stack traces, the correct release tag, and the user/org context. The verification step ships in the project (Chapter 099) but the lesson asserts the shape so the student isn't relying on "the wizard said it worked."
- **Watch-outs.** Forgetting `onRequestError` so framework-boundary throws never reach Sentry (the most common omission); forgetting `captureException` inside `authedAction` catches so handled errors silently disappear; missing source maps in production (build step skipped — release tagging works but stack traces are minified); using `Sentry.captureMessage` for what should be a log line — `captureException` is for thrown errors with stack traces; high-cardinality tags blowing out the quota; the `digest` not threaded from `error.tsx`, so server-side group fingerprinting misses; setting `NEXT_PUBLIC_SENTRY_DSN` as a separate "client" DSN when one DSN covers both — extra config to maintain; the wizard's example page left in production; ad-blocker-eaten events in client SDK fixed by the tunnel route — leave it on.

What this lesson does not cover:

- The structured-log discipline and correlation IDs — lesson 2 of chapter 096.
- The PII/secret exclusion rule applied to logs — lesson 3 of chapter 096.
- Log destinations and Vercel Log Drains — lesson 4 of chapter 096.
- VS Code debugger attachment — lesson 5 of chapter 096.
- PostHog and session replay — Chapter 097.
- Performance traces at depth — Chapter 098.
- The user/operator message split itself — lesson 2 of chapter 084 owns it.
- Error fingerprinting / deduplication beyond the one-line mention — out of scope.

Estimated student time: 50 to 60 minutes. The chapter's largest lesson; the wizard walkthrough plus the manual-capture pattern carries weight.

## Lesson 2 — Structured logs with correlation IDs

Set up `pino` with a fixed JSON key set, thread a per-request `requestId` through AsyncLocalStorage, and emit child loggers from every server-side seam.

Topics to cover:

- **The senior question.** A webhook handler 500s for a single org. Sentry shows the stack trace and the org ID tag. The operator wants to know: what was the last successful step before the throw, what was the rate-limit headroom at the time, what was the upstream API response. The reach is the request-scoped log — every server-side operation emits one or more JSON log lines tagged with a `requestId`, and the operator filters the log destination by that ID. The lesson installs the structured-logging discipline.
- **Why structured (JSON), not strings.** A string log line is grep-only; a JSON line is queryable (`level:error AND org_id:"org_123" AND duration_ms:>1000`). The destination (lesson 4 of chapter 096) treats every key as an index. The discipline: every log line is one JSON object with a fixed key set; humans never write `console.log('user ' + userId + ' did X')` in server code after this lesson.
- **The fixed key set.** Every log line carries: `level` (debug / info / warn / error), `time` (ISO 8601), `msg` (a short human-readable phrase, low-cardinality), `requestId`, `userId?`, `orgId?`, `service` (e.g. `'app'`, `'worker'`), `env` (production / preview / development). Domain-specific keys append (`invoiceId`, `webhookEventType`, `durationMs`). The fixed set makes dashboards consistent across services and ages.
- **`pino` as the default.** Default. The 2026 Node logger of record: fast (asynchronous, structured), small, has the child-logger API that makes correlation IDs ergonomic. Alternatives named once: `winston` (heavier, slower, traditional), `bunyan` (legacy), `consola` (DX-focused, not production-shaped). Trigger to flip: none for the course. Set the default level via `LOG_LEVEL` env (`info` in production, `debug` in development).
- **The `pino` setup file — `lib/logger.ts`.** Exports a `logger` singleton with `base: { service, env, release }` so every line carries those keys for free. Production: default JSON output. Development: `pino-pretty` via env-gated import for readable lines. The base also carries `release: process.env.VERCEL_GIT_COMMIT_SHA` so logs join Sentry events on release tag.
- **Vercel and `pino` worker threads — the known footgun.** `pino`'s transport runs in a worker thread for performance; serverless functions on Vercel terminate workers between invocations, which breaks transports on cold paths. The reach: use `pino`'s synchronous output to `stdout` in production (the default when no transport is configured), let Vercel capture `stdout`, let the log drain (lesson 4 of chapter 096) ship it on. Transports for dev only (`pino-pretty`). Name the trap once so the student doesn't paste a transport from a blog post.
- **The correlation ID — `requestId`.** Every server-rendered request and every server action invocation gets a `requestId` (a ULID or UUID generated at the entry seam). All log lines emitted during that request carry it. Sentry events for the request carry the same ID as a tag. When the operator finds a Sentry event, they pivot to the log destination filtered by `requestId` and read the per-request narrative.
- **Where the requestId comes from.** Three options. (1) Read the incoming `x-request-id` header if present (Vercel's edge sets one; some reverse proxies add it). (2) Generate via `crypto.randomUUID()` or `ulid()` at the entry seam if absent. (3) Surface as `x-request-id` on the response so the client can include it in error reports. Default: read-or-generate at the entry seam, attach to the AsyncLocalStorage context, echo on response.
- **Threading the requestId — AsyncLocalStorage.** The reach: `import { AsyncLocalStorage } from 'node:async_hooks'`; a `RequestContext` ALS instance stores `{ requestId, userId, orgId }`. The entry seam (`proxy.ts` for routes, the `authedAction` wrapper for server actions) calls `als.run({ requestId, ... }, () => handler())`. Anywhere in the call stack, `als.getStore()?.requestId` returns it. The `logger.child(als.getStore())` returns a child logger with the request keys pre-bound — every log line inside the request carries the IDs without per-call-site discipline. Same pattern as the request-scoped context from prior chapters (audit log enrichment in lesson 3 of chapter 085).
- **The `proxy.ts` and `authedAction` seams.** Two entry seams. (1) `proxy.ts` runs for every server-rendered route and route handler; wrap the response in `als.run({...}, () => next())`. (2) `authedAction` (the wrapper from Chapter 7) runs for every server action; same wrap. Server components running outside both seams (rare — they fire under proxy) need no special handling. The discipline: there are two seams and they're owned.
- **Child loggers per scope.** `const log = baseLogger.child({ webhookEventId: event.id, stripeEventType: event.type })` inside a webhook handler. All subsequent `log.info(...)` calls carry those keys plus the request IDs. Cheap; idiomatic in `pino`. Pattern reads as scope-narrowing — the file's logger knows what the file is doing.
- **Log levels and what each means.** `debug` — high-volume tracing, off in production. `info` — significant successful events (signed in, webhook processed, job completed). `warn` — recoverable issues (cache miss when cache hit was expected, retry triggered, fail-open carve-out fired). `error` — handled or unhandled exceptions, with stack trace; Sentry sees these too. The discipline: every log line has the right level; `info` is not a synonym for `console.log`. CI greps for `console.log` in server code and fails the build.
- **Threading the log into Sentry — `Sentry.addBreadcrumb`.** When a log line carries diagnostic value for a possible upcoming error, add it as a breadcrumb in the same call. The reach: a `log.info({ webhookEventType, eventId }, 'webhook received')` also calls `Sentry.addBreadcrumb({ category: 'webhook', message: 'webhook received', data: { ... } })`. Wrap in a `logWithBreadcrumb` helper to avoid two call sites for one event. Logs are persistent; breadcrumbs are per-event context.
- **The console.log retirement.** After this lesson, `console.log` in server code is a lint failure (`no-console` with a server-files glob). Frontend client code keeps `console.error` for browser DevTools (matched by the client SDK's `captureConsoleIntegration` from lesson 1 of chapter 096). Server logs go through `logger`. Tests can keep `console.log`; the lint rule excludes them.
- **The cost of cardinality.** A log destination indexes every key. A `userId` field is high-cardinality but bounded by user count; a `requestId` is high-cardinality and ephemeral (TTL drops it). Don't index free-text in keys (`{ note: 'user clicked the green button at 2pm wearing a hat' }`); index structured values, put narrative in `msg`. Watch for accidentally indexing stack traces — the `error` key holds a structured `{ name, message, stack }` object, the stack is the long string.
- **The `Error` serializer.** `pino`'s default `Error` serializer (`pino.stdSerializers.err`) renders `{ type, message, stack, cause }`. Configure it on the logger so `log.error({ err }, '...')` produces structured error fields. With the cause-chain discipline from Chapter 012, the serializer's `cause` recursion preserves the wrap-and-throw chain in the log.
- **Watch-outs.** `console.log` left in server code falls through to Vercel's plain stdout and bypasses the structured shape; missing `requestId` on a log line means the line can't be joined back to the request; logging via the base `logger` instead of the child poisons cardinality (the request keys never appear); `pino` transports in production break on Vercel — synchronous stdout only; logging large objects (whole DB rows) blows out the log volume and cost; logging `Error.stack` as a string field instead of via the `err` serializer fragments the search index; high-cardinality keys (per-request `referer` URL) make the destination's UI slow; logging in a hot loop without sampling buries the signal — log once per batch with a count; the AsyncLocalStorage context leaking between invocations is a worker reuse bug — confirm by setting the ALS at the entry, never module-scope.

What this lesson does not cover:

- The "log inputs at 3am" rule and PII/secrets exclusion — lesson 3 of chapter 096.
- Shipping logs to a destination — lesson 4 of chapter 096.
- Sentry's setup and capture surface — lesson 1 of chapter 096.
- The audit log (an append-only domain event store, separate from operational logs) — lesson 3 of chapter 085.
- OpenTelemetry traces — out of scope for this chapter; named once as the alternative path.
- Frontend logging beyond breadcrumbs — out of scope.

Estimated student time: 45 to 55 minutes. Mechanics-and-pattern lesson; the `logger.child(als.getStore())` shape recurs across every server-side file.

## Lesson 3 — The 3am rule and PII exclusion

Decide what each seam logs for incident reconstruction and enforce a structural redaction config that keeps passwords, tokens, headers, and GDPR-class PII out of the log stream.

Topics to cover:

- **The senior question.** It's 3am. A pager fires. The operator opens the log destination filtered by the `requestId` from the Sentry event. What do they need to see to diagnose without reproducing? The answer drives what gets logged. The lesson installs two complementary rules: **log what an operator at 3am would want; exclude what a regulator at 9am would object to.**
- **Rule 1 — Inputs you'd want at 3am.** For every meaningful operation, log the inputs and outcome a remote operator would need to reconstruct the decision. The reach: log the *what* and the *which*, not the *how*. Examples by seam. (a) Server action: action name, validated input shape (not raw, see redaction), userId, orgId, outcome (`ok` / error code), duration. (b) Webhook handler: event type, event ID, signature-verified, idempotency hit/miss, downstream effect summary. (c) Background job: job ID, input keys (`invoiceId`, `userId`), result code, retry count. (d) External API call: endpoint, status code, duration, retry attempt. The shape: one structured `info` line per successful operation; one structured `error` line per failure with the cause chain.
- **Rule 2 — What never reaches the log.** The exclusion list, named explicitly. **Passwords** (plaintext or hashed — neither belongs in logs). **API keys, tokens, secrets** (Stripe keys, OAuth tokens, JWTs, session cookies, signed URLs, API tokens). **Full request bodies** (may contain passwords or PII). **Full Authorization / Cookie headers**. **PII per GDPR** — full name, postal address, phone, IP address (logged as the last octet zeroed or via a hashed token), full credit card or bank account, government IDs, date of birth, geolocation precise enough to identify. **Sensitive categories per GDPR Art. 9** — health, religion, ethnicity, political opinion, sexual orientation, biometric. **What's safe** — user ID, email (operator-side identifier in this codebase, per lesson 2 of chapter 084's user/operator split), org ID, plan, role; non-sensitive request shape (the validated Zod payload with sensitive fields stripped).
- **The split: operator-side vs. user-side identifiers.** The course's anchor is lesson 2 of chapter 084. Internal IDs and email addresses live on the operator side — they're necessary for support, fraud investigation, and incident correlation. Names, postal addresses, phone numbers live on the user side — not load-bearing for diagnosis. The split decides what the logger emits; redaction enforces the boundary on the fields that slip through.
- **Structural redaction — `pino`'s `redact` option.** The reach: configure `redact: { paths: ['*.password', '*.token', 'req.headers.authorization', 'req.headers.cookie', '*.apiKey', 'user.fullName', 'user.phone', '*.creditCard'], censor: '[REDACTED]' }`. Redaction runs at serialize time, before the line leaves the process. The path syntax supports wildcards (`*`) and deep paths (`a.b.c`). Set once, applies everywhere. Auditable: the redact config is a file in the repo, reviewed in PR, tested with a fixture log line.
- **Why structural over per-call-site.** Per-call-site redaction (`log.info({ user: { id: user.id /* skip name */ } })`) relies on every developer remembering, every code review catching. Structural redaction is a single file the security review reads; new fields added to logged objects are redacted by name without code edits. The senior posture: redaction is the logger's job, not the caller's. Backstop the structural redaction with a CI check that grep-fails specific patterns (literal `password`, `secret`, `_key:` followed by a value) in committed code.
- **Request-body logging — the bounded shape.** When logging a request body is necessary (a Zod validation failure where the operator needs to see the input shape), log the *validated and redacted* output, never `req.body` raw. The Zod parse already enforces the shape; the result has known keys; the redact config strips sensitive ones. Don't log `req.body` before parse — it can carry anything an attacker injects.
- **Errors and stack traces — what's safe.** The `Error.stack` is operator-side (file paths and line numbers, no user data). `Error.message` may carry user input concatenated (`Could not find user with email alice@example.com`); the user/operator split says the message is operator-side, so it logs. *But* — don't include raw user-supplied strings in `message` if the message reaches surfaces beyond the log (the audit log, the user-facing toast). Constants for error messages, structured fields for context: `new NotFoundError('user', { email })`, not `new Error('Could not find user with email ' + email)`.
- **IP addresses — the GDPR-specific reach.** Under GDPR an IP is personal data. Log it for operator diagnosis (rate-limit context, geographic anomaly detection) but the EU-favored discipline is: log the last octet zeroed (`192.168.1.0`) for routine info, log full IP for security events (sign-in failures, rate-limit breaches) under the legitimate-interest basis tied to security. Set retention shorter than other operational logs (30 days vs 90). The audit-log chapter (lesson 3 of chapter 085) covers retention; this lesson names the IP-specific gotcha.
- **The "log everything" anti-pattern.** A common reach is `log.info({ ctx }, 'entering handler')` and `log.info({ ctx }, 'leaving handler')` on every function. The cost (volume, money, signal-to-noise) outpaces the value. The discipline: log the request entry, log significant decisions inside (cache miss, rate-limit hit, branch taken), log the outcome. Function entry/exit pairs add no signal; the request's `requestId` ties the lines together.
- **The sampling threshold.** A read-heavy endpoint at 1000 req/s producing one info line each costs more than its diagnostic value. Drop to `debug` (off in production) for hot read paths; keep `info` for state-changing operations and errors. Trigger to flip: when the log volume on a single endpoint dominates the destination's bill, sample (1-in-N info lines, all errors).
- **Auditing the rule — a quick exercise.** A worked example walks through a sign-in flow's log lines. The student identifies: which fields are redacted, which keys are safe, which keys violate the rule. The pattern lands by review, not by reading.
- **The recurrence on the audit log.** The audit log (lesson 3 of chapter 085) carries domain events for compliance, with its own PII discipline (also redacted, retained per legal basis, append-only). The operational logger in this chapter and the audit log are two streams with overlapping but different rules; the audit log preserves payloads the operational log doesn't. Cross-reference, don't re-cover.
- **Watch-outs.** Logging `req.headers` whole, then assuming `authorization` is redacted by name — case-sensitive paths trip on `Authorization`; logging `error` objects via JSON.stringify drops `Error.message` and `Error.stack` (non-enumerable); logging Zod errors with the full input payload leaks PII — log `error.issues` (paths and codes) without the input; logging a Stripe webhook event raw includes customer email, address, and last-4 — log `event.id`, `event.type`, the customer ID, and the redacted summary; assuming `requestId` is enough for compliance — the destination's retention governs the rule, not the requestId; redact config drift (a new field added, redact not updated) is a known cost — pair adds with redact PR review; logging `JSON.stringify(err)` instead of `{ err }` and the serializer; sending PII to the log destination then asking the vendor to delete it — the deletion has limits, prevention is cheaper.

What this lesson does not cover:

- The logger setup and child-logger shape — lesson 2 of chapter 096.
- Log destinations and drains — lesson 4 of chapter 096.
- The audit log (append-only domain events for compliance) — lesson 3 of chapter 085.
- GDPR retention timers and deletion — lesson 4 of chapter 085.
- The user/operator message split itself — lesson 2 of chapter 084.
- The secrets management baseline (preventing secrets in env / code) — lesson 6 of chapter 085.

Estimated student time: 35 to 45 minutes. Pattern-and-decision lesson; the redact config plus the rule together carry the chapter's compliance posture.

## Lesson 4 — Shipping logs with Vercel Drains

Wire a Vercel Drain to Axiom, verify indexed fields, and walk the Sentry-to-logs pivot that resolves a real production incident by `requestId`.

Topics to cover:

- **The senior question.** The structured log lines from lesson 2 of chapter 096 currently land in Vercel's function logs UI — searchable by free text for the most recent hour, not retained beyond a short window, no joins, no dashboards. The reach is the **log drain**: a one-way fan-out from Vercel's runtime to a long-lived, queryable destination where the operator opens a dashboard, filters by `requestId`, joins to Sentry by tag, and sees a week of context. The lesson installs Vercel Drains.
- **Vercel Drains in 2026.** Renamed from "Log Drains" to "Drains" — the same primitive now also carries traces and Speed Insights. For this lesson, drains forward structured `stdout` JSON lines to a destination over HTTP. Plan requirement: Pro or Enterprise (the course assumes Pro for the production scenario).
- **The destination choice.** **Axiom** — default for this course. Vercel-native marketplace integration, automatic drain setup, generous free tier (500 GB/month ingest), schema-on-read JSON, dataset-per-environment, good UI. **Better Stack (formerly Logtail)** — solid alternative, similar shape, smaller free tier. **Datadog** — reach for it when the team already runs Datadog for infra; pricier, heavier UI. **Logflare / Grafana Cloud Loki** — open-source-friendly; more setup. Default to Axiom; name the alternatives in one line.
- **Why not the Vercel UI alone.** Vercel's built-in log viewer is for "the last hour, eyeballed" — retention is short, query is grep, joins don't exist. Drains exist because the UI doesn't replace a real logging tool. Name the threshold: once the team is paged once a week, the UI is no longer the destination.
- **Setup — installing the Axiom integration.** From the Vercel marketplace: install Axiom, authorize, pick the project, pick the dataset. The integration creates the drain automatically (the manual HTTP drain setup is the conditional path). The student walks the install, names what each step writes (the `AXIOM_TOKEN` and `AXIOM_DATASET` envs aren't required for the integration path — Axiom's ingest is auth'd via the integration).
- **The manual drain alternative.** When the destination isn't on the marketplace: configure a drain in Vercel project settings → Drains → New Drain → "Logs" → JSON → destination URL → optional auth header. The destination receives a POST per batch of structured lines. Reach for the manual path only for custom destinations.
- **JSON drain format and discovery.** Vercel batches stdout lines, wraps each in a payload (with `timestamp`, `requestId`, `projectId`, `deploymentId`, `host`, plus the raw `message` which is the pino JSON). The destination indexes the wrapped fields and the inner JSON. Axiom auto-parses; Datadog needs a pipeline; Better Stack auto-parses. Verify in the destination that `level`, `requestId`, `orgId` are indexed as top-level fields.
- **The first useful query.** "Errors for org X in the last 24 hours." `level:error AND orgId:"org_123" AND _time > now-24h`. The result feeds the on-call narrative. Pin the query as a saved view; build the dashboard in Chapter 099's project.
- **The Sentry-to-Logs pivot.** A Sentry event carries the `requestId` as a tag (the logger and Sentry's scope share the ALS context, per lesson 2 of chapter 096). The on-call opens the Sentry event, copies the `requestId`, opens the drain destination, filters by `requestId`. Three clicks from "the user-facing error message" to "the per-request narrative." Sentry can deep-link to the destination via the Sentry-Axiom integration; for Datadog and Better Stack, the link is manual but the workflow is the same.
- **Retention and cost shape.** Free tier covers the course's traffic. Retention defaults: Axiom 30 days, Better Stack 30 days, Datadog plan-dependent. The chapter's defaults assume 30 days for operational logs; the audit log (lesson 3 of chapter 085) is retained separately per legal basis (longer, different store).
- **Reading production logs in anger — the post-incident drill.** A modeled drill: a webhook handler 500s for a single org for 3 hours starting at 02:00 UTC. Sentry shows 47 grouped events. The on-call workflow: (1) open the Sentry event, copy a sample `requestId`; (2) open the drain destination, filter by `requestId` — read the per-request narrative; (3) widen to `orgId` over the time window — confirm the scope (one org, not all); (4) read the `info` lines leading up to the error — find the divergence (the org's webhook signing secret was rotated upstream, the rotation wasn't applied locally); (5) note the fix path. The workflow lands by walking it, not by listing it abstractly. This drill recurs in chapter 099 with a seeded failure.
- **The Vercel-native checks that don't need a drain.** Build logs, deploy events, function-duration p95 spikes — Vercel surfaces these in its own UI. Don't drain everything; drain what the production debugging workflow needs. The drain carries application JSON; the platform stays platform-native.
- **The "Vercel + Sentry + drain" minimum viable observability stack.** Sentry for errors with stack traces; the drain destination for structured per-request logs; Vercel's UI for platform metadata. Three tools, one workflow. Anything beyond — APM, full-text traces, custom dashboards — earns its weight against this baseline.
- **Watch-outs.** Drains are per-environment (production vs. preview vs. development) — production-only is the default, preview drains add noise; high-volume logging without retention tuning bills more than the team expects; the destination's free tier resets monthly — a single bad deploy can burn it; the drain doesn't replay history (lines emitted before the drain was wired are lost); ad-blocked client logs never reach the destination — the drain is server-side stdout, the client's log path is via Sentry (browser SDK); the destination's parser may not auto-extract nested JSON — verify `level`, `requestId`, `orgId` are indexed; relying on the drain for compliance audit (it's operational, not the audit log); Vercel's stdout buffer can drop lines on function timeout — for critical events, the audit log's transactional write is the durable path, not stdout.

What this lesson does not cover:

- The logger setup and structured-log shape — lesson 2 of chapter 096.
- The PII redaction rule applied at the logger — lesson 3 of chapter 096.
- Sentry's setup — lesson 1 of chapter 096.
- The audit log's separate retention and store — lesson 3 of chapter 085 / lesson 4 of chapter 085.
- Vercel Speed Insights / Web Analytics drains — Chapter 097 / chapter 098.
- OpenTelemetry traces — out of scope.
- Custom drain destinations — named once.

Estimated student time: 35 to 45 minutes. Setup-and-walkthrough lesson; the post-incident drill is the load-bearing exercise.

## Lesson 5 — Server-side debugging with the inspector

Launch `next dev --inspect`, attach VS Code, and use breakpoints, conditional breakpoints, and logpoints to resolve a failing server action that logs and Sentry alone couldn't crack.

Topics to cover:

- **The senior question.** A server action returns a generic error. Sentry shows the throw, the log lines show the inputs and the cause chain, but the operator can't tell *why* the validation predicate returned false for an input that looked correct. The next move is a debugger — attach to the local server, set a breakpoint on the line, run the request, inspect the local variables and the call stack live. Sentry plus logs cover 90% of incidents; the debugger covers the rest, and the senior knows when to reach for which.
- **The decision: when the debugger earns its weight.** Logs and Sentry breadcrumbs are *retroactive* and *static* — what was captured is what you see. The debugger is *interactive* — read every local, every closure, every property of every object, mutate state and resume. Reach for it when: (1) the bug doesn't reproduce in logs (the state is on the wire, not in a logged variable), (2) the call stack reads as "something inside this library does something we don't expect," (3) the failure is a heisenbug that disappears when you add logging. Don't reach for it: (1) for known bugs with clear reproductions (add the assertion / test), (2) for production debugging (use logs / Sentry), (3) for performance work (use profiling tools — Chapter 098).
- **`node --inspect` — the protocol.** Node's built-in V8 inspector listens on a port (default `127.0.0.1:9229`) and speaks the Chrome DevTools Protocol. Any client (VS Code, Chrome DevTools, WebStorm) can attach over the protocol. `--inspect-brk` pauses on the first line; `--inspect` runs but is attachable. Production Node never runs with `--inspect`; dev only.
- **Starting Next.js with the inspector.** Two reach points. (1) `next dev --inspect` — the integrated flag (Next.js 15+) passes the inspector through to the dev server's Node process. (2) `NODE_OPTIONS='--inspect' next dev` — the env-var alternative for setups that wrap the dev command. Verify the inspector is listening (Node prints "Debugger listening on ws://127.0.0.1:9229/...").
- **The Turbopack interaction — the known footgun.** Turbopack (the default bundler in Next.js 16 dev) historically interfered with source maps for server-side breakpoints, leaving "unbound breakpoint" markers in VS Code. The May 2026 status: improved but not universally smooth; if a breakpoint shows as unbound, the workaround is `next dev --webpack` (the fallback bundler) for the debugging session. Name the trap; verify against current Next.js docs before claiming smoothness.
- **VS Code launch config — `launch.json`.** Three configurations: "Next.js: debug server-side" (attaches to `--inspect`'s port), "Next.js: debug client-side" (launches Chrome with debugging on the dev port), "Next.js: debug full stack" (compound — both at once). The lesson ships a `launch.json` snippet: `{ "type": "node", "request": "attach", "name": "Next.js: server", "port": 9229, "skipFiles": ["<node_internals>/**", "**/node_modules/**"] }`. `skipFiles` keeps step-into shallow.
- **Setting breakpoints in server code.** Click the gutter in VS Code on a line inside a server action or a `/lib` helper. The marker is red when bound, gray ("unbound") when source-mapped resolution failed. Run the request that triggers the code. Execution pauses; VS Code highlights the line, populates Variables, Watch, Call Stack, and the Debug Console.
- **The Debug Console.** A REPL bound to the paused frame's scope. Evaluate expressions in context: `user.role`, `await db.query.invoices.findFirst({ where: eq(invoices.id, 'inv_123') })`. Mutate state and resume to test hypotheses. The Debug Console is faster than adding a log line and restarting.
- **Conditional breakpoints.** Right-click a breakpoint, set a condition (`user.id === 'usr_problem'`, `attempt > 3`). The breakpoint fires only when the condition matches. Reach for it when the bug is intermittent or per-user; avoids stepping through every request to hit the one that matters.
- **Logpoints — the non-pausing alternative.** A breakpoint that prints to the Debug Console instead of pausing. `Logpoint message: input was {input}, user is {user.id}`. Equivalent to adding a log line without restarting the server. Reach for it for quick-add diagnostics on hot paths where pausing breaks the flow.
- **The server-action-failed workflow.** Modeled end-to-end. The user clicks a button, the toast shows the generic error. The student: (1) opens Sentry, reads the stack trace, identifies the failing server action and the line; (2) opens the log destination, filters by `requestId`, reads the per-request narrative — the failure is in a validation predicate that should pass for this input; (3) starts the dev server with `--inspect`, attaches VS Code, sets a breakpoint on the predicate's `return false` line, replays the same input locally; (4) the breakpoint hits, the Variables panel reveals the closure captured the wrong tenant scope; (5) the fix is one line in `/lib`. The workflow ties together all four prior lessons.
- **How logs and Sentry breadcrumbs interleave with the live debugger.** The breadcrumbs that fired before the breakpoint are visible in the Sentry UI; the log lines for the request are in the destination; the debugger sees the live state. The senior reads Sentry's stack, *then* the log narrative, *then* attaches the debugger. Each surface answers a different question: Sentry — "what threw," logs — "what happened," debugger — "what's in scope right now."
- **Chrome DevTools as the alternative client.** `chrome://inspect` shows the Node process; click "inspect" — Chrome DevTools attaches. Equivalent surface to VS Code; reach for it when stepping through code without an editor open. Same protocol; same source maps; same Turbopack caveat.
- **Source maps and the debugger.** The debugger reads the same source maps Sentry reads (lesson 1 of chapter 096); when source maps drift (rare in dev), breakpoints land on transpiled lines. Verify the dev source map output is current (`next dev` writes maps to `.next/`); restart the dev server if breakpoints look misaligned.
- **What not to debug live.** Production. Never run `--inspect` against a production Node process — it opens a remote-code-execution surface. The debugger is a dev-only tool; production debugging is the logs + Sentry workflow from prior lessons. Vercel doesn't expose the inspector port; the discipline matches the platform.
- **The `debugger` statement.** A line of `debugger;` in code pauses execution when the debugger is attached. Reach for it when a gutter breakpoint won't bind (Turbopack source-map miss) or when the location is dynamically computed (a generated function). Treat as a temporary; remove before commit (CI greps for it).
- **Performance debugging is a separate tool.** Profiling (CPU, memory, async waterfalls) uses `--inspect`'s "Profiler" tab in Chrome DevTools or VS Code's CPU profile. Out of scope for this lesson; named once with a forward pointer to Chapter 098's perf work.
- **Watch-outs.** "Unbound breakpoint" with Turbopack — fall back to `next dev --webpack` for the session; running `--inspect` in production — never; leaving `debugger;` in committed code — CI grep fails the build; setting a breakpoint inside a tight loop pauses 10000 times — use a conditional breakpoint; stepping into `node_modules` wastes time — `skipFiles` keeps it shallow; the inspector port (9229) collides with another process — pass `--inspect=9230` to override; client-side React 19 server components don't run in the Node process — they run during server render, breakpoint in the server config; HMR can stale the file the breakpoint references — restart the dev server when in doubt; the Debug Console's `await` returns a Promise unless the surrounding context supports top-level await — VS Code's REPL does; expecting the debugger to step *inside* a Drizzle generated query at the SQL level — it sees the JS, the SQL is generated lazily; debugging a worker thread or edge runtime function — different launch config, not covered here.

What this lesson does not cover:

- Performance profiling and bundle analysis — Chapter 098.
- React DevTools (component tree, hooks, profiler) — client-side, owned earlier.
- Test debugging in Vitest with the inspector — Chapter 19 references the same protocol.
- Production hot-debugging — never, out of scope by design.
- Edge runtime debugging — out of scope (the course defaults to Node runtime).
- Heap snapshots and memory leak diagnosis — out of scope.

Estimated student time: 40 to 50 minutes. Setup-and-pattern lesson; the modeled server-action-failed workflow is the chapter's payoff.
