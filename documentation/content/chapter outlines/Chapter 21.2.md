# Chapter 21.2 — The CI gate on GitHub Actions

## Chapter framing

Chapter 21.2 wires the structural enforcement Lesson 21.1.4 promised. The branch protection ruleset names required status checks by string; until something runs and reports those names, the gate is hollow. This chapter is what *produces* those checks. The thesis is a four-job baseline — type-check, lint, test, build — that runs on every PR, gates the merge, and ships with the repo as `.github/workflows/ci.yml`. Past the baseline, the chapter names the supplementary checks that earn their weight on a SaaS codebase (`pnpm audit`, link-check on docs, `actionlint` on the workflow itself, Dependabot version updates) and the few platform primitives that matter in 2026 (least-privilege `permissions:`, OIDC for deploy credentials, `concurrency:` to cancel superseded PR runs, dependency caching the right way for pnpm).

Threads that run through every lesson. **CI is the structural enforcement of the rules the team would otherwise hope hold** — Lesson 21.1.4's ruleset only enforces what CI produces; rename a job and the gate disappears silently. **Four jobs, named once, listed in the ruleset by name string** — `typecheck`, `lint`, `test`, `build`; everything else is supplementary and gated separately. **Fast feedback or the gate gets bypassed** — the senior reflex is to keep the baseline under five minutes on warm cache, which dictates caching, parallelism, and what doesn't belong in CI. **Security defaults are non-negotiable** — pin actions by SHA where the action handles secrets, `permissions:` set at the workflow level to the minimum, no long-lived cloud credentials in `secrets:` when OIDC is available. **The platform default is GitHub Actions** — GitLab CI / CircleCI / Buildkite get named once.

## Lesson 21.2.1 — GitHub Actions primitives

The workflow/job/step model, the trigger surface, pnpm-aware caching, least-privilege `permissions:`, secrets, and `concurrency:` — the GHA surface a senior needs to author the workflows that produce required status checks.

Topics to cover:

- **The senior question.** The ruleset from 21.1.4 lists required status checks by name. Where do those names come from, and what's the smallest surface of GitHub Actions a senior needs to read and write the workflows that produce them? The lesson lands workflows, jobs, steps, secrets, caching, permissions, concurrency — the primitives the rest of the chapter composes.
- **The mental model.** A **workflow** is a YAML file under `.github/workflows/`. Each workflow has one or more **jobs**. Each job runs on a fresh **runner** (`ubuntu-latest` by default) and is a sequence of **steps**. Steps either `run:` a shell command or `uses:` a reusable **action** from the Marketplace or a path in the repo. Jobs in the same workflow run in parallel by default; `needs:` makes one wait for another. Each job's name string is what the ruleset matches against.
- **The trigger surface, named.** `on: push` (every push to any branch), `on: pull_request` (PR opened/synchronized/reopened), `on: schedule` (cron), `on: workflow_dispatch` (manual button), `on: workflow_call` (reusable from another workflow). The CI baseline uses `pull_request` plus `push: branches: [main]` — every PR runs; merges to `main` re-run so the post-merge state is also verified.
- **The minimum-viable workflow shape.** Walkthrough of a worked file: `name`, `on`, `permissions` (set to `contents: read` at the top), `concurrency` block, `jobs:` with one job that checks out (`actions/checkout`), sets up Node + pnpm (`pnpm/action-setup` then `actions/setup-node` with `cache: pnpm`), installs (`pnpm install --frozen-lockfile`), and runs `pnpm typecheck`. The whole file is ~30 lines and the student can read every one.
- **`actions/checkout`, `pnpm/action-setup`, `actions/setup-node` — the three actions every JS CI uses.** `checkout@v4` clones the repo at the PR head SHA; `pnpm/action-setup@v4` installs pnpm at the version pinned in `package.json`'s `packageManager` field; `setup-node@v4` with `cache: 'pnpm'` installs Node and configures the built-in dependency cache against `pnpm-lock.yaml`. The pnpm action goes *before* setup-node so the cache integration finds pnpm. Pin major versions; pin by SHA for actions that touch secrets.
- **Caching pnpm dependencies.** `setup-node`'s `cache: 'pnpm'` option handles the common case: it caches pnpm's store directory keyed on `pnpm-lock.yaml`, restores it before install, saves it after. A cold install on a 2026 SaaS app runs ~80 seconds; a warm cache is ~30 seconds. The senior diff: don't roll a manual `actions/cache` step unless `setup-node`'s built-in misses (rare). For Next.js build cache (separate from pnpm store), `actions/cache` against `.next/cache` keyed on `pnpm-lock.yaml` plus source files — earns its weight when build time exceeds two minutes.
- **`pnpm install --frozen-lockfile` as the CI invocation.** In CI, pnpm automatically uses frozen-lockfile mode (no `pnpm-lock.yaml` mutations); naming the flag explicitly makes the intent clear and works in non-CI environments that re-use the workflow. Without it, a stale lockfile silently writes a new one and CI passes against a different dep tree than `main` has.
- **`permissions:` at the workflow level — least privilege.** Default `GITHUB_TOKEN` historically had broad write permissions; the senior reflex is to explicitly set `permissions: contents: read` at the workflow level and grant additional scopes at the job level only where needed. A typecheck/lint/test/build CI needs only `contents: read`. A workflow that posts PR comments needs `pull-requests: write` on that job. Set the floor at the workflow, raise per job, never the reverse.
- **Secrets — `secrets:` mapping, `${{ secrets.NAME }}`.** Repo-level secrets (Settings → Secrets and variables → Actions); environment-level secrets (gated to specific deployment environments, named once). Secrets injected as environment variables at the step level: `env: DATABASE_URL: ${{ secrets.DATABASE_URL }}`. Never log secrets — GitHub redacts known secret values from logs but echoing a derived value (a JWT signed with the secret) leaks. **OIDC, forward-referenced** — for cloud-credential access (Vercel deploys, AWS roles) prefer OIDC-issued short-lived tokens over long-lived secrets; Chapter 21.3.7 owns the wiring on the Vercel side.
- **`concurrency:` — cancel superseded PR runs.** When a PR gets two pushes in two minutes, the second run should kill the first; otherwise the queue piles up and the gate runs out of CI minutes. `concurrency: group: ${{ github.workflow }}-${{ github.ref }} cancel-in-progress: true`. Reach: set on every PR-triggered workflow. Watch-out: setting it on the `push: main` job too aggressively will cancel a deploy mid-run when a fast follow-up lands; the senior reflex is to scope concurrency tighter when production deploys are in the workflow (don't cancel deploys mid-flight).
- **The expression syntax, named.** `${{ github.ref }}`, `${{ github.event.pull_request.number }}`, `${{ secrets.X }}`, `${{ env.Y }}`, `${{ matrix.node }}`. Used in `if:` conditions, env values, and concurrency groups. Documented in the GHA docs; the chapter doesn't memorize it.
- **Job parallelism vs. sequencing.** Four jobs in `jobs:` without `needs:` run in parallel — the right shape for typecheck/lint/test/build when each is independent. `needs: [build]` on a deploy job is the right shape for "only deploy after build passes." The senior reflex: parallelize aggressively; sequence only for true dependencies. Watch-out: four parallel jobs each install dependencies separately; either accept the duplicate install time (acceptable when cached) or fold into one job with sequential steps (acceptable when total wall-clock matters more than per-job retry granularity).
- **Matrix strategy — named, mostly cut.** `strategy: matrix:` runs the job once per combination of variables. The legitimate SaaS use is rare in 2026 (one Node version, one OS); cross-version testing is for libraries, not products. Named once; the course defaults to single-config jobs.
- **Reusable workflows and composite actions — named once.** `workflow_call` lets one workflow be invoked from another with inputs; useful at organization scale for "every repo's CI looks the same." Composite actions in `.github/actions/setup/` extract repeated setup steps (checkout + pnpm + node + install) into one `uses:` line. For a single-repo SaaS, the inline form is clearer; the abstractions earn weight at 5+ repos.
- **`actionlint` — lint the workflows themselves.** A workflow YAML typo (wrong indentation, wrong `uses:` reference, undefined expression) fails at run time, not at commit time. `rhysd/actionlint` runs locally and in CI; catches the typos at the speed of `eslint`. Setup is one `uses:` step in a workflow-lint job, plus an optional pre-commit hook. Named here; configured in 21.2.3 as one of the supplementary checks.
- **The `.github/workflows/` directory.** All workflow YAML lives here, version-controlled, reviewed in PRs. Naming convention: `ci.yml` for the PR gate, `deploy.yml` for production deploys (Chapter 21.3), one workflow per concern. Per-job env files can live elsewhere; workflows themselves are concentrated in this directory.
- **The runner image — `ubuntu-latest` and the watchout.** `ubuntu-latest` is a moving target (currently 24.04 in 2026); pinning to `ubuntu-24.04` shields against runner upgrades that break the build. The senior call: pin when the team has been burned, accept the moving target until then. macOS and Windows runners exist; SaaS builds run on Linux.
- **Self-hosted runners — named, cut.** For specialized hardware or private network access. The course defaults to GitHub-hosted; self-hosted is named once and forwarded to the platform team's playbook.
- **Watch-outs:** unpinned actions (`@main`) — a malicious tag rewrite (the March 2026 trivy-action incident) leaks every secret in scope; SHA-pin actions that handle secrets; default `permissions:` too broad — set `contents: read` at the workflow level; missing `concurrency:` on PR workflows — burns CI minutes and queue time; `pnpm install` without `--frozen-lockfile` — silent lockfile drift; not caching pnpm store — every run pays cold-install cost; secrets echoed in step output — known-value redaction misses derived values; renaming a job after the ruleset references it — gate silently disappears (21.1.4 watch-out, repeated here because it lands in this chapter's code).

- Dependabot version updates — Lesson 21.2.3.

## Lesson 21.2.2 — The four-job merge gate

The `.github/workflows/ci.yml` baseline of parallel typecheck / lint / test / build jobs, why these four and no others gate the merge, the five-minute speed budget, and the job-name discipline that keeps the ruleset enforceable.

Topics to cover:

- **The senior question.** What's the smallest set of checks a SaaS repo must run on every PR to make the branch-protection ruleset structurally enforceable? Four: type-check, lint, test, build. Each catches a class of regression the others won't. The lesson lands the four-job `.github/workflows/ci.yml`, the trade-offs (parallel jobs vs. one fat job), the failure modes each catches, and the speed budget.
- **Why these four, and no others, in the gate.** **Type-check** (`pnpm typecheck` → `tsc --noEmit`) catches anything the type system can prove wrong; the bar the rest of the codebase relies on. **Lint** (`pnpm lint` → Biome) catches code-style violations and a band of correctness rules (unused imports, missing await, etc.) that types don't cover. **Test** (`pnpm test` → Vitest) catches behavioral regressions; the safety net for refactors. **Build** (`pnpm build` → `next build`) catches the cross-module integration problems type-check misses (bad imports across files, missing env at build time, RSC/client boundary violations); also produces the artifact the deploy job will consume in 21.3. Anything else (audit, link-check, actionlint, dependabot) is supplementary — useful, but not part of the merge gate.
- **The workflow shape — four parallel jobs.** One `jobs:` map with `typecheck`, `lint`, `test`, `build`. Each job: checkout, setup pnpm + Node with cache, `pnpm install --frozen-lockfile`, then the one command that *is* the job. No `needs:` between them — parallel. The trade: each job pays install time separately, but the wall-clock is bounded by the slowest job (typically `build`) instead of the sum. With pnpm store cached, install is ~30s; total wall-clock for the four parallel jobs is ~3–5 minutes on a typical SaaS app.
- **The fat-job alternative — when it earns weight.** One job that runs all four commands sequentially. Pays install once; total wall-clock is the sum. Reach when CI minutes matter more than feedback granularity (cheap plan with limited concurrent jobs) or when the install cost dominates run time and caching can't fix it. The default is four parallel jobs because failure granularity matters — a failing typecheck shouldn't make you wait for build to finish to see lint and test results too.
- **`pnpm typecheck` — the script and the failure mode.** `package.json`: `"typecheck": "tsc --noEmit"`. Runs the TypeScript compiler against the whole project; reports any type error with file/line. `--noEmit` skips the actual JS emission (Next.js handles that). The CI failure mode it catches: `pnpm dev` was happy because TS-loader was lazy, but `tsc --noEmit` over the whole project finds the import cycle that breaks once nothing's lazy.
- **`pnpm lint` — Biome as the formatter + linter in one.** Established in Unit 1. `package.json`: `"lint": "biome ci ."`. The `ci` subcommand exits non-zero on any violation (unlike `biome check` which can auto-fix). Biome covers what ESLint+Prettier used to; the senior diff in 2026 is one tool, one config, ~10x faster. The watch-out: `biome ci` is read-only; running `biome check --write` locally before pushing is the dev-side workflow that prevents the CI failure.
- **`pnpm test` — Vitest in CI mode.** `package.json`: `"test": "vitest run"`. The `run` subcommand runs once and exits (vs. `vitest` which watches). `--reporter=github-actions` emits annotations that GitHub renders inline in the PR's "Files changed" view — failed assertions show on the line that caused them. Reach for this reporter when the test base is large enough that "go find the failure in the log" is friction. Unit 19 owns the test framework wiring; this lesson uses it as a black box.
- **Sharding tests, named.** Large test suites (>500 tests, >2 minutes) can shard across parallel jobs using Vitest's `--shard=1/4` flag plus a matrix. Earns weight past the typical SaaS startup's surface area; named once, cut at the baseline.
- **`pnpm build` — `next build` in CI.** `package.json`: `"build": "next build"`. Runs the full production build: bundle, route compilation, type-checking again (but only of build-relevant code), static-page generation. The watch-out: `next build` requires environment variables that are read at build time (`NEXT_PUBLIC_*`, database URLs read by Drizzle for static generation, etc.); CI needs them set as secrets or as inert defaults. The course's env strategy (Unit 5) ships a `T3-env`/`@t3-oss/env-nextjs`-style validator; CI sets `SKIP_ENV_VALIDATION=true` only when build-time DB access would fail, never in production.
- **The env-at-build-time problem.** Some builds (RSC pre-rendering of public pages) hit the DB at build time. CI either needs a DB (Neon branch via `DATABASE_URL` secret) or needs the routes to be marked dynamic. The senior diff: prefer dynamic rendering for anything DB-backed; reserve build-time DB reads for genuinely static public pages. The CI workflow exposes whichever env vars build needs as `env:` from `secrets:` — `DATABASE_URL`, `BETTER_AUTH_SECRET`, etc.; minimum scope each time. Chapter 21.3.7 owns the env scoping story.
- **Reporting and annotations.** Vitest's `--reporter=github-actions` and `tsc --pretty` both produce GitHub-aware output. Biome's CI mode emits problem-matcher-friendly format. The senior reflex: configure reporters so failures land *on the PR diff*, not buried in a 1000-line log.
- **The speed budget — under five minutes wall-clock on warm cache.** Cold cache: ~6–8 minutes. Warm: ~3–5 minutes. If the build alone exceeds three minutes the senior reflex is to investigate (the project changed shape; some route fetches everything at build time; the test base sharded already and didn't help; the runner needs upgrading from the free tier). CI past five minutes gets bypassed — by "merge anyway, I'll fix it after," by branch-protection exceptions, by frustration. The budget is load-bearing.
- **Cache strategy past pnpm store.** The Next.js build cache (`.next/cache`) speeds incremental `next build` significantly. `actions/cache` against `.next/cache` keyed on `pnpm-lock.yaml` + the source-file hash. Reach when build time exceeds two minutes; cut when build is already fast. Stale cache is the watch-out — a bad cache can produce a passing CI run that ships broken; key carefully and accept occasional cache misses.
- **The required-status-check list — the names that must match the ruleset.** Job IDs become check names. The ruleset (21.1.4) lists `typecheck`, `lint`, `test`, `build` by string. **Watch-out, repeated**: renaming a job (e.g., `test` → `tests`) breaks the gate silently — the ruleset's "required" line no longer matches any check; the PR is merge-eligible without that check passing. The reflex: any rename of a CI job is a two-file PR (workflow + ruleset).
- **When the CI gate is also the release gate.** In Chapter 21.3, the deploy workflow either triggers on `push: main` (after merge) or is a separate job in this same `ci.yml` gated on `needs: [typecheck, lint, test, build]`. The decision is forward-referenced; what matters here is that the baseline produces the artifact deploy can consume.
- **Worked file outline (not full code).** `.github/workflows/ci.yml`: name, on (pull_request + push to main), permissions (`contents: read`), concurrency block, jobs (`typecheck`, `lint`, `test`, `build`) each running checkout, pnpm setup, node setup with cache, frozen install, then the single command. ~80 lines total. The student writes this file in the exercise.
- **Watch-outs:** the four job names must match the ruleset string-for-string; build that doesn't run in CI (only locally) — broken `main` is one push away; tests that pass locally but fail in CI — usually env, timezone, or `Date.now()` differences; flaky tests in the gate — retry-on-failure is the wrong reflex (named, cut), fixing the test is the only durable answer; environment validation that fails in CI — either ship `SKIP_ENV_VALIDATION` *only* for build, never runtime, or provide inert CI values; oversized PR build that times out — the speed budget is the canary, not the CI minutes bill.

What this lesson does not cover:

- The GHA primitives — Lesson 21.2.1.
- Supplementary checks (audit, link-check, actionlint, dependabot) — Lesson 21.2.3.
- The deploy workflow — Chapter 21.3.
- The test framework itself — Unit 19.
- Biome configuration — Unit 1.
- Migration testing against the Neon branch — Chapter 21.4.
- Bundle-size checks, Lighthouse, perf budgets in CI — Unit 20.

## Lesson 21.2.3 — Signal checks and dependency hygiene

The gate-vs-signal split for non-blocking jobs (`pnpm audit`, `actionlint`, scheduled link-check), the 2026 supply-chain layer (signature verification, `minimumReleaseAge`, SHA-pinning), and Dependabot grouping with patch auto-merge.

Topics to cover:

- **The senior question.** The four-job baseline is the merge gate. What else runs on every PR (or on a schedule against `main`) without blocking merges, and what's the senior diff on each: `pnpm audit` for known CVEs, `markdown-link-check` for docs rot, `actionlint` for workflow typos, and Dependabot (or Renovate) for keeping the dep tree alive. The lesson lands the supplementary surface and the threshold each crosses to earn its weight.
- **The "gate vs. signal" split.** Supplementary checks are **signal**, not **gate**. They run, they report, they may comment on the PR, but they don't block merge. The reasoning: a CVE in a transitive dep with no exploitable path shouldn't block a Friday-evening fix; a broken link in the README shouldn't block a hotfix. Reach for blocking only when the failure means production-broken; otherwise the cost of a false-positive gate is bypass habit, which corrodes the real gates.
- **`pnpm audit` — known-CVE scanning.** Runs against the registry's advisory database; reports vulnerabilities by severity (low/moderate/high/critical) with affected paths. CI job: `pnpm audit --audit-level=high` (exit nonzero on high or critical; ignore low/moderate noise). Reach: separate job in `ci.yml`, not in the gate list; runs on every PR. Watch-out: a critical CVE in a dev-only dep (a test framework, a build tool) is still flagged; the path-aware filter (`--prod`) scopes to production deps only when that's the relevant signal.
- **`pnpm audit signatures` and provenance — the 2026 supply-chain layer.** After the spring 2026 OSS incidents (compromised packages on npm, force-pushed action tags), the senior reflex is to add a second layer: verify package signatures and provenance. `pnpm audit signatures` (when stable) checks ECDSA signatures against registry public keys; pnpm's `minimumReleaseAge` setting in `.npmrc` blocks installs of packages published in the last N hours (catching active compromises before they propagate); `strictDepBuilds` and `blockExoticSubdeps` raise the prevention bar. Named here; the senior call is to enable `minimumReleaseAge: 72h` and SHA-pin GitHub Actions that touch secrets, both done once in the repo.
- **The audit-ci alternative — fine-grained allowlisting.** When `pnpm audit` flags a CVE the team has triaged (no exploitable path, upstream is dragging on a fix), the IBM `audit-ci` tool allowlists by advisory ID with an expiry date. Reach when the dep tree has long-tail unfixed advisories that aren't real risks; cut when audit findings are zero or near-zero. Named once; the default is plain `pnpm audit`.
- **`markdown-link-check` — docs rot.** External and internal links in `README.md`, `AGENTS.md`, `docs/**` go stale (404s, domain changes, repo renames). A scheduled job (`on: schedule: cron: '0 0 * * 1'` — Monday mornings) runs the check across `**/*.md` and posts a GitHub issue when links break. PR-level checking is too noisy (docs PRs would re-check the whole tree); a weekly sweep is the right cadence. Reach when the docs surface exceeds a handful of files; cut for a single-README repo.
- **`actionlint` — lint the workflow YAML itself.** Catches: undefined `${{ }}` expressions, wrong `uses:` action names or versions, invalid event triggers, shellcheck issues in `run:` blocks. CI job: `rhysd/actionlint@<sha>` action runs on PRs that touch `.github/workflows/`. Local pre-commit hook for the dev-side catch. The watch-out the lint catches: a typo in `pnpm/action-setup@v4` that fails at run time, blocking the gate's gate; better to catch on commit.
- **Dependabot — the GitHub-native dep updater.** `.github/dependabot.yml` configures Dependabot to open PRs for outdated dependencies on a cadence. Three update streams worth enabling: `npm` (the JS deps), `github-actions` (workflow action versions; this is the answer to the "pin actions but keep them current" problem), and `docker` if the repo has a Dockerfile. Configuration: weekly cadence, group minor+patch into one PR per ecosystem to avoid PR-noise, separate PRs for major bumps. PRs run the full CI gate; merge once green. Forward-reference: Unit 17.2.8 covered the dep-hygiene story; this lesson is the workflow wiring.
- **Dependabot's grouping rules.** Pre-2024, Dependabot opened one PR per dep, drowning the team in 50 PRs/week. Modern config: `groups:` block per ecosystem, with `applies-to: version-updates`, `patterns:` (e.g., `["*"]` for everything), `update-types: ["minor", "patch"]`. One weekly PR for minor+patch across all deps; majors separate. Reach this config; the per-dep flood is the deprecated default.
- **Renovate — the alternative.** Renovate is the more configurable alternative (multi-platform support, regex managers for non-package version pinning, shared org config presets, more granular automerge rules). Setup is more involved (GitHub App install + onboarding PR + config tuning). Reach when Dependabot's capabilities don't fit (auto-merging patch updates with a stability window; updating versions in arbitrary files; shared config across many repos); the course default is Dependabot for simplicity. Named, deferred.
- **Auto-merge for green dep PRs.** GitHub's auto-merge feature (21.1.3) queues PRs to merge once required checks pass. Pair with Dependabot patch-update PRs that pass CI: the PR opens, runs CI, merges itself. The senior call: enable for patch updates only (lowest risk); require human approval for minor and major. The watch-out: a patch update with a behavior change slips through; the test base is the safety net, which is why the four-job gate runs on every PR including Dependabot's.
- **Bundle-size check — named, deferred.** A separate workflow that compares the production bundle size to `main`'s and comments on the PR if delta exceeds a threshold. Tools: `bundlewatch`, `size-limit`, or Next.js's built-in `--analyze`. Earns weight when bundle size matters to the product (consumer-facing, mobile-heavy); cut for B2B SaaS where shipping a megabyte more isn't the bottleneck. Unit 20 owns the perf story; named here as the supplementary-check pattern extends to bundle reports.
- **CodeQL and SAST — named, deferred.** GitHub Advanced Security's CodeQL runs static analysis for security issues. Free on public repos; paid on private. Earns weight when the team handles authn/authz/crypto code directly; the SaaS using Better Auth offloads most of that. Named once.
- **License scanning — named, deferred.** Tools like `license-checker-rseidelsohn` verify dep licenses match an allowlist (no AGPL in commercial code, etc.). Earns weight when the company has a legal policy; cut at the typical SaaS-startup scope.
- **The supplementary-check workflow shape.** Either folded into `ci.yml` as additional jobs not listed in the required-checks list, or split into `.github/workflows/security.yml`, `.github/workflows/docs.yml`, etc., by concern. The senior call: split when the concerns are independent (audit runs on every PR, link-check runs weekly, dep updates run on Dependabot's schedule); fold when one trigger covers all. Three workflows is a reasonable steady state.
- **The schedule trigger.** `on: schedule: - cron: '0 9 * * 1'` (Monday 9 AM UTC). Useful for non-PR signals — link rot, dep audits on `main`, weekly security scans. Watch-out: scheduled workflows run on the default branch's version of the workflow file; testing a schedule change requires merging first or using `workflow_dispatch` for one-off runs.
- **Worked configuration outline (not full code).** Three files. `.github/workflows/ci.yml` adds two non-blocking jobs: `audit` (running `pnpm audit --audit-level=high`) and `actionlint` (running the rhysd action). `.github/workflows/links.yml` runs on a Monday schedule, runs `markdown-link-check`, opens an issue on failure via `peter-evans/create-issue-from-file`. `.github/dependabot.yml` enables `npm` and `github-actions` ecosystems, weekly, grouped minor+patch.
- **Watch-outs:** supplementary checks bypassed into the required list — defeats the gate-vs-signal split; pnpm audit's noise leading to "ignore the report" habit — tune the audit level and accept that some advisories don't apply; dependabot PRs accumulating without review — auto-merge for patches, batched review for the rest; actionlint not run locally — same surface as biome, catch on commit; broken links flagged on PRs that didn't touch the broken file — schedule the check, don't run per-PR; minimumReleaseAge too aggressive (e.g., 14 days) — slows legitimate fast-fix updates; renovate config sprawl when Dependabot was sufficient — cut the complexity until you need it; scheduled workflow changes that never run because the schedule is on `main`'s old version — `workflow_dispatch` is the test hook.

What this lesson does not cover:

- License scanning — out of scope.

## Lesson 21.2.4 — Quizz

Top 10 topics to quiz:

- The four-job baseline (typecheck / lint / test / build) — what each catches that the others don't; why these and only these gate the merge; the speed budget of under five minutes wall-clock on warm cache; the senior reflex that flakes in CI are fixed, not retried.
- The required-status-check name-string discipline — the ruleset from 21.1.4 references jobs by string; renaming a job silently disables the gate; reflex of editing workflow and ruleset in the same PR.
- pnpm caching the right way — `setup-node` with `cache: 'pnpm'`, `pnpm install --frozen-lockfile`, pnpm action ordered before setup-node, when to add `actions/cache` against `.next/cache` (build time > 2 min).
- The minimum-viable workflow shape — `name`/`on`/`permissions`/`concurrency`/`jobs`; `permissions: contents: read` at workflow level with per-job raises; `concurrency` block to cancel superseded PR runs.
