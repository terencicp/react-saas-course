# Chapter 065 — Soft delete, archive, and concurrency

## Chapter framing

Chapter 065 ships the two destructive-action disciplines every SaaS list view needs the day after launch: rows that disappear without being lost, and writes that detect they were racing another tab. The student arrives with a working URL-state list view from chapter 064, a Server Action / Drizzle stack from Units 6 and 7, the tenant-aware query helper from Chapter 060, and the Server Action Result shape from Chapter 047. This chapter lands `deletedAt` as the soft-delete column, archive as a distinct user-facing lifecycle state (not the same thing as soft delete), the restore action, and a base-query helper layered on top of `tenantDb` so a hand-written query physically cannot forget `deletedAt IS NULL`. Then it lands the version-column precondition that turns silent last-write-wins into an honest 409, and the React 19 surface that lets the second tab refresh and retry instead of overwriting.

Threads that run through every lesson: soft-delete is a database mechanism for recovery and audit, archive is a product decision about lifecycle state — same column shape often, different user contract, different UI affordance; never expose internal `deletedAt` to the user as "deleted" — either it's gone from the UI entirely (soft delete) or it's labeled "archived" with a restore path; missing `deletedAt IS NULL` is the canonical SaaS data-leak bug, structurally prevented by the base-query helper rather than by code review; the tenant-aware helper from chapter 060 composes with the visibility helper here — both filters get added by the helper, neither is hand-typed at the call site; partial unique indexes (`WHERE deleted_at IS NULL`) keep uniqueness sane after soft delete; optimistic concurrency is the senior default for any user-editable record where two tabs is realistic — the `version` column (or `updatedAt` precondition) goes on the entity at schema design time, the precondition lives in the UPDATE's `WHERE`, the zero-rows-affected result returns 409 through the Server Action Result shape; React 19's flow on conflict is refresh-and-retry, not silent overwrite — the user is told their copy is stale, sees the current values, and decides; last-write-wins remains correct for low-collision append-only writes (audit-log style) and for fields the user expects to clobber (a single toggle owned by one human) — name the threshold, don't dogmatize. The chapter ships three teaching lessons plus a quiz.

## Lesson 1 — Two timestamps, three actions

Teaches the soft-delete-vs-archive distinction, the `deletedAt`/`archivedAt` schema with partial unique and composite indexes, and the `softDelete`/`archive`/`restore` Server Actions that drive the lifecycle.

Topics to cover:

- **The senior question.** A user clicks "Delete invoice." What disappears from the UI, what stays in the database, what can come back, and what cannot? And separately: a project is "done" but the user wants it off the active list without deleting it — is that the same column, the same action, the same UI? The lesson lands the soft-delete-vs-archive distinction and the three actions (`delete`, `archive`, `restore`) as the canonical lifecycle surface.
- **Soft delete vs. hard delete — when each earns its weight.** Soft delete sets `deletedAt = now()`; the row stays in the table, every query filters it out, an admin can restore it, the audit trail survives. Hard delete is `DELETE FROM`; the row is gone, foreign keys cascade or block, recovery is from backup. Soft delete is the default for any record a user can re-reference (invoices, projects, customers, comments); hard delete is right for transient artifacts (session rows, password reset tokens, draft autosaves the user never sees). The senior call: pick soft for anything with cross-references or compliance shadows, hard for ephemera.
- **Soft delete vs. archive — the user-facing distinction.** Soft delete is invisible: the user clicks "Delete", the row vanishes from every list, only an admin (or a "deleted items" view, often hidden) ever sees it again. Archive is explicit: the user clicks "Archive", the row moves to an "Archived" tab the user can browse, restore is a normal user action. Same column shape often (`deletedAt`/`archivedAt` are both timestamps), different product contract. Many SaaS products use only archive (Notion, Linear projects), some use both (archive for "I'm done with this", soft delete for "I deleted this by mistake"), some use only soft delete (admin-only restore). The lesson names the three product shapes and picks "archive as the primary user surface, soft delete as the admin recovery surface" as the course default.
- **The schema — one timestamp or two.** Single-column approach: `deletedAt` plus `archivedAt`, both nullable timestamps, both nullable, two distinct lifecycle states. Single-column-with-state approach: a `status` enum (`active`, `archived`, `deleted`) plus `deletedAt`/`archivedAt` for the timestamp. The course picks two timestamp columns — simpler schema, the column name is self-documenting, the partial-index story is the same shape, and a row can be archived then deleted (both set, deleted overrides). The senior anchor: optimize for the query and the UI affordance, not for "look one column".
- **The Drizzle schema fragment.** `deletedAt: timestamp('deleted_at', { withTimezone: true })` and `archivedAt: timestamp('archived_at', { withTimezone: true })` — both nullable, both indexed via partial indexes. Pair with `updatedAt` for the audit trail and the concurrency precondition (lesson lesson 3 of chapter 065). The shape lives in a shared `lifecycleColumns` helper that gets spread into every entity table — Architectural Principle #5 (use conventions, don't reinvent) applied at the schema layer.
- **The three Server Actions — `softDelete`, `archive`, `restore`.** Each returns the canonical Result shape (lesson 2 of chapter 047). `softDelete` sets `deletedAt = now()`. `archive` sets `archivedAt = now()`. `restore` clears whichever timestamp is set — the action is idempotent on already-active rows. All three update `updatedAt` and bump `version` (lesson lesson 3 of chapter 065 makes this load-bearing). Each action runs under the tenant-aware helper from chapter 060 and the RBAC wrapper from chapter 061 — the lesson restates the composition; the action body is three lines.
- **The "deleted" vs "archived" UI affordances.** Archive is a button on every row (and a bulk action). Restore is a button on the "Archived" tab. Hard delete is a destructive button on the archive view, gated behind a confirm dialog, often role-restricted. The list view's filter has a tri-state: `active` (default, archived hidden), `archived` (archive tab), `all` (admin view, includes soft-deleted). The URL-state hook from chapter 064: `?status=archived` flows through the same `parseAsStringEnum` parser the chapter already taught.
- **Restore semantics — what comes back.** Setting `deletedAt`/`archivedAt` to null restores the row. The references the row had (foreign keys, child rows) survived the soft delete, so they snap back. Cascading restore (restoring an invoice restores its line items) is right when the children were soft-deleted together; the senior call: cascade restore only when the parent action cascaded the soft delete. If line items were hard-deleted on parent soft-delete (a bad pattern, but it happens), restore can't bring them back — name the case, name why soft-delete cascades belong together.
- **The "soft delete is a write, not a delete" invariant for joins and counts.** A `count(invoices)` that doesn't filter `deletedAt IS NULL` returns the wrong number. A join from `customers` to `invoices` that doesn't filter pulls deleted rows. Every query that touches the table is a code path that could miss the filter. The lesson names this as the canonical failure mode and previews the structural fix (lesson lesson 2 of chapter 065) — the base-query helper is the only sustainable answer past two or three tables.
- **Partial unique indexes — uniqueness survives soft delete.** A unique constraint on `(orgId, slug)` breaks after soft delete: the user soft-deletes a project named "Acme", creates a new project named "Acme", the unique constraint blocks because the soft-deleted row still has that slug. The fix: a partial unique index `CREATE UNIQUE INDEX ... ON projects (orgId, slug) WHERE deletedAt IS NULL`. The Drizzle migration spells this out; the `.unique()` shorthand doesn't support the `WHERE` clause, so the senior reach is a raw `index()` with `.where()`. The lesson names this as the most-frequent soft-delete bug after missing filters.
- **Partial indexes for the visibility filter — the hot path.** The "active rows" query runs constantly; a partial index `CREATE INDEX ... ON invoices (orgId, createdAt) WHERE deletedAt IS NULL` keeps the index small and the scan tight. The senior reach: pair every hot composite index with `WHERE deletedAt IS NULL` when the read path always filters them out. Skip when the table is small or the deleted ratio is tiny — a regular index covers both cases without the partial-index complexity.
- **Foreign keys and cascade behavior.** A soft-deleted row still has foreign-key children. `ON DELETE CASCADE` doesn't fire on soft delete (no DELETE happened). The senior call: soft-delete cascades are application-level (the Server Action soft-deletes the parent and the children in the same transaction), not database-level. Drizzle's `db.transaction(async (tx) => ...)` from lesson 9 of chapter 047 is the wrapper; the lesson restates the shape and points at the failure mode (orphaned children visible after soft-delete) as the bug to test for.
- **The audit-log angle.** Every soft-delete, archive, and restore writes an audit-log row (lesson 3 of chapter 085 owns the audit-log table; this lesson names the touch-point). The Server Action and the audit-log insert happen in the same transaction. The senior anchor: the lifecycle state on the row is the current truth; the audit log is the history. Restoring a row doesn't erase the delete from the log.
- **GDPR posture — soft delete is not a deletion record for compliance.** A user's GDPR delete request requires the data actually gone, or anonymized beyond recovery — soft delete doesn't satisfy it. The senior anchor: the GDPR pipeline (lesson 4 of chapter 085) is a separate path that nullifies PII columns and may hard-delete the row; the regular "delete" button users see is soft delete. Two distinct flows, named here, owned at depth in chapter 085.
- **Worked example — the invoice with archive and soft delete.** The schema fragment with `deletedAt` and `archivedAt`. The `archiveInvoice` Server Action, the `restoreInvoice` action, the `softDeleteInvoice` action. The migration with the partial unique index and the partial composite index. The "Archived" tab in the list view, filtered via `?status=archived` from chapter 064's URL-state vocabulary. The example is the chapter's reference shape.
- **Watch-outs:** soft delete without partial unique indexes blocks re-creation with the same slug; cascade soft-deletes that aren't in a transaction can half-finish on error; restoring a row whose parent is still deleted produces an orphan visible in the UI — restore checks parent state or surfaces a 409; counting `select count(*) from invoices` without the filter is the canonical reporting bug; `db.query.invoices.findMany()` without the filter is the same bug at the application layer — lesson lesson 2 of chapter 065 is the structural fix; foreign keys with `ON DELETE CASCADE` are irrelevant for soft delete and lull a reviewer into thinking deletes are handled; archive without a visible "Archived" tab is a usability bug — the user can't find the row again; hard delete from the archive tab without a confirm dialog is destructive UX; soft-delete tables grow unbounded — lesson 4 of chapter 085 retention policy applies.

What this lesson does not cover:

- The base-query helper that enforces `deletedAt IS NULL` — lesson lesson 2 of chapter 065.
- The `version` column and the 409 mechanics — lesson lesson 3 of chapter 065.
- The URL-state visibility toggle — Chapter 064.
- Audit-log table schema — lesson 3 of chapter 085.
- GDPR retention and anonymization — lesson 4 of chapter 085.
- Trigger-based archive tables and partition pruning — beyond course scope; named as an alternative.
- Postgres Row-Level Security as an alternative enforcement layer — named as an alternative; not the course default.

Estimated student time: 45 to 55 minutes. The lifecycle vocabulary and the schema shape are the foundation for the rest of the chapter.

## Lesson 2 — Making the missing filter impossible

Teaches the base-query helper that composes on top of `tenantDb` to expose `active()`, `archived()`, and `includingDeleted()` so every read carries the lifecycle and tenancy filters by construction.

Topics to cover:

- **The senior question.** A list view filters `deletedAt IS NULL`. The detail page filters it. The count query filters it. Every join filters it. One forgotten filter and deleted rows leak into a report, a customer's archived data shows up in a new view, or — worst case — a deleted row in one tenant shows up to another tenant because the filter and the tenancy check were both missed. How do you make missing the filter physically impossible? The lesson lands the base-query helper as the structural fix: every read goes through it, hand-written `from(invoices)` is a code-review red flag.
- **The failure mode, named in production terms.** A developer adds a new route, writes `db.select().from(invoices).where(eq(invoices.orgId, orgId))`, ships. The query returns rows the user soft-deleted last week. The bug is invisible in dev (no soft-deleted rows yet); shows up the day a customer notices. Defense by code review fails because the omission looks correct. Defense by typing — the helper returns a query builder that already has `deletedAt IS NULL` applied — works because the wrong shape doesn't compile.
- **The composition with `tenantDb` from chapter 060.** Chapter 060's `tenantDb(orgId)` returns a query layer that pre-scopes every read to the organization. This chapter's helper composes on top: `tenantDb(orgId).invoices.active()` returns a query builder for active invoices in the org; `.archived()` for the archived tab; `.includingDeleted()` for the admin recovery view (gated by role, never the default). One source of truth for both filters — neither is typed at the call site.
- **The API surface — three methods per entity.** `active()` returns `deletedAt IS NULL AND archivedAt IS NULL`. `archived()` returns `deletedAt IS NULL AND archivedAt IS NOT NULL`. `includingDeleted()` returns everything — the escape hatch for admin tooling. The lesson names the three states as the canonical surface; resist adding `onlyDeleted()` or `withTrashed()` — the three states cover the product surface, and the escape hatch is intentionally named to feel weighty.
- **The Drizzle implementation shape.** A factory function `createScopedQuery(tx, orgId, table)` returns an object with the three methods, each returning a builder pre-filtered by `orgId`, `deletedAt`, `archivedAt`. The builder is a normal Drizzle select chain; the caller adds `.where(extra)`, `.orderBy(...)`, `.limit(...)`. The senior call: the helper returns a chainable builder, not a fully-formed query — composability matters, the lesson keeps the helper minimal.
- **TypeScript discipline — the helper's return type tells the truth.** The returned builder is typed such that the columns include `archivedAt` (nullable timestamp) so the UI can render "Archived on…" but the deleted state is unreachable through `active()` and `archived()`. The senior anchor: types are the second layer of the structural defense — the helper makes the wrong shape harder to write, and TypeScript makes it harder to read code that bypassed the helper.
- **The "escape hatch" rule.** `includingDeleted()` exists for admin tooling, GDPR exports, and audit views. It's marked with a code search-friendly name so a grep for `includingDeleted` returns every place the escape hatch fires. The senior anchor (from Architectural Principle #5): the escape hatch is part of the API, not a workaround — name it loud and gate it on role at the action layer.
- **Composing with joins — the trickier case.** A query joining invoices to line items needs the soft-delete filter on both. The helper handles single-table reads cleanly; joins need either a join-aware variant (`scoped.invoices.active().withLineItems()`) or a discipline of "any join goes through a named function in `/lib/queries`". The course picks the second: hand-written joins live in `/lib/queries/invoices.ts`, each function is reviewed, each query passes the helper's filters explicitly via a shared `activeFilter(table)` predicate. The senior anchor: the helper covers the 80%, joins get named functions and reviewed once.
- **The shared `activeFilter` and `archivedFilter` predicates.** Two named SQL fragments — `sql\`${table.deletedAt} IS NULL\`` and the archived variant — exported from the helper module, used inside hand-written joins so the same definition flows everywhere. The senior anchor: one definition, used by both the helper and the carve-out queries; a refactor of the lifecycle shape (adding a third state, say) touches one file.
- **The lint angle — banning raw `.from(invoices)`.** An ESLint rule (or a code-review checklist item) flags any direct `db.select().from(<entityTable>)` outside the helper module. The senior reach: when the type system can't enforce a discipline, lint can. The lesson names the rule as a one-line override (an ESLint custom rule or a `no-restricted-syntax` configuration) rather than implementing it inline; the broader course owns lint setup in Unit 1.
- **The "but I need a raw query for performance" case.** A complex reporting query hand-tuned with `EXPLAIN ANALYZE` (lesson 3 of chapter 043) may not fit through the helper. The senior call: hand-tuned queries live in `/lib/queries/reports/*.ts`, each carries a comment naming the perf reason and the `WHERE` clauses that replace the helper's filters, each is unit-tested for the soft-delete and tenancy filters. Performance escapes are fine; escaping the discipline silently is the bug.
- **The Server Action update/delete side — same helper applies.** `softDelete` and `archive` actions go through the helper's update form (`scoped.invoices.update().where(...)`) so the tenancy and the lifecycle precondition are on the UPDATE too — a malicious request that crafts a different `orgId` or targets a soft-deleted row gets zero rows affected. The lesson restates the symmetry; lesson lesson 3 of chapter 065 builds on this with the `version` precondition.
- **Worked example — the helper module.** The factory, the three methods for invoices, the shared filter predicates, a sample read (`scoped.invoices.active().where(...).orderBy(...).limit(20)`), a sample join through `/lib/queries/invoices.ts`. The example is small — ten or twelve lines of helper code — and the rest of the chapter uses it.
- **Watch-outs:** the helper that only works for single-table reads silently leaks deleted rows through joins — name the join discipline explicitly; the helper that returns a fully-formed query (not a builder) forces callers to construct two query paths and tempts a hand-rolled fallback — return a builder; the escape hatch that's not gated on role is the bug the helper was meant to prevent — wrap `includingDeleted` calls in the RBAC action wrapper; raw SQL through `db.execute(sql\`...\`)` bypasses the helper entirely — the lint rule names this case; refactoring an entity to add `archivedAt` without updating the helper leaves callers writing `.active()` queries that include archived rows — keep the helper's filters and the schema in lockstep; importing `db` directly from `/lib/db` in a route handler is the smell — route handlers reach for `scopedQuery(req.orgId)` instead.

What this lesson does not cover:

- The Drizzle query-builder mechanics — Chapter 042.
- The `tenantDb` helper itself — lesson 2 of chapter 060.
- The RBAC wrapper that gates `includingDeleted` — Chapter 061.
- Row-Level Security as the database-layer alternative — named, not implemented.
- The `version` precondition on UPDATEs — lesson lesson 3 of chapter 065.
- Audit logs around the escape hatch — lesson 3 of chapter 085.

Estimated student time: 35 to 45 minutes. The lesson is short because the API surface is small; the value is the discipline, not the code.

## Lesson 3 — Version columns and the honest 409

Teaches version-based optimistic concurrency, the Drizzle UPDATE precondition that turns a race into a typed 409 Result, and the React 19 refresh-and-retry surface built on `useActionState` and `useOptimistic`.

Topics to cover:

- **The senior question.** Two users open the same invoice in two tabs. Both edit. Both click Save. The naive UPDATE silently overwrites whichever lands second — the first user's edits are gone and nobody knows. What's the precondition that detects the race, what's the response that lets the user recover, and when is it correct to skip the precondition? The lesson lands version-based optimistic concurrency as the default, with `updatedAt` as the alternative, the 409 surface, and the React 19 refresh-and-retry flow.
- **Optimistic concurrency vs. pessimistic locking.** Optimistic: the client reads a version, sends it back on UPDATE, the server checks the version still matches before writing — no lock, no contention. Pessimistic: the client holds a row lock from read to write — guaranteed consistency, no surprises, but a network round-trip's worth of locking on every edit, and a forgotten release leaves rows stuck. For SaaS web traffic, optimistic is the default — collisions are rare, locks at scale are a perf bug, and the 409 path makes the rare collision honest. Pessimistic earns its weight only in narrow batch-processing or financial-ledger paths the course doesn't reach for.
- **The `version` column shape.** `version: integer().notNull().default(1)` — an integer that increments on every UPDATE. The Server Action's UPDATE includes `WHERE version = :clientVersion` and `SET version = version + 1` in the same statement. Rows affected: 0 means the version moved, the write is rejected with 409; 1 means the write landed and the version is now `clientVersion + 1`. The senior call: integer over UUID for the version — small, ordered, and the increment is atomic in SQL.
- **The `updatedAt` alternative — when to reach.** `WHERE updatedAt = :clientUpdatedAt` works as a precondition without adding a column. The pros: one less column, the timestamp is useful for display anyway. The cons: timestamps at second precision can collide on rapid writes; clock skew (between the client read and the server write) is a non-issue because both reads come from the server, but timezone bugs in serialization are common. The senior call: prefer `version` for entities with structured editing (forms, drafts, multi-field updates); reach for `updatedAt` only when adding a column is impossible (an existing table, frozen schema) and the precision is high enough (millisecond timestamps).
- **The Drizzle UPDATE shape.** `db.update(invoices).set({ ..., version: sql\`${invoices.version} + 1\`, updatedAt: sql\`NOW()\` }).where(and(eq(invoices.id, id), eq(invoices.orgId, orgId), eq(invoices.version, clientVersion), isNull(invoices.deletedAt))).returning()` — the `returning()` clause produces zero rows on conflict, one row on success. The Server Action branches on `result.length === 0` to return the 409. The senior anchor: every precondition is in the `WHERE` clause — tenancy, lifecycle, and version. Miss any, write the wrong row.
- **The Server Action Result shape on 409.** The action returns `{ ok: false, error: { code: 'CONFLICT', userMessage: 'This invoice was changed in another tab. Refresh to see the latest version.', current: <fresh row> } }` — the `current` field carries the server's current state so the client can render it without a second round-trip. The shape ties to the canonical Result from lesson 2 of chapter 047 and extends it with a typed error code the UI branches on.
- **HTTP 409 as the status code, RFC 9457 as the body shape.** Server Actions don't surface HTTP status codes directly to the client (they return the Result object), but route-handler endpoints that wrap the same action do — `409 Conflict` with an RFC 9457 Problem Details body (lesson 2 of chapter 015 owns the shape). External integrators get the right status; the in-app form gets the typed Result. The senior anchor: one error semantics, two transport shapes.
- **The React 19 client surface — `useActionState` plus `useOptimistic`, conflict path.** The form holds the current `version` as a hidden input (or as part of the controlled state). On submit, `useActionState` invokes the Server Action. On `{ ok: true }`, the form replaces local state with the returned row and shows a success indicator. On `{ ok: false, error: { code: 'CONFLICT' } }`, the form renders a "This was changed elsewhere" banner with the conflicting row's values, the user reviews, then clicks "Use latest and edit again" (which replaces the form state with `current` and resets `version`) or "Overwrite anyway" (rare, calls the action with `force: true` that bypasses the precondition — gated, often role-restricted).
- **`useOptimistic` interaction with conflicts.** Optimistic UI shows the user's edit instantly. On 409, the optimistic state is rolled back automatically by React (the action returned, the optimistic value is dropped). The form then renders the conflict banner against the user's original state (still in the controlled inputs) plus the server's `current`. The senior anchor: optimistic UI is fine for the common case; the rollback on conflict is automatic; the conflict UI is what the user actually sees on the rare collision.
- **The "last-write-wins is fine" carve-out.** A single-user toggle (a personal preference, a "show archived" flag scoped to one user), an append-only log entry (a comment, a status note that can never be edited), a counter increment driven by SQL itself (`SET count = count + 1` — no read-modify-write loop on the client). Each of these has no read-modify-write race because no client read informs the write. The senior call: skip the version column when there's no client-side read in the loop; add it when there is.
- **Idempotency keys vs. version preconditions — different bugs.** Idempotency keys (Chapter 067, foreshadowed at lesson 10 of chapter 047) prevent the same write from landing twice (double-click, retry on network blip). Version preconditions prevent two different writes from clobbering each other (two tabs editing the same row). Both can apply to the same Server Action — the idempotency key dedupes a retry, the version check detects the cross-user collision. The lesson names the distinction so the student doesn't conflate them.
- **The migration shape — adding `version` to existing tables.** `ALTER TABLE invoices ADD COLUMN version integer NOT NULL DEFAULT 1` — the default backfills existing rows in one statement, no rolling update needed. The migration is small; the wiring (the Server Actions, the forms) is the work. The senior anchor: the schema change is cheap, the discipline of including `version` in every UPDATE is what costs — start with the entities where two tabs is plausible.
- **Schema-level enforcement — the trigger option, named and dismissed.** A Postgres trigger `BEFORE UPDATE` that increments `version` automatically would centralize the increment. The trade-off: the trigger fires for all writes, including admin tooling that bypasses the helper. Hidden behavior is harder to reason about than an explicit `SET version = version + 1` in the Drizzle update. The senior call: keep the increment in the application layer alongside the precondition check — both are visible in the same call site.
- **The optimistic-mutation triplet, restated from lesson 2 of chapter 009.** The state machine: `idle -> submitting -> { success, conflict, error }`. The conflict state is the named transition this lesson exists to teach. The lesson restates the machine so the student sees the new state slotted into a model they already have.
- **Worked example — the invoice edit form.** The form takes the current invoice (with `version`) as a Server Component prop, renders the inputs with `useActionState` wrapping the update action, includes `version` as a hidden field. The Server Action validates, runs the UPDATE with the version precondition, returns the Result. The client renders success, conflict banner, or generic error. The example is the canonical shape for every edit form in the rest of the course.
- **Watch-outs:** UPDATE without the version precondition is a silent last-write-wins bug — name the entities where this applies; the version precondition in the WHERE without bumping the version in the SET creates an infinite-loop UI (every save succeeds, no retry needed) — pair the two; returning `current` in the error payload is necessary or the client needs a second fetch round-trip; the "Overwrite anyway" affordance is a sharp edge — gate it on role or hide it entirely depending on the product; `updatedAt`-based preconditions broken by clock skew between two app servers (not client clocks) are a real failure mode — version columns sidestep it; forgetting to update `updatedAt` on UPDATE breaks the audit trail even if `version` is correct — `sql\`NOW()\`` on every update; the version precondition that doesn't include the tenancy filter is the cross-tenant overwrite bug — the base helper from lesson 2 of chapter 065 carries both; serializing a `version` field through a form's hidden input requires it to not be user-tampered — Zod parses on the server (lesson 4 of chapter 047) is the defense, but a tampered version just produces a 409, so the impact is bounded.

What this lesson does not cover:

- Pessimistic locking (`SELECT ... FOR UPDATE`) — named as the alternative; not implemented.
- Idempotency keys for webhook ingestion — Chapter 067.
- The Server Action Result base shape — lesson 2 of chapter 047.
- `useOptimistic` mechanics in depth — lesson 6 of chapter 048.
- The transaction wrapper around multi-row updates — lesson 9 of chapter 047.
- Conflict resolution for collaborative editing (CRDTs, operational transforms) — out of course scope.
- Audit-log entries on edits — lesson 3 of chapter 085.

Estimated student time: 50 to 60 minutes. The most senior-mindset lesson of the chapter; the conflict UX is what separates a toy edit form from a production one.

## Lesson 4 — Quizz

Top 10 topics to quiz:

- Soft delete vs. hard delete — the senior trigger for each (cross-referenced records and compliance shadows go soft, ephemera goes hard); soft delete is a write, not a delete.
- Soft delete vs. archive — the user-facing distinction (invisible recovery vs. explicit lifecycle state), the two-timestamp schema (`deletedAt`, `archivedAt`), and the three Server Actions (`softDelete`, `archive`, `restore`).
- Partial unique indexes (`WHERE deleted_at IS NULL`) as the fix for the "can't re-create a soft-deleted slug" bug, and partial composite indexes on the hot active-row read path.
- Cascade-on-soft-delete as an application-level transaction (Drizzle `db.transaction`), not a database `ON DELETE CASCADE`.
- The base-query helper as the structural fix for missing `deletedAt IS NULL` — `active()`, `archived()`, `includingDeleted()` as the three-method API; composition with `tenantDb` from chapter 060.
- The shared `activeFilter`/`archivedFilter` predicates and the discipline that hand-written joins go through named functions in `/lib/queries`, each carrying the same filters.
- Optimistic vs. pessimistic concurrency — the SaaS web-traffic default is optimistic; pessimistic is for narrow batch/ledger paths the course doesn't reach.
- The `version` column as the precondition (preferred) and `updatedAt` as the alternative (when adding a column is impossible); the Drizzle UPDATE that increments `version` in the SET and checks it in the WHERE; zero rows affected becomes 409.
- The 409 Server Action Result shape — typed error code (`CONFLICT`), `userMessage`, and `current` row payload so the client renders the conflict without a second fetch; RFC 9457 Problem Details on the HTTP transport for external endpoints.
- The React 19 conflict UX — `useActionState` plus `useOptimistic`, automatic rollback on conflict, the "Use latest" vs. "Overwrite anyway" affordances, and the carve-out where last-write-wins is correct (single-user toggles, append-only writes, SQL-increment counters).
