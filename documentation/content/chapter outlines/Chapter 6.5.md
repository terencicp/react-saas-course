# Chapter 6.5 — Migrations and seeding

## Chapter framing

Chapter 6.5 closes Unit 6 by turning the schema from 6.2 and the queries from 6.3/6.4 into a workflow that ships. Two questions earn full treatment: how does a change to `db/schema.ts` become a SQL file that runs against production without dropping data or locking tables, and how does the dev/test database get realistic, repeatable data without hand-written `insert` scripts. The chapter installs Drizzle Kit as the migration tool, the `generate` → review → `migrate` discipline as the daily loop, the production hand-edits the generator can't emit on its own (`CREATE INDEX CONCURRENTLY`, triggers from 6.2.4, custom constraints, the safe-rename two-step), and `drizzle-seed` as the deterministic seeder the project chapter and every test in later units consume.

Threads that run through every lesson: the migration file is checked-in source code, reviewed in PRs, and immutable once merged — never edited after the fact, never re-ordered; `drizzle-kit generate` is the senior default everywhere except a local prototype, `drizzle-kit push` is the local-only escape hatch with the failure mode (silent column drops) named explicitly; the migration runner in production is `drizzle-kit migrate` against the unpooled `DATABASE_URL_UNPOOLED` from 6.1.4, never the pooled URL; every migration on a table with material write traffic uses `CONCURRENTLY` for indexes and the expand-backfill-contract pattern for column changes — the rename-in-place is the production footgun this chapter teaches you to avoid; seed scripts are deterministic, idempotent, and FK-aware — running them twice produces the same database state, running them against a non-empty dev branch doesn't corrupt it. The chapter ships three teaching lessons plus a quiz, ordered by the workflow: daily mechanic, production discipline, seeding, quiz.

## Lesson 6.5.1 — The Drizzle Kit daily loop

Teaches the `drizzle.config.ts` contract, the migration folder layout, the `generate` → review → `migrate` workflow, and Drizzle Studio as the in-stack dev GUI.

Topics to cover:

- **The senior question.** A change in `db/schema.ts` (a new column, a new table, a new index) — what's the smallest, repeatable sequence that turns it into a SQL file checked into the repo and applied to every environment? Drizzle Kit's `generate` and `migrate` commands are the answer; the studio is the in-stack GUI for the dev database.
- **Drizzle Kit as the migration tool, named.** A standalone CLI that reads `db/schema.ts` and emits versioned SQL migration files. Lives in `devDependencies`. Configured via `drizzle.config.ts` at the repo root. The senior reach because it stays in the same TypeScript codebase as the schema — no separate migration DSL, no manual SQL drift.
- **`drizzle.config.ts` — the contract.** Exports a `defineConfig({ ... })` with: `dialect: 'postgresql'`, `schema: './src/db/schema.ts'` (or the glob if split across files), `out: './drizzle'` (the migration folder), `dbCredentials: { url: process.env.DATABASE_URL_UNPOOLED! }` (the unpooled URL from 6.1.4), `casing: 'snake_case'` (matches the convention from 6.2.2), and `verbose`/`strict` flags. The same file drives `generate`, `migrate`, `push`, and `studio` — one config, four commands.
- **The migration folder layout.** `drizzle/0000_<adjective>_<noun>.sql` files plus a `meta/` directory holding `_journal.json` (ordered list of applied migrations) and per-snapshot JSON files (the schema state at each step, used by `generate` to diff). Both the SQL and the meta files are committed; nothing else.
- **`drizzle-kit generate` — the daily move.** Reads `db/schema.ts`, diffs it against the latest snapshot in `meta/`, and emits a new numbered `.sql` file plus an updated snapshot. The dev's job: edit the schema, run `generate`, review the emitted SQL, edit if needed (6.5.2), commit both files together. The reflex: the migration file is generated, not hand-written from scratch — but it can be edited before commit.
- **`drizzle-kit migrate` — applying it.** Connects to the database at `DATABASE_URL_UNPOOLED` (unpooled because migration DDL holds long transactions that don't fit transaction-mode pooling, 6.1.4) and applies pending files in journal order. Tracks state in a `__drizzle_migrations` table the runner manages. Idempotent: applying twice is a no-op. Reach: `pnpm drizzle-kit migrate` in dev; CI runs it against staging; the production deploy pipeline runs it against production before the new app version goes live.
- **The `migrate()` function in app code — when to reach.** `drizzle-orm/node-postgres/migrator`'s `migrate(db, { migrationsFolder })` runs the same operation programmatically. Reach when CI scripts or app startup needs to apply migrations without spawning the CLI. Same `__drizzle_migrations` table, same journal. Named for recognition; the course's default is the CLI in the deploy pipeline.
- **The PR review loop.** A PR that changes the schema includes the generated `.sql` and `meta/` snapshot. The reviewer reads the SQL, not the schema diff — the SQL is what runs. This is the moment the silent-column-drop failure mode of `push` is avoided (full coverage in 6.5.2).
- **Naming migrations — the `--name` flag.** `drizzle-kit generate --name add_invoices_status_index` makes the filename grep-able later. The default random name (`0007_brave_phoenix.sql`) is fine for a course example but the senior reach in production is named migrations so `git log -- drizzle/` reads as a changelog.
- **`drizzle-kit studio` — the GUI.** Spins up a local web UI at `https://local.drizzle.studio` that talks to the database in `dbCredentials.url` and shows the tables, lets the dev browse rows, run ad-hoc queries, and edit data. Schema-aware: it knows the relations from `db/relations.ts` and lets the dev walk them. The senior reach: dev-database inspection, sanity-checking a seed run, eyeballing a query result during development. Not a production tool — credentials sit in `drizzle.config.ts`, the UI has no auth model.
- **Studio vs. external GUI tools, briefly.** TablePlus, pgAdmin, DataGrip, Neon's web console all work against the same Postgres. Studio's edge: schema-aware traversal via `defineRelations`, zero setup beyond `drizzle.config.ts`. Most other tools win on power-features (query plan visualization, schema export, multi-tab editing). Pick one; the course uses Studio.
- **The CLI commands at a glance.** `drizzle-kit generate` (emit migration), `drizzle-kit migrate` (apply), `drizzle-kit push` (skip the file, apply diff direct — 6.5.2 owns this), `drizzle-kit pull` (introspect existing DB into a schema file — named for recognition, reach when adopting Drizzle on an existing database), `drizzle-kit studio` (GUI), `drizzle-kit drop` (revert a migration from the journal — 6.5.2 owns the production rule against it), `drizzle-kit check` (lint the migration history for collisions when two branches each added one).
- **Worked example.** Add an `invoices.archivedAt` column to `db/schema.ts`. Run `drizzle-kit generate --name add_invoices_archived_at`. Show the emitted SQL (`ALTER TABLE "invoices" ADD COLUMN "archived_at" timestamptz`). Run `drizzle-kit migrate`. Confirm in Studio that the column exists with the expected type. Commit the schema change, the migration file, and the updated snapshot together.
- **The package.json scripts — the senior shape.** `db:generate`, `db:migrate`, `db:studio`, `db:push` as wrapper scripts so the team types one command, not the full CLI invocation. The unpooled URL is loaded by the script (typically via `dotenv-cli` or the framework's env loader). The course's project starters ship these scripts.
- **Watch-outs:** running `migrate` against the pooled URL trips on long DDL transactions — always the unpooled URL; two branches that each add a migration produce a journal conflict at merge — `drizzle-kit check` flags it, the senior fix is to renumber and re-generate the second one; editing an already-applied migration file in main is the worst footgun in this stack — the snapshot drifts from the database state, the next `generate` produces nonsense; the `meta/` snapshot files must be committed alongside the SQL — leaving them out breaks every teammate's next `generate`; Studio's edit-rows affordance writes to the dev database — same `DELETE` footgun as a missing-`where` mutation, just behind a button.

What this lesson does not cover:

- The push vs. generate decision in depth, hand-editing the SQL, zero-downtime patterns — Lesson 6.5.2.
- Seeding the database with realistic fixtures — Lesson 6.5.3.
- The pooled vs. unpooled URL decision — Chapter 6.1.4.
- Drizzle's `db` client wiring (separate from Drizzle Kit) — Chapter 6.2.
- The schema authoring itself (`pgTable`, columns, relations) — Chapter 6.2.
- Expand-backfill-contract for column changes at depth — Chapter 21.4 owns the production rollout pattern; Lesson 6.5.2 introduces it.

## Lesson 6.5.2 — Production-safe migrations

Teaches the push-vs-generate decision, the five-question SQL review checklist, `CREATE INDEX CONCURRENTLY` with statement breakpoints, the column-change patterns that bite, and expand-backfill-contract as the zero-downtime reflex.

Topics to cover:

- **The senior question.** Drizzle Kit emits SQL that's correct in form but not always production-safe — when does the dev hand-edit the file before commit, when does the auto-generated SQL ship as-is, and when is `push` the right tool instead? The lesson lands the production-safety checklist, the push-vs-generate decision, and the three column-change patterns that survive a live deploy.
- **`drizzle-kit push` — the prototype tool.** Reads the schema and applies the diff directly to the database, skipping the SQL file. Wins for: a solo prototype before the schema has stabilized, throwaway feature branches against an ephemeral Neon branch, tutorial/demo work. Loses for: any environment with data the team cares about. The failure mode: ambiguous diffs (a column rename looks identical to a drop-and-add) get resolved by silent column drop, data lost without a SQL review step. Senior rule: `push` only against a database whose data you'd happily wipe; `generate` for everything else.
- **`generate` as the default.** The reasons: the SQL file is reviewable in a PR (the migration is code, the review catches destructive operations); the file is replayable (CI applies it against staging, production applies the same bytes, drift is zero); the file is editable (hand-tweaks for `CONCURRENTLY`, triggers, or comments survive). Any team larger than one defaults to generate.
- **Reading the emitted SQL — the review checklist.** Five questions for every PR with a migration: (1) Does anything DROP — column, table, constraint, index? If yes, is it safe in this deploy? (2) Is any rename a true rename, or is Drizzle Kit being asked to disambiguate a remove-and-add? (3) Does any new index on a hot table need `CONCURRENTLY`? (4) Does any column-type change rewrite the table (the planner-blocking case)? (5) Does any new `NOT NULL` column without a default fail against existing rows? Each question maps to a hand-edit or a multi-step rollout.
- **The three column-change patterns that bite.**
  - **Adding a `NOT NULL` column with no default.** Fails on existing rows. The fix: add nullable first, backfill, then alter to `NOT NULL` in a later migration. Or: ship with a `DEFAULT` value the existing rows accept. Drizzle Kit emits the naive form; the dev's job to recognize it.
  - **Renaming a column.** The auto-generated SQL is correct, but the deploy is the trap: the old app version queries the old column name; the new app version queries the new name; the migration runs at the boundary, leaving one of them broken for the cutover. The pattern: add the new column, dual-write from app code, backfill, switch reads, drop the old column in a later deploy. Three migrations, two deploys. Owned at depth in Chapter 21.4; named here as the production reflex.
  - **Changing a column type.** Postgres rewrites the whole table for incompatible type changes, holding an `ACCESS EXCLUSIVE` lock the entire time — every read and write blocks. The fix: add a new column with the new type, backfill in batches, switch reads, drop the old column.
- **The expand-backfill-contract pattern — named, owned later.** The shape every safe schema change takes when zero-downtime matters: expand (add the new shape alongside the old), backfill (write the new values for existing rows), contract (drop the old shape once nothing reads it). Three deploys minimum for a destructive change. Foreshadowed for Chapter 21.4; the student needs the vocabulary here so they recognize the trigger.
- **Indexes — the `CONCURRENTLY` hand-edit.** Drizzle Kit emits `CREATE INDEX "..." ON "..." (...)`. On any table with material write traffic, this locks writes for the duration of the index build — minutes to hours on a large table. The fix is one hand-edit: change `CREATE INDEX` to `CREATE INDEX CONCURRENTLY`. Caveats: `CONCURRENTLY` can't run inside a transaction (Drizzle Kit wraps each migration in one), so the migration file needs `--> statement-breakpoint` markers around the index DDL so Drizzle's runner commits the surrounding work first, then runs the index outside the transaction. Course rule: every index on a non-trivial table gets `CONCURRENTLY` plus the breakpoint dance.
- **Statement-breakpoint — the Drizzle migration runner's split marker.** A comment-style directive in the SQL file (`--> statement-breakpoint`) that tells the runner to commit and start a new transaction at that line. Reach: `CREATE INDEX CONCURRENTLY` (can't be in a transaction), `CREATE EXTENSION` (must commit before use), some `ALTER TYPE` patterns. Drizzle Kit inserts breakpoints automatically for some statements; the dev adds them when hand-editing in others.
- **DDL Drizzle Kit doesn't emit cleanly — the manual SQL block.** Generated columns with complex expressions, triggers (the `updatedAt` trigger from 6.2.4), `CREATE EXTENSION`, custom constraints, partial unique indexes with `WHERE` clauses that reference computed values. The pattern: write the schema in `db/schema.ts` for the parts Drizzle Kit handles, write a `sql\`...\`` block at the bottom of the generated migration file for the rest. The migration file is the source of truth for the database state; the schema is the source of truth for the Drizzle types — the two stay in sync because the next `generate` re-reads the schema.
- **Down migrations — named, deferred, mostly avoided.** Drizzle Kit emits `up` migrations only; there's no `down` file. The senior reach in 2026: forward-only migrations, never roll back in production — instead, the next migration corrects the previous one (forward-fix). Reach for `drizzle-kit drop` only in dev to remove the most recent unapplied migration before re-generating. Production never reverses; it forward-fixes.
- **Migration failures mid-run — the recovery shape.** A migration that fails partway leaves the `__drizzle_migrations` table without an entry for that file (most failures roll back the transaction). The dev investigates, fixes the SQL or the data, re-runs `migrate`. If the failure was outside a transaction (a `CONCURRENTLY` index that built halfway), Postgres marks the index invalid; the recovery is `DROP INDEX` and re-create. The senior reflex: every migration's failure mode is rehearsed against staging before it touches production.
- **The CI and deploy story.** CI runs `drizzle-kit migrate` against the staging branch (a long-lived Neon branch from 6.1.3) on every merge to `main`; the production deploy pipeline runs `migrate` as a pre-deploy step before swapping app versions. Vercel's "build and deploy" hook is the typical seam. Naming: `db:migrate:prod` in `package.json` so the script is grep-able in the pipeline. The unpooled URL is wired into the deploy env, separate from the pooled URL the app uses.
- **Verifying the migration applied — the production reflex.** After a deploy, query `select * from __drizzle_migrations order by created_at desc limit 5;` against the Neon production branch. The latest row's hash matches the latest committed file's hash. If they don't match, the deploy applied something else — investigate before serving traffic.
- **The push escape hatch, restated.** Local prototyping against a throwaway database: `drizzle-kit push` is faster and fine. Against anything else — shared dev branch, staging, production — generate-and-migrate is the only rule.
- **Watch-outs:** editing a migration file after it's been applied anywhere (even just on the dev's laptop) drifts the `meta/` snapshot from the database state — the senior fix is to revert the file and generate a corrective forward migration; running migrations against the pooled URL truncates long DDL silently — always the unpooled URL (course rule from 6.1.4); the rename-in-place + same-deploy is the canonical "I broke prod" mistake — expand-contract is non-negotiable for renames in any live system; `CREATE INDEX` without `CONCURRENTLY` on a hot table is an outage waiting to happen — the hand-edit is the senior reflex; trusting `drizzle-kit push` against staging because "it's not prod" still loses data when staging is shared across the team; `auto_explain` on slow migration statements is a useful production-debug tool (named in 6.4.3).

What this lesson does not cover:

- The expand-backfill-contract pattern at full depth, including app-code dual-writes and the migration choreography — Chapter 21.4.
- Generating the migration files and the daily workflow — Lesson 6.5.1.
- Seeding the database — Lesson 6.5.3.
- Index types and the `CONCURRENTLY` flag in the context of when to add an index — Chapter 6.4.1 (this lesson is the migration-file mechanic).
- The `updatedAt` trigger or generated columns mechanics — Chapter 6.2.4.
- Production deployment pipelines at depth — Unit 21.

Estimated student time: 50 to 60 minutes. Load-bearing for every schema change a student ships in a later unit and for the production-deploy patterns in Chapter 21.

## Lesson 6.5.3 — Deterministic seeding with drizzle-seed

Teaches the `seed().refine(...)` call shape, FK-aware insertion order, the idempotent reset-and-seed script, the same-seed determinism guarantee, and the test-factory boundary.

Topics to cover:

- **The senior question.** A fresh dev branch or a CI test database — how does it get filled with enough realistic data to render the project's list pages, paginate, and exercise edge cases, without hand-writing `INSERT` statements? `drizzle-seed` is the schema-aware, deterministic, FK-resolving seeder Drizzle ships; the lesson lands the call shape, the determinism guarantee, the idempotent reset pattern, and the test-factory boundary.
- **`drizzle-seed` named, characterized.** A devDependency that reads the same Drizzle schema and generates fake data shaped to the columns' types and constraints. Deterministic: a fixed seed number produces the exact same data every run. FK-aware: it reads `references()` declarations and inserts parents before children. Versioned: the library pins value-generation behavior to a version so upgrades don't silently change seed output.
- **The minimal call shape.** `import { seed } from 'drizzle-seed'; await seed(db, schema, { seed: 1 }).refine((f) => ({ ... }));`. `db` is the Drizzle client; `schema` is the bag of `pgTable` exports; `seed` is the determinism knob. `.refine(...)` is the per-table fine-tuning: column-level value generators, row counts, weighted distributions, `with` for parent-child counts.
- **Default behavior — ten rows per table.** Without `.refine`, the seeder inserts ten rows per table with type-appropriate fake values. Useful for a smoke test; rarely enough for a real dev experience. The senior reach: `.refine` for any table the project's list pages exercise.
- **`.refine` — the per-table generator object.** `(f) => ({ invoices: { count: 200, columns: { status: f.weightedRandom([{ weight: 0.6, value: f.valuesFromArray({ values: ['paid'] }) }, { weight: 0.4, value: f.valuesFromArray({ values: ['pending', 'overdue'] }) }]), total: f.number({ minValue: 100, maxValue: 10000, precision: 100 }) }, with: { lineItems: 5 } } })`. The shape: `count` for row count, `columns` for per-column generators, `with` for child-row counts per parent.
- **The value generators.** `f.firstName`, `f.lastName`, `f.email`, `f.companyName`, `f.streetAddress`, `f.country`, `f.boolean`, `f.number`, `f.int`, `f.date`, `f.datetime`, `f.timestamp`, `f.uuid`, `f.valuesFromArray` (pick from a fixed set), `f.weightedRandom` (weighted pick), `f.default` (use the column's schema default). Faker-style breadth without bundling Faker. Senior signal: read the generator list once, reach for `valuesFromArray` when an enum is involved, reach for `weightedRandom` when the distribution matters (the "most invoices are paid" shape is what makes a seeded dataset feel real).
- **FK resolution.** `drizzle-seed` reads the schema's `references()` and inserts in topological order — organizations before invoices, invoices before line items, no manual ordering. The `with` option declares the parent-child fanout (`with: { lineItems: 5 }` makes each invoice get five line items). The student doesn't compute IDs; the seeder picks valid parent IDs per child.
- **Determinism — the same-seed guarantee.** Two runs with `{ seed: 1 }` produce identical data. Two devs on the team running the seed get the same dataset to debug against. CI's seed run produces the same data as local. The reflex: pin the seed number in the script; vary it only to explore different shapes.
- **The reset workflow — idempotent dev databases.** The course's seed script is two steps: `await reset(db, schema)` (clear all rows in FK-safe order, leaving the schema intact) then `await seed(db, schema, { seed: 1 }).refine(...)`. Re-runnable; running it twice ends in the same state. The dev's reflex: when the local data gets weird, `pnpm db:seed` and start clean.
- **The `db:seed` script — the package.json contract.** `tsx scripts/seed.ts` invoked via a `db:seed` script. The script imports the unpooled `db`, the schema, and the seed config, runs reset then seed, exits. Production never runs it; the script's environment is dev or test only. Course rule: the seed script `process.exit(0)`s on success and `process.exit(1)`s on failure so CI can gate on it.
- **Seeding for tests — the per-test reset.** Integration tests against a real database (Vitest pattern in later units) run the seeder in a `beforeEach`: reset, seed a small dataset for the test, run the assertions. Determinism keeps tests reproducible. For tests where the seeded distribution doesn't match the scenario, the test inserts the specific rows it needs after the seed — small targeted inserts on top of a known baseline.
- **The test-factory alternative — named, scoped.** For unit-level tests that want one-off rows without the full seed, factory helpers (`makeInvoice({ status: 'paid', total: '100.00' })`) are the cleaner reach. The factory inserts directly via Drizzle and returns the row. `drizzle-seed` is the dataset shape; factories are the per-test row shape. The course defaults to the seeder for fixtures, factories for ad-hoc test setup.
- **What `drizzle-seed` doesn't generate well.** Domain-specific data that real users would enter (the body of a support ticket, the description of an invoice with vocabulary that fits the SaaS) — the seeder produces lorem-ipsum-grade text, which is fine for shape but flat for screenshots. The senior fix: `valuesFromArray` with a hand-curated list of realistic strings for the columns demos will surface; lorem-grade is fine for the rest.
- **Production seeding — named, dropped.** Some apps need "fixture" data in production (the default workspace on signup, the seed roles for RBAC, the system organization). That's not `drizzle-seed`'s job — it's a one-shot data migration written by hand, run once, then tombstoned. Named so the student doesn't reach for the seeder for prod data.
- **Worked example.** A seed script for the project: 5 organizations (`f.companyName`), 200 invoices total (`with: { lineItems: 3-8 }`), weighted status distribution (`paid` 60%, `pending` 30%, `overdue` 10%), realistic `customerName` and `customerEmail` from `valuesFromArray` of curated names, `createdAt` spread across the last six months via `f.date({ minDate, maxDate })`. Run, open Studio, eyeball the list. Run again with `seed: 1`, confirm identical rows.
- **Watch-outs:** seeding against the pooled URL is fine for `drizzle-seed`'s short inserts but the reset uses `TRUNCATE ... CASCADE` which holds locks — unpooled URL is the safe default for both reset and seed; forgetting to call `reset` before `seed` on a non-empty database produces FK violations or duplicates from the previous run — wrap both into one script; the `with` count is per parent, not total — 5 orgs with `invoices: 200` is 1000 invoices, not 200; `valuesFromArray` with a list of values smaller than the row count makes duplicates the norm — fine for status, wrong for emails (use a unique generator or compose with the row index); changing the seeder's `.refine` config in a way that shifts column order or table count invalidates the determinism contract on the same seed number — bump the seed value when the shape changes.

What this lesson does not cover:

- The schema authoring itself — Chapter 6.2.
- Migrations and Drizzle Kit — Lessons 6.5.1 and 6.5.2.
- Integration testing patterns at depth (Vitest, the per-test database setup) — Unit 18.
- Production data migrations (the one-shot kind) — Chapter 21.4.
- Faker.js or Mockoon as external alternatives — `drizzle-seed` covers the in-stack workload; named and dropped.
- Generating production-realistic data at scale (millions of rows for performance testing) — out of scope; `drizzle-seed` handles the dev and test volumes the course needs.
