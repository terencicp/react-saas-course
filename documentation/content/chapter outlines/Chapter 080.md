# Chapter 080 — TanStack Query

## Chapter framing

TanStack Query is the first of two conditional client-state tools in Unit 16. Every prior chapter has pushed read state to Server Components, write state to Server Actions, transient UI state to `useState`, and shareable view state to the URL with `nuqs`. Those four cover the SaaS surface a senior actually ships. TanStack Query is the conditional reach past them — the library you bring in when the workload genuinely is client-side server-state (polling, frequent refetches, complex optimistic mutations with rollback across views, infinite-scroll lists that need cache reuse) and the framework's defaults stop pulling their weight. The chapter's job is to make the threshold explicit, teach the primitives the student will write at the call site, wire the library correctly against the App Router (this is the trap — wrong wiring leaks the singleton across requests on the server), and walk one concrete case in our app where the threshold is met. The chapter ends without a project; chapter 081 turns the threshold case into a buildable surface.

Threads through every lesson: TanStack Query is **conditional**, not default — the lesson opens by naming what already works and only then names what doesn't; the four named triggers (polling/frequent refetches, cross-view client cache, optimistic with rollback, infinite scroll with cache reuse) are the only justifications the course accepts; cache invalidation responsibility shifts from `updateTag`/`revalidateTag` (Server Component world) to `queryClient.invalidateQueries({ queryKey })` (TanStack world), and the two systems are kept separate on purpose; the QueryClient is created per-request on the server (never module-scoped) to avoid the cross-request leak that has its own discussion thread on the library's repo; SSR-hydrated initial data is the default integration shape, not client-only fetching — the page is still a Server Component that prefetches, then `<HydrationBoundary>` rehydrates on the client; mutations use the v5 optimistic-update-via-`variables` shape when the optimism is local, the cache-write shape when downstream queries need the optimistic row to flow through; the library is **client-only** — never imported into Server Components, never inside a Server Action.

## Lesson 1 — When TanStack Query earns its weight

The four named triggers (polling, cross-view caching, optimistic with rollback into cached queries, infinite scroll with reuse) that justify reaching past Server Components, Server Actions, `useState`, and `nuqs` for a client-side server-state library.

Topics to cover:

- **The senior question.** Server Components fetch and stream data. Server Actions mutate and call `updateTag` for read-your-writes. `useState` holds transient UI state. `nuqs` puts shareable view state in the URL. What's left? When does a SaaS feature genuinely need a client-side server-state library, and what's the threshold the four defaults stop covering? The lesson answers it explicitly so the rest of the chapter has a justification.
- **The four named triggers — what crosses the threshold.** The course accepts four (and only four) reasons to bring TanStack Query into a SaaS codebase:
  - **Polling and frequent refetches.** A comment thread, a notification badge, a job-status panel — anything where the client expects to learn about server changes on a tight cadence without a user action. Server Components plus `router.refresh()` works for occasional refresh; once the cadence drops below "user-initiated", the per-refresh request waterfall and full-segment re-render is the wrong shape.
  - **Complex client-side caching across views.** A detail page and a list page reading the same row; navigating between them should not refetch from the network. The Next.js Router Cache covers prefetched soft navigations; `use cache` covers the server side. When the surface is a deeply interactive client tree (a sidebar that mounts/unmounts the same data, a tab strip switching views of the same query) the explicit client cache is the right tool.
  - **Optimistic mutations with rollback.** `useOptimistic` handles the single-list, immediate, rolled-back-on-failure case (lesson 5 of chapter 048). When the optimism needs to flow into multiple cached queries simultaneously, when the rollback shape is complex (mutation A's failure should restore values mutation B depended on), or when the optimistic update must persist across a navigation — `useMutation`'s `onMutate`/`onError` rollback against the client cache is the senior reach.
  - **Infinite scroll with cache reuse.** A long thread, a feed, a chat — the user scrolls down to load more, then scrolls back up; the already-loaded pages should still be there. `useInfiniteQuery` plus `maxPages` is the canonical shape. The Server Component cursor pagination from lesson 4 of chapter 064 produces the right URL but loses already-fetched pages on each "Next" click; infinite scroll is a different UX with a different cache contract.
- **The conditional framing — what does NOT clear the bar.** A list view with filter/sort/search/cursor pagination — no, `nuqs` plus Server Components is the right shape (Chapter 064). A form that submits and revalidates — no, Server Actions plus `useActionState` (Chapter 048). A single optimistic toggle — no, `useOptimistic`. Reading the session — no, `auth()` in a Server Component. The senior reflex: enumerate which default the workload would otherwise use; only if every default is wrong does TanStack Query earn its weight.
- **The cost of bringing the library in.** A second cache to reason about; a second invalidation surface (`invalidateQueries` vs. `updateTag`); a second mental model for "where does this query live and when does it refetch"; a runtime dependency (well-maintained, but a dependency); a Client Component boundary forced wherever a `useQuery` lives. Each bullet shrinks the surface the senior should be willing to apply it on. The library is right for the four triggers, wrong for everything else.
- **The integration shape — Server Components own the initial render, TanStack owns the live cache.** The default shape, not negotiable in this course: the page is a Server Component that prefetches (`queryClient.prefetchQuery`) the same data the client will read, then dehydrates into a `<HydrationBoundary>` so the first paint is server-rendered and the client cache is populated. No `useQuery` ever fires a cold request on initial page load; the network round-trip happens in the RSC payload. Subsequent interactions (polling, optimistic mutations, scroll-fetching the next page) use the client cache. This shape pays the price of TanStack Query only for the parts of the surface that need it; the static/initial parts stay Server-Component-fast.
- **Comparison vs. SWR — one paragraph, why TanStack Query is the pick.** SWR is the older, smaller alternative from the Vercel team; works for simple revalidation patterns. TanStack Query has the richer mutation surface (optimistic-via-variables, paused mutations, the `useMutationState` shared-state hook), infinite queries with `maxPages`, the better devtools, and broader ecosystem reach. The course picks TanStack Query as the canonical client-side server-state library and does not teach SWR. If a codebase already runs SWR for these workloads, it's fine; the mental model transfers.
- **The "do we even own this state on the client?" check.** Before reaching for any of this, ask: is this state actually the server's? URL state, form-input state, theme preference — none of those are server state. The optimistic-update-with-rollback shape only earns its weight when the rollback target is server state. The mental check keeps the library scoped to its job.
- **Decision tree — the four-trigger checklist as the gate.** A diagram (mermaid) showing the funnel: "is this server state?" → "would Server Components plus `router.refresh()` cover it?" → "would `useOptimistic` cover it?" → "would Server-Component cursor pagination cover it?" → "would Server Components and the soft-navigation cache cover it?" — only if all four fall through does TanStack Query earn its weight.
- **The "in our app" pointer.** This chapter's worked case is the per-invoice comment thread (lesson 4 of chapter 080). The chapter 081 project builds the full surface against it. The lesson names the surface once and lets the later lessons drill in.
- **Watch-outs:** reaching for TanStack Query because "we always used React Query" — that was 2022; the defaults have moved; restate the trigger; using `useQuery` for static data that a Server Component would render — pure overhead, no win; treating `useQuery` as a generic state manager — that's what `useState` and Zustand are for, query is server state only; running `useQuery` in a route that's already a Server Component with prefetch — duplicate fetches; bringing the library in for one screen and never using it again — the cost ledger doesn't justify it, push back to `useEffect` plus an action, or accept the slight UX cost of `router.refresh()`.

What this lesson does not cover:

- The TanStack Query primitives themselves — lesson 2 of chapter 080.
- The App Router wiring (provider, hydration boundary) — lesson 3 of chapter 080.
- The worked screen in our app — lesson 4 of chapter 080.
- Zustand and global client state — Chapter 082.
- `useOptimistic` and the React 19 optimistic primitive — lesson 5 of chapter 048 (prerequisite).
- The Server Component cache (`use cache`, `cacheTag`, `cacheLife`) — Chapter 036 / chapter 076.
- Real-time via WebSockets or Server-Sent Events — out of scope for this course.

Estimated student time: 35 to 45 minutes. The chapter's load-bearing decision lesson — every other lesson is conditional on the threshold landing here.

## Lesson 2 — The four primitives the project reaches for

`useQuery` (with `staleTime` and the `isPending`/`isFetching` split), `useMutation` (the five-callback lifecycle), the v5 optimistic-update two-shape decision (via-variables vs. cache-update), and `useInfiniteQuery` with `maxPages` — plus query keys as the cache contract, `refetchInterval` polling, and the `invalidateQueries`/`setQueryData`/`removeQueries` trio.

Topics to cover:

- **The senior question.** Once the threshold is crossed, what's the API surface a senior actually writes? The lesson is a focused tour of the four primitives the chapter's project will reach for — `useQuery`, `useMutation`, the v5 optimistic-update patterns, and `useInfiniteQuery` — with the senior call on each.
- **`useQuery` — the read primitive.** `const { data, error, isPending, isFetching } = useQuery({ queryKey, queryFn, staleTime })`. The query key is the cache identity — an array, structured (`['comments', invoiceId]`), serialized internally for cache lookups; the `queryFn` returns a Promise; `staleTime` controls when a refetch is allowed (default 0 means every mount refetches, which is rarely what you want — the senior default for SaaS reads is `staleTime: 30_000` or higher). The `isPending` vs. `isFetching` distinction: `isPending` is "have I ever resolved?" (true until the first success), `isFetching` is "is a network request currently in flight?" (true on background refetches). Render the skeleton on `isPending`, the subtle spinner on `isFetching`.
- **Query keys as the cache contract.** Hierarchical arrays (`['comments', invoiceId]`, `['comments', invoiceId, { sort: 'newest' }]`) so `invalidateQueries({ queryKey: ['comments', invoiceId] })` matches both. The senior pattern: a `queryKeys.ts` file with helper functions (`commentKeys.list(invoiceId)`, `commentKeys.detail(commentId)`) so raw arrays don't leak across the codebase — same discipline as `tags.ts` in lesson 1 of chapter 076, applied at the client cache layer.
- **`queryFn` — what it returns.** A Promise that resolves to the data shape `useQuery` exposes, or throws on failure. The course pattern: the `queryFn` calls a route handler (Chapter 050), the route handler is the seam between the client cache and the database. Server Actions are not the right call site — they're designed for form submissions with redirects, not pure-read fetches that need stable error semantics. The senior call: TanStack Query reads from route handlers, mutates can go either way (handlers for the contract-driven case, Server Actions for the in-app shortcut).
- **`useMutation` — the write primitive.** `const { mutate, mutateAsync, isPending, error } = useMutation({ mutationFn, onMutate, onError, onSuccess, onSettled })`. The five-callback lifecycle: `onMutate` fires before the mutation, returns a `context` rolled forward; `onError` fires on failure with access to that context for rollback; `onSuccess` fires on success; `onSettled` fires either way (the cleanup seat). The senior call: `mutate` is fire-and-forget for the common case; `mutateAsync` returns a Promise for compose-with-await flows (chained mutations, redirect-after-mutate).
- **Optimistic updates — the v5 two-shape decision.** Two patterns, named explicitly:
  - **The "via variables" shape (newer, simpler).** `useMutation` exposes the in-flight `variables` and `isPending` to the component; render the optimistic row inline by reading `variables` directly. No cache write, no rollback code — the optimistic row vanishes automatically when the mutation settles (success refetches the list, failure shows the error). Right for the simple case: one list, one optimistic add, one rollback path.
  - **The "cache update" shape (older, more powerful).** `onMutate` snapshots the current cache (`queryClient.getQueryData`), writes the optimistic value (`queryClient.setQueryData`), returns the snapshot as context; `onError` restores from the snapshot; `onSettled` invalidates. Right when the optimistic update must flow into multiple queries, persist across navigations, or interact with other in-flight mutations. For paged data from `useInfiniteQuery`, type the snapshot explicitly with TanStack Query's `InfiniteData<TPage>` (e.g. `queryClient.getQueryData<InfiniteData<CommentPage>>(commentKeys.list(invoiceId))`) so the `pages` / `pageParams` shape is typed when the updater rewrites `data.pages[0]`.
  - The senior reach: start with the "via variables" shape, escalate to "cache update" only when the simpler shape doesn't fit. The chapter 081 project uses cache-update because the optimistic comment must appear in the `useInfiniteQuery`'s first page — a cache write, not an inline render.
- **`useInfiniteQuery` — the cursor-paginated cache.** `const { data, fetchNextPage, hasNextPage, isFetchingNextPage } = useInfiniteQuery({ queryKey, queryFn, getNextPageParam, initialPageParam, maxPages })`. The `queryFn` takes `{ pageParam }` (the cursor); `getNextPageParam(lastPage)` returns the next cursor or `undefined` to stop; `data.pages` is an array of page results, flattened with `data.pages.flatMap(p => p.rows)` at the render site. `maxPages` (v5) caps the number of pages held in the cache — beyond it, the oldest page drops, refetched if the user scrolls back. The senior call: `maxPages: 10` for chat-style threads; `maxPages: undefined` (unbounded) for feeds where re-entry to old positions is rare.
- **Cache invalidation — `invalidateQueries`, `setQueryData`, `removeQueries`.** Three calls cover the surface. `invalidateQueries({ queryKey })` marks every matching query as stale and refetches the active ones; `setQueryData(key, updater)` writes a value directly (the optimistic path); `removeQueries({ queryKey })` evicts entries entirely (rare, used on sign-out or org switch). The senior anchor: `invalidateQueries` is the default after a mutation; `setQueryData` is the optimistic shortcut; `removeQueries` is the tenancy-switch sledgehammer.
- **Polling — `refetchInterval`.** `useQuery({ queryKey, queryFn, refetchInterval: 10_000 })` polls every 10 seconds. The function form `refetchInterval: (data) => done ? false : 5000` stops polling when a condition is met (job complete). The senior call: pair `refetchInterval` with `refetchIntervalInBackground: false` (the default) so polling pauses when the tab is hidden — the user's battery and the database's connection pool both benefit. Polling is one of the four triggers; this is the syntactic shape.
- **`useQueryClient` — the imperative escape hatch.** `const queryClient = useQueryClient()` inside any Client Component returns the singleton wired up by `<QueryClientProvider>`. Use it in event handlers and mutation callbacks for `invalidateQueries`, `setQueryData`, `cancelQueries` (cancel in-flight before an optimistic update to prevent the network response from overwriting the optimism). The cancel-before-optimistic-update pattern is in the official optimistic-update example for a reason.
- **`useMutationState` — shared mutation state across components.** Read the state of every in-flight mutation matching a filter, from any Client Component, without prop drilling. The 2026 use case: a navbar badge that shows "Saving..." when any mutation in the app is pending. Reach when the shared-pending UX is real; otherwise skip.
- **The Zod parse on the client side.** Every `queryFn` and `mutationFn` parses its response with the same Zod schema the route handler emits — the contract from lesson 2 of chapter 050. The cached data is typed by construction, parse failures surface as `useQuery` errors instead of silent type drift. The senior reflex: schemas live in `/lib`, both the route handler's response writer and the `queryFn`'s response parser import the same schema.
- **What gets a query and what doesn't.** A query is a read of server state with an identity (the key). A computed derivation of multiple queries (`useMemo` over their `data`) is not its own query — don't create synthetic keys. A one-shot fetch in response to a button click is not a query, it's a mutation or a route-handler `fetch`. The discipline: queries have keys, mutations have actions, derived values use `useMemo`.
- **Worked example — the comment thread query.** A 25-line Client Component reading `useInfiniteQuery({ queryKey: commentKeys.list(invoiceId), queryFn: ({ pageParam }) => fetchComments(invoiceId, pageParam), getNextPageParam, initialPageParam: null, refetchInterval: 10_000, maxPages: 10 })`, rendering `data.pages.flatMap(p => p.comments)`, with a `<button onClick={fetchNextPage}>` and an "isFetchingNextPage" spinner. The example is the project's center of gravity in compact form.
- **Watch-outs:** `staleTime: 0` (the default) refetches on every mount and tab focus — the right default for live data, the wrong default for most SaaS reads; reach for `30_000` or higher unless you mean it; mutating `data` returned from `useQuery` directly mutates the cache and skips re-renders — treat it as frozen; query keys that include `Date`, function references, or non-serializable values break cache lookups — primitives and plain objects only; `useInfiniteQuery` without `maxPages` on a thread the user scrolls deep into accumulates pages forever — set a cap; the `onSuccess`/`onError` callbacks were removed from `useQuery` in v5 (only on mutations now) — handle render-time effects in the component, not the hook config; running `mutate` inside `onMutate` is a recursion bug — call the second mutation in `onSuccess` instead.

What this lesson does not cover:

- The `QueryClient` instantiation and provider wiring — lesson 3 of chapter 080.
- The SSR hydration shape — lesson 3 of chapter 080.
- The full project against a real screen — lesson 4 of chapter 080 and Chapter 081.
- Devtools, persistence (the persist-query-client plugin), online/offline handling, prefetching on hover — named once, not full-treatment.
- Suspense mode (`useSuspenseQuery`) — named once as an alternative to the `isPending` check, used selectively in our setup.
- `useQueries` (parallel queries) and `useQuery({ enabled })` (conditional queries) — named in passing, not load-bearing for the project.

Estimated student time: 55 to 70 minutes. The chapter's heaviest mechanics lesson — every primitive the project reaches for lives here.

## Lesson 3 — Wiring without leaking the cache across requests

The `'use client'` `<Providers>` shell, the per-request `getQueryClient()` helper via React's `cache()` (the multi-tenant isolation trap), SSR-hydrated initial data through `prefetchQuery` plus `<HydrationBoundary>`, the senior `staleTime`/`gcTime` defaults, the two-system invalidation reality after a Server Action, and the org-switch `queryClient.clear()` discipline.

Topics to cover:

- **The senior question.** TanStack Query needs a `QueryClientProvider` to expose the cache to every `useQuery`. The provider must be a Client Component. The Server Component layer wants to prefetch data so the initial paint isn't blank. How are these wired in Next.js 16's App Router without leaking the singleton across requests on the server, and without re-fetching what the server already prefetched? The lesson is the setup/wiring chapter — terminal commands, file shapes, and the closure rule that gates the entire library.
- **Install.** `pnpm add @tanstack/react-query @tanstack/react-query-devtools`. Versions match what lesson 2 of chapter 080 covered. Names the devtools as part of the dev experience; not an afterthought.
- **The `Providers` Client Component — where the QueryClient lives.** A `src/components/providers.tsx` file marked `'use client'` that creates a `QueryClient` and wraps `children` in `<QueryClientProvider client={queryClient}>`. The file exports `<Providers>` as the single integration point; `app/layout.tsx` wraps `children` in `<Providers>{children}</Providers>`. Standard provider pattern.
- **The per-request rule for the server-side QueryClient — the load-bearing trap.** Creating a `QueryClient` at module scope (`const queryClient = new QueryClient()` at the top of a Server Component) leaks state across requests — every request sees the previous request's cache, which on a multi-tenant SaaS is a data-isolation bug. The senior pattern: a `getQueryClient()` helper in `src/lib/query-client.ts` that returns a **new** `QueryClient` per call on the server (using React's `cache()` to memoize per-request — same primitive from lesson 5 of chapter 036). On the client, the helper returns a singleton via a module-scoped variable so the cache persists across renders. The same file, branching on `typeof window`. Show both branches, name the bug pattern explicitly.
- **The senior default config — `staleTime` and `gcTime`.** The `QueryClient` constructor takes `defaultOptions` — set `queries: { staleTime: 60_000, gcTime: 5 * 60_000 }`. `staleTime: 60s` means refetches don't fire on every mount/focus inside a minute; `gcTime: 5min` keeps unused cache entries around for navigations back. The defaults that shipped with the library are right for the "always live data" use case and wrong for SaaS — the lesson sets the SaaS-appropriate defaults at the provider and lets queries override per-call when needed.
- **`<HydrationBoundary>` — the SSR-hydrated initial-data shape.** The page is still a Server Component. Inside the Server Component, instantiate the per-request `QueryClient` via `getQueryClient()`, call `queryClient.prefetchQuery({ queryKey, queryFn })` (or `prefetchInfiniteQuery`) for the data the client will read, then `dehydrate(queryClient)` and pass the dehydrated state to a `<HydrationBoundary state={dehydratedState}>` wrapping the Client Component that runs `useQuery`. The Client Component reads from the populated cache on first render — no loading state, no waterfall. The senior win: SSR-fast initial paint, plus TanStack Query's cache populated for the rest of the session.
- **The data-fetching function shared between server and client.** The `queryFn` lives in `src/lib/comments/fetch.ts` (or wherever the feature owns its data layer). The Server Component imports it and calls it directly inside `prefetchQuery`; the Client Component's `useQuery` references the same function. One function, two call sites — the contract is structural, not by convention.
- **The route handler the `queryFn` calls — what owns the request seam.** On the client, the `queryFn` calls `fetch('/api/invoices/${id}/comments?cursor=...')`. On the server inside `prefetchQuery`, calling `fetch()` to your own host is wasteful (loopback over HTTP for data you can read directly). The senior pattern: the `queryFn` accepts a "fetcher" parameter, the client passes a `fetch`-based fetcher, the server passes a direct DB read. Alternative: the `queryFn` calls a thin function from `/lib` that branches on `typeof window` — the lesson picks the explicit-fetcher version because it's testable. The route handler is still the public contract for any external client; the in-app server-side prefetch just skips the network hop.
- **Devtools setup — `<ReactQueryDevtools />`.** A small floating panel showing every query's state in development. Mount inside the `<Providers>` component, gated on `process.env.NODE_ENV !== 'production'` so it tree-shakes out of the production bundle. The lesson positions devtools as part of the daily loop — open the panel, see queries by key, watch `isStale`, click "Invalidate" to test the refresh path.
- **`'use client'` boundary placement.** Every component that calls `useQuery`/`useMutation` is `'use client'`. The Server Component page passes the dehydrated state through the boundary, then the `<HydrationBoundary>` (Client) wraps the `useQuery`-using leaf. Drawing the boundary at the leaf, not at the page, keeps the static parts of the page Server Components — the architectural principle from lesson 2 of chapter 034 applied here.
- **Cleanup — `queryClient.clear()` after the server response.** The dehydrated state is sent in the RSC payload; the server-side `QueryClient` then sits in memory until garbage-collected. For high-throughput servers, calling `queryClient.clear()` after `dehydrate` is the discipline named in the official advanced-SSR guide. The course names it; in practice React's `cache()` per-request scoping plus normal GC handles it.
- **The Server-Action interaction — invalidate from the client after the action returns.** A Server Action that mutates data (e.g., `addCommentAction`) returns the canonical `Result`. The Client Component awaits the action's promise, then calls `queryClient.invalidateQueries({ queryKey: commentKeys.list(invoiceId) })` to mark the cached list stale. The Server Action's own `revalidatePath`/`revalidateTag` invalidates the **Server Component** cache; TanStack Query's `invalidateQueries` invalidates the **client** cache; both fire because both layers hold data. The lesson surfaces this two-system reality explicitly so the student doesn't expect `updateTag` to invalidate TanStack queries (it doesn't).
- **Tenant scope at the cache layer — the org-switch reset.** When the user switches active org (Chapter 060), the entire client cache is for the wrong tenant. The pattern: on org switch, call `queryClient.clear()` (or `removeQueries` for the org-scoped subtrees) before the navigation. The TanStack Query cache holds no org filter by itself — wrong-org data leaking across switches is a data-isolation bug at the cache layer the same way it would be at the data layer. Name the bug, name the fix.
- **Worked example — the full wiring for the comment thread.** `providers.tsx` (Client) creates the `QueryClient` with the senior defaults; `app/layout.tsx` wraps children; `lib/query-client.ts` exports `getQueryClient()` with the server/client branch; `app/(app)/invoices/[id]/page.tsx` (Server Component) prefetches the infinite query and renders `<HydrationBoundary state={dehydrate(queryClient)}><CommentThread invoiceId={id} /></HydrationBoundary>`; `comment-thread.tsx` (Client, `'use client'`) runs `useInfiniteQuery` and reads from the populated cache. One worked path from layout to leaf.
- **Watch-outs:** module-scoped `new QueryClient()` on the server is the cross-request leak — use the `cache()`-wrapped `getQueryClient()`; missing `'use client'` on the provider component produces a "createContext is not a function" error at build — the canonical mistake; calling `useQueryClient()` inside a Server Component fails silently in some configurations and loudly in others — `useQueryClient` is client-only by definition; prefetching with `prefetchQuery` but rendering a `useQuery` with a different key produces a cold fetch — the keys must match exactly, hence the `queryKeys.ts` discipline from lesson 2 of chapter 080; `dehydrate` and `<HydrationBoundary>` skipped means the client refetches on mount even though the server already had the data — a network hop wasted; running `<ReactQueryDevtools />` in production ships the devtools bundle — gate on `NODE_ENV`; `staleTime: 0` at the provider level overrides per-query defaults and produces a refetch storm on tab focus — set the SaaS default at the provider, override per-query when live freshness matters.

What this lesson does not cover:

- The primitives themselves — lesson 2 of chapter 080.
- The worked screen end-to-end with optimistic mutation and polling — lesson 4 of chapter 080 and Chapter 081.
- Suspense mode integration with the App Router's streaming — named in passing; the SaaS surface here uses the imperative `isPending` shape.
- Server-side prefetch from a Server Action context — Server Actions handle their own invalidation; the prefetch pattern is for initial page loads.
- Edge runtime considerations — not relevant; the QueryClient runs in Node.
- React Server Components calling `useQuery` directly — that's not a thing; `useQuery` is Client-only by definition.

Estimated student time: 45 to 55 minutes. The wiring lesson — the one place the SSR-hydrated shape lands; everything else assumes this is correct.

## Lesson 4 — The per-invoice comment thread clears the bar

Running the four-trigger funnel against one concrete screen, splitting the read side (`useInfiniteQuery` with 10s polling, `maxPages: 10`, SSR-prefetched first page) from the write side (Server Action plus client-side `invalidateQueries`), and the cache-update optimistic-add shape — framing the chapter 081 build.

Topics to cover:

- **The senior question.** With the threshold, the primitives, and the wiring in place, what screen in our SaaS actually clears the bar? The lesson names the surface, walks the four-trigger check against it, and frames the project (chapter 081) that builds it. Concrete, specific, end of chapter.
- **The screen — per-invoice comment thread.** On every invoice detail page (`/invoices/[id]`), a comment thread the user can read, scroll back through, and add to. The product motivation: invoice disputes, internal notes, customer correspondence — the SaaS standard. The screen is the canonical "client-side server state" surface in the course.
- **The four-trigger check, applied.** Run the lesson 1 of chapter 080 funnel against the comment thread:
  - **Polling.** A coworker may post a comment from another session; the user expects to see new comments without manually refreshing. The cadence is "every 10 seconds while the tab is focused" — sub-user-action frequency. `router.refresh()` on a `setInterval` would re-render the entire route segment every poll — too expensive, too jarring. Trigger met.
  - **Optimistic with rollback.** Posting a comment should show immediately; on a 500 from the route handler the optimistic row rolls back with an error banner. The optimism flows into a cached infinite-query (the first page must include the new row immediately); `useOptimistic` covers single-list inline rollback but doesn't intersect with the infinite-query cache shape. Trigger met.
  - **Infinite scroll with cache reuse.** The thread can be hundreds of comments deep; the user scrolls back, then forward again. Server-Component cursor pagination from lesson 4 of chapter 064 would refetch every page on each navigation. `useInfiniteQuery` with `maxPages` keeps the loaded pages in the client cache for the session. Trigger met.
  - **Cross-view caching.** The same comment thread might be peeked at from a "recent activity" sidebar that mounts/unmounts as the user navigates. The client cache means the thread renders instantly on re-mount. Trigger met — though weaker than the first three.
  - Three out of four cleared, the strongest case for the library being right.
- **What gets a query, what stays a Server Action.** The read side (the thread): `useInfiniteQuery` against `/api/invoices/[id]/comments` with cursor pagination, 10-second `refetchInterval`, `maxPages: 10`. The write side (posting a new comment): a Server Action (`addCommentAction`) with `useActionState` for the form, then a client-side `queryClient.invalidateQueries({ queryKey: commentKeys.list(invoiceId) })` after the action resolves. Why split: Server Actions own form submission semantics (the `FormData` shape, validation, the audit-log write, the canonical `Result`); TanStack Query owns the read cache that the action invalidates. The senior call: the two systems don't fight, they cooperate at the seam.
- **The route handler as the read seam.** `GET /api/invoices/[id]/comments?cursor=...` (Chapter 050) returns `{ comments, nextCursor }`. The route handler runs `authedRoute(role, schema, fn)` (lesson 3 of chapter 061) for the auth/tenancy check, returns the validated response shape. The `queryFn` calls this endpoint; the response Zod schema is shared with the route handler. The endpoint is also the public contract — a future mobile client or third-party integration uses the same surface.
- **The optimistic-update shape — cache-write, not via-variables.** The new comment needs to appear at the top of the first page of the `useInfiniteQuery`. That requires writing into `data.pages[0].comments`, which is a cache write — `setQueryData(commentKeys.list(invoiceId), (old) => ...)` inside `onMutate`, snapshot in context, restore in `onError`, invalidate in `onSettled`. The lesson walks the exact updater function: it clones `old.pages`, prepends the optimistic comment to page 0, returns the new shape. The cancel-before-set pattern: `await queryClient.cancelQueries({ queryKey })` first so in-flight refetches don't overwrite the optimism.
- **The polling cadence — `refetchInterval` plus `refetchIntervalInBackground: false`.** Poll every 10 seconds while the tab is focused; stop polling when the tab is hidden. The senior call: 10 seconds is the right cadence for a comment thread (faster feels jittery, slower feels stale); the in-background pause is the battery-and-connection-pool discipline. The threshold from lesson 1 of chapter 080 was "the cadence drops below user-initiated" — 10s clears that bar.
- **The SSR-hydrated initial page.** The Server Component page (`app/(app)/invoices/[id]/page.tsx`) prefetches the first page of the infinite query via `queryClient.prefetchInfiniteQuery`. The dehydrated state goes into `<HydrationBoundary>`; the `<CommentThread />` Client Component renders the prefetched data on first paint. No loading skeleton on initial load; subsequent scroll-down calls hit the route handler directly. The win: server-rendered first paint, client-cached scroll.
- **The two invalidation systems, side by side.** When `addCommentAction` succeeds:
  - The Server Action calls `revalidateTag(invoiceTag(invoiceId), 'max')` so the **Server Component** invoice page (the parent layout's summary cards, any other server-rendered comment counts) refreshes.
  - The client awaits the action's `Result`, then calls `queryClient.invalidateQueries({ queryKey: commentKeys.list(invoiceId) })` so the **TanStack Query** cache refetches.
  - Both fire because both layers hold data. The lesson surfaces this two-system reality as the architectural cost of bringing TanStack Query in — it's not a bug, it's the price tag.
- **What stays in Server-Component shape — the rest of the page.** The invoice header, the customer card, the line items table, the version banner — all Server Components, no `useQuery`. The comment thread is the one part of the screen that needs TanStack Query; the rest stays with the framework defaults. The senior reflex: scope the library to the leaf that needs it.
- **The mobile / third-party surface — already covered.** Because the read side flows through a route handler, the comment thread is reachable from anything that can speak HTTP — a future mobile app, a Zapier integration, a script. Bringing TanStack Query in didn't sacrifice the API; it added a client cache on top of it. The senior win: the architecture stayed open.
- **The forward pointer to chapter 081.** The project chapter builds this surface end to end: the seeded comments, the `QueryClientProvider` wiring, the `<HydrationBoundary>` on the page, the `useInfiniteQuery` with polling, the `useMutation` with cache-update optimistic add, the verify recipe (cross-session arrival, optimistic visibility, forced 500 rollback). This lesson is the framing; chapter 081 is the build.
- **Anti-pattern callouts — what NOT to do on this screen.** Don't push the entire invoice detail page into Client Components — only the comment thread. Don't replace the action with a route-handler POST + `useMutation` — Server Actions own the form contract, the audit-log write, the canonical `Result`; the action stays, TanStack Query handles the cache after. Don't poll the entire invoice detail — only the comment thread. Don't skip the SSR prefetch — the initial blank state is jarring and unnecessary. Don't drop the route handler and call a `/lib` function directly from the client — the client can't import `/lib` (it would bundle Drizzle), and even if it could, the seam matters.
- **Watch-outs:** posting a comment with a synchronous `mutate` followed by `queryClient.invalidateQueries` produces a double-render (optimistic, then refetch, then optimistic clears) — the cancel-before-set pattern smooths it; polling that fires while the optimistic update is pending overwrites the optimism — the cancel pattern again; the new comment from a coworker arriving via poll between the optimistic add and the action resolve duplicates the row temporarily — accept the brief duplicate or dedupe by client-generated UUID; the action's `revalidateTag` plus the client's `invalidateQueries` are both needed — skipping either leaves one layer stale; bumping `refetchInterval` to 1s "for snappiness" floods the server and burns mobile battery — 10s is the right call.
