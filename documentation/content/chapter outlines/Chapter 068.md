# Chapter 068 — Object storage

## Chapter framing

Chapter 068 is the conditional half of Unit 12: most SaaS products in 2026 ship without ever writing a byte to object storage — the database covers everything until users start handing the app files (avatars, contracts, generated assets) or the app starts generating files large enough to be wrong to inline (the chapter 067 CSV export). The chapter lands the threshold up front so the student can defend *not* adding R2 to a project, then installs the three primitives a senior reaches for when the threshold is crossed: a Cloudflare R2 bucket plus scoped credentials, presigned URLs for direct browser-to-bucket transfer (the bytes never touch the Next.js function), and a `file_metadata` row in Postgres that is the canonical reference — the object in R2 is a blob keyed by the row's path, not the other way around. The chapter closes the Blob/File/`URL.createObjectURL` thread opened in chapter 016 and the upload-issue-endpoint seam from Architectural Principle #3.

Threads that run through every lesson: storage and metadata are split — Postgres owns identity, ownership, and lifecycle; R2 owns the bytes; the join is the object key; the row is the source of truth for "does this file exist for this user." The Next.js function is never a byte pipe — the function signs a URL, the browser PUTs to R2 directly, the function records the metadata. Presigned URLs are scoped: short TTL, single content-type and content-length constraint, scoped R2 token with the minimum operations the app actually does. Tenancy is on the object key (`org/${organizationId}/...`) and re-checked at every metadata read, never trusted from the client. CORS is configured on the bucket, not worked around in the app. The chapter is conditional — no project until chapter 069 — and the prose pins the threshold explicitly so the student is equipped to leave R2 out of an app that does not need it.

## Lesson 1 — Defending the no — when object storage earns its weight

Names the three trigger conditions (user uploads, generated assets the app serves back, third-party media) that put R2 on the table, the conditions that do not, and why R2 beats S3 and the upload SaaS wrappers on unit economics for any read-heavy product.

Topics to cover:

- **The senior question.** A SaaS launches. Does it need object storage? The default answer for most B2B SaaS is no — the product surface is forms, lists, and dashboards, the database holds every byte that matters, and adding a second storage system is operational complexity for nothing. The lesson lands the threshold: object storage earns its weight only when the app handles user-supplied or app-generated binary payloads that would be wrong to put in Postgres.
- **The trigger conditions.** Three named, observable conditions. **(1) User uploads** — avatars, document attachments, contract PDFs, profile or org logos; the moment the app accepts a file from a user, R2 is on the table. **(2) Generated assets the app must serve back** — the chapter 067 CSV export, generated PDF invoices, server-rendered images for OG cards, exported reports; if the file is too large to inline in an email and lives longer than the request, R2 wins. **(3) Third-party media** — partner imports, downloaded artifacts, cached external assets; same shape as (1) from the function's view.
- **Conditions that do not justify object storage.** A 2 KB JSON document — fits in a `jsonb` column. A handful of static marketing images — ship them with the Next.js build, the CDN handles it. A few hundred KB of binary config — Postgres `bytea` is fine at small scale. "I want a separate bucket for cleanliness" — that is not a threshold.
- **Why not Postgres for binary blobs.** Database backups balloon (every snapshot pulls the bytes), connection-pool memory pressure on reads, no built-in HTTP delivery (every read is a function invocation streaming bytes), `pg_dump` becomes unusable at any real volume. Names the failure modes once.
- **Why R2 over S3 for SaaS unit economics.** Cloudflare R2 has zero egress fees; S3 charges $0.09/GB out after 100 GB free. For any SaaS that serves files back to users — the entire trigger condition (2) — egress dominates the bill. Storage at $0.015/GB/month and Class A/B operation pricing covers everything else. The 2026 numbers for a typical SaaS workload (10 TB storage, 50 TB/month egress): R2 ~$200/month total, S3 several thousand in egress alone. The decision is operational unit economics, not a feature gap.
- **Why R2 over UploadThing / Vercel Blob / Supabase Storage.** UploadThing wraps S3 with a managed upload widget and per-GB markup — wins for prototypes, loses on cost and lock-in at SaaS scale. Vercel Blob is fine for Vercel-tight apps but priced at retail S3 with no egress break. Supabase Storage is good if Supabase is already the database — outside that, it pulls a second platform in. R2's S3-compatible API means the AWS SDK works unchanged, so the off-ramp to S3 or any S3-compatible provider is structural. Names the alternatives once, picks R2.
- **The architectural shape.** Three nouns. **Bucket** in R2 — a namespace, scoped credentials, CORS configured for the app's origin. **Object key** — the path inside the bucket; the SaaS pattern keys objects by tenancy (`org/${organizationId}/files/${fileId}`). **Metadata row** in Postgres — the canonical identity (`id`, `organizationId`, `objectKey`, `contentType`, `byteSize`, `uploadedBy`, `uploadedAt`). The row is the source of truth; the object is the bytes the row references.
- **The bytes-never-touch-the-function rule.** Architectural Principle #3 — the upload-issue endpoint is the seam. The function signs a URL; the browser PUTs to R2 directly; the function records the metadata after. Function CPU and bandwidth bill is unchanged whether the upload is 5 KB or 5 GB. The naive shape (multipart POST to the function, function pipes to R2) is restated and rejected: doubles the bandwidth bill, doubles the time, hits the function timeout for large files. Forecasts lesson 3 of chapter 068.
- **The chapter's scope cuts.** No image resizing or transformation (Cloudflare Images is a separate product; reach for it when the product calls for it). No streaming multipart for >100 MB uploads (presigned PUT caps at 5 GB; the streaming-multipart pattern is named once for recognition). No virus scanning (named as the production hardening that lives between upload and "available"). No CDN-cache invalidation tactics (R2 ships with a public bucket option; the course's default is private + presigned GET).
- **The course's wiring — preview.** chapter 069 builds the user-upload path end-to-end (presigned PUT, `file_metadata`, `Files` list, presigned GET); the chapter also wires back the chapter 067 CSV export so the email carries a real R2 download link instead of `console.log`. Both call sites use the same `presignedPut` / `presignedGet` helpers — one shape, two consumers.
- **Watch-outs:** reaching for R2 before any trigger condition fires — operational complexity for nothing; storing avatars or attachments as base64 in a `text` column — works at first, kills the database backup; routing user uploads through the function — eats function time and bandwidth, breaks at any real file size; treating "I have a `file_metadata` table" as enough — orphaned objects in R2 and orphaned rows in Postgres are both real failure modes the next four lessons name.

What this lesson does not cover:

- The R2 bucket setup and CORS — lesson 2 of chapter 068.
- Presigned URL mechanics — lesson 3 of chapter 068.
- The `file_metadata` row design — lesson 4 of chapter 068.
- The full upload flow code — Chapter 069.
- Image transformation and CDN behaviors — out of scope, named once.

Estimated student time: 25 to 35 minutes. The "defend the no" lesson — most students will not reach for R2 in their first SaaS and need to know why that is correct.

## Lesson 2 — Standing up R2 — buckets, scoped tokens, and CORS

Sets up the minimum R2 surface a SaaS trusts in production: one bucket per environment, bucket-scoped tokens with minimum operations, the S3-compatible client in `lib/r2.ts`, the CORS rule with explicit origins and headers, and the `org/${orgId}/files/${id}` object-key tenancy convention.

Topics to cover:

- **The senior question.** The trigger conditions from lesson 1 of chapter 068 fire. What is the minimum R2 setup that gets bytes flowing, and which configuration knobs matter for production trust boundaries vs. which are dashboard noise?
- **The bucket.** A namespace inside an R2 account, named once at creation (`acme-saas-prod`, `acme-saas-staging`). Buckets do not auto-name by environment — the project explicitly creates one bucket per environment so a staging upload can never land in production. Location hint: pick the region closest to the user base (R2's location is a hint, not a hard pin — R2 replicates globally).
- **S3-compatible endpoint.** R2 speaks the S3 API. The endpoint URL is `https://${ACCOUNT_ID}.r2.cloudflarestorage.com`, the bucket appears as a path. The AWS SDK v3 (`@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner`) talks to R2 unchanged once `endpoint` and `region: 'auto'` are set on the client. The off-ramp to S3 is structural — change the endpoint and credentials, keep the code.
- **Scoped API tokens.** The R2 dashboard issues account-level tokens (full access) and bucket-scoped tokens (per-bucket, per-operation). The senior default: one scoped token per environment, scoped to the single bucket, with the minimum operations the app uses (Object Read, Object Write — Admin and Bucket-Create are not granted). The senior anchor: the production credentials never see `localhost`; staging and production are separate tokens.
- **Env surface.** `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET_NAME`. Validated in `lib/env.ts` via the Zod env pattern from Chapter 034 — non-optional in production, present in `.env.local` for development. Forecasts the chapter 069 starter's `.env.example`.
- **The S3 client construction.** A single `lib/r2.ts` module exports a configured `S3Client` instance — `new S3Client({ region: 'auto', endpoint: ..., credentials: { accessKeyId, secretAccessKey } })`. Constructed once, imported everywhere — the same one-client-per-process discipline as `lib/db.ts` and `lib/resend.ts`. Architectural Principle #5 — the SDK is not wrapped beyond the call site; helpers (`presignedPut`, `presignedGet`) compose the SDK, they do not abstract it.
- **CORS — the browser upload trust boundary.** R2 enforces CORS for browser-originating requests. Without a CORS rule on the bucket, the browser blocks the presigned PUT even though the signature is valid. The CORS configuration declares: `AllowedOrigins: [APP_URL]` (the SaaS origin — `http://localhost:3000` in development, the production URL in production), `AllowedMethods: ['GET', 'PUT']`, `AllowedHeaders: ['content-type']`, `MaxAgeSeconds: 3600`. Configured once per bucket via the dashboard or a `PutBucketCors` API call; the project repo ships the JSON.
- **CORS — the security watch-outs.** `AllowedOrigins: ['*']` is the most common production leak — it lets any origin upload through any presigned URL leaked from the app's logs or browser dev tools. `AllowedHeaders: ['*']` works on AWS but fails on R2 — the lesson restates the R2-specific quirk. The lesson shows the wrong-then-right shape once.
- **Bucket privacy.** R2 buckets are private by default — no public access, every read requires either a presigned URL or a Cloudflare Workers route. The course keeps everything private + presigned. The "public bucket" mode is named once; useful for fully public assets (marketing media), unsuitable for tenant-scoped files because anyone with the URL can read.
- **The object-key tenancy convention.** The path inside the bucket carries tenancy. The course's pattern: `org/${organizationId}/files/${fileId}`. Two structural wins: (1) a fabricated client key cannot land outside the org's prefix without the function constructing it; (2) bucket-level diagnostics (R2 dashboard, lifecycle rules, retention policies) can scope by prefix. Forecasts lesson 4 of chapter 068 where the row's `objectKey` is constructed server-side, never accepted from the client.
- **Lifecycle rules — named once.** R2 supports prefix-scoped lifecycle rules (delete objects older than N days under prefix `tmp/`). The default is "no rules"; the project does not need them. Named so the student knows the surface exists.
- **Local development.** The dev environment talks to the real R2 bucket (the staging or a dedicated `dev` bucket). The senior call: one `dev` bucket per developer is overkill; one shared `dev` bucket with a `local/${developer}/` prefix is the small-team default. Lifecycle rule (24 h deletion) on the `local/` prefix is the cleanup. The local-stub alternative (MinIO in Docker) is named once — only worth it if R2 access is gated.
- **Observability.** R2 dashboard surfaces per-bucket request counts (Class A: writes / list / multipart-init, Class B: reads / head) and storage bytes. The senior reach: track weekly the same way Vercel function invocations are tracked — anomalies usually mean a leaked presigned URL or a polling client.
- **Watch-outs:** using the account-level token in app code — full bucket-create / delete blast radius; sharing one token across staging and production — same leak surface; CORS with `*` in `AllowedOrigins` — public PUT to your bucket; CORS with `*` in `AllowedHeaders` — silently broken on R2; constructing the S3 client per-request — connection pool churn; constructing the object key on the client and trusting the server with it — tenancy bypass; forgetting `region: 'auto'` — SDK throws on signing.

What this lesson does not cover:

- The actual presigned PUT / GET call mechanics — lesson 3 of chapter 068.
- The `file_metadata` row — lesson 4 of chapter 068.
- The browser-side upload code — Chapter 069.
- Cloudflare Workers in front of R2 — out of scope.
- Self-hosting MinIO past the named mention — out of scope.

Estimated student time: 35 to 45 minutes. The "I can stand up the bucket and trust the boundary" lesson — every later lesson assumes this surface is configured.

## Lesson 3 — Presigned URLs — signing the upload seam

Teaches the presigned PUT and GET mechanics that move bytes from browser to R2 without touching the function: signed `ContentType` and `ContentLength`, 5-10 minute expiries, the layered size defense, and the sign-then-finalize two-step write with post-upload HEAD verification.

Topics to cover:

- **The senior question.** The bucket exists, the credentials are scoped, CORS is configured. How does the byte actually get from the user's browser to R2 without ever touching the function, and what does the function sign so the upload can't be abused?
- **What a presigned URL is.** A regular S3-style URL with query parameters carrying a signature (HMAC over the bucket, key, method, expiry, and any pinned headers using the R2 credentials). Anyone with the URL can perform the signed operation until the expiry; no further credential is needed. The function generates the URL; the browser uses it. Architectural Principle #3 — the upload-issue endpoint is the seam.
- **The three flavors.** **Presigned PUT** — the browser uploads a single object to the signed key with `fetch(url, { method: 'PUT', body: file })`. The course's default for uploads <100 MB. **Presigned GET** — anyone with the URL reads the object until expiry; the course's default for serving private files back. **Presigned POST (policy-based)** — an alternative for uploads that needs server-side size and content-type constraints; named once for recognition, not built (PUT covers the project's needs, POST policy adds glue).
- **The PUT shape.** `getSignedUrl(s3Client, new PutObjectCommand({ Bucket, Key, ContentType, ContentLength }), { expiresIn: 300 })`. Five-minute expiry, content-type and content-length pinned in the signed parameters, returns a string URL. The browser does `await fetch(url, { method: 'PUT', body: file, headers: { 'Content-Type': file.type } })`. The browser's `Content-Type` header must match the signed one exactly — mismatch returns 403 SignatureDoesNotMatch.
- **The GET shape.** `getSignedUrl(s3Client, new GetObjectCommand({ Bucket, Key }), { expiresIn: 600 })`. Ten-minute expiry, signed URL goes into `<img src>`, `<a href>`, or an email body. Senior anchor: presigned GET URLs are fresh-issued per render — never persisted to the database, never cached past the expiry. A 24-hour-expiry GET in an email is a known anti-pattern; the user opens the email 25 hours later and the link is dead.
- **Expiry — the trust trade-off.** Short expiry (300 s for PUT, 600 s for GET) is the senior default. PUTs need long enough for the user's network to complete the upload; GETs need long enough for the page to render and the user to act on it. Long-expiry URLs (24 h, 7 d) are a leak surface — every URL in browser history, every analytics tracker, every shared screenshot extends the trust window. The course pins 5-10 minutes everywhere.
- **The size pin — `ContentLength`.** A PUT presigned URL has known limits in 2026: the AWS SDK signs `ContentLength` for S3, which enforces it server-side; R2 does not enforce the signed length the same way S3 does, so a malicious client can attempt to PUT a larger body than declared. The defense is layered: (1) client-side validates `file.size` before requesting the URL; (2) server-side checks `file.size` claim against a per-tenant cap before signing; (3) post-upload, the metadata-save action HEADs the object via `HeadObjectCommand` and verifies the actual size matches the claim before persisting the row. The lesson names the layered defense — neither the client nor the presigned URL alone is the boundary.
- **The content-type pin.** The signed URL carries the `ContentType` from the request; the browser must send the matching header. This is the structural constraint that prevents "upload an executable to a `.png` key" — the function rejects unsupported content types before signing, and the signed URL only accepts that one type. The lesson shows the rejection shape (`if (!ALLOWED_TYPES.includes(file.type)) return error('UNSUPPORTED_TYPE')`).
- **The two-step write — sign, then save.** The canonical flow: (1) client requests presigned PUT, server validates inputs (size, type, role), constructs the object key (`org/${orgId}/files/${nanoid()}.${ext}`), signs, returns `{ url, objectKey, uploadId }`; (2) client PUTs the file directly to R2; (3) on PUT success, client calls a second Server Action (`finalizeUpload`) with `{ uploadId, objectKey, actualSize }`; (4) server HEADs the object, compares to the claim, inserts the `file_metadata` row, returns success. Forecasts lesson 4 of chapter 068. The senior anchor: the metadata is written *after* the upload completes, not before — orphaned rows in Postgres are worse than orphaned bytes in R2 (lifecycle rules clean orphaned bytes; orphaned rows mislead the UI).
- **What happens on partial / failed uploads.** The browser PUT can fail (network drop, user navigates away, file too large for size cap). The client never calls `finalizeUpload` — no row is written, the partial bytes in R2 are orphaned. The cleanup: a daily Trigger.dev or Vercel Cron job listing the bucket for objects with no matching `file_metadata` row and deleting them after a 24 h grace window. Named once; the project does not build it.
- **The Server Action signature.** `presignedPut(input: { fileName: string, contentType: string, claimedSize: number }): Promise<Result<{ url: string, objectKey: string }>>`. Wrapped in `authedAction('member', schema, ...)` — the role boundary is at the action, not at R2. Forecasts the chapter 069 reference signature.
- **CORS preflight — the watch-out.** The browser fires a `OPTIONS` preflight before the PUT for any cross-origin request with a non-simple Content-Type. R2 must respond with the matching CORS headers (set in lesson 2 of chapter 068). The lesson names the failure shape: the network tab shows a 200 OPTIONS followed by a 403 PUT — that is CORS allowing the preflight but the signed Content-Type not matching the sent header.
- **No-server-pipe verification.** The senior anchor: open the Network tab during the upload; the PUT request goes directly to `${bucket}.r2.cloudflarestorage.com`, the function call only carries the small JSON `{ fileName, contentType, claimedSize }` and returns the small JSON `{ url, objectKey }`. The function's response size is bytes, not megabytes. Forecasts the verify step in lesson 7 of chapter 069.
- **Watch-outs:** signing without `ContentType` — the browser can upload anything; signing without `ContentLength` — no size pin; setting a 24 h `expiresIn` for a button click — the URL leaks into analytics with a long trust window; constructing `objectKey` from client input verbatim — tenancy bypass; trusting `claimedSize` without HEAD-after-upload — size-bomb attack; persisting the presigned GET URL in `file_metadata.url` — the URL goes stale, the row lies; mismatched `Content-Type` between sign and PUT — silent 403.

What this lesson does not cover:

- The `file_metadata` row design — lesson 4 of chapter 068.
- The full upload UI and client-side code — Chapter 069.
- Multipart presigned uploads for >100 MB files — named, not built.
- POST policy uploads with size constraints — named, not built.
- Virus scanning and content moderation — out of scope.

Estimated student time: 45 to 55 minutes. The "I can sign safely" lesson — the trust boundary every later upload depends on.

## Lesson 4 — Postgres owns identity, R2 owns bytes

Designs the `file_metadata` row as the canonical record (id-as-key-segment, tenant-scoped reads, soft-delete with cooled-off object cleanup, HEAD-verified `byteSize`, no persisted URL) and names the orphan failure modes in both directions.

Topics to cover:

- **The senior question.** The bytes are in R2, keyed by `org/${orgId}/files/${fileId}`. What does Postgres need to remember about them, what's the canonical row shape, and how does it stay in sync with the bucket without periodic full-bucket scans?
- **The split.** Postgres owns identity, ownership, and lifecycle metadata; R2 owns the bytes. The `file_metadata` row is the canonical record — "does this file exist for this org and this user, at this object key, with this declared content type and this byte count" — and the bucket is dumb storage keyed by the row's path. Reads always join through `file_metadata`; nothing in the app ever lists the bucket directly.
- **The row shape.** Drizzle schema: `id` (uuid, primary key, used as the `${fileId}` segment in `objectKey`), `organizationId` (uuid, foreign key to `organizations`, indexed), `uploadedBy` (uuid, foreign key to `users`), `objectKey` (text, not null, unique), `originalFileName` (text — the user's filename, displayed in the UI), `contentType` (text — from the upload), `byteSize` (bigint — the HEAD-verified actual size), `uploadedAt` (timestamptz), `softDeletedAt` (timestamptz nullable — the chapter 061 soft-delete shape carried over). Names the unique constraint on `objectKey` once — no two rows reference the same blob.
- **Why `id` is also the key segment.** The row's UUID *is* the unique identifier inside the `org/${orgId}/files/${id}.${ext}` path. Wins: the row-to-object lookup is a function (`id → objectKey`); listing files by org is a Postgres query, never a bucket list; deletion of the row marks the soft-delete and a background sweep deletes the object. The alternative (random `objectKey` independent of the row ID) needs a secondary lookup table and breaks recovery.
- **Why `originalFileName` is separate from `objectKey`.** The user uploaded `Q4 Financials FINAL.pdf` — the row preserves that for display; the object key uses the row UUID + sanitized extension so it is URL-safe and collision-free. The download link sends `Content-Disposition: attachment; filename="${originalFileName}"` from the function to set the browser-side filename. Names the rule.
- **Tenancy enforcement at the metadata layer.** Every read goes through `tenantDb(orgId).fileMetadata.findById(id)` — the `where` clause pins `organizationId`, so a fabricated `fileId` from one org cannot resolve a row in another org. The same shape from chapter 056; the `file_metadata` table joins to `organizations` and the helper enforces the scope. Forecasts the chapter 069 `getFile(id)` shape.
- **The presigned-GET helper sits on the metadata layer.** `getFileDownloadUrl(id): Promise<Result<{ url: string, fileName: string }>>` — looks up the row in tenant scope, presigns a 10-minute GET against `objectKey`, returns `{ url, fileName }` for the client to put in `<a href>`. The senior shape: the URL is generated fresh per call, never stored on the row, never cached past expiry.
- **Soft delete, not hard delete.** Carrying the chapter 061 discipline: deletion sets `softDeletedAt`, the row stays. Two reasons: (1) audit and recovery — "did this user have access to this file two weeks ago" is answerable; (2) decoupled object cleanup — the row marks deletion-intent, a Trigger.dev or cron sweep deletes the R2 object N days later under a `softDeletedAt < now() - interval '30 days'` predicate. Hard-deleting the row without cleaning the object leaves orphaned bytes; deleting the object before the soft-delete cooling window risks losing genuinely-needed files. Names the rule.
- **Indexes.** Primary key on `id`. Index on `(organizationId, softDeletedAt, uploadedAt DESC)` for the canonical "list this org's files, newest first, non-deleted" query. Restates the chapter 039 index discipline — index the access pattern, not the columns.
- **The orphan problem — both directions.** **Orphan bytes:** PUT succeeded, `finalizeUpload` action failed (network, function timeout). Cleanup: daily bucket-list sweep for `org/*/files/*` objects with no matching `file_metadata.objectKey`, delete after 24 h grace. **Orphan rows:** row inserted, the post-upload HEAD failed before commit (shouldn't happen with the right shape from lesson 3 of chapter 068 but defense-in-depth). Cleanup: hourly query for rows where `uploadedAt < now() - 1 hour AND` R2 HEAD returns 404, hard-delete. Both sweeps named, neither built; project hands off to 14.x or stays as a forward note.
- **Auditing.** Every upload writes one `audit_logs` row (`event: 'file.uploaded', payload: { fileId, byteSize, contentType }`); every download URL issued writes one (`event: 'file.download_url_issued'`); every soft-delete writes one (`event: 'file.soft_deleted'`). Names the discipline; the project audits at the action boundary.
- **Why no `url` column.** The presigned URL goes stale; storing it lies. The row carries the parts that are permanent (org, key, type, size, names) and the helper generates a fresh URL per request. Restates Architectural Principle #5 — the R2 SDK is not abstracted, but the call site has a thin helper that generates URLs on demand.
- **Watch-outs:** persisting the presigned URL in the row — stale URL lies, regenerate per render; constructing the `objectKey` from `originalFileName` directly — special characters, length, collision; allowing the client to choose `id` — clobbering existing rows; deleting the R2 object the same instant the row is soft-deleted — no recovery window; hard-deleting the row without cleaning the object — orphan bytes pile up; reading `file_metadata` outside `tenantDb` — cross-tenant leak; storing `byteSize` from the client's claim, not the post-upload HEAD — size-claim trust failure.

What this lesson does not cover:

- The full upload flow code — Chapter 069.
- Background orphan sweeps — named, not built (forward note).
- Image transformation by content type — out of scope.
- Per-file ACLs (private to one user inside an org) — out of scope (organization-level scope is the course's default tenancy).
- Versioning — out of scope.

Estimated student time: 35 to 45 minutes. The "Postgres owns identity, R2 owns bytes" lesson — the split every later upload-bearing feature assumes.

## Lesson 5 — Wiring R2 into our app — two workloads, one mechanism

Picks the two call sites R2 covers (the chapter 069 user-upload path with `file_metadata` and the chapter 067 CSV export retrofit with a server-side PUT and no metadata row), names the workloads kept off R2, and pre-loads the lib surface and env shape for the chapter 069 project.

Topics to cover:

- **The senior question.** Student has the threshold, the bucket, the presigned mechanics, the metadata row. Where in the course's app does R2 get wired in, and where deliberately not?
- **The two workloads the course wires through R2.** **(1)** User uploads — the chapter 069 project ships the `<input type="file">` direct-to-R2 PUT, the `file_metadata` insert, the `Files` list with presigned GETs, the per-row download link. **(2)** Generated-asset retrofit — the chapter 067 CSV export's `console.log`-placeholder `downloadUrl` becomes a real R2 PUT inside the paginate-page child task, the email's `downloadUrl` is the matching presigned GET (10-minute expiry refreshed on the email render, the user clicks the link inside the window). Both flows use the same `lib/r2.ts` and the same `file_metadata` table — one mechanism, two consumers.
- **The workloads the course keeps off R2.** Avatar uploads from Better Auth's hosted account UI in Unit 8 — if used, Better Auth's own avatar handling covers it (the course's own avatar field stays opt-in for the project's UI). Background-job output that lives only inside Trigger.dev's run state (every `paginatePage` child's intermediate CSV string) — stays in-memory inside the run, never written to R2 until the parent assembles the final blob. The 8.x React Email rendered output — that is HTML the email provider hosts, not bytes the app stores. Each "no" is named so the student does not over-extend.
- **The env surface.** Carried in from lesson 2 of chapter 068: `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET_NAME`. Plus a derived `R2_ENDPOINT` computed from the account ID in `lib/r2.ts`. `APP_URL` is reused — it goes in the bucket's CORS `AllowedOrigins`. Forecasts the chapter 069 `.env.example`.
- **The lib surface.** `lib/r2.ts` exports the configured `S3Client`. `lib/files/presigned-put.ts` exports `presignedPut({ fileName, contentType, claimedSize, organizationId, uploadedBy }) -> { url, objectKey }`. `lib/files/presigned-get.ts` exports `presignedGet({ objectKey, expiresIn }) -> { url }`. `lib/files/finalize.ts` exports the post-upload HEAD-and-save. `lib/files/file-metadata.ts` wraps the row reads / writes in `tenantDb` scope. All four are 30-60 lines each; the SDK shows at the call site.
- **The Trigger.dev wire-back.** The chapter 067 `exportInvoices` parent task body grows one step: after the page loop, the accumulator string becomes a `Buffer`, the parent does `await s3Client.send(new PutObjectCommand({ Bucket, Key: \`org/${organizationId}/exports/${ctx.run.id}.csv\`, Body: buffer, ContentType: 'text/csv' }))` directly (server-side PUT — the bytes do touch this function, but it's a background Trigger.dev worker, not a user-facing request, so the byte-pipe rule does not apply). The email step's payload gains `downloadUrl: await presignedGet({ objectKey, expiresIn: 600 })`, the email template renders the link, the user clicks within the window. The `file_metadata` table is *not* used for export outputs — those are short-lived, the row would be noise; a 7-day R2 lifecycle rule on the `org/*/exports/*` prefix cleans them up.
- **The two patterns side by side.** User uploads use `file_metadata` + presigned PUT from the browser + soft-delete. Export outputs use no metadata row + server-side PUT from a worker + lifecycle-rule cleanup. The senior call: short-lived generated outputs with one consumer (the email recipient) do not need a metadata row; long-lived user-managed files do. The lesson names the split explicitly so lesson 6 of chapter 069 lands without a "wait, why is this different" stall.
- **CORS in production.** The dev bucket has `AllowedOrigins: ['http://localhost:3000']`; the prod bucket has `AllowedOrigins: ['https://app.example.com']`. The lesson restates the rule: never `*` in production. Deploy order: configure CORS on the prod bucket *before* the first production user upload, or the upload fails.
- **Deploy surface.** No new CI step — the R2 credentials are environment variables; bucket configuration is a one-time setup. Restates the deploy-Trigger-first ordering from lesson 7 of chapter 066 only insofar as: deploying a new feature that issues presigned URLs is fine; rotating R2 credentials needs the staged rollover discipline from Unit 20.
- **Cost in operational terms.** Per-GB storage, per-thousand Class A operations (writes), per-thousand Class B operations (reads). The chapter 067 export job: one PUT per export, one GET per email click — pennies per thousand exports. The chapter 069 user-upload flow: one PUT per file, one GET per render — Class B reads dominate for a heavily-browsed gallery, so the senior reach is "cache the presigned GET inside the page render for the page's lifetime, do not re-issue on every component re-render." Names the rule once.
- **The off-ramp.** Apache 2.0 / non-existent — R2 is a managed service. The structural off-ramp is the S3-compatible API: changing four env variables and the endpoint moves the code unchanged to S3, Backblaze B2, MinIO, or DigitalOcean Spaces. Restates the structural-portability win from lesson 1 of chapter 068.
- **Forward note for chapter 069.** The project ships the user-upload path end-to-end: starter clone, `presignedPut` Server Action, browser-side direct PUT, `file_metadata` migration + `finalizeUpload` action, `Files` list with fresh presigned GETs per render, and the retrofit on the chapter 067 export job so the email carries a real R2 link.
- **Watch-outs:** putting the function in the byte path for user uploads — undoes the entire architecture; using `file_metadata` for short-lived export outputs — table grows unbounded; sharing a single bucket across staging and production — leak surface; reusing the same presigned GET URL across renders past expiry — silent dead links; rotating credentials without staging — production reads fail in the gap; treating R2 as a database — every list / search / aggregation belongs in Postgres against `file_metadata`.

What this lesson does not cover:

- The chapter 069 project's code — Chapter 069.
- Background orphan-byte cleanup — named, not built.
- Multipart upload for files >100 MB — named, not built.
- Cloudflare Images for transformation — out of scope.
- Migration off R2 to S3 in operational terms — Unit 20 cuts the deploy story.

Estimated student time: 25 to 30 minutes. The bridge lesson — short, names the workloads and pre-loads the chapter 069 project.

## Lesson 6 — Quizz

Top 10 topics to quiz:

- The threshold — when object storage earns its weight (user uploads, generated assets the app serves back, third-party media) and the conditions that do not qualify (small blobs, static marketing assets, "for cleanliness").
- Why R2 over S3 for SaaS unit economics — zero egress flipping the per-month bill for any read-heavy SaaS; the S3-compatible API as the structural off-ramp.
- The three nouns — bucket (namespace + CORS + scoped credentials), object key (`org/${orgId}/files/${id}`), `file_metadata` row (canonical identity in Postgres) — and which one owns what.
- Bucket-scoped tokens, one per environment, with minimum operations — the trust-boundary discipline, plus the CORS pitfalls (`*` in origins or headers, the R2-specific behavior on `AllowedHeaders: ['*']`).
- Architectural Principle #3 in practice — the function signs a URL, the browser PUTs to R2 directly, the function never sees the bytes; why the multipart-POST-to-function shape is wrong.
- Presigned PUT signature mechanics — `ContentType` and `ContentLength` pinned in the signed URL, short expiry (5 min), the browser must send the matching `Content-Type` header, the layered size defense (client check, server cap, post-upload HEAD).
- Presigned GET mechanics — fresh-issued per render, never persisted to the row, 10-minute expiry, the "URL in email" pitfall.
- The two-step write — sign → upload → finalize-with-HEAD → row insert. Why the row is written after the upload, not before, and what the orphan failure modes look like in both directions.
- The `file_metadata` row shape — `id` as the key segment, `originalFileName` separate from `objectKey`, `byteSize` from post-upload HEAD not client claim, soft-delete with cooled-off object cleanup, tenancy enforced at every read.
- The course wiring — user uploads use `file_metadata` + presigned PUT + soft-delete; export outputs use no metadata row + server-side PUT from the Trigger.dev worker + lifecycle-rule cleanup; one `lib/r2.ts`, two consumers.
