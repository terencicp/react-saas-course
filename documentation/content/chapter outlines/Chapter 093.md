# Chapter 093 — Component tests, off by default

## Chapter framing

Chapter 093 is the **thin component band of the honeycomb** from chapter 090 — conditional in the same sense Chapter 16's TanStack Query and Zustand are conditional. The default for a Next.js 16 SaaS in 2026 is *no component tests at all*: most of the rendered surface is framework-orchestrated (Server Components, `<Link>`, `redirect()`), most of the business behavior tests at the seam (chapter 092), and most "did the click do the thing" is either inside the Server Action or, for the 20–30 money paths, an E2E concern (chapter 094). Reaching for React Testing Library earns its weight on three specific surfaces, and only those: a **shared component library** consumed in many places (one bug, many regressions); a **complex stateful interactive component** where state transitions are the lesson (a multi-step form with conditional fields, a virtualized table with selection, a date-range picker built on Temporal); and a **critical UX path** that is too granular for E2E but too consequential to leave to manual review (the cookie consent gate from Unit 17, the checkout summary line in Unit 12). Outside those three triggers, the senior reach is to delete the test, not write it.

Threads through every lesson. **The unit under test is a client component or a pure presentational component, never a Server Component** — async Server Components are not a 2026 RTL surface; they get tested through their data layer at the seam (chapter 092) and at the money-path boundary in E2E (chapter 094). **The Vitest `component` project is a separate slice** (`environment: 'jsdom'`, glob `*.dom.test.tsx`) so the unit and integration projects stay node-fast. **`@testing-library/react` v16 with React 19, `@testing-library/user-event` v14, `@testing-library/jest-dom/vitest` for matchers** is the pinned surface. **Queries follow the priority ladder** — accessible role and name first, label and placeholder for forms, text for content, `data-testid` only as last resort — because tests written to the accessibility tree double as accessibility regressions. **`userEvent.setup()` once per test, `await` every interaction** — `fireEvent` is the v14-era escape hatch named once for completeness. **Behavior is what the user observes** — what's rendered, what's announced to assistive tech, what changes after an interaction — never component internals, props, or state. **MSW carries through from chapter 092** — the same boundary mocks serve network calls a client component fires (`useFormStatus`, `fetch`-in-client-effect, optimistic updates).

## Lesson 1 — When RTL earns its weight

The three triggers (shared library, complex state, critical UX) plus the accessibility-regression trigger that make React Testing Library worth its cost, and the anti-triggers that say delete the test instead of writing it.

Topics to cover:

- **The senior question.** A new feature lands with three new components. Without a rule, the reflex is "write tests for the components." Two months later the suite has 200 component tests, 80% are re-tests of presentational primitives (`<Card>` renders children, `<Button>` accepts `onClick`), CI is slow, the suite catches no production bugs, and the team learns to skip the component-test review. The reach is the inverse rule: **component tests are off by default**; they earn weight against three named triggers, and a fourth implicit one — accessibility regressions on a high-traffic page. Reach for RTL only where the bug density justifies the cost.
- **Why "off by default."** A Next.js 16 SaaS renders most of its tree on the server. A Server Component fetching invoices and rendering a list has its bugs at the seam (the query, the auth check, the formatter) — caught by chapter 092. The rendered output is framework-orchestrated; testing it with RTL re-tests Next.js. The Client Components left are mostly thin: forms wired to Server Actions (covered by the action's integration test), modals, tooltips, navigation. Their behavior is either trivial or visible on first manual interaction.
- **Trigger 1 — shared component library.** A `<DataTable>`, `<Combobox>`, `<DatePicker>` consumed in 30+ places. One bug ships 30 regressions. The cost-per-test math flips: a test that prevents one regression in a shared primitive pays for itself ten times. Shared in the course's codebase: the form field set built in Unit 7, the toast surface from Unit 14, the date-range picker from Unit 18. Reach for RTL here.
- **Trigger 2 — complex stateful interactive component.** A multi-step form with conditional branches (Unit 7's `<SubscribeForm>`), a virtualized list with multi-select and keyboard navigation, a command palette with async search. State transitions are the lesson; manual walk-through is unreliable; the integration test at the action seam doesn't see the form's branch logic. Reach when the state graph has more than three nodes or interactions span more than two events.
- **Trigger 3 — critical UX path.** Behaviors too granular for E2E but too consequential to leave to manual QA. The course's two: the **cookie consent gate** (Unit 17) — gating analytics on consent must work and a one-line bug ships GDPR exposure; the **checkout summary line** (Unit 12) — the user reads the total, the tax, the trial-end date before committing money. Each is a single component with branching content; E2E covers the *happy path* but not the seven content variants. Reach here.
- **The implicit fourth — accessibility regressions on high-traffic pages.** Tests written to the priority ladder (lesson 3 of chapter 093) double as accessibility audits. A login form, a primary CTA, a navigation header are surfaces where a missed label or a button without an accessible name is a real bug. The test that finds the submit button by role and name fails the moment the name disappears; the accessibility regression and the test failure are the same event.
- **The anti-triggers — when not to reach for RTL.** A purely presentational component (`<Card>`, `<Section>`) with no state and no branching content. A Server Component (async or otherwise). Anything covered end-to-end by the Server Action integration test from lesson 7 of chapter 092. Anything covered by Playwright's money-path tests in chapter 094. The framework-owned surface (`<Link>`, `<Image>`, route segment behavior). Plugin internals (TipTap commands, virtualization library internals — the library owns those).
- **The cost shape.** Unit `/lib`: 5 ms per test. Integration with rollback: 20–80 ms. Component with jsdom: 100–300 ms (jsdom boot, render, queries). E2E: seconds. Component tests sit in the middle — slow enough that 200 of them double the local watch loop, fast enough that 20 well-chosen ones cost nothing. The discipline is in the count.
- **What component tests catch that integration tests don't.** Conditional render branches driven by client state; keyboard navigation order; focus management after a modal closes; an error message rendered from a `useFormState` reducer; an optimistic-update reconciliation; the accessible name on a dynamic button (`"Delete invoice INV-001"`, computed from props). All of these live in the component, not at the seam.
- **What integration tests catch that component tests don't.** Whether the Server Action wrote the row. Whether the cross-tenant filter held. Whether the audit log fired. Whether the rate limiter allowed the request. All of these live at the seam, not in the component. Component tests asserting on a Server Action's effect are mocking too deep.
- **The senior's "before you write this test" checklist.** (1) Is this a Server Component? Stop — wrong surface. (2) Is this covered by the Server Action integration test? Stop — duplicate. (3) Is this a one-off non-shared presentational component? Stop — not earning weight. (4) Is this on the money path Playwright tests? Stop — duplicate at higher cost. (5) Is this a shared library, a complex state machine, a critical UX path, or an accessibility-sensitive primitive? Reach. The checklist is a five-second mental gate.
- **Year-one SaaS may ship zero component tests.** Same shape as chapter 094 on E2E. A small team shipping fast on a stack where most of the UI is server-rendered and most of the behavior is at the seam is correct to write zero RTL tests in year one. The trigger language is what tells them when to start, not "we should have component tests."
- **Forward references.** lesson 2 of chapter 093 owns the setup and the render helper. lesson 3 of chapter 093 owns the query priority and what "behavior" means at this layer. lesson 4 of chapter 093 walks the four to six components in the course's app where the trigger has actually been met. lesson 5 of chapter 093 is the quiz.
- **Watch-outs.** Writing component tests for every Client Component as a coverage exercise — the bug density doesn't justify the cost; testing Server Components with RTL — the framework's surface and async-component support is fragile in 2026; mocking a Server Action import and asserting the mock was called — re-tests the integration test from lesson 7 of chapter 092 at higher cost; snapshot tests on every component — churn destroys signal; "the component renders" as a test name and asserting only that it didn't throw — coverage theatre; reaching for RTL when the behavior is visual (spacing, color, animation) — visual-regression tooling is the surface, not RTL.

What this lesson does not cover:

- jsdom and RTL setup, render helper, user-event configuration — lesson 2 of chapter 093.
- The query priority ladder and what "behavior" means at this layer — lesson 3 of chapter 093.
- The specific components in the course app — lesson 4 of chapter 093.
- E2E money-path trigger — Chapter 094.
- Server-Action integration testing — lesson 7 of chapter 092.
- Visual-regression testing (Chromatic, Percy) — out of scope.
- Storybook play functions as a parallel surface — named once if at all.

Estimated student time: 30 to 40 minutes. Decision lesson; the rest of the chapter sits under the trigger this one installs.

## Lesson 2 — The jsdom project and the render helper

Wiring the third Vitest project against jsdom, pinning the `@testing-library/*` set for React 19, and building the `render` wrapper that pre-applies providers and returns a ready `userEvent` instance.

Topics to cover:

- **The senior question.** RTL is greenlit (lesson 1 of chapter 093). What does it cost to get the first test running, and what's the smallest surface that holds up across the chapter? The reach: a third Vitest project (`component`, `environment: 'jsdom'`), the `@testing-library/*` pin set, a thin `render` helper that wires the providers the codebase already has, `userEvent.setup()` as the per-test reflex.
- **The third Vitest project.** Extending `test.projects` from lesson 1 of chapter 090 with `{ name: 'component', test: { environment: 'jsdom', include: ['src/**/*.dom.test.tsx'], setupFiles: ['./vitest.setup.dom.ts'] } }`. The `.dom.test.tsx` suffix is the discriminator — keeps jsdom out of the node-fast unit and integration lanes. `vitest --project component` runs the slice; `vitest` runs all three.
- **The pin set, May 2026.** `@testing-library/react` v16+ (the React 19 / RSC-aware line; v16 dropped the React 18 codepath), `@testing-library/user-event` v14, `@testing-library/jest-dom` v6 (the `/vitest` entrypoint registers matchers against `expect`), `jsdom` as the environment. `pnpm add -D @testing-library/react @testing-library/user-event @testing-library/jest-dom jsdom`. Course pins jsdom over happy-dom — better Testing Library compatibility, fewer surprises on `Element.checkVisibility`, `ResizeObserver`, focus semantics. Happy-dom is named once as the faster alternative; the course doesn't switch.
- **`vitest.setup.dom.ts` — the one setup file.** Three lines doing four jobs: `import '@testing-library/jest-dom/vitest'` (registers `toBeInTheDocument`, `toHaveAccessibleName`, etc.), `import { afterEach } from 'vitest'; import { cleanup } from '@testing-library/react'; afterEach(cleanup)` (unmounts after each test — RTL v16 doesn't auto-cleanup with the right runner-config combination; name it), polyfills for jsdom gaps the codebase touches (`ResizeObserver`, `matchMedia`, `IntersectionObserver` shims — the smallest stub that satisfies the contract, not a real impl).
- **The `render` helper.** `src/test/render.tsx` exports `render(ui, options?)` that wraps RTL's `render` and pre-applies the providers the codebase wraps `app/layout.tsx` in: theme, `next-intl` locale (Unit 18), the `Toaster` portal target. Tests call the wrapper, never RTL's `render` directly. Helper signature returns `{ ...rtlReturn, user }` where `user = userEvent.setup({ delay: null })` — pre-built `user` object the test calls without re-setup. The wrapper is the seam: a new provider lands in production, the helper updates, every test inherits.
- **`userEvent.setup()` discipline.** v14's API: `const user = userEvent.setup()` once per test (or returned from `render`), then `await user.click(...)`, `await user.type(...)`, `await user.keyboard('{Enter}')`. Every interaction is awaited — v14 events are async and resolve after their downstream effects (state updates, microtasks, transitions) settle. Forgetting `await` is the same trap as the unawaited-promise bug class from lesson 5 of chapter 091. Vitest 3's silent-pass and v4's hard-fail apply.
- **`userEvent` over `fireEvent`.** `fireEvent.click(button)` dispatches one synthetic event; `user.click(button)` dispatches `pointerdown`, `mousedown`, `focus`, `pointerup`, `mouseup`, `click` and waits for React to flush. The difference catches the bug where a component listens to `focus` (not just `click`) to fire validation. Default to `userEvent`; reach for `fireEvent` only when a specific synthetic event needs isolation (a `scroll` listener under test where `user.scroll` doesn't exist — name it, don't habit it).
- **Async queries — `findBy`.** A render that flips state asynchronously (a `useEffect`-triggered fetch resolved by an MSW handler) won't have its content on first `getBy`. `findBy*` returns a promise that retries until the element appears or the default 1000 ms timeout elapses. `await screen.findByRole('alert', { name: /payment failed/i })`. Default reach for any post-interaction observable. `waitFor` is the lower-level escape hatch for non-DOM assertions (a mock got called, a prop changed) — name it; prefer `findBy`.
- **`screen` vs. destructuring queries.** `screen.getByRole(...)` is the senior reach — `screen` reflects the global document, no destructuring drift if the test refactors. Destructured `getByRole` from `render`'s return scopes to the rendered container; useful when the test mounts two separate trees, rare in practice. Default to `screen`.
- **Providers and the i18n thread.** The course's i18n setup (Unit 18) wraps `<NextIntlClientProvider>` around client trees. The `render` helper accepts `{ locale = 'en-US', messages = enMessages }` and wires the provider so localized text queries (`findByText(t('checkout.confirm'))`) work without each test recreating provider boilerplate. Switching locale in a single test is one option flip.
- **MSW carried in.** The integration project's MSW server (lesson 5 of chapter 092) is reused: the `component` project's setup file imports the same `server` and registers `listen`/`resetHandlers`/`close` lifecycle. Tests that hit `fetch` from a client component (rare in the course's stack, but real for `useSWR`-style or `useFormState` patterns) get the same boundary mocks. No new MSW boilerplate.
- **Mocking `next/navigation` and `next/headers` in jsdom.** A client component reading `useRouter`, `usePathname`, `useSearchParams` needs the module mocked because jsdom has no Next router. Setup file registers `vi.mock('next/navigation', () => ({ useRouter: () => ({ push: vi.fn(), refresh: vi.fn() }), usePathname: () => '/invoices', useSearchParams: () => new URLSearchParams() }))`. Per-test overrides via `vi.mocked(useRouter).mockReturnValue(...)`. Same shape as lesson 3 of chapter 092's auth mock — named once, reused everywhere.
- **What the `render` helper does NOT own.** Test data — that's the factories from lesson 2 of chapter 091 / lesson 3 of chapter 092. Auth state — components reading `auth()` are usually Server Components; client components receive auth as a prop, not a hook, and the test passes the prop. Network — that's MSW. The helper is exactly providers and `user`.
- **The watch loop.** `vitest --project component` in watch mode triggers on file save. The full DOM suite at the chapter's end is 15–30 tests; runs in 3–5 seconds. If it grows past 100 tests, revisit the trigger (lesson 1 of chapter 093) — likely over-reach.
- **Why not Storybook + play.** Storybook's `play` function with `@storybook/test` is a parallel interactive-component test surface. Two reasons the course doesn't pin it: (1) Storybook setup carries its own configuration tax and a separate runner story; (2) the same RTL queries and `user-event` calls already work inside Vitest with zero new runtime. Named once for awareness; the course writes against Vitest.
- **Watch-outs.** Forgetting `afterEach(cleanup)` leaves rendered trees alive across tests — queries return stale elements; missing `jsdom` polyfills (`ResizeObserver`, `matchMedia`) crash on first import — name the smallest stub; using the wrong env (`environment: 'node'` for a `.tsx` test) — `document` undefined; mixing `userEvent` and `fireEvent` in one test makes the timeline ambiguous; `userEvent.setup()` inside `beforeEach` and then awaiting inside `it` works but obscures the per-test seam — return `user` from `render`; pinning `@testing-library/react` v15 with React 19 — known peer-dep mismatch, jump to v16+; running the `component` project against the integration DB by accident — keep test data in props and MSW, no DB.

What this lesson does not cover:

- The query priority ladder and "behavior" at this layer — lesson 3 of chapter 093.
- The course's actual component coverage — lesson 4 of chapter 093.
- React Server Component testing — out of scope (E2E in chapter 094 or seam in chapter 092).
- Visual-regression tooling — out of scope.
- Storybook play functions — named once, not pinned.
- E2E browser testing — Chapter 094.

Estimated student time: 45 to 55 minutes. Setup-and-wiring; one-time but everything downstream rides on it.

## Lesson 3 — The query ladder is the accessibility audit

The role-plus-name priority ladder, the `getBy` / `queryBy` / `findBy` split, and the rule that behavior is what a user observes — never internal state, props, or class names.

Topics to cover:

- **The senior question.** A button under test has class `btn-primary`, id `submit`, text `Confirm`, aria-label `Confirm purchase`, role `button`, and a `data-testid="submit-btn"`. Which query finds it? The naive reach picks `getByTestId` — fastest to write, most fragile in maintenance. The senior reach follows the **priority ladder**: `getByRole` with `name`, then label / placeholder, then text, then `getByTestId` only as a last resort. The ladder is the same axis as accessibility — a test that fails because the accessible name disappeared is an accessibility regression caught for free.
- **The priority ladder, in order.** (1) **`getByRole(role, { name })`** — semantic role plus accessible name. The single highest-value query. Works for `button`, `link`, `textbox`, `combobox`, `checkbox`, `radio`, `heading`, `dialog`, `alert`, `region`. (2) **`getByLabelText(text)`** — form inputs by their associated `<label>`. (3) **`getByPlaceholderText(text)`** — only when no label exists (usually a sign the form is inaccessible, but real). (4) **`getByText(text)`** — non-interactive content. (5) **`getByDisplayValue(value)`** — current value of an input. (6) **`getByAltText(text)`** — images. (7) **`getByTitle(text)`** — `title` attribute. (8) **`getByTestId(id)`** — last resort for elements with no semantic surface (a styling wrapper, a portal anchor). The ladder is the published Testing Library guidance and the course's enforced reach.
- **Why role-first.** Role + name is *what assistive tech sees*. A test passing the role-first query proves the element is reachable by keyboard and announced correctly to a screen reader. A test passing only `getByTestId` proves nothing about accessibility — the element could be a `<div>` with a click handler and the screen reader would never find it. The query is the audit.
- **Accessible names and `name` matching.** `name` accepts a string (exact match), a regex (`{ name: /confirm/i }`), or a function `(name, element) => boolean`. Regex is the default reach — content shifts (`"Confirm purchase"` to `"Confirm payment"`) and the test stays green if the regex still matches the stable part. Strict-equality string match for tests that *should* break on copy change (a privacy-relevant label, a legally-required string).
- **`getBy` vs. `queryBy` vs. `findBy`.** `getBy*` throws if missing, returns one element — assertion that "this is here now." `queryBy*` returns `null` if missing — the only correct query for "this is NOT here" (`expect(screen.queryByRole('alert')).not.toBeInTheDocument()`). `findBy*` returns a promise, retries up to 1000 ms — "this will be here after async work." The three serve different intentions; using the wrong one produces misleading failures (`getBy` for "not present" throws before the assertion runs).
- **`*AllBy` variants.** For multiple elements (`getAllByRole('listitem')`). Return arrays; assert `.toHaveLength(n)` then index. A test that needs to address a specific item by content (`getByRole('button', { name: /delete invoice INV-001/i })`) is usually better than indexing into an array.
- **Behavior at the component layer — definition.** Behavior is what a user (sighted or not) observes from outside the component: rendered text, accessible names, button states (disabled, pressed), images alt text, form values, navigation focus order, what's announced after an interaction. Behavior is *not*: which hook ran, which prop a child received, internal state shape, the order of `useEffect` calls, which classes were applied. The same rule as lesson 4 of chapter 090, specialized to the DOM.
- **The "user sees" reflex.** When writing an assertion: phrase it as "the user sees X" or "the user can do X." `expect(screen.getByRole('alert', { name: /payment failed/i })).toBeInTheDocument()` reads as "the user sees a payment-failed alert." `expect(mockOnSubmit).toHaveBeenCalledWith(...)` reads as "the test's mock got called" — implementation surface, lower-value test. The first is durable through refactor; the second breaks on a rename.
- **The matchers from `jest-dom`.** `toBeInTheDocument`, `toBeVisible`, `toBeDisabled`, `toBeEnabled`, `toBeChecked`, `toHaveValue`, `toHaveAccessibleName`, `toHaveAccessibleDescription`, `toHaveTextContent`, `toHaveAttribute`, `toHaveClass` (avoid — couples to implementation), `toHaveFocus`. Prefer the matchers that read as observations (`toBeVisible`, `toHaveAccessibleName`) over the ones that read as DOM detail (`toHaveClass`, `toHaveAttribute`).
- **Async assertions and `findBy`.** Any assertion after a `user.click` that triggers state, fetch, or transition uses `findBy` or `await waitFor(...)`. `findBy` is the default. `waitFor` is for non-DOM observations (a mock got called: `await waitFor(() => expect(mock).toHaveBeenCalled())`). Both have 1000 ms default; lengthen only for genuinely slow async (server-action effects MSW resolves with a delay), never to paper over flake.
- **`screen.debug()` and `logRoles`.** Debug aids. `screen.debug()` prints the current DOM. `screen.logTestingPlaygroundURL()` opens the Testing Playground in the browser. `logRoles(container)` lists every role and accessible name in the tree — the diagnostic when "I can't find this button by role" hits. Named once; remove from committed code.
- **The query-name reflex during writing.** Senior writes the assertion *first* — "the user sees the success toast with text 'Invoice sent'" — and the query falls out: `await screen.findByRole('status', { name: /invoice sent/i })`. If the query doesn't fit the role ladder, the *component* is wrong — missing `role="status"`, missing accessible name, no `<label>`. The test failing is the prompt to fix the component, not to drop down the ladder to `getByTestId`.
- **`data-testid` — the carve-out.** Three legitimate cases: (1) elements with no semantic role (a portal mount node, a styling wrapper); (2) third-party widgets the team doesn't own (a charting library's container — assert on its presence, not internals); (3) a screen with two semantically-identical regions (two `<main>`s during a transition) where role-plus-name is ambiguous. Outside those, `getByTestId` is a smell. The `data-testid="submit-btn"` reach is the lazy reach — fix the component, write the role query.
- **Implicit and explicit roles.** `<button>` has implicit role `button`. `<input type="checkbox">` has implicit role `checkbox`. `<a href>` has implicit role `link`. `<a>` without `href` has *no* role. `<div role="button">` has explicit role `button` but needs `tabIndex={0}` and a keyboard handler — RTL queries find it, accessibility tools fail it. Tests reveal the gap.
- **Naming the unhappy DOM tree.** A component that ships without `role="dialog"` on its modal, without `aria-labelledby` on its title, without an accessible name on its close `<button>` *renders* but is untestable by the role-first ladder. The course's discipline: write the test, watch the query fail, fix the component to expose the semantic structure. Then the test passes and the component is accessible.
- **What component tests assert across the layers.** **Render** — "the user sees X." **Interaction** — "after the user clicks, the user sees Y." **Branching content** — "given prop Z, the user sees X; given prop Z', the user sees Y." **Accessibility** — "the submit button has accessible name 'Subscribe to Pro plan'." **Focus management** — "after the modal closes, focus returns to the trigger button" (`toHaveFocus`). **Negative assertions** — "before submission, no error alert is in the document" (`queryByRole`).
- **Counter-example tests, named for what they smell of.** `expect(component.state.isOpen).toBe(true)` — testing internal state, the component refactors and the test breaks; `expect(wrapper.find('Button').props().onClick).toBeDefined()` — Enzyme-era implementation peek; `expect(container.querySelector('.btn-primary')).toBeInTheDocument()` — CSS-class coupling; `toMatchSnapshot()` on the entire tree — churn without signal. Each ships with its remediation.
- **Custom queries.** `within(element).getByRole(...)` scopes a query to a sub-tree — useful for a list item with a delete button (`within(row).getByRole('button', { name: /delete/i })`). Standard reach, no extension required. Custom matchers (`expect.extend`) are the same threshold as lesson 1 of chapter 091 — domain-repeated three times, then extract.
- **Watch-outs.** Defaulting to `getByTestId` for speed — the suite ages badly; `getByText('Confirm')` matching a tooltip and a button — narrow with role; `getByRole('button')` when the page has many buttons — add `name`; `findByRole` without `name` on an `alert` matching the first alert that ever renders — wrong alert, wrong test; `expect(...).toHaveClass('error')` — coupling to Tailwind class names that get refactored; `waitFor` wrapping a synchronous assertion — slows the suite for no reason; asserting on a child component's props through a render-prop pattern — testing implementation; `screen.debug()` left in committed code; `getByTitle` on a `<button title="…">` — `title` is not an accessible name (it shows on hover only), use `aria-label` and `getByRole({ name })`.

What this lesson does not cover:

- The components the course applies this to — lesson 4 of chapter 093.
- Mocking Server Actions imports — lesson 4 of chapter 093.
- Setup, render helper, user-event configuration — lesson 2 of chapter 093.
- E2E browser testing — Chapter 094.
- Accessibility audits at depth (axe-core, Lighthouse a11y) — out of scope here; named once.
- Visual-regression testing — out of scope.

Estimated student time: 50 to 60 minutes. The highest-value lesson in the chapter; the priority ladder is the durable reach.

## Lesson 4 — The catalog: five components that earn the test

A walk through the course app's component tests — cookie consent, subscribe form, date-range picker, data table, checkout summary — naming the trigger met, the behaviors to assert, and what to leave to the seam or to E2E.

Topics to cover:

- **The senior question.** The trigger from lesson 1 of chapter 093 names three surfaces; the priority ladder from lesson 3 of chapter 093 names the queries. What does that look like applied to the codebase the student has been building since Unit 4? The lesson walks a small, named catalog — four to six components — and for each one names *which trigger was met*, *what behaviors to assert*, *what to deliberately leave to the seam or to E2E*. Pattern lesson; the catalog is the canon.
- **Catalog component 1 — the cookie consent gate (Unit 17).** Trigger: critical UX path with legal weight. Behaviors to assert: the banner renders for a first-time visitor (no consent cookie present); clicking "Accept" sets the consent cookie and dismisses the banner; clicking "Reject" sets a different cookie value and dismisses; re-mount with the cookie set does not render the banner; the analytics gate (`useConsent()` hook) returns `false` before consent and `true` after. Mocks: `next/headers`-style cookie reading is replaced by a client cookie helper the component already uses; mock the cookie module. Assertions: `findByRole('dialog', { name: /cookie/i })`, `user.click(getByRole('button', { name: /accept/i }))`, `expect(...).not.toBeInTheDocument()` after, plus an assertion that the consent module's `setConsent` function was called with the right value. Left to E2E: the actual PostHog gate firing — Playwright's job in chapter 099 / chapter 094.
- **Catalog component 2 — the multi-step subscribe form (Unit 7 / Unit 12).** Trigger: complex stateful interactive component. Behaviors to assert: initial step renders only the plan selection; selecting "Pro" reveals the seat-count field; selecting "Free" hides the seat-count field; submit is disabled until required fields are filled; submitting with valid data calls the Server Action prop (a Server Action import mocked at the function level — the action's own seam-test owns its body, this test owns the *form's contract* with the action); the action returning `{ ok: false, error: { code: 'CARD_DECLINED' } }` renders the error alert with the right accessible name. Mocks: the imported Server Action via `vi.mock('@/server/actions/createSubscription', () => ({ createSubscription: vi.fn() }))`; per-test it returns `Result.ok` or `Result.err`. Assertions: role + name queries throughout; `findByRole('alert', { name: /card was declined/i })` for the error branch. Left to integration (lesson 7 of chapter 092): the action's actual database write, audit log, Stripe call. Left to E2E (chapter 094): the full Stripe Checkout redirect path.
- **Catalog component 3 — the date-range picker (Unit 18, built on Temporal).** Trigger: shared component library — consumed across Reports, Invoices, Filters. Behaviors to assert: rendering with a default range shows the dates in the user's locale (the i18n thread); selecting a new start date past the current end date snaps the end date forward; keyboard navigation (`{Tab}`, `{Enter}`, `{ArrowRight}`) moves the focused day; `Esc` closes the popover and returns focus to the trigger; selecting a range fires the `onChange` prop with two `Temporal.PlainDate`s. Mocks: the Temporal clock seam from lesson 3 of chapter 091 — `vi.setSystemTime(new Date('2026-05-14'))` so "today" is deterministic. Assertions: role + name queries for the calendar grid (`getByRole('grid')`), keyboard via `user.keyboard('{ArrowRight}')`, focus assertions via `toHaveFocus()`. Left out: the calendar's internal layout, which library cell wraps which day — implementation.
- **Catalog component 4 — the data table with selection (Unit 11).** Trigger: shared component library. Behaviors to assert: rows render with the expected accessible names (`getAllByRole('row')` and content checks); clicking a row checkbox selects it and updates the toolbar (`"1 selected"`); the header checkbox toggles all rows; selecting rows enables the "Delete" toolbar button (`toBeEnabled` flips); clicking "Delete" with two rows selected opens the confirm dialog with the count in its accessible name (`getByRole('dialog', { name: /delete 2 invoices/i })`). Mocks: the Server Action import for delete, mocked to return `Result.ok`. Assertions: role-first throughout, `within(row)` for per-row addressing. Left out: pagination (URL-state, integration territory); virtualization internals (the library owns).
- **Catalog component 5 — the checkout summary line (Unit 12).** Trigger: critical UX path on the money surface. Behaviors to assert: with `plan: 'pro'`, `seats: 5`, `couponCode: undefined`, the line reads the expected total; with a coupon applied, the discount line appears and the total updates; the trial-end date renders in the user's locale and the user's timezone; the "Subtotal," "Tax," "Total" labels are present and announced. The component is a pure presentational variant table — perfect RTL surface. Behaviors are content under prop variants; six to eight `it` blocks cover the matrix. No interactions, no mocks beyond locale provider. Left out: the actual Stripe Checkout redirect (E2E in chapter 094); tax computation (a `/lib` unit test from chapter 091 owns it).
- **Catalog component 6 (optional, conditional reach) — the command palette / combobox.** Trigger: shared component library *if* the course's app ships one. Behaviors to assert: typing filters options; `{ArrowDown}` moves focus through results; `{Enter}` fires the action; `{Esc}` closes and returns focus; the listbox has `role="listbox"` and items have `role="option"`. Include only if the course has built one by this chapter; mention as a typical shape if not.
- **What does NOT make the catalog.** Every `<Card>`, `<Section>`, `<PageHeader>` — no trigger met. The Stripe-redirect button on the pricing page — covered by E2E (chapter 094). The page-level Server Components rendering each surface — framework-orchestrated. The Server Actions themselves — covered at the seam (lesson 7 of chapter 092). The catalog is short on purpose.
- **The Server Action mock-import pattern.** When a client component imports a Server Action and the test isn't end-to-end, `vi.mock('@/server/actions/createSubscription', () => ({ createSubscription: vi.fn() }))` at the top of the test file. Per-test, `vi.mocked(createSubscription).mockResolvedValue(Result.ok({...}))` or `Result.err({ code: '...' })`. The test asserts the component's reaction; the action's body is the integration test's job (lesson 7 of chapter 092). The two tests *compose* — the form test trusts the action's contract, the action test trusts no client.
- **Locale-aware assertions.** The i18n thread from Unit 18 means content shifts per locale. Tests run with a default locale (`'en-US'` in the `render` helper). Locale-sensitive components get a second test in `'es-ES'` only when the locale-switching behavior is itself the lesson. Don't multiply locales pointlessly.
- **Per-component file layout.** `src/components/checkout/SubscribeForm.dom.test.tsx` next to the component. One `describe` per component; one `it` per observable behavior. Six to ten `it` blocks is a healthy file; past fifteen, the component is doing too much or the tests are duplicating the integration test.
- **The reviewer checklist for a new component test PR.** (1) Which trigger from lesson 1 of chapter 093 was met? Name it in the PR description. (2) Are the queries role-first? `data-testid` appearances need justification. (3) Is the test asserting on what the user observes? Internal state, prop calls, or class names are smells. (4) Is the Server Action mocked at the import — and does its real test exist at the seam? (5) Does the test pass with `vitest --project component --shuffle`? Each item is a one-second review reflex.
- **The "year-one zero" stance, applied.** A small team's chapter-chapter 093 work might be one or two components, not six. The catalog above is the *shape* the course presents; the *count* in a real codebase is whatever earns weight. Better five well-chosen tests than fifty box-ticking ones.
- **Watch-outs.** A form test that mocks the Server Action *and* asserts on the database (cross-layer confusion — pick a layer); writing the same form test at the integration layer *and* the component layer (duplicate coverage, double maintenance — keep the action's behavior at the seam, keep the form's branching at the component); a test that asserts on which icon library rendered which SVG path — implementation; a date-picker test asserting on the calendar library's internal class names — library internals; component tests for the cookie consent that don't reset `document.cookie` in `afterEach` — state leak across tests; a `toMatchSnapshot()` on the checkout summary that updates every time the copy changes — write explicit assertions on the labels and amounts; reaching for `screen.getByText('Total')` when two regions both have "Total" — `within` to scope.

- E2E money-path tests — Chapter 094.

- Visual-regression — out of scope.
- Storybook play functions — out of scope.
- Specific accessibility audits beyond what role queries surface — out of scope; pointer to axe-core if the team reaches for it.

## Lesson 5 — Quizz

Top 10 topics to quiz:

- The three triggers for RTL — shared component library, complex stateful interactive component, critical UX path; plus the implicit accessibility-regression trigger; component tests are off by default for a Next.js SaaS in 2026.
- Anti-triggers — presentational primitives, Server Components, anything covered by the Server Action integration test, anything on the E2E money path; year-one SaaS may ship zero RTL tests.
- The Vitest `component` project — `environment: 'jsdom'`, `*.dom.test.tsx` glob, separate setup file with `cleanup` in `afterEach`, jsdom over happy-dom for Testing Library compatibility.
- The pin set — `@testing-library/react` v16+ for React 19, `@testing-library/user-event` v14, `@testing-library/jest-dom/vitest` for matchers; the project's `render` helper wraps the codebase's providers and returns `{ ...rtl, user }`.
- `userEvent.setup()` once per test, `await` every interaction — v14 events are async; default `userEvent` over `fireEvent`; `fireEvent` is the carve-out for specific synthetic events.
- The query priority ladder — role+name first, then label, placeholder, text, displayValue, alt, title, testid as last resort; the ladder doubles as an accessibility audit.
- `getBy` (throws if missing), `queryBy` (returns null — for "not present"), `findBy` (async, retries up to 1000 ms — for post-interaction observables); `waitFor` for non-DOM assertions.
- Behavior at the component layer — what the user observes (rendered content, accessible names, focus, branching content under prop variants); never internal state, props passed to children, class names, or hook execution order.
- Server Action mocking pattern at the import — `vi.mock('@/server/actions/...', () => ({ name: vi.fn() }))`, per-test `mockResolvedValue` with `Result.ok`/`Result.err`; the action's body lives at the seam (lesson 7 of chapter 092), not in the component test.
- The catalog of components in the course's app where the trigger was met — cookie consent gate (critical UX), subscribe form (complex state), date-range picker (shared library + Temporal seam), data table (shared library), checkout summary (critical UX, money surface).
