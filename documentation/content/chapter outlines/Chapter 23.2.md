# Chapter 23.2 — Text, objects, and the chat surface

## Chapter framing

Chapter 23.1 settled the decisions — does the surface earn an LLM, how is cost bounded, how is the provider abstracted. Chapter 23.2 is the mechanics chapter: the four AI SDK Core primitives the senior reaches for to put text on the page (`streamText`, `generateText`, `generateObject`, `streamObject`), and the v5 UI hooks (`useChat`, `useCompletion`, `useObject`) that bridge the server stream to the React tree. By the end of the chapter the student can wire a streaming chat box or a structured-output form against any provider behind `lib/llm/models.ts`, with the messages array, the system prompt, the Zod schema, and the parts-array rendering all in the right place. Tool calling, agents, and generative UI are 23.3; the worked surface against the course's invoice data is 23.4.

Threads through every lesson: **every LLM call runs server-side**, the client only ever sees the stream (the route handler / Server Action is the seam, the SDK enforces it by hook shape); **the messages array is the conversation contract** — `system`/`user`/`assistant` roles, prompt + messages composition, the system prompt as the controller and user input as data (the prompt-injection posture from 23.1.2); **`maxOutputTokens` is mandatory on every call** — the cost cap from 23.1.2 is a syntactic discipline, not an operational reminder; **the model handle comes from `lib/llm/models.ts`** — every snippet imports a named handle, never inlines `openai('gpt-X')` (the abstraction discipline from 23.1.3); **v5 shapes are canonical** — `UIMessage` with a `parts` array, `sendMessage` / `regenerate`, the transport-based architecture, manually managed input state, `convertToModelMessages` at the route-handler seam; **structured output is the swap-friendlier shape** — `generateObject` with a Zod schema absorbs prompt-shape differences across providers, reach for it whenever the workload is extraction / classification / form-fill; **streaming is the default UX** — text deltas and partial objects render as they arrive, the user perceives the first token, not the last.

## Lesson 23.2.1 — streamText, generateText, and the route-handler seam

The two text-generation primitives, the messages-array conversation contract, the system prompt as controller, and the Next.js 16 route handler shape that wraps every LLM call with auth, quotas, `onFinish` audit writes, `finishReason` handling, and `abortSignal` cancellation.

Topics to cover:

- **The senior question.** Once the trigger from 23.1.1 has landed, what is the smallest call that puts model output on the page, and what does that call site look like in a Next.js 16 route handler? The lesson is the two text-generation primitives, the messages contract that feeds them, and the streaming-vs.-batch decision at the call site.
- **`streamText` vs. `generateText` — the streaming-vs.-batch cut.** `generateText` runs the model to completion and returns the full string in one Promise; `streamText` returns a stream of text deltas the route handler pipes to the client. The default for any user-facing surface is `streamText` — the user perceives first-token latency, which is half the time-to-completion for a long answer. Reach for `generateText` when the output is small and a downstream step depends on the complete string (a one-line classification result that feeds a Drizzle query, a JSON tag the route handler logs). Senior call: streaming for human readers, generation for code consumers.
- **The minimal call site — inline shape.** `streamText({ model: smartModel, system, messages, maxOutputTokens: 1000 })` and `generateText({ model: fastModel, prompt, maxOutputTokens: 200 })` — two argument shapes, one returning a stream, one returning a string. The model handle is the named export from `lib/llm/models.ts` (23.1.3); `maxOutputTokens` is non-optional in this course's style (23.1.2). Show both; do not show the call without the cap.
- **The messages array — the conversation contract.** Every multi-turn call passes `messages: ModelMessage[]` where each message has `{ role: 'system' | 'user' | 'assistant', content }`. The system message owns the instructions and persona; user and assistant messages alternate to form the conversation history. The SDK distinguishes `ModelMessage` (what the model sees, lossy) from `UIMessage` (what the app stores, full) — at the route-handler seam the client sends `UIMessage[]` and the handler calls `convertToModelMessages(messages)` before passing to `streamText`. Name both types here; the UI side lands in 23.2.3.
- **`prompt` vs. `messages` — when each fits.** Single-turn calls (one-shot classification, one-shot summary) pass `prompt: string` and the SDK wraps it in a single user message. Multi-turn or conversational calls pass `messages`. A `system` prop sits alongside both, prepending the system message. The senior cut: `prompt` for stateless single-turn calls inside backend pipelines; `messages` for any user-facing chat or any call that needs context.
- **The system prompt — the controller, not the conversation.** The system prompt sets the model's role, the answer constraints, the refusal rules, and the output format. The 23.1.2 posture against prompt injection is structural: the system prompt is the controller (trusted, code-authored), the user messages are data (untrusted, never spliced into the system prompt as a string). A senior writes the system prompt in a constant or a `lib/llm/prompts.ts` file, never templates user input into it. Name the rule; show a system prompt as a short inline example for a single domain (e.g., an invoice-domain assistant).
- **The Next.js 16 route handler shape — the seam.** Every user-facing LLM call lands in a Route Handler under `app/api/.../route.ts` (Chapter 7.5 / 5.x), wrapped by the `authedRoute(role, schema, fn)` from 10.2.3 plus the rate-limit and quota guards from 23.1.2. The handler shape: parse the request body (validated `UIMessage[]`), call `convertToModelMessages`, call `streamText`, return `result.toUIMessageStreamResponse()`. The senior call: the stream-return helper is the contract between the SDK and the `useChat` hook — using `Response` directly with a custom stream breaks the parts protocol.
- **`onFinish` and the post-call hook.** Both `streamText` and `generateText` accept an `onFinish` callback fired with the final `{ text, usage, finishReason, response }`. The senior reflex: this is where the `usage` field (input/output tokens) and the audit-event write from 23.1.2 live. Without `onFinish`, the post-call accounting (counter increment, audit event, cost-cents computation) has no place to land. Name the slot; the handler imports the audit helper and writes the `llm.call.completed` event inside `onFinish`.
- **`finishReason` and graceful caps.** The result carries a `finishReason` — `'stop'` (the model finished), `'length'` (`maxOutputTokens` hit), `'content-filter'` (provider moderation tripped), `'error'`. The senior surfaces this to the UI: a chat answer truncated by `'length'` shows a "response was cut off" affordance; a `'content-filter'` shows the policy message. A handler that silently ignores `finishReason` produces a UX where the answer ends mid-sentence with no explanation.
- **Aborting a stream — `abortSignal` for cancellation.** `streamText` accepts an `abortSignal`; the route handler wires it to the request's signal so a user navigating away cancels the model call mid-stream. The cost discipline matters here — an abandoned request that keeps generating burns tokens for nothing. The senior pattern: pass `request.signal` through; the SDK and provider observe it.
- **Temperature and the determinism dial — a one-line treatment.** The `temperature` argument controls randomness; the senior default for SaaS workloads is low (0–0.3 for classification, summarization, extraction) and only raised when creative variance is the feature. Skip the deep ML treatment; name the dial, name the default.
- **A short diagram (mermaid) — the call lifecycle.** Client `useChat` `sendMessage` → POST `/api/chat` with `UIMessage[]` → `authedRoute` guards → `convertToModelMessages` → `streamText` → `onFinish` writes audit event → `toUIMessageStreamResponse` → client receives stream parts. The diagram makes the seams visible; each seam has a name from prior chapters.
- **Watch-outs:** inlining the model handle (`openai('gpt-X')`) at the call site instead of importing from `lib/llm/models.ts`; omitting `maxOutputTokens` "for now"; templating user input into the system prompt as a string (`` `You are an assistant for ${userInput}` ``) — the prompt-injection class; returning `new Response(stream)` instead of `toUIMessageStreamResponse()` — the parts protocol breaks and the client renders garbage; doing the `usage` write outside `onFinish` (e.g., before `await streamText`) — the call hasn't completed yet, the counts are wrong; using `generateText` for a long user-facing answer — the user stares at a spinner for 8 seconds; not forwarding `request.signal` — abandoned streams keep burning tokens; treating `finishReason` as informational — the UX needs to react to `'length'` and `'content-filter'`.

What this lesson does not cover:

- `generateObject` and `streamObject` — Lesson 23.2.2.
- The client-side `useChat` / `useCompletion` hooks and the `UIMessage` `parts` array — Lesson 23.2.3.
- Tool calling and the agentic loop — Chapter 23.3.
- `experimental_telemetry` and OpenTelemetry spans — out of scope here; observability lives in Unit 20.
- Provider-specific options (reasoning tokens, prompt caching, structured tool outputs) — out of scope; the AI Gateway from 23.1.3 absorbs the differences.
- Image / file inputs (`UIMessage` parts of type `file`) — out of scope.

Estimated student time: 40 to 50 minutes. The chapter's load-bearing mechanics lesson.

## Lesson 23.2.2 — Zod schemas as the model contract

`generateObject` and `streamObject` for type-safe structured output, Zod schema design with `.describe` as prompt-shape carrier, the schema constraints the model can render, `enum` and `array` modes, the `maxRetries` cost trade-off, and why structured output is the more provider-swap-friendly call shape.

Topics to cover:

- **The senior question.** When the workload is "map free text to a typed row in our database" — classify an inbound email, extract line items from a description, fill a form from a paragraph — the answer is structured data, not prose. How is that call shaped so the output is type-safe, schema-validated, and the model retries automatically when it returns shape it shouldn't have? The lesson is `generateObject` and `streamObject` and the Zod schema as the contract.
- **Why structured output over `streamText` with JSON prompt-engineering.** A senior on the 2026 stack does not ask a model to "respond in JSON" and parse the string. `generateObject` constrains the model's output to a Zod schema, retries on malformed output, returns a typed `object`, and absorbs provider differences (the same Zod schema works against OpenAI, Anthropic, Google). The prompt-engineered shape is a class of bug: hallucinated keys, extra prose around the JSON, schema drift on model updates. Name the anti-pattern; this is the discipline.
- **The portability win — the cross-link to 23.1.3.** Structured output is the more swap-friendly call shape. Free-form `streamText` is sensitive to prompt-engineering tuned to a specific model; `generateObject` puts the contract in the Zod schema, and the schema is provider-independent. A surface built on `generateObject` swaps providers more cleanly — the senior calls for it whenever the workload allows.
- **The minimal call — inline shape.** `generateObject({ model: smartModel, schema: invoiceLineItemSchema, prompt, maxOutputTokens: 500 })` returns `{ object, usage, finishReason }` where `object` is typed by the Zod schema. The TypeScript win is real: `result.object.amount` is a `number` at the call site, not `any`. Show the schema, the call, the typed access; do not show JSON parsing or post-validation.
- **Schema design — Zod descriptions as prompt-shape carriers.** The Zod schema is not just a runtime guard; the SDK serializes it to JSON Schema and sends it to the model as part of the call. The `.describe()` call on each field is the model's documentation. A field named `dueDate` with `.describe('ISO 8601 date string, the date by which the invoice must be paid')` extracts cleanly; a bare `dueDate: z.string()` produces inconsistent formats. The senior reflex: every non-obvious field gets a `.describe`; the schema reads like a spec. Senior cross-link to Chapter 7.1 (Zod 4 schemas) — the same Zod, with a new audience (the model).
- **Schema shape constraints — what the model can and can't render.** The top level should be an object (`z.object(...)`) — most providers do not accept a bare array or primitive at the root. Inside: strings, numbers, booleans, enums (`z.enum([...])`), nested objects, arrays of objects all work cleanly. Unions are supported but cost the model accuracy; tagged unions with a discriminator field (`z.discriminatedUnion('kind', [...])`) work better than untagged. Avoid `z.unknown`, `z.any`, `z.transform` — the SDK can't serialize them. Avoid recursive schemas — the JSON Schema export blows up. Name the constraints once; the student writes schemas that survive.
- **Refining vs. validating — when to lean on Zod, when to lean on the prompt.** Zod `.refine` runs at validation time on the returned object; the SDK retries the model on validation failure but each retry costs tokens. A schema that's too strict (`.refine(x => x.startsWith('INV-'))` for a model-generated invoice number) thrashes the retry loop and burns budget. The senior cut: hard structural constraints in the schema (types, enums, required fields), soft constraints in the prompt ("invoice numbers follow the pattern `INV-XXXX`"). The schema is the floor; the prompt is the suggestion.
- **`streamObject` and progressive rendering.** When the schema is large (a multi-section summary, a list of extracted line items, a long-form report), `streamObject` streams partial objects as fields populate. The route handler returns `result.toTextStreamResponse()` or pipes via the SDK helpers; the client uses `useObject` (covered in 23.2.3) to render fields as they arrive. The UX win: a four-second generation feels like four progressive updates instead of a four-second spinner. The senior call: `generateObject` for short / single-field outputs, `streamObject` when there are multiple fields the user reads sequentially.
- **`enum` mode and single-value outputs.** For workloads where the answer is one value from a known set (sentiment classification, intent labeling, priority bucket), `generateObject({ output: 'enum', enum: ['low', 'medium', 'high'] })` skips the schema overhead. Same retry behavior, cheaper call, simpler return shape. Name the mode; show the one-liner.
- **`array` mode — when the workload is a list of objects.** `generateObject({ output: 'array', schema: lineItemSchema })` returns `{ object: LineItem[] }`. The same effect can be had with `z.object({ items: z.array(lineItemSchema) })` at the root, but `output: 'array'` is the cleaner idiom for "extract a list." Senior reflex: pick the mode that names the workload.
- **The route-handler shape — same seam as 23.2.1.** `generateObject` lives behind the same `authedRoute` + rate-limit + quota stack as `streamText`. The `onFinish` callback fires with the typed object and `usage`; the audit-event write goes there. For `streamObject`, the handler returns `result.toTextStreamResponse()` and the post-call write lives in `onFinish` the same way. Name the consistency; the student does not learn a different route-handler shape for structured output.
- **The retry knob — `maxRetries`.** When the model returns malformed JSON or output that fails Zod parsing, the SDK retries — the default is two retries. The senior cut: a strict schema with a high retry count is a cost amplifier (each retry pays a full call). Drop `maxRetries` to 1 when the schema is well-described and the model is reliable; raise it only when the workload genuinely benefits. Name the trade; show the knob.
- **Where structured output earns its weight in the course's running app.** The invoice-domain examples from prior chapters — extracting line items from a free-text description, classifying inbound payment emails into status buckets, drafting an invoice description from a brief. Each is a `generateObject` call with a domain Zod schema. The worked surface is 23.4 (Q&A with tool calling); the senior reflex is that even within Q&A, the tool's input and output schemas use this same Zod discipline.
- **A short diagram (mermaid) — the structured-output flow.** User input → route handler → `generateObject({ schema })` → SDK serializes schema to JSON Schema → model → SDK validates with Zod → retry on failure → typed object returned → `onFinish` audit write → response. The diagram makes the schema's role as both prompt-shape and runtime-guard visible.
- **Watch-outs:** asking `streamText` to return JSON and parsing the string — wrong tool, the SDK already solves this; bare Zod fields with no `.describe` — the model guesses the shape; recursive or `z.any` schemas — the JSON Schema export breaks; `.refine` with a hard pattern that the model can't reliably hit — retry-loop cost amplifier; `streamObject` for a small fixed shape — overkill, `generateObject` is the right call; forgetting that the Zod schema is sent to the model — large schemas blow input token cost, trim unused fields; using structured output as a way to "force the model to follow instructions" — the schema constrains shape, not behavior; not handling the rare case where `object` is `null` (the model genuinely couldn't comply) — defensive code around the typed access.

What this lesson does not cover:

- Client-side rendering of partial objects via `useObject` — Lesson 23.2.3.
- Tool-call inputs and outputs (which also use Zod schemas) — Chapter 23.3.1.
- The Zod 4 schema fundamentals (`z.object`, `z.enum`, `.refine`) — Chapter 7.1 (prerequisite).
- Image extraction / vision inputs feeding a structured schema — out of scope.
- Reasoning-token cost when reasoning models are used for structured output — out of scope.
- Caching identical structured-output calls — out of scope; the AI Gateway from 23.1.3 handles it.

Estimated student time: 40 to 50 minutes. The chapter's portability lesson — structured output is the swap-friendly call shape.

## Lesson 23.2.3 — useChat, useObject, and the parts array

The v5 transport-based UI hooks (`useChat`, `useCompletion`, `useObject`), the `UIMessage` `parts` array as rendering source of truth, manually managed input state with plain `useState`, `sendMessage` / `regenerate` / `stop` / `status`, the `convertToModelMessages` / `toUIMessageStreamResponse` handler contract, and the Server-Component-loads-`initialMessages` / Client-Component-mounts-`useChat` boundary.

Topics to cover:

- **The senior question.** The server-side primitives from 23.2.1 and 23.2.2 push tokens out as a stream. How does a Client Component subscribe to that stream, render text deltas and partial objects progressively, manage the input box, and persist messages — under React 19's streaming primitives and v5's transport-based architecture? The lesson is the three UI hooks (`useChat`, `useCompletion`, `useObject`), the `UIMessage` `parts` array as the rendering source of truth, and the route-handler-to-client contract via `toUIMessageStreamResponse` / `convertToModelMessages`.
- **The v5 transport architecture — what changed and why.** v5 replaced v4's monolithic `useChat` with a transport-based architecture: the hook delegates message sending to a `ChatTransport`, defaulting to `DefaultChatTransport` which speaks HTTP POST + SSE against `/api/chat`. The student does not write a custom transport in this lesson — the default is the right call — but the architecture matters because three v4-era assumptions break: input state is no longer auto-managed by the hook, the `messages` array uses `UIMessage` with a `parts` array (not flat `.content`), and the call surface is `sendMessage` / `regenerate` (not `append` / `reload`). Name v5 as canonical; v4 shapes the student may find on the web are wrong by construction.
- **The `UIMessage` `parts` array — the rendering source of truth.** A v5 message is `{ id, role, parts: UIMessagePart[] }`. Each part has a `type` — `'text'` for prose, `'tool-<name>'` for tool invocations (23.3), `'file'` for attachments, `'reasoning'` for chain-of-thought traces, plus user-defined data parts. The render pattern: `message.parts.map((part) => ...)` and switch on `part.type`. A v4 codebase rendering `message.content` as a string is broken — the rich rendering (tool calls, files, reasoning) lives in `parts`. Name the pattern; show a minimal render snippet inline.
- **`useChat` — the minimal call surface.** `const { messages, sendMessage, regenerate, status, error, stop } = useChat({ api: '/api/chat' })`. `messages` is `UIMessage[]`; `sendMessage({ text })` posts to the route handler and appends the response; `regenerate()` re-runs the last assistant turn; `status` is `'submitted' | 'streaming' | 'ready' | 'error'`; `stop()` cancels the in-flight stream. Each rename from v4 (`append` → `sendMessage`, `reload` → `regenerate`) is named once; the student updates their mental model.
- **Manually managed input state — the v5 change with the biggest footprint.** v5's `useChat` does not return `input`, `handleInputChange`, `handleSubmit`. The client component owns input state via plain `useState`; the form's `onSubmit` calls `sendMessage({ text: input })` and clears the state. The senior reflex: this is the same shape as any other React form (Chapter 4.8) — the hook returning input state was always a leaky abstraction. Show the pattern inline; the student writes one `useState<string>('')` and one `<form>` and the conversation works.
- **`status` and `stop` — the streaming UX.** `status === 'streaming'` is the boolean that drives the "stop" button, the input disabled state, the typing indicator. `stop()` is wired to the cancel control; under the hood it aborts the fetch and the route handler's `abortSignal` propagates to the model call (23.2.1's cost discipline). The senior pattern: every chat surface has a visible cancel affordance, and `stop` is wired the same way the input is.
- **`onFinish` on the client — the persistence seam.** `useChat({ onFinish: ({ messages }) => save(messages) })` fires when the stream completes. The senior cut: messages persist as `UIMessage[]` (the source of truth), not as `ModelMessage[]` (lossy). On the next mount the client loads `UIMessage[]` from the database via a Server Component, hydrates `useChat({ initialMessages })`, and the conversation resumes. The route handler still calls `convertToModelMessages` before `streamText`. Name the two-message-types boundary; it's the v5 architectural keystone.
- **The route handler half — `convertToModelMessages` and `toUIMessageStreamResponse`.** The seam back to 23.2.1: the route handler receives `UIMessage[]` from the client, calls `convertToModelMessages(messages)` to produce `ModelMessage[]` for `streamText`, calls `result.toUIMessageStreamResponse()` to return the v5 stream protocol. The two helpers are the contract; deviating from them means hand-writing the stream protocol. Show the four-line handler inline; the rest of the file is the auth/quota wrapper from 23.1.2.
- **`useCompletion` — the simpler hook for single-shot text.** When the surface is one input field and one streaming output (a free-form prose generator, a summary box, an autocomplete drawer), `useCompletion` is the right tool: `const { completion, complete, isLoading, stop } = useCompletion({ api: '/api/complete' })`. No message history, no `parts` array, just text in / text out. The route handler shape is the same as `useChat`'s but returns `result.toTextStreamResponse()`. The senior cut: reach for `useCompletion` when the workload is stateless; reach for `useChat` when there's a conversation.
- **`useObject` — streaming partial objects to the UI.** `const { object, submit, isLoading, stop, error } = useObject({ api: '/api/extract', schema: lineItemSchema })`. `object` is a `Partial<LineItem>` that builds up field-by-field as the stream arrives; the component renders it progressively. The schema is the same Zod object used server-side. Senior cross-link to 23.2.2: when `streamObject` is the right server primitive, `useObject` is the right client primitive — they share the schema.
- **Progressive rendering patterns — text deltas and partial objects.** For `useChat`, the `'text'` parts append characters as they arrive; the component re-renders on each delta and the user sees the answer build. For `useObject`, `object` is a `Partial` that gains keys as they parse; conditional rendering (`object?.title && <h2>{object.title}</h2>`) lights up the UI section by section. The senior pattern is to design the surface so partial output is useful, not jarring — a chat box reads naturally as it streams; a form field that "appears and disappears" as the model thinks does not. Name the UX rule; the student designs around it.
- **Error states — `error`, `status === 'error'`, and the UI affordance.** Every hook surfaces an `error` field. The senior reflex: render a retry control plus a sanitized message ("the model couldn't complete that request"), never the raw error string (provider error messages leak vendor identifiers and stack traces). For provider 429 / 5xx (23.1.2's seventh abuse shape), the surface degrades gracefully with a "try again in a moment" message; for the user's daily quota (23.1.2), the surface shows "you've used your daily limit, resets at midnight UTC." Both messages come from the route handler's HTTP response, not from the model.
- **Server Component for initial load — the prerendered shell.** The chat page is a Server Component (Chapter 5.2) that loads the conversation's `UIMessage[]` from the database and passes them to a Client Component as `initialMessages`. The Client Component renders `useChat({ initialMessages })`. The senior architecture: the static shell is server-rendered, the live stream is client-rendered, the two compose naturally under React 19. Name the boundary; the student structures the route this way by default.
- **A short diagram (mermaid) — the client/server message lifecycle.** Server Component loads `UIMessage[]` from DB → Client Component mounts `useChat({ initialMessages })` → user types → `sendMessage` → POST `/api/chat` → route handler converts to `ModelMessage[]` → `streamText` → stream parts back → client `messages` updates → render walks `parts` → `onFinish` saves `UIMessage[]` back to DB. The diagram makes the two message types and the persistence seam visible.
- **Watch-outs:** rendering `message.content` instead of `message.parts` — the v4 shape, breaks tool calls and rich parts; expecting `input` / `handleInputChange` / `handleSubmit` from `useChat` — gone in v5, use plain `useState`; calling `append` / `reload` — renamed to `sendMessage` / `regenerate`; returning `new Response(stream)` from the handler — breaks the parts protocol, use `toUIMessageStreamResponse`; persisting `ModelMessage[]` instead of `UIMessage[]` — loses tool calls, reasoning, and metadata on the next mount; missing `stop` wiring — abandoned streams keep generating; rendering raw `error.message` to the UI — leaks vendor identifiers and is bad UX; mixing `useChat` and direct fetch calls to the same endpoint — the parts protocol and a custom fetch don't compose; loading `initialMessages` from a Client Component fetch instead of a Server Component prop — adds a roundtrip the architecture doesn't need.

What this lesson does not cover:

- Tool calls rendered as `'tool-<name>'` parts and the agentic loop — Chapter 23.3.1.
- Generative UI with `streamUI` and `ai/rsc` — Chapter 23.3.2.
- Custom transports (WebSocket, durable, in-process) — named once; the default is the right call for this course.
- Multi-modal inputs (file / image parts on `UIMessage`) — out of scope.
- Reasoning-trace rendering for reasoning models — out of scope.
- Optimistic updates and client-side dedupe — out of scope; the SDK handles streaming state.
- The worked surface against the invoice data — Chapter 23.4.

Estimated student time: 45 to 55 minutes. The chapter's client-side mechanics lesson — the surface the user actually sees.
