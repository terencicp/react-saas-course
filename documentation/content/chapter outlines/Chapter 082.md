# Chapter 082 — Zustand (conditional)

## Chapter framing

Zustand is the second of two conditional client-state tools in Unit 16. The course has spent twenty-two chapters pushing read state to Server Components, write state to Server Actions, transient UI state to `useState`/`useReducer`, shareable view state to the URL with `nuqs`, and server-state caches (when the four triggers from lesson 1 of chapter 080 land) to TanStack Query. Zustand is the conditional reach past *those* — the library for **genuinely shared, non-server, non-URL client state** that lives in memory across deeply nested or cross-route trees: multi-step wizard payloads, a cart, global UI flags (command palette open, theme override, sidebar collapsed) that several disjoint subtrees read and write. The chapter's job is to make the threshold explicit so the library doesn't sprawl into the role of "ambient global state for everything," to teach the v5 primitives a senior actually writes (vanilla `createStore`, the slices composition pattern, selector-driven subscriptions), to wire the per-request store provider correctly against the App Router (the same cross-request leak shape as lesson 3 of chapter 080, with a different fix), and to walk the one concrete case in our app where the threshold is met. The chapter ends without a project; chapter 083 turns the wizard case into a buildable surface.

Threads through every lesson: Zustand is **conditional and per-feature**, not a global state bus — every store is named, scoped, and lives next to the feature that owns it (the per-feature scoping rule — each store has a single feature owner and lives next to it, never one global ambient store for the app); the four prior defaults (`useState`, `useReducer` for related local state, lifted state plus context for narrow shared cases, `nuqs` for shareable view state) are named at the top of every conditional discussion so the threshold stays visible; Zustand is **client-only** — never imported into a Server Component, never inside a Server Action, never holding server state that TanStack Query or Server Components already own; selectors are mandatory at the call site — components subscribe to the *slice* they read, never the whole store, to keep re-renders surgical; on App Router, stores are instantiated **per request** via a `useRef`-pinned provider so SSR doesn't leak one tenant's state into the next user's response; sensitive data (billing fields, draft PII) lives in the store *only because* the URL is the wrong place for it — the senior call is named on the wizard surface; refresh loses store state by design — anything that must survive refresh goes to the server (an action draft) or to `sessionStorage` via `persist`, and that's a product decision, not a library limitation.

## Lesson 1 — When Zustand earns its weight

The three triggers that justify reaching for Zustand past the five client-state defaults, the per-feature rule, and the "client-only, never global ambient" framing for SaaS codebases.

Topics to cover:

- **The senior question.** Five client-state shapes have already been covered with stronger defaults — `useState` for transient local UI, `useReducer` for related local state with transitions, lifted state plus React Context for narrowly shared trees, `nuqs` for shareable view state in the URL, and TanStack Query for server state on the four lesson 1 of chapter 080 triggers. What's left? When does a SaaS feature genuinely need an in-memory global client store, and what's the threshold the five defaults stop covering? The lesson answers it explicitly so the rest of the chapter has a justification.
- **The three named triggers — what crosses the threshold.** The course accepts three (and only three) reasons to bring Zustand into a SaaS codebase:
  - **Genuinely shared client state across deeply nested or cross-route components.** A multi-step wizard whose steps live on different route segments (`/new/step-1`, `/new/step-2`, …) and need to share draft payload; a global command palette whose open/close state is read by a topbar, a layout overlay, and a keybinding handler in three different subtrees; a shopping cart read by a header badge, a slide-over panel, and a checkout page. Lifting plus Context works when the consumers all live under one provider in one subtree; once the consumers are in disjoint trees or across route boundaries inside the same layout, Context becomes prop drilling by another name.
  - **Cross-component imperative actions on shared UI state.** "Open the global toast", "trigger the confirmation modal from any leaf", "flip the sidebar from a deeply nested button." A small store with an `open()`/`close()`/`fire()` action surface beats threading a callback through six layers. The senior call: the trigger is real only when more than two unrelated subtrees fire the action.
  - **Frequent reads of the same client state by many components, where Context's whole-tree re-render is the cost.** Context re-renders every consumer when *any* value changes; Zustand's selector-based subscriptions re-render only the components whose selected slice changed. When the shared state mutates often (a cart updating per keystroke, a wizard validating per field) and dozens of consumers read disjoint slices, Zustand's selector model is the right shape. The trigger: profiled re-render cost, not aesthetic preference.
- **The conditional framing — what does NOT clear the bar.** A single page's form state — no, `useState` (or React Hook Form on the chapter 049 threshold). A theme/locale that the whole app reads from one place — no, Context is fine because reads are rare. Filter/sort/search/cursor pagination — no, `nuqs` plus Server Components (Chapter 064). Server state that the client polls or optimistically updates — no, TanStack Query. Auth session — no, `auth()` in a Server Component or the Better Auth client hook in a leaf. The senior reflex: enumerate which default the workload would otherwise use; only if every default is wrong does Zustand earn its weight. State the rule plainly: "if `useState` covers it, ship `useState`."
- **The cost of bringing the library in.** Another mental model for "where does this state live"; a per-feature store file that future readers must locate; an SSR wiring trap (the per-request rule); a `'use client'` boundary at every consumer; a reset discipline (sign-out, org-switch, submit-success) that, if forgotten, leaks state across sessions in the same tab. Each cost shrinks the surface the senior should be willing to apply it on.
- **The "per-feature, never global ambient" principle.** The per-feature scoping rule lands here — an unbound architectural rule (not numbered against the principle list, which already owns #6 for a different rule from lesson 3 of chapter 034) that says every client store has a single feature owner and lives next to the feature that owns it. Each store has a single feature owner: `useWizardStore` for the customer-onboarding wizard, `useCartStore` for the cart, `useCommandPaletteStore` for the palette. There is no `useAppStore` that holds everything; that pattern is the 2016 Redux-store-of-the-universe in fewer lines, and it carries the same coupling cost. The lesson states this rule and gives the file-layout convention (`/lib/<feature>/store.ts`).
- **Comparison vs. the alternatives — one paragraph each.** Redux Toolkit (heavier API surface, reducer/action ceremony that v5 Zustand doesn't need; pick Zustand by default in 2026, name RTK if a codebase already uses it). Jotai (atom-based, fine-grained, picks fight with the slices model — different mental shape; Jotai is right when the state is genuinely many small independent atoms with derivation graphs, which is rare in SaaS UI). Valtio (proxy-based mutation API; nicer ergonomics in spots, smaller ecosystem; the team's other libs are pmndrs, and pmndrs picks Zustand for the canonical 2026 SaaS shape). The course's call: Zustand v5, no detours.
- **The "do we even own this state on the client?" check.** Before reaching for any of this, ask: is this state actually the server's? A draft that must survive refresh is server state (write it as a `draft` row); a list view's filter is URL state; a comment thread is server state (TanStack Query on the chapter 080 trigger). Zustand owns client-only state that is allowed to vanish on refresh by product decision. The mental check keeps the library scoped to its job.
- **Decision tree — the funnel as the gate.** A mermaid diagram showing: "is this server state?" → "would `useState` cover it?" → "would lifting plus Context cover it (consumers under one subtree, low write frequency)?" → "would URL state cover it?" → only if all four fall through does Zustand earn its weight; then sub-funnel "single store per feature, never one global store."
- **The "in our app" pointer.** This chapter's worked case is a four-step customer-onboarding wizard at `/customers/new/step-1` through `step-4` (lesson 3 of chapter 082). The chapter 083 project builds the full surface against it. The lesson names the screen once and lets the later lessons drill in.
- **Watch-outs:** reaching for Zustand because "global state feels cleaner" — that's the Redux-of-the-universe trap restated; using Zustand to cache server data the user can see refresh-stale — that's TanStack Query's job; one giant `useAppStore` with twelve unrelated slices — per-feature stores instead; persisting everything to `localStorage` by default — only persist when product semantics demand it (cart yes, draft sometimes, wizard typically no); selecting the whole store (`useStore((s) => s)`) — every state change re-renders every consumer, defeats the point.

What this lesson does not cover:

- The Zustand primitives themselves — lesson 2 of chapter 082.
- The App Router per-request wiring — lesson 2 of chapter 082.
- The worked screen in our app — lesson 3 of chapter 082.
- TanStack Query and client-side server state — Chapter 080.
- The wizard project build, file by file — Chapter 083.

Estimated student time: 30 to 40 minutes. The chapter's load-bearing decision lesson — every other lesson is conditional on the threshold landing here.

## Lesson 2 — Primitives and the per-request provider

The v5 API surface — `createStore`, the slices pattern with `StateCreator`, selector subscriptions with `useShallow`, the `useRef`-pinned Context provider for App Router SSR, and the middleware lineup named once.

Topics to cover:

- **The senior question.** Once the threshold is crossed, what's the API surface a senior actually writes? And how is the store wired into the App Router without leaking one request's state into the next response? The lesson is a tour of the v5 primitives plus the setup pattern, because Zustand's footprint is small enough that primitives and wiring belong in one place — the project (chapter 083) needs both before it can start.
- **Install.** `pnpm add zustand`. Named once; v5 is the May 2026 baseline.
- **The store — what it is.** A Zustand store is a function-call subscription primitive: it holds a state object, exposes `getState()`, `setState()`, `subscribe()`, and integrates with React via a `useStore` hook that re-renders the calling component only when the selected slice changes. No reducers, no action types, no dispatcher. The state and the actions that mutate it live in one closure.
- **`create` vs. `createStore` — the v5 split that matters for SSR.** `create` (from `zustand`) returns a React-bound hook ready to use directly: `const useCartStore = create<CartState>()((set) => ({ ... }))`. `createStore` (from `zustand/vanilla`) returns a vanilla store you wrap yourself with React Context and the generic `useStore` hook from `zustand`. The senior reach in App Router code: **always `createStore` plus a provider**, because the React-bound `create` makes the store module-scoped, which is the per-request leak. Show both, name the rule. `create` is fine in a non-SSR React app; this course's stack is SSR by default.
- **The `set`/`get` API.** Actions live inside the store creator and call `set` to update state, `get` to read current state when an action needs to derive from existing values. `set((state) => ({ count: state.count + 1 }))` for functional updates, `set({ count: 5 })` for absolute writes; `set(partial, true)` (the rare replace-mode flag) wipes the whole state and is almost never what you want. `get` is the inside-the-store read used in actions like `addItem: (item) => { if (get().items.find(...)) return; set(...); }`. Show the canonical action-shape pattern: each action is a method on the state object, colocated with the values it touches.
- **The slices pattern — composing multiple feature areas of one store.** A slice is a factory function `createWizardContactSlice: StateCreator<WizardState, [], [], ContactSlice> = (set, get) => ({ contact: {...}, setContactField: (k, v) => set(...) })`. The full store composes slices: `createStore<WizardState>()((...a) => ({ ...createContactSlice(...a), ...createBillingSlice(...a), ...createPreferencesSlice(...a) }))`. The senior win: each slice is a self-contained file (`/lib/wizard/contact-slice.ts`), the store assembly is a thin composition file (`/lib/wizard/store.ts`), and the type surface stays narrow per slice. The lesson shows the `StateCreator` generic shape because that's the load-bearing TypeScript move — every slice tutorial that skips it ends with `any` later.
- **Selectors — the subscription contract.** Components call `useWizardStore((state) => state.contact.email)` to subscribe only to that field. State changes elsewhere in the store don't re-render the component. Reaching for the whole state (`useWizardStore((state) => state)`) defeats the model. The senior pattern: a `/selectors.ts` file alongside the store exporting named selectors (`selectContactEmail = (s: WizardState) => s.contact.email`) so call sites stay terse and the selector logic is reused and testable. For derived values that combine multiple slices, the selector is a plain function; React re-runs it on every state change but the re-render only fires if the *returned value* changes (referential equality by default; use `useShallow` from `zustand/react/shallow` for object/array returns).
- **`useShallow` and the equality function — when default equality bites.** Default selector equality is `Object.is`. Returning a new object/array each call (`useWizardStore((s) => ({ a: s.a, b: s.b }))`) re-renders every state change because the literal is a new reference. Two fixes, name both: pick atomic values with separate selector calls (`const a = useStore(s => s.a); const b = useStore(s => s.b);`), or wrap in `useShallow`: `useWizardStore(useShallow((s) => ({ a: s.a, b: s.b })))`. The senior default: separate atomic selectors when there are two or three fields; `useShallow` when the selection is a list mapped from a slice.
- **Action access — pulling the function out of the store.** Actions are stable references inside the store (defined once in the creator); selecting them is just `const setContactField = useWizardStore((s) => s.setContactField)`. No subscription cost because the function reference doesn't change. The lesson establishes the read/write split at the call site: components select the read slices they actually render and the actions they actually call, nothing more.
- **The per-request provider — the load-bearing App Router pattern.** Module-scoped `create((...))` in `/lib/wizard/store.ts` lives in the server bundle's module memory: every request reads and writes the same instance, so one user's wizard payload becomes another user's initial state. The fix: a `WizardStoreProvider` Client Component that creates a new store with `useRef`:
  - `const storeRef = useRef<WizardStore>(null); if (!storeRef.current) { storeRef.current = createWizardStore(initialState); }`
  - `return <WizardStoreContext value={storeRef.current}>{children}</WizardStoreContext>;`
  - A `useWizardStore<T>(selector: (s: WizardState) => T)` hook reads the store from context and calls `useStore(store, selector)` from `zustand`.
  - Files: `/lib/wizard/store.ts` (the `createWizardStore` factory using `createStore`), `/lib/wizard/store-provider.tsx` (`'use client'`, Context + `useRef`), `/lib/wizard/use-wizard-store.ts` (the typed hook). Show the three-file shape; it's the canonical Zustand-on-App-Router pattern.
- **Why `useRef` not `useState`.** `useState` would create a new store on every render before commit; `useRef` creates exactly once per component instance. The pattern is in the official Zustand Next.js setup guide for a reason; name it.
- **The reset action — submit-success and tenancy boundaries.** Every feature store exposes a `reset()` action that puts state back to initial. Call it after submit-success, on sign-out, and on org-switch (Chapter 060). The store doesn't auto-reset on navigation because that defeats the whole purpose of cross-route persistence; *the developer* resets when product semantics demand it. Forgetting the reset is the senior watch-out the project's verify recipe catches.
- **`persist` middleware — name it once, don't reach for it by default.** `persist((set) => ({ ... }), { name: 'cart-v1', storage: createJSONStorage(() => sessionStorage) })` survives refresh. The senior call: `sessionStorage` over `localStorage` for ephemeral session data (the cart, the wizard if product wants refresh-survives semantics); never persist server state, auth tokens, or anything an org-switch invalidates. Hydration mismatch is the trap (server has empty state, client has persisted state); the `onRehydrateStorage` callback or a `hasHydrated` boolean gates the first render. The course's wizard project does **not** persist — refresh-loses is the explicit product call (lesson lesson 3 of chapter 082 names why) — but the middleware lives in the surface and gets named.
- **`subscribeWithSelector` middleware — when imperative subscriptions matter.** For non-React listeners (a `useEffect` that fires a side effect when a specific slice changes, or an analytics subscriber outside React), `subscribeWithSelector` lets `store.subscribe((state) => state.slice, (val, prev) => {...})`. Name it once for the analytics-on-cart-change use case; the wizard project doesn't need it.
- **Devtools middleware — optional, dev-only.** `devtools` wraps the creator and exposes the store to the Redux DevTools browser extension. Worth mentioning once for debugability; gate on `NODE_ENV` so it doesn't ship to production.
- **TypeScript discipline.** The `StateCreator<Store, Mws, Mws, Slice>` generic is verbose but worth typing once per slice. Name the type-flow rule: state shape is one source-of-truth `WizardState` type composed from slice types (`ContactSlice & BillingSlice & PreferencesSlice & MetaSlice`); each slice's `StateCreator` is parameterized so `set`/`get` see the full store, not just the slice. The lesson shows the four-line type setup; the project chapter applies it.
- **Worked example — the wizard store skeleton.** Five files in compact form: `wizard-types.ts` (the four slice types), `contact-slice.ts` (one slice with three fields and a setter action), `store.ts` (`createWizardStore` factory composing all slices plus `reset`), `store-provider.tsx` (the `useRef`+context Client Component), `use-wizard-store.ts` (the typed hook). The example is the project's center of gravity in skeleton form.
- **Watch-outs:** `create` instead of `createStore` in an App Router codebase — the cross-request leak the per-request rule prevents; module-scoped `const wizardStore = createStore(...)` outside the provider — same leak; selecting the whole store (`useWizardStore((s) => s)`) — every change re-renders every consumer; new-literal returns from selectors without `useShallow` — re-render storm; missing `'use client'` on the provider — the canonical "createContext is not a function" build error; one giant store with everything in it — per-feature stores instead; persisting an org-scoped store without clearing on org-switch — wrong-tenant data after switching; running actions inside the store creator's top-level body (rather than inside an action function) — they fire at construction, not on demand; calling `setState` inside a render — infinite render loop; expecting state to survive refresh without `persist` — it doesn't, that's the design.

What this lesson does not cover:

- The threshold decision — lesson 1 of chapter 082.
- The worked screen end-to-end — lesson 3 of chapter 082 and Chapter 083.
- React Server Components reading from a Zustand store — that's not a thing; Zustand is Client-only by definition.
- Cross-tab synchronization via `BroadcastChannel` or storage events — out of scope.
- The `combine` and `redux` middlewares — named in passing, the slices pattern covers the composition need.

Estimated student time: 60 to 75 minutes. The chapter's heaviest mechanics-plus-wiring lesson — every primitive and the per-request trap land here.

## Lesson 3 — The routed wizard, end to end

The four-step customer-onboarding wizard as the in-app trigger — the four-slice store shape, per-step Zod gates, the Server-Action submit boundary, back/forward preserves vs. refresh loses, and reset discipline at submit-success and org-switch.

Topics to cover:

- **The senior question.** With the threshold and the primitives in place, what screen in our SaaS actually clears the bar? The lesson names the surface, walks the three-trigger check against it, and frames the project (chapter 083) that builds it.
- **The screen — the four-step "new customer" wizard.** At `/customers/new/step-1` through `/customers/new/step-4`, a wizard for creating a customer: step 1 contact (name, email, phone), step 2 billing (address, tax ID, payment terms), step 3 preferences (notification channels, default currency, language), step 4 review (read-only summary, final submit). Each step is its own route segment under a shared layout; the layout hosts the `<WizardStoreProvider>` so the store survives across the four route changes. The product motivation: onboarding a customer is the SaaS standard high-value form; breaking it into four steps reduces abandonment vs. one tall form and makes validation feedback per-section.
- **The three-trigger check, applied.** Run the lesson 1 of chapter 082 funnel against the wizard:
  - **Genuinely shared state across cross-route components.** Each step lives on its own route segment; step 4 reads the data from steps 1–3 for the review. Lifting state to a common parent fails because routing replaces the page-level Client Components on each navigation; the parent layout is one level up and persists across the four pages, but threading state through it as props requires every step to be a Client Component receiving every other step's data — that's prop drilling. Trigger met.
  - **Action surface across disjoint components.** The header shows a progress indicator that reads `currentStep`; the footer "Next" button reads the current slice's validity and calls `goNext()`; the review step reads all slices and calls `submit()`. Three disjoint subtrees in the layout all need access. Trigger met.
  - **Cost of Context's whole-tree re-render.** Every keystroke in step 2's address field would re-render the header, the footer, and any other consumer if Context held the state. Selector-based subscriptions keep step 2's input local. Trigger met — though weakest of the three for a four-step wizard; the strong case is the first two.
- **Why not URL state.** URL state is the default for shareable view state (filters, sort, cursor), but billing data and PII don't belong in the URL: they leak to server logs, browser history, analytics referrers, screenshot tools, and copy-paste support tickets. Trying to encode the entire wizard payload as URL parameters also pushes URL length and shape past what's reasonable. The senior call: URL is wrong for sensitive draft data; the in-memory store is right.
- **Why not server state.** The wizard is *draft* — the customer doesn't exist until step 4 submits. Persisting partial drafts to the database before submit means writing a `customer_drafts` table, garbage-collecting abandoned drafts, surfacing them on return, handling tenancy on the draft rows. That's product scope, not stack scope. The course's call: refresh loses the wizard, by explicit product decision; the trade is named on the screen.
- **Why not just `useState` plus lifted state.** The four steps are on four routes. Lifting requires a single Client Component owning the state and rendering all four step UIs — fine for a modal wizard, wrong for a routed wizard where step 1 should be shareable URL `/customers/new/step-1` and back/forward should work. The senior call: routed steps lock in Zustand-or-equivalent.
- **The store shape — four slices.** `ContactSlice` (firstName, lastName, email, phone, plus per-field setters and a slice-level validity computed via Zod), `BillingSlice` (address fields, taxId, paymentTerms, with the same shape), `PreferencesSlice` (channels array, currency, language), `MetaSlice` (currentStep, completedSteps, the cross-cutting `goNext()`/`goBack()` actions plus the global `reset()`). Four slice files, one store composition file, one provider, one hook. The slices pattern from lesson 2 of chapter 082 lands on a concrete shape.
- **The Zod-per-step contract.** Each step's slice has a sibling `contact-schema.ts` exporting `contactSchema: z.ZodObject<...>`. The slice exposes a `validate()` action that runs `contactSchema.safeParse(this.contact)` and returns the result; the "Next" button is gated on `validate().success`. The senior pattern: same schema parses the slice client-side at the gate **and** the submit action server-side on step 4 — the contract from Chapter 046.
- **The submit boundary — Server Action, not store action.** Step 4's submit calls `createCustomerAction(fullPayload)` (a Server Action) that re-parses the entire composite schema, performs the DB insert, and on success returns the canonical `Result`. The store does **not** submit directly — it doesn't talk to the database. On success, the client awaits the action's promise, then calls `wizardStore.reset()` and `router.push('/customers/[newId]')`. The senior reflex: store owns the draft, action owns the persistence, redirect closes the loop.
- **Back/forward navigation — what works, what doesn't.** Browser back/forward triggers App Router navigation between the four route segments. Because the store provider lives on the shared layout and the store is held in a `useRef`, the store instance survives those navigations; all slice state is preserved. The user can fill step 1, advance to step 2, hit back, and find step 1 intact. The senior win: zero ceremony to make this work, *if* the provider sits on the shared layout (not on each step page — the canonical mistake).
- **What loses state — refresh and the back-button-to-pre-wizard exit.** Hard refresh kills the store (no `persist` middleware). Navigating to a non-wizard route (clicking the logo, hitting back past `/customers/new/step-1`) unmounts the provider and discards the store. Both are explicit product calls: refresh-loses keeps the surface honest; exit-discards prevents stale drafts from haunting future visits. The lesson names both and explains why this is the senior call: drafts that survive refresh need server persistence with all the garbage-collection cost.
- **Reset discipline at the boundaries.** Three resets, named explicitly: after submit-success on step 4 (so a quick "create another" doesn't show the previous customer's data); on sign-out (handled at the provider unmount, but call the reset for safety); on org-switch (Chapter 060 — wrong tenant's draft is a data-isolation bug at the client cache layer the same way it would be at the data layer). Same shape as the TanStack Query `queryClient.clear()` discipline from lesson 3 of chapter 080, applied to the Zustand surface.
- **The Next-gate validation pattern.** The "Next" button on each step is `<Button disabled={!isStepValid} onClick={goNext}>`. `isStepValid` is a selector running `validate()` on the current slice; `goNext` advances `currentStep` and `router.push`-es to the next route. The validation runs on every keystroke (via the selector subscription) so the button enables the moment the slice is valid; show the error messages from `safeParse` under each field. The senior call: client-side gate is UX, server-side parse on submit is correctness; both fire.
- **Anti-pattern callouts — what NOT to do on this screen.** Don't put each step's `<WizardStoreProvider>` on the page itself — only on the shared layout, or the store resets every navigation. Don't reach for `persist` "just in case" — refresh-loses is the call. Don't store the new customer's ID in the wizard store after submit — that's server state, redirect to the detail page instead. Don't drop the per-step Zod validation in favor of "we'll validate on submit" — abandonment on step 4 with errors discovered there is the worst UX. Don't query the store from a Server Component — there isn't one inside the wizard tree, but the watch-out is worth naming because a senior reviewer flags any attempt.
- **The forward pointer to chapter 083.** The project chapter builds this surface end to end: the four route segments, the per-step Zod schemas, the wizard store with four slices, the provider on the shared layout, each step's form wired to the store with the Next-gate, the submit action, the success-reset, and the verify recipe (back/forward preserves, refresh loses by design, no double-submit on step 4). This lesson is the framing; chapter 083 is the build.
- **Watch-outs:** the provider on each step page instead of the shared layout — store resets every navigation, the canonical mistake; submitting from a `useEffect` instead of a button handler — double-submit risk on re-render; reading `currentStep` from the URL instead of the store — they should match but the URL is the source of truth for routing while the store is the source of truth for which step is *valid* to reach (no skipping to step 3 if step 2 is invalid); persisting the wizard to `sessionStorage` without telling the user — surprise data hanging around after they abandoned, especially on shared computers; clearing the store with `set({})` instead of `set(initialState, true)` — partial state shapes break selectors that assume slice presence.

What this lesson does not cover:

- The actual build steps (file by file) — Chapter 083.
- The starter walkthrough and verify recipe — Chapter 083.
- Customer-draft persistence to the server — explicitly out of scope, the product call.
- Wizard-as-modal (single-route) — different shape, the routed wizard is the case in our app.
- Step-by-step animations and transitions — UX polish, out of scope.

## Lesson 4 — Quizz

Top 10 topics to quiz:

- The three named triggers that make Zustand earn its weight (genuinely shared state across cross-route or disjoint subtrees, cross-component imperative action surface, selector-vs-Context re-render cost on frequently-mutated shared state); the senior reflex of running the funnel before reaching for the library.
- What does NOT cross the threshold — single-form state, narrowly shared state under one subtree, URL-shaped view state, server state — and which default (`useState`, lifted state + Context, `nuqs`, TanStack Query / Server Components) handles each.
- The per-feature, never-global rule (the per-feature scoping rule — each Zustand store has a single feature owner and lives next to it); why one `useAppStore` with twelve unrelated slices is the wrong shape.
- `create` vs. `createStore` — when each is right; why App Router SSR forces `createStore` plus a provider.
- The slices pattern — `StateCreator` typing, composing multiple slices in one store, per-feature slice files.
- Selectors as the subscription contract — atomic-selector default, `useShallow` for object/array returns, why selecting the whole store defeats the model.
