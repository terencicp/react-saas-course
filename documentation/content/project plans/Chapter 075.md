# Chapter 075 — Project plan: Upstash rate limits on the auth surface

## Design decision (resolved — read first)

The aligner flagged a conflict: the chapter-075 outline's "Done when" clauses, inspector spec, and Moment-of-truth sections describe the gated **Server Action** surfaces as returning an HTTP **429 with `RateLimit-*` headers and a JSON 429 body**. The chapter-074 teaching lessons (lesson 3, confirmed in its Continuity notes) establish the opposite and the student built it that way: **a Next.js Server Action cannot set arbitrary response headers** — `headers()` from `next/headers` is read-only — so a gated action carries its rate-limit budget **inside its `Result` payload** (`rateLimitBudget(ipLimit)` on the `ok` branch) and rejects with `rateLimited(...) → err('rate_limited', <opaque message>)` on the `useActionState` channel. Literal `RateLimit-*` HTTP headers + a JSON 429 body exist **only on a route-handler twin** (`rateLimitedResponse` / `rateLimitHeaders`), used for webhooks/public APIs, never Server Actions; and this project's `/api/auth/[...all]` catch-all is **left unwrapped** (limits land at the action seam that calls `auth.api.*` directly).

**Decision: option (a).** The inspector reads the budget **off the action `Result`** — the success-path `ok` payload's `rateLimit` field, and the `err('rate_limited', …)` code on rejection — matching exactly what the lessons taught. **No real `RateLimit-*` HTTP headers on the action path; the catch-all stays unwrapped.** Option (b) (routing the action surfaces through a real 429 route handler) is rejected: it would contradict the lesson-3 architecture ("limits land at the action seam") and re-teach a shape the student never built. The `rateLimitHeaders` + `rateLimitedResponse` route-handler twin still **ships in `lib/rate-limit-headers.ts`** (taught "for parity", statically checked for shape) and is exercised by **one read-only demo route** (`/api/limit-demo`) so the student can *see* real headers exist somewhere — but **no auth action routes through it**.

Consequence threaded through this whole plan: every place the outline says "11th response is a 429 with `Retry-After` and a JSON body" becomes "**11th `signInAction` call returns `err('rate_limited', 'Too many attempts. Please try again later.')`** while the inspector's per-key readout (`getRemaining`) shows `0/10`." The inspector's recent-responses log shows an **outcome** — the `Result`'s real code (`ok` / `rate_limited` / `unauthorized` / `validation` / `internal`) — + the budget fields off the `Result`, **not** an HTTP status + headers. The opaque message is the user-safe contract; the operator-honest detail (which gate, the key) lands in the `rate_limit_log` table read by the structured-log tail.

## Project goals

Cement chapter 074 by layering three `@upstash/ratelimit` limiters onto the chapter-055 email+password auth surface at the **Server Action seam**, then watching every gate fire on a provided `/inspector`. Coding develops five durable, "make-the-wrong-shape-impossible" skills. (1) **The limiter is a named seam** — `lib/rate-limit.ts` is the one and only place `new Ratelimit(...)` appears; call sites import `signInLimiter` / `signUpLimiter` / `resetLimiter`, never construct inline, because an in-handler instance defeats the in-memory `ephemeralCache` and corrupts the prefix namespace. (2) **Gate before work, dual-keyed where it matters** — sign-in runs two ordered `safeLimit` calls (`ip:` then `email:`, cheaper first) **before** `auth.api.signInEmail`, both must pass; sign-up is per-IP only (the email is the attacker's choice); reset is per-IP + per-email (the per-email gate protects a third party's inbox and Resend cost). (3) **The budget rides on the `Result`, not on HTTP headers** — the reject path returns `rateLimited(...) → err('rate_limited', <opaque message>)`; the success path returns `ok({ redirectTo, rateLimit: rateLimitBudget(ipLimit) })`, because `headers()` is read-only in a Server Action. (4) **One place owns the fail-open policy** — `safeLimit` wraps every `limit()` call: a Redis outage logs `rate_limit_unavailable` and returns `{ success: true, … }` so the auth path stays up, and flipping to fail-closed is a one-line change there. (5) **The architectural swap** — Better Auth's built-in in-memory limiter goes off (`rateLimit: { enabled: false }`), making the application-level limiters the single enforcement point. The student practices each by filling six stub files against a brief and reading every outcome off the provided `/inspector` (remaining-token panel, recent-responses log, structured-log tail, failure-mode toggles).

The build is deliberately one surface — three limiters, three action wraps, one inspector — so the student completes it quickly and the structural lesson ("the action wrapper is the single enforcement point; `safeLimit` is the fail-open knob; the budget rides the `Result`") lands without app-feature noise. The point is not a realistic app; it is a realistic workflow walked end-to-end.

## Locked decisions

- **The budget rides the action `Result`; no `RateLimit-*` HTTP headers on the action path; the catch-all stays unwrapped (option (a) — the chapter-074-taught shape).** Server Actions cannot set HTTP headers (`headers()` is read-only). The reject path is `rateLimited(r, gate, key) → err('rate_limited', 'Too many attempts. Please try again later.')`; the sign-in/sign-up success path returns `ok({ redirectTo, rateLimit: rateLimitBudget(ipLimit) })`; reset returns `ok({ sent: true })`. Literal `RateLimit-*` headers + a JSON 429 body exist only on the route-handler twin (`rateLimitHeaders` / `rateLimitedResponse`), exercised by `/api/limit-demo` for parity. `/api/auth/[...all]` is **never** wrapped — limits land at the action seam that calls `auth.api.*`. The inspector reads outcomes + budget off the `Result`, never HTTP status/headers.
- **`lib/rate-limit.ts` is the only place `new Ratelimit(...)` appears.** Call sites import `signInLimiter` / `signUpLimiter` / `resetLimiter`. An in-handler instance defeats `ephemeralCache` and corrupts the prefix namespace — a static grep for `new Ratelimit` outside this file is a regression.
- **Module-scope declaration is load-bearing.** `redis` and all three `Ratelimit` instances are declared once at module scope; each limiter has its **own** `ephemeralCache: new Map()` and a distinct prefix (`rl:signin`, `rl:signup`, `rl:reset`); all carry `analytics: true`.
- **Budgets (senior calls, carried from chapter 074):** sign-in `slidingWindow(10, '1 m')` (legitimate typos tolerated), sign-up `slidingWindow(5, '10 m')`, reset `slidingWindow(3, '15 m')` (tightest — concrete third-party abuse cost). `LIMITER_MAX = { signin: 10, signup: 5, reset: 3 }` is the locally-declared cap the inspector readout pairs with `getRemaining(key).remaining` (the `2.0.8` `getRemaining` also returns a `limit` field — the *effective*/dynamic limit, unused here; the explicit `LIMITER_MAX` keeps the static cap at the call site).
- **`getRemaining`, not `limit`, for readouts.** The inspector's "Remaining tokens" panel calls `limiter.getRemaining(key)` — it consumes no budget. Using `limit()` for a readout would burn a token per render and lock the user out via the panel itself.
- **Gate before work; dual-keyed where it matters; cheaper first.** Sign-in: `ip:` then `email:`, both `safeLimit`, both before `auth.api.signInEmail`, early-return on each `!success`. Sign-up: per-IP only (email is attacker-chosen). Reset: `ip:` then `email:` (per-email survives an IP switch). The order is the lint-by-eye rule: parse → resolve ip+email → safeLimit gate(s) → `auth.api.*` → return.
- **`safeLimit` is the one place the fail-open policy lives.** `safeLimit(limiter, prefix, key)`: `try { return await limiter.limit(key) } catch { await logRateLimit({ event: 'rate_limit_unavailable', limiter: prefix, key }); return { success: true, … } }`. The limiter's `prefix` is `protected readonly` in `@upstash/ratelimit@2.0.8` (reading `limiter.prefix` externally is `TS2445`), so the prefix literal is the wrapper's third argument — call sites pass `'rl:signin'` / `'rl:signup'` / `'rl:reset'` alongside their limiter. Flipping to fail-closed is changing the returned `success` here, once. Every `limit()` call on the auth path goes through `safeLimit`; only `/api/limit-demo` calls `limit()` bare (the headers demo).
- **The 429 body / opaque message is identical regardless of which gate tripped.** `rateLimited` returns the same `err('rate_limited', 'Too many attempts. Please try again later.')` for `ip` and `email`; `rateLimitedResponse` returns the same JSON. The gate + key land only in `rate_limit_log` (operator-honest), never in the user-facing message — no information leak.
- **`reset` is delta-seconds in headers/budget, milliseconds from the library.** `limit().reset` is a Unix ms timestamp; `rateLimitBudget` and `rateLimitHeaders` convert via `Math.ceil((reset - Date.now()) / 1000)`. Raw ms is a named bug.
- **`pending` analytics flush via `after()`, never `await` on the path.** Each `safeLimit` result's `pending` is handed to `after(...)` (from `next/server`) on both the reject-via-catch and success paths. Awaiting on the path adds 5–10ms (the inspector's "Await pending" toggle proves it).
- **Better Auth's built-in limiter is off.** `rateLimit: { enabled: false }` is the only `rateLimit` entry in `lib/auth.ts`; the application-level limiters are the single enforcement point. `secondaryStorage` (the framework-owns-it alternative) is named-not-built.
- **`getClientIp` trust boundary + `normalizeEmail` rule.** `getClientIp` reads `x-forwarded-for` (first entry), falls back to `x-real-ip`, then `'unknown'` (loose fallback deliberate; strict rejection is chapter 081). `normalizeEmail` is `trim().toLowerCase()` only (no `+`-strip); the **same** normalization runs at the limiter key and the DB lookup, achieved by the action schema's `z.string().trim().toLowerCase().pipe(z.email())` so `parsed.data.email` is key-ready (the action does not call `normalizeEmail` manually for the schema-validated email).
- **The action success path returns `redirectTo`; the form navigates.** The chapter-055 `redirect()`-on-success is replaced by `ok({ redirectTo, rateLimit })` (sign-in/sign-up) because a redirect-throw cannot carry the budget and the inspector must call the action without a redirect throw. The provided `sign-in-form.tsx` / `sign-up-form.tsx` read `state?.ok === true` and `router.push(state.data.redirectTo)` in a `useEffect` (one effect, deps `[state]`; `useRouter` from `next/navigation`). On `state?.ok === false` they render the existing error `Card` (the opaque rate-limit message lands there on `code === 'rate_limited'`). Reset's form renders an enumeration-uniform confirmation on `ok`, no navigation.
- **`Result<T>` carries real success payloads here.** The carried 7-code `ErrorCode` union is **unchanged** (`rate_limited` is already in it); only the `data` type parameter is used: `signInAction`/`signUpAction` → `Result<{ redirectTo: string; rateLimit: RateLimitBudget }>`, `resetAction` → `Result<{ sent: true }>`. `RateLimitBudget = { limit: number; remaining: number; reset: number }` is plain-number serializable (crosses the `useActionState` RSC boundary cleanly). No new `Result` codes; no two-type-param `Result` (the project uses the carried single-param `Result<T>`).
- **`rate_limit_log` is the operator-honest surface; pino is named-not-built.** `logRateLimit` writes `{ event, limiter, key, remaining, reset, firedAt }` rows; the structured-log tail reads them. The real pino + AsyncLocalStorage logger is chapter 092 (a comment names it). Log rows key on `id`; the responses-log keys on an inspector `seq` counter — never an array index (Biome `noArrayIndexKey` under `biome ci`).
