# Chapter 047 — Project plan: CRUD via Server Actions on the invoicing data layer

Sixth project; **forks the Chapter 041 `solution/`** (the org-scoped invoicing data layer — schema, relations, seed, the two read helpers, the postgres-js client, env, config, toolchain) and ships a full CRUD surface on it: a "new invoice" form, an "edit invoice" form, and a delete-with-confirmation button. Every mutation flows through one of three Server Actions, each parsing `FormData` with a `drizzle-zod`-derived Zod schema, each returning the canonical `Result`, each `revalidatePath`-ing the list. The create form layers `useOptimistic`; the delete runs in a transaction; field errors render inline from `Result.error.fieldErrors`; the whole surface works with JavaScript disabled. The standards bar: `safeParse` (never `parse`) opens every action, the tenant filter lives in the `where` (never a post-load check), the schema is the single contract both the action and the form `name`s obey, and the no-JS path still creates/edits/deletes — all checkable against a real render of `/invoices`, not just a green build.

**Inherited-toolchain reality (carried from Ch 041, overrides the chapter outline).** Pinned to `drizzle-orm@0.45.x` + `drizzle-kit@0.31.x`: **Relations v1** (`relations(<table>, ({ one, many }) => ({...}))`, callback `where`/`orderBy`), the **postgres-js** driver via `drizzle-orm/postgres-js` (local Docker Postgres 18), `casing: 'snake_case'` on the client, `drizzle-zod@^0.8` as a **separate package** (`createInsertSchema` imports from `'drizzle-zod'`, not `'drizzle-orm/zod'`). Do NOT normalize any of this to Drizzle 1.0 shapes.

**Two deliberate divergences from the chapter outline's reference build, both grounded in the Ch 044 continuity findings:**
1. **`status` and `customerId` use shadcn `native-select` (a real `<select name>`), not the Radix `<Select>`.** The Radix `<Select>` emits no native `FormData` entry without JS (it needs a hidden-input mirror), which would silently break the no-JS create/edit path this chapter is built to demonstrate. `native-select` posts through `<form action>` with zero JS. (The outline's `<Select>` predates this PE call.)
2. **The hand-rolled field cluster wraps in a plain `<div className="grid gap-2">`, and the project installs NO shadcn `form.tsx`.** Per the Ch 044 L6 finding, every shadcn form primitive except `<FormItem>` calls `useFormField()` and throws outside a React Hook Form `<FormField>`; `<FormItem>` itself is only a `grid gap-2` spacing wrapper. Pulling in `form.tsx` just to use one spacing div is over-engineering and a footgun. Use the plain div.

## Project goals

The project cashes in Unit 6 by shipping the canonical SaaS mutation surface every later unit extends. The student practices five durable skills at once, each cemented by writing real code that a live `/invoices` render confirms: **deriving a mutation schema from the Drizzle table** with `createInsertSchema` + refinement, then treating that one schema as the contract both the action's `safeParse` and the form's input `name`s obey (drift impossible by construction); **writing Server Actions in the five-seam shape** — parse → authorize → mutate → revalidate → return — that hand back a `Result<T>` instead of throwing, with the tenant guard in the `where` and the unique-violation mapped to a `conflict`; **building native React 19 forms** with `useActionState`, uncontrolled `defaultValue` inputs, the reusable `<SubmitButton>`/`<FieldError>`, Constraint Validation attributes that mirror the schema, and the JS-disabled path intact; **layering `useOptimistic`** with a client-generated UUIDv7 so the new row appears instantly and rolls back implicitly when the transition fails; and **reaching for a Drizzle transaction** when a delete is multi-step, keeping external calls and `revalidatePath` outside it. The coding is sliced so each step adds one surface a student can exercise against a seeded database: create → edit → delete → optimistic create → transactional delete. The data layer is inherited whole; the student writes only the schemas, the three actions, and the form components.

`createInvoiceInputSchema` — `createInsertSchema(invoices, { number: (s) => s.min(1).max(50), total: (s) => s.regex(/^\d+(\.\d{1,2})?$/, 'Enter a valid amount (max 2 decimals)').refine((v) => Number(v) >= 0, 'Total must be non-negative') }).omit({ organizationId: true, createdBy: true, createdAt: true })`. **Code conventions (taught syntax — from `Code conventions.md` and the Ch 042–044 lessons):**
- **`safeParse`, never `parse`, at the action boundary** — a throw trips `error.tsx`, the wrong destination for a bad field. `Object.fromEntries(formData)` then `safeParse` is the literal first line; branch on `parsed.success`; read everything downstream from `parsed.data`, never `formData.get('x')` field-by-field in the body.
- **The schema is the single contract** — the action's `safeParse` schema and the form's input `name`s are the same set; derive `updateInvoiceInputSchema` from `createInvoiceInputSchema` (`.extend`), never copy-and-edit. `createInsertSchema(table, overrideMap).omit({...})` for the create base; callback override form `{ col: (s) => s.refine(...) }` preserves nullability.
- **Field errors via `z.flattenError(parsed.error).fieldErrors`** (flat `Record<string, string[]>`), read on the form as `state.error.fieldErrors?.<field>?.[0]` — NOT `treeifyError`/`.properties` (the projection Ch 043 rejected for this channel).
- **`Result<T>`** is canonical at `lib/result.ts`, imported by every action — no per-action `{ success, error }` variant. `ok`/`err` are positional (`err(code, userMessage, fieldErrors?)`); `ErrorCode` is the seven-member set. `redirect()` and `revalidatePath()` ride the framework, return nothing; `redirect()` is **last and outside any try/catch**.
- **Tenant filter in the `where`**, AND-combined (`and(eq(invoices.id, id), eq(invoices.organizationId, organizationId))`), never a post-load check (the IDOR failure mode). `organizationId`/`userId` come from `getActiveContext()` (read **after** the parse), never from the client or `cookies()`.
- **Transaction discipline (L6):** `db.transaction(async (tx) => ...)`; every query inside uses `tx`, not `db`; the callback returns a discriminated value for "not found" (throw only for real rollback); `revalidatePath`/`redirect`/external calls live **outside** the callback. The transaction is the mutate seam; error mapping wraps it from outside.
- **Forms:** uncontrolled inputs with `name` + `defaultValue` (never `value`); `<form action={formAction}>`; `useActionState(action, null)` at the form root with the `(prevState, formData)` action signature; the bound `formAction` on the form (never the raw action); `useFormStatus()` (from `react-dom`) read by the descendant `<SubmitButton>`; Constraint Validation attributes mirror the schema (`required`/`type`/`step`/`min`/`inputMode`/`autoComplete`); `:user-invalid` (not `:invalid`) for error styling via Tailwind variants; `aria-invalid`/`aria-describedby` with stable ids on every errorable field.
