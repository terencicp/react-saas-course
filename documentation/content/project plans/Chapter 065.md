# Chapter 065 — Project plan: From Stripe webhook to plan entitlement

## Project goals

The student wires the production async edge every SaaS grows into: the `POST /api/webhooks/stripe` route that ingests three Stripe events and the derived `plan_entitlements` row the app reads on every request. Coding cements six durable skills, each a "make the wrong shape impossible" lever: (1) the webhook trust boundary — read the raw body once, verify the signature before parsing or logging, answer 400 `application/problem+json` on failure so Stripe treats the delivery as terminal; (2) verify-then-claim-then-mutate inside one `db.transaction` so a crash mid-handler can never leave partial state and a replayed event ID never mutates twice; (3) the derived-view projection — a pure `subscriptionToEntitlement` function plus the single-writer rule (only the webhook writes the row, every other surface reads); (4) the out-of-order guard — the `last_event_at < event.created` predicate lives in the UPDATE's `WHERE`, so a stale event silently no-ops under row-lock; (5) the thin SDK interface — `lib/billing/` as the only place `stripe` is imported, exposing exactly `upgrade`/`openPortal`/`requirePlan`, with `requirePlan` as the load-bearing Server-Component gate; (6) tenancy hardening — cross-checking the Customer's owning org against the metadata carry-channel so a forged `organization_id` cannot write to the wrong tenant. The student practices each by filling stubs against a brief, proving the local loop with `stripe listen` + `stripe trigger`, and reading the inspector's panels as the live forensic surface.

## Slices

### Slice S1 — Verify before you parse

Scope: **Lesson 2.** Write the webhook route's verification skeleton: read the raw body once, verify the Stripe signature, answer 400 `application/problem+json` on any failure — with structured logging on every disposition. No claim, no dispatch, no DB write yet.

In scope:
- `src/app/api/webhooks/stripe/route.ts`: replace the 404 stub. `export const runtime = 'nodejs';` (the Stripe SDK is Node-only; `constructEvent` is synchronous on Node). `export const POST = async (request: Request): Promise<Response> => { … }`: read `const body = await request.text()` **exactly once**; read `const signature = request.headers.get('stripe-signature')`; if the header is missing, log disposition `missing_header` (keyed by no event id yet) and return the 400 problem+json; wrap `stripe.webhooks.constructEvent(body, signature, env.STRIPE_WEBHOOK_SECRET)` in `try`/`catch`; `catch` discriminates `instanceof Stripe.errors.StripeSignatureVerificationError` → log `invalid_signature`, return 400 problem+json with `title: 'invalid_signature'` (no body echo, no detail leakage) — any other thrown error rethrows (a genuine 500). On success, log `verified` with `event.id` + `event.type` and return `Response.json({ received: true }, { status: 200 })` (the claim/dispatch land in S2). A missing header and a bad signature are the **same** 400 answer.
- The `problemJson(status, title)` helper: if not already carried in (059 has `lib/result.ts` but no problem-json helper), add `src/lib/problem.ts` exporting `problemJson(status: number, title: string): Response` returning `application/problem+json` with `{ type: 'about:blank', title, status }`, no detail leakage. (If a 063-era helper already exists in the fork, reuse it.)
- Structured logging: `const log = logger.child({ seam: 'webhook.stripe' })`; one line per disposition (`missing_header` / `invalid_signature` / `verified`), each carrying the event id when available. Never log the body before verification (log-injection vector).

Out of scope: claiming the event, the dispatch switch, any DB write (S2); the projection + handlers (S3); the `billing.*` interface (S4); the metadata cross-check (S5).

Contracts created: the `POST` handler's verified→200 / unverified→400 shape; `problemJson` (`@/lib/problem`). The inspector's "Tamper signature" / "Missing header" debugs consume this route.

Screenshot: none (S2 captures the inspector's `processed_events` tail once it gains a row; the route's 400 behavior is a rendered check against the debug panel, not a figure).

### Slice S2 — Claim the event inside one transaction

Scope: **Lesson 3.** Wrap the post-verification path in `db.transaction`, claim the event against `processed_events` to dedupe, and stub the dispatch switch so every event type is logged. A first-seen event writes one `processed_events` row; a replay writes none. The `plan_entitlements` panel stays `free` (handlers still empty).

In scope:
- `route.ts`: after `constructEvent` succeeds, open `await db.transaction(async (tx) => { const claimed = await claimEvent(tx, 'stripe', event.id, event.type); if (!claimed) { log.info({ eventId: event.id }, 'duplicate'); return; } log.info({ eventId: event.id }, 'claimed'); await dispatch(tx, event); })`. Return `Response.json({ received: true, duplicate: <bool> }, { status: 200 })` on both the claimed and the duplicate path (a duplicate is a **success** path — never a 4xx/5xx, which would tell Stripe to retry forever). Every DB call inside the closure uses `tx`, never the global `db` (mis-routing to `db` opens a racing outer transaction).
- `src/lib/webhooks/stripe.ts`: implement `dispatch(tx, event)` as an exhaustive `switch (event.type)` over `'checkout.session.completed'` → `onCheckoutCompleted`, `'customer.subscription.updated'` → `onSubscriptionUpdated`, `'customer.subscription.deleted'` → `onSubscriptionDeleted`, with a `default` that logs `unhandled` and returns (Stripe sends events from dashboard misconfigurations the app never subscribed to — refusing them is dashboard noise, answered 200). The three handler bodies stay log-and-return stubs this slice.
- Structured logging covers `verified` / `duplicate` / `claimed` / `dispatched` / `unhandled`, each keyed by event id.

Out of scope: the projection + the real handler bodies + the ordering predicate + audit writes (S3); `billing.*` (S4); the cross-check (S5).

Contracts created: the transaction wrapper + `{ received, duplicate }` response shape; `dispatch(tx, event)` (`@/lib/webhooks/stripe`). The inspector's `processed_events` tail + "Replay last event" debug consume this.

## Locked decisions

- **`getEntitlement(orgId)` and `hasActiveAccess(e)` live in `src/db/queries/entitlements.ts`, NOT `lib/billing/`** (ch064 L6 continuity contract, overriding the chapter outline's lines 70/114–116 which place them in `lib/billing/entitlement.ts`). - **`lib/billing/**` is the only place `stripe` is imported** (064 L6). The Stripe singleton is `lib/billing/stripe.ts` with `apiVersion: '2026-05-27.dahlia'` (the `stripe@22.2.0` `LatestApiVersion` — the only literal the SDK's `apiVersion` field type accepts; an older string is a `tsc` error). The `current_period_end`-on-item gotcha 064 taught against basil carries forward to dahlia unchanged (still on `sub.items.data[0]`, gone from the Subscription root), so the lesson is intact; do NOT pin an older literal. - **The webhook is the only writer for `plan_entitlements`.** Server Actions, the Portal return, and the Checkout success page all **read** (`getEntitlement`), never write. The success page reads-and-polls via `router.refresh()` (provided) — it never calls `sessions.retrieve` (the webhook owns writes; trusting `session_id` is the named anti-pattern).
- **Verify → claim → mutate in ONE `db.transaction`.** The signature verify happens before the body is parsed or logged (log-injection + RFC 9457 discipline); `claimEvent` and all entitlement/audit writes share one `tx`; a duplicate (lost claim) returns 200 with `{ duplicate: true }` (never 4xx/5xx). Every DB call inside a handler uses `tx`, never the global `db`. - **`event.created` and `current_period_end` at the Stripe seam are plain numbers/Dates, no Temporal** (063 L3 / 064 L4 simplification): `lastEventAt`/`currentPeriodEnd` are `timestamptz` columns, so each takes a `Date` — both `event.created` and `item.current_period_end` are Unix-**seconds** integers converted `new Date(<n> * 1000)` before they land in a column or feed the `lt(lastEventAt, …)` comparison (a raw seconds integer in a `timestamptz` `.set(...)` is a `tsc` error). The seconds value is never stored or compared raw. No `Temporal` import, no `lib/temporal.ts`.

The write runs inside the same route `db.transaction` as the entitlement write, only on a non-zero UPDATE (or every UPSERT); a logging failure rolls the change back. The webhook does NOT wrap in `withTenant` (no `app.org_id` RLS session is needed — the route runs as the superuser `postgres`, which has `BYPASSRLS`, and the org is carried on the explicit event, not the session variable). `audit_logs.organizationId` is `text` (the Better Auth org id is `text`, not `uuid`).
