# Chapter 059 — Project plan: org, RBAC, and invitations end-to-end

## Project goals

Cement Unit 9 by turning the chapter 055 single-user dashboard into a multi-tenant SaaS, wiring every Unit 9 structural defense end-to-end on the carried-in auth + email scaffold.
The student practices the four patterns the unit installed, each one a "make the wrong shape not compile" lever: organizations as the tenancy unit with `activeOrganizationId` on the session and a `requireOrgUser()` that returns `{ user, orgId, role }`; `tenantDb(orgId)` as the only scoped data facade (the missing-org-filter bug class is unwriteable); `authedAction(role, schema, fn)` as the only privileged Server Action shape (the missing-role-check bug class is unwriteable); an append-only `audit_logs` table the Postgres RLS policies themselves protect (deny UPDATE/DELETE), written in the *same* `withTenant(orgId, …)` transaction as the work via a `logAudit(tx, event)` whose `Transaction`-typed first arg refuses off-transaction calls; and a capability-bearing invite URL — a 32-byte token plus an HMAC signature, hashed at rest, mailed only after the transaction commits — that carries a stranger from email to a seat across four arrival shapes.
The codebase is deliberately backend-shaped with one provided verification surface: a single `/inspector` Server Component (members panel, role-change selects rendered to *every* role on purpose, invite form, pending list, live audit tail, raw-helpers sanity panel) plus the student-built `/accept-invite` page. The inspector renders privileged controls to every identity so the server-side refusal — not a client-side hide — is the observable, load-bearing defense. Features stay minimal (one role-change action, one send, one accept; no remove/leave/transfer, no teams, no fine-grained permissions) so the student completes the workflow quickly and the structural lessons land without app-feature noise.

### Slice S2

Scope: **Lesson 3 — Append-only audit_logs with RLS.** Lay down the `audit_logs` table with org-isolation + deny-UPDATE/DELETE RLS policies, the `withTenant(orgId, fn)` transaction helper, and the transaction-required `logAudit(tx, event)` writer. Win: in `psql` as the app role, an audit insert succeeds inside a `withTenant` transaction and fails outside one; UPDATE/DELETE match zero rows; the inspector's raw-helpers panel shows an `auditLogs` count of 0 for the current org.

Build:
- `src/db/audit.ts`: define `auditLogs = pgTable('audit_logs', { … }, (t) => [ …indexes…, …policies… ])`. Columns (continuity-note canonical shape): `id uuid pk $defaultFn(() => uuidv7())`, `organizationId text notNull` FK→`organization` `onDelete: 'cascade'`, `actorUserId text` nullable FK→`user` `onDelete: 'set null'`, `actorIp text` (NOT `inet` — Drizzle has no first-class `inet` builder), `actorUserAgent text`, `action text notNull`, `subjectType text notNull`, `subjectId text notNull`, `payload jsonb().$type<Record<string, unknown>>().notNull().default({})`, `createdAt timestamp({ withTimezone: true }).defaultNow().notNull()`. **`organizationId`/`actorUserId` are `text`, NOT `uuid`** — Better Auth generates `organization.id`/`user.id` as `text` (its id strategy is a base62 string, not a UUID); a `uuid` FK→`text` makes `drizzle-kit generate` emit DDL Postgres rejects at apply with *"foreign key constraint … cannot be implemented … incompatible types: uuid and text."* Only the standalone `audit_logs.id` PK (no incoming FK) stays `uuid`/`uuidv7()`. NO `updatedAt`/`deletedAt` (append-only tell). Indexes (explicit names): `idx_audit_logs_org_created` on `(organizationId, createdAt desc)`, `idx_audit_logs_org_actor_created` on `(organizationId, actorUserId, createdAt desc)`. `.enableRLS()`. Policies: `pgPolicy('audit_logs_org_isolation', { as: 'permissive', for: 'all', to: authenticatedRole, using: sql\`${t.organizationId} = current_setting('app.org_id', true)\`, withCheck: sql\`${t.organizationId} = current_setting('app.org_id', true)\` })`, `pgPolicy('audit_logs_no_update', { for: 'update', to: authenticatedRole, using: sql\`false\` })`, `pgPolicy('audit_logs_no_delete', { for: 'delete', to: authenticatedRole, using: sql\`false\` })`. (No `::uuid` cast in the policy — `organization_id` is now `text`, so `current_setting('app.org_id', true)` compares text-to-text directly; a `::uuid` cast would error against the text column.) Also export `AuditEvent` type and the `auditLogs` row types via `$inferSelect`/`$inferInsert`. `authenticatedRole` from `drizzle-orm/neon`; `sql` from `drizzle-orm`. The `, true` on `current_setting` is load-bearing (missing → NULL → policy false → fail-closed, not 500).
- `src/db/tenant.ts` (the `withTenant` half): `import 'server-only'` first. Export `const withTenant = async <T>(orgId: string, fn: (tx: Transaction) => Promise<T>): Promise<T> => db.transaction(async (tx) => { await tx.execute(sql\`select set_config('app.org_id', ${orgId}, true)\`); return fn(tx); })`. `set_config(..., true)` is transaction-local (the `SET LOCAL` equivalent that accepts a bind parameter) — never plain `SET` (pooled-connection leak). `Transaction` imported from `@/db`. (The `tenantDb` facade is added in S3 in this same file.)
- `src/db/audit-log.ts`: `import 'server-only'` first. Export `const logAudit = async (tx: Transaction, event: AuditEvent): Promise<void> => { … }`. **The first param is the `Transaction` type — no overload accepting bare `db`**, so off-transaction calls don't typecheck. Derives actor/org context itself: `const { user, orgId } = await requireOrgUser()` → `actorUserId = user.id`, `organizationId = orgId`; `const h = await headers()` → `actorIp = h.get('x-forwarded-for')`, `actorUserAgent = h.get('user-agent')?.slice(0, 512)`. Single `tx.insert(auditLogs).values({ organizationId, actorUserId, actorIp, actorUserAgent, action: event.action, subjectType: event.subjectType ?? '', subjectId: event.subjectId ?? ## Locked decisions

Cross-cutting calls every slice honors:

- **Table names are the plugin-owned singulars: `organization` / `member` / `invitation`.** Do NOT rename to `organizations`/`org_members` (the chapter-framing prose used plurals — the lessons corrected this; the plugin owns the names). `audit_logs` is the one hand-authored table (snake-case via client `casing`).
- **File locations (continuity-note canonical, overriding the chapter outline's `src/lib/*` guesses):** `roles.ts` → `src/lib/auth/roles.ts`; `authedAction` → `src/lib/auth/authed-action.ts`; `requireOrgUser` lives **in `src/lib/auth.ts`** (the file and the `src/lib/auth/` directory coexist); `tenantDb` + `withTenant` → `src/db/tenant.ts`; `logAudit` → `src/db/audit-log.ts`; `auditLogs` table → `src/db/audit.ts`; tenant-scoped reads → `src/db/queries/<entity>.ts`; invitation logic → `src/lib/invitations/{url,send,accept,manage}.ts`; invite template → `src/emails/invite.tsx`; accept page → `src/app/(auth)/accept-invite/page.tsx`; inspector → `src/app/(protected)/inspector/`.
- **Action naming: verb+noun, NO `Action` suffix** (Ch057 L2 convention, overriding the chapter outline's `…Action` names): `changeMemberRole`, `sendInvitation`, `acceptInvitation`. The dev-only inspector helpers keep `Action` only to disambiguate (`switchUserAction`, `resetAndReseedAction`).
- **The `Result` union is the carried-in 7-code set, UNCHANGED** (`'validation'|'conflict'|'not_found'|'unauthorized'|'forbidden'|'rate_limited'|'internal'`). Transport codes the wrapper emits: `'forbidden'` (role) + `'validation'` (parse). Domain reasons (`'last-owner'`, `'cannot-change-owner-role'`, already-member, duplicate-pending) live **inside action bodies** and map to `'conflict'`/`'not_found'` by message — never invent `'already-invited'`/`'already-member'`/`'last-owner'` as `Result` codes (the student never learned them).
- **`authedAction` is the only privileged Server Action shape; `tenantDb`/`withTenant` are the only scoped data paths.** Four fixed-order steps (resolve → authorize → parse → call), authorize before parse. Refusals return a `Result`, never throw (a throw 500s the action and loses the typed contract `useActionState` renders). Do not inline the wrapper's checks in an action body — the import shape is what review catches. Do NOT add steps to the wrapper (no logging/entitlements/rate-limit).
- **`tenantDb` is app-layer scoping only and does NOT set `app.org_id`.** The audit-bearing transaction is the separate `withTenant(orgId, …)` from `@/db/tenant`, which runs `set_config('app.org_id', …, true)` so the `audit_logs` org-isolation policy clears the INSERT. Reads stay on `ctx.db`. The mutation + its `logAudit` row co-transact in one `withTenant` — never `ctx.db.transaction`, never a deferred/post-commit audit write. `set_config(..., true)` (transaction-local), never plain `SET`.
- **Direct Drizzle writes through `tx`, never `auth.api.*`, for member/invitation mutations** — the plugin runs its `after` hooks post-commit, breaking the one-transaction audit contract. The **single sanctioned `auth.api` write** is `auth.api.setActiveOrganization` in `acceptInvitation` (an in-process session write, post-the-other-writes, before the redirect).
- **`logAudit(tx, event)`'s first param is the `Transaction` type with no bare-`db` overload** — off-transaction calls don't typecheck. The caller passes only the event; the helper derives actor/org from `requireOrgUser()` + `headers()`.
- **RLS is wired ONLY on `audit_logs`** (org-isolation `FOR ALL` + deny `UPDATE`/`DELETE`, all `to: authenticatedRole`). The `authenticated` role is **created by a `--custom` migration before the policy migration** (drizzle-kit never creates it — it is a Neon-provided role absent on local Docker, where `CREATE POLICY … TO authenticated` would otherwise fail to apply). Every RLS table is **enabled AND forced** (a `--custom` migration adds `FORCE ROW LEVEL SECURITY` — drizzle-kit emits ENABLE only). The org-isolation predicate compares `organization_id` (a **`text`** column) to `current_setting('app.org_id', true)` with **no `::uuid` cast** (text-to-text); the `, true` makes a missing setting NULL → policy false → fail-closed, not 500. The app connects as the superuser `postgres` in local dev (which bypasses RLS); the policies demonstrate via a `SET ROLE authenticated` `psql` session. The non-owner request role is Chapter 081 (named-not-built). All other tables use `tenantDb` application scoping only.
- **The accept URL is a capability**: 32-byte `crypto.getRandomValues` token, base64url; only `sha256(token)` (64-char hex) stored; HMAC over `${id}.${token}` with `INVITATION_SIGNING_SECRET` (distinct from `BETTER_AUTH_SECRET`); key imported once, non-extractable; verify via `crypto.subtle.verify` (constant-time, never `===`). Email send sits **outside** the DB transaction (send-after-commit). The accept page's verify-ladder first three failures (signature/row/hash) collapse to one generic refusal. The action re-verifies independently (the POST is a separate request); `sig` is not an action input.
- **`requireOrgUser` reads the role fresh from the membership row** via `auth.api.getActiveMember`, never from a query/route param; the active-org id comes only from the server-validated session. React-`cache`d per request, resolved through the single cached `getSession`. Redirects: no session → `/sign-in`; no active org or no membership → `/onboarding/create-org`.
- **`organization()` goes BEFORE `nextCookies()` in `plugins`** (nextCookies last to flush Set-Cookie); the client plugin set mirrors the server (`organizationClient()` registered or `authClient.organization.*` doesn't typecheck). `additionalFields` type-inference on `auth.api.*` is buggy (better-auth#5122) → the Drizzle `$inferSelect` table type is the source of truth for `tokenHash`/`acceptedAt`, never `auth.api` returns.
