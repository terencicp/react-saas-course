# Chapter 079 — Project plan: routed customer wizard with Zustand

## Project goals

Layer a four-step routed "new customer" wizard onto the Ch062 customers surface, with one per-feature Zustand store owning the draft across four route segments.
The student practices standing up the canonical Zustand-on-App-Router skeleton end to end and cementing the Ch078 reflexes against one runnable screen: the `createStore` (vanilla) factory composed from four typed slices, the `useRef`-pinned `WizardStoreProvider` mounted on the **shared `/customers/new` layout** (the per-request rule that stops one tenant's draft leaking into another's SSR render — the load-bearing reason from Ch078 L2), the typed `useWizardStore<T>(selector)` hook, atomic selectors that keep keystroke re-renders surgical, **derived** per-step validity (`selectIsStepValid` runs the current step's Zod schema against its own slice — no stored `isValid`, no `validate()` action), the Next-gate that bundles `goNext()` + `router.push` in one handler, the step-4 review reading all three data slices through one `useShallow` pick, and the `createCustomer` Server Action that re-parses the composite payload at the boundary then triggers success-reset + redirect.
The coding develops these skills by forcing the student to: split a React boundary (provider + hook in `_components/`) from pure store code (slices, types, selectors, factory in `_lib/`) under the no-React-in-`lib/` convention; derive validity rather than store it; and keep the store strictly client-only so the only client↔server seam is the submit button calling the action.
Scope is deliberately minimal: a four-field contact step, a six-field billing step, a three-control preferences step, a read-only review — the smallest surface that exercises four slices, per-step gating, and the submit boundary. No autosave, no draft persistence, no step-skipping UI, no field-level "touched" tracking.

- `store.ts`: `import { createStore } from 'zustand/vanilla'` and `import type { StateCreator } from 'zustand'`. Type `composeSlices` **as a `StateCreator`** so `...a` is the exact slice-arg tuple — `Parameters<...>` over `createStore<WizardStore>` resolves to `undefined`/`never` (the factory arg is optional in the overload), so it does not compile. Define the composition so `reset` produces a complete `WizardStore`:
  ```ts
  const composeSlices: StateCreator<WizardStore, [], [], WizardStore> = (...a) => ({
    ...createContactSlice(...a),
    ...createBillingSlice(...a),
    ...createPreferencesSlice(...a),
    ...createMetaSlice(...a),
    reset: () => a[0]({ ...composeSlices(...a), ...initialWizardData }, true),
  });
  export const createWizardStore = () => createStore<WizardStore>()(composeSlices);
  export type WizardStoreApi = ReturnType<typeof createWizardStore>;
  ```
  Typing `composeSlices` as `StateCreator<WizardStore, [], [], WizardStore>` (the **full store** as the produced slice) gives `...a` the `(set, get, store)` tuple every slice receives, so `a[0]` is `set` and the spread of each slice factory typechecks. `reset` re-spreads the slices (fresh action identities + blank data via `initialWizardData`) and writes with the replace flag `true` — a **complete** store object, so v5's replace type-check passes and no stale sub-object survives. Vanilla `createStore`, never `create`. Empty middleware generics. No module-scoped store instance. Export `WizardStoreApi = ReturnType<typeof createWizardStore>` (the `StoreApi<WizardStore>` handle) for the provider's ref and the broadcast hook.

### Slice S2 — Field bindings and the derived Next-gate

Scope: bind every step-1/2/3 field through atomic selectors + slice setters, render inline Zod errors from a whole-slice `safeParse`, and wire the footer so Next gates on `selectIsStepValid` and advances store + URL together. After this slice steps 1–3 write into their slices with per-step gating; step 4 is still a stub.

Implement:
- `selectors.ts`:
  - Atomic field selectors used at call sites where it reads cleaner as a named function (`selectContactFirstName`, etc.) — optional, but `selectCurrentStep = (s: WizardState) => s.currentStep` and the validity/error selectors below are mandatory.
  - `type Step = { schema: z.ZodType; slice: (s: WizardState) => unknown }` then `const steps: readonly Step[] = [{ schema: contactSchema, slice: (s) => s.contact }, { schema: billingSchema, slice: (s) => s.billing }, { schema: preferencesSchema, slice: (s) => s.preferences }]`. **Annotate the element type `Step` (`schema: z.ZodType`) — do not use `as const`.** Under `as const` each `step.schema` keeps its distinct `ZodObject` type, so `step.schema.safeParse(...).error` is a *union* of three `ZodError<…>` types and `z.flattenError(r.error)` fails to compile (`flattenError<T>` cannot unify the union). Widening `schema` to `z.ZodType` collapses the error to a single `$ZodError`, which `z.flattenError` accepts.
  - `export const selectIsStepValid = (state: WizardState): boolean => { const step = steps[state.currentStep - 1]; return step ? step.schema.safeParse(step.slice(state)).success : true }` — **derived, not stored**; the `step ? … : true` guard covers the review step (step 4) and any out-of-range index under `noUncheckedIndexedAccess`. The returned value is a primitive boolean, so `Object.is(true,true)` short-circuits re-render — the footer re-renders only when validity flips.
  - `export const selectStepErrors = (state: WizardState): Record<string, string[]> => { const step = steps[state.currentStep - 1]; if (!step) return {}; const r = step.schema.safeParse(step.slice(state)); return r.success ? {} : z.flattenError(r.error).fieldErrors }` — flat error shape; each step page reads its own field's `[0]`.
- `step-1/page.tsx`: `'use client'`; four field inputs, each `const firstName = useWizardStore((s) => s.contact.firstName)` + `const setContactField = useWizardStore((s) => s.setContactField)` paired (atomic — a whole-slice read returns a fresh object every state change and re-renders on every keystroke; the re-render counter is the demo). Inline error per field read **atomically as a primitive**: `const firstNameError = useWizardStore((s) => selectStepErrors(s).firstName?.[0])` rendering the `string | undefined` as short red text under the field. **Do not read the whole error object** (`const errors = useWizardStore(selectStepErrors)`): `selectStepErrors` constructs a fresh object every call, so a bare `useWizardStore(selectStepErrors)` returns a new reference each `getSnapshot`, fails `Object.is`, and crashes the page at runtime with React's "The result of getSnapshot should be cached to avoid an infinite loop" (the dev overlay shows *This page couldn't load*) — the exact whole-object hazard the atomic-selector rule exists to prevent. ## Locked decisions

- **In-memory base, no DB/auth/Docker.** The carried-in Ch062 solution is in-memory (`src/server/store.ts` + cookie session). Do not add Postgres, Drizzle, `tenantDb`, Better Auth, `db.transaction`, `drizzle-kit`, `docker-compose`, or `@t3-oss/env-nextjs`. Customers live in the store; the action calls `pushCustomer` + `logAudit`; tenancy is the store's `orgId` filter via the session's `orgId`. The outline's Drizzle/migration carry-in is an idealized lineage the real base lacks — follow the real base.
- **File placement (load-bearing — reconciles the outline's `/lib/wizard/` shorthand to the Ch078 L2/L3 canonical split).** The store lives **route-private**, never under `src/lib/`. Pure/type-only files under `src/app/(app)/customers/new/_lib/wizard/`: `wizard-types.ts`, `contact-slice.ts`, `billing-slice.ts`, `preferences-slice.ts`, `meta-slice.ts`, `selectors.ts`, `store.ts`, `schemas.ts`, `actions.ts`. The two React files under `src/app/(app)/customers/new/_components/`: `wizard-store-provider.tsx`, `use-wizard-store.ts`. Forced by the no-React-in-`lib/` convention (the provider and hook use React) — exactly the layout Ch078 L2 ships and flags as a deliberate divergence. (`actions.ts` is `'use server'`, not React, so it sits in `_lib/` legitimately.)
- **Slices hold data + setters only — no `validate()` actions (load-bearing — reconciles the outline's per-slice `validateContact()/.success`).** Validity is **derived** by `selectIsStepValid` in `selectors.ts`, which indexes a `steps` array (`{ schema, slice }`) and runs `step.schema.safeParse(step.slice(state)).success`. No stored `isValid` boolean, no `validate()` action to keep in sync. Field errors come from `selectStepErrors` running the same `safeParse` and returning `z.flattenError(result.error).fieldErrors`. `selectors.ts` is mandatory and owned by S2.
- **MetaSlice shape (load-bearing — reconciles the outline's `Set`/`markStepComplete`/in-slice `reset`).** `MetaSlice = { currentStep: number; completedSteps: number[]; goNext: () => void; goBack: () => void }`. `completedSteps` is a **`number[]`**, never a `Set` (the RSC/postMessage snapshot must serialize, and the lessons standardized on the array). **No `markStepComplete`** — `goNext` records the current step into `completedSteps` inline. **`reset` is store-level**, living on `WizardStore = WizardState & { reset }`, not inside `MetaSlice`. `WizardState` is the four-slice intersection (data + per-slice actions) only.
- **`currentStep` is 1-based** (1…4) so it maps onto the `step-N` URL; `steps` is indexed `currentStep - 1`.
- **Store factory uses `createStore` from `zustand/vanilla`, never `create`** — the per-request leak from Ch078 L2 is the reason. No module-scoped store instance anywhere. `StateCreator` parameterized on the full `WizardStore` (so `set`/`get` see every slice); empty middleware generics (`[], []`) — no `persist`/`devtools`/`subscribeWithSelector`. **`reset` writes a complete store with the replace flag**: `set({ ...composeSlices(...a), ...initialWizardData }, true)` (re-spread the slices for fresh actions, overlay `initialWizardData` for blank data). v5 compile-checks replace completeness, so a partial `set({}, true)` can't slip through and no stale sub-object survives a reset. A plain `set(partialData, true)` typed as data-only fails the replace type-check — the slice re-spread is what makes it complete.
- **Provider on the shared `/customers/new/layout.tsx`, never per step page.** The `useRef`-pinned store (`if (storeRef.current === null)` init, React-19/Compiler-safe form, not `useState`) survives back/forward across the four segments because the segment layout persists. Per-page mounting is the canonical mistake (resets the draft on every navigation) and is what the `PROVIDER_ON_STEP_PAGE` debug flag demonstrates. React 19 `<Context value={…}>` form, no `.Provider`.
- **`useWizardStore<T>(selector)` is the only store access** — `'use client'`, reads the store from Context, throws `useWizardStore must be used within a WizardStoreProvider` when the Context is `null`, binds with `useStore(store, selector)`. No module-level `.getState()` handle. Selector typed on `WizardState`.
- **Atomic selectors are the default; `useShallow` only on the step-4 review.** Each field binds one atomic selector + one setter. A whole-slice read (`useWizardStore((s) => s.contact)`) returns a fresh object each state change, fails `Object.is`, and re-renders on every keystroke. `useShallow` from `zustand/react/shallow` appears exactly once (step-4 review's three-slice pick). Selecting an action carries no subscription cost (action identity is stable). **`selectStepErrors` is a fresh-object selector and must be read atomically too** — `useWizardStore((s) => selectStepErrors(s).<field>?.[0])` per field (a `string | undefined` primitive), never `useWizardStore(selectStepErrors)`. Because `useStore` runs the selector inside `useSyncExternalStore`, a selector returning a new object reference each `getSnapshot` does not merely re-render too often — it **hard-crashes** the page (React: "The result of getSnapshot should be cached to avoid an infinite loop"; the dev overlay reads *This page couldn't load*). This applies to any object/array-returning selector outside the step-4 `useShallow` pick.
- **Next-gate bundles `goNext()` + `router.push` in one handler; the slice never calls the router.** No `useEffect` watching `currentStep` to orchestrate the push (effects-as-orchestrators is rejected). The gate is UX only — the action re-parses server-side, so a bypass still returns `{ ok: false }`.
- **Submit is a programmatic `createCustomer(input)` call inside a `useTransition`, not `<form action>` + `useActionState`** — the payload already lives parsed in the client store, so re-encoding it as `FormData` is ceremony with no upside. `isPending` drives both the "Creating…" label and the double-submit guard. **`reset()` runs before `router.push` on success only**; the failure branch sets local `useState` error and never resets. Do not stash the new customer's id in the store (it is server state the redirect transitions to).
- **`createCustomer` is a direct-input Server Action via `authedInputAction('member', createCustomerInput, fn)`** (added to `src/lib/authed-action.ts`). Ch062's FormData `authedAction` returns `(prev, formData) => Promise<Result>` — that signature cannot be called with a plain object. Re-parse the **composite** `createCustomerInput` at the boundary (correctness, not UX). Audit action string `customer.created`. `revalidatePath('/customers')` after the write, before return (no tag invalidation — Unit 14 not taught).
- **Result codes are the canonical Ch062 shape.** Malformed payload → `authedInputAction`'s `safeParse` failure → `err('validation', …)` → `{ ok: false, error: { code: 'validation' } }` (the outline's `'invalid-input'` literal does not exist in this `Result`). Forced failure → `err('internal', 'Forced action failure for verification')` (not `'forced_failure'`). Duplicate email → `conflict(...)` → `code: 'conflict'`. Zod 4 top-level builders (`z.email()`), `z.strictObject`, `z.flattenError(...).fieldErrors`.
- **Zod schemas (provided in `_lib/wizard/schemas.ts`):**
  - `contactSchema = z.strictObject({ firstName: z.string().min(1).max(80), lastName: z.string().min(1).max(80), email: z.email(), phone: z.string().min(7).max(20) })`
  - `billingSchema = z.strictObject({ line1: z.string().min(1), line2: z.string(), city: z.string().min(1), region: z.string().min(1), postalCode: z.string().min(1), country: z.string().length(2), taxId: z.string().min(1), paymentTerms: z.enum(['net15','net30','net60']) })`
  - `preferencesSchema = z.strictObject({ channels: z.array(z.enum(['email','sms','inApp'])).min(1), defaultCurrency: z.string().length(3), language: z.enum(['en-US','en-GB','fr-FR']) })`
  - `createCustomerInput = z.strictObject({ contact: contactSchema, billing: billingSchema, preferences: preferencesSchema })`
  - Ids in the store are non-UUID (`cust-acme-0001`, `org-acme`, `user-acme-admin`); never `z.uuid()` on them.
- **Zustand scoped to the wizard surface only.** `useWizardStore`/`createWizardStore`/`WizardStoreProvider`/`WizardStoreContext` appear only under `src/app/(app)/customers/new/`. No Server Component imports the store; the action file imports schemas only (no store-provider/hook). Customers list/detail and the inspector are Server Components (the inspector reads the store snapshot through the iframe + postMessage bridge, not by mounting the provider).
- **Stable selectors (`data-testid`):** `wizard-progress` (single element — the layout places it as one slot, must resolve to one element), `wizard-back`, `wizard-next`, `step-1`/`step-2`/`step-3`/`step-4` (each the step's form container, one element), `field-firstName`/`field-lastName`/`field-email`/`field-phone`/`field-line1`/`field-city`/`field-region`/`field-postalCode`/`field-country`/`field-taxId`/`field-paymentTerms`/`field-defaultCurrency`/`field-language`, `channel-email`/`channel-sms`/`channel-inApp`, `error-<field>` (inline field error), `review-contact`/`review-billing`/`review-preferences`, `wizard-submit`, `submit-error`. The wizard layout drops `{children}` as one slot — each step page's root must resolve to a single element, never split into siblings.
- **Toolchain (from constraints, carried from Ch062, keep idempotent):** `tsconfig.json` — `"jsx": "react-jsx"`, `"skipLibCheck": true`, `"incremental": true`, `include` has both `".next/types/**/*.ts"` and `".next/dev/types/**/*.ts"`, no `baseUrl` (TS6 hard error — `"paths": { "@/*": ["./src/*"] }` resolves under `moduleResolution: "bundler"`), `strict` + `noUncheckedIndexedAccess`, `verbatimModuleSyntax`, `allowJs: false`. `biome.json` — `quoteStyle: 'single'`, exclude `next-env.d.ts` via `"files": { "includes": ["**", "!next-env.d.ts", "!.next", "!node_modules"] }` (no trailing `/**`), `css.parser.tailwindDirectives: true`. `next.config.ts` — `cacheComponents: true`, `typedRoutes: true`, `reactCompiler: true`, `turbopack: { root: __dirname }` (carried verbatim). `typedRoutes` is live: the footer's and submit button's dynamic-href `router.push` must cast `as Route` per the constraint; `verify` runs `next typegen` to regenerate `.next/types/` **before** `tsc --noEmit`, so the standalone typecheck validates the casts (a cold checkout has no `.next/types/`, so a `tsc` placed before typegen passes vacuously — the constraint's order trap). `babel-plugin-react-compiler@1.0.0` devDep. `pnpm-workspace.yaml` with `allowBuilds:\n  sharp: true` in both `solution/` and `start/`. (No `esbuild` entry needed — no `drizzle-kit`/`tsx` in this in-memory base.)
- **React Compiler is on (`reactCompiler: true`); no manual `useMemo`/`useCallback`/`React.memo`.** `useShallow` is a Zustand equality helper, not manual memoization — it stays. The `useRef` store pin and `useTransition` are the only non-render hooks the wizard needs.
