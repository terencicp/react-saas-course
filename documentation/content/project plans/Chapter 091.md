# Chapter 091 — Project plan: testing the Stripe webhook and Checkout money path

## Substrate decision (architect's call — read first)

This project is unlike every prior project chapter: **the student's deliverable is test files, not app features.** The runnable app is the **Chapter 065 Stripe-webhook solution, carried in unchanged** as the system under test (SUT). The student writes four files — three Vitest integration tests against the real `POST /api/webhooks/stripe` route handler + a real test Postgres with per-test rollback, and one Playwright spec driving the Upgrade-to-Pro money path against a production build. Everything else (the 065 app, the full test harness, both DB lifecycles, MSW, the auth/event/rollback fixtures, both Playwright and Vitest configs) ships **provided**.

Consequences that shape this plan and bind every coder:

- **The 065 SUT is byte-identical to `projects/Chapter 065/solution/` and is never edited.** No `runtime` export is added (065's route deliberately omits it — `cacheComponents: true` makes Next reject an explicit `runtime` export; Node is already the default). No `getDb()`/ALS seam is grafted into `route.ts`/`stripe.ts`/`entitlements.ts`. The test transaction reaches the route through a **`vi.mock('@/db')` proxy** wired in the integration setup (see Locked decisions) — invisible to the student and to production. Editing the SUT to make it testable would defeat the chapter's whole thesis (test the real handler).
- **Two independent test systems coexist, do not conflate them:**
  1. The **lesson-test gate** — `pnpm test:lesson <n>` → `lesson-verification/Lesson <n>.ts`, written later by `project-lesson-test-coder`, run by the reviewer. Node env, no DOM. It inspects the **source shape** of the student's test files (does `webhook-checkout-completed.int.test.ts` call `postWebhook`, assert on `plan_entitlements`/`processed_events`/`audit_logs`/`resendCalls`?). It does **not** boot Postgres or run the student's suite.
  2. The **student's own suite** — `pnpm test:integration` (Vitest, `tests/integration/**/*.int.test.ts`, real test Postgres) and `pnpm test:e2e` (Playwright, `tests/e2e/**/*.spec.ts`, production build). This is what the chapter teaches the student to write and run.
- **The render pipeline (`project-screenshotter` / `project-inspector`) boots the 065 app and serves its inspector deterministically against seeded Postgres.** Because this chapter teaches *test files* (no new visible surface), almost every slice has `Screenshot: none`. The carried 065 app has **no `/billing` page** — only `/billing/success` (the Checkout return poller) — and the Checkout **entry** (the `checkout-pro-button` → `upgrade('pro')`) lives on `/inspector`. So the chapter outline's `/billing` framing is corrected: the money path the Playwright spec drives enters at `/inspector` and returns to `/billing/success`. The one screenshottable rendered surface a lesson displays is the **`/inspector` Checkout-entry region** at seeded `free` (entitlement panel + enabled Upgrade button), deterministic without live Stripe; carried-065 inspector checks confirm the SUT still boots and renders. The live Checkout→success→Pro round-trip, `stripe trigger`, and the integration suite running green are the lessons' **by-hand checklist** (they need `stripe listen` + a test-mode Stripe account the pipeline has no access to) — exactly the 065/085 posture for flows requiring a live third party.
- **`pnpm verify` (`biome ci . && tsc --noEmit && next build`) must pass in both `solution/` and `start/`.** The student's test files import devDependency-only modules (`vitest`, `@playwright/test`, `msw`) and reference test-only types. To keep `verify` green and honest, `tests/` is governed precisely (see Scaffolding recipe → tsconfig and Locked decisions): both the standalone `tsc --noEmit` and `next build`'s internal typecheck honor the root tsconfig's `include`/`exclude`, so `solution/` type-checks the four completed `tests/` files (they pass) while `start/` adds `"tests"` to `tsconfig` `exclude` (the four stubs would otherwise need full signatures that give the lesson away) — the established posture from prior project starts. `next build` never *bundles* `tests/` regardless (no `app/**` import reaches it).

This diverges from the chapter outline's `degit`-starter framing only in mechanics (the pipeline uses `start/` + `solution/`, not a published starter repo); every concept the unit teaches — integration tests at the webhook seam, per-test rollback, MSW at the network boundary, signed-payload webhook testing, idempotency, fail-closed signature rejection, and one Playwright money path with `storageState` + `frameLocator` — is preserved and built faithfully.

## Project goals

The project cements Unit 18 (Testing, ch086–090) as one runnable layered suite proving the Chapter 065 Stripe money path. Writing these four files develops six durable senior skills: (1) **read a test as a behavior contract** — each `it` name states what it proves, and the assertion targets only the caller-observable surface (the `processed_events` row, the `plan_entitlements` fields, the `audit_logs` row, the typed `Response`, the recorded `resendCalls`), never `lib/webhooks/stripe.ts` internals or private helpers; (2) **integration tests against real Postgres with per-test transaction rollback** — the suite runs green twice in a row with no reset because every test wraps in a `withRollback` body and the route's transaction joins it; (3) **mock at the boundary, not inside the function** — MSW intercepts Resend's `POST /emails` (the genuine taught network boundary), and Stripe's `subscriptions.retrieve` is stubbed at the SDK seam (`vi.mock('@/lib/billing/stripe')` — MSW can't intercept `stripe@22`'s Node HTTP client, see Locked decisions); either way the test never reaches into `onCheckoutCompleted`/`dispatch` internals, and `onUnhandledRequest: 'error'` turns a stray outbound call into a loud failure; (4) **sign webhook payloads the way Stripe does** — `stripe.webhooks.generateTestHeaderString` produces a header the real `constructEvent` verifies, and a one-character tamper proves the fail-closed front door rejects before any work; (5) **prove idempotency structurally** — a pinned `event.id` sent twice yields `200`/`duplicate:false` then `200`/`duplicate:true`, one claim row, an unchanged `updatedAt`, one audit row; (6) **drive one money path end-to-end** — Playwright on a production build via `webServer`, `storageState` for one-time auth, role-first locators, `frameLocator` for the Stripe Checkout iframe, asserting on what the **user sees** (the success-page poll flipping from "Finalizing…" to "Your plan is now pro"), never re-asserting the DB write the integration tests own. The capstone is the behavior-anchored proof: deliberate handler mutations localize failure to the one test that asserts the mutated behavior.

The goal is not to build an app; it is to walk the canonical 2026 "prove the money path with the right test at each layer" workflow so the honeycomb shape, the rollback discipline, and the network-boundary reflex are cemented — adding the fourth/fifth integration test (the ordering predicate, the `subscription.deleted` path) becomes minutes of work on the same harness. Scope stays minimal: three integration tests + one Playwright spec; no component tests (089's surface is not this app), no contract tests against live Stripe, no CI wiring (Unit 20).

### The `@/db` test mock (the load-bearing harness piece — provided)

Wired in `src/test/integration-setup.ts`:
```ts
vi.mock('@/db', async (importActual) => {
  const actual = await importActual<typeof import('@/db')>();
  const proxy = new Proxy({} as typeof actual.db, {
    get(_t, prop) {
      const current = testTxContext.getStore() ?? getTestDb();
      if (prop === 'transaction') {
        // The route opens db.transaction(fn). Inside withRollback a tx is already
        // open, so run fn(tx) directly on it (the route's "transaction" becomes a
        // no-op join, and the OUTER withRollback transaction owns the rollback).
        return (fn: (tx: Transaction) => Promise<unknown>) =>
          fn(testTxContext.getStore() ?? (current as Transaction));
      }
      return Reflect.get(current, prop);
    },
  });
  return { ...actual, db: proxy, dbUnpooled: proxy };
});
```
This keeps `route.ts`/`stripe.ts`/`entitlements.ts`/`audit-log.ts` **byte-identical to 065** (they `import { db } from '@/db'`): every `db.query`/`db.insert`/`db.transaction` resolves to the test transaction in scope, so the real handler runs against the rollback tx with zero SUT edits. The `Transaction` type is re-exported from `@/db`. Confirm the proxy threads correctly before locking (a happy-path `postWebhook` must write a row readable via `tx`).

The **same `integration-setup.ts`** registers a second `vi.mock` for the Stripe SDK (Stripe cannot ride MSW — see Locked decisions): `vi.mock('@/lib/billing/stripe', async (importActual) => { const actual = await importActual<typeof import('@/lib/billing/stripe')>(); return { ...actual, stripe: { ...actual.stripe, webhooks: actual.stripe.webhooks, subscriptions: { retrieve: async (id: string) => lookupSubscription(id) } } }; })`. Only `subscriptions.retrieve` is replaced (reading the per-test registry); `webhooks.constructEvent`/`generateTestHeaderString` stay the real implementation, so signing in `post-webhook.ts` and verification in the route both run for real.
