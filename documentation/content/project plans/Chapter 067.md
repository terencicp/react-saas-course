# Chapter 067 — Project plan: Durable CSV export with Trigger.dev

## Project goals

The student builds the canonical Trigger.dev v4 durable-job shape: a paginated CSV export of an org's invoices, fired fire-and-forget from a Server Action and surviving a mid-run worker kill. Coding cements six "make the wrong shape impossible" skills, each a primitive the student must defend: (1) the trigger boundary — `schemaTask` validates a `z.strictObject` payload at the trigger edge (never inside the body), because a task has no request context so `organizationId`/`requestedBy` travel as payload cargo and tenancy is re-derived via `tenantDb(organizationId)` inside the run; (2) multi-tenant back-pressure — one predeclared `queue({ name, concurrencyLimit: 1 })` at module scope plus `concurrencyKey: organizationId` at the trigger call, which serializes within an org and parallelizes across orgs (the v4-native shape; a dynamically-named queue is rejected); (3) durability at the seam — each page is its own `paginatePage` child via `triggerAndWait`, and every `triggerAndWait` is a checkpoint, so a crash between pages 3 and 4 resumes at page 4 when the parent re-issues the same idempotency keys; (4) two-scope idempotency — a business key `idempotencyKeys.create([orgId, userId, dayBucket()], { scope: 'global' })` dedupes a same-day re-trigger at the action, while per-step `idempotencyKeys.create([orgId, 'page', String(page)])` (`scope: 'run'`) keys make completed children return cached on retry; (5) live progress — the parent writes `metadata.set('pagesTotal'/'pagesDone')` and the inspector's poller reads run state structurally via the Trigger.dev REST API, never log strings; (6) guarded side effects — the ready-email is its own `triggerAndWait` child keyed by `idempotencyKeys.create([orgId, 'export-email'])`, and the run closes by updating the `exports` row and writing one `export.invoices.completed` audit entry in a single `tenantDb` transaction, with `AbortTaskRunError` reserved for the permanent empty-resultset case. The student writes only the three task files and the `startExport` action against a brief; the cloud-linked project, the inspector, the seed, the React Email template, and the REST-read helpers are provided.

## Scaffolding recipe

**Fork the org/RBAC/audit/auth backend lineage** (the chapter-059 solution shape: Postgres 18 + Drizzle 0.45 + Better Auth organization plugin + `audit_logs` + `tenantDb` + `authedAction` + `requireOrgUser` + the `(protected)` surface + the provided-inspector pattern), then graft on the Chapter 062 invoices data layer (the `invoices` table + the cursor-paginatable `listInvoices` read) and the Chapter 050 Resend send path. The dependency map points at 062, but 062 is the no-DB in-memory production-list; this project's carry-in (tenancy via `tenantDb`, `audit_logs`/`logAudit`, `authedAction`, real Drizzle/Postgres, Resend) is the 059 backend, and 062 contributes the invoices schema + `listInvoices` shape re-homed onto Drizzle. **Trigger.dev is layered on top:** `@trigger.dev/sdk` + the CLI, a provided `trigger.config.ts`, the root-level `trigger/` folder, and the provided REST-read helpers + inspector.

**The Trigger.dev worker is NOT available in the build/render pipeline** (the marquee constraint, parallel to 065's Stripe CLI). The render pipeline boots Docker Postgres + `db:migrate` + `db:seed` only — no Trigger.dev cloud project, no `TRIGGER_SECRET_KEY` round-trip, no local worker. So the provided inspector renders **deterministically from the seeded `exports` table + audit tail**, and ships a dev **"Simulate run" debug** (a Server Action that writes an `exports` row directly to a chosen state — `queued` / `running` with `pagesDone`/`pagesTotal` / `completed` with `downloadUrl` — no Trigger.dev call) so the run-panel figures (progress bar mid-run, completed panel) are reproducible. The live loop (click Export → real run progresses → kill-resume → email arrives) is the lessons' by-hand checklist, which requires the two-terminal CLI setup.

### Slice S2 — One checkpoint per page

Scope: **Lesson 3.** Turn the placeholder body into a real paginated export where every page is its own durable `paginatePage` child, drive the progress bar through `run.metadata`, and abort permanently on an empty resultset.

In scope:

## Locked decisions

- **Persistence is real Postgres 18 + Drizzle + Better Auth + `tenantDb` + `audit_logs`** (the 059 backend lineage), NOT the 062 in-memory store. The whole teaching point is a task with no request context re-deriving tenancy via `tenantDb(organizationId)` from its payload and writing a co-transacted audit row — both require the real backend. The 062 contribution is the `invoices` schema + the cursor-paginatable `listInvoices`/`countInvoices` read, re-homed onto Drizzle. Do import `drizzle-orm`, `postgres`, `better-auth`, `@t3-oss/env-nextjs`; ship `docker-compose.yml` + `.env`.
- **Trigger.dev is NOT live in the build/render pipeline.** The render pipeline boots Docker Postgres + `db:migrate` + `db:seed` only — no Trigger.dev cloud project, no local worker, no `TRIGGER_SECRET_KEY` round-trip (the env block validates against dummy `tr_dev_…`/`proj_…` values so `next build`'s env check passes). Every rendered check targets what seeded Postgres renders deterministically at first paint (the recent-exports table, the seeded `completed` run panel + audit row) or what the provided `simulateRun(state)` debug writes directly (`queued`/`running`-with-progress/`completed` — a direct `exports` write, no Trigger.dev call). The live loop — clicking Export against a real worker, the progress bar advancing across real pages, the kill-resume drill, the email arriving, the per-org serialization / cross-org parallelism proofs — is the lessons' by-hand checklist (requires `pnpm trigger:dev` + a linked cloud project), matching the 065 posture for flows that need a live third party.
- **The student writes exactly four files:** `trigger/export-invoices.ts`, `trigger/paginate-page.ts`, `trigger/send-export-email.ts`, `src/lib/exports/start.ts`. Everything else (schema, queries, inspector, REST helpers, CSV/email/error/day-bucket libs, config, seed) is provided. No schema/migration work this chapter (the `invoices` + `exports` tables ship complete in the scaffold).
- **`schemaTask` validates at the boundary, never the body.** Every task is a `schemaTask` with a `z.strictObject` payload; `z.uuid()`/`z.int()` are the top-level builders (Zod 4). A malformed payload fails at the trigger edge before the body runs and before retries are spent. The body never re-parses.
- **Queues are predeclared at module scope; per-tenant isolation is `concurrencyKey`, not a dynamic queue name.** `const exportQueue = queue({ name: 'export', concurrencyLimit: 1 })` lives at module scope in `export-invoices.ts` (attached via `queue: exportQueue`), and `concurrencyKey: organizationId` passed at the `tasks.trigger` call is the per-org lane — sequential within an org, parallel across orgs. The v3 dynamically-named-queue / trigger-time-`concurrencyLimit` shape is rejected by v4. **(Overrides Code conventions §"Background work" line which shows `queue({ name: \`org:${orgId}:exports\` })` dynamic-per-tenant phrasing — that is the stale v3 shape; Ch066 L4 continuity notes explicitly correct it to `concurrencyKey`. The lesson syntax wins.)** `concurrencyLimit: 1` is the read-heavy-export back-pressure choice; raise it only if a tenant repeatedly stacks up.
- **`tasks.trigger` (fire-and-forget) from the action; `triggerAndWait` only inside a task body.** `startExport` fires with `tasks.trigger` and returns the handle's `id` immediately — calling `triggerAndWait` from a Server Action would block the request past `maxDuration`. The parent body uses `triggerAndWait` for each `paginatePage` and the `sendExportEmail` child (in-task children, returns the result, each a durable checkpoint).
- **Idempotency keys use `idempotencyKeys.create([...parts], { scope })`, never hand-spliced `${ctx.run.id}:…` strings.** The business key is `idempotencyKeys.create([orgId, userId, dayBucket()], { scope: 'global' })` + `idempotencyKeyTTL: '24h'` at the action (one export per org-user-day). The per-page key is `idempotencyKeys.create([organizationId, 'page', String(page)])` (default `scope: 'run'` — namespaced to the parent run id, so a parent retry re-issues the same keys and completed children return cached; the parts array is `string[]`, so the numeric `page` is stringified — `idempotencyKeys.create` rejects a raw `number`). The email key is `idempotencyKeys.create([organizationId, 'export-email'])` (`scope: 'run'`). **(Overrides Code conventions §"Background work" idempotency-key naming lines which show `${ctx.run.id}:${step}:${n}` / `${userId}:${taskId}:${ymd}` template strings — Ch066 L5/L6 continuity notes explicitly supersede the manual prefix with `idempotencyKeys.create`. The lesson syntax wins; the template string is named only as the underlying mental model.)** Dropping `scope: 'run'` for `'global'` on the page key would collide across runs; folding `Date.now()` into any key would break the cache on retry.
- **Durability lives at the seam.** Each page is its own `triggerAndWait` child (one checkpoint per page); the loop is sequential `for...of`/`for(;;)` + `await` (parallel `Promise.all` races the concurrency limit and reorders rows). Cursor pagination is the stable restart point. The body re-derives tenancy via `tenantDb` inside `listInvoices` — `requireOrgUser()` does not exist in a task; `organizationId`/`requestedBy` ride in the payload.
- **`metadata.set` from the parent is the live-progress channel.** The parent writes `metadata.set('pagesTotal', total)` once and `metadata.set('pagesDone', page + 1)` each iteration, plus `metadata.set('downloadUrl', …)` at the end. Progress is set from the parent (its view is user-facing); omitting the `pagesDone` write is the zero-bar bug. The inspector reads run state structurally via the Trigger.dev REST API (`retrieveRun`), never by scanning logs.
- **`AbortTaskRunError` for permanents only; plain throw for transients.** The empty-resultset case (`total === 0`) is a permanent failure on the same inputs → `AbortTaskRunError` (stops immediately, spawns no children). A transient (DB blip, Resend 5xx) throws normally and retries with the configured backoff. Never wrap a child call in a manual try/catch+retry loop (stacks two retry layers).
- **Side effects are exactly-once across a parent retry.** The email is a `sendExportEmail` child keyed by `[organizationId, 'export-email']` (a parent retry returns the cached child result, not a second send). The `exports`-row update + the `export.invoices.completed` audit write share one `tenantDb(organizationId).transaction`; the audit write comes **after** the email (audit the shipped outcome). `logAudit(tx, …)` is called with explicit `{ organizationId, actorUserId: null, action, subjectType, subjectId, payload }` (the task has no session — `actorUserId: null` is the system-actor pattern; never the 059 session-deriving form inside a task). A Resend suppression is an expected Result (`err('suppressed', …)`), not a throw — the run still completes and the audit note records it.
- **Timestamps are plain `Date` at the Postgres `timestamptz` boundary; `dayBucket()` is a `YYYY-MM-DD` string. No Temporal, no `lib/temporal.ts`** (the 062/065 simplification carried forward — the student has not seen Temporal in domain code). `requestedAt`/`completedAt` are `timestamptz` columns taking a `Date`; `dayBucket` is `text`. **(Code conventions §"Time" makes Temporal the default and forbids `Date` in domain code — this project carries the established Stripe/list-seam exception: `Date` only, at the DB column boundary. No Temporal import.)**
