# Chapter 071 ‚Äî Project plan: notification dispatcher

## Project goals

The student assembles the dispatcher seam taught across chapter 070 into one runnable surface: a single `dispatch(event)` entry point that every call site fires through, fanning three events across an email channel and an in-app inbox channel. Coding cements five durable, "make-the-wrong-shape-impossible" skills. (1) **The one seam** ‚Äî `dispatch` is the only place channel knowledge lives; call sites build a typed `NotificationEvent` and `await dispatch(...)`, never importing `sendEmail` or writing `notifications` directly (a grep for either outside `lib/notifications/` is the regression check). (2) **The registry as source of truth** ‚Äî `notifiableEvents` (`as const satisfies Record<string, NotifiableEvent>`) enumerates what is notifiable; adding an event is one entry, adding a channel later is one function of the same signature. (3) **Preference resolution read once, default-on** ‚Äî a single batched prefs read per dispatch, `?? true` so a missing row receives everything, plus the critical-channel override that keeps billing email flowing even when the user toggled it off. (4) **Time-windowed dedup** ‚Äî a 60-second window keyed `(eventType, dedupKey, recipientUserId)` collapses a burst to one notification, the check sitting after prefs and before channels. (5) **Fire-after-commit discipline** ‚Äî at three real call sites (invite, role-change, billing webhook) the dispatch runs only after the action's transaction commits, so rolled-back work notifies nobody, and each channel runs behind its own `try/catch` so one failing channel never kills the other. The student practices each by filling stubs against a brief, driving the loop from a provided `/inspector`, and reading `/inbox` as the live inbox read.

**Carry-in from chapter 070's lessons (NOT re-derived ‚Äî these are the canonical shapes the project must match):**
- `dispatch(event)` takes one options object `NotificationEvent = { type: EventType; recipientUserIds: string[]; subjectId: string; payload: Record<string, unknown> }` and returns `DispatchResult = { sent: number; deduped: number; suppressedByPrefs: number }` ‚Äî **a flat count summary, deliberately NOT a `Result<T>` and NOT per-channel.**
- `notifiableEvents` registry `as const satisfies Record<string, NotifiableEvent>`; keys are `domain.entity.action` dotted literals; `EventType = keyof typeof notifiableEvents`. Entry fields: `channels`, `templates: { email, inbox }`, `preferenceCategory`, `dedup: { windowSeconds, keyBy }`, optional `criticalChannel`, `description`. The `templates.email` field type in the provided `types.ts` must be the permissive `React.ComponentType<any>` (equivalently `(props: any) => ReactElement`) ‚Äî **never** `(props: Record<string, unknown>) => ReactElement` and never `ComponentType<Record<string, unknown>>`: each shipped template (`InviteSentEmail` etc.) declares its own typed payload, and a typed-prop component does NOT assign to a `Record<string, unknown>`-param field under TS 6 strict (parameter contravariance ‚Äî verified, `TS2322`), nor would the channel be able to call it with `rendered.emailProps` (a `Record<string, unknown>`). Only the `any`-prop form accepts every typed template AND stays callable with the rendered props.
- `ChannelFn = (args: { recipient: Recipient; event: NotificationEvent; payload: Record<string, unknown>; rendered: RenderedContent }) => Promise<void>`; `channelFns = { email: sendEmailChannel, inbox: writeInboxChannel } satisfies Record<ChannelName, ChannelFn>`; the dispatcher loops over resolved channels and calls `channelFns[channel](args)` with no branch. `Recipient = { userId: string }`. `RenderedContent` is pre-computed once per recipient: `{ emailProps, inbox: { title, body }, orgId, ...display }`.
- `userNotificationPreferences` `(userId, category)` named-unique, per-channel `email`/`inbox`/`push` booleans default-on at the column; `resolveChannels(event, prefs)` with `prefs?.[channel] ?? true` and the `criticalChannel` override; one batched read across recipients (`inArray(userId, ids)` + `eq(category, ‚Ä¶)`) into `prefsByUser: Map`.
- `notification_dedup` `(eventType, dedupKey, recipientUserId, firedAt)` + composite index; `isDuplicate({event,userId,payload})`/`recordDedup({event,userId,payload})`; per-recipient loop order: `resolveChannels` ‚Üí skip-if-empty ‚Üí `isDuplicate` ‚Üí skip+`deduped++` ‚Üí fan-out ‚Üí `recordDedup`.

## Locked decisions

- **`DispatchResult` is a flat count object `{ sent: number; deduped: number; suppressedByPrefs: number }`** ‚Äî NOT per-channel, NOT a `Result<T>` (chapter-070 L1/L4 continuity, overriding the chapter-071 outline's `sent: { email, inbox }`). `sent` is a single running total incremented once per successful channel send; the inspector's counters, the `DispatchResult` panel, every requirement, and the structured log (`logger.info({ seam: 'notifications.dispatch', ...result })`) read the flat shape. `isDuplicate` returns a bare `boolean`, also not a `Result<T>`.
- **No HMAC unsubscribe token, no `List-Unsubscribe` header, no `NOTIFICATION_UNSUBSCRIBE_SECRET`, no unsubscribe-token inspector tool, no RFC 8058 callout** (chapter-070 L2/L3 cut it explicitly ‚Äî "Do not add it back" ‚Äî and it is correct for 2026: one-click unsubscribe is required only for promotional bulk email, not transactional notifications). `sendEmailChannel` calls `sendEmail` with no `headers` arg. Opt-out is exclusively the per-category preference toggle. There is no `tokens.ts` in `lib/notifications/`. There is no `/api/email/unsubscribe` handler in this project.
- **Channel-function name is `writeInboxChannel`** (chapter-070 L2, overriding the outline's `writeInboxRow`); the registry join field is **`preferenceCategory`** (chapter-070 L1/L3, overriding the outline's `category`). The dedup helpers are **`isDuplicate`/`recordDedup`** (chapter-070 L4, overriding the outline's `claimDedup`). The prefs read is **`readPrefsForCategory`** returning `Map<userId, PrefRow | undefined>`; the dispatcher map is **`prefsByUser`**.
- **The dispatcher is the only seam.** `dispatch` imports from `@/lib/notifications` only at the call sites; a grep for `sendEmail(` or `db.insert(notifications)` (or `.insert(notifications)`) outside `src/lib/notifications/` is a regression ‚Äî zero hits required. Call sites build a `NotificationEvent` and `await dispatch(...)`; they never import a channel or write `notifications`.
- **The registry is the source of truth**, `as const satisfies Record<string, NotifiableEvent>` so `EventType` is inferred and an unknown key is a compile error; adding an event stays a one-entry change. `notifiableEvents[type] === undefined` ‚Üí `throw new NotificationError('REGISTRY_MISS', type)` before the per-recipient loop, never swallowed (a programmer error, not a channel failure).
- **Preferences read once per dispatch, batched, default-on.** One `WHERE userId IN (...) AND category = ?` query before the loop (never per-recipient); `prefs?.[channel] ?? true` (a missing row receives everything ‚Äî silence-by-default is worse than friction); `|| c === eventDef.criticalChannel` forces a critical channel back on. `userNotificationPreferences` has **no `orgId`** (user-scoped ‚Äî prefs follow the user across orgs); `notification_dedup` has **no `orgId`** (user-scoped bookkeeping). Both override the tenant-leading-column rule deliberately.
- **Dedup sits inside the dispatcher, after prefs, before channels**, keyed `(eventType, dedupKey, recipientUserId)` ‚Äî `recipientUserId` is load-bearing (two recipients getting the same event is not a duplicate). The 60-second window is per-event in the registry. The dedup row is inserted **after** a successful fan-out (`recordDedup` last in the per-recipient order), and skipped for a fully-suppressed recipient (no channels ‚Üí no dedup row, so the count and window stay honest). Check-then-insert race accepted in v1; the unique-constraint upgrade and the Redis-backed swap (chapter 075) are named, deferred.
- **Channels are uniform and independent.** `ChannelFn = ({ recipient, event, payload, rendered }) => Promise<void>`; `channelFns = { email, inbox } satisfies Record<ChannelName, ChannelFn>`; the dispatcher loop is `await channelFns[channel](args)` with no branch on channel name. Each channel runs behind a **per-channel** `try/catch` (one failing channel never kills the other); an expected channel failure (`sendEmail` error, `RECIPIENT_NOT_FOUND`) is logged and swallowed, never thrown out of `dispatch`. Channel functions return `Promise<void>`, never `Result<T>`. Every channel/dispatcher/registry file starts with `import 'server-only'`.
- **Render-at-dispatch.** The inbox row's `title`/`body` are rendered once in the dispatcher (`eventDef.templates.inbox(payload)` into `RenderedContent`) and frozen onto the `notifications` row ‚Äî the inbox UI is a pure read with no joins, immune to later actor-name drift. The email idempotency key is deterministic from event identity: `${event.type}:${event.subjectId}:${recipient.userId}`.
- **Fire-after-commit at every call site.** `await tx`-work resolves, **then** `await dispatch(...)` ‚Äî a notification for rolled-back state is the failure mode the seam prevents. The webhook is the awkward case: the handler runs **inside** `route.ts`'s `db.transaction`, so it cannot dispatch; it reads the org owners inside the tx and pushes a `NotificationEvent` onto a closure-captured array, and `route.ts` dispatches after the transaction commits ‚Äî **importing the notification entry as `dispatchNotification` because `route.ts` already binds `dispatch` to the carried Stripe event router (`dispatch(tx, event)`); the two are different functions with different arities and must not share a name.** The transactional-outbox alternative is named, deferred. The dispatcher trusts its caller (gating is at the `authedAction` boundary).
- **`sendEmail` is called without `from`/`replyTo`** (the wrapper owns them from env) and without `headers`. Passing any of them from the channel is a regression.
- **The email mock is the inspector's verification proxy.** `EMAIL_MOCK=1` (default) makes `sendEmail` short-circuit before Resend, bump an in-process counter, and return `ok` ‚Äî so the email-sent counter is deterministic and no live Resend round-trip happens in dev or the render pipeline. The student never edits `lib/email.ts`. `setEmailShouldFail(true)` (the `Make email fail` debug) makes the mock return `err` to prove channel independence.
- **The inbox reads `userId` from the session / acting identity, never `orgId` from a query string** ‚Äî `/inbox` and the inspector inbox panel resolve the user from `requireOrgUser()` (or the dev acting cookie), so a hand-crafted cross-tenant URL cannot leak another user's notifications.
- **No caching of the notification surfaces.** The inbox feed is not cached; `/inbox` and `/inspector` re-render via `revalidatePath('/inspector')` (the carried action pattern) / `router.refresh()`. No `cacheTag`/`revalidateTag`/`'use cache'` anywhere (Unit 14 unlearned). The `cacheTag('notifications', userId)` reach is a named forward reference.
- **Structural single-slot / single-element invariants:**

Canonical ids: `inspector-page`, `inspector-header`, `user-switcher`, `org-switcher`, `prefs-panel`, `pref-toggle-team-email`, `pref-toggle-team-inbox`, `pref-toggle-billing-email`, `pref-toggle-security-email`, `fire-invite-sent`, `fire-role-changed`, `fire-billing-past-due`, `rapid-fire-invite-sent`, `dispatch-result`, `result-sent`, `result-deduped`, `result-suppressed`, `dedup-badge`, `email-sent-counter`, `inbox-panel`, `inbox-row`, `inbox-empty`, `processed-events-tail`, `debug-controls`, `force-registry-miss`, `make-email-fail`, `wrap-invite-rollback`, `reset-reseed`, `inbox-page`, `inbox-page-row`, `inbox-page-empty`.
