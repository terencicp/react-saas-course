# Chapter 069 — Project plan: presigned R2 upload

## Project goals

The student builds the canonical browser-to-R2 upload surface plus the Chapter 067 export retrofit, cashing in the Chapter 068 R2 primitives as one runnable feature. Coding cements seven "make the wrong shape impossible" skills: (1) **the function is never a byte pipe** — `presignedPut` signs a direct-to-R2 `PutObjectCommand` so the multi-MB body goes straight to the virtual-hosted R2 host `${bucket}.${R2_ACCOUNT_ID}.r2.cloudflarestorage.com` (the AWS SDK's default addressing, which is exactly the form R2 expects) while only small JSON crosses the action; (2) **the two-step write** — sign → browser PUT → `finalizeUpload` HEADs the object → row insert, never row-before-bytes, so a never-completed upload leaves no orphan row lying in the UI; (3) **server-observed identity over client claims** — `byteSize`/`contentType` come from the post-upload `HeadObjectCommand`, never the client's claim, the layered size defense that catches a lying client (the R2 `ContentLength` non-enforcement quirk makes the HEAD the real boundary); (4) **server-constructed object keys** — `objectKey = org/${orgId}/files/${uploadId}.${ext}` is built from the org and a server-generated UUIDv7, never anything the client sends, because a client-chosen key/id is the tenancy-bypass shape; (5) **fresh-per-render presigned GETs** — the `/files` list mints a new short-lived GET per row per render and never persists or caches it (`getSignedUrl` is local HMAC, effectively free), so a copied URL dies at expiry while a refresh re-issues a working one; (6) **tenancy at every read** — every download/list read goes through `tenantDb(orgId)` filtered to non-deleted rows, so one org never sees another's files; (7) **one mechanism, two consumers** — one `lib/r2.ts` client powers both the browser-PUT user uploads and the server-PUT export retrofit, where the byte-pipe rule flips (a worker with the CSV in memory PUTs directly — presigning back to itself is ceremony). The student writes six surfaces — `presignedPut`, the `file_metadata` migration, `finalizeUpload`, the XHR upload form, the fresh-GET list query + `/files` page, and the export retrofit — against a brief; `lib/r2.ts`, the key/error helpers, the inspector, the seed, and the whole Chapter 067 backend are provided.

- **R2 is NOT live in the build/render pipeline.** The render pipeline boots Docker Postgres + `db:migrate` + `db:seed` only — no R2 bucket, no live PUT/HEAD/GET round-trip, no `pnpm r2:cors`/`r2:lifecycle` run. The four R2 env vars validate against **dummy values** so `next build`'s env check passes and `getSignedUrl` (local HMAC, no R2 call) can sign real-shaped URLs at render time. Every rendered check targets what seeded Postgres renders deterministically (the `/files` list off seeded `file_metadata` rows with locally-signed GET hrefs; the upload form's resting markup; the seeded `completed` export on `/inspector`). The live loop — a real browser PUT landing bytes, the HEAD verifying true size/type, a copied URL 403ing at 11 minutes while a refresh re-issues a working one, the export emailing a working R2 link, the cross-org tenancy `object-not-found` proof, the CORS allow/deny — is the lessons' by-hand checklist (requires the student's own R2 bucket + `pnpm r2:cors`), matching the 067/065 posture for flows needing a live third party.
- **The student writes exactly six surfaces:** `lib/files/presigned-put.ts`, `lib/files/finalize.ts`, `db/queries/file-metadata.ts`, `app/files/page.tsx`, `app/files/upload-form.tsx`, and the closing R2 write in `trigger/export-invoices.ts`. Everything else (`lib/r2.ts`, the key/error/soft-delete helpers, the CORS/lifecycle scripts, the schema/tenant/seed, the inspector, the whole 067 backend) is provided.
- **The `lib/files/` vs `db/queries/` home is split by responsibility, honoring the taught Chapter 068 layout** (the outline-aligner's resolution): **tenant-scoped reads live in `db/queries/file-metadata.ts`** (`getFile`/`getFileDownloadUrl`/`getSignedGetForKey`/`listFiles`) — Chapter 068 lesson 4/5 continuity is explicit this is the delivered home, NOT `lib/files/file-metadata.ts` (the outline brainstorm was wrong), and Code conventions §"Data layer" places tenant-scoped read helpers in `db/queries/`, one file per entity. **The Server Actions + pure helpers live in `lib/files/`** (`presigned-put.ts`, `finalize.ts`, `keys.ts`, `errors.ts`, `soft-delete.ts`) — matching the Chapter 067 `lib/exports/{start,to-csv,errors,day-bucket}.ts` precedent for a feature's actions + pure seams. The `lib/files/presigned-put.ts`-style filenames the Chapter 068 outline mentioned were outline-only and never delivered there — this project delivers them as the actions' home (they are Server Actions, not tenant reads). No `lib/files/file-metadata.ts` and no `lib/files/list.ts`/`presigned-get.ts` (those reads are in `db/queries/file-metadata.ts`).
- **The function is never a byte pipe for user uploads; the worker PUTs directly.** `presignedPut` signs a direct-to-R2 URL — the multi-MB body never crosses the action (only small JSON does). The export worker, with the CSV already in memory, issues a server-side `PutObjectCommand` directly — presigning a PUT back to itself is ceremony (Chapter 068 lesson 5). One `lib/r2.ts` client, two consumers, the byte-pipe rule flipping by caller.
- **The two-step write: sign → PUT → finalize-with-HEAD → insert. No row before the bytes.** `presignedPut` writes no DB row; `finalizeUpload` inserts only after the HEAD confirms the object. A never-completed upload leaves an orphan object (cheap, lifecycle-swept), never an orphan row (a UI lie). The HEAD-then-insert is not transactional with R2 (a microsecond gap, accepted).
- **Server-observed identity over client claims.** `byteSize`/`contentType` come from the post-upload `HeadObjectCommand`, never the client's claim. The signed `ContentLength` documents intent but R2 does not enforce it — the HEAD is the real size boundary (the layered defense: client pre-check → signed `ContentLength` → post-upload HEAD). `objectKey` is server-constructed via `buildObjectKey` from `ctx.orgId` + the server-generated `uploadId`, never client-supplied; the extension comes from the validated `contentType` via the static map, never the user filename.
- **Object key + upload id are server-generated.** `uploadId = uuidv7()` in `presignedPut`; `finalizeUpload` inserts the row with `id: uploadId`. A client-chosen id/key is the tenancy-bypass shape (a crafted value could clobber another org's row); the `unique(objectKey)` constraint (global, not partial) is the backstop that rejects a replayed finalize.
- **Presigned GETs are fresh-per-render, never persisted, never cached.** The `/files` page mints a new `getFileDownloadUrl` per row per render; `file_metadata` has **no `url` column**. The page MUST NOT opt into `'use cache'` (a cached response freezes URLs that then expire — the page would lie). `getSignedUrl` is local HMAC (no R2 round trip), so N signs per render is not a hot spot. PUT expiry 300s, GET expiry 600s.
- **Tenancy at every read via `tenantDb(orgId)`, filtered to non-deleted rows.** Every `file_metadata` read (`getFile`, `getFileDownloadUrl`, `listFiles`) goes through `tenantDb(orgId).query.fileMetadata` with `isNull(softDeletedAt)` as the OUTER predicate — a cross-org `fileId` returns null (→ `object-not-found`), a soft-deleted row stays hidden. `getSignedGetForKey` is the lone tenant-free helper (the export worker is inside the trust boundary, signs a raw key).
- **`file_metadata` for user uploads; no row + lifecycle rule for exports.** User uploads are long-lived, user-managed, multi-consumer → a `file_metadata` row. The export is a throwaway single-consumer artifact → no row, a 7-day lifecycle rule on the literal `exports/` prefix segment (R2 prefix matching is a literal string prefix, not a glob). One bucket per environment; the prefix (`files/` vs `exports/`), not a separate bucket, carries the workload split.
