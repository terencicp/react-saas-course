# Chapter 055 — Project plan: email+password auth with verification

## Project goals

Cement Unit 8 by wiring the canonical first auth flow end-to-end on the carried-in invoicing/email scaffold: configure the Better Auth `auth` instance, generate the four-table Drizzle schema, mount the one catch-all handler, and ship sign-up → verification email → click-to-verify → sign-in → protected `/dashboard` → sign-out (the `session` row disappears).
The student practices: composing `betterAuth({...})` with `nextCookies()`, the Drizzle adapter, and the `emailAndPassword` + `emailVerification` blocks at the call-site defaults the course tightened to (12-char min, `requireEmailVerification: true`, `autoSignIn: false`, `__Host-` cookie prefix); the one cached `getSession` seam every read flows through (`getCurrentUser` / `requireUser`); enumeration-safe Server Actions that parse with Zod first and return the canonical 7-code `Result<T>`; the verification email riding the existing Unit 7 `sendEmail` pipeline through a purpose-built React Email template; the two-layer request-time gate (`proxy.ts` cookie-presence + layout validating read), the `?next=` open-redirect closure via `safeNext`, the inverse gate, and a sign-out Server Action that revokes by deleting the row.
The codebase is deliberately one flow — two auth-page groups, one protected group, one catch-all route, four stub modules — so the student completes it quickly and the structural lesson ("the `auth` instance is the single seam; the proxy gates, the layout validates, the action authorizes") lands without app-feature noise. No OAuth, passkeys, 2FA, magic links, or password reset: the project stays in the email+password lane.
