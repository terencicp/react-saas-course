# Chapter 073 — Project plan: caching the invoices list with tag-driven invalidation

## Project goals

The student layers Next.js 16 tag-driven caching onto the chapter-062 invoices list and watches every cache decision play out on a `fetchedAt` strip. Coding cements five durable, "make-the-wrong-shape-impossible" skills. (1) **Opt-in caching on a dynamic-by-default surface** — only three reads (list, per-org summary, detail) earn `'use cache'`; the toolbar and every Client Component stay dynamic, and adding the directive to a request-scoped read is the trap named and avoided. (2) **`tags.ts` as the single source of truth** — three namespaced `invoiceTags` helpers (`.list` / `.record` / `.summary`); read sites and write sites import the same functions, and a grep for a raw `org:` / `invoice:` string outside `tags.ts` and the cached reads is the regression check. (3) **The cached read is a pure function of its arguments** — `orgId` arrives as an argument, never read from the session inside the cached body, so the cache key can never silently bake one tenant's data into a shared entry. (4) **`updateTag` vs `revalidateTag` picked by who is waiting** — the four lifecycle actions fan `updateTag` out (read-your-writes, the same user is on the redirect) after commit and before redirect; a background summary job calls `revalidateTag(tag, 'max')` (eventual, nobody waiting, stale-while-revalidate). (5) **The deliberate failure modes** — a misuse branch that swaps `revalidateTag('max')` into an action for the list tag (the eventual primitive where read-your-writes wants `updateTag`; the visible single-render break is the cross-process chapter-074 reality, the in-app signal is the logged `action`-sourced `revalidateTag` row), and a "force `updateTag` from the job" button shows the framework throw the API's architectural rule. The student practices each by filling five stubs against a brief and reading every cache-state change off the `/inspector` and the `/invoices` strip.

**Persistence substrate (architect's call — read this first, it overrides the chapter outline).** This project continues the **chapter-062 codebase**, which is a deterministic in-memory module store (`src/server/store.ts`) — **no Postgres, no Drizzle, no Docker, no Better Auth, no Trigger.dev, no `.env`** (chapter 062 made that call following the chapter-035 precedent; see `documentation/content/project code outlines/Chapter 062.md`). The chapter-073 outline's Postgres / Drizzle / `docker-compose` / `pnpm db:migrate` / Trigger.dev `schemaTask` framing is a preliminary brainstorm describing a DB-backed base that **does not exist on this dependency path**; this plan translates it faithfully onto the in-memory base. The translation is loss-free because **Next.js Cache Components cache the return value of a function, not a database query** — `'use cache'`, `cacheTag`, `cacheLife`, `updateTag`, and `revalidateTag` behave identically whether the cached body reads Postgres or an in-memory array, so the entire teaching surface (cache hit/miss via `fetchedAt`, tag-driven invalidation, read-your-writes vs. eventual) renders deterministically under `pnpm dev` with zero bring-up. The two DB-shaped artifacts become: the `org_invoice_summaries` table → an in-store `summaries` map; the Trigger.dev `summaryRecomputeTask` → a plain in-process async job (`src/server/jobs/summary-recompute.ts`) validated with Zod at its boundary and invoked by an inspector button — which is exactly what the outline already specified the inspector does ("invokes the task body directly… does not require a working Trigger.dev cloud account"). The `cache_invalidation_log` table → an in-store array with the same `logCacheInvalidation(tag, source)` helper.

**Single-process caveat (true and intended).** The dev server is one process; the in-memory store and the Next.js default cache handler both live in it, so an action's `updateTag` and the next read see the same store. The `fetchedAt`-stable-across-refresh demonstration relies on the dev server's **persistent in-memory cache** — Next.js docs note self-hosted / persistent-memory environments reuse a fresh cache entry across requests, whereas serverless instances may not persist entries between requests; `pnpm dev` (and the inspector boot) is the persistent-memory case, so hits are observable. This is sufficient for every project outcome. The cross-process reality (Vercel's shared cache backend) is the explicit forward reference to chapter 074 (Upstash) — named in a comment, not built.

**Cached-read store-capture rule (a real Cache Components footgun, defused here).** Next.js folds a cached function's closed-over outer-scope **variables** into its cache key and warns that awaiting a runtime Promise pulled from shared storage (a module-level `Map`/array) inside `'use cache'` hangs the build. This project's store holds **plain synchronous data** (arrays/maps of plain objects), not Promises, so reading it inside a cached body is a computation, not a runtime-Promise await — safe. To keep the cache key clean, every cached read touches the store **only through the imported helper functions** (`scopedInvoices(orgId)`, `getSummaryRow(orgId)`) — never by capturing a raw mutable `invoices`/`summaries` value into the cached scope — exactly as the chapter-062 `listInvoices` already does. See Locked decisions.

### Slice S2 — Read-your-writes invalidation

Scope: **chapter-073 lesson 3.** Make every edit and lifecycle change refresh the list, the affected record, and the summary on the same render via `updateTag`, after commit and before redirect; wire the misuse-`revalidateTag` failure-mode branch.

In scope:
- `src/lib/invoices/actions.ts` — for each of the four actions (`updateInvoice`, `archiveInvoice`, `restoreInvoice`, `softDeleteInvoice`), **after the store mutation + audit push (the "commit") and before the return/redirect**, fan three `updateTag` calls out through the `tags.ts` helpers: `updateTag(invoiceTags.list(ctx.orgId))`, `updateTag(invoiceTags.record(ctx.orgId, row.id))`, `updateTag(invoiceTags.summary(ctx.orgId))` — the minimum complete set for an invoice mutation (the row moves in/out of the list, its record display changes, the totals shift). Call `logCacheInvalidation(tag, 'action')` **after** each `updateTag` returns (a throwing `updateTag` must not leave a log row claiming success). Keep the chapter-062 `revalidatePath('/invoices')` as well (belt-and-suspenders; the tags are the teaching surface). Ordering, named as a lint-by-eye rule: mutate (commit) → `updateTag` ×3 + log → `revalidatePath`/redirect. `updateTag` is correct here because a specific user sits on the redirect; it is Server-Action-only and is called only through the helpers, never a raw string.

## Locked decisions

- **No database, no auth, no Drizzle/Better Auth/Postgres, no Trigger.dev, no Upstash.** Persistence is the chapter-062 in-memory `src/server/store.ts` (extended with `summaries` + `invalidationLog`); identity is the `acting-identity` cookie via `src/server/session.ts`; the "background job" is a plain in-process async function. Do not import `drizzle-orm`, `postgres`, `better-auth`, `@t3-oss/env-nextjs`, `@trigger.dev/sdk`, or `@upstash/*`; do not add `docker-compose.yml`/`.env`. This is the substrate that boots under `pnpm dev` with no bring-up and renders every cache outcome deterministically. (Overrides the chapter-073 outline's Postgres/Trigger.dev framing — preliminary brainstorm.)
- **Cache primitives are real and load-bearing.** `'use cache'`, `cacheTag`, `cacheLife`, `updateTag`, `revalidateTag` all import from **`next/cache`** (confirmed against Next.js 16.2.9 docs). The directives cache the function's return value, independent of the in-memory substrate, so the entire teaching surface is faithful. `cacheComponents: true` is already in `next.config.ts` (chapter-062 carry-in) — dynamic by default, opt-in only via `'use cache'`.
- **Exactly three reads earn `'use cache'`:** `listInvoices`, `getOrgInvoiceSummary`, `getInvoiceDetail`. The toolbar, view-tabs, pagination, chips, edit-form, and every Client Component stay dynamic (they read request-scoped URL/session state) — adding `'use cache'` to any of them is the trap. The cached reads are **async** functions (a sync function cannot carry `'use cache'`); the page/detail/inspector `await` them.
- **`tags.ts` is the only place tag strings exist.** Read sites and write sites import `invoiceTags.{list,record,summary}`; a grep for a raw `org:`/`invoice:` literal anywhere except `src/lib/cache/tags.ts` is a regression — zero hits required. The three helpers are pure functions of their arguments.
- **Cached reads are pure functions of their arguments.** `orgId` (and `view`/`status`/`sort`/`q`/`cursor`/`pageSize`/`role`) arrive as arguments; **no `getSession()`/`cookies()`/`headers()` call appears inside any cached body** — the page resolves the session and passes `orgId` in. The detail read carries the tag union (record + list).
- **Cached reads touch the store only through imported helper functions** (`scopedInvoices(orgId)`, `getSummaryRow(orgId)`), never by closing over a raw mutable `invoices`/`summaries` value — a closed-over data variable is folded into the cache key and a mutating snapshot would defeat hits; a module-level function reference is not. The store holds plain synchronous data (no Promises), so reading it inside `'use cache'` is a computation, not the runtime-Promise-from-shared-storage await that hangs the build. Hit/miss is proven by the frozen `fetchedAt`, not by data identity.
- **`cacheLife` choices:** `listInvoices` and `getInvoiceDetail` → `'minutes'` (read+edited often; a few minutes is the staleness ceiling if an invalidation ever misfires); `getOrgInvoiceSummary` → `'hours'` (refreshed explicitly by the job, dashboard-only). `profiles.ts` mirrors these for the inspector readout.
- **`fetchedAt` is the only cache-state window.** Each cached read returns `fetchedAt: new Date().toISOString()` computed once inside the cached body (frozen into the entry): stable across requests = hit, advancing = miss/invalidated. Every verification reads it off the `<FetchedAtStrip />`. The framework does not surface hit/miss to user code.
- **Invalidation primitive picked by who is waiting.** Server Action + user on the redirect → `updateTag(tag)` (read-your-writes, synchronous, Server-Action-only, single-arg). Background job + nobody waiting → `revalidateTag(tag, 'max')` (eventual, stale-while-revalidate; the second `cacheLife` argument is **required** in Next.js 16 — the single-arg form is deprecated, warns but does not throw). The misuse branch (`revalidateTag` from an action) and the force-`updateTag`-from-job throw are the two deliberate failure-mode demos; the misuse branch's in-app signal is the logged `action`-sourced `revalidateTag` row (its single-render visible staleness is cross-process, deferred to ch074 — see `r-misuse-stale`).
- **After-commit, then invalidate, then redirect — never inside the mutation.** Each action: mutate the store + push audit (the "commit") → `updateTag` ×3 + `logCacheInvalidation(tag,'action')` → `revalidatePath`/redirect. Invalidate-then-commit and redirect-then-invalidate are both named wrong. `logCacheInvalidation` is called **after** the real invalidation returns.
- **The three-tag fan-out is the minimum complete set per invoice mutation:** list + record + summary, for all four lifecycle actions (each moves a row in/out of the active set, changes the record display, and shifts the totals). Missing one is the silent stale-read bug.
- **The summary job validates its payload with Zod and calls `revalidateTag` only.** `z.strictObject({ orgId: z.string().min(1) }).parse(input)` at the boundary; recompute over active rows; `upsertSummaryRow`; `revalidateTag(invoiceTags.summary(orgId), 'max')`; `logCacheInvalidation(..., 'job')`. `updateTag` is unavailable in the job (the framework throws) — that throw is the API enforcing the rule, exposed via the force-throw button.
- **Single-process truth.** The dev server is one process; the store mutation, the cache handler, and the next read share it, so the **`updateTag` read-your-writes** path demonstrates end-to-end without a shared backend (a non-misused edit advances `list-fetched-at`/`summary-fetched-at` on the next render). The **eventual** path's *single-render staleness* (the stale value briefly served while the background recompute runs) is **not visually observable here** — the inspector actions `redirect()` to a fresh navigation and the in-process recompute lands before that render, so `revalidateTag('max')` entries (the misuse list edit, the summary job) advance on the very next render too. The observable signals for the eventual path are the logged invalidation row and the recomputed value; the *served-stale-then-fresh* divergence is the cross-process reality (Vercel + Upstash) deferred to chapter 074, named in a comment, not built. Server-action module state (the store arrays/maps and the `misuseFlag` object) is shared across the page-render + action graph; a Route Handler runs in a separate module instance, so the force-`updateTag` route must not depend on shared mutable store state (it reads only `session.orgId` + the pure tag helper).
