# Chapter 3.3 — URLs, origins, and security boundaries

## Chapter framing

Chapter 3.1 installed the request lifecycle from URL bar to interactive page. Chapter 3.2 installed the HTTP contract — methods, status codes, headers — at the depth a SaaS engineer needs. This chapter zooms into the *first* part of every request (the URL the browser parsed) and the *boundary* every cross-context request crosses (the origin the browser enforces). The URL is the addressable surface of the application; the origin is the security boundary the browser polices around it. Both are load-bearing for every later unit — the URL spec lands at Next.js routing and `URLSearchParams`-driven list views (Chapter 11.1), the origin model lands at cookies and the auth trust model (Chapter 3.4, Unit 9), and CORS lands the moment the application's frontend calls a sibling API on a different host (the canonical setup for any SaaS that fronts a `web.app.com` UI with an `api.app.com` backend).

The senior framing: **the URL is a structured value the application reads and writes constantly, and the origin is the unit of trust the browser enforces around it.** A SaaS engineer writes URLs into route definitions, parses them in middleware, mutates their search params to drive list state, and constructs them when redirecting to third parties. Doing any of that with string concatenation rather than the `URL` constructor produces a class of bugs — wrong encoding, path traversal, double slashes, accidental open redirects — that the platform's URL APIs are designed to eliminate. And every cross-origin request the application makes (or receives) gets evaluated against the same-origin policy by the browser before a byte of body data lands at the application. The senior knows what's an origin, what's a site, what the browser blocks by default, and what the application has to opt into to allow.

The chapter ships three teaching lessons. The first installs the URL spec at the depth a SaaS engineer reads and writes — the four anatomical parts (origin, pathname, search, hash), the percent-encoding rules, the `URL` constructor as the parser, `URLSearchParams` as the query-string surface, and the senior reach (parse, don't concatenate). The second installs the same-origin policy as the browser's default trust boundary — the origin tuple, the origin-vs-site distinction that governs cookies and modern security headers, what the policy blocks and what it always allows, and the load-bearing security frame the rest of the chapter and Unit 9 rely on. The third installs CORS as the opt-in mechanism that loosens the same-origin policy for legitimate cross-origin APIs — the simple-vs-preflighted request decision, the response-header surface (`Access-Control-Allow-Origin`, `Access-Control-Allow-Credentials`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`, `Access-Control-Max-Age`), the credentials-with-wildcard trap, and the failure modes the student will hit the first time they wire a cross-origin fetch.

Threads that must run through every lesson:

- **Parse, don't concatenate.** Every URL the application touches goes through `new URL()` or `URLSearchParams`. String concatenation is the source of every URL bug — wrong encoding on a search param, double slashes from a missing trim, an open-redirect vulnerability from an unvalidated query param. The lessons demonstrate the constructors, name what they prevent, and refuse to show concatenation as the "easy way."
- **Origin is the unit of browser trust.** Every restriction the browser enforces (cookie scope, CORS, COOP/COEP, the storage partition, Service Worker scope) is keyed on origin. The student learns "origin = scheme + host + port" as a single load-bearing tuple and reads it on the address bar by reflex. "Site" (scheme + eTLD+1) is named because cookies use it (the `SameSite` attribute, Chapter 3.4), but origin is the primary unit.
- **The same-origin policy is the default; CORS is the opt-out.** The lessons frame it that way — the browser blocks cross-origin reads by default, and CORS is the server-side mechanism that says "I trust this other origin." Naming the default first makes the CORS mechanics legible. The senior reflex: when a fetch fails with a CORS error, the question isn't "how do I disable CORS" — it's "what does the server need to say to authorize this origin."
- **CORS is a browser policy, not a server one.** This is the single most-misunderstood point. CORS doesn't protect the server (the request still hits the server; the server still processed it) — it protects the *user* whose cookies the browser was about to send and whose response data the browser was about to expose to the requesting origin's JavaScript. The senior knows this because it changes how the application reasons about auth, logging, and security boundaries: a CORS-blocked request *was processed*, just not exposed to the calling page. CSRF is a separate concern (Chapter 17.2) and `SameSite` cookies (Chapter 3.4) are the better mitigation.
- **The 2026 default for the course's stack: same-origin everywhere.** Next.js 16's App Router renders the UI and serves Route Handlers on the same origin by default. The student building the course's reference SaaS never hits CORS for first-party API calls. CORS shows up when the application *consumes* a third-party API from the browser (rare in 2026 — the senior reach is to proxy through a server-side Route Handler), when the application *exposes* an API to a partner's origin (the explicit cross-origin contract), or when the dev loop spans two ports (`localhost:3000` vs. `localhost:4000`). Each scenario gets named where it lands.
- **Don't fight the framework when it's right; intervene when the contract demands it.** Next.js handles same-origin routing and most URL parsing implicitly. The student writes URL parsing by hand when wiring URL-driven list state (Chapter 11.1), redirect targets (login flows in Unit 9), webhook callback URLs (Chapter 12.1), and external API constructions (Stripe, Resend). The lesson shows the senior surface for each.
- **Senior anchors for later units are seeded here.** `URLSearchParams` lands again at URL-state list views (Chapter 11.1). The origin tuple lands at cookies (Chapter 3.4) and at the security baseline (Chapter 17.2). The CORS preflight model lands at WebCrypto-signed webhook receivers (Chapter 12.1) and at third-party integrations more broadly. The "parse, don't concatenate" reflex lands at redirect validation (Chapter 9.3) and at open-redirect prevention (Chapter 17.2).

This chapter ships short code snippets — `new URL()`, `URLSearchParams` operations, a Next.js Route Handler returning CORS headers, a `fetch` that fails-then-succeeds with credentials — and a few `TabbedContent`, `Matching`, `Buckets`, and `CodeReview` exercises that lock the vocabulary in. Diagrams carry weight: a hand-authored SVG anatomy of a URL with each part labeled, a Mermaid sequence diagram for the CORS preflight dance, and an interactive widget for the origin-vs-site comparison. The chapter is meaningful on its own — by the end the student designs URLs by parsing through the constructor, names the origin of any URL on sight, knows what the browser blocks and what CORS authorizes, and recognizes a CORS preflight on the DevTools Network panel by its `OPTIONS` method and `Access-Control-Request-Method` header.

The chapter ordering follows the dependency. The URL spec comes first because the same-origin policy is defined in terms of the URL's components — you can't reason about "same origin" without naming origin as `scheme + host + port`. The same-origin policy comes second because CORS is its loosening — CORS is incomprehensible without the default it modifies. CORS comes third because it's the most mechanically detailed lesson and lands on the model the previous two installed. The quiz closes the chapter.

The TOC packs the chapter into three lessons. The slicing holds — each lesson has a distinct mental model, fits comfortably under an hour, and closes on a self-contained piece the student can name and use. The URL lesson is slightly heavier because `URLSearchParams` and percent-encoding both need full treatment; the CORS lesson is the densest because the preflight mechanics, the response-header palette, and the credentials trap all have to land together for the model to be usable.

## Lesson 3.3.1 — Parse, don't concatenate

Teaches the URL as a structured value with origin/pathname/search/hash, `new URL()` and `URLSearchParams` as the senior reach, percent-encoding rules (including the `%20`-vs-`+` split), and the bug classes string concatenation produces.

Topics to cover:

- The chapter-opening senior question: the student has been writing URLs as strings since Chapter 1 — `'/api/invoices'`, `'/dashboard?tab=billing'`, `'https://api.stripe.com/v1/charges'`. The naive operation when they need to add a query param is `'/dashboard?tab=' + tab`. The senior reach is `new URL('/dashboard', base)` and `url.searchParams.set('tab', tab)`. The lesson installs the URL constructor as the parser, `URLSearchParams` as the query-string surface, and the senior reflex of "parse, don't concatenate" that prevents the entire class of URL bugs.
- **The URL as a structured value, not a string.** A URL parses into a fixed set of components defined by the WHATWG URL Standard (the spec all modern browsers and Node.js share — there is no longer a "Node.js URL" vs. "browser URL" split as of Node.js 18+). The full anatomy named with a single canonical example like `https://app.example.com:3000/invoices/123?status=paid&page=2#summary`:
  - **Scheme** (`https:`) — the protocol. `http:` and `https:` for web traffic; `mailto:`, `tel:`, `data:`, `blob:` named in one line for recognition.
  - **Host** (`app.example.com`) — the hostname or IP.
  - **Port** (`3000`) — explicit only when non-default (80 for HTTP, 443 for HTTPS).
  - **Pathname** (`/invoices/123`) — the path the server routes on.
  - **Search** (`?status=paid&page=2`) — the query string, accessible as a structured `searchParams` object.
  - **Hash** (`#summary`) — the fragment, never sent to the server; client-only.
  - **Origin** (`https://app.example.com:3000`) — the security-boundary tuple. Named here because the next lesson lives on it. Origin is `scheme + host + port`; that's the definition the same-origin policy uses.
- **The `URL` constructor as the parser.** `new URL('/invoices/123?status=paid', 'https://app.example.com')` returns a `URL` object with every component as a property. The two-argument form (path + base) is the canonical reach when building URLs in application code that's portable between client and server. The single-argument form (`new URL('https://...')`) parses an absolute URL. A malformed URL throws `TypeError` — the senior reach: parse in a `try/catch` when the input comes from a user or external source, treat the failure as invalid input. The `URL` object's properties are mutable — `url.pathname = '/invoices/456'; url.searchParams.set('page', '3'); url.toString()` produces the rewritten URL. The senior reach for any URL manipulation: get a `URL` object, mutate it, serialize.
- **What `new URL()` prevents.** The bug class string concatenation produces:
  - **Wrong percent-encoding.** A search param containing `&` or `=` or a space inserted by concatenation breaks the query-string format. `URLSearchParams.set('q', 'a&b')` produces `q=a%26b` automatically.
  - **Double slashes.** `base + '/' + path` produces `https://example.com//invoices` when `base` ends in `/`. `new URL('/invoices', 'https://example.com/')` normalizes.
  - **Path traversal.** `'/files/' + filename` is exploitable if `filename` contains `../`. `new URL(filename, 'https://example.com/files/')` resolves the path against the base and prevents escape (the result stays under `/files/` only if the base ends with `/files/` — the senior reach is to validate the resulting `pathname` starts with the expected prefix).
  - **Scheme injection.** `'https://' + userInput + '/path'` is exploitable if `userInput` contains `@` or `?` — the parsed URL points somewhere else entirely. `new URL(userInput, 'https://example.com')` either parses to a safe URL or throws.
  - **Open-redirect injection.** Building a redirect target by string concatenation is the canonical open-redirect vulnerability — Chapter 17.2 owns the full treatment; this lesson names parsing as the first line of defense.
- **Percent-encoding** (also called URL encoding):
  - **The model.** URLs can contain only a restricted set of ASCII characters. Any byte outside that set — non-ASCII characters, reserved characters with special meaning (`/`, `?`, `#`, `&`, `=`), spaces — must be encoded as `%XX` where `XX` is the byte's hex value. UTF-8 is the byte encoding for non-ASCII characters.
  - **The percent-encode sets**, named with the canonical four the spec defines:
    - **C0 control percent-encode set** — control characters and ASCII characters above 0x7E. Applied everywhere.
    - **Path percent-encode set** — adds space, `"`, `<`, `>`, `` ` ``, `#`, `?`, `{`, `}`. Applied to the pathname.
    - **Query percent-encode set** — adds the previous set plus `'`. Applied to the query string.
    - **Application/x-www-form-urlencoded percent-encode set** — the strictest — adds the query set plus `!`, `$`, `%`, `&`, `'`, `(`, `)`, `*`, `+`, `,`, `/`, `:`, `;`, `=`, `?`, `@`, `[`, `\`, `]`, `^`, `` ` ``, `{`, `|`, `}`, `~`. Applied by `URLSearchParams` and HTML form submissions.
  - **The space-vs-plus trap.** In a pathname, a space is encoded as `%20`. In a query string serialized by `URLSearchParams` (which uses the `application/x-www-form-urlencoded` encoding), a space is encoded as `+`. On the parsing side, `URLSearchParams` decodes `+` as space; a literal `+` in a value must be sent as `%2B`. The senior watch-out: hand-rolling query strings produces inconsistent encoding; `URLSearchParams` produces the form encoding that every server-side parser (Express, Next.js, every framework) decodes correctly.
  - **`encodeURIComponent` and `encodeURI` as legacy primitives.** Named in one paragraph for recognition. `encodeURIComponent` encodes a single value (a key or a value), `encodeURI` encodes a full URL while preserving the structural characters (`/`, `?`, `#`, `&`, `=`). Neither matches the WHATWG spec exactly — the senior reach in 2026 is `URL` and `URLSearchParams`, not the legacy `encode*` functions. The exception: when constructing a `data:` URL or a URL fragment by hand, `encodeURIComponent` is still the right primitive.
- **`URLSearchParams` as the query-string surface.** The class for reading and writing query strings without touching encoding by hand:
  - **Construction.** `new URLSearchParams('?status=paid&page=2')` parses a query string. `new URLSearchParams({ status: 'paid', page: '2' })` builds from an object (string values only — numbers and booleans coerce to strings). `new URLSearchParams([['status', 'paid'], ['page', '2']])` builds from a key-value array (useful when keys repeat).
  - **Reading.** `params.get('status')` returns the first value or `null`. `params.getAll('tag')` returns all values for a key (multi-valued params are first-class — `?tag=urgent&tag=invoice` returns `['urgent', 'invoice']`). `params.has('archived')` returns a boolean. `params.has('archived', 'true')` (the two-argument form added in 2023) checks key-and-value.
  - **Writing.** `params.set('page', '2')` replaces all values for the key. `params.append('tag', 'urgent')` adds without replacing. `params.delete('page')` removes all values for the key. `params.delete('tag', 'urgent')` removes a specific value (the two-argument form added in 2023). `params.sort()` sorts keys lexicographically — useful when the URL needs to be canonical for caching keys.
  - **Iteration.** `for (const [key, value] of params) { ... }` walks every pair (including repeated keys). `params.entries()`, `params.keys()`, `params.values()` are the iterator methods.
  - **Serialization.** `params.toString()` produces the query string (without the leading `?`). The reach via the `URL` object: `url.searchParams.set(...)` mutates the underlying string and `url.search` and `url.toString()` reflect it immediately.
- **`URLSearchParams` in 2026 SaaS code.** The canonical use sites:
  - **Server-side URL parsing** — Next.js Route Handlers receive a `Request` object; `new URL(request.url)` gives the full URL, and `url.searchParams` is the typed surface for reading query params. Better than the legacy `request.query` pattern.
  - **Client-side state in the URL** — Chapter 11.1 owns this pattern in depth. The list page's filters, sort, and page number live in the URL so the page is bookmarkable, shareable, and back-button-correct. `URLSearchParams` is the read/write surface; Next.js's `useSearchParams` returns one.
  - **Constructing third-party URLs** — building a Stripe checkout URL, a Resend reset-password link, an OAuth authorization URL. The senior reach: `new URL(base); url.searchParams.set(...); url.toString()`. The reasons: correct encoding, no double slashes, no scheme injection.
- **The hash fragment is client-only.** The browser never sends `#` and everything after it to the server. The senior consequences: hash routing (legacy SPAs) is invisible to SSR; OAuth implicit-flow tokens that arrive in the hash never hit the server; any anchor scroll target is purely a client concern. The 2026 senior reach: don't put state the server needs in the hash; don't put secrets in the hash (the page's JavaScript and any browser extension can read it). Named at the depth that prevents the bug.
- **Punycode and IDN**, named in one paragraph for recognition. International domain names (`bücher.example`, `北京.cn`) are encoded as ASCII punycode (`xn--bcher-kva.example`) on the wire. The `URL` constructor handles the conversion transparently — `url.hostname` returns the Unicode form for display, but the on-the-wire form is punycode. The watch-out: visually identical Unicode characters (the "homoglyph" attack — `аpple.com` with a Cyrillic `а` looks like `apple.com`) — modern browsers detect and display the punycode form in the address bar when the domain mixes scripts suspiciously. The application doesn't usually need to care; the senior knows the mechanism.
- **The `URLPattern` API**, named as the 2026 standard. `URLPattern` is the W3C standard for matching URLs against templates (`/invoices/:id`, `/posts/:slug/edit`). It powers the URL Pattern API and underlies modern routing approaches. Next.js 16's App Router uses its own file-based routing (Chapter 5.1) so the student doesn't author `URLPattern` instances directly, but the syntax is the same shape — `:param` for named groups, `*` for wildcards. Named here for recognition; the lesson points to Chapter 5.1 for the file-routing translation.
- **The watch-outs a senior names**:
  - **Trailing slashes are not idempotent semantically.** `/invoices` and `/invoices/` are different URLs as far as the URL spec is concerned. Frameworks (Next.js) usually normalize them with a redirect or a canonical-form rule, but the application's code shouldn't assume equivalence — pick one form and stick to it in route definitions and links.
  - **Encoding twice is a real bug.** A URL passed through `encodeURIComponent` and then concatenated into another URL that's also passed through `encodeURIComponent` produces `%2520` (an encoded `%20`) instead of `%20` (an encoded space). The reflex: pass raw values to `URLSearchParams.set`; let it encode once.
  - **The `URL` constructor is permissive on the hostname.** It accepts hostnames like `0x7f.0.0.1` and `2130706433` (numeric IP encodings of `127.0.0.1`). Server-Side Request Forgery (SSRF) protection that string-checks for `127.0.0.1` is bypassable; the senior reach is to parse with `URL`, resolve the hostname, and check the resolved IP against allow/deny lists. Chapter 17.2 owns SSRF; named here.
  - **`URL` parses but doesn't validate semantic shape.** `new URL('https://') ` throws, but `new URL('https://x')` parses successfully even though `x` isn't a real hostname. The senior reach when accepting user-supplied URLs: parse with `URL`, then check the protocol is `https:` (or whitelist of allowed schemes), check the hostname is a real domain (or an allowed list), check the path matches expectations. Don't trust `new URL` as validation.
  - **`URLSearchParams` does not deduplicate.** Setting a key three times with `set` replaces the value each time, but `append` accumulates. Iterating gives repeated keys. The senior watch-out: code that assumes one value per key (`Object.fromEntries(params)` collapses to the last one) loses data on multi-valued params.
  - **Query string values are always strings.** A search param like `?page=2` parses as the string `'2'`. The application must coerce — `Number(params.get('page'))` or, with Zod (Chapter 7.1), parse-and-validate-in-one-step with `z.coerce.number()`. Forgetting this is the bug class behind "the page=2 doesn't work" off-by-one.

What this lesson does not cover:

- The same-origin policy and the origin's security semantics (3.3.2).
- CORS, preflights, and the cross-origin response-header surface (3.3.3).
- Cookies and the `SameSite` attribute that uses the site (not origin) definition (Chapter 3.4).
- Next.js routing, file-based routes, dynamic segments, and `useSearchParams` (Chapter 5.1, Chapter 5.5).
- The URL-state list-view pattern at depth (Chapter 11.1).
- Open-redirect prevention and SSRF mitigations (Chapter 17.2).
- Server-Side Request Forgery, the full hostname-allowlist treatment (Chapter 17.2).
- The Fetch API's `Request` and `Response` objects (Chapter 3.6.1).
- OAuth flows and the authorization-URL construction (Unit 9).
- Webhooks and signed callback URLs (Chapter 12.1).
- The full WHATWG URL Standard's edge cases (the file URL parser, the URL host parser's IPv6 syntax) — out of scope.

Pedagogical approach:

Mechanics archetype with a strong reference beat. The lesson teaches a specific API surface (`URL` and `URLSearchParams`) and the senior reflex of parsing rather than concatenating. Open with the senior question — "you need to add `?status=paid` to a URL that already has `?page=2`; what do you write?" — and a small `MultipleChoice` exercise contrasting three approaches (string concatenation that breaks on existing params, manual `?` and `&` insertion that breaks on encoding, `new URL` + `searchParams.set` that handles everything). The discrimination — concatenation is structurally fragile — frames the lesson.

A `Figure` with a hand-authored SVG renders the URL anatomy: a single canonical URL (`https://app.example.com:3000/invoices/123?status=paid&page=2#summary`) with each component labeled by name (scheme, host, port, pathname, search, hash) and bracketed at the top with "origin" wrapping scheme+host+port. The student sees the structure at a glance.

A `ScriptCoding` block has the student construct a `URL` object from a base and a path, read its `pathname`, `searchParams`, `hash`, and `origin`, then mutate the search params (`set`, `append`, `delete`) and observe `url.toString()` updating. The hands-on with the API surface in the console is the lesson's primary deliverable.

A `TabbedContent` block organizes `URLSearchParams` operations into four tabs: read (`get`, `getAll`, `has`, `entries`), write (`set`, `append`, `delete`, `sort`), construct (from string, object, array), and serialize (`toString`, embedded in a `URL`'s `search`). Each tab has the canonical SaaS use case and a one-line watch-out.

A `CodeVariants` block contrasts three implementations of "build a redirect URL with a `next` query param": tab 1 is string concatenation (the broken default — fails on encoded values, double slashes, scheme injection), tab 2 is `encodeURIComponent` + concatenation (better but still fragile — wrong encoding for the form set, no path normalization), tab 3 is `new URL` + `searchParams.set` (the senior reach — correct on every input). The annotations name the bug class each prior tab leaves.

A `Buckets` exercise sorts ten URL operations into "needs `new URL()`" or "needs `URLSearchParams` directly" — parsing a request URL in a Route Handler, building a Stripe checkout URL, reading a single query param from `useSearchParams`, normalizing a redirect target, constructing a sort+filter URL for a list view, parsing a webhook callback URL, encoding a single form value, building a multi-tag query string. The decision discipline locks in.

A `CodeReview` exercise on a 30-line snippet with three URL bugs: a `redirectTo` parameter passed to `res.redirect(req.query.next)` without parsing or validating (open-redirect vulnerability — named here, depth in Chapter 17.2), a search-param construction using `'?' + 'q=' + searchTerm` (encoding bug on special characters), and a pathname built with `'/files/' + filename` (path-traversal risk if filename contains `../`). The student leaves a comment per issue with the senior fix.

A small `PredictOutput` exercise on three `URLSearchParams` operations: `new URLSearchParams('a=1&a=2&a=3').get('a')` returns `'1'` (first value); `new URLSearchParams({ name: 'Alice Smith' }).toString()` returns `'name=Alice+Smith'` (space-as-plus in form encoding); `new URL('/path with space', 'https://example.com').toString()` returns `'https://example.com/path%20with%20space'` (space-as-%20 in pathname). The space-vs-plus distinction is the load-bearing point.

Close with a `Matching` exercise pairing five URL components with their canonical use case — origin (the security-boundary tuple the same-origin policy uses), pathname (the route the server matches on), searchParams (URL-driven list state), hash (the client-only fragment), port (rarely written; significant for the origin definition). The vocabulary is locked in and the lesson hands off to the same-origin policy lesson by naming origin as the next lesson's primary unit.

Estimated student time: 45 to 55 minutes. Load-bearing for Chapter 11.1 and across every URL-touching surface in the course.

## Lesson 3.3.2 — Origin is the unit of browser trust

Teaches origin as the `(scheme, host, port)` tuple versus site as `(scheme, eTLD+1)`, what the same-origin policy blocks versus what it always allows, and the load-bearing point that the policy protects the user (not the server) by gating the response, not the request.

Topics to cover:

- The senior question: the application's frontend runs on `https://app.example.com`. A piece of JavaScript on that page tries to `fetch('https://api.stripe.com/v1/customers')` with the user's Stripe cookie. What does the browser do? The naive answer is "the request goes out." The senior answer names the same-origin policy as the default boundary the browser enforces — the request *does* go out, but the browser refuses to expose Stripe's response to the page's JavaScript unless Stripe's response opts in via CORS headers. The lesson installs the policy at the depth that lets the student reason about every cross-origin scenario the application will hit.
- **Origin defined sharply.** Origin is the **tuple** of `(scheme, host, port)`. Two URLs are *same-origin* if and only if all three components match exactly:
  - `https://app.example.com` and `https://app.example.com/dashboard` — **same origin** (path doesn't matter).
  - `https://app.example.com` and `http://app.example.com` — **different origin** (scheme differs).
  - `https://app.example.com` and `https://app.example.com:8080` — **different origin** (port differs, even though `:443` is the implicit HTTPS port).
  - `https://app.example.com` and `https://api.example.com` — **different origin** (subdomain differs, so host differs).
  - `https://app.example.com` and `https://app.example.com:443` — **same origin** (default port).
  - The lesson lands this with a side-by-side table and a small exercise; the tuple is the load-bearing primitive of the chapter.
- **Site vs. origin**, the second distinction every SaaS engineer needs:
  - **Site** is the tuple of `(scheme, eTLD+1)` where eTLD+1 is the registrable domain — the public suffix plus one label. For `app.example.com`, the eTLD is `com`, and eTLD+1 is `example.com`. For `co.uk` (which is a public suffix maintained by the Public Suffix List), the eTLD+1 of `app.example.co.uk` is `example.co.uk`.
  - **Same-site** means scheme matches and eTLD+1 matches. `https://app.example.com` and `https://api.example.com` are **same-site** (both schemes are HTTPS, both eTLD+1 is `example.com`) but **different origin** (different hosts).
  - **Why both definitions exist.** The same-origin policy uses origin (the strict, scheme + host + port tuple) for everything related to script-level access — what JavaScript can read across documents. Cookies use site (the broader scheme + eTLD+1) for `SameSite` enforcement so a user signed in at `accounts.example.com` doesn't lose their session when they navigate to `app.example.com`. The senior reach: when you read a security spec and it says "same-origin," it means the strict tuple; when it says "same-site," it means the registrable-domain tuple. They are *not* synonyms.
  - The lesson points forward to Chapter 3.4 where the `SameSite` cookie attribute uses the site definition.
- **What the same-origin policy blocks.** Named with the canonical list a senior reaches for:
  - **Cross-origin reads** by JavaScript — `fetch`, `XMLHttpRequest`, `EventSource`, `WebSocket` (the policy applies differently per API, but the principle is the same — reads are restricted). The browser sends the request, the server responds, and the browser examines the response's CORS headers before deciding whether to expose the body, status, and headers to the calling JavaScript.
  - **Cross-origin DOM access** — a page on `app.example.com` that opens a popup on `accounts.example.com` cannot read the popup's `document` or `window` properties. The browser throws on the access attempt.
  - **Cross-origin `iframe` introspection** — same as DOM access. A page that embeds an `<iframe>` from a different origin cannot read into the iframe's document.
  - **Cross-origin storage access** — `localStorage`, `sessionStorage`, IndexedDB, and the Cache API are partitioned by origin. JavaScript on `app.example.com` cannot read storage written by `accounts.example.com`.
  - **Cross-origin Service Worker scope** — Service Workers are scoped to their registering origin (out of scope for the course; named once for recognition).
- **What the same-origin policy always allows.** The exceptions that bite if the student doesn't know them:
  - **Cross-origin writes** — submitting a form `POST` to a different origin is allowed (the legacy HTML form behavior). The browser sends cookies. This is the canonical vector for CSRF (Chapter 17.2 owns the mitigation; `SameSite` cookies are the modern primary defense, Chapter 3.4).
  - **Cross-origin embedding** — `<img>`, `<video>`, `<audio>`, `<script>`, `<link rel="stylesheet">`, `<iframe>` can load resources from any origin. The browser fetches and renders them; JavaScript on the page can't read their contents (the cross-origin image is a black box; the cross-origin script's source is invisible). The exceptions: `<script>` source is opaque to JavaScript, but the script *runs* with the embedding page's origin (which is why a malicious script tag is a serious vulnerability — Chapter 17.2's CSP owns this).
  - **Cross-origin navigation** — clicking a link to a different origin navigates the top-level browsing context with no restriction.
- **The policy in practice — what the developer sees.** When JavaScript calls `fetch('https://api.different-origin.com/data')`:
  - The browser sends the request (with or without credentials depending on the `credentials` option).
  - The server processes the request and sends a response.
  - The browser checks the response for the `Access-Control-Allow-Origin` header. If absent, or if it doesn't authorize the calling origin, the browser blocks JavaScript from reading the response. The fetch's promise rejects with a `TypeError` whose message mentions CORS.
  - The DevTools Network panel shows the request *succeeded* (status 200, content downloaded). The DevTools Console shows a CORS error. This is the most-misleading-feeling debugging experience in web development; the senior knows the pattern.
- **What CORS errors actually mean.** Three classes the student must distinguish:
  - **The request didn't have permission to be sent in the first place** — a preflight failed (Lesson 3.3.3 owns the mechanics). The DevTools shows an `OPTIONS` request that returned without the right CORS headers, followed by the real request never going out.
  - **The request went out and got a response, but the response wasn't authorized for the calling origin** — the canonical CORS error. The Network panel shows the real request with a 200. The Console shows "blocked by CORS policy."
  - **The request went out, got a response, but the response is authorized only for credentials-free reads** — a credentialed fetch (with cookies) against a server that didn't set `Access-Control-Allow-Credentials: true`. The error message names credentials specifically.
- **The same-origin policy is for the user, not the server.** The lesson's load-bearing point. The policy doesn't exist to keep the server safe — the server *received* and *processed* the cross-origin request. The policy exists to protect the **user**:
  - The user is signed in at `https://bank.example.com` with a session cookie.
  - The user visits `https://evil.example.com`.
  - `evil.example.com` runs JavaScript that calls `fetch('https://bank.example.com/api/accounts', { credentials: 'include' })`.
  - Without the same-origin policy, the browser would send the cookie (because it's a request to `bank.example.com`), the bank's server would treat it as an authenticated request, and `evil.example.com`'s JavaScript would read the user's account balance.
  - With the same-origin policy, the browser sends the request (the bank's server still processes it), but the browser refuses to expose the response to `evil.example.com`'s JavaScript. The user's data stays protected.
  - The senior consequence: a CORS-blocked request *was processed by the server*. Logging, side effects, and database writes happened. The server cannot pretend it didn't receive the request. The mitigation for unauthorized requests is server-side auth (cookies + `SameSite`, CSRF tokens, origin validation), not CORS.
- **CSRF, named for the boundary.** Cross-Site Request Forgery is the attack the same-origin policy *doesn't* prevent — the policy allows cross-origin writes (form submissions) because that's how the web's hyperlink graph works. The mitigation in 2026:
  - **`SameSite=Lax` cookies** (the modern browser default) refuse to send the cookie on cross-site POSTs entirely. This is the primary defense and the reason CSRF tokens are no longer load-bearing for typical SaaS apps. Chapter 3.4 owns the cookie attribute.
  - **CSRF tokens** for legacy/cross-site flows that need them. Chapter 17.2 owns the depth.
  - **`Sec-Fetch-Site` headers** that the browser auto-attaches and the server can check. Named for recognition.
  - The lesson states the same-origin policy doesn't fix CSRF, names the modern primary defense (`SameSite`), and points forward.
- **Origin in 2026 frameworks**, named where the student will see it:
  - **In Next.js 16 Server Components and Route Handlers**, the `Request.headers.get('origin')` returns the origin of the document that initiated the request (or `null` for same-origin navigation requests, depending on the browser). The senior reach for explicit cross-origin validation in a Route Handler: read the `Origin` header and check it against an allowlist; reject if not allowed. CORS does this declaratively; manual origin checks do it imperatively for finer control.
  - **In middleware (Chapter 5.5)**, the `Origin` header is one of the request properties available before route resolution. Useful for CORS policy decisions at the edge.
  - **In the browser**, `window.origin` returns the current page's origin as a string. `URL.parse(href).origin` (or `new URL(href).origin`) extracts the origin from any URL.
- **`Sec-Fetch-*` headers**, named in one paragraph for recognition. Modern browsers attach metadata headers to every request describing where it came from and what triggered it:
  - **`Sec-Fetch-Site`** — `same-origin`, `same-site`, `cross-site`, or `none` (for top-level navigation typed in the address bar).
  - **`Sec-Fetch-Mode`** — `cors`, `navigate`, `no-cors`, `same-origin`, `websocket`.
  - **`Sec-Fetch-Dest`** — what kind of resource is being requested (`document`, `script`, `image`, etc.).
  - The senior reach: a Route Handler can read `Sec-Fetch-Site` and reject any request that wasn't `same-origin` for endpoints that should never be called cross-site. The browser sets these unmodifiably (they're forbidden header names — JavaScript can't override them). Chapter 17.2 owns the security baseline that reads them; the lesson names the mechanism for recognition.
- **Cross-Origin-Opener-Policy (COOP) and Cross-Origin-Embedder-Policy (COEP)**, named in one paragraph. These response headers let a page opt into stricter isolation than the same-origin policy provides — preventing cross-origin windows from accessing each other's `window.opener` references (COOP) and ensuring all loaded subresources have explicitly opted into cross-origin embedding (COEP). The 2026 reality: required for SharedArrayBuffer and high-resolution timers (workloads like WebContainers, certain WebAssembly applications). The 2026 senior watch-out: COOP breaks OAuth-popup-style integrations because it severs the window-opener link the popup uses to message back; the `COOP: same-origin-allow-popups` value preserves popup interaction. The course's stack rarely needs this — named once for recognition, the depth lives in the rare project that needs SharedArrayBuffer or in Chapter 17.2's full security-baseline treatment.
- **The watch-outs a senior names**:
  - **`http://localhost:3000` and `http://localhost:4000` are different origins.** The most common dev-time CORS bite — the Next.js dev server runs on `:3000`, the student spins up an Express API on `:4000`, the fetch fails. The fix is either to proxy the API through Next.js (route the call through a Route Handler on `:3000` that fetches the upstream), to configure CORS on the API, or to use a shared dev port via a reverse proxy. The senior preference in 2026: proxy through Next.js Route Handlers so the production architecture is the same.
  - **`http` and `https` are different origins.** `http://localhost:3000` and `https://localhost:3000` (after `mkcert`) are different origins. Cookies set on one don't apply to the other; CORS-blocked between them.
  - **Subdomains are different origins by default.** `https://app.example.com` and `https://api.example.com` are different origins, even though they're same-site. CORS must be configured on the API host for the app host to fetch with credentials.
  - **The browser's CORS error message hides in the Console, not the Network panel.** The Network row shows a successful response — students new to the model assume the request "worked." The Console is where the CORS rejection is logged. The senior reflex: when a fetch fails inexplicably, open the Console first.
  - **`mode: 'no-cors'`** on a `fetch` call is *not* a way to bypass CORS — it tells the browser "I want an opaque response I can't read." Useful for fire-and-forget pings; useless for reading data. Named for recognition.
  - **`window.postMessage`** is the cross-origin escape hatch for explicit message-passing between windows or iframes. The receiver must validate the `event.origin` against an allowlist — otherwise any origin can post messages it'll process. The senior reflex: `postMessage` is the right tool for embedded widgets that need to talk to their parent; always validate the origin. Out of scope for the course at depth; named once.
  - **The address bar's origin is what cookies and CORS use, not the page's `<base href>`.** A page at `https://app.example.com` with a `<base href="https://other.example.com/">` still belongs to the `app.example.com` origin. The browser uses the document's URL, not the base href, for origin decisions.

What this lesson does not cover:

- The URL spec and `URLSearchParams` (3.3.1).
- CORS mechanics — preflights, response headers, credentials, the failure modes (3.3.3).
- Cookies — `Secure`, `HttpOnly`, `SameSite`, scope (Chapter 3.4).
- CSRF tokens, the security baseline, COOP/COEP at depth (Chapter 17.2).
- The Fetch API surface — `Request`, `Response`, the `credentials` option (Chapter 3.6.1).
- `window.postMessage` and the embedded-widget messaging pattern — out of scope for the course's primary surface.
- Service Workers and their scope rules — out of scope.
- The full storage partitioning model in 2026 browsers (third-party cookie deprecation, Storage Access API) — out of scope; named once.
- The Public Suffix List's update cadence and edge cases — out of scope.

Pedagogical approach:

Concept archetype. The lesson teaches a mental model — origin as the unit of browser trust — that the next lesson's CORS mechanics land on. Open with the senior question — "evil.com runs JavaScript that fetches bank.com with the user's cookie; what does the browser do?" — and let the student predict before the lesson resolves it.

A `Figure` with a hand-authored SVG renders the origin tuple as three boxes (scheme, host, port) with a label "origin" wrapping all three. A second row below shows the site tuple as two boxes (scheme, eTLD+1) with a label "site." A side annotation calls out that `https://app.example.com` and `https://api.example.com` are different origins (host differs) but same site (eTLD+1 matches).

A small `Tokens` or `Matching` exercise: given five URL pairs (`https://app.example.com` vs. `https://app.example.com/dashboard`, `https://app.example.com` vs. `http://app.example.com`, `https://app.example.com` vs. `https://api.example.com`, `https://app.example.com:443` vs. `https://app.example.com`, `https://app.example.com` vs. `https://example.com`), the student classifies each as same-origin, same-site-only, or cross-site. The discrimination locks in the tuple definitions.

A second `Figure` with a Mermaid sequence diagram walks the "evil.com fetches bank.com" scenario: the JavaScript call, the browser sending the request (with cookie), the server processing the request and responding, the browser checking the `Access-Control-Allow-Origin` header on the response, the browser blocking JavaScript from reading the body, the `fetch` promise rejecting. The annotation calls out the load-bearing point: **the request was processed by the server; CORS does not protect the server from the request — it protects the user from the response leak**.

An interactive widget (a small toggle-driven simulator) shows three scenarios side by side: tab 1 is "same-origin fetch" (request sent with cookies, response readable); tab 2 is "cross-origin fetch without CORS" (request sent with or without cookies depending on the `credentials` option, response unreadable, console error); tab 3 is "cross-origin fetch with proper CORS" (request sent, response readable). Each tab shows the Network panel state and the Console state. The model is tangible.

A `Buckets` exercise sorts eight cross-origin operations into "browser blocks" or "browser allows" — JavaScript fetching cross-origin and reading the response, a form POST to a different origin, embedding a cross-origin image, reading the `document` of a cross-origin iframe, reading localStorage written by another origin, navigating the top window to a different origin, loading a cross-origin script tag, reading the cross-origin script tag's source. The recognition discipline locks in.

A `CodeReview` exercise on a 25-line Next.js middleware file with three same-origin-policy bugs: a `postMessage` handler that doesn't validate `event.origin` (any origin can send commands), a server-side handler that trusts the `Origin` request header for authorization without validating it (the header can be set arbitrarily by non-browser clients), and a fetch call from the client that includes `credentials: 'include'` to a third-party origin (will fail in production; the student is expected to recognize and either remove credentials or proxy through Next.js). The student leaves a comment per issue.

A `MultipleChoice` exercise on the load-bearing point: given the scenario "a fetch from evil.com to your API was blocked by CORS," pick what is true (the request was processed by your server — right; the request was rejected before reaching your server — wrong; CORS protected your server from the request — wrong; CORS protected the user from the response being exposed — right). The discrimination is the deliverable.

A `TrueFalse` round of five statements on origin and same-origin: "Subdomains of the same domain are the same origin" (false — same site, different origin), "An HTTP and HTTPS version of the same host are the same origin" (false — scheme matters), "The same-origin policy prevents the server from receiving cross-origin requests" (false — the request still arrives; the policy blocks the response from being read), "`SameSite=Lax` cookies are the modern primary CSRF defense" (true), "`Sec-Fetch-Site` is settable by the page's JavaScript" (false — it's a forbidden header name; only the browser sets it). The vocabulary is locked in.

Close with a `PredictOutput` exercise: given three fetch calls from `https://app.example.com` to (a) `https://app.example.com/api/data`, (b) `https://api.example.com/data` without CORS headers, (c) `https://api.example.com/data` with `Access-Control-Allow-Origin: https://app.example.com`, the student predicts which succeed and which fail. The CORS mechanics get installed in the next lesson; this prepares the model.

Estimated student time: 45 to 55 minutes. Load-bearing for Lesson 3.3.3, Chapter 3.4, and Chapter 17.2.

## Lesson 3.3.3 — The preflight dance

Teaches CORS as the opt-in that loosens same-origin, the simple-vs-preflighted decision, the `Access-Control-Allow-*` response-header palette, the wildcard-with-credentials trap, and the canonical browser error messages with their fixes.

Topics to cover:

- The senior question: the same-origin policy blocks cross-origin reads by default. The application has a legitimate cross-origin scenario — the SaaS frontend at `https://app.example.com` needs to call its own API at `https://api.example.com`, or a partner integration needs to call the SaaS API from the partner's origin. CORS is the server-side mechanism that opts in to specific cross-origin requests. The lesson installs the CORS protocol at the depth the student needs to author a CORS-enabled endpoint, recognize a preflight on the Network panel, and debug the canonical failure modes (the wildcard-with-credentials trap, the missing preflight handler, the disallowed header).
- **CORS as the opt-in mechanism.** The browser checks every cross-origin response for CORS headers and uses them to decide whether to expose the response to the calling JavaScript. The server controls authorization by setting (or omitting) the right headers. The two-line summary: **the server says which origins it trusts; the browser enforces it on the client side.**
- **The simple-request vs. preflighted-request distinction.** The browser categorizes cross-origin requests into two buckets:
  - **Simple requests** — the request can be sent directly; the browser checks the response's CORS headers when it arrives. A request is "simple" if and only if it meets *all* of these conditions:
    - Method is `GET`, `HEAD`, or `POST`.
    - Only headers the browser sets automatically, plus a short whitelist of headers the application is allowed to set without triggering a preflight: `Accept`, `Accept-Language`, `Content-Language`, `Content-Type` (restricted to three values — see below), and a few `Range` cases. Custom headers (`Authorization`, `X-Tenant-Id`, `Idempotency-Key`, etc.) are *not* in the whitelist.
    - `Content-Type`, if present, is one of `text/plain`, `application/x-www-form-urlencoded`, or `multipart/form-data`. `application/json` is *not* in this list — sending JSON triggers a preflight. The 2026 SaaS reality: almost every API request triggers a preflight because either the `Content-Type` is `application/json` or there's an `Authorization` header or both.
  - **Preflighted requests** — the browser sends an `OPTIONS` request to the same URL first, asking the server whether the real request is allowed. The server responds with the CORS headers describing what it allows; the browser then sends the real request only if the response approves it. The senior reach: nearly every fetch from a SaaS frontend to a cross-origin API is preflighted.
- **The preflight `OPTIONS` request**, walked in detail:
  - The browser sends `OPTIONS /endpoint HTTP/1.1` with:
    - `Origin: https://app.example.com` — the requesting origin.
    - `Access-Control-Request-Method: POST` — the method the real request will use.
    - `Access-Control-Request-Headers: content-type, authorization` — the headers the real request will set (lowercase, comma-separated).
  - The server responds with `204 No Content` (or `200 OK`) and the CORS headers:
    - `Access-Control-Allow-Origin: https://app.example.com` (must echo the requesting origin, or be `*` for credential-free requests).
    - `Access-Control-Allow-Methods: POST, GET, DELETE` (or whatever the server allows).
    - `Access-Control-Allow-Headers: Content-Type, Authorization` (echoing the requested headers).
    - `Access-Control-Max-Age: 86400` (how many seconds the browser can cache this preflight result; up to 7200 in Chrome, 86400 in Firefox, 600 in Safari — the browsers cap it).
    - `Access-Control-Allow-Credentials: true` (only if the real request will include cookies; see below).
  - The browser caches the preflight result (per origin, per URL, per method) for the `Max-Age` duration and skips the preflight for subsequent requests within that window.
- **The real request after preflight passes.** The browser sends the actual request with `Origin: https://app.example.com` and any custom headers. The server responds with the body *and* with `Access-Control-Allow-Origin` (and, if credentials, `Access-Control-Allow-Credentials: true`) on the *response*. Missing or wrong headers on the response cause the browser to block JavaScript from reading the body, even though the preflight passed. The senior watch-out: CORS headers must be set on *both* the preflight `OPTIONS` response *and* the real request's response.
- **The CORS response headers**, named with their canonical values:
  - **`Access-Control-Allow-Origin`** — the single origin to allow, or `*` for any origin. The most-misconfigured CORS header. Two valid forms:
    - `Access-Control-Allow-Origin: https://app.example.com` — explicit origin. Required for credentialed requests. The server typically reads the `Origin` request header, checks it against an allowlist, and echoes it back. Setting `Vary: Origin` on the response is required so the CDN doesn't serve the wrong origin's response to another origin.
    - `Access-Control-Allow-Origin: *` — wildcard. Allowed only for credential-free requests. Useful for public APIs. The browser refuses to send cookies if `*` is set, even if `Access-Control-Allow-Credentials: true` is also set (the wildcard-with-credentials trap, see below).
  - **`Access-Control-Allow-Credentials`** — `true` if the server allows the request to include cookies, `Authorization` header, or TLS client certificates. The header is only meaningful in the value `true` (lowercase, no other values are valid). Required on *both* the preflight response and the real-request response when credentials are involved. The browser's `fetch` must also opt in with `credentials: 'include'` for cookies to be sent.
  - **`Access-Control-Allow-Methods`** — comma-separated list of allowed methods. Sent on the preflight response. The browser caches this with the preflight result and won't preflight again for these methods.
  - **`Access-Control-Allow-Headers`** — comma-separated list of allowed request headers. Sent on the preflight response. Custom headers the real request will set must be listed here.
  - **`Access-Control-Expose-Headers`** — comma-separated list of response headers the browser exposes to JavaScript. By default, JavaScript can read only a short whitelist of "CORS-safelisted response headers" (`Cache-Control`, `Content-Language`, `Content-Type`, `Expires`, `Last-Modified`, `Pragma`). Custom response headers (like `X-RateLimit-Remaining`, `Request-Id`) must be in this list for `response.headers.get(...)` on the client to return them.
  - **`Access-Control-Max-Age`** — how long (in seconds) the browser can cache the preflight result. The senior reach: set this high (an hour or a day) so most requests skip the preflight.
- **The wildcard-with-credentials trap.** The single most-common CORS bug. Setting `Access-Control-Allow-Origin: *` and `Access-Control-Allow-Credentials: true` together does *not* work. The browser refuses the credentialed request and logs an error. The fix is to echo the specific origin: read the `Origin` request header, validate it against an allowlist, and set `Access-Control-Allow-Origin` to that exact origin (with `Vary: Origin` to keep the CDN from poisoning the cache). The senior reach for SaaS APIs that accept credentials: a tiny middleware that reads `Origin`, checks against `process.env.ALLOWED_ORIGINS`, and echoes if allowed.
- **CORS in Next.js 16**, named at the framework level:
  - **The 2026 reality: first-party API endpoints are same-origin.** Next.js renders the UI and serves Route Handlers on the same origin by default. The student building the course's reference SaaS never hits CORS for their own API. The first-party calls go through Server Actions (which are post-and-redirect, no CORS concern) or Route Handlers on the same origin.
  - **When CORS shows up.** Three scenarios:
    1. The dev loop spans ports — Next.js on `:3000`, a sidecar API on `:4000`. The senior preference: proxy through a Next.js Route Handler so production is single-origin.
    2. The SaaS exposes a public API for partner integrations. The Route Handler sets the right CORS headers for the allowed partner origins.
    3. The frontend consumes a third-party browser-callable API (rare in 2026 — most third-party APIs are server-to-server). The senior preference: proxy through a Next.js Route Handler so the third-party API key stays server-side.
  - **The implementation.** In a Route Handler, an `OPTIONS` export handles the preflight; the `GET`/`POST`/etc. exports set CORS headers on the response. The pattern is small enough to write inline; for shared use, a tiny `withCors` helper applies the headers from an allowlist. Middleware (Chapter 5.5) is the other place; CORS headers can be set globally before route resolution.
  - **The `next.config.js` `headers` option** can set CORS headers declaratively for a path pattern. Useful for simple cases (all routes under `/api/public/*` get the same CORS policy). Named for recognition.
- **Reading a CORS interaction on DevTools**, the recognition recipe:
  - In the Network panel, a preflighted request appears as **two rows** for the same URL: an `OPTIONS` row (the preflight) and the real request (`POST`, `GET`, etc.) below it.
  - The preflight row's request headers show `Access-Control-Request-Method` and `Access-Control-Request-Headers` (and `Origin`).
  - The preflight row's response headers show the `Access-Control-Allow-*` headers and the status (204 or 200).
  - The real request's response headers also show `Access-Control-Allow-Origin` (and possibly `Access-Control-Allow-Credentials`).
  - The Console shows a CORS error message if either response fails the check.
  - The senior reflex: filter the Network panel by `OPTIONS` to see all the preflights; check the response headers row by row.
- **The canonical CORS failure modes**, each with the recognition signal and the fix:
  - **"No 'Access-Control-Allow-Origin' header is present on the requested resource."** The server didn't set the header (or set it to a different origin). The fix: configure CORS on the server to allow the calling origin.
  - **"The value of the 'Access-Control-Allow-Origin' header in the response must not be the wildcard '*' when the request's credentials mode is 'include'."** The wildcard-with-credentials trap. The fix: echo the specific origin from the `Origin` request header.
  - **"Request header field X is not allowed by Access-Control-Allow-Headers in preflight response."** The real request will send a custom header the preflight didn't approve. The fix: add the header to `Access-Control-Allow-Headers`.
  - **"Method X is not allowed by Access-Control-Allow-Methods in preflight response."** The real request's method wasn't approved. The fix: add the method to `Access-Control-Allow-Methods`.
  - **"Response to preflight request doesn't pass access control check: It does not have HTTP ok status."** The preflight `OPTIONS` request returned a 4xx or 5xx. Often because the framework's route doesn't have an `OPTIONS` handler. The fix: add an `OPTIONS` export to the Route Handler.
  - **"Credentials flag is 'true', but the 'Access-Control-Allow-Credentials' header is not 'true' or is missing."** The fetch sent `credentials: 'include'` but the server didn't set the credentials header. The fix: set `Access-Control-Allow-Credentials: true` on both the preflight and real responses.
- **The `fetch` client-side surface**, named with the credential options:
  - **`credentials: 'same-origin'`** (the default) — send cookies for same-origin requests, omit for cross-origin.
  - **`credentials: 'include'`** — send cookies for all requests, including cross-origin. Requires the server's CORS to set `Access-Control-Allow-Credentials: true` and a specific (non-wildcard) `Access-Control-Allow-Origin`.
  - **`credentials: 'omit'`** — never send cookies. Useful for public-API calls where credentials would be pointless.
  - **`mode: 'cors'`** (the default for cross-origin) — apply CORS rules.
  - **`mode: 'no-cors'`** — receive an opaque response (status 0, no body access). Not a CORS bypass; useful only for fire-and-forget cross-origin pings.
- **CORS does not protect against**, named for the boundary:
  - **CSRF on `SameSite=Lax`-or-stronger cookies.** The cookie isn't sent on cross-site POSTs at all. CORS isn't relevant; `SameSite` is the defense (Chapter 3.4).
  - **Server-side abuse.** A non-browser client (`curl`, a Python script, a Go service) sends whatever headers it wants. `Origin` can be spoofed; `Sec-Fetch-Site` cannot (browsers refuse to let JavaScript set it, but non-browser clients omit it entirely). The server's authorization model — auth tokens, signed requests, IP allowlists — is the real defense.
  - **API scraping.** The CORS error only fires in browsers; a script that scrapes the API from a server has no CORS layer to defeat.
  - **The same-origin policy itself.** CORS *relaxes* the policy; it doesn't add new restrictions. A page can still embed cross-origin images, submit cross-origin forms, and navigate to cross-origin URLs without CORS being involved.
- **Private Network Access**, named as a 2026 reality. Chrome enforces an additional preflight when a public website tries to fetch a resource on a private network (the user's home router at `192.168.1.1`, a corporate intranet). The preflight requires `Access-Control-Allow-Private-Network: true`. The course's stack rarely hits this; named once for recognition because the error message is cryptic and the student will Google it the first time it happens (a "blocked by CORS" message that has nothing to do with regular CORS configuration). Out of scope for application code.
- **The watch-outs a senior names**:
  - **Set `Vary: Origin` whenever you echo `Access-Control-Allow-Origin`.** Without it, a CDN may cache the response and serve it to a different origin, breaking CORS for everyone. The framework usually handles this; the senior verifies it's set in the response headers.
  - **CORS is per-origin, not per-path.** A preflight for `/api/users` doesn't approve `/api/invoices`. The browser caches per-URL (and per-method). The Max-Age cap reduces but doesn't eliminate the preflight cost.
  - **A 401 on a CORS-protected endpoint is a CORS-shaped error in the browser.** The 401 response must still set the CORS headers, or the browser strips them and JavaScript sees a CORS error rather than a 401. The senior reflex: set CORS headers on *all* responses, including error responses, when the route is CORS-enabled.
  - **Cookies sent on credentialed cross-origin requests must have `SameSite=None; Secure`.** A cookie with `SameSite=Lax` (the modern default) is *not* sent on cross-origin fetches. The senior consequence: cross-origin authenticated APIs need `SameSite=None` cookies, which require `Secure`, which requires HTTPS. Chapter 3.4 owns the cookie attributes.
  - **`OPTIONS` should not require auth.** The browser doesn't include `Authorization` headers or cookies on the preflight (browsers vary, but the safe assumption is "no auth on OPTIONS"). The server's `OPTIONS` handler must respond with 204 and the CORS headers without requiring authentication; otherwise the preflight 401s and the real request never goes out.
  - **`fetch` does not retry CORS failures.** A CORS error is a `TypeError`, distinct from HTTP errors (which resolve with `response.ok = false`). The senior reach: client-side fetch wrappers should distinguish — a `TypeError` is a CORS or network failure; an HTTP 4xx/5xx is a server response that succeeded the CORS check.
  - **Misconfigured CORS hides real bugs.** If a CORS error masks every fetch failure, the developer reaches for "fix CORS" before debugging the real failure (the server is down, the URL is wrong, the auth token is invalid). The senior reflex: confirm the response actually arrived at the server (check server logs) before concluding the failure is CORS.

What this lesson does not cover:

- The URL spec and `URLSearchParams` (3.3.1).
- The same-origin policy as the default trust boundary (3.3.2).
- Cookie attributes including `SameSite=None; Secure` requirements (Chapter 3.4).
- The Fetch API's `Request` and `Response` objects, the `credentials` and `mode` options at depth (Chapter 3.6.1).
- Next.js Route Handler authoring conventions, the `OPTIONS` export pattern (Chapter 7.5).
- Next.js middleware (Chapter 5.5).
- CSRF protection at the security baseline (Chapter 17.2).
- WebSocket cross-origin behavior — out of scope.
- gRPC-Web's CORS conventions — out of scope.
- `Access-Control-Request-Private-Network` and the full Private Network Access flow at depth — out of scope.

Pedagogical approach:

Mechanics archetype with a strong reference beat. The lesson teaches a specific protocol (request, preflight, response) and the senior recipe for reading and authoring CORS. Open with the senior question — "you wired your Next.js dev server to fetch a sidecar API on a different port and got a CORS error; walk me through what the browser actually did" — and a `MultipleChoice` exercise pitting four explanations of the error (the request never reached the server — wrong; the preflight failed because the server didn't handle OPTIONS — right; the server returned 401 — wrong; the framework's CORS is misconfigured — partially right but not specific). The discrimination installs the preflight as the first failure point.

A `Figure` with a Mermaid sequence diagram walks the full preflight dance: the browser's JavaScript calls `fetch`; the browser detects the cross-origin + non-simple request; the browser sends an `OPTIONS` request with `Access-Control-Request-Method` and `Access-Control-Request-Headers`; the server responds with `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`, `Access-Control-Max-Age`; the browser caches and sends the real request; the server responds with the body and `Access-Control-Allow-Origin`; the browser exposes the response to JavaScript. The diagram is the lesson's anchor for the rest of the prose.

A `TabbedContent` block organizes the CORS response headers as a reference card. Tab 1 is `Access-Control-Allow-Origin` (the specific-vs-wildcard distinction). Tab 2 is `Access-Control-Allow-Credentials` (the credentials toggle and the wildcard trap). Tab 3 is `Access-Control-Allow-Methods` and `Access-Control-Allow-Headers` (the preflight-approval pair). Tab 4 is `Access-Control-Expose-Headers` (the rarely-used but load-bearing reverse-direction header). Tab 5 is `Access-Control-Max-Age` (the preflight cache duration). Each tab has the canonical value and a watch-out.

An `AnnotatedCode` block walks a minimal Next.js 16 Route Handler with a `GET` and an `OPTIONS` export, plus a helper that builds the CORS response headers from an origin allowlist and the request's `Origin` header. The annotations highlight the four pieces: the allowlist check, the echo of the origin (not wildcard, for credentials), the `Vary: Origin` for CDN safety, and the `OPTIONS` handler returning 204 with the same headers. The senior pattern is concrete.

A second `AnnotatedCode` block walks the client-side `fetch` call that triggers the preflight — `fetch(url, { method: 'POST', credentials: 'include', headers: { 'Content-Type': 'application/json', 'Authorization': 'Bearer ...' }, body: JSON.stringify(...) })`. The annotations identify why each property triggers the preflight: `POST` is simple by method, but the `Content-Type: application/json` and the `Authorization` header push it into preflight territory; `credentials: 'include'` requires the server's CORS to set `Access-Control-Allow-Credentials: true` and a specific origin.

A `CodeVariants` block contrasts three CORS misconfigurations against the senior fix. Tab 1: `Access-Control-Allow-Origin: *` with `Access-Control-Allow-Credentials: true` — the wildcard-with-credentials trap, browser refuses. Tab 2: CORS headers set on the `GET` handler but no `OPTIONS` handler — the preflight 404s, real request never sent. Tab 3: CORS headers on the success response but not on the 401 — the browser sees a CORS error, not a 401. Tab 4 is the fix — echo origin from allowlist, `Vary: Origin`, separate `OPTIONS` handler, CORS headers on every response including errors. The annotations name the failure mode and the senior fix per tab.

A `Buckets` exercise sorts ten `fetch` scenarios into "simple (no preflight)" or "preflighted" — a `GET` to a different origin with no custom headers, a `POST` with `Content-Type: application/x-www-form-urlencoded`, a `POST` with `Content-Type: application/json`, a `GET` with an `Authorization` header, a `DELETE` request, a `POST` with no body, a `GET` with `credentials: 'include'` and no custom headers (simple — credentials don't force a preflight on their own), a `POST` with a custom `X-Tenant-Id` header. The recognition discipline is concrete.

A `CodeReview` exercise on a 40-line CORS middleware with five issues: hardcoded wildcard `Access-Control-Allow-Origin: '*'` with `credentials: true` (wildcard-with-credentials trap), missing `Vary: Origin` (CDN cache poisoning), no `OPTIONS` handler (preflights 404), CORS headers only on success responses (errors break in browser), `Access-Control-Allow-Headers` missing the `Idempotency-Key` header the real request sends (preflight blocks it). The student leaves a comment per issue with the senior fix.

A `Matching` exercise pairs five CORS error messages with their root cause and fix — "No 'Access-Control-Allow-Origin' header is present" (server didn't set it; add header), "the wildcard '*' when credentials mode is 'include'" (wildcard-with-credentials trap; echo specific origin), "header field X is not allowed by Access-Control-Allow-Headers" (preflight didn't approve the header; add it to the preflight response), "Response to preflight does not have HTTP ok status" (no OPTIONS handler; add one), "Credentials flag is 'true', but the 'Access-Control-Allow-Credentials' header is not 'true'" (server didn't set the credentials header; add it). The error-message vocabulary is locked in.

A short setup/wiring walkthrough has the student build a tiny demo: spin up a Next.js Route Handler at `/api/echo` that returns `{ origin: request.headers.get('origin') }` with CORS headers for `http://localhost:3000`, then open a separate page on `http://localhost:4000` (or use a different port) and call the endpoint with `fetch`. The student observes the preflight in DevTools, swaps the allowlist to break it, observes the CORS error in Console, fixes it, and confirms the response is readable. Five minutes; the recognition pattern is hands-on.

Close with a `TrueFalse` round of five statements on CORS: "A CORS error means the server rejected the request" (false — server processed it; browser blocked the response from JavaScript), "Setting `Access-Control-Allow-Origin: *` works with credentialed requests" (false — wildcard-with-credentials trap), "A `GET` request with `Content-Type: application/json` triggers a preflight" (true — JSON content-type isn't in the simple list), "An `OPTIONS` preflight is sent before every cross-origin request" (false — simple requests skip it), "Setting `Access-Control-Max-Age: 86400` caches the preflight for 24 hours" (true on Firefox; Chrome caps at 2 hours, Safari at 10 minutes). The vocabulary is locked in.

Estimated student time: 50 to 60 minutes. Load-bearing for Chapter 3.6 (Fetch), Chapter 7.5 (Route Handlers), and Unit 9 (auth across origins).

## Lesson 3.3.4 — Quizz

Top ten topics to quiz:

1. The URL anatomy — scheme, host, port, pathname, search, hash, origin — and what origin includes (scheme + host + port, not path).
2. `new URL(path, base)` and `URLSearchParams` as the senior reach for parsing and building URLs; the bug class string concatenation produces (wrong encoding, double slashes, path traversal, scheme injection, open redirects).
3. The percent-encoding split — pathname uses `%20` for space, query string (`URLSearchParams`) uses `+` for space; `URLSearchParams` uses the `application/x-www-form-urlencoded` encoding set.
4. Origin defined as `(scheme, host, port)`; site defined as `(scheme, eTLD+1)`. Subdomains are same-site but different-origin. `https://` and `http://` versions are different origins.
5. What the same-origin policy blocks (cross-origin reads by JavaScript, DOM access, storage access) vs. what it always allows (cross-origin writes via form POST, cross-origin embedding via `<img>`/`<script>`/`<iframe>`, cross-origin navigation).
6. CORS does not protect the server — the request still hits the server and is processed; CORS protects the user by blocking the response from being exposed to the calling origin's JavaScript. The mitigation for unauthorized requests is server-side auth (cookies + `SameSite`, signed requests), not CORS.
7. The simple-vs-preflighted decision — simple requests are GET/HEAD/POST with no custom headers and a content-type in `{text/plain, application/x-www-form-urlencoded, multipart/form-data}`. Almost every SaaS API request (JSON body or auth header) is preflighted.
8. The CORS preflight `OPTIONS` request and the response-header palette — `Access-Control-Allow-Origin`, `Access-Control-Allow-Credentials`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`, `Access-Control-Max-Age`, `Access-Control-Expose-Headers`.
9. The wildcard-with-credentials trap — `Access-Control-Allow-Origin: *` and `Access-Control-Allow-Credentials: true` together don't work. The fix is to echo the specific origin from the `Origin` request header with `Vary: Origin` on the response.
10. The canonical CORS failure modes and their console messages — missing `Access-Control-Allow-Origin`, wildcard-with-credentials, disallowed header in preflight, missing `OPTIONS` handler (preflight non-2xx), missing `Access-Control-Allow-Credentials` on a credentialed request, CORS headers missing on error responses (the 401 that looks like a CORS error).
