# Chapter 22.1 — Docs that live next to the truth

## Chapter framing

Documentation is part of shipping; a feature without docs is a feature one promotion or one quarter away from being unmaintainable. This chapter teaches the four-document surface a 2026 SaaS codebase actually maintains: a thin README for first contact, an `AGENTS.md` for coding agents and humans skimming for conventions, an `/docs/adr/` log of one-decision-per-file architectural decisions in Nygard format, and the schema file as the data-model reference. The chapter's job is to lock down *which* artifact owns *which* job so they don't sprawl, drift, or duplicate — and to give the student the Diataxis vocabulary that explains why each artifact has the shape it does.

Threads through every lesson: docs live next to the truth they describe — the schema file IS the data model doc, the Server Action signature IS the API doc, the env vars in `process.env` access sites point at the env entry that owns them; every doc has exactly one audience and one job (first contact, agent context, decision history, reference); the senior reflex is to ask "could this paragraph be a link to the canonical source instead?" before writing it; documentation discipline is structural, not aspirational — the doc that lives in the same PR as the code change stays accurate, the doc that lives in a wiki rots; Diataxis (tutorial / how-to / reference / explanation) is the vocabulary that names what each artifact is for, and a doc that mixes types is the doc nobody reads; the architectural decisions this course already made (Drizzle, Better Auth, Biome, R2, Node runtime, native forms) are the worked examples of "what earns an ADR," so the student sees the form on real choices instead of toy ones.

## Lesson 22.1.1 — Diataxis: the four jobs a doc can do

Learn the Diataxis vocabulary (tutorial, how-to, reference, explanation), the 2x2 axes behind it, the "mixing trap" that wrecks most READMEs, where each type lives in a 2026 SaaS repo, and the "could this be a link?" reflex.

Topics to cover:

- **The senior question.** Why do docs rot, why do READMEs balloon into novels nobody reads, and why is "we have docs" not the same as "we have working docs"? Because the team writes one document trying to do four different jobs at once. Diataxis names the four jobs, and the rest of the chapter rests on the vocabulary it gives you.
- **The four document types — one definition each.**
  - *Tutorial.* Learning-oriented. The reader is a stranger to the codebase; the doc takes them by the hand through one path to a working end state. The course itself is a tutorial. In a SaaS repo: the "first time setup" section of the README, *if* it exists, and nothing else.
  - *How-to guide.* Task-oriented. The reader already knows the codebase and wants to accomplish one specific thing: "how to add a new Server Action," "how to seed a tenant for local dev," "how to rotate the Stripe webhook secret." Lives in `/docs/how-to/` or in the README's "common tasks" section.
  - *Reference.* Information-oriented. Dry, complete, alphabetically or structurally organized; describes the API surface as it is. The Drizzle schema file is reference. TSDoc on a public function is reference. The env var list is reference.
  - *Explanation.* Understanding-oriented. Background, rationale, the *why*. ADRs are explanation. A `CONTRIBUTING.md` note on "why we chose Server Actions over tRPC" is explanation.
- **The two axes — what the framework actually is.** Diataxis arranges the four on a 2x2: *acquisition vs. application* (learning vs. doing) and *action vs. cognition* (practical steps vs. theoretical knowledge). Tutorials = acquisition+action. How-to = application+action. Reference = application+cognition. Explanation = acquisition+cognition. The diagram is worth one mermaid; the senior win is recognizing that a doc fighting itself usually mixes two quadrants.
- **The mixing trap — why most READMEs fail.** A typical README opens with a tutorial ("install this, run that"), then a how-to ("to add a new feature, do X"), then reference ("env vars are…"), then explanation ("we chose Postgres because…"). Each section pulls in a different direction; the reader who came for one job has to scan past three others. The senior call is to put each type in its right home — keep the README a tutorial-and-pointer document, send reference to the schema and the TSDoc, send explanation to ADRs, send how-to to either a `/docs/how-to/` folder or to dedicated AGENTS.md sections.
- **Where each Diataxis type lives in a 2026 SaaS repo — the map.** Tutorial → `README.md` "Getting started" section only. How-to → `AGENTS.md` "Common tasks" and `/docs/how-to/` for the heavyweight ones. Reference → the code itself (Drizzle schema, Zod schemas, TSDoc on public functions, the `env.ts` file). Explanation → `/docs/adr/` for architectural decisions, inline `// why` comments for narrow rationale. Each subsequent lesson in this chapter is one cell of this table; name them inline so the chapter shape is visible.
- **The "could this be a link?" reflex.** Whenever a doc-writer is about to paraphrase canonical truth (a function signature, a schema column list, a config value), the senior reflex is to link to the canonical source instead. Paraphrased docs go stale; linked docs stay correct because the linked source is the source of truth. The reflex is the structural enforcement that keeps docs aligned with code.
- **What "good" looks like for each type — the quality test per quadrant.** A tutorial works if a new hire follows it to a green checkmark in under thirty minutes without asking anyone. A how-to works if the reader doing the task can scan-find the answer in under a minute. A reference works if a senior debugging at 2am can find the exact answer without reading anything they don't need. An explanation works if a future maintainer reading it understands the *trade-off* well enough to decide whether the decision still holds under new constraints. Different quality bars; the lesson states each one.
- **Watch-outs:** treating Diataxis as four mandatory folders the project must have on day one — most SaaS repos need a `README.md`, an `AGENTS.md`, and `/docs/adr/` and nothing else; the four-type vocabulary is for *thinking*, not folder structure; trying to write one document that's all four types ("comprehensive docs") — readers bounce; deleting reference docs because "the code is self-documenting" — the schema is self-documenting, a complex Server Action's TSDoc isn't optional; writing explanation docs ("our philosophy on testing") before any reference docs exist — philosophy without artifacts is decoration.

What this lesson does not cover:

- README structure and contents — Lesson 22.1.2.
- AGENTS.md structure and contents — Lesson 22.1.3.
- ADR template and rationale — Lesson 22.1.4.
- TSDoc syntax and where to put it — Chapter 22.2.
- Wiki/Notion/Confluence docs — the course's stance is that durable docs live in the repo; out-of-repo docs are out of scope.

Estimated student time: 25 to 30 minutes. The chapter's framing lesson — short, vocabulary-only, every following lesson references it.

## Lesson 22.1.2 — The thin README and source-as-doc

Learn the README's one job (first contact), its five-section template, the "what does NOT belong" list, and the rule that the schema file, `env.ts`, and Server Action signatures ARE the docs for what they describe so the README links instead of duplicating.

Topics to cover:

- **The senior question.** What goes in the README, what doesn't, and where does everything that *doesn't* go? The answer dictates two things at once: a thin README that stays accurate because it's short, and a discipline of letting canonical artifacts (the schema, the `env.ts`, the Server Action file) be the doc for what they describe.
- **The README's one job — first contact.** The README is read by exactly two audiences: a new contributor in their first hour, and a recruiter/reviewer looking at the repo on GitHub. Both want the same five things and nothing else: what this project is (one paragraph), how to run it locally (three to seven shell commands), how to run the tests, where the docs live (link to `AGENTS.md`, link to `/docs/adr/`), and the license. Anything else belongs in another file.
- **The five-section structure — the template.**
  - *Title and one-paragraph description.* "A multi-tenant invoice management SaaS built on Next.js 16, Postgres, and Stripe billing." One sentence on what; one sentence on stack; done.
  - *Getting started.* The minimum copy-paste sequence to a running dev server: clone, `pnpm install`, `cp .env.example .env.local`, `pnpm db:push`, `pnpm db:seed`, `pnpm dev`. Each command on its own line; no narration between them; the user trusts the README more if it's short.
  - *Common tasks.* A bulleted list of the four or five tasks done weekly with their commands: run tests (`pnpm test`), run a single test (`pnpm test path/to/file`), run a migration (`pnpm db:generate && pnpm db:migrate`), seed (`pnpm db:seed`), reset the DB (`pnpm db:reset`). Not every task — the four or five that come up most.
  - *Where the docs live.* Links, not duplication. "Conventions for working in this codebase: `AGENTS.md`. Architectural decisions: `/docs/adr/`. Data model: `src/db/schema.ts`. Environment variables: `src/env.ts`." Five links; that's the section.
  - *License.* One line.
- **The "what does NOT belong in the README" list.** API reference, full env var list with descriptions, contribution guidelines (those go to `AGENTS.md` or `CONTRIBUTING.md`), philosophy/architecture explanation (ADRs), changelog (a `CHANGELOG.md` or release notes on the platform), team conventions (AGENTS.md), deployment notes (a `/docs/how-to/deploy.md` or runbook), screenshots beyond one or two for context. Each addition past first-contact dilutes the README's only job.
- **The badges question — when one or two earns its weight.** A CI status badge and a license badge are useful at a glance for the recruiter audience; everything else (test coverage percentages, dependency versions, last commit) is noise. The senior call: zero or two, never more.
- **Docs live next to the truth — the rule.** The schema file (`src/db/schema.ts`) IS the data-model documentation. The `env.ts` file (validated with Zod, Chapter 21.3) IS the env var documentation. The Server Action file with TSDoc on the action function IS the API documentation for that action. The Stripe webhook handler with one paragraph of TSDoc IS the webhook contract doc. Each doc lives in the source file that owns the truth it describes; the README links to those files instead of copying their contents. The structural reason: paraphrased docs drift because nothing forces them to update when the code changes; the source-file-as-doc updates because the developer is editing the file anyway.
- **The schema-as-data-model-doc — worked example.** Drizzle schema with `pgTable`, column types, indexes, and relations is reference documentation in code form: a reader scrolling the file sees every table, every column, every constraint, every foreign key. Adding a paragraph of comment above each table explaining its purpose ("the `invoices` table — one row per issued invoice; tenant-scoped on `organization_id`; `status` transitions are append-only via `invoice_events`") turns reference into reference-plus-explanation in the place where it can't go stale. The senior pattern: a one-paragraph header comment per table, no per-column comments unless the column's purpose isn't obvious from its name and type.
- **The env-vars-as-doc — worked example.** A typed `env.ts` (Chapter 21.3) with each variable Zod-parsed and TSDoc-commented serves both reference (what's the type, is it optional, what's the default) and explanation (why is this variable here, what breaks without it). A paragraph in the README pointing at `env.ts` plus a `.env.example` file with the variable names and placeholder values is the entire env-var doc surface. No `ENVIRONMENT.md` separately.
- **The Server-Action-as-API-doc — worked example.** Server Actions are the course's API surface (Chapter 7.2). The action function with a Zod input schema and TSDoc on the signature IS the API contract: input shape, error shape, success shape. A REST/tRPC project would generate an API reference document; the Server Action project doesn't need one because the function is the doc. The senior reflex: keep the action small, type it precisely, comment the non-obvious failure modes inline; that's the API doc.
- **The "link, don't duplicate" enforcement.** When a paragraph wants to say "the user table has these columns…" the senior reflex is to rewrite as "the user table is defined in [`src/db/schema/users.ts`](link); see the file for column list and constraints." The link survives schema changes; the paragraph doesn't.
- **The `.env.example` discipline.** A committed `.env.example` with every required variable, fake/placeholder values, and a one-line comment per non-obvious entry. The file is the contract: if the dev's `.env.local` is missing keys present in `.env.example`, the `env.ts` Zod parse fails loudly at startup (Chapter 21.3). The README's "Getting started" section's `cp .env.example .env.local` step assumes this file.
- **The `CONTRIBUTING.md` question — when it earns a separate file.** For a closed-source SaaS, no — contribution conventions live in `AGENTS.md` (Lesson 22.1.3). For an open-source library or an open-by-policy repo where external contributors exist, yes — `CONTRIBUTING.md` covers the PR process, code-of-conduct pointer, and how to claim an issue. The course's stack assumes closed-source SaaS by default; the lesson states the conditional once.
- **Watch-outs:** the 400-line README — nobody reads past line 50, and lines 51–400 drift; copying the schema column list into a "Data model" README section — guaranteed to be wrong within a month; documenting env vars in three places (README, `ENVIRONMENT.md`, `env.ts`) — three sources of truth, two will drift; writing an "Architecture" section in the README — that's explanation, send it to ADRs; including screenshots of the running app in the README — they become outdated and never get updated; "TODO" sections in the README — TODOs live in issue tracker, not in user-facing docs; including team meeting notes or roadmap — wrong audience, wrong file.

What this lesson does not cover:

- AGENTS.md contents and structure — Lesson 22.1.3.
- ADR template and the course's opinionated picks — Lesson 22.1.4.
- TSDoc syntax for the schema/action/env comments — Chapter 22.2.
- Auto-generated docs (TypeDoc, Drizzle Studio) — out of scope; the typed source is the doc.
- Multi-package monorepo READMEs — out of scope; the course is single-app.

Estimated student time: 35 to 45 minutes. The chapter's grounding lesson on where docs *live* — every later lesson rests on this rule.

## Lesson 22.1.3 — AGENTS.md, the conventions file

Learn the 2026 `AGENTS.md` standard — its two audiences (coding agents and humans), the canonical sections (overview, layout, commands, conventions, don'ts, pointers), the hierarchical lookup rule, and the boundaries against README and ADRs.

Topics to cover:

- **The senior question.** Once the README is thin and the source files own their own reference docs, where do conventions, build steps, test commands, and "how we do things here" notes live? In 2026 the answer is `AGENTS.md` — a file written for two audiences at once: coding agents (the canonical case) and humans skimming for conventions. The chapter introduced `AGENTS.md` in passing at Chapter 1.4 when the project was scaffolded; this lesson is the full treatment.
- **What AGENTS.md is — the 2026 standard.** A markdown file at the repo root, content open-form but with conventional sections, read by coding agents (Codex, Claude Code, Cursor, Copilot agent mode, Factory CLI, et al.) to bootstrap their context on the codebase. The format is stewarded by the Agentic AI Foundation; the filename is uppercase `AGENTS.md`; agents read the nearest one in the directory tree, so monorepos can ship per-package overrides. Name the canonical homepage once; don't go deeper than necessary.
- **The two audiences — and why one file serves both.** A coding agent needs the build/test/lint commands, the style conventions, the entry points, and the watch-outs that aren't obvious from reading the code. A human contributor skimming "how do we work in this repo" needs the same information. The 2026 norm is: write one file for both, in plain markdown, with no agent-specific syntax — agents parse the same English a human does.
- **What earns a place — the inclusion test.** The senior test is "would a competent developer joining the team next Monday need this to be productive in week one?" If yes, it goes in `AGENTS.md`. If no, it's noise that pads the file and dilutes the parts that matter. Specifically: build/test/lint commands yes; the team's Slack channel no; the "we use Drizzle" yes; the history of why we used Prisma before no (that's an ADR).
- **The canonical sections — what most SaaS AGENTS.md files contain.**
  - *Project overview.* One paragraph: what the product does, who the user is, the stack core. Same shape as the README's opening, terser.
  - *Repo layout.* The top-level folders and what lives in each (`/src/app` Next.js routes, `/src/components` Client Components, `/src/db` Drizzle schema and queries, `/src/lib/<feature>` feature modules, `/src/server` Server Actions, `/tests` integration tests).
  - *Build, test, lint commands.* The exact commands an agent (or a new hire) runs: `pnpm install`, `pnpm dev`, `pnpm test`, `pnpm test:watch`, `pnpm lint`, `pnpm typecheck`. Listed plainly; no narration.
  - *Conventions.* The non-obvious rules: Server Components by default, Client Components only at the leaves; Server Actions handle mutations, route handlers only for third-party callbacks (webhooks); Zod schemas live next to the action that uses them; never `any`, use `unknown`; the file-naming convention; the test-file location convention. Each convention one line.
  - *What NOT to do.* The traps the codebase has structural rules against: don't import server modules into client components (Next.js will catch it but name it), don't hand-write SQL outside Drizzle, don't reach for `useEffect` for data fetching (Chapter 4.9), don't ship `console.log` (the lint rule will catch it, name the rule). The negative space saves the agent (and the human) cycles.
  - *PR and commit conventions.* If the team uses conventional commits, name the prefixes. If PRs require a passing CI before merge, name it. One paragraph; not a process essay.
  - *Pointers.* Link to the ADR folder, the schema file, the env file, the relevant how-to docs. Same "link, don't duplicate" reflex from Lesson 22.1.2.
- **The hierarchical structure — nested AGENTS.md files.** Agents read the nearest `AGENTS.md` walking up the tree, so a monorepo or a subfolder with idiosyncratic conventions can ship its own. The course's stack is single-app, so the file at the root is the only one; name the hierarchy rule once for the case where a `/packages/email-templates/AGENTS.md` would later add email-specific guidance.
- **Signal density over format compliance — the writing posture.** The spec deliberately doesn't mandate sections because the format's value is signal-per-line, not header structure. A senior writes AGENTS.md the way they'd brief a new hire over coffee — direct, no filler, examples where helpful. The failure mode is the file that's been "expanded for completeness" until every section is full and none is high-signal; the senior reflex is to delete sections that have nothing load-bearing in them.
- **AGENTS.md vs. README — the boundary.** The README is for the recruiter and the first-hour new contributor. The `AGENTS.md` is for the agent and the first-week contributor doing real work. They share the project-overview paragraph and the local-setup commands, by design — those are first-hour and first-week needs both — and after that they diverge: README links out, AGENTS.md goes into convention depth.
- **AGENTS.md vs. ADRs — the boundary.** AGENTS.md describes *what the conventions are*; ADRs describe *why the architectural decisions behind those conventions were made*. AGENTS.md says "Drizzle for all DB access"; the ADR says "Drizzle over Prisma because…". The link from AGENTS.md to the ADR closes the loop for the agent or contributor who wants the *why*.
- **Versioning and rot — keeping it accurate.** The structural enforcement is the same as for any in-repo doc: the AGENTS.md change ships in the PR that changed the convention. If the test command moved from `pnpm test` to `pnpm vitest`, the AGENTS.md update is in the same PR. The senior reviewer (Chapter 22.3) checks this. Mention the rule once; the discipline lesson lives in Chapter 22.2.3.
- **The `.cursorrules` / `.clinerules` / vendor-specific question — and why AGENTS.md replaces them.** Pre-2026, every agent had its own config filename: `.cursorrules`, `.clinerules`, `.github/copilot-instructions.md`. The fragmentation forced teams to maintain three files saying the same things. 2026's `AGENTS.md` consolidates: most major agents (Codex, Claude Code, Cursor, Cline, Aider, Factory, Kilo) read `AGENTS.md` either natively or via a one-line bridge file pointing at it. Name the consolidation once; don't list every vendor.
- **A worked AGENTS.md for this course's stack — sketch the shape.** Not a full file (too long); a sketch with the section headers and one-line content per section so the student sees the shape: "## Project overview — a multi-tenant invoice SaaS… ## Repo layout — `/src/app`… ## Commands — `pnpm dev`… ## Conventions — Server Components by default; Server Actions for mutations; Drizzle for DB; Zod next to actions; never `any`. ## Don't — no SQL outside Drizzle; no `useEffect` for data fetch; no `console.log`. ## Pointers — ADRs: `/docs/adr/`; schema: `src/db/schema.ts`; env: `src/env.ts`."
- **Watch-outs:** treating AGENTS.md as the dumping ground for everything that doesn't fit elsewhere — the file balloons, signal drops; including the full env var list in AGENTS.md instead of pointing at `env.ts` — duplication, drift; agent-specific instructions ("when you are an agent, do X") that don't make sense to humans — write for humans, agents parse the same prose; sections without content because "the spec recommends them" — delete the empty section; the meta-section explaining what AGENTS.md is — the agent and the human both already know; AGENTS.md as the home for ADR-worthy explanation ("we chose Drizzle because of these eight trade-offs") — that goes in `/docs/adr/`, AGENTS.md just states the convention.

What this lesson does not cover:

- The ADR template and the course's opinionated picks — Lesson 22.1.4.
- TSDoc on functions — Chapter 22.2.
- Specific agent product configuration (settings inside Codex/Cursor/Claude Code apps) — out of scope; AGENTS.md is the in-repo surface.
- Auto-generated AGENTS.md sections — out of scope; this is a hand-maintained file.

Estimated student time: 40 to 50 minutes. The chapter's longest lesson — the full treatment of the file the course itself ships with.

## Lesson 22.1.4 — ADRs: one decision per file

Learn the Nygard ADR template (Title, Status, Context, Decision, Consequences), the three-test inclusion check, the "write while deciding" discipline, the supersession lifecycle, and six worked ADR sketches on the course's own opinionated picks (Drizzle, Better Auth, Biome, R2, Node runtime, native forms).

Topics to cover:

- **The senior question.** Six months from now, when a new senior joins the team and asks "why did you pick Drizzle?" or two years from now when someone wants to swap Better Auth for Clerk, where does the *reasoning* behind that decision live? Not in commit messages (one-line, lost in `git log`), not in Slack (gone), not in the README (wrong audience), not in `AGENTS.md` (wrong purpose). It lives in an ADR — one markdown file per decision in `/docs/adr/`, written *as the decision is being made*, not reconstructed from memory after the fact.
- **What an ADR is — the definition.** A short markdown document (typically half a page to two pages) capturing one architectural decision: the context that forced the choice, the decision itself, and the consequences the team accepted by choosing it. Originated in Michael Nygard's 2011 blog post; remains the dominant template in 2026; lives in `/docs/adr/` as numbered files (`0001-use-drizzle-not-prisma.md`, `0002-use-better-auth-not-clerk.md`).
- **The Nygard template — the three load-bearing sections.**
  - *Title.* Numbered, short noun phrase: "ADR 0001: Use Drizzle, not Prisma." Not a question, not a sentence, not vague — the noun phrase of the decision itself.
  - *Status.* Proposed / accepted / superseded / deprecated. Most live ADRs are "accepted." When a later ADR supersedes an earlier one, the older file's status becomes "superseded by ADR 0017"; the file is never deleted, the history is the value. (Some templates add *Date*; useful, not load-bearing — git tracks the file's birthdate.)
  - *Context.* The forces at play when the decision was made. What problem needed solving; what constraints existed; what alternatives were on the table; what was *not* known. Written so a reader two years from now can model the situation. A paragraph, sometimes two; not a history of the company.
  - *Decision.* One sentence, declarative: "We will use Drizzle as the ORM for all database access." Not hedged, not "we are considering," not "we should." A senior writes the decision crisply because the document records what was *decided*, not what was discussed.
  - *Consequences.* What changes as a result. Both the good (the constraints the decision releases) and the costs (the constraints it imposes). What the team commits to maintaining; what becomes harder; what new patterns the team now needs to learn. The honest list is the senior signal — a Consequences section with only upsides is a sales pitch, not an ADR.
- **One decision per file — the atomicity rule.** ADR 0001 covers Drizzle. ADR 0002 covers Better Auth. Not "ADR 0001: our stack choices." Each file is one decision because each decision has its own context, alternatives, and consequences; bundling them defeats the per-decision history when one decision later gets revisited.
- **Write the ADR *while* deciding, not after.** The senior call is to draft the ADR as the team is having the conversation, with a status of "proposed," and merge it as part of the PR that ships the decision. The reasoning: a week after the decision the rationale starts compressing; a month after, the alternatives get rationalized; six months after, the ADR is reconstructed from memory and is wrong. The PR that introduces Drizzle to the codebase is the PR that adds `/docs/adr/0001-use-drizzle.md`. The discipline is structural.
- **What earns an ADR — the inclusion test.** A decision earns an ADR if (a) it affects multiple files or future PRs (architectural), (b) reasonable engineers could pick differently, and (c) reversing it would cost more than one PR. "Use Drizzle" passes all three. "Name this variable `userId` instead of `uid`" passes none. The lesson states the test plainly so the student doesn't ADR every micro-decision.
- **What does NOT earn an ADR.** Variable naming, function decomposition, file structure within a feature, library version bumps that don't change the API, choice of CSS class name. These are code-review topics, not ADR topics. The structural rule: if it's reversible in one PR by one engineer, it's not an architectural decision.
- **The course's opinionated picks worth an ADR — the worked examples.** The course itself has made architectural decisions that are perfect ADR fodder. The lesson walks each as a one-paragraph sketch of the ADR that would justify it, so the student sees the form on real choices:
  - *Drizzle over Prisma.* Context: the SaaS needs a typed ORM with control over generated SQL and a small client bundle; Prisma owns the market but has heavier runtime, schema-as-DSL, and migration ergonomics that don't fit our migration story (Chapter 21.4). Decision: Drizzle. Consequences: leaner runtime, raw-SQL escape hatch, manual relation typing (handled with `relations()`), less mature ecosystem of plugins, fewer Stack Overflow hits — accept the trade.
  - *Better Auth over Clerk / Auth.js.* Context: the SaaS needs email+password, OAuth, sessions, RBAC, org-scoped auth in a Next.js 16 codebase under our control. Decision: Better Auth. Consequences: full data sovereignty (the users table is *ours*, in *our* Postgres), no per-MAU billing, RBAC and orgs built into the plugin surface, the operational responsibility of running auth is ours (Chapter 9.4).
  - *Biome over ESLint+Prettier.* Context: the team wants linting and formatting that runs in milliseconds in pre-commit and CI; ESLint+Prettier is two tools, two configs, slower runtime. Decision: Biome. Consequences: single config, ~10x faster, fewer rules than the union of ESLint plugins (accept the trade), modern stack alignment.
  - *Cloudflare R2 over S3.* Context: the SaaS needs object storage with predictable cost and no egress fees for the typical SaaS read/download pattern. Decision: R2. Consequences: zero egress, S3-compatible API (the SDK is the same), one extra account to manage, slightly newer ecosystem maturity than S3 — accept the trade for the egress savings.
  - *Node runtime, not edge, for App Router routes by default.* Context: Vercel offers Edge runtime for latency wins, but Drizzle + Postgres + Resend + Stripe + Better Auth all want Node APIs and persistent connections; Edge runtime forbids Node modules and prefers stateless HTTP. Decision: Node runtime as default; Edge only for specific routes that genuinely benefit (none in the course's surface). Consequences: simpler module compatibility, full Node ecosystem, slightly higher cold-start latency vs. Edge (acceptable for SaaS dashboards).
  - *Native React 19 forms before reaching for React Hook Form.* Context: React 19 ships `useActionState`, native `<form action>` with progressive enhancement, server-action integration; RHF is a strong library but adds a dependency, a hook learning curve, and a different mental model from the platform default. Decision: native forms by default; RHF only when the threshold from Chapter 7.4 fires. Consequences: less code, fewer deps, server-action integration is direct, complex client-only validation flows take more lifting when they do come up (accept the trade and reach for RHF when they do).
  
  Each one is a one-paragraph sketch in the lesson — not a full ADR — to model the shape. The point is that *every* one of these had a reasonable alternative and the team would benefit from the next maintainer seeing the reasoning.
- **The numbering convention.** Sequential, zero-padded, never reused. `0001-use-drizzle.md`, `0002-use-better-auth.md`. Numbers are stable identifiers; they appear in PRs ("see ADR 0007") and survive renames. Don't number by date (the date already lives in git); don't number by category.
- **Status transitions — the lifecycle in practice.** A decision that gets revisited two years later: the new ADR is `0019-replace-drizzle-with-X.md`, status "accepted"; ADR 0001's status updates to "superseded by ADR 0019" with a one-line reference. The original file's content stays intact — the historical record is the value. Deletion is never the move; supersession is.
- **The "one paragraph per section" discipline.** ADRs are not essays. The reader is a future maintainer with a question; the document should answer it in two minutes of reading. Context: one paragraph. Decision: one sentence. Consequences: a bulleted list of three to seven points. The discipline keeps the ADR write-time low (fifteen minutes, not an afternoon) and the read-time low.
- **The "where do ADRs live" answer.** `/docs/adr/` at the repo root. Plain `.md` files. Indexed by a short `README.md` inside the folder that lists each ADR with its title and status; the index is hand-maintained because it doubles as a table of contents.
- **MADR — the alternative template, named once.** MADR (Markdown Any Decision Records) is the other widely-used template; it adds *Considered Options* and *Decision Outcome* sections, more structured. Some teams prefer MADR's structure; the course uses Nygard because the three-section shape is the smallest one that does the job, and signal-per-line is the senior posture. Name MADR once so the student recognizes it when they see it; the course's call is Nygard.
- **What `git blame` and PR descriptions do NOT replace.** A common pushback: "the PR description and `git blame` already capture this." They don't — PR descriptions are searchable only by PR-tooling-specific queries (and they're often a checklist, not prose); `git blame` answers "who wrote this line" not "why is this whole pattern here"; commit messages compress the rationale to one line and don't survive squash merges. The ADR is the durable, queryable, file-system-grep-able artifact; it's worth the fifteen minutes per decision.
- **Watch-outs:** writing the ADR three months after the decision shipped — the reasoning is already compressed and wrong; bundling multiple decisions into one ADR — defeats the per-decision lifecycle; "ADR 0001: Coding conventions" as the first ADR — coding conventions are not architectural decisions, they're `AGENTS.md` content; an empty Consequences section because "we don't see any downsides" — every architectural decision has costs, the senior writer names them or the ADR isn't honest; hedged Decision lines ("we will consider Drizzle") — the Decision section is for the *decision*, not the deliberation; deleting an ADR when the decision is superseded — supersede in place, don't delete; ADR sprawl (one per feature flag, one per dependency upgrade) — apply the three-test inclusion check.

What this lesson does not cover:

- The PR-review side of catching missing ADRs — Chapter 22.3.
- The project-chapter exercise of writing an ADR from a seeded diff — Chapter 22.4.
- TSDoc, comments, and inline documentation — Chapter 22.2.
- Process tools for ADRs (adr-tools CLI, log4brains) — out of scope; plain markdown files are the deliverable.
- ADRs for non-code decisions (team process, hiring) — out of scope; this is architectural decisions only.

Estimated student time: 50 to 60 minutes. The chapter's heaviest lesson — both the Nygard template and the six course-level worked examples land here.
