# Chapter 3.6 — Fetch and live streams

## Chapter framing

Chapters 3.1–3.4 walked the request side of the browser platform from the outside in — what crosses the wire, the HTTP contract, origins and CORS, cookies. Chapter 3.5 turned to the DOM and event substrate. Chapter 3.6 closes the platform unit on the JavaScript surface that issues network requests and reads streamed responses: the universal `fetch` API on one side and the `ReadableStream` plus SSE half on the other. The senior framing is that `fetch` is the single network primitive every 2026 codebase reaches for — from a Server Component pulling JSON, to a Server Action's third-party SDK underneath, to a Client Component subscribing to a live feed. The chapter installs the canonical call shape, the response-handling reflexes that don't come for free, and the streaming primitives that let the server push progress to a UI without polling.

Threads that run through every lesson. `fetch` does not throw on 4xx/5xx — every call branches on `response.ok` and parses the success body with Zod (forward reference to 7.1, named once); the unhappy branches map to typed errors and never leak raw response bodies. Every network call carries an `AbortSignal` — `AbortSignal.timeout(ms)` for deadline-bound calls and `AbortSignal.any([...])` for composing user-cancel, timeout, and request-shutdown signals (the surface from 2.7.4, re-applied here). The senior posture is "trigger before tool" for live updates: polling is the default until reads are frequent enough or pushes have to be timely; SSE is the conditional reach when the channel is server-to-client and stays on HTTP; WebSockets are the conditional past SSE, reached for only when the channel needs to be bidirectional. The chapter is deliberately platform-level — no React hooks, no Next.js caching wrappers (`fetch` augmentation lands in Chapter 5.4), no AI SDK abstractions (those land in Unit 23). The student finishes able to write a hardened `fetch` call from memory, recognize the augmented `fetch` Next.js layers on top later, read a streaming response chunk by chunk, emit and consume an SSE stream, and pick the right live-channel primitive against a concrete trigger.

The chapter ships two teaching lessons plus a quiz.

## Lesson 3.6.1 — The universal HTTP client

Installs the canonical `fetch` call shape — `Request`, `Response`, `Headers`, `FormData`, the body consumer methods, the `response.ok` branch, `AbortSignal.timeout` and `AbortSignal.any`, and the Zod parse on the success path — as the one HTTP surface a 2026 codebase reaches for from the browser, from a Server Component, and from inside a Server Action.

Topics to cover:

- **The senior question.** A button click needs to POST a JSON body to `/api/invoices`, parse the response, render the new row, and behave correctly when the request times out, the user navigates away, the server returns 422, or the body isn't the shape the client expected. What's the call shape that handles all four without retries or wrappers, and what reflexes does `fetch` not give you for free? The lesson installs the canonical 2026 form.
- **`fetch` as the one HTTP primitive.** The same `fetch(input, init)` function works in every runtime the course touches — Chrome and Safari, Node 22+, the Edge runtime, the Server Components renderer, the Server Action body, the route handler. The 2026 reflex from 2.7.4 (Promises) and 2.8.2 (catch narrowing): one async function call, awaited, branched on the typed `Response`. No `axios`, no `node-fetch`, no custom client.
- **The `Request` / `Response` / `Headers` triad as Web Platform types.** The three named types every `fetch` call uses. `Request` is the request object (URL, method, headers, body) — passable directly to `fetch` or constructed inline via the `init` shape. `Response` is the typed result (status, `ok`, `headers`, body). `Headers` is the case-insensitive multimap of header names to values, with `.get`, `.set`, `.append`, `.entries`. The student leaves recognizing these as the same types route handlers receive and return — symmetry across the seam.
- **The canonical call shape, end to end.** The form every lesson and project in the course writes against:

  parse-input → build `init` (method, headers, body, signal) → `await fetch(url, init)` → branch on `response.ok` → on `ok`, `await response.json()` then `Schema.parse(...)` → on not-`ok`, read a typed error body and return a `Result` failure (foreshadow 7.2.3) → catch the thrown errors (network / abort / timeout) and re-shape into the same `Result`.

  The five seams are: build, send, ok-branch, parse, catch. The senior reflex: every `fetch` call has all five, even when some collapse to one line.
- **The body consumer methods — pick one.** A `Response` body is a `ReadableStream` consumed exactly once. The five consumers: `response.json()` (the default for SaaS JSON APIs), `response.text()` (raw text, HTML, plain logs), `response.formData()` (the inverse of a `<form>` post), `response.blob()` (binary — images, PDFs, file downloads), `response.arrayBuffer()` (typed-array reads, crypto, low-level binary). Reading twice throws — the second `.json()` returns a "body already used" error. The senior reflex when you need both raw and parsed: read once into a variable, then act.
- **Setting the request body — the four shapes.** `string` (handcrafted JSON via `JSON.stringify`, the common case), `FormData` (multipart with files; the browser sets the `Content-Type` boundary header for you; do not set it manually), `URLSearchParams` (URL-encoded form posts), `Blob` / `ArrayBuffer` / `ReadableStream` (binary uploads, the substrate the 3.7.3 file-upload chapter and the 13.4 R2 chapter build on). The watch-out: setting `Content-Type` to `application/json` when sending `FormData` corrupts the multipart boundary; the senior call is to let the browser pick the content type when sending `FormData`.
- **The `Headers` surface every call touches.** `Accept` (the response content type the caller wants), `Content-Type` (the request body's type, set explicitly for JSON), `Authorization` (the bearer-token site for service-to-service calls; cookies handle the browser case), `Idempotency-Key` (the retry-safety header from 3.2.1, foreshadowed for 12.1). The senior reflex: send what you need, no more — every header is a fingerprint. Cross-reference 3.2.3 for the full header surface.
- **`response.ok` is the load-bearing branch.** `fetch` resolves the promise for any response the server returns, including 404, 422, and 500. The promise only rejects on network failure, CORS preflight rejection, abort, or DNS error. The 2026 reflex: every call has `if (!response.ok) { ... }` immediately after the await. Inside that branch, the senior reads the typed error body (RFC 9457 Problem Details from 3.2.2 — `application/problem+json`), maps it to the codebase's `Result` failure shape, and returns; outside that branch, parsing proceeds on the happy path.
- **Reading and validating the happy-path body with Zod.** Once the call clears the `ok` branch, the response body is `unknown` from TypeScript's point of view (`response.json()` returns `Promise<unknown>` in modern types). The senior reflex: `Schema.parse(await response.json())` — never trust the wire. The schema lives in `/lib` (Architectural Principle #3, foreshadowed) and is shared between the caller and the route handler when both sit in the same codebase. The full Zod call surface is owned by 7.1; this lesson names the move and writes the one-line `parse` on the success path.
- **`AbortSignal.timeout(ms)` as the deadline default.** Every outbound `fetch` to a third party or a slow internal route carries a deadline. The 2026 reflex from 2.7.4: `init.signal = AbortSignal.timeout(5_000)`. The signal fires a `TimeoutError` (not `AbortError`) at the deadline — narrow on `error.name === 'TimeoutError'` in the catch and map to a typed `Result` failure. Senior calibration: 5s for an internal call, up to 30s for a known-slow third party, never "no deadline" for a user-facing path.
- **`AbortSignal.any([...])` for composing cancellation.** When the caller needs both a user-cancel signal and a deadline, compose: `AbortSignal.any([userController.signal, AbortSignal.timeout(5_000)])`. The composite signal aborts on whichever fires first; the catch narrows on `error.name` to tell them apart. The pattern from 2.7.4 re-applied at the `fetch` call site.
- **Catching the right errors.** Five throwable cases at the `fetch` await: `TypeError` (network failure, CORS, DNS), `AbortError` (manual abort), `TimeoutError` (signal timeout), `DOMException` of various kinds in rare runtimes, and downstream throws from body consumers (`SyntaxError` from `response.json()` on malformed JSON). The senior reflex: narrow with `if (error instanceof Error && error.name === '...')` per 2.8.2, never `catch (e: any)`, never swallow.
- **Where `fetch` calls live in a 2026 codebase.** Four call sites the student maps to mentally:
  - From a Server Component or a Server Action calling an internal `/api/...` route — rare, almost always re-architected to a direct function call (Principle #3 thread).
  - From a Server Component or a Server Action calling a third-party API (Stripe, Resend, an analytics service) — the common pattern, often hidden behind the vendor SDK.
  - From a Client Component calling an internal route handler — the canonical site for live feeds, search-as-you-type, and any work the action surface can't carry.
  - From a Client Component calling a third-party API directly — rare, gated by CORS; the senior call is to proxy through a route handler.
- **What Next.js layers on top, named once.** Next.js extends the global `fetch` with caching, deduplication, and revalidation options (`cache: 'force-cache'`, `next: { revalidate, tags }`). The student does not learn that surface here — it's owned by Chapter 5.4. The lesson names the augmentation so the student recognizes the same `fetch` signature behaving differently inside a Server Component. The principle: the senior recognizes the platform default first, then the framework's augmentation.
- **What `XMLHttpRequest` and SDK wrappers are for in 2026.** Recognition only. `XMLHttpRequest` still exists for upload-progress events (the one capability `fetch` doesn't natively expose; the 13.4 R2 upload chapter names the trigger). Wrappers like `ky`, `ofetch`, and `axios` exist but the 2026 default is plain `fetch` plus a thin in-house helper — the senior posture is "don't import an HTTP client until your `fetch` boilerplate has visibly cost you something."
- **The small in-house `apiFetch` helper.** When the codebase has accumulated three identical-looking `fetch` calls — same base URL, same auth header, same error mapping — extract a typed helper into `/lib/http.ts`: `apiFetch<T>(path, init, schema)` that owns the base URL, the default headers, the `response.ok` branch, the Zod parse, and the `Result` mapping. Named once here as the natural extraction; not built in this lesson. The senior posture: extract on the third repetition, not the first.
- **Watch-outs.** Treating `await fetch(...)` as the parsed body — it's the `Response` object, the body needs a second await; calling `response.json()` twice throws — read once into a typed variable; setting `Content-Type: application/json` on a `FormData` post strips the multipart boundary — let the browser set it; assuming `fetch` throws on 404 or 500 — it doesn't, the `ok` branch is the failure surface; passing the raw URL string with `?q=${userInput}` for the query string — use `URL` and `URLSearchParams` from 3.3.1; forgetting the `signal` on a third-party call and watching the request hang — `AbortSignal.timeout` belongs on every outbound call; using `try`/`catch` to mean "the request failed" and lumping HTTP errors with network errors — they're different failure modes with different `Result` shapes; sending credentials cross-origin without `credentials: 'include'` — the browser strips cookies by default cross-origin (foreshadowed from 3.3.3); reading `response.headers` case-sensitively — `Headers.get` is case-insensitive and that's the contract.

What this lesson does not cover:

- The Zod call surface in depth — owned by Chapter 7.1.
- The `Result<T>` shape and its helpers — named here, owned by 7.2.3.
- Next.js's `fetch` augmentation (`cache`, `next.revalidate`, `next.tags`) — owned by Chapter 5.4.
- Route-handler authoring (the receiving side) — owned by Chapter 7.5.
- File uploads with `Blob` / `File` — primitive named here, owned by 3.7.3 and the 13.4 R2 project.
- Retries with exponential backoff — out of scope for this chapter; foreshadowed at the third-party-SDK seam (Chapter 8 for email, Chapter 12 for billing).
- Cookies, CORS, and `credentials` mode in depth — owned by Chapters 3.3 and 3.4.
- Progress events for uploads — `XMLHttpRequest` recognition only; the 13.4 chapter handles the trigger.

## Lesson 3.6.2 — Streaming and live channels

Teaches reading a `Response` body as a `ReadableStream` of chunks, emitting Server-Sent Events from a Next.js Route Handler, consuming them with `EventSource` (and where plain `fetch` over the stream earns its weight instead), and the polling / SSE / WebSocket decision tree with the trigger that flips each step.

Topics to cover:

- **The senior question.** A CSV export takes 12 seconds; the user needs a progress bar. A notifications panel needs to show new items as they arrive without a 5-second poll. An LLM endpoint streams tokens. Each is a server-to-client live update over HTTP, none of them is a request-response. What primitive carries the bytes, what shape goes on the wire, how does the client consume them, and when does polling beat SSE and SSE beat WebSockets? The lesson installs the substrate and the decision.
- **`Response.body` is a `ReadableStream`.** Every `Response` body is already a stream of `Uint8Array` chunks the browser exposes as a `ReadableStream`. The body-consumer methods from 3.6.1 (`.json()`, `.text()`) buffer the stream end-to-end and resolve once; calling `response.body` instead hands the stream to the caller to read chunk by chunk. The 2026 reflex: `for await (const chunk of response.body) { ... }` — `ReadableStream` is async-iterable in modern runtimes; the loop yields a `Uint8Array` per chunk. The `getReader()` / `read()` form is the older surface, recognized but not the default.
- **Decoding chunks back to text.** `Uint8Array` is bytes; the consumer almost always wants strings. The `TextDecoder` API is the surface: `new TextDecoder().decode(chunk, { stream: true })` decodes the bytes preserving multi-byte UTF-8 boundaries across chunk edges, a final `decoder.decode()` flushes any trailing buffered byte. The senior reflex: always `stream: true` on chunks, always one final flush. The watch-out: decoding chunk-by-chunk without the `stream: true` option corrupts emoji and accented characters that straddle a chunk boundary.
- **Reframing chunks into messages.** Chunks are network frames, not application messages — a single SSE event can span two chunks, two events can land in one chunk. The senior reflex: buffer chunks into a string accumulator, split on the protocol's framing token (for SSE, the blank-line `\n\n` separator), emit one parsed message per separator. The pattern: append, split, keep the tail, parse the rest. Named once; the lesson writes the loop.
- **Server-Sent Events as the senior default for one-way live updates.** SSE is a thin convention layered on a `text/event-stream` HTTP response that stays open. The server writes lines of the form `data: <payload>\n\n` (with optional `event:`, `id:`, and `retry:` fields); the client reads the stream and dispatches per event. The browser's native `EventSource` API handles reconnection automatically and replays the last seen event ID on reconnect via the `Last-Event-ID` header. The 2026 reflex: when the channel is server-to-client only and the payload is JSON or text, SSE is the default.
- **Emitting SSE from a Next.js Route Handler.** A `GET` handler that returns a `Response` constructed from a `ReadableStream`. The route handler's full file shape is owned by Chapter 7.5; the lesson uses the minimum form:

  inline pseudo-code shape: `new Response(stream, { headers: { 'Content-Type': 'text/event-stream', 'Cache-Control': 'no-cache, no-transform', 'Connection': 'keep-alive' } })`

  Inside the `ReadableStream` controller's `start`, write `data: ${JSON.stringify(payload)}\n\n` per event with `controller.enqueue(encoder.encode(...))`, call `controller.close()` when the stream is done. The senior anchor: SSE is bytes, not strings — encode through `TextEncoder` before enqueueing.
- **The three load-bearing headers.** `Content-Type: text/event-stream` (the protocol opt-in — browsers and proxies key on this), `Cache-Control: no-cache, no-transform` (`no-transform` prevents middle boxes from buffering the stream into a single response — the failure mode that breaks SSE in production behind a CDN), `Connection: keep-alive` (recognition; HTTP/2 and HTTP/3 ignore it, HTTP/1.1 needs it). The 2026 reflex: all three together, every time.
- **Cleaning up when the client disconnects.** The route handler must respect the client closing the tab — every modern runtime exposes `request.signal` (an `AbortSignal` that fires on disconnect). The 2026 reflex: subscribe to `request.signal`'s `'abort'` event inside the stream, clear timers and close database cursors, then `controller.close()`. The pattern composes with the 3.5.3 `AbortController` thread — one signal per setup site, abort fan-out on cleanup. The watch-out: a stream that doesn't watch `request.signal` leaks a timer (and a database connection) per disconnected client.
- **Consuming SSE from a Client Component with `EventSource`.** `const source = new EventSource('/api/events')` opens the connection, `source.onmessage = (e) => ...` handles default `data:` events, `source.addEventListener('<name>', ...)` handles named events, `source.close()` shuts the connection. The browser auto-reconnects with the last `id:` replayed in the `Last-Event-ID` header. The senior reflex inside React: the subscription lives in a `useEffect` (or future `use` boundary — Chapter 4.9), the cleanup function calls `source.close()`, the `AbortController` pattern from 3.5.3 carries through.
- **The `EventSource` ceiling — when plain `fetch` earns its weight.** `EventSource` can't set custom request headers (no `Authorization`, no CSRF token), can't issue a non-`GET` request, and can't carry a request body. The 2026 reflex when any of those bite: consume the `text/event-stream` response with `fetch` directly — same SSE byte protocol, the client side reads `response.body` and reframes events itself. The named callsite: cookie-authenticated SSE works with `EventSource` (cookies travel automatically with `withCredentials: true` for cross-origin), bearer-auth SSE requires the `fetch` consumer.
- **The live-channel decision tree.** Three options, three triggers, in the order a senior considers them:
  - *Polling* — the default. `setInterval` or React Query's `refetchInterval` on a normal request-response endpoint. Trigger to leave it: the staleness window the product requires is shorter than the polling cadence the server can carry, or the polling fan-out (clients × cadence) becomes a load problem. The 2026 reflex: most "real-time" SaaS features ship on 5-30s polling and never need anything more; cross-reference forward to React Query in Chapter 16.
  - *SSE* — the conditional past polling. Trigger: server-to-client only, payload is JSON or text, the connection budget per client is acceptable (one open HTTP connection per subscribed tab, persisting). Examples: LLM token streaming, build-progress feed, notifications panel, live order status. The senior anchor: SSE rides existing HTTP infrastructure (CDN-friendly with `no-transform`, plays with HTTP/2 multiplexing) and re-uses the auth model the rest of the app uses (cookies).
  - *WebSockets* — the conditional past SSE. Trigger: the channel needs to be bidirectional (collaborative cursors, chat with typing indicators, multiplayer state), or message framing on the upstream side has to be tighter than HTTP can carry. Examples: Liveblocks-style presence, real-time chat with delivery acks, multiplayer canvases. The senior anchor: WebSockets are not a default — they need a separate connection model, can't use the HTTP cache, and need their own auth handshake. The full WebSocket surface is recognition only in this course; the trigger is named so the student recognizes when a feature crosses the line.
- **Why the AI SDK's stream lands on this substrate.** Forward reference, named once: the Vercel AI SDK's `streamText` / `useChat` (Unit 23) is `ReadableStream` + an SSE-shaped protocol on the wire, plus typed deltas on top. The student finishes recognizing that the AI surface in Unit 23 is the same primitives this lesson installs, not a parallel transport.
- **What this surface is not for.** Two cuts named at adult depth so the student doesn't reach wrong:
  - *Large file downloads* are `response.blob()` or `response.body` piped to a save-to-disk destination — streaming bytes, not streaming events. Recognition only.
  - *Background work that takes longer than 30 seconds* is a background job, not an SSE stream — Chapter 16 owns the queue surface. Holding a connection open for minutes is fragile across proxies, mobile networks, and serverless platforms.
- **Watch-outs.** A stream that doesn't write the three SSE headers ends up buffered by a CDN and the client receives a single 12-second-late blob — the `no-transform` directive is load-bearing; encoding chunks without `stream: true` on `TextDecoder` corrupts multi-byte characters straddling a chunk boundary; parsing SSE by splitting every chunk on `\n\n` without buffering loses events that span a chunk — accumulate then split; calling `response.body.getReader()` and forgetting `reader.releaseLock()` (or to consume to completion) leaks the underlying network connection; an `EventSource` opened without a `close()` cleanup keeps the connection (and the client tab's connection budget) alive after the component unmounts — the cleanup is non-optional; serverless platforms cap connection duration (Vercel's per-plan limits, the 2026 default for SSE handlers on Hobby is bounded) — long-lived streams need the platform's streaming or background-jobs surface, named for Chapter 21; SSE on HTTP/1.1 is limited to 6 connections per origin per browser — multiple tabs subscribing to the same stream compete for the budget, HTTP/2 multiplexing removes the cap (cross-reference 3.1.1 on HTTP/3 over QUIC); writing to a `ReadableStreamDefaultController` after `close()` throws — guard with the `abort` signal before every write; sending an SSE payload that isn't JSON-stringified and contains an embedded newline corrupts the protocol — every line of `data:` is one logical line, multi-line payloads either escape newlines or repeat the `data:` prefix per line.

What this lesson does not cover:

- The route-handler file shape, the `Params`/`Body` schemas, and the auth wrapper — owned by Chapter 7.5.
- Next.js page streaming and Suspense — different primitive, same `ReadableStream` underneath — owned by Chapter 5.3.
- The Vercel AI SDK's `streamText`, `useChat`, and the `UIMessage` protocol — owned by Unit 23, named here as a forward reference.
- React Query polling configuration (`refetchInterval`, `staleTime`) — owned by Chapter 16.1.
- WebSocket APIs (the `WebSocket` constructor, `ws://`, framing, ping/pong) — out of scope, recognition only at the decision tree.
- Background jobs and queue dispatching — owned by Chapter 16.
- Server-to-server streaming patterns (gRPC, message buses) — out of scope.
- The `WritableStream` / `TransformStream` surface — `ReadableStream` is the only stream side this chapter installs.
- Backpressure and stream piping (`pipeThrough`, `pipeTo`) — recognition only; the SaaS surface rarely needs the full pipe API.

## Lesson 3.6.3 — Quizz

Top 10 topics to quiz:

- The `fetch` resolution model — what makes the promise resolve, what makes it reject, why `response.ok` is the load-bearing branch and why `fetch` never throws on a 4xx or 5xx.
- The `Request` / `Response` / `Headers` triad — what each carries, why `Headers.get` is case-insensitive, the same types appearing on the receiving side of a route handler.
- The body-consumer methods (`.json`, `.text`, `.formData`, `.blob`, `.arrayBuffer`) — why a body can be consumed only once, which to pick per content type, the "let the browser set Content-Type on FormData" rule.
- The canonical five-seam call shape — build, send, ok-branch, parse, catch — and the Zod parse on the success path as the "never trust the wire" reflex.
- `AbortSignal.timeout(ms)` versus `AbortSignal.any([...])` — why every outbound call carries a deadline, how `TimeoutError` is distinguished from `AbortError` in the catch.
- `Response.body` as an async-iterable `ReadableStream` — chunk-by-chunk reading, `TextDecoder` with `{ stream: true }`, and why decoding without `stream: true` corrupts multi-byte UTF-8.
- The SSE wire format — `data:` lines, the `\n\n` event terminator, named events with `event:`, the `id:` field that auto-replays via `Last-Event-ID` on reconnect.
- Emitting SSE from a Next.js route handler — the three load-bearing headers (`Content-Type: text/event-stream`, `Cache-Control: no-cache, no-transform`, `Connection: keep-alive`), the `ReadableStream` with `TextEncoder`, and respecting `request.signal` for client-disconnect cleanup.
- `EventSource` consumption and its ceiling — auto-reconnect with `Last-Event-ID`, why custom headers (bearer auth) and non-`GET` requests force the plain-`fetch` consumer instead.
- The polling / SSE / WebSocket decision tree — the trigger that flips each step (staleness budget exceeds poll cadence → SSE; channel needs bidirectionality → WebSockets) and why polling is the senior default in 2026 SaaS.
