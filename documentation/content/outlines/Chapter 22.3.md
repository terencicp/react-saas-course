# Chapter 22.3 — The review surface

## Chapter framing

Unit 22 has covered the artifacts that document a SaaS codebase (Chapter 22.1) and the in-source discipline that keeps them honest (Chapter 22.2). Code review is the third leg of the same stool — the checkpoint where one human (or one agent) reads a diff with attention before it becomes production behavior. The PR review is also where every architectural principle and SaaS pattern the course taught either gets defended or gets quietly eroded. This chapter teaches the senior posture at the review surface: *what* to look for so the review tracks the course's earlier teaches instead of devolving into style commentary, and *how* to write the comment so the feedback lands as collaboration instead of gatekeeping.

Threads through every lesson: the review's job is to defend the architectural principles (#1–#7) and SaaS patterns (#1–#15) the codebase already paid to establish — style, formatting, and naming polish belong to the linter and the formatter, not the reviewer; correctness, security, and contract integrity sit at the top of the review stack and are never bumped by aesthetic preference; every review comment carries a severity that the author can read in one second (blocking vs. suggesting vs. nit), so the author knows which comments hold the merge and which are polish; the language of disagreement is opinionated, evidence-led, and short — the reviewer states the failure mode, points at the principle or pattern that catches it, and proposes the fix or asks the question, then stops; reviews are mentorship channels and the senior reviewer earns trust by being right often and humble when wrong; AI agents now write code that lands in the same PR queue — the review surface is the same, the comments are the same, the discipline doesn't bifurcate.

## Lesson 22.3.1 — Where the eyes go first

The five-layer review stack a senior runs top-down on every diff, and the principle-and-pattern map that turns the diff into a small set of checks instead of a line-by-line crawl.

Topics to cover:

- **The senior question.** Open a PR diff. Where do the eyes go first, and what are they looking for? The wrong answer — and the most common one — is "from the top, line by line, flag anything that looks off." That review surfaces formatting noise and misses the load-bearing failures. The senior reflex is to map the diff onto the course's principles and patterns and ask, for each one the diff touches, *did this change defend the invariant the principle or pattern exists to enforce, or did it punch a hole through it?*
- **The review stack — what gets checked first and why.** The reviewer's attention is finite; spend it where it matters. A five-layer stack the senior runs in order, top-down, on every PR:
  1. *Correctness and security.* Does the code do what the PR description claims? Does it leak secrets, drop the auth check, expose the tenant boundary, log PII, trust client input the server should validate?
  2. *Architectural principles (#1–#7).* Does the diff respect co-location, schema-as-source-of-truth, the side-effect boundary, naming-for-intent, framework conventions over invention, explicit-over-magic, and impossible-states-unrepresentable?
  3. *SaaS patterns (#1–#15).* Does the diff use the tenant-aware query helper, the authed-action wrapper, the seat-management primitives, the billing carve-out, the webhook idempotency key, the Result return shape, the URL-state list view convention, the cache-decision posture, the soft-delete shape, the notification layer, the expand-migrate-contract cadence, the security baseline, the time/date/i18n primitives, the performance vigilance reach — wherever the diff touches the surface?
  4. *Tests and contract integrity.* Are the new code paths covered, do the tests assert behavior the user cares about, does the TSDoc still match the signature, did `.env.example` update with `env.ts`?
  5. *Style and naming polish.* The linter caught the formatting; the reviewer reads names against the intent and flags the genuinely confusing one. This is the last layer, not the first; if the review only got this far the review missed.
- **The principle-and-pattern map — the reviewer's reference.** The lesson lays out the seven principles and fifteen patterns as a compact reference the reviewer scans against the diff. The map names, for each principle and pattern, *the failure mode it prevents* and *the diff signature that signals "this didn't use the established surface."* Not a re-teach — a one-line reminder per item with a link back to the lesson that owns it. The senior reflex after a quarter of practice: scan the diff once, hold the map in peripheral memory, the comments write themselves.
- **The principles — what each one catches at review time.**
  - *#1 Co-locate by feature, not by layer.* Diff signature: a new file landing in a layer folder (`/utils/`, `/services/`, `/types/`) instead of the feature module it belongs to. The comment: "this belongs in `/lib/<feature>/`, see Lesson 5.1.1."
  - *#2 The schema is the source of truth.* Diff signature: a hand-written type that duplicates a Drizzle/Zod shape instead of inferring from it; a hardcoded option list that the schema already enumerates. The comment: derive from the schema.
  - *#3 Pure functions in `/lib`, side effects at named boundaries.* Diff signature: a `db.insert` or `fetch` call inside a `/lib/<feature>/` helper; a Server Action that doesn't own the side effect; business logic inside a route handler. The comment: move the effect to the action seam.
  - *#4 Name for intent, not implementation.* Diff signature: `handleClick`, `processData`, `helper`, `utils`, `manager`. The comment: name what the function *means*, not what it *does*.
  - *#5 Use framework conventions; don't invent.* Diff signature: a custom router, a hand-rolled action-call wrapper, a homemade form library when native forms and Server Actions cover the case. The comment: the framework already has this.
  - *#6 Prefer explicit over magic.* Diff signature: a global side-effect import that mutates module-level state; a magic string driving control flow; a directive missing on a file that needs it. The comment: name the boundary.
  - *#7 Make impossible states unrepresentable.* Diff signature: a `boolean` flag pair that admits a contradictory state (`isLoading: true, data: <value>`); an optional pair where both fields should be required-together; a `string` field that should be a discriminated union. The comment: tighten the type.
- **The SaaS patterns — what each one catches at review time.** The fifteen patterns each pin a specific concern; the lesson runs them as one-liners. Examples (full list named in the lesson, every item gets one line):
  - *#1 Tenant-aware query.* Diff signature: `db.select().from(table)` without the org filter. Comment: use `tenantDb(orgId)`.
  - *#2 Authed action wrapper.* Diff signature: a Server Action that opens with manual `auth()` and `if (!user) throw`. Comment: wrap in `authedAction(role, schema, fn)`.
  - *#4 Billing carve-out.* Diff signature: `import Stripe from 'stripe'` outside `/lib/billing/`. Comment: go through the billing interface.
  - *#5 Webhook ingestion.* Diff signature: a webhook handler that doesn't dedup on the provider's event id; missing idempotency. Comment: add the dedup key.
  - *#6 Result return shape.* Diff signature: a Server Action that `throw`s on a validation failure instead of returning `{ ok: false, error }`. Comment: return Result, throw only at the framework edge.
  - *#7 URL state list view.* Diff signature: filter/sort/page state stored in `useState` instead of `searchParams`. Comment: lift state to the URL.
  - *#8 Cache decisions.* Diff signature: a fetch without an explicit cache directive when the data is mutable; a `revalidatePath` missing after a mutation. Comment: name the cache decision.
  - *#9 Soft delete and archive.* Diff signature: a `DELETE FROM` that should be a `deletedAt` update; a unique constraint that doesn't account for soft-deleted rows. Comment: soft-delete or partial-unique.
  - *#10 Notification layer.* Diff signature: a direct Resend or Twilio call inside a feature module. Comment: route through the notification layer.
  - *#11 Expand-migrate-contract.* Diff signature: a `NOT NULL` on an existing column, a column rename, a column drop landing in one PR. Comment: split into the three-step cadence.
  - *#12 Security baseline.* Diff signature: untyped `JSON.parse` of request body; missing CSRF on a mutating route; secret in a client component. Comment: name the baseline being violated.
  - *#13/#14 Time and i18n.* Diff signature: `new Date()` in a server component for user-visible formatting; a hardcoded English string in a translatable surface. Comment: use the primitives.
  - *#15 Performance vigilance.* Diff signature: a list query without pagination; an N+1 on a relation that has a join helper. Comment: paginate or join.
- **The "what doesn't get a comment" cut — the reviewer's negative space.** Formatting (the formatter owns it). Bikeshedding on variable casing the linter already accepts. Personal style preferences ("I would have written it as a ternary"). Architectural rewrites that aren't load-bearing for this PR ("you could have used the visitor pattern here"). The review's job is to defend the principles and patterns; novel-architecture proposals belong in a separate design doc or ADR thread, not in a line comment on someone else's PR. Name this explicitly: the senior reviewer practices *restraint* on the diff in front of them.
- **PR size and the review's reach.** The reviewer's accuracy collapses past a certain diff size — public data and team practice both peg the threshold around 400 lines of meaningful change. Past that, the reviewer either rubber-stamps or misses load-bearing issues. The senior reflex: if the PR is too big, the right comment is "split this — the review can't catch what it can't read." Not a soft suggestion, a structural one. The lesson states the threshold once.
- **The PR description and the "what am I reviewing" check.** Before reading the diff, the reviewer reads the PR description and asks: *do I understand what this PR claims to do, and do I have the context to verify the claim?* If the description is one line and the diff is 600 lines, the first comment is "what does this PR change and why" — not a line comment on line 47. The description carries the contract for the review; an empty description is a contract gap.
- **Tests as part of the diff, not separate.** The reviewer reads tests with the same posture as production code: do the assertions check behavior the user cares about, or do they only verify the implementation the test was written alongside? A test that asserts the implementation tautologically (the mock was called, the spy fired) is not a test, it's a snapshot. The lesson states the cut: tests defend contract, not implementation. Surface mocks that mock the thing under test as a watch-out.
- **Security and the always-look-for items.** A short, non-exhaustive list the senior runs every review: secrets in client-bundle paths (anything imported from a `'use client'` file that reads `process.env.STRIPE_SECRET_KEY`); missing tenant filters on queries; user input crossing into SQL/HTML without validation; auth checks living in components instead of the action seam; PII in logs. Not a security re-teach — a five-item scan, the deep treatment lives in Unit 17.
- **The performance scan — when it earns its weight.** The reviewer doesn't profile every PR. The scan triggers when the diff touches a list query (pagination?), a join (N+1?), a render path (a server component fetching on every render?), or a hot loop. The threshold: if the diff signature matches a known performance pattern, the comment is one line and points at the lesson; otherwise it's not the review's job.
- **The "first reviewer is the linter and CI" frame.** Anything CI catches is not the human reviewer's beat. The senior reviewer's time is too expensive to spend on what `tsc`, `biome`, or the test suite would catch. Beyond those structural defaults, name additional CI signals to consider — a type-coverage check (owned by Chapter 19.2), a migration linter (owned by Chapter 21.4), an env-parity linter (owned by Chapter 17.2) — as possible promotions when a class of comment recurs. State the corollary: when a class of comment recurs across PRs (the same forgotten check, the same missing wrapper), the senior promotes it to a lint rule or a CI check. Reviews encode patterns, the patterns become structural enforcement; the reviewer's beat narrows as the structural net widens.
- **Reviewing AI-generated code — the same surface, the same posture.** When a coding agent opens a PR, the diff lands in the same queue and gets the same review. The agent's failure modes have a characteristic shape (re-inventing a wrapper the codebase already has, adding plausible-looking tests that don't exercise the contract, drifting from the conventions in `AGENTS.md`). The reviewer doesn't soften the bar; the bar is the bar. Name the property once: agent code is reviewed by the same checklist, and the checklist works because the principles and patterns are stable.
- **The review's tempo — first response, total turnaround.** A review that arrives a week after the PR opens is a review that didn't happen — the author moved on, the context evaporated, the merge happens under pressure without the review's input. The senior reflex: first response within a business day, ideally within a few hours; total turnaround on small PRs same-day. State the asymmetry: a slow review costs more than a fast imperfect one, because the alternative is no review.
- **What the lesson is and isn't.** The lesson is a *Decision/Reference* hybrid — it lays out the principle-and-pattern map as the reviewer's reference, and it teaches the prioritization decision (which layer gets the attention first). It is not a checklist tour disconnected from the course's earlier work; every item points back at the lesson that owns the principle or pattern.
- **Watch-outs:** treating review as a power play ("gatekeeping") — the review serves the codebase, not the reviewer's ego; reviewing for personal style instead of established convention — the codebase has conventions, review against those; missing the security and correctness layer because the reviewer started at line 1 and ran out of attention by line 100 — top-down on the stack, not top-down on the file; nitpicking AI-generated code disproportionately because "the AI should get it right" — same bar for everyone; rubber-stamping the senior teammate's PR because "they know what they're doing" — same bar for everyone the other direction; reviewing every PR with the same depth — the diff size and risk should set the depth; letting CI-catchable comments creep into human review — promote them to lint instead.

What this lesson does not cover:

- How to phrase the comment once you've decided to leave one — Lesson 22.3.2.
- The PR review project (an actual seeded PR to review) — Chapter 22.4.
- Re-teaching the seven principles or fifteen patterns — they live in the lessons that established them; this lesson references, not duplicates.
- Specific PR-review tooling tours (Graphite, GitHub Copilot review, CodeRabbit, Qodo) — out of scope; the lesson teaches the posture, not the platform.
- Branching strategies and merge-queue mechanics — out of scope.

Estimated student time: 50 to 60 minutes. The chapter's load-bearing lesson — the principle/pattern map is the deliverable, the rest is the prioritization stack and the watch-outs.

## Lesson 22.3.2 — The comment that lands

The four-part comment anatomy, the five severity labels (`blocking:`, `suggestion:`, `question:`, `nit:`, `praise:`), the blocking-vs.-suggesting cut, and the language of disagreement that keeps reviews actionable.

Topics to cover:

- **The senior question.** You've decided a line in the diff needs a comment. What do you write, in what shape, at what severity, so the author can act on it in five seconds instead of debating it for thirty minutes? The default failure mode is the comment that's *technically right but unactionable* — too vague, wrong severity, mixed up "I wouldn't" with "this is broken," buried the fix under three paragraphs of context. The senior reflex is a small, opinionated comment shape that the author parses without translation.
- **The comment anatomy — the four-part shape.** Every comment fits a small structure: *severity label*, *the observation*, *the reason*, *the proposed action or question*. The label tells the author whether the comment blocks the merge. The observation names the line or behavior. The reason names the principle, pattern, or failure mode (link the lesson when relevant). The action either proposes a fix or asks a question that the author answers. Four parts, often one or two sentences total. State the shape, give two or three sample comments in the shape, move on.
- **The severity labels the course uses.** The convention is small and stable. The reviewer prefixes the comment with one of:
  - *`blocking:`* — must change before merge. Used for correctness, security, principle/pattern violations the codebase has paid to establish.
  - *`suggestion:`* — strong recommendation, would-merge-with-this-discussed. Used when the alternative is genuinely better but the current code is not broken.
  - *`question:`* — the reviewer doesn't know if this is right and wants the author to explain. Often resolves into either nothing (the explanation is fine) or a blocking/suggestion after the answer.
  - *`nit:`* — non-blocking polish; the author can ignore. Used sparingly because nits drown signal; one or two per PR, not twelve.
  - *`praise:`* — used when something genuinely deserves it (a clever simplification, a clean factoring). Real praise is a free morale win; rote praise (`// great job!`) is condescending. The lesson states the cut: praise specific things, never reflexively.
- **The Conventional Comments shape — the named external standard.** The course's labels above are a pragmatic subset of *Conventional Comments* (the public spec at conventionalcomments.org). Name the standard once so the student can adopt the full vocabulary (`issue:`, `thought:`, `chore:`, `todo:`) when joining a team that uses it. State the property the standard buys: severity is parseable without reading the comment body, so the author triages quickly. The course's smaller subset is a default; the full spec is the conditional.
- **The "blocking vs. suggesting" cut — the load-bearing distinction.** The single most useful discipline in code review: the reviewer knows, *before writing the comment*, whether the line must change. Blocking comments name an objective failure (the principle violation, the security gap, the contract drift); suggesting comments name a subjective preference (the cleaner factoring, the alternative pattern). Mixing them is the source of most review friction — the author treats every comment as blocking and over-corrects, or treats every comment as optional and misses the real failures. The senior reflex: label every comment with the severity it deserves, *especially* the comments that aren't blocking. The author then knows what to act on.
- **The language of disagreement — opinionated, evidence-led, short.** When the reviewer thinks the code is wrong, the comment names the failure mode, points at the principle or pattern, and proposes the fix or asks the question. No hedging that obscures the position (`maybe consider perhaps possibly`), no aggression (`this is wrong, see the docs`), no novel-length essays. The voice: a peer talking to a peer who shares the codebase's conventions. State the bad shapes ("hedged into uselessness," "blame-laden," "essay-length") and the good shape (one to three sentences, position + reason + ask) by contrast.
- **"You" vs. "we" vs. "the code" — the second-person reflex.** The comment talks about the *code*, not the *author*. "This bypasses `tenantDb` — let's route the query through the helper" beats "You bypassed `tenantDb`." Not because the author can't take feedback, but because the code is what's being reviewed and the de-personalization keeps the conversation on the change. The lesson states the heuristic: when in doubt, address the code; reserve "you" for praise and direct questions ("can you explain why this needs the manual auth check?").
- **Asking a question vs. asserting a position.** When the reviewer isn't sure, the question is the right shape: "is there a reason this doesn't go through `authedAction`?" The question signals that the author may have context the reviewer doesn't, and resolves quickly. When the reviewer is sure, the question shape is dishonest ("can we use `authedAction` here?" when the reviewer means "this must use `authedAction`") — it adds a step and softens the signal. State the cut: questions when uncertain, positions when certain, never use questions to soften positions.
- **The suggestion block — when GitHub's `suggestion` markdown earns its weight.** Most platforms (GitHub, GitLab, Graphite) support a `suggestion:` markdown block that lets the author apply the fix with one click. Use it for short, mechanical fixes (a renamed identifier, a missing `await`, a one-line wrapper substitution) — saves the author the diff. Don't use it for large rewrites; the comment is the wrong surface for those. State the threshold: under five lines and one obvious correct edit, use suggestion; otherwise, prose.
- **The "one comment, one concern" rule.** A comment that bundles three observations forces the author to address them as a unit — accepted or rejected together. Split them: one line each, one concern each. The author can ack, fix, push back per comment; the thread stays small and actionable. State the corollary: if three concerns cluster on the same line, the line is doing too much and the comment is "this function needs to split before any of these can be addressed cleanly."
- **Mentorship through review — the secondary channel.** Junior teammates learn faster from review comments than from any other channel because the feedback is on their own code. The senior reflex: when explaining a principle violation, link the lesson or the convention doc that established it (`see Chapter 10.1 on `tenantDb`'). The link adds a minute to the comment and saves the author an hour of context-hunting. The discipline scales the team's competence over months. State the property; don't moralize.
- **Praise as a deliberate move, not a reflex.** Real praise (named, specific, on a non-trivial choice) is a morale win and a signal of taste. Reflex praise ("looks good!", "nice work!") is noise. The senior reflex: when the author made a non-obvious right call, name it specifically — "good call using `authedAction` here even though the path is internal-only; keeps the pattern uniform." Three praise comments per quarter that the author remembers; not thirty rote ones.
- **Receiving review — the symmetric posture.** The lesson covers the reverse: when the senior is the author, not the reviewer. The reflex is to respond to every comment (even with a thumbs-up), to push back on incorrect comments with evidence rather than defer, and to mark resolved threads honestly (resolving "I'll do it next PR" is dishonest; resolving "fixed in commit X" is correct). State the asymmetry: a reviewer's bad day shouldn't merge into the codebase, and the author defends the design where they have evidence.
- **The "approve with comments" vs. "request changes" calculus.** Most platforms have three review actions: approve, comment, request changes. The senior cut: `request changes` is reserved for blocking comments — if a comment is `blocking:`, the review state should reflect that. `approve with comments` is for suggestions and nits where the author can merge after acking; the author is trusted to address or not. `comment only` is the default during back-and-forth before the final state. State the property: the review state matches the comment severities, not the reviewer's mood.
- **Async review hygiene — when to escalate to a call.** Most reviews resolve async. When a thread crosses three round-trips without convergence, the disagreement is structural and the surface is wrong — pull it into a 15-minute call, decide, capture the decision in the PR (or as an ADR if it's architectural). State the threshold: three round-trips, escalate. The PR comment thread is not the place to debate large design moves.
- **The reviewer's "what would I want to see" mirror.** Before submitting the review, the senior re-reads the comments as if they were the author: are the severities right, are the asks actionable, are the reasons named? The mirror catches the comment that landed as "I don't like this" rather than "this conflicts with pattern #N." Sixty seconds, every review, before submit.
- **Reviewing in the era of AI-generated comments — calibration.** Tools like GitHub Copilot review, CodeRabbit, and Qodo now leave their own comments on PRs. The senior reflex: read them with the same skepticism as a junior reviewer's comments — sometimes load-bearing, often noise. The human reviewer dismisses the noise, ack's the right ones, and doesn't outsource judgment. State the property once; don't make the lesson about the tools.
- **The "fast first response" discipline.** A review that lands within a few hours, even if it's an early "I'll do a full pass tomorrow, here are three quick concerns," keeps the author moving. The senior reflex: first contact fast, depth follows. Reviews that sit for days train the team that PRs don't matter and the author should merge without them.
- **Watch-outs:** mixing `nit:` and `blocking:` so the author can't tell which to fix first; writing essay-length comments where a one-liner with a link would do; using `question:` to soften a position the reviewer is certain about (epistemic cowardice); the "drive-by approval" — clicking approve without reading, especially on a teammate's PR you trust; the "drive-by request-changes" — blocking on style; bundling three concerns into one comment so the author has to address them as a unit; sarcastic or snarky tone that ages badly when the PR archive is read months later; rubber-stamping AI-generated PRs because they "look clean" — same bar; demanding rewrites that aren't load-bearing for this PR scope ("you could have refactored X while you were here" — that's a different PR); resolving threads on the author side without addressing the comment (resolved-as-ignored); responding to a `blocking:` comment with a thumbs-up and merging anyway — the blocking label is structural.

What this lesson does not cover:

- What to look for in the first place (the principle/pattern map) — Lesson 22.3.1.
- The end-to-end PR review project — Chapter 22.4.
- Specific platform UI tours (GitHub PR UI, Graphite, GitLab) — out of scope; the conventions are platform-agnostic.
- Team-process meta (review-load rotation, ownership files, CODEOWNERS) — out of scope; the lesson is about the comment, not the routing.
- Synchronous-review/pair-programming protocols — out of scope; the lesson is async-review-default.

Estimated student time: 40 to 50 minutes. A *Pattern* lesson — the comment anatomy and the severity labels are the deliverables, the rest is the disagreement language and the watch-outs.

## Lesson 22.3.3 — Quizz

Top 10 topics to quiz:

- The review stack and the order the senior runs it — correctness/security first, principles, patterns, tests/contracts, style/polish last; why starting at line 1 of the diff is the wrong default.
- The principle-and-pattern map as the reviewer's reference — the diff signature that signals each principle violation (#1–#7) and each pattern bypass (#1–#15), and the lesson that owns each.
- The "what doesn't get a comment" cut — formatting (the formatter), naming the linter accepts, personal-style preferences, off-topic refactors; the review's beat is principles and patterns, not aesthetics.
- The PR-size threshold and the "split this" structural comment — past ~400 meaningful lines, review accuracy collapses; the right comment is to split, not to power through.
- The CI/lint-first frame — anything `tsc`, the formatter, the test suite, or a custom lint rule can catch is not the human reviewer's beat; recurring comments get promoted to structural checks.
- The four-part comment anatomy — severity label, observation, reason, proposed action or question; usually one or two sentences total.
- The severity labels the course uses (`blocking:`, `suggestion:`, `question:`, `nit:`, `praise:`) and the Conventional Comments standard they're a subset of; how the label drives the author's triage.
- The blocking-vs.-suggesting cut — objective failure (must change) vs. subjective preference (would be better); mixing them is the source of most review friction.
- The language of disagreement — opinionated, evidence-led, short; address the code, not the author; questions when uncertain, positions when certain; never use questions to soften positions.
- The receiving-review posture — respond to every comment, push back with evidence on incorrect comments, resolve threads honestly; the "request changes" vs. "approve with comments" calculus tracks the comment severities, not mood.
