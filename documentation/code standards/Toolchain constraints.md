# Toolchain constraints

Config knobs the toolchain forces, breaks, or mishandles. Plans locking these must follow the listed resolution. Append entries when reviewer/corrector finds a new framework conflict.

## Next.js 16

- **`tsconfig.json: "jsx"`** — `next build` rewrites `"preserve"` → `"react-jsx"` ("mandatory change"). Pin `"jsx": "react-jsx"`.
- **`next-env.d.ts`** — auto-regenerated with double-quoted imports; conflicts with Biome `quoteStyle: 'single'`. Exclude: `"files": { "includes": ["**", "!next-env.d.ts"] }`.
- **Multi-lockfile warning** — emitted when a parent dir has its own lockfile (monorepo). Silence with `turbopack: { root: __dirname }` in `next.config.ts`.
- **`tsconfig.json` auto-injection** — each `next build` appends type-dir globs to `include` and reformats arrays multiline. Next 16.2.x injects **two** globs across successive builds: first adds `".next/types/**/*.ts"`, second adds `".next/dev/types/**/*.ts"`. Only `.next/types/` is actually generated; `.next/dev/types/` is a phantom Next still re-requests. Bake **both** into the locked `tsconfig.json` `include` so it's drift-free/idempotent from the first build (committing only one leaves it drifting). Also suggests `"allowJs": true` but *only when absent* — an explicit `"allowJs": false` survives. Of printed changes, only `jsx` / `module` / `moduleResolution` / `esModuleInterop` / `isolatedModules` / `resolveJsonModule` are "mandatory"; the include addition and `allowJs` are "suggested." Bake the include addition and the `jsx: "react-jsx"` pin into the locked `tsconfig.json`.
- **Cache Components + slot `default.tsx` re-export** — under `cacheComponents: true`, a parallel-route slot `default.tsx` re-exporting its `page.tsx` (`export { default } from './page'`) propagates the page's dynamic data reads into sibling routes' prerender. The slot-scoped `@<slot>/loading.tsx` is a Suspense boundary only for the slot's *own* `page.tsx`, not when composed into a sibling. Build fails with "Uncached data was accessed outside of `<Suspense>`" on every sibling route. Fix: add segment-level `loading.tsx` at the parallel-routes parent (`app/<segment>/loading.tsx`), or move dynamic reads behind `'use cache'`. Hand-written `<Suspense>` is last resort; segment-level `loading.tsx` is the cleanest seam.
- **Parallel routes implicit `children` slot needs `default.tsx`** — when a layout declares named slots, the implicit `children` slot also needs its own `default.tsx` at the same level. Without it, URLs matching only the named slots render broken: HTTP **200**, named slots render, but the children area shows Next's "This page could not be found" fallback (a `NEXT_HTTP_ERROR_FALLBACK;404` template inlined into the slot's DOM). Always enumerate `app/<segment>/default.tsx` alongside `@<slot>/default.tsx`.
- **`typedRoutes: true` + dynamic href strings** — template-literal `href={`/path/${id}`}` and `router.push("/path?x=y")` with built strings fail typecheck (`Route` is a branded string union). Caveat: the augmentation lives in `.next/types/link.d.ts`, but Next 16's default `next-env.d.ts` only references `routes.d.ts`. The `next` TS plugin loads `link.d.ts` in the IDE, but **`tsc --noEmit` silently passes** unless plans add `/// <reference path="./.next/types/link.d.ts" />` to `next-env.d.ts` (or include it manually). Fix: cast site-by-site (`as Route` or `as Parameters<typeof router.push>[0]`), or route href construction through a typed helper. Plans flipping `typedRoutes: true` must pre-state both the reference-path wiring and the cast pattern.
- **`new Date()` in Server Components under Cache Components** — reading `new Date()` (or any IO clock) in a Server Component without first awaiting a dynamic source breaks `next build` under `cacheComponents: true`. Common in inherited footers (`© {new Date().getFullYear()}`). Fix: hardcode the literal year, move into a Client Component, or precede with an `await` on a dynamic source.
- **A plain page's uncached request-time reads need a `<Suspense>` seam under Cache Components** — a Server Component page that reads uncached, request-time data (a DB query, `fetch` with `no-store`, etc.) in its body fails `next build` at prerender with `Uncached data was accessed outside of <Suspense>`, even when it `await searchParams`/`await params` first (that marks the route dynamic but does not wrap the reads). Cheapest fix: ship a segment-level `app/<segment>/loading.tsx` (default-export component) — the route then builds as Partial Prerender (static shell + dynamic streamed content). Hand-written `<Suspense>` or `'use cache'` are alternatives; segment `loading.tsx` is the cleanest. Plans whose pages read live data per request must enumerate the `loading.tsx` in the file tree.
- **`reactCompiler: true` requires `babel-plugin-react-compiler`** — Next 16 promoted `reactCompiler` to stable but doesn't bundle the Babel plugin it runs through. Setting `reactCompiler: true` without `babel-plugin-react-compiler` in `devDependencies` fails `next build` immediately: "Failed to resolve package babel-plugin-react-compiler while attempting to resolve React Compiler." Pin `babel-plugin-react-compiler` (latest stable `1.0.0`) as a devDep.
- **pnpm blocks `sharp`'s native build → `next build` fails on clean install** — `next` pulls `sharp` (image optimizer) transitively, and pnpm 10+ (pinned `11.3.0`) won't run its native build script unless approved. `pnpm install` only warns, but Next 16's `next build` re-runs `pnpm install` (dep-status check) and surfaces the ignored build as a hard `ERR_PNPM_IGNORED_BUILDS`, failing `pnpm verify`. Surfaces only on a **cold** `node_modules` (a warm tree passes), so it ships latent. Ship `pnpm-workspace.yaml` (no `packages:` key — approval only) with exactly `allowBuilds:\n  sharp: true` in **both** `solution/` and `start/`, list it in the File tree, and overwrite (not keep) the stub `create-next-app` emits. **pnpm 11 removed `onlyBuiltDependencies`, `ignoredBuiltDependencies`, `neverBuiltDependencies`, and `onlyBuiltDependenciesFile` — `allowBuilds` (a map of `pkg: true|false`) is the sole replacement**, so do NOT ship `onlyBuiltDependencies` (dead key) and do NOT leave `sharp` under the emitted `ignoredBuiltDependencies` list or an unresolved `allowBuilds: { sharp: set this to true or false }` placeholder — either keeps the build blocked. `strictDepBuilds` defaults to `true` on this line, so an unlisted/placeholder native build is a hard error, not a warning. The `package.json#pnpm` field is no longer read. Needed even without `next/image` — it's pnpm's ledger, not a runtime dep. (Verified against pnpm 11.3.0, June 2026.)
- **`tsconfig.json: "skipLibCheck"`** — `next build` adds `"skipLibCheck": true` every run (ships in Next 16's recommended config; prints "reconfigured your tsconfig.json … skipLibCheck was set to true"). Omitting it drifts the committed file on first build and makes standalone `tsc --noEmit` fail when nested under a monorepo whose parent `node_modules` hoists incompatible `@types/*` (e.g. `@types/mdx` → "Cannot find namespace 'JSX'"). Bake `"skipLibCheck": true` into the locked `tsconfig.json`.
- **`tsconfig.json: "incremental"`** — `next build` injects `"incremental": true` on the first run (prints "incremental was set to true" under the suggested-values block). It is idempotent after that, but a committed `tsconfig.json` lacking it drifts on the first build. Bake `"incremental": true` into the locked `tsconfig.json`. It composes with `"noEmit": true` (standalone `tsc --noEmit` stays green).
- **Parallel-route slot returning a fragment flattens into the parent layout** — a `@<slot>` `page.tsx`/`default.tsx` returning `<><A /><B /></>` adds no DOM wrapper, so when the layout drops `{slot}` into a CSS `grid`/`flex` container each child becomes its own grid/flex item — a region meant to fill one slot silently splits into several cells (e.g. a two-column grid receiving 3 children: filter + list + detail). Each multi-child slot return must resolve to a single wrapping element. (Chapter 035.)

## TypeScript 6

- **`baseUrl` is a hard error** — `typescript@latest` is 6.x (6.0 GA). TS 6 turns the `baseUrl` deprecation into an error-level diagnostic: `tsc --noEmit` emits `error TS5101: Option 'baseUrl' is deprecated and will stop functioning in TypeScript 7.0` and **exits 1**, failing any `pnpm verify` that runs standalone `tsc`. (Notably `next build`'s own typecheck does NOT surface this — only standalone `tsc` does, so it ships latent if the gate is build-only.) Fix: drop `baseUrl` entirely. `"paths": { "@/*": ["./src/*"] }` resolves correctly without it under `moduleResolution: "bundler"` (paths resolve relative to the tsconfig location), and Next does not re-inject `baseUrl`. Alternative (not preferred): keep `baseUrl` and add `"ignoreDeprecations": "6.0"`.

## Biome 2.4

- **`css.parser.tailwindDirectives`** — Tailwind v4 directives (`@theme`, `@apply`, `@custom-variant`, `@import "tailwindcss"`) emit parse errors without this flag. Always include `"css": { "parser": { "tailwindDirectives": true } }`.
- **`lint/performance/noImgElement`** — top-level performance rule warning on raw `<img>` (no `next` domain exists in Biome 2.4; framework filtering is via `--only=next`, not a domain block). Plans mandating raw `<img>` (theme-aware swaps, plain logos) must override: `"lint": { "rules": { "performance": { "noImgElement": "off" } } }`.
- **`files.includes` folder-glob normalization** — `useBiomeIgnoreFolder` rejects `"!folder/**"` and `biome check --write` auto-strips to `"!folder"`. Plans pinning the `/**` suffix fail `biome ci` until rewritten. Write ignores without trailing glob: `["!.next", "!node_modules"]`, not `["!.next/**", "!node_modules/**"]`.
- **`<ul role="list">` Safari VoiceOver workaround** — the standard Safari fix for `list-style: none` stripping AT list semantics trips `lint/a11y/noRedundantRoles` (only — `useSemanticElements` doesn't target this in Biome 2.4). Fix: suppress per-element with `biome-ignore`, or disable the rule globally when the project commits to the workaround. Architects mandating the role must pre-declare the chosen reconciliation.
- **`lint/suspicious/noArrayIndexKey` on placeholder lists** — recommended-on, errors on `Array.from({ length: 8 }).map((_, i) => <li key={i}>…</li>)` even when static and non-reorderable (biomejs/biome#9469). A `biome-ignore` above `.map(...)` does not suppress the diagnostic on the nested `<li key={i}>` line. Fix: declare a stable string-key tuple (`const KEYS = ['row-0', …] as const`) and map over that, or replace with a fixed number of explicit elements. Skeleton/placeholder components should ship the tuple pattern up front.
- **`pnpm <script> --write` doubles the flag** — projects pin `"check": "biome check --write ."`, so `pnpm check --write` runs `biome check --write . --write`, rejected with "argument `--write` cannot be used multiple times in this context." Write `pnpm check` (the script already carries `--write`); never append the flag again.

## Shadcn UI

- **Primitive formatting** — `pnpm dlx shadcn add` writes primitives with double quotes and `import * as React` (not `import type`), triggering Biome warnings on first lint. Accept on first commit; let `pnpm check --write` normalize.
- **Radix umbrella package** — shadcn's v4-style preset depends on the `radix-ui` umbrella package, not per-primitive `@radix-ui/react-*` (older `default`/`new-york` styles still use per-primitive). For v4-style installs, pin `radix-ui@^1.4.3` instead of enumerating per-primitive deps.
- **CLI v4 init defaults to Base UI, not Radix — pass `--base radix`** — the March-2026 shadcn CLI v4 replaced the old `style` concept with design-system *presets*. `init -y` still prompts interactively for the preset (hangs in CI); `init -y -d` skips all prompts but its default preset is `base-nova`, which installs **Base UI** (`@base-ui/react`) and writes `"style": "base-nova"` — a different primitive library that clashes with a `radix-ui`-locked plan. To get the Radix umbrella non-interactively, pass `--base radix` (`init -y -d --base radix` → `"style": "radix-nova"`, `iconLibrary: lucide`, `radix-ui` kept). `add` also needs `-y` (it prompts to create `components.json` otherwise). Side-effect: a bare `init` (no existing project marker) adds `shadcn` (and sometimes `msw`) to `package.json` `dependencies` — strip them, they are registry tooling, not app deps. (Verified against `shadcn@4.10`, June 2026.)
- **`Card` has no `asChild` slot** — shadcn's `Card` renders a `<div>` with no `asChild` prop, so it can't be retargeted to `<article>` etc. Plans needing a semantic root (e.g. `FeatureCard` typed as `ComponentProps<'article'>`) must inline `Card`'s container classes onto the chosen element and nest `CardHeader`/`CardTitle`/`CardDescription` inside, rather than wrapping `Card`. Compose by hand, don't double-wrap.
- **Radix primitives ship `'use client'`** — `dialog`, `sheet`, `separator`, and other Radix-wrapping primitives carry the directive. Any verification counting `'use client'` files must include the installed `components/ui/*` set, not just authored islands.

## eslint-plugin-react-hooks 7 (Compiler diagnostics layer)

- **`flat.recommended` has no `files` glob or parser** — `reactHooks.configs.flat.recommended` is a bare rules/plugins object with no `files` and no `languageOptions.parser`. Spread alone into a flat config (`export default [reactHooks.configs.flat.recommended, …]`) on a TS-only project, ESLint 9 matches nothing (default flat-config extensions exclude `.ts`/`.tsx`), so `eslint .` exits 0 having linted zero files — the layer is silently inert. Plans wiring this layer must add `files: ['**/*.{ts,tsx}']` and `languageOptions: { parser: tseslint.parser }` (ship `typescript-eslint@^8.60` as a devDep) inside the spread, or the diagnostics never run. Verify non-vacuously: a `--print-config`/`-c` run on a `.tsx` source must report it as checked, not "File ignored because no matching configuration was supplied."
- **`react-hooks/set-state-in-effect` false-positives on the next-themes mount gate** — the rule (error in `flat.recommended`) flags the canonical `useEffect(() => setMounted(true), [])` hydration gate (facebook/react#34743, pacocoursey/next-themes#374). Plans locking that mount-gate AND wiring the hooks layer must turn it off (`'react-hooks/set-state-in-effect': 'off'`); the remaining compiler diagnostics (`set-state-in-render`, `rules-of-hooks`, `purity`, memoization) stay on.

## lucide-react 1.x

- **All brand icons removed** — lucide-react 1.0 dropped every brand glyph (`Github`, `Twitter`, `Linkedin`, `Youtube`, `Facebook`, `Instagram`, `Slack`, `Gitlab`, …); importing them throws "export X doesn't exist in target module" and fails `next build` (trademark/maintenance scope; Simple Icons is the recommended brand-logo replacement). ## Tailwind CSS 4

- **`scale-*` sets the CSS `scale` property, not `transform`** — Tailwind v4's `scale-105` compiles to `scale: var(--tw-scale-x) var(--tw-scale-y)` (the standalone `scale` property), so `getComputedStyle(el).transform` is always `"none"` for a scaled element. Rendered checks asserting a scale/lift must read the computed `scale` property (e.g. `"1.05"`), never `transform`. An assertion written against `transform` passes trivially in both states and is not falsifiable.
- **`motion-reduce:` must be stacked under a responsive variant to override it** — to suppress a responsive utility under reduced motion (`md:scale-105` ↔ no scale), a bare `motion-reduce:scale-100` is **inert**: v4 emits unprefixed-state variants (`motion-reduce:`) earlier in source order than responsive variants (`md:`), and since both target the same property with equal (single-class) specificity, at `md`+ the `md:scale-105` rule wins by source order. Stack the guard as `md:motion-reduce:scale-100` — same specificity, later in source — so it overrides at `md`+ under reduced motion while leaving the non-reduced lift intact. Plans pairing a responsive transform/scale with a reduced-motion fallback must stack the fallback under the same breakpoint.

Unified package: `react-email@^6.5.0` exports every primitive (`Body`, `Button`, `Container`, `Head`, `Heading`, `Html`, `Img`, `Link`, `Preview`, `Row`, `Section`, `Tailwind`, `Text`, …), plus `pixelBasedPreset` and `render`, and ships the `email` CLI bin — import from `react-email`, never the deprecated `@react-email/components`. (Verified against `react-email@6.5.0`.)

- **`email dev` blocks on an interactive `@react-email/ui` install prompt** — the preview server's UI is NOT bundled into `react-email` and is NOT pulled in transitively. Running `email dev` on a tree without `@react-email/ui` prints `To run the preview server, the package "@react-email/ui" must be installed. Would you like to install it? (Y/n)` and **hangs waiting for input** (fails outright in CI/non-interactive shells). It is build-time tooling only — never imported by app code, so it stays out of `dependencies`.

Write `` <title>{`Welcome to ${appName}`}</title> ``. React Email's own `<Preview>` component tolerates the split form (it is a component, not a raw `<title>`).
- **Preview server runs in its own `.react-email` working dir** — templates under `src/emails/` must import siblings with same-folder **relative** paths (`./email-tailwind-config`, `./components/email-layout`), never `@/` aliases (may not resolve), and read NO `process.env`/`NEXT_PUBLIC_*` (undefined there) — keep chrome on module-level literal constants. Verified: a template doing so renders identically in `email dev`, a Next inspector iframe (`render(<T {...PreviewProps} />)`), and a real send.

## Better Auth 1.6 (Drizzle adapter + organization plugin)

- **Plugin-owned ids are `text`, not `uuid` — a `uuid` FK to them fails migration apply** — `auth:generate` emits every Better Auth table id (`user.id`, `organization.id`, `member.id`, `invitation.id`, …) as `text` (its id strategy is a base62 random string, e.g. `xBibNvkrZLOcaOc14ILubYLDxls2vhXy`, not a UUID). A hand-authored table whose FK column is `uuid` referencing one of these `text` ids passes `tsc` (Drizzle's `.references()` does not check column-type compatibility) but `drizzle-kit migrate` fails at apply: `foreign key constraint … cannot be implemented … incompatible types: uuid and text`. Declare FK columns to Better Auth tables as `text`. A standalone PK on the same table (no incoming FK) may still be `uuid`/`uuidv7()`. Knock-on: any RLS `current_setting(...)::uuid` cast comparing such a column must drop the cast (the column is `text`).
- **A hand-rolled `tx.insert` into a plugin table must supply `id` (and any no-default column)** — the plugin normally generates `id` and sets `createdAt` in its own write path. When a plan mandates direct Drizzle `tx.insert(member|invitation)(…)` (to keep an audit row in the same transaction, bypassing `auth.api.*`), the generated schema's `id text primaryKey()` has **no DB default** and some `createdAt` columns lack `defaultNow()` — the insert hits a not-null violation unless the caller passes them (`id: crypto.randomUUID()`, `createdAt: new Date()`). The seed's direct inserts need the same.
- **`z.uuid()` rejects plugin ids** — validating a Better Auth id (member id, invitation id) with Zod's `z.uuid()` rejects every real value (they are not RFC-4122 UUIDs). Use `z.string().min(1)` for plugin-id inputs in action schemas.
- **`additionalFields` inviter relation is aliased by the FK target, not the field name** — when the organization plugin generates `invitationRelations`, the `one(user, …)` join on `invitation.inviterId` is named `user` (the target table), not `inviter`. `db.query.invitation.findMany({ with: { inviter: true } })` is a type error; use `with: { user: true }` and read the inviter off `.user`. (Better Auth `additionalFields` type-inference on `auth.api.*` returns is separately buggy — better-auth#5122 — so the Drizzle `$inferSelect` table type is the source of truth for plugin columns.)
- **Regenerating clobbers hand-added schema edits** — `pnpm auth:generate` overwrites the output schema file wholesale. An application-only index/constraint added to a plugin table's Drizzle callback is silently dropped on the next regenerate. Add such indexes as a `--custom` migration, never as a callback edit on the generated file.

## Postgres 18 (Docker)

- **Volume mount path** — Postgres 18 stores data under `/var/lib/postgresql/18/docker` (new major-versioned layout). Mount the parent `/var/lib/postgresql` so `pg_upgrade --link` works across major-version bumps; the historic `/var/lib/postgresql/data` is still writable but breaks the upgrade flow. Use:
  ```yaml
  volumes:
    - pgdata:/var/lib/postgresql
  ```
- **`@neondatabase/serverless` cannot reach vanilla Postgres** — the Neon driver speaks HTTP/WebSocket only, not the Postgres wire protocol; against local Docker it errors before connect. For local Docker chapters use the course-pinned `postgres` (postgres-js) driver via `drizzle-orm/postgres-js` — it speaks the wire protocol, bundles cleanly under Turbopack, and needs no `serverExternalPackages` entry.
- **RLS policy `TO authenticated` fails to apply — the role does not exist on Docker** — `drizzle-orm/neon`'s `authenticatedRole` (and `crudPolicy`) emit `CREATE POLICY … TO authenticated`, but `authenticated`/`anonymous` are **Neon/Supabase-provided** roles. Vanilla Docker Postgres has only `postgres`, so `pnpm db:migrate` fails the policy migration with `role "authenticated" does not exist`. drizzle-kit never creates the role. Plans using `authenticatedRole` against local Docker must ship a `--custom` migration that creates it idempotently — `DO $$ BEGIN IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'authenticated') THEN CREATE ROLE authenticated NOLOGIN; END IF; END $$;` — **ordered before** the first policy migration. Roles are cluster-wide, so one create suffices. The app still connects as the superuser `postgres` locally (which has `BYPASSRLS`); the role exists only so the DDL applies and a `SET ROLE authenticated` `psql` session can prove the deny behavior. Per-surface non-owner request roles are a later-chapter reach.

## Drizzle 0.45

Pinned to `drizzle-orm@0.45.x` + `drizzle-kit@0.31.x` until Better Auth ships `drizzle-orm@^1.0` support ([better-auth#6766](https://github.com/better-auth/better-auth/issues/6766), target 1.8+). Constraints on this line:

- **Runtime `casing` lives on the client.** `drizzle({ client, casing: 'snake_case', schema })` snake-cases all columns from one declaration. Per-column overrides via explicit names (`uuid('organization_id')`) are the escape hatch. (Removed in 1.0 — flips to `pgTableCreator((name) => toSnakeCase(name), 'snake_case')`; the second arg is mandatory or columns stay camelCase.)
- **Migration file layout is flat.** `drizzle-kit generate` emits `drizzle/<timestamp>_<name>.sql` plus top-level `drizzle/meta/_journal.json` and `drizzle/meta/<id>_snapshot.json`. Pass `--name <verb>_<noun>` so the suffix is intentional. Renaming the timestamp prefix breaks journal lookup. (1.0 moves to per-migration dirs `drizzle/<timestamp>_<name>/{migration.sql, snapshot.json}` with no top-level `meta/`.)
- **Relations v1 is the API.** `relations(<table>, ({ many, one }) => ({ ... }))` per-table; `db.query.<table>.findMany({ with: { ... } })` is the N+1-safe traversal. (1.0 replaces this with a single `defineRelations(schema, (r) => ({ ... }))`; the `r.one` form defaults to `optional: true` and must be flipped to `false` on NOT NULL FKs.)
- **Zod helpers ship as a separate package.** `drizzle-zod@^0.8` peer-requires `drizzle-orm@>=0.36` and `zod@^4`. `createSelectSchema`/`createInsertSchema`/`createUpdateSchema` import from `drizzle-zod`, not `drizzle-orm/zod`. (1.0 drops the separate package, moving them under `drizzle-orm/zod`.)
- **`drizzle-seed` `with` keys are table names, not relation aliases** — refining a parent with children uses the schema's table binding (`with: { invoiceLines: [...] }`), not the relation alias (`lines:`). Junction tables hand-populated post-seed need `{ count: 0 }` in the refine to suppress the default fill. General across both versions.
- **`drizzle-seed@0.3.x` cannot seed constraint-heavy schemas — use direct inserts, keep only `reset()`.** Verified against `drizzle-seed@0.3.1` + `drizzle-orm@0.45`: (1) any table with a `unique`/composite-unique constraint throws `Error: Values length equals zero.` at `.refine()` init (and `valuesFromArray({ isUnique: true })` throws even with a pool larger than the row count) — [drizzle-orm#4354](https://github.com/drizzle-team/drizzle-orm/issues/4354); (2) the default `numeric` generator emits large NEGATIVE, out-of-precision values that violate `check (col >= 0)` and `numeric(p,s)`; (3) `uuid` PK columns are filled with the generator's own **v4** value (its `uuid` branch fires before its `hasDefault` branch), overriding a `default(sql\`uuidv7()\`)` even when `id` is omitted from the refine — and those v4 ids are NOT RFC-variant-valid, so Zod 4 `z.uuid()` rejects them downstream (breaks cursor/id validation); (4) `with`-fanned children do not inherit the parent's tenant FK (`organizationId`), and per-parent sequential columns (`position`) can't be expressed (independent draws collide on a `(parent, position)` unique). Resolution for schemas with these shapes: call `reset(db, schema)` for the idempotent wipe (works fine), then **direct-insert** every row with a fixed-seed PRNG for determinism, leaving `uuid` PKs to the schema's SQL default (direct inserts honor it; `drizzle-seed` does not). Reserve `drizzle-seed`'s `.refine()` generators for unconstrained scratch tables only.
- **`tsx`/`drizzle-kit` pull `esbuild` → pnpm ignored-build hard-fails `next build`** — adding `tsx` and/or `drizzle-kit` (DB chapters) drags in multiple `esbuild` majors whose native build script pnpm 11.3.0 refuses to run unattended. Even a warm `pnpm install` then exits non-zero with `ERR_PNPM_IGNORED_BUILDS: esbuild@…`, and Next 16's `next build` re-runs the dep-status check and surfaces it as a hard failure (same mechanism as the sharp ledger). esbuild needs NO build step (its native binary ships in the `@esbuild/<platform>` optional dep), so acknowledge-but-skip in `pnpm-workspace.yaml`: `allowBuilds: { sharp: true, esbuild: false }` (do not add `esbuild` to `onlyBuiltDependencies`). Needed in both `solution/` and `start/`.
- **CLI `import.meta.url` entry-point guard breaks on paths with spaces** — `tsx`/`node` scripts that self-detect "run directly" via `if (import.meta.url === \`file://${process.argv[1]}\`)` silently no-op when the absolute path contains a space (course project dirs are `Chapter NNN/...`): `import.meta.url` percent-encodes the space (`Chapter%20041`) while `process.argv[1]` keeps it literal, so the comparison is always false and the script's body (e.g. a seed) never runs but still exits 0. Use `import { pathToFileURL } from 'node:url'; const e = process.argv[1]; if (e && import.meta.url === pathToFileURL(e).href) { … }`.
