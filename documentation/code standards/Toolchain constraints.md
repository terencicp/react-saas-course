# Toolchain constraints

Config knobs the toolchain forces, breaks, or mishandles. Plans locking these must follow the listed resolution. Append entries when reviewer/corrector finds a new framework conflict.

## Next.js 16

- **`tsconfig.json: "jsx"`** — `next build` rewrites `"preserve"` → `"react-jsx"` ("mandatory change"). Pin `"jsx": "react-jsx"`.
- **`next-env.d.ts`** — auto-regenerated with double-quoted imports; conflicts with Biome `quoteStyle: 'single'`. Exclude: `"files": { "includes": ["**", "!next-env.d.ts"] }`.
- **Multi-lockfile warning** — emitted when a parent dir has its own lockfile (monorepo). Silence with `turbopack: { root: __dirname }` in `next.config.ts`.
- **`tsconfig.json` auto-injection** — each `next build` appends type-dir globs to `include` and reformats arrays multiline. Next 16.2.x injects **two** globs across successive builds: first adds `".next/types/**/*.ts"`, second adds `".next/dev/types/**/*.ts"`. Only `.next/types/` is actually generated; `.next/dev/types/` is a phantom Next still re-requests. Bake **both** into the locked `tsconfig.json` `include` so it's drift-free/idempotent from the first build (committing only one leaves it drifting). Also suggests `"allowJs": true` but *only when absent* — an explicit `"allowJs": false` survives. Of printed changes, only `jsx` / `module` / `moduleResolution` / `esModuleInterop` / `isolatedModules` / `resolveJsonModule` are "mandatory"; the include addition and `allowJs` are "suggested." Bake the include addition and the `jsx: "react-jsx"` pin into the locked `tsconfig.json`.
- **Cache Components + slot `default.tsx` re-export** — under `cacheComponents: true`, a parallel-route slot `default.tsx` re-exporting its `page.tsx` (`export { default } from './page'`) propagates the page's dynamic data reads into sibling routes' prerender. The slot-scoped `@<slot>/loading.tsx` is a Suspense boundary only for the slot's *own* `page.tsx`, not when composed into a sibling. Build fails with "Uncached data was accessed outside of `<Suspense>`" on every sibling route. Fix: add segment-level `loading.tsx` at the parallel-routes parent (`app/<segment>/loading.tsx`), or move dynamic reads behind `'use cache'`. Hand-written `<Suspense>` is last resort; segment-level `loading.tsx` is the cleanest seam.
- **Parallel routes implicit `children` slot needs `default.tsx`** — when a layout declares named slots, the implicit `children` slot also needs its own `default.tsx` at the same level. Without it, URLs matching only the named slots render broken: HTTP **200**, named slots render, but the children area shows Next's "This page could not be found" fallback (a `NEXT_HTTP_ERROR_FALLBACK;404` template inlined into the slot's DOM). Always enumerate `app/<segment>/default.tsx` alongside `@<slot>/default.tsx`.
- **`typedRoutes: true` + dynamic href strings** — template-literal `href={`/path/${id}`}` and `router.push("/path?x=y")` with built strings fail typecheck (`Route` is a branded string union). Caveat: the augmentation lives in `.next/types/link.d.ts`, but Next 16's default `next-env.d.ts` only references `routes.d.ts`. The `next` TS plugin loads `link.d.ts` in the IDE, but **`tsc --noEmit` silently passes** unless plans add `/// <reference path="./.next/types/link.d.ts" />` to `next-env.d.ts` (or include it manually). Fix: cast site-by-site (`as Route` or `as Parameters<typeof router.push>[0]`), or route href construction through a typed helper. Plans flipping `typedRoutes: true` must pre-state both the reference-path wiring and the cast pattern.
- **`new Date()` in Server Components under Cache Components** — reading `new Date()` (or any IO clock) in a Server Component without first awaiting a dynamic source breaks `next build` under `cacheComponents: true`. Common in inherited footers (`© {new Date().getFullYear()}`). Fix: hardcode the literal year, move into a Client Component, or precede with an `await` on a dynamic source.
- **`reactCompiler: true` requires `babel-plugin-react-compiler`** — Next 16 promoted `reactCompiler` to stable but doesn't bundle the Babel plugin it runs through. Setting `reactCompiler: true` without `babel-plugin-react-compiler` in `devDependencies` fails `next build` immediately: "Failed to resolve package babel-plugin-react-compiler while attempting to resolve React Compiler." Pin `babel-plugin-react-compiler` (latest stable `1.0.0`) as a devDep.
- **pnpm blocks `sharp`'s native build → `next build` fails on clean install** — `next` pulls `sharp` (image optimizer) transitively, and pnpm 10+ (pinned `11.3.0`) won't run its native build script unless approved. `pnpm install` only warns, but Next 16's `next build` re-runs `pnpm install` (dep-status check) and surfaces the ignored build as a hard `ERR_PNPM_IGNORED_BUILDS`, failing `pnpm verify`. Surfaces only on a **cold** `node_modules` (a warm tree passes), so it ships latent. Ship `pnpm-workspace.yaml` (no `packages:` key — approval only) with `onlyBuiltDependencies: [sharp]` + `allowBuilds: { sharp: true }` in **both** `solution/` and `start/`, list it in the File tree, and don't delete the one `create-next-app` emits. pnpm 11.3.0 honors `allowBuilds` (the `package.json#pnpm` field is no longer read). Needed even without `next/image` — it's pnpm's ledger, not a runtime dep.
- **`tsconfig.json: "skipLibCheck"`** — `next build` adds `"skipLibCheck": true` every run (ships in Next 16's recommended config; prints "reconfigured your tsconfig.json … skipLibCheck was set to true"). Omitting it drifts the committed file on first build and makes standalone `tsc --noEmit` fail when nested under a monorepo whose parent `node_modules` hoists incompatible `@types/*` (e.g. `@types/mdx` → "Cannot find namespace 'JSX'"). Bake `"skipLibCheck": true` into the locked `tsconfig.json`.
- **Parallel-route slot returning a fragment flattens into the parent layout** — a `@<slot>` `page.tsx`/`default.tsx` returning `<><A /><B /></>` adds no DOM wrapper, so when the layout drops `{slot}` into a CSS `grid`/`flex` container each child becomes its own grid/flex item — a region meant to fill one slot silently splits into several cells (e.g. a two-column grid receiving 3 children: filter + list + detail). Each multi-child slot return must resolve to a single wrapping element. (Chapter 035.)

## Biome 2.4

- **`css.parser.tailwindDirectives`** — Tailwind v4 directives (`@theme`, `@apply`, `@custom-variant`, `@import "tailwindcss"`) emit parse errors without this flag. Always include `"css": { "parser": { "tailwindDirectives": true } }`.
- **`lint/performance/noImgElement`** — top-level performance rule warning on raw `<img>` (no `next` domain exists in Biome 2.4; framework filtering is via `--only=next`, not a domain block). Plans mandating raw `<img>` (theme-aware swaps, plain logos) must override: `"lint": { "rules": { "performance": { "noImgElement": "off" } } }`.
- **`files.includes` folder-glob normalization** — `useBiomeIgnoreFolder` rejects `"!folder/**"` and `biome check --write` auto-strips to `"!folder"`. Plans pinning the `/**` suffix fail `biome ci` until rewritten. Write ignores without trailing glob: `["!.next", "!node_modules"]`, not `["!.next/**", "!node_modules/**"]`.
- **`<ul role="list">` Safari VoiceOver workaround** — the standard Safari fix for `list-style: none` stripping AT list semantics trips `lint/a11y/noRedundantRoles` (only — `useSemanticElements` doesn't target this in Biome 2.4). Fix: suppress per-element with `biome-ignore`, or disable the rule globally when the project commits to the workaround. Architects mandating the role must pre-declare the chosen reconciliation.
- **`lint/suspicious/noArrayIndexKey` on placeholder lists** — recommended-on, errors on `Array.from({ length: 8 }).map((_, i) => <li key={i}>…</li>)` even when static and non-reorderable (biomejs/biome#9469). A `biome-ignore` above `.map(...)` does not suppress the diagnostic on the nested `<li key={i}>` line. Fix: declare a stable string-key tuple (`const KEYS = ['row-0', …] as const`) and map over that, or replace with a fixed number of explicit elements. Skeleton/placeholder components should ship the tuple pattern up front.
- **`pnpm <script> --write` doubles the flag** — projects pin `"check": "biome check --write ."`, so `pnpm check --write` runs `biome check --write . --write`, rejected with "argument `--write` cannot be used multiple times in this context." Write `pnpm check` (the script already carries `--write`); never append the flag again.

## Shadcn UI

- **Primitive formatting** — `pnpm dlx shadcn add` writes primitives with double quotes and `import * as React` (not `import type`), triggering Biome warnings on first lint. Accept on first commit; let `pnpm check --write` normalize.
- **Radix umbrella package** — shadcn's `new-york-v4` style depends on the `radix-ui` umbrella package, not per-primitive `@radix-ui/react-*` (older `default`/`new-york` styles still use per-primitive). For v4-style installs, pin `radix-ui@^1.4.3` instead of enumerating per-primitive deps.
- **`Card` has no `asChild` slot** — shadcn's `Card` renders a `<div>` with no `asChild` prop, so it can't be retargeted to `<article>` etc. Plans needing a semantic root (e.g. `FeatureCard` typed as `ComponentProps<'article'>`) must inline `Card`'s container classes onto the chosen element and nest `CardHeader`/`CardTitle`/`CardDescription` inside, rather than wrapping `Card`. Compose by hand, don't double-wrap.
- **Radix primitives ship `'use client'`** — `dialog`, `sheet`, `separator`, and other Radix-wrapping primitives carry the directive. Any verification counting `'use client'` files must include the installed `components/ui/*` set, not just authored islands.

## eslint-plugin-react-hooks 7 (Compiler diagnostics layer)

- **`flat.recommended` has no `files` glob or parser** — `reactHooks.configs.flat.recommended` is a bare rules/plugins object with no `files` and no `languageOptions.parser`. Spread alone into a flat config (`export default [reactHooks.configs.flat.recommended, …]`) on a TS-only project, ESLint 9 matches nothing (default flat-config extensions exclude `.ts`/`.tsx`), so `eslint .` exits 0 having linted zero files — the layer is silently inert. Plans wiring this layer must add `files: ['**/*.{ts,tsx}']` and `languageOptions: { parser: tseslint.parser }` (ship `typescript-eslint@^8.60` as a devDep) inside the spread, or the diagnostics never run. Verify non-vacuously: a `--print-config`/`-c` run on a `.tsx` source must report it as checked, not "File ignored because no matching configuration was supplied."
- **`react-hooks/set-state-in-effect` false-positives on the next-themes mount gate** — the rule (error in `flat.recommended`) flags the canonical `useEffect(() => setMounted(true), [])` hydration gate (facebook/react#34743, pacocoursey/next-themes#374). Plans locking that mount-gate AND wiring the hooks layer must turn it off (`'react-hooks/set-state-in-effect': 'off'`); the remaining compiler diagnostics (`set-state-in-render`, `rules-of-hooks`, `purity`, memoization) stay on.

## lucide-react 1.x

- **All brand icons removed** — lucide-react 1.0 dropped every brand glyph (`Github`, `Twitter`, `Linkedin`, `Youtube`, `Facebook`, `Instagram`, `Slack`, `Gitlab`, …); importing them throws "export X doesn't exist in target module" and fails `next build` (trademark/maintenance scope; Simple Icons is the recommended brand-logo replacement). Plans needing "social" buttons must use non-brand glyphs that still ship (`Mail`, `MessageCircle`, `Globe`, `Rss`, `Send`, `AtSign`, `Share2`) or pull in a separate brand-icon package. Do not reference brand-name lucide icons.

## Postgres 18 (Docker)

- **Volume mount path** — Postgres 18 stores data under `/var/lib/postgresql/18/docker` (new major-versioned layout). Mount the parent `/var/lib/postgresql` so `pg_upgrade --link` works across major-version bumps; the historic `/var/lib/postgresql/data` is still writable but breaks the upgrade flow. Use:
  ```yaml
  volumes:
    - pgdata:/var/lib/postgresql
  ```
- **`@neondatabase/serverless` cannot reach vanilla Postgres** — the Neon driver speaks HTTP/WebSocket only, not the Postgres wire protocol; against local Docker it errors before connect. For local Docker chapters use the course-pinned `postgres` (postgres-js) driver via `drizzle-orm/postgres-js` — it speaks the wire protocol, bundles cleanly under Turbopack, and needs no `serverExternalPackages` entry.

## Drizzle 0.45

Pinned to `drizzle-orm@0.45.x` + `drizzle-kit@0.31.x` until Better Auth ships `drizzle-orm@^1.0` support ([better-auth#6766](https://github.com/better-auth/better-auth/issues/6766), target 1.8+). Constraints on this line:

- **Runtime `casing` lives on the client.** `drizzle({ client, casing: 'snake_case', schema })` snake-cases all columns from one declaration. Per-column overrides via explicit names (`uuid('organization_id')`) are the escape hatch. (Removed in 1.0 — flips to `pgTableCreator((name) => toSnakeCase(name), 'snake_case')`; the second arg is mandatory or columns stay camelCase.)
- **Migration file layout is flat.** `drizzle-kit generate` emits `drizzle/<timestamp>_<name>.sql` plus top-level `drizzle/meta/_journal.json` and `drizzle/meta/<id>_snapshot.json`. Pass `--name <verb>_<noun>` so the suffix is intentional. Renaming the timestamp prefix breaks journal lookup. (1.0 moves to per-migration dirs `drizzle/<timestamp>_<name>/{migration.sql, snapshot.json}` with no top-level `meta/`.)
- **Relations v1 is the API.** `relations(<table>, ({ many, one }) => ({ ... }))` per-table; `db.query.<table>.findMany({ with: { ... } })` is the N+1-safe traversal. (1.0 replaces this with a single `defineRelations(schema, (r) => ({ ... }))`; the `r.one` form defaults to `optional: true` and must be flipped to `false` on NOT NULL FKs.)
- **Zod helpers ship as a separate package.** `drizzle-zod@^0.8` peer-requires `drizzle-orm@>=0.36` and `zod@^4`. `createSelectSchema`/`createInsertSchema`/`createUpdateSchema` import from `drizzle-zod`, not `drizzle-orm/zod`. (1.0 drops the separate package, moving them under `drizzle-orm/zod`.)
- **`drizzle-seed` `with` keys are table names, not relation aliases** — refining a parent with children uses the schema's table binding (`with: { invoiceLines: [...] }`), not the relation alias (`lines:`). Junction tables hand-populated post-seed need `{ count: 0 }` in the refine to suppress the default fill. General across both versions.
