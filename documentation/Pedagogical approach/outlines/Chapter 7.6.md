# Chapter 7.6 — Project: CRUD via Server Actions: pedagogical approach

## Concept 1 — The cuts panel: what 7.6 ships and what later units own

**Why it's hard.** The student opens the project brief and the reflex is *more is better* — auth, RBAC, soft delete, audit log, idempotency, rate limits, all reachable from where the chapter ends. The senior move is the inverse: ship the canonical Server Action shape and *nothing else*, leaving every later layer to its owning unit. Without the cut named explicitly, the student either pads the scope (loses the deadline, blurs the discipline) or shortcuts the disciplines the cut is designed to install (skips the no-JS test, invents a session reader to replace the auth stub, wraps `revalidatePath` in a `try`). The cut *is* the lesson, and the unit graph the cuts point at is what the chapter cashes in for in Units 8–15.

**Ideal teaching artifact.** Pattern archetype delivered as a **cuts-and-carry-forwards table** in the brief lesson, echoing 6.6 Concept 1's shape so the student recognizes the project framing across the curriculum. Two columns: *In scope* (three actions, three forms, the optimistic create, the transactional delete, the JS-disabled verify) and *Cut, with the unit that owns it* (real auth → 9.5, RBAC + `authedAction` wrapper → 10.4, soft delete + optimistic concurrency → 11.3, transactional email after the commit → 8.3, idempotency-key dedup → 12.1, audit log inside the tx → 10/17, cache tags replacing `revalidatePath` → 15.2). A third row at the bottom: *forward-loaded disciplines* (the `getActiveContext()` call site that 10.2 will wrap with `authedAction`; the `db.transaction` block that holds the audit-log and notification slots open; the client-generated UUIDv7 hidden input that 12.1 repurposes for replay safety). The senior posture lands inline: the cut items are not missing — they are owned by their owning units, and this chapter's job is the canonical shape they all extend.

**Engagement.** A `Buckets` sort: twelve feature descriptions ("a `safeParse` at the top of every action", "an `authedAction('member', schema, fn)` wrapper around every action", "an optimistic create with `useOptimistic`", "a soft-delete that moves rows to a `deletedAt` column", "a `revalidatePath('/invoices')` after every mutation", "an `Idempotency-Key` hidden field that dedups double-clicks", "a Resend email send after the create commits", "a `<form action={createInvoice}>` that works without JS", "a `version` column for optimistic concurrency on edits", "a `?deleted=INV-0042` URL-param success toast", "an `org_audit_log` row written inside the delete tx", "a Sentry capture on the `internal` Result branch") into "in scope for 7.6" or "owned by a later chapter (name it)." Wrong-answer feedback names which discipline the cut protects.

**Components.**
- `Figure` wrapping a hand-coded HTML table (in-scope / cut-and-owned-by / forward-loaded) for the cuts-and-carries panel. Single-use static composition; no bespoke component earns its weight, and the shape is the same one 6.6 Concept 1 uses.
- `Buckets` for the in-scope-vs-owned-elsewhere sort.
- `Aside` (`tip`) below the panel: "the cut items are not missing — they are owned by their owning units; this chapter's job is the canonical Server Action shape they all extend."

**Project link.** This concept *is* lesson 7.6.1's framing — the student returns to the panel at the end of every build lesson to confirm they're still on the inside of the cut.

## Concept 2 — Reading the starter as a contract: the page is the spec, the form earns its right to exist by satisfying it

**Why it's hard.** The student opens `app/invoices/new/page.tsx`, sees a Server Component shell that imports `<NewInvoiceForm />` from a sibling file, and reaches for the keyboard to write the form however feels right. The provided page is *not* a sketch — it's the contract: it imports the form by exact name and exact path, it passes specific props (the loaded invoice on the edit page; the customer list on the new page), and it expects the form to render in the slot the page already laid out. The senior reflex is to read the call site *first* and treat the form's signature, props, file path, and exported name as the spec the implementation satisfies. The junior reflex is the opposite — write the form, then "wire it up" — and the wiring breaks because the page already wrote it and was waiting for the form to arrive. The chapter's three forms all live behind the same contract; reading them as contracts is the load-bearing posture for the rest of the chapter.

**Ideal teaching artifact.** Pattern archetype delivered as a **call-site trace** through the three provided page files, walked once in `AnnotatedCode`. The student walks `/invoices/new/page.tsx` (passes `customers` prop, expects `<NewInvoiceForm customers={customers} />`), then `/invoices/[invoiceId]/page.tsx` (loads the invoice via the provided `getInvoiceDetail`, passes `invoice` prop, renders both `<EditInvoiceForm invoice={invoice} />` and `<DeleteInvoiceForm invoice={invoice} />`), then `app/invoices/page.tsx` (wraps the rendered list rows with `<OptimisticInvoicesList initialInvoices={...}>`). For each call site, three annotation tracks: (a) the file path and exported name the page imports — the student-written file must match exactly; (b) the props the page passes — the form's prop type is fixed by the page; (c) what would break in the page if the student renamed the export, moved the file, or changed the prop shape. The closing line names the senior reflex: when the page is the spec, the implementation satisfies it; the temptation to "improve" the call site is the smell. A second short beat lingers on `lib/result.ts`, `lib/auth-stub.ts`, and the shadcn `<Form>` primitives in `components/ui/form.tsx` — provided code the student reads but doesn't touch, with the boundary line drawn explicitly so the student stops trying to rebuild what the starter already ships.

**Engagement.** A `Dropdowns` exercise on the three form components' file headers — given the page's import line, the student fills in the file path, the exported name, and the prop type for each of the three forms. Wrong picks surface the wiring break each one would produce ("the page imports `NewInvoiceForm`, your file exports `CreateForm` — the page renders `undefined`"). Confirm with a `MultipleChoice`: "a teammate's PR renames `<DeleteInvoiceForm>` to `<DeleteDialog>` and updates the export. What does the senior reviewer ask for?" — right answer names *the page is the contract; the form's name is fixed by the import in `/invoices/[invoiceId]/page.tsx` — revert and keep the contract*.

**Components.**
- `AnnotatedCode` walking the three call sites with the file-path / props / break-mode annotation tracks.
- `Dropdowns` for the three-header signature exercise.
- `MultipleChoice` for the contract-rename PR-review.
- `Aside` (`note`) below: "the page is the contract; the forms earn their right to exist by satisfying it. The starter draws the line — provided code stays untouched, stub files match the page's import."

**Project link.** This concept *is* lesson 7.6.2's senior posture — the student leaves knowing exactly what they have to write for each of the three forms, before they write a line, with the page-side imports as the four-corner constraint.

## Concept 3 — Mutation schemas: `createInsertSchema` + override + `.omit`, with `id` kept in for 7.6.5

**Why it's hard.** Three structural facts collapse into one beat the student stitches wrong on first read. (1) The schema is *derived*, not hand-written — `createInsertSchema(invoices, { number: s => s.min(1).max(50), total: s => s.refine(n => n >= 0) })` produces the base; the override callback receives the *generated* column schema and the student chains `.refine` / `.min` on top, never rebuilds from `z.string()` (the 7.1 Concept 14 trap, lifted to the project's keyboard). (2) `.omit({ organizationId: true, createdBy: true, createdAt: true })` strips columns the action sets server-side from the auth context — but the `id` column stays in the schema, optional and defaulted, because 7.6.5's optimistic-create flow will pass a client-generated UUIDv7 as a hidden field and the schema must accept it without a re-shape mid-chapter. The "decided in 7.6.3 for 7.6.5's benefit" is the load-bearing *foresight* the senior reaches for. (3) `FormData` arrives as strings; the per-field overrides layer the coercion (`total: z.coerce.number().nonnegative().multipleOf(0.01)`, `issuedAt: z.coerce.date()`, `customerId: z.uuid()`) so the action's `safeParse(Object.fromEntries(formData))` produces the typed shape the database insert wants. Without the three landing together, the schema either drifts from the table, breaks 7.6.5's optimistic flow, or fails the `FormData` parse on the first submit.

**Ideal teaching artifact.** Pattern archetype delivered as a **decision-walked authoring beat** in `AnnotatedCode` with the source-of-truth chain echoed visually beside it. The student watches `createInvoiceInputSchema` get authored line by line in eight annotations: (1) `createInsertSchema(invoices, ...)` — the base shape derives from the Drizzle table; one column change in `db/schema.ts` propagates here; (2) the override map's callback parameter is the generated column schema — chain on top, don't rebuild; (3) `total: s => s.refine(n => n >= 0, 'Total must be non-negative')` — local rule lives on the schema, the message authors here, the form renders later (the 7.1 Concept 11 contract); (4) the `FormData` coercion overrides for `total`, `issuedAt`, `dueAt`, `customerId` — the bridge from strings to the typed domain (the 7.1 Concept 12 wire walk, applied); (5) `.omit({ organizationId: true, createdBy: true, createdAt: true })` — strip columns the action fills from `getActiveContext()`, never from the form; (6) `id` is *not* omitted — it stays optional, the column has a `$defaultFn` UUIDv7, the create form will supply a client-generated UUIDv7 in 7.6.5; (7) `updateInvoiceInputSchema = createInvoiceInputSchema.extend({ id: z.uuid() })` — the update form posts the `id` as a hidden input, and `.extend` is the 7.1 Concept 8 derivation reflex (don't re-author the field list); (8) `deleteInvoiceInputSchema = z.object({ id: z.uuid() })` — the smallest schema that earns its weight, two lines, no `.omit`. Beside the walk, a tiny one-box-three-arrows hand SVG echoing 7.1 Concept 14's source-of-truth chain — the box is `db/schema.ts`, three arrows fan out to the three input schemas, every consumer downstream typed off the same root.

**Engagement.** A `MultipleChoice` round of three traps: (a) "a teammate's PR omits `id` from `createInvoiceInputSchema` — what breaks in 7.6.5?" → the optimistic-create flow's hidden UUIDv7 field has nowhere to land; the schema would need re-shaping mid-chapter; (b) "a teammate writes the `total` override as `total: z.number().refine(...)` instead of `total: s => s.refine(...)` — what's wrong?" → the override callback's parameter is the generated column schema; rebuilding from `z.number()` loses the table-derived defaults and the column-aware nullability; (c) "a teammate adds the `organizationId` to the form as a hidden input instead of `.omit`-ing it — what's the smell?" → the form is the wrong source of truth for tenant identity; the action reads it from `getActiveContext()`, the form has no business carrying it. Wrong-answer feedback names the discipline each trap violates.

**Components.**
- `AnnotatedCode` walking the eight-annotation `createInvoiceInputSchema` authoring beat.
- `Figure` wrapping a small hand SVG of the source-of-truth chain (one schema box, three arrows to the three input schemas) — confirms the forward-link from 7.1 Concept 14.
- `MultipleChoice` reused three times for the trap round.
- `Aside` (`tip`) below the authoring beat: "the override callback receives the generated column schema; chain on top. `id` stays optional in the create schema for 7.6.5's UUIDv7 hidden input — decided here, used there."

**Project link.** This concept *is* the schema-authoring posture for lesson 7.6.3's first half — the student writes the three schemas from this walk, and 7.6.5's optimistic flow rests on the `id` decision made here.

## Concept 4 — The five-seam action body, shipped three times

**Why it's hard.** The student arrives with the five-seam stencil installed (7.2 Concept 4 — parse, authorize, mutate, revalidate, return) and now has to ship it *three times* against three different operations whose substitutions diverge: create returns `ok({ id })` and `redirect`s; update returns `ok({ id })` and stays on the page; delete returns `ok(null)` and `redirect`s after revalidating. The order is non-negotiable, but the per-seam content varies, and the student who treats each action as a fresh structure ends up with three actions in three different shapes — the reviewer can't scan the file at speed, the failure modes (logging unparsed input, hitting the DB before authorizing, revalidating before the write) re-emerge per action. The reflex install is the *pattern repetition* — three actions, identical seams, varying contents — so the shape becomes the thing the student reaches for at the next action's blank cursor.

**Ideal teaching artifact.** Pattern archetype delivered as a **stencil-and-substitute walkthrough** in `AnnotatedCode`, lifting the 7.2 Concept 4 five-seam stencil and applying it three times in one block with the substitutions called out per seam. The student sees the create body laid out in five labeled regions (`// 1. parse`, `// 2. authorize`, `// 3. mutate`, `// 4. revalidate`, `// 5. return`) with the canonical lines: `safeParse(Object.fromEntries(formData))` → `getActiveContext()` → `db.insert(invoices).values({ ...parsed.data, organizationId, createdBy: userId }).returning({ id })` → `revalidatePath('/invoices')` → `return ok({ id })` followed by `redirect('/invoices/' + id)`. Below it, two parallel stencils for `updateInvoice` and `deleteInvoice` showing the *same five seams* with the substitutions: update's mutate becomes `db.update(invoices).set(parsed.data).where(and(eq(invoices.id, parsed.data.id), eq(invoices.organizationId, organizationId)))`, return becomes `ok({ id })` *without* the redirect (the edit form stays on the page, 7.6.4's reason); delete's mutate is the simple `db.delete(...)` form (7.6.6 wraps in tx), return is `ok(null)` followed by `redirect('/invoices')`. Across all three actions, the parse line is identical, the authorize line is identical, the revalidate line is identical — the *shape* is the thing the student feels three times. A small Aside names the senior anti-patterns the order prevents: parsing after the auth read wastes a session lookup on bad input; revalidating before the write ships stale data on next navigation; logging unparsed input leaks PII to production logs (the 7.2 watch-outs made operational).

**Engagement.** A `Sequence` drag-to-order drill: the student is shown the three actions' six lines each shuffled and orders them into the five seams plus the redirect. Wrong placements produce targeted feedback ("`getActiveContext()` before `safeParse` runs the auth read on bad input — wasted session lookup once 10.2's wrapper lands"; "`revalidatePath` before the database write revalidates against unchanged state — the user navigates back and sees the old list"; "the `redirect` inside the try/catch swallows the framework's control-flow exception — the redirect never fires"). The redirect-inside-the-catch trap is the chapter-specific one to surface here.

**Components.**
- `AnnotatedCode` walking the create body with five-seam annotations, with the update and delete stencils rendered alongside as parallel substitutions. Reuses the five-seam stencil shape from 7.2 Concept 4 — confirms the forward-link.
- `Sequence` for the drag-to-order drill, with the three traps as wrong-placement feedback.
- `Aside` (`note`) below: "five seams, this order, every action. The substitutions vary per operation; the shape is the thing."

**Project link.** This concept *is* the action-authoring posture for lesson 7.6.3's second half — the student writes all three actions from this stencil, and the project's reviewer checklist names the order explicitly.

## Concept 5 — Tenant guard in the `where`, redirect outside the catch

**Why it's hard.** Two structural traps converge on the action body and the chapter is where they get installed at the keyboard. (1) **The IDOR-class tenant guard.** The naive update reads the row first, checks `row.organizationId === currentOrgId`, then writes — the row was already loaded, the auth check fires *after* the data left the database, the leak vector exists even when the write doesn't fire. The structural fix from 6.6 Concept 6 is the one-line shape: `where: and(eq(invoices.id, parsed.data.id), eq(invoices.organizationId, organizationId))` — the database returns no row when the IDs don't match, nothing was loaded, nothing leaked. The chapter's update and delete actions both ride this rule. (2) **The redirect-inside-the-catch trap.** `redirect` from `next/navigation` works by throwing a control-flow exception the framework catches at the route boundary — so `try { await db.insert(...); return ok({ id: row.id }); redirect(...) } catch (e) { if (isUniqueViolation(e)) return err('conflict', ...); throw e; }` looks innocuous but the redirect, placed inside the try, gets caught by the unique-violation branch and silently swallowed. The senior pattern is to put the redirect *after* the try/catch on the success branch, never inside it. Both traps are surface-level and failure-fatal; they have to land as named *reflexes*, not as paragraphs the student reads past.

**Ideal teaching artifact.** Misconception-first ambush delivered as a **two-trap predict-and-fix round** in `CodeVariants`. **Trap one (the IDOR shape)**: tab one shows `updateInvoice` written load-then-check — `const existing = await db.query.invoices.findFirst({ where: eq(invoices.id, parsed.data.id) }); if (!existing || existing.organizationId !== organizationId) return err('not_found', ...); await db.update(invoices).set(parsed.data).where(eq(invoices.id, parsed.data.id))` — annotated as "the row was read; in a logging or audit context, the read is recorded against org A's session even when the row belongs to org B (the 6.6.5 leak vector). Tab two shows the structural fix — `db.update(invoices).set(parsed.data).where(and(eq(invoices.id, parsed.data.id), eq(invoices.organizationId, organizationId)))` — the database returns zero affected rows when the orgs don't match, nothing was loaded, nothing leaked. The closing rule lifts 6.6 Concept 6 directly: the tenant guard is a query-shape, not a post-condition. **Trap two (the swallowed redirect)**: tab one shows `try { await db.insert(...); revalidatePath('/invoices'); return ok({ id }); redirect('/invoices/' + id) } catch (e) { if (isUniqueViolation(e)) return err('conflict', ...); throw e; }` — annotated as "the redirect throws, the catch fires, the redirect's exception matches the catch but isn't a unique violation, the rethrow surfaces a generic 500. The success path *never reaches the navigation*." Tab two moves the redirect outside: `try { ... } catch (e) { ... }` followed by `redirect('/invoices/' + id)` on the success branch. The closing rule: framework redirects are control-flow throws; the catch must not see them.

**Engagement.** Each fix is a small repair the student applies in the action file. Confirm with a `MultipleChoice` round of two: (a) "a teammate's PR loads the invoice first, checks the org ID, then deletes — the test passes, the action is approved. What's the failure mode?" → the row was read; even if the delete doesn't fire, the read is logged against the caller; structural fix is the org check in the `where`; (b) "a teammate's create action puts `redirect('/invoices/' + id)` inside the try block, after the `return ok({ id })`. The redirect never fires on success. What's the bug?" → the `return` already exited the function before the redirect line; even if the order were swapped, the redirect's control-flow throw would be caught by the unique-violation catch — move the redirect after the try/catch, on the success branch.

**Components.**
- `CodeVariants` reused twice — once for the IDOR contest, once for the swallowed-redirect contest, each tab carrying the failure-mode annotation.
- `MultipleChoice` for the two-fix recall.
- `Aside` (`caution`) below the two contests: "tenant guard in the `where`, never after the load. Framework redirects are control-flow throws; place the redirect after the try/catch on the success branch."

**Project link.** This concept lands in lesson 7.6.3's update and delete actions — the student writes the `and(eq(id), eq(organizationId))` `where` clause from the reflex installed here, and the create/delete redirects sit on the success branch outside the unique-violation catch.

## Concept 6 — The `Result` → `<FieldError>` contract: form is a renderer

**Why it's hard.** The student finished 7.2 with the canonical `Result<T>` shape and 7.3 with the `<FieldError>` reader pattern, but the project is where the contract lands end-to-end against three actions and three forms. The trap surfaces as drift: one form reads `state?.fieldErrors?.email`, the next reads `state?.error?.fieldErrors?.email`, the third writes its own copy because "the schema's message wasn't quite right for this UI." Each drift breaks the discipline that makes the contract worth installing — the schema authors the message, `z.treeifyError` produces the wire shape, the form is a mechanical reader. The student has to *feel* the contract three times: the action returns the shape, the `<FieldError>` reads it, the input wires `aria-describedby` and `aria-invalid` to it. Once it lands at the third form, the reflex is automatic and any future form in the codebase joins the contract for free.

**Ideal teaching artifact.** Pattern archetype delivered as a **three-panel contract walk** in `AnnotatedCode`, recalling the 7.3 Concept 6 spine and instantiating it against the project's create form. The student sees three coordinated panels in one block: (a) the action's failure branch — `if (!parsed.success) return err('validation', 'Check the highlighted fields.', z.treeifyError(parsed.error).properties)`; (b) the `<FieldError>` component — `function FieldError({ name, fieldErrors }: { name: string; fieldErrors: Record<string, string[]> | undefined }) { const message = fieldErrors?.[name]?.[0]; return message ? <p id={`${name}-error`} className="text-destructive text-sm mt-1">{message}</p> : null }`; (c) the form's wiring — `const fieldErrors = state?.ok === false ? state.error.fieldErrors : undefined;` followed by the input `<Input name="total" required aria-describedby="total-error" aria-invalid={!!fieldErrors?.total?.[0]} />` paired with `<FieldError name="total" fieldErrors={fieldErrors} />`. Three annotation tracks walk through: (1) the schema-authored message flows through `treeifyError` into `Result.error.fieldErrors` (the 7.1 Concept 11 contract, instantiated); (2) the `<FieldError>` is a mechanical reader — no copy-writing, no fallback strings, no `if (name === 'email') return 'Email is invalid'`; (3) the input's ARIA wiring is non-optional — the visual error is for sighted users, the `aria-describedby` / `aria-invalid` is for assistive tech (the 4.11.5 dividend made operational). The closing line names the rule: schema authors the message, the action ships the wire shape, the form renders mechanically — three layers, one contract, no per-form copy.

**Engagement.** A `MultipleChoice` PR-review round of three: (a) "a teammate's `<FieldError>` writes `fieldErrors?.[name]?.[0] ?? 'This field is required'` as the fallback. What's the smell?" → the schema authors the message; the fallback drifts the moment a new validation rule lands; drop the fallback, the schema covers it; (b) "a teammate's edit form reads `state?.fieldErrors?.total` directly. What's the bug?" → the `Result` shape is `state.ok === false ? state.error.fieldErrors : undefined`; the discriminator check unlocks the typed access; without it, the success branch's `state.data` would crash the read; (c) "a teammate omits `aria-describedby` and `aria-invalid` from the input because the visual error is rendered. What does the senior reviewer ask for?" → the ARIA wiring is the assistive-tech surface; the visual is one channel, the ARIA is the other — both are non-optional.

**Components.**
- `AnnotatedCode` walking the three-panel action → `<FieldError>` → form-wiring contract. Recalls 7.3 Concept 6's spine — confirms the forward-link.
- `MultipleChoice` reused three times for the PR-review round.
- `Aside` (`note`) below: "schema authors the message, action ships the wire shape, form renders mechanically. Three layers, one contract, no per-form copy."

**Project link.** This concept *is* the form-wiring posture for lesson 7.6.4 — the `<FieldError>` and `<SubmitButton>` primitives ship as project-level reusables, every form in the project consumes them, and the reviewer's "is this a one-off render or a reusable primitive?" question maps onto the contract.

## Concept 7 — Uncontrolled inputs, Constraint API mirrored to the schema

**Why it's hard.** Two structural facts collapse here at the project's keyboard. (1) Every input is uncontrolled — `name="total"`, `defaultValue={invoice?.total ?? ''}`, no `useState` per field — and the edit form's `defaultValue` flows from the page's loaded invoice prop, not from any client state. The student who reaches for controlled inputs to "make the edit form easier" recreates the 7.3 Concept 1 reflex this chapter has to override; `revalidatePath` re-renders the Server Component, the new `defaultValue` flows down, the DOM reconciles by uncontrolled-input identity. (2) Every input's HTML5 attributes mirror the schema's rules — `required` where the schema says `.min(1)`, `type="number" step="0.01" min="0"` where the schema says `z.coerce.number().nonnegative().multipleOf(0.01)`, `inputMode="decimal"` where the schema says decimal numbers, `type="date"` where the schema says `z.coerce.date()`. The mirror is the discipline that makes the cheap layer (the constraint API, browser-native, free, no JS) catch the same shape rules the server's schema enforces, with the server as the boundary of correctness for stale-bundle and curl-and-no-JS callers (the 7.2 Concept 5 ambush, lifted to the project). Drift between the schema and the attributes is the bug class the mirror prevents.

**Ideal teaching artifact.** Pattern archetype delivered as a **schema-to-attribute mirror table** rendered as a `Figure`-wrapped HTML table, paired with one canonical input authoring beat in `AnnotatedCode`. The mirror table has six rows, one per project field: `customerId` (schema: `z.uuid()`, input: `<Select name="customerId" required>` with the customer list as options); `number` (schema: `s => s.min(1).max(50)`, input: `<Input name="number" required minLength={1} maxLength={50} autoComplete="off">`); `status` (schema: `z.enum([...])`, input: `<Select name="status" required>`); `total` (schema: `z.coerce.number().nonnegative().multipleOf(0.01)`, input: `<Input name="total" type="number" step="0.01" min="0" required inputMode="decimal">`); `currency` (schema: `z.string().default('USD')`, input: `<Input name="currency" defaultValue="USD" required>`); `issuedAt` / `dueAt` (schema: `z.coerce.date()`, input: `<Input name="issuedAt" type="date" required>`). Three columns per row: the schema rule, the matching input attributes, the failure mode if the mirror breaks (a stale client running the old bundle accepts a malformed payload the server rejects — round-trip cost, bad UX). The paired `AnnotatedCode` walks one full input — the `total` field — being authored with five annotations: `name="total"` matches the schema key; `type="number"` triggers the browser's numeric keypad on mobile; `step="0.01"` mirrors `multipleOf(0.01)`; `required` mirrors the non-optional schema field; `inputMode="decimal"` is the touch-keyboard hint with no schema mirror (UX-only); `defaultValue=""` keeps the input uncontrolled; `aria-describedby` and `aria-invalid` carry the FieldError wiring from Concept 6.

**Engagement.** A `Tokens` round on a forty-line invoice form authored *without* the constraint attributes — the student clicks every input where a `required`, `type`, `min`, `step`, `inputMode`, or `autoComplete` attribute is missing, and the wrong-answer feedback names what the schema's rule is and what the attribute's mirror should have been. Confirm with a `MultipleChoice`: "a teammate's PR adds `total: z.coerce.number().min(0).max(1000000)` to the schema but doesn't change the input. What does the senior reviewer ask for?" → mirror the bound on the input — `max="1000000"` — so the cheap client check matches the server's contract; the schema is the source, the attributes mirror it.

**Components.**
- `Figure` wrapping a hand-coded HTML six-row mirror table (schema rule / input attributes / failure-if-broken).
- `AnnotatedCode` walking the canonical `total` input with the five-annotation authoring beat.
- `Tokens` for the missing-attribute spotting round.
- `MultipleChoice` for the schema-changed-but-input-didn't PR-review.
- `Aside` (`tip`) below the mirror table: "every shape rule lives on both — the constraint API on the client (UX), the schema on the server (correctness). Drift between them is the bug the mirror prevents."

**Project link.** This concept lands across all three forms in lesson 7.6.4 — the `name`, `defaultValue`, and constraint attributes follow the mirror; the edit form's `defaultValue`s flow from the loaded invoice; the `:user-invalid` Tailwind variant (from 7.3 Concept 11) styles the post-interaction red ring.

## Concept 8 — Shadcn `<Form>` primitives without RHF, and the `<Dialog>` PE fallback

**Why it's hard.** Two project-specific decisions land in this lesson and both go against student instinct. (1) **Shadcn's `<Form>` family (`<FormField>`, `<FormItem>`, `<FormLabel>`, `<FormControl>`, `<FormMessage>`) ships in the docs paired with React Hook Form** — the docs imply they require RHF — but the project deliberately uses them as *pure layout primitives* under `useActionState` to keep the PE story intact (7.3 Concept 6's discipline). The chapter's discipline is to use the layout primitives without the RHF instance, with the per-field error rendered through `<FieldError>` (Concept 6) instead of `<FormMessage />`. ## Concept 9 — Optimistic create with a client-generated UUIDv7, and the observable rollback

**Why it's hard.** Three structural facts from 7.3 Concept 9 land in the project simultaneously and the student has to *feel* each one in the keyboard. (1) **The hook is transition-bound** — `addOptimistic(value)` outside a `startTransition` (or outside the action call's implicit transition) makes the optimistic state stick; the rollback never fires. (2) **The rollback is implicit** — React owns it; the form layer doesn't write rollback logic, doesn't track temp/real ID mappings, doesn't manually remove failed updates. (3) **The reconciliation key is a client-generated UUIDv7** — `useState(() => uuidv7())` at form-mount, rendered as a hidden `<input type="hidden" name="id" value={tempId}>`, the schema accepts the optional `id` from Concept 3's foresight, the action passes it through to the database, the optimistic and persisted rows reconcile by key with no flicker. The fourth fact the project adds: a `?fail=1` debug branch that returns `err('internal', ...)` after a 500ms delay, so the student can *observe* the rollback at the project's keyboard — submit, watch the row appear in the list with a spinner, watch it disappear when the action returns failure, watch the form's banner show the userMessage with the typed values still intact. The observation is the load-bearing moment; without it, the rollback is a paragraph the student reads past.

**Ideal teaching artifact.** Mechanics archetype delivered in two beats — the wiring is one model, the observable rollback is another. **Beat one — the wiring walk** in `AnnotatedCode` over the project's `OptimisticInvoicesList` plus the refactored `NewInvoiceForm`. The student sees five annotations: (1) `useOptimistic(initialInvoices, (current, next) => [next, ...current])` in the list component — the reducer is pure, the optimistic state is reactive, the hook returns `[optimisticInvoices, addOptimistic]`; (2) the `addOptimisticInvoice` function exposed via a small React context so the create form can call it without prop-drilling — the project's pattern, named once; (3) at form-mount in `NewInvoiceForm`, `const [tempId] = useState(() => uuidv7())` — the UUIDv7 is generated *once*, stable across re-renders, rendered as `<input type="hidden" name="id" value={tempId}>`; (4) the submit handler wraps the action call in a transition: `const handleSubmit = (formData: FormData) => { startTransition(() => { addOptimisticInvoice({ id: tempId, ...rawValuesFromFormData, status: 'draft', pending: true }); formAction(formData); }) }` — the optimistic add fires *before* the action, both inside the transition; (5) the action's success path: the database row writes with the client-supplied `id` (the schema accepted the optional UUIDv7, Concept 3's decision), the redirect fires; on return to `/invoices`, the server-rendered list now includes the new invoice with the same UUID, React reconciles by `key={invoice.id}` and the optimistic row swaps to the real one without a flicker. **Beat two — the controllable-failure observation** in `ReactCoding` (or, if the project's runtime can't host the slow-action simulator, a `DiagramSequence` walking the four observable states). The observation is the assessment because the student must *watch the rollback fire* to internalize that React owns it.

**Engagement.** Beat two's observation carries the assessment — submit, watch, observe. Confirm with a `MultipleChoice` after both beats: (a) "a teammate's optimistic update sticks on success — the row duplicates with the real one. What's the bug?" → the `addOptimistic` call wasn't inside the action's transition; React's implicit rollback only fires for transition-bound updates; (b) "a teammate uses `temp-${Date.now()}` as the optimistic row's ID instead of a UUIDv7. What flickers?" → the optimistic row's key differs from the persisted row's key, so React unmounts the temp item and mounts the real one on revalidation — a brief visual flicker; the UUIDv7 hidden field reconciles cleanly because the keys match.

## Concept 11 — The delete in a transaction: the shape that holds for tomorrow

**Why it's hard.** The student's first reaction to "wrap the delete in a `db.transaction`" is reasonable: "the FK `ON DELETE CASCADE` on `invoice_lines → invoices` already deletes the children atomically; the tx is overhead." The senior framing inverts the impulse — the transaction earns its weight not for *today's* atomicity, which the FK provides, but for the *shape* it installs for tomorrow's extensions: Unit 10 will add an audit-log row inside the same tx, Unit 11 will add a soft-delete branch and a `deletedAt` write, Unit 8 will add an email send *after* the tx commits, Unit 14 will fire a notification. The reviewer who reads the tx-wrapped delete sees the multi-step shape and knows where each future addition lands; the reviewer who reads the bare `db.delete(...)` sees a single-step operation that has to be re-architected when the next requirement arrives. The trade is named explicitly — the FK alone would work today; the wrapper holds for tomorrow — and the no-external-calls-inside-the-tx rule (7.2 Concept 10's landmine, made operational) gets installed at the right moment so Unit 8's email send and Unit 13's file cleanup land *outside* the tx block when those layers arrive.

**Ideal teaching artifact.** This concept needs two paired beats — the tx shape is one model, the no-external-calls rule is another, and conflating them loses the second beat. **Beat one — the before-and-after refactor with the slot-for-tomorrow walkthrough** in `AnnotatedCode`. The student sees the bare `deleteInvoice` body (single `db.delete(...)` call) on the left, then the refactored body on the right: `const result = await db.transaction(async tx => { const existing = await tx.query.invoices.findFirst({ where: and(eq(invoices.id, id), eq(invoices.organizationId, organizationId)) }); if (!existing) return { notFound: true }; await tx.delete(invoiceLines).where(eq(invoiceLines.invoiceId, id)); await tx.delete(invoices).where(and(eq(invoices.id, id), eq(invoices.organizationId, organizationId))); return { notFound: false, deletedNumber: existing.number } });`. Six annotations: (1) the `tx` parameter is a different value from `db`; every query inside uses `tx`; missing one (`db.delete` inside the callback) breaks atomicity (the 6.4.4 / 7.2 watch-out); (2) the tenant-scoped `findFirst` reads existing inside the tx so the row-count check is atomic with the delete; (3) the `tx.delete(invoiceLines)` line is the *first* future-extension slot — Unit 10's audit-log write, Unit 11's soft-delete flag write, all land here; (4) the `tx.delete(invoices)` line is the second slot — Unit 11's `deletedAt` update replaces it for soft delete; (5) the return-a-discriminated-value pattern for the not-found case (throwing inside the callback would roll back the transaction; returning lets the action map to the right `Result`); (6) `revalidatePath` lives *outside* the tx (the 7.2 Concept 10 rule made operational once more — calling it inside would invalidate the cache even if the transaction rolled back). The closing line names the trade explicitly: the FK alone would work today; the tx wrapper is the shape that holds for tomorrow. **Beat two — the no-external-calls-inside-the-tx ambush**, recalling 7.2 Concept 10's two-lane transaction-rollback timeline. The student sees one frame: the temptation — "while we're here, also email the customer that their invoice was deleted" — and the consequence: the `await sendDeletionEmail(...)` inside the `tx => ...` callback succeeds and sends the email; a unique-constraint violation on the next line rolls the tx back; the database has the row, the customer received an email saying it was deleted; the divergence is permanent. The fix is one line: the email send fires *after* `await db.transaction(...)` returns, on the success branch. Hold the line: external side effects fire after the commit; Unit 8 owns the email send, Unit 14 owns the notification, both land outside the tx.

**Engagement.** A `Sequence` drag-to-order drill on the future-extended delete body: the student orders eight steps (`safeParse`, authorize, `db.transaction(async tx => { tx.findFirst existing; tx.insert audit-log; tx.delete invoiceLines; tx.delete invoices; })`, `revalidatePath`, `await sendDeletionEmail`, `redirect`, `return ok`) including the distractors of "`revalidatePath` *inside* the transaction" and "`sendDeletionEmail` *inside* the transaction." Wrong placements produce targeted feedback ("revalidating inside the tx invalidates the cache even if the write rolled back"; "the email inside the tx is the canonical production divergence — the row rolls back, the customer received an email saying their invoice was deleted"). Reuses the 7.2 Concept 10 drill shape — confirms the forward-link. Confirm with a `MultipleChoice`: "a teammate's PR adds an audit-log row inside the delete tx and an email send inside the same tx. What does the senior reviewer ask for?" → the audit-log stays inside (atomicity with the delete), the email moves outside (no rollback semantics for external side effects).

## Concept 12 — The verify lesson as failure-rehearsal pass

**Why it's hard.** Verification reads as bureaucracy — a checklist the student ticks to ship — until the student understands what each Done-when clause is *for*. Each clause corresponds to a discipline that has a failure mode if skipped: the JS-disabled submit fails when the form regresses to controlled inputs; field errors don't render when the form layer reads from local state instead of the action's `Result`; the optimistic rollback never fires when `addOptimistic` runs outside the transition; the delete confirmation breaks when the dialog has no no-JS fallback; the transactional atomicity fails when one of the deletes uses `db` instead of `tx`; the revalidation never lands when `revalidatePath` is missing. The verify lesson's job is not to *check the boxes* but to *rehearse each failure* — to make the student feel what would have shipped if the discipline weren't installed. Without the rehearsal, the verify lesson is a checklist; with it, the student leaves the chapter knowing exactly what a senior CRUD surface guards against.

**Ideal teaching artifact.** Pattern archetype delivered as a **failure-rehearsal pass** through the six Done-when clauses, each as a small three-beat scrub in `DiagramSequence` (or, if the chapter's scrub model needs simplification, a `Steps`-shaped vertical list with one expandable row per clause). For each clause, three beats: (1) *the verify step* — the action the student takes (e.g., "DevTools → Disable JavaScript → reload `/invoices/new` → submit a valid invoice"); (2) *the failure that would have shipped* — what the user would have seen if the discipline were missing (e.g., "the form would show 'JavaScript required to use this app' or hang on submit; the row never writes; the user assumes the app is broken"); (3) *the discipline that prevented it* — the named concept and the structural rule (e.g., "Concept 7's uncontrolled inputs + Concept 4's `redirect` on the success branch — the browser submits to the action's URL natively, the action runs, the redirect fires, the row writes"). The student walks the six clauses (JS-disabled submit, inline field errors, conflict path via duplicate `number`, optimistic rollback via `?fail=1`, transactional atomicity via temporary `throw`, `revalidatePath` after every mutation) each as a three-beat scrub, and leaves with the verify checklist *and* the failure-mode catalog the verification protects against. The closing beat is the chapter's senior posture distilled in six rules: every action parses on entry; every action returns `Result`, never throws on expected failures; every action revalidates after the write, before the return; every form is a Client Component with uncontrolled inputs and `defaultValue`; field errors render from the action's `Result`, never from local state; the delete transaction's shape holds for the audit-log, notification, and external-call extensions of Units 8–14.
