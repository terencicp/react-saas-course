# Chapter 8.2 — Authoring templates: pedagogical approach

## Concept 1 — Email HTML is its own constrained subset, and that decides the entire chapter

**Why it's hard.** The student arrives with 2026 web reflexes — `flex`, `grid`, custom properties, container queries, the modern cascade — and reads "React Email" as "React for the browser, served as email." It isn't. The interoperable rendering baseline across the mailbox-client matrix (Gmail web/iOS/Android, Apple Mail macOS/iOS, Outlook Windows legacy and Edge-based, Outlook Mac, Yahoo, Proton, Spark, enterprise webmail) is roughly HTML 4 plus table layouts plus inline styles plus a sliver of media queries. Without that picture on the page, every later constraint in the chapter — why `<Tailwind>`'s utility set is narrower, why images need explicit `width`/`height`, why dark mode needs head meta tags — reads as React Email's fussiness rather than the substrate's reality. The whole chapter pivots on this frame landing first.

**Ideal teaching artifact.** Concept archetype delivered as a **compatibility-grid reveal with a same-template-three-renderings contrast**. The grid is rendered as `Figure`-wrapped hand-coded HTML — rows are modern CSS features the student reaches for daily (`flex`, `grid`, custom properties, `@layer`, container queries, viewport units, `position: sticky`, `gap`, the modern color functions), columns are the mailbox clients that matter (Gmail web, Gmail iOS, Apple Mail, Outlook Windows). Cells are `supported` / `partial` / `not supported` / `silently breaks` (the worst category — the email *renders*, just wrong). The grid ends with one row underlined: "table layouts + inline styles + a small subset of media queries — the interoperable baseline." Below the grid, a side-by-side three-panel comparison renders the same simple email template three ways: the browser's view (clean, modern), Gmail web's view (close, but the `flex` collapse is visible), and Outlook Windows' view (the table layout is intact, but the `flex`-styled hero stacks wrong). The reveal lands the chapter's spine: the template the team writes in JSX is *not* the HTML that ships, and the renderer's job is to bridge that gap.

**Engagement.** A `Buckets` sort of fifteen CSS features into "use freely in email," "use only inside `<Tailwind>` and check the preview," and "do not reach for — substrate doesn't render it." Items include `flex`, `grid`, `padding`, `font-size`, `@media (max-width: 600px)`, custom properties, `border-radius`, `position: absolute`, `gap`, `text-align`, `box-shadow`, `transform`, viewport units, `aspect-ratio`, and `<table>` layout. Wrong-answer feedback names the worst-client failure mode for each.

**Components.**
- `Figure` wrapping a hand-coded HTML compatibility grid (CSS features × mailbox clients). Single-use static composition — the picture is the teaching.
- `Figure` wrapping a hand SVG of the three-panel "same template, three renderings" contrast beneath the grid. Single-use static composition; the kinetic moment is the visual mismatch, not interaction.
- `Buckets` for the fifteen-item CSS sort.
- `Aside` (`note`) below: "the substrate is the constraint; React Email's whole job is to let you write 2026 React while emitting the HTML the worst client still parses."

**Project link.** Chapter 8.3.4's welcome template is built directly against this grid's "use freely" column; the student returns to the picture whenever a stray `flex` reflex slips into a template and the preview server's HTML tab shows the inline-style fallback.

## Concept 2 — The component vocabulary as one read-once map, anchored on the failure mode each primitive prevents

**Why it's hard.** The React Email package ships ~15 components and the temptation is to learn them as a flat API list — `<Container>`, `<Section>`, `<Row>`, `<Column>`, `<Heading>`, `<Text>`, `<Link>`, `<Button>`, `<Img>`, `<Hr>`, `<Preview>`, `<Html>`, `<Head>`, `<Body>`, `<Tailwind>`. The student memorizes names, ships templates that use raw `<div>`s anyway, and silently breaks Gmail. The senior reach is the inverse: each component exists because a specific failure mode in raw email-HTML kept biting authors, and the component's job is to make that failure unwriteable. `<Button>` exists because Outlook's VML fallback for bulletproof CTAs is a paragraph of XML the author shouldn't hand-write. `<Img>` requires `width`/`height` because Outlook ignores CSS dimensions and renders at natural pixel size. `<Preview>` exists because the inbox preheader pulls from the first visible body line otherwise and that line is almost always wrong. Without the failure-per-primitive frame, the vocabulary is pure memorization and the student reaches for `<div>` under deadline pressure.

**Ideal teaching artifact.** Reference archetype delivered as an **anatomy poster** of a worked welcome email template. A `Figure`-wrapped hand SVG renders the rendered email on the left (a realistic preview of the welcome template — header with logo, hero heading, body paragraph, CTA button, footer) with annotation lines pointing from each visible region to a labeled card on the right. Each card carries: the React Email component name, the one-sentence "what it produces in the wire HTML" (e.g., `<Container>` → centered 600px-max table cell), and the failure-mode-it-prevents tag (e.g., `<Img>` → "Outlook ignores CSS width — `width`/`height` attributes are non-negotiable"; `<Preview>` → "the inbox preheader pulls from the first body line if you skip this, almost always wrong"; `<Button>` → "Outlook needs a VML fallback — hand-rolling it is a paragraph of XML"). The poster shows the *whole template at once*, with the vocabulary laid over it as labels. The student leaves with a single picture they re-scan when authoring, not a flat list to memorize.

**Engagement.** A `Tokens` round on the same template's *source JSX* — the student clicks each component tag in turn and is asked to name (a) what wire HTML it emits and (b) the failure mode it prevents. The token surface forces recall of both halves; the wrong-answer feedback names the call site where the failure would surface (a `<div>` instead of `<Section>` collapses to default block flow in Gmail; a missing `<Preview>` lets the inbox surface the wrong line).

**Components.**
- `Figure` wrapping a hand SVG of the rendered template + annotation cards (the anatomy poster). Single-use static composition; the picture is the teaching.
- `Tokens` for the source-JSX click round.
- `Aside` (`tip`) below: "the vocabulary isn't ~15 names to memorize — it's ~15 failure modes the renderer makes unwriteable; reach for the component, not the raw tag."

**Project link.** Chapter 8.3.4's welcome template is the same template the poster annotates; the student arrives at 8.3.4 already knowing which component owns each region.

## Concept 3 — The `<Tailwind>` wrapper: same vocabulary, narrower subset, inline-style output

**Why it's hard.** The Unit 4 muscle memory — `flex`, `grid`, `dark:`, arbitrary properties — collides head-on with the email substrate. `<Tailwind>` looks like the same wrapper, the preview in Chrome renders `flex` correctly, and the template ships. Gmail collapses to default block flow; the student ships a broken hero section and can't see why. Three things have to land together: (1) the wrapper *compiles to inline styles per element*, not a CSS class on a stylesheet — the Unit 4 mental model of "classes in the markup, rules in the stylesheet" doesn't apply; (2) the supported utility subset is narrower than the web build (`flex`, `grid`, container queries, `dark:` without head plumbing — all dropped silently with a console warning), and the safe set is roughly typography, spacing, color, basic borders, and `max-w-*`; (3) arbitrary values (`max-w-[37.5rem]`) are first-class because email layout often needs exact pixel widths the design tokens don't cover. The misframe — "it's just Tailwind" — produces broken templates whose failure mode is invisible in Chrome.

**Ideal teaching artifact.** Mechanics archetype delivered in two coordinated beats. **Beat one — the compile reveal.** A `CodeVariants` two-tab block. Tab one: a small `<Tailwind>`-wrapped JSX snippet (`<Section className="bg-zinc-50 px-6 py-8 max-w-[600px]">…</Section>`). Tab two: the rendered HTML output the renderer emits — the same `<table>` with `style="background-color:#fafafa;padding:1.5rem 2rem;max-width:600px"` inlined on the cell, no class attribute anywhere. The student sees the "Tailwind classes in source, inline styles on the wire" pipeline once and the mental model snaps into place. **Beat two — the supported-subset map.** A `Figure` wrapping a hand-coded HTML 3-column reference: "use freely" (typography, spacing, color, basic borders, `max-w-*`, `text-center`/`text-left`, hex via `bg-[#...]`), "use with caution — preview verifies" (`@media`-driven responsive prefixes, `dark:` with the head plumbing from concept 7), "drops silently" (`flex`, `grid`, container queries, `gap` on non-table layouts, transforms, viewport units). Each row carries a one-line "what to reach for instead" — `<Row>`+`<Column>` instead of `flex`, `<Section>` stacks instead of `grid`, `padding` on `<Column>` instead of `gap`.

**Engagement.** A `MultipleChoice` PR-review round of four scenarios: *teammate ships `<div className="flex items-center gap-4">` inside a `<Tailwind>` wrapper "because the preview looks fine"* → block, the preview is Chrome, Gmail collapses to block flow, reach for `<Row>`+`<Column>`; *teammate uses `bg-[#1a1a1a]` instead of `bg-zinc-900`* → approve, arbitrary hex is first-class in `<Tailwind>` and avoids the brand-color drop on clients that strip class-derived properties; *teammate ships `dark:bg-zinc-900` without the `<meta name="color-scheme">` head plumbing* → flag, Apple Mail ignores the variant entirely without it (concept 7); *teammate uses `mx-auto max-w-[600px]` for centering* → approve, that's the canonical pattern. Wrong-answer feedback names the failure mode each PR introduces.

**Components.**
- `CodeVariants` for the two-tab compile reveal (JSX source vs. emitted inline-style HTML).
- `Figure` wrapping a hand-coded HTML 3-column supported-subset reference. Single-use static composition.
- `MultipleChoice` for the PR-review round.
- `Aside` (`caution`) below: "the preview's render is Chrome, not Gmail — an unsupported utility is a console warning the student doesn't read; the failure mode arrives at send time."

**Project link.** Chapter 8.3.4's welcome template uses `<Tailwind>` for the brand color, typography, and spacing only; the layout is `<Container>`/`<Section>`/`<Row>`/`<Column>`. The student reads the project's template as the supported-subset reference made concrete.

## Concept 4 — Props-only templates and the `PreviewProps` contract

**Why it's hard.** The student's first instinct is to read the current user, environment variables, or the database row from inside the template ("it's just a React component, the data's right there"). Three failure modes converge: (1) the preview server can't render the template — there's no user session, no DB connection, the render throws; (2) the unit test or snapshot can't render the template for the same reason; (3) the production send and the preview render diverge silently because they pull data from different sources. The senior shape is the inverse: the template is a *pure renderer* with typed props; the caller (the Server Action) computes every dynamic value and passes it in; `PreviewProps` is the mock-data contract that lets the preview, the test, and the production send render *identically*. The misconception that hits hardest is treating `PreviewProps` as "the preview's fake data" rather than "the contract the production caller must satisfy" — once the student sees that the prop shape *is* the template's API, the discipline lands.

**Ideal teaching artifact.** Pattern archetype delivered as a **three-render parity walk**. A `Figure`-wrapped hand SVG diagram with three lanes converging on a single `<WelcomeEmail />` template at the center: lane one — the *preview server* fires the template with `WelcomeEmail.PreviewProps = { firstName: 'Ada', verifyUrl: 'https://...' }`; lane two — a *unit test* renders the template with the same prop shape (`render(<WelcomeEmail firstName="Ada" verifyUrl="https://..." />)`); lane three — the *production Server Action* fetches the user row, computes the verify URL, calls `sendEmail({ react: <WelcomeEmail firstName={user.firstName} verifyUrl={url} /> })`. All three lanes feed the same component with the same prop shape; all three produce the same HTML. Below the diagram, a `CodeVariants` two-tab contrast: tab one — the *bad shape*, the template reads `useSession()` and `process.env.APP_URL` and fetches the user inside; the preview server crashes, the test crashes, the production send diverges from the preview by a path the team can't reproduce. Tab two — the *good shape*, the template's signature is `({ firstName, verifyUrl }: WelcomeEmailProps) => …`, the action does the data work, the prop shape *is* the API. The reveal: `PreviewProps` isn't preview-only fake data — it's the prop-shape contract every caller obeys.

**Engagement.** A `MultipleChoice` round of four scenarios: *the template needs the current year for the footer copyright* → derive in props from the caller, never `new Date()` inside the template (snapshot tests fail at midnight, preview drifts from production); *the template needs the brand logo URL* → pass via props from a single shared `EmailLayout` config or read from a build-time constant module, *not* from a DB row; *the template needs to know if the user is on a paid plan to show a different CTA* → derive in the action, pass `ctaVariant: 'paid' | 'free'` as a prop; *the template needs the recipient's email address for the unsubscribe link* → pass via props, the template never reads from session or DB. Wrong-answer feedback names the failure mode the misframe produces (preview crashes, test flakes, snapshot drift, production-vs-preview divergence).

**Components.**
- `Figure` wrapping a hand SVG of the three-render parity diagram (preview / test / production lanes converging on the template). Single-use static composition.
- `CodeVariants` for the two-tab bad-shape vs. good-shape contrast.
- `MultipleChoice` for the four-scenario derive-where round.
- `Aside` (`tip`) below: "the prop shape *is* the template's API — `PreviewProps` is the contract every caller satisfies, not the preview's private mock data."

**Project link.** Chapter 8.3.4 builds the welcome template with a typed `WelcomeEmailProps` interface and a `PreviewProps` named export with the canonical mock values; the `sendWelcomeEmail` Server Action computes the same props from the user row. The chapter's "Done when" verification asserts the preview render and the production send produce the same HTML for the same prop values.

## Concept 5 — The preview server as the inner loop, and the canonical save-eyeball-fix discipline

**Why it's hard.** Two anti-patterns compete for the student. (1) The "send-from-staging-and-refresh-Gmail" loop — every iteration is a real send to a real inbox, the feedback latency is 30–90 seconds, the student stops iterating after three tries and ships a template they only half-checked. (2) The "preview only in the browser tab without the dev server running" loop — the student writes the template, opens the rendered HTML in a browser once, eyeballs it, ships. The mobile view never gets checked, the dark mode toggle never gets clicked, the plain-text tab is invisible, and the failure modes (the 40-character first name that wraps the heading, the dark-mode inversion that hides the logo, the empty preheader) ship to production. The senior loop is neither: `pnpm email dev` runs in a side terminal during *any* template work; every save triggers a hot reload; every check (desktop, mobile, dark, plain-text, edited `PreviewProps`) is a single click in the chrome; the test-send is the *verification gate*, fired once at the end, not the iteration tool. The discipline is the lesson; the syntax (one CLI flag, one keyboard shortcut) is small.

**Ideal teaching artifact.** Setup-and-pattern archetype delivered as a **canonical loop walkthrough** in `DiagramSequence`. Seven frames the student scrubs through, each frame a hand SVG showing the developer's screen state at that step: (1) write the template in `emails/welcome.tsx`; (2) define `PreviewProps` with realistic mock data (a 12-character `firstName`, a 60-character `verifyUrl` — the realistic shape, not "Test Test"); (3) `pnpm email dev --port 3001` running in a side terminal, the preview chrome on the second monitor; (4) save the file, the preview hot-reloads, eyeball at desktop; (5) toggle the chrome's viewport switch to mobile, eyeball at 375px — the heading wraps, fix the copy, save, re-eyeball; (6) toggle the dark-mode switch, eyeball — the logo on dark is invisible, note the fix for concept 7, save; (7) read the plain-text tab — the CTA reads as `Verify your email: https://...` (good) not `Verify [https://...]` with no context (bad), reword the link text, save. Below the seven-frame sequence, a single eighth frame breaks out as the *test-send gate*: fire the preview's "Send test" button to a personal Gmail and a personal Apple Mail, eyeball the actual client renders, *then* wire the template into the Server Action. The frames make the loop *visible* — every step has a failure mode the previous step doesn't catch, and the student sees that pattern.

**Engagement.** A `Sequence` drag-and-order drill: nine action cards from a real iteration session — *write the template*, *define `PreviewProps` with realistic mock values*, *start `pnpm email dev` in a side terminal*, *save and eyeball at desktop*, *toggle to mobile and re-eyeball*, *toggle to dark and re-eyeball*, *read the plain-text tab*, *test-send to a real Gmail and a real Apple Mail*, *wire into the Server Action* — plus three trap cards the student rejects by leaving them out (*ship the template after a single desktop preview*, *test-send to the team's `team@yourapp.com` shared inbox nobody opens*, *iterate by re-deploying staging and refreshing Gmail*). Wrong-answer feedback names which loop-failure each trap produces.

**Components.**
- `DiagramSequence` for the seven-frame canonical loop walkthrough plus the eighth test-send gate frame. Each frame is a hand SVG showing the dev screen state — the editor, the preview chrome with the active toggle highlighted, the side counter for "checks completed."
- `Sequence` for the nine-action drag-and-order drill.
- `Aside` (`tip`) below: "the preview server is the iteration loop, the test-send is the verification gate — never use the test-send as the iteration tool, and never ship without firing the gate."

**Project link.** Chapter 8.3.4 walks the student through this exact loop against the welcome template; the project's `package.json` ships with the `email` script pre-wired and the README documents the port-3001 convention from this concept.

## Concept 6 — Plain-text fallback as a generated artifact, never hand-maintained

**Why it's hard.** The plain-text part of a `multipart/alternative` email is invisible to most authors — the receiving client picks `text/html` when both parts exist, and the developer never sees the text version. The instinct, once the student learns the part exists, is to hand-maintain it as a `text` field in the send call. Within two releases the text version has drifted: the HTML's CTA is "Verify your email," the text version still reads "Confirm your account" from a copy change three sprints back. Three audiences silently miss the message — the screen-reader user whose client renders text-only, the corporate-policy plain-text mailbox, the spam filter that scans the text part for keyword matching (a missing or empty text part is itself a small spam signal). The senior reflex is structural: the text part is *generated from the same JSX* via `render({ plainText: true })`, the SDK's `react` prop runs both renders internally, the application code never touches the text part directly. The text-version *coherence check* — reading the generated text in the preview's plain-text tab and confirming it reads as a standalone message — is the only manual step.

**Ideal teaching artifact.** Pattern archetype delivered in two coordinated beats. **Beat one — the multipart anatomy reveal.** A `Figure`-wrapped hand SVG of a single MIME message on the wire, with the `multipart/alternative` boundary visible and both parts shown side by side: `text/plain` on the left (the generated text version), `text/html` on the right (the inlined-style HTML). Three labeled arrows point from the message into three audience lanes: a screen-reader icon (`text/plain` rendered by the assistive tech), a corporate-mailbox icon (`text/plain` shown because the client strips HTML), a spam-filter icon (the filter scores the `text/plain` part for keyword matching). The reveal: the text part is non-optional, and three real audiences see it instead of the HTML. **Beat two — the generated-vs-hand-maintained contrast** in `CodeVariants`. Tab one — the *bad shape*, the call site passes a hand-written `text` field; the JSX changes for the marketing team's copy update, the `text` field doesn't, the screen-reader user reads the old copy for a quarter. Tab two — the *good shape*, the call site passes only the `react` prop, the SDK runs `render({ plainText: true })` internally, both parts ship from the same JSX, the coherence check happens in the preview's plain-text tab. The closing rule: the `text` field is *only* for plain-text-only messages (rare for transactional); the `react` prop is the default and ships both parts.

**Engagement.** A `MultipleChoice` round on the *coherence check* — four plain-text outputs from the preview, the student picks the failure mode in each: *"Verify your email [https://yourapp.com/verify/abc-123]"* → good, the link is labeled and parseable; *"Verify [https://yourapp.com/verify/abc-123]"* → bad, the link text doesn't say what's being verified, fix in the JSX; *"Logo image View in browser image"* → bad, decorative images are leaking — set `alt=""` on the logo and hero decoration; *the footer text appears at the top of the plain-text version* → bad, a `<Container>` nesting bug in the JSX is reordering sections, fix the structure. Forces the student to *read* the plain-text tab the way a senior would, not just confirm it exists.

**Components.**
- `Figure` wrapping a hand SVG of the multipart-anatomy reveal (MIME message + three audience lanes) for beat one. Single-use static composition.
- `CodeVariants` for the two-tab generated-vs-hand-maintained contrast in beat two.
- `MultipleChoice` for the coherence-check round.
- `Aside` (`caution`) below: "hand-maintaining the text field is a drift bug shipping in two releases — the SDK's `react` prop generates both parts from one JSX, and the only manual step is reading the preview's plain-text tab."

**Project link.** Chapter 8.3.4's welcome action calls `sendEmail({ react: <WelcomeEmail … /> })` with no `text` field; 8.3.5's verification asserts the multipart message arrived with both parts and the plain-text version reads coherently. The student returns to the coherence-check vocabulary from this concept to do the verification.

## Concept 7 — The dark-mode three-tier posture, the head-meta plumbing, and the WCAG floor that holds in light

**Why it's hard.** Three failures stack. (1) The student treats `prefers-color-scheme: dark` as a reliable signal because that's how the web works; mailbox clients fall into three groups — *no transformation* (Apple Mail desktop, recent Outlook), *partial inversion* (Gmail iOS, Outlook iOS — light backgrounds invert, dark elements preserved), *full inversion* (Gmail Android in some configs, some Outlook installs — everything inverts, logos become photo negatives, brand colors hue-shift). The single posture that works on the web breaks on email. (2) The student opts into Tailwind's `dark:` variant and ships, not knowing that without two `<meta>` tags and an inline `<style>` block in `<Head>`, Apple Mail ignores the dark styles entirely even when the user is in dark mode. (3) The student treats dark mode as the *accessibility* fix and lets the light-mode template fail WCAG contrast (4.5:1 body, 3:1 headings/large text), not knowing that most clients don't honor `prefers-color-scheme: dark` and the *default* template must already meet contrast in its light shape. The senior posture is the inverse: the light template meets WCAG AA on its own; dark mode is opt-in plumbing for the clients that respect it; full-inversion clients are designed *defensively* (logo with a `<picture>` swap, brand colors that survive hue-shift). The accessibility checklist (lang, headings, link text, alt, contrast, font size, touch targets) sits *underneath* the dark-mode posture as the substrate, not the dark-mode story.

**Ideal teaching artifact.** Decision-and-pattern archetype delivered in two coordinated beats — neither carryable alone.

**Beat one — the three-tier client behavior simulator.** A `DiagramSequence` walks four frames the student scrubs through, each frame the same welcome template rendered under one client behavior. Frame one — *no transformation* (Apple Mail dark mode): the template ships as authored, the inline `@media (prefers-color-scheme: dark)` block applies, the brand-color CTA stays brand, the logo swaps via `<picture>`, the result is clean. Frame two — *partial inversion* (Gmail iOS): light backgrounds inverted by the client, brand colors preserved, the dark-mode `@media` block doesn't fire (Gmail iOS uses its own algorithm), the result is mostly fine because the *light template's brand color survives*. Frame three — *full inversion* (Gmail Android, blanket inversion config): everything inverted by the client, the logo is a photo negative, the brand-color CTA hue-shifted, the *light template's defensive choices* (logo with transparent background, brand color chosen to survive inversion) are what carries the result through. Frame four — *the WCAG-floor reveal*: a fifth panel below shows the light template's *contrast scorecard* — body text 7.2:1 (passes AA and AAA), CTA 4.6:1 (passes AA), heading 5.1:1 (passes AA on large text). The light template is *already accessible* without dark mode; dark mode is the bonus for clients that honor it, not the accessibility fix.

**Beat two — the head-plumbing anatomy.** A `Figure` wrapping a hand SVG of the rendered email's `<head>` section with the three required pieces highlighted: `<meta name="color-scheme" content="light dark" />`, `<meta name="supported-color-schemes" content="light dark" />`, an inline `<style>` block carrying `:root { color-scheme: light dark; } @media (prefers-color-scheme: dark) { … }`. Each piece annotated with the client behavior it unlocks (the meta tags signal Apple Mail to apply the template's dark styles; without them, Apple Mail ignores the styles entirely even in dark mode). A small reference panel below maps the two styling approaches: the inline `@media` block (most reliable, write once for the brand-color and logo swap) versus Tailwind's `dark:` variant (supported by Apple Mail and recent Outlook only, useful for simple swaps, not a substitute for the inline block). The `<picture>` element with a `media="(prefers-color-scheme: dark)"` source is named here as the logo-swap pattern for the "logo on dark" failure case from beat one's frame two.

**Engagement.** A two-stage round. Stage one — a `Buckets` sort on the *accessibility checklist*, eight items sorted into "ships in every template" (lang attribute on `<Html>`, document title in `<Head>`, one h1 per email, link text describes destination, decorative images get `alt=""`, body text >= 14px, CTAs >= 44×44px touch target, color contrast meets WCAG AA in light mode) and "marketing-template concern, not transactional" (full RTL pass with `dir="auto"`, marketing one-click unsubscribe header, preferences-center link). Stage two — a `MultipleChoice` round on dark-mode design choices: *teammate sets a near-white background and a near-black logo, no `<picture>` swap* → flag, the logo will invert to invisible on full-inversion clients; *teammate ships `dark:bg-zinc-900 dark:text-zinc-50` without the `<meta name="color-scheme">` head plumbing* → flag, Apple Mail ignores the variant; *teammate uses brand color #3b82f6 for body text on white* → flag, contrast is ~3.6:1, fails WCAG AA on body (4.5:1 required); *teammate ships only the `<meta>` tags and skips the `@media (prefers-color-scheme: dark)` block* → approve as the minimum for transactional, the brand color on a white background will survive partial-inversion clients without the explicit dark block.

**Components.**
- `DiagramSequence` for the four-frame three-tier client behavior simulator + the contrast-scorecard reveal in beat one. Each frame is a hand SVG showing the rendered template, the client's transformation, and the visible result; the scorecard frame shows numeric contrast ratios on each text region.
- `Figure` wrapping a hand SVG of the head-plumbing anatomy + styling-approach reference panel for beat two.
- `Buckets` for the accessibility-checklist sort.
- `MultipleChoice` for the dark-mode-design round.
- `Aside` (`caution`) below: "the light template meets WCAG AA on its own — dark mode is opt-in plumbing, defensive design carries the full-inversion clients, and `prefers-color-scheme: dark` is never the accessibility fix."

**Project link.** Chapter 8.3.4's welcome template ships the `<meta name="color-scheme">` plumbing and one `@media (prefers-color-scheme: dark)` block for the brand color and logo swap; 8.3.5's verification includes a real test-send to Apple Mail (dark mode), Gmail iOS (partial inversion), and Gmail Android (full inversion if available) to confirm the three-tier posture lands.

## Component proposals

None — the chapter ships entirely on existing components.

The chapter's heaviest teaching beats — Concept 5's seven-frame canonical loop walkthrough and Concept 7's four-frame three-tier client simulator — each ride on `DiagramSequence` hosting hand SVGs as its frame slots. The kinetic moment in each is the *scrub* that exposes the cause-and-effect (each loop step has a failure mode the previous step doesn't catch; each client behavior tier transforms the same template differently); `DiagramSequence`'s slider already carries that vocabulary from Chapter 8.1's six-stage deliverability chain and Chapter 6.1's pool-exhaustion scrub. No bespoke component earns its weight against that toolkit, and a "PreviewLoopWalker" or "DarkModeSimulator" widget would be single-use without a credible forward-link — the chapter is the only place these visuals live in the curriculum, with Chapter 8.3 reusing the *vocabulary* by reference rather than re-rendering the components.

The remaining concepts each reach a strong fit with `Figure` + hand-coded HTML tables (Concept 1's compatibility grid, Concept 3's supported-subset reference), `Figure` + hand SVG (Concept 1's three-rendering contrast, Concept 2's anatomy poster, Concept 4's three-render parity diagram, Concept 6's multipart anatomy, Concept 7's head-plumbing anatomy), `CodeVariants` (Concepts 3, 4, 6 for the source-vs-output and bad-vs-good contrasts), and the existing exercise components (`Buckets`, `Tokens`, `MultipleChoice`, `Sequence`).

The chapter validates the same posture as Chapter 8.1 and Chapter 7.1: when the existing toolkit's visual primitives (the scrub, the annotated table, the side-by-side contrast, the anatomy poster) carry the kinetic moment that teaches, the bespoke component proposal is the wrong reach.
