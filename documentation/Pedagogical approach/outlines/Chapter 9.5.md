# Chapter 9.5 — Project: email+password auth with verification: pedagogical approach

## Concept 1 — The cuts panel: what 9.5 ships and what later chapters and units own

**Why it's hard.** Unit 9 is a buffet — OAuth, magic links, TOTP, passkeys, account linking, password reset, change-email, the active-sessions surface, rate limits — and the student arrives at the project chapter with all of them on the table. ## Concept 2 — Reading the starter as a contract: the catch-all handler is the spine, not three wired routes

**Why it's hard.** The student opens the starter, sees `proxy.ts`, `lib/auth.ts`, `lib/auth-client.ts`, `lib/auth-helpers.ts`, the catch-all `app/api/auth/[...all]/route.ts`, and the empty `(auth)/sign-up`, `(auth)/sign-in`, `(auth)/verify-email`, `(protected)/dashboard` segments. Two reflexes ship the chapter wrong. (1) **The catch-all is the only auth route file** — the student who reads "I need a sign-in endpoint" and writes `app/api/auth/sign-in/route.ts` shadows the catch-all, breaks every endpoint Better Auth expects to expose, and the failure surfaces as opaque 404s from the `authClient`. (2) **The Server Actions are the form-side wire, not the auth endpoint** — the student who treats `signUp.email` as the action's full body misses that the action wraps the library call with the project's `Result` shape, the Zod parse, the conflict mapping, and the `redirect`. The page is the contract; the action satisfies it; the catch-all carries everything the form-side actions don't. The student needs to read the file tree as a *layered surface* — one catch-all for the library, two thin modules at the lib seam, three form-side actions that thread the library calls into the project's Result discipline — before any code is written.

**Ideal teaching artifact.** Pattern archetype delivered as a **call-site trace** through the provided files in `AnnotatedCode`. The student walks five seams in one block: (1) `app/api/auth/[...all]/route.ts` — three lines, the catch-all that mounts every Better Auth endpoint (`sign-up/email`, `sign-in/email`, `callback/google`, `session`, `verify-email/...`); the `[...all]` catch-all segment is annotated as the **only** auth route file the project ever writes; (2) `lib/auth.ts` — the `'server-only'` first line, the `betterAuth({...})` body the student fills, the `nextCookies()` plugin marker; (3) `lib/auth-client.ts` — the `createAuthClient({ baseURL })` line, the only browser-side door, the `useSession()` hook the protected layout *doesn't* use (the layout reads server-side via `requireUser`); (4) `app/(auth)/sign-up/actions.ts` — the action's five-seam shape, calls `auth.api.signUpEmail` *without* `headers` (the `nextCookies()` plugin attaches the cookie to the response), maps unique-violation to `'conflict'`, redirects to `/verify-email`; (5) `proxy.ts` — empty body with a TODO header, framing the cookie-presence-only gate the student writes in 9.5.5. Three annotation tracks per file: (a) what the file owns (library mount vs. lib seam vs. action wire vs. proxy gate); (b) what the file imports (`auth` vs. `authClient` — never both in one file); (c) what would break if the student wrote the wrong shape (a per-action route file would shadow the catch-all; importing `auth` into a `'use client'` form would trip `'server-only'`; reaching for `auth.api.getSession` in the proxy collapses the two-layer model). The closing line names the senior reflex: one catch-all, two lib modules, one form-side action per surface — never per-action route files.

**Engagement.** A `Dropdowns` exercise on the file headers — given each file's role, the student fills in the import (`auth` from `@/lib/auth` vs. `authClient` from `@/lib/auth-client`), the directive (`'use server'` vs. `'use client'` vs. `'server-only'` vs. none), and the function the file exports. Wrong picks surface the wiring break each one would produce ("you imported `auth` into a Client Component — `server-only` refuses the bundle, the page won't compile"). Confirm with a `MultipleChoice`: "a teammate's PR adds `app/api/auth/sign-in/route.ts` to 'have tighter control over the sign-in endpoint.' What does the senior reviewer ask for?" → the catch-all is the only auth route file; the per-action file shadows the library's expected mount point and breaks endpoints the team didn't even write yet (verify-email, callback, session); revert and add the wrapping discipline at the form-side action layer if it's needed.

**Components.**
- `AnnotatedCode` walking the five-seam call-site trace across the provided files.
- `Dropdowns` for the file-header signature exercise.
- `MultipleChoice` for the per-action-route PR-review.
- `Aside` (`note`) below: "one catch-all for the library, two lib modules at the seam, one form-side action per surface. Never per-action route files."

**Project link.** This concept *is* lesson 9.5.2's senior posture — the student leaves knowing exactly what they have to write for each file, before they write a line, with the catch-all as the four-corner constraint.

## Concept 3 — `nextCookies()` as the silent-fail closer: the line that makes sign-up visible to the browser

**Why it's hard.** This is the chapter's first canonical footgun and the one that hurts most when missed because *every individual signal lies*. Server Actions in Next.js 16 don't auto-flush `Set-Cookie` headers the way route handlers do — the action returns 200, the `auth.api.signUpEmail` call writes the `user` row + `account` row + `session` row in Postgres, the network tab shows a clean 200 response, the action's `Result` is `ok({ id })`, and *no cookie reaches the browser*. The student refreshes, sees no signed-in state, opens DevTools, sees no cookie, and concludes the sign-up failed — but Postgres says it didn't. Without the failure mode rendered visceral, the student debugs every other dimension first (cookie attributes, CORS, the action's return shape, the `redirect` placement) and the actual fix — `plugins: [nextCookies()]` — is the last thing they try. The teaching has to make the symptom *recognizable on sight* so the student writes the plugin line in muscle memory and recognizes the bug pattern in a teammate's PR a year later.

**Ideal teaching artifact.** Misconception-first ambush delivered as a **wrong-by-default before/after replay** in `CodeVariants`, mirroring 9.2 Concept 2's shape and instantiated at the project's keyboard. Tab one *(broken)*: the `betterAuth({...})` config the student is about to write, with `plugins: [...]` empty. Beside it, a hand-SVG-inside-`Figure` triple panel: (a) the network tab showing the sign-up Action's response — `200 OK`, the `Set-Cookie` header line empty; (b) a small Postgres readout showing the `user`, `account`, and `session` rows freshly inserted; (c) the browser's cookie jar, empty. The student is asked to predict what `/dashboard` will do on the next render — the layout calls `requireUser`, the cookie isn't there, the user is treated as anonymous, redirect to `/sign-in`, the user thinks the sign-up failed. Tab two *(fixed)*: the same config with `plugins: [nextCookies()]` added on a highlighted line, the network panel now showing the `Set-Cookie: __Host-better-auth.session_token=...` header attached to the same Action response, the cookie jar populated, the same Postgres state. A short paragraph below names the mechanism — the plugin hooks Next.js's Server Action response pipeline and copies the cookies the library tried to set onto the outgoing response. The closing line lands the senior reflex: *if a Better Auth call appears to succeed but the user stays signed-out, check `nextCookies()` first — every other failure mode produces a non-200 or a console error; the silent failure is uniquely this one.*

**Engagement.** The before/after replay carries the assessment because the student must *make the change* in lesson 9.5.3 and watch the cookie land in DevTools. Confirm with a `MultipleChoice`: "the team reports `signUpEmail` is silently failing — the `user` row appears in Postgres, the Action returns 200, but the user isn't signed in on the next render. What's the single most likely cause?" The four options frame the wrong diagnoses (cookie attribute mismatch, CORS preflight, missing handler mount, missing `nextCookies()`); the right answer ties the symptom to the plugin and explains why the symptom is uniquely *silent* — every other failure mode produces a non-200 response or a console error.

**Components.**
- `CodeVariants` for the broken/fixed two-tab replay, with a hand-SVG-inside-`Figure` triple panel (network tab + Postgres readout + cookie jar) embedded in each tab. Reuses 9.2 Concept 2's shape — confirms the forward-link.
- `MultipleChoice` for the diagnosis confirmation.
- `Aside` (`caution`) below the variants: "no plugin, no cookie. Server Actions don't flush `Set-Cookie` on their own; `nextCookies()` is the one line that makes every downstream auth feature visible to the browser."

**Project link.** This concept *is* the load-bearing reflex of lesson 9.5.3 — the student writes the `betterAuth` config with `nextCookies()` from this replay, runs the sign-up smoke test, and the cookie landing in DevTools is the chapter's first verifiable artifact.

## Concept 4 — `requireEmailVerification: true` as the senior default, sign-up as an *unsigned* response

**Why it's hard.** Three reflexes converge on the wrong shape. (1) **The library default is `requireEmailVerification: false`** — the student copies the docs example, ships sign-up that issues a session immediately, and the unverified email is now a working credential. The chapter's first verifiable bug class is the typo-squat: anyone who types `dad@gmail.com` (the typo) instead of `dan@gmail.com` (the real address) gets a working account at the typo address; the real address's owner has nothing to verify and no recovery path. (2) **`autoSignIn: false` is the second non-negotiable** — without it, the unverified user gets a cookie attached on sign-up, signs into the app, and the verification email becomes optional decoration; the student sees the dashboard, ignores the inbox, the gate dies. (3) **`autoSignInAfterVerification: true` is the senior call on the *verify-click*** — the user just proved control of the inbox by clicking the link; dropping them at a sign-in form to re-type the password they just typed is a UX regression with no security upside. The three knobs are *one posture*, not three independent dials, and the student needs to feel the posture shift the response shape — sign-up becomes "credential stored, verification queued, *no cookie*", verify-click becomes "credential proven, session issued, cookie lands". Without the posture made visceral, the student picks two of three and ships the half-state.

**Ideal teaching artifact.** Pattern archetype delivered as a **two-config side-by-side replay** in `CodeVariants`, mirroring 9.3 Concept 1's shape and instantiated at the project's keyboard. Tab one *(naive default — the broken state lesson 9.5.3 deliberately ships, with the TODO comment to flip)*: `emailAndPassword: { enabled: true }` only. Beside it, a hand-SVG-inside-`Figure` panel showing what lands on submit: a `user` row with `emailVerified: false`, an `account` row with `providerId: 'credential'` and the bcrypt hash, *and* a `__Host-` cookie attached to the response. The student predicts the next-page render — the unverified user is signed in, the inbox is bypassed, the typo-squat hole is open. Tab two *(senior baseline — what 9.5.4 flips to)*: the same config plus `requireEmailVerification: true`, `autoSignIn: false`, `minPasswordLength: 12`, plus the `emailVerification: { sendVerificationEmail, sendOnSignUp: true, autoSignInAfterVerification: true, expiresIn: 60 * 60 }` block. The same panel now shows: the `user` and `account` rows with `emailVerified: false`, *plus* a `verification` row with the hashed token, *no* cookie, the response carrying an `ok({ id })` Result, the action's `redirect` lands at `/verify-email?email=...`. A side caption pins the rule three ways: the password lands on `account` (not `user` — the 9.2.3 four-table decomposition holds); the cookie shows up only after the verify-click, never on sign-up; the verify-click *does* attach the cookie because `autoSignInAfterVerification: true` is the senior call. The closing line names the failure mode the posture closes: *unverified email is not a credential — every SaaS that ships with `requireEmailVerification: false` ships with the typo-squat hole built in.*

**Engagement.** A `Tokens` exercise on the canonical `auth.ts` config block: the student clicks the four load-bearing tokens (`requireEmailVerification: true`, `autoSignIn: false`, `sendOnSignUp: true`, `autoSignInAfterVerification: true`) and gets a one-line reveal per token naming the bug each default ships if left wrong. Confirm with a `MultipleChoice`: "the team enables `emailAndPassword: { enabled: true }` and ships sign-up. QA reports the user is signed in immediately and never sees the verification email — three knobs in the config are missing or wrong. Which is the senior baseline?" The four options frame the misconfigurations; the right answer names the senior triple plus the reasoning for `autoSignInAfterVerification: true` on the verify-click side.

**Components.**
- `CodeVariants` for the naive/senior two-tab replay, with a hand-SVG-inside-`Figure` row-and-cookie panel embedded in each tab. Reuses 9.3 Concept 1's shape — confirms the forward-link.
- `Tokens` on the senior config block for the four-knob reveal.
- `MultipleChoice` for the four-misconfig diagnosis.
- `Aside` (`caution`) below the variants: "the password lives on `account`, never on `user`. The cookie shows up after the verify-click, never on sign-up. The library defaults ship the typo-squat hole; the senior baseline closes it."

**Project link.** This concept lives in lesson 9.5.3 (where `requireEmailVerification: false` is *temporarily* set with the TODO marker) and lesson 9.5.4 (where the flip happens with the `emailVerification` block landing at the same time). The two lessons are designed as a deliberate two-step so the student feels the posture shift, not as a pacing accident.

## Concept 5 — The `verification` row as the substrate: hashed token, expiry, single-use, row deletes on consume

**Why it's hard.** The student has met the `verification` table in 9.2.2 and 9.3.5's substrate concept, but the project chapter is where they watch the row appear in Postgres on sign-up, watch the row disappear on verify-click, and have to be able to read the row's columns without checking the schema file. Three properties have to land at the keyboard: (1) the `value` column is the *hash* of the token, never the raw token — the raw token rides in the email URL, the hash sits in the table, the lookup is by the hashed value (server SHA-256s the incoming token from the URL and compares). The student who reads `value` and assumes "that's the token" ships the wrong mental model and fails to recognize why a database snapshot leak doesn't yield working tokens. (2) The `identifier` column carries the namespace (`email-verification:<email>`) — the project ships only one namespace, but the substrate is shared with reset, magic-link, 2FA-setup; recognizing the namespace pattern is what makes 9.3.4–9.3.7 read as "different namespace, same primitive." (3) The row *deletes* on consume — the verify-click flips `user.emailVerified` to `true` *and* deletes the `verification` row in the same transaction. A second click on the same link returns "invalid or expired token" because the row is gone, not because the token didn't match. Single-use is enforced by deletion, not by a `consumed` flag.

**Ideal teaching artifact.** Mechanics archetype delivered as a **four-frame token lifecycle scrub** in `DiagramSequence`, instantiated against the project's send → verify → consume flow. Frame 1: the user clicks Submit on `/sign-up` — the action calls `auth.api.signUpEmail`, the library generates 32 random bytes via CSPRNG, base64url-encodes (the *raw* token, ~256 bits), inserts a `verification` row with `identifier: 'email-verification:dan@example.com'`, `value: SHA-256(token)`, `expiresAt: now + 1 hour`. The diagram shows the row landing in a small Postgres panel; the raw token is rendered in a "ride in the email" badge, never in the table. Frame 2: the `sendVerificationEmail` callback fires, the React Email template renders with `verifyUrl` carrying the raw token, the email lands in the inbox. Frame 3: the user clicks the link — the verify endpoint reads the raw token from the URL, hashes it (`SHA-256`), queries `verification WHERE identifier = ? AND value = SHA-256(incoming) AND expiresAt > now()`. Frame 4: match — `user.emailVerified` flips to `true`, the `verification` row is deleted in the same transaction, a `session` row is inserted (the `autoSignInAfterVerification: true` payoff), the cookie attaches, the redirect lands on `/dashboard`. A small ledger underneath each frame names what changed in Postgres and what's in the browser. The closing line names the substrate rule: *raw tokens in the email, hashed tokens in the table, lookup by the hash, single-use enforced by row deletion, expiry enforced by the `WHERE` clause* — same primitive every later chapter (9.3.4 reset, 9.3.5 magic-link, 9.3.6 2FA-setup) reuses.

**Engagement.** A `Sequence` drag-to-order drill of the eight token-lifecycle steps with two distractors that map to common misconceptions: *"store the raw token in `verification.value` so the lookup is direct"* (no — database-snapshot leak yields working tokens for every pending verification; the lookup *is* by the hash) and *"flag the row as `consumed: true` instead of deleting it"* (no — single-use is enforced by deletion; the flag adds storage and a second condition to every read for no security gain). Wrong placements name the failure mode each shipping move enables. Confirm with a `MultipleChoice`: "the user clicks the verify link a second time after a successful verify — what does the server return?" The right answer names the row deletion as the cause: the second click can't find the row, the response is the same `'invalid-or-expired-token'` regardless of whether the token was wrong or already consumed (the enumeration discipline from 9.3.2 holds at the verify endpoint too).

**Components.**
- `DiagramSequence` hosting the four-frame token-lifecycle scrub with the Postgres-row + browser-cookie panels per frame. Reuses 9.3 Concept 5's substrate shape — confirms the forward-link.
- `Sequence` for the eight-step ordering drill with the two misconception distractors.
- `MultipleChoice` for the second-click confirmation.
- `Aside` (`note`) below the scrub: "raw tokens in the email, hashed tokens in the table, single-use by row deletion. Same primitive as session tokens and reset tokens — recognize the shape every time."

**Project link.** This concept lives in lesson 9.5.4 (the `emailVerification` block lands and the React Email template wires up) and lesson 9.5.6 (the verify clauses watch the row land in Postgres and disappear on consume). The substrate is the *frame* the student carries into 9.3.4 / 9.3.5 / 9.3.6 — same row, different namespace.

## Concept 6 — The verification email rides the Unit 8 pipeline: one wrapper, one new template, no re-implementation

**Why it's hard.** The student has the Unit 8 send pipeline carry-in (`lib/email.ts`'s `sendEmail({ to, subject, react, idempotencyKey })`) but the reflex when wiring `sendVerificationEmail` is to reach back into the Resend SDK directly — "the verification email is auth's concern, it should live next to `lib/auth.ts`." That reflex breaks four things at once: the suppression-list read (Unit 8's wrapper checks `email_suppressions` before calling Resend; bypassing the wrapper means a hard-bounced email re-hits Resend and burns the verified domain's reputation); the idempotency-key discipline (Resend dedups on the key; a fresh UUID per call defeats it); the verified `EMAIL_FROM` lock (per-call `from` is how multi-tenant sends accidentally land on the wrong subdomain); the typed `Result` shape (the wrapper returns `Result<{ id }>`, the action passes it through; reaching Resend directly with try/catch collapses the failure surface). The senior shape is the inverse: Better Auth's `sendVerificationEmail` callback is a thin adapter — receives `{ user, url }` from the library, calls `sendEmail` with the project's chokepoint, returns. The new code in this chapter is the React Email template (`WelcomeVerification.tsx`) and the callback signature; the send pipeline is unchanged.

**Ideal teaching artifact.** Pattern archetype delivered as a **two-layer authoring walk** in `AnnotatedCode`, instantiated at the project's keyboard with the Unit 8 wrapper as the boundary the student doesn't cross. The walk has two passes. *Pass one — the React Email template:* `WelcomeVerification.tsx` authored in seven annotations — typed props (`{ firstName: string; verifyUrl: string }`), the `<EmailLayout>` wrapper from 8.2 (brand surface lives there, the template stays content-only), the `<Preview>` line for the inbox preheader, the `<Heading>` greeting, the `<Text>` paragraph, the `<Button href={verifyUrl}>` CTA, the plain-text fallback `<Text>` ("If the button doesn't work, paste this link: {verifyUrl}"), and the `WelcomeVerification.PreviewProps = { firstName: 'Ada', verifyUrl: 'https://example.com/verify/abc-123' }` declaration as the production caller's contract. The 8.2 props-only rule holds: every dynamic value comes from the caller, the template never reads from env or session. *Pass two — the `sendVerificationEmail` callback:* the `emailVerification.sendVerificationEmail` body authored in five annotations — the destructured `{ user, url }` from the library's call; the `await sendEmail({ to: user.email, subject: 'Verify your email', react: <WelcomeVerification firstName={user.name} verifyUrl={url} />, idempotencyKey: \`verify:${user.id}:${url}\` })` invocation; the *idempotency key derived from stable inputs* (the user ID + the URL — the same retry mints the same key, Resend dedups; the URL changes per token issue, so a token re-issue fires a fresh send); the wrapper's `Result` is *not* unwrapped or re-thrown (the library callback expects a Promise; if `sendEmail` returns `err('suppressed', ...)`, the library proceeds — the user row exists, the inbox just never gets the mail; the resend button on `/verify-email` is the user's escape hatch); no try/catch around the call. The closing line names the seam: the callback is a thin adapter, the wrapper is the chokepoint, the template is the new content; nothing else changes.

**Engagement.** A `MultipleChoice` round of three scenarios: *"the team writes `new Resend(env.RESEND_API_KEY).emails.send(...)` directly inside `sendVerificationEmail` 'because the verification email is auth's concern'"* — the right answer names the four bypassed disciplines (suppression read, idempotency, `EMAIL_FROM` lock, `Result` shape) and points at `lib/email.ts` as the chokepoint Unit 8 installed exactly so this kind of "but my surface is special" reach doesn't ship; *"the team uses `crypto.randomUUID()` as the idempotency key for each send 'so retries don't collide'"* — the right answer inverts the framing (collisions are *the point* of idempotency keys; a fresh UUID defeats Resend's dedup and the user gets two emails on a Vercel cold-start retry); *"the team passes the bcrypt-hashed verification token to the email template instead of the URL"* — the right answer names the substrate confusion (the hash sits in `verification.value` for the lookup; the URL carries the *raw* token, which is what the user clicks; the template never sees the hash). Confirm with a `Buckets` sort: six call-shapes into two columns — *valid `sendVerificationEmail` body* / *footgun*. Examples: `await sendEmail({...})` (valid), `await new Resend(...).emails.send({...})` (footgun — bypasses the chokepoint), `idempotencyKey: \`verify:${user.id}:${url}\`` (valid — derived from stable inputs), `idempotencyKey: crypto.randomUUID()` (footgun — defeats dedup), `from: \`Acme <noreply@\${tenantDomain}>\`` (footgun — per-call `from`), `<WelcomeVerification firstName={user.name} verifyUrl={url} />` (valid — props-only template).

**Components.**
- `AnnotatedCode` for the two-layer authoring walk — first the seven-annotation template walk, then the five-annotation callback walk. Reuses 8.2 Concept 4's props-only rule and 8.3 Concept 7's chokepoint shape — confirms both forward-links.
- `MultipleChoice` reused three times for the PR-review round.
- `Buckets` for the six-call-shape sort.

## Concept 7 — Sign-in's enumeration discipline: opaque `'invalid-credentials'`, distinct `'email-not-verified'`

**Why it's hard.** The sign-in action's failure paths are where the chapter's enumeration discipline (9.3 Concept 2) lands at the keyboard, and the trade is subtle. *Same opaque error for "no such email" and "wrong password"* (`'invalid-credentials'`) because differentiating them hands the attacker an account-discovery oracle: if "no such email" is distinct from "wrong password", the attacker enumerates valid emails by the response shape alone, then runs credential-stuffing against the validated list. *Distinct error for "email not verified"* (`'email-not-verified'`) is fine because that information leaks only after the password matched, which is itself the proof of identity — at that point the attacker already proved credential control, the verification status is a continuation state the legitimate user needs surfaced. The student's reflex is the inverse on both counts: differentiate "no such email" for friendlier UX ("did you mean to sign up?"), collapse "email not verified" into "invalid credentials" for "consistency." The first ships the enumeration leak; the second loses the resend-link continuation that gets unverified users back into the flow. The discipline has to land as *one* rule with two halves, not as two unrelated decisions.

**Ideal teaching artifact.** Pattern archetype delivered as a **three-branch decision panel paired with the wrong-by-default action body** rendered in `AnnotatedCode` plus `CodeVariants`. *Beat one — the decision panel:* a `Figure`-wrapped hand-coded HTML three-row table with three columns — *the library response*, *the action returns*, *the form replaces itself with*. Row 1: `INVALID_EMAIL_OR_PASSWORD` (covers both "no such email" and "wrong password" at the library boundary — Better Auth's enumeration discipline is structural) → `err('invalid-credentials', 'Invalid email or password.')` → form stays, opaque message renders inline, no UI replacement. ## Concept 8 — The two-layer gate at the keyboard: cookie parse in proxy, validating read in layout

**Why it's hard.** The student has the two-layer model installed conceptually (9.4 Concept 1) but the project chapter is where the discipline gets enforced *at the keyboard*, and the temptations are operational. ## Concept 9 — `?next=` round-trip with open-redirect closure, plus the inverse gate

**Why it's hard.** Two adjacent disciplines land in the same proxy file and the student treats them as separate puzzles instead of one composed surface. (1) **The `?next=` round-trip preserves the original destination through the sign-in detour** — signed-out user types `/dashboard/invoices/inv_42`, proxy redirects to `/sign-in?next=%2Fdashboard%2Finvoices%2Finv_42`, sign-in form threads `next` into the action, action redirects to the validated `next` after success. The trap is reflecting `next` raw into `redirect()` — a hostile `?next=//evil.com` becomes a protocol-relative URL the browser resolves to `https://evil.com`, the user signs in and lands on the attacker's site (open redirect, common phishing primitive). The `sanitizeNext()` helper closes it: paths starting with `/`, no `//`, no `:`, fall back to `/dashboard` on rejection. (2) **The inverse gate** — auth pages refuse signed-in users — uses the same matcher logic in the opposite direction: a signed-in user navigating to `/sign-in` is bounced to `/dashboard` instead of seeing a stale form (sometimes invalidating the current session by re-signing-in with the same credentials). The two pieces share the same proxy file, the same matcher list, the same cookie read; treating them as one composed surface is the discipline.

## Concept 10 — Sign-out as a Server Action, the session row's absence as the verification

**Why it's hard.** Three reflexes converge on the wrong sign-out shape. (1) **`onClick={() => fetch('/api/auth/sign-out')}`** — the JS-only path, breaks progressive enhancement, fails when JS is disabled or the bundle hasn't loaded yet, the redirect doesn't compose cleanly with the auth flow. (2) **Reaching for `signOut` from `authClient`** — the client-side library's signout is for SPAs that want to handle the redirect in JS-land; the project's pattern is server-driven, the cookie clears on the server response, the redirect lands cleanly. (3) **Trusting the cookie clear and skipping the row delete** — the student who reads "sign-out clears the cookie" assumes that's the whole story; the senior reflex is the inverse: the *row's deletion* is the revocation, the cookie clear is just the client-side cleanup. A stale cookie pointing at a deleted session reads as 401 on the next request; a deleted cookie pointing at a still-extant session is the bug — the user thinks they signed out but the session row is alive on the server, an attacker who somehow re-acquires the cookie still has access. The opaque-server-session model from 9.1.2 lands here as the operational property: revocation is the row delete, *the row's absence is the verification*. The sign-out Server Action is a four-line file: `'use server';` then `await auth.api.signOut({ headers: await headers() });` then `redirect('/sign-in');` — the library does both halves (cookie clear + row delete) in one call.

**Ideal teaching artifact.** Pattern archetype delivered as a **two-frame contrast scrub** in `DiagramSequence`, instantiated at the project's keyboard. Frame 1 — *the wrong shape*: `<button onClick={() => fetch('/api/auth/sign-out')}>` with the JS-disabled annotation showing "click does nothing"; the cookie maybe clears (depending on the fetch response handling), the session row's disposition is opaque from the client. Frame 2 — *the senior shape*: `<form action={signOutAction}><button>Sign out</button></form>` paired with the four-line `signOutAction` file, with a Postgres-row panel beside it showing the `session` row before the click and the row's *absence* after the click. The library's `auth.api.signOut({ headers })` reads the session token from the cookie, deletes the matching `session` row in Postgres, sets a `Set-Cookie` header that clears the cookie (zero `Max-Age`), the response redirects to `/sign-in`, the next request from the same browser reads the cleared cookie, the proxy redirects to `/sign-in` again. The closing rule lands as one line: *sign-out is a row delete; the row's absence is the verification*. A small annotated panel below shows the canonical `signOutAction` file and the canonical `<form action={signOutAction}>` markup — both as code, both as the JS-disabled-friendly shape that just works.

**Engagement.** A `MultipleChoice` round of three scenarios: (a) "the team ships the sign-out as `<button onClick={() => authClient.signOut()}>` and considers it done — works in the browser, JS is everywhere, what's the smell?" — the right answer names the PE regression (no JS = no sign-out), the redundant client-side library reach (the server action shape is one line shorter and works without JS), and the senior reflex to use server-driven for auth surfaces; (b) "a teammate writes `auth.api.signOut()` without passing `{ headers: await headers() }` — what fails?" — the call has no way to read the cookie, can't identify which session to delete, returns success against zero rows; the user's session row stays alive, the cookie clears, the user thinks they signed out but a stolen cookie is still valid against the live row; (c) "the team ships the sign-out without verifying the row deletion in Postgres — what's the verify clause?" — open Studio (or `pnpm db:studio`), filter `session WHERE userId = ?`, sign out, refresh — the row count drops by one.
