# Chapter 7.4 — When the platform isn't enough: React Hook Form: pedagogical approach

## Concept 1 — The four triggers: RHF is a conditional reach, not a default

**Why it's hard.** The student just finished 7.3 with a working native form pattern; the moment they read "React Hook Form" they install it on every form they write next. The senior posture inverts the impulse — RHF is what the student reaches for when one of *four* specific UX shapes breaks the platform default — and unless the threshold lands as a *closed set with named triggers*, every form in the codebase grows an unnecessary client-form-state library and the chapter's core decision is lost.

**Ideal teaching artifact.** Decision archetype delivered as a **four-trigger gate** rendered as a `Figure`-wrapped HTML decision matrix. Four rows, one per trigger: *per-field validation timing past submit* (blur-to-validate, change-as-you-type strength meter), *dynamic field arrays* (variable line items, add/remove rows), *multi-step wizards across components* (onboarding flows, configurators), *controlled UI library inputs* (Combobox, DatePicker, RichText, Select). Three columns per row: *the failure mode of the native pattern* (per-keystroke handlers and ref reads, manual array bookkeeping, state-prop-drilling across steps, `FormData` doesn't see Radix-controlled values), *the RHF feature that fixes it* (`mode: 'onBlur'` + resolver, `useFieldArray`, `FormProvider` + `useFormContext`, `Controller`), *what survives unchanged* (the Server Action, the Zod schema, the shadcn `<Form>` layout). At the bottom, the threshold restated as one quotable line — *reach for RHF when validation timing, dynamic shape, multi-step state, or controlled inputs are the trigger; otherwise the 7.3 native pattern.* A small forward-link callout names Conform and TanStack Form as the two alternative reaches with the axis each optimizes for (PE-with-actions, smallest bundle + TS inference) so the student leaves recognizing the landscape without surveying it.

**Engagement.** A `Buckets` sort: ten SaaS forms ("a customer create form with five text inputs", "an invoice with a variable list of line items", "a five-step onboarding wizard with back-edit", "a settings form with a Radix-built tag combobox", "a comment box with a submit button", "a password form with a live strength meter", "an admin search form with two filters", "a permissions matrix with row-level controls", "a checkout form with country-dependent state dropdown but only validate on submit", "a contact form with three required fields") into "native 7.3 pattern" vs "reach for RHF — name the trigger." Wrong-answer feedback names which trigger fires (or doesn't) for each.

**Components.**
- `Figure` wrapping a hand-coded HTML four-trigger decision matrix.
- `Buckets` for the ten-form trigger sort, with the wrong-answer feedback naming the specific trigger.
- `Aside` (`tip`) below: "RHF is conditional. Four triggers — past-submit timing, dynamic arrays, multi-step state, controlled inputs. Otherwise 7.3."

**Project link.** The 7.6 project is deliberately built on the native pattern to anchor the senior default. This concept is the reason the project doesn't reach for RHF; the chapter's later concepts only earn their weight on forms that genuinely fire one of the four triggers (the project does not).

## Concept 2 — The cost: controlled inputs, JS-required submit, PE as a named casualty

**Why it's hard.** The four triggers from Concept 1 only earn the reach if the *cost* is honest. The student who reads "RHF is the reach" without internalizing what flips when adopting it ships forms that look like 7.3 but quietly broke three of the chapter's prior disciplines: inputs are now controlled (DOM no longer owns the live value), the submit is JS-required (the action prop's no-JS POST stops working), and `useFormStatus` is replaced by `formState.isSubmitting` because RHF intercepts the submit. Each switch is correct *under the trigger*; ignoring them produces forms that ship with a regressed PE story the student didn't notice.

**Ideal teaching artifact.** Concept archetype delivered as a **side-by-side property diff** in `TabbedContent` with two panes: *7.3 native pattern* vs *RHF pattern*. Each pane lists the same six properties with the value each pattern ships: (1) **input control model** — uncontrolled (`defaultValue`, DOM-owned) vs controlled (`register` wires `value`/`onChange`); (2) **submit primitive** — `<form action={action}>` vs `<form onSubmit={form.handleSubmit(onSubmit)}>`; (3) **pending state** — `useFormStatus().pending` (descendant) or `useActionState().isPending` (owner) vs `form.formState.isSubmitting`; (4) **client validation** — Constraint Validation API + `:user-invalid` styling vs `mode: 'onBlur'` + resolver-driven errors; (5) **no-JS submit** — works (native POST to the action URL) vs broken (RHF intercepts; without JS the form never submits); (6) **what stays the same** — the Server Action, the Zod schema, the shadcn `<Form>` layout primitives, the `Result` shape, `revalidatePath`. Below the panes, a one-line rule the student can quote: *RHF replaces the client form-state layer; the server seam is unchanged. PE degrades on the affected forms — accepted under the trigger, named in code review, never silent.*

**Engagement.** A `MultipleChoice` PR-review round of four scenarios: "a teammate ports a customer-create form to RHF for consistency with the wizard the team just shipped" (revert — none of the four triggers fired, PE lost for nothing); "a teammate's RHF form keeps the action prop on the `<form>`" (the action prop and `handleSubmit` compete — RHF must own the submit); "a teammate's RHF-managed wizard ships without naming PE in the PR description" (the senior reviewer asks for the PE casualty to be acknowledged on the wizard ticket, not just accepted silently); "a teammate's RHF form uses `useFormStatus` for the submit button's pending state" (the hook reads from the action-prop's context which RHF didn't establish — switch to `formState.isSubmitting`). Each wrong answer names a real cost the swap exposes.

**Components.**
- `TabbedContent` with two panes for the 7.3-vs-RHF property diff (each pane is a small static composition listing the six properties).
- `MultipleChoice` for the four-scenario PR-review round.
- `Aside` (`caution`) below: "RHF flips the inputs to controlled, the submit to JS-required, and the pending hook from `useFormStatus` to `formState.isSubmitting`. PE is a casualty on the affected forms — accept it under the trigger, never silently."

**Project link.** Because 7.6 stays on the native pattern, the cost diff is the precondition the student would name when *not* reaching for RHF; the same diff is the checklist any future RHF migration in the project's codebase would walk through.

## Concept 3 — `useForm` and the `defaultValues` discipline: the form's identity

**Why it's hard.** RHF's `useForm` looks like a hook call with optional config, but two of its options are load-bearing structurally and the student misses both. (1) **`defaultValues` declares the field set RHF tracks**, not just the initial values — a missing default for a field makes the input render as uncontrolled on first paint, switch to controlled the first time the user types, and emit a React warning. The trap is silent until the warning fires in production. (2) **`mode` is the validation-timing trigger from Concept 1 made concrete** — the default `'onSubmit'` defeats most of the reason the student reached for RHF in the first place; `'onBlur'` is the senior reach for "validate-on-blur, fix-as-you-type" UX, paired with `reValidateMode: 'onChange'`. Without both calls landing at the keyboard, the `useForm` line reads as "boilerplate," not as the form's identity declaration.

**Ideal teaching artifact.** Mechanics archetype delivered as a **wrong-by-default sandbox** in `ReactCoding`. The starter ships a six-field create form with `useForm()` called with *no* arguments and `mode` left at the default. Three observable bugs are wired into the preview pane: (1) typing into the `email` field emits a "changing an uncontrolled input to be controlled" warning in the dev console; (2) the form validates only on submit — the user can type a malformed email, blur, and see no error until they hit Save; (3) the resolver is wired but the field-set typing surfaces `unknown` because `defaultValues` is missing. The student's job is two repairs: (a) declare `defaultValues: { email: '', name: '', ... }` for every field — the warning vanishes; (b) set `mode: 'onBlur'` plus `reValidateMode: 'onChange'` — the email error now appears on blur and clears as the user fixes it. The fixtures pinned alongside show the before/after console state and the validation timing. The repair *is* the assessment because the bugs are observable.

**Engagement.** The repair carries the assessment. Confirm with a `Dropdowns` exercise on a one-block `useForm` skeleton with three blanks: the resolver position (`zodResolver(InvoiceSchema)`), the `mode` value for the canonical "blur to validate, fix as you type" UX (`'onBlur'`), and the `defaultValues` shape that drives the field set (`{ email: '', name: '', ... }` — not `undefined`, not omitted). Each blank's wrong picks surface the failure mode each one would produce.

**Components.**
- `ReactCoding` for the wrong-by-default sandbox with the missing `defaultValues` warning, the wrong `mode` validation timing, and the typing gap as the three observable bugs.
- `Dropdowns` for the three-blank `useForm` skeleton.
- `Aside` (`caution`) below: "`defaultValues` declares the field set, not just initial values; missing defaults produce silent uncontrolled-to-controlled warnings. `mode: 'onBlur'` is the senior reach for forms past submit-only validation."

**Project link.** Concept lands as the precondition for any future RHF form in the project's codebase. When 7.6's create-invoice form is hypothetically migrated to RHF (the student practices the migration in the chapter's examples), `defaultValues` and `mode: 'onBlur'` are the two non-negotiable opens.

## Concept 4 — `register` vs `Controller`: pick by the input's control model

**Why it's hard.** The student arrives with a binary mental model — "RHF wires inputs" — and reaches for `register` everywhere because the docs show it first. Half the chapter's forms then fail silently: a Radix-built `Combobox` ignores the `ref` and `onChange` `register` spreads onto it, the form's state never sees the user's selection, the resolver passes (the field is `undefined` and the schema's check missed it), the action receives an empty value. The senior reach is structural: *`register` for native inputs the DOM owns, `Controller` (or `useController`) for any component whose value lives in `value`/`onChange` props*. The decision is the input's control model, not RHF's API surface.

**Ideal teaching artifact.** Decision archetype delivered as a **decision-by-component-shape catalog** rendered as a `Figure`-wrapped HTML table, paired with a **same-form-two-paths contest** in `CodeVariants`. The table has three columns: *the input's control model* (native `<input>`/`<textarea>`/`<select>`/checkbox/radio vs. controlled component with `value`/`onChange` props), *the RHF reach* (`register('name')` spread onto the input vs. `<Controller name="..." control={form.control} render={({ field }) => ...}>`), *the canonical examples* (text/email/password/number/textarea/checkbox/radio vs. shadcn `Combobox`/Radix `Select`/date picker/rich text editor/file uploader with preview). The contest in `CodeVariants` shows one form with a text input and a Combobox written two ways: tab one wires *both* via `register` — the text input works, the Combobox is silently dead (the schema thinks the role is `undefined`, the action would write `null`); tab two wires the text input via `register` and the Combobox via `Controller` with the `field.value`/`field.onChange` bridge — both fields land in `formState`, the resolver validates both, the submit ships the typed object. A side beat names `useController` as the *hook-form* of `Controller` for reusable field components that own their UI (a project-internal `<RoleSelect form={form} />` calls `useController` internally instead of forcing the consumer to write the render prop).

**Engagement.** A `Buckets` sort: eight inputs ("text input for email", "shadcn Combobox for customer", "checkbox for archived", "Radix Select for status", "textarea for notes", "shadcn DatePicker for dueAt", "number input for amount", "shadcn rich-text editor for description") into "`register`" vs "`Controller`/`useController`." Wrong-answer feedback names the input's control model that decided the reach.

**Components.**
- `Figure` wrapping a hand-coded HTML decision-by-control-model table.
- `CodeVariants` with two tabs for the same-form-two-paths contest, with the silently-dead Combobox as the load-bearing reveal.
- `Buckets` for the eight-input sort.
- `Aside` (`tip`) below: "the input's control model decides the reach. Native DOM inputs → `register`; controlled-prop components → `Controller` or `useController`."

**Project link.** Concept maps onto any project chapter that ships a Combobox or DatePicker on a form past the native pattern (Chapter 11.2's filter sidebar, Chapter 17's settings form). The 7.6 project doesn't reach for either, so the concept's cash-in is the forward-link.

## Concept 5 — The canonical RHF form shape: `handleSubmit` as the submit interceptor

**Why it's hard.** The student arrives carrying the 7.3 spine — `<form action={formAction}>` plus `useActionState` — and the swap to `<form onSubmit={form.handleSubmit(onSubmit)}>` is *not* a one-line rewrite. The action prop is gone (RHF must intercept the submit to run client validation), the action becomes a *callback* invoked from inside `onSubmit` after validation passes, and the `useActionState` wrapper falls away. The `Result` round-trip from 7.2 still happens, but on the imperative call site inside `onSubmit`. Without a *new* canonical shape that names every region — the hook call, the `Form` wrapper that consumes the form instance, the `<form onSubmit>`, the inline `<FormField>`s, the action call inside `onSubmit`, the `Result` reader — the student stitches together a working form whose layout drifts from every other RHF form in the codebase.

**Ideal teaching artifact.** Pattern archetype delivered as a **stencil-and-fill canonical shape** in `AnnotatedCode`, with explicit cross-references to the 7.3 spine it replaces. The student sees one canonical form component — `<CreateInvoiceForm>` — laid out in six labeled regions: (1) **the hook call**: `const form = useForm<z.input<typeof InvoiceSchema>>({ resolver: zodResolver(InvoiceSchema), defaultValues: {...}, mode: 'onBlur' })` (the Concept 3 + Concept 7 + Concept 8 payload); (2) **the shadcn `<Form>` wrapper**: `<Form {...form}>` provides the form context the `<FormField>` primitives consume (the same shadcn primitives 7.3 used, now consuming the RHF instance); (3) **the `<form onSubmit>`**: `<form onSubmit={form.handleSubmit(onSubmit)} noValidate>` — the `noValidate` is intentional, RHF owns validation now; (4) **the `<FormField>`s**: each field rendered with `<FormField control={form.control} name="email" render={({ field }) => <FormItem><FormLabel>...</FormLabel><FormControl><Input {...field} /></FormControl><FormMessage /></FormItem>} />` — the `<FormMessage />` automatically reads `formState.errors[name]`; (5) **the imperative action call** inside `onSubmit`: `const result = await createInvoice(values); if (!result.ok) applyServerErrors(form, result);` (the 7.2 Server Action is unchanged; the call site moved from the form's `action` prop to the `onSubmit` body); (6) **the submit button**: `<SubmitButton disabled={form.formState.isSubmitting}>Save</SubmitButton>` (the same shadcn `Button` primitive, the pending fact now reads from `formState`, not `useFormStatus`). The annotation track walks each region in order, naming the principle behind each — the resolver as the validation source (Concept 7), the `<Form>` context bridge, the `noValidate` rationale, the `<FormField>` slot pattern, the imperative action call, the `applyServerErrors` helper (forward-linked to Concept 9). The closing line names the principle: *every RHF form in the codebase follows this shape; if a form deviates, a new component has emerged or a rule has bent.*

**Engagement.** A `Sequence` drag-to-order drill on a shuffled form-component skeleton: the student orders seven fragments — the `useForm` call, the `<Form>` wrapper, the `<form onSubmit>`, the `<FormField>` block, the imperative `await createInvoice(values)`, the `if (!result.ok) applyServerErrors(...)`, and the `<SubmitButton disabled={isSubmitting}>` — plus a distractor: a `<form action={createInvoice}>` left over from the 7.3 spine. The distractor has no correct slot — feedback names "the action prop competes with `handleSubmit`; RHF must own the submit." Confirm with a `MultipleChoice`: "a teammate's RHF form keeps `useActionState` and writes `<form action={formAction}>` — what does the senior reviewer ask for?" — the right answer names *remove the wrapper, call the action imperatively from `onSubmit`*; wrong answers name symptom mitigations.

**Components.**
- `AnnotatedCode` walking the six-region canonical RHF form-component shape, with explicit cross-references to the 7.3 spine each region replaces.
- `Sequence` for the seven-fragment drag-to-order drill with the leftover-`action`-prop distractor.
- `MultipleChoice` for the `useActionState`-doesn't-belong-here PR-review.
- `Aside` (`note`) below: "one canonical RHF form shape. `handleSubmit` owns the submit; the action is called imperatively inside `onSubmit`; the shadcn `<Form>` consumes the RHF instance via `<Form {...form}>`."

**Project link.** Forward-links to any future migration of a 7.6 form to RHF (a hypothetical line-item editor on the invoice). The shape installed here is the spine the chapter's later concepts (`useFieldArray`, wizards) extend without reinventing.

## Concept 6 — `formState` and the per-field re-render isolation: `useWatch` for derived values

**Why it's hard.** RHF's value proposition past the resolver is *performance via subscription*: a registered input doesn't re-render the form root on every keystroke. The student who hasn't internalized this writes `const { values } = form` (or `form.getValues()` inside the form root) to derive a character count or a conditional-render flag, and accidentally reads from a value that doesn't trigger re-renders — the count is stale. The naive fix is `watch('field')` *in the form root*, which works but re-renders the entire form on every keystroke (the resolver runs again, every field re-renders, the performance win is lost). The senior reach is structural: `useWatch({ control, name })` *in a small child component* that owns the derived render — the parent stays stable, only the child re-renders. Symmetric trap on `formState`: destructuring `errors` from `formState` triggers a re-render whenever *any* `formState` property changes, not just `errors`; reach for the property explicitly.

**Ideal teaching artifact.** Mechanics archetype delivered as a **render-counter visualization sandbox** in `ReactCoding`. The starter ships a five-field form with three render-counter badges visible in the preview pane: one on the form root, one on a derived child component (`<CharacterCount field="notes" />`), one on the submit button (`<SubmitButton>` reading `isSubmitting`). Three states the student walks through: (1) **the wrong reach** — `<CharacterCount>` reads `form.watch('notes')` from the form root passed via prop; the badge shows the form root re-renders on every keystroke (count climbs visibly); (2) **the right reach** — `<CharacterCount>` calls `useWatch({ control, name: 'notes' })` internally; only the child's badge ticks, the form root stays stable; (3) **the `formState` trap** — the form root destructures `const { errors, isDirty, isSubmitting } = form.formState`; the badge shows the form root re-renders when *any* `formState` field changes, including `isSubmitting` flipping at submit time. The student modifies the destructure to `form.formState.errors` (read inline) and observes the form root stays stable on submit. The closing rule lands on the badge: *`useWatch` for derived state in a small child; explicit property access on `formState` to avoid over-subscription.*

**Engagement.** The render-counter observation carries the assessment. Confirm with a `MultipleChoice` after the sandbox: "a teammate's RHF form re-renders on every keystroke even though `register` is wired correctly. What's the most likely cause?" — the right answer names `watch(...)` called in the form root or `formState` destructured wholesale; wrong answers name unrelated mitigations ("memoize the `<FormField>` blocks", "switch from `mode: 'onChange'` to `'onBlur'`").

**Components.**
- `ReactCoding` for the render-counter sandbox with three badges and three observable states (wrong `watch`, right `useWatch`, `formState` over-subscription).
- `MultipleChoice` for the cause-of-re-render confirmation.
- `Aside` (`note`) below: "RHF's win is per-field re-render isolation. Derived values use `useWatch` *inside the child that renders them*; `formState` is read by explicit property to avoid over-subscription."

**Project link.** Concept compounds in any future form-heavy chapter (Chapter 11.3's inline editors, Chapter 17.2's settings panels) where the form's parent surfaces would re-render expensively if the per-field isolation is lost. The 7.6 project doesn't ship RHF, so the cash-in is the forward-link.

## Concept 7 — `zodResolver`: one schema, both sides of the wire

**Why it's hard.** The student finished 7.1 (the schema) and 7.2 (the action's `safeParse` on entry) and now needs the form to validate against the *same* schema before the action runs. The lazy reach is to declare a parallel client-side schema — or worse, hand-write per-field rules in `register({ required: true, minLength: 8 })` — both of which produce drift the day the schema gains a field. The drift is silent: the resolver accepts a payload the action rejects, the user sees the resolver's "form looks valid" UX, then the action's failure surfaces as a banner that contradicts the inline state. The senior reach is structural — *one Zod schema, imported into both the action and the form's resolver, change-once propagates everywhere* — and the discipline is non-negotiable.

**Ideal teaching artifact.** Pattern archetype delivered as a **single-source-of-truth chain diagram** rendered as a `Figure`-wrapped hand SVG, paired with a **before-and-after refactor contest** in `CodeVariants`. The chain diagram (the same visual vocabulary as 7.1's source-of-truth chain): one box at the root labeled `/lib/schemas/invoice.ts` exporting `InvoiceSchema`; two arrows out — one to the Server Action (`Schema.safeParse(Object.fromEntries(formData))` in `/app/.../actions.ts`), one to the form's resolver (`useForm({ resolver: zodResolver(InvoiceSchema) })` in the Client Component). A second loop closes back: the schema's error messages flow into both `treeifyError` (for the action's `Result.fieldErrors`) and `formState.errors` (for the resolver's inline errors) — *one set of strings, two render sites*. The contest: tab one (the drift trap) shows a form with hand-written `register('email', { required: 'Email required', pattern: { value: /.+@.+/, message: 'Invalid email' } })` plus the action's separate `z.email()` schema — the messages diverge, the rules diverge; tab two shows the resolver wiring with the imported schema feeding both sides — the messages and rules are identical because they originate from one declaration. Below the contest, the rule: *any form past the native pattern imports the same schema the action parses with; per-field `register` rules are a code-review smell.*

**Engagement.** A `MultipleChoice` PR-review round of three scenarios: "a teammate's PR adds a new field to the schema and updates the form's `register` rules to match" (revert — the form should consume the schema via the resolver, not mirror it); "a teammate's PR adds the resolver but keeps `required: true` on the `register` calls 'as a safety net'" (the safety net is duplicate validation that drifts — drop the `register` rules, the schema is the contract); "a teammate's PR uses `zodResolver` but with a *different* schema than the action's" (the whole point is one schema — make them the same import). Each wrong answer names a real drift class.

**Components.**
- `Figure` wrapping a hand SVG of the single-source-of-truth chain (one schema box, two arrows to action and resolver, the loop-back for error messages).
- `CodeVariants` with two tabs for the drift-vs-resolver contest.
- `MultipleChoice` for the three-scenario PR-review.
- `Aside` (`caution`) below: "one schema, imported into both the action and the form's resolver. Per-field `register` rules duplicate the schema — drop them."

**Project link.** Reuses the source-of-truth chain visual vocabulary from 7.1's Concept 14 (`drizzle-zod` capstone). When the chapter's examples reach for RHF, the form imports the same `drizzle-zod`-derived schema the project's actions parse with; the chain has one more node feeding off the root.

## Concept 8 — `z.input` vs `z.output`: the type bridge transforms create

**Why it's hard.** For schemas without transforms, `z.infer<typeof Schema>` is the only helper the student reached for in Chapter 7.1. The moment a schema gains `z.coerce.number()` or `z.iso.datetime().transform(s => new Date(s))`, the parse acquires *two* types — the input it accepts (string from the form) and the output it returns (number, `Date`). RHF's `defaultValues` and the field-registration types want the *input* type (what the form holds in its registered fields); the `onSubmit(values)` callback wants the *output* type (what the action receives after coercion). The student who types `useForm<z.infer<typeof Schema>>()` works fine until the first transform — then `defaultValues: { quantity: 0 }` fails type-checking against the input type `{ quantity: string }`, and the student "fixes" it by widening to `string | number`, losing the type discipline. The teaching has to make the split visible at the form's site.

**Ideal teaching artifact.** Concept archetype delivered as a **type-pane probe** in `ZodCoding`, recalling and extending the 7.1 Concept 9 visualization but with RHF's two type slots pinned alongside. The student sees one schema — `InvoiceSchema = z.object({ quantity: z.coerce.number().positive(), issuedAt: z.iso.datetime().transform(s => new Date(s)), notes: z.string() })` — with three Twoslash queries pinned underneath. Query one: `z.input<typeof InvoiceSchema>` resolves to `{ quantity: unknown; issuedAt: string; notes: string }` (what the form holds — strings before coercion). Query two: `z.output<typeof InvoiceSchema>` resolves to `{ quantity: number; issuedAt: Date; notes: string }` (what the action's body receives). Query three: a `useForm` skeleton — `useForm<z.input<typeof InvoiceSchema>, unknown, z.output<typeof InvoiceSchema>>({ ... })` — the new generic accepts the input type for `register`/`defaultValues` *and* the output type for the `onSubmit` callback's `values` parameter. The student adjusts the schema (adds a non-transform field, swaps a coerce out) and watches all three queries update in lockstep. The closing rule: *`z.input` for the form's field types and `defaultValues`; `z.output` for the `onSubmit` parameter; `z.infer` is wrong the moment a transform is present.*

**Engagement.** A `MultipleChoice` round of three pairings: "the type to pass as the first generic to `useForm`" → `z.input`; "the type of the `values` parameter inside `onSubmit`" → `z.output` (or `z.infer` only if no transforms); "the type of the value RHF holds for an `<input type="number">` registered as `z.coerce.number()`" → `string` (the input type — RHF tracks what the DOM provides, the resolver runs the coercion). Each wrong answer tags the typing error or the runtime mismatch the wrong helper produces.

**Components.**
- `ZodCoding` for the three-pinned-`^?`-query probe with the `useForm` skeleton's two generics visible.
- `MultipleChoice` for the three-pairing round.
- `Aside` (`note`) below: "`z.input` for the form's holdings; `z.output` for the action's parameter. `z.infer === z.output` — fine without transforms, wrong the moment one is added."

**Project link.** Cross-references 7.1 Concept 9 directly. The 7.6 project's schemas (created via `drizzle-zod` with coercions for `FormData`) would diverge `input` and `output` the moment any RHF migration happens; this concept is the precondition for that migration's typing.

## Concept 9 — Server-pushed errors → `setError`: closing the round-trip without overwriting state

**Why it's hard.** The resolver runs client-side only and validates *shape*. Cross-resource rules — invoice number unique within the org, customer belongs to the same org, plan limit, suppression list — can't validate without the server's truth, so the action returns them as `Result.error.fieldErrors` (the 7.2 Concept 8 mapping). The form has to *land those errors back on the right fields* without overwriting field state, and the student reaches for the wrong tool: `setValue('field', value, { shouldValidate: true })` re-runs the resolver and overwrites the error with the resolver's verdict (the field looks valid client-side); a manual `useState` for "server errors" duplicates `formState` and competes with `<FormMessage />`'s renderer; a top-level banner loses the per-field anchoring. The senior reach is `form.setError(name, { type: 'server', message })` — RHF's bridge for "this error came from outside the resolver, render it under the field, don't re-validate." The discipline is one helper (`applyServerErrors`) called once after the action returns; the rest is mechanical.

**Ideal teaching artifact.** Mechanics archetype delivered as a **controllable-failure simulator** in `ReactCoding` paired with the **`applyServerErrors` helper** in `AnnotatedCode`. The simulator: a small create-customer form wired to a stub action with a `?fail=` query toggle in the preview pane: `?fail=conflict` returns `{ ok: false, error: { fieldErrors: { email: ['That email is already taken'] } } }`; `?fail=validation` returns a shape-error the resolver should have caught (asserting the round-trip path even when client validation passed against a stale schema); `?fail=plan` returns a banner-level error with no field. The student observes the form *without* `applyServerErrors` first: the action returns, the form shows nothing — the user is confused. The student then wires `applyServerErrors(form, result)` from `/lib/forms/server-errors.ts`. The helper's body, walked in the paired `AnnotatedCode`: iterate `result.error.fieldErrors`, call `form.setError(name, { type: 'server', message: messages[0] })` for each; if the result has a top-level `userMessage` with no `fieldErrors`, surface it via a separate banner state (or `form.setError('root.serverError', { message })` for a form-level slot). After the wire, the same `?fail=conflict` produces an inline error under the email field (rendered by `<FormMessage />` because it reads from `formState.errors`), the same shape errors the resolver would render. The rule lands at the keyboard: *server-pushed errors are `setError`, not `setValue`; the `<FormMessage />` renders both client and server errors uniformly.*

**Engagement.** The simulator's three failure modes carry the assessment because the student must observe each behavior. Confirm with a `MultipleChoice` after the wire: "a teammate's RHF form swallows server-returned uniqueness errors — the user sees no message, the form looks valid. What's the bug?" — the right answer names *the form never called `setError` on the action's `fieldErrors`*; wrong answers name symptom mitigations ("add a `useEffect` to copy `result.error` into a local state", "move the conflict check into the schema's `.refine`").

**Components.**
- `ReactCoding` for the controllable-failure simulator with `?fail=conflict|validation|plan` toggles in the preview pane.
- `AnnotatedCode` walking the `applyServerErrors(form, result)` helper with one annotation per branch (field errors → `setError`, top-level → banner or `root.serverError` slot).
- `MultipleChoice` for the post-wire confirmation.
- `Aside` (`tip`) below: "server-pushed errors are `setError`, not `setValue`. One helper (`applyServerErrors`) in `/lib/forms/`, called once after the action returns."

**Project link.** The `applyServerErrors` helper is the canonical reach for any form in the project (RHF or otherwise) that surfaces action-returned `fieldErrors` per field. The pattern installed here forward-links into 11.3's inline editors and 17.2's settings forms, where conflict errors and plan-limit rejections need the same per-field anchoring.

## Concept 10 — `useFieldArray`: dynamic lists with stable identity

**Why it's hard.** The student reaches for the chapter's most kinetic feature without first internalizing the two non-obvious mechanics: (1) **`field.id` is RHF's bookkeeping**, separate from the entity's domain ID — it's stable across reorders, used for React's `key` prop and *nothing else*; the student who keys by index breaks input focus and dirty state on every reorder; the student who keys by `field.id` *and* persists it as the database ID ships data with random RHF-internal IDs that don't survive a page reload. (2) **The schema shape pairs with the hook**: `z.array(z.object(...)).min(1, '...')` produces error trees keyed by `formState.errors.lineItems?.[index]?.fieldName` — the access path is the only non-obvious part of the integration, and the student who tries `errors.lineItems.email` (without the index) reads `undefined` and renders nothing. The teaching has to land both as kinetic moments — feel the reorder break with the wrong key, feel the per-row error path with the right access — not as paragraphs.

**Ideal teaching artifact.** This concept needs two paired beats — the hook's mechanics are one model, the field.id-vs-domain-ID discipline is another, and conflating them loses the second beat.

**Beat one — the invoice-line-items sandbox** in `ReactCoding`. The starter ships a partial invoice form with `useFieldArray({ control, name: 'lineItems' })` already wired and three rows pre-populated; the student fills in the canonical render — the `fields.map((field, index) => <li key={field.id}>...<FormField name={`lineItems.${index}.description`} />...<Button onClick={() => remove(index)}>Remove</Button></li>)` plus the `<Button onClick={() => append({ description: '', amount: 0 })}>Add line</Button>` — and observes the live add/remove/reorder behavior. A toggle in the preview pane swaps `key={field.id}` for `key={index}` and the student sees the consequence: typing into row 2, then removing row 1, the focused input loses focus and the value visibly jumps; with `key={field.id}` the focus stays put. Validation is wired so the per-row error access path is observable — emptying a row's `description` shows the error rendered by the `<FormMessage />` keyed at `lineItems.1.description`. The repair *is* the assessment.

**Beat two — the field.id-vs-domain-ID contest** in `CodeVariants`. The student sees the same edit-an-existing-invoice form rendered two ways. Tab one (the bug): the form stores `field.id` (RHF's bookkeeping ID) into a hidden input as the row's identifier; the action receives a payload with random per-render IDs that don't match anything in the database, the diff logic explodes. Tab two (the senior shape): the schema's array element is `z.object({ id: z.uuid().optional(), description: z.string().min(1), amount: z.coerce.number().positive() })`; `defaultValues` includes the existing rows' domain IDs (from the database fetch), `append({ description: '', amount: 0 })` creates rows with no `id` (the action treats `id`-less rows as inserts), `remove(index)` is local until submit (the action computes the diff on submit and runs the inserts/updates/deletes in a transaction — Chapter 6.4.4 wrapping). Below both tabs, the rule: *`field.id` is React's key only; the entity's `id` is a separate optional field on the schema that distinguishes new rows from edits.*

**Engagement.** Beat one's reorder-with-wrong-key observation carries the assessment. Confirm with a `MultipleChoice` after both beats: "a teammate's `useFieldArray`-driven invoice form ships rows whose IDs change on every page reload. What's the bug?" — the right answer names *the form persisted `field.id` (RHF's bookkeeping) as the entity's ID — the schema's `id` field is separate and absent for new rows, present for existing*; wrong answers name unrelated mitigations.

**Components.**
- `ReactCoding` for beat one's invoice-line-items sandbox with the `key={index}`-vs-`key={field.id}` toggle and the per-row error access path observable.
- `CodeVariants` with two tabs for beat two's `field.id`-vs-domain-ID contest.
- `MultipleChoice` for the post-beat confirmation.
- `Aside` (`caution`) below beat two: "`field.id` is React's key — RHF-internal, not stable across reloads. The entity's domain ID is a separate optional schema field; absent means insert, present means update."

**Project link.** Forward-link to a hypothetical line-item editor on the 7.6 invoice (the project ships single-row CRUD; the array editor lives in the chapter's example, not the project). The diff-on-submit transaction pattern reuses the 6.4.4 wrapping discipline directly.

## Concept 11 — Multi-step wizards: `FormProvider`, per-step `trigger` + `pick`, and the unmount question

**Why it's hard.** The wizard is the chapter's cumulative payoff and the load-bearing facts collapse into one beat. (1) **One `useForm` at the root, shared via `FormProvider`** — the student who calls `useForm` inside each step component creates a new form per step, the values vanish on navigation, the resolver fragments. (2) **Per-step validation reuses the same schema via `.pick()`** — advancing from step 1 should validate step 1's fields only, not the whole form; `form.trigger(['company.name', 'company.taxId'])` runs the resolver against the named subset and returns boolean. The student who calls `form.trigger()` (no args) validates the entire form and the user gets stuck on step 1 because step 5's empty fields fail. (3) **The unmount question** — `shouldUnregister: false` (the default) keeps unmounted fields' values; `true` clears them. The student who flips it to `true` thinking "clean up" loses the user's entries on every "back" press. (4) **The wizard's state is the form** — every step writes through `register`/`Controller` into the same instance; the *current step* is local component state, separate from the form. Without all four facts landing as one structural model, the student ships a wizard that breaks on the first back-edit.

**Ideal teaching artifact.** Pattern archetype delivered as a **scrubbable wizard-state-trace animation** in `DiagramSequence`, paired with a **canonical wizard skeleton** in `AnnotatedCode`. The animation has six frames the student steps through with the slider: (1) **the wizard mounts** — `useForm` fires once at the root, `<FormProvider>` wraps all five step components, the wizard's local `useState` for `step` initializes to 1; the form's state container is shown as one shared box, the steps as five tabs above it; (2) **the user fills step 1** — `register('company.name')` and `register('company.taxId')` write into the form's container, the box visibly accrues the values; (3) **"Next" on step 1** — the handler calls `await form.trigger(['company.name', 'company.taxId'])` (the field set derived from `WizardStep1Schema = FullSchema.pick({ company: true })`), the resolver runs against those two fields only, returns `true`, the wizard's local state advances to step 2; (4) **the user fills step 2, then "Back"** — `step` drops to 1, step 2's component unmounts, but the form's container *still holds* step 2's values because `shouldUnregister: false` is the default; the box shows both step 1 and step 2's values pinned; (5) **the user re-edits step 1, then "Next" + "Next"** — the wizard advances to step 2; the form's container has the new step 1 values plus the preserved step 2 values, the user sees their step 2 entries intact; (6) **"Submit" on step 5** — `form.handleSubmit(onSubmit)` validates the *whole* schema (catches cross-step rules via `.refine`), `onSubmit(values)` calls the Server Action with the typed object, the action returns `Result`, `applyServerErrors` maps any `fieldErrors` back into `formState.errors`, the wizard's UX layer navigates the user to the step that owns the offending field. Across every frame, two callouts: *one `useForm` at the root, one shared instance via `FormProvider`*; *current step is local state, field values are RHF state*. The paired `AnnotatedCode` walks the canonical skeleton with five annotations: the root `useForm` + `<FormProvider>`; the step component reading the shared instance via `useFormContext`; the per-step `trigger(StepNSchema.keyof().options)` call on "Next"; the `shouldUnregister: false` default; the final `handleSubmit` on the last step.

**Engagement.** A `Sequence` drag-to-order drill on the wizard's "Next from step N" handler — the student orders five steps (`pick the schema for the current step`, `derive the field name list`, `await form.trigger(fieldNames)`, `if (false) return` to bail on validation failure, `setStep(step + 1)` to advance) plus a distractor (`form.handleSubmit(onSubmit)` mid-wizard, which would submit instead of advance). The distractor has no correct slot and feedback names "the wizard submits *once*, on the last step — every other Next is a `trigger`." Confirm with a `MultipleChoice` round of three scenarios: "a teammate's wizard loses values when the user goes back" (set `shouldUnregister: false` — or, if already false, check that the step components don't call `useForm` themselves); "a teammate's wizard fails advancing from step 1 because step 5's empty fields fail validation" (the handler called `form.trigger()` with no args — pass the step's field set or use the schema-`pick` approach); "a teammate's wizard navigates the user to the wrong step on a server-pushed error" (the UX layer needs to map the failed field's path back to the step that owns it — name the field-to-step map as a small `/lib` constant the wizard reads).

**Components.**
- `DiagramSequence` hosting the six-frame wizard-state-trace animation; each frame is a hand SVG inside `Figure` showing the form's shared state container, the current step indicator, and the active write/read.
- `AnnotatedCode` for the canonical wizard skeleton (root `useForm` + `<FormProvider>`, per-step `useFormContext` + `trigger` + `.pick`, final `handleSubmit`).
- `Sequence` for the five-step Next-handler drill with the mid-wizard `handleSubmit` distractor.
- `MultipleChoice` for the three-scenario PR-review.
- `Aside` (`caution`) below the animation: "one `useForm` at the root, shared via `FormProvider`. Per-step validation is `trigger(fieldNames)` against `Schema.pick(...)`; the final `handleSubmit` validates the whole schema and submits *once*."

**Project link.** Wizards are out of scope for the 7.6 project (single-page CRUD), but forward-links to onboarding flows in Chapter 17 and Chapter 13's setup wizards. The discipline installed here — one `useForm`, per-step `trigger`, the field-to-step map for server errors — is the canonical pattern those chapters build on.

The chapter ships entirely on existing components. The visual vocabulary established by 7.1 (the source-of-truth chain), 7.2 (the five-seam stencil and the wire-crossing animation), and 7.3 (the canonical form-component spine, the wrong-by-default `ReactCoding` repair sandbox, the hand-SVG decision matrices) carries this chapter's full teaching load. Concepts 3, 6, 9, and 10 (beat one) all reach for the wrong-by-default `ReactCoding` sandbox shape introduced in 7.3 Concepts 7 and 9 — the kinetic-repair format proves general across the unit. Concept 11's wizard animation extends 7.3 Concept 3's submit-lifecycle `DiagramSequence` to a longer stateful flow, but the component machinery is identical.

The two closest calls for a bespoke component are Concept 6's render-counter sandbox and Concept 11's wizard-state-trace animation. The render-counter could earn a `RenderCountedSandbox` widget that ticks visible badges on every component re-render — but the same shape doesn't recur in this chapter (the per-field isolation is taught once and the threading lives in the kinetic moment), and the closest forward-link is Chapter 9's React performance unit which would build a different shape (a profiler-driven view, not three badges in a sandbox preview). Per the single-use discipline, the badges live as instrumented children inside the existing `ReactCoding` runtime — the build cost is the badge component's twenty lines, not a one-off widget. The wizard-state-trace could earn a `WizardStateTrace` widget that animates the form's container across step transitions — but the chapter teaches one wizard, the visual recurs only in Chapters 13 and 17 as a re-reference, and the six-frame `DiagramSequence` of hand SVGs already carries the kinetic moment with the build cost in the SVG content.

Two latent forward-links to flag rather than build now. The `applyServerErrors(form, result)` helper from Concept 9 is a small `/lib/forms/` primitive that recurs in any RHF form ever shipped — it's a code artifact, not a component, but the *teaching shape* (the controllable-failure simulator with `?fail=` toggles in the preview pane) recurs in Chapter 11.3 (inline editors with conflict failures), Chapter 12 (Stripe webhooks pushing errors back to forms in rare cases), and Chapter 17 (settings forms with plan-limit rejections). If the `?fail=` toggle pattern emerges as the canonical "show me the failure path" affordance across these chapters, a small `FailureToggleSandbox` wrapper around `ReactCoding` may earn its weight — but the toggle is a query-param read inside the existing `ReactCoding` preview pane, and the build cost is the action stub's branch, not a new widget.

The second forward-link is the wizard's field-to-step map (Concept 11's third PR-review scenario): when a server-pushed error needs to navigate the user to the step that owns the offending field, a small `/lib/wizards/` constant maps schema field paths to step numbers. This is a code pattern, not a component, but it recurs in every multi-step wizard in the curriculum (Chapter 13's organization setup, Chapter 17's account configuration). The pattern lands inline here; the abstraction is deferred until the third use materializes.
