# Chapter 10.1 ŌĆö Organizations as the tenancy model: pedagogical approach

## Concept 1 ŌĆö Active-org as server-side session state, not URL state, not client state

**Why it's hard.** The student arrives carrying three competing reflexes for "where the current org lives." Reflex one ŌĆö *put it in the URL* (`/o/acme/dashboard`) ŌĆö feels familiar from REST shops and from path-based routers; the failure mode is that the URL becomes a security boundary it can't enforce (a user types `/o/competitor/dashboard` into the address bar and the route handler trusts it). Reflex two ŌĆö *put it in `localStorage` or a separate cookie* ŌĆö drifts from the server's truth, breaks across devices, and turns "switch org" into a client-only event the server never sees. Reflex three ŌĆö *put it on the `user` row* ŌĆö collapses the multi-device case (the "I review my own invoices in Personal in one tab while triaging Acme in another" reflex). The senior reading is that the active org is *per-session* server state living on the `session` row, validated by membership at every read, never readable from any client-supplied surface. The student needs to feel why each of the three wrong reflexes ships a specific bug before the right placement reads as obvious.

**Ideal teaching artifact.** Concept archetype delivered as a **four-lane storage-placement comparison** rendered in `TabbedContent`, mirroring the per-flow comparison shape installed across Unit 9. Each tab is a candidate placement: *URL param*, *`localStorage` / separate cookie*, *`user.activeOrganizationId`*, *`session.activeOrganizationId` (senior baseline)*. Inside each tab, three small panels stacked: *(a)* a one-line schema sketch (what gets stored where), *(b)* a hand-SVG inside `Figure` of two browser tabs and a phone, all signed in as Ada ŌĆö annotated to show what each surface "knows" the active org is, *(c)* a *failure-or-fit ledger* ŌĆö for the three wrong tabs, the named bug (URL: cross-org access by typing; `localStorage`: device drift, lost on logout, not readable server-side; `user`: only one org at a time across all sessions); for the senior tab, the three properties that fall out structurally (per-session, server-validated, switchable without touching the client). A small closing one-liner under the tabs: *the active org is what the session says it is ŌĆö the URL is a route, the client is a view, the `user` row is the human; the `session` row is where "currently operating as" lives.*

**Engagement.** A `Buckets` sort on eight pieces of state into two columns ŌĆö *belongs on `session`* / *belongs elsewhere (and where)*. Examples: *active organization* (session), *user's email* (user), *theme preference* (user or client), *which tab is selected on the dashboard* (URL), *the JWT for an external API* (session, encrypted), *user's display name* (user), *cart contents during a guided sales flow* (session if server-validated, URL if shareable), *device label for the active session* (session ŌĆö `userAgent`). Wrong-answer feedback names the bug class each misplacement enables. Confirmed by a `MultipleChoice`: *"a teammate's PR adds `?orgId=...` to every protected URL 'so deep links can carry the org context.' What's the failure mode?"* ŌĆö the right answer names the trust-the-URL bug (a user editing the param to a non-member org), names the right shape (`session.activeOrganizationId` is the only authoritative read; deep links into another org should redirect through a switch-org Server Action that validates membership).

**Components.**
- `TabbedContent` for the four-lane placement comparison, each tab containing a small `Code` schema sketch, a hand-SVG inside `Figure` for the multi-tab/phone visualization, and a hand-coded HTML failure-or-fit ledger.
- `Buckets` for the eight-piece state-placement sort.
- `MultipleChoice` for the URL-param diagnosis.
- `Aside` (`caution`) below the tabs: "the active org lives on the `session` row. URL params, client storage, and the `user` table all ship a specific bug class ŌĆö name them once, never reach for them."

**Project link.** The 10.4 project's `requireOrgUser()` helper (lesson 10.4.3) reads `activeOrganizationId` from `auth.api.getSession({ headers })` and never from a route param; the four-lane comparison is the *why* the project's inspector page rejects any `?orgId=` query string outright as a probe of the discipline.

## Concept 2 ŌĆö The `databaseHooks.session.create` callback as the one place that covers every entry point

**Why it's hard.** "Set the active org on sign-in" feels like the right install point ŌĆö and it ships a real bug. Sign-in is one of *several* paths that mint a `session` row: sign-up, OAuth callback, magic-link redemption, invitation accept, even the rare admin "impersonate as user" flow all create sessions. Wiring the active-org logic in the sign-in action leaves every other path landing a session with `activeOrganizationId` null; the user signs up via Google, gets a working session, and the protected layout's `requireOrgUser()` redirect-loops to `/onboarding/create-org` for the wrong reason ŌĆö because the entry path forgot to set the slot. The senior reflex is to find *the one place every session-creation path passes through* and put the logic there. Better Auth's `databaseHooks.session.create.before` is exactly that surface; the discipline isn't writing the lookup, it's recognizing that any per-action wiring is structurally incomplete.

**Ideal teaching artifact.** Pattern archetype delivered as a **multi-entry-point convergence diagram with a wrong-then-right fork**. A hand-SVG inside `Figure` draws six entry points fanning in from the left ŌĆö *email + password sign-in*, *email + password sign-up*, *Google OAuth callback*, *magic-link redemption*, *invitation-accept handshake*, *admin impersonate* ŌĆö each labeled with the action it's wired in. The arrows converge on a single node labeled `session row created`. *Branch A (broken)*: the active-org-set logic lives in a small box hanging off the *sign-in* arrow only. The diagram is annotated: five of the six entry points pass through *without* hitting the box; the resulting session row's `activeOrganizationId` is null; the layout redirects to `/onboarding/create-org` and the user has no idea why. *Branch B (senior baseline)*: the active-org-set logic moves to a box wrapping the convergence node ŌĆö `databaseHooks.session.create.before` ŌĆö and every arrow passes through it before the row is written. A small `Code` block below shows the four-line hook body: look up the user's most-recent active org, fall back to first membership, fall back to null. The closing rule lands as one line: *if the wiring lives on an entry point, every new entry point is a missed wiring; if it lives on the convergence point, every entry point is covered structurally.*

**Engagement.** A `MultipleChoice` round of three scenarios. *"A teammate adds passkey sign-in next sprint. They write the action and ship the PR. The active-org slot wiring lives in the sign-in action ŌĆö what's the new bug?"* ŌĆö the right answer names the missing-wiring-per-entry-point pattern, names the structural fix (the hook). *"A teammate moves the active-org logic into the OAuth callback as well as the sign-in action 'to cover both paths.'"* ŌĆö the right answer names this as the wrong direction (every new entry point is one more place to remember; the hook is the one place that doesn't drift). *"The hook is in place; the team adds an admin 'impersonate' flow that calls a custom session-mint endpoint that bypasses the hook for performance reasons."* ŌĆö the right answer names this as the bypass that re-opens the bug class, ties to the senior reflex (if you bypass the hook, you re-implement the lookup; never two implementations of the same invariant). Confirmed by a `Tokens` exercise on the canonical Better Auth config block ŌĆö the student clicks `databaseHooks`, `session`, `create`, `before` (the four load-bearing tokens that name where the logic lives) with feedback naming what each one closes.

**Components.**
- `Figure` wrapping a hand-SVG convergence diagram with the six entry points fanning into the session-creation node, with the broken-vs-senior fork shown as two stacked variants.
- `Code` block showing the four-line hook body.
- `MultipleChoice` for the three-scenario round.
- `Tokens` on the Better Auth config snippet for the four load-bearing path tokens.
- `Aside` (`note`) below the diagram: "find the convergence point; every per-entry-point wiring is structurally incomplete."

**Project link.** Lesson 10.4.3 of the project ships exactly this hook in the Better Auth config; the convergence diagram is the *why* the project's verification probes include "sign up via Google, confirm `activeOrganizationId` is set without any sign-up-specific code path."

## Concept 3 ŌĆö Switch-org is a Server Action: cache wipe, cookie-rewrite, and what happens to in-flight work

**Why it's hard.** Three orthogonal mechanics fire when the user picks a different org from the switcher, and the student treats them as one event. *(1) The active-org cookie cache* ŌĆö Better Auth's cookie cache from 9.2.3 holds the session payload including `activeOrganizationId`; `setActive` rewrites it, but the student has to know the rewrite happens immediately (no waiting on `maxAge`) so the next request reads the new value. *(2) The Next.js route cache* ŌĆö if the action returns without `revalidatePath('/', 'layout')`, the protected layout serves a stale render with the old org's data; the user sees Acme's invoices for ten more seconds and concludes the switch didn't work. *(3) The in-flight Server Action that started under the old org* ŌĆö a slow save that read `orgId` at its start completes against the old org *correctly*; the student's instinct is to call this a race condition and try to cancel it, but the senior reading is that actions resolve in the context they started, and `router.refresh()` after switch is what closes the visible-state ambiguity. Conflating the three ships a UX where the user can't trust either the org switcher or any unsaved form they had open.

**Ideal teaching artifact.** Mechanics archetype delivered as a **three-track timeline scrub** in `DiagramSequence`, with each track resolving a different mechanic on a shared timebase. The student scrubs through eight frames; each frame the three tracks update independently. *Track 1 ŌĆö cookie cache state* shows the cookie's payload (org A ŌåÆ org B at the `setActive` frame, no waiting). *Track 2 ŌĆö route cache state* shows the cached layout (org A's data ŌåÆ invalidated ŌåÆ org B's data, gated on the `revalidatePath` call inside the action). *Track 3 ŌĆö in-flight Server Action* shows a slow `saveInvoiceDraft` action that started 200ms before the switch ŌĆö it holds `orgId = A` in its closure, runs against org A's tenant scope, commits an org-A row, and a green check lands on the timeline. A small ledger below the tracks resolves what the user sees in three frames: *frame 4 ŌĆö switcher shows org B, layout still shows org A's data (stale-cache window if `revalidatePath` is missing)*; *frame 5 ŌĆö `revalidatePath` fires inside the action, cache invalidated, next render reads org B*; *frame 8 ŌĆö the slow action completes against org A; this is correct, not a bug; the in-flight write belonged to the old session context*. A second small beat below the scrub names the three lines of the canonical switch action body: `await authClient.organization.setActive({ organizationId })`, `revalidatePath('/', 'layout')`, return `Result` ŌĆö three lines, three mechanics closed.

**Engagement.** A `MultipleChoice` round of three scenarios. *"After shipping the switcher, users report 'it shows the new org in the dropdown but the dashboard still has the old company's data for a while'"* ŌĆö the right answer names the missing `revalidatePath` and the stale layout cache. *"The team adds an aggressive `router.refresh()` plus a hard-reload-after-switch as a workaround for the staleness bug instead of fixing the action."* ŌĆö the right answer names this as papering over the missing structural call; `revalidatePath` inside the action is the right surface, `router.refresh()` after is complementary, the hard reload is a tell. *"A user reports 'I started saving an invoice in Acme, switched to Personal mid-save, and the invoice ended up in Acme. Is that a bug?'"* ŌĆö the right answer names the action-context invariant (actions resolve in the context they started; the in-flight save against Acme is correct), names the UX move (the form's optimistic state should show "saved to Acme" rather than imply the new org). Confirmed by a `Sequence` ordering of the seven steps that fire when the user clicks switch: *(1) client calls `setActiveOrgAction(orgId)`*, *(2) action validates membership server-side (library-internal)*, *(3) library updates `session.activeOrganizationId` row*, *(4) library rewrites the session cookie payload*, *(5) action calls `revalidatePath('/', 'layout')`*, *(6) action returns `{ status: 'ok' }`*, *(7) client calls `router.refresh()` and the next render reads `activeOrganizationId = B`*. One distractor: *"clear `localStorage` to drop client-side org state"* ŌĆö no, the active org never lived in `localStorage` (Concept 1).

**Components.**
- `DiagramSequence` for the three-track timeline scrub (cookie cache / route cache / in-flight action) sharing a timebase ŌĆö the parallel-lanes pattern from Unit 9 extended to three tracks.
- `Code` block showing the three-line canonical switch action body.
- `MultipleChoice` for the three-scenario round.
- `Sequence` for the seven-step switch lifecycle with the `localStorage` distractor.
- `Aside` (`caution`) below the scrub: "three mechanics, three closing calls ŌĆö `setActive` rewrites the cookie, `revalidatePath` wipes the layout cache, the in-flight action belongs to the old context. Never paper over a missing `revalidatePath` with a hard reload."

**Project link.** The 10.4 project's `setActiveOrgAction` ships the canonical three-line shape; the three-track scrub is the *why* lesson 10.4.3 calls out the `revalidatePath` line as load-bearing rather than as boilerplate, and the *frame* the project's verification step uses to test "switch org, confirm the layout reads the new org's data on the very next navigation."

## Concept 4 ŌĆö The missing-`where` bug class: making the wrong query *not compile*, not *not pass review*

**Why it's hard.** The student's working mental model is "the team will catch missing org filters in code review." This is the wrong frame because the bug is structurally invisible at review time ŌĆö `db.select().from(invoices).where(eq(invoices.id, id))` looks identical whether the table is tenant-owned or global; nothing on the call site flags the missing scope. The reviewer would have to remember, for every query, which tables carry `organizationId`, and check that the predicate is present. That's a discipline that decays the moment the team grows past three engineers or ships under deadline. The senior shift is from "remember to filter" to "make the unfiltered call shape *not exist*" ŌĆö the helper makes the org predicate the only path that compiles, the unwrapped client is reserved for explicit cross-org reach, and the bug class moves from "review responsibility" to "type system responsibility." The student needs to *feel* the review-invisibility before the helper signature reads as load-bearing rather than as ceremony.

**Ideal teaching artifact.** Pattern archetype delivered as a **mock pull-request review with a delayed reveal**, rendered as a `Figure`-wrapped hand-coded HTML PR view followed by a `CodeVariants` wrong-then-right replay. *Beat one ŌĆö the PR ambush:* the student is shown a small PR diff inside `Figure` (mocked GitHub-style: file header `app/invoices/[id]/page.tsx`, three changed lines visible). The diff shows a clean-looking server-component fetch ŌĆö `const invoice = await db.query.invoices.findFirst({ where: eq(invoices.id, params.id) })` ŌĆö surrounded by a credible PR description ("fix: render single invoice"). The student is asked, in prose under the figure, *"would you approve this PR?"* with a `MultipleChoice` (approve / request changes / approve with comment). The reveal lands after the answer: *the approve answer is wrong; the bug is the missing `eq(invoices.organizationId, orgId)` predicate; a user can navigate to any invoice ID and the route returns it; the PR is a cross-tenant data leak in two lines.* The reveal annotation calls out that nothing in the call site visually flags the bug ŌĆö the file looks normal, the call shape is the same as for global tables, the reviewer has to *remember* the table is tenant-owned to spot it. *Beat two ŌĆö the structural cure:* `CodeVariants` with two tabs. Tab 1 (the broken shape just reviewed): unwrapped `db.query.invoices.findFirst(...)` with the missing predicate, captioned *the bug is invisible in code review*. Tab 2 (the senior baseline): `tenantDb(orgId).query.invoices.findFirst({ where: eq(invoices.id, params.id) })` ŌĆö the helper injects `and(eq(invoices.organizationId, orgId), ...)` automatically; calling the helper without an `orgId` doesn't typecheck; calling it on a global table (`tenantDb(orgId).query.users.findMany(...)`) doesn't typecheck. A small ledger underneath: *broken ŌĆö bug class moves from "missed predicate" (invisible) to "missed PR review" (impossible to catch reliably); senior ŌĆö bug class moves to "imported the wrong client" (one grep, visible at every call site).*

**Engagement.** The PR ambush itself is the recall (the wrong-answer is the assessment that surfaces the misframe). Confirmed by a `CodeReview` exercise: the student is given a six-file PR with three tenant-table reads and three legitimately-global reads (`user`, `verification`, `organization`); the rubric asks the student to leave a "request changes" comment on the unwrapped tenant reads naming the missing helper, leave the global reads alone, and flag any case where the unwrapped client is imported into a request-handler file (which, even when the call is correct, is a smell). Wrong comments on the global reads get rubric feedback naming the registry distinction (Concept 5).

**Components.**
- `Figure` wrapping a hand-coded HTML mock PR diff (file header, three changed lines, credible PR description) ŌĆö the visual ambush.
- `MultipleChoice` immediately under the PR ambush for the approve / request-changes / comment decision, with the delayed-reveal feedback as the teaching beat.
- `CodeVariants` for the two-tab broken/senior replay with the helper signature visible.
- `CodeReview` for the six-file PR exercise on the structural-vs-stylistic distinction.
- `Aside` (`caution`) below the variants: "the bug is invisible in code review; the helper makes the bug not compile. The reviewer's job changes from 'check every where clause' to 'flag any unwrapped tenant import.'"

**Project link.** The 10.4 project ships the unwrapped `db` client and the `tenantDb(orgId)` facade side by side; the PR ambush is the *why* the project's verification step (lesson 10.4.8) includes a grep probe ŌĆö `grep -r "from '@/lib/db'" app/` should return zero hits in request-handler files ŌĆö as the structural check the team would run on every PR.

## Concept 5 ŌĆö The `tenantDb` shape: typed wrapper, table registry, FK-child inheritance

**Why it's hard.** Once the student accepts that the helper is the structural cure (Concept 4), the next question is *what does it actually wrap, and how does the type system enforce the contract?* Three sub-bites land together. *(a) Wrap, don't replace*: the helper exposes the same shape Drizzle does (`.query.X.findMany`, `.select().from(X)`, `.insert(X)`, `.update(X)`, `.delete(X)`) so the student's existing Drizzle muscle memory carries over; the helper is a thin facade, not a new ORM. *(b) The table registry*: some tables carry `organizationId` (tenant-owned), some don't (`user`, `verification`, lookup tables); the helper has to refuse the global-table call shape at the type level, otherwise the student writes `tenantDb(orgId).query.users.findMany(...)` and gets a runtime error or, worse, a no-op filter that passes silently. *(c) FK-child inheritance*: `query.invoices.findMany({ with: { lines: true, customer: true } })` joins through children; the helper scopes the *root*, and children whose FK chains back to a tenant table inherit the scope through the join ŌĆö the student has to see the schema's FK shape as the second half of the discipline (the helper is one layer; the FK graph is the other). The misconception is treating the helper as a runtime filter to remember, rather than as a *type-shape that documents which tables are tenant-owned and which aren't*.

**Ideal teaching artifact.** Concept + Pattern composite delivered as a **type-shape diagram paired with a relational-API walkthrough**. *Beat one ŌĆö the type shape:* a hand-SVG inside `Figure` drawing the helper's type plumbing in three layers, top to bottom. *Top layer ŌĆö the schema*: ten tables drawn as small rectangles, four annotated with a green `organizationId` badge (`invoices`, `customers`, `documents`, `audit_logs`) and six annotated as global (`user`, `session`, `verification`, `organization`, `member`, `invitation`). *Middle layer ŌĆö the registry*: a `TENANT_TABLES` type-level set keyed off the four green tables, drawn as a labeled box with edges to the schema's tenant tables only. *Bottom layer ŌĆö the helper signature*: `tenantDb(orgId).query.X` where `X` is constrained to `keyof TENANT_TABLES` ŌĆö the type system refuses the global tables. Two callouts on the diagram: a green "Ō£ō" arrow from the schema's `invoices` to the helper's `query.invoices` ("compiles"), and a red "Ō£Ś" arrow from the schema's `user` to a hypothetical `query.users` call ("type error: `users` not in tenant registry"). *Beat two ŌĆö the relational-API walkthrough:* an `AnnotatedCode` block on the canonical multi-table read ŌĆö `tenantDb(orgId).query.invoices.findMany({ with: { lines: true, customer: true }, where: eq(invoices.status, 'open') })` ŌĆö with three highlighted spans. *Span 1 ŌĆö the root scope*: the helper composes the caller's `where` with `and(eq(invoices.organizationId, orgId), ...)`; pinned at the root. *Span 2 ŌĆö the child via FK*: `customer` is tenant-owned; the FK `invoices.customerId` joins to `customers.id`; the customer row is reachable only because the parent invoice was scoped, so the customer is implicitly scoped through the join graph. *Span 3 ŌĆö the leaf via FK*: `lines` is a child of `invoices` (no `organizationId` of its own ŌĆö the FK to the parent tenant table is the scope); the join structure carries the discipline. The closing rule: *the helper scopes the root; the schema's FK graph carries the rest; tables that don't chain to a tenant table read globally and the registry has to know they're allowed to.*

**Engagement.** A `TypeCoding` exercise where the student writes (or fills in) the `TENANT_TABLES` type and the helper's `query` accessor type so that *(a)* `tenantDb(orgId).query.invoices.findMany({})` typechecks, *(b)* `tenantDb(orgId).query.users.findMany({})` produces a *named* type error (`"'users' is not a tenant-scoped table ŌĆö use the unwrapped db client"`), *(c)* the helper's return type matches Drizzle's so existing autocomplete still works. The grader runs Twoslash `^?` queries on three positions to confirm the types resolve correctly. Confirmed by a `Buckets` sort on twelve table names from a SaaS schema into two columns ŌĆö *tenant-owned (carries `organizationId`)* / *global (no `organizationId`)* ŌĆö with examples like `invoices`, `customers`, `documents`, `comments`, `attachments` (tenant), `user`, `session`, `verification`, `organization`, `member`, `country_codes` (lookup, global), `feature_flags` (depends ŌĆö flag this as the trap, the right answer is "depends; if flags are per-org, tenant-owned; if global, global; the registry is where this decision is made explicit").

**Components.**
- `Figure` wrapping a hand-SVG three-layer type-shape diagram (schema / registry / helper signature) with the green-Ō£ō and red-Ō£Ś annotation pair.
- `AnnotatedCode` for the relational-API walkthrough with three highlighted spans (root scope, child via FK, leaf via FK).
- `TypeCoding` for the registry-and-helper-signature exercise with three Twoslash positions.
- `Buckets` for the twelve-table tenant-vs-global sort with the `feature_flags` trap.
- `Aside` (`note`) below the walkthrough: "the helper scopes the root; the FK graph carries the children; the registry is the type-level record of which tables are tenant-owned. Adding a new table is a registry update, not a helper change."

**Project link.** Lesson 10.4.5 of the project ships the `tenantDb(orgId)` facade with the `TENANT_TABLES` registry; the type-shape diagram is the *why* the project's `tenantDb` source file leads with the registry declaration before the helper body, and the `TypeCoding` exercise is the rehearsal the student carries into the project build.

## Concept 6 ŌĆö The `orgId` provenance rule and the "different file, different client" cross-org reach

**Why it's hard.** The helper's correctness is conditional on *where the `orgId` came from*. If `orgId` is a parameter the function received, the helper is only as safe as the caller ŌĆö and the natural reflex when wiring multi-tenant features is to thread `orgId` through a few layers of business logic, eventually accepting it as a function argument or, worst case, as a route param or form field. The senior reflex is the inverse: `orgId` has *one acceptable provenance* ŌĆö `requireOrgUser()`, which derives it from the validated session ŌĆö and any other source is a smell. The second half of the concept is the "I need to read across orgs" reach: admin tools, batch jobs, BI exports legitimately need to span tenants, and the senior call is *not* to add an `allOrgs: true` mode to `tenantDb` (which normalizes the bypass) but to reach for the unwrapped `db` client *in a different file under `src/lib/admin/` or `scripts/`* ŌĆö the cross-org call is a deliberate decision the file path makes visible. The student conflates these as "the helper has an escape hatch when you need to cross orgs"; the senior shape is "the helper has no escape hatch; the escape lives in a different file."

**Ideal teaching artifact.** Decision archetype delivered as a **provenance flow diagram paired with a two-file directory comparison**. *Beat one ŌĆö the provenance flow:* a hand-SVG inside `Figure` showing four candidate `orgId` sources flowing into the helper, with three of them marked with a red Ō£Ś and one with a green Ō£ō. The four sources: *(1) function parameter* (Ō£Ś ŌĆö only as safe as the caller; the helper can't validate the caller checked membership), *(2) route param `[orgId]` from the URL* (Ō£Ś ŌĆö the user can edit the URL; this is the cross-tenant attack from Concept 1 at the data layer), *(3) form field / request body* (Ō£Ś ŌĆö the client can submit any value; never trust client-supplied identifiers), *(4) `requireOrgUser()`* (Ō£ō ŌĆö derived from the session, validated against `member` table, single source of truth). Each Ō£Ś has a short callout naming the bug class it ships. The diagram's bottom names the canonical action shape: `const { user, orgId } = await requireOrgUser(); const db = tenantDb(orgId);` ŌĆö three tokens, no other valid composition. *Beat two ŌĆö the cross-org reach as a directory layout:* a `FileTree` block showing two directories side by side. Directory A ŌĆö `src/app/(app)/invoices/actions.ts`: imports `tenantDb`, calls `requireOrgUser()`, every read scoped. Directory B ŌĆö `src/lib/admin/cross-org-reports.ts`: imports the unwrapped `db` directly, has a comment header *"runs as super-admin; bypasses tenant scope by design; reach only when the request is non-tenant-bound"*; the file path itself signals the intent. A small annotation pairs the two: *the helper has no `allOrgs: true` mode because the file path is the escape hatch; if a cross-org call shows up under `app/`, it's a bug; if it shows up under `lib/admin/`, it's a feature.* The closing rule lands as one line: *one provenance for `orgId`; one location for the unwrapped client; mixing either is a review-loud smell.*

**Engagement.** A `Buckets` sort on ten code locations into two columns ŌĆö *uses `tenantDb(orgId)` from `requireOrgUser()`* / *uses unwrapped `db` (and the file lives under `lib/admin/` or `scripts/`)*. Examples: *the invoice-list Server Action* (helper), *the nightly retention-cleanup script* (unwrapped, scripts), *the `requireOrgUser` helper itself* (unwrapped ŌĆö it reads `member` to validate the active org, and `member` is the boundary table), *the org-switcher action* (unwrapped ŌĆö it's reading the user's full membership list across orgs to render the switcher), *the customer-detail page* (helper), *the admin "list users across all orgs" page* (unwrapped, lib/admin), *the audit-log writer for a tenant action* (helper, with `withTenant` from Concept 8 stacked on), *the BI export job that runs nightly under cron* (unwrapped, scripts), *the support-tool "view this customer's tickets across all their orgs" surface* (unwrapped, lib/admin), *the per-org dashboard count widget* (helper). Wrong-answer feedback names whether the misplacement is a security bug (helper-needed file using unwrapped) or a smell (helper used in a place where the unwrapped is correct because the surface is genuinely cross-tenant). Confirmed by a `MultipleChoice`: *"a teammate proposes adding a `tenantDb(orgId, { allOrgs: true })` mode 'so the admin tools don't have to import a different client.' What does the senior reviewer say?"* ŌĆö the right answer names the normalization-of-bypass argument (every call site that wants to skip can pass the flag; the discipline collapses), names the alternative (file-path discipline; the unwrapped import is a one-line review signal).

**Components.**
- `Figure` wrapping a hand-SVG four-source provenance diagram with three Ō£Ś and one Ō£ō annotations.
- `FileTree` (or two side-by-side `Code` blocks with file headers) showing the directory layout ŌĆö `app/.../actions.ts` vs `lib/admin/.../cross-org-reports.ts`.
- `Buckets` for the ten-location sort with feedback distinguishing security-bug from smell.
- `MultipleChoice` for the `allOrgs: true` proposal diagnosis.

## Concept 7 ŌĆö The RLS threshold: three triggers, defense-in-depth, and a per-table decision

**Why it's hard.** RLS sits at a misframe collision point. The student has two failure modes available. *Mode one ŌĆö RLS-everywhere maximalism:* "the database can enforce tenancy structurally; let's RLS every table"; the bill is operational (every connection needs a session variable, every local debug session breaks, policy authoring becomes the team's full-time job, and the helper-discipline atrophies because "the database catches it"). *Mode two ŌĆö RLS-never minimalism:* "we have `tenantDb`, we're done"; the bill is a real one-missed-application-layer-scope leak on a regulatory-sensitive table where the team didn't have the budget for the post-mortem. The senior shape is *threshold-driven, per-table*: three triggers (highest-stakes data class, many writer paths, external writers) and a default-otherwise. RLS is *defense-in-depth* ŌĆö the application still uses `tenantDb` even on RLS-protected tables; the two layers fail independently. The student needs the threshold as a *decision rule*, not as a yes/no project-level flag, and they need to feel why each trigger crosses the cost threshold rather than memorize the list.

**Ideal teaching artifact.** Decision archetype delivered as a **per-table decision tree paired with a cost-and-coverage comparison table**. *Beat one ŌĆö the comparison table:* a hand-coded HTML table inside `Figure` with two columns ŌĆö *application-layer scope (`tenantDb`)* / *RLS on the table* ŌĆö and five rows. *Bug class caught*: helper catches missed `where` in the *normal request path*; RLS catches it for *every* path including hand-rolled scripts and external writers. *Operational cost*: helper is one wrapper file, no DB-side state; RLS demands per-request `SET LOCAL`, pooled-connection discipline, owner-bypass awareness, policy authoring as ongoing work. *Local debugging story*: helper is invisible to `psql`; RLS makes `psql` queries return empty unless the variable is set ("why are there no rows" loop). *When the wrong layer fails*: helper fails when an engineer reaches for the unwrapped client in an `app/` file (review-visible, grep-visible); RLS fails silently (a bad policy returns empty, no error ŌĆö the app shows empty lists, nobody notices for a week). *When to reach*: helper is the year-1 default for every tenant-owned table; RLS is the conditional reach when the threshold is met. *Beat two ŌĆö the per-table decision tree:* a hand-SVG inside `Figure` with three diamond decision nodes flowing top to bottom. *Diamond 1 ŌĆö "is a cross-tenant leak from this table unrecoverable?"* (regulatory, audit, security-sensitive): yes ŌåÆ RLS; no ŌåÆ continue. *Diamond 2 ŌĆö "is this table written by code paths outside the request handler?"* (jobs, scripts, external integrations, partner ETLs): yes ŌåÆ RLS; no ŌåÆ continue. *Diamond 3 ŌĆö "is the team the only writer?"*: no (third party with DB credentials) ŌåÆ RLS; yes ŌåÆ application-layer scope via `tenantDb`. The tree's leaves carry one line each: RLS leaf ŌĆö *"add to RLS table list, wire `withTenant`, write the integration test (Concept 9)"*; helper leaf ŌĆö *"`tenantDb` enforces the discipline, ship"*. Worked example below the tree: walk three tables ŌĆö *`invoices`* (no on all three diamonds ŌåÆ helper), *`audit_logs`* (yes on diamond 1 ŌåÆ RLS), *`payment_methods`* (yes on diamond 1 if storing card data ŌåÆ RLS) ŌĆö and name the decision per table.

**Engagement.** A `MultipleChoice` round of three scenarios. *"The team's instinct is 'RLS on every table for safety.' What's the senior pushback?"* ŌĆö the right answer names the operational cost catalog (debugging loop, pool discipline, policy drift), names the helper-discipline-atrophy second-order effect (when "the database catches it" becomes the mental model, the unwrapped-client smells stop being noticed). *"The team ships a partner integration that runs nightly with a service-account DB credential to write a `partner_events` table. The application layer never touches it. RLS or no?"* ŌĆö the right answer names diamond 3 (external writer), names RLS as the right call. *"The team puts RLS on `audit_logs` and removes `tenantDb` from audit-log writes 'because the database catches it.'"* ŌĆö the right answer names defense-in-depth (the application's explicit `organizationId` value catches a policy bug; the policy catches an application bug; removing one layer halves the protection). Confirmed by a `Buckets` sort across eight tables from the course's stack ŌĆö *`invoices`*, *`customers`*, *`documents`*, *`audit_logs`*, *`partner_webhook_events` (third-party writer)*, *`comments`*, *`api_keys` (high-stakes credential)*, *`feature_flags`* ŌĆö into two columns *RLS earns its cost* / *helper-only is sufficient*. Wrong-answer feedback names which diamond was missed.

**Components.**
- `Figure` wrapping a hand-coded HTML two-column comparison table (helper vs RLS) across five rows.
- `Figure` wrapping a hand-SVG three-diamond decision tree with worked examples below.
- `MultipleChoice` for the three-scenario round.
- `Buckets` for the eight-table per-table classification.
- `Aside` (`note`) below the tree: "RLS is per-table, not per-project. The same project can have RLS on `audit_logs` and not on `invoices` and be correctly designed. Defense-in-depth ŌĆö never remove `tenantDb` from an RLS-protected table."

**Project link.** Lesson 10.4.4 of the project applies RLS to `audit_logs` only; the decision tree is the *frame* the project's brief uses to justify why `invoices` (added in Unit 11's project) does not get RLS, and the *why* the inspector verifies the per-table choice rather than testing a project-level flag.

## Concept 8 ŌĆö `SET LOCAL` inside a transaction: the pooled-connection footgun and the `withTenant` helper

**Why it's hard.** The naive shape ŌĆö "set `app.org_id` once on the connection and let RLS handle the rest" ŌĆö ships a real cross-tenant leak in 2026. Connection pooling is the default everywhere (Drizzle's Postgres adapter, Neon's serverless driver, PgBouncer in transaction mode), and pooled connections are reused across requests. Two failure modes ride this. *(1) Variable-not-set leak:* request 2 reuses request 1's connection without setting the variable; the policy reads the still-set value from request 1; request 2 reads org A's data while authenticated as org B. *(2) Variable-not-cleared leak:* request 1 sets the variable, completes; request 2 reuses the connection, forgets to set, inherits request 1's value; same outcome. The mitigation isn't "remember to set on every request" (decays under deadline); it's `SET LOCAL` *inside an explicit transaction* ŌĆö `SET LOCAL` is bound to the transaction, automatically cleared on commit/rollback, structurally impossible to leak across requests. The student needs to feel the leak vector concretely (not as abstraction), then see `withTenant(orgId, fn)` as the wrapper that makes the leak structurally impossible.

**Ideal teaching artifact.** Concept + Pattern composite delivered as a **two-track pooled-connection simulator**, mirroring the pool simulator vocabulary installed in 6.1's `ConnectionPoolSim`. *Beat one ŌĆö the leak walkthrough:* a `DiagramSequence` with five frames showing two requests sharing a pooled connection over time, side by side. *Track A (broken) ŌĆö `SET` without `LOCAL` and without a transaction:* frame 1 ŌĆö request 1 starts, checks out connection from pool, runs `SET app.org_id = 'A'`; frame 2 ŌĆö request 1 inserts an audit row for org A, returns response, *does not unset the variable*, releases connection back to pool with `app.org_id = 'A'` still set; frame 3 ŌĆö request 2 (authenticated as org B) checks out the same connection from the pool; frame 4 ŌĆö request 2 inserts what it thinks is an audit row for org B, but `app.org_id` is still 'A' from request 1 ŌĆö the policy's `WITH CHECK` *succeeds* if the application code passed `organizationId: 'B'` and the policy's `WITH CHECK` is `organization_id = current_setting('app.org_id')::uuid` ('A' != 'B'), the insert fails with a policy violation; or *worse*, if the application code reuses the variable's value (which it shouldn't, but if), the audit row lands in org A's data; frame 5 ŌĆö annotated reveal: *the variable persisted across the request boundary; the connection pool is the leak vector; the request did the wrong thing for a structural reason*. *Track B (senior) ŌĆö `SET LOCAL` inside a transaction:* same five frames, but every database operation in request 1 is wrapped in `BEGIN; SET LOCAL app.org_id = 'A'; INSERT ...; COMMIT;`; on `COMMIT`, the transaction-scoped variable is cleared; frame 5 ŌĆö annotated reveal: *the variable's lifetime is the transaction's lifetime; pool reuse cannot leak it; the next request starts with no variable set, and the policy fails closed (`current_setting('app.org_id', true)` returns NULL, the predicate against NULL excludes every row, the request fails loudly rather than silently leaking)*. *Beat two ŌĆö the `withTenant` helper:* a `Code` block showing the wrapper signature and body ŌĆö `await withTenant(orgId, async (tx) => { await tx.insert(auditLogs).values({ organizationId: orgId, ... }) })` ŌĆö annotated to call out three things. *Annotation 1*: the helper opens an explicit `db.transaction(...)`, runs `SET LOCAL` as the first statement, then yields the transaction handle to the callback. *Annotation 2*: the callback receives `tx`, not the unwrapped `db` ŌĆö every operation inside the callback runs inside the transaction (and therefore inside the `SET LOCAL` scope). *Annotation 3*: the helper composes with `tenantDb` from Concept 5 ŌĆö the `tx` can be wrapped by the same registry-typed facade so the application-layer scope holds inside the RLS-protected transaction; defense-in-depth, both layers, both writes look the same.

**Engagement.** A `MultipleChoice` round of three scenarios. *"A teammate's PR sets `app.org_id` in a connection-init hook on the pool ('once per connection, every query benefits'). What's the failure mode?"* ŌĆö the right answer names the per-request-not-per-connection invariant (the variable's value is per-request data, not per-connection setup); ties to the senior reflex (`SET LOCAL` inside a transaction is the only safe primitive). *"A teammate's PR uses `SET app.org_id` (no `LOCAL`) inside a transaction 'because the transaction commits and we're done.'"* ŌĆö the right answer names the trap (`SET` without `LOCAL` is session-scoped even inside a transaction; the COMMIT does not clear it; the next checkout from the pool inherits it); names the fix (`SET LOCAL` is the only correct shape). *"A teammate proposes a test that sets `app.org_id` and runs `SELECT * FROM audit_logs` outside a transaction to 'verify the policy works' ŌĆö the test passes."* ŌĆö the right answer names two bugs: the test passes for the wrong reason (the variable is set at session scope, not transaction scope, so it persists across the test's queries); and if the test is running as the table owner, it bypasses the policy entirely (the test passes regardless of whether the policy is correctly written) ŌĆö both are setups for Concept 9's proof-of-isolation discipline. Confirmed by a `Sequence` ordering of the seven steps inside `withTenant`: *(1) `db.transaction(async (tx) => { ... })` opens a transaction (BEGIN under the hood)*, *(2) `await tx.execute(sql\`SET LOCAL app.org_id = ${orgId}\`)` scopes the variable to this transaction*, *(3) callback runs, every read and write inside it is filtered by the policy against the set variable*, *(4) callback returns successfully*, *(5) Drizzle issues COMMIT*, *(6) Postgres clears `app.org_id` because it was `SET LOCAL`*, *(7) connection returns to pool with no tenant variable set; the next request starts clean*. One distractor: *"set `app.org_id = 'reset'` after the transaction commits to be safe"* ŌĆö no, `SET LOCAL` clears itself; reaching for a manual reset is a tell that the wrong primitive was used.

**Components.**
- `DiagramSequence` for the five-frame two-track pooled-connection simulator (broken `SET` vs senior `SET LOCAL`-in-transaction) ŌĆö reuses the parallel-lanes shape from 9.1 and 9.4.
- `Code` block with the `withTenant` helper signature and body, with three callout annotations.
- `MultipleChoice` for the three-scenario round.
- `Sequence` for the seven-step `withTenant` lifecycle with the manual-reset distractor.
- `Aside` (`caution`) below the simulator: "`SET LOCAL` inside a transaction is the only safe primitive on a pooled connection. `SET` (session-scoped) leaks across requests. Connection-init hooks are the wrong layer. The helper is the only call shape; reach for raw `SET` and the leak is back."

**Project link.** Lesson 10.4.4 of the project ships `withTenant(orgId, fn)` as the only audit-log write surface; the two-track simulator is the *why* the project's `logAudit(tx, event)` signature *requires* a transaction handle (no overload accepts the unwrapped `db`), and the *frame* the project's verification probes use to confirm ŌĆö by killing a request mid-transaction in `psql` ŌĆö that the variable doesn't leak to the next checkout from the pool.

## Concept 9 ŌĆö Drizzle policy authoring, `ENABLE` + `FORCE`, and the integration test that proves isolation

**Why it's hard.** Three tightly-coupled mechanics land together in lesson 10.1.4 and any one missed silently neutralizes the other two. *(1) The policy itself*: `USING (organization_id = current_setting('app.org_id', true)::uuid) WITH CHECK (...)` ŌĆö the `, true` argument and the `::uuid` cast are load-bearing (fail-closed when unset; loud failure on bad input); writing the policy without them ships a fail-open default. *(2) `ENABLE` *and* `FORCE`*: `ALTER TABLE audit_logs ENABLE ROW LEVEL SECURITY` makes the policy take effect for non-owners; without `FORCE`, the table owner (and any code running as owner ŌĆö including most migrations and most local-dev scripts) bypasses the policy entirely. The trap is that the team writes the policy, runs the migration as owner, runs tests as owner, sees rows, declares it working ŌĆö and ships an RLS-protected table that protects against nothing. *(3) The integration test that runs as the app role*: the only proof-of-isolation that doesn't deceive ŌĆö insert two rows for two orgs, set `app.org_id = A` inside a transaction, query without a `where`, assert one row returned, repeat for B. Run this as the *non-owner app role*, not the migration-running owner role; testing as owner makes every policy test a false positive. The student needs the three mechanics as a *single composed defense*, with the test as the only honest verification.

**Ideal teaching artifact.** Setup + Pattern composite delivered as a **stepped policy walkthrough culminating in a wrong-then-right test**. *Beat one ŌĆö the policy and its modifiers, annotated:* an `AnnotatedCode` block on the canonical Drizzle policy declaration in the schema file ŌĆö `pgPolicy('audit_logs_org_isolation', { as: 'permissive', for: 'all', to: ['authenticated'], using: sql\`organization_id = current_setting('app.org_id', true)::uuid\`, withCheck: sql\`organization_id = current_setting('app.org_id', true)::uuid\` })` followed by `.enableRLS()` ŌĆö with five highlighted spans. *Span 1 ŌĆö `, true`*: returns `NULL` instead of erroring when the variable is unset; the predicate against `NULL` excludes every row; *fails closed*. *Span 2 ŌĆö `::uuid` cast*: forces a typed domain check; a typo or a `'%'` value errors out loudly rather than returning surprising rows. *Span 3 ŌĆö `using` vs `withCheck`*: `using` filters reads, `withCheck` constrains writes; both predicates pinned, both layers covered. *Span 4 ŌĆö `for: 'all'`*: covers select/insert/update/delete in one policy; separate policies per command earn their weight only when the predicates diverge. *Span 5 ŌĆö `.enableRLS()`*: the modifier that makes Drizzle Kit emit `ENABLE ROW LEVEL SECURITY` in the generated migration. *Beat two ŌĆö the `FORCE` trap, made concrete:* a `CodeVariants` with two tabs on the generated migration SQL. Tab 1 (broken ŌĆö what some Drizzle versions emit by default): `ENABLE ROW LEVEL SECURITY` only; the panel underneath shows a `psql` session running as the table owner ŌĆö `SELECT * FROM audit_logs` returns every row, across every org, because the owner bypasses; the team's manual smoke test passes; the table is unprotected. Tab 2 (senior baseline): the same migration plus `ALTER TABLE audit_logs FORCE ROW LEVEL SECURITY` (added manually as a follow-up migration if Drizzle Kit didn't emit it); the same `psql` session as owner now also respects the policy and returns no rows unless the variable is set; the protection is real. The senior reflex rides the caption: *read the generated migration on first RLS table; never trust the codegen blindly when the cost of being wrong is total cross-tenant exposure*. *Beat three ŌĆö the only honest test:* a `Code` block showing the integration test shape ŌĆö *(a)* connect using the *app role* `DATABASE_URL`, not the owner's; *(b)* insert one `audit_logs` row for org A, one for org B, using the owner role (test setup); *(c)* run the assertion under the app role: open a transaction, `SET LOCAL app.org_id = A`, `SELECT * FROM audit_logs` (no `where`), assert one row returned and it's A's; commit; repeat for B; *(d)* assert that `SELECT * FROM audit_logs` *outside* the `withTenant` wrapper returns zero rows (the variable is unset, the policy fails closed). A small annotation underneath: *if this test passes when run as the owner role, it's lying; the test must run as the app role to be a real proof; this is the lesson's verification step and Chapter 19.3's integration suite is where the test permanently lives.*

**Engagement.** A `MultipleChoice` round of three scenarios. *"The CI runs the RLS isolation test using `DATABASE_URL_OWNER` (the migration credential) because 'it's already in CI'. What's wrong?"* ŌĆö the right answer names the owner-bypass false-positive (the test passes regardless of whether the policy is correct), names the fix (CI must have a separate app-role `DATABASE_URL` for the isolation test). *"The team writes the policy with `current_setting('app.org_id')::uuid` (no `, true`). The dev environment seeds the variable; CI doesn't. CI starts erroring on every RLS query with a Postgres error."* ŌĆö the right answer names two things: the missing `, true` is what makes the policy fail-loud instead of fail-closed in the unset case, *and* the senior reading is that the loud failure is correct ŌĆö the request that didn't go through `withTenant` should fail, not silently succeed; the fix is to make sure every reach for the table goes through `withTenant`, not to suppress the error by removing the cast. *"The team adds `BYPASSRLS` to the app role 'so local debugging is easier.'"* ŌĆö the right answer names this as defeating the entire defense, names the senior alternative (a `dev:psql` script that connects as the owner role for ad-hoc exploration; or a `\set` snippet in `~/.psqlrc` that sets `app.org_id` to the seed's first org). Confirmed by a `Tokens` exercise on the canonical policy declaration with five clickable load-bearing tokens (`, true`, `::uuid`, `using`, `withCheck`, `.enableRLS()`) ŌĆö the student clicks each, gets a one-line reveal naming the bug class each closes; decoys include `for: 'all'` (load-bearing only when the predicates are uniform ŌĆö flagged as conditional rather than wrong) and `as: 'permissive'` (the default ŌĆö reaching for `'restrictive'` is a legitimate-but-rare conditional, not a wrong choice).

**Components.**
- `AnnotatedCode` for the policy declaration with five highlighted spans.
- `CodeVariants` for the broken/senior migration SQL with `psql`-session annotations underneath each tab demonstrating the owner-bypass ŌĆö both panels reuse the wrong-then-right pattern from Concept 4.
- `Code` block for the integration test shape with the app-role-not-owner-role annotation.
- `MultipleChoice` for the three-scenario round.
- `Tokens` on the canonical policy declaration for the five load-bearing tokens with conditional-decoy feedback.
- `Aside` (`caution`) below the test block: "the test that runs as the owner role is lying. The integration suite must connect as the app role for the isolation proof to mean anything. `BYPASSRLS` on the app role defeats the entire defense ŌĆö never reach for it, ever."

**Project link.** Lesson 10.4.4 of the project ships the `auditLogs` schema with `pgPolicy` declarations including the `, true` and `::uuid` shape, the explicit `FORCE ROW LEVEL SECURITY` follow-up migration, and the integration test that runs against a non-owner role; the three-beat artifact is the *frame* lesson 10.4.8's verification rehearsal uses when probing `psql UPDATE` and `psql DELETE` refusals on `audit_logs` and confirming the cross-tenant probe returns the expected isolation.
