# Chapter 9.1 — The auth mental model: pedagogical approach

## Concept 1 — Authn vs authz, the request-lifecycle order, and the 401/403 split

**Why it's hard.** The student carries one undifferentiated bucket called "auth," and inside it "the user is signed in" silently doubles as "the user is allowed." The chapter's first move is to crack the bucket open: authentication answers *who is this request from*, authorization answers *is this principal allowed to do this thing on this resource*, and the two checks live at different boundaries with different status codes. Until the student can run a request through the request lifecycle in their head and predict at which step a 401 versus a 403 fires, every later lesson reads as a syntax tour.

**Ideal teaching artifact.** Concept archetype delivered as a **scrubbable request-lifecycle simulator** with two control knobs. The student sees one request — `POST /invoices/inv_42/delete` — animate from the browser through the proxy, into a Server Action, and out as a response. Two switches at the top: *session?* (none / valid / expired) and *role?* (member / admin). The student flips switches and steps through five frames: (1) request leaves the browser carrying the cookie; (2) proxy reads the session — fork: missing or expired → 401, valid → through; (3) Server Action runs `safeParse`; (4) `authedAction` wrapper reads role — fork: insufficient → 403, sufficient → through; (5) the row is deleted, `Result<{ ok: true }>` returns. The two fork points are the visual punchline — the student sees authn fail at step 2 with 401 and authz fail at step 4 with 403, never the other way around. A side panel lists every (session × role) combination with the predicted status code so the student can match the simulation against the table. Concept archetype; the artifact is a simulator and the boundary-vs-status mapping is the takeaway.

**Engagement.** A `MultipleChoice` round of four scenarios after the simulator: a curl with no cookie hitting `/invoices/delete` (401, before authz runs); a member trying to delete an admin-only invoice (403, after authn passes); an expired-session cookie hitting any protected route (401, the cookie's presence isn't proof of identity); a valid session whose user was just removed from the org (403, the cookie still authenticates the principal — the role check is what fails). Wrong-answer feedback names the boundary the misframe collapses.

**Components.**
- New component **`RequestLifecycleSimulator`** (sketched below) — the (session × role) controls plus the five-frame stepper. Recurs at 9.4.1 (the proxy gate — frame 2 of the simulator), 10.2.2 (the `authedAction` wrapper — frame 4 of the simulator), and 10.2.3 (`authedRoute`'s 401/403/422 status map) — clear forward-link.
- Alternative if not built: hand-SVG five-frame `DiagramSequence` showing the lifecycle with color-coded fork branches per (session × role) combination, plus a hand-coded HTML matrix below mapping every combination to the predicted status code. Loses the "flip-and-watch" cause-and-effect but renders the same mapping.
- `MultipleChoice` for the four-scenario round.

**Project link.** The 9.5 project's `proxy.ts` gate fires the 401 path and the action-boundary check fires the 403 path; the simulator is the first place the student sees the two boundaries that the project's verify lesson rehearses against curl probes.

## Concept 2 — Identification, authentication, authorization, and the misframe catalog

**Why it's hard.** Three concepts collapse into one in the student's vocabulary, and each collapse maps to a specific production bug class. *Identification* is the claim ("I am `ada@acme.com`"); *authentication* is the proof of the claim; *authorization* is the permission to act on the verified identity. The senior-recognized misframes — email-in-DB read as authenticated, signed-in read as allowed, paid read as authorized — each ship a real CVE-shape. On top of the three-way distinction sits the principal-state ladder (anonymous, authenticated, elevated) where elevation is *authentication triggered by an authorization policy* — a re-prompt that lives at the intersection of the two concepts, not in either one alone. The student needs the vocabulary first and the misframes attached to it second, so the catalog reads as a diagnostic rather than as trivia.

**Ideal teaching artifact.** Concept archetype delivered as a **misframe-first ambush followed by the dictionary**. The lesson opens with three short bug stories, one per misframe, each presented as a paragraph plus a one-line code excerpt: (1) *the unverified-email reset* — a password-reset link sent to an email row that was never proven, attacker takes over by typing the victim's email at signup; (2) *the layout-only gate* — a `/admin` layout that renders nothing for non-admins, but the underlying Server Action is unguarded and a curl POST works; (3) *the paid-equals-allowed entitlement* — a billing-tier check that gates a feature, the user is on the right plan but in the wrong org, the per-org scoping was never written. For each story, the student is asked *which collapse caused this?* before the dictionary appears. The dictionary then lands as a small three-row card stack — *identification / authentication / authorization* — with the boundary that owns each (form input / credential check / action handler). A second card stack to the right shows the principal-state ladder (*anonymous / authenticated / elevated*), with elevation defined as the re-authentication an authorization policy demands for high-value actions, and the canonical triggers named (credential change, billing change, destructive admin). The two stacks share the same visual vocabulary so the student reads the ladder as a *time series* of states the same principal moves through.

**Engagement.** A `Buckets` sort: twelve short scenarios into three buckets — *identification problem* / *authentication problem* / *authorization problem*. Examples seeded across the buckets: "the signup form accepts an email and saves it" (identification — the claim is recorded, nothing is proven); "the password matched" (authentication); "the role check at the action handler said no" (authorization); "the unverified email got a password-reset link" (identification confused for authentication); "the admin layout renders nothing for non-admins but the action still runs" (authorization at the wrong boundary); "the paid-tier user from org A tried to read org B's invoice" (authorization, scope dimension). Wrong-answer feedback names the bug class each misclassification ships. Confirmed by a `MultipleChoice`: "what triggers an elevation re-prompt, and which concept is being re-checked?" — correct answer names the policy-side trigger (changing credentials, changing billing, destructive admin) and identifies the re-prompt as *authentication* invoked by an *authorization* rule.

**Components.**
- `Figure` wrapping a hand-SVG card stack with the three-concept dictionary on the left and the principal-state ladder on the right, sharing the visual vocabulary. Single-use, no forward-link strong enough to compound — hand-SVG is the right scope.
- `Buckets` for the twelve-scenario sort across three columns.
- `MultipleChoice` for the elevation-trigger confirmation.
- `Aside` (`note`) below the dictionary: "identification is the claim, authentication is the proof, authorization is the permission. The misframes are the bugs that ship when any two collapse."

**Project link.** The 9.5 project ships email verification as the gate that closes the identification-equals-authentication misframe — the student arrives at lesson 9.5.4 already knowing *why* `requireEmailVerification` matters, not just *that* Better Auth has the flag.

## Concept 3 — Sessions versus JWTs: revocation as the load-bearing trade

**Why it's hard.** The student arrives with the JWT-as-modern-default reflex installed by years of "stateless is scalable" framing. The senior call is the inverse — for browser-driven SaaS in 2026, server-stored opaque sessions are the default and JWTs are the conditional reach — and the lever that flips the call is *revocation*, not performance, not statelessness, not edge-rendering. The student has to feel the asymmetry: with sessions, "stop this stolen cookie in the next second" is one DELETE; with JWTs, the same action is structurally impossible until the token expires unless a denylist re-introduces the very database read JWTs claimed to escape. Without the asymmetry made visceral, the student picks JWT for the wrong reason and the operational cost of revocation arrives later as an incident.

**Ideal teaching artifact.** Decision archetype delivered as a **side-by-side incident replay** in `DiagramSequence`. The student scrubs through the same five-frame incident in two parallel lanes — *session lane* on the left, *JWT lane* on the right. Frame one: a user's laptop gets stolen at a coffee shop with the browser logged in. Frame two: the user signs in from their phone, opens *Settings → Security*, clicks *Sign out everywhere*. Frame three: the action fires. *Session lane:* `DELETE FROM session WHERE userId = ?` — one statement, the row is gone. *JWT lane:* the server has nothing to delete; the existing JWT is signed and still verifies; the only path is to add the token's `jti` to a denylist and check it on every request. Frame four: the thief opens the laptop and clicks anything. *Session lane:* the next request's lookup returns no row, 401, redirect to sign-in. *JWT lane:* the request hits the denylist check; the database round-trip JWT was meant to avoid is now mandatory, and statelessness is gone. Frame five: a small ledger underneath shows what each lane bought and what it cost — *session: 1 DB lookup per request, instant revocation, stateful by definition; JWT-with-denylist: 1 DB lookup per request, eventual revocation if denylist is missed, stateful in everything but name; pure JWT: 0 DB lookups per request, no revocation, the stolen token is valid until `exp`.* The closing rule lands as one line: revocation is the trade, statelessness is what you give up to keep it, and for browser-driven SaaS the trade goes one way.

A second short beat below the replay names the threshold where pure JWTs *do* earn their weight — edge-rendered routes where the round-trip is the bottleneck, service-to-service calls, and the hybrid pattern (5–15 min access JWT + server-side refresh token) as the production-grade reach the early-stage SaaS doesn't pay for yet.

**Engagement.** A `MultipleChoice` round of four scenarios after the replay: "user reports their account was compromised — when does the stolen credential stop working?" (sessions: instantly on row delete; pure JWT: at `exp`, possibly hours away; hybrid: at access-token `exp`, minutes); "team wants to revoke all sessions of users on a deprecated app version" (one indexed UPDATE/DELETE; with JWTs needs a version claim plus a global cutoff check); "team wants to add a denylist to JWTs to fix revocation" (the denylist re-introduces the DB read; statelessness is the lie); "early-stage SaaS picks JWT to save the per-request DB hop" (the operational debt of revocation isn't visible at month one and arrives at month six as an incident — the choice is wrong for the wrong reason). Wrong-answer feedback names the property that was traded.

**Components.**
- `DiagramSequence` hosting the five-frame side-by-side incident replay, each frame a hand-SVG inside `Figure` with the session lane and JWT lane drawn parallel and a ledger row below. Reuses the parallel-lanes shape from 7.2 Concept 5's three-attacker DiagramSequence — confirms the forward-link.
- `MultipleChoice` for the four-scenario round.
- `Aside` (`note`) below the replay: "revocation is the trade. Server-stored sessions for browser-driven SaaS, JWTs for service-to-service or measured edge bottlenecks, hybrid only when both pressures land at once."

**Project link.** The 9.5 project ships sessions as the only carrier — no JWT path — and the chapter's verify lesson rehearses sign-out as a row delete; the incident replay is the *why* the project never reaches for the JWT lane.

## Concept 4 — The `__Host-` cookie line as the carrier, with each token keyed to the failure it prevents

**Why it's hard.** The student saw the cookie attribute palette in Chapter 3.4, but the auth-cookie call site applies *all* of them at once and the line — `Set-Cookie: __Host-session=<128-bit token>; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=2592000` — has to be defensible attribute-by-attribute under code review. Static prose lists six attributes; recall a week later is poor and the student's reach for `Domain=acme.com` "to share across subdomains" or `SameSite=None` "to support the marketing iframe" undoes a structural defense each time. The teaching has to chunk the line into one memorable artifact and tie every token to a specific browser-enforced refusal.

**Ideal teaching artifact.** Reference archetype rendered as a **clickable-token reveal of the senior auth-cookie line**, mirroring 3.4 Concept 2's shape so the visual vocabulary is recognized on sight. The student sees the canonical line as a single `Set-Cookie` header where every attribute is a clickable token. Clicking `__Host-` opens "the browser refuses `Domain=` and refuses to set the cookie without `Secure` and `Path=/` — moves scope-tightness from 'the developer remembered' to 'the browser enforced.'" Clicking `HttpOnly` opens "JS cannot read it via `document.cookie` — XSS that runs arbitrary script in the page still cannot exfiltrate the session token." Clicking `SameSite=Lax` opens "withheld on cross-site `POST`s — closes the classic CSRF vector for the Server Action surface; magic-link top-level navigation still attaches because `Lax` permits safe-method top-level nav." Clicking `Max-Age` opens the lifetime trade: 30 days with sliding renewal is the senior default; "remember me" toggles the cookie's persistence. Each token's reveal names the *attack class* the attribute closes, not just what it does.

**Engagement.** A `Tokens` exercise on a *deliberately broken* `Set-Cookie` header presented without commentary — `Set-Cookie: session=abc123; Domain=acme.com; Path=/admin; SameSite=None`. The student clicks every attribute that is wrong, with feedback naming the structural defense each break undoes (`Domain=` with no `__Host-` opens subdomain planting; `Path=/admin` is not a security boundary and breaks `__Host-` rules; `SameSite=None` re-opens CSRF and requires `Secure` it didn't get; missing `HttpOnly` opens XSS exfiltration; the cookie name has no `__Host-` prefix to enforce any of the above). Confirmed by a `MultipleChoice`: "a teammate's PR adds `Domain=.acme.com` to the auth cookie 'so the marketing site can read it.' What does the senior reviewer ask for?" — the marketing site is a different application, doesn't need session access; sharing the cookie is the subdomain-takeover vector; the right answer is the marketing site signs in independently, the auth cookie stays host-locked.

**Components.**
- `Tokens` for the clickable-token reveal of the canonical line and again for the broken-header recall — the existing exercise component is the exact fit and the same shape as 3.4 Concept 2, confirming the forward-link.
- `MultipleChoice` for the PR-review confirmation.
- `Aside` (`caution`) below the second `Tokens`: "the prefix is the structural enforcement, not a style choice. Reaching for `Domain=` or `SameSite=None` undoes a browser-enforced refusal — the right move is to keep the cookie host-locked and add what the cross-surface flow actually needs at a different layer."

**Project link.** The 9.5 project's first verifiable artifact is a `__Host-session` cookie landing in the browser after sign-up; the chapter's verify lesson reads the cookie's attributes off DevTools and matches them against this canonical line.

## Concept 5 — The session row and the lifecycle: issue, refresh, revoke, expire, plus the fixation defense

**Why it's hard.** Three structural facts have to land together or the student's mental model of "session" stays as "the cookie." (1) The cookie carries an *opaque ID*; the source of truth — the session row — lives on the server with `(id, userId, expiresAt)` as the floor and a small set of load-bearing columns (`lastActiveAt`, `userAgent`, `ipAddress`, `activeOrganizationId`) that each feed a downstream feature the student will build later in the unit. (2) The lifecycle has four named transitions — *issue* on sign-in, *refresh* on activity, *revoke* on sign-out / password change / admin action, *expire* by `expiresAt` — and confusing them produces specific bugs (rotating on every request races concurrent tabs; not regenerating on sign-in opens session-fixation; deleting the cookie without deleting the row leaves a zombie). (3) The fixation defense — *regenerate the session token at sign-in, never reuse a pre-auth ID for the post-auth session* — is the one transition the student must recognize even when the library handles it, because hand-rolled auth gets it wrong by default.

**Ideal teaching artifact.** Two-beat composite. *Beat one:* a **session-row anatomy diagram** rendered as a hand-SVG inside `Figure` — the row drawn as a rectangle with each load-bearing column labeled, and an arrow off each column pointing to the downstream feature it serves (`lastActiveAt` → "*last seen 3 hours ago* in 9.4.3"; `userAgent`/`ipAddress` → "active-sessions list and anomaly detection in 9.4.3"; `activeOrganizationId` → "the multi-tenancy hook in Unit 10"; `expiresAt` → "treated as absent by every read"). The arrows make the columns earn their place — every column is *because of* a feature, not just because the schema defaults to it. *Beat two:* a **lifecycle state machine** rendered as a Mermaid `stateDiagram-v2` with four states (*pre-auth*, *active*, *expired*, *revoked*) and the labeled transitions: *pre-auth → active* on sign-in (annotated *regenerate token* — the fixation defense made structural); *active → active* on activity (annotated *update `lastActiveAt`, optionally rotate on a configurable cadence*); *active → revoked* on sign-out / password change / admin action (annotated *DELETE the row*); *active → expired* on time elapsed (annotated *no DB write needed; reads filter `WHERE expires_at > now()`*); *revoked / expired → 401* on the next request from the stale cookie. The two artifacts together answer the row-shape question and the time-evolution question — both are the lessons the student needs to read Better Auth's defaults in 9.2.3.

**Engagement.** A `Matching` exercise pairing six columns from the row diagram against the downstream feature each one feeds (`lastActiveAt` ↔ idle-timeout policy and last-seen UI; `userAgent` ↔ active-sessions list per device; `ipAddress` ↔ anomaly detection and audit; `activeOrganizationId` ↔ Unit 10's multi-tenancy reads; `expiresAt` ↔ filter on every read; `impersonatedBy` ↔ admin support reach, named-not-built). Confirmed by a `Sequence` drag-to-order of seven lifecycle events — *user types password*, *server validates credentials*, *server **regenerates** session token* (the fixation-defense beat), *server inserts row*, *server sets `__Host-session` cookie*, *user closes tab and reopens hours later*, *server reads cookie and updates `lastActiveAt`*. Wrong placements name the bug each mis-ordering ships ("setting the cookie before inserting the row leaves a window where the next request reads no session and 401s the user back to sign-in"; "skipping the regenerate step is session fixation — the pre-auth ID becomes the post-auth ID").

**Components.**
- `Figure` wrapping a hand-SVG row-anatomy diagram with arrows from columns to downstream features. Single-use within the chapter; downstream chapters (9.4.3, 10.1) will reach for their own scoped versions of the same row, but the *anatomy with annotations* style here is single-use — hand-SVG is the right scope.
- Mermaid `stateDiagram-v2` for the lifecycle state machine. Recurring shape; no new component needed.
- `Matching` for the column-to-feature pairing.
- `Sequence` for the seven-event lifecycle ordering, with the regenerate beat as the load-bearing position.
- `Aside` (`note`) below the lifecycle: "regenerate the token at sign-in. Better Auth does it; recognize the move when reviewing hand-rolled code."

**Project link.** The 9.5 project's `proxy.ts` reads the cookie and the action layer reads the session row; the chapter's verify lesson opens Postgres and matches the row against the lifecycle states the student just walked.

## Concept 6 — OAuth's name, the four roles, and OIDC as the authentication layer on top

**Why it's hard.** The student has clicked "Sign in with Google" a thousand times and parses OAuth as "the protocol that signs you in." It isn't — OAuth is *Open Authorization*, a delegated-authorization protocol whose *common use* in SaaS is authentication via the OIDC layer that adds the `id_token` and a standardized `userinfo` shape on top. Until the student separates the two layers in their head, the eight-step flow in Concept 7 reads as ceremony rather than as two protocols cooperating. The four roles — *resource owner*, *client*, *authorization server*, *resource server* — exist precisely so the spec can describe what each party does without the same word meaning three things, and the student needs the role names installed before the flow steps can use them.

**Ideal teaching artifact.** Concept archetype delivered as a **two-layer cake diagram with a worked-example cast list**, rendered as a hand-SVG inside `Figure`. The cake is drawn as two stacked layers: the bottom layer labeled *OAuth 2.1 — delegated authorization* with one-line description ("third-party app acts on the user's behalf at a resource server, with consent, without the password"); the top layer labeled *OIDC — authentication on top of OAuth* with one-line description ("`openid` scope opts in, `id_token` and `userinfo` standardize *who is the user*"). Beside the cake, a **cast-list sidebar** instantiates the four roles for the worked example *Sign in with Google to ledger.app*: *resource owner* — the human user (Ada); *client* — `ledger.app` (this SaaS); *authorization server* — `accounts.google.com` (the identity endpoint); *resource server* — Google's `userinfo` endpoint (the only resource read for a pure-login flow; for a Google-Drive-integrated app the resource server would be Drive's API). A footer line names the senior shorthand: "the team says *OAuth login* in conversation; what's underneath is *OIDC over OAuth* — OAuth gives you tokens, OIDC tells you who the user is."

**Engagement.** A `Matching` exercise pairing the four roles against five labeled boxes (the four roles plus a distractor box labeled *the user's password*, which has no slot — naming why it has no slot is the point: OAuth's delegation model exists *so* the password never travels to the client). Confirmed by a `MultipleChoice`: "a teammate explains the social-login button as 'OAuth signs the user in.' What does the senior reviewer add?" — the precise answer names OAuth as authorization, OIDC as the layer that adds authentication, the `userinfo` endpoint as where identity is read, and the senior shorthand that conflates the two in casual speech but never in design.

**Components.**
- `Figure` wrapping a hand-SVG two-layer cake with the cast-list sidebar. Single-use; the role vocabulary is installed once and re-used in prose for the rest of the chapter — hand-SVG is the right scope.
- `Matching` for the four-roles-plus-distractor pairing.
- `MultipleChoice` for the senior-shorthand confirmation.
- `Aside` (`note`) below the diagram: "OAuth is delegated authorization; OIDC is authentication on top. Every *Sign in with X* button in this stack is OIDC-shaped. The shorthand says OAuth, the design names OIDC."

**Project link.** The 9.5 project explicitly cuts OAuth — sticks to email+password — but the cake diagram is the *why* the cut is a cut and not a default; lesson 9.5.1's scope panel cites the diagram when naming the carve-out.

## Concept 7 — The eight-step authorization-code-with-PKCE flow, end to end

**Why it's hard.** The flow is the chapter's densest single artifact — eight steps spanning two protocols (OAuth + OIDC), three actors (user, app, identity provider), two cryptographic bindings (`state` for CSRF, `code_verifier`/`code_challenge` for PKCE), and one data exchange (`code` for tokens) — and the student must be able to draw it from memory by the end of the lesson. A static eight-step list reads as memorization ceremony. The teaching has to make every step *necessary* — each step closes an attack class or carries a value the next step consumes — so the student learns the flow as a chain of cause-and-effect rather than as a recipe.

**Ideal teaching artifact.** Concept + Mechanics archetype delivered as a **scrubbable three-lane sequence** in `DiagramSequence` paired with a **value-flow ledger**. The three lanes are the *user's browser*, the *app server (`ledger.app`)*, and the *Google authorization server* — drawn as three vertical columns with arrows between them. The student scrubs through eight frames, each frame highlighting the active arrow plus one short explanation: (1) **client prepares** — app generates `code_verifier` (high-entropy random) and derives `code_challenge = BASE64URL(SHA256(code_verifier))`; generates `state` (random nonce) and stashes verifier + state server-side in a pre-auth session; (2) **client redirects browser** — full-page redirect to `https://accounts.google.com/o/oauth2/v2/auth?response_type=code&client_id=...&redirect_uri=...&scope=openid+email+profile&state=...&code_challenge=...&code_challenge_method=S256`; (3) **user consents** — Google shows the consent screen, the user picks an account and approves the requested scopes; (4) **authorization server redirects back** — browser lands on the app's registered `redirect_uri` carrying `?code=AUTH_CODE&state=...`; (5) **client verifies `state`** — fetches the stashed value, rejects on mismatch (the CSRF beat); (6) **client exchanges `code` for tokens** — back-channel `POST /token` from the app server (not the browser) carrying `grant_type=authorization_code`, `code`, `redirect_uri`, `client_id`, `client_secret`, *and* `code_verifier`; Google checks the redirect-URI exact match, the client credentials, and that `SHA256(verifier) == challenge` — the PKCE beat; (7) **authorization server returns tokens** — `{ access_token, refresh_token?, id_token, expires_in, token_type, scope }`; client verifies the `id_token` signature against Google's JWKS endpoint and the `aud`/`iss`/`exp` claims; (8) **client provisions and signs in** — looks up `(provider, providerAccountId)` in the local `account` table, links or creates the user row, and issues *its own* session cookie (the 9.1.2 session — the OAuth tokens are *not* the app's session, they're the proof-of-identity input).

The **value-flow ledger** sits below the sequence as a small hand-coded HTML table with three columns — *value*, *generated by*, *consumed by* — listing the seven moving values: `code_verifier` (app, kept), `code_challenge` (app, sent in step 2), `state` (app, sent in step 2 and verified in step 5), `code` (Google, returned in step 4 and exchanged in step 6), `client_secret` (app, sent in step 6), `access_token` (Google, returned in step 7), `id_token` (Google, returned in step 7 and verified in step 7). The ledger answers the *which value goes where* question the sequence's arrows can't hold without overcrowding.

**Engagement.** A `Sequence` drag-to-order drill of the eight steps with two distractors that map to common misconceptions: *"client posts the code-for-tokens request from the browser"* (no — back-channel from the server, the `client_secret` never touches the browser) and *"client trusts the `id_token` claims without signature verification"* (no — verify against JWKS first, the JWT is only an `id_token` if its signature checked out). Wrong placements name the attack class each mis-ordering enables (verifying `state` *after* the token exchange is a window where a forged callback already had the code). Confirmed by a `MultipleChoice`: "the callback URL contains `?code=ABC&state=XYZ` — what does the app do *first*?" — verify `state` against the server-stashed value before doing anything else with the `code`.

**Components.**
- `DiagramSequence` hosting the eight-frame three-lane scrubbable sequence. Each frame is a hand-SVG inside `Figure` showing the three lanes with the active arrow highlighted and the short explanation pinned. Reuses 7.2 Concept 1's three-lane round-trip shape — confirms the forward-link.
- `Figure` wrapping a hand-coded HTML value-flow ledger (three columns: value / generated by / consumed by). Single-use, but the ledger is the artifact that answers the question the sequence can't — hand-coded HTML is the right scope.
- `Sequence` for the eight-step drag-to-order with two misconception distractors.
- `MultipleChoice` for the state-first confirmation.
- `Aside` (`note`) below the sequence: "the OAuth tokens are not the app's session. They are the proof of identity the app's session is built on top of. Step 8 is where Concept 5's session-row insert happens — the chapter's two halves meet here."

**Project link.** The 9.5 project cuts OAuth, but the eight-step flow is the *frame* lesson 9.3.8 (OAuth providers, in-chapter not in-project) builds Better Auth's `socialProviders` config against. The chapter gives the student the model that makes 9.3.8's `clientId / clientSecret / scope` config legible without re-deriving the protocol.

## Concept 8 — The 2.1 hardenings: PKCE-everywhere, `state`-mandatory, exact-match redirect URIs

**Why it's hard.** The student finished Concept 7 knowing *what* the flow does. Concept 8 closes the *why* — three hardenings OAuth 2.1 (`draft-15`, March 2026) makes mandatory versus 2.0's defaults, each closing a real attack class that the 2024–2025 incident wave made operational. The student will see the hardenings as library defaults in 9.3.8 and will feel them at config time (the registrar exact-match rule will bite when they configure Google with a trailing-slash mismatch). The teaching has to make each hardening *visceral* — show the attack the 2.0 default permits and the way the 2.1 mandate closes it — so the student recognizes the mandate as a defense rather than as red tape, and so they don't reach for "let me skip PKCE on the server-side flow because we have a client secret" the moment they hand-roll anything.

**Ideal teaching artifact.** Pattern archetype delivered as a **three-attack-defense walkthrough** in `DiagramSequence`. The student steps through three short attack scenarios, each as a two-frame before/after: the *2.0-default frame* showing the attack landing, then the *2.1-mandated frame* showing the attack closed.

(1) *Authorization-code interception* — frame A: a malicious browser extension (or a logging proxy, or a mis-deployed access log) captures the `?code=...` from the redirect URL; the attacker replays the code at the token endpoint with the (leaked or guessable) client secret; tokens land in the attacker's hands. Frame B: with PKCE mandatory, the attacker has the code but not the `code_verifier` (kept server-side at the app, never transmitted); the token endpoint computes `SHA256(verifier) == challenge` and the exchange fails. The mandate's payoff: PKCE on every flow regardless of client type, because confidential clients running in Node behind a CDN with logs and shared secrets across deployments survive the "client secret is enough" model only theoretically.

(2) *Callback CSRF without `state`* — frame A: an attacker initiates an OAuth flow on their own account, captures the `code`, then tricks the victim's browser into hitting the app's callback with the attacker's code; the app links the attacker's Google identity to the victim's app account. Frame B: with `state` mandatory and verified before the code exchange, the callback's `state` doesn't match the app's stashed value (the attacker can't write to the victim's pre-auth session) and the request is rejected. Closing rule: `state` is not optional, libraries that omit it are broken.

(3) *Open-redirect via flexible redirect-URI matching* — frame A: a 2.0-era app registers `https://app.example.com/api/auth/callback/google*` (prefix match) so any subpath works; the attacker convinces the user to start the flow with `redirect_uri=https://app.example.com/api/auth/callback/google.attacker.com`; the auth code lands on the attacker's domain. Frame B: with exact-string-match mandatory, only the registered URIs work, no prefix tricks, no query-string flexibility. The senior reflex this installs: register every environment's redirect URI explicitly (dev, staging, production), the path matters, the trailing slash matters, and never reflect untrusted input into the post-sign-in destination.

The walkthrough closes with the two grants 2.1 *removed* — implicit (`response_type=token` returned tokens in the URL fragment, never survived browser history and analytics scripts) and ROPC (sending the password to the third-party app defeats delegation) — named once with the one-line "why it died" so the student knows to recognize them in legacy code.

**Engagement.** A `Buckets` sort: ten specifications into three buckets — *2.1 mandatory*, *2.1 removed*, *unchanged from 2.0*. Examples: "PKCE on every flow" (mandatory), "implicit grant" (removed), "ROPC grant" (removed), "redirect URI exact match" (mandatory), "`state` parameter" (mandatory, was recommended in 2.0), "client credentials grant" (unchanged — service-to-service still uses it), "device code flow" (unchanged — TVs and CLIs), "authorization code grant" (unchanged — the surviving browser-side flow), "implicit grant for SPAs" (removed — SPAs use code+PKCE now), "refresh token rotation" (recommended in 2.1, library-default in 2026). Wrong-answer feedback names the attack class each mandate closes or the reason the removed grant died. Confirmed by a `MultipleChoice`: "a teammate's PR sets `redirect_uri` to `https://app.example.com/api/auth/callback/google` in dev but the production environment uses `.../google/` (trailing slash). What's the failure mode?" — exact-string-match means one of the two breaks silently; the production deploy works in staging and 500s on the first user sign-in; the fix is registering both URIs explicitly or normalizing the path before registration.

## Component proposals

**`RequestLifecycleSimulator`** — a request animation with two pickable inputs (*session: none / valid / expired*, *role: member / admin*) that steps the student through five frames (browser → proxy authn read → action `safeParse` → action authz read → response), forking the path at the two boundary points and surfacing the resulting status code (200 / 401 / 403). Inputs: the two pickers' value sets, the per-(session × role) outcome map, the five-frame template. Renders as a small simulator with controls at the top and a labeled lane diagram below.
- **Uses in this chapter:** Concept 1.
- **Forward-links:** 9.4.1 (the production-shaped two-layer proxy gate — the simulator's frame 2 *is* the gate the chapter implements), 10.2.2 (the `authedAction` wrapper — the simulator's frame 4 *is* the wrapper's authz read), 10.2.3 (`authedRoute`'s 401/403/422 status map). The same lifecycle reappears across every protected-surface chapter in Units 9–10 — clear and recurring forward-link.

The full proposal (frame-by-frame stepper, animated arrows) is a v2 if the simulator proves to compound across chapters as expected.

## Build priority

`RequestLifecycleSimulator` is the only new component proposed in this chapter and carries the highest forward-link weight in Unit 9 — every protected-surface chapter (9.4.1, 10.2.2, 10.2.3) reaches for the same lifecycle picture, and the simulator's two-knob shape is what makes the (session × role) → status mapping click instead of memorize. Build the leanest v1 (dropdowns + static lane SVG with the active fork highlighted) before this chapter ships; budget the v2 stepper interaction for after 9.4.1 confirms the second use.

The chapter's other artifacts ride on existing components — the parallel-lanes `DiagramSequence` from 7.2 carries Concepts 3, 7, and 8; the `Tokens` shape from 3.4 carries Concept 4; the lifecycle Mermaid carries Concept 5; hand-SVG inside `Figure` carries Concepts 2, 5 (anatomy diagram), 6 (cake), and 7 (value-flow ledger). The simulator earns its weight precisely because the lifecycle mapping recurs across the unit, not because this chapter alone needs it.
