# Chapter 7.5 — Route handlers and API contracts: pedagogical approach

## Concept 1 — The five triggers: route handler is a conditional reach, not the default

**Why it's hard.** The student finished 7.2–7.4 with the Server Action installed as the mutation reflex, then opens this chapter and reads "route handlers" — the immediate over-correction is to ship every endpoint as `route.ts` "for symmetry with REST," losing the action's automatic revalidation, typed call site, and progressive enhancement on every form in the codebase. The senior posture inverts the impulse: the action stays the default for in-app mutations; route handlers earn their weight only when one of *five* specific triggers fires. Without the threshold landing as a closed set with named triggers, "REST feels right" wins and the chapter's core decision is lost.

**Ideal teaching artifact.** Decision archetype delivered as a **five-trigger gate** rendered as a `Figure`-wrapped HTML decision matrix, echoing 7.4's four-trigger gate so the student recognizes the threshold-before-tool pattern across the unit. Five rows, one per trigger: *non-React callers* (mobile, partner backends, CLI, Zapier), *webhooks* (raw body for HMAC verification), *GET endpoints* (cacheable reads, autocomplete, JSON feeds), *streaming and large bodies* (>1 MB, SSE, AI streaming, file downloads), *custom HTTP semantics* (Cache-Control, ETag, conditional requests, status codes the action's Result can't express). Three columns per row: *the failure mode of staying with the action* (no HTTP endpoint exists, FormData is wrong shape for HMAC, POST-only protocol blocks GET, 1 MB cap, Result is JS-shape not HTTP-shape), *what the route handler offers that fixes it* (a public URL, raw `request.text()`, GET method, streaming Response, direct status/header control), *what stays as the action* (every in-app form, every authenticated dashboard mutation a React component invokes). At the bottom, the threshold restated as one quotable line: *if the caller is a React component on the same Next.js app, the answer is the action; past the five triggers, the route handler earns its weight.* A small forward-link callout names Hono-inside-Next as the rare reach when the API is published externally with versioning and an SDK (out of scope, named once).

**Engagement.** A `Buckets` sort: ten endpoints ("a Stripe webhook receiver", "a customer-create form on the dashboard", "a public RSS feed of blog posts", "a mobile-app-driven invoice creator", "an autocomplete dropdown for customers fetched from a Client Component", "a CSV export of invoices over 5 MB", "a delete-invoice button on a list row", "an OpenAI streaming chat response", "a settings-update form on the account page", "a partner-backend-driven org-provisioning call") into "Server Action" vs "route handler — name the trigger." Wrong-answer feedback names which trigger fires (or doesn't fire) for each.

**Components.**
- `Figure` wrapping a hand-coded HTML five-trigger decision matrix.
- `Buckets` for the ten-endpoint trigger sort.
- `Aside` (`tip`) below: "Route handler is conditional. Five triggers — non-React callers, webhooks, GETs, streaming/large bodies, custom HTTP semantics. Otherwise the Server Action."

**Project link.** The 7.6 project ships entirely on Server Actions because none of the five triggers fire. This concept is the reason the project doesn't reach for `route.ts`, and the trigger table is the reviewer's checklist for any future endpoint added to the codebase.

## Concept 2 — `route.ts` shape and the dynamic-by-default caching posture

**Why it's hard.** Two structural facts about route handlers collapse into one beat the student stitches wrong on first read. (1) The file convention is opinionated — a file named `route.ts` exports HTTP-method-named functions (`GET`, `POST`, etc.), receives `NextRequest` plus `{ params }` (now a `Promise` in Next.js 16, mirror of the page convention), and *cannot coexist with `page.tsx` at the same segment*. The student who tries to put both in `app/api/invoices/[id]/` discovers the conflict at build, the student who writes one fat handler with a manual method switch loses the framework's auto-dispatch and the auto-`OPTIONS`. (2) The caching default in Next.js 15/16 is *dynamic* — no automatic full-route cache — and most route handlers serve authenticated requests where caching is wrong by default; opt into caching deliberately at the endpoint where it's correct, never globally. The senior anchor: the file convention is the framework's contract, and caching is opt-in at the handler that earns it.

**Ideal teaching artifact.** Mechanics archetype delivered as a **stencil-and-fill anatomy walkthrough** in `AnnotatedCode`, paired with a **conflict-and-caching contest** in `CodeVariants`. The `AnnotatedCode` walks one canonical handler — `app/api/invoices/[invoiceId]/route.ts` exporting `GET` and `PATCH` — with six labeled annotations: (1) the file path convention (`route.ts`, not `page.tsx`); (2) the HTTP-method-named exports as the framework's dispatch table; (3) the `(request: NextRequest, { params }: { params: Promise<{ invoiceId: string }> })` signature with the `await params` line tagged as the Next.js 16 break the student must remember; (4) the auto-`OPTIONS` and auto-`405` from the methods exported (no hand-written method switch, no hand-written `Allow` header); (5) the dynamic-by-default caching — no `'use cache'`, no `Cache-Control` header — annotated as "this is correct for authenticated handlers"; (6) the closing return as a `Response` or `NextResponse.json(...)`, not the action's `Result`. Below the walk, a `CodeVariants` with two tabs: tab one ships `app/dashboard/route.ts` next to `app/dashboard/page.tsx` — the student sees the build error from the segment conflict; tab two ships the same handler with `Cache-Control: public, s-maxage=60` set on a per-user response — the senior reviewer's red flag is annotated alongside ("the CDN caches and serves another user's body — the privacy bug bites in production"). Below both tabs, the rule: *one segment, one role; opt into caching at the endpoint that earns it, never globally.*

**Engagement.** A `Sequence` drag-to-order drill on the canonical handler skeleton: the student orders six fragments — the `export async function GET`, the `await params`, the `safeParse` of params, the database read, the `NextResponse.json` return, plus a distractor (a manual `if (request.method === 'POST')` switch). The distractor has no correct slot — feedback names "the framework dispatches by method-named exports; the `if` competes with the platform." Confirm with a `MultipleChoice`: "a teammate's PR adds `Cache-Control: public, s-maxage=60` to a route handler returning user-specific data. What does the senior reviewer ask for?" — right answer names *remove the header; per-user data must not be CDN-cached*; wrong answers name symptom mitigations.

**Components.**
- `AnnotatedCode` walking the six-annotation `route.ts` anatomy with the Next.js 16 `await params` break tagged.
- `CodeVariants` with two tabs for the segment-conflict and per-user-cache contests.
- `Sequence` for the six-fragment drag-to-order drill with the manual method-switch distractor.
- `MultipleChoice` for the cache-on-private-data PR-review.
- `Aside` (`caution`) below: "`route.ts` and `page.tsx` cannot coexist at the same segment. Caching is opt-in at the handler that earns it; the default is dynamic, and most authenticated handlers stay there."

**Project link.** The 7.6 project ships no route handlers, but the file-vs-page conflict and the caching-by-default rule are the preconditions for any future endpoint added to the codebase. The handler skeleton installed here is the spine the chapter's later concepts (parse order, status codes, list endpoints) extend.

## Concept 3 — The four input sources, parsed cheapest-first

**Why it's hard.** The action had one input source — `formData` — and the parse was a one-liner. The route handler has *four* (path params, query, headers, body) and the student who treats them as one block — "parse everything at the top" — pays for it three ways: a malformed UUID in the path triggers a body read before the request is rejected; a missing `Idempotency-Key` header is discovered after the database is touched; a 500 is returned on a request that should have produced a 400 in the first ten lines. The senior posture inverts the impulse: parse cheapest-first, fail fast on the cheapest disqualifier, the handler's first ten lines are four `safeParse` calls and four short-circuits, no business logic until everything parses. The order is structural — path params (cheapest, format check), headers (auth/content-type/idempotency), query, body — and the discipline has to land at the keyboard, not as a paragraph.

**Ideal teaching artifact.** Pattern archetype delivered as a **scrubbable parse-pipeline animation** in `DiagramSequence` paired with a **stencil-and-fill canonical handler** in `AnnotatedCode`. The animation has five frames the student steps through: (1) a malformed-UUID request hits `/api/invoices/not-a-uuid?status=sent` with a 2 MB JSON body; the path-params parse fires first, fails on the `z.uuid()` check, the handler short-circuits with 400 — *the body is never read, the database is never touched*, the cost lane shows zero downstream work; (2) a request with a valid UUID but a missing `Idempotency-Key` header on a side-effecting POST; the headers parse fires second, fails the required header check, short-circuits with 400 *before the body is read*; (3) a request with valid path and headers but `?limit=999999` in the query; the query parse fires third, the schema's `.max(100)` rejects with 400, body still not read; (4) a request with valid path/headers/query but a body whose `email` field is missing; the body parse fires fourth, returns 422 with Problem Details (the next concept's payload); (5) the happy path — all four parses pass, the business logic runs. Across every frame, the cost lane on the right shows what was avoided by short-circuiting. The paired `AnnotatedCode` walks one canonical handler body with the four parses laid out in order, each annotated with what it gates: format check, auth/content-type, query bounds, body shape. The closing line names the principle: *the handler's first ten lines are four `safeParse` calls and four short-circuits; everything below depends on parsed inputs.*

**Engagement.** A `Sequence` drag-to-order drill on a shuffled handler body — the student orders the four parses (path params, headers, query, body) plus the database call and the response. Wrong placements produce targeted feedback ("parsing the body before the path UUID makes a malformed-UUID request read 2 MB of bytes for nothing", "checking the database before the auth header runs a query for an unauthenticated request"). Confirm with a `MultipleChoice`: "a teammate's handler returns 500 on a request with a malformed path UUID — the database fails on the bad ID. What did the handler skip?" — right answer names *the path-params `safeParse` at the top, return 400 before the database touch*; wrong answers name database-side mitigations.

**Components.**
- `DiagramSequence` hosting the five-frame parse-pipeline animation; each frame is a hand SVG inside `Figure` showing the request, the four parse stages, and the cost lane on the right.
- `AnnotatedCode` walking the canonical four-parse handler body with each parse's gating role annotated.
- `Sequence` for the shuffled-handler drag-to-order drill.
- `MultipleChoice` for the skipped-parse confirmation.
- `Aside` (`tip`) below: "Path params, headers, query, body — cheapest-first. Fail fast on the cheapest disqualifier; no business logic until everything parses."

**Project link.** No direct cash-in inside 7.6 (the project is action-only), but the four-parse discipline is the spine the chapter's later concepts (status codes, idempotency, list endpoints) build on, and the pattern lands directly in 12.1 (webhook receiver — header parse before body parse) and 16's TanStack-Query-driven endpoints.

## Concept 4 — Request and response as Zod schemas: the wire contract is the schema

**Why it's hard.** The student arrives carrying the action's contract — *the schema's input type, the canonical Result* — and reaches for the wire boundary expecting symmetry, then discovers the action's `Result` is a *JavaScript* shape the React form layer reads, while the route handler returns *HTTP* — a status code plus a JSON body, no implicit shape, no typed call site. The lazy reach is to type the response by hand and let the handler return whatever feels right; the failure mode is the handler that drifts from its declared response type one PR at a time, and the OpenAPI generator (when the API leaves the codebase) can't be honest because the schema doesn't describe what the handler returns. The senior reach is structural: one Zod schema per response shape, validated on the way out for public APIs, typed for internal-only handlers, the schema is the contract regardless of who's reading. The discipline mirrors 7.1's source-of-truth posture, lifted to the wire boundary.

**Ideal teaching artifact.** Pattern archetype delivered as a **bidirectional contract diagram** rendered as a `Figure`-wrapped hand SVG, paired with a **before-and-after refactor contest** in `CodeVariants`. The diagram (visual vocabulary of 7.1 Concept 14's source-of-truth chain) shows one box at the root labeled `/lib/schemas/invoice.ts` exporting four named schemas — `ParamsSchema`, `QuerySchema`, `BodySchema`, `InvoiceResponseSchema` — with two arrow tracks. *Inbound* track: the four input schemas feed the handler's four `safeParse` calls (Concept 3's parse order). *Outbound* track: the response schema feeds two consumers — the handler's `NextResponse.json(InvoiceResponseSchema.parse(data))` (validation on the way out, public APIs) and the OpenAPI generator (named once, out of scope). A loop closes back: the same schemas feed the external client's typed SDK via `z.infer`, so the wire types on both sides originate from one declaration. The contest in `CodeVariants` shows one handler authored two ways. Tab one (the drift trap): the handler returns `NextResponse.json({ id, total, status, ...whateverElseTheServiceComputed })` with a hand-typed response in a separate file the docs page references — the docs say `total: number`, the handler ships `total: string` on the day someone refactors the service, the external client breaks silently. Tab two (the senior reach): the handler returns `NextResponse.json(InvoiceResponseSchema.parse(data))` — the response is *validated on the way out*, a shape mismatch fails the handler at runtime in dev (and triggers a 500 in production with a logged correlation ID), the schema is the contract, the OpenAPI generator picks it up unchanged. Below the contest, a small reference catalog matches the four input schemas to their parse source: `Params` ← `await params`; `Query` ← `request.nextUrl.searchParams`; `Headers` ← `request.headers.get(...)`; `Body` ← `await request.json()` / `request.text()` / `request.formData()`.

**Engagement.** A `MultipleChoice` PR-review round of four scenarios: "a teammate's PR adds a new field to the response and updates the docs page by hand to match" (revert — the schema is the docs source; update `InvoiceResponseSchema` and let the doc generator regenerate); "a teammate's handler returns `NextResponse.json(invoice)` with no schema parse on the way out, on a public API endpoint" (the handler can drift from the declared response — wrap with `InvoiceResponseSchema.parse(...)`); "a teammate declares the request body schema in the handler file as a private const" (the schema lives in `/lib/schemas` so both the action and the handler import the same one — mirror of 7.4.3); "a teammate uses `z.infer<typeof InvoiceResponseSchema>` for the external client's TypeScript SDK type" (the right reach — one source, two sides). Each wrong answer names a real drift class.

**Components.**
- `Figure` wrapping a hand SVG of the bidirectional contract diagram (one schema box, four inbound arrows from the request, two outbound arrows to the response and the OpenAPI generator).
- `CodeVariants` with two tabs for the drift-vs-validated-response contest.
- `MultipleChoice` for the four-scenario PR-review round.
- `Aside` (`note`) below: "One schema per input source, one schema per response shape, validated on the way out for public APIs. The schema is the contract; the OpenAPI generator picks it up unchanged."

**Project link.** Reuses the source-of-truth chain visual vocabulary from 7.1 Concept 14 and 7.4 Concept 7 — the chain has one more pair of nodes (the response schema and its OpenAPI consumer) feeding off the root. The 7.6 project's actions parse with the same schemas; the handler concept is the extension surface for any future API endpoint added to the codebase.

## Concept 5 — RFC 9457 Problem Details: one error envelope, one helper, every handler

**Why it's hard.** The student reads "return an error response" and reaches for whatever shape feels right at the call site — `{ error: 'invalid' }`, a string, a 200 with `{ ok: false }`, a 4xx with the raw Zod issue tree. By the third handler, the codebase has four error shapes, the client can't write one error renderer, and the form-mapping helper from 7.4.3 (which expects `fieldErrors`) breaks the moment a route handler is the caller. The senior reach is structural and named: every error response is `application/problem+json` with the RFC 9457 fields (`type`, `title`, `status`, `detail`, `instance`) plus the `errors` extension carrying `fieldErrors` for 422 validation failures, returned by *one* `problem(status, code, options?)` helper in `/lib/api`. The shape is consistent across handlers; clients write one renderer; the form-mapping helper from 7.4.3 still works because the field-error shape matches.

**Ideal teaching artifact.** Pattern archetype delivered as a **same-error-three-shapes contest** in `CodeVariants` paired with a **canonical helper walkthrough** in `AnnotatedCode`. The contest shows one error case (a 422 validation failure on a missing `email` field) returned three ways. Tab one (the ad-hoc): `return NextResponse.json({ error: 'Invalid input' }, { status: 422 })` — the client gets a string, doesn't know which field, can't render inline; the form-mapping helper from 7.4.3 reads `undefined` for `fieldErrors` and renders the generic banner instead of the inline error; the failure mode is annotated alongside. Tab two (the leaky): `return NextResponse.json(parsed.error.issues, { status: 422 })` — the client gets the raw Zod tree, has to write a parser per handler, the shape changes when Zod ships v5; the dependency leak is annotated. Tab three (the senior shape): `return problem(422, 'validation_failed', { fieldErrors: z.treeifyError(parsed.error).properties })` — the response is `application/problem+json`, body is `{ type: 'https://example.com/problems/validation-failed', title: 'Validation failed', status: 422, detail: '...', instance: '/api/invoices', errors: { fieldErrors: { email: ['Required'] } } }`; the client renders inline; the form-mapping helper reads `body.errors.fieldErrors.email[0]` and the field-error renderer from 7.4.3 / 7.6 lights up. The paired `AnnotatedCode` walks the `problem(status, code, options?)` helper body — a five-line function in `/lib/api/problem.ts` that maps the internal `code` to a stable `type` URI (the docs page anchors), pulls the matching `title` from a small map, sets `status` and `instance` from the request, and merges in `options` (like `fieldErrors`) as extensions. Five annotations: the `application/problem+json` content type as the wire signal; the `type` URI as the stable error identifier (one URI per error class, never reused); the `detail` field as the human-readable instance-specific message; the `errors.fieldErrors` extension matching the `Result.error.fieldErrors` shape from 7.2 and 7.4.3; the `correlationId` extension on 500s for the support team to look up in observability.

**Engagement.** A `MultipleChoice` round of four error scenarios: "a 422 validation failure on a missing email" (Problem with `errors.fieldErrors`); "a 409 unique-constraint conflict on an invoice slug" (Problem with `code: 'conflict'`, no `fieldErrors`, the `detail` carries the conflict reason); "a 500 unhandled exception" (Problem body that does *not* leak the stack trace; the trace goes to the observability sink, the body carries a `correlationId`); "a 401 from the auth wrapper before any handler logic ran" (Problem with `code: 'unauthenticated'`, generic `detail`, no field info). Each wrong answer surfaces a real production leak (raw stack trace, ad-hoc shape, fields omitted) the helper prevents.

**Components.**
- `CodeVariants` with three tabs for the ad-hoc / leaky / senior-shape contest.
- `AnnotatedCode` walking the five-annotation `problem` helper body.
- `MultipleChoice` for the four-scenario error round.
- `Aside` (`caution`) below: "One Problem helper, every error response, RFC 9457 wire format. The `type` URI is the error identifier; the `errors.fieldErrors` extension matches the action's `Result` shape so one renderer covers both."

**Project link.** No direct cash-in inside 7.6 (the project's actions return `Result`, not Problem responses), but the field-error shape is intentionally aligned: the same `<FieldError>` reader the project ships in 7.6.4 lights up unchanged when a future route handler returns Problem with `fieldErrors`. The pattern installed here forward-links into 12.1 (webhook signature failures returned as Problem 401), 11.2.3 (optimistic-concurrency 409s with Problem detail), and 17.2 (rate-limit 429s with the `Retry-After` field).

## Concept 6 — One operation, two callers: the shared mutator and the shared `where`-builder

**Why it's hard.** The chapter installs two architectural moments where the same operation has both an action caller and a route-handler caller, and the student who treats them as separate code paths ships duplicated logic that drifts on the next column change. The first case: a `createInvoice` operation that needs both a form-driven Server Action *and* a public API endpoint — the lazy reach is to copy-paste the action body into the handler with the parse swapped from `FormData` to `JSON`; the senior reach is one shared pure mutator in `/lib/invoices/create.ts`, two thin wrappers (action parses `FormData`, handler parses JSON body, both call the mutator). The second case: a list query that the route handler serves to external callers *and* the in-app Server Component reads directly from the database (Chapter 11.1) — the lazy reach is to build the SQL `where` clause inline in each surface; the senior reach is one shared pure `where`-builder in `/lib/invoices/list-query.ts`, both surfaces import it. Both cases are Architectural Principle #3 made operational: the seam is the wire format (FormData vs JSON, handler vs Server Component), the business logic and the query shape are shared. The student who hasn't installed the pattern at the keyboard ships drift the day the schema gains a field or the filter set grows.

**Ideal teaching artifact.** Architectural pattern archetype delivered in two paired beats — the shared-mutator and the shared-`where`-builder are one principle with two different surfaces; teaching them as one beat conflates the wire format with the query shape and the student misses the second.

**Beat one — the shared-mutator three-design contest** in `CodeVariants` showing the same `createInvoice` flow three ways. Tab one (everything duplicated): the Server Action body and the route handler body are nearly identical (parse, authorize, mutate, revalidate/respond), with the field-restatement count rendered as a badge: "47 lines duplicated across two files." Tab two (the wrapper trap): a generic `safeMutation(schema, fn)` wraps both surfaces with a custom DSL — the duplication drops but Architectural Principle #5 is violated, the action no longer reads as a React 19 action, the handler no longer reads as HTTP. Tab three (the senior reach): one pure mutator in `/lib/invoices/create.ts` — `createInvoice(input: CreateInvoiceInput, ctx: RequestContext): Promise<Invoice>` — called by both surfaces; the action parses `FormData`, calls the mutator, returns `Result`; the handler parses JSON, calls the mutator, returns `Response` (or Problem on failure). The five seams (parse, authorize, mutate, revalidate, return) stay visible at each call site; only the *mutate* seam is shared. Annotated as "Principle #3 honored, Principle #5 honored — the seam is the wire format, the business logic is the shared pure function." Below the three tabs, a small `Figure`-wrapped directory tree showing the canonical layout: `/app/.../actions.ts` (thin action), `/app/api/.../route.ts` (thin handler), `/lib/invoices/create.ts` (shared mutator).

**Beat two — the shared `where`-builder dual-consumer diagram** rendered as a `Figure`-wrapped hand SVG, paired with a small `AnnotatedCode` of the builder. The diagram shows one box at the root labeled `/lib/invoices/list-query.ts` exporting `buildInvoicesWhere(parsed: ListInvoicesQuery, ctx: { orgId: string })` returning a Drizzle condition tree — with two arrow tracks. *Track one*: the route handler at `app/api/invoices/route.ts` parses `searchParams` against `ListInvoicesQuery`, calls `buildInvoicesWhere(parsed, ctx)`, runs the query, returns the `{ data, pageInfo }` envelope. *Track two*: the in-app Server Component at `app/(app)/invoices/page.tsx` reads `searchParams` directly (Chapter 11.1's URL-state pattern), parses it against the same `ListInvoicesQuery`, calls the same `buildInvoicesWhere(parsed, ctx)`, runs the query, renders the table. ## Concept 7 — Method by intent: the method is documentation, not transport

**Why it's hard.** The student reaches for POST for everything because the spec is loose enough to permit it and `fetch('/api/invoices/123', { method: 'POST', body: JSON.stringify({ status: 'cancelled' }) })` ships and works. The cost is invisible until the second reviewer reads the codebase: nobody can scan the routes and predict which mutations are idempotent, which are safe to retry, which are creates vs updates vs cancels — the method is supposed to be the first signal of intent, and POST-for-everything erases the signal. The senior posture is structural: methods are documentation. GET is safe and cacheable, POST creates or invokes non-idempotent operations, PUT is idempotent full replacement, PATCH is partial update, DELETE is idempotent removal. The 2026 reviewer rejects `POST /invoices/[id]/status` carrying `{ status: 'cancelled' }` as a PATCH wearing a POST; accepts `POST /invoices/[id]/cancel` because *cancel* is itself a non-idempotent intent (sends an email, fires a webhook). The decision is the operation's *intent*, not the URL's literal shape.

**Ideal teaching artifact.** Decision archetype delivered as a **intent-to-method decision matrix** rendered as a `Figure`-wrapped HTML table, paired with a **same-operation-three-methods contest** in `CodeVariants`. The matrix has five rows, one per method, with three columns: *intent* (read / create-without-known-id / non-idempotent action / idempotent full replace / partial update / idempotent remove), *idempotency* (yes / no / yes / yes-if-the-change-is-itself-idempotent / yes), *canonical example from a SaaS surface* (`GET /invoices?status=sent`, `POST /invoices`, `POST /invoices/[id]/cancel`, `PUT /users/me/profile`, `PATCH /invoices/[id]` with `{ status: 'sent' }`, `DELETE /invoices/[id]`). The contest in `CodeVariants` shows three ways to express "set this invoice's status to cancelled." Tab one: `POST /invoices/[id]/status` carrying `{ status: 'cancelled' }` — annotated as "PATCH wearing POST; the reviewer rejects." Tab two: `PATCH /invoices/[id]` carrying `{ status: 'cancelled' }` — annotated as "correct for the *update* intent; idempotent because setting a status to a fixed value is itself idempotent." Tab three: `POST /invoices/[id]/cancel` with no body — annotated as "correct for the *cancel action* intent because cancel triggers an email and a webhook (non-idempotent side effects); the action endpoint is the senior reach when the operation is more than 'set a field.'" The two correct shapes are presented as *both legitimate*, distinguished by whether the operation is a field update (PATCH) or a non-idempotent action (POST to an action endpoint). Below the contest, the rule: *the method names the intent; if the intent is "do this thing with side effects," POST to an action endpoint; if the intent is "set these fields," PATCH the resource.*

## Concept 8 — The status code table the reviewer enforces

**Why it's hard.** Status codes are the protocol's failure-routing channel and the senior reviewer reads them first on every PR — but the student arrives with a binary mental model ("200 if it works, 4xx if it doesn't, 500 if it crashed") and ships handlers where every error is 400, every success is 200, every auth failure is 401. The cost is structural: clients can't programmatically branch on outcome (a `409` conflict and a `422` validation failure both return as `400`, the form layer can't tell which to render); the protocol's vocabulary is wasted; the reviewer has to read the body to understand the failure. Three lines are load-bearing in the 2026 senior subset: *400 is "the wire payload is malformed" and 422 is "the wire payload parsed but failed validation"* (the schema's `safeParse` failure returns 422, never 400); *401 is "no valid credentials," 403 is "credentials but no permission," and 404 is "the resource doesn't exist or isn't yours" — tenant-scoped resources return 404 over 403 to avoid leaking existence*; *409 is the conflict line — duplicate unique key, optimistic-concurrency mismatch, re-invite where one is pending*. Without the table as a reviewer-enforceable reference, every handler reinvents the failure routing.

**Ideal teaching artifact.** Reference archetype delivered as a **trigger-driven status-code catalog** rendered as a `Figure`-wrapped HTML table, paired with **two misconception-first ambushes** in `DiagramSequence` for the two lines the senior reviewer enforces hardest. The catalog has fifteen rows covering the 2026 senior subset (200, 201, 202, 204, 303, 400, 401, 403, 404, 409, 413, 415, 422, 429, 500), three columns each: *status* (the code and the official name), *trigger* (the precise condition that earns it — "the wire payload is malformed JSON", "the JSON parsed but the schema rejected the shape", "the caller is identified but lacks permission"), *canonical body* (a one-line reference to the Problem helper from Concept 5 — `problem(422, 'validation_failed', { fieldErrors })`, etc.). The catalog reads top-to-bottom as the reviewer's checklist; the SaaS-relevant cuts are emphasized (200 vs. 201 vs. 204 in three rows; 400 vs. 422 in two rows side by side; 401 vs. 403 vs. 404 in three rows side by side). Below the catalog, **two ambushes** as `DiagramSequence` frames. *Ambush one — 422 vs. 400.* Two frames: a request with truncated JSON ("`{ "email": "a@b.com`" — no closing brace) hits a handler that wraps the whole body in a `try { JSON.parse } catch { return 400 }` and the schema parse never runs; a request with valid JSON but missing the required `email` field hits the same handler, the JSON parses, the schema's `safeParse` fails, the handler returns 422 with `fieldErrors` — the diff is the line: *400 if the bytes don't parse as JSON; 422 if the schema rejects the shape*. *Ambush two — 404 over 403.* Two frames: a request for `/api/invoices/abc-123` where `abc-123` belongs to another tenant returns 403 — the diagram shows the attacker probing the URL space, distinguishing 403 (resource exists, you're not allowed) from 404 (resource doesn't exist), enumerating the org's invoice IDs by status-code timing; the same request returns 404 because the tenant-scoped query returned no row — the diagram shows the attacker getting the same 404 for both "doesn't exist" and "exists but isn't yours," the existence leak closed. The closing line lands the rule: *prefer 404 over 403 for tenant-scoped resources; the `tenantDb` helper from 10.4.5 enforces the scope structurally so the handler doesn't have to choose.*

**Engagement.** A `Buckets` sort: twelve failure scenarios ("the JSON body is truncated", "the body is valid JSON but the email field is missing", "the request has no `Authorization` header", "the request has a valid token but the user lacks the `admin` role", "the request asks for an invoice that belongs to another org", "the request asks for an invoice that doesn't exist anywhere", "the request POSTs an invoice number that already exists in the org", "the request body is 5 MB on an endpoint that caps at 1 MB", "the request sends `Content-Type: text/plain` to a JSON endpoint", "the request hits the rate limit", "the handler caught an unexpected exception in the database call", "the request POSTs a status update that violates the optimistic-concurrency `version`") sorted into the status-code buckets (400, 401, 403, 404, 409, 413, 415, 422, 429, 500). Wrong-answer feedback names the line the wrong status crosses (`400` for `422` loses the `fieldErrors` mapping; `403` for `404` leaks existence; `500` for `409` masks a recoverable conflict). Confirm with a `MultipleChoice` PR-review: "a teammate's handler returns 200 with `{ ok: false, error: 'Email already taken' }` on a unique-constraint violation. What does the senior reviewer ask for?" — right answer names *return 409 with a Problem body; clients that read only the status get the wrong picture from a 200*.

**Components.**
- `Figure` wrapping a hand-coded HTML status-code reference catalog.
- `DiagramSequence` hosting ambush one (the 422-vs-400 byte-vs-shape distinction).
- `DiagramSequence` hosting ambush two (the 404-over-403 existence-leak walk-through).
- `Buckets` for the twelve-scenario sort.
- `MultipleChoice` for the 200-with-error-body PR-review.
- `Aside` (`caution`) below: "The status code is the protocol's failure channel; the reviewer reads it first. 422 for schema failures, 404 over 403 on tenant-scoped resources, 409 for conflicts, 500 hides the trace."

## Concept 9 — `Idempotency-Key` operationalized: the dedup row, the claim primitive, the lifetime

**Why it's hard.** The student reads "every POST that has a side effect accepts `Idempotency-Key`" and mentally checks the box without internalizing the *operation*: what the client sends, what the handler does with it, what the database stores, what happens on replay, when the row expires. The naive reach is to read the header and call it a day — the first careless retry from a flaky mobile network produces two charges; a slightly-less-naive reach is to check `if (await db.select().where(eq(processed_requests.key, key)))` then insert, which is a TOCTOU race that lets two concurrent requests both pass the check and both insert. The senior reach is structural: the `INSERT ... ON CONFLICT DO NOTHING` claim primitive on a `(key, route, tenant)` composite uniqueness constraint — the database is the arbiter, only one inserter wins, the loser reads the cached response from the dedup row. The lifetime is tunable (24 hours by default, the Stripe convention). The teaching has to land the *flow* — header → claim → dispatch (winner runs the operation, loser returns the cached response) → cache-the-response — as a kinetic moment, not as a rule.

**Ideal teaching artifact.** Mechanics archetype delivered as a **two-client race simulator** in `DiagramSequence` paired with a **canonical claim helper walkthrough** in `AnnotatedCode`. The simulator has six frames the student steps through: (1) **client A submits** — generates `Idempotency-Key: 8f2a-...`, POSTs to `/api/invoices` with the body; the handler hashes `(key, route, tenant)`, prepares the dedup row; (2) **client A claims** — `INSERT INTO processed_requests (key_hash, response, status) VALUES (..., NULL, 'pending') ON CONFLICT DO NOTHING RETURNING id` succeeds, the handler runs the operation (charges the card, sends the email, fires the webhook); (3) **client A persists the response** — the handler `UPDATE`s the dedup row with the response body and the final status code; the client gets `201 Created` with `Location: /api/invoices/[id]`; (4) **the user double-clicks** — client B submits with the *same* `Idempotency-Key: 8f2a-...` (a network retry, an accidental double-click); the handler hashes again, attempts the same `INSERT ... ON CONFLICT DO NOTHING`, returns *zero rows* this time (the unique constraint fired); (5) **client B reads the cached response** — the handler does `SELECT response, status FROM processed_requests WHERE key_hash = ...`, returns the same `201` with the same `Location` and the same body; *one charge, one email, one webhook*, two HTTP responses; (6) **24 hours later** — the dedup row's `expires_at` passes, the cleanup job (Chapter 13.1's territory) deletes the row; a replay with the same key produces a *new* operation, named here as the lifetime contract. Across every frame, a side-counter shows "charges fired: 1, emails sent: 1, webhooks: 1" — the success criterion the operationalization protects. The paired `AnnotatedCode` walks the `withIdempotency(key, route, tenant, fn)` helper body — a six-line wrapper that does the hash, the claim, the dispatch (run the operation if winner, return cached response if loser), the persist on success — with five annotations: the composite hash as the lookup key (so `key=8f2a` for `/api/invoices` and `key=8f2a` for `/api/payments` are different rows); the `INSERT ... ON CONFLICT DO NOTHING` as the atomic claim primitive (the database is the arbiter, no TOCTOU); the dispatch branching on whether the insert returned a row; the response persistence inside the *same* transaction as the operation (so a rolled-back operation also rolls back the claim — the dedup row must reflect what actually happened); the `expires_at` set to `now() + interval '24 hours'`. The closing line names the rule: *every public POST with a side effect accepts `Idempotency-Key`; the database's unique constraint is the arbiter; the row caches the response for 24 hours.*

**Engagement.** A `Sequence` drag-to-order drill on the dedup-claim flow: the student orders six steps (`hash key + route + tenant`, `INSERT ... ON CONFLICT DO NOTHING`, `if returned zero rows: SELECT cached response and return`, `else: run the operation`, `UPDATE processed_requests SET response = ...`, `return response`) plus two distractors (`SELECT first to check existence` — the TOCTOU trap; `INSERT after the operation completes` — the second-request-runs-the-op-too trap). Wrong placements produce targeted feedback (the SELECT-then-INSERT race is named explicitly; the late-insert lets the second concurrent request also run the operation). Confirm with a `MultipleChoice`: "a teammate's PR adds idempotency by checking `if (await db.select(...))` then inserting if no row exists. What does the senior reviewer ask for?" — right answer names *use `INSERT ... ON CONFLICT DO NOTHING` so the database is the arbiter; the SELECT-then-INSERT pattern is a TOCTOU race*; wrong answers name symptom mitigations (locking, retries with delay).

**Components.**
- `DiagramSequence` hosting the six-frame two-client race simulator; each frame is a hand SVG inside `Figure` showing the client lane, the handler, the dedup row's state, and the side-counter for charges/emails/webhooks fired.
- `AnnotatedCode` walking the `withIdempotency` helper with five annotations.
- `Sequence` for the six-step drag-to-order drill with the TOCTOU and late-insert distractors.
- `MultipleChoice` for the SELECT-then-INSERT PR-review.
- `Aside` (`caution`) below the simulator: "The database's unique constraint is the arbiter. `INSERT ... ON CONFLICT DO NOTHING` claims atomically; the SELECT-then-INSERT pattern is a TOCTOU race that lets two concurrent requests both pass the check."

**Project link.** No direct cash-in inside 7.6 (the project's `useOptimistic` UUIDv7 in 7.6.5 is the form-driven equivalent foreshadow). The pattern installed here is the spine 12.1 builds on for webhook receivers — the same dedup row, the same `INSERT ... ON CONFLICT DO NOTHING` primitive, with the webhook signature verification wrapping it.

## Concept 10 — The query-schema dimensions: filter, sort, search, paginate

**Why it's hard.** A list endpoint serves a query string with four orthogonal dimensions — *filter* (which rows match), *sort* (in what order), *search* (free-text match), *paginate* (which page) — and the student who treats them as one undifferentiated input ships handlers that mix dimensions in incoherent ways: a `?status=sent` filter shipped as a comma-separated string in one handler and as a repeated key in the next, a `?sort=issuedAt_desc` ad-hoc syntax that doesn't match any team convention, a `?limit` with no ceiling that returns a million rows on the first careless request, an empty-string filter that matches "no value" instead of "not specified." The senior reach is structural and named: each dimension has a canonical schema shape — *filters as enums or arrays of enums with allowlists*, *sort as a single string parsed into `{ field, direction }` with a field-allowlist*, *search as a bounded string the SQL layer routes to ILIKE or FTS*, *paginate as `{ cursor, limit }` with `limit.max(100)` as the contract ceiling* — and the four dimensions parse together at the top of the handler, then translate into the Drizzle query in five lines. The discipline is the schema's job; the SQL composition is mechanical once the schema lands.

**Ideal teaching artifact.** Pattern archetype delivered as a **dimensions-and-conventions catalog** rendered as a `Figure`-wrapped HTML table, paired with a **stencil-and-fill canonical query schema** in `AnnotatedCode`. The catalog has four rows, one per dimension: *filter* (the canonical Zod shape `z.enum([...]).optional()` for single-value, `z.array(z.enum([...]))` for multi-value; the `?status=sent&status=overdue` repeated-key form as the senior reach over CSV; the empty-string normalization that drops `?status=` before the enum check), *sort* (the prefix-form convention `?sort=-issuedAt` parsed into `{ field, direction }` with a field-allowlist enum so arbitrary fields can't sort and scan-the-table; the comma-separated multi-sort `?sort=-issuedAt,total` as the rare reach), *search* (the bounded string `z.string().min(1).max(100).optional()`; the routing decision — ILIKE on a small searchable column, Postgres FTS on stored generated columns, external search past Postgres FTS — named once, the schema sees `q` as a string regardless), *paginate* (the `cursor` as an opaque base64-encoded string and the `limit` as `z.coerce.number().int().positive().max(100).default(20)`; the hard ceiling as the contract). Each row carries one canonical line of code per dimension. The paired `AnnotatedCode` walks one full `ListInvoicesQuery` schema being authored — `z.object({ status: z.array(z.enum(['draft', 'sent', 'paid', 'overdue'])).optional(), sort: z.string().regex(/^-?(issuedAt|total|createdAt)$/).default('-createdAt').transform(parseSort), q: z.string().min(1).max(100).optional(), cursor: z.string().optional(), limit: z.coerce.number().int().positive().max(100).default(20) })` — with five annotations: the multi-value status filter parsed via `searchParams.getAll('status')`; the sort field-allowlist as a regex over the *enum* of sortable columns (every sortable column has a covering index, Chapter 6.5); the search field's max length as the contract ceiling; the cursor's opacity declared at the schema level; the limit's default and ceiling as part of the API contract. Below the walkthrough, a small reference catalog matches each schema field to its `searchParams` parse: `getAll` for multi-value, `get` for single-value, `Object.fromEntries` only when no key repeats.

**Engagement.** A `Buckets` sort: ten query-string fragments ("`?status=sent&status=overdue`" → multi-value filter via `getAll`; "`?sort=-issuedAt`" → prefix-form sort, descending; "`?q=acme`" → search; "`?cursor=eyJpZCI6...`" → opaque cursor; "`?limit=999999`" → rejected by `.max(100)`; "`?status=`" → empty-string normalization drops the filter; "`?sort=password`" → rejected by the field-allowlist; "`?status=sent,overdue`" → CSV form, not the canonical reach; "`?withCount=1`" → the count-knob, named once; "`?sort=-issuedAt,total`" → comma-separated multi-sort, rare reach) into "filter / sort / search / paginate / rejected by the schema / not the canonical reach." Wrong-answer feedback names the dimension and the convention each fragment violates or honors. Confirm with a `MultipleChoice`: "a teammate's PR adds `z.coerce.number()` for the `limit` query param with no ceiling. What does the senior reviewer ask for?" — right answer names *add `.int().positive().max(100)`; without the ceiling, the first careless `?limit=999999` brings the database down*; wrong answers name app-side mitigations.

**Components.**
- `Figure` wrapping a hand-coded HTML dimensions-and-conventions catalog.
- `AnnotatedCode` walking the five-annotation `ListInvoicesQuery` schema.
- `Buckets` for the ten-fragment query-string sort.
- `MultipleChoice` for the unbounded-`limit` PR-review.
- `Aside` (`tip`) below: "Four dimensions, four canonical shapes. Filter as enum or enum array; sort as prefix-form with a field-allowlist; search as bounded string; paginate as cursor + limit-with-ceiling. The schema is the contract; the SQL composition is mechanical."

**Project link.** No direct cash-in inside 7.6 (the project ships single-row CRUD, not list queries), but the schema shape is the precondition for 11.1's URL-state list view (which reads the same query string from the in-app side) and for 16.2's TanStack-Query-driven endpoints. The shared `where`-builder from Concept 6 closes the loop.

## Concept 11 — Cursor pagination at the wire boundary: opacity, the envelope, the read-one-extra trick

**Why it's hard.** Three structural facts about wire-boundary pagination collapse into one beat the student stitches wrong on first read. (1) **The cursor is opaque** — the client treats it as a string, the server decodes the inner JSON; the senior reach is base64-encoded `{ id, sortKey }` for keyset pagination on `(sortKey, id)`, *never* exposed as URL-readable JSON. The student who lets `?cursor={"id":"abc"}` ship watches clients construct cursors by hand, the contract calcify, and the day the inner shape needs to change become the day every external client breaks. (2) **The response is an envelope, not a bare array** — `{ data: Invoice[], pageInfo: { nextCursor: string | null, hasMore: boolean } }`, even when v1 only has `data` and `nextCursor`, because future fields (total counts, applied-filter echo, search-rank metadata) land in the envelope without breaking the contract. The naked-array v1 that v2 has to break is the failure mode; envelope from day one. (3) **The `hasMore` signal comes from reading one extra row** — `limit + 1` returned from the database, the handler slices to `limit` and sets `hasMore = rows.length > limit`, the `nextCursor` is built from the last item in the sliced array; the alternative (a separate `COUNT(*)` query) scans the table on every list request and is wrong by default. The teaching has to land all three as one structural model, not three rules.

**Ideal teaching artifact.** Pattern archetype delivered as a **scrubbable cursor-walk animation** in `DiagramSequence` paired with a **canonical handler-body walkthrough** in `AnnotatedCode`. The animation has five frames the student steps through: (1) **first request, no cursor** — `GET /api/invoices?limit=20` hits the handler, the query runs with `limit=21`, returns 21 rows; the handler slices to 20, sets `hasMore = true` (because 21 > 20), builds `nextCursor` by base64-encoding `{ id: rows[19].id, sortKey: rows[19].issuedAt }`, returns `{ data: rows.slice(0, 20), pageInfo: { nextCursor: 'eyJpZCI6...', hasMore: true } }`; (2) **the client uses the cursor** — `GET /api/invoices?limit=20&cursor=eyJpZCI6...` hits the handler, the cursor decodes to `{ id, sortKey }`, the SQL `WHERE (issuedAt, id) < ($cursor.sortKey, $cursor.id)` runs (the keyset predicate from Chapter 6.6), returns the next 21 rows, the handler slices and re-builds; (3) **the last page** — the request returns only 7 rows (less than `limit + 1`); the handler sets `hasMore = false`, `nextCursor = null`, the client knows to stop; (4) **a write happens between pages** — a new invoice is inserted between the first request and the second; the diagram shows the keyset predicate skipping the new row (because `issuedAt` is later than the cursor's sortKey) — *no double-counting*, no row vanishing across pages, the senior payoff of cursor over offset; (5) **the malformed-cursor probe** — a client sends `?cursor={"id":"abc"}` (handcrafted JSON instead of base64); the handler's cursor schema fails the parse, returns 400 with a Problem body — the opacity contract held. The paired `AnnotatedCode` walks the canonical handler body in seven lines: parse the query (Concept 10's schema); decode the cursor if present; build the Drizzle `where` from `buildInvoicesWhere(parsed, ctx)` (Concept 6); execute with `limit + 1`; slice to `limit` and compute `hasMore`; build `nextCursor` from the last sliced row; return `NextResponse.json({ data, pageInfo })`. Five annotations: the `limit + 1` trick as the `hasMore` signal source; the slice as the discard of the probe row; the `nextCursor` built from the last *included* row, not the discarded probe; the envelope as the future-extension contract; the malformed-cursor return as 400 with the opacity-contract message.

**Engagement.** A `MultipleChoice` round of four scenarios: "a teammate ships a list endpoint that returns a bare array `[invoice1, invoice2, ...]` because v1 only has `data`" (revert — envelope from day one; future fields land without breaking the contract); "a teammate exposes the cursor as URL-readable JSON `?cursor={"id":"abc"}`" (revert — base64-wrap; the opacity is the contract that lets the inner shape evolve); "a teammate runs a separate `SELECT COUNT(*)` query on every list request to populate `pageInfo.hasMore`" (revert — read `limit + 1`, slice, compute `hasMore` from the row count; the COUNT scans the table); "a teammate uses offset pagination on a list that gets writes during scroll" (revert — offset double-counts and skips rows when the underlying set changes; cursor is the API default). Each wrong answer names the failure mode of the wrong reach.

**Components.**
- `DiagramSequence` hosting the five-frame cursor-walk animation; each frame is a hand SVG inside `Figure` showing the request, the SQL predicate, the rows returned, and the envelope built.
- `AnnotatedCode` walking the seven-line canonical handler body with five annotations.
- `MultipleChoice` for the four-scenario PR-review.
- `Aside` (`note`) below: "Cursor as opaque base64; envelope from day one; `limit + 1` for `hasMore`. The keyset SQL is Chapter 6.6; the wire shape is yours to design now."

**Project link.** No direct cash-in inside 7.6 (the project ships single-row CRUD), but the envelope shape is the precondition for 11.1's URL-state list view (which reads the same envelope from the route handler's caller side when TanStack Query gets involved in 16.2). The keyset SQL composition reuses Chapter 6.6's pattern directly.
