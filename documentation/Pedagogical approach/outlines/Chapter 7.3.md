# Chapter 7.3 ‚Äî Forms the platform way: pedagogical approach

## Concept 1 ‚Äî Uncontrolled by default: the form's state lives in the DOM

**Why it's hard.** The student arrives carrying the 2018 React reflex: every input is `value={state}` plus `onChange={setState}`, six fields means six `useState` calls plus a manual `preventDefault` plus a `fetch`. The reflex isn't *wrong* ‚Äî it ships forms ‚Äî it's *misaligned*: per-keystroke re-renders, per-field state, and per-form handler glue that the platform now does for free. Until the inversion lands at the fingers ‚Äî uncontrolled is the default, controlled is the conditional ‚Äî every following lesson reads as "another way to do the same thing."

**Ideal teaching artifact.** Misconception-first ambush delivered as a **two-implementation contest** in `CodeVariants` over the same six-field signup form. Tab one is the 2018 reflex: six `useState` calls, six `onChange` handlers, an `onSubmit` with `preventDefault` and a `fetch`. A small render-counter badge in the corner ticks every render ‚Äî typing "alex" into the email field racks up four renders across the form, visible. Tab two is the 2026 default: a native `<form action={signup}>`, six `<input name="..." defaultValue="...">`s, no state, no handlers. The render-counter sits at one. The fixtures pinned below both tabs submit the same payload ‚Äî both produce the same row in the action's POST body ‚Äî so the student sees the runtime is equivalent and the *cost* is the diff. Below the contest, the threshold lands as one line: *the only trigger to flip an input controlled is some other part of the UI must re-render per keystroke* (a debounced search, a dependent dropdown, a live character counter rendered elsewhere). Everything else is uncontrolled.

**Engagement.** A `Buckets` sort: ten field shapes ("a six-field signup form", "a debounced product-search input that filters the list below", "an edit-profile form with prefilled values", "a country picker that gates which states show in the next dropdown", "a notes textarea on a CRM record", "a password field that drives a strength meter rendered next to it", "a quantity input on a cart line", "an email field on a contact form", "a markdown editor with a live preview pane", "a checkbox on a settings page") into "uncontrolled" vs "controlled ‚Äî some other UI reacts per keystroke." Wrong-answer feedback names the threshold each case crosses or doesn't.

**Components.**
- `CodeVariants` with two tabs for the contest, with the render-counter badge sitting in each tab's preview pane.
- `Buckets` for the threshold sort.
- `Aside` (`tip`) below: "uncontrolled by default; reach for controlled only when another part of the UI has to re-render per keystroke."

**Project link.** Every form in 7.6 is uncontrolled ‚Äî the create form, the edit form (with `defaultValue` from the fetched invoice), the delete confirmation. The student should never see a `useState` per field in the project starter.

## Concept 2 ‚Äî `name` is the contract; FormData is the wire shape

**Why it's hard.** Three layers ‚Äî the input's `name` attribute, the `FormData` keys, the Zod schema's object keys ‚Äî must be one set of strings, and any drift between them silently drops the field. The trap compounds with non-text inputs whose semantics break the typed-domain mental model: a checkbox sends `"on"` when checked and is *absent* when unchecked (not `false`); a multi-value field needs `getAll`, not `get`; a file is a `File` instance, not a string filename. The student who writes `formData.get('archived')` for an unchecked checkbox reads `null`, "fixes" it by switching to `formData.get('archived') ?? 'off'`, and ships a form that's invisibly wrong on every untouched checkbox.

**Ideal teaching artifact.** Concept archetype delivered as a **three-lane wire-shape walk** rendered as a `Figure`-wrapped hand SVG. Top lane: the form, with seven labeled inputs covering every non-text shape ‚Äî `<input name="email">`, `<input name="archived" type="checkbox">`, three `<input name="role" type="radio" value="...">`s sharing a name, `<select name="status">`, `<textarea name="notes">`, `<input name="avatar" type="file">`, two `<input name="tags[]" type="checkbox">`s sharing a name. Middle lane: the resulting `FormData` after submit, with each entry's *exact* shape ‚Äî `email: 'alex@‚Ä¶'`, `archived: 'on'` (and a sibling annotation showing `archived` *absent* when unchecked), `role: 'admin'` (only the selected radio's value), `status: 'draft'`, `notes: '...'`, `avatar: File`, `tags[]: 'urgent'` plus `tags[]: 'overdue'` (two entries under the same key). Bottom lane: the action's first line ‚Äî `Object.fromEntries(formData)` for the flat shape, `formData.getAll('tags[]')` for the multi-value field, then `Schema.safeParse(...)`. Between lanes, two callouts: *the input's `name` is the FormData key is the schema's object key ‚Äî one set of strings*; and *unchecked checkboxes are absent, not present-with-false; multi-value fields need `getAll`*. The diagram is the picture the student returns to whenever they cross the form-action seam.

**Engagement.** A `PredictOutput` round of five expressions on a fixture form where the user checked one of two `tags[]` boxes, left the `archived` checkbox unchecked, selected the "admin" radio, didn't pick a file, and typed an email. The student predicts: `formData.get('email')` (`'alex@‚Ä¶'`), `formData.get('archived')` (`null`, not `'off'`), `formData.get('tags[]')` (`'urgent'` ‚Äî only the *last* value, the trap), `formData.getAll('tags[]')` (`['urgent']`), `formData.get('avatar')` (a `File` instance with size `0`, not `null`). Each surprise locks one quirk.

**Components.**
- `Figure` wrapping a hand SVG of the three-lane wire-shape diagram with the multi-value, checkbox, and file callouts pinned.
- `PredictOutput` for the five-expression round.
- `Aside` (`caution`) below: "the `name` attribute is the contract with the schema; missing names silently drop fields; unchecked checkboxes are *absent*; multi-value fields need `getAll`."

**Project link.** The 7.6 invoice form's `name` attributes (`number`, `total`, `dueAt`, `customerId`, `archived`) are the same strings the `drizzle-zod`-derived schema's object keys carry; the `archived` checkbox uses the absence-aware `z.preprocess` from 7.1.6.

## Concept 3 ‚Äî The `action` prop: React 19's form primitive replaces `onSubmit` + `fetch`

**Why it's hard.** The student's reach for "wire the form to the server" is `onSubmit={async (e) => { e.preventDefault(); await fetch(...) }}` ‚Äî the pattern from every form they've written for five years. The action prop reads like a regression to the 1999 HTML form's `action="/handler.php"` attribute, and the student doesn't see that React rewrote the platform's submit lifecycle to do *more* than the legacy `onSubmit` did: serialize the form to `FormData`, invoke the imported Server Action, surface pending state, reset the form on success, and fall back to a native POST when JavaScript hasn't loaded. The teaching has to make the lifecycle visible, not just state the swap.

**Ideal teaching artifact.** Concept archetype delivered as a **scrubbable submit-lifecycle animation** in `DiagramSequence`. Six frames the student steps through with the slider: (1) the form rendered ‚Äî `<form action={createInvoice}>` in a Client Component, three uncontrolled inputs, a submit button; (2) the click ‚Äî React intercepts the submit, runs the browser's constraint validation (the `required` attribute fires here, foreshadowing Concept 10); (3) the serialization ‚Äî React walks the named inputs and builds a `FormData`; (4) the call ‚Äî the Server Action's opaque-ID POST fires (the wire-crossing visualized in 7.2 Concept 1, recalled here as one frame); (5) the response ‚Äî the action's `Result` returns; if `ok` and the form is uncontrolled, React resets the inputs to their `defaultValue`s (the auto-reset is annotated as a *surprise* the student will reach for `useActionState` to override on edit forms); (6) the revalidation ‚Äî `revalidatePath` from inside the action triggers the surrounding Server Component to re-render with fresh data. Across every frame, a callout names what the legacy `onSubmit` + `fetch` would have to do manually ‚Äî `preventDefault`, build a JSON body, set the `Content-Type` header, parse the response, manage form state, refetch the list. The closing rule: the `action` prop is the senior default for every mutation form; `onSubmit` survives only for non-action submits (a third-party SDK call, an analytics ping).

A side beat names the **`formAction` per-button override** as the "two submits, one form" reach: a save-draft button and a publish button live inside the same form, each with its own `formAction={...}`, each consuming its own `Result`.

**Engagement.** A `MultipleChoice` PR-review round of four scenarios: "a teammate writes `<form onSubmit={handleSubmit}>` with `e.preventDefault()` and a `fetch` to a route handler the action could replace" (move to `<form action={action}>`); "a teammate wraps the action: `<form action={() => createInvoice(formData)}>` to add a console.log" (the wrapper breaks the `FormData` argument and PE ‚Äî pass the function reference); "a teammate writes an edit form and complains the typed values vanish on save" (auto-reset on success ‚Äî pair with `useActionState`'s `state.data`); "a teammate adds a save-draft button and a publish button as two separate forms with shared inputs" (one form, two `formAction`s). Wrong answers each name a real misuse the structure prevents.

**Components.**
- `DiagramSequence` hosting the six-frame submit-lifecycle animation. Each frame is a hand SVG inside `Figure`.
- `MultipleChoice` for the four-scenario PR-review round.
- `Aside` (`note`) below the sequence: "the `action` prop is the React 19 form primitive ‚Äî pass the function reference, not a wrapper; the legacy `onSubmit` survives only for non-action submits."

**Project link.** 7.6.4 wires the create and edit forms with `<form action={formAction}>`; the auto-reset behavior is observed on the create form (correct) and overridden on the edit form via `useActionState` (the surprise lands as a feature). The 7.6 delete is a separate form with its own action.

## Concept 4 ‚Äî Native `<form>` vs Next.js `<Form>`: the trigger names the choice

**Why it's hard.** The framework ships *two* form primitives that look interchangeable in JSX ‚Äî `<form>` and `<Form>` (capital F, imported from `next/form`) ‚Äî and the student picks the framework one because it's framework-branded, surely better. The cost is invisible: `<Form>` adds prefetching of the destination route and client-side navigation on submit, both of which only fire when the action is a *string URL* (the form GETs to `/search?q=...`). When the action is a Server Action, the destination isn't known until the action runs; the prefetching can't apply, and `<Form>` collapses to `<form>` with extra weight. The senior call is small and structural: `<form>` for mutations, `<Form>` for search-and-filter URL-action forms. Picking the wrong one isn't a runtime error ‚Äî it's a missed performance win on search forms and a misplaced reach on mutation forms.

**Ideal teaching artifact.** Decision archetype delivered as a **trigger-driven decision rule** rendered as a `Figure`-wrapped HTML table. Two rows, one per primitive. Column one: the call shape ‚Äî `<form action={serverAction}>` vs `<Form action="/search">`. Column two: what the framework adds ‚Äî *nothing extra, the React 19 default lifecycle* vs *route prefetch + client-side nav on submit*. Column three: the trigger that earns it ‚Äî *every mutation form (create, update, delete)* vs *search forms, filter forms, any form whose action is a URL string*. Column four: the failure mode of picking wrong ‚Äî *misplaced reach, no functional cost* vs *missed prefetch, slower perceived navigation on a hot path*. Below the table, a one-line rule the student can quote at code-review: *string action ‚Üí `<Form>`; function action ‚Üí `<form>`*. A small forward-link callout names 11.1 as the chapter that picks up the search-form case in full.

**Engagement.** A `MultipleChoice` round of four scenarios: "a customer-create form posting to `createCustomer`" (`<form>` ‚Äî function action); "a top-of-page search box that navigates to `/search?q=...`" (`<Form>` ‚Äî string action, prefetch wins); "a filter sidebar that updates URL search params and refetches the list" (`<Form>` ‚Äî string action, prefetch wins); "a delete button on each list row" (neither ‚Äî imperative `await action()` from an event handler, named in 7.3.2's three call shapes). Wrong-answer feedback names the trigger each case crosses.

**Components.**
- `Figure` wrapping a hand-coded HTML decision table.
- `MultipleChoice` for the four-scenario round.
- `Aside` (`tip`) below: "string action ‚Üí `<Form>`; function action ‚Üí `<form>`. Mismatching costs nothing functionally ‚Äî but on search forms, `<form>` leaves a perceptible prefetch on the table."

**Project link.** Every form in 7.6 is `<form>` (function action ‚Äî `createInvoice`, `updateInvoice`, `deleteInvoice`); the project doesn't reach for `<Form>` because the list view's URL-state filtering lives in 11.3.

## Concept 5 ‚Äî `useActionState`: three returns, one signature change

**Why it's hard.** The hook's surface is small ‚Äî three returns, two arguments ‚Äî but each return changes how the form is wired, and the action's signature *changes* under the wrapper. The student who writes `<form action={createInvoice}>` after `useActionState(createInvoice, null)` ships a form where the hook never sees the submit (the raw action runs, `state` never updates). The student who keeps the action's signature as `(formData) => ...` ships a TypeScript error and a runtime crash because the hook's wrapper passes `(prevState, formData)`. Both mistakes are surface-level but failure-fatal; the hook's mechanics have to land before the canonical form shape (Concept 6) does.

**Ideal teaching artifact.** Mechanics archetype delivered as an **anatomy walkthrough** in `AnnotatedCode`. One canonical hook call sits in the block ‚Äî `const [state, formAction, isPending] = useActionState(createInvoice, null);` ‚Äî paired with the action's signature underneath: `async function createInvoice(prevState: Result<Invoice> | null, formData: FormData): Promise<Result<Invoice>>`. Five annotation steps walk through: (1) **`state`** ‚Äî the latest `Result` from the action, or `initialState` before any submit; the form reads `state?.ok === false` to render errors and `state?.data` to read the saved entity; (2) **`formAction`** ‚Äî the *wrapped* action; this is what `<form action={...}>` receives, *not* the raw `createInvoice`; the trap is wiring the raw action and getting a form that submits but whose state never updates; (3) **`isPending`** ‚Äî true from invocation to result; drives submit-button text and `disabled`, optionally a `<fieldset disabled={isPending}>` to lock the whole form; (4) **the action's `prevState` parameter** ‚Äî the framework-bound first argument; the trap is keeping the action's signature as `(formData) => ...` and seeing a runtime type mismatch; (5) **the optional `permalink`** ‚Äî third argument, the no-JS fallback URL; named once, default omit (Concept 12 picks up the PE story). Each annotation step shows the failure mode of getting it wrong.

**Engagement.** A `Dropdowns` exercise on a small form skeleton with five blanks: the hook's three return positions (`state`, `formAction`, `isPending`), the value passed to the form's `action` prop (the wrapped one, not the raw), and the action's first parameter name (`prevState`, not `formData`). Each blank has three options ‚Äî the right answer plus the two adjacent mistakes. Wrong picks surface the failure mode each one would produce.

**Components.**
- `AnnotatedCode` walking the five-annotation hook anatomy with the action's signature pinned alongside.
- `Dropdowns` for the five-blank form skeleton.
- `Aside` (`caution`) below: "the form receives `formAction` (the wrapped one), and the action's first parameter is `prevState` (framework-bound), not `formData`. Either mistake silently breaks the wiring."

**Project link.** Every form in 7.6.4 calls `useActionState` with the wrapped `formAction` passed to the `<form>` and the action signed as `(prevState, formData)`. The project's reviewer checklist names both wires explicitly.

## Concept 6 ‚Äî The canonical form-component shape: the spine pattern

**Why it's hard.** With Chapter 7.1 (the schema), Chapter 7.2 (the action returning `Result`), and Concept 5 (the hook), the student has every piece ‚Äî but assembling them into one *readable* form component is the load-bearing move that gets repeated dozens of times across the rest of the course. Without a canonical shape, every form in the codebase grows a different layout: one renders the form-level error above the inputs, another renders it inside the submit button's tooltip, a third forgets to render it; one reads `state.error.fieldErrors.email` directly (and crashes when `state` is `null`), another wraps every read in a `try`. The senior posture is structural: declare *one* shape, ship it as the project's `<FieldError>` and `<SubmitButton>` primitives, and reuse it across every form. The shape is the chapter's spine ‚Äî every later form chapter (7.4, 7.6, 11.3, 17.2) re-references it.

**Ideal teaching artifact.** Pattern archetype delivered as a **stencil-and-fill canonical shape** in `AnnotatedCode`. The student sees one canonical form component ‚Äî the `<CreateInvoiceForm>` ‚Äî laid out in five labeled regions: (1) **the hook call**: `const [state, formAction, isPending] = useActionState(createInvoice, null);`; (2) **the form-level error banner**: `{state?.ok === false && state.error.fieldErrors == null && <Banner variant="destructive">{state.error.userMessage}</Banner>}` ‚Äî the discriminator check unlocks the typed access, the `fieldErrors == null` branch keeps the banner from doubling with field-level renders; (3) **the form**: `<form action={formAction}>` wrapping the inputs, each with `name`, `defaultValue`, and a sibling `<FieldError name="..." />`; (4) **the field-error component**: a tiny `<FieldError>` that reads `state?.ok === false ? state.error.fieldErrors?.[name]?.[0] : null` and renders it under the input, anchored by `aria-describedby`; (5) **the submit**: `<SubmitButton>Save</SubmitButton>` (the `useFormStatus`-driven variant from Concept 7). The annotation track walks each region in order, naming the principle behind each ‚Äî the discriminator check, the schema-as-message-author rule from 7.1, the typed `Result` reader, the spine-component reuse. The closing line names the principle: *every form in the codebase follows this shape; if a form deviates, a new component has emerged or a rule has bent.*

**Engagement.** A `Sequence` drag-to-order drill on a shuffled form-component skeleton: the student orders six fragments ‚Äî the hook call, the form-level banner, the `<form>` wrapper, the input + `<FieldError>` pair, the submit button, and a distractor (a `useEffect` that resets the form manually on `state.ok`). The distractor has no correct slot ‚Äî the feedback names "the action's auto-reset already does this; the `useEffect` competes with the platform." Confirm with a `MultipleChoice`: "a teammate's PR adds a third form to the codebase with its own inline error rendering ‚Äî what does the senior reviewer ask for?" ‚Äî correct answer names *extract `<FieldError>` and `<SubmitButton>`, reuse them*; wrong answers name symptom mitigations.

**Components.**
- `AnnotatedCode` walking the five-region canonical form-component shape.
- `Sequence` for the six-fragment drag-to-order drill with the `useEffect` distractor.
- `MultipleChoice` for the spine-reuse PR-review.
- `Aside` (`note`) below: "one canonical form shape, two reusable primitives (`<FieldError>`, `<SubmitButton>`), every form in the codebase. Drift starts the first time someone inlines an error renderer."

**Project link.** 7.6.4 ships `<FieldError>` and `<SubmitButton>` as starter primitives the student fills; every form in the project (create, edit, delete) uses this exact five-region shape. The reviewer's "is this a one-off render or a reusable primitive?" question maps onto this concept's rule.

## Concept 7 ‚Äî `useFormStatus`: descendant-only, and the SubmitButton pattern

**Why it's hard.** The student has `isPending` from `useActionState` already ‚Äî why a *second* hook for the same fact? The answer is structural: `useActionState` returns `isPending` to the form's *owning* component; `useFormStatus` returns `pending` to *descendant* components inside the form. Calling `useFormStatus` in the form's own render scope returns `pending: false` always (the hook reads from a context the form establishes for *its* descendants), and the failure is silent ‚Äî the button never disables, the spinner never appears, and the student "fixes" it by reaching back for prop-drilling. The reflex install is two-part: the descendant rule, and the `<SubmitButton>` pattern as the canonical reach (the button reads context, the form imports the component, no prop crosses the boundary).

**Ideal teaching artifact.** Mechanics archetype delivered as a **wrong-by-default sandbox** in `ReactCoding`. The student opens a small form with three components: a `<Form>` parent that calls `useActionState`, an `<InvoiceFields>` middle layer that renders the inputs, and a `<MySubmitButton>` leaf that needs to disable on pending. The starter wires `<MySubmitButton>` *next to* the form (a sibling, not a descendant) and calls `useFormStatus()` inside it ‚Äî the button never disables, the spinner never appears. The student's job is two repairs: (1) move the `<MySubmitButton>` *inside* the `<form>` element (the descendant rule lands at the keyboard); (2) the lesson next shows the same button placed inside a `<fieldset>` two layers deep ‚Äî the hook still works because it reads context, not props (the boundary discipline lands). The fixtures simulate a slow action (a 1500ms delay) so the pending state is observable. The wrong-by-default state is the assessment ‚Äî the student fixes the bug to pass.

A side beat names the **`data` field** of `useFormStatus` as the niche reach: a delete confirmation reads `data.get('id')` to render "Deleting invoice INV-104..." inside the disabled state. One paragraph, named once for the project's delete dialog.

**Engagement.** The repair carries the assessment because the student must move the component to make the form work. Confirm with a `MultipleChoice` after the repair: "where can `useFormStatus()` be called and return a useful value?" with picks ‚Äî *only inside a descendant of `<form>`*, *anywhere in the form's owning component*, *only inside the form's own render scope*, *anywhere on the page*. The right answer reinforces the descendant rule.

**Components.**
- `ReactCoding` for the wrong-by-default sandbox with the descendant rule and the boundary discipline as the two repair steps. The fixtures simulate a slow action so pending state is observable.
- `MultipleChoice` for the descendant-rule confirmation.
- `Aside` (`caution`) below: "`useFormStatus` is descendant-only and reads from the form's context ‚Äî `useActionState`'s `isPending` is for the form's owning component. They report the same fact from different sides of the boundary."

**Project link.** 7.6.4 ships `<SubmitButton>` (the shadcn `Button` + `useFormStatus` composition) as the project's reusable primitive; the dialog's delete-confirmation button uses the `data.get('id')` reach to render "Deleting invoice {id}...".

## Concept 8 ‚Äî The threshold for optimistic UI: a decision before the hook

**Why it's hard.** `useOptimistic` is shiny ‚Äî the student reads "instant UI updates" and reaches for it on every mutation. The cost is a worse UX, not a better one: optimistically updating a payment that fails 30% of the time produces a "charged ‚Üí not charged" flicker the user reads as a bug; optimistically updating a server-validated mutation forces the form to render a partial entity (no server-generated timestamp, no computed total) the user notices as broken. The senior reach is to install the *threshold* before the *hook*: optimism earns its weight when the success rate is overwhelmingly high *and* the immediate feedback is visible *and* the UI change is small. Everything else uses the in-flight pending state from Concepts 5 and 7. Without the threshold first, the student writes optimism into mutations that should wait.

**Ideal teaching artifact.** Decision archetype delivered as a **three-axis decision triangle** rendered as a `Figure`-wrapped hand SVG. Three labeled vertices: *high success rate* (does this mutation succeed >95% of the time?), *visible-to-user* (does the user watch the affected pixel?), *small UI change* (one row, one flag, one count ‚Äî not a wholesale page refresh). The vertices form a triangle; the centroid is the only zone where optimism earns its weight. Outside the triangle: the alternative is the in-flight pending state. Below the triangle, eight SaaS mutations are dropped onto the diagram in their respective zones ‚Äî *like a comment*, *toggle starred*, *mark notification read*, *cart quantity +/-* land in the centroid; *create invoice with validation*, *complete checkout*, *change subscription plan*, *upload avatar* land outside (one or more vertices missing). Each off-centroid case carries a one-line annotation: "validation fails 15% of the time ‚Äî the rollback feels like a bug," "the user navigates away during the upload ‚Äî they don't watch the optimistic frame," "the plan-change cascades a recalculation the optimistic frame can't compute." The closing rule: *the default is no optimism; the trigger is all three axes lit.*

**Engagement.** A `Buckets` sort: ten mutations into "optimistic" vs "pending state" buckets, each with the dominant disqualifier from the three axes named in the wrong-answer feedback. The mutations span the SaaS surface ‚Äî a comment like, a profile-photo upload, a cart line removal, a tier upgrade, a notification dismiss, a calendar event drag-to-reschedule, a Stripe webhook retry, a soft-delete with undo, a project-name inline edit, a Stripe subscription cancellation.

**Components.**
- `Figure` wrapping a hand SVG of the three-axis decision triangle with the eight SaaS mutations placed in their zones.
- `Buckets` for the ten-mutation sort.
- `Aside` (`tip`) below: "the default is no optimism; the trigger is high success rate *and* visible-to-user *and* small UI change. One missing vertex disqualifies."

**Project link.** 7.6 reaches for optimism *once* ‚Äî the create-invoice flow in 7.6.5 ‚Äî because all three axes are lit (validation passes >95% from the form's own constraint check, the user watches the new row appear, the UI change is one row in a list). The edit and delete flows use pending state only.

## Concept 9 ‚Äî `useOptimistic`: implicit rollback, transition-bound, UUID-keyed

**Why it's hard.** Three structural facts collapse into one beat. (1) **The hook is transition-bound**: `addOptimistic(value)` outside a `startTransition` (or outside the action call's implicit transition) makes the optimistic state *stick* ‚Äî the rollback never fires. (2) **The rollback is implicit**: React owns it; the form layer doesn't write rollback logic, doesn't track temp/real ID mappings, doesn't manually remove failed updates. The student who writes manual rollback ships a competing implementation that breaks on the next React minor. (3) **The temp-ID problem and the UUIDv7 reach**: the optimistic entity needs a key React can reconcile against the server-returned entity; the senior fix is a client-generated UUIDv7 (passed to the action as the entity's ID), which lets the optimistic and persisted rows reconcile by key with no flicker. The temp-string fallback (`temp-${Date.now()}`) works but flickers on revalidation as the temp item unmounts and the real item mounts. Each fact has to land at the keyboard, not be paragraphed past.

**Ideal teaching artifact.** Mechanics archetype delivered in two beats. **Beat one ‚Äî the controllable-failure simulator** in `ReactCoding`. The student sees a comments list with an "add comment" form wired to `useActionState(addComment, null)` plus `useOptimistic(comments, (cur, next) => [...cur, next])`. A `?fail=1` query-param toggle in the preview pane flips the action's success/fail outcome. The student submits with `fail=0`, sees the optimistic comment land instantly and reconcile to the server-returned row; toggles to `fail=1`, submits again, and watches the optimistic row appear, then *vanish on its own* when the action returns failure ‚Äî the implicit rollback is felt, not asserted. A second toggle in the same pane wraps the optimistic call *outside* a transition (a deliberate misuse) ‚Äî the optimistic row appears and *stays* even on success, the actual list re-renders with the real row alongside, the duplicate is the bug. The student sees the transition rule as a failure mode, not a footnote.

**Beat two ‚Äî the temp-ID contest** in `CodeVariants`. Tab one: the temp-string approach (`temp-${Date.now()}`) ‚Äî the optimistic row mounts with the temp ID, revalidation re-renders the list with the real ID, React reconciles by key, the temp item unmounts and the real item mounts; the brief visual flicker is shown in a small annotated GIF or step diagram. Tab two: the client-generated UUIDv7 approach ‚Äî `crypto.randomUUID()` is computed, passed to the action as a hidden field (the action accepts the client-provided ID, UUIDv7 reconciles cleanly with the 6.2 PK convention), the optimistic and persisted rows share the same key, no flicker. Below both tabs, the rule: *temp strings are the prototype reach; UUIDv7 is the project default.*

**Engagement.** Beat one's `?fail=1` toggle and the transition-misuse toggle carry the assessment because the student must observe both behaviors. Confirm with a `MultipleChoice` after both beats: "a teammate's optimistic update sticks on success ‚Äî the row duplicates with the real one. What's the bug?" with the right answer naming *the `addOptimistic` call wasn't inside the action's transition*; wrong answers name symptom mitigations ("write a `useEffect` to dedupe", "key by index instead of ID").

**Components.**

## Concept 10 ‚Äî Constraint Validation: the cheap layer with a hard rule

**Why it's hard.** Two distinct calls collapse into one for the student. (1) **The platform already ships client-side validation**: `required`, `pattern`, `type`, `min`/`max`, `minlength`/`maxlength`, `inputmode`, `autocomplete` ‚Äî checks the browser runs before any JS, with focus management and native error bubbles, for free. The student who reaches for a JS-based validation library before knowing this surface ships extra weight that does less. (2) **The hard rule on top**: every check that runs on the client *also* runs on the server in the Zod schema. Stale clients running an older bundle, JS-disabled submits, scripted requests ‚Äî the client check is UX, the server check is correctness. The two are mirrored for shape rules; cross-resource rules (uniqueness, plan limits) live exclusively in the action body. The teaching has to land both ‚Äî the surface as a usable catalog *and* the rule as a non-negotiable structural contract.

**Ideal teaching artifact.** Reference archetype delivered in two beats. **Beat one ‚Äî the trigger-driven attribute catalog** rendered as a `Figure`-wrapped HTML table. Three columns: *the rule the input enforces* (presence, length range, numeric range, format, soft-keyboard hint, autofill hint), *the attribute that fires it* (`required`, `minlength`/`maxlength`, `min`/`max`/`step`, `pattern`/`type="email|url|tel|date"`, `inputmode="numeric|decimal|email|tel"`, `autocomplete="email|new-password|one-time-code|tel|street-address"`), and *the schema mirror* (the Zod equivalent: `.min(1)`, `.length`, `.gte`/`.lte`, `z.email()`/`z.url()`, n/a ‚Äî input-mode is UX-only, n/a ‚Äî autocomplete is UX-only). The catalog reads top-to-bottom as a one-page reference the student returns to whenever they author an input, with the schema mirror making the *parallel-but-mirrored* discipline visible at a glance.

**Beat two ‚Äî the line between client and server** rendered as a `Figure`-wrapped two-axis decision matrix. Two axes: *what the rule needs to know* (only the input value vs database/external state) and *who enforces it* (the constraint API on the client, the Zod schema on the server, the action body). The cells produce three rules: shape rules (presence, length, format) ‚Üí *both* the client constraint API and the schema, mirrored; cross-resource rules (uniqueness, plan limits, suppression lists) ‚Üí action body only, after the parse; programmatic shape rules the attributes can't express (cross-field "passwords match", custom strength predicates) ‚Üí `setCustomValidity` on the client *and* a schema `.refine(...)` on the server, named once for the rare reach. Below both beats, the rule: *the client surface is UX; the server surface is correctness; every shape rule lives on both, every business rule lives on the server only.*

**Engagement.** A `Tokens` round on a forty-line user-signup form authored *without* constraint attributes ‚Äî the student clicks every input where a `required`, `type`, `minlength`, `inputmode`, or `autocomplete` attribute is missing, and the wrong-answer feedback names what the platform was about to do for free. Confirm with a `Buckets` sort of eight rules ("email must be valid format", "the email must not already be taken", "password must be ‚â•8 characters", "password must match the org's complexity policy stored in the DB", "the slug must match `[a-z0-9-]+`", "the plan must be permitted by the org's billing tier", "the date must be in the future", "the file must be ‚â§5MB") into "client + server (both)" vs "server only (action body)."

**Components.**
- `Figure` wrapping a hand-coded HTML attribute catalog for beat one.
- `Figure` wrapping a hand-coded HTML two-axis decision matrix for beat two.
- `Tokens` for the missing-attribute spotting round.
- `Buckets` for the rule-placement sort.
- `Aside` (`caution`) below the matrix: "client validation is UX; server validation is correctness. Every shape rule lives on both; cross-resource rules live exclusively in the action body."

**Project link.** Every input in 7.6's invoice forms carries the right `required`, `type`, `min`, `pattern`, `inputmode`, and `autocomplete` attributes; the `drizzle-zod`-derived schema mirrors each shape rule; the cross-resource rules (invoice number unique within the org, customer belongs to the same org) live exclusively in the action body.

## Concept 11 ‚Äî Production-grade error rendering: `:user-invalid`, the bubble, and the FieldError reader

**Why it's hard.** The constraint API's default UX has two production-blocking defaults the student doesn't think to override. (1) **`:invalid` styles fields red the moment the page renders** ‚Äî before the user has typed anything, the empty `required` email field is already styled as an error, which is *wrong*. The fix is `:user-invalid` (post-interaction), the W3C pseudo-class added precisely for this case. (2) **The native error bubble** ("Please fill in this field") doesn't match the design system, doesn't position predictably, and disappears the moment the user clicks elsewhere. The senior reach is to suppress the bubble (via `onInvalid={e => e.preventDefault()}`) and render the design-system inline error from the `Result.error.fieldErrors` tree (the `<FieldError>` component from Concept 6). The two fixes together make the form feel *finished*; without them, the form ships with two visible polish gaps.

**Ideal teaching artifact.** Pattern archetype delivered as a **before-and-after polish contest** in `HtmlCssCoding`. The student sees a four-field form rendered three ways in side-by-side panes the lesson advances through: (1) **default** ‚Äî every field rendered red on first paint (the `:invalid` styling fires on empty required fields), the native bubble appearing on submit; (2) **`:user-invalid`** ‚Äî the styling now waits for the user to interact with the field (blur or first invalid-submit attempt); the same form looks calm on first paint; (3) **suppressed bubble + inline error** ‚Äî `onInvalid={e => e.preventDefault()}` on each input, the `<FieldError>` component rendering the schema-authored message under the field, anchored by `aria-describedby` for screen readers, the input itself carrying `aria-invalid={!!fieldError}`. The student modifies the starter to add the `user-invalid:` Tailwind variant (replacing `invalid:`) and the `onInvalid` handler ‚Äî the form transitions from "default browser" to "design-system-aligned" at the keyboard. The fixtures show the contrast at three points: empty initial render (no red), first blur with empty required field (red appears), submit attempt with invalid email (the inline error renders, not the native bubble).

A side beat names **`noValidate` on the form** as the rare-reach: when the design system fully owns inline error rendering and the latency cost of a server roundtrip is acceptable, `<form noValidate action={...}>` disables the cheap pre-submit check entirely. Named once; the project chapter doesn't reach for it.

**Engagement.** The polish-contest repair carries the assessment because the student must apply the two fixes. Confirm with a `MultipleChoice` after the repair: "a teammate's PR styles invalid inputs with `:invalid`; the design review flags that empty required fields appear red on first render. What's the senior fix?" ‚Äî correct answer names *swap to `:user-invalid` for the post-interaction case*; wrong answers name symptom mitigations ("delay the styling with a `setTimeout`", "remove the `required` attribute and check in JS").

**Components.**
- `HtmlCssCoding` for the three-pane before-and-after polish contest with the styling and the inline-error reader as the two repair steps.
- `MultipleChoice` for the `:user-invalid` confirmation.
- `Aside` (`tip`) below: "`:user-invalid` for post-interaction styling; suppress the native bubble and render the schema-authored message via `<FieldError>` anchored with `aria-describedby` and `aria-invalid`."

**Project link.** 7.6.4 ships the `<FieldError>` component reading from `state.error.fieldErrors`, with `aria-describedby`/`aria-invalid` wiring; every input in the project's forms uses `user-invalid:border-destructive` for the post-interaction red state; the native bubble is suppressed on every input.

## Concept 12 ‚Äî Progressive enhancement: the property that falls out (and the test that catches regressions)

**Why it's hard.** The instinct from the JS-heavy 2018‚Äì2023 era is to treat PE as a niche concern ‚Äî "nobody disables JavaScript." The 2026 reality is the inversion: PE is what the platform gives you for free *if you don't break it*, and the bundle-not-loaded-yet case (a flaky mobile connection, a slow first paint, a crashed SDK) is the more common production failure mode than deliberately disabled JS. The senior posture is structural: build the form the way the rest of the chapter taught (action prop, uncontrolled inputs, schema on the server), and PE *falls out*. The five disciplines are the "don't break it" list; the JS-disabled manual test is the verify reflex that catches regressions before they ship. Without both as named structural commitments, the student treats the chapter's prior six lessons as recipes instead of as a system whose property is PE.

**Ideal teaching artifact.** Concept archetype delivered in two beats. **Beat one ‚Äî the side-by-side JS-on/JS-off comparison** in `DiagramSequence`. Four frames the student steps through: (1) **submit with JS** ‚Äî the form's `<form action={createInvoice}>` fires, React intercepts, the action runs via the opaque-ID POST, the `Result` returns to `useActionState`, the form re-renders with the inline field errors (the lifecycle from Concept 3, recalled); (2) **submit without JS** ‚Äî the same form with JavaScript disabled in DevTools; the browser submits to the framework-rewritten action URL natively, the same Server Action runs, the action `redirect()`s on success or returns a `Result` whose error path renders via the server's HTML response (degraded ‚Äî a full page reload with field errors as `searchParams`-driven render); (3) **what works without JS** ‚Äî annotated as a checklist over the prior frames: the action prop's submit, the constraint API's pre-submit checks, the action's `redirect()`, `revalidatePath()`'s effect on the next request, the Server Component re-render after the redirect; (4) **what doesn't work without JS** ‚Äî the inverse checklist: `useActionState`'s in-page error render, `useFormStatus`'s pending state, `useOptimistic`'s instant update, the auto-reset on success. The closing rule lands on the frame: *the form's function is the same in both modes; the experience degrades gracefully ‚Äî no JS, no in-page error, but the row is still created.*

**Beat two ‚Äî the five disciplines as a checklist plus the JS-disabled verify** rendered as a `Figure`-wrapped checklist with each discipline cross-linked to the chapter lesson it derives from: (1) **use `action`, not `onSubmit` + `fetch`** (Concept 3); (2) **uncontrolled inputs only** (Concept 1) ‚Äî controlled inputs need JS to maintain values; (3) **don't put critical UI inside `useOptimistic` or hooks that need React** (Concept 9) ‚Äî the optimistic state is a JS-only enhancement; (4) **don't gate the submit behind a JS-only condition** ‚Äî the button is enabled by default, the server is the boundary of correctness; (5) **keep the form's HTML semantic** ‚Äî `<button type="submit">`, `<label htmlFor>`, `<form action>`. Below the checklist, the verify reflex: *every important form gets one manual JS-disabled test pass at feature-launch* (DevTools ‚Üí Settings ‚Üí Disable JavaScript ‚Üí submit). The closing line names the senior posture: *don't try to make the form PE-compatible; don't break the PE that the platform gives you.*

**Engagement.** A `Buckets` sort: ten form behaviors ("the row is created on submit", "the spinner appears on the submit button", "field errors render under the input on the same page", "the user is redirected to the new resource's detail page", "the optimistic comment appears instantly in the list", "an unchecked checkbox sends `'on'`", "the form resets to its `defaultValue`s on success", "the cached list refreshes after the mutation", "the `required` attribute blocks empty submits", "the `:user-invalid` styling appears after blur") into "works without JS" vs "JS-only enhancement." Wrong-answer feedback names which discipline (or which absence of one) governs each.

**Components.**
- `DiagramSequence` hosting the four-frame JS-on/JS-off comparison; each frame is a hand SVG inside `Figure`.
- `Figure` wrapping a hand-coded HTML checklist of the five disciplines, with cross-links to the prior concepts.
- `Buckets` for the works-without-JS sort.
- `Aside` (`note`) below: "PE is what the platform gives you for free if you build the form the chapter's way. The five disciplines are the *don't break it* list; the JS-disabled manual test is the verify."

**Project link.** 7.6.7 verifies PE on the create-invoice form as a named "Done when" clause: the student opens DevTools, disables JavaScript, submits the form, and confirms the row is created and the redirect lands. The five disciplines map onto the project's reviewer checklist line by line.
