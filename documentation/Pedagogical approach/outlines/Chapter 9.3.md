# Chapter 9.3 ‚Äî Authentication flows: pedagogical approach

## Concept 1 ‚Äî `requireEmailVerification` reshapes the post-sign-up flow

**Why it's hard.** A junior reads `emailAndPassword: { enabled: true }` and assumes "submit form, get session." The senior reflex is the inverse: the sign-up handler that *issues a session* is the broken one ‚Äî the email is unproven, the form is the identification-equals-authentication misframe from 9.1 made operational. Three knobs reshape the flow at once: `requireEmailVerification: true` flips the response from *credential stored, session issued* to *credential stored, verification queued, no session*; `autoSignIn: false` keeps the unverified row from leaking a cookie; `minPasswordLength: 12` raises the floor above the library's 8. The student needs the three knobs as one *posture*, not three independent dials, and they need the password's destination ‚Äî `account.password`, not `user.password` ‚Äî re-anchored from 9.2 because the response shape and the row layout are what the next eight lessons compose against.

**Ideal teaching artifact.** Pattern archetype delivered as a **two-config side-by-side replay** in `CodeVariants`. Tab one *(naive default)*: `emailAndPassword: { enabled: true }` ‚Äî under it, a hand-SVG panel inside `Figure` showing what lands on submit: a `user` row with `emailVerified: false`, an `account` row with the Argon2id hash in `account.password`, *and* a `__Host-` cookie attached to the response. The student is asked to predict the next-page render ‚Äî the unverified user is signed in, the inbox is bypassed, the misframe is shipped. Tab two *(senior baseline)*: the same config with `requireEmailVerification: true`, `autoSignIn: false`, `minPasswordLength: 12` added on highlighted lines ‚Äî same `user` and `account` rows, *plus* a `verification` row, *no* cookie, the response carries an "ok" Result and the form transitions to a "check your inbox" view. A side caption pins the rule: the password lands on `account` (one of many proofs), `user.emailVerified` is the gate Lesson 9.3.2 reads, the cookie shows up only after the verify-click in 9.3.3.

**Engagement.** A `MultipleChoice` post-replay: "the team enables `emailAndPassword` and ships sign-up. QA reports the user is signed in immediately and never sees the verification email ‚Äî three knobs in the config are missing or wrong." The four options frame the misconfigurations (only `requireEmailVerification`; only `autoSignIn: false`; both with `minPasswordLength: 6`; the senior triple). Confirmed by a `Tokens` exercise on the canonical config block: the student clicks the three load-bearing tokens (`requireEmailVerification`, `autoSignIn`, `minPasswordLength`) and gets a one-line reveal per token naming the bug each default ships if left wrong.

**Components.**
- `CodeVariants` for the broken/fixed two-tab replay, with a hand-SVG-inside-`Figure` row-and-cookie panel embedded in each tab.
- `Tokens` on the senior config block for the three-knob reveal.
- `MultipleChoice` for the four-misconfig diagnosis.
- `Aside` (`caution`) below the variants: "the password lives on `account`, never on `user`. The cookie shows up *after* the verify-click, never on sign-up. The library's defaults ship the misframe ‚Äî the senior baseline closes it."

**Project link.** The 9.5 project's lesson 9.5.4 flips `requireEmailVerification` on as the gate that turns the spine from "anyone can sign in" into "verification is the floor"; the replay is the *why* the project doesn't accept the library defaults and the *what* lesson 9.5.6 verifies in Postgres + browser cookies.

## Concept 2 ‚Äî User-enumeration discipline as the chapter-long thread

**Why it's hard.** Enumeration is the one threat that runs through every flow in the chapter ‚Äî sign-up, sign-in, verification, reset, magic-link ‚Äî and the misframe is identical at every entry point: the response shape *answers a question the form should refuse to answer*. "This email is already registered" undoes the sign-up defense; "no such email" undoes the sign-in defense; "we sent a reset link to this account" with a different copy when the email doesn't exist undoes the reset defense. Each leak hands an attacker a yes/no oracle for *does this email have an account?*, which is the precondition for credential-stuffing, targeted phishing, and password-spray. The student knows the rule abstractly from 9.1 but breaks it by reflex because friendlier UX always seems to demand the distinction. The discipline has to land as a *response-shape ledger across the chapter* so the student carries the rule into every flow they wire, not just the one in front of them.

**Ideal teaching artifact.** Concept archetype delivered as an **enumeration ledger across five flows** rendered as a hand-coded HTML table inside `Figure`. Five rows ‚Äî *sign-up*, *sign-in*, *resend verification*, *request reset*, *magic-link send* ‚Äî and four columns: *what the user types*, *what the server does internally*, *the response shape (always identical regardless of email-existence)*, *the attack the response-shape closure prevents*. Sign-up: types email+password; server creates row or skips; *response is the same "check your inbox" view in both cases*; closes account-discovery via sign-up. Sign-in: types email+password; server validates or doesn't; *response is the same `'invalid-credentials'` for "wrong password" and "no such email"*; closes account-discovery via sign-in. Resend: types email; server queues or skips; *response is the same "if it exists, we sent it"*; closes the resend oracle. Reset: same shape. Magic-link: same shape (with `disableSignUp: false`; the `disableSignUp: true` case re-opens enumeration unless the rule is held). The table's last column is the load-bearing one ‚Äî every closure ties back to a specific attacker capability, so the rule reads as a defense, not as friction. Below the table, a one-paragraph callout names the *deliberate* friendlier-UX trade: a product can choose to surface "this email is already registered" at sign-up to reduce confused-user support, but the trade is *named* ‚Äî the enumeration cost is paid, the choice is conscious, the default stays closed.

**Engagement.** A `Buckets` sort: ten error-message strings into two columns ‚Äî *enumeration-safe (ship)* and *enumeration-leak (rewrite)*. Examples: "If an account exists for that email, we sent a reset link" (safe); "No account found for that email" (leak); "Wrong password" (leak ‚Äî distinguishes from "no such email"); "Invalid email or password" (safe); "We re-sent the verification email" (safe ‚Äî same shape regardless of whether the account was unverified or didn't exist); "This email is already in use, sign in instead" (leak ‚Äî discovery via sign-up); "Check your inbox" (safe). Wrong-answer feedback names the attacker capability the leak hands away. Confirmed by a `MultipleChoice`: "a teammate's PR adds 'this email is already registered, sign in instead' to the sign-up form's error path 'so users don't get confused.' What does the senior reviewer ask for?" ‚Äî the right answer names the trade explicitly (it's a deliberate enumeration-cost choice, not a free UX win), names the alternative (keep the response identical, add a "trouble signing in?" link in the inbox-confirmation view that surfaces the reset path), and ties the decision to the threat model the product actually faces.

**Components.**
- `Figure` wrapping a hand-coded HTML five-row, four-column ledger table. The table recurs *in spirit* across the chapter ‚Äî every later concept tags the relevant row when it touches a flow's response surface ‚Äî but the table itself is the install-once artifact.
- `Buckets` for the ten-message safe-vs-leak sort.
- `MultipleChoice` for the friendlier-UX-trade confirmation.
- `Aside` (`caution`) below the ledger: "every flow's response shape is identical regardless of email existence. Friendlier copy is a deliberate enumeration-cost trade ‚Äî name it, don't sleepwalk into it."

**Project link.** The 9.5 project explicitly pins enumeration discipline as a non-negotiable across sign-up, sign-in, and resend (lesson 9.5.4 ships opaque errors); the ledger is the *why* the project doesn't reach for the friendlier copy and the *checklist* lesson 9.5.6 verifies against curl probes that the response shape doesn't branch on email existence.

## Concept 3 ‚Äî The sign-in `Result` discriminant catalog

**Why it's hard.** Sign-in's response is not binary. *Five* discriminants travel back from the action ‚Äî `'invalid-credentials'`, `'email-not-verified'`, `'too-many-attempts'`, `'requires-second-factor'`, `'ok'` ‚Äî and each one drives a different UI continuation: a re-prompt with opaque copy, a "check your inbox" replacement, a "try again in N minutes" lockout view, a transition into the TOTP/passkey form, or a `redirect`. The student's reflex is to treat sign-in like login forms in 2018 ‚Äî pass/fail with a single error string ‚Äî and the misframe ships specific bugs: rendering `'requires-second-factor'` as an error (the user sees "sign-in failed" and tries the password again instead of being routed to the TOTP prompt); collapsing `'email-not-verified'` into `'invalid-credentials'` (the user can't tell why they're stuck and can't find the resend path); skipping `'too-many-attempts'` (the form re-submits and the lockout never surfaces). The five branches are *the lesson*, not a footnote.

**Ideal teaching artifact.** Reference archetype delivered as a **five-branch decision tree paired with a response-to-UI mapping table**, both inside `Figure`-wrapped hand-SVGs. The tree draws the library's internal flow as a top-down branch from "credentials submitted" through five forks: (1) lookup fails or password mismatch ‚Üí `'invalid-credentials'`; (2) lookup succeeds, `emailVerified: false` ‚Üí `'email-not-verified'`; (3) failed-attempt counter tripped ‚Üí `'too-many-attempts'`; (4) verified user, 2FA enabled ‚Üí `'requires-second-factor'` *with `factor-token`*; (5) verified, no 2FA ‚Üí `'ok'` *with session row inserted, cookie attached*. The mapping table to the right has five rows ‚Äî one per discriminant ‚Äî and three columns: *the discriminant*, *what the form replaces itself with*, *what the user does next*. `'invalid-credentials'`: form stays, opaque "invalid email or password" error, user retries. `'email-not-verified'`: form replaced by "check your inbox" view, library re-sent the email automatically (`sendOnSignIn: true`), user clicks the link. `'too-many-attempts'`: form replaced by lockout view with countdown and a reset-password link, user waits or resets. `'requires-second-factor'`: form replaced by the TOTP/passkey prompt, the `factor-token` is held in pre-auth state and consumed by the second-factor verify call. `'ok'`: redirect to `callbackURL`. The last column ties every branch to its lesson ‚Äî the failure paths are *not* errors in the framework sense, they are continuation states each driving a specific UI move.

**Engagement.** A `Matching` exercise pairing the five discriminants against five UI-move descriptions (form-stays / replace-with-check-inbox / replace-with-lockout-countdown / replace-with-TOTP-prompt / redirect). Confirmed by a `Sequence` ordering: the student arranges the seven steps of a 2FA-enabled sign-in ‚Äî *user types password*, *server validates credentials*, *server checks `emailVerified`*, *server checks failed-attempt counter*, *server detects 2FA enabled and returns `'requires-second-factor'` with `factor-token`*, *form swaps to TOTP prompt*, *user types code, library validates, session issued, redirect*. Two distractors test the misframe: *"render `'requires-second-factor'` as an error toast"* (no ‚Äî it's a continuation state, the form must transition); *"verify the TOTP code without the `factor-token`"* (no ‚Äî the token is what binds the second-factor call to the validated first factor).

**Components.**
- `Figure` wrapping a hand-SVG five-branch decision tree with the discriminant as the leaf label.
- `Figure` wrapping a hand-coded HTML five-row mapping table (discriminant / form replacement / user's next action).
- `Matching` for the discriminant-to-UI-move pairing.
- `Sequence` for the 2FA-sign-in seven-step ordering with two misframe distractors.
- `Aside` (`note`) below the tree: "`'requires-second-factor'` is a continuation, not an error. Every branch except `'ok'` *replaces the form* ‚Äî render the right view, don't pad an error string."

**Project link.** The 9.5 project ships only `'invalid-credentials'`, `'email-not-verified'`, and `'ok'` (no 2FA scope); the tree is the *map* lesson 9.5.4 leans on when wiring `useActionState` against the action's return, with the unimplemented branches greyed out as the cuts the project owns.

## Concept 4 ‚Äî The two defenses: per-account lockout and per-IP rate limit

**Why it's hard.** Brute-force defenses look redundant on paper ‚Äî both count failures, both reject requests past a threshold ‚Äî and the student's instinct is to keep one ("the IP rate limit is enough, the lockout just frustrates real users"). They protect against *different* attacks and dropping either opens a real hole. *Per-IP rate limit* defends against credential-stuffing from one machine ‚Äî an attacker tries `password1` against ten thousand emails. *Per-account lockout* defends against the *inverse* attack ‚Äî an attacker rotates IPs (or runs from a botnet) and tries a thousand passwords against one email. Without the lockout, IP rotation evades the per-IP cap and the account is open. Without the per-IP cap, one IP can evade the per-account counter by spreading across many emails. The student needs the two attacks visualized in parallel so the *complementary* shape lands, not as "defense in depth" platitudes but as the specific holes each closes.

**Ideal teaching artifact.** Concept archetype delivered as a **two-attack parallel walkthrough** in `DiagramSequence`. Two attack scenarios, each as a three-frame scrub; the same parallel-lanes shape Concept 3 of 9.1 used so the visual vocabulary is recognized. *Attack A ‚Äî credential stuffing from one IP* (left lane): frame 1: attacker has a leaked password, tries it against `ada@acme.com`, `ben@acme.com`, `cara@acme.com`, ... from one machine; frame 2: with *no per-IP cap*, the attempts go through, one in ten thousand hits and the account falls. Frame 3: with *per-IP cap at 10/min*, the eleventh request 429s, the campaign collapses; the per-account counter doesn't fire because no individual account hit the threshold. *Attack B ‚Äî password spray with IP rotation* (right lane): frame 1: attacker targets `ada@acme.com` specifically, tries the top thousand passwords; frame 2: with *no per-account counter*, the attempts go through, one of the thousand might hit; the per-IP cap doesn't fire because each request comes from a different IP in the botnet. Frame 3: with *per-account counter at 5 attempts then 15-min lockout*, the sixth request locks Ada's account regardless of IP; the attacker either waits 15 minutes per password (intractable) or moves on. A small ledger underneath names the asymmetry: *per-IP rate limit defends the breadth of accounts, per-account counter defends the depth of one account; remove either and the corresponding attack works*.

**Engagement.** A `MultipleChoice` round of three scenarios: *"team disables the per-account counter because 'real users get locked out and complain'"* ‚Äî the right answer names the password-spray hole the disable opens, then names the alternative (clearer copy on lockout, a one-click reset link in the lockout view, never disabling); *"team raises the per-IP cap to 1000/min because 'the office wifi has many users behind one NAT'"* ‚Äî the right answer names the credential-stuffing window the loosening opens and the alternative (per-account counter still holds even when per-IP loosens, plus distinguishing real-NAT traffic via session-cookie-presence); *"team keeps the per-IP cap and removes the per-account counter because 'they're redundant'"* ‚Äî the right answer names them as orthogonal, not redundant, and runs the password-spray scenario through the leaky stack to show the hole. Wrong-answer feedback names the attack each misconfiguration opens.

**Components.**
- `DiagramSequence` hosting the two-attack three-frame scrubs in parallel lanes ‚Äî reuses the parallel-lanes shape from 9.1 Concept 3 and 9.1 Concept 8, confirming the in-unit reuse.
- `MultipleChoice` for the three-scenario round.
- `Aside` (`note`) below the scrub: "per-IP defends the breadth, per-account defends the depth. They protect different attacks; never disable the lockout for 'convenience' ‚Äî the right move is clearer copy and a one-click reset path."

**Project link.** The 9.5 project doesn't wire rate limits (full Upstash wiring is Chapter 15.3); the walkthrough is the *why* the project's lesson 9.5.6 production-checklist names "rate limits enabled in 15.3" as a real shipping prerequisite rather than a hand-wave.

## Concept 5 ‚Äî The shared `verification` substrate, with one table and four namespaces

**Why it's hard.** Four distinct flows ‚Äî verify-email, password-reset, magic-link, 2FA-setup ‚Äî all need a *short-lived, one-time token bound to a fact about the user*. The naive instinct is one table per flow (`verifications`, `password_resets`, `magic_links`, `pending_2fa`) ‚Äî same shape, four times. Better Auth ships *one* `verification` table with `identifier` as the namespacing key ‚Äî `email-verification:<email>`, `reset-password:<userId>`, `magic-link:<email>`, `2fa-setup:<userId>` ‚Äî and the same security primitives apply to all of them: CSPRNG token, hash at rest, constant-time compare, expiry, single-use, row deletes on success. Until the student sees the *substrate* ‚Äî and sees that storing the raw token would be the load-bearing bug, not a stylistic choice ‚Äî every flow reads as a separate puzzle and the security properties drift across implementations.

**Ideal teaching artifact.** Concept + Mechanics composite. *Beat one ‚Äî the substrate diagram:* a `Figure`-wrapped hand-SVG drawing the `verification` table as a single rectangle in the center with four arrows feeding into it, one per flow. Each arrow is labeled with the `identifier` namespace it writes (`email-verification:ada@acme.com`, `reset-password:user_42`, `magic-link:ada@acme.com`, `2fa-setup:user_42`) and the `expiresIn` it carries (1 hour, 10 minutes, 5 minutes, 10 minutes). The row's columns are annotated: `identifier` (the namespace + key ‚Äî what this token is *for*), `value` (the SHA-256 hash of the token, *never the raw token*), `expiresAt` (the cap), and a small "row deletes on use" label on the row's outline. Below the diagram, a one-line rule: *one table, many namespaces, one set of security properties enforced once*. *Beat two ‚Äî the token-shape walkthrough:* a small `DiagramSequence` (or `AnnotatedCode`) showing the four-frame lifecycle of one token ‚Äî *(1) server generates 32 random bytes via CSPRNG and base64url-encodes (256 bits)*, *(2) raw token goes into the email, `SHA-256(token)` goes into `verification.value`*, *(3) user clicks the link, server hashes the incoming token and compares constant-time against the stored hash*, *(4) match + within `expiresAt` ‚Üí delete the row, perform the side effect (flip `emailVerified`, reset password, issue session, complete 2FA setup)*. The two beats together answer the *why one table* question and the *what's in a token* question ‚Äî both lessons the student needs to read every later flow's security properties as the same shape, not as four ad-hoc implementations.

**Engagement.** A `Matching` exercise pairing six rows from "what just happened" against the `identifier` namespace each one writes ‚Äî *user clicks "send me a reset link"* ‚Üî `reset-password:<userId>`; *new user signs up with email+password* ‚Üî `email-verification:<email>`; *user types email into magic-link form* ‚Üî `magic-link:<email>`; *user enables 2FA and gets the QR code* ‚Üî `2fa-setup:<userId>`; *user changes their email from settings* ‚Üî `email-verification:<userId>` (different namespace because the user already exists ‚Äî keyed by user not email); *user clicks "resend verification"* ‚Üî `email-verification:<email>` (same namespace as sign-up, the previous row is invalidated). Confirmed by a `MultipleChoice`: "a teammate's PR stores the raw token in `verification.value` 'because hashing would break the lookup.' What's the failure mode and the fix?" ‚Äî the right answer names the database-snapshot threat (an attacker with read access to the table gets working tokens for every pending verification, reset, and magic-link), then names the fix (the lookup *is* by the hash ‚Äî server SHA-256s the incoming token and queries on the hashed value, no decryption needed; this is the same property session tokens have in 9.1 Concept 4).

**Components.**
- `Figure` wrapping a hand-SVG substrate diagram (one table, four arrows, four namespaces, columns annotated with what each holds).
- `DiagramSequence` (or `AnnotatedCode`, whichever fits the lesson rhythm best) for the four-frame token lifecycle.
- `Matching` for the six-flow ‚Üî namespace pairing.
- `MultipleChoice` for the raw-token storage diagnosis.
- `Aside` (`caution`) below the substrate diagram: "raw tokens in the email, hashed tokens in the table. The lookup is by the hash, not by decryption. Same primitive as session tokens in 9.1 ‚Äî recognize the shape every time."

**Project link.** The 9.5 project ships only `email-verification:<email>`; the substrate diagram is the *frame* lessons 9.5.4 and 9.5.6 use to verify the row's shape against `verification` in Postgres without re-deriving the security properties from scratch.

## Concept 6 ‚Äî Password reset's session-invalidation invariant

**Why it's hard.** The six steps of a password reset look like the verify-email flow with one more column ‚Äî request ‚Üí email ‚Üí click ‚Üí submit new password ‚Üí done. The *load-bearing* difference is buried in step five: the library deletes *every existing session for the user* on a successful reset. To the student this looks like punitive UX ("I changed my password and got logged out everywhere ‚Äî annoying"). To the senior it is the *whole point*: a password reset only fires when the user thinks the old credential might be in attacker hands; if the existing sessions stay alive, the attacker keeps acting *after* the legitimate user resets. Without the invalidation, the reset doesn't actually reset access ‚Äî it just changes a string the attacker no longer needs because they already have a session cookie. The invariant has to land as the *defining property* of a secure reset, not as an implementation detail.

**Ideal teaching artifact.** Decision archetype delivered as a **two-lane attacker-replay** in `DiagramSequence`, mirroring 9.1 Concept 3's stolen-laptop incident replay so the visual vocabulary is recognized. The student scrubs through the same five-frame attack in two parallel lanes ‚Äî *with session invalidation* (the senior baseline) on the left, *without session invalidation* (the broken implementation) on the right. Frame 1: attacker has phished Ada's password, signs in from their laptop at 09:00, gets a session cookie, starts exfiltrating data. Frame 2: Ada notices the suspicious activity, requests a password reset at 09:15. Frame 3: Ada clicks the link, types a new password, submits. *Left lane:* library hashes the new password, updates `account.password`, *deletes every session row for Ada's `userId`*; the attacker's cookie now points at a session row that doesn't exist. *Right lane:* library only updates `account.password`; the attacker's cookie still points at a valid session row ‚Äî the password change touched nothing the attacker is using. Frame 4: at 09:16 the attacker clicks anything. *Left lane:* the next request reads the session row, it's gone, 401, the attacker is signed out. *Right lane:* the next request reads the session row, it's there, the attacker continues. Frame 5: a small ledger names what each lane bought ‚Äî *left: reset means "every cookie that knew the old credential is dead"; right: reset means "the password string changed, the access didn't"*. The closing rule lands as one line: *the session-invalidation property is what makes "reset" mean reset*.

**Engagement.** A `MultipleChoice` round of three scenarios: *"team adds a 'keep me signed in on this device' checkbox to the reset form so the user doesn't have to sign back in"* ‚Äî the right answer names the threat the checkbox re-opens (the attacker's session would be the one kept alive if they'd hit reset themselves; even if not, the policy of "all sessions die" is what keeps the property structural rather than user-policy-dependent), then names the alternative (the *current* session ‚Äî the one initiating the reset ‚Äî gets re-issued automatically, which is the library's default; "remember me on other devices" is the dangerous flip); *"team applies the same property to password *change* from settings (9.4.2) ‚Äî every session except the current one is revoked"* ‚Äî the right answer names this as the consistent application of the same invariant (a credential change is a credential change; the attacker who phished the old password might also have the old session, the same revocation closes it); *"team turns off session invalidation for 'enterprise customers who don't want users disrupted'"* ‚Äî the right answer names the trade explicitly (the customer is asking for "reset that doesn't reset access"), names the precedent (no compliance regime accepts this), and surfaces the alternative (session invalidation with clear "we logged you out everywhere because your password changed" copy, not skipping the invalidation).

**Components.**
- `DiagramSequence` hosting the five-frame two-lane attacker replay ‚Äî reuses the parallel-lanes shape from 9.1 Concept 3 and 9.3 Concept 4.
- `MultipleChoice` for the three-scenario round.
- `Aside` (`caution`) below the replay: "session invalidation is what makes reset mean reset. The same invariant applies to password change at settings (9.4.2). Don't trade it away for UX."

**Project link.** The 9.5 project explicitly cuts password reset; the replay is the *why* the project's scope panel (lesson 9.5.1) names reset as "Chapter 9.3.4's territory" rather than a casual omission, and the *why* the chapter's quiz (9.3.10) returns to the property as a non-negotiable.

## Concept 7 ‚Äî Magic-link as a product decision: when passwordless earns the call

**Why it's hard.** "No passwords" sounds like an obvious win ‚Äî no leaks, no reuse, no reset flow ‚Äî and the student's reflex is either to default to magic-link for everything or to dismiss it as "not real auth." Both reflexes are wrong, and the correct call is *conditional on use-frequency, user base, and product centerpiece*. Magic-link wins when sign-in is infrequent (Slack-style monthly login), the user base is non-technical and reset-volume swamps support, or the product already centers email (Substack, mailing-list tools). It loses when sign-in is frequent (the inbox round-trip is friction every time), deliverability is shaky (corporate Outlook with aggressive filtering blocks the credential), or hard MFA is required (TOTP composes less cleanly without a password). The student needs the *threshold table* ‚Äî the specific product shapes that flip the call ‚Äî not "consider the trade-offs," because the trade-offs are exactly the hand-wave the chapter is designed to replace.

**Ideal teaching artifact.** Decision archetype delivered as a **three-column verdict matrix** rendered as a hand-coded HTML table inside `Figure`. Three columns: *email+password (default)*, *magic-link only (deliberate reach)*, *both (coexistence)*. Six rows, one per product axis: *sign-in frequency* (daily ‚Üí password; monthly ‚Üí magic-link), *user technical fluency* (technical ‚Üí password handles itself; non-technical, password-reset-volume problem ‚Üí magic-link), *product centerpiece* (any ‚Üí password is neutral; email-centric ‚Üí magic-link reads as native), *MFA requirement* (TOTP/passkey required ‚Üí password baseline composes more cleanly; optional ‚Üí either works), *deliverability environment* (consumer Gmail/Apple/Outlook ‚Üí either works; corporate Exchange/Outlook with aggressive filtering ‚Üí password as the resilient floor), *recovery footprint* (password reset is the recovery ‚Üí simple; magic-link broken delivery has no recovery ‚Üí password as fallback non-negotiable). Each cell carries either a green check, a red X, or a "neutral" dash. The third column ("both") shows green checks across the row ‚Äî coexistence is the safe-by-default reach when the call isn't decisive, with the trade being a slightly busier sign-in form. Below the table, a one-paragraph closing rule: *year-1 SaaS default is email+password + optional 2FA + Google OAuth; magic-link is a deliberate product call, not a default; magic-link as the only path is a deliberate reach with a deliverability-resilience cost*. A short second beat below the matrix walks the four-step flow as a hand-SVG inside `Figure` ‚Äî submit email, server queues token, user clicks link in inbox, server validates and issues session ‚Äî so the *mechanics* land alongside the *decision*; the mechanics are short because they reuse Concept 5's substrate.

**Engagement.** A `MultipleChoice` round of four product scenarios: *"a B2B SaaS where engineers sign in daily from their dev machines"* (password+optional 2FA; magic-link adds inbox-round-trip friction every day); *"a newsletter platform where authors sign in roughly monthly to schedule posts"* (magic-link wins; password-reset volume on monthly users would swamp support); *"a healthcare app with HIPAA-grade requirements"* (password+TOTP/passkey required ‚Äî the MFA composition column rules out magic-link-only); *"a consumer photo-sharing app where re-engagement is the dominant metric"* (both ‚Äî magic-link as the friction-free re-engagement path, password as the option for users who want it). Wrong-answer feedback names the column of the matrix that decides each. Confirmed by a short `Sequence` ordering of the four-step magic-link flow with one distractor: *the link can be re-used by anyone who has the URL* (no ‚Äî single-use, row deletes on success; the chapter's substrate from Concept 5 enforces this).

**Components.**
- `Figure` wrapping a hand-coded HTML six-row, three-column verdict matrix.
- `Figure` wrapping a hand-SVG four-step flow drawing reusing Concept 5's substrate diagram visual vocabulary.
- `MultipleChoice` for the four-scenario product-fit round.
- `Sequence` for the four-step flow ordering with one substrate-aware distractor.

## Concept 8 ‚Äî TOTP enrollment, recovery codes, and elevation on every toggle

**Why it's hard.** TOTP enrollment is six steps and three of them are silent failure modes if the student gets the order wrong. The *elevation check* ‚Äî `enable` requires a fresh authentication, not just a valid session ‚Äî closes the "stale-session hijacker disables 2FA" hole; without it, an attacker with a stolen cookie can switch off the second factor and lock the legitimate user out. The *show-recovery-codes-once* step is non-negotiable because the codes are stored hashed and unrecoverable post-enrollment; surfacing them later is structurally impossible. The *confirm-the-first-code-before-flipping-`enabled`* step prevents the orphaned-secret state where the user closes the browser between scanning the QR code and verifying, the secret is in their authenticator but the server thinks 2FA is off. The student sees six steps in the docs and the natural reading is "ceremony"; the senior reading is "every step closes a specific hole, miss any and the security tier is broken or the user is locked out."

**Ideal teaching artifact.** Setup + Pattern composite delivered as a **six-step enrollment walkthrough** in `Steps`, with each step carrying a *failure-if-skipped* annotation. Step 1 ‚Äî user clicks "Enable 2FA"; *if skipped*, no entry point, no enrollment. Step 2 ‚Äî `authClient.twoFactor.enable({ password })` ‚Äî *the elevation check*; *if skipped or weakened to "valid session is enough"*, a stolen-cookie attacker disables the security tier silently. Step 3 ‚Äî server returns secret + QR-code URI; client renders QR + base32 fallback. Step 4 ‚Äî user scans with authenticator app, app starts generating codes. Step 5 ‚Äî *user types first code, server verifies* before flipping `enabled: true`; *if skipped*, the user can leave the form with the secret in their app but `enabled: false` on the server, getting them into a state where they think 2FA is on but the next sign-in skips the second factor. Step 6 ‚Äî *recovery codes returned and shown once*; *if skipped*, lost-phone means lost-account, with no recovery path because the codes are stored hashed and the server can't re-show them. Each step's annotation names the bug class the missed step ships. Alongside the walkthrough, a small `Figure`-wrapped hand-SVG draws the *elevation requirement triangle* ‚Äî `enable`, `disable`, and `verify-recovery-code` all sit at the top requiring elevation; the senior rule lands as one line: *every 2FA toggle requires fresh authentication, never a stale session*.

**Engagement.** A `Sequence` drag-to-order of the six steps with two distractor steps placed mid-sequence: *"flip `enabled: true` immediately after rendering the QR code"* (no ‚Äî the user might leave before scanning, the secret is server-side but unverified) and *"surface recovery codes only when the user later requests them from settings"* (no ‚Äî the codes are stored hashed at generation, the server cannot show them again). Wrong placements name the failure mode each shipping move enables. Confirmed by a `MultipleChoice`: "a teammate's PR removes the elevation check on `disable` 'because the user is already signed in.' What's the threat?" ‚Äî the right answer names the stolen-cookie + 2FA-disable hole, ties it to the elevation tier from 9.1 (the principal-state ladder where destructive actions live above active-session), and names the rule (every 2FA toggle is elevated; period).

- `Figure` wrapping a hand-SVG elevation-requirement triangle.
- `Sequence` for the six-step ordering with two misordering distractors.
- `MultipleChoice` for the elevation-on-disable diagnosis.
- `Aside` (`caution`) below the triangle: "every 2FA toggle requires fresh auth. A stolen-cookie attacker who can disable the second factor has defeated the security tier."

**Project link.** The 9.5 project explicitly cuts 2FA; the walkthrough is the *why* the cut is "Chapter 9.3.6 owns this" rather than "we'll add it later," and the *frame* the chapter's quiz (9.3.10) uses to test the elevation invariant.

## Concept 9 ‚Äî The two-step sign-in: `factor-token` as the continuation primitive

**Why it's hard.** When 2FA is enabled, sign-in stops being a single round-trip. The first call validates the password and returns `'requires-second-factor'` *with a `factor-token`*; the second call carries the `factor-token` plus the TOTP code (or the recovery code, or the passkey assertion) and gets the session. The `factor-token` is the *continuation primitive* ‚Äî a short-lived signed token representing "first factor validated, second factor pending" ‚Äî and the student's reflex is to either skip it (try to verify the second factor against the session, which doesn't exist yet) or persist it long-term (it's a session credential in disguise). The misframes ship specific bugs: a long-lived `factor-token` becomes a re-usable bypass; passing the wrong `factor-token` to the verify call lets one user's first-factor validation authenticate another user's second factor. The student needs the `factor-token` as a *named continuation* ‚Äî bound to one first-factor validation, single-use against the second-factor verify, expires in seconds ‚Äî not as an implementation detail.

**Ideal teaching artifact.** Mechanics + Pattern composite delivered as a **state-machine animation of the two-step flow** rendered as a Mermaid `stateDiagram-v2` paired with a short `CodeVariants` showing the wrong-and-right shapes. The state machine has four states ‚Äî *anonymous*, *first-factor-validated* (the state the `factor-token` represents), *authenticated*, *signed-in* ‚Äî and the labeled transitions carry the API calls and the values exchanged: *anonymous ‚Üí first-factor-validated* on `signIn.email({ email, password })` returning `'requires-second-factor'` *with `factor-token`*; *first-factor-validated ‚Üí authenticated* on `twoFactor.verifyTotp({ code, factorToken })` (or `verifyBackupCode`, or the passkey assertion); *authenticated ‚Üí signed-in* on session-row insert + cookie attach. Two failure transitions: *first-factor-validated ‚Üí expired* on `factor-token` TTL elapsed (back to *anonymous*, user retries from password); *first-factor-validated ‚Üí first-factor-validated* on a wrong second-factor code (per-`factor-token` attempt counter increments, locks the continuation after N failures). The `CodeVariants` to the right shows two tabs: *(broken)* the second-factor verify call without the `factor-token` argument ‚Äî the call has no idea which first-factor validation it's continuing, the library returns an error or worse silently authenticates against a wrong context; *(correct)* the verify call with `factor-token` threaded through, the form holds it in pre-auth state across the second-factor prompt, the call consumes it.

**Engagement.** A `MultipleChoice` round of three scenarios: *"the form persists `factor-token` in `localStorage` 'so the user can refresh the page mid-2FA'"* ‚Äî the right answer names the long-lived-continuation bypass (the `factor-token` is a credential; persisting it past the form's lifetime turns it into a session-shaped artifact without the session's revocation properties), then names the alternative (the `factor-token` lives in the form's React state only, refresh means restart the sign-in); *"the verify call doesn't pass `factor-token`, the developer 'just figures the session can be inferred from the cookie'"* ‚Äî the right answer names the no-context bug (no session yet, no inference possible) and ties it back to the state machine; *"the second-factor failure increments a counter on the `factor-token`, not on the user"* ‚Äî correct: the per-`factor-token` counter prevents an attacker who already has the password from brute-forcing 6-digit codes against a single first-factor validation, while keeping the user's broader account-level counter from being tripped by their own typos. Confirmed by a `Matching` exercise pairing the four state transitions against the API call (or expiry trigger) that fires each.

**Components.**
- Mermaid `stateDiagram-v2` for the four-state two-step-sign-in machine with labeled transitions.
- `CodeVariants` for the wrong/right shape of the second-factor verify call with `factor-token` threading.
- `MultipleChoice` for the three-scenario round.
- `Matching` for the four-transition ‚Üî API-call pairing.
- `Aside` (`note`) below the state machine: "`factor-token` is a continuation, not a session. Hold it in pre-auth form state, never persist it. The state machine for passkeys-as-second-factor is the same ‚Äî 9.3.7 swaps the verify call, the rest of the flow is identical."

**Project link.** The 9.5 project cuts 2FA, so the `factor-token` doesn't fire at the keyboard there; the state machine is the *frame* lessons 9.3.6 and 9.3.7 share ‚Äî TOTP and passkey-as-second-factor are different verify-call shapes against the same continuation primitive.

## Concept 10 ‚Äî WebAuthn mental model: the ceremonies, the keypair, origin-bound

**Why it's hard.** Passkeys read as magic until the student sees the cryptographic shape underneath. *Three roles* ‚Äî relying party (SaaS), user agent (browser), authenticator (secure enclave) ‚Äî and *two ceremonies* (registration and authentication), with one keypair generated at registration: the public key registered with the server, the private key locked in the device's secure enclave (or synced via iCloud/Google Password Manager). Authentication is a signed-challenge exchange ‚Äî server issues a random challenge, the device signs it with the private key only after a local biometric or PIN unlock, the server verifies the signature against the stored public key. The *origin-bound* property is the phishing-resistance: the browser binds the assertion to the origin that requested it, so a proxied attacker site at `evil.com` cannot get the device to sign a challenge that the real `app.example.com` will accept. Until the student sees the keypair, the ceremonies, and the origin binding as one mechanism, passkey config (`rpID`, `origin`) reads as ceremony, not as the wiring that makes phishing-resistance work.

**Ideal teaching artifact.** Concept archetype delivered as a **three-actor two-ceremony walkthrough** in `DiagramSequence`. Three lanes drawn vertically ‚Äî *user's device + authenticator* (left), *browser* (middle), *SaaS server* (right). The student scrubs through both ceremonies in two short three-frame sequences. *Ceremony A ‚Äî registration:* frame 1: server generates registration challenge with `rpID: 'app.example.com'`, sends to browser; frame 2: browser presents prompt (Face ID, Touch ID, YubiKey), user authenticates locally; *device generates keypair*, stores private key in secure enclave (or syncs encrypted via iCloud Keychain), returns public key + attestation + credential ID; frame 3: server verifies attestation, checks origin matches `rpID`, stores `(userId, credentialID, publicKey, counter, deviceType, transports, name)` in the `passkey` table. *Ceremony B ‚Äî authentication:* frame 1: server issues a fresh random challenge bound to `rpID`, sends to browser; frame 2: browser presents prompt, user authenticates locally; *device signs the challenge with the stored private key*, returns signature + credential ID + counter; frame 3: server looks up the row by credential ID, verifies the signature against the stored public key, checks origin against `rpID`, checks counter is greater than last-seen (cloned-authenticator detection), issues session. The lanes' visual punchline: *the private key never leaves the device, only signatures cross the wire, the origin is the binding that makes phishing-resistance structural*. A small caption pins the senior rule: *`rpID` must match the actual hostname; `app.example.com` works at that exact subdomain only ‚Äî `example.com` registrations would silently fail at assertion time*.

**Engagement.** A `Matching` exercise pairing six WebAuthn terms against their definitions ‚Äî *relying party* ‚Üî the SaaS site (your app); *user agent* ‚Üî the browser; *authenticator* ‚Üî the secure enclave (TPM, Secure Enclave, TitanM, YubiKey); *credential ID* ‚Üî the public-key handle the server stores; *attestation* ‚Üî the signed metadata proving the authenticator is genuine; *counter* ‚Üî monotonic increment per assertion, detects cloned authenticators. Confirmed by a `MultipleChoice`: "a teammate registers passkeys with `rpID: 'example.com'` because 'the app might move to the apex domain later.' The app is at `app.example.com`. What's the failure mode?" ‚Äî the right answer names the silent registration-vs-assertion mismatch (registration succeeds because `app.example.com` is a subdomain of `example.com`, but assertions originating from `app.example.com` *can* succeed against `rpID: example.com`, *while* assertions from `example.com` would fail; the deeper bug is that moving the app between subdomains later breaks every existing passkey), names the rule (`rpID` is the exact domain the credentials are bound to, treat it as immutable per environment), and ties it to the chapter's substrate (the two-step sign-in from Concept 9 ‚Äî passkey-as-second-factor is a `factor-token`-driven assertion ceremony with the same origin binding).

**Components.**
- `DiagramSequence` hosting the two three-frame ceremony walkthroughs (registration + authentication) across three lanes (device / browser / server) ‚Äî reuses 9.1 Concept 7's three-lane shape.
- `Matching` for the six-term WebAuthn vocabulary pairing.
- `MultipleChoice` for the `rpID` mismatch diagnosis.
- `Aside` (`note`) below the sequence: "the private key never leaves the device, only signatures cross the wire, the origin binding makes phishing-resistance structural. `rpID` is the immutable scope per environment."

**Project link.** The 9.5 project cuts passkeys; the ceremonies walkthrough is the *why* lesson 9.3.7 doesn't re-derive the protocol from scratch and the *frame* the quiz (9.3.10) uses to test the `rpID` / origin-binding rule.

## Concept 11 ‚Äî Synced versus device-bound passkeys: the trade

**Why it's hard.** Passkeys come in two flavors and the student will pick the wrong default if they don't see the trade. *Synced* passkeys (iCloud Keychain, Google Password Manager, 1Password) replicate the private key across the user's devices via cloud sync, encrypted end-to-end ‚Äî the user signs in on a new device without re-registering, recovery is trivial, but cloud-account compromise yields the credential. *Device-bound* passkeys (YubiKey, Windows Hello credential without sync) keep the private key on the device only ‚Äî losing the device loses the credential, recovery is hard, but cloud-account compromise can't reach the credential. The naive default is to pick one over the other on cryptographic purity grounds (device-bound feels stronger), but the *operational trade* is the deciding axis: consumer SaaS where users lose phones constantly *needs* synced-by-default with recovery codes as the floor; enterprise/regulated SaaS where the ownership chain matters *needs* device-bound as a senior reach with explicit recovery procedure. The student needs the trade as a *use-case-driven verdict*, not as a feature comparison.

**Ideal teaching artifact.** Decision archetype delivered as a **two-column comparison panel with a recovery-path footer** rendered as a hand-coded HTML table inside `Figure`. Two columns ‚Äî *synced (consumer default)* and *device-bound (enterprise reach)*. Five rows: *where the private key lives* (cloud-encrypted in iCloud Keychain / Google PM / 1Password vs. secure enclave on one device only); *new-device sign-in* (works automatically ‚Äî credential follows the user vs. requires re-registration on every new device); *device loss* (low-friction ‚Äî sign in from another device that has the synced credential vs. high-friction ‚Äî credential is gone, need recovery codes or another sign-in method); *cloud-account compromise* (the attacker gets the credential vs. the credential is unaffected); *typical user fit* (consumer SaaS, frequent device-switching vs. high-stakes admin / regulated industries / users who explicitly want the harder property). Below the table, a *recovery-path footer* lands the non-negotiable: *every passkey enrollment surfaces recovery codes alongside the credential* (synced or device-bound); a user with one synced passkey and a lost iCloud account is in the same place as a user with one device-bound passkey and a lost device ‚Äî the recovery codes are the universal fallback, and "no recovery" is an explicit opt-in the user has to acknowledge.

**Engagement.** A `MultipleChoice` round of three scenarios: *"a consumer photo-app where users routinely upgrade phones every two years"* ‚Äî synced (the new-iPhone sign-in just works, the user expects continuity); *"a treasury-management SaaS where ownership chain is the audit requirement"* ‚Äî device-bound (the credential being un-syncable is the property the auditor wants), with explicit recovery via support-driven identity verification; *"a consumer SaaS where users will not engage with recovery codes"* ‚Äî synced as the default, recovery codes still surfaced at enrollment as the fallback, plus a second sign-in method (password or magic-link) as the recovery floor; the right answer names the trade explicitly and never picks "passkeys as the only credential without recovery." Confirmed by a short `MultipleChoice`: "a user with one synced passkey loses their iCloud account ‚Äî what's the recovery path?" ‚Äî the right answer names the recovery-codes-from-enrollment as the structural fallback, names the implication (every passkey enrollment must surface recovery codes), and ties it back to Concept 8's "show recovery codes once" non-negotiable.

**Components.**
- `Figure` wrapping a hand-coded HTML two-column, five-row comparison table with the recovery-path footer below. Single-use composition; the comparison shape recurs in spirit across the chapter's decision concepts (Concept 7's magic-link matrix, this) ‚Äî fine to standardize on hand-coded HTML inside `Figure`.
- `MultipleChoice` for the three-scenario fit round.
- `MultipleChoice` for the recovery-path follow-up.
- `Aside` (`note`) below the table: "synced is the consumer default, device-bound the enterprise reach. Recovery codes are universal ‚Äî passkeys without a recovery path is a deliberate opt-in, never a default."

**Project link.** The 9.5 project cuts passkeys; the trade table is the *why* lesson 9.3.7 ships the synced default with `discoverableCredentials: 'preferred'` and the *why* the chapter's quiz (9.3.10) tests recovery-codes-on-passkey-enrollment as a non-negotiable.

## Concept 12 ‚Äî OAuth callback: the lookup tree, and the redirect-URI exact-match trap

**Why it's hard.** The OAuth callback runs a three-fork lookup tree the student has to be able to draw from memory: (1) lookup by `(providerId, accountId)` ‚Äî does an `account` row already link this Google identity to a user? Yes ‚Üí sign in; (2) lookup by email ‚Äî does a `user` with this email already exist? Yes + linking enabled + provider trusted ‚Üí link; (3) neither ‚Üí create new `user` + `account`. Mis-modeling the tree ships specific bugs: skip step 1 and a returning Google user with a renamed email creates a duplicate account; skip step 2 and a password-first user who later clicks "Sign in with Google" gets a duplicate account split off from their original; conflate steps 2 and 3 and the linking-vs-create call gets reversed. Compounding the lookup-tree confusion is the *redirect URI exact-match rule* ‚Äî OAuth 2.1 requires the registered URI to match the callback URI down to the trailing slash, and the trap is environment drift (`localhost:3000/api/auth/callback/google` vs `app.example.com/api/auth/callback/google/` with a trailing slash), where dev works and prod 500s on the first user sign-in. The student needs the tree as a *decision diagram* and the exact-match rule as a *config-time discipline*, not as two unrelated topics.

**Ideal teaching artifact.** Pattern + Mechanics composite. *Beat one ‚Äî the lookup tree:* a `Figure`-wrapped hand-SVG drawing the three-fork tree top-down. Root: callback fires with `code` exchanged for tokens, server reads `userinfo`. Fork 1: query `account WHERE providerId = 'google' AND accountId = <google sub>`. Found ‚Üí sign in this user, end. Not found ‚Üí continue. Fork 2: query `user WHERE email = <google email>`. Not found ‚Üí *create user + account*, sign in, end. Found ‚Üí check linking config: `accountLinking.enabled` AND `'google' IN trustedProviders` AND `email_verified` from Google. All true ‚Üí *insert account row, link to existing user*, sign in, end. Any false ‚Üí return `account-not-linked`, surface "this email signs in with X ‚Äî sign in there first" to the user. Each leaf carries the `account`-table change it produces (insert one row vs. zero) and the user-facing copy. *Beat two ‚Äî the redirect URI trap:* a small `CodeVariants` showing two tabs: *(broken)* one tab where dev uses `http://localhost:3000/api/auth/callback/google` and prod uses `https://app.example.com/api/auth/callback/google/` (trailing slash) ‚Äî registered at the provider's console as different URIs, the trailing-slash one was forgotten. The dev sign-in works; prod 500s on the callback because the provider's redirect doesn't match any registered URI. *(correct)* same env-by-env paths, registered explicitly at the provider with no trailing slash, normalized at the registration step. A short caption names the rule: *register every environment's redirect URI explicitly, normalize the trailing slash, never share credentials across environments ‚Äî exact-string-match means a trailing slash is a different URI*.

**Engagement.** A `Sequence` drag-to-order of the seven-step OAuth callback flow with the lookup tree as the central beat: *(1) provider redirects browser to callback URI with `?code=...&state=...`*, *(2) server verifies `state` against pre-auth session*, *(3) server exchanges `code` for tokens via back-channel POST*, *(4) server reads `userinfo` from `id_token` (signature-verified) or userinfo endpoint*, *(5) server queries `account` by `(providerId, accountId)`*, *(6) on miss, server queries `user` by email and applies linking rules*, *(7) server creates / links account, issues session, redirects to `callbackURL`*. Two distractors map to common misconceptions: *"server creates a user before checking the email lookup"* (no ‚Äî the lookup decides whether to create or link; reversing the order spawns duplicate accounts) and *"server skips the `(providerId, accountId)` check because email lookup is sufficient"* (no ‚Äî the email might have changed at the provider since the original sign-in, the `(providerId, accountId)` is the stable identity key). Confirmed by a `MultipleChoice`: "a teammate's PR registers `http://localhost:3000/api/auth/callback/google` for dev and `https://app.example.com/api/auth/callback/google/` for prod (trailing slash). Dev signs in fine; prod 500s on the first callback. What's the failure mode?" ‚Äî the right answer names exact-string-match, names the silent failure shape (registered URIs are checked exactly, the trailing-slash URI doesn't match the URI the provider redirects to), names the fix (register both with and without the trailing slash, or normalize the redirect URI at registration time).

**Components.**
- `Figure` wrapping a hand-SVG three-fork lookup tree with each leaf annotated with the `account`-table change it produces.
- `CodeVariants` for the two-environment trailing-slash trap (broken + correct).
- `Sequence` for the seven-step callback flow with two misconception distractors.
- `MultipleChoice` for the trailing-slash diagnosis.
- `Aside` (`caution`) below the tree: "the lookup is `(providerId, accountId)` first, email second, create third. Reversing the order ships duplicate accounts. Redirect URIs are exact-string-match ‚Äî trailing slashes count, environments are registered explicitly."

## Concept 13 ‚Äî Account linking as a trust transfer: `trustedProviders`, `allowDifferentEmails`, and the unlink guard

**Why it's hard.** Linking sounds like a UX feature ‚Äî Ada signs up with email+password, later clicks "Sign in with Google" with the same email, the app recognizes her and links the providers. The senior reading is the inverse: linking is a *trust transfer* ‚Äî the system is saying "whoever controls this Google identity is the same person who knows this password." If `email_verified` is wrong (a provider bug, a Workspace edge, a domain takeover), an attacker who controls Google `ada@acme.com` can sign in as Ada *without her password*. The defenses are three knobs: `trustedProviders` (only providers with strong `email_verified` claims qualify ‚Äî Google and GitHub yes, Twitter/X no), `allowDifferentEmails` (false by default ‚Äî linking requires email match, the implicit-trust signal; true is a deliberate reach for products where the user explicitly wants multi-email), and the *unlink guard* that refuses to remove the user's only sign-in method. The student's reflex is to enable linking with all defaults and ship; the senior reflex is to treat each knob as a security decision and surface "we linked a new provider" as a notification email. Without the trust-transfer framing, linking reads as plumbing.

**Ideal teaching artifact.** Decision archetype delivered as a **three-knob configuration walkthrough with a threat-and-defense panel** rendered as a `TabbedContent`. Three tabs, one per knob. *Tab 1 ‚Äî `trustedProviders`:* a small hand-coded HTML table with three columns ‚Äî *provider*, *`email_verified` claim*, *trust verdict*. Google: `email_verified: true` for consumer accounts, sometimes false for Workspace ‚Äî trusted with senior caveat. GitHub: provides verified-email list with `primary` + `verified` flags, pick the primary verified ‚Äî trusted. Twitter/X: no `email_verified` claim at all ‚Äî *not trusted*, do not place on the list. The threat panel names the attack the allowlist closes: enabling linking with no `trustedProviders` means *every provider is implicitly trusted*, including providers whose email claims are unverified or weak; an attacker who controls a Twitter account claiming Ada's email can sign in as Ada without her password. *Tab 2 ‚Äî `allowDifferentEmails`:* a comparison panel ‚Äî *false (senior default)* enforces email match, the implicit-trust signal that "this is the same person" rests on the email being a known identifier; *true (deliberate reach)* allows linking a personal Google to a work-email account, useful for products where the user explicitly wants multi-email, but loses the email-match signal entirely ‚Äî must be paired with explicit link-from-settings (no link-on-sign-in) to preserve the consent step. *Tab 3 ‚Äî the unlink guard:* a four-frame walkthrough showing Ada with `'credential'` + Google linked, removing the password (no `'credential'` row), then trying to unlink Google ‚Äî the library refuses, surfaces "you can't unlink your only sign-in method, set up a password first." Each tab closes with the threat the knob defends and the senior default. Below the tabs, a closing rule: *linking is a trust transfer; every knob is a security decision; the unlink guard prevents the user from locking themselves out, which the library doesn't enforce automatically without the right config*.

**Engagement.** A `MultipleChoice` round of three scenarios: *"team enables linking with `accountLinking: { enabled: true }` and ships"* ‚Äî the right answer names the missing `trustedProviders` allowlist (every provider becomes implicitly trusted), names the bug (Twitter sign-in could link to Ada's account if she has Twitter `ada@acme.com`), and names the senior-default config (`enabled: true, trustedProviders: ['google', 'github'], allowDifferentEmails: false`); *"team enables `allowDifferentEmails: true` to support users with multi-email setups, with link-on-sign-in still on"* ‚Äî the right answer names the dangerous combination (implicit linking with a different email collapses both defenses at once ‚Äî no email match required AND linking happens implicitly without explicit user consent), names the fix (`allowDifferentEmails: true` is reserved for explicit link-from-settings only); *"the unlink endpoint succeeds even when it leaves the user with no sign-in method"* ‚Äî the right answer names the lockout bug, names the guard that closes it (refuse the unlink and surface "set up an alternate sign-in first"). Confirmed by a `Buckets` sort: ten config snippets into *safe-to-ship* / *missing-defense* / *dangerous-combination* ‚Äî examples ranging from the senior triple to `allowDifferentEmails: true` without link-from-settings to no `trustedProviders` to the senior triple with the unlink guard, with feedback naming the threat each falls under.

**Components.**
- `TabbedContent` for the three-knob walkthrough, one tab per knob, with hand-coded HTML tables and panels inside.

- `Buckets` for the ten-config safe/missing-defense/dangerous-combination sort.

## Open pedagogical questions

- Concept 5's two-beat composite (substrate diagram + token-shape walkthrough) sits inside lesson 9.3.3 but the substrate *frames* lessons 9.3.4, 9.3.5, and 9.3.6. The open question is whether the substrate diagram lives in 9.3.3 only (with later lessons referencing it inline) or repeats as a thumbnail at the head of each consuming lesson. The first reads cleaner (no visual repetition); the second reinforces the recurrence and trades a small amount of repetition for stronger recall. The current default is the first; field testing on inter-lesson recall would settle the call.
- Concept 7's verdict matrix and Concept 11's comparison table are both six-row, multi-column hand-coded HTML tables inside `Figure`. They sit two lessons apart and serve different decisions (when to ship magic-link vs. synced-vs-device-bound passkeys). The open question is whether the visual repetition of the table shape risks reading as filler or whether the consistent shape across decision concepts is itself a teaching signal. Current default is to keep both as tables on the grounds that the *form* recurs across decision concepts in the chapter and the recurrence is the calibration signal; if the chapter authors find them hard to distinguish at a glance, the second can compress to a paragraph plus a one-row use-case-fit table.
- Concept 9's state machine could be rendered as a hand-SVG instead of Mermaid `stateDiagram-v2` if the labeled-transitions density (state name + API call + value passed + counter behavior on failure) overflows what Mermaid renders cleanly. Mermaid is the default per the guidelines; the open question is whether this particular state machine's density tips into the "layout itself carries meaning" territory hand-SVG owns. If it does, the hand-SVG version trades some authoring overhead for clearer per-transition annotation space.
