# Chapter 9.2 — Better Auth setup: pedagogical approach

## Concept 1 — The library surface: catch-all handler, `auth.api.*` server side, `authClient.*` browser side

**Why it's hard.** Better Auth ships one library that exposes itself through three doors at once — a server module, a browser module, and a Next.js route handler — and the student's first reflex is to import the wrong one in the wrong place. Pulling `auth` into a Client Component bundles the server library into the browser and only `server-only` catches it (with an opaque error if the student hasn't met the pattern). Calling `authClient` from a Server Component fails for the inverse reason. The teaching has to install the boundary as a wiring picture before the student writes the first import.

**Ideal teaching artifact.** Concept archetype delivered as a **wiring map with three doors and four call sites**. A single hand-SVG diagram inside `Figure` shows the running app as three vertical zones — *Browser (Client Component)*, *Next.js server (Server Component, Server Action, route handler, proxy)*, and *Better Auth library* on the right. Three doors connect them: the browser zone holds `authClient` which posts to `/api/auth/*`; the server zone holds `auth.api.*` which calls the library directly; the catch-all `app/api/auth/[...all]/route.ts` is drawn as the public seam where browser HTTP traffic meets the server library. Each door is annotated with what it *can't* do — the `auth` arrow into the browser zone is drawn crossed-out with the legend *"`server-only` refuses the import"*; the `authClient` arrow into the server zone is crossed-out with *"the cookies aren't on the server-side request"*. The `[...all]` segment label points at a small list of paths (`sign-up/email`, `sign-in/email`, `callback/google`, `session`) that the same one route handler dispatches — the senior reflex *never write individual route files for sign-in / sign-up / callback* lands as a label, not a paragraph. The student leaves with a map they can point at when reviewing a PR that imports the wrong door.

**Engagement.** A `Buckets` sort: ten code snippets into three buckets — *belongs in a Server Component / Action / handler* (uses `auth.api.*`, runs on the server, has access to `headers()`), *belongs in a Client Component* (uses `authClient.*`, runs in the browser), *broken by definition* (`auth` imported into a `'use client'` file; `authClient.useSession()` called from a Server Component; an individual `route.ts` for `/api/auth/sign-in/email` shadowing the catch-all). Wrong-answer feedback names the boundary the misimport crosses and which guard catches it (`server-only`, runtime "headers()" failure, route precedence).

**Components.**
- `Figure` wrapping a hand-SVG three-zone wiring map with the catch-all seam labeled and the two crossed-out arrows. Single-use within the chapter; the same picture is the *why* every Unit 9 chapter respects the boundary, but the diagram itself doesn't recur — hand-SVG is the right scope.
- `Buckets` for the ten-snippet three-bucket sort.
- `Aside` (`note`) below the diagram: "the catch-all handler is the contract. The browser posts to `/api/auth/*`; the server calls `auth.api.*` directly; never write individual `route.ts` files inside `/api/auth/`."

**Project link.** The 9.5 project's `lib/auth.ts` and `lib/auth-client.ts` are the two files the diagram pre-explains; lesson 9.5.3 mounts the catch-all handler and the student already knows why one route file covers every endpoint.

## Concept 2 — `nextCookies()` and the canonical silent-fail

**Why it's hard.** Server Actions in Next.js 16 don't auto-flush `Set-Cookie` headers the way route handlers do — the action returns 200, the server-side `signUp` call writes the session row, and *no cookie reaches the browser*. The student refreshes, sees no signed-in state, and assumes the call failed. The bug is uniquely cruel because every individual piece works: the row is in Postgres, the Action returned successfully, the network tab shows 200. The fix is one line in the plugin array — `plugins: [nextCookies()]` — but the teaching has to make the failure mode *visible* so the student recognizes it on sight in their own code or a teammate's PR, even months later.

**Ideal teaching artifact.** Pattern archetype delivered as a **wrong-by-default before/after replay** in `CodeVariants`. Tab one *(broken)*: the `betterAuth({ … })` config without `nextCookies()` in the `plugins` array, paired with a network-tab screenshot annotation showing the Action's response — `200 OK`, an empty `Set-Cookie` line, the session row visible in a side Postgres readout, the browser's cookie jar empty. The student is asked to predict what happens on the next page render: the layout calls `getCurrentUser()`, the cookie isn't there, the user is treated as anonymous. Tab two *(fixed)*: same config with `plugins: [nextCookies()]` added on a highlighted line, the network tab now showing the `Set-Cookie: __Host-better-auth.session_token=...` header attached to the same Action response, the cookie jar populated. A short paragraph below names the mechanism — the plugin hooks Next.js's Server Action response pipeline and copies the cookies the library tried to set onto the outgoing response. The artifact ends with the senior reflex as a one-liner: *if a Better Auth call appears to succeed but the user stays signed-out, check `nextCookies()` first.*

**Engagement.** A `MultipleChoice` post-replay: "the team reports `signUpEmail` is silently failing — the row appears in Postgres, the Action returns 200, but the user isn't signed in on the next render. What's the single most likely cause?" The four options name the four wrong diagnoses each shape (cookie attribute mismatch, CORS, missing handler mount, missing `nextCookies()`); the right answer ties the symptom to the plugin and explains why the symptom is uniquely *silent* — every other failure mode produces a non-200 response or a console error.

**Components.**
- `CodeVariants` for the broken/fixed two-tab replay, with a hand-SVG-inside-`Figure` panel embedded in each tab showing the network-tab + Postgres-row + cookie-jar triple.
- `MultipleChoice` for the diagnosis confirmation.
- `Aside` (`caution`) below the variants: "no plugin, no cookie. Server Actions don't flush `Set-Cookie` on their own; `nextCookies()` is the single line that makes every downstream auth feature visible to the browser."

**Project link.** The 9.5 project's first verifiable artifact (lesson 9.5.3) is the `__Host-` cookie landing in the browser after sign-up; the replay is the *why* lesson 9.5.3 explicitly checks for `nextCookies()` in the plugin array before declaring the spine wired.

## Concept 3 — The four-table decomposition: why `password` lives on `account`, not `user`

**Why it's hard.** A junior reads "auth schema" and reaches for one `users` table with `email`, `password_hash`, `created_at` — the shape every framework tutorial showed for a decade. Better Auth's four-table decomposition (`user`, `session`, `account`, `verification`) reads as overkill until the student sees what the single-table model breaks: *change password* mutates the user row (mixing identity with proof), *link Google* requires a column added per provider (or a JSON blob), *unlink Google* needs to know whether the password account also exists or the user gets locked out, and *email verification before user exists* has nowhere to live. The decomposition isn't ceremony — it's the shape that survives every Chapter 9.3 flow without re-shaping. Until the student feels the flow-by-flow stress against the wrong model, the right model reads as gratuitous.

**Ideal teaching artifact.** Concept archetype delivered as a **misframe-first stress test followed by the canonical ER picture**. The lesson opens with a one-table strawman drawn as a small `Figure` — `users(id, email, password_hash, google_id, github_id, email_verified, …)` — then walks four short flow scenarios as a hand-SVG-inside-`Figure` annotated table: (1) *user signs up with Google then later adds a password* — the strawman has no `password_hash` row to attach the password to without writing it onto the user, blurring identity-and-proof; (2) *user links a second OAuth provider* — strawman adds another column, then a third; (3) *user unlinks Google but keeps password* — strawman has no way to know if any other proof method exists without reading every nullable column; (4) *user requests email verification before any account exists* — strawman has no row to attach the token to until the user is created, and creating the user before verification is the identification-equals-authentication misframe from 9.1. After the four stress scenarios, a single Mermaid `erDiagram` draws the canonical four-table shape with the cardinalities (`user 1—* account`, `user 1—* session`, `verification` standalone) and a label on each table naming what it owns: `user` = identity, `session` = "this browser is currently signed in," `account` = "this is one way the user can prove they're them," `verification` = "this token proves something short-lived." The cascades (`session.userId ON DELETE CASCADE`, `account.userId ON DELETE CASCADE`, `verification` keyed by `identifier` not `userId`) land as labels on the relationship lines.

**Engagement.** A `Matching` exercise pairing six rows from "what just happened" against the table that owns the change — *user changes password* ↔ `account` (update one row); *user signs out from one device* ↔ `session` (delete one row); *admin deletes user* ↔ cascade through `session` and `account`; *user requests email verification before signing up* ↔ `verification` (no FK to user, the row exists *before* the user does); *user adds a passkey* ↔ `account` (insert another proof row); *user changes their displayed name* ↔ `user` (the only column on `user` that isn't pure identity). Confirmed by a `MultipleChoice`: "a teammate's migration moves `password_hash` onto `user` to 'simplify the joins.' What does the senior reviewer ask?" — the right answer names the four flows the change breaks, with the unlink-without-lockout flow as the load-bearing one.

**Components.**
- `Figure` wrapping the one-table strawman as the misframe opener.
- `Figure` wrapping a hand-SVG four-scenario stress table — each row showing the flow on the left and what the strawman breaks on the right.
- Mermaid `erDiagram` for the canonical four-table shape with cardinalities and cascade labels.
- `Matching` for the six-flow ↔ table-owns-it pairing.
- `MultipleChoice` for the migration-PR confirmation.

**Project link.** The 9.5 project ships exactly the four canonical tables (lesson 9.5.3 generates them via the CLI); the stress-test concept is the *why* the project doesn't second-guess the shape and the *why* lesson 9.5.6 verifies sign-up by reading rows from `user`, `account`, and `verification` separately.

## Concept 4 — CLI generates, Drizzle Kit migrates: one history, two tools

**Why it's hard.** Better Auth ships its own `migrate` CLI; the project already has Drizzle Kit. The student's reflex is to use whichever tool's docs they read most recently — and using both fragments the migration history into two ledgers that don't see each other, leaving the schema in a state where neither tool can reliably re-apply it. The senior call is *Drizzle Kit owns migrations end-to-end, Better Auth's CLI is generation-only* (re-run on every plugin change, output reviewed like a PR). Until the student sees the two-tool fragmentation as a real failure mode, "just run `better-auth migrate` once to get going" looks harmless.

**Ideal teaching artifact.** Decision archetype delivered as a **side-by-side comparison panel** with a small worked-failure underneath. A `TabbedContent` with three tabs: *(1) Generate via Better Auth CLI, migrate via Drizzle Kit (default)*, *(2) Use Better Auth migrate CLI for everything*, *(3) Hand-write the schema*. Each tab carries the same four rows in a small hand-coded HTML table — *who owns the schema source* / *what re-runs on plugin changes* / *who generates the migration SQL* / *what reviews the SQL before apply*. Tab one shows the senior path: schema lives in `src/db/schema/auth.ts` (CLI-generated, manually reviewed, treated as code), migrations generated by `pnpm drizzle-kit generate` and reviewed before `migrate`. Tab two shows the fragmentation: Better Auth's CLI writes its own migration files outside the Drizzle Kit `migrations/` directory, the two histories drift, the next `drizzle-kit generate` proposes to drop tables it doesn't know about. Tab three is the dead-end: hand-writing the columns means reading Better Auth's source for every required field name and type — the moment a plugin is added, the schema falls out of sync silently and the first failing call is opaque. Below the tabs, a one-paragraph closing rule: *one history, one tool, regenerate the schema when plugins change, migrate via Drizzle Kit always.*

**Engagement.** A `Sequence` drag-to-order of the canonical workflow: *(1) install Better Auth*, *(2) write the `lib/auth.ts` config naming the plugins*, *(3) run `npx @better-auth/cli generate` against `lib/auth.ts`*, *(4) review the generated `src/db/schema/auth.ts` file*, *(5) run `pnpm drizzle-kit generate` to produce the SQL migration*, *(6) review the generated SQL*, *(7) run `pnpm drizzle-kit migrate` to apply*. Two distractor steps test the misconception — *run `npx @better-auth/cli migrate`* (wrong path, fragments the history) and *skip the schema review because the CLI is reliable* (the CLI is generation, the review catches plugin-added columns the team doesn't want yet). Wrong placements name the failure mode each shipping move enables.

**Components.**
- `TabbedContent` for the three-path comparison with the hand-coded HTML four-row table inside each tab.
- `Sequence` for the seven-step workflow ordering with two misconception distractors.
- `Aside` (`note`) below the comparison: "schema is the source of truth. The Better Auth CLI generates the schema file; Drizzle Kit owns the migration history. One ledger, one tool, one review pass per change."

**Project link.** The 9.5 project's lesson 9.5.3 runs exactly this workflow — the student arrives knowing why the migration goes through Drizzle Kit and not the Better Auth CLI, and lesson 9.5.6 verifies the generated SQL against the four canonical tables before declaring the spine done.

## Concept 5 — `expiresIn`, `updateAge`, `freshAge`: three knobs, three downstream features

**Why it's hard.** The three numbers look interchangeable in the docs (all are durations, all default to a number of days), and the student's instinct is to set all three to the same value or copy whatever the example uses. The trap: each number gates a different downstream feature, and getting them wrong produces specific bugs — `expiresIn` set too long means stolen cookies survive too long; `updateAge` set too short means write amplification on every request; `freshAge` set too long means destructive actions skip re-auth and the elevation tier from 9.1 dies. The student needs the three numbers attached to the three features they feed, not to "session lifetime" as one undifferentiated knob.

**Ideal teaching artifact.** Reference archetype delivered as a **three-column knob-to-feature reference card** rendered as a hand-coded HTML table inside `Figure`. The table has three rows — one per knob — and four columns: *knob name*, *what it controls*, *which downstream feature reads it*, *senior default + threshold to flip*. `expiresIn`: absolute session lifetime / Chapter 9.4.3's "session expired" detection / 7 days for low-sensitivity, 30 days for typical SaaS, 24h for compliance. `updateAge`: sliding-renewal cadence / the UPDATE on `session.updatedAt` that extends `expiresAt` / 1 day default; shortening trades write amplification for tighter expiry, never set to seconds. `freshAge`: "this session is freshly authenticated" window / Chapter 9.4.2's elevation gate (change password / change email / disable 2FA require re-auth past this window) / 1 day default; shortening for high-sensitivity destructive actions, never disable. Below the table, a small annotated `Code` block shows the `session: { expiresIn: 60 * 60 * 24 * 30, updateAge: 60 * 60 * 24, freshAge: 60 * 60 * 24 }` config block with each value's downstream consumer noted as an inline comment.

**Engagement.** A `Matching` exercise pairing six scenarios against the knob that addresses each — *a stolen cookie keeps working for weeks* ↔ `expiresIn` (the absolute cap); *the database shows an UPDATE on `session` for every page load* ↔ `updateAge` (set too short); *the user changed their password and was immediately allowed to disable 2FA without re-auth* ↔ `freshAge` (set too long, elevation gate skipped); *the user gets logged out mid-task three days into a 7-day session* ↔ `updateAge` (sliding renewal isn't firing because activity is slower than the cadence — or `disableSessionRefresh` is true); *the compliance team needs forced re-login every 24h regardless of activity* ↔ `disableSessionRefresh: true` + `expiresIn: 24h`; *destructive admin actions need re-auth even moments after sign-in* ↔ `freshAge` (with a separate elevation check at the action boundary, named-not-built here). Confirmed by a `MultipleChoice`: "the team sets `expiresIn: 1 year` for UX. What does the senior call out?" — sessions that never expire are sessions that never rotate; pick a finite window and lean on `updateAge` for sliding renewal.

**Components.**
- `Figure` wrapping a hand-coded HTML three-row knob-to-feature reference table. Single-use shape but the *card* form recurs across the chapter (cookie cache trade in Concept 6 uses the same shape) — fine to standardize on hand-coded HTML inside `Figure`.
- `Code` block showing the canonical `session: {…}` config with downstream-consumer comments.
- `Matching` for the six-scenario knob pairing.
- `MultipleChoice` for the year-long-session confirmation.

**Project link.** The 9.5 project sticks to Better Auth defaults for the three knobs (lesson 9.5.3 doesn't override `session`); the reference card is the *why* the defaults are accepted as defaults rather than copied without thought, and the *why* lesson 9.5.6's checklist names "session config reviewed against defaults" as one of the production-readiness items.

## Concept 6 — The cookie cache: when zero DB hits earns the staleness window

**Why it's hard.** The cookie cache is a textbook second-order optimization — it removes a database round-trip from every authenticated page load by serializing a signed copy of `{ user, session }` into a second cookie that the server reads instead of querying. The trade is asymmetric: reads get faster everywhere, but server-side changes (admin revokes a session, user changes their email, org switch) take *up to the cache TTL* to propagate. The student's reflex is binary — "caching is good, turn it on" — and the senior reflex is to weigh revocation latency against the read load per surface. Without seeing the staleness collision against a destructive-action surface, the cache reads as a free win.

**Ideal teaching artifact.** Decision archetype delivered as a **two-state side-by-side comparison with a staleness-collision walkthrough**. A `TabbedContent` with two tabs — *cookie cache off (default-DB)* and *cookie cache on, 5-minute TTL (default-on for 2026 templates)*. Each tab carries a small hand-SVG-inside-`Figure` showing one page-load request: the *off* tab draws an arrow from the request through `auth.api.getSession` to a database query and back; the *on* tab draws the same request resolving entirely against the signed cookie payload, with the database lane greyed-out. Below each tab, a short ledger: *latency per session read*, *DB load per signed-in user per page*, *propagation delay on revoke*, *propagation delay on user-data change*. Off: ~1ms DB round-trip, 1 query per session read, instant revoke, instant data change. On: ~0.1ms cookie verify, 0 queries per session read until TTL expiry, *up to 5 minutes* revoke delay, *up to 5 minutes* user-data change delay. Below the two tabs, a third panel runs the staleness-collision walkthrough as a four-frame hand-SVG inside `Figure`: *(1) admin opens `/admin/users` and revokes user Bob's session at 10:00:00*; *(2) Bob's browser still holds the cached cookie payload, valid until 10:04:30 by signature*; *(3) Bob navigates to `/dashboard` at 10:01:00 — the server reads the cached payload, sees a valid session, and serves the page*; *(4) at 10:04:30 the cache TTL expires, the next request hits the DB, the row is gone, 401, redirect to sign-in.* The walkthrough names the surfaces where this collision is unacceptable (admin panels, billing changes) and the mitigation (force-refresh the session at the action boundary, or shorten `maxAge`, or disable for that surface).

**Engagement.** A `MultipleChoice` round of three scenarios after the comparison: *a marketing-page-heavy SaaS with thousands of authenticated reads per second per user — cookie cache?* (on, the read-load wins); *a compliance app where session revoke must propagate within 30 seconds — cookie cache?* (off, or `maxAge: 30` if the read load demands it, or refresh-at-action-boundary mitigation); *a billing-changes admin surface where a fired employee must lose access immediately — cookie cache?* (the destructive surface explicitly forces a session re-read regardless of cache, the cache stays on for everything else). Wrong-answer feedback names the trade each misconfiguration accepts.

**Components.**
- `TabbedContent` for the two-state comparison with hand-SVG-inside-`Figure` panels and the four-row ledger inside each tab.
- `Figure` wrapping a four-frame hand-SVG staleness-collision walkthrough.
- `MultipleChoice` for the three-scenario surface-fit round.
- `Aside` (`caution`) below the walkthrough: "cookie cache on is the 2026 template default; the default is *not* universal. For destructive-action surfaces, force a session refresh at the action boundary or accept the revocation latency."

**Project link.** The 9.5 project leaves the cookie cache at Better Auth's default; the comparison is the *why* the project's lesson 9.5.6 production-checklist line *"cookie cache trade reviewed against destructive surfaces"* is a real review step rather than a checkbox.

## Concept 7 — One `getSession` call, five surfaces

**Why it's hard.** Sessions can be read in five places — `proxy.ts` middleware, layouts, Server Components, Server Actions, route handlers — and the student's instinct is to assume each surface needs a different API. They don't. The single call shape `await auth.api.getSession({ headers: await headers() })` works identically in all five, returns the same `{ user, session } | null`, and the differences live in *what each surface does with the null* (proxy redirects, layout calls `requireUser`, Server Action returns the `unauthenticated` Result, route handler returns 401). Until the student sees the same call drawn into all five surfaces from one source, they reach for `getServerSession` flashbacks and write five wrappers where one helper suffices.

**Ideal teaching artifact.** Pattern archetype delivered as a **five-surface call-site map** rendered as a hand-SVG inside `Figure`. The diagram draws five small panels arranged around a central call-site box — `await auth.api.getSession({ headers: await headers() })` as the trunk, with five branches reaching into *(1) `proxy.ts`* (panel content: `if (!session) redirect('/sign-in')` on matched routes), *(2) layout.tsx* (panel content: `const user = await requireUser(); return <Header user={user} />`), *(3) Server Component leaf* (panel content: `const user = await getCurrentUser(); return user ? <UserMenu /> : <SignInLink />`), *(4) Server Action* (panel content: `const user = await getCurrentUser(); if (!user) return { kind: 'unauthenticated' }`), *(5) route handler* (panel content: `const session = await auth.api.getSession({ headers: await headers() }); if (!session) return new Response(null, { status: 401 })`). Each panel is annotated with what the surface does with `null` — *redirect*, *redirect via `requireUser`*, *render anonymous UI*, *return Result discriminant*, *401 Problem Details*. The diagram's punchline is structural: one call, five reactions, never five calls. A small caption underneath names what *isn't* on the map — `useSession()` from `authClient` lives on the Client Component side and drives display only, never gates.

**Engagement.** A `Tokens` exercise on the canonical call line `await auth.api.getSession({ headers: await headers() })` — the student clicks the load-bearing tokens and gets a one-line reveal per token: `await` (the call is async, missing `await` returns a Promise that's truthy and looks signed-in); `auth.api.getSession` (server-side library, never `authClient`); `headers()` (from `next/headers`, the cookie-bearing source); `await headers()` (Next.js 16 made `headers()` async — the canonical "I'm signed in but the layout says I'm not" bug is `headers()` without the inner `await`, the call silently sees no cookies). Confirmed by a `Buckets` sort: ten code lines into five buckets named for the surfaces — the student matches each line to the surface it belongs in (`redirect('/sign-in')` → proxy; `return new Response(null, { status: 401 })` → route handler; `return { kind: 'unauthenticated' }` → Server Action; `<Header user={user} />` after `requireUser()` → layout; `authClient.useSession()` → none of the server surfaces, distractor for the Client Component side).

**Components.**
- `Figure` wrapping a hand-SVG five-surface call-site map with the central call-site box and five annotated branches.
- `Tokens` on the canonical call line for the per-token reveal.
- `Buckets` for the ten-line five-surface sort.
- `Aside` (`note`) below the map: "one call, five reactions. The differences live in what each surface does with `null`, not in how the session is read."

**Project link.** The 9.5 project's `proxy.ts` (lesson 9.5.5), `requireUser()` in the protected layout (9.5.5), and the sign-out Server Action (9.5.5) are three of the five surfaces lit up on the map; the student arrives knowing the same call shape covers all three and writes the proxy, layout read, and action with the same idiom.

## Concept 8 — `React.cache` per-request memoization, and why `'use cache'` is the wrong reach

**Why it's hard.** The session is read multiple times per request — the proxy reads it, the layout reads it for the header, a Server Component leaf reads it for the sidebar, a Server Action might read it again. Without memoization each call hits the cookie cache (or the DB). The fix is `cache()` from React — a request-scoped wrapper that makes every call inside the same request return the same Promise, evaluated once. The trap: React 19 / Next.js 16 also ship `'use cache'`, a *cross-request* directive that caches across users. Wrapping `getCurrentUser` in `'use cache'` is a critical bug — user A's session-derived UI gets served to user B because the cache key doesn't include the cookie. The student needs the two caches as different tools with different scopes, not as variants of the same idea.

**Ideal teaching artifact.** Mechanics + Pattern archetype delivered as a **two-scope side-by-side wiring panel** inside `CodeVariants`. Tab one *(`React.cache`, correct)*: shows `getCurrentUser = cache(async () => { const s = await auth.api.getSession({ headers: await headers() }); return s?.user ?? null; })`, with a small hand-SVG-inside-`Figure` to the right showing one request's lifecycle — the layout calls `getCurrentUser()`, returns Promise A; the Server Component leaf calls `getCurrentUser()`, gets the *same* Promise A back; the Action calls `getCurrentUser()`, same Promise A. *One DB hit (or one cookie verify), three reads.* Tab two *(`'use cache'`, broken)*: shows `getCurrentUser = async () => { 'use cache'; … }` with the same diagram, but now showing two requests from two different users — user A's first request populates the cache; user B's request hits the same cache key (no cookie in the key, the function takes no arguments) and gets user A's session back. *One cookie's worth of data leaked across users.* The caption underneath names the rule: *`React.cache` is request-scoped, perfect for per-request deduplication; `'use cache'` is cross-request, perfect for shared static data, never for per-user reads.* A second short paragraph names the two helpers downstream chapters call — `getCurrentUser()` (returns `User | null`) and `requireUser()` (calls `redirect('/sign-in')` when null) — both wrapped in the same `cache()` so the request-scoped property holds at every call site.

**Engagement.** A `MultipleChoice`: "a teammate's PR wraps `getCurrentUser` in `'use cache'` to 'avoid re-reading the session three times per page.' What's the failure mode?" The four options frame the candidate answers (slower because cache is too wide; eventual consistency on revoke; the cache key skips cookies and one user gets another user's data; nothing, the cache key auto-includes auth state). The right answer ties the bug to the cache-key shape — `'use cache'` keys on the function arguments, the function takes none, the cookie isn't an argument, every user collides on the same cache entry. Confirmed by a `Sequence` ordering: the student arranges four memoization choices on a per-data-shape ladder — *static marketing copy* → `'use cache'`; *per-org dashboard query* → `'use cache'` with the org-id as an argument; *per-request user session* → `React.cache`; *per-request derived computation off the user* → `React.cache` (composes with the cached `getCurrentUser`).

**Components.**
- `CodeVariants` for the two-tab `React.cache` vs `'use cache'` panel, with hand-SVG-inside-`Figure` request-lifecycle diagrams embedded in each tab.
- `MultipleChoice` for the cross-user-leak diagnosis.
- `Sequence` for the four-shape memoization ladder.
- `Aside` (`caution`) below the variants: "`React.cache` is per-request; `'use cache'` is cross-request. For per-user reads, the second one is a data leak. The helpers `getCurrentUser` and `requireUser` exist precisely so the right wrapper is centralized."

**Project link.** The 9.5 project's `lib/auth.ts` (lesson 9.5.5) exports `requireUser()` wrapped in `cache()`; the layout calls it once per request and the per-request memoization is the *why* the layout doesn't need to thread `user` as a prop down every tree.
